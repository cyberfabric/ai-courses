The Big Idea

AI models never see text — they see token IDs. BPE builds a vocabulary by repeatedly merging the most frequent character pairs: count pairs → merge the winner → repeat. Common words become single tokens; rare words split into known subwords.

📊 5 Merges on “low lower newest widest”

  1. Merge 1: l + o → lo

    Most frequent pair (count: 7). Vocab: 12 tokens.

  2. Merge 2: lo + w → low

    Next winner (count: 7). “low” becomes a recognizable piece.

  3. Merge 3: e + s → es

    Common suffix pattern emerges (count: 5).

  4. Merge 4: es + t → est

    The “-est” suffix is captured (count: 5).

  5. Merge 5: low + _ → low_

    “low” becomes a single token (count: 5). Most common word = most compressed.

🎒 Core Concepts

📑
Merge Table
The tokenizer's recipe — a priority-ordered list of merge rules. Apply in order to encode any text.
🔢
Token ID
Each token maps to a unique number. “hello” might be 15339. The model only ever sees these numbers.
👁
Character Blindness
Models see tokens, not letters. “strawberry” → [str, aw, berry]. Counting r's requires reverse-engineering tokens.
🌐
Multilingual Tax
Languages underrepresented in training get 2–3× more tokens for the same meaning. Less context, slower, more expensive.
⚫
Special Tokens
BOS (start), EOS (stop), PAD (fill), UNK (unknown). Control signals the model uses to manage sequences.
💰
Vocab Size Tradeoff
Bigger vocab = fewer tokens per text (faster, more context) but larger embedding table (more parameters).

⚠️ Tokenization Gotchas

␣
Leading Spaces
“ hello” and “hello” can be different tokens. Invisible whitespace changes the token sequence.
💻
Code Is Expensive
Unusual patterns like === or </div> don't merge well, producing many small tokens.
📏
Inconsistent Limits
4K tokens ≠ 4K words. English averages ~1.3 tokens/word; Japanese can be 3–4 tokens/word.

📖 Key Terms

Tokenization
Converting text into a sequence of token IDs that a model can process.
BPE (Byte Pair Encoding)
Algorithm that builds a vocabulary by repeatedly merging the most frequent adjacent pairs.
Subword Token
A piece between a character and a full word. Common words stay whole; rare words split into subwords.
Merge Table
Priority-ordered list of merge rules. Applied in sequence to encode new text.
Token ID
The integer a token maps to. The model only sees these numbers, never raw text.
Special Tokens
Control tokens: BOS (start), EOS (stop generating), PAD (fill batches), UNK (unknown).
Vocabulary Size
Number of unique tokens. Tradeoff: bigger = fewer tokens per text but larger embedding table.
Byte-level BPE
Starts with 256 bytes as base vocab. Can encode anything — no UNK tokens ever needed.

❔ Quick Quiz

Why can’t models easily count the r’s in “strawberry”?

Because “strawberry” is split into tokens like [str, aw, berry]. The model sees 3 tokens, not 10 letters. It was never trained on the character composition of each token.

What is the BPE algorithm in one sentence?

Count adjacent character pairs, merge the most frequent, repeat until vocabulary reaches target size.

Why does Japanese text use more tokens than English for the same meaning?

BPE training data is mostly English, so English patterns get merged aggressively. Japanese characters remain less merged, producing 2–3× more tokens.

A model has 100K vocab and 4096-dim embeddings. How many embedding parameters?

100,000 × 4,096 = 409 million parameters just for the embedding layer.

Why do subword tokens beat word-level tokens?

Word-level can’t handle words not in its vocabulary. Subword splits unknown words into known pieces — it can encode any text by composing familiar subwords.

📌 Key Takeaways

  • Models see token IDs, not text. Tokenization is the bridge between human language and neural network math.
  • BPE algorithm: start with characters, count pairs, merge the most frequent, repeat. That’s it.
  • Subwords win because common words stay whole and rare words split into known pieces.
  • The merge table is the tokenizer’s recipe — stored alongside the model, applied in order.
  • Character blindness is fundamental: letter-counting and anagrams are hard because tokens hide individual characters.
  • Vocab size tradeoff: larger vocab = fewer tokens but bigger embedding table.
  • Multilingual tax: non-English languages often use 2–3× more tokens for the same content.
