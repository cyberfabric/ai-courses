Chapter 10

Context, Memory & the KV Cache

Memory Map
💡

The Big Idea

The KV cache trades memory for compute: instead of recomputing K and V for every token at every step, we store them and look them up. This makes generation practical — but the cache must fit in GPU memory, which is why context windows have hard limits.

🎒 Core Concepts

💾
KV Cache
Store K and V vectors for already-processed tokens. Only compute new ones. Trades memory for compute.
📷
Sliding Window
Only attend to the last W tokens. Memory = O(W) instead of O(n). Info propagates indirectly through layers.
🤝
GQA
Groups of Q heads share K/V. 8 KV groups for 64 Q heads = 75% cache savings with minimal quality loss.
🚫
Lost in the Middle
Models attend more to the beginning and end of context. Info in the middle gets less attention. Put important stuff first or last.
🔄
RoPE
Rotary Position Embedding. Rotates Q/K vectors by position-dependent angles. Enables context extrapolation beyond training length.
⚡
Prompt Caching
Reuse KV cache from shared prompt prefixes across requests. Same system prompt = skip that prefill. Up to 15× faster TTFT.

📖 Key Terms

KV Cache
Stored Key and Value vectors from previous tokens, reused to avoid redundant computation during generation.
Prefill
Phase 1: process the entire prompt in parallel, building the initial KV cache. Determines TTFT.
Decode
Phase 2: generate tokens one at a time, appending to cache each step. Determines TPS.
TTFT (Time to First Token)
Latency before the first output token. Grows with prompt length (more prefill work).
TPS (Tokens Per Second)
Generation speed once streaming starts. Roughly constant regardless of prompt length.
Sliding Window Attention
Attend only to the last W tokens. Caps memory at O(W). Info propagates indirectly through layers.
GQA (Grouped Query Attention)
Share K/V across groups of Q heads. Reduces KV cache size by 75–90% with minimal quality loss.
Prompt Caching
Server-side reuse of KV cache from shared prompt prefixes across API requests.

❔ Quick Quiz

Why can K and V be cached but Q can’t?

In a causal (decoder-only) model, a token’s K and V never change regardless of what comes after. But Q is only needed for the current token — each step has a new Q for the new token being generated.

Why does a 10K-token prompt make you wait longer before text appears?

The prefill phase must process all 10K tokens to build the KV cache before the first output token can be generated. More prompt tokens = longer TTFT. But once streaming starts, TPS is the same.

How much KV cache does one token need at GPT-3 scale?

~4.5 MB. That’s 96 layers × 96 heads × 128 dim × 2 (K+V) × 2 bytes = 4,608 KB per token.

Why doesn’t “128K context” mean the model equally understands all 128K tokens?

The “lost in the middle” problem: models attend more to the beginning and end of context. Information buried in the middle receives less attention and is more likely to be missed.

How does GQA with 8 KV groups and 64 Q heads reduce cache by 75%?

Standard MHA stores 64 K + 64 V = 128 vectors per layer. GQA stores 8 K + 8 V = 16 vectors. 16/128 = 12.5%, so savings = 87.5%. (The exact savings depend on the ratio.)

📌 Key Takeaways

  • KV cache = memory for compute tradeoff. Store K/V from previous tokens instead of recomputing them. Savings grow quadratically.
  • Two phases: Prefill (process prompt, build cache, determines TTFT) and Decode (stream tokens, append to cache, determines TPS).
  • ~4.5 MB per token at GPT-3 scale. 128K tokens = 576 GB of KV cache alone.
  • Context limits are memory limits. The KV cache must fit in GPU memory. More context = more memory, period.
  • Attention is O(n²). Double the context = 4× the attention compute.
  • GQA shares K/V across Q head groups — 75–90% cache savings.
  • Lost in the middle: models attend unevenly. Put important info at the start or end.
