Chapter 10

Context, Memory & the KV Cache

How the KV cache works mechanically, why context windows have hard limits, and the memory math behind it all

Companion: This expands on AI-Intro Ch 8: Context Windows. There you learned that models have a maximum context size. Now you’ll understand why that limit exists and what the KV cache actually does under the hood.

In Chapter 8, you traced autoregressive generation: predict one token, append it, predict the next. In Chapter 9, you saw how sampling strategies shape each prediction. But there’s a critical performance question we skipped: every time the model generates a new token, does it really reprocess the entire sequence from scratch?

Without optimization, yes. And that’s brutally expensive. The KV cache is the trick that makes generation practical. By the end of this chapter, you’ll understand exactly what gets cached, how much memory it costs, and why context windows have hard limits.

The Problem: Redundant Computation

Recall from Chapter 6: attention requires computing Q, K, and V for every token, then computing attention scores between all pairs. In autoregressive generation, we add one new token at each step.

The Naive Approach (No Cache)

At each generation step, recompute Q, K, V for every token in the sequence, including all the ones that haven’t changed. Then compute the full attention matrix.

  • Step 1: Process tokens [The] — compute Q, K, V for 1 token
  • Step 2: Process tokens [The, cat] — recompute Q, K, V for 2 tokens
  • Step 3: Process tokens [The, cat, sat] — recompute Q, K, V for 3 tokens
  • Step 100: Process all 100 tokens — recompute Q, K, V for 100 tokens

Total K/V computations across 100 steps: 1 + 2 + 3 + … + 100 = 5,050. But most of that work is wasted — “The” doesn’t change between steps!

The Key Insight

In a decoder-only model (GPT, Claude, Llama), tokens can only attend to previous tokens (the causal mask from Chapter 7). This means the K and V vectors for token 1 are always the same, regardless of what comes after. Token 2’s K and V are always the same. Only the newest token’s K and V need to be computed.

So why not save the K and V vectors as we go and just look them up next time?

The KV Cache: Save and Reuse

How the KV Cache Works

Instead of recomputing K and V for all tokens at every step:

  1. Compute K and V for the new token only
  2. Append them to the cached K and V from previous steps
  3. Compute Q for the new token only
  4. Attend: use Q (new token) against all cached K and V

Result: each generation step only computes K, V, and Q for one token instead of all N tokens.

The Tradeoff

The KV cache trades memory for compute. Instead of recomputing K and V vectors (expensive), we store them in GPU memory (takes space). For a 5-token generation, we computed 7 K/V pairs instead of 25. For a 1000-token generation, it’s 1,003 instead of 501,500. The savings grow quadratically with sequence length.

Quick Check

If you’re generating 200 tokens after a 50-token prompt, how many K/V computations does the cache save?

Answer: With cache: 50 (prefill) + 200 (one per step) = 250. Without cache: 50 + 51 + 52 + … + 250 = 30,100. Savings: 29,850 redundant computations avoided!

Two Phases of Generation

The KV cache creates two distinct phases with very different performance characteristics:

Phase 1: Prefill

  • What: Process the entire prompt
  • Computes: Q, K, V for all prompt tokens
  • Parallelism: High — all tokens processed at once
  • Bottleneck: Compute (matrix multiplications)
  • Produces: The KV cache + first token prediction
  • You experience this as: The pause before text starts appearing

Phase 2: Decode

  • What: Generate tokens one at a time
  • Computes: Q, K, V for one token per step
  • Parallelism: Low — inherently sequential
  • Bottleneck: Memory bandwidth (reading the cache)
  • Produces: One token, then appends K/V to cache
  • You experience this as: Text streaming in token by token

This is why you experience two distinct latencies when using ChatGPT or Claude:

Practical impact: If you paste a 10,000-token document and ask a question, you’ll wait several seconds for the first token (long prefill). But once generation starts, it streams at the same speed as a short prompt. The KV cache from the long prefill is already built — each decode step only adds one row.

How Much Memory Does the KV Cache Need?

Here’s where the hard limits come from. The KV cache stores two vectors (K and V) per token, per layer, per head. Let’s compute the exact size.

KV Cache Memory Formula

Memory = tokens × layers × 2 × num_heads × head_dim × bytes_per_value

The “2” is for K and V. Everything else is architecture-specific.

KV Cache Size for a Toy Model

Let’s start small. Our toy model:

  • Layers: 4
  • Attention heads: 2
  • Head dimension: 3
  • Precision: float16 (2 bytes per value)
  • Sequence: 10 tokens

Per token, per layer:

2 (K+V) × 2 heads × 3 dimensions = 12 values = 24 bytes

Per token, all layers:

24 bytes × 4 layers = 96 bytes

Full 10-token sequence:

96 bytes × 10 tokens = 960 bytes

Less than 1 KB. No problem! But watch what happens when we scale up.

KV Cache Size for Real Models

Now let’s use GPT-3-scale numbers:

  • Layers: 96
  • Attention heads: 96
  • Head dimension: 128
  • Precision: float16 (2 bytes)

Per token, per layer:

2 × 96 heads × 128 dim × 2 bytes = 49,152 bytes ≈ 48 KB

Per token, all layers:

48 KB × 96 layers = 4,608 KB ≈ 4.5 MB

That’s 4.5 megabytes per token. Now let’s see what that means for different context lengths:

Why Context Windows Have Hard Limits

The context window limit is fundamentally a memory limit. Every token you add costs ~4.5 MB of GPU memory (for a GPT-3-scale model). When you hit 128K tokens, the KV cache alone needs 576 GB — and that’s on top of the model weights (350 GB for GPT-3) and other overhead. There’s no algorithm trick to make this disappear. More context = more memory, period.

Watch out: The KV cache size is per user, per request. When a cloud provider serves thousands of simultaneous users, each one has their own KV cache. This is a major reason why API pricing scales with token count — you’re literally renting GPU memory.

The Quadratic Attention Problem

The KV cache solves the redundant computation problem, but there’s a second issue: attention itself is O(n²) where n is the sequence length.

Why? Each token attends to every previous token. The attention score matrix has dimensions n × n. Double the sequence length, and you have 4× the attention scores to compute.

This quadratic scaling is the main reason long context is expensive. Going from 4K to 128K context isn’t 32× more expensive — it’s 1,024× more expensive for the attention computation.

Quick Check

If processing a 1,000-token prompt takes 100ms, how long would a 10,000-token prompt take with standard attention?

Answer: 10× longer = 100× more attention scores = 10,000ms (10 seconds). That’s the quadratic tax. In practice, optimizations like FlashAttention reduce this, but the scaling relationship holds.

Solutions: Making Long Context Possible

Researchers have developed several techniques to push context windows further. Here are the most important ones:

Sliding Window Attention

Instead of attending to ALL previous tokens, only attend to the last W tokens. This turns the memory requirement from O(n) to O(W), where W is fixed.

Sliding Window with W=4

An 8-token sequence with a window of 4. Each token can only attend to itself and the 3 tokens before it:

t1t2t3t4t5t6t7t8
t1✔
t2✔✔
t3✔✔✔
t4✔✔✔✔

t6✔✔✔✔
t7✔✔✔✔
t8✔✔✔✔

Token 8 cannot directly see token 1. But information can propagate indirectly through the layers — token 5 saw token 1, and token 8 can see token 5. With enough layers stacked, information from early tokens can still reach later ones.

Mistral’s models use sliding window attention with W=4,096. Combined with many layers, information propagates across much longer distances than the window size suggests.

Grouped Query Attention (GQA)

Standard multi-head attention has separate K and V for every attention head. GQA shares K and V across groups of heads, drastically reducing cache size.

GQA: Sharing K/V Across Heads

A model with 8 query heads and 2 KV groups:

Q HeadsShared K/V Group
Q1, Q2, Q3, Q4K/V Group 1
Q5, Q6, Q7, Q8K/V Group 2

Standard MHA: 8 Q heads × K + 8 Q heads × V = 16 vectors per token per layer

GQA with 2 groups: 8 Q heads + 2 K + 2 V = 4 K/V vectors per token per layer

Memory savings: 4/16 = 75% reduction in KV cache size!

Llama 2 (70B) uses GQA with 8 KV groups for 64 Q heads. Llama 3 continues this pattern. The quality loss is minimal because K/V vectors within a group are often similar anyway.

Multi-Query Attention (MQA)

The extreme version of GQA: all query heads share a single K and single V. Maximum memory savings, but quality can suffer.

ApproachK/V sets per layerCache savingsQuality impact
Standard MHA= num Q heads (e.g., 64)None (baseline)Best quality
GQA= num groups (e.g., 8)~75–90%Minimal loss
MQA= 1~98%Some loss

RoPE: Rotary Position Embedding

Remember positional encoding from Chapter 7? The original sinusoidal approach added a fixed position vector to each token’s embedding. RoPE takes a different approach: it rotates the Q and K vectors by an angle that depends on position.

The key advantage: RoPE allows models to extrapolate to longer sequences than they were trained on. A model trained with 4K context can sometimes handle 8K or 16K with RoPE-based techniques like “NTK-aware scaling” — you essentially adjust the rotation frequency to handle longer distances.

Why this matters: RoPE is used in Llama, Mistral, and most modern open models. It’s a big part of how the field went from 2K context windows to 128K+ in just a few years.

The Meeting Notes Analogy

The KV cache is like taking notes during a long meeting.

  • Without cache (naive): Every time someone asks a question, you re-listen to the entire meeting recording from the start. The more the meeting goes on, the longer each re-listen takes.
  • With KV cache: You write notes as the meeting progresses. When someone asks a question, you consult your notes instead of rewinding. Much faster — but your notebook has limited pages.
  • Context window limit: You run out of notebook pages. You can’t take more notes, so the meeting has to end (or you start a new notebook, losing the first half).
  • Sliding window: You erase old notes to make room for new ones. You can’t look up what was said 2 hours ago, but recent discussion is always available.
  • GQA: Instead of each team member keeping their own notes, groups share a single set. Less paper used, and most people’s notes looked similar anyway.

The “Lost in the Middle” Problem

A longer context window doesn’t automatically mean better performance. Research shows that models attend unevenly — they pay most attention to the beginning and end of the context, with reduced attention to the middle.

The lost-in-the-middle effect: If you paste a 100-page document and ask about information on page 50, the model is less likely to find it than information on page 1 or page 100. This isn’t a bug in a specific model — it’s a property of how attention distributes across long sequences.

Practical implications:

  • Put the most important context at the beginning or end of your prompt
  • Don’t assume “128K context” means the model equally processes all 128K tokens
  • For document Q&A, retrieval (RAG) + short context often beats stuffing everything into one giant prompt

Prompt Caching: Reusing Across Requests

The KV cache we’ve discussed is within a single request. But what if you make multiple requests with the same system prompt? Do you recompute K/V for that shared prefix every time?

Providers like Anthropic and OpenAI offer prompt caching: if the beginning of your request matches a previously computed prefix, the server reuses the cached K/V from that prefix. You only pay the prefill cost for the new, unique part of your prompt.

Example: Your application sends a 3,000-token system prompt with every request, plus a ~200-token user message. Without prompt caching, every request processes 3,200 tokens in prefill. With caching, only the 200 new tokens need prefill — the 3,000-token system prompt’s K/V are already in cache. That’s up to 15× faster TTFT.
