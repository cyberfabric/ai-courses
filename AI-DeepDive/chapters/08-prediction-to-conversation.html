Chapter 8

From Prediction to Conversation

Autoregressive generation traced token by token — how sentences form one prediction at a time

Companion: This expands on AI-Intro Ch 6: Text Generation. There you learned the loop. Now you’ll trace every probability distribution and selection step with real numbers.

You now know everything that happens inside a transformer: tokenization (Chapter 5), attention (Chapter 6), and the full block with residual connections, layer norm, and FFN (Chapter 7). But there’s a crucial question left: how does all of that produce a sentence?

The answer is surprisingly simple: predict one token, append it, repeat. In this chapter, you’ll trace this loop with real numbers and see how different decoding strategies shape the output.

The Fundamental Loop

Autoregressive Generation

The model predicts one token at a time. After each prediction, the new token is appended to the input, and the model runs again to predict the next one. This continues until a stop condition is met.

The key word is autoregressive: each output becomes part of the input for the next step. The model “feeds itself” its own predictions.

1. Prompt

[The, cat, sat]

→

2. Predict

Get probability for every token

→

3. Select

Pick one token

→

4. Append

[The, cat, sat, on]

→

5. Repeat

Go to step 2

That’s it. Every chatbot response, every code completion, every AI-generated story is produced by this loop. The only difference between strategies is step 3: how you pick the next token.

Every token sees everything before it

At step 3, the model processes all 5 tokens [The, cat, sat, on, the] through the transformer. The causal mask (Chapter 7) ensures each token attends only to previous tokens. The last position’s output gives the probability distribution for the next token. The model doesn’t “remember” step 1 — it reprocesses the entire sequence every time.

Greedy Decoding: Always Pick the Best

Greedy Decoding

At each step, pick the token with the highest probability. No randomness. The same prompt always produces the same output.

Greedy decoding is fast (one forward pass per token) and deterministic. But it has a problem: it can get stuck in repetitive loops.

Greedy decoding can get boring

Consider a model generating a story. At each step, the most likely continuation is the most generic one:

StepGeneratedTop token (greedy)
1The weather wasnice (0.35)
2The weather was niceand (0.40)
3The weather was nice andthe (0.30)
4The weather was nice and theweather (0.25)
5The weather was nice and the weatherwas (0.35)
6The weather was nice and the weather wasnice (0.40)

The model is stuck in a loop! Each individual choice is the most probable, but the sequence is terrible. This is the greedy trap: local optima don’t guarantee a globally good sequence.

Checkpoint

If the model always picks the most probable next token, will it always generate good text?

Answer: No. Greedy decoding can produce repetitive, boring, or looping text. The highest-probability token at each step doesn’t guarantee the best overall sequence.

Beam Search: Keeping Multiple Options Open

Beam search addresses the greedy trap by exploring multiple candidate sequences simultaneously. Instead of committing to one token at each step, it keeps the top B best sequences (called beams).

Beam Search

At each step: expand every beam with all possible next tokens, score each expanded sequence by its total probability, keep only the top B sequences. Continue until all beams produce EOS or reach max length.

Beam search with beam width B=2

Start: Prompt = “The cat”

Step 1: Get probabilities for next token. Keep top 2:

BeamSequenceP(next)Total Score
AThe cat sat0.45log(0.45) = -0.80
BThe cat is0.30log(0.30) = -1.20

Step 2: Expand each beam. Beam A tries all tokens, Beam B tries all tokens. That’s 2 × 8 = 16 candidates. Keep top 2:

BeamSequenceP(new)Cumulative Score
AThe cat sat on0.55-0.80 + (-0.60) = -1.40
BThe cat is sleeping0.40-1.20 + (-0.92) = -2.12

Step 3: Expand again. Keep top 2:

BeamSequenceCumulative Score
AThe cat sat on the-1.73
BThe cat sat on a-2.30

Notice: “The cat is sleeping” was pruned because both extensions of beam A scored better.

Why beam search helps

Greedy would commit to “sat” immediately and never explore “is sleeping.” Beam search keeps both options alive until it has enough information to decide. It finds sequences that are globally better, not just locally best. The tradeoff: B beams means B forward passes per step — it’s B times slower than greedy.

Beam search is common in translation and summarization, where accuracy matters more than creativity. For chat and creative writing, sampling (Chapter 9) is usually preferred.

When Does Generation Stop?

Three conditions can end generation:

Stop ConditionHow It WorksExample
EOS token Model generates the special <EOS> token Natural end of a response
Max length Hit the maximum token limit (e.g., 4096) Response is cut off mid-sentence
Stop sequence A specific string appears (e.g., </answer>) Structured output formats

If you’ve ever seen a chatbot response end mid-sentence, the likely cause is hitting the max length limit before generating an EOS token.

How Chat Actually Works

When you send a message to Claude or ChatGPT, it doesn’t maintain a separate conversation state. Instead, the entire conversation history is concatenated into one long sequence every time:

What the model actually sees

<BOS> [system prompt: ~500 tokens]
[user turn 1: ~50 tokens]
[assistant turn 1: ~200 tokens]
[user turn 2: ~30 tokens]
[assistant turn 2: ← generating here]

Total input for turn 2: ~780 tokens. The model processes all of them through every transformer block just to generate the first token of the response.

By turn 20 of a conversation, the input might be 15,000+ tokens. Every single one is processed through attention in every layer, every time you get a response.

Memory is an illusion

The model doesn’t “remember” the conversation. It re-reads the entire history every time. What feels like memory is actually the conversation text being fed back in as input. This is why conversations have a context window limit — eventually the history gets too long to fit. It’s also why the model can’t learn from past conversations: each interaction starts fresh.

The attention mask in chat

The causal mask (Chapter 7) still applies: each token can only attend to tokens before it. But the system prompt tokens, all previous user messages, and all previous assistant responses are all visible. This means the system prompt influences every token the assistant generates — it shapes the “personality” consistently across the entire conversation.

When you chat with Claude, you see text appearing one chunk at a time. This isn’t a dramatic effect — it’s literally how the model works.

Two phases of latency

There are two distinct delays:

  • Time to first token (TTFT): Processing the entire prompt through the transformer. Longer prompts = longer wait. This is the “thinking” phase.
  • Tokens per second (TPS): Once generation starts, each new token takes roughly the same time (~30ms for a fast model). This is constant regardless of prompt length (thanks to the KV cache, which we’ll cover in Chapter 10).

A 10,000-token prompt might take 2 seconds before you see anything, but then tokens flow at the same speed as a 100-token prompt.

Sampling vs. Greedy: A Preview

We traced greedy decoding above, but most chat models use sampling: randomly selecting from the probability distribution instead of always picking the top token.

Chapter 9 dives deep into sampling: temperature, top-k, top-p, and penalties. For now, just know that sampling introduces controlled randomness to avoid the repetitive monotony of greedy decoding.

The Committee Analogy

Autoregressive generation is like writing a story by committee where each committee member only gets to add one word:

  1. Committee member 1 reads the prompt and writes one word.
  2. Committee member 2 reads the prompt + word 1 and writes one word.
  3. Committee member 3 reads everything so far and writes one word.
  4. And so on, until someone writes “THE END.”

Here’s the twist: every committee member is the same person. They just don’t remember what they were thinking last time — they only see what was written. The model has no hidden state between tokens. Its entire “thinking” is captured in the sequence of tokens it can see.

Why This Architecture Works So Well

It seems crude: predict one token at a time, with no memory, no planning ahead. And yet this produces coherent essays, working code, and nuanced conversations. Why?

  • Enormous context: With 128K+ tokens of context, the model can “see” massive amounts of preceding text.
  • Attention is powerful: Each token can attend to any relevant token in the entire context, no matter how far back.
  • Training on trillions of tokens: The model has seen virtually every common pattern in human language.
  • The next token is surprisingly informative: Predicting the next word well requires understanding grammar, facts, logic, style, and intent. It’s a simple objective that requires deep understanding to optimize.

Key Takeaways

  • Autoregressive generation: predict one token, append, repeat. The model feeds itself its own output.
  • Greedy decoding always picks the highest-probability token. Fast but prone to repetition.
  • Beam search keeps multiple candidates alive, finding globally better sequences at B× the cost.
  • Sampling introduces randomness for more varied, natural-sounding output.
  • Chat = concatenation: the entire conversation history is re-sent every turn. “Memory” is an illusion.
  • Streaming sends each token as it’s generated — what you see is literally the model working in real time.
  • Two latency phases: TTFT (processing the prompt) and TPS (generating tokens). Long prompts increase TTFT but not TPS.
  • Stop conditions: EOS token, max length, or stop sequence.
