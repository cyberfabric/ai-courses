Chapter 3

Task Decomposition

Breaking big problems into agent-sized pieces — the art of knowing how small is small enough

Still Domain 1 — Still 27%

We're continuing our journey through Domain 1: Agentic Architecture & Orchestration (27%). In Chapter 2, you learned why multi-agent systems exist and how agents communicate. Now the question is: how do you decide what each agent should do?

Task decomposition is the process of taking a complex goal and breaking it into subtasks that individual agents can handle. Get it right and your system is fast, reliable, and debuggable. Get it wrong and you'll drown in overhead, gaps, or circular dependencies.

Think of it like project management. A project manager doesn't hand a developer a ticket that says "build the entire product." They break it into epics, stories, and tasks. Task decomposition for agents follows the same logic — just with different constraints.

Why Decomposition Matters

Three reasons you can't just throw a complex task at a single agent:

Context Smaller tasks fit in smaller context windows
Parallel Independent subtasks run simultaneously
Reliable Focused tasks produce fewer hallucinations

Context windows: A single agent processing a 50-page document, running 10 tool calls, and synthesizing results will burn through its context window. Breaking this into "extract sections" → "analyze each section" → "synthesize" keeps each step lean.

Parallelism: If you decompose a research task into five independent sub-questions, five subagents can work on them simultaneously. Sequential processing would be 5x slower.

Reliability: LLMs reason better over focused tasks. Asking "analyze this 200-line function for security vulnerabilities" produces better results than "audit this entire codebase for all possible issues." Decomposition reduces cognitive overload on the model.

Two Approaches: Static vs. Dynamic

This is one of the most exam-tested distinctions in Domain 1. The exam calls them prompt chaining (static) and adaptive decomposition (dynamic).

Static Decomposition (Prompt Chaining)

The task structure is predetermined. You know the steps in advance, and each step's input and output are predictable. Think of it as a recipe: Step 1 always leads to Step 2, which always leads to Step 3.

# Static: Fixed pipeline for invoice processing
Step 1: Extract text from PDF  →  raw_text
Step 2: Identify line items     →  items[]
Step 3: Validate against schema →  validated_items[]
Step 4: Calculate totals        →  invoice_summary

When to use it: The workflow is well-defined. Each step's input/output is predictable. There are no conditional branches based on intermediate results.

Dynamic Decomposition (Adaptive)

The agent decides the next steps at runtime based on what it discovers. You can't predict the exact workflow in advance because intermediate results change the approach.

# Dynamic: Adaptive research task
Step 1: Analyze the research question
Step 2: Based on analysis, decide sub-questions (could be 2 or 10)
Step 3: For each sub-question, choose research strategy
        → If academic: search papers
        → If technical: search docs + run code
        → If opinion-based: search forums
Step 4: Based on findings, decide if more research is needed
Step 5: Synthesize (approach depends on what was found)

When to use it: The task is unpredictable. Intermediate results may change the approach. The number of subtasks isn't known in advance.

The exam's favorite question format: "Given this scenario, should you use prompt chaining or adaptive decomposition?" The rule is simple: if the task has conditional branches or unknown complexity, the answer is adaptive decomposition. If it's a fixed, linear pipeline, the answer is prompt chaining. The distractors usually describe a rigid pipeline for an unpredictable task, or adaptive decomposition for something that's clearly sequential.
Prompt Chaining (Static)
Invoice processing: extract → parse → validate → calculate. Same steps every time. Input format is known. Output schema is fixed.
Predictable pipeline. Static decomposition is the right call.
Adaptive (Dynamic)
Debug a production error: read logs → form hypothesis → investigate → maybe check database → maybe check network → test fix → verify. Different every time depending on what you find.
Unpredictable task. Dynamic decomposition is the right call.

Hierarchical Decomposition

The most intuitive decomposition strategy. You break a big task into smaller ones, then break those into even smaller ones. It looks exactly like a project management hierarchy:

Epic: "Build customer support agent"
├── Story: "Implement ticket classification"
│   ├── Task: Classify ticket by type (billing/tech/shipping)
│   ├── Task: Extract priority from ticket text
│   └── Task: Route to correct specialist
├── Story: "Implement knowledge base search"
│   ├── Task: Query knowledge base with customer question
│   ├── Task: Rank results by relevance
│   └── Task: Format response for customer
└── Story: "Implement escalation logic"
    ├── Task: Detect unresolvable issues
    ├── Task: Prepare handoff summary for human agent
    └── Task: Transfer conversation state

In this model, the orchestrator handles the "Epic" level. Subagents handle "Story" level tasks. Individual tool calls handle the "Task" level.

The Decomposition Depth Decision. How many levels do you need? The answer depends on agent capability. If a single agent can reliably handle a "Story" level task (classify, extract, route), you don't need to decompose further. If it struggles, break the Story into Tasks and assign them to separate tool calls or subagents.

Rule of thumb: Decompose until each piece is something a single agent can handle in one focused conversation, without its context window running out. Don't go further than that.

DAG-Based Workflows

Not all subtasks are sequential or hierarchical. Sometimes you have a Directed Acyclic Graph (DAG) — a set of tasks with dependencies between them, but no cycles.

Don't let the name intimidate you. A DAG is just a fancy way of saying "some tasks depend on others, and there are no loops." Here's an example:

# DAG for a code review system
#
#   ┌─── Lint Check ──────────┐
#   │                         │
# Parse Code ─── Security Scan ─── Generate Report
#   │                         │
#   └─── Type Check ──────────┘
#
# Parse Code runs first (no dependencies)
# Lint, Security, and Type checks run in PARALLEL (all depend on Parse)
# Generate Report runs last (depends on all three checks)

The key insight: tasks that don't depend on each other can run in parallel. In this example, after parsing the code, three independent checks run simultaneously. The report waits for all three to finish.

Dependency Management

When working with DAGs, you need to manage three things:

  1. Dependency tracking: Which tasks must complete before this task can start?
  2. Data passing: What output from Task A does Task B need as input?
  3. Failure handling: If Task A fails, what happens to Tasks B, C, and D that depend on it?

In the Claude Agent SDK, the orchestrator handles this. It spawns subagents for independent tasks (parallel), waits for results, then spawns the next batch that depends on those results.

You don't need to implement a full DAG execution engine for the exam. You need to recognize when a workflow has DAG-like dependencies, identify which tasks can run in parallel, and explain the dependency chain. The exam tests architectural thinking, not implementation.

The Fan-Out / Fan-In Pattern

This is the most common DAG pattern you'll see on the exam, and it's beautifully simple:

Decompose
→
Fan Out (parallel)
→
Fan In (aggregate)
→
Synthesize
  1. Fan-out: The orchestrator breaks the task into N independent subtasks and spawns N subagents in parallel.
  2. Fan-in: The orchestrator collects all N results and aggregates them into a single output.

Think of it like MapReduce: fan-out is the Map phase (distribute work), fan-in is the Reduce phase (combine results).

# Fan-out / Fan-in for multi-source research
orchestrator_prompt = """
Research the impact of AI on healthcare.
Spawn these subagents in parallel:
"""

# Fan-out: 3 parallel subagents
subagent_1: "Search academic papers for AI in diagnostics"
subagent_2: "Search news articles for AI in drug discovery"
subagent_3: "Search regulatory databases for AI medical device approvals"

# Fan-in: orchestrator collects all 3 results
# Synthesize: orchestrator writes final report combining all findings

In the Multi-Agent Research System scenario, this is the primary pattern. The orchestrator decomposes a research query into sub-questions, fans out to parallel researcher agents, fans in the results, and synthesizes a report.

When the exam describes a scenario where "multiple independent data sources need to be queried and results combined," the answer is fan-out/fan-in. Watch for distractors that suggest sequential processing of each source — if sources are independent, parallel execution is always preferred.

Granularity: Too Coarse vs. Too Fine

One of the hardest judgment calls in decomposition is getting the granularity right. Both extremes are bad:

Too Coarse
"Analyze this entire 500-page legal contract and extract all relevant terms, obligations, deadlines, penalties, and exceptions."
One agent can't handle this reliably. Context will overflow. The task is too broad for focused reasoning.
Too Fine
"Read paragraph 3 of page 12. Extract the date mentioned. Now read paragraph 4 of page 12. Extract the dollar amount. Now read paragraph 5..."
Too many tiny subtasks. Each has overhead (spawning, context setup, result passing). The overhead exceeds the value.

The Goldilocks Zone

The right granularity depends on the task, but here are guidelines:

Signal Too Coarse Too Fine Just Right
Context usage Agent runs out of context Most context is overhead Agent uses 30–70% of window
Task clarity Agent gets confused by scope Task is trivially simple Agent can focus without ambiguity
Error rate Agent makes mistakes from overload Coordination errors dominate Each subtask succeeds reliably
Parallelism Can't parallelize — it's one blob Too many parallel tasks to manage 3–7 parallel subtasks
Anti-pattern: over-decomposition. A common exam distractor suggests breaking every task into the smallest possible pieces "for maximum parallelism." But each subagent has overhead: spawning time, context setup, result passing back to the orchestrator, and aggregation cost. If you decompose a 10-minute task into 100 subtasks, the orchestration overhead alone might take longer than doing the task sequentially. The exam tests whether you recognize this tradeoff.

Error Handling at the Subtask Level

When a subtask fails, the orchestrator has three options:

1. Retry

Run the same subtask again. Works for transient failures (API timeout, rate limit). Use exponential backoff. Set a max retry count.

2. Skip

Mark the subtask as failed and continue without it. Works when the subtask's output is optional or when partial results are acceptable. In the research scenario, if one of five sources is unavailable, the report can still use the other four.

3. Escalate

Stop the workflow and escalate to a human or a higher-level agent. Required when the failed subtask is critical and the overall task can't proceed without it.

# Error handling in an orchestrator
for subtask in subtasks:
    try:
        result = run_subagent(subtask)
        results.append(result)
    except TransientError:
        # Retry with exponential backoff
        result = retry_with_backoff(subtask, max_retries=3)
        results.append(result)
    except CriticalError:
        # Escalate — can't continue without this subtask
        return escalate_to_human(subtask, error_context)
    except NonCriticalError:
        # Skip — continue with partial results
        results.append({"status": "skipped", "reason": str(e)})
Design pattern: classify subtasks by criticality upfront. Before decomposing, tag each subtask as "critical" (must succeed for the whole task to succeed) or "optional" (nice to have, can skip on failure). This lets the orchestrator make error-handling decisions automatically without case-by-case logic.

In the Customer Support Agent scenario: looking up the customer's account is critical (you can't help them without it). Searching the knowledge base for similar issues is optional (you can still help based on the account info alone).

Real-World Decomposition: The Code Review Example

Let's walk through a realistic exam-style decomposition. Suppose the Code Generation with Claude Code scenario asks you to design a system that reviews a large pull request:

Step 1: Assess the Input

The PR has 15 changed files across 3 directories. Some files are related, others are independent.

Step 2: Choose the Strategy

This is a hybrid: static decomposition for the overall structure (every PR review follows the same phases), dynamic within each phase (what to look for depends on what you find).

Step 3: Decompose

# Phase 1: Per-file analysis (fan-out, parallel)
for each changed_file:
    subagent: "Review {file} for bugs, style, and logic issues"

# Phase 2: Cross-file integration (sequential, depends on Phase 1)
subagent: "Check for consistency across all 15 files.
           Look for: conflicting changes, missing imports,
           broken cross-references"

# Phase 3: Summary (sequential, depends on Phase 2)
subagent: "Synthesize all findings into a PR review comment.
           Prioritize by severity: critical → warning → suggestion"

Notice the structure: fan-out for independent per-file reviews, then sequential for cross-cutting concerns, then synthesis. This is a common pattern the exam expects you to recognize.

For code review scenarios, the exam specifically looks for per-file analysis plus a separate cross-file integration pass. A common trap is answering "just review all files in sequence" (too slow, no parallelism) or "review everything in one pass" (context overflow on large PRs). The correct architecture separates independent analysis from cross-cutting analysis.

Decomposition Decision Framework

When the exam gives you a scenario and asks how to decompose it, use this decision tree:

  1. Is the workflow predictable?
    • Yes → Use static decomposition (prompt chaining)
    • No → Use dynamic (adaptive) decomposition
  2. Are there independent subtasks?
    • Yes → Fan-out for parallel execution
    • No → Sequential pipeline
  3. Do subtasks have complex dependencies?
    • Yes → Model as a DAG, identify the critical path
    • No → Simple fan-out/fan-in or pipeline
  4. Is each subtask a manageable size?
    • Too big → Decompose further (hierarchical)
    • Too small → Merge related subtasks
    • Right-sized → You're done

Scenario Connections

Here's how task decomposition shows up in each exam scenario:

Scenario Decomposition Pattern
Customer Support Dynamic: classify issue → route to specialist → resolve or escalate. Branching depends on issue type.
Code Generation Hybrid: static phases (plan → implement → test), dynamic within each phase.
Multi-Agent Research Fan-out/fan-in: decompose query into sub-questions, parallel research, aggregate results.
Developer Productivity Dynamic: adapt workflow based on developer's current task and discovered context.

Exercise: Decompose a Research Task

You're building the Multi-Agent Research System for a question: "What are the environmental, economic, and social impacts of electric vehicles in Europe?"

  1. Is this static or dynamic decomposition? Why?
  2. Identify the natural sub-questions (hint: the question itself suggests a fan-out structure).
  3. For each sub-question, what type of source should the subagent search?
  4. Which subtasks can run in parallel? Which have dependencies?
  5. How should the orchestrator aggregate the results?
  6. What's your error-handling strategy if one research subagent fails?

Check yourself: Three natural sub-questions (environmental, economic, social). All three are independent → fan-out in parallel. Aggregation requires a synthesis step. If one fails, you can produce a partial report (skip strategy) since each sub-question is independently valuable.

Quick Review

  • Static decomposition (prompt chaining) for predictable pipelines; dynamic (adaptive) decomposition for unpredictable tasks with conditional branches
  • Fan-out/fan-in is the key pattern: parallelize independent subtasks, then aggregate results — think MapReduce
  • Granularity matters: too coarse overwhelms the agent, too fine drowns in overhead — aim for 3–7 parallel subtasks
  • Error handling: retry (transient), skip (optional subtask), escalate (critical) — classify subtask criticality upfront
  • DAG dependencies: identify which tasks can run in parallel and which must wait — this directly impacts overall execution time
