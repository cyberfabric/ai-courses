Chapter 5

Designing Agent Tools

Tool schemas, descriptions, granularity, and error handling — the interface between the agent's brain and the outside world

Welcome to Domain 2

We're entering Domain 2: Tool Design & MCP Integration, which makes up 18% of the exam (~11 questions). This chapter covers tool design; Chapter 6 covers MCP. Together they prepare you for every Domain 2 question.

Tools are how Claude interacts with the outside world. Without tools, an agent is just a chatbot — it can only generate text. With tools, it can search databases, call APIs, read files, create resources, and take real actions. Tool design is where architecture meets implementation.

Why Tool Design Matters on the Exam

The biggest source of exam mistakes in Domain 2 is not understanding how tool descriptions drive Claude's tool selection. Claude picks tools based on their descriptions, not their names. A poorly described tool won't get used — or worse, it'll get used incorrectly. The exam tests this heavily.

Anatomy of a Tool Definition

Every tool in the Claude API has three components:

{
  "name": "search_knowledge_base",
  "description": "Search the customer support knowledge base for articles
    matching a query. Returns the top 5 most relevant articles with titles,
    URLs, and relevance scores. Use this tool when a customer asks a question
    that might be answered by existing documentation. Do NOT use this for
    account-specific queries — use lookup_account instead.",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "The search query. Use keywords from the customer's
          question. Prefer specific terms over broad ones."
      },
      "category": {
        "type": "string",
        "enum": ["billing", "technical", "shipping", "general"],
        "description": "Filter results by category. Omit for cross-category search."
      },
      "max_results": {
        "type": "integer",
        "description": "Maximum number of results to return. Default: 5.",
        "default": 5
      }
    },
    "required": ["query"]
  }
}

Let's break down each component:

1. Name

Use descriptive, distinct names. search_knowledge_base is clear. search is ambiguous when you have multiple search tools. skb is cryptic. The name should immediately tell Claude (and you) what the tool does.

2. Description (The Most Important Part)

This is by far the most critical field. Anthropic's official guidance: aim for at least 3–4 sentences per tool description. Include:

  • What the tool does
  • When to use it (and when NOT to use it)
  • What it returns
  • Important caveats or limitations
The exam tests the principle that tool descriptions are the primary mechanism for tool selection. Claude reads the descriptions to decide which tool to call. If two tools have similar names but different descriptions, Claude picks based on the descriptions. If a description is vague, Claude may call the tool at the wrong time or with wrong parameters. Exam questions frequently present a scenario where the agent picks the wrong tool, and the answer is "improve the tool description."

3. Input Schema

A JSON Schema object defining the parameters Claude should pass. Key design decisions:

  • Required vs. optional: Only mark parameters as required if the tool literally can't function without them
  • Enums: Use enums when a parameter has a fixed set of valid values — this constrains Claude's output
  • Descriptions on parameters: Each parameter should have its own description explaining format and usage
  • Defaults: Specify defaults so Claude doesn't need to guess

Tool Description Design: Good vs. Bad

This is the most exam-relevant section in this chapter. Let's see the difference:

Bad Description
"Searches the database."
Which database? What kind of search? What does it return? When should Claude use it vs other search tools? This description gives Claude nothing to work with.
Good Description
"Search the customer support knowledge base for articles matching a query. Returns top 5 articles with titles, URLs, and relevance scores. Use when the customer asks a question that might be in existing docs. Do NOT use for account-specific queries — use lookup_account instead."
What it does, what it returns, when to use it, when NOT to use it. Claude can make an informed decision.

The "When NOT to Use" Pattern

Including when NOT to use a tool is just as important as saying when to use it. This is especially critical when you have similar tools:

// Tool 1
"name": "search_knowledge_base",
"description": "...Use for general questions about policies and procedures.
  Do NOT use for customer-specific data like orders or account details."

// Tool 2
"name": "lookup_account",
"description": "...Use when you need customer-specific information like
  order history, account status, or billing details. Do NOT use for
  general policy questions — use search_knowledge_base instead."

Each tool explicitly says "use me for X, not for Y — use that other tool for Y." This cross-referencing eliminates ambiguity.

The tool description is an instruction manual for an agent. Think of Claude as a new employee who's never seen your system before. The tool descriptions are the only documentation they have. Would they know which tool to use, when, and how? If not, your descriptions need work.

Design principle: Write tool descriptions as if you're training someone who's smart but has zero context about your system. Include the what, when, when-not, returns, and caveats. This single practice eliminates most tool selection errors.

Tool Granularity: One-Purpose vs. Swiss Army Knife

How many tools should you give an agent? And how broad should each tool be?

One-Purpose Tools

Each tool does exactly one thing: get_order_status, process_refund, update_address. Clear, testable, easy to describe. But if you have 30 of them, Claude wastes reasoning tokens deciding which one to use.

Swiss Army Knife Tools

One tool that does many things: customer_operations(action="refund"), customer_operations(action="lookup"). Fewer tools to choose from, but the description gets complex and Claude might pass the wrong action.

The Sweet Spot

Anthropic's recommendation: consolidate related operations into fewer, more capable tools. This reduces selection ambiguity. But each tool should still have a clear purpose. The ideal is 5–10 well-described tools per agent, not 30 micro-tools or 1 mega-tool. Think of it like REST API design: you don't want one endpoint for everything, but you also don't want 100 endpoints when 10 would do.
Approach Pros Cons When to Use
Many small tools (20+) Clear responsibility Selection confusion, token cost Only when tool overlap is zero
Few broad tools (2–3) Easy selection Complex parameters, hard to describe Simple agents with few capabilities
Balanced (5–10) Clear + manageable Requires good descriptions Most production agents
When the exam says an agent "frequently selects the wrong tool" or "fails to use available tools," the most common correct answer is either: (1) improve the tool descriptions, or (2) reduce the number of tools. If an agent has 25 tools, the exam expects you to recognize that tool overload is the likely cause. The fix is consolidation and better descriptions.

The Reasoning Boundary

One of the most important architectural decisions: what should the agent decide, and what should the tool enforce?

The agent (Claude) is good at:

  • Understanding natural language
  • Deciding which tool to call and with what parameters
  • Interpreting results and deciding next steps
  • Synthesizing information from multiple sources

The tool (your code) is good at:

  • Deterministic operations (calculations, database queries)
  • Enforcing business rules (validation, authorization)
  • Accessing external systems (APIs, databases)
  • Returning structured, reliable data
# BAD: Agent does the calculation
Agent: "The customer ordered 3 items at $29.99 each.
        The total is... $89.97"  # What if it hallucinates?

# GOOD: Tool does the calculation
Agent calls: calculate_order_total(order_id="12345")
Tool returns: {"total": 89.97, "items": 3, "subtotal": 89.97, "tax": 0}
# Deterministic. Correct every time.
Never let the agent do math, date calculations, or business logic. LLMs are unreliable at arithmetic. If the agent needs to calculate a refund amount, total, or date difference, that's a tool's job. The agent decides when to calculate; the tool does the calculation. This boundary is tested on the exam.

Parallel Tool Calls

Claude can request multiple tool calls in a single response. When it does, those calls can execute in parallel:

# Claude's response includes multiple tool_use blocks
[
  {"type": "tool_use", "name": "lookup_account", "input": {"customer_id": "C123"}},
  {"type": "tool_use", "name": "search_knowledge_base", "input": {"query": "return policy"}},
  {"type": "tool_use", "name": "get_order_status", "input": {"order_id": "O456"}}
]

# Your code executes all three in parallel
# Then returns all three tool_results in the next message

This is a significant performance optimization. Instead of three round trips to the API (call tool 1 → return result → call tool 2 → ...), everything happens in one round trip.

For parallel tool calls to work well, the tools should be independent — the output of one shouldn't be needed as input to another. Claude is smart enough to call tools in parallel only when their inputs don't depend on each other.

Error Responses That Help the Agent Recover

When a tool fails, what you return in the tool_result determines whether the agent recovers gracefully or spirals into confusion.

Structured Error Responses

The exam specifically tests a structured error format with three fields:

# Structured error response
{
  "type": "tool_result",
  "tool_use_id": "toolu_abc123",
  "is_error": true,
  "content": {
    "error": "Customer not found",
    "errorCategory": "NOT_FOUND",
    "isRetryable": false,
    "suggestion": "Verify the customer ID. The customer may have
      been merged or deleted. Try searching by email instead."
  }
}

The key fields:

Field Purpose Why It Matters
is_error Marks the result as an error Tells Claude not to treat this as a successful result
errorCategory Classifies the error type Helps Claude decide whether to retry, try another approach, or give up
isRetryable Whether retrying might succeed Prevents infinite retry loops on permanent failures
suggestion What the agent should try instead Guides recovery without requiring the agent to guess
Bad Error Response
"Error: something went wrong"
Claude has no idea what failed, whether to retry, or what to do next. It'll either give up or retry blindly.
Good Error Response
"Error: Rate limit exceeded (429). isRetryable: true. Suggestion: Wait 5 seconds and retry. The API allows 100 requests per minute."
Claude knows it's temporary, should retry after a delay, and understands the constraint.
The exam tests whether you understand the is_error, isRetryable, and errorCategory fields in tool results. A common exam pattern: "The agent keeps retrying a failed tool call. What's the most likely cause?" Answer: the error response doesn't include isRetryable: false, so the agent doesn't know the error is permanent.

Strict Tool Use

The Claude API supports a strict: true option on tool definitions. When enabled, Claude's tool calls are guaranteed to conform to your JSON Schema exactly — no extra fields, no wrong types, no missing required parameters.

{
  "name": "process_refund",
  "description": "Process a customer refund...",
  "strict": true,  // Guarantees schema conformance
  "input_schema": {
    "type": "object",
    "properties": {
      "order_id": {"type": "string"},
      "amount": {"type": "number"},
      "reason": {"type": "string", "enum": ["defective", "wrong_item", "changed_mind"]}
    },
    "required": ["order_id", "amount", "reason"]
  }
}

Without strict: true, Claude usually follows the schema — but occasionally it might add extra fields or use slightly wrong types. With it enabled, schema conformance is guaranteed. Use it for any tool where parameter correctness is critical.

Tool Testing Strategies

Tools should be testable independently of the model. This is a key architectural principle:

  1. Unit test the tool itself: Does search_knowledge_base return the right results for a given query? Test this without Claude in the loop.
  2. Test the schema: Does the input_schema validate correctly? Are required fields actually required?
  3. Test error handling: What happens when the tool gets invalid input? Does it return a structured error that helps the agent?
  4. Integration test with the agent: Does Claude call the tool at the right time with reasonable parameters? This is the last step, not the first.
Design pattern: tools as a testable API layer. Treat your tools like a REST API. Each tool has a well-defined interface (input schema), documented behavior (description), predictable error responses, and unit tests. The agent is just another client calling this API.

This separation is powerful: you can improve tool reliability without touching the agent, and you can improve agent behavior without changing tools. It's the same reason we separate frontend from backend in web development.

Tool Versioning and Backward Compatibility

In production, tools evolve. You might add parameters, change return formats, or deprecate old tools. Best practices:

  • Add optional parameters instead of changing existing ones — old agents still work
  • Return format changes should be additive (add fields, don't remove them)
  • Deprecation path: If you must remove a tool, update the agent's system prompt to stop using it before removing the tool from the available tools list

Tool Choice Parameter

The tool_choice parameter in the API controls how Claude selects tools:

Value Behavior Use Case
"auto" Claude decides whether to use a tool Default for most agents
"any" Claude must use at least one tool Force tool use when you know it's needed
{"type": "tool", "name": "..."} Claude must use this specific tool When you know exactly which tool should run
"none" Claude can't use any tools Force text-only response
In most agent loops, you use "auto" and let Claude decide. But sometimes you know a tool call is needed — for example, in the Structured Data Extraction scenario, you might force tool_choice: {"type": "tool", "name": "extract_fields"} to guarantee structured output rather than freeform text.

Scenario Connections

Here's how tool design shows up in each exam scenario:

  • Customer Support: Tools for ticket lookup, knowledge base search, refund processing. Description design is critical for routing to the right tool. Error responses guide recovery when account lookups fail.
  • Code Generation: Tools like FileRead, FileWrite, Bash. These follow Anthropic's own tool design patterns from Claude Code.
  • Multi-Agent Research: Search tools for each subagent. Consistent return formats across tools so the orchestrator can aggregate results.
  • Developer Productivity: MCP-connected tools for internal systems. Tool descriptions must explain the internal system's quirks.
  • CI/CD: Tools for running tests, checking linting, deploying. Each tool needs clear success/failure signaling.
  • Data Extraction: Tools with strict schemas for structured output. strict: true is essential here.
Exercise: Design a Tool Set for Customer Support

Design 5 tools for the Customer Support Resolution Agent scenario. For each tool, write:

  1. The name
  2. A description (at least 3 sentences: what it does, when to use it, when NOT to use it)
  3. The input_schema with required and optional parameters
  4. An example error response with errorCategory and isRetryable

Suggested tools: lookup_customer, search_knowledge_base, get_order_status, process_refund, escalate_to_human.

Check yourself: Each description should cross-reference related tools ("For X, use Y instead"). The process_refund tool should have a reason enum, not a freeform string. escalate_to_human should require a summary parameter. Error responses should distinguish retryable (API timeout) from permanent (customer not found) failures.

Quick Review

  • Tool descriptions are the #1 factor in tool selection — write 3–4 sentences minimum, include when-to-use and when-NOT-to-use
  • Aim for 5–10 well-described tools per agent — too many causes selection confusion, too few means over-complex parameters
  • The reasoning boundary: agent decides what to do, tool does the deterministic work — never let the agent do math or business logic
  • Structured error responses need is_error, errorCategory, isRetryable, and a suggestion — vague errors cause blind retries
  • Use strict: true for tools where parameter correctness is critical — guarantees schema conformance
