Plan Mode: Think Before You Act

Plan mode is a permission mode where Claude reads and analyzes code without making changes. It's the "look before you leap" approach.

How It Works

  • Claude can read files, search code, and analyze the codebase
  • Claude cannot write files, execute commands, or make changes
  • Claude uses AskUserQuestion to gather requirements and clarify goals
  • The output is a plan — a detailed proposal of what to do

How to Enter Plan Mode

# Start a session in plan mode
claude --permission-mode plan

# Switch during a session: press Shift+Tab twice
# Normal → Auto-Accept → Plan Mode

# Run a one-off analysis in plan mode
claude --permission-mode plan -p "Analyze the auth system and suggest improvements"

When to Use Plan Mode

  • Multi-step implementation: When a feature requires editing many files, plan first
  • Code exploration: When researching an unfamiliar codebase
  • Architecture review: When evaluating tradeoffs before committing to an approach
  • Interactive development: When iterating on direction with Claude before executing
The exam tests the distinction between plan mode and direct execution. A common question: "A developer wants Claude to analyze a large codebase and propose a refactoring strategy without making any changes. Which mode should they use?" Answer: plan mode (--permission-mode plan). The key phrase is "without making any changes" — that's plan mode's defining characteristic.
The plan-first pattern. For non-trivial tasks, the recommended workflow is:

1. Plan: Start in plan mode. Explore the codebase, understand dependencies, propose an approach.

2. Review: The developer reviews the plan, asks questions, refines.

3. Execute: Switch to normal mode and implement the approved plan.

This mirrors how senior engineers work: understand the problem fully before writing code. The exam tests this as a best practice for complex changes.

Git Worktrees: Parallel Isolation

Git worktrees give each Claude session its own isolated copy of the codebase. This means multiple Claude sessions can work simultaneously without interfering with each other.

The Problem

If you run two Claude sessions on the same repository, they'll fight over the same files. Session A writes to auth.js while Session B is reading it — chaos. Worktrees solve this by creating separate working directories with their own branches.

How to Use Worktrees

# Start Claude in an isolated worktree
claude --worktree feature-auth
# Creates .claude/worktrees/feature-auth/ with branch worktree-feature-auth

# Start another session in a different worktree
claude --worktree bugfix-123
# Creates .claude/worktrees/bugfix-123/ with branch worktree-bugfix-123

# Auto-generate a worktree name
claude --worktree
# Creates something like .claude/worktrees/bright-running-fox/

Worktrees are created at <repo>/.claude/worktrees/<name> and branch from origin/HEAD (the remote's default branch).

Worktree Cleanup

  • No changes made: Worktree and branch are removed automatically
  • Changes exist: Claude prompts you to keep or remove the worktree

Subagent Worktrees

Subagents can also use worktree isolation. Add isolation: worktree to a subagent's frontmatter, and each spawned subagent gets its own worktree. This is how Agent Teams work: multiple agents, each in their own worktree, working in parallel.

# .claude/agents/feature-builder.md
---
name: feature-builder
isolation: worktree
allowedTools:
  - Read
  - Write
  - Bash
---
You are a feature builder. Implement the assigned feature
in your isolated worktree...
When the exam describes a scenario where "multiple developers (or agents) need to work on different features simultaneously in the same repository," the answer is git worktrees. The key signal is "parallel work, same repo." Without worktrees, concurrent sessions create conflicts. With worktrees, each gets an isolated copy.

Headless Mode: claude -p

The -p (or --print) flag runs Claude Code in non-interactive (headless) mode. It processes the prompt and outputs results to stdout with no human interaction. This is the foundation of CI/CD integration.

# Basic headless execution
claude -p "Explain the main architecture of this codebase"

# Pipe input
cat build-error.txt | claude -p "Explain the root cause of this build error"

# Output to file
claude -p "Review src/auth.js for security issues" > review.txt

# Control output format
claude -p "Analyze code" --output-format json > analysis.json
claude -p "Analyze code" --output-format stream-json  # streaming

# Limit agent iterations for cost control
claude -p "Review changes" --max-turns 10

# Cap the dollar amount for a single command
claude -p "Review changes" --max-turns 10 --max-budget-usd 0.50

Output Formats

Format Description Use For
text Plain text (default) Simple integrations, human-readable output
json JSON array with metadata (cost, duration) Programmatic parsing, detailed logging
stream-json Streaming JSON objects in real-time Real-time monitoring, progressive display
Cost management matters in headless mode. Without a human to say "stop," a headless Claude can run up significant API costs. Always use --max-turns to cap the number of agent loop iterations. This prevents runaway conversations that burn tokens on unproductive cycles. The exam specifically tests max_turns as a cost control mechanism.

CI/CD Integration

Running Claude Code in CI/CD pipelines is the core of the Claude Code for CI/CD exam scenario. The official way is through GitHub Actions, but the pattern applies to any CI system.

GitHub Actions Integration

# .github/workflows/claude-review.yml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: anthropics/claude-code-action@v1
        with:
          prompt: |
            Review this pull request for:
            1. Security vulnerabilities
            2. Performance issues
            3. Code style violations
            Post your findings as PR comments.
          max_turns: 10
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}

The anthropics/claude-code-action@v1 GitHub Action runs the full Claude Code runtime inside a GitHub Actions runner. No local installation needed.

Common CI/CD Use Cases

PR
Automated Code Review
Triggered on PR open/update. Claude reviews changes, posts inline comments on issues.
T
Test Generation
Triggered on PR. Claude generates tests for new/changed code, commits them to the PR branch.
S
Security Audit
Scheduled weekly. Claude scans the codebase for vulnerabilities, creates issues for findings.
D
Deployment Checks
Pre-deploy gate. Claude verifies config changes, migration scripts, and environment compatibility.

Security Considerations for CI/CD

  • API key management: Store the ANTHROPIC_API_KEY in GitHub Secrets, never in code or logs
  • Permissions: Use the most restrictive permission mode. CI/CD Claude typically needs read + limited write
  • Cost control: Always set max_turns to prevent runaway costs
  • Output filtering: Don't log raw Claude output that might contain sensitive information from the codebase
A frequent CI/CD exam question: "How should the API key be provided to Claude Code in a GitHub Actions workflow?" The answer is GitHub Secrets (${{ secrets.ANTHROPIC_API_KEY }}). Distractors include hardcoding in the workflow file, committing to .env, or using environment variables in .mcp.json. Only secrets are secure in CI.

Scheduled Agents and Remote Triggers

Claude Code supports several scheduling options for recurring tasks:

Option Where It Runs Best For
Routines (research preview) Anthropic-managed cloud Cloud-scheduled work, API-triggered tasks, GitHub-event reactions — all from one saved configuration
Desktop scheduled tasks Your machine (desktop app) Tasks needing local file access
GitHub Actions (cron) CI pipeline Repo-event-driven or cron-based tasks
/loop Current CLI session Quick polling while session is open

Routines: Three Triggers, One Config

Routines (launched April 14, 2026, currently in research preview) replace what was previously called "cloud scheduled tasks." A routine is a saved Claude Code configuration — prompt, repositories, environment, connectors — with one or more triggers attached:

  • Scheduled: recurring cron-style schedules (minimum interval: 1 hour) or one-off runs at a specific timestamp
  • API: a per-routine HTTPS endpoint (POST /v1/claude_code/routines/<id>/fire) authenticated with a bearer token — wire it to alerting tools, deploy pipelines, or internal apps
  • GitHub: reacts to repository events like pull_request.opened or release.created, with optional filters on author, labels, branch, etc.

A single routine can combine triggers (e.g., a PR-review routine that runs nightly, fires from a deploy script, and reacts to every new PR). Manage routines from claude.ai/code/routines or from the CLI with /schedule, /schedule list, /schedule update, and /schedule run.

# Schedule via GitHub Actions cron
on:
  schedule:
    - cron: '0 9 * * MON'  # Every Monday at 9 AM
jobs:
  weekly-review:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          prompt: "Review all open PRs labeled 'needs-review'. Post summaries."
When writing prompts for scheduled tasks, be explicit about what success looks like and what to do with results. The task runs autonomously — it can't ask clarifying questions. Bad: "Check the PRs." Good: "Review open PRs labeled 'needs-review', leave inline comments on issues, and post a summary in the #eng-reviews Slack channel."

The Dual-Agent Pattern

For complex automated workflows, use the dual-agent pattern: an initializer agent that plans, then spawns worker subagents that execute.

Initializer Agent
→
Plan & Decompose
→
Worker Agents (parallel)
→
Aggregate Results
# Dual-agent: initializer plans, workers execute
# Initializer (runs in plan mode)
claude --permission-mode plan -p "
  Analyze the PR and create a review plan:
  1. List all changed files
  2. Categorize changes (feature, bug fix, refactor)
  3. Identify files that need security review
  4. Output a JSON task list for worker agents
"

# Workers (run in parallel with worktrees)
for task in tasks:
    claude --worktree "review-${task.id}" -p "${task.prompt}"
The dual-agent pattern separates planning from execution. The initializer agent has broad read access but no write access (plan mode). It produces a structured task list. Worker agents have focused write access but limited scope (one task each, isolated worktrees).

This is safer than giving one agent full access to everything. It mirrors the hub-and-spoke pattern from Chapter 2, applied specifically to Claude Code workflows: the initializer is the hub, workers are the spokes.

Session Management

Claude Code sessions persist and can be resumed, which is important for long-running workflows:

# Resume the most recent session
claude --continue

# Resume a named session
claude --resume auth-refactor

# Name a session at startup
claude -n auth-refactor

# Resume a session linked to a PR
claude --from-pr 123

Sessions store the full conversation history, tool results, and configuration state. When you resume, everything is restored. This is how you maintain continuity across multiple work sessions on the same feature.

Session Forking

Use /branch to create a fork of the current session — explore an alternative approach without losing the original conversation. Think of it like git branch for conversations.

Cost Management

For the exam, know the key cost control mechanisms:

Mechanism How When
--max-turns Caps agent loop iterations CI/CD, headless mode — prevents runaway
--max-budget-usd Hard dollar ceiling for a single -p command Unattended execution — pairs with --max-turns as the cost-control duo
Effort level /effort low|medium|high Adaptive thinking — lower = fewer thinking tokens
Model selection --model claude-sonnet-4-6 Sonnet for routine tasks, Opus for complex ones
Subagent models Specify model in agent frontmatter Use cheaper models for simple subagent tasks
JSON output format --output-format json Includes cost metadata for tracking
Cost trap in CI/CD: Running Claude on every PR commit with no max_turns limit. If a PR has 20 commits in a day, that's 20 full Claude reviews. Use types: [opened] instead of types: [opened, synchronize] if you only want one review per PR, or add a label-based trigger.

Quick Review

  • Plan mode (--permission-mode plan) is read-only analysis — use it for exploration and multi-step planning before execution
  • Git worktrees (--worktree name) provide isolated copies for parallel work — subagents can use isolation: worktree
  • Headless mode (claude -p) runs non-interactively for CI/CD — always pair --max-turns (iteration cap) with --max-budget-usd (dollar ceiling) for cost control
  • The dual-agent pattern separates planning (initializer, read-only) from execution (workers, isolated worktrees) for safer automation
  • API keys in CI/CD go in GitHub Secrets — never in code, logs, or environment files committed to git
