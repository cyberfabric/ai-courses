Chapter 10

Structured Output & Validation

JSON schema enforcement, validation loops, extraction pipelines — making agent output predictable and parseable

Completing Domain 4

This chapter finishes Domain 4: Prompt Engineering & Structured Output (20%). Chapter 9 covered prompting techniques. Now we cover the other half: how to make Claude's output machine-parseable, schema-compliant, and reliably structured.

This is where the Structured Data Extraction scenario lives. It's also where you'll find questions about validation loops, JSON schemas, and production-grade extraction pipelines. If you've ever called JSON.parse() on Claude's output and gotten an error, this chapter explains how to never have that happen again.

Why Structured Output Matters

Without structured output, Claude's responses are freeform text. That's fine for conversations, but useless for downstream systems. APIs expect JSON. Databases expect typed fields. Dashboards expect numbers. Structured output is the bridge between Claude's reasoning and your application logic.

Two Approaches: JSON Outputs vs. Strict Tool Use

The Claude API offers two complementary features for structured output. Understanding when to use each is heavily exam-tested.

JSON output_config.format — controls response format
Strict strict: true on tools — guarantees input schema
Both Can be used together in the same request

JSON Outputs (output_config.format)

This controls the format of Claude's response. You provide a JSON Schema, and Claude's output is guaranteed to match it through constrained decoding — the model literally cannot produce tokens that would violate your schema.

import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": "Extract info: John Smith (john@example.com) wants the Enterprise plan."
    }],
    output_config={
        "format": {
            "type": "json_schema",
            "schema": {
                "type": "object",
                "properties": {
                    "name": {"type": "string"},
                    "email": {"type": "string"},
                    "plan": {"type": "string"},
                    "demo_requested": {"type": "boolean"}
                },
                "required": ["name", "email", "plan", "demo_requested"],
                "additionalProperties": False
            }
        }
    }
)

# Guaranteed valid JSON matching the schema
data = json.loads(response.content[0].text)

When to use: Data extraction, report generation, API response formatting, any time you need Claude's final answer as structured JSON.

Strict Tool Use (strict: true)

This guarantees that Claude's tool call inputs match the tool's input schema exactly. We covered this in Chapter 5 — here's how it connects to structured output.

# strict: true guarantees schema conformance on tool inputs
{
  "name": "extract_invoice",
  "strict": true,
  "input_schema": {
    "type": "object",
    "properties": {
      "vendor": {"type": "string"},
      "total": {"type": "number"},
      "line_items": {
        "type": "array",
        "items": {
          "type": "object",
          "properties": {
            "description": {"type": "string"},
            "amount": {"type": "number"}
          },
          "required": ["description", "amount"]
        }
      }
    },
    "required": ["vendor", "total", "line_items"]
  }
}

When to use: When you need structured data as input to a tool call, not as the final response. Common in extraction pipelines where Claude calls a "save" or "process" tool with the extracted data.

The exam tests the distinction between these two approaches. JSON outputs = structured final response. Strict tool use = structured tool call inputs. A common question: "An agent needs to extract invoice data and return it to the user as JSON." Answer: use output_config.format with a JSON schema. If the question says "extract data and pass it to a validation tool," use strict: true on the tool.

Constrained Decoding: How It Works

Under the hood, structured outputs use constrained decoding. Your JSON Schema is compiled into a grammar, and during token generation, the model is physically restricted to only produce tokens that would result in valid output matching your schema.

This means:

  • Always valid JSON syntax — no JSON.parse() errors, ever
  • Type-safe fields — a "type": "number" field always contains a number
  • Required fields always present — no missing fields
  • No extra fields when additionalProperties: false

This is fundamentally different from "prompting Claude to return JSON." Prompting is probabilistic — it usually works, but occasionally fails. Constrained decoding is deterministic — schema violations are impossible.

Constrained decoding vs. validation loops: the architect's choice.

Constrained decoding (structured outputs): Schema compliance is guaranteed at generation time. No retries needed. Fastest approach. Use when the schema is known in advance and the Claude API supports it.

Validation loops (generate → validate → retry): Schema compliance is checked after generation. Retries needed on failure. More flexible — works with any validation logic, not just JSON Schema. Use when you need custom validation beyond what JSON Schema can express (business rules, cross-field constraints, external lookups).

Best practice: Use constrained decoding as the first layer (guarantees structure), then add validation loops on top for business logic (guarantees semantics).

The Validation Loop Pattern

Even with guaranteed structure, you might need to validate content. A valid JSON response with "email": "not-an-email" passes schema validation but fails business validation. The validation loop handles this:

Validate

Pass? Done

Fail? Retry with feedback
MAX_RETRIES = 3

for attempt in range(MAX_RETRIES):
    response = client.messages.create(
        model="claude-opus-4-7",
        max_tokens=1024,
        messages=messages,
        output_config={"format": {"type": "json_schema", "schema": schema}}
    )

    data = json.loads(response.content[0].text)  # Always valid JSON

    # Business validation (beyond schema)
    errors = validate_business_rules(data)

    if not errors:
        return data  # Success!

    # Retry with error feedback
    messages.append({"role": "assistant", "content": response.content})
    messages.append({
        "role": "user",
        "content": f"Validation errors: {errors}. Please fix and try again."
    })

raise ValidationError(f"Failed after {MAX_RETRIES} attempts")

The key insight: feed the validation errors back to Claude. Don't just retry with the same input — tell Claude what went wrong so it can fix it. This dramatically improves success rates on retries.

The exam tests the generate → validate → retry pattern specifically. A common question format: "An extraction pipeline produces valid JSON but the email field contains an invalid format. What's the best approach?" Answer: validation loop with error feedback. The distractor "tighten the JSON schema" is wrong because email format validation goes beyond what JSON Schema handles well. You need business-level validation in code, with error messages fed back to Claude.

Building Extraction Pipelines

The Structured Data Extraction scenario is all about multi-step extraction from unstructured text. Here's the standard architecture:

# Multi-step extraction pipeline
#
# ┌──────────────┐
# │ Raw Document │  (PDF, email, contract, invoice)
# └──────┬───────┘
#        │
# ┌──────▼───────┐
# │ Step 1:       │  Claude extracts fields into JSON
# │ Extract       │  (constrained decoding for structure)
# └──────┬───────┘
#        │
# ┌──────▼───────┐
# │ Step 2:       │  Code validates business rules
# │ Validate      │  (email format, date ranges, amounts)
# └──────┬───────┘
#     Pass│  Fail
#        │  └── Retry with feedback (up to 3x)
# ┌──────▼───────┐
# │ Step 3:       │  Transform for downstream system
# │ Transform     │  (normalize dates, currency, etc.)
# └──────┬───────┘
#        │
# ┌──────▼───────┐
# │ Step 4:       │  Store in database / send to API
# │ Load          │

Extraction with Confidence Scores

For documents where some fields are ambiguous, include confidence scores in the schema:

{
  "type": "object",
  "properties": {
    "vendor_name": {
      "type": "object",
      "properties": {
        "value": {"type": "string"},
        "confidence": {"type": "string", "enum": ["high", "medium", "low"]}
      },
      "required": ["value", "confidence"]
    },
    "total_amount": {
      "type": "object",
      "properties": {
        "value": {"type": "number"},
        "confidence": {"type": "string", "enum": ["high", "medium", "low"]}
      },
      "required": ["value", "confidence"]
    }
  }
}

Low-confidence fields can be flagged for human review. This is a common pattern in the Data Extraction scenario — the agent extracts what it can, marks uncertain fields, and escalates ambiguity.

Schema Design Best Practices

Practice Do Don't
Nesting Flat when possible, nested only when logical Deeply nested objects (3+ levels)
Required fields Mark truly required fields, use nullable for optional Make everything required (causes extraction failure)
Enums Use for fields with fixed valid values Freeform strings for categorical data
Descriptions Add description to every property Rely on property names alone
Additional properties Set additionalProperties: false Allow arbitrary extra fields
Schema design trap: Making all fields required when some may not be present in the source document. If your schema requires "phone_number" but the document doesn't contain one, Claude is forced to hallucinate a value to satisfy the schema. Use "type": ["string", "null"] for fields that might be absent, so Claude can return null instead of inventing data.

When to Use Structured Output vs. Freeform Text

Use Structured Output
- Data extraction (invoices, contracts) - Classification (ticket routing, sentiment) - API responses consumed by code - Metadata generation (tags, categories) - Any output parsed by downstream systems
Use Freeform Text
- Explanations for humans - Creative writing - Code generation (use code blocks) - Conversational responses - Analysis with nuanced reasoning
The exam rule of thumb: if the output is consumed by code, use structured output. If consumed by a human, freeform is usually better. Many scenarios involve both — the agent's internal processing uses structured output, but the final customer-facing response is freeform text.

Handling Malformed Output Gracefully

Even with constrained decoding, you need fallback strategies for edge cases (API errors, timeout mid-generation, etc.):

1. Partial Extraction

If Claude extracts 8 of 10 fields successfully but fails on 2, accept the partial result rather than rejecting everything. Flag the missing fields for human review.

2. Fallback to Freeform

If structured extraction fails after max retries, fall back to asking Claude for a freeform summary. You lose machine parseability but still get useful information.

3. Multi-Pass Extraction

For complex documents, extract in multiple passes: first pass gets the easy fields (names, dates, amounts), second pass gets the nuanced fields (contract terms, obligations). Each pass has a simpler schema, improving reliability.

Design pattern: extraction pipeline with escalation tiers.

Tier 1 — Constrained decoding: Fast, guaranteed structure. Handles 80-90% of documents.

Tier 2 — Validation loop: For documents that pass schema but fail business rules. Retry with feedback, 2-3 attempts.

Tier 3 — Human review: For documents that fail after max retries. Flag specific fields that need human judgment.

This tiered approach ensures high throughput for easy cases while maintaining quality for edge cases. The exam tests whether you can design this kind of graduated fallback.

The Batch API for Cost Optimization

For high-volume extraction (thousands of documents), the Message Batches API offers 50% cost savings with a 24-hour processing window:

# Batch extraction for cost optimization
batch = client.messages.batches.create(
    requests=[
        {
            "custom_id": f"invoice-{i}",
            "params": {
                "model": "claude-sonnet-4-6",
                "max_tokens": 1024,
                "messages": [{"role": "user", "content": doc}],
                "output_config": {
                    "format": {"type": "json_schema", "schema": invoice_schema}
                }
            }
        }
        for i, doc in enumerate(documents)
    ]
)

The tradeoff: 50% cheaper, but results arrive within 24 hours, not in real-time. Use batch for overnight processing jobs, not for real-time agent interactions.

The exam tests the Batch API as a cost optimization strategy for latency-tolerant workloads. Key detail: 50% cost savings, 24-hour processing window. If the question describes a scenario where "1,000 invoices need to be processed overnight," Batch API is the correct answer. If it says "each invoice must be processed in real-time as it arrives," use the standard Messages API.

Scenario Deep Dive: Data Extraction

The Structured Data Extraction scenario is the exam's primary test of this chapter. Here's what it typically involves:

  • Extracting structured data from invoices, contracts, or reports
  • Enforcing schema compliance with constrained decoding
  • Validating business rules beyond the schema (amount ranges, date validity)
  • Handling ambiguous or missing fields with confidence scores
  • Building a multi-step pipeline: extract → validate → transform → load

In the Customer Support scenario, structured output also appears: extracting ticket metadata (category, priority, customer ID) from freeform messages. The agent needs to produce structured data that the ticketing system can consume.

Design a JSON Schema for extracting data from invoices. The schema should handle:

  1. Vendor name and address
  2. Invoice number and date
  3. Line items (description, quantity, unit price, total)
  4. Subtotal, tax, and grand total
  5. Payment terms and due date

Then design the validation layer:

  1. What business rules can't be expressed in JSON Schema? (Hint: line item totals should equal quantity × unit price)
  2. Which fields should be nullable (might not be on every invoice)?
  3. How would you handle a 3-page invoice that exceeds easy extraction? (Multi-pass?)
  4. What's your retry strategy when validation fails?

Check yourself: Line item math validation (total = qty × price) is business logic, not schema. Payment terms might be absent — make nullable. Long invoices benefit from per-page extraction with aggregation. Retry up to 3 times with error feedback, then escalate to human review.

Quick Review

  • Two approaches: JSON outputs (output_config.format) for structured responses, strict tool use (strict: true) for structured tool inputs
  • Constrained decoding guarantees schema compliance at generation time — no JSON.parse() errors, no retries needed for structure
  • Validation loops handle business logic beyond schema: generate → validate → retry with error feedback — max 3 attempts
  • Use nullable types for fields that may not exist in the source — forcing required fields causes hallucination
  • Batch API offers 50% cost savings for high-volume extraction with 24-hour processing window
