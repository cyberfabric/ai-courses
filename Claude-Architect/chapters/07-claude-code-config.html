Chapter 7

Claude Code Configuration

CLAUDE.md, settings layers, hooks, permissions, and skills — persistent context engineering for teams

Welcome to Domain 3

We're entering Domain 3: Claude Code Configuration & Workflows, which makes up 20% of the exam (~12 questions). This chapter covers the configuration side; Chapter 8 covers workflows. Together they prepare you for every Domain 3 question.

Claude Code is more than a CLI — it's a configurable agent harness. The configuration system controls what Claude knows about your project, what it's allowed to do, and how it responds to events. Think of it as programming the environment around the agent, not the agent itself.

Why Configuration Matters on the Exam

The exam gets surprisingly specific about Claude Code configuration. You'll be asked about CLAUDE.md hierarchy, settings scope precedence, hook events, permission rules, and custom skills. This isn't theoretical — it's "where does this config file go?" practical.

CLAUDE.md: Persistent Context Engineering

CLAUDE.md is the most important concept in Claude Code configuration. It's a markdown file that gives Claude persistent context — project conventions, instructions, constraints, and coding standards. It's loaded into Claude's system prompt automatically every session.

Think of CLAUDE.md as programming your agent's memory. Without it, every session starts from zero. With it, Claude already knows your project's tech stack, coding conventions, testing approach, and team preferences.

The CLAUDE.md Hierarchy

CLAUDE.md files exist at three levels, and they merge together:

Level Location Scope Use For
User ~/.claude/CLAUDE.md All your projects Personal preferences, default conventions
Project CLAUDE.md or .claude/CLAUDE.md in repo root All collaborators Project-specific context, team conventions
Directory CLAUDE.md in any subdirectory That directory's files Module-specific instructions

All three levels merge into one system prompt. If there's a conflict, more specific levels win. A project CLAUDE.md overrides the user CLAUDE.md; a directory CLAUDE.md adds to the project one for files in that directory.

The exam tests the three-level hierarchy and the merge behavior. A common question: "A developer has a personal CLAUDE.md that says 'always use tabs.' The project CLAUDE.md says 'always use spaces.' Which one applies?" Answer: project-level wins because more specific scopes take precedence. The user-level is the lowest priority.

What Goes in CLAUDE.md

Put in CLAUDE.md
- Project tech stack and architecture - Coding conventions and style rules - Testing approach and frameworks - Build and deployment commands - File structure conventions - Common patterns the team uses - Things Claude should never do
Don't Put in CLAUDE.md
- Secrets, API keys, passwords - Temporary task context - Things derivable from the code itself - Long copy-pasted documentation - Entire file contents
CLAUDE.md is persistent context engineering. Every token in CLAUDE.md is loaded every session, consuming context window space. Keep it focused and concise. A 2,000-token CLAUDE.md is loaded into every conversation — if it's full of irrelevant information, you're wasting context budget on noise.

Rule of thumb: If you'd tell a new team member something on their first day, it belongs in CLAUDE.md. If you'd only tell them when they're working on a specific task, it doesn't.

Settings Scopes: The Four Layers

Beyond CLAUDE.md, Claude Code has a structured settings system with four scopes:

Scope Location Shared with Team? Priority
Managed Server-managed / MDM / system-level Yes (deployed by IT) Highest (can't override)
Local .claude/settings.local.json No (gitignored) High
Project .claude/settings.json Yes (committed to git) Medium
User ~/.claude/settings.json No Lowest

The precedence order: Managed > Local > Project > User. If a permission is allowed in user settings but denied in project settings, the project setting wins.

// .claude/settings.json — project-level, committed to git
{
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test *)",
      "Read(~/.zshrc)"
    ],
    "deny": [
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  }
}
The exam tests scope precedence. Key rule: managed settings are the highest priority and cannot be overridden by anything. If IT deploys a managed setting that denies a tool, no user or project setting can re-allow it. This is by design for security compliance.

Permission Configuration

Permissions control what Claude Code is allowed to do. They use tool names with glob patterns:

// Permission rules with glob patterns
{
  "permissions": {
    "allow": [
      "Read",                    // Allow all file reads
      "Bash(git *)",             // Allow git commands
      "Bash(npm run *)",         // Allow npm scripts
      "Write(./src/**)"          // Allow writing to src/
    ],
    "deny": [
      "Bash(rm -rf *)",          // Block destructive commands
      "Write(./.env)",           // Block writing to .env
      "Bash(curl *)",            // Block network calls
      "Read(./secrets/**)"       // Block reading secrets
    ]
  }
}

The Trust Spectrum

Claude Code offers different permission modes that represent a trust spectrum:

Full Manual
→
Default
→
Allowlisted
→
Auto Mode
  • Full Manual: Every tool call requires human approval. Safest, but slowest.
  • Default: Safe operations (reads) are auto-approved. Writes and commands require approval.
  • Allowlisted: Specific tools and patterns are pre-approved. Everything else requires approval.
  • Auto Mode: An AI classifier decides what's safe. Maximum autonomy, but requires trust.

For the exam, understand that the trust spectrum is about balancing safety vs. speed. More autonomy means faster work but more risk. The right level depends on the context: CI/CD pipelines might use auto mode (no human to approve); production systems might use allowlisted mode.

Hooks System

Hooks are the most powerful configuration mechanism. They let you run custom code at specific points in Claude's lifecycle. Remember from Chapter 2: hooks are deterministic (100% reliable), unlike prompts (probabilistic).

Key Hook Events

Hook Event When It Fires Use For
PreToolUse Before every tool call Validate inputs, enforce rules, block dangerous actions
PostToolUse After tool execution Validate outputs, log results, inject context
UserPromptSubmit When user sends a message Add context, modify prompts, enforce policies
SessionStart When a session begins Set up environment, load context
PreCompact Before context compaction Preserve critical information from being summarized away
Stop When agent stops Cleanup, logging, notifications

PreToolUse and PostToolUse are the most exam-tested hooks. PreToolUse can return allow/deny decisions that override the permission system. This is how you enforce business rules programmatically.

// Hook configuration in settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "python3 ~/.claude/validate-command.py"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint --fix ${file}"
          }
        ]
      }
    ]
  }
}
A recurring exam pattern: "How do you ensure Claude never writes to production config files?" The distractors include system prompt instructions ("never modify production configs") and CLAUDE.md rules. The correct answer is a PreToolUse hook on the Write tool that checks the file path and returns a deny decision for production directories. Hooks are deterministic; prompts are not.

Hooks in the Customer Support Scenario

In the Customer Support Agent scenario, hooks enforce business rules:

  • PreToolUse on ProcessRefund: Check if refund amount exceeds $500 limit. If yes, deny and return escalation instructions.
  • PostToolUse on SearchKB: Log which knowledge base articles were accessed for compliance auditing.
  • PreToolUse on EscalateToHuman: Validate that a summary is included in the escalation payload.

Agent Skills

Skills extend Claude with reusable instructions, playbooks, and parameterized workflows. Each skill is a SKILL.md file with YAML frontmatter (name, description, etc.) and a markdown body. Claude can auto-invoke skills when the description matches a request, or you call them explicitly with /skill-name.

The legacy .claude/commands/*.md slash-command format still works — Anthropic merged custom commands into the skills system, so a file at .claude/commands/deploy.md and a skill at .claude/skills/deploy/SKILL.md both create /deploy and behave the same way. Skills add three things on top: a directory for supporting files, frontmatter for invocation control, and auto-invocation.

Where Skills Live

Scope Location Applies To Shared?
Enterprise Managed settings (org-deployed) All users in your org Yes (centrally deployed)
Personal ~/.claude/skills/<name>/SKILL.md All your projects No (personal)
Project .claude/skills/<name>/SKILL.md This project only Yes (committed to git)
Plugin <plugin>/skills/<name>/SKILL.md Wherever the plugin is enabled Yes (via the plugin)

When skills share a name across levels, enterprise overrides personal, and personal overrides project. Plugin skills use a plugin-name:skill-name namespace so they cannot conflict with other levels.

A SKILL.md bundles its description (so Claude knows when to load it) with the playbook itself:

# .claude/skills/review-pr/SKILL.md
---
name: review-pr
description: Review the current pull request for quality, security, and test coverage. Use when the user asks for a PR review or before merging.
---

Review the current pull request for:
1. Code quality and adherence to project conventions
2. Security vulnerabilities (OWASP top 10)
3. Test coverage gaps
4. Performance concerns

Focus on actionable feedback. Skip style nits handled by the linter.
Use the project's CLAUDE.md conventions as your reference.

Invoke it explicitly with /review-pr, or let Claude load it automatically when someone asks for a PR review. Every team member gets the same review criteria, ensuring consistency. The same content saved as .claude/commands/review-pr.md (no frontmatter) creates the same slash command via the legacy path.

Design pattern: skills as team knowledge codification. When a senior developer has a great code review approach, it lives in their head. When that approach becomes a skill file committed to .claude/commands/, the entire team benefits. Skills turn individual expertise into team-wide capability.

Project skills vs user skills: Project skills (.claude/commands/) are shared with the team via git. Use them for workflows everyone should follow. User skills (~/.claude/commands/) are personal. Use them for your own shortcuts and preferences.

Subagents: Named Agent Definitions

Beyond skills, Claude Code supports subagent definitions — named agent configurations with custom system prompts, tool restrictions, and model settings:

# .claude/agents/code-reviewer.md
---
name: code-reviewer
description: Reviews code for quality, security, and performance
allowedTools:
  - Read
  - Glob
  - Grep
  - Bash(git diff *)
model: claude-sonnet-4-6
---

You are a senior code reviewer. Focus on:
- Security vulnerabilities
- Performance issues
- Code style violations per project CLAUDE.md
- Missing error handling

Never modify files. Only read and report.

Subagents use frontmatter to specify their configuration. They can be invoked as the main agent (claude --agent code-reviewer) or spawned by the Agent tool during multi-agent workflows.

The distinction between skills and subagents: skills are prompts (text that gets sent to Claude). Subagents are agent configurations (system prompt + tool restrictions + model). A skill says "do this task." A subagent says "be this kind of agent." The exam tests this distinction.

Plugins: Packaging Configuration for Distribution

Plugins are the architect-level packaging primitive. A plugin bundles skills, MCP servers, slash commands, hooks, and subagents into a single installable unit. Instead of telling each team member "drop these five files into your .claude/," you publish a plugin and they install it with one command.

Browse the official marketplace with /plugin (Discover tab) or at claude.com/plugins. Install a plugin from the official marketplace with:

/plugin install code-review@claude-plugins-official

The official marketplace ships ~101 plugins as of early 2026 — 33 Anthropic-built (language servers, dev workflow, setup tools, output styles) plus 68 partner plugins (GitHub, Playwright, Supabase, Figma, Vercel, Linear, Sentry, Stripe, Firebase, and more).

For the architect: think of plugins as the deployment primitive for standard configurations. A "code-review" plugin can ship the review skill, a Linter MCP server, a PreToolUse hook that blocks unsafe writes, and a code-reviewer subagent — all together. Inside an organization, you publish private plugins to enforce conventions across many repos. The exam may not test plugin internals directly, but the architect-level reader will encounter them in production Domain 3 work.

Configuration for the Code Generation Scenario

In the Code Generation with Claude Code scenario, configuration is central:

# CLAUDE.md for a TypeScript project
## Project
- TypeScript 5.x, React 19, Vite bundler
- pnpm for package management

## Conventions
- Functional components only, no class components
- Use React Query for server state
- All API calls go through src/api/ layer
- Tests: Vitest + React Testing Library
- Naming: camelCase for variables, PascalCase for components

## Commands
- Build: pnpm build
- Test: pnpm test
- Lint: pnpm lint

## Rules
- Never import from node_modules directly
- Always handle loading and error states
- No any types — use proper TypeScript generics

This CLAUDE.md ensures every session starts with full project context. Claude won't suggest class components, won't use npm instead of pnpm, and will follow the team's testing approach without being told each time.

Best Practices: What to Configure Where

Configuration Need Where Why
Project conventions CLAUDE.md (project) Shared context for all collaborators
Security policy Managed settings Can't be overridden
Team tool permissions .claude/settings.json Shared, committed to git
Personal tool permissions .claude/settings.local.json Personal, gitignored
Experimental MCP server ~/.claude.json Personal, not shared
Business rules enforcement Hooks (PreToolUse) Deterministic, not skippable
Code review workflow Skill (.claude/commands/) Shared, reusable
Module-specific instructions CLAUDE.md in subdirectory Scoped to that directory
Common exam trap: Using CLAUDE.md to enforce security rules. CLAUDE.md is part of the system prompt — it's guidance, not enforcement. Claude usually follows CLAUDE.md instructions, but it's not guaranteed. For security-critical rules (don't access production, don't expose secrets), use permissions (deny rules) or hooks, not CLAUDE.md.
Exercise: Configure Claude Code for a New Project

You're setting up Claude Code for a Python FastAPI project. Create the configuration:

  1. Write a CLAUDE.md with project context (tech stack, conventions, commands, rules)
  2. Write a .claude/settings.json with permissions that allow running tests and linting but block curl and access to .env files
  3. Define a PreToolUse hook that validates all Write operations target only ./src/ or ./tests/ directories
  4. Create a skill in .claude/commands/add-endpoint.md that guides Claude through adding a new API endpoint (router, schema, tests)
  5. Decide: which of these files are committed to git (shared) and which are personal?

Check yourself: CLAUDE.md, .claude/settings.json, hooks in settings, and skills are all committed to git (shared with team). .claude/settings.local.json is gitignored (personal). User-level settings stay in ~/.claude/ (personal).

Quick Review

  • CLAUDE.md has three levels (user, project, directory) that merge together — more specific scopes override less specific ones
  • Settings have four scopes: Managed > Local > Project > User — managed settings cannot be overridden by anything
  • Hooks (PreToolUse, PostToolUse) provide deterministic enforcement — use them for business rules, not the system prompt
  • Skills live at .claude/skills/<name>/SKILL.md (modern, with YAML frontmatter and auto-invocation) or the legacy .claude/commands/ slash-command path; subagents (.claude/agents/) are full agent configurations with tool restrictions; plugins package skills + MCP + hooks + subagents as one installable unit
  • Permission rules use tool names with glob patterns — allow/deny lists control what Claude Code can do
