Multi-Agent Systems

Why one agent isn't enough — and how to design systems where multiple agents collaborate

Welcome to Domain 1

This chapter begins our deep dive into Domain 1: Agentic Architecture & Orchestration, which makes up 27% of the exam — the single largest domain. We'll spend three chapters here (Chapters 2, 3, and 4) because there's a lot of ground to cover and the exam tests this material heavily.

Why Single Agents Hit Walls

Let's start with the obvious question: why do you need more than one agent?

A single Claude agent is powerful. It can reason, call tools, write code, and manage multi-step tasks. But in production, you'll hit three walls pretty quickly:

Wall 1: Context Window Limits

Every tool call, every tool result, every message — it all goes into the context window. A single agent handling a complex research task might burn through 100K tokens just on intermediate results. Once you're near the limit, the agent starts losing track of earlier information.

Wall 2: Specialization

A single agent's system prompt tries to cover everything: "you're a researcher AND a coder AND a reviewer AND a deployment specialist." The more roles you pile on, the worse the agent performs at each one. It's like asking one person to be the chef, waiter, and dishwasher simultaneously.

Wall 3: Parallelism

A single agent processes sequentially. If you need to search three databases, analyze five documents, and check two APIs — a single agent does them one at a time. Multiple agents can do them in parallel.

The exam loves to present a scenario where a single agent is struggling, then ask you to identify which wall the system is hitting. The correct answer usually maps to one of these three: context exhaustion, role overload, or sequential bottleneck. Learn to recognize the symptoms of each.

The Agent Loop: Plan → Act → Observe → Repeat

Before we go multi-agent, you need to understand the fundamental loop that every Claude agent runs. Whether it's a single agent or one of twenty subagents, they all follow the same cycle:

Receive Task
→
Reason / Plan
→
Call Tool
→
Observe Result
→
Decide: Continue or Stop

In the Claude API, this loop is controlled by the stop_reason field in the response:

  • stop_reason: "tool_use" — The agent wants to call a tool. Execute it, append the result, and send the conversation back to Claude.
  • stop_reason: "end_turn" — The agent is done. It has a final response ready.

Here's the minimal agent loop in Python:

import anthropic

client = anthropic.Anthropic()
messages = [{"role": "user", "content": task}]

while True:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=4096,
        tools=my_tools,
        messages=messages
    )

    # Append assistant response to conversation
    messages.append({"role": "assistant", "content": response.content})

    if response.stop_reason == "end_turn":
        break  # Agent is done

    # Execute tool calls and append results
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            result = execute_tool(block.name, block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result
            })

    messages.append({"role": "user", "content": tool_results})
Exam trap: A common distractor answer suggests parsing the agent's text output to decide whether to continue the loop (e.g., looking for phrases like "I'm done" or "task complete"). The correct approach is always to check stop_reason. Never parse natural language to control loop execution — it's unreliable and the exam specifically tests this.

Agent Roles and Specialization

In a multi-agent system, each agent gets a focused role. Think of it like a software team: you wouldn't ask the same person to write code, review it, test it, and deploy it. Here are common agent roles you'll encounter on the exam:

Role Responsibility Example Tools
Orchestrator Plans, delegates, aggregates results Task (subagent spawning)
Researcher Searches and retrieves information WebSearch, DatabaseQuery
Coder Writes and refactors code FileWrite, FileEdit, Bash
Reviewer Validates output, checks quality FileRead, Lint, Test
Extractor Pulls structured data from unstructured text tool_use (schema enforcement)

The key insight: each agent has a focused system prompt and a limited set of tools. A researcher agent doesn't need file-writing tools. A coder agent doesn't need web search. By constraining each agent's capabilities, you get better performance from each one.

The Principle of Least Privilege for Agents. Just as you give microservices only the permissions they need, give each agent only the tools it needs. An orchestrator's allowedTools should include "Task" for spawning subagents. A research subagent's tools should be limited to search and retrieval. This isn't just security — it improves agent focus. When a model has 20 tools available, it spends more reasoning tokens deciding which one to use. With 3–5 focused tools, decisions are faster and more accurate.

Topology 1: Hub-and-Spoke

The hub-and-spoke pattern is the most common multi-agent architecture, and the one the exam tests most heavily. Think of it like an orchestra conductor:

  • The hub (orchestrator/coordinator) receives the task, breaks it down, and delegates subtasks to specialized agents
  • The spokes (subagents) each handle their subtask independently and return results to the hub
  • The hub aggregates results and produces the final output
# Hub-and-spoke: orchestrator spawns subagents
# In the Agent SDK, the orchestrator uses the Task tool

Orchestrator (hub)
    ├── Researcher Agent (spoke) → searches knowledge base
    ├── Analyst Agent (spoke)    → processes retrieved data
    └── Writer Agent (spoke)     → synthesizes final report

Context Isolation: The Critical Detail

Here's the most exam-tested aspect of hub-and-spoke: subagents do NOT inherit the orchestrator's conversation history. Each subagent starts with a clean context that includes only:

  1. Its own system prompt (role, tools, constraints)
  2. The specific task description the orchestrator passes to it

This is by design. If subagents inherited the full orchestrator context, you'd be wasting tokens and potentially confusing the subagent with irrelevant information.

Correct: Explicit Context Passing
Task to subagent: "Search the knowledge base for articles about refund policies. Return the top 3 most relevant article titles and their URLs."
Subagent gets only what it needs — a clear, focused task with specific output format.
Wrong: Sharing Full Context
Task to subagent: [entire 50-message conversation history including user complaints, previous tool calls, and orchestrator reasoning]
Wastes tokens, confuses the subagent with irrelevant details, and may exceed context limits.
The exam frequently asks: "In a hub-and-spoke architecture, what context does a subagent receive?" The answer is only the task-specific context explicitly passed by the orchestrator. Distractors will suggest the subagent inherits the full conversation, gets the orchestrator's system prompt, or has access to other subagents' results. All wrong. Context isolation is a fundamental principle.

When to Use Hub-and-Spoke

  • When you have a clear coordinator role that can decompose the task
  • When subagents don't need to communicate with each other
  • When you want centralized observability (all communication goes through the hub)
  • When parallel execution matters (multiple spokes can run simultaneously)

In the Customer Support Agent scenario, the orchestrator classifies the customer issue, then delegates to specialized agents: one looks up account info, another searches the knowledge base, a third checks order status. Results flow back to the orchestrator, which synthesizes a response.

In the Multi-Agent Research System scenario, the orchestrator breaks a research query into sub-questions, spawns researcher agents for each, and aggregates their findings into a cohesive report.

Topology 2: Peer-to-Peer

In a peer-to-peer topology, agents collaborate as equals without a central coordinator. Think of a jazz ensemble: each musician listens to the others and adapts in real time, with no conductor.

  • Agents communicate directly with each other
  • No single point of control or failure
  • Each agent can trigger actions in other agents

This sounds elegant, but in practice it's harder to debug, harder to observe, and harder to control. Without a central coordinator:

  • How do you know when the task is done?
  • How do you prevent circular communication (Agent A asks B, B asks A, infinite loop)?
  • How do you aggregate results if there's no aggregator?
Hub-and-spoke vs. peer-to-peer — the architect's tradeoff.

Hub-and-spoke gives you centralized control, clear observability, and predictable data flow. The tradeoff: the coordinator is a bottleneck and a single point of failure.

Peer-to-peer gives you resilience and flexibility. The tradeoff: it's complex to implement, hard to debug, and risky for circular dependencies.

For the exam: Unless the question specifically describes a scenario that requires decentralized coordination (rare), hub-and-spoke is almost always the right answer. Anthropic's Agent SDK is built around the hub-and-spoke model with the Task tool for subagent spawning.

The Orchestra vs. Jazz Analogy

This analogy shows up frequently in study materials and can help you reason about exam questions:

Aspect Orchestra (Hub-and-Spoke) Jazz Ensemble (Peer-to-Peer)
Leadership Conductor controls everything Musicians self-coordinate
Communication All through conductor Direct between players
Predictability High — follows a score Lower — improvised
Failure mode Conductor = single point of failure Resilient but chaotic
Best for Structured, repeatable tasks Creative, adaptive tasks

Autonomous vs. Orchestrated Agents

Beyond topology, there's another axis to consider: how much freedom each agent has.

Orchestrated Agents

The orchestrator tells them exactly what to do: "Search for X," "Analyze Y," "Summarize Z." They follow instructions and return results. Predictable, testable, easy to debug.

Autonomous Agents

The agent decides its own course of action based on a high-level goal: "Resolve this customer's issue." It chooses which tools to use, in what order, and when to stop. More flexible, but harder to control.

In practice, production systems use a spectrum. The orchestrator might be fairly autonomous (deciding how to decompose a task), while subagents are tightly orchestrated (given specific instructions). This gives you flexibility at the top and predictability at the bottom.

Think of autonomy like a dial, not a switch. The exam tests whether you can identify the right level of autonomy for a given scenario. Customer support? More orchestrated — you want predictable, policy-compliant behavior. Research? More autonomous — the agent needs freedom to explore unexpected connections.

Agent Communication: Structured Messages and Handoffs

How do agents talk to each other? In the Claude Agent SDK, communication happens through two mechanisms:

1. Task Tool (Subagent Spawning)

The orchestrator uses the Task tool to create a subagent. It passes a task description (plain text) and receives the subagent's final output. The orchestrator's allowedTools must include "Task" for this to work.

2. Shared State (External)

For more complex coordination, agents can read and write to shared external state — a database, a file system, or a task board. This is how Anthropic's Agent Teams feature works: agents share a task board tracking dependencies and coordinate via inter-agent messaging.

The exam tests the first pattern heavily. Know how the Task tool works and what context gets passed.

Common Pitfalls

The exam loves to present anti-patterns and ask you to identify what's wrong. Here are the big ones:

1. Over-Decomposition

Breaking a task into too many tiny subtasks. Each subagent has overhead (spawning, context setup, result passing). If you decompose "write a function" into "write line 1," "write line 2," etc., you've created more overhead than value.

2. Chatty Agents

Agents that pass too much information back and forth. If your subagent returns a 10,000-token response when the orchestrator only needs a 3-sentence summary, you're wasting context budget.

3. Unclear Responsibilities

When two agents overlap in their roles, they might duplicate work or produce conflicting results. Every agent should have a clear, non-overlapping responsibility.

4. Missing Error Handling

What happens when a subagent fails? If the orchestrator doesn't handle failures, the entire system breaks. Always design for: retry, skip, or escalate.

5. Arbitrary Iteration Caps

Setting max_iterations = 10 as the primary way to stop an agent loop. The correct stopping mechanism is stop_reason == "end_turn". Iteration caps are a safety net, not a control mechanism.

Anti-pattern alert: Using confidence scores or sentiment analysis to decide when to escalate to a human. The exam considers this unreliable. Valid escalation triggers are: explicit human request, policy gap (no rule covers the situation), task exceeds agent capability, or a business threshold is exceeded. Not "the agent seems unsure."

Hooks: Deterministic vs. Probabilistic Enforcement

One more Domain 1 concept that crosses into multi-agent design: the distinction between hooks and prompts for enforcing rules.

Mechanism Reliability Use For
Hooks (PreToolUse, PostToolUse) 100% deterministic Refund limits, compliance rules, security constraints
Prompts (system prompt instructions) Probabilistic (model may ignore) Tone, style preferences, soft guidelines

In the Customer Support Agent scenario, imagine a refund limit of $500. Do you enforce it via the system prompt ("never approve refunds over $500") or via a PostToolUse hook that checks the refund amount before it executes?

Answer: hook. A prompt can be ignored or misinterpreted. A hook is code — it runs every time, no exceptions.

Whenever the exam asks about enforcing a critical business rule (refund limits, compliance requirements, security policies), the answer is almost always hooks, not prompts. Prompts are for style and tone. Hooks are for hard constraints. This is one of the most predictable patterns on the exam.

Session Management: Resume vs. Fork

The Agent SDK provides two ways to handle ongoing agent sessions:

  • --resume — Continues an existing session with its full conversation history. Use when the user comes back to continue a task.
  • fork_session — Creates a new session that branches from an existing one. Use when you want to explore an alternative approach without losing the original conversation state.

Think of it like Git: --resume is continuing to commit on the same branch; fork_session is creating a new branch from a specific point.

Exercise: Design a Multi-Agent Customer Support System

You're building a customer support agent that handles three types of issues: billing, technical, and shipping. Design the multi-agent architecture:

  1. What agent roles do you need? (Hint: at minimum an orchestrator + 3 specialists)
  2. What tools does each agent get?
  3. Draw the communication flow: who talks to whom?
  4. How does the orchestrator decide which specialist to invoke?
  5. What happens if a specialist can't resolve the issue?

Bonus: Which business rules would you enforce with hooks vs. prompts?

Check yourself: Your design should use hub-and-spoke. The orchestrator classifies the issue (billing/technical/shipping), delegates to the right specialist, and handles escalation if the specialist fails. Refund limits = hook. Friendly tone = prompt.

Quick Review

  • Single agents hit walls on context limits, specialization, and parallelism — multi-agent systems solve all three
  • The agent loop checks stop_reason: "tool_use" means continue, "end_turn" means stop — never parse text to control the loop
  • Hub-and-spoke is the primary pattern: orchestrator delegates to subagents with isolated contexts — subagents do NOT inherit coordinator history
  • Use hooks for critical business rules (100% deterministic), prompts for style preferences (probabilistic)
  • Valid escalation triggers: explicit human request, policy gap, capability exceeded, business threshold — NOT sentiment or confidence scores
