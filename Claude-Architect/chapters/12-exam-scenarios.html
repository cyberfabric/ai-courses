Exam Day: Scenarios & Final Review

All 6 scenarios dissected, practice questions, time management, and your final game plan

The Capstone

This is it. You've covered all five domains across 11 chapters. Now we bring everything together. This chapter walks through each exam scenario with the patterns and traps you'll encounter, gives you practice questions to test your readiness, and delivers your exam-day strategy.

Remember: 4 of 6 scenarios are randomly selected for your exam sitting. You must prepare for all six — no shortcuts.

Exam Reminders

60 questions, 120 minutes, 720/1000 to pass. Closed-book, live proctored via ProctorFree. Each question has one correct answer and three distractors. Every question is anchored to one of the 4 selected scenarios. The scoring is scaled — harder questions count more.

Scenario 1: Customer Support Resolution Agent

Domains tested: Domain 1 (orchestration), Domain 2 (tools), Domain 5 (context/escalation)

What You're Building

A multi-step agent that classifies customer issues, looks up account data, searches knowledge bases, processes refunds, and escalates to humans when needed.

Patterns the Exam Expects

  • Routing: Classify issue type (billing/technical/shipping) and route to specialist
  • Hub-and-spoke: Orchestrator delegates to tool-calling subagents
  • Hooks for business rules: PreToolUse on ProcessRefund to enforce $500 limit
  • Escalation: Policy gap, customer request, or threshold exceeded → handoff with summary
  • Context management: Long conversations need compaction; key customer data must survive

Common Traps

  • Using sentiment analysis for escalation decisions (wrong — use explicit triggers)
  • Enforcing refund limits via system prompt instead of hooks (wrong — hooks are deterministic)
  • Sharing full orchestrator context with subagents (wrong — context isolation)

Scenario 2: Code Generation with Claude Code

Domains tested: Domain 3 (Claude Code), Domain 4 (prompts), Domain 1 (architecture)

What You're Building

Using Claude Code to generate, refactor, and review code in a development workflow. Covers CLAUDE.md, worktrees, permissions, and code-quality prompts.

Patterns the Exam Expects

  • CLAUDE.md: Project conventions, tech stack, coding standards as persistent context
  • Worktrees: --worktree feature-name for isolated parallel work
  • Plan mode: Analyze before implementing — plan → review → execute
  • Skills: .claude/commands/ for reusable team workflows (review, test, deploy)
  • Temperature 0.0: Deterministic code generation for consistency

Common Traps

  • Putting secrets in CLAUDE.md (wrong — CLAUDE.md is committed to git)
  • Using CLAUDE.md to enforce security rules (wrong — use permissions/hooks)
  • Running multiple Claude sessions on the same repo without worktrees (wrong — file conflicts)

Scenario 3: Multi-Agent Research System

Domains tested: Domain 1 (orchestration), Domain 5 (reliability), Domain 2 (tools)

What You're Building

Multiple specialized agents collaborate to research a topic — one searches, one analyzes, one summarizes. Orchestrator decomposes the query and aggregates results.

Patterns the Exam Expects

  • Fan-out/fan-in: Decompose query into sub-questions, parallel research, aggregate
  • Dynamic decomposition: Number of sub-questions depends on the research topic
  • Context isolation: Each researcher gets only its sub-question, not the full query context
  • Reduce ≠ concatenation: The synthesis step must resolve contradictions and identify gaps
  • Error handling: If one researcher fails, produce partial results from the others (skip strategy)

Common Traps

  • Sequential research when sub-questions are independent (wrong — parallelize with fan-out)
  • Concatenating research results instead of synthesizing (wrong — reduce means synthesize)
  • Over-decomposing a simple question into 20 sub-questions (wrong — 3–7 is the sweet spot)

Scenario 4: Developer Productivity with Claude

Domains tested: Domain 3 (Claude Code), Domain 2 (MCP), Domain 4 (prompts)

What You're Building

Integrating Claude into a developer's daily workflow through MCP servers for internal tools, hooks for automation, and custom skills.

Patterns the Exam Expects

  • MCP servers: Connect Claude to internal tools (CI, deployment, monitoring) via standard protocol
  • Transport selection: stdio for local tools, Streamable HTTP for shared team services
  • Configuration scoping: .mcp.json (team-shared) vs ~/.claude.json (personal)
  • Tool descriptions: Must explain internal system quirks that Claude wouldn't know
  • Hooks: Automating workflows — PostToolUse to run linter after file writes

Common Traps

  • Using stdio transport for a shared team MCP server (wrong — use Streamable HTTP)
  • Hardcoding credentials in .mcp.json (wrong — use ${ENV_VAR} expansion)
  • One mega-MCP server for everything (wrong — separate servers per system for least privilege)

Scenario 5: Claude Code for CI/CD

Domains tested: Domain 3 (Claude Code), Domain 5 (reliability), Domain 1 (architecture)

What You're Building

Running Claude Code in headless mode (claude -p) inside CI/CD pipelines for automated code review, test generation, and deployment checks.

Patterns the Exam Expects

  • Headless mode: claude -p for non-interactive execution in pipelines
  • GitHub Actions: anthropics/claude-code-action@v1 for PR-triggered workflows
  • Cost control: --max-turns to prevent runaway conversations
  • API key security: GitHub Secrets, never in code or logs
  • Dual-agent pattern: Initializer (plan mode) + workers (parallel worktrees)

Common Traps

  • Running on every PR synchronize event without cost limits (wrong — use opened only or add max_turns)
  • Storing API keys in environment variables in the workflow file (wrong — use GitHub Secrets)
  • Giving CI Claude full write permissions (wrong — use restrictive permission mode)

Scenario 6: Structured Data Extraction

Domains tested: Domain 4 (structured output), Domain 2 (tools), Domain 5 (context)

What You're Building

Extracting structured information from unstructured documents — invoices, contracts, reports — using JSON schema enforcement and validation pipelines.

Patterns the Exam Expects

  • Constrained decoding: output_config.format with JSON schema for guaranteed structure
  • Validation loops: Generate → validate business rules → retry with error feedback
  • Nullable fields: Use ["string", "null"] for fields that may not exist in source
  • Strict tool use: strict: true when extraction feeds into a processing tool
  • Batch API: 50% cost savings for high-volume overnight processing

Common Traps

  • Making all fields required when some may be absent (wrong — forces hallucination)
  • Using only prompting for JSON structure (wrong — use constrained decoding for guarantees)
  • Retrying without error feedback (wrong — tell Claude what failed so it can fix it)

Practice Questions

Test yourself with these exam-style questions. Answers are at the end.

Q1 (Domain 1)

In the Multi-Agent Research scenario, the orchestrator decomposes a research query into 5 sub-questions and spawns parallel subagents. What context should each subagent receive?

A) The full orchestrator conversation history
B) Only the specific sub-question and relevant instructions
C) The original user query plus all other sub-questions
D) The orchestrator's system prompt and the sub-question

Q2 (Domain 1)

A customer support agent needs to enforce a $500 refund limit. What is the most reliable enforcement mechanism?

A) Add "never approve refunds over $500" to the system prompt
B) Use a PreToolUse hook on the refund tool that validates the amount
C) Include the limit in the tool description for ProcessRefund
D) Set a confidence threshold that escalates high-amount refunds

Q3 (Domain 2)

A team wants to share an MCP server that connects to their staging database. Multiple developers on different machines need access. Which transport should they use?

A) stdio
B) Streamable HTTP
C) Server-Sent Events (SSE)
D) WebSocket

Q4 (Domain 3)

A developer wants to test an experimental MCP server without affecting their teammates. Where should they configure it?

A) .mcp.json in the project root
B) .claude/settings.json
C) ~/.claude.json
D) CLAUDE.md

Q5 (Domain 4)

An extraction pipeline produces valid JSON matching the schema, but the "email" field contains "not-available" instead of a valid email. What's the best fix?

A) Add email format validation to the JSON schema
B) Add a validation loop that checks business rules and retries with error feedback
C) Make the email field an enum of known valid emails
D) Use a lower temperature for more deterministic extraction

Q6 (Domain 5)

A customer support agent is at turn 45 and approaching its context limit. The customer's order number was mentioned in turn 3. What's the best approach?

A) Truncate all messages before turn 35
B) Summarize older turns, preserving key facts like order numbers in the summary
C) Increase the context window limit
D) Start a new session and ask the customer to repeat their order number

Q7 (Domain 1)

An agent loop should continue executing when:

A) The agent's response contains the phrase "I'll continue"
B) stop_reason is "tool_use"
C) The iteration count is below the maximum
D) The agent's confidence score is above 0.8

Answers:

Q1: B — Context isolation. Subagents get only task-specific context.

Q2: B — Hooks are deterministic (100% reliable). Prompts are probabilistic.

Q3: B — Streamable HTTP for remote, multi-client access. SSE is deprecated.

Q4: C — ~/.claude.json is user-level, not shared with team.

Q5: B — Business validation beyond schema. JSON Schema can't reliably enforce email format in this context.

Q6: B — Summarization preserves key facts while freeing context. Truncation loses the order number.

Q7: B — Always check stop_reason. Never parse text or use confidence scores for loop control.

Exam-Day Strategy

Time Management: The 3-Pass Approach

Pass 1: Easy (60 min)
→
Pass 2: Medium (40 min)
→
Pass 3: Review (20 min)
  • Pass 1 (60 min): Answer everything you're confident about. Skip anything taking more than 2 minutes. Build momentum.
  • Pass 2 (40 min): Return to skipped questions. With less pressure, answers often become clearer.
  • Pass 3 (20 min): Review flagged answers. Double-check you didn't misread scenario contexts. Trust your first instinct unless you spot a clear error.

Domain-Weighted Prioritization

If you're running short on time, prioritize questions from the heaviest domains:

Never leave a question blank. There's no penalty for wrong answers. Even a guess has a 25% chance. If you're stuck with 30 seconds left, pick the answer that sounds most like "use hooks for business rules" or "context isolation for subagents" — these are overwhelmingly common correct answers on the exam.

The Top 10 Exam Traps

  1. Sentiment-based escalation — Always wrong. Use explicit triggers.
  2. Parsing text for loop control — Always wrong. Check stop_reason.
  3. Prompts for business rules — Almost always wrong. Use hooks.
  4. Full context to subagents — Always wrong. Pass only task-specific context.
  5. stdio for shared remote servers — Always wrong. Use Streamable HTTP.
  6. Required fields for optional data — Wrong. Use nullable types to avoid hallucination.
  7. Sequential processing of independent tasks — Wrong. Use fan-out/parallel.
  8. API keys in code or workflow files — Wrong. Use secrets management.
  9. Over-prompting with aggressive language — Wrong for newer Claude models. Use calm, direct instructions.
  10. Single-agent for complex multi-step tasks — Usually wrong. Consider multi-agent with decomposition.
When you're stuck between two answers, ask yourself: "Which answer is more deterministic and architectural?" The exam consistently favors engineering solutions (hooks, schemas, structured patterns) over probabilistic solutions (prompts, confidence scores, sentiment analysis). If one answer involves code-level enforcement and the other involves prompt-level guidance, pick the code-level answer.
