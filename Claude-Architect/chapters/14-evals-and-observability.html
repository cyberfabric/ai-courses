Chapter 14

Agent Evaluations & Observability

Designing Eval Suites and Production Telemetry — The Architect's Reliability Story for the CCA-F

Why This Chapter Matters

You finished Chapter 13 ready to architect a Managed Agents deployment. There's one thing missing: how do you know it works? Not "works on my machine," not "works in the demo." How do you know it works on Tuesday at 2pm when the prompt got swapped, the model got upgraded, and a new edge case appears in production?

The answer is evaluations — eval suites for short. And the architectural twin of evals is observability: the metrics, traces, and alerts that tell you when production behavior drifts from the eval baseline. Together, evals plus observability are how an architect proves reliability instead of hoping for it.

Domain 5 Connection: Context & Reliability is 15% of the CCA-F. Chapter 11 covered the patterns (retries, circuit breakers, escalation, compaction). This chapter covers the discipline — how you measure that the patterns are doing their job. Both are exam-relevant and tested through the 6 scenarios.

Anthropic shipped a flurry of eval-related tooling between January and April 2026: Harbor + Terminal-Bench 2.0 in January, the skill-creator evals workflow in March, the engineering post Demystifying evals for AI agents in April, and a planning/generation/evaluation three-agent harness pattern. The bar for "production-ready agent" rose. An architect today is expected to design the eval suite alongside the agent, not after it.

The Failure Mode: Ship-Without-Eval Drift

Here's the failure mode every architect should fear. You ship an agent with no eval suite. It works great on Monday. On Tuesday, you tweak the system prompt to "be more concise." On Wednesday, an Anthropic model upgrade ships. On Thursday, your agent now answers half the customer-support tickets with a single dismissive sentence. By Friday, your CSAT is in the basement, and you can't tell which of those three changes broke things.

That's drift. It's silent, it's compounding, and it's invisible to anyone watching only "is the API returning 200?" Evals are how you catch drift. Observability is how you spot it before the eval suite does.

The Three Eval Types

Anthropic's official taxonomy from Demystifying evals for AI agents divides evals into three layers, ordered by complexity:

Single-Turn
Prompt → response → grading. The simplest case. Good for testing classifiers, extractors, summarizers.
Multi-Turn
Multiple back-and-forth turns with tool calls. Tests whether the model maintains context and chooses tools correctly.
Agent
Most complex — agents modify state, adapt to results, and mistakes compound across steps. Closest to production.

That's the development lens. The architect's deployment lens is slightly different but complementary — the same evals, named for where they run:

Pre-Launch Evals
  • Unit checks on discrete steps (tool calls, output schemas)
  • Integration runs that exercise full agent loops
  • Regression suites that gate deployment
  • Run in CI/CD, fast feedback, deterministic where possible
Production Evals
  • Sampled traces from live sessions
  • LLM-as-judge scoring on subjective dimensions
  • Long-tail edge-case capture from real users
  • Run continuously, slower feedback, ground-truthed at scale

An architect's eval suite usually contains both. Pre-launch evals catch obvious regressions before deploy; production evals catch the long-tail cases your dataset never imagined.

Capability vs Regression Evals

Within those layers, Anthropic distinguishes two flavors that behave differently as the agent matures:

Capability Evals
  • Test "can the agent do this thing yet?"
  • Start with a low pass rate — that's the point
  • Watch the rate climb as you iterate on prompts and tools
  • Once saturated (near 100%), they graduate into the regression suite
Regression Evals
  • Test "does the agent still do this thing?"
  • Should sit near 100% always
  • Any dip is a signal — investigate immediately
  • Block deployment when the rate drops below threshold
Exam Tip: If a question describes a team that ships a feature, sees the eval pass rate go from 40% to 85%, and asks what they should do next, the right answer is usually graduate the eval into the regression suite and start writing harder capability evals. The wrong answers will say "stop iterating" or "ship it as-is" — both miss the eval-driven development loop.

Designing Eval Datasets

Anthropic's headline number for getting started: 20–50 simple tasks drawn from real failures. Early-stage evals see large effect sizes, so a small dataset is enough. Mature agents need bigger and harder evals to detect smaller wins.

Quality tasks share four properties:

  • Unambiguous: Two domain experts grading independently should reach the same pass/fail decision. If they disagree, the task is the problem.
  • Has a reference solution: Proves the task is solvable and validates that your grader is configured correctly.
  • Balanced: Test both positive and negative cases. A dataset that's 95% happy paths optimizes for over-triggering. A dataset that's 95% adversarial optimizes for under-triggering. Avoid both.
  • Isolated: Each trial starts from a clean environment. No leftover files, no cached state, no shared databases. Dirty environments inflate scores by introducing correlated failures across trials.
Warning: If a task pass rate is 0% across many trials, your task is probably broken — not the agent. Anthropic explicitly calls this out. Read the actual transcripts before you write a "the model can't do X" report.
Architect Pattern — Scenario-Driven Eval Datasets: The 6 CCA-F scenarios (Customer Support, Code Generation, Multi-Agent Research, Developer Productivity, CI/CD, Structured Data Extraction) are each a natural eval-dataset organization unit. Treat each scenario as a small "suite": 5–10 happy-path tasks, 3–5 edge cases, 2–3 adversarial cases. The exam loves "the team is failing in one scenario, not others" patterns — that's a sign the eval dataset is unbalanced or missing entirely for that scenario.

The Eval Frameworks Landscape

You don't have to build the harness from scratch. As of 2026 the landscape has consolidated around a handful of options:

Harbor
Container-based agent eval framework. Official harness for Terminal-Bench 2.0. Supports Claude Code, Codex CLI, OpenHands, Mini-SWE-Agent. Released January 2026.
skill-creator
Anthropic's official skill workflow with built-in evals, benchmarks, and blind A/B testing. Shipped March 2026.
Braintrust
Vendor-agnostic eval platform. Strong UI for transcript review and dataset curation.
MLflow
Autologs Claude Agent SDK calls. Slots into existing MLOps pipelines.
Agent SDK Hooks
PreToolUse, PostToolUse, SubagentStop hooks (Chapter 7) double as native eval probes — capture every tool call as a graded checkpoint.
Custom + OTel
Roll-your-own with OpenTelemetry traces. Right when you need full data residency or a non-standard environment.

For most architects the right answer is "use Harbor or Braintrust for the harness, instrument with OpenTelemetry, route critical events through Agent SDK hooks." You don't pick one tool; you pick a layered story.

Here's what a minimal Agent-SDK-native eval harness looks like in practice. The hooks from Chapter 7 do most of the work:

# Minimal eval harness using Agent SDK hooks
from anthropic import Anthropic
from anthropic.tools import PreToolUseHook, PostToolUseHook

client = Anthropic()
results = []

def grade_tool_choice(event):
    # Did the agent pick the expected tool for this task?
    expected = event.task.expected_tool
    actual = event.tool_use.name
    results.append({
        "task_id": event.task.id,
        "step": "tool_choice",
        "passed": expected == actual,
        "expected": expected,
        "actual": actual,
    })

def grade_final_output(event):
    # LLM-as-judge on the final response
    judgment = client.messages.create(
        model="claude-haiku-4-5-20251001",
        system="You grade customer-support replies. "
               "Reply only with PASS, FAIL, or UNKNOWN.",
        messages=[{
            "role": "user",
            "content": f"Task: {event.task.prompt}\n"
                       f"Reply: {event.final_output}\n"
                       f"Rubric: {event.task.rubric}"
        }],
        max_tokens=20,
    )
    results.append({
        "task_id": event.task.id,
        "step": "final_output",
        "verdict": judgment.content[0].text.strip(),
    })

# Run all 50 tasks through the harness
for task in load_eval_dataset():
    run_agent(task, hooks=[
        PreToolUseHook(grade_tool_choice),
        PostToolUseHook(grade_final_output),
    ])

print_summary(results)

Two things make this harness real rather than toy. First, the LLM-judge call uses a small fast model (Haiku) and an explicit PASS/FAIL/UNKNOWN rubric — no inflated scores, no hallucinated numbers. Second, every grader is itself a deterministic function. The grader can fail; the result is recorded; nothing is silent.

Eval-Aware Design Patterns

This is where eval design loops back into agent design. Some architectural choices make agents easy to evaluate. Others make evals nearly impossible to write. The architect's job is to bias toward the easy side.

  • Structured outputs as testable boundaries. If your agent produces JSON conforming to a schema (Chapter 10), every field is a discrete grader target. Compare that to free-form prose, which only an LLM judge can score reliably. Whenever you can, push the boundary toward structured.
  • Hooks as deterministic checkpoints. Chapter 7's hooks (PreToolUse, PostToolUse, SubagentStop) fire at known points in the agent lifecycle. They're free eval probes — capture state at every checkpoint and you have a deterministic trace your grader can walk through.
  • Structured error responses as scoring signals. Chapter 5's {is_error, errorCategory, isRetryable, suggestion} error format isn't just for the agent's recovery logic. Each category becomes a graded outcome: did the agent handle retryable errors correctly? Did it escalate non-retryable ones?
Architect Pattern — The Three-Agent Eval Harness: A pattern Anthropic surfaced in April 2026: separate the agent into three roles — planner (decomposes), generator (executes), and evaluator (grades). The evaluator is itself an agent that scores the generator's output against the planner's intent. This decoupling makes long-running autonomous workflows debuggable: when something fails, you can ask which of the three roles broke. It's also the cleanest place to insert LLM-as-judge logic without contaminating the generator.

LLM-as-Judge: When the Grader Is Itself a Model

Some things you want to score don't reduce to "did this string match this string?" Things like "was the tone empathetic?" or "did the explanation cover the key risks?" That's where LLM-as-judge comes in — you ask another model to grade the output.

LLM-as-judge has three failure modes worth designing against:

  1. The judge is sycophantic. If you don't give it permission to fail outputs, it will inflate scores. Solve this with explicit pass/fail rubrics and a reference solution.
  2. The judge is overconfident on subjective dimensions. "Empathy" is hard. Calibrate by sampling 30–50 outputs, having a human grade them, and comparing scores. If the judge and the human diverge by more than ~10%, your rubric needs work.
  3. The judge hallucinates a score. Always offer an "Unknown" or "Cannot determine" option. It's better than a fabricated 7-out-of-10.
Exam Tip: When the exam asks how to evaluate "the agent's tone in customer support replies" or "the appropriateness of a code review comment," the right answer involves an LLM-as-judge with a calibrated rubric — not regex, not exact string match. When the exam asks how to evaluate "did the agent return JSON with the right schema?" the right answer is a deterministic schema validator, not LLM-as-judge. Pick the simplest grader that does the job.

Non-Determinism: pass@k vs pass^k

Agents are stochastic. Run the same eval ten times and you might get nine successes and one failure — or one success and nine failures. Anthropic publishes two metrics that capture this directly:

pass@k

"In k attempts, what's the probability the agent succeeds at least once?"

Use this when one good answer is all you need — a code-generation tool where the user can re-roll, or a research agent where you'll synthesize across attempts.

pass^k

"In k attempts, what's the probability the agent succeeds on every single one?"

Use this when consistency is the product — a customer-support agent where every reply must be on-policy, or an automated CI/CD reviewer that can't flake.

The two metrics diverge fast. At k=1 they're identical. By k=10, pass@k often approaches 100% while pass^k can fall to 0% on the same agent. Choose the metric that matches your product's risk tolerance, not the one that gives the better-looking number.

Production Observability

Pre-launch evals tell you the agent is good on your dataset. Production observability tells you the agent is good on real traffic. They're different questions and they need different instrumentation.

The exam-relevant metric categories for an agent system:

  • Tokens per session. Track median and p95. Budget overruns mean either prompts have grown or the agent is looping.
  • Tool-success rate. Per tool. A tool that quietly drops from 99% to 92% is a regression even if no agent traces error out.
  • Escalation rate. What percent of sessions hand off to a human? Compare to baseline. Spikes indicate the agent's confidence dropped or a new failure mode emerged.
  • First-contact resolution. The customer-support twin of escalation rate. Did the agent solve it without a human?
  • Cost per resolution. Tokens + tool-execution costs / resolutions. The dollar lens on every other metric.
  • Latency. Time-to-first-token, time-to-last-token, end-to-end session duration. SLOs live here.

OpenTelemetry has emerged as the vendor-neutral standard for agent telemetry in 2026. Datadog, Sentry, Arrize, Arthur AI, and Anthropic-internal tooling all emit OTel traces. Spend the hour wiring it in once and you can swap visualizers later for free.

Common Pitfalls: (1) Sampling only happy-path traces — the failures are where the lessons live. (2) Evaluating only at deploy time — drift happens between deploys, not at them. (3) Aggregating metrics across all users when one segment is silently broken — segment by user type, scenario, and tool. (4) Treating the eval pass rate as a vanity number rather than a deployment gate.

The Eval-Driven Iteration Loop

Put it all together and you get an iteration loop that mirrors test-driven development for traditional software:

1. Capture

Pull failures from prod traces and user reports

→

2. Codify

Convert each failure into a deterministic eval task

→

3. Reproduce

Verify the eval fails on the current agent

→

4. Iterate

Tweak prompt, tool, or harness until eval passes

→

5. Graduate

Move passing eval into the regression gate

A/B testing fits between steps 4 and 5. When you have two candidate prompts, run both against the eval suite and against a slice of real production traffic. Promote the winner. The losing candidate becomes a regression task — if your prod metrics ever look like the loser's profile, you know you've regressed.

Anti-Patterns to Avoid

Four anti-patterns the exam tests:
  1. Evaluating only at deploy time. Drift between deploys is invisible. Continuous sampling is non-negotiable.
  2. Sampling only happy-path traces. The 5% of sessions that fail badly tell you more than the 95% that work fine.
  3. Over-trusting LLM-as-judge for subjective dimensions. Without human calibration, the score is fiction.
  4. Treating evals as a one-time setup. Eval suites rot. Schedule a weekly transcript review and a monthly dataset refresh.

Scenario Connections

Architect Pattern — Eval Design Per Exam Scenario:
  • Customer Support: Multi-turn evals with LLM-judge on tone + deterministic checks on escalation triggers. pass^k matters — every reply must be on-policy.
  • Code Generation: Run-the-tests style grader. Did the generated code compile? Pass tests? Has the lint footprint regressed? pass@k acceptable — users iterate.
  • Multi-Agent Research: Multi-turn agent evals with synthesized-judgment LLM judge. Cite-coverage metrics on top of the judge.
  • Developer Productivity: Tool-success rate per MCP server. Track which tools the agent picks for which intents.
  • CI/CD: Deterministic schema graders on PR-comment outputs. SLO on latency — CI/CD has a wall clock.
  • Structured Data Extraction: Field-level grading. Precision and recall per field. Schema-validity gate before any LLM-judge runs.

Practice Questions

Test Yourself: Try answering these before checking the explanations below.

Q1: A team ships a customer-support agent. Their eval suite has 100% pass rate on 30 happy-path tasks. After two weeks in production, complaints spike about "rude" replies. The eval still shows 100%. What's the most likely diagnosis?

A) The model degraded silently — file a bug with Anthropic
B) The eval dataset is unbalanced — no adversarial or tone-sensitive cases
C) The deployment pipeline corrupted the system prompt
D) Production traffic is fundamentally different from any benchmark

Q2: An architect is choosing between pass@k and pass^k for an automated PR-review agent that posts comments. The agent runs once per PR and the comment goes directly to GitHub. Which metric, and why?

A) pass@k — users can re-run the agent if a comment looks off
B) pass^k — every comment ships, so consistency is the product
C) Either is fine — they're identical at k=1
D) Neither — use raw accuracy instead

Q3: A developer wants to evaluate "did the agent's reply express empathy?" Which grading approach is most appropriate?

A) Regex match for empathy keywords ("sorry," "understand," "frustrating")
B) Schema validator on a structured-output empathy field
C) LLM-as-judge with a calibrated rubric and an "Unknown" out-clause
D) Skip the evaluation — empathy is too subjective to measure

Answers:

Q1: B. The eval shows 100% because the dataset doesn't contain the failure mode. This is the classic happy-path-only-sampling anti-pattern. The fix is to expand the dataset with adversarial and tone-sensitive cases, not to blame the model or the deployment pipeline. Production traffic being different (D) is the same answer phrased differently — but B is more diagnostic and actionable.

Q2: B. The PR-review agent has no human-in-the-loop re-roll. Every comment lands. That's the textbook pass^k case — consistency is the product, not just one good answer. pass@k would mask flakiness behind retries that don't exist in production.

Q3: C. Empathy isn't reducible to keyword matching (A) and isn't a structured field (B). It's exactly the subjective dimension LLM-as-judge was designed for — with the critical safeguards: a calibrated rubric and an "Unknown" out-clause. (D) is the lazy answer the exam loves to bait with — the architect's job is to evaluate the hard things, not skip them.

Hands-On Exercise

Design a 10-Task Eval Suite: Pick the Customer Support Resolution Agent scenario from Chapter 12. Design a 10-task eval dataset for it.

Specify:

  1. Five happy-path tasks — common ticket types the agent should resolve cleanly. For each, list the success criterion.
  2. Three edge-case tasks — tickets with ambiguous intent, missing information, or competing constraints. For each, decide what counts as a pass.
  3. Two adversarial tasks — users trying to extract data they shouldn't have, or attempting prompt injection. For each, define the must-not-do behavior.
  4. For each of the 10 tasks, pick a grader: deterministic, LLM-as-judge with rubric, or human review.
  5. Decide which 3 of these 10 will become your regression gate — the ones that block deployment if they regress.

Bonus: Pick the right metric for the suite — pass@k or pass^k? Justify your answer in two sentences.

Quick Review

  • Evals come in three layers: single-turn, multi-turn, and agent — from simple to production-realistic
  • Capability evals start low and graduate into regression evals once they saturate near 100%
  • Start with 20–50 tasks drawn from real failures — small samples have large effect sizes early on
  • Tasks must be unambiguous, balanced (positive + negative), have reference solutions, and run in isolated environments
  • Pick the simplest grader: deterministic for structured output, LLM-as-judge for subjective dimensions, human for calibration
  • Always offer LLM judges an "Unknown" out-clause and calibrate against humans every release cycle
  • pass@k = "at least one success in k tries"; pass^k = "every one of k tries succeeds." They diverge fast — pick the one that matches your product
  • Production observability tracks tokens, tool success, escalation, first-contact resolution, cost, and latency — OpenTelemetry is the 2026 standard
  • Eval-driven iteration: capture → codify → reproduce → iterate → graduate
  • The same eval suite that catches regressions is the architect's portfolio artifact — it's the proof that you designed for reliability, not hoped for it
