Chapter 6

MCP: The Universal Adapter

The Model Context Protocol — how Claude connects to any system through one standardized interface

Completing Domain 2

This chapter finishes Domain 2: Tool Design & MCP Integration (18%). In Chapter 5 you learned how to design tools. Now you'll learn how the Model Context Protocol (MCP) makes those tools available to Claude through a standardized, pluggable architecture.

MCP is to AI agents what USB is to computers. Before USB, every device had its own proprietary connector. Before MCP, every AI integration needed custom code. MCP gives you one protocol that works everywhere.

Why MCP Matters on the Exam

The exam expects you to understand MCP at an implementation level, not just conceptually. You need to know the architecture (hosts, clients, servers), the three primitives (tools, resources, prompts), transport types (stdio vs Streamable HTTP), and how to configure MCP servers in .mcp.json. A 985/1000 scorer stressed that MCP is "not optional" and appears across multiple question types.

The Problem MCP Solves

Without MCP, you have the N×M integration problem. If you have N AI applications and M external tools, you need N×M custom integrations. Each app needs its own code to talk to each tool.

# Without MCP: N×M integrations
Claude Code  → custom code → Slack API
Claude Code  → custom code → GitHub API
Claude Code  → custom code → Database
Claude Desktop → custom code → Slack API  (duplicate!)
Claude Desktop → custom code → GitHub API (duplicate!)
VS Code      → custom code → Slack API  (triplicate!)

With MCP, each tool exposes itself through a standard server. Any MCP-compatible client can connect to it. The math drops from N×M to N+M:

# With MCP: N+M integrations
Claude Code    → MCP Client → MCP Server (Slack)
Claude Desktop → MCP Client → MCP Server (Slack)  (same server!)
VS Code        → MCP Client → MCP Server (Slack)  (same server!)

Each tool builds ONE MCP server. Each app builds ONE MCP client.
Done.
The exam tests whether you understand the N×M problem and how MCP reduces it to N+M. A common question format: "An organization has 4 AI applications and 6 external services. Without MCP, how many integrations are needed? With MCP?" Answer: 24 without, 10 with. Simple multiplication vs addition.

MCP Architecture: The Three Participants

MCP has three key participants. Understanding their roles and relationships is critical for the exam.

Host The AI application (Claude Code, Claude Desktop, VS Code)
Client A component inside the host that connects to one server
Server A program that provides tools, resources, or prompts

MCP Host

The AI application that coordinates everything. Claude Code is a host. Claude Desktop is a host. VS Code with Claude extension is a host. The host creates one MCP client for each MCP server it connects to.

MCP Client

A component inside the host that maintains a dedicated connection to one MCP server. If Claude Code connects to 3 MCP servers, the runtime creates 3 MCP client instances. Each client handles lifecycle management, capability negotiation, and message exchange with its server.

MCP Server

A program that provides context to clients. It can expose tools, resources, and prompts. An MCP server might be a local process (accessing your filesystem) or a remote service (connecting to Sentry, Slack, or a database).

# MCP Architecture
MCP Host (Claude Code)
├── MCP Client 1 ──── MCP Server A (Filesystem - local, stdio)
├── MCP Client 2 ──── MCP Server B (Database - local, stdio)
└── MCP Client 3 ──── MCP Server C (Sentry - remote, HTTP)
Don't confuse "server" with "remote." An MCP server can run locally on your machine (e.g., the filesystem server). The word "server" refers to its role in the protocol (it serves context), not where it runs. Local servers typically use stdio transport; remote servers use Streamable HTTP.

The Two Layers

MCP has two architectural layers:

Data Layer (Inner)

Defines the JSON-RPC 2.0 protocol for communication: how messages are structured, lifecycle management (initialize → negotiate capabilities → exchange data → terminate), and the core primitives (tools, resources, prompts).

Transport Layer (Outer)

Defines how messages travel between client and server. The same JSON-RPC messages work regardless of transport.

Transport Types: stdio vs. Streamable HTTP

This is heavily tested on the exam. You need to know when to use each transport.

Transport How It Works When to Use Example
stdio Standard input/output streams between local processes Local servers on the same machine Filesystem access, local database, local scripts
Streamable HTTP HTTP POST + optional Server-Sent Events for streaming Remote servers across a network Sentry, Slack, cloud APIs, shared team services

stdio is faster (no network overhead) and simpler (no authentication needed for local processes). It serves a single client at a time.

Streamable HTTP supports multiple clients, authentication (bearer tokens, API keys, OAuth), and works across networks. It's what you use when the server is hosted as a service.

A classic exam question: "A team wants to share an MCP server that connects to their internal database. Multiple developers need to connect from different machines. Which transport should they use?" Answer: Streamable HTTP. The key signals are "shared" and "different machines." stdio is for local, single-client use. HTTP is for remote, multi-client use.
Note on SSE: Older documentation mentions Server-Sent Events (SSE) as a separate transport type. SSE has been deprecated in favor of Streamable HTTP, which incorporates SSE for streaming capabilities. If the exam mentions SSE as a standalone option, it's a legacy reference — Streamable HTTP is the current standard for remote transport.

The Three MCP Primitives

MCP servers expose functionality through three primitives. Think of them as three types of things a server can offer:

1. Tools (Actions)

Executable functions that the AI application can invoke. This is what you learned in Chapter 5 — tools have a name, description, and input schema. MCP tools follow the same design principles.

# MCP tool discovery: tools/list response
{
  "tools": [
    {
      "name": "query_database",
      "description": "Execute a read-only SQL query against the project database.
        Returns rows as JSON. Maximum 100 rows per query.",
      "inputSchema": {
        "type": "object",
        "properties": {
          "sql": {"type": "string", "description": "The SQL SELECT query to execute"}
        },
        "required": ["sql"]
      }
    }
  ]
}

2. Resources (Data)

Data sources that provide contextual information. Unlike tools (which do something), resources are things you read. A resource might be a file, a database schema, a configuration, or an API response.

# MCP resource: database schema
{
  "resources": [
    {
      "uri": "db://schema/customers",
      "name": "Customers Table Schema",
      "description": "Column definitions for the customers table",
      "mimeType": "application/json"
    }
  ]
}

The key difference: tools are actions (verbs), resources are data (nouns). A database MCP server might expose a query_database tool (action) and a db://schema resource (data).

3. Prompts (Templates)

Reusable templates that help structure interactions. An MCP server can provide prompt templates with parameters that the client fills in. Think of them as pre-built conversation starters.

# MCP prompt template
{
  "prompts": [
    {
      "name": "analyze_table",
      "description": "Generate a data analysis prompt for a specific table",
      "arguments": [
        {
          "name": "table_name",
          "description": "Name of the database table to analyze",
          "required": true
        }
      ]
    }
  ]
}
Design pattern: combine all three primitives in one server. A well-designed MCP server for a database might expose:

Tools: query_database (execute queries), insert_record (add data)

Resources: Database schema, table descriptions, index information

Prompts: Templates for "analyze this table," "generate report," "find anomalies"

This gives the AI application everything it needs: data for context, tools for action, and prompts for structured interaction. The exam tests whether you can identify which primitive type is appropriate for a given need.

MCP Lifecycle: How Connections Work

When an MCP host connects to a server, it follows a lifecycle:

Initialize
→
Negotiate Capabilities
→
Discover Primitives
→
Exchange Data
→
Terminate
  1. Initialize: Client sends an initialize request with its capabilities and protocol version
  2. Negotiate: Server responds with its capabilities (which primitives it supports)
  3. Discover: Client calls tools/list, resources/list, prompts/list to discover what's available
  4. Exchange: Client calls tools/call, resources/read, etc. as needed
  5. Terminate: Connection closes gracefully

The capability negotiation step is important: it means client and server agree upfront on what they support. If a server doesn't support resources, the client won't try to list them.

Configuring MCP Servers

In Claude Code, MCP servers are configured in JSON files. There are two levels:

File Scope Shared via VCS? Use For
.mcp.json Project-level Yes Team-wide servers (shared database, project APIs)
~/.claude.json User-level No Personal/experimental servers 
// .mcp.json — project-level, committed to git
{
  "mcpServers": {
    "project-db": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres",
               "postgresql://localhost:5432/mydb"],
      "env": {
        "DB_PASSWORD": "${DB_PASSWORD}"  // Environment variable expansion
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "./src"]
    }
  }
}

Notice the ${DB_PASSWORD} syntax — MCP configuration supports environment variable expansion, so you don't hardcode secrets in committed files.

The exam tests the two-level configuration: .mcp.json (project, shared) vs ~/.claude.json (user, personal). A common question: "A developer wants to add an experimental MCP server without affecting teammates. Where should they configure it?" Answer: ~/.claude.json (user-level, not committed to VCS).

Security Boundaries

MCP servers have access to whatever systems they connect to. Security is critical:

Authentication & Authorization

  • Local servers (stdio): Run with the user's permissions. No additional auth needed, but they inherit all the user's access rights.
  • Remote servers (HTTP): Support standard HTTP authentication — bearer tokens, API keys, OAuth. MCP recommends OAuth for token management.

Sandboxing

  • MCP servers should follow the principle of least privilege. A database MCP server should have read-only access unless writes are explicitly required.
  • Filesystem servers should be scoped to specific directories, not the entire filesystem.
Security trap on the exam: An MCP server that connects to a production database with full read-write access, exposed to all team members via .mcp.json. The correct answer is to scope the server to read-only access and use environment variables for credentials, not to embed credentials in the config file.

Common MCP Patterns

The exam references several standard MCP server patterns:

DB
Database Access
Tools for querying, resources for schema. Read-only by default. Scoped to specific tables when possible.
FS
Filesystem
Tools for reading/writing files, resources for directory structure. Scoped to project directory.
API
API Gateway
Wraps internal REST/GraphQL APIs as MCP tools. Handles auth, rate limiting, and response formatting.
Git
Version Control
Tools for git operations, resources for repo metadata (branches, recent commits, diff summaries).

Notifications: Real-Time Updates

MCP servers can send notifications to clients when things change. For example, if a server adds a new tool or a resource is updated, it sends a notifications/tools/list_changed message. The client then re-fetches the tool list.

This is important for long-running sessions where server capabilities might change mid-conversation. The AI application can dynamically adapt to new tools without restarting.

MCP server design tradeoff: granularity of servers.

One big server that wraps everything (database + filesystem + APIs): simpler configuration, single point of failure, hard to scope permissions.

Many small servers with one per service: more configuration overhead, independent failure domains, precise permission scoping.

The exam favors small, focused servers — it aligns with the principle of least privilege. Each server connects to one system and exposes only what's needed. This mirrors the tool design principle from Chapter 5: focused, well-described, single-responsibility.

Scenario Connections

Here's where MCP shows up in the exam scenarios:

  • Customer Support: MCP server for the ticketing system (tools for ticket CRUD, resources for ticket schemas), MCP server for the knowledge base (tools for search, resources for article index).
  • Developer Productivity: MCP servers for internal tools — CI system, deployment platform, monitoring dashboard. This is the primary MCP scenario. The developer connects Claude to their entire toolchain through MCP.
  • Data Extraction: MCP servers for data sources — document storage, external APIs, validation services. Resources provide schema definitions; tools do the extraction.
  • CI/CD: MCP servers integrated into the pipeline for test runners, linters, and deployment tools. Configuration via .mcp.json committed to the repo.
Exercise: Design an MCP Server Configuration

Your team needs Claude Code to access three systems: a PostgreSQL database, a Jira project tracker, and a private NPM registry. Design the MCP configuration:

  1. For each system, decide: local server (stdio) or remote server (Streamable HTTP)?
  2. Write the .mcp.json configuration with all three servers
  3. For each server, list what you'd expose as tools vs. resources vs. prompts
  4. Where do credentials go? (Hint: environment variables, never hardcoded)
  5. Which servers should be in .mcp.json (team-shared) vs ~/.claude.json (personal)?

Check yourself: PostgreSQL is likely local (stdio) if it's a dev database on localhost, or remote (HTTP) if it's a shared staging database. Jira is definitely remote (HTTP) — it's a cloud service. NPM registry depends on whether it's local or hosted. Credentials go in environment variables via ${VAR} syntax. Database and Jira servers are team-shared (.mcp.json). A personal experimental server goes in ~/.claude.json.

MCP vs. Direct Tool Integration

A final question the exam tests: when should you use MCP, and when should you just define tools directly in the API?

Use MCP When... Use Direct Tools When...
Multiple AI apps need the same tool Only one app uses the tool
You want the team to share tool configs The tool is specific to one agent's role
You need standard discovery and lifecycle You need tight control over tool behavior
The tool connects to an external system The tool is pure computation (no I/O)

Quick Review

  • MCP solves the N×M integration problem by standardizing how AI apps connect to tools — reducing integrations to N+M
  • Three participants: host (AI app), client (connection component), server (provides context) — one client per server connection
  • Two transports: stdio (local, single client, no auth) vs Streamable HTTP (remote, multi-client, auth support) — choose based on locality
  • Three primitives: tools (actions/verbs), resources (data/nouns), prompts (templates) — know which to use for each need
  • Configuration: .mcp.json (project, shared via VCS) vs ~/.claude.json (user, personal) — use env vars for credentials
