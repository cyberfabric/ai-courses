Managed Agents for Architects

Anthropic's Fully Managed Agent Runtime — Architecture, Decisions, and Exam Connections

Why This Chapter Matters

Chapters 1–12 gave you everything you need to pass the CCA-F. So why Chapter 13? Because Anthropic launched Claude Managed Agents on April 8, 2026 — a fully managed agent runtime that changes how you think about building production agent systems. Even if the exam was designed before this launch, Managed Agents implements the exact patterns tested across all five domains. Understanding it deepens your grasp of every exam concept.

Think of Managed Agents as the "cloud-hosted Claude Code." Instead of running agents on your own infrastructure, Anthropic provides the harness, the sandbox, the tool execution, and the session management. You just define what the agent should do.

The Four Pillars

Managed Agents is built around four core concepts. If you remember nothing else from this chapter, remember these:

Agent

Model, system prompt, tools, MCP servers, skills

→

Environment

Container template: packages, network access

→

Session

A running agent instance performing a task

→

Events

Messages between your app and the agent

Here's how they come together in code. Creating an agent is just an API call:

# Python SDK
from anthropic import Anthropic
client = Anthropic()

agent = client.beta.agents.create(
    name="Support Agent",
    model="claude-sonnet-4-6",
    system="You are a customer support agent. Follow escalation policies.",
    tools=[{"type": "agent_toolset_20260401"}],
)

environment = client.beta.environments.create(
    name="support-env",
    config={
        "type": "cloud",
        "networking": {"type": "unrestricted"},
    },
)

session = client.beta.sessions.create(
    agent=agent.id,
    environment_id=environment.id,
    title="Ticket #4521",
)
Beta Header Required: Claude Managed Agents is in beta — every API call requires the anthropic-beta: managed-agents-2026-04-01 header. The official Anthropic SDKs set this automatically; raw HTTP clients must add it manually or every call will fail.

The agent_toolset_20260401 tool type enables all built-in tools: bash, file operations, web search, and more. You create the agent once and reuse its ID across many sessions.

Key Insight: Agents and environments are templates. Sessions are instances. One agent definition can power thousands of concurrent sessions, each in its own isolated environment. This maps directly to the hub-and-spoke pattern from Chapter 4.

Architecture: Decoupling the Brain from the Hands

The most important architectural insight in Managed Agents is the separation of three concerns:

Brain
Claude + stateless harness — decides what to do
Hands
Sandboxed containers — execute tool calls
Session Log
Append-only event store — durable state outside the context window

Why does this matter? In a traditional agent harness, the orchestrator lives inside the container. If the container crashes, the agent dies. In Managed Agents, the harness is stateless — if it fails, a new instance picks up the session log and resumes exactly where it left off. Anthropic's engineering team calls this turning harnesses from "pets" (hand-maintained, irreplaceable) into "cattle" (interchangeable, disposable).

Containers are provisioned on-demand. If a session only needs Claude to reason (no bash commands, no file writes), no container spins up at all. This is why Managed Agents achieved a ~60% reduction in median time-to-first-token and >90% at the 95th percentile compared to the coupled architecture.

Architectural Pattern — Brain/Hands Decoupling: This is the same principle behind microservices vs monoliths. By separating reasoning (brain) from execution (hands) and state (session log), each component scales independently. The brain scales with inference demand, the hands scale with tool execution demand, and the session log provides durability. When the exam asks about agent lifecycle management or error recovery, think about this pattern.

Domain 1: Agentic Architecture (27%)

Managed Agents is a concrete implementation of the orchestration patterns from Chapters 2–4. Here's how the exam concepts map:

  • Hub-and-spoke: The managed harness is the hub. It routes tool calls to containers (the spokes). You configure the routing through the agent definition.
  • Agent loop: Managed Agents runs the plan → act → observe loop for you. Claude decides which tools to use, the harness executes them in the container, and results stream back as events.
  • Task decomposition: In the multi-agent research preview, one agent can spawn sub-agents to parallelize work — exactly the fan-out/fan-in pattern from Chapter 3.
  • Error handling: When a container crashes, the harness treats it as a tool-call error and passes it to Claude for retry. Claude can reinitialize containers using standard recipes. No information is lost because the session log is durable.
Exam Tip: If a question asks about agent lifecycle management or what happens when a tool execution fails, the correct answer usually involves the orchestrator catching the error, logging it, and giving the model a chance to retry or escalate — not silently failing or restarting the entire session. Managed Agents implements this pattern exactly.

Domain 2: Tool Integration (18%)

Remember Chapter 5's tool design principles? They all apply to Managed Agents, with some platform-specific additions:

Built-in tools include bash, file read/write/edit, glob, grep, web search, and web fetch. You enable the full set with agent_toolset_20260401, or configure individual tools for finer control.

MCP server attachment lets you connect external tool providers. This is the same MCP architecture from Chapter 6 — your agent can call tools exposed by MCP servers running alongside the container.

# Agent with MCP server for database access
agent = client.beta.agents.create(
    name="Data Analyst",
    model="claude-sonnet-4-6",
    system="You analyze data. Use SQL queries through the database tool.",
    tools=[
        {"type": "agent_toolset_20260401"},
    ],
    mcp_servers=[
        {
            "name": "postgres",
            "url": "https://your-mcp-server.example.com/mcp",
        }
    ],
)

The tool design principles from Chapter 5 apply directly: clear descriptions, well-defined schemas, one-purpose tools. The difference is that Managed Agents handles the tool execution infrastructure for you.

Domain 3: Claude Code Workflows (20%)

Think of Managed Agents as the cloud counterpart to local Claude Code. Here's a comparison:

Claude Code (Local)
  • Runs on your machine
  • CLAUDE.md for project context
  • Git worktrees for isolation
  • Hooks for automation
  • claude -p for CI/CD
  • Full local file system access
Managed Agents (Cloud)
  • Runs on Anthropic's infrastructure
  • System prompt for agent context
  • Environments for isolation
  • Events for automation
  • API calls for CI/CD
  • Sandboxed container file system

For CI/CD integration (the Claude Code for CI/CD scenario), Managed Agents offers a compelling alternative: instead of installing Claude Code in your pipeline runner, you make API calls to create sessions and stream results. This is cleaner for environments where you can't install CLI tools.

# In a CI/CD pipeline script
session = client.beta.sessions.create(
    agent="agent_01abc...",          # pre-configured code review agent
    environment_id="env_01xyz...",   # env with project dependencies
    title=f"PR #{pr_number} review",
)

# Send the PR diff as a user message
client.beta.sessions.events.send(session.id, events=[{
    "type": "user.message",
    "content": [{"type": "text", "text": f"Review this PR diff:\n{diff}"}],
}])

# Stream results back to the PR comment
with client.beta.sessions.events.stream(session.id) as stream:
    for event in stream:
        if event.type == "agent.message":
            post_pr_comment(event.content)
        elif event.type == "session.status_idle":
            break

Domain 4: Prompt Engineering (20%)

The system prompt in a Managed Agent serves the same role as CLAUDE.md in Claude Code — it's persistent context that shapes every interaction. But there are differences:

  • Agent-level system prompt: Defined once in the agent creation call. Applied to every session.
  • Session-level context: Passed through user events at session start. Varies per task.
  • Tool descriptions: Built-in tools have pre-optimized descriptions. Custom MCP tools need your own clear descriptions (Chapter 5 principles).

The prompt engineering techniques from Chapter 9 all apply: role prompting, explicit criteria, few-shot examples. The key architectural decision is what goes in the agent system prompt vs what goes in the session's first user message.

Rule of Thumb: Put stable instructions in the agent system prompt (role, constraints, output format). Put task-specific context in the session's user events (the specific ticket, the PR diff, the research query). This mirrors the CLAUDE.md vs conversation split in Claude Code.

Domain 5: Context & Reliability (15%)

This is where Managed Agents truly shines from an architectural perspective. The platform handles the hardest parts of reliability engineering:

  • Session log as durable state: The append-only event log lives outside Claude's context window. The harness can query it with getEvents(), selecting positional slices — rewind, skip forward, or reread context before a specific action.
  • Built-in compaction: The managed harness includes prompt caching and compaction strategies. You don't implement sliding-window truncation yourself.
  • Graceful degradation: Container failures become tool-call errors. The model can retry, reinitialize, or escalate.
  • Event streaming: Server-sent events (SSE) provide real-time visibility into agent execution. You can steer or interrupt the agent mid-task by sending additional user events.

The session lifecycle follows a clear pattern:

Created

Session initialized, no work started

→

Running

Agent is processing, tools executing

→

Idle

Agent finished, waiting for input

→

Ended

Session complete or timed out

Exam Tip: When the exam asks about long-running agent sessions or maintaining state across multiple interactions, the correct pattern is external state management (like a session log) rather than cramming everything into the context window. Managed Agents implements this with its durable event log. The "lost in the middle" problem from Chapter 11 is mitigated by the harness's ability to selectively retrieve context from the log.

The Architectural Decision: Managed vs Self-Hosted

This is the most important section for exam-style thinking. When should you use Managed Agents vs building your own agent harness with the Messages API and Agent SDK?

Factor Managed Agents Self-Hosted (Messages API + Agent SDK)
Infrastructure overhead Near-zero — Anthropic runs everything You manage containers, sandboxing, scaling
Customization Configure within platform constraints Full control over every component
Latency control Depends on Anthropic's infrastructure You choose compute location and resources
Compliance Data processed on Anthropic infrastructure Data stays in your environment
Cost model Token rates + $0.08/session-hour Token rates + your own infrastructure costs
Agent loop control Platform-managed loop with steering via events You implement the full plan-act-observe cycle
Tool execution Sandboxed containers with built-in tools You build tool execution and sandboxing
Best for Long-running tasks, async work, rapid prototyping Custom loops, data-sensitive workloads, edge deployment
Architectural Tradeoff — Managed vs Custom: This is a classic build-vs-buy decision. Managed Agents trades customization for operational simplicity. Choose managed when your agent fits the platform's patterns (long-running tasks, standard tools, cloud-first). Choose self-hosted when you need custom agent loops, strict data residency, or non-standard execution environments. The exam loves testing this kind of tradeoff — there's rarely a universally "right" answer, only a right answer for the scenario.

Environment Design

Environments define the container where your agent executes tools. Key configuration options:

# Environment with specific packages and restricted networking
environment = client.beta.environments.create(
    name="data-analysis-env",
    config={
        "type": "cloud",
        "networking": {
            "type": "restricted",
            "allowed_domains": ["api.internal.company.com", "pypi.org"],
        },
    },
)

Security boundaries in Managed Agents follow two patterns:

  • Resource-bundled auth: Git tokens are used during container setup to clone repos. They're wired into local git remotes — the agent never handles credentials directly. Push/pull just works.
  • Vault-proxied access: OAuth tokens for custom tools (via MCP) live in secure vaults outside the sandbox. A dedicated proxy retrieves credentials and makes external calls. The agent never sees the actual tokens.
Security Pattern: Never pass API keys or credentials directly in the agent's system prompt or user messages. Use environment-level configuration or vault-proxied MCP servers. This applies to both Managed Agents and self-hosted setups — it's a universal agent security principle the exam tests.

Research Previews and Future Directions

Three features round out the platform: Memory recently graduated to public beta (April 2026), while Multi-Agent and Outcomes remain research previews (separate access required). All three signal where the platform is heading:

Multi-Agent
One agent spawns and directs sub-agents to parallelize work. Implements the fan-out/fan-in pattern from Chapter 3 at the platform level.
Memory (Public Beta)
Agents learn across sessions through a file-backed memory layer. Memories are stored as files with scoped permissions and audit logs — org-wide read-only stores or per-user read/write stores. Addresses the context limits from Chapter 11.
Outcomes
Agents define success criteria and self-evaluate. They iterate until criteria are met, enabling more autonomous operation.
For the Exam: These research previews aren't directly tested (the exam predates them), but they implement patterns that are tested: multi-agent orchestration, context persistence, and self-evaluation loops. Understanding them reinforces your grasp of the core concepts.

Exam Scenario Connections

Here's how Managed Agents maps to the 6 exam scenarios:

  • Customer Support Resolution Agent: Long-running sessions with escalation. Managed Agents handles session persistence, so the agent can maintain context across a multi-turn support conversation. Events let you inject supervisor guidance mid-session (the escalation pattern from Chapter 4).
  • Code Generation with Claude Code: Environments pre-loaded with project dependencies. The agent writes code, runs tests, and iterates — all inside a sandboxed container. This is exactly what Claude Code does locally, but cloud-hosted.
  • Multi-Agent Research System: The multi-agent preview enables one orchestrator to spawn researcher sub-agents in parallel. Each runs in its own session with a shared environment template. Results are aggregated through event streaming.
  • Developer Productivity: Agents with MCP servers connected to internal tools (Jira, GitHub, Slack). The environment provides network access to internal APIs while sandboxing tool execution.
  • Claude Code for CI/CD: Sessions triggered by pipeline events (PR opened, merge to main). The API-first design makes integration straightforward — no CLI installation required.
  • Structured Data Extraction: Agents that process documents in the container, extract structured data, and validate against schemas. The environment provides the parsing tools; the agent handles the extraction logic from Chapter 10.

Practice Questions

Test Yourself: Try answering these before checking the explanations below.

Q1: A financial services company needs an agent that processes sensitive customer data and must comply with data residency requirements (data cannot leave their AWS region). Which approach is most appropriate?

A) Managed Agents with restricted networking
B) Self-hosted agent using Messages API + Agent SDK
C) Managed Agents with vault-proxied authentication
D) Claude Code running in their CI/CD pipeline

Q2: In Managed Agents, what happens when a container crashes during tool execution?

A) The entire session is terminated and must be restarted
B) The harness automatically retries the exact same tool call
C) The failure is passed to Claude as a tool-call error for decision-making
D) The session enters an error state and waits for manual intervention

Q3: An architect is designing a multi-agent research system using Managed Agents. Each sub-agent needs different packages (one needs Python scientific libraries, another needs Node.js). What's the correct approach?

A) Install all packages in a single shared environment
B) Create separate environment templates for each agent type
C) Use the system prompt to instruct agents to install packages at runtime
D) Use a single agent with all tools instead of multiple agents

Q4: What is the primary architectural benefit of Managed Agents' brain/hands decoupling?

A) Lower token costs
B) Faster model inference
C) Independent scaling and failure recovery of reasoning and execution
D) Support for more programming languages

Answers:

Q1: B. Data residency requirements mean data must stay in their environment. Managed Agents processes data on Anthropic's infrastructure, which doesn't satisfy the compliance constraint. Self-hosted gives them full control over where data lives.

Q2: C. The harness treats container failures as tool-call errors and passes them to Claude. Claude can then decide to retry, reinitialize the container, or take an alternative approach. This is the graceful degradation pattern from Chapter 11.

Q3: B. Environments are templates. Create separate environments optimized for each agent's needs. A Python-heavy researcher gets a Python environment; a Node.js agent gets a Node.js environment. This follows the principle of agent specialization from Chapter 2.

Q4: C. The decoupling allows the brain (reasoning) and hands (execution) to scale independently and recover from failures independently. The session log provides durability. This is the core architectural insight.

  • Choose managed for long-running async tasks with standard tools; choose self-hosted for custom loops, data residency, or edge deployment
  • All 5 exam domains map to Managed Agents: orchestration patterns, tool integration, Claude Code parallels, prompt design, and session reliability
