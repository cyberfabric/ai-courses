Chapter 11

Context Management & Reliability

Token budgets, compaction, escalation, graceful degradation — keeping agents on track when things get messy

Domain 5: The Final Domain

This chapter covers Domain 5: Context Management & Reliability, which makes up 15% of the exam (~9 questions). It's the lightest domain by weight, but don't underestimate it. Context management questions are tricky because they test nuance — there's rarely a single right answer, and the distractors are plausible.

This domain is about what happens when things don't go perfectly: the context window fills up, the API returns an error, the agent can't handle a request, or a conversation runs for 50 turns. Production systems need strategies for all of these.

Domain 5 on the Exam

This domain covers context degradation prevention, escalation design, error propagation in multi-agent systems, and human-in-the-loop review workflows. The exam scenarios most tied to this chapter are Customer Support (long conversations, escalation) and Multi-Agent Research (context across agents).

Context Window Arithmetic

Every Claude request has a token budget. Understanding how it's consumed is fundamental:

Total context = System prompt
             + Conversation history (all messages)
             + Tool definitions
             + Tool results
             + Current input
             ─────────────────
             Must fit within model's context window

Current context windows:

200K Standard context (Opus 4.6, Sonnet 4.6)
1M Extended context (Opus 4.6 with 1M)
~346 Tokens added by tool use system prompt

The key insight: every token matters in long-running agent sessions. A system prompt uses tokens on every request. Tool definitions use tokens on every request. Conversation history grows with every turn. In a 50-turn customer support conversation, the conversation history alone might consume 80% of the context window.

The exam tests context window arithmetic. A question might describe: "An agent has a 200K context window, a 3K system prompt, 5 tools at ~500 tokens each, and a 40-turn conversation averaging 2K tokens per turn. Is the agent approaching its context limit?" You need to calculate: 3K + 2.5K + 80K + 346 = ~86K tokens. That's 43% used — plenty of room. But if it says 120-turn conversation: 3K + 2.5K + 240K + 346 = ~246K. That exceeds the 200K limit. Know how to do this math quickly.

The "Lost in the Middle" Problem

Language models have better recall for information at the beginning and end of the context window. Information in the middle gets weaker attention — the so-called "lost in the middle" problem.

In practice, this means:

  • System prompt (at the beginning) is always well-attended to
  • Recent messages (at the end) get strong attention
  • Messages from 30 turns ago (in the middle) may be forgotten or misremembered

This has real architectural implications. In the Customer Support scenario, a customer mentions their order number in turn 3, and by turn 40 the agent may have lost track of it because it's buried in the middle of the conversation.

Design pattern: critical information at the edges. Place the most important context in positions where recall is strongest:

Beginning (system prompt): Agent identity, constraints, business rules, tool descriptions

End (recent turns): Current task, recent tool results, active conversation context

For critical middle-context: Use summarization to resurface important early information near the end of the conversation. Or store key facts (customer ID, order number) in the system prompt dynamically.

Compaction Strategies

When context grows too large, you need to compress it. This is called compaction. Claude Code has automatic compaction built in; for custom agents, you implement it yourself.

1. Summarization

Replace older conversation turns with a summary. A 20-turn exchange becomes a 500-token summary. This preserves the key information while freeing up context space.

# Summarization compaction
if token_count(messages) > THRESHOLD:
    old_messages = messages[:-10]  # Keep last 10 turns intact
    summary = claude_summarize(old_messages)
    messages = [
        {"role": "user", "content": f"[Previous conversation summary: {summary}]"},
        *messages[-10:]  # Recent turns preserved in full
    ]

2. Truncation

Simply drop the oldest messages. Fast and simple, but you lose information permanently. Use when the old context is genuinely irrelevant.

3. Sliding Window

Keep only the last N turns. As new turns arrive, the oldest drop off. Good for conversations where recent context is all that matters.

4. Selective Pruning

Remove specific types of content: verbose tool results, intermediate reasoning, or repeated information. Keep high-value content (decisions, user requests, key data) and prune low-value content (raw API responses, exploratory tool calls).

Strategy Preserves Info? Complexity Best For
Summarization Mostly yes High (needs LLM call) Long, information-dense conversations
Truncation No Low Conversations where old context is irrelevant
Sliding window Partial Low Real-time interactions, chat
Selective pruning Mostly yes Medium Tool-heavy agent sessions
The exam frequently asks: "An agent is approaching its context limit after a long customer support interaction. What's the best compaction strategy?" The answer depends on what's important. If the customer's issue details matter: summarization (preserves key info). If only the recent discussion matters: sliding window. If tool results are bloating context: selective pruning. The exam tests whether you can match the strategy to the scenario.

Claude Code's Built-in Compaction

Claude Code automatically compacts context when approaching the window limit. It summarizes the conversation history and continues seamlessly. You can customize what's preserved using the PreCompact hook (from Chapter 7) to ensure critical information survives compaction.

  "hooks": {
    "PreCompact": [{
      "matcher": "",
      "hooks": [{

      }]
    }]
  }
}

Escalation Patterns

We covered escalation triggers in Chapter 4. Now let's go deeper into the architecture of escalation — how agent-to-human handoff actually works in production.

Tiered Support Architecture

Tier 1: Agent
→
Tier 2: Specialist Agent
→
Tier 3: Human Agent
→
Tier 4: Manager

Each tier handles progressively more complex issues:

  • Tier 1 (AI agent): Handles common questions — password resets, order status, FAQ lookups. Resolves 70-80% of tickets.
  • Tier 2 (specialist agent): Handles complex issues requiring multiple tool calls — billing disputes, technical troubleshooting.
  • Tier 3 (human agent): Issues requiring judgment, policy exceptions, or empathy that AI can't provide.
  • Tier 4 (manager): Escalation authority for high-value customers or policy changes.

The Escalation Handoff

When escalating, the agent must pass enough context for the next tier to continue without re-asking the customer. The handoff payload should include:

{
  "escalation": {
    "from": "tier-1-agent",
    "to": "tier-3-human",
    "reason": "refund_exceeds_policy_limit",
    "customer_id": "C-12345",
    "conversation_summary": "Customer reports double charge of $299.99 on order #O-789. Verified duplicate in billing system. Refund amount ($299.99) exceeds tier-1 limit ($200).",
    "actions_taken": [
      "Verified customer identity",
      "Confirmed duplicate charge in billing system",
      "Attempted standard refund — blocked by limit"
    ],
    "recommended_action": "Approve $299.99 refund for duplicate charge",
    "full_transcript_id": "session-abc-123"
  }
}
Escalation anti-pattern: the "cold handoff." The agent transfers to a human with just "Customer needs help." The human has no context, asks the customer to repeat everything, and the customer is frustrated. Always include a summary, actions taken, and recommendation. The exam specifically tests the quality of escalation handoffs.

Error Handling and Retry Logic

Production agents encounter errors constantly: API timeouts, rate limits, tool failures, invalid responses. Here's the standard approach:

Exponential Backoff

import time

def retry_with_backoff(func, max_retries=3, base_delay=1):
    for attempt in range(max_retries):
        try:
            return func()
        except RateLimitError:
            delay = base_delay * (2 ** attempt)  # 1s, 2s, 4s
            time.sleep(delay)
        except PermanentError:
            raise  # Don't retry permanent errors
    raise MaxRetriesExceeded()

Circuit Breaker Pattern

If a service fails repeatedly, stop trying for a cooldown period. This prevents cascading failures and gives the service time to recover.

# Circuit breaker states
CLOSED  → requests flow normally
OPEN    → all requests fail immediately (after N consecutive failures)
HALF-OPEN → allow one test request; if it succeeds, close the circuit
The reliability stack for production agents.

Layer 1 — Retry with backoff: For transient errors (timeouts, rate limits). 3 retries, exponential backoff.

Layer 2 — Circuit breaker: For persistent service failures. Stop retrying, return a fallback response.

Layer 3 — Graceful degradation: When a capability is unavailable, provide reduced functionality rather than failing completely.

Layer 4 — Escalation: When all else fails, hand off to a human with full context.

These four layers are how you build a system that stays up even when individual components fail. The exam tests whether you know which layer to apply for which failure type.

Graceful Degradation

When a component fails, the system should degrade gracefully rather than crash completely:

Failure Graceful Response Bad Response
Knowledge base unavailable Answer from general knowledge, flag as "unverified" Crash or return empty response
Context window overflow Compact and continue with summary Drop the entire conversation
Tool call fails Try alternative tool or ask user directly Stop processing without explanation
Rate limit hit Queue request, inform user of delay Return error to customer
When the exam presents a failure scenario, the correct answer almost always involves continuing with reduced capability rather than stopping entirely. "The knowledge base is temporarily unavailable. What should the agent do?" The answer is not "tell the customer to try later." It's "answer from available context, mark the response as unverified, and offer to follow up when the KB is available."

Rate Limiting and Quota Management

Claude API has rate limits (requests per minute) and spending limits. Production agents must handle both:

  • Request queuing: When approaching rate limits, queue requests instead of dropping them
  • Priority queuing: High-priority requests (escalations, VIP customers) skip the queue
  • Token tracking: Monitor token usage per session and per day for cost control
  • Model fallback: If Opus hits its limit, fall back to Sonnet for non-critical tasks

Monitoring and Observability

You can't fix what you can't see. Production agent systems need monitoring at three levels:

1. Agent-Level Metrics

  • Tokens consumed per session
  • Tool call success/failure rates
  • Average turns to resolution
  • Escalation rate

2. System-Level Metrics

  • API latency (p50, p95, p99)
  • Error rates by type (rate limit, timeout, validation failure)
  • Context window utilization
  • Cost per interaction

3. Business-Level Metrics

  • Customer satisfaction score
  • First-contact resolution rate
  • Time to resolution
  • Cost per resolution vs. human agent cost
The exam doesn't test specific monitoring tools, but it does test whether you understand what to monitor. If a question asks "How would you detect that an agent is degrading in performance?", the answer involves metrics like increasing escalation rate, growing token usage per session, or declining first-contact resolution rate — not specific dashboards or tools.

Long-Running Interactions

Some agent sessions run for hours or span multiple days. Session persistence and state management are critical:

Session Persistence Strategies

  • Full transcript storage: Save the entire conversation history. Resume by replaying it. Accurate but context-expensive.
  • State snapshot: Save a summary + key state variables (customer ID, issue status, actions taken). Resume from the snapshot. More efficient.
  • External state store: Keep conversation state in a database. The agent reads/writes state on each turn. Scales to indefinite sessions.

Claude Code's session management (from Chapter 8) uses full transcript storage with automatic compaction. For custom agents, state snapshots are often the best balance of accuracy and efficiency.

Scenario Connections

  • Customer Support: Long conversations that push context limits. Escalation to human agents. Graceful degradation when knowledge base is down. Token budget management across many turns.
  • Multi-Agent Research: Context isolation between agents prevents one agent's work from polluting another's context. Result aggregation must fit in the orchestrator's context window.
  • CI/CD: Rate limiting when many PRs trigger Claude reviews simultaneously. Circuit breaker for API outages. Cost monitoring for automated pipelines.
  • Data Extraction: Batch processing with token budget awareness. Graceful handling of documents that exceed single-context capacity (multi-pass extraction).

Tier 1 (agent, <$200 refunds), Tier 2 (specialist, complex billing), Tier 3 (human, policy exceptions). Metrics: escalation rate (alert >30%), avg tokens/session (alert >100K), first-contact resolution (alert <60%), API error rate (alert >5%), cost/ticket (alert >$2). Priority queue + model fallback (Sonnet for simple queries) for the spike.

Quick Review

  • Context arithmetic: system prompt + history + tools + input must fit the window — long conversations are the biggest consumer
  • "Lost in the middle": information at the edges gets better recall — place critical context in system prompt (beginning) or recent turns (end)
  • Compaction strategies: summarization (preserves info), truncation (loses info), sliding window (recent only), selective pruning (remove low-value content)
  • Reliability stack: retry with backoff → circuit breaker → graceful degradation → escalation to human
  • Escalation handoffs must include summary, actions taken, and recommendation — never do a "cold handoff"
