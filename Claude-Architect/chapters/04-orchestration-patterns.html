Chapter 4

Orchestration Patterns

The playbook for coordinating agents — hub-and-spoke, pipelines, map-reduce, routing, and when each one wins

Completing Domain 1

This is our final chapter on Domain 1: Agentic Architecture & Orchestration (27%). In Chapter 2, you learned why multi-agent systems exist. In Chapter 3, you learned how to break tasks apart. Now we bring it together: which orchestration pattern do you choose, and why?

The exam doesn't just ask you to name patterns — it presents a scenario and asks you to pick the right one. This chapter gives you the decision framework to do that confidently.

The Five Patterns You Must Know

Every orchestration question on the exam maps to one of these: hub-and-spoke, pipeline/chain, map-reduce, routing, or escalation/fallback. Know what each one looks like, when to use it, and when it's the wrong choice.

Pattern 1: Hub-and-Spoke (In Depth)

We introduced hub-and-spoke in Chapter 2. Now let's go deeper into the orchestrator's responsibilities, because this is where the exam gets specific.

The Orchestrator's Job

The hub (coordinator) has four responsibilities:

  1. Task decomposition: Break the incoming request into subtasks
  2. Delegation: Assign each subtask to the right subagent with the right context
  3. Aggregation: Collect results from all subagents and synthesize a final output
  4. Error handling: Detect subagent failures and respond (retry, skip, or escalate)
# Hub-and-spoke orchestrator pseudocode
def orchestrate(user_request):
    # 1. Decompose
    subtasks = decompose(user_request)

    # 2. Delegate (parallel where possible)
    results = {}
    for task in subtasks:
        subagent = spawn_subagent(
            prompt=task.description,
            tools=task.required_tools,
            context=task.relevant_context  # NOT full conversation
        )
        results[task.id] = subagent.run()

    # 3. Aggregate
    synthesis = synthesize(results)

    # 4. Error handling (simplified)
    failed = [t for t in results if results[t].failed]
    if any(t.critical for t in failed):
        return escalate(failed)

    return synthesis
When the exam asks about the orchestrator's role, all four responsibilities matter. A common distractor answer describes an orchestrator that only delegates but doesn't aggregate — or one that aggregates but doesn't handle errors. The complete orchestrator does all four: decompose, delegate, aggregate, and handle errors.

The allowedTools Requirement

For the orchestrator to spawn subagents in the Agent SDK, its allowedTools must include "Task". This is a specific implementation detail the exam tests. Without it, the orchestrator can't create subagents.

# Orchestrator configuration in Agent SDK
{
  "agent": "coordinator",
  "allowedTools": ["Task"],  // Required for subagent spawning
  "system_prompt": "You are a coordinator that delegates research tasks..."
}

# Subagent configuration (spawned by Task tool)
{
  "agent": "researcher",
  "allowedTools": ["WebSearch", "ReadDocument"],
  "system_prompt": "You are a researcher. Search for..."
}

Parallel Subagent Execution

When the orchestrator makes multiple Task tool calls in a single response, they execute in parallel. This is how you get the performance benefit of multi-agent systems. The orchestrator doesn't need to wait for one subagent to finish before starting the next.

The number of parallel subagents matters. Too few (1–2) and you lose the parallelism benefit. Too many (20+) and you overwhelm the system with coordination overhead. The sweet spot for most scenarios is 3–7 parallel subagents. The exam occasionally tests this judgment.

Pattern 2: Pipeline / Chain

In a pipeline, agents process data sequentially. Each agent transforms the input and passes it to the next. Think of an assembly line: each station adds something, and the product moves forward.

Agent A: Extract
→
Agent B: Transform
→
Agent C: Validate
→
Agent D: Load

Each agent in the pipeline follows the single responsibility principle: it does one thing well. Agent A's output becomes Agent B's input. No branching, no parallel paths.

# Pipeline for document processing
raw_text = agent_extract(pdf_document)      # Extract text from PDF
structured = agent_parse(raw_text)          # Parse into structured data
validated = agent_validate(structured)      # Validate against schema
enriched = agent_enrich(validated)          # Add metadata, lookups
final = agent_format(enriched)             # Format for output

When to Use Pipelines

  • Each step has a clear, single responsibility
  • Output of one step is the input of the next
  • The order is fixed and deterministic
  • Each agent specializes in one transformation

When NOT to Use Pipelines

  • When steps can run in parallel (use fan-out instead)
  • When you need conditional branching (use routing instead)
  • When the number of steps isn't known in advance (use adaptive decomposition)
Pipeline vs. single agent with multiple tools. A common exam question: should you use a pipeline of 4 specialized agents, or a single agent with 4 tools? The answer depends on context window pressure. If each step produces large intermediate output (e.g., extracting text from a 100-page document), a pipeline is better because each agent has a fresh context window. If the steps are lightweight (a few tool calls each), a single agent is simpler and avoids inter-agent overhead.

Rule of thumb: Use a pipeline when intermediate results are large or when each step needs a specialized system prompt. Use a single agent when steps are lightweight and share context.

Pattern 3: Map-Reduce

Map-reduce is fan-out/fan-in formalized. You already learned it in Chapter 3, but here we focus on the reduce step — the part that trips people up.

The Map Phase (Fan-Out)

Distribute the same type of task across multiple data points. Each subagent handles one chunk independently.

# Map: parallel analysis of 10 customer reviews
for review in reviews:
    spawn_subagent(
        prompt=f"Analyze this review for sentiment and key themes: {review}",
        tools=["SentimentAnalyzer"]
    )

The Reduce Phase (Fan-In)

Collect all results and combine them into a single output. This is where the orchestrator (or a dedicated reducer agent) does the synthesis.

# Reduce: aggregate all analysis results
all_results = collect_subagent_results()
summary = synthesize_agent(
    prompt=f"""Combine these {len(all_results)} review analyses into
    a single report: overall sentiment trend, top 5 themes,
    notable outliers. Results: {all_results}"""
)
The reduce step bottleneck. If you have 20 subagents each returning 2,000 tokens of results, the reducer needs to fit 40,000 tokens in its context window just for the input. This is a real constraint. Solutions: (1) ask each subagent to return a concise summary instead of raw results, (2) do hierarchical reduce (reduce groups of 5, then reduce the 4 partial results), or (3) have each subagent write to a shared data store and have the reducer read selectively.

Exam Scenario: Multi-Agent Research

The Multi-Agent Research System scenario is map-reduce in action:

  1. Map: Orchestrator decomposes "Research topic X" into 4 sub-questions, spawns 4 researcher agents in parallel
  2. Each researcher: Searches its assigned sub-question, returns a summary of findings
  3. Reduce: Orchestrator (or a dedicated writer agent) synthesizes all 4 summaries into a cohesive report
In the Multi-Agent Research scenario, the exam tests whether you understand that the reduce step is not just concatenation. The reducer must synthesize, resolve contradictions between sources, identify gaps, and produce a coherent narrative. A distractor will suggest "combine all results into one document" — that's concatenation, not synthesis.

Pattern 4: Routing

A routing pattern directs tasks to specialized agents based on classification. Think of a phone tree: "Press 1 for billing, 2 for technical support, 3 for shipping."

# Routing pattern for customer support
def route_request(customer_message):
    # Step 1: Classify the request
    category = classifier_agent(customer_message)
    # Returns: "billing", "technical", or "shipping"

    # Step 2: Route to specialist
    if category == "billing":
        return billing_agent(customer_message, account_info)
    elif category == "technical":
        return technical_agent(customer_message, product_info)
    elif category == "shipping":
        return shipping_agent(customer_message, order_info)
    else:
        return escalate_to_human(customer_message)

Routing is the bread-and-butter of the Customer Support Resolution Agent scenario. The exam tests three aspects:

  1. Classification accuracy: How does the router decide which specialist to invoke? (Usually an LLM call with a constrained output schema)
  2. Context passing: What information does the specialist receive? (The customer message plus relevant data — not the entire conversation)
  3. Fallback handling: What happens when the classifier is unsure? (Escalate to human, don't guess)
Routing can be nested. The first router classifies by category (billing/technical/shipping). The technical router then sub-classifies (hardware/software/network). Each level narrows the specialist. This hierarchical routing is common in production — and on the exam.

The key decision: Should you use an LLM for classification (flexible but probabilistic) or deterministic rules (rigid but 100% reliable)? For simple routing (3–5 clear categories), an LLM classifier works well. For critical routing where misclassification has consequences, combine LLM classification with a programmatic hook that validates the routing decision.

Pattern 5: Escalation and Fallback

Every production agent system needs a "what if the agent can't handle it?" path. Escalation patterns define when and how to hand off to a human or a higher-level agent.

Valid Escalation Triggers

Trigger Example Action
Explicit customer request "I want to speak to a human" Immediate escalation
Policy gap No rule covers this situation Escalate with context summary
Capability exceeded Task requires external system access the agent lacks Escalate with attempted actions
Business threshold Refund amount exceeds $500 limit Escalate for human approval

Invalid Escalation Triggers

Sentiment-Based
"The customer seems angry, so escalate to a human."
Sentiment analysis is unreliable. An angry customer might have a simple issue the agent can solve. Don't escalate based on tone.
Confidence-Based
"The agent's confidence score is below 0.7, so escalate."
LLM confidence scores are poorly calibrated. A model might be "90% confident" in a wrong answer. Don't use self-reported confidence for escalation decisions.

The Escalation Handoff

When escalating, the agent should provide:

  • Summary of the conversation — what the customer asked for
  • Actions already taken — what the agent tried
  • Reason for escalation — why the agent can't resolve it
  • Relevant data — account info, order numbers, etc.

A human receiving an escalation with this context can pick up immediately instead of re-asking the customer everything.

The exam consistently marks sentiment-based and confidence-based escalation as wrong. Expect at least one question where these are distractors. The correct escalation triggers are always explicit, rule-based, or threshold-based. This is one of the most predictable patterns on the exam.

State Passing Between Agents

How do agents share information? This is a critical design decision with real tradeoffs:

Option 1: Through the Orchestrator

All state flows through the hub. Agent A returns results to the orchestrator, which passes relevant parts to Agent B. This is clean and observable, but the orchestrator becomes a bottleneck.

Option 2: Shared External State

Agents read and write to a shared database, file system, or message queue. This allows direct communication without bottlenecking the orchestrator, but introduces coordination complexity.

Option 3: Message Passing

Agents send structured messages directly to each other. Fast and flexible, but harder to debug because there's no central log.

Approach Pros Cons Best For
Through orchestrator Observable, controlled Bottleneck, token cost Most hub-and-spoke systems
Shared external state No bottleneck, persistent Coordination complexity Long-running workflows
Message passing Fast, flexible Hard to debug Peer-to-peer systems
For the exam, "through the orchestrator" is the default correct answer unless the scenario specifically describes a need for direct agent-to-agent communication. Hub-and-spoke systems pass state through the hub — that's the entire point of having a hub.

Anti-Patterns: What NOT to Build

The exam loves presenting broken architectures and asking you to identify the problem. Here are the big anti-patterns:

1. The God Orchestrator

An orchestrator that does everything: classifies, processes, validates, formats, and delegates. If the orchestrator is doing the work itself instead of delegating, it's not an orchestrator — it's a monolithic agent pretending to be a coordinator.

2. Circular Dependencies

Agent A delegates to Agent B, which delegates back to Agent A. This creates an infinite loop. In a DAG, there are no cycles by definition. If your workflow graph has a cycle, you have a design error.

3. Missing Error Propagation

A subagent fails, but the orchestrator doesn't detect it and proceeds as if the subtask succeeded. Always check subagent results for errors before aggregating.

4. Cascading Hallucinations

Agent A hallucinates a fact ("refunds over $100 require manager approval"). Agent B receives this as context and acts on it. Agent C cites B's output as evidence. The hallucination becomes "fact" through the pipeline. Prevent this with validation at each step and authoritative data sources.

5. Context Bleed

Sharing the full orchestrator context with subagents, causing them to act on irrelevant information or exceed their context windows. Always pass only task-specific context.

Cascading hallucinations is the sneakiest anti-pattern. On the exam, look for scenarios where one agent's output feeds directly into another without validation. The fix is always to validate against an authoritative source (a database, an API, a knowledge base) rather than trusting another agent's output at face value.

Choosing the Right Pattern: Decision Framework

When the exam presents a scenario, use this framework to pick the right orchestration pattern:

If the scenario has... Use this pattern
A coordinator managing specialists Hub-and-spoke
Sequential transformations on data Pipeline / chain
Same task applied to many items, then combined Map-reduce
Classification leading to different handlers Routing
Agent can't handle something, needs a human Escalation / fallback
Multiple independent queries combined into a report Fan-out / fan-in (map-reduce)
A complex system with many types of work Combination (routing + hub-and-spoke + escalation)

Most real-world systems and most exam scenarios use a combination. The Customer Support Agent, for example, uses routing (classify issue type) + hub-and-spoke (specialist delegates to tool-calling subagents) + escalation (hand off when stuck).

Scenario Deep Dive: Customer Support

Let's wire up the full orchestration for the Customer Support scenario, since it tests the most patterns at once:

# Complete Customer Support orchestration
#
#  User Message
#       │
#  ┌────▼────┐
#  │ Router  │  ← Classifies: billing / technical / shipping
#  └────┬────┘
#       │
#  ┌────▼──────────────────────────┐
#  │ Specialist Agent (e.g. billing)│
#  │   ├── Tool: LookupAccount     │
#  │   ├── Tool: SearchKB          │
#  │   └── Tool: ProcessRefund     │  ← Hook validates amount ≤ $500
#  └────┬──────────────────────────┘
#       │
#  ┌────▼────┐
#  │ Check   │  ← Can the specialist resolve it?
#  └────┬────┘
#    Yes│    No
#       │  ┌────▼─────┐
#       │  │ Escalate  │  ← Summary + context → human agent
#       │  └───────────┘
#  ┌────▼────┐
#  │ Respond │  ← Final answer to customer
#  └─────────┘

This combines routing (classify the issue), hub-and-spoke (specialist uses tools), hooks (validate refund amounts), and escalation (hand off on failure). That's why it's the exam's most comprehensive scenario.

Exercise: Choose the Pattern

For each scenario below, identify the primary orchestration pattern and justify your choice:

  1. A system that reviews 50 pull requests overnight and generates a summary report of all changes.
  2. A chatbot that answers questions about three different products, each with its own knowledge base.
  3. A document processing pipeline: OCR → entity extraction → classification → database insert.
  4. A research assistant that takes a broad question, finds relevant papers, and writes a literature review.
  5. A support agent that detects when a customer's request requires access to a system the agent doesn't have credentials for.

Answers: (1) Map-reduce: same task (review) applied to many items (PRs), then synthesized. (2) Routing: classify which product, then route to the right knowledge base agent. (3) Pipeline: fixed sequential transformations. (4) Hub-and-spoke with fan-out: decompose question, parallel research, synthesize. (5) Escalation: capability exceeded → hand off with context.

Quick Review

  • Hub-and-spoke: orchestrator decomposes, delegates, aggregates, and handles errors — needs "Task" in allowedTools
  • Pipeline: sequential agents, each with single responsibility — use when steps are ordered and context-heavy
  • Map-reduce: parallel processing of similar items + synthesis — watch for the reduce-step bottleneck
  • Routing: classify then dispatch to specialist — combine with hooks for critical routing decisions
  • Escalation triggers must be explicit (human request, policy gap, capability exceeded, threshold) — never sentiment or confidence-based
