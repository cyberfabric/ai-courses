Claude Code Configuration

Memory Map

The Big Idea

Claude Code's configuration system is persistent context engineering — you program the environment around the agent (CLAUDE.md, settings, hooks, permissions, skills) so every session starts informed, constrained, and team-aligned.

🧩 Core Concepts

📄
CLAUDE.md
Persistent context loaded into every session. Three levels merge together: user → project → directory.
⚙️
Settings Layers
Four scopes with strict precedence: Managed > Local > Project > User. Managed can't be overridden.
🔒
Permissions
Allow/deny rules with glob patterns on tool names. Control what Claude can read, write, and execute.
🪝
Hooks
Deterministic code that runs at lifecycle events. PreToolUse and PostToolUse are the most exam-tested.
🎯
Skills
SKILL.md with YAML frontmatter at .claude/skills/<name>/. Claude auto-invokes by description, or you call /skill-name. Legacy .claude/commands/ still works.
🤖
Subagents
Named agent configs in .claude/agents/ with system prompts, tool restrictions, and model settings.
📦
Plugins
Bundle skills + MCP + hooks + commands + subagents as one installable. /plugin install <name>@claude-plugins-official. ~101 in the official marketplace.
🛡️
Trust Spectrum
Full Manual → Default → Allowlisted → Auto Mode. Balances safety vs. speed based on context.
🏗️
Domain 3
20% of the exam (~12 questions). Chapters 7 + 8 together cover all of Domain 3.

🪝 Hook Lifecycle Events

  1. SessionStart

    Session begins — set up environment, load context.

  2. UserPromptSubmit

    User sends a message — add context, enforce policies.

  3. PreToolUse ⭐

    Before every tool call — validate inputs, block dangerous actions. Can return allow/deny decisions.

  4. PostToolUse ⭐

    After tool execution — validate outputs, log results, inject context.

  5. PreCompact

    Before context compaction — preserve critical information from being summarized away.

  6. Stop

    Agent stops — cleanup, logging, notifications.

📖 Key Terms

Persistent Context Engineering
Using CLAUDE.md to give Claude project knowledge that persists across sessions.
Managed Settings
Server-deployed settings with highest priority — cannot be overridden by any user or project setting.
PreToolUse Hook
Fires before every tool call; can return allow/deny decisions that override the permission system.
Trust Spectrum
The range from Full Manual (safest, slowest) to Auto Mode (fastest, most trust required).
Glob Patterns
Wildcard patterns in permission rules — e.g., Bash(npm run *) allows any npm script.
Project Skills
SKILL.md files at .claude/skills/<name>/, committed to git. YAML frontmatter (name, description) lets Claude auto-invoke them. Invoked explicitly with /skill-name.
Plugins
Distribution bundles for skills + MCP servers + hooks + commands + subagents. Browse with /plugin; install with /plugin install <name>@claude-plugins-official.
Subagent Definition
A markdown file with frontmatter specifying system prompt, allowed tools, and model for a named agent.
Context Budget
Every token in CLAUDE.md is loaded every session — keep it concise to avoid wasting context window space.

🚨 Exam Traps to Remember

Quick Quiz

Personal CLAUDE.md says "use tabs." Project CLAUDE.md says "use spaces." Which wins?

Project-level wins. More specific scopes override less specific ones. User-level has the lowest priority in the CLAUDE.md hierarchy.

How do you ensure Claude NEVER writes to production config files?

Use a PreToolUse hook on the Write tool that checks file paths and returns deny for production directories. Not CLAUDE.md (that's guidance, not enforcement). Not system prompt instructions (probabilistic, not deterministic).

Can a user setting override a managed setting?

No. Managed settings have the highest priority and cannot be overridden by local, project, or user settings. This is by design for security compliance.

What's the difference between a skill and a subagent?

A skill is a prompt template ("do this task"). A subagent is a full agent configuration with system prompt, tool restrictions, and model ("be this kind of agent").

Why not put security rules in CLAUDE.md?

CLAUDE.md is part of the system prompt — it's guidance, not enforcement. Claude usually follows it but it's not guaranteed. Security-critical rules need permissions (deny rules) or hooks for deterministic enforcement.

Key Takeaways

  • CLAUDE.md = persistent context engineering. Three levels merge: user → project → directory. More specific wins.
  • Settings precedence: Managed > Local > Project > User. Managed settings are unbreakable.
  • Hooks are deterministic; prompts are probabilistic. For security-critical rules, always use hooks or permissions, never CLAUDE.md alone.
  • PreToolUse is the most powerful hook — it can override the permission system with allow/deny decisions.
  • Skills = prompt templates ("do this"). Subagents = agent configurations ("be this"). Plugins = bundles of skills + MCP + hooks. The exam tests skill-vs-subagent.
  • The trust spectrum (Manual → Default → Allowlisted → Auto) balances safety vs. speed. Context determines the right level.
