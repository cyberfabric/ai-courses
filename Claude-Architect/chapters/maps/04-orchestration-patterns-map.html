Orchestration Patterns

Memory Map
💡

The Big Idea

Five orchestration patterns cover every exam scenario: hub-and-spoke, pipeline, map-reduce, routing, and escalation. Real systems combine them — learn to match the pattern to the scenario's characteristics.

🏗️ The Five Patterns

🎯
Hub-and-Spoke
Central orchestrator decomposes, delegates, aggregates, handles errors. Needs "Task" in allowedTools.
🔗
Pipeline / Chain
Sequential agents — each transforms data and passes it forward. Assembly line. Single responsibility per step.
🗺️
Map-Reduce
Same task on many items in parallel (map), then combine results into one output (reduce). Fan-out/fan-in formalized.
🔀
Routing
Classify input, dispatch to the right specialist. Phone tree: "Press 1 for billing, 2 for tech support."
🆘
Escalation / Fallback
When agent can't handle it — hand off to human with conversation summary, actions taken, and reason.

🗺️ Map-Reduce: The Reduce Trap

📤
Map (Fan-Out)
Distribute same task across N items. N parallel subagents, each handles one chunk independently.
📥
Reduce (Fan-In)
Collect all results → synthesize (not concatenate!). Resolve contradictions, identify gaps, create narrative.
⚠️
Reduce Bottleneck
20 agents × 2K tokens = 40K tokens in reducer input. Fix: concise summaries, hierarchical reduce, or shared data store.

🔀 Routing: Customer Support Pattern

  1. 📨 Receive Message

    Customer sends their request to the system.

  2. 🏷️ Classify (Router)

    LLM or rule-based classifier: billing / technical / shipping. Fallback: escalate if unsure.

  3. 👤 Specialist Agent

    Focused agent with domain tools (LookupAccount, SearchKB, ProcessRefund). Hook validates refund ≤ $500.

  4. ✅ Resolve or 🆘 Escalate

    Specialist resolves → respond. Can't resolve → escalate with summary + actions taken + reason.

🚫 Anti-Patterns the Exam Tests

👑
God Orchestrator
Does everything itself instead of delegating. It's a monolith pretending to be a coordinator.
🔄
Circular Dependencies
A delegates to B, B delegates back to A → infinite loop. DAGs have no cycles by definition.
🙈
Missing Error Propagation
Subagent fails but orchestrator proceeds as if it succeeded. Always check results before aggregating.
🦠
Cascading Hallucinations
Agent A hallucinates → B acts on it → C cites B. Fix: validate against authoritative sources at each step.
💧
Context Bleed
Sharing full orchestrator context with subagents. Pass only task-specific context, never the whole conversation.

🧪 Quick Quiz

What 4 things must a hub-and-spoke orchestrator do?

Decompose, delegate, aggregate, and handle errors. Missing any one is an incomplete orchestrator.

50 PRs need overnight review + summary report. Which pattern?

Map-reduce. Same task (review) on many items (PRs) in parallel, then synthesize into one report.

The reduce step — is it concatenation or synthesis?

Synthesis. The reducer must resolve contradictions, identify gaps, and produce a coherent narrative — not just paste results together.

Pipeline vs. single agent with tools — what decides?

Context window pressure. Large intermediate results → pipeline (fresh context per step). Lightweight steps → single agent (simpler, less overhead).

Agent A hallucinates a policy, Agent B acts on it. What anti-pattern is this?

Cascading hallucinations. Fix: validate each agent's output against authoritative sources before passing it forward.

🔑 Key Takeaways

  • 5 patterns: hub-and-spoke, pipeline, map-reduce, routing, escalation — most systems combine several.
  • Orchestrator has 4 jobs: decompose, delegate, aggregate, handle errors. Needs "Task" in allowedTools.
  • Pipeline for sequential transforms with heavy intermediate data. Single agent for lightweight steps.
  • Map-reduce: synthesis, not concatenation. Watch for the reduce-step context window bottleneck.
  • Anti-patterns: God orchestrator, circular deps, missing error propagation, cascading hallucinations, context bleed.
