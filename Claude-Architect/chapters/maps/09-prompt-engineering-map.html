Chapter 9

Prompt Engineering for Architects

Memory Map
💡

The Big Idea

Production prompts aren't conversations — they're infrastructure code. Version them, test them against evals, structure them for caching. A system prompt that runs thousands of times must be clear, specific, and robust to edge cases.

🧩 Core Techniques

🎯
Be Clear & Direct
Show your prompt to a colleague — if they'd be confused, Claude will be too. Add why to rules so Claude can generalize.
📋
Few-Shot Prompting
3–5 examples in <example> tags. Cover diversity of inputs. Label diversity matters more than individual label correctness.
🔗
Chain-of-Thought
Step-by-step reasoning for complex tasks. Skip it when adaptive thinking is enabled — adjust the effort parameter instead.
🎭
Role Prompting
A single role sentence changes perspective. "Senior security auditor" catches different things than "junior developer."
🏷️
XML Tags
Claude's native structuring mechanism. Separates instructions from data. Key defense against prompt injection.
📐
Explicit Criteria
Define "good" with rubrics (1–5 scales). Makes output consistent and auditable across thousands of runs.
🌡️
Temperature Control
Default API temp is 1.0. For agents: use 0.0–0.3. Predictable beats creative in production.
📝
Prompt Templates
Use parameterized templates with variables. Static parts get cached; only variable parts change per request.

🔄 The Multi-Pass Review Pattern

Pass 1: Generate

Agent produces the output — code review, analysis, or response.

Pass 2: Evaluate

Second API call reviews output against explicit criteria.

Pass 3: Refine

Based on evaluation, agent improves the output. Each pass is logged independently.

⚖️ When to Use Chain-of-Thought

ScenarioUse Explicit CoT?Why
Adaptive thinking disabled ✅ Yes Claude needs prompting to reason step-by-step
Adaptive thinking enabled ❌ No Already does internal CoT — explicit is redundant, wastes tokens
Simple factual lookup ❌ No No reasoning needed — CoT adds latency for no gain
Multi-step debugging ✅ Yes (if thinking off) Complex reasoning benefits from structured steps

📖 Key Terms

Few-Shot Prompting
Providing input/output examples to establish the desired pattern. 3–5 diverse examples recommended.
Chain-of-Thought (CoT)
Encouraging step-by-step reasoning before the final answer. Redundant when adaptive thinking is on.
System Prompt
Persistent instructions that run with every request. The architectural foundation of an agent.
Role Prompting
Setting behavioral perspective ("You are a senior security auditor") to focus the model's approach.
Prompt Template
Parameterized prompt with variables ({language}, {diff}). Static parts cached, variables change per request.
Explicit Criteria
Defined rubrics (1–5 scales) that make agent output consistent and auditable.
Prompt Injection
Attack where untrusted input tricks the model. XML tags wrapping user data are the primary defense.
Adaptive Thinking
Claude's built-in internal reasoning. Controlled via effort parameter (low/medium/high). On Opus 4.7 it's the only thinking mode supported — manual budget_tokens returns a 400 error.
Context Assembly
The process of gathering the right context for a prompt. The #1 real-world failure mode — not bad prompts, but wrong context.

🧪 Quick Quiz

What's wrong with adding "CRITICAL: YOU MUST" to prompts for Claude 4.5+?

Aggressive language hurts newer models. Claude 4.5+ responds better to calm, direct instructions. Emphatic language was useful for older models that under-triggered on tools, but now causes over-triggering.

Should you add explicit chain-of-thought when adaptive thinking is enabled?

No — adaptive thinking already handles internal CoT. Explicit CoT on top is redundant and wastes tokens. Instead, adjust the effort parameter (low/medium/high).

What are the 5 sections of the System Prompt Contract?

Role, Success Criteria, Constraints, Output Format, and Uncertainty Handling.

How do XML tags defend against prompt injection?

By wrapping untrusted user input in data tags (e.g., <input>), Claude treats the content as data rather than instructions. Even if the user types "Ignore all previous instructions," it stays inside the tags.

What does Anthropic say is the real failure mode in production?

"The real failure mode isn't a bad prompt — it's bad context assembly." Most agent failures happen because wrong context was loaded, tool results were truncated, or relevant info was compacted away.

📌 Key Takeaways

  • Be clear and direct — add why to rules. Avoid aggressive language on Claude 4.5+ models.
  • Few-shot: 3–5 diverse examples in <example> tags. Format diversity > label perfection.
  • Skip explicit CoT when adaptive thinking is on — use the effort parameter instead.
  • System prompts = infrastructure: version-controlled, structured for caching (static first), tested against evals.
  • XML tags are the primary prompt injection defense — wrap untrusted input as data.
  • The real production failure mode is bad context assembly, not bad prompts.
  • For agents: set temperature to 0.0–0.3 explicitly (API default is 1.0).
  • Use the multi-pass pattern (generate → evaluate → refine) for critical outputs.
