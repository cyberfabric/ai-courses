Chapter 14

Agent Evaluations & Observability

Memory Map
💡

The Big Idea

An architect doesn't hope the agent works — they prove it. Evals catch regressions before deploy. Observability catches drift between deploys. Together they're how you ship a system that keeps working when models, prompts, and tools change.

🔬 The Three Eval Types

1️⃣
Single-Turn
Prompt → response → grade. Simplest. Good for classifiers, extractors, summarizers.
2️⃣
Multi-Turn
Multiple turns + tool calls. Tests context retention and tool selection.
3️⃣
Agent
State changes, mistakes compound. Closest to production. Hardest to grade.

⚖️ Picking the Right Grader

⚙️
Deterministic
Schema validators, regex, exact match, run-the-tests. Fast, cheap, reproducible. Use whenever possible.
🤖
LLM-as-Judge
For subjective dimensions (tone, empathy, completeness). Needs calibrated rubric + "Unknown" out-clause.
👤
Human
Gold standard. Slow, expensive. Use to calibrate LLM judges and spot-check production.

Rule: pick the simplest grader that does the job.

🎲 pass@k vs pass^k

MetricWhat it asksUse when…
pass@k"At least one success in k tries?"One good answer is enough; users can re-roll
pass^k"Every one of k tries succeeds?"Consistency is the product; no human re-roll

At k=1 they're identical. By k=10 they often diverge to ~100% vs ~0%.

📡 Production Observability

💰
Tokens / Session
Median + p95. Spikes mean prompts grew or the agent looped.
🔧
Tool-Success Rate
Per tool. Quiet 99% → 92% drops are silent regressions.
🆘
Escalation Rate
% of sessions handed to a human. Spikes signal new failure modes.
✅
First-Contact Resolution
Customer-support twin of escalation rate. Did the agent solve it solo?
💵
Cost / Resolution
The dollar lens. Tokens + tool execution ÷ resolutions.
⏱️
Latency
TTFT, TTLT, end-to-end. SLOs live here.

OpenTelemetry is the 2026 vendor-neutral standard. Wire it once.

🚨 Anti-Patterns

  1. Deploy-time evals only

    Drift happens between deploys, not at them. Continuous sampling is non-negotiable.

  2. Happy-path-only sampling

    The 5% that fail teach you more than the 95% that pass.

  3. Uncalibrated LLM judges

    Without human spot-checks, the score is fiction.

  4. Eval-suite rot

    Eval suites need owners, weekly transcript review, and monthly dataset refresh.

📖 Key Terms

Capability eval
Tests whether the agent can do something yet. Starts low; graduates to regression once saturated.
Regression eval
Sits near 100%. Any dip blocks deployment. The deploy gate.
LLM-as-judge
Using a model to grade subjective output dimensions. Always pair with calibration + "Unknown" out-clause.
pass@k
Probability of at least one success in k attempts. For products where one good answer is enough.
pass^k
Probability that all k attempts succeed. For products where consistency is the deliverable.
Three-agent harness
Planner / generator / evaluator separated into roles. Apr 2026 pattern that makes long autonomous workflows debuggable.
OpenTelemetry (OTel)
Vendor-neutral observability standard. The 2026 baseline for agent telemetry — portable across visualizers.
Eval-driven development
Test-driven development applied to agents: write evals first, iterate until they pass, graduate them into the gate.

🧪 Quick Quiz

How many tasks should you start an eval suite with?

20–50 simple tasks drawn from real failures. Early evals have large effect sizes — small samples are enough.

Eval shows 100% pass rate but production complaints are spiking. Diagnosis?

The dataset is unbalanced or missing the failure mode. Capture the failing transcripts, codify them as new eval tasks, and re-evaluate.

An automated PR-review agent posts comments straight to GitHub. pass@k or pass^k?

pass^k. No human re-roll. Every comment ships, so consistency across attempts is the product.

How do you evaluate "did the reply express empathy?"

LLM-as-judge with a calibrated rubric and an "Unknown" out-clause. Not regex (won't catch paraphrase). Not skip-it (the architect's job is to evaluate the hard things).

What graduates a capability eval into a regression eval?

Saturation — when the pass rate sits near 100% reliably. At that point, it stops driving improvement and starts catching regressions.

📌 Key Takeaways

  • Three eval types: single-turn → multi-turn → agent. Each adds complexity and realism.
  • Capability evals start low and graduate to regression evals once saturated.
  • Start with 20–50 tasks from real failures. Unambiguous, balanced, isolated.
  • Pick the simplest grader: deterministic > LLM-as-judge > human (in that order of preference).
  • LLM judges need calibration against humans + an "Unknown" out-clause.
  • pass@k for products with re-rolls; pass^k for products that ship every output.
  • Production observability covers tokens, tool success, escalation, FCR, cost, latency — via OpenTelemetry.
  • Design agents to be evaluable: structured outputs, hook checkpoints, structured errors.
  • Avoid: deploy-time-only evals, happy-path-only sampling, uncalibrated judges, eval-suite rot.
  • The eval suite that catches regressions is also the architect's portfolio artifact — proof of reliability discipline.
