Designing Agent Tools

Memory Map
💡

The Big Idea

Claude picks tools based on their descriptions, not their names. A well-described tool gets used correctly; a vague one gets misused or ignored. Descriptions are the #1 lever for tool selection accuracy.

🔧 Anatomy of a Tool Definition

🏷️
Name
Descriptive and distinct. search_knowledge_base not search or skb.
📝
Description (Most Important!)
3–4 sentences minimum. What it does, when to use, when NOT to use, what it returns, caveats.
📐
Input Schema
JSON Schema: required vs optional params, enums for fixed values, descriptions on each parameter, defaults.

📝 The Perfect Tool Description

  1. What it does

    "Search the customer support knowledge base for articles matching a query."

  2. What it returns

    "Returns top 5 articles with titles, URLs, and relevance scores."

  3. When to use it

    "Use when customer asks a question that might be answered by existing docs."

  4. When NOT to use it

    "Do NOT use for account-specific queries — use lookup_account instead."

🎯 Cross-reference similar tools: "For X, use tool Y instead." Eliminates ambiguity.

🧠 The Reasoning Boundary

Task Type 🧠 Agent Decides (Claude) 🔧 Tool Executes (Your Code)
Natural language ✅ Understanding, interpreting —
Which tool to call ✅ Selection + parameters —
Math / calculations ❌ Never! LLMs are unreliable ✅ Always deterministic
Business rules ❌ Can hallucinate policies ✅ Enforced by code
Database queries — ✅ Reliable, structured
Synthesizing results ✅ Narrative + interpretation —

📏 Tool Granularity Sweet Spot

❌
Too Many (20+)
Selection confusion. Claude wastes reasoning tokens deciding. "Tool overload" — the exam's term for this.
❌
Too Few (2–3)
Parameters get complex. Descriptions become essays. Hard for Claude to use correctly.
✅
Just Right (5–10)
Clear purpose per tool. Manageable selection. Like a well-designed REST API. Anthropic's recommendation.

🚨 Structured Error Responses

is_error: true
Marks the result as an error — tells Claude not to treat this as a success.
errorCategory
Classifies the error (NOT_FOUND, RATE_LIMIT, AUTH_FAILED, etc.). Helps Claude choose recovery strategy.
isRetryable
true = try again (transient). false = don't retry (permanent). Prevents infinite retry loops.
suggestion
What to try instead. "Verify the customer ID" or "Wait 5 seconds and retry."

🧪 Quick Quiz

Agent frequently picks the wrong tool. Most likely fix?

Improve tool descriptions (add when-to-use, when-NOT-to-use, cross-references) or reduce the number of tools if there are 20+.

Agent keeps retrying a permanently failed tool call. Why?

The error response doesn't include isRetryable: false, so the agent doesn't know the error is permanent.

Customer ordered 3 items at $29.99. Should the agent calculate the total?

No! Never let the agent do math. Use a tool like calculate_order_total(order_id) — deterministic every time.

When should you use strict: true on a tool?

When parameter correctness is critical — especially for data extraction and structured output. Guarantees the tool call conforms to your JSON Schema exactly.

How many tools per agent is ideal?

5–10 well-described tools. 20+ causes tool overload/selection confusion. 2–3 means overly complex parameters.

🔑 Key Takeaways

  • Descriptions are the #1 lever — 3–4 sentences, include when-to-use, when-NOT-to-use, cross-reference similar tools.
  • 5–10 tools per agent. Too many = selection confusion (tool overload). Too few = parameter complexity.
  • Agent decides what to do; tool does the deterministic work. Never let the agent do math or enforce business rules.
  • Structured errors: is_error, errorCategory, isRetryable, suggestion. Vague errors cause blind retries.
  • strict: true guarantees schema conformance. Use it for critical tools, especially data extraction.
