Chapter 2

Multi-Agent Systems

Memory Map
💡

The Big Idea

Single agents hit walls on context, specialization, and parallelism. Multi-agent systems solve all three by giving each agent a focused role, isolated context, and the ability to work in parallel.

🧱 Why Single Agents Break

📦
Wall 1: Context Limits
Tool calls and results eat the context window. A complex task can burn 100K+ tokens on intermediate results alone.
🎭
Wall 2: Specialization
One agent = one system prompt trying to be researcher + coder + reviewer. More roles piled on = worse at each. Chef-waiter-dishwasher problem.
⏳
Wall 3: Parallelism
Single agent processes sequentially. Multiple agents search, analyze, and check APIs simultaneously.

🔄 The Agent Loop

📥 Receive Task
→
🧠 Reason / Plan
→
🔧 Call Tool
→
👁️ Observe Result
→
🔀 Continue or Stop
🔧
stop_reason: "tool_use"
Agent wants to call a tool. Execute it, append result, send back to Claude. Loop continues.
🏁
stop_reason: "end_turn"
Agent is done. Final response ready. Break the loop.
⚠️
Exam Trap!
NEVER parse text ("I'm done") to control the loop. Always use stop_reason. The exam specifically tests this.

🏗️ Two Topologies: Orchestra vs. Jazz

Aspect 🎻 Hub-and-Spoke (Orchestra) 🎷 Peer-to-Peer (Jazz)
Leadership Central conductor/orchestrator Self-coordinating equals
Communication All through the hub Direct between agents
Predictability High — follows a plan Lower — improvised
Failure mode Hub = single point of failure Resilient but chaotic
Debugging Easy — centralized logs Hard — distributed state
Best for Structured, repeatable tasks Creative, adaptive tasks
Exam default? ✅ Almost always the right answer Only if explicitly required

🕸️ Hub-and-Spoke Architecture

🔍 Researcher
💻 Coder
🎯 Orchestrator
✅ Reviewer
📊 Extractor

🔒 Context Isolation (Exam Favorite!)

✅
What Subagents GET
Their own system prompt + the specific task description from the orchestrator. That's it.
❌
What Subagents DON'T Get
No orchestrator history. No other subagent results. No inherited conversation. Clean context by design.

👥 Common Agent Roles

🎯 Orchestrator
Plans, delegates, aggregates. Uses the Task tool to spawn subagents. The "hub" in hub-and-spoke.
🔍 Researcher
Searches and retrieves information. Tools: WebSearch, DatabaseQuery. Focused retrieval only.
💻 Coder
Writes, edits, and refactors code. Tools: FileWrite, FileEdit, Bash. No search tools needed.
✅ Reviewer
Validates output and checks quality. Tools: FileRead, Lint, Test. Read-only access.
📊 Extractor
Pulls structured data from unstructured text. Uses tool_use for schema enforcement.

⚖️ Hooks vs. Prompts: The Enforcement Rule

Mechanism Reliability Use For
🔩 Hooks (PreToolUse / PostToolUse) 100% deterministic — runs as code Refund limits, compliance, security
💬 Prompts (system prompt instructions) Probabilistic — model may ignore Tone, style, soft guidelines

🚫 Anti-Patterns the Exam Loves

  1. Over-Decomposition

    Breaking tasks too small. Each subagent has overhead — don't decompose "write a function" into individual lines.

  2. Chatty Agents

    Subagent returns 10K tokens when orchestrator needs 3 sentences. Wasted context budget.

  3. Unclear Responsibilities

    Overlapping roles = duplicated work or conflicting results. Each agent needs non-overlapping scope.

  4. Missing Error Handling

    No plan for subagent failure. Always design: retry, skip, or escalate.

  5. Arbitrary Iteration Caps

    Using max_iterations as primary stop mechanism. Use stop_reason — caps are safety nets only.

🚨 Valid Escalation Triggers

🙋
Human Request
User explicitly asks for a human agent.
📋
Policy Gap
No rule covers this situation — needs human judgment.
🚧
Capability Exceeded
Task is beyond what the agent can do.
📈
Business Threshold
Amount or risk exceeds automated limits.

🧪 Quick Quiz

In hub-and-spoke, what context does a subagent receive?

Only its own system prompt + the specific task description from the orchestrator. NOT the orchestrator's history or other subagents' results.

How should you control the agent loop — text parsing or stop_reason?

stop_reason, always. "tool_use" = continue, "end_turn" = stop. Never parse "I'm done" from text output.

A refund limit of $500 — enforce with hook or prompt?

Hook (PostToolUse). Critical business rules = hooks (deterministic). Style/tone = prompts (probabilistic).

The agent "seems uncertain" — is that a valid escalation trigger?

No. Confidence scores and sentiment analysis are unreliable triggers. Valid triggers: human request, policy gap, capability exceeded, business threshold.

What are the 3 walls that single agents hit?

1) Context window limits, 2) Specialization (role overload), 3) Parallelism (sequential bottleneck).

🔑 Key Takeaways

  • Single agents break on context limits, specialization, and parallelism — multi-agent solves all three.
  • The agent loop: check stop_reason — "tool_use" continues, "end_turn" stops. Never parse text.
  • Hub-and-spoke is the exam default — orchestrator delegates, subagents get isolated context only.
  • Hooks for hard rules (refund limits, compliance), prompts for soft guidelines (tone, style).
  • Least privilege for agents: each gets only the tools it needs — fewer tools = better focus.
