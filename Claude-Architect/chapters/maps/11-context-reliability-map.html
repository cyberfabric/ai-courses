The Big Idea

Production agents fail — APIs time out, context overflows, tools break. The exam tests whether you design for graceful degradation: continue with reduced capability rather than crash. The correct answer is almost never "stop and tell the user to try later."

🏗️ The Reliability Stack

  1. Layer 1: Retry with Backoff

    For transient errors (timeouts, rate limits). 3 retries, exponential backoff: 1s → 2s → 4s.

  2. Layer 2: Circuit Breaker

    For persistent failures. Stop retrying, return fallback. States: Closed → Open → Half-Open.

  3. Layer 3: Graceful Degradation

    When a capability is unavailable, provide reduced functionality rather than failing completely.

  4. Layer 4: Escalation

    When all else fails, hand off to a human with full context — never a "cold handoff."

Anti-Pattern: The Cold Handoff

Never transfer with just "Customer needs help." Always include: summary, actions taken, and recommended action. The exam specifically tests handoff quality.

🛡️ Graceful Degradation Examples

📚

💥

Try alternative tool or ask the user directly. Don't stop processing silently.

📖 Key Terms

Context Window Arithmetic
Calculating total token usage: system prompt + history + tools + input. Must fit the model's window.
Lost in the Middle
Models recall information better at the beginning and end of context. Middle content gets weaker attention.
Compaction
Compressing conversation context when it grows too large. Four strategies: summarize, truncate, sliding window, prune.
Exponential Backoff
Retry strategy where wait time doubles each attempt (1s → 2s → 4s). For transient errors only.
Circuit Breaker
Pattern that stops retrying after N consecutive failures. States: Closed → Open → Half-Open.
Graceful Degradation
Continuing with reduced functionality when a component fails, rather than crashing entirely.
Cold Handoff
Anti-pattern: escalating without context. Always include summary, actions taken, and recommendation.
PreCompact Hook
Claude Code hook that lets you specify what to preserve during automatic context compaction.
Model Fallback
Falling back to a cheaper/faster model (Opus → Sonnet) when rate limits or cost thresholds are hit.

🧪 Quick Quiz

An agent has 200K context, 3K system prompt, 5 tools (~2.5K), and a 120-turn conversation at 2K/turn. What happens?

3K + 2.5K + 240K + 346 ≈ 246K tokens. Exceeds the 200K window. The agent needs compaction before it can continue.

A customer mentions their order number in turn 3. By turn 40, the agent forgets it. Why?

"Lost in the middle" — the order number is buried in the middle of the context where attention is weakest. Fix: store it in the system prompt dynamically, or resurface it via summarization.

The knowledge base is down. What should the agent do?

Graceful degradation: answer from available context, mark the response as "unverified," and offer to follow up when the KB is available. Never just say "try later."

What's wrong with escalating to a human agent with "Customer needs help"?

That's a "cold handoff" — the human has no context and must re-ask everything. Always include: conversation summary, actions already taken, and recommended next action.

Which compaction strategy for a 50-turn customer support conversation?

Summarization — it preserves key information (customer details, issue history, decisions made) while compressing the token count. Truncation would lose critical early context.

📌 Key Takeaways

  • Context arithmetic: system prompt + history + tools + input must fit the window. Long conversations are the biggest consumer.
  • "Lost in the middle": place critical info at the edges (system prompt or recent turns), not buried in old history.
  • 4 compaction strategies: summarization (preserves info), truncation (fast, lossy), sliding window (recent only), selective pruning (remove low-value).
  • Reliability stack: retry with backoff → circuit breaker → graceful degradation → escalation to human.
  • Escalation handoffs must include summary + actions taken + recommendation. Never do a cold handoff.
  • Graceful degradation = continue with reduced capability. The exam answer is almost never "stop and tell the user to try later."
  • Monitor at 3 levels: agent metrics, system metrics, business metrics.
