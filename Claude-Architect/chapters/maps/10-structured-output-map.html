Structured Output & Validation

Memory Map

The Big Idea

Constrained decoding makes structured output deterministic, not probabilistic. Your JSON Schema compiles into a grammar — the model physically cannot produce tokens that violate it. No more JSON.parse() errors. Ever.

🧩 Core Concepts

📋
JSON Outputs
output_config.format — controls the format of Claude's response. Provide a JSON Schema, get guaranteed-valid JSON back.
🔧
Strict Tool Use
strict: true on tools — guarantees tool call inputs match the schema. For extraction pipelines where Claude calls a processing tool.
🔒
Constrained Decoding
Schema compiled into a grammar at generation time. Always valid JSON, type-safe fields, required fields present, no extras.
🔄
Validation Loops
Generate → Validate → Retry with feedback. Handles business logic beyond schema (email format, math checks). Max 3 retries.
📊
Confidence Scores
Include "confidence": "high|medium|low" per field. Low-confidence fields get flagged for human review.
📦
Batch API
50% cost savings, 24-hour processing window. Use for high-volume extraction (thousands of docs), not real-time agents.

🏗️ Escalation Tiers

  1. Tier 1: Constrained Decoding

    Fast, guaranteed structure. Handles 80–90% of documents automatically.

  2. Tier 2: Validation Loop

    For docs that pass schema but fail business rules. Retry with error feedback, 2–3 attempts.

  3. Tier 3: Human Review

    For docs that fail after max retries. Flag specific fields that need human judgment.

🧪 Quick Quiz

An agent extracts invoice data and returns it as JSON to the user. Which API feature?

output_config.format with a JSON schema (JSON Outputs). This controls the response format. Strict tool use controls tool input format — different thing.

Valid JSON is returned but the email field contains "not-an-email." What's the fix?

Validation loop with error feedback. JSON Schema can't validate email format well — you need business-level validation in code, with errors fed back to Claude for retry.

Your schema requires "phone_number" but some invoices don't have one. What happens?

Claude is forced to hallucinate a phone number to satisfy the schema. Fix: use "type": ["string", "null"] so Claude can return null for absent fields.

1,000 invoices need overnight processing. What API should you use?

The Message Batches API — 50% cost savings, 24-hour processing window. Perfect for latency-tolerant, high-volume extraction.

What's the difference between constrained decoding and validation loops?

Constrained decoding guarantees structure (valid JSON, correct types) at generation time. Validation loops check semantics (business rules, cross-field logic) after generation. Best practice: use both — constrained decoding as the first layer, validation loops on top.

📌 Key Takeaways

  • Two approaches: output_config.format for structured responses, strict: true for structured tool inputs.
  • Constrained decoding = deterministic schema compliance. No JSON.parse() errors. No retries for structure.
  • Validation loops handle business logic: generate → validate → retry with error feedback (max 3 attempts).
  • Use nullable types for optional fields — required fields on absent data forces hallucination.
  • Extraction pipeline: Extract → Validate → Transform → Load, with escalation tiers.
  • Confidence scores let agents flag uncertain fields for human review.
  • Batch API: 50% savings, 24-hour window — for high-volume, not real-time.
