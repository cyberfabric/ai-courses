Chapter 3

Task Decomposition

Memory Map
💡

The Big Idea

Break complex tasks into agent-sized pieces — like a project manager splitting an epic into stories and tasks. Get the granularity right: too coarse overwhelms the agent, too fine drowns in overhead.

🎯 Why Decompose?

📦
Context Fits
Smaller tasks fit in smaller context windows. A 50-page doc with 10 tool calls burns through context fast.
⚡
Parallelism
Independent subtasks run simultaneously. 5 parallel subagents = 5x faster than sequential.
🎯
Reliability
LLMs reason better on focused tasks. "Analyze this function" beats "audit the entire codebase."

⚖️ Static vs. Dynamic (Exam Favorite!)

Aspect 📋 Static (Prompt Chaining) 🔀 Dynamic (Adaptive)
Structure Predetermined — steps known in advance Decided at runtime based on results
Analogy Following a recipe Debugging — depends on what you find
Branching No conditional branches Branches based on intermediate results
# of subtasks Known in advance Unknown — could be 2 or 10
Best for Invoice processing, ETL pipelines Research, debugging, customer support
Exam keyword "fixed pipeline," "same steps every time" "depends on," "conditional," "unpredictable"

🌊 The Fan-Out / Fan-In Pattern

🧩 Decompose
task into parts
→
📤 Fan Out
N parallel agents
→
📥 Fan In
collect N results
→
📝 Synthesize
final output

Think MapReduce: fan-out = Map (distribute), fan-in = Reduce (combine)

🏗️ Hierarchical Decomposition

  1. 🎯 Epic Level — Orchestrator

    "Build customer support agent" — the top-level goal that gets broken into stories.

  2. 📖 Story Level — Subagents

    "Implement ticket classification," "Build KB search," "Add escalation logic" — each handled by a focused subagent.

  3. ✅ Task Level — Tool Calls

    "Classify by type," "Extract priority," "Route to specialist" — individual tool calls within a subagent.

Parse first → three checks in parallel → report waits for all three

📖 Key Terms

Prompt Chaining
Static decomposition — predetermined steps, same every time. Exam term for fixed pipelines.
Adaptive Decomposition
Dynamic — agent decides next steps at runtime. Exam term for conditional/branching workflows.
Fan-Out / Fan-In
Parallelize independent subtasks (fan-out), collect and combine results (fan-in). Like MapReduce.
DAG (Directed Acyclic Graph)
Tasks with dependencies but no cycles. "Some tasks wait for others, and there are no loops."
Granularity
How big or small each subtask is. Sweet spot: 3–7 parallel subtasks, each using 30–70% of context window.
Subtask Criticality
Tag each subtask as "critical" (must succeed) or "optional" (can skip). Drives error-handling strategy.

📏 The Goldilocks Zone

Signal ❌ Too Coarse ❌ Too Fine ✅ Just Right
Context Agent runs out Mostly overhead Uses 30–70% of window
Focus Confused by scope Trivially simple Clear, no ambiguity
Errors Overload mistakes Coordination errors Reliable per subtask
Parallelism One blob — can't split Too many to manage 3–7 parallel tasks

🚨 Error Handling: Retry, Skip, Escalate

🔄
Retry
For transient failures (API timeout, rate limit). Use exponential backoff + max retry count.
⏭️
Skip
For optional subtasks. Mark as failed, continue with partial results. Research: 4 of 5 sources still useful.
🆘
Escalate
For critical subtasks. Stop and hand off to human or higher-level agent. Can't proceed without this result.

🎯 Scenario Cheat Sheet

🎧
Customer Support
Dynamic: classify → route → resolve/escalate. Branches on issue type.
💻
Code Generation
Hybrid: static phases (plan → implement → test), dynamic within each.
🔬
Multi-Agent Research
Fan-out/fan-in: decompose query → parallel research → aggregate.
🔄
CI/CD
DAG: lint + test + security in parallel → deploy depends on all passing.
📄
Data Extraction
Static pipeline: extract → parse → validate → transform. Same every time.
⚡
Dev Productivity
Dynamic: adapt workflow based on developer's task and discovered context.

🧭 Decomposition Decision Tree

  1. Predictable workflow?

    Yes → static (prompt chaining). No → dynamic (adaptive).

  2. Independent subtasks?

    Yes → fan-out for parallel execution. No → sequential pipeline.

  3. Complex dependencies?

    Yes → model as DAG, find critical path. No → simple fan-out/fan-in or pipeline.

  4. Right-sized subtasks?

    Too big → decompose more. Too small → merge related tasks. Just right → done!

🧪 Quick Quiz

Invoice processing: extract → parse → validate → calculate. Static or dynamic?

Static (prompt chaining). Fixed steps, predictable input/output, no conditional branches.

Debugging a production error. Static or dynamic?

Dynamic (adaptive). Next steps depend on what you find — logs, hypothesis, investigation path all change based on intermediate results.

"Multiple independent data sources need to be queried and results combined." What pattern?

Fan-out/fan-in. Independent sources = parallel execution, then aggregate. Sequential processing is a common distractor.

A research subagent fails. The other 4 succeeded. What error strategy?

Skip — produce a partial report with 4 sources. Each sub-question is independently valuable. Don't abort the whole task for one optional failure.

What's the ideal number of parallel subtasks?

3–7. Fewer means you're not parallelizing enough. More means orchestration overhead dominates.

🔑 Key Takeaways

  • Static (prompt chaining) for predictable pipelines; dynamic (adaptive) for tasks with conditional branches.
  • Fan-out/fan-in is the star pattern: parallelize independent work, then aggregate. Think MapReduce.
  • Goldilocks granularity: 3–7 parallel subtasks, each using 30–70% of context window.
  • Error handling by criticality: retry transient failures, skip optional subtasks, escalate critical ones.
  • Use the 4-step decision tree: predictable? → independent? → complex deps? → right-sized?
