The Big Idea

The Architect thinks in modules. The Developer thinks in tasks. A planning step bridges the gap — breaking modules into right-sized, dependency-ordered tasks that the orchestrator can dispatch one at a time.

🎯 Task Granularity: The Goldilocks Zone

Aspect ❌ Too Large ✅ Right-Sized ❌ Too Small
Files per task 20+ files 3-8 files 1-2 files
Scope Entire module One layer, one module Single class
Problem Context overflow, quality drops Focused & coherent Orchestration overhead
Testability Hard to isolate failures Compiles & tests alone Can't test partial layer

🏃 Four Dependency-Boxed Sprints

  1. Sprint 1: Foundation (Tasks 1-4, ~$0.40)

    Domain entities, use cases, UI theme, test fakes. No dependencies — build first.

  2. Sprint 2: Data Layer (Tasks 5-8, ~$0.40)

    Room entities, DAOs, AppDatabase, repositories. Depends on Sprint 1.

  3. Sprint 3: Features (Tasks 9-16, ~$0.80-$1.60)

    ViewModels + Screens for all 4 features. Parallelizable — no cross-feature dependencies.

  4. Sprint 4: Integration (Tasks 17-20, ~$0.40)

    Navigation graph, Hilt wiring, feature integration, polish. Depends on everything.

Total: ~20 tasks • 4 sprints • ~$2.00-$2.80 • ~30-45 min

📋 Task Spec Anatomy

🏷️
Header
Task number, module, title, depends_on (which tasks must be complete), sprint number.
🎯
Objective
One sentence describing what this task produces.
📁
Files to Create
Exact file paths the Developer should create. 3-8 files per task.
📦
Dependencies Available
What already exists from previous tasks. The Developer knows what to import.
✅
Acceptance Criteria
Pulled from specs.md. The Developer knows when the task is done.
📏
Rules to Follow
Relevant RULE-NNN references from the Architect's rules.md.

⚠️ Risk Assessment

Risk Level Tasks Review Strategy
🟢 Low Domain entities, test fakes, Room entities, DAOs Skip Code Reviewer → straight to QA
🟡 Medium ViewModels, repositories, Compose screens Standard review loop
🔴 High AppDatabase, navigation graph, integration wiring Review + security check, plan for re-dispatch

Risk also drives model selection: route 🟢 low-risk tasks to Haiku 4.5 ($1/$5, 3× cheaper than Sonnet 4.6). Sprint 1 drops from ~$0.40 to ~$0.13 with no quality cost on mechanical work.

📖 Key Terms

Task Decomposition
Breaking modules into layer-level tasks of 3-8 files each, sized for a single Developer dispatch.
Goldilocks Zone
3-8 files per task. Fewer wastes orchestration overhead; more causes context overflow and quality drops.
Dependency-Boxed Sprint
A group of tasks at the same dependency level. Sprints advance when all tasks at that level are complete.
Task Spec
Structured format sent to the Developer: header, objective, files, dependencies, criteria, rules.
Risk Assessment
Low/medium/high classification per task. Drives review routing: skip review, standard loop, or review + security.
Parallel Opportunities
Tasks with no cross-dependencies (e.g., all feature modules in Sprint 3) can be dispatched simultaneously.
Validation Checkpoint
After each sprint: does the code compile? Do tests pass? If not, re-dispatch before advancing.

🧪 Quick Quiz

Why not hand the Architect's raw backlog directly to the Developer?

The Architect's backlog is module-sized (20+ files). That's too large for one Developer dispatch — context overflows and quality degrades. Tasks need to be 3-8 files each.

What are the four Goldilocks rules for task sizing?

3-8 files per task. One layer, one module. Clear inputs and outputs. Testable in isolation (compiles and passes its own tests).

Why are Sprint 3 tasks parallelizable?

Feature modules share no dependencies between them — they all depend on the data layer (Sprint 2) but not on each other. All 8 tasks can run simultaneously.

How does risk level affect the review pipeline?

Low risk: skip Code Reviewer, go to QA. Medium: standard review loop. High: review + security check, with re-dispatch planned.

What does the orchestrator do between sprints?

Validates: does the code compile? Do existing tests pass? If validation fails, re-dispatches the failing task before moving to the next sprint.

📌 Key Takeaways

  • Modules → tasks: Break each module by layer. Each layer becomes a 3-8 file task.
  • Goldilocks zone: 3-8 files per task. One layer, one module. Testable in isolation.
  • 4 sprints: Foundation → Data → Features (parallel) → Integration. ~20 tasks total.
  • Task specs: Header + objective + files + dependencies + criteria + rules = Developer's entire world.
  • Risk flags: Low/medium/high drives review routing AND model selection (Haiku 4.5 for low-risk, Sonnet 4.6 for medium/high).
  • Validate between sprints: Compile + test before advancing. Fix failures before adding complexity.
  • 10 min planning saves hours: Right-sized tasks = fewer re-dispatches = lower cost.
