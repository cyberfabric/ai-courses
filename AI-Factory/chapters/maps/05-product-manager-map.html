The AI Product Manager: Specs for Machines

Memory Map
💡

The Big Idea

Specs written for AI consumption are fundamentally different from specs written for humans. Every behavior must be enumerated, every edge case listed, every acceptance criterion testable by code. The PM eliminates ambiguity so downstream roles never guess.

🔄 Human Specs vs. AI Specs

Aspect ❌ Human Spec ✅ AI Spec
Speed "Search should be fast" "Results within 200ms for 1,000 recipes"
Behavior "Implement search" "300ms debounce, min 2 chars, title + ingredients + tags"
Empty state (not mentioned) "No results shows 'No recipes found' message"
Edge cases 3-5 obvious ones 15-20 including boundary values and concurrent ops
Decisions Left to developer judgment Every decision made and documented

The "Should" Smell

The Technology Leak
"Store in a local database" = fine (what). "Store in Room" = technology leak (how). The PM defines needs; the Architect picks tools.

🔬 Feature Spec Anatomy

  1. Persona Link

    Which persona does this feature serve? Prevents orphan features nobody needs.

  2. User Stories (Given/When/Then)

    Maps directly to test code. The QA role converts these into test cases with minimal interpretation.

  3. Behaviors

    Every action with exact values: debounce timing, max lengths, result ordering, default states.

  4. Edge Cases (5-10 per feature)

    Boundary values, empty states, concurrent access, low storage, rapid input. AI's strength.

  5. Acceptance Criteria

    Testable assertions: if you can't write assertEquals() for it, rewrite it.

🎯 Each Section Feeds a Downstream Role

📖 Stories → Architect (screens, navigation, state)
⚙️ Behaviors → Developer (implementation instructions)
📋 specs.md
🧪 Edge Cases → QA Engineer (test scenarios)
✅ Criteria → Code Reviewer (pass/fail decisions)

📖 Key Terms

AI-Consumable Spec
A specification with every behavior enumerated, every value explicit, and every criterion testable by code.
Given/When/Then
User story format that maps directly to test assertions. The QA role converts these into automated tests.
Edge Case Enumeration
Listing 5-10 boundary scenarios per feature. AI produces 15-20 where humans typically find 3-5.
The "Should" Smell
Any use of "should," "probably," or "might" signals an unmade decision. Replace with specific values.
Technology Leak
When the PM names a specific library or tool instead of describing the need. "Local database" = OK. "Room" = leak.
Non-Functional Requirements
App-wide targets: performance budgets, accessibility minimums, data limits. Every number becomes a test assertion.
API Contracts
Shared data models and cross-feature navigation rules that prevent integration bugs between modules.
PM Output Contract
The 7-item checklist the Architect expects: frontmatter, feature specs, stories, edge cases, criteria, NFRs, contracts.

🧪 Quick Quiz

Why are AI specs different from human specs?

Humans fill gaps with experience and judgment. AI needs every behavior enumerated, every value explicit, and every criterion testable. Vague specs produce inconsistent output.

What is the "should" smell?

Any use of "should," "probably," or "might" in a spec is a decision the PM didn't make. Replace each with a specific value: "search should be fast" → "search returns results within 200ms."

Why does the PM skill check the vision's status field?

It enforces the handoff protocol. If the vision is still draft or rejected, the PM refuses to work — preventing specs based on unapproved vision.

How do edge cases save tokens downstream?

Every edge case the PM catches upfront is one fewer Developer-Reviewer loop. The Developer handles it in the first pass instead of in a re-dispatch bug-fix cycle.

Why is there no human checkpoint after the PM?

The PM's output is a deterministic expansion of the already-approved vision. If the vision is right, the specs will be right. You can add a checkpoint in the orchestrator if needed.

📌 Key Takeaways

  • Specs for machines: Every behavior explicit, every value specific, every criterion testable by code.
  • 5-part feature spec: Persona link, Given/When/Then stories, behaviors, edge cases, acceptance criteria.
  • The "should" smell: Search PM output for vague words. Each one is an unmade decision.
  • Edge case strength: AI finds 15-20 edge cases where humans find 3-5. Prevents re-dispatch loops.
  • No technology leaks: PM defines needs (what), never tools (how). That's the Architect's job.
  • Input validation: PM checks status: approved before working. Roles enforce the contract.
  • Fast and cheap: ~3 min, under $0.10 on Opus 4.7 for 15 stories, 39 behaviors, 33 edge cases, 20 criteria.
