Security Review and QA Engineering

The Big Idea

Code that passes architecture rules isn't necessarily secure or thoroughly tested. Two final roles complete the quality stack: the Security Reviewer finds vulnerabilities, the QA Engineer generates the tests the Developer skipped.

🔒 Security Review: 5 OWASP Categories

  1. Data Storage (M2)

    No plaintext secrets, no logged PII, encrypted storage for sensitive data.

  2. Input Validation (M8)

    Sanitized inputs, parameterized queries only, no path traversal.

  3. Data Export (M1)

    Scoped FileProvider, permissions on content providers, backup rules.

  4. Dependencies (M7)

    No known CVEs, no unnecessary permissions in manifest.

  5. Cryptography

    No custom crypto, Android Keystore, TLS 1.2 minimum.

⚠️ Security Severity & Verdicts

🔴
CRITICAL
Exploitable vulnerability. Data leak, injection, exposed secrets. Escalate to human immediately.
🟠
HIGH
Likely exploitable with effort. Missing encryption, overly broad permissions. Must fix.
🟡
MEDIUM
Defense-in-depth gap. Missing validation on non-sensitive field, verbose logging.
🟢
LOW
Best practice suggestion. Safe but could be hardened. Fix if time allows.
Verdict Criteria Action
SECURE 0 CRITICAL, 0 HIGH, ≤2 MEDIUM Advance. Log LOW findings.
FIX-REQUIRED Any HIGH, or 3+ MEDIUM Send to Developer for remediation.
BLOCK Any CRITICAL Escalate to human. No auto-fix.

🧪 QA Test Generation Priorities

1️⃣
Edge Cases
Empty inputs, null values, max-length strings, boundary values, special characters, unicode.
2️⃣
State Transitions
Loading → Success, Loading → Error, Error → Retry → Success, Empty → Non-Empty.
3️⃣
Integration Tests
ViewModel + Repository end-to-end, in-memory Room operations, navigation actions.
4️⃣
Regression Guards
One test per Code Reviewer bug found, one test per security finding remediation.

📖 Key Terms

Quality Stack
Three quality roles in sequence: Code Reviewer (rules) → Security Reviewer (vulnerabilities) → QA Engineer (tests).
OWASP Mobile Top 10
Industry-standard mobile vulnerability categories. The Security Reviewer checks M1, M2, M7, M8, and cryptography.
OWASP LLM Top 10 (2025 v2.0)
Vulnerabilities of the AI Factory itself, not the recipes app. Most relevant: LLM01 Prompt Injection, LLM02 Sensitive Info Disclosure, LLM05 Improper Output Handling, LLM06 Excessive Agency.
OWASP Agentic Top 10 (2026)
Risks specific to multi-agent pipelines like the Factory: goal misalignment, tool misuse, inter-agent communication, persistent memory poisoning. Core principle: Least-Agency — grant each role only the autonomy it needs.
Ship Readiness
All three quality roles pass, all tests pass, zero warnings, no TODOs. Pipeline produces the report; human makes the decision.
BLOCK Verdict
CRITICAL security finding = escalate to human immediately. The Developer shouldn't auto-fix exploitable vulnerabilities.
QA Test Generation
QA doesn't just evaluate — it generates new tests: edge cases, state transitions, integration, regression guards.
PREFER Accumulation
Security LOWs and Code Review PREFERs accumulate. If the same finding recurs across 5+ tasks, promote its severity.

🧪 Quick Quiz

Why does the Security Reviewer use Opus 4.7 while the QA Engineer uses Sonnet 4.6 (or Haiku 4.5)?

Security requires judgment — understanding intent, weighing exploitability, deciding severity. Test generation is mechanical — following patterns from existing tests. Different cognitive demands, different models. In 2026, QA's high-volume mechanical work makes it a strong Haiku 4.5 candidate (3× cheaper than Sonnet, 4–5× faster, ~5pp behind on SWE-bench), while Security still earns Opus 4.7's deeper reasoning.

Why do CRITICAL security findings always go to a human?

The Developer role shouldn't auto-fix exploitable vulnerabilities — the fix itself could introduce new vulnerabilities. Security-critical code needs human judgment to verify both the finding and the remediation.

What's the difference between what the QA Engineer and Developer produce?

The Developer writes one test per public class (the minimum). The QA Engineer generates edge case tests, state transition tests, integration tests, and regression guards — the comprehensive coverage the Developer skipped.

Why does the quality stack cost ~55% of the entire pipeline?

Quality is expensive but cascading failures are more expensive. A MUST finding in an early task that goes undetected breaks 10+ downstream tasks. The quality cost pays for itself by preventing rework.

When does "ship readiness" not equal "ship decision"?

Always. The pipeline produces a readiness report, but a human makes the ship decision. The human spot-checks code, verifies business logic, and makes the final call. The pipeline reduces work, not responsibility.

📌 Key Takeaways

  • Two final roles: Security Reviewer (Opus 4.7, high-risk, OWASP) + QA Engineer (Sonnet 4.6 default / Haiku 4.5 for high-volume runs, all tasks, generates tests).
  • Security verdicts: SECURE (ship), FIX-REQUIRED (remediate), BLOCK (escalate CRITICAL to human).
  • QA generates, not just evaluates: Edge cases, state transitions, integration tests, regression guards.
  • 128 tests total: 48 Developer (minimum) + 80 QA-generated, 92% criteria coverage.
  • Quality stack ~$5.30 (2026): Code Review (~$2.90) + Security (~$0.40) + QA (~$2.00 Sonnet / ~$0.60 Haiku 4.5) under Opus 4.7 pricing — roughly half the older Opus-era total of $10.10.
  • CRITICAL = human always: No auto-fix for exploitable vulnerabilities.
  • Ship readiness ≠ ship decision: Pipeline produces the report, human makes the call.
