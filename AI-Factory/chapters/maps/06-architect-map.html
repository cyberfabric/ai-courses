Chapter 6

The AI Architect: Design for the Machine

Memory Map
💡

The Big Idea

The Architect doesn't just design the system — it generates the architecture rules that keep 10 separate Developer dispatches building a coherent app instead of 10 disconnected fragments. Rules are the Architect's primary contribution to code quality.

📦 Three Artifacts, Not One

📋 specs.md
→
🏗️ Architect (Opus 4.7)
→
📐 architecture.md
📏 rules.md
📋 backlog/*.md
📐
architecture.md
Module list with responsibilities, dependency graph, data models at all layers, technology stack with versions.
📏
rules.md
15-20 numbered rules with ✅ correct / ❌ wrong examples. Covers naming, layers, DI, state, errors, testing.
📋
backlog/*.md
One file per module in dependency order. Each lists files to create, dependencies, and acceptance criteria.

🧱 RecipeVault: 10-Module Structure

Layer Modules Responsibility
Core :core:domain, :core:ui Domain entities + use cases, design system + shared composables
Data :data:local, :data:repository Room DB + DAOs + mappers, repository implementations
Feature :feature:recipe-list, :feature:recipe-detail, :feature:recipe-books, :feature:recipe-create One screen/flow per module, each with ViewModel + UI
App :app Hilt setup, navigation graph, entry point
Test :test:shared Test fakes, builders, shared utilities

🔗 Dependency Graph & Build Order

:core:domain
→
:core:ui
:data:local
→
:data:repository
→
:feature:*
→
:app
⬆️
Dependencies Point Inward
presentation → domain ← data. Domain has zero framework imports. Feature modules never depend on other features.
⚡
Parallel Opportunities
:core:ui and :data:local can build in parallel. All four :feature:* modules can build in parallel.
🚫
No Circular Dependencies
If A depends on B and B depends on A, neither can build first. The Architect must ensure one-directional flow.
🎯
10 Modules = 10 Dispatches
Each module is scoped for a single Developer invocation. The Architect designs work units, not just architecture.

📊 Data Models: Three Layers

  1. Domain Entities (:core:domain)

    Pure Kotlin data classes. No framework annotations. Used by use cases, shared across all features.

  2. Database Entities (:data:local)

    Room @Entity classes with table/column annotations. Never leak outside data layer.

  3. Mappers (:data:local)

    Extension functions converting between domain ↔ database. Tested independently.

📏 Architecture Rules: The Developer's Guardrails

Rule ✅ Correct ❌ Wrong
RULE-001: Domain = zero framework imports data class Recipe(val id, val title) @Entity data class Recipe(...)
RULE-002: Features never depend on features :recipe-list → :core:domain :recipe-list → :recipe-detail
RULE-003: Constructor injection only @Inject constructor(dao) val dao = RecipeDao()
RULE-005: Single sealed UiState per screen sealed interface RecipeListUiState Multiple LiveData for loading/error/data
RULE-006: Repos return Result<T> fun get(): Result<Recipe> fun get(): Recipe // throws
🛡️
Self-Enforcing
✅/❌ examples let the Developer pattern-match and the Code Reviewer cite rule numbers when sending code back.
⚠️
The Rules Gap
Most common failure: beautiful architecture doc but weak rules. Without strong rules, 10 modules look like 10 different codebases.

📖 Key Terms

Module Decomposition
Breaking the app into 10 modules, each scoped for a single Developer dispatch.
Dependency Graph
Which modules depend on which. Determines build order and parallel opportunities.
Architecture Rules
Numbered constraints (RULE-001, etc.) with ✅/❌ examples that the Developer must follow and the Reviewer checks.
Three-Layer Data Models
Domain entities (pure Kotlin), database entities (Room @Entity), and mappers between them.
Implementation Backlog
One markdown file per module with files to create, dependencies, and acceptance criteria from specs.
The Rules Gap
When the Architect produces good design docs but weak rules, causing inconsistent Developer output.
Architecture Checkpoint
Second human review point. Wrong architecture is the most expensive mistake — fix it here, not during implementation.

🧪 Quick Quiz

Why does the Architect produce three artifacts instead of one?

architecture.md describes the system. rules.md constrains the Developer. backlog/*.md gives the Developer specific tasks. Each serves a different consumer.

Why are architecture rules more important than the architecture document?

Rules keep 10 separate Developer dispatches consistent. Without rules, each invocation makes its own style decisions, producing 10 modules that look like 10 different codebases.

Why does the Architect use Opus 4.7 instead of Sonnet 4.6?

Sonnet 4.6 produces generic output (textbook patterns, oversized tasks). Opus 4.7 reasons about tradeoffs and makes project-specific decisions (domain-tailored rules, properly scoped work units). Sonnet is still ~40% cheaper, but the gap shrunk dramatically from older Opus generations — for a leverage-heavy planning role, paying for Opus 4.7 judgment is the right tradeoff.

What makes circular dependencies a pipeline-killer?

If module A depends on B and B depends on A, neither can build first. The Developer can't start either module, blocking the entire implementation phase.

When should you re-dispatch the Architect vs. edit the files directly?

Edit directly when 90% right — add a missing rule, fix a dependency, rename a module. Re-dispatch only if the architecture is fundamentally wrong (wrong module structure, missing layers).

📌 Key Takeaways

  • Three outputs: architecture.md (design) + rules.md (constraints) + backlog/*.md (tasks).
  • 10 modules = 10 dispatches: The Architect designs work units, not just architecture.
  • Dependency graph = build order: Core first, then data, then features (parallel), then app.
  • Rules > docs: Numbered rules with ✅/❌ examples keep Developer output consistent.
  • Three-layer data: Domain entities (pure), database entities (Room), mappers between them.
  • Second checkpoint: Wrong architecture is the most expensive mistake. Review before building.
  • Opus 4.7 for judgment: Architect needs deep reasoning for project-specific tradeoffs. Cost gap to Sonnet shrunk dramatically; spend Opus on roles that set rules, save Sonnet for roles that follow them.
