Orchestration With Claude Code

Memory Map

The Big Idea

Three mechanisms make multi-role AI pipelines work: the Agent tool dispatches isolated subagents, plan mode structures thinking before action, and skill files encode each role's contract.

🏗️ The Three Pillars of Orchestration

🚀
Agent Tool
Spawns isolated subagents with their own context and tools. Like a function call where the function is an AI.
📋
Plan Mode
Forces structured thinking before dispatching. Decides role order, checkpoints, and error recovery strategy.
📄
Skills & Sub-agents
Skills (.claude/skills/ or .claude/commands/) are user-invokable; sub-agents (.claude/agents/) are dispatched via the Agent tool. Both encode a role contract.

🔬 Anatomy of a Skill File

  1. Identity

    Who you ARE and what you do NOT do. "You design systems. You do NOT write code."

  2. Input

    What the role receives from the previous stage. Explicit format and source.

  3. Output

    Exact sections the role must produce. Specific enough that the next role can consume without guessing.

  4. Rules

    Hard boundaries the role must never cross. Prevents role bleeding.

📖 Key Terms

Subagent
An isolated Claude instance spawned by the Agent tool with its own context and tools.
Orchestrator
A master skill that coordinates all role skills — plans, dispatches, captures, and routes.
Artifact Chain
The data flow: each role's output becomes the next role's input (vision → specs → design → code).
File-Based Handoff
Saving artifacts to artifacts/ directory so roles read files instead of inline content.
Role Registry
A mapping of which skill file + which model handles each role, with fallback strategies.
The "NOT" Pattern
Every skill explicitly states what the role does NOT do, preventing role bleeding.
Checkpoint
A point where the pipeline pauses for human approval before continuing.

🧪 Quick Quiz

Why does the Agent tool run subagents in isolation?

To prevent roles from overstepping their contracts. Each role sees only the context it needs, not the full conversation history.

What are the four sections of a production-grade skill file?

Identity (who you are/aren't), Input (what you receive), Output (exact sections to produce), Rules (hard boundaries).

Why save artifacts to files instead of passing them inline?

Files survive interruption, are editable between runs, work with any size, and don't bloat the prompt.

Why is there no human checkpoint after the Developer role?

The Code Reviewer handles quality automatically. Humans only step in where AI judgment is unreliable: product direction, architecture, and security risk.

When should a role use Opus vs Sonnet vs Haiku?

Opus 4.7 for judgment (Architect, Code Reviewer, Security) — price drop to $5/$25 made it economical for everyday review. Sonnet 4.6 for rule-following work where speed matters (Developer, QA). Haiku 4.5 ($1/$5) for the most mechanical, high-volume tasks — ~3× cheaper than Sonnet.

What's the difference between a skill and a sub-agent?

A skill in .claude/skills/ (or .claude/commands/) is user-invokable via /name and can be auto-loaded by Claude. A sub-agent in .claude/agents/ is delegated to via the Agent tool with its own context window. The orchestrator is best as a skill; the seven roles work well as sub-agents.

📌 Key Takeaways

  • Agent tool: Dispatches isolated subagents. Each gets only the context it needs.
  • Plan first: The orchestrator writes a pipeline plan before dispatching any role.
  • Skill anatomy: Identity + Input + Output + Rules = a complete role definition.
  • File-based handoffs: Save artifacts to disk. Roles read files, not inline blobs.
  • Error recovery: Detect → Diagnose → Re-dispatch → Escalate after 2 retries.
  • Human checkpoints: After vision, after architecture, on security criticals. Not after developer.
  • Model selection: opus-4.7 for judgment (cost gap to Sonnet collapsed), sonnet-4.6 for execution, haiku-4.5 for the most mechanical work.
  • Skills vs sub-agents: skills/commands are user-invokable; sub-agents in .claude/agents/ are dispatched via the Agent tool.
