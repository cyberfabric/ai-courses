The AI Product Owner: Vision and Scope

The Big Idea

The PO answers one question: "What are we building and for whom?" It produces a vision document specific enough that the PM role can consume it without asking follow-up questions.

📄 Vision Document: 6 Required Sections

  1. App Description

    2-3 paragraphs: core value proposition and why the app needs to exist.

  2. User Personas (3-5)

    Name, demographics, goals, pain points, usage patterns. Real people, not abstractions.

  3. Feature List (MoSCoW)

    Every feature categorized: Must / Should / Could / Won't. Ruthlessly honest.

  4. MVP Scope

    3-5 Must Have features ONLY. If it's more than 5, the PO isn't being ruthless enough.

  5. Release Roadmap

    MVP → v1 (Should Haves) → v2 (Could Haves). Pipeline runs MVP first.

  6. Success Metrics

    3-5 measurable outcomes. "User can create a recipe in under 2 minutes."

📊 MoSCoW for the Recipes App

Priority Features When
Must CRUD recipes, recipe books, search, offline MVP
Should Grocery list, URL import, photos, tags v1
Could Sharing, meal planner, timers, nutrition v2
Won't Social features, cloud sync, video Never (this project)

👥 Strong vs Weak Personas

❌
Weak Persona
"User A wants to save recipes." Too generic — every feature "helps User A." Can't make priority decisions.
✅
Strong Persona
"Maria, 34, working mom. Needs 30-min dinners. Recipes in 5 apps. No signal at grocery store." Drives real decisions.
🎯
Personas Drive Priorities
Maria drives offline + search + grocery list. If a feature doesn't serve a persona, it's probably a Could Have.
📋
PM Uses Personas
"As Maria, I want to search recipes so I can find a weeknight dinner in under 10 seconds." Stories reference real people.

🔬 PO Skill File Structure

🎭 Identity: WHAT to build, not HOW

(or legacy .claude/commands/product-owner.md)

🚫 Rules: no tech, no code, no architecture

👤 The Vision Checkpoint

✅
Approve
Set status: approved. PM proceeds with the vision as-is.
✏️
Edit & Approve
Fix the file directly. Faster and cheaper than re-dispatching. The recommended approach for 90%-right output.
🔄
Reject & Re-dispatch
Send back to PO with specific feedback. Use only when the vision is fundamentally wrong.

📖 Key Terms

Product Owner (PO)
First role in the pipeline. Answers "What are we building and for whom?" Produces the vision document.
Vision Document
The PO's output: 6 sections covering description, personas, features, MVP, roadmap, and metrics.
MoSCoW
Prioritization framework: Must Have (MVP), Should Have (v1), Could Have (v2), Won't Have (out of scope).
User Persona
A specific fictional user with name, goals, pain points, and usage patterns. Drives feature priorities.
Vision Checkpoint
First human review point. Approve, edit, or reject the vision before the PM proceeds.
Output Contract
The 7-item checklist the PM expects: frontmatter, description, personas, MoSCoW, MVP, roadmap, metrics.

🧪 Quick Quiz

What is the PO's single question?

"What are we building and for whom?" The PO defines vision and scope, not architecture or code.

How many Must Have features should the MVP include?

3-5 maximum. If your Must Have list is longer, the PO isn't being ruthless enough about priorities.

What makes a persona "strong" vs "weak"?

Strong personas have a name, demographics, specific goals, pain points, and usage patterns. Weak personas are generic ("User A wants to save recipes").

When should you edit the vision file directly vs re-dispatching the PO?

Edit directly when the output is 90% right — it's faster and cheaper. Re-dispatch only when the vision is fundamentally wrong.

What 3 things does the PO skill explicitly NOT do?

Suggest technologies, describe architecture patterns, or write code. These are the Architect's and Developer's jobs.

📌 Key Takeaways

  • PO = vision: WHAT to build and for WHOM. Never HOW.
  • 6 required sections: Description, Personas, MoSCoW Features, MVP Scope, Roadmap, Metrics.
  • Strong personas: Real people with names, goals, and pain points. They drive priorities.
  • Ruthless MoSCoW: Most features are Should/Could, not Must. MVP = 3-5 features.
  • Edit, don't re-dispatch: Fix a 90%-right vision in 30 seconds instead of spending tokens.
  • First checkpoint: Human approves vision before PM proceeds. Wrong vision = wrong everything.
