One Sentence to App

Memory Map
💡

The Big Idea

One sentence + constraints = a complete pipeline execution plan. Configure everything before dispatching the first role — iterate on the plan, not the pipeline.

🌱 Seed + Constraints

💬
The Seed
"I want an offline-first Android app for managing recipes, organizing them into books, and generating grocery lists."
🔒
Constraints
Platform, UI, architecture, DI, storage, network strategy, audience. These are the pipeline's parameters.
🎯
Why Constraints?
A seed without constraints is like a function without parameters — the output could be anything.
🔄
Swap & Replay
Change a constraint (Room → DataStore), re-run the pipeline. Same factory, different parameters, different output.

🤝 Handoff Protocol

📝
YAML Frontmatter
Every artifact has: artifact type, produced_by, consumed_by, version, and status.
✅
Status Field
draft → approved at checkpoints. Roles refuse to consume rejected artifacts.
🔢
Versioning
Dev-Reviewer loop produces v1, v2, v3. Without versioning, you can't tell which iteration you're on.

⚡ The Four Pipeline Phases

  1. Discovery (~5 min, ~$0.10)

    PO → vision.md → CHECKPOINT → PM → specs.md (Opus 4.7)

  2. Design (~4 min, ~$0.10)

    Architect → architecture.md + rules.md + backlog/*.md → CHECKPOINT (Opus 4.7)

  3. Implementation (~10 min, ~$2.00)

    Developer × 10 modules on Sonnet 4.6 in dependency order: :core:domain → :core:ui → :data:* → :feature:* → :app

  4. Quality (~8 min, ~$1.15)

    Code Reviewer + Security Reviewer (Opus 4.7) → QA Engineer (Sonnet 4.6) with re-dispatch loops if needed

📖 Key Terms

Seed Idea
The one-sentence input that starts the pipeline. 20 words that become thousands of lines.
Constraints
Pipeline parameters: platform, architecture, DI, storage, audience. Change one, get different output.
Artifact Chain
The data flow: each role's output file becomes the next role's input file.
Handoff Protocol
YAML frontmatter standard (type, producer, consumer, version, status) on every artifact.
Plan-Only Mode
Run the orchestrator without dispatching roles — produces just the execution plan for review.
Dependency Order
Modules are built in dependency sequence: core first, then data, then features, then app.

🧪 Quick Quiz

Why does a seed need constraints?

Without constraints, the output could be anything. Constraints narrow the solution space so every role has concrete boundaries.

What fields must every artifact's YAML frontmatter include?

artifact (type), produced_by, consumed_by, version, and status (draft/approved/rejected).

Which role is the biggest cost driver and why?

The Developer — it runs 10 times (once per module) on Sonnet 4.6, generating ~80K tokens total (~$2.00). Volume makes it bigger than all the Opus 4.7 roles combined, since Opus 4.7's price dropped to $5/$25 per MTok.

Why use plan-only mode before a full pipeline run?

Planning costs almost nothing. Executing a wrong plan wastes real tokens. Review the plan first — fix the blueprint, not the building.

What does the status field in the handoff protocol control?

It tracks human approval at checkpoints. Roles should refuse to consume artifacts marked as rejected.

📌 Key Takeaways

  • Seed + constraints: One sentence plus 7 parameters define the entire pipeline run.
  • Artifact chain: 9 artifacts flow through 7 roles — each output is the next input.
  • Handoff protocol: YAML frontmatter with type, producer, consumer, version, status on every artifact.
  • Four phases: Discovery → Design → Implementation → Quality (~25 min, ~$2-5 total under Opus 4.7 + Sonnet 4.6).
  • Plan first: Run plan-only mode, review, iterate. Then commit real tokens.
  • Project structure: Skills in .claude/commands/, artifacts in artifacts/, code in app/src/.
