The AI Code Reviewer: Quality at Machine Speed

Memory Map
💡

The Big Idea

The Developer can't judge its own work. A separate AI invocation with fresh context loads the same architecture rules and produces structured findings with rule IDs, severity levels, and specific fixes — the automated first pass before human eyes see anything.

👀 Why Not Self-Review?

Aspect ❌ Self-Review (Same Context) ✅ Separate Reviewer (Fresh Context)
Bias Same biases that wrote the code Fresh eyes, no anchoring
Context priority Code first, rules second Rules first, code second
Catch rate ~60% of issues ~90% of rule violations
Output Checklist (✅/❌) Structured findings with fixes

⚖️ Severity Levels & Verdicts

🔴
MUST
Breaks the pipeline or other modules. Wrong package, missing @Inject, circular dependency.
🟡
SHOULD
Degrades quality but won't block. Naming drift, missing edge case test, verbose code.
🔵
PREFER
Style suggestion for consistency. Parameter ordering, import ordering. Logged but never affects verdict.
Verdict Criteria Action
PASS 0 MUST, ≤2 SHOULD Ship it. Advance to next task.
REVISE Any MUST, or 3+ SHOULD Send findings back to Developer.
REJECT Structural failure Escalate. Task spec needs revision.

📋 Anatomy of a Finding

  1. Rule ID

    Which RULE-NNN from rules.md was violated.

  2. Severity

    MUST / SHOULD / PREFER — determines the verdict.

  3. Location

    Exact file path and line number for the Developer to fix.

  4. Issue

    What's wrong and why it matters.

  5. Fix

    Specific code change to make. Each finding is a mini task spec.

🎯 Risk-Based Review Routing

Low Risk — Skip Review

ViewModels, repositories, Compose screens. Full review loop with structured findings. 12 tasks, ~$1.20 (Opus 4.7).

AppDatabase, navigation, Hilt wiring. Review + Security check. Plan for at least one cycle. 4 tasks, ~$0.80.

🤝 AI + Human = Full Coverage

🤖
AI Catches (Mechanical)
Rule violations, naming drift, missing tests, dependency direction, pattern compliance.
👤
Human Catches (Judgment)
Business logic, UX implications, performance, edge cases specs missed, "there's a better way."

The human only reviews code that already passes all rule checks — minimal time, maximum impact.

📖 Key Terms

Structured Finding
A review result with five fields: rule ID, severity, location, issue, and fix. Each finding is a mini task spec.
Review-Fix Loop
Developer generates → Reviewer checks → REVISE sends findings back → Developer fixes → Reviewer re-checks. Max 2 cycles.
Acceptance Threshold
PASS = 0 MUST + ≤2 SHOULD. REVISE = any MUST or 3+ SHOULD. REJECT = structural failure.
Risk-Based Routing
Low-risk tasks skip review. Medium-risk gets standard review. High-risk gets review + security check.
Fresh-Eyes Principle
A separate AI invocation with fresh context catches ~90% of violations vs. ~60% for self-review.
PREFER Accumulation
PREFER findings don't affect verdicts, but if the same one appears across 5+ tasks, promote it to SHOULD.
Opus 4.7 for Judgment
The Reviewer uses Opus 4.7 (~$0.10/task under 2026 pricing) because judging rule intent vs. literal compliance requires deep reasoning. Cost is now near parity with Sonnet, so the choice is purely about quality.

🧪 Quick Quiz

Why does the Reviewer run on Opus 4.7 instead of Sonnet 4.6?

Writing code mechanically is Sonnet's strength. Judging whether code correctly follows rules — catching subtle violations, weighing severity, deciding pass vs. reject — requires Opus's deeper reasoning. The 2026 economics also help: at Opus 4.7's $5/$25 vs Sonnet's $3/$15, an Opus review is now near cost parity with a Sonnet review (~$0.10/task), so the choice is purely about quality.

What happens after 2 failed review-fix cycles?

The orchestrator escalates to the human. After 2 cycles, the problem is likely a bad task spec or a rule conflict — not something the Developer can fix by retrying.

Why skip reviews for low-risk tasks?

Low-risk tasks (domain entities, test fakes, DAOs) follow simple patterns where the Developer's self-verification is sufficient. Skipping saves ~$0.10 per task and a couple of minutes of wall-clock time.

What's the most common finding across RecipeVault?

Wrong dependency direction — a feature module importing another feature module. This is always MUST severity because it breaks module isolation and the build graph.

How do PREFER findings become actionable?

If the same PREFER finding appears across 5+ tasks, it signals the Developer needs a stronger constraint. Promote it to SHOULD in the rules to enforce it.

📌 Key Takeaways

  • Separate invocation: Fresh context catches ~90% of violations vs. ~60% for self-review.
  • Structured findings: Rule ID + severity + location + issue + fix = mini task specs for the Developer.
  • Three verdicts: PASS (ship it), REVISE (fix and resubmit), REJECT (escalate).
  • Severity math: 0 MUST + ≤2 SHOULD = PASS. Any MUST or 3+ SHOULD = REVISE.
  • Risk routing: Low = skip review. Medium = standard. High = review + security.
  • Max 2 cycles: If Developer can't satisfy Reviewer in 2 rounds, escalate to human.
  • ~$2.90 total (Opus 4.7): Review adds ~20–25% cost (down from ~40% on the older Opus rate card) but still catches issues that cascade through 10+ downstream tasks.
