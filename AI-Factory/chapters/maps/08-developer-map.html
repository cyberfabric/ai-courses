Chapter 8

The AI Developer: Writing Code at Scale

Memory Map
💡

The Big Idea

The Developer is the pipeline's workhorse — 20 dispatches, ~60% of total cost, running on Sonnet 4.6 as the default because it follows rules mechanically, not making judgment calls. Drop low-risk tasks to Haiku 4.5 for 3× savings. Quality depends on what context it sees and a 5-step process: Read → Plan → Write → Test → Verify.

🔄 The 5-Step Developer Process

  1. Read

    Load rules, task spec, existing code dependencies. Identify every type and import needed.

  2. Plan

    List every file to create. List classes/functions and signatures. Verify all imports resolve.

  3. Write

    Write each file completely. No TODOs, no placeholders. Follow naming conventions exactly.

  4. Test

    Write a unit test for every public class. Include happy path and edge cases.

  5. Verify

    Re-read rules. Check every file. List acceptance criteria met (✅/❌). Report issues.

Key Trick
Load interfaces, not implementations. Interfaces are 10-20 lines. Implementations are 100+. Biggest context savings.
⚠️
Priority Matters
If context fills up, skip similar code before skipping rules. A Developer that forgets rules produces inconsistent code.

📖 Key Terms

Build-Test-Fix Loop
Generate code → compile → test → pass or re-dispatch with error. Max 2 retries before escalating.
Context Loading
Prioritized strategy for what the Developer sees: rules first, then task, then dependencies, then architecture.
Completion Report
Developer's output summary: files created, tests created, criteria met (✅/❌), issues encountered.
Interfaces Over Implementations
Load 10-20 line interfaces instead of 100+ line implementations to save context budget.
Naming Drift
When different dispatches invent different names for the same concept (e.g., RecipeRepository vs RecipeRepo).
The "It Compiles" Trap
Code that compiles isn't necessarily correct. The output contract includes tests AND rule compliance, not just compilation.

🧪 Quick Quiz

Why does the Developer skill include a "Plan" step before "Write"?

Without planning, the AI starts writing code before understanding the full scope. This leads to inconsistent naming, missing imports, and files that don't fit together. Planning prevents the most common AI coding failures.

What's the most important context savings trick?

Load interfaces, not implementations. Interfaces are 10-20 lines vs 100+ for implementations. The Developer needs to know what to call, not how it's implemented.

Why is Sonnet 4.6 the default Developer model in 2026?

The Developer follows rules mechanically — it needs reasoning, but not Opus-tier judgment. Sonnet 4.6 ($3/$15) at 79.6% SWE-bench Verified handles the messy 80% of code gen. Opus 4.7's price drop made the old "Sonnet vs Opus" cost story obsolete; the new question is Sonnet vs Haiku 4.5 ($1/$5, 73.3% SWE-bench), which works well for low-risk mechanical tasks like domain entities, fakes, and DAO boilerplate.

What happens when the fix loop fails twice?

Escalate to the human. After 2 retries, the error likely needs judgment the AI can't provide — cross-module wiring, navigation bugs, or complex type conversions.

What does the Developer output contract require beyond "it compiles"?

All files created, test files for every public class, all tests pass, all architecture rules followed, acceptance criteria met, and no TODOs or placeholders.

📌 Key Takeaways

  • Pipeline workhorse: 20 dispatches, ~60% of cost. Sonnet 4.6 by default; Haiku 4.5 (3× cheaper) for mechanical low-risk tasks.
  • 5-step process: Read → Plan → Write → Test → Verify. Planning prevents naming drift and missing imports.
  • Context priority: Rules (always) → task spec → dependencies (interfaces!) → architecture → similar code.
  • Build-test-fix: Generate → compile → test → pass or re-dispatch with error. Max 2 retries.
  • Success rates: ~70% first pass, ~25% one retry, ~5% escalate to human.
  • Common fixes: Explicit imports, exact file paths, naming rules, mandatory test files.
  • Compiles ≠ correct: Output contract includes tests, rule compliance, and completion report.
