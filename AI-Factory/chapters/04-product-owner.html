Chapter 4

The AI Product Owner: Vision and Scope

Build the PO skill that turns one sentence into a product vision

The First Role Clocks In

The pipeline is configured. The execution plan is written. Now it's time to build the first actual role skill and watch it work. The Product Owner is the pipeline's starting point — it takes the one-sentence seed from Chapter 3 and transforms it into a complete product vision that the PM role can consume.

This chapter builds a production-grade PO skill file, runs it against the recipes app idea, and reviews the output at the first human checkpoint. By the end, you'll have artifacts/vision.md — the foundation for everything downstream.

The Product Owner Role

Product Owner
Inputs: One-sentence seed idea + platform constraints + target audience description
Outputs: Product Vision Document containing: app description, 3-5 user personas, MoSCoW feature list, MVP scope, release roadmap (v1/v2/v3), success metrics

The PO answers the question: “What are we building and for whom?” It doesn't decide how to build it (that's the Architect), doesn't write specs (that's the PM), and definitely doesn't write code. Its entire job is to produce a vision document specific enough for the PM role to consume without asking follow-up questions.

The Complete PO Skill File

Here's the full .claude/commands/product-owner.md skill file. This is copy-paste ready — save it and invoke it with /product-owner: 

# .claude/commands/product-owner.md

You are the **Product Owner** in the AI Factory pipeline.

## Identity
You define product vision and scope. You decide WHAT to build and
for WHOM. You do NOT decide HOW to build it (Architect's job). You
do NOT write specifications (PM's job). You do NOT write code.

## Input
You will receive:
- A **one-sentence app idea** describing the core product
- **Platform constraints** (e.g., Android, iOS, web)
- **Target audience** description

## Output
Produce a **Product Vision Document** in markdown with YAML
frontmatter. The document MUST contain all of these sections:

```yaml
---
artifact: vision
produced_by: product-owner
consumed_by: [product-manager]
version: 1
status: draft
---
```

### 1. App Description
2-3 paragraphs describing the app, its core value proposition,
and why it needs to exist.

### 2. User Personas (3-5)
For each persona:
- **Name** and demographic (e.g., "Maria, 34, working mom")
- **Goals** — what they want to achieve
- **Pain points** — what frustrates them today
- **Usage pattern** — when and how they'd use the app

### 3. Feature List (MoSCoW Prioritized)
Categorize every feature as:
- **Must Have** — MVP, ship-blocking
- **Should Have** — v1 after MVP
- **Could Have** — v2 nice-to-haves
- **Won't Have** — explicitly out of scope (for now)

### 4. MVP Scope
List ONLY the Must Have features. Define what "done" looks
like for the MVP. The MVP must be achievable by the AI
pipeline in a single run.

### 5. Release Roadmap
- **MVP** — core features, target: 2-week AI pipeline run
- **v1** — Should Have features added
- **v2** — Could Have features, polish, advanced UX

### 6. Success Metrics
3-5 measurable outcomes. Examples: "User can create a recipe
in under 2 minutes", "App works fully offline."

## Rules
- NEVER suggest specific technologies or libraries
- NEVER describe architecture patterns or code structure
- NEVER mention implementation details
- Every feature MUST have a clear user benefit
- Personas must feel like real people, not abstractions
- MoSCoW categories must be ruthlessly honest — most features
  are Should/Could, not Must
- MVP scope must be SMALL — 3-5 core features maximum
Notice the YAML frontmatter requirement. The skill tells the PO to include the handoff protocol header from Chapter 3. This ensures the PM role can verify it's consuming the right artifact type and check its status before proceeding.
Skill file location (2026): The example above uses the legacy .claude/commands/product-owner.md path, which still works. The canonical 2026 form is .claude/skills/product-owner/SKILL.md — the directory layout lets you bundle scripts, templates, and reference docs alongside the skill. Both forms create /product-owner and can be auto-loaded by Claude when relevant; the directory form is preferred for any skill that grows beyond a single markdown file. See Chapter 2 for the skill vs sub-agent distinction.

What the PO Produces

Let's run the PO skill against the recipes app seed and see what comes out. Here's a representative excerpt of the vision document:

Product Owner
Product Vision Document — RecipeVault (excerpt)
---
artifact: vision
produced_by: product-owner
consumed_by: [product-manager]
version: 1
status: draft
---

# RecipeVault — Product Vision

## 1. App Description

RecipeVault is an offline-first Android app for home cooks who want
to collect, organize, and use recipes without relying on internet
connectivity. The app solves a common frustration: recipes scattered
across bookmarks, screenshots, and paper notes with no way to
organize or search them.

The core value proposition is simple: one place for all your recipes,
organized your way, available anywhere — even without signal.

## 2. User Personas

### Maria, 34, Working Mom
- **Goals:** Quick weeknight dinners, meal planning for the family
- **Pain points:** Recipes scattered across 5 apps, can't find them
  when she's at the grocery store with no signal
- **Usage:** Evenings (planning) and weekends (cooking)

### David, 28, Hobby Cook
- **Goals:** Organize recipes by cuisine, track what he's tried
- **Pain points:** Loses recipes from YouTube videos and Reddit posts
- **Usage:** Weekends, browses and saves recipes during commute

### Sofia, 62, Grandmother
- **Goals:** Preserve family recipes digitally for grandchildren
- **Pain points:** Handwritten recipe cards are fading, no backup
- **Usage:** Occasional, enters recipes one at a time with care

## 3. Feature List (MoSCoW)

### Must Have (MVP)
- Create, edit, delete recipes with title, ingredients, steps
- Organize recipes into recipe books (collections)
- Full-text search across all recipes
- Fully offline — all data stored locally

### Should Have (v1)
- Generate grocery list from selected recipes
- Import recipe from URL (with parsing)
- Photo attachment for recipes
- Tags and filtering

### Could Have (v2)
- Share recipes with other users
- Meal planning calendar
- Cooking timer integrated with recipe steps
- Nutritional information auto-calculation

### Won't Have (this project)
- Social features (comments, likes, following)
- Cloud sync between devices
- Video recording of cooking process

User Persona Generation

Personas are the PO's secret weapon. They prevent the entire pipeline from building features nobody wants. The skill file requires personas with specific structure: name, demographics, goals, pain points, and usage patterns.

Weak Persona
User A: Wants to save recipes and find them later.
Too generic. Doesn't tell the PM what to prioritize. Every feature “helps User A.”
Strong Persona
Maria, 34, working mom: Needs weeknight dinners under 30 min. Has recipes in 5 different apps. Can't find them at the grocery store when she has no signal. Uses the app evenings (planning) and weekends (cooking).
Specific enough to make decisions. Maria drives the offline requirement, the search feature, and the grocery list feature. She's a real person the PM can write stories for.
Personas drive priorities. When the PM writes user stories, each story references a persona: “As Maria, I want to search recipes so I can find a weeknight dinner in under 10 seconds.” If a feature doesn't serve a persona, it's probably a Could Have, not a Must Have.

MoSCoW Prioritization

The PO's most critical skill is saying no. Every developer wants to build everything. Every stakeholder wants their pet feature. MoSCoW forces ruthless honesty about what's actually required for launch:

Must
Ship-blocking. Without these, the app isn't the app. For RecipeVault: CRUD, books, search, offline. That's it.
Should
Important but not blocking. v1 features. Grocery lists, import, photos, tags. Users want them, but the app works without them.
Could
Nice-to-haves. v2 territory. Sharing, meal planning, timers, nutrition. Delightful, not essential.
Won't
Explicitly out of scope. Social features, cloud sync, video. Saying “we won't build this” is as important as saying what we will.
The MVP trap: If your Must Have list has more than 5 features, your PO skill isn't being ruthless enough. Add a rule to the skill file: “MVP scope must be SMALL — 3-5 core features maximum.” The AI will naturally try to be helpful by including everything. Constrain it.

Release Roadmap

The roadmap gives the PM and Architect a phased target. The pipeline executes MVP first — all seven roles focus on Must Have features only. Should and Could features are recorded but deferred:

MVP
CRUD + Books + Search + Offline
→
v1
Grocery + Import + Photos + Tags
→
v2
Sharing + Meal Plan + Timers

The pipeline will build the MVP in a single run. After that, you can re-run the pipeline with the v1 features, and the Developer role will extend the existing codebase rather than starting from scratch.

The Vision Checkpoint

The first human checkpoint happens right here. After the PO produces artifacts/vision.md, the orchestrator pauses and shows you the document. This is where you ask:

  • Do the personas match my actual users?
  • Is the MVP scope right? Too big? Too small?
  • Are the MoSCoW priorities honest?
  • Did the PO hallucinate features I didn't ask for?
  • Are the Won't Haves truly things we should exclude?
Orchestrator → Product Owner with seed idea
PO returns: artifacts/vision.md (2,400 tokens)
CHECKPOINT: Human reviews vision document
Human approves → status changes to approved
Human rejects → orchestrator re-dispatches PO with feedback
PO dispatch: ~2 min • under $0.10 (Opus 4.7) • Ready for PM
Edit, don't re-dispatch. If the vision is 90% right, just edit artifacts/vision.md directly. Fix the persona, adjust a priority, add a missing Won't Have. Then mark it status: approved and let the PM proceed. Re-dispatching costs tokens for something you can fix in 30 seconds.

Exercise: Build and Run Your PO Skill

  1. Create the skill file. Copy the PO skill from this chapter into .claude/commands/product-owner.md in your project.
  2. Customize it for your app. If your app isn't recipes, adjust:
    • The persona structure (does your app need B2B personas? Developer personas?)
    • The MoSCoW rules (how many Must Haves are acceptable?)
    • The success metrics format (downloads? retention? latency?)
  3. Run it. Invoke /product-owner with your seed idea from Chapter 1 and constraints from Chapter 3.
  4. Review the output. Check every item in the output contract checklist above. Is anything missing? Is the MVP scope small enough?
  5. Iterate. If the output isn't right, don't re-run the skill. Edit the vision document directly. Then save it as artifacts/vision.md with status: approved.
