Chapter 1

The AI Factory: Roles, Not Prompts

The paradigm shift from “using AI” to orchestrating AI team members

What If You Could Hire a Full Team... of AI?

You know what building software really takes. Not just code — but a Product Owner who clarifies the vision, a PM who writes specs machines can follow, an Architect who decomposes modules, a Developer who writes the code, a Reviewer who enforces standards, a Security Engineer who catches vulnerabilities, and a QA Engineer who breaks everything before users do.

What if you could assemble that entire team in under a minute — each member specialized, consistent, and tireless — and run the full pipeline from a single sentence?

That's what this course builds. Not a toy demo. Not a “let me ask ChatGPT to write some code.” A reproducible AI development pipeline with formalized roles, structured handoffs, and quality gates — the same way a real team operates, except every member is an AI skill you own, version, and improve.

The Problem With Ad-Hoc Prompting

Everyone starts the same way. You open your AI tool, type a request, and hope for the best. Sometimes it's great. Sometimes it's garbage. You tweak the prompt, try again, get a different kind of garbage. Sound familiar?

The Ad-Hoc Approach
Build me an offline-first Android recipes app with Jetpack Compose, Room, Hilt, Clean Architecture, and MVI. Include recipe management, recipe books, grocery list generation, and make it production-ready.
This prompt asks one AI to be PO, PM, Architect, Developer, Reviewer, and QA — all at once. The result? A 500-line file that compiles (maybe) but has no architecture, no tests, no separation of concerns, and no way to reproduce or improve it.

The fundamental problem isn't the AI. It's asking one generalist to do seven specialists' jobs simultaneously. You wouldn't hire one person to be your entire engineering department. Why do you treat AI that way?

The core insight: Ad-hoc prompting fails at scale because it produces inconsistent results with no reproducibility, no quality gates, and no way to improve individual stages of the process.

The Shift: From Prompts to Roles

The fix is surprisingly simple: stop prompting and start hiring. Give each AI invocation a specific role with a defined contract — what it receives as input, what it must produce as output, and what quality criteria it must meet.

Ad-Hoc Prompt
Write me the data layer for a recipes app using Room with Clean Architecture.
No context about what features exist. No spec to implement against. No rules to follow. Every run produces different entity designs, different DAO signatures, different repository patterns.
Role-Based Dispatch
You are the Developer role. Your input is the Architect's module design (attached) and the architecture rules (attached). Implement the :data:recipe module. Output: compilable Kotlin files following all MUST rules. Run tests before reporting done.
Clear role. Defined inputs. Specific output contract. Quality criteria built in. Every run follows the same rules and produces compatible results.

The role-based approach isn't just “better prompting.” It's a fundamentally different mental model. You're not asking AI to do something — you're dispatching a specialized team member to execute their part of a pipeline.

Here's what each role owns:

Role Responsibility Key Output
Product Owner Turns a one-sentence idea into a product vision Vision document, personas, MoSCoW priorities
Product Manager Converts vision into machine-parseable specs Feature specs, acceptance criteria, edge cases
Architect Designs module structure and architecture rules Module graph, data models, architecture rules
Developer Writes compilable, tested code one module at a time Kotlin source files, passing unit tests
Code Reviewer Reviews code against architecture rules Structured findings with severity and fixes
Security Reviewer Checks OWASP compliance, secrets, auth patterns Security findings with remediation steps
QA Engineer Generates tests, discovers edge cases, measures coverage Test suite, coverage report, edge case matrix
Why seven roles? This mirrors a real development team. You could use fewer roles for smaller projects (Chapter 12 covers that), but starting with the full pipeline teaches you the complete pattern. Removing a role is easy once you understand what it contributes.

The Role Contract Pattern

Every role in the pipeline follows the same contract pattern. Think of it like an interface in code — each role implements the same structure but with different specialization:

  • Inputs: What the role receives from the previous stage
  • Outputs: What the role must produce for the next stage
  • Quality Criteria: What “done” means — measurable, verifiable conditions

Here's the contract for the Product Owner role — the first role in the pipeline:

Product Owner
Inputs: One-sentence app idea + platform constraints (Android, iOS, web) + target audience description
Outputs: Product vision document with: app description, 3-5 user personas, feature list with MoSCoW priorities, MVP scope, release roadmap (v1/v2/v3), success metrics

Notice how specific the output contract is. It's not “write a product vision.” It's “produce a document with exactly these sections, in this structure, ready for the PM role to consume.” The contract is what makes the pipeline reproducible.

Common mistake: Vague output contracts. If a role's output isn't specific enough for the next role to consume without guessing, your pipeline will break at the handoff. Be ruthlessly specific about what each role produces.

Why Roles Beat Prompts

This isn't just a theoretical improvement. Roles produce measurably better results than ad-hoc prompting across every dimension that matters:

Consistent
Same role + same input = same quality every time. No more “run it again and hope for better output.”
Reproducible
Skills are files. Check them into git. Share with your team. Run them on CI. The pipeline is code, not tribal knowledge.
Improvable
When the Reviewer finds too many false positives, you fix the Reviewer skill — not the entire pipeline. Each role evolves independently.
Auditable
Every artifact has a source role. You can trace any line of code back through the pipeline to the spec that required it.

Claude Code Skills as Role Implementations

So how do you actually implement a role? In Claude Code, roles become skills — markdown files that live in your project's .claude/commands/ directory. Each skill file is a complete role definition: system prompt, input expectations, output format, and quality rules. 

# .claude/commands/product-owner.md

You are the **Product Owner** for this project.

## Your Input
You will receive:
- A one-sentence app idea
- Platform constraints (e.g., Android, Kotlin, Compose)
- Target audience description

## Your Output
Produce a **Product Vision Document** in markdown with:
1. **App Description** — 2-3 paragraphs
2. **User Personas** — 3-5 personas with name, role, goals, pain points
3. **Feature List** — MoSCoW prioritized (Must/Should/Could/Won't)
4. **MVP Scope** — Features for first release only
5. **Release Roadmap** — v1, v2, v3 milestones
6. **Success Metrics** — How we measure if the app works

## Rules
- NEVER suggest technologies — that's the Architect's job
- NEVER write code — that's the Developer's job
- Focus on WHAT, not HOW
- Every feature must have a clear user benefit
- MVP must be buildable in 2 weeks by the AI pipeline
Skills vs. prompts: A prompt lives in your clipboard and dies when you close the tab. A skill lives in .claude/commands/, is version-controlled with git, and can be invoked by name: /product-owner. Skills are infrastructure. Prompts are improvisation.

When you invoke /product-owner in Claude Code, the skill file becomes the system context. The AI doesn't guess what to do — it follows the contract. Every time. For every team member who runs it.

The Agent Tool: Your Pipeline Dispatcher

Individual skills are useful, but the real power comes from orchestration — running roles in sequence, passing artifacts between them, and letting the pipeline flow without manual intervention.

Claude Code's Agent tool lets you dispatch work to specialized subagents. Think of it as a team lead assigning tasks: “Hey PO, take this idea and produce a vision doc. Hey PM, take this vision doc and write specs. Hey Architect, take these specs and design the system.”

Orchestrator → dispatches to Product Owner with input: “offline-first Android recipes app”
Product Owner returns: vision-document.md (2,400 tokens)
Orchestrator → dispatches to Product Manager with input: vision-document.md
Product Manager returns: feature-specs.md (4,100 tokens)
Orchestrator → dispatches to Architect with input: feature-specs.md
Architect returns: architecture-design.md (5,800 tokens)
Pipeline stage 1-3 complete • 12,300 tokens • ~$0.15

Each dispatch is isolated. The Product Owner doesn't see the Architect's work. The Developer doesn't see the PO's brainstorming. Each role gets exactly the context it needs — no more, no less. This prevents roles from overstepping their boundaries.

Chapter 2 dives deep into the Agent tool mechanics, plan mode, and how to build the orchestrator skill that coordinates all seven roles. For now, just understand the pattern: one orchestrator dispatches to many specialists.

Exercise: Define Your AI Team

Before we start building, think about your own project:

  1. Write a one-sentence seed idea for an app you want to build. Be specific about platform and key features. Example: “I want a cross-platform habit tracker with streak visualization and smart reminders.”
  2. For each of the 7 roles, write one sentence describing what that role would produce for your app:
    • PO: What's the vision?
    • PM: What are the key specs?
    • Architect: What's the module structure?
    • Developer: What's the first module to build?
    • Code Reviewer: What rules matter most?
    • Security Reviewer: What's the biggest security risk?
    • QA: What's the hardest thing to test?
  3. Identify one role where you think AI will struggle most. That's the role you'll want to invest the most time defining in your skill file.
