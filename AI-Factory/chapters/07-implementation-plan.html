Chapter 7

From Design to Implementation Plan

Break architecture into tasks sized for AI execution, ordered by dependency

The Blueprint Is Ready. Now What?

The Architect produced three artifacts: architecture.md, rules.md, and a set of backlog/*.md files. You have module structure, data models, and architecture rules. But if you hand the Architect's raw backlog directly to the Developer role, you'll hit problems fast.

The Architect thinks in modules. The Developer thinks in tasks. A module like :data:local might contain a Room database, three DAOs, six entity classes, and twelve mapper functions. That's too much for a single Developer dispatch — the context overflows and quality drops.

This chapter bridges the gap. You'll take the Architect's backlog and produce an implementation plan: a prioritized, dependency-ordered list of right-sized tasks that the orchestrator can dispatch one at a time.

Why a Separate Planning Step?

Implementation Planner
Inputs: Architecture document, rules file, and raw backlog from the Architect (artifacts/architecture.md, artifacts/rules.md, artifacts/backlog/*.md)
Outputs: Refined implementation plan (artifacts/plan.md) with 30+ right-sized tasks in dependency order, each with a task spec

This isn't a separate AI role — it's a refinement step you do yourself or delegate to the orchestrator. The Architect designs; the planner sizes. The key questions:

Size
Is each task small enough for one Developer dispatch? A task should produce 3-8 files, not 30.
Order
Can the Developer build this task without code that doesn't exist yet? Dependencies must be built first.
Scope
Does each task have clear acceptance criteria? The Developer needs to know when it's done.
Risk
What could go wrong? Tasks with external dependencies or complex logic need fallback strategies.

Task Decomposition Strategy

The Architect's backlog has one file per module. The planner breaks each module into tasks by layer:

📦 Module
e.g., :data:local
→
Task 1
Entities
→
Task 2
DAOs
→
Task 3
Database + Mappers

Each layer becomes a task. Within a module, layers still have dependencies: DAOs reference entities, the database class references DAOs. The decomposition follows the same inward-dependency principle as the module graph.

Too Large (Module-Level)
Task: Build :data:local
Files: RecipeEntity, IngredientEntity, StepEntity, RecipeBookEntity, RecipeDao, RecipeBookDao, SearchDao, AppDatabase, 12 mapper functions
Result: 20+ files. Developer loses coherence around file 12. Naming inconsistencies. Mapper bugs.
Too much for one dispatch. The Developer's context fills up and quality degrades in the second half of the output.
Right-Sized (Layer-Level)
Task 3a: Create Room entities (RecipeEntity, IngredientEntity, StepEntity, RecipeBookEntity)
Task 3b: Create DAOs (RecipeDao, RecipeBookDao, SearchDao)
Task 3c: Create AppDatabase + entity-to-domain mappers
Result: 4-6 files each. Focused. Coherent. Testable.
Each task is focused enough for the Developer to hold in context and produce consistent, high-quality output.

Task Granularity: The Goldilocks Zone

Tasks that are too small waste orchestration overhead. Tasks that are too large produce low-quality output. The sweet spot:

The Goldilocks Rules

  • 3-8 files per task. Fewer than 3 means you're splitting too fine — the orchestrator spends more time dispatching than the Developer spends coding. More than 8 means context overflow risk.
  • One layer, one module. Don't cross module boundaries in a single task. Don't mix data entities with UI composables.
  • Clear inputs and outputs. The Developer should know exactly what files to create, what dependencies to import, and what tests to write.
  • Testable in isolation. Each task should produce something that compiles and passes its own tests without needing later tasks.
The file-count heuristic. Count the files listed in the Architect's backlog for each module. If a module lists more than 8 files, split it into tasks of 3-6 files each, grouped by layer (entities, DAOs, ViewModels, screens).

Dependency Ordering

Tasks execute in dependency order. The orchestrator dispatches them one at a time (or in parallel where dependencies allow). The RecipeVault execution order:

Implementation Planner
Task Dependency Order — RecipeVault
## Execution Order

### Sprint 1: Foundation (tasks 1-4)
  Task 1: :core:domain — Domain entities (Recipe, Ingredient, Step, RecipeBook)
  Task 2: :core:domain — Use case interfaces + Result wrapper
  Task 3: :core:ui — Theme, colors, typography, shared composables
  Task 4: :test:shared — Test fakes, builders, assertions

### Sprint 2: Data Layer (tasks 5-8)
  Task 5: :data:local — Room entities + type converters
  Task 6: :data:local — DAOs (RecipeDao, RecipeBookDao, SearchDao)
  Task 7: :data:local — AppDatabase + entity-to-domain mappers
  Task 8: :data:repository — Repository implementations + use cases

### Sprint 3: Features (tasks 9-16, parallelizable)
  Task 9:  :feature:recipe-list — RecipeListViewModel + UiState
  Task 10: :feature:recipe-list — RecipeListScreen composable
  Task 11: :feature:recipe-detail — RecipeDetailViewModel + UiState
  Task 12: :feature:recipe-detail — RecipeDetailScreen composable
  Task 13: :feature:recipe-create — RecipeCreateViewModel + validation
  Task 14: :feature:recipe-create — RecipeCreateScreen composable
  Task 15: :feature:recipe-books — RecipeBookViewModel + UiState
  Task 16: :feature:recipe-books — RecipeBookScreen composable

### Sprint 4: Integration (tasks 17-20)
  Task 17: :app — Navigation graph + Hilt modules
  Task 18: :app — MainActivity + app-level setup
  Task 19: Integration — Wire all features together, verify navigation
  Task 20: Polish — Error states, empty states, loading states
Sprint 3 tasks are parallelizable. Once the data layer is complete (Sprint 2), all four feature modules can be built simultaneously because they share no dependencies between them. The orchestrator can dispatch tasks 9-16 in parallel if you want speed over cost.

The Task Spec Format

Each task in the plan needs a structured spec that the Developer role can consume. Here's the format the orchestrator passes to each Developer dispatch:

Implementation Planner
Task Spec — Task 6: Room DAOs
---
task: 6
module: :data:local
title: Create Room DAOs
depends_on: [5]  # Task 5 must be complete (entities exist)
sprint: 2
---

## Objective
Create the Room DAO interfaces for recipe, recipe book, and search.

## Files to Create
- data/local/src/main/kotlin/.../dao/RecipeDao.kt
- data/local/src/main/kotlin/.../dao/RecipeBookDao.kt
- data/local/src/main/kotlin/.../dao/SearchDao.kt

## Dependencies Available
- RecipeEntity, IngredientEntity, StepEntity (from Task 5)
- Domain entities (from Tasks 1-2)

## Requirements
- RecipeDao: insert, update, delete, getById, getAll (Flow)
- RecipeBookDao: insert, update, delete, getAll, getRecipesInBook
- SearchDao: searchByTitle, searchByIngredient, searchByTag
- All queries return Flow for reactive updates
- Use @Transaction for multi-table operations

## Acceptance Criteria (from specs.md)
- Search returns results for title, ingredient, and tag queries
- Deleting a book does NOT delete its recipes
- All read operations return Flow for reactive updates

## Architecture Rules to Follow
- RULE-001: No domain imports in entity files
- RULE-003: Constructor injection for all classes
- RULE-006: Use Result<T> wrapper in repository layer (not here, but
  design DAOs to support it)
The task spec is the Developer's entire world. When the orchestrator dispatches a Developer, it sends: the task spec, the architecture rules, and pointers to existing code from previous tasks. The Developer shouldn't need to read anything else.

Sprint Planning With AI

Sprints aren't time-boxed in the AI pipeline — they're dependency-boxed. A “sprint” is a group of tasks that share the same dependency level:

Sprint 1
Foundation: domain entities, use cases, UI theme, test utilities. No dependencies on anything. ~4 tasks, ~$0.40.
Sprint 2
Data layer: Room entities, DAOs, database, repositories. Depends on Sprint 1. ~4 tasks, ~$0.40.
Sprint 3
Features: all ViewModels and screens. Depends on Sprint 2. ~8 tasks, $0.80-$1.60 (parallel or sequential).
Sprint 4
Integration: navigation, Hilt wiring, polish. Depends on all above. ~4 tasks, ~$0.40.

Total: ~20 tasks, 4 sprints, ~$2.00-$2.80 for the full implementation phase. The orchestrator advances sprints automatically — no human intervention needed between sprints unless a task fails.

Risk Assessment Per Task

Not all tasks are equal. Some are straightforward (entity classes), others are risky (navigation graph wiring). The plan flags high-risk tasks so the orchestrator knows when to expect failures:

Implementation Planner
Risk Assessment (excerpt)
## Risk Assessment

### Low Risk (expect first-pass success)
- Tasks 1-2: Domain entities — pure data classes, no frameworks
- Tasks 5-6: Room entities + DAOs — well-defined patterns
- Task 4: Test fakes — mirrors domain interfaces

### Medium Risk (may need one re-dispatch)
- Tasks 9,11,13,15: ViewModels — state management complexity
- Task 8: Repositories — bridging data and domain layers
- Tasks 10,12,14,16: Screens — Compose layout decisions

### High Risk (plan for re-dispatch + reviewer loop)
- Task 7: AppDatabase — type converters, migration setup
- Task 17: Navigation graph — cross-module routing
- Task 19: Integration — wiring all modules together
Risk drives review strategy. Low-risk tasks can skip the Code Reviewer and go straight to QA. Medium-risk tasks get a standard review. High-risk tasks get a review and a security check. The orchestrator adjusts the pipeline based on risk level.
Risk also drives model selection: the Haiku 4.5 lane. Sonnet 4.6 ($3/$15 per MTok) is the default Developer model, but Haiku 4.5 ($1/$5 per MTok) is exactly 3× cheaper and handles mechanical, low-risk code generation comfortably. Route low-risk tasks — domain data classes, test fakes, UI theme tokens, simple mappers — to Haiku 4.5. For RecipeVault, that's Tasks 1, 4, and the entity-creation portions of Task 5. Sprint 1 alone drops from ~$0.40 to ~$0.13 with no quality cost when the work is straightforward enough. Keep Sonnet 4.6 for medium and high-risk tasks (ViewModels, repositories, navigation wiring) where its adaptive thinking earns its keep. The orchestrator picks the model per task based on the risk tier in your plan.

The Orchestrator's Execution Plan

The orchestrator reads the implementation plan and produces an execution plan showing exactly what to dispatch and in what order:

Sprint 1: Dispatch Developer × 4 (sequential)
Tasks 1-4: domain entities → use cases → UI theme → test fakes
Est: ~8 min, ~$0.40
Sprint 2: Dispatch Developer × 4 (sequential)
Tasks 5-8: entities → DAOs → database → repositories
Est: ~8 min, ~$0.40
Sprint 3: Dispatch Developer × 8 (4 parallel pairs)
Tasks 9-16: ViewModel + Screen for each feature
Est: ~8-16 min (parallel vs sequential), ~$0.80-$1.60
Sprint 4: Dispatch Developer × 4 (sequential)
Tasks 17-20: navigation → app setup → integration → polish
Est: ~8 min, ~$0.40
Total implementation: ~20 dispatches • ~30-45 min • ~$2.00-$2.80

After each sprint, the orchestrator runs a quick validation: does the code compile? Do existing tests pass? If not, it re-dispatches the failing task before moving to the next sprint.

With Plan vs. Without Plan

Some developers skip the planning step and let the Architect's raw backlog drive the Developer directly. Here's what happens:

No Planning Step
• Module-sized tasks overwhelm the Developer (20+ files)
• No dependency ordering — Developer tries to use classes that don't exist
• No risk assessment — high-risk tasks fail silently
• No sprint boundaries — can't tell when to validate
The Architect's backlog is a design artifact, not an execution plan.
With Planning Step
• Tasks are 3-8 files each — right-sized for AI
• Dependency order ensures every import resolves
• Risk flags tell the orchestrator when to expect failures
• Sprint boundaries provide natural validation checkpoints
10 minutes of planning saves hours of re-dispatching.

Exercise: Build Your Implementation Plan

  1. Read the Architect's backlog. Open each artifacts/backlog/*.md file. Count the files listed per module.
  2. Apply the Goldilocks rules. Any module with more than 8 files? Split it into layer-level tasks (entities, DAOs, ViewModels, screens).
  3. Order by dependency. Draw the task dependency graph. Which tasks have no dependencies (Sprint 1)? Which can run in parallel (Sprint 3)?
  4. Write task specs. For each task, write the structured spec: module, title, depends_on, files to create, requirements, acceptance criteria, relevant architecture rules.
  5. Assess risk. Flag each task as low / medium / high risk. Plan for re-dispatch on medium and high-risk tasks.
  6. Save the plan. Write artifacts/plan.md with the complete task list, sprint grouping, and execution order. This becomes the orchestrator's input for the implementation phase.
