Chapter 6

The AI Architect: Design for the Machine

Build the Architect skill that turns specs into module structure, data models, and architecture rules

The Technical Roles Take Over

The PO defined what to build. The PM specified exactly what each feature does. Now the Architect answers the most consequential question in the pipeline: “How do we structure the code so the Developer role can build it module by module?”

The Architect is unique in the pipeline. It doesn't just produce a design document — it generates the architecture rules that every Developer dispatch must follow. These rules are the guardrails that keep 10 separate Developer invocations building a coherent app instead of 10 disconnected fragments.

This is also the second human checkpoint. A wrong architecture is the most expensive mistake in the pipeline — every downstream role builds on the Architect's decisions.

The Architect Role

The Architect produces three artifacts, not one. The architecture document describes the system. The rules file constrains the Developer. The backlog gives the Developer specific tasks. This is the most complex role output in the pipeline.

The Complete Architect Skill File

Here's .claude/commands/architect.md — the full Architect skill: 

# .claude/commands/architect.md

You are the **Architect** in the AI Factory pipeline.

## Identity
You design the technical structure of the application. You decide
HOW to build it: modules, layers, data models, dependencies, and
technology choices. You do NOT write implementation code (Developer's
job). You do NOT define features (PM already did that).

## Input
Read `artifacts/specs.md`. Verify:
- YAML frontmatter has `artifact: specs` and `status: approved`
- Contains feature specifications, NFRs, and API contracts
- If status is not `approved`, STOP and report

## Output
Produce THREE artifacts:

### 1. Architecture Document (`artifacts/architecture.md`)
```yaml
---
artifact: architecture
produced_by: architect
consumed_by: [developer, code-reviewer]
version: 1
status: draft
---
```

Include these sections:
- **Module Structure**: list every module with its responsibility
- **Dependency Graph**: which modules depend on which
- **Data Models**: domain entities, database entities, DTOs, mappers
- **Technology Stack**: specific libraries and versions
- **Layer Rules**: what each layer can and cannot do

### 2. Architecture Rules (`artifacts/rules.md`)
A numbered list of rules the Developer MUST follow. Format:
- **RULE-001**: [Category] Rule description
  - ✅ Correct: example
  - ❌ Wrong: counter-example

Cover: naming, dependencies, layers, error handling, testing.

### 3. Implementation Backlog (`artifacts/backlog/`)
One markdown file per module, in dependency order:
- `01-core-domain.md`, `02-core-ui.md`, `03-data-local.md`, etc.
- Each file: module name, dependencies, files to create, acceptance
  criteria from the specs

## Rules
- Use Clean Architecture with domain/data/presentation layers
- Use the technology stack: Kotlin, Compose, Hilt, Room, Navigation
- Every module MUST have a single, clear responsibility
- Dependency arrows point INWARD: presentation → domain ← data
- NEVER put framework imports in domain layer
- Design for testability: every class gets its dependencies injected
The Architect is opinionated by design. Unlike the PO and PM who avoid technology decisions, the Architect skill explicitly names the stack: Kotlin, Compose, Hilt, Room. This is deliberate — the Developer needs concrete technology choices, not abstract patterns.

Module Decomposition

The Architect reads the four MVP features and decomposes them into modules. For RecipeVault, this produces a 10-module structure:

Architect
Module Structure — RecipeVault (excerpt)
## Module Structure

### Core Modules (shared foundation)
- `:core:domain` — Domain entities, repository interfaces, use cases
- `:core:ui` — Design system: theme, shared composables, navigation

### Data Modules (persistence layer)
- `:data:local` — Room database, DAOs, entity mappers
- `:data:repository` — Repository implementations, data source coordination

### Feature Modules (one per MVP feature)
- `:feature:recipe-list` — Recipe list screen, search, filtering
- `:feature:recipe-detail` — Recipe detail/edit screen, CRUD operations
- `:feature:recipe-books` — Book management, recipe-to-book assignment
- `:feature:recipe-create` — Recipe creation flow with validation

### App Module
- `:app` — Application class, Hilt setup, navigation graph, entry point

### Test Infrastructure
- `:test:shared` — Test fakes, builders, shared test utilities
10 modules = 10 Developer dispatches. Each module is scoped small enough for a single Developer invocation. The Architect doesn't just design the system — it designs the work units that the pipeline will execute.

Dependency Graph

Dependencies determine build order. The Developer can't build :feature:recipe-list until :core:domain exists because it depends on domain entities. The Architect makes this explicit:

Architect
Dependency Graph — RecipeVault
## Dependency Graph

:core:domain     → (no dependencies — build FIRST)
:core:ui         → :core:domain
:data:local      → :core:domain
:data:repository → :core:domain, :data:local
:feature:recipe-list   → :core:domain, :core:ui, :data:repository
:feature:recipe-detail → :core:domain, :core:ui, :data:repository
:feature:recipe-books  → :core:domain, :core:ui, :data:repository
:feature:recipe-create → :core:domain, :core:ui, :data:repository
:app → all modules
:test:shared → :core:domain

Build order: :core:domain → :core:ui + :data:local (parallel)
→ :data:repository → :feature:* (parallel) → :app

The dependency graph doubles as the implementation plan. Chapter 7 will formalize this into a task backlog, but the Architect has already defined the critical path.

Circular dependencies kill the pipeline. If module A depends on B and B depends on A, the Developer can't build either first. The Architect must ensure all dependencies point in one direction. Add this as a rule in the skill file if your domain is prone to cycles.

Data Model Design

The Architect defines data models at three levels. Each serves a different purpose and lives in a different layer:

Domain
Pure Kotlin data classes. No framework annotations. Used by use cases and shared across all features. Lives in :core:domain.
Database
Room @Entity classes with table/column annotations. Lives in :data:local. Never leaks outside data layer.
Mappers
Extension functions converting between domain and database entities. Lives in :data:local. Tested independently.
Architect
Data Models — Recipe (excerpt)
## Data Models

### Domain Entity (`:core:domain`)
Recipe(id, title, ingredients: List<Ingredient>, steps: List<Step>,
       bookId?, tags: List<String>, createdAt, updatedAt, photoUri?)

Ingredient(name, quantity: Double, unit: String)
Step(order: Int, instruction: String)

### Database Entity (`:data:local`)
RecipeEntity(@PrimaryKey id, title, bookId?, tags: String/*JSON*/,
             createdAt, updatedAt, photoUri?)
IngredientEntity(@PrimaryKey id, recipeId/*FK*/, name, quantity, unit)
StepEntity(@PrimaryKey id, recipeId/*FK*/, order, instruction)

### Mappers
RecipeEntity.toDomain(ingredients, steps) → Recipe
Recipe.toEntity() → RecipeEntity + List<IngredientEntity> + List<StepEntity>
The PM's API contracts become the Architect's data models. The shared Recipe shape from specs.md maps directly to the domain entity. The Architect adds the database layer on top. This is the artifact chain in action.

Technology Selection

The Architect makes explicit technology decisions. For RecipeVault, these choices are driven by the constraints from Chapter 3 and the NFRs from Chapter 5:

Architect's Technology Decisions
DI: Hilt — industry standard, compile-time safety
DB: Room — offline-first requires local persistence
UI: Jetpack Compose — modern declarative UI
Navigation: Navigation Compose — type-safe routes
Async: Kotlin Coroutines + Flow — reactive data layer
Testing: JUnit 5 + Turbine (Flow) + Mockk
Each choice is justified by a constraint or NFR. The Developer doesn't need to decide which library to use — the Architect already did.

Notice that these are the same technologies you already know. The Architect isn't teaching you the stack — it's documenting the stack so the Developer role uses the right tools consistently across all 10 module dispatches.

Architecture Rules: The Developer's Guardrails

This is the Architect's most important output. The rules file constrains every Developer dispatch. Without it, each Developer invocation would make its own style decisions, and the result would be 10 modules that look like they were built by 10 different teams.

Architect
Architecture Rules — RecipeVault (excerpt)
## Architecture Rules

**RULE-001**: [Layers] Domain layer has ZERO framework imports
  ✅ Correct: `data class Recipe(val id: String, val title: String)`
  ❌ Wrong: `@Entity data class Recipe(...)`

**RULE-002**: [Dependencies] Feature modules never depend on other features
  ✅ Correct: `:feature:recipe-list` → `:core:domain`
  ❌ Wrong: `:feature:recipe-list` → `:feature:recipe-detail`

**RULE-003**: [DI] All classes receive dependencies via constructor injection
  ✅ Correct: `class RecipeRepository @Inject constructor(private val dao: RecipeDao)`
  ❌ Wrong: `class RecipeRepository { val dao = RecipeDao() }`

**RULE-004**: [Naming] ViewModels: `{Feature}ViewModel`, Screens: `{Feature}Screen`
  ✅ Correct: `RecipeListViewModel`, `RecipeListScreen`
  ❌ Wrong: `RecipeVM`, `RecipeListView`

**RULE-005**: [State] UI state is a single sealed interface per screen
  ✅ Correct: `sealed interface RecipeListUiState { data class Success(...) }`
  ❌ Wrong: Multiple LiveData fields for loading/error/data

**RULE-006**: [Error] Repositories return Result<T>, ViewModels map to UiState
  ✅ Correct: `suspend fun getRecipe(id: String): Result<Recipe>`
  ❌ Wrong: `suspend fun getRecipe(id: String): Recipe // throws`

**RULE-007**: [Testing] Every ViewModel has a corresponding test file
  ✅ Correct: `RecipeListViewModelTest.kt` tests all UiState transitions
  ❌ Wrong: No tests, or tests that only check the happy path
Rules with examples are self-enforcing. The ✅ Correct / ❌ Wrong format gives the Developer and Code Reviewer concrete patterns to match against. The Reviewer checks code against these rules — if RULE-002 is violated, the Reviewer cites the rule number and sends the code back.

For more on the architecture rules pattern and how it keeps AI-generated code consistent, see AI-Ready Engineering Chapter 7.

Running the Architect Skill

The orchestrator dispatches the Architect with Opus 4.7 — this role requires deep reasoning about tradeoffs:

Orchestrator → Architect (Opus 4.7) with artifacts/specs.md
Architect reads specs → decomposes into 10 modules → generates dependency graph
Architect produces 3 artifacts:
artifacts/architecture.md (module structure + data models + tech stack)
artifacts/rules.md (15-20 numbered rules with examples)
artifacts/backlog/01-core-domain.md through 10-app.md
Architect dispatch: ~4 min • under $0.10 (Opus 4.7) • Ready for human checkpoint

The Architect is the most token-intensive planning role, but on Opus 4.7 it still runs under $0.10. The bigger cost of a wrong architecture is human time, not tokens: even re-dispatching all 10 Developer invocations would cost ~$1–2, but the days you spend reviewing the wrong code and re-aligning the team dwarf the token spend. The architecture checkpoint exists to catch problems before they cascade.

The Architecture Checkpoint

This is the second human checkpoint and arguably the most important one. A wrong vision wastes planning tokens. A wrong architecture wastes implementation tokens. Review these items:

Edit, don't re-dispatch. Same principle as the vision checkpoint. If the module structure is 90% right, fix the 10% directly in the file. Add a missing rule. Move a dependency. Rename a module. Then mark status: approved and let the Developer proceed.

The Architect Output Contract

The Developer and Code Reviewer expect exactly this from the Architect:

architecture.md
Module list with responsibilities, dependency graph, data models at all layers, technology stack with versions.
rules.md
15-20 numbered rules covering naming, layers, DI, state, error handling, and testing. Each with ✅/❌ examples.
backlog/*.md
One file per module in dependency order. Each lists files to create, dependencies to import, and acceptance criteria.
The rules gap. The most common Architect failure: producing a beautiful architecture document but weak rules. Without strong rules, the Developer generates inconsistent code across modules, and the Code Reviewer has nothing concrete to check against. Rules are the Architect's primary contribution to code quality.

Why the Architect Uses Opus 4.7

The Architect is one of three Opus 4.7 roles (with Code Reviewer and Security Reviewer). Sonnet 4.6 follows instructions well but tends toward generic output: module names like :data, textbook rules, oversized backlog tasks. Opus 4.7 reasons about tradeoffs and makes project-specific decisions: :feature:recipe-books, domain-tailored rules, tasks scoped to a single Developer dispatch.

The cost gap between the two used to be the deciding factor. With Opus 4.7 at $5/$25 and Sonnet 4.6 at $3/$15, Sonnet is still ~40% cheaper per token, but the dramatic gap from older Opus generations (3× the price) has shrunk to a much smaller margin. For a role this leverage-heavy — where every decision constrains 10 Developer dispatches downstream — paying a few extra cents for Opus 4.7's judgment is the right tradeoff. Save Sonnet for roles that follow rules; spend Opus 4.7 on roles that set them.

Exercise: Build and Run Your Architect Skill

  1. Create the skill file. Copy the Architect skill from this chapter into .claude/commands/architect.md.
  2. Run it. Make sure artifacts/specs.md is approved. Invoke /architect.
  3. Count the rules. Did the Architect produce at least 10 numbered rules? Do they all have ✅/❌ examples? If not, add to the skill: “Generate at least 15 rules, each with a correct and incorrect example.”
  4. Check the dependency graph. Draw it out. Are there any circular dependencies? Can you identify which modules can be built in parallel?
  5. Verify backlog files. One per module, in dependency order, each referencing acceptance criteria from specs. If backlog tasks are too large (more than 10 files to create), split them.
  6. Customize for your stack. If you're not building Android, replace the technology stack in the skill's Rules section. The pattern works for any platform — the rules just name different frameworks.
