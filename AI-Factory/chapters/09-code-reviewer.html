Chapter 9

The AI Code Reviewer: Quality at Machine Speed

Build the Reviewer skill that checks every Developer output against architecture rules

The Quality Gate

The Developer finished Task 1 — four domain entity files, four test files, a completion report claiming all acceptance criteria are met. Do you trust it?

You shouldn't. The Developer follows rules mechanically, but it can't judge its own work. It can miss a naming drift it doesn't notice, forget a rule it loaded but didn't apply, or produce code that compiles but violates the dependency graph. The Developer's "it compiles and tests pass" is necessary but not sufficient.

The Code Reviewer is the pipeline's quality gate — a separate AI invocation that loads the same architecture rules, reads the Developer's output, and produces structured findings with rule IDs, severity levels, and specific fix suggestions. It's the automated first pass before human eyes see anything.

For more on the human side of reviewing AI-generated code, see Reviewing AI Code For Dummies. This chapter automates the first pass.

The Code Reviewer Role

Code Reviewer
Input: Developer output (source files + test files), rules.md, architecture.md, task spec, acceptance criteria
Output: Structured review with findings (rule ID + severity + location + explanation + fix), overall verdict (PASS / REVISE / REJECT), summary of issues found
Model: Opus 4.7 — judgment calls require deep reasoning about rule intent vs. literal compliance

Notice: the Reviewer runs on Opus 4.7, not Sonnet. Writing code mechanically is Sonnet's strength. Judging whether code correctly follows rules — catching subtle violations, weighing severity, deciding pass vs. reject — requires the deeper reasoning Opus provides. The 2026 economic argument has flipped: with Opus 4.7 at $5/$25 per MTok and Sonnet 4.6 at $3/$15, an Opus review is now roughly at cost parity with a Sonnet review (~$0.10/task once Opus 4.7's ~35% larger tokenizer is factored in). The decision is purely about quality, not cost — and on judgment-heavy review work, Opus 4.7 wins.

Why Not Let the Developer Self-Review?

Self-Review (Same Context)

  • Same biases that wrote the code judge the code
  • "Verify" step at the end of the 5-step process
  • Catches ~60% of issues at best
  • Can't detect its own blind spots
  • No structured output — just a checklist

Separate Reviewer (Fresh Context)

  • Fresh context — no anchoring to the code it wrote
  • Loads rules as primary context, code as secondary
  • Catches ~90% of rule violations
  • Structured findings with severity and fix suggestions
  • Clear verdict: PASS, REVISE, or REJECT
The fresh-eyes principle: The Developer's "Verify" step (Step 5 in its process) is like a programmer re-reading their own PR. The Code Reviewer is like a teammate reviewing it. Fresh context catches what self-review misses.

The Code Reviewer Skill

The complete skill file lives at .claude/commands/review-code.md. Here's the full implementation:

Skill File — Code Reviewer 
# Identity
You are the Code Reviewer for the AI Factory pipeline.
Check Developer output against architecture rules and produce
structured findings. Model: Opus.

# Input
1. rules.md — architecture rules with ✅/❌ examples
2. Task spec dispatched to the Developer
3. All source + test files the Developer created
4. architecture.md (summary) for structural context

# Review Process
1. LOAD rules.md. Every rule is a checklist item.
2. READ each source file — check naming, imports, layer
   boundaries, patterns (sealed UiState, @Inject, Result<T>),
   and completeness (no TODOs or placeholders).
3. READ each test file — check coverage (test per public class),
   assertions (real, not "does not throw"), naming.
4. CHECK acceptance criteria from task spec — all must be met.
5. PRODUCE structured findings (see Output Format).

# Severity Levels
- MUST: Breaks pipeline or other modules (wrong package,
  missing @Inject, circular dependency).
- SHOULD: Degrades quality but won't block (naming drift,
  missing edge case test).
- PREFER: Style suggestion (parameter ordering, import order).

# Verdicts
- PASS: 0 MUST, ≤2 SHOULD. Ship it.
- REVISE: Any MUST OR >2 SHOULD. Send back to Developer.
- REJECT: Fundamental structural problem. Escalate.

# Output Format
## Review: Task [N] — [Title]
### Verdict: [PASS | REVISE | REJECT]
### Findings
- **[MUST|SHOULD|PREFER]-[NNN]**: [Summary]
  - Rule: RULE-NNN | File: path.kt | Line: N
  - Issue: [What's wrong]
  - Fix: [Specific change]
### Summary
- Files/Tests reviewed: N/N
- Findings: N MUST, N SHOULD, N PREFER
- Acceptance criteria: N/N met

Structured Review Output

The Reviewer doesn't write prose. It produces machine-parseable findings that the orchestrator can act on. Each finding has five fields:

Each finding has five fields: rule ID (which RULE-NNN was violated), severity (MUST/SHOULD/PREFER), location (file + line), issue (what's wrong), and fix (specific change). When findings go back to the Developer, each one is a mini task spec — no vague "improve error handling."

The Severity Math

PASS = 0 MUST + at most 2 SHOULD.
REVISE = any MUST, or 3+ SHOULD.
REJECT = structural failure (wrong module, wrong pattern entirely).
PREFER findings are logged but never affect the verdict.

Live Review: Task 1 Domain Entities

The Developer generated four domain entities (Recipe, Ingredient, Step, RecipeBook) with four test files. Let's run the Reviewer.

Review Output — Task 1 
## Review: Task 1 — Domain Entities
### Verdict: REVISE

### Findings
- **MUST-001**: Recipe uses java.util.UUID instead of String for ID
  - Rule: RULE-001 | File: core/domain/model/Recipe.kt | Line: 8
  - Issue: java.util.UUID creates platform coupling. Domain
    entities must use only Kotlin stdlib types.
  - Fix: Change `val id: UUID` to `val id: String`

- **SHOULD-001**: Ingredient.quantity uses Double (precision issues)
  - Rule: RULE-001 | File: core/domain/model/Ingredient.kt | Line: 5
  - Issue: 0.1 + 0.2 ≠ 0.3. Kitchen measurements need exact values.
  - Fix: Use `val quantity: String` with a display formatter.

- **SHOULD-002**: RecipeBook allows empty name (missing validation)
  - Rule: RULE-007 | File: core/domain/model/RecipeBook.kt | Line: 3-7

### Summary
- Files/Tests reviewed: 4/4
- Findings: 1 MUST, 2 SHOULD, 1 PREFER
- Acceptance criteria: 3/4 met

Review-Fix Loop — Task 1

Reviewer REVISE: 1 MUST, 2 SHOULD, 1 PREFER ~$0.10 • 1 min
Developer Fixed UUID → String, added init block, renamed tests ~$0.10 • 1 min

Total Task 1 complete (1 review-fix cycle) ~$0.40 • ~5 min
Max 2 review-fix cycles. If the Developer can't satisfy the Reviewer after two rounds, the orchestrator escalates to the human. After 2 cycles, the problem is likely a bad task spec or a rule conflict — not something the Developer can fix by retrying.

The Acceptance Threshold

Not every finding needs fixing before the task can advance. The Reviewer uses a graduated acceptance model:

Verdict Criteria Action
PASS 0 MUST, ≤2 SHOULD Advance to next task. Log PREFER findings for future reference.
REVISE Any MUST, or 3+ SHOULD Re-dispatch Developer with findings. Developer fixes and resubmits.
REJECT Structural failure Don't re-dispatch. The task spec or architecture needs revision. Escalate.
PREFER findings accumulate. They don't affect any single verdict, but if the same PREFER finding appears across 5+ tasks, promote it to a SHOULD in the rules. That's a signal the Developer needs a stronger constraint.

The Top 10 Findings

After reviewing 20 tasks for RecipeVault, certain findings appear repeatedly. Knowing them in advance helps you write better rules and task specs:

MUST Severity (always)

  • #1 Wrong dependency direction — feature imports another feature
  • #2 Missing @Inject annotation on constructor
  • #3 Domain entity with framework import (Room, Hilt)
  • #4 Repository returning raw type instead of Result<T>
  • #5 ViewModel exposing MutableStateFlow (should be StateFlow)

SHOULD Severity (promoted to MUST if recurring)

  • #6 Naming drift — RecipeRepo vs RecipeRepository
  • #7 Test with no assertions — just "does not throw"
  • #8 Missing sealed interface for UiState
  • #9 Wildcard imports instead of explicit
  • #10 Missing error state in sealed UiState hierarchy

Risk-Based Review Routing

Not every task needs the full Reviewer treatment. The implementation plan assigns a risk level to each task, and the orchestrator uses it to decide the review path:

Low Risk
Domain entities, test fakes, Room entities, DAOs. Skip the Reviewer — go straight to QA. These tasks follow simple patterns where the Developer's self-verification (Step 5) is sufficient. Saves ~$0.10 per task and a couple of minutes of wall-clock time.
Medium Risk
ViewModels, repositories, Compose screens. Standard review loop. Reviewer checks all rules, returns structured findings. Most tasks (12 of 20) fall in this category.
High Risk
AppDatabase, navigation graph, Hilt wiring, integration. Reviewer + Security check (Chapter 10). Cross-module wiring has the highest failure rate. Plan for at least one review-fix cycle.

Review Cost Budget (Opus 4.7)

Low risk (4 tasks × $0): $0.00
Medium risk (12 tasks × $0.10): $1.20
High risk (4 tasks × $0.20): $0.80
Review-fix retries (~6 × $0.15): $0.90
Total review budget: ~$2.90 (about 20–25% of the pipeline cost)

Recomputed under Opus 4.7's $5/$25 per MTok pricing (down 3× from the Opus 4/4.1 era). The old budget of ~$7.10 was based on the older Opus rate card.

AI Reviewer + Human = Full Coverage

AI Catches (Mechanical)

  • Rule violations, naming drift
  • Missing test coverage and annotations
  • Dependency direction violations
  • Pattern compliance (sealed UiState, Result<T>)

Human Catches (Judgment)

  • Business logic correctness
  • Performance and UX implications
  • Edge cases the spec missed
  • "This works but there's a better way"

The AI handles mechanical checks, the human handles judgment. The human only reviews code that already passes all rule checks — minimal time, maximum impact.

Exercise: Customize the Reviewer Skill

Build Your Own Review Checklist

  1. Copy the review-code.md skill file into your project's .claude/commands/ directory.
  2. Add a domain-specific check with ✅/❌ examples: decimal precision for money, timezone handling for dates, input sanitization for user-facing fields.
  3. Customize severity: define what MUST vs. SHOULD means for your domain. In a healthcare app, missing validation might always be MUST.
  4. Run the Reviewer against code from a Developer dispatch. Compare AI findings to what you'd catch manually. Tune the acceptance threshold if too lenient or strict.

The Code Reviewer Output Contract

  • Every architecture rule checked against every file
  • Every acceptance criterion verified (met or not met)
  • Every finding has: rule ID, severity, file, line, issue, fix
  • Verdict is one of: PASS, REVISE, REJECT
  • Summary includes counts of findings by severity
  • PREFER findings logged but don't affect verdict
  • No false positives — every finding cites a specific rule
