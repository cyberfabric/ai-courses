Chapter 2

Orchestration With Claude Code

The technical foundation for multi-role AI pipelines

From Concept to Machinery

Chapter 1 introduced the idea: roles, not prompts. Seven specialized AI team members, each with a contract, assembled into a pipeline. Great concept. But how does it actually work?

This chapter is the technical foundation. You'll learn the three mechanisms that make multi-role orchestration possible in Claude Code: the Agent tool for dispatching to subagents, plan mode for structured thinking before action, and skill files as the building blocks that encode each role. By the end, you'll have a working orchestrator skill that coordinates the entire pipeline.

The Agent Tool: Dispatching to Specialists

The Agent tool is the core mechanism for orchestration. It lets you spawn a subagent — an isolated Claude instance with its own context, tools, and instructions — that executes a specific task and returns a result.

Think of it as a function call, except the function is an AI with full coding capabilities: 

Agent({
  description: "Generate product vision",
  prompt: `You are the Product Owner role.

Input: "${seedIdea}"
Platform: Android (Kotlin, Compose, Room, Hilt)
Audience: Home cooks who want to organize recipes

Produce a Product Vision Document with:
1. App Description (2-3 paragraphs)
2. User Personas (3-5)
3. Feature List (MoSCoW prioritized)
4. MVP Scope
5. Release Roadmap
6. Success Metrics`,
  subagent_type: "general-purpose"
})

The subagent runs in isolation. It doesn't see the orchestrator's conversation history. It doesn't see other roles' outputs unless you explicitly pass them in the prompt. This isolation is a feature, not a limitation — it prevents roles from overstepping their contracts.

Key insight: The Agent tool's prompt parameter is where you embed the role contract. Everything the subagent needs — role identity, inputs, output format, rules — goes in this prompt. The subagent has access to all tools (Read, Write, Edit, Bash, Grep, Glob) so it can read existing code, write files, and run commands.
Built-in vs custom sub-agents: subagent_type: "general-purpose" works for one-off dispatches. For roles you'll re-use, define a custom sub-agent in .claude/agents/<role>.md with YAML frontmatter (name, description, tools, model) followed by the system prompt — then dispatch with subagent_type: "<role>". Custom sub-agents are the 2026 canonical pattern for reusable role definitions; this chapter uses general-purpose in inline examples to keep them self-contained.

What the Agent Tool Returns

When a subagent finishes, it returns a single text result to the orchestrator. This is the artifact — the role's output. The orchestrator captures it and passes it to the next role:

Orchestrator → Agent(prompt: "You are the PO role...")
Agent returns: Product Vision Document (2,400 tokens)
Orchestrator → Agent(prompt: "You are the PM role. Here is the vision: ${poResult}...")
Agent returns: Feature Specifications (4,100 tokens)
Orchestrator → Agent(prompt: "You are the Architect role. Here are the specs: ${pmResult}...")
Agent returns: Architecture Design (5,800 tokens)
3 Agent dispatches • 12,300 output tokens • ~45 seconds • ~$0.10 (Opus 4.7)

Notice the pattern: each dispatch includes the previous role's output as context. This is the artifact chain — the pipeline's data flow.

Plan Mode: Think Before You Build

Before the orchestrator dispatches any role, it should plan. Claude Code's plan mode forces structured thinking: analyze the request, design the approach, then execute. For an orchestrator, planning means deciding:

  • Which roles to invoke and in what order
  • What context each role needs
  • Where human oversight checkpoints go
  • What happens if a role fails

The orchestrator skill uses plan mode as its first step. Before spawning any subagents, it writes a pipeline execution plan:

Orchestrator
Pipeline Execution Plan
## Pipeline Plan for: Recipes App

### Phase 1: Discovery
1. Product Owner → vision document
   - Input: seed idea + constraints
   - Human review: YES (approve vision before specs)

2. Product Manager → feature specs
   - Input: approved vision document
   - Human review: NO (proceed to architecture)

### Phase 2: Design
3. Architect → architecture design + rules
   - Input: feature specs
   - Human review: YES (approve architecture before coding)

### Phase 3: Implementation
4. Developer → code (per module, 10 iterations)
   - Input: architecture + rules + task spec
   - Human review: NO (reviewer handles quality)

### Phase 4: Quality
5. Code Reviewer → findings
   - Input: generated code + architecture rules
   - If FAIL: re-dispatch Developer with findings

6. Security Reviewer → security findings
   - Input: generated code
   - If CRITICAL: halt pipeline, require human

7. QA Engineer → test suite
   - Input: code + specs + acceptance criteria

### Estimated: 15-20 Agent dispatches, ~$2-5 total
Don't skip planning. Without a plan, the orchestrator will make ad-hoc decisions about role ordering, context passing, and error handling. That's exactly the problem we're trying to solve. The orchestrator should be as disciplined as the roles it dispatches.

Skill Files: Anatomy of a Role

In Chapter 1, you saw a basic skill file for the Product Owner. Now let's look at the full anatomy of a production-grade skill that embodies a role: 

# .claude/commands/architect.md

You are the **Architect** role in the AI Factory pipeline.

## Identity
You design software systems. You do NOT write code (that's the
Developer's job). You do NOT write specs (that's the PM's job).

## Input
A **Feature Specifications** document from the PM role containing:
feature list, acceptance criteria, user stories, NFRs.

## Output
Produce an **Architecture Design Document** with:

### 1. Module Structure
- Every Gradle module (e.g., `:core:domain`, `:data:recipe`)
- Module dependencies and layer assignments

### 2. Data Models
- Domain entities (pure Kotlin, no framework deps)
- Room entities, DTOs, mapper interfaces

### 3. Architecture Rules (BOTH formats)
- Prose format for CLAUDE.md
- Structured checklist for the Reviewer role

### 4. Technology Decisions
- Justify each library choice with alternatives considered

## Rules
- Domain layer MUST have zero Android framework dependencies
- Data layer MUST NOT expose Room entities to domain
- All cross-module communication through interfaces
- NEVER produce code — only design documents and rules

Notice the structure: Identity (who you are and aren't), Input (what you receive), Output (what you produce with exact sections), and Rules (boundaries you must not cross).

Architect (Skill Anatomy)
Identity: "You design systems. You do NOT write code."
Input: Feature specs from PM role
Rules: Scope boundaries the role must never cross
Output: Architecture design with exact section structure
Format: Markdown with module list, data models, rules, decisions
The "NOT" pattern: Every skill should explicitly state what the role does not do. "You do NOT write code" prevents the Architect from generating Kotlin files. "You do NOT suggest technologies" keeps the PO focused on the product. Boundaries prevent role bleeding.
Skills, sub-agents, slash commands — what's what (2026): .claude/skills/<name>/SKILL.md and .claude/commands/<name>.md are the same thing now — both create /<name> and can be auto-loaded by Claude when relevant. .claude/agents/<name>.md is different: a sub-agent is delegated to via the Agent tool with its own context window. In this course, each role can be authored as a skill (user invokes it, or Claude loads it when relevant) or as a sub-agent (the orchestrator dispatches it). The orchestrator itself is best as a skill; the seven roles work well as sub-agents because each runs in isolation.

The Orchestrator Skill Pattern

The orchestrator is itself a skill — a master coordinator that dispatches to the other role skills. It's the team lead that assigns work, collects results, and decides what happens next. 

# .claude/commands/orchestrate-pipeline.md

You are the **Pipeline Orchestrator**. Coordinate the AI Factory
pipeline from idea to tested code.

## Input
- A one-sentence app idea
- Platform and technology constraints

## Execution Protocol

### Step 1: Plan
Create a Pipeline Execution Plan in `pipeline-plan.md`.

### Step 2: Discovery
1. **Product Owner** → save to `artifacts/vision.md`
   - CHECKPOINT: Show vision to user. Wait for approval.
2. **Product Manager** → save to `artifacts/specs.md`

### Step 3: Design
3. **Architect** → save to `artifacts/architecture.md`
   - CHECKPOINT: Show architecture to user. Wait for approval.

### Step 4: Implementation
4. **Developer** → code in `src/`, one module at a time
   - If code doesn't compile: re-dispatch with errors

### Step 5: Quality
5. **Code Reviewer** → if MUST-fix: re-dispatch Developer
6. **Security Reviewer** → if CRITICAL: HALT, notify user
7. **QA Engineer** → tests in `src/test/`

## Rules
- NEVER skip a human checkpoint
- ALWAYS save artifacts to files (not just in memory)
- Max retries per role: 2. Then ask the human.
Why save artifacts to files? Two reasons. First, files survive if the pipeline is interrupted — you can resume from the last saved artifact. Second, files let you inspect and edit artifacts between roles. If the PO's vision is 90% right, you fix it in the file and let the PM role consume the corrected version.

Passing Context Between Roles

The artifact chain is the pipeline's data flow. Each role reads the previous role's artifact and produces its own. But how do you actually pass artifacts between Agent dispatches?

There are two strategies. Inline: paste the artifact into the Agent prompt. Simple but fragile — bloats the prompt and can't be edited between runs. File reference: tell the subagent to read the artifact from disk. This is the recommended approach:

File Reference (Recommended)
Read artifacts/architecture.md for the module design. Read artifacts/rules.md for architecture rules. Implement task-03 in :data:recipe module.
Subagent reads files itself. Works with any artifact size. Artifacts are editable between dispatches. Subagent sees the latest version.
Inline Embedding (Fragile)
Here is the architecture: [4,000 tokens of pasted content]. Here are the rules: [3,000 tokens of pasted content]. Now implement task-03.
Bloats the prompt. Can't be edited between runs. Risk of hitting context limits. Content is frozen at dispatch time.
The file-based handoff pattern: Save artifacts to artifacts/ directory. Each role reads what it needs. The orchestrator doesn't need to manage artifact content — just file paths.

Error Handling and Re-Dispatch

Roles fail. The Developer produces code that doesn't compile. The Reviewer finds MUST-fix violations. The Architect forgets a module. When a role fails, the orchestrator needs a strategy:

  1. Detect: Check output against the role's contract. All sections present? Code compiles? Review passed?
  2. Diagnose: Include failure details in the re-dispatch: "Your output had these issues: [specifics]."
  3. Re-dispatch: Send the role back with original input PLUS failure feedback.
  4. Escalate: After 2 retries, stop and ask the human.

The Developer-Reviewer loop is the most common re-dispatch pattern:

Orchestrator → Developer: "Implement :data:recipe module"
Developer returns: 12 Kotlin files written
Orchestrator → Code Reviewer: "Review :data:recipe"
Reviewer returns: FAIL — 3 MUST-fix findings
Orchestrator → Developer: "Fix these 3 findings: [details]"
Developer returns: 3 files updated
Orchestrator → Code Reviewer: "Re-review :data:recipe"
Reviewer returns: PASS — 0 MUST-fix, 2 SHOULD-fix (accepted)
4 dispatches • Developer-Reviewer loop converged in 1 iteration

The Role Registry

The orchestrator needs to know which skill handles which role. The role registry is a simple mapping, defined at the top of the orchestrator skill: 

## Role Registry
| Role               | Skill / Sub-agent File   | Model        | Cheaper fallback                          |
|--------------------|--------------------------|--------------|-------------------------------------------|
| Product Owner      | /product-owner           | opus-4.7     | sonnet-4.6 (now ~equivalent cost)         |
| Product Manager    | /product-manager         | opus-4.7     | sonnet-4.6                                |
| Architect          | /architect               | opus-4.7     | none (critical judgment)                  |
| Developer          | /developer               | sonnet-4.6   | haiku-4.5 (3× cheaper, mechanical tasks) |
| Code Reviewer      | /code-reviewer           | opus-4.7     | sonnet-4.6                                |
| Security Reviewer  | /security-reviewer       | opus-4.7     | none (critical judgment)                  |
| QA Engineer        | /qa-engineer             | sonnet-4.6   | haiku-4.5 (high-volume test generation)   |
Opus 4.7
For roles requiring deep judgment: Architect, Code Reviewer, Security. Opus 4.7's price drop ($5/$25 per MTok) made it economical for everyday review work, not just high-stakes design.
Sonnet 4.6
For roles following clear rules: Developer, QA. Sonnet's edge over Opus is now mostly speed; for the cheapest mechanical work, Haiku 4.5 ($1/$5 per MTok) is the better lane.
Cost optimization: Not every role needs Opus. The Developer follows explicit rules — Sonnet handles this well and is faster, and Haiku 4.5 handles the most mechanical tasks for ~3× less. The Architect makes design decisions with ambiguous tradeoffs — Opus territory, but the price gap to Sonnet has collapsed. Chapter 12 covers cost optimization in detail.

Human Oversight Insertion Points

A fully autonomous pipeline sounds exciting, but it's dangerous. Some decisions need human judgment. The orchestrator skill defines explicit checkpoints where the pipeline pauses and waits for human approval:

Checkpoint After Role Why
Vision Review Product Owner Wrong vision = wrong everything downstream. Catch it early.
Architecture Review Architect Module structure is expensive to change later. Validate before coding.
Security Escalation Security Reviewer CRITICAL findings may need domain expertise the AI doesn't have.

Notice there's no checkpoint after the Developer role. That's intentional — the Code Reviewer handles quality automatically. Humans only step in where AI judgment is unreliable: product direction, architectural tradeoffs, and security risk assessment.

Rule of thumb: Insert checkpoints where a wrong decision is expensive to reverse. Vision and architecture affect everything downstream. A coding bug is cheap to fix. A wrong module structure requires rewriting the whole app.

Exercise: Build Your First Orchestrator Skill

Create a minimal orchestrator skill that coordinates just the first two roles:

  1. Create .claude/commands/mini-orchestrator.md in your project
  2. Define the orchestrator identity: "You coordinate the discovery phase of the AI Factory pipeline."
  3. Add the execution protocol:
    • Step 1: Dispatch to PO role with the seed idea
    • Step 2: Save PO output to artifacts/vision.md
    • Step 3: Show vision to user, wait for approval
    • Step 4: Dispatch to PM role with the approved vision
    • Step 5: Save PM output to artifacts/specs.md
  4. Add error handling: if either role produces empty output, retry once.
  5. Test it by running /mini-orchestrator with your seed idea from Chapter 1's exercise.
