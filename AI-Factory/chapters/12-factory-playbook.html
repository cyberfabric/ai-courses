Your AI Factory Playbook

Adapt, customize, optimize, and share the pipeline

The Factory Is Yours

You built a 7-role pipeline that turns a single sentence into a reviewed, secured, tested Android app in 90 minutes for $9. But RecipeVault was just the demo. The real value is the factory itself — the skills, the contracts, the orchestration patterns.

This chapter makes the factory portable. You'll learn to adapt it to other platforms, add custom roles, remove roles you don't need, optimize costs, and share the pipeline with your team. By the end, you'll have a complete playbook for running your own AI development factory.

Adapting to Other Platforms

The pipeline's structure is platform-agnostic. The 7 roles, the 4 phases, the 3 checkpoints — none of these change. What changes are the Architect's rules, the Developer's technology context, and the Security Reviewer's checklist.

Component Android (RecipeVault) iOS (SwiftUI) Web (Next.js)
Architect rules Hilt, Room, Compose, sealed UiState SwiftData, MVVM, @Observable, NavigationStack App Router, Server Components, Prisma, Zustand
Developer context Kotlin, Gradle, Android SDK Swift, SPM, UIKit/SwiftUI TypeScript, npm, React
Security checks OWASP Mobile, SQLCipher, Keystore OWASP Mobile, Keychain, App Transport Security OWASP Web, CSP, CSRF, XSS
Unchanged PO, PM, Planner, Code Reviewer structure, QA strategy, Orchestrator
To adapt the pipeline to a new platform: Copy all 7 skill files. Modify the Architect skill's rules section, the Developer skill's technology context, and the Security Reviewer's checklist. Everything else stays the same.
Compose Multiplatform — one Architect, two platforms: Compose Multiplatform 1.8.0 (May 2025) made Compose stable for iOS. Real-world apps now share 80–96% of UI code across Android and iOS, with ~9 MB added to iOS binaries. For the Factory, this means the Architect can produce a single Compose Multiplatform stack and the Developer skill stays Kotlin-bound across both platforms. You skip the duplicate iOS skill set entirely and only branch out for genuinely platform-specific concerns (Keychain vs Keystore, ATS vs Network Security Config). Treat Compose Multiplatform as a fourth column in the platform table when both mobile platforms are in scope.

Adding Custom Roles

The 7-role pipeline covers the core development workflow. But some projects need specialized roles that don't fit neatly into the existing seven:

Custom Role Template
Input: What artifacts from previous roles does this role consume?
Output: What artifact does this role produce? Who consumes it next?
Model: Does this role need judgment (Opus) or follow patterns (Sonnet)?
Designer
Input: PO vision. Output: wireframes, color scheme, component specs. Runs after PM, before Architect.
DevOps
Input: architecture.md. Output: CI/CD config, Dockerfile, deploy scripts. Runs after Architect.
Tech Writer
Input: specs + code. Output: API docs, README, user guide. Runs after QA.
Perf Engineer
Input: code + profiling data. Output: optimization plan, benchmark results. Runs after QA.
Skill File — Tech Writer (example) 
# Identity
You are the Technical Writer for the AI Factory pipeline.
Generate documentation from code and specs. Model: Sonnet.

# Input
1. specs.md — feature descriptions and acceptance criteria
2. architecture.md — module structure and data models
3. Source code — public APIs and key classes
4. rules.md — naming conventions for consistency

# Output
1. README.md — project overview, setup, architecture summary
2. API.md — public interfaces with usage examples
3. CONTRIBUTING.md — how to add features following the pipeline

# Process
1. READ specs for feature descriptions.
2. READ architecture for module overview.
3. SCAN source code for public classes and functions.
4. GENERATE README with setup instructions from build.gradle.
5. GENERATE API docs with one example per public interface.
6. GENERATE contributing guide referencing the pipeline roles.

Removing or Combining Roles

Not every project needs all 7 roles. Smaller projects can combine or skip roles:

Solo Dev
3 roles: PO + Architect + Developer. Skip PM (you know your specs), skip Reviewer (you review your own code), skip Security and QA (manual testing is fine for a personal project). Cost: ~$2-3.
Small Team
5 roles: PO + PM + Architect + Developer + Code Reviewer. Skip Security Reviewer (team does manual security review). Skip QA Engineer (developers write their own thorough tests). Cost: ~$5-7.
Enterprise
7+ roles: all 7 plus custom roles. Add Designer, DevOps, Tech Writer, Performance Engineer. Full quality stack. Full documentation. Cost: ~$12-18.
Don't skip the Architect. Every other role can be removed or combined. But without the Architect, you lose architecture rules — and without rules, 20 Developer dispatches produce 20 inconsistent modules. The Architect is the one non-negotiable role.

Cost & Time Optimization

The RecipeVault pipeline costs ~$6.50 under Opus 4.7 + Sonnet 4.6 pricing (down from ~$9.20 on the Opus 4/4.1 rate card). With Opus's 3× price drop, the model-assignment decision is now primarily about judgment vs mechanical work, not cost. The bigger lever is wall-clock time, not dollars.

Use Opus 4.7 For (judgment)

  • Architect — design decisions and tradeoffs
  • Code Reviewer — rule interpretation and severity
  • Security Reviewer — vulnerability assessment
  • Orchestrator — advance/re-dispatch/escalate

Use Sonnet 4.6 / Haiku 4.5 For (mechanical)

  • Product Owner — follows template to generate vision (Sonnet)
  • Product Manager — follows template to write specs (Sonnet)
  • Developer — follows rules to write code (Sonnet, or Haiku 4.5 for low-risk tasks)
  • QA Engineer — follows patterns to generate tests (Sonnet, or Haiku 4.5 for high-volume runs)
  • Planner — follows dependency order (Sonnet)
The 2026 axis is judgment vs mechanical, not cost. Opus 4.7 ($5/$25 per MTok) is roughly at parity with Sonnet 4.6 ($3/$15) once you factor in Opus 4.7's larger tokenizer. Pick Opus where you need genuine reasoning; Sonnet where you need pattern-following at scale; Haiku 4.5 ($1/$5) where the task is high-volume and mechanical.

Time Levers (Opus 4.7 era)

  • Skip low-risk reviews: Save ~$0.30 (down from ~$1.00 on old Opus pricing) AND ~3 minutes of wall-clock time per task by skipping Code Reviewer on domain entities, test fakes, and DAOs. The wall-clock saving is now the bigger win.
  • Batch QA: Run QA per sprint instead of per task. Fewer dispatches, same coverage. Save ~$0.80 (Sonnet pricing unchanged) plus ~5 minutes of orchestration overhead.
  • Route mechanical work to Haiku 4.5: 3× cheaper than Sonnet, 4–5× faster on typical workloads, 73.3% SWE-bench Verified vs Sonnet's 79.6%. Sprint 1 domain entities, test fakes, and high-volume QA test generation are good candidates. Save ~$1.40 on the pipeline.
  • Reuse architecture: If building a similar app, reuse rules.md from a previous project. Skip Architect entirely. Save ~$0.08 (down from ~$0.25) plus the 4-minute Architect dispatch.
  • Reduce sprints: For simple apps, 2 sprints instead of 4. Foundation + everything else. Save ~15% on orchestration overhead and ~10 minutes of wall-clock time.

The Skill Library as a Team Asset

Your .claude/ directory is a team asset, not a personal tool. Version it, share it, evolve it:

Three locations, three roles in 2026:
  • .claude/skills/<name>/SKILL.md — auto-loaded by Claude when relevant or invoked directly with /<name>. The 2026 canonical home for reusable role playbooks (PO, PM, Architect, Developer, QA, Planner, Orchestrator).
  • .claude/agents/<name>.md — sub-agent definitions, delegated via the Agent tool with their own context window. Best for the review roles (Code Reviewer, Security Reviewer) where context isolation matters.
  • .claude/commands/<name>.md — legacy slash commands, now merged into skills. Existing files keep working, but new roles should go in .claude/skills/ for the supporting-files and auto-load features.

Skill Library Best Practices

  • Version in git: Skills are markdown files. They belong in your repo, reviewed in PRs like any other code.
  • One skill per role: Keep skills focused. A skill that does PO + PM is harder to test and iterate than two separate skills.
  • Include examples: Every skill should have ✅/❌ examples in its output format. The AI follows examples more reliably than instructions.
  • Log and iterate: After each pipeline run, note which skills produced good output and which needed re-dispatch. Update the skill file.
  • Share across projects: The PO and PM skills are almost identical across projects. The Architect and Developer skills are platform-specific. Build a library with both generic and platform-specific skills.

  commands/                                # legacy (still works, merged with skills)
    # Existing files keep working; new roles go in skills/
Why split skills and agents this way? Reviewers run in isolation on a single artifact (the diff or the security scan target) and benefit from a fresh context window each invocation — that's what sub-agents give you. The 7 generative roles, by contrast, work better as skills because Claude can auto-load them when conversation context matches (the Architect skill activates when you say "design the data layer"; the Developer skill activates when you ask for an implementation). Same role library, two execution modes.

The Factory as a Competitive Advantage

Without the Factory

  • Ad-hoc prompting, different quality every time
  • No architecture rules — 10 modules, 10 styles
  • Manual code review catches maybe 60% of issues
  • Tests are "I'll write them later" (never)
  • 2-4 days to build a basic app

With the Factory

  • Reproducible pipeline, same quality every run
  • Architecture rules keep 20 dispatches consistent
  • AI review catches ~90% + human catches the rest
  • 128 tests generated automatically, 92% coverage
  • 90 minutes, ~$9, 17 minutes of your time

Exercise: Build Your Factory

Your Complete AI Factory Setup

  1. Choose your project. Pick a real app you want to build. Write the one-sentence seed idea (Chapter 3 exercise).
  2. Choose your pipeline size. Solo dev (3 roles), small team (5 roles), or enterprise (7+ roles)? Start smaller than you think you need.
  3. Copy the skill files. All 7 core skills are in Chapters 4-11. Copy them into .claude/commands/ and customize the Architect's rules for your platform.
  4. Run Discovery + Design. Execute Phase 1 and Phase 2. Review at both checkpoints. Get the architecture right before writing any code.
  5. Run one sprint. Execute Sprint 1 of Phase 3. Review the output. Tune your skills based on what worked and what needed correction.
  6. Scale up. Once Sprint 1 is solid, run the remaining sprints. Each sprint gets better as the pattern stabilizes.
  7. Share with your team. Commit the skill files to your repo. Write a README explaining the pipeline. Your teammates can run the same factory on their features.

From Zero to Factory

Twelve chapters ago, you had a single sentence: "I want an offline-first Android app for managing recipes." Now you have a complete development pipeline with 7 specialized AI roles, structured contracts, quality gates, and human oversight points.

The pipeline isn't magic. It's engineering — applying the same principles of separation of concerns, clear interfaces, and quality gates that you already use in your code to the process of building code.

The AI doesn't replace you. It replaces the 80% of work that's mechanical: writing boilerplate, checking rules, generating tests, catching naming inconsistencies. You focus on the 20% that matters: product vision, architecture decisions, UX quality, and the final ship call.

The factory is yours. Build it. Run it. Evolve it. Ship faster, ship better, and spend your time on the work that only humans can do.
