Chapter 3

One Sentence to App

From a single idea to a full pipeline execution plan

The Starting Line

You have the factory blueprint (Chapter 1) and the machinery (Chapter 2). Now it's time to feed the machine its first input. Everything in this course begins with a single sentence — one idea that the 7-role pipeline will transform into a production-grade Android app.

This chapter configures the pipeline for our recipes app. You'll map out every artifact the pipeline produces, define the handoff protocol between roles, estimate time and cost, and set up the project structure. By the end, you'll have a complete execution plan ready for the first role to consume.

The One-Sentence Seed

Human Input
The Seed Idea

“I want an offline-first Android app for managing recipes, organizing them into books, and generating grocery lists.”

One sentence. 20 words. From this, the pipeline will produce thousands of lines of specifications, architecture documents, Kotlin code, review findings, and tests. But the seed alone isn't enough — it needs constraints.

Why Constraints Matter

A seed without constraints is like a function without parameters — the output could be anything. Constraints narrow the solution space and give every role in the pipeline concrete boundaries to work within.

Here are the constraints for the recipes app:

Constraint Value Impact on Pipeline
Platform Android (Kotlin) Developer generates Kotlin, QA writes JUnit/Espresso
UI Framework Jetpack Compose Developer uses Compose APIs, no XML layouts
Architecture Clean Architecture + MVI Architect designs layers, Reviewer checks boundaries
DI Hilt Developer uses @Inject, @HiltViewModel, @Module
Local Storage Room Architect designs entities, Developer implements DAOs
Offline-First Yes No network calls in MVP. All data local.
Target Audience Home cooks PO generates personas around cooking workflows
Constraints are pipeline parameters. When you change a constraint (say, swap Room for DataStore), the Architect produces a different design, the Developer writes different code, and the Reviewer checks different rules. The pipeline is the same — only the parameters change.

The Handoff Protocol

Artifacts need a consistent format so roles can consume them reliably. The handoff protocol defines the structure every artifact must follow:

Protocol
Standard Artifact Header
---
artifact: vision | specs | architecture | rules | task | review | security
produced_by: product-owner | product-manager | architect | developer | ...
consumed_by: [list of roles that read this]
version: 1
status: draft | approved | rejected
---

# [Artifact Title]

[Content follows the role's output contract]

The YAML frontmatter tells every role exactly what it's looking at. The status field is especially important — it tracks whether a human has approved the artifact at a checkpoint. Roles should refuse to consume artifacts with status: rejected.

Handoff Protocol Rules
Every artifact MUST include: YAML frontmatter with artifact type, producer, consumer list, version, and status fields
Every consuming role MUST: Verify the artifact type matches what it expects, check status is approved or draft, and reference the artifact version in its own output
Why version artifacts? When the Developer-Reviewer loop sends code back for fixes, the Developer produces version 2 of the code, and the Reviewer re-reviews version 2. Without versioning, you can't tell which iteration you're looking at. It also helps when you resume a pipeline after interruption.

Time and Cost Estimation

How long does the full pipeline take? How much does it cost? Here's the realistic breakdown for the recipes app:

~25 min
Total pipeline time for discovery + design + one module implementation
~$2-5
Total API cost for the full pipeline (all 7 roles, 10 modules) under Opus 4.7 ($5/$25) + Sonnet 4.6 ($3/$15)
~150K
Tokens generated across all roles (input + output combined). Opus 4.7's tokenizer can run up to 1.35× for code-heavy content.
15-20
Agent dispatches for the complete pipeline including retries

Here's the per-role breakdown:

Role Model Time Output Tokens Est. Cost
Product Owner Opus 4.7 ~2 min ~2,500 ~$0.04
Product Manager Opus 4.7 ~3 min ~4,000 ~$0.06
Architect Opus 4.7 ~4 min ~6,000 ~$0.10
Developer (x10) Sonnet 4.6 ~10 min ~80,000 ~$2.00
Code Reviewer Opus 4.7 ~3 min ~5,000 ~$0.08
Security Reviewer Opus 4.7 ~2 min ~3,000 ~$0.06
QA Engineer Sonnet 4.6 ~3 min ~40,000 ~$1.00
Total ~140K output ~$3.34
Cost vs. value: The full pipeline costs about the same as a fancy coffee. The Developer role is the biggest cost because it runs 10 times (once per module) on Sonnet 4.6. The discovery and design phases (PO + PM + Architect) cost under $0.25 total under Opus 4.7's $5/$25 pricing — the price drop made judgment roles dramatically cheaper than the Sonnet-bound execution roles.

Configuring the Pipeline

Before running the pipeline, you need to set up the project structure. Every AI role needs to find its skill file, its input artifacts, and its output location: 

recipes-app/
  CLAUDE.md                          # Project config + architecture rules
  .claude/
    commands/
      product-owner.md               # PO role skill
      product-manager.md             # PM role skill
      architect.md                   # Architect role skill
      developer.md                   # Developer role skill
      code-reviewer.md               # Code Reviewer role skill
      security-reviewer.md           # Security Reviewer role skill
      qa-engineer.md                 # QA Engineer role skill
      orchestrate-pipeline.md        # Master orchestrator skill
  artifacts/
    vision.md                        # PO output (created by pipeline)
    specs.md                         # PM output
    architecture.md                  # Architect output
    rules.md                         # Architect output
    backlog/                         # Architect output
      task-01-core-domain.md
      task-02-recipe-entity.md
      ...
    review.md                        # Reviewer output
    security.md                      # Security output
  app/
    src/                             # Developer output (actual app code)
    src/test/                        # QA output (test code)
The CLAUDE.md file is critical. It tells Claude Code about your project — architecture rules, coding conventions, module structure. Every role reads it automatically. This is where the Architect's rules land after the design phase, making them available to the Developer and Reviewer roles without explicit passing.

For more on configuring CLAUDE.md for AI productivity, see AI-Engineering Chapter 3.

The Pipeline Execution Plan

Before launching the first role, the orchestrator produces a complete execution plan. Here's what the plan looks like for the recipes app:

Phase 1: Discovery — 2 dispatches, ~5 min, ~$0.10 (Opus 4.7)
1. Product Owner → artifacts/vision.md
2. CHECKPOINT: Human reviews vision
3. Product Manager → artifacts/specs.md
Phase 2: Design — 1 dispatch, ~4 min, ~$0.10 (Opus 4.7)
4. Architect → artifacts/architecture.md + rules.md + backlog/*.md
5. CHECKPOINT: Human reviews architecture
Phase 3: Implementation — 10 dispatches, ~10 min, ~$2.00
6-15. Developer → one module per dispatch, dependency order:
:core:domain → :core:ui → :data:recipe → :data:book → :data:grocery → :feature:recipe → :feature:book → :feature:grocery → :feature:search → :app
Phase 4: Quality — 3-5 dispatches, ~8 min, ~$1.15
16. Code Reviewer (may trigger re-dispatch to Developer)
17. Security Reviewer (may HALT on CRITICAL)
18. QA Engineer → src/test/**/*.kt
Total: 15-20 dispatches • ~25 min • ~$2-5 estimated (Opus 4.7 + Sonnet 4.6)

The First Run: Pipeline Overview

Let's do a dry run. Instead of executing all seven roles, let's invoke the orchestrator in plan-only mode — it will analyze the seed idea and produce the execution plan without dispatching any subagents:

Invoking the Orchestrator
/orchestrate-pipeline

Seed: "I want an offline-first Android app for managing recipes, organizing them into books, and generating grocery lists."

Constraints: Android, Kotlin, Compose, Room, Hilt, Clean Architecture + MVI
Audience: Home cooks

Mode: plan-only (don't execute roles, just produce the execution plan)
The orchestrator reads the seed and constraints, maps them to the 7-role pipeline, and produces a detailed execution plan. No code is generated yet — this is the blueprint for the pipeline run.

The plan-only mode is your safety net. Review the plan before committing tokens and time. Does the module list look right? Are the checkpoints in the right places? Is the dependency order correct? Fix the plan now — changing it mid-pipeline is expensive.

Iterate on the plan, not the pipeline. If the orchestrator's plan misses a module or puts checkpoints in the wrong places, edit the plan and re-run. Planning costs almost nothing. Executing a wrong plan wastes real tokens.

Exercise: Configure Your Own Pipeline

Using your seed idea from Chapter 1's exercise, set up the pipeline:

  1. Define your constraints. Create a table with: platform, UI framework, architecture, DI, storage, network strategy, and target audience.
  2. Map the artifact chain. List every artifact your pipeline will produce: filename, producing role, consuming role, and contents summary.
  3. Create the directory structure. Set up your project with .claude/commands/ for skills and artifacts/ for outputs. You don't need to write the skills yet — just create empty placeholder files for all 8 skills (7 roles + orchestrator).
  4. Estimate time and cost. Based on the recipes app numbers, adjust for your app's complexity. More modules = more Developer dispatches = higher cost.
  5. Write the execution plan. List all phases, dispatches, and checkpoints. Where will you insert human review?
