Chapter 13

Running the Factory Programmatically — The Claude Agent SDK

From interactive Claude Code to autonomous library: the same factory, now scriptable

The Factory, Now as a Library

Twelve chapters in, you have a 7-role pipeline that runs interactively in Claude Code: you type /orchestrate "build the recipes app", hit enter, and watch the roles fire one after another. That's perfect for greenfield work where a human wants to be in the loop.

But what about nightly app generation? Triggering the factory on a config change? Running it inside CI for client work, or embedding it inside an existing dev tool? Interactive Claude Code can't do that — it's a CLI, not a library.

Enter the Claude Agent SDK: the exact same agent loop, tools, sub-agents, hooks, skills, and MCP support that power Claude Code, packaged as an importable Python and TypeScript library. The factory you built in chapters 4–11 runs unchanged — just headless.

The SDK was renamed in 2026. What used to be the "Claude Code SDK" is now the Claude Agent SDK. The old @anthropic-ai/claude-code package still exists for the CLI binary; the library you import for programmatic use is @anthropic-ai/claude-agent-sdk (TypeScript) or claude-agent-sdk (Python). If you have older code referencing claude-code-sdk, see the migration guide in the official docs.

Claude Code vs Agent SDK: One Loop, Two Surfaces

The Agent SDK and Claude Code share an engine. The CLI is a UI on top of the SDK. Anything you can configure for Claude Code — skills, sub-agents, hooks, MCP servers, tool permissions, settings — is configurable in the SDK with the same semantics.

Aspect Claude Code (CLI) Claude Agent SDK
Surface Terminal session, you type prompts Library function, you call query()
Best for Interactive development, one-off tasks CI/CD, batch jobs, custom apps, production automation
Skills Loaded from .claude/skills/ Loaded from .claude/skills/ (same files, same semantics)
Sub-agents .claude/agents/<name>.md Same file format, OR programmatic AgentDefinition
Tools Built-ins + MCP + bash/edit Identical set, configured via allowed_tools
Authentication Browser login or API key API key only (Bedrock/Vertex/Foundry supported)

The headline: same factory, two execution modes

Workflows translate directly. A team can run the factory interactively during design (CLI), then re-run it in CI for a nightly regression build (SDK), without rewriting a single skill file. The .claude/ directory is the shared contract between the two surfaces.

Install and First Run

Install in whichever runtime fits your stack. Python 3.10+ or Node.js 18+ are the minimums. The TypeScript SDK bundles a native Claude Code binary as an optional dependency, so you don't need to install Claude Code separately. 

# TypeScript / Node.js
npm install @anthropic-ai/claude-agent-sdk

# Python
pip install claude-agent-sdk

# Set your API key
export ANTHROPIC_API_KEY=your-api-key

# Opus 4.7 requires SDK v0.2.111 or later
First Agent — Python 
import asyncio
from claude_agent_sdk import query, ClaudeAgentOptions

async def main():
    async for message in query(
        prompt="What files are in this directory?",
        options=ClaudeAgentOptions(allowed_tools=["Bash", "Glob"]),
    ):
        if hasattr(message, "result"):
            print(message.result)

asyncio.run(main())
First Agent — TypeScript 
import { query } from "@anthropic-ai/claude-agent-sdk";

for await (const message of query({
  prompt: "What files are in this directory?",
  options: { allowedTools: ["Bash", "Glob"] }
})) {
  if ("result" in message) console.log(message.result);
}

That's it. query() is an async iterator that streams messages as Claude works through the agent loop — tool calls, results, intermediate reasoning, and the final response. No tool-execution loop to implement; the SDK handles it.

Loading the Factory's Skills (Zero Code Changes)

Here's the pivotal feature for our purposes: the SDK auto-loads every skill, sub-agent, and CLAUDE.md from your working directory's .claude/ folder by default. The exact same skill files you wrote in chapters 4–11 work in the SDK without any edits.

FeatureLocationAuto-loaded by SDK?
Skills.claude/skills/<name>/SKILL.mdYes
Slash commands.claude/commands/<name>.mdYes (merged with skills)
Sub-agents.claude/agents/<name>.mdYes
MemoryCLAUDE.md or .claude/CLAUDE.mdYes
PluginsProgrammatic plugins optionOn request
Restrict what the SDK loads. Set setting_sources (Python) or settingSources (TypeScript) in ClaudeAgentOptions to ["project"] or ["user"] if you want to scope down. For CI, restrict to ["project"] so the runner only sees the repo's skills, never a developer's ~/.claude/.

That means the orchestrator skill from chapter 11 already runs under the SDK unchanged. The Architect, Developer, Reviewer, and friends all work the same way — auto-loaded from the project's .claude/ directory.

The Factory as a Python Script

Let's stitch the pipeline together programmatically — not because we can't invoke /orchestrate via the SDK (we can), but because explicit Python gives us per-stage control: budget caps, structured outputs, custom retries, and observability.

Programmatic Orchestrator (SDK form)
Input: Seed idea (string), config (model lanes, budget, output dir)
Output: Generated codebase + ship-readiness report + cost ledger
Model: Opus 4.7 for the orchestrator's judgment calls (advance / re-dispatch / escalate)
orchestrate.py — minimal pipeline 
import asyncio, json, pathlib
from claude_agent_sdk import query, ClaudeAgentOptions

PROJECT_DIR = pathlib.Path("./recipes-app")
SETTINGS = ClaudeAgentOptions(
    cwd=str(PROJECT_DIR),
    setting_sources=["project"],     # only this repo's .claude/
    allowed_tools=["Read", "Write", "Edit", "Bash", "Glob", "Grep", "Agent"],
    model="claude-opus-4-7",         # orchestrator runs on Opus 4.7
)

async def run_role(prompt: str, options: ClaudeAgentOptions) -> str:
    final = ""
    async for message in query(prompt=prompt, options=options):
        if hasattr(message, "result"):
            final = message.result
    return final

async def main(idea: str):
    # Phase 1 — Discovery
    vision = await run_role(f"/product-owner {idea}", SETTINGS)
    specs  = await run_role(f"/product-manager (use vision.md from previous step)", SETTINGS)

    # Phase 2 — Design
    arch   = await run_role("/architect (use specs.md)", SETTINGS)
    plan   = await run_role("/implementation-planner (use architecture.md)", SETTINGS)

    # Phase 3 — Build (loop per task)
    for task in json.loads(plan)["tasks"]:
        await run_role(f"/developer {task['id']}", SETTINGS)
        verdict = await run_role(f"/code-reviewer {task['id']}", SETTINGS)
        if "BLOCK" in verdict:
            await run_role(f"/developer fix {task['id']}", SETTINGS)

    # Phase 4 — Ship
    sec = await run_role("/security-reviewer", SETTINGS)
    qa  = await run_role("/qa-engineer", SETTINGS)
    print({"sec": sec, "qa": qa})

asyncio.run(main("offline-first Android recipes app"))

Each run_role call dispatches a fresh agent loop with the project's skills auto-loaded. The skill names (/product-owner, /architect, etc.) are exactly the same ones from chapters 4–11 — no rewrite needed.

Headless: CI, Cron, Webhooks

Once the orchestrator is a script, you can wire it into anything that runs scripts. Three common patterns:

CI Pipeline
GitHub Actions, GitLab CI, or Jenkins. A push to main triggers a job that runs python orchestrate.py. The skills directory ships with the repo, so the CI runner has everything it needs. Use it for nightly regression builds, PR-triggered scaffolding, or release-branch validation.
Cron / Scheduled
Daily idea-to-prototype runs. A cron job pulls the latest seed from a "ideas" backlog file or Linear ticket and spins up a prototype branch. Useful for spike-style explorations or content factories (the same pipeline can generate a marketing site as easily as an Android app).
Webhook
Trigger on config or schema change. A small HTTP service wraps orchestrate.py. POST a payload (idea, target platform, budget cap) and the service runs the pipeline and posts the result. Great for client-work shops where each engagement starts with a one-page brief.
Headless = no humans in the loop. All three patterns lose the interactive checkpoints that Claude Code gives you for free (the Architect review, the Code Reviewer block-then-confirm, the Security Reviewer escalation). You'll re-add those as code: an approval gate that emails a human between Phase 2 and Phase 3, a Slack-prompt before deploy. See AI-Workflow for automation patterns that fit here.

Structured Outputs

Two ways to get JSON instead of Markdown back: (1) Skill-side — have the role emit a JSON block at the end of its output and parse it from the orchestrator. Keeps the human-readable artifact intact. (2) SDK-side — pass a Pydantic/Zod schema via the SDK's structured-output option; the response validates and retries once on mismatch. Use skill-side for design roles where prose helps human review; SDK-side for verdict roles (Reviewer, Security, QA) where you only want typed pass/fail + findings.

The Factory Reports Its Own Spend

Each query() result includes a usage block: input tokens, output tokens, total cost. Aggregate across role calls and you have a per-pipeline cost ledger that beats the CLI's anecdotal "~$6.50" estimate.

Budget enforcement via PreToolUse hook 
from claude_agent_sdk import query, ClaudeAgentOptions, HookMatcher

BUDGET_USD = 8.00
spent = {"total": 0.0}

async def enforce_budget(input_data, tool_use_id, context):
    if spent["total"] >= BUDGET_USD:
        return {"decision": "block", "reason": f"budget exceeded: ${spent['total']:.2f}"}
    return {}

async def track_spend(input_data, tool_use_id, context):
    cost = input_data.get("usage", {}).get("cost_usd", 0)
    spent["total"] += cost
    return {}

opts = ClaudeAgentOptions(
    hooks={
        "PreToolUse":  [HookMatcher(matcher="*", hooks=[enforce_budget])],
        "PostToolUse": [HookMatcher(matcher="*", hooks=[track_spend])],
    },
)

Now the pipeline auto-halts when it hits the cap. Pair this with the model lanes from chapter 12 (Opus 4.7 for judgment, Sonnet 4.6 for daily work, Haiku 4.5 for mechanical) and you have a self-governing cost system: the Factory knows what it costs, when to slow down, and when to escalate. That's the kind of feedback loop you can't build interactively.

Hybrid Usage and When to Graduate

The pragmatic 2026 pattern is to use both surfaces. Architect-stage iteration runs in Claude Code (humans iterate on rules with interactive feedback); Discovery + Build + Review runs in CI via the SDK once the architecture is locked. Both share the same .claude/ directory, so a skill edit made interactively is immediately live for the next CI run — no library version drift.

Stay on Claude Code (CLI) when…

  • The pipeline runs less than once a week
  • Each run is bespoke (different ideas, different scopes)
  • You want a human in the loop for design decisions
  • Team is one or two engineers

Move to Agent SDK when…

  • The pipeline runs daily or on every push
  • Multiple projects or clients run the same skills
  • You need cost ceilings and audit logs
  • The Factory is part of a product or needs Slack/PR approvals

Security Caveats: Re-add the Humans

Headless mode loses every safety rail Claude Code gives you for free. The CLI prompts before destructive operations; the SDK doesn't unless you wire it. For the Factory specifically:

Three places to re-add a human gate in the SDK:
  • After Architect (end of Phase 2): ship the architecture.md + rules.md to a review channel. Block the loop until a human responds.
  • After Security verdict (Phase 4): if any HIGH/CRITICAL finding exists, do NOT auto-merge. Send to security oncall.
  • Before any deploy or external API call: the SDK has a permission_mode: "acceptEdits" setting that auto-approves edits but still prompts for shell commands. Use this aggressively.

Also be deliberate about which tools you allow. A pipeline that should write code to a directory does not need WebFetch or WebSearch. Tighten allowed_tools to the minimum set per role; the SDK will refuse anything outside that list. (For more on this, the OWASP LLM Top 10's "Excessive Agency" is exactly this category — see chapter 10 for the Factory-side mitigations.)

Worked Example: RecipeVault in CI

Putting it all together, here's the .github/workflows file that runs RecipeVault nightly. The skill directory lives in the repo; the orchestrator script lives in scripts/; the API key is a GitHub secret.

.github/workflows/nightly-factory.yml 
name: Nightly Factory Build
on:
  schedule:
    - cron: '0 2 * * *'   # 2am UTC
  workflow_dispatch:

jobs:
  factory:
    runs-on: ubuntu-latest
    timeout-minutes: 120
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with: { python-version: '3.11' }

      - run: pip install claude-agent-sdk

      - name: Run the AI Factory
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: python scripts/orchestrate.py "offline-first Android recipes app"

      - name: Upload artifacts
        uses: actions/upload-artifact@v4
        with:
          name: recipes-app-${{ github.run_id }}
          path: |
            recipes-app/
            ship-readiness-report.json
            cost-ledger.json

The total per-run cost is bounded by the budget hook (~$8). Each morning the team gets a fresh build artifact, a ship-readiness report, and a cost ledger. The skills evolved interactively in Claude Code during business hours; the nightly run validates them at scale.

Exercise: Headless Your Factory

Run the RecipeVault Pipeline via SDK

  1. Install the SDK in your factory project's environment (pip install claude-agent-sdk or npm install @anthropic-ai/claude-agent-sdk).
  2. Write a 30-line orchestrate.py that loops through your existing skills (/product-owner → /architect → one Developer dispatch). Verify it loads your .claude/skills/ without modification.
  3. Add a budget hook. Set the cap at $2 for testing. Confirm the loop halts when the cap is hit.
  4. Add an approval gate. Between Architect and Developer, print the architecture.md and prompt the user to type "yes" on stdin. If they say no, exit. (In production, replace with a Slack hook.)
  5. Wire it to GitHub Actions. Add the workflow above; trigger it via workflow_dispatch first to validate before scheduling.
  6. Compare costs. Run the same pipeline interactively in Claude Code and via the SDK. The token counts should match within a few percent — same loop, same model, same skills.

The Factory is Now a Library

The CLI taught you the moves. The SDK lets you compose them. The 80/20 split from chapter 12 still holds — the SDK doesn't replace the human in the loop; it lets the human set the loop instead of being inside it. Write the skills once, wire the budget once, wire the approvals once — then the factory runs without you, on the cron of your choice. That's the production end-state: a 7-role pipeline, a script that runs it, and a ledger that proves what it cost.
