The AI Developer: Writing Code at Scale

Build the Developer skill that turns task specs into compilable, tested code

The Pipeline's Workhorse

Five roles have spoken: the PO defined the vision, the PM wrote specs, the Architect designed the structure, and you planned the implementation. Now the Developer role does the heavy lifting — turning 20 task specs into actual Kotlin code.

The Developer is the most-dispatched role in the pipeline. It runs 20 times for RecipeVault — once per task. It's also the biggest cost driver, accounting for roughly 60% of the total pipeline spend. The default model is Sonnet 4.6 because the Developer follows rules mechanically rather than making judgment calls — and for the lowest-risk mechanical tasks, Haiku 4.5 drops the cost another 3× with no quality loss.

This chapter builds the Developer skill, shows the build-test-fix loop, and walks through generating the first module.

Context Loading Strategy

The Developer's quality depends entirely on what context it sees. Too little context and it invents types that don't exist. Too much and it drowns in noise. The skill file specifies exactly what to load:

1. Rules
Always load artifacts/rules.md first. These are non-negotiable constraints. The Developer reads them before writing a single line.
2. Task Spec
The specific task from artifacts/plan.md. Lists files to create, dependencies available, acceptance criteria, relevant rules.
3. Existing Code
Source files from previous tasks that this task depends on. The Developer reads interfaces it needs to implement or types it needs to import.
4. Architecture
The relevant section of artifacts/architecture.md — module structure, data models, technology stack. Not the whole document.
Context priority matters. If the context window fills up, rules and task spec take priority over existing code. A Developer that forgets the rules produces inconsistent code. A Developer that can't see a prior file can at least follow the interface contract from the architecture doc.

The Complete Developer Skill File

Here's .claude/commands/developer.md — the full Developer skill: 

# .claude/commands/developer.md

You are the **Developer** in the AI Factory pipeline.

## Identity
You write production-quality Kotlin code that compiles and passes
tests. You follow architecture rules exactly. You do NOT make
design decisions (Architect already did). You do NOT skip tests.

## Input
You will receive:
1. **Architecture rules** — read `artifacts/rules.md`. Follow every rule.
2. **Task spec** — the specific task to implement (module, files, criteria)
3. **Existing code** — source files from prior tasks you depend on
4. **Architecture doc** — relevant section of `artifacts/architecture.md`

## Process
Follow this loop for EVERY task:

### Step 1: Read
- Read all rules. Read the task spec. Read existing code you depend on.
- Identify every type, interface, and import you need.

### Step 2: Plan
- List every file you will create.
- For each file, list the classes/functions and their signatures.
- Verify all imports will resolve against existing code.

### Step 3: Write
- Write each file completely. No TODOs, no placeholders.
- Follow naming conventions from the rules exactly.
- Include all imports. Use fully qualified names if unsure.

### Step 4: Test
- Write a unit test file for every public class.
- Test all public methods. Include happy path and edge cases.
- Use fakes from `:test:shared` where available.

### Step 5: Verify
- Re-read the rules. Check every file against each rule.
- Verify all acceptance criteria from the task spec are met.
- List any criteria you could NOT meet and explain why.

## Output Format
For each file, use this structure:
```
### File: path/to/File.kt
```kotlin
// Full file content
```
```

End with a completion report:
```
## Task Completion Report
- Files created: [list]
- Tests created: [list]
- Acceptance criteria met: [list with ✅/❌]
- Issues encountered: [list or "none"]
```

## Rules
- NEVER use placeholder code, TODOs, or stub implementations
- NEVER skip test files — every public class gets a test
- NEVER make architecture decisions — follow the architecture doc
- NEVER add dependencies not specified in the architecture
- If a required type doesn't exist yet, check if it's in a later task.
  If so, create a minimal interface in the current module.
- Prefer explicit imports over wildcard imports
- Follow Kotlin coding conventions and the project's naming rules
The 5-step process is deliberate. Read → Plan → Write → Test → Verify mirrors how a careful human developer works. The “Plan” step prevents the common AI failure of starting to write code before understanding the full scope, which leads to inconsistent naming and missing imports.

The Build-Test-Fix Loop

The Developer skill includes a built-in verification step, but the orchestrator adds an external validation layer: after each dispatch, it compiles the code and runs the tests.

✅ Pass / 🔄 Fix

The Fix Loop

When compilation or tests fail:

  1. Capture the error. The orchestrator collects the compiler output or test failure stacktrace.
  2. Re-dispatch with feedback. Send the Developer the original task spec PLUS the error output. The Developer reads the error and fixes the code.
  3. Max 2 retries. If the fix loop runs twice and the code still doesn't compile, escalate to the human. Some errors need judgment.

Common Developer Failures

Even with a good skill file, the Developer role hits predictable failure modes. Knowing them helps you tune the skill:

Missing Imports
The Developer references a type from another module but forgets the import. Fix: add a rule — “Use explicit imports, never wildcard.”
Wrong Package
Files end up in the wrong package path. Fix: include exact file paths in the task spec, not just file names.
Naming Drift
Task 1 uses RecipeRepository, task 8 invents RecipeRepo. Fix: RULE-004 naming conventions + reference existing code.
Missing Tests
The Developer writes code but skips or partially writes tests. Fix: add a rule — “NEVER submit without a test file for every public class.”
Vague Task Spec → Failures
Task: “Create the DAOs for the data layer.”
Result: Wrong package name, inconsistent method naming, missing @Transaction annotations, no test files.
Without explicit file paths, available dependencies, and relevant rules, the Developer fills in the blanks differently each time.
Detailed Task Spec → Clean Output
Task 6: Create Room DAOs. Files: dao/RecipeDao.kt, dao/RecipeBookDao.kt, dao/SearchDao.kt. Depends on: RecipeEntity (Task 5). Follow: RULE-003, RULE-006. Criteria: search by title, ingredient, tag. Multi-table ops use @Transaction.
Every decision is pre-made. The Developer just writes the code.

Handling Context Overflow

By Sprint 3, the existing codebase has grown. The Developer needs to see prior code to implement correctly, but loading everything won't fit in context. Strategy:

Context Budget Priorities

  1. Rules (always): ~800 tokens. Never skip.
  2. Task spec (always): ~400 tokens. The task at hand.
  3. Direct dependencies (always): Only files this task imports from. Load interfaces, not implementations.
  4. Architecture doc (summary): ~600 tokens. Module structure and data models, not the full document.
  5. Similar completed code (if room): If building RecipeDetailViewModel, show the completed RecipeListViewModel as a pattern reference.
Load interfaces, not implementations. The Developer needs to know what RecipeRepository looks like (its interface), not how it's implemented. Interfaces are 10-20 lines. Implementations are 100+. This is the single biggest context savings trick.

Why Sonnet 4.6 — and When to Drop to Haiku 4.5

The Developer is a rule-following role, not a judgment role. It reads rules and follows them. It reads task specs and implements them. The 2026 economic story is no longer "Sonnet vs Opus" — with Opus 4.7 at $5/$25 and Sonnet 4.6 at $3/$15, Sonnet's cost edge over Opus shrunk dramatically. The interesting model choice now is Sonnet 4.6 vs Haiku 4.5.

Sonnet 4.6 — the default Developer
• $3/$15 per MTok — ~$0.10 per task
• 79.6% SWE-bench Verified — the daily-driver coding model
• Adaptive thinking — reasons longer on hard tasks, faster on easy ones
• Best for: ViewModels, repositories, navigation wiring, cross-module work
Sonnet 4.6 is the default for the Developer role in this course. It handles the messy 80% of code generation where you need real reasoning but not Opus-tier judgment.
Haiku 4.5 — the mechanical lane
• $1/$5 per MTok — exactly 3× cheaper than Sonnet
• 73.3% SWE-bench Verified — within ~5 pp of Sonnet 4.6
• 4–5× faster than Sonnet on typical workloads
• Best for: domain entities, test fakes, theme tokens, simple mappers, DAO boilerplate
Route the lowest-risk tasks to Haiku 4.5. For RecipeVault, Sprint 1 (Tasks 1, 4, 5) drops from ~$0.40 to ~$0.13 with no quality loss on mechanical work. The orchestrator picks the model per task using the risk tier from your implementation plan.
Cost math under 2026 pricing. 20 Sonnet 4.6 dispatches × ~$0.10 = $2.00 for the full Developer pass. Splitting low-risk tasks to Haiku 4.5 (~6 of 20 at ~$0.03 each) drops the bill to ~$1.60 with no review friction added. The savings get more interesting in Chapter 10 where QA's high-volume test generation is an even stronger Haiku candidate.

Exercise: Build and Run Your Developer Skill

  1. Create the skill file. Copy the Developer skill into .claude/commands/developer.md.
  2. Pick Task 1. Take the first task from your implementation plan (domain entities). Run /developer with the task spec.
  3. Compile. Run ./gradlew :core:domain:build. Does it compile? If not, re-dispatch with the error output.
  4. Check tests. Did the Developer create test files? Do they pass? If tests are missing, strengthen the rule: “NEVER submit without tests.”
  5. Check rules. Read the completion report. Are all rules checked? Open each file and verify RULE-001 (no framework imports in domain).
  6. Run Task 2. Now try a task that depends on Task 1. Does the Developer correctly import types from the first task? If not, make sure you're loading the prior task's code as context.
