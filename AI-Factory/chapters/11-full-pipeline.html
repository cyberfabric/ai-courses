Chapter 11

The Full Pipeline: All Roles in Concert

From one sentence to reviewed, secured, tested code — the complete end-to-end run

Pressing Play

You've built every role. You've seen each one in isolation — the PO generating a vision, the PM writing specs, the Architect designing modules, the Developer writing code, the Reviewer checking rules, the Security Reviewer scanning for vulnerabilities, and the QA Engineer generating tests.

Now let's press play and watch them all run together. This chapter walks through the complete pipeline execution for RecipeVault, from the one-sentence idea to a reviewed, secured, tested codebase. You'll see the orchestrator skill that coordinates everything, the full execution log, and where the pipeline excels versus where it needs human help.

The Orchestrator Skill

Orchestrator
Input: One-sentence idea, project configuration (platform, architecture style, target audience)
Output: Complete codebase with tests, review reports, security audit, QA coverage report
Model: Opus 4.7 — the orchestrator makes judgment calls about when to advance, when to re-dispatch, and when to escalate

The Orchestrator isn't just a sequencer. It's the decision maker that reads each role's output, decides whether it meets the contract, and routes accordingly. It's the only role that sees the entire pipeline state.

Skill File — Orchestrator (core loop) 
# Identity
You are the Orchestrator for the AI Factory pipeline.
Coordinate all roles from idea to ship-ready code. Model: Opus.

# Pipeline Phases
Execute these phases in order. After each phase, validate
the output before advancing.

## Phase 1: Discovery
1. Dispatch Product Owner with the seed idea
2. CHECKPOINT: Present vision doc to human for approval
3. Dispatch Product Manager with approved vision
4. Validate: specs have acceptance criteria for every feature

## Phase 2: Design
5. Dispatch Architect with approved specs
6. CHECKPOINT: Present architecture to human for approval
7. Dispatch Implementation Planner with architecture
8. Validate: plan has 15-25 tasks, dependency order is valid

## Phase 3: Build (repeat per sprint)
For each sprint in the plan:
  For each task in the sprint:
    9. Dispatch Developer with task spec + rules
    10. If task risk is MEDIUM or HIGH:
        Dispatch Code Reviewer with output + rules
        If REVISE: re-dispatch Developer (max 2 cycles)
    11. If task risk is HIGH:
        Dispatch Security Reviewer with output
        If BLOCK: CHECKPOINT — escalate to human
    12. Dispatch QA Engineer with output + specs
  End task loop
  13. Validate sprint: compile all code, run all tests
  If validation fails: re-dispatch failing task
End sprint loop

## Phase 4: Ship Readiness
14. Compile full project
15. Run complete test suite
16. Generate ship readiness report
17. CHECKPOINT: Present report to human for ship decision

# Error Recovery
- Role produces empty output → re-dispatch once, then escalate
- Role output doesn't match contract → re-dispatch with
  specific feedback about what's missing
- 2 consecutive failures → escalate to human with full context
- CRITICAL security finding → immediate human escalation

The Three Checkpoints

The pipeline runs autonomously between checkpoints, but pauses for human judgment at three critical points:

1
After PO Vision (Phase 1). Is this the right product? Are the priorities correct? Remove features that don't belong in MVP. This takes ~2 minutes but prevents building the wrong thing for $9.
2
After Architect Design (Phase 2). Is the module structure sound? Are the dependency directions correct? Wrong architecture is the most expensive mistake — fix it here, not during Sprint 3. ~5 minutes.
3
Ship Readiness (Phase 4). Review the report. Spot-check 2-3 files. Run the app. Does it feel right? The pipeline ensures correctness; the human ensures quality of experience. ~10 minutes.

Total Human Time: ~17 minutes

Out of 90 minutes of pipeline execution, you spend ~17 minutes actively engaged. The rest runs autonomously. Your job shifted from writing code to reviewing decisions and output.

Down from ~$9.20 on the Opus 4/4.1 rate card ($15/$75). Routing QA to Haiku 4.5 ($1/$5) drops it further to ~$5.10. The pipeline got noticeably cheaper without any quality tradeoff.

Cost scales with complexity, not linearly. A 40-task app doesn't cost 2× more — the Discovery and Design phases are one-time costs. Build and Review scale linearly, but the 2026 model lineup gives you three speed/cost lanes: Opus 4.7 for judgment, Sonnet 4.6 for everyday code generation, and Haiku 4.5 for mechanical high-volume work like test generation.

Error Recovery in Practice

The RecipeVault run wasn't perfect. Here's what went wrong and how the orchestrator recovered:

What Failed

  • Task 7 (AppDatabase): missing type converter — Code Reviewer caught it
  • Task 7: unencrypted database — Security Reviewer caught it
  • Tasks 12, 14: naming drift in 2 screens — Code Reviewer caught it
  • Tasks 18, 19: wrong Hilt scoping + nav argument types — Code Reviewer caught it

How It Recovered

  • Re-dispatched Developer with specific findings — fixed in 1 cycle
  • Re-dispatched with SQLCipher remediation steps — fixed in 1 cycle
  • Re-dispatched with naming corrections — fixed in 1 cycle each
  • Re-dispatched with Hilt scope rules + nav type fix — fixed in 1 cycle each

Every failure was caught by a quality role, fixed in one re-dispatch cycle, and verified by re-review. No escalations to human during the Build phase. The architecture rules and structured findings made fixes targeted and fast.

What would have happened without the quality stack? The type converter bug in Task 7 would have caused compile failures in Tasks 9-16 (all features depend on the database). That's 8 failing tasks instead of 1 re-dispatch. The quality stack cost $6.50 but saved an estimated $15-20 in rework.

Where the Pipeline Excels

Consistency
20 Developer dispatches, 1 coding style. Rules enforce uniformity across all modules.
Coverage
128 tests. No human would write 80 edge case tests for a side project.
Speed
90 minutes total. A human building the same app: 2-4 days minimum.
Reproducibility
Same pipeline, same skills, same rules = same quality every time you run it.

Where It Needs Human Help

  • Product judgment: The PO generates features, but only a human knows which features actually matter to real users.
  • UX quality: Code compiles and tests pass, but does the app feel right? Spacing, animation timing, loading states — these need human eyes.
  • Performance: No quality role measures runtime performance. A ViewModel that allocates in a loop won't be caught until profiling.
  • Business edge cases: The QA Engineer tests technical edge cases. It doesn't know that "a recipe with 0 servings makes no real-world sense."
  • Integration testing: The pipeline tests modules in isolation. End-to-end flows (create recipe → add to book → generate grocery list) need manual verification.
The pipeline doesn't replace you. It replaces the 80% of work that's mechanical. You focus on the 20% that requires human judgment: product decisions, UX feel, performance tuning, and the final ship call.

Exercise: Run Your Own Pipeline

Execute the Full Pipeline on Your App Idea

  1. Pick your one-sentence idea (from the Chapter 3 exercise, or a new one).
  2. Set up the skills directory: copy all 7 skill files into .claude/commands/. You have them from Chapters 4-10.
  3. Run Phase 1 manually: invoke the PO skill, review the vision, then invoke the PM skill. This is your discovery phase.
  4. Run Phase 2: invoke the Architect skill, review the architecture, then invoke the Planner. This is your design phase.
  5. Run Phase 3 for Sprint 1 only: dispatch the Developer for the first 3-4 tasks, then run the Code Reviewer and QA Engineer on the output.
  6. Compare the pipeline's output to what you'd produce manually. Where was the AI better? Where did it need correction? What would you change in the skills?

You don't need to run the full pipeline on your first try. Start with Discovery + Design, validate the architecture, then run one sprint. Build confidence before scaling to the full 4-sprint execution.
