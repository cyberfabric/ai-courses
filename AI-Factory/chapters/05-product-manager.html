Chapter 5

The AI Product Manager: Specs for Machines

Build the PM skill that turns vision into AI-implementable specifications

The PO Clocked Out. The PM Clocks In.

You have artifacts/vision.md — a product vision with personas, MoSCoW features, and a release roadmap. The PO answered “What are we building and for whom?” Now the Product Manager answers a different question: “What exactly does each feature do, and how do we know it works?”

Here's the key insight of this chapter: specs written for AI consumption are fundamentally different from specs written for humans. A human developer can read “implement search” and fill in the gaps with experience and judgment. An AI developer needs every behavior enumerated, every edge case listed, every acceptance criterion testable by code.

The PM is the translator between the PO's business language and the Architect's technical input. Get this wrong, and every downstream role suffers.

The Product Manager Role

Product Manager
Inputs: Approved vision document (artifacts/vision.md) with personas, MoSCoW features, and MVP scope
Outputs: Feature Specification Document (artifacts/specs.md) containing: feature specs with acceptance criteria, user stories with testable outcomes, edge cases, non-functional requirements, and API contracts between features

The PM sits between the PO and the Architect in the artifact chain. It consumes the vision and produces specs. It does NOT design architecture (Architect's job), does NOT prioritize features (PO already did that), and does NOT suggest technologies.

Specs for Humans vs. Specs for Machines

This is the insight that makes or breaks your pipeline. Look at the difference:

Human-Readable Spec
Search: Users should be able to search their recipes by title, ingredients, or tags. Search should be fast and work offline.
A human developer reads this and fills in 50 implicit decisions: debounce timing, minimum query length, result ordering, empty state UI, partial matches. An AI developer has to guess at all of them — and it will guess differently every time.
AI-Consumable Spec
Feature: Recipe Search
Trigger: User types in the search bar on the recipe list screen
Behavior:
• Search is triggered after 300ms debounce
• Minimum query length: 2 characters
• Searches: title (exact + prefix), ingredients (contains), tags (exact)
• Results ordered by relevance: title match > ingredient match > tag match
• Empty query shows all recipes sorted by last modified
• No results shows empty state with message “No recipes found”
Acceptance: Given 100 recipes, search for “chicken” returns all recipes containing “chicken” in title or ingredients within 200ms
Every decision is made. The Architect knows what to design. The Developer knows what to build. The QA Engineer knows what to test. Zero ambiguity.
The rule of thumb: if a spec sentence contains the words “should,” “probably,” “might,” or “as needed,” it's a human spec, not an AI spec. Rewrite it with specific values and explicit behavior.

The Complete PM Skill File

Here's .claude/commands/product-manager.md — the full PM skill, copy-paste ready: 

# .claude/commands/product-manager.md

You are the **Product Manager** in the AI Factory pipeline.

## Identity
You translate product vision into detailed, AI-implementable feature
specifications. You decide WHAT each feature does in precise detail.
You do NOT decide HOW to implement it (Architect's job). You do NOT
suggest technologies. You do NOT write code.

## Input
Read `artifacts/vision.md`. Verify:
- YAML frontmatter has `artifact: vision` and `status: approved`
- Contains: App Description, User Personas, MoSCoW Features, MVP Scope
- If status is `draft` or `rejected`, STOP and report

## Output
Produce `artifacts/specs.md` with YAML frontmatter:
```yaml
---
artifact: specs
produced_by: product-manager
consumed_by: [architect]
version: 1
status: draft
---
```

For EACH MVP feature, produce:
### Feature: [Name]
**Persona:** Which persona this serves | **Priority:** Must Have

#### User Stories (Given/When/Then format)
#### Behaviors (every action with exact values)
#### Edge Cases (5-10 per feature: condition → behavior)
#### Acceptance Criteria (testable by automated tests)

After all features, add:
### Non-Functional Requirements (performance, accessibility, data limits)
### Feature API Contracts (shared data models, cross-feature navigation)

## Rules
- NEVER suggest technologies, UI design, or database schemas
- Every behavior MUST have an explicit, testable outcome
- Every edge case MUST specify the expected behavior
- User stories MUST reference a persona from the vision
- If a feature is ambiguous, make a decision — do NOT leave it open
Notice the input validation. The PM skill checks the vision's status field before proceeding. If it's still draft, the PM refuses to work. This is the handoff protocol from Chapter 3 in action — roles enforce the contract.

The Feature Specification Template

Every MVP feature gets the same structured treatment. The template forces the PM to make every decision upfront. Each section feeds a specific downstream role:

Stories
The Architect uses stories to identify required screens, navigation flows, and state management patterns.
Behaviors
The Developer reads behaviors as implementation instructions: “search triggers after 300ms debounce” = code.
Edge Cases
The QA Engineer converts edge cases directly into test scenarios without interpretation.
Criteria
The Code Reviewer checks the implementation against acceptance criteria to decide PASS or FAIL.

Acceptance Criteria That AI Can Verify

If you can't write assertEquals() or assertTrue() for it, it's not testable:

Untestable Criteria
• Search should be fast
• The app should handle large recipe collections
• Error messages should be user-friendly
“Fast” is not a number. “Large” is not a number. “User-friendly” is not verifiable by code.
Testable Criteria
• Search returns results within 200ms for 1,000 recipes
• App supports up to 10,000 recipes without exceeding 100MB storage
• Error messages include: what went wrong, what the user can do
Every criterion has a number or a structure. The QA role can write a test for each one.
The “should” smell. Search your PM output for the word “should.” Every “should” is a decision the PM didn't make. Replace each one with a specific value: “search should be fast” becomes “search returns results within 200ms.”

Edge Case Enumeration

This is where AI PMs outperform human PMs. A human produces 3-5 obvious edge cases. An Opus 4.x PM produces 15-20, including scenarios humans rarely consider:

Product Manager
Edge Cases — Recipe CRUD (excerpt)
### Edge Cases: Create Recipe

1. **Empty title** → Show validation error "Recipe title is required"
2. **Title > 200 chars** → Truncate at 200, show warning
3. **No ingredients** → Allow save (some recipes are just instructions)
4. **Duplicate title** → Allow (recipes can share names)
5. **Ingredient quantity = 0** → Show warning but allow save
6. **Ingredient with no unit** → Default to "piece(s)"
7. **Steps with no content** → Remove empty steps on save
8. **Paste 50 ingredients at once** → Process all, no limit
9. **Recipe with only a title** → Allow save as minimal recipe
10. **Special characters in title** → Allow all Unicode, escape for search
11. **Create while offline** → Save locally, no sync needed (offline-first)
12. **Create during low storage** → Check storage before save, warn if < 50MB
13. **Rapid create (spam save button)** → Debounce, prevent duplicates
Edge cases prevent re-dispatches. Items 8, 10, 12, and 13 rarely appear in a human PM's initial spec but surface in QA testing. Every edge case the PM catches upfront is one fewer Developer-Reviewer loop — saving time and tokens.

User Stories With Testable Outcomes

The PM uses Given/When/Then because this format maps directly to test code:

Product Manager
User Stories — Recipe Search (excerpt)
### Feature: Recipe Search
**Persona:** Maria (weeknight dinner planner)

#### User Stories

- As Maria, I want to search recipes by title so that I can quickly
  find a specific recipe I remember.
  - **Given** 50 saved recipes, **When** I type "pasta" in the search
    bar, **Then** I see all recipes with "pasta" in the title within 200ms
  - **Given** no saved recipes, **When** I type any query, **Then**
    I see "No recipes yet. Add your first recipe!" with a create button

- As Maria, I want to search by ingredient so that I can find recipes
  for what I have in the fridge.
  - **Given** recipes containing "chicken," **When** I search "chicken,"
    **Then** results include recipes with "chicken" in any ingredient
  - **Given** a search for "chkn" (typo), **When** no exact match
    exists, **Then** I see "No recipes found" (no fuzzy matching in MVP)

Every story references a persona from the vision, keeping features grounded in real user needs. The QA Engineer converts each Given/When/Then directly into a test case with minimal interpretation.

Non-Functional Requirements

NFRs apply to the whole app, not any single feature. The PM defines them as measurable targets:

Product Manager
Non-Functional Requirements (excerpt)
### Non-Functional Requirements

#### Performance
- Cold start: < 2 seconds on mid-range device (Pixel 6a)
- Screen transitions: < 300ms
- Search results: < 200ms for 1,000 recipes

#### Accessibility
- All images have content descriptions
- Minimum touch target: 48dp × 48dp
- Screen reader navigation order matches visual order

#### Data Limits
- Max recipes: 10,000 (warn at 8,000)
- Max ingredients per recipe: 100
- Max title length: 200 characters
- Max photo size: 5MB (compressed to 1MB on save)

Every number here becomes a constant or test assertion downstream. The Architect designs the data layer to meet these limits. The Developer implements validation. The QA Engineer writes performance tests against them.

API Contracts Between Features

Features don't exist in isolation. Search queries recipes, books reference recipes, grocery lists read ingredients. The PM defines how features communicate — without specifying the technology:

Product Manager
Feature API Contracts (excerpt)
### Feature API Contracts

#### Shared Data: Recipe
- id (unique, auto-generated)
- title (string, required, max 200 chars)
- ingredients (list of: name, quantity, unit)
- steps (ordered list of: step number, instruction text)
- recipe_book_id (optional, links to a recipe book)
- tags (list of strings)
- created_at, updated_at (timestamps)

#### Cross-Feature Navigation
- Recipe List → Detail: navigates with recipe ID, back preserves scroll
- Recipe List → Search: search replaces list content, clear restores it
- Recipe → Book: recipe belongs to 0-1 books, deleting book orphans recipes
API contracts prevent integration bugs. Without the shared Recipe shape, each module might model “recipe” differently. The PM defines the data contract; the Architect implements it.

Running the PM Skill

The orchestrator dispatches the PM after the vision is approved. Here's the execution trace:

Orchestrator → Product Manager with artifacts/vision.md (status: approved)
PM reads vision → validates frontmatter → extracts MVP features
PM produces artifacts/specs.md (4 MVP features × 5 sections each)
Output: ~4,000 tokens • 4 feature specs + NFRs + API contracts
PM dispatch: ~3 min • under $0.10 (Opus 4.7) • Ready for Architect
Why no PM checkpoint? The PM's output is a structured expansion of the already-approved vision. If the vision is right, the specs will be right. If you want to review specs anyway, add a checkpoint in your orchestrator skill. The pipeline is configurable.

The PM Output Contract

The Architect expects this exact structure. Missing items trigger a re-dispatch:

The technology leak. The most common PM failure: slipping in technology decisions. “Store recipes in a local database” is fine (what). “Store recipes in Room” is a technology leak (how). The PM defines the need; the Architect picks the tool. Add this to your PM skill's rules section to prevent it.

Exercise: Build and Test Your PM Skill

  1. Create the skill file. Copy the PM skill from this chapter into .claude/commands/product-manager.md in your project.
  2. Run it against your vision. Make sure you have an approved artifacts/vision.md from Chapter 4. Invoke /product-manager.
  3. Count the edge cases. Did the PM find at least 5 per feature? If not, add a rule: “List at least 8 edge cases per feature including boundary values, empty states, and concurrent operations.”
  4. Check for “should” smell. Search the output for “should,” “probably,” and “might.” Replace each with a specific value or behavior.
  5. Verify the contract. Run through the output contract checklist above. Every item must be checked or the Architect won't have what it needs.
  6. Customize for your domain. API-dependent apps need request/response contracts. Real-time apps need latency budgets. Multi-user apps need conflict resolution specs. Add the sections your Architect needs.
