Chapter 10

Security Review and QA Engineering

Two final quality gates: vulnerability scanning and comprehensive test generation

The Last Two Gates

The Code Reviewer checked architecture rules. Code compiles, tests pass, naming is consistent. But two questions remain: Is it secure? and Is it thoroughly tested?

This chapter builds the final two roles — the Security Reviewer and the QA Engineer. Together with the Code Reviewer, they form the quality stack: rules compliance, security posture, and test coverage. All three must pass before code ships. For deep dives into mobile security, see App Security For Dummies.

The Security Reviewer Role

Security Reviewer
Input: Source code from Developer, dependency list (build.gradle), architecture.md (for data flow), task spec (for context)
Output: Security findings with OWASP category, severity (CRITICAL / HIGH / MEDIUM / LOW), location, explanation, and remediation steps
Model: Opus 4.7 — security judgment requires understanding intent, not just pattern matching

The Security Reviewer only runs on high-risk tasks (AppDatabase, navigation, Hilt wiring, integration) — typically 4 of 20 tasks. For an offline-first app like RecipeVault, the attack surface is smaller than a networked app, but data storage, input handling, and export features still need scrutiny.

The Security Reviewer Skill

Skill File — Security Reviewer 
# Identity
You are the Security Reviewer for the AI Factory pipeline.
Check Developer output for security vulnerabilities. Model: Opus.

# Input
1. Source files from the Developer
2. build.gradle.kts (dependency versions)
3. architecture.md (data flow context)
4. Task spec (what this code does)

# Review Checklist
For each file, check against these categories:

## 1. Data Storage (OWASP M2)
- No plaintext secrets, hardcoded keys, or logged PII
- Sensitive data: EncryptedSharedPreferences or SQLCipher
- No raw SharedPreferences for tokens/passwords

## 2. Input Validation (OWASP M8)
- All user input sanitized before DB queries
- No raw SQL — Room parameterized @Query only
- File paths validated (no path traversal)

## 3. Data Export (OWASP M1)
- Content providers require permissions
- FileProvider paths scoped (not root)
- Backup rules exclude sensitive data

## 4. Dependencies (OWASP M7)
- No known CVEs in dependency versions
- No unnecessary permissions in manifest

## 5. Cryptography
- No custom crypto — use Android Keystore
- Minimum TLS 1.2 for network calls

# Severity Levels
- CRITICAL: Exploitable vulnerability. Data leak, injection,
  exposed secrets. Must fix before merge.
- HIGH: Likely exploitable with effort. Missing encryption,
  overly broad permissions. Must fix.
- MEDIUM: Defense-in-depth gap. Missing input validation
  on non-sensitive field, verbose logging. Should fix.
- LOW: Best practice suggestion. Code is safe but could
  be hardened. Fix if time allows.

# Output Format
## Security Review: Task [N] — [Title]
### Verdict: [SECURE | FIX-REQUIRED | BLOCK]
### Findings
- **[SEVERITY]**: [Summary]
  - Category: OWASP [MN] — [Name]
  - File: path.kt | Line: N
  - Issue: [What's exploitable and how]
  - Remediation: [Specific fix with code example]
### Summary
- Files scanned: N
- CRITICAL: N | HIGH: N | MEDIUM: N | LOW: N

Live Security Review: Task 17 (AppDatabase)

Task 17 creates the Room database, type converters, and migration strategy. This is a high-risk task because it handles all data persistence.

Security Review Output — Task 17 
## Security Review: Task 17 — AppDatabase & Migrations
### Verdict: FIX-REQUIRED

### Findings
- **HIGH**: Database not encrypted — recipe data in plaintext
  - Category: OWASP M2 | File: data/local/db/RecipeVaultDatabase.kt
  - Issue: Default SQLite. Rooted device or extracted backup
    exposes all recipe data.
  - Remediation: Add SQLCipher + SupportFactory for encrypted Room.

- **MEDIUM**: Type converter logs full recipe content at DEBUG level
  - Category: OWASP M2 | File: data/local/db/Converters.kt
  - Issue: Log.d() prints ingredient lists with possible PII.
  - Remediation: Remove debug logging or use Timber release tree.

- **LOW**: fallbackToDestructiveMigration causes silent data loss
  - Category: OWASP M2 | File: data/local/db/RecipeVaultDatabase.kt
  - Issue: Schema change deletes all recipes without warning.
  - Remediation: Write explicit Migration objects.

### Summary
- Files scanned: 4
- CRITICAL: 0 | HIGH: 1 | MEDIUM: 1 | LOW: 1
CRITICAL vs. HIGH for offline apps: For RecipeVault (no network, no auth), most storage findings are HIGH rather than CRITICAL. An unencrypted database on a non-rooted device requires physical access to exploit. For a banking app, the same finding would be CRITICAL.

Security Verdicts

SECURE
0 CRITICAL, 0 HIGH, ≤2 MEDIUM. Code can advance. LOW findings logged for future hardening.
FIX-REQUIRED
Any HIGH, or 3+ MEDIUM. Send findings back to Developer for remediation. Same fix loop as Code Reviewer.
BLOCK
Any CRITICAL finding. Escalate to human immediately. CRITICAL = exploitable vulnerability that must be reviewed by a security-aware human before any fix is attempted.
CRITICAL findings always go to a human. The Developer role shouldn't attempt to fix exploitable vulnerabilities on its own — the fix itself could introduce new vulnerabilities. Security-critical code needs human judgment.

The Factory Itself Has an Attack Surface

The OWASP Mobile categories above protect the recipes app. They don't protect the AI Factory pipeline that built it. The pipeline runs skill files, dispatches sub-agents, reads task specs, and writes code — all of which are themselves a target. Two newer OWASP frameworks apply specifically:

OWASP Top 10 for LLM Applications (2025 v2.0)

  • LLM01 Prompt Injection — a poisoned task spec, architecture doc, or imported file can hijack a Developer or Reviewer dispatch
  • LLM02 Sensitive Information Disclosure — secrets in CLAUDE.md or skill files leaking into outputs
  • LLM05 Improper Output Handling — the orchestrator running un-sanitized AI output through ./gradlew
  • LLM06 Excessive Agency — sub-agents that can run shell commands or write files outside the project

OWASP Top 10 for Agentic Applications (2026)

  • Goal misalignment between the orchestrator and a sub-agent
  • Tool misuse — agent calling a tool outside its mandate
  • Inter-agent communication — unauthenticated handoffs between roles
  • Persistent memory poisoning — tainted context carried across dispatches
  • Least-Agency principle — only grant the minimum autonomy each role needs
Practical mitigations for the Factory: scope each role's tool access to what it actually needs (Architect should not be able to run shell commands; QA should not be able to push to git), validate every artifact handoff against its contract before passing to the next role, treat any external input (third-party docs, GitHub issues, fetched URLs) as untrusted, and keep human checkpoints at the Architect and pre-merge stages so no single dispatch can write production code unattended. See OWASP LLM Top 10 and OWASP Agentic Top 10.

The QA Engineer Role

QA Engineer
Input: Source code + existing tests from Developer, specs.md (acceptance criteria), rules.md (testing rules)
Output: Additional test files (edge cases, integration tests), coverage report, test quality assessment, "ship readiness" verdict
Model: Sonnet 4.6 by default; Haiku 4.5 for high-volume test generation runs — test generation is mechanical, following patterns from the Developer's existing tests

Unlike other quality roles, the QA Engineer doesn't just evaluate — it generates new tests. The Developer writes one test per public class (the minimum). The QA Engineer adds edge case, integration, and scenario tests.

The QA Engineer Skill

Skill File — QA Engineer 
# Identity
You are the QA Engineer for the AI Factory pipeline.
Generate comprehensive tests beyond the Developer's minimum.
Model: Sonnet.

# Input
1. Source files from the Developer
2. Existing test files from the Developer
3. specs.md — acceptance criteria and edge cases
4. rules.md — testing conventions

# Test Generation Strategy
1. READ existing tests. Identify what's covered.
2. READ acceptance criteria from specs.md.
3. GENERATE tests for each gap:

## Edge Cases (priority 1)
- Empty inputs, null values, max-length strings
- Boundary values (0 items, 1 item, max items)
- Special characters in recipe names/ingredients
- Unicode and emoji in text fields

## State Transitions (priority 2)
- Loading → Success, Loading → Error
- Error → Retry → Success
- Empty → First Item Added → Non-Empty

## Integration Tests (priority 3)
- ViewModel + Repository (fake) end-to-end
- Database operations with in-memory Room
- Navigation actions trigger correct destinations

## Regression Guards (priority 4)
- One test per bug found by Code Reviewer
- One test per security finding remediation

# Output Format
## QA Report: Task [N] — [Title]
### Tests Generated
- File: [path/to/NewTest.kt]
  - [N] edge case tests
  - [N] state transition tests
  - [N] integration tests

### Coverage Assessment
- Public classes with tests: N/N (before → after)
- Acceptance criteria with tests: N/N
- Edge cases from specs covered: N/N

### Verdict: [SHIP | MORE-TESTS | BLOCK]
- SHIP: All acceptance criteria tested, ≥80% edge cases.
- MORE-TESTS: Missing critical path coverage. Re-run.
- BLOCK: Untestable code. Needs Developer refactor.

QA in Action: Task 9 (RecipeList ViewModel)

The Developer wrote a happy-path test for RecipeListViewModel. The QA Engineer generates the tests the Developer missed:

QA-Generated Tests — RecipeListViewModel 
// Edge case: empty list
@Test
fun `initial state with no recipes shows empty state`() {
    val vm = RecipeListViewModel(FakeRecipeRepository(emptyList()))
    assertIs<RecipeListUiState.Empty>(vm.uiState.value)
}

// Edge case: boundary value
@Test
fun `recipe with 500-char name displays without crash`() {
    repository.emit(listOf(RecipeFixtures.recipe(title = "A".repeat(500))))
    assertIs<RecipeListUiState.Success>(vm.uiState.value)
}

// State transition: loading → error
@Test
fun `repository error transitions to error state`() {
    repository.emitError(IOException("DB locked"))
    assertIs<RecipeListUiState.Error>(vm.uiState.value)
}

// State transition: error → retry → success
@Test
fun `retry after error loads recipes successfully`() {
    repository.emitError(IOException("timeout"))
    vm.retry()
    repository.emit(RecipeFixtures.sampleRecipes())
    assertIs<RecipeListUiState.Success>(vm.uiState.value)
}

// Integration: delete refreshes list
@Test
fun `deleting recipe removes it from list`() {
    repository.emit(RecipeFixtures.sampleRecipes()) // 3 recipes
    vm.deleteRecipe(RecipeFixtures.sampleRecipes().first().id)
    assertEquals(2, (vm.uiState.value as Success).recipes.size)
}

The Developer wrote 1 test. The QA Engineer generated 5 more covering edge cases and state transitions the spec requires but the Developer didn't test.

QA model selection in 2026. Sonnet 4.6 is the default (~$0.10/task × 20 tasks = ~$2.00 added to the pipeline). Test generation is mechanical and pattern-following, so it's also a strong Haiku 4.5 candidate at $1/$5 per MTok — exactly 3× cheaper than Sonnet, and 4–5× faster on typical workloads. Routing high-volume QA to Haiku 4.5 drops the QA bill from ~$2.00 to ~$0.60 with no quality loss on the boilerplate test cases that dominate the run. Keep Sonnet 4.6 for tasks that need cross-state-machine reasoning (complex ViewModel test scenarios, integration paths).

The Ship Readiness Assessment

After all three quality roles pass, the orchestrator produces a ship readiness report — the final artifact before human review:

Ship Readiness Criteria

  • Code Reviewer: all tasks PASS (0 MUST findings remaining)
  • Security Reviewer: all high-risk tasks SECURE (0 CRITICAL, 0 HIGH remaining)
  • QA Engineer: all tasks SHIP (≥80% edge case coverage)
  • All 128 tests pass
  • Code compiles with zero warnings
  • No TODO comments in source code
Ship readiness is not ship decision. The pipeline produces a readiness report, but a human makes the ship decision. The human reviews the report, spot-checks code, and decides whether to merge. The pipeline reduces human work from "review everything" to "review the report + spot-check."

Where Humans Must Decide

The quality stack automates most checks, but some findings require human judgment:

1
CRITICAL security findings. Exploitable vulnerabilities need a security-aware human to verify the finding and approve the remediation.
2
REJECT verdicts from Code Reviewer. Structural failures mean the task spec or architecture may need revision — that's an Architect-level decision.
3
BLOCK verdicts from QA. Untestable code means the Developer produced tightly coupled code that needs refactoring, not more tests.
4
Business logic correctness. No AI role checks whether the recipe search algorithm actually returns useful results. That's a product judgment.

Exercise: Build Your Security Checklist

Customize the Quality Stack for Your Domain

  1. Copy both skill files (security-review.md and qa-engineer.md) into your project's .claude/commands/ directory.
  2. Security: Add domain-specific checks. If your app handles payments, add PCI-DSS checks. If it handles health data, add HIPAA storage requirements. Write each check with an Issue and Remediation example.
  3. QA: Adjust the test generation priorities. For a real-time app, prioritize concurrency tests. For a data-heavy app, prioritize boundary value tests.
  4. Run both skills against a high-risk task in your project. Compare the findings to what a manual security review and manual test planning would produce.
