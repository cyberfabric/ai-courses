Chapter 11

Code Review and Quality

Reviewing AI-generated code, catching common smells, the 10-point checklist

The Review Mindset

AI-generated code is like code from a talented junior developer: syntactically correct, often clever, but missing the context that experience provides. It compiles, it usually works on the happy path, and it looks reasonable at first glance. The bugs hide in the gaps between what you specified and what the AI assumed.

Your review mindset should be: "This code is guilty until proven innocent." Not because AI is bad at coding — it's often quite good. But because the cost of a missed bug in reviewed code is yours to bear, not the AI's. You are the last line of defense before production.

The Ownership Principle

You are responsible for every line of AI-generated code you commit. "The AI wrote it" is not an excuse in a post-mortem. If it's in your PR, it's your code. Review it with the same rigor you'd apply to code from a teammate whose work you're putting your name on.

Common AI Code Smells

AI-generated code has characteristic patterns that should trigger closer inspection. Learn to recognize these smells and you'll catch the majority of issues before they reach production.

1. Over-Engineering

AI loves abstractions. Ask for a simple utility function and you might get a factory pattern with dependency injection, three interfaces, and a configuration system. The code works, but it's 10x more complex than the task requires.

Backend You asked for a function to format user display names. You got a DisplayNameFormatterFactory with pluggable strategies, locale-aware formatters, and a caching layer. When a three-line function was all you needed:

# What you needed:
def format_display_name(first: str, last: str) -> str:
    return f"{first} {last}".strip()

# What AI generated (over-engineered):
class DisplayNameFormatter:
    def __init__(self, strategy: FormattingStrategy, cache: Cache):
        self._strategy = strategy
        self._cache = cache

    def format(self, user: User) -> str:
        cache_key = f"display_name:{user.id}"
        cached = self._cache.get(cache_key)
        if cached:
            return cached
        result = self._strategy.format(user.first_name, user.last_name)
        self._cache.set(cache_key, result, ttl=3600)
        return result

2. Hallucinated APIs

AI models sometimes generate calls to APIs, methods, or parameters that don't exist. They look plausible because they follow naming conventions of real APIs, but they'll fail at runtime.

Android Dev Watch for Compose functions that look right but don't exist: Modifier.fadeIn(), LazyColumn.stickyFooter {}, or rememberScrollableState() with parameters that the real API doesn't accept. Always verify that imported APIs actually exist in your dependency versions.

3. Inconsistent Patterns

AI may use different approaches in different parts of the same file: callbacks here, coroutines there. Result wrapping in one function, raw exceptions in the next. This happens when the model generates code section by section without maintaining a coherent style.

4. Missing Error Handling

AI often generates the happy path perfectly and either ignores error cases or handles them generically (catch-all exceptions, empty catch blocks, returning null where it shouldn't). Every try/catch block deserves scrutiny: is the catch clause actually handling the error correctly?

5. Stale or Wrong Dependencies

AI may import deprecated libraries, use old API versions, or add dependencies that conflict with your existing stack. Check every new import and dependency against your project's current versions. This is especially common in fast-moving ecosystems like Android Jetpack, where API surfaces change frequently between versions.

The 30-second sniff test: Before reading AI-generated code line by line, scan it quickly: How many files were created? How many new classes/functions? How many new dependencies? If any of these seem disproportionate to the task, something is over-engineered.

Security Review

AI-generated code frequently contains security vulnerabilities. Not because the AI is malicious, but because security is a constraint that needs to be explicitly specified — and often isn't.

Common AI Security Issues

Vulnerability What AI Does What to Check
SQL Injection Uses string interpolation in queries All database queries use parameterized statements
Hardcoded Secrets Puts API keys, passwords in source code Secrets come from env vars or secret managers
Missing Auth Creates endpoints without authentication checks Every endpoint has appropriate auth middleware
XSS Renders user input without sanitization All user input is escaped before rendering
Excessive Data Returns full objects including sensitive fields Response models exclude passwords, tokens, internal IDs
Insecure Defaults Disables SSL verification, uses HTTP instead of HTTPS All external connections use secure defaults

Full-Stack AI-generated React code sometimes uses dangerouslySetInnerHTML to render dynamic content. AI-generated Node.js code sometimes uses eval() or child_process.exec() with unsanitized input. These are XSS and command injection vectors that look like reasonable solutions until you think about malicious input.

Backend Here's a real example of AI-generated code that looks fine but has multiple security issues:

# AI-generated endpoint — spot the problems
@app.get("/api/users/{user_id}")
async def get_user(user_id: int):
    user = await db.execute(
        f"SELECT * FROM users WHERE id = {user_id}"
    )
    return user.first()

# Issues:
# 1. SQL injection via f-string interpolation
# 2. SELECT * exposes password_hash, reset_token, etc.
# 3. No authentication — anyone can access any user
# 4. No authorization — users can see other users' data
# 5. Returns raw DB row, not a response model

# Fixed version:
@app.get("/api/users/{user_id}", response_model=UserPublicResponse)
@require_auth
async def get_user(user_id: int, current_user: User = Depends(get_current_user)):
    if current_user.id != user_id and not current_user.is_admin:
        raise HTTPException(status_code=403)
    user = await db.execute(
        select(User.id, User.name, User.email).where(User.id == user_id)
    )
    result = user.first()
    if not result:
        raise HTTPException(status_code=404)
    return result
Add security rules to CLAUDE.md. Explicitly tell the AI what it must never do: "NEVER use string interpolation in SQL queries. NEVER hardcode secrets. NEVER disable SSL verification. ALWAYS use parameterized queries. ALWAYS add authentication to new endpoints." These rules prevent many security issues at generation time, but never replace manual security review.

Performance Review

AI optimizes for correctness, not performance. It will generate code that works but may be needlessly slow. Common performance issues to watch for:

  • N+1 queries: A loop that makes one database query per iteration instead of a batch query
  • Missing pagination: Loading entire tables into memory instead of paginating
  • Redundant computation: Recalculating values that could be cached or computed once
  • Blocking the main thread: Synchronous operations where async is needed (especially in Android and Node.js)
  • Unbounded collections: SELECT * without LIMIT, lists that grow without bounds

Android Dev AI-generated Compose code often triggers unnecessary recompositions: using unstable types in state, creating new lambda instances on every recomposition, or putting expensive computations directly in composable functions instead of remember or derivedStateOf. These won't crash your app but will make it feel sluggish.

// AI-generated: creates a new lambda on every recomposition
@Composable
fun UserList(users: List<User>, onUserClick: (User) -> Unit) {
    LazyColumn {
        items(users) { user ->
            // BAD: new lambda instance every recomposition
            UserCard(user = user, onClick = { onUserClick(user) })
        }
    }
}

// Better: use key parameter and remember the callback
@Composable
fun UserList(users: List<User>, onUserClick: (User) -> Unit) {
    LazyColumn {
        items(users, key = { it.id }) { user ->
            UserCard(user = user, onClick = { onUserClick(user) })
        }
    }
}

DevOps AI-generated Terraform sometimes creates oversized resources (db.r6g.2xlarge when db.r6g.large suffices), missing auto-scaling configurations, or leaving logging at verbose levels that generate excessive CloudWatch costs. Always check resource sizing and cost implications in AI-generated infrastructure code — the AI defaults to "safe" (large) rather than "efficient" (right-sized).

Freelancer Performance issues in AI-generated code are particularly costly for client projects. A client who sees a slow app doesn't care that "the AI generated it efficiently for correctness." Profile AI-generated code on realistic data volumes before delivering. A list that works with 10 items but freezes with 1,000 is a bug the review should catch.

Architecture Review

The subtlest and most important review: does the AI-generated code fit your existing architecture?

  • Layer boundaries: Does the code respect your layered architecture? Routes shouldn't access repositories directly. ViewModels shouldn't import UI framework classes.
  • Pattern consistency: Does it use the same patterns as existing code? If your app uses sealed interfaces for UI state, the new code shouldn't use data classes with nullable fields.
  • Dependency direction: Do dependencies point the right way? Domain code shouldn't depend on framework code.
  • Module boundaries: Does new code stay within its module? A feature module shouldn't reach into another feature's internals.
  • Naming conventions: Do names follow your project's conventions? Services, repositories, ViewModels, routes should all follow existing naming patterns.

Team Lead Architecture violations are the most expensive AI code smells because they compound. One bypassed layer boundary becomes the template for the next ten features. Catch them early. Add architectural rules to CLAUDE.md and enforce them in PR review. Consider adding architecture tests (like ArchUnit for JVM projects) that automatically catch layer violations.

Android Dev Common Android architecture violations by AI: a ViewModel that imports android.widget.Toast (UI framework leak), a repository that directly accesses SharedPreferences instead of going through a DataStore abstraction, a Compose screen that creates its own ViewModel instance instead of receiving it via Hilt, or a data layer class that references Compose state types.

Use AI to check AI. After the agent generates code, ask it to review its own output against your architecture rules: "Review the code you just generated against the architecture rules in CLAUDE.md. Flag any violations of layer boundaries, naming conventions, or dependency direction." This catches 60-70% of architecture issues before you even start reading.

The 10-Point Review Checklist

Use this checklist for every AI-generated PR. Print it, pin it, make it part of your PR template.

AI Code Review Checklist

  • 1. Correctness: Does the code actually do what was specified? Walk through the acceptance criteria.
  • 2. Security: No SQL injection, XSS, hardcoded secrets, missing auth, or excessive data exposure.
  • 3. Performance: No N+1 queries, unbounded collections, missing pagination, or blocking operations.
  • 4. Architecture: Respects layer boundaries, dependency direction, and module separation.
  • 5. Pattern consistency: Uses the same patterns, naming conventions, and approaches as existing code.
  • 6. Error handling: All error paths handled appropriately. No empty catch blocks or swallowed exceptions.
  • 7. Edge cases: Nulls, empty inputs, boundary values, concurrent access, and failure modes are handled.
  • 8. Tests: Meaningful tests exist that test behavior, not implementation. Edge cases and error paths are covered.
  • 9. Scope: The AI didn't add unrequested features, refactor unrelated code, or introduce new patterns.
  • 10. Dependencies: No new unnecessary dependencies. Existing dependencies used at correct versions. No hallucinated APIs.

Not every item applies to every change. A one-line bug fix doesn't need an architecture review. Scale the checklist to the size and risk of the change. For high-risk changes (auth, payments, data migrations), apply every point. For low-risk changes (formatting, docs, simple refactors), focus on correctness and scope.

Team Workflows for AI Code Review

PR Template for AI-Assisted Code

## Summary
[What this PR does and why]

## AI Assistance
- [ ] Primarily AI-generated
- [ ] AI-assisted (human + AI collaboration)
- [ ] Human-written (AI used only for review/suggestions)

## Review Checklist (for AI-generated code)
- [ ] Correctness verified against spec/requirements
- [ ] Security review: no injection, secrets, or auth gaps
- [ ] Performance: no N+1 queries or unbounded operations
- [ ] Architecture: respects existing patterns and layers
- [ ] Tests: meaningful coverage of behavior and edge cases
- [ ] Scope: no unrequested changes or new dependencies

## How to Test
[Steps to verify the changes]

Team Lead Make the "AI Assistance" section mandatory. It signals to reviewers how much scrutiny the code needs. Primarily AI-generated code gets the full 10-point checklist. Human-written code with minor AI suggestions gets a lighter review. This transparency builds trust and calibrates effort.

Review Guidelines for the Team

  • Don't trust "the AI wrote it" as a quality signal. Review AI code with the same rigor as human code.
  • Read the spec first, then the code. Review against the specification, not just the code's internal logic.
  • Run the code locally before approving. Build it, test it, click through it. AI code that "looks right" may not behave right.
  • Check the diff size. If a simple feature produced a 500-line diff, something may be over-engineered.
  • Look for scope creep. AI often "helpfully" refactors adjacent code or adds features you didn't request.

Code Health Assessment

CodeScene's research (Adam Tornhill, February 2026) recommends assessing code health before deploying AI changes. Their finding: AI performs best when modifying healthy code and tends to amplify problems in unhealthy code.

9.5+
Target code health score (out of 10) before applying AI changes to a module
2x
More defects introduced by AI in low-health code vs high-health code

Before letting an AI agent loose on a module, ask: is this code healthy enough to receive AI modifications? If the module is already tangled, poorly tested, and full of tech debt, AI will generate code that's consistent with the mess — making it worse. Consider cleaning up the module first, then applying AI.

The health-first strategy: Use AI to improve code health (add tests, extract functions, simplify logic) before using AI to add features. Healthy code produces healthier AI output. It's a virtuous cycle: cleaner code → better AI suggestions → even cleaner code.

Exercise: Review AI-Generated Code

Practice the 10-point review checklist on real AI-generated code:

  1. Generate a non-trivial feature with AI. Use Claude Code to implement something substantial: a new API endpoint with tests, a Compose screen with ViewModel, or a data pipeline with error handling.
  2. Apply the 10-point checklist systematically. For each point, write down:
    • Pass/Fail
    • Specific issues found (if any)
    • The exact line or code block with the issue
  3. Categorize your findings: How many issues are security-related? Performance? Architecture? Pattern consistency?
  4. Fix the issues — either manually or by re-prompting the AI with specific corrections.
  5. Reflect: Which checklist items caught the most issues? Which are hardest to evaluate? Where does AI-generated code in your project tend to fall short?

Run this exercise 3-5 times across different types of changes. You'll develop an intuition for where AI code typically needs correction in your specific project and stack.
