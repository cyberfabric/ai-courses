Test-Driven Development with AI

AI writes tests from specs, then implements to pass them — TDD without the tedium

The Classic TDD Problem

TDD has always been the discipline most developers admire in theory and abandon in practice. The idea is sound: write the tests first, then write code to make them pass. The reality: writing tests is slow, repetitive, and feels like it's slowing you down when you already know what the implementation should be.

AI changes this equation completely. The slow part — writing test boilerplate, setup code, assertion scaffolding, edge case permutations — is exactly what AI excels at. The hard part — deciding what to test and what the correct behavior is — still requires your judgment. AI-powered TDD gives you the discipline of test-first development without the tedium that made you abandon it.

The AI-Powered TDD Workflow

The traditional TDD cycle is Red → Green → Refactor. The AI-powered version adds spec-driven test generation:

Write Spec
→
AI Generates Tests
→
Review Tests
→
AI Implements
→
All Green
  1. Write the spec — Define what the code should do, its edge cases, and its constraints (Chapter 7).
  2. AI generates tests — Feed the spec to AI: "Write tests for this behavior." Tests are written against the spec, not the implementation.
  3. Review the tests — This is your critical checkpoint. Are the tests testing the right things? Are edge cases covered? Are assertions meaningful?
  4. AI implements — Feed the failing tests to AI: "Make these tests pass." The tests serve as an executable specification.
  5. Verify — All tests pass, the implementation matches the spec, and you review the code for quality.

Why Tests-First Matters Even More with AI

When AI writes the implementation, tests are your safety net against hallucinated behavior. Without tests, you're trusting the AI got it right. With tests written from the spec, you have automated verification that the code does what you specified — not what the AI assumed.

Generating Tests from Specs

The quality of AI-generated tests depends entirely on what you ask for. The prompt structure matters.

Android Dev Kotlin / JUnit Tests

$ claude "Write unit tests for the FavoriteRepository based on the
  spec in docs/specs/offline-favorites.md.

  Test framework: JUnit 5 + MockK + Turbine (for Flow testing)
  Follow patterns in: data/repository/CartRepositoryTest.kt

  Test these behaviors:
  - Adding a favorite stores it locally with syncStatus=PENDING
  - Removing a favorite marks it as DELETED locally
  - getFavorites() returns a Flow that updates when favorites change
  - syncFavorites() uploads PENDING items and deletes DELETED items
  - Sync conflict: server version wins (overwrite local)
  - Network failure during sync: items stay PENDING for retry

  Do NOT test Room database internals — mock the DAO."

The resulting test file gives you executable requirements:

class FavoriteRepositoryTest {

    private val favoriteDao = mockk<FavoriteDao>()
    private val favoritesApi = mockk<FavoritesApi>()
    private val repository = FavoriteRepository(favoriteDao, favoritesApi)

    @Test
    fun `adding favorite stores with PENDING sync status`() = runTest {
        val productId = "product-123"

        repository.addFavorite(productId)

        coVerify {
            favoriteDao.insert(match {
                it.productId == productId &&
                it.syncStatus == SyncStatus.PENDING
            })
        }
    }

    @Test
    fun `sync uploads pending items to server`() = runTest {
        val pending = listOf(
            FavoriteEntity("p1", SyncStatus.PENDING),
            FavoriteEntity("p2", SyncStatus.PENDING)
        )
        coEvery { favoriteDao.getBySyncStatus(SyncStatus.PENDING) } returns pending
        coEvery { favoritesApi.addFavorite(any()) } returns Unit

        repository.syncFavorites()

        coVerify(exactly = 2) { favoritesApi.addFavorite(any()) }
        coVerify { favoriteDao.updateSyncStatus("p1", SyncStatus.SYNCED) }
        coVerify { favoriteDao.updateSyncStatus("p2", SyncStatus.SYNCED) }
    }

    @Test
    fun `network failure keeps items as PENDING`() = runTest {
        coEvery { favoriteDao.getBySyncStatus(any()) } returns listOf(
            FavoriteEntity("p1", SyncStatus.PENDING)
        )
        coEvery { favoritesApi.addFavorite(any()) } throws IOException()

        repository.syncFavorites()

        coVerify(exactly = 0) {
            favoriteDao.updateSyncStatus(any(), SyncStatus.SYNCED)
        }
    }
}

These tests now serve as an executable contract. When you tell the AI to implement FavoriteRepository, the tests define exactly what "correct" means.

Backend API and Integration Tests

For backend services, AI-generated tests shine at covering the full request-response cycle:

$ claude "Write integration tests for the PATCH /api/v2/users/{id}/preferences
  endpoint based on the spec in docs/specs/user-preferences.md.

  Test framework: pytest + httpx (async)
  Database: use the test database fixture from conftest.py
  Follow patterns in: tests/integration/test_users_profile.py

  Cover:
  - 200: valid update with all fields
  - 200: partial update (only theme)
  - 404: non-existent user ID
  - 422: invalid theme value ('rainbow')
  - 422: language code too long (>10 chars)
  - 401: missing auth token
  - 403: updating another user's preferences
  - Idempotency: same request twice produces same result"
@pytest.mark.asyncio
async def test_update_preferences_valid(client, test_user, auth_headers):
    response = await client.patch(
        f"/api/v2/users/{test_user.id}/preferences",
        json={
            "theme": "dark",
            "language": "en",
            "notifications": {"email": True, "push": False}
        },
        headers=auth_headers,
    )
    assert response.status_code == 200
    data = response.json()
    assert data["theme"] == "dark"
    assert data["notifications"]["push"] is False


@pytest.mark.asyncio
async def test_update_preferences_partial(client, test_user, auth_headers):
    response = await client.patch(
        f"/api/v2/users/{test_user.id}/preferences",
        json={"theme": "light"},
        headers=auth_headers,
    )
    assert response.status_code == 200
    assert response.json()["theme"] == "light"


@pytest.mark.asyncio
async def test_update_preferences_invalid_theme(client, test_user, auth_headers):
    response = await client.patch(
        f"/api/v2/users/{test_user.id}/preferences",
        json={"theme": "rainbow"},
        headers=auth_headers,
    )
    assert response.status_code == 422


@pytest.mark.asyncio
async def test_update_preferences_forbidden(client, test_user, other_auth_headers):
    response = await client.patch(
        f"/api/v2/users/{test_user.id}/preferences",
        json={"theme": "dark"},
        headers=other_auth_headers,
    )
    assert response.status_code == 403
Integration tests > unit tests for APIs. For endpoints, integration tests that hit the real route handler with a test database catch more real bugs than unit tests with mocked layers. Use AI to generate the integration tests first, then add unit tests for complex business logic.

What Makes AI-Generated Tests Good or Bad

Not all AI-generated tests are equal. Here's what to watch for during review:

Signs of Good AI Tests

  • Test names describe the behavior, not the implementation ("updates theme" not "calls setTheme")
  • Each test verifies one behavior (single assertion concept)
  • Edge cases are covered: nulls, empty inputs, boundary values, concurrent access
  • Error paths are tested: network failures, invalid input, auth failures
  • Tests are independent — no shared mutable state between tests
  • Setup is minimal and focused on what each test needs

Signs of Bad AI Tests

  • Testing implementation details: "verifies repository.save() is called" instead of checking the actual outcome
  • Tautological assertions: assert mock.called when you set up the mock to be called — this tests your mock, not your code
  • Missing edge cases: Only happy-path tests with no error scenarios
  • Brittle assertions: Testing exact string output instead of structural properties
  • Copy-paste tests: Same test repeated with trivially different inputs instead of parameterized tests
The dangerous anti-pattern: "Generate tests for this code." When you ask AI to write tests after the implementation exists, the AI often tests the implementation rather than the behavior. It sees that the function calls repository.save() and writes a test verifying that call, rather than testing whether the data actually persists correctly. Always generate tests from specs, not from code.

Coverage Strategies

AI can find test gaps you didn't think of. Use it as a coverage auditor:

# Find untested edge cases
$ claude "Read the tests in tests/unit/test_order_service.py and the
  implementation in src/services/order_service.py. Identify behaviors
  that are implemented but not tested. Focus on error paths, boundary
  conditions, and concurrent scenarios."

# Generate missing tests
$ claude "Based on your analysis, generate additional test cases for
  the gaps you identified. Follow the existing test patterns."

Full-Stack For frontend components, AI catches visual edge cases you might miss: "What happens when the user name is 200 characters long? What about right-to-left text? What about an empty state with no items?" Ask AI to generate snapshot or visual regression tests for these scenarios.

AI-Powered Mutation Testing

Mutation testing verifies that your tests actually catch bugs. The idea: mutate the code (change a < to >, remove a null check) and verify that tests fail. If they don't, the tests aren't testing what they claim to.

$ claude "Perform a manual mutation analysis on src/services/order_service.py.
  For each public method:
  1. Identify 3-5 mutations that would change behavior (flip conditions,
     change operators, remove error handling)
  2. Check whether existing tests would catch each mutation
  3. For mutations that aren't caught, write a new test that would catch it"
Coverage as behavioral guardrails. CodeScene's research recommends treating code coverage not as a metric to game but as behavioral guardrails that prevent AI from accidentally changing functionality during refactors. When AI modifies code, existing tests verify that behavior is preserved. Target meaningful coverage (testing real behaviors) over percentage targets.

Android testing has unique challenges: Compose UI tests, instrumented tests, and lifecycle-aware components. AI handles all three well when guided properly.

The Test Review Checkpoint

Before accepting AI-generated tests, run through this quick review:

  • Does each test name describe a behavior? If you read only the test names, do they form a readable spec?
  • Are assertions checking outcomes, not calls? Verify state changes, not that methods were called.
  • Are error paths tested? At minimum: invalid input, auth failure, network error, not found.
  • Run the tests with the implementation intentionally broken. Do they actually fail? A test that never fails is worse than no test.
  • Are tests independent? Shuffle the order — do they still pass?
  • Is the setup minimal? Each test should create only what it needs, not a full system state.

Team Lead Establish a team rule: AI-generated tests must be reviewed with the same rigor as AI-generated production code. Tests are code. Bad tests are worse than no tests because they create false confidence. Add "test quality review" to your PR template checklist.

Putting It All Together

Here's the complete AI-powered TDD workflow for a real feature:

# Step 1: Write the spec (you, 15 minutes)
# docs/specs/search-filters.md — acceptance criteria, edge cases

# Step 2: Generate tests from spec (AI, 2 minutes)
$ claude "Read docs/specs/search-filters.md and generate test files:
  - tests/unit/test_search_service.py (business logic)
  - tests/integration/test_search_api.py (endpoint behavior)
  Follow existing test patterns. Cover all acceptance criteria
  and edge cases from the spec."

# Step 3: Review tests (you, 10 minutes)
# Read the generated tests. Add missing cases. Fix bad assertions.
# Run them — they should all FAIL (no implementation yet).

# Step 4: Implement to pass tests (AI, 5 minutes)
$ claude "Implement the search filters feature to make all tests
  in tests/unit/test_search_service.py and
  tests/integration/test_search_api.py pass.
  Follow the spec in docs/specs/search-filters.md.
  Follow existing patterns in src/services/ and src/routes/."

# Step 5: Verify (you + CI, 5 minutes)
# Run the full test suite. Review the implementation.
# Check spec compliance. Commit.

Total time: ~37 minutes for a feature that would have taken 2-3 hours with manual testing. The tests exist from day one, covering edge cases you might have skipped under time pressure.

Freelancer This workflow is especially valuable for client projects where you need to deliver fast but can't afford to skip tests. AI-generated tests give you confidence that the feature works, protect against regressions, and show clients that quality is baked in — not bolted on.

Exercise: TDD an Untested Class

Find an existing class in your codebase that has no tests (every project has at least one). Apply the AI-powered TDD workflow:

  1. Write a mini-spec for the class: what should it do? What are the edge cases? What are the error conditions? (10 minutes)
  2. Generate tests from the spec using Claude Code. Explicitly tell it to test behaviors from the spec, not the current implementation. (2 minutes)
  3. Review the tests: Do they test behaviors, not implementation details? Are edge cases covered? Add or remove tests as needed. (10 minutes)
  4. Run the tests. Some should pass (existing behavior), some may fail (spec describes behavior the code doesn't implement). This reveals gaps between intent and implementation.
  5. Fix any failures by either updating the code (if the spec is right) or updating the spec (if the code is right).
  6. Perform a mutation check: Ask AI to identify 3 mutations that should cause test failures. Verify they do.

This exercise retrofits tests onto existing code using the spec-first approach. The class gets tested, and you practice the AI-TDD workflow on something real.
