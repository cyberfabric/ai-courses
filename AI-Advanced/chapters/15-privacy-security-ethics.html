Chapter 15

Privacy, Security, and Ethics

What not to share, secure AI coding, license compliance, and building team policies

The Responsibility Layer

AI makes you dramatically more productive. It also introduces risks that didn't exist when you wrote every line by hand. When you paste code into an AI tool, where does that data go? When AI generates code, who owns it? When AI introduces a vulnerability, who's liable? This chapter gives you practical answers and a policy framework you can apply immediately.

What NOT to Share with AI

Every prompt you send to a cloud-based AI model leaves your machine. Even tools with strong privacy policies (no-training commitments, data encryption) involve data transit. The safest data is data that never leaves.

Never send these to any AI tool:
  • API keys, tokens, and secrets — even in configuration files or environment examples
  • Passwords and credentials — including database connection strings with passwords
  • PII (Personally Identifiable Information) — real user data, email addresses, phone numbers
  • Proprietary algorithms — trade secrets, patented code, competitive advantages
  • Customer data — real transaction records, health data, financial information
  • Internal infrastructure details — server IPs, VPN configs, security architecture

Backend A common mistake: pasting a .env file or docker-compose.yml with real database credentials into an AI chat to debug a connection issue. Instead, sanitize first:

# BAD: Sending real credentials
DATABASE_URL=postgresql://admin:s3cr3tP@ss@prod-db.company.com:5432/users
PAYMENT_API_KEY=<your-real-live-key-here>
AWS_SECRET_ACCESS_KEY=<your-real-aws-secret-here>

# GOOD: Sanitized version
DATABASE_URL=postgresql://user:password@hostname:5432/dbname
PAYMENT_API_KEY=REDACTED
AWS_SECRET_ACCESS_KEY=REDACTED
Use .gitignore patterns in your CLAUDE.md. Add rules like: "NEVER read or display the contents of .env, secrets.yaml, credentials.json, or any file matching *.key or *.pem. If you need to reference environment variables, use placeholder values." This prevents the AI agent from accidentally loading secrets into context.

Data Sanitization Techniques

When you need AI help with code that touches sensitive data, sanitize the context before sharing.

Data Type Sanitization Approach Example
Credentials Replace with placeholders password=REDACTED
PII Use synthetic data user@example.com, Jane Doe
Real database records Use schema only, or generate fake data Share the model, not the data
Internal URLs Replace with generic URLs https://api.example.com
Business logic Abstract the pattern, share the structure "Calculate discount based on tier" (omit actual tiers/rates)

Android Dev Android apps often embed API keys in local.properties or BuildConfig. When asking AI for help with networking code, share the Retrofit interface and model classes — never the actual API keys or base URLs that point to production servers.

License Compliance

AI models are trained on open-source code. This creates legal gray areas around AI-generated output.

The Core Questions

  • Can AI-generated code violate open-source licenses? Potentially, yes. If the AI reproduces substantial portions of GPL-licensed code, your project may inherit GPL obligations.
  • Who owns AI-generated code? Generally, the person who prompted and directed the AI. But legal precedent is still evolving. Check your jurisdiction.
  • Can I use AI-generated code in commercial products? Yes, with the same diligence you'd apply to any code. Review it, understand it, and ensure it doesn't reproduce copyrighted material verbatim.

Practical Guidelines

  • Review AI output for verbatim reproduction. If a generated function looks suspiciously specific (exact variable names, unique comments), search for it online to check if it's copied from an open-source project.
  • Don't ask AI to reproduce specific copyrighted code. "Rewrite the React source code" is problematic. "Generate a virtual DOM diffing algorithm" is fine.
  • Document AI involvement. Note in your project that AI tools were used in development. This transparency helps with future compliance audits.
  • Use license-aware tools. GitHub Copilot has a filter that blocks suggestions matching public code. Enable it for commercial projects.
  • Apply your existing open-source policy. If your company has rules about using open-source code, extend them to AI-generated code. The risk profile is similar.

Team Lead Make license compliance part of your AI code review checklist. Add a step: "Does this AI-generated code look like it could be reproduced from a specific open-source project? If so, verify the license is compatible." This is the same diligence you'd apply if a developer copy-pasted from GitHub — the source just happens to be less visible.

Security Implications

Chapter 11 covered security in code review. This section addresses the systemic security risks of AI-driven development.

OWASP Top 10 in AI-Generated Code

AI-generated code frequently contains vulnerabilities from the OWASP Top 10. Here are the most common patterns and how to prevent them:

Note on OWASP Top 10:2025. The categories below use the OWASP Top 10:2021 numbering. The OWASP Top 10:2025 (confirmed January 2026) reshuffled the list — Software Supply Chain Failures is now A03, Security Misconfiguration rose to A02, SSRF was folded into Broken Access Control, and Mishandling of Exceptional Conditions entered at A10. The category-by-category mapping below is still durable; the patterns AI gets wrong haven't changed, only their position in the list.
OWASP Category AI Pattern Prevention
A01: Broken Access Control Generates endpoints without auth checks CLAUDE.md rule: "ALL endpoints require authentication middleware"
A02: Cryptographic Failures Uses MD5/SHA1 for password hashing, HTTP instead of HTTPS Specify: "Use bcrypt for passwords, TLS for all connections"
A03: Injection String interpolation in SQL, unsanitized shell commands CLAUDE.md rule: "ALWAYS use parameterized queries"
A04: Insecure Design Missing rate limiting, no input validation Include in specs: rate limits, input constraints, abuse scenarios
A07: Auth Failures Weak token generation, missing session management Reference existing auth patterns: "Follow our auth middleware"
A08: Data Integrity No input validation, trusts client-side data Spec requirement: "Server-side validation for all inputs"
A09: Logging Failures Logs sensitive data, insufficient audit trails CLAUDE.md rule: "NEVER log passwords, tokens, PII, or card numbers" 
# Add to your CLAUDE.md:
## Security Rules
- NEVER use string interpolation in database queries
- NEVER log passwords, tokens, PII, or payment card data
- NEVER hardcode secrets — use environment variables
- NEVER disable SSL/TLS verification
- ALWAYS add authentication to new API endpoints
- ALWAYS validate and sanitize user input server-side
- ALWAYS use parameterized queries (SQLAlchemy, Room @Query params)
- ALWAYS set secure defaults (HTTPS, secure cookies, CORS restrictions)
Security rules in CLAUDE.md are preventive, not sufficient. They catch the most common issues at generation time, but they're not a substitute for security review. Always run security-focused code review (Chapter 11) and consider automated security scanning tools (SAST/DAST) in your CI pipeline.

Navigating Corporate AI Policies

DevOps Many enterprises restrict which AI tools can be used, what data can be sent to them, and where the models are hosted. Before adopting AI tools on a corporate project:

  • Check your company's AI usage policy. Many organizations have explicit rules about which tools are approved.
  • Understand data residency requirements. Some industries (healthcare, finance, government) require data to stay in specific regions.
  • Verify the tool's data handling policy. Does it train on your code? Where is data stored? What's the retention policy?
  • Use enterprise plans when available. Enterprise tiers typically offer stronger data protection guarantees (no training, SOC 2, BAA).
  • Consider self-hosted alternatives. For the most sensitive codebases, self-hosted models (via Ollama, vLLM, or similar) keep data entirely on your infrastructure.

Freelancer When working on client projects, ask about their AI policy before using any tools. Some clients explicitly prohibit sending their code to third-party AI services. Others are fine with it. Get this in writing as part of your contract. Violating a client's AI policy is a trust-destroying event.

The Attribution Question

Who "wrote" AI-generated code? The legal and ethical landscape is evolving, but practical guidelines are clear:

Your Responsibility

  • You directed the AI
  • You reviewed the output
  • You committed it to the repository
  • You are accountable for bugs, vulnerabilities, and license issues

Appropriate Transparency

  • Note in PR descriptions when AI was heavily used
  • Document AI tools in your project's CONTRIBUTING.md
  • Don't claim AI-generated code as solely your own creative work
  • Be honest when asked about your development process

Team Lead Establish team norms around AI disclosure. Some teams add a checkbox to PR templates: "This PR includes AI-generated code." Others use commit message conventions: Co-Authored-By: Claude <noreply@anthropic.com>. The key is transparency without bureaucracy.

Bias in Code Generation

AI models reproduce patterns from their training data — including biased ones. Common manifestations in code:

  • Default assumptions: AI may generate user models with a binary gender field, assume Western name formats (first/last), or default to English-only UIs.
  • Example data bias: Generated sample data often skews toward certain demographics, regions, or cultural contexts.
  • Accessibility gaps: AI-generated UI code frequently lacks ARIA labels, screen reader support, keyboard navigation, and proper color contrast ratios.
  • Language and locale: Date formats, currency handling, and text direction may default to US English patterns.
Add inclusivity to your specs. When writing specs (Chapter 7), include requirements for internationalization, accessibility, and inclusive design. "Support RTL languages." "Include ARIA labels on all interactive elements." "Use culturally neutral sample data." These requirements flow through to AI-generated code when they're part of the spec.

Environmental Considerations

AI inference consumes energy. Every prompt you send, every code generation request, every long agentic session draws computational resources in a data center.

This doesn't mean you should stop using AI — the productivity gains are real and significant. But it does mean being intentional: don't run the AI in a loop generating code you throw away. Don't use the largest model when a smaller one suffices for the task. Don't spam the same prompt hoping for a different result. Efficient AI usage is both good engineering practice and responsible resource consumption.

Building an AI Usage Policy for Your Team

Team Lead Every team using AI tools should have an explicit, written policy. Here's a template you can adapt:

# AI Usage Policy — [Team Name]

## Approved Tools
- Claude Code (Max plan — enterprise data protection)
- GitHub Copilot (enterprise tier)
- Cursor (team plan)
- Android Studio Gemini (included)

## Data Rules
- NEVER send: credentials, PII, customer data, proprietary algorithms
- ALWAYS sanitize: env files, database dumps, config files
- OK to send: source code, test files, documentation, build configs

## Code Quality
- AI-generated code follows the same review standards as human code
- Use the 10-point review checklist (Chapter 11) for AI-generated PRs
- All AI-generated code must have tests

## Disclosure
- PRs should indicate level of AI assistance (template checkbox)
- CLAUDE.md and .cursorrules are team resources — keep them updated

## Security
- Security rules in CLAUDE.md are mandatory
- AI-generated code must pass security scanning in CI
- Quarterly review of AI-generated code for vulnerability patterns

## Compliance
- Check license compatibility for AI suggestions that look specific
- Document AI tool usage for audit trail
- Review this policy quarterly and update as tools evolve

The Policy Principle

The best AI policy is one that enables productivity while preventing the specific risks your team faces. Too restrictive and people work around it. Too permissive and incidents happen. Start with the template above, customize for your context, and iterate based on real incidents and team feedback.

Exercise: Security and Privacy Audit

Audit your recent AI interactions for potential security and privacy issues:

  1. Review your last 5 AI interactions (Claude Code sessions, Cursor chats, ChatGPT conversations). For each, check:
    • Did you share any credentials, API keys, or secrets?
    • Did you share any PII or customer data?
    • Did you share any proprietary business logic?
  2. Review the AI-generated code from those sessions. For each piece:
    • Does it have any OWASP Top 10 vulnerabilities?
    • Does it hardcode any secrets or use insecure defaults?
    • Does it handle errors securely (no stack traces leaked to users)?
  3. Create a CLAUDE.md security section for your project with rules that would have prevented any issues you found.
  4. Draft a one-page AI usage policy for your team based on the template in this chapter. Share it for feedback.

This audit takes 30-45 minutes and often reveals surprises. Most developers discover at least one instance of accidentally sharing sensitive data or accepting insecure AI-generated code.
