Spec-Driven Development

Memory Map

The Big Idea

Garbage in, garbage out — at scale. AI produces 500 lines of wrong code just as fast as 500 lines of correct code. The spec is the contract between your intent and AI's output. Tighter spec = more predictable result.

Human dominates first & last phases. AI dominates the middle. This is the 80/20 rule in practice.

📝 Anatomy of a Good Spec

  1. Description

    What we're building and why it exists. 1-2 sentences.

  2. Acceptance Criteria

    User-visible behaviors as checkboxes. The definition of "done."

  3. Edge Cases

    Empty/null input, concurrent access, scale (1000+ items), error states.

  4. Constraints

    Required patterns, things not to break, performance requirements, security needs.

  5. Non-Goals ⚠️

    What we're explicitly NOT building. Prevents AI scope creep at AI speed.

Plus Technical Notes: relevant files, patterns to follow, DB/API changes needed.

⚖️ How Much Spec Do You Need?

Task Type Spec Level Why
Production feature Full spec Users interact with it. Bugs have consequences.
API endpoint Full spec Contracts consumed by other systems. Changes break things.
Database migration Full spec Data changes are hard to reverse.
Refactoring Light spec Goal + scope + what NOT to change.
Bug fix Bug report Expected vs actual + regression test.
Prototype / POC None Exploring, not shipping. Vibe code it.

⚡ Implementation Best Practices

🪜
Step-by-Step > All-at-Once
Implement one plan step at a time. Review each before proceeding. Catch wrong data models before they propagate everywhere.
📎
Reference the Spec
"Implement Step 1 from docs/specs/feature.md" — always point AI to the spec file, not just a vague description.
🔗
Reference Existing Patterns
"Follow the pattern in CartRepository.kt" — existing code is the best example for AI to match your conventions.
📄
Specs Are Living Documents
When you discover the spec is wrong: update the spec first, then the code. Keeps spec as source of truth.

✅ Validation Checklist

Acceptance criteria
Walk through each criterion. Does the code satisfy it?
Edge cases
Are all listed edge cases handled? Test them explicitly.
Constraints
Does the code use required patterns and respect stated limits?
Non-goals
Did the AI add features from the non-goals list? Remove them.
Tests
Do tests cover acceptance criteria AND edge cases?
Architecture fit
Does the code follow existing project patterns?

Pro tip: ask AI to compare implementation against the spec and flag gaps.

🧪 Quick Quiz

Why do specs matter MORE with AI than without?

Speed amplifies ambiguity. Without AI, vague requirements produce slow mediocre code. With AI, they produce fast mediocre code — and a lot more of it. AI doesn't pause to ask "what about this edge case?" It fills gaps with assumptions.

What are the four phases of the SDD cycle and who owns each?

Specify (human), Plan (human + AI), Implement (AI, human reviews), Validate (human + CI). Human dominates first and last; AI dominates the middle.

Why are non-goals as important as goals?

Without explicit non-goals, AI will add "helpful" features you didn't ask for. Non-goals prevent scope creep at AI speed. "Sharing favorites" seems natural but if it's out of scope, it should be explicitly excluded.

When should you NOT write a full spec?

Prototypes/POCs (you're exploring, not shipping), one-line fixes (the prompt IS the spec), and bug fixes (a bug report with expected/actual behavior suffices). Scale spec investment to task risk.

📌 Key Takeaways

  • Garbage in, garbage out — at scale. Speed amplifies ambiguity. The spec is the highest-leverage investment in AI-driven development.
  • SDD cycle: Specify → Plan → Implement → Validate. Human owns the first and last phases.
  • Five spec sections: description, acceptance criteria, edge cases, constraints, and non-goals. Non-goals prevent AI scope creep.
  • Step-by-step beats all-at-once. Implement one plan step at a time. Review each before proceeding.
  • Agents excel within constraints. The spec doesn't limit AI — it enables it to do its best work.
  • Specs are living documents. When the spec is wrong, update the spec first, then the code.
