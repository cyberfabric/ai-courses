Chapter 10

Vibe Coding vs Structured Development

Memory Map
💡

The Big Idea

Vibe coding and structured development aren't opposing philosophies — they're two modes you switch between. Vibe-code to explore and learn, then formalize for production. Never ship the prototype.

⚖️ Two Modes of AI Development

Aspect 🎵 Vibe Coding 📐 Structured Development
Approach Conversational, exploratory Spec-driven, planned
Upfront work None — start talking Spec + acceptance criteria
Direction Emerges through iteration Defined before coding starts
Motto "Let's see what happens" "Let's build exactly this"
Speed Fast start, uncertain destination Slower start, predictable outcome
Tests None (or minimal) Written before implementation

🎯 When to Use Each

🎵
Vibe: Prototypes & POCs
Exploring feasibility. You don't know what you're building yet. The point is to learn, not to ship.
🎵
Vibe: Learning New Stacks
First time with KMP, new framework, unfamiliar library. Learn by building, steering the AI as questions arise.
🎵
Vibe: Hackathons & Side Projects
You're the only stakeholder, no production consequence. Speed and creativity over maintainability.
📐
Structured: Production Features
Users interact with it. Bugs have consequences. Spec catches edge cases before AI generates code at speed.
📐
Structured: Team Projects
Multiple devs = different assumptions. Spec is the coordination mechanism that prevents integration failures.
📐
Structured: Irreversible Changes
DB migrations, API contracts, regulated industries. Can't vibe-code it back. Spec provides the paper trail.

Never Ship the Prototype

The prototype is a learning tool full of shortcuts, hardcoded values, and accidental architecture. Rebuild for production. The rebuild is fast because the vibe session taught you what to build.

❓ The Four-Question Decision Framework

Will this code be maintained?
Yes → structured. Throwaway → vibe.
Are other people depending on it?
Yes → structured. Solo experiment → vibe.
Do I know the requirements?
Yes → structured. Exploring → vibe first, then formalize.
What's the cost of getting it wrong?
High (data loss, security, users) → structured. Low (can redo tomorrow) → vibe.

All four point to vibe? Vibe freely. Any one points to structured? Add structure proportionally.

🚫 The Two Danger Zones

💀
Vibe Coding in Production
800 lines deployed with no tests, columns named temp_fix, three patterns for the same thing. Tech debt at AI speed.
🐌
Over-Structuring Experiments
2 hours writing specs for a throwaway POC. Comprehensive tests for code deleted tomorrow. In a hackathon, still planning.

Mode-switch signal: If you catch yourself thinking "wait, what about..." more than twice while vibing → stop and write a spec. If your spec is half unanswerable questions → vibe-code a prototype first.

👥 Structure by Team Size

Team Size Recommended Approach
Solo Vibe freely. Structure only for production features you'll maintain.
2-3 devs Light specs for shared interfaces (APIs, data models). Vibe individual components.
4-8 devs Full specs for features. Shared conventions. Vibe only for prototypes.
8+ devs Formal SDD. Code review requirements. ADRs. Vibe sandboxed to POC branches only.

🧪 Quick Quiz

What's the "vibe coding sweet spot"?

High uncertainty about requirements, low consequence of mistakes, single developer, short timeframe. If all four conditions are true, vibe freely. If any breaks, add structure proportionally.

What's the hybrid approach?

Vibe-code the prototype to discover requirements, then write a spec from what you learned, then build the production version with SDD + TDD. Never ship the prototype — rebuild for production.

What's the "mode-switch signal"?

If vibing and you think "wait, what about..." more than twice → stop and write a spec (complexity crossed the threshold). If writing a spec and half the questions are unanswerable → vibe-code a prototype first to discover them.

Why does vibe coding break down in teams?

Two developers vibing the same feature make different assumptions and choose different patterns, producing code that doesn't integrate. Teams need structure proportional to their size.

📌 Key Takeaways

  • Two modes, not opposing philosophies. Vibe coding = explore and learn. Structured = build for production. Switch between them deliberately.
  • The hybrid approach: vibe the prototype, write spec from learnings, build properly. Never ship the prototype.
  • Four-question framework: Maintained? Others depending? Know requirements? High cost of failure? Any "yes" = add structure.
  • Two danger zones: vibe coding in production (tech debt at AI speed) and over-structuring experiments (planning paralysis).
  • Team size dictates structure: solo = vibe freely. 2-3 = light specs for interfaces. 4-8 = full specs. 8+ = formal SDD.
