Chapter 15

Privacy, Security, and Ethics

Memory Map
💡

The Big Idea

AI makes you faster, but every prompt leaves your machine. Sanitize before sharing, review for vulnerabilities, and own every line you commit. Responsible AI development isn't a constraint on productivity — it's what makes productivity sustainable.

🚫 Never Send to Any AI Tool

🔑
API Keys & Tokens
Even in config files or .env examples. Replace with REDACTED placeholders.
🔒
Passwords & Credentials
Database connection strings, SSH keys, certificates. Share the structure, not the values.
👤
PII / Customer Data
Real emails, phone numbers, health/financial records. Use synthetic data: user@example.com.
🧬
Proprietary Algorithms
Trade secrets, patented logic, competitive advantages. Abstract the pattern instead.
🏗️
Infrastructure Details
Server IPs, VPN configs, security architecture. Replace with generic URLs.
💳
Transaction Records
Real orders, payments, financial data. Share the schema/model only.
🛡️ CLAUDE.md protection: Add rules like "NEVER read .env, secrets.yaml, *.key, *.pem files." Prevents the agent from accidentally loading secrets into context.

🧹 Data Sanitization Quick Reference

Data Type Approach Example
Credentials Replace with placeholders password=REDACTED
PII Use synthetic data Jane Doe, user@example.com
DB Records Share schema only Model definition, not row data
Internal URLs Generic URLs https://api.example.com
Business Logic Abstract the pattern "Calculate discount by tier" (omit rates)

🔓 OWASP Top 10 in AI-Generated Code

Categories below use OWASP Top 10:2021 numbering. OWASP Top 10:2025 (Jan 2026) renumbered some entries; the underlying patterns are unchanged.

  1. Broken Access Control

    AI generates endpoints without auth. Fix: CLAUDE.md rule requiring auth middleware on all endpoints.

  2. Cryptographic Failures

    Uses MD5/SHA1 for passwords, HTTP instead of HTTPS. Fix: specify bcrypt + TLS always.

  3. Injection

    String interpolation in SQL, unsanitized shell commands. Fix: mandatory parameterized queries.

  4. Insecure Design

    Missing rate limiting, no input validation. Fix: include in specs — rate limits, abuse scenarios.

  5. Logging Failures

    Logs passwords, tokens, PII, card numbers. Fix: CLAUDE.md rule — NEVER log sensitive data.

⚡ Prevention + Review: Security rules in CLAUDE.md are preventive but not sufficient. Always pair with manual security review (Ch 11) and automated SAST/DAST scanning in CI.

⚖️ License Compliance

Verbatim Reproduction
If AI output looks suspiciously specific (exact names, unique comments), search online — it may be copied from a GPL project.
Code Ownership
Generally belongs to the person who prompted and directed the AI. Legal precedent is still evolving.
Commercial Use
Yes, with the same diligence as any code. Review, understand, verify no copyrighted material reproduced verbatim.
Copilot Filter
GitHub Copilot can block suggestions matching public code. Enable for commercial projects.
Document AI Usage
Note in the project that AI tools were used. Helps future compliance audits.

👀 You reviewed output

💾 You committed it

Transparency without bureaucracy: PR checkbox for AI assistance level, commit message co-author tags, and AI tools listed in CONTRIBUTING.md. Don't claim AI output as solely your own creative work.

🌍 Bias in Code Generation

👤
Default Assumptions
Binary gender fields, Western name formats (first/last), English-only UIs.
📊
Example Data Bias
Sample data skews toward certain demographics, regions, cultural contexts.
♿
Accessibility Gaps
Missing ARIA labels, screen reader support, keyboard navigation, color contrast.
🌐
Locale Defaults
US English date formats, currency, text direction. Spec RTL, i18n, and a11y upfront.

🧪 Quick Quiz

You need AI help debugging a database connection issue. What do you share?

Share the error message, code structure, and sanitized config (DATABASE_URL=postgresql://user:password@hostname:5432/dbname). NEVER share real credentials, server IPs, or production connection strings.

Who owns AI-generated code?

Generally the person who directed the AI, reviewed the output, and committed it. You are fully accountable for bugs, vulnerabilities, and license issues. "The AI wrote it" is never an excuse.

How do you prevent AI from generating insecure code?

Add security rules to CLAUDE.md (preventive), include security requirements in specs, and always run security review + SAST/DAST scanning. Prevention catches common issues; review catches the rest.

What should you do if AI-generated code looks suspiciously specific?

Search for it online. If it matches code from an open-source project, verify the license is compatible with your project. Treat it with the same diligence as copy-pasting from GitHub.

How do you address bias in AI-generated code?

Include inclusivity requirements in your specs: RTL language support, ARIA labels, culturally neutral sample data, non-binary-assumption data models. Requirements flow through to generated code when they're part of the spec.

Key Takeaways

  • Never send credentials, PII, customer data, or proprietary algorithms to any AI tool. Sanitize first.
  • Add CLAUDE.md rules to prevent the agent from reading secret files (.env, *.key, *.pem).
  • AI-generated code frequently contains OWASP vulnerabilities — security rules prevent; manual review catches the rest.
  • You own every line of AI-generated code you commit. Bugs, vulnerabilities, and license issues are yours.
  • Check for verbatim reproduction from open-source projects — especially for suspiciously specific output.
  • AI reproduces training data biases: gender assumptions, accessibility gaps, locale defaults. Spec inclusivity upfront.
  • Check corporate AI policies before using tools on company or client code. Get freelance agreements in writing.
  • Build a written team AI usage policy covering approved tools, data rules, disclosure, and security requirements.
