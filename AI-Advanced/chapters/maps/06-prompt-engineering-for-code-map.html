Chapter 6

Prompt Engineering for Code

Memory Map
💡

The Big Idea

Code prompting isn't general prompting — the margin for "close enough" is zero. Scale the specificity of your prompt to the criticality of the task. A quick utility? One-liner. Payment logic? Spell out every constraint.

🧬 Anatomy of a Good Coding Prompt

📋 Task
What action: implement,
refactor, fix, test, review
+
📂 Context
Relevant files, existing
patterns, problem background
+
🚧 Constraints
Must/must not, framework
choices, perf requirements
+
📎 Examples
"Follow the pattern in
OrderService.kt"

You don't need all four every time — but knowing the structure helps you identify what's missing when output is wrong.

🎯 The Four Prompt Scenarios

⚡
Generating Code
Describe what you need + constrain how it's built + reference what it should look like. Include file paths, layout details, pattern references.
🐛
Debugging
Expected vs actual behavior, exact error/stack trace, repro steps, what you've tried, relevant files. Never paraphrase errors — copy-paste them.
🔧
Refactoring
Goal (why), scope (which files), DO NOT change (boundaries!), target pattern, verification. Without exclusions, AI "helpfully" touches what you didn't ask.
🔍
Code Review
Specify what kind of review: security, performance, architecture, or newcomer-friendly. "Review this code" = generic. Targeted = actionable.

🐛 The Debugging Template

What I expected
Correct behavior described precisely.
What actually happens
Exact error message or stack trace. Copy-paste, never paraphrase.
Steps to reproduce
How to trigger the issue. Specific user actions or test cases.
What I've already tried
Approaches that didn't work and why. Saves the AI from repeating them.
Relevant code
File paths or specific functions. Let the AI read them directly.

🔧 The Refactoring Template

Goal
What the refactoring achieves — the why, not just the what.
Scope
Exactly which files, functions, or classes to change.
Do NOT change ⚠️
Public APIs, tests, behavior, adjacent code. Critical — prevents noisy diffs and unwanted "improvements."
Target pattern
What code should look like after. Reference an existing example file.
Verification
Existing tests should pass. Specify which test suite to run.

🔍 Code Review Variants

Review Type Focus On Ignore
🔒 Security Injection, access control, token handling, secrets Style issues
⚡ Performance N+1 queries, missing indexes, unbounded results Naming conventions
🏗️ Architecture Pattern consistency, layer boundaries, new dependencies Implementation details
🆕 Newcomer Business context, patterns used, what to know for future edits Bugs (separate pass)

AI handles first pass (style, bugs, security). You focus on architecture fit and business logic. Review throughput doubles.

🔄 The Iterative Refinement Loop

  1. Prompt

    Initial request with task + context + constraints + examples. Invest time here to reduce iterations.

  2. Review

    Read the generated code critically. Check correctness, patterns, edge cases, security.

  3. Refine

    Give specific, surgical feedback: "Change X to Y because Z." Don't re-describe the whole task — make targeted corrections.

  4. Accept

    Code meets your standards. Run tests, verify behavior, merge.

A precise first prompt + one refinement beats a vague prompt + five rounds of "no, I meant..."

🪄 Meta-Prompting

🤖

AI Writes Your Prompts

Ask AI to analyze your codebase and generate prompt templates customized to your actual file structure, naming conventions, and patterns. Review and refine once, reuse forever. Works for any task type: endpoints, migrations, PR reviews, feature modules.

🚫 Common Prompting Mistakes

🌫️
Too Vague
"Fix the bug" → Add which bug, where, with what symptoms and expected behavior.
📜
Too Verbose
500-word prompt for a one-line change. Wastes tokens and confuses focus.
❓
No Verification
"Add feature X" without "verify by running these tests." Always include how to confirm correctness.
🎯
No Pattern Reference
"Create a service" → "following the pattern in ExistingService." Without it, output is generic.
🔀
Ambiguous Scope
"Improve the code" → Specify: performance? readability? security? The AI can't read your mind.
🚧
Missing Constraints
"Add logging" without specifying where, what level, what format, and what NOT to log (PII!).

🧪 Quick Quiz

What are the four components of a good coding prompt?

Task (what action), Context (relevant files/patterns), Constraints (must/must not), Examples (pattern references). Not all four needed every time, but knowing the structure helps diagnose bad output.

Why is "Do NOT change" critical in refactoring prompts?

Without explicit exclusions, AI will "helpfully" rename variables, reorganize imports, update adjacent code, and refactor things you didn't ask about. This makes diffs noisy and code review harder.

What's the difference between a security review and a performance review prompt?

Security: focus on injection, access control, token handling, secrets (ignore style). Performance: focus on N+1 queries, missing indexes, unbounded results (ignore naming). Targeted reviews produce actionable findings; generic "review this" doesn't.

What is meta-prompting?

Using AI to generate prompt templates customized to your project. The AI analyzes your codebase, identifies patterns, and creates reusable templates referencing your actual file structure and conventions.

📌 Key Takeaways

  • Four components: Task + Context + Constraints + Examples. Scale specificity to task criticality.
  • Four scenarios, four templates: generation (describe + constrain + reference), debugging (expected/actual/error/tried), refactoring (goal/scope/DO NOT/pattern/verify), review (specify the type).
  • Iterate surgically: Prompt → Review → Refine → Accept. Give targeted corrections, don't re-describe the whole task.
  • "Do NOT change" prevents scope creep — always specify boundaries in refactoring prompts.
  • Build a prompt library — store templates in .claude/commands/ for team reuse. Use meta-prompting to bootstrap.
