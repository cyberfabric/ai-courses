The Big Idea

AI-driven development requires more thinking, not less. The thinking just moved upstream — from typing code to specifying intent, reviewing output, and making architectural decisions.

🚫 The Six Anti-Patterns

🙏
1. Prompt-and-Pray
Vague one-liner prompts hoping for the best. Fix: invest in precise descriptions with context and constraints.
😌
2. Over-Trusting
Accepting output because "it compiled." AI code can hide SQL injection, SELECT *, missing pagination. Always review.
👀
3. Under-Reviewing
Skimming instead of reading. You see structure but miss the hardcoded timeout, wrong column selection, missing index.
🤖
4. Full Delegation
Letting AI make all decisions including architecture. 17% comprehension drop. Builds code you can't debug or own.
🧲
5. Automation Bias
Assuming AI is right because it's AI. A confident-sounding model suggesting GlobalScope.launch is still wrong for Android.
🕳️
6. Context Neglect
Prompting without project context. You know your codebase intimately; AI starts from zero. Fix: CLAUDE.md, project rules.

🧠 Five Habits to Stay Sharp

  1. Ask "Why?"

    When AI generates code, ask it to explain choices. "Why StateFlow instead of SharedFlow?" Takes 30 seconds, builds understanding.

  2. Ask for Alternatives

    Don't accept the first approach. "What are two other ways? What are the trade-offs?" Exposes patterns you wouldn't have considered.

  3. Review Before Running

    Read code before running tests. Form your own hypothesis. If your prediction was wrong, you learned something.

  4. Implement Hard Parts Yourself

    Novel algorithms, tricky state machines, performance-critical paths — write these by hand. Use AI for scaffolding around them.

  5. Teach Back

    Explain AI-generated code in your own words. If you can't explain why it works, you don't own it yet. (Feynman technique.)

⏸️ Before You Prompt

What exactly am I building?
Not vaguely — specifically. Input, output, constraints, edge cases.
What context does the AI need?
Which files, patterns, conventions are relevant? Reference them explicitly.
What should it NOT do?
Which approaches to avoid? Which code not to touch?
How will I verify the result?
What tests pass? What behavior do I see? What edge cases matter?
What's the right scope?
Too much in one prompt? Break into steps.

2-5 minutes of planning saves 30 minutes of iteration.

⚡ The Speed Trap

⚠️

Speed Amplifies Everything

Before AI, bad architecture slowed you down naturally. Now you can build an entire module on a flawed foundation in minutes. Be fast at generating, slow at deciding. Invest more in planning, less in iteration — iteration is cheap, misdirection is expensive.

👥 The Shift by Role

📱
Android Dev
Stop thinking in XML nodes. Describe intent ("card with avatar left, name right"). Verify Modifier chains, recomposition, accessibility.
🖥️
Backend
Think in contracts, not implementations. Define API shape, error codes, auth. Let AI implement. Review queries and security.
☁️
DevOps
Think in constraints, not configurations. "Encrypted, same VPC, 30-day backups, 2 AZ replicas." Review for security and cost.
👔
Team Lead
Think in processes. Design team workflows where AI amplifies everyone. Set review standards. Mentor judgment, not syntax.
💼
Freelancer
Think in value, not hours. Bill for expertise that directs AI to the right outcome. Clients pay for results, not keystrokes.

Universal rule: Move your effort upstream. More time understanding & deciding, less time on mechanical translation.

🧪 Quick Quiz

What replaced the "typing" step in the new development loop?

Two new steps: describing precisely (specifying intent with context and constraints) and reviewing output (critical reading of AI-generated code).

What's the "Lazy Developer" fallacy?

The myth that AI means less work. Reality: AI requires more thinking — the effort shifts from typing to specifying, reviewing, and making architectural decisions.

Name the 6 anti-patterns of AI-driven development.

1) Prompt-and-Pray, 2) Over-Trusting, 3) Under-Reviewing, 4) Full Delegation, 5) Automation Bias, 6) Context Neglect.

What's the 70/20/10 rule?

70% AI generates fully (boilerplate, standard patterns). 20% you design, AI implements, you review closely. 10% you write entirely by hand (novel logic, critical paths).

How do you avoid skill atrophy while using AI?

Ask "why" about AI choices, request alternatives, review before running tests, implement hard parts yourself, and teach back (explain code in your own words).

📌 Key Takeaways

  • More thinking, not less — effort moves upstream to specifying, reviewing, and deciding. The "lazy developer" myth is dangerously wrong.
  • Think before you prompt — 2-5 minutes of planning (what, context, constraints, verification) saves 30 minutes of iteration.
  • Six anti-patterns to avoid: Prompt-and-Pray, Over-Trusting, Under-Reviewing, Full Delegation, Automation Bias, Context Neglect.
  • Your 20% drives quality — architecture, specs, review, and integration judgment are what separate great AI-assisted code from bad.
  • Stay sharp with the 70/20/10 rule — ask "why," review before running, implement hard parts yourself, and teach back.
  • Be fast at generating, slow at deciding — speed amplifies both good and bad decisions.
