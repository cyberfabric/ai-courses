Test-Driven Development with AI

Memory Map

The Big Idea

AI eliminates the tedium that made you abandon TDD. It writes the boilerplate, setup, and edge case permutations. You decide what to test and review whether it's right. Tests-first is your safety net against hallucinated behavior.

🛡️ Why Tests-First Matters MORE with AI

❌ Without tests:
trusting AI "got it right"
🤖 AI generates 10x
faster than you type
🧪 Tests =
Safety Net
✅ With tests:
automated verification
📝 Tests from spec =
executable contract

When AI writes code, tests verify it does what you specified — not what the AI assumed. Zero excuses left for not having tests.

⚠️ The Dangerous Anti-Pattern

Approach What Happens Result
✅ Tests from spec AI tests the intended behavior Tests catch real bugs, verify contract
❌ Tests from code AI tests the current implementation Tautological tests that verify mocks, not behavior

"Generate tests for this code" is the #1 AI testing mistake. Always generate tests from specs.

✅ Good Tests vs ❌ Bad Tests

✅
Behavior-Named
"updates theme to dark" not "calls setTheme." Test names read like a spec.
✅
Outcome Assertions
Check state changes and results, not that methods were called on mocks.
✅
Edge Cases Covered
Nulls, empty inputs, boundaries, concurrency, auth failures, network errors.
❌
Implementation Testing
"Verifies repository.save() is called" — tests your mock, not your behavior.
❌
Tautological Assertions
assert mock.called when you set up the mock to be called. Always passes, proves nothing.
❌
Copy-Paste Tests
Same test with trivially different inputs. Use parameterized tests instead.

🔍 Test Review Checklist

  1. Names Describe Behaviors?

    Read only test names — do they form a readable spec for the feature?

  2. Assertions Check Outcomes?

    Verify state changes, not that methods were called on mocks.

  3. Error Paths Tested?

    At minimum: invalid input, auth failure, network error, not found.

  4. Break the Implementation

    Intentionally break the code. Do tests actually fail? A test that never fails is worse than no test.

  5. Tests Independent?

    Shuffle the order — do they still pass? No shared mutable state.

🎯 Coverage Strategies

🔎
Gap Analysis
Ask AI to read tests + implementation, identify untested behaviors. Focus on error paths and boundary conditions.
🧬
Mutation Testing
Ask AI to flip conditions, remove null checks, change operators. Verify tests catch each mutation. Write tests for gaps.
🛡️
Behavioral Guardrails
Coverage = guardrails that prevent AI from accidentally changing behavior during refactors. Target meaningful coverage, not percentages.
📱
Visual Edge Cases
For UI: 200-char names, RTL text, empty states, loading states. AI catches cases humans skip.

📝 The Test Generation Prompt Pattern

Source
"Based on the spec in docs/specs/feature.md" — never "based on the code in src/".
Framework
Specify exactly: JUnit 5 + MockK + Turbine, or pytest + httpx async, etc.
Pattern reference
"Follow patterns in tests/unit/ExistingTest.kt" — ensures consistent test style.
Behaviors to test
List explicitly: happy path, error paths, edge cases, concurrency, auth scenarios.
Exclusions
"Do NOT test Room internals — mock the DAO." Prevents testing framework behavior.

📱 Platform-Specific Testing

Android: Compose UI Tests
Use compose-ui-test with semantics matchers. Never Thread.sleep — use waitUntil. Follow existing *ScreenTest.kt patterns.
Android: ViewModel Tests
Use Turbine for Flow assertions. Mock the repository. Test state transitions: Loading → Success → Error.
Backend: Integration Tests
Hit the real route handler with a test database. Cover all HTTP status codes. Integration tests > unit tests for APIs.
DevOps: Infrastructure Tests
Terratest (Go) for Terraform modules. Verify instance class, encryption, security groups, backup policies.

🧪 Quick Quiz

Why should you generate tests from specs, not from existing code?

When AI sees existing code, it tests the implementation (verifying method calls) rather than the behavior (verifying outcomes). Spec-generated tests catch real bugs; code-generated tests create false confidence with tautological assertions.

What's the AI-powered TDD workflow in order?

Write Spec (you, 15 min) → AI Generates Tests (2 min) → Review Tests (you, 10 min) → AI Implements to pass tests (5 min) → Verify all green (you + CI, 5 min).

How do you verify that AI-generated tests actually catch bugs?

Mutation testing: intentionally break the implementation (flip conditions, remove null checks, change operators). If tests don't fail, they're not testing what they claim. Write new tests for uncaught mutations.

What's the difference between testing outcomes vs testing calls?

Outcome: "verify the favorite was persisted with PENDING status" (checks real state). Call: "verify repository.save() was called" (checks mock interaction). Outcomes test behavior; calls test implementation details that break on refactor.

📌 Key Takeaways

  • AI eliminates the tedium that made you abandon TDD. 10x faster test generation, 3-5x more edge cases covered.
  • Always generate tests from specs, never from code. Code-derived tests verify mocks, not behavior.
  • Tests are your safety net against AI hallucinations. Without them, you're trusting the AI got it right.
  • Review AI tests with the same rigor as production code. Check: behavior names, outcome assertions, error paths, independence.
  • Use mutation testing to verify tests actually catch bugs. A test that never fails is worse than no test.
  • Coverage = behavioral guardrails that prevent AI from accidentally changing functionality during refactors.
