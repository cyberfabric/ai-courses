Code Review and Quality

Memory Map
💡

The Big Idea

AI-generated code is guilty until proven innocent. You own every line you commit — "the AI wrote it" is never an excuse. Review AI output with the same rigor as code from a talented junior developer whose work bears your name.

👃 The 5 AI Code Smells

🏗️
Over-Engineering
AI loves abstractions. You ask for a 3-line function, you get a factory pattern with DI, caching, and strategy classes.
👻
Hallucinated APIs
Methods, parameters, and imports that look plausible but don't exist. They compile in your head but crash at runtime.
🔀
Inconsistent Patterns
Callbacks here, coroutines there. Result wrapping in one function, raw exceptions in the next. No coherent style.
🕳️
Missing Error Handling
Happy path: perfect. Error paths: empty catch blocks, swallowed exceptions, or generic catch-alls that hide real issues.
📦
Stale Dependencies
Deprecated libraries, old API versions, or conflicting dependencies. Especially common in fast-moving ecosystems like Jetpack.
🔍 The 30-second sniff test: Before reading line by line, scan quickly — how many files, classes, and dependencies? If disproportionate to the task, it's over-engineered.

🔒 Security Vulnerabilities in AI Code

Vulnerability What AI Does What to Check
SQL Injection String interpolation in queries All queries use parameterized statements
Hardcoded Secrets API keys and passwords in source Secrets from env vars or secret managers
Missing Auth Endpoints without auth checks Every endpoint has auth middleware
XSS Renders user input unsanitized All user input escaped before rendering
Excessive Data Returns full objects with sensitive fields Response models exclude passwords, tokens
Insecure Defaults Disables SSL, uses HTTP All connections use secure defaults
🛡️ Prevention tip: Add security rules directly to CLAUDE.md — "NEVER use string interpolation in SQL. ALWAYS add authentication to endpoints." Prevents issues at generation time.

📝 The 10-Point Review Checklist

  1. Correctness

    Does the code actually do what was specified? Walk through acceptance criteria.

  2. Security

    No SQL injection, XSS, hardcoded secrets, missing auth, or excessive data exposure.

  3. Performance

    No N+1 queries, unbounded collections, missing pagination, or blocking operations.

  4. Architecture

    Respects layer boundaries, dependency direction, and module separation.

  5. Pattern Consistency

    Uses the same patterns, naming conventions, and approaches as existing code.

  6. Error Handling

    All error paths handled. No empty catch blocks or swallowed exceptions.

  7. Edge Cases

    Nulls, empty inputs, boundary values, concurrent access, failure modes handled.

  8. Tests

    Meaningful tests that test behavior, not implementation. Edge cases covered.

  9. Scope

    No unrequested features, unrelated refactors, or new patterns introduced.

  10. Dependencies

    No unnecessary new deps. Existing deps at correct versions. No hallucinated APIs.

If it's in your PR, it's your code. No post-mortem excuse of "the AI wrote it."

⚡ Performance Pitfalls to Catch

🔁
N+1 Queries
A loop making one DB query per iteration instead of a batch query.
📜
Missing Pagination
Loading entire tables into memory. Works with 10 rows, crashes with 10,000.
🧮
Redundant Computation
Recalculating values that could be cached or computed once.
🧊
Blocking Main Thread
Synchronous operations where async is needed. Especially in Android and Node.js.

🧪 Quick Quiz

What's the right mindset when reviewing AI-generated code?

Guilty until proven innocent. AI code is like code from a talented junior developer — often correct on the happy path but missing context. You bear the cost of any missed bugs.

What's the "30-second sniff test"?

Before reading line by line, scan for proportionality: how many files, classes, and dependencies were created? If disproportionate to the task, it's likely over-engineered.

Why should you assess code health before deploying AI changes?

AI amplifies existing code quality. In healthy code (9.5+ score), AI output is good. In unhealthy code, AI generates code consistent with the mess — introducing 2x more defects.

What are the three levels of AI disclosure in PRs?

1) Primarily AI-generated (full checklist review), 2) AI-assisted (human + AI collaboration), 3) Human-written (AI used only for review/suggestions). Each signals different review effort.

How can you use AI to catch its own architectural violations?

Ask the AI to review its output against your CLAUDE.md architecture rules. This self-review catches 60-70% of architecture issues before you even start reading the code.

Key Takeaways

  • You own every line of AI code you commit — review it like it bears your name.
  • Learn the 5 AI code smells: over-engineering, hallucinated APIs, inconsistent patterns, missing error handling, stale dependencies.
  • Apply the 10-point review checklist scaled to change risk — full checklist for auth/payments, lighter for formatting.
  • Security review is non-negotiable: check for injection, hardcoded secrets, missing auth, XSS, and excessive data exposure.
  • AI optimizes for correctness, not performance — watch for N+1 queries, missing pagination, and blocking operations.
  • Assess code health (target 9.5+) before letting AI modify a module — AI amplifies existing quality, good or bad.
  • Use AI to check AI: self-review against architecture rules catches most violations before human review.
  • Disclose AI usage level in PRs to calibrate review effort across the team.
