Chapter 12

Working with Large Codebases

Context strategies, refactoring at scale, migrations, and legacy code archaeology

The Context Problem at Scale

Everything we've covered so far works beautifully on small-to-medium projects. But enterprise codebases are different beasts: hundreds of modules, millions of lines, decades of history. Even with 1M token context windows, you can't fit the whole thing in. And if you could, the model would drown in noise.

The challenge isn't having too little context — it's having too much. The skill is knowing what to load, what to summarize, and what to leave out entirely.

1M
Max tokens in context — roughly 25,000 lines of code, or one medium module of a large codebase
10-100x
Typical enterprise codebase exceeds context window by this factor
80%
Of the codebase is irrelevant to any given task — the art is identifying the 20% that matters

Strategic Context Loading

For large codebases, you need a deliberate strategy for what the AI sees. Think in three tiers:

Tier 1: Always Loaded (CLAUDE.md)

Project-level instructions, architecture overview, build commands, conventions. This is your permanent context — the AI sees it on every invocation. Keep it concise: 500-1000 lines maximum for the root CLAUDE.md.

Tier 2: Task-Relevant Files

The files directly involved in the current task plus their immediate dependencies. Let the agent discover these through search rather than pre-loading everything.

Tier 3: Reference Only

Example files the AI should follow but doesn't need to modify. Point to them by path ("follow the pattern in src/services/OrderService.ts") rather than loading their contents.

# Good: Let the agent navigate strategically
$ claude "Add email verification to the signup flow.
  The relevant code is in:
  - src/auth/ (authentication module)
  - src/email/ (email service)
  Follow the pattern of password reset in src/auth/reset.ts
  Don't modify anything outside src/auth/ and src/email/"

# Bad: Trying to pre-load everything
$ claude "Here is our entire auth module [3000 lines],
  our email module [2000 lines], our user model [500 lines],
  our config [200 lines], our middleware [800 lines]...
  Now add email verification."
Trust the agent's navigation. Claude Code can search your codebase, read files on demand, and follow import chains. Don't try to manually load every relevant file into context. Give the agent the starting point and let it explore. It's often better at finding relevant code than you are because it can scan systematically.

Architecture Documentation as AI Enabler

In large codebases, architecture documentation isn't a nice-to-have — it's the map that makes AI effective. Without it, the AI navigates blind.

Essential Documents for Large Projects

Document Purpose for AI Where to Put It
Architecture overview Tells AI how modules relate, where boundaries are CLAUDE.md or docs/architecture.md
Module map Tells AI where to find code for specific domains CLAUDE.md section listing modules
ADRs Tells AI why decisions were made, preventing it from reverting them docs/adr/ directory
API contracts Tells AI the interfaces between services OpenAPI specs, protobuf definitions
Directory CLAUDE.md Module-specific context, conventions, and rules Each major module root

Team Lead Invest in Architecture Decision Records (ADRs). When AI encounters a pattern it wants to "improve," an ADR explains why the current approach was chosen: "We use polling instead of WebSockets for notification delivery because our load balancer doesn't support sticky sessions (ADR-047)." Without this context, the AI will helpfully refactor to WebSockets and break your infrastructure.

Directory-Level CLAUDE.md

For large projects, a single root CLAUDE.md isn't enough. Add CLAUDE.md files to major module directories with module-specific instructions:

# services/payments/CLAUDE.md
# Module: Payment Processing

## Critical Rules
- ALL changes require review by the payments team
- NEVER modify the payment gateway integration without updating
  the rollback procedure in docs/payments/rollback.md
- All monetary values use Decimal, NEVER float
- All API responses must include idempotency_key
- PCI compliance: no card numbers in logs or error messages

## Architecture
- Gateway adapter pattern: see adapters/ for Stripe and PayPal
- All transactions go through TransactionService (single entry point)
- Event sourcing for audit trail: see events/

## Testing
- Integration tests require PAYMENT_TEST_MODE=sandbox env var
- Mock gateway responses are in tests/fixtures/gateway/

Backend The payments module above has strict rules that a generic CLAUDE.md can't capture. Directory-level instructions ensure the AI respects module-specific constraints even in a massive codebase. The agent loads the relevant CLAUDE.md when it enters that directory.

Refactoring with AI at Scale

Large-scale refactoring is where AI shines brightest — and where it's most dangerous. The key is breaking massive changes into safe, incremental steps.

The Safe Refactoring Pattern

Understand
→
Plan
→
Verify Tests
→
Refactor Module
→
Run Tests
→
Commit
  1. Understand: Ask AI to map the code that will be affected. "Find all usages of UserService.getUser() across the codebase."
  2. Plan: Break the refactoring into independently deployable steps. Each step should leave the codebase in a working state.
  3. Verify tests: Ensure adequate test coverage before refactoring. If coverage is low, generate tests first (Chapter 8).
  4. Refactor one module: Apply the change to one module at a time. Commit each module separately.
  5. Run tests: Full suite after every module. Catch regressions immediately.
  6. Commit: One commit per module. If something breaks, you can revert one module without losing the rest.
# Example: Rename UserService.getUser() to UserService.findById()
$ claude "Plan a safe refactoring to rename UserService.getUser()
  to UserService.findById() across the entire codebase.

  Requirements:
  - Each module should be updated in a separate commit
  - Keep backward compatibility during the migration (deprecate old method)
  - Run tests after each module change
  - List all files that will be affected before making any changes"
Never refactor across the whole codebase in a single commit. If something breaks, you need to know which module's change caused it. One commit per module gives you precise revert granularity. This is slower but dramatically safer for large codebases.

AI-Assisted Migrations

Migrations are the most tedious work in software engineering: thousands of files, mechanical changes, occasional tricky edge cases. AI handles the mechanical part; you handle the edge cases.

Android Dev Java to Kotlin Migration

# Phase 1: Identify migration candidates
$ claude "List all Java files in the project, sorted by dependency count
  (fewest dependencies first). These are the safest to migrate first."

# Phase 2: Migrate one file at a time
$ claude "Convert src/data/models/User.java to idiomatic Kotlin.
  - Use data class where appropriate
  - Replace nullable types with Kotlin null safety
  - Use Kotlin property syntax instead of getters/setters
  - Ensure Java interop (add @JvmStatic, @JvmOverloads where needed)
  - Run ./gradlew testDebug after conversion"

# Phase 3: Gradually remove Java interop annotations
# as callers are also migrated to Kotlin

Full-Stack REST to GraphQL Migration

# Strangler fig approach: wrap existing REST in GraphQL
$ claude "Create a GraphQL schema that mirrors our existing REST API
  in routes/api/v2/. For each endpoint:
  1. Create a GraphQL type matching the response model
  2. Create a resolver that calls the existing service layer
  3. Add the resolver to the schema
  Don't modify the REST endpoints — they stay active during migration.
  Start with the read-only endpoints (GET) before mutations (POST/PUT/DELETE)."

DevOps For infrastructure migrations (on-prem to cloud, one cloud to another), AI generates the new IaC while you manage the cutover strategy. "Convert this CloudFormation template to Terraform, preserving all resource configurations, tags, and IAM policies. Output a checklist of manual steps needed for the actual migration."

Legacy Code: AI as Archaeologist

Every large codebase has dark corners: modules nobody understands, code written by developers who left years ago, undocumented business logic hidden in 2000-line functions. AI is an excellent archaeologist for these situations.

# Understanding undocumented code
$ claude "Read src/billing/invoice_calculator.py (800 lines) and:
  1. Explain what this module does in plain language
  2. Identify the main public functions and their purposes
  3. Map the data flow from input to output
  4. List any business rules embedded in the logic
  5. Flag any suspicious patterns (hardcoded values, commented-out
     code, TODO comments, catch-all exception handlers)
  6. Identify which parts are most fragile (complex conditionals,
     implicit dependencies, global state mutations)"

# AI reads the code and produces a documentation report.
# You verify the report against your domain knowledge.
# The report becomes the starting point for a proper CLAUDE.md
# or architecture document for this module.
AI-generated documentation needs verification. The AI can describe what the code does, but it can't tell you why certain business rules exist. After AI generates the documentation, review it with someone who understands the domain. The combination of AI's code-reading ability and human domain knowledge produces documentation that neither could create alone.

Monorepo Strategies

Monorepos amplify the context problem: one repository, many services, shared libraries, different tech stacks in different directories. The key is scoping the AI to the relevant slice.

Scoping AI to Modules

# Navigate to the specific module first
$ cd services/order-service
$ claude "Add rate limiting to the order creation endpoint.
  Only look at code within this module.
  Use the shared rate-limiter from libs/rate-limiter/."

# Or scope with explicit boundaries
$ claude "Working only within services/notification-service/:
  Add support for SMS notifications.
  Use the Twilio SDK already in our package.json.
  Follow the email notification pattern in notifiers/email.ts."

Team Lead For monorepos, establish ownership boundaries in the root CLAUDE.md:

# Root CLAUDE.md excerpt for monorepo
## Module Ownership
- services/auth/ — Auth team. Changes require auth-team review.
- services/payments/ — Payments team. PCI compliance rules apply.
- services/orders/ — Commerce team.
- libs/shared/ — Platform team. Changes affect all services.
- infra/ — DevOps team. All changes need terraform plan output.

## Cross-Module Rules
- Services communicate ONLY via the API gateway or message queue
- NEVER import directly from another service's internal modules
- Shared types go in libs/shared-types/
- Database schemas are service-private (no shared databases)

The Strangler Fig Pattern with AI

The strangler fig pattern — incrementally replacing legacy code with new code while both run in parallel — pairs perfectly with AI. AI handles the high-volume mechanical work of wrapping and replacing, while you manage the architectural cutover.

Wrap Legacy
→
Build New
→
Route Traffic
→
Verify Parity
→
Remove Legacy
  1. Wrap: AI creates adapter interfaces around legacy modules, exposing clean APIs
  2. Build: AI implements the new version behind the same interfaces
  3. Route: Feature flags toggle between legacy and new implementations
  4. Verify: Both implementations run in parallel; compare outputs for parity
  5. Remove: Once parity is confirmed, remove the legacy code and adapters

Freelancer The strangler fig is your friend for client projects with legacy codebases. You can modernize incrementally without a risky big-bang rewrite. AI generates the adapters and new implementations; you manage the migration plan and client communication. Each phase delivers visible progress while keeping the existing system running.

Exercise: Document an Undocumented Module

Find a poorly documented module in your codebase and use AI to create comprehensive documentation:

  1. Identify the module: Choose a module or class that lacks documentation and that people on your team find confusing or intimidating.
  2. AI archaeology: Ask Claude Code to read the module and generate:
    • A plain-language explanation of what it does
    • A list of public functions/classes with descriptions
    • A data flow diagram (described in text)
    • Business rules embedded in the logic
    • Suspicious patterns and potential risks
  3. Verify with domain experts: Share the AI-generated documentation with someone who knows the business context. What did the AI get right? What did it miss or misunderstand?
  4. Create a CLAUDE.md for the module directory, incorporating the verified documentation. Include conventions, constraints, and warnings for future AI sessions.
  5. Test the CLAUDE.md: Ask Claude Code to make a small change in the module. Does the new CLAUDE.md help it make better decisions?

This exercise produces real value: a previously opaque module becomes documented, and future AI interactions with it become more effective. It's one of the highest-ROI exercises in this book.
