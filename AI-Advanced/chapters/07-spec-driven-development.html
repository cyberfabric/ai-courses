Chapter 7

Spec-Driven Development

The Specify → Plan → Implement → Validate cycle that makes AI predictable

Why Specs Matter More With AI

Without AI, vague requirements produced slow, mediocre code. With AI, vague requirements produce fast, mediocre code — and a lot more of it. Speed amplifies ambiguity. A developer who types slowly has time to realize "wait, what about the edge case where..." midway through implementation. An AI agent that generates the entire feature in 90 seconds doesn't pause to ask clarifying questions. It fills gaps with assumptions.

This is why spec-driven development (SDD) becomes critical in the AI era. The spec is the contract between your intent and the AI's output. The tighter the spec, the more predictable the result.

The SDD Principle

Garbage in, garbage out — at scale. An AI can produce 500 lines of incorrect code just as quickly as 500 lines of correct code. The difference is entirely determined by the specification you provide. SDD is not bureaucracy; it's the highest-leverage investment you can make in AI-driven development.

The SDD Cycle

Spec-driven development follows a four-phase cycle. Each phase has a clear input, output, and owner:

Specify
→
Plan
→
Implement
→
Validate
Phase Owner Input Output
Specify Human Requirements, domain knowledge Specification document
Plan Human + AI Specification Implementation plan (files, order, approach)
Implement AI (human reviews) Plan + spec + codebase context Working code + tests
Validate Human + CI Code + spec Approved changes (or feedback loop)

The critical insight: the human dominates the first and last phases. You define what to build and you verify it was built correctly. AI dominates the middle. This is the 80/20 rule from Chapter 3 in practice.

The Specification Document

A good spec answers five questions: What are we building? How should it behave? What are the edge cases? What are the constraints? What are we NOT building?

Spec Template

## Feature: [Name]

### Description
[1-2 sentences: what this feature does and why it exists]

### Acceptance Criteria
- [ ] [User-visible behavior 1]
- [ ] [User-visible behavior 2]
- [ ] [User-visible behavior 3]

### Edge Cases
- [What happens when input is empty/null/malformed?]
- [What happens under concurrent access?]
- [What happens at scale (1000+ items)?]

### Constraints
- [Must use existing pattern X]
- [Must not break existing behavior Y]
- [Performance requirement: response under N ms]
- [Security requirement: authentication, authorization]

### Non-Goals (explicitly out of scope)
- [Feature we're NOT building in this iteration]
- [Optimization we're deferring]

### Technical Notes
- [Relevant existing code: file paths, patterns]
- [Database changes needed]
- [API contract changes]

Android Dev Real Spec Example

## Feature: Offline Favorites

### Description
Users can mark products as favorites while offline. Favorites
sync to the server when connectivity is restored.

### Acceptance Criteria
- [ ] Heart icon on ProductCard toggles favorite state
- [ ] Favorites persist locally (survive app restart)
- [ ] Favorites sync to server when network is available
- [ ] Conflict resolution: server wins on sync conflicts
- [ ] FavoritesScreen shows all favorites with offline indicator
- [ ] Optimistic UI: toggle is instant, sync happens in background

### Edge Cases
- Tap favorite while sync is in progress → queue the change
- 1000+ favorites → paginate the local query
- Server returns 401 during sync → re-authenticate, retry
- Airplane mode toggle rapid-fire → debounce sync trigger

### Constraints
- Use Room for local storage (existing database)
- Use WorkManager for background sync (existing pattern in CartSync)
- Follow MVVM with UiState pattern (like CartViewModel)
- Material 3 icon animations for the heart toggle

### Non-Goals
- Favorite categories or folders (future iteration)
- Sharing favorites with other users
- Server-side favorite recommendations

### Technical Notes
- New Room entity: FavoriteEntity(productId, addedAt, syncStatus)
- New DAO: FavoriteDao with Flow-based queries
- Modify ProductCard.kt to accept onFavoriteToggle callback
- New WorkManager worker: FavoriteSyncWorker
- API: POST/DELETE /api/v2/users/{id}/favorites/{productId}
Non-goals are as important as goals. Without explicit non-goals, AI will often add "helpful" features you didn't ask for. "Sharing favorites" seems like a natural addition — but if it's not in scope, it's not in scope. Non-goals prevent scope creep at AI speed.

The Planning Phase

With the spec in hand, the planning phase breaks it into implementable chunks. This is where AI becomes a powerful collaborator — it can analyze your codebase and suggest an implementation plan.

# Use Claude Code's plan mode for collaborative planning
$ claude "Read the spec in docs/specs/offline-favorites.md and
  create an implementation plan. Break it into ordered steps,
  identify which files need to be created or modified, and
  note any dependencies between steps.

  Consider our existing patterns — look at how CartSync works
  for the WorkManager pattern, and CartViewModel for the MVVM
  approach."

A good implementation plan looks like this:

## Implementation Plan: Offline Favorites

### Step 1: Data Layer
- Create FavoriteEntity.kt and FavoriteDao.kt
- Add migration to AppDatabase (version N+1)
- Create FavoriteRepository.kt with local CRUD + sync logic
Dependencies: None

### Step 2: Sync Worker
- Create FavoriteSyncWorker.kt (follow CartSyncWorker pattern)
- Register in WorkManagerInitializer
Dependencies: Step 1 (needs Repository)

### Step 3: ViewModel + UI State
- Create FavoritesViewModel.kt with UiState
- Create FavoritesUiState sealed interface
Dependencies: Step 1

### Step 4: UI Changes
- Modify ProductCard.kt (add heart icon + callback)
- Create FavoritesScreen.kt
- Add navigation route
Dependencies: Step 3

### Step 5: Tests
- Unit tests for FavoriteRepository
- Unit tests for FavoritesViewModel
- Unit test for FavoriteSyncWorker
Dependencies: Steps 1-4

Team Lead The planning phase is where you add the most value. AI produces a reasonable plan, but you catch: "Step 2 should come after Step 3 because we need the ViewModel's error handling defined before the worker can report sync failures." These dependency insights come from project experience the AI doesn't have.

Implementation with AI

With the spec and plan ready, implementation becomes the most predictable part. Feed the spec to your AI agent, reference the plan, and let it work step by step.

Without spec (unpredictable)
Add offline favorites to the app

The AI guesses at behavior, data model, sync strategy, UI placement. Every guess is a potential wrong turn.

With spec (predictable)
Implement Step 1 from the plan in docs/specs/offline-favorites.md: Create FavoriteEntity, FavoriteDao, database migration, and FavoriteRepository. Follow the patterns in data/local/CartEntity.kt and data/repository/CartRepository.kt.

Specific step, spec reference, pattern references. The AI produces code that matches your architecture.

Backend For backend features, feed the spec one layer at a time: data model first, then repository, then service, then route handler, then tests. Each step validates against the spec before moving to the next. This prevents the AI from building an entire vertical slice on a flawed data model.

Step-by-step beats all-at-once. Even though AI can generate the entire feature in one shot, implementing step by step produces better results. You catch issues early (wrong data model, bad sync strategy) before they propagate through the entire codebase. Review each step, then proceed.

Validation: Spec Compliance

Validation closes the loop. You check the implementation against the spec — not just "does it compile" but "does it match what we specified?"

Validation Checklist

  • Acceptance criteria: Walk through each criterion. Does the code satisfy it?
  • Edge cases: Are all listed edge cases handled? Test them explicitly.
  • Constraints: Does the code use the required patterns and respect the stated limits?
  • Non-goals: Did the AI add any features from the non-goals list? Remove them.
  • Tests: Do tests cover the acceptance criteria and edge cases?
  • Architecture fit: Does the code follow existing project patterns?

You can even use AI to help validate:

$ claude "Compare the implementation in feature/favorites/ against
  the spec in docs/specs/offline-favorites.md. For each acceptance
  criterion, confirm whether it's implemented and tested. Flag any
  gaps or non-goals that were accidentally included."

DevOps For infrastructure specs, validation means running terraform plan, kubectl --dry-run, or integration tests against a staging environment. The spec should define what "done" looks like for infrastructure just as precisely as for application code: "RDS instance is Multi-AZ, encrypted, and accessible only from the ECS security group."

When Specs Are Overkill

Not everything needs a formal spec. The investment should match the risk:

Task Type Spec Level Why
Production feature Full spec Users will interact with it. Bugs have consequences.
API endpoint Full spec Contracts are consumed by other systems. Changes are breaking.
Database migration Full spec Data changes are hard to reverse. Specify exactly.
Refactoring Light spec Goal + scope + what not to change. No acceptance criteria needed.
Bug fix Bug report Expected vs actual behavior + regression test requirement.
Prototype / POC None You're exploring, not shipping. Vibe code it (Chapter 10).
One-line fix None Just do it. The prompt IS the spec.

Freelancer For client work, a spec doubles as project documentation and a shared understanding of what "done" means. Even when the technical risk is low, the spec protects you from scope creep and "that's not what I meant" conversations. Write the spec, get client sign-off, then implement. AI makes the implementation fast; the spec makes the client relationship smooth.

Tools for Spec-Driven Development

SK
spec-kit (GitHub)
Open-source toolkit for spec-driven development. Provides markdown templates, validation tools, and integration with GitHub issues and PRs. Created by GitHub to formalize the SDD workflow. See spec-kit on GitHub.
PM
Claude Code Plan Mode
Built-in planning mode in Claude Code. Activate with /plan or Shift+Tab. AI reads your spec, explores the codebase, and produces an implementation plan without making any changes. Review and approve before execution begins.
MD
Markdown Spec Templates
Store reusable spec templates in your project's docs/templates/ directory. Create a custom Claude Code command (/new-spec) that generates a pre-filled spec from the template for each new feature.

Full-Stack Create a /new-spec custom command that auto-generates a spec from a feature name:

# .claude/commands/new-spec.md
Create a specification document for a new feature.

1. Ask me for the feature name and a one-line description
2. Read the template in docs/templates/feature-spec.md
3. Generate a filled-in spec based on my description
4. Analyze the codebase to pre-fill the Technical Notes section
   with relevant existing files and patterns
5. Save to docs/specs/[feature-name].md

Include realistic edge cases based on similar features in our codebase.

The Spec as Contract

McKinsey's QuantumBlack research on agentic workflows describes a two-layer approach that maps perfectly to SDD:

Orchestration Layer (Human)

  • Defines the specification
  • Sets constraints and guidelines
  • Reviews and approves output
  • Makes architectural decisions

Execution Layer (AI Agent)

  • Reads the specification
  • Plans the implementation
  • Generates code within constraints
  • Runs tests and iterates

The key insight from their research: agents excel within constraints with explicit guidelines. Remove the constraints and the agent wanders. Add clear constraints and the agent produces focused, high-quality work. The spec IS the constraint document. It's not limiting the AI — it's enabling it to do its best work.

Specs are living documents. When you discover during implementation that the spec was wrong or incomplete, update the spec first, then update the code. This keeps the spec as the source of truth. If you only update the code, the spec becomes stale documentation — and the next person (or AI) who reads it will get the wrong picture.

Exercise: The Full SDD Cycle

Practice the complete spec-driven development cycle on a real feature:

  1. Choose a feature you need to build (or pick one: "add search functionality to a list screen" or "add export-to-CSV for a data table").
  2. Write the specification using the template from this chapter. Include all five sections: description, acceptance criteria, edge cases, constraints, and non-goals. Spend 15-20 minutes on this.
  3. Generate the plan using Claude Code's plan mode: claude "/plan Read the spec in [path] and create an implementation plan." Review and refine the plan.
  4. Implement step by step — feed each step to Claude Code with the spec as reference. Review each step before proceeding to the next.
  5. Validate against the spec — walk through every acceptance criterion and edge case. Use the AI to check for gaps.
  6. Retrospective: What did the spec catch that you would have missed? Where was the spec too vague? What would you add next time?

The first time through the cycle feels slow. By the third time, it's faster than ad-hoc development because you spend zero time on rework and "wait, what about..." moments.
