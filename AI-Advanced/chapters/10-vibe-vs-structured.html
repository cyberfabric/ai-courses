Chapter 10

Vibe Coding vs Structured Development

When to freestyle and when to formalize — the hybrid approach for real projects

Two Modes of AI Development

Chapters 7-9 taught you the disciplined path: specs, tests, agentic workflows with guardrails. But if you've ever sat down with Claude Code and just started talking — "Let's build a todo app... actually make it a habit tracker... add some charts..." — you've experienced something very different. That's vibe coding.

These aren't opposing philosophies. They're two modes you switch between depending on context. Understanding when to use each — and when to transition from one to the other — is what separates experienced AI-driven developers from everyone else.

Vibe Coding

  • Conversational and exploratory
  • No upfront spec or plan
  • You describe intent, AI builds, you react
  • Direction emerges through iteration
  • Fast start, uncertain destination
  • "Let's see what happens"

Structured Development

  • Spec-driven and planned
  • Acceptance criteria defined upfront
  • You specify, AI implements, you validate
  • Direction defined before coding starts
  • Slower start, predictable outcome
  • "Let's build exactly this"

When Vibe Coding Wins

Vibe coding is the right choice when the cost of planning outweighs the cost of iterating. In these situations, the fastest path to a good outcome is to start building and discover the requirements along the way.

Prototypes and POCs

You're exploring whether an idea is feasible. You don't need a spec because you don't know what you're building yet. The point is to learn, not to ship.

$ claude "Let's prototype a real-time collaborative editor.
  Start with a basic textarea that syncs between two browser tabs
  using WebSockets. Use Node.js and plain HTML. Keep it simple —
  I want to see if the basic approach works before we invest in
  CRDT libraries."

# 5 minutes later: working prototype
# "Cool, now add cursor position sharing"
# "Actually, let's try operational transforms instead of last-write-wins"
# "What if we add offline support with a queue?"

# Each iteration builds on the previous one.
# The "spec" emerges from the conversation.

Learning a New Stack

Android Dev You're learning Kotlin Multiplatform for the first time. Vibe coding lets you explore without the overhead of specs: "Build a simple KMP app that shares a data layer between Android and iOS. Use Ktor for networking. Show me how the expect/actual pattern works for platform-specific code." You learn by building, steering the AI as questions arise.

Personal Projects and Hackathons

When you're the only stakeholder and there's no production consequence, formality is overhead. Vibe code your side project. Vibe code your hackathon entry. The goal is speed and creativity, not maintainability.

Exploratory Bug Investigation

Backend "Something weird is happening with our caching layer. Let me explore." You use the AI conversationally: "Read the cache module. What's the eviction policy? Is there a race condition between the TTL check and the refresh? Show me the code path when two requests hit an expired cache entry simultaneously." This is debugging as conversation, not as spec.

The vibe coding sweet spot: High uncertainty about requirements, low consequence of mistakes, single developer, and short timeframe. If all four conditions are true, vibe code freely. If any condition breaks, add structure proportionally.

When Structured Development Wins

Structured development wins when mistakes are expensive, requirements are knowable, and multiple people need to coordinate.

Production Features

Users will interact with this code. Bugs have consequences. A spec ensures you've thought through edge cases before the AI starts generating code at speed.

Team Projects

Team Lead When three developers are building different parts of the same feature, a spec is the coordination mechanism. Without it, developer A builds the API returning arrays while developer B's frontend expects objects. The spec prevents integration failures that waste days.

Regulated Industries

Healthcare, finance, government. "The AI vibed it into existence" is not an acceptable answer during an audit. Specs provide the paper trail that compliance requires: here's what we intended, here's what we built, here's how we verified it.

Database Migrations and API Contracts

Irreversible or externally-consumed changes need full specs. A database migration that drops the wrong column can't be vibe-coded back. An API that breaks mobile clients can't be casually iterated on in production.

The production vibe coding trap. It starts innocently: "Let me just quickly add this feature..." Three hours later, you have 800 lines of AI-generated code in production with no tests, no spec, and a data model that made sense at iteration 3 but not at iteration 15. Vibe coding in production is tech debt at AI speed.

The Hybrid Approach

The most effective workflow combines both modes. Vibe code the prototype, then formalize for production.

The prototype phase answers: Is this idea feasible? Does it feel right? What are the hard parts? The structured phase answers: How do we build it correctly for production?

Full-Stack You vibe-code a real-time notification system in 2 hours. It works on your machine. Now you write the spec: WebSocket connection management, reconnection strategy, message ordering guarantees, offline queue, push notification fallback, database schema for notification history, rate limiting. The prototype taught you what the spec needs to say. Then you build the production version with tests, following the spec.

The Critical Rule

Never ship the prototype. The prototype is a learning tool. It's full of shortcuts, hardcoded values, missing error handling, and architecture that evolved through conversation rather than design. Rebuild for production. The rebuild is fast because you already know what to build — the vibe coding session taught you.

Freelancer This hybrid approach is gold for client work. Vibe-code a prototype in the first meeting: "Here's roughly what it would look like and how it would work." The client sees something tangible immediately. Then take the prototype home, write the real spec, build it properly, and deliver production-quality code. You get the speed of vibe coding for the demo and the quality of structured development for the delivery.

The Danger Zones

Danger Zone 1: Vibe Coding in Production

Signs you've fallen into this trap:

  • There are no tests for a feature that's already deployed
  • The data model has columns named temp_fix or old_value_backup
  • Nobody can explain why a particular approach was chosen — "the AI suggested it"
  • Three different patterns are used for the same thing across the codebase
  • There's no spec and nobody agrees on what the feature should actually do

Danger Zone 2: Over-Structuring Experiments

Signs you've fallen into this trap:

  • You spent 2 hours writing a spec for a throwaway POC
  • You wrote comprehensive tests for a prototype that might be deleted tomorrow
  • You're in a hackathon and you haven't written any code yet because you're still planning
  • A quick investigation turned into a formal project with tickets and milestones

Both traps waste time and energy. The skill is recognizing which mode the current task demands and switching cleanly between them.

The mode-switch signal: If you're vibe coding and you catch yourself thinking "wait, I need to handle the case where..." more than twice, that's your signal to stop and write a spec. The complexity has crossed the threshold where structure pays for itself. Conversely, if you're writing a spec and realize you don't know the answers to half the questions, vibe-code a prototype first to discover them.

Team Considerations

Team Lead Vibe coding works for solo developers. It breaks down in teams. When two developers vibe-code the same feature, they make different assumptions, choose different patterns, and produce code that doesn't integrate. Teams need structure proportional to their size.

Team Size Recommended Approach
Solo Vibe code freely. Add structure only for production features you'll maintain.
2-3 developers Light specs for shared interfaces (API contracts, data models). Vibe code individual components.
4-8 developers Full specs for features. Shared CLAUDE.md and conventions. Vibe coding only for prototypes and personal exploration.
8+ developers Formal SDD process. Code review requirements. Architectural decision records. Vibe coding sandboxed to POC branches only.

The Morning/Evening Split

Some teams adopt a temporal split between the two modes:

Morning: Creative Mode

  • Vibe-code new features and prototypes
  • Explore solutions conversationally
  • Sketch out approaches
  • Human energy high → creative decisions

Evening/Overnight: Execution Mode

  • Agent-driven structured implementation
  • Test generation from morning's specs
  • Documentation updates
  • Human energy low → delegate to agents

DevOps This split works especially well for infrastructure work. Morning: vibe-code a prototype Terraform module, test it in a sandbox account, explore different configurations. Evening: write the spec from what you learned, let the agent generate the production module with all the tags, encryption, monitoring, and compliance requirements that make infrastructure code tedious but critical.

Case Study: Same Feature, Both Ways

Let's compare building the same feature — a search-with-filters system — using each approach.

Vibe Coded Version

# Session starts at 2:00 PM
$ claude "Add search to the products page"
# AI adds a basic text search box and filter function

$ claude "Add filters for category, price range, and rating"
# AI adds filter dropdowns and updates the query

$ claude "Actually, make the price filter a range slider"
# AI swaps the dropdown for a slider component

$ claude "The results should update as you type, with debouncing"
# AI adds debounced search with 300ms delay

$ claude "Add a 'no results' state with suggestions"
# AI adds empty state UI

# 2:45 PM — working search with filters
# Total: 45 minutes, no spec, no tests, works on happy path
# Issues: no loading states, no error handling, no URL params,
# no accessibility, search doesn't survive page refresh

Structured Version

# Day 1: Write spec (20 min)
# - Acceptance criteria: debounced search, filter persistence in URL,
#   loading states, error states, empty states, keyboard navigation
# - Edge cases: special characters, very long queries, rapid filter changes
# - Performance: results under 200ms, debounce at 300ms
# - Accessibility: ARIA labels, keyboard navigation, screen reader support

# Day 1: Generate tests from spec (5 min + 10 min review)
# Day 1: Implement with agent following spec (15 min)
# Day 1: Validate against spec (10 min)

# Total: ~60 minutes, spec exists, tests exist, all edge cases handled
# Issues: none — the spec caught them before implementation

Vibe Coded Result

  • 45 minutes to working demo
  • 0 tests
  • Happy path works
  • 5+ known gaps
  • Good for: demo, prototype, exploration

Structured Result

  • 60 minutes to production-ready
  • Full test coverage
  • All edge cases handled
  • Spec documents the behavior
  • Good for: production, team, long-term

The vibe-coded version is 25% faster to a working demo. The structured version is production-ready from day one. The right choice depends entirely on your goal.

The Decision Framework

Before starting any task, ask yourself these four questions:

  • Will this code be maintained? If yes → structured. If it's throwaway → vibe.
  • Are other people depending on it? If yes → structured. Solo experiment → vibe.
  • Do I know the requirements? If yes → structured. Exploring → vibe first, then formalize.
  • What's the cost of getting it wrong? High (data loss, security, users affected) → structured. Low (can redo tomorrow) → vibe.

If all four answers point to vibe: vibe freely. If any answer points to structured: add structure proportionally. If you're unsure, default to the hybrid approach: vibe the prototype, formalize for production.

Exercise: Build It Twice

Build the same small feature using both approaches and compare the results:

  1. Choose a feature that's small enough to build twice in one session. Suggestions: a search/filter UI, a settings page, a data export function, a notification system.
  2. Round 1 — Vibe code it. No spec, no plan. Open Claude Code and just start talking. Build iteratively for 30-45 minutes. Note what you build, what you miss, and how the design evolves.
  3. Round 2 — Build it structured. Write a spec first (15 min). Generate tests from the spec. Implement with the agent following the spec. Validate against acceptance criteria.
  4. Compare:
    • Time to working code
    • Number of edge cases handled
    • Test coverage
    • Code quality and consistency
    • How confident are you shipping each version?
  5. Reflect: In what scenarios would each approach be the right choice for your team?

This exercise builds the intuition for mode-switching that you'll use daily. Most developers develop a strong preference for one mode — the exercise forces you to practice the other.
