Chapter 9

Agentic Coding Workflows

Multi-step autonomous AI, custom skills, parallel agents, and the trust spectrum

What "Agentic" Means

In previous chapters, you directed the AI step by step: write this test, generate that function, review this code. Agentic coding is different. You give the AI a goal, and it figures out the steps on its own — reading files, making decisions, writing code, running tests, fixing failures, and iterating until the task is done.

An agentic workflow is an AI that operates in a loop:

Observe
→
Plan
→
Act
→
Verify
→
Iterate

The agent observes the codebase (reads files, understands structure). It plans an approach (which files to change, in what order). It acts (writes code, runs commands). It verifies (runs tests, checks output). If something fails, it iterates — diagnosing the error and trying again. This loop continues until the task succeeds or the agent hits a problem it can't solve.

The Key Difference

Chat-based AI responds to your messages. Agentic AI takes initiative. It doesn't wait for you to tell it to run the tests — it runs them, sees the failure, reads the stack trace, fixes the bug, and runs them again. Your role shifts from "driver" to "supervisor."

Claude Code as an Agent

Claude Code is a full-featured coding agent. Here's what it can do autonomously in a single session:

  • Read files — Navigate and understand your entire codebase
  • Search code — Find patterns, usages, definitions across the project
  • Edit files — Create new files and modify existing ones with precise diffs
  • Run commands — Execute build tools, test suites, linters, and scripts
  • Interpret output — Read error messages, test results, and command output
  • Iterate on failures — Fix bugs caught by tests or build errors, then retry
  • Manage git — Stage changes, write commit messages, create branches

This is the tool chain that achieves the 79% task automation rate from Anthropic's research. The agent doesn't just generate code — it integrates it into your project, validates it, and delivers working results.

Backend Example: End-to-End Feature

$ claude "Add a product reviews feature to the API:
  - POST /api/v2/products/{id}/reviews (create review)
  - GET /api/v2/products/{id}/reviews (list reviews, paginated)
  - DELETE /api/v2/reviews/{id} (delete own review)
  - Include star rating (1-5), title, body, author
  - Add database migration for the reviews table
  - Write integration tests for all endpoints
  - Follow existing patterns in the orders module
  - Run the test suite when done"

# Claude Code will:
# 1. Read the orders module to understand patterns
# 2. Create the Review model and migration
# 3. Create the repository, service, and route handlers
# 4. Write integration tests
# 5. Run pytest
# 6. Fix any failures
# 7. Report the results

You walk away, come back in a few minutes, and review the complete implementation. That's agentic coding.

Multi-Step Tasks

The power of agentic workflows shows most in multi-step tasks that span multiple files and require coordination:

Android Dev "Add offline caching to the product catalog." The agent reads your network layer, understands your Room setup, creates a CachePolicy, modifies the repository to check cache first, adds a WorkManager task for periodic refresh, writes tests, and updates the ViewModel to show cached data while loading fresh data. This involves 8-10 files across 4 layers — exactly the kind of task where manual implementation is tedious but AI orchestration shines.

DevOps "Set up GitHub Actions CI for the new service." The agent reads your project structure, identifies the build system, creates a workflow YAML with build/test/lint stages, adds caching for dependencies, sets up parallel test runners, creates the Dockerfile if missing, and validates the workflow syntax. A task that requires knowing multiple tools and file formats, done coherently in a single pass.

Full-Stack "Add user avatar upload: React component, API endpoint, S3 storage, and database migration." The agent coordinates across the full stack: creates the upload component with drag-and-drop, adds the multer middleware, configures S3 upload in the service layer, adds the avatar_url column migration, and writes tests at every layer.

Custom Skills and Commands

Custom commands turn your team's recurring workflows into one-line invocations. They're markdown files in .claude/commands/ that encode multi-step processes the agent follows.

Team Lead Standardized Team Workflows

# .claude/commands/new-feature.md
# Invoke with: /new-feature

Implement a new feature following our team workflow:

1. Read the spec in docs/specs/$ARGUMENTS.md
2. Create an implementation plan (show me before proceeding)
3. Implement step by step:
   a. Data layer first (models, migrations, repositories)
   b. Service layer (business logic)
   c. API layer (routes, request/response models)
   d. Tests at every layer
4. Run the full test suite: `make test`
5. Run the linter: `make lint`
6. Fix any failures
7. Create a git branch: feature/$ARGUMENTS
8. Commit with a conventional commit message
9. Summarize what was built and any decisions made
# .claude/commands/fix-ci.md
# Invoke with: /fix-ci

The CI pipeline is failing. Fix it:

1. Read the latest CI output from .github/workflows/
2. Identify the failing step and error message
3. Find the root cause in the codebase
4. Fix the issue
5. Run the same checks locally to verify:
   - `make test`
   - `make lint`
   - `make build`
6. Commit the fix with message: "fix: [description of CI fix]"

With custom commands, a new team member can run /new-feature user-preferences and get a correctly structured, convention-following feature implementation without knowing all the team's patterns by heart.

The $ARGUMENTS variable in custom commands captures everything after the command name. /new-feature user-preferences sets $ARGUMENTS to "user-preferences". Use this to make commands reusable across different features, specs, or modules.

The Trust Spectrum

How much autonomy should you give the agent? It depends on the task's risk and reversibility:

Trust Level Agent Autonomy Best For
Full supervision Agent proposes, you approve each step Database migrations, security changes, production configs
Milestone review Agent runs freely, pauses at checkpoints New features, large refactors, API changes
Test-gated Agent runs freely; tests are the guardrail Bug fixes, test generation, documentation
Full autonomy Agent runs to completion, you review the diff Formatting, linting fixes, boilerplate generation

Start with more supervision and loosen over time as you build confidence in the agent's output for your specific codebase. A well-configured CLAUDE.md and strong test suite let you trust the agent more because the guardrails are built in.

Never give full autonomy to destructive operations. Database migrations, file deletions, production deployments, and git force-pushes should always require human approval. Configure Claude Code's permissions to deny these by default and require explicit confirmation.

Error Recovery

Agents make mistakes. The question isn't whether the agent will hit errors — it's how well it recovers. Good agentic workflows include built-in error handling:

What Agents Handle Well

  • Build failures — Missing imports, type mismatches, syntax errors. The agent reads the error, fixes it, rebuilds.
  • Test failures — Assertion mismatches, missing setup. The agent reads the failure, understands what went wrong, fixes the code or test.
  • Linter errors — Style violations, unused imports. Easy to auto-fix.

What Agents Handle Poorly

  • Architectural misalignment — The agent chose the wrong pattern and keeps building on it. More iterations make it worse, not better.
  • Silent logic bugs — Code compiles, tests pass (because the tests are also wrong), but the behavior doesn't match the spec.
  • Scope drift — The agent starts "improving" things outside the task scope. It fixes the requested feature and also refactors three unrelated files.

Freelancer When the agent goes off track, the best recovery is often to start fresh with a more precise prompt rather than trying to steer it back. Sunk cost fallacy applies to agent sessions too. If the agent built 200 lines on a wrong foundation, discarding and re-prompting with better constraints is faster than fixing 200 lines of misdirected code.

The 3-attempt rule: If the agent fails to fix an error after 3 attempts, intervene. Either the error requires context the agent doesn't have, or the approach is fundamentally wrong. Step in, diagnose, and either provide the missing context or redirect the approach.

Long-Running Agents

For complex tasks that take many steps, you need strategies to keep the agent effective over extended sessions.

Context Compaction

Long sessions accumulate context: files read, errors encountered, approaches tried. This eventually fills the context window with stale information. Use /compact in Claude Code to compress old context and free up space for what matters now.

Checkpointing

For multi-hour tasks, commit working intermediate states. If the agent goes off track in step 7, you can revert to the checkpoint at step 6 instead of starting over:

# In your custom command, add checkpoints:
# After each major step:
# - Run tests
# - If passing, commit: "checkpoint: completed [step name]"
# - If failing, stop and report

Sub-Agents and Agent Teams

Claude Code can spawn sub-agents for parallel work. For example, a main agent orchestrates the feature implementation while sub-agents handle independent tasks:

# The main agent can delegate:
# - Sub-agent 1: Generate database migration and repository
# - Sub-agent 2: Generate API route handlers
# - Sub-agent 3: Generate test suite
# Each works in isolation, main agent integrates the results

This mirrors how a tech lead delegates to team members. The orchestrating agent maintains the big picture while sub-agents handle focused tasks.

As of Opus 4.6 (February 2026), this pattern is also available as a first-class product called Agent Teams: one Claude session acts as the team lead, spawns multiple teammates (each with its own context window), assigns work via a shared task list, and synthesizes results — with the option to require teammates to plan in read-only plan mode until the lead approves. It's the same Writer/Reviewer or fan-out-by-file workflow you'd build by hand, but with messaging, locking, and oversight baked in. Be aware: token usage scales linearly with active teammates, so reach for it on tasks that genuinely benefit from parallel context. See Effective Harnesses for Long-Running Agents for advanced patterns and the Agent Teams docs for setup details.

Multi-Layer Safeguards

Trusting an agent doesn't mean removing all guardrails. The best agentic workflows have multiple safety layers, as recommended by CodeScene's research:

Generation Checks
→
Test Suite
→
Pre-Commit Hooks
→
PR Review
→
CI Pipeline
  • Generation checks: CLAUDE.md rules that prevent known bad patterns ("NEVER use GlobalScope")
  • Test suite: The agent runs tests after every change. Failures are caught immediately.
  • Pre-commit hooks: Linters, formatters, secret scanners run before any commit.
  • PR review: Human reviews the complete diff. AI can assist but doesn't replace this.
  • CI pipeline: Full build, test, and security scan in a clean environment.

Each layer catches different failure modes. Generation checks catch convention violations. Tests catch behavioral bugs. Pre-commit hooks catch secrets and formatting. PR review catches architectural issues. CI catches environment-specific problems. No single layer is sufficient; together, they make agentic coding safe for production.

Auto mode and /sandbox: built-in guardrails for autonomous runs. When you run Claude Code with --permission-mode auto, a separate classifier model gates approvals: routine work proceeds, but scope escalation, unknown infrastructure, or hostile-content-driven actions get blocked. For non-interactive runs (claude -p) used in CI or fan-out scripts, auto mode aborts if the classifier repeatedly blocks — a safe failure mode rather than a stall. Pair it with /sandbox for OS-level filesystem and network isolation when the task warrants stronger boundaries (untrusted dependencies, exploratory builds, agentic file rewrites). These two are the cheapest "free" layers in the safeguard stack — turn them on before you start trusting longer autonomous runs.

A Real Agentic Workflow

Backend Here's a real end-to-end agentic session for adding a feature:

# 1. You write the spec (15 min)
# docs/specs/product-reviews.md

# 2. You invoke the agent (1 min)
$ claude "/new-feature product-reviews"

# 3. Agent works autonomously (5-10 min):
#    - Reads the spec
#    - Reads existing code patterns
#    - Shows you the implementation plan (you approve)
#    - Creates ReviewModel + migration
#    - Creates ReviewRepository
#    - Creates ReviewService with business logic
#    - Creates route handlers
#    - Generates integration tests
#    - Runs pytest — 2 failures
#    - Reads the errors, fixes both issues
#    - Runs pytest — all passing
#    - Runs linter — clean
#    - Creates branch: feature/product-reviews
#    - Commits with conventional message

# 4. You review the diff (10 min)
$ git diff main...feature/product-reviews

# 5. You push and create PR (1 min)
$ git push -u origin feature/product-reviews
$ gh pr create

Total human time: ~27 minutes. Feature is complete with tests, following conventions, on a clean branch ready for PR review. The agent handled roughly 80% of the work; you handled the 20% that required judgment (spec, plan approval, diff review).

Exercise: Create a Custom Skill

Build a custom Claude Code command for a task your team does repeatedly:

  1. Identify a recurring task in your project. Examples:
    • Adding a new API endpoint with all layers
    • Creating a new screen/page with ViewModel and tests
    • Setting up a new microservice with boilerplate
    • Running a release process (version bump, changelog, tag)
  2. Write the command in .claude/commands/[task-name].md. Include:
    • Step-by-step instructions the agent should follow
    • References to existing patterns ("follow the pattern in...")
    • Verification steps ("run tests," "run linter")
    • Use $ARGUMENTS for the variable parts
  3. Test the command on a real task. Invoke it with /task-name argument.
  4. Refine based on results. Did the agent miss a step? Add it. Did it make incorrect assumptions? Add constraints. Did it touch files it shouldn't have? Add exclusions.
  5. Commit the command so your team can use it.

A good custom command pays for the 20 minutes of writing it within its first three uses. The best ones become team standards that new members use on day one.
