Chapter 17

Claude Skills, Subagents, Plugins & MCP — Extending Claude Code

The extension stack: file-based primitives that turn Claude Code from a coding assistant into a project-shaped collaborator

The Extension Stack at a Glance

Chapter 9 introduced "custom skills/commands" as one tool in the agentic toolbox. In the year since, that single bullet has grown into a coherent extension stack. Claude Code now ships five named, file-based primitives plus two runtime modes — and authoring each is a real skill of its own. If you only know about CLAUDE.md and slash commands, you're using maybe a third of the tool.

Here's the stack, in order of "weight" (from advisory hint to deterministic guardrail):

Five Primitives

  • CLAUDE.md — ambient instructions loaded every session
  • Skills — on-demand domain knowledge or repeatable workflows
  • Subagents — isolated context with a scoped tool whitelist
  • MCP servers — external integrations (Notion, Linear, Figma, databases)
  • Hooks — deterministic scripts at lifecycle points

Two Runtime Modes

  • Auto Mode — classifier-gated permissions instead of click-through approvals
  • Sandboxing — OS-level filesystem and network isolation

Plus Plugins (installable bundles of all of the above) and Agent Teams (multi-session orchestration).

Choose by Question, Not by Vibe

The official best-practices doc frames the choice as a single question: what is this thing supposed to guarantee? If it's persistent context, it's CLAUDE.md. If it's knowledge or a workflow you want available on demand, it's a Skill. If it's a focused task that should run in its own context window, it's a Subagent. If it's access to an external system, it's an MCP server. If it's an action that must happen every time, with zero exceptions, it's a Hook. Match the primitive to the guarantee and you'll rarely pick wrong.

Skills: File-Based Domain Knowledge

A Skill is a folder under .claude/skills/<name>/ containing a SKILL.md with frontmatter. Claude loads the description into context at session start, then loads the body on demand when the description matches what you're doing — or you can invoke it manually with /<skill-name>. Files in .claude/commands/ still work for backwards compatibility, but the unified, recommended path is .claude/skills/.

There are two flavors. Ambient skills describe how things work; Claude pulls them in when relevant. Side-effect skills describe a workflow you trigger explicitly; you set disable-model-invocation: true to keep Claude from running them on its own.

Backend Ambient Skill

.claude/skills/api-conventions/SKILL.md
---
name: api-conventions
description: REST API design conventions for our services
---
# API Conventions
- Use kebab-case for URL paths
- Use camelCase for JSON properties
- Always include pagination for list endpoints (cursor-based, default limit 50)
- Version APIs in the URL path (/v1/, /v2/)
- Errors follow RFC 7807 problem+json
- All timestamps are ISO 8601 in UTC

This file lives next to the code. Anyone touching the API repo gets the same conventions. The CLAUDE.md doesn't need to know about them — the description in the frontmatter is enough for Claude to find the skill when the task is "add a new endpoint."

Full-Stack Side-Effect Skill

.claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: Fix a GitHub issue end-to-end
disable-model-invocation: true
---
Analyze and fix the GitHub issue: $ARGUMENTS.

1. Use `gh issue view` to get the issue details
2. Understand the problem described in the issue
3. Search the codebase for relevant files
4. Implement the necessary changes to fix the issue
5. Write and run tests to verify the fix
6. Ensure code passes linting and type checking
7. Create a descriptive commit message
8. Push and create a PR linking the issue

You run it as /fix-issue 1234. The $ARGUMENTS token is substituted with whatever you typed. The disable-model-invocation flag is what makes this safe — without it, Claude might decide to "fix issue 1234" on its own when you only meant to ask about it.

The CLAUDE.md vs Skill test. Read the rule out loud. Does it apply to every conversation in this repo? Then it goes in CLAUDE.md. Does it apply only when a specific kind of task comes up? Then it's a Skill. Bloated CLAUDE.md files cause Claude to ignore your actual instructions — the doc says this in nearly the same words. Move conditional knowledge into Skills and let CLAUDE.md stay short.

Subagents: Isolated Context, Scoped Tools

A Subagent is a single Markdown file under .claude/agents/<name>.md. When Claude delegates to it, the subagent runs in its own context window with its own tool whitelist and reports back a summary. The main conversation never sees the dozens of files the subagent read — only the conclusion.

This is the single most powerful tool for managing context. The best-practices doc calls subagents "one of the most powerful tools available" precisely because context is the fundamental constraint, and subagents let you spend other people's context on research while keeping yours clean for implementation.

DevOps Security Reviewer

.claude/agents/security-reviewer.md
---
name: security-reviewer
description: Reviews code for security vulnerabilities
tools: Read, Grep, Glob, Bash
model: opus
---
You are a senior security engineer. Review code for:
- Injection vulnerabilities (SQL, XSS, command injection)
- Authentication and authorization flaws
- Secrets or credentials in code
- Insecure data handling and missing input validation
- Insecure cryptographic primitives or key handling

Provide specific line references and suggested fixes.

Three things to notice. First, tools: Read, Grep, Glob, Bash — an explicit whitelist. The reviewer can read the codebase but can't write to it. Second, model: opus — this subagent will use Opus 4.7 even if the main session is on Sonnet, because security review is the kind of work that justifies the strongest model. Third, the prompt is concrete. Vague subagent prompts produce vague reviews.

Trigger it with: "Use a subagent to review this code for security issues." You'll get back five lines of summary instead of two thousand lines of file dumps. The subagents docs list every frontmatter field, including disallowedTools as a denylist alternative when you want "everything except X."

Subagents inherit MCP servers by default. If your main session has a database MCP server connected, your subagent has it too unless you scope tools explicitly. For anything that touches production, use tools: as an allowlist. "Trust by default" is the wrong default for delegation.

The Plugin Marketplace

A Plugin is an installable bundle of skills, hooks, subagents, and MCP servers, distributed through marketplaces. /plugin opens the browser. The official Anthropic-managed marketplace (claude-plugins-official) is auto-available the moment you start Claude Code — no setup required.

Plugins matter for two reasons. They let teams package and distribute their conventions as a single install, and they let the community share workflows that would otherwise live in a thousand README gists. The discover-plugins doc highlights one category in particular: code-intelligence plugins for typed languages. If you work in TypeScript, Kotlin, Rust, or Go, install the relevant code-intelligence plugin so Claude has precise symbol navigation and automatic error detection after edits, instead of running grep over the codebase.

Enterprise: Private Plugin Marketplaces

In February 2026, Anthropic added private plugin marketplaces as part of the Claude Cowork enterprise update. Admins set up a private marketplace through the "Customize" menu, populate it with public plugins, partner-built plugins, or custom plugins built from templates, and then control who sees what via per-user provisioning and team-scoped visibility. If you're at a regulated company that can't pull random plugins from public marketplaces, this is the path: a curated internal catalog where your security team reviews each plugin before publishing.

Team Lead Adoption Path

Don't roll your own bundle on day one. Pattern that works:

  1. Browse the official marketplace with /plugin. Install the code-intelligence plugin for your language.
  2. Adopt one or two community plugins your team agrees on (a test-runner plugin, a migration plugin).
  3. After three months of using primitives directly, identify the three Skills/Hooks/Subagents your team copies between repos. That is your plugin's surface area.
  4. Bundle, publish to your private marketplace, and onboard new repos with one install.

MCP Servers in Practice

The Model Context Protocol (MCP) is the universal connector between Claude Code and external systems. You install a server with claude mcp add <name>, and from then on Claude can talk to that system inside any session. Notion for docs, Linear for tickets, Figma for design specs, your own database for schema-aware queries — pick the server, install once, use everywhere.

The Cowork update on 2026-02-24 added twelve new connectors at once: Google Calendar, Google Drive, Gmail, DocuSign, Apollo, Clay, Outreach, Similarweb, MSCI, LegalZoom, FactSet, WordPress, and Harvey. The fact that legal-tech, sales-ops, and financial-data servers all ship in the same release is the signal: MCP isn't an "engineering integration" anymore, it's the company-wide tool fabric.

Android Dev The Same MCP Servers in Android Studio

This is the part that surprises people. The MCP server you set up for Claude Code also works in Android Studio's Agent Mode. Google explicitly designed Agent Mode to consume remote MCP servers, and the Android Studio team has gone on record about Notion, Figma, Canva, and Linear integrations specifically. That means your Linear MCP server gives Claude Code and Gemini-in-Android-Studio the same view of your sprint board. The investment compounds across tools.

Start with one. Pick the system you reach for ten times a day — for most teams that's the issue tracker — and add it first. Once Claude can fetch ticket descriptions, paste reproduction steps, and update status without you tab-switching, you'll never go back. The other eleven connectors can wait.

Hooks vs Skills vs CLAUDE.md

Hooks are scripts that run automatically at points in the agentic loop — before a tool call, after a file edit, before a commit. They live in .claude/settings.json, and they are deterministic: unlike CLAUDE.md (advisory) and Skills (advisory + on-demand), a Hook always runs. If you say "block writes to migrations/," writes to migrations/ are blocked. No model interpretation, no negotiation, no "the rule got lost in a long context."

The single most useful test from the best-practices doc: "For each rule, ask: would removing this cause Claude to make mistakes? If not, cut it. If yes, ask: should this be advisory (Skill) or guaranteed (Hook)?"

The Three-Layer Defense

DevOps Hook Examples

  • Pre-edit lint: run ESLint on every file Claude touches; block the edit if it would introduce errors
  • Pre-commit secret scan: reject commits where git diff contains an API key pattern
  • Path block: deny writes to migrations/, infra/prod/, or .github/workflows/ without explicit approval
  • Post-tool log: append every Bash command to a session audit log for compliance

Claude can write Hooks for you. Try: "Write a hook that blocks writes to migrations/ and runs ./scripts/typecheck.sh after every edit. Edit .claude/settings.json to add it, and run /hooks to confirm it's active." The doc lists this exact pattern as an idiomatic prompt.

Auto Mode and /sandbox

By default, Claude Code asks permission for every action that modifies your system: file writes, Bash commands, MCP tool calls. Safe but tedious — after the tenth approval you're not reviewing, you're clicking. Anthropic shipped two complementary mechanisms to fix this in 2026.

Auto Mode (March 24, 2026)

Auto Mode replaces click-through approvals with a classifier model. A separate Sonnet 4.6 instance reviews each action and decides whether to approve it. The classifier checks three risk categories:

  • Scope escalation — is Claude doing something beyond what you asked for?
  • Untrusted infrastructure — is the action targeting systems the classifier doesn't recognize?
  • Prompt injection — does the action look driven by hostile content Claude encountered in a file or web page?

Implementation note: it's a two-stage filter. A fast single-token decision ("yes block / no allow") for the common case, then chain-of-thought reasoning only when the first stage flags the transcript. This keeps latency tolerable while preserving thoroughness on suspicious actions. Engage it with --permission-mode auto:

$ claude --permission-mode auto -p "fix all lint errors"

Between launch and 2026-04-16, Auto Mode rolled from research preview to GA on Max, Team, Enterprise, and the API. For non-interactive runs (the -p flag), there's no human to fall back to — if the classifier blocks an action repeatedly, Auto Mode aborts the session rather than hanging.

Sandboxing

Sandboxing is the OS-level companion: Claude runs inside a restricted environment with bounded filesystem and network access. Use /sandbox in an interactive session to enable it. Inside the sandbox, Bash commands are auto-allowed without prompts, because the sandbox itself is the boundary. This is materially different from Auto Mode: Auto Mode reasons about intent, the sandbox enforces capability.

Combine Them

The doc explicitly recommends combining the two: Auto Mode for "what is the agent trying to do?" and the sandbox for "what could it physically reach if it tried?" In production-adjacent environments, run Claude inside a container or VM and use Auto Mode. Anthropic's own guidance: "Do not run it directly on production systems without a backup strategy."

Auto Mode is not "yolo mode." The classifier blocks the most common destructive actions, but it's not infallible. Use it where the blast radius is bounded — a feature branch, a sandbox, a containerized dev env. Don't use it on a checkout that has uncommitted work you can't reproduce. Auto Mode + a fresh worktree + clean git status is the sweet spot.

Agent Teams and Parallel Sessions

Subagents are isolated workers reporting to a boss. Agent Teams are something different: a collaborative squad where one Claude is the lead and others are teammates who can talk to each other directly, self-assign tasks from a shared list, and challenge each other's findings. Anthropic shipped Agent Teams alongside Opus 4.6 on 2026-02-05.

The default size is 3–5 teammates. Bigger teams produce more coordination overhead than parallelism gain. Smaller teams are usually just fine without Agent Teams — subagents work.

The Writer/Reviewer Pattern

The single most useful pattern, straight from the doc:

Session A (Writer)Session B (Reviewer)
"Implement a rate limiter for our API endpoints."  
  "Review the rate limiter in @src/middleware/rateLimiter.ts. Look for edge cases, race conditions, and consistency with our existing middleware patterns."
"Here's the review feedback: [Session B output]. Address these issues."  

Why a separate session? Fresh context defeats author bias. A reviewer that didn't write the code spots problems the writer missed. The same trick works for testing: have one session write tests, then a second session write code to pass them. TDD, parallelized.

Backend Fan-Out Across Files

For large migrations, distribute work across many parallel claude -p invocations:

$ for file in $(cat files.txt); do
    claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
      --allowedTools "Edit,Bash(git commit *)"
  done

--allowedTools is the safety belt. In a batch loop, no human is reviewing each step — you constrain the tool surface up front. Test the prompt on three files, refine, then run on two thousand. The doc walks through this pattern as the canonical migration example.

Worked Example: An Android Repo with the Full Stack

One Android Dev project. Four primitives, one MCP server, working together.

1. The compose-migration Skill

.claude/skills/compose-migration/SKILL.md
---
name: compose-migration
description: Migrate an XML layout to Jetpack Compose, preserving behavior
---
When migrating an XML layout file:
1. Read the XML and identify root layout type
2. Map ConstraintLayout → Box with constraints, LinearLayout → Row/Column
3. Preserve view IDs as testTags
4. Migrate dimens/colors to MaterialTheme tokens
5. Generate a Composable preview with sample data
6. Add a screenshot test using Paparazzi

2. The security-reviewer Subagent

The same one from earlier — tools scoped to Read, Grep, Glob, Bash, model: opus. After every Compose migration, the team lead manually triggers: "Use the security-reviewer subagent on the migrated file." Five lines of summary back; no context bloat.

3. The Linear MCP Server

claude mcp add linear once at setup. Now Claude can read ticket descriptions, paste acceptance criteria into the spec, and update status when the migration ships. Same MCP server is available in Android Studio's Agent Mode — one integration, two tools.

4. The Pre-Commit Hook

In .claude/settings.json: a hook that runs ./gradlew detekt ktlintCheck before every commit and blocks if either fails. Deterministic. The lint rules don't drift, the formatting doesn't drift, and Claude can't accidentally commit code that fails CI.

Engineer types: "Migrate the screen for LIN-4421 to Compose." Claude pulls the ticket from the MCP server, runs the Skill, the subagent reviews, the hook lints, and the PR opens. Half an hour of XML migration, three lines of human input.

Anti-Patterns

Don't

  • Stuff domain knowledge into CLAUDE.md when it only applies to a specific task
  • Reach for a Subagent when a Skill would do (subagents pay context cost on the round-trip)
  • Install every shiny plugin from the marketplace "just in case"
  • Run Auto Mode on a checkout with uncommitted experimental work you can't reproduce
  • Use Hooks for advisory rules that should be Skills
  • Spin up Agent Teams for tasks a single session would finish in five minutes

Do

  • Keep CLAUDE.md short; move conditional knowledge to Skills
  • Use Subagents when context savings outweigh delegation overhead
  • Install plugins deliberately; uninstall ones you haven't used in a month
  • Combine Auto Mode with sandboxing or a worktree for blast-radius control
  • Reserve Hooks for invariants you'd write a CI check for anyway
  • Use Agent Teams for genuinely parallelizable work (cross-layer features, large migrations)
Plugin sprawl is a real failure mode. A plugin that's installed but unused still loads its descriptions into context, still appears in /plugin, and still adds cognitive load when you're picking the right tool. Treat plugins like dependencies: audit them quarterly. "Have I used this in the last 30 days?" If no, uninstall.

Exercise: Author the Stack

Build the Three Primitives in Your Own Project

In a real repository you work in, author the following and commit them:

  1. One Skill. Pick a repeatable task you do in this codebase (release notes, dependency upgrade, schema migration). Write .claude/skills/<name>/SKILL.md. Decide whether it's ambient or side-effect; if side-effect, set disable-model-invocation: true.
  2. One Subagent. Define a focused reviewer or investigator. Whitelist tools narrowly. Pick the model deliberately (opus for hard work, sonnet for everything else).
  3. One Hook. Pick one invariant you wish your CI enforced — secrets in commits, lint pre-commit, blocked paths. Add it to .claude/settings.json. Run /hooks to verify.

Then run a task that exercises all three:

  • Invoke the Skill on a real input
  • Ask Claude to "use the <subagent-name> subagent to review the change"
  • Commit and watch the Hook run

Reflection:

  • Which primitive saved the most context? (Probably the Subagent.)
  • Which felt most "always on"? (Probably the Hook.)
  • Which would a new teammate reach for first? (Whichever you described best in its frontmatter.)

Bonus: Bundle them as a Plugin and publish to a private repo. Onboard a teammate's project with a single /plugin install. That's the moment the extension stack stops being a per-engineer trick and starts being team infrastructure.
