Chapter 5

Context Engineering

The single most important skill: CLAUDE.md, project memory, and feeding the right tokens

Why Context Is King

If you take one thing from this entire book, let it be this: the quality of AI output is determined by the quality of context you provide. Not the quality of your prompt. Not the model you choose. The context.

Context is everything the AI can "see" when processing your request: the system prompt, your project instructions, the files it has read, the conversation history, and the specific prompt you typed. The model reasons over this combined context to produce its output. Bad context produces bad output regardless of how well you phrase your prompt.

As Anthropic's engineering team defines it: context engineering is "curating and maintaining the optimal set of tokens during LLM inference." The key word is curating. Not dumping everything in. Not hoping the model figures it out. Deliberately selecting the smallest set of high-signal tokens that give the model what it needs.

1M
Maximum tokens in Claude's context window — roughly 3,000 pages of text
~4
Characters per token on average — a 500-line file is roughly 2,000-4,000 tokens
10-50x
Impact of good context vs good prompting on output quality

Context Engineering vs Prompt Engineering

These terms are often confused. They're related but distinct skills:

Prompt Engineering

  • How you phrase your request
  • "Write a function that..." vs "Implement..."
  • Instruction clarity, format, constraints
  • What you say in the current message
  • Important but overrated

Context Engineering

  • What information surrounds your request
  • Project structure, conventions, existing code
  • CLAUDE.md, file references, documentation
  • Everything the model sees, not just your message
  • Underrated and far more impactful

A perfectly phrased prompt with poor context produces generic code. A rough prompt with rich context produces code that fits your project. Context always wins.

Backend Compare these two scenarios for the same task — "add a caching layer to the user service":

Rich prompt, no context
Please implement a Redis caching layer for the user service. Use a TTL of 5 minutes. Invalidate on updates. Handle cache misses gracefully with fallback to the database. Use the decorator pattern.

Detailed prompt, but the AI doesn't know your existing cache utilities, Redis client setup, or how your service layer is structured. It'll generate something generic.

Simple prompt, rich context
Add caching to UserService like we did in OrderService.

Short prompt, but the AI has read your CLAUDE.md, knows your cache utilities exist in src/cache/, has seen OrderService's caching pattern, and understands your Redis setup. It produces code that matches your project exactly.

CLAUDE.md: Your Project's Brain

The CLAUDE.md file is the single most impactful thing you can create for AI-driven development. It's a markdown file that Claude Code reads automatically on every invocation. Think of it as onboarding documentation for an AI teammate that starts fresh every session.

The Layered Approach

CLAUDE.md files work in layers, from general to specific:

~/.claude/CLAUDE.md
Global (all projects)
→
project/CLAUDE.md
Project-level
→
project/src/api/CLAUDE.md
Directory-level

Global CLAUDE.md (~/.claude/CLAUDE.md): Your personal preferences that apply everywhere. Coding style, preferred languages, response format.

Project CLAUDE.md (project root): Tech stack, build commands, architecture, conventions. This is the most important one.

Directory CLAUDE.md (any subdirectory): Module-specific context. Only loaded when working in that directory.

Anatomy of a Great CLAUDE.md

# MyApp — Project Instructions for Claude

## Project Overview
E-commerce Android app built with Kotlin, Jetpack Compose, and
a Python FastAPI backend. Monorepo with android/ and backend/ dirs.

## Tech Stack
- Android: Kotlin 2.0, Compose + Material 3, Room, Retrofit, Hilt
- Backend: Python 3.12, FastAPI, SQLAlchemy 2.0, PostgreSQL 16
- CI: GitHub Actions, runs on every PR

## Build & Test Commands
- Android: `./gradlew :app:assembleDebug` and `./gradlew :app:testDebug`
- Backend: `cd backend && pytest` and `cd backend && ruff check .`
- Full CI: `make ci` (runs both)

## Architecture
- Android follows MVVM with UiState sealed interfaces
- Backend follows clean architecture: routes → services → repositories
- All API endpoints are versioned under /api/v2/
- See docs/architecture.md for the full diagram

## Conventions
- Compose: one composable per file, named after the component
- ViewModels: suffix with ViewModel, use StateFlow not LiveData
- Backend: all routes return Pydantic models, never raw dicts
- Tests: follow Arrange-Act-Assert pattern
- Commit messages: conventional commits (feat:, fix:, refactor:)

## Important Rules
- NEVER use runBlocking in Android production code
- NEVER expose database IDs in API responses (use UUIDs)
- ALWAYS add migration scripts for database schema changes
- All new endpoints need integration tests

Android Dev Notice how the CLAUDE.md specifies Android-specific patterns (MVVM, StateFlow, one-composable-per-file). Without these, Claude Code would produce valid Kotlin that doesn't match your team's conventions.

The "new developer" test: Read your CLAUDE.md as if you were a senior developer joining the team on day one. Does it contain everything they'd need to write code that passes code review? If a human would struggle with it, the AI will too. If it's clear enough for a new hire, it's clear enough for Claude.

Project Memory: The .claude/ Directory

Beyond CLAUDE.md, the .claude/ directory stores persistent project configuration that enhances Claude Code's capabilities:

project/
  .claude/
    settings.json          # Project-level permissions and tool config
    commands/
      generate-api.md      # Custom command: /generate-api
      review-pr.md         # Custom command: /review-pr
      run-migration.md     # Custom command: /run-migration

Custom Commands (Skills)

Custom commands are reusable prompts that encode your team's workflows. Create a markdown file in .claude/commands/ and invoke it with /command-name:

# .claude/commands/generate-api.md
# Invoke with: /generate-api

Generate a new API endpoint following our conventions:
1. Create route handler in backend/routes/
2. Create Pydantic request/response models in backend/models/
3. Create service function in backend/services/
4. Create repository function in backend/repositories/
5. Add integration test in backend/tests/integration/
6. Add route to backend/routes/__init__.py
7. Run `cd backend && pytest` to verify

Follow the patterns in the existing orders endpoint as a reference.
Use UUID for all public-facing IDs.

Team Lead Custom commands are how you scale AI workflows across a team. Instead of every developer writing their own prompts for common tasks, you create commands once, commit them to the repo, and everyone uses the same standardized workflow. A team of ten developers all running /generate-api produces consistent, convention-matching code.

DevOps Create commands for your infrastructure workflows:

# .claude/commands/new-service.md
# Invoke with: /new-service

Scaffold a new microservice:
1. Create directory under services/
2. Generate Dockerfile from our template in templates/Dockerfile
3. Create Kubernetes deployment, service, and HPA manifests
4. Add to the CI pipeline in .github/workflows/ci.yml
5. Create Terraform module for the service's AWS resources
6. Add health check endpoint at /health
7. Run `terraform validate` and `kubectl --dry-run=client` to verify

The Context Budget

Even with 1M token contexts, treating the context window as unlimited is a mistake. More tokens means more noise, more chance of contradictory information, and slower responses. Think of context like RAM — you have a lot, but you still need to be intentional about what you load.

What to Include

  • CLAUDE.md — always (loaded automatically)
  • Files directly relevant to the task — the files being modified and their direct dependencies
  • Example files — one or two files that demonstrate the pattern you want followed
  • Test files — existing tests that show your testing conventions
  • Error messages — full stack traces when debugging

What to Exclude

  • Entire codebases — don't dump everything; let the agent search for what it needs
  • Generated files — build artifacts, node_modules, compiled output
  • Irrelevant history — long conversation threads about unrelated topics
  • Redundant documentation — don't paste the same info that's already in CLAUDE.md
More context is not always better. Anthropic's research found that stuffing the context window with marginally relevant information can actually decrease output quality. The model has to sort through noise to find the signal. Curate aggressively: include what's necessary, exclude what's merely related.

Context Rot

Context rot happens when the information in your context becomes stale, contradictory, or misleading. It's the AI equivalent of outdated documentation — except the AI trusts it implicitly.

Common Causes

  • Stale CLAUDE.md — your project evolved but the instructions didn't. The CLAUDE.md says "use LiveData" but you migrated to StateFlow six months ago.
  • Long conversations — early messages in a long session may describe a state that no longer exists. You changed the approach on message 15, but messages 1-14 still describe the old plan.
  • Contradictory instructions — a directory-level CLAUDE.md contradicts the project-level one. Or your prompt contradicts CLAUDE.md.
  • Cached understanding — the AI read a file early in the session. The file changed (by you or another tool). The AI still reasons from the old version.

Prevention

  • Review CLAUDE.md monthly — add it to your sprint retrospective checklist
  • Start fresh sessions for new tasks — don't reuse long conversations
  • Use /compact in Claude Code to compress old context and focus on what's current
  • Be explicit when changing direction: "Ignore what I said earlier about X. We're now doing Y instead."

Full-Stack Context rot is especially dangerous in monorepos where frontend and backend conventions differ. A single CLAUDE.md that says "use camelCase" works for your React frontend but not your Python backend (which uses snake_case). Use directory-level CLAUDE.md files to avoid this contradiction: one for frontend/, one for backend/.

Repo Structure as Context

Your file and directory organization is itself context. A well-structured repo helps AI navigate and generate code in the right place. A messy repo confuses both humans and AI.

Poor structure (ambiguous)
src/ utils.py (3,000 lines) helpers.py (2,000 lines) main.py (5,000 lines) models.py (all models) tests.py (all tests)

AI can't tell where new code should go. Everything lands in the wrong file.

Clear structure (self-documenting)
src/ orders/ routes.py service.py repository.py models.py tests/ users/ routes.py service.py ...

AI immediately understands where to put new order-related code. The structure IS the context.

Android Dev Feature-based package structure (feature/settings/, feature/profile/) gives AI clear boundaries. If you ask Claude to "add a settings screen," it knows exactly where to create the files. With a flat structure (viewmodels/, screens/, repositories/), the AI has to guess which existing files to reference.

Documentation as AI Fuel

Documentation that was "nice to have" for humans becomes critical infrastructure for AI. Every piece of documentation you write is context that makes AI output better:

  • README.md — Project overview, setup instructions, architecture summary
  • Architecture Decision Records (ADRs) — Why decisions were made, not just what they are
  • API documentation — Endpoint contracts, request/response examples
  • Inline comments on non-obvious code — The "why", not the "what"
  • CHANGELOG — What changed and why, helps AI understand project evolution

The Documentation ROI Shift

In the pre-AI era, documentation had a delayed, uncertain payoff — maybe a future developer would read it. In the AI era, documentation has an immediate, guaranteed payoff: the AI reads it on every single invocation. Good docs make AI produce better code today, not hypothetically tomorrow. This changes the cost-benefit analysis of writing documentation entirely.

Runtime Retrieval: Just-in-Time Context

Not all context needs to be pre-loaded. Claude Code and other agentic tools can retrieve context at runtime — reading files, searching codebases, and pulling in information as needed. This "just in time" approach is often better than "just in case" loading.

Backend Instead of pre-loading every model file into context, let the agent discover what it needs:

# Instead of manually listing files:
$ claude "Add a new endpoint for product reviews. Here are the relevant
  files: models/user.py, models/product.py, models/order.py..."

# Let the agent find what it needs:
$ claude "Add a new endpoint for product reviews. Follow the patterns
  in our existing endpoints. Check the models/ and routes/ directories
  for conventions."

# The agent reads the directory, finds the relevant files, and loads
# only what it needs. Often more effective because it discovers
# dependencies you forgot to mention.
Trust the agent's search. CLI agents like Claude Code are good at finding relevant files. Rather than manually curating every file reference, give the agent a clear goal and let it navigate. You save time, and the agent often finds context you'd have missed.

Multi-File Context Strategies

For complex tasks that span multiple files, how you reference context matters:

In Claude Code

# Reference specific files
$ claude "Refactor the auth middleware. See src/middleware/auth.py
  for the current implementation and src/middleware/rate_limit.py
  for the pattern to follow."

# Reference by pattern
$ claude "Update all route handlers in src/routes/ to use the
  new error response format defined in src/utils/errors.py"

# Let the agent explore
$ claude "Find all places where we handle user permissions and
  ensure they follow the same RBAC pattern."

In Cursor

// Use @ mentions in the chat panel:
// @src/services/UserService.ts @src/models/User.ts
// "Add email verification to the user signup flow"

// Reference documentation:
// @docs/api-spec.yaml
// "Generate the client-side types from this API spec"

Freelancer When working on an unfamiliar client codebase, use the exploration strategy: ask Claude Code to map the architecture first ("Explain the directory structure and how modules interact"), then use its findings as context for subsequent tasks. This builds your understanding and the AI's context simultaneously.

Exercise: Write a CLAUDE.md for Your Project

Create a comprehensive CLAUDE.md for a project you're actively working on. Use this template as a starting point, then customize:

  1. Start with claude /init to generate a baseline
  2. Add your project overview: What does this project do? What's the tech stack?
  3. Add build and test commands: How to build, test, lint, and deploy
  4. Document architecture: Main modules, data flow, key design patterns
  5. List conventions: Naming, file organization, code style, testing approach
  6. Add "important rules": Things the AI must never do (or always do)
  7. Reference example files: Point to specific files that exemplify your patterns
  8. Test it: Ask Claude Code to perform a task and verify it follows your CLAUDE.md

Spend 20-30 minutes on this. A good CLAUDE.md pays for itself within a day of AI-assisted development. Review it with your team and commit it to version control.
