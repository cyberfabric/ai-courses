Phase 1: Ideation and Research

AI accelerates the earliest stage of a project: validating whether an idea is worth building and understanding the technical landscape.

# Explore technical feasibility
$ claude "I want to build a habit-tracking app for Android that:
  - Tracks daily habits with streaks
  - Shows progress charts (weekly/monthly)
  - Sends smart reminders based on user patterns
  - Works fully offline with cloud sync

  Questions:
  1. What's the best tech stack for this in 2026?
  2. What are the hardest technical challenges?
  3. Are there open-source libraries that solve the hard parts?
  4. What's a realistic MVP scope for a solo developer?"

Freelancer For client projects, use AI to quickly assess scope and complexity during the proposal phase. "Analyze this feature list and estimate relative complexity (low/medium/high) for each item. Identify technical risks and dependencies between features." This produces a better proposal in 30 minutes than a day of manual estimation.

AI for competitive research: Ask the AI to analyze existing solutions and identify gaps. "What are the top 5 habit-tracking apps on Android? What do users complain about in their reviews? What features are missing that we could differentiate on?" This isn't market research — it's focused technical scoping using the AI's knowledge base.

Phase 2: Architecture Design

Architecture is where your expertise matters most. AI can propose architectures, but you evaluate them against your specific constraints: team size, deployment targets, scalability needs, and maintenance capacity.

$ claude "Design the architecture for the habit tracker app.

  Constraints:
  - Solo Android developer, may add iOS later (consider KMP)
  - Must work fully offline (offline-first architecture)
  - Cloud sync via simple REST API (Firebase or custom backend)
  - Budget: minimal infrastructure cost
  - Target: 10,000 users in year one

  Produce:
  1. High-level architecture diagram (described in text)
  2. Tech stack recommendations with justification
  3. Module structure for the Android app
  4. Data model (entities and relationships)
  5. Sync strategy for offline-first
  6. List of key architectural decisions (ADRs)"

Backend For backend projects, have the AI produce architecture alternatives with trade-offs: "Compare a monolithic FastAPI backend vs microservices for our e-commerce platform. Consider: team size (4 devs), expected traffic (1000 RPM initially, 10x in 18 months), deployment complexity, and operational overhead. Recommend one approach with justification."

Validate AI architecture proposals. AI produces plausible architectures, but it lacks the context of your team's operational maturity, your infrastructure constraints, and your tolerance for complexity. An AI might suggest Kubernetes for a project that would be perfectly served by a single VM with Docker Compose. Always filter architectural suggestions through your real-world constraints.

Phase 3: Project Scaffolding

With the architecture decided, AI generates the project structure: boilerplate, configuration, CI/CD, and the foundational code that every project needs.

Android Dev Android Project Scaffold

$ claude "Scaffold the habit tracker Android project:

  1. Create the module structure:
     - app/ (main application)
     - core/data/ (Room database, repositories)
     - core/network/ (Retrofit API, sync)
     - core/ui/ (shared Compose components, theme)
     - feature/habits/ (habit list and creation)
     - feature/stats/ (charts and progress)
     - feature/settings/ (preferences)

  2. Set up:
     - Gradle version catalog (libs.versions.toml)
     - Hilt dependency injection
     - Room database with initial migration
     - Retrofit with OkHttp for API
     - Material 3 theme with dark mode support
     - Navigation Compose with type-safe routes
     - GitHub Actions CI (build + test on every PR)

  3. Create CLAUDE.md with project conventions
  4. Create .cursorrules for Cursor users
  5. Run ./gradlew assembleDebug to verify"

DevOps Scaffold the deployment infrastructure alongside the application:

$ claude "Create the infrastructure scaffold:
  - Dockerfile for the backend service
  - docker-compose.yml for local development (app + postgres + redis)
  - Terraform module for AWS deployment (ECS Fargate + RDS + ElastiCache)
  - GitHub Actions workflow: build → test → deploy to staging
  - Monitoring: basic CloudWatch dashboards and alarms
  - Put all infra in infra/ directory with its own CLAUDE.md"
Scaffold once, customize forever. The scaffolding phase is where AI saves the most time relative to manual work. A project scaffold that takes a day to set up manually takes 15-20 minutes with AI. The key is being specific about your conventions upfront so the generated boilerplate matches your standards from the start.

  Run tests after each layer."

# Step 5: Review and validate (10 min)

# Total: ~50 min for a complete feature with tests

Full-Stack For web projects, apply the same cycle per feature but across the full stack: "Implement the user registration feature. Start with the database migration, then the API endpoint with validation, then the React form component, then end-to-end tests. Follow the spec in docs/specs/registration.md."

Repeat this cycle for each feature. By Feature 3, the pattern is muscle memory. By Feature 5, you're moving faster because earlier features established patterns the AI follows automatically.

Phase 5: Testing and QA

If you've followed the TDD workflow from Chapter 8, you already have tests for each feature. Phase 5 adds the project-wide quality pass.

# Coverage audit
$ claude "Analyze our test coverage across the entire project.
  For each module, identify:
  - Current test coverage percentage
  - Untested public functions
  - Missing edge case tests
  - Missing error path tests
  Generate the missing tests for the top 10 gaps."

# Integration testing
$ claude "Write end-to-end integration tests that verify:
  1. Create a habit → check it in → see streak increment
  2. Create multiple habits → view stats → charts show correct data
  3. Offline: create habit while offline → sync when online → verify
  Use our test fixtures and follow existing integration test patterns."

# Security audit
$ claude "Review the entire codebase for security issues.
  Focus on: input validation, SQL injection, data exposure in logs,
  authentication bypass, and insecure defaults.
  Produce a report with severity (critical/high/medium/low) and fix
  suggestions for each finding."

Team Lead For team projects, add a pre-release quality gate: AI reviews the complete codebase against a quality checklist, produces a report, and the team reviews the findings before release. This catches cross-feature issues that per-feature reviews miss.

Phase 6: Documentation

AI generates documentation from the code it helped create. Since it understands the codebase, the documentation is accurate and comprehensive.

# Generate README
$ claude "Generate a comprehensive README.md for this project:
  - Project description and screenshots placeholder
  - Setup instructions (prerequisites, clone, build, run)
  - Architecture overview (reference docs/architecture.md)
  - Contributing guidelines
  - License section"

# Generate API documentation
$ claude "Generate OpenAPI documentation for all endpoints in
  the backend. Include request/response examples, error codes,
  and authentication requirements."

# Update CLAUDE.md
$ claude "Update CLAUDE.md to reflect the current state of the
  project. Add any conventions that emerged during development
  that aren't documented yet."
Documentation as a project milestone, not an afterthought. Schedule documentation as a formal phase. AI makes it fast enough that there's no excuse to ship without it. Good documentation also improves future AI interactions with your project — it's an investment in your own productivity.

Phase 7: Deployment and Monitoring

DevOps AI assists with the deployment pipeline and monitoring setup:

# CI/CD pipeline
$ claude "Create a complete GitHub Actions deployment pipeline:
  - On PR: build, test, lint, security scan
  - On merge to main: build, test, deploy to staging
  - On release tag: deploy to production
  - Include rollback step if health check fails
  - Use our existing AWS ECS setup in infra/"

# Monitoring
$ claude "Set up monitoring for the production deployment:
  - CloudWatch alarms for: error rate > 1%, latency p99 > 500ms,
    CPU > 80%, memory > 85%
  - Structured logging with correlation IDs
  - Health check endpoint at /health with dependency status
  - Add to the Terraform in infra/modules/monitoring/"

Time Comparison: Traditional vs AI-Driven

Here's a realistic comparison for a medium-complexity Android app (7 features, backend API, CI/CD):

Traditional (Solo Dev)

  • Architecture: 2-3 days
  • Scaffolding: 1-2 days
  • 7 features: 4-6 weeks
  • Testing: 1-2 weeks
  • Documentation: 2-3 days
  • Deployment: 2-3 days
  • Total: 8-10 weeks

AI-Driven (Solo Dev)

  • Architecture: 0.5-1 day
  • Scaffolding: 2-4 hours
  • 7 features: 1.5-2.5 weeks
  • Testing: 2-3 days (mostly built with features)
  • Documentation: 0.5-1 day
  • Deployment: 0.5-1 day
  • Total: 3-4 weeks

The speedup isn't uniform. Architecture gets a 2-3x boost (AI proposes, you decide). Scaffolding gets 5-10x (pure boilerplate generation). Feature development gets 2-3x (spec + AI implementation). Testing gets 3-5x (AI-generated from specs). Documentation gets 5-10x (AI generates from code). Deployment gets 2-3x (AI-generated IaC and pipelines).

Where the Time Goes

In traditional development, 60-70% of your time is implementation (typing code). In AI-driven development, 60-70% of your time is thinking: writing specs, reviewing output, making architectural decisions, and validating quality. The total project time shrinks by 2-3x, but the nature of your work shifts dramatically. You're not typing less — you're thinking more productively.

Exercise: Start a New Project with AI

Apply the full AI-driven workflow to a real project from scratch:

  1. Choose a project idea — something you'd actually want to build. A personal tool, a side project, or a client prototype. Keep scope small: 3-5 features for the MVP.
  2. Phase 1 (30 min): Use AI for ideation and feasibility. Define your MVP scope.
  3. Phase 2 (1 hour): AI-assisted architecture design. Get a tech stack, module structure, and data model. Write 2-3 key ADRs.
  4. Phase 3 (1 hour): Scaffold the project with AI. Generate boilerplate, CI, and CLAUDE.md.
  5. Phase 4 (remainder): Build your first feature using the full SDD + TDD cycle: spec → tests → implement → review.
  6. Reflect: After the first feature, note what worked and what was friction. Adjust your CLAUDE.md and workflow for the second feature.

Don't try to finish the whole project in one session. The exercise is about experiencing the full lifecycle, not rushing to completion. Build one feature properly using every technique from this book, then iterate.
