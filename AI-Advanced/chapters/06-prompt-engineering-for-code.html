Chapter 6

Prompt Engineering for Code

Code-specific prompting for generation, debugging, refactoring, and review

Beyond Generic Prompting

Chapter 5 taught you to curate context — the information surrounding your request. This chapter focuses on the request itself: how to write prompts that produce correct, convention-matching, production-ready code.

Code prompting is different from general prompting. You're not asking for an essay or a summary. You're asking for a precise artifact that must compile, pass tests, handle edge cases, and fit into an existing system. The margin for "close enough" is zero — code either works or it doesn't.

Anatomy of a Good Coding Prompt

Every effective coding prompt has four components. You don't need all four for every prompt, but knowing the structure helps you identify what's missing when output is wrong.

Task
+
Context
+
Constraints
+
Examples
  • Task — What you want done. Be specific about the action: "implement," "refactor," "fix," "add tests for," "review."
  • Context — What the AI needs to know. Relevant files, existing patterns, the problem being solved. (Chapter 5 covered this in depth.)
  • Constraints — What the AI must or must not do. Framework choices, performance requirements, files to leave untouched.
  • Examples — Reference implementations. "Follow the pattern in OrderService.kt" or "Match the style of existing tests in tests/unit/".

The Precision Principle

The specificity of your prompt should match the criticality of the task. Quick utility function? A one-liner is fine. Payment processing logic? Spell out every constraint, edge case, and security requirement. Scale your investment in the prompt to the risk of getting it wrong.

Generating Code

Code generation is the most common use case. The pattern: describe what you need, constrain how it should be built, and reference what it should look like.

Android Dev Generating a Compose Screen

Vague
Create a profile screen

No tech stack, no design details, no patterns to follow. You'll get generic code that doesn't match your project.

Precise
Create a ProfileScreen composable in feature/profile/ProfileScreen.kt. Layout: user avatar (circular, 80dp), name and email below it, then a LazyColumn with settings items (icon + label + chevron). Follow the pattern in feature/settings/SettingsScreen.kt for the list items. Use our AppTheme from ui/theme/. ViewModel should use UiState sealed interface like HomeViewModel. Include @Preview with sample data.

Specific file location, layout details, pattern references, and verification (preview).

Backend Generating an API Endpoint

Full-stack endpoint prompt
Add a PATCH /api/v2/users/{id}/preferences endpoint. Request body: { theme: "light"|"dark", language: string, notifications: { email: bool, push: bool } } Response: 200 with the updated preferences object, 404 if user not found, 422 if validation fails. Implementation: - Route handler in backend/routes/users.py - Pydantic models in backend/models/user_preferences.py - Service function in backend/services/user_service.py (add to existing) - Repository: use existing UserRepository.update_preferences() - Add integration test in backend/tests/integration/test_users.py Follow the pattern of the existing PATCH /api/v2/users/{id}/profile endpoint.

Complete contract, file locations, layers, pattern reference, and test requirement.

DevOps Generating Infrastructure

# Generating a Terraform module
$ claude "Create a Terraform module in infra/modules/rds-postgres/ for
  a PostgreSQL 16 RDS instance.

  Requirements:
  - db.r6g.large instance in our existing VPC (var.vpc_id)
  - Multi-AZ with one read replica
  - Encrypted at rest with our KMS key (var.kms_key_arn)
  - Automated backups retained 30 days
  - Security group allowing access only from our ECS tasks
  - Parameter group with shared_buffers=256MB, max_connections=200
  - Output: endpoint, port, connection string (as SSM parameter)

  Follow the structure of our existing infra/modules/rds-mysql/ module."

Debugging Prompts

Debugging with AI is where many developers get the most immediate value. The key: give the AI everything it needs to reproduce your mental model of the problem.

The Debugging Prompt Template

# What I expected:
[Expected behavior]

# What actually happens:
[Actual behavior, exact error message or stack trace]

# Steps to reproduce:
[How to trigger the issue]

# What I've already tried:
[Approaches that didn't work and why]

# Relevant code:
[File paths or snippets]

Android Dev Example: Compose Recomposition Bug

Effective debugging prompt
Bug: The item count badge in CartScreen.kt recomposes on every frame, causing UI jank. Expected: Badge updates only when cart items change. Actual: Logcat shows "CartBadge recomposing" logging continuously, even with no user interaction. FPS drops to ~30. Setup: CartScreen observes cartViewModel.uiState (StateFlow). The badge reads uiState.value.items.size. I suspect the UiState is being recreated on every emission even when items haven't changed, but I'm not sure where the unnecessary emission originates. Relevant files: feature/cart/CartScreen.kt, feature/cart/CartViewModel.kt, data/repository/CartRepository.kt

Specific symptoms, performance data, hypothesis, and file references. The AI can investigate the exact emission chain.

Always include the actual error. Don't paraphrase error messages or stack traces. Copy-paste the exact output. AI models are trained on real error messages and can often identify the root cause from the precise wording. "It crashes when I tap the button" is useless. The full NullPointerException stack trace is gold.

Refactoring Prompts

Refactoring prompts are uniquely tricky because you need to specify what should change and what must not. Without explicit boundaries, AI tends to "improve" things you didn't ask it to touch.

The Refactoring Prompt Template

# Goal:
[What the refactoring achieves — why, not just what]

# Scope:
[Exactly which files/functions/classes to change]

# Do NOT change:
[What must stay the same — public APIs, tests, behavior]

# Target pattern:
[What the code should look like after — reference examples]

# Verification:
[How to confirm the refactoring is correct — existing tests should pass]

Full-Stack Example: Extracting a Hook

Bounded refactoring prompt
Refactor: Extract the authentication logic from LoginPage.tsx and SignupPage.tsx into a shared useAuth() hook. Goal: Eliminate duplicated auth state management across two components. Scope: Only modify src/pages/LoginPage.tsx, src/pages/SignupPage.tsx, and create src/hooks/useAuth.ts. Do NOT change: - The visual UI of either page - The API calls in src/api/auth.ts - Any test files (I'll update those separately) The hook should expose: { user, isLoading, login, signup, logout, error } Follow the pattern of our existing useCart() hook in src/hooks/useCart.ts.

Clear goal, explicit scope, explicit exclusions, target interface, and pattern reference.

"Do NOT change" is critical. Without explicit exclusions, AI will often "helpfully" rename variables, reorganize imports, update adjacent code, or refactor things you didn't ask about. This makes diffs noisy and code review harder. Always specify boundaries.

Code Review Prompts

AI-assisted code review works best when you specify what kind of review you want. "Review this code" produces generic feedback. Targeted prompts produce actionable findings.

Team Lead Review Prompt Variants

# Security-focused review
$ claude "Review the authentication changes in this diff for security
  vulnerabilities. Focus on: injection attacks, improper access
  control, token handling, and secret exposure. Ignore style issues."

# Performance-focused review
$ claude "Review the database queries in src/repositories/ for
  performance issues. Look for: N+1 queries, missing indexes,
  unbounded result sets, and unnecessary JOINs."

# Architecture-focused review
$ claude "Review this PR for architectural consistency. Does the new
  code follow our existing patterns? Are there any violations of our
  layer boundaries (routes should not access repositories directly)?
  Does it introduce any new dependencies that should be discussed?"

# Newcomer review (for onboarding)
$ claude "Explain what this PR does as if I'm a new developer joining
  the team. What's the business context? What patterns does it use?
  What would I need to understand to modify this code in the future?"
Combine AI review with human review. Let AI handle the first pass: style, common bugs, security patterns, test coverage. You focus on architecture fit, business logic correctness, and the things that require project context to evaluate. This division doubles your effective review throughput.

The Iterative Loop

Rarely does the first prompt produce perfect code. The real workflow is iterative:

Prompt
→
Review
→
Refine
→
Accept

The refinement step is where your judgment matters. Instead of reprompting from scratch, give specific, targeted feedback:

# Round 1: Initial generation
$ claude "Add pagination to the /api/users endpoint"

# Round 2: Targeted refinement
$ claude "Good, but three changes needed:
  1. Use cursor-based pagination, not offset-based (we need stable
     pagination for the mobile client)
  2. Default page size should be 20, max 100
  3. Add the pagination metadata to the response envelope,
     not as separate headers"

# Round 3: Final adjustment
$ claude "The cursor encoding looks correct. One issue: the 'next'
  cursor should be null (not empty string) when there are no more
  results. Fix that and add a test case for the last page."

Each iteration gets more specific. You're not re-describing the entire task — you're making surgical corrections. This is faster and produces better results than starting over.

Freelancer The iterative loop is where your development speed actually lives. A vague first prompt followed by five rounds of "no, I meant..." is slower than a precise first prompt with one refinement. Invest time upfront to reduce iteration cycles.

Prompt Templates for Common Tasks

Templates eliminate the "blank page" problem. Start with a template, fill in the specifics, and you'll consistently produce better prompts than freestyling.

Template: New UI Component

Create a [component name] composable/component in [file path].

Visual: [describe the layout, spacing, and elements]
State: [what state it needs, where it comes from]
Interactions: [what happens on tap, swipe, etc.]
Follow the pattern in [reference file].
Include preview/storybook with sample data.

Template: Data Model + Repository

Create a [Model] entity with fields: [field list with types].

Database: [Room entity / SQLAlchemy model / Prisma schema]
Repository: CRUD operations + [any custom queries]
Location: [file paths for model and repository]
Follow naming and patterns in [reference model].
Include unit tests for the repository.

Template: Bug Fix

Fix: [one-line description of the bug]

Expected: [correct behavior]
Actual: [current broken behavior]
Error: [exact error message or stack trace]
Cause (if known): [your hypothesis]
Files: [relevant file paths]

Fix the bug and add a regression test to prevent recurrence.

Template: Test Suite

Write tests for [class/function] in [file path].

Test these scenarios:
- [happy path]
- [edge case 1]
- [edge case 2]
- [error condition]

Use [test framework]. Follow patterns in [reference test file].
Mock [external dependencies] but use real [internal components].

Backend Store templates in your project's .claude/commands/ directory so they're reusable by the whole team. A /new-endpoint command with a pre-filled template is faster than remembering the template structure each time.

Meta-Prompting: AI Writes Your Prompts

Here's a power technique: use AI to generate prompt templates for your specific project. This bootstraps your prompt library faster than writing everything by hand.

$ claude "Analyze our codebase and create 5 reusable prompt templates
  for the most common tasks in this project. Each template should:
  - Reference our actual file structure and naming conventions
  - Include placeholders for variable parts
  - Reference specific example files to follow
  - Include verification steps

  Save them as custom commands in .claude/commands/ so the team
  can use them with /command-name."

The AI reads your project, identifies patterns, and generates templates that are already customized for your codebase. Review and refine them, but you'll be surprised how good the starting point is.

Meta-prompting works for any task type. "Write me a prompt template for reviewing PRs in this project." "Write me a prompt template for adding a new database migration." "Write me a prompt template for creating a new feature module." The AI generates the template; you refine it once and reuse it forever.

Common Prompting Mistakes

  • Too vague: "Fix the bug" → Add which bug, where, with what symptoms
  • Too verbose: A 500-word prompt for a one-line change wastes tokens and confuses focus
  • No verification criteria: "Add feature X" → Add "and verify by running these tests"
  • No pattern reference: "Create a service" → Add "following the pattern in ExistingService"
  • Ambiguous scope: "Improve the code" → Specify exactly what "improve" means (performance? readability? security?)
  • Missing constraints: "Add logging" → Specify where, what level, what format, what NOT to log (PII)
  • Re-explaining context: Don't repeat what's in CLAUDE.md — the AI already has it

Exercise: Build Your Prompt Library

Create a personal prompt library for the 10 tasks you do most frequently. This library becomes a reusable asset that saves time on every future task.

  1. List your top 10 recurring development tasks (e.g., new endpoint, new screen, bug fix, migration, test writing, PR review, refactoring, docs update, CI fix, dependency update)
  2. For each task, write a prompt template using the structures from this chapter. Include:
    • Task description with placeholders
    • Context references specific to your project
    • Constraints and exclusions
    • Verification criteria
  3. Save as custom commands in your project's .claude/commands/ directory
  4. Test each template on a real task. Refine based on the output quality.
  5. Share with your team — commit the commands to version control

Bonus: Use meta-prompting to bootstrap the library. Run: claude "Analyze this project and generate custom command templates for our most common tasks. Save them in .claude/commands/"
