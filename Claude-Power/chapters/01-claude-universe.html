Chapter 1

The Claude Universe

A complete map of every Claude product — and when to reach for each one

You use Claude. Maybe you chat with it on claude.ai, maybe you run Claude Code in your terminal, maybe you've hit the API a few times. But here's the thing — most developers only use one piece of the Claude ecosystem, and they use it for everything.

That's like owning a Swiss Army knife and only using the corkscrew.

This chapter gives you the full map. By the end, you'll know exactly which Claude product to reach for in any situation — and you'll stop forcing a screwdriver into a job that needs pliers.

The Ecosystem at a Glance

Claude isn't one product. It's a family of interfaces, all powered by the same underlying models but designed for very different jobs. Here's the quick overview:

5 Product surfaces
3 Model tiers
1M Token context (Opus & Sonnet)
Claude.ai → Claude Code CLI → IDE Extensions → Claude API → Agent SDK

Same brain, five different bodies. Let's meet each one.

Claude.ai — The Conversational Interface

This is where most people start. You open claude.ai in your browser (or the mobile app), type a question, get an answer. Simple. But Claude.ai is much more than a chat box.

When to use Claude.ai

  • Brainstorming and ideation — exploring ideas, weighing options, thinking out loud
  • Writing and editing — drafts, emails, documentation, technical writing
  • Research and analysis — summarizing papers, comparing approaches, exploring tradeoffs
  • Learning — understanding new concepts, getting explanations, asking "why"
  • Quick prototypes — using Artifacts to build interactive demos right in the chat
Claude.ai has features most users never discover: Projects (persistent context), Artifacts (interactive code outputs), Styles (tone presets), and custom instructions. We'll deep-dive into all of these in Chapter 2.

When NOT to use Claude.ai

  • Editing files on your machine — use Claude Code instead
  • Running terminal commands — use Claude Code instead
  • Automated pipelines — use the API instead
  • Building AI features into your app — use the API or Agent SDK

Claude Code CLI — The Developer's Power Tool

If Claude.ai is the friendly conversation partner, Claude Code is the hands-on pair programmer who can actually touch your files, run your tests, and commit your code.

Claude Code runs in your terminal. It reads your codebase, understands your project structure (via CLAUDE.md files), and can execute real actions: editing files, running commands, searching code, managing git — all through natural language.

When to use Claude Code CLI

  • Writing and refactoring code — it reads your files, understands context, makes edits
  • Git workflows — commits, branch management, PR creation
  • Debugging — it can read logs, run your code, trace errors
  • Code review — feed it a diff and get systematic feedback
  • Automation — use --print mode in CI/CD pipelines and scripts
  • File operations — searching, creating, moving, bulk editing
Find all the TODO comments in this project, group them by priority, and create a GitHub issue for each high-priority one.
I'll search for TODOs across the codebase, categorize them, and use the GitHub CLI to create issues. Let me start by scanning your files...

That's not a hypothetical. Claude Code can actually do that — it searches your files, parses the results, runs gh issue create, and reports back. This is the kind of multi-step task that would take you 20 minutes manually.

Basic usage: "Fix the bug in auth.js"

You paste code, get a suggestion, manually apply it, hope it works.

Power user: "Find the failing test in auth, trace the root cause, fix it, run the tests to verify, and commit with a descriptive message."

Claude Code does it all — reads, edits, runs, verifies, commits.

Claude Code in Your IDE

Don't want to leave your editor? You don't have to. Claude Code is available as extensions for both VS Code and JetBrains IDEs.

VS Code Extension

The VS Code extension gives you a native graphical panel right inside your editor. You can:

  • Chat with Claude in a dedicated panel (or multiple tabs)
  • @-mention files with specific line ranges from your selection
  • Review Claude's proposed edits as inline diffs before accepting
  • Auto-accept edits in real time as Claude works
  • Access your full conversation history

JetBrains Plugin

The JetBrains plugin runs the Claude Code CLI inside your IDE's integrated terminal and opens proposed changes in the IDE's native diff viewer. It's a slightly different experience — less graphical, more terminal-native — but the workflow is the same: ask, review, accept.

When to use IDE extensions vs the CLI

Situation Best choice
Quick question about a selected block of code IDE extension
Multi-file refactoring across the project CLI (better context management)
You want to see diffs inline before accepting IDE extension
You're running complex commands (git, tests, deploys) CLI
You're already in the terminal CLI
You want to @-mention specific files and line ranges IDE extension
Open Claude Code in VS Code: Press Ctrl+Shift+P (or Cmd+Shift+P on Mac), then type "Claude" to find all Claude Code commands. Or just use the Claude icon in the sidebar.

The Claude API — Building Claude Into Your Apps

The API is for when you want Claude inside your software. Not as a chat partner, not as a coding assistant — as a component in your system.

When to use the API

  • Custom integrations — Claude as part of your product's backend
  • Automated pipelines — batch processing, content generation, data analysis
  • Product features — AI-powered search, summarization, classification in your app
  • Fine-tuned control — you need specific parameters, system prompts, tool definitions

The API is available through official SDKs in Python and TypeScript, or via direct HTTP calls. Key capabilities include:

  • Messages API — the core interface for sending prompts and receiving responses
  • Streaming — get tokens in real time instead of waiting for the full response
  • Tool use (function calling) — let Claude call your functions and act on the results
  • Extended thinking — Claude reasons step-by-step before answering
  • Vision — send images for analysis alongside text
  • Prompt caching — reduce costs dramatically for repeated context
Don't use the API for one-off questions. That's what Claude.ai is for. The API makes sense when you need programmatic access — automation, integration, or scale. If you're copying API responses into a chat window, you're doing it wrong.

The Claude Agent SDK — Autonomous Agents

The Agent SDK is the newest and most powerful piece of the puzzle. It gives you the same tools, agent loop, and context management that power Claude Code — but programmable in Python and TypeScript.

What makes agents different from API calls?

A regular API call is: send prompt → get response. Done.

An agent is: send goal → Claude plans steps → uses tools → observes results → adjusts plan → uses more tools → keeps going until done.

API approach:

You write the loop. You define when to call which tool. You handle errors. You decide when to stop.

You are the orchestrator.

Agent SDK approach:

You define the tools and the goal. Claude decides when to call which tool, handles errors, and keeps going until the goal is met.

Claude is the orchestrator.

When to use the Agent SDK

  • Multi-step tasks — anything that requires planning, execution, and adaptation
  • Complex workflows — tasks where the next step depends on the result of the previous one
  • Building your own "Claude Code" — specialized coding agents for your domain
  • Autonomous operations — agents that monitor, analyze, and act on their own

The Model Family: Opus, Sonnet, Haiku

Every Claude product runs on the same model family. Think of them as three engine sizes — you pick the one that matches your job.

Model Best for Context API Cost (per MTok) Speed
Opus 4.7 (flagship) Agentic coding, long-horizon tasks, deep analysis 1M tokens $5 in / $25 out Moderate
Sonnet 4.6 Everyday coding, balanced quality/speed 1M tokens $3 in / $15 out Fast
Haiku 4.5 Quick tasks, classification, high-volume pipelines 200K tokens $1 in / $5 out Fastest
Opus 4.6 (legacy) Previous flagship — still active on the API 1M tokens $5 in / $25 out Moderate

Opus 4.7 launched April 16, 2026 at the same price as 4.6 and jumped from 80.8% to 87.6% on SWE-bench Verified — the biggest single-version coding gain in the family so far. One thing to know if you call the API directly: 4.7 supports adaptive thinking only (the older extended-thinking request shape returns a 400 error). We’ll cover what that means for your code in Chapter 7.

Sonnet 4.6 scores 79.6% on SWE-bench Verified — about 8 points behind Opus 4.7’s 87.6%, but at 60% of the price and noticeably faster on every turn. For most everyday development tasks, Sonnet is still the sweet spot. Reach for Opus 4.7 when you’re running deep agentic loops or analyzing very long context. Use Haiku for bulk operations where speed and cost matter more than depth.

Which model runs where?

  • Claude.ai — defaults to Sonnet, you can switch to Opus or Haiku in the model selector
  • Claude Code CLI — defaults to Sonnet, switch with --model flag or /model command
  • IDE extensions — follows your Claude Code model selection
  • API — you specify the model in every request (claude-opus-4-7, claude-sonnet-4-6, claude-haiku-4-5-20251001; the older claude-opus-4-6 ID still resolves for legacy code)

How They All Connect

Here's what makes the Claude ecosystem powerful: it's all the same brain. The same model that chats with you on claude.ai is the one editing your files in Claude Code and processing API requests in your backend.

That means:

  • Techniques you learn in one surface transfer to others
  • Prompt patterns that work in Claude.ai work in the API
  • The quality and capabilities are consistent across all products

The difference is the interface and the tools available:

The Decision Framework

Here's the cheat sheet. When you think "I want to ___", reach for the right tool:

Which Claude Product Should I Use?

"I want to brainstorm / write / research" → Claude.ai

"I want to build an interactive prototype" → Claude.ai (Artifacts)

"I want to edit code in my project" → Claude Code (CLI or IDE)

"I want to automate a dev workflow" → Claude Code with --print

"I want Claude inside my app" → Claude API

"I want to build an autonomous agent" → Claude Agent SDK

"I want to do something quick without leaving my editor" → IDE extension

Common Mistakes (and What to Do Instead)

Mistake 1: Using Claude.ai for coding tasks

You copy code from your editor into Claude.ai, describe what you want, get a response, then manually paste the changes back into your file.

Slow. Error-prone. You lose context every time.

Use Claude Code. It reads your files directly, makes edits in place, and understands your project structure. No copy-paste needed.

Mistake 2: Using the API for one-off questions

If you're writing a Python script just to ask Claude a single question, stop. Open claude.ai or type your question into Claude Code. The API is for programmatic access, not casual conversation.

Mistake 3: Always using the most expensive model

Opus is amazing, but Sonnet handles 90% of everyday tasks just as well — faster and cheaper. Don't default to Opus out of habit. Match the model to the task.

Mistake 4: Ignoring Claude Code's project context

Claude Code automatically reads your CLAUDE.md files, understands your project structure, and maintains conversation context. If you're not using CLAUDE.md, you're making Claude start from scratch every time. We'll fix that in Chapter 5.

What people do
*Opens Claude.ai* "Here's my file auth.js: [pastes 200 lines]. I also have this file user-service.js: [pastes 150 lines]. Can you refactor the authentication logic to use JWT instead of sessions?"
Manual copy-paste, lost file structure, no ability to verify changes work.
What power users do
*In Claude Code* "Refactor the authentication in src/auth.js to use JWT instead of sessions. Update the user service to match. Run the tests after to make sure nothing broke."
Claude Code reads the files, makes edits, runs tests, reports results. All in one go.

MCP: The Integration Layer

There's one more piece that ties it all together: the Model Context Protocol (MCP). MCP is an open standard that lets Claude connect to external tools and services — GitHub, Slack, Jira, Google Drive, databases, and thousands more.

Think of MCP as giving Claude hands. Instead of just thinking about your question, Claude can reach out and interact with your actual tools.

MCP servers can be configured in both Claude Code and Claude.ai, and the ecosystem has grown to over 6,000 integrations. You don't need to worry about MCP internals right now — just know that when you see Claude creating GitHub issues, reading Slack messages, or querying a database, MCP is what makes that possible.

Beyond the Dev Stack: Cowork and Design

The five surfaces above are the developer-facing stack. Two newer products sit alongside them and are worth knowing about even if you won’t live in them daily.

Claude Cowork — launched January 2026. An agentic assistant for complex business tasks: calendar, email, docs, research, and Cowork-managed MCP connectors (Drive, Gmail, Calendar, and dozens more). It’s where Claude does end-to-end work instead of coding — the closest thing to a Claude-powered teammate.
Claude Design — launched April 17, 2026 as a research preview from Anthropic Labs. Generates design systems, prototypes, slide decks, and one-pagers from a prompt. Runs on Opus 4.7 vision and ships as a plugin inside Claude Cowork. Built for founders and PMs without a design background, but useful for any developer who needs a quick UI mock.

You don’t need either of these to be a Claude power user as a developer — but if a teammate asks “does Claude do X?” and X is a calendar workflow or a Figma-shaped task, this is your answer.

Mapping Your Current Usage

Take 5 minutes and audit your Claude usage. Open a notes app and answer these questions:

  1. What do you currently use Claude for? List your top 5 tasks.
  2. Which Claude product do you use for each? (Claude.ai? Claude Code? API?)
  3. Are you using the right product for each task? Check against the decision framework above.
  4. What are you NOT doing yet? Look at the list of capabilities for each product. What jumps out as something you could be using but aren't?

If you find even one task where you've been using the wrong tool, you've already found a win. The rest of this book will show you how to maximize each one.
