Chapter 5

CLAUDE.md & Memory Systems

Persistent instructions and memory — teach Claude your project once, benefit forever

In the last chapter, you wired Claude into your IDE so it's always within reach. But here's a frustrating pattern you've probably noticed: every time you start a new conversation, Claude acts like it's never seen your project before. It doesn't know your coding standards, your preferred test framework, or that weird build command your project requires.

That's about to change. CLAUDE.md is the single most underused power feature in Claude Code. It's a plain Markdown file that Claude reads at the start of every conversation — like an onboarding document for an AI teammate who just joined your project.

Combined with Claude's auto memory system, you can build a persistent knowledge layer that makes Claude dramatically more useful over time. Let's set it up.

What Is CLAUDE.md?

CLAUDE.md is a special file that Claude Code reads automatically before it sees your first message in any session. Think of it as a project briefing — it tells Claude:

  • What this project is and how it's structured
  • What coding conventions to follow
  • What tools and commands to use (and avoid)
  • What patterns matter to your team

It's plain Markdown. No special syntax. No configuration format. Just instructions written in natural language that Claude ingests as context.

Without CLAUDE.md

Every conversation starts cold. You repeat yourself constantly:

"Use Kotlin, not Java. We use Hilt for DI. Tests go in the test/ directory. Our build command is ./gradlew assembleDebug..."

With CLAUDE.md

Claude already knows all of this. You jump straight to the actual work:

"Add a new repository class for user preferences with proper DI setup and tests."

The CLAUDE.md Hierarchy

Here's where it gets interesting. Claude doesn't just read one file — it reads a hierarchy of CLAUDE.md files, merging them from broadest to most specific scope:

~/.claude/CLAUDE.md

User-level — your personal preferences across ALL projects

→

Project Root CLAUDE.md

Project-level — shared with team via git

→

Subdirectory CLAUDE.md

Module-level — loaded on demand for specific areas

Each level adds context. They don't replace each other — they merge. So your personal preferences combine with project conventions, and subdirectory instructions add specifics when Claude works in that area.

Monorepo Tip: Parent directories above your project root can also have CLAUDE.md files. In a monorepo, you might have conventions at the repo root that apply to all packages, plus package-specific CLAUDE.md files in each subdirectory.

What Goes Where?

File Location Scope What to Put Here Git?
~/.claude/CLAUDE.md All projects Language preferences, formatting style, communication tone No
./CLAUDE.md (project root) This project Tech stack, build commands, architecture, conventions Yes
./src/api/CLAUDE.md Subdirectory API-specific patterns, endpoint conventions, auth requirements Yes

Writing an Effective CLAUDE.md

The quickest way to start is the /init command. Run it in Claude Code, and it generates a starter CLAUDE.md based on your project structure. But the auto-generated version is just a starting point — the real power comes from what you add.

Quick Start

Run /init inside Claude Code to auto-generate a CLAUDE.md from your project. Then refine it with the patterns below.

The Golden Rule

For every line in your CLAUDE.md, ask: "Would Claude make a mistake without this?" If Claude already does something correctly on its own, that instruction is noise. Only include things that prevent errors or enforce conventions Claude wouldn't know about.

The ~150 instruction budget: Research shows that compliance starts to drop off after roughly 150–200 distinct instructions. Keep your CLAUDE.md focused and concise. Quality over quantity — every instruction should earn its place.

What to Include

A well-structured CLAUDE.md: 
# MyApp

Android app using Kotlin, Jetpack Compose, and Hilt.
Min SDK 26, target SDK 35.

## Build & Test
- Build: `./gradlew assembleDebug`
- Test: `./gradlew testDebugUnitTest`
- Lint: `./gradlew lintDebug`
- Always run tests before suggesting a PR

## Architecture
- MVVM with Repository pattern
- ViewModels in `feature/*/viewmodel/`
- Repositories in `data/repository/`
- Use cases in `domain/usecase/`

## Code Conventions
- Kotlin, never Java for new code
- Compose for all new UI (no XML layouts)
- Use `StateFlow` not `LiveData`
- Hilt for DI — annotate with `@HiltViewModel`, `@Inject`
- Coroutines for async — no RxJava

## Testing
- Unit tests: JUnit 5 + MockK
- UI tests: Compose testing library
- Name pattern: `should_expectedBehavior_when_condition`

## Git
- Branch format: `feature/JIRA-123-short-description`
- Commit messages: conventional commits (feat:, fix:, etc.)
- Always create a new branch, never commit to main

What NOT to Include

Anti-patterns that waste your instruction budget:

  • Obvious language rules: "Use proper indentation" — Claude already does this
  • Entire file trees: Don't paste your whole directory structure — Claude can explore it
  • Long explanations: "We use Kotlin because in 2019 Google announced..." — Claude doesn't need the history
  • Contradictory rules: "Always add comments" + "Keep code clean and minimal" — pick one
  • Rules you don't enforce: If nobody on the team actually follows it, don't teach it to Claude

Power Tip: The Correction Capture Pattern

When Claude makes a mistake during a session, don't just correct it — add that correction to your CLAUDE.md. If Claude used LiveData when you wanted StateFlow, add "Use StateFlow not LiveData" to your conventions. Over a few sessions, your CLAUDE.md evolves into a precisely-tuned instruction set that prevents every mistake Claude has ever made on your project.

The .claude/ Directory Deep Dive

CLAUDE.md is just the tip of the iceberg. Your project also has a .claude/ directory that holds configuration, commands, and settings. Let's explore what lives there.

📁
.claude/settings.json
Project-level settings shared with your team via git. Hooks, permissions, tool configurations.
🔒
.claude/settings.local.json
Your personal project settings, automatically git-ignored. API keys, local overrides, experiments.
⚡
.claude/commands/
Custom slash commands (skills) for this project. Reusable prompt templates invoked with /command-name.

The Settings Hierarchy

Just like CLAUDE.md files merge, settings have a strict precedence chain. When the same setting exists at multiple levels, the most specific one wins:

→

→

.claude/settings.local.json

→

.claude/settings.json

→

Check what's active: Run /status inside Claude Code to see which settings sources are loaded and where each setting comes from. Super handy for debugging why Claude is (or isn't) following a particular configuration.

What to Commit vs. Keep Local

Commit to Git Keep Local

.claude/commands/ (shared skills) ~/.claude/commands/ (personal skills)

The Auto Memory System

CLAUDE.md is what you write for Claude. Auto memory is what Claude writes for itself.

As you work with Claude Code across sessions, it automatically saves notes about things it discovers: build commands that work, debugging insights, architecture patterns, code style preferences, and workflow habits. These notes live in ~/.claude/projects/<project-hash>/memory/ as plain Markdown files.

Session 1

You tell Claude: "Our CI uses make test-integration, not npm test."

Claude saves this to auto memory.

Session 5

Claude already knows the right test command. You never mention it again.

It just works.

How Auto Memory Works

Auto memory is organized around a MEMORY.md index file, with individual memory files linked from it. Each memory has a type:

User
Your role, preferences, and expertise level
Feedback
Corrections and confirmed approaches
Project
Ongoing work, goals, decisions, deadlines
Reference
Pointers to external resources and systems

Each memory is a separate Markdown file with frontmatter (name, description, type) and a short body. The MEMORY.md file acts as an index — one-line pointers to each memory file, kept under 200 lines.

Example: What a memory file looks like 
---
name: Build system quirk
description: Integration tests require Docker running
type: project
---

Integration tests (`make test-integration`) require Docker
daemon to be running. The CI handles this automatically,
but locally you need `docker compose up -d` first.

**Why:** Tests spin up a Postgres container for real DB access.
**How to apply:** Before suggesting test commands, check
if the user is running locally vs CI.

Browsing and Editing Memory

Memory Commands

  • /memory — Browse and manage Claude's auto memory files
  • Ask Claude to "remember that..." — saves a new memory immediately
  • Ask Claude to "forget about..." — finds and removes the relevant memory

Everything in auto memory is plain Markdown — you can read, edit, or delete any of it. Claude doesn't hide anything. If a memory is wrong or outdated, just delete the file or ask Claude to forget it.

Memory has a size limit: The recommended maximum for any single memory file is about 40,000 characters. Files that exceed this get flagged, and Claude may not read the full content. Keep individual memories focused and concise.

Auto Dream: Memory Consolidation

After 20+ sessions, auto memory can get messy. Contradictory entries pile up. Relative dates like "yesterday" lose meaning. Three different sessions might note the same thing.

Auto Dream is Claude's solution — think of it as a periodic cleanup routine. When it runs, it:

  • Converts relative dates to absolute dates ("yesterday" → "2026-03-15")
  • Deletes contradicted facts (switched from Express to Fastify? removes the old entry)
  • Merges overlapping entries (three sessions noted the same build quirk → one clean entry)
  • Removes stale debugging notes that reference files that no longer exist
  • Reorganizes everything into clean, indexed topic files

Auto Dream runs roughly every 24 hours, but only after you've accumulated at least 5 sessions of auto memory data. It's automatic — you don't need to trigger it.

Check your dream status: Run /memory and look for "Auto-dream: on" in the output. If it's enabled, Claude is already consolidating your memory files between sessions.

User-Level Instructions

Your ~/.claude/CLAUDE.md file applies to every project you work on. This is where you put personal preferences that don't belong in any specific project:

Example ~/.claude/CLAUDE.md: 
# Personal Preferences

- I prefer concise responses — skip the preamble
- When showing code, include the file path as a comment
- I'm an Android developer — use Kotlin examples by default
- For new projects, suggest Gradle with Kotlin DSL (build.gradle.kts)
- Don't explain basic concepts I already know (coroutines, DI, MVVM)
- When I say "test it", run the relevant unit tests, not the full suite

These instructions travel with you. Switch to a different project? Your personal preferences still apply. The project's CLAUDE.md adds project-specific context on top.

CLAUDE.md in Practice: Real Patterns

Pattern 1: The Forbidden List

Prevent Claude from making specific mistakes you've seen before:

## Forbidden Patterns
- NEVER use `!!` (non-null assertion) in Kotlin — use safe calls or requireNotNull
- NEVER create God objects — split into focused classes under 200 lines
- NEVER use Thread.sleep() in tests — use coroutines test dispatcher
- Do NOT add println for debugging — use Timber.d()

Pattern 2: The Decision Record

Explain why things are the way they are, so Claude doesn't "fix" intentional decisions:

## Architecture Decisions
- We use Room instead of SQLDelight (team familiarity, not a TODO to change)
- API responses are mapped to domain models at the repository layer
  (don't map in ViewModels or Composables)
- We keep the navigation graph in a single file (nav_graph.kt)
  even though it's large — splitting caused merge conflicts

Pattern 3: The Context Provider

Give Claude the context it can't get from code alone:

## Project Context
- This is a fintech app — security is critical, never log sensitive data
- We support offline mode — all data operations must work without network
- The app is used by elderly users — accessibility is non-negotiable
- We're migrating from Fragments to Compose — new features use Compose,
  but don't refactor existing Fragments unless specifically asked

Pattern 4: Splitting Large Instructions with Imports

If your CLAUDE.md grows too large, split it using the .claude/rules/ directory. Create individual Markdown files for different topics, and Claude will discover them when it works in relevant areas:

# Project root CLAUDE.md — kept focused and short

# MyApp
Android fintech app. See .claude/rules/ for detailed conventions.

## Quick Reference
- Build: ./gradlew assembleDebug
- Test: ./gradlew testDebugUnitTest
- Lint: ./gradlew lintDebug

Then in .claude/rules/testing.md:

# Testing Conventions
- JUnit 5 + MockK for unit tests
- Compose testing library for UI tests
- Test naming: should_expectedBehavior_when_condition
...

Power Tip: Keep It Under 200 Lines

The official recommendation is to target under 200 lines per CLAUDE.md file. Longer files consume more of your context window and reduce Claude's adherence to individual instructions. If your CLAUDE.md is growing past 200 lines, it's time to split into .claude/rules/ files.

The Context Window Connection

Everything in CLAUDE.md (and loaded memory) counts toward your context budget. This is a real tradeoff:

More Instructions

  • Claude knows more about your project
  • Fewer mistakes and corrections
  • More consistent output

Less Context Available

  • Less room for your actual conversation
  • Conversations hit limits sooner
  • Compliance drops if instructions are too long

The sweet spot is a focused CLAUDE.md that prevents real mistakes without bloating your context. Remember the golden rule: if Claude would do it correctly without the instruction, remove it.

Maintaining CLAUDE.md Over Time

A CLAUDE.md file isn't a "set it and forget it" document. Your project evolves, and your instructions should evolve with it.

  • Review your CLAUDE.md monthly — remove rules for patterns that are no longer relevant
  • After a major refactor, update architecture descriptions and conventions
  • When a team member joins, have them read the CLAUDE.md — if it confuses a human, it'll confuse Claude
  • Track what corrections you repeat most often — those should become CLAUDE.md rules
  • Remove instructions for things Claude now handles correctly by default (the model improves over time)
Stale instructions cause real problems. If your CLAUDE.md says "use LiveData" but your project migrated to StateFlow six months ago, Claude will actively fight you by suggesting the wrong approach. Outdated instructions are worse than no instructions.

Common CLAUDE.md Mistakes

Mistake: The Novel

A 500-line CLAUDE.md with your entire project's history, every decision ever made, and a full style guide. Claude can't prioritize when everything is "important."

Mistake: The Wishlist

"Write perfect code. Always handle errors. Follow best practices." — These are too vague to be actionable. Be specific: which errors? Which best practices?

Mistake: The Contradiction

"Keep functions under 20 lines" + "Always add comprehensive error handling with detailed logging" = Claude can't satisfy both. Prioritize or be explicit about when each applies.

Try It Yourself: Write Your CLAUDE.md

  1. Open your current project in Claude Code
  2. Run /init to generate a starter CLAUDE.md
  3. Read the generated file and remove anything obvious (things Claude would do correctly anyway)
  4. Add your project's specific conventions:
    • Build and test commands
    • Architecture patterns and where key files live
    • 3–5 "forbidden patterns" — mistakes you've seen Claude make
    • 1–2 architecture decisions with the why
  5. Keep it under 200 lines
  6. Start a new Claude Code session and test it — does Claude follow your conventions without being reminded?

Bonus: Create a ~/.claude/CLAUDE.md with your personal preferences. Notice how both files merge seamlessly in your next session.
