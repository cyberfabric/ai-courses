Chapter 9

Claude for Documentation

READMEs, changelogs, API docs, architecture decisions — the docs you never write

In the last chapter, you built a code review workflow that catches bugs before they ship. Now let's tackle the other thing developers perpetually promise to do "later": documentation.

Here's the uncomfortable truth: your code is useless if nobody can understand it. That brilliant architecture? Meaningless if the next developer can't figure out how to run the project. Those API endpoints? Dead to anyone without tribal knowledge of what parameters to pass.

Documentation solves this. But nobody writes it because it's tedious, it goes stale immediately, and it feels like work that doesn't ship features. Claude changes the equation: what used to take an hour now takes two minutes. Let's make docs happen.

Generating READMEs

A README is your project's front door. Most projects either have no README or a stale one from three years ago. Claude can generate a comprehensive one in seconds.

The prompt:

Generate a README.md for this project. Read the project structure, key config files, and CLAUDE.md to understand what this project does. Include: project description, prerequisites, setup instructions, how to run, how to test, architecture overview, and contributing guidelines.

Claude reads your project files, understands the tech stack, and generates a complete README with accurate commands, directory structure, and descriptions — not generic boilerplate.

Your Current README

# MyApp
TODO: add docs

(Committed 2 years ago. Never updated.)

Claude-Generated README

Project description, prerequisites, setup steps, run commands, test commands, architecture overview, environment variables, contributing guide — all accurate to the current codebase.

Key insight: Claude generates better docs when it can read your actual code, not just a description. Always run this in Claude Code (where it has file access), not in Claude.ai with a vague description.

The README Prompt That Works

Good: Specific and scoped

Generate a README.md for this project. Read the following
files first for context:
- package.json (or build.gradle.kts)
- CLAUDE.md
- src/ directory structure
- Any existing docs/

Include these sections:
1. One-paragraph project description
2. Prerequisites (with version numbers)
3. Setup: clone, install, configure
4. Running locally (dev mode)
5. Running tests
6. Project structure (key directories only)
7. Environment variables (from .env.example)
8. Deployment notes
9. Contributing guidelines

Use real commands from the project, not generic examples.

Bad: Vague and generic

Write a README for my project.

Claude has no idea what to emphasize, what commands to include, or who the audience is. You'll get a generic template.

API Documentation

Documenting API endpoints is one of Claude's strongest use cases. It can read your route definitions, controllers, and models, then generate accurate documentation.

The prompt:

Read all the API route files in src/api/routes/. For each endpoint, document: HTTP method, path, description, request parameters, request body schema, response schema, example request, example response, and possible error codes. Output as Markdown.

## POST /api/users

Create a new user account.

**Request Body:**
| Field    | Type   | Required | Description          |
|----------|--------|----------|----------------------|
| email    | string | Yes      | User's email address |
| name     | string | Yes      | Display name         |
| password | string | Yes      | Min 8 characters     |

**Response (201):**
```json
{
  "id": "usr_abc123",
  "email": "user@example.com",
  "name": "Jane Doe",
  "createdAt": "2026-04-02T10:30:00Z"
}
```

**Errors:**
- `400` — Invalid request body
- `409` — Email already registered

The key is letting Claude read the actual code. It extracts the real fields, types, and validation rules — not guesses.

Architecture Decision Records (ADRs)

ADRs document why you made architectural decisions. They're invaluable six months later when someone asks "why are we using Redis instead of Memcached?" and nobody remembers.

Generating an ADR:

We just decided to use Room instead of SQLDelight for our local database. Write an ADR documenting this decision. We chose Room because: the team has more experience with it, it integrates better with our existing Hilt setup, and the migration from our current SQLite helper is simpler. The tradeoff is that SQLDelight has better multiplatform support if we ever go KMP.

# ADR-007: Use Room for Local Database

## Status
Accepted (2026-04-02)

## Context
We need a structured local database solution for
offline data caching. The candidates were Room and
SQLDelight.

## Decision
We will use Room as our local database abstraction.

## Reasons
- Team has existing Room experience (3 of 4 devs)
- Native Hilt integration via @Database + @Dao
- Simpler migration path from current SQLiteOpenHelper
- Better Android Studio tooling (schema inspector)

## Tradeoffs
- SQLDelight offers better KMP support if we pursue
  multiplatform in the future
- SQLDelight's type-safe SQL is arguably more flexible
  for complex queries

## Consequences
- All new data access goes through Room DAOs
- Existing SQLiteOpenHelper code migrates incrementally
- If KMP becomes a priority, revisit this decision

Notice that you don't need to write the ADR yourself. Just tell Claude the decision and the reasoning — it handles the structure and format.

Power Tip: Retroactive ADRs

Most projects have dozens of undocumented decisions. Ask Claude: "Read the codebase and identify 5 architectural decisions that should have ADRs. For each one, infer the likely reasoning from the code and dependencies, and draft an ADR." Claude will analyze your tech stack, dependency choices, and patterns, then generate ADRs for decisions that were made implicitly but never documented. In 5 minutes, you'll have documentation that would have taken hours to write.

Changelog Generation

Nobody wants to write changelogs manually. But users and stakeholders need to know what changed. Claude can generate human-readable changelogs from your git history:

Changelog Prompts

  • Generate a changelog from git log between v2.3.0 and HEAD. Group by: Features, Bug Fixes, Breaking Changes.
  • Summarize the last 2 weeks of commits into release notes for non-technical stakeholders.
  • Write a CHANGELOG.md entry for this release based on all commits since the last tag.

Code Comments: When and How

Controversial opinion: most code comments are noise. Claude can help you add the right comments — the ones that explain why, not what.

Useless Comments

// Get the user
val user = repo.getUser(id)

// Check if null
if (user == null) {
    // Return error
    return Result.Error("Not found")
}

Useful Comments

// Fetch from local cache first — network call
// happens only if cache miss (see UserRepo L.45)
val user = repo.getUser(id)

if (user == null) {
    // Don't throw — caller expects Result type
    // per error handling convention (see CLAUDE.md)
    return Result.Error("Not found")
}

Good prompt for adding comments:

Review this file and add comments ONLY where:
1. The "why" isn't obvious from the code
2. There's a non-obvious side effect
3. A performance or security decision needs explanation
4. A workaround references a bug or limitation

Do NOT add comments that just restate what the code does.
Less is more — every comment should earn its place.

Onboarding Guides

New team member starting Monday? Instead of spending half a day walking them through the codebase, let Claude generate an onboarding guide:

The prompt:

Generate a "Getting Started" guide for a new developer joining this project. Read the project structure, CLAUDE.md, build files, and test configuration. Include: how to set up the development environment, how to run the app, how to run tests, key architectural concepts they need to understand, where the important code lives, and common gotchas.

Claude generates a step-by-step guide with real paths, real commands, and explanations tailored to your actual project — not a generic template.

Commit this to your repo as docs/onboarding.md. New developers can follow it independently, and you update it with Claude whenever the project changes.

Generating Diagrams

Sometimes a picture is worth a thousand lines of documentation. Claude excels at generating Mermaid diagrams — text-based diagrams that render in GitHub, GitLab, and most Markdown viewers.

The prompt:

Read the auth flow in src/auth/ and generate a Mermaid sequence diagram showing the complete login process from user tap to token storage.

```mermaid
sequenceDiagram
    participant U as User
    participant VM as LoginViewModel
    participant R as AuthRepository
    participant API as AuthAPI
    participant DB as TokenStore

    U->>VM: tap Login
    VM->>R: login(email, password)
    R->>API: POST /auth/login
    API-->>R: { accessToken, refreshToken }
    R->>DB: store(tokens)
    R-->>VM: Result.Success
    VM-->>U: navigate to Home
```

Paste this into any Markdown file, and it renders as a clean sequence diagram. No Visio, no draw.io, no exporting PNGs — just text that's version-controlled alongside your code.

Context efficiency: Mermaid diagrams are roughly 3–6x more token-efficient than equivalent prose descriptions. A sequence diagram that takes 400 tokens in Mermaid syntax would take 2,000+ tokens as a written explanation. Great for CLAUDE.md architecture sections too.

Diagram Types That Work Well

Flowchart
Decision logic, feature flags, branching workflows
Sequence
API calls, auth flows, multi-service interactions
Class
Data models, inheritance, dependency relationships
State
UI states, order lifecycles, connection status
Claude.ai Artifacts: In Claude.ai, Mermaid diagrams render live in Artifacts. Ask Claude to "create a Mermaid diagram of..." and you'll get an interactive, editable visualization. Great for iterating on architecture diagrams before committing them to code.

Keeping Docs in Sync

The biggest documentation problem isn't writing docs — it's keeping them updated. Here's how to use skills and hooks (from Chapter 6) to keep docs fresh:

Strategy 1: The Update Skill

File: .claude/commands/update-docs.md

Check if any documentation is outdated.

1. Read README.md and compare against current project state
2. Read docs/api.md and compare against current route files
3. Read docs/architecture.md and compare against current code

For each outdated section:
- Explain what changed
- Update the documentation to match current reality
- Show what you changed

Do NOT rewrite sections that are still accurate.

Run /update-docs periodically (before a release, after a sprint, or when onboarding someone) to catch stale docs.

Strategy 2: The Documentation Hook

Want docs to update automatically? Use a PostToolUse hook that reminds Claude to check related docs after editing code:

// In your CLAUDE.md:
## Documentation Rules
- When modifying API routes, update docs/api.md
- When changing the database schema, update docs/data-model.md
- When adding a new feature, add a section to docs/features.md

This isn't automatic doc generation — it's a convention that Claude follows because it's in CLAUDE.md. Every time Claude edits an API route, it remembers to update the API docs too.

Documentation Anti-Patterns

Anti-pattern: Over-documentation

Asking Claude to "add comprehensive documentation to every function" creates noise. Most functions don't need doc comments. A well-named getUserById(id: String): User? is self-documenting.

Anti-pattern: The Boilerplate Factory

Generating generic class-level comments like "This class handles user operations" adds zero value. If the comment doesn't tell you something the class name doesn't already say, delete it.

Anti-pattern: Generated and Forgotten

Generating beautiful docs and never updating them is worse than no docs. Stale docs actively mislead. Use the /update-docs skill pattern to keep them current.

Building a Documentation Skill

Here's a comprehensive /generate-docs skill you can adapt for your project:

File: .claude/commands/generate-docs.md 
Generate or update project documentation based on the
current codebase state.

## What to Generate
1. **README.md** — if missing or outdated
2. **docs/architecture.md** — high-level architecture with
   Mermaid diagrams
3. **docs/api.md** — API endpoint documentation (if applicable)
4. **CHANGELOG.md** — recent changes since last documented release

## Rules
- Read the actual code, don't guess
- Use Mermaid for all diagrams
- Include real commands from the project
- Keep it concise — devs don't read novels
- Don't document obvious things
- Update existing docs, don't regenerate from scratch unless
  they're badly outdated

Target audience: $ARGUMENTS
(default: "a mid-level developer joining the team")

Try It Yourself: Document Your Project

  1. README check: Open your current project. Does it have a README? Is it accurate? Ask Claude to generate or update it.
  2. Architecture diagram: Ask Claude to read your project structure and generate a Mermaid diagram showing the key components and their relationships. Paste it into a Markdown file.
  3. Retroactive ADR: Ask Claude to identify one architectural decision in your codebase that should be documented. Have it draft an ADR explaining what was decided and why.
  4. Changelog: Run git log --oneline for the last 20 commits. Ask Claude to generate a human-readable changelog grouped by category.
  5. Bonus: Create the /generate-docs skill from this chapter and run it on your project.
