Plugins & the Marketplace Ecosystem

Memory Map
💡

The Big Idea

A plugin is the packaging layer of Claude Code — one installable bundle for skills, hooks, subagents, MCP servers, LSP servers, monitors, and commands. Marketplaces are the discovery layer. Together they turn "copy these files" into "/plugin install".

📦 What Goes In a Plugin?

ComponentWhere in the plugin
Manifest (required).claude-plugin/plugin.json
Skillsskills/<name>/SKILL.md
Subagentsagents/<name>.md
Hookshooks/hooks.json
MCP servers.mcp.json
LSP servers.lsp.json
Background monitorsmonitors/monitors.json

⚠️ Only plugin.json goes inside .claude-plugin/ — everything else lives at the plugin root.

🏪 Three Kinds of Marketplaces

🅰️
Official
claude-plugins-official. Auto-available on launch. Browse at claude.com/plugins or in /plugin.
🧰
Community
Add manually with /plugin marketplace add. Examples: buildwithclaude.com, claudemarketplaces.com, awesome-claude-plugins.
🔒
Private (Cowork)
Enterprise-only since Feb 24, 2026. Per-user provisioning, auto-install, restricted visibility. Customize menu in admin settings.

⚡ The Five Commands You'll Use

/plugin
Plugin manager UI: Discover / Installed / Marketplaces / Errors tabs.
/plugin install <name>@<marketplace>
Install a plugin. Example: /plugin install code-quality@claude-plugins-official.
/plugin marketplace add <source>
Add a marketplace by git URL, GitHub owner/repo, or local path.
/plugin marketplace refresh
Pull the latest manifests — do this after a teammate publishes.
/reload-plugins
Hot-reload plugin files during development. No restart needed.
--plugin-dir ./path
CLI flag — load a local plugin directory without publishing. Pass it multiple times to load several at once.

🛠️ Authoring Flow (5 Steps)

  1. Make the manifest

    mkdir -p my-plugin/.claude-plugin + plugin.json with name, description, version.

  2. Add components

    Skills in skills/, subagents in agents/, hooks in hooks/hooks.json, MCP in .mcp.json. Plugin root, not inside .claude-plugin/.

  3. Test locally

    claude --plugin-dir ./my-plugin. Edit + /reload-plugins to iterate without restart.

  4. Push to a git repo

    That repo IS your marketplace. Public or private; works the same way.

  5. Onboard teammates

    They run /plugin marketplace add <repo> + /plugin install <name>@<marketplace>. Done.

🛡️ Security Cheat Sheet

⚠️
Untrusted = arbitrary code
Plugins can ship hooks that run anything. Vet community plugins like npm dependencies. Official first.
🚫
Plugin subagent restrictions
Plugin subagents can't use hooks, mcpServers, or permissionMode. Copy locally if you need them.
📌
Pin versions
No version field = every commit is a new release. Pin in shared environments to prevent surprise drift.
🔍
Watch the Errors tab
"Claude is broken" is often "one plugin is broken." Check /plugin → Errors before deeper debugging.

🧪 Quick Quiz

What's the only file that belongs inside .claude-plugin/?

plugin.json. Everything else (skills/, agents/, hooks/, .mcp.json, monitors/, etc.) lives at the plugin root.

What's the syntax for installing a plugin from the official marketplace?

/plugin install <name>@claude-plugins-official. The official marketplace is auto-available; community marketplaces require /plugin marketplace add first.

How do you test a plugin without publishing it?

Use the --plugin-dir ./path CLI flag. Pair with /reload-plugins for hot reload after edits. The flag accepts multiple values to load several plugins at once.

Why are plugin skill names namespaced (e.g., /my-plugin:hello)?

To prevent conflicts when multiple plugins ship skills with the same name. Standalone skills in .claude/ get short names; plugin skills always get the namespace prefix.

What launched on February 24, 2026?

Private enterprise plugin marketplaces in Cowork, plus 10 department-specific plugins, 12 new MCP connectors, and the Customize admin menu. The biggest enterprise plugin update of the year.

What's the conversion path from .claude/ to a plugin?

Make the manifest, move commands/ + agents/ + skills/ into the plugin root, copy hooks from settings.json into hooks/hooks.json, test with --plugin-dir, push to git.

🎯 Key Takeaways

  • A plugin bundles skills + hooks + subagents + MCP servers + LSP + monitors + commands into one installable unit.
  • Three marketplace flavors: official (auto), community (add manually), private enterprise (Cowork-managed).
  • Standalone (.claude/) for personal/experimental; plugins for sharing, versioning, and cross-project reuse.
  • Manifest at .claude-plugin/plugin.json; everything else at the plugin root. Don't nest.
  • Skill namespacing prevents conflicts: standalone /deploy vs plugin /my-pack:deploy.
  • Develop locally with --plugin-dir + /reload-plugins — tightest possible iteration loop.
  • Publish by pushing to a git repo — the repo IS the marketplace. Pin version for stability.
  • Cowork's Feb 24, 2026 release added private marketplaces, 10 dept plugins, 12 MCP connectors, and the Customize admin menu.
  • Security: untrusted plugins can run arbitrary code. Vet community plugins like npm deps. Watch the Errors tab.
  • Plugins are the delivery mechanism; skills/hooks/subagents are the payload. The conversion path is mechanical.
