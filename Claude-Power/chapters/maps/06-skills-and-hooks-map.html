Chapter 6

Custom Skills & Hooks

Memory Map
💡

The Big Idea

CLAUDE.md teaches Claude what to know. Skills teach Claude what to do on demand. Hooks make things happen automatically. Together, they turn Claude Code into a customized development machine.

🎯 The Core Analogy

📖
CLAUDE.md
The employee handbook — what Claude should know about your project
📋
Skills
Standard operating procedures — reusable steps you invoke with a command
⚙️
Hooks
Automatic quality checks — run without anyone asking, every single time

🔧 Part 1: Skills (Custom Slash Commands)

Write a .md file
→
Put it in .claude/commands/ or .claude/skills/
→
Type /command-name
→
Claude follows your template

📁 Where Skills Live

Location Scope Best For
.claude/commands/ Project (shared via git) Team workflows
.claude/skills/*/SKILL.md Project (shared via git) Rich skills with templates & supporting files
~/.claude/commands/ All your projects (personal) Your personal toolkit

📝 Two Formats

📄
Simple Command
A plain .md file in .claude/commands/. Filename = command name. Just write your prompt instructions.
📦
Rich Skill (SKILL.md)
A folder in .claude/skills/ with a SKILL.md file. Supports frontmatter, supporting files, and advanced config.

⚙️ SKILL.md Frontmatter Fields

description
What the skill does — helps Claude decide when to auto-load it
argument-hint
Hints at what argument to pass (e.g., "file path or class name")
allowed-tools
Which tools Claude can use during this skill (e.g., Bash, Read, Edit)
paths
Only activate when working with matching file patterns (e.g., "src/**/*.kt")
disable-model-invocation
Set to true — only run when explicitly called, never auto-loaded
$ARGUMENTS
Variable replaced with whatever the user types after the command name

🛠️ Example Skills

📝
/daily-standup
Summarize last 24h of git activity into standup format
🔍
/review-pr
Review a PR for bugs, security, and style violations
♻️
/refactor
Refactor code while preserving behavior, then run tests
📖
/explain
Explain a file or function in plain English
🚀
/release-notes
Generate release notes from git history between tags
🧪
/generate-tests
Generate unit tests for a file with edge cases and mocks

✍️ Writing Good Skills

✅
Do: Be Directive
Use imperative verbs: "Read the file", "Generate tests", "Run lint." Be specific about format, tools, and output.
❌
Don't: Be Conversational
"Hey Claude, could you maybe..." is vague. Skills are instructions, not conversations.

🪝 Part 2: Hooks (Automated Triggers)

🎯 Event Fires
(Claude uses a tool)
→
⚡ Hook Runs
(your shell command)
→
✨ Result
(format, block, notify)

📡 Hook Events (~30 in total, grouped by cadence)

Event Cadence Can Block? Use For
SessionStart / SessionEnd Per session ❌ No Auto-load context; persist transcripts
UserPromptSubmit Per turn ✅ Yes Inject hidden context, redact secrets
UserPromptExpansion Per turn ✅ Yes Modify slash command after expansion
Stop Per turn ✅ Yes Final checks before Claude is allowed to stop
PreToolUse Per tool call ✅ Yes (exit 2) Validate, block dangerous ops
PostToolUse Per tool call ✅ Yes Auto-format, lint, log
SubagentStop Per tool call ✅ Yes Validate or filter subagent output
PreCompact Async ✅ Yes Save state right before context is compacted
Notification Async ❌ No Slack alerts, desktop notify

Most teams wire up only 2–3 of these. Full reference: code.claude.com/docs/en/hooks.

🔑 Exit Codes (PreToolUse)

0️⃣
Exit 0
Allow the action — hook ran, proceed normally
2️⃣
Exit 2
Block the action — Claude is told and tries an alternative
❓
Other Codes
Hook error — logged, but action still proceeds

🎯 The Matcher Filter

Bash
Shell commands Claude runs
Edit|Write
Any file modification
Read
File reads (useful for auditing)
.*
Every single tool — use carefully!

🔨 Practical Hook Recipes

  1. Auto-Format on Edit

    PostToolUse + Edit|Write matcher → run Prettier on the edited file. Add || true so it doesn't fail on unsupported types.

  2. Block Dangerous Commands

    PreToolUse + Bash matcher → grep for rm -rf, drop table, force push. Exit 2 to block.

  3. Run Lint After Changes

    PostToolUse + Edit|Write matcher → run linter only on matching file extensions (e.g., *.kt files).

  4. Desktop Notification

    Notification event + .* matcher → notify-send (Linux) or osascript (macOS) with the message.

🧑‍🤝‍🧑 Beyond Skills and Hooks: Subagents

🔎
Explore
Built-in. Haiku-powered, read-only codebase search. Keeps exploration noise out of your main context.
📋
Plan
Built-in. Read-only researcher used by plan mode to design changes before doing them.
🛠️
general-purpose
Built-in. Full tool access for tasks needing both exploration and modification.
📁
.claude/agents/
Drop your own Markdown files here. YAML frontmatter + system prompt body — like a SKILL.md, but for an isolated worker.

Mental model: Skills run a procedure on demand. Hooks fire on events. Subagents run in their own conversation and return only a summary — great for fan-out work without flooding main context.

🔗 Skills + Hooks Together

The Pattern

Skills handle big, intentional workflows. Hooks handle small, automatic quality checks. Together, quality is built in, not bolted on.

🐛 Debugging Checklist

👻
Skill Not Showing?
Check filename & directory. Must be in .claude/commands/ or .claude/skills/*/SKILL.md
🔇
Hook Not Firing?
Run /status to verify hooks loaded. Check matcher regex matches tool name exactly.
💥
Hook Not Working?
Test the command in terminal first. $CLAUDE_TOOL_INPUT_* vars only exist during hook execution.
🐌
Hook Too Slow?
Hooks run synchronously — keep under 2s. Add 2>/dev/null and scope to specific file types.

🧠 Quick Quiz

What's the difference between a skill and a hook?

A skill is a reusable slash command you invoke manually (e.g., /review-pr). A hook is an automated shell command that runs automatically in response to Claude Code events (e.g., auto-format after every file edit).

Which exit code blocks a PreToolUse hook?

Exit code 2. Exit 0 allows the action, and any other exit code is treated as a hook error (action still proceeds).

Where do you put a skill to share it with your team?

In .claude/commands/ or .claude/skills/*/SKILL.md — these are project directories committed to git. Personal skills go in ~/.claude/commands/.

What variable captures the argument a user passes to a skill?

$ARGUMENTS — it gets replaced with everything the user types after the slash command name.

How do you create namespaced skills like /android:lint?

Use subdirectories: commands/android/lint.md becomes /android:lint.

📌 Key Takeaways

  • Skills = reusable slash commands from .md files. Filename → command name. $ARGUMENTS for user input.
  • Two formats: simple (.claude/commands/*.md) and rich (.claude/skills/*/SKILL.md with frontmatter).
  • Hooks = automated shell commands triggered by Claude Code events. There are ~30 events grouped by cadence (per session / turn / tool call / async); most teams use 2–3.
  • Subagents are the third primitive (alongside skills and hooks): isolated workers like Explore / Plan / general-purpose, plus your own in .claude/agents/.
  • PreToolUse exit 2 blocks the action. All other exit codes allow it.
  • Matcher is a regex filtering which tools trigger hooks (e.g., Edit|Write).
  • Skills + Hooks together = skills for intentional workflows, hooks for automatic quality checks.
  • Share skills by committing .claude/commands/ to git. Keep personal ones in ~/.claude/commands/.
  • Keep hooks fast — they run synchronously. Under 2 seconds is the target.
