The Big Idea

Claude doesn't skim, doesn't get tired, and is available at 2 AM. Combine AI thoroughness (mechanical bugs, security, conventions) with human judgment (architecture, design, business logic) for review that's better than either alone.

📈 Review Workflow Maturity

  1. ⚡ Quick /review

    Start here. Type /review before every push. Instant feedback, zero setup.

  2. 👀 Fresh Eyes Pre-Push

    Review your own code as a stranger would. Ask Claude to explain it back and check assumptions.

  3. 📝 Custom Review Skill

    Build /review-pr tailored to your project — conventions, tech stack, common pitfalls.

  4. 🔄 CI/CD Automation

    Every PR auto-reviewed via GitHub Actions + claude --print. Humans start with findings already posted.

  5. 👥 Team-Wide Adoption

    Shared review skills, project conventions in CLAUDE.md, triage process for findings.

🚫 Handling False Positives

Don't dismiss all feedback
If 3 of 10 findings are wrong, the other 7 might save you from a production bug.
Teach Claude what's intentional
Add architectural decisions to CLAUDE.md so Claude stops flagging known choices.
Tune your review prompt
If Claude keeps flagging a pattern, add "don't flag this" to the review skill.
Use severity levels
Categorize as Critical / Suggestion / Good so you can triage quickly.
Track patterns
Consistent false positives = a CLAUDE.md update waiting to happen.

🧪 Quick Quiz

What's the fastest way to get a code review in Claude Code?

Type /review — it analyzes all uncommitted changes and gives categorized feedback instantly.

What's the "Rubber Duck Review" technique?

Ask Claude to explain your code back to you. If its explanation doesn't match your intent, the code says something different from what you meant — you've found a semantic bug.

What should Claude review vs what should humans review?

Claude: logic bugs, security, performance, conventions, edge cases. Humans: architecture, business logic, UX, design decisions, long-term maintainability.

How do you automate Claude reviews in CI/CD?

Use claude --print in a GitHub Actions workflow — pipe the diff in, capture the review output as markdown, post it as a PR comment.

What's the anti-pattern to avoid with AI code review?

Using Claude as a rubber stamp — "Claude didn't find anything, so the code must be fine." AI review reduces bugs but doesn't eliminate the need for human review.

📌 Key Takeaways

  • /review gives instant, zero-setup feedback on uncommitted changes — use it before every push.
  • The multi-agent Code Review system analyzes PRs automatically with parallel bug hunters, verification, and severity ranking.
  • Specific review prompts (security-only, performance-only) catch more real bugs than generic "review this code."
  • The Fresh Eyes and Rubber Duck techniques catch bugs in your own code that self-review always misses.
  • Build a custom /review-pr skill so the whole team gets consistent, thorough reviews.
  • Automate with claude --print in GitHub Actions — every PR reviewed before humans even look.
  • AI catches mechanical bugs; humans catch design problems. Use both. Never use Claude as a rubber stamp.
