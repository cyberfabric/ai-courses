Chapter 5

CLAUDE.md & Memory Systems

Memory Map
💡

The Big Idea

CLAUDE.md is a project briefing for an AI teammate — write it once, and Claude starts every session already knowing your conventions, stack, and patterns. Combined with auto memory, Claude gets smarter about your project over time.

🧩 Core Concepts

📄
CLAUDE.md
A plain Markdown file Claude reads at the start of every session. Your project's conventions, build commands, architecture — written once, applied always.
🪜
The Hierarchy
CLAUDE.md files merge from broadest to most specific: user-level → project root → subdirectory. They stack, never replace.
🧠
Auto Memory
What Claude writes for itself across sessions — build quirks, workflow habits, preferences. Lives in ~/.claude/projects/ as Markdown files.
💤
Auto Dream
Periodic cleanup of auto memory: merges duplicates, fixes stale dates, removes contradictions. Runs ~every 24 hours after 5+ sessions.
⚙️
Settings Hierarchy
Five levels of settings, from highest to lowest priority: Managed → CLI Flags → Local → Project → User. Most specific wins.
📊
Context Budget
Every CLAUDE.md line costs context tokens. More instructions = less room for conversation. Keep it focused — ~150 instructions max.

🛠️ Building an Effective CLAUDE.md

  1. Run /init

    Auto-generate a starter CLAUDE.md from your project structure.

  2. Remove the Obvious

    Delete rules Claude already follows (proper indentation, standard syntax). Ask: "Would Claude make a mistake without this?"

  3. Add Conventions

    Tech stack, build commands, architecture patterns, test frameworks.

  4. Add Forbidden Patterns

    Specific mistakes to prevent: "NEVER use LiveData", "No Thread.sleep() in tests".

  5. Add Decision Records

    Explain the why behind intentional choices so Claude doesn't "fix" them.

  6. Keep It Under 200 Lines

    Compliance drops past ~150 instructions. Split into .claude/rules/ if needed.

📋 Four Essential CLAUDE.md Patterns

🚫
The Forbidden List
Explicit "NEVER do X" rules that prevent known mistakes. Example: "NEVER use !! (non-null assertion) in Kotlin."
📝
The Decision Record
Explain why things are intentional so Claude doesn't "fix" them. Example: "We use Room, not SQLDelight (team familiarity)."
🌍
The Context Provider
Business context code can't convey. Example: "Fintech app — never log sensitive data. Offline mode required."
✂️
The Import Split
When CLAUDE.md grows too large, split into .claude/rules/ files by topic. Keep main file as a focused quick reference.

🧠 Auto Memory Types

👤
User
Your role, preferences, and expertise. Helps Claude tailor responses to your level and style.
🔄
Feedback
Corrections ("don't do X") and confirmed approaches ("yes, keep doing Y"). Both matter equally.
🎯
Project
Ongoing work, goals, decisions, deadlines. Decays fast — includes "why" for future judgment calls.
🔗
Reference
Pointers to external systems: "bugs in Linear project INGEST", "latency dashboard at grafana.internal/..."

⚠️ Three CLAUDE.md Anti-Patterns

📚
The Novel
500+ lines with full project history. Claude can't prioritize when everything is "important." Keep it under 200 lines.
🌈
The Wishlist
"Write perfect code. Follow best practices." Too vague to act on. Be specific: which practices? What errors to handle?
💥
The Contradiction
"Keep functions under 20 lines" + "Add comprehensive error handling with logging." Pick one, or say when each applies.

⚡ Quick Quiz

What's the golden rule for CLAUDE.md content?

For every line, ask: "Would Claude make a mistake without this?" If Claude already does it correctly, the instruction is noise — remove it.

How many levels does the CLAUDE.md hierarchy have?

Three main levels: user-level (~/.claude/CLAUDE.md), project root (./CLAUDE.md), and subdirectory (./src/api/CLAUDE.md). They merge from broadest to most specific.

What are the four auto memory types?

User (role & preferences), Feedback (corrections & confirmed approaches), Project (goals, decisions, deadlines), and Reference (pointers to external resources).

What happens when your CLAUDE.md exceeds ~150 instructions?

Compliance starts to drop. Split your instructions into .claude/rules/ files organized by topic, and keep the main CLAUDE.md under 200 lines.

What's the Correction Capture Pattern?

When Claude makes a mistake during a session, add that correction to your CLAUDE.md. Over time, your instructions evolve into a precisely-tuned set that prevents every mistake Claude has ever made on your project.

📌 Key Takeaways

  • CLAUDE.md is a plain Markdown file that Claude reads at the start of every session — like onboarding an AI teammate.
  • Three levels merge: user → project → subdirectory. They stack, never replace each other.
  • Only include instructions that prevent real mistakes — if Claude already does it right, skip it.
  • Auto memory lets Claude learn across sessions; Auto Dream consolidates and cleans up memory automatically.
  • Keep CLAUDE.md under 200 lines. Split into .claude/rules/ when it grows too large.
  • Commit team settings (CLAUDE.md, settings.json, commands/) to git. Keep personal prefs local.
  • Use the Correction Capture Pattern: every Claude mistake becomes a new CLAUDE.md rule.
