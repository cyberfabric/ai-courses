The Big Idea

One power user = 10% improvement. Ten power users = transformation. Build shared infrastructure (CLAUDE.md, skills, workflows) that makes everyone effective — don't force everyone to use Claude the same way.

📋 Team CLAUDE.md: The AI Constitution

Build & Test Commands
How to build, test, lint — so Claude runs the right commands for everyone.
Architecture Overview
Patterns, key directories, data flow — so Claude generates code that fits.
Code Conventions
Language choices, naming, forbidden patterns — consistency across the team.
Git Conventions
Branch naming, commit format, workflow rules — no more inconsistent commits.
"What NOT to Do"
Anti-patterns your team has encountered — prevent repeating past mistakes.
Available Skills
List of team skills so devs know what's available.

💡 Keep it under 200 lines. If Claude already does it correctly, don't waste a line on it.

🚀 New Developer Onboarding Path

  1. 📅 Week 1: Claude.ai

    Questions and brainstorming. Learn /review and /commit. Understand the project CLAUDE.md.

  2. 📅 Week 2: Claude Code

    Coding tasks and reviews. Use team skills: /review-pr, /generate-tests.

  3. 📅 Week 3: Personal Setup

    Create ~/.claude/CLAUDE.md for personal preferences. Build first custom skill.

  4. 📅 Week 4: Optimization

    Custom skills, personal workflow optimization, sharing discoveries with the team.

💰 Cost & 🔒 Security

📊
Right-Size Models
Haiku for simple tasks, Sonnet for coding, Opus for complex reasoning.
🗜️
Compact Often
Use /compact regularly. Start fresh when context is stale.
🔑
Keys in Env Vars
Never hardcode API keys. Use secrets managers at scale. Rotate regularly.
🚫
No Secrets to Claude
Never paste production credentials, customer data, or PII. Add rules to CLAUDE.md.

🧪 Quick Quiz

Which group should you target first when introducing Claude to a team?

The enthusiasts — not the skeptics. Build shared tools with enthusiasts, let visible results attract the curious, and skeptics follow once they see the impact.

What's the single highest-impact thing for team Claude adoption?

A shared CLAUDE.md committed to git. It ensures every developer gets the same baseline behavior — conventions, build commands, forbidden patterns — regardless of experience.

How long should a team CLAUDE.md be?

Under 200 lines. Focus on rules that prevent real mistakes. If Claude already does something correctly without being told, don't waste a line on it.

What's the recommended onboarding order for a new Claude user?

Week 1: Claude.ai for questions → Week 2: Claude Code for coding → Week 3: Team skills → Week 4: Custom skills and personal optimization.

What's the common onboarding mistake to avoid?

Information overload — dumping 50 features at once. Start with 3 things that produce immediate value: ask questions about code, review changes, generate tests.

📌 Key Takeaways

  • Start with enthusiasts, build shared tools, let results attract the rest. Don't try to convert skeptics first.
  • A shared CLAUDE.md (under 200 lines, committed to git) is the highest-impact team investment.
  • Shared skills in .claude/commands/ give everyone consistent quality — reviews, tests, onboarding.
  • Onboard gradually: Claude.ai → Claude Code → team skills → personal customization over 4 weeks.
  • Knowledge sharing ritual: 5 minutes per standup for "Claude wins this week" builds collective expertise.
  • Security is non-negotiable: env vars for keys, no secrets in prompts, AI code gets same review as human code.
  • Don't over-measure ROI. Ship faster + fewer bugs + happier devs = it's working.
