The Big Idea

The Claude API gives you programmatic access to the same models behind Claude.ai and Claude Code — but you control everything: system prompt, tools, conversation flow, and output delivery. It's how you put Claude inside your apps.

🎯 Why Use the API?

🤖
Automation
Run Claude in CI/CD, cron jobs, and scripts — zero human interaction needed.
🧩
Integration
Embed Claude in Slack bots, dashboards, review tools — anywhere your code runs.
🎛️
Control
Fine-tune system prompts, temperature, tool definitions, and output format precisely.
📈
Scale
Process hundreds of files or PRs in parallel — not one conversation at a time.

⚡ Power Features at a Glance

  1. 🔊 Streaming

    Tokens arrive in real-time — users see text appear word by word instead of waiting for the full response.

  2. 🧠 Adaptive Thinking

    Opus 4.7 only accepts thinking.type: "adaptive" — the old "enabled" form returns 400. Steer with output_config.effort (low / medium / high / xhigh / max).

  3. 🖼️ Vision

    Send images (base64 or URL) alongside text. Great for UI reviews, error screenshots, and whiteboard-to-code.

  4. 💰 Prompt Caching

    Add cache_control: {"type": "ephemeral"}. Write: 1.25× (5-min) or 2× (1-hour). Read: 0.1×. Pays off after one re-read (5-min) or two (1-hour).

  5. 🗂️ Memory + Context Editing

    Beta header context-management-2025-06-27 unlocks a managed memory file directory and tool-result pruning — essential for hours-long agent runs.

🛡️ Error Handling Essentials

429 — Rate Limited
Implement exponential backoff. SDK has built-in retries via max_retries.
529 — Overloaded
API at capacity. Respect the Retry-After header and back off.
Token Limit Exceeded
Input too large for context window. Truncate or summarize before sending.
Timeouts
Set reasonable timeouts. Complex Opus calls can take 30+ seconds.
API Key Security
Never hardcode keys. Use ANTHROPIC_API_KEY env var or a secrets manager.

🧪 Quick Quiz

What are the 3 required parameters for every Messages API call?

model, max_tokens, and messages.

How much do you save with prompt caching on repeated requests?

Cache reads are 0.1× base input (90% off). Writes are 1.25× (5-min TTL) or 2× (1-hour TTL), so caching pays off after a single re-read for 5-min and two re-reads for 1-hour.

Why does the old thinking.type: "enabled" form fail on Opus 4.7?

Opus 4.7 only supports "adaptive" thinking and rejects "enabled" with a 400. You steer depth via output_config.effort (low…max, plus xhigh exclusive to 4.7).

What's the "tool use loop"?

Send message → Claude requests a tool call → your code executes it → send result back → repeat until Claude gives a final text response.

When should you use extended thinking?

For complex problems: debugging, architecture reviews, multi-file refactors — anything that benefits from a dedicated reasoning phase before responding.

What's the difference between the Agent SDK and raw tool use?

Raw tool use requires you to manage the loop manually. The Agent SDK handles the entire agentic loop automatically — think, call tools, observe, decide, repeat.

📌 Key Takeaways

  • The API gives you the same Claude models with full programmatic control — system prompts, tools, streaming, and more.
  • Three things you always need: model, max_tokens, messages.
  • System prompts are your biggest lever — they transform generic answers into expert, targeted responses.
  • Tool use turns Claude from a text generator into an agent that can call your functions and act on real data.
  • Prompt caching: writes are 1.25× (5-min) or 2× (1-hour), reads are 0.1× — 90% off after the breakeven re-read.
  • Opus 4.7 only accepts adaptive thinking; steer it with output_config.effort, including the new xhigh level.
  • For long-running agents, pair the memory tool and context editing (context-management-2025-06-27 beta) with prompt caching.
  • The Agent SDK handles the agentic loop for you — define a goal and tools, and let it run.
  • Always use environment variables for API keys — never hardcode them.
