The Big Idea

Documentation that used to take an hour now takes two minutes. The key: let Claude read the actual code — it extracts real fields, commands, and patterns, not generic boilerplate. Then keep docs alive with update skills.

📚 The Documentation Toolbox

📋
READMEs
Your project's front door. Give Claude a specific list of sections to include — description, setup, run, test, structure.
🔌
API Docs
Point Claude at route files. It extracts methods, paths, params, schemas, and error codes from real code.
🏛️
ADRs
Architecture Decision Records — document why you made choices. Tell Claude the decision + reasoning, it handles the format.
📝
Changelogs
Feed git log to Claude → get human-readable release notes grouped by Features, Fixes, Breaking Changes.
💬
Code Comments
Only add comments that explain why, not what. If the code is self-documenting, skip the comment.
🚀
Onboarding Guides
Generate "Getting Started" docs for new devs — real paths, real commands, key concepts, common gotchas.
📊
Mermaid Diagrams
Text-based diagrams that render in GitHub/GitLab. 3–6x more token-efficient than prose descriptions.

🔄 Keeping Docs Alive

  1. 📝 The /update-docs Skill

    Reads README, API docs, and architecture docs — compares against current code — updates only what's stale.

  2. 📋 CLAUDE.md Conventions

    "When modifying API routes, update docs/api.md." Claude follows these rules every time it edits code.

  3. ⏰ Periodic Runs

    Run /update-docs before releases, after sprints, or when onboarding. Don't let docs rot.

🚫 Documentation Anti-Patterns

Over-Documentation
Adding doc comments to every function creates noise. getUserById(id) is self-documenting.
The Boilerplate Factory
Generic class comments like "This class handles user operations" add zero value beyond the class name.
Generated and Forgotten
Beautiful docs that are never updated are worse than no docs — stale docs actively mislead.
Vague Prompts
"Write a README for my project" produces generic templates. Be specific about sections and audience.
⚡

Power Move: Retroactive ADRs

Ask Claude to read your codebase and identify undocumented architectural decisions. It infers reasoning from code, deps, and patterns — then drafts ADRs for decisions that were made implicitly. 5 minutes → hours of documentation.

🧪 Quick Quiz

Why should you generate docs in Claude Code, not Claude.ai?

Claude Code has file access — it reads your actual code and extracts real fields, commands, and patterns. Claude.ai without project context can only generate generic templates.

What makes Mermaid diagrams better than image-based diagrams for docs?

They're text-based (version-controlled), render in GitHub/GitLab natively, 3–6x more token-efficient than prose, and require no external tools.

What's a retroactive ADR and when should you use it?

An ADR documenting a past architectural decision that was never formally recorded. Ask Claude to analyze your codebase and infer the reasoning. Use it when your project has undocumented decisions.

How do you keep docs from going stale?

Build an /update-docs skill that compares docs against current code. Add doc-update rules to CLAUDE.md. Run the skill before releases and after sprints.

When should you NOT add a code comment?

When the comment just restates what the code does. Only add comments that explain why — non-obvious side effects, workarounds, performance decisions, or references to external constraints.

📌 Key Takeaways

  • Always let Claude read the actual code — it generates accurate docs, not generic boilerplate.
  • Be specific in your prompts: list the sections you want, the audience, and the files to read.
  • ADRs document why decisions were made — Claude can draft retroactive ADRs from your existing codebase.
  • Mermaid diagrams are text-based, version-controlled, and 3–6x more token-efficient than prose.
  • Comments should explain why, never what. If code is self-documenting, skip the comment.
  • Build an /update-docs skill and add doc conventions to CLAUDE.md to keep docs alive.
  • Stale docs are worse than no docs — the biggest problem isn't writing them, it's maintaining them.
