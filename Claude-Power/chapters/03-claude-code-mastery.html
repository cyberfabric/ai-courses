Chapter 3

Claude Code CLI Mastery

Flags, modes, slash commands, and the techniques that separate beginners from power users

In the last chapter, you unlocked Claude.ai's hidden features. Now we shift to the tool that developers actually spend most of their time with: Claude Code.

If you've used Claude Code at all, you've probably typed a question, watched it edit a file, and thought "that's cool." But you're driving a race car in first gear. This chapter teaches you to shift up.

Installation and Setup

If you already have Claude Code installed, make sure you're on the latest version:

npm install -g @anthropic-ai/claude-code

To check your version:

claude --version

Authentication

Claude Code supports multiple authentication methods:

  • Anthropic API key — set ANTHROPIC_API_KEY in your environment
  • Claude.ai account — log in through the browser flow with claude login
  • OAuth / SSO — for Team and Enterprise plans
If you're on a Pro or Max plan for Claude.ai, you can use Claude Code through your subscription — no separate API key needed. Just run claude login and authenticate through the browser.

The Basics (You Probably Know These)

Start an interactive session:

claude

Ask a question with initial context:

claude "explain the authentication flow in this project"

Once inside, you type naturally. Claude Code reads your files, understands your project structure, and can make edits, run commands, search code, and manage git. You already know this.

Now let's learn what you don't know.

Slash Commands: The Built-In Toolkit

Slash commands are typed inside an active Claude Code session. They control Claude's behavior without consuming tokens on a prompt. Here are the ones that matter:

Command What it does When to use it
/help Shows all available commands When you forget what's available
/clear Clears the conversation history Fresh start without exiting
/compact Compresses conversation history When the session gets long and Claude starts losing context
/cost Shows token usage and cost for this session Tracking spend, budget awareness
/model Switch models mid-session Start with Sonnet, switch to Opus for a hard problem
/review Review code changes Quick code review of recent changes
/commit Create a git commit with AI-generated message After Claude makes changes you're happy with
/diff Show all changes Claude has made Before accepting or committing
/resume Resume a previous session Pick up where you left off yesterday
/init Auto-generate a CLAUDE.md for your project First time setting up Claude Code in a repo
/config Open settings TUI Change settings without editing JSON
/permissions Audit what Claude can read/write Security check, understanding access
The /compact command is your secret weapon for long sessions. As your conversation grows, Claude's context window fills up and it starts forgetting earlier context. Instead of starting fresh and losing everything, /compact compresses the history — keeping the important parts while freeing up space. Use it whenever Claude starts giving answers that ignore things you discussed earlier.

Flags That Change Everything

CLI flags are passed when you start Claude Code. These transform how the entire session behaves.

The Big Five

--print (or -p) — Headless Mode

This is the flag that turns Claude Code from an interactive tool into an automation engine. With --print, Claude processes your prompt and exits — no interactive session, no waiting for input. Perfect for scripts and CI/CD.

# Generate a changelog from recent commits
claude -p "generate a changelog from the last 10 commits in markdown format"

# Pipe input into Claude
cat error.log | claude -p "what's causing the errors in this log?"

# Use in a script
REVIEW=$(claude -p "review the staged changes for security issues")
echo "$REVIEW" > review-output.md

--model — Choose Your Engine

# Use Opus for a complex refactoring task
claude --model opus "refactor the auth module to use the new token system"

# Use Haiku for a quick lookup
claude -p --model haiku "what's the syntax for a Kotlin sealed interface?"

--output-format — Machine-Readable Output

# Get JSON output for parsing in scripts
claude -p --output-format json "list all the public functions in src/api/"

# Stream JSON for real-time processing
claude -p --output-format stream-json "analyze this project's dependencies"

--permission-mode — Control What Claude Can Do

# Auto-accept file edits (skip "allow edit?" prompts)
claude --permission-mode acceptEdits

# Plan-only mode (Claude analyzes but can't change anything)
claude --permission-mode plan

--allowedTools — Fine-Grained Tool Access

# Only allow reading and searching, no editing
claude --allowedTools "Read,Grep,Glob"

# Allow everything except bash
claude --allowedTools "Edit,Read,Write,Grep,Glob"

Manual workflow:

Open Claude Code. Ask for a review. Read the output. Copy it. Paste into a file. Run the next command. Repeat.

Automated with --print:

claude -p "review staged changes" > review.md
claude -p "run tests and summarize failures" >> review.md
gh pr create --body-file review.md

Three commands. No interaction needed. Runs in CI.

The --bare flag for scripting

When using -p in scripts or CI pipelines, add --bare to skip hooks, LSP startup, plugin sync, and skill directory walks. This makes headless calls significantly faster:

claude -p --bare "summarize the changes in this PR"

Permission Modes Deep Dive

Permission modes control whether Claude asks before acting. This is one of the most important configuration choices you'll make.

Mode Behavior Best for
default Asks permission for edits and commands Learning, unfamiliar codebases, sensitive repos
acceptEdits Auto-accepts file edits, still asks for shell commands Everyday coding when you trust Claude's edits
plan Claude analyzes and plans but cannot make changes Architecture review, understanding code, pre-flight check
auto Claude decides what needs permission (launched March 2026) Experienced users on trusted projects
Cycle permission modes mid-session: Press Shift+Tab to cycle through default → acceptEdits → plan → auto. The current mode is displayed below the prompt. No need to restart the session.
Auto mode is available on Max, Team, Enterprise, and API plans (not Pro). On Team and Enterprise, an admin must enable it in Claude Code admin settings before users can turn it on. It works with Sonnet 4.6, Opus 4.6, or Opus 4.7 on Team / Enterprise / API; on the Max plan, auto mode is Opus 4.7 only. Haiku and older models aren’t supported. Anthropic API only — not on Bedrock, Vertex, or Foundry.
Two more modes the docs list but most users skip: dontAsk auto-denies anything you haven’t pre-approved (great for locked-down CI), and bypassPermissions skips the permission layer entirely (containers and VMs only — equivalent to --dangerously-skip-permissions). Both are opt-in via --permission-mode; neither shows up in the default Shift+Tab cycle.

Plan Mode: Think Before You Act

Plan mode deserves special attention. When you're about to tackle something complex — a big refactor, an unfamiliar codebase, a multi-file change — start in plan mode.

/plan Migrate our REST API from Express to Fastify. What files need to change, what's the migration order, and what are the risks?
I'll analyze the codebase and create a migration plan. Let me read your route definitions, middleware, and configuration... **Migration Plan: Express → Fastify** **Files to change (in order):** 1. `package.json` — swap dependencies 2. `src/server.ts` — replace Express app setup with Fastify 3. `src/middleware/auth.ts` — convert to Fastify plugin pattern 4. `src/routes/*.ts` (12 files) — update route syntax 5. `tests/**/*.ts` — update test setup for Fastify's inject **Risks:** - Middleware ordering works differently in Fastify (plugins are scoped) - Error handling patterns need to change (no more `next()`) - Your custom `req.user` typing needs Fastify declaration merging **Estimated impact:** ~30 files, ~400 lines changed Want me to switch to execution mode and start with step 1?

See the power? Claude analyzed the codebase, identified every file that needs changing, ordered them logically, and flagged the tricky parts — all without touching a single file. Now you can review the plan, adjust it, and then let Claude execute.

The ! Prefix: Shell Commands Inside Claude Code

Need to run a quick shell command without leaving your Claude Code session? Prefix it with !:

! git status
! npm test
! cat package.json | head -20

The output goes directly into the conversation context, so Claude can see and react to it. This is faster than switching to another terminal.

Context Management

Understanding what Claude Code "sees" is critical to getting good results.

What goes into context automatically

  • CLAUDE.md files — project root, subdirectories, user-level (~/.claude/CLAUDE.md)
  • Your conversation history — everything you've said and Claude has responded
  • Files Claude has read — any file Claude opened during the session
  • Tool results — output from commands, searches, edits

What does NOT go into context automatically

  • Your entire codebase (Claude reads files on demand, not upfront)
  • Files you haven't mentioned or that Claude hasn't opened
  • Previous sessions (unless you /resume)
When to start fresh vs continue: Continue the session when you're iterating on the same task. Start fresh (/clear or a new session) when you're switching to an unrelated task. Stale context from a previous task can confuse Claude on the new one.

Cost Awareness

Claude Code charges per token. Being smart about cost doesn't mean being cheap — it means not wasting money.

/cost

Run this periodically to see your session's token usage and cost. Here are strategies to keep spend efficient:

Sonnet Default for 90% of tasks
/compact Compress long sessions
Haiku Quick lookups and bulk ops
  • Use Sonnet as your default — it's fast, capable, and 40% cheaper than Opus
  • Switch to Opus 4.7 for hard problems — the new flagship (April 16, 2026) at the same $5/$25 price as 4.6, with a step-change jump in agentic coding. On Enterprise pay-as-you-go and the API, it’s also been the default since April 23, 2026.
  • Use Haiku for quick questions — syntax lookups, simple formatting, bulk classification
  • Compact regularly — long conversations burn tokens on repeated context
  • Be specific in your prompts — vague prompts cause Claude to read more files searching for context
  • Start fresh when switching tasks — carrying irrelevant context wastes tokens

The .claude/ Directory

Every project (and your home directory) can have a .claude/ folder. Here's what lives there:

Path Purpose
.claude/settings.json Project settings (committed, shared with team)
.claude/settings.local.json Local settings (gitignored, your personal prefs)
.claude/commands/ Custom slash commands (skills) for this project
~/.claude/settings.json User-level settings (all projects)
~/.claude/commands/ User-level slash commands (available everywhere)
~/.claude/CLAUDE.md User-level instructions (loaded in every project)

We'll dive deep into CLAUDE.md in Chapter 5 and custom commands in Chapter 6. For now, just know this directory is where Claude Code stores its configuration.

Keyboard Shortcuts in a Session

Shortcut Action
Ctrl+C Cancel current input or generation
Ctrl+D Exit the session
Ctrl+L Clear screen
Ctrl+R Reverse search through history
Shift+Tab Cycle permission modes
Alt+T Toggle extended thinking
Alt+P Quick model selection
Ctrl+G Open prompt in your editor
Ctrl+B Background the current running task
Ctrl+T Toggle task list
Alt+T toggles extended thinking. When Claude is working on a complex problem and the answer seems shallow, toggle thinking on. Claude will reason step-by-step before responding, dramatically improving quality on hard tasks. Toggle it off for simple questions to save tokens and time.

Automate something you do manually today.

  1. Think of a repetitive task: generating a changelog, reviewing staged changes, summarizing TODOs, updating a version number
  2. Write it as a --print command: 
    claude -p "generate a changelog from the last 5 commits in keep-a-changelog format"
  3. Run it and check the output
  4. If it works, save it as a shell alias or script: 
    alias changelog='claude -p "generate a changelog from the last 5 commits in keep-a-changelog format"'
  5. Bonus: pipe something into it: 
    git diff --staged | claude -p "review these changes for potential bugs"

You've just built your first Claude automation. In Chapter 6, we'll turn these into proper reusable skills.
