Chapter 7

The Claude API

Build Claude into your own apps — from first API call to autonomous agents

In the last chapter, you extended Claude Code with custom skills and hooks. But all of that still happens inside Claude Code's world. What if you want Claude's intelligence in your app, your script, your CI pipeline?

That's what the API is for. The Claude API gives you direct, programmatic access to the same models powering Claude.ai and Claude Code — but you control everything: the system prompt, the conversation flow, the tools Claude can use, and how the output gets delivered.

This chapter won't make you an API expert overnight, but it will get you from "I've never called the API" to "I built a working tool" in one sitting.

Why Use the API?

You already have Claude Code and Claude.ai. Why bother with the API? Because the API unlocks things the other interfaces can't do:

Automation
Run Claude in CI/CD pipelines, cron jobs, and scripts with zero human interaction
Integration
Embed Claude in your own app — Slack bots, code review tools, internal dashboards
Control
Fine-tune system prompts, temperature, tool definitions, and output format precisely
Scale
Process hundreds of files, PRs, or documents in parallel — not one conversation at a time

Getting Started: Your First API Call

Step 1: Get an API Key

Go to console.anthropic.com, create an account (or sign in), and generate an API key. Store it securely — treat it like a password.

Never hardcode API keys in source code. Use environment variables (ANTHROPIC_API_KEY) or a secrets manager. If you accidentally commit a key, rotate it immediately.

Step 2: Install the SDK

Anthropic provides official SDKs for Python and TypeScript:

Python

pip install anthropic

TypeScript

npm install @anthropic-ai/sdk

Step 3: Make Your First Call

Python — Hello, Claude API: 
import anthropic

client = anthropic.Anthropic()  # reads ANTHROPIC_API_KEY from env

message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "Explain Kotlin coroutines in 3 sentences."}
    ]
)

print(message.content[0].text)
TypeScript — Same thing: 
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic(); // reads ANTHROPIC_API_KEY from env

const message = await client.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 1024,
  messages: [
    { role: "user", content: "Explain Kotlin coroutines in 3 sentences." }
  ],
});

console.log(message.content[0].text);

That's it. Three lines of real code and you're talking to Claude programmatically.

The Messages API: Anatomy of a Request

Every API call follows the same structure. Here are the key parameters:

ParameterRequiredWhat It Does
modelYesWhich Claude model to use
max_tokensYesMaximum tokens in the response
messagesYesThe conversation history (user/assistant turns)
systemNoSystem prompt — sets Claude's behavior and context
toolsNoTools Claude can call (function definitions)
temperatureNoRandomness (0 = deterministic, 1 = creative)
streamNoEnable streaming for real-time output

System Prompts: Setting the Stage

The system prompt is your most powerful lever. It defines who Claude is and how it should behave for your specific use case:

No System Prompt

Claude gives generic, verbose answers. It doesn't know your context or constraints.

"Kotlin coroutines are a concurrency design pattern that you can use on Android to simplify code that executes asynchronously..."

With System Prompt

system: "You are a senior Android engineer. Be concise. Use code examples. Skip basics."

"Coroutines give you structured concurrency with suspend functions. Launch from viewModelScope for auto-cancellation: viewModelScope.launch { val data = repo.fetch() }"

Streaming: Real-Time Output

For anything user-facing, you want streaming. Instead of waiting for the entire response, you get tokens as they're generated:

Python streaming example: 
with client.messages.stream(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a Kotlin data class for a User"}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)

The user sees text appear word by word instead of staring at a blank screen for 5 seconds. Much better UX.

Tool Use (Function Calling)

This is where the API gets really powerful. You can define custom tools — functions that Claude can call during a conversation. Claude decides when to use them, calls them with the right arguments, and incorporates the results into its response.

1. Define Tools

Tell Claude what functions are available

→

2. Claude Decides

Claude calls a tool when needed

→

3. You Execute

Your code runs the function

→

4. Return Result

Send the result back to Claude

Defining a tool: 
tools = [
    {
        "name": "get_build_status",
        "description": "Check the CI build status for a branch",
        "input_schema": {
            "type": "object",
            "properties": {
                "branch": {
                    "type": "string",
                    "description": "The git branch name"
                }
            },
            "required": ["branch"]
        }
    }
]

message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "Is the feature/login branch passing CI?"}]
)

Claude will respond with a tool_use block asking to call get_build_status with {"branch": "feature/login"}. Your code executes the actual check, and you send the result back as a tool_result message. Claude then crafts its final answer using the real data.

The tool use loop: For multi-step tasks, Claude might call several tools in sequence. Your code needs a loop: send a message → check if Claude wants to use a tool → execute it → send the result back → repeat until Claude gives a final text response.

Extended Thinking

For complex problems — debugging, architecture decisions, multi-file refactors — you can enable extended thinking. This gives Claude a dedicated reasoning phase before it responds:

Enabling extended thinking (Opus 4.7+): 
message = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=16000,
    thinking={
        "type": "adaptive",
        "display": "summarized"  # default on 4.7 is "omitted"
    },
    output_config={"effort": "high"},  # low | medium | high | xhigh | max
    messages=[{"role": "user", "content": "Review this architecture and find potential issues: ..."}]
)

# The response includes thinking blocks and text blocks
for block in message.content:
    if block.type == "thinking":
        print(f"Thinking: {block.thinking}")
    elif block.type == "text":
        print(f"Answer: {block.text}")

On Opus 4.7, only "adaptive" thinking is supported — the older {"type": "enabled", "budget_tokens": N} form returns a 400 error. Adaptive lets Claude decide when (and how much) to think; you steer it with the effort parameter inside output_config. On Opus 4.6 and Sonnet 4.6, the "enabled" form still works but is deprecated.

Effort levels: low, medium, high (default), xhigh (Opus 4.7 only), max. Higher effort means deeper reasoning at the cost of latency and tokens. display: "omitted" skips streaming the thinking text but you still pay for the tokens — use it when you don't show the reasoning to users.

Vision: Sending Images

Claude can analyze images sent via the API — screenshots, diagrams, whiteboard photos, UI mockups:

Sending an image for analysis: 
import base64

with open("screenshot.png", "rb") as f:
    image_data = base64.standard_b64encode(f.read()).decode("utf-8")

message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": [
            {
                "type": "image",
                "source": {
                    "type": "base64",
                    "media_type": "image/png",
                    "data": image_data
                }
            },
            {
                "type": "text",
                "text": "What accessibility issues do you see in this UI?"
            }
        ]
    }]
)

This is incredibly useful for UI reviews, analyzing error screenshots, or converting whiteboard sketches into code.

Prompt Caching: Saving Money

If you're sending the same large context repeatedly (a system prompt, a codebase, documentation), prompt caching can cut your costs dramatically.

Without Caching

Every request re-processes your 50,000-token system prompt. Expensive and slow.

With Prompt Caching

Cache write: 1.25× base input (5-min TTL) or 2× (1-hour TTL). Cache read: 0.1× base input. So 5-min caching pays off after a single re-read; 1-hour pays off after two.

Pick the 5-min TTL for chatty workloads where the same context is reused inside a session. Pick the 1-hour TTL when calls are sparser but still grouped (e.g., a nightly batch where requests trickle in over 30 minutes). Enable caching by adding cache_control to the content blocks you want cached:

message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[{
        "type": "text",
        "text": "You are a code reviewer for our Android app. Here are our conventions: ...[large text]...",
        "cache_control": {"type": "ephemeral"}
    }],
    messages=[{"role": "user", "content": "Review this function: ..."}]
)

Power Tip: Cache Your Codebase Context

For batch operations (reviewing 50 PRs, analyzing 100 files), put your project conventions and architecture docs in a cached system prompt. The first call pays a small premium, but every subsequent call reads at 10% of base input. On a batch of 100 calls with a 50K-token system prompt on Sonnet 4.6, that's roughly ~$13 saved vs uncached ($15 → ~$1.70). Same shape on Opus 4.7: ~$22 saved ($25 → ~$2.80).

Token Counting and Cost Management

Every API response includes usage information:

# After any API call:
print(f"Input tokens:  {message.usage.input_tokens}")
print(f"Output tokens: {message.usage.output_tokens}")

# With caching:
print(f"Cache read:    {message.usage.cache_read_input_tokens}")
print(f"Cache write:   {message.usage.cache_creation_input_tokens}")
Cost rule of thumb: Output tokens are exactly 5× input across the current line-up ($5/$25 Opus 4.7, $3/$15 Sonnet 4.6, $1/$5 Haiku 4.5). Prompt caching makes input tokens even cheaper (10% of base on cache reads). For cost-sensitive applications, use Haiku for simple tasks and reserve Opus 4.7 for complex reasoning.

Long-Running Agents: Memory & Context Editing

For agents that run for hours or accumulate hundreds of tool calls, two newer primitives keep your context window from blowing up:

  • Memory tool: a managed file-directory Claude can read, write, update, and delete — persistent across requests. Instead of stuffing everything into the prompt, the agent stores facts in memory and pulls them back on demand.
  • Context editing: server-side strategies (like clear_tool_uses_20250919) that prune stale tool results once the conversation crosses a threshold. Old file reads and search results stop costing you tokens after Claude has digested them.

Both are gated by the context-management-2025-06-27 beta header and pair well with prompt caching: keep the cached system prompt stable, let memory hold the long-tail state, and let context editing drop expired tool noise.

The Claude Agent SDK

If you want to build something that works like Claude Code itself — an autonomous agent that reads files, runs commands, searches the web, and makes decisions — the Agent SDK gives you that power:

A minimal agent in Python: 
from claude_agent_sdk import Agent

agent = Agent(
    model="claude-sonnet-4-6",
    prompt="Find all TODO comments in the src/ directory and create GitHub issues for each one.",
    allowed_tools=["Read", "Glob", "Grep", "Bash"],
    permission_mode="accept-edits"
)

async for event in agent.run():
    if event.type == "text":
        print(event.text)
    elif event.type == "tool_use":
        print(f"Using tool: {event.name}")
    elif event.type == "result":
        print(f"Done: {event.text}")

The Agent SDK handles the entire agentic loop: Claude thinks, calls tools, observes results, decides what to do next, and repeats until the task is done. You just define the goal and the available tools.

Built-in Agent Tools
Read, Write, Edit, Bash, Glob, Grep, WebSearch, WebFetch — the same tools Claude Code uses, available in your agent.

Error Handling and Rate Limits

Production API code needs to handle failures gracefully:

  • Rate limits (429): Implement exponential backoff. The SDK has built-in retry logic — configure max_retries
  • Overloaded (529): The API is temporarily at capacity. Back off and retry after the Retry-After header
  • Token limits: If your input exceeds the context window, you'll get an error. Truncate or summarize input before sending
  • Timeouts: Set reasonable timeouts for your use case. Complex Opus calls can take 30+ seconds
# The SDK handles retries automatically:
client = anthropic.Anthropic(max_retries=3)

# For manual control:
try:
    message = client.messages.create(...)
except anthropic.RateLimitError:
    # Back off and retry
    time.sleep(60)
except anthropic.APIError as e:
    print(f"API error: {e.status_code} - {e.message}")

Putting It All Together: A Practical Script

Here's a real-world example — a script that analyzes a file and suggests improvements:

file-reviewer.py — a CLI code review tool: 
import anthropic
import sys

client = anthropic.Anthropic()

def review_file(filepath: str) -> str:
    with open(filepath) as f:
        code = f.read()

    message = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=2048,
        system="You are a senior code reviewer. Be specific and actionable. "
               "Categorize issues as: BUG, SECURITY, PERFORMANCE, or STYLE. "
               "Include line numbers. Skip obvious things.",
        messages=[{
            "role": "user",
            "content": f"Review this file ({filepath}):\n\n```\n{code}\n```"
        }]
    )
    return message.content[0].text

if __name__ == "__main__":
    for path in sys.argv[1:]:
        print(f"\n{'='*60}\n{path}\n{'='*60}")
        print(review_file(path))

Run it: python file-reviewer.py src/UserRepository.kt src/LoginViewModel.kt

In 20 lines of Python, you have a reusable code review tool that works on any file. Pipe it into CI, run it pre-commit, or build a Slack bot around it.
