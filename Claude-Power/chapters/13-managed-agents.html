Chapter 13

Claude Managed Agents

Offload long-running tasks to cloud-hosted Claude instances — the “set it and forget it” power move

You built your playbook. You have skills, hooks, memory, and a daily routine that compounds. You're a power user. So what's left?

Here's the thing: everything you've done so far requires you to be present. You start Claude Code, you watch it work, you approve tool calls, you wait for results. That's great for interactive coding. But what about the 30-minute analysis task? The overnight code review of a massive PR? The weekly report that someone has to remember to generate?

That's where Managed Agents come in. Instead of running Claude on your laptop and babysitting the process, you send the task to Anthropic's cloud infrastructure and walk away. Claude runs in a secure container with its own filesystem, tools, and network access. You get results when it's done. No laptop required. No terminal open. No watching a cursor blink.

The one-sentence version: Managed Agents are Claude instances that run on Anthropic's servers instead of your machine. You define what to do and what tools to use, then fire and forget.

Why Should You Care?

If you're already using Claude Code effectively (and after 12 chapters, you are), Managed Agents unlock a new category of tasks:

  • Long-running work — tasks that take 30 minutes to several hours. Your laptop doesn't need to stay open.
  • Scheduled tasks — daily code quality reports, weekly dependency audits, nightly test suite analysis.
  • Parallel workloads — spin up 10 agents to review 10 repos simultaneously.
  • CI/CD integration — trigger an agent from your pipeline for automated review, documentation, or migration checks.

Think of it this way: Claude Code is like having a brilliant colleague sitting next to you. Managed Agents are like having that colleague work from their own office, on their own computer, and sending you a report when they're done.

When to Use What: The Decision Framework

You now have three ways to use Claude programmatically. Here's when each one shines:

Claude Code (CLI / IDE)

  • Interactive coding sessions
  • You're at your desk, working alongside Claude
  • Quick tasks: < 5 minutes
  • You want to review each step
  • Local file access is essential

Claude API (Direct)

  • Building Claude into your own app
  • Custom UI or workflow
  • You manage the agent loop yourself
  • Fine-grained control over every call
  • Token-level cost optimization

Managed Agents

  • Fire-and-forget tasks
  • Long-running analysis (30min+)
  • Scheduled / recurring jobs
  • No infrastructure to manage
  • Built-in sandboxing and tools

Rule of Thumb

If you need to watch Claude work → use Claude Code. If you need to embed Claude in your product → use the API. If you need to send Claude off to do a job → use Managed Agents.

The Four Core Concepts

Managed Agents are built on four objects. Understand these and everything else falls into place:

1. Agent

The definition of what Claude should do. You specify the model, system prompt, available tools, and MCP servers. Create it once, reuse it across many sessions.

→

2. Environment

The workspace where Claude runs. A cloud container with pre-installed packages, network rules, and optional mounted files. Think of it as Claude's office.

→

3. Session

A running instance of work. You combine an Agent + Environment, give it a task, and the session executes. Each session is independent and isolated.

→

4. Events

The communication channel. Events stream from the session via SSE (Server-Sent Events), telling you what Claude is doing, what tools it's calling, and what results it produces.

Here's the mental model: you create an Agent (the "who"), set up an Environment (the "where"), start a Session (the "go"), and listen to Events (the "status updates").

Creating Your First Managed Agent

Let's build a real example. We'll create an agent that can analyze code, run commands, and search the web — then send it a task.

Step 1: Install and configure the SDK 
pip install anthropic

# Set your API key
export ANTHROPIC_API_KEY="sk-ant-..."
Step 2: Create the Agent and Environment 
from anthropic import Anthropic

client = Anthropic()

# Create the agent definition
agent = client.beta.agents.create(
    name="Code Analyst",
    model="claude-sonnet-4-6",
    system="""You are a senior code analyst. When given a
repository or codebase, you:
1. Analyze architecture and patterns
2. Identify potential bugs and security issues
3. Suggest improvements with specific examples
4. Produce a structured markdown report""",
    tools=[
        {"type": "agent_toolset_20260401"},
    ],
)
print(f"Agent ID: {agent.id}")

# Create the environment
environment = client.beta.environments.create(
    name="analysis-env",
    config={
        "type": "cloud",
        "networking": {"type": "unrestricted"},
    },
)
print(f"Environment ID: {environment.id}")
Step 3: Start a session with a task 
# Start the session
session = client.beta.sessions.create(
    agent=agent.id,
    environment_id=environment.id,
    title="Analyze my-app repository",
)
print(f"Session ID: {session.id}")

# Send the task
client.beta.sessions.send_event(
    session_id=session.id,
    event={
        "type": "user_message",
        "content": """Clone https://github.com/myorg/my-app
and produce a full code quality report.
Focus on: architecture, error handling,
test coverage gaps, and security.""",
    },
)
Step 4: Stream the results 
# Watch the agent work in real-time
with client.beta.sessions.stream(
    session_id=session.id
) as stream:
    for event in stream:
        if event.type == "agent_message":
            print(event.content)
        elif event.type == "tool_use":
            print(f"[Tool: {event.name}]")
        elif event.type == "session_complete":
            print("Done!")
            break

That's it. Four API calls: create agent, create environment, start session, stream events. The agent runs on Anthropic's infrastructure, clones the repo, analyzes the code, runs commands, and produces a report — all without your laptop being involved.

Environments: Your Agent's Workspace

The environment is where your agent actually runs. It's a cloud container that you configure with:

  • Pre-installed packages — Python, Node.js, Go, and standard dev tools come ready to use
  • Network access — control whether the agent can reach the internet (unrestricted, restricted, or no network)
  • Mounted files — upload files the agent needs (config files, datasets, reference docs)
# Environment with specific configuration
environment = client.beta.environments.create(
    name="secure-analysis-env",
    config={
        "type": "cloud",
        "networking": {"type": "restricted",
                        "allowed_domains": [
                            "github.com",
                            "api.github.com",
                            "pypi.org",
                        ]},
    },
)

Environment Reuse

You create an environment once and reference it by ID in every session. Think of it like a Docker image: define it, then spin up as many containers (sessions) as you need. Don't create a new environment for every task — reuse them.

Sessions and Events: The Lifecycle of Work

A session is a single unit of work. Here's the lifecycle:

  1. Create — combine an Agent ID + Environment ID, give it a title
  2. Send a message — tell the agent what to do via a user event
  3. Stream events — watch progress via SSE (or poll for status)
  4. Complete — the agent finishes and the session enters a terminal state

Events flow in two directions:

  • You → Agent: user_message events to send tasks and follow-up instructions
  • Agent → You: agent_message, tool_use, tool_result, and session_complete events

The key insight: sessions persist even if your connection drops. You can start a session, close your laptop, come back an hour later, and retrieve all the events that happened while you were away.

Managed Agents API Quick Reference

POST /v1/agentsCreate an agent definition
POST /v1/environmentsCreate an environment
POST /v1/sessionsStart a new session
POST /v1/sessions/{id}/eventsSend a user event
GET /v1/sessions/{id}/streamStream events via SSE
GET /v1/sessions/{id}/eventsRetrieve all events (for debugging)
GET /v1/sessions/{id}Check session status

Built-in Tools: What Your Agent Can Do

When you pass {"type": "agent_toolset_20260401"} as a tool, your agent gets access to a full suite of pre-built capabilities:

💻

Bash

Run shell commands in the container. Install packages, clone repos, run scripts, process data.

📁

File Operations

Read, write, edit, and search files. Full filesystem access within the sandboxed environment.

🔍

Web Search

Search the web for documentation, Stack Overflow answers, or any reference material the agent needs.

🌐

Web Fetch

Download files, fetch API responses, or read web pages directly.

🔌

MCP Servers

Connect to any MCP-compatible server for integrations: databases, APIs, Slack, GitHub, and more.

The agent_toolset_20260401 bundles all of these into a single tool type. You don't need to configure each one individually — just enable the toolset and the agent picks the right tool for each step.

Before / After: The Power of Fire-and-Forget

Let's make this concrete. Here's a real task: reviewing a large PR with 47 changed files across 12 modules.

Before: Manual Review

  1. Open Claude Code in your terminal
  2. Paste the PR diff (hope it fits in context)
  3. Wait 5-10 minutes for analysis
  4. Ask follow-up questions
  5. Wait again for each response
  6. Manually copy results into PR comments
  7. Your laptop is busy the whole time
  8. Total: ~30 minutes of your time

After: Managed Agent

  1. Trigger agent via API (or CI/CD hook)
  2. Agent clones the repo, checks out the branch
  3. Runs tests, analyzes diff, checks for issues
  4. Produces a structured review report
  5. Posts comments directly to the PR via MCP
  6. You get a notification when it's done
  7. Your laptop is free the whole time
  8. Total: ~2 minutes of your time

The agent does the same quality of work. The difference is that you're not waiting around for it. You kicked it off and went to lunch. Or started another task. Or went home for the day.

Practical Use Cases for Power Users

Here are the patterns that get the most value from Managed Agents:

1. Automated Code Review

Trigger an agent on every PR. It clones the repo, runs the test suite, analyzes the diff for bugs, security issues, and style violations, then posts a review comment. Your team gets instant, thorough feedback on every PR without anyone spending time on the first pass.

2. Scheduled Reports

Set up a daily agent that analyzes your codebase: test coverage trends, dependency freshness, TODO count, dead code detection. The report lands in Slack every morning before standup.

3. Research Tasks

Need to evaluate 5 different logging libraries for your project? Spin up an agent with web access. It researches each library, tests compatibility with your stack, and delivers a comparison matrix.

4. Data Processing Pipelines

Got 500 CSV files that need cleaning, transforming, and loading? An agent can bash-script its way through the batch while you focus on the business logic.

Power Move: Chain Agents Together

Don't limit yourself to one agent per task. Use a "coordinator" script that launches multiple agents in parallel — one to analyze backend code, one for frontend, one for infrastructure — and aggregates their reports. You're not just delegating to one colleague; you're managing a whole team of specialists. Each agent gets its own session, its own context, and its own tools. The coordinator waits for all sessions to complete, then merges the results.

Let's do some real math:

  • A 30-minute code review with moderate token usage: ~$0.04 runtime + ~$0.30 tokens = ~$0.34 total
  • A daily automated report running 15 minutes: ~$0.02 runtime + ~$0.15 tokens = ~$5/month
  • A deep 2-hour analysis with heavy token usage: ~$0.16 runtime + ~$2.00 tokens = ~$2.16 total

For context, the $0.08/hour runtime fee is essentially free compared to the token costs. The real expense is tokens, just like the regular API. The runtime fee is Anthropic's charge for hosting the container, filesystem, and tools.

Cost Optimization Tips

  • Default to Sonnet 4.6 for most agent tasks — fast, capable, and 5× cheaper than Opus per token
  • Reach for Opus 4.7 when the loop is long-horizon, deeply agentic, or hits failure modes Sonnet can't recover from — Anthropic recommends starting at high or xhigh effort for these
  • Write focused system prompts so the agent doesn't waste tokens on irrelevant analysis
  • Use restricted networking to prevent the agent from wandering into unnecessary web searches
  • Reuse agent and environment definitions — don't recreate them for every session

Connecting MCP Servers

One of the most powerful features is connecting MCP servers to your agent. This lets it interact with external services — GitHub, Slack, databases, your internal tools — without you building custom integrations.

# Agent with MCP server for GitHub integration
agent = client.beta.agents.create(
    name="PR Reviewer",
    model="claude-sonnet-4-6",
    system="You review PRs and post comments directly.",
    tools=[
        {"type": "agent_toolset_20260401"},
    ],
    mcp_servers=[
        {
            "type": "url",
            "url": "https://mcp.github.com/sse",
            "token": "ghp_your_token_here",
        }
    ],
)

With GitHub MCP connected, your agent can read PRs, post review comments, check CI status, and even merge — all autonomously. Combine this with a CI/CD trigger and you have a fully automated review pipeline.

Real-World Pattern: The Nightly Audit Agent

Let's put it all together with a practical example. Here's a script that runs a nightly code quality audit:

#!/usr/bin/env python3
"""Nightly code audit agent - run via cron or CI scheduler."""

from anthropic import Anthropic

client = Anthropic()

# Reuse existing agent and environment (create once, use forever)
AGENT_ID = "agent_01J..."   # your saved agent ID
ENV_ID = "env_01J..."       # your saved environment ID

# Start the audit session
session = client.beta.sessions.create(
    agent=AGENT_ID,
    environment_id=ENV_ID,
    title=f"Nightly audit",
)

# Send the task
client.beta.sessions.send_event(
    session_id=session.id,
    event={
        "type": "user_message",
        "content": """
        Clone https://github.com/myorg/my-app (main branch).
        Run the full test suite and record results.
        Analyze: code coverage, lint warnings, TODO/FIXME count,
        dependencies with known vulnerabilities (npm audit),
        and files changed in the last 24 hours.
        Output a markdown report to /workspace/audit-report.md.
        """,
    },
)

print(f"Audit started: session {session.id}")
print("Results will be available when the session completes.")

Schedule this with cron, GitHub Actions, or any task scheduler. The agent runs, produces the report, and you check results in the morning. Zero babysitting required.

Try It Yourself: Build a Code Quality Agent

Create a Managed Agent that analyzes a GitHub repository and produces a code quality report. Here's your step-by-step plan:

  1. Set up: Install the Anthropic Python SDK (pip install anthropic) and set your ANTHROPIC_API_KEY
  2. Create the agent: Use the code from "Creating Your First Agent" above. Customize the system prompt for the type of analysis you want.
  3. Create the environment: Use "networking": {"type": "unrestricted"} so the agent can clone from GitHub.
  4. Start a session: Point it at a public GitHub repository you maintain (or any open-source repo).
  5. Stream the events: Watch the agent clone, analyze, and report in real-time.
  6. Review the report: Check the agent's output. How does it compare to your own manual review?

Bonus challenge: Modify the script to post the report as a GitHub issue or Slack message using an MCP server.
