Chapter 10

Claude for Debugging

From "it's broken" to "here's the fix" — systematic AI-assisted debugging

In the last chapter, you learned to generate documentation that keeps your team moving fast. Now let's tackle the activity that consumes more developer time than almost anything else: debugging.

Most developers debug the same way: stare at the code, add some print statements, stare some more, Google the error, find a Stack Overflow answer from 2019, try it, break something else. Repeat.

Claude changes this. Instead of staring and guessing, you have an assistant that can read your entire codebase, trace execution paths across files, understand error messages in context, and suggest fixes backed by actual code analysis. Let's build a debugging workflow that gets you to root cause faster.

The Debugging Mindset

Before we talk about tools and prompts, let's establish the key principle: how you present the problem determines how useful Claude's help will be.

Bad Bug Report to Claude

"My app crashes. Fix it."

Claude has nothing to work with. It'll ask 10 clarifying questions before it can help.

Good Bug Report to Claude

"The app crashes with a NullPointerException in UserRepository.getUser() when the user has no profile photo. Here's the stack trace: [paste]. This started after commit a1b2c3."

Claude can immediately trace the issue and suggest a fix.

The Bug Report Template

When you bring a bug to Claude, include as much of this as you can:

  • What happened: The actual behavior (crash, wrong output, unexpected state)
  • What you expected: The correct behavior
  • Error message/stack trace: The full text, not a summary
  • How to reproduce: Steps, input data, environment
  • When it started: Recent commit, deployment, dependency update
  • What you've tried: So Claude doesn't repeat your failed attempts

You don't need all of these every time. But the more context you give, the faster Claude gets to root cause.

Pattern 1: Stack Traces and Error Messages

The simplest debugging pattern: paste the error and ask Claude to decode it. But how you paste matters.

Good: Full context

I'm getting this crash on Android 14 when the user
taps "Save Profile" with no photo selected:

java.lang.NullPointerException:
  at com.myapp.data.UserRepository.getUser(UserRepository.kt:47)
  at com.myapp.ui.ProfileViewModel.saveProfile(ProfileViewModel.kt:83)
  at com.myapp.ui.ProfileScreen$SaveButton.onClick(ProfileScreen.kt:142)

This works fine when a photo IS selected. It started
after I added the profile photo feature in commit a1b2c3.

Read UserRepository.kt and ProfileViewModel.kt to find the root cause.

Bad: Minimal context

NullPointerException in UserRepository line 47. Fix it.

Claude might fix the symptom (add a null check) without understanding the root cause (the photo URL should default to empty string, not null).

The key difference: the good prompt tells Claude when it happens (no photo selected), when it started (specific commit), and asks it to read the actual files. Claude can trace the execution path and find the real problem.

Pattern 2: The "Explain This Error" Technique

Some error messages are genuinely cryptic. Instead of Googling, ask Claude to translate:

You:

Explain this error in plain English. What's causing it and how do I fix it?

Error: Compose runtime: Remember observers count is inconsistent. Expected 3, got 2. This typically indicates a composition was restarted...

Claude: This means a Composable function is being recomposed, but the number of remember calls changed between compositions. This usually happens when you have a remember inside a conditional block (if/else) — Compose expects the same number of remember calls every time.

Fix: Move the remember call outside the conditional, or wrap the entire conditional block in a key() composable to reset state when the condition changes.

Claude doesn't just translate the error — it explains the underlying concept and gives you a concrete fix. This is especially valuable when working in a language or framework you're less familiar with.

Pattern 3: Rubber Duck Debugging

Rubber duck debugging is the classic technique: explain the problem out loud and the solution often reveals itself. Claude is the ultimate rubber duck — one that actually responds with insights.

The magic of rubber duck debugging with Claude: Claude serves as an intelligent rubber duck that can spot where your mental model diverges from the code's actual behavior. It's especially effective for bugs where the code "looks right" but doesn't work — the act of explaining often reveals the issue even before Claude responds.

The rubber duck prompt:

I'm going to walk you through a bug. Don't try to fix
it yet — just listen and ask clarifying questions.

The user taps "Sync" and nothing happens. No error, no
crash, just... nothing. The sync button's onClick calls
viewModel.sync(), which calls repository.syncData(),
which should call the API and update the database.

I've verified the onClick fires (added a log). The
ViewModel method is called. But the repository method
seems to return immediately without doing anything.

What should I check next?

By telling Claude "don't fix it yet — ask questions," you force a dialogue that helps you think through the problem systematically instead of jumping to a premature fix.

Power Tip: The Reverse Explanation

Ask Claude to explain what your code actually does, step by step, without seeing the bug report. Then compare its explanation with what you intended the code to do. Where they diverge, you've found the bug. This works because Claude reads the code literally — it sees what the code does, not what you think it does. Example: "Read syncData() in DataRepository.kt and explain exactly what happens when it's called, step by step." If Claude says "it returns early when isOffline is true," and you didn't know about that early return, there's your bug.

Pattern 4: Log Analysis

When you have log files full of data, Claude can spot patterns that would take you hours to find manually:

Log Analysis Prompts

  • Analyze this log file. Find any errors, warnings, or patterns that suggest a problem.
  • This API started timing out at 2pm. Here are the logs from 1pm to 3pm. What changed?
  • Compare these two log files — one from a working session and one from a broken session. What's different?
  • Find all unique error messages in this log and group them by root cause.

Claude is especially good at finding temporal patterns: "errors started 3 minutes after the database connection pool maxed out" or "this timeout always follows a garbage collection pause."

Large log files: If a log file is too large for context, ask Claude to filter first: "Read server.log and extract only lines containing ERROR or WARN from the last hour." This keeps the context focused on relevant data.

Pattern 5: The Git Bisect Approach

"It was working last week" is one of the most common debugging scenarios. Something changed, but you don't know which commit broke it. Claude + git bisect is a powerful combination:

The prompt:

The login feature was working in v2.3.0 but is broken on the current branch. Help me find which commit broke it. Run git bisect to narrow down the breaking commit, then analyze what changed.

Claude runs git bisect start, marks the known good and bad commits, and systematically tests each midpoint. Once it finds the breaking commit, it reads the diff and explains exactly what change caused the regression.

1. Start

Identify the last known good commit and the first known bad commit

→

2. Bisect

Claude runs git bisect, testing each midpoint

→

3. Found It

The breaking commit is identified

→

4. Analyze

Claude reads the diff and explains the root cause

This binary search approach is far faster than reading through 50 commits manually. Claude handles the mechanical work; you just tell it how to verify if the bug is present.

Pattern 6: Reproducing Bugs

Can't reproduce a bug? Claude can help generate minimal reproduction cases:

Generating a minimal repro:

Users report a crash when they enter special characters
in the search field. I can't reproduce it. Read the
SearchViewModel and SearchRepository code and:

1. Identify which special characters could cause issues
2. Generate a list of test inputs that would trigger
   the bug (SQL chars, Unicode, emoji, etc.)
3. Write a unit test that reproduces the crash
4. Suggest a fix

Claude will analyze the search logic, find where input isn't properly escaped or validated, generate specific test cases that trigger the bug, and provide both a test and a fix.

Pattern 7: Performance Debugging

Performance bugs are the trickiest — no crash, no error, just "it's slow." Claude can help analyze profiler output and identify bottlenecks:

The prompt:

The app takes 8 seconds to load the user list. Here's the output from Android Studio Profiler showing the top 10 methods by CPU time. What's causing the slowdown and how do I fix it?

getUsersList() — 6.2s
mapToUiModel() — 4.8s
formatDate() — 4.1s
SimpleDateFormat.parse() — 3.9s

Claude: The bottleneck is SimpleDateFormat.parse() being called inside mapToUiModel() for every user. SimpleDateFormat is notoriously slow, and you're creating a new instance per call.

Fix: Replace SimpleDateFormat with java.time.format.DateTimeFormatter (thread-safe, faster) and cache the formatter instance. For 1000 users, this should drop from 4 seconds to ~50ms.

What Claude Catches

  • N+1 database queries
  • Object allocation in hot loops
  • Blocking calls on the main thread
  • Missing indexes on queries
  • Unnecessary recompositions

What You Still Need

  • Actual profiler measurements
  • Real-device testing
  • Production-scale data volumes
  • Memory leak detection tools
  • Network latency testing

Debugging Across Languages

If your project spans multiple languages (Kotlin + TypeScript, Python + SQL, etc.), Claude handles cross-language debugging better than most humans:

Kotlin/Java
Null safety, coroutine issues, Android lifecycle, Compose recomposition
TypeScript
Type narrowing, async/await pitfalls, React state, undefined vs null
Python
Import issues, async generators, type coercion, dependency conflicts
SQL
Query plans, index usage, join performance, migration issues

Claude can trace a bug from your Kotlin ViewModel through a Retrofit API call to a backend TypeScript handler to a SQL query — following the data flow across languages and network boundaries.

When Claude Gets It Wrong

AI debugging isn't infallible. Here's how to spot and handle Claude's mistakes:

Debugging hallucination red flags:
  • Claude suggests a fix for a function that doesn't exist in your codebase
  • The explanation references API behavior that doesn't match the documentation
  • Claude proposes using a library method with wrong parameters
  • The root cause analysis sounds plausible but doesn't match the stack trace

Rule of thumb: Always verify Claude's diagnosis before applying the fix. Read the code Claude references. Check that the proposed fix addresses the actual root cause, not just the symptom.

The verification prompt:

Before I apply this fix, verify your analysis:
1. Read the actual file at the line you referenced
2. Confirm the variable/function you mentioned exists
3. Trace the execution path to make sure your root
   cause analysis is correct
4. Explain why the fix won't break anything else

Building a Debugging Skill

Create a structured debugging command for consistent, thorough bug analysis:

File: .claude/commands/debug.md 
Investigate and fix a bug based on the user's description.

## Process
1. **Understand:** Read the error/description carefully.
   Ask clarifying questions if anything is ambiguous.
2. **Locate:** Find the relevant code. Read files mentioned
   in stack traces. Trace the execution path.
3. **Diagnose:** Identify the root cause (not just the
   symptom). Explain WHY the bug happens.
4. **Verify:** Confirm the diagnosis by reading surrounding
   code. Check that your understanding matches reality.
5. **Fix:** Implement the minimal fix. Don't refactor
   unrelated code.
6. **Test:** Write a test that reproduces the bug BEFORE
   the fix, and passes AFTER.
7. **Explain:** Summarize: what was wrong, why, and what
   you changed.

## Rules
- Always read the actual code — never guess from memory
- Prefer root cause fixes over symptom patches
- A null check is rarely the right fix — find why
  the value is null in the first place
- If the bug involves multiple files, trace the full
  execution path before suggesting a fix

The user's bug report: $ARGUMENTS

Now /debug the app crashes when saving with no photo triggers a structured investigation every time.

Debugging Prompts That Work

Copy-Paste Debugging Prompts

  • Explain this error in plain English: [paste error]
  • Read [file] and explain what actually happens at line N, step by step
  • This worked before commit [hash]. Read the diff and find what broke
  • Compare these two log outputs — one works, one doesn't. What's different?
  • This code looks correct but produces wrong output. Read it and find the logical error
  • Generate test inputs that would expose edge cases in [function]

Try It Yourself: The Debugging Workflow

  1. Error translation: Find a cryptic error message from your project (check recent CI failures or crash logs). Ask Claude to explain it in plain English and suggest a fix.
  2. Reverse explanation: Pick a function in your project that's longer than 30 lines. Ask Claude to explain what it does step by step. Compare the explanation to what you intended — is there a discrepancy?
  3. Log analysis: If you have log files available, feed Claude a section and ask it to identify patterns or anomalies.
  4. Build the skill: Create the /debug command from this chapter. Use it on a real bug from your backlog.
  5. Bonus: Take a real bug you're currently stuck on and work through it with Claude using the rubber duck approach. Tell Claude not to fix it yet — just ask questions. See if the dialogue helps you find the root cause.
