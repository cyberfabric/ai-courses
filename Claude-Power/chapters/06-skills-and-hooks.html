Chapter 6

Custom Skills & Hooks

Reusable slash commands and automated triggers — extend Claude Code to fit your workflow

In the last chapter, you taught Claude what to know about your project with CLAUDE.md and memory. Now it's time to teach Claude what to do — automatically, consistently, and with a single command.

Think of it this way: CLAUDE.md is the employee handbook. Skills are the standard operating procedures. Hooks are the automatic quality checks that run without anyone asking.

Together, they turn Claude Code from a smart assistant into a customized development machine that follows your workflows.

Part 1: Skills (Custom Slash Commands)

You already know slash commands like /help, /commit, and /review. Skills let you create your own. Type /my-command, and Claude follows a reusable prompt template you wrote — same quality, same format, every single time.

Without Skills

You type the same long prompt every time you want to generate tests:

"Write unit tests for this file using JUnit 5 and MockK. Follow the should_expected_when_condition naming pattern. Mock all external dependencies. Include edge cases..."

With a Custom Skill

You type one command and get the same quality every time:

/generate-tests UserRepository.kt

Where Skills Live

Skills are Markdown files stored in specific directories. Where you put them determines their scope:

Location Scope Use Case
.claude/commands/ This project (shared via git) Team skills: project-specific workflows
.claude/skills/*/SKILL.md This project (shared via git) Rich skills with supporting files
~/.claude/commands/ All your projects (personal) Your personal toolkit
Commands vs. Skills: Files in .claude/commands/ and .claude/skills/ both create slash commands. The skills/ format is newer and supports additional features like supporting files (templates, scripts), frontmatter configuration, and dynamic context. New skills should use the .claude/skills/ format. Existing commands still work perfectly.

Creating Your First Skill (Simple Command)

The simplest skill is a Markdown file in .claude/commands/. The filename becomes the command name.

File: .claude/commands/generate-tests.md 
Write comprehensive unit tests for the file or class
the user specifies.

## Requirements
- Use JUnit 5 and MockK
- Follow naming: `should_expectedBehavior_when_condition`
- Mock all external dependencies (repositories, APIs)
- Include: happy path, edge cases, error cases
- Place tests in the matching test/ directory
- Run the tests after generating them

## Context
- Read the source file first to understand all public methods
- Check existing tests to match the project's testing style
- Use @MockK for mocks, @InjectMockKs for the class under test

The user will specify which file to test as: $ARGUMENTS

Now you type /generate-tests UserRepository.kt and Claude follows these instructions exactly. The $ARGUMENTS variable gets replaced with whatever you type after the command name.

Quick Reference: Skill Variables

  • $ARGUMENTS — everything the user types after the command name
  • The filename (minus .md) becomes the slash command: generate-tests.md → /generate-tests
  • Subdirectories create namespaced commands: commands/android/lint.md → /android:lint

Creating a Rich Skill (SKILL.md Format)

For skills that need supporting files, templates, or advanced configuration, use the .claude/skills/ directory with a SKILL.md file:

File: .claude/skills/review-pr/SKILL.md 
---
description: Review a pull request for bugs, security, and style
argument-hint: PR number or branch name
allowed-tools:
  - Bash
  - Read
  - Grep
  - Glob
---

Review the PR specified by $ARGUMENTS.

## Steps
1. Run `git diff main...$ARGUMENTS` to see all changes
2. Read each changed file in full context
3. Check for:
   - Bugs and logic errors
   - Security vulnerabilities (SQL injection, XSS, etc.)
   - Performance issues (N+1 queries, unnecessary allocations)
   - Code style violations per CLAUDE.md
4. Provide feedback in this format:
   - 🔴 **Critical:** must fix before merge
   - 🟡 **Suggestion:** would improve the code
   - 🟢 **Nice:** things done well

## Rules
- Be specific: reference file names and line numbers
- Suggest fixes, not just problems
- Don't nitpick formatting — focus on substance

Frontmatter Fields Explained

Field Purpose Example
description What the skill does (helps Claude decide when to auto-load) "Generate unit tests for a file"
argument-hint Hints at what argument to pass "file path or class name"
allowed-tools Which tools Claude can use during this skill [Bash, Read, Edit]
paths Only activate when working with matching files ["src/**/*.kt"]
disable-model-invocation Only run when explicitly called (never auto-loaded) true

Practical Skill Examples

Here are five skills that cover the most common developer workflows:

📝
/daily-standup
Summarizes your git activity from the last 24 hours into a standup-ready format. Reads git log, groups by feature, and generates "Yesterday I... Today I plan to..." format.
🔍
/review-pr
Reviews a PR for bugs, security issues, and style. Takes a branch name or PR number as an argument. Outputs categorized feedback.
♻️
/refactor
Refactors a file or function while preserving behavior. Reads the code, identifies improvement opportunities, makes changes, and runs tests to verify nothing broke.
📖
/explain
Explains a file or function in plain English. Covers what it does, why it exists, how it connects to the rest of the codebase, and common gotchas.
🚀
/release-notes
Generates human-readable release notes from git history between two tags. Groups changes by category (features, fixes, breaking changes).

Power Tip: Build a Standup Skill in 2 Minutes

Create .claude/commands/daily-standup.md with this content:

Summarize my git activity from the last 24 hours.

1. Run `git log --oneline --author="$(git config user.name)" --since="24 hours ago"`
2. Group commits by feature/area
3. Format as:
   **Yesterday:** [bullet list of what was done]
   **Today:** [suggest next logical steps based on recent work]
   **Blockers:** [note any failing tests or unresolved conflicts]

Keep it concise — this goes in a Slack standup channel.

Now /daily-standup gives you a formatted standup in seconds. Share this with your team by committing it to the repo.

Writing Good Skill Instructions

Good: Be directive and specific

Read the file at $ARGUMENTS.
Identify all public methods without unit tests.
For each untested method, generate a test that covers:
- The happy path with typical inputs
- At least one edge case
- Error handling if the method can throw
Run the tests and fix any failures.

Bad: Vague and conversational

Hey Claude, could you maybe look at this file and
write some tests if you think it needs them? Use
whatever testing framework seems appropriate. Thanks!

Skills are instructions, not conversations. Use imperative verbs: "Read the file", "Generate tests", "Run lint." Be specific about format, tools, and expected output.

Part 2: Hooks (Automated Triggers)

If skills are things you ask Claude to do, hooks are things that happen automatically. They're shell commands that run in response to specific Claude Code events — like a Git hook, but for Claude's actions.

How Hooks Work

Event fires (file write, command, etc.) → Your shell command runs → Format, block, notify, log…

Hook Events

Hooks aren’t limited to PreToolUse / PostToolUse. Claude Code now exposes around 30 lifecycle events grouped into three cadences: once per session, once per turn, and once per tool call (plus a handful of asynchronous events for things like compaction and worktrees). You don’t need to know all of them — here are the ones you’ll actually reach for:

Event Cadence Can Block? Common Use
SessionStart Per session No Auto-load context (read git status, inject reminders)
SessionEnd Per session No Persist transcripts, ship logs, cleanup
UserPromptSubmit Per turn Yes Add hidden context to the prompt; redact secrets
UserPromptExpansion Per turn Yes Modify a slash command after it expands into a prompt
Stop Per turn Yes Run final checks before Claude is allowed to stop
PreToolUse Per tool call Yes (exit 2) Validate commands, block dangerous operations
PostToolUse Per tool call Yes Auto-format, lint, log actions
SubagentStop Per tool call Yes Validate or filter what a subagent returns
PreCompact Async Yes Save state right before context is compacted
Notification Async No Route alerts to Slack, desktop, logging

The full reference (including StopFailure, PostToolBatch, PermissionRequest, InstructionsLoaded, WorktreeCreate, and friends) lives in the official hooks docs. Most teams only ever wire up two or three of these, so don’t feel obligated to learn them all.

Configuring Hooks

Hooks live in your settings files. Here's the JSON format in .claude/settings.json:

Example: Auto-format files after Claude edits them 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\""
          }
        ]
      }
    ]
  }
}

Every time Claude edits or writes a file, Prettier auto-formats it. No asking, no forgetting — it just happens.

The Matcher Filter

The matcher field is a regex that filters which tools trigger the hook. Common matchers: Bash (shell commands), Edit|Write (any file modification), Read (file reads, for auditing), .* (every tool — use carefully).

Practical Hook Examples

1. Auto-Format on Edit (PostToolUse)

Ensure every file Claude touches is properly formatted:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\" 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}

The || true ensures the hook doesn't fail for non-supported file types.

2. Block Dangerous Commands (PreToolUse)

Prevent Claude from running destructive operations:

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CLAUDE_TOOL_INPUT_COMMAND\" | grep -qE 'rm -rf|drop table|force push'; then exit 2; fi"
          }
        ]
      }
    ]
  }
}

Exit code 2 tells Claude Code to block the action. Claude gets told the hook rejected the command and can try a safer alternative.

3. Run Lint After File Changes (PostToolUse)

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "if [[ \"$CLAUDE_TOOL_INPUT_FILE_PATH\" == *.kt ]]; then ./gradlew lintDebug --quiet 2>/dev/null; fi"
          }
        ]
      }
    ]
  }
}

4. Desktop Notification on Completion (Notification)

{
  "hooks": {
    "Notification": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "notify-send 'Claude Code' \"$CLAUDE_NOTIFICATION_MESSAGE\""
          }
        ]
      }
    ]
  }
}

On macOS, replace notify-send with osascript -e 'display notification \"$CLAUDE_NOTIFICATION_MESSAGE\" with title \"Claude Code\"'

Hook performance matters. Hooks run synchronously — a slow hook blocks Claude from proceeding. Keep hook commands fast (under 2 seconds). If you need something slower like a full test suite, consider running it asynchronously or only on specific file patterns.

Blocking vs. Non-Blocking Hooks

PreToolUse exit codes: 0 allows the action (proceed normally), 2 blocks it (Claude is told the hook rejected it and can try an alternative), and any other code is treated as a hook error (logged, but the action still proceeds).

Beyond Skills and Hooks: Subagents

Skills and hooks are two of the three first-class extension points in Claude Code. The third is subagents — specialized workers Claude can delegate to that run in their own context window. Built-in subagents include Explore (Haiku-powered, read-only codebase search), Plan (research-only, used by plan mode), and general-purpose (full tool access). You can also drop your own Markdown files into .claude/agents/ — same shape as a SKILL.md, but the body becomes a system prompt for an isolated worker.

The mental model: Skills run a procedure when you call them. Hooks run on events. Subagents run in their own conversation, returning only a summary — great for fan-out work like running three reviewers in parallel without flooding your main context. Subagents are a topic in their own right; this chapter stays scoped to skills and hooks.

Combining Skills and Hooks

The real power comes from using skills and hooks together. Here's a workflow that shows how:

/feature — skill scaffolds branch, files, tests → You code — hook auto-formats every edit → /review-pr — skill reviews changes → /commit — hook lints before commit

Skills handle the big, intentional workflows. Hooks handle the small, automatic quality checks. Together, they create a pipeline where quality is built in, not bolted on.

Debugging Skills and Hooks

Things will go wrong. Here's how to fix them:

  • Skill not showing up? Check the filename and directory. It must be in .claude/commands/ or .claude/skills/*/SKILL.md. Filename becomes the command name.
  • Hook not firing? Run /status to verify hooks are loaded. Check that the matcher regex matches the tool name exactly.
  • Hook running but not working? Test the command manually in your terminal first. Environment variables like $CLAUDE_TOOL_INPUT_FILE_PATH are only set during hook execution.
  • Hook blocking when it shouldn't? Check your exit codes. Only exit code 2 blocks — other non-zero codes are treated as errors but don't block.
  • Hook too slow? Add 2>/dev/null to suppress stderr noise and ensure commands are scoped to relevant file types only.
Quick test: Claude Code watches your settings files for changes. If you edit .claude/settings.json while Claude is running, the hooks update automatically — no restart needed.

Sharing Skills With Your Team

One of the best things about skills in .claude/commands/ or .claude/skills/: they're just files in your repo. Commit them to git, and every team member gets the same slash commands.

Recommended: Document your skills in CLAUDE.md

## Available Skills
- `/generate-tests [file]` — Generate unit tests for a file
- `/review-pr [branch]` — Review a PR for bugs and style
- `/daily-standup` — Summarize git activity for standup
- `/explain [file]` — Explain what a file does in plain English
- `/release-notes [tag1] [tag2]` — Generate release notes between tags

This tells both Claude and your teammates what's available.

Personal skills go in ~/.claude/commands/ — they travel with you across projects but aren't shared with your team.

Try It Yourself: Build Your First Skill and Hook

Part 1: Create a skill

  1. Create .claude/commands/daily-standup.md in your project
  2. Write instructions to summarize your last 24 hours of git activity
  3. Include the $ARGUMENTS variable so you can optionally specify a time range
  4. Test it: type /daily-standup in Claude Code

Part 2: Add a hook

  1. Open .claude/settings.json (create it if it doesn't exist)
  2. Add a PostToolUse hook that auto-formats files after Claude edits them (use Prettier, ktlint, or your project's formatter)
  3. Edit a file with Claude and verify the formatter runs automatically

Bonus: Create a PreToolUse hook that blocks rm -rf commands. Test it by asking Claude to delete a temporary directory with rm -rf — the hook should block it.
