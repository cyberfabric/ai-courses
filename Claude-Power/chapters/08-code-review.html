Chapter 8

Claude for Code Review

From quick feedback to automated PR pipelines — catch bugs before they ship

In the last chapter, you learned to call Claude from code via the API. Now let's put that power — and everything else you've learned — to work on the one task every developer does but nobody loves: code review.

Code review is critical. It catches bugs, enforces standards, and spreads knowledge across the team. But it's also slow, inconsistent, and often superficial. Reviewers skim 500-line diffs, miss edge cases, and rubber-stamp with "LGTM" because they're busy.

Claude doesn't get busy. Claude doesn't skim. And Claude is available at 2 AM when you want feedback on that late-night fix before pushing it. Let's build a review workflow that combines human judgment with AI thoroughness.

The Quick Review: /review

The fastest way to get a code review is the built-in /review command in Claude Code. It analyzes your current changes and gives you instant feedback:

Instant Code Review

  • /review — Review all uncommitted changes in your working directory
  • Review the changes in feature/login branch — Review a specific branch's diff against main
  • Review my last 3 commits — Review recent commits

That's it. Claude reads the diff, understands the context from your codebase, and gives you categorized feedback. No setup, no configuration.

Self-Review Without Claude

You re-read your own code. Your brain auto-corrects mistakes because you wrote it 10 minutes ago. You miss the off-by-one error in that loop. You push. CI fails.

Self-Review With Claude

You type /review. Claude spots the off-by-one error, a potential null pointer, and a missing edge case in your error handling. You fix all three before pushing. CI passes.

What to Ask Claude to Look For

A generic "review this code" gets generic results. Power users tell Claude exactly what to focus on. Here are the review angles that catch real bugs:

Logic
Off-by-one errors, wrong conditions, missing branches, state management bugs
Security
SQL injection, XSS, insecure auth, exposed secrets, missing input validation
Edge Cases
Null/empty inputs, concurrency, timeout handling, large data sets
Performance
N+1 queries, unnecessary allocations, missing caching, blocking main thread

Prompt Templates for Different Review Types

Security-focused review: 
Review this diff for security vulnerabilities only.
Focus on:
- Input validation and sanitization
- Authentication and authorization checks
- Data exposure (logs, error messages, API responses)
- Injection risks (SQL, command, XSS)
- Secrets or credentials in code
- Insecure cryptography or random number generation

For each finding, explain:
1. What the vulnerability is
2. How it could be exploited
3. How to fix it

Ignore style, naming, and performance issues.
Performance-focused review: 
Review this code for performance issues only.
Focus on:
- Database queries inside loops (N+1 problem)
- Unnecessary object allocations in hot paths
- Missing indexes or inefficient queries
- Blocking operations on the main thread
- Unbounded collections or memory leaks
- Missing pagination for list endpoints

For each finding, estimate the impact (low/medium/high)
and suggest a concrete fix.

Building a Custom Review Skill

In Chapter 6, you learned to create skills. A review skill is one of the most valuable you can build:

File: .claude/commands/review-pr.md 
Review the current branch's changes against main.

## Steps
1. Run `git diff main...HEAD --stat` to see which files changed
2. Run `git diff main...HEAD` to see the full diff
3. Read each changed file in full (not just the diff) for context
4. Read CLAUDE.md for project conventions

## What to Check
- **Bugs:** Logic errors, off-by-one, null safety, missing error handling
- **Security:** Input validation, auth checks, data exposure
- **Performance:** N+1 queries, blocking calls, unnecessary allocations
- **Conventions:** Violations of CLAUDE.md rules
- **Tests:** Are new code paths tested? Are edge cases covered?

## Output Format
### Critical (must fix)
- [file:line] Description of issue + suggested fix

### Suggestions (should fix)
- [file:line] Description + why it matters

### Good (well done)
- [file:line] What was done well and why

Keep it actionable. No generic advice. Reference specific
lines and suggest specific fixes.

Now /review-pr gives your team a consistent, thorough review every single time. Because it's in .claude/commands/, everyone on the team gets the same review quality.

The "Fresh Eyes" Technique

One of the most powerful review patterns doesn't involve PRs at all. It's reviewing your own code before you push — using Claude as a pair of fresh eyes.

The fresh eyes prompt:

I just wrote this code and I'm about to push it.
Review it as if you've never seen this codebase before.

Specifically:
1. What assumptions am I making that might be wrong?
2. What would break if the input is unexpected?
3. Is there a simpler way to do this?
4. What would a new team member find confusing?

Be direct. I'd rather hear about problems now than
in production.

This is especially valuable late at night or when you've been staring at the same code for hours. Claude doesn't have context fatigue — it sees your code exactly as a reviewer would see it for the first time.

Power Tip: The Rubber Duck Review

Ask Claude to explain your code back to you before reviewing it. "Explain what this function does, step by step." If Claude's explanation doesn't match your intent, you've found a bug — the code says something different from what you meant. This catches semantic bugs that traditional reviews miss, because the reviewer (Claude) is working from the code alone, not your mental model of what the code should do.

Automated Review in CI/CD

The ultimate goal: every PR gets reviewed automatically before any human looks at it. Here's how to set that up using Claude Code's --print flag:

GitHub Actions workflow (.github/workflows/claude-review.yml): 
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Review PR
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          DIFF=$(git diff origin/main...HEAD)
          echo "$DIFF" | claude --print \
            "Review this diff for bugs, security issues, and \
             convention violations. Output as markdown. \
             Focus on critical issues only." \
            > review.md

      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            await github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: `## Claude Code Review\n\n${review}`
            });

Every PR now gets an automatic review comment. Human reviewers start with Claude's findings already posted, so they can focus on architecture and design decisions instead of hunting for bugs.

Don't replace human review. AI review catches mechanical bugs, security issues, and convention violations. Human review catches architectural problems, business logic errors, and design decisions. Use both. Claude handles the tedious part so humans can focus on the hard part.

Handling False Positives

Claude will sometimes flag code that's actually fine. This is normal — no reviewer (human or AI) is perfect. Here's how to handle it:

  • Don't dismiss all AI feedback. If Claude flags 10 things and 3 are wrong, the other 7 might save you from a production bug
  • Teach Claude what's intentional. Add architectural decisions to your CLAUDE.md (see Chapter 5): "We use Room instead of SQLDelight (intentional, not a TODO)"
  • Tune your review prompt. If Claude keeps flagging your error handling style, add "Our error handling pattern is X — don't flag this" to the review skill
  • Use severity levels. A good review skill categorizes findings by severity so you can quickly skip low-confidence suggestions
  • Track false positive patterns. If Claude consistently gets something wrong, that's a CLAUDE.md update waiting to happen

Review Checklists: Human + AI

The best review workflow combines AI thoroughness with human judgment. Here's a checklist that covers both:

Claude Reviews

  • Logic errors and edge cases
  • Security vulnerabilities
  • Performance antipatterns
  • Convention violations
  • Missing error handling
  • Test coverage gaps

Humans Review

  • Architecture and design
  • Business logic correctness
  • UX implications
  • Team knowledge sharing
  • Long-term maintainability
  • Is this the right approach?

Reviewing Different Types of Changes

Not all code changes need the same review. Match your prompt to the type of change:

🐛
Bug Fix Review
"Does this fix actually address the root cause? Could it introduce a regression? Is there a test that would have caught this bug originally?"
🚀
New Feature Review
"Are all edge cases handled? Is the API surface clean? Are there sufficient tests? Does it follow existing patterns?"
♻️
Refactor Review
"Is the behavior preserved exactly? Were all callers updated? Are the tests still testing the right things?"
🔒
Security-Sensitive Review
"Are there injection vectors? Is auth properly checked? Are secrets handled correctly? Is input validated at every boundary?"

Team Review Workflows

Once you've built a review skill that works for you, scale it to your team:

  • Commit the review skill to .claude/commands/review-pr.md so everyone uses the same criteria
  • Include project-specific rules in CLAUDE.md so Claude knows your conventions
  • Set up CI review so every PR gets baseline feedback before human review starts
  • Establish a triage process: fix all "Critical" findings, discuss "Suggestions," acknowledge "Good"
  • Review the reviewer: periodically check Claude's review quality and update your prompts

Anti-pattern: Using Claude as a rubber stamp

"Claude didn't find anything, so the code must be fine." Wrong. Claude misses things, especially business logic errors and design problems. AI review reduces bugs — it doesn't eliminate the need for human review.

Reviewing Your Last Commits

A quick way to catch issues before they become PR feedback:

Pre-Push Review Commands

  • Review my last commit for bugs — Quick check before pushing
  • Review all changes on this branch vs main — Full branch review
  • What could go wrong with these changes in production? — Pessimistic review
  • Review this for security before I push to production — Security gate

Try It Yourself: Build a Review Workflow

  1. Quick review: Make some changes to a file in your project (or use uncommitted changes). Run /review in Claude Code. How many issues did it find that you missed?
  2. Custom skill: Create a .claude/commands/review-pr.md file tailored to your project. Include your specific conventions, tech stack, and common pitfalls. Test it on a recent branch.
  3. Compare reviews: Review your last 3 commits with Claude, then re-read the diffs yourself. Make a tally:
    • Issues Claude found that you missed
    • Issues you found that Claude missed
    • False positives from Claude
  4. Bonus: Set up the GitHub Actions workflow from this chapter to auto-review PRs in one of your repositories
