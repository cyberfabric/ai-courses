Chapter 14

Subagents — Specialized Workers in .claude/agents/

Spin up focused workers in their own context window — the third leg of the skills/hooks/subagents stool

Back in Chapter 6 you learned two ways to extend Claude Code: skills (reusable procedures you invoke with a slash) and hooks (shell commands that fire on lifecycle events). Those two cover a lot of ground — but there's a third primitive that the docs now treat as a peer to both, and it's the one that quietly changes how you run any non-trivial task: subagents.

A subagent is a specialized assistant that runs in its own context window, with its own system prompt, its own tool restrictions, and (optionally) its own model. You hand it a job; it goes off, does the work in isolation, and returns only the summary you actually need. Your main conversation never sees the 80 grep results, the 12 log files, or the 47 tool calls. It just sees the answer.

The mental model in one sentence: skills run procedures, hooks run on events, subagents run in their own context window. Each solves a different problem. Once you internalize that split, you stop forcing the wrong tool on the wrong job.

Skills vs Hooks vs Subagents at a Glance

Before we dive in, a quick comparison so you know which tool to grab when:

MechanismWhat it isWhen to reach for it
Skill A reusable prompt template invoked with /name or auto-loaded by description. You keep typing the same kind of instruction. Centralize it once.
Hook A shell command the harness fires on a lifecycle event (PreToolUse, Stop, etc.). You want something to happen automatically on every X — format, lint, notify, audit.
Subagent A specialist running in its own context window with its own prompt, tools, and model. The work is verbose, parallelizable, or needs different permissions than your main session.

Plenty of real workflows combine all three: a hook that auto-launches a skill, a skill that delegates to a subagent, a subagent with its own internal hooks. They compose — that's the point.

Why Subagents Exist (the Context Problem)

Every prompt you send Claude consumes context. Tool results consume context. File reads consume context. By the third hour of a long session, your context is a swamp of half-relevant search results and stack traces, and the model is paying real money to re-read them on every turn.

Subagents solve this by giving verbose, self-contained work its own private window. Run the test suite over there. Trawl the codebase for usages over there. Diff a 47-file PR over there. Only the conclusion comes back.

Without subagents

You ask Claude to "find every place we call the deprecated v1 API." It runs grep -r, dumps 200 hits into context, then walks them one by one. By the time it answers, half your context budget is gone. You ask one follow-up question and you're at compaction.

With a subagent

Claude delegates to the Explore subagent. It does the grep, reads the files, classifies the hits in its own context, and returns: "12 production callers in billing/, 8 test fixtures, 3 dead config strings. List of files attached." Your main thread sees a paragraph, not 200 lines.

The Three Built-in Subagents

Claude Code ships with three subagents that the model uses automatically when it spots the right kind of task. You don't have to configure them — they're there the moment you launch.

🔍
Explore
Fast, read-only codebase search. Runs on Haiku for speed. Denied Write and Edit. Claude picks one of three thoroughness levels: quick, medium, or very thorough.
🗺️
Plan
Read-only research used during plan mode. Inherits the parent's model. Gathers context so Claude can present a plan without infinite nesting (subagents can't spawn subagents).
🛠️
General-purpose
Full tool access. Inherits the parent's model. Use it for multi-step work that needs both exploration and modification — the catch-all worker.

There's also a "helpers" tier that fires automatically and you'll rarely think about: statusline-setup (Sonnet, runs when you invoke /statusline) and Claude Code Guide (Haiku, answers questions about Claude Code itself). These are good citizens — let them work.

When to Delegate (and When Not To)

Subagents are not free. They start with empty context, so they have to gather their own bearings, which adds latency. They cost tokens to spin up and tokens to summarize their findings on the way back. Use them when the win is real.

Use a subagent when
- The task produces verbose output you don't need in your main context (test runs, log scans, large diffs) - You want to enforce specific tool restrictions (read-only review, no shell access) - The work is self-contained and can return a summary - You can run multiple investigations in parallel
Stay in the main loop when
- The task needs frequent back-and-forth with you - Multiple phases share significant context (plan → implement → test) - You're making a quick, targeted edit - Latency matters — subagents start cold

Defining a Custom Subagent

A custom subagent is a Markdown file with YAML frontmatter. The frontmatter is the configuration; the body becomes the system prompt the subagent runs under.

Save it to one of these locations and Claude Code picks it up at session start:

  • .claude/agents/<name>.md — project subagent (check it into git so the team gets it)
  • ~/.claude/agents/<name>.md — user subagent (yours alone, available across all projects)

The fastest way to author one is the interactive /agents command — it walks you through the fields and can even draft the system prompt from a description.

/agents
Opens a tabbed UI: Library (browse + create) and Running (live subagents in this session). Pick "Create new agent" → "Personal" or "Project" → "Generate with Claude" and describe what you want. Claude drafts the name, description, system prompt, then asks you which tools and which model. Save and it's available immediately.

Or write the file by hand. Here's a complete example:

---
name: code-reviewer
description: Expert code review specialist. Use proactively after writing or modifying code.
tools: Read, Grep, Glob, Bash
model: sonnet
---

You are a senior code reviewer ensuring high standards of code quality and security.

When invoked:
1. Run git diff to see recent changes
2. Focus on modified files
3. Begin review immediately

Review checklist:
- Code is clear and readable
- Functions and variables are well-named
- No exposed secrets or API keys
- Input validation implemented
- Good test coverage

Provide feedback organized by priority:
- Critical issues (must fix)
- Warnings (should fix)
- Suggestions (consider improving)

Two fields are required: name and description. Everything else is optional and inherits sensible defaults. Here's the full surface:

FieldWhat it does
nameLowercase, hyphenated identifier. Used in @-mentions and the --agent flag.
descriptionOne sentence telling Claude when to delegate. Add "use proactively" for aggressive auto-delegation.
toolsAllowlist of tools. Omit to inherit everything from the parent (including MCP).
disallowedToolsDenylist. Applied before the allowlist, so you can subtract from "everything inherited."
modelsonnet, opus, haiku, a full ID like claude-opus-4-7, or inherit (default).
permissionModeOne of: default, acceptEdits, auto, dontAsk, bypassPermissions, plan.
maxTurnsHard cap on agentic turns before the subagent stops.
skillsList of skills to preload (full content injected) instead of leaving for discovery.
mcpServersMCP servers scoped to this subagent only — keeps their tool descriptions out of the parent's context.
hooksLifecycle hooks (e.g., PreToolUse, PostToolUse) that fire only while this subagent is active.
memoryuser, project, or local. Gives the subagent a persistent MEMORY.md directory.
backgroundtrue to always run concurrently rather than blocking the main thread.
effortOverride session-level thinking effort: low, medium, high, xhigh, max.
isolationworktree spawns the subagent in a temporary git worktree so its edits don't touch your checkout.
colorUI badge color. Picks from red, blue, green, yellow, purple, orange, pink, cyan.
initialPromptAuto-submitted as the first user turn when this agent runs as the main session via --agent.

If you set both tools and disallowedTools, the deny list applies first, then the allow list resolves against what's left. A tool listed in both is removed.

Permission Modes in Plain English

The permissionMode field is the most under-used knob in the spec. Six choices, each tuned for a different shape of work:

ModeWhat it does
defaultStandard prompts — you approve each new tool call interactively.
acceptEditsAuto-accept file edits and common filesystem commands inside the working directory.
autoA background classifier reviews each command and prompts you only when it spots something risky.
dontAskAuto-deny everything that wasn't explicitly allowed up front. Tightest mode short of read-only.
bypassPermissionsSkip every prompt. Convenient and dangerous — reach for it only in tightly-scoped automation.
planPlan mode (read-only exploration). The subagent can look but not touch.

One important inheritance rule: if the parent session is in bypassPermissions or acceptEdits, the subagent inherits that — its own permissionMode field is ignored. Same with auto mode. You can't quietly tighten security from inside a permissive parent; you can only relax it. Set the right floor at the session level.

Persistent Memory: Subagents That Learn

Set memory: project (or user, or local) on a subagent and it gets a directory it can read from and write to across conversations. The first 200 lines (or 25 KB) of MEMORY.md in that directory get prepended to its system prompt every time it spawns — so the subagent literally accumulates knowledge.

ScopeLocationUse when
user~/.claude/agent-memory/<name>/The subagent should remember across all your projects.
project.claude/agent-memory/<name>/Project-specific knowledge, shareable via git. Recommended default.
local.claude/agent-memory-local/<name>/Project-specific but should NOT be committed (gitignored).

Tell the subagent in its system prompt to consult its memory before starting and update it after finishing. Over a few weeks, your code-reviewer learns your project's recurring issues; your migration-helper learns your schema conventions; your onboarding-buddy learns the questions juniors keep asking.

Whole-Session Subagents: the --agent Flag

Pass --agent <name> when you launch Claude Code and the main thread itself takes on that subagent's system prompt, tool restrictions, and model — the same way --system-prompt works. Now your whole session behaves like that role.

$ claude --agent code-reviewer
@code-reviewer ready. Tools: Read, Grep, Glob, Bash. Model: sonnet.
Working directory: /Users/you/projects/your-app

I'll review code with focus on quality, security, and best practices. Show me what you'd like me to look at.

Pin it as the project default by adding "agent": "code-reviewer" to .claude/settings.json. Now everyone who opens this repo with Claude Code gets a code reviewer instead of a generalist — great for "this is the security-audit branch, behave accordingly" workflows. The CLI flag overrides the setting, so you can still launch a normal session with --agent "" when you need to.

Scope Priority: Who Wins When Names Collide

Multiple subagents can share a name — the highest-priority definition wins. Here's the order:

PriorityLocationScope
1 (highest)Managed settingsOrganization-wide deployment
2--agents CLI flagCurrent session only (JSON inline)
3.claude/agents/Current project (git-shared)
4~/.claude/agents/All your projects
5 (lowest)Plugin's agents/ directoryWherever the plugin is enabled

Quick guidance: project subagents for codebase-specific specialists (test-runner, migration-checker), user subagents for personal preferences (your favorite review style), CLI-defined subagents for one-off automation scripts. Plugin subagents are perfect for cross-team distribution but they cannot use hooks, mcpServers, or permissionMode for security reasons.

Four Ways to Invoke a Subagent

Claude usually decides on its own when to delegate, based on each subagent's description field. But you can take the wheel four different ways:

1. Natural language — just name it. Claude usually delegates.

"Use the test-runner subagent to fix failing tests"

2. @-mention — guarantees that specific subagent runs for the next task.

"@code-reviewer look at the auth changes"

3. /agents — interactive UI for create/edit/run/stop and live status.

4. --agent <name> — whole session takes on that subagent's prompt, tools, and model. Pin it with "agent": "<name>" in .claude/settings.json to make it the project default.

The --agent flag is the strongest move — it replaces the default Claude Code system prompt entirely, the same way --system-prompt does. Useful for "I want this whole session to behave like a security reviewer."

Parallel Execution: The Code-Review Power Move

The single most-cited reason people reach for subagents is parallelism. Three slow checks running in sequence take three slow check's worth of wall time. Three subagents running in parallel take whichever one is slowest.

Review my staged changes — in parallel, run a style check, a security scan, and a test-coverage analysis. Consolidate the findings.
Spawning three subagents…
🟦 style-checker (read-only, Haiku) — scanning 14 modified files
🟥 security-scanner (read-only, Sonnet) — auditing auth and input handling
🟩 test-coverage (Bash + Read, Sonnet) — running coverage report on diffed files

Consolidated review:
• 2 style nits (line 47, line 112) — non-blocking
• 1 security finding: missing CSRF check on /api/billing/webhook — blocking
• Coverage on changed lines: 87% (was 84%) — passing

That's a single prompt that does what would otherwise take three sequential review passes — and your main context stays clean because each subagent reports a summary, not its working notes.

Foreground vs Background

By default subagents run foreground: they block the main conversation until done, and any permission prompts pass through to you. Background subagents run concurrently while you keep working — before launch, Claude Code prompts for any tool permissions the subagent will need so it can proceed without interrupting you. Failures auto-deny rather than asking.

Claude usually picks the right mode. You can also ask for it ("run this in the background") or hit Ctrl+B to background a running task. Set CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1 if you want all spawns to stay synchronous.

Common Pitfalls

  • Over-delegating. A subagent for a one-line typo fix is theater. They start cold and have setup overhead. Reach for them when isolation, parallelism, or context savings actually pay off.
  • Losing context across the boundary. Only the subagent's final summary returns. If you need an artifact (a list of files, a diff), tell the subagent explicitly to include it in the summary. Don't assume it'll be there.
  • Model mismatch. Don't put Opus on a job Haiku can do. Explore runs on Haiku on purpose — it's fast and cheap, and codebase scanning doesn't need a flagship model. Match model to job.
  • Forgetting subagents can't spawn subagents. If your workflow needs nested delegation, chain subagents from the main thread, or look at Skills, or use Agent Teams (below).
  • Trusting bypassPermissions. The mode skips permission prompts entirely. Convenient and dangerous. Use it for tightly-scoped automation, never for "I just want fewer prompts in my interactive session" — that's what auto mode is for.

Power Move: Forked Subagents

Forks are subagents that inherit your full conversation history instead of starting cold. Same system prompt, same tools, same messages — but their tool calls stay out of your main window and only the final result comes back. Perfect for "try a side approach without re-explaining the situation."

Enable with CLAUDE_CODE_FORK_SUBAGENT=1 (requires Claude Code v2.1.117+). Once on, three things change:

  • Claude spawns a fork whenever it would otherwise use the general-purpose subagent. Named subagents like Explore still spawn as themselves.
  • Every spawn runs in the background.
  • The /fork command spawns a fork instead of acting as an alias for /branch.

You can kick one off explicitly: /fork draft unit tests for the parser changes so far. The fork appears in a panel below your prompt and runs while you keep working. Forks are still experimental, but they're the cheapest "side quest" mechanism you have — their first request reuses the parent's prompt cache.

Agent Teams (Experimental, but Worth Watching)

Subagents work within a single session. If you need multiple agents collaborating across sessions — passing messages back and forth, sharing memory, resuming each other's work — that's agent teams. Enable with CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1. Once on, you get a SendMessage tool that can resume a stopped subagent by ID, which means you can build durable multi-agent pipelines that survive restarts. Treat this as "preview territory" today, but it's where parallel-agent workflows are heading.

Practical Patterns

The Code-Review Fan-Out

Three specialists in parallel: style-checker, security-scanner, test-coverage. Each is a read-only subagent with a focused system prompt. Trigger from a single command, consolidate results before they hit your context. (You'll build this in the exercise below.)

Research-Then-Implement Split

One Explore subagent gathers context (find every caller, every test, every config touch-point). The main thread reads only the summary, then writes the change with full context budget intact. This is the pattern Plan Mode automates, but you can drive it manually too.

Multi-Language Reviewer Pool

One kotlin-reviewer, one swift-reviewer, one python-reviewer, each with a description that mentions its language. Claude routes a polyglot PR to the right specialist per file. Each reviewer can preload its own skill library via the skills field.

Isolated Worktree Edits

Set isolation: worktree on a subagent that runs experiments — it gets a temporary git worktree, makes its edits there, and the worktree is auto-cleaned if it produces no changes. Your working tree stays clean even if the experiment fails midway.

Build a code-review-fanout workflow

Create three project subagents and a slash command that runs them in parallel. By the end you'll have a single prompt that does what used to be three sequential review passes.

  1. Create .claude/agents/style-checker.md — tools: Read, Grep, Glob, model: haiku. System prompt: scan diffed files for naming, formatting, and idiomatic issues; report top 5 nits with line numbers.
  2. Create .claude/agents/security-scanner.md — tools: Read, Grep, Glob, Bash, model: sonnet. System prompt: focus on auth, input validation, secrets, and common OWASP categories; classify findings as critical / warning / informational.
  3. Create .claude/agents/test-coverage.md — tools: Read, Bash, model: sonnet. System prompt: run the project's coverage command, focus on diffed files, report coverage delta.
  4. Restart Claude Code (or run /agents) so the new files load.
  5. Add a slash command .claude/commands/fanout-review.md with body: "Use the style-checker, security-scanner, and test-coverage subagents in parallel on the staged diff. Consolidate findings into one report grouped by severity."
  6. Stage a real change in your project, run /fanout-review, and watch three colored badges fire concurrently.

Bonus: add a PostToolUse hook in settings.json that auto-fires /fanout-review after every git add, so the review happens before you even reach for the keyboard.

The pattern to internalize: when a task is going to flood your main context with output you won't reference again, that's a subagent. The mental switch from "I do this" to "I delegate this" is what separates a Claude Code user from a Claude Code architect.
