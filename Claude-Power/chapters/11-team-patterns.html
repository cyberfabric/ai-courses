Chapter 11

Team Collaboration Patterns

Scale Claude from individual productivity to team-wide leverage

Everything up to now has been about your productivity. You've mastered Claude Code, built custom skills, automated reviews, and streamlined debugging. But here's the thing: one power user on a team of ten creates a 10% improvement. Ten power users create a transformation.

This chapter is about scaling what you've learned across your team. Not by forcing everyone to use Claude the same way, but by building shared infrastructure that makes everyone more effective — while respecting different skill levels and preferences.

The Team Adoption Challenge

When you introduce Claude to a team, you'll find three groups of people:

Enthusiasts
Already using Claude daily. Want more power. Likely why you're reading this book.
Curious
Tried Claude a few times. Saw potential but didn't stick. Need a clear path in.
Skeptical
Think AI is overhyped. Worry about code quality. Need to see concrete results.

The mistake most teams make: they try to convert the skeptics first. Don't. Start with the enthusiasts, build shared tools that produce visible results, and let the curious people come to you. The skeptics will follow when they see the impact.

Shared CLAUDE.md: Your Team's AI Constitution

The single highest-impact thing you can do for your team: create a shared CLAUDE.md in your project root and commit it to git. This ensures every developer gets the same baseline behavior from Claude, regardless of their experience level.

Without Shared CLAUDE.md

Alice tells Claude to use Hilt. Bob doesn't know about the convention and Claude uses manual DI. Carol's Claude generates XML layouts because she never mentioned Compose. Three developers, three different code styles.

With Shared CLAUDE.md

Everyone's Claude uses Hilt, Compose, StateFlow, and conventional commits — automatically. The project CLAUDE.md teaches Claude once, and every developer benefits.

What to Put in a Team CLAUDE.md

Team CLAUDE.md template: 
# ProjectName

Brief description of the project and tech stack.

## Build & Test
- Build: `./gradlew assembleDebug`
- Test: `./gradlew testDebugUnitTest`
- Lint: `./gradlew lintDebug`
- Always run relevant tests before suggesting changes

## Architecture
- [Architecture pattern, key directories, data flow]
- [Where ViewModels, repositories, and use cases live]

## Code Conventions
- [Language, framework, and library choices]
- [Naming patterns, forbidden patterns]
- [Error handling approach]

## Git Conventions
- Branch: feature/JIRA-123-short-description
- Commits: conventional commits (feat:, fix:, chore:)
- Always create a new branch, never commit to main

## What NOT to Do
- [Specific anti-patterns your team has encountered]
- [Libraries or approaches to avoid and why]

## Available Skills
- /review-pr — Review current branch against main
- /generate-tests [file] — Generate unit tests
- /daily-standup — Summarize recent git activity
Keep it under 200 lines. A CLAUDE.md that's too long gets ignored (by both Claude and developers). Focus on rules that prevent real mistakes. If Claude already does something correctly without being told, don't waste a line on it.

Shared Skills Library

In Chapter 6, you learned to build individual skills. A team skills library multiplies the impact:

📝
/daily-standup
Everyone generates standup summaries the same way. Consistent format, same information, less standup meeting friction.
🔍
/review-pr
Every PR gets reviewed with the same criteria. Junior devs get the same review quality as senior devs.
🧪
/generate-tests
Tests follow the same patterns, naming conventions, and coverage expectations across the team.
📖
/explain
New team members can ask Claude to explain any file using the team's shared context. Reduces "hey, what does this do?" interruptions.

All of these live in .claude/commands/ and get committed to git. When a new developer clones the repo, they instantly have the full skill set.

Power Tip: The Onboarding Skill

Create a .claude/commands/onboard.md skill that walks new developers through the project:

Welcome the new developer to the project.

1. Read CLAUDE.md for project overview
2. Read the README for setup instructions
3. List the key directories and explain what each contains
4. Show the available custom skills (/review-pr, etc.)
5. Explain the git workflow and branch conventions
6. List 3 "easy win" issues from the backlog that would
   make good first tasks
7. Suggest which files to read first to understand
   the architecture

Be friendly and thorough — this is someone's first day.

New developer joins? They run /onboard and get a personalized walkthrough of the project. No senior dev needs to stop what they're doing for a 2-hour onboarding session.

Onboarding New Team Members to Claude

When introducing Claude to someone who's never used it (or barely used it), there's a specific order that works best:

Week 1

Claude.ai for questions and brainstorming

→

Week 2

Claude Code for coding tasks and reviews

→

Week 3

Team skills: /review-pr, /generate-tests, etc.

→

Week 4

Custom skills and personal workflow optimization

What to Teach First

  • Day 1: How to start a Claude Code session and ask a question about the codebase
  • Day 1: How to use /review and /commit
  • Day 2: How CLAUDE.md works and why our project has one
  • Day 3: How to use the team's custom skills (/review-pr, /generate-tests)
  • Week 2: Creating their own ~/.claude/CLAUDE.md for personal preferences
  • Week 3: Building their first custom skill for a personal workflow

Common Onboarding Mistakes

Mistake: Information overload

"Here's everything Claude can do!" dumps 50 features on someone at once. They'll forget 48 of them. Start with 3 things that produce immediate value: ask questions about code, review changes, generate tests.

Mistake: No guardrails

"Just use Claude however you want!" leads to inconsistent code, wasted tokens, and security issues. The shared CLAUDE.md and team skills provide guard rails without being restrictive.

Knowledge Sharing: What Worked, What Didn't

Teams that get the most out of Claude actively share their discoveries. Set up a simple system for capturing wins and failures:

Share This

  • Prompts that produced great results
  • Skills that saved significant time
  • Debugging patterns that found real bugs
  • Claude failures (so others avoid them)

Where to Share

  • A Slack channel (#claude-tips)
  • A shared doc or wiki page
  • The team's CLAUDE.md (for conventions)
  • New skills in .claude/commands/
Weekly ritual: Dedicate 5 minutes of your team standup or retro to "Claude wins this week." One person shares a technique that saved them time. Over months, this builds a collective knowledge base that makes the whole team faster.

Cost Management Across Teams

AI isn't free. When a whole team uses Claude, costs add up. Here's how to stay productive without burning through budget:

Understanding the Cost Stack

Claude.ai
Team plan: fixed per-seat cost. Predictable. Includes Claude Code.
API
Pay per token. Variable. Use prompt caching to cut costs by up to 90%.
Code Review
Token-based per review. Scope reviews to relevant files to reduce cost.

Cost Optimization Strategies

  • Right-size your model: Use Haiku for simple tasks (formatting, quick questions), Sonnet for most coding tasks, Opus for complex reasoning and architecture
  • Use /compact regularly: Compress long conversations to free up context and reduce token usage
  • Start fresh when context is stale: A new conversation is often cheaper than continuing a bloated one
  • Prompt caching for API: Cache system prompts and large contexts (see Chapter 7)
  • Focused CLAUDE.md: A lean CLAUDE.md uses fewer tokens per session than a 500-line novel
  • Track with /cost: Monitor per-session spending and identify expensive patterns

Security Considerations

When a team uses Claude, security becomes a shared responsibility. Here are the non-negotiable rules:

Security essentials for teams:
  • API keys: Use environment variables, never hardcode in source. Use a secrets manager at scale. Rotate keys regularly.
  • Sensitive data: Never paste production credentials, customer data, or secrets into Claude. Add rules to CLAUDE.md: "Never log sensitive fields: password, token, ssn, creditCard"
  • Code review: AI-generated code gets the same review as human-written code. No exceptions.
  • .gitignore: Ensure .claude/settings.local.json is ignored (Claude Code handles this automatically)
  • MCP servers: Only use trusted MCP servers. Audit what data they can access.

Quick Security Checklist for CLAUDE.md

## Security Rules
- Never log or print: passwords, tokens, API keys, PII
- Never commit: .env files, credentials, secret configs
- Never use --no-verify to skip git hooks
- Always validate user input at API boundaries
- Always use parameterized queries (no string concatenation for SQL)

Team Workflows: PR and Release Processes

Here's a team workflow that combines everything from this book into a coherent process:

Create Branch

Claude creates feature branch following team conventions

→

Develop

Code with Claude, auto-formatted by hooks

→

Self-Review

/review-pr before opening PR

→

Open PR

CI runs Claude Code Review automatically

→

Human Review

Reviewer focuses on architecture, not bugs

The human reviewer starts with Claude's findings already posted. They don't hunt for missing null checks or convention violations — Claude caught those. They focus on design, business logic, and maintainability. Reviews get better and faster.

Measuring Impact

How do you know if Claude is actually making your team more productive? Track these metrics:

Leading Indicators

  • Time from PR open to first review
  • Number of review round-trips
  • Test coverage on new code
  • Bug count in post-review code

Lagging Indicators

  • Sprint velocity (story points delivered)
  • Production bug rate
  • Developer satisfaction surveys
  • Onboarding time for new members
Don't over-measure. The goal isn't to prove ROI on every token. If your team is shipping faster with fewer bugs and higher confidence, Claude is working. Focus on the vibe: are developers happier? Are reviews more thorough? Is onboarding smoother?

Try It Yourself: Team Setup Sprint

  1. CLAUDE.md audit: Does your project have a CLAUDE.md? If not, create one using the template from this chapter. If it exists, review it — is it under 200 lines? Does it cover build commands, conventions, and forbidden patterns?
  2. Team skills: Create at least 2 shared skills in .claude/commands/. Start with /review-pr and /generate-tests customized for your project.
  3. Onboarding skill: Create the /onboard skill from this chapter. Test it by pretending you're a new developer and running it.
  4. Security check: Add the security rules section to your CLAUDE.md. Verify that .claude/settings.local.json is in .gitignore.
  5. Share: Show one teammate your favorite Claude technique from this book. Teach them one skill they can use today.
