Why Bots: Meet People Where They Are

Your team lives in Slack. Your friends live in Discord. Your family lives in WhatsApp and Telegram. Everyone already has a messaging app open all day.

So instead of building another web app nobody bookmarks, build a bot that lives where your users already are. No new app to install. No URL to remember. Just type a message and get an answer.

In Chapter 8, we built invisible AI services behind APIs. Now we're giving those services a face — a conversational one. Bots are the most natural interface for AI because talking to an AI should feel like talking.

3B+ people use messaging apps daily
0 new apps your users need to install
~2 hrs to build your first useful bot

The Good Bot vs. The Annoying Bot

Before we build anything, let's talk about what makes a bot people want to use vs. one they immediately mute:

The Annoying Bot
- Messages you without being asked - Sends walls of text nobody reads - Asks 5 questions when 1 would do - Responds to every message in a channel - Has a cutesy personality that gets old fast - Breaks when you say something unexpected - "I'm sorry, I don't understand" on repeat
The Good Bot
- Responds only when spoken to (or on schedule) - Gives concise, actionable answers - Asks clarifying questions only when necessary - Stays out of conversations it wasn't invited to - Has a consistent, professional tone - Handles unexpected input gracefully - Knows when to say "let me check" vs. guessing
The #1 rule of bot design: be useful, not chatty. Nobody wants a bot that says "Great question! Let me think about that..." before every answer. Get to the point. Deliver value. Shut up.

The Projects

Team Standup Bot
Week
Python Claude API Slack Bolt SQLite

Collects async standups via DM every morning, summarizes them for the team channel, identifies blockers, and spots patterns over time. "Alice has been blocked on the API integration for 3 days — this might need escalation."

What it does: DMs each team member at their preferred time, collects standup responses (what I did, what I'm doing, any blockers), posts an AI-summarized digest to the team channel, and tracks blockers across days.

Why it's cool: Async standups in a channel are noisy — 8 people posting walls of text nobody reads. This bot condenses them into a 30-second summary with the important stuff highlighted. Blockers that persist across days get automatically flagged.

How to build it:

  1. Use Slack Bolt to create a bot that listens for slash commands and DMs
  2. Schedule daily DMs using Slack's scheduled messages or a cron job
  3. Collect responses and store in SQLite with timestamps
  4. At a configured time, send all responses to Claude: "Summarize these standups. Highlight blockers, flag any that have persisted 2+ days."
  5. Post the summary to the team channel with @mentions for items needing attention
  6. Add a /standup-trends command that shows weekly patterns

On-Call Companion Bot
Week
Python Claude API python-telegram-bot ChromaDB

Forward a PagerDuty alert to Telegram, and the bot gives you: what the alert means, relevant runbook steps, similar past incidents, and investigation commands to run. Your 3am self will thank you.

What it does: Receives alert messages, matches them against indexed runbooks and incident history using RAG, and provides context-aware investigation guidance — all in your phone's messaging app.

Why it's cool: At 3am, you're half-asleep staring at "CRITICAL: latency p99 > 500ms." This bot instantly tells you: "This usually means the database connection pool is exhausted. Check: kubectl get pods -n db. Last time this happened (March 12), it was caused by a connection leak in the batch job."

How to build it:

  1. Index your runbooks and past incident reports in ChromaDB (RAG setup from Chapter 5)
  2. Create a Telegram bot with python-telegram-bot
  3. On receiving an alert: search runbooks for relevant procedures, find similar past incidents
  4. Send context to Claude: "Given this alert and these runbook excerpts, what should the on-call engineer do?"
  5. Reply with: summary, investigation steps, relevant commands, and links to full runbooks
  6. Add a follow-up flow: "I ran that command, here's the output" → bot analyzes and suggests next steps
On-call bots should assist, not replace human judgment. Always include a "this is AI-generated guidance, verify before acting" disclaimer. For destructive actions (restarting services, scaling down), require human confirmation.

Daily Briefing Bot
Weekend
Python Claude API python-telegram-bot Various APIs

Every morning at 8am, your Telegram pings with a personalized briefing: weather, calendar highlights, top news in your interests, GitHub notifications, and anything else you configure. One message, 30-second read, you're caught up.

What it does: Aggregates data from multiple sources (weather API, Google Calendar, news APIs, GitHub), sends everything to Claude for a concise AI-narrated summary, and delivers it as a Telegram message at your preferred time.

Why it's cool: Instead of checking 6 different apps every morning, you check one message. The AI doesn't just list data — it narrates it: "Rainy today, bring an umbrella. You have 3 meetings, including the design review at 2pm. Two PRs need your review, both from Alice."

How to build it:

  1. Create a Telegram bot and save your chat ID for direct messaging
  2. Build data fetchers: weather (OpenWeatherMap), calendar (Google Calendar API), GitHub (REST API), news (NewsAPI or RSS feeds)
  3. Aggregate all data into a structured prompt
  4. Send to Claude: "Create a concise morning briefing from this data. Be conversational, highlight what matters."
  5. Schedule with cron or APScheduler to send at your preferred time
  6. Add a /briefing command for on-demand briefings

Conversation State Management

A bot that forgets what you said 5 seconds ago is useless. Here's how to maintain context across messages:

from collections import defaultdict

# Simple in-memory conversation store
conversations = defaultdict(list)

async def handle_message(user_id, message):
    # Retrieve conversation history
    history = conversations[user_id]
    history.append({"role": "user", "content": message})

    # Keep last 10 messages to stay within token limits
    recent = history[-10:]

    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=512,
        system="You are a helpful team assistant bot. Be concise.",
        messages=recent
    )

    reply = response.content[0].text
    history.append({"role": "assistant", "content": reply})
    return reply

For persistence across restarts, swap the dict for SQLite or Redis. Key the conversation by user_id (for DMs) or channel_id + user_id (for channels).

Set a conversation TTL (time-to-live). Clear context after 30 minutes of inactivity. This prevents stale context from confusing the bot when a user comes back hours later with a different topic. Fresh conversations are better than confused ones.

Multi-Turn Interactions

Sometimes a bot needs to ask follow-up questions. The pattern:

User Request → Bot Clarifies → User Answers → Bot Acts

The key: use a state machine. Track where each user is in a flow:

user_states = {}  # {user_id: {"flow": "expense", "step": "confirm", "data": {...}}}

async def handle_message(user_id, message):
    state = user_states.get(user_id)

    if state and state["flow"] == "expense":
        if state["step"] == "confirm":
            if message in ["👍", "yes", "correct"]:
                save_expense(state["data"])
                del user_states[user_id]
                return "Saved!"
            else:
                # Let Claude parse the correction
                corrected = parse_correction(message, state["data"])
                state["data"] = corrected
                return f"Updated: {corrected}. Confirm?"

    # No active flow — handle normally
    return await normal_response(user_id, message)

Deploying Bots: Always-On vs. Serverless

Bots need to be running 24/7. Two approaches:

Approach Best For Cost
Always-On Server Real-time bots, WebSocket connections (Discord, Slack). Use a VPS ($5/mo on Hetzner, DigitalOcean) or Railway. $5-10/month
Serverless (webhook mode) Telegram bots (natively webhook-based), Slack bots with event subscriptions. Use Lambda, Cloud Functions, or Vercel. Often free tier

Telegram bots work great with webhooks — set the webhook URL and Telegram pushes messages to your serverless function. Discord and Slack bots typically need a persistent WebSocket connection, so they need an always-on server.

What if you built a "meta-bot" that routes to specialized bots? One bot in your Slack workspace that handles everything: "expense" routes to the Expense Bot, "standup" triggers the Standup Bot, "quiz me on K8s" activates the Learning Buddy, and "what's my morning look like?" calls the Daily Briefing. One unified @assistant that intelligently routes to the right specialist. Use Claude to classify the request and dispatch to the right handler — you've built a personal AI assistant that lives in your team's chat.

Build It: Daily Briefing Bot

Let's build the Daily Briefing Bot for Telegram. You'll get your first AI-narrated morning digest tomorrow.

Step 1: Create a Telegram bot (2 min)

Open Telegram, find @BotFather, send /newbot, follow the prompts. Save the API token.

Step 2: Set up (2 min)

mkdir briefing-bot && cd briefing-bot
python -m venv .venv && source .venv/bin/activate
pip install anthropic python-telegram-bot requests apscheduler

Step 3: Create bot.py (12 min)

import anthropic, requests, asyncio
from telegram import Bot
from apscheduler.schedulers.asyncio import AsyncIOScheduler

TELEGRAM_TOKEN = "your-telegram-bot-token"
CHAT_ID = "your-chat-id"  # Send /start to your bot, check updates API
WEATHER_KEY = "your-openweathermap-key"  # Free tier

client = anthropic.Anthropic()
bot = Bot(token=TELEGRAM_TOKEN)

def get_weather(city="London"):
    r = requests.get(f"https://api.openweathermap.org/data/2.5/weather"
        f"?q={city}&appid={WEATHER_KEY}&units=metric")
    d = r.json()
    return f"{d['weather'][0]['description']}, {d['main']['temp']}°C"

def get_github_notifications():
    # Use a personal access token for authenticated requests
    headers = {"Authorization": "token YOUR_GITHUB_TOKEN"}
    r = requests.get("https://api.github.com/notifications",
                     headers=headers, params={"per_page": 5})
    if r.status_code == 200:
        return [n["subject"]["title"] for n in r.json()[:5]]
    return ["(configure GitHub token for notifications)"]

async def send_briefing():
    weather = get_weather()
    github = get_github_notifications()

    data = f"Weather: {weather}\nGitHub: {', '.join(github)}"

    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=512,
        system="Create a concise, friendly morning briefing. "
               "Use short paragraphs. Be conversational.",
        messages=[{"role": "user", "content": data}]
    )

    await bot.send_message(chat_id=CHAT_ID, text=response.content[0].text)

async def main():
    scheduler = AsyncIOScheduler()
    scheduler.add_job(send_briefing, "cron", hour=8, minute=0)
    scheduler.start()
    print("Briefing bot running. Next briefing at 8:00 AM.")

    # Send one now for testing
    await send_briefing()
    print("Test briefing sent!")

    # Keep running
    while True:
        await asyncio.sleep(60)

if __name__ == "__main__":
    asyncio.run(main())

Step 4: Run it (1 min)

# Set your environment variables and run
python bot.py

You'll get a test briefing immediately and then one every morning at 8am. Add more data sources (calendar, news, Jira) as you see fit — each one is just another function that fetches data and adds it to the prompt.
