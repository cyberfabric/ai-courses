The "What Should I Build?" Framework

You've seen 55+ project ideas across 11 chapters. CLI tools, dashboards, bots, web apps, APIs, creative experiments, workflow automations. The question isn't "what can I build?" anymore. It's "what should I build first?"

Here's a simple framework. Score each project idea on four dimensions:

F Feasibility — can you build it this weekend?
V Value — will you or others actually use it?
L Learning — will it teach you something new?
Fun Enjoyment — will you have fun building it?

Rate each 1-5. Multiply: F × V × L × Fun. The highest score wins. It's not scientific, but it forces you to think about what matters.

A project that's feasible (5), valuable (5), teaches nothing (1), and boring (1) scores 25. A project that's feasible (4), moderately useful (3), teaches a lot (5), and fun (4) scores 240. Pick the 240.

Don't optimize for "impressive." Build what's useful to you. The Terminal Translator from Chapter 1 isn't flashy, but if you use it 10 times a day, it's worth more than a fancy dashboard nobody opens. Personal utility beats demo appeal every time.

Scoping: The Weekend MVP

The #1 project killer isn't technical difficulty. It's scope creep. "I'll build a full-featured dashboard with auth, multi-tenancy, real-time updates, and..." — dead on arrival. Three weeks later, nothing is shipped.

The fix: the Weekend MVP. Force yourself to build the smallest useful version in 48 hours.

Friday Night → Saturday Build → Sunday Polish → Sunday Night Ship
  • Friday night: Define the ONE thing this tool does. Write it in one sentence. If you can't, the scope is too big.
  • Saturday: Build the core feature. No auth, no database, no fancy UI. Just the thing that makes it useful.
  • Sunday morning: Polish. Add the minimum UX to make it usable (not beautiful, usable).
  • Sunday evening: Ship. Push to GitHub, deploy to Railway/Vercel, tell someone about it.
Scope Creep Project
"I'll build a full expense tracker with: - Multi-currency support - Receipt OCR with ML model - Bank API integration - Monthly reports with charts - User accounts with OAuth - Mobile-responsive dashboard - Export to QuickBooks - Team features with roles" Status after 3 weeks: 20% done, abandoned
Weekend MVP
"Send a receipt photo to Telegram, get the amount and category back." That's it. One feature. One weekend. Status after 2 days: shipped, working, used daily. Features added over time.

Tech Stack Decision Framework

Don't overthink this. Here's when to use what:

If you need... Use this Why
A CLI tool Python + Click Fastest to build, easiest to distribute
A web app Python + FastAPI (or Next.js) FastAPI for speed, Next.js for rich UI
A dashboard Python + Streamlit Zero frontend code, live in minutes
A bot Python + platform SDK All bot SDKs have great Python support
A backend API Python + FastAPI Async, typed, auto-docs, production-ready
Something with a nice UI TypeScript + React + Next.js When the UI IS the product

Notice a pattern? Python for almost everything. It has the best AI libraries, the simplest deployment, and Claude Code generates excellent Python. Only reach for TypeScript when you need a rich interactive frontend.

Prompt Engineering Patterns

Every project in this course uses Claude API calls. Here are the patterns that apply everywhere:

The Persona Pattern

Start your system prompt with who the AI should be: "You are a senior database architect" or "You are a witty dungeon master." This sets the tone, vocabulary, and expertise level.

The Constraint Pattern

Tell the AI what NOT to do: "Output ONLY valid JSON. No markdown, no explanation." "Never break character." "Don't suggest solutions that require paid services." Constraints prevent the most common failure modes.

The Example Pattern

Show the AI what you want: "Here's an existing endpoint. Generate a new one following the exact same patterns." One good example is worth 100 words of description.

The Structured Output Pattern

Use tool use for anything that needs to be parsed programmatically. Don't ask the AI to "return JSON" in a text response — use the tool use API to guarantee the schema. (See Chapter 8 for details.)

Don't over-engineer your prompts. Start with the simplest possible system prompt. Only add constraints when you see failure modes. A 50-word prompt that works is better than a 500-word prompt that's fragile. Iterate based on real failures, not hypothetical ones.

Cost Management

Keeping Claude API costs under $5/month for personal projects is easy:

  • Use Haiku for simple tasks: Classification, formatting, summarization. ~$0.001 per request.
  • Use Sonnet for complex tasks: Code generation, analysis, creative writing. ~$0.01 per request.
  • Cache aggressively: Same input = same output. Hash the input, check the cache first.
  • Set max_tokens wisely: A tweet needs 100 tokens, not 4096.
  • Use prompt caching: Long system prompts cached at 90% discount.
  • Batch when possible: "Categorize these 20 emails" in one call, not 20 calls.

At 100 Haiku requests/day, you'll spend about $3/month. That's less than a coffee. Even heavy Sonnet usage rarely exceeds $10/month for personal projects.

The Build Loop

Idea → Prototype → Test → Iterate → Ship
  1. Idea: One sentence. What does it do? Who is it for?
  2. Prototype: Build the core in one sitting. Use Claude Code to help. Don't polish.
  3. Test: Use it yourself for a day. Does it solve the problem?
  4. Iterate: Fix what's broken, add what's missing. One thing at a time.
  5. Ship: Push to GitHub. Deploy. Tell someone. It doesn't have to be perfect.

Shipped Beats Perfect

A working tool with rough edges that you use every day is infinitely more valuable than a polished masterpiece sitting in a private repo. Ship early, improve later. Your first user should be you. Your second user should be a friend. Everything after that is a bonus.

Open-Sourcing Your Project

Built something useful? Share it. Here's the minimum for a good open-source release:

README Template

# Project Name
One-sentence description of what it does.

## Quick Start
```bash
pip install your-tool
your-tool "do the thing"
```

## What It Does
2-3 sentences explaining the value.

## Installation
Step-by-step setup instructions.

## Usage
3-5 example commands with expected output.

## Configuration
Environment variables, config files, options.

## License
MIT (or your choice)

Share Your Work

  • GitHub: Public repo with a good README, license, and example usage
  • Blog post: Write about what you built and why — the story matters more than the code
  • Show HN: Hacker News loves developer tools. "Show HN: I built an AI-powered git log beautifier" will get upvotes
  • Twitter/X: Short demo GIF + one-line pitch. Developer tools go viral when they're visual
  • Product Hunt: For more polished tools with a web interface

The Build Loop in Practice

Let's walk through a real example. Say you pick the CSV Whisperer from Chapter 4:

Phase What You Do Time
Idea "Chat with any CSV file using natural language questions." 5 min
Prototype Follow the exercise: load CSV, send schema to Claude, execute generated pandas code, print results. 20 min
Test Try it with your bank statement, a sales export, and an analytics CSV. Note what breaks. 30 min
Iterate Fix: date parsing, large files, chart generation. Add: --output flag, error recovery. 2 hrs
Ship Push to GitHub, add README with usage examples, pip-installable via pyproject.toml. 30 min

Total: under 4 hours. You now have a shipped, useful tool that you'll reach for every time someone sends you a CSV.

Use Claude Code itself to help you build. Describe what you want, let it write the first draft, then iterate. You're building AI tools with AI tools. It's turtles all the way down, and it's fast.

From Side Project to Product

Some projects deserve more than a weekend. How do you know when to invest more?

  • You use it every day and can't imagine going back
  • Other people ask you to share it
  • You keep adding features because you want to, not because you have to
  • Someone says "I'd pay for this"
  • You see competitors charging real money for something worse
  • It solves a problem for a specific, reachable audience

If 3+ of these are true, you might have a product. Add user accounts, a payment page (Stripe is one API call), and a landing page (use the generator from Chapter 7). Charge $5-20/month. See if people pay.

The Portfolio Advantage

5 small shipped projects beat 1 big unfinished one. Every time.

A portfolio of AI projects tells employers and clients: "I don't just know about AI — I build with it. Repeatedly. Successfully." Each project demonstrates a different skill: CLI tools show systems thinking, web apps show product sense, bots show user empathy, APIs show architecture.

Here's what a strong AI portfolio looks like:

  1. A CLI tool you use daily (shows utility)
  2. A web app with a deployed URL (shows product skills)
  3. A bot running in a real Slack/Discord (shows integration skills)
  4. An open-source project with stars (shows community engagement)
  5. A creative project that makes people smile (shows personality)

You can build all five in a month using projects from this course.

Your 30-Day Challenge

Ship one project per week for 4 weeks. Here's a suggested schedule:

The simplest build. A CLI tool you'll use every day. Gets you comfortable with the Claude API pattern. Ship to GitHub, add a README, install globally on your machine.

Week 2: PR Description Generator (Chapter 2)
