Chapter 3

Code Generation & Scaffolding

Generate boilerplate, scaffolds, and production-ready starter code from plain English descriptions

Why Scaffolding Matters

Think about the last time you started a new feature. What did you do first? Write the exciting business logic? Nope.

You created a file. Copied a similar file. Renamed things. Set up the boilerplate. Wired the imports. Created a test file. Added the route. Updated the config. Two hours later, you finally wrote the first line of interesting code.

That first two hours? It's the same every time. And it's exactly the kind of repetitive, pattern-based work that AI devours.

~2 hrs wasted on boilerplate per feature
80% of scaffolding is copy-paste-rename
5 min with a good code generator

In Chapter 2, we automated the process around code — PR descriptions, reviews, docs. Now we're automating the code itself. Not the clever parts — the boring parts. The parts that follow a pattern. The parts you can describe in one sentence.

"I need a REST endpoint for user profiles with CRUD operations and validation."

That sentence has enough information to generate 300 lines of production code. Let's build the tools that do it.

API Endpoint Generator
Week
Python Claude API FastAPI / Express OpenAPI

Describe an API endpoint in plain English, and get the full implementation: route handler, request/response models, validation, error handling, tests, and an OpenAPI spec — all matching your project's conventions.

What it does: Takes a natural language description like "POST endpoint for creating orders with items, quantities, and shipping address" and generates complete, tested endpoint code ready to drop into your project.

Why it's cool: Every REST endpoint follows the same pattern: parse input, validate, call service, handle errors, return response. This tool knows that pattern and writes it faster than you can type.

How to build it:

  1. Accept a description and a --framework flag (FastAPI, Express, Flask, etc.)
  2. Optionally scan your project for existing endpoints to learn your conventions
  3. System prompt includes the framework's patterns, your project's style, and output format
  4. Generate: route handler, request/response schemas, validation rules, error handlers, and unit tests
  5. Also generate an OpenAPI spec snippet that can be merged into your existing spec
  6. Write each to the correct directory with --write, or print to stdout for review

Test Suite Generator
Weekend
Python Claude API pytest / Jest

Point it at any class or module, and it generates comprehensive tests: happy paths, edge cases, error scenarios, and integration stubs. No more staring at an empty test file wondering where to start.

What it does: Reads your source code, analyzes every public method, and generates a test suite that covers normal usage, boundary conditions, error handling, and common edge cases.

Why it's cool: Writing tests is the thing everyone agrees is important and nobody wants to do first. This tool generates 80% of your test suite in seconds. You review, tweak, and add the tricky edge cases that need human insight.

How to build it:

  1. Accept a file path or module name as input
  2. Read the source code and any existing tests for context
  3. Send to Claude: "Generate comprehensive tests for this code. Include: happy path, edge cases, error handling, boundary values. Use [pytest/Jest/JUnit] conventions."
  4. Generate the test file with proper imports, fixtures, and assertions
  5. Add a --coverage flag that runs the tests and reports coverage immediately
  6. Add --style tdd to generate tests first, before the implementation exists
AI-generated tests are a starting point, not the finish line. They're great at covering the obvious cases, but they can miss subtle business logic edge cases. Always review generated tests — and add the cases that require domain knowledge the AI doesn't have. The goal is 80% coverage in 5 minutes, not 100% coverage on autopilot.

Template Engines: Matching Your Conventions

The biggest complaint about code generators? "The output doesn't look like our code." Here's how to fix that.

Feed It Examples

The simplest approach: include an existing file from your project in the prompt. "Generate a new endpoint following the exact same patterns as this one."

# Read an existing endpoint as the "style guide"
existing = open("src/routes/users.py").read()

prompt = f"""Generate a new endpoint for orders.
Follow the EXACT same patterns as this existing endpoint:

{existing}

Requirements: POST /orders, validates items exist in inventory,
returns order ID and estimated delivery."""

Convention Files

For team-wide consistency, create a .codegen/ directory with convention files:

.codegen/
  conventions.md        — naming, structure, error handling rules
  templates/
    endpoint.py.tmpl    — skeleton with your patterns
    model.py.tmpl       — model class skeleton
    test.py.tmpl        — test file skeleton

Your generator reads these before generating anything. The AI fills in the blanks, but the structure is yours.

Convention files are the difference between a toy and a tool. Without them, every developer gets slightly different output. With them, the generated code looks like your team wrote it. Spend 30 minutes writing conventions once — it pays off on every generation.

The Validation Loop

Generated code that doesn't compile is useless. Here's the pattern that makes code generation reliable:

Generate → Write Files → Run Tests → Fix Errors → Ship It

The key insight: after generating code, automatically run the compiler and tests. If something fails, send the error back to Claude and ask it to fix it. Most issues resolve in one retry.

import subprocess

# Generate the code
generated_code = call_claude(prompt)
write_file("src/routes/orders.py", generated_code)

# Try to run tests
result = subprocess.run(["pytest", "tests/test_orders.py"],
                       capture_output=True, text=True)

if result.returncode != 0:
    # Send the error back for a fix
    fix_prompt = f"The generated code had errors:\n{result.stderr}\n\nFix the code."
    fixed_code = call_claude(fix_prompt)
    write_file("src/routes/orders.py", fixed_code)

This generate-test-fix loop is what separates a "paste AI output and pray" approach from a real code generation tool. The tool verifies its own work.

What if you took this to the extreme? Imagine a "project bootstrap" tool: describe your entire project in one paragraph — "a recipe sharing platform with user auth, recipe CRUD, ingredient search, and a rating system" — and it generates the entire skeleton. Database schema, API endpoints, model classes, test stubs, Dockerfile, CI config, README. Not a finished product, but a running starting point you can build on in minutes instead of days. The Database Schema Designer + API Generator + Test Generator, all chained together with the validation loop.

Teaching the Generator Your Patterns

Every team has unwritten rules. "We always use repository pattern." "Error messages come from a constants file." "ViewModels never import Android framework classes." Here's how to encode those rules:

The Project Scan

Before generating anything, scan the project and build a context document:

def scan_project(project_path):
    """Build a context document from the existing project."""
    context = {
        "framework": detect_framework(project_path),
        "architecture": detect_architecture(project_path),
        "naming": detect_naming_conventions(project_path),
        "examples": get_sample_files(project_path, count=3),
        "dependencies": read_dependency_file(project_path),
    }
    return format_context(context)

This context goes into every prompt. The AI doesn't just generate generic code — it generates code that fits your project.

The Conventions Checklist

Add a post-generation check that verifies the output matches your rules:

  • Uses correct import paths for your project
  • Follows naming conventions (camelCase, snake_case, etc.)
  • Error handling matches your project's pattern
  • Tests use your project's fixtures and helpers
  • Generated files are in the correct directories
  • No hardcoded strings that should be in constants
  • No direct dependencies that should be injected

The 80/20 Rule of Code Generation

AI generates 80% of the code perfectly. The remaining 20% needs your expertise — business logic edge cases, performance optimization, and domain-specific patterns the AI hasn't seen. Don't try to make the generator perfect. Make it fast, review the output, and polish the 20% yourself. That's still 80% less boilerplate you had to write.

Build It: Database Schema Designer

Let's build the Database Schema Designer. We'll generate a complete schema for a recipe app in under 20 minutes.

Step 1: Set up (2 min)

mkdir schema-designer && cd schema-designer
python -m venv .venv && source .venv/bin/activate
pip install anthropic click

Step 2: Create schemadesign.py (10 min)

import click
import anthropic
import json

client = anthropic.Anthropic()

SYSTEM = """You are a database architect. Given a description
of an application, design the database schema.

Output a JSON object with:
- "tables": array of tables, each with:
  - "name": table name (snake_case, plural)
  - "columns": array of {name, type, constraints}
  - "indexes": array of index definitions
- "relationships": array of {from, to, type, via}
- "sql": complete CREATE TABLE SQL statements
- "seed": INSERT statements with 5+ realistic rows per table
- "diagram": Mermaid ER diagram code

Use PostgreSQL types. Include created_at/updated_at on all tables.
Add appropriate indexes for foreign keys and common queries."""

@click.command()
@click.argument("description", nargs=-1, required=True)
@click.option("--output", "-o", default=None, help="Output directory")
@click.option("--diagram", is_flag=True, help="Print Mermaid diagram")
def main(description, output, diagram):
    query = " ".join(description)

    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=4096,
        system=SYSTEM,
        messages=[{"role": "user", "content": query}]
    )

    text = response.content[0].text

    # Extract JSON from response
    try:
        start = text.index("{")
        end = text.rindex("}") + 1
        data = json.loads(text[start:end])
    except (ValueError, json.JSONDecodeError):
        click.echo(text)
        return

    if diagram and "diagram" in data:
        click.echo(data["diagram"])
    elif output:
        import os
        os.makedirs(output, exist_ok=True)
        with open(f"{output}/schema.sql", "w") as f:
            f.write(data.get("sql", ""))
        with open(f"{output}/seed.sql", "w") as f:
            f.write(data.get("seed", ""))
        click.echo(f"Written to {output}/")
    else:
        click.echo(data.get("sql", text))

if __name__ == "__main__":
    main()

Step 3: Generate a schema (2 min)

python schemadesign.py "recipe app with users, recipes that have
  ingredients with amounts, star ratings with comments, and
  the ability to bookmark favorite recipes" --output ./db

# Check what was generated
cat db/schema.sql
cat db/seed.sql

Step 4: Visualize it (1 min)

# Generate a Mermaid diagram and paste into mermaid.live
python schemadesign.py "recipe app with users, recipes,
  ingredients, ratings, bookmarks" --diagram

You just designed a database without writing SQL. Copy the schema into your next project and start building.

Pick Your Project

Five generators, each one saves hours every time you start something new:

  • API Endpoint Generator — if you write the same CRUD routes over and over (Week)
  • Android Feature Scaffolder — if creating an Android feature means 8 files of boilerplate (Week)
  • Database Schema Designer — if you want to go from idea to schema in 2 minutes (Weekend)
  • Test Suite Generator — if you always "plan to add tests later" (Weekend)
  • Component Library Builder — if building UI components is 70% prop types and stories (Week)

My recommendation? Start with the Database Schema Designer. It's a weekend build, the exercise above gets you running in 15 minutes, and it's the one tool that helps at the very start of every project. A good schema is the foundation everything else is built on.

And once you've generated a schema, you'll immediately want the API Endpoint Generator to build routes for it. That's project number two, and suddenly you're building projects in a fraction of the time.
