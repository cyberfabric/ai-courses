Build intelligent automations that connect services and make decisions — smarter than Zapier because they think

Beyond Zapier

Zapier, Make, n8n — they're great at moving data from A to B. "When a form is submitted, add a row to the spreadsheet." Simple. Reliable. Dumb.

But what happens when the automation needs to think? "When an email arrives, figure out if it's an invoice, a meeting request, or a newsletter — then route it accordingly." Zapier can't do that. It can't read an email and understand it.

That's where AI-powered automation gets exciting. You're not just moving data. You're building pipelines that analyze, decide, and act.

Trigger → Analyze → Decide → Act

The "Analyze" and "Decide" steps are where AI lives. A traditional automation skips them entirely — it just moves data on a trigger. An AI automation reads the data, understands it, and makes a judgment call about what to do next.

80% of useful automations need a "decide" step
10x more flexible than rule-based automation
$0 monthly subscription fees (you own the code)

In Chapter 9, we built bots that respond to messages. Now we're building automations that respond to events — emails, git pushes, form submissions, app reviews — and take intelligent action without anyone asking.

Smart Email-to-Action Pipeline
Week
Python Claude API IMAP Google Calendar API

Monitors your inbox and automatically routes emails to the right workflow. Invoices get forwarded to accounting and logged in a spreadsheet. Meeting requests get added to your calendar. Newsletters get summarized into a daily digest. Spam gets archived.

What it does: Connects to your email via IMAP, classifies each new message using AI, and triggers the appropriate workflow: forward, file, calendar entry, summary digest, or archive.

Why it's cool: Email is the universal inbox — and it's chaos. This pipeline imposes intelligent order. You stop being a human email router and let AI sort, classify, and act on the 80% of emails that don't need your brain.

How to build it:

  1. Connect to your inbox via IMAP (imaplib) or Gmail API
  2. For each new email, send subject + body to Claude: classify as invoice, meeting, newsletter, action-required, or noise
  3. Route based on classification:
    • Invoice: extract amount, vendor, due date → log to spreadsheet, forward to accounting
    • Meeting: extract date, time, attendees → create calendar event
    • Newsletter: summarize key points → add to daily digest
    • Action-required: extract task → create todo item, send you a notification
    • Noise: archive silently
  4. Generate a daily email digest: "Today you received 3 invoices ($4,200 total), 2 meeting invites, and 5 newsletters (here are the highlights)."

Customer Feedback Analyzer
Week
Python Claude API App Store API Slack API

Collects customer feedback from multiple sources — App Store reviews, support tickets, social mentions, NPS surveys — categorizes sentiment and topics, identifies trends, and generates a weekly report with actionable insights.

What it does: Aggregates feedback from multiple channels, runs AI-powered sentiment analysis and topic extraction, identifies emerging trends (positive and negative), and generates a weekly report with recommendations.

Why it's cool: Customer feedback is gold, but it's scattered everywhere and nobody has time to read 200 reviews. This tool reads all of them, finds the patterns, and tells you what matters: "Login issues mentioned 47 times this week, up 300% from last week. Likely related to the v3.2 update."

How to build it:

  1. Build collectors: App Store reviews (RSS feed or scraper), support tickets (Zendesk/Intercom API), social mentions (Twitter search)
  2. For each piece of feedback, classify: sentiment (positive/negative/neutral), topic, specific feature mentioned, urgency
  3. Aggregate weekly: count by topic, track sentiment trends, identify spikes
  4. Generate a weekly report with Claude: "Summarize this week's customer feedback. Highlight new issues, persistent problems, and positive trends."
  5. Post to Slack and email to product team
  6. Add alerts for sudden sentiment drops (>20% negative change triggers immediate notification)
App Store review scraping may violate terms of service. Use official APIs where available (Apple provides RSS feeds for app reviews). For Google Play, use the official Google Play Developer API. Always respect platform ToS.
Hiring Pipeline Automator
Month
Python Claude API Gmail API Calendar API

Screens job applications against your requirements, scores candidates, sends personalized responses, and schedules interviews for promising applicants. The recruiter's tedious work, automated intelligently.

What it does: Processes incoming applications, extracts key qualifications, scores against job requirements, sends personalized acknowledgments, and moves top candidates forward in the pipeline.

Why it's cool: Hiring involves reading 200 resumes to find 10 worth interviewing. This automator reads all of them, scores them consistently against your criteria, and surfaces the best candidates — while sending personalized (not generic!) responses to everyone.

How to build it:

  1. Ingest applications: monitor an email inbox or integrate with an ATS (Lever, Greenhouse)
  2. For each application, extract: experience, skills, education, relevant projects
  3. Score against job requirements (configurable rubric): must-haves, nice-to-haves, red flags
  4. Top 20%: send personalized "we'd love to talk" email, schedule interview via Calendly link
  5. Middle 60%: add to "review" queue with AI summary for human decision
  6. Bottom 20%: send respectful, personalized rejection with specific feedback
AI-assisted hiring is sensitive territory. Always have a human review the top candidates before scheduling interviews. Use AI for the initial screen (reading 200 resumes) but never let it make final decisions. Bias in AI is real — audit your scoring rubric regularly and test with diverse sample applications.

Integration Patterns

Every automation needs a trigger. Here are the three patterns:

Webhooks (Push)

The source pushes data to your endpoint when something happens. Best for: GitHub, Stripe, Slack events, form submissions. Instant and efficient.

Polling (Pull)

Your automation checks the source periodically. Best for: email (IMAP), RSS feeds, APIs without webhook support. Use cron for scheduling.

Event Streams

Subscribe to a continuous stream of events. Best for: real-time monitoring, high-volume data. Use WebSockets, Kafka, or platform-specific event APIs.

# Polling pattern with schedule
import schedule

def check_for_new_emails():
    emails = fetch_unread_emails()
    for email in emails:
        classification = classify_email(email)
        route_email(email, classification)

schedule.every(5).minutes.do(check_for_new_emails)

# Webhook pattern with FastAPI
@app.post("/webhook/github")

The Orchestrator Pattern

Complex automations have multiple steps. The orchestrator pattern keeps them manageable:

Trigger → Orchestrator → Step 1 → Step 2 → Step N

The orchestrator is a central function that coordinates the workflow. It tracks which steps have completed, handles failures, and decides whether to continue or abort.

async def orchestrate_email_pipeline(email):
    """Central orchestrator for email processing."""
    try:
        # Step 1: Classify
        classification = await classify_email(email)
        log_step("classify", "success", classification)

        # Step 2: Extract data
        data = await extract_entities(email, classification)
        log_step("extract", "success", data)

        # Step 3: Route and act
        result = await route_email(email, classification, data)
        log_step("route", "success", result)

        # Step 4: Log for daily digest
        await add_to_digest(email, classification, data)
        log_step("digest", "success")

    except Exception as e:
        log_step("error", str(e))
        await alert_on_failure(email, e)  # Don't lose the email!

Error Recovery

What happens when step 3 of 5 fails? Three strategies:

  • Retry with backoff: Try again in 1s, 5s, 30s. Most failures are transient (API timeouts, rate limits).
  • Skip and continue: If one step is optional, skip it and continue the pipeline. Log the failure for later.
  • Dead letter queue: Move the failed item to a "needs attention" queue for manual review. Never lose data.

The Golden Rule of Automation

Never silently drop data. If an email can't be classified, put it in a "review" queue. If a release note can't be posted to Slack, save it to a file. If a candidate can't be emailed, flag them for manual outreach. Automation that loses data is worse than no automation at all.

Build It: Release Notes Generator

Let's build the Release Notes Generator. Tag a release and get formatted notes in 10 seconds.

Step 1: Set up (2 min)

mkdir release-notes && cd release-notes
python -m venv .venv && source .venv/bin/activate
pip install anthropic click

Step 2: Create releasenotes.py (10 min)

import click
import anthropic
import subprocess

client = anthropic.Anthropic()

SYSTEM = """You are a release notes writer. Given a list of git
commits between two tags, generate user-facing release notes.

Group into sections:
### New Features (things users will notice)
### Bug Fixes (things that were broken, now fixed)
### Improvements (performance, UX, internal upgrades)

Write for end users, not developers. Be specific but concise.
Each item should be one clear sentence.
End with a count of commits and contributors."""

def run(cmd):
    return subprocess.run(cmd, shell=True, capture_output=True, text=True).stdout.strip()

@click.command()
@click.argument("tag")
@click.option("--prev", default=None, help="Previous tag (auto-detected if omitted)")
@click.option("--post-slack", is_flag=True, help="Post to Slack webhook")
def main(tag, prev, post_slack):
    if not prev:
        # Get the tag before this one
        tags = run("git tag --sort=-version:refname").split("\n")
        current_idx = tags.index(tag) if tag in tags else 0
        prev = tags[current_idx + 1] if current_idx + 1 < len(tags) else ""

    if prev:
        commits = run(f"git log {prev}..{tag} --oneline")
        range_label = f"{prev}..{tag}"
    else:
        commits = run(f"git log {tag} --oneline -20")
        range_label = f"up to {tag}"

    if not commits:
        click.echo(f"No commits found for {range_label}")
        return

    click.echo(f"Generating notes for {range_label}...")

    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        system=SYSTEM,
        messages=[{"role": "user",
            "content": f"Tag: {tag}\nCommits:\n{commits}"}]
    )

    notes = response.content[0].text
    click.echo(f"\n## {tag}\n")
    click.echo(notes)

    if post_slack:
        import requests
        WEBHOOK = "your-slack-webhook-url"
        requests.post(WEBHOOK, json={"text": f"*Release {tag}*\n{notes}"})
        click.echo("\nPosted to Slack!")

if __name__ == "__main__":
    main()

Step 3: Test it (3 min)

# In any git repo with tags:
python releasenotes.py v1.0.0

# Or create a quick test:
git tag v0.1.0  # tag an earlier commit
# make some commits...
git tag v0.2.0
python releasenotes.py v0.2.0

Step 4: Make it a GitHub Action (bonus)

Add to .github/workflows/release.yml to auto-generate notes on every tag push.

You now have automatic release notes. Never write "bumped version" as your release description again.
