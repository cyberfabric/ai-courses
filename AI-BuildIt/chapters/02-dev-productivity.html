Developer Productivity Tools

Automate the boring parts of development so you can spend more time writing actual code

The Hidden Time Sink

Here's a number that should make you angry: developers spend 30-40% of their time NOT writing code.

Writing PR descriptions. Reviewing other people's PRs. Sitting in meetings and then manually turning notes into tickets. Updating documentation that nobody reads until someone yells "where are the docs?"

All of these are important. None of them are fun. And every single one of them can be automated with AI.

30-40% of dev time spent on non-coding tasks
~6 hrs per week writing PR descriptions & docs
5 tools to automate most of it

In Chapter 1, we built CLI tools that think. Now we're building tools that think about your workflow. These tools plug into the processes you already have — git hooks, CI pipelines, IDE extensions — and make them smarter.

Let's build.

The Projects

PR Description Generator
Weekend
Python Claude API Git GitHub CLI

Run prdesc in your branch, and it reads your git diff, analyzes the changes, and writes a detailed PR description with summary, motivation, list of changes, and test notes. Then optionally opens the PR for you.

What it does: Reads the diff between your branch and main, sends it to Claude with a structured prompt, and generates a PR description in your team's format.

Why it's cool: Be honest — how many of your PRs have a description that says "fixes stuff" or "WIP please review"? This tool writes the description your reviewers wish you wrote.

How to build it:

  1. Run git diff main...HEAD to get the full diff for the branch
  2. Run git log main..HEAD --oneline to get commit messages for context
  3. Send both to Claude with a system prompt defining your PR template
  4. Output the description to stdout, or use --open to create the PR via gh pr create
  5. Add a --template flag to load team-specific PR formats from a config file
Example Output
$ prdesc ## Summary Adds retry logic to the payment webhook handler to prevent duplicate charge failures during network timeouts. ## Motivation Users reported intermittent double-charges when webhook delivery failed and retried. The existing handler wasn't idempotent — it processed the same event multiple times. ## Changes - Added idempotency key check in `webhook_handler.py` - New `ProcessedEvents` table for deduplication - Retry logic with exponential backoff in `payment_service.py` - Unit tests for duplicate event scenarios ## Test Plan - [x] Unit tests for idempotent handler - [x] Integration test with mock webhook delivery - [ ] Manual test with Stripe test mode webhooks
Make the system prompt configurable. Store your team's PR template in ~/.prdesc/template.md so every developer on the team gets consistent, well-structured descriptions. Include sections your team actually uses — skip the ones they don't.
Code Review Prep Tool
Week
TypeScript Claude API GitHub API Node.js

Before you review a PR, run reviewprep 42 (where 42 is the PR number). It fetches the diff, analyzes the changes, and gives you a prep sheet: summary of what changed, potential issues it spotted, and questions to ask the author.

What it does: Fetches a PR from GitHub, analyzes the diff with Claude, and generates a structured review guide: what changed, why it matters, what looks risky, and what tests are missing.

Why it's cool: Code reviews are more effective when you understand the big picture before diving into line-by-line comments. This tool gives you that big picture in 10 seconds.

How to build it:

  1. Use the GitHub API (via octokit) to fetch the PR diff and description
  2. Send to Claude with a code review system prompt that asks for structured analysis
  3. Generate sections: Summary, Risk Areas, Missing Tests, Questions for Author
  4. Optionally post the prep as a PR comment with --post
  5. Add --focus security or --focus performance flags for specialized reviews

Meeting Notes to Tickets
Weekend
Python Claude API Jira API Linear API

Paste your messy meeting notes (or a transcript from Otter.ai / Fireflies), and get structured tickets with titles, descriptions, acceptance criteria, and assignees — ready to push to Jira or Linear.

What it does: Takes unstructured meeting notes or transcripts, extracts action items and decisions, and creates well-structured tickets with acceptance criteria.

Why it's cool: Every meeting generates action items. Most of them live in someone's head until they're forgotten. This tool captures them while they're fresh and turns them into trackable work.

How to build it:

  1. Accept input: file path, clipboard, or piped stdin
  2. First pass with Claude: extract action items, decisions, and owners
  3. Second pass: for each action item, generate a ticket with title, description, and acceptance criteria
  4. Output as JSON, Markdown, or push directly to Jira/Linear via API
  5. Add --assignees flag with a team roster to auto-assign based on names mentioned

Documentation Autopilot
Week
Python Claude API AST parsing TypeScript

Point it at your codebase and it finds every undocumented public function, class, and API endpoint. Then it generates docs, README sections, and usage examples — in your project's existing documentation style.

What it does: Scans source files for public APIs, detects which are undocumented, generates documentation that matches your existing style, and outputs it as docstrings, Markdown, or both.

Why it's cool: Documentation is always out of date because writing it is painful. This tool makes it painless. Run it after every sprint and your docs stay current.

How to build it:

  1. Use AST parsing (ast module in Python, ts-morph in TypeScript) to find public functions and classes
  2. Filter: which ones have docstrings/JSDoc? Which don't?
  3. For undocumented ones, read the source and send to Claude: "Write documentation for this function in the style of [existing example]"
  4. Output options: insert docstrings in-place, generate a Markdown file, or create a README section
  5. Add a --check mode for CI that fails if coverage drops below a threshold
AI-generated documentation is only as good as the code it reads. If your function is named do_stuff(x, y) with no types, the generated docs will be vague. Good naming and type hints make the docs dramatically better. Garbage in, garbage out.
Dependency Auditor
Weekend
Python Claude API pip / npm

Run depaudit in your project and get a clear report: what each dependency does, which are outdated, which have known vulnerabilities, and which have lighter alternatives you should consider.

What it does: Reads requirements.txt, package.json, or build.gradle, looks up each dependency, and generates a human-readable audit with actionable recommendations.

Why it's cool: Every project has that one dependency nobody remembers adding. And that other one that hasn't been updated in 3 years. And that massive library you're using for one tiny function. This tool finds all of them.

How to build it:

  1. Parse the dependency file to extract package names and versions
  2. For each dependency, gather info: latest version, last publish date, size, license
  3. Send the full list to Claude: "Audit these dependencies. For each: explain what it does, flag if outdated or risky, suggest alternatives if oversized."
  4. Generate a table with traffic-light ratings (green/yellow/red)
  5. Add a --ci flag that returns exit code 1 if any critical issues are found

Integrating With Your Workflow

The real power of these tools isn't running them manually. It's wiring them into the workflows you already have.

Git Hooks

Use a prepare-commit-msg hook to auto-generate PR descriptions, or a pre-push hook to run the dependency auditor before you ship:

# .git/hooks/prepare-commit-msg
#!/bin/bash
# Auto-generate commit message from staged changes
DIFF=$(git diff --cached)
if [ -n "$DIFF" ]; then
    SUGGESTION=$(echo "$DIFF" | prdesc --commit-mode)
    echo "$SUGGESTION" > "$1"
fi

CI/CD Pipeline

Add the Documentation Autopilot and Dependency Auditor to your CI pipeline. Fail the build if docs coverage drops or a vulnerable dependency appears:

# .github/workflows/ai-checks.yml
- name: Check documentation coverage
  run: docpilot --check --min-coverage 80

- name: Audit dependencies
  run: depaudit --ci --fail-on critical

IDE Extensions

Wrap any of these tools in a VS Code extension. The Code Review Prep tool is perfect for this — right-click a PR notification, get the prep sheet without leaving your editor.

Start with git hooks — they're the easiest integration point. No CI config, no extension scaffolding, just a shell script in .git/hooks/. You can always upgrade to CI or an extension later.

The "Review Before Commit" Pattern

Here's a pattern that will level up your entire team's code quality:

Write Code → AI Review → Fix Issues → Commit → Human Review

Before your code ever reaches a human reviewer, it goes through an AI review pass. The AI catches the obvious stuff — typos, missing error handling, inconsistent naming, missing tests — so the human reviewer can focus on architecture and business logic.

Build this as a pre-commit hook using the Code Review Prep tool's logic. The AI checks your staged changes and either passes or flags issues. If it flags something, you fix it before committing. Your human reviewers see cleaner code, spend less time on nitpicks, and can focus on what matters.

The Productivity Multiplier

Each of these tools saves 15-30 minutes per use. If you use three of them daily, that's over 5 hours per week of reclaimed development time. That's an extra day of actual coding every two weeks. Multiply by your team size and the math gets ridiculous.

Building VS Code Extensions With AI Backends

Want to go beyond CLI tools? Any of these projects can become a VS Code extension with a few extra steps:

  1. Scaffold with yo code (the Yeoman VS Code extension generator)
  2. Add a command that reads the current file or git diff
  3. Call your CLI tool as a subprocess, or embed the Claude API call directly
  4. Display results in a Webview panel, output channel, or inline decoration

The PR Description Generator as a VS Code extension: one click, and the PR description appears in a side panel. Edit it, then click "Create PR." The Code Review Prep tool: click a PR notification, see the prep sheet inline.

// VS Code extension command example
import * as vscode from 'vscode';
import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic();

export function activate(context: vscode.ExtensionContext) {
  const cmd = vscode.commands.registerCommand('prdesc.generate', async () => {
    const terminal = vscode.window.createTerminal('PR Desc');
    // Get diff, call Claude, show in webview panel
    const diff = await exec('git diff main...HEAD');
    const response = await client.messages.create({
      model: 'claude-sonnet-4-20250514',
      max_tokens: 1024,
      system: 'Generate a PR description from this diff...',
      messages: [{ role: 'user', content: diff }]
    });
    // Display in a webview panel
    const panel = vscode.window.createWebviewPanel('prdesc', 'PR Description', vscode.ViewColumn.Two);
    panel.webview.html = markdownToHtml(response.content[0].text);
  });
  context.subscriptions.push(cmd);
}

Build It: PR Description Generator

Let's build the PR Description Generator right now. This is the highest-value tool in this chapter — you'll use it on every PR from now on.

Step 1: Set up (2 min)

mkdir pr-desc && cd pr-desc
python -m venv .venv && source .venv/bin/activate
pip install anthropic click

Step 2: Create prdesc.py (10 min)

import click
import anthropic
import subprocess

client = anthropic.Anthropic()

SYSTEM = """You are a senior developer writing a PR description.
Given a git diff and commit messages, generate:

## Summary
1-2 sentence overview of the change.

## Motivation
Why this change is needed.

## Changes
Bulleted list of what changed and why.

## Test Plan
What was tested and how to verify.

Be specific. Reference file names and functions."""

def run(cmd):
    return subprocess.run(cmd, shell=True, capture_output=True, text=True).stdout

@click.command()
@click.option("--base", default="main", help="Base branch")
@click.option("--open", "open_pr", is_flag=True, help="Create PR via gh")
def main(base, open_pr):
    diff = run(f"git diff {base}...HEAD")
    commits = run(f"git log {base}..HEAD --oneline")

    if not diff.strip():
        click.echo("No changes found vs " + base)
        return

    prompt = f"Commits:\n{commits}\n\nDiff:\n{diff[:12000]}"

    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        system=SYSTEM,
        messages=[{"role": "user", "content": prompt}]
    )

    desc = response.content[0].text
    click.echo(desc)

    if open_pr:
        title = desc.split("\n")[0].replace("## Summary", "").strip()
        subprocess.run(
            ["gh", "pr", "create", "--title", title[:72], "--body", desc],
        )

if __name__ == "__main__":
    main()

Step 3: Test on a real branch (5 min)

# Make sure you're on a feature branch with changes
git checkout -b test-prdesc
# Make some changes, commit them
python prdesc.py
python prdesc.py --open  # Creates the actual PR!

Step 4: Hook it up (5 min)

Add to your shell aliases for instant access:

# ~/.bashrc or ~/.zshrc
alias prdesc="python ~/tools/pr-desc/prdesc.py"
alias prnew="python ~/tools/pr-desc/prdesc.py --open"

You now have AI-generated PR descriptions. Your reviewers will thank you.
