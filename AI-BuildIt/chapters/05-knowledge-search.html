Chapter 5

Knowledge Base & Search Projects

Build systems that make information findable using embeddings, RAG, and semantic search

Why Keyword Search Fails

You read an article last week about fixing memory leaks in Android. Now you need it. You search "memory leak Android" in your notes. Nothing. You search "out of memory." Nothing. "OOM." Still nothing.

The article was titled "Taming the Garbage Collector." No keyword match. Your information is there — you just can't find it.

This is the fundamental problem with keyword search: you have to remember the exact words, not just the concept. And humans don't think in keywords. We think in concepts, vibes, and half-remembered ideas.

Semantic search fixes this. Instead of matching exact words, it matches meaning. "That article about memory leaks" finds "Taming the Garbage Collector" because the AI understands they're about the same thing.

~60% of searches fail due to vocabulary mismatch
1536 dimensions in an embedding vector
<100ms to search 1M+ documents semantically

Embeddings in 60 Seconds

Here's the core idea: you can turn any text into a list of numbers (a vector). Texts with similar meaning get similar numbers. Then "searching" is just finding the closest vectors.

Text → Embedding Model → Vector [0.23, -0.41, ...] → Vector Database

When you search, your query gets turned into a vector too, and the database finds the stored vectors that are closest. "Memory leaks in Android" and "Taming the Garbage Collector" end up near each other in vector space because they mean similar things.

import voyageai  # or use OpenAI embeddings

vo = voyageai.Client()

# Turn text into a vector
result = vo.embed(["Taming the Garbage Collector"], model="voyage-3")
vector = result.embeddings[0]  # [0.023, -0.41, 0.87, ...] (1024 dims)

# Similar texts → similar vectors
# "memory leaks in Android" would produce a nearby vector
You don't need to understand the math behind embeddings to build with them. Think of it like GPS coordinates: two addresses that are "close" in meaning are close in coordinate space. The embedding model is the GPS that assigns coordinates to text.

In Chapter 4, we made data queryable through conversation. Now we're making knowledge searchable through meaning. Same AI-powered approach, different superpower.

  6. Add a --ask mode that sends retrieved chunks to Claude for a synthesized answer (that's RAG!)

Team FAQ Generator
Month
Python Claude API Slack API ChromaDB

Monitors your Slack or Teams channels, identifies questions that keep getting asked, and automatically generates a living FAQ with sourced answers. "How do I set up the dev environment?" appears for the third time? The FAQ entry writes itself.

What it does: Listens to team channels, detects question patterns, clusters similar questions, extracts the best answers, and maintains a searchable FAQ that updates automatically.

Why it's cool: Every team has tribal knowledge trapped in Slack threads. The same questions get asked every month by new joiners. This tool captures the answers once and makes them searchable forever.

How to build it:

  1. Connect to Slack using the Bolt framework, listen for messages in target channels
  2. Use Claude to classify each message: question, answer, discussion, or noise
  3. Embed questions and cluster similar ones (cosine similarity > 0.85 = same question)
  4. When a question cluster reaches 3+ instances, extract the best answer from the threads
  5. Generate a clean FAQ entry with the question, answer, and Slack thread links as sources
  6. Publish to a Notion page, Confluence, or static site that auto-updates
Start with a backfill: export your Slack channel history (Slack API can pull the last 90 days on free plans), process all of it, and generate an initial FAQ. You'll be surprised how many questions have already been answered three times.

Choosing a Vector Database

You need somewhere to store your embeddings. Here's a quick guide:

Database Best For Complexity
ChromaDB Getting started fast. Runs in-process, no server needed. Perfect for personal projects. Minimal
SQLite + extensions When you want everything in one file. Use sqlite-vss for vector search. Low
Qdrant Production use with filtering. Great performance, nice REST API, runs as a Docker container. Medium
Pinecone Fully managed cloud. Zero ops. Good when you don't want to manage infrastructure. Low (managed)
pgvector Already using PostgreSQL? Add vector search to your existing database. Low (if you have PG)
Start with ChromaDB. It's a pip install, runs in-process, and handles everything you need for personal projects. When you outgrow it, migrating to Qdrant or pgvector is straightforward — the embedding format is the same.

Chunking Strategies

Chunking is how you split documents before embedding them. Get it right and your search is precise. Get it wrong and you get irrelevant results or miss important context.

By Paragraphs

Simple and effective for articles and notes. Each paragraph becomes a chunk. Works well when paragraphs are self-contained.

By Headers/Sections

Best for documentation. Split at ## Header boundaries. Each section is a natural unit of meaning.

By Code Functions

For codebases, use Tree-sitter to split at function/class boundaries. Each function is a chunk with its docstring and signature.

Sliding Window

For dense text, use overlapping chunks of ~500 tokens with 100-token overlap. Prevents information from being split across chunks.

def chunk_text(text, chunk_size=500, overlap=100):
    """Split text into overlapping chunks."""
    words = text.split()
    chunks = []
    for i in range(0, len(words), chunk_size - overlap):
        chunk = " ".join(words[i:i + chunk_size])
        chunks.append(chunk)
    return chunks
Chunks that are too small lose context ("he" — who?). Chunks that are too large dilute the embedding (the vector represents too many topics). Aim for 200-500 tokens per chunk. Test with real queries to find the sweet spot for your data.

The RAG Pattern

RAG — Retrieval-Augmented Generation — is the pattern that makes all of these projects work. It's simple and powerful:

Query → Retrieve → Augment → Generate
  1. Query — the user asks a question in natural language
  2. Retrieve — embed the query, find the most relevant chunks from your vector database
  3. Augment — insert the retrieved chunks into the LLM prompt as context
  4. Generate — Claude reads the context and generates an answer, citing specific sources
# The RAG pattern in ~20 lines
import chromadb
import anthropic

chroma = chromadb.PersistentClient(path="./vault_db")
collection = chroma.get_collection("knowledge")
client = anthropic.Anthropic()

def ask(question):
    # Retrieve: find relevant chunks
    results = collection.query(query_texts=[question], n_results=5)
    context = "\n\n".join(results["documents"][0])
    sources = results["metadatas"][0]

    # Augment + Generate: answer using the context
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        system="Answer based ONLY on the provided context. Cite sources.",
        messages=[{"role": "user",
                   "content": f"Context:\n{context}\n\nQuestion: {question}"}]
    )
    return response.content[0].text, sources

Why RAG Beats Fine-Tuning

You might wonder: why not just fine-tune the model on your data? RAG is better for most cases because: it's always up to date (re-index, don't retrain), it cites sources (you can verify), it's cheap (no training costs), and it works with any model. Fine-tuning is for changing the model's behavior. RAG is for giving it knowledge.

What if you added a chat interface to your Knowledge Vault? Instead of one-off searches, have a conversation with your knowledge. "Tell me what I know about microservices." Follow up: "What are the trade-offs I've noted?" Then: "Based on my notes, should I use microservices for this new project?" The vault remembers context across the conversation, pulling in relevant chunks as the discussion evolves. You're having a conversation with your past self — the version of you that read all those articles.

Build It: Bookmark Intelligence

Let's build Bookmark Intelligence. Export your bookmarks and make them searchable by meaning in 20 minutes.

Step 1: Set up (2 min)

mkdir bookmark-intel && cd bookmark-intel
python -m venv .venv && source .venv/bin/activate
pip install anthropic chromadb trafilatura beautifulsoup4

Step 2: Export bookmarks

In Chrome: Bookmarks → Bookmark Manager → Export. Saves as bookmarks.html.

Step 3: Create bookmarks.py (12 min)

import click, anthropic, chromadb
from bs4 import BeautifulSoup
import trafilatura

client = anthropic.Anthropic()
chroma = chromadb.PersistentClient(path="./bookmarks_db")

@click.group()
def cli():
    pass

@cli.command()
@click.argument("html_file")
def ingest(html_file):
    """Import bookmarks from exported HTML."""
    collection = chroma.get_or_create_collection("bookmarks")
    soup = BeautifulSoup(open(html_file), "html.parser")
    links = soup.find_all("a")
    click.echo(f"Found {len(links)} bookmarks. Processing...")

    for i, link in enumerate(links[:100]):  # Start with first 100
        url = link.get("href", "")
        title = link.text.strip()
        if not url.startswith("http"):
            continue
        try:
            content = trafilatura.fetch_url(url)
            text = trafilatura.extract(content) or ""
            summary = summarize(title, text[:2000])
            collection.add(
                documents=[summary],
                metadatas=[{"url": url, "title": title}],
                ids=[f"bm-{i}"]
            )
            click.echo(f"  [{i+1}] {title[:50]}")
        except Exception:
            continue

    click.echo(f"Indexed {collection.count()} bookmarks.")

def summarize(title, text):
    r = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=200,
        messages=[{"role": "user",
            "content": f"Summarize in 2 sentences:\nTitle: {title}\n{text}"}]
    )
    return r.content[0].text

@cli.command()
@click.argument("query", nargs=-1)
def search(query):
    """Search bookmarks by meaning."""
    collection = chroma.get_collection("bookmarks")
    q = " ".join(query)
    results = collection.query(query_texts=[q], n_results=5)
    for doc, meta in zip(results["documents"][0], results["metadatas"][0]):
        click.echo(f"\n{meta['title']}")
        click.echo(f"  {doc}")
        click.echo(f"  {meta['url']}")

if __name__ == "__main__":
    cli()

Step 4: Index and search (5 min)

# Import your bookmarks
python bookmarks.py ingest bookmarks.html

# Search by meaning
python bookmarks.py search "database performance optimization"
python bookmarks.py search "that article about React state management"
python bookmarks.py search "debugging production issues"

Your bookmarks are now searchable by meaning. Try searching for concepts you remember reading about — you'll be amazed at what you rediscover.
