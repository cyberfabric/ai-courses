API Design & Microservices

Build backend services that use LLMs for intelligent processing behind the scenes

The AI Microservice Pattern

In Chapter 7, we built web apps where users see the AI. Now we're going behind the curtain. AI microservices are the invisible intelligence layer — other services call them, users never know AI is involved.

The pattern is beautifully simple:

HTTP Request → Validate Input → Build Prompt → Call LLM → Parse + Return JSON

A thin API wrapper around a structured LLM call. The consumer sends data, gets structured JSON back, and never knows there's an LLM involved. It looks like any other API — except it can understand natural language, classify intent, extract meaning, and make judgments that would take thousands of lines of hand-coded rules.

~50 lines of code for a working AI microservice
1-3 sec typical response time
$0.001 per request with Haiku

Think about all the places in your backend where you have messy if/else chains, regex patterns, or keyword matching. Every one of those is a candidate for an AI microservice that actually understands the data instead of pattern-matching it.

The Projects

Smart Webhook Processor
Week
Python Claude API FastAPI Redis

A universal webhook receiver that routes and processes incoming webhooks based on natural language rules instead of hard-coded logic. Define rules like "if it's a payment failure, alert the finance Slack channel and create a Jira ticket" — in English, not code.

What it does: Receives webhooks from any source (Stripe, GitHub, Shopify, etc.), uses AI to classify the event, match it against your rules, and trigger the appropriate actions — all configured in plain English.

Why it's cool: Every webhook integration starts simple and becomes a tangled mess of if/else. This approach lets you define routing rules in English and add new ones without writing code. "If someone opens a high-priority issue, page the on-call engineer" just works.

How to build it:

  1. Build a FastAPI endpoint that accepts any JSON webhook payload
  2. Store routing rules in a config file as natural language statements
  3. On each webhook: send the payload + rules to Claude: "Which rules match this event? Return the matching rule IDs."
  4. For each matched rule, execute the associated action (Slack message, Jira ticket, email, HTTP call)
  5. Log every decision for auditability
  6. Add a web UI to manage rules without editing config files
Webhook Processor Config
# webhook_rules.yaml rules: - id: payment-failure match: "Payment failed or charge was declined" actions: - slack: "#finance-alerts" - jira: {project: "PAY", type: "Bug", priority: "High"} - id: new-signup match: "A new user signed up or created an account" actions: - slack: "#growth" - email: "welcome-sequence" - id: deploy-complete match: "A deployment or release was completed successfully" actions: - slack: "#engineering" - id: critical-error match: "A server error, crash, or 5xx status occurred" actions: - pagerduty: {severity: "critical"} - slack: "#incidents"

Send a raw email to this API and get back structured intelligence: intent (question, request, complaint, FYI), urgency (low/medium/high/critical), required action, key entities, and a suggested reply draft.

What it does: Accepts raw email text (or parsed subject + body) and returns a structured JSON analysis with intent classification, urgency scoring, entity extraction, action items, and an optional reply draft.

  4. Add a POST /batch endpoint for analyzing multiple emails at once
  5. Include confidence scores for each classification

Content Moderation API
Weekend
Python Claude API FastAPI

Goes way beyond keyword matching. This API understands context, sarcasm, coded language, and subtle policy violations. Returns severity scores, specific policy violations, and recommended actions.

What it does: Accepts user-generated content (comments, posts, messages) and returns a moderation decision: safe, warning, or block — with specific reasons, severity scores, and the policy rules that were violated.

Why it's cool: Keyword-based moderation is a joke. "I'm going to kill this presentation" gets flagged. Actual harassment with no banned words gets through. AI understands meaning, not just words.

How to build it:

  1. Define your moderation policy in plain English (the system prompt)
  2. Build a POST /moderate endpoint accepting content text
  3. Use Claude to analyze: harassment, hate speech, threats, spam, NSFW, self-harm
  4. Return: decision (allow/warn/block), categories triggered, severity (1-10), reasoning
  5. Add context awareness: include the conversation thread for better judgment
  6. Build a review queue for edge cases (severity 4-7) that need human review
Keyword Moderation
"I'm going to kill this presentation" → BLOCKED (contains "kill") ✗ False positive "You're so worthless, everyone hates you" → ALLOWED (no banned words) ✗ Missed harassment "Nice job, Einstein" (sarcastically) → ALLOWED (positive words) ✗ Missed bullying

Translation API With Context
Week
Python Claude API FastAPI

Not just word-for-word translation. This API accepts domain context and a glossary, producing translations that match the register (formal, casual, technical) and use your specific terminology consistently.

What it does: Translates text while respecting: domain context (medical, legal, tech), tone register (formal letter vs. casual chat), custom glossary terms, and cultural adaptation.

Why it's cool: Generic translation APIs translate "controller" the same way whether it's a game controller or an MVC controller. This API knows the difference because you tell it the context.

How to build it:

  1. Build a POST /translate endpoint with: text, source lang, target lang, domain, glossary
  2. Include the glossary in the system prompt: "Always translate 'sprint' as 'Sprint' (keep English), 'deploy' as 'mise en production'"
  3. Add register detection: infer formality from the source text or accept it as a parameter
  4. Return: translated text, glossary terms used, confidence, alternative translations for ambiguous terms
  5. Add batch translation with consistent terminology across all texts
Data Enrichment Service
Weekend
Python Claude API FastAPI

Send a company name, get back structured data: what they do, industry, estimated size, tech stack, recent news, and key people. Perfect for CRM enrichment, lead scoring, or competitive analysis.

What it does: Takes a company name (or URL) and returns structured data extracted from Claude's training knowledge: description, industry, products, tech stack, funding status, and competitors.

Why it's cool: CRM data enrichment services charge $0.10-$1.00 per lookup. This API does it for $0.001. It's not as fresh as a paid service, but for initial enrichment and lead qualification, it's incredibly effective.

How to build it:

  1. Build a POST /enrich endpoint accepting a company name or domain
  2. Use Claude to extract: description, industry, size estimate, founded year, HQ location, key products, tech stack, competitors
  3. Force structured JSON output using tool use
  4. Cache results (company data doesn't change daily) to reduce costs
  5. Add batch enrichment for processing CSV lists of companies
  6. Include a confidence score — Claude knows "Google" well but may hallucinate about "Bob's Local Bakery"
Claude's knowledge has a cutoff date. For recent news, funding rounds, or current employee counts, you'll need to supplement with real-time data sources (web search, LinkedIn API, Crunchbase). Use Claude for the analysis and interpretation, external APIs for fresh data.

Structured Output: Getting JSON Every Time

AI microservices must return consistent, parseable JSON. Here's how to guarantee it:

Tool Use (Best Approach)

Claude's tool use feature forces structured output by defining a schema the model must follow:

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    tools=[{
        "name": "email_analysis",
        "description": "Analyze an email and return structured data",
        "input_schema": {
            "type": "object",
            "properties": {
                "intent": {"type": "string", "enum": ["question", "request",
                           "complaint", "fyi", "urgent"]},
                "urgency": {"type": "string", "enum": ["low", "medium",
                            "high", "critical"]},
                "action_required": {"type": "string"},
                "entities": {"type": "object"},
                "suggested_reply": {"type": "string"}
            },
            "required": ["intent", "urgency", "action_required"]
        }
    }],
    tool_choice={"type": "tool", "name": "email_analysis"},
    messages=[{"role": "user", "content": email_text}]
)

# Guaranteed structured output
result = response.content[0].input  # Already a dict!
Tool use is the gold standard for structured output in AI microservices. The schema acts as a contract — just like a typed API response. Your consumers always get the fields they expect, in the types they expect. No more parsing markdown or hoping the JSON is valid.

Error Handling: When the LLM Returns Garbage

It will happen. The LLM will occasionally return something unexpected. Build for it:

import json
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(min=1, max=10))
def call_ai_service(input_data):
    response = client.messages.create(...)

    # Validate the response
    result = response.content[0].input
    if not result.get("intent"):
        raise ValueError("Missing required field: intent")
    if result["urgency"] not in ["low", "medium", "high", "critical"]:
        raise ValueError(f"Invalid urgency: {result['urgency']}")

    return result

Key principles: retry on failure (most issues are transient), validate the response against your schema, fall back to a safe default if all retries fail, and log every failure for monitoring.

Rate Limiting and Queuing

AI API calls take 1-3 seconds. If 100 requests hit your service simultaneously, you'll exhaust your API rate limit. Use a queue:

Request In → Queue (Redis) → Worker Pool → Claude API → Response Out

For simple services, use Python's asyncio.Semaphore to limit concurrent calls. For production, use Redis + Celery or a proper message queue. The goal: never hit Claude's rate limit, never drop a request.

Testing AI APIs

Testing AI services is different from testing regular APIs because the output is non-deterministic. Here's the playbook:

Snapshot Tests

Save known-good responses as snapshots. On each test run, compare the structure (not the exact text) against the snapshot. Fields present? Types correct? Enums valid?

Evaluation Sets

Build a set of 20-50 inputs with expected outputs. Run them periodically and track accuracy. If it drops below your threshold, investigate.

# eval_set.json
[
  {
    "input": "Hi, invoice #123 is overdue. Please pay by Friday.",
    "expected": {"intent": "request", "urgency": "high"}
  },
  {
    "input": "Just FYI, the meeting is moved to 3pm.",
    "expected": {"intent": "fyi", "urgency": "low"}
  }
]

# Run evaluations
correct = 0
for case in eval_set:
    result = call_api(case["input"])
    if result["intent"] == case["expected"]["intent"]:
        correct += 1
accuracy = correct / len(eval_set)
assert accuracy >= 0.9, f"Accuracy dropped to {accuracy}"

Build It: Email Intelligence API

Let's build the Email Intelligence API. We'll have it classifying real emails in 15 minutes.

Step 1: Set up (2 min)

mkdir email-intel && cd email-intel
python -m venv .venv && source .venv/bin/activate
pip install anthropic fastapi uvicorn

Step 2: Create app.py (10 min)

from fastapi import FastAPI
import anthropic

app = FastAPI(title="Email Intelligence API")
client = anthropic.Anthropic()

@app.post("/analyze")
def analyze_email(data: dict):
    email_text = f"From: {data.get('from','')}\n"
    email_text += f"Subject: {data.get('subject','')}\n"
    email_text += f"Body: {data.get('body','')}"

    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        tools=[{
            "name": "email_analysis",
            "description": "Analyze email and return structured data",
            "input_schema": {
                "type": "object",
                "properties": {
                    "intent": {"type": "string", "enum":
                        ["question","request","complaint","fyi","urgent"]},
                    "urgency": {"type": "string", "enum":
                        ["low","medium","high","critical"]},
                    "sentiment": {"type": "string"},
                    "action_required": {"type": "string"},
                    "suggested_reply": {"type": "string"},
                    "entities": {"type": "object"}
                },
                "required": ["intent","urgency","action_required"]
            }
        }],
        tool_choice={"type": "tool", "name": "email_analysis"},
        messages=[{"role": "user", "content": email_text}]
    )
    return response.content[0].input

Step 3: Run and test (3 min)

uvicorn app:app --reload &

# Test with curl
curl -X POST http://localhost:8000/analyze \
  -H "Content-Type: application/json" \
  -d '{"from":"boss@company.com","subject":"Urgent: deploy broken",
       "body":"The production deploy failed. Customers are seeing errors.
       Please roll back immediately."}'

You've got a working Email Intelligence API. Test it with 10 different email types — you'll be surprised how accurately it classifies intent and urgency.

Pick Your Project

Five backend services, each one an invisible intelligence layer:

  • Smart Webhook Processor — if your webhook handlers are spaghetti code (Week)
  • Email Intelligence API — if you process emails and need structured data (Weekend)
  • Content Moderation API — if keyword filters cause more problems than they solve (Weekend)
  • Translation API With Context — if generic translations miss your domain's nuances (Week)
  • Data Enrichment Service — if you need company data without paying $1 per lookup (Weekend)

My recommendation? Start with the Email Intelligence API. It's a weekend build, the exercise above gets you running in 15 minutes, and it teaches you the tool use pattern for structured output — the most important technique in this chapter. Once you can force structured JSON from Claude, every other microservice is the same pattern with a different schema.
