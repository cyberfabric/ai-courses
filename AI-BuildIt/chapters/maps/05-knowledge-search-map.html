💡 The Core Insight

🔍

Search by Meaning, Not by Words

Keyword search fails because you have to remember the exact words, not just the concept. Semantic search matches meaning — "that article about memory leaks" finds "Taming the Garbage Collector" because AI understands they're about the same thing. ~60% of searches fail due to vocabulary mismatch. Embeddings fix this.

🧬 How Embeddings Work

Turn any text into a list of numbers (a vector). Similar meaning → similar numbers → searchable.

Think of embeddings like GPS coordinates: two texts that are "close" in meaning are close in coordinate space. The embedding model is the GPS that assigns coordinates to text.

RAG vs Fine-Tuning

RAG wins for most use cases: always up to date (re-index, don't retrain), cites sources (verifiable), cheap (no training costs), works with any model. Fine-tuning changes behavior; RAG gives knowledge.

✂️ Chunking Strategies

How you split documents before embedding determines search quality. Too small = lost context. Too large = diluted meaning. Aim for 200-500 tokens per chunk.

  1. 📄 By Paragraphs

    Simple and effective for articles and notes. Each paragraph = one chunk. Best when paragraphs are self-contained.

  2. 📑 By Headers / Sections

    Best for documentation. Split at ## Header boundaries — each section is a natural unit of meaning.

  3. 🔧 By Code Functions

    For codebases: use Tree-sitter to split at function/class boundaries. Each function with its docstring and signature.

  4. 🪟 Sliding Window

    For dense text: overlapping chunks of ~500 tokens with 100-token overlap. Prevents info from being split across chunks.

📖 Key Terms

Embedding
A numerical representation (vector) of text where similar meanings produce similar numbers.
Vector
A list of numbers (e.g., 1536 dimensions) that represents a piece of text in "meaning space."
Cosine Similarity
A measure of how "close" two vectors are — higher similarity = more related meaning.
RAG (Retrieval-Augmented Generation)
The pattern of retrieving relevant context from a database and injecting it into an LLM prompt before generating an answer.
Chunking
Splitting documents into smaller pieces before embedding — the unit of retrieval for search.
Vector Database
A database optimized for storing and searching vectors by similarity (e.g., ChromaDB, Qdrant, Pinecone).
Semantic Search
Finding information by meaning rather than exact keyword matches — powered by embeddings.

🧪 Quick Quiz

Test Your Knowledge

Why does keyword search fail for "that article about memory leaks"?

Because the article might be titled "Taming the Garbage Collector" — no keyword overlap. Semantic search matches meaning, so it finds the article even without shared words.

What are the four steps of the RAG pattern?

Query (user asks), Retrieve (embed query, find closest chunks), Augment (inject chunks into prompt), Generate (LLM answers citing sources).

Why is RAG usually better than fine-tuning for giving a model knowledge?

RAG is always up to date (re-index, don't retrain), cites verifiable sources, costs nothing to "train," and works with any model. Fine-tuning changes behavior; RAG gives knowledge.

What's the ideal chunk size, and what happens if chunks are too small or too large?

Aim for 200-500 tokens. Too small: chunks lose context ("he" — who?). Too large: the vector represents too many topics, diluting the embedding.

Which vector database should you start with and why?

ChromaDB — it's a single pip install, runs in-process with no server, and handles everything for personal projects. Migrating later is easy since the embedding format is universal.
