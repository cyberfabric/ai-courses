Chapter 1

CLI Tools That Think

Build AI-powered command-line tools that understand plain English and supercharge your terminal

Why the CLI Is the Perfect Home for AI

Here's a hot take: the humble command line is the single best place to start building AI-powered tools. Not a web app. Not a mobile app. The terminal.

Why? Because the CLI gives you superpowers that other interfaces don't:

  • Speed — no UI to build, no CSS to fight, no deploy pipeline. Just code and run.
  • Scriptable — your tool becomes part of every automation you'll ever write.
  • Composable — pipe the output of one AI tool into another. Unix philosophy meets LLMs.
  • Pipe-friendly — cat error.log | your-ai-tool just works.
  • Low friction — no auth screens, no loading spinners, no cookie banners. Just results.

Think about it: every developer already lives in the terminal. Your AI tools should meet them there.

~70% of dev time spent in terminal
30 min to build your first AI CLI tool
0 frontend frameworks required

The Anatomy of an AI CLI Tool

Every AI-powered CLI tool follows the same basic pattern. Once you see it, you can build anything:

Parse Input → Build Prompt → Call LLM → Structure Output → Take Action

Let's break that down:

  1. Parse Input — read arguments, flags, or piped stdin. Libraries like argparse (Python) or commander (Node) handle this.
  2. Build Prompt — combine the user's input with a system prompt that defines your tool's personality and output format.
  3. Call LLM — send the prompt to Claude API and get a response. That's one API call.
  4. Structure Output — parse the response into something useful: JSON, a formatted table, or a runnable command.
  5. Take Action — print, copy to clipboard, execute the command, or write to a file.
The magic is in step 2. A well-crafted system prompt turns a generic LLM call into a specialized tool. You're not building an AI — you're building a wrapper that makes AI useful for one specific job.

Here's the skeleton in Python — every project in this chapter starts from something like this:

import anthropic
import sys

client = anthropic.Anthropic()

# Read from stdin or arguments
user_input = sys.stdin.read() if not sys.stdin.isatty() else " ".join(sys.argv[1:])

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    system="You are a [specific tool]. Output [specific format].",
    messages=[{"role": "user", "content": user_input}]
)

print(response.content[0].text)

That's it. Seriously. Everything from here is just making that skeleton smarter for specific use cases.

The Projects

Time to build. Here are five CLI tools you can ship this weekend. Each one solves a real problem, and each one is a building block for something bigger.

Smart Git Companion
Weekend
Python Claude API subprocess Git

Type gitai "show me what changed in auth last week" and get the actual git commands executed with results. No more googling git log --since --author --pretty flag combinations.

What it does: Translates natural language questions about your repo into the right git commands, runs them, and presents the results in a human-readable format.

Why it's cool: You know git has the answer — you just never remember the exact command. This tool does.

How to build it:

  1. Accept a natural language query as the argument
  2. Send to Claude with a system prompt: "You are a git expert. Given a question, return ONLY the git command(s) needed. Format as JSON: {commands: [string]}"
  3. Parse the JSON, execute each command with subprocess.run()
  4. Send the raw output back to Claude to summarize in plain English
  5. Print both the commands used and the summary
Example Usage
$ gitai "who touched the payments module this month?" Running: git log --since="2026-03-01" -- src/payments/ --format="%an" | sort | uniq -c | sort -rn Results: 12 Alice Chen 7 Bob Martinez 3 You Summary: Alice has been the most active in payments this month with 12 commits, primarily around the refund flow refactor.
Log Detective
Weekend
Python Claude API stdin pipes

Pipe any log file into this tool and get root cause analysis, not just pattern matching. cat server.log | logdetective tells you why your server crashed, not just that it crashed.

What it does: Reads log output from stdin, identifies errors, traces the root cause through the log timeline, and suggests specific fixes.

Why it's cool: Traditional log analysis means ctrl+F for "ERROR" and reading backwards. This tool reads the whole story and tells you the plot.

How to build it:

  1. Read from stdin (pipe-friendly by default)
  2. If the log is huge, chunk it — send the last 500 lines plus any lines containing ERROR/WARN/FATAL
  3. System prompt: "You are a log analyst. Identify the root cause, the chain of events, and suggest fixes. Be specific — reference line numbers and timestamps."
  4. Add a --fix flag that also generates a patch or config change
Be careful piping sensitive logs (credentials, PII) through external APIs. Add a --redact flag that strips emails, IPs, and tokens before sending. Your users will thank you.
Config Generator
Weekend
Python Claude API YAML/TOML

Describe what you want in English, get a production-ready config file. confgen "nginx reverse proxy for a Node app on port 3000 with SSL and rate limiting" — done.

What it does: Generates config files for nginx, Docker, Docker Compose, Kubernetes, Gradle, GitHub Actions, and more — from plain English descriptions.

Why it's cool: Config files are 90% boilerplate and 10% "where does this one setting go?" This tool nails both.

How to build it:

  1. Detect the config type from the description (or accept a --type flag)
  2. System prompt includes best practices for each config type
  3. Output the config with inline comments explaining each section
  4. Add a --validate flag that checks syntax before writing
  5. Write to file with --output or print to stdout
Without Config Generator

With Config Generator
$ confgen "nginx reverse proxy, port 3000, SSL with Let's Encrypt, rate limit 100 req/min" --output nginx.conf ✓ Generated nginx.conf (47 lines) ✓ Includes SSL configuration ✓ Rate limiting: 100 req/min ✓ Syntax validated
Smart File Organizer
Weekend
Python Claude API pathlib shutil

Point it at your messy Downloads folder and watch it rename, sort, and organize files by content, not just extension. PDFs about taxes go to finance/, screenshots of bugs go to bugs/, random memes go to memes/.

What it does: Analyzes file names, metadata, and (optionally) content to intelligently categorize and rename files into a clean directory structure.

Why it's cool: Your Downloads folder is a graveyard of Screenshot_2026-04-01_14.32.15.png and document(3).pdf. This tool gives it a proper burial.

How to build it:

  1. Scan the target directory and collect file names + metadata
  2. For text-based files, extract the first few hundred characters
  3. Send file info to Claude: "Suggest a directory structure and new names for these files. Return JSON."
  4. Show a preview (--dry-run by default — never move files without confirmation)
  5. On confirmation, execute the moves and renames
Always default to --dry-run mode for tools that modify files. Show the user what would happen, then ask for confirmation. This builds trust and prevents disasters. One wrong move command and someone's thesis is gone.
Terminal Translator
Weekend
Python Claude API Click / Typer

Type what you want to do in plain English, get the exact bash command. tt "find all Python files larger than 1MB modified this week" returns the command, and with --explain it breaks down every flag.

What it does: Translates natural language descriptions of terminal tasks into the precise bash/zsh commands, with optional explanations and direct execution.

Why it's cool: You know there's a find command that does what you want. You just can't remember if it's -mtime or -mmin or -newer. This tool remembers for you.

How to build it:

  1. Accept natural language as arguments
  2. Detect the current shell and OS for platform-specific commands
  3. System prompt: "You are a terminal expert. Return ONLY the command. No markdown, no explanation, just the command."
  4. Add flags: --explain (break down each part), --run (execute directly), --copy (clipboard)
  5. For --run, always show the command first and ask for confirmation

How to Distribute Your CLI Tool

You built something cool. Now let other people use it. Here are three ways to ship a CLI tool:

Python: pip install

Use pyproject.toml with a [project.scripts] entry. Publish to PyPI and anyone can pip install your-tool. Use uv for even faster installs.

[project.scripts]
logdetective = "logdetective.cli:main"
tt = "terminal_translator.cli:main"

Node.js: npm install -g

Add a "bin" field to package.json. Publish to npm and it's npm install -g your-tool away.

{
  "bin": {
    "gitai": "./dist/cli.js"
  }
}

Homebrew Tap

Create a GitHub repo called homebrew-tools, add a Ruby formula, and users get brew install yourusername/tools/your-tool. Fancy? Yes. Worth it for tools you share with your team? Absolutely.

The Composability Advantage

Here's where CLI tools get really powerful. Because they read from stdin and write to stdout, you can chain them:

# Analyze logs AND get fix suggestions
cat server.log | logdetective --json | confgen --from-suggestions

# Find what changed, then explain it
gitai "changes to auth this week" | tt "create a summary email from this"

# Organize, then verify
fileorg ~/Downloads --dry-run | tt "create a shell script from these moves"

Each tool does one thing well. Together, they're an AI-powered pipeline. This is the Unix philosophy upgraded for the LLM era.

The Golden Rule of AI CLI Tools

Read from stdin, write to stdout, use stderr for status messages. If your tool follows this convention, it plays nicely with every other tool in the terminal — including your other AI tools.

Performance Tips

AI CLI tools have one big weakness: latency. An API call takes 1-3 seconds. Here's how to keep things snappy:

Stream Responses

Don't wait for the full response. Stream tokens as they arrive so the user sees output immediately:

with client.messages.stream(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    messages=[{"role": "user", "content": user_input}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)

Cache Aggressively

If someone asks the same question twice, don't hit the API again. Use a simple file-based cache keyed on the input hash:

import hashlib, json, os

cache_dir = os.path.expanduser("~/.cache/your-tool/")
cache_key = hashlib.md5(prompt.encode()).hexdigest()
cache_file = os.path.join(cache_dir, cache_key)

if os.path.exists(cache_file):
    result = open(cache_file).read()
else:
    result = call_claude(prompt)
    os.makedirs(cache_dir, exist_ok=True)
    open(cache_file, "w").write(result)

Choose the Right Model

Not every task needs the most powerful model. A simple command translation? Haiku is fast and cheap. Complex log analysis? Sonnet has you covered. Save Opus for when you truly need deep reasoning.

Pro move: add a --fast flag that uses Haiku for quick tasks and a --think flag that uses extended thinking for complex analysis. Let the user choose the speed/quality tradeoff.

Build It: Terminal Translator in 30 Minutes

Let's build the Terminal Translator right now. Open your terminal and follow along. You'll have a working tool in 30 minutes.

Step 1: Set up the project (2 min)

mkdir terminal-translator && cd terminal-translator
python -m venv .venv && source .venv/bin/activate
pip install anthropic click

Step 2: Create the CLI (10 min)

Create tt.py with this structure:

import click
import anthropic
import subprocess

client = anthropic.Anthropic()

SYSTEM = """You are a terminal command expert. Given a natural language
description, return ONLY the exact command. No markdown, no backticks,
no explanation. Just the raw command."""

EXPLAIN_SYSTEM = """You are a terminal command expert. Given a natural
language description:
1. First line: the exact command
2. Blank line
3. Then explain each part with → arrows"""

@click.command()
@click.argument("description", nargs=-1, required=True)
@click.option("--explain", is_flag=True, help="Explain each part")
@click.option("--run", is_flag=True, help="Execute the command")
def main(description, explain, run):
    query = " ".join(description)
    system = EXPLAIN_SYSTEM if explain else SYSTEM

    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=512,
        system=system,
        messages=[{"role": "user", "content": query}]
    )

    output = response.content[0].text
    print(output)

    if run:
        cmd = output.split("\n")[0]
        if click.confirm(f"\nExecute: {cmd}?"):
            subprocess.run(cmd, shell=True)

if __name__ == "__main__":
    main()

Step 3: Test it (5 min)

python tt.py "list all files modified today"
python tt.py "find processes using port 8080" --explain
python tt.py "count lines of code in this project" --run

Step 4: Make it a real command (5 min)

Add a pyproject.toml, set up the script entry point, and pip install -e . — now you can use tt from anywhere.

Bonus: add clipboard support (3 min)

# pip install pyperclip
@click.option("--copy", is_flag=True, help="Copy to clipboard")
# In main():
if copy:
    import pyperclip
    pyperclip.copy(output.split("\n")[0])
