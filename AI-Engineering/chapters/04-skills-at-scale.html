Skills That Scale: Your Team's Command Library

Build a shared skill library with categories, parameters, and quality standards

From Ad-Hoc Prompts to Reusable Commands

In Chapters 2 and 3, you built the brain (CLAUDE.md) and the memory (context layer). Now it's time to build the muscle — skills.

A skill is a reusable prompt stored as a Markdown file in your project's .claude/commands/ directory. When you type /skill-name in Claude Code, it loads that file as a prompt and executes it. Think of skills as functions for AI — named, parameterized, documented, and version-controlled.

Without skills, your team re-invents the same complex prompts over and over. "How do I scaffold a new feature again?" "What was the review prompt we used last time?" Skills turn tribal knowledge into shared infrastructure.

The Shift: Individual developers write prompts. Teams write skills. The difference is the same as writing one-off scripts vs building a CLI tool — skills are tested, documented, and shared through version control.

Skill Library Architecture

A professional skill library has structure. Here's the directory layout that scales from 5 skills to 50:

.claude/commands/ — skill library structure
.claude/
  commands/
    # Scaffolding skills — create new code structures
    scaffold-feature.md
    scaffold-test-suite.md
    scaffold-module.md

    # Review skills — analyze existing code
    run-review.md
    review-pr.md
    check-architecture.md

    # Testing skills — generate and run tests
    generate-tests.md
    generate-integration-test.md

    # Documentation skills — generate docs
    document-module.md
    generate-api-docs.md

    # Migration skills — transform code patterns
    migrate-to-compose.md
    migrate-to-stateflow.md

    # Workflow skills — multi-step processes
    prepare-release.md
    generate-next-chapter.md

Naming Conventions

Consistent naming is what makes a library navigable. Follow this pattern:

Category Prefix Examples
Scaffolding scaffold- scaffold-feature, scaffold-module, scaffold-test-suite
Review review- or check- run-review, review-pr, check-architecture
Testing generate-test- generate-tests, generate-integration-test
Documentation document- document-module, generate-api-docs
Migration migrate- migrate-to-compose, migrate-to-stateflow
Workflow verb-noun prepare-release, generate-next-chapter
Tip: Use kebab-case for skill file names. The filename becomes the command: scaffold-feature.md is invoked as /scaffold-feature. Keep names short but descriptive — you'll type them often.

Anatomy of a Well-Designed Skill

Let's build a real skill step by step. A scaffold skill for new features is the most common starting point:

.claude/commands/scaffold-feature.md
Scaffold a new feature module for the recipes app.

## Arguments
Feature name: $ARGUMENTS (e.g., "nutrition", "shopping-list")

## Steps

1. Read CLAUDE.md for architecture rules and module structure
2. Read docs/architecture/presentation-layer-ctx.md for UI conventions
3. Read docs/architecture/package-structure-ctx.md for package layout

4. Create the feature module directory:
   - feature/$ARGUMENTS/src/main/kotlin/com/recipes/feature/$ARGUMENTS/

5. Create the following files:
   - presentation/{Name}Screen.kt — Composable with Modifier parameter
   - presentation/{Name}ViewModel.kt — MVI with StateFlow<UiState>
   - presentation/{Name}UiState.kt — data class
   - presentation/{Name}Event.kt — sealed interface
   - domain/Get{Name}UseCase.kt — single-responsibility use case
   - domain/{Name}Repository.kt — repository interface
   - data/{Name}RepositoryImpl.kt — repository implementation with @Inject

6. Create build.gradle.kts with dependencies on :core:domain and :core:ui

7. Register the module in settings.gradle.kts

8. Create test files:
   - presentation/{Name}ViewModelTest.kt with 3 test cases
   - domain/Get{Name}UseCaseTest.kt with success, error, edge case

9. Run `./gradlew :feature:$ARGUMENTS:testDebugUnitTest` to verify

Notice the structure: context loading first (steps 1-3), then creation (steps 4-7), then verification (steps 8-9). This pattern — read, create, verify — works for virtually any skill.

Parameterized Skills

The $ARGUMENTS variable is what makes skills reusable. Whatever text follows the skill name becomes $ARGUMENTS:

Invoking a parameterized skill
/scaffold-feature nutrition → $ARGUMENTS = "nutrition" → Creates :feature:nutrition module with all files
One skill, infinite features. The parameter makes it reusable.

Parameters can be more complex than a single word. Design your skills to parse structured arguments when needed:

.claude/commands/generate-tests.md
Generate comprehensive tests for the specified file.

## Arguments
$ARGUMENTS — the file path to generate tests for (e.g., "feature/recipes/presentation/RecipesViewModel.kt")

## Steps
1. Read the target file: $ARGUMENTS
2. Read CLAUDE.md for testing standards
3. Read docs/architecture/testing-ctx.md for test conventions
4. Identify the layer (presentation/domain/data) from the file path
5. Generate test file in the corresponding test directory:
   - Match the package structure
   - Use test naming: `should {expected} when {condition}`
   - Use fakes from :core:testing, not mocks
   - Include at least: success case, error case, edge case
   - For ViewModels: test state transitions
   - For UseCases: test input/output mapping
   - For Repositories: test data source interaction
6. Run the test file to verify it compiles and passes
Warning: Don't over-parameterize. If a skill needs more than 2-3 pieces of information, AI should ask follow-up questions rather than parsing a complex argument string. Skills should be easy to invoke, not require a manual to use.

Skill Composition: The Macro Pattern

Advanced skills can reference other skills, creating multi-step workflows. This is the "macro skill" pattern — one skill orchestrates several others:

.claude/commands/prepare-release.md
Prepare the project for a release build.

## Steps
1. Run the equivalent of /check-architecture to verify no rule violations
2. Run the equivalent of /run-review on all files changed since the last tag
3. Verify all tests pass: `./gradlew testDebugUnitTest`
4. Check for TODO/FIXME comments in changed files — list any found
5. Verify version bump in build.gradle.kts
6. Generate release notes from git log since last tag
7. Create a summary of changes with:
   - Features added
   - Bugs fixed
   - Breaking changes (if any)
   - Files changed count

The macro skill doesn't literally call /check-architecture and /run-review as sub-commands. Instead, it tells AI to perform the equivalent actions. AI reads those skill files for guidance on what "check architecture" and "run review" mean in your project.

For Tech Leads: Macro skills are your secret weapon for standardizing complex workflows. Instead of writing a wiki page that says "before release, do X, Y, Z" — encode it as a skill. The wiki page becomes outdated. The skill stays in the repo, version-controlled and always current.

Case Study: The /generate-next-chapter Skill

You've been watching this skill in action throughout this course. Every chapter was generated by invoking /generate-next-chapter. Let's dissect why it works so well:

9
Steps in the workflow
from reading state to updating navigation
5
Files read for context
CLAUDE.md, style.css, index, previous chapter, chapter details
3
Files modified after generation
new chapter + CLAUDE.md + index.html

What makes this skill excellent:

  • Self-discovering: It reads CLAUDE.md to find which chapter is next — no parameter needed
  • Context-loading: It reads the style.css, previous chapter, and chapter details before generating
  • Comprehensive: It handles the full lifecycle — create file, update tracking, update navigation
  • Quality-gated: It specifies line count requirements (400-600) and HTML validation rules
  • Idempotent: Running it again skips completed chapters and generates the next one
Design Principle: The best skills are self-contained workflows. They load their own context, do all their work, and clean up after themselves. The developer types one command and gets a complete result.

Quality Standards for Team Skills

When skills are shared across a team, they need quality standards. A bad skill wastes everyone's time. Here's the checklist for skill review:

Context Loading: Skill reads CLAUDE.md and relevant .ctx files before acting
Clear Steps: Numbered, sequential, unambiguous instructions
Verification: Skill includes a validation step (tests, lint, compile check)
Documentation: Top-level description explains what the skill does
Parameters: $ARGUMENTS usage is documented with examples
Naming: Follows the team naming convention (category-verb-noun)
Tested: Skill has been run at least once successfully before merging

The Skill Review Process

Treat skill PRs like code PRs. When someone adds or modifies a skill:

  1. PR includes a test run — screenshots or output showing the skill worked
  2. Another team member reviews — checks clarity, naming, context loading
  3. Skill is documented — description at the top explains purpose and parameters
  4. Edge cases considered — what happens if $ARGUMENTS is empty? What if the target already exists?
Warning: Never merge a skill that hasn't been run at least once. Unlike code, skills fail silently — AI will try its best with a badly-written skill, producing subtly wrong output. Always verify the output matches expectations.

Version Control for Skills

Skills live in your repo and evolve with your codebase. Here's how to manage changes without breaking existing workflows:

Breaking Change
  • Rename a widely-used skill
  • Change parameter format
  • Remove steps that team depends on
  • Team members get confused
Safe Evolution
  • Add new skills alongside old ones
  • Extend parameters (new optional info)
  • Add steps at the end, don't remove middle ones
  • Deprecate in CLAUDE.md before removing

When a skill needs a major overhaul, create the new version alongside the old one. Add a deprecation note to the old skill:

.claude/commands/scaffold-feature.md (deprecated header)
⚠️ DEPRECATED: Use /scaffold-feature-v2 instead. This skill will be removed next sprint.

Scaffold a new feature module...
[rest of old skill]

After the team has transitioned, remove the old skill in a subsequent PR. This gives everyone time to update their muscle memory.

The Starter Library: Your First 5 Skills

Every project should start with these five skills. They cover the most common workflows and provide a template for building more:

1
/scaffold-feature
Creates a complete feature module with presentation (Compose + MVI), domain (UseCase + Repository interface), data (RepositoryImpl), and test scaffolding. The most-used skill on any team.
2
/run-review
Loads all review checklists from docs/architecture/ and runs a comprehensive code review against them. Outputs a structured report with MUST/SHOULD/PREFER findings.
3
/generate-tests
Takes a file path, reads the file, determines the layer, and generates a matching test file with success, error, and edge-case scenarios using fakes from :core:testing.
4
/check-architecture
Scans for architecture violations: wrong dependency directions, forbidden patterns from CLAUDE.md, naming convention violations. Outputs a pass/fail report.
5
/document-module
Takes a module path, reads all public interfaces, and generates a MODULE_README.md with purpose, public API, dependencies, and usage examples.

These five skills cover the daily development loop: create (scaffold), verify (review + check), test (generate-tests), and document. Start here, then add skills as patterns emerge from your team's workflows.

Your Chapter Exercise

Exercise: Build Your Starter Skill Library

Create the first three skills for your project. Start with the highest-impact ones from the starter library.

  1. Create the directory: mkdir -p .claude/commands
  2. Build /scaffold-feature — adapt the template from this chapter to your project's architecture. Replace recipes-specific patterns with your own module structure, naming conventions, and test standards.
  3. Build /generate-tests — create a skill that takes a file path and generates a test file following your testing conventions.
  4. Build /run-review — create a skill that loads your .ctx files (from Chapter 3) and runs an architecture review.
  5. Test each skill — run it at least once and verify the output.
  6. Update CLAUDE.md — add a Workflow section that references your new skills:
## Workflow
- Use /scaffold-feature for new features
- Use /generate-tests to create test coverage
- Use /run-review before submitting PRs

Acceptance criteria: 3 skill files in .claude/commands/, each following the read-create-verify pattern, with documentation and parameter descriptions. CLAUDE.md updated with skill references.
