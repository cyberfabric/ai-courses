Chapter 1

Why Your Project Isn't AI-Ready Yet

The 5-pillar framework for AI readiness and the real cost of bad context

The Moment It Clicks

You've been using Claude Code for months. You're good at prompting. You know when to use extended thinking, when to let the agent run, when to step in. And yet — every new task still starts with a correction cycle. "No, we don't use that pattern here." "Actually, our DI setup works differently." "That's the wrong module."

Sound familiar? Here's the uncomfortable truth: your prompting isn't the problem. Your project is.

The difference between a project where AI consistently nails it and one where you're constantly course-correcting has nothing to do with how clever your prompts are. It's about how much your project teaches the AI before you even type a word.

The Core Insight: An AI-ready project isn't one that uses AI — it's one where AI already knows your architecture, your conventions, your forbidden patterns, and your team's preferences before every single interaction.

Warning: Most teams think they're at "AI-Assisted" when they're actually "AI-Unaware." Having a CLAUDE.md that says "use Clean Architecture" doesn't count — that's like telling a new hire "write good code" and expecting them to know your conventions.

The 5 Pillars of AI Readiness

After configuring dozens of projects for AI productivity, a clear pattern emerges. AI readiness isn't one thing — it's five things working together. Miss one pillar and the whole structure wobbles.

1

Think of it like building a house. CLAUDE.md is the blueprint. Context engineering is choosing which materials to bring to the job site. Skills and hooks are the power tools. Architecture rules are the building codes. Team configuration is making sure every worker reads the same blueprint.

Each pillar gets its own chapter (or two) in this course. By Chapter 12, you'll have all five pillars solid.

Why "Just Use Claude Code" Isn't Enough

Here's a conversation that plays out in every team without proper AI configuration:

What happens without configuration
You: "Add a new feature to show recipe nutrition info" Claude Code: *creates a NutritionActivity with direct Room queries, inline string formatting, and a ViewModel that calls the repository directly* You: "No, we use Compose not Activities. And the ViewModel should go through a UseCase. And strings go in resources. And..." — 4 correction cycles later, you've spent more time fixing AI output than writing it yourself.
Without project context, AI falls back to generic patterns that rarely match your specific architecture.
What happens WITH configuration
You: "Add a new feature to show recipe nutrition info" Claude Code: *reads CLAUDE.md, loads architecture rules, creates a Compose screen in the nutrition feature module with a MVI ViewModel, a GetNutritionUseCase in the domain layer, a NutritionRepositoryImpl in the data layer, and string resources* You: "Ship it."
Same prompt, same model, same developer. The only difference is project configuration.

The Compounding Effect

Here's what makes AI readiness so powerful: every rule you encode pays dividends on every future interaction.

Write "NEVER create Activities, always use Compose navigation" once in your CLAUDE.md. That one line prevents a wrong-pattern suggestion in every single feature request, refactor, and code review for the life of the project. Across a team of five developers making ten AI requests a day, that single line saves hundreds of corrections per month.

Without Configuration
  • Each developer re-explains conventions per task
  • AI makes the same mistakes across the team
  • New team members have no AI context at all
  • Knowledge lives in people's heads
  • Cost scales linearly with team size
With Configuration
  • Conventions are encoded once, applied everywhere
  • AI learns from shared rules, not individual corrections
  • New team members get AI context on day one
  • Knowledge lives in the repo
  • Cost stays flat as the team grows
The Math: If encoding one architecture rule takes 10 minutes and saves 2 minutes per AI interaction across 5 developers making 10 requests/day — that's 100 minutes saved per day from a 10-minute investment. The ROI is absurd. The only question is why you haven't done it yet.

The Quick Readiness Check

Before we go further, let's see where your project actually stands. Answer these five questions honestly:

AI Readiness Quick Check — Score Your Project
  1. Does your project have a CLAUDE.md?
    0 = No  |  1 = Basic (just a description)  |  2 = Detailed (architecture rules, forbidden patterns, conventions)
  2. Can AI find your architecture rules without asking?
    0 = No rules documented  |  1 = Rules in README/wiki  |  2 = Rules in AI-readable format (.claude/rules/*.md files with paths: frontmatter, CLAUDE.md references)
  3. Do you have reusable skills for common tasks?
    0 = No skills  |  1 = A few personal skills  |  2 = Team-shared skill library with categories
  4. Are there automated guardrails (hooks) for AI actions?
    0 = No hooks  |  1 = Basic lint/format hooks  |  2 = Safety hooks + quality hooks + subagent oversight
  5. Could a new team member be AI-productive in 30 minutes?
    0 = No onboarding for AI tools  |  1 = Some docs exist  |  2 = Structured onboarding with first-task guide

Scoring: 0-3 = AI-Hostile/Unaware  |  4-6 = AI-Assisted  |  7-8 = AI-Configured  |  9-10 = AI-Native

Be honest with yourself. Most projects score 2-4 on their first check. That's not a failure — that's a starting point. By the end of this course, you'll have a clear path to 9-10.

Case Study: The Recipes App

Throughout this course, we'll reference a real Android project — a multi-module recipes app built with Clean Architecture, Jetpack Compose, Hilt, and Room. It's a project that scores 9/10 on the readiness check. Here's why:

recipes-app/CLAUDE.md (excerpt)
# Recipes App — AI Configuration

## Architecture
- Clean Architecture: presentation → domain → data
- NEVER add feature-to-feature module dependencies
- All new screens use Compose + MVI pattern
- ViewModels MUST use StateFlow, NEVER LiveData
- All data operations go through UseCases, NEVER call Repository from ViewModel

## Module Structure
- :feature:* — one module per feature (recipes, nutrition, shopping)
- :core:ui — shared Compose components
- :core:data — shared data layer (Room, network)
- :core:domain — shared domain models and base UseCases

## Forbidden Patterns
- NEVER create Activity classes (we use single-Activity + Compose navigation)
- NEVER use findViewById or View Binding
- NEVER add suspend functions to UI layer
- NEVER put business logic in Composables
- NEVER create God ViewModels (max 3 public functions per ViewModel)

## Naming Conventions
- Screens: {Feature}Screen.kt (e.g., RecipeListScreen.kt)
- ViewModels: {Feature}ViewModel.kt
- UseCases: {Verb}{Noun}UseCase.kt (e.g., GetRecipesUseCase.kt)
- Repositories: {Feature}Repository (interface) / {Feature}RepositoryImpl (implementation)

Notice what this CLAUDE.md does not do. It doesn't explain what Clean Architecture is. It doesn't teach MVI. It doesn't describe what a UseCase does. The reader — and the AI — already know these patterns. What it does is encode this project's specific decisions: which patterns, which boundaries, which naming conventions.

That's the key insight. AI knows Clean Architecture. What it doesn't know is that your project uses StateFlow instead of LiveData, single-Activity navigation, and a max of 3 public functions per ViewModel. Those specific decisions are what make the difference.

Key Takeaway: A good CLAUDE.md doesn't teach architecture — it encodes decisions. The more specific your rules, the less you correct AI output.

What Makes It Work: The Configuration Stack

The recipes app doesn't just have a CLAUDE.md. It has a complete configuration stack:

Pillar What the Recipes App Has Chapter
CLAUDE.md Root-level + directory-level files with forbidden patterns, naming rules, module boundaries Ch 2
Context 10 architecture rule files in dual format (checklist + path-scoped .claude/rules/*.md), ARCHITECTURE.md, module README files Ch 3
Skills & Hooks 12 team skills (scaffold-feature, run-review, generate-test), 5 hooks (lint, format, test, safety, oversight) Ch 4-5
Architecture Rules 10 topic areas with MUST/SHOULD/PREFER severity, condensed .claude/rules/*.md versions for context windows Ch 7
Team Config Onboarding checklist, shared MCP config, skill review process, monthly CLAUDE.md review Ch 8

Each piece reinforces the others. The CLAUDE.md references the architecture rules. The skills use the hooks. The team config ensures everyone has the same setup. It's a system, not a collection of files.

The New Developer Test

Here's the ultimate test for your AI readiness: could a new developer clone your repo, open Claude Code, and produce architecture-compliant code within 30 minutes?

Not "could they read the docs and figure it out." Not "could they ask a teammate." Could they sit down, open Claude Code, and have the AI guide them to correct output from their very first prompt?

The 30-Minute Test
  1. New developer clones the repo
  2. Opens Claude Code in the project directory
  3. Asks: "Add a new screen that shows a list of favorite recipes"
  4. AI reads CLAUDE.md, loads rules, generates code
  5. Pass: Generated code follows all architecture rules, uses correct patterns, lands in the right module
  6. Fail: Developer has to explain conventions, correct patterns, or move files

If your project fails this test, your AI configuration isn't specific enough. The good news: that's exactly what the next 11 chapters will fix.

This test also reveals a hidden benefit of AI readiness: it's the best onboarding documentation you'll ever write. A CLAUDE.md that teaches AI your conventions also teaches new humans your conventions. Two audiences, one investment.
