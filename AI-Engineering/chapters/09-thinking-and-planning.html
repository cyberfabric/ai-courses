Chapter 9

Extended Thinking and Plan Mode

When to let AI think harder — and when to stop it from overthinking

Your project is configured: CLAUDE.md is solid, skills are shared, hooks are guarding quality, architecture rules are in .ctx files, and the team is onboarded. Now it's time to tune how AI thinks about your project — not just what it knows.

Extended thinking and plan mode are Claude Code's two power tools for complex reasoning. Used wisely, they turn multi-file refactors and architecture decisions from coin flips into reliable outcomes. Used carelessly, they burn tokens on tasks that don't need them.

This chapter teaches you when to turn up the reasoning dial, when to leave it alone, and how to encode those decisions into your project's configuration so your team doesn't have to guess.

What Extended Thinking Actually Does

Extended thinking gives Claude a private "scratchpad" before it responds. Instead of immediately generating code, it reasons through the problem — considering constraints, evaluating trade-offs, and planning its approach. You never see the thinking content directly, but you see the result: higher-quality output on complex tasks.

Think of it like the difference between a developer who immediately starts typing code and one who stares at the whiteboard for five minutes first. The second developer writes less code, but the code is better structured.

The core trade-off: Extended thinking produces better output on complex tasks but costs more tokens and takes longer. The configuration challenge is encoding when to use it into your project setup so decisions are automatic, not ad-hoc.

When Extended Thinking Helps

Extended thinking isn't universally better — it's specifically better for tasks that require multi-step reasoning. Here's the decision matrix:

Task TypeExtended Thinking?Why
Architecture decisions (new module boundaries, API design)YesNeeds to weigh trade-offs across the codebase
Multi-file refactors (rename + update all callers)YesMust track dependencies across files
Debugging complex issues (race conditions, state corruption)YesNeeds to reason about execution paths
Migration planning (LiveData → StateFlow across 40 files)YesMust consider ordering, breaking changes, rollback
Writing a single new functionNoStraightforward generation, constraints are clear
Formatting or linting fixesNoMechanical transformation, no reasoning needed
Boilerplate generation (new feature scaffold)NoFollowing a known pattern, not designing one
Writing unit tests for existing codeNoThe code defines the test structure
The rule of thumb: If the task involves one file and a clear pattern, skip extended thinking. If it involves multiple files, trade-offs, or architecture impact, enable it.

Configuring Extended Thinking in CLAUDE.md

Don't leave extended thinking to individual judgment. Encode when to use it directly in your CLAUDE.md so every team member — and every skill — follows the same guidelines.

CLAUDE.md
## Thinking Guidelines

### When to use extended thinking
- Architecture decisions: module boundaries, API design, dependency changes
- Multi-file refactors: changes touching 3+ files
- Debugging: race conditions, state management bugs, data flow issues
- Migration planning: any change affecting more than one module

### When to skip extended thinking
- Single-file edits following existing patterns
- Test generation for existing code
- Boilerplate scaffolding (use /scaffold-feature instead)
- Formatting, linting, or mechanical transformations
- README and documentation updates

This isn't just documentation — it's an instruction to Claude Code. When the AI reads your CLAUDE.md and sees "use extended thinking for multi-file refactors," it will actually engage deeper reasoning for those tasks. Specific instructions beat vague ones: "use extended thinking for changes touching 3+ files" is better than "use extended thinking for complex tasks."

Plan Mode: The Strategic Layer

Plan mode is different from extended thinking. Extended thinking happens inside a single response — the AI reasons privately before answering. Plan mode is a workflow — you ask Claude to create a plan, review it together, then execute step by step.

Without Plan Mode

"Extract the user profile into its own module"

→ AI starts coding immediately

→ Moves files you didn't expect

→ Breaks 3 features you forgot about

→ 45 minutes fixing the mess

With Plan Mode

"Plan: extract user profile into its own module"

→ AI produces a file-by-file plan

→ You review, adjust, approve

→ AI executes the approved plan

→ Clean extraction, no surprises

Plan mode is especially powerful when combined with your architecture rules. The AI creates a plan, you check it against your .ctx rules, and only then does execution begin. This is the "measure twice, cut once" workflow for AI-assisted development.

The Plan-Execute-Validate Pattern

The most effective way to use plan mode is a structured three-phase approach. This isn't just a suggestion — you can encode it directly into skills.

1

Plan

AI analyzes the codebase and produces a detailed plan: which files change, what the changes are, what could break, and what to test.

→
2

Execute

After human approval, AI implements the plan step by step. Each step is small and reviewable.

→
3

Validate

AI runs tests, checks architecture compliance, and reports what changed. Human does final review.

The key insight: the Plan phase is where extended thinking earns its cost. A well-thought-out plan prevents rework during execution. The Execute phase is usually straightforward — following the plan doesn't need deep reasoning. The Validate phase checks the work against your rules.

Integrating Plan Mode into Skills

The real power comes from encoding the plan-execute-validate pattern directly into your skills. Here's a skill that forces planning before any module extraction:

.claude/commands/extract-module.md
# Extract Module

Extract a feature into its own module using the plan-execute-validate pattern.

## Phase 1: Plan (DO NOT write any code yet)

Analyze the feature specified in $ARGUMENTS and produce a plan:

1. **Scope**: List every file that references this feature
2. **New module structure**: Proposed package structure for the new module
3. **Dependency analysis**: What does this feature depend on? What depends on it?
4. **Breaking changes**: What will break when we move these files?
5. **Migration order**: Which files to move first to minimize breakage
6. **Test impact**: Which tests need updating?

Present the plan and STOP. Wait for approval before proceeding.

## Phase 2: Execute (only after explicit approval)

Implement the plan step by step:
- Move files in the order specified
- Update imports and references
- Update dependency injection modules
- Update Gradle module dependencies

## Phase 3: Validate

After execution:
- Run `./gradlew :feature-$ARGUMENTS:testDebugUnitTest`
- Run `./gradlew lint`
- Check for circular dependencies: no feature module should import another
- Report: files moved, files modified, tests passing/failing

Notice how Phase 1 explicitly says "DO NOT write any code yet." This is critical. Without that instruction, AI tends to skip straight to execution. The plan-execute boundary must be explicit.

For Team Leads: Create plan-mode skills for every high-risk operation: module extraction, database schema changes, API versioning, dependency upgrades. The 5-minute planning overhead prevents hours of rework. Review the generated plans in your standup to build team-wide architectural awareness.

When to Force Plan Mode vs When to Allow Direct Execution

Not every task needs a plan. Forcing planning on simple tasks wastes time and annoys developers. Here's how to configure the boundary:

CLAUDE.md
## Plan Mode Guidelines

### Always use plan mode for:
- Module extraction or restructuring
- Database schema migrations
- API contract changes (breaking or additive)
- Dependency upgrades affecting 3+ modules
- Any change that requires a feature flag

### Never use plan mode for:
- Bug fixes in a single file
- Adding a new screen that follows existing patterns
- Writing or updating tests
- Documentation changes
- Config file updates

### How to invoke:
- High-risk: use /extract-module, /plan-migration, /plan-api-change (built-in planning)
- Medium-risk: start with "Plan first:" prefix in your prompt
- Low-risk: direct execution is fine

The Recipes App: Thinking vs Not Thinking

Let's see when extended thinking and plan mode make a real difference with concrete examples from the recipes app.

Example 1: Module Extraction (Plan Mode + Extended Thinking)

With Plan Mode

Prompt: "Plan: extract the user profile feature from :app into a new :feature-profile module"

Plan output includes:

  • 12 files to move (screens, viewmodels, DI, navigation)
  • 8 files to modify (imports, navigation graph, Gradle)
  • 2 potential circular dependencies to resolve first
  • Suggested migration order: DI → domain → data → presentation
  • 3 tests that need new module references

Result: Clean extraction, zero regressions, 20 minutes total.

Without Plan Mode

Prompt: "Extract the user profile feature into a new module"

What happens:

  • AI starts moving files immediately
  • Misses a ViewModel dependency on SharedPreferences in :app
  • Creates a circular dependency between :feature-profile and :feature-settings
  • Navigation breaks because the nav graph still references :app paths

Result: 45 minutes of cleanup, plus a force-push to undo the mess.

Example 2: Test Generation (No Plan Mode, No Extended Thinking)

Direct Execution

Prompt: "Write unit tests for RecipeListViewModel following the testing-ctx.md patterns"

What happens: AI reads the ViewModel, reads testing-ctx.md, generates tests following the Given-When-Then pattern with fake repositories. Done in 30 seconds.

Why no planning needed: The .ctx file already defines the test patterns. The ViewModel defines the test cases. There's nothing to decide — just execute.

Budget Thinking: The Cost Reality

Extended thinking isn't free. Understanding the cost structure helps you make informed decisions about when it's worth the investment.

3-5x
Token cost increase with extended thinking vs standard
10-30s
Additional latency for extended thinking responses
60-80%
Reduction in rework when plan mode is used for complex tasks

The math works out clearly: if extended thinking costs 3x more tokens but prevents a 45-minute rework session, the ROI is massive. But if you enable it for every single-file edit, you're burning budget with no quality improvement.

Watch out: Extended thinking costs scale with the thinking budget. More thinking = more tokens. For your CLAUDE.md, specify which tasks warrant extended thinking rather than a blanket "always use extended thinking" rule. Chapter 10 covers the full cost engineering picture.

Encoding Model Selection into Skills

Different skills benefit from different reasoning levels. You can guide model selection in your skill descriptions:

.claude/commands/plan-migration.md
# Plan Migration

Plan a codebase migration using extended thinking for thorough analysis.

This skill requires careful reasoning about dependencies, ordering,
and breaking changes. Use extended thinking for the planning phase.

## Instructions

Analyze the migration described in $ARGUMENTS:

1. **Impact assessment** (use extended thinking):
   - Which modules are affected?
   - What are the dependency chains?
   - What's the risk of each change?
   - What's the optimal migration order?

2. **Generate migration plan**:
   - Ordered list of changes with estimated complexity
   - Rollback strategy for each step
   - Test checkpoints between steps

3. **Risk summary**:
   - High-risk changes (need manual review)
   - Medium-risk changes (automated tests sufficient)
   - Low-risk changes (mechanical transformations)

Present the plan. Do NOT execute until approved.
Skill design principle: Skills that plan should request extended thinking. Skills that execute a known pattern should not. The skill description itself is the right place to encode this — it becomes part of the AI's context when the skill is invoked.

Plan Mode for Architecture Decisions

One of the highest-value uses of plan mode is architectural decision records (ADRs). Instead of writing ADRs manually, use plan mode to generate a draft that captures the trade-offs:

.claude/commands/draft-adr.md
# Draft Architecture Decision Record

Generate an ADR for the decision described in $ARGUMENTS.

Use extended thinking to analyze trade-offs thoroughly.

## Output format (in docs/adr/):

### ADR-NNN: [Decision Title]

**Status:** Proposed
**Date:** [today]
**Context:** Why this decision is needed now
**Decision:** What we chose and why
**Alternatives considered:**
- Option A: [description] — rejected because [reason]
- Option B: [description] — rejected because [reason]
**Consequences:**
- Positive: [what improves]
- Negative: [what gets harder]
- Neutral: [what changes but isn't better/worse]
**Compliance:** Which architecture rules (.ctx) this affects

Save to docs/adr/ with the next available number.

This skill leverages extended thinking where it matters most: weighing trade-offs. The resulting ADR becomes both a human-readable decision record and AI context for future development — Claude will read it and understand why the architecture is shaped the way it is.

When NOT to Use Extended Thinking

Extended thinking can actually hurt in some scenarios. Watch for these anti-patterns:

Anti-PatternWhat HappensBetter Approach
Overthinking simple edits AI spends 20 seconds "thinking" about a one-line fix Direct execution — the edit is obvious
Planning boilerplate Plan says "create file, add imports, add class" — no insight Use a scaffold skill with templates instead
Thinking about formatting Extended thinking on a lint fix wastes tokens Use a PostToolUse hook for auto-formatting
Re-planning known patterns Planning how to write a ViewModel when .ctx defines the pattern The .ctx file IS the plan — just follow it
Planning in parallel Multiple devs plan the same refactor independently One plan, shared via PR, executed collaboratively
The golden rule: Extended thinking is for decisions, not execution. If the decision is already made (by a .ctx rule, a skill template, or a previous plan), skip the thinking and just do the work.

The Thinking Audit

Is Your Project Using Thinking Effectively?

0 No thinking guidelines in CLAUDE.md. Team uses extended thinking randomly or never.
1 CLAUDE.md mentions extended thinking but doesn't specify when to use it.
2 CLAUDE.md has specific thinking guidelines (use for X, skip for Y). Some skills use plan mode.
3 All high-risk skills enforce plan-execute-validate. Thinking guidelines match task types. Team follows consistently.
4 Thinking strategy is tuned by task category. ADR generation uses plan mode. Budget thinking is tracked. Skills explicitly specify reasoning requirements.
