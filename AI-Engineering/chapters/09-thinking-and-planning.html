Chapter 9

Extended Thinking and Plan Mode

When to let AI think harder — and when to stop it from overthinking

Your project is configured: CLAUDE.md is solid, skills are shared, hooks are guarding quality, architecture rules live in .claude/rules/, and the team is onboarded. Now it's time to tune how AI thinks about your project — not just what it knows.

Extended thinking and plan mode are Claude Code's two power tools for complex reasoning. Used wisely — through the right /effort setting, the right skill frontmatter, or the right per-turn cue — they turn multi-file refactors and architecture decisions from coin flips into reliable outcomes. Used carelessly, they burn tokens on tasks that don't need them.

This chapter teaches you when to turn up the reasoning dial, when to leave it alone, and how to encode those decisions into your project's configuration so your team doesn't have to guess.

What Extended Thinking Actually Does

Extended thinking gives Claude a private "scratchpad" before it responds. Instead of immediately generating code, it reasons through the problem — considering constraints, evaluating trade-offs, and planning its approach. You never see the thinking content directly, but you see the result: higher-quality output on complex tasks.

Think of it like the difference between a developer who immediately starts typing code and one who stares at the whiteboard for five minutes first. The second developer writes less code, but the code is better structured.

The core trade-off: Extended thinking produces better output on complex tasks but costs more tokens and takes longer. Your job is to set the right depth — through /effort, an effort:-tuned skill, or a per-turn cue — so the depth matches the task.

How to Engage Extended Thinking

Claude Code gives you four controls for thinking depth. Pick the one that matches the scope of the work:

ControlScopeWhen to use it
/effort low|medium|high|xhigh|maxWhole sessionLong planning or debugging session. xhigh is the recommended starting point on Opus 4.7; medium is the Claude Code default on Opus 4.6. Run bare /effort to open an arrow-key slider.
effort: frontmatter on a skillThe skill's lifetimeA skill that always needs more (or less) reasoning than the session default. Pair with ${CLAUDE_EFFORT} in the body to adapt the instructions to the active level.
Adaptive thinking (Opus 4.6 / 4.7, Sonnet 4.6)Per-turn, automaticAlways on. The model calibrates depth from the effort setting and the complexity of each query. You set the dial; the model reads the room.
ultrathink in the promptOne turnA deep one-off burst — typically an architecture decision or a tricky debug — when you don't want to bump the session-wide setting.

For most teams the right baseline is /effort high for planning sessions, effort: xhigh on the high-stakes skills (module extraction, schema migration, ADR drafting), and the default elsewhere. Plan Mode — covered later — gives you the structured workflow that wraps around all of this.

When Extended Thinking Helps

Extended thinking isn't universally better — it's specifically better for tasks that require multi-step reasoning. Here's the decision matrix:

Task TypeExtended Thinking?Why
Architecture decisions (new module boundaries, API design)YesNeeds to weigh trade-offs across the codebase
Multi-file refactors (rename + update all callers)YesMust track dependencies across files
Debugging complex issues (race conditions, state corruption)YesNeeds to reason about execution paths
Migration planning (LiveData → StateFlow across 40 files)YesMust consider ordering, breaking changes, rollback
Writing a single new functionNoStraightforward generation, constraints are clear
Formatting or linting fixesNoMechanical transformation, no reasoning needed
Boilerplate generation (new feature scaffold)NoFollowing a known pattern, not designing one
Writing unit tests for existing codeNoThe code defines the test structure
The rule of thumb: If the task involves one file and a clear pattern, skip extended thinking. If it involves multiple files, trade-offs, or architecture impact, enable it.

Documenting Thinking Conventions in CLAUDE.md

A short section in CLAUDE.md tells the team which control to reach for in which situation. The point is consistency: everyone runs /effort high for planning sessions, everyone reaches for the same skills on high-stakes work, and nobody guesses.

CLAUDE.md
## Thinking Conventions

For deep-reasoning work — engage extra thinking depth:
- Architecture decisions, ADR drafting, API design
- Multi-file refactors (3+ files)
- Migration planning, schema changes, dependency upgrades
- Debugging race conditions or state-management bugs

How to engage it:
- Long planning sessions: run `/effort high`
  (or `/effort xhigh` on Opus 4.7) once at the start
- One-off deep prompt: open with `ultrathink`
- High-stakes operations: use the dedicated skills
  (`/plan-migration`, `/draft-adr`, `/extract-module`)
  Their SKILL.md sets `effort: xhigh` so the depth is engaged
  the moment the skill runs.

For routine work — adaptive thinking on Opus 4.6 / 4.7 handles it:
- Single-file edits following existing patterns
- Test generation for existing code
- Boilerplate scaffolding (use `/scaffold-feature`)
- Formatting, linting, or mechanical transformations
- README and documentation updates

This section is short on purpose. It exists so a developer who joins the team in week two knows the answer to "should I bother with deep thinking for this?" without having to ask. Pair it with skills whose frontmatter does the work, and the convention starts running itself.

Plan Mode: The Strategic Layer

Plan mode is different from extended thinking. Extended thinking happens inside a single response — the AI reasons privately before answering. Plan mode is a workflow — you ask Claude to create a plan, review it together, then execute step by step.

Without Plan Mode

"Extract the user profile into its own module"

→ AI starts coding immediately

→ Moves files you didn't expect

→ Breaks 3 features you forgot about

→ 45 minutes fixing the mess

With Plan Mode

"Plan: extract user profile into its own module"

→ AI produces a file-by-file plan

→ You review, adjust, approve

→ AI executes the approved plan

→ Clean extraction, no surprises

Plan mode is especially powerful when combined with your architecture rules. The AI creates a plan, you check it against the rule files in .claude/rules/, and only then does execution begin. This is the "measure twice, cut once" workflow for AI-assisted development.

Plan Mode in Claude Code today: invoke it with /plan <task> (the inline argument lets you skip the chat round-trip), or set permissions.defaultMode: "plan" in settings.json so every session in a high-risk module starts there. Once a plan exists, Ctrl+G opens it in your editor for direct edits, and Claude uses the AskUserQuestion tool to clarify gaps before proposing the next step.

The Plan-Execute-Validate Pattern

The most effective way to use plan mode is a structured three-phase approach. This isn't just a suggestion — you can encode it directly into skills.

1

Plan

AI analyzes the codebase and produces a detailed plan: which files change, what the changes are, what could break, and what to test.

→
2

Execute

After human approval, AI implements the plan step by step. Each step is small and reviewable.

→
3

Validate

AI runs tests, checks architecture compliance, and reports what changed. Human does final review.

The key insight: the Plan phase is where extended thinking earns its cost. A well-thought-out plan prevents rework during execution. The Execute phase is usually straightforward — following the plan doesn't need deep reasoning. The Validate phase checks the work against your rules.

Integrating Plan Mode into Skills

The real power comes from encoding the plan-execute-validate pattern directly into your skills. Here's a skill that forces planning before any module extraction:

.claude/skills/extract-module/SKILL.md
---
name: extract-module
description: Extract a feature into its own module using plan-execute-validate.
effort: xhigh
model: claude-opus-4-7
---

# Extract Module

Extract a feature into its own module.

## Phase 1: Plan (DO NOT write any code yet)

Analyze the feature specified in $ARGUMENTS and produce a plan:

1. **Scope**: List every file that references this feature
2. **New module structure**: Proposed package structure for the new module
3. **Dependency analysis**: What does this feature depend on? What depends on it?
4. **Breaking changes**: What will break when we move these files?
5. **Migration order**: Which files to move first to minimize breakage
6. **Test impact**: Which tests need updating?

Present the plan and STOP. Wait for approval before proceeding.

## Phase 2: Execute (only after explicit approval)

Implement the plan step by step:
- Move files in the order specified
- Update imports and references
- Update dependency injection modules
- Update Gradle module dependencies

## Phase 3: Validate

After execution:
- Run `./gradlew :feature-$ARGUMENTS:testDebugUnitTest`
- Run `./gradlew lint`
- Check for circular dependencies: no feature module should import another
- Report: files moved, files modified, tests passing/failing

Notice how Phase 1 explicitly says "DO NOT write any code yet." Combined with the effort: xhigh frontmatter, the planning phase gets both the depth and the explicit instruction to stop before execution.

For Team Leads: Create plan-mode skills for every high-risk operation: module extraction, database schema changes, API versioning, dependency upgrades. Set effort: xhigh in the frontmatter on every high-risk skill so the depth is engaged from the moment the skill runs. Review the generated plans in your standup to build team-wide architectural awareness.

When to Force Plan Mode vs When to Allow Direct Execution

Not every task needs a plan. Forcing planning on simple tasks wastes time and annoys developers. Here's how to configure the boundary:

CLAUDE.md
## Plan Mode Guidelines

### Always use plan mode for:
- Module extraction or restructuring
- Database schema migrations
- API contract changes (breaking or additive)
- Dependency upgrades affecting 3+ modules
- Any change that requires a feature flag

### Never use plan mode for:
- Bug fixes in a single file
- Adding a new screen that follows existing patterns
- Writing or updating tests
- Documentation changes
- Config file updates

### How to invoke:
- High-risk: use /extract-module, /plan-migration, /plan-api-change
  (their frontmatter sets `effort: xhigh`)
- Medium-risk: open with `/plan` for an interactive plan, or run
  `/effort high` then start the prompt with "Plan first:"
- Long planning sessions: run `/effort high` once at the start
- Low-risk: direct execution, default effort

The Recipes App: Thinking vs Not Thinking

Let's see when extended thinking and plan mode make a real difference with concrete examples from the recipes app.

Example 1: Module Extraction (Plan Mode + Extended Thinking)

With Plan Mode

Prompt: "Plan: extract the user profile feature from :app into a new :feature-profile module"

Plan output includes:

  • 12 files to move (screens, viewmodels, DI, navigation)
  • 8 files to modify (imports, navigation graph, Gradle)
  • 2 potential circular dependencies to resolve first
  • Suggested migration order: DI → domain → data → presentation
  • 3 tests that need new module references

Result: Clean extraction, zero regressions, 20 minutes total.

Without Plan Mode

Prompt: "Extract the user profile feature into a new module"

What happens:

  • AI starts moving files immediately
  • Misses a ViewModel dependency on SharedPreferences in :app
  • Creates a circular dependency between :feature-profile and :feature-settings
  • Navigation breaks because the nav graph still references :app paths

Result: 45 minutes of cleanup, plus a force-push to undo the mess.

Example 2: Test Generation (No Plan Mode, No Extended Thinking)

Direct Execution

Prompt: "Write unit tests for RecipeListViewModel following the .claude/rules/testing.md patterns"

What happens: AI reads the ViewModel, reads .claude/rules/testing.md, generates tests following the Given-When-Then pattern with fake repositories. Done in 30 seconds.

Why no planning needed: The rule file in .claude/rules/ already defines the test patterns. The ViewModel defines the test cases. There's nothing to decide — just execute.

Budget Thinking: The Cost Reality

Extended thinking isn't free. Understanding the cost structure helps you make informed decisions about when it's worth the investment.

3-5x
Token cost increase with extended thinking vs standard
10-30s
Additional latency for extended thinking responses
60-80%
Reduction in rework when plan mode is used for complex tasks

The math works out clearly: if extended thinking costs 3x more tokens but prevents a 45-minute rework session, the ROI is massive. But if you enable it for every single-file edit, you're burning budget with no quality improvement.

Watch out: Thinking depth scales cost. Reserve /effort xhigh and effort: xhigh skills for the task categories that earn it; let adaptive thinking handle the rest. On Opus 4.7 the API rejects manual thinking budgets outright — adaptive is the only mode, and /effort is the knob you turn. Chapter 10 covers the full cost engineering picture.

Declaring Reasoning Depth in Skill Frontmatter

Different skills want different depths. Planning skills declare effort: xhigh so the deep budget is engaged the moment they run; pattern-following skills (test generation, scaffolds, formatters) leave the field unset and inherit the session default. The migration planner looks like this:

.claude/skills/plan-migration/SKILL.md
---
name: plan-migration
description: Plan a codebase migration with deep reasoning about dependencies and ordering.
effort: xhigh
model: claude-opus-4-7
---

# Plan Migration

Plan a codebase migration. Current effort level: ${CLAUDE_EFFORT}.

## Instructions

Analyze the migration described in $ARGUMENTS:

1. **Impact assessment**:
   - Which modules are affected?
   - What are the dependency chains?
   - What's the risk of each change?
   - What's the optimal migration order?

2. **Generate migration plan**:
   - Ordered list of changes with estimated complexity
   - Rollback strategy for each step
   - Test checkpoints between steps

3. **Risk summary**:
   - High-risk changes (need manual review)
   - Medium-risk changes (automated tests sufficient)
   - Low-risk changes (mechanical transformations)

Present the plan. Do NOT execute until approved.
Skill design principle: Set effort: xhigh in the frontmatter of skills that need deep reasoning. Leave it unset on skills that follow a known pattern. The frontmatter declares the depth; the skill body describes the work. ${CLAUDE_EFFORT} lets the body adapt its instructions to the active level.

Plan Mode for Architecture Decisions

One of the highest-value uses of plan mode is architectural decision records (ADRs). Instead of writing ADRs manually, use plan mode to generate a draft that captures the trade-offs:

.claude/skills/draft-adr/SKILL.md
---
name: draft-adr
description: Generate an architecture decision record with deep trade-off analysis.
effort: xhigh
model: claude-opus-4-7
---

# Draft Architecture Decision Record

Generate an ADR for the decision described in $ARGUMENTS.

## Output format (in docs/adr/):

### ADR-NNN: [Decision Title]

**Status:** Proposed
**Date:** [today]
**Context:** Why this decision is needed now
**Decision:** What we chose and why
**Alternatives considered:**
- Option A: [description] — rejected because [reason]
- Option B: [description] — rejected because [reason]
**Consequences:**
- Positive: [what improves]
- Negative: [what gets harder]
- Neutral: [what changes but isn't better/worse]
**Compliance:** Which architecture rules (in .claude/rules/) this affects

Save to docs/adr/ with the next available number.

This skill leverages extended thinking where it matters most: weighing trade-offs. The resulting ADR becomes both a human-readable decision record and AI context for future development — Claude will read it and understand why the architecture is shaped the way it is.

Auto Mode and Task Budgets

Plan Mode produces a plan. Auto Mode runs it. Released for Max subscribers on Opus 4.7, Auto Mode is a classifier-based execution layer: a Haiku fast-path tags each tool call as safe or risky, and the Opus thinking path takes over on the ambiguous ones. The result is a continuous run that pauses on real risk and powers through the rest. The new PermissionDenied hook lets the model react to a denied command — adjusting its approach instead of giving up — so a single rejected rm or git push doesn't kill the whole task.

Task budgets (currently in public beta) cap the token spend of a single long-running operation. Set a budget once, and any skill that loops too long stops gracefully instead of running up the bill. They pair naturally with Auto Mode: bound the autonomy with a token ceiling.

.claude/settings.json
{
  "permissions": {
    "defaultMode": "plan"
  },
  "tasks": {
    "budget": 250000
  },
  "hooks": {
    "PermissionDenied": [
      {
        "command": ".claude/hooks/log-denied.sh",
        "comment": "Track which commands Auto Mode wants to retry"
      }
    ]
  }
}

The combined flow for a high-risk multi-file change in May 2026: /plan <task> → review and edit the plan with Ctrl+G → accept → Auto Mode (Opus 4.7, xhigh effort) executes the plan with hooks as guardrails and the task budget as the safety net.

When NOT to Use Extended Thinking

Extended thinking can actually hurt in some scenarios. Watch for these anti-patterns:

Anti-PatternWhat HappensBetter Approach
Overthinking simple edits AI spends 20 seconds "thinking" about a one-line fix Direct execution — the edit is obvious
Planning boilerplate Plan says "create file, add imports, add class" — no insight Use a scaffold skill with templates instead
Thinking about formatting Extended thinking on a lint fix wastes tokens Use a PostToolUse hook for auto-formatting
Re-planning known patterns Planning how to write a ViewModel when a .claude/rules/ file defines the pattern The rule file IS the plan — just follow it
Planning in parallel Multiple devs plan the same refactor independently One plan, shared via PR, executed collaboratively
The golden rule: Extended thinking is for decisions, not execution. If the decision is already made (by a rule in .claude/rules/, a skill template, or a previous plan), skip the thinking and just do the work.

Exercise: Add Thinking Conventions to Your Project

  1. Open your project's CLAUDE.md.
  2. Add a "## Thinking Conventions" section with two lists:
    • For deep reasoning (run /effort high, or invoke a skill whose frontmatter sets effort: xhigh): list 3-5 task types specific to your project — e.g. module extraction, schema migration, ADR drafting.
    • Adaptive thinking handles: list 3-5 pattern-based task types — e.g. test generation, formatting, README updates.
  3. Create one plan-mode skill for your highest-risk operation (module extraction, schema migration, or API change).
  4. Set effort: xhigh in the skill's frontmatter and structure the body as Phase 1 (Plan, "DO NOT write code yet") → Phase 2 (Execute, "wait for approval") → Phase 3 (Validate).
  5. Commit both files: the updated CLAUDE.md and the new skill.
