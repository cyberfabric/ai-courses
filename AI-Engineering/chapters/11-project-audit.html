Chapter 11

The AI-Ready Audit: Your Project Scorecard

Score your project across all 5 pillars — and build a plan to close the gaps

You've learned the full stack: CLAUDE.md, context engineering, skills, hooks, architecture rules, team onboarding, thinking strategy, and cost management. Now it's time to turn that knowledge inward and audit your own project.

This chapter gives you a structured scorecard you can run against any project. You'll score each of the 5 pillars from 0 to 4, identify the biggest gaps, and build a prioritized improvement plan. By the end, you'll have a concrete action list — not abstract theory, but specific files to create, rules to write, and skills to build.

This chapter is an exercise. Don't just read it — actually run the audit on your project. Open a new document, score each pillar, and record what you find. The value is in the doing, not the reading.

The 5-Pillar Audit Framework

Every AI-ready project rests on five pillars. Each one supports the others — a weakness in one reduces the effectiveness of all five. Here's the framework from Chapter 1, now expanded into a scoring rubric.

Pillar 1: CLAUDE.md & Project Identity

P1: Score Your CLAUDE.md

0 No CLAUDE.md exists. AI has zero project-specific context.
1 CLAUDE.md exists but is generic ("This is a Kotlin project using Clean Architecture"). No specific rules or forbidden patterns.
2 CLAUDE.md has specific rules: module structure, naming conventions, 3+ NEVER rules. Enough for AI to produce mostly-correct code.
3 CLAUDE.md is comprehensive: architecture rules, module boundaries, build commands, testing conventions, model selection guidelines. Organized with stable content first for caching.
4 Multi-level CLAUDE.md hierarchy: root + directory-level files. Monthly review cadence. New developers productive within 30 minutes using only the CLAUDE.md and skills.

Common gaps at each level:

ScoreMost Common GapQuick Fix
0 → 1No file existsCopy the universal starter template (Ch 2) and customize for 30 minutes
1 → 2Too vague — "use Clean Architecture"Add 5 specific NEVER rules from your last 10 code reviews
2 → 3Missing build commands and testing conventionsAdd ## Build, ## Testing, ## Model Selection sections
3 → 4Single file, no directory-level CLAUDE.mdAdd CLAUDE.md in top 3 feature directories with layer-specific rules

Pillar 2: Context Engineering

P2: Score Your Context Strategy

0 No documentation-as-context. No .claude/rules/ files. AI reads only code and comments.
1 README exists but is stale. No architecture documentation. AI context is whatever it finds in the codebase.
2 ARCHITECTURE.md or similar exists. Some path-scoped rule files in .claude/rules/ for key layers. Context budget is roughly managed.
3 Path-scoped rule files in .claude/rules/ for all major layers (25 lines each). CLAUDE.md references them with load-on-demand pattern. No context pollution — everything loaded serves a purpose.
4 Full dual-format documentation: review checklists + .claude/rules/ files for all 10 topic areas. Context audit completed. ~2K tokens covers entire architecture. Cache hit rate tracked.

Common gaps:

ScoreMost Common GapQuick Fix
0 → 1No docs at allWrite a 1-page ARCHITECTURE.md: layers, modules, key patterns
1 → 2Docs are prose-heavy, not AI-consumableCreate 3 rule files in .claude/rules/: presentation, domain, testing (25 lines each)
2 → 3CLAUDE.md loads everything alwaysSwitch to reference pattern: inline NEVER rules, reference path-scoped rule files on demand
3 → 4No review checklists, only condensed rule filesWrite verbose checklists for top 3 layers. Use for code review, condense to .claude/rules/ for dev.

Pillar 3: Skills & Hooks

P3: Score Your Automation Layer

0 No custom skills. No hooks. Developers type everything from scratch every time.
1 1-2 personal skills (not shared). No hooks. One developer benefits, the team doesn't.
2 3-5 team skills committed to the repo. Basic hooks (auto-format on save). Most of the team uses them.
3 Complete skill library covering scaffold, review, test, and document categories. PreToolUse safety hooks. PostToolUse quality hooks. Skills use plan-execute-validate for high-risk operations.
4 Skills versioned and reviewed like code. Hook matrix covers all risk levels. Skill quality checklist enforced. Team members contribute new skills monthly.

Common gaps:

ScoreMost Common GapQuick Fix
0 → 1No skills at allCreate /scaffold-feature and /run-review — the two highest-value skills
1 → 2Skills are personal, not sharedMove to .claude/skills/<name>/SKILL.md and commit (legacy .claude/commands/ still works). Add documentation headers.
2 → 3No safety hooksAdd the dangerous-command blocker and sensitive-file protector (Ch 5)
3 → 4Skills accepted without reviewAdd skill quality checklist to PR template. Require docs + test for every new skill.

Pillar 4: Architecture Rules

P4: Score Your Architecture Rules

0 No architecture rules encoded. AI invents its own patterns.
1 Some rules in CLAUDE.md, but vague: "follow our architecture." No rule IDs or severity levels.
2 MUST/SHOULD/PREFER rules for 3+ topic areas. Rule IDs assigned. .claude/rules/ files for key layers.
3 Full dual-format rules for 7+ of the 10 topic areas. /check-architecture skill validates compliance. Rules updated during migrations.
4 All 10 topic areas covered. Monthly review cadence. Rules evolve with the codebase. Compliance rate above 90%. Team treats architecture rules like tests — they must pass.

Common gaps:

ScoreMost Common GapQuick Fix
0 → 1No rules at allExtract 5 rules from your last 10 code review comments. Add to CLAUDE.md.
1 → 2Rules lack specificityAdd MUST/SHOULD/PREFER severity. Assign rule IDs (P1, D1, T1).
2 → 3No compliance checkingCreate /check-architecture skill that validates rules against code.
3 → 4Rules go stale during migrationsAdd "update architecture rules" to every migration checklist.

Pillar 5: Team Configuration

P5: Score Your Team Readiness

0 No team adoption. One developer uses AI tools; everyone else ignores them.
1 Multiple developers use AI, but each has their own setup. No shared configuration.
2 Shared configuration in the repo. Onboarding checklist exists. Most team members use skills regularly.
3 Full onboarding playbook. Adoption metrics tracked. Model selection and thinking guidelines followed. MCP configured project-wide with ${ENV_VAR} pattern.
4 AI-native team: configuration is part of Definition of Done. Team creates and evolves skills independently. 50%+ members contributed a skill or rule. Cost tracking in place.

Common gaps:

ScoreMost Common GapQuick Fix
0 → 1No one else is using AIRun a 30-minute demo session (Ch 8). Show 3 skills in action.
1 → 2Each dev has their own setupCommit CLAUDE.md, skills, and MCP config. Everyone clones the same setup.
2 → 3No onboarding processWrite AI_ONBOARDING_CHECKLIST.md. Add to new-hire checklist.
3 → 4Team consumes but doesn't contributeSprint goal: every team member creates one skill or one path-scoped rule file.

The Prioritized Improvement Plan

Don't try to fix everything at once. Use this priority matrix to focus on the highest-impact improvements first:

Priority Rule: Fix your lowest-scoring pillar first. A score of 0 in any pillar undermines all the others. A project with scores of 4-4-4-4-0 performs worse than one with 2-2-2-2-2 because the missing pillar creates a bottleneck.

One-Week Quick Wins

5 Improvements in 5 Days

  • Day 1: Create or expand CLAUDE.md with 5 NEVER rules from recent code reviews
  • Day 2: Write 3 rule files in .claude/rules/ for your most-edited layers (presentation, testing, data)
  • Day 3: Create /scaffold-feature and /run-review skills
  • Day 4: Add the dangerous-command blocker hook and auto-format hook
  • Day 5: Run a 30-minute team demo. Commit everything. Celebrate.

One-Month Deep Improvements

5 Transformative Changes

  • Week 1-2: Write dual-format rules (checklist + path-scoped rule file) for your top 5 architecture areas
  • Week 2-3: Build plan-mode skills for your 3 highest-risk operations
  • Week 3: Configure project-level MCP servers (database, issue tracker, docs)
  • Week 4: Implement the full onboarding playbook. Every team member runs through it.
  • Ongoing: Add thinking guidelines and model selection to CLAUDE.md

For Team Leads: Run the audit together as a team exercise. Each person scores independently, then you compare and discuss. The disagreements are the most valuable part — they reveal where the team's understanding of the configuration differs. Consensus building is part of adoption.

The Audit Skill

Turn this entire audit into a runnable skill that checks your project automatically:

.claude/skills/audit-readiness/SKILL.md
# Audit AI Readiness

Run the 5-pillar AI readiness audit on this project.

## Check each pillar:

### P1: CLAUDE.md
- Does CLAUDE.md exist at the project root?
- Does it contain NEVER rules?
- Does it have build commands, testing conventions, module structure?
- Are there directory-level CLAUDE.md files?

### P2: Context Engineering
- Do path-scoped rule files exist in .claude/rules/?
- Does CLAUDE.md reference them with load-on-demand pattern?
- Is there an ARCHITECTURE.md?

### P3: Skills & Hooks
- How many skills are in .claude/skills/ (current shape: SKILL.md per skill directory)?
- How many legacy skills remain in .claude/commands/ (still works, but consider migrating)?
- Are there PreToolUse or PostToolUse hooks in settings?
- Do any skills use plan-execute-validate pattern?
- Does any skill use frontmatter to let Claude auto-invoke it?

### P4: Architecture Rules
- Do rules use MUST/SHOULD/PREFER severity?
- Do rules have IDs (P1, D1, T1)?
- How many of the 10 topic areas are covered?

### P5: Team Configuration
- Does an onboarding checklist exist?
- Is MCP configured at the project level?
- Are there model selection guidelines?

## Output:
Score each pillar 0-4 with justification.
List the top 3 gaps and recommended fixes.
Save the report to docs/ai-readiness-audit.md.
Run this quarterly. The audit is most valuable as a recurring practice, not a one-time event. Add it to your quarterly planning — "run /audit-readiness before sprint planning" keeps configuration quality on the team's radar.
Don't game the score. The goal isn't to reach 20/20 — it's to identify the gaps that hurt most. A project with genuine 14/20 that knows its weaknesses outperforms a project that inflated to 18/20 by checking boxes without real depth.

Exercise: Run the Full Audit

  1. Copy the scorecard template above into a new file: docs/ai-readiness-audit.md
  2. Score each of the 5 pillars honestly (use the rubrics in this chapter)
  3. Identify your top 3 gaps — the improvements with the highest impact
  4. Write 3 concrete action items with deadlines for the next 2 weeks
  5. Commit the audit file to your repo
  6. Bonus: Create the /audit-readiness skill and run it to compare automated vs manual scoring
