Chapter 3

Context Engineering for Real Projects

Control what AI reads, how much, and in what format — at the project level

Your CLAUDE.md Is Only the Beginning

In Chapter 2, you built a production-grade CLAUDE.md. That file is the single most important piece of your AI configuration. But it's not the only thing AI reads.

Every time you start a Claude Code session, the model assembles a context window from multiple sources: your CLAUDE.md, the files you reference, the conversation history, any documentation it discovers, and whatever it reads while exploring your project. That context window has a budget — and how you manage that budget determines whether AI gets the right information or drowns in noise.

Context engineering isn't about writing better prompts. It's a project-level discipline: deciding what AI should read, in what format, and making sure irrelevant information stays out of the way.

The Principle: Context engineering is to AI what mise en place is to a chef. Before cooking starts, every ingredient is measured, prepped, and within reach. Before AI starts coding, every relevant rule, pattern, and reference should be available in the right format at the right size.

Context Budget Planning

Every AI model has a context window — the total amount of text it can process in a single interaction. Claude's context window is large (200K tokens), but that doesn't mean you should fill it. More context isn't always better. The right question is: what's the minimum context AI needs to produce correct output?

Here's how context typically breaks down in a well-configured project:

Context Source Typical Size Caching? Your Control
CLAUDE.md (root) 500-2K tokens Cached (stable) Full
CLAUDE.md (directory) 200-500 tokens each Cached (stable) Full
Files AI reads during task 2-20K tokens Partial Indirect (project structure)
Conversation history Grows over session Partial Limited
.claude/rules/<topic>.md 200-400 tokens each Cached; path-scoped rules load with matching files Full
Architecture docs 1-5K tokens When stable Full

The key insight: CLAUDE.md and unscoped .claude/rules/ files get cached because they don't change between interactions. This means they're essentially "free" after the first read in a session. Path-scoped rules (paths: frontmatter, covered in Chapter 7) cost zero context until Claude reads a matching file. Invest in making these files excellent — they're the highest-ROI context you can provide.

Budget Rule of Thumb: Keep your total "always-loaded" context (project CLAUDE.md + unscoped rules) under 5K tokens. Path-scoped rules don't count against the always-loaded budget, since they only enter context when a matching file is read.

Context Window Anatomy: What Loads at Startup

Before you ever type a prompt, Claude Code has already loaded a stack of automatic context. Knowing the order is the difference between optimizing the right thing and chasing phantom token costs.

# Component Typical tokens What it is
1System prompt~4,200Core behavior, tool-use rules, response format. You never see it.
2Auto memory (MEMORY.md)~680Claude's notes from prior sessions; first 200 lines or 25KB.
3Environment info~280cwd, platform, shell, OS, git status block.
4MCP tool names (deferred)~120Names only; full schemas load on demand via tool search.
5Skill descriptions~450One-liners so Claude knows what skills exist.
6User CLAUDE.md (~/.claude/CLAUDE.md)~320Your global preferences across every project.
7Project CLAUDE.md~1,800Team rules. The most important file you maintain.

That's roughly 8K tokens of context loaded before your first message — about 4% of a 200K window. Numbers vary with your setup (more MCP servers, longer CLAUDE.md, bigger MEMORY.md), but the order is fixed.

Defer MCP tool schemas. By default, Claude Code lists MCP tool names at startup and pulls the full schema on demand via tool search. Set ENABLE_TOOL_SEARCH=auto to load schemas upfront only when they fit within 10% of the window, or =false to load everything (rarely the right call). As of Claude Code v2.1.69, even built-in tool schemas (Read, Edit, Bash, etc.) are deferred this way, dropping startup tool overhead from ~14K to under 1K tokens.

What Survives /compact

Long sessions eventually fill the context window. /compact summarizes the conversation so far and frees space for more work. Not every kind of context survives. Knowing what reloads automatically and what evaporates is essential when you architect rules across CLAUDE.md, .claude/rules/, skills, and hooks.

What happens to each mechanism after /compact
System prompt + output style       → Unchanged (not part of message history)
Project-root CLAUDE.md             → Re-injected from disk
Unscoped rules (no paths:)         → Re-injected from disk
Auto memory                        → Re-injected from disk
Path-scoped rules (paths:)         → LOST until matching file is read again
Nested CLAUDE.md (subdirectories)  → LOST until file in that dir is read again
Invoked skill bodies               → Re-injected, capped 5K tokens/skill,
                                     25K total; oldest dropped first
Skill descriptions (the index)     → NOT re-injected after compact
Hooks                              → Not applicable (run as code, not context)

This drives architecture: if a rule MUST survive compaction, drop the paths: frontmatter or move it into the project-root CLAUDE.md. If a skill MUST survive, keep it under 5K tokens and put the most critical instructions near the top of SKILL.md (truncation cuts from the bottom).

The skill-index gotcha: after /compact, only skills you actually invoked stay in context. Skill descriptions are not re-injected. If your team relies on Claude auto-discovering a skill mid-session, it won't see un-invoked ones after the first compact.

Diagnostics: /context, /memory, and Subagents

Three mechanisms let you observe and control context as it grows:

  • /context — live breakdown of what's in your window right now, by category (system, MCP, memory, conversation, autocompact buffer, free space). Run it before starting heavy work or when responses feel slow. Aim to start substantial tasks below 80% utilization.
  • /memory — lists every CLAUDE.md, CLAUDE.local.md, and rules file that loaded at startup. Use it to confirm a file you expected is actually visible to Claude.
  • Subagent context isolation — each subagent (Explore, Plan, custom) runs in its own separate context window. Files it reads don't touch yours; only the final summary returns. Built-in Explore and Plan agents skip CLAUDE.md for an even smaller starting context.
Subagents as a context budget tool: when you need to research a large codebase or read many files, delegate to a subagent. The 6,000+ tokens of file reads stay in the subagent's window; you get a 400-token summary back. AI Harness Engineering covers subagents in depth — for context engineering, the rule is: if a task involves reading more than ~10K tokens of files, do it in a subagent.

Documentation-as-Context

Most project documentation is written for humans. It uses narrative prose, includes background history, explains "why" at length. That's great for onboarding — but terrible for AI context. AI needs structured, scannable, rule-based information.

The solution isn't to rewrite your docs. It's to maintain two formats: human-readable documentation and AI-readable context files.

Human-Readable (README.md) 
## Architecture

Our app follows Clean Architecture principles,
which we adopted in Q2 2024 after the monolith
became unmaintainable. The presentation layer
uses Jetpack Compose with MVI pattern, which
replaced our previous MVVM approach. The domain
layer contains all business logic and is
intentionally kept free of Android dependencies
to ensure testability...
AI-Readable (CLAUDE.md) 
## Architecture
- Clean Architecture: presentation → domain → data
- Presentation: Compose + MVI (StateFlow)
- Domain: pure Kotlin, no Android imports
- Data: Room + Retrofit, Repository pattern
- NEVER reverse dependency direction
- NEVER put business logic in Composables

The human version tells a story. The AI version states constraints. Both are valuable — but for different audiences. Your CLAUDE.md should read like a rulebook, not a history book.

ARCHITECTURE.md: The Bridge Document

For larger projects, create an ARCHITECTURE.md that sits between the narrative README and the terse CLAUDE.md. It provides structured documentation that serves both humans and AI:

ARCHITECTURE.md (structured for both audiences)
# Architecture Overview

## Layer Diagram
```
┌─────────────────────────────────┐
│  :app (Navigation, DI setup)    │
├─────────────────────────────────┤
│  :feature:*  (Compose + MVI)    │
│  ┌──────────┐  ┌──────────┐    │
│  │ recipes  │  │nutrition │    │
│  └──────────┘  └──────────┘    │
├─────────────────────────────────┤
│  :core:ui    :core:domain       │
│  :core:data  :core:testing      │
└─────────────────────────────────┘
```

## Module Dependencies (allowed)
- :feature:* → :core:domain, :core:ui
- :core:data → :core:domain
- :app → :feature:*, :core:*

## Module Dependencies (forbidden)
- :feature:A → :feature:B (features never depend on each other)
- :core:domain → any Android library
- :feature:* → :core:data (features access data through domain)

## Key Decisions
| Decision | Choice | Alternative Rejected |
|----------|--------|---------------------|
| UI framework | Compose | Views (legacy) |
| State management | MVI + StateFlow | MVVM + LiveData |
| DI | Hilt (constructor) | Koin, manual |
| Async | Coroutines + Flow | RxJava |
| Navigation | Compose Navigation | Fragment Navigation |

This format works for both audiences: humans get the visual diagram and decision rationale, AI gets the structured dependency rules and decision table. Reference it from your CLAUDE.md: See ARCHITECTURE.md for module dependency diagram.

Auto Memory: Claude's Notes to Itself

Everything we've covered so far is context you write. Auto memory is context Claude writes for itself, between sessions. When Claude notices a build command, a recurring debugging pattern, or a preference you've corrected twice, it can save a note to a per-project memory directory. Next session, the notes load automatically alongside your CLAUDE.md.

Auto memory layout (machine-local; one dir per git repo)
~/.claude/projects/<project>/memory/
  MEMORY.md          ← concise index, loaded every session
  debugging.md       ← detailed debugging notes
  api-conventions.md ← API design decisions Claude observed
  ...                ← any topic files Claude creates

Three rules govern auto memory:

  • Only MEMORY.md auto-loads. The first 200 lines or 25KB (whichever comes first) load at session start. Topic files like debugging.md stay on disk and load only when Claude reads them.
  • Machine-local, repo-scoped. All worktrees and subdirectories of the same git repo share one auto-memory directory. Memory is not synced across machines or cloud environments.
  • Plain markdown. Run /memory to browse and edit; the files have no special format.
.claude/settings.json — auto memory controls
{
  "autoMemoryEnabled": true,                 // default; toggle in /memory
  "autoMemoryDirectory": "~/.claude/memory"  // override storage location
}

You can also set CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 as an environment variable to disable it entirely. Auto memory requires Claude Code v2.1.59 or later (claude --version).

When to use which: CLAUDE.md = team-shared rules (in git). Auto memory = personal accumulated learnings (machine-local, gitignored by default location). If a rule must apply to every teammate, it goes in CLAUDE.md. If it's "Claude figured this out by working with you," let auto memory keep it.
Audit it. Auto memory accumulates silently. Run /memory periodically and skim MEMORY.md — outdated or misremembered notes turn into confidently wrong context. Treat them like CLAUDE.md: prune what's stale, especially after a major refactor.

The Condensed-Rules Pattern (Now Native)

The most efficient way to deliver architectural rules to AI is in condensed form — one topic per file, ~25 lines, rule IDs you can reference. Claude Code now supports this natively through path-scoped rules in .claude/rules/. Each rule file uses YAML frontmatter to declare which paths it applies to, and the rule auto-loads when Claude reads a matching file. Zero manual referencing, zero context cost when the rule isn't relevant.

The Condensed-Rules Contract: Maximum ~25 lines. One topic per file. One-line summaries with rule IDs. No explanations, no prose, no history — just rules. If AI needs more detail, it reads the corresponding full checklist. Chapter 7 covers the dual-format pattern in implementation depth; this section gives you the concept.
.claude/rules/presentation-layer.md
---
paths:
  - "**/presentation/**/*.kt"
  - "**/ui/**/*.kt"
---

# Presentation Layer — Condensed Rules

P1: All screens are @Composable functions, never Activities/Fragments
P2: Every screen takes a Modifier parameter as last param
P3: ViewModels expose StateFlow<UiState> — never LiveData
P4: UiState is a data class — never sealed class
P5: Events are sealed interface — one per user action
P6: Max 3 public functions per ViewModel
P7: No business logic in Composables — delegate to ViewModel
P8: No direct coroutine launches in Composables — use LaunchedEffect
P9: String resources via stringResource() — never hardcoded
P10: Preview functions for every screen with @Preview annotation
P11: Navigation args via type-safe route objects
P12: Loading/error/content states handled via UiState properties

Twelve rules, twelve lines of body. That's roughly 200 tokens — a fraction of what a full documentation page would cost. And the paths: frontmatter means this rule contributes zero tokens to your startup context; it loads only when Claude reads a Kotlin file under presentation/ or ui/.

The Dual-Format Pattern

Every condensed rule file has a longer companion: a review checklist with detailed explanations, examples, and severity levels. The condensed file is for everyday AI context (loaded automatically by path). The checklist is for thorough code review (loaded by an explicit skill or import).

.claude/rules/<topic>.md
~25 lines, loaded by paths
↔
Review Checklist
100+ lines, loaded for reviews
docs/architecture/presentation-layer-review-checklist.md
# Presentation Layer — Review Checklist

## P1: Compose-Only UI [MUST]
- All screens MUST be @Composable functions
- Activities are forbidden (single-Activity architecture)
- Fragments are forbidden (Compose Navigation only)
- **Check:** No new files extending Activity or Fragment

## P2: Modifier Parameter [MUST]
- Every screen Composable MUST accept Modifier as last parameter
- Default value: Modifier (not Modifier.fillMaxSize())
- **Why:** Enables parent-controlled sizing and testing
- **Check:** grep for `fun.*Screen(` without `modifier: Modifier`

## P3: StateFlow Exposure [MUST]
- ViewModels MUST expose StateFlow<UiState>
- LiveData is forbidden in new code
- MutableStateFlow is private, exposed via .asStateFlow()
- **Check:** No public MutableStateFlow or LiveData in ViewModels

... (continues for all 12 rules)

The condensed rule file tells AI the rules during development. The checklist tells AI (or a reviewer) how to verify compliance during review. Two files per topic, two purposes, minimal duplication.

Avoiding Context Pollution

Context pollution is when AI reads information that actively hurts its output. Watch for these five polluters:

  • Outdated Rules — CLAUDE.md says "use LiveData" but you migrated to StateFlow. AI follows the old rule with full confidence.
  • Conflicting Rules — Root CLAUDE.md says "sealed class for state" but a rule in .claude/rules/ says "data class." AI picks one randomly or merges both.
  • Irrelevant Context — Backend API docs loaded for a UI-only task. Wastes tokens and causes over-engineering.
  • Verbose Documentation — 5-page ADR loaded for a bug fix. Actual rules get buried under narrative.
  • Legacy Patterns — Old code examples in docs using deprecated approaches. AI treats them as patterns to follow.
Warning: Conflicting rules are the most dangerous polluter. AI will try to satisfy both rules simultaneously, producing Frankenstein code that follows neither pattern correctly. If you maintain both .claude/rules/ files and CLAUDE.md rules, they must agree. One source of truth, multiple formats.

The Context Audit

Run this audit quarterly (or after any major refactor):

List every file AI reads automatically (CLAUDE.md, unscoped .claude/rules/ files, auto memory MEMORY.md, ARCHITECTURE.md)
For each file, ask: "Is every statement still accurate?"
Check for conflicts between files (same topic, different rules)
Check for outdated references (files that no longer exist, renamed modules)
Check for verbose sections that could be condensed
Remove or update anything that fails the above checks
Run /context mid-session and confirm utilization is below 80% before starting heavy work

Directory-Level CLAUDE.md Strategy

As covered in Chapter 2, directory CLAUDE.md files add tokens every time AI works in that directory. The context engineering rule: only create directory CLAUDE.md files for modules that genuinely differ from root conventions. 3-4 override files total, never repeating what's in root.

build-logic/CLAUDE.md (example of a justified directory override)
# Build Logic — CLAUDE.md

## This directory contains Gradle convention plugins
- Written in Kotlin DSL, NOT application code
- No Clean Architecture here — these are build scripts
- Follow Gradle plugin conventions, not app conventions
- Test with `./gradlew :build-logic:test`
- Files: convention plugins in src/main/kotlin/
- Naming: {Purpose}ConventionPlugin.kt

The Recipes App Context Strategy

Here's how the recipes app's context layer works as a complete system:

The strategy works in layers:

Layer Files When Loaded Tokens
Always CLAUDE.md (root) Every session, automatically ~800
On Demand 10 .claude/rules/ files Auto-load when paths: frontmatter matches ~200 each
Review Only 10 review checklists Only during /run-review skill ~800 each
Structural ARCHITECTURE.md When creating modules or changing deps ~1,200
Override 3 directory CLAUDE.md files When working in that directory ~300 each

For a typical development task, AI loads the root CLAUDE.md (~800 tokens) plus 1-2 relevant rule files (~400 tokens). Total overhead: ~1,200 tokens. That's less than 1% of the context window, yet it provides comprehensive architectural guidance.

For a code review task, the /run-review skill loads the full checklists (~8,000 tokens). Still manageable, and only when needed.

For Tech Leads: Track context usage across your team. If everyone is manually pasting architectural rules into their prompts, that's a sign your context layer isn't working. The goal is zero manual context — everything AI needs should be discoverable from CLAUDE.md references and project structure.
