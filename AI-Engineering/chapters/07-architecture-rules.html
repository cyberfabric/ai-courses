Chapter 7

Architecture Rules as AI Context

Encode your team's standards in dual-format rules — so AI follows your architecture, not its own instincts

Why Rules Before Code

In Chapter 6, you connected AI to your external tools. Now it can read your database, check your CI, and query your issue tracker. But access to information doesn't guarantee good decisions. AI can query your database perfectly and still generate code that violates your architecture.

The problem isn't capability — it's knowledge. AI doesn't know that your team uses MVI, not MVVM. It doesn't know that repository implementations must inject via Hilt, not manual construction. It doesn't know that feature modules can't depend on each other. Unless you tell it.

Chapter 2 introduced CLAUDE.md rules. This chapter takes that concept to its logical conclusion: a comprehensive rule system covering every layer of your architecture, in two formats optimized for different use cases.

The Rule of Prevention: One architecture rule in a path-scoped rule file prevents more bugs than a hundred lines of review comments after the fact. Rules are proactive. Reviews are reactive. Invest in rules.

The Dual-Format Pattern

Architecture rules serve two distinct purposes, and each needs a different format:

Review Checklists (Verbose)
  • Detailed rules with explanations
  • MUST/SHOULD/PREFER severity levels
  • Examples of correct and incorrect code
  • Used for: thorough code reviews, onboarding
  • Length: 50–100 lines per topic area
Path-Scoped Rules (Condensed)
  • One-line rule summaries with IDs
  • 25-line maximum per file
  • No examples, no explanations
  • Auto-load via paths: frontmatter
  • Live in .claude/rules/<topic>.md

The two formats work together. During development, AI auto-loads the matching condensed rule file from .claude/rules/ when it reads a file the rule applies to. During code review, the verbose checklists provide detailed criteria, loaded by an explicit skill or import. Same rules, two perspectives.

Start where it hurts: You don't need all 10 on day one. Identify the 3 areas where AI most often generates wrong code, and write rules for those first. For most teams, that's presentation, package structure, and testing.

Writing Review Checklists

A review checklist is a structured document with categorized rules. Each rule has a severity level that tells AI (and reviewers) how seriously to treat it:

  • MUST — Mandatory. Violations are bugs. AI must never generate code that breaks these.
  • SHOULD — Strong recommendation. Exceptions require justification in a comment.
  • PREFER — Soft guideline. Follow when practical, skip when it adds complexity.

Here's a real review checklist for the presentation layer:

docs/architecture/presentation-layer-ai-review-checklist.md
# Presentation Layer Review Checklist

## Screen Composables
- **[P1] MUST** accept a `Modifier` parameter as the first optional parameter
- **[P2] MUST** be stateless — state comes from ViewModel via `collectAsStateWithLifecycle()`
- **[P3] MUST** NOT call repository or use case functions directly
- **[P4] SHOULD** use `@Preview` with sample data for each visual state
- **[P5] PREFER** extracting complex UI logic into separate composable functions

## ViewModels
- **[P6] MUST** expose state as `StateFlow<UiState>` (not LiveData, not mutableStateOf)
- **[P7] MUST** handle events through a sealed interface (not individual methods)
- **[P8] MUST** inject dependencies via constructor using @HiltViewModel
- **[P9] SHOULD** use `viewModelScope.launch` for coroutines (not GlobalScope)
- **[P10] SHOULD** map domain errors to user-facing messages in the ViewModel

## UiState
- **[P11] MUST** be a data class (not a regular class)
- **[P12] MUST** have a default/initial state defined as companion object or default constructor
- **[P13] SHOULD** use sealed interfaces for loading/success/error states
- **[P14] PREFER** flattened state over nested objects for simple screens

## Events
- **[P15] MUST** be a sealed interface (not sealed class, not enum)
- **[P16] MUST** be processed in a `when` block with no `else` branch
- **[P17] SHOULD** carry only the data needed for the action

Notice the structure: categorized by component, each rule has an ID ([P1], [P2]), a severity level, and a concrete instruction. The IDs matter — they let you reference specific rules in reviews: "This violates P3 — screens must not call use cases directly."

For Tech Leads: Rule IDs make architectural discussions precise. Instead of "I think we should use StateFlow here," you say "P6 requires StateFlow." The rule is the authority, not the reviewer's preference. This depersonalizes reviews and makes them more objective.

Condensing to a Path-Scoped Rule

A review checklist is great for deep reviews, but it's too large to load into AI context for every interaction. A path-scoped rule file solves this — same rules, compressed into 25 lines or fewer, scoped to the files it applies to:

.claude/rules/presentation-layer.md
---
paths:
  - "**/presentation/**/*.kt"
  - "**/ui/**/*.kt"
---

# Presentation Layer Rules (condensed)
P1: Screen composables accept Modifier as first optional param
P2: Screens are stateless — state via collectAsStateWithLifecycle()
P3: Screens NEVER call repositories or use cases directly
P4: Previews for each visual state with sample data
P5: Extract complex UI logic into separate composables
P6: ViewModel exposes StateFlow<UiState> (not LiveData)
P7: Events via sealed interface (not individual methods)
P8: ViewModel uses @HiltViewModel with constructor injection
P9: Use viewModelScope.launch (not GlobalScope)
P10: Map domain errors to user-facing messages in ViewModel
P11: UiState is a data class with default/initial state
P12: Use sealed interfaces for loading/success/error
P13: Events are sealed interface (not class, not enum)
P14: Process events in when block without else branch

Compare the two: the checklist is 40+ lines with explanations and categories. The condensed rule body is 14 lines with one rule per line. Both encode the same knowledge, but the path-scoped rule loads into context only when Claude reads a file matching the paths: patterns — zero startup cost, fits any context window.

The 25-line limit is a discipline, not a suggestion. If your rule file body exceeds 25 lines, you're including too much detail. Ruthlessly compress. "P6: ViewModel exposes StateFlow<UiState>" is enough — AI knows what StateFlow is. You don't need to explain it.

Native Path-Scoped Rules with .claude/rules/

Older AI configurations relied on referencing rule files manually from CLAUDE.md (or worse, inlining everything in one giant CLAUDE.md). Claude Code now loads rules natively from .claude/rules/: drop a markdown file in there with a paths: frontmatter, and Claude auto-injects it whenever a matching file enters context. No manual references, no token cost when the rule isn't relevant.

Directory Layout

Place markdown files in your project's .claude/rules/ directory. Each file should cover one topic, with a descriptive filename. All .md files are discovered recursively, so you can organize rules into subdirectories like frontend/ or backend/:

Project layout
your-project/
├── .claude/
│   ├── CLAUDE.md           # Main project instructions (always loaded)
│   └── rules/
│       ├── code-style.md   # Unscoped: loads at launch
│       ├── presentation-layer.md   # paths: **/presentation/**
│       ├── data-layer.md           # paths: **/data/**
│       ├── testing.md              # paths: **/*Test.kt
│       └── frontend/               # subdirectory: discovered recursively
│           └── components.md       # paths: **/components/**/*.tsx

Rules without paths: frontmatter are loaded at launch with the same priority as .claude/CLAUDE.md. Rules with paths: frontmatter only enter context when Claude reads a file matching one of the patterns.

Frontmatter Syntax

Add YAML frontmatter at the top of the rule file. The paths: field takes a list of glob patterns:

.claude/rules/api-design.md
---
paths:
  - "src/api/**/*.ts"
  - "src/handlers/**/*.ts"
---

# API Development Rules

A1: All API endpoints validate input with zod schemas
A2: Use the standard error response shape ({ code, message })
A3: Include OpenAPI documentation comments
A4: NEVER expose internal stack traces in error responses

Glob patterns support extension matching, recursive directories, and brace expansion. Stick to the documented quoted-string list form — some unquoted variants don't parse cleanly:

PatternMatches
**/*.tsAll TypeScript files in any directory
src/**/*All files under src/
*.mdMarkdown files in the project root
src/components/*.tsxReact components in a specific directory
src/**/*.{ts,tsx}Brace expansion: TS or TSX under src/

Sharing Rules Across Projects

The .claude/rules/ directory supports symlinks. Maintain a shared rules repository and link it into multiple projects so a single update propagates everywhere:

Symlinking shared rules
# Link a whole shared directory
ln -s ~/shared-claude-rules .claude/rules/shared

# Link an individual file (e.g., company-wide security standard)
ln -s ~/company-standards/security.md .claude/rules/security.md

Symlinks are resolved and loaded normally; circular symlinks are detected and handled gracefully. For preferences that aren't project-specific, use user-level rules in ~/.claude/rules/. They apply to every project on your machine and load before project rules — project rules have higher priority and override personal ones.

Read-only trigger (intentional limitation): path-scoped rules load when Claude reads a matching file, not when it writes a new file matching the pattern. If you ask Claude to create src/api/new-endpoint.ts from scratch, the API rule won't fire because no Read happened. Workaround: include a "read an existing file in this directory first" step in your skill, or move the most critical rules into project-root CLAUDE.md so they're always in context. (See issue #23478, closed as not planned — this is documented behavior, not a pending fix.)
Path-scoped rules don't survive /compact. Once compacted, a path-scoped rule is gone from context until Claude reads a matching file again. Chapter 3 has the full survival table. Trade-off: native auto-loading vs survives-compaction. If a rule MUST hold across long sessions, drop the paths: frontmatter (so it loads unconditionally) or move it into project-root CLAUDE.md.

AGENTS.md Cross-Tool Compatibility

Path-scoped rules in .claude/rules/ are Claude-specific — Codex, Cursor, Aider, and other tools won't read them. If your team uses multiple coding agents, keep cross-tool baseline rules in AGENTS.md and let CLAUDE.md import it (covered in Chapter 2). Use .claude/rules/ for the deeper architectural rules that only Claude needs to load.

Case Study: The Recipes App Rules

Let's walk through how the recipes app implements the full dual-format pattern for its presentation layer — from problem to solution.

The Problem

Before rules, AI generated presentation code with these recurring violations:

Common AI mistakes without rules
class RecipesViewModel : ViewModel() { // No @HiltViewModel val recipes = MutableLiveData<List<Recipe>>() // LiveData instead of StateFlow fun loadRecipes() { GlobalScope.launch { // GlobalScope instead of viewModelScope recipes.value = repository.getAll() // No error handling } } }
Four violations in one class: no Hilt, LiveData, GlobalScope, no error handling. Every one of these would be caught by the rules.

The Solution

With .claude/rules/presentation-layer.md auto-loaded (because Claude read a Kotlin file under presentation/), AI generates:

AI output with rules loaded
@HiltViewModel // P8: @HiltViewModel class RecipesViewModel @Inject constructor( private val getRecipes: GetRecipesUseCase // P8: constructor injection ) : ViewModel() { private val _state = MutableStateFlow(UiState()) // P6: StateFlow val state: StateFlow<UiState> = _state.asStateFlow() fun onEvent(event: RecipesEvent) { // P7: sealed interface events when (event) { // P14: when without else is RecipesEvent.Load -> loadRecipes() is RecipesEvent.Refresh -> refresh() } } }
Every pattern matches the rules. The rule IDs in the comments show which rule drives each decision.

The file structure for the recipes app's rule system:

recipes-app — complete rules layout
.claude/rules/
  # Path-scoped rules — auto-load on matching file reads
  presentation-layer.md     (paths: **/presentation/**, 14-line body)
  domain-layer.md           (paths: **/domain/**,        10-line body)
  data-layer.md             (paths: **/data/**,          12-line body)
  testing.md                (paths: **/*Test.kt,         15-line body)
  di.md                     (paths: **/di/**,             9-line body)

  # Full review checklists — loaded for reviews via /run-review skill
  presentation-layer-ai-review-checklist.md    (65 lines)
  domain-layer-ai-review-checklist.md          (40 lines)
  data-layer-ai-review-checklist.md            (50 lines)

  di-ai-review-checklist.md                    (30 lines)

Six topic areas, 12 files, ~70 lines of condensed rules — but only the unscoped package-structure.md sits in startup context. The rest enter context on demand. That's the entire architecture encoded as AI context, with near-zero baseline cost.

Measuring Rule Compliance

Rules are only valuable if AI follows them. Here's how to measure compliance:

The Review Audit

After AI generates code, run a quick manual check against the rules. Score each rule area:

Presentation: StateFlow used? MVI pattern correct? @HiltViewModel present?
Package structure: File in correct module? No cross-feature dependencies?
Testing: Fakes used instead of mocks? Naming convention followed?
DI: Constructor injection? Correct Hilt scoping?

The Compliance Skill

Path-scoped rules don't fire on Write (see the limitation above), so a compliance skill that explicitly reads each rule file before checking git diff output is still essential — it's how you bridge the Write gap. Build it like this:

.claude/commands/check-architecture.md
Check the most recently modified files for architecture rule compliance.

## Steps
1. Read .claude/rules/presentation-layer.md
2. Read .claude/rules/domain-layer.md
3. Read .claude/rules/data-layer.md
4. Read .claude/rules/package-structure.md
5. Read .claude/rules/testing.md
6. Read .claude/rules/di.md
7. Get the list of files changed since the last commit: `git diff --name-only`
8. For each changed file, identify its layer from the file path
9. Check every applicable rule from the corresponding rule file
10. Report: list each file, the rules checked, and any violations found
11. Format as: ✅ P6 compliant / ❌ P3 violation — [explanation]
For Tech Leads: Run /check-architecture as the last step before every PR. If you've built the skill and the rules, this takes 30 seconds and catches violations that manual review would miss. Over time, track which rules are violated most often — those need clearer .ctx wording.

Evolving Rules as Architecture Changes

Architecture evolves. Your rules must evolve with it. A stale rule that contradicts your current codebase is worse than no rule — it causes AI to fight against your actual patterns.

The Rule Maintenance Lifecycle

  1. Monthly review: Compare .ctx rules against recent PRs. Do rules match what's actually being shipped?
  2. Quarterly audit: Check all 10 topic areas. Are there new patterns that need rules? Deprecated patterns with lingering rules?
  3. Migration update: When you adopt a new pattern (e.g., switching from Flow to StateFlow), update the rule immediately — don't wait for the migration to finish.
  4. Rule retirement: When a rule no longer applies (you've fully migrated away from the old pattern), remove it. Dead rules add noise.
Stale Rules (Danger)
  • "MUST use LiveData" (but you migrated to StateFlow)
  • "MUST use dagger-android" (but you switched to Hilt)
  • AI generates outdated patterns
  • Developers lose trust in the rules
Current Rules (Trust)
  • "MUST use StateFlow" (matches your current code)
  • "MUST use @HiltViewModel" (matches your DI setup)
  • AI generates modern patterns
  • Rules are the team's source of truth
The #1 rule maintenance failure: Completing a migration but forgetting to update the rules. You spend 2 weeks migrating from LiveData to StateFlow, and the rules still say "MUST use LiveData." AI happily generates LiveData code. Add "update architecture rules" to every migration checklist.

Your Chapter Exercise

Exercise: Create Your First 3 Dual-Format Rule Sets

Write architecture rules for the three topic areas where AI most often generates incorrect code in your project.

  1. Create both directories: mkdir -p .claude/rules docs/architecture
  2. Pick your 3 topic areas — typically presentation, package structure, and testing are the highest-impact starting points.
  3. Write review checklists for each topic in docs/architecture/. For each rule: assign an ID, set a severity (MUST/SHOULD/PREFER), and write a concrete instruction. Aim for 15–25 rules per topic.
  4. Create matching path-scoped rule files in .claude/rules/ — YAML frontmatter with paths: globs, then ~25 lines of one-line rules with the same IDs. Strip explanations, keep instructions.
  5. Inline your top 5 NEVER rules in project-root CLAUDE.md — these survive /compact and apply unconditionally, regardless of which file Claude is reading.
  6. Build /check-architecture — create the compliance skill from this chapter (it bridges the Write-tool limitation by reading the rules explicitly before checking diffs).
  7. Test the rules — ask AI to read an existing file in one of your ruled directories, then generate or modify code there. Verify it follows the rules.

Acceptance criteria: 3 review checklists in docs/architecture/ + 3 matching path-scoped rules in .claude/rules/ with valid paths: frontmatter. CLAUDE.md updated with the top NEVER rules inlined. /check-architecture skill created and tested. Run /memory in a session to confirm the rules load.
