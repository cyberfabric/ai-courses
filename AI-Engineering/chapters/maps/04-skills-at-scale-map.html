The Big Idea

Individual developers write prompts. Teams write skills. Skills are functions for AI — named, parameterized, documented, version-controlled in .claude/commands/. They turn tribal knowledge into shared infrastructure.

🚀 The Starter Library: 5 Essential Skills

  1. /scaffold-feature

    Creates complete feature module: Compose screen, MVI ViewModel, UseCase, Repository, tests.

  2. /run-review

    Loads all review checklists, runs architecture review, outputs MUST/SHOULD/PREFER report.

  3. /generate-tests

    Takes file path, determines layer, generates test with success/error/edge cases using fakes.

  4. /check-architecture

    Scans for violations: wrong dependencies, forbidden patterns, naming errors. Pass/fail report.

  5. /document-module

    Reads public interfaces, generates MODULE_README.md with API, dependencies, usage examples.

Covers the daily loop: create → verify → test → document. Start here, add more as patterns emerge.

📖 Key Terms

Skill
Reusable prompt stored as .md in .claude/commands/. Invoked as /skill-name. Functions for AI.
$ARGUMENTS
Variable substituted with whatever text follows the command. Makes skills parameterized and reusable.
Macro Skill
A skill that orchestrates multiple other skills into a single complex workflow (e.g., /prepare-release).
Read-Create-Verify
The universal 3-phase skill pattern: load context → generate output → validate results.
Skill Library
Categorized collection of team-shared skills in .claude/commands/ with naming conventions and quality standards.
Idempotent Skill
A skill that can be run multiple times safely — skips completed work, generates the next item.

🧪 Quick Quiz

What's the 3-phase pattern every skill should follow?

Read Context → Create → Verify. Load CLAUDE.md and .ctx files first, generate output, then run tests/checks to validate.

What are the 6 skill categories and their prefixes?

scaffold-* (create), review-*/check-* (analyze), generate-test-* (test), document-* (docs), migrate-* (transform), verb-noun (workflow).

What makes the macro skill pattern different from a regular skill?

A macro skill orchestrates multiple tasks — it references other skills' logic ("run the equivalent of /check-architecture") to create complex, multi-step workflows like release preparation.

Why should you never merge a skill that hasn't been run?

Skills fail silently — AI will try its best with a badly-written skill, producing subtly wrong output. Always verify output matches expectations before sharing with the team.

How do you safely deprecate a skill?

Create the new version alongside the old one, add a "⚠️ DEPRECATED" header to the old skill pointing to the replacement, then remove the old skill in a subsequent PR after the team transitions.

📌 Key Takeaways

  • Skills are functions for AI — named, parameterized, version-controlled in .claude/commands/.
  • Every skill follows Read Context → Create → Verify. Context loading first, always.
  • 6 categories: scaffold, review/check, generate-test, document, migrate, workflow.
  • $ARGUMENTS makes skills reusable. Don't over-parameterize — max 2-3 pieces of info.
  • Macro skills orchestrate multiple tasks into one command (e.g., /prepare-release).
  • Start with 5 skills: scaffold-feature, run-review, generate-tests, check-architecture, document-module.
  • Treat skill PRs like code PRs — test before merging, deprecate before removing.
