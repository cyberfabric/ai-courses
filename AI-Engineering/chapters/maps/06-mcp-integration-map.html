The Big Idea

Without MCP, you are the integration layer — copying and pasting between tools and AI. With MCP, AI connects to your tools directly. Your job shifts from "relay information" to "configure access boundaries."

🚀 First 3 Servers Setup

  1. GitHub MCP

    Create PAT with repo:read + actions:read. Export GITHUB_TOKEN. Add to settings.json. Test: "What's the latest CI status?"

  2. PostgreSQL MCP (Read-Only)

    CREATE USER ai_readonly with SELECT only. Export DB_READONLY_URL. Add to settings.json. Test: "Show me all tables."

  3. Issue Tracker (Linear/Jira)

    Create read-only API key. Export LINEAR_API_KEY. Add to settings.json. Test: "What issues are assigned to me?"

🏗️ The Configuration Pattern

  1. Choose Servers

    Apply the manual bridge test. Pick tools you copy-paste from 2+ times/day.

  2. Create Read-Only Credentials

    Dedicated DB user with SELECT only. API tokens with minimum permissions. 90-day expiration.

  3. Configure settings.json

    Add server entries with ${ENV_VAR} references. Commit to repo for team consistency.

  4. Document in CLAUDE.md

    List each server, its purpose, and restrictions. "NEVER suggest write queries" = AI config.

  5. Test & Verify

    Run one query per server. Don't move on until each is confirmed working.

📖 Key Terms

MCP (Model Context Protocol)
An open standard for connecting AI tools to external data sources and services via a client-server model.
MCP Server
A lightweight process that exposes tools from an external system (DB, API) for AI to call.
Manual Bridge Test
Decision framework: if you copy-paste from a tool into AI more than 2x/day, it's an MCP candidate.
${ENV_VAR} Pattern
Referencing credentials via environment variables in settings.json so configs are shareable but secrets stay personal.
Read-Only Pattern
Defense-in-depth for database MCP: dedicated user with SELECT-only permissions. Even if AI tries to write, the DB blocks it.
Token Scoping
Creating API tokens with the minimum permissions needed. Read-only, no admin, 90-day expiration.
Pillar 4a
MCP Integration — part of the broader Architecture & Integration pillar of AI readiness.

❓ Quick Quiz

Test Yourself

Where should project MCP server configs go?

In .claude/settings.json (project-level), committed to version control, with ${ENV_VAR} references for credentials.

What's the single most important rule for database MCP access?

Always use a dedicated read-only database user. Never use application credentials. Even for dev databases.

When should you NOT use an MCP server?

For files already in your project, CLI tools AI can call via Bash, one-off lookups, or tools with interactive auth flows. Use the "manual bridge test" — if you copy-paste less than twice a day, skip MCP.

Why document MCP servers in CLAUDE.md?

It serves double duty: tells the team what's available AND tells AI how to use each server appropriately (e.g., "NEVER suggest write queries" is an AI configuration instruction, not just docs).

What are the recommended first 3 MCP servers?

GitHub (PRs, CI, issues), PostgreSQL with read-only user (data exploration), and your issue tracker (Linear/Jira for requirements).

🚫 Common Mistakes

💀
Using App DB Credentials
Always create a dedicated read-only user. Your app's credentials can write, and AI might try.
🔓
Over-Scoped Tokens
GitHub token with admin + write access? That's a blast radius problem. Read-only is enough for MCP.
🏭
Connecting to Production
DB_READONLY_URL must point to dev. One typo connecting to prod can expose real user data.
🔑
Hardcoding Tokens
Never put actual tokens in settings.json. Use ${ENV_VAR} references. The file is committed to git.

📌 Key Takeaways

If You Remember Nothing Else

  • MCP connects AI to external tools (databases, issue trackers, CI) — so you stop being the copy-paste middleman.
  • Use the manual bridge test: if you copy-paste from a tool 2+ times/day, configure MCP for it.
  • Project-level config in .claude/settings.json with ${ENV_VAR} for secrets — never hardcode tokens.
  • Read-only credentials always. Dedicated DB user, scoped API tokens, dev-only connections.
  • Document servers in CLAUDE.md with usage instructions and restrictions — it configures AI behavior, not just team docs.
  • Start with 3 servers (GitHub + DB + Issue Tracker), test each before adding more.
