Team Onboarding: From Solo to Squad

Memory Map

💡 The Big Idea

👥

Adoption Is a Progression, Not an Event

Day 1 is installation. Week 1 is using skills. Month 1 is creating skills. Quarter 1 is contributing to configuration. Meet people where they are — don't dump the entire stack on them at once.

📊 The Configuration Maturity Model

  1. Level 0: No Config

    No CLAUDE.md, no skills, no rules. AI output quality is random.

  2. Level 1: Personal

    One dev has a CLAUDE.md and a few skills. Not shared. Knowledge in one head.

  3. Level 2: Shared Basics

    CLAUDE.md committed. 3-5 skills in repo. Some team members use them.

  4. Level 3: Team Standard

    Full 5-pillar config. All members using skills & hooks. Regular maintenance.

  5. Level 4: AI-Native

    Config is part of Definition of Done. Team creates & evolves skills independently.

🤝 The People Challenges

🤨
The Skeptic
Thinks AI is overhyped. Convert with evidence, not arguments. Do a paired task: manual vs AI. Let the time/quality difference speak.
🚀
The Enthusiast
Wants AI for everything. Set guardrails: always review output, never skip tests, skills need quality standards before merging.
🤷
The Pragmatist
Just wants to know what changes Monday morning. Show them the 3 skills they'll use daily. Keep it concrete.

How many team members created or modified a skill? Target: 50%+ by Month 2.
📝
CLAUDE.md Freshness
Is the team updating CLAUDE.md? Stale file = superficial adoption.

🚧 Guardrails for Enthusiasts

👀
Always Review Output
Treat AI code like a PR from a junior dev. Read every line before committing.
🧪
Never Skip Tests
Tests verify correctness, not authorship. AI-generated code still needs testing.
🏃
Run Before Commit
Never commit AI-generated code without running it locally first. Compile + test.
📏
Skills Need Standards
New skills must pass the quality checklist (Ch 4) and be reviewed before merging.

❓ Quick Quiz

Test Yourself

What are the 5 maturity levels?

Level 0: No Config, Level 1: Personal, Level 2: Shared Basics, Level 3: Team Standard, Level 4: AI-Native. Most teams start at 0-1 and should target 2-3.

What's the 30-minute session format?

10 min demo (show 3 skills), 10 min setup (install + env vars), 10 min hands-on (everyone runs a skill). Don't flip this — demos should be fast, hands-on time matters most.

How do you convert a skeptic?

Evidence, not arguments. Do a paired task: they write manually, you use AI. Compare time and quality. Concrete evidence beats philosophical debate. Also respect their expertise — quality is their real concern.

What's the biggest onboarding mistake?

Spending 25 minutes presenting slides and 5 minutes doing hands-on. People learn by doing, not watching. Flip the ratio.

When does adoption either stick or fade?

Month 1. If team members are still just running existing skills (consumers), push them to create one skill and one rule in .claude/rules/ (contributors). Creation = investment = stickiness.

📌 Key Takeaways

If You Remember Nothing Else

  • Adoption is a progression: Day 1 (install), Week 1 (use skills), Month 1 (create skills). Don't skip levels.
  • The 30-minute session: Demo (10 min) → Setup (10 min) → Hands-on (10 min). Show the result first, explain later.
  • Skeptics need evidence (paired tasks, time comparisons). Enthusiasts need guardrails (review output, never skip tests).
  • Shared config in repo (CLAUDE.md, skills, hooks, MCP). Personal config stays local (~/.claude/, env vars).
  • Measure quality, not just usage. Skill contribution rate, rule compliance, and CLAUDE.md freshness matter more than skill invocation count.
  • Month 1 is make-or-break. Push every team member from consumer to contributor — creating one skill or rule cements adoption.
