📐 3 CLAUDE.md Caching Patterns

  1. Stable Content First

    Architecture rules and naming conventions (rarely change) go at the top. Sprint context and known issues (change weekly) go at the bottom. Cache invalidates from the point of change downward.

  2. Minimal but Sufficient

    Enough rules for correct output, not a 500-line architecture document. "VMs use UseCases, never repos directly" beats explaining Clean Architecture from scratch.

  3. Reference Instead of Inline

    Inline only NEVER rules (~50 tokens). Reference .ctx files for layer details (~200 tokens each, loaded on demand). Saves 600+ tokens on interactions that don't touch those layers.

❓ Quick Quiz

Test Yourself

Why is a rich CLAUDE.md nearly free in practice?

Because prompt caching matches from the beginning of the prompt. CLAUDE.md content is sent first, stays stable between requests, and gets cached at a 90% discount. After the first request in a session, it's nearly free.

What happens to the cache after a 10-minute break?

The cache expires — it has a 5-minute TTL. Your next request pays full price for all input tokens. The cache rebuilds from that request for subsequent interactions.

Why should stable content go at the top of CLAUDE.md?

Cache matching works on a prefix basis — from the beginning forward. If the top is stable, those tokens stay cached even when the bottom changes. If the top changes, the entire cache is invalidated.

What's cheaper: more input context or more regeneration cycles?

More input context, especially when cached. Output tokens cost ~3x more than input. And regeneration means paying for output multiple times. Rich cached context + 1 correct output beats minimal context + 3 wrong outputs.

When does high cost signal a configuration problem?

When you see: one dev 5x above average (workflow issue), one skill 10x more expensive (context bloat), high regeneration rates (missing rules), or cache hit rate below 60% (unstable CLAUDE.md). Fix the configuration, not the budget.

📌 Key Takeaways

If You Remember Nothing Else

  • Prompt caching makes rich context nearly free. CLAUDE.md + .ctx files are sent first, cached at 90% discount. Invest in context — it pays for itself.
  • Output tokens cost ~3x more than input. Preventing regeneration (by providing precise rules) saves the expensive kind of tokens.
  • Stable content first in CLAUDE.md. Architecture rules (top, rarely change) → sprint context (bottom, changes weekly). Cache invalidates from the change point down.
  • Match models to tasks: Opus for architecture, Sonnet for implementation, Haiku for formatting. Don't over-optimize — rework costs more than the right model.
  • 3–5x cost difference between configured ($0.03–0.08/interaction) and unconfigured ($0.15–0.25). At team scale, that's $1,000–$2,000/month saved.
  • Fix configuration, not budgets. High regeneration = missing rules. Low cache hits = unstable CLAUDE.md. Never restrict usage — improve the context.
