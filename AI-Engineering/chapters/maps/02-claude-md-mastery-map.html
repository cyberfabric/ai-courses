The Big Idea

A production CLAUDE.md doesn't teach architecture — it encodes your specific decisions. Tell AI what NOT to do (negative space), be specific enough that violations are binary, and lean on @imports, .claude/rules/, and diagnostics so the file stays under 200 lines.

🔗 Sharing Instructions: @imports, AGENTS.md, Excludes

📎
@path Imports
Inline expansion: @README.md, @docs/git.md, @~/.claude/personal.md. Max 5 hops deep. Loads at launch — for organization, not token savings.
🤝
AGENTS.md Compatibility
Claude reads CLAUDE.md, not AGENTS.md. Use @AGENTS.md at the top of CLAUDE.md, then add a "## Claude Code" section with overrides below.
🙈
claudeMdExcludes
Glob list in .claude/settings.local.json. Skip noisy ancestor CLAUDE.md files in monorepos. Managed-policy files cannot be excluded.
🔐
First-Time Approval
External imports trigger an approval dialog the first time. Decline once and they stay disabled — no re-prompt.

⚖️ Vague vs Specific Rules

❌ Vague (Don't)✅ Specific (Do)
"Use Clean Architecture""presentation → domain → data. NEVER reverse."
"Follow best practices""Max 3 public functions per ViewModel"
"Keep code clean""All Composables take a Modifier parameter"
"Use proper DI""Constructor injection via @Inject. NEVER field inject."
"Write good tests""Every UseCase has a matching test with 3+ scenarios"

The test: Could AI violate it without ambiguity? If yes, it's specific enough.

🗿 The Negative Space Technique

🗿

The Sculptor's Approach

Don't describe the statue — describe which marble to remove. Stack enough NEVER rules and AI is funneled toward your exact architecture through elimination. 10-15 forbidden patterns is the sweet spot. Past 30, split into .claude/rules/ (the peer of .claude/skills/) instead.

🚫
NEVER Repo → ViewModel
Always go through UseCase.
🚫
NEVER Business in Composable
Extract to ViewModel.
🚫
NEVER Feature → Feature
Features are isolated modules.
🚫
NEVER Room → Domain
Map entities to domain models.

🧬 CLAUDE.md Anatomy: 6 Sections

  1. 📛 Project Identity

    One paragraph max. What is this? Tech stack. Platform.

  2. 🏛️ Architecture

    Layers with dependency direction. Patterns. UI framework.

  3. 📦 Module Structure

    Every module, its purpose, its boundaries.

  4. 🚫 Forbidden Patterns

    10-15 NEVER rules. The negative space.

  5. 📝 Naming & Testing

    Templates for every file type. Test standards and naming.

  6. ⚙️ Workflow & Imports

    Build commands, lint, skills, plus @imports for shared rules.

Order matters: AI processes top-to-bottom. Most important context first.

🩺 Diagnostics: What Actually Loaded?

📋
/memory
Lists every CLAUDE.md, CLAUDE.local.md, and rules file loaded in the session. Toggles auto-memory. First place to look when a rule is ignored.
📡
InstructionsLoaded hook
Audit-only observability. Logs which files load, when, and why — perfect for path-scoped rules and lazy subdirectory loads.
🪄

Interactive multi-phase /init. Asks which artifacts to set up, explores via subagent, presents a reviewable proposal before writing.
🤐
HTML Comments Hide
Block-level <!-- maintainer notes --> are stripped before injection. Free space for human-only commentary.

⚠️ The 4 Pitfalls

📚
The Novel
Don't explain what Clean Architecture is. AI already knows. Target under 200 lines per file — longer reduces adherence.
📋
Copy-Paste from Docs
CLAUDE.md speaks to AI (terse, rules). Docs speak to humans (narrative). Reference, don't duplicate.
🕳️
Rules Without Enforcement
Pair critical rules with hooks (Ch 5) or review checklists (Ch 7) that catch violations.
🧑
Personal as Project
Trailing commas go in ~/.claude/CLAUDE.md. Project file = team-agreed decisions only.

🧪 Quick Quiz

What's wrong with the rule "Use Clean Architecture"?

It's too vague — AI knows dozens of valid Clean Architecture interpretations. Instead, specify: "presentation → domain → data. NEVER reverse dependency direction."

Where can a project CLAUDE.md live, and where does subdirectory CLAUDE.md fit in?

Project CLAUDE.md can live at ./CLAUDE.md or ./.claude/CLAUDE.md — both load at launch. Subdirectory CLAUDE.md files (e.g., ./feature/billing/CLAUDE.md) are not a fifth tier; they load on demand only when Claude reads files in that directory.

Why is "negative space" (NEVER rules) often more effective than positive rules?

AI already knows thousands of valid patterns. Positive rules pick one; NEVER rules eliminate wrong ones. Stack enough and AI is funneled toward your exact architecture through elimination.

Your repo already has an AGENTS.md for Codex. How do you make Claude Code use the same rules?

Create a CLAUDE.md with @AGENTS.md at the top, then add a "## Claude Code" section underneath for Claude-specific overrides (plan mode, hooks, skills). Both tools read the same baseline.

Claude seems to ignore a rule you added to CLAUDE.md. What's the first thing you check?

Run /memory. It lists every instruction file loaded in the session. If your file isn't there, Claude can't see it — usually a path typo, the wrong working directory, or an entry in claudeMdExcludes.

What's the "3-strike rule" for CLAUDE.md maintenance?

If AI makes the same mistake 3 times across sessions, add a new NEVER rule. Don't keep correcting AI — correct the CLAUDE.md instead.

📌 Key Takeaways

  • Four-tier hierarchy: Managed Policy → Project (./CLAUDE.md or ./.claude/CLAUDE.md) → User → Local. Subdirectory files load on demand, not at launch.
  • Use @imports for organization (max 5 hops); they load at launch, so they don't save context.
  • Mixed-tool repos: @AGENTS.md at the top of CLAUDE.md, Claude-specific overrides below.
  • Climb the specificity ladder: "NEVER field inject" beats "use proper DI."
  • Negative space works: 10-15 NEVER rules define architecture through elimination. Past 30, split into .claude/rules/ (peer of .claude/skills/).
  • Target under 200 lines per file — longer reduces adherence. Use HTML comments for human-only notes (stripped before injection).
  • Diagnose load issues with /memory, the InstructionsLoaded hook, and CLAUDE_CODE_NEW_INIT=1 for interactive scaffolding.
  • Maintain with the 3-strike rule. Run /memory after every edit to confirm what loaded.
