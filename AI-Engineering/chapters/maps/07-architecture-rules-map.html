💡 The Big Idea

🏗️

Rules Prevent, Reviews React

One architecture rule in a path-scoped .claude/rules/ file prevents more bugs than a hundred review comments after the fact. Encode your standards before AI writes code, and let Claude auto-load them when matching files enter context.

🎯 Native Path-Scoped Rules

🎯

Auto-Load on Match

Drop a markdown file in .claude/rules/ with a paths: frontmatter, and Claude injects it whenever a matching file enters context. No manual references, no token cost when irrelevant. Unscoped rules (no frontmatter) load at launch like CLAUDE.md.

📁
Recursive Discovery
All .md files under .claude/rules/ are discovered, including subdirectories like frontend/, backend/.
🪄
Brace Expansion
"src/**/*.{ts,tsx}" matches multiple extensions in one pattern. Lists in paths: are quoted strings.
🔗
Symlinks Supported
ln -s ~/shared-claude-rules .claude/rules/shared — one canonical rules repo, linked into many projects.
🏠
User-Level Rules
~/.claude/rules/ applies everywhere on your machine. Loads before project rules — project rules win on conflict.

⚠️ Three Limitations to Know

📖
Read-Only Trigger
Path-scoped rules fire on Read, NOT Write. Creating a new file from scratch won't trigger the rule (issue #23478, closed as not planned). Workaround: a compliance skill that reads rules explicitly, or critical rules in CLAUDE.md.
🌪️
Lost After /compact
Path-scoped rules summarize away during compaction and reload only when a matching file is read again. If a rule MUST persist, drop the paths: frontmatter or move it into project-root CLAUDE.md.
🐛
paths: YAML Bugs
Open issues #17204, #13905 report parser failures with quoted YAML-list paths: in some configs; #21858 reports user-level ~/.claude/rules/ silently ignores paths:. After changes, run /memory to confirm the rule actually loaded.

🤝 Cross-Tool: AGENTS.md

🤝

Mixed-Tool Shops

Path-scoped rules in .claude/rules/ are Claude-only. If your team also uses Codex/Cursor/Aider, keep cross-tool baseline rules in AGENTS.md and let CLAUDE.md import it. Use .claude/rules/ for the deep architectural rules only Claude needs.

🛠️ The Compliance Skill

  1. Read rule files explicitly

    Bridges the Write-tool limitation: skill reads each rule file before checking diffs.

  2. Get changed files

    git diff --name-only lists what was modified.

  3. Match files to layers

    Identify each changed file's layer from its path.

  4. Check applicable rules

    Walk through each rule with the matching ID and severity.

  5. Report violations

    ✅ P6 compliant / ❌ P3 violation — [explanation].

♻️ Rule Lifecycle

  1. Monthly Review

    Compare rule files against recent PRs — do they match shipped code?

  2. Quarterly Audit

    Walk all 10 topic areas. New patterns? Deprecated rules?

  3. Migration Update

    Adopting a new pattern? Update the rule during migration, not after.

  4. Rule Retirement

    Old pattern fully removed? Delete the rule. Dead rules add noise.

📖 Key Terms

.claude/rules/
Per-project directory for modular rule files. All .md files discovered recursively.
paths: frontmatter
YAML field listing glob patterns. Rule loads when Claude reads a matching file.
Unscoped rule
Rule file without paths: frontmatter. Loads at launch with the same priority as .claude/CLAUDE.md.
Brace expansion
Glob shorthand: **/*.{ts,tsx} matches both extensions in one pattern.
User-level rules
~/.claude/rules/ applies to every project. Loaded before project rules; project rules override on conflict.
Read-only trigger
Documented limitation: path-scoped rules fire on Read, not Write. Issue #23478, closed as not planned.
/memory command
Run inside a session to list every CLAUDE.md, CLAUDE.local.md, and rule file actually loaded. Use it to verify a new paths: rule is being picked up — especially given open YAML parser bugs (#17204, #13905, #21858).
MUST/SHOULD/PREFER
Severity levels in review checklists. MUST = mandatory; SHOULD = strong; PREFER = soft.
Rule ID
Short identifier (P1, D3) per rule. Lets reviewers cite rules by name: "violates P3."
AGENTS.md
Cross-tool fallback for shops using multiple coding agents. Claude reads CLAUDE.md, which can @AGENTS.md.

🧪 Quick Quiz

What's the difference between an unscoped rule and a path-scoped rule?

Unscoped (no paths: frontmatter) loads at launch like CLAUDE.md and survives /compact. Path-scoped (with paths:) loads only when Claude reads a matching file, costs zero tokens otherwise — but is lost after /compact until re-triggered.

You ask Claude to create src/api/new-endpoint.ts. Why doesn't your .claude/rules/api.md trigger?

Path-scoped rules fire on Read, not Write. No file was read, so no rule injection. Workaround: have Claude read an existing file in the directory first, or build a compliance skill that reads rules explicitly, or move the most critical rules to project-root CLAUDE.md.

A rule must apply to every long session, even after /compact. Where does it live?

Either remove the paths: frontmatter (so it loads unconditionally) or inline it in project-root CLAUDE.md. Both survive compaction.

How do you share architecture rules across multiple projects?

Symlink them. ln -s ~/shared-claude-rules .claude/rules/shared links a whole directory; ln -s ~/company-standards/security.md .claude/rules/security.md links a single file. One canonical source, many projects.

You also use Cursor and Codex. Where do shared rules go?

In AGENTS.md (cross-tool standard). Have CLAUDE.md import it via @AGENTS.md, then put Claude-only deep architectural rules in .claude/rules/.

✅ Key Takeaways

📌 Remember This

  • Rules prevent. Reviews react. Invest in .claude/rules/ — it pays compounding interest.
  • Native auto-loading: drop a file in .claude/rules/ with paths: frontmatter and Claude injects it on matching reads.
  • Dual-format: condensed path-scoped rule (development) + verbose review checklist (review).
  • Two limits to know: (1) Read-only trigger — rules don't fire on Write. (2) Lost after /compact — reload only on next matching read.
  • Critical rules go in project-root CLAUDE.md so they survive /compact and apply unconditionally.
  • Symlink .claude/rules/ to share rules across projects. Use ~/.claude/rules/ for personal preferences.
  • Mixed-tool shops: AGENTS.md for cross-tool baseline, .claude/rules/ for Claude-only deep rules.
  • Build /check-architecture — the compliance skill that bridges the Write-tool gap by reading rules explicitly.
  • Update rules with every migration. Stale rules are worse than no rules — AI fights against your real patterns.
