Chapter 5

Hooks and Guardrails: Automated Quality Gates

Safety nets that run before and after every AI action — so mistakes get caught automatically

From Trust to Verify Automatically

In Chapter 4, you built a library of skills that encode your team's workflows. Skills tell AI what to do. Hooks tell AI what it can't get away with.

A hook is a shell command that Claude Code runs automatically at specific moments — before a tool is used, after a tool completes, or when a subagent finishes. If the hook exits with a non-zero code, the action is blocked or flagged. Think of hooks as CI/CD checks, but running locally on every AI action — not just on push.

Without hooks, your safety strategy is "review every AI change manually." With hooks, dangerous operations get blocked before they happen, formatting gets applied automatically, and tests run after every code change. The shift from manual review to automated guardrails is what makes AI development production-grade.

The Key Insight: Skills define what AI should do. Hooks enforce what AI must NOT do. Together, they form a complete quality system — proactive guidance (skills) plus reactive safety (hooks).

How Hooks Work

Hooks are configured in your Claude Code settings file. Each hook specifies when it runs (the event), what it checks (the matcher), and what it does (the command):

.claude/settings.json — hook configuration structure
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hook": "bash .claude/hooks/check-dangerous-commands.sh"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hook": "bash .claude/hooks/post-edit-lint.sh"
      }
    ],
    "SubagentStop": [
      {
        "matcher": "",
        "hook": "bash .claude/hooks/subagent-review.sh"
      }
    ]
  }
}

Three hook events cover the full AI action lifecycle:

PreToolUse
Before action
→
AI Action
Tool executes
→
PostToolUse
After action

The matcher field filters which tool triggers the hook. Use tool names like Bash, Write, Edit, Read, or pipe-separated combinations like Write|Edit. An empty matcher matches everything.

How hooks receive context: Hooks receive the tool name and input as environment variables. The tool's input is passed via stdin as JSON. Your script reads it with INPUT=$(cat) and parses fields with jq. This is how hooks make intelligent decisions about whether to allow or block specific actions.

The Hook Matrix

Not all hooks are created equal. Some prevent data loss. Others catch style issues. Organize your hooks by risk level to prioritize what to implement first:

RiskHookPurposePriority
CriticalPreToolUse / BashBlock destructive commandsMust-have
CriticalPreToolUse / WritePrevent writes to protected filesMust-have
HighPostToolUse / EditRun linter after every code editShould-have
HighPostToolUse / WriteRun formatter on new filesShould-have
MediumPostToolUse / EditRun unit tests for modified moduleNice-to-have
MediumSubagentStopSummary check for agent tasksNice-to-have
Rule of Thumb: Implement critical hooks first. A missing formatter hook means messy code that you can clean up. A missing destructive-command blocker means deleted databases that you can't recover. Prioritize by damage potential, not convenience.

PreToolUse Hooks: The Safety Gates

PreToolUse hooks are your first line of defense. They run before AI executes a tool, giving you a chance to block dangerous operations entirely.

Blocking Destructive Bash Commands

This is the single most important hook in any project. It prevents AI from running commands that could cause irreversible damage:

.claude/hooks/check-dangerous-commands.sh
#!/bin/bash
# PreToolUse hook: Block dangerous bash commands
# Reads tool input from stdin as JSON

INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.command // empty')

# List of dangerous patterns
DANGEROUS_PATTERNS=(
  "rm -rf /"
  "rm -rf ~"
  "rm -rf \."
  "DROP TABLE"
  "DROP DATABASE"
  "git push.*--force.*main"
  "git push.*--force.*master"
  "git reset --hard"
  "chmod -R 777"
)

for pattern in "${DANGEROUS_PATTERNS[@]}"; do
  if echo "$COMMAND" | grep -qiE "$pattern"; then
    echo "BLOCKED: Command matches dangerous pattern: $pattern" >&2
    echo "Command was: $COMMAND" >&2
    exit 1
  fi
done

exit 0

When this hook detects a dangerous pattern, it exits with code 1, which blocks the tool from executing. The error message goes to stderr and is shown to both AI and the developer, explaining why the action was blocked.

Protecting Sensitive Files

Some files should never be modified by AI — environment files, credentials, CI configurations, or production database configs:

.claude/hooks/protect-sensitive-files.sh
#!/bin/bash
# PreToolUse hook: Prevent writes to sensitive files
# Applies to Write and Edit tools

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.file_path // empty')

PROTECTED_PATTERNS=(
  "\.env$"
  "\.env\."
  "credentials"
  "secret"
  "\.pem$"
  "\.key$"
  "google-services\.json"
  "\.github/workflows/"
)

for pattern in "${PROTECTED_PATTERNS[@]}"; do
  if echo "$FILE_PATH" | grep -qiE "$pattern"; then
    echo "BLOCKED: Cannot modify protected file: $FILE_PATH" >&2
    echo "Matches protection pattern: $pattern" >&2
    echo "Modify this file manually if changes are needed." >&2
    exit 1
  fi
done

exit 0
For Tech Leads: The protected files list should be reviewed quarterly. As your project evolves, new sensitive files appear — new CI configs, new credential stores, signing keystores. Add them to the hook before they get accidentally modified. One team learned this the hard way when AI helpfully "fixed" their CI pipeline and broke the release process.

PostToolUse Hooks: Automatic Quality Enforcement

PostToolUse hooks run after AI completes an action. They're your automatic quality layer — formatting, linting, and testing happen without anyone remembering to run them.

Auto-Format After Every Edit

.claude/hooks/post-edit-lint.sh
#!/bin/bash
# PostToolUse hook: Format and lint after code edits
# Runs after Write and Edit tools

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.file_path // empty')

# Only process code files
case "$FILE_PATH" in
  *.kt)
    ktlint --format "$FILE_PATH" 2>/dev/null
    ;;
  *.java)
    google-java-format --replace "$FILE_PATH" 2>/dev/null
    ;;
  *.ts|*.tsx|*.js|*.jsx)
    npx prettier --write "$FILE_PATH" 2>/dev/null
    ;;
  *.py)
    black "$FILE_PATH" 2>/dev/null
    ;;
  *.swift)
    swiftformat "$FILE_PATH" 2>/dev/null
    ;;
esac

# Exit 0 — formatting issues shouldn't block the action
exit 0

Notice that this hook always exits with code 0. The action has already completed — there's nothing to block. PostToolUse hooks that exit with non-zero surface the error as feedback to AI, which can use that information to self-correct. Use non-zero exits for warnings, not blocking.

Run Tests After Module Changes

.claude/hooks/post-edit-test.sh
#!/bin/bash
# PostToolUse hook: Run relevant tests after code edits
# Only runs for source files in feature/ or core/ directories

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.file_path // empty')

# Skip test files themselves to avoid infinite loops
if echo "$FILE_PATH" | grep -qE "Test\.kt$|_test\.go$|\.test\.(ts|js)$"; then
  exit 0
fi

# Extract module from file path (e.g., feature/recipes → :feature:recipes)
if echo "$FILE_PATH" | grep -qE "^(feature|core)/"; then
  MODULE=$(echo "$FILE_PATH" | sed -E 's|^(feature|core)/([^/]+)/.*|:\1:\2|')

  echo "Running tests for module $MODULE..." >&2
  ./gradlew "${MODULE}:testDebugUnitTest" --quiet 2>&1

  if [ $? -ne 0 ]; then
    echo "WARNING: Tests failed for $MODULE after edit" >&2
    echo "File changed: $FILE_PATH" >&2
    exit 1  # Non-zero signals the issue back to AI
  fi
fi

exit 0
Without Post-Edit Hooks
  • AI edits files, developer reviews
  • Manually runs formatter — finds issues
  • Manually runs tests — 2 failures
  • Back to AI for fixes, another review round
With Post-Edit Hooks
  • AI edits files, hooks run automatically
  • Formatter and tests execute per file/module
  • AI sees failures, fixes in same session
  • Developer reviews clean, tested output

SubagentStop Hooks: Watching the Watchers

When you use Claude Code's agent capabilities for long-running tasks — like a background agent refactoring across modules — SubagentStop hooks run when the agent completes. They provide oversight for work you weren't watching in real time.

.claude/hooks/subagent-review.sh
#!/bin/bash
# SubagentStop hook: Summarize what a subagent changed

# Count changed files
CHANGED=$(git diff --name-only | wc -l)
ADDED=$(git diff --name-only --diff-filter=A | wc -l)
DELETED=$(git diff --name-only --diff-filter=D | wc -l)

echo "=== Subagent Completion Report ===" >&2
echo "Files modified: $CHANGED" >&2
echo "Files added: $ADDED" >&2
echo "Files deleted: $DELETED" >&2

# Flag large changes for manual review
if [ "$CHANGED" -gt 20 ]; then
  echo "WARNING: Large change set (${CHANGED} files). Review recommended." >&2
  exit 1
fi

# Check for changes to protected directories
if git diff --name-only | grep -qE "^(\.github|\.claude/settings|build\.gradle)"; then
  echo "WARNING: Infrastructure files were modified. Review carefully." >&2
  exit 1
fi

exit 0

SubagentStop hooks are especially valuable when you run agents with --background or in headless CI mode. They provide the oversight layer that makes autonomous agent work safe for production codebases. You can also pipe the report to a shared log (.claude/agent-activity.log) for team-wide audit trails.

Combining Hooks with CI/CD

Hooks and CI/CD are complementary, not competing. Hooks catch issues at development time (fast, local). CI/CD catches issues at integration time (thorough, remote). The best strategy layers both:

CheckHook (Local)CI/CD (Remote)
Dangerous commandsPreToolUse — block immediatelyN/A — never reaches CI
Code formattingPostToolUse — auto-fixLint check — fail if unformatted
Unit testsPostToolUse — affected moduleFull test suite — all modules
Architecture rulesNot practical per-editFull scan on PR
Security scanPreToolUse — obvious patternsSAST/DAST — deep analysis
The Layered Defense Model: Hooks are your fast, local first responder. CI/CD is your thorough, comprehensive inspector. Together, they create a quality system where issues are caught at the earliest possible moment — and nothing slips through to production.

Hook Debugging: When Things Go Wrong

Hooks can fail silently, misconfigure subtly, or block legitimate actions. Here's the debugging toolkit:

The four most common hook failures, in order of frequency:

  1. Missing execute permission — fix with chmod +x .claude/hooks/*.sh
  2. Wrong jq path for input parsing — test with echo '{}' | jq -r '.field'
  3. Matcher doesn't match tool name — use exact names: Bash, Write, Edit
  4. Overly broad pattern matching — hook blocks legitimate operations

The Hook Debugging Workflow

  1. Check registration: Open .claude/settings.json and verify the event, matcher, and path.
  2. Check permissions: Run ls -la .claude/hooks/ — all scripts need execute permission.
  3. Test in isolation: Pipe simulated JSON input to your hook and check the exit code: echo '{"command":"rm -rf /"}' | bash .claude/hooks/check-dangerous-commands.sh; echo $?
  4. Check stderr output: Hook messages must go to stderr (>&2). If you see no output, the echo is going to stdout.
  5. Narrow the matcher: If a hook triggers too broadly, make it more specific. Use Write instead of Write|Edit if only writes need checking.
Warning: Never disable all hooks to "get past" a blocking issue. If a hook blocks a legitimate action, fix the hook's pattern — don't remove the safety net. Disabling hooks for one task and forgetting to re-enable them is how databases get dropped.

Performance Considerations

Hooks run on every matching tool use. A slow hook on Edit means every code change takes longer. Design hooks with performance in mind:

Hook Type Acceptable Latency Guidance
PreToolUse pattern check < 100ms Pure text matching — should be instant
PostToolUse formatter < 2s Single-file formatting is fast enough
PostToolUse unit tests 5–30s Only run for affected module, not full suite
SubagentStop summary 1–5s Git commands are fast — keep reports brief

The biggest performance trap is running full test suites on every edit. Scope your test hook to the modified module only. Running :feature:recipes:testDebugUnitTest takes 10 seconds. Running the entire project's test suite takes 3 minutes. One is acceptable per-edit latency. The other kills your workflow. If hooks slow you down during exploratory work, add a conditional CLAUDE_SKIP_TESTS=1 environment variable — but be disciplined about re-enabling it.
