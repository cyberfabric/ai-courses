Chapter 10

Cost Engineering: Caching and Token Strategy

A well-configured project isn't just better — it's cheaper

Everything you've built so far — CLAUDE.md, skills, hooks, architecture rules, thinking guidelines — has a price tag measured in tokens. At the individual level, it's barely noticeable. At team scale, with 10 developers running 50+ AI interactions per day, costs compound fast.

The good news: the same configuration that makes AI produce better code also makes it cheaper. A well-structured CLAUDE.md gets cached. Precise .ctx files reduce context bloat. Skills that skip extended thinking on simple tasks save tokens. Cost engineering isn't a separate concern — it's a natural outcome of good configuration.

This chapter teaches you how the cost mechanics work and how to optimize your project configuration for maximum value per token.

How Prompt Caching Works

Prompt caching is the single most important cost mechanism to understand. When you send a request to Claude, the system checks whether the beginning of your prompt matches a recently cached prompt. If it does, those cached tokens are charged at a fraction of the normal price.

The key insight: Your CLAUDE.md, .ctx files, and skill definitions are sent at the beginning of every interaction. If they're stable (not changing between requests), they get cached and become nearly free after the first request. This means rich project configuration costs almost nothing in practice.

The 5-Minute TTL

Cached content expires after 5 minutes of inactivity. This has practical implications:

ScenarioCache StatusCost Impact
Rapid back-and-forth conversation (every 1-2 min)Cache stays warmCLAUDE.md + rules nearly free
Thinking for 4 minutes, then respondingCache still warmSame as above
Taking a 10-minute coffee break mid-taskCache expiresFull price on next request
Starting a new conversationCache coldFirst request is full price
Practical tip: If you're about to step away for more than 5 minutes during a complex task, consider finishing the current step first. The next interaction will re-cache your context, but you'll pay full price for that one request.

What Gets Cached

The cache works on a prefix basis — it matches from the beginning of the prompt forward. This means the order of your context matters:

1

System Prompt

Always first. Includes CLAUDE.md content. Most cacheable — rarely changes.

→
2

Referenced Files

.ctx files, skill definitions. Cacheable when stable.

→
3

Conversation History

Prior messages. Grows with each turn. Partially cacheable.

→
4

Current Request

Your latest message. Never cached (it's new).

The implication: CLAUDE.md content at position 1 gets cached most reliably. This is why investing in a comprehensive CLAUDE.md is almost free in terms of ongoing token cost — you pay once, then it's cached for every subsequent interaction in the session.

The Context Budget

Every AI interaction has two cost components: input tokens (what you send) and output tokens (what AI generates). Understanding this split is essential for cost optimization.

~3x

The math is clear: sending 2,000 tokens of architecture rules as cached input is negligible cost. But generating 2,000 tokens of incorrect code that you then have to regenerate? That's expensive. More context upfront = less wasted output.

The poorly-configured project uses 3x more output tokens — the expensive kind. The well-configured project spends a little more on input but gets it right the first time. Over a team of 10 developers doing 50 interactions daily, this difference is massive.

Model Selection by Task Type

Not every task needs the most powerful model. Different Claude models have different cost profiles, and matching the right model to the right task is a significant cost lever.

ModelBest ForCost LevelWhen to Use
OpusArchitecture, complex reasoning$$$Module extraction, migration planning, ADRs, debugging race conditions
SonnetImplementation, code generation$$Feature development, test writing, code review, refactoring within patterns
HaikuSimple tasks, formatting$Linting, formatting, simple edits, documentation updates, commit messages
Don't over-optimize: Using Haiku for architecture decisions saves money but produces lower quality. The rework cost of fixing a bad architectural plan far exceeds the savings from using a cheaper model. Match the model to the task complexity, not just the price.

Encoding Model Selection in CLAUDE.md

You can guide your team's model choices by documenting recommendations directly in your project configuration:

CLAUDE.md
## Model Selection Guidelines

### Use Opus (highest capability) for:
- Architecture decisions and module design
- Migration planning and dependency analysis
- Complex debugging (race conditions, state corruption)
- ADR generation and trade-off analysis
- Code review of critical system components

### Use Sonnet (balanced) for:
- Feature implementation following established patterns
- Test generation and test refactoring
- Code review of standard feature PRs
- Refactoring within a single module
- Documentation with technical depth

### Use Haiku (fastest, cheapest) for:
- Formatting and linting fixes
- Simple string/config changes
- Commit message generation
- Boilerplate that follows strict templates
- Quick lookups and explanations
For Team Leads: Model selection isn't about restricting developers — it's about setting defaults. A developer who's debugging a subtle race condition should reach for Opus without feeling guilty about cost. A developer fixing a typo shouldn't need to think about it — Haiku is the natural choice. Document the defaults so the decision is automatic.

Cost Optimization Patterns for CLAUDE.md

The structure of your CLAUDE.md affects how well it caches. Here are patterns that maximize cache efficiency:

Pattern 1: Stable Content First

Put content that rarely changes at the top of CLAUDE.md. Content that changes frequently should go near the bottom. Cache matching works on a prefix — if the top of the file is identical between requests, those tokens stay cached even if the bottom differs.

CLAUDE.md — Caching-Optimized Order
# Project Name

## Architecture Rules (rarely changes)
- NEVER add feature-to-feature dependencies
- NEVER use LiveData in new code — use StateFlow
- NEVER call repositories directly from composables

## Module Structure (rarely changes)
:app → :feature-* → :core-* → :shared
No reverse dependencies allowed.

## Naming Conventions (rarely changes)
- ViewModels: {Feature}ViewModel
- Screens: {Feature}Screen
- UseCases: {Verb}{Noun}UseCase

## Current Sprint Context (changes weekly)
Sprint 24: Focus on profile module extraction.
Avoid touching :feature-search — active refactor by another dev.

## Known Issues (changes frequently)
- Flaky test in RecipeDetailViewModelTest — skip if red
- Room migration 15→16 needs manual review
Why this order matters: If "Architecture Rules" and "Module Structure" are at the top and stable, they cache across every interaction. "Current Sprint Context" at the bottom changes weekly — when it changes, only the last portion invalidates the cache, not the entire CLAUDE.md.

Pattern 2: Minimal but Sufficient Context

More context isn't always better. Each additional token in your CLAUDE.md adds cost (even if cached, it's not zero). The goal is sufficient context — enough for AI to produce correct output, not more.

Too LittleJust RightToo Much
"Use Clean Architecture" "Layers: presentation → domain → data. No reverse deps. VMs use UseCases, never repos directly." A 500-line architecture document explaining Clean Architecture from scratch
No testing rules "Tests use fakes, not mocks. Given-When-Then naming. Each ViewModel test covers: init state, success, error, loading." Every test pattern example for every ViewModel in the project
AI guesses wrong → rework AI follows rules → correct output Wasted tokens, potential confusion from contradictions

Pattern 3: Reference Instead of Inline

For detailed rules that only apply to specific tasks, reference .ctx files instead of inlining everything in CLAUDE.md:

CLAUDE.md — Reference Pattern
## Architecture Rules

### Always loaded (inlined):
- NEVER add feature-to-feature dependencies
- NEVER use LiveData in new code
- VMs expose StateFlow<UiState>, not mutableStateOf

### Load on demand (referenced):
- Before editing UI: read docs/rules/presentation-layer-ctx.md
- Before editing data layer: read docs/rules/data-layer-ctx.md
- Before writing tests: read docs/rules/testing-ctx.md
- Before modifying Gradle: read docs/rules/gradle-ctx.md

The inlined NEVER rules (50 tokens) are always in context. The .ctx files (200 tokens each) are loaded only when relevant. This saves 600+ tokens on every interaction that doesn't touch those specific layers.

Team Spend Management

For larger teams, you'll want guardrails around spending. Not to restrict usage — to prevent waste and catch configuration problems early.

Cost Management Checklist

  • Set per-developer daily alerts — flag when someone exceeds 2x the team average. This usually indicates a workflow problem, not abuse.
  • Review per-skill costs monthly — if a skill costs more than expected, audit its context loading and thinking requirements.
  • Track cache hit rates — below 60% means your CLAUDE.md is changing too often or conversations are too fragmented.
  • Monitor regeneration rates — high regeneration = insufficient context. Fix the rules, not the budget.
  • Compare across projects — if one project costs 3x more per interaction, its configuration needs work.
For Team Leads: Never restrict AI usage as a cost measure — it sends the message that the tool isn't worth using. Instead, invest in better configuration. A $200 investment in CLAUDE.md and .ctx rules saves $1,000+/month in wasted tokens. Fix the root cause (bad context), not the symptom (high cost).
