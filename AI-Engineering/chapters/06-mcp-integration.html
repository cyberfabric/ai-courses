Chapter 6

MCP Servers: Connecting AI to Your World

Extend AI's reach beyond the codebase — into databases, issue trackers, documentation, and team tools

Beyond Reading and Writing Files

So far, you've built CLAUDE.md (Chapter 2), context files (Chapter 3), skills (Chapter 4), and hooks (Chapter 5). Everything until now has been about AI reading and writing files in your project. But real development involves far more than code files.

You check Jira for requirements. You query the database to understand data shapes. You search Confluence for architecture decisions. You look at GitHub PRs for context on recent changes. Every time AI can't access one of these systems, you become the manual bridge — copying, pasting, and re-explaining.

MCP (Model Context Protocol) eliminates that bridge. MCP servers are lightweight services that give AI structured access to external tools — databases, APIs, documentation systems, project management tools. Instead of you copy-pasting a Jira ticket into the chat, AI queries Jira directly.

The Key Shift: Without MCP, you are the integration layer between AI and your tools. With MCP, AI connects to your tools directly. Your job shifts from "relay information" to "configure access boundaries."

How MCP Works (The 60-Second Version)

MCP follows a client-server model. Claude Code is the client. Each MCP server exposes a set of tools that AI can call. When you configure an MCP server, you're telling Claude Code: "here's a service you can talk to, and here are the tools it offers."

Each MCP server runs as a separate process. It can be a local command (started via npx, uvx, or a binary), or a remote server accessed over HTTP. The server declares what tools it provides, and Claude Code discovers them automatically. You don't need to teach AI the API — the server does that.

Important distinction: This chapter is about configuring MCP servers, not building them. You'll select from existing servers, configure them for your team, and set up security boundaries. For building custom MCP servers, see Automating Your Dev Workflow Chapter 10.

Configuring MCP in settings.json

MCP servers are configured in Claude Code's settings files. There are two levels, and choosing the right one matters:

Project-Level (.claude/settings.json)
  • Shared via version control
  • Same config for every team member
  • Use for: project DB, GitHub, issue tracker
  • No personal credentials in the file
User-Level (~/.claude/settings.json)
  • Personal, not shared
  • Different per developer
  • Use for: personal Slack, email, calendar
  • Safe for personal API tokens

Here's the project-level configuration for a typical development team:

.claude/settings.json — project MCP configuration
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "POSTGRES_CONNECTION": "${DB_READONLY_URL}"
      }
    },
    "linear": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-linear"],
      "env": {
        "LINEAR_API_KEY": "${LINEAR_API_KEY}"
      }
    }
  }
}

Notice the ${VARIABLE} syntax for credentials. The actual tokens live in each developer's environment variables — never in the settings file. This way the configuration is shareable but the secrets stay personal.

For Tech Leads: Add a setup section to your project README or onboarding docs that lists the required environment variables for MCP servers. Something like: "Set GITHUB_TOKEN, DB_READONLY_URL, and LINEAR_API_KEY in your shell profile. See 1Password vault 'Dev Tokens' for values."

Team-Shared MCP Configurations

The whole point of project-level MCP configuration is consistency. When everyone on the team has the same MCP servers, AI conversations are reproducible — the same question gets the same quality of answer regardless of who asks it.

Version-controlled: .claude/settings.json is committed to the repo
Environment variables for secrets: No tokens in the file, only ${VAR} references
Onboarding documented: README lists which env vars to set and where to get tokens
Read-only by default: Database connections use read-only credentials
Tested: Each MCP server verified working before merging the config

Document your MCP configuration in CLAUDE.md so AI knows what tools are available:

CLAUDE.md — MCP documentation section
## Available MCP Servers
The following MCP servers are configured for this project:
- **GitHub**: Query PRs, issues, actions status. Use for understanding
  recent changes and CI results.
- **PostgreSQL**: Read-only access to the development database.
  Use for understanding data shapes and debugging data issues.
  NEVER suggest write queries — the connection is read-only.
- **Linear**: Access project issues and sprint context.
  Use for understanding requirements when working on tickets.

Required environment variables (see 1Password "Dev Tokens" vault):
- GITHUB_TOKEN
- DB_READONLY_URL
- LINEAR_API_KEY

This documentation serves double duty: it tells your team what's available, and it tells AI how to use each server appropriately. The "NEVER suggest write queries" instruction is AI configuration, not just documentation.

Database Access via MCP: The Read-Only Pattern

Database MCP access is the highest-value, highest-risk MCP integration. High value because AI can explore data shapes, write queries, and debug data issues without you copy-pasting schema definitions. High risk because a write query in the wrong context can corrupt data.

The solution is simple: always use read-only credentials.

Database setup — creating a read-only user for MCP
-- PostgreSQL: Create a read-only user for AI access
CREATE USER ai_readonly WITH PASSWORD 'generate-a-strong-password';
GRANT CONNECT ON DATABASE recipes_dev TO ai_readonly;
GRANT USAGE ON SCHEMA public TO ai_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO ai_readonly;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT ON TABLES TO ai_readonly;

-- Connection string for MCP (set as DB_READONLY_URL):
-- postgresql://ai_readonly:password@localhost:5432/recipes_dev

With read-only credentials, even if AI generates a DELETE FROM users query, the database rejects it. This is defense in depth — the CLAUDE.md instruction says "never write," the database user enforces it.

Never use your application's database credentials for MCP. Create a dedicated read-only user. Even for development databases. Even for local databases. It takes 2 minutes to set up and prevents an entire category of accidents.

Internal Tool Integration

Beyond databases, the most valuable MCP integrations connect AI to your team's workflow tools. Here's how to configure the big four:

GitHub: The Development Hub

GitHub MCP is the highest-value integration for most teams. AI can look up PR context, check CI status, search for related issues, and understand recent changes — all without you copying links.

With GitHub MCP configured
"Look at the failing CI check on PR #342 and fix the issue" AI queries GitHub for PR #342, reads the failing check details, identifies the error, and makes the fix — all in one step.
Without MCP, you'd need to open the PR, copy the error log, paste it into the conversation, then ask AI to fix it. Three steps become one.

Issue Trackers: Jira / Linear

When AI can read your issue tracker, it understands requirements without you re-explaining them. "Implement RECIPE-456" becomes a complete instruction because AI can read the ticket description, acceptance criteria, and linked designs.

Confluence / Notion: Documentation Access

Architecture Decision Records (ADRs), design documents, and API specifications often live in documentation tools. MCP gives AI access to these documents, so it can reference the team's actual decisions instead of making assumptions.

Slack: Conversation Context

Slack MCP is useful for finding decisions that were made in conversation but never documented. "What did the team decide about the caching strategy?" can be answered by searching the relevant channel. Use cautiously — Slack search results can be noisy.

Start small: Don't configure all four on day one. Start with GitHub (highest value for most teams), add your issue tracker next, then add others as the need becomes clear. Each MCP server adds complexity — only add what your team will actually use.

MCP Security Boundaries

Every MCP server is an access point. More access means more capability, but also more risk. Design your MCP configuration with the principle of least privilege:

PrincipleImplementationExample
Read-only defaultUse read-only credentials/tokensDB read-only user, GitHub read-only token
Scoped tokensTokens with minimal permissionsGitHub token with repo read, no admin
Development onlyNever connect to productionDB_READONLY_URL points to dev, not prod
Explicit documentationCLAUDE.md states what's allowed"NEVER suggest write queries"

Token Scoping Examples

GitHub token scoping — minimum permissions for MCP
# When creating a GitHub Personal Access Token for MCP:
#
# Required permissions (read-only):
#   - repo (read)          — access code, PRs, issues
#   - actions (read)       — check CI status
#
# DO NOT grant:
#   - repo (write)         — no push, no PR merge
#   - admin                — no settings changes
#   - delete_repo          — obviously not
#
# Create at: github.com/settings/tokens
# Set expiration: 90 days (rotate quarterly)
For Tech Leads: Add MCP security review to your quarterly security checklist. Verify: Are all tokens still scoped correctly? Have any been over-provisioned? Are database connections still pointing to dev, not prod? Are there any MCP servers configured that nobody uses anymore? Remove unused servers — they're attack surface for no benefit.

When NOT to Use MCP

MCP is powerful, but it's not always the right tool. Here are situations where MCP adds complexity without sufficient value:

Don't Use MCP For
  • Reading files already in your project
  • Running CLI tools AI can invoke via Bash
  • One-off lookups you do once a month
  • Tools that require interactive auth flows
Do Use MCP For
  • External data sources (databases, APIs)
  • Tools you query multiple times per day
  • Shared team tools (issue tracker, CI)
  • Systems with stable, well-documented APIs

The "manual bridge test" is your decision framework: if you find yourself copy-pasting from a tool into AI conversations more than twice a day, that tool is a strong MCP candidate. If you do it once a week, manual copy-paste is fine.

MCP is not a substitute for good context engineering. If your project needs AI to understand your architecture, put it in CLAUDE.md and .ctx files (Chapter 3). MCP is for dynamic, external data — not static project knowledge. A .ctx file is faster, cheaper, and always available. MCP requires a running server, network access, and API tokens.

Setting Up Your First 3 MCP Servers

Here's the practical, step-by-step guide for the three most valuable MCP servers. Start with these, then add more as your team's needs evolve.

Server 1: GitHub

  1. Create a Personal Access Token at github.com/settings/tokens with repo:read and actions:read permissions
  2. Add to your shell profile: export GITHUB_TOKEN="ghp_your_token_here"
  3. Add the server config to .claude/settings.json (see configuration example above)
  4. Test: ask AI "What's the status of the latest CI run on main?"

Server 2: PostgreSQL (Read-Only)

  1. Create a read-only database user (see SQL script earlier in this chapter)
  2. Add to your shell profile: export DB_READONLY_URL="postgresql://ai_readonly:password@localhost:5432/mydb"
  3. Add the server config to .claude/settings.json
  4. Test: ask AI "What tables exist in the database and what are their schemas?"

Server 3: Your Issue Tracker (Linear/Jira)

  1. Create an API key with read-only permissions in your issue tracker
  2. Add to your shell profile: export LINEAR_API_KEY="lin_api_your_key_here"
  3. Add the server config to .claude/settings.json
  4. Test: ask AI "What are the open issues assigned to me this sprint?"
Verification pattern: After configuring each server, run a simple query that proves the connection works. Don't move on to the next server until the current one is verified. A misconfigured MCP server that fails silently is worse than no server at all — AI will try to answer without the data and may hallucinate instead.
