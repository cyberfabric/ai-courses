Your AI-Ready Playbook

Everything from Chapters 1–11, consolidated into one actionable reference

This is the capstone. Eleven chapters of configuration wisdom, distilled into a playbook you can reference every week. No new concepts here — just the complete checklist, the maintenance schedule, and the 90-day plan that takes any project from zero to AI-native.

Bookmark this chapter. Print it. Tape it to your monitor. It's the one page you'll come back to long after you've forgotten the details of .claude/rules/ and hook matrices.

The Complete Configuration Checklist

Every item below maps to a specific chapter. Check off what your project already has, then work through the gaps in priority order.

Pillar 1: CLAUDE.md & Project Identity (Ch 2)

  • CLAUDE.md exists at the project root (or ./.claude/CLAUDE.md)
  • Under 200 lines (the official adherence target)
  • Contains 5+ specific NEVER rules (not vague guidelines)
  • Documents module structure and dependency directions
  • Includes naming conventions for all major types
  • Lists build commands (./gradlew, test commands, lint)
  • Has testing conventions (fakes vs mocks, naming patterns)
  • Organized with stable content first for caching (Ch 10)
  • Uses @import for shared sections (e.g., @docs/git-instructions.md) instead of inlining everything
  • @AGENTS.md at the top if the repo also serves Codex/Cursor/Aider
  • CLAUDE.local.md gitignored for personal sandbox URLs and throwaway notes
  • claudeMdExcludes in .claude/settings.local.json if you're inside a noisy monorepo
  • Model selection guidelines (Opus/Sonnet/Haiku by task type)
  • Thinking guidelines (when to use extended thinking, when to skip)
  • Monthly review cadence scheduled
  • Habit: run /memory after any CLAUDE.md change to confirm what loaded

Pillar 2: Context Engineering (Ch 3)

  • ARCHITECTURE.md exists (1-page structure overview)
  • .claude/rules/ directory with path-scoped rules for presentation, domain, data, testing (paths: frontmatter, ~25 lines body each)
  • Review checklists for top 5 architecture areas (verbose, for review)
  • Critical NEVER rules in project-root CLAUDE.md so they survive /compact; deep rules path-scoped in .claude/rules/
  • No context pollution — every loaded file serves a purpose
  • Total always-loaded context under 5K tokens (path-scoped rules don't count — they cost zero until matched)
  • Auto memory reviewed: skim ~/.claude/projects/<project>/memory/MEMORY.md for stale notes after major refactors
  • Habit: run /context mid-session and confirm utilization is below 80% before starting heavy work

Pillar 3: Skills & Hooks (Ch 4, 5)

  • /scaffold-feature skill (generates screen + ViewModel + DI + test)
  • /run-review skill (architecture compliance check)
  • /generate-tests skill (follows testing-ctx.md patterns)
  • At least 2 plan-mode skills for high-risk operations (Ch 9)
  • PreToolUse hook: dangerous-command blocker
  • PreToolUse hook: sensitive-file protector
  • PostToolUse hook: auto-formatter on file write
  • PostToolUse hook: module test runner on file write
  • Skills documented with headers and $ARGUMENTS
  • Skill quality checklist enforced on PR

Pillar 4: Architecture Rules (Ch 7)

  • Rules use MUST/SHOULD/PREFER severity levels
  • Rules have IDs (P1, D1, T1, etc.)
  • At least 5 of 10 topic areas covered
  • Dual-format: review checklists + path-scoped .claude/rules/ files with paths: frontmatter
  • /check-architecture skill bridges the Read-only-trigger limitation by reading rules explicitly before checking diffs
  • Rules updated during every migration
  • "Update architecture rules" in every migration checklist

Pillar 5: Team Configuration (Ch 8)

  • AI_ONBOARDING_CHECKLIST.md committed to repo
  • 30-minute onboarding session format documented
  • Shared MCP configuration with ${ENV_VAR} pattern
  • API tokens in shell profiles, never in repo
  • Adoption metrics tracked (skill usage, compliance rate)
  • 50%+ team members have contributed a skill or rule
  • Cost monitoring in place (per-dev, per-skill alerts)

# Monthly Maintenance Review

Run the monthly AI configuration maintenance review.

## Check CLAUDE.md
1. Read CLAUDE.md and compare against recent git log (last 30 days)
2. Flag any rules that contradict recent merged changes
3. Flag any new patterns in PRs not yet captured in rules

## Check .claude/rules/
4. For each rule file in .claude/rules/, verify rules match current code patterns
5. Flag stale rules (reference patterns no longer in codebase)
6. Verify paths: frontmatter still matches the actual layout (e.g., directories haven't been renamed)

## Check skills
7. List all skills in .claude/commands/
8. Flag skills not invoked in the last 30 days (candidate for removal)
9. Flag skills without documentation headers

## Report
Generate a maintenance report in docs/maintenance-review-[date].md:
- Rules to update
- Rules to add
- Skills to review
- Recommended actions
For Team Leads: Add "AI config maintenance" as a recurring calendar event. Monthly reviews take 30 minutes and prevent the slow drift that makes configuration stale. The most dangerous state isn't "no configuration" — it's outdated configuration that gives AI wrong instructions.

Staying Current

Claude Code evolves. New features appear, capabilities expand, and best practices shift. Here's how to stay current without chasing every update:

When Claude Code Updates

  • Read the changelog — focus on new hook events, skill features, and context handling changes
  • Check your hooks — new hook events may enable better safety gates
  • Test your skills — behavior changes might affect skill outputs
  • Review model updates — new models may change your model selection guidelines
  • Update CLAUDE.md — if new features change best practices, update your guidelines
Don't chase every feature. A new capability is only worth adopting if it improves your specific workflow. Evaluate against your audit scores: does this feature improve your lowest-scoring pillar? If not, note it and move on.
Track updates: Follow the Claude Code changelog and release notes. When a major update ships, schedule 15 minutes to assess impact on your configuration. Most updates need no changes — but the ones that do matter a lot.

Your AI Engineering Manifesto

You've spent 12 chapters learning the craft of AI-ready engineering. Here's the manifesto — the principles that tie everything together:

The AI-Ready Engineer's Manifesto

  1. Configuration over conversation. Every rule in CLAUDE.md prevents a hundred correction prompts.
  2. Prevention over review. A .ctx file that guides generation beats a code review that catches mistakes.
  3. Shared over personal. Configuration in the repo benefits everyone. Configuration on your laptop benefits only you.
  4. Specific over vague. "NEVER call repositories from composables" beats "follow Clean Architecture."
  5. Maintained over created. Stale configuration is worse than no configuration. Monthly reviews are non-negotiable.
  6. Balanced over deep. Score 3 across all 5 pillars before scoring 4 on any single one.
  7. Evidence over opinion. Track metrics. Run audits. Let the numbers guide investment.
  8. Invest in context, not prompts. Rich cached context produces correct output. No prompt engineering compensates for missing rules.
