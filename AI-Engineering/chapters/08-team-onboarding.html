Chapter 8

Team Onboarding: From Solo to Squad

The playbook for taking a team from zero AI configuration to full adoption

The Human Challenge

Chapters 2 through 7 built the complete technical infrastructure: CLAUDE.md, context engineering, skills, hooks, MCP servers, and architecture rules. If you've been doing this solo, your project is AI-ready. But a project isn't a team.

The best configuration in the world is worthless if your team doesn't use it. And getting people to change their workflows is harder than writing any CLAUDE.md file. You'll face skeptics who think AI is overhyped, enthusiasts who want to use it for everything, and pragmatists who just want to know what to do differently on Monday morning.

This chapter is the operational onboarding guide — the concrete playbook for taking your team from "what's Claude Code?" to "I can't work without my skills."

The Key Insight: Onboarding is not a one-time event. It's a progression: Day 1 is installation. Week 1 is using existing skills. Month 1 is creating new skills. Quarter 1 is contributing to the configuration itself. Meet people where they are.

The Configuration Maturity Model

Before you onboard anyone, understand where your project sits today. The maturity model gives you a shared vocabulary for talking about AI readiness:

LevelNameWhat It Looks Like
0 No Config No CLAUDE.md, no skills, no rules. Developers use AI with default settings. Output quality is random.
1 Personal One developer has a CLAUDE.md and a few skills. Not shared with the team. Knowledge lives in one person's head.
2 Shared Basics CLAUDE.md is committed. 3-5 skills in the repo. Basic rules exist. Some team members use them, others don't.
3 Team Standard Full 5-pillar configuration. All team members using skills and hooks. Architecture rules enforced. Regular maintenance cycle.
4 AI-Native AI configuration is part of the definition of done. New features include updated rules and skills. Team creates and evolves skills independently. Configuration quality is a review criterion.

Most teams start at Level 0 or 1. The goal of this chapter is to get you to Level 2-3, with a clear path to Level 4.

Don't skip levels. A team at Level 0 can't jump to Level 4 in a sprint. Each level builds on the previous one. Trying to deploy the full configuration stack to a team that hasn't seen Claude Code before will overwhelm them and create resistance.

The Team Onboarding Playbook

Here's the timeline that works. It's structured as Day 1 / Week 1 / Month 1 because adoption happens in stages, not all at once.

Day 1: Install and First Win

The goal of Day 1 is simple: every team member has Claude Code installed and has completed one task using the project's skills. Nothing more.

Install Claude Code — walk through setup together (5 minutes)
Clone the repo — project configuration loads automatically
Run one skill — e.g., /scaffold-feature test-feature and watch it work
Delete the test output — the point was seeing it work, not keeping the code
Set up MCP environment variables — follow the README instructions

The first win matters more than you think. When a developer sees /scaffold-feature generate a complete, architecturally-correct module in 30 seconds, the skepticism drops immediately. Don't explain the theory first — show the result.

Week 1: Use Existing Skills Daily

The goal of Week 1 is habit formation. Team members should use at least one skill per day as part of their normal workflow:

Use /generate-tests on a file they're working on
Use /run-review before submitting a PR
Use /scaffold-feature if they start a new feature
Read the CLAUDE.md — understand the project's AI configuration
Ask AI a question using MCP — e.g., "What issues are assigned to me?"

Month 1: Customize and Create

By Month 1, team members should transition from consumers to contributors:

Create a personal skill — automate something they do repeatedly
Suggest a CLAUDE.md improvement — add a rule that would have caught a recent mistake
Add a .ctx rule — document a pattern they noticed AI getting wrong
Review someone else's skill PR — participate in skill quality control
For Tech Leads: The Month 1 transition is where adoption either sticks or fades. If team members are still just running existing skills after a month, they're consumers, not contributors. Push them to create one skill and one .ctx rule. Once they've created something, they're invested.

The 30-Minute Onboarding Session

This is the session you run on Day 1. Keep it tight — 30 minutes, not an hour. Developers lose interest in demos that run long.

What to Cover (10 Minutes)

  1. Demo /scaffold-feature — show a complete module generated in 30 seconds
  2. Demo /run-review — show an architecture review with rule IDs and findings
  3. Show the CLAUDE.md — "This is the brain. Everything AI knows about our project is here."

What to Skip

  • Don't explain how LLMs work — irrelevant for adoption
  • Don't explain the full 5-pillar framework — too abstract for day 1
  • Don't show hooks or MCP internals — they work invisibly, that's the point
  • Don't compare Claude to GPT or other tools — not the time for debate
The biggest onboarding mistake: Spending 25 minutes presenting slides and 5 minutes doing hands-on. Flip it. People learn by doing, not by watching. The demo should be fast enough that people are itching to try it themselves.

Dealing with Skeptics

Every team has at least one person who thinks AI-assisted development is overhyped. They're not wrong to be cautious — bad AI output does exist. The way to convert skeptics is not with arguments but with evidence.

The Evidence-Based Approach

30s
Time to scaffold a feature
vs 20 min manually
5 min
Architecture review via skill
vs 45 min manual review
0
Architecture violations with rules
vs 3-4 per PR without

Don't argue about AI in the abstract. Instead, pick a task the skeptic does regularly, do it with AI in front of them, and compare the time and quality. Concrete evidence beats philosophical debate every time.

Converting a skeptic — the paired task
"Let's both write tests for RecipesViewModel. You write them manually, I'll use /generate-tests. Let's compare output quality and time when we're done."
The skeptic sees the AI-generated tests follow all project conventions (fakes, naming, edge cases) in a fraction of the time. The evidence is undeniable.
Respect the skeptic's expertise. Many skeptics are your best developers. Their concern is quality, not laziness. Show them that the configuration (CLAUDE.md, rules, hooks) addresses their quality concerns directly. Hooks catch the very issues they'd catch in review — automatically.

Dealing with Enthusiasts

The opposite problem: developers who want to use AI for everything. They'll try to generate entire features without reading the output, skip tests because "AI wrote it so it's probably fine," and create 50 skills in a week without documentation.

The Guardrails for Enthusiasts

CLAUDE.md — AI usage guidelines section
## AI Usage Guidelines
- ALWAYS review AI-generated code before committing — treat it like a PR from a junior developer
- NEVER skip tests because AI generated the code — tests verify correctness, not authorship
- NEVER commit AI-generated code without running it locally first
- Skills must pass the quality checklist (Chapter 4) before merging
- New rules require at least one real example of the violation they prevent
- Use /run-review before every PR — no exceptions

The key principle: AI augments your judgment, it doesn't replace it. Enthusiasts need this boundary explicitly stated, preferably in CLAUDE.md where AI will also read it and self-moderate.

Enthusiast Without Guardrails
  • Generates entire features without review
  • Creates 20 untested skills in a week
  • Skips manual testing: "AI wrote tests"
  • Quality drops, team loses trust in AI
Enthusiast With Guardrails
  • Generates features, reviews output critically
  • Creates skills with documentation and testing
  • Uses AI-generated tests AND manual verification
  • Quality stays high, AI becomes trusted tool

Shared vs Personal Configuration

Not everything belongs in the repo. Clear boundaries prevent configuration conflicts:

Config TypeLocationShared?Examples
Project rulesCLAUDE.md, .ctx filesYes — in repoArchitecture rules, forbidden patterns, naming conventions
Team skills.claude/commands/Yes — in repo/scaffold-feature, /run-review, /generate-tests
Team hooks.claude/hooks/Yes — in repoDangerous command blocker, sensitive file protector
Project MCP.claude/settings.jsonYes — in repoGitHub, database, issue tracker (with ${ENV_VAR})
Personal MCP~/.claude/settings.jsonNo — local onlyPersonal Slack, email, calendar integrations
Personal skills~/.claude/commands/No — local onlyPersonal workflow shortcuts, project-switching helpers
API tokensShell profile / env varsNo — never in repoGITHUB_TOKEN, DB_READONLY_URL, LINEAR_API_KEY
For Tech Leads: A common conflict arises when a developer puts personal preferences into the shared CLAUDE.md. "Use verbose variable names" is a personal preference. "Use StateFlow not LiveData" is an architecture rule. CLAUDE.md should only contain team-agreed rules, not individual style preferences. Settle disagreements before they reach the file.

Measuring Adoption

Usage alone doesn't tell you if adoption is working. A developer who runs /scaffold-feature once and then goes back to manual coding isn't adopted. Measure configuration quality, not just activity:

The Adoption Dashboard

Skill usage frequency: How often are team skills invoked per week? Increasing trend = good.
Rule compliance rate: Run /check-architecture on PRs. What % of rules pass? Target: 90%+.
Skill contribution rate: How many team members have created or modified a skill? Target: 50%+ by Month 2.
CLAUDE.md update frequency: Is the team updating CLAUDE.md? Stale = adoption is superficial.
Hook trigger rate: How often do hooks block dangerous actions? Some blocking is healthy. Zero might mean hooks aren't configured.
.claude/commands/adoption-report.md
Generate an AI adoption report for the current project.

## Steps
1. Count skill files in .claude/commands/: `ls .claude/commands/*.md | wc -l`
2. Count hook scripts in .claude/hooks/: `ls .claude/hooks/*.sh | wc -l`
3. Count .ctx rule files: `ls docs/architecture/*-ctx.md | wc -l`
4. Check CLAUDE.md last modified: `git log -1 --format="%ar" -- CLAUDE.md`
5. Count recent skill usage in git log: `git log --oneline --since="1 week ago" | grep -i "skill\|generate\|scaffold" | wc -l`
6. Report: number of skills, hooks, rules, last CLAUDE.md update,
   and estimated team maturity level (0-4) based on findings.

The Onboarding Checklist Template

Give every new team member this checklist. It replaces the "ask the senior dev" approach with a self-service guide:

docs/AI_ONBOARDING_CHECKLIST.md
# AI-Ready Onboarding Checklist

## Day 1: Setup
- [ ] Install Claude Code (see https://claude.ai/code)
- [ ] Clone the project repo (config loads automatically from .claude/)
- [ ] Set environment variables (see README → "MCP Setup"):
      - GITHUB_TOKEN
      - DB_READONLY_URL
      - LINEAR_API_KEY
- [ ] Run `/scaffold-feature test-onboarding` — verify it generates correctly
- [ ] Delete the test output: `rm -rf feature/test-onboarding`
- [ ] Read CLAUDE.md — understand the project's AI rules

## Week 1: Daily Usage
- [ ] Use `/generate-tests` on a file you're working on
- [ ] Use `/run-review` before submitting your first PR
- [ ] Ask AI to query the database via MCP (e.g., "Show me the users table schema")
- [ ] Ask AI to look up an issue in Linear/Jira

## Month 1: Contribution
- [ ] Create one personal skill for a task you repeat often
- [ ] Propose one new rule for CLAUDE.md (based on a mistake you observed)
- [ ] Review one skill PR from a teammate
- [ ] Update your AI_READINESS_BASELINE.md with current scores

## Verification
- [ ] All MCP servers responding (test each one)
- [ ] Hooks active (try a blocked command to verify)
- [ ] Skills accessible (run `/help` to see available skills)
Commit this checklist to the repo. It's a living document that evolves with your configuration. When you add a new MCP server, add it to the checklist. When you create a new must-use skill, add it to Week 1. The checklist is the onboarding source of truth, not a wiki page that gets outdated.

Your Chapter Exercise

Build the onboarding infrastructure that makes AI adoption self-service for your team.

  1. Assess your current maturity level (0-4) using the model from this chapter.

Commit everything and update your AI_READINESS_BASELINE.md with your Pillar 5 score.
