Chapter 2

CLAUDE.md: Your Project's AI Brain

Production-grade CLAUDE.md files that actually constrain AI behavior

Beyond the Basics

In Chapter 1, you saw a CLAUDE.md excerpt from the recipes app and learned that encoding specific decisions beats vague guidelines. Now we go deep. This chapter is about writing CLAUDE.md files that make AI behave like a senior developer who's been on your team for a year.

If you've read basic CLAUDE.md tutorials, you know how to add a project description and a few rules. That's Level 1. We're going to Level 4 — multi-level hierarchies, forbidden-pattern encoding, the "negative space" technique, and a maintenance process that keeps your CLAUDE.md accurate as your codebase evolves.

The Goal: By the end of this chapter, you'll have a production-grade CLAUDE.md committed to your repo that covers architecture, naming, module boundaries, forbidden patterns, and workflow preferences — and you'll know when to split it across directories.

The CLAUDE.md Hierarchy

CLAUDE.md isn't a single file — it's a four-tier hierarchy. Claude Code walks up the directory tree from the working directory, concatenates every CLAUDE.md and CLAUDE.local.md it finds, and layers them so more specific instructions take precedence. Understanding the four tiers is the difference between a flat config and a scalable one.

Managed Policy

→
User
~/.claude/CLAUDE.md
→
Project
./CLAUDE.md or ./.claude/CLAUDE.md
→
Local
./CLAUDE.local.md (gitignored)
Tier Location What Goes Here Checked Into Git?
Managed Policy /Library/Application Support/ClaudeCode/CLAUDE.md (macOS)
/etc/claude-code/CLAUDE.md (Linux/WSL)
C:\Program Files\ClaudeCode\CLAUDE.md (Windows)		 Org-wide standards pushed by IT/DevOps: security policies, compliance, banned libraries No (deployed via MDM/Group Policy/Ansible)
Project ./CLAUDE.md or ./.claude/CLAUDE.md Architecture, naming conventions, module structure, forbidden patterns, workflow — the file your team writes together Yes (shared with team)
User ~/.claude/CLAUDE.md Personal preferences across every project: response format, tooling shortcuts, prompt habits No (personal)
Local ./CLAUDE.local.md Project-specific personal notes: sandbox URLs, preferred test data, throwaway todos No (add to .gitignore)

Subdirectory CLAUDE.md files load on demand — they are not a fifth tier of the launch hierarchy. A ./feature/billing/CLAUDE.md only enters the context window when Claude reads a file inside feature/billing/. That's exactly what you want for module-specific rules: zero token cost until they're relevant.

Rule of Thumb: Put 80% of your rules in the project CLAUDE.md (root or .claude/ — both work, both load at launch). Use module-level CLAUDE.md only when a directory has genuinely different conventions (legacy module, build-tools, generated code).
Hidden notes for humans: Block-level HTML comments in CLAUDE.md (<!-- maintainer notes -->) are stripped before the file is injected into Claude's context. Use them to document why a rule exists for future maintainers without spending context tokens. Comments inside fenced code blocks are preserved.

When to Use Module-Level CLAUDE.md

Don't split prematurely. Directory-level files add complexity and can cause confusion when rules conflict. Use them only for these situations:

  • Legacy modules — a module that hasn't been migrated yet and uses different patterns
  • Generated code directories — "NEVER modify files in this directory manually"
  • Test directories — different naming conventions, allowed patterns (mocks, fakes)
  • Build/config directories — Gradle plugins, CI scripts with their own conventions
feature/legacy-auth/CLAUDE.md
# Legacy Auth Module — CLAUDE.md Override

## IMPORTANT: This module uses legacy patterns
- This module still uses Activities + View Binding (NOT Compose)
- ViewModels use LiveData (NOT StateFlow) — migration planned for Q3
- Do NOT refactor to Compose unless explicitly asked
- All other root CLAUDE.md rules still apply

## Migration Notes
- When migration happens, delete this file and follow root conventions
- See JIRA-4521 for the migration plan

Sharing Instructions With @imports and AGENTS.md

Once your CLAUDE.md grows past a couple of topics, you'll want to split it. Claude Code supports inline imports, plus a clean compatibility pattern for repos that also serve other AI coding tools.

The @path/to/file Import Syntax

Anywhere in CLAUDE.md, write @ followed by a relative or absolute path. Claude expands the import at launch and concatenates the imported file's contents alongside the parent. Imports can chain — an imported file can @import others — up to five hops deep. Relative paths resolve against the file containing the import, not the working directory.

CLAUDE.md — using @imports
# Recipes App

See @README.md for project overview and @package.json for available scripts.

## Workflow
- git workflow @docs/git-instructions.md
- review checklist @docs/review-checklist.md

## Personal preferences (gitignored, optional)
- @~/.claude/my-recipes-overrides.md

The first time Claude Code encounters external imports in a project, it shows an approval dialog listing the files. If you decline, imports stay disabled and the dialog won't appear again. Use @~/.claude/... for personal instructions you want to share across worktrees of the same repo — a gitignored CLAUDE.local.md only exists in one worktree.

Imports don't save context: imported files load at launch alongside the parent CLAUDE.md. Use @imports for organization, not for token savings. If you actually need lazy loading, move topic-specific instructions to .claude/rules/ with paths: frontmatter (Chapter 7).

AGENTS.md Compatibility

Claude Code reads CLAUDE.md, not AGENTS.md. If your repo already maintains an AGENTS.md for Codex, Cursor, Aider, or other tools, don't duplicate the content — create a CLAUDE.md that imports it and append Claude-specific overrides below:

CLAUDE.md — AGENTS.md compatibility pattern
@AGENTS.md

## Claude Code

- Use plan mode for changes under :feature:billing
- NEVER use field injection — constructor @Inject only
- When generating tests, use the fakes in :core:testing (not Mockito)

This gives every coding tool the same baseline rules while letting you encode Claude-specific behavior (plan mode, hooks, skills) without polluting the shared file.

Excluding Noisy Ancestors in Monorepos

In a large monorepo, ancestor CLAUDE.md files from other teams can leak into your context window and contradict your local rules. The claudeMdExcludes setting takes glob patterns and skips matching files entirely. Configure it locally so the exclusion stays on your machine:

.claude/settings.local.json
{
  "claudeMdExcludes": [
    "**/monorepo/CLAUDE.md",
    "/home/me/monorepo/other-team/.claude/rules/**"
  ]
}

Patterns match against absolute paths. You can set claudeMdExcludes at any settings layer (user, project, local, managed) and arrays merge across layers. Managed-policy CLAUDE.md cannot be excluded — that's intentional, so org-wide rules always apply.

Writing Rules That Actually Constrain

Here's the most common mistake in CLAUDE.md files: rules that are too vague to be actionable. AI can interpret vague rules in many ways — and it will choose the wrong interpretation approximately half the time.

Vague Rules (Don't Do This)
  • "Use Clean Architecture"
  • "Follow best practices"
  • "Keep code clean"
  • "Use proper dependency injection"
  • "Write good tests"
Specific Rules (Do This)
  • "presentation → domain → data. NEVER reverse."
  • "Max 3 public functions per ViewModel"
  • "All Composables take a Modifier parameter"
  • "Use constructor injection via @Inject. NEVER use field injection."
  • "Every UseCase has a matching UseCaseTest with at least 3 scenarios"

The test for a good rule: could AI violate it without ambiguity? If the rule is "use proper DI," AI can't tell if it violated it or not — the definition of "proper" is subjective. But if the rule is "NEVER use field injection, always constructor injection via @Inject" — that's binary. Either the code uses constructor injection or it doesn't.

The Specificity Ladder

When writing rules, climb the specificity ladder:

Category
"Use DI"
→
Tool
"Use Hilt"
→
Pattern
"Constructor injection"
→
Constraint
"NEVER field inject"

Each step up the ladder reduces AI misinterpretation. The sweet spot is usually "Pattern" or "Constraint" level. Going all the way to "Constraint" for every rule creates an overwhelming CLAUDE.md, so reserve it for patterns where AI consistently gets it wrong.

The "Negative Space" Technique

Here's a counterintuitive truth about AI configuration: telling AI what NOT to do is often more effective than telling it what to do.

Why? Because AI already knows thousands of valid patterns. When you say "use Clean Architecture," it picks from dozens of valid interpretations. But when you say "NEVER call a Repository from a ViewModel" — that eliminates exactly one wrong path. Stack enough forbidden patterns and AI is funneled toward your exact architecture through elimination.

The Negative Space Approach
## Forbidden Patterns - NEVER call Repository directly from ViewModel (always go through UseCase) - NEVER put business logic in Composables (extract to ViewModel) - NEVER use mutableStateOf in ViewModel (use MutableStateFlow) - NEVER create a UseCase that returns a DAO entity (always map to domain model) - NEVER add a dependency from :feature:A to :feature:B (features are isolated)
Each "NEVER" eliminates one wrong path. Together, they define your architecture through constraint — like a sculptor removing marble to reveal the statue.

Think of it as sculpting. You don't describe the statue — you describe which marble to remove. What remains is your architecture.

Warning: Don't go overboard. 10-15 forbidden patterns is the sweet spot. More than 20 and AI starts getting confused by conflicting constraints. If you find yourself writing 30+ NEVER rules, you probably need to split them into .claude/rules/ files instead — see Chapter 7 for the path-scoped rules pattern.
The peer concept: .claude/rules/ sits next to .claude/skills/ in your project. Rules load at session start (or when matching files are read) and shape how Claude thinks about the codebase. Skills, in .claude/skills/<name>/SKILL.md, are on-demand workflows Claude invokes when relevant. You'll meet skills in Chapter 4 — for now, just know that the .claude/ directory has these two peers, plus settings and hooks, all shipped together as your project's AI configuration.

Encoding Architecture Decisions

Your CLAUDE.md should encode the decisions that make your project unique. Not textbook knowledge — your specific choices. Here are the categories every project needs:

1. Module Boundaries

CLAUDE.md — Module Boundaries Section
## Module Structure
- :app — single-Activity host, navigation graph, Hilt setup
- :feature:* — one module per user-facing feature
- :core:ui — shared Composables, theme, design system tokens
- :core:data — Room database, Retrofit services, shared DTOs
- :core:domain — domain models, base UseCase interface, shared business logic
- :core:testing — shared test utilities, fakes, test fixtures

## Module Rules
- Features NEVER depend on other features
- Features depend on :core:domain and :core:ui only
- :core:domain has ZERO Android dependencies (pure Kotlin)
- Only :app knows about all features (for navigation graph)
- New features MUST be in their own module from day one

2. Naming Conventions

Name every file type AI will create. Leave nothing to interpretation:

CLAUDE.md — Naming Conventions Section
## Naming Conventions
### Files
- Screens: {Feature}Screen.kt (e.g., RecipeDetailScreen.kt)
- ViewModels: {Feature}ViewModel.kt
- UI State: {Feature}UiState.kt (data class, NEVER sealed class for state)
- UI Events: {Feature}Event.kt (sealed interface)
- UseCases: {Verb}{Noun}UseCase.kt (e.g., GetRecipesUseCase.kt)
- Repositories: {Feature}Repository.kt (interface) + {Feature}RepositoryImpl.kt
- DAOs: {Entity}Dao.kt
- Entities: {Name}Entity.kt (Room) vs {Name}.kt (domain model)
- Mappers: {Source}To{Target}Mapper.kt

### Packages
- feature.{name}.presentation — screens, viewmodels, components
- feature.{name}.domain — usecases, models
- feature.{name}.data — repository impls, data sources, mappers

3. Workflow Preferences

Don't forget to encode how AI should work, not just what it should produce:

CLAUDE.md — Workflow Section
## Workflow Preferences
- Run `./gradlew :feature:{name}:testDebugUnitTest` after modifying any feature module
- Run `./gradlew detekt` before considering any task complete
- When creating a new feature, use `/scaffold-feature` skill first
- When modifying data layer, always check if migration is needed
- Prefer extending existing Composables over creating new ones
- When in doubt about architecture, reference the :feature:recipes module as the gold standard

Real-World Teardown: The Recipes App CLAUDE.md

Let's walk through a complete, production-grade CLAUDE.md section by section. This is what a Level 4 CLAUDE.md looks like — not because it's long, but because it's specific.

recipes-app/CLAUDE.md (full production version)
# Recipes App

Android app for discovering, saving, and cooking recipes.
Multi-module Clean Architecture with Compose, Hilt, Room, Retrofit.

## Architecture
- Layers: presentation → domain → data. NEVER reverse dependency direction.
- Presentation: Compose + MVI. ViewModel exposes StateFlow<UiState>.
- Domain: Pure Kotlin. No Android imports. UseCases are single-responsibility.
- Data: Room for local, Retrofit for remote. Repository pattern.

## Module Structure
- :app — host Activity, NavGraph, Hilt entry point
- :feature:recipes, :feature:nutrition, :feature:shopping — isolated features
- :core:ui, :core:data, :core:domain, :core:testing

## Forbidden Patterns
- NEVER add feature-to-feature dependencies
- NEVER call Repository from ViewModel (always UseCase)
- NEVER use LiveData (use StateFlow)
- NEVER create Activities (single-Activity app)
- NEVER use field injection (constructor @Inject only)
- NEVER put business logic in Composables
- NEVER create God ViewModels (max 3 public functions)
- NEVER return Room entities from domain layer (map to domain models)
- NEVER use GlobalScope or Dispatchers.IO directly (inject dispatchers)
- NEVER write UI tests without test tags

## Naming
- Screens: {Feature}Screen.kt | ViewModels: {Feature}ViewModel.kt
- UseCases: {Verb}{Noun}UseCase.kt | Repos: {Feature}RepositoryImpl.kt
- State: {Feature}UiState.kt (data class) | Events: {Feature}Event.kt (sealed interface)

## Testing
- Every UseCase → UseCaseTest with success, error, and edge case
- Every ViewModel → ViewModelTest with state transitions
- Fakes over mocks. Shared fakes live in :core:testing.
- Test names: `should {expected} when {condition}`

## Workflow
- Run tests after any feature module change
- Run detekt before marking tasks complete
- Reference :feature:recipes as the gold-standard module
- Use /scaffold-feature for new features, /run-review for code review

## Imports
@AGENTS.md
@docs/architecture/presentation-layer.md
@docs/architecture/data-layer.md

## Context References
- Review checklists live in docs/architecture/*-checklist.md (load via @import only when reviewing that layer)

Count the lines: about 45 lines of actual configuration. That's it. Forty-five lines that prevent hundreds of architecture violations per month. Notice the patterns:

  • Description first — one sentence, so AI knows what the project is
  • Architecture is directional — arrows show allowed dependency flow
  • Forbidden patterns are specific — each one names exactly what's banned
  • Naming is templated — AI can generate correct filenames without asking
  • Testing has standards — AI knows what "good enough" looks like
  • Workflow is explicit — AI knows to run tests and linting automatically
  • Context files are referenced — AI knows where to find deeper rules

The CLAUDE.md Anatomy: Section by Section

Every production CLAUDE.md should have these sections, in this order:

1
Project Identity
What is this? One paragraph max.
2
Architecture
Layers, patterns, direction of dependencies
3
Module Structure
Every module, its purpose, its boundaries
4
Forbidden Patterns
10-15 NEVER rules that define boundaries
5
Naming & Testing
Templates for every file type + test standards
6
Workflow & References
What to run, where to look for more context

This order matters. AI processes CLAUDE.md top-to-bottom, and the most important context should come first. Project identity and architecture set the frame. Forbidden patterns narrow the solution space. Naming and testing handle the details. Workflow and references are the finishing touches.

For Tech Leads: The sections map to review concerns. When reviewing a PR from an AI-assisted developer, check: Did it follow architecture? (Section 2) Did it avoid forbidden patterns? (Section 4) Did it use correct naming? (Section 5) Did it run the right checks? (Section 6) If the answer is "no" to any of these, the fix isn't in the PR — it's in the CLAUDE.md.

Warning: Don't fill in the template with generic best practices. The whole point is to encode your decisions. If you find yourself writing "follow SOLID principles" — stop. That's not specific enough. Write what SOLID means in your project: "Single-responsibility: each UseCase does one thing. Open/closed: extend via interfaces, never modify existing implementations."

Keeping CLAUDE.md in Sync

A CLAUDE.md that was accurate six months ago is worse than no CLAUDE.md at all. Outdated rules cause AI to produce confidently wrong code. Here's the maintenance process:

The Monthly Review Checklist

Are all modules listed still accurate? (Any renamed, removed, or added?)
Do the forbidden patterns still apply? (Has the team adopted new patterns?)
Are naming conventions still current? (Any new file types or changed templates?)
Do workflow commands still work? (Any changed build scripts or CI paths?)
Are context file references valid? (Do referenced .claude/rules/*.md files still exist?)
Any new recurring AI mistakes that need a new rule?
Practical Process: Add "Review CLAUDE.md" to your sprint retrospective template. It takes 5 minutes. One team member reads through the CLAUDE.md and asks: "Is this still true?" If anything changed, update it on the spot. The PR for the CLAUDE.md change should be tiny and fast to merge.

Trigger-Based Updates

Don't wait for the monthly review if any of these happen:

  • New module added — add it to Module Structure
  • Pattern migration — update Forbidden Patterns (e.g., migrating LiveData to StateFlow)
  • AI makes the same mistake 3+ times — add a new NEVER rule
  • New skill created — add it to Workflow section
  • Architecture decision changed — update Architecture section immediately

The "3-strike rule" is particularly effective: if AI makes the same mistake three times across different sessions, that's a signal that your CLAUDE.md is missing a constraint. Don't correct the AI — correct the CLAUDE.md.

Diagnostics: Verifying What Actually Loaded

Three built-in tools tell you exactly what Claude is reading at the start of a session. Use them whenever you're not sure why a rule is being ignored — almost always the answer is "the file you expected didn't load."

  • /memory slash command — lists every CLAUDE.md, CLAUDE.local.md, and rules file loaded in the current session. Toggles auto-memory on or off, and gives you a link to open the auto-memory folder. Run this first whenever Claude seems to ignore a rule: if the file isn't listed, Claude can't see it.
  • InstructionsLoaded hook — an observability hook that fires every time an instruction file gets loaded. It can't block or modify loading, but it logs which file loaded, when, and why. Wire it into a log file and you have a paper trail for path-scoped rules and on-demand subdirectory loads.
  • CLAUDE_CODE_NEW_INIT=1 — opt into the interactive multi-phase /init. Instead of generating a one-shot CLAUDE.md, Claude asks which artifacts to set up (CLAUDE.md, skills, hooks), explores your codebase with a subagent, fills gaps via follow-ups, and presents a reviewable proposal before writing any files.
Habit: after every CLAUDE.md edit, run /memory in a fresh session to confirm the file actually loads from where you expect. A typo in a path or a misplaced .gitignore rule can silently exclude the most important file in your config.

Common Pitfalls

Pitfall 1: The Novel

Your CLAUDE.md is not a knowledge base. It shouldn't explain what Clean Architecture is or how Hilt works. AI already knows. Long explanations waste context tokens and bury the actual rules. The official guidance is to target under 200 lines per CLAUDE.md file — longer files consume more context and reduce adherence. If you're crowding 200, split into @imports or move topic-specific rules into .claude/rules/ so they only load when relevant.

Pitfall 2: Copy-Paste from Docs

Don't paste your README or ARCHITECTURE.md into CLAUDE.md. They serve different audiences. CLAUDE.md speaks to AI — terse, rule-based, constraint-focused. Documentation speaks to humans — explanatory, contextual, narrative. Reference docs from CLAUDE.md; don't duplicate them.

Pitfall 3: Rules Without Enforcement

A CLAUDE.md rule is only as good as the feedback loop. If AI can violate a rule and nobody notices, the rule might as well not exist. Pair critical rules with hooks (Chapter 5) or review checklists (Chapter 7) that catch violations.

Pitfall 4: Personal Preferences as Project Rules

Your preference for trailing commas or single-line vs multi-line imports doesn't belong in the project CLAUDE.md unless the team has agreed on it. Put personal preferences in your user-level ~/.claude/CLAUDE.md. Project CLAUDE.md is for team-wide decisions only.

Your Chapter Exercise

Exercise: Create Your Production CLAUDE.md

Using the starter template above, create a production-grade CLAUDE.md for your project. Follow these steps:

  1. Start from the template — copy it into your project root
  2. Fill in Project Identity — one sentence, include tech stack
  3. Map your architecture — write the layer direction with arrows, name the pattern
  4. List every module — with its purpose and dependency rules
  5. Write 10 forbidden patterns — think about the last 10 times you corrected AI output. What did it get wrong? Each correction becomes a NEVER rule.
  6. Template every file type — screens, viewmodels, services, tests, configs
  7. Add testing standards — naming, coverage expectations, fakes vs mocks
  8. Define the workflow — build commands, lint commands, reference modules

Acceptance criteria: Your CLAUDE.md has all 6 sections, at least 10 forbidden patterns, and naming templates for every file type AI will create. Under 200 lines total.

Test it: Open Claude Code in your project and ask it to create a new feature. Does it follow your rules? If not, add the missing constraint and try again.

Commit the CLAUDE.md and update your AI_READINESS_BASELINE.md with your new Pillar 1 score.
