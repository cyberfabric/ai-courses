Chapter 7

Temperature, Top-p & Friends

Controlling randomness — what those API parameters actually do

In Chapter 6, you learned that text generation involves sampling from a probability distribution. Sometimes you want predictable, reliable output. Sometimes you want creative, surprising output. But how do you control that?

The answer is a handful of parameters — temperature, top-p, top-k, and a few others — that act as "creativity knobs" on the generation process. If you've ever seen these in an API call or tool settings and wondered what they do, this chapter is for you.

By the end, you'll know exactly which knob to turn for code generation vs. brainstorming vs. factual Q&A.

Temperature: The Creativity Dial

Temperature is the single most important generation parameter. It controls how "sharp" or "flat" the probability distribution is before sampling.

Remember from Chapter 6: for each position, the model produces probabilities for every possible next token. Temperature modifies those probabilities before the model picks one.

Analogy: The Adventure Level on a GPS

Imagine your GPS has an "adventure level" setting. At the lowest setting, it always picks the fastest, most predictable route — the highway, every time. At medium, it might suggest an interesting scenic road that's almost as fast. At the highest setting, it takes you on wild detours through villages you've never heard of.

Temperature works exactly like this. Low temperature = safest, most predictable route (highest-probability tokens). High temperature = adventurous, surprising routes (lower-probability tokens get a real chance).

Here's what different temperature values actually do:

Temperature = 0: Always Pick the Top Token

At temperature 0, the model always picks the single most likely token. No randomness at all. This is deterministic — the same prompt will produce the exact same output every time.

This is what you want when correctness matters more than variety. Factual lookups, structured data extraction, code that needs to follow a specific pattern exactly.

The downside? Output can be repetitive and boring. And if the most likely path isn't the best path, you're stuck with it — there's no chance of the model finding a better answer through a less obvious route.

Temperature = 1: Use the Probabilities As-Is

At temperature 1, the model uses its raw probabilities without modification. A token with 60% probability gets picked 60% of the time. A token with 5% probability gets picked 5% of the time.

This is the "default" behavior — what the model was actually trained to do. It produces natural, varied text that balances quality and creativity.

Temperature > 1: Flatten the Distribution

Higher temperatures make the distribution flatter. Unlikely tokens become more likely. The model gets more "adventurous" and picks surprising tokens more often.

At temperature 1.5 or 2.0, you'll get creative, unexpected output — but also more errors, nonsense, and incoherence. The model is essentially more willing to take risks, and sometimes those risks don't pay off.

Temperature 0 (deterministic)

Prompt: "Name a color" Response: "Blue" Every. Single. Time. Reliable but boring.

Temperature 1.5 (creative)

Prompt: "Name a color" Response 1: "Cerulean" Response 2: "Burnt sienna" Response 3: "The color of a Tuesday afternoon in November" Varied but unpredictable.

Here's a practical way to think about it:

Temperature Behavior Good For
0 Always the most likely token Code generation, factual Q&A, data extraction
0.3–0.5 Mostly predictable, slight variety Technical writing, code with some flexibility
0.7–1.0 Balanced creativity and coherence General conversation, explanations
1.0–1.5 More creative, some surprises Brainstorming, creative writing
> 1.5 Wild, often incoherent Experimental, rarely useful

Top-p (Nucleus Sampling): The Smart Filter

Temperature changes how the probabilities are distributed. Top-p takes a completely different approach: it limits which tokens are even considered.

Here's how it works: instead of considering all ~100,000 tokens, top-p only considers the smallest set of tokens whose combined probability adds up to p.

For example, with top-p = 0.9:

How top-p = 0.9 works

Token probabilities (sorted): "Paris" → 70% ← included (running total: 70%) "the" → 10% ← included (running total: 80%) "Lyon" → 5% ← included (running total: 85%) "a" → 4% ← included (running total: 89%) "located" → 3% ← included (running total: 92%) STOP Everything below "located" is excluded. We've reached 90% cumulative probability. Sample only from these top 5 tokens.

The beauty of top-p is that it's adaptive. When the model is confident (one token dominates), very few tokens pass the filter. When the model is uncertain (many tokens have similar probabilities), more tokens pass. The filter automatically adjusts to the situation.

Aha Moment

This is why top-p is often preferred over temperature in practice. Temperature blindly reshapes all probabilities. Top-p intelligently focuses on only the tokens that matter — the ones the model considers plausible.

With top-p = 0.95, the model can be creative when many options are viable (creative writing) and focused when only one or two options make sense (code syntax). It adapts automatically. Temperature doesn't do this — it applies the same adjustment everywhere.

Top-k: The Simpler Filter

Top-k is a simpler version of the same idea: only consider the top k most likely tokens, then sample from those.

  • Top-k = 1: Only consider the single most likely token. Same as temperature 0 — deterministic.
  • Top-k = 10: Consider the top 10 tokens, sample from those.
  • Top-k = 50: Consider the top 50 tokens — more variety.
  • Top-k = 100,000: Consider all tokens — effectively no filter.

The problem with top-k is that it's not adaptive. It always considers exactly k tokens regardless of how the probabilities are distributed. If one token has 99% probability, top-k = 50 still considers 49 mostly irrelevant alternatives. If the probabilities are spread across 200 plausible options, top-k = 50 cuts off potentially good candidates.

For this reason, top-p is generally preferred in modern AI systems. But you'll still see top-k in some APIs and tools.

Temperature vs. Top-p: When to Use What

In practice, most AI tools use either temperature or top-p (or both, carefully). Here's a simple decision guide:

Use temperature when:

You want direct control over how "creative" the output is. Set to 0 for deterministic output. Set to 0.7 for balanced output. Set higher for brainstorming. Simple mental model: creativity dial.

Use top-p when:

You want the model to adapt its creativity to each token position. Set to 0.9-0.95 for most tasks. Lower for more focused output. Higher for more exploration. Smart: adjusts per-token automatically.

Don't Combine Carelessly

Setting both temperature and top-p simultaneously can produce unexpected results. If you set temperature high (more randomness) AND top-p low (fewer candidates), the effects can fight each other. Most API documentation recommends adjusting one and leaving the other at its default. When in doubt, use top-p and leave temperature at 1.

Other Generation Parameters

Max Tokens: Setting the Length Limit

This one is straightforward: max tokens sets the maximum number of tokens the model can generate in its response. Hit the limit and the response stops abruptly, even mid-sentence.

A few things to know:

  • This is an upper limit, not a target. The model might stop earlier (when it generates a stop token).
  • If you set it too low, you'll get truncated responses. If you set it too high, you might get unnecessarily long responses (and pay for all those tokens).
  • Remember: max tokens is only about the output. Your input tokens don't count against this limit (they count against the context window, which is different — see Chapter 8).

Stop Sequences: Telling the Model When to Stop

Stop sequences are specific strings that tell the model: "When you generate this, stop immediately." This is useful when you want the model to produce a specific format and stop.

Example: Using stop sequences

Stop sequence: "\n\n" The model generates text and automatically stops when it produces two consecutive newlines. Useful for "generate one paragraph" type tasks.

Frequency Penalty & Presence Penalty: Fighting Repetition

These two parameters reduce repetition:

  • Frequency penalty: Penalizes tokens based on how many times they've already appeared in the output. Higher penalty = less repetition. The more a word has been used, the more it's penalized.
  • Presence penalty: Penalizes tokens that have appeared at all in the output, regardless of how many times. Even one occurrence triggers the penalty. This encourages the model to bring up new topics and ideas.

Most of the time, the defaults work fine. But if you notice the model repeating itself (especially in long outputs), a small frequency penalty (0.3–0.5) can help.

The Practical Guide: Best Settings for Common Tasks

Here's what you actually came here for — a cheat sheet for different use cases:

Task Temperature Top-p Why
Code generation 0–0.3 0.9–0.95 Code needs to be correct, not creative
Code review / explanation 0.3–0.5 0.9 Some variety in phrasing, but focused
Factual Q&A 0–0.2 0.9 You want the most likely (correct) answer
General conversation 0.7–1.0 0.95 Natural, varied responses
Creative writing 0.8–1.2 0.95–1.0 Encourage unexpected word choices
Brainstorming 1.0–1.3 0.95–1.0 Maximum variety in ideas
Data extraction / JSON 0 0.9 Strict format, no surprises

Aha Moment

This is why coding tools like Claude Code and Cursor use low temperature by default — they need predictable, syntactically correct code. And it's why creative AI tools use higher temperature — they need surprising, novel output.

The same model can behave like a precise engineer or a wild artist — the only difference is these generation parameters. Understanding this gives you the power to configure any AI tool for exactly the behavior you want.

Tip

If you're building an app with an AI API, the single most impactful thing you can do is set the temperature correctly for your use case. Most AI "quality issues" that developers report turn out to be a temperature problem — either too high (incoherent output) or too low (repetitive, boring output).

With low temperature, the model reliably picks the most likely tokens — which for code means syntactically correct, conventionally structured, well-known patterns. You don't want your code assistant to "get creative" with syntax.

If someone accidentally set the temperature to 1.5, you might get code that uses bizarre variable names, unconventional patterns, or even syntax that doesn't compile. The model would be "exploring" less likely tokens, which in the context of code means straying from correct patterns.

This is also why AI coding tools tend to produce "boring" but reliable code. They're optimized for correctness over style — which is exactly what you want when the output has to actually run.

Putting It All Together

  • Temperature controls how sharp/flat the probability distribution is. Low = predictable, high = creative.
  • Temperature 0 = always pick the top token. Deterministic and boring.
  • Temperature 1 = use raw probabilities as-is. The "default."
  • Top-p filters to only consider tokens within the top p% of probability. Adaptive and generally preferred.
  • Top-k considers only the top k tokens. Simple but not adaptive.
  • Max tokens limits output length. Stop sequences halt generation at specific strings.
  • Frequency/presence penalties reduce repetition.
  • Coding tools use low temperature for correctness. Creative tools use higher temperature for variety.

Exercise: Feel the Temperature Difference

If your AI tool allows setting temperature (the API, Playground, or some chat interfaces), try this experiment:

  1. Pick a coding prompt: "Write a Python function that reverses a string"
  2. Run it at temperature 0. Note the result.
  3. Run the same prompt at temperature 1.0. Compare.
  4. Run it at temperature 1.5. What happens?

Now try a creative prompt: "Write a one-sentence story about a robot who learns to dance"

  1. Run it 5 times at temperature 0. Are the results identical?
  2. Run it 5 times at temperature 1.0. How much variety do you see?

What to notice: For the coding prompt, low temperature produces reliable, standard code. For the creative prompt, higher temperature produces more diverse and interesting results. Same model, same mechanism — different knob settings.

No API access? You can still observe this indirectly: ask ChatGPT the same factual question 5 times (notice the consistency) vs. the same creative question 5 times (notice the variety). The underlying parameters are what cause this difference.
