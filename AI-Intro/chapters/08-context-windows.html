Chapter 8

Context Windows & How Conversations Work

System prompts, chat history, the memory illusion, and why AI “forgets”

You've been having "conversations" with ChatGPT, Claude, and other AI tools for a while now. You ask a question, get an answer, ask a follow-up, and the model references things you said earlier. It follows up on previous messages. It seems to track the thread of discussion. It feels like a conversation with someone who's paying attention.

But here's the uncomfortable truth we teased at the end of Chapter 7:

AI has no memory between requests. None. Zero. Every single response is generated from scratch, as if the model has never seen you before.

What you experience as a "conversation" is actually the application sending the entire conversation history with every single message.

This chapter explains how the illusion works, what context windows are, why conversations "break" after a while, and why understanding this will make you significantly better at using AI tools.

The Fundamental Truth: No Memory

Let's be absolutely clear about this: when you send message #5 in a ChatGPT conversation, the model does not "remember" messages #1 through #4. It has no persistent state between requests. No internal notebook. No memory bank. No hidden database of past conversations.

The model doesn't even know it's in a "conversation." From its perspective, every request is a brand new, standalone prompt that happens to contain some text formatted like a conversation.

So how does it work? The application (ChatGPT, Claude, Cursor) does something clever: it takes your entire conversation history — every message you've sent and every response the model has given — and sends it all as a single, giant prompt with your new message appended at the end.

Analogy: The Amnesia Friend

Imagine you have a friend with complete amnesia. Every time you talk to them, they've forgotten everything. But you've been keeping a journal of all your previous conversations.

Each time you want to talk, you hand them the entire journal and say "Read this first, then answer my new question." They read the journal, understand the full context, and give you a great response. Then they immediately forget everything again.

Next time you talk, you hand them the journal again — now with the previous response added — plus your new question. They read the whole thing from scratch, respond, forget.

That's exactly how AI conversations work. The "journal" is the conversation history. The "amnesia" is the model's lack of persistent memory.

The Conversation Stack

Every time you send a message, the application constructs a single prompt that looks something like this:

What actually gets sent to the model (message #3 of a conversation)

[System] You are a helpful assistant... [User] What's the best way to sort an array in Python? [Assistant] There are several ways to sort arrays in Python. The most common approach is... [User] What about if I need to sort by a custom key? [Assistant] For custom sorting, you can use the key parameter in sorted() or .sort()... [User] Can you show me an example with a list of dictionaries?

The entire conversation — system prompt, all previous messages and responses, and your latest message — is sent as one prompt. The model processes everything from scratch each time.

This stack has three layers:

1. System Prompt: The Hidden Instructions

The system prompt is a set of instructions that comes before everything else. You usually never see it. It tells the model how to behave: "You are a helpful assistant," "Always respond in JSON," "You are an expert Python developer," etc.

In tools like Claude Code, the system prompt includes detailed instructions about how to interact with your codebase, what tools are available, and how to format responses. When you create a CLAUDE.md file in your project, its contents get included in this system prompt. That's why CLAUDE.md works — it's literally part of the prompt the model sees every time.

System prompts can be quite long — sometimes thousands of tokens. This is context window space that you "pay" before you've even typed your first message. It's usually worth it because good system instructions dramatically improve response quality, but it's important to know they're there.

2. Conversation History: The Replay

Every previous message and response is replayed in order. The model reads through all of it, building up context through its attention mechanism (Chapter 5). By the time it reaches your latest message, it has "attended to" the entire conversation.

This means the model re-reads your first message every single time. Message #1 gets processed when you send message #2, again when you send #3, again for #4, and so on. It's wasteful in a sense — but it's the only way a stateless system can maintain the illusion of continuity.

3. Your Latest Message: The Actual Question

Your new message comes last. The model generates a response to this, informed by everything above it in the stack. The position of your message at the end is important — it means the model has maximum context when it starts generating the response.

Tip

This architecture explains why the first message in a conversation feels just as fast as the fifth — message #5 isn't built on top of message #4. It's a complete re-read of everything from scratch. The model is doing more work (reading more tokens), but it's the same process every time.

Context Window: How Much the Model Can See

The context window is the maximum number of tokens the model can process in a single request — the combined total of system prompt + conversation history + your message + the response.

Context windows have grown dramatically:

In Chapter 4, we discussed that 1 token ≈ 0.75 words. So:

Context Window Approximate Words Roughly Equivalent To
4K tokens ~3,000 words A few pages
32K tokens ~24,000 words A long article or report
128K tokens ~96,000 words A short novel
1M tokens ~750,000 words Several novels, or a large codebase

What Happens When You Hit the Limit

As your conversation grows longer, the total token count (system prompt + history + new message + response) approaches the context window limit. When it gets too long, something has to give. Different tools handle this differently:

  • Truncation: The oldest messages get dropped from the history. The model literally can't see them anymore. This is the most common approach and usually happens silently — you won't get a warning.
  • Summarization: Older parts of the conversation are summarized into a shorter version, preserving key points while reducing token count. More sophisticated but harder to implement well.
  • Sliding window: Keep only the most recent N messages. Simple and predictable, but can lose important early context.
  • Error: Some API implementations simply refuse the request if it exceeds the limit, returning an error that you need to handle.

Aha Moment

This is why AI "forgets" things in long conversations. It's not getting confused or losing track. The old messages are literally being removed from the prompt. The model can't reference what it can't see.

When you have a 50-message conversation and the model seems to have "forgotten" what you discussed at the beginning, it probably has — because those early messages were truncated to fit within the context window. It's not a bug. It's a space constraint.

This is also why starting a new conversation sometimes helps. A fresh conversation has no history consuming your context window — the model can dedicate more capacity to your actual question.

And this explains a common frustration: you're debugging something with an AI, and after many back-and-forth messages, the model starts making contradictory suggestions or forgetting earlier findings. It's not getting "confused" — it's losing context. The earliest (and often most important) messages are being dropped as the conversation outgrows the window.

Practical Advice

If a long conversation starts going sideways, start a new one. Summarize the key findings and context from the old conversation in your first message. This gives the model a fresh context window with the most important information front and center, instead of buried under dozens of messages.

The Context Window Is RAM, Not a Hard Drive

Here's the best mental model for context windows:

Analogy: RAM vs Hard Drive

Think of the context window as RAM, not a hard drive. It's the model's working memory — what it can actively think about right now. And just like RAM, it has a fixed size.

When your computer runs out of RAM, it starts swapping things to disk (slow) or killing processes. When an AI conversation runs out of context window, it starts dropping old messages (forgetting) or summarizing.

The model has no "hard drive" — no persistent storage between conversations. Everything it knows about your conversation exists only in this RAM-like context window. Close the window, and it's gone.

This analogy also explains why bigger context windows help but don't solve everything. More RAM lets your computer handle more tasks simultaneously, but it doesn't make the CPU faster. Similarly, a 1M-token context window lets the model see more text, but it doesn't necessarily make the model smarter at using that text.

And just like closing unused browser tabs frees up RAM, removing unnecessary conversation history or irrelevant files from the context frees up tokens for what actually matters. Context window management is resource management.

How Coding Tools Manage Context

This is where things get really practical. Tools like Claude Code, Cursor, and Copilot face a particular challenge: your codebase has thousands of files, but the context window can only hold so much.

So these tools make smart decisions about what to include:

CC
Claude Code
Reads your project structure, CLAUDE.md instructions, and selectively includes relevant files based on your prompt. Uses tool calls to read specific files as needed rather than loading everything upfront.
Cu
Cursor
Indexes your codebase and includes currently open files, recent edits, and files that seem relevant to your query. Uses embeddings (Chapter 9) to find the most related code.
CP
GitHub Copilot
Primarily uses the current file and nearby files as context. Focuses on the code immediately around your cursor position for inline completions.

The quality of these tools largely comes down to how well they select context. A tool that picks the right 5 files to include will outperform one that dumps 50 irrelevant files into the prompt. This is why different coding assistants can use the same underlying model but produce very different results — the model is only as good as the context it receives.

This context curation is one of the hardest engineering problems in AI tooling. How do you figure out which files are relevant to a user's question before you even know the answer? Tools use a combination of heuristics (current file, recently edited files, imported modules) and semantic search (embeddings, which we'll explore in Chapter 9) to make these decisions.

Tip

This is why CLAUDE.md files are so powerful. They give the model permanent, always-present context about your project's conventions, architecture, and preferences. Instead of the model guessing what framework you use or how you name variables, CLAUDE.md tells it directly — and it's included in every single request.

Why "Just Make It Bigger" Isn't the Answer

Context windows have grown from 4K to 1M tokens. Problem solved, right? Not quite.

Cost

You pay per token. A 100K-token prompt costs roughly 100x more than a 1K-token prompt. Stuffing your entire codebase into every request would be extraordinarily expensive. For an API-heavy application processing thousands of requests per day, unnecessary context tokens add up to real money fast.

Speed

More input tokens means more computation. Processing 500K tokens of context takes significantly longer than processing 5K tokens. The model's attention mechanism has to compare every token with every other token — that's roughly quadratic in cost. Double the context, roughly quadruple the computation for the attention layers.

Attention Degradation (The "Lost in the Middle" Problem)

Here's the subtlest problem: models get worse at finding relevant information as the context gets longer. When you dump 100K tokens of text into a prompt, the model has to sort through all of it to find what's relevant to your question.

Research has shown a striking pattern: models are best at using information at the beginning and end of the context, and tend to overlook information buried in the middle. If the critical detail is on "page 50 of 100," the model is more likely to miss it than if it were on page 1 or page 100.

This has real implications for how you structure prompts. Important instructions belong at the beginning (system prompt) or end (just before your question), not buried in the middle of a long context dump.

More Context Is Not Always Better

1M tokens of capacity doesn't mean you should use 1M tokens. Including irrelevant files, old conversation history, and unnecessary boilerplate actively hurts the model's ability to focus on what matters. Think of it like giving someone a 500-page document when the answer is on page 237 — they'll find it eventually, but they'd find it much faster if you just gave them page 237.

Bad: Dump everything

"Here are all 200 files in my project. Also here is the full git log. And the complete test output. Now fix the bug in user.py." → Model drowns in irrelevant context. Misses the key information. Slow and expensive.

Good: Curate the context

"Here is user.py and the error traceback. The User model is in models/user.py. We use SQLAlchemy for the ORM. Fix the bug on line 42." → Model has exactly what it needs. Fast, cheap, accurate.

KV Cache: Why Long Conversations Use More Memory

There's a technical detail worth knowing if you're curious about why AI providers charge more for longer contexts: the KV cache.

Remember how attention works (Chapter 5)? Each token computes keys and values that other tokens reference. When generating a response, the model stores the keys and values for every token in the conversation — this is the KV cache.

As the conversation grows, the KV cache grows with it. A 100K-token conversation requires storing keys and values for all 100K tokens across every attention head in every layer. This takes a lot of GPU memory.

This is a significant part of why longer contexts are more expensive. It's not just the computation — it's the memory needed to hold the KV cache for the entire conversation while the model generates each output token.

Some providers use "prompt caching" to optimize this: if the first part of your prompt is the same across multiple requests (which it often is — same system prompt, same conversation history), they can reuse the cached KV values and only compute the new tokens. This is why subsequent messages in a conversation can sometimes feel faster than the first one.

Tip

This is why some providers offer different pricing tiers for different context lengths. A 4K-context request is not only faster but uses dramatically less GPU memory than a 128K-context request. When designing AI-powered features, the most cost-effective approach is to use the minimum necessary context, not the maximum available.

Putting It All Together

  • AI has zero memory between requests. Every response is generated from scratch.
  • "Conversations" work by sending the complete history (system prompt + all messages) with every request.
  • The context window is the maximum tokens the model can handle. It's shared between input and output.
  • When conversations exceed the context window, old messages get dropped (truncation) or compressed (summarization).
  • The context window is RAM, not a hard drive. Bigger helps, but there are costs: money, speed, and attention degradation.
  • Coding tools succeed by curating context well — including the right files, not all files.
  • CLAUDE.md files work because they're included in the context with every request.
  • More context isn't always better. Focused, relevant context outperforms massive dumps of text.

Exercise: Test the Memory Illusion

Try this experiment in any AI chatbot:

  1. Start a new conversation. In your first message, tell the model: "My favorite color is turquoise. Remember that."
  2. Send 20-30 messages on various unrelated topics (just ask random questions).
  3. Then ask: "What's my favorite color?"
  4. Does it remember? If you're using a tool with a large context window, it probably does — because your first message is still in the context. If the context window is small, it might have been truncated.

Now try the same thing but start the conversation in a new chat window:

  1. In a brand new chat, ask: "What's my favorite color?"
  2. The model will have no idea. Because this is a new context window — your previous conversation doesn't exist.

What to notice: The model "remembers" within a conversation (because the history is re-sent each time) but has zero memory across conversations. That's the illusion at work.

Going further: Some tools (like ChatGPT's "Memory" feature or Claude's memory system) add a layer on top by saving key facts to a persistent store and injecting them into the system prompt. This gives the appearance of long-term memory, but it's still the same mechanism — the saved facts are just added to the context window of future conversations.
