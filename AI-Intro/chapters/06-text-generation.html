Chapter 6

How Text Generation Actually Works

Next-token prediction, autoregressive generation, and the probability game

You know what tokens are (Chapter 4). You know how the transformer processes them (Chapter 5). Now it's time for the big reveal: how does text actually come out?

When you type a prompt into Claude or ChatGPT and words start streaming across your screen — what's actually happening, step by step? This is the chapter where everything from Chapters 1–5 clicks together into one complete picture.

Fair warning: this chapter might change how you see AI tools forever. Once you understand how text generation actually works, you'll never look at a streaming response the same way again.

And the answer starts with a surprisingly simple truth.

The Fundamental Truth: One Token at a Time

LLMs generate text one token at a time. Not one sentence. Not one paragraph. One token.

Every response you've ever received from ChatGPT, Claude, or any LLM was produced by predicting a single next token, appending it, and repeating — hundreds or thousands of times.

This process is called autoregressive generation, and it works like this:

  1. You provide a prompt: "Write a function that sorts a list"
  2. The model predicts the most likely first token of the response. Let's say it picks "def".
  3. Now the input becomes: "Write a function that sorts a list" + "def"
  4. The model predicts the next token given this extended input. It picks " sort".
  5. Now the input is: "Write a function that sorts a list" + "def sort"
  6. The model predicts the next token: "_list".
  7. And so on, token by token, until it generates a special "stop" token or hits the maximum length.

Each new token is generated based on everything that came before it — both your original prompt and all the tokens the model has generated so far. The response builds on itself, one piece at a time.

This means the model is essentially making a fresh decision at every single step. "Given everything so far, what's the best next token?" It doesn't plan ahead. It doesn't outline a response structure. It doesn't think "I'll need to close this function bracket later." It just predicts the single next token, and the structure of the response emerges one step at a time.

This is both amazing (it works!) and a fundamental limitation (it can lose track of complex, long-range structure). Keep this in mind — it explains a lot about why AI occasionally loses the plot in long responses.

Analogy: Autocomplete on Steroids

You know the autocomplete on your phone keyboard? It suggests 3 possible next words based on what you've typed so far. You tap one, and it suggests 3 more.

LLMs work the exact same way, but instead of 3 suggestions, they have 100,000 (one for every token in the vocabulary). And instead of you tapping to choose, the model picks one automatically based on learned probabilities. Then it uses that choice to generate the next 100,000 suggestions. And again. And again.

That's it. That's the entire mechanism behind every AI response you've ever seen. Phone autocomplete, but with 100,000 options and a transformer with billions of parameters doing the prediction.

The Probability Distribution

For each step, the model doesn't just pick one token. It computes a probability score for every single token in its vocabulary.

Imagine this: you've prompted the model with "The capital of France is" and it needs to predict the next token. The output of the transformer's final layer is a score for each of the ~100,000 tokens:

Example: Probability distribution for "The capital of France is ___"

" Paris" → 92.1% " the" → 1.3% " a" → 0.8% " Lyon" → 0.4% " located" → 0.3% " known" → 0.2% ... " banana" → 0.00001% " def" → 0.000001% (~100,000 tokens, each with a probability, all adding up to 100%)

For a factual question like this, the distribution is very "sharp" — one token dominates with over 90% probability. The model is very confident about what comes next. But for more open-ended prompts, the distribution can be much flatter, with many tokens having reasonable probabilities.

Example: Probability distribution for "Once upon a ___"

" time" → 87.5% " midnight" → 2.1% " day" → 1.4% " warm" → 0.9% " cold" → 0.7% " dreary" → 0.5% ... Still "time" is dominant, but more alternatives are plausible.

For a creative writing prompt like "The dragon opened its mouth and"? The distribution could be very flat, with dozens of tokens having similar probabilities: "roared," "spoke," "breathed," "whispered," "laughed." That's when things get interesting — and when the choice of sampling strategy really matters.

Why Not Just Pick the Most Likely Token?

You might think: "Just always pick the token with the highest probability. Problem solved." This approach is called greedy decoding, and it actually works — but it produces terrible output.

Greedy decoding (always pick #1)

Prompt: "Write a poem about the ocean" The ocean is a beautiful place. The ocean is a beautiful place. The ocean is a beautiful place. It is a beautiful place to be. It is a beautiful place to be.

Repetitive, boring, loops. The "most likely" next token after each sentence tends to restart the same pattern.

With sampling (pick randomly, favoring likely)

Prompt: "Write a poem about the ocean" Beneath the silver veil of moon, the ocean hums a restless tune. Salt and shadow, foam and light — the tide exhales into the night.

Creative, varied, interesting. Occasional "unlikely" token choices produce fresh combinations.

Greedy decoding gets stuck in loops because the most probable next token is often the same one that was most probable before. The model keeps predicting the same high-probability patterns over and over. It's like a GPS that always takes the "most popular" road — you'd drive the same route every time, even when there are perfectly good alternatives.

The solution? Don't always pick the most likely token. Instead, randomly sample from the top candidates, with higher-probability tokens being more likely to be chosen.

Sampling: The Art of Controlled Randomness

Instead of always picking the #1 token, the model randomly selects a token from the probability distribution, where tokens with higher probabilities are more likely to be chosen.

Think of it like a weighted lottery. If "Paris" has a 92% probability, it will be selected most of the time. But occasionally, "Lyon" or "the" might get picked instead. For factual contexts, this almost never matters because the top token is so dominant. For creative contexts, it's what makes AI interesting.

This randomness is a feature, not a bug. It's what makes AI responses feel natural and creative rather than robotic and repetitive. Without it, every AI chatbot would sound like a broken record, repeating the most statistically common phrases over and over.

Tip

The exact behavior of this sampling is controlled by parameters like temperature, top-p, and top-k. These are the "creativity knobs" that determine how much randomness to allow. We'll explore them in detail in Chapter 7. For now, just know: sampling = controlled randomness = the reason AI output feels alive.

Now let's put the complete pipeline together. Here's what happens every time you hit Enter:

Through all layers

5. Sample
Pick one token

After step 5, the selected token is appended to the input, and the loop repeats from step 3. The model generates one token, feeds it back in, generates the next, feeds that back in, and so on until it's done.

Let's trace a concrete example. You type into Claude Code: "Write a hello world function in Python"

  1. Tokenize: Your prompt becomes something like [8144, 264, 24748, 1917, 734, 304, 13325]
  2. Embed: Each token ID is converted into a rich numerical representation (embedding)
  3. Transform: These embeddings flow through 80+ transformer layers, with attention connecting every token to every other token
  4. Score: The final layer outputs probabilities for all ~100K tokens. "def" scores highest.
  5. Sample: "def" is selected.
  6. Repeat: Now the input is your prompt + "def". Process again. " hello" is next. Then "_world". Then "():". Then "\n". Then " print". Then ("(". Then '"Hello'. And so on, token by token, until the function is complete.

Notice something: the model didn't plan the function first and then type it out. It discovered the function structure one token at a time. Each token was simply the most likely continuation of everything before it. The fact that the result looks like a coherent, well-structured function is a testament to how good the patterns in the training data are.

Each token requires a complete forward pass through the entire transformer. For a 200-token response, that's 200 full passes through billions of parameters. For a long code block of 1,000 tokens? That's 1,000 full passes. The compute adds up fast.

Why Responses Feel "Creative"

Now you understand why AI doesn't sound like a robot. The randomness in sampling means the model doesn't just regurgitate the most common patterns — it explores alternatives. Sometimes a slightly less likely token leads to a much more interesting continuation.

It's like jazz improvisation. A jazz musician knows the most "correct" note for each moment, but deliberately plays unexpected notes that create tension and surprise. The music is more interesting because of the unpredictability.

AI writing works the same way. The model knows the "most likely" continuation, but by sometimes picking less likely tokens, it produces text that feels fresh and varied. This is especially important for creative tasks — poetry, brainstorming, storytelling — where predictability is the enemy.

For code generation, though, you usually want predictability. You don't want "creative" variable names or "surprising" syntax. This is why coding tools tend to use less randomness than chatbots. The same model, the same mechanism — but with the randomness dialed way down. We'll see exactly how this works in Chapter 7.

Why the Same Prompt Gives Different Answers

Have you noticed that if you send the exact same prompt to ChatGPT twice, you often get different responses? Now you know exactly why: sampling is random.

At every token position, the model randomly selects from the probability distribution. Even if the probabilities are the same each time, the random selection might pick a different token. That different choice cascades — because the next token's probabilities depend on the previous token, a single different choice early on can lead to a completely different response.

The Butterfly Effect of Token Sampling

One different token early in the response can change everything that follows. If the model picks "However" instead of "But" as the first word, the entire sentence structure shifts. The next token probabilities change, leading to different word choices, which lead to different ideas, which lead to a completely different response.

This is why you can ask the same question five times and get five genuinely different answers. It's not random noise — it's the natural consequence of sampling from a probability distribution at each step.

For code generation, this variability is usually lower because code has more constrained patterns (there are fewer "correct" ways to write a function), so the probability distributions tend to be sharper. But it still happens — which is why Cursor or Claude Code might suggest slightly different variable names or code structures on different runs.

This also explains a useful trick: if you're not happy with an AI's response, just regenerate it. You might get a much better answer on the second or third try, simply because different random samples lead to a different (and potentially better) response path.

Why AI "Hallucinates"

This is the section that ties the biggest loose end of this book.

You've probably seen AI confidently state something that's completely wrong. Made-up library names. Nonexistent API methods. Fabricated historical events. Citations to papers that don't exist. Why?

Now you have the tools to understand: the model is not trying to be correct. It's trying to predict the most likely next token.

When you ask "What year was the Battle of Thermopylae?", the model isn't looking up the answer in a database. It's generating a response token by token, each chosen because it's statistically likely given the context. Usually, the statistically likely answer happens to be the correct one (because the training data contained accurate information about Thermopylae many times).

But when you ask about something obscure, or something the model has seen conflicting information about, or something that wasn't in its training data at all — the model doesn't know it doesn't know. It has no built-in uncertainty detector. It doesn't pause and think "hmm, I'm not sure about this." It just generates the most plausible-sounding continuation. And plausible-sounding is not the same as correct.

This is especially dangerous with code. When you ask Claude Code for an API method from a less-popular library, it might generate a method name that looks right (follows the naming conventions, fits the context) but doesn't actually exist. The model isn't lying — it's just predicting what tokens are most likely in that context, and sometimes the most likely tokens form a hallucination.

Aha Moment

Hallucination isn't a bug in the system. It's the system working exactly as designed.

The model's entire job is to generate plausible next tokens. It's extraordinarily good at this — so good that most of the time, "plausible" and "correct" overlap. But when they don't, the model has no mechanism to prefer truth over plausibility. It will confidently generate a wrong answer that sounds right, because "sounding right" is literally what it's optimized to do.

This isn't something that will be "fixed" with a software update. It's a fundamental property of next-token prediction. Models are getting better at saying "I don't know" (through fine-tuning), but the underlying architecture will always generate something — it can't simply stay silent when it's uncertain.

Practical Takeaway

Never trust AI output on critical facts without verification. This isn't a temporary limitation — it's inherent to how generation works. Use AI for drafting, brainstorming, and code patterns, but verify facts, API references, and library names independently. The model is a writing partner, not a fact database.

The Speed Question: Why Generation Is Slow

In Chapter 5, we celebrated that transformers process input tokens in parallel. So why does generation still feel slow? Why do you see text appearing one word at a time?

Because reading and writing are fundamentally different operations:

Operation How It Works Speed
Processing your prompt All tokens in parallel Fast (even for long prompts)
Generating the response One token at a time, sequentially Slow (each token needs a full forward pass)

Your 10,000-token prompt gets processed all at once in a single pass. But the 500-token response has to be generated one token at a time, because each token depends on all previous tokens. You can't generate token #5 until you know what tokens #1–4 are — it's inherently sequential.

This is why you sometimes notice a brief pause after hitting Enter (the model is processing your entire prompt in parallel), followed by text that appears gradually (the model is generating tokens one at a time). The pause is the fast part. The gradual appearance is the slow part.

It's also why output tokens cost more than input tokens in API pricing — generating each output token requires more computation than simply processing an input token.

Streaming: Why Text Appears Word by Word

The word-by-word appearance isn't a fancy animation effect. It's the actual generation happening in real time. Each word appears as soon as the model generates it, without waiting for the full response to be complete.

This is called streaming, and it's a practical design choice. Without streaming, you'd stare at a blank screen for 10-30 seconds and then the entire response would appear at once. With streaming, you get something to read while the model is still working on the rest.

Streaming also lets you interrupt early. If you see the model going in the wrong direction — generating the wrong approach, using the wrong library, misunderstanding your intent — you can stop it before it wastes tokens (and your time and money) completing a bad response. In tools like Claude Code, this is especially valuable: you can see the code taking shape and hit stop if it's not what you want.

When you see Claude Code generating a long code block and the text appears smoothly token by token — you're watching the model run through its entire neural network (all layers, all attention heads, all parameters) once per token, in real time. Every word you see is the result of trillions of mathematical operations.

Faster models aren't necessarily "smarter" — they might just be running on faster hardware, using a smaller architecture, or employing optimization tricks like quantization (reducing the precision of those parameter numbers to make computation faster). There's always a trade-off between speed and quality.

Putting It All Together

  • LLMs generate text one token at a time — autoregressive generation.
  • For each position, the model computes probabilities for every possible next token (~100K candidates).
  • Always picking the #1 token (greedy) produces repetitive, boring output.
  • Sampling with randomness produces natural, creative, varied text.
  • The same prompt gives different answers because sampling is random — one different token cascades into a completely different response.
  • Hallucination is inherent: the model generates plausible tokens, not necessarily correct ones. Plausible ≠ true.
  • Reading your prompt is fast (parallel). Generating a response is slow (sequential, one token at a time).
  • Streaming shows tokens as they're generated — it's the actual process, not an animation.

Exercise: See the Randomness Yourself

Open ChatGPT, Claude, or any AI chatbot and try this experiment:

  1. Send the exact same prompt 3 times: "Write me a haiku about programming"
  2. Compare the three responses. Are they identical? Similar? Completely different?
  3. Now try a factual prompt 3 times: "What is the capital of Japan?"
  4. Compare again. Are these more consistent? Why?

Explanation: Creative prompts have flat probability distributions (many plausible next tokens), so sampling produces varied results. Factual prompts have sharp distributions (one token dominates), so responses are more consistent — but still not always identical.

Bonus: Try asking the model to name a specific, obscure API method from a less-common framework. Does it give a real method or a hallucinated one? Check the documentation to find out. This is hallucination in action.
