Building Your Own

You don't need to build from scratch — you're already customizing a harness

You're Already a Harness Engineer

Here's the thing most people miss: you don't need to build a harness from scratch. Every time you write a CLAUDE.md file, create a Cursor rule, configure a custom command, or add an MCP server — you're engineering a harness. You're deciding what the model knows, what tools it can use, and how it should behave.

Think of it like customizing your development environment. You didn't build VS Code from source. But you've spent hours on settings, extensions, keybindings, and snippets. That configuration is your IDE — the raw editor is just the starting point. The same is true for AI coding tools.

This chapter is your practical guide to making the harness work for your projects, your team, and your workflows. We'll cover five layers of customization, from simplest to most powerful.

The Five Layers of Harness Customization

1. Project instructions — CLAUDE.md, AGENTS.md, Cursor rules

2. Custom commands — reusable workflows your team can share

3. Hooks — automated verification and enforcement

4. MCP servers — custom tools for external systems

5. Permission tuning — balancing autonomy and safety

Each layer builds on the previous one. Start with project instructions and add complexity only when you need it.

CLAUDE.md: Your Most Important File

If you do nothing else from this entire course, write a good CLAUDE.md. It's the single highest-impact harness customization you can make. Every conversation with Claude Code starts by reading this file — it shapes every decision the model makes in your project.

The Hierarchy

Claude Code loads CLAUDE.md files in layers, from broadest to most specific:

~/.claude/CLAUDE.md
Global — your personal defaults across all projects
→
./CLAUDE.md
Project root — shared with team via git
→
./src/api/CLAUDE.md
Directory-level — loaded when Claude works in that area

Each layer adds context. A directory-level file doesn't replace the project file — it supplements it. This is progressive disclosure: broad guidance at the top, specific guidance where it's needed.

What to Include

A well-structured project CLAUDE.md
# My Project

## Build & Test
- Build: `npm run build`
- Test: `npm run test` (vitest)
- Lint: `npm run lint` (eslint + prettier)
- Single test: `npm run test -- path/to/file.test.ts`

## Code Conventions
- Use TypeScript strict mode everywhere
- Prefer named exports over default exports
- Error handling: use Result types from `@/lib/result`, not try-catch
- State management: Zustand, never Redux

## Testing Rules
- IMPORTANT: Use msw for API mocking, never jest.mock()
- Test user behavior, not implementation
- Every new component needs a test file

## Architecture
- /src/api/ — Express routes, one file per resource
- /src/lib/ — Shared utilities, no side effects
- /src/components/ — React components, colocated tests

What to Leave Out

Bloated CLAUDE.md — information Claude can figure out itself
# Don't include these:
- Write clean, readable code           ← Claude already does this
- Use TypeScript                        ← Claude can see your tsconfig.json
- Files in /src contain source code     ← Obvious from the directory name
- Our API returns JSON                  ← Claude will see this in the code
- Follow SOLID principles               ← Generic advice that adds noise
- [500 lines of API documentation]      ← Link to docs instead

For every line, ask: "Would removing this cause Claude to make a mistake?" If not, cut it. Bloated files cause the model to ignore actual instructions.

The @import Trick

CLAUDE.md supports importing other files with the @path/to/file syntax. Instead of duplicating your git workflow in CLAUDE.md, create a docs/git-workflow.md and import it: @docs/git-workflow.md. This keeps CLAUDE.md lean while making detailed guidance available.

AGENTS.md: The Cross-Tool Alternative

If your team uses multiple AI tools, AGENTS.md is worth knowing. Now stewarded by the Linux Foundation's Agentic AI Foundation, it's an open format supported by Codex, Cursor, Google Jules, Amp, and Factory.

The format is similar to CLAUDE.md but uses a hierarchical discovery chain: files are concatenated from the git root down to the current directory. Codex adds Skills — a progressive disclosure pattern where the agent sees only skill names and descriptions until it decides to use one, then loads the full instructions.

AGENTS.md with skill reference
# Project Agent Instructions

## Overview
E-commerce API built with Express + PostgreSQL.

## Standards
- All endpoints must validate input with Zod
- Database queries use the query builder in /src/db/

## Skills
See `.agents/skills/` for specialized workflows:
- `add-endpoint` — scaffold a new API endpoint
- `write-migration` — create a database migration

Custom Commands: Reusable Workflows

CLAUDE.md tells the model what to know. Custom commands tell it what to do. They're markdown files that become slash commands — reusable workflows your entire team can share via git.

How They Work

Drop a markdown file in .claude/commands/ and it becomes a slash command. The filename becomes the command name.

.claude/commands/review.md — becomes /project:review
Review the current git diff for:
1. Security issues (injection, XSS, secrets in code)
2. Error handling gaps
3. Missing test coverage
4. Performance concerns (N+1 queries, unbounded loops)

For each issue found:
- Cite the file and line number
- Explain the risk
- Suggest a specific fix

If no issues found, say so briefly. Don't invent problems.

Now anyone on the team types /project:review and gets a consistent, thorough code review. No remembering what to check. No variation between reviewers.

Dynamic Arguments

Use $ARGUMENTS as a placeholder for whatever the user types after the command:

.claude/commands/explain.md — /project:explain auth middleware
Explain $ARGUMENTS in this codebase.

1. Find the relevant source files
2. Trace the execution flow from entry point
3. Explain what it does in plain language
4. Note any non-obvious behavior or gotchas
5. List the key functions and where they're defined

Organizing Commands

Subdirectories create namespaces. Put frontend commands in .claude/commands/frontend/ and they become /project:frontend:component, /project:frontend:storybook, etc.

Personal commands go in ~/.claude/commands/ — available across all your projects but not shared with the team.

Commands vs CLAUDE.md

CLAUDE.md defines what Claude should always know (conventions, standards, architecture). Commands define what Claude should do on demand (review, scaffold, audit, migrate). Commands can reference standards in CLAUDE.md without repeating them — "Follow the testing rules from CLAUDE.md" is enough.

Hooks: Automated Guardrails

CLAUDE.md instructions are advisory — the model should follow them, but might not. Hooks are deterministic — they always execute. They're shell commands that run at specific lifecycle points, giving you programmatic control over what the agent does.

When Hooks Fire

The most useful hook events for day-to-day work:

  • PreToolUse — runs before a tool executes. Can block the action (exit code 2) or modify inputs.
  • PostToolUse — runs after a tool executes. Great for auto-formatting, logging, or validation.
  • UserPromptSubmit — runs when you press Enter. Can transform or validate your input.
  • Stop — runs when the agent finishes. Good for final checks.

Practical Examples

.claude/settings.json — auto-format after every file edit
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
      }]
    }]
  }
}

Every time Claude edits or writes a file, Prettier formats it automatically. No "please format" reminders needed.

Block dangerous commands with PreToolUse
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "if echo \"$TOOL_INPUT\" | grep -qE 'rm -rf|drop table|force push'; then echo 'Blocked: dangerous command' >&2; exit 2; fi"
      }]
    }]
  }
}

Exit code 2 blocks the action. The stderr message becomes feedback to Claude, who will adjust its approach.

Hooks vs CLAUDE.md Instructions

Don't use hooks for things that CLAUDE.md handles well (style preferences, naming conventions). Use hooks for things that must happen reliably: formatting, security checks, test execution. If "usually follows the rule" is good enough, use CLAUDE.md. If "always follows the rule" is required, use a hook.

MCP Servers: Extending Your Toolbox

Model Context Protocol servers give your AI coding tool access to systems it can't reach natively — databases, APIs, monitoring dashboards, internal tools. Each MCP server exposes a set of tools that the model can call just like built-in tools.

Adding Servers

Adding MCP servers via CLI
# GitHub integration — PRs, issues, repo management
claude mcp add github -- npx @anthropic-ai/github-mcp

# PostgreSQL — query your database directly
claude mcp add postgres -- npx @anthropic-ai/postgres-mcp

# Brave Search — web search from within Claude Code
claude mcp add brave-search --env BRAVE_API_KEY=your-key -- \
  npx @anthropic-ai/brave-search-mcp

# Custom internal API
claude mcp add internal-api --transport http \
  https://mcp.internal.yourcompany.com/mcp

Servers are stored in your settings file and started automatically. Project-scoped servers go in .claude/settings.json (shared with team). Personal servers go in ~/.claude/settings.json.

The Essential Three

If you're starting from zero, these three MCP servers cover most development needs:

GitHub
PRs, issues, code review — the model can read issues and create PRs directly
Web Search
Documentation lookups, error research — the model finds answers instead of hallucinating
Playwright
Browser automation — the model can test web UIs and take screenshots

Don't Over-Tool

Remember Stripe's lesson from Chapter 9: they have ~500 tools available but give each agent a small default set. Every MCP server you add increases the model's tool selection burden. Add servers for tools you'll actually use regularly, not "just in case." You can always add more later.

Permission Tuning: The Trust Spectrum

Every AI coding tool balances two forces: the speed of autonomous action and the safety of human oversight. Permission configuration is how you set that balance for your project.

Claude Code Permission Rules

.claude/settings.json — tuned permissions for a typical project
{
  "permissions": {
    "allow": [
      "Bash(npm run test *)",
      "Bash(npm run lint *)",
      "Bash(npm run build)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(git log *)",
      "Edit",
      "Write"
    ],
    "deny": [
      "Bash(git push *)",
      "Bash(rm -rf *)",
      "Bash(npm publish *)",
      "Read(.env*)"
    ]
  }
}

Allow safe, reversible actions. Deny dangerous or irreversible ones. Everything else prompts for approval.

The evaluation order is: deny → ask → allow. A deny rule always wins, even if an allow rule matches too. This means you can be generous with allow rules and precise with deny rules.

Cursor's Approach

Cursor uses a different model: rule types with application modes. Rules marked alwaysApply: true are always active. Rules with globs patterns activate when matching files are involved. Rules with descriptions let the agent decide relevance.

.cursor/rules/api-conventions.md
---
description: "REST API design conventions for backend routes"
alwaysApply: false
globs: ["src/api/**/*.ts"]
---

- Validate all inputs with Zod schemas
- Return consistent error format: { error: string, code: number }
- Use middleware for auth, not inline checks
- Log all 5xx errors with request context

This rule only activates when Cursor works with files in src/api/ — it doesn't pollute context for frontend work.

When to Build from Scratch: The Agent SDK

Most developers should extend existing tools, not build new ones. But if you need a custom agent application — an internal code review bot, an automated migration tool, a specialized testing agent — the Claude Agent SDK gives you the same harness infrastructure that powers Claude Code.

The SDK provides the agentic loop (plan → act → observe → repeat), built-in tools (Read, Write, Edit, Bash, Grep, Glob), hooks for validation, MCP integration, and session management. You bring the specific workflow.

Minimal agent with the Claude Agent SDK (TypeScript)
import { query, ClaudeAgentOptions } from "@anthropic-ai/claude-agent-sdk";

const options: ClaudeAgentOptions = {
  allowedTools: ["Read", "Edit", "Bash"],
  systemPrompt: "You review TypeScript files for type safety issues.",
  maxTurns: 10,
};

for await (const message of query("Review src/auth/", options)) {
  console.log(message);
}

Ten lines to create an agent with file access, a focused system prompt, and bounded execution. The SDK handles the tool loop, context management, and model communication.

Extend First, Build Second

Before building a custom agent, ask: can I achieve this with a CLAUDE.md section + a custom command + a hook? Custom agents are powerful but require maintenance — you're taking on the responsibility of the harness. For most workflows, configuring an existing tool is faster, cheaper, and more reliable than building your own.

Putting It All Together

Here's what a complete harness configuration looks like for a real project. This is a mid-size TypeScript API with a React frontend — the kind of project where good configuration pays for itself within a week.

🔍 Inside the Harness: Your Project Before and After

Before: Out of the Box

You open Claude Code in your project directory. No CLAUDE.md, no custom commands, no hooks. Claude Code works — but it guesses your test runner, uses default formatting, doesn't know your architecture, and occasionally runs commands you'd rather it didn't.

After: Configured Harness

Here's what you've added:

  1. CLAUDE.md (project root) — build commands, code conventions, architecture overview. 40 lines.
  2. CLAUDE.md (in src/api/) — API-specific patterns, validation rules, error format. 15 lines. Loaded only when working on backend.
  3. .claude/commands/review.md — security-focused code review checklist. Team standard.
  4. .claude/commands/add-endpoint.md — scaffold a new API endpoint with test, route, and validation.
  5. PostToolUse hook — auto-formats with Prettier after every edit.
  6. PreToolUse hook — blocks rm -rf, git push --force, and reads of .env files.
  7. GitHub MCP server — Claude can read issues and create PRs directly.
  8. Permission rules — allow test/lint/build, deny push/publish/delete.

The result: Claude Code now follows your conventions on the first try. Reviews are consistent. Formatting is automatic. Dangerous commands are blocked before they execute. And you didn't write a single line of model training code — just configuration.

The Minimal Viable Harness

Feeling overwhelmed? Here's the simplest harness configuration that still delivers meaningful value. Do this first, add more later.

Step 1: Run claude /init to generate a starter CLAUDE.md from your project
Step 2: Edit it down to what matters — build commands, test runner, top 3 conventions the model gets wrong
Step 3: Create one custom command for your most common workflow
Step 4: Commit all of it to git so your team benefits too

That's it. Four steps, maybe 20 minutes. Everything else in this chapter is a refinement you can add when you need it.

What's Next: The Future of Harness Engineering

You've reached the end of this course, but harness engineering is just getting started. Here's where things are heading:

Self-improving harnesses. Today you manually tune your CLAUDE.md and hooks. Tomorrow, agents will analyze their own failure patterns and suggest harness improvements. Stripe's eval-driven approach (Chapter 9) is an early version of this.

Standardization. AGENTS.md is already governed by the Linux Foundation. MCP is becoming the universal tool protocol. We're moving toward portable harness configurations that work across tools — write once, run in Claude Code, Cursor, or Codex.

Autonomous agents. As harnesses get more sophisticated — better verification, tighter guardrails, more reliable tool use — the human-in-the-loop can move further from the keyboard. Not disappear entirely (we covered why in Chapter 9), but shift from approving every action to reviewing outcomes.

Model-harness co-design. SWE-1.5 showed that training model and harness together produces better results than optimizing each separately. Expect more tools to follow this path.

Try It Yourself

✍️ Build Your Complete Harness Configuration

This is the capstone exercise. Set up a real harness configuration for a project you work on daily.

  1. Write a CLAUDE.md (or AGENTS.md if you use Codex/Cursor). Include: build/test commands, your top 5 code conventions, and a brief architecture overview. Target: 30–50 lines. Run claude /init to start, then edit ruthlessly.
  2. Create one custom command for your most repeated workflow. Good candidates:
    • Code review with your team's checklist
    • Scaffolding a new component/endpoint/migration
    • Running the full test + lint + build pipeline
  3. Add one hook. The easiest high-value hook: auto-format after file edits with your project's formatter (Prettier, Black, gofmt).
  4. Test before and after. Give the AI a task you've done before without configuration. Then do the same task with your new configuration. Notice the difference.
  5. Commit everything to git. Your team gets the benefits on their next git pull.

Here's a starter file tree for Claude Code:

your-project/
  CLAUDE.md                        # Project instructions
  .claude/
    settings.json                  # Permissions + hooks + MCP
    commands/
      review.md                    # /project:review
      scaffold-endpoint.md         # /project:scaffold-endpoint

You don't need all five layers from this chapter on day one. Start with CLAUDE.md and one command. Add hooks next week. Add MCP servers when you hit a real need. The best harness configuration is one that grows with your workflow.
