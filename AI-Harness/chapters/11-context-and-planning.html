Chapter 11

Plan Mode & Context Strategy

When to think before acting — and when to forget on purpose

The Ten-Chapter Dilemma

Imagine you're generating a ten-chapter HTML course using Claude Code. You have a custom command — /generate-next-chapter — that reads your CLAUDE.md, checks which chapter is next, searches the web for fresh material, and writes 400–600 lines of HTML. You run it once. Beautiful chapter. You run it again. Another good one. By chapter five, you start wondering.

Should I enter plan mode before each generation? Would the extra thinking time produce better results? Or am I burning tokens on something the command already handles?

Should I type /clear between chapters? The context is getting heavy — five chapters' worth of file reads, tool calls, and conversation history. Is all that context helping or hurting? And what about the cache — am I throwing away something expensive when I clear?

These aren't just UI decisions. They're harness engineering decisions. Plan mode controls how the model reasons. /clear controls what the model knows. Together, they're two of the most powerful knobs you can turn on the Claude Code harness — and most developers never think about when to turn them.

What Plan Mode Actually Is

In Chapter 6, we saw plan mode mentioned as "read-only operations." Let's look at what that actually means for the harness.

When you press Shift+Tab to enter plan mode (or pass --mode plan when starting Claude Code), the harness does three things:

  1. Swaps the tool registry. Write, Edit, and destructive Bash commands are removed. The model keeps Read, Grep, Glob, and Agent — it can look at anything but touch nothing.
  2. Injects planning instructions. The system prompt gains a new section telling the model to explore, reason, and write a plan file — not to execute. Remember from Chapter 6: Claude Code has 110+ conditional prompt fragments. Plan mode activates a different subset.
  3. Creates a plan file. The model writes its approach to a markdown file you can review, edit, and approve before any code changes happen.

Think of it like putting on safety glasses before working in a machine shop. You can walk around, inspect every machine, measure parts, sketch designs on paper. But the power tools are locked out. You can't accidentally cut something.

The key insight: plan mode isn't just "the model thinking harder." It's a constraint mechanism. The harness physically prevents execution, which forces the model to reason about its approach instead of jumping straight to implementation. It's the difference between a contractor who sketches before building and one who starts swinging a hammer.

How to Toggle Plan Mode

In a session: Press Shift+Tab to toggle between plan and act modes. You can plan first, review the plan, then switch to act mode for implementation.

From the command line: Start with claude --mode plan to begin in plan mode. Or claude --mode auto to let the model decide when to plan.

When Plan Mode Earns Its Keep

Plan mode costs tokens — the model reads files, reasons through approaches, and writes a plan before doing any actual work. That's time and money. When is it worth it?

Complex multi-file changes. In Chapter 6 we noted the rule of thumb: if a change touches 3 or more files, use plan mode. Each additional file adds compounding decisions. Planning catches architectural mistakes before they're coded into existence.

Unfamiliar codebases. First day on a new project? Plan mode forces the model to explore before acting. It reads the directory structure, checks existing patterns, understands conventions — then proposes an approach that fits what's already there, instead of guessing.

Expensive-to-undo changes. Database migrations, API changes with downstream consumers, configuration that affects production environments. If a wrong implementation is hard to reverse, the planning tax pays for itself.

Ambiguous requirements. When the user's request could be interpreted multiple ways, plan mode surfaces those ambiguities. The plan file becomes a conversation checkpoint: "Here's what I think you mean — is this right?" Much better than implementing the wrong interpretation.

When Plan Mode Is Overhead

Not every task needs a blueprint. Sometimes the blueprint costs more than the building.

Well-defined repetitive tasks. Remember our ten-chapter scenario? If you have a /generate-next-chapter command that already specifies the structure, tone, CSS classes, and content requirements — the "plan" is baked into the command. Plan mode would re-derive what the command already states.

Single-file edits with clear patterns. "Add a new endpoint that follows the same pattern as the existing ones." The model can see the pattern in the file, replicate it, and it's done. Planning would just say "I'll follow the existing pattern" — not worth the round trip.

Tasks the model has just done successfully. If you just generated chapter 3 and it was perfect, the model understands the template. Chapters 4 through 10 don't need fresh plans — the pattern is established in context.

The Plan Mode Paradox

Plan mode is most valuable when you know the least about a task, and least valuable when you know the most. If you find yourself reading the plan and thinking "yes, obviously" — you probably didn't need plan mode. If you find yourself thinking "oh, I hadn't considered that" — it just saved you a failed implementation.

For the ten-chapter scenario: use plan mode for chapter 1 (establishing the template, making structural decisions). Skip it for chapters 2–10 (the pattern is set, the command is clear, repetition is the point).

What /clear Actually Does

Type /clear in Claude Code and the entire conversation vanishes. Not summarized. Not compressed. Gone.

Here's what happens mechanically:

  1. Conversation history is deleted. Every message you sent, every response, every tool call and result — erased from the context.
  2. The system prompt is rebuilt from scratch. Claude Code re-reads your CLAUDE.md files, reloads tool definitions, re-evaluates conditional prompt fragments for your platform and mode.
  3. You start with a fresh context window. The only tokens in the window are the system prompt, tools, and CLAUDE.md content — exactly what you'd get if you'd just opened a new terminal session.

What survives /clear: everything on disk. Your CLAUDE.md, settings, hooks, MCP servers, permission rules, and any files the model created or edited. The filesystem is permanent; the conversation is ephemeral.

What doesn't survive: discovered conventions ("I noticed your tests always use this pattern"), accumulated understanding of project structure, cross-references between files the model read, in-flight reasoning, and any preferences learned during the conversation.

One catch worth knowing: /clear looks nuclear, but every session is auto-saved to disk. claude --resume or /resume reloads the full transcript — including tool results — so /clear is reversible if you named the session first with /rename. We'll come back to this in section 11.

/clear vs /compact: Nuclear vs Surgical

/clear is a full reset — everything goes. /compact is a lossy summary — the harness condenses the conversation into a compact form, preserving key facts (what files were read, what decisions were made, what's in progress) while dropping the raw details. In Chapter 3, we saw that compaction is like reading meeting notes instead of the full transcript. /clear is like shredding the notes entirely.

The Prompt Cache and the 5-Minute Window

Here's something most developers don't realize: there's a cache involved, and /clear interacts with it in non-obvious ways.

Anthropic's API uses prompt caching. Every time Claude Code sends a request, it includes the system prompt, CLAUDE.md content, tool definitions, and conversation history. The first part of that request — the system prompt and tool definitions — tends to be identical across requests. This prefix gets cached on the server side.

The cache has a 5-minute TTL (time to live). If you make another request within 5 minutes using the same prefix, it's a cache hit — faster and cheaper. If you wait longer, the cached prefix expires and the next request pays full price.

What does this mean for /clear?

  • Clearing within 5 minutes: Your system prompt prefix may still be cached. The next request after /clear can hit the cache for the system prompt portion. You lose conversation history but keep the cache benefit on the static prefix.
  • Clearing after 5+ minutes: Full cache miss. The entire request — system prompt, tools, everything — is processed from scratch. This is the most expensive restart.
  • Not clearing at all: Each request includes the growing conversation history. The prefix stays cached (as long as you're within 5 minutes between requests), but you're paying for all those accumulated tokens on every subsequent request.

🔍 Inside the Harness: Before and After /clear

Before /clear — At Chapter 10 of a Sequential Generation

Your context window contains:

  • System prompt + CLAUDE.md + tool definitions: ~12K tokens
  • 9 rounds of "generate chapter" prompts: ~5K tokens
  • 9 chapter HTML outputs: ~45K tokens
  • File reads (style.css, index.html, previous chapters): ~30K tokens
  • Web search results accumulated over 9 chapters: ~20K tokens
  • Tool call metadata and model reasoning: ~15K tokens

Total: ~127K tokens in context. The model sees everything. Cross-references are easy. Consistency is automatic. But you're paying for 127K input tokens on every request, and older chapters are deep in the "lost in the middle" zone from Chapter 3.

After /clear — Fresh Start for Chapter 10

Your context window contains:

  • System prompt + CLAUDE.md + tool definitions: ~12K tokens

Total: ~12K tokens. Clean, cheap, focused. But the model doesn't know what tone was used, what CSS classes worked well, what cross-reference patterns were established, or what the previous 9 chapters even covered. It has to re-discover all of that from CLAUDE.md and the files on disk.

Context: Asset or Liability?

This is the core question. That accumulated context — is it helping or hurting?

The answer depends on one thing: how relevant the accumulated context is to the current task.

When Context Is an Asset

  • Consistency. If the model generated chapters 1–5, it has internalized the tone, terminology, and structure. Chapter 6 will be consistent without being told to "match the style of previous chapters."
  • Cross-references. The model knows what was covered in chapter 3 and can reference it naturally in chapter 7. Without that context, cross-references become generic or wrong.
  • Discovered conventions. Maybe the model noticed that your CSS has a .inside-look-box class and started using it effectively. After /clear, it needs to re-discover this from the CSS file.
  • Error correction. If you corrected the model's behavior in chapter 2 ("don't use Python examples, use JSON config snippets"), that correction carries forward. After /clear, the same mistake may reappear.

When Context Is a Liability

  • Irrelevant noise. When generating chapter 10 about building your own harness, the detailed web search results about MCP protocol history from chapter 4 are just noise. They take up tokens and attention without helping.
  • Attention degradation. The "lost in the middle" problem from Chapter 3. In a 127K-token context, the model pays strong attention to the beginning (system prompt) and the end (recent messages) but may gloss over the middle — where your chapter 3 corrections and chapter 5 conventions live.
  • Token cost. Every token in the context is billed on every request. 127K tokens of context means you're paying for 127K input tokens for every model response, even if 80% of that context is irrelevant to the current chapter.
  • Auto-compaction risk. Hit 95% context capacity and the harness auto-compacts (Chapter 3). You don't control what gets preserved and what gets dropped. A well-timed /clear or /compact is better than an emergency auto-compaction.

The Relevance Heuristic

If more than half the accumulated context is relevant to your current task — keep it. If less than half is relevant — /compact or /clear. If you're switching task domains entirely (frontend to database, generation to debugging) — always /clear.

Token Economics: What Context Actually Costs

Let's make this concrete with the ten-chapter scenario.

Strategy A: Never clear. Chapter 1 starts with ~12K tokens of context. By chapter 10, you're at ~127K. Each chapter generation uses more input tokens than the last. Total input tokens across all 10 chapters: roughly 700K. Plus you risk auto-compaction around chapters 7–8 depending on your model's window size.

Strategy B: Clear before every chapter. Each chapter starts at ~12K tokens. But the model re-reads style.css (~1.5K), index.html (~250 lines), and the previous chapter (~500 lines) each time. That's maybe 15–20K of re-discovery per chapter. Total input tokens: roughly 200K. Cheaper, but you lose consistency and cross-references.

Strategy C: The sweet spot. Run chapters 1–4 without clearing (context is still manageable, consistency matters while establishing patterns). /compact before chapter 5 (preserve key conventions, drop raw details). Run 5–7 with fresh compacted context. /clear before chapter 8 (CLAUDE.md captures the conventions by now). Run 8–10 fresh. Total input tokens: roughly 350K — a middle ground that preserves quality where it matters.

The Re-Discovery Tax

Clearing context isn't free. Every file the model needs to re-read, every convention it needs to re-discover, every correction it needs to re-learn — those are tokens too. The question isn't "is clearing cheaper?" It's "is carrying the old context more expensive than re-discovering what I need?" Sometimes yes, sometimes no.

The Ten-Chapter Playbook

Applying this to sequential course generation:

Chapters 1–3: No /clear, no plan mode. Context is light, you're establishing patterns, and consistency between early chapters matters most. (Exception: use plan mode for chapter 1 if the template isn't well-defined yet.)
Chapters 4–5: Run /compact before chapter 4 or 5. Context is getting heavy but the established patterns are valuable. Compaction preserves what matters.
Chapters 6–7: Consider /clear before chapter 6 or 7 — but only if your CLAUDE.md captures the conventions the model learned. If your CLAUDE.md is thin, use /compact instead.
Chapters 8–10: /clear is safe. By now, your CLAUDE.md and command template define the pattern. The model doesn't need conversation history from chapter 2 to write chapter 9.

What Gets Lost — and How to Protect It

Every /clear kills knowledge. The question is: does it kill knowledge that matters?

Types of Knowledge That Die

  • Learned corrections. "Don't use Python examples" — gone. The model may revert to its default behavior.
  • Discovered patterns. "This project uses msw for API mocking" — the model figured this out by reading test files. After /clear, it hasn't read those files yet.
  • Cross-file understanding. The model read 8 files and understands how they relate. After /clear, each file is unknown again.
  • Conversation-specific preferences. "Make the analogies more concrete" — a preference you stated in turn 3 that shaped every subsequent response.

The Protection Strategy: Write It Down

The filesystem survives /clear. The conversation doesn't. So the protection strategy is simple: move important knowledge from the conversation to the filesystem before clearing.

CLAUDE.md updated with session-discovered conventions
## Content Generation Rules
- Code examples: config snippets only (JSON, YAML, shell). NOT Python.
- Analogies: concrete and specific, not abstract. "Like a restaurant
  kitchen" not "like a system."
- Each chapter MUST include at least one .inside-look-box
- Use .stats-grid for any numerical evidence
- Cross-reference earlier chapters by number: "In Chapter 3, we saw..."
- Line target: 400-600 lines. Count before finalizing.

The Pre-Clear Checklist

Before typing /clear, ask yourself: did the model learn anything in this session that isn't already in CLAUDE.md? If yes, add it to CLAUDE.md first. Two minutes of updating CLAUDE.md saves twenty minutes of re-correcting the model.

What Happens to Tool Results — and How to Keep Web Research

You just spent eight tool calls and a generous web-search budget gathering material for a chapter. Then you hit /compact — or worse, /clear. Did the research survive?

The question matters because every refresh of a chapter starts with WebSearch and WebFetch dumping pages of raw HTML into the conversation. That research is the most expensive thing in the window, and the most repetitive thing to redo. Here's what the official Claude Code docs say happens to tool output — and four ways to keep it.

1. /compact summarizes tool output, it doesn't delete it

Auto-compaction is not a deletion event. The official docs describe it like this:

"Claude Code automatically optimizes costs through prompt caching (which reduces costs for repeated content like system prompts) and auto-compaction (which summarizes conversation history when approaching context limits)." — code.claude.com/docs/en/costs

In practice the bulky tool outputs — full WebFetch pages, large file reads — are the first things condensed. Your prompts, recent decisions, and the citation URLs the model wrote down stay; the raw 40KB HTML pages get replaced by a few sentences of summary. So when you compact mid-chapter, you don't lose the research — you compress the part of the research that has already done its job.

2. /compact takes a focus argument — and it's the lever most people miss

Bare /compact lets the model decide what to keep. /compact <instruction> tells it explicitly:

Steering compaction toward what matters
# Mid-chapter, after WebFetch dumped 5 pages
/compact Focus on verified facts and citation URLs from the web research

# Between two related chapters
/compact Focus on the conventions and CSS classes established so far

# Before switching from research to writing
/compact Focus on the chapter outline and source quotes

You can also bake this into CLAUDE.md so it applies to every compaction in the project. The official docs show the syntax:

CLAUDE.md — persistent compaction instructions
# Compact instructions

When compacting, preserve:
- Citation URLs and exact quotes from web research
- The chapter outline and section list
- CSS classes the model has decided to use
- User corrections from this session

Drop:
- Raw HTML page contents from WebFetch
- File contents already written to disk
- Search-result preambles

Reference: code.claude.com/docs/en/costs — Manage context proactively.

3. The subagent pattern: search without bloating context

If you want web research to never enter your main context in the first place, delegate it to a subagent. The subagent runs in its own context window, gets flooded with raw search results, and returns only a summary — the noise stays in the subagent's window, not yours. The official docs are explicit that this is the intended use:

"Use one when a side task would flood your main conversation with search results, logs, or file contents you won't reference again: the subagent does that work in its own context and returns only the summary." — code.claude.com/docs/en/sub-agents

And from the cost-management page:

"Delegate verbose operations to subagents. Running tests, fetching documentation, or processing log files can consume significant context. Delegate these to subagents so the verbose output stays in the subagent's context while only a summary returns to your main conversation." — code.claude.com/docs/en/costs

🔍 Inside the Harness: Web Research With and Without a Subagent

Without a subagent

Main session WebFetches five pages. Each is 8–15KB. The conversation now contains roughly 60KB of raw HTML, which the model re-reads on every subsequent turn. Cost over the next 10 turns: 60KB × 10 = 600KB of input tokens for content the model only needed once.

With a subagent

Main session asks an Explore subagent: "Research X, return verified facts with URLs, under 400 words." The subagent's context fills with the same 60KB of HTML. The subagent returns a 1KB summary. The main session sees only the 1KB. Cost over the next 10 turns: 1KB × 10 = 10KB — a 60× reduction. The raw research is gone, but the citations survive.

4. Saved sessions preserve tool results across /clear

This is the single most important thing missing from most users' mental model: /clear looks nuclear, but every session is auto-saved to disk, and --resume brings the whole conversation back, tool results included.

"Conversation Storage: All conversations are automatically saved locally with their full message history. ... Tool State: Tool usage and results from the previous conversation are preserved." — code.claude.com/docs/en/common-workflows

The trick is naming the session before you clear, so you can find it later:

The "/clear but keep the research" workflow
# After the chapter is written, before you reset
/rename ai-advanced-ch04-research

# Now safe to start fresh for the next chapter
/clear

# ...later, if you need that chapter's research again
claude --resume ai-advanced-ch04-research

# Or, from inside any session:
/resume ai-advanced-ch04-research

This is the literal answer to "can I clear context but keep the WebSearch?" — yes, by parking each chapter's session under a memorable name and pulling it back on demand. The cost-management page recommends this exact pattern:

"Use /clear to start fresh when switching to unrelated work. Stale context wastes tokens on every subsequent message. Use /rename before clearing so you can easily find the session later, then /resume to return to it." — code.claude.com/docs/en/costs

5. The filesystem is your zero-cost cache

If you'd rather not depend on resume (or you want research to outlive a single project), tell the model to dump verified facts to a markdown file before any reset:

Persisting research to disk
# Mid-chapter
"Before we continue, write the verified facts and citation URLs
to chapters/.research/04-prompt-caching.md so they survive /clear."

# Later, in a fresh session
"Read chapters/.research/04-prompt-caching.md and use those
citations for the rewrite."

Files cost zero context tokens until you @-reference them, and they survive both /compact and /clear. They're also reusable across courses — the same research file can feed an AI-Harness chapter and an AI-Economics chapter.

The Refresh-Chapter Playbook

A concrete recipe for sequential chapter-refresh workflows:

  1. Skip plan mode for routine refreshes — the existing chapter file is the plan. Reach for plan mode only when you're rewriting a chapter's structure, not its content.
  2. Compact mid-chapter after the WebSearch / WebFetch phase, before the writing phase: /compact Focus on verified facts and citation URLs. Raw pages collapse into citable notes; the cache stays warm for the writing turn.
  3. Between two related refreshes in the same course: /compact. Conventions and CSS choices stay; the prior chapter's research compresses to a summary.
  4. Between two unrelated refreshes (or when switching courses entirely): /rename <memorable-name> then /clear. The next refresh starts cheap; the old research is one --resume away if you ever need it.
  5. Anything you wish CLAUDE.md already knew, write to CLAUDE.md before clearing — that's the only knowledge that auto-loads in every future session, with zero re-discovery cost.

Putting It All Together

Plan mode and /clear aren't competing features. They're complementary tools for different problems.

Feature What It Controls Use When
Plan mode Reasoning depth — forces exploration before execution Complex tasks, unfamiliar codebases, 3+ file changes
/compact Context density — preserves key facts, drops raw details Context is heavy but conventions in conversation still matter
/clear Context scope — full reset to system prompt + CLAUDE.md Switching tasks, context is mostly irrelevant, CLAUDE.md captures conventions

The best practitioners combine them naturally. Plan mode for the first task in a new domain. Skip plan mode for repetitive tasks. /compact when context is heavy but still useful. /clear when switching gears entirely. Update CLAUDE.md before clearing to preserve what matters.

Try It Yourself

✍️ Audit Your Context Strategy

Take a multi-step task you do regularly and test the impact of context management.

  1. Establish a baseline. Run 3 related tasks (e.g., create 3 similar components or fix 3 related bugs) without /clear or plan mode. Note the quality and consistency of the output.
  2. Try plan mode first. Do the same task, but enter plan mode for the first one. Review the plan, then switch to act mode. Was the plan useful? Did it catch anything you wouldn't have?
  3. Try clearing between tasks. Do the same 3 tasks but type /clear between each one. Note what the model "forgets" — does it re-discover conventions? Does it make mistakes it didn't make before?
  4. Try the hybrid approach. No clear for tasks 1–2, then /compact before task 3. Compare quality and cost.
  5. Update your CLAUDE.md. Whatever the model keeps forgetting after /clear — add it to CLAUDE.md. That's the knowledge worth persisting.

The goal isn't to find the "right" strategy. It's to develop an intuition for when accumulated context helps and when it hurts — for your specific workflow.
