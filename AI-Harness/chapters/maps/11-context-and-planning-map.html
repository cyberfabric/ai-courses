Chapter 11

Plan Mode & Context Strategy

Memory Map
💡

The Big Idea

Plan mode and /clear are harness engineering knobs, not just UI features. Plan mode constrains tool access to force better reasoning. /clear resets context to start fresh. Both optimize the context window — knowing when to use each is the difference between burning tokens and investing them.

🛠️ The Three Context Controls

🔎
Plan Mode
Swaps tool registry to read-only. Model explores and writes a plan file. Can't edit or create files. A constraint mechanism that forces reasoning before action.
💥
/clear
Full context reset. Conversation history deleted. System prompt + CLAUDE.md rebuilt from scratch. Nuclear option — nothing from the conversation survives.
📦
/compact
Lossy summary of conversation. Key facts preserved, raw details dropped. Surgical option — meeting notes instead of the full transcript.
⚡
Prompt Cache
System prompt prefix cached server-side. 5-minute TTL. Cache hit = 90% cheaper. Clearing within 5 min keeps cache; after 5 min = full miss.

🧠 When to Use Plan Mode

Use Plan Mode ✅ Skip Plan Mode ❌
Multi-file changes (3+ files) Single-file edits with clear patterns
Unfamiliar codebases Repetitive tasks with established templates
Expensive-to-undo changes (migrations, APIs) Tasks the model just did successfully
Ambiguous requirements Well-defined commands (e.g., /generate-next-chapter)

Rule of thumb: plan mode is most valuable when you know the least about a task.

📚 The Ten-Chapter Playbook

  1. Chapters 1–3: Keep Everything

    Context is light, patterns are being established, consistency between early chapters matters most. Use plan mode only for chapter 1 if the template isn't defined yet.

  2. Chapters 4–5: /compact

    Context is getting heavy but conventions are valuable. Compaction preserves what matters, drops raw details like old web search results.

  3. Chapters 6–7: /clear (if CLAUDE.md is ready)

    Safe to clear if your CLAUDE.md captures discovered conventions. If CLAUDE.md is thin, use /compact instead to preserve learned patterns.

  4. Chapters 8–10: /clear freely

    CLAUDE.md and command templates define the pattern. The model doesn't need conversation history from chapter 2 to write chapter 9.

📖 Key Terms

Plan Mode
Harness constraint that swaps tool registry to read-only. Activated via Shift+Tab, --mode plan, or /plan prefix. Creates a plan file for review before execution.
/clear
Full conversation reset. Deletes all history, rebuilds system prompt from scratch. Filesystem survives; conversation doesn't.
/compact
Lossy conversation summary. Preserves key facts (files read, decisions made, conventions learned), drops raw tool output and detailed reasoning.
Prompt Cache
Server-side cache for the system prompt prefix. 5-minute TTL. Cache hits cost ~90% less than uncached tokens.
Context Window
The model's "working desk" — everything it can see at once. Grows with conversation. Auto-compacts at ~95% capacity.
Lost in the Middle
Attention degradation in long contexts. The model attends strongly to the beginning and end, less to the middle. From Chapter 3.
Re-Discovery Tax
The token cost of re-reading files and re-learning conventions after /clear. Clearing isn't free — it trades conversation tokens for re-discovery tokens.
Constraint Mechanism
A harness feature that limits what the model can do. Plan mode is a constraint mechanism; so are permission rules and hook-based blocks.

🧪 Quick Quiz

What does plan mode change about the harness?

It swaps the tool registry to read-only: Write, Edit, and destructive Bash are removed. The system prompt gains planning-specific instructions. A plan file is created for the model to write its approach. The model can explore and reason but cannot execute changes.

What survives /clear? What doesn't?

Survives: Everything on disk — CLAUDE.md, settings, hooks, MCP servers, files the model created. The system prompt is rebuilt from scratch. Doesn't survive: Conversation history, discovered conventions, learned corrections, cross-file understanding, accumulated preferences.

When should you use /compact instead of /clear?

When accumulated context is heavy but still useful — especially when the model has learned corrections or conventions that aren't in CLAUDE.md. /compact preserves key facts (files read, decisions made) while dropping raw details. Use /clear only when the old context is mostly irrelevant or you're switching task domains.

How does the 5-minute prompt cache TTL affect /clear timing?

If you /clear within 5 minutes of your last request, the system prompt prefix may still be cached — your next request is faster and cheaper on the static portion. After 5 minutes, it's a full cache miss and the entire prefix is re-processed. So if you're going to clear, do it quickly rather than waiting.

In a 10-chapter sequential task, what's the recommended strategy?

Chapters 1–3: no clear (building patterns, context is light). Chapters 4–5: /compact (heavy context, but conventions are valuable). Chapters 6–7: /clear if CLAUDE.md captures conventions. Chapters 8–10: /clear freely. Use plan mode only for chapter 1 if the template isn't well-defined.

📋 Key Takeaways

  • Plan mode is a constraint mechanism — it swaps tools to read-only, not just "makes the model think harder."
  • Use plan mode when the cost of wrong implementation exceeds the cost of planning (3+ files, unfamiliar code, expensive-to-undo).
  • /clear is a full reset. /compact is a lossy summary. Know which you need before reaching for either.
  • The prompt cache has a 5-minute TTL — clearing within 5 minutes keeps the cache; after 5 minutes is a full miss.
  • Accumulated context is an asset when >50% relevant (consistency, cross-references) and a liability when <50% relevant (noise, cost, attention decay).
  • The filesystem survives /clear; the conversation doesn't. Write important discoveries to CLAUDE.md before clearing.
  • For sequential tasks: keep context early (patterns forming), /compact in the middle (heavy but valuable), /clear late (CLAUDE.md has the patterns).
  • Clearing isn't free — account for the re-discovery tax of re-reading files and re-learning conventions.
