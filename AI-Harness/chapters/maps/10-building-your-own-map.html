The Big Idea

You don't need to build a harness from scratch — you're already customizing one. CLAUDE.md, custom commands, hooks, MCP servers, and permissions are five layers of harness engineering. Start with CLAUDE.md and add complexity only when you need it.

🏗️ The Five Layers of Harness Customization

  1. 📋 Project Instructions

    CLAUDE.md / AGENTS.md / .cursorrules. What the model should always know. Build commands, conventions, architecture. The highest-impact single change.

  2. ⚡ Custom Commands

    Markdown files in .claude/commands/ that become slash commands. Reusable workflows shared via git. $ARGUMENTS for dynamic inputs.

  3. ⚙️ Hooks

    Shell commands at lifecycle points: PreToolUse, PostToolUse, UserPromptSubmit, Stop. Turn "should" into "always." Auto-format, block dangerous commands, validate.

  4. 🔌 MCP Servers

    Custom tools for external systems: GitHub, databases, web search, internal APIs. Same tool-use mechanism as built-in tools. Start with 3: GitHub, Web Search, Playwright.

  5. 🛡️ Permission Tuning

    Allow safe/reversible actions. Deny dangerous/irreversible ones. Everything else prompts. Deny always wins, even over allow rules.

📋 CLAUDE.md: What to Include vs Skip

Include ✅ Skip ❌
Build, test, lint commands "Write clean, readable code" (already does this)
Non-obvious conventions (msw over jest.mock) "Use TypeScript" (can see tsconfig.json)
Architecture overview (where things live) "Files in /src contain source code" (obvious)
Error handling patterns (Result types, not try-catch) 500 lines of API docs (link to them instead)
Testing rules (vitest, not jest) "Follow SOLID principles" (generic noise)

For every line, ask: "Would removing this cause Claude to make a mistake?" If not, cut it.

📖 Key Terms

CLAUDE.md
Project instructions loaded into every Claude Code request. Three levels: global → project → directory. The single highest-impact harness customization.
AGENTS.md
Cross-tool alternative to CLAUDE.md. Supported by Codex, Cursor, Google Jules. Governed by Linux Foundation. Progressive disclosure with Skills.
Custom Commands
Markdown files in .claude/commands/ that become slash commands. Reusable workflows shared via git. Support $ARGUMENTS for dynamic input.
Hooks
Shell commands at lifecycle points (PreToolUse, PostToolUse, etc.). Deterministic guardrails: auto-format, block dangerous commands, validate. Exit code 2 = block.
MCP Servers
External tool integrations via Model Context Protocol. Project-scoped in .claude/settings.json, personal in ~/.claude/settings.json.
Claude Agent SDK
Programmable library with Claude Code's harness infrastructure. Use for custom agents when CLAUDE.md + commands + hooks isn't enough.
@import Syntax
CLAUDE.md can import other files with @path/to/file. Keeps CLAUDE.md lean while making detailed guidance available.
Progressive Disclosure
Broad guidance at top level, specific guidance in subdirectories. Load details only when the agent enters that area.

🧪 Quick Quiz

What should you put in CLAUDE.md vs leave out?

Include things the model can't infer: build commands, non-obvious conventions (msw over jest.mock), architecture rules, testing standards. Leave out what it can see: "use TypeScript" (sees tsconfig), "files in /src are source code" (obvious), generic advice like "follow SOLID." For every line, ask: would removing it cause a mistake?

When should you use a hook vs a CLAUDE.md instruction?

CLAUDE.md = advisory ("should follow but might not"). Hooks = deterministic ("always executes"). Use CLAUDE.md for style preferences and naming conventions. Use hooks for formatting, security checks, and blocking dangerous commands — things that must happen reliably.

What are the "essential three" MCP servers?

GitHub (PRs, issues, code review), Web Search (docs, error research), and Playwright (browser testing, screenshots). Start with these, add more only when you hit a real need. Don't over-tool.

How does the permission evaluation order work?

Deny → Ask → Allow. Deny rules always win, even if an allow rule matches. Be generous with allow rules (safe, reversible actions) and precise with deny rules (dangerous, irreversible actions). Everything else prompts for approval.

Should you build a custom agent or extend an existing tool?

Extend first, build second. Ask: can I achieve this with CLAUDE.md + a custom command + a hook? Custom agents require maintenance — you own the harness. For most workflows, configuring an existing tool is faster, cheaper, and more reliable.

📋 Key Takeaways

  • You're already a harness engineer. Every CLAUDE.md, custom command, and hook is harness configuration.
  • Five layers: project instructions → custom commands → hooks → MCP servers → permission tuning. Start simple.
  • CLAUDE.md is the highest-impact single change. Include what the model can't infer. Skip what it can see.
  • Commands = what to do on demand. Hooks = what must always happen. CLAUDE.md = what to always know.
  • Permission pattern: allow safe/reversible, deny dangerous/irreversible. Deny always wins.
  • Extend existing tools before building custom agents. CLAUDE.md + commands + hooks covers most workflows.
  • Minimal viable harness: claude /init, edit to essentials, one command, commit to git. 20 minutes.
  • The fundamentals are durable: context curation, tool design, verification, progressive disclosure, constraint-based architecture.
