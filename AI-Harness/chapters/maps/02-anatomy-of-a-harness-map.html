Chapter 2

The Anatomy of a Harness

Memory Map
💡

The Big Idea

Every harness is built from six components — System Prompt, Tools, Context, Memory, Guardrails, and Persistence. They form a cycle, not a pipeline. The sophistication of the harness matches the complexity of the tasks it handles.

🍳 The Restaurant Kitchen Analogy

👨‍🍳
Chef = LLM
Skilled, creative, does the actual cooking (text generation). But chaos without kitchen infrastructure.
📖
Recipe Book = System Prompt
Rules the chef reads before every shift. Tone, behavior, constraints — the employee handbook.
🔪
Kitchen Tools = Tool Layer
Knives, ovens, mixers. Without them, the chef can only describe food, not cook it.
🥕
Prep Station = Context
A prep cook selects the right ingredients from the warehouse. The chef never sees the full inventory.
📝
Order Notes = Memory
"Table 4 is allergic to nuts." Notes that fake continuity across orders and shifts.
🔍
Health Inspector = Guardrails
Rules the chef can't override. Permission checks before every action.
📋
Shift Log = Persistence
So the next shift knows what happened today. Files, git commits, saved state.

🧩 The Six Components

📜 1. System Prompt
Hidden instructions loaded before every interaction. Claude Code uses 110+ modular fragments that activate based on OS, mode, and context. The single biggest lever a harness has.
🔧 2. Tools
The model's hands. ~18 built-in tools in Claude Code (Read, Edit, Bash, Grep, etc.) + MCP extensions. The model requests a tool call; the harness executes it.
📦 3. Context Engineering
Curating what reaches the model. System prompt + CLAUDE.md + conversation history + tool results + system reminders. Uses compaction when the window fills up.
🧠 4. Memory & State
Three layers: working memory (context window), session state (harness tracking), persistent memory (files on disk). LLMs have zero memory — the harness fakes all of it.
🛡️ 5. Guardrails
Five-layer permission evaluation: Hooks → Deny Rules → Permission Mode → Allow Rules → User Approval. Controls what the model is allowed to do.
💾 6. Persistence
What survives when the session ends. Git commits, JSON history, progress files, config. The filesystem is the most reliable persistence strategy.

🔬 Inside the Harness: "Refactor to async/await"

  1. 📜 System Prompt Activates

    Thousands of tokens of hidden instructions load: tone, rules, capabilities, constraints.

  2. 📦 Context Assembled

    CLAUDE.md loaded, conversation history attached, system reminders injected.

  3. 🔧 Tools Registered

    Read, Edit, Bash, Grep, Glob, and MCP tools — all described with schemas.

  4. 🔍 Model Searches

    Calls Grep to find the function → reads the file → plans the refactor.

  5. 🛡️ Guardrails Gate Edits

    Read-only tools pass through. Edit triggers permission check — you approve.

  6. ✅ Model Verifies

    Runs npm test. If pre-approved in allow rules, executes immediately.

  7. 💾 State Persisted

    Conversation saved. Edited file on disk. Next session can pick up where this left off.

🧪 Quick Quiz

What are the six components of a harness?

System Prompt, Tools, Context Engineering, Memory & State, Guardrails, and Persistence.

In the restaurant analogy, what does the health inspector represent?

Guardrails — rules the model can't override. Permission checks that run before every tool call.

What are the five layers in Claude Code's permission evaluation?

Hooks → Deny Rules → Permission Mode → Allow Rules → User Approval.

Why does Claude Code use "system reminders" mid-conversation?

To combat "instruction fade" — models pay less attention to instructions buried in the middle of long contexts. Periodic re-injection keeps key rules top of mind.

What's the most reliable persistence strategy for coding agents?

The filesystem itself — git commits, progress files, config files. These survive crashes, context loss, and session endings.

📋 Key Takeaways

  • Six components: System Prompt, Tools, Context, Memory, Guardrails, Persistence — forming a cycle, not a pipeline.
  • The system prompt is the single biggest lever — change it and the model behaves completely differently.
  • Tools are the model's hands. Without them, an LLM can only talk about code.
  • LLMs have zero memory. Everything that feels like memory is the harness faking it across three layers.
  • Claude Code's five-layer permission check runs before every tool call: Hooks → Deny → Mode → Allow → User.
  • Not all harnesses need all six components. Simpler tools use fewer; complexity matches the task.
  • The filesystem is the most reliable persistence — git commits and files survive anything.
