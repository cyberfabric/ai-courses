Context Engineering

Memory Map
💡

The Big Idea

Context engineering is packing a suitcase — you can't take everything, so you pick what matters most for this trip. How you present information matters as much as what you present. Your prompt is ~5% of what the model sees; the harness provides the other 95%.

📦 What Fills the Context Window

  1. 📜 System Prompt (5K–10K tokens)

    Identity, rules, tone, tool descriptions. Conditional fragments activate based on OS, shell, mode.

  2. 📋 CLAUDE.md Chain

    Global → project → directory-level overrides. Loaded fresh every turn. Your most powerful context lever.

  3. 🔧 Tool Descriptions (2K–4K tokens)

    Every registered tool with name, description, and parameter schema. Deferred tools get lighter descriptions.

  4. 💬 Conversation History

    Every message, tool call, and result. Accumulates fast — by turn 20, easily 100K+ tokens.

  5. 🔔 System Reminders

    Key instructions re-injected at intervals to combat "instruction fade." Near the end, where attention is highest.

  6. ✏️ Your Prompt (~5%)

    Finally, your actual message. Often the smallest piece in the assembled context.

📖 Key Terms

Context Window
The fixed amount of text a model can process at once — its "working desk."
Context Engineering
The art of curating what information reaches the model. The harness's most critical job.
CLAUDE.md / AGENTS.md
User-written project context files loaded automatically into every request.
System Reminders
Key instructions re-injected at intervals to fight attention fade in long conversations.
Compaction
Summarizing older conversation turns to free up context space. Lossy but necessary.
Lost in the Middle
Models pay less attention to information in the middle of long contexts. U-shaped attention curve.
Hashline
Adding line numbers and structured delimiters to code, dramatically improving edit success rates.
Progressive Disclosure
Loading context incrementally (top-level first, details on demand) instead of all at once.

🧪 Quick Quiz

What percentage of the context window does your prompt typically fill?

About 5%. The other 95% is system prompt, CLAUDE.md, conversation history, tool descriptions, and system reminders.

What is the "lost in the middle" problem?

Models show a U-shaped attention curve — they pay the most attention to the beginning and end of context, and the least to the middle. Performance drops 30%+ for info buried in the middle.

What should you put in a CLAUDE.md file?

Things the model can't infer from the code: custom build commands, non-obvious conventions, architecture decisions, specific tooling choices. Skip things like "this is a TypeScript project" — the model can see that.

What did the Hashline technique prove?

That how you present information matters as much as what you present. Just adding line numbers and file headers improved coding success from 6.7% to 68.3% — same model, same code, different formatting.

What triggers context compaction in Claude Code?

Auto-compaction triggers at ~95% capacity. You can also trigger it manually with /compact. It preserves task state and drops individual tool call details.

📋 Key Takeaways

  • Your prompt is ~5% of the context. The harness assembles the other 95%: system prompt, CLAUDE.md, history, tools, reminders.
  • Every unnecessary token degrades performance. Curate the smallest, highest-signal context.
  • CLAUDE.md is your most powerful lever — include what the model can't infer, skip what it can see.
  • System reminders combat "lost in the middle" by re-injecting key instructions near the end of context.
  • Compaction summarizes old turns at ~95% capacity. It's lossy — the model may need to re-read files.
  • The Hashline technique (line numbers + delimiters) improved coding success 10x. Presentation matters as much as content.
  • Different harnesses use different strategies: model-driven retrieval (Claude Code), automatic indexing (Cursor), real-time awareness (Windsurf), progressive disclosure (Codex).
