Chapter 4

Tools and Tool Use

Memory Map
💡

The Big Idea

LLMs can't do anything except generate text. Tools are how they act on the world. The model proposes a structured request, the harness executes it, and the result goes back. Without tools, an LLM is a phone with no apps.

🔄 The Tool Use Loop

User Message
→
Model Reasons
→
Tool Call Request
→
Harness Validates & Executes
→
Result Back to Model
→
Reason Again (or finish)

A single user request can trigger 2, 5, or 20+ loops before the model produces a final response.

🔬 Example: "Add input validation"

  1. 🔍 Grep — Find the code

    Grep(pattern="function createUser", type="ts") → src/services/user.ts:28

  2. 📖 Read — See the function

    Read(file_path="src/services/user.ts", offset=20, limit=30) → 30 lines of TypeScript

  3. ✏️ Edit — Add validation

    Edit(file_path=..., old_string=..., new_string=...) → Permission check → approved → applied

  4. ✅ Bash — Run tests

    Bash(command="npm test -- --grep createUser") → "4 tests passed"

🗂️ Tool Categories

📁
File System
Read, Write, Edit, Glob, Grep. Navigate, read, and modify code. The foundation of coding agents.
💻
Terminal
Bash / shell execution. Run tests, builds, git commands, system operations.
🌐
Web Access
WebSearch, WebFetch. Search the web for docs, fetch URLs for reference.
🧠
Code Intelligence
Semantic search, linting. Understand code structure beyond plain text.
🤖
Agent Management
Spawn sub-agents for parallel work. Isolated contexts for complex tasks.
🔌
External Services (MCP)
GitHub, Slack, databases — anything connected via MCP servers.

⚡ Sequential vs. Parallel Tool Calls

Aspect Sequential Parallel
How it works One tool at a time, wait for each Multiple tools in one response
Round trips N round trips for N calls 1 round trip for N calls
Latency Sum of all call durations Duration of the slowest call
Speedup Baseline 2–4x for I/O-heavy tasks
When to use Calls depend on previous results Calls are independent of each other

🎯 Tool Selection & Scaling

🎲
Selection = Text Generation
No special "tool selector." The model reads descriptions and picks the best match — just like generating any text.
📝
Descriptions Matter
Vague descriptions → misuse. Precise descriptions → accurate selection. Tool naming is harness engineering.
⚠️
Too Many Tools = Confusion
Accuracy drops from 43% (4 tools) to 2% (51 tools). More isn't always better.
💤
Deferred Loading
Register tools as "deferred." Model uses ToolSearch to discover them on demand. 85%+ token savings.

📖 Key Terms

Tool Use (Function Calling)
The mechanism where a model outputs a structured request instead of text, asking the harness to execute an action.
Tool Schema
A structured definition of a tool: name, description, and parameter types. Directly influences how well the model uses it.
Tool Use Loop
The cycle of: model reasons → requests tool → harness executes → result returns → model reasons again.
Parallel Tool Calls
Requesting multiple independent tools in one response. Reduces round trips and latency 2–4x.
Deferred Loading
Registering tools without full descriptions to save tokens. Model discovers them via ToolSearch when needed.
MCP (Model Context Protocol)
Open standard for connecting AI tools to external systems. Three primitives: Tools, Resources, Prompts.
Tool Selection
How the model picks which tool to use — it's just another text generation decision, guided by descriptions.

🧪 Quick Quiz

Can an LLM execute a tool call directly?

No. The model generates a structured request. The harness intercepts it, checks permissions, executes the action, and sends the result back. The model proposes, the harness disposes.

Why does tool selection degrade with more tools?

Tool selection is just text generation — no special selector module. With 50+ tools, accuracy drops from 43% to 2% because tools get buried in a "blind spot." Deferred loading solves this.

How much faster are parallel tool calls vs sequential?

2–4x faster for I/O-heavy tasks. Latency drops from the sum of all calls to the duration of the slowest one. Same cost, fewer round trips.

What are MCP's three primitives?

Tools (executable functions with side effects), Resources (read-only data sources), and Prompts (predefined templates exposed as slash commands).

Why did Cursor name tools to match Unix commands?

Models were trained on Unix documentation. Using familiar names (like rg for ripgrep) significantly improved tool adoption — the model is more likely to use a tool it "recognizes."

📋 Key Takeaways

  • LLMs can only generate text. Tools are how they act — the model requests, the harness executes.
  • Tool use is a loop: reason → call tool → get result → reason again. One request can trigger 20+ loops.
  • Tool descriptions directly affect selection quality. Good naming and descriptions are harness engineering.
  • More tools ≠ better. Accuracy drops sharply past 30–50 tools. Use deferred loading to scale.
  • Parallel tool calls give 2–4x speedup for independent operations — zero extra cost.
  • MCP is the USB-C of AI tools — an open standard for connecting external systems to any harness.
  • File system + terminal tools are what make coding agents possible. Everything else builds on top.
