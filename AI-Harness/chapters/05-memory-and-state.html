Chapter 5

Memory and State

LLMs have no memory — how harnesses create the illusion of continuity

In Chapter 4, we gave the model hands (tools). Now let's talk about what happens between uses of those hands. You ask Claude Code to fix a bug. It reads a file, edits it, runs the tests. Ten minutes later, you ask a follow-up question. The model seems to "remember" the bug, the file, and the fix.

It doesn't. LLMs have zero memory. Not "limited memory." Not "short-term memory." Zero. Every single API call starts from a completely blank slate. The model has no idea who you are, what you said five seconds ago, or what project you're working on.

Everything that feels like memory is the harness faking it. And it fakes it brilliantly.

The Desk, the Filing Cabinet, and the Warehouse

Think of memory in layers, like storage at an office:

  • Your desk (working memory) — What's right in front of you. Limited space, but instant access. This is the context window.
  • The filing cabinet (session state) — Nearby, organized by project. Things you pull out when needed. This is what the harness tracks in memory during a session.
  • The warehouse archive (persistent storage) — In another building. Takes effort to retrieve, but survives forever. This is files on disk, git history, and persistent memory systems.

Each layer trades off speed for durability. Your desk is fast but gets cleared when you go home. The warehouse survives a fire but takes 20 minutes to find anything. The harness manages all three layers simultaneously.

Working Memory: The Context Window

This is the memory you interact with most. The conversation you see on screen — your messages, the model's responses, tool calls and their results — all of it lives in the context window.

Here's the trick: the harness replays the entire conversation with every single API call. When you send your 15th message, the harness doesn't send just that message. It sends messages 1 through 15, all tool calls and results, plus the system prompt and project context. The model processes the whole thing from scratch, as if it's seeing the conversation for the first time.

The Movie Analogy

Imagine an actor who has amnesia between every take. To film scene 15, the director plays scenes 1 through 14 before rolling the camera. The actor performs scene 15 perfectly — but only because they just watched the whole movie. That's how LLM "memory" works. The harness is the director, replaying the tape every time.

This works beautifully — until the conversation gets too long and the tape doesn't fit in the player anymore. That's when compaction kicks in (we covered this in Chapter 3). Older turns get summarized, detail is lost, and the model works from meeting notes instead of the full transcript.

Practical implication: if you need the model to remember something specific across a very long conversation, don't rely on the conversation history alone. Write it to a file (CLAUDE.md, a progress file, or a comment in the code). Files survive compaction. Conversation turns don't.

Session State: What the Harness Tracks

Beyond the conversation itself, the harness maintains its own state — things the model can't see directly but that affect how the harness behaves:

  • Token usage — How much of the context window is consumed. Triggers compaction at ~95%.
  • Permission grants — Which tools you've approved during this session.
  • Active MCP connections — Which external servers are connected and available.
  • Modified files — Which files the model has changed (tracked by the harness, not the model).
  • Cost tracking — API calls made, tokens consumed, estimated cost.
  • Session metadata — Start time, session ID, working directory.

This state lives in the harness process, not in the model. If the harness crashes, this state is lost — but the files on disk and the git history survive. This is why the filesystem is the most reliable memory backend.

An important nuance: session state isn't sent to the model directly. The model doesn't see a variable called tokensUsed: 142000. Instead, the harness uses that state to make decisions — like triggering compaction when tokens are high, or blocking a tool when permissions haven't been granted. The model experiences the effects of session state without seeing the state itself.

How Chat Apps Fake Memory

ChatGPT feels like it remembers you. You open a new conversation, and it greets you by name and knows your preferences. How?

Two mechanisms, both harness-level:

1. Conversation Replay

Within a single chat, ChatGPT does exactly what we described: replays the full conversation with every message. The model "remembers" your first message because the harness sent it again.

2. Saved Memories

Across chats, ChatGPT uses a separate "memory" system. When the model notices something it thinks is worth remembering — your name, your job, your preferences — it saves a fact to an external database. On the next conversation, these facts are injected into the system prompt.

The key insight: ChatGPT's memory is intentionally vague. It stores "user prefers Python over JavaScript" rather than exact code snippets. It learns high-level facts, not precise details. This is a deliberate design choice — storing exact templates would be fragile and create data management headaches.

ChatGPT's "memory" and "chat history" are separate features with separate controls in settings. Memory stores facts you explicitly or implicitly want retained. Chat history allows the model to search past conversations for context. You can disable either one independently. If you're wondering why ChatGPT remembers something surprising, check both.

The Filesystem IS the Memory

Here's one of the most important insights about coding agent memory: the filesystem is the most reliable memory system. Not a database. Not an in-memory store. Files on disk.

Think about it: when Claude Code edits your code, that edit persists on your filesystem. If the process crashes, the context window is wiped, and the session state is gone — but the file changes are still there. When you start a new session, the model can read the files and reconstruct what happened.

This is why sophisticated harnesses use progress files for long-running work: 

# claude-progress.txt

## Completed
- [x] Set up Express server with TypeScript
- [x] Added PostgreSQL connection with Drizzle
- [x] Created user CRUD endpoints
- [x] Added JWT authentication middleware

## In Progress
- [ ] Rate limiting on public endpoints
- [ ] Input validation with Zod schemas

## Blockers
- Redis not configured yet for rate limiting store

## Key Decisions
- Using Drizzle over Prisma for type-safe queries
- JWT stored in httpOnly cookies, not localStorage

A new session (or a new agent) reads this file and immediately understands the project state. No conversation replay needed. No memory system. Just a file.

This is perhaps the most practical memory tip in the entire course: if something is important enough that a future session needs it, write it to a file. CLAUDE.md for permanent project context. A progress file for long-running features. Code comments for decisions that future developers (human or AI) need to understand. Files are the only memory layer that survives everything — crashes, compaction, new sessions, even switching to a different AI tool entirely.

Git: The Version-Controlled Memory

Git is even more powerful than progress files because it captures what changed and why. A new agent can run git log --oneline -20 and see the last 20 commits. Each commit message is a memory entry.

Anthropic's guide for long-running agents recommends a specific pattern:

  • Initializer agent — Sets up the environment, creates progress files, makes an initial git commit
  • Working agents — Each session reads the progress file and git log, does work, commits with descriptive messages, updates the progress file
  • HANDOFF.md — For feature branches, a handoff document at .agents/handoffs/{branch}/{session}.md records what was done and what's next

The result: an agent can work across 10 sessions on a complex feature, and each new session picks up exactly where the last one left off — by reading files, not by having memory.

The Memory Hierarchy at a Glance

Layer What It Holds Lifetime Survives Crash?
Context Window Conversation history, tool results, system prompt Current turn (replayed each time) No
Session State Token count, permissions, MCP connections Current session No
Auto Memory Learned preferences, patterns, insights Across sessions (per project) Yes
Project Files CLAUDE.md, progress files, HANDOFF.md Permanent (until deleted) Yes
Git History Commits, diffs, branch state Permanent (version-controlled) Yes

The pattern is clear: the higher the layer, the faster and more detailed but less durable. The lower the layer, the slower to access but virtually permanent. Good harness engineering uses all five layers — and good users learn which layer to trust for which kind of information.

Cross-Session Memory Compared

Claude Code

Auto Memory: Silently learns build commands, debugging insights, code style preferences from your sessions. Stored at ~/.claude/projects/.

CLAUDE.md: Manual project instructions loaded every turn. Most reliable — you control exactly what's remembered.

Session summaries: Automatic background summaries stored for future reference.

Scope: Project-level. Memories stay with the project directory.

ChatGPT

Saved Memories: Facts extracted from conversations and stored externally. Injected into system prompt on next chat.

Chat History: Past conversations searchable for context. Separate from saved memories.

Style: Intentionally vague — stores preferences, not exact templates.

Scope: User-level. Memories follow you across all projects and topics.

Windsurf (Cascade)

Autonomous Memories: Cascade generates memories on its own when encountering useful context. Stored at ~/.codeium/windsurf/memories/.

Manual Memories: You can ask "Create a memory of..." for critical decisions.

End-of-session review: Prompts you to confirm what was captured.

Scope: Workspace-level. Memories are tied to the workspace, not the repo.

Cursor

Git Worktrees: Each parallel agent works in an isolated worktree with its own branch and full git history.

Session IDs: Saved in worktree metadata, allowing resumption with full context.

.cursorrules: Project-level instructions (similar to CLAUDE.md).

Scope: Per-worktree/branch. Agents in different worktrees don't see each other's work.

The Compaction Challenge

We covered compaction in Chapter 3, but let's revisit it through the lens of memory. When the context window fills up and the harness compacts, you lose memory. Specifically, you lose the detailed working memory that let the model reason precisely about earlier turns.

What this looks like in practice:

Before compaction

You: "Fix the auth bug" Model: [Reads auth.ts, finds null check missing on line 42, edits line 42, runs tests — all details in context] You: "What did you change on line 42?" Model: "I added a null check for the user object before accessing user.email"

Full details available — the model can answer precisely.

After compaction

You: "What did you change on line 42?" Model: "Based on the conversation summary, I fixed an auth bug in auth.ts. I can read the file again to see the current state."

Details compacted away — the model knows WHAT happened but not the specifics.

The model isn't wrong — it's working from a summary. It offers to re-read the file, which is the right move. But the precision is gone.

This is why experienced users develop habits to work with compaction rather than against it:

  • Keep conversations focused on one task — start a new session for a new topic
  • Write critical information to files (CLAUDE.md, progress files, code comments)
  • Use /compact manually at natural breakpoints, not when you're mid-task
  • Commit work frequently so git history serves as a memory backup
  • Rely on conversation history for precise details after 50+ turns
  • Expect the model to remember exact line numbers from 30 turns ago

Inside the Harness: What Happens When You Hit the Context Limit

You've been working with Claude Code for 40 turns. You've read 8 files, made 12 edits, run tests 5 times. The context window is at 92% capacity. You send another message. Here's what happens:

  1. The harness checks token count. Your message plus the accumulated history would push the context to 97%. That's past the compaction threshold (~95%).
  2. Compaction triggers automatically. The harness sends the full conversation to a summarization step, asking: "Compress this conversation while preserving: completed tasks, current state, files modified, active constraints, and next steps."
  3. A summary replaces the history. The 180,000-token conversation becomes a 5,000-token summary. Everything before this point is gone from working memory.
  4. Your new message is added to the fresh context window, which now contains: system prompt + CLAUDE.md + compacted summary + your new message. Plenty of room.
  5. The model processes your request using the summary as context. If it needs details that were compacted away, it calls tools to re-read files or check git history.
  6. The conversation continues from this new baseline, accumulating history again until the next compaction.

The model doesn't know compaction happened. It sees a summary that looks like conversation context, processes your request, and continues working. The transition is seamless — but the precision of earlier turns is gone. The harness traded detail for continuity.

Try It Yourself

Test the Memory Boundaries

Pick your AI coding tool and run this experiment:

  1. Early reference test: Start a conversation. In your first message, mention a specific, unusual detail: "My favorite color is chartreuse and my dog's name is Fibonacci." Continue the conversation about coding for 20+ turns. Then ask: "What's my dog's name?" Does it remember?
  2. Cross-session test: Close the conversation. Start a completely new one. Ask the same question. Does it remember across sessions? (It will in ChatGPT if memory is on. It won't in most coding tools unless you wrote it to a file.)
  3. Compaction test: In Claude Code, have a long conversation (read several large files, make many edits). When compaction triggers (you'll see the context shrink), ask about something specific from early in the conversation. Notice how the model's answer changes from precise to summary-level.

The takeaway: Once you see the boundaries, you'll naturally adapt. You'll start writing critical info to files, keeping conversations focused, and treating the filesystem as your real long-term memory.
