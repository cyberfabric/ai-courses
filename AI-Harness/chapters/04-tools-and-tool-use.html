Chapter 4

Tools and Tool Use

How LLMs call functions — the mechanism that turns a text generator into an agent

Chapter 3 covered what the model sees. This chapter is about what the model does. And the answer is deceptively simple: LLMs can't do anything except generate text. Every action you've ever seen an AI tool take — reading a file, editing code, running tests, searching the web — happened because the harness gave the model tools.

Apps on a Smartphone

Think of tools like apps on a smartphone. Your phone's processor can do basic things on its own — calculate, display text. But it's the apps that make it genuinely useful: a camera app takes photos, a maps app gives directions, a messaging app sends texts.

The phone doesn't take photos itself. It sends a signal to the camera hardware, which takes the photo and returns the result to the phone for display. Tools work the same way: the model sends a structured request to the harness, the harness executes the action, and the result goes back to the model.

Without tools, an LLM is a phone with no apps — it can have a conversation, but it can't do anything.

What Tool Use Actually Is

Tool use (also called "function calling") is a mechanism where the model outputs a structured request instead of regular text. The model says: "I'd like to call this tool with these parameters." The harness intercepts, executes, and returns the result.

Here's the actual shape of a tool call from the Claude API: 

// Model's response (tool call request)
{
  "type": "tool_use",
  "id": "toolu_01A09q90qw90lq917835lq9",
  "name": "Read",
  "input": {
    "file_path": "/src/auth.ts",
    "limit": 50
  }
}

// Harness sends back (tool result)
{
  "type": "tool_result",
  "tool_use_id": "toolu_01A09q90qw90lq917835lq9",
  "content": "1\timport express from 'express';\n2\t..."
}

Notice: the model didn't read the file. It asked for the file to be read. The harness did the reading, checked permissions, and sent back the contents. This distinction matters — the model proposes, the harness disposes.

The Tool Use Loop

Tool use isn't a single call. It's a loop that repeats until the model has enough information to give a final answer.

User Message → Model Reasons → Tool Call → Harness Executes → Result Back → Model Reasons Again

That last step is the key: the model reasons again with the tool result in context. It might decide to call another tool, or it might have enough to answer. A single user request might trigger 2, 5, or 20 tool calls in a loop before the model produces a final response.

When you ask Claude Code to "refactor the login function," the loop might go:

  1. Grep for the function → finds src/auth.ts:42
  2. Read the file around line 42 → sees the callback-based code
  3. Edit the file → replaces callbacks with async/await
  4. Bash npm test → tests pass
  5. Final response: "Done. Refactored login to async/await. Tests pass."

Five tool calls, one user message. The harness orchestrated all of it.

Tool Schemas: Teaching the Model What's Available

The model doesn't magically know what tools exist. The harness describes each tool using a schema — a structured definition that includes the tool's name, what it does, and what parameters it accepts. 

{
  "name": "Bash",
  "description": "Executes a bash command and returns its output. Use for running tests, builds, git commands, and system operations. Avoid using for tasks that have dedicated tools (use Read instead of cat, Grep instead of grep).",
  "input_schema": {
    "type": "object",
    "properties": {
      "command": {
        "type": "string",
        "description": "The command to execute"
      },
      "timeout": {
        "type": "number",
        "description": "Optional timeout in milliseconds (max 600000)"
      }
    },
    "required": ["command"]
  }
}

That description is doing heavy lifting. It's not just documentation — it directly influences when and how the model uses the tool. A vague description leads to misuse. A precise description leads to accurate tool selection.

Tool descriptions are one of the most underrated pieces of harness engineering. Cursor discovered that naming tools to match Unix commands the model was trained on (like rg for ripgrep) significantly improved tool adoption. The model is more likely to use a tool it "recognizes." Good tool naming and descriptions can improve tool selection as much as a prompt improvement.

Categories of Tools

Not all tools are created equal. They fall into clear categories based on what they do:

Category Examples What They Do
File System Read, Write, Edit, Glob, Grep Navigate, read, and modify code
Terminal Bash, shell execution Run commands, tests, builds
Web Access WebSearch, WebFetch Search the web, fetch URLs
Code Intelligence Semantic search, linting Understand code beyond text
Agent Management Agent (subagent), Task tools Spawn sub-agents for parallel work
External Services MCP tools (GitHub, Slack, DB) Interact with external systems

The first two categories (file system and terminal) are what make coding agents possible. Without them, the model can only talk about code. With them, it can read, write, and test code.

How the Model Picks a Tool

Here's something that surprises people: tool selection is just another text generation decision. The model doesn't have a special "tool selector" module. It reads the tool descriptions, considers the user's request, and generates the tool call that seems most appropriate — the same way it generates any other text.

This means tool selection quality depends on:

  • Tool descriptions — Clear, specific descriptions lead to better selection
  • Number of tools — More tools means more confusion (more on this in a moment)
  • System prompt guidance — The harness can steer the model: "Use Read instead of cat, Grep instead of grep"
  • Training data — Models trained with specific tools will use them more naturally
More tools isn't always better. Research shows tool selection accuracy drops significantly after 30–50 available tools. One study found accuracy plummeting from 43% with 4 tools to just 2% with 51 tools on a scheduling task. Tools get buried in a "blind spot" and the model overlooks the right one.

Parallel Tool Calls

Sometimes the model needs to do multiple independent things at once. Instead of calling tools one by one (sequential), some harnesses let the model request multiple tools in a single response.

Sequential (slow)

Turn 1: Read file A → wait → result Turn 2: Read file B → wait → result Turn 3: Read file C → wait → result Total: 3 round trips

Parallel (fast)

Turn 1: Read file A + Read file B + Read file C → all execute concurrently → all results return Total: 1 round trip

The latency drops from the sum of all calls to the duration of the slowest one. For I/O-heavy tasks (reading multiple files, making API calls), this is a 2–4x speedup with zero additional cost.

Claude Code uses parallel tool calls extensively. When it needs to explore your codebase, it might spawn multiple Read and Grep calls simultaneously rather than waiting for each one to finish.

Deferred Tool Loading: When You Have Too Many Tools

Every tool description costs tokens. A typical tool definition is 200–500 tokens. If you have 50 tools, that's 10,000–25,000 tokens just for tool descriptions — before any conversation even starts.

Now imagine connecting 5 MCP servers with 400 total tools. That could eat 67,000+ tokens — a third of your entire context window, gone before you type a word.

The solution: deferred loading. The harness registers tools with a flag that says "don't load this tool's full description until it's needed." Instead, the model gets access to a ToolSearch tool that can discover relevant tools on demand. 

// Tool registered with deferred loading
{
  "name": "github_create_issue",
  "description": "Create a new GitHub issue",
  "defer_loading": true
}

// Model discovers it via ToolSearch when needed
ToolSearch(query: "create github issue")
→ Returns full schema for github_create_issue
→ Model can now call it

Result: 85%+ token reduction in tool descriptions. The model loads only the 3–5 tools it actually needs for each task, not the entire catalog.

MCP: The Universal Tool Connector

Imagine if every phone manufacturer required different charger cables. That was the state of AI tool integration until recently — every harness had its own way of connecting external tools.

The Model Context Protocol (MCP) is the USB-C of AI tools. It's an open standard (introduced by Anthropic in late 2024, adopted by OpenAI, Cursor, and others in 2025) that standardizes how AI applications connect to external systems.

MCP defines three primitives:

  • Tools — Executable functions with side effects (write files, call APIs, send messages)
  • Resources — Read-only data sources (database schemas, document libraries, configs)
  • Prompts — Predefined templates exposed as slash commands

In practice, you configure MCP servers in your settings, and the harness connects to them automatically: 

// .claude/settings.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "..." }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": { "DATABASE_URL": "..." }
    }
  }
}

Now the model can create GitHub issues, query your database, and do whatever else these servers expose — through the same tool-use mechanism as built-in tools. Same loop, same permission checks, same harness orchestration.

Cursor

Built-in tools: 15+ specialized tools including semantic code search, rg (ripgrep), read_lints.

Extension: MCP integration + subagents (Explore, Bash, Browser) with isolated contexts.

Key design: Unix-aligned tool names match model training data. Custom embedding model for semantic search.

Codex

Architecture: JSON-RPC App Server with bidirectional communication. Items, turns, and threads as core abstractions.

Key design: Rich session semantics (streaming diffs, approval flows) designed for IDE integration. Chose JSON-RPC over MCP for richer interaction patterns.

Windsurf

Architecture: Cascade uses real-time awareness tracking (edits, terminal, clipboard) as implicit tool inputs.

Extension: Built-in tool suite focused on code editing and terminal. MCP support added.

Key design: Persistent agentic approach — tools operate within a continuous context rather than stateless calls.

Inside the Harness: A File Read, an Edit, and a Test

Let's trace the complete tool-use loop for a simple task: reading a file, editing it, and running tests.

You type: add input validation to the createUser function

Round 1 — Finding the code:
The model generates: Grep(pattern="function createUser", type="ts")
The harness checks permissions (Grep is read-only — auto-approved), executes the search, returns: src/services/user.ts:28. Total round-trip: ~200ms.

Round 2 — Reading the file:
The model generates: Read(file_path="src/services/user.ts", offset=20, limit=30)
Harness executes, returns 30 lines of TypeScript. The model now sees the function and its parameters.

Round 3 — Making the edit:
The model generates: Edit(file_path="src/services/user.ts", old_string="function createUser(data) {", new_string="function createUser(data) {\n if (!data.email) throw new Error('Email required');\n if (!data.name) throw new Error('Name required');")
Harness checks permissions — Edit modifies a file. Depending on your permission mode, it either auto-approves or prompts you. You approve. The edit is applied.

Round 4 — Verifying:
The model generates: Bash(command="npm test -- --grep createUser")
Harness checks permissions: npm test matches your allow list. Runs the tests. Returns output: "4 tests passed." The model summarizes the result for you.

Four tool calls, one user message. The model decided which tools to use, generated structured requests, and reasoned about each result. The harness validated permissions, executed each action, and fed results back. Neither could have done this alone.

Try It Yourself

Catalog Your Tool's Actions

Use your AI coding tool for a real task — something that involves multiple steps (fix a bug, add a feature, write tests). As you work, keep a running list:

  1. List every action the tool takes beyond generating text. Did it read files? Search? Run commands? Edit code?
  2. Categorize each action using the table above: file system, terminal, web access, code intelligence, agent management, or external service.
  3. Count the tool calls. How many tool calls did a single request trigger? Were any done in parallel?
  4. Spot the permission checks. Which actions did the tool ask permission for? Which did it do automatically?

If you're using Claude Code, try running with --verbose to see tool calls as they happen. You'll see the exact loop: model reasons, calls a tool, gets a result, reasons again.

Bonus: If you have MCP servers configured, ask the model to do something that requires an MCP tool. Watch how it discovers and uses the tool through the same mechanism as built-in ones.
