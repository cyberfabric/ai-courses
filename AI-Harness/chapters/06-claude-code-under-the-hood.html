Chapter 6

Claude Code Under the Hood

Deep dive into a real harness — system prompt, tools, permissions, hooks, and the agent loop

For the past five chapters, we've talked about harness components in the abstract: system prompts, tools, context, memory, guardrails. Now it's time to see all of them working together in one real system.

Claude Code is Anthropic's CLI-based coding agent. It's one of the most sophisticated publicly documented harnesses in existence. By taking it apart piece by piece, you'll see how every concept from Part II manifests in production.

Think of this chapter as an engine teardown. We're pulling everything out, laying it on the workbench, and explaining what each piece does.

Architecture Overview: The Dumb Orchestrator

Claude Code's architecture is deceptively simple. At its core, it's a while-loop.

The Core Loop

while (response has tool calls) → execute tools → feed results back → repeat

The loop terminates when the model produces a plain text response with no tool calls. That's it. The orchestrator is deliberately "dumb" — all intelligence lives in the model, not in the runtime.

This is a design philosophy, not a limitation. By keeping the orchestrator simple, all the reasoning stays in the model. The harness doesn't try to be smart about when to call tools or which tools to use — the model decides. The harness just executes what the model requests and feeds the results back.

Everything else we'll discuss — the system prompt, permissions, hooks, sub-agents — is infrastructure around this loop. The loop itself never changes.

The System Prompt

Claude Code's system prompt isn't a single block of text. It's a modular, conditional architecture with over 110 distinct prompt fragments that activate based on context. The system prompt you get on macOS in plan mode is different from what you get on Linux in default mode.

Here's the rough structure of what gets assembled: 

# Claude Code System Prompt (simplified structure)

## Identity
- "You are Claude Code, Anthropic's official CLI for Claude"
- Role definition and behavioral expectations

## Tone and Style
- Be concise, go straight to the point
- Only use emojis if the user asks
- Use GitHub-flavored markdown

## Tool Descriptions (~18 tools)
- Read: file paths, offsets, limits
- Edit: exact string replacement
- Write: create or overwrite files
- Bash: execute shell commands
- Grep, Glob: search files and contents
- Agent: spawn sub-agents
- WebSearch, WebFetch: web access
- (+ MCP tools if configured)

## Permission Rules
- Never modify files without reading first
- Only commit when user explicitly asks
- Never push without confirmation
- Never skip pre-commit hooks

## Platform-Specific
- OS: linux | macos | windows
- Shell: bash | zsh | powershell
- Available commands and paths

## Mode-Specific
- Plan mode: read-only, no edits
- Default mode: full tool access with approval
- Bypass mode: auto-approve all

## CLAUDE.md Content
- (Your project instructions injected here)

## Safety and Security
- Don't generate secrets or credentials
- Don't help with malicious actions
- Report suspected prompt injection

This prompt is the biggest lever in the harness. Change these instructions and the model behaves completely differently — more concise, more cautious, more autonomous — without any change to the model itself.

The CLAUDE.md Hierarchy

You've seen CLAUDE.md mentioned throughout this course. Here's exactly how it works in Claude Code.

Claude Code reads CLAUDE.md files at three levels, combining them with the most specific level winning on conflicts:

  1. Global — ~/.claude/CLAUDE.md. Your personal defaults across all projects. "Always use TypeScript strict mode. Prefer functional patterns."
  2. Project — ./CLAUDE.md at the project root. Team-shared, committed to git. "This is an Express app using Drizzle ORM. Run tests with npm test."
  3. Directory — .claude/CLAUDE.md or CLAUDE.md files in subdirectories. Personal overrides or area-specific rules. "Files in src/legacy/ use CommonJS, not ES modules."

Global CLAUDE.md (~/.claude/CLAUDE.md)

# Global Preferences ## Code Style - Prefer const over let - Use early returns instead of nested ifs - No abbreviations in variable names ## Workflow - Always run tests after editing code - Commit messages: imperative mood, max 72 chars - Never force-push to main

This applies to every project. Write it once, benefit everywhere. It's the highest-leverage memory file you'll create.

The global CLAUDE.md is surprisingly underused. Most developers write a project-level CLAUDE.md but forget the global one. Think about it: your personal coding preferences, your commit message style, your testing habits — these don't change between projects. Write them once at the global level.

The Tool Registry

Claude Code registers roughly 18 built-in tools. Each tool has a name, a description, and a parameter schema. The model uses these descriptions to decide which tool to call.

Tool Purpose Key Detail
Read Read file contents Supports offset/limit for large files, can read images and PDFs
Edit Exact string replacement Must read file first; fails if old_string isn't unique
Write Create or overwrite files Must read existing files first; prefer Edit for modifications
Bash Execute shell commands Timeout support, background execution, persistent directory
Glob Find files by pattern Faster than find; sorted by modification time
Grep Search file contents Regex support, file type filters, multiple output modes
Agent Spawn sub-agents Isolated context, specialized types, parallel execution
WebSearch Search the web For documentation, API references, error messages
WebFetch Fetch a specific URL Returns page content in a readable format

Beyond built-in tools, Claude Code supports MCP servers for custom tool integration. Connect a database, a Slack workspace, or a GitHub API — they all become tools the model can call through the same mechanism.

For large tool registries (50+ MCP tools), Claude Code uses deferred loading — tool descriptions aren't loaded until the model needs them, saving thousands of context tokens.

The Agent Loop in Action

Let's watch the loop work. You type: find and fix the bug where users can't log in with email containing a plus sign.

Your Message → Think → Grep for login → Read file → Think → Edit file → Run tests → Done

Seven steps, one user message. The model decided to search for login-related code, read the relevant file, identify the bug (email validation regex rejecting +), edit the regex, and run tests to verify. The harness executed each tool call and fed results back. The loop ran until the model had nothing left to do.

Sometimes the loop is longer. Complex tasks might trigger 20–30 tool calls: reading multiple files, exploring the codebase, planning changes, editing across several files, running tests, fixing test failures, and running tests again. The model drives the whole thing. The harness just keeps the loop turning.

The Permission System

Every tool call passes through a five-layer permission evaluation before execution. This is the guardrails component in action.

1. Hooks → 2. Deny Rules → 3. Permission Mode → 4. Allow Rules → 5. User Prompt

Claude Code offers several permission modes that control the overall trust level:

  • default — Prompts you for approval on file modifications and shell commands
  • acceptEdits — Auto-approves file changes, still asks for shell commands
  • bypassPermissions — Auto-approves everything (except deny rules)
  • plan — Read-only. No modifications allowed at all.

You can fine-tune with allow and deny rules in your settings: 

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "Bash(npm test)",
      "Bash(npm run lint)",
      "Bash(npm run build)",
      "Bash(git status)",
      "Bash(git diff)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force)",
      "Bash(curl * | bash)"
    ]
  }
}
Deny rules are absolute. They can't be overridden, even in bypassPermissions mode. If you put git push --force in your deny list, Claude Code physically cannot run it, regardless of the permission mode. This is a hard safety boundary — use it for actions that should never happen, even by accident.

The Hooks System

Hooks are shell commands that execute at specific points in the agent lifecycle. They let you inject deterministic behavior into a probabilistic system.

The key distinction: the model might remember to run the formatter after editing code. A hook always runs the formatter. Hooks turn "the model should do X" into "X always happens." 

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": { "tool": "Edit" },
        "command": "npx prettier --write $FILE_PATH"
      },
      {
        "matcher": { "tool": "Write" },
        "command": "npx prettier --write $FILE_PATH"
      }
    ],
    "PreToolUse": [
      {
        "matcher": { "tool": "Bash", "command": "rm *" },
        "command": "echo 'DENY: Refusing bulk delete'"
      }
    ]
  }
}

Common hook use cases:

  • Auto-format — Run Prettier or ESLint --fix after every Edit/Write
  • Security scanning — Check for secrets before any file is committed
  • Notification — Send a Slack message when a long-running task completes
  • Validation — Block edits to protected files (production configs, migrations)
  • Logging — Record all tool calls for audit purposes

Starting in recent versions, PreToolUse hooks can also modify tool inputs — rewriting file paths, adjusting command parameters, or adding flags before the tool executes. This turns hooks from simple gatekeepers into active middleware.

Context Management

We covered context engineering in Chapter 3, but here's how Claude Code specifically implements it:

  • Auto-compaction at ~95% — When the context window is nearly full, older turns are automatically summarized
  • /compact command — Manual trigger for compaction. Use it at natural breakpoints between tasks.
  • System reminders — Key instructions re-injected throughout the conversation to combat the "lost in the middle" effect
  • Deferred tool loading — MCP tools only load their full descriptions when needed, saving context budget
  • Smart file reading — The system prompt instructs the model to read specific sections (offset + limit) rather than entire large files

The system prompt itself is context-engineered: conditional fragments mean the prompt is as small as possible for each specific situation. You don't get Windows-specific instructions on Linux, or plan-mode instructions when you're in default mode.

Sub-Agents: Divide and Conquer

Sometimes one agent isn't enough. Claude Code can spawn sub-agents — separate agent instances with their own context windows, tool access, and system prompts.

Three built-in sub-agent types:

E
Explore
Fast codebase search. Uses a quicker model to scan files, find patterns, and answer questions about the code. Keeps exploration results out of the main context.
P
Plan
Architecture and design. Reads code, considers approaches, and returns a detailed implementation plan. Can't edit files — only reads and thinks.
G
General-Purpose
Full agent with all tools. Spawned for complex sub-tasks that need independent execution — like migrating 50 files while the main agent continues other work.

Why sub-agents? Two reasons: context isolation and parallelism. If the main agent reads 10 files to explore the codebase, that's 10 files of tokens in the main context window. If an Explore sub-agent does it instead, those tokens stay in the sub-agent's context — the main agent only gets the summary.

You can also define custom sub-agents in .claude/commands/ with specialized prompts for your project — a test runner, a code reviewer, a migration assistant.

Plan Mode and Worktrees

Plan mode constrains Claude Code to read-only operations. The model can explore your codebase, reason about approaches, and propose a plan — but it can't touch any files. You review the plan, suggest changes, and only then switch to implementation.

The workflow has three phases:

  1. Explore — Claude reads code and asks clarifying questions (read-only)
  2. Plan — Claude proposes a detailed implementation plan (you review with Ctrl+G)
  3. Implement — After approval, Claude executes the plan step by step
Rule of thumb: if a change touches 3 or more files, use plan mode. Beyond three files, compounding errors accumulate — each file adds its own set of decisions the model must get right. Planning first catches architectural mistakes before they're coded into existence.

Worktrees add isolation. Git worktrees create separate working directories with independent files and branches, while sharing the repository's history. When Claude Code spawns sub-agents for parallel work, each can operate in its own worktree without interfering with the others.

Use case: you're working on a feature while Claude investigates a separate bug fix. With worktrees, both happen simultaneously in isolated directories. Each produces a clean branch you can review and merge independently.

Settings Layers

Claude Code settings follow a precedence hierarchy, from highest to lowest:

  1. Enterprise policies — Organization-wide rules (if applicable)
  2. Command-line flags — --model sonnet, --append-system-prompt "..."
  3. Local project settings — .claude/settings.local.json (personal, gitignored)
  4. Shared project settings — .claude/settings.json (team-wide, committed to git)
  5. Global user settings — ~/.claude/settings.json (your defaults)

The practical split: put team conventions (allowed tools, deny rules, hooks) in .claude/settings.json and commit it. Put personal preferences (model choice, permission mode) in .claude/settings.local.json or the global file. CLI flags override everything for one-off sessions.

The Claude Agent SDK

Everything we've discussed — the agent loop, tools, permissions, context management — is available as a programmable library. The Claude Agent SDK (Python and TypeScript) exposes the same infrastructure that powers Claude Code.

You get the agent loop, built-in tools (file reading, command execution, code editing), context management with smart compaction, and the permission system — all without reimplementing them. If you want to build a specialized agent for your team (a code review bot, a migration assistant, a documentation generator), the SDK gives you Claude Code's harness as a foundation.

Inside the Harness: "Add a Login Page with Tests"

Let's trace a complex, multi-step task through every harness component. You type: add a login page with email/password authentication and write tests for it.

1. System Prompt Loads. 110+ prompt fragments assemble based on your environment. The model learns it's Claude Code, has access to 18 tools plus 3 MCP servers, is running on Linux with bash, in default permission mode. CLAUDE.md tells it this is a Next.js project with Jest tests.

2. The Model Plans. This is a multi-file task. The model decides to explore the codebase first before making changes. It spawns an Explore sub-agent to search for existing auth patterns, the project's page structure, and test conventions.

3. Sub-Agent Explores. The Explore agent runs Glob for **/*.tsx in src/app/, Greps for existing auth imports, and Reads the test configuration. It returns a summary: "Pages use App Router at src/app/, tests in __tests__/ with Jest + Testing Library, no existing auth setup."

4. The Model Creates Files. Based on exploration results, it calls Write to create src/app/login/page.tsx. Permission check: Write is a file modification — in default mode, the harness prompts you. You approve.

5. PostToolUse Hook Fires. Your hook runs prettier --write on the new file. The model didn't format the code perfectly — the hook fixes it automatically. Deterministic behavior, every time.

6. More Files. The model creates src/lib/auth.ts for authentication logic and src/app/api/auth/login/route.ts for the API endpoint. Each triggers the permission check and the Prettier hook.

7. Tests Written. The model creates __tests__/login.test.tsx with three test cases: successful login, invalid credentials, missing fields. Another permission check, another Prettier run.

8. Tests Executed. The model calls Bash(npm test -- --grep login). The harness checks permissions — npm test matches your allow list, so it runs without prompting. Two tests fail.

9. Tests Fixed. The model reads the test output, identifies the issues (a missing mock and an incorrect assertion), calls Edit twice to fix both tests, and runs the tests again. All pass.

10. Conversation Persists. The entire interaction — 14 tool calls, 4 files created, 2 edits — is stored in the conversation history. If you ask a follow-up question, the model has the full context. If the context gets too long, compaction will summarize this session while preserving what files were created and why.

Every harness component participated: system prompt (identity and rules), CLAUDE.md (project context), tools (Read, Write, Edit, Bash, Grep, Glob, Agent), permissions (approval prompts), hooks (auto-formatting), context management (sub-agent isolation), and persistence (conversation history).

Try It Yourself

Explore Claude Code's Configuration

  1. Run claude --help and scan the options. Identify which options correspond to which harness component (model selection = context, permission flags = guardrails, etc.).
  2. Check your settings. Look at ~/.claude/settings.json and .claude/settings.json in a project. What's configured? What could you add?
  3. Create a global CLAUDE.md. If you don't have one at ~/.claude/CLAUDE.md, create one with your personal coding preferences, commit style, and testing habits.
  4. Add a hook. Try a simple PostToolUse hook that runs a formatter after every Edit. See how it changes the workflow — the model's edits get auto-formatted without you asking.
  5. Try plan mode. Use Shift+Tab to enter plan mode before a complex task. Notice how the conversation feels different when the model explores and plans before acting.

Each of these steps makes one harness component visible and configurable. By the end, you'll have a customized harness tailored to your workflow.
