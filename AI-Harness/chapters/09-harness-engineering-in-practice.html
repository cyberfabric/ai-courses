Harness Engineering in Practice

From understanding to results — the techniques that actually move the needle

The Biggest Lever You're Not Pulling

You've spent eight chapters learning what harnesses are, how they work, and how every major AI coding tool builds one. Now for the punchline: improving the harness consistently delivers bigger gains than waiting for the next model.

This isn't a vague claim. Teams have measured it.

52.8% → 66.5%
LangChain improved GPT-5.2-Codex scores by changing only the harness — no model upgrade
42% → 78%
One team nearly doubled success on programming benchmarks — same model, different harness
69%
Of frequent AI coding users report deployment problems with AI-generated code

That last number is the AI Velocity Paradox: teams ship code faster with AI, but testing, securing, and deploying hasn't kept up. Raw model output without proper harness engineering creates as many problems as it solves. The harness is where you fix that.

From Prompt Engineering to Harness Engineering

Think of it like tuning a race car. Prompt engineering is shouting instructions at the driver through the radio — useful, but limited. Context engineering is planning which tires, fuel strategy, and route to use — now you're thinking about the whole race. Harness engineering is redesigning the car itself — the transmission, the suspension, the telemetry. Same driver, dramatically different lap times.

Each phase subsumes the previous one. You still write good prompts and curate context, but you're also engineering the entire system around the model.

The Three Phases of Working with LLMs

Prompt Engineering (2022–2023): Craft the perfect instruction. "Let's think step by step."

Context Engineering (2024): Curate what information reaches the model. CLAUDE.md, retrieval, compaction.

Harness Engineering (2025–2026): Design the entire runtime — tools, verification, permissions, eval loops. The model is one component, not the whole system.

The Hashline Breakthrough: When Format Beats Intelligence

In early 2026, researcher Can Boluk published a finding that stunned the industry: changing only the edit tool format — not the model, not the prompt, not the tools — improved 15 different LLMs by 5–14 percentage points on coding benchmarks.

Let that sink in. No retraining. No new model. Just a different way of presenting code to the model and asking it to describe edits.

6.7% → 68.3%
Grok Code Fast 1 — tenfold improvement from a format change alone
5–14 pts
Average improvement across 15 different LLMs
61% fewer
Output tokens for Grok 4 Fast — less wasted work, fewer retry loops

Here's what happened. The major AI coding tools use different formats for code edits:

  • Patch format (used by Codex): the model writes a unified diff. Works well for OpenAI models, but failure rates hit 50% for non-OpenAI models.
  • str_replace (used by Claude Code): the model reproduces the exact original text and the replacement. Reliable but brittle — one whitespace mismatch and the edit fails.
  • Hashline (the new approach): each line gets tagged with a short content hash. The model references lines by hash instead of reproducing text verbatim.
Hashline format — the model sees tagged lines
11:a3|function validateUser(input) {
12:f7|  if (!input.email) {
13:b2|    return { valid: false, error: "Missing email" };
14:c9|  }
15:d1|  return { valid: true };
16:e5|}

The model says "replace lines 12:f7 through 14:c9 with..." instead of reproducing the original code character-by-character. Hash mismatches reject edits before corruption happens.

It's Not a Silver Bullet

Hashline effects are language-dependent. Python actually penalizes hashline (indentation-sensitive languages struggle), TypeScript is roughly neutral, and Rust benefits significantly. A second independent study confirmed there's no universal winner — which is exactly why you need to measure your specific use case.

The big takeaway isn't "use hashline." It's that the same model can perform dramatically differently depending on the harness. Format is a harness decision, not a model decision. And it can be worth more than a model upgrade.

Stripe's Minions: 1,300 AI PRs Per Week

If hashline shows what a single harness tweak can do, Stripe shows what happens when you engineer the entire system. Their internal agent platform, Minions, produces over 1,300 pull requests per week containing zero human-written code — on a codebase that handles $1 trillion in annual payments.

This isn't a research experiment. It's production engineering at one of the most demanding codebases in the world.

1,300+
AI-generated PRs per week
~500
Tools available via MCP — but agents get a small default set
3M+
Tests in CI — the safety net that makes autonomy possible

The key architectural insight is Blueprints: workflows that alternate between deterministic steps and agentic steps. "Checkout the repo" is hardcoded. "Run the linter" is hardcoded. "Figure out how to fix the bug" gets full agentic freedom. This means the boring stuff always happens, and the creative stuff gets room to breathe.

Their tool management is equally deliberate. Stripe's internal "Toolshed" provides around 500 tools via MCP — but each agent starts with a small default set. More tools are added only when the task requires them. Flooding an agent with 500 tools doesn't make it capable; it makes it confused.

The verification approach is pragmatic: at most two rounds of CI. If code doesn't pass after the second push, it goes back to a human. Every PR gets human review before merging. Partial success counts — a 70% correct PR that an engineer can polish in twenty minutes is still a significant win.

Stripe's Key Insight

"Start with your developer environment, your test infrastructure, and your feedback loops. If those are solid, agents will benefit from them." The secret to Stripe's success isn't the AI model — it's the infrastructure the model runs on.

Measuring What Matters: Evals and Benchmarks

If you're not measuring, you're guessing. The eval-driven approach treats harness changes exactly like code changes: measure before, make the change, measure after. No vibes-based assessment.

SWE-Bench: The Industry Standard (and Its Limits)

SWE-Bench asks coding agents to solve real GitHub issues. The "Verified" leaderboard has become the headline number everyone cites: the top models score around 81% (Claude Opus 4.5 leads at 80.9%).

But there's a catch. SWE-Bench Pro — a harder, private variant designed to resist data contamination — tells a very different story. The best models score around 23%. That gap is enormous, and it suggests that headline benchmark numbers dramatically overstate real-world agent capability.

OpenAI actually stopped reporting Verified scores after their own audit found models could reproduce verbatim patches from the dataset. Contamination is real, and it makes public benchmarks unreliable as the sole measure of quality.

On Cursor Bench — harness matters more than model size

Cursor Bench: Measuring What Developers Actually Do

Cursor built their own evaluation suite that's closer to reality. Cursor Bench uses "Cursor Blame" to trace committed code back to the agent request that produced it, creating authentic query-solution pairs from real Cursor sessions — not synthetic puzzles.

The most striking finding: Haiku can match or exceed GPT-5 on Cursor Bench. A much smaller, faster model performs comparably when the harness is tuned for it. This is the clearest evidence that harness quality can compensate for model size.

The lesson: build your own evals based on your actual tasks. Public benchmarks tell you about general capability. Private evals tell you whether your harness is getting better.

The Tuning Playbook: Four Levers That Work

You don't need to build Stripe's infrastructure to improve your results. Here are four levers that consistently deliver measurable improvements, ordered from easiest to most involved.

1. System Prompt Iteration

The instructions you give the model shape everything it does. Small changes in system-level instructions produce measurable differences in success rates. The key is to be specific and opinionated rather than vague and generic.

CLAUDE.md — specific instructions that improve test generation
## Testing Conventions
- Use vitest, not jest
- Every test file must import from @testing-library/react
- Mock API calls with msw, never with jest.mock()
- Test user behavior, not implementation details
- Name test files: [component].test.tsx

This isn't prompt engineering in the old sense — it's engineering the persistent context that shapes every interaction. Test different instructions, measure how often the agent gets it right, iterate.

2. Tool Design

Better tool descriptions improve tool selection accuracy. The model picks tools based on their names and descriptions — if those are ambiguous, the model picks wrong.

MCP tool definition — clear description guides correct selection
{
  "name": "search_codebase",
  "description": "Semantic search across the repository. Use this when you need to find functions, classes, or patterns by meaning rather than exact text. Returns ranked results with file paths and surrounding context. Prefer this over grep when the exact identifier name is unknown.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": { "type": "string", "description": "Natural language description of what to find" },
      "maxResults": { "type": "number", "default": 10 }
    }
  }
}

Notice the explicit guidance: "prefer this over grep when..." That one sentence prevents a category of tool-selection errors.

3. Context Curation

What you exclude matters as much as what you include. Large codebases contain thousands of files that are irrelevant to any given task. Excluding noise is free improvement.

.claude/settings.json — excluding irrelevant context
{
  "contextExclusions": [
    "node_modules/**",
    "dist/**",
    "*.min.js",
    "**/*.generated.ts",
    "coverage/**",
    "*.lock"
  ]
}

Every file you exclude frees up tokens for files that actually matter. Lock files, generated code, and build artifacts consume context without providing value.

4. Output Format Control

Structured outputs reduce parsing errors and keep the agent on track. When the model knows exactly what format to produce, it wastes fewer tokens on formatting decisions.

System prompt snippet — constraining output for migration tasks
When generating database migrations:
- Output exactly one migration file
- Use the format: YYYYMMDD_HHMMSS_description.sql
- Include both UP and DOWN sections
- Never modify existing migration files
- After writing, run: npm run migrate:check

The Boring Improvements That Work

The most effective harness improvements are rarely glamorous. Better error messages when tools fail. Clearer descriptions on existing tools. Smarter default file exclusions. Explicit naming conventions. These "boring" changes consistently deliver 2–5x reliability gains in agentic workflows. Don't chase novelty — chase consistency.

Model-Harness Co-Design: The Frontier

Most teams treat model selection and harness design as separate decisions: pick a model, then build around it. Windsurf (now part of Cognition) asked a different question: what if you train them together?

SWE-1.5 was trained using end-to-end reinforcement learning within the Cascade agent harness. The model didn't just learn to generate code — it learned to use specific tools, follow specific workflows, and recover from specific failure modes. Model, harness, and inference system were designed as one unit.

Anti-Patterns: The Traps That Waste Good Models

Knowing what works is half the battle. Knowing what doesn't work saves you from expensive mistakes. Here are the four most common traps.

1. Context Stuffing

Dumping your entire knowledge base into one giant context file feels helpful. It's not. More noise means more context rot, more things silently ignored, and worse performance on the things that actually matter.

CLAUDE.md — context stuffing (don't do this)
# Project Info
[... 2,000 lines covering every API endpoint, database schema,
deployment procedure, team roster, coding standards,
historical decisions, and lunch menu ...]

The model pays less attention to information buried in the middle of long contexts. A 2,000-line CLAUDE.md means most of your instructions are effectively invisible.

The fix: keep your top-level context short and focused. Use directory-level CLAUDE.md files for area-specific instructions. Progressive disclosure — provide details only when the agent enters that area of the codebase.

2. Tool Sprawl

Registering every possible tool "just in case" overwhelms the model's selection ability. Stripe's lesson is instructive: they have ~500 tools available, but each agent starts with a small default set. More tools get added only when the task demands them.

3. Over-Prompting

System prompts so long and contradictory that the model ignores parts of them. If your system prompt contains competing instructions ("always write tests" and "keep changes minimal" for a complex refactor), the model will silently pick one and ignore the other. Keep instructions consistent and prioritized.

4. Ignoring Verification

The Self-Evaluation Fallacy

Models cannot reliably evaluate their own work. They exhibit overconfidence even when outputs are broken. Asking an agent "did you do this correctly?" is like asking a student to grade their own exam — the answer is almost always "yes." Use external verification: linters, type checkers, test suites, CI pipelines. If your harness doesn't verify, your harness doesn't work.

The core principle behind all of these: constraining the agent's solution space dramatically increases its productivity. More rules, better guardrails, and tighter boundaries paradoxically produce better outcomes. Freedom without structure is chaos.

Inside the Harness: Before and After

🔍 Same Model, Same Task, Different Harness

A team wants their AI agent to generate database migration tests. Here's what changes when they improve the harness — with the exact same model underneath.

Before: The Kitchen-Sink Approach

  1. System prompt: Generic — "You are a helpful coding assistant. Follow best practices."
  2. Tools available: All 200+ registered tools, including ones for UI testing, deployment, and documentation generation.
  3. Context: The entire src/ directory (12,000 files) indexed and available.
  4. Verification: The agent is asked "does this look correct?" after generating tests.
  5. Result: 34% success rate. The agent frequently picks wrong tools, writes tests against incorrect schemas, and self-reports success on broken test files.

After: Engineered Harness

  1. System prompt: Specific — "You generate migration tests using our vitest setup. Always read the migration file first. Use the db_schema tool to check current table state. Test both UP and DOWN migrations. Follow examples in tests/migrations/."
  2. Tools available: 8 curated tools — file read, file write, db_schema, run_migration, run_tests, git_diff, list_migrations, rollback_migration.
  3. Context: Only migrations/, tests/migrations/, and the database schema file. Exclusions for everything else.
  4. Verification: Harness automatically runs npm run test:migrations after generation. Agent gets pass/fail output and can retry once.
  5. Result: 79% success rate. Same model. The constraints eliminated whole categories of errors.

The improvement came from four changes: a focused system prompt, a curated tool set, restricted context, and external verification replacing self-evaluation. No model upgrade required.

Try It Yourself

✍️ Design a Harness Improvement

Pick one AI coding task you do at least twice a week — writing tests, reviewing PRs, generating API endpoints, debugging errors — and design a targeted harness improvement.

  1. Identify the task and estimate your current success rate. How often does the AI get it right on the first try?
  2. Write a focused CLAUDE.md section (or AGENTS.md, or Cursor rules file) with specific instructions for that task. Include naming conventions, tool preferences, and verification steps.
  3. Create a custom command that encodes the workflow. For Claude Code:
# .claude/commands/write-migration-test.md
Generate a migration test for the most recent migration file.

Steps:
1. Read the latest file in migrations/
2. Check the current schema with: npm run db:schema
3. Write a test in tests/migrations/ following existing examples
4. Run: npm run test:migrations
5. If tests fail, fix and retry once
  1. Test before and after. Run the task 5 times without the improvement, then 5 times with it. Track success/failure.
  2. Measure the difference. Did your success rate improve? Did the agent need fewer retries? Did it produce more consistent output?

You don't need statistical significance. Even a rough before/after comparison will show you whether the harness change helped — and it almost always does.
