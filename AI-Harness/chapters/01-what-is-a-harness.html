Chapter 1

What Is a Harness?

The invisible layer between you and the LLM — and why it matters more than the model itself

You Already Use Harnesses Every Day

You open Claude Code, type refactor this function to use async/await, and hit Enter. A few seconds later, the tool reads your file, thinks about the change, edits the code, and runs your tests to make sure nothing broke. Feels like magic, right?

Here's the thing: that magic isn't the LLM.

The LLM — the large language model at the center of it all — is the part that generates text. That's it. It reads text in, it writes text out. Everything else — the file reading, the code editing, the test running, the permission asking, the awareness of your project structure — that's something else entirely.

That "something else" is called a harness. And you've been using one every single time you've touched Claude Code, ChatGPT, Cursor, Codex, or Windsurf. You just never noticed it, because a good harness is invisible.

By the end of this chapter, you'll never look at these tools the same way again.

The Raw LLM — It's Underwhelming

To understand what a harness does, you first need to see what life is like without one. Let's call a model API directly — no tools, no system prompt, no harness at all. Just raw text in, raw text out. 

curl https://api.anthropic.com/v1/messages \
  -H "x-api-key: $ANTHROPIC_API_KEY" \
  -H "content-type: application/json" \
  -H "anthropic-version: 2023-06-01" \
  -d '{
    "model": "claude-sonnet-4-20250514",
    "max_tokens": 1024,
    "messages": [
      {"role": "user", "content": "What files are in my project and which ones need refactoring?"}
    ]
  }'

What do you get back? Text. Just text. The model will politely explain that it can't see your files, doesn't know what project you're working on, and would be happy to help if you paste some code. No file reading. No code execution. No memory of what you said five minutes ago. No awareness of your project.

Raw API response (no harness)

I don't have access to your filesystem or project directory, so I can't see what files you have. However, if you share the code or file structure, I'd be happy to suggest refactoring opportunities...

The model is smart, but it's blind. No files, no tools, no context. It can only work with the text you send it.

The raw model is like a brilliant consultant locked in a windowless room. Smart? Absolutely. Useful on its own? Barely. You'd have to manually copy-paste every file, re-explain your project from scratch in every conversation, and then manually apply every suggested change yourself.

So what turns this windowless-room consultant into the tool that reads your files, edits your code, and runs your tests?

The harness.

The Harness, Defined

The Key Idea

A harness is the complete software infrastructure that wraps an LLM — everything between the user's input and the model's raw output. It decides what the model sees, what tools it can use, what it remembers, and what it's allowed to do.

LangChain put it in a neat equation: Agent = Model + Harness. The model provides the intelligence. The harness makes that intelligence useful and autonomous.

When Anthropic engineers talk about Claude Code, when OpenAI describes Codex, when Cursor blogs about their agent — they're all talking about their harness. The model is a shared ingredient (sometimes literally the same model). The harness is what makes each tool different.

This is the thing nobody told you: most of what you experience when using an AI tool is the harness, not the model.

The Harness Is the Operating System

Think about a CPU. It does raw math — billions of calculations per second. Incredibly powerful. But have you ever tried to use a CPU without an operating system? No filesystem, no memory management, no device drivers, no user interface. Just a very fast, very useless chip.

The operating system is what turns a CPU into a computer. It provides the filesystem so programs can read and write data. It manages memory so things don't crash. It handles permissions so one program can't wreck another. It gives you a screen and a keyboard so you can actually interact with the thing.

The harness does the same thing for an LLM:

  • CPU = LLM (raw intelligence, generates text)
  • Filesystem = Tools (file read/write, terminal access)
  • Memory management = Context engineering (what fits in the window)
  • Permissions = Guardrails (what the model is allowed to do)
  • Device drivers = Tool integrations (MCP servers, APIs)
  • User interface = The CLI or chat experience you see
When someone says "Claude is so good at coding," what they really mean is "Claude Code's harness is so good at making Claude useful for coding." The same model through a raw API call would struggle with the same task — not because it's dumber, but because it has no tools, no context, and no awareness of your project.

The Race Car Driver and the Pit Crew

Here's another way to think about it. The LLM is a Formula 1 driver — fast reflexes, incredible skill, makes split-second decisions about where to steer next.

But a driver alone doesn't win races. You need:

  • A pit crew (tools) — they change tires, refuel, make repairs
  • A track layout (context) — the driver needs to know where the turns are
  • Race rules (guardrails) — speed limits in the pit lane, penalties for dangerous moves
  • A radio (memory) — the team relays tire condition, weather changes, strategy updates
  • Telemetry data (persistence) — recorded from past laps so the next session starts informed

A Formula 1 driver in a go-kart, with no pit crew, no radio, and no strategy, is just a talented person going in circles. That's your LLM without a harness.

Why Prompt Engineering Is Only One Small Piece

If you've spent the last couple of years learning to write better prompts — "be specific," "give examples," "use chain of thought" — that's great. But you've been optimizing one room in a house with six rooms.

The prompt is what you type. The harness controls everything else the model sees and does. Watch the difference:

Prompt Engineering Only

You are an expert software engineer. Please carefully analyze the file src/auth.ts and refactor the login function to use async/await instead of callbacks. Be thorough and consider edge cases. Return the complete updated file.

A carefully crafted prompt — but the model still can't read src/auth.ts, doesn't know your project structure, and can't test the result.

Harness Engineering

refactor the login function to use async/await

A simple prompt — but the harness provides: a system prompt with project context, file reading tools, code editing tools, test execution, and permission guardrails. The harness does the heavy lifting.

See the irony? The worse prompt gets better results — because the harness provides the context, tools, and guardrails that no amount of prompt crafting can replace.

This connects to something Martin Fowler observed: harness engineering is a shift from "generate anything" to deliberate constraint. You don't need a perfect prompt when the harness gives the model the right context, the right tools, and the right boundaries. You constrain the problem space so the model can focus on what it does best — reasoning and generating.

The 2025–2026 Realization

Here's the industry's biggest insight of the past year: improving the harness often delivers more improvement than improving the model. Same model, better wrapping, dramatically better results.

30% Performance gain Cursor achieved through harness tweaks alone — same model, better instructions and tools
#30 → #5 LangChain's Deep Agents jumped on Terminal Bench through harness engineering, not model upgrades
Same LLM Codex, Claude Code, and Cursor often use the same frontier models — the harness is what differentiates them

This is the most exciting thing about harness engineering: you don't need to wait for the next model release to get dramatically better results. You can improve the harness today — right now — by writing better project instructions, configuring tools, and tuning context.

This doesn't mean the model doesn't matter. A great harness can't fix a weak model. But a mediocre harness absolutely can waste a great model. Most developers today are running great models through mediocre harnesses — and that's where the opportunity is.

The Six Components — A Quick Preview

Every harness is built from some combination of six components. Not all harnesses use all six, but every serious AI coding tool uses most of them. Here's the map:

System Prompt → Tools → Context → Memory → Guardrails → Persistence

1. System Prompt

Hidden instructions that shape the model's behavior before you say a word. Claude Code's system prompt is thousands of tokens long — you never see it, but it tells the model how to behave, what tone to use, and what rules to follow.

2. Tools

File reading, code editing, terminal access, web search — the model's hands. Without tools, an LLM can only talk about code. With tools, it can actually work with code.

3. Context Engineering

Your codebase might be 500,000 lines. The model's context window might hold 200,000 tokens. Context engineering is the art of picking the right information for this specific task — like packing a suitcase for a trip.

4. Memory & State

LLMs have zero memory — every API call starts from scratch. The harness fakes continuity by tracking conversation history, session state, and sometimes writing persistent notes to disk.

5. Guardrails

Permission boundaries, output validation, safety rails. The harness decides what the model is allowed to do — can it delete files? Run arbitrary commands? Push to production?

6. Persistence

Saving progress, resuming sessions, handing off between conversations. When you close Claude Code and reopen it tomorrow, persistence is what lets the next session pick up where you left off.

Chapter 2 breaks all six of these wide open. For now, just know the map.

Where You'll See Harnesses

Every AI tool you use daily is a harness around a model. Here's a quick roll call:

  • Claude Code — CLI agent harness with file tools, bash execution, MCP integrations, and a granular permission system
  • ChatGPT — Browser harness with web search, code interpreter, DALL-E, file upload, and a memory feature
  • Cursor — IDE-integrated harness with 15+ specialized tools, subagents, and model-specific tuning
  • Codex — Cloud-based harness with sandboxed execution, AGENTS.md context, and a JSON-RPC architecture
  • Windsurf — Persistent agentic harness with a Cascade architecture that carries context forward across sessions

Same frontier LLMs under the hood. Very different harnesses. That's why the same question gets different results in different tools — and why you prefer one tool for one task and another tool for a different task.

What Happens When You Press Enter

Let's make this concrete. You type a message in Claude Code and hit Enter. Here's what happens before the model sees a single word of your prompt — and after it responds.

  1. Your message is captured by the CLI interface and added to the conversation history.
  2. The harness loads the system prompt — thousands of tokens of hidden instructions that tell the model how to behave, what tools are available, and what rules to follow.
  3. The harness checks for CLAUDE.md files — project-level instructions you've written that get injected as additional context.
  4. Conversation history is assembled. If the conversation is long, older turns may be compacted (summarized) to free up space.
  5. Available tools are registered — file read, file write, bash, grep, glob, web search, plus any MCP tools you've configured.
  6. All of this is packaged into a single API request. Your prompt is maybe 5% of what the model actually sees. The other 95% is harness-provided context.
  7. The model responds — possibly with a tool call request like "I need to read the file src/auth.ts."
  8. The harness checks permissions, executes the tool, and sends the result back to the model.
  9. The model processes the tool result and may request another tool call — "now edit lines 42–58." This loop continues until the model has a final answer.
  10. The harness formats and displays the final response in your terminal.

That "simple" Enter keypress triggered system prompt loading, context assembly, tool registration, permission checks, and a multi-turn tool loop. The model did the thinking. The harness did everything else.

Try It Yourself

Call a Raw API vs. Use an AI Tool — Spot the Harness

Step 1: Call a model API directly using the command below (or use the Anthropic Console / OpenAI Playground in "raw" mode). Ask: "What files are in my current project and which ones need refactoring?" 

curl https://api.anthropic.com/v1/messages \
  -H "x-api-key: $ANTHROPIC_API_KEY" \
  -H "content-type: application/json" \
  -H "anthropic-version: 2023-06-01" \
  -d '{
    "model": "claude-sonnet-4-20250514",
    "max_tokens": 1024,
    "messages": [{"role": "user",
      "content": "What files are in my current project and which ones need refactoring?"}]
  }'

Step 2: Ask Claude Code (or ChatGPT, or Cursor) the exact same question.

Step 3: Compare the two experiences. Make a list of everything the tool-based version does that the raw API cannot:

  • Did it read your files?
  • Did it remember your previous messages?
  • Did it ask for permission before acting?
  • Did it use tools (web search, file reading, code execution)?
  • Did it format the output in a special way?

Every item on your list is something the harness provides — not the model.
