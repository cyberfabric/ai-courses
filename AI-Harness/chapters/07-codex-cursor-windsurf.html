Chapter 7

Codex, Cursor, and Windsurf

Same LLMs, different harnesses — how three tools solve the same problems differently

Chapter 6 took Claude Code apart bolt by bolt. Now let's line up three other major AI coding tools and see how they solve the same fundamental problems — system prompts, tools, context, memory, guardrails, and persistence — with completely different architectures.

The point isn't to crown a winner. It's to show that the same model can produce vastly different results depending on the harness. Each tool makes different tradeoffs, and understanding those tradeoffs will make you a better user of whichever tool you choose.

Think of it like comparing three different car manufacturers. They all have engines, steering wheels, and brakes. But a Tesla, a Toyota, and a Ferrari feel completely different to drive — because the engineering around the engine is different.

OpenAI Codex

Architecture: Cloud-First, Sandbox by Default

Codex takes a fundamentally different approach from Claude Code. While Claude Code runs locally on your machine, Codex runs in the cloud — inside sandboxed containers managed by OpenAI. Your code gets cloned into a cloud VM, the agent works there, and you review the results as a diff or PR.

This has a big architectural implication: Codex is sandboxed by default. The agent runs in an isolated environment with Landlock and seccomp security, meaning it physically cannot access your local filesystem, your browser, or your network (except for allowed endpoints). This is the most restrictive security model of any major coding agent.

Codex's sandbox eliminates an entire category of risk. The agent can't accidentally delete your files, leak secrets to the network, or modify anything outside the sandbox. The tradeoff: it can't access local services, databases, or tools that exist only on your machine.

The Agent Loop: Items, Turns, and Threads

Codex structures its agent loop around three abstractions:

  • Items — The atomic unit. Each item is an input or output with a lifecycle: "started" → optional "delta" events (streaming) → "completed." Items can be user messages, agent messages, tool executions, or approval requests.
  • Turns — A sequence of items from a single unit of work. One turn might include multiple tool calls and iterations.
  • Threads — Durable containers for ongoing sessions. They support creation, resumption, forking, and archival.

Communication happens over a JSON-RPC protocol (a bidirectional "JSON-RPC lite" variant over stdio). This is different from Claude Code's simpler tool-call-and-response pattern — Codex's protocol supports richer interactions like streaming diffs, approval workflows, and server-initiated requests.

AGENTS.md and Progressive Disclosure

Where Claude Code uses CLAUDE.md, Codex uses AGENTS.md — but with a different philosophy. Codex emphasizes progressive disclosure: start with a small entry point (~100 lines) and let the agent navigate to deeper context on demand. 

# AGENTS.md (root — kept small)

## Overview
E-commerce platform: Next.js frontend, Python FastAPI backend.

## Structure
- frontend/ — Next.js 15 app (see frontend/AGENTS.md)
- backend/ — FastAPI service (see backend/AGENTS.md)
- docs/ — Architecture decisions and API specs

## Key Commands
- make test — run all tests
- make lint — run linters
- make dev — start development servers

The root AGENTS.md is intentionally brief. Subdirectories have their own AGENTS.md files with progressively more detail. The agent loads what it needs, when it needs it — keeping the context lean.

Codex also supports a Skills system: reusable task definitions with metadata loaded first and full instructions loaded only when needed. Skills turn recurring engineering tasks (PR review, release prep, verification) into repeatable workflows.

Interestingly, OpenAI evaluated MCP for Codex's tool protocol but decided it was insufficient. MCP's tool-oriented model didn't support IDE-specific features like streaming diffs, approval flows, and thread persistence. So they built JSON-RPC instead — a reminder that standards don't always fit every use case.

Cursor

Architecture: IDE-Native, Model-Specific Tuning

Cursor takes the opposite approach from Codex: instead of running in the cloud, it's embedded in your IDE. The agent lives inside your editor, sees your open tabs, your cursor position, and your recent edits. This gives it context that cloud-based agents can't access.

But Cursor's most distinctive feature is per-model harness tuning. Every model in Cursor's agent gets different instructions and different tool configurations optimized for how that specific model was trained.

What per-model tuning looks like in practice:

When Cursor integrated OpenAI's Codex model, they found they needed to:

  • Limit reasoning summaries to 1–2 sentences (the model responded better to concise instructions)
  • Remove mid-turn communication language (Codex couldn't communicate mid-turn like Claude can)
  • Use more direct instructions ("implement this" instead of "propose and then implement")
  • Name tools to match Unix commands the model was trained on (rg instead of "search")

These changes alone produced a 30% performance difference on Cursor's internal benchmarks. Same model, different harness instructions, dramatically different results.

Subagents and Parallel Work

Cursor supports up to 8 parallel background agents, each working in an isolated git worktree. This means you can have multiple agents working on different tasks simultaneously without interfering with each other.

The subagent architecture forms a tree: a parent agent can spawn child agents, which can spawn their own children. Each agent gets:

  • An independent context window (exploration doesn't bloat the parent)
  • Its own git worktree (isolated file changes)
  • A custom system prompt (specialized for the task)
  • Filtered tool access (only what the sub-task needs)

Three built-in subagent types: Explore (search with a faster model), Bash (isolate verbose terminal output), and Browser (control browsers via MCP, filter noisy DOM snapshots).

CursorBench: Measuring Harness Quality

Cursor built their own evaluation suite, CursorBench, based on real engineering sessions from their own team. Why? Because public benchmarks like SWE-Bench don't capture real developer workflows — they test isolated bug fixes, not the messy reality of multi-file feature work.

CursorBench measures solution correctness, code quality, efficiency, and interaction behavior. It's better at distinguishing models than public benchmarks because it tests actual Cursor usage patterns, not academic scenarios.

This is a harness engineering insight worth remembering: how you evaluate your harness matters as much as how you build it. Public benchmarks test narrow scenarios. Cursor built their own benchmark based on real usage, and it tells them things public benchmarks can't — like whether a model handles multi-file refactoring well, or how it behaves in the middle of a long session.

Windsurf

Architecture: Persistent, Aware, Co-Designed

Windsurf's Cascade is the most architecturally different of the three. While Codex and Cursor use general-purpose frontier models with harness tuning, Windsurf took a radical step: they trained their model and harness together.

Their SWE-1 model family was built using reinforcement learning directly on the Cascade agent harness in real coding environments. The model learned not just how to code, but how to use the specific tools in the Cascade harness. This is like training a driver specifically for one car, rather than teaching a generic driver to adapt to any vehicle.

40% SWE-1.5 score on SWE-Bench Pro — second only to Claude Sonnet 4.5's 43.6%
~14x Faster than Claude Sonnet 4.5 — ~950 tokens/sec vs 69 tokens/sec
Co-designed Model + harness trained as a unified system via reinforcement learning

Code Mode vs. Chat Mode

Windsurf explicitly separates two operating modes:

  • Code Mode — Full agent with file modification. Building features, refactoring, fixing bugs. The model can read and write your code.
  • Chat Mode — Read-only assistant. Questions, suggestions, architectural advice. The model can see your code but won't touch it.

This is similar to Claude Code's plan mode vs. default mode, but Windsurf makes the separation more prominent in the UI. The two modes use different harness configurations — different system prompts, different tool access, different guardrails.

Real-Time Awareness

Cascade's most distinctive feature is Flows — real-time awareness of your actions. Unlike other tools that only get context when you send a message, Cascade tracks your edits, terminal commands, clipboard activity, and cursor position continuously. It infers your intent from your actions.

This means Windsurf can adapt context before you ask. If you're editing a test file, Cascade already knows the relevant source file. If you copy an error from the terminal, the context updates automatically. No explicit context management required from the user.

The tradeoff with real-time awareness: it can include noise. If you're jumping between unrelated files or experimenting, Cascade might incorporate irrelevant context. The implicit approach works well for focused work but can misfire during exploration.

Side-by-Side: The Six Components

Let's compare how each tool handles the six harness components we defined in Chapter 2.

Codex — System Prompt

AGENTS.md with progressive disclosure. Small root file, deeper context in subdirectories. Skills system for reusable task definitions. Context loaded on demand.

Cursor — System Prompt

Per-model customized instructions. Each frontier model gets different prompts tuned to its strengths. .cursorrules for project-level context. Most harness-aware approach to prompt design.

Codex — Tools & Context

Skills-based tools with dynamic loading. Cloud sandbox execution. Progressive disclosure keeps context lean. Prompt caching enables efficient retrieval across turns.

Windsurf — Tools & Context

Real-time Flows tracking all user actions. Context updates automatically as you work. Model co-designed with tool harness via RL. Checkpoint system for reverting changes.

Codex — Guardrails

Strongest sandbox: Landlock + seccomp by default. Three modes: workspace-write (default), danger-full-access, on-request. Approval workflow via JSON-RPC protocol. Zero Data Retention for enterprise.

Cursor — Guardrails

Git worktree isolation per agent. Approval system for parallel operations. Per-agent tool filtering. IDE-level file access controls. Self-hosted option for enterprise deployment.

Inside the Harness: Same Task, Three Approaches

Let's trace what happens when you give the same task — add rate limiting to the API endpoints — to each tool.

Codex: Your repo is cloned into a cloud sandbox. The agent reads AGENTS.md, navigates to backend/AGENTS.md for API details, identifies the middleware layer, writes the rate limiting code, runs tests inside the sandbox. You get a PR with a clean diff. The sandbox means the agent couldn't access your local Redis — it had to mock it or use an in-sandbox alternative.

Cursor: The agent works in your IDE. It uses semantic code search to find all route definitions, reads your existing middleware patterns, and crafts a solution that matches your project's style. Because it can see your open tabs and recent edits, it notices you were just working on the auth middleware and follows that pattern. Per-model instructions ensure the LLM uses tools efficiently. Your local Redis is accessible for testing.

Windsurf: Cascade's real-time awareness already knows you've been editing API routes. Before you even finish typing, it has context on your route structure. SWE-1.5, co-trained with Cascade's tools, navigates your codebase naturally. It creates a checkpoint before changes so you can revert if the approach doesn't work. Everything happens locally, fast.

Same task, same quality of result — but different paths. Codex prioritized security (sandbox isolation). Cursor prioritized accuracy (per-model tuning + IDE context). Windsurf prioritized speed (co-designed model + real-time awareness). The harness shaped the experience.

Try It Yourself

Compare Two Tools on the Same Task

If you have access to two or more of these tools (including Claude Code from Chapter 6), try this experiment:

  1. Pick a concrete task — something specific like "add input validation to the signup form" or "write integration tests for the user API."
  2. Give the exact same task to both tools, in the same project if possible.
  3. Observe the differences:
  • How did each tool explore the codebase? (Tool calls, file reads, search patterns)
  • How many steps did each take?
  • What context did each seem to understand? (Did it know your testing framework? Your code style?)
  • What permissions did each ask for?
  • How did the final code differ in style, structure, and completeness?

The differences you observe are almost entirely harness differences. The underlying models may be similar or even identical. The harness — system prompt, tools, context strategy, guardrails — is what shapes the result.
