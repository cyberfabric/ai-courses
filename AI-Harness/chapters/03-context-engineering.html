Chapter 3

Context Engineering

How the right information gets into the window — the most critical job a harness performs

In Chapter 2, we called context engineering the "ingredient prep station" in our restaurant kitchen. The chef doesn't walk into a warehouse of 10,000 ingredients. Someone curates what's on the station for today's dish.

That analogy understates it. Context engineering is arguably the single most impactful thing a harness does. Get it right, and a mediocre model produces great results. Get it wrong, and the best model in the world hallucinates, misses files, and makes changes that break your project.

This chapter is about what goes into that context window, why most of it isn't your prompt, and the techniques harnesses use to make every token count.

The Context Window Is Prime Real Estate

Every model has a context window — a fixed amount of text it can "see" at once. Think of it as the model's working desk. Everything it needs to think about must fit on this desk.

200K Tokens in Claude's context window (~150,000 words)
1M Tokens in Claude's extended context (available to some tools)
5% How much of the window your actual prompt typically fills

That last number is the important one. When you type a one-line request in Claude Code, your words might be 50–100 tokens. But the full API request sent to the model could be 20,000+ tokens. The other 95% is context the harness assembled: system prompt, conversation history, tool descriptions, project files, system reminders.

Here's the analogy that makes this click: context engineering is like packing a suitcase for a trip. You can't take your entire wardrobe. You have a fixed amount of space. So you pick what matters most for this specific trip — the weather, the activities, the dress code. Pack wrong, and you're stuck at a business dinner in hiking boots.

What Actually Goes Into the Window

Let's be specific about what fills that context window for a typical Claude Code request. The harness assembles these pieces, in roughly this order:

  1. System prompt — Identity, rules, tone, tool descriptions. This alone can be 5,000–10,000 tokens.
  2. CLAUDE.md content — Your project instructions, architecture notes, coding standards. Loaded fresh every turn.
  3. Conversation history — Every message you've sent and every response, plus every tool call and its result.
  4. Tool results — The contents of files read, search results returned, command outputs. These accumulate fast.
  5. System reminders — Re-injected instructions at intervals throughout the conversation.

Notice what's happening: a long conversation with many tool calls will fill the window quickly. Read three large files, run a few searches, and you might already be at 50% capacity — before any actual reasoning happens.

Every unnecessary token actively degrades performance. It's not just about fitting within the limit — models perform worse when the context is cluttered with irrelevant information. A 50,000-token context with high signal beats a 200,000-token context stuffed with noise. The harness's job isn't to fill the window — it's to curate the smallest, highest-signal set of tokens.

The Curation Problem

Your codebase is 500,000 lines. The context window holds maybe 200,000 tokens. Even if you could convert lines to tokens one-to-one (you can't — it's roughly 1 line = 5–15 tokens), you still couldn't fit the whole codebase.

So what do you include? This is the curation problem, and different harnesses solve it differently:

Strategy 1: User-Driven Retrieval

The simplest approach — the model reads files when it decides it needs them. In Claude Code, the model calls the Read tool to fetch a file, the Grep tool to search for a pattern, or the Glob tool to find files by name. The model is the curator, guided by the system prompt.

Strategy 2: Automatic Context Scanning

Some harnesses scan your project automatically. Cursor indexes your codebase and uses semantic code search to pull in relevant files before the model even starts reasoning. Windsurf tracks your edits, terminal commands, and clipboard in real-time to infer what context is relevant.

Strategy 3: Static Project Files

CLAUDE.md and AGENTS.md are context you curate yourself. They're loaded into every request automatically, so they're always available. This is the most reliable method — but it only works for information that's always relevant.

Claude Code

Context strategy: Model-driven retrieval. The model decides what to read using tools (Read, Grep, Glob). CLAUDE.md is loaded automatically. System reminders re-inject key instructions.

Strength: Precise — only loads what's needed for the task.

Weakness: The model might miss relevant files it doesn't know to look for.

Cursor

Context strategy: Automatic indexing + semantic search. Cursor pre-indexes your codebase and retrieves relevant snippets before the model reasons. Combines lexical search (for identifiers) with vector search (for concepts).

Strength: Finds relevant code the model didn't explicitly ask for.

Weakness: May include irrelevant snippets, wasting tokens.

Codex

Context strategy: AGENTS.md with progressive disclosure. Starts with top-level AGENTS.md, loads subdirectory files as needed. Optimizes repositories for agent consumption with structured documentation.

Strength: Scales well for large monorepos with clear documentation.

Weakness: Requires upfront effort to write and maintain AGENTS.md files.

Windsurf

Context strategy: Real-time awareness. Tracks your edits, terminal commands, clipboard activity, and conversation history continuously. Infers intent from your actions.

Strength: Context updates automatically as you work — no explicit retrieval needed.

Weakness: Can include noise from unrelated actions in the same session.

CLAUDE.md and AGENTS.md: Your Context Files

These are context files you write that the harness loads automatically. They're the most powerful context tool you have direct control over.

CLAUDE.md lives in your project root and becomes part of Claude Code's system prompt on every turn. It tells the model about your project without you having to explain it in every conversation.

Effective CLAUDE.md (excerpt)

# Project: API Gateway Service ## Stack - Node.js 20 with TypeScript (strict mode) - Express 4.x with Zod validation - PostgreSQL 16 via Drizzle ORM - Jest for testing, Biome for linting ## Key Commands - `npm test` — run all tests - `npm run lint` — check with Biome - `npm run db:migrate` — apply migrations ## Architecture - Routes in src/routes/, one file per resource - Business logic in src/services/ - Database queries in src/repositories/ - All errors go through src/middleware/error-handler.ts ## Rules - Never use `any` type — use `unknown` and narrow - All new endpoints need integration tests - Migrations must be backward-compatible

This gives the model project context on every turn. No need to explain your stack, conventions, or test commands in every conversation.

AGENTS.md is the equivalent for OpenAI's Codex and a growing cross-tool standard. Same idea: structured project context that agents load automatically.

Recent research from ETH Zurich suggests a nuance: only include information the model can't figure out on its own. Things like "this project uses TypeScript" are redundant — the model can see that from the files. Focus on things the model can't infer: custom build commands, non-obvious conventions, specific tooling decisions, and architectural rules that aren't visible in the code.

System Reminders: Fighting Instruction Fade

Here's a problem: when a context window gets long, the model starts "forgetting" the instructions from the beginning. Not literally forgetting — the tokens are still there — but paying less attention to them. Researchers call this "lost in the middle."

Stanford researchers found that LLM performance degrades by more than 30% when relevant information is in the middle of a long context. Models show a U-shaped attention curve — they pay the most attention to the beginning and end, and the least to the middle.

Claude Code's solution: system reminders. These are key instructions re-injected at intervals throughout the conversation. It's not just the system prompt at the top — the harness periodically inserts reminders like: 

<system-reminder>
Remember: prefer editing existing files over creating new ones.
Only create commits when requested by the user.
Use the Read tool before editing a file you haven't read.
The following deferred tools are now available: WebSearch, WebFetch
</system-reminder>

Think of it as a meeting facilitator who periodically restates the agenda. Without reminders, the model drifts. With them, it stays on track even in conversations that span hundreds of turns.

Context Compaction: When the Suitcase Is Full

Long conversations fill up. You read files, run commands, get search results — and suddenly you're at 90% capacity. What happens next?

The harness compacts the conversation. It takes the existing history, summarizes it into a condensed form, and uses that summary as the starting point for a fresh context window. The old details are gone, but the essential information — what was done, what's in progress, what constraints matter — survives.

Claude Code triggers auto-compaction at roughly 95% capacity. You can also trigger it manually with the /compact command. Here's what a good compaction preserves:

  • What tasks have been completed
  • Current state of the work in progress
  • Files that were modified and why
  • Constraints and rules that are still active
  • Next steps the model was planning
  • Individual tool call details (summarized away)
  • Exact file contents that were read (summarized away)
  • Exploratory dead-ends (summarized away)

The unchecked items are what gets dropped. This is a trade-off: you lose precision but gain space. The model's understanding after compaction is like reading meeting notes instead of the full transcript — you get the decisions, not the discussion.

Compaction is lossy. If the model read a file earlier and the details were compacted away, it may need to re-read that file. This is why Claude Code's system prompt tells the model to read files before editing them — it can't trust that earlier file reads are still in the context.

The "Lost in the Middle" Problem

We mentioned the U-shaped attention curve. Let's look at why it matters practically.

Imagine you're in turn 50 of a conversation. The system prompt is at the beginning. Your latest message is at the end. But that critical file you read in turn 12? It's buried in the middle — and the model is paying the least attention to it.

This is why harnesses employ several counter-strategies:

  • System reminders — Re-inject critical instructions near the end of the context, where attention is highest.
  • Compaction — Summarize away the bulky middle, so there's less "middle" to get lost in.
  • Strategic placement — Put the most important context at the beginning (system prompt) and end (recent messages). Let the middle hold conversation history that serves as background.
  • CLAUDE.md reload — Project instructions are re-loaded on every turn, typically near the top, not left to drift into the middle.

As a user, you can work with this: if you need the model to pay attention to something specific in a long conversation, mention it again in your latest message. Don't assume it remembers turn 5 when you're on turn 50.

The Hashline Technique: Format Changes That Change Everything

Here's a story that perfectly illustrates why context engineering matters more than model engineering.

In 2025–2026, researchers discovered that simply changing how code is formatted in the context window dramatically improved coding success rates. The technique, called Hashline, adds structured delimiters and line numbers to code before sending it to the model.

Standard format

function login(user, pass) { db.query('SELECT * FROM users', (err, rows) => { if (err) callback(err); const match = rows.find(r => r.name === user); callback(null, match); }); }

Hashline format

### src/auth.js 1 | function login(user, pass) { 2 | db.query('SELECT * FROM users', (err, rows) => { 3 | if (err) callback(err); 4 | const match = rows.find(r => r.name === user); 5 | callback(null, match); 6 | }); 7 | }

The results were staggering: one model went from 6.7% success to 68.3% — a tenfold improvement — just from reformatting the code. The model's intelligence didn't change. The harness changed how information was presented.

Why does this work? Because the problem often isn't the model's ability to understand code — it's the model's ability to express precise edits. Line numbers give the model a coordinate system. Structured delimiters clarify file boundaries. The model can say "replace lines 2–5" instead of trying to reproduce the exact surrounding code from memory.

The Harness Engineering Lesson

The Hashline discovery is the purest example of context engineering in action. Same model, same task, same code — different formatting, dramatically different success rate. This is why harness engineering delivers returns that rival model improvements. How you present information matters as much as what information you present.

Inside the Harness: Opening a Project in Claude Code

Let's trace what Claude Code puts into the context window when you open a project and type your first message.

Before your message reaches the model, the harness assembles:

  1. Base system prompt (~5,000–10,000 tokens) — Conditional fragments activate based on your OS, shell, permission mode, and available tools.
  2. Tool descriptions (~2,000–4,000 tokens) — Every registered tool with its name, description, and parameter schema. Deferred tools get lighter descriptions to save space.
  3. CLAUDE.md chain — Global (~/.claude/CLAUDE.md), then project (./CLAUDE.md), then any directory-level overrides. All concatenated into the context.
  4. Memory files — If you have persistent memory configured, relevant memories are loaded based on the current project.
  5. Your message — Finally, your actual prompt. Often the smallest piece of the puzzle.

For a first message in a project with a well-written CLAUDE.md and 18 registered tools, the context might already be 15,000–20,000 tokens before the model processes your request. Your 20-word prompt is riding on a foundation of carefully assembled context.

On subsequent turns, conversation history gets added — every previous message, tool call, and tool result. By turn 20 with several file reads, you could easily be at 100,000+ tokens. This is when compaction becomes critical.

Try It Yourself

Write a CLAUDE.md (or AGENTS.md) for Your Project

Pick a project you work on regularly. Create a CLAUDE.md file in the project root with:

  1. Stack description — Language, framework, database, key libraries.
  2. Key commands — How to run tests, lint, build, and deploy.
  3. Architecture overview — Where different types of code live (routes, services, models).
  4. Non-obvious rules — Things the model can't figure out from the code. Custom conventions, forbidden patterns, specific tooling decisions.

Skip things the model can infer: "this is a TypeScript project" (it can see that), "use camelCase for variables" (it can see that too). Focus on what's invisible in the code.

Then test it: ask Claude Code (or your tool of choice) a question about your project without the context file, and then with it. Notice the difference in answer quality — especially for questions about conventions, architecture decisions, and "how do we do X here?"
