Chapter 9

Debug-Build Inspectors

Chucker for HTTP, a debug-menu pattern, debug-only feature flags — gated so none of it ships

The "open the hood" problem

StrictMode and LeakCanary watch your app from the outside. They yell when something is wrong. But there's a different category of bug-hunting where you don't want a yell — you want to look inside the running app: what request did it just send, what did the server send back, which feature flags are currently on, what build is this exactly, what would happen if I forced this screen into the error state.

For years, the answer was Logcat plus prayer. You'd add Log.d("API", response.toString()) in three places, ship a debug build, navigate around, scroll through 80 KB of unrelated logs, and squint. By the time you found the request, you'd forgotten what you were looking for.

This chapter wires up the kit that experienced Android teams quietly take for granted: an in-app HTTP inspector (Chucker), a debug menu reachable from the running app, and a tiny pattern for overriding feature flags locally. None of it ships to users. All of it is gated through debugImplementation and BuildConfig.DEBUG, so the release APK stays exactly as small and exactly as locked-down as it was before.

The goal of this chapter: turn your debug build into a tool, not just a slower version of release. By the end you'll have HTTP requests one notification away, a hidden debug menu reachable from your app, and feature flags you can flip without redeploying remote config.

The debug-only kit, in one box

Meet the Debug-Build Kit

What it is: a curated bundle of three things — Chucker (HTTP inspector), a debug menu screen, and a debug-only feature-flag override layer — all delivered via debugImplementation and BuildConfig.DEBUG gates.

What it catches that other tools don't: live API regressions, server-side bugs masquerading as client bugs, "works on staging breaks on prod" issues, feature-flag mismatches, "what build did the tester actually run" confusion, and any bug that needs state inspection rather than crash signals.

Install (one-line summary): debugImplementation("com.github.chuckerteam.chucker:library:VERSION") + releaseImplementation("...:library-no-op:VERSION") + a Compose DebugMenuScreen guarded by BuildConfig.DEBUG.

Runtime cost: Chucker adds an OkHttp interceptor (microseconds per request) and a small SQLite store of the last N transactions. Both vanish in release. Debug menu is one extra entry point in your nav graph, also gone in release.

When to skip it: if your app makes no HTTP calls and has zero feature flags, you don't need Chucker. You probably still want a debug-menu pattern for build info.

Why the debugImplementation distinction matters

Three Gradle dependency configurations carry this entire chapter:

implementation

Ships in every variant, including release. Anything you put here is in your APK. Treat as your default only for code users are meant to run.

debugImplementation

Compiled into the debug build variant only. The release APK never sees it. This is where Chucker, your debug-menu library code, LeakCanary, and any in-app inspector go.

releaseImplementation

Ships only in release. Useful for the no-op half of a debug/release pair (e.g. Chucker's no-op artifact) so call sites don't need if (BuildConfig.DEBUG) wrappers.

The pattern: "real" library on debugImplementation, matching no-op library (same package, empty implementations) on releaseImplementation. Your app code calls the API unconditionally, and the no-op silently does nothing in release. Less branching, fewer mistakes, smaller release APK.

Chucker: HTTP requests in your notification shade

Chucker is an OkHttp Interceptor that records every request and response your app makes — URL, method, status, headers, body, timing — and surfaces them in two places: a persistent notification (tap to open) and an in-app screen. You see what the network actually did, in plain English, without leaving the device.

It is the single most-installed dependency in any Android team that does API integration regularly, and the one that gets installed last. Don't be that team.

Install & wire it up

app/build.gradle.kts

dependencies {
    // Chucker: real library in debug, no-op in release
    debugImplementation("com.github.chuckerteam.chucker:library:4.0.0")
    releaseImplementation("com.github.chuckerteam.chucker:library-no-op:4.0.0")
}

The no-op artifact is the trick. It exposes the same ChuckerInterceptor class with empty methods, so the call sites below compile and run in release without an if (BuildConfig.DEBUG) wrapper.

NetworkModule.kt — plug into your OkHttpClient

import com.chuckerteam.chucker.api.ChuckerInterceptor
import com.chuckerteam.chucker.api.RetentionManager

@Provides
@Singleton
fun provideOkHttpClient(@ApplicationContext context: Context): OkHttpClient {
    val chuckerInterceptor = ChuckerInterceptor.Builder(context)
        .collector(ChuckerCollector(
            context = context,
            showNotification = true,
            retentionPeriod = RetentionManager.Period.ONE_HOUR
        ))
        .maxContentLength(250_000L)              // truncate big bodies
        .redactHeaders("Authorization", "Cookie") // never log secrets
        .alwaysReadResponseBody(true)             // capture even on stream errors
        .build()

    return OkHttpClient.Builder()
        .addInterceptor(chuckerInterceptor)       // last in chain = sees final request
        .build()
}

redactHeaders is non-negotiable: bearer tokens and session cookies must never be visible in your in-app inspector, even on a debug build that lives on a tester's phone.

Reading Chucker output, fast

  • Pull down the notification shade — Chucker shows the live count of recent requests.
  • Tap to open the in-app list — sorted newest-first, color-coded by status code.
  • Open a request — three tabs: Overview, Request, Response. Body is rendered with JSON pretty-printing.
  • Long-press the row to share — exports the transaction as text or as a curl command, perfect for pasting into a bug report.
Headers redaction is mandatory. Anyone with the debug APK on their phone can open Chucker and read every request body in plain text. If you ship debug builds to QA, beta testers, or contractors, scrub Authorization, Cookie, and any custom auth header at minimum. Treat the debug APK as a build that could be reverse-engineered — it won't, but build like it could.

The debug menu pattern

Every shipping team eventually grows a "secret" screen that QA, support, and engineers use to inspect and override behavior. Most teams build it accidentally, one ad-hoc button at a time. The result is hostile and forgotten.

The right move: design the debug menu as a deliberate feature with a stable home. Reachable from a debug-only entry point (long-press your logo, settings → "Debug" link, a 5-tap version-number Easter egg if you're a traditionalist), gated on BuildConfig.DEBUG, and kept in source control so the whole team contributes to it.

What goes inside

📋
Build info: version name, version code, commit SHA, build time, build type, flavor, server URL
🚩
Feature flag overrides: every flag with a local toggle that beats remote config
💥
Force-crash, force-ANR, force-OOM buttons for crash-reporter testing
🧹
Clear app data, clear OkHttp cache, clear shared prefs — one-tap state reset
🚪
Deep-link launcher: jump to any screen by name without a real flow
📤
Log export: dump the last N seconds of Logcat to a shareable file

A minimal Compose debug menu

DebugMenuScreen.kt (in src/debug/java/...)

@Composable
fun DebugMenuScreen(
    flags: DebugFeatureFlagRepository,
    onClearData: () -> Unit,
    onForceCrash: () -> Unit,
) {
    Column(modifier = Modifier.padding(16.dp)) {
        Text("Build info", style = MaterialTheme.typography.titleMedium)
        Text("Version: ${BuildConfig.VERSION_NAME} (${BuildConfig.VERSION_CODE})")
        Text("Commit: ${BuildConfig.GIT_SHA}")
        Text("Build time: ${BuildConfig.BUILD_TIME}")
        Text("Server: ${BuildConfig.API_BASE_URL}")

        HorizontalDivider(modifier = Modifier.padding(vertical = 12.dp))

        Text("Feature flags", style = MaterialTheme.typography.titleMedium)
        flags.allFlags().forEach { flag ->
            Row(verticalAlignment = Alignment.CenterVertically) {
                Text(flag.key, modifier = Modifier.weight(1f))
                Switch(
                    checked = flag.localOverride ?: flag.remoteValue,
                    onCheckedChange = { flags.override(flag.key, it) }
                )
            }
        }

        HorizontalDivider(modifier = Modifier.padding(vertical = 12.dp))

        Button(onClick = onClearData) { Text("Clear app data") }
        Spacer(Modifier.height(8.dp))
        Button(onClick = onForceCrash) { Text("Force crash (test Crashlytics)") }
    }
}

Putting this file under src/debug/java/ instead of src/main/java/ means it literally does not exist in the release source set. Even if you accidentally import it from a release-shared file, the build will fail — a much better failure mode than silently shipping debug UI.

Wiring BuildConfig with build info

app/build.gradle.kts — expose git SHA & build time

android {
    defaultConfig {
        // Resolve git SHA at configuration time
        val gitSha = providers.exec {
            commandLine("git", "rev-parse", "--short", "HEAD")
        }.standardOutput.asText.get().trim()

        buildConfigField("String", "GIT_SHA", "\"$gitSha\"")
        buildConfigField("long", "BUILD_TIME", "${System.currentTimeMillis()}L")
    }

    buildTypes {
        debug {
            buildConfigField("String", "API_BASE_URL", "\"https://staging.example.com\"")
        }
        release {
            buildConfigField("String", "API_BASE_URL", "\"https://api.example.com\"")
        }
    }

    buildFeatures {
        buildConfig = true   // required to use buildConfigField in AGP 8+
    }
}

buildFeatures.buildConfig = true is required from AGP 8.0 onward — the default flipped to false. If your BuildConfig mysteriously stops compiling after a Gradle bump, this is the line you forgot.

Debug-only feature flag overrides

You almost certainly already have a feature-flag system — Firebase Remote Config, LaunchDarkly, Statsig, an in-house wrapper, or a homegrown SharedPreferences map. The problem with all of them: they answer "what should this flag be for this user, on this device, right now?" without giving you a way to override locally during development.

The pattern is a thin wrapper that checks a debug-only override store first, then falls back to the real source.

DebugFeatureFlagRepository.kt (debug source set only)

class DebugFeatureFlagRepository @Inject constructor(
    private val real: RemoteConfigRepository,
    @ApplicationContext context: Context,
) : FeatureFlagRepository {

    private val overrides = context.getSharedPreferences(
        "debug_flag_overrides", Context.MODE_PRIVATE
    )

    override fun isEnabled(key: String): Boolean {
        // Debug override beats remote config
        if (overrides.contains(key)) return overrides.getBoolean(key, false)
        return real.isEnabled(key)
    }

    fun override(key: String, value: Boolean) {
        overrides.edit().putBoolean(key, value).apply()
    }

    fun clear(key: String) = overrides.edit().remove(key).apply()
}

Bind this implementation only in the debug Hilt module. The release Hilt module binds RemoteConfigRepository directly. Same interface, different implementation per variant — the call site doesn't change.

Why this is leverage: testing a flag that gates a paywall used to mean editing remote config (visible to the entire team), waiting for cache TTL, restarting the app, and praying. Now: open debug menu, flip switch, hit back. 3 seconds. Repeat 50 times in a session without coordinating with PMs.

Verifying nothing leaked into release

The single most important verification step in this whole chapter: your release APK must not contain any of the debug-only code. If Chucker, your debug menu, or the override repository ends up in a release build, you've shipped a free reverse-engineering kit to attackers and a confused experience to users.

The 60-second sanity check

Inspect the release APK

# Build the release APK
./gradlew :app:assembleRelease

# Inspect with apkanalyzer (ships with Android SDK)
$ANDROID_HOME/cmdline-tools/latest/bin/apkanalyzer dex packages \
    app/build/outputs/apk/release/app-release.apk \
    | grep -E "(chuckerteam|DebugMenuScreen|DebugFeatureFlag)"

# Should print: nothing.

Wire this as a CI check — a single grep on the release APK that fails the build if debug symbols leak. Cheap, deterministic, runs in seconds.

The Gradle-side belt-and-braces

app/build.gradle.kts — force build to fail on misuse

dependencies {
    // Compile-time check: anything in :app:debug source set
    // CANNOT be referenced from :app:main or :app:release.
    // Putting DebugMenuScreen.kt under src/debug/ enforces this for free.
}

android {
    buildTypes {
        release {
            isMinifyEnabled = true
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
        }
    }
}

R8 in release will also catch dead-code references to debug classes — they'd be unused after stripping the wrapper, which gets stripped, and so on. The source-set boundary catches it earlier and clearer.

Other debug-only inspectors worth knowing

Chucker is the workhorse but not the only option. Quick mentions, not deep dives:

Flipper (deprecated, but you'll see it)

Meta's desktop-app inspector. Powerful, but Meta wound it down in 2024. If you inherit a project using it, plan a migration to Chucker + a debug menu rather than reviving Flipper.

Beagle

An in-app debug-menu library that pre-bundles many of the patterns in this chapter (build info, feature toggles, force-crash, log viewer). If you'd rather not roll your own, Beagle gets you 80% of a debug menu in 20 minutes.

Rebugger / Compose recomposition counters

Compose-specific. Logs which composable recomposed and why — helpful when chasing performance issues. Pair with the Layout Inspector's recomposition-count overlay.

Hyperion

A Willowtree-built debug menu shaped like a drawer. Older but still maintained. Comparable to Beagle in scope.

Pick one debug-menu library or roll your own — not both. Two competing debug menus in the same project is a smell that nobody owns the developer-experience layer. Make it one person's responsibility, even part-time.

AI-Assisted Setup: /draft-debug-menu

The debug menu is the kind of code you write once per project, then keep extending forever. Each new feature gets one new toggle, one new shortcut, one new override. Writing the initial scaffolding by hand is busywork. Hand it to Claude.

Good prompt

"I'm adding a Compose-based debug menu to my Android app. Tech stack: Hilt, Compose, Coil, Retrofit + OkHttp, Firebase Remote Config. I want sections for: build info (version, commit, build time, API base URL), feature-flag overrides (read from Firebase, toggle locally), force-crash button, clear-app-data button, deep-link launcher with hardcoded routes [home, profile, paywall, settings]. Place the file under app/src/debug/java/com/example/debug/. Use BuildConfig fields where possible. The route into this screen is from a long-press on the app logo in the home screen, gated on BuildConfig.DEBUG. Output one Composable file plus the route registration code."

Bad prompt

"Make me a debug menu."

Why it fails: no tech-stack signal, no list of what goes in it, no source-set instruction, no hint about which screens to deep-link to. You'll get a generic Toast.makeText example from a 2018 blog post.

Save it as a reusable skill

.claude/commands/draft-debug-menu.md

# /draft-debug-menu

Draft a Compose-based debug-menu screen for this project.

Steps:
1. Read app/build.gradle.kts to detect: Compose, Hilt, OkHttp, Coil, Retrofit usage.
2. Read any existing FeatureFlagRepository or RemoteConfig wrapper to match its API.
3. Generate a single DebugMenuScreen.kt under app/src/debug/java/.../debug/.
4. Include sections: build info, feature-flag overrides, force-crash, clear-data,
   deep-link launcher.
5. Wire BuildConfig fields (GIT_SHA, BUILD_TIME, API_BASE_URL) — add them to
   build.gradle.kts if missing.
6. Output the Compose file + the nav-graph route registration + the entry-point
   gesture (long-press on logo, BuildConfig.DEBUG-gated).

Constraints:
- All code goes in src/debug/, never src/main/.
- No hardcoded server URLs — always read from BuildConfig.
- Force-crash uses `throw RuntimeException(...)` so Crashlytics catches it.
- Feature-flag toggles persist in SharedPreferences "debug_flag_overrides".

Run /draft-debug-menu on a new project, get the scaffold in under a minute, then spend your time on the parts that are actually project-specific — which screens to deep-link to, which flags exist, which buttons your QA team has been begging for.

Adoption ladder

This is one of the easier ladders in the course — the install is two lines, the gating is automatic, and the value is immediate.

Day 1

Add Chucker to debugImplementation + matching no-op to releaseImplementation. Plug the interceptor into your OkHttpClient. Redact Authorization and Cookie. Ship a debug build to one tester — verify the notification appears and a request is captured. Total: 30 minutes, zero config files touched outside build.gradle.kts.

Week 1

Month 1

Document the debug-menu contract in CONTRIBUTING.md: every new feature flag PR adds a toggle here. Optionally adopt Beagle if rolling-your-own is taking too long.

The before/after

Before

QA reports "the avatar isn't loading on the profile screen." You ask: which build? Tester pastes a screenshot of the version banner. You ask: was the new image-CDN flag on? Tester doesn't know. You ask for the request URL and headers. Tester opens an emulator, can't reproduce, files an incomplete bug. 90 minutes lost.

After

QA opens the debug menu, sees commit SHA + flag state, notes the avatar flag is off, taps the avatar request in Chucker, long-presses to share as curl, pastes the curl + flag state into the bug report. Engineer reproduces it in 2 minutes. Total triage time: 10 minutes.

The pitfalls

Do not gate via runtime flag, gate via source set. An if (isDebug) check at runtime still ships the gated code in the release APK. Use debugImplementation dependencies and src/debug/ source sets so the code literally is not in the release artifact.
Don't ship Chucker without redacted headers. Beta testers' phones get plugged into chargers in coffee shops. A debug build with bearer tokens visible in a notification is one stolen phone away from credential leakage. Redact headers before the first build.
Don't let the debug menu rot. Without ownership it becomes archeology — six obsolete buttons, two broken deep links, one crashing flag. Add a "debug menu spring cleaning" task to your quarterly tech-debt sprint. Or, as a forcing function: the QA lead owns the menu's roadmap.

Exercise

Wire up the debug-build kit on your real project

  1. Add Chucker (real + no-op) to app/build.gradle.kts with redacted Authorization and Cookie headers. Build a debug APK and a release APK. Diff their sizes — the release should be unchanged within 1 KB.
  2. Run apkanalyzer dex packages on the release APK and grep for chuckerteam. It must return nothing. If it does, your no-op artifact isn't wired correctly.
  3. Create app/src/debug/java/.../debug/DebugMenuScreen.kt with at minimum: version name, version code, git SHA (via buildConfigField), API base URL, and a force-crash button.
  4. Wire one debug-only entry point in your nav graph or main screen, gated on BuildConfig.DEBUG.
  5. Hand a fresh debug APK to one teammate. Ask them to find the build's commit SHA without using Logcat. Time it. Aim for under 30 seconds.

Deliverable: a PR adding Chucker, a debug-menu screen, and a CI step that fails the build if release contains debug-only packages. Bonus: a /draft-debug-menu command in .claude/commands/ for next time.
