Chapter 12

Convention Plugins & The CI Quality Gate

Package the whole stack into one plugin; wire it as a PR-blocking CI check — the payoff chapter

The "we have it everywhere" problem

Look at your project right now. If you've followed the previous eleven chapters, every module's build.gradle.kts is bigger than it was — lint config, detekt config, ktlint, the StrictMode wiring, R8 rules, version catalogs. The good kind of bigger. But also the bad kind: the same 30-line block of "configure quality stack" copied into every module's build file.

You added a feature module last month and forgot to copy the lint block over. It was building green for two weeks until a code review noticed a missing HardcodedText error that the rest of the project would have failed on. The standard exists, but it's not enforced; it's opt-in by copy-paste.

This chapter fixes that, permanently. Convention plugins are Gradle's way of packaging build logic into a plugin you apply with one line. Once you have my.app.android.application and my.app.quality, every module gets the entire stack for free, and there's no way to opt out without a deliberate, reviewable change.

Then we wire that stack as a CI quality gate: every PR runs lint + detekt + ktlint + tests + buildHealth, and a failing check blocks the merge button. The 12-chapter journey ends here. After this, "we'll add it later" is structurally impossible — the merge button literally won't work.

The frame for this final chapter: you've been adding tools. Now you're going to operationalize them — one place to configure, one CI gate to enforce, zero copy-paste. The quality stack stops being something engineers must remember to do and becomes something the system does for them.

Meet the tool

Meet Convention Plugins + the CI Quality Gate

What it is: a Gradle build-logic/ included build that hosts custom plugins (e.g. my.app.android.application, my.app.quality) every module applies in one line, plus a GitHub Actions workflow that runs the full quality stack on every PR and gates merging via branch protection.

What it catches that humans miss: "I forgot to copy the lint block to this new module," "this module's detekt config drifted from the rest," "this PR slipped through without ktlintCheck running," "the team has no enforcement for the standard we wrote down."

Install: create build-logic/ as an included build, add a convention module with the kotlin-dsl plugin, register your plugins in gradlePlugin {}, then apply id("my.app.android.application") in modules.

Runtime cost: zero at app runtime. Build-time cost is negative — less repeated config, faster sync, single source of truth for upgrades.

When to skip it: only if you have one module. As soon as you have three modules with the same lint/detekt/ktlint config, this is overdue.

The two ways to do convention plugins

Gradle gives you two mechanisms for shared build logic. Pick one and don't mix.

buildSrc/

The old way. A magic directory at the project root that's automatically a "compile-this-first" project. Compiles every Gradle sync, even when nothing changed. Invalidates the configuration cache for the whole project on every edit. Avoid for new work.

build-logic/ (included build)

The current best practice. A separate Gradle build registered via includeBuild("build-logic") in settings.gradle.kts. Compiles on demand, plays nicely with the configuration cache, can be released as a standalone artifact if you ever scale to multiple repos.

If your project already uses buildSrc/, migration is mostly mechanical: rename the directory to build-logic/convention/, add a settings.gradle.kts in build-logic/, register the included build in your root settings.gradle.kts. The plugin code itself usually doesn't change.

The build-logic/ skeleton

settings.gradle.kts (root) — register the included build

pluginManagement {
    includeBuild("build-logic")
    repositories {
        google()
        mavenCentral()
        gradlePluginPortal()
    }
}

dependencyResolutionManagement {
    repositories {
        google()
        mavenCentral()
    }
}

rootProject.name = "MyApp"
include(":app", ":feature:profile", ":feature:home", ":core:network", ":core:ui")

includeBuild tells Gradle "before configuring this build, configure the one over there and make its plugins available." That's the magic that lets you reference plugins from a sibling directory without publishing them.

build-logic/settings.gradle.kts

dependencyResolutionManagement {
    repositories {
        google()
        mavenCentral()
        gradlePluginPortal()
    }

    // CRITICAL: share the version catalog with the main build
    versionCatalogs {
        create("libs") {
            from(files("../gradle/libs.versions.toml"))
        }
    }
}

rootProject.name = "build-logic"
include(":convention")

Sharing the version catalog means your convention plugins reference the same Kotlin/AGP/Compose versions as your app. Edit one TOML file; everything stays in lockstep.

build-logic/convention/build.gradle.kts

plugins {
    `kotlin-dsl`
}

group = "com.example.buildlogic"

dependencies {
    // Pin the same versions of the AGP & Kotlin plugins your app uses
    compileOnly(libs.android.gradlePlugin)
    compileOnly(libs.kotlin.gradlePlugin)
    compileOnly(libs.detekt.gradlePlugin)
    compileOnly(libs.ktlint.gradlePlugin)
}

gradlePlugin {
    plugins {
        register("androidApplication") {
            id = "my.app.android.application"
            implementationClass = "AndroidApplicationConventionPlugin"
        }
        register("androidLibrary") {
            id = "my.app.android.library"
            implementationClass = "AndroidLibraryConventionPlugin"
        }
        register("quality") {
            id = "my.app.quality"
            implementationClass = "QualityConventionPlugin"
        }
    }
}

The gradlePlugin { plugins { register(...) }} block is what makes your plugin classes addressable by ID. compileOnly rather than implementation — you don't ship Gradle & AGP inside your plugin, the host build provides them.

Your first convention plugin

Plugins live in build-logic/convention/src/main/kotlin/. Convention is to put each plugin's class in the source-set root with no package, but you can package them if you prefer.

AndroidApplicationConventionPlugin.kt

import com.android.build.gradle.LibraryExtension
import com.android.build.api.dsl.ApplicationExtension
import org.gradle.api.Plugin
import org.gradle.api.Project
import org.gradle.kotlin.dsl.configure
import org.gradle.kotlin.dsl.dependencies

class AndroidApplicationConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        // Apply the AGP application plugin and Kotlin
        pluginManager.apply("com.android.application")
        pluginManager.apply("org.jetbrains.kotlin.android")

        extensions.configure<ApplicationExtension> {
            compileSdk = 35
            defaultConfig {
                minSdk = 24
                targetSdk = 35
                vectorDrawables { useSupportLibrary = true }
            }
            buildTypes {
                getByName("release") {
                    isMinifyEnabled = true
                    isShrinkResources = true
                    proguardFiles(
                        getDefaultProguardFile("proguard-android-optimize.txt"),
                        "proguard-rules.pro"
                    )
                }
            }
            buildFeatures {
                buildConfig = true
            }
            compileOptions {
                sourceCompatibility = JavaVersion.VERSION_17
                targetCompatibility = JavaVersion.VERSION_17
            }
        }

        // Apply the quality stack to every Android app module
        pluginManager.apply("my.app.quality")
    }
}

One file. Every Android app module in the repo applies id("my.app.android.application"), gets the same compileSdk, R8 config, Java 17 source level, and the quality stack — for free.

The library variant is the same shape with a few changes: com.android.library instead of com.android.application, LibraryExtension instead of ApplicationExtension, no defaultConfig.targetSdk, no buildTypes R8 (libraries don't ship release builds in the same sense). Otherwise identical.

The quality convention plugin — the payoff

QualityConventionPlugin.kt

import io.gitlab.arturbosch.detekt.extensions.DetektExtension
import org.gradle.api.Plugin
import org.gradle.api.Project
import org.gradle.kotlin.dsl.configure

class QualityConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        pluginManager.apply("io.gitlab.arturbosch.detekt")
        pluginManager.apply("org.jlleitschuh.gradle.ktlint")

        // Detekt: shared config + per-module baseline
        extensions.configure<DetektExtension> {
            config.setFrom(rootProject.file("config/detekt/detekt.yml"))
            baseline = file("$projectDir/detekt-baseline.xml")
            buildUponDefaultConfig = true
            allRules = false
            autoCorrect = true
        }

        // Ktlint: editor config + reporters
        extensions.configure<org.jlleitschuh.gradle.ktlint.KtlintExtension> {
            android.set(true)
            ignoreFailures.set(false)
            reporters {
                reporter(org.jlleitschuh.gradle.ktlint.reporter.ReporterType.HTML)
                reporter(org.jlleitschuh.gradle.ktlint.reporter.ReporterType.PLAIN)
            }
        }

        // Lint: shared lint.xml
        plugins.withId("com.android.base") {
            extensions.configure<com.android.build.api.dsl.CommonExtension<*, *, *, *, *, *>> {
                lint {
                    abortOnError = true
                    warningsAsErrors = false
                    checkReleaseBuilds = true
                    baseline = file("lint-baseline.xml")
                    lintConfig = rootProject.file("config/lint/lint.xml")
                    htmlReport = true
                    xmlReport = true
                }
            }
        }
    }
}

Three tools, one plugin. Apply id("my.app.quality") in any module and you instantly get detekt + ktlint + lint with shared config files at config/detekt/detekt.yml and config/lint/lint.xml. Per-module baselines stay per-module (because legacy code is per-module). Shared rules stay shared.

The shared config files at the root deserve their own note. Convention is:

project-root/
├── build-logic/
│   └── convention/
│       └── src/main/kotlin/
│           ├── AndroidApplicationConventionPlugin.kt
│           ├── AndroidLibraryConventionPlugin.kt
│           └── QualityConventionPlugin.kt
├── config/
│   ├── detekt/detekt.yml
│   └── lint/lint.xml
├── gradle/libs.versions.toml
└── (modules)

The ratio matters. Modules go from 60-line build files (mostly repeated config) to 15-line build files (only the module-specific bits: namespace, applicationId, dependencies). Adding a new module becomes a 30-second copy-paste of an existing one with two strings changed.

The CI quality gate

Convention plugins make the standards declarable. The CI gate makes them enforced. Below is a working GitHub Actions workflow for a real Android project.

.github/workflows/quality.yml

name: Quality

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

concurrency:
  group: quality-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: ${{ github.event_name == 'pull_request' }}

jobs:
  static-analysis:
    name: Static Analysis
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0   # needed for some lint baseline tools

      - uses: actions/setup-java@v4
        with:
          distribution: 'zulu'
          java-version: '17'

      - uses: gradle/actions/setup-gradle@v4
        with:
          cache-read-only: ${{ github.ref != 'refs/heads/main' }}

      - name: Ktlint check
        run: ./gradlew ktlintCheck --continue

      - name: Detekt
        run: ./gradlew detekt --continue

      - name: Lint
        run: ./gradlew :app:lintRelease --continue

      - name: Upload reports
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: quality-reports
          path: |
            **/build/reports/detekt/*.html
            **/build/reports/ktlint/*.html
            **/build/reports/lint-results-*.html
          retention-days: 14

  tests:
    name: Unit tests
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          distribution: 'zulu'
          java-version: '17'
      - uses: gradle/actions/setup-gradle@v4
      - name: Run unit tests
        run: ./gradlew testReleaseUnitTest --continue
      - name: Upload test reports
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-reports
          path: '**/build/reports/tests/'
          retention-days: 14

  build-health:
    name: Dependency health
    runs-on: ubuntu-latest
    timeout-minutes: 15
    if: github.event_name == 'pull_request'
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          distribution: 'zulu'
          java-version: '17'
      - uses: gradle/actions/setup-gradle@v4
      - name: buildHealth (advisory)
        run: ./gradlew buildHealth || true
        # Advisory only until noise floor is low — flip to required later.

Three jobs in parallel. --continue is the trick — it makes Gradle run all the checks even if one fails, so a single PR shows you all the violations at once instead of one-at-a-time. concurrency.cancel-in-progress cancels old PR pushes so you don't burn CI minutes on every force-push.

Branch protection — making the gate real

The workflow above runs on every PR. By default, it doesn't block anything — engineers can merge red builds. Branch protection turns advisory checks into required checks.

GitHub branch protection (Settings → Branches → Add rule)

Branch name pattern: main

✅ Require a pull request before merging
   ✅ Require approvals (1 minimum)
   ✅ Dismiss stale pull request approvals when new commits are pushed

✅ Require status checks to pass before merging
   ✅ Require branches to be up to date before merging
   Required status checks:
     - Static Analysis
     - Unit tests

✅ Require conversation resolution before merging
✅ Do not allow bypassing the above settings (for non-admins)

Note: don't make build-health required from day 1 — the first buildHealth reports on a legacy project are noisy. Keep it advisory until the unused-deps section is empty, then promote it to required.

The "Do not allow bypassing" toggle is what turns this from a guideline into a rule. Without it, admins (which often means tech leads under deadline pressure) can override and the gate becomes performance art. Turn it on. Talk to your team about it first.

Inline PR comments with Reviewdog or Danger

Gradle reports are great if engineers click into the CI log. They almost never do. Surface findings as inline PR comments and review-time engagement skyrockets.

Reviewdog

Lightweight Go-based tool. Hooks ktlint, detekt, lint output into PR comments via the ktlint-action and friends. Five lines of YAML. Recommended starting point.

Danger / DangerKotlin

Heavier framework, more flexible. Lets you write custom rules ("PR title must follow Conventional Commits", "no TODO in modified lines") in a Dangerfile. Worth it for teams ready to invest in PR-bot tooling.

.github/workflows/reviewdog.yml — ktlint inline comments

name: Reviewdog

on: [pull_request]

jobs:
  ktlint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: ScaCap/action-ktlint@master
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          reporter: github-pr-review
          fail_on_error: false
          ktlint_version: 1.5.0

Set fail_on_error: false while rolling this out. Engineers see the comments, get used to fixing them, then you flip to true in a follow-up PR once the team's used to the workflow.

AI-Assisted Setup: /draft-quality-convention-plugin

Setting up build-logic/ from scratch is the kind of mechanical-but-error-prone work where AI shines. Hand it the existing module configs and let it consolidate.

Good prompt

"Create a convention-plugin build-logic/ for this project. Read every module's build.gradle.kts to identify the common config patterns (lint, detekt, ktlint, compileSdk, R8, Java level). Output: (1) build-logic/settings.gradle.kts, (2) build-logic/convention/build.gradle.kts with plugin registrations for my.app.android.application, my.app.android.library, and my.app.quality, (3) the three plugin classes consolidating the common config, (4) a unified diff showing what to remove from each module's build file. Preserve any module-specific overrides as inline overrides in the post-migration files. Use the existing gradle/libs.versions.toml for version references — share the catalog with the included build."

Bad prompt

"Add convention plugins to my project."

Why it fails: doesn't say to read existing configs, doesn't define plugin IDs, doesn't ask for a diff, doesn't constrain to preserve overrides. You'll get a tutorial-quality stub plus zero application.

.claude/commands/draft-quality-convention-plugin.md

# /draft-quality-convention-plugin

Set up build-logic/ as an included build with three convention plugins:
  - my.app.android.application
  - my.app.android.library
  - my.app.quality (lint + detekt + ktlint)

Inputs (read from disk):
  - All module build.gradle.kts files
  - Root settings.gradle.kts
  - gradle/libs.versions.toml
  - Any existing buildSrc/ directory (deprecate it)

Outputs:
  1. build-logic/settings.gradle.kts (sharing the version catalog)
  2. build-logic/convention/build.gradle.kts (kotlin-dsl + plugin registrations)
  3. AndroidApplicationConventionPlugin.kt
  4. AndroidLibraryConventionPlugin.kt
  5. QualityConventionPlugin.kt
  6. Move shared config files: config/detekt/detekt.yml, config/lint/lint.xml
  7. A unified diff showing each module's build.gradle.kts shrunk
     (only namespace, applicationId, dependencies remain)
  8. Updated root settings.gradle.kts with includeBuild("build-logic")

Constraints:
  - Use included build (build-logic/), NOT buildSrc/.
  - All plugin IDs must use the format my.app.X.
  - Preserve any module-specific overrides — flag them as candidates
    for migration into the convention plugin if they appear in 3+ modules.
  - Java 17, AGP from libs.versions.toml.

The 30-day rollout

Day 1 (Week 1)

Stand up build-logic/ as an included build. Write AndroidApplicationConventionPlugin and AndroidLibraryConventionPlugin — just the AGP/Kotlin/Java level config, no quality stack yet. Migrate one leaf module to apply the convention. Verify it builds.

Week 2

Add QualityConventionPlugin consolidating lint + detekt + ktlint. Migrate the rest of the modules. Move shared config files to config/lint/ and config/detekt/. Each module's build.gradle.kts should now be 60-70% shorter.

Week 3

Add the GitHub Actions workflow. Three jobs: static-analysis, tests, build-health (advisory). Run on every PR + push to main. Set --continue for full visibility per PR.

Week 4 (Month 1)

Turn on branch protection: require static-analysis + tests, require approvals, dismiss stale approvals on push. Add Reviewdog for inline ktlint comments (fail_on_error: false initially). Schedule the buildHealth promotion-to-required for 60 days out, once you've cleaned the unused-deps backlog.

The pitfalls

Don't try to do all 12 chapters in 30 days. The course is designed to be adopted one tool at a time, over months. The convention-plugin chapter assumes the previous tools are in place. If you've only adopted lint and detekt, your convention plugin only consolidates lint and detekt — that's still a win.
Don't merge convention-plugin migration with adding new rules. Migration is mechanical — consolidating config that already exists. Adding new rules is a separate concern. Two PRs.
Don't enable branch protection on day one. Run the CI workflow as advisory for at least a sprint. Once team velocity is unaffected and engineers know how to read failures, flip the required-checks toggle. Forced adoption with broken tooling is the fastest way to make the team hate the quality stack.
Don't build a "perfect" convention plugin first. Ship a 60%-good one, migrate two modules, learn what breaks, iterate. Convention plugins are normal code — they get refactored.

Exercise — the capstone

Operationalize the whole stack

  1. Create build-logic/ as an included build with the skeleton from this chapter. Write AndroidApplicationConventionPlugin, AndroidLibraryConventionPlugin, and QualityConventionPlugin.
  2. Migrate at least one app module and one library module to use the convention plugins. Verify both build green.
  3. Move detekt/lint config to config/detekt/ and config/lint/ — no per-module copies of the rules.
  4. Add .github/workflows/quality.yml with three jobs: static-analysis, tests, build-health (advisory). Verify it runs on a test PR.
  5. Configure branch protection on main: require static-analysis & tests, require approvals, do not allow bypassing for non-admins.
  6. Add Reviewdog for inline ktlint comments with fail_on_error: false.
  7. Open a PR titled "Operationalize the quality stack." Body: link to the convention plugin, before/after diff for one module's build.gradle.kts, screenshot of branch protection settings, link to a green CI run.

Deliverable: a repo where the entire 12-chapter quality stack is one id("my.app.android.application") per module away. Bonus: /draft-quality-convention-plugin in .claude/commands/ so the next project you join takes 90 minutes instead of a week.

The closing message

Twelve chapters ago you were the engineer who'd been putting this off. You shipped real apps, debugged real crashes, but never invested in detekt, lint, StrictMode, LeakCanary, version catalogs, R8, convention plugins, or a CI gate. Today's the day that changed.

You went from "we don't have time for quality" to "the quality stack runs on every PR, automatically, with one line per module." The compounding starts now. Every new feature lands on a foundation that catches NPEs, leaks, formatting drift, dependency rot, and APK bloat before they hit users. Every new engineer who joins inherits the standards via the convention plugin, not via a wiki page nobody reads. Every "what's our convention for X?" question has a one-file answer.

The trap of "I'll add quality tooling when I have time" is that it's recursive: you don't have time precisely because you don't have quality tooling. PR reviews drag, regressions creep in, debugging takes hours instead of minutes. Adopting the stack creates the time you needed to adopt the stack. That's the leverage. That's the whole game.

Ship one convention plugin this week. Ship one CI gate next week. The next time you read a "we don't have time for that" thread on your team's Slack, you'll know: yes, you do, and here's how.
