Chapter 7

StrictMode: Find Bugs Before Your Users Do

The 15-year-old Android API that nobody turns on. Ten lines of code, and your app starts shouting at you about every disk read on the main thread, every leaked Activity, every SQLite query that has no business running where it's running.

The runtime layer

You've spent six chapters making the build smarter. Lint, detekt, ktlint, hooks — all running before the app starts. They catch a lot. They cannot catch what only happens when the app is running. Disk reads on the main thread that take 30ms on a Pixel 7 and 800ms on a budget Samsung in Brazil. Activities that leak because a static field never let go. SharedPreferences cold-loading on a UI thread because someone forgot.

That's where StrictMode comes in. It's not a linter; it's a runtime watchdog. You enable it in your Application's onCreate, gate it on BuildConfig.DEBUG, and from that point on Android logs (or crashes) every time your code does something it shouldn't. Disk on UI thread. Network on UI thread. Leaked Cursor. Closeable left open. ProgressBar registered in onCreate and never unregistered.

StrictMode shipped in API 9 (2010). It is one of the longest-standing, most-overlooked tools in Android. The only reason your app doesn't have it on is that nobody on the team flipped the switch.

StrictMode

Android's built-in runtime offender detector for thread and VM policy violations. Free, no dependency, no APK size impact, ships on every device since 2010.

What it catches
ThreadPolicy: disk reads/writes on UI thread, network on UI, custom slow calls. VmPolicy: leaked Activities, leaked closables (Cursors, FileInputStreams), unregistered receivers, file URIs exposed to other apps, cleartext HTTP.
Install
10 lines in Application.onCreate(), guarded by if (BuildConfig.DEBUG). No Gradle dependency. No setup outside the Application class.
Runtime cost
Negligible in most modes. penaltyLog just writes to logcat. penaltyDeath crashes the process — use it in dev when you want zero tolerance. Never enable in release builds.
When to skip it
Never in dev builds. Always in release builds (the cost is fine, but the crashes from unfixed third-party SDK violations are not).

What "violation" actually means

StrictMode has two policies, and both are configured separately.

  • ThreadPolicy watches the UI thread. If anything blocking happens on it — disk read, disk write, network call, a custom call you flagged as slow — you get a violation. The bar is intentionally low: even a 5ms disk read is a violation. The point is to flush them all out so you make conscious decisions.
  • VmPolicy watches the process. If an Activity is GC-eligible and its reference count is wrong, if a Cursor never had close() called, if a BroadcastReceiver was registered and never unregistered, that's a violation.

Each violation has a stack trace. The line number near the top is usually not the offender — it's framework code. The offender is somewhere 5–10 frames deeper. Reading the trace is a skill; we'll cover that later in the chapter.

The 10-line install

Open your Application class. If you don't have one, make one (and register it in the manifest). The whole setup is in onCreate.

app/src/main/java/com/example/MyApp.kt
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        if (BuildConfig.DEBUG) {
            enableStrictMode()
        }
    }

    private fun enableStrictMode() {
        // ThreadPolicy: catch slow operations on the UI thread.
        StrictMode.setThreadPolicy(
            StrictMode.ThreadPolicy.Builder()
                .detectDiskReads()
                .detectDiskWrites()
                .detectNetwork()
                .detectCustomSlowCalls()
                .detectResourceMismatches()
                .penaltyLog()         // Start with log; promote to death later.
                .build()
        )

        // VmPolicy: catch leaks and lifecycle violations.
        StrictMode.setVmPolicy(
            StrictMode.VmPolicy.Builder()
                .detectLeakedSqlLiteObjects()
                .detectLeakedClosableObjects()
                .detectLeakedRegistrationObjects()
                .detectActivityLeaks()
                .detectFileUriExposure()
                .detectCleartextNetwork()
                .penaltyLog()
                .build()
        )
    }
}
The BuildConfig.DEBUG guard is non-negotiable. penaltyDeath on a release build means a third-party SDK violation crashes your app for users. Always wrap StrictMode in a debug check.

Register the Application in the manifest if it isn't already.

app/src/main/AndroidManifest.xml
<application
    android:name=".MyApp"
    android:label="@string/app_name"
    ... >
    <!-- activities, services, etc. -->
</application>
Without android:name, your custom Application is never instantiated, and StrictMode is never enabled. Easy to miss; classic "I added the code but it doesn't work" trap.

The penalty knob: log, death, dropbox

Detection is one half. The other half is what happens when a violation fires. StrictMode offers three.

.penaltyLog()
Writes the violation + stack trace to logcat. App keeps running. Use this on Day 1. You'll be flooded; you need to triage without crashing every screen.
.penaltyDeath()
Crashes the process the moment a violation fires. Zero tolerance. Use this in Week 2 once your top offenders are fixed and you want regressions to be loud and obvious.

There's also .penaltyDropBox() — sends violations to Android's DropBox manager (a system log). Useful in CI integration tests where you want a structured artifact, less useful for day-to-day. .penaltyDialog() exists too — pops a dialog on every violation — which is amusing for about 30 seconds before you turn it back off.

The natural progression is penaltyLog → (fix top 5 offenders) → penaltyDeath. Don't start at penaltyDeath. The first run on a real codebase will crash before the launcher activity finishes loading, and you'll think StrictMode is broken when actually your app is.

Reading a violation

You enable StrictMode, run the app, scroll logcat. You see something like this:

D/StrictMode: StrictMode policy violation; ~duration=143 ms:
android.os.strictmode.DiskReadViolation
    at android.os.StrictMode$AndroidBlockGuardPolicy.onReadFromDisk
    at java.io.UnixFileSystem.checkAccess
    at java.io.File.exists
    at android.app.SharedPreferencesImpl.<init>
    at android.app.ContextImpl.getSharedPreferences
    at com.example.auth.TokenStore.getToken(TokenStore.kt:23)
    at com.example.auth.AuthInterceptor.intercept(AuthInterceptor.kt:14)
    at okhttp3.internal.http.RealInterceptorChain.proceed
    ...
    at com.example.MainActivity.onCreate(MainActivity.kt:42)

The top of the trace is framework code: StrictMode$AndroidBlockGuardPolicy, UnixFileSystem, SharedPreferencesImpl. That's not your bug. Scroll down past the framework frames until you hit the first one in your package — com.example.auth.TokenStore.getToken(TokenStore.kt:23). That's the call site. Above it: AuthInterceptor, onCreate. The trace tells you a story: "MainActivity.onCreate triggered an OkHttp interceptor that called TokenStore.getToken which loaded SharedPreferences which read from disk — on the UI thread."

Find the first frame in your package. That's almost always the right line to fix. Frameworks are doing what they advertise; your code is the one calling them in the wrong place. The fix is usually "move this off the main thread" — a coroutine, a Dispatchers.IO, a WorkManager.

The legitimate suppression

Sometimes you genuinely have to do disk I/O on the main thread. App-startup config that downstream code depends on. A migration that has to complete before any UI shows. For these cases, StrictMode provides a scoped suppression.

val token = StrictMode.allowThreadDiskReads().use {
    sharedPrefs.getString("auth_token", null)
}

This says: "I know I'm reading from disk on the UI thread; I've thought about it; this specific block is OK." StrictMode disables the policy for the lifetime of the block and re-enables it afterward. The use block ensures the restore happens even if the code throws.

Don't bulk-suppress. The anti-pattern is "the team had a bad week with StrictMode crashes, so we wrapped setupApp() in allowThreadDiskReads and moved on." That kills the entire purpose. Suppress only the specific call you've decided is OK; never a whole function. If you find yourself suppressing more than 5 spots in your codebase, you have a real architectural problem (probably synchronous I/O in the wrong place) that suppression is hiding.

StrictMode in instrumented tests

Logcat is a developer signal. CI is a regression signal. You can wire StrictMode into instrumented tests so any new violation fails the build. The trick is to set policy per test and assert it.

app/src/androidTest/java/.../StrictModeRule.kt
class StrictModeRule : TestRule {
    override fun apply(base: Statement, description: Description): Statement =
        object : Statement() {
            override fun evaluate() {
                val originalThread = StrictMode.getThreadPolicy()
                val originalVm = StrictMode.getVmPolicy()
                StrictMode.setThreadPolicy(
                    StrictMode.ThreadPolicy.Builder()
                        .detectAll()
                        .penaltyDeath()
                        .build()
                )
                StrictMode.setVmPolicy(
                    StrictMode.VmPolicy.Builder()
                        .detectAll()
                        .penaltyDeath()
                        .build()
                )
                try {
                    base.evaluate()
                } finally {
                    StrictMode.setThreadPolicy(originalThread)
                    StrictMode.setVmPolicy(originalVm)
                }
            }
        }
}

class LoginFlowTest {
    @get:Rule val strictMode = StrictModeRule()

    @Test fun login_doesNotBlockMainThread() {
        // any I/O on the UI thread crashes this test
        ActivityScenario.launch(LoginActivity::class.java).use {
            onView(withId(R.id.email)).perform(typeText("a@b.com"))
            onView(withId(R.id.signInButton)).perform(click())
            // ...
        }
    }
}
The rule saves and restores policy so tests are independent. detectAll() + penaltyDeath() in tests is the strongest possible regression guard — any new violation crashes the test instantly.

What you'll see on Day 1

The first run after enabling StrictMode is alarming. Your launcher Activity will trigger 6–15 violations before the user even sees the home screen. SharedPreferences cold loads, lazy Retrofit init that does a DNS lookup, a Bitmap decode in a static initializer, a database migration check — all of it on the UI thread.

This is not a sign your app is bad. It's a sign your app is normal. Every Android codebase that hasn't run StrictMode looks like this. The fix is iterative: triage by frequency (which violation fires the most?), fix the top 3, repeat. After a couple weeks, the count drops to a handful that you've consciously decided to live with.

Day 1 Add the 10-line setup in your Application class, guarded by BuildConfig.DEBUG, with .penaltyLog() on both policies. Run the app for 5 minutes of normal use. Save the logcat output. Run /explain-strictmode-violation on the top 5 by frequency.
Week 1 Fix the top 5. Move SharedPreferences off-main (DataStore is the right answer for new code). Replace commit() with apply(). Wrap legitimate exceptions in allowThreadDiskReads { } with a comment. Promote ThreadPolicy to .penaltyDeath(); keep VmPolicy on log.
Month 1 Promote VmPolicy to death too. Add a StrictModeRule to the instrumented test suite for your top 3 user flows (login, navigation, checkout). Document any remaining suppressions in a STRICTMODE_SUPPRESSIONS.md doc with reasons.

The "untrip and restore" anti-pattern

This deserves its own section because it's how teams quietly kill StrictMode. Here's the pattern:

  1. Team enables StrictMode with penaltyDeath.
  2. App crashes constantly during testing — way more than expected.
  3. Frustrated dev wraps StrictMode.allowThreadDiskReads around the entire onCreate of every Activity to "buy time."
  4. Six months later, every onCreate is wrapped, nobody remembers why, and StrictMode is effectively off.
  5. The codebase has accumulated a year of new violations nobody noticed.

The right answer when StrictMode is too noisy: start with penaltyLog, not death. Triage. Fix. Then tighten. Wholesale suppression is the developer equivalent of putting tape over a smoke alarm.

Why StrictMode is leverage

Your users will eventually see slow main-thread I/O as ANRs — the dreaded "App not responding" dialog. Without StrictMode, you find out from the Play Console's vitals report, weeks after the offending PR landed. With StrictMode, you find out the moment the dev opens the screen on their own machine. Same bug, two timelines: 6 weeks and a 1-star review, or 5 minutes and a coroutine dispatch.

Common pitfalls

  • Forgetting the manifest android:name. Code is in onCreate, but the Application class never runs because the manifest still points to the default. Verify with a logcat statement.
  • Enabling in release builds. A third-party SDK violation crashes for real users. Always gate on BuildConfig.DEBUG.
  • Confusing ThreadPolicy with VmPolicy. ThreadPolicy is per-thread (mostly UI). VmPolicy is process-wide. You set both because they catch different bugs.
  • Reading the top of the stack trace. The top is always framework. The line to fix is the first one in your package, usually 5–10 frames down.
  • penaltyDeath on Day 1. The launcher Activity won't survive. Start with log; promote when the count is manageable.
  • Wrapping suppression around whole functions. Suppress the specific call, not the whole block. Otherwise StrictMode is theater.

StrictMode and Compose

Compose surfaces some unique violations. painterResource(R.drawable.large_image) can trigger a DiskReadViolation because the image is loaded synchronously the first time. remember { File(...).readText() } in a Composable is a guaranteed violation — the read runs on the composition thread (which is usually main).

The general rule: if a Composable reads anything off disk or network, you're doing it wrong. Put the I/O in a ViewModel + a StateFlow, expose state to the Composable. Compose isn't different from XML in this regard; it just makes the violations more concentrated because Composables run frequently.

Your turn.
  1. Open your Application class. If it doesn't exist, create one and register it in the manifest with android:name.
  2. Add the 10-line StrictMode setup from the recipe above. Use .penaltyLog() on both policies. Build and install the debug APK.
  3. Use the app for 5 minutes of normal flow: launch, sign in, navigate to two main screens, sign out. Capture adb logcat | grep StrictMode the whole time.
  4. Sort the captured violations by frequency. Pick the top 3.
  5. For each one, run the /explain-strictmode-violation skill. Get a diagnosis and a fix.
  6. Apply the smallest fix you're comfortable with. Re-run the app. Confirm the violation no longer fires.
  7. Open a PR with the StrictMode setup + the 3 fixes. Title: "Enable StrictMode in debug builds + fix 3 main-thread offenders."

Stretch goal: pick one user flow (login, settings, etc.) and write a StrictModeRule-based instrumented test that asserts no violations occur during that flow. That's regression protection forever.

What's next

StrictMode catches main-thread offenses and obvious leaks. But there's a category of leaks even StrictMode struggles with — the slow ones, where an Activity is held by something five references deep, garbage collection runs and runs, and the leak grows quietly until the app OOMs at hour two of usage. Chapter 8 brings in LeakCanary — Square's heap-watcher that automatically detects these, captures a snapshot, and tells you exactly which reference is the culprit.
