Chapter 9

Debug-Build Inspectors

Memory Map
🔧

The Big Idea

Turn your debug build into a tool, not a slower release. Chucker for HTTP, a debug menu for state, debug-only feature flags — all delivered through debugImplementation and src/debug/ so the release APK is byte-identical to before.

🧰 The Three-Piece Kit

🌐
Chucker
OkHttp interceptor that records every request & response. Tap the notification, see the body, share as curl.
📋
Debug Menu
Hidden screen with build info, force-crash, clear-data, deep-link launcher. Lives in src/debug/.
🚩
Flag Overrides
Local override layer in front of remote config. Flip flags in 3 seconds, no PM coordination.

🚪 Three Gradle Configurations You Live On

ConfigurationWhere it shipsUse for
implementation Every variant (incl. release) Production code only
debugImplementation Debug variant only Chucker, debug-menu code, LeakCanary
releaseImplementation Release variant only The matching no-op artifact

🎯 The No-Op Pattern

📞 Call site
chucker.intercept(...)
→
🔀 Build variant decides
→
🐛 Debug: real lib runs
→
🚀 Release: no-op silently does nothing

Same call site, no if (BuildConfig.DEBUG) wrappers, smaller release APK.

🌐 Chucker Setup — 4 Steps

  1. Add the dependency pair

    debugImplementation the real library + releaseImplementation the no-op artifact at the same version.

  2. Plug into OkHttpClient

    OkHttpClient.Builder().addInterceptor(chuckerInterceptor) — last in the chain so it captures the final outgoing request.

  3. Redact secrets

    .redactHeaders("Authorization", "Cookie") — non-negotiable. Set retention to 1 hour, max content 250 KB.

  4. Verify the release APK

    apkanalyzer dex packages + grep for chuckerteam. Must return nothing.

🔐 Source-Set Gating Beats Runtime Gating

❌ if (BuildConfig.DEBUG)
✅ debugImplementation
Source-Set Gate
✅ src/debug/java/
✅ no-op releaseImplementation

Runtime check still ships the code. Source set guarantees it isn't in the release artifact at all.

📚 Key Terms

Source set
A directory whose contents are compiled only for a specific build variant (e.g. src/debug/java/).
No-op artifact
A library variant that exposes the same API as the real one but with empty implementations — ships in release with zero behavior.
OkHttp interceptor
A function that wraps every HTTP request/response — where Chucker plugs in.
Debug menu
A hidden in-app screen exposing build info, flag overrides, and dev shortcuts — gated by build variant.
Feature-flag override
A local-only value that takes precedence over remote-config for the current debug build.
buildConfigField
Gradle DSL that injects a constant into the generated BuildConfig class at compile time.
apkanalyzer
Android SDK CLI that lists DEX packages, manifest entries, and resource sizes inside an APK or AAB.

🚫 Three Things to Never Do

⛔
Runtime-only gating
An if (isDebug) still ships the code. Use source sets, not flags.
🔓
Unredacted headers
Bearer tokens visible in a tester's notification shade is a credential leak waiting to happen.
👻
Unowned debug menu
Without an owner it rots into broken buttons and obsolete deep links. Assign someone.

🧪 Quick Quiz

Why pair a real library on debugImplementation with a no-op on releaseImplementation?

Same API in both variants — call sites compile and run unchanged in release without if (BuildConfig.DEBUG) wrappers, and the real library never enters the release APK.

Where should DebugMenuScreen.kt live in the project?

Under app/src/debug/java/..., never src/main/. The source-set boundary makes it impossible for the file to compile into a release build.

What single CLI check verifies no debug code leaked into release?

apkanalyzer dex packages app-release.apk | grep chuckerteam — should return nothing. Wire it as a CI step.

Why redact Authorization and Cookie headers in Chucker?

Anyone with the debug APK on their phone can open Chucker and read every request body. A stolen tester phone with bearer tokens visible is a credential leak.

What does buildFeatures.buildConfig = true do, and when do you need it?

Enables BuildConfig generation including buildConfigField entries. Required from AGP 8.0+ — the default flipped to false.

Key Takeaways

  • Three pieces: Chucker (HTTP) + debug menu (state) + flag overrides (behavior).
  • Gate via source set (src/debug/ + debugImplementation) — not via runtime if (DEBUG).
  • Real library on debugImplementation, no-op artifact on releaseImplementation — same API, zero release cost.
  • Redact Authorization and Cookie headers in Chucker before the first build.
  • Verify with apkanalyzer + grep that no debug packages leaked into release. Wire as CI.
  • Debug menu must show: build info, flag overrides, force-crash, clear-data, deep-link launcher.
  • Reusable skill: /draft-debug-menu — scaffolds a Compose debug menu in under a minute.
  • Assign an owner to the debug menu, or it rots into broken buttons within a quarter.
