The Big Idea

Package the whole stack into one Gradle plugin every module applies in one line. Wire a CI workflow that runs it on every PR. Turn on branch protection. The standard stops being something engineers must remember and becomes something the system enforces.

🔐 Branch Protection — Making it Real

  1. Require PR & approvals

    1+ reviewer; dismiss stale approvals on push.

  2. Require status checks

    Static Analysis + Unit tests required. build-health stays advisory at first.

  3. Require up-to-date branches

    Forces rebase before merge — checks always run on the merged state.

  4. "Do not allow bypassing"

    The toggle that turns guideline into rule. Without it, admins can override and the gate becomes theater.

📚 Key Terms

Convention plugin
A Gradle plugin that consolidates shared build logic so every module applies it in one line.
Included build
A separate Gradle build registered via includeBuild() — the modern alternative to buildSrc/.
kotlin-dsl
The Gradle plugin that lets you write convention plugins in idiomatic Kotlin.
Plugin ID
The string used in plugins {} blocks (e.g. my.app.android.application) — registered in gradlePlugin {}.
CI quality gate
A CI workflow whose checks must pass before a PR can merge — enforced via branch protection.
Branch protection
GitHub repo settings that enforce required reviewers, required status checks, and disallow bypassing.
Reviewdog
A tool that surfaces ktlint/detekt/lint findings as inline PR comments instead of buried CI logs.
--continue
Gradle flag that runs all checks even after one fails — surfaces every violation in a single CI run.

🧪 Quick Quiz

Why use build-logic/ instead of buildSrc/?

buildSrc/ compiles every Gradle sync and invalidates the configuration cache on every edit. build-logic/ as an included build compiles on demand and plays nicely with the cache.

How do you share the version catalog with the included build?

In build-logic/settings.gradle.kts: versionCatalogs { create("libs") { from(files("../gradle/libs.versions.toml")) } }.

What's the one critical branch-protection toggle?

"Do not allow bypassing the above settings (for non-admins)." Without it, admins can override and the gate becomes theater.

Why use --continue in CI Gradle calls?

It runs all the checks even after one fails — engineers see every violation per PR instead of a one-at-a-time fix-rerun loop.

Why register my.app.quality separately and have the application plugin apply it?

Composition. Tests-only Kotlin modules can apply just my.app.quality without the Android plugin. Application/library plugins layer Android-specific config on top.

Key Takeaways

  • Use build-logic/ as an included build — not buildSrc/. Faster, cache-friendly, scalable.
  • Three plugins minimum: my.app.android.application, my.app.android.library, my.app.quality.
  • Share the version catalog with the included build: one TOML, everything in lockstep.
  • Module build files shrink from 60-80 lines to 15-20 lines (namespace + applicationId + dependencies).
  • CI workflow: 3 jobs (static-analysis, tests, build-health). --continue for visibility. Cancel old PR runs.
  • Branch protection: required checks + required reviewers + dismiss stale + no bypassing.
  • Roll out advisory first. Sprint of soft-running. Then flip required.
  • Reviewdog or Danger for inline PR comments — engagement skyrockets when findings appear at the diff line.
  • The 12-chapter journey ends here: every PR runs the full stack, every standard is enforced, "we don't have time" becomes structurally impossible.
