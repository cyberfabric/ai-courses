Chapter 10

Version Catalogs & Dependency Health

Memory Map
📚

The Big Idea

One libs.versions.toml file becomes the source of truth for every dependency and version in your project. Two plugins (dependency-analysis + versions) tell you which deps are unused, misused, or outdated. Bot-driven PRs (Renovate / Dependabot) keep them current.

📦 The Four Sections of libs.versions.toml

🔢
[versions]
Named version constants. kotlin = "2.0.21" — referenced by the rest of the file.
📚
[libraries]
Module + version pairs. retrofit = { module = "com.squareup.retrofit2:retrofit", version.ref = "retrofit" }
🔌
[plugins]
Plugin IDs + versions. Consumed via alias(libs.plugins.X) in plugins {} blocks.
🎁
[bundles]
Named groups of libraries pulled in with one line. Networking, Compose UI, testing.

🔁 The Naming Rule

📛 TOML key
androidx-core-ktx
→
⚙️ Gradle DSL
libs.androidx.core.ktx

Hyphens in TOML keys become dots in build.gradle.kts. Pick short, hyphenated names — not the verbose form.

🧹 Migration Order — Don't Skip Steps

  1. Create the catalog

    Copy every existing dep + version into gradle/libs.versions.toml verbatim. No upgrades yet.

  2. Migrate one leaf module first

    Smallest module, not :app. Build it. If it builds, your catalog entries are right.

  3. Migrate the rest, module by module

    Build between each. One PR per migration unit if the diff gets big.

  4. Migrate the plugins block

    Top-level build.gradle.kts uses alias(libs.plugins.X) instead of id() version.

  5. Delete the old version constants

    buildSrc/Versions.kt, ext {} blocks, hardcoded strings — all gone.

  6. Now consider upgrades

    Separate PR. Don't mix migration with version bumps.

🔍 The Two Health Plugins

PluginQuestion it answersTask
com.autonomousapps.dependency-analysis Which deps are unused, misused, or transitively pulled? ./gradlew buildHealth
com.github.ben-manes.versions Which deps have newer stable versions available? ./gradlew dependencyUpdates -Drevision=release

📊 Three Categories of buildHealth Advice

🗑️
Remove
Unused dependencies. Pure deletion. Lowest risk, biggest APK win.
🔄
Change
api → implementation or vice versa. Catches leaky abstractions across modules.
➕
Add
Transitive deps you should declare directly — protects you from upstream removal.

📚 Key Terms

Version catalog
A Gradle-native, type-safe registry of dependency coordinates and versions, declared once in libs.versions.toml.
Bundle
A named group of library entries you can pull in with a single libs.bundles.X reference.
BOM (Bill of Materials)
A "platform" artifact that pins compatible versions of related libraries (e.g. compose-bom).
buildHealth
The dependency-analysis plugin task that audits your dependency declarations across all modules.
api vs implementation
Two Gradle configurations — api exposes a dep to consuming modules, implementation hides it. Picking wrong leaks abstractions.
Transitive dep
A dependency you don't declare directly but inherit through another dep. Use it > declare it directly so upstream changes don't break you.
Dependabot Alerts
GitHub's free CVE-notification feature for repo dependencies, distinct from the upgrade-PR Dependabot.
rejectVersionIf
The filter on the versions plugin that rejects alpha/RC candidates when your current version is stable.

🚫 Pitfalls to Skip

⛔
Don't migrate & upgrade together
Two PRs. Migration is mechanical; upgrades carry risk. Don't conflate.
📚
Don't fix all of buildHealth at once
First report has hundreds of items. Tackle "remove" first — lowest risk.
🌊
Don't skip Dependabot grouping
Ungrouped Dependabot opens 50 PRs/week. Group from day one or get drowned.

🧪 Quick Quiz

Where does the version catalog file live, and does it need registration?

gradle/libs.versions.toml. Gradle 7.0+ picks it up automatically — no settings.gradle.kts registration needed.

How does TOML key androidx-core-ktx look in build.gradle.kts?

libs.androidx.core.ktx. Hyphens become dots.

Why declare compose-bom as platform(libs.compose.bom) and the individual artifacts without versions?

The BOM pins compatible versions for all Compose artifacts in lockstep. You upgrade the BOM once, every Compose dep follows.

What are the three categories of advice from buildHealth?

Remove (unused), Change (api/implementation reshuffle), Add (transitive deps used directly).

What's the one filter you must add to the versions plugin and why?

rejectVersionIf { !isStable(candidate.version) && isStable(currentVersion) } — otherwise the report includes alphas/snapshots and becomes pure noise.

Key Takeaways

  • One gradle/libs.versions.toml = source of truth for every dep + version. Picked up automatically by Gradle 7+.
  • Four sections: [versions], [libraries], [plugins], [bundles].
  • Hyphens in TOML keys become dots in DSL. Pick short, hyphenated names.
  • Migrate before upgrading. Two PRs, two diffs, two reviews. Faster overall.
  • dependency-analysis-gradle-plugin → finds unused, misused, and transitive deps via ./gradlew buildHealth.
  • ben-manes/versions → finds outdated deps. Add the isStable filter or drown in alpha noise.
  • CVE default: GitHub Dependabot Alerts. Step up to OWASP DC for build gating.
  • Dependabot or Renovate → weekly grouped PRs. Group, or you'll get 50 PRs/week.
  • Reusable skill: /migrate-to-version-catalog — one-shot for any new project.
