Auditing & Baselining

Memory Map
💡

The Big Idea

Spend 30 minutes looking before you install anything. The output is a one-page quality-debt.md. The unlock for adoption on a legacy codebase is the baseline file — freeze the past, enforce the future.

🔍 The 30-Minute Audit (3 Commands)

📦
:app:dependencies
Reveals direct vs. transitive deps, version conflicts, duplicate libraries (e.g., gson AND moshi).
🔎
:app:lintDebug
HTML report at app/build/reports/. What lint-with-defaults already thinks of your code.
📋
tasks --group verification
Surfaces what quality gates already exist. On legacy projects, the list is short.
⏱️
~5 min run, ~25 min read
Runtime is fast. Reading carefully is the work. Don't fix anything yet — just see.

📂 The Three Categories of Quality Debt

📝 Codebase debt
lint · smells · format drift · dead code
📦 Dependency debt
outdated · unused · CVEs · bloat
Quality Debt
⚙️ Process debt

🎯 Pick the loudest

Each category has a different toolset and a different rollout strategy. Sort findings by category before recommending a starting point.

🪜 Baseline File: IS vs IS NOT

The baseline IS... The baseline IS NOT...
A snapshot of existing violations A TODO list to fix later
Committed to the repo, versioned Regenerated every CI run
"We will not regress" "We'll get to it eventually"
For rules you agree with but can't fix all of yet For silencing rules you disagree with (just disable them)
Shrinks naturally as people fix incidentally Permanent or sacred
An XML/text file you can diff in PRs A black box

🗓️ The Audit Rollout

  1. Day 1 — Run + Write

    Three Gradle commands. Skim the lint HTML. Skim deps. Write the one-page quality-debt.md. Open the audit PR. 45 minutes.

  2. Week 1 — Install First Tool

    Install whichever tool you recommended. Generate baseline. Wire into CI. Append to the adoption-log table in quality-debt.md.

  3. Month 1 — Re-Audit

    Re-run the audit. Compare numbers. The lint count should be flat or down; the process-gaps list should have one fewer entry. 30 min/month.

🎯 Picking Your Starting Tool

If your audit screams... Start with Chapter
Style debates on every PR Ktlint + Spotless 5
Mystery release crashes R8 + Crashlytics mapping 11
Unsafe nullables, complex methods Detekt 4
Memory leaks after long sessions LeakCanary 8
Frame drops, ANRs, jank StrictMode 7
Dependency mess, version drift Version catalogs + dep-analysis 10

If two apply, pick the one with the highest team-morale impact. Style debates feel terrible; eliminating them earns social capital.

🔑 Key Terms

Audit
A 45-minute exercise that produces a one-page quality-debt.md. Not a deep-dive — a forcing function to describe problems in writing.
Baseline file
A committed snapshot of existing violations that the tool ignores on subsequent runs. The unlock for legacy adoption.
Codebase debt
Lint findings, Kotlin smells, formatting drift, dead code. Surfaced by static analysis.
Dependency debt
Outdated, unused, vulnerable, or duplicated libraries. Surfaced by dep-analysis tools.
Process debt
No CI gate, no formatter, no hooks, no branch protection. The reason "we already had detekt" turned into "nobody runs detekt."
quality-debt.md
The single document that lives in your repo: top 10 lint categories, top 5 dep risks, process gaps, recommendation. Plus a growing adoption log.
Cleanup-PR trap
Trying to fix all the violations in one giant PR. Always blows up. Fix incidentally over weeks instead.
/audit-android-project
The reusable Claude Code skill that takes your build files and produces the one-page audit automatically.

⚠️ Audit Pitfalls

🌳
Mistaking transitive for direct
A library you don't recognize is usually pulled in by something else. Remove the parent, not the child.
📊
Misreading lint counts
412 HardcodedText findings might live in 4 screens. Severity beats raw count.
📅
"Old" ≠ "risky"
A 2-year-old lib that hasn't moved is fine. Risk lives where the lib HAS moved and you haven't.
🛠️
Fixing instead of describing
If you're past 45 min, stop. The audit describes; the next 11 chapters fix.

🧪 Quick Quiz

Test yourself

What three commands give you the audit?

./gradlew :app:dependencies --configuration releaseRuntimeClasspath, ./gradlew :app:lintDebug, and ./gradlew tasks --group verification. Total runtime ~5 minutes.

What's the difference between baseline IS and baseline IS NOT?

A baseline IS "we will not regress." A baseline IS NOT "we'll fix this later." Treating it as a TODO list is the path to never fixing it.

What are the three categories of quality debt?

Codebase debt (lint findings, smells), dependency debt (outdated/unused libs), process debt (no CI gate, no formatter, no hooks). Each has its own toolset.

Why is the cleanup PR an anti-pattern?

It's always too big, the reviewer is overwhelmed, subtle behavior changes slip in, it never merges, and quality tooling becomes associated with that bad experience. Fix incidentally instead.

What does the audit deliverable look like?

One page. docs/quality-debt.md. Sections: top 10 lint categories, top 5 dep risks, process gaps, recommendation. Plus an adoption log that grows over time.

Key Takeaways

  • Audit before you install. Three Gradle commands, 45 minutes, one page.
  • Sort findings into codebase debt · dependency debt · process debt — each gets a different tool.
  • The baseline file is the unlock: freeze existing violations, enforce on new code only.
  • Baseline IS "we will not regress." Baseline IS NOT "we'll fix this later."
  • Universal workflow: install → generate baseline → commit → wire CI → ship.
  • Pick the starting tool by which debt is loudest — usually ktlint + Spotless for the team-morale win.
  • Skip the cleanup PR. Fix violations incidentally over months, not in one sprint.
  • Save /audit-android-project as a reusable Claude Code skill — you'll use it on every new repo.
