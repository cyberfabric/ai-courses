Pre-commit & Local Loop

Memory Map
⚡

The Big Idea

The same check that takes 7 minutes in CI takes 300 milliseconds locally. Move every cheap check to git commit; demote the rest to git push; let CI be the safety net, not the workhorse.

Each rung catches what the rung below missed; each is faster than the rung above.

⏱️ Latency budget per rung

⌨️
IDE inspection
Live red squiggles as you type. 0 sec. Lint inspections, ktlint inline highlights.
💾
Format on save
Reformat + optimize imports on Ctrl+S. ~50 ms. Catches 95% of style issues.
📦
Pre-commit
Spotless + detekt on staged files only. ~300 ms. Catches the remaining 5%.
🚀
Pre-push
Full lint, full detekt, spotlessCheck across the repo. under 2 min. Catches what scope-limiting hid.
🛡️
CI gate
Same checks again on a clean machine. ~5 min. Safety net, not the front line.

🎯 Picking the right framework

🥇
lefthook (recommended)
Single Go binary. No language deps. Fast, parallel. Best for Android-only repos.
🥈
husky
Node-based. Use if your repo already has a JS/TS layer (web, RN). Otherwise it drags Node in for hooks.
🥉
pre-commit
Python-based. Rich plugin ecosystem; slower startup. Best for polyglot orgs that already run Python.
🚫
Native .git/hooks
Don't. Not in source control = new hires forget = quality theater.

📋 Lefthook setup — in 3 steps

  1. Install the binary

    brew install lefthook (Mac) or scoop install lefthook (Windows). One-time, per machine.

  2. Wire it into the repo

    lefthook install from repo root. Writes the actual .git/hooks/* scripts that delegate to lefthook.

  3. Commit lefthook.yml

    The config at the repo root. Add the install lines to README so new contributors run them on first checkout.

🔑 Magic tokens & flags

{staged_files}
Lefthook substitutes this with the list of files Git is about to commit. Pass it to your task to scope work to just what's changing.
stage_fixed: true
If the hook modified files (e.g., spotlessApply), auto-stage them so the cleaned version is what gets committed.
parallel: true
Run multiple commands in the same hook simultaneously. Cuts pre-push latency by 2–3×.
glob
Filter {staged_files} by pattern (*.{kt,kts}). Skips the hook entirely when no matching files are staged.
commit-msg hook
Validates the commit message itself. Useful for enforcing Conventional Commits (feat(auth): ...) without a bot.
--no-verify
Git's escape hatch that bypasses all hooks. Don't ban it — treat its frequency as a quality signal that hooks are too slow or wrong.

⚡ Pre-commit vs pre-push split

Pre-commit (under 2 sec)Pre-push (under 2 min)
spotlessApply on {staged_files}spotlessCheck (full repo)
detekt on {staged_files}detekt (all modules)
commit-msg format check:app:lintDebug
(no full lint):app:assembleDebug (compile sanity)
(no unit tests)unit tests of changed modules

If pre-push runs over 2 minutes, demote the slowest task to CI. Past 2 min, devs --no-verify their way around it — and the local layer becomes useless.

🚀 Speeding up Gradle hooks

🔁
org.gradle.daemon=true
Reuse the JVM between invocations. Cold starts cost ~3 seconds; warm starts are sub-second.
🗄️
configuration-cache=true
Caches the configuration phase. The single biggest hook-latency win on Gradle 8+.
🧵
parallel=true + caching=true
Run modules in parallel; reuse outputs of unchanged tasks. Compounds with the daemon.
🎯
Scope to staged files
Pass {staged_files}. Don't run the whole-repo task in pre-commit; that's pre-push's job.

🚨 The --no-verify signal

FrequencyWhat it means
Once a quarter (per dev)Genuine emergency. The escape hatch exists for a reason.
Once a week (same dev)Hooks too slow or rules don't fit reality. Investigate which.
Silently across the teamYour hooks are theater. One slow check is the culprit; find and fix.

The right answer is never "ban it." It's "make hooks fast and accurate enough that nobody wants to skip them."

⚠️ Pitfalls to dodge

🏚️
Whole-repo task in pre-commit
Running unscoped detekt on every commit = 30 sec per commit = devs skip the hook within a week.
🗑️
Forgetting stage_fixed
Spotless rewrites a file but the un-rewritten version is what commits. Five commits later CI still complains.
🐢
4-minute pre-push
Devs --no-verify around it. Demote slow tasks to CI — pre-push is for "fast enough to wait through."
🧪
Tests in pre-commit
Unit tests are great, just not on every commit. Pre-commit is for milliseconds; tests live in pre-push at the earliest.

Quick Quiz

Why not just use .git/hooks/pre-commit directly?

Because .git/ is local-only and never checked in. Every developer has to install hooks manually; new hires forget; CI agents don't have them. A hook framework puts the config in source control and standardizes setup with one command.

What does {staged_files} do?

Lefthook substitutes it with the list of files Git is about to commit. Passing it to your task scopes the work to just the changing files — turning a 30-second whole-repo detekt run into a 300-millisecond per-file check.

What's the right home for a 90-second full-repo lint check?

Pre-push, not pre-commit. Pre-commit must be sub-2-second to stay invisible; anything heavier goes to pre-push, which devs only run when they explicitly invoke git push. CI catches what pre-push misses.

What does stage_fixed: true protect against?

A scenario where Spotless rewrites a file but the original (unformatted) version is what gets committed, because the hook ran after staging. stage_fixed: true auto-re-stages the cleaned files so the commit captures the formatted version.

How should a team treat a teammate using --no-verify weekly?

As a quality signal, not a discipline problem. Hooks that get bypassed are too slow, too noisy, or enforce rules that don't fit. Investigate the specific hook and either fast-track it or demote it to CI.

Key Takeaways

  • Left-shift is arithmetic: a 7-minute CI failure becomes a 300-millisecond local rejection. Same check, different layer.
  • Build the 5-rung ladder: IDE inspection → format-on-save → pre-commit → pre-push → CI. Each catches what the previous missed.
  • Use lefthook for Android-only repos (single Go binary, no language deps); husky for repos with a JS layer; never raw .git/hooks.
  • Pre-commit must be under 2 seconds; pre-push under 2 minutes. Past those budgets, devs --no-verify around you.
  • Always scope pre-commit work with {staged_files}. stage_fixed: true ensures auto-fixed files end up in the commit.
  • Configuration cache + Gradle daemon + parallel = the difference between a snappy hook and a hated one.
  • --no-verify usage is data, not insubordination. Make hooks fast enough that nobody wants to skip them.
