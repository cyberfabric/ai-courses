Android Lint, Properly Configured

Memory Map
💡

The Big Idea

Android Lint already ships with AGP and knows things detekt and ktlint never will: the manifest, your minSdk, accessibility, deprecated APIs. The work is configuration, not installation. Three knobs: lint {} in Gradle, lint.xml for severity, GitHub Actions for the gate.

🛠️ Tool Profile — Android Lint

🎯
What it catches
Android-specific: NewApi, MissingPermission, HardcodedText, ContentDescription, manifest issues, R8 incompat. ~400 built-in checks.
📦
Install
Already installed. Comes with AGP. The work is configuration.
⏱️
Run cost
:app:lintDebug typically 30s–3min. Fully cacheable.
🚫
When to skip
Never. You already have it.

🤔 Why Default Lint Output Gets Ignored

📊 Volume
thousands of findings, mostly yellow

"Lint = noise"
🚦 No CI gate
no merge blocker = homework, doesn't get done

🚨 The Five Categories to Promote Today

Issue ID Catches Default Should be
NewApi API call above your minSdk error error (verify)
MissingPermission Permission not declared in manifest error error (verify)
WrongConstant Bad @IntDef argument error error
HardcodedText Untranslatable XML strings warning error
InvalidPackage Library uses APIs above your minSdk error error (verify)

📊 Severity Strategy

🛑
error — showstoppers
Will crash, leak, or violate a platform contract. Fail the build.
⚠️
warning — visible, not blocking
Real but not always actionable. IDE + CI report; doesn't block merge.
ℹ️
informational — taste
Edge cases, opt-in. Leave at defaults; don't promote without reason.
🔇
ignore — honestly off
If you'll never fix it, ignore it. Don't pretend it's a warning.

Anti-pattern: 200+ permanent warnings nobody fixes. Either promote or ignore. Permanent yellow is rot.

  3. Month 1 — CI Gate

    Add the GitHub Actions workflow. Enable branch protection requiring "Android Lint" status check. Re-audit baseline shrinkage in quality-debt.md.

⚠️ Common Pitfalls

🌐
Globally disabling for one file
Use <ignore path="..."> in lint.xml. Disabling project-wide kills the rule for future code too.
🔁
Auto-regen baseline in CI
Defeats the entire point. The baseline is generated locally, intentionally, by a human.
💥
warningsAsErrors = true
Sounds rigorous; produces fragile builds that break on every AGP minor. Use per-issue severity instead.
🐌
checkDependencies = true
Recursively lints library modules. 8+ minute jobs. For most repos the default (off) is correct.

🔑 Key Terms

lint {} block
The Gradle DSL inside android {} that controls failure behavior, baseline path, report formats. Where abortOnError lives.
lint.xml
The per-issue severity file at the module root. Promote, demote, or ignore individual rules; scope rules with path/regex filters.
lint-baseline.xml
Auto-generated snapshot of existing findings. Lint silently ignores anything in it; new findings fail the build. Generated by :app:updateLintBaseline.
checkReleaseBuilds
Runs lint on the release variant. Catches release-only issues like ProGuard/R8 incompatibilities and resource problems.
Custom lint check
Kotlin module packaged as lintChecks(...) dependency. Worth it for in-house API rules; usually overkill — detekt is the better home for Kotlin smells.
/triage-lint-baseline
Reusable Claude Code skill that reads lint-baseline.xml and groups findings by category, severity, and likely effort.
Branch protection
GitHub setting that makes a CI status check a hard merge requirement. Without this, the lint job is advisory.

🧪 Quick Quiz

Test yourself

Why is Android Lint different from detekt and ktlint?

Lint knows the Android platform: manifest, minSdk, accessibility, deprecated APIs, R8 keep rules. Detekt knows Kotlin idioms; ktlint knows formatting. They're complementary, not interchangeable.

What does the lint {} block control vs. what does lint.xml control?

The Gradle lint {} block sets global behavior: abortOnError, baseline path, report formats, release-build checks. The lint.xml file sets per-issue severity and path/regex scope filters.

Which 5 issue IDs should every team promote to error?

NewApi, MissingPermission, WrongConstant, HardcodedText, InvalidPackage. Most are already error by default; declare them explicitly so the team owns the contract.

Why is auto-regenerating the baseline in CI an anti-pattern?

It silently absorbs every new finding, defeating the entire purpose. Baselines are intentional, human-generated, and committed deliberately. CI just reads them.

What's the lint-IDE-CI consistency trap and the fix?

By default Android Studio doesn't respect your lint.xml, so dev sees warning where CI shows error. Fix: tick "Use lint settings from lint.xml" under Settings → Editor → Inspections → Android → Lint.

Key Takeaways

  • You already have Android Lint. It's the only static analyzer that knows the manifest, minSdk, accessibility, and R8.
  • Three knobs: lint {} in Gradle (global behavior), lint.xml (per-issue severity), GitHub Actions (the gate).
  • Promote these 5 to error: NewApi, MissingPermission, WrongConstant, HardcodedText, InvalidPackage.
  • Severity strategy: error / warning / informational / ignore. Permanent yellow is rot — promote or ignore.
  • Baseline is generated locally, committed by a human, never auto-regenerated in CI.
  • Sync the IDE: tick "Use lint settings from lint.xml" so dev sees the same severity as CI.
  • The CI workflow + branch protection together is what turns lint from advisory to mandatory.
  • Skill to save: /triage-lint-baseline — groups your baseline by category and ranks quick wins.
