Chapter 4

Detekt + Konsist

Memory Map
💡

The Big Idea

Lint knows Android. Detekt knows Kotlin. Konsist knows your architecture. Defaults are sane on day one; the work is enabling type resolution and picking 8–10 high-value rules. The same baseline pattern from Lint applies.

🛠️ Tool Profile — Detekt

🎯
What it catches
Kotlin smells: !! (UnsafeCallOnNullableType), swallowed exceptions, long methods, complex conditions, magic numbers. ~250 rules across 10 sets.
📦
Install
One Gradle plugin at the root. Apply via allprojects {}. Defaults are sane — just override what you care about.
⏱️
Run cost
10–30s without type resolution; 1–3 min with it. Per-module parallelism built in.
🚫
When to skip
Pure-Java codebase only. For any Kotlin project, day-one defaults already pay for themselves.

🔺 The Three-Tool Triangle

Tool Knows Catches Chapter
Android Lint Android platform Manifest, SDK levels, accessibility, R8 keep rules 3
Detekt Kotlin idioms Smells, complexity, exceptions, naming, performance 4
Ktlint Formatting Indentation, line length, wildcard imports — auto-fixes 5

Overlap is <5%. You want all three.

⭐ The Eight Day-One Rules

💥
UnsafeCallOnNullableType
Catches naked !! — the single largest source of NPEs in Kotlin codebases.
🤐
SwallowedException
Empty catch blocks. The silent killer. Hides bugs for years.
🎰
TooGenericExceptionCaught
catch (e: Exception) at random masks NPEs and IO errors.
📏
LongMethod
Methods over your threshold (default 60 lines). Best signal of a function that should be split.
🤯
ComplexCondition
Boolean expressions with >4 ops. Extract to a named function.
🪜
NestedBlockDepth
Loops/ifs nested 4 deep. Same fix as LongMethod.
🔄
ReturnCount
Multiple returns are fine; ten is spaghetti.
🌟
WildcardImport
import foo.*. Hides what your file actually depends on.

✏️ Custom Rule — Pick the Right visit*

If your smell touches... Override
Function calls (System.currentTimeMillis(), runBlocking()) visitCallExpression
Property declarations visitProperty
Class shapes / inheritance / suffixes visitClassOrObject
Import lines visitImportDirective
Inheritance (extends / :) visitSuperTypeList

🏛️ Konsist — Architecture as Tests

🏷️
Naming invariants
ViewModel classes must end with ViewModel. Repository impls must be RepositoryImpl.
📦
Layer isolation
domain must not import from data. presentation must not import from data.
📍
Package placement
Anything ending in RepositoryImpl must live in ..data... Anything ending in UseCase must live in ..domain...
🧪
JUnit-native
Tests live in src/test, run with ./gradlew test, fail like any other test.

🪜 Adoption Ladder

  1. Day 1 — Plugin + Baseline

    Apply detekt at the root. Generate detekt.yml & baseline. Commit. Run ./gradlew detekt — passes (baseline absorbs everything). 2 hours.

  2. Week 1 — Type Res + 8 Rules

    Enable type resolution. Promote the eight day-one rules. Install IntelliJ detekt plugin. Add ./gradlew detekt to CI.

  3. Month 1 — Konsist + Custom

    Add Konsist with 3–4 architecture invariants. Write your first custom rule via /draft-custom-detekt-rule. Update quality-debt.md.

🔑 Key Terms

buildUponDefaultConfig
Tells detekt to start from bundled defaults so you only override what you care about — not redefine ~250 rules from scratch.
Type resolution
Detekt mode where the analyzer resolves real Kotlin types against the classpath. Unlocks rules like UnsafeCallOnNullableType; ~3x slower.
autoCorrect
Detekt setting (default false). Leave it off — auto-fixing is ktlint's job, not detekt's.
Rule
A Kotlin class extending io.gitlab.arturbosch.detekt.api.Rule with an Issue and a visit* override. The unit of detekt extensibility.
Konsist
Architecture-as-code testing library. JUnit tests that scan your code's PSI and assert structural invariants.
scopeFromProject()
Konsist entry point that loads the entire project for assertions. .classes(), .files, .functions() are common downstream filters.
/draft-custom-detekt-rule
Reusable Claude Code skill that generates a Rule skeleton + tests from an English description of the smell.

🧪 Quick Quiz

Test yourself

What does detekt catch that Android Lint doesn't?

Kotlin idioms: complexity (LongMethod, NestedBlockDepth), exception-handling smells (SwallowedException, TooGenericExceptionCaught), naked !!, magic numbers, naming. Lint knows the platform; detekt knows the language.

Why enable type resolution even though it's 3x slower?

It unlocks the highest-value rules — UnsafeCallOnNullableType, UnreachableCatchBlock, ImplicitDefaultLocale — that depend on knowing real Kotlin types, not just textual matches. 1–3 minutes is fine for CI.

What is buildUponDefaultConfig and why use it?

It tells detekt to start from the ~250 bundled defaults and only override what you change. Without it, you'd have to redefine every rule from scratch.

What is Konsist and how does it differ from detekt?

Konsist is an architecture-as-code testing library. Detekt catches code smells inside files; Konsist enforces structural invariants across files (e.g., domain doesn't import from data). Konsist tests are JUnit tests in src/test.

If you're banning runBlocking() outside tests, which visit* method?

visitCallExpression — because runBlocking() is a function call. For class-shape rules use visitClassOrObject; for imports use visitImportDirective.

Key Takeaways

  • Detekt fills the Kotlin-shaped hole left by Android Lint — install it at the root for every module to inherit.
  • Defaults are sane: buildUponDefaultConfig = true, then override 8–10 rules you care about.
  • Eight day-one rules: UnsafeCallOnNullableType, SwallowedException, TooGenericExceptionCaught, LongMethod, ComplexCondition, NestedBlockDepth, ReturnCount, WildcardImport.
  • Enable type resolution. 3x slower; worth it. CI-only, not pre-commit.
  • Same baseline pattern as Lint: detektBaseline, commit, ignore-after.
  • Custom rules are one Kotlin class — pick the right visit* for the smell. Use /draft-custom-detekt-rule.
  • Konsist enforces architecture invariants as JUnit tests. 3–4 tests per module is enough to start.
  • Together with Lint and Ktlint (next chapter) you have the full static-analysis pillar.
