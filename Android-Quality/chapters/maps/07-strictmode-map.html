Chapter 7

StrictMode

Memory Map
🚨

The Big Idea

StrictMode is a 15-year-old Android API that nobody turns on. Ten lines of code, gated on BuildConfig.DEBUG, and your app starts shouting at you about every disk read on the main thread, every leaked Activity, every SQLite query in the wrong place — on your machine, before users ever see the ANR.

🧰 Two policies, two halves

🧵
ThreadPolicy
Watches the UI thread. Detects disk reads, disk writes, network calls, custom slow calls, resource mismatches. Anything blocking on the main thread is a violation.
📦
VmPolicy
Watches the process. Detects leaked Activities, leaked closables (Cursors, FileInputStreams), unregistered receivers, file-URI exposure, cleartext network.

📋 The 10-line install

  1. Create / open Application

    Make sure android:name=".MyApp" is in the manifest. Without it, your Application class never runs.

  2. Gate on BuildConfig.DEBUG

    Non-negotiable. Release builds never see StrictMode — a third-party SDK violation would crash for real users.

  3. Set ThreadPolicy + VmPolicy

    Use the builders. Call all the detect*() methods. End with .penaltyLog() on Day 1.

  4. Run app, capture logcat

    5 minutes of normal use. Filter D/StrictMode. Sort by frequency. Triage the top 5.

🔓 Legitimate suppression

✅
Right way
Wrap the specific call: StrictMode.allowThreadDiskReads().use { sharedPrefs.getString(...) }. Restored on block exit.
🚫
Wrong way
Wrapping a whole onCreate or setupApp() in suppression. Kills StrictMode wholesale; tape over the smoke alarm.
📊
The 5-spot rule
More than 5 suppressions in your codebase = real architectural problem (synchronous I/O in the wrong place) that suppression is hiding.

🧪 StrictMode in instrumented tests

@Rule
detectAll()
StrictModeRule
penaltyDeath()
save / restore
CI gate

A JUnit TestRule sets the strictest policy per test, runs the test, restores on finish. Any new violation crashes the test — regression protection forever.

📖 Key terms

ThreadPolicy
Per-thread (mostly UI) detector for blocking work: disk, network, slow calls, resource mismatches.
VmPolicy
Process-wide detector for leaks: Activities, Closeables, registrations, file URIs, cleartext network.
BuildConfig.DEBUG
The build-type flag your StrictMode setup must be gated on. Release builds must never enable it.
penaltyLog vs penaltyDeath
Log writes to logcat; death crashes the process. Day-1 use log; promote to death once top offenders are fixed.
allowThreadDiskReads
Scoped suppression block. Disables the policy for the lifetime of one specific call, then restores it.
StrictModeRule
JUnit TestRule that activates the strictest policy for one test, with save/restore around it. CI regression protection.
Untrip-and-restore anti-pattern
The team-quietly-kills-StrictMode trap: bulk-wrapping onCreate in suppression because penaltyDeath was too noisy on Day 1. Start with log instead.

Quick Quiz

Why is BuildConfig.DEBUG the non-negotiable guard?

Because in release builds, a violation in third-party SDK code (which you can't fix) would crash for real users. StrictMode is a developer-feedback tool, not a production safety net. Always wrap the setup block in if (BuildConfig.DEBUG) { ... }.

Where is the actual bug in a StrictMode trace?

Not at the top — that's framework code (StrictMode$AndroidBlockGuardPolicy, SharedPreferencesImpl). Scroll down past framework frames until you hit the first one in your package; that's the call site you need to fix.

Why start with penaltyLog instead of penaltyDeath?

A typical first run produces 6–15 violations before the home screen even loads — penaltyDeath would crash the launcher. Use penaltyLog to triage; promote to death once the top offenders are fixed and the count is manageable.

What does StrictMode.allowThreadDiskReads().use { } do?

Disables the disk-read policy for the lifetime of the block (one specific call), then restores it. The right tool for genuinely-must-be-on-main-thread cases — like cold app-startup config. Wrap one call, not a whole function.

Why is wrapping setupApp() in allowThreadDiskReads an anti-pattern?

Because it kills StrictMode wholesale — the function spans many calls, only some of which are legitimately on-main, but the suppression covers them all. Six months later nobody remembers why, and StrictMode is effectively off. Suppress per-call, not per-function.

Key Takeaways

  • StrictMode is a runtime watchdog Android shipped in 2010 — 10 lines of code in your Application class, gated on BuildConfig.DEBUG.
  • Two policies: ThreadPolicy (UI-thread blocking work) and VmPolicy (process-wide leaks). Set both.
  • Progress: penaltyLog → fix top 5 offenders → penaltyDeath. Never penaltyDeath on Day 1.
  • Reading a trace: skip the framework frames at the top, find the first frame in your package — that's the call site.
  • Suppress specific calls with allowThreadDiskReads().use { }, never whole functions. More than 5 suppressions = real architectural problem.
  • A StrictModeRule in instrumented tests with detectAll() + penaltyDeath() is regression protection forever.
  • Same bug, two timelines: 6 weeks and a 1-star review (without StrictMode), or 5 minutes and a coroutine dispatch (with).
