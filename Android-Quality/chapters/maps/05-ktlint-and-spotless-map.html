Chapter 5

Ktlint & Spotless

Memory Map
💡

The Big Idea

Detekt and Lint catch bugs. Formatters catch shape. Pick an opinion once, apply it everywhere, and reviewers stop arguing about indentation forever — senior engineering time gets back to questions that actually matter.

🧰 The two pieces

⚙️
Ktlint (the engine)
Pinterest's opinionated Kotlin formatter. Indent, line length, import order, trailing commas, wildcard imports, spacing — configured by .editorconfig.
🚗
Spotless (the chassis)
Gradle plugin that wraps ktlint plus formatters for Gradle Kotlin DSL, XML, JSON, YAML, Markdown, and license headers. One config block, one task.
📄
.editorconfig
The single source of truth. Read by ktlint, IntelliJ, Android Studio, VS Code — so the IDE formats identically to spotlessApply.
🪞
Format on save
Settings → Tools → Actions on Save → "Reformat code" + "Optimize imports". 95% of issues fixed before they ever exist.

🎯 The one-shot mega-PR

  1. Pin ktlint to an exact version

    ktlint("1.2.1") — never +. Transitive bumps that rewrite your codebase on a Tuesday are a nightmare.

  2. Run ./gradlew spotlessApply

    Brace yourself: ~12k lines change on first run for a mid-size repo. That's the one-time tax.

  3. Open a dedicated format PR

    Only the formatter's diff. No logic. No renames. Reviewers spot-check 3 files and approve.

  4. Merge Friday afternoon, low traffic

    30-min heads-up to the team. Everyone rebases Monday. Pain is concentrated and over in a day.

  5. Never look back

    From here on, every PR is held to the standard. Old code is already there. The bikeshedding tax is gone.

🚪 The escape hatches

Three rules teams legitimately disable. Each costs you a small piece of consistency — don't accumulate more than 5 unconsciously.

🌟
no-wildcard-imports
Usually on. Disable only if your Compose codebase leans heavily on material3.* imports.
🅿️
function-naming
Disable for Compose. @Composable fun MyScreen() is PascalCase by convention; ktlint flags it as a violation.
📁
filename
Disable if you use multiple top-level types per file (sealed-class hierarchies, paired data classes).

📖 Key terms

Bikeshedding tax
The cumulative time reviewers spend on style nits — invisible because it never appears on a ticket, real because it burns hours weekly.
Format-the-whole-repo PR
The dedicated, one-time PR that contains only the formatter's diff. Merged in isolation, then everyone rebases.
Spotless target glob
A pattern (**/*.kt) that selects which files Spotless formats. Use targetExclude for generated code, never for "code we don't want to clean up."
spotlessApply vs spotlessCheck
Apply rewrites files to match the rules. Check validates without modifying — the CI version.
editorConfigOverride
Spotless DSL block that lets you flip individual ktlint rules in build.gradle.kts instead of in .editorconfig. Useful for project-specific quirks.
license header
Spotless feature that injects a copyright comment at the top of every source file. Use a year placeholder ($YEAR) so you never hand-update it.

⚠️ Pitfalls to dodge

📌
Floating ktlint version
A transitive bump rewrites your codebase overnight. Pin exactly; bump deliberately.
🎭
IDE vs ktlint disagree
Save reformats one way, spotlessApply reformats another. Fix .editorconfig; never paper over with non-checked-in IDE settings.
🚧
Permanent dark zones
targetExclude("**/legacy/**") creates code that never gets cleaned up. Format it once, accept the diff, move on.
🗓️
Hand-updated copyrights
Use $YEAR in your license-header template. Updating "2024" to "2025" across 800 files is exactly the manual work the formatter is supposed to kill.

Quick Quiz

Why use Spotless instead of ktlint's official Gradle plugin?

Because the moment you also want to format build.gradle.kts files, XML resources, or strip trailing whitespace from Markdown, you need it anyway. Two formatters, one config, one task — less surface to maintain than one plugin per file type.

What goes in the format-the-whole-repo PR?

Only the formatter's diff. No logic changes, no renames, no val-to-var tweaks. Reviewers don't read it line by line; they spot-check 3 files. Mixing logic in destroys the spot-check assumption and forces a slow review.

Why pin ktlint to an exact version?

Ktlint occasionally changes rules between releases. A floating version means a transitive bump can rewrite your codebase on a random Tuesday, mid-sprint. Pin to 1.2.1 (or whatever) and bump it deliberately when you're ready for the diff.

Where does .editorconfig belong?

At the repo root, with root = true. Every modern editor (IntelliJ, Android Studio, VS Code) reads it, and so does ktlint. One file ensures the IDE formats identically to spotlessApply.

What's the right move when ktlint complains about @Composable fun MyScreen()?

Disable ktlint_standard_function-naming in .editorconfig. Compose convention is PascalCase for @Composable functions, which conflicts with ktlint's default. Don't fight it with @Suppress on every Composable — that's hundreds of files of noise.

Key Takeaways

  • Detekt and Lint catch bugs; ktlint catches shape. Formatters end style debates by simply picking an answer.
  • Use Spotless + ktlint together: ktlint formats Kotlin, Spotless covers Gradle DSL, XML, license headers — one config, one task.
  • .editorconfig at the repo root is the single source of truth — ktlint, Spotless, and every IDE read it.
  • Land the format-the-whole-repo PR in isolation. No logic. No mixing. Friday afternoon, low traffic.
  • Pin ktlint exactly — floating versions can rewrite your codebase mid-sprint.
  • Format-on-save catches 95%, pre-commit catches 4%, spotlessCheck in CI catches the last 1% — PRs cannot merge unformatted.
  • The win isn't the diff. It's the category of PR comments that disappears: indentation, imports, trailing commas. Reviewers finally ask about logic.
