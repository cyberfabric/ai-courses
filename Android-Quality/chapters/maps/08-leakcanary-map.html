LeakCanary

Memory Map
🐦

The Big Idea

Two lines of Gradle. Zero code changes. Auto-installs in debug. LeakCanary catches leaked Activities, Fragments, and Views before they OOM your users — and tells you the exact reference chain holding them, in plain English.

If a destroyed Activity / Fragment / View is still in memory after 5 seconds, the heap is dumped, walked from every GC root, and the shortest reference path to the leaked object becomes your trace.

📦 The two-line install

  1. Add the dependency

    debugImplementation("com.squareup.leakcanary:leakcanary-android:2.14") — one line in app/build.gradle.kts.

  2. Sync & install debug APK

    LeakCanary auto-initializes via a ContentProvider. No Application code, no setup.

  3. Use the app for 10 min

    Launch, sign in, navigate, rotate, sign out. LeakCanary runs in the background; leaks surface as a notification.

  4. Verify it's debug-only

    ./gradlew :app:dependencies --configuration releaseRuntimeClasspath | grep leakcanary — should print nothing.

📖 Reading a leak trace

🌳
GC root (top)
Always alive. The chain starts here. Skip past until you hit a class in your package.
〰️
The squiggle (~~~)
LeakCanary's call-out under a field name: "this is the suspicious link." That field's assignment is what to fix.
💀
Leaked object (bottom)
Marked Leaking: YES. Usually Activity#mDestroyed = true — should have been GC'd, isn't.
📊
Retained heap
The MB number at the bottom — the memory freed if you fix this leak. Triage by size: 14MB = fire; 200KB = paper cut.

🎨 Compose-era considerations

🌐
Captured Activity context
LocalContext.current in a remember referenced by a singleton = leaked Activity. Watch your captures.
🔄
Manually-created scopes
CoroutineScope(Job() + Dispatchers.Main) in a remember needs onDispose cancellation.
🧭
Compose nav backstacks
Some libraries cache backstack entries longer than expected; Activity-scoped repos can leak through that chain.
⚡
Less, but not never
remember, LaunchedEffect, and rememberCoroutineScope are auto-managed — but you can still leak through static state.

📖 Key terms

ObjectWatcher
The lifecycle hook that adds destroyed Activities/Fragments/Views to a watched-set with a weak reference.
5-second window
The grace period between "should have been GC'd" and "let's dump the heap." Long enough that normal GC catches up; short enough to feel responsive.
Shark
Square's heap analyzer that walks the HPROF dump from GC roots to find the shortest reference path to the leaked object.
Retained heap
The total memory that would be freed if you fixed this one leak. Triage signal: 14MB = fire; 200KB = paper cut.
GC root
An object that's always alive (system class, JVM stack, JNI ref). The starting point of every reference chain.
The squiggle (~~~)
LeakCanary's annotation under a field name: "this is the suspicious link." The fix is wherever that field gets set.
debugImplementation
The Gradle scope that compiles a dependency only into debug variants. Non-negotiable for LeakCanary — implementation would ship it to users.
IgnoredReferenceMatcher
The suppression API for known third-party leaks. Lives in app/src/debug/; needs a comment and a follow-up date.

⚠️ Pitfalls

🚀
implementation instead of debugImplementation
LeakCanary ends up in your release APK — size + exposed internals. Always use the debug scope.
📏
Ignoring sub-1MB leaks
Tempting, but a 200KB leak per screen visit becomes 50MB by hour 2. Triage by frequency, not just size.
🤐
Reasonless suppressions
Every suppression needs a comment (which SDK, why) and a follow-up date. Otherwise nobody re-evaluates.
🏃
No CI integration
The notification only fires for the dev who tripped the leak. Without instrumentation tests, others' leaks slip through.

Quick Quiz

Why debugImplementation and not implementation?

Because implementation ships the dependency in release builds. LeakCanary is a developer tool that adds APK size, exposes internal class names, and hooks lifecycle callbacks — none of which belong in production. debugImplementation guarantees it only compiles into your debug variant.

What does the squiggle (~~~) under a field name mean?

It's LeakCanary's annotation marking the suspicious link in the reference chain — the field that's holding the leaked object alive. The fix is wherever that field gets set: search the codebase for the field name, find the assignment, and either null it out at the right lifecycle moment or change to a weak reference.

How do you triage which leak to fix first?

Sort by retained heap size, then weight by frequency. A 14MB leak fixed first buys more headroom than five 200KB leaks fixed first. But a 200KB leak that fires per screen visit can outgrow a one-time 14MB leak quickly — consider both axes.

Why is _binding = null the canonical Fragment fix?

Fragment views are destroyed earlier than Fragment instances (e.g., when the Fragment is on the back stack but not yet popped). Without nulling _binding in onDestroyView, the entire view hierarchy stays in memory until the Fragment itself dies — sometimes minutes later, sometimes never.

How do you make leak detection a CI gate?

Add the leakcanary-android-instrumentation dep with androidTestImplementation scope. Wrap your top user-flow instrumented tests with LeakAssertions.assertNoLeaks(). Run on every PR — any new leak fails the build before it merges.

Key Takeaways

  • Two lines of Gradle — debugImplementation("...:leakcanary-android:2.14") — auto-installs, no code changes, debug-only.
  • Pipeline: ObjectWatcher → 5-sec window → heap dump → Shark analyzer → leak trace + retained heap.
  • Read traces top-to-bottom: skip framework frames, find the squiggle (~~~), fix where the suspicious field is set.
  • Triage by retained heap size. 14MB = fire; 200KB = paper cut. Fix the biggest first.
  • The Fragment + ViewBinding leak is the most common Android leak in 2025. _binding = null in onDestroyView. Lift to a Konsist test.
  • Suppressions live in app/src/debug/, need a comment + follow-up date, audit quarterly. Without it, suppressions become noise that hides real leaks.
  • CI gate via androidTestImplementation + LeakAssertions.assertNoLeaks() on top user flows = regression protection forever.
  • Highest leverage debug-build dep in the entire Android ecosystem — if you adopt one runtime tool from this course, adopt LeakCanary.
