Chapter 4

Detekt + Konsist: Kotlin Smells & Architecture Rules

Lint knows Android. Detekt knows Kotlin. Konsist knows your architecture. Together they cover most of the codebase debt category from your audit.

The Kotlin-shaped hole in your stack

You just wired up Android Lint. It's catching NewApi, HardcodedText, manifest issues. Great. But Lint has a blind spot: it understands the Android platform, not Kotlin idioms. It does not flag a 200-line method, an exception that's caught and silently swallowed, a deeply nested when, or six different NullPointerException-prone !! calls in a row.

Detekt does. And on a real Kotlin codebase, the count of detekt findings is usually 5–10x the count of lint findings — not because your code is worse than the average, but because Kotlin gives you that many more dimensions on which to be inconsistent. The good news: the same baseline-file pattern works, the same adoption ladder works, and most of the value lives in the default rule sets.

Detekt
What it catches
Kotlin smells: LongMethod, ComplexCondition, NestedBlockDepth, UnsafeCallOnNullableType (the !! trap), SwallowedException, TooGenericExceptionCaught, MagicNumber, WildcardImport. Architecture-shaped concerns like ForbiddenComment, UseRequire, naming conventions. Roughly 10 rule sets, ~250 rules.
Install
One Gradle plugin at the root project: id("io.gitlab.arturbosch.detekt") version "1.23.7". Default config is sane on day one.
Run cost
Without type resolution: ~10–30s on a medium app, fully cacheable. With type resolution (recommended): 1–3 minutes. Per-module parallelism is built in.
When to skip
You're a pure-Java codebase. Otherwise, install it — the day-one defaults already pay for themselves.

Where detekt fits next to lint and ktlint

The three Kotlin-friendly static analyzers in this course look similar from a distance and are very different up close:

📐 Android Lint

Knows the Android platform: manifest, SDK levels, accessibility, R8 keep rules. Catches platform-contract bugs.

🧩 Detekt

Knows Kotlin idioms: complexity, exception handling, smells, code shape, naming, performance hot-spots. Catches design-quality bugs.

✏️ Ktlint

Knows formatting: indentation, line length, wildcard imports, trailing commas. Catches style drift — and auto-fixes most of it. (Chapter 5.)

You want all three. They overlap by less than 5%, and where they overlap, configuring one to disable the duplicate rule is a 10-second job.

Installing detekt

The cleanest setup applies detekt at the root project so every module gets it for free. The plugin is small; the config file is the meaningful part.

build.gradle.kts (root)
plugins {
  // Detekt at the root means every subproject inherits the task graph.
  id("io.gitlab.arturbosch.detekt") version "1.23.7"
}

allprojects {
  apply(plugin = "io.gitlab.arturbosch.detekt")

  detekt {
    // Start from the bundled defaults so you only override what you care about.
    buildUponDefaultConfig = true

    // Project-wide config file — one source of truth.
    config.setFrom("$rootDir/config/detekt/detekt.yml")

    // Baseline lives next to the config. Generated once, committed, ignored after.
    baseline = file("$rootDir/config/detekt/baseline.xml")

    // Run analysis in parallel across modules. Free speedup.
    parallel = true

    // Don't fail on the first issue; collect all findings, fail at the end.
    autoCorrect = false
  }

  // The detekt JVM target should match your Kotlin target.
  tasks.withType<io.gitlab.arturbosch.detekt.Detekt>().configureEach {
    jvmTarget = "17"
    reports {
      html.required.set(true)
      xml.required.set(true)
      sarif.required.set(true)   // For GitHub code-scanning integration
    }
  }
}
If your project is multi-module, applying at the root via allprojects is the lowest-friction path. For finer control, switch to a convention plugin (Chapter 12) once you've earned the right to.

The detekt.yml config file

Detekt's defaults are sensible. The first time, generate the default config and commit it — then override only what you want to change. This file is the heart of your team's Kotlin standard.

Generate the default config
./gradlew detektGenerateConfig
mv detekt.yml config/detekt/detekt.yml
The generated file is ~700 lines and exhaustively documents every rule. Don't trim it; it's a reference. Just override the rules you care about at the top or with comments.

The rule sets you'll touch most:

config/detekt/detekt.yml — meaningful overrides
build:
  maxIssues: 0                # Any new finding fails the build (we have a baseline)
  excludeCorrectable: false

complexity:
  LongMethod:
    active: true
    threshold: 60             # Default 60. Drop to 40 if you're brave.
  ComplexCondition:
    active: true
    threshold: 4              # >4 boolean ops in one expression = unreadable
  NestedBlockDepth:
    active: true
    threshold: 4              # Loops/ifs nested 4 deep is a function-extraction signal

exceptions:
  TooGenericExceptionCaught:
    active: true              # catch (Exception e) is almost always a bug
  SwallowedException:
    active: true              # catch ... { /* nothing */ } — the silent killer
  ReturnCount:
    active: true
    max: 3                    # Multiple returns are fine; ten is a smell

style:
  MagicNumber:
    active: true
    ignoreNumbers: ['-1', '0', '1', '2', '100']
    ignorePropertyDeclaration: true
  WildcardImport:
    active: true              # `import foo.bar.*` — nope
  ForbiddenComment:
    active: true
    comments: ['FIXME:', 'STOPSHIP:', 'TODO:']  # surface latent debt
Don't try to enable everything at once. Pick 8–10 rules you actually want, override their thresholds, and let the baseline absorb the rest. You can promote more rules over time.

Generating the baseline

Same pattern as Chapter 3. Detekt has its own task; the file is XML; you commit it.

Generate & commit the baseline
# Generates config/detekt/baseline.xml
./gradlew detektBaseline

git add config/detekt/baseline.xml
git commit -m "chore(detekt): baseline existing findings"
If you split detekt across multiple modules with their own baselines, you'll get one baseline per module. That's fine — one source of truth per scope.

Type resolution — the upgrade you'll want

By default detekt analyzes your code without resolving types — faster, but blind to whole categories of bugs. With type resolution turned on, detekt knows what foo.bar() actually returns, can flag UnsafeCallOnNullableType, UnreachableCatchBlock, ImplicitDefaultLocale, and dozens more rules that depend on knowing real Kotlin types.

app/build.gradle.kts — type resolution
tasks.withType<io.gitlab.arturbosch.detekt.Detekt>().configureEach {
  jvmTarget = "17"
  // Tell detekt to type-resolve against the main classpath
  classpath.setFrom(
    android.sourceSets["main"].kotlin.srcDirs,
    configurations["releaseRuntimeClasspath"]
  )
}

// And in your detekt.yml — rules that require type resolution:
// potential-bugs:
//   UnsafeCallOnNullableType: { active: true }
//   ExitOutsideMain:          { active: true }
//   ImplicitDefaultLocale:    { active: true }
Type resolution roughly triples runtime. On a medium app that's still under 3 minutes. The trade is firmly worth it; just don't put type-resolution detekt on a pre-commit hook (Chapter 6) — that's a CI job.

The eight rules worth their weight on day one

If you read nothing else from detekt.yml, these eight rules pay for themselves in the first week of use:

RuleCatchesWhy it matters
UnsafeCallOnNullableTypeNaked !! callsSingle largest source of NPEs in Kotlin codebases
SwallowedExceptionEmpty catch blocksHides bugs for years; almost always wrong
TooGenericExceptionCaughtcatch (e: Exception) at randomMasks NPEs, IO errors, and your real intent
LongMethodMethods over your thresholdThe single best signal of a method that should be split
ComplexConditionBoolean expressions with >4 opsUnreadable; should be extracted to a named function
NestedBlockDepth4+ nested blocksThe other side of LongMethod — same fix
ReturnCountFunctions with too many returnsDoesn't ban early-returns; just flags spaghetti
WildcardImportimport foo.*Hides what your file actually depends on; breaks reviews

Writing a custom detekt rule

The default rule sets cover most cases, but every team eventually wants something specific: “don't call System.currentTimeMillis() directly — use Clock.now(),” “ban runBlocking outside of tests,” “ViewModel classes must end in ViewModel.” Detekt makes this surprisingly approachable.

detekt-rules/src/main/kotlin/.../NoSystemTime.kt
package com.example.detekt

import io.gitlab.arturbosch.detekt.api.*
import org.jetbrains.kotlin.psi.KtCallExpression

class NoSystemTime(config: Config) : Rule(config) {
  override val issue = Issue(
    "NoSystemTime",
    Severity.Defect,
    "Use Clock.now() instead of System.currentTimeMillis()",
    Debt.FIVE_MINS,
  )

  override fun visitCallExpression(expression: KtCallExpression) {
    super.visitCallExpression(expression)
    val text = expression.text
    if (text.contains("System.currentTimeMillis")
        || text.contains("System.nanoTime")) {
      report(CodeSmell(issue, Entity.from(expression),
        "Replace System time call with Clock.now() so tests can fake it"))
    }
  }
}
A rule is one Kotlin class extending Rule, with a visit* override for the AST node you care about. The hard part is knowing which visit* to override — which is exactly where the AI skill below earns its keep.

AI-Assisted Setup: /draft-custom-detekt-rule

The AST visitor pattern is the wall most teams hit when they first try to write a detekt rule. You know what smell you want to ban; you don't know whether it's a visitCallExpression, a visitProperty, or a visitClassOrObject. Claude Code can draft the skeleton from a one-sentence description.

.claude/commands/draft-custom-detekt-rule.md
---
description: Draft a custom detekt rule from an English description
---

The user will describe a Kotlin smell they want to ban. Your output is:

1. A complete Kotlin file at `detekt-rules/src/main/kotlin/.../<RuleName>.kt`
   that:
   - extends `io.gitlab.arturbosch.detekt.api.Rule`
   - declares an `Issue` with appropriate `Severity` and `Debt`
   - overrides the right `visit*` method based on what the smell touches:
       * function calls   → visitCallExpression
       * property decls   → visitProperty
       * class shapes     → visitClassOrObject
       * import lines     → visitImportDirective
       * inheritance      → visitSuperTypeList
   - uses `report(CodeSmell(...))` with a clear human message

2. A `tests/` file with at least 2 positive cases and 2 negative cases
   using detekt's KtTestCompiler.

3. The YAML stanza to add to `config/detekt/detekt.yml` to enable it.

Be precise. Don't write generic visitor code — pick the most specific
visit method for the smell described.
The first time you call this on a new rule, paste the failing detekt output (or your test case for the smell) so Claude has a concrete example. The skeleton is rarely 100% right, but it gets you 80% of the way.
Good prompt
/draft-custom-detekt-rule Ban any direct call to runBlocking() outside of files in src/test/. Severity: Defect. Reason: in production code, runBlocking blocks the caller thread and is almost always misused. Use viewModelScope or a coroutine context instead.

Konsist: architecture-as-code

Detekt catches code smells at the function/class level. Konsist enforces architecture invariants at the module level — the things you currently rely on code review to catch, like “no presentation class imports from data” or “every ViewModel ends with the suffix ViewModel.” Konsist tests are JUnit tests; they live in src/test; they fail your build the moment a violation is introduced.

app/src/test/.../ArchitectureTest.kt
import com.lemonappdev.konsist.api.Konsist
import com.lemonappdev.konsist.api.verify.assertTrue
import org.junit.Test

class ArchitectureTest {

  @Test fun `viewmodels end with ViewModel`() {
    Konsist.scopeFromProject()
      .classes()
      .filter { it.resideInPackage("..presentation..") }
      .filter { it.hasParent { it.name == "ViewModel" } }
      .assertTrue { it.name.endsWith("ViewModel") }
  }

  @Test fun `domain does not depend on data`() {
    Konsist.scopeFromProject()
      .files
      .filter { it.resideInPackage("..domain..") }
      .assertTrue { file ->
        file.imports.none { it.name.contains(".data.") }
      }
  }

  @Test fun `repository implementations live in data`() {
    Konsist.scopeFromProject()
      .classes()
      .filter { it.name.endsWith("RepositoryImpl") }
      .assertTrue { it.resideInPackage("..data..") }
  }
}
Konsist scans your code at test time using the Kotlin compiler's PSI — same engine detekt uses. The tests are fast (a few seconds); they're real JUnit; they run with ./gradlew test alongside everything else.

Three or four Konsist tests per module is enough to start. They codify the architecture decisions you currently re-explain in every code review. After the first month, "we don't import from data in domain" stops being a review comment and becomes a build failure — which is exactly where you want it.

The adoption ladder

Day 1

Apply detekt at the root. Generate detekt.yml, generate the baseline, commit. Run ./gradlew detekt — it should pass (the baseline absorbs everything). 2 hours.

Week 1

Enable type resolution. Promote the eight day-one rules. Install the IntelliJ detekt plugin so devs see findings in-editor. Add ./gradlew detekt to your CI workflow.

Month 1

Add Konsist with 3–4 architecture invariants. Write your first custom detekt rule (use /draft-custom-detekt-rule). Update quality-debt.md with detekt's row in the adoption log.

Exercise: detekt + one Konsist test, in one PR

Bolt detekt onto the project, with one architectural assertion

Set aside a half-day. From your project root:

  1. Apply the detekt plugin per the recipe above. Run ./gradlew detektGenerateConfig. Move the generated file to config/detekt/detekt.yml.
  2. Customize the eight day-one rules. Save.
  3. Generate the baseline: ./gradlew detektBaseline. Commit the file.
  4. Run ./gradlew detekt. It should pass. Open the HTML report at build/reports/detekt/detekt.html — just to see what's been baselined.
  5. Add Konsist as a test dependency: testImplementation("com.lemonappdev:konsist:0.16.1"). Write one test that asserts a real architectural rule from your codebase. Make it pass.
  6. Add detekt to your CI workflow. Push. Verify the green check.
  7. Open the PR. Title: “chore(detekt): install + baseline + first Konsist test.”
  8. Save /draft-custom-detekt-rule in .claude/commands/. You'll use it the first time the team asks "can we ban X?"

Done. Two new merge blockers in one PR. Both will earn their keep this sprint.

What's next

Chapter 5 closes out static analysis with ktlint and Spotless — the formatters that end PR style debates forever. Detekt and lint catch bugs; ktlint catches inconsistency. By the end of Chapter 5 every PR runs through all three, and your reviewers can finally focus on the code's meaning.
