Chapter 8

LeakCanary: Memory Leaks Made Visible

Two lines of Gradle config and your app starts auto-detecting Activity, Fragment, and View leaks. Then it tells you the exact reference chain holding them, in plain English, on a notification.

The leaks StrictMode can't catch

Chapter 7 turned on StrictMode. It catches the obvious: a leak that fires on every Activity creation, a closeable left open in the same scope. But Android leaks are sneakier than that. The classic pattern: an Activity is rotated, a static field still holds it, garbage collection runs and runs, and the leak grows quietly. By hour two of usage, the heap is 40MB heavier than it should be. Hour three, the OOM. Your user blames the app. You blame the device.

LeakCanary catches these. It runs in your debug build, watches every Activity and Fragment for the moment they should be GC-able, waits 5 seconds, and if they're still in memory, captures a heap snapshot. Then it walks the reference chain from a GC root all the way to the leaked object and tells you, in plain English, which line of which class is the culprit.

It is the single highest-leverage debug-build dependency in the entire Android ecosystem. Two lines, lifetime memory-leak insurance.

LeakCanary

Square's automatic memory-leak detector. Wraps a heap analyzer (Shark) and an Activity/Fragment lifecycle watcher into a debug-only library that surfaces leaks as notifications and an in-app screen.

What it catches
Leaked Activities, Fragments, Views, ViewModels, and any object you explicitly tell it to watch. Identifies the retention chain (which reference is keeping it alive) and the GC root.
Install
One line in app/build.gradle.kts: debugImplementation("com.squareup.leakcanary:leakcanary-android:2.14"). No code changes. No Application setup. It auto-installs on debug builds.
Runtime cost
Negligible until a leak is suspected. Heap analysis runs on a background thread when triggered — takes 10–30 seconds, no impact on UI thread, only on debug builds.
When to skip it
Never in debug. Always in release (the debugImplementation scope already enforces this). For instrumented tests on CI, opt in deliberately — the heap analysis adds time.

The two-line install

It really is two lines. Add the dependency, sync Gradle, install the debug APK. LeakCanary auto-initializes via a ContentProvider, no Application code required.

app/build.gradle.kts
dependencies {
    // ... your other deps

    // LeakCanary — debug only, never ships in release.
    debugImplementation("com.squareup.leakcanary:leakcanary-android:2.14")

    // Optional: enable on release-debug builds for QA testing
    // (do NOT use 'implementation' — that ships to users).
    // releaseImplementation("com.squareup.leakcanary:leakcanary-android-release:2.14")
}
The debugImplementation scope is what guarantees LeakCanary never reaches a release APK — it's only compiled into your debug build variant. Verify after install with ./gradlew :app:dependencies --configuration releaseRuntimeClasspath | grep leakcanary — should return nothing.

How it works (in 60 seconds)

Three pieces, working together.

ObjectWatcher → 5-second window → Heap dump → Shark analyzer → Leak trace
  1. ObjectWatcher hooks Android's lifecycle callbacks. When an Activity reaches onDestroy, a Fragment its onDestroyView, a ViewModel its onCleared, or a destroyed View, it adds the object to a watched-set with a weak reference.
  2. 5-second grace. If the object is GC'd within 5 seconds (the normal case), LeakCanary clears the watch and moves on.
  3. Heap dump. If the object is still in memory after 5 seconds and a forced GC, LeakCanary takes an HPROF heap snapshot of the entire process. This takes a couple of seconds.
  4. Shark analyzes. Square's Shark engine walks the heap from every GC root looking for the shortest reference path to the leaked object.
  5. Leak trace. The path is rendered as a chain: GC root → static field → collection → the leaked Activity. Each link gets a per-line "this is leaking?" annotation: yes / no / unknown.

Reading a leak trace

You install LeakCanary, use the app, get a notification: "Tap to see leak." The screen shows something like this.

┬───
│ GC Root: System class
│
├─ android.app.ActivityThread instance
│    Leaking: NO (Activity not destroyed)
│    ↓ ActivityThread.mInstrumentation
├─ com.example.MyInstrumentation instance
│    Leaking: NO
│    ↓ MyInstrumentation.cachedActivityRef
│           ~~~~~~~~~~~~~~~~~~~
├─ com.example.MainActivity instance
│    Leaking: YES (Activity#mDestroyed is true)
│
═════════════════════
Retained heap: 14.3 MB

Read it top to bottom. The GC root is alive (always). It holds MyInstrumentation, which has a field called cachedActivityRef — the squiggle (~~~) under it is LeakCanary's call-out: "this is the suspicious link." That field references a MainActivity that is destroyed (mDestroyed = true). So the leak is: someone is caching the Activity in MyInstrumentation and never clearing it on destroy.

14.3 MB is the retained heap — the total memory that would be freed if you fixed this one leak. That number is what you triage on. A 14MB leak is a fire; a 40KB leak is a paper cut.

The fix is upstream of the squiggle. The ~~~ tells you the field that's holding the leaked object. The fix is wherever that field is set. Search the code for the field name, find every assignment, and either null it out at the right lifecycle moment, or change the reference to a weak one.

The classic Android leaks

After running LeakCanary across a few projects, you see the same patterns. Knowing them by sight makes the leak trace much faster to read.

Inner class with implicit outerOuterActivity$1.this$0 link in the traceMake the inner class static (Java) or top-level (Kotlin)

The Fragment + ViewBinding canonical leak

This one deserves its own section because it's the single most common Android leak in 2025. The pattern:

// LEAKING:
class ProfileFragment : Fragment() {
    private var _binding: FragmentProfileBinding? = null
    private val binding get() = _binding!!

    override fun onCreateView(...): View {
        _binding = FragmentProfileBinding.inflate(inflater, container, false)
        return binding.root
    }
    // Missing onDestroyView!
}

Fragment views live shorter than Fragment instances. When the Fragment goes to the back stack, its view is destroyed but the Fragment isn't. If _binding isn't nulled, the entire view hierarchy (and any data it holds) is kept alive until the Fragment itself dies — sometimes minutes later, sometimes never.

The fix is one line.

ProfileFragment.kt (correct)
class ProfileFragment : Fragment() {
    private var _binding: FragmentProfileBinding? = null
    private val binding get() = _binding!!

    override fun onCreateView(
        inflater: LayoutInflater,
        container: ViewGroup?,
        savedInstanceState: Bundle?
    ): View {
        _binding = FragmentProfileBinding.inflate(inflater, container, false)
        return binding.root
    }

    override fun onDestroyView() {
        super.onDestroyView()
        _binding = null  // The whole game is this one line.
    }
}
You can enforce this team-wide with a Konsist test (Chapter 4): "every class extending Fragment with a non-null _binding field must override onDestroyView." Catches it once, in CI, forever.

Compose-era considerations

Compose generally leaks less than the View system — remember is tied to composition lifecycle, LaunchedEffect is automatically cancelled, rememberCoroutineScope is scoped to the composable. But "less" is not "never."

  • Captured Activity context. If a Composable captures LocalContext.current in a remember and that remember ends up referenced by something longer-lived (a static cache, a singleton), you've leaked the Activity.
  • Manually-created scopes. A CoroutineScope(Job() + Dispatchers.Main) created in a remember is yours to cancel. Forget to cancel it on onDispose, and the coroutine keeps running — with whatever it captured.
  • Side-effect navigation libraries. Some Compose navigation graphs cache backstack entries longer than expected; if your screen ViewModel pulls in an Activity-scoped repository, you can leak the Activity through that chain.

LeakCanary works the same way on Compose — it watches the underlying Activity and any objects you explicitly add. If you want it to watch a Composable's specific state, use AppWatcher.objectWatcher.expectWeaklyReachable(yourObject, "description") in a DisposableEffect's cleanup block.

Suppressing legitimate non-leaks

Sometimes a "leak" isn't yours. A third-party SDK holds a reference it shouldn't, the OEM ROM has a bug, an Android framework class on a specific API level genuinely retains your Activity. LeakCanary ships with a list of known issues (see AndroidReferenceMatchers), but you'll occasionally hit a new one.

app/src/debug/.../DebugApp.kt
// Only runs in debug builds.
class DebugApp : Application() {
    override fun onCreate() {
        super.onCreate()
        configureLeakCanary()
    }
}

private fun configureLeakCanary() {
    LeakCanary.config = LeakCanary.config.copy(
        referenceMatchers = AndroidReferenceMatchers.appDefaults +
            listOf(
                // Known leak in BadVendor SDK 2.3 — remove when 2.4 lands.
                IgnoredReferenceMatcher(
                    pattern = InstanceFieldPattern(
                        className = "com.badvendor.SdkSingleton",
                        fieldName = "lastActivity"
                    )
                )
            )
    )
}
Suppressions go in app/src/debug/ — never in main, never in release. Each one needs a comment explaining why and when to remove it. Audit suppressions every quarter; expired ones (the SDK was updated, the bug is gone) become noise that hides real leaks.

LeakCanary in instrumented tests

The notification is for developers. CI needs structured signal. LeakCanary publishes a test mode that runs heap analysis after each test and fails if a leak occurred.

app/build.gradle.kts (additional dep)
dependencies {
    // ...

    // Instrumented-test integration: fails any test that leaks.
    androidTestImplementation("com.squareup.leakcanary:leakcanary-android-instrumentation:2.14")
}
Add a @Rule or RunListener that calls LeakAssertions.assertNoLeaks(). Wrap your top user flows (login, navigation, checkout) in instrumented tests; any leak introduced by a future PR fails the build.

AI-Assisted Setup — the /explain-leak-trace skill

LeakCanary's traces are clear once you've read 20 of them. The first 5, you'll squint. Claude Code is great at the squinting phase — paste the trace, get a one-paragraph explanation plus the smallest fix.

The heap size tells me whether to triage this as a fire (>10MB) or a paper cut (<100KB). 2. Walk the reference chain from GC root to the leaked object. State which pattern this matches (or note "novel" if none fit). 5. Give the smallest fix. Show the before/after diff for the offending class if you have enough context. If the leak is in a third-party SDK, give me the IgnoredReferenceMatcher snippet for LeakCanary config and flag it for follow-up. Here's the trace: [paste]
Save to .claude/commands/explain-leak-trace.md. Use it on every new leak for the first month. After ~20 runs, you'll recognize the patterns at a glance and the skill becomes optional — but it stays useful for novel leaks.

A second skill worth standing up: /draft-leakcanary-suppression. When you've decided a leak is third-party and not actionable, generate the IgnoredReferenceMatcher with the right pattern type (InstanceFieldPattern, StaticFieldPattern, JavaLocalPattern) and a comment block explaining the decision and the expected resolution date.

Skill: /explain-leak-trace (don't do this)
My app is leaking memory. Help.
No trace, no retained heap size, no clue what type of object is leaking. Claude can't run LeakCanary for you — the trace it produces is the input. Always paste the full trace (GC root through to the leaked object, with the squiggle annotations); skipping the trace turns this from a 30-second answer into a guessing game.

What you'll see on Day 1

Most apps have 1–3 leaks they didn't know about. The first run will surface them in the first 10 minutes of regular use. The two most common results:

  1. A Fragment + ViewBinding leak in code written before _binding = null was a known pattern.
  2. A custom singleton or DI-scoped class that's caching an Activity context "for convenience."

Both are genuine bugs. Both have small fixes. Don't panic: this is what every Android codebase looks like before LeakCanary runs. The fact that you found them is good news.

Triage by retained heap size. Sort the leaks by the "Retained heap" number at the bottom of each trace. A 14MB leak fixed first buys you more headroom than five 200KB leaks fixed first — even if the small ones are easier. Spend an hour on the biggest leak before you spend ten minutes on the smallest.
Day 1 Add the debugImplementation dependency. Install the debug APK. Use the app for 10 minutes of real flow. Tap each leak notification, copy the trace into /explain-leak-trace, fix the top 1.
Week 1 Fix all currently-surfaced leaks. Add Konsist tests for the Fragment + ViewBinding pattern. Configure suppressions for any third-party leaks you can't act on (with comments and follow-up dates).
Month 1 Add the androidTestImplementation instrumentation library. Wire LeakAssertions.assertNoLeaks() into your top 3 user-flow instrumented tests. Run them on every PR — any new leak fails the build.

False positives — what to genuinely ignore

Most LeakCanary findings are real. A few aren't, and the difference matters because chasing a false positive is hours you don't have.

  • OEM-specific framework leaks. Some Samsung / Huawei ROMs have known framework bugs that retain Activities. LeakCanary's defaults already filter the well-known ones; if you hit a new one, file in their tracker and add a suppression.
  • "Leaks" during config changes. An Activity may briefly appear leaked during rotation because the new instance is alive while the old hasn't been GC'd yet. LeakCanary's 5-second window usually catches this; if it doesn't, force a GC and re-test.
  • WebView native leaks. WebView occasionally retains native references LeakCanary can see but can't clean up. Confirmed WebView leaks tied to Android version are usually filed upstream; suppress with a follow-up date.

The default rule: assume the leak is real until you've spent at least 15 minutes confirming it isn't. Most "false positives" turn out to be real on inspection.

Common pitfalls

  • Using implementation instead of debugImplementation. LeakCanary ends up in your release APK — it adds size and exposes internal class names. Always use debugImplementation.
  • Ignoring leaks under 1MB. Tempting, but those add up. A 200KB leak that fires per-screen-visit becomes a 50MB leak by hour two.
  • Suppressing without a reason. Every suppression should have a comment explaining the third-party SDK and an expected-resolution date. Without it, you'll never re-evaluate.
  • Not running on instrumented CI. The notification only fires for the developer who tripped the leak. Without CI checks, leaks shipped by other team members slip through. The instrumentation library is the regression net.

Why this is the highest-leverage debug-build dep

Two lines of Gradle. Zero code changes. Auto-installs. Catches the most expensive class of bugs (memory leaks → OOMs → 1-star reviews). Tells you the exact line to fix in plain English. Works on every Android codebase regardless of architecture, regardless of legacy. There is no other tool in the ecosystem with this ratio of effort to payoff. If you only adopt one runtime tool from this course, adopt LeakCanary.
