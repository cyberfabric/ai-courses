Chapter 11

R8: Minification Without Mysteries

Smaller APKs, faster startups — without ClassNotFoundException on release

The "we'll turn on minification later" problem

You know R8 exists. You know your release APK is bigger than it needs to be. You may have even tried turning minifyEnabled = true once — the build worked, you shipped, an hour later Crashlytics lit up with NoSuchMethodError from an obfuscated class name nobody could read, and you reverted in a panic. Since then: nothing. Release APK still 30% too big.

This is the chapter that fixes that, permanently. R8 is not magic and not scary. It strips unused code, optimizes what's left, and renames symbols to short tokens. Your job is to tell it which symbols can't be renamed (because they're referenced by string — through reflection, JNI, XML, or framework lookup). Once you know the patterns, R8 becomes one of the cheapest wins in the whole quality stack.

The frame: your release APK has 20-40% of code that nobody runs. R8 removes it. The pricetag is a few keep-rule lines per library that uses reflection. After your first mapping.txt-driven debugging session, those rules are obvious; before the first one, they feel like voodoo. This chapter takes you across that line.

Meet the tool

Meet R8

What it is: the bundled minifier in Android Gradle Plugin (replaced ProGuard in AGP 3.4+). Four phases — shrink, optimize, obfuscate, preverify — applied to release builds.

What it does that humans can't: dead-code analysis across your entire app + every library, inlining of small methods, class merging, value propagation, name shortening to single letters. Manual equivalents: impossible at any reasonable cost.

Install: none required — bundled with AGP. You enable it via minifyEnabled = true + shrinkResources = true in your release buildType.

Runtime cost: negative — smaller APK, faster class loading, faster cold start. Build-time cost: 30-90 seconds added to release builds depending on size.

When to skip it: never for production releases. Some teams skip for debug builds (the default). Don't ship a release without it — the APK-size and obfuscation gains compound.

What R8 actually does, in 60 seconds

R8 reads your compiled bytecode, your AndroidManifest.xml, your resources, and your keep rules. From the manifest entry points (Application, Activity, BroadcastReceiver, etc.) it walks the call graph, marks every reachable class/method/field, and discards the rest. Then it optimizes the survivors. Then it renames them to a.b.C.d()-shaped tokens so the APK is smaller and the bytecode is harder to read.

1. Shrink

Remove unreachable classes, methods, fields. Anything not referenced from a manifest entry, a keep rule, or another reachable symbol — gone.

→

2. Optimize

Inline small methods, propagate constant values, merge sibling classes, devirtualize calls, eliminate dead branches.

→

3. Obfuscate

Rename packages, classes, methods, fields to single letters. Saves bytes; makes reverse-engineering harder.

→

4. Preverify

Add metadata so Android's runtime can verify the bytecode faster. Tiny but free.

Turn it on

app/build.gradle.kts — minimal release config

android {
    buildTypes {
        release {
            isMinifyEnabled = true       // shrink + optimize + obfuscate
            isShrinkResources = true     // strip unused PNG/XML/string resources

            proguardFiles(
                // Default Android-aware optimizations (use the "-optimize" variant)
                getDefaultProguardFile("proguard-android-optimize.txt"),
                // Your app-specific keep rules
                "proguard-rules.pro"
            )

            signingConfig = signingConfigs.getByName("release")
        }

        debug {
            // Default: minifyEnabled = false. Leave it. Debug builds want fast iteration.
        }
    }
}

Two lines. isMinifyEnabled turns on the R8 pipeline; isShrinkResources additionally strips unused resources. The latter requires the former — resource shrinking can't happen without code shrinking, because R8 needs to know which classes survive to know which resources are referenced.

The keep-rule mental model

R8 strips everything not reachable from a manifest entry or a keep rule. So the question becomes: which of my code is reachable in a way R8 can't see? Three usual suspects:

🔍 Reflection

Anything looked up by string at runtime: Gson/Moshi DTOs, Hilt DI, Dagger components, Retrofit interfaces, room entities accessed via Class.forName, Compose @Composable previews referenced from XML.

🌉 JNI

Native code calling back into Java/Kotlin. Method names matter to C++. Strip the name, native crashes on JNIEnv::FindClass.

📄 XML / String references

Custom views in layouts, fragments referenced by class name in nav graphs, services and receivers in the manifest, broadcast actions matched by package-qualified strings.

Anything reflected, JNI'd, or named-by-string needs a keep rule. Most modern libraries ship their own — you'll see them under META-INF/proguard/ in the AAR — but not all, and not always completely.

The keep-rule patterns you'll actually use

Below is a battle-tested proguard-rules.pro covering the libraries every Android team ships. Add only the rules you need; don't paste the whole thing.

app/proguard-rules.pro — common patterns

# --- Kotlin metadata (for reflection-using libraries) ---
-keep class kotlin.Metadata { *; }

# --- Gson / Moshi DTOs ---
# Keep all classes annotated with @JsonClass (Moshi)
-keep,allowobfuscation,allowshrinking @com.squareup.moshi.JsonClass class *

# Keep generated adapters
-keep class **JsonAdapter { *; }
-keep class **_MoshiJsonAdapter { *; }

# Or with Gson — keep model classes by package
-keep class com.example.app.api.dto.** { *; }
-keepclassmembers class com.example.app.api.dto.** {
    <fields>;
    <init>(...);
}

# --- Retrofit ---
# Retrofit interfaces are looked up by name
-keep,allowobfuscation,allowshrinking interface retrofit2.Call
-keep,allowobfuscation,allowshrinking class retrofit2.Response
# Your own API interfaces
-keep interface com.example.app.api.** { *; }

# --- Hilt / Dagger ---
# Library ships rules; this catches edge cases
-keep class dagger.hilt.** { *; }
-keep class * extends dagger.hilt.android.HiltAndroidApp { *; }

# --- Coroutines (already shipped by the library, but pin the safety net) ---
-keepnames class kotlinx.coroutines.internal.MainDispatcherFactory {}
-keepnames class kotlinx.coroutines.CoroutineExceptionHandler {}

# --- ViewBinding generated classes ---
# Usually not needed (AGP injects rules), but here for completeness:
-keep class * implements androidx.viewbinding.ViewBinding {
    public static * inflate(...);
    public static * bind(...);
}

# --- Custom Views referenced from XML layouts ---
-keep public class * extends android.view.View {
    public <init>(android.content.Context);
    public <init>(android.content.Context, android.util.AttributeSet);
    public <init>(android.content.Context, android.util.AttributeSet, int);
}

# --- Native methods (JNI) ---
-keepclasseswithmembernames class * {
    native <methods>;
}

# --- Crashlytics: keep useful stack frames ---
-keepattributes SourceFile,LineNumberTable
-renamesourcefileattribute SourceFile

Most teams need fewer than 10 of these for their app code — modern AndroidX, Compose, Retrofit, Moshi, and Hilt all ship their own keep rules. The two you almost always need to write yourself: your API DTO package and your Retrofit interface package.

The SourceFile + LineNumberTable attributes are the difference between a useful Crashlytics stack trace and a useless one. Without them, your retraced stack still shows real class names but no file/line info. With them, it tells you exactly which line crashed. Always keep these. The 1-2 KB they cost is nothing compared to debugging time saved.

The mapping.txt golden rule

Every R8 build produces a mapping.txt file at app/build/outputs/mapping/release/mapping.txt. It records every original-name ↔ obfuscated-name pair. Without it, an obfuscated stack trace reads:

java.lang.NullPointerException
    at a.b.C.d(:0)
    at a.b.C.e(:0)
    at a.f.G.h(:0)

Useless. With it — via the Android Studio "Retrace" tool or retrace.jar — the same trace becomes:

java.lang.NullPointerException
    at com.example.app.profile.ProfileViewModel.loadAvatar(ProfileViewModel.kt:74)
    at com.example.app.profile.ProfileViewModel.observeUser(ProfileViewModel.kt:51)
    at com.example.app.di.AppGraph.getProfileViewModel(AppGraph.kt:128)
mapping.txt is build-specific. If you lose it, the obfuscated stack traces from that build are unrecoverable forever. Either: (1) commit it to a private storage bucket per release, (2) upload it automatically to Crashlytics/Bugsnag/Sentry as part of the release pipeline. Both is even better. Never just leave it in build/.

Auto-uploading to Crashlytics

app/build.gradle.kts — the Crashlytics + Firebase rule

plugins {
    alias(libs.plugins.google.services)
    alias(libs.plugins.firebase.crashlytics)
}

android {
    buildTypes {
        release {
            // Crashlytics plugin uploads mapping.txt automatically on every release build
            // when both plugins above are applied. Verify in build logs:
            //   "Successfully uploaded mapping file"
        }
    }
}

If you don't use Crashlytics, the equivalent for Sentry / Bugsnag is one Gradle config block and an API key. The key behavior is the same: every release build triggers an upload, retracing happens server-side automatically.

Reading a release-only crash

You ship a release. Within an hour Crashlytics flags a crash in a.b.c.d(). The trace looks gibberish. You have mapping.txt for that build version. Walk through:

  1. Open Crashlytics — if mapping was uploaded, the trace is already retraced. Done. Skip to step 4.
  2. If not, find mapping.txt for the exact build version. Exact: a mismatched mapping retraces to wrong lines, not "no line."
  3. Run retrace.jar mapping.txt obfuscated-trace.txt, or paste both into Android Studio's "ReTrace" tool (Build → Analyze APK → ProGuard ReTrace).
  4. Read the retraced trace. The first frame in your package is the entry point. Walk down the trace until you hit framework code — that's the boundary.
  5. If the crash is NoSuchMethodError, ClassNotFoundException, NoClassDefFoundError, or "unable to instantiate," it's a missing keep rule. Add it, ship a patch.

The "rules from libraries" pattern

Most modern libraries ship a consumer-proguard-rules.pro inside their AAR. R8 picks it up automatically — you don't need to write rules for them. AndroidX, Retrofit, OkHttp, Moshi, Hilt, Coroutines, Compose, Coil, Glide all do this.

The libraries that don't tend to be older ones (pre-2020) or smaller open-source projects. Symptoms: release-only crashes referencing the library's package, or worse, your own package using a library API. Diagnose by:

Check what rules a library actually ships

# 1. Find the AAR in your gradle cache
find ~/.gradle/caches/modules-2 -name "library-name-*.aar"

# 2. AARs are zip files
unzip -l path/to/library.aar | grep proguard

# 3. Inspect the rules
unzip -p path/to/library.aar proguard.txt
# or
unzip -p path/to/library.aar META-INF/proguard/library.pro

If the AAR has no proguard rules and you're crashing on it, you'll need to write your own keep rules for that library. Start with -keep class com.thatlibrary.** { *; }, ship a patch, then narrow down once you see what actually gets used.

The dangerous optimizations — and which to enable

The default proguard-android-optimize.txt is conservative. There are three knobs you can turn for more aggressive optimization. Two are safe; one isn't.

FlagWhat it doesSafe?
android.enableR8.fullMode=true (in gradle.properties) Default since AGP 8.0. More aggressive optimizations & obfuscation. Some libraries needed extra rules at first; ecosystem has caught up. ✅ Yes
-allowaccessmodification Lets R8 widen access modifiers (private → public) when it enables an optimization. ✅ Yes — only used internally to the optimizer.
-repackageclasses '' Moves all classes to the empty package (extreme name shortening). ⚠️ Mostly — breaks anything that does package-name-based reflection (rare, but real).
Default recommendation: turn on fullMode (or just leave it — it's the AGP 8+ default), enable -allowaccessmodification in your rules file, leave -repackageclasses off unless you're benchmarking the last few KB. The first two get you 90% of the gains with zero ongoing maintenance burden.

Baseline Profiles — the next-level optimization

Once R8 is solid and your APK is shrunk, the next lever is cold start time. Baseline Profiles are AOT-compilation hints — you record which classes/methods run during your critical user journeys, the Play Store ships that profile alongside the APK, and the runtime pre-compiles them. 20-40% startup improvements are typical.

Outside this chapter's scope, but: once R8 is wired and you have a working release pipeline, Baseline Profiles via the Macrobenchmark library is the right next investment. The official guide is sufficient. Mention them on the next "we have time for a perf project" cycle.

AI-Assisted Setup: /draft-keep-rules

Writing keep rules from scratch is busywork — you stare at a release-only stack trace and a library doc, then translate. Hand it to Claude.

Good prompt

"I have a release-only crash. Stack trace is below (already retraced via mapping.txt). Build is using minifyEnabled = true, fullMode = true, default proguard-android-optimize.txt. Library involved: com.example:legacy-sdk:3.4.0. Output: (1) the precise -keep rule(s) to add, (2) where in proguard-rules.pro they should go (with a comment header), (3) any narrower variant that would still fix the crash but keep more bytes shrunk. Do not output a wildcard -keep class com.example.** { *; } unless it's the only safe answer — explain why if it is."

Bad prompt

"Fix this crash."

Why it fails: no signal that R8 is the cause, no library identification, no constraints on rule scope. You'll get a try/catch Stack-Overflow snippet and zero R8 understanding.

.claude/commands/draft-keep-rules.md

# /draft-keep-rules

Diagnose a release-only Android crash and produce ProGuard/R8 keep rules.

Inputs (paste in the conversation):
- The retraced stack trace (after mapping.txt application).
- The library/dependency involved.
- Current proguard-rules.pro content.
- Current minify config from build.gradle.kts.

Output:
1. The minimal keep rule(s) that fix the crash.
2. Placement: where in proguard-rules.pro to add them, with a section comment.
3. A narrower variant if any (preferring -keepclasseswithmembers
   over wildcard -keep when possible).
4. Brief explanation of WHY R8 stripped this — reflection? JNI? XML?
   The reader should understand the pattern, not just the patch.

Constraints:
- Prefer -keep,allowobfuscation,allowshrinking when only signature
  preservation is needed.
- Avoid -keep class **.* { *; } unless no narrower rule exists.
- Always output keep rules in proper R8 syntax — single-quoted method
  signatures, <init>(...) constructors, etc.

Run the skill once per release-only crash. After ~5 invocations, you'll start spotting the patterns yourself and only need it for unfamiliar libraries.

Adoption ladder

Day 1

Enable isMinifyEnabled = true + isShrinkResources = true in a feature branch's release build type. Add the standard keep rules from this chapter. Build a release APK. Diff its size against the previous release: expect 20-35% reduction. Don't ship yet.

Week 1

Smoke-test the release APK end-to-end on at least one device. Use the /draft-keep-rules skill to fix any release-only crashes that surface. Wire Crashlytics (or your crash reporter) to auto-upload mapping.txt. Ship the minified release behind a staged rollout (10% → 50% → 100%).

Month 1

Wire android.enableR8.fullMode=true (or accept the AGP 8+ default). Audit your proguard-rules.pro and remove rules that newer library versions now ship themselves. Schedule a Baseline Profile project as the next perf investment.

The before/after

Before

Release APK 38 MB. Cold start 1.8s on a mid-tier device. Reverse-engineering the APK takes 10 minutes with jadx: every class and method named exactly as in source. "We tried minification once but it broke."

After

Release APK 24 MB (37% smaller). Cold start 1.4s. jadx output is a.b.C.d()-shaped — same logic, much harder to read. Crashlytics retraces every stack server-side. Adding a new library? Run /draft-keep-rules, ship.

The pitfalls

Don't disable minification "until we figure it out." Once you skip a release, the next "let's enable it" gets pushed forever. Enable it in a branch, fix the keep rules, ship. Two hours of work. Permanent payoff.
Don't use -keep class com.example.** { *; } as a default sledgehammer. It works, but every wildcard you keep is bytes R8 can't shrink and a class an attacker can read. Narrow down with -keepclasseswithmembers, package boundaries, or annotation-based keeps as soon as you have time.
Do not lose mapping.txt. Once a release is in users' hands and you've lost the mapping, every crash from that build is unreadable forever. Auto-upload to Crashlytics or commit to a private store. Belt-and-braces.
Don't put keep rules in proguard-android-optimize.txt. That file is the default Android one and gets overwritten on AGP upgrades. Your rules go in proguard-rules.pro. Both files are listed in proguardFiles(...); both are applied; only the second is yours.

Exercise

Turn on R8 for real, end-to-end

  1. On a feature branch, enable isMinifyEnabled = true + isShrinkResources = true in your release build type.
  2. Build a release APK. Note its size. Compare to the previous unminified release. Record the percentage reduction.
  3. Install and exercise the release APK on at least one real device for 5 minutes. Note any crashes.
  4. For each release-only crash: locate the mapping.txt, retrace the stack with Android Studio's ReTrace tool, run /draft-keep-rules with the trace, apply the suggested rule, rebuild.
  5. Wire com.google.firebase.crashlytics Gradle plugin (or your crash reporter's equivalent) to auto-upload mapping.txt. Verify in build logs that the upload happens.
  6. Open a PR titled "Enable R8 minification + resource shrinking." Body: pre/post APK sizes, crashes found and fixed, link to the verified mapping upload.

Deliverable: a green build with minification on, an APK that's 20%+ smaller, and a crash-reporter pipeline that retraces every future release stack automatically. Bonus: /draft-keep-rules committed to .claude/commands/.

What's next

You've got the whole stack working — lint, detekt, ktlint, pre-commit hooks, StrictMode, LeakCanary, debug inspectors, version catalogs, dependency health checks, and a minified release pipeline. Each module's build.gradle.kts probably has 30+ lines of repeated configuration to wire all of it. That's the last problem.

Chapter 12 is the payoff: convention plugins. You'll package the entire quality stack into a single Gradle plugin (my.app.quality) that every module applies in one line, and wire a CI quality gate that runs the full stack on every PR. The result is a repo where every new module gets the standards for free, every PR gets the same checks, and "we'll add it later" becomes structurally impossible.
