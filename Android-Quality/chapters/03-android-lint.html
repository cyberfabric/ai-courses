Chapter 3

Android Lint, Properly Configured

The linter that came with AGP knows things detekt and ktlint will never know. Stop ignoring it.

The free linter you've been ignoring for six years

Android Lint shipped with the Android Gradle Plugin. You did not install it. You probably did not configure it. You almost certainly do not look at its output. And yet it is the only static analyzer in the entire stack that knows what an AndroidManifest.xml is, that knows what your minSdk is, that knows your strings are user-facing, and that knows which APIs were added in which platform version.

This chapter is about turning Android Lint from a Gradle task you have never run into a high-signal first line of defense, with a baseline file and a CI gate, in roughly an afternoon.

Android Lint
What it catches
Android-specific issues other linters can't see: NewApi calls without SDK_INT guards, MissingPermission, HardcodedText in user-facing strings, ContentDescription for accessibility, deprecated APIs, manifest problems, WrongConstant on annotations, R8 keep-rule incompatibilities. Around 400 built-in checks; many ship with the AndroidX/Compose libraries you already depend on.
Install
Already installed. It comes with AGP. The work is configuration, not installation.
Run cost
./gradlew :app:lintDebug typically 30s–3 min on a medium app, fully cacheable. :app:lintRelease can be a bit slower because it analyzes more variants.
When to skip
Never. You already have it. The only question is whether you've turned it into something useful.

Why the default output is ignored

If you've ever run ./gradlew lint on a real project, you know the experience: thousands of findings, mostly yellow, mixed severity, scattered across modules, no clear “these are the bugs.” You scroll, you sigh, you close the report. That is the default state. We are going to fix all three reasons it happens.

  • Volume — default lint runs every check at default severity. The signal-to-noise ratio is bad without tuning.
  • Severity — defaults are mostly warning. Warnings don't fail builds. If nothing fails the build, nobody fixes anything.
  • No CI gate — if lint isn't a merge blocker, it's homework. Homework doesn't get done.

The fix is the same three knobs in reverse: tune the rules with lint.xml, configure failure modes in build.gradle.kts, and put it in CI. Let's do them in order.

The lint {} block in build.gradle.kts

Step one is wiring up failure behavior in your app module's build.gradle.kts. This is the contract between Lint and Gradle: which findings stop the build, which reports get generated, where the baseline lives.

app/build.gradle.kts
android {
  // ...

  lint {
    // Stop the build on errors. This is the only way new errors actually get fixed.
    abortOnError = true

    // Don't escalate warnings yet — we'll tune severity per-issue in lint.xml.
    warningsAsErrors = false

    // Run lint on the release variant too. Catches release-only issues like
    // ProGuard/R8 incompatibilities and resource problems.
    checkReleaseBuilds = true

    // The baseline file: existing findings live here, new ones fail the build.
    baseline = file("lint-baseline.xml")

    // Reports for both humans and CI. The HTML is for you; the XML is for CI tools.
    htmlReport = true
    htmlOutput = file("$buildDir/reports/lint/lint-results.html")
    xmlReport  = true
    xmlOutput  = file("$buildDir/reports/lint/lint-results.xml")

    // Severity-by-issue lives in lint.xml — these are project-wide overrides only
    // for issues you really want to globally turn off (e.g., a rule that's broken).
    disable += setOf("MissingTranslation")  // Example only — only disable if you're sure.

    // Make these issues fatal regardless of their default severity.
    fatal += setOf("StopShip")
  }
}
Most teams don't need disable at all. Reach for it only when a rule is misfiring on your codebase — and prefer per-issue severity in lint.xml for everything else.

The lint.xml file — per-issue severity

The lint {} block sets global behavior. The lint.xml file in your module root sets per-issue severity. This is where most of the actual tuning happens. The structure is XML, the schema is forgiving, the file is checked into source control.

app/lint.xml
<?xml version="1.0" encoding="UTF-8"?>
<lint>
  <!-- Promote: showstoppers must fail the build, not whisper a warning -->
  <issue id="NewApi"            severity="error" />
  <issue id="MissingPermission" severity="error" />
  <issue id="WrongConstant"     severity="error" />
  <issue id="InvalidPackage"    severity="error" />
  <issue id="HardcodedText"     severity="error" />

  <!-- Demote: things you don't currently care about (be honest, not lazy) -->
  <issue id="IconLauncherShape" severity="ignore" />

  <!-- Scope a rule to specific files: don't lint generated code -->
  <issue id="HardcodedText">
    <ignore path="**/build/generated/**" />
    <ignore path="**/test/**" />
  </issue>

  <!-- Per-file regex ignores when one file is a legitimate exception -->
  <issue id="ObsoleteSdkInt">
    <ignore regexp=".*LegacyCompat\.kt" />
  </issue>
</lint>
Place lint.xml next to your module's build.gradle.kts. Lint discovers it automatically. For multi-module repos you can also put a shared lint.xml at the root and reference it via lintConfig = file(...).

Generating the baseline

You have a tuned config but a four-year-old codebase that will produce thousands of findings. The baseline file is what makes that survivable. Generate it once, commit it, and Lint will read it forever after — failing only on new findings.

Generate the baseline
# AGP 8.1+ — recommended task name
./gradlew :app:updateLintBaseline

# Older: just delete the existing baseline file and run lint — it auto-creates
rm app/lint-baseline.xml
./gradlew :app:lintDebug
git add app/lint-baseline.xml
git commit -m "chore: lint baseline"
The baseline file ends up at app/lint-baseline.xml by default. Commit it. Yes, it's noisy. Yes, that's fine. Configure your repo's .gitattributes to mark it as linguist-generated=true so GitHub stops counting its lines in your stats.

The baseline only suppresses findings that match the existing entries exactly. If you move a class to a new file, Lint may treat the same finding as new and fail the build. Treat occasional baseline regenerations (e.g. quarterly) as normal — not as a failure of the workflow.

Severity tuning — the strategy

You have four severity levels: fatal, error, warning, informational. Most teams use too many. The strategy that survives contact with reality:

🛑 error

Showstoppers. Things that will crash, leak, or violate a platform contract. NewApi, MissingPermission, WrongConstant, InvalidPackage, HardcodedText. Fail the build.

⚠️ warning

Real issues, but not always actionable in the immediate context. UnusedResources, ContentDescription, SetTextI18n. Visible in IDE; visible in CI report; doesn't block merge.

ℹ️ informational

Taste rules, edge cases, opt-in checks. Most teams should leave the default here. If a rule is informational by default, don't promote it without a reason.

🔇 ignore

Rules that are wrong for your codebase. Be honest: if you're not going to fix it, ever, ignore it. Don't pretend it's a warning — that's just permanent log noise.

Anti-pattern: every team I've audited has at least one warning-level rule that is silently producing 200+ findings nobody will ever fix. Either fix the rule (promote to error and clear it) or ignore it (admit you're not going to fix it). Permanent yellow is just rot.

The five categories worth promoting today

If you do nothing else after reading this chapter, promote these five. They catch real bugs that ship to real users. None of them are stylistic.

A short detour: custom lint checks

Lint is extensible. You can write your own checks in Kotlin, package them as a separate module, and the AGP will pick them up automatically. The reasons to do this are narrow but valuable: enforcing in-house API rules (“don't call SimpleDateFormat directly — use SafeDateFormat”), catching leaks specific to your architecture, banning a deprecated internal API.

lint-rules/build.gradle.kts (skeleton)
plugins {
  alias(libs.plugins.kotlin.jvm)
  alias(libs.plugins.android.lint)
}

dependencies {
  compileOnly("com.android.tools.lint:lint-api:31.4.0")
  compileOnly("com.android.tools.lint:lint-checks:31.4.0")
}

// And in your :app module:
// dependencies { lintChecks(project(":lint-rules")) }
Don't write a custom rule until you've felt the pain three times. The setup is small but maintaining it indefinitely is real work. Detekt (Chapter 4) is usually a better home for in-house Kotlin smells.

AI-Assisted Setup: /triage-lint-baseline

Your first lint baseline is going to be hundreds or thousands of entries. Reading through it manually is morale-destroying. This is exactly the kind of structured-text triage that Claude Code does well — we'll build a reusable skill that takes the baseline, groups findings, and produces a punch list ranked by likely effort.

.claude/commands/triage-lint-baseline.md
---
description: Triage a lint baseline by category, severity, and likely effort
---

You are looking at the file `app/lint-baseline.xml`. Read it.

Produce a markdown report with these sections:

  ## Summary
    - total entries
    - top 5 issue IDs by count

  ## By category (one section per top-5 issue ID)
    For each: count, 3 representative file paths, 1-2 sentence description
    of the rule, estimated effort to clear all instances (S/M/L), and
    whether the rule is currently a warning or error in lint.xml.

  ## Quick wins
    Categories where effort is S and impact is real. List up to 3.

  ## Recommend severity changes
    Issue IDs that are currently warning but should be error based on what
    you see in the baseline (e.g., HardcodedText with 400+ entries means
    the team has been ignoring localization — promote and bulk-fix).

Be terse. The whole report should fit on one printed page.
The first time you run this on a real baseline, you'll typically discover one or two categories you didn't realize were dominant. Those are your quick wins.
Good prompt
/triage-lint-baseline Project is a 5-year-old Kotlin Android app. minSdk 24, targetSdk 34. Currently 1,847 entries in app/lint-baseline.xml. We want the top 3 quick wins for next sprint.

The CI hook

Lint that runs on your machine is lint that runs nowhere. The CI gate is what turns “I tuned lint.xml” into “the team has lint.” The minimum useful CI job runs lint on the release variant on every PR and surfaces the report.

.github/workflows/lint.yml
name: Android Lint

on:
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '17'

      - uses: gradle/actions/setup-gradle@v3

      # The actual gate: lint on the release variant, abortOnError=true
      # means a new finding fails the build, blocking the merge.
      - name: Run Android Lint
        run: ./gradlew :app:lintRelease

      # Always upload the report — even on success — so reviewers can read it.
      - name: Upload lint report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: lint-report
          path: app/build/reports/lint/
Once this job is green on main, enable branch protection: Settings → Branches → Branch protection rule → Require status checks → tick "Android Lint." Now the gate is real.

IDE integration — close the loop

Android Studio runs Lint inspections in real time as you type. The catch: by default it does not respect your project's lint.xml severities. Until you tell it to, the IDE shows HardcodedText as a warning while CI rejects it as an error — a confusing developer experience.

Two settings to flip per developer machine (or commit to .idea/inspectionProfiles/ if you want it standardized):

  • Settings → Editor → Inspections → Android → Lint — tick “Use lint settings from lint.xml.” This makes the IDE match the build's severities exactly.
  • Settings → Tools → Actions on Save — enable “Run code cleanup” and “Reformat code.” Lint findings show up as save-time fixes the developer sees before they even commit.

If you commit .idea/inspectionProfiles/Project_Default.xml, the whole team gets the same IDE inspection severities automatically. Most teams gitignore the entire .idea/ folder — you don't have to. Selectively committing the inspection profile is a high-leverage move.

Common lint config pitfalls

Three traps every team falls into at least once:

  • Globally disabling a rule because one file misfires it. Use <ignore path="..."> in lint.xml instead. Disabling a rule project-wide because one legacy file violates it disables it for all your future code, too.
  • Regenerating the baseline on every CI run. Some teams add ./gradlew :app:updateLintBaseline to CI “to keep it fresh.” This silently absorbs every new finding and defeats the entire point. The baseline is generated locally, intentionally, and committed by a human.
  • Treating warningsAsErrors = true as the goal. It sounds rigorous; in practice it produces a fragile build that breaks on every new AGP minor that adds checks. Use per-issue severity in lint.xml. Be explicit about what blocks merges.

If your lint job in CI is taking 8+ minutes, you've probably enabled checkDependencies = true. That recursively lints every library module from your app module — useful for shared rule enforcement, but expensive. For most repos the default (off) is correct.

The adoption ladder

Day 1

Add the lint {} block. Add a minimal lint.xml promoting the five high-signal categories. Generate the baseline. Commit. 2–4 hours.

Week 1

Run /triage-lint-baseline. Pick one quick-win category. Fix it across the codebase, regenerate baseline. Enable IDE inspections to match the lint.xml severity for the whole team.

Month 1

Add the GitHub Actions workflow. Enable branch protection requiring it. Optionally write your first custom lint check (only if you have a clear use case). Re-audit the baseline shrinkage in quality-debt.md.

Exercise: turn lint into a merge blocker

Promote and gate, in one PR

Set aside a half-day. From your project root:

  1. Add the lint {} block from earlier in this chapter to your app module's build.gradle.kts. Adjust paths if your output dir differs.
  2. Create app/lint.xml with the five promoted categories: NewApi, MissingPermission, WrongConstant, HardcodedText, InvalidPackage.
  3. Run ./gradlew :app:updateLintBaseline. Verify app/lint-baseline.xml exists and has entries.
  4. Run ./gradlew :app:lintDebug — it should pass (the baseline absorbs everything).
  5. Save .claude/commands/triage-lint-baseline.md from this chapter. Run /triage-lint-baseline on your real baseline. Read the report.
  6. Add the GitHub Actions workflow. Push the branch. Verify the CI job runs and is green.
  7. Open the PR. Title: “chore(lint): tune severity, baseline existing findings, gate in CI.”

Done. The day you merge this PR, every new HardcodedText in the codebase is a build failure. That is what enforcement looks like.
