Chapter 2

Auditing Your Project + Baselining Existing Sins

Thirty minutes of honest looking, one page of findings, and the single concept that unlocks everything else: the baseline file.

From "we should do this" to "here's the audit"

Last chapter we agreed: you've been putting quality tooling off, the cost is invisible-but-real, and the way out is one tool a week with a baseline file in hand. Good. This chapter is where it stops being a pep talk.

The first thing every team does wrong is install detekt before they understand their own project. They generate 4,000 findings, panic, disable half the rules, and abandon the whole thing by Friday. We're going to do the opposite. Before we touch a single plugin, we're going to spend 30 minutes looking. The output is a one-page audit you can show your team. It tells you which tool to install first and why.

The audit is not optional and it is not for show. It's a forcing function: it makes you describe your project's actual problems in writing, which prevents the most common adoption failure — installing whatever tool is currently fashionable instead of the one your codebase needs.

The 30-minute honest audit

You can learn an embarrassing amount about your project's quality posture by running three Gradle tasks. None of them require new plugins. None of them modify anything. All of them work on your real project, today.

Run from your project root
# 1. What's actually in your dependency graph?
./gradlew :app:dependencies --configuration releaseRuntimeClasspath > deps.txt

# 2. What does the built-in linter already know about your code?
./gradlew :app:lintDebug

# 3. What verification tasks does your project already expose?
./gradlew tasks --group verification
Replace :app with whichever module is your application. On a multi-module repo, also run ./gradlew dependencies at the root to see plugin dependencies.

Total runtime: ~5 minutes on a typical project. Total reading time: ~25 minutes. Output: three text files you'll read with the eyes of someone who has never seen this codebase before.

What each command tells you

  • :app:dependencies — the actual dependency tree. You'll see direct deps (your build.gradle.kts) and transitive deps (what they pulled in). Look for: version conflicts, unexpected libraries, duplicate logging frameworks, three different JSON parsers.
  • :app:lintDebug — HTML report at app/build/reports/lint-results-debug.html. This is what Android Lint, with its default config, already thinks of your code. You probably haven't read it in months. Read it now.
  • tasks --group verification — tells you what quality gates already exist (the test task, lint, anything custom). On most legacy projects this list is short.

Reading the dependency tree without crying

The output of ./gradlew dependencies is intimidating but the structure is simple. Each line is a dependency, indentation shows the parent, and the right-hand annotations tell you what Gradle did:

Excerpt from deps.txt — reading the annotations
+--- androidx.core:core-ktx:1.12.0
|    \--- androidx.core:core:1.12.0           # ← transitive: pulled in by core-ktx
+--- com.squareup.retrofit2:retrofit:2.9.0
|    \--- com.squareup.okhttp3:okhttp:3.14.9 -> 4.12.0   # ← (*) version override
+--- com.squareup.okhttp3:okhttp:4.12.0       # ← direct: you declared this
+--- com.google.code.gson:gson:2.10.1
\--- com.squareup.moshi:moshi:1.15.0          # ← also direct: TWO JSON parsers
The -> 4.12.0 arrow means Gradle resolved a conflict by upgrading. The (*) at the end of a line (not shown) means "shown earlier, see above." Two JSON parsers is a smell — standardize.

Three things to flag while reading:

  1. Version conflicts — lines with -> arrows. Usually fine, but worth knowing about. If a critical lib is being downgraded, that's a bug waiting to happen.
  2. Duplicate solutions to the same problem — two JSON parsers, two image loaders, two networking stacks. This is technical debt you can pay down once you have a version catalog (Chapter 10).
  3. Surprises — libraries you don't recognize. Find them. Sometimes they're transitive deps of a SDK you forgot about. Sometimes they're a former colleague's experiment that nobody removed.

Do not "fix" anything you find here yet. The point of the audit is to see, not to solve. Cleanup PRs that touch dependencies in week one are a classic way to lose buy-in — you'll inevitably bump something subtle and break a build. We bump dependencies behind the safety net of the version catalog and dep-analysis tooling in Chapter 10.

The three categories of quality debt

What you find in the audit will fall into three buckets. Sorting them this way isn't pedantic — each bucket has a different set of tools and a different rollout strategy.

📝 Codebase debt

Things wrong with your code: lint findings, Kotlin smells, formatting drift, dead code, unsafe nullables, deprecated APIs. Surfaced by static analysis (lint, detekt, ktlint).

Tools: Chapters 3, 4, 5.

📦 Dependency debt

Things wrong with your libraries: outdated versions, unused declarations, transitive bloat, CVEs, two libs doing one job. Surfaced by dependency-analysis tooling.

Tools: Chapter 10.

⚙️ Process debt

Things wrong with your workflow: no CI gate, no formatter, no pre-commit hooks, no branch protection. The reason "we already had detekt" turned into "nobody runs detekt."

Tools: Chapters 6, 12.

Most teams have all three, in different proportions. A 6-year-old codebase with a small team usually has a lot of codebase debt. A team that's grown fast has more dependency debt (everyone added their favorite library). A team that hasn't invested in CI has overwhelming process debt regardless of the code itself.

The baseline file: the entire unlock

This is the concept the rest of the course depends on. Read it twice.

A baseline file is a snapshot of every existing rule violation in your codebase, captured the first time you run the tool. The tool reads the baseline on every subsequent run and silently ignores anything in it. New violations — in new code — fail the build. Old violations — in old code — do not.

That's the whole idea. It's deceptively simple, and it is the only reason you can adopt these tools on a real, messy, four-year-old codebase without committing to a multi-week cleanup PR.

What a baseline is

  • An XML or text file, committed to your repo
  • Generated by the tool itself, on demand, with one Gradle task
  • Read by the tool on every subsequent run
  • Updated when you intentionally regenerate it (after fixing a batch of issues, or as part of a quarterly cleanup)
  • Versioned. You can diff it across PRs. New entries appearing means somebody silenced a violation rather than fixing it — that's a code-review signal, not a normal occurrence.

What a baseline is not

  • Not a TODO list. The whole team treating it as "we'll fix this later" is the path to never fixing any of it. The right framing: we will not regress.
  • Not permanent. It shrinks naturally as people fix things while working in a file (most tools re-emit a violation if it moves; many tools auto-update on next regenerate).
  • Not a way to silence rules you disagree with. If a rule is wrong for your codebase, disable it — don't baseline its findings. Baselines are for rules you agree with but can't fix all the violations of immediately.
  • Not something you regenerate every CI run. The whole point is the diff — new violations should fail the build, not silently disappear into the baseline.

How baselines work across the stack

Almost every tool in this course supports baselines. The mechanics differ slightly per tool, but the concept is universal. Here's the cheat sheet — deep dives are in the relevant chapters.

app/build.gradle.kts — baseline declarations
android {
  // Android Lint — the file Gradle reads/writes
  lint {
    baseline = file("lint-baseline.xml")    // Chapter 3
  }
}

// Detekt — the baseline file path is a property
detekt {
  baseline = file("$rootDir/config/detekt/baseline.xml")  // Chapter 4
}

// Ktlint — via the spotless plugin, no separate file: format runs in-place
spotless {
  kotlin { ktlint("1.3.1") }                // Chapter 5 — you "baseline" by formatting once
}
Each tool has a "generate the baseline" command: ./gradlew :app:updateLintBaseline, ./gradlew detektBaseline, etc. Generate, commit, ship.

The pattern across all of them is the same:

Install plugin
→
Generate baseline
→
Commit baseline
→
Wire into CI
→
Ship

The one-page audit report

Here's what you walk away from this chapter with: a single page, in your repo, that summarizes what you found. This is the document you bring to the team standup. The point is that it's short — nobody reads a 12-page audit; everyone reads one page.

docs/quality-debt.md
# Quality Debt Audit — 2026-04-25

## Top 10 lint categories (by count, from lintDebug report)
1. HardcodedText                 412
2. UnusedResources               189
3. ContentDescription            156
4. NewApi                         87
5. MissingPermission              43
6. SetTextI18n                    38
7. ObsoleteSdkInt                 22
8. AppCompatCustomView            18
9. InvalidPackage                 12
10. WrongConstant                   7

## Top 5 dependency risks
- okhttp resolved 3.14.9 -> 4.12.0 (transitive override)
- gson AND moshi both present (standardize on moshi)
- com.android.support:* still on classpath (migrate to androidx)
- 3 dependencies last updated >2 years ago (kotlin-stdlib-jdk7, etc.)
- No CVE scanning configured

## Process gaps
- No CI quality checks (only :app:assembleDebug runs)
- No formatter (PR style debates frequent)
- No pre-commit hooks
- No branch protection on main
- No mapping.txt uploaded on release

## Recommendation: start with Ktlint + Spotless (Chapter 5)
PR style debates are the most-felt pain. Format-on-save kills them in week 1
and earns social capital for installing detekt next.
This is the entire deliverable. One page. Top 10 / Top 5 / Process gaps / Recommendation. Not 12 pages.

Be specific in the recommendation. "Adopt the quality stack" is a non-decision. "Start with ktlint + Spotless because PR debates are our biggest pain" is a decision. The audit is what lets you make a defensible one.

Picking your starting point

Once you have the audit, the question is which tool to install first. There is no universal right answer; it depends on which kind of debt is loudest in your project. Here's the rough heuristic:

If your audit screams...Start withChapter
Lots of unsafe nullable handling, complex methods, exception-swallowDetekt4
Mystery release crashes you can't decodeR8 + Crashlytics mapping11
Team conflicts on PR styleKtlint + Spotless5
App leaks memory after long sessionsLeakCanary8
Frame drops, ANRs, jank you can't reproStrictMode7
Dependency mess, version drift, unused libsVersion catalogs + dep-analysis10
Quality is high in IDEs but bad code still mergesCI gate + convention plugin12

If two of these apply, pick the one with the highest team morale impact. PR style debates feel terrible even when the actual cost is small — eliminating them earns you the social capital to do everything else.

"Don't fix it all" — the cleanup-PR trap

The single most common way teams blow up an adoption is the multi-week cleanup PR. The story goes: someone gets ambitious, decides "let's just fix all the lint warnings in one big PR," and disappears for two weeks. The PR is 4,000 files. Three of them have subtle behavior changes. The reviewer is dead inside. It never merges. Quality tooling is now associated with that experience.

The first PR after installing a tool should add the tool, generate the baseline, and ship. That's it. Fix the existing violations incidentally, while doing other work, over weeks and months — never as a single sprint goal. The whole point of the baseline is to make this possible.

If your team really wants to do a cleanup, scope it: "this sprint, we'll fix the top 3 categories from the lint baseline" — that's a defensible chunk. "Fix the entire baseline" is not.

The audit-and-baseline rollout

Even the audit itself follows the adoption-ladder pattern from Chapter 1. The audit is not a one-and-done event; it's the first checkpoint on a recurring cadence.

Day 1

Run the three Gradle commands. Skim the lint HTML report. Skim the dependency tree. Write the one-page quality-debt.md. Open the audit PR.

Effort: 45 minutes. Output: a document.

Week 1

Install the first tool you recommended. Generate its baseline. Commit. Wire into CI. Append the row to the adoption log in quality-debt.md.

Effort: a half-day.

Month 1

Re-run the audit at the end of the month. Compare. The lint count should be flat or down. The dep risks should be unchanged (we haven't gotten there yet). The process gaps should have one fewer entry.

Effort: 30 minutes per month.

The recurring audit is what surfaces drift. Without it, you'll genuinely forget what the codebase looked like before you started, and that makes it impossible to demonstrate progress to your manager. With it, you have a number every month that goes down.

Common audit pitfalls

Three traps to watch for as you read your own audit output:

  • Confusing transitive deps with direct ones. A library you've never heard of in deps.txt is usually transitive — pulled in by something else you depend on. Don't try to remove it. Remove the parent if you don't need it.
  • Misreading lint counts. 412 HardcodedText findings sounds catastrophic, but most of them are probably in three or four screens. Severity matters more than count. Prioritize error-level findings first; warning-level can wait.
  • Confusing “old” with “risky”. A 2-year-old library is not automatically a problem. kotlin-stdlib-jdk7 hasn't moved because it doesn't need to. The risk is libraries that have moved and you haven't, especially if security patches landed in the gap.

If you find yourself spending more than 45 minutes on the audit, you're trying to fix things instead of describe them. Stop. Write what you have. The fixing happens in the next 11 chapters.

Exercise: audit your real project

Run the audit on the project you actually ship

Set aside 45 minutes. From your project root:

  1. Run the three Gradle commands from earlier in this chapter. Save outputs.
  2. Open app/build/reports/lint-results-debug.html. Scroll. Read the top 10 issue categories. Note the counts.
  3. Skim deps.txt. Flag: any version conflicts? Any duplicate solutions to the same problem? Any libraries you don't recognize?
  4. Create docs/quality-debt.md in your repo following the template above.
  5. Pick one recommendation and write it as the last section. Be specific about why.
  6. (Optional but recommended) Save the /audit-android-project skill in .claude/commands/. Re-run it. See if Claude's audit matches yours.
  7. Open a PR with just the new docs/quality-debt.md. No code changes. The PR is a document.

Done. You now have a written, defensible starting point. Next chapter we install the first tool.
