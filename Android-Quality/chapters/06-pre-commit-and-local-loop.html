Chapter 6

Pre-commit Hooks & The Local Dev Loop

Stop finding out about formatting failures 7 minutes into a CI run. Move every cheap check to the moment of git commit — feedback in seconds, not coffee breaks.

The 7-minute round trip

Picture the loop without local hooks. You write code. You commit. You push. You open a PR. You context-switch to Slack. Seven minutes later GitHub Actions emails you: spotlessCheck failed. You switch back, stare at a one-line error, run ./gradlew spotlessApply, commit, push, wait again. Total cost of that one missing trailing comma: 14 minutes plus two context switches.

Now picture it with a pre-commit hook. You write code. You type git commit. Half a second later: "spotlessCheck failed on Foo.kt — fixed; please re-stage." You re-stage, commit again, push. Total cost: 5 seconds. Same check, same rule, different layer. Left-shift is not a mantra; it's arithmetic.

This chapter wires up the local dev loop — the IDE, the save action, the pre-commit hook, the pre-push hook — so by the time CI runs, it's a formality. The same checks already passed locally, in seconds, before anything left your laptop.

lefthook

A fast, language-agnostic Git hook manager written in Go. Single binary, no Python or Node dependency, and it runs hooks in parallel.

What it catches
Anything you tell it to — ktlint, detekt, lint, custom scripts — on the staged subset of files only, so commits stay snappy.
Install
brew install lefthook or scoop install lefthook, then lefthook install in the repo. The config lives at lefthook.yml in the repo root.
Runtime cost
Pre-commit on staged files: typically 200ms–2s. Pre-push (full module check): 30s–2min. Native parallelism keeps it tight.
When to skip it
If your repo already has a working JS/TS toolchain, husky is fine — you save a binary install. For Android-only repos, lefthook avoids dragging Node in just for hooks.

The dev loop ladder

You don't pick one layer. You build the ladder. Each rung catches what the rung below missed; each is faster than the one above.

IDE inspection → Format on save → Pre-commit → Pre-push → CI gate
  • IDE inspection — live highlighting as you type. Fastest feedback in existence (zero seconds).
  • Format on save — ktlint/Spotless rewrites the file on Ctrl+S. Catches 95% of formatting issues before you even notice.
  • Pre-commit — runs on the staged file subset. Catches what the IDE missed (custom rules, files edited outside the IDE, contributors with different settings).
  • Pre-push — runs the heavier checks (full lint, detekt, unit tests of changed modules) before the network ever sees your branch.
  • CI gate — the safety net. If the local layers worked, CI is a stamp; if not, it's the wall.

Native Git hooks — and why they're not enough

Git ships with hooks already. Drop a script at .git/hooks/pre-commit, mark it executable, and Git runs it. So why not stop there?

  • Per-clone, not per-repo. The .git/ directory is local-only; hooks aren't checked in. Every developer has to install them manually. New hire? Hooks won't run until they remember to copy a script. CI agent? Doesn't have them. Inconsistent.
  • Easy to skip. git commit --no-verify bypasses hooks silently. That's sometimes legitimate; usually it's "I'm in a hurry, who cares."
  • Bash-only by default. Cross-platform shell scripting is a tax most teams don't want to pay.
  • No parallelism, no scoping. A vanilla hook that runs ./gradlew spotlessApply on every commit takes 8 seconds. Run it on staged files only and it's 300ms. The native system gives you no help with that.

Hook frameworks fix all four. Config goes in source control, install is one command, scripts are declarative, and most frameworks parallelize and scope to staged files automatically.

For most Android teams, the answer is lefthook. It's the cleanest "no extra runtime" choice on a JVM project. The rest of this chapter assumes lefthook; the concepts translate one-to-one to husky and pre-commit if your repo's situation pushes you that way.

Installing lefthook

Three steps. The first two are per-developer setup; the third is what gets checked in.

  1. Install the binary. macOS: brew install lefthook. Windows: scoop install lefthook. Linux: curl from GitHub releases or use the Snap. Verify with lefthook version.
  2. Wire it into the repo. Run lefthook install from the repo root. This writes the actual .git/hooks/* scripts that delegate to lefthook's config.
  3. Commit lefthook.yml. The config lives at the repo root. Every clone of the repo, every CI agent, every new hire's first checkout sees it — and a lefthook install in the README setup steps wires it up.
lefthook.yml (repo root)
# Pre-commit: fast checks on staged files only.
# Goal: under 2 seconds total. Anything slower goes to pre-push.
pre-commit:
  parallel: true
  commands:
    spotless:
      glob: "*.{kt,kts}"
      run: ./gradlew spotlessApply -PspotlessFiles="{staged_files}"
      stage_fixed: true
    detekt-staged:
      glob: "*.{kt,kts}"
      run: ./gradlew detekt -PdetektInput="{staged_files}"

# Pre-push: heavier checks before the network ever sees your branch.
# Goal: under 2 minutes. If it's slower, demote it to CI only.
pre-push:
  parallel: true
  commands:
    lint:
      run: ./gradlew :app:lintDebug
    detekt-full:
      run: ./gradlew detekt
    spotless-check:
      run: ./gradlew spotlessCheck
The {staged_files} token is lefthook magic — it's substituted with the list of files Git is about to commit. stage_fixed: true auto-stages files that the hook modified (e.g., spotlessApply rewriting indentation), so the commit captures the cleaned version.

Format on commit vs format on save

Both are good. They cover different gaps.

Format on save
Triggers on Ctrl+S in the IDE. Catches 95% of formatting before the file even hits disk in "unformatted" form. Limit: only fires for the dev's chosen IDE. Files edited from the terminal, contributors with different IDEs, AI-generated patches — none of those go through it.
Format on commit (pre-commit)
Triggers on git commit. Catches the remaining 5% — files edited from the terminal, contributor IDEs without format-on-save, generated code, patches dropped in via cherry-pick. Limit: ~300ms–2s of latency on the commit. Fast enough to be invisible if scoped to staged files.

Run both. Format-on-save is the convenience; pre-commit is the guarantee.

The pre-push hook — bigger checks, still local

Some checks are too slow for pre-commit (full lint takes 30 seconds; detekt across all modules takes a minute). They don't belong in the half-second window between Enter and your prompt returning. They do belong before git push, where you've mentally moved on but haven't paid the CI tax yet.

The pre-push hook is the right home for: full lint, detekt across all modules (not just staged files), ./gradlew :app:assembleDebug if you want a compile sanity check, and unit tests of modules whose files changed since last push.

The 2-minute rule. If your pre-push hook takes longer than two minutes, demote the slowest task to CI. Anything beyond that and developers will --no-verify their way around it — and your local layer becomes useless. Pre-push is for "fast enough to wait through"; the heaviest stuff (instrumented tests, R8 release builds, dependency-vulnerability scans) belongs in CI.

Speeding up local checks

The fastest hook is one that doesn't fire on irrelevant files. Beyond that, three Gradle settings are worth flipping.

gradle.properties
# Keep the daemon alive between invocations — cold starts cost ~3s.
org.gradle.daemon=true

# Cache configuration phase between builds. Massive win for hooks
# that invoke Gradle 5+ times a day.
org.gradle.configuration-cache=true

# Parallel execution across modules.
org.gradle.parallel=true

# Build cache: reuse outputs of unchanged tasks.
org.gradle.caching=true

# Don't rebuild the world when only one module changed.
org.gradle.configureondemand=true
Configuration cache is the biggest single win for hook latency — Gradle 8+ supports it for almost all configurations. If you hit a plugin that's incompatible, the build will tell you exactly which one. File an issue upstream and disable selectively, not globally.

For pre-commit specifically, scope the work. Don't run ./gradlew detekt across the whole repo on every commit; pass only the staged files via {staged_files}. Don't run ./gradlew lint at all in pre-commit — it touches the manifest and resources, which means full module recompilation. Save it for pre-push.

The IDE rung

Hooks are great. The IDE catching it half a second after you type it is better. Flip these in Android Studio:

  • Settings → Editor → Inspections → Android → Lint. Enable inline lint surfacing for the categories you promoted to error in Chapter 3 — NewApi, HardcodedText, WrongConstant. They show up as red squiggles before you save.
  • Settings → Tools → Actions on Save. Enable "Reformat code," "Optimize imports," "Run code cleanup." All firing on Ctrl+S.
  • Settings → Version Control → Commit. Enable "Run inspections before commit" — the IDE runs its own inspection set inside the commit dialog, complementing the lefthook hook.
Don't make IDE settings your source of truth. Anything you change in the IDE is per-developer and not checked in. The settings above are convenience — the actual gates are .editorconfig, lefthook.yml, and CI YAML. If you're tempted to "ask the team to enable X in their IDE" as a quality gate, you're building a system that depends on memory. Don't.

AI-Assisted Setup — the /draft-lefthook-config skill

Lefthook configs are short but project-specific. The right glob, the right Gradle task names, the right modules to check — these depend on your repo. Hand-writing one means reading the lefthook docs and remembering all your task names. Letting Claude draft it from your build.gradle.kts + your project structure is a 60-second job.

Skill: /draft-lefthook-config
Read the root build.gradle.kts and settings.gradle.kts to understand this project's modules and the registered Gradle tasks. Then produce a lefthook.yml at the repo root with: - pre-commit: spotlessApply on staged Kotlin files (use the {staged_files} token), with stage_fixed: true so cleaned files re-enter the commit. Add detekt on staged files only. - pre-push: full ./gradlew spotlessCheck, ./gradlew detekt, and ./gradlew :app:lintDebug. Mark them parallel: true. Use only tasks that actually exist in this project (verify by running ./gradlew tasks first). If a task is missing, suggest the most likely candidate and flag it for me to confirm. Optimize for "all under 2 seconds on pre-commit, all under 2 minutes on pre-push." Comment any line that has a non-obvious choice.
Save this to .claude/commands/draft-lefthook-config.md. The "verify tasks first" line matters — without it, you sometimes get a config referencing a task that doesn't exist in this particular module setup, and the hook fails on first commit.

Companion skill worth standing up: /explain-hook-failure. When a teammate's commit is blocked by a hook and they don't know why, paste the hook output and have Claude diagnose it. Most often it's a missing IDE setting, occasionally a stale Gradle daemon.

Skill: /draft-lefthook-config (don't do this)
Generate a lefthook config for an Android project.
No grounding in your project's task names, modules, or scoping conventions. You'll get a generic template that often references tasks that don't exist (:app:detekt when your project has it at the root, :app:lintRelease when only :lintDebug is wired up). The hook fails on the first commit and the team thinks lefthook is broken. Always feed the skill your real build.gradle.kts first.

The bonus: commit-msg hook for conventional commits

If your team has decided on Conventional Commits (or any structured commit format), a commit-msg hook is the cheap enforcement. It runs after the commit message is typed but before the commit lands, and rejects messages that don't match the pattern.

lefthook.yml (additional block)
# commit-msg: validate the message format itself.
# Runs after the dev types their message, before the commit lands.
commit-msg:
  commands:
    conventional-commits:
      run: |
        if ! head -n 1 "{1}" | grep -qE '^(feat|fix|docs|style|refactor|perf|test|chore|build|ci|revert)(\(.+\))?: .{1,72}$'; then
          echo "Commit message must follow Conventional Commits."
          echo "Format: type(scope): description"
          echo "Example: feat(auth): add biometric login fallback"
          exit 1
        fi
The {1} token is the path to the commit message file Git is about to use. The regex covers the standard Conventional Commit prefixes; extend as needed. Don't go overboard — teams that try to validate the whole message body in a hook lose their patience within a week.

The --no-verify question

Sooner or later someone on the team will hit git commit --no-verify. Don't ban it — that's a recipe for resentment. But treat its frequency as a quality signal.

  • Used once a quarter for a genuine emergency (production hotfix, hook is broken)? Fine. The escape hatch exists for a reason.
  • Used once a week by the same person? Either the hooks are too slow (fix that) or the rules don't fit reality (fix that).
  • Used silently across the team? Your hooks are theater. Find out why — usually one slow check is the culprit.

The right answer to repeated --no-verify is never "ban it." It's "make the hook fast and accurate enough that nobody wants to skip it." A 200ms pre-commit doesn't get bypassed; an 8-second one gets bypassed by everyone, eventually.

Day 1 Install lefthook (brew install lefthook). Add a minimal lefthook.yml with one pre-commit task: spotlessApply on staged files, stage_fixed: true. Add lefthook install to README setup. Commit and push.
Week 1 Add detekt-on-staged to pre-commit. Add spotlessCheck, detekt, and :app:lintDebug to pre-push. Enable Gradle configuration cache. Standardize IDE Actions-on-Save settings via a team doc.
Month 1 Track --no-verify usage in the team channel for a sprint. Fix the slow hook(s) people skip. Add a commit-msg hook for conventional-commit format if your team cares. Document the local-loop expectations in CONTRIBUTING.md.

What it looks like in practice

Day in the life, post-Chapter-6: you change a ViewModel. Save. Format-on-save reflows imports. Compile errors light up red as you type. You stage. You type git commit -m "fix: refresh state on resume". Lefthook fires — spotless rewrites a trailing comma you missed, detekt clears, the commit captures the cleaned version. You type git push. Pre-push runs full lint and detekt in 90 seconds. Push goes through. PR opens. CI runs the same gates — passes in 4 minutes flat because everything was already clean.

You did not see a single style nit. You did not wait 7 minutes to learn about a missing import. The local loop carried the weight, exactly as designed.

Left-shift, quantified

A team of 8 devs, each making 5 commits a day, runs ~200 commits a week. If pre-commit catches just two of those before they need a CI round-trip, you've saved 14 minutes × 2 = 28 minutes of someone's full attention every week. Multiply by 50 weeks. That's a developer-day per quarter, recovered with a 30-line YAML file.

Common pitfalls

  • Hooks that ignore {staged_files}. Running ./gradlew detekt with no scoping on every commit means every commit takes 30 seconds. Always scope to staged files in pre-commit.
  • Forgetting stage_fixed: true. Without it, spotless rewrites a file but the un-rewritten version is what gets committed. Five commits later you wonder why CI keeps complaining about already-fixed code.
  • Pre-push that takes 4 minutes. People will --no-verify their way around it. Demote the slowest task to CI.
  • Running tests in pre-commit. Unit tests are great, just not on every commit. They take seconds-to-minutes; pre-commit is for things that take milliseconds. Tests live in pre-push at the earliest.
  • Forgetting to commit lefthook.yml. If the file isn't in source control, new hires don't get the hooks. Verify with git ls-files | grep lefthook.

Onboarding new contributors

Update your README's setup section with three lines:

# Install lefthook (one-time, per machine):
brew install lefthook   # or scoop install lefthook on Windows

# Wire it into this checkout (one-time, per clone):
lefthook install

That's it. Three lines added to onboarding, infinite hours of CI round-trips removed. Add a sanity step too: lefthook run pre-commit — runs the hook on the current state without committing, useful for "did I set this up right?"

Your turn.
  1. Install lefthook on your machine (brew install lefthook or your platform's equivalent).
  2. Run the /draft-lefthook-config skill against your repo. Save the output to lefthook.yml in the repo root.
  3. Run lefthook install. Confirm .git/hooks/pre-commit now exists and is one line that delegates to lefthook.
  4. Make a deliberately bad commit: edit a Kotlin file, break the indentation, stage, and run git commit. Watch lefthook reformat it and re-stage. Note the latency.
  5. Run lefthook run pre-push manually. Time the full pre-push pass. If it's over 2 minutes, identify the slowest task and decide whether to demote it to CI-only.
  6. Add the brew install lefthook + lefthook install lines to your project README.
  7. Commit and push the lefthook.yml + README changes. Brag in your team channel about the new local loop.

Stretch goal: track --no-verify usage for a week (a quick git log --grep won't find it — ask the team to self-report). The result tells you whether your hooks are too slow or just right.

What's next

Static analysis is now wired into every layer of your loop — IDE, save, commit, push, and CI. The build itself is on tight rails. But there's a whole category of bugs your build cannot see: the ones that only happen when the app is running. Disk reads on the main thread. Leaked Activities. Network on UI. Chapter 7 turns on StrictMode — Android's runtime safety net — so you find those before your users do, in dev, where they're cheap.
