Chapter 10

Version Catalogs & Dependency Health

Switch to libs.versions.toml and add tooling that flags unused, outdated, and risky deps

The dependency mess you've been living with

Open app/build.gradle.kts in any 3+ year-old Android project and you'll find roughly the same fossil layer: retrofit:2.6.0 in one module, retrofit:2.9.0 in another, kotlinx.coroutines hardcoded in five places with three different versions, a compileSdk integer literal, and at least one comment that says "TODO: bump this when we have time."

It's not a code-quality problem in the sense lint or detekt would catch. It's an operational problem: every dependency upgrade requires a grep + multi-file edit. Every "which version of OkHttp are we on?" question requires reading three modules. And nobody knows which deps are actually used vs. which were added in 2022 for a feature that got cut.

This chapter solves that. Step one is the modern way to declare dependencies: a Gradle version catalog, a single libs.versions.toml file that becomes the source of truth. Step two is wiring two small plugins that automatically tell you which deps are unused, which are outdated, and which have known CVEs.

The frame for this chapter: your dependency list is a tax surface. Every line is something to upgrade, audit, or remove. Catalogs centralize the surface. Health plugins give you a report on it. Together they convert "I have no idea what's in this APK" into "I know exactly, and I get a weekly diff."

Meet the Tools

Meet Version Catalogs + Dependency Health

What it is: a Gradle-native dependency catalog (gradle/libs.versions.toml) plus two plugins — com.autonomousapps.dependency-analysis (find unused/misused deps) and com.github.ben-manes.versions (find outdated deps).

What it catches that humans miss: unused dependencies bloating your APK, version drift across modules, deps declared api when they should be implementation (and vice versa), transitive deps you should declare directly, libraries your team is 4+ minor versions behind on.

Install snippet: create gradle/libs.versions.toml (built-in to Gradle 7.0+, no plugin needed) and apply com.autonomousapps.dependency-analysis + com.github.ben-manes.versions to your root project.

Runtime cost: zero. These are build-time checks. The catalog adds milliseconds to configuration; the plugins are tasks you invoke explicitly (./gradlew buildHealth, ./gradlew dependencyUpdates) or wire to a weekly cron.

When to skip it: if your project has fewer than 10 dependencies and one module, just inline them. Catalogs pay back at multi-module scale.

What a version catalog actually looks like

The catalog is a single TOML file at gradle/libs.versions.toml. Gradle picks it up automatically — no settings.gradle.kts registration needed. It has four sections: [versions], [libraries], [plugins], and [bundles].

[versions]
agp = "8.7.3"
kotlin = "2.0.21"
ksp = "2.0.21-1.0.28"
compose-bom = "2024.12.01"
core-ktx = "1.15.0"
lifecycle = "2.8.7"
retrofit = "2.11.0"
okhttp = "4.12.0"
moshi = "1.15.1"
hilt = "2.52"
coroutines = "1.9.0"
junit = "4.13.2"
mockk = "1.13.13"

[libraries]
# AndroidX
androidx-core-ktx       = { module = "androidx.core:core-ktx",        version.ref = "core-ktx" }
androidx-lifecycle-rt   = { module = "androidx.lifecycle:lifecycle-runtime-ktx", version.ref = "lifecycle" }
androidx-lifecycle-vm   = { module = "androidx.lifecycle:lifecycle-viewmodel-ktx", version.ref = "lifecycle" }

# Networking

retrofit-moshi          = { module = "com.squareup.retrofit2:converter-moshi", version.ref = "retrofit" }
okhttp                  = { module = "com.squareup.okhttp3:okhttp", version.ref = "okhttp" }

# Testing

mockk                   = { module = "io.mockk:mockk", version.ref = "mockk" }

[plugins]
android-application = { id = "com.android.application", version.ref = "agp" }
android-library     = { id = "com.android.library",     version.ref = "agp" }
kotlin-android      = { id = "org.jetbrains.kotlin.android", version.ref = "kotlin" }
kotlin-compose      = { id = "org.jetbrains.kotlin.plugin.compose", version.ref = "kotlin" }
ksp                 = { id = "com.google.devtools.ksp", version.ref = "ksp" }
hilt                = { id = "com.google.dagger.hilt.android", version.ref = "hilt" }

[bundles]
networking = ["retrofit", "retrofit-moshi", "okhttp", "okhttp-logging", "moshi", "moshi-kotlin"]
compose-ui = ["compose-ui", "compose-material3", "compose-tooling-preview"]
testing    = ["junit", "mockk", "coroutines-test"]

The naming convention matters. Hyphens in TOML keys become dots in Gradle: androidx-core-ktx in the catalog becomes libs.androidx.core.ktx in your build.gradle.kts. Pick a convention and stick to it — usually {group}-{artifact} with the noisy parts dropped (e.g. androidx-core-ktx, not androidx-androidx-core-core-ktx).

What a module's build.gradle.kts looks like after

app/build.gradle.kts — consuming the catalog

plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.android)
    alias(libs.plugins.kotlin.compose)
    alias(libs.plugins.ksp)
    alias(libs.plugins.hilt)
}

dependencies {
    implementation(libs.androidx.core.ktx)
    implementation(libs.androidx.lifecycle.rt)
    implementation(libs.androidx.lifecycle.vm)

    // Compose: BOM aligns versions; individual artifacts have none
    implementation(platform(libs.compose.bom))
    implementation(libs.bundles.compose.ui)
    debugImplementation(libs.compose.tooling)

    // Networking — one bundle, six artifacts
    implementation(libs.bundles.networking)

    implementation(libs.hilt.android)
    ksp(libs.hilt.compiler)

    implementation(libs.coroutines.android)

    testImplementation(libs.bundles.testing)
}

Notice three things: (1) plugin IDs come from libs.plugins, (2) bundles let you pull in groups of artifacts in one line, (3) the compose-bom trick — you declare the BOM as a platform() and the individual Compose artifacts inherit the BOM's pinned versions.

The "where is this version pinned?" answer is now one file. When you bump retrofit from 2.11.0 to 2.12.0, you change one line in libs.versions.toml. Every module that uses it picks up the new version automatically. Gone is the multi-grep-multi-edit ritual.

Migrating an existing project

The migration is mechanical and tedious. It is the textbook AI-assisted task — deterministic transforms across many files. Don't do it by hand.

The order that won't break things

  1. Create gradle/libs.versions.toml with all your existing deps & their current versions copied verbatim. Don't upgrade anything yet.
  2. Migrate one module (start with the smallest leaf, not app). Run ./gradlew :feature-foo:assembleDebug — if it builds, you've validated the catalog entries.
  3. Migrate the rest module by module, building each time. Don't try to do all of them in one PR.
  4. Migrate plugins in your top-level build.gradle.kts last. The plugins {} block uses alias(libs.plugins.X) instead of id("...") version "...".
  5. Delete the old version constants — buildSrc/Versions.kt, ext { } blocks, hardcoded version strings.
  6. Now consider upgrades. Don't mix migration with upgrades in the same PR.
Don't use the auto-migration IDE action blindly. Android Studio has a "Replace with new dependency notation" option that converts your existing build files. It works for simple cases but produces ugly names (libs.com.squareup.retrofit2.retrofit) and doesn't do bundles. Use it as a starting point, then clean up the names by hand or with the AI skill below.

The dependency-analysis plugin: find unused deps

Now that all your deps live in one catalog, you'd love to know which ones are actually used. Enter dependency-analysis-gradle-plugin by Tony Robalik (@autonomousapps on GitHub). It analyzes your bytecode and reports: unused implementations, abused api declarations, transitive deps you depend on directly, and more.

Root build.gradle.kts — apply once at the root

plugins {
    // ... your existing plugins
    id("com.autonomousapps.dependency-analysis") version "2.6.1"
}

This plugin is unique — it's applied at the root project, not per-module. It then introspects every subproject automatically.

Running it

Run buildHealth across the whole project

$ ./gradlew buildHealth

> Task :buildHealth FAILED

Advice for :feature:profile
  Unused dependencies which should be removed:
    implementation("com.squareup.retrofit2:converter-gson:2.9.0")
    implementation("com.airbnb.android:lottie:6.1.0")

  Existing dependencies which should be modified to be as indicated:
    api("androidx.core:core-ktx:1.15.0") (was implementation)

  Transitively used dependencies that should be declared directly:
    implementation("androidx.annotation:annotation:1.7.0")

Three categories of advice: remove (unused), change (wrong configuration), add (transitive used directly). All three matter. The "wrong configuration" advice is what catches the worst kind of fragility — an api declaration leaking implementation details to consumer modules.

The first buildHealth run on a legacy project will return hundreds of items. Don't try to fix them all in one PR. Pick the unused-deps category first — pure deletion, lowest risk, biggest APK-size payoff. Save the api/implementation tweaks for later when you have time to test consumer modules.

The versions plugin: find outdated deps

Different problem, different plugin. com.github.ben-manes.versions answers: which of your declared dependencies have newer versions available? It scans Maven Central, JitPack, and your other configured repos and tells you what's behind.

Root build.gradle.kts

plugins {
    id("com.github.ben-manes.versions") version "0.51.0"
}

// Filter: ignore alphas, betas, RCs unless you opt in
fun isStable(version: String): Boolean {
    val stableKeywords = listOf("RELEASE", "FINAL", "GA")
        .any { version.uppercase().contains(it) }
    val regex = "^[0-9,.v-]+(-r)?$".toRegex()
    return stableKeywords || regex.matches(version)
}

tasks.withType<com.github.benmanes.gradle.versions.updates.DependencyUpdatesTask> {
    rejectVersionIf {
        !isStable(candidate.version) && isStable(currentVersion)
    }
}

The rejectVersionIf filter is critical. Without it, you'll get noise reporting every alpha and snapshot. With it, you only see stable upgrades you'd actually consider.

Run dependencyUpdates

$ ./gradlew dependencyUpdates -Drevision=release

------------------------------------------------------------
 Project Dependency Updates (report to plain text file)
------------------------------------------------------------

The following dependencies are using the latest release version:
 - androidx.core:core-ktx:1.15.0
 - junit:junit:4.13.2

The following dependencies have later release versions:
 - com.squareup.retrofit2:retrofit [2.9.0 -> 2.11.0]
 - com.squareup.okhttp3:okhttp     [4.10.0 -> 4.12.0]
 - org.jetbrains.kotlinx:kotlinx-coroutines-android [1.7.3 -> 1.9.0]

 - com.example:internal-sdk
   1.2.3

The "failed to determine" section usually means the dep is in a private repo the plugin can't reach. Add the repo to your repositories {} block or accept that you'll have to check those manually.

The vulnerability layer: CVE scanning

Outdated isn't the same as unsafe. A 2-year-old version of OkHttp may have a known CVE; a brand-new version might too. CVE scanning is its own discipline. Mention here, deep dives in the security course.

OWASP Dependency-Check

The free option. Gradle plugin org.owasp.dependencycheck. Cross-references your deps against the National Vulnerability Database. Slow first run (downloads the NVD), fast after. Generates HTML/JSON reports. Hooks into CI.

Snyk / Sonatype OSS Index / GitHub Dependabot Alerts

Hosted alternatives. Snyk and Sonatype are paid but more polished. Dependabot Alerts are free with GitHub repos and surface CVEs as repo notifications — the lowest-friction starting point for most teams.

For most teams the right starting point is GitHub's Dependabot Alerts. Free, on-by-default for public repos, opt-in for private. It will tell you within a day when a CVE drops on a dep you ship. Step up to OWASP Dependency-Check when you need build-time gating, not just notifications.

Automated upgrade PRs: Renovate & Dependabot

Knowing about outdated deps is half the win. The other half is upgrading them without the manual ritual of bump-build-test-PR. Two robots compete for this job.

Renovate

Most flexible. Configurable groups, schedules, automerge rules. Best for teams that want to batch all weekly patch updates into one PR. Slightly more config to learn.

Dependabot

Built into GitHub. Zero-setup beyond a YAML file. Excellent for "open one PR per dep upgrade." Less granular grouping than Renovate but vastly less to think about.

.github/dependabot.yml — weekly batch upgrades

version: 2
updates:
  - package-ecosystem: "gradle"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
      time: "08:00"
      timezone: "Europe/London"
    open-pull-requests-limit: 10
    groups:
      androidx:
        patterns:
          - "androidx.*"
      kotlin-stack:
        patterns:
          - "org.jetbrains.kotlin*"
          - "org.jetbrains.kotlinx*"
      networking:
        patterns:
          - "com.squareup.retrofit2*"
          - "com.squareup.okhttp3*"
          - "com.squareup.moshi*"
    labels:
      - "dependencies"
    commit-message:
      prefix: "deps"

The groups section turns 30 individual PRs ("bump retrofit," "bump okhttp," "bump moshi") into one weekly PR per logical group. This is the difference between dependency upgrades being a thing your team does and a thing your team avoids.

AI-Assisted Setup: /migrate-to-version-catalog

The biggest lift in this chapter is the initial migration. Hand it to Claude with a clear contract.

Good prompt

"Migrate this Android project to a Gradle version catalog. Read every build.gradle.kts in the project, every buildSrc/Versions.kt file, and any ext {} version blocks in the root build.gradle.kts. Produce: (1) a gradle/libs.versions.toml with [versions], [libraries], [plugins], and [bundles] sections (suggest bundles for: networking, compose UI, testing); (2) the corresponding build.gradle.kts diffs for every module. Use hyphenated naming — androidx-core-ktx, not the verbose form. Don't change versions during migration — preserve them exactly. Output the TOML and a unified diff."

Bad prompt

"Make a version catalog for me."

Why it fails: doesn't say to preserve versions, doesn't define naming, doesn't ask for bundles, doesn't ask for diffs. You'll get a generic example file and zero application of it to your project.

.claude/commands/migrate-to-version-catalog.md

# /migrate-to-version-catalog

Migrate this Gradle project to a version catalog at gradle/libs.versions.toml.

Inputs (read from disk):
- All build.gradle.kts files
- buildSrc/src/main/kotlin/**/*.kt (especially Versions.kt, Deps.kt)
- Root build.gradle.kts ext {} blocks
- gradle.properties (kotlin.version, agp.version, etc.)

Outputs:
1. gradle/libs.versions.toml with [versions], [libraries], [plugins], [bundles]
2. Unified diffs for every build.gradle.kts that needs to change
3. A list of files to delete after migration (Versions.kt, Deps.kt)

Naming convention: hyphenated, group-artifact short form.
  - androidx.core:core-ktx     → androidx-core-ktx
  - com.squareup.retrofit2:retrofit → retrofit
  - com.google.dagger:hilt-android   → hilt-android

Bundles to suggest:
  - networking: retrofit + okhttp + moshi (and converters)
  - compose-ui: ui + material3 + tooling-preview (BOM-managed)
  - testing: junit + mockk + coroutines-test

Constraints:
  - Preserve every version exactly. Migration is mechanical, not an upgrade.
  - Use plugin aliases (`alias(libs.plugins.foo)`) instead of the legacy `id() version` form.
  - Compose artifacts that are BOM-managed should NOT have a version in [libraries].
  - Output the diffs in a way I can apply with `git apply`.

The skill becomes one-shot for any Android project you join. Run it, review the output, apply the diff, build — one PR, done.

Adoption ladder

Day 1

Create gradle/libs.versions.toml with current versions copied verbatim. Migrate one leaf module to it. Verify it builds. Don't change versions, don't add bundles yet, don't migrate plugins. Total: 60 minutes with the AI skill above.

Week 1

Migrate the remaining modules + plugins block. Add bundles for the obvious groupings (networking, compose UI, testing). Apply com.autonomousapps.dependency-analysis + com.github.ben-manes.versions at the root. Run ./gradlew buildHealth — remove the top 5 unused deps in a separate PR.

Month 1

Add Dependabot or Renovate config — weekly grouped PRs for AndroidX, Kotlin stack, and networking. Wire buildHealth as a CI step (warning, not blocker, until the noise floor is low). Enable Dependabot Alerts for CVE notifications.

The before/after

Before

"Bump Retrofit to 2.12.0." You grep for retrofit2, find 8 occurrences across 5 modules, edit each, hope you got them consistent. Build fails because module 6 still has 2.9.0 transitively. APK size mysteriously grows. Six months later: nobody knows whether lottie is still used, but everyone is afraid to remove it.

After

"Bump Retrofit to 2.12.0." Edit one line in libs.versions.toml. Build. Done. ./gradlew buildHealth reports lottie as unused; you delete it in a 2-line PR. ./gradlew dependencyUpdates tells you OkHttp 5.0 is out; you decide whether to take the major bump or wait.

The pitfalls

Don't migrate and upgrade in the same PR. If a build breaks during migration, you need to know whether it's the catalog or the version bump. Two PRs, two diffs, two reviews. Faster overall.
The first buildHealth report is overwhelming. Resist the urge to fix everything. Fix the "unused dependencies" section first — lowest risk, biggest payoff. Save the api/implementation reshuffles for a dedicated tech-debt week.
Beware ungrouped Dependabot. Without groups, Dependabot opens one PR per dep per week. With 50 deps, that's a 50-PR review pile every Monday morning. Group aggressively from day one.

Exercise

Migrate & audit your real project

  1. Copy your largest module's dependency list into a gradle/libs.versions.toml. Preserve every version exactly. Hyphenated naming.
  2. Migrate that one module's build.gradle.kts to consume from the catalog. Build it. If it fails, fix the catalog entries until it doesn't.
  3. Apply com.autonomousapps.dependency-analysis at the root. Run ./gradlew buildHealth. Save the output.
  4. Apply com.github.ben-manes.versions with the isStable filter from this chapter. Run ./gradlew dependencyUpdates -Drevision=release.
  5. Open a PR with: the libs.versions.toml, the migrated module, and the two plugins applied. Title: "Migrate to version catalog + dep health plugins."
  6. Open a separate PR removing the top 3 unused deps from the buildHealth report. Title: "Remove unused dependencies."

Deliverable: two PRs, an opinion on Dependabot vs. Renovate for your team, and a written-down number for "deps removed" and "APK bytes saved." That number is what you show the next person who says "we don't have time for that."
