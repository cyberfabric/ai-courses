Chapter 5

Ktlint & Spotless: One-Click Formatting

End the tabs-vs-spaces wars. Format on save, format on commit, fail the CI build on unformatted code — and never argue about a trailing comma again.

The bikeshedding tax

Every team has had this PR comment. "Can you fix the indentation on line 47?" "Trailing comma here, please." "Why are these imports in this order?" "Wildcard import — expand it." Multiply by every PR, every reviewer, every week. That's the bikeshedding tax. It's invisible because it never appears on a Jira ticket, but it's burning hours of senior-engineer time on questions a 30-line script could answer.

Detekt and Lint catch bugs. Formatters catch shape. Indentation, line length, import order, trailing commas, wildcard imports, blank lines, semicolons (don't), spaces around operators. None of it is a bug; all of it is a debate; and a formatter ends every debate by simply picking an answer and applying it everywhere.

This chapter installs the two pieces you need. Ktlint — the de-facto Kotlin formatter, opinionated by design, configured by .editorconfig. And Spotless — the Gradle plugin that wraps ktlint (and a dozen other formatters) into one task, so a single ./gradlew spotlessApply formats your Kotlin, your Gradle scripts, your XML, your Markdown, and your license headers in one shot.

The numbers above aren't sourced from a study; they're what you'll see on your own team. Ask three reviewers how often they comment on indentation, import order, or trailing commas. The answer is "every PR, multiple times." That's hours per week, sprayed across senior calendars. Spotless eliminates the entire category in an afternoon.

ktlint

Pinterest's anti-bikeshedding Kotlin formatter, derived from the official Kotlin and Android style guides.

What it catches
Indentation, line length, import order, wildcard imports, trailing commas, blank lines, spacing, parameter wrapping, chain wrapping — the entire surface area of "what does Kotlin code look like."
Install
id("com.diffplug.spotless") version "6.25.0" in your root build.gradle.kts (we use ktlint through Spotless, not directly).
Runtime cost
Roughly 5–15 seconds for a medium Android repo on a clean check; sub-second for incremental formatting via the Gradle plugin or IDE.
When to skip it
Never. Even a one-person side project benefits — you stop having style fights with past-you. The only legitimate reason to skip is if your team is already on detekt-formatting (which uses ktlint internally anyway).

Why two tools, not one?

You might be wondering why this chapter is "ktlint and Spotless" and not just one. Short answer: ktlint is the engine, Spotless is the chassis. Ktlint formats Kotlin. Spotless wraps ktlint plus formatters for Java, Groovy, XML, JSON, YAML, Markdown, and adds license-header management, trailing-newline enforcement, and target-glob filters.

Could you use ktlint's official Gradle plugin (org.jlleitschuh.gradle.ktlint) directly? Yes, and it's a fine choice. We're recommending Spotless because the moment you want to also format your build.gradle.kts files, your XML resources, or strip trailing whitespace from Markdown docs, you need it anyway. Two formatters, one config, one task. Less surface to maintain.

Spotless approach
./gradlew spotlessApply Formats Kotlin (via ktlint), build scripts, XML, Markdown, license headers — all at once. One config block, one task.
Tool-per-format approach
./gradlew ktlintFormat ./gradlew xmlFormat ./gradlew prettierFormat ./gradlew licenseFormat Four plugins. Four config blocks. Four version pins to babysit.

The install — Spotless + ktlint

Two files change. The root build.gradle.kts applies the plugin and configures it; the .editorconfig at the repo root tells ktlint your house rules. Both go in source control.

build.gradle.kts (root)
plugins {
    id("com.diffplug.spotless") version "6.25.0"
}

spotless {
    // Format Kotlin source files in every module.
    kotlin {
        target("**/*.kt")
        targetExclude("**/build/**", "**/generated/**")
        ktlint("1.2.1")
            .editorConfigOverride(
                mapOf(
                    "ktlint_standard_filename" to "disabled",
                    "ktlint_standard_no-wildcard-imports" to "enabled"
                )
            )
        trimTrailingWhitespace()
        endWithNewline()
    }
    // Format Gradle Kotlin DSL scripts.
    kotlinGradle {
        target("*.gradle.kts", "**/*.gradle.kts")
        ktlint("1.2.1")
        trimTrailingWhitespace()
        endWithNewline()
    }
    // Format XML resources (layouts, drawables, values).
    format("xml") {
        target("**/src/**/*.xml")
        targetExclude("**/build/**")
        trimTrailingWhitespace()
        endWithNewline()
    }
}
Pin ktlint by exact version — ktlint releases occasionally change rules and you don't want a transitive bump rewriting your codebase on a Tuesday. The same goes for the Spotless plugin itself.

The companion file is .editorconfig at the repo root. Ktlint reads it; so does IntelliJ, Android Studio, VS Code, and every modern editor. One file to rule them all.

.editorconfig
root = true

[*]
charset = utf-8
end_of_line = lf
insert_final_newline = true
trim_trailing_whitespace = true
indent_style = space
indent_size = 4

[*.{kt,kts}]
max_line_length = 140
ij_kotlin_imports_layout = *,java.**,javax.**,kotlin.**,^
ij_kotlin_allow_trailing_comma = true
ij_kotlin_allow_trailing_comma_on_call_site = true
ktlint_standard_no-wildcard-imports = enabled
ktlint_standard_filename = disabled
ktlint_standard_function-naming = disabled

[*.{xml,yml,yaml,json}]
indent_size = 2

[*.md]
max_line_length = off
trim_trailing_whitespace = false
The ij_kotlin_* keys are read by IntelliJ and Android Studio so the IDE formats identically to ktlint. Disable filename if your project has classes-per-file conventions ktlint dislikes (test files, sealed-class hierarchies). Disable function-naming if you use @Composable functions named in PascalCase (Compose convention conflicts with the rule).

The "format-the-whole-repo-once" PR

You install Spotless. You run ./gradlew spotlessApply. git diff shows 12,000 changed lines. Welcome to the one-time tax.

Don't panic, and absolutely don't try to merge it with feature work. The right move is a dedicated format-the-whole-repo PR that does nothing else. No logic changes, no refactors, no renames. Just the formatter's diff. Reviewers don't read it line by line; they spot-check 5 files, see only whitespace and import-order changes, and approve.

Land it on a Friday afternoon. Merge conflicts on the format PR are nasty if there are 12 in-flight feature branches. Pick a low-traffic window, give a 30-minute heads-up in your team channel, merge it, then ask everyone to rebase on Monday morning. The pain is concentrated and over in a day. The alternative — a slow rolling format that touches every file someone edits — bleeds for months.

Once the big-bang PR is in, every subsequent PR has a clean baseline. New code is held to the standard. Old code is already there. The "we'll format as we touch it" strategy sounds appealing but produces years of half-formatted files where reviewers can't tell which lines are actually changed.

IDE integration — format on save

Spotless on the command line is fine. Spotless running on every Ctrl+S is the actual win. Two settings in Android Studio close the loop.

  1. Settings → Editor → Code Style → Kotlin → Set from… → Kotlin style guide. This aligns the IDE's built-in formatter with Kotlin's official style. Combined with .editorconfig in your repo, the IDE will pick up your project-specific overrides automatically.
  2. Settings → Tools → Actions on Save. Enable "Reformat code" and "Optimize imports" on save. Now every save reflows the file to ktlint-compatible formatting before you even think about it.

Bonus: install the official ktlint IntelliJ plugin (Ktlint by Nikky Bonnici on the marketplace). It surfaces ktlint warnings as inline highlights in the editor — the same warnings the build will fail on, shown in real time. The plugin reads your .editorconfig, so it stays in sync with the Spotless config.

One source of truth. If the IDE's formatter and ktlint disagree, you'll have a nightmarish ping-pong where save reformats one way and spotlessApply reformats it another. Always set the IDE to "Kotlin style guide" and let .editorconfig override the parts you care about. Don't customize formatting in the IDE settings — it's per-developer, won't be in source control, and your CI build doesn't see it.

The CI gate

Local format-on-save is for the conscientious. The CI gate is for everyone else. Add a single check job: ./gradlew spotlessCheck. If anything is unformatted, the build fails. The PR cannot merge. The fix is one command (./gradlew spotlessApply + commit), and it's clearly the developer's responsibility, not the reviewer's.

.github/workflows/format.yml
name: Format check

on:
  pull_request:
    branches: [ main, develop ]

jobs:
  spotless:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: 17

      - name: Setup Gradle
        uses: gradle/actions/setup-gradle@v3

      # Fast: only validates formatting, does not modify files.
      - name: Run spotlessCheck
        run: ./gradlew spotlessCheck --no-daemon

      # If it fails, the error message tells the dev to run
      # ./gradlew spotlessApply locally and commit.
Run spotlessCheck separately from your build job. It's fast (under a minute on most repos) and gives a clean, dedicated failure signal — "your code is unformatted" — instead of burying it among compilation errors.

Custom rules — the escape hatches

Ktlint is opinionated, which is the point. But occasionally a rule is wrong for your codebase. Three legitimate reasons to disable a rule, in order of frequency.

  • ktlint_standard_no-wildcard-imports — usually you keep this on, but if you have a heavy import androidx.compose.material3.* codebase, you might disable it. Or just live with the explicit imports; the IDE handles them.
  • ktlint_standard_function-naming — disable for Compose codebases. @Composable fun MyScreen() is PascalCase by convention; ktlint flags it as a violation.
  • ktlint_standard_filename — disable if you have files containing multiple top-level types (sealed-class hierarchies, paired data classes). Ktlint wants the filename to match the single public type.

Each disabled rule costs you something — a small piece of consistency. Don't accumulate disabled rules unconsciously. Every entry in .editorconfig is a deliberate choice. If you ever find yourself disabling more than 5 rules, ask whether you've configured around your team's actual conventions or around individual preferences that should adapt instead.

Don't ban @file:Suppress as a habit. It exists for a reason — sometimes a generated file or a mock has weird shape that ktlint can't reasonably know about. But if you're suppressing in normal code, you're either fighting a rule that should be globally disabled, or you're avoiding a refactor that the rule is correctly nudging you toward. Audit suppressions monthly. They tend to multiply.

AI-Assisted Setup — the /draft-editorconfig skill

The most boring part of installing ktlint is writing the .editorconfig. You don't know your team's conventions in formal terms; you know them as "yeah, that looks right." A short Claude Code skill bridges the gap: feed it five representative Kotlin files from your codebase, get back a draft .editorconfig that matches what you already do.

Skill: /draft-editorconfig
Read 5 representative .kt files from src/main/kotlin (pick from different packages: a ViewModel, a Repository, a data class, a Composable, and a util). Detect the prevailing conventions: - indent size and style (tabs vs spaces) - max line length (look at the longest non-string line) - import order (standard, by package prefix, etc.) - wildcard imports: present or absent - trailing comma usage on multi-line calls/declarations - function naming (camelCase, PascalCase for Composables, etc.) Produce a complete .editorconfig file at the repo root that locks in these conventions. Annotate any rule you disable with a brief comment explaining why (cite the file/line that justifies it). Flag any inconsistencies you found between files — those are spots where the team needs to make a decision before formatting.
Save this to .claude/commands/draft-editorconfig.md. Run it before the format-the-whole-repo-once PR. The output gives you a starting point and, more importantly, surfaces the inconsistencies in your codebase that the formatter is about to silently resolve.

One more skill worth standing up: /explain-spotless-failure. When CI fails on Spotless, the error log lists the files but not why. Have Claude diff the offending lines against the ktlint rule and explain in English what's wrong. Most often it's something the IDE would have caught if format-on-save were enabled.

Skill: /draft-editorconfig (don't do this)
Make me an .editorconfig for an Android project.
No grounding in your codebase. You'll get a generic Android template that may or may not match your conventions, and once you commit it the format-the-whole-repo PR will rewrite thousands of lines to match an opinion nobody on your team agreed to. Always feed the skill real files first; let it observe before it prescribes.

The format pipeline at a glance

By the end of this chapter you have four checkpoints, in order of when they fire. Each catches what the previous missed.

IDE save → Pre-commit → spotlessCheck → PR block

IDE save catches 95% of issues with zero developer effort. Pre-commit (Chapter 6) catches the remaining 4% before push. spotlessCheck in CI catches the last 1% — usually from contributors whose IDE isn't configured. The PR cannot merge until the check passes. There is no "I'll fix it later" path, because there is no path that bypasses the gate.

The pre-commit angle (teaser)

The CI gate is the safety net. Format-on-save is the convenience. The pre-commit hook is the layer between — runs spotlessApply on staged files only, in well under a second, before the commit ever lands. You'll wire that up in Chapter 6.

Day 1 Add the Spotless plugin to root build.gradle.kts. Drop a starter .editorconfig in the repo root. Run ./gradlew spotlessApply. Open the format-the-whole-repo PR. Merge it before any new feature lands on top.
Week 1 Get the team on format-on-save (Settings → Tools → Actions on Save). Install the ktlint IntelliJ plugin. Tune .editorconfig for the 2–3 rules that genuinely don't fit your codebase. Add a spotlessCheck CI job that warns on PRs.
Month 1 Promote the CI spotlessCheck job from warn to required — PRs cannot merge unformatted. Add a pre-commit hook (Chapter 6) that runs Spotless on staged files. Audit suppressions and disabled rules quarterly.

Common pitfalls

  • Floating ktlint version. Pinning to "+" or even a major version means a transitive update can rewrite your codebase overnight. Pin to a specific version (ktlint("1.2.1")) and bump it deliberately when you're ready for the diff.
  • IDE and ktlint disagree. If the IDE auto-format produces different output than ./gradlew spotlessApply, your .editorconfig is incomplete or the IDE's code style is overriding it. Fix the config; don't paper over it with project-level IDE settings that aren't checked in.
  • Suppressing whole packages. A targetExclude("**/legacy/**") sounds pragmatic but creates a permanent dark zone. Better: format it once, accept the diff, move on. Exclusions should be for generated code, not "code we don't feel like cleaning up."
  • License headers as moving targets. If you enable Spotless's licenseHeader feature, pick a year placeholder ($YEAR) and never touch it again. Hand-updating copyright years across 800 files every January is exactly the manual work the formatter is supposed to eliminate.
  • Forgetting the Gradle DSL files. If you only configure kotlin { } and skip kotlinGradle { }, your build.gradle.kts files drift in shape and someone eventually files a PR with their personal favorite indentation. Format both.

What you'll see in PRs after this

The biggest visible change is what stops happening. No more "fix the indentation" comments. No more "alphabetize the imports." No more debate over whether trailing commas are good (Kotlin says yes, ktlint says yes, the formatter applies it, conversation over).

The reviewers' attention finally turns to the part of the code that actually matters: does this logic work? Is this abstraction the right one? Did you forget a null check? Those are the questions that justify hiring senior engineers. Style is the noise that drowns them out.

The compounding payoff

Ktlint pays for itself the first week, in PR-review hours saved. Then it keeps paying every week, forever, with no maintenance. New hires learn the style by reading code and saving it. Old hires stop having opinions. Code review becomes about logic, not shape. This is what "leverage" means: a 50-line config block that gives you back senior engineering time, indefinitely.

Pre-flight checklist for the format-the-whole-repo PR

Before you click "merge" on the big-bang formatter PR, walk this list. Each item has burned at least one team that skipped it.

  • All in-flight feature branches have been notified to rebase Monday morning.
  • Ktlint version is pinned exactly (no +, no version range) in build.gradle.kts.
  • The PR contains only the formatter's diff — no rename, no logic change, no val-to-var tweaks.
  • .editorconfig is committed in the same PR as the format diff (so the next person who runs spotlessApply gets the same result).
  • CI runs spotlessCheck on the format PR and passes green.
  • The PR description explicitly tells reviewers: "do not line-by-line review; spot-check 3 files."
  • Merge window is low-traffic (Friday afternoon, end of sprint) and a 30-minute heads-up has gone to the team channel.
Your turn.
  1. Add the Spotless plugin to your root build.gradle.kts (use the recipe above).
  2. Drop the starter .editorconfig at your repo root. Don't tune it yet — defaults first.
  3. Run ./gradlew spotlessApply. Note how many files changed (git status | wc -l).
  4. Run ./gradlew spotlessCheck. Confirm it now passes.
  5. Open a feature branch and commit only the formatter's diff. Do not include any logic changes.
  6. Run the /draft-editorconfig skill against 5 representative files. Compare its output to your starter config. Adjust 1–2 rules if the AI's suggestions match your team's actual conventions better.
  7. Enable "Reformat code on save" in Android Studio. Confirm that saving a file with deliberate bad formatting auto-corrects.

Stretch goal: add the GitHub Actions workflow above and run it on the format PR. Watch spotlessCheck pass green for the first time. That's the moment the bikeshedding tax ends.

What's next

You now have three static analyzers (Lint, detekt) plus a formatter (Spotless+ktlint) running on every PR. That's the whole static-analysis pillar. But there's a problem: every check runs in CI, 7 minutes after push. By then you've context-switched. Chapter 6 moves the checks left — pre-commit hooks, pre-push hooks, save actions — so feedback comes in seconds instead of minutes. Same checks, same rules, just a much faster loop.
