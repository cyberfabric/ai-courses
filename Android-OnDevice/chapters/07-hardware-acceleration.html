Chapter 7

Hardware Acceleration — NPU, GPU, and DSP

Same model, same accuracy, 5-20x faster — unlock your phone's AI hardware

You've Been Using the Slowest Processor

Every model you've run so far — the MobileNet classifier in Chapter 3, the MediaPipe detectors in Chapter 4, the ML Kit features in Chapter 5 — has been running on the CPU. The CPU is a general-purpose workhorse, great for most code, but it's the worst choice for ML inference.

Your phone has at least three other processors that are much better at the matrix multiplications that models rely on. And you've been ignoring all of them.

Time to fix that.

The Hardware Zoo Inside Your Phone

A modern Android phone ships with four types of compute hardware, each with different strengths:

C
CPU — General Purpose
Good at everything, great at nothing (for ML). Handles complex branching logic well but processes tensor operations sequentially. 4-8 cores. Slowest for ML, but always available.
G
GPU — Massively Parallel
Hundreds of small cores designed for parallel math — exactly what matrix multiplication needs. Great for large models with many operations. 2-5x faster than CPU for most models. Available on every Android device.
N
NPU / TPU — Purpose-Built for AI
Dedicated silicon designed specifically for tensor operations. Runs quantized models at maximum efficiency with minimal power consumption. 5-20x faster than CPU. Found in 2022+ flagships and mid-range phones.
D
DSP — Digital Signal Processor
Optimized for signal processing: audio, sensor data, always-on tasks. Very power-efficient. Good for audio models (keyword spotting, speech recognition) that need to run continuously. Found in Qualcomm Snapdragon (Hexagon DSP).

Think of it like this: the CPU is a Swiss Army knife, the GPU is a chainsaw, and the NPU is a laser cutter. All can cut things — but for precise, repetitive cuts (tensor operations), the laser cutter wins every time.

The Performance Gap Is Huge

Here's what happens when you move the exact same model from CPU to GPU to NPU — no changes to accuracy, no changes to the model file:

TFLite GPU Delegate

The GPU delegate is the easiest acceleration to add and works on virtually every Android device. Two lines of code transform your Chapter 3 classifier:

dependencies {
    implementation("org.tensorflow:tensorflow-lite-gpu:2.16.1")
}

// Add GPU delegate to interpreter options
val gpuDelegate = GpuDelegate()
val options = Interpreter.Options().apply {
    addDelegate(gpuDelegate)
}
val interpreter = Interpreter(model, options)

That's it. Same model file, same preprocessing, same postprocessing. The interpreter now runs tensor operations on the GPU instead of the CPU.

How the GPU delegate works under the hood:

TFLite Model → GPU Delegate Analyzes Ops → Supported Ops → GPU → Unsupported Ops → CPU → Combined Result

The delegate inspects every operation in the model. Standard ops (conv2d, depthwise_conv, matmul, relu, softmax) go to the GPU. Uncommon ops that the GPU can't handle fall back to CPU automatically. You don't need to worry about compatibility — it just works.

GPU delegate works best with float32 models. Quantized int8 models are already optimized for CPU/NPU and may actually run slower on the GPU (which prefers float math). Use GPU delegate with your original float32 model, and save int8 quantization for CPU/NPU execution.

NNAPI Delegate — The Hardware Abstraction Layer

NNAPI (Neural Networks API) is Android's official hardware abstraction for ML. Instead of targeting a specific chip, you tell NNAPI "run this model on the best available hardware" and it figures out the rest:

val nnApiDelegate = NnApiDelegate(
    NnApiDelegate.Options().apply {
        // Let NNAPI choose the best accelerator
        setUseNnapiCpu(false)  // don't fall back to NNAPI's CPU impl
    }
)
val options = Interpreter.Options().apply {
    addDelegate(nnApiDelegate)
}
val interpreter = Interpreter(model, options)

NNAPI automatically routes operations to the NPU, GPU, or DSP based on what the device supports. On a Pixel phone, it hits the Tensor TPU. On a Galaxy, it targets the Hexagon NPU or Exynos NPU. On a budget phone without an NPU, it falls back to GPU or CPU.

NNAPI behavior varies wildly across devices. The same NNAPI call may route to a world-class NPU on a Pixel 9, a mediocre GPU driver on a mid-range Xiaomi, or fail entirely on an older device. Always benchmark on your target devices and have a fallback plan. Never assume NNAPI = fast.

Chipset-Specific Acceleration

Qualcomm QNN — Snapdragon Hexagon NPU

For maximum performance on Snapdragon devices (the most widely deployed mobile chipset), Qualcomm's QNN SDK provides direct NPU access bypassing NNAPI overhead:

dependencies {
    implementation("com.qualcomm.qti:qnn-runtime:2.25.0")
}

// QNN delegate targets the Hexagon NPU directly
val qnnDelegate = QnnDelegate(
    QnnDelegate.Options().apply {
        setSocModel(QnnDelegate.Options.SocModel.SM8650) // Snapdragon 8 Gen 3
        setBackendType(QnnDelegate.Options.BackendType.HTP) // Hexagon Tensor Processor
    }
)
val options = Interpreter.Options().apply {
    addDelegate(qnnDelegate)
}

Google Tensor TPU

Pixel phones access their Tensor TPU through NNAPI — no special SDK needed. The NNAPI driver on Pixel devices is highly optimized and routes compatible operations directly to the TPU.

Samsung Exynos NPU

Galaxy devices with Exynos chips access their NPU through NNAPI. Samsung also offers the ENN SDK for direct NPU access in specific use cases, but NNAPI is sufficient for most applications.

Building a Dynamic Delegate Selector

In production, you can't hardcode a delegate — your app runs on thousands of device configurations. Build a delegate selector that benchmarks at first run and caches the best choice:

class DelegateSelector(private val context: Context) {

    private val prefs = context.getSharedPreferences("ml_delegate", MODE_PRIVATE)

    fun getBestInterpreter(modelBuffer: MappedByteBuffer): Interpreter {
        val cached = prefs.getString("best_delegate", null)
        if (cached != null) {
            return createInterpreter(modelBuffer, cached)
        }
        // First run: benchmark each delegate
        val results = benchmarkDelegates(modelBuffer)
        val best = results.minByOrNull { it.value }!!.key
        prefs.edit().putString("best_delegate", best).apply()
        return createInterpreter(modelBuffer, best)
    }

    private fun benchmarkDelegates(model: MappedByteBuffer): Map<String, Long> {
        val results = mutableMapOf<String, Long>()
        val dummy = ByteBuffer.allocateDirect(224 * 224 * 3 * 4)

        // Try NNAPI
        tryDelegate("nnapi", model, dummy, results) {
            Interpreter.Options().apply { addDelegate(NnApiDelegate()) }
        }
        // Try GPU
        tryDelegate("gpu", model, dummy, results) {
            Interpreter.Options().apply { addDelegate(GpuDelegate()) }
        }
        // CPU baseline (always works)
        tryDelegate("cpu", model, dummy, results) {
            Interpreter.Options().apply { numThreads = 4 }
        }
        return results
    }

    private fun tryDelegate(
        name: String, model: MappedByteBuffer,
        input: ByteBuffer, results: MutableMap<String, Long>,
        optionsBuilder: () -> Interpreter.Options
    ) {
        try {
            val interp = Interpreter(model, optionsBuilder())
            val output = ByteBuffer.allocateDirect(1000 * 4)
            // Warm up
            repeat(3) { interp.run(input, output); input.rewind(); output.rewind() }
            // Measure
            val start = System.nanoTime()
            repeat(10) { interp.run(input, output); input.rewind(); output.rewind() }
            results[name] = (System.nanoTime() - start) / 10 / 1_000_000
            interp.close()
        } catch (e: Exception) {
            Log.w("DelegateSelector", "$name failed: ${e.message}")
        }
    }
}

The Fallback Chain Pattern

Always implement a fallback chain: NPU/TPU → GPU → CPU. If the NPU delegate fails (unsupported operations, driver bug), try GPU. If GPU fails, fall back to CPU. CPU is your safety net — it always works, on every device, for every model. Your app should never crash because a delegate failed.

Benchmarking Methodology

Measuring inference speed correctly is trickier than it looks. Here's how to get reliable numbers:

Warm-Up Runs

The first 3-5 inferences are always slow — the GPU compiles shaders, the NPU initializes its pipeline, memory is allocated. Always discard warm-up runs before measuring:

// Warm up (discard results)
repeat(5) { interpreter.run(input, output) }

// Measure (real results)
val times = mutableListOf<Long>()
repeat(100) {
    val start = System.nanoTime()
    interpreter.run(input, output)
    times.add((System.nanoTime() - start) / 1_000_000)
}

// Report percentiles, not averages
times.sort()
val p50 = times[times.size / 2]
val p95 = times[(times.size * 0.95).toInt()]
val p99 = times[(times.size * 0.99).toInt()]
Log.d("Benchmark", "p50=${p50}ms, p95=${p95}ms, p99=${p99}ms")

Why Percentiles, Not Averages

Averages hide the truth. If 99 inferences take 3ms and one takes 200ms (thermal throttle, GC pause), the average says "5ms" but the user saw a 200ms stutter. Report p50 (typical), p95 (occasional), and p99 (worst case).

Thermal Throttling

Run inference continuously for 60 seconds and watch what happens. Most devices throttle after 10-20 seconds of sustained NPU/GPU usage, and inference times increase 20-50%. Your production benchmarks should measure sustained performance, not just burst speed.

Reliable benchmarking

5 warm-up runs, then 100 measured runs Report p50, p95, p99 percentiles Test sustained performance (60+ seconds) Test on 3+ device tiers (flagship, mid, budget) Measure memory consumption alongside latency

Misleading benchmarking

Run once and report the number Report only the average Test only on your flagship dev phone Measure only cold-start (first inference) Ignore memory usage

Handling Delegate Failures Gracefully

Delegates can fail for many reasons: unsupported operations, driver bugs, insufficient memory, or missing hardware. Your app must handle every failure gracefully:

fun createSafeInterpreter(model: MappedByteBuffer): Interpreter {
    // Try NNAPI first
    try {
        val options = Interpreter.Options().apply {
            addDelegate(NnApiDelegate())
        }
        return Interpreter(model, options).also {
            Log.i("ML", "Using NNAPI delegate")
        }
    } catch (e: Exception) {
        Log.w("ML", "NNAPI failed: ${e.message}")
    }

    // Try GPU
    try {
        val options = Interpreter.Options().apply {
            addDelegate(GpuDelegate())
        }
        return Interpreter(model, options).also {
            Log.i("ML", "Using GPU delegate")
        }
    } catch (e: Exception) {
        Log.w("ML", "GPU failed: ${e.message}")
    }

    // CPU fallback (always works)
    Log.i("ML", "Using CPU fallback")
    return Interpreter(model, Interpreter.Options().apply {
        numThreads = 4
    })
}
Never let a delegate failure crash your app. Wrap every delegate initialization in a try/catch. Log which delegate succeeded — this telemetry is invaluable for understanding your real-world device mix. In production, report the delegate used alongside your performance metrics.

Take the image classifier from Chapter 3 and measure the real performance impact of hardware acceleration.

  1. Add the GPU delegate dependency to your project

  3. Run each one with 5 warm-up inferences followed by 50 measured inferences
  4. Record for each delegate:

Expected results: GPU should be 2-3x faster than CPU. NNAPI should be 3-8x faster on flagship devices, but may fail on budget phones. CPU should always work as the reliable fallback.

What's Next

Hardware acceleration makes your models run faster on the available hardware. But what if the model itself is too big? A 100MB float32 model is too heavy for mobile no matter how fast your NPU is — it takes too much RAM, too much storage, and too long to load.

In Chapter 8: Model Optimization — Making Models Mobile-Ready, we'll learn to shrink models: quantization (32-bit → 8-bit, 4x smaller), pruning (remove unnecessary weights), and distillation (train a smaller model to mimic a big one). Combine optimization with hardware acceleration and you get models that are both tiny and blazing fast.
