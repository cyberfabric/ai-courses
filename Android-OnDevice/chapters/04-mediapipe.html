Chapter 4

MediaPipe — Production-Ready AI Features

Face detection, hand tracking, pose estimation, and object detection — all in an afternoon

From Raw TFLite to Batteries Included

In Chapter 3 you built an image classifier from scratch — loading a model, writing preprocessing code, wiring up CameraX, handling threading. It worked, but it was a lot of plumbing for one feature.

MediaPipe is the shortcut. It's Google's cross-platform framework that bundles optimized ML models with all the preprocessing, postprocessing, and threading logic built in. What took you an entire chapter with raw TFLite takes about 30 lines with MediaPipe.

Think of the difference like this: raw TFLite is cooking from scratch (you buy ingredients, chop, season, cook). MediaPipe is a meal kit (pre-portioned, pre-prepped, just assemble and heat). Same quality result, a fraction of the work.

What MediaPipe Gives You

MediaPipe's Tasks API provides high-level classes for the most common ML features. Each one handles model loading, preprocessing, inference, and postprocessing internally:

6+ Vision tasks (detection, tracking, segmentation)
3 Model variants per task (lite, full, heavy)
~30 Lines of code for a complete feature

The key tasks you'll use most:

  • ObjectDetector — detects objects with bounding boxes and labels
  • FaceLandmarker — tracks 468 face landmarks in real time (eyes, nose, lips, jawline)
  • HandLandmarker — tracks 21 landmarks per hand (fingertips, knuckles, wrist)
  • PoseLandmarker — tracks 33 body landmarks (shoulders, elbows, knees, ankles)
  • ImageClassifier — classifies images (like Chapter 3, but simpler API)
  • GestureRecognizer — detects hand gestures (thumbs up, peace, pointing, etc.)

Setting Up MediaPipe

Add the MediaPipe Tasks Vision dependency to your build.gradle.kts:

dependencies {
    implementation("com.google.mediapipe:tasks-vision:0.20250101")
}

That single dependency includes all vision tasks. No separate model downloads for setup — but you do need to download the specific .task model bundle file for each feature you use and place it in assets/.

Building a Real-Time Object Detector

Let's build something more useful than a classifier — an object detector that draws bounding boxes around everything it sees. This is what camera apps use to highlight faces, what warehouse apps use to scan shelves, and what accessibility apps use to describe scenes.

MediaPipe object detection pipeline — what happens under the hood:

Camera Frame → MPImage Wrapper → ObjectDetector → Bounding Boxes + Labels → Canvas Overlay

MediaPipe handles all the preprocessing (resize, normalize, pad) and postprocessing (NMS, score filtering) internally. You just feed it an image and read the results.

Step 1: Create the Detector

Download the efficientdet_lite0.task model bundle (~4.4MB) from the MediaPipe website and place it in assets/. Then initialize the detector:

val baseOptions = BaseOptions.builder()
    .setModelAssetPath("efficientdet_lite0.task")
    .build()

val options = ObjectDetector.ObjectDetectorOptions.builder()
    .setBaseOptions(baseOptions)
    .setRunningMode(RunningMode.LIVE_STREAM)
    .setMaxResults(5)
    .setScoreThreshold(0.5f)
    .setResultListener { result, inputImage ->
        // Called on every frame with detection results
        handleDetections(result)
    }
    .build()

val detector = ObjectDetector.createFromOptions(context, options)

That's your entire detector. No Interpreter, no ImageProcessor, no TensorBuffer. MediaPipe handles it all.

Step 2: Feed Camera Frames

analysis.setAnalyzer(Dispatchers.Default.asExecutor()) { proxy ->
    val mpImage = BitmapImageBuilder(proxy.toBitmap()).build()
    val timestampMs = proxy.imageInfo.timestamp / 1000
    detector.detectAsync(mpImage, timestampMs)
    proxy.close()
}

Step 3: Draw Bounding Boxes on Compose Canvas

@Composable
fun DetectionOverlay(
    detections: List<Detection>,
    imageWidth: Int,
    imageHeight: Int
) {
    Canvas(modifier = Modifier.fillMaxSize()) {
        val scaleX = size.width / imageWidth
        val scaleY = size.height / imageHeight

        detections.forEach { detection ->
            val box = detection.boundingBox()
            val rect = Rect(
                box.left * scaleX, box.top * scaleY,
                box.right * scaleX, box.bottom * scaleY
            )
            // Draw bounding box
            drawRect(Color(0xFFFBBF24), rect.topLeft, rect.size,
                style = Stroke(3.dp.toPx()))

            // Draw label
            val label = detection.categories().first()
            val text = "${label.categoryName()} ${(label.score() * 100).toInt()}%"
            drawContext.canvas.nativeCanvas.drawText(
                text, rect.left, rect.top - 8f, labelPaint
            )
        }
    }
}
Coordinate scaling is essential. The model runs on a smaller input image (320×320), but your camera preview fills the screen. You must scale the bounding box coordinates from model space to screen space, or your boxes will be in the wrong position.

Face Mesh — 468 Landmarks in Real Time

MediaPipe's FaceLandmarker tracks 468 points on a face — every contour of the eyes, nose, lips, eyebrows, and jawline. This powers features like AR filters (Snapchat-style lenses), face-controlled UI, driver drowsiness detection, and emoji puppeting.

val faceLandmarker = FaceLandmarker.createFromOptions(context,
    FaceLandmarker.FaceLandmarkerOptions.builder()
        .setBaseOptions(BaseOptions.builder()
            .setModelAssetPath("face_landmarker.task")
            .build())
        .setRunningMode(RunningMode.LIVE_STREAM)
        .setNumFaces(1)
        .setResultListener { result, _ ->
            val landmarks = result.faceLandmarks()
            if (landmarks.isNotEmpty()) {
                val face = landmarks[0]  // 468 NormalizedLandmark points
                val noseTip = face[1]    // index 1 = nose tip
                val leftEye = face[33]   // index 33 = left eye center
                val rightEye = face[263] // index 263 = right eye center
                updateFaceOverlay(face)
            }
        }
        .build()
)

Each landmark is a NormalizedLandmark with x, y, and z coordinates normalized to [0, 1]. Multiply by your view dimensions to get pixel positions.

The z coordinate represents depth relative to the face center. Negative z means closer to the camera, positive means farther away. This lets you detect head rotation (turning left/right) and build 3D face effects.

Hand Tracking — 21 Landmarks Per Hand

The HandLandmarker detects 21 points on each hand: wrist, every knuckle, and every fingertip. This is the foundation for gesture recognition — mapping hand poses to app actions.

val handLandmarker = HandLandmarker.createFromOptions(context,
    HandLandmarker.HandLandmarkerOptions.builder()
        .setBaseOptions(BaseOptions.builder()
            .setModelAssetPath("hand_landmarker.task")
            .build())
        .setRunningMode(RunningMode.LIVE_STREAM)
        .setNumHands(2)
        .setResultListener { result, _ ->
            result.landmarks().forEachIndexed { i, hand ->
                val thumbTip = hand[4]    // thumb tip
                val indexTip = hand[8]    // index finger tip
                val middleTip = hand[12]  // middle finger tip
                val wrist = hand[0]       // wrist
                detectGesture(hand)
            }
        }
        .build()
)

Simple Gesture Recognition

You can recognize gestures by checking finger positions relative to each other. Here's a simple approach that works surprisingly well:

fun detectGesture(landmarks: List<NormalizedLandmark>): String {
    val thumbTip = landmarks[4]
    val indexTip = landmarks[8]
    val middleTip = landmarks[12]
    val ringTip = landmarks[16]
    val pinkyTip = landmarks[20]
    val wrist = landmarks[0]

    // Fingers are "up" if their tip is above their base
    val indexUp = indexTip.y() < landmarks[6].y()
    val middleUp = middleTip.y() < landmarks[10].y()
    val ringUp = ringTip.y() < landmarks[14].y()
    val pinkyUp = pinkyTip.y() < landmarks[18].y()

    return when {
        indexUp && middleUp && !ringUp && !pinkyUp -> "peace"
        !indexUp && !middleUp && !ringUp && !pinkyUp -> "fist"
        indexUp && middleUp && ringUp && pinkyUp -> "open_palm"
        else -> "unknown"
    }
}

Good gesture-to-action mapping

Open palm → Pause/play video Peace sign → Take screenshot Thumbs up → Like / approve Fist → Dismiss notification

Gestures are distinct and hard to trigger accidentally. Each maps to a single clear action.

Risky gesture-to-action mapping

Slight finger curl → Delete file Two-finger tap → Send message Wrist rotate → Undo purchase Any hand visible → Start recording

Ambiguous gestures lead to false positives. Destructive actions should never be triggered by gestures alone.

Pose Estimation — Full-Body Tracking

The PoseLandmarker tracks 33 body landmarks — from head to toes. This powers fitness apps (count reps, check form), physical therapy (measure range of motion), and gesture-controlled games.

val poseLandmarker = PoseLandmarker.createFromOptions(context,
    PoseLandmarker.PoseLandmarkerOptions.builder()
        .setBaseOptions(BaseOptions.builder()
            .setModelAssetPath("pose_landmarker_lite.task")
            .build())
        .setRunningMode(RunningMode.LIVE_STREAM)
        .setResultListener { result, _ ->
            val pose = result.landmarks().firstOrNull() ?: return@setResultListener
            val leftShoulder = pose[11]
            val leftElbow = pose[13]
            val leftWrist = pose[15]
            val angle = calculateAngle(leftShoulder, leftElbow, leftWrist)
            // angle > 160° = arm extended, angle < 90° = arm bent
        }
        .build()
)

A classic use case — counting bicep curls by tracking the elbow angle:

fun calculateAngle(a: NormalizedLandmark, b: NormalizedLandmark,
                    c: NormalizedLandmark): Double {
    val radians = atan2(
        (c.y() - b.y()).toDouble(), (c.x() - b.x()).toDouble()
    ) - atan2(
        (a.y() - b.y()).toDouble(), (a.x() - b.x()).toDouble()
    )
    var degrees = Math.toDegrees(abs(radians))
    if (degrees > 180) degrees = 360 - degrees
    return degrees
}

MediaPipe Configuration Deep Dive

Running Modes

Every MediaPipe task supports three running modes. Pick the right one for your use case:

Mode Use When Behavior
IMAGE Processing a single photo Synchronous. Call detect(image), get result immediately. Blocks the calling thread.
VIDEO Processing recorded video frames Synchronous with timestamps. Call detectForVideo(image, timestampMs). Frames must be in order.
LIVE_STREAM Real-time camera processing Asynchronous. Call detectAsync(image, timestampMs). Results arrive via callback. Drops frames if busy.
Always use LIVE_STREAM for camera feeds. The IMAGE and VIDEO modes are synchronous — they block until inference completes. With a live camera, this creates backpressure, dropped frames, and UI stutter. LIVE_STREAM automatically manages the frame queue and delivers results via a non-blocking callback.

Model Variants: Lite vs Full vs Heavy

Most MediaPipe tasks offer three model variants. Choose based on your accuracy vs. speed needs:

MediaPipe hand tracking — model variants on a Pixel 8 Pro:

Variant Model Size Inference Time Landmark Accuracy Best For
Lite 2.3 MB ~5 ms Good Real-time, budget devices
Full 3.8 MB ~8 ms Better Balanced — default choice
Heavy 5.6 MB ~15 ms Best Precision-critical (AR, medical)

Start with Full for development. Switch to Lite if you need faster inference on lower-end devices. Use Heavy only when landmark precision is critical (AR overlays that must perfectly align with facial features).

Combining Multiple MediaPipe Tasks

What if you want face detection and hand tracking running simultaneously? You can create multiple task instances, but be mindful of resource usage:

// Both tasks share the camera feed but run independently
val faceDetector = FaceLandmarker.createFromOptions(context, faceOptions)
val handDetector = HandLandmarker.createFromOptions(context, handOptions)

analysis.setAnalyzer(executor) { proxy ->
    val mpImage = BitmapImageBuilder(proxy.toBitmap()).build()
    val timestamp = proxy.imageInfo.timestamp / 1000

    // Both run asynchronously — results arrive via their callbacks
    faceDetector.detectAsync(mpImage, timestamp)
    handDetector.detectAsync(mpImage, timestamp)
    proxy.close()
}

Memory impact of running multiple tasks simultaneously:

  • 1 task — ~30-50 MB additional RAM. Fine on any device with 3GB+ total RAM.
  • 2 tasks — ~60-90 MB. Comfortable on devices with 4GB+ RAM.
  • 3+ tasks — ~100-150 MB. Flagship territory (6GB+). Watch for onTrimMemory callbacks and consider unloading idle tasks.

On budget devices (2-3 GB RAM), run only one task at a time and switch between them based on what the user needs.

MediaPipe vs. Raw TFLite

When should you use MediaPipe, and when should you drop down to raw TFLite like in Chapter 3?

Criteria MediaPipe Raw TFLite
Setup time 30 minutes Half a day
Model choice Pre-selected, optimized Any TFLite model
Preprocessing Built in You write it
Threading Managed for you You manage it
Custom models No — use their models only Yes — any model you want
Fine-tuning Limited configuration Full control

The Decision Rule

Start with MediaPipe. If one of its pre-built tasks does what you need, use it. You'll ship faster and the results will be better than most custom implementations. Drop down to raw TFLite only when you need a custom model that MediaPipe doesn't support — like a model trained on your own data, or a specialized architecture not in MediaPipe's catalog.

Performance Tuning Tips

  • Use GPU delegate for sustained inference — add .setDelegate(Delegate.GPU) to BaseOptions. Reduces CPU load and heat.
  • Lower setMaxResults — if you only need the top detection, don't ask for 10.
  • Raise setScoreThreshold — filtering low-confidence results early saves postprocessing work.
  • Skip frames on slow devices — process every 2nd or 3rd camera frame instead of every one.
  • Close tasks when done — call detector.close() when leaving the screen to free GPU/NPU resources.

Try It: Build a Hand Gesture Controller

Build a hand gesture recognizer that maps 3 gestures to 3 app actions.

  1. Set up MediaPipe HandLandmarker with the hand_landmarker.task model
  2. Connect it to CameraX using LIVE_STREAM mode
  3. Implement gesture detection for three poses:
    • Open palm (all fingers extended) → show "Pause" on screen
    • Peace sign (index + middle up) → show "Screenshot" on screen
    • Fist (all fingers closed) → show "Dismiss" on screen
  4. Display the detected gesture name and the 21 hand landmarks drawn on a Compose Canvas overlay

Bonus: Try it on two hands simultaneously — does accuracy change? How about in different lighting conditions (bright room vs. dim room)?
