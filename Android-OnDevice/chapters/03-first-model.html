Chapter 3

Your First On-Device Model

Build a real-time image classifier with TFLite and CameraX from scratch

Time to Stop Reading, Start Building

In Chapter 2 you learned the theory: models, tensors, preprocessing, postprocessing. Now we put it all together. By the end of this chapter, you'll have an app on your phone that points the camera at objects and tells you what they are — in real time, running entirely on-device.

We'll use MobileNetV3 for classification, TensorFlow Lite for inference, CameraX for camera frames, and Jetpack Compose for the UI. The full inference loop from Chapter 2 — load, preprocess, infer, postprocess, display — implemented end-to-end.

What we're building — the complete image classification pipeline:

CameraX Frame → Resize 224×224 → Normalize [0, 1] → TFLite Inference → Top-5 Labels → Compose UI

Step 1: Set Up the Project

Create a new Android project in Android Studio (Empty Compose Activity). Then add the TensorFlow Lite and CameraX dependencies to your module-level build.gradle.kts:

dependencies {
    // TensorFlow Lite — core runtime
    implementation("org.tensorflow:tensorflow-lite:2.16.1")
    // TFLite Support Library — image helpers, label loading
    implementation("org.tensorflow:tensorflow-lite-support:0.4.4")
    // TFLite Metadata — read model metadata
    implementation("org.tensorflow:tensorflow-lite-metadata:0.4.4")

    // CameraX — camera integration
    val cameraVersion = "1.4.1"
    implementation("androidx.camera:camera-camera2:$cameraVersion")
    implementation("androidx.camera:camera-lifecycle:$cameraVersion")
    implementation("androidx.camera:camera-view:$cameraVersion")
}

And in the android block, tell the build system not to compress the model file:

android {
    // ...
    androidResources {
        noCompress += "tflite"
    }
}
Why noCompress? TFLite can memory-map model files directly from the APK — this avoids copying the entire model into RAM. But it only works if the file isn't compressed. Without this setting, Android would GZIP the .tflite file during build, and loading would require decompressing the entire model into memory first.

Step 2: Add the Model to Your Project

Download MobileNetV3-Large from TensorFlow Hub or Kaggle Models. You'll get a file called something like mobilenet_v3_large_100_224.tflite (about 5.4MB). This model classifies images into 1,000 ImageNet categories.

Place the model file and its labels file in your app's assets:

app/
  src/
    main/
      assets/
        mobilenet_v3.tflite    <-- the model (5.4 MB)
        labels.txt             <-- 1,000 class names, one per line

The labels file has one class name per line — "tench", "goldfish", "great_white_shark", ... all the way to "toilet_tissue" at line 1,000. The line number corresponds to the model's output index.

Step 3: Load the Model

Loading a TFLite model is a two-step process: read the file into a memory-mapped buffer, then create an Interpreter:

class ImageClassifier(private val context: Context) {

    private val interpreter: Interpreter
    private val labels: List<String>

    init {
        // Memory-map the model file from assets
        val model = FileUtil.loadMappedFile(context, "mobilenet_v3.tflite")

        // Configure the interpreter
        val options = Interpreter.Options().apply {
            numThreads = 4  // use multiple CPU cores
        }
        interpreter = Interpreter(model, options)

        // Load class labels
        labels = FileUtil.loadLabels(context, "labels.txt")
    }
}
Common crash: "Model file not found." Make sure the file is in src/main/assets/ (not res/). The name must match exactly — mobilenet_v3.tflite, not MobileNet_v3.tflite. Android assets are case-sensitive on most devices.

Step 4: Build the Preprocessing Pipeline

Remember from Chapter 2: the model expects a [1, 224, 224, 3] float tensor with values between 0.0 and 1.0. Your camera gives you a full-resolution Bitmap. TFLite's support library makes the conversion easy:

private val imageProcessor = ImageProcessor.Builder()
    .add(ResizeOp(224, 224, ResizeOp.ResizeMethod.BILINEAR))
    .add(NormalizeOp(0.0f, 255.0f))  // 0-255 → 0.0-1.0
    .build()

fun classify(bitmap: Bitmap): List<Classification> {

Step 5: Run Inference and Read Results

Now for the exciting part — actually asking the model "what is this?":

fun classify(bitmap: Bitmap): List<Classification> {
    val tensorImage = TensorImage(DataType.FLOAT32)
    tensorImage.load(bitmap)
    val processed = imageProcessor.process(tensorImage)

    // Prepare output buffer — 1000 classes
    val output = TensorBuffer.createFixedSize(
        intArrayOf(1, 1000), DataType.FLOAT32
    )

    // Run inference
    interpreter.run(processed.buffer, output.buffer.rewind())

    // Find top-5 predictions
    val probabilities = output.floatArray
    return probabilities
        .mapIndexed { index, score -> Classification(labels[index], score) }
        .sortedByDescending { it.score }
        .take(5)
}

data class Classification(val label: String, val score: Float)

That's the complete inference code. Load the image, preprocess it, run the model, sort the output. Five lines of real work.

Step 6: Connect to CameraX

To classify objects in real time, you need a continuous stream of camera frames. CameraX's ImageAnalysis use case gives you exactly that:

@Composable
fun CameraClassifierScreen(classifier: ImageClassifier) {
    val results = remember { mutableStateOf(emptyList<Classification>()) }
    val context = LocalContext.current
    val lifecycleOwner = LocalLifecycleOwner.current

    // Camera preview reference
    val previewView = remember { PreviewView(context) }

    LaunchedEffect(Unit) {
        val cameraProvider = ProcessCameraProvider.awaitInstance(context)

        val preview = Preview.Builder().build().also {
            it.surfaceProvider = previewView.surfaceProvider
        }

        val analysis = ImageAnalysis.Builder()
            .setBackpressureStrategy(STRATEGY_KEEP_ONLY_LATEST)
            .build()

        analysis.setAnalyzer(Dispatchers.Default.asExecutor()) { proxy ->
            val bitmap = proxy.toBitmap()
            results.value = classifier.classify(bitmap)
            proxy.close()
        }

        cameraProvider.bindToLifecycle(
            lifecycleOwner, CameraSelector.DEFAULT_BACK_CAMERA,
            preview, analysis
        )
    }

    // UI: camera preview + results overlay
    Box(modifier = Modifier.fillMaxSize()) {
        AndroidView({ previewView }, Modifier.fillMaxSize())
        ClassificationOverlay(results.value)
    }
}
STRATEGY_KEEP_ONLY_LATEST is critical. This tells CameraX: "If I'm still processing the previous frame, drop new frames instead of queuing them." Without this, frames queue up faster than you can process them, memory balloons, and your app crashes. Always use this strategy for real-time ML.

Step 7: Display Results in Compose

The overlay composable shows the top predictions with confidence bars:

@Composable
fun ClassificationOverlay(results: List<Classification>) {
    Column(
        modifier = Modifier
            .fillMaxWidth()
            .padding(16.dp)
            .background(Color.Black.copy(alpha = 0.6f), RoundedCornerShape(12.dp))
            .padding(16.dp)
    ) {
        results.forEach { result ->
            Row(
                modifier = Modifier.fillMaxWidth(),
                horizontalArrangement = Arrangement.SpaceBetween
            ) {
                Text(
                    text = result.label.replace("_", " "),
                    color = Color.White,
                    fontSize = 14.sp
                )
                Text(
                    text = "${(result.score * 100).toInt()}%",
                    color = Color(0xFFFBBF24),
                    fontWeight = FontWeight.Bold
                )
            }
            Spacer(Modifier.height(4.dp))
        }
    }
}

Threading Strategy — Never Block the UI

Notice that our ImageAnalysis analyzer runs on Dispatchers.Default — a background thread pool. This is essential. Model inference takes 6-100ms depending on the model and hardware. Running that on the main thread means 6-100ms of frozen UI per frame — your app would stutter horribly and eventually ANR.

The pattern is simple:

  • Camera frames arrive on the CameraX thread
  • Inference runs on Dispatchers.Default (background)
  • Results update a mutableStateOf (Compose recomposes on the main thread automatically)

Correct threading

// Analyzer runs on background executor analysis.setAnalyzer( Dispatchers.Default.asExecutor() ) { proxy -> results.value = classifier.classify(proxy.toBitmap()) proxy.close() }

Inference on background thread. UI updates via Compose state. Smooth 60fps UI.

ANR waiting to happen

// DON'T: Analyzer on main thread analysis.setAnalyzer( Executors.mainThreadExecutor() ) { proxy -> results.value = classifier.classify(proxy.toBitmap()) proxy.close() }

Inference blocks the UI thread. Every frame causes 50-100ms jank. ANR dialog after 5 seconds.

Measuring Performance — Your CPU Baseline

Before optimizing (that's Chapter 7), you need to know where you stand. Add simple timing to your classifier:

fun classify(bitmap: Bitmap): Pair<List<Classification>, Long> {
    val startTime = System.nanoTime()

    // ... preprocessing and inference (same as before) ...

    val inferenceTimeMs = (System.nanoTime() - startTime) / 1_000_000
    Log.d("Classifier", "Inference: ${inferenceTimeMs}ms")

    return Pair(results, inferenceTimeMs)
}

MobileNetV3-Large CPU baseline — what to expect on first run:

Device Chipset CPU Inference Frames/sec
Pixel 8 Pro Tensor G3 ~18 ms ~55 fps
Samsung Galaxy S24 Snapdragon 8 Gen 3 ~15 ms ~66 fps
Pixel 7a Tensor G2 ~25 ms ~40 fps
Samsung Galaxy A54 Exynos 1380 ~55 ms ~18 fps
Redmi Note 12 Snapdragon 4 Gen 1 ~80 ms ~12 fps

These are CPU-only numbers (no GPU or NPU acceleration). In Chapter 7, we'll add hardware acceleration and see these drop by 3-10x. But even CPU-only, flagships can classify faster than the camera produces frames.

First-inference warm-up: The very first inference after loading the model is always 2-5x slower than subsequent ones. This is because TFLite performs internal optimizations (memory allocation, operation scheduling) on the first run. Always run 2-3 warm-up inferences before measuring performance or showing results to users.

// Warm up the model after loading
val dummyBitmap = Bitmap.createBitmap(224, 224, Bitmap.Config.ARGB_8888)
repeat(3) { classifier.classify(dummyBitmap) }  // discard results

Common Pitfalls and Fixes

Every Android developer hits these the first time. Here's the cheat sheet:

Symptom Cause Fix
Crash: "Cannot load model" Wrong file path or missing from assets Check src/main/assets/mobilenet_v3.tflite exists. Exact case.
Crash: "Input tensor shape mismatch" Feeding raw camera resolution instead of 224×224 Always resize to the model's expected input shape first.
Garbage output: every image is "digital clock" Forgot to normalize pixels (model gets 0-255 instead of 0.0-1.0) Add NormalizeOp(0.0f, 255.0f) to the image processor.
ANR dialog after a few seconds Running inference on the main thread Use Dispatchers.Default.asExecutor() for the analyzer.
Camera image rotated 90° ImageProxy has rotation metadata you're ignoring Apply proxy.imageInfo.rotationDegrees to the bitmap before classifying.
OutOfMemoryError Creating new Bitmap objects every frame without recycling Reuse a single Bitmap, or let the GC handle it with STRATEGY_KEEP_ONLY_LATEST.

Handling Image Rotation

Camera sensors are often physically mounted sideways. The ImageProxy gives you rotation metadata that you need to apply:

analysis.setAnalyzer(Dispatchers.Default.asExecutor()) { proxy ->
    val bitmap = proxy.toBitmap()

    // Apply rotation if needed
    val rotated = if (proxy.imageInfo.rotationDegrees != 0) {
        val matrix = Matrix().apply {
            postRotate(proxy.imageInfo.rotationDegrees.toFloat())
        }
        Bitmap.createBitmap(bitmap, 0, 0,
            bitmap.width, bitmap.height, matrix, true)
    } else bitmap

    results.value = classifier.classify(rotated)
    proxy.close()
}
For classification (not detection), rotation usually doesn't matter much — MobileNet handles rotated objects reasonably well. But for object detection and segmentation in later chapters, correct rotation is essential for accurate bounding boxes.

The Complete Classifier Class

Here's the full classifier assembled from all the pieces above, ready to drop into your project:

class ImageClassifier(context: Context) {

    private val interpreter: Interpreter
    private val labels: List<String>
    private val imageProcessor = ImageProcessor.Builder()
        .add(ResizeOp(224, 224, ResizeOp.ResizeMethod.BILINEAR))
        .add(NormalizeOp(0.0f, 255.0f))
        .build()

    init {
        val model = FileUtil.loadMappedFile(context, "mobilenet_v3.tflite")
        val options = Interpreter.Options().apply { numThreads = 4 }
        interpreter = Interpreter(model, options)
        labels = FileUtil.loadLabels(context, "labels.txt")

        // Warm up
        val dummy = Bitmap.createBitmap(224, 224, Bitmap.Config.ARGB_8888)
        classify(dummy)
    }

    fun classify(bitmap: Bitmap): List<Classification> {
        val tensorImage = TensorImage(DataType.FLOAT32)
        tensorImage.load(bitmap)
        val processed = imageProcessor.process(tensorImage)

        val output = TensorBuffer.createFixedSize(
            intArrayOf(1, 1000), DataType.FLOAT32
        )
        interpreter.run(processed.buffer, output.buffer.rewind())

        return output.floatArray
            .mapIndexed { i, score -> Classification(labels[i], score) }
            .sortedByDescending { it.score }
            .take(5)
    }

    fun close() = interpreter.close()
}

data class Classification(val label: String, val score: Float)

Try It: Build Your Image Classifier

Build a working image classifier and test it on real objects.

  1. Create a new Compose project and add the TFLite + CameraX dependencies
  2. Download MobileNetV3-Large .tflite and the ImageNet labels.txt into assets/
  3. Implement the ImageClassifier class from this chapter
  4. Add CameraX integration with ImageAnalysis
  5. Display top-5 results overlaid on the camera preview
  6. Add inference time logging

Then test it: Photograph 10 different objects around your desk or home. For each:

  • Record the top prediction and confidence score
  • Mark whether it got it right or wrong
  • Note the inference time on your device

Expected outcome: MobileNetV3 should correctly identify 6-8 out of 10 common objects (keyboards, monitors, cups, bottles). It struggles with unusual angles, partial views, and objects not in ImageNet's 1,000 classes. That's normal — and it's why we'll explore specialized models in the next chapters.
