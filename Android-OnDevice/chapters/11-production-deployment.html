Chapter 11

Model Management and Production Deployment

Ship on-device AI to millions of devices — with confidence

The Production Gap

"It works on my Pixel" is not a shipping strategy. The feature you built and tested on your flagship dev phone will run on 24,000+ distinct Android device configurations — different chipsets, different RAM amounts, different Android versions, with or without NPUs.

24,000+ Distinct Android device models
8+ Major chipset families
512MB-16GB RAM range across active devices

This chapter is about bridging the gap between "cool demo" and "shipped feature." We'll cover model packaging, OTA updates, A/B testing, error handling, monitoring, and graceful degradation — everything you need to ship with confidence.

Model Packaging Strategies

Your first decision: how does the model get onto the user's device?

Strategy APK Impact First-Use Experience Updates Best For
Bundled in APK +model size Instant — works immediately Requires app update Small models (< 10MB)
Download on first use No impact Delayed — download required Download new version Large models (10-100MB)
Firebase ML hosting No impact Delayed — download required OTA — no app update Models that improve over time
Play Asset Delivery Separate asset pack Installed with app or on-demand Via Play Store Very large models (100MB+)

Bundle in APK when:

Model is < 10MB Feature must work on first launch Model rarely changes Used by 100% of users

MobileNet (5MB), MediaPipe tasks (2-4MB), keyword spotters (<1MB) — these belong in the APK.

Download separately when:

Model is > 10MB Feature is optional (not all users need it) Model updates frequently (weekly/monthly) Multiple model variants for different use cases

Translation models (50MB each), speech models (100MB+), custom models that improve — download on demand.

APK Size Management

A 15MB model added to your APK means a 15MB larger download for every user — including those who never use the AI feature. Strategies to minimize impact:

// build.gradle.kts — use Android App Bundles
android {
    bundle {
        language { enableSplit = true }
        density { enableSplit = true }
        abi { enableSplit = true }
    }
    // Don't compress model files (enables memory mapping)
    androidResources {
        noCompress += listOf("tflite", "task", "onnx")
    }
}
App Bundles deliver only what the device needs. A user on an arm64 device gets only arm64 native code, not x86. Combined with split APKs, your model adds its full size only to users who actually need it.

Firebase ML Model Management

Firebase ML lets you host TFLite models on Google's infrastructure and push updates without app releases. This is the gold standard for production ML on Android:

OTA model update workflow — from training to production:

Train Model → Validate Offline → Upload to Firebase → 5% Rollout → Monitor Metrics → 100% Rollout 
dependencies {
    implementation("com.google.firebase:firebase-ml-modeldownloader:25.0.1")
}

// Download model from Firebase
val conditions = CustomModelDownloadConditions.Builder()
    .requireWifi()
    .build()

FirebaseModelDownloader.getInstance()
    .getModel("my_classifier_v2", DownloadType.LOCAL_MODEL_UPDATE_IN_BACKGROUND,
        conditions)
    .addOnSuccessListener { model ->
        val modelFile = model.file ?: return@addOnSuccessListener
        val interpreter = Interpreter(modelFile)
        // Ready to use — and it will auto-update in background
    }
    .addOnFailureListener { e ->
        Log.w("ML", "Download failed, using bundled fallback: ${e.message}")
        useBundledModel()
    }

Model Versioning and A/B Testing

Treat models like code: version them, test them, and roll them out gradually.

Versioning Strategy

// Embed version in SharedPreferences when model loads
val modelVersion = model.name.substringAfterLast("_v") // "classifier_v2.3"
prefs.edit().putString("active_model_version", modelVersion).apply()
analytics.logEvent("model_loaded") {
    param("model_version", modelVersion)
    param("model_size", model.file?.length() ?: 0)
}

A/B Testing with Remote Config

val remoteConfig = Firebase.remoteConfig
remoteConfig.fetchAndActivate().addOnSuccessListener {
    val modelVariant = remoteConfig.getString("classifier_model")
    // "classifier_v2" for 50% of users, "classifier_v3" for 50%

    FirebaseModelDownloader.getInstance()
        .getModel(modelVariant, DownloadType.LOCAL_MODEL, conditions)
        .addOnSuccessListener { model ->
            loadModel(model)
            // Log which variant this user got
            analytics.logEvent("ab_model_assigned") {
                param("variant", modelVariant)
            }
        }
}
Measure what matters in A/B tests: not just accuracy on test data, but real user engagement. Does the new model lead to more feature usage? Fewer corrections? Higher satisfaction scores? A model that's 2% more accurate but 3x slower might lose the A/B test because users stop using the feature.

Device Capability Detection

Not every device can run every model. Detect capabilities at runtime and adapt:

class DeviceCapability(context: Context) {

    val totalRamMB: Long
    val availableRamMB: Long
    val hasNpu: Boolean
    val apiLevel: Int = Build.VERSION.SDK_INT

    init {
        val memInfo = ActivityManager.MemoryInfo()
        (context.getSystemService(Context.ACTIVITY_SERVICE) as ActivityManager)
            .getMemoryInfo(memInfo)
        totalRamMB = memInfo.totalMem / (1024 * 1024)
        availableRamMB = memInfo.availMem / (1024 * 1024)

        // Check for NPU via NNAPI device list
        hasNpu = if (Build.VERSION.SDK_INT >= 29) {
            val devices = android.ml.NeuralNetworksDevice.getDevices()
            devices.any { it.type == NeuralNetworksDevice.DEVICE_TYPE_ACCELERATOR }
        } else false
    }

    fun getModelTier(): ModelTier = when {
        totalRamMB >= 8000 && hasNpu -> ModelTier.FLAGSHIP
        totalRamMB >= 4000 -> ModelTier.MID_RANGE
        totalRamMB >= 2000 -> ModelTier.BUDGET
        else -> ModelTier.UNSUPPORTED
    }
}

enum class ModelTier { FLAGSHIP, MID_RANGE, BUDGET, UNSUPPORTED }

Recommended model strategy per device tier:

Tier RAM Model Strategy Delegate
Flagship 8GB+, NPU Full-size models, Gemini Nano, multiple concurrent models NNAPI / QNN
Mid-range 4-8GB Quantized int8 models, one model at a time GPU delegate
Budget 2-4GB Smallest model variants only, aggressive unloading CPU (4 threads)
Unsupported <2GB Disable AI features, show cloud fallback or static alternative N/A

Error Handling and Graceful Degradation

In production, everything that can fail will fail. Plan for it:

class SafeModelManager(private val context: Context) {

    private var interpreter: Interpreter? = null

    fun loadModel(): Boolean {
        return try {
            val model = FileUtil.loadMappedFile(context, "classifier.tflite")
            interpreter = createSafeInterpreter(model) // Chapter 7 fallback chain
            true
        } catch (e: OutOfMemoryError) {
            logError("model_oom", e)
            false
        } catch (e: Exception) {
            logError("model_load_failed", e)
            false
        }
    }

    fun classify(bitmap: Bitmap): Result<Classification> {
        val interp = interpreter ?: return Result.failure(
            IllegalStateException("Model not loaded")
        )
        return try {
            val result = runInference(interp, bitmap)
            if (result.score < 0.3f) {
                // Low confidence — don't show unreliable results
                Result.failure(LowConfidenceException(result))
            } else {
                Result.success(result)
            }
        } catch (e: Exception) {
            logError("inference_failed", e)
            Result.failure(e)
        }
    }

    private fun logError(event: String, e: Throwable) {
        analytics.logEvent(event) {
            param("error", e.message ?: "unknown")
            param("device", Build.MODEL)
            param("ram_mb", getAvailableRam())
        }
    }
}
Never show garbage output to users. If the model returns low-confidence results (below your threshold), show a graceful fallback: "Unable to identify" is always better than a confident wrong answer. Users lose trust in AI features after one bad experience.

Memory Management in Production

Models consume significant RAM. Manage their lifecycle carefully:

class ModelLifecycleManager : DefaultLifecycleObserver {

    private var interpreter: Interpreter? = null

    override fun onResume(owner: LifecycleOwner) {
        // Load model when screen is visible
        if (interpreter == null) {
            interpreter = loadModel()
        }
    }

    override fun onStop(owner: LifecycleOwner) {
        // Unload model when screen is not visible
        interpreter?.close()
        interpreter = null
    }
}

// Also handle system memory pressure
class MyApp : Application(), ComponentCallbacks2 {
    override fun onTrimMemory(level: Int) {
        if (level >= TRIM_MEMORY_MODERATE) {
            // System is running low — release models
            ModelRegistry.unloadAll()
        }
    }
}
Load lazily, unload eagerly. Don't preload models at app startup — load them when the feature screen opens. Unload when the user leaves the screen. For models used across multiple screens, use a reference count and unload when the count hits zero.

Performance Monitoring

You can't improve what you don't measure. Track these metrics in production:

Key production metrics to track via Firebase Analytics:

Metric What to Track Alert Threshold
Inference latency p50, p95 per device tier p95 > 100ms (real-time features)
Model load success rate % of users who loaded model successfully < 95% success rate
Inference crash rate Crashes per 1K inferences > 0.1% crash rate
Download completion % of model downloads that finish < 90% completion
Delegate distribution % users on NPU / GPU / CPU Unexpected shifts after updates
Low-confidence rate % of inferences below confidence threshold > 30% low-confidence 
fun logInferenceMetrics(
    modelVersion: String,
    delegate: String,
    latencyMs: Long,
    confidence: Float,
    success: Boolean
) {
    Firebase.analytics.logEvent("ml_inference") {
        param("model_version", modelVersion)
        param("delegate", delegate)
        param("latency_ms", latencyMs)
        param("confidence", confidence.toDouble())
        param("success", if (success) 1L else 0L)
        param("device_model", Build.MODEL)
        param("device_tier", DeviceCapability(context).getModelTier().name)
    }
}

Privacy and Compliance

On-Device AI Is a Privacy Superpower

Every on-device AI feature in this course processes data locally. No images, text, or audio ever leave the device. This gives you genuine privacy advantages:

  • GDPR / CCPA — on-device processing typically doesn't constitute "data collection" since nothing is transmitted. Simpler compliance, less legal review.
  • User trust — you can truthfully say "processed entirely on your device" in your privacy policy and in-app messaging.
  • No consent required — processing data on the user's own device generally doesn't require explicit consent the way sending data to a server does.
  • No data breach risk — if data never leaves the device, it can't be leaked from your servers.

Communicate this clearly to users. "Processed on your device — your data stays private" is a feature worth highlighting.

The Production Deployment Checklist

  • Device capability check — detect RAM, NPU, API level; choose model tier or disable feature
  • Model loading with fallback — try NNAPI → GPU → CPU; never crash on delegate failure
  • Error handling — catch OOM, load failures, inference crashes; log to analytics
  • Confidence threshold — don't show results below your quality bar
  • Memory lifecycle — load on screen enter, unload on exit, handle onTrimMemory
  • Performance telemetry — log latency, success rate, delegate used, model version
  • Model versioning — track which version runs on which devices
  • OTA update path — Firebase ML or equivalent for updating models without app updates
  • Graceful degradation UI — meaningful fallback when the feature can't run
  • Privacy messaging — tell users their data is processed locally

Try It: Production-Harden a Feature

Take the image classifier from Chapter 3 (or any feature from Chapters 4-10) and add full production hardening.

  1. Add device capability detection — check RAM and NPU availability, choose model tier
  2. If the model isn't bundled, add a download flow with:
    • Progress indicator
    • Wi-Fi-only option
    • Retry on failure
    • Bundled fallback model if download fails
  3. Wrap all model operations in error handling — catch every exception, log to analytics
  4. Add a confidence threshold — show "Unable to identify" for low-confidence results
  5. Add performance telemetry — log inference latency, delegate used, model version, success/failure on every run
  6. Add graceful degradation — if the device is UNSUPPORTED tier, show a message explaining why and offer a cloud alternative (or disable the feature cleanly)
  7. Implement model lifecycle management — load on screen enter, unload on exit

Test by simulating failures: Force a delegate failure (pass an invalid delegate), force low memory (allocate large arrays before loading), and force a bad model file. Verify the app never crashes and always shows a meaningful UI.
