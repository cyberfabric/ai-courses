Chapter 8

Model Optimization — Making Models Mobile-Ready

Quantization, pruning, and distillation — shrink models without losing accuracy

Why You Can't Just Ship Any Model

In Chapter 7 you learned to run models faster using GPU and NPU delegates. But speed isn't your only constraint. A 100MB float32 model is a problem even on the fastest NPU:

  • APK size — a 100MB model turns your 20MB app into a 120MB download. Users abandon installs over 50MB on mobile data.
  • RAM usage — the model loads into memory during inference. On a 4GB device with 2GB free, a 100MB model plus its working memory can trigger OOM kills.
  • Load time — memory-mapping a 100MB file from flash storage takes 200-500ms. Users notice.

The goal is to take a model that works but is too big or too slow and optimize it for mobile without meaningfully hurting accuracy.

< 20 MB Target model size for bundling in APK
< 15 ms Target inference time on NPU
< 50 MB Target runtime memory usage

Quantization — The Single Biggest Win

Most ML models store their weights as 32-bit floating point numbers (float32). Each weight takes 4 bytes. Quantization reduces that to 16-bit (2 bytes) or 8-bit integers (1 byte).

Think of it like image compression. A RAW photo and a JPEG photo look almost identical to human eyes, but the JPEG is 10x smaller. Quantization does the same thing for model weights — the model's "knowledge" is almost the same, but the file is 2-4x smaller and inference is 2-4x faster.

What quantization does to a model:

Float32 Model (14 MB) → Quantize Weights → Int8 Model (3.5 MB) → Same Accuracy (±1-2%)

4x smaller file, 2-4x faster inference, fits in cache better, runs efficiently on NPU — all from a single conversion step that takes minutes.

Three Types of Post-Training Quantization

Post-training quantization (PTQ) is the easiest optimization — you take an existing model and convert it. No retraining needed. TFLite offers three variants:

1. Dynamic Range Quantization

Quantizes weights only from float32 to int8. Activations stay float32 at runtime. The simplest option with zero data required:

# Python — run this once to convert your model
import tensorflow as tf

converter = tf.lite.TFLiteConverter.from_saved_model("saved_model/")
converter.optimizations = [tf.lite.Optimize.DEFAULT]

quantized_model = converter.convert()
with open("model_dynamic.tflite", "wb") as f:
    f.write(quantized_model)
Metric Result
Size reduction~2x smaller (weights go from 4 bytes to 1 byte, but activations stay float)
Speed improvement~1.5-2x faster on CPU
Accuracy impactMinimal (< 1% drop typically)
Data requiredNone
Best forQuick wins, first attempt at optimization

2. Full Integer Quantization (int8)

Quantizes both weights and activations to int8. This is the production standard — maximum compression and speed. Requires a small representative dataset (100-500 samples) to calibrate activation ranges:

# Python — full int8 quantization
import tensorflow as tf
import numpy as np

converter = tf.lite.TFLiteConverter.from_saved_model("saved_model/")
converter.optimizations = [tf.lite.Optimize.DEFAULT]

# Representative dataset for calibrating activation ranges
def representative_dataset():
    for _ in range(200):
        # Generate or load sample inputs matching your model's input shape
        sample = np.random.rand(1, 224, 224, 3).astype(np.float32)
        yield [sample]

converter.representative_dataset = representative_dataset
converter.target_spec.supported_ops = [tf.lite.OpsSet.TFLITE_BUILTINS_INT8]
converter.inference_input_type = tf.uint8
converter.inference_output_type = tf.uint8

quantized_model = converter.convert()
with open("model_int8.tflite", "wb") as f:
    f.write(quantized_model)
Metric Result
Size reduction~4x smaller
Speed improvement~2-4x faster on CPU, 5-10x with NPU
Accuracy impactSmall (1-2% drop typically)
Data required100-500 representative samples
Best forProduction models, NPU deployment, maximum performance
Int8 models are the sweet spot for NPU execution. NPUs are designed to crunch int8 operations at maximum throughput. A float32 model on NNAPI may actually fall back to CPU for some ops, while an int8 model runs entirely on the NPU. Quantize to int8 before deploying with NNAPI for best results.

3. Float16 Quantization

Reduces precision from float32 to float16 (half precision). Minimal accuracy loss, and GPUs are optimized for float16 math:

# Python — float16 quantization
converter = tf.lite.TFLiteConverter.from_saved_model("saved_model/")
converter.optimizations = [tf.lite.Optimize.DEFAULT]
converter.target_spec.supported_types = [tf.float16]

quantized_model = converter.convert()
with open("model_fp16.tflite", "wb") as f:
    f.write(quantized_model)
Metric Result
Size reduction~2x smaller
Speed improvement~1.5-2x faster on GPU
Accuracy impactNegligible (< 0.5% drop)
Data requiredNone
Best forGPU delegate deployment, accuracy-sensitive models

MobileNetV2 — quantization impact (Pixel 8 Pro, NNAPI):

Variant Model Size CPU Inference NPU Inference Top-1 Accuracy
Float32 (original) 14.0 MB 25 ms 8 ms 71.8%
Float16 7.0 MB 22 ms 6 ms 71.7%
Dynamic range 4.3 MB 15 ms 5 ms 71.2%
Full int8 3.5 MB 10 ms 2.5 ms 70.5%

Full int8 is 4x smaller, 10x faster on NPU, with only 1.3% accuracy loss. For most mobile use cases, that's a fantastic trade. Float16 is the safest bet when you can't afford any accuracy loss.

Quantization-Aware Training (QAT)

Sometimes post-training quantization drops accuracy too much — especially on small models where every bit of precision matters. QAT solves this by simulating quantization during training, so the model learns to compensate:

# Python — QAT with TensorFlow Model Optimization Toolkit
import tensorflow_model_optimization as tfmot

# Take your already-trained model
model = tf.keras.models.load_model("my_model")

# Apply QAT — inserts fake quantization nodes
quantize_model = tfmot.quantization.keras.quantize_model
qat_model = quantize_model(model)

# Fine-tune for a few epochs with your training data
qat_model.compile(optimizer="adam", loss="categorical_crossentropy")
qat_model.fit(train_dataset, epochs=5)

# Convert to TFLite (already quantization-ready)
converter = tf.lite.TFLiteConverter.from_keras_model(qat_model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
tflite_model = converter.convert()
Only use QAT when post-training quantization fails. QAT requires the original training pipeline and dataset, which you may not have for pre-trained models. Try PTQ first — if accuracy drops more than 2-3%, then invest in QAT.

Pruning — Removing Unnecessary Weights

Pruning removes weights that contribute little to the model's output — setting them to zero. A pruned model has many zero weights, which means less computation and smaller file size after compression.

Think of pruning like removing dead branches from a tree. The tree still grows the same fruit; it's just lighter and more efficient.

Structured Pruning

Removes entire channels or layers. Result: physically smaller model, fewer operations. Speedup is real — less work to do. Works with any hardware. Typical: 30-50% of channels removed.

Best for production: the model is genuinely smaller and faster because entire computation paths are removed.

Unstructured Pruning

Zeros individual weights scattered throughout. Result: sparse model, same shape, many zeros. Speedup needs sparse hardware support. Most mobile NPUs don't accelerate sparse models. Typical: 70-90% of weights zeroed.

Looks impressive on paper (90% sparse!), but most mobile hardware can't exploit sparsity. Use structured pruning for mobile instead.

Knowledge Distillation — Training a Smaller Model

When quantization and pruning aren't enough, you can train a small "student" model to mimic a large "teacher" model. The student learns not just the correct answers, but the teacher's confidence patterns — which makes it smarter than if trained from scratch on the data alone.

This is how models like MobileNet were created: a huge model was trained on ImageNet, then a tiny model was trained to match the big model's output. The result is a 5MB model that punches far above its weight.

# Python — simplified distillation
# Teacher: big accurate model (EfficientNet-B7, 66MB)
# Student: small fast model (MobileNetV3-Small, 2MB)

for images, labels in training_data:
    # Get teacher's soft predictions (probabilities, not just labels)
    teacher_probs = teacher_model(images, training=False)

    with tf.GradientTape() as tape:
        student_probs = student_model(images, training=True)
        # Loss combines: match the teacher + match the labels
        loss = 0.7 * kl_divergence(teacher_probs, student_probs) + \
               0.3 * cross_entropy(labels, student_probs)

    gradients = tape.gradient(loss, student_model.trainable_variables)
    optimizer.apply_gradients(zip(gradients, student_model.trainable_variables))
Distillation requires the full training pipeline. You need the training dataset, a GPU for training (free via Google Colab), and Python ML expertise. It's the most powerful optimization technique, but also the most effort. Only reach for distillation when pre-trained mobile models don't meet your accuracy requirements.

Model Format Conversion

Many models you'll find online are in PyTorch or TensorFlow SavedModel format. To run them on Android, you need to convert to TFLite or ONNX:

Common conversion paths to mobile formats:

PyTorch → ONNX → TFLite

TensorFlow SavedModel → TFLite (direct) 
# PyTorch → ONNX
import torch
dummy_input = torch.randn(1, 3, 224, 224)
torch.onnx.export(model, dummy_input, "model.onnx",
                   input_names=["input"], output_names=["output"])

# ONNX → TFLite (using onnx2tf tool)
# pip install onnx2tf
# onnx2tf -i model.onnx -o saved_model/
# Then use TFLiteConverter on the saved_model/
Conversion pitfalls to watch for:
  • Unsupported ops — some PyTorch operations don't have TFLite equivalents. The converter will error with the op name — search for TFLite custom op or restructure the model.
  • Dynamic shapes — TFLite prefers static input shapes. Models with variable-length inputs may need reshaping.
  • Accuracy drift — always verify accuracy after conversion. Run the same test images through both the original and converted model.

ONNX Runtime Mobile

If your model originates from PyTorch and doesn't convert cleanly to TFLite, consider ONNX Runtime Mobile as an alternative:

dependencies {
    implementation("com.microsoft.onnxruntime:onnxruntime-android:1.19.0")
}

val session = OrtEnvironment.getEnvironment().use { env ->
    val options = OrtSession.SessionOptions()
    options.addNnapi()  // Use NNAPI for hardware acceleration
    env.createSession(modelBytes, options)
}

// Run inference
val inputTensor = OnnxTensor.createTensor(env, inputArray)
val results = session.run(mapOf("input" to inputTensor))
val output = results[0].value as Array<FloatArray>

TFLite vs. ONNX Runtime Mobile — when to choose which:

  • Use TFLite when: the model originates from TensorFlow/Keras, you want the widest hardware acceleration support (NNAPI, GPU, QNN), or you're using Google's ecosystem (ML Kit, MediaPipe).
  • Use ONNX when: the model originates from PyTorch and doesn't convert to TFLite cleanly, you need cross-platform compatibility with iOS (ONNX works on both), or the model uses ops not supported by TFLite.
  • Both support: NNAPI hardware acceleration, quantized models, and run on all Android devices back to API 21.

The Optimization Decision Tree

When a model is too big or too slow for mobile, follow this decision tree:

Step-by-Step Optimization Strategy

  1. Start with post-training int8 quantization — 4x smaller, 2-4x faster, 5 minutes of work. If accuracy is acceptable (< 2% drop), you're done.
  2. If accuracy drops too much, try float16 — 2x smaller, almost no accuracy loss. Pair with GPU delegate.
  3. If still too large, try structured pruning — remove 30-50% of channels, then quantize the pruned model.
  4. If still not good enough, consider distillation — train a smaller architecture to match the big model's output.
  5. If nothing works, pick a different base model — switch from EfficientNet to MobileNet, or from a generic model to a domain-specific one.

Most mobile models only need step 1. Seriously — int8 quantization solves 90% of "model too big" problems.

Putting It Together — Optimization + Acceleration

The real magic happens when you combine Chapter 7 (hardware acceleration) with Chapter 8 (model optimization). Take MobileNetV2 and apply both:

Configuration Size Inference Speedup vs. Baseline
Float32 + CPU (baseline) 14 MB 25 ms 1x
Float32 + GPU delegate 14 MB 8 ms 3x
Int8 + CPU 3.5 MB 10 ms 2.5x
Int8 + NNAPI (NPU) 3.5 MB 2.5 ms 10x

From 14MB / 25ms to 3.5MB / 2.5ms. 4x smaller and 10x faster. Same model, same accuracy (within 1.3%), same app. That's the power of optimization + acceleration combined.

Try It: Quantize and Benchmark MobileNetV2

Take a float32 MobileNetV2 model, quantize it to int8, and measure the real-world impact.

  1. Download MobileNetV2 float32 from TensorFlow Hub (~14 MB)
  2. Run int8 post-training quantization using the Python script above (use Google Colab if you don't have a local Python environment)
  3. Load both models in your Android app and run the same 20 test images through each
  4. For each model, measure and record:
    • File size (original vs. quantized)
    • Inference time on CPU (4 threads)
    • Inference time with NNAPI delegate
    • Top-1 prediction match (did both models agree?)
  5. Calculate: size reduction, speedup, and accuracy agreement percentage

Expected results: ~4x size reduction (14MB → 3.5MB), ~2.5x CPU speedup, ~3x NPU speedup, and 18-20 out of 20 images should produce the same top prediction.
