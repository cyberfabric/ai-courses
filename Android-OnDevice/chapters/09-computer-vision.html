Chapter 9

Computer Vision Features

Object detection, segmentation, custom training, and real-time video at 30fps

Beyond "What Is This?"

In Chapter 3 you built an image classifier that answers "what is this?" — one label for the whole image. Useful, but real apps need more. They need to know where objects are, which pixels belong to which object, and how to track objects across frames.

This chapter covers the four levels of computer vision, from simplest to most complex:

Classification "What is it?" — one label per image
Detection "Where is it?" — bounding boxes + labels
Segmentation "Which pixels?" — per-pixel masks

Object Detection with SSD-MobileNet

Object detection is the workhorse of mobile vision. Instead of one label for the whole image, you get a list of objects, each with a class name, confidence score, and bounding box telling you exactly where it is.

Object detection pipeline — from camera frame to bounding boxes:

Camera Frame → Resize 320×320 → SSD-MobileNet → NMS Filter → Boxes + Labels → Canvas Overlay

SSD-MobileNetV2 is the standard choice: 6.8MB, detects 90 COCO object categories (person, car, dog, chair, phone, etc.), and runs in under 10ms on an NPU.

class ObjectDetector(context: Context) {

    private val interpreter: Interpreter
    private val labels: List<String>
    private val inputSize = 320

    init {
        val model = FileUtil.loadMappedFile(context, "ssd_mobilenet_v2.tflite")
        val options = Interpreter.Options().apply {
            addDelegate(NnApiDelegate())
        }
        interpreter = Interpreter(model, options)
        labels = FileUtil.loadLabels(context, "coco_labels.txt")
    }

    fun detect(bitmap: Bitmap): List<Detection> {
        val resized = Bitmap.createScaledBitmap(bitmap, inputSize, inputSize, true)
        val input = TensorImage(DataType.FLOAT32).apply { load(resized) }

        // SSD outputs: [1,N,4] boxes, [1,N] classes, [1,N] scores, [1] count
        val boxes = Array(1) { Array(10) { FloatArray(4) } }
        val classes = Array(1) { FloatArray(10) }
        val scores = Array(1) { FloatArray(10) }
        val count = FloatArray(1)

        val outputs = mapOf(
            0 to boxes, 1 to classes, 2 to scores, 3 to count
        )
        interpreter.runForMultipleInputsOutputs(arrayOf(input.buffer), outputs)

        return (0 until count[0].toInt()).map { i ->
            Detection(
                label = labels[classes[0][i].toInt()],
                score = scores[0][i],
                box = RectF(boxes[0][i][1], boxes[0][i][0],
                            boxes[0][i][3], boxes[0][i][2])
            )
        }.filter { it.score > 0.5f }
    }
}

data class Detection(val label: String, val score: Float, val box: RectF)
SSD output boxes are normalized to [0, 1]. The bounding box coordinates represent percentages of the image dimensions, not pixels. Multiply by the display width/height to get screen coordinates: screenX = box.left * viewWidth.

Non-Maximum Suppression (NMS)

When a detector sees a cat, it doesn't return one box — it returns 20 overlapping boxes for the same cat. NMS filters these duplicates by keeping only the highest-confidence box and removing any that overlap too much.

The algorithm is simple:

  1. Sort detections by confidence (highest first)
  2. Take the top detection — keep it
  3. Remove any remaining detections that overlap with it by more than a threshold (IoU > 0.5)
  4. Repeat with the next remaining detection
fun nonMaxSuppression(
    detections: List<Detection>, iouThreshold: Float = 0.5f
): List<Detection> {
    val sorted = detections.sortedByDescending { it.score }.toMutableList()
    val kept = mutableListOf<Detection>()

    while (sorted.isNotEmpty()) {
        val best = sorted.removeFirst()
        kept.add(best)
        sorted.removeAll { other ->
            calculateIoU(best.box, other.box) > iouThreshold
        }
    }
    return kept
}

fun calculateIoU(a: RectF, b: RectF): Float {
    val intersection = RectF()
    if (!intersection.setIntersect(a, b)) return 0f
    val intersectionArea = intersection.width() * intersection.height()
    val unionArea = a.width() * a.height() + b.width() * b.height() - intersectionArea
    return intersectionArea / unionArea
}
Many TFLite detection models include NMS as a built-in postprocessing step — check the model metadata before implementing your own. The SSD-MobileNet model from TensorFlow Hub already outputs NMS-filtered results.

Image Segmentation with DeepLab

Segmentation goes pixel by pixel: instead of a box around a person, you get a mask that covers exactly the person's outline. This powers portrait mode (blur the background), background replacement, and visual effects.

class Segmenter(context: Context) {

    private val interpreter: Interpreter
    private val inputSize = 257

    init {
        val model = FileUtil.loadMappedFile(context, "deeplabv3_257.tflite")
        interpreter = Interpreter(model, Interpreter.Options().apply {
            addDelegate(GpuDelegate())
        })
    }

    fun segment(bitmap: Bitmap): IntArray {
        val resized = Bitmap.createScaledBitmap(bitmap, inputSize, inputSize, true)
        val input = TensorImage(DataType.FLOAT32).apply { load(resized) }

        // Output: [1, 257, 257, 21] — 21 class probabilities per pixel
        val output = Array(1) { Array(inputSize) { Array(inputSize) { FloatArray(21) } } }
        interpreter.run(input.buffer, output)

        // For each pixel, find the class with highest probability
        val mask = IntArray(inputSize * inputSize)
        for (y in 0 until inputSize) {
            for (x in 0 until inputSize) {
                mask[y * inputSize + x] = output[0][y][x].indices.maxByOrNull {
                    output[0][y][x][it]
                } ?: 0
            }
        }
        return mask  // 0=background, 15=person, etc.
    }
}

Building a Portrait Mode Effect

@Composable
fun PortraitModeOverlay(mask: IntArray, size: Int) {
    Canvas(modifier = Modifier.fillMaxSize()) {
        val scaleX = this.size.width / size
        val scaleY = this.size.height / size

        for (y in 0 until size) {
            for (x in 0 until size) {
                if (mask[y * size + x] != 15) { // 15 = person class
                    // Draw semi-transparent blur overlay on non-person pixels
                    drawRect(
                        color = Color.Black.copy(alpha = 0.5f),
                        topLeft = Offset(x * scaleX, y * scaleY),
                        size = Size(scaleX, scaleY)
                    )
                }
            }
        }
    }
}

Custom Model Training for Your Domain

COCO's 90 categories are a good start, but your app probably needs to detect objects that aren't in any pre-trained model — your company's products, specific document types, or domain-specific items. You need a custom model.

The fastest path: TensorFlow Model Maker — a high-level library that fine-tunes a pre-trained detector on your images with minimal code:

# Python — in Google Colab (free GPU)
from tflite_model_maker import object_detector

# Your labeled images in Pascal VOC or COCO format
train_data = object_detector.DataLoader.from_pascal_voc(
    "train_images/", "train_annotations/", ["product_a", "product_b", "product_c"]
)

# Fine-tune EfficientDet-Lite0 on your data
model = object_detector.create(
    train_data,
    model_spec=object_detector.EfficientDetLite0Spec(),
    epochs=50,
    batch_size=8
)

# Export as TFLite (automatically quantized)
model.export(export_dir=".", tflite_filename="my_detector.tflite")

How Much Data Do You Need?

Images per Class Expected Accuracy Good For
50-100 70-80% Prototype / proof of concept
200-500 85-92% Production MVP — usually sufficient
1,000+ 93-98% High-accuracy production
Use data augmentation to multiply your dataset. Flip, rotate, crop, adjust brightness/contrast, and add noise to your training images. A 200-image dataset with augmentation can perform like a 1,000-image dataset. TensorFlow Model Maker applies standard augmentations automatically.

Real-Time Video Processing — The 30fps Challenge

Running detection on a single image is straightforward. Running it at 30fps on a live camera feed requires careful engineering. At 30fps, you have 33ms per frame — and you need to capture, preprocess, run inference, postprocess, and render within that budget.

The Producer-Consumer Pattern

The key insight: don't process every frame. The camera produces frames at 30fps, but inference might take 10ms. You don't need the result of every single frame — the user can't perceive the difference between 30fps and 20fps detection updates.

// Producer: camera pushes frames at 30fps
// Consumer: inference processes as fast as it can, skipping stale frames
val latestFrame = AtomicReference<Bitmap?>(null)

// Camera side — always update to latest frame
analysis.setAnalyzer(executor) { proxy ->
    latestFrame.set(proxy.toBitmap())
    proxy.close()
}

// Inference side — process latest available frame
val detectionJob = scope.launch(Dispatchers.Default) {
    while (isActive) {
        val frame = latestFrame.getAndSet(null) ?: continue
        val results = detector.detect(frame)
        withContext(Dispatchers.Main) {
            detectionState.value = results
        }
    }
}

Efficient frame processing

Process latest frame, skip stale ones. Inference runs on Dispatchers.Default. UI updates via StateFlow (non-blocking). Reuse Bitmap buffers when possible. Frame skip: process every 2nd frame on slow devices.

Smooth camera preview, inference runs as fast as the hardware allows, no frame queue buildup.

Frame processing anti-patterns

Queue every frame for processing (OOM). Run inference on the main thread (ANR). Create new Bitmap every frame (GC thrashing). Block camera callback until inference completes. No frame skipping on slow devices (stutters).

Memory pressure, UI freezes, and eventually a crash. Never queue frames — always use latest-only.

Drawing Results on Compose Canvas

Scaling detection coordinates from model space to screen space is the trickiest part of any vision feature. The model operates on a 320×320 input, but your camera preview fills the screen at an arbitrary aspect ratio:

@Composable
fun DetectionCanvas(
    detections: List<Detection>,
    imageWidth: Int,  // model input width (e.g., 320)
    imageHeight: Int  // model input height (e.g., 320)
) {
    val paint = remember {
        Paint().apply {
            color = android.graphics.Color.parseColor("#FBBF24")
            textSize = 40f
            typeface = Typeface.DEFAULT_BOLD
        }
    }

    Canvas(modifier = Modifier.fillMaxSize()) {
        val scaleX = size.width / imageWidth
        val scaleY = size.height / imageHeight

        detections.forEach { det ->
            val left = det.box.left * imageWidth * scaleX
            val top = det.box.top * imageHeight * scaleY
            val right = det.box.right * imageWidth * scaleX
            val bottom = det.box.bottom * imageHeight * scaleY

            drawRect(
                color = Color(0xFFFBBF24),
                topLeft = Offset(left, top),
                size = Size(right - left, bottom - top),
                style = Stroke(3.dp.toPx())
            )
            drawContext.canvas.nativeCanvas.drawText(
                "${det.label} ${(det.score * 100).toInt()}%",
                left, top - 10f, paint
            )
        }
    }
}

Aspect ratio handling between camera and model:

  • Camera sensor typically outputs 4:3 (e.g., 4032×3024)
  • Preview display is often 16:9 or 20:9
  • Model input is usually square (320×320 or 640×640)

When you resize a 4:3 frame to a 1:1 model input, objects get squished. Two strategies:

  • Letterbox (pad) — add black bars to maintain aspect ratio. More accurate boxes, but wastes model input space.
  • Stretch (distort) — stretch to fill the square. Uses all input pixels, but boxes need inverse-distortion scaling. Most mobile models handle mild distortion well.

Testing Vision Features

Vision features need automated tests just like any other code. Build a test suite with saved images and expected results:

@Test
fun testDetectsPersonInOfficePhoto() {
    val bitmap = loadTestImage("test_images/office_scene.jpg")
    val detections = detector.detect(bitmap)

    val personDetections = detections.filter { it.label == "person" }
    assertTrue("Should detect at least one person", personDetections.isNotEmpty())
    assertTrue("Confidence should be > 70%", personDetections[0].score > 0.7f)

    // Bounding box should be in the expected region (center of image)
    val box = personDetections[0].box
    assertTrue("Person should be in center region", box.centerX() in 0.2f..0.8f)
}

@Test
fun testModelPerformance() {
    val bitmap = loadTestImage("test_images/office_scene.jpg")
    val times = (1..50).map {
        measureTimeMillis { detector.detect(bitmap) }
    }
    val p95 = times.sorted()[(times.size * 0.95).toInt()]
    assertTrue("p95 latency should be under 15ms", p95 < 15)
}
Keep a folder of 20-30 test images covering your key scenarios: well-lit, dim, close-up, far away, multiple objects, empty scene. Run these tests whenever you change models or update preprocessing. They catch regressions that manual testing misses.

Build a production-quality real-time object detector that draws labeled bounding boxes on the live camera feed.

  1. Download SSD-MobileNetV2 (int8 quantized) and COCO labels into assets/
  2. Load the model with NNAPI delegate (with GPU and CPU fallbacks)
  3. Connect to CameraX with the producer-consumer pattern (process latest frame only)
  4. Draw bounding boxes with labels and confidence scores on a Compose Canvas overlay
  5. Display the current FPS and inference time in a debug overlay
  6. Verify the detector maintains 30fps camera preview with detection updates at 15+ fps

Test with:

  • Your desk — should detect monitor, keyboard, mouse, cup, bottle, phone
  • A room with people — should draw boxes around each person
  • A street view — should detect cars, bicycles, traffic lights

Bonus: Add frame skipping for slow devices — process every 2nd or 3rd frame when inference exceeds 30ms.
