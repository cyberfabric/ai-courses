NLP and Audio On-Device

Text classification, sentiment analysis, speech-to-text, and keyword spotting — all on the phone

Beyond Images — Understanding Words and Sounds

Chapters 3-9 focused on what your phone can see. This chapter is about what it can read and hear. The inference pattern is identical — load model, preprocess input, run inference, postprocess output — but the input is text or audio instead of pixels.

On-device NLP and audio are already shipping in features you use daily: Gboard's next-word prediction, Google Assistant's offline voice commands, and spam call filtering. Now you'll build the same kind of features.

What's Practical On-Device in 2026

Text classification (spam, category, intent) Sentiment analysis (positive/negative/neutral) Entity extraction (dates, phones, addresses) Language detection Keyword spotting / wake words Audio classification (sounds, events) Short speech-to-text (commands, dictation) Semantic similarity / search

Text Classification with MobileBERT

Text classification is the NLP equivalent of image classification — you give it text, it tells you which category it belongs to. Spam filter, support ticket router, content moderator, intent classifier — they're all the same pattern.

Text classification pipeline — from raw text to category:

Raw Text → Tokenize → Token IDs + Mask → MobileBERT → Category Scores → Label

MobileBERT is the go-to model for on-device NLP. It's a compressed version of BERT — 25MB, ~12ms inference on NPU, and surprisingly capable for classification tasks.

Tokenization — Turning Text into Numbers

Models don't understand words — they understand numbers. Tokenization converts text into a sequence of integer IDs that the model can process. It's the NLP equivalent of resizing an image to 224×224.

// WordPiece tokenization (used by BERT / MobileBERT)
// "I love this app" → ["[CLS]", "i", "love", "this", "app", "[SEP]"]
// → [101, 1045, 2293, 2023, 10439, 102]

class WordPieceTokenizer(context: Context) {
    private val vocab: Map<String, Int>

    init {
        // Load vocabulary file (30,522 tokens for BERT)
        val vocabText = context.assets.open("vocab.txt").bufferedReader().readLines()
        vocab = vocabText.withIndex().associate { (i, word) -> word to i }
    }

    fun tokenize(text: String, maxLength: Int = 128): IntArray {
        val tokens = mutableListOf(101) // [CLS] token
        text.lowercase().split(" ").forEach { word ->
            tokens.add(vocab[word] ?: vocab["[UNK]"]!!)
        }
        tokens.add(102) // [SEP] token

        // Pad or truncate to fixed length
        return IntArray(maxLength) { i ->
            if (i < tokens.size) tokens[i] else 0
        }
    }
}
The tokenizer must match the model. BERT uses WordPiece. GPT-style models use BPE (Byte Pair Encoding). SentencePiece is used by many multilingual models. Using the wrong tokenizer produces garbage input IDs and meaningless output. Always use the tokenizer that shipped with your model.

Running Text Classification

class TextClassifier(context: Context) {

    private val interpreter: Interpreter
    private val tokenizer: WordPieceTokenizer
    private val labels = listOf("urgent", "informational", "social", "spam")

    init {
        val model = FileUtil.loadMappedFile(context, "text_classifier.tflite")
        interpreter = Interpreter(model, Interpreter.Options().apply {
            addDelegate(NnApiDelegate())
        })
        tokenizer = WordPieceTokenizer(context)
    }

    fun classify(text: String): Pair<String, Float> {
        val inputIds = tokenizer.tokenize(text)
        val attentionMask = IntArray(128) { if (inputIds[it] != 0) 1 else 0 }

        val output = Array(1) { FloatArray(labels.size) }
        interpreter.runForMultipleInputsOutputs(
            arrayOf(inputIds, attentionMask), mapOf(0 to output)
        )

        val maxIdx = output[0].indices.maxByOrNull { output[0][it] } ?: 0
        return labels[maxIdx] to output[0][maxIdx]
    }
}

Sentiment Analysis

Sentiment analysis is just text classification with three (or five) classes: positive, negative, neutral. You can use the same MobileBERT architecture, fine-tuned on sentiment data:

val sentimentLabels = listOf("negative", "neutral", "positive")

fun analyzeSentiment(text: String): String {
    val (label, confidence) = sentimentClassifier.classify(text)
    return when {
        label == "positive" && confidence > 0.7f -> "Positive 😊"
        label == "negative" && confidence > 0.7f -> "Negative 😞"
        else -> "Neutral 😐"
    }
}

On-Device Speech-to-Text

Three approaches, from easiest to most flexible:

A
Android SpeechRecognizer (Offline)
Built-in Android API. Download offline language packs in Settings → Languages. Zero code for basic use. Limited customization, but works on almost all devices. Best for: voice commands, simple dictation.
V
Vosk
Lightweight open-source speech recognition. Models are 50-200MB per language. Runs as a library in your app — no system dependency. Supports streaming recognition. Best for: always-on recognition, custom vocabularies.
W
Whisper.cpp
OpenAI's Whisper model compiled for mobile via C++. Highest accuracy. Whisper-tiny is 39MB, processes ~1 second of audio in ~200ms. Best for: high-accuracy transcription where latency isn't critical.

Android SpeechRecognizer — The Quick Path

// Check for offline recognition support
val recognizer = SpeechRecognizer.createOnDeviceSpeechRecognizer(context)

val intent = Intent(RecognizerIntent.ACTION_RECOGNIZE_SPEECH).apply {
    putExtra(RecognizerIntent.EXTRA_LANGUAGE_MODEL,
        RecognizerIntent.LANGUAGE_MODEL_FREE_FORM)
    putExtra(RecognizerIntent.EXTRA_LANGUAGE, "en-US")
    putExtra(RecognizerIntent.EXTRA_PARTIAL_RESULTS, true)
}

recognizer.setRecognitionListener(object : RecognitionListener {
    override fun onResults(results: Bundle) {
        val matches = results.getStringArrayList(
            SpeechRecognizer.RESULTS_RECOGNITION
        )
        val transcription = matches?.firstOrNull() ?: ""
        updateUI(transcription)
    }
    override fun onPartialResults(partialResults: Bundle) {
        val partial = partialResults.getStringArrayList(
            SpeechRecognizer.RESULTS_RECOGNITION
        )?.firstOrNull() ?: ""
        updateUI(partial) // Show text as user speaks
    }
    // ... other callbacks
})
recognizer.startListening(intent)
createOnDeviceSpeechRecognizer() requires Android 12+ (API 31). On older devices, use createSpeechRecognizer() which may fall back to cloud recognition. Always check SpeechRecognizer.isOnDeviceRecognitionAvailable(context) at runtime.

Keyword Spotting — Always-On Wake Word Detection

Keyword spotting uses a tiny model (< 1MB) that listens for specific phrases — like "Hey Google" or your app's custom wake word. The model is so small it can run continuously in a background service with minimal battery impact.

class KeywordSpotter(context: Context) {

    private val interpreter: Interpreter
    private val keywords = listOf("hey_app", "stop", "next", "unknown")

    init {
        val model = FileUtil.loadMappedFile(context, "keyword_model.tflite")
        interpreter = Interpreter(model) // CPU is fine for < 1ms inference
    }

    fun detectKeyword(audioFrame: FloatArray): String? {
        // audioFrame: 1 second of audio at 16kHz = 16,000 floats
        // Extract mel spectrogram features
        val features = extractMelSpectrogram(audioFrame)

        val output = Array(1) { FloatArray(keywords.size) }
        interpreter.run(features, output)

        val maxIdx = output[0].indices.maxByOrNull { output[0][it] } ?: 0
        val confidence = output[0][maxIdx]

        return if (confidence > 0.85f && keywords[maxIdx] != "unknown") {
            keywords[maxIdx]
        } else null
    }
}

Battery impact of always-on audio processing:

  • Keyword spotting model (< 1MB, ~1ms inference) — negligible battery impact. Can run continuously for days. This is how "Hey Google" works without draining your battery.
  • Full speech recognition (~50MB, ~200ms/sec) — significant battery drain. Only activate after a keyword is detected. Never run continuously.
  • Audio classification (YAMNet, ~4MB, ~5ms) — moderate impact. Suitable for short sessions (fitness app listening for rep sounds) but not 24/7 monitoring.

The pattern: use a tiny keyword spotter as a gate, then activate the heavy model only when triggered.

Audio Classification with YAMNet

YAMNet classifies sounds into 521 categories — speech, music, dog barking, siren, cough, applause, and hundreds more. It's useful for accessibility features (alerting deaf users to sounds), smart home triggers, and content analysis.

class SoundClassifier(context: Context) {

    private val interpreter: Interpreter
    private val labels: List<String> // 521 sound classes

    init {
        val model = FileUtil.loadMappedFile(context, "yamnet.tflite")
        interpreter = Interpreter(model)
        labels = FileUtil.loadLabels(context, "yamnet_labels.txt")
    }

    fun classify(audioWaveform: FloatArray): List<SoundResult> {
        // YAMNet expects: 0.975 seconds of audio at 16kHz (15,600 samples)
        val output = Array(1) { FloatArray(521) }
        interpreter.run(audioWaveform, output)

        return output[0].mapIndexed { i, score -> SoundResult(labels[i], score) }
            .sortedByDescending { it.score }
            .take(5)
    }
}

data class SoundResult(val label: String, val score: Float)

Text Embeddings — On-Device Semantic Search

Text embeddings convert text into numerical vectors that capture meaning. Similar texts produce similar vectors. This enables powerful features without a server: semantic search ("find messages about meetings"), duplicate detection, and recommendation.

class TextEmbedder(context: Context) {

    private val interpreter: Interpreter
    private val tokenizer: WordPieceTokenizer

    init {
        val model = FileUtil.loadMappedFile(context, "all_minilm_l6.tflite")
        interpreter = Interpreter(model, Interpreter.Options().apply {
            addDelegate(NnApiDelegate())
        })
        tokenizer = WordPieceTokenizer(context)
    }

    fun embed(text: String): FloatArray {
        val inputIds = tokenizer.tokenize(text, maxLength = 64)
        val attentionMask = IntArray(64) { if (inputIds[it] != 0) 1 else 0 }
        val embedding = Array(1) { FloatArray(384) } // 384-dim embedding

        interpreter.runForMultipleInputsOutputs(
            arrayOf(inputIds, attentionMask), mapOf(0 to embedding)
        )
        return embedding[0]
    }

    fun similarity(a: FloatArray, b: FloatArray): Float {
        val dot = a.zip(b).sumOf { (x, y) -> (x * y).toDouble() }.toFloat()
        val normA = sqrt(a.sumOf { (it * it).toDouble() }).toFloat()
        val normB = sqrt(b.sumOf { (it * it).toDouble() }).toFloat()
        return dot / (normA * normB)  // cosine similarity: -1 to 1
    }
}

// Usage: find the most relevant message
val query = embedder.embed("when is the meeting?")
val messages = listOf("Team standup at 3pm", "Great job on the PR!", "Lunch?")
val scored = messages.map { it to embedder.similarity(query, embedder.embed(it)) }
    .sortedByDescending { it.second }
// scored[0] = ("Team standup at 3pm", 0.78)

Combining NLP with Generative AI

Traditional NLP models and Gemini Nano (Chapter 6) serve different purposes. Use the right tool for each job:

When to Use What

  • Classification / sentiment / detection → MobileBERT (25MB, ~12ms, deterministic, reliable)
  • Summarization / rewriting / generation → Gemini Nano (system-managed, ~30 tok/sec, creative but variable)
  • Entity extraction → ML Kit (Chapter 5) — no model management needed
  • Wake word detection → Tiny custom model (< 1MB, always-on)
  • Full speech-to-text → Android SpeechRecognizer or Whisper

Traditional NLP models are fast, cheap, and deterministic — same input always produces same output. Generative models are flexible but variable — great for creative tasks, unreliable for classification. Combine both in your app for the best experience.

Multilingual Considerations

If your app serves multiple languages, you have two strategies:

Approach Pros Cons
One multilingual model Simple deployment, one model for 100+ languages Larger (50-100MB), slightly lower per-language accuracy
Per-language specialist Smaller (15-25MB each), better accuracy per language Multiple downloads to manage, storage adds up
Start with a multilingual model (like multilingual MobileBERT) for broad coverage. If a specific language underperforms, add a specialist model for that language and download it on demand. This gives you instant global coverage with targeted improvements where needed.
