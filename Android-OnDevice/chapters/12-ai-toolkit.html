The On-Device AI Decision Framework

Given a new feature requirement, here's how to pick the right framework:

Question Answer → Framework
Is it OCR, barcode scanning, translation, or entity extraction? ML Kit (Chapter 5) — done in an hour, production-ready
Is it real-time face, hand, pose, or object tracking? MediaPipe (Chapter 4) — optimized pipelines, done in an afternoon
Is it text generation, summarization, or rewriting? Gemini Nano (Chapter 6) — on supported devices, cloud fallback for rest
Do you need a custom model trained on your own data? TFLite (Chapters 3, 8, 9) — maximum flexibility
Does the model come from PyTorch and not convert to TFLite? ONNX Runtime (Chapter 8) — PyTorch native, cross-platform

The decision flow in practice:

Feature Requirement → ML Kit covers it? → MediaPipe covers it? → Generative task? → Custom TFLite/ONNX

Always start from the left. The further right you go, the more work you take on. Most features can be built with ML Kit or MediaPipe — only reach for raw TFLite when you genuinely need a custom model.

Building a Reusable Inference Wrapper

Every on-device AI feature you build follows the same pattern. Abstract it into a reusable class:

abstract class OnDeviceModel<I, O>(
    private val context: Context,
    private val modelPath: String
) {
    private var interpreter: Interpreter? = null
    private val delegateSelector = DelegateSelector(context)

    fun initialize() {
        val model = FileUtil.loadMappedFile(context, modelPath)
        interpreter = delegateSelector.getBestInterpreter(model)
        // Warm up
        val dummy = createDummyInput()
        repeat(3) { runInference(dummy) }
    }

    fun predict(input: I): Result<O> {
        val interp = interpreter ?: return Result.failure(
            IllegalStateException("Model not initialized")
        )
        return try {
            val startTime = System.nanoTime()
            val preprocessed = preprocess(input)
            val rawOutput = runRawInference(interp, preprocessed)
            val result = postprocess(rawOutput)
            val latencyMs = (System.nanoTime() - startTime) / 1_000_000
            logMetrics(latencyMs, true)
            Result.success(result)
        } catch (e: Exception) {
            logMetrics(0, false)
            Result.failure(e)
        }
    }

    fun release() {
        interpreter?.close()
        interpreter = null
    }

    abstract fun preprocess(input: I): Any
    abstract fun postprocess(output: Any): O
    abstract fun createDummyInput(): I
    abstract fun runRawInference(interpreter: Interpreter, input: Any): Any
}

Now every new model becomes a simple subclass:

class ImageClassifierModel(context: Context) : OnDeviceModel<Bitmap, Classification>(

The Model Evaluation Checklist

Before shipping any on-device AI feature, measure these six metrics:

The six metrics to measure before shipping:

Metric What to Measure Target How to Measure
Accuracy Does it get the right answer? > 90% for your domain Test set of 200+ labeled examples
Latency Fast enough for real-time? p95 < 33ms (30fps) 100 runs, report p50/p95/p99
Model Size Fits the APK budget? < 10MB bundled, < 50MB downloaded File size after quantization
Memory Runs without OOM? < 50MB runtime on mid-range Android Profiler during inference
Power Drains battery? < 5% per hour of continuous use Battery Historian during sustained test
Device Coverage What % of users can run it? > 80% of your user base Play Console device catalog + RAM/API checks
A model that's 95% accurate but only runs on 30% of devices is worse than a model that's 88% accurate and runs on 95% of devices. Always evaluate accuracy and coverage together. The quantized, smaller model almost always wins for mobile.

Multi-Model Architectures

Advanced features chain multiple models together in a pipeline. Each model handles one focused task:

Example: emotion-aware response pipeline

Camera Frame → Face Detector → Crop Face → Emotion Classifier → Gemini Nano Response 
class EmotionResponsePipeline(context: Context) {
    private val faceDetector = MediaPipeFaceDetector(context)
    private val emotionClassifier = EmotionModel(context) // custom TFLite
    private val textGenerator = GeminiNanoWrapper(context)

    suspend fun process(frame: Bitmap): String? {
        // Step 1: Detect face
        val faces = faceDetector.detect(frame)
        val face = faces.firstOrNull() ?: return null

        // Step 2: Crop and classify emotion
        val cropped = cropFace(frame, face.boundingBox)
        val emotion = emotionClassifier.predict(cropped).getOrNull() ?: return null

        // Step 3: Generate contextual response
        return if (emotion.label == "sad" && emotion.score > 0.7f) {
            textGenerator.generate("Write a short encouraging message")
        } else null
    }
}
Memory budget for multi-model pipelines: Each loaded model consumes 15-80MB of RAM. On a 4GB device with 2GB available, you can comfortably run 2-3 small models simultaneously. Load models on demand and release them when the pipeline step completes. Never keep all models loaded at once unless you're on a flagship device.

Edge Cases and Failure Modes

Real-world inputs are messy. Your model trained on clean, well-lit, centered images will encounter:

Handling edge cases well

Low confidence → show "Unable to identify" Unknown object → don't force a wrong label Blurry input → prompt user to hold steady No face detected → hide the face feature quietly Model timeout → show cached last-known result

Graceful degradation. The feature works great when conditions are good and fails silently when they're not.

Handling edge cases poorly

Low confidence → show the guess anyway (wrong 40% of the time) Unknown object → label it as the closest known class Blurry input → show flickering results every frame No face → crash with NPE Model timeout → freeze the UI

Users lose trust after one wrong confident answer. It's better to say nothing than to say the wrong thing confidently.

The Capstone Architecture

Here's how all the pieces from this course combine in a single app — a "smart assistant" that sees, reads, listens, and generates:

Capstone: Smart Document Assistant

  • Vision (Chapter 9) — Camera detects documents via object detection, auto-crops to document bounds
  • OCR (Chapter 5) — ML Kit extracts all text from the document image
  • Classification (Chapter 10) — MobileBERT classifies the document type (receipt, invoice, contract, letter)
  • Entity extraction (Chapter 5) — ML Kit pulls out dates, amounts, phone numbers, addresses
  • Summarization (Chapter 6) — Gemini Nano generates a 3-bullet summary of the document
  • Voice (Chapter 10) — Keyword spotter activates on "Hey Scanner" for hands-free operation
  • Production (Chapter 11) — Device capability detection, delegate selection, error handling, telemetry

All running on-device. No cloud. No API keys. No per-request cost. The user points their camera at a document and gets a classified, extracted, summarized result in under 2 seconds.

Capstone resource budget on a Pixel 9 Pro:

  • Object detection (SSD-MobileNet int8) — 4.4MB model, ~6ms, ~20MB RAM
  • Text recognition (ML Kit) — system-managed, ~35ms, ~30MB RAM
  • Text classification (MobileBERT int8) — 6MB model, ~8ms, ~35MB RAM
  • Entity extraction (ML Kit) — system-managed, ~15ms, ~10MB RAM
  • Summarization (Gemini Nano) — system-managed, ~3sec, ~shared with OS
  • Keyword spotter — 0.8MB model, ~1ms, ~5MB RAM
  • Total pipeline — ~11MB bundled models, ~100MB peak RAM, ~2sec end-to-end

On mid-range devices: drop summarization (no Gemini Nano), use CPU delegates, and accept slightly higher latency. On budget devices: use ML Kit OCR only with entity extraction — still a useful feature.

What's Coming Next

The on-device AI landscape is moving fast. Here's what's on the horizon:

  • On-device multimodal models — image + text input on a single model (Gemini Nano is heading here). Point your camera at something and ask "what is this?" in natural language.
  • LoRA adapters for personalization — small adapter layers that customize a base model to individual users, without retraining the full model. Your keyboard's autocomplete learns your writing style.
  • Federated learning — improve models across millions of devices without collecting any data centrally. Each device trains locally, shares only anonymized gradients.
  • Always-on ambient AI — models running continuously in the background, understanding context from sensors, audio, and camera without user interaction.
  • On-device fine-tuning — adapting models directly on the phone using the user's own data. A photo classifier that learns to recognize your pet specifically.

Cross-Platform Considerations

If your team also builds for iOS, good news: most of what you've learned transfers directly.

Technology Android iOS Shared?
TFLite Native support TFLite Swift/ObjC Same .tflite model files
ONNX Runtime Android SDK iOS SDK Same .onnx model files
MediaPipe Tasks API Tasks API Same .task model bundles
ML Kit Full support Full support Same features, similar APIs
Hardware accel. NNAPI / GPU delegate Core ML / Metal Different APIs, same concept
Share models, not code. The .tflite model file is identical on both platforms. The preprocessing/postprocessing code differs (Kotlin vs Swift). If using Kotlin Multiplatform, you can share the preprocessing logic — tensor manipulation, normalization, label mapping — in common code.

Your 30-Day Action Plan

From Reading to Shipping in 30 Days

  • Week 1: Prototype — Pick one feature from Chapter 1's exercise (the 3 features you identified). Build it with ML Kit or MediaPipe. Get it running on your phone. Don't optimize yet.
  • Week 2: Optimize — Apply int8 quantization (Chapter 8). Add hardware acceleration (Chapter 7). Benchmark on 2-3 devices. Get inference under 15ms on your target device.
  • Week 3: Harden — Add device capability detection, error handling, confidence thresholds, and graceful degradation (Chapter 11). Set up Firebase Analytics for performance telemetry.
  • Week 4: Ship — Deploy to beta users. Monitor metrics for one week. Fix any device-specific issues. Roll out to all users. Celebrate.

Four weeks from "I've never done on-device AI" to "I shipped an AI feature to production." You have all the knowledge. Now execute.

Try It: Design Your On-Device AI Feature

Design a complete on-device AI feature for your own app. Write a technical design document covering:

  1. Problem statement — what user problem does this feature solve? Why on-device instead of cloud?
  2. Model selection — which framework (ML Kit / MediaPipe / TFLite / Gemini Nano)? Which specific model? What's the rationale?
  3. Optimization strategy — quantization type, target model size, target inference time
  4. Hardware acceleration plan — which delegates? Fallback chain? Benchmark targets?
  5. Fallback behavior — what happens on unsupported devices? Cloud fallback? Feature disabled? Simplified version?
  6. Production monitoring — which metrics to track? Alert thresholds? A/B testing plan?
  7. Privacy impact — what data is processed? Does it stay on-device? GDPR implications?
  8. 30-day timeline — week-by-week plan following the action plan above

This is the document you'd present to your team before starting implementation. It forces you to think through every decision from model selection to production monitoring — exactly the skills this course taught you.
