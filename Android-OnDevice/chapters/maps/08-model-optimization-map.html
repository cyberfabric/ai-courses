The Big Idea

Quantization is like JPEG compression for models — the "knowledge" stays almost the same, but the file is 2-4x smaller and 2-4x faster. Int8 quantization solves 90% of "model too big" problems in 5 minutes of work. Combine it with hardware acceleration (Ch. 7) for 10x total speedup.

📏 Three Types of Post-Training Quantization

Type What It Quantizes Size Speed Accuracy Loss Data Needed
Dynamic Range Weights only ~2x smaller ~1.5-2x CPU < 1% None
Full Int8 ⭐ Weights + activations ~4x smaller ~2-4x CPU, 5-10x NPU 1-2% 100-500 samples
Float16 Precision halved ~2x smaller ~1.5-2x GPU < 0.5% None

⭐ Full int8 is the production standard. NPUs are designed for int8 — a float32 model on NNAPI may fall back to CPU, while int8 runs entirely on NPU.

🛠️ Beyond Quantization

✂️
Structured Pruning
Remove entire channels/layers. Model is genuinely smaller — fewer ops. Works on any hardware. 30-50% channels removed.
🕸️
Unstructured Pruning
Zero individual weights (70-90% sparse). Looks impressive but most mobile NPUs can't exploit sparsity. Avoid for mobile.
🎓
Knowledge Distillation
Train a small "student" to mimic a big "teacher." Most powerful technique, but needs full training pipeline + dataset.
🔬
QAT (Quantization-Aware Training)
Simulate quantization during training so model compensates. Use only when PTQ drops accuracy too much (>2-3%).

🌳 Pruning is like removing dead branches from a tree — same fruit, lighter tree. Distillation is like a student learning from a master — smaller person, same knowledge.

🌳 The Optimization Decision Tree

  1. Start: Int8 Post-Training Quantization

    4x smaller, 2-4x faster, 5 minutes of work. If accuracy loss < 2%, you're done. (Solves 90% of cases.)

  2. Accuracy too low? Try Float16

    2x smaller, almost zero accuracy loss. Pair with GPU delegate. No representative data needed.

  3. Still too large? Structured Pruning

    Remove 30-50% of channels, then quantize the pruned model. Genuinely fewer operations.

  4. Still not enough? Knowledge Distillation

    Train a smaller architecture to match the big model. Requires training pipeline, dataset, and Python ML skills.

  5. Nothing works? Pick a Different Base Model

    Switch EfficientNet → MobileNet, or generic → domain-specific. Sometimes the right model beats the right optimization.

⚖️ TFLite vs ONNX Runtime Mobile

Use TFLite when
Model is from TensorFlow/Keras, you want widest NNAPI/GPU/QNN support, or you use Google's ecosystem (ML Kit, MediaPipe).
Use ONNX when
Model is from PyTorch and won't convert cleanly, you need cross-platform (iOS too), or the model uses ops TFLite doesn't support.
Both support
NNAPI hardware acceleration, quantized models, and all Android devices back to API 21.

🧪 Quick Quiz

What's the analogy for quantization?

JPEG compression for models. A RAW photo and JPEG look almost identical, but JPEG is 10x smaller. Quantization does the same — the model's "knowledge" is almost the same, but the file is 2-4x smaller and inference is 2-4x faster.

Which quantization type should you try first?

Full int8 post-training quantization. It's the production standard: 4x smaller, 2-4x faster on CPU, 5-10x faster on NPU, and only needs 100-500 representative samples. It solves 90% of "model too big" problems.

Why is int8 better than float32 on NPU, but worse on GPU?

NPUs are purpose-built for int8 tensor operations — they run at maximum throughput. GPUs prefer float32/float16 math. Use int8 for NPU/CPU, float16 for GPU delegate.

When should you use QAT instead of post-training quantization?

Only when PTQ drops accuracy more than 2-3%. QAT requires the original training pipeline and dataset. Try PTQ first — it works well for most models.

Why is structured pruning better than unstructured for mobile?

Structured pruning removes entire channels — the model is physically smaller with fewer ops. Unstructured pruning just zeros scattered weights (sparse model, same shape). Most mobile NPUs can't exploit sparsity, so unstructured pruning gives no real speedup.

📌 Key Takeaways

  • Quantization = JPEG for models. Int8 gives 4x smaller, 2-4x faster, ~1-2% accuracy loss. Try this first — it solves 90% of cases.
  • Three PTQ types: dynamic range (easy, 2x), full int8 (production standard, 4x), float16 (safe, GPU-friendly, 2x).
  • Int8 + NPU = best combo. NPUs are built for int8. Float32 on NNAPI may fall back to CPU; int8 stays on NPU.
  • Decision tree: int8 PTQ → float16 → structured pruning → distillation → different model.
  • Structured pruning removes real channels (fewer ops). Unstructured pruning creates sparsity mobile hardware can't exploit.
  • Knowledge distillation is the most powerful but needs full training pipeline. Reserve for when nothing else works.
  • Optimization + acceleration combined: 14 MB/25 ms → 3.5 MB/2.5 ms. 4x smaller, 10x faster.
  • TFLite for TensorFlow models + Google ecosystem. ONNX for PyTorch models + cross-platform needs.
