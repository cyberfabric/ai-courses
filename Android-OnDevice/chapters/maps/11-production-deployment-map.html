Model Management and Production Deployment

Memory Map

The Big Idea

"It works on my Pixel" is not a shipping strategy. 24,000+ Android device configurations means you need device capability detection, graceful degradation, error handling, and telemetry. On-device AI is also a privacy superpower — data never leaves the device, simplifying GDPR/CCPA compliance.

📱 The Fragmentation Reality

📊
24,000+ Devices
Distinct Android device models in the wild. Your Pixel 9 test phone is not representative.
🔧
8+ Chipset Families
Tensor, Snapdragon, Exynos, Dimensity, Kirin — each with different NPU/GPU capabilities.
💾
512MB–16GB RAM
Budget phones can't hold a 50MB model in memory. Flagship phones can run multiple models.
🎚️
4 Device Tiers
Flagship (8GB+, NPU), Mid-range (4-8GB, GPU), Budget (2-4GB, CPU), Unsupported (<2GB, disable).

🧠 Model Memory Lifecycle

  1. 📱 Screen Opens (onResume)

    Load model lazily. Don't preload at app startup — only when the feature screen is visible.

  2. 🔄 Feature Active

    Run inference. Track reference count if model is shared across screens.

  3. 👋 Screen Exits (onStop)

    Call interpreter.close() and null the reference. Free the RAM immediately.

  4. ⚠️ System Pressure (onTrimMemory)

    If level ≥ TRIM_MEMORY_MODERATE, unload all models via ModelRegistry.unloadAll().

Mantra: load lazily, unload eagerly.

📊 Production Metrics to Track

⏱️
Inference Latency
p50 and p95 per device tier. Alert if p95 > 100ms for real-time features.
✅
Load Success Rate
% of users who loaded the model successfully. Alert if < 95%.
💥
Crash Rate
Crashes per 1K inferences. Alert if > 0.1%. Always catch exceptions.
⬇️
Download Completion
% of model downloads that finish. Alert if < 90%. Offer Wi-Fi-only option.
🎯
Low-Confidence Rate
% of inferences below your quality threshold. Alert if > 30% — model may need retraining.
🔧
Delegate Distribution
% users on NPU / GPU / CPU. Watch for unexpected shifts after model updates.

📖 Key Terms

Firebase ML Model Downloader
Hosts TFLite models on Google's infra. Supports OTA updates, background downloads, Wi-Fi-only conditions, and automatic caching.
A/B Testing (Models)
Run two model versions on different user segments via Remote Config. Compare accuracy, latency, and engagement — not just offline metrics.
Device Tier
Classification based on RAM + NPU availability. Flagship/Mid/Budget/Unsupported determines which model variant and delegate to use.
Graceful Degradation
When AI can't run (OOM, unsupported device, download failure), show a meaningful fallback UI instead of crashing or showing garbage.
onTrimMemory
Android callback when system needs RAM. At TRIM_MEMORY_MODERATE or higher, unload all models to prevent being killed.
Confidence Threshold
Minimum score (e.g., 0.3f) below which results are hidden. Prevents showing unreliable predictions to users.
Play Asset Delivery
Google Play mechanism for delivering large assets (100MB+) separately from the base APK. Install-time, fast-follow, or on-demand modes.
Semantic Versioning (Models)
Version models like code (v1.2.3). Track which version runs on which devices. Enable rollback when a new model performs worse.

🧪 Quick Quiz

When should you bundle a model in the APK vs download it separately?

Bundle in APK when the model is <10MB, needed on first launch, rarely changes, and used by all users (e.g., MobileNet, keyword spotter). Download separately when the model is >10MB, optional, or updates frequently (e.g., translation models, speech models).

Why is "load lazily, unload eagerly" the right memory strategy?

Models consume significant RAM (25-100MB). Loading at startup wastes memory for users who may never use the AI feature. Loading on screen enter and unloading on screen exit keeps RAM free for other app functions. Handling onTrimMemory prevents the system from killing your app.

What should you do when inference returns low-confidence results?

Never show garbage to users. If confidence is below your threshold (e.g., 0.3f), show a graceful fallback like "Unable to identify" instead of a confident wrong answer. Users lose trust in AI features after a single bad experience — it's better to say nothing than be wrong.

Why is on-device AI a GDPR/CCPA advantage?

On-device processing means data never leaves the user's phone — no data transmission, no server storage, no data collection. This typically doesn't require explicit consent the way server-side processing does, and eliminates data breach risk entirely. It's a genuine privacy feature worth communicating.

What's the right way to A/B test model versions?

Use Firebase Remote Config to assign model variants to user segments. Don't just measure offline accuracy — track real user engagement: feature usage, corrections, satisfaction. A model that's 2% more accurate but 3x slower might lose because users stop using the feature.

📌 Key Takeaways

  • Detect device capabilities at runtime: check RAM, NPU, API level. Assign a tier (Flagship/Mid/Budget/Unsupported) and load the right model variant.
  • Four packaging options: bundled (<10MB), download on first use (10-100MB), Firebase ML (OTA updates), Play Asset Delivery (100MB+).
  • OTA updates via Firebase ML: push improved models without app releases. Gradual rollout: 5% → monitor → 100%.
  • Load lazily, unload eagerly: load on onResume, close on onStop, release all on onTrimMemory. Never preload at app startup.
  • Never show garbage: confidence threshold + graceful degradation. "Unable to identify" beats a confident wrong answer.
  • Track 6 metrics: inference latency (p50/p95), load success rate, crash rate, download completion, delegate distribution, low-confidence rate.
  • Privacy superpower: on-device = no data leaves phone = simpler GDPR/CCPA = no breach risk. Communicate it to users.
