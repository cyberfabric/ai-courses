Chapter 7

Hardware Acceleration — NPU, GPU, and DSP

Memory Map
💡

The Big Idea

Your phone has 4 processors — you've been using the slowest one. The CPU is a Swiss Army knife, the GPU is a chainsaw, the NPU is a laser cutter. For ML's repetitive tensor math, the laser cutter wins. Same model, same accuracy, 5-20x faster — just switch the delegate.

🏛️ The Hardware Zoo

🔧
CPU
General purpose, always available, slowest for ML. 4-8 cores. Your safety net fallback.
⚡
GPU
Hundreds of parallel cores. 2-5x faster than CPU. Available on every device. Best with float32 models.
🧠
NPU / TPU
Purpose-built for tensor ops. 5-20x faster, minimal power. Found in 2022+ flagships.
🎵
DSP
Signal processing specialist. Very power-efficient. Best for always-on audio (keyword spotting). Qualcomm Hexagon.

🔌 The Three Delegates

GPU Delegate
Easiest to add. Works on every device. Two lines of code: GpuDelegate() + addDelegate(). Best with float32 models — int8 may run slower on GPU.
NNAPI Delegate
Android's hardware abstraction. Routes to the best available hardware automatically (NPU, GPU, or DSP). Behavior varies wildly across devices — always benchmark.
QNN Delegate (Qualcomm)
Direct Hexagon NPU access on Snapdragon chips. Bypasses NNAPI overhead. Maximum performance on the most common mobile chipset.

🛡️ The Fallback Chain Pattern

  1. Try NPU/TPU (via NNAPI or QNN)

    Fastest option. May fail on budget devices or with unsupported ops. Wrap in try/catch.

  2. Try GPU Delegate

    2-5x speedup, works on almost every device. Float32 models preferred. Catches most NPU failures.

  3. Fall Back to CPU

    Always works, every device, every model. Your safety net. Set numThreads = 4.

  4. Cache the Winner

    Benchmark on first run, save best delegate to SharedPreferences. Don't re-benchmark every launch.

Rule: Never let a delegate failure crash your app. CPU always works.

📏 Benchmarking Done Right

🔥
Warm-Up First
First 3-5 inferences are slow (shader compilation, NPU init). Discard them before measuring.
📊
Percentiles, Not Averages
Report p50 (typical), p95 (spikes), p99 (worst). Averages hide 200ms GC stutters behind 3ms medians.
🌡️
Test Thermal Throttling
Run 60+ seconds sustained. Most devices throttle after 10-20s — inference times jump 20-50%.
📱
3+ Device Tiers
Test flagship, mid-range, and budget. Your dev Pixel doesn't represent your user base.
💾
Measure Memory Too
Latency isn't everything. GPU delegate uses extra VRAM. NPU may buffer data. Watch total consumption.
📝
Log Which Delegate Won
Telemetry in production: which delegate, what latency, which device. Invaluable for understanding your real user mix.

⚠️ Key Gotchas

GPU + int8 = slower
GPU prefers float32. Quantized int8 models are optimized for CPU/NPU and may actually slow down on GPU.
NNAPI varies wildly
Same call may hit a world-class NPU on Pixel, a mediocre GPU driver on Xiaomi, or fail entirely on an older device.
First inference is always slow
GPU compiles shaders, NPU initializes pipeline. Run 3-5 warm-ups before showing results to users.
Thermal throttling is real
After 10-20s of sustained NPU/GPU usage, speeds drop 20-50%. Measure sustained performance, not burst.
setUseNnapiCpu(false)
Without this, NNAPI may silently fall back to its own CPU implementation — slower than TFLite's CPU. Disable it.

🧪 Quick Quiz

What's the analogy for CPU, GPU, and NPU?

CPU = Swiss Army knife (general purpose, okay at everything). GPU = chainsaw (massively parallel, great for big cuts). NPU = laser cutter (purpose-built for precise, repetitive tensor operations). For ML's repetitive math, the laser cutter wins.

Why might a quantized int8 model run slower on GPU than CPU?

GPUs prefer float32 math — their parallel cores are optimized for floating point operations. Int8 quantized models are designed for CPU/NPU efficiency. Use GPU delegate with float32 models; save int8 for CPU/NPU.

Why should you report p50/p95/p99 instead of averages?

Averages hide spikes. If 99 inferences take 3ms and one takes 200ms (thermal throttle, GC pause), the average says "5ms" but the user experienced a 200ms stutter. Percentiles reveal the real experience.

What's the fallback chain and why is it essential?

NPU/TPU → GPU → CPU. Each delegate may fail (unsupported ops, driver bugs, missing hardware). CPU always works on every device for every model. Wrap each delegate init in try/catch. Never let a delegate failure crash the app.

Why should you benchmark on first run and cache the result?

You can't predict which delegate is fastest on a given device — NNAPI varies wildly across manufacturers. Benchmark all three on first launch, pick the winner, save to SharedPreferences. Don't re-benchmark every session.

📌 Key Takeaways

  • 4 processors in your phone: CPU (slowest for ML), GPU (2-5x faster), NPU/TPU (5-20x faster), DSP (power-efficient audio).
  • GPU delegate: 2 lines of code, works everywhere, best with float32. Don't use int8 models on GPU.
  • NNAPI: auto-routes to best hardware, but behavior varies wildly by device. Always benchmark.
  • Fallback chain: NPU → GPU → CPU. Wrap each in try/catch. CPU is your safety net — it always works.
  • DelegateSelector: benchmark all delegates on first run, cache the winner in SharedPreferences.
  • Benchmark correctly: 5 warm-up runs, report p50/p95/p99 percentiles, test sustained (60s+), test 3+ device tiers.
  • Thermal throttling drops performance 20-50% after 10-20s. Measure sustained speed, not burst.
  • Log which delegate is used in production — understand your real device mix.
