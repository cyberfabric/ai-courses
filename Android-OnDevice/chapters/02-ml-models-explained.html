How ML Models Work

Just enough theory to be dangerous — no PhD required

What Is a Model, Really?

In Chapter 1 we said on-device AI means running models on the phone. But what is a model?

Here's the analogy that will save you: a model is a compiled function you didn't write. It takes an input (an image, a sentence, an audio clip) and returns an output (a label, a bounding box, a transcription). You can call it, but you can't read its source code. The "source code" is millions of numerical weights that were learned during training.

In Android terms, think of it like a .so native library. You didn't write the C++ inside it. You don't need to understand every line. You just know: "I call this function with these arguments, and I get this result back." A model file works the same way — except the "function" was learned from data, not written by a programmer.

What a model IS

A file (2-50MB) containing millions of numerical weights. You load it, feed it input, and read its output. It's a black-box function: input → numbers → output. You don't modify it at runtime — it's read-only.

What a model is NOT

It's NOT code you write or compile. It's NOT a rules engine or if/else chain. It's NOT a database you query. It does NOT "think" — it multiplies matrices, very fast.

The Inference Loop — The Pattern Behind Every Feature

Every on-device AI feature you build in this course follows the exact same five-step pattern. Learn it once, use it forever:

The inference loop — the core pattern for every on-device AI feature:

1. Load Model → 2. Preprocess Input → 3. Run Inference → 4. Postprocess Output → 5. Display Result

Load the model file into memory once (expensive). Preprocess your raw input (resize image, tokenize text) into the format the model expects. Run inference — the model crunches numbers and produces raw output. Postprocess the output into something meaningful (label + confidence). Display the result to the user.

Steps 1 and 5 are Android work you already know (asset loading, Compose UI). Steps 3 is handled by the ML runtime. Your new skills are steps 2 and 4 — preprocessing and postprocessing — and that's what most of this chapter is about.

Model Formats You'll Encounter

Just like images come in PNG, JPEG, and WebP, ML models come in different formats. Here are the ones that matter on Android:

Format Extension Origin When You'll Use It
TensorFlow Lite .tflite Google / TensorFlow Most common on Android. Chapters 3, 7-11. Best ecosystem support, widest hardware acceleration.
MediaPipe Task Bundle .task Google / MediaPipe Pre-packaged TFLite model + metadata + preprocessing config. Chapters 4, 9. Easiest to use.
ONNX .onnx Microsoft / Linux Foundation Models from PyTorch, scikit-learn, or cross-platform projects. Chapter 8. Good iOS compatibility.
PyTorch Mobile .ptl Meta / PyTorch Direct PyTorch deployment. Less common on Android than TFLite — usually converted to TFLite or ONNX first.
Default to TFLite. When you have a choice, use .tflite format on Android. It has the best hardware acceleration support (NNAPI, GPU delegate), the most mature tooling, and the largest collection of pre-optimized mobile models. You'll only reach for ONNX when a model doesn't convert cleanly to TFLite.

Tensors Explained for Android Developers

The word "tensor" sounds intimidating, but you already work with them every day. A tensor is just a multi-dimensional array. That's it.

You know IntArray — a 1D tensor. You know Array<IntArray> — a 2D tensor. A Bitmap? That's a 3D tensor: height × width × color channels (RGBA).

Android Concept Tensor Shape What It Means
A single number (confidence score) [1] — scalar 0 dimensions, one value
A list of class probabilities [1000] — 1D 1,000 numbers, one per ImageNet class
A grayscale image [224, 224] — 2D 224 rows, 224 columns of pixel values
An RGB image [224, 224, 3] — 3D 224×224 pixels, 3 color channels
A batch of RGB images [1, 224, 224, 3] — 4D 1 image of 224×224×3 — the standard model input shape

That last row — [1, 224, 224, 3] — is the shape you'll see most often. It means: "I'm giving you a batch of 1 image, sized 224×224 pixels, with 3 color channels (RGB)." Nearly every image classification model expects exactly this shape (sometimes with 320 or 512 instead of 224).

Shape mismatches are the #1 beginner mistake. If a model expects input shape [1, 224, 224, 3] and you feed it a [1, 640, 480, 3] tensor (your raw camera frame), it will crash or produce garbage. Always check the model's expected input shape and resize your data to match.

Input Preprocessing — Why You Can't Just Throw a Bitmap at a Model

Your camera gives you a 12-megapixel Bitmap. The model expects a tiny 224×224 float tensor with values between 0.0 and 1.0. You need to bridge that gap. This is preprocessing.

Here are the three preprocessing steps you'll do for almost every image model:

Step 1: Resize to the Expected Dimensions

Most mobile vision models expect small, square inputs — 224×224, 320×320, or 512×512. You need to resize (and potentially crop) your camera frame to match.

// Resize a Bitmap to the model's expected size
val resized = Bitmap.createScaledBitmap(
    originalBitmap,
    224,  // model expects 224 wide
    224,  // model expects 224 tall
    true  // bilinear filtering
)

Step 2: Normalize Pixel Values

A Bitmap stores each color channel as an integer from 0 to 255. Most models want floats between 0.0 and 1.0 (or sometimes -1.0 to 1.0). You divide by 255:

// Convert pixel (0-255) to normalized float (0.0-1.0)
val normalized = pixelValue / 255.0f

// Some models use a different normalization:
// mean/std normalization (ImageNet standard)
val normalizedImageNet = (pixelValue / 255.0f - 0.485f) / 0.229f

Step 3: Arrange Channels Correctly

Android Bitmap pixels are ARGB (alpha, red, green, blue). Most models expect RGB (no alpha) or sometimes BGR. You need to extract the right channels in the right order.

Thankfully, you rarely do this by hand. TFLite's ImageProcessor class handles all three steps in a clean pipeline:

val imageProcessor = ImageProcessor.Builder()
    .add(ResizeOp(224, 224, ResizeOp.ResizeMethod.BILINEAR))
    .add(NormalizeOp(0.0f, 255.0f))  // normalize to [0, 1]
    .build()

val tensorImage = TensorImage(DataType.FLOAT32)
tensorImage.load(bitmap)
val processed = imageProcessor.process(tensorImage)
// processed is now ready for inference

Image preprocessing pipeline — what happens to your camera frame before inference:

Camera Bitmap (4032×3024) → Resize (224×224) → Normalize (0-255 → 0.0-1.0) → Float Tensor [1, 224, 224, 3]

Output Postprocessing — Making Sense of Raw Numbers

A model doesn't return "golden retriever, 92% confident." It returns an array of 1,000 floating-point numbers — one for each class it knows. These raw numbers are called logits or probabilities (depending on the model's final layer).

Your job is to interpret them:

// Model output: FloatArray of 1000 probabilities
val output = FloatArray(1000)
interpreter.run(inputBuffer, output)

// Find the index with the highest probability
val maxIndex = output.indices.maxByOrNull { output[it] } ?: 0
val confidence = output[maxIndex]

// Map index to human-readable label
val labels = loadLabels(context, "labels.txt")  // 1000 lines, one per class
val prediction = labels[maxIndex]

// Result: "golden_retriever" with 0.92 confidence
println("$prediction: ${(confidence * 100).toInt()}%")

That's postprocessing in a nutshell: find the highest number, look up what it means, and show the result. Different model types have different output formats — object detectors return bounding box coordinates, segmentation models return per-pixel masks — but the principle is always the same: raw numbers in, meaningful result out.

Model Metadata — The README Inside the Model

Good .tflite models come with embedded metadata — think of it as a README file packed inside the model. The metadata tells you:

  • Input tensor shape — what size and format the model expects (e.g., [1, 224, 224, 3])
  • Output tensor shape — what the model returns (e.g., [1, 1000] for 1,000 classes)
  • Normalization parameters — how to scale pixel values (mean=0.0, std=255.0)
  • Labels file — the list of class names the model can predict
  • Model description — what the model does, its accuracy, and limitations

You can read metadata programmatically:

val metadataExtractor = MetadataExtractor(modelBuffer)
val inputTensor = metadataExtractor.getInputTensorShape(0)
// inputTensor = [1, 224, 224, 3]

// Extract the labels file bundled inside the model
val labelsStream = metadataExtractor
    .getAssociatedFile("labels.txt")
val labels = labelsStream.bufferedReader()
    .readLines()
Always check model metadata first. Before writing preprocessing code, check what the model actually expects. A model that wants [1, 320, 320, 3] with mean/std normalization will produce garbage if you feed it [1, 224, 224, 3] with simple 0-255 normalization. The metadata is your contract with the model.

Common Model Architectures for Android

You don't need to understand how these models are built. But you do need to know what they do, because you'll be choosing between them:

M
MobileNet
Image classification — "What is this?" Takes an image, returns the most likely class label. MobileNetV3-Small (2.4MB) and MobileNetV3-Large (5.4MB) are the go-to choices. Extremely fast on mobile hardware.
S
SSD / YOLO
Object detection — "What is this, and where is it?" Returns bounding box coordinates + class labels for every detected object. SSD-MobileNetV2 (6.8MB) for speed; YOLOv8-nano (6.2MB) for better accuracy.
D
DeepLab
Image segmentation — "Which pixels belong to what?" Returns a per-pixel mask labeling every pixel as person, background, car, etc. Powers portrait mode and background replacement features.
B
MobileBERT
Text classification — "What kind of text is this?" Analyzes text for sentiment, category, intent, or spam detection. 25MB, ~12ms inference. The NLP equivalent of MobileNet.

The Size-Accuracy Tradeoff

In mobile ML, there's always a tradeoff between model size and accuracy. Smaller models run faster and use less memory, but they make more mistakes. Here's the real data:

Image classification models — size vs. accuracy on ImageNet (Pixel 8 Pro, NNAPI):

Model Size Top-1 Accuracy Inference Time Memory Usage
MobileNetV3-Small 2.4 MB 67.5% ~3 ms ~15 MB
MobileNetV3-Large 5.4 MB 75.2% ~6 ms ~25 MB
EfficientNet-Lite0 4.7 MB 75.1% ~7 ms ~30 MB
EfficientNet-Lite4 13 MB 80.4% ~20 ms ~55 MB
EfficientNet-B7 66 MB 84.3% ~120 ms ~250 MB

Going from 2.4MB to 66MB gets you 17 percentage points of accuracy — but costs 40x the inference time and 16x the memory. For most mobile features, MobileNetV3-Large is the sweet spot.

Remember the "good enough" principle from Chapter 1: a 5MB model fine-tuned for your specific 10-class task will outperform a 66MB general-purpose model. Pick the smallest model that meets your accuracy requirements, then fine-tune it for your domain.

Where to Find Pre-Trained Models

You don't need to train models from scratch. There are thousands of pre-trained, mobile-optimized models ready to download. Think of these as the "package registries" for ML models:

TF
TensorFlow Hub / Kaggle Models
Google's model registry. Largest collection of .tflite models optimized for mobile. Filter by "TFLite" and "mobile" to find models ready for Android. Your first stop for any vision or NLP task.
MP
MediaPipe Solutions
Pre-built ML solutions with models included. Face detection, hand tracking, pose estimation, object detection, text classification — production-ready, no model hunting required.
HF
Hugging Face
The largest open model community. Filter by "tflite" or "onnx" for mobile-compatible models. Especially strong for NLP models (text classification, sentiment, translation). Many models need conversion to TFLite.
OX
ONNX Model Zoo
Curated collection of models in ONNX format. Good for PyTorch-originated models. Use ONNX Runtime Mobile on Android or convert to TFLite.

Device compatibility for common model sizes:

  • Under 5MB models (MobileNet, MediaPipe) — run on virtually any Android device from the last 5 years, including budget phones with 2GB RAM
  • 5-25MB models (EfficientNet-Lite, MobileBERT) — comfortable on mid-range devices (4GB+ RAM). May need fallbacks on budget devices
  • 25-100MB models (larger EfficientNet, Whisper small) — flagship territory (8GB+ RAM). Consider dynamic download rather than bundling in APK
  • Gemini Nano (system-managed, ~2GB) — Pixel 8 Pro+, Samsung Galaxy S24+. The OS manages the model; your app just calls the API

Peeking Inside a Model with Netron

There's a free tool called Netron that lets you visually inspect any model file — think of it as "Android Studio's Layout Inspector, but for ML models." Open a .tflite file in Netron and you can see:

  • The input shape (what the model expects)
  • The output shape (what the model returns)
  • Every layer and operation in the model
  • The file size and number of parameters
  • Embedded metadata and labels

You can use Netron at netron.app in your browser — no install required. Just drag and drop a .tflite file. This is the fastest way to answer "what does this model expect?" when documentation is sparse.

Putting It All Together

Let's trace through a complete example end-to-end. You want to classify a photo of a dog:

End-to-End Inference Example

  1. Load model — read mobilenet_v3.tflite from assets into an Interpreter
  2. Check metadata — input: [1, 224, 224, 3], normalize: 0.0/255.0, labels: 1,000 ImageNet classes
  3. Preprocess — resize Bitmap to 224×224, normalize pixels to [0, 1], create float tensor
  4. Inference — call interpreter.run(input, output) — takes ~6ms on NPU
  5. Postprocess — find max index in output array (249), look up label ("golden_retriever"), get confidence (0.92)
  6. Display — show "Golden Retriever — 92%" in your Compose UI

That's the entire mental model. Every on-device AI feature in this course — image classification, object detection, hand tracking, text analysis, speech recognition — is a variation of this pattern. The model changes, the preprocessing changes, the output format changes, but the five-step loop stays the same.

Try It: Explore a Real Model

Download and inspect a real model — no code required yet.

  1. Go to Kaggle Models and search for "MobileNetV3 TFLite"
  2. Download the MobileNetV3-Large .tflite file
  3. Open it in Netron (netron.app — works in your browser)
  4. Find and write down:
    • The input tensor name and shape (hint: look for serving_default_input)
    • The output tensor name and shape
    • The file size
    • The number of operations in the model graph
  5. Now download MobileNetV3-Small and compare — how do the sizes and shapes differ?
