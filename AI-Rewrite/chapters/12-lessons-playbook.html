Lessons From the Rewrite: Your Playbook

Retrospective, reusable playbook, metrics, and connecting to AI-Factory

The Walkthrough Is Over

The house is built, inspected, and occupied. The monolith — that single-room shack with the plumbing in the walls and the wiring duct-taped to the ceiling — has been replaced by a 10-room house with clean architecture, fire alarms in every zone, and a building inspector's stamp of approval.

But the real value isn't the house. It's the blueprint. You've built a repeatable process for transforming any monolith into clean architecture using AI. This chapter extracts that process into a playbook you can hand to any team, for any app, on any platform.

Let's look back at what worked, what didn't, and what surprised us.

The Rewrite Retrospective

What Worked

  • Rules before code: Writing 50+ architecture rules before touching code (Ch 2) meant AI generated compliant code from the start. Without rules, every generation pass required manual fixes.
  • Dual-format rules: Review checklists for humans + .ctx files for AI. Same rules, two audiences. This prevented drift between what we review and what AI follows.
  • Bottom-up generation: Domain → data → presentation → DI → navigation. Each layer built on tested foundations. Top-down generation produces code that looks right but doesn't connect.
  • Fakes over mocks: FakeRecipeRepository in :core:testing was reused across 60+ tests. Mocks would have required 60+ separate setups. Fakes also survived every refactoring pass.
  • The MVI Contract: State + Event + Effect made ViewModel testing mechanical. Given state + event → assert new state. No ambiguity about what to test.

What Didn't Work

  • AI's first pass on cross-cutting concerns: Navigation, error handling, and Gradle conventions required 2-3 iterations. AI struggled with module boundaries and always defaulted to dumping code in :app.
  • Trusting AI for module placement: AI put error types in the wrong modules, qualifiers in the wrong packages, and routes in shared modules. Every generated file needed a module-placement review.
  • Generating tests without explicit rules: Without TEST-001/002/003 in the prompt, AI generated mock-heavy, happy-path-only tests with JUnit 4 patterns. Rules were essential for test quality.
  • One-shot migration prompts: "Migrate this Activity to Clean Architecture" produced inconsistent results. Step-by-step prompts (extract domain → build data → create presentation) were far more reliable.

What Surprised Us

  • AI excels at mechanical extraction: Pulling domain models from Activities, generating Room entities from raw SQL, creating mappers — AI does this faster and more consistently than humans.
  • AI's biggest weakness is boundaries: Module boundaries, layer boundaries, error translation boundaries. Anything that requires understanding "what belongs where" was AI's weakest area.
  • Rules compound: Each chapter's rules built on previous ones. By Chapter 9, AI had 50+ rules as context — and followed them. The investment in rules paid exponential dividends.
  • Convention plugins were underrated: A 30-line convention plugin eliminated 90% of Gradle configuration issues across 10 modules.

The Reusable Rewrite Playbook

Rule PLAY-001: The Eight-Step Rewrite

Every AI-driven rewrite follows the same eight steps, in order. Skip a step and the downstream steps produce worse output. The order isn't arbitrary — it's the dependency chain of the rewrite itself.

1

Audit & Decide (Ch 1)

AI health check on the monolith. Rewrite vs refactor decision. Define the target architecture.

2

Write Rules (Ch 2-3)

50+ architecture rules across 10 areas. Dual-format: review checklists + .ctx files. Load into CLAUDE.md.

3

Create Module Structure (Ch 4)

AI generates the multi-module Gradle structure. Convention plugins + version catalog. Empty modules that compile.

4

Extract Domain (Ch 5)

Pure Kotlin models, repository interfaces, use cases. Zero Android dependencies. The foundation.

5

Build Data Layer (Ch 6)

Room entities, DAOs, mappers, repository implementations. Error handling at the boundary.

6

Create Presentation (Ch 7-8)

MVI Contracts, ViewModels, Composables, Hilt modules. The visible layer built on tested foundations.

7

Wire Cross-Cutting (Ch 9)

Navigation, errors, network, Gradle conventions. The connective tissue between layers.

8

Test & Integrate (Ch 10-11)

128+ tests. Feature parity verification. Performance validation. Release build.

class GetRecipesUseCase @Inject constructor(private val repo: RecipeRepository) {
    suspend operator fun invoke(): Result<List<Recipe>> = repo.getRecipes()

// :data:repository — RecipeRepositoryImpl.kt (20 lines)
class RecipeRepositoryImpl @Inject constructor(private val dao: RecipeDao) : RecipeRepository {

Adapting the Playbook

PlatformWhat ChangesWhat Stays the Same
iOS (SwiftUI) Room → Core Data/SwiftData. Hilt → manual DI or Swinject. Compose → SwiftUI. Gradle → SPM/Xcode. 8-step process. Rules-first. Bottom-up. Domain extraction. Fakes over mocks. Contract testing.
Web (React/Next.js) Room → Prisma/Drizzle. Hilt → dependency injection container. Compose → React components. Module boundaries. Domain layer purity. Error hierarchy. Convention-based tooling.

Multiplatform (KMP) Platform-specific: UI layer per target. Shared: domain + data. Everything in :core:domain and :data:* is already pure Kotlin — it's KMP-ready.

Tip: The Domain Layer Is Platform-Agnostic

The domain layer from this rewrite (pure Kotlin models, use cases, Result type) works unchanged in KMP, backend, or any JVM target. That's the power of dependency inversion — the most valuable layer has zero platform dependencies.

The Blueprint Is Yours

Twelve chapters. One monolith transformed. Ten modules, 128+ tests, 50+ rules, and a playbook that works for any codebase.

The recipes app was the vehicle. The real deliverable is the process: audit → rules → structure → domain → data → presentation → DI → cross-cutting → tests → integration → ship. That process doesn't expire when this project ends. It's how you'll approach every legacy codebase from now on.

AI didn't replace the architect. It replaced the repetitive work that made rewrites take months instead of weeks. You provided the judgment — what belongs where, which rules matter, when AI's output needed correction. That judgment is what makes the rewrite succeed.
