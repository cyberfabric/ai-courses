Chapter 4

From Monolith to Multi-Module

AI creates the 10-module Gradle structure with dependency analysis

Breaking Ground

Chapters 1-3 were planning. This chapter is where you pick up the sledgehammer. You're about to go from a single app module to 10 Gradle modules — the empty rooms that will receive the migrated code over the next seven chapters.

This is the first time AI generates real project files for your rewrite. It's also the first real test of your PKG-* rules from Chapter 2. If the module structure is wrong, everything built on top of it will be wrong too.

The Goal: By the end of this chapter, you'll have 10 Gradle modules with correct dependency wiring, convention plugins, a version catalog, and a project that compiles with zero code in the new modules. The skeleton is up. The rooms are framed. Nothing moves in yet.

Step 1: The Dependency Graph Prompt

Before creating modules, you need to understand what depends on what in the monolith. AI can map this for you — but only if you give it the right prompt.

Weak Prompt
Analyze the dependencies in my Android project.
Too vague. AI will list Gradle dependencies, not internal code dependencies. You need the coupling map between your own classes.
Dependency Graph Prompt
Analyze the internal dependencies in app/src/main/java/. For each file: 1. List every other file it imports or references 2. Classify the relationship: UI→Logic, Logic→Data, Data→Framework, or Utility 3. Identify circular dependencies (A uses B, B uses A) 4. Flag "God files" that have 10+ internal dependencies Output as a dependency matrix table. Then draw a simplified graph showing the major clusters and their relationships. Target modules for context: :core:domain, :core:common, :data:local, :data:repository, :feature:recipe, :feature:books, :feature:grocery
Asks for internal coupling, not library dependencies. The classification reveals which layer each dependency crosses. Circular deps must be broken before modularization.

The dependency graph reveals the surgery plan. Files that reference each other heavily must end up in the same module or communicate through an interface in :core:domain.

Circular dependencies are the #1 blocker. If RecipeActivity imports RecipeDbHelper and RecipeDbHelper imports something from RecipeActivity, you can't just split them into different modules. The circle must be broken first — usually by extracting an interface into :core:domain.

Step 2: Generating the Module Structure

Now AI creates the actual Gradle files. This is a two-part operation: settings.gradle.kts declares the modules, and each module gets its own build.gradle.kts.

Single Module → 10 Modules

settings.gradle.kts (Before)

rootProject.name = "RecipesApp"
include(":app")

settings.gradle.kts (After)

rootProject.name = "RecipesApp"

include(":app")

// Core modules
include(":core:common")
include(":core:design-system")
include(":core:domain")
include(":core:testing")

// Data modules
include(":data:local")
include(":data:repository")

// Feature modules
include(":feature:recipe")
include(":feature:books")
include(":feature:grocery")
Module Generation Prompt
Generate the Gradle module structure for the recipes app rewrite. Create these 10 modules with their build.gradle.kts files: :app, :core:common, :core:design-system, :core:domain, :core:testing, :data:local, :data:repository, :feature:recipe, :feature:books, :feature:grocery Requirements: - Use convention plugins (not copy-pasted plugin blocks) - Version catalog (libs.versions.toml) for all dependency versions - KSP only, no KAPT - Each module's dependencies follow PKG-* rules: • :core:domain depends on NOTHING (pure Kotlin) • :core:common depends on :core:domain • :data:local depends on :core:domain (Room, no feature deps) • :data:repository depends on :core:domain and :data:local • :feature:* depends on :core:domain and :core:design-system • :app depends on everything (wires DI + navigation) • :core:testing depends on :core:domain (test utilities) Generate: settings.gradle.kts, each module's build.gradle.kts, and the convention plugin for shared Android configuration. Each module should compile but contain no source files yet.
Explicit dependency graph. Convention plugins instead of duplicated config. Version catalog. KSP-only. The output is a compilable empty project.
PKG-004 Every module's build.gradle.kts MUST declare only the dependencies listed in the approved dependency graph. AI MUST NOT add transitive shortcuts (e.g., :feature:recipe depending directly on :data:local instead of going through :core:domain interfaces).

Step 3: Convention Plugins

Without convention plugins, every build.gradle.kts is 40+ lines of duplicated configuration. AI loves to copy-paste. Convention plugins prevent that.

Convention Plugin Prompt
Create convention plugins in build-logic/ for the recipes app: 1. recipes.android.library — shared Android library config (compileSdk, minSdk, Kotlin options, test runner) 2. recipes.android.feature — extends library + adds Compose + Hilt (applies org.jetbrains.kotlin.plugin.compose; Compose Compiler is a separate Kotlin plugin since Kotlin 2.0) 3. recipes.android.application — app module config 4. recipes.kotlin.library — pure Kotlin module (for :core:domain) Use the version catalog (libs.versions.toml) for all versions. No hardcoded version strings anywhere.
Four plugins cover all module types. Pure Kotlin plugin for :core:domain ensures no Android dependency leaks. Version catalog as single source of truth.

Here's what a feature module's build file looks like with convention plugins:

// feature/recipe/build.gradle.kts
plugins {
    id("recipes.android.feature")
}

dependencies {
    implementation(project(":core:domain"))
    implementation(project(":core:design-system"))
    testImplementation(project(":core:testing"))
}

Three lines of dependencies. No duplicated Android config. No version numbers. That's the power of convention plugins — and it's exactly the kind of structure AI can generate reliably when given the right constraints.

If AI generates a build.gradle.kts with inline compileSdk = 34 or direct plugin application like id("com.android.library"), that's a GRADLE-* rule violation. Convention plugins are non-negotiable for a multi-module project.

Step 4: The Version Catalog

Every dependency version lives in one place: gradle/libs.versions.toml. AI generates this alongside the module structure.

GRADLE-001 All dependency versions MUST be defined in gradle/libs.versions.toml. No hardcoded version strings in build.gradle.kts files. No version ranges. Convention plugins reference the catalog, modules reference the catalog.

Step 5: The Dependency Graph as AI Context

The dependency graph isn't just a planning tool — it's a constraint document that AI should see on every session. Add it to your CLAUDE.md.

Dependency Graph (for CLAUDE.md)

## Module Dependency Graph
:app → [:feature:recipe, :feature:books, :feature:grocery,
        :data:repository, :data:local, :core:domain,
        :core:common, :core:design-system]

:feature:recipe → [:core:domain, :core:design-system]
:feature:books → [:core:domain, :core:design-system]
:feature:grocery → [:core:domain, :core:design-system]

:data:repository → [:core:domain, :data:local]
:data:local → [:core:domain]

:core:common → [:core:domain]
:core:design-system → [:core:domain]
:core:domain → [] (NOTHING — pure Kotlin)
:core:testing → [:core:domain]

FORBIDDEN:
- :feature:* → :feature:* (no cross-feature deps)
- :feature:* → :data:* (features use domain interfaces)
- :data:* → :feature:* (data never knows about UI)
- :core:domain → anything (must stay pure)

When this is in CLAUDE.md, AI reads it at the start of every session. Every time it generates a build.gradle.kts or an import statement, it checks against this graph. The FORBIDDEN section is the critical part — it tells AI what connections must never exist.

The dependency inversion principle is your friend here. Features don't depend on data implementations. They depend on repository interfaces in :core:domain. The implementations in :data:repository are wired by Hilt in :app. If AI tries to add implementation(project(":data:repository")) to a feature module, that's a PKG violation.

Step 6: Validating the Module Graph

AI generated the structure. Now verify it's correct before writing any migration code.

Validation Prompt
Review the multi-module Gradle structure I just generated. Check for: 1. Circular dependencies — does any module chain lead back to itself? 2. Forbidden dependencies — does any dependency violate the graph in CLAUDE.md? 3. Missing dependencies — can each module compile with its declared deps? 4. Visibility leaks — are internal classes accidentally exposed? 5. Convention plugin consistency — does every module use the right plugin? For each issue found, state the violation, the rule it breaks, and the fix.
Five-point verification. AI reviews its own work against the rules. Catches issues before they become entrenched in the project structure.

AI Validation Output (Example)

No circular dependencies detected ✓
All dependencies match approved graph ✓
:core:design-system missing dependency on :core:domain ✗ — Fix: add implementation(project(":core:domain"))
Convention plugins correctly assigned ✓
:data:local uses id("com.android.library") instead of id("recipes.android.library") ✗ — Fix: switch to convention plugin

Two issues caught before any code was written. That's the value of validation — structural problems are cheap to fix now and catastrophic to fix later.

Common compile errors and their AI-assisted fixes:

Error Cause Fix
Module not found Module declared in settings.gradle.kts but directory doesn't exist Create the directory with the correct path
Plugin not found Convention plugin not registered in build-logic Add plugin to build-logic/build.gradle.kts gradlePlugin block
Could not resolve project :core:domain Colon-separated path doesn't match directory structure Verify core/domain/ directory exists under project root
Namespace not specified Android module missing namespace in build.gradle.kts Add namespace = "com.recipes.core.domain" to android block
Compile Error Fix Prompt
I'm getting this Gradle error after generating the multi-module structure: [paste the error] The module dependency graph is: [paste from CLAUDE.md] Fix the error while maintaining compliance with PKG-* and GRADLE-* rules. Do NOT add dependencies that aren't in the approved graph.
Gives AI the error AND the constraints. Without the graph, AI might "fix" the error by adding a forbidden dependency.

Common AI Mistakes in Module Generation

Even with good rules, AI makes predictable mistakes when generating multi-module structures. Watch for these:

AI Gets Right

  • Basic module declaration in settings.gradle.kts
  • Simple dependency chains (feature → domain)
  • Version catalog structure
  • Standard plugin application
  • Package naming from conventions

AI Gets Wrong

  • Transitive dependency shortcuts (:feature → :data:local)
  • Circular deps through shared utilities
  • Wrong visibility (public instead of internal)
  • Hardcoded versions despite version catalog
  • KAPT instead of KSP for annotation processing
The sneakiest mistake: AI adds implementation(project(":data:repository")) to a feature module because "the feature needs to access recipes." It's technically correct but architecturally wrong. The feature should depend on the interface in :core:domain, not the implementation in :data:repository. Hilt wires the implementation at runtime via :app.
PKG-005 Feature modules MUST depend on :core:domain for repository interfaces, NEVER on :data:* modules directly. The concrete implementation is provided by Hilt at runtime through :app's DI wiring.

Try It: Generate Your Multi-Module Structure

Exercise: From 1 Module to N Modules

Using your project and the module plan from Chapter 1:

  1. Run the dependency graph prompt: Map internal dependencies in your monolith. Identify circular dependencies that must be broken.
  2. Generate the module structure: Use the module generation prompt with your specific module names and dependency graph.
  3. Create convention plugins: Generate build-logic/ with plugins matching your module types.
  4. Generate the version catalog: Move all dependency versions to libs.versions.toml.
  5. Validate: Run the 5-point validation prompt. Fix any issues found.
  6. Compile: Run ./gradlew assembleDebug. Fix errors using the compile error fix prompt.
  7. Add the dependency graph to CLAUDE.md: Include the FORBIDDEN section.
  8. Commit: This is your rewrite/setup branch.

Deliverable: A compiling multi-module project with empty modules, convention plugins, version catalog, and the dependency graph in CLAUDE.md.
