Chapter 1

The Rewrite Decision

When to rewrite vs refactor, and how AI changes the time/risk equation

The Renovation Analogy

You don't demolish a house because the paint is peeling. You grab a roller and fix it over a weekend. But when the plumbing runs through load-bearing walls, the electrical doesn't meet code, and every "quick fix" cracks the foundation a little more — at some point, gutting it to the studs is cheaper than patching.

Software rewrites work the same way. The question is never "should we rewrite?" in the abstract. It's: does the cost of continued patching exceed the cost of a controlled demolition and rebuild?

For most of software history, the answer was almost always "keep patching." Rewrites were expensive, risky, and famously prone to failure. Joel Spolsky called them "the single worst strategic mistake that any software company can make."

That was before AI could read your entire codebase, understand the patterns, generate the replacement code, and write the tests — all in the same afternoon.

The Core Thesis: AI doesn't just make rewrites faster. It makes them feasible for the first time — by compressing months of mechanical transformation into days of guided generation and review.

Rewrite vs Refactor: The Decision Framework

Before you commit to anything, you need a clear-eyed assessment. Here's the framework that separates "we should rewrite" from "we're just frustrated."

Signal Refactor Rewrite
Pain is localized One module is messy, the rest are fine Every module has the same structural problems
Test coverage Existing tests catch regressions Zero tests — every change is a gamble
Architecture Sound foundation, just needs cleanup No architecture — everything depends on everything
Team velocity Slowing down but still shipping Features take 3x longer than they should
New hires Can onboard in a week Need months of tribal knowledge
Tech debt ratio Isolated pockets you can chip away at Debt is the architecture
If you checked "Rewrite" for 4+ signals, you're not being dramatic. You have a structural problem that incremental refactoring won't solve — it'll just spread the pain over more sprints.

The critical distinction: refactoring improves code within an existing architecture. Rewriting replaces the architecture itself. If your problem is the architecture, refactoring is rearranging deck chairs.

How AI Changes the Calculus

The traditional rewrite was terrifying for three reasons: it took forever, it broke things you didn't expect, and the team couldn't ship features while it was happening. AI attacks all three.

Industry data on AI-assisted coding shows wide variance — from sharply faster to occasionally slower — depending on codebase size, prior test coverage, and team familiarity with AI tooling. Treat these numbers as a feasibility signal, not a delivery promise.

Here's why the math changes:

Without AI

  • Manually trace every dependency
  • Write migration code line by line
  • Write tests after the rewrite (if at all)
  • One developer per module extraction
  • Review by reading every line

With AI

  • AI maps the dependency graph in minutes
  • AI generates migration code from rules
  • AI writes tests before you touch the code
  • One developer orchestrating AI across modules
  • Review by checking against architecture rules

The biggest shift: AI can generate tests for the legacy code before you rewrite it. That means you can verify the rewrite preserves behavior — something that was prohibitively expensive before. You're not rewriting blind anymore.

The risk equation flips: with AI, the risk of not rewriting (accumulating more untested, unstructured code) becomes higher than the risk of the rewrite itself.

The 10-Module Target

Here's the architecture we're building toward. You already know these patterns — the point isn't to learn them, it's to see how AI will construct them.

Target Module Structure

:app                    ← Wires everything together, navigation host
├── :core:common        ← Shared utilities, extensions, base classes
├── :core:design-system ← Theme, shared Composables, design tokens
├── :core:domain        ← Entities, repository interfaces, use cases
├── :core:testing       ← Test utilities, fakes, shared fixtures
├── :data:local         ← Room database, entities, DAOs, migrations
├── :data:repository    ← Repository implementations, data mappers
├── :feature:recipe     ← Recipe list/detail screens, MVI contracts
├── :feature:books      ← Cookbook management screens
└── :feature:grocery    ← Grocery list generation screens

The dependency rules are strict:

  • Features depend on :core:domain and :core:design-system, never on each other
  • Data modules depend on :core:domain for interfaces, never on features
  • :core:domain depends on nothing — pure Kotlin
  • :app is the only module that sees everything — it wires DI and navigation
RULE PKG-001 Feature modules MUST NOT depend on other feature modules. All inter-feature communication goes through :core:domain interfaces, wired by :app.

This rule will be one of the first you teach your AI. Every piece of generated code will be checked against it. Get used to seeing rule IDs — by Chapter 2, you'll have a full library of them.

The AI Codebase Health Check

Before you start ripping things apart, let AI give you a diagnosis. Think of it as the home inspector's report before renovation begins.

Weak Prompt
Review my codebase and tell me what's wrong with it.
Too vague. AI will give you generic advice about "improving code quality" that you already know. No actionable output.
Effective Prompt
Audit this Android project for rewrite readiness. For each file in the app module, classify it as: 1. **God class** — Does it handle UI + business logic + data access? 2. **Coupled** — Does it directly instantiate its dependencies (no DI)? 3. **Untestable** — Does it have Android framework dependencies that prevent unit testing? 4. **Raw SQL** — Does it use SQLiteDatabase directly instead of Room? Output a table: file name, classifications, estimated migration complexity (low/medium/high), and which target module it should be split into. Here are the target modules: :core:domain, :data:local, :data:repository, :feature:recipe, :feature:books, :feature:grocery
Specific classifications. Concrete output format. Target modules provided as context. AI can produce an actionable migration map.

The output you get is your migration inventory — a file-by-file map of what needs to move where and how hard it'll be. This becomes the backbone of your entire rewrite plan.

Run this audit with your architecture rules loaded into context (CLAUDE.md or system prompt). AI will flag violations against your specific standards, not just generic best practices. We'll set up those rules in Chapter 2.

Estimating Rewrite Scope With AI

Once you have the migration inventory, AI can estimate the scope. This isn't project management theater — it's a concrete breakdown that maps to your sprint capacity.

Scope Estimation Prompt
Based on this migration inventory, estimate the effort for each target module: [paste the inventory table from the health check] For each module, estimate: - Number of files to create - Number of files to migrate from the monolith - Lines of code (approximate) - Number of tests needed for adequate coverage - Dependencies on other modules (which must be done first) - Risk level: LOW (mechanical move), MEDIUM (logic extraction), HIGH (complex refactoring) Suggest a migration order that minimizes blocked dependencies.
Uses the health check output as input. Asks for dependency ordering — critical for planning which module to extract first.

The migration order matters enormously. The typical sequence:

:core:domain
→
:data:local
→
:data:repository
→
:feature:*
→
:app

Domain first because everything depends on it. Data next because features need repositories. Features last because they consume everything below. The :app module gets cleaned up at the end — it becomes just the navigation host and DI wiring.

The Rewrite Manifesto

Before you write a single line of migration code, you need a manifesto. This is the contract between you and AI — the non-negotiable rules that every generated file must follow.

RULE RW-001 The Rewrite Manifesto: (1) No new code in the monolith module — all new code goes to target modules. (2) Every migrated file gets tests before deletion from the monolith. (3) Feature parity is non-negotiable — nothing gets cut during migration. (4) Architecture rules are loaded into every AI session — no "quick fixes" that violate the target.

The manifesto breaks into three categories:

What Changes

  • Single module → 10 modules with clear dependency rules
  • God Activities → Composables + ViewModels + Use Cases
  • Raw SQLite → Room with typed DAOs and entities
  • Manual instantiation → Hilt dependency injection
  • Zero tests → 128+ tests across all layers

What Stays

  • All existing features — every screen, every flow, every edge case
  • The database schema (Room wraps it, doesn't replace it)
  • API contracts — the server doesn't know you rewrote
  • User-facing behavior — the user shouldn't notice the rewrite

What Gets Deleted

  • God Activities (replaced by Compose screens + ViewModels)
  • Raw SQL helpers (replaced by Room DAOs)
  • Manual dependency wiring (replaced by Hilt modules)
  • Duplicated utility functions (consolidated into :core:common)
The manifesto isn't aspirational — it's a gate. Load it into your CLAUDE.md. Every AI-generated file gets checked against it. If the AI generates code in the monolith module, that's a rule violation, full stop. Chapter 3 shows how to enforce this with quality gate hooks.

Risk Assessment: What Can Go Wrong

Not all migrations carry equal risk. Here's the risk matrix for the recipes app, ordered from highest to lowest danger.

Migration Risk Why
Database → Room HIGH Data loss potential. Existing raw queries may rely on SQLite quirks that Room doesn't support.
Navigation overhaul HIGH Deep links, back stack behavior, and implicit intents all change. Regression surface is massive.
Activity → Compose MEDIUM UI parity is hard to verify. Custom views, animations, and edge-case layouts need manual review.
Domain extraction LOW Mostly mechanical: pull logic out, define interfaces. Tests verify correctness easily.
DI setup LOW Hilt setup is template-driven. AI gets this right consistently with proper rules.
The #1 rewrite killer: tackling high-risk migrations first. Start with LOW risk modules to build confidence and establish patterns. Save database migration and navigation for when you have a working test suite protecting you.

This is why the migration order matters. You build the safety net (domain + tests) before you touch the dangerous parts (database + navigation).

Try It: Your Own AI Health Check

Exercise: Run the AI Codebase Health Check

Pick a project you've been wanting to rewrite (or the messiest module in your current app). Run this sequence:

  1. Prepare context: List your project's source files, or point AI at the root directory. If it's too large, pick the worst module.
  2. Run the health check prompt from earlier in this chapter. Adapt the target modules to your app's domain.
  3. Review the output: Does the classification match your intuition? Did AI find coupling you didn't know about?
  4. Run the scope estimation prompt using the health check output. Does the migration order make sense?
  5. Write your manifesto: Three columns — what changes, what stays, what gets deleted. Keep it to one page.

Deliverable: A migration inventory table and a one-page rewrite manifesto for your app. You'll use both throughout this course.
