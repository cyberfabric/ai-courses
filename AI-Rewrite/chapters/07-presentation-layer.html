Chapter 7

The Presentation Layer: MVI With AI

ViewModels with Contract pattern, Compose screens, and the reducer pattern

Painting the Walls

Foundation poured. Plumbing installed. Now it's time for the part everyone actually sees — the walls, the paint, the fixtures. In app terms: ViewModels, state management, and Compose screens.

The presentation layer is where AI makes the most mistakes. It's also where AI saves the most time. A God Activity with 800 lines of UI logic, state management, navigation, and business rules can be decomposed into a ViewModel + Contract + Composable in minutes — if AI has the right constraints.

This chapter fills the :feature:recipe and :feature:books modules with ViewModels and Compose screens, consuming the domain use cases from Chapter 5.

The MVI Contract Pattern

Every screen in the rewritten app follows the same pattern: a Contract interface that defines the screen's State, Events, and Effects. This is the single most important constraint for AI-generated presentation code.

Rule PRES-001: Contract Pattern

Every screen MUST have a Contract interface containing: (1) a single immutable data class State, (2) a sealed interface Event for user actions, and (3) a sealed interface Effect for one-shot side effects (navigation, snackbar, etc.).

Rule PRES-002: Single State Source

Each ViewModel MUST expose exactly one StateFlow<State>. No scattered MutableStateFlows. No LiveData. The Contract's State data class is the single source of truth for everything the screen displays.

Rule PRES-003: Effects via Channel

One-shot side effects (navigation, toasts, snackbars) MUST use a Channel<Effect> collected as a Flow in the Composable. Never put one-shot events in State — they'll replay on configuration change.

The Contract Template

interface RecipeListContract {
    data class State(
        val recipes: List<Recipe> = emptyList(),
        val searchQuery: String = "",
        val isLoading: Boolean = false,
        val error: String? = null
    )

    sealed interface Event {
        data class SearchQueryChanged(val query: String) : Event
        data class RecipeClicked(val id: Long) : Event
        data class DeleteRequested(val id: Long) : Event
        data class FavoriteToggled(val id: Long) : Event
        data object RetryClicked : Event
    }

    sealed interface Effect {
        data class NavigateToDetail(val id: Long) : Effect
        data class ShowSnackbar(val message: String) : Effect
    }
}

This template goes into every AI prompt that generates a ViewModel. It constrains the output format so tightly that AI produces consistent, reviewable code every time.

After: RecipeListViewModel in :feature:recipe
class RecipeListViewModel(
    private val getRecipes: GetRecipesUseCase,
    private val searchRecipes: SearchRecipesUseCase,
    private val toggleFavorite: ToggleFavoriteUseCase,
    private val deleteRecipe: DeleteRecipeUseCase
) : ViewModel() {

    private val _state = MutableStateFlow(RecipeListContract.State())
    val state: StateFlow<RecipeListContract.State> = _state.asStateFlow()

    private val _effects = Channel<RecipeListContract.Effect>()
    val effects: Flow<RecipeListContract.Effect> = _effects.receiveAsFlow()

    init { loadRecipes() }

    fun onEvent(event: RecipeListContract.Event) {
        when (event) {
            is Event.SearchQueryChanged -> onSearchChanged(event.query)
            is Event.RecipeClicked -> viewModelScope.launch {
                _effects.send(Effect.NavigateToDetail(event.id))
            }
            is Event.DeleteRequested -> deleteRecipe(event.id)
            is Event.FavoriteToggled -> toggleFavorite(event.id)
            is Event.RetryClicked -> loadRecipes()
        }
    }

    private fun loadRecipes() {
        viewModelScope.launch {
            _state.update { it.copy(isLoading = true, error = null) }
            getRecipes().onSuccess { recipes ->
                _state.update { it.copy(recipes = recipes, isLoading = false) }
            }.onError { msg, _ ->
                _state.update { it.copy(error = msg, isLoading = false) }
            }
        }
    }
    // ... other private methods follow same pattern
}

The Reducer Pattern

For complex screens, extract state transitions into a pure function. This makes the ViewModel testable without coroutines or flows — just input state + event = output state + effects.

Reducer as Pure Function

// Pure function — no side effects, no coroutines, fully testable
object RecipeListReducer {
    data class ReduceResult(
        val state: RecipeListContract.State,
        val effects: List<RecipeListContract.Effect> = emptyList()
    )

    fun reduce(
        current: RecipeListContract.State,
        event: RecipeListContract.Event
    ): ReduceResult = when (event) {
        is Event.SearchQueryChanged -> ReduceResult(
            state = current.copy(searchQuery = event.query)
        )
        is Event.RecipeClicked -> ReduceResult(
            state = current,
            effects = listOf(Effect.NavigateToDetail(event.id))
        )
        is Event.RetryClicked -> ReduceResult(
            state = current.copy(isLoading = true, error = null)
        )
        // ... other events
    }
}

When to Use a Reducer

Simple screens (3-4 events) don't need a separate reducer — the when block inside onEvent() is fine. Extract to a reducer when: (1) the screen has 6+ events, (2) state transitions are complex, or (3) you want to test transitions without coroutines. For the recipes app, RecipeList and RecipeEdit get reducers; RecipeDetail doesn't.

After: Compose screen driven by State
@Composable
fun RecipeListScreen(
    viewModel: RecipeListViewModel = hiltViewModel(),
    onNavigateToDetail: (Long) -> Unit
) {
    val state by viewModel.state.collectAsStateWithLifecycle()

    LaunchedEffect(Unit) {
        viewModel.effects.collect { effect ->
            when (effect) {
                is Effect.NavigateToDetail -> onNavigateToDetail(effect.id)
                is Effect.ShowSnackbar -> { /* snackbar host */ }
            }
        }
    }

    Column(modifier = Modifier.fillMaxSize()) {
        SearchBar(
            query = state.searchQuery,
            onQueryChange = { viewModel.onEvent(Event.SearchQueryChanged(it)) }
        )
        when {
            state.isLoading -> LoadingIndicator()
            state.error != null -> ErrorState(
                message = state.error!!,
                onRetry = { viewModel.onEvent(Event.RetryClicked) }
            )
            state.recipes.isEmpty() -> EmptyState()
            else -> RecipeList(
                recipes = state.recipes,
                onRecipeClick = { viewModel.onEvent(Event.RecipeClicked(it)) },
                onFavoriteToggle = { viewModel.onEvent(Event.FavoriteToggled(it)) }
            )
        }
    }
}

Manual visibility toggling is gone. The when block renders exactly one state at a time. Every user action flows through onEvent(). The Composable is a pure function of State — no logic, no side effects, no database calls.

Watch: Business Logic in Composables

AI's #1 presentation mistake: putting logic in Composables. If you see if (recipes.size > 10), filtering, sorting, formatting, or any remember { calculation } that transforms data — it belongs in the ViewModel or a use case. Composables render State. Period.

State Management Patterns

AI generates several state management anti-patterns. Load these rules to prevent them.

State Update Patterns

// ✅ CORRECT: Single atomic update via copy()
_state.update { it.copy(isLoading = true, error = null) }

// ❌ WRONG: Multiple separate updates = intermediate invalid states
_state.value = _state.value.copy(isLoading = true)
_state.value = _state.value.copy(error = null)  // Renders between these!

// ✅ CORRECT: Derive display values in State data class
data class State(
    val recipes: List<Recipe> = emptyList(),
    val searchQuery: String = ""
) {
    val filteredRecipes: List<Recipe>  // Derived in State, not Composable
        get() = if (searchQuery.length >= 2)
            recipes.filter { it.title.contains(searchQuery, ignoreCase = true) }
        else recipes
}

// ❌ WRONG: Deriving in the Composable
val filtered = remember(state.recipes, state.searchQuery) {
    state.recipes.filter { ... }  // Logic in UI layer!
}

AI Review: Presentation Compliance

Presentation Layer Review Checklist

Review generated presentation code:

✅ PASS: Contract with State + Event + Effect | Single StateFlow<State>
| Channel<Effect> for one-shots | onEvent() dispatches all actions
| State updates via .copy() | No business logic in Composables
| collectAsStateWithLifecycle() | Navigation via lambda callbacks

❌ FAIL: Multiple StateFlows | LiveData | Mutable state in Composable
| Business logic in Composable | SharedFlow for effects (replays)
| Direct navigation in ViewModel | Android imports beyond ViewModel

Common AI Mistakes in the Presentation Layer

Mistake What AI Does Rule / Fix
Mutable state var isLoading by mutableStateOf() in ViewModel PRES-002: Single immutable StateFlow<State>
SharedFlow for effects Uses SharedFlow instead of Channel PRES-003: Channel prevents replay on config change
Logic in Composable Filtering, sorting, formatting in @Composable Composables render State. Logic in ViewModel.
Multiple StateFlows Separate flows for recipes, loading, error PRES-002: single State data class, single flow
Direct navigation navController.navigate() in ViewModel Effects → Composable → navigation callback
Missing lifecycle collectAsState() without lifecycle Always collectAsStateWithLifecycle()

Try It: Convert Your Screens

Exercise: From God Activities to MVI

  1. Define the Contract: For your main screen, write State (all display data), Event (all user actions), Effect (navigation + one-shots).
  2. Generate the ViewModel: Feed AI the Contract + use case signatures + PRES rules. Verify: single StateFlow, Channel for effects, onEvent() dispatches all.
  3. Extract the Reducer: If the screen has 6+ events, extract state transitions to a pure function. Test it without coroutines.
  4. Generate the Compose screen: Feed AI the Contract + State shape. Verify: collectAsStateWithLifecycle(), no logic, navigation via lambdas.
  5. Run the review checklist: Check PRES-001, PRES-002, PRES-003. No mutable state, no shared effects, no logic in Composables.
  6. Repeat: For each screen in your app. The pattern is identical — only the Contract changes.

Deliverable: Feature modules with Contract + ViewModel + Screen for every screen. All consuming domain use cases. All following the same MVI pattern.
