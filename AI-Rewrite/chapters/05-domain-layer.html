Chapter 5

Extracting the Domain Layer

Pure Kotlin models, repository interfaces, use cases, and the Result pattern

The Heart of the House

In renovation terms, the domain layer is the load-bearing structure — the beams and columns that everything else attaches to. You don't install drywall before the framing is solid. You don't wire plumbing before the walls are framed.

That's why :core:domain is the first module you fill. Everything depends on it. Nothing depends on anything else yet. It's the lowest-risk, highest-value migration target.

In Chapter 4, you built 10 empty rooms. Now you're installing the structural beams in the most important one.

Why Domain First?

:core:domain depends on nothing. Every other module depends on it. That means you can extract, test, and validate domain code in complete isolation — no Android, no Room, no Hilt, no Compose. Pure Kotlin. If you get this layer right, every subsequent extraction has a stable foundation to build on.

Code Archaeology: Finding Domain Concepts

Your monolith has domain logic. It's just buried. Business rules live inside Activities, data classes share files with UI code, and validation logic hides in onClick handlers. AI is excellent at excavating these buried concepts — if you give it the right prompt.

Good: The Code Archaeology Prompt

Analyze these legacy files to identify domain concepts for extraction:
- RecipeListActivity.kt, RecipeDetailActivity.kt, RecipeEditActivity.kt
- DatabaseHelper.kt, RecipeAdapter.kt

For each file, identify:
1. Data classes or data structures (explicit or implicit)
2. Business rules (validation, calculation, transformation)
3. Repository-like operations (CRUD, queries, filters)
4. Enums or sealed types (status, category, filter type)

Output: domain models, repository interfaces, use cases, and inline
business rules to extract. Everything must be pure Kotlin — no Android types.

Bad: Vague Extraction Prompt

Extract the domain layer from my app.

No file references, no output format, no constraints. AI will guess your architecture and probably leak Android types into domain.

The Domain Layer Rules as AI Constraints

Before AI writes a single line of domain code, load the domain rules from Chapter 2. These aren't suggestions — they're compile-time constraints for the AI.

Rule DOM-001: Pure Kotlin Only

:core:domain MUST contain only pure Kotlin. No Android imports, no Room annotations, no Hilt annotations, no Compose types. If import android.* appears anywhere in this module, the extraction has failed.

Rule DOM-002: Result Sealed Class

All repository interface methods MUST return Result<T> (custom sealed class). No raw exceptions crossing the domain boundary. No nullable returns for error cases.

Rule DOM-003: Use Case Pattern

Each use case MUST have exactly one public method: operator fun invoke(). Constructor injection for dependencies. No state. No side effects beyond the return value.

The Result Sealed Class

Before defining any repository interfaces, you need the error handling contract. The Result pattern standardizes how errors flow through the domain layer.

The Domain Result Type

// com.example.recipes.domain.model.Result
sealed class Result<out T> {
    data class Success<T>(val data: T) : Result<T>()
    data class Error(val message: String, val cause: Throwable? = null) : Result<Nothing>()
}

// Extension functions for ergonomic usage
inline fun <T> Result<T>.onSuccess(action: (T) -> Unit): Result<T> {
    if (this is Result.Success) action(data); return this
}
inline fun <T> Result<T>.onError(action: (String, Throwable?) -> Unit): Result<T> {
    if (this is Result.Error) action(message, cause); return this
}

Why custom instead of Kotlin's built-in kotlin.Result? The built-in type can't be used as a normal return type (compiler restriction). A custom sealed class gives you full control over the error model and extension functions.

Repository Interfaces: The Domain's Contract

Repository interfaces live in :core:domain. Implementations live in :data:repository. This is the dependency inversion that makes the whole architecture work — and the boundary AI gets wrong most often.

Good: Repository Interface Prompt

Generate repository interfaces for :core:domain. Operations found in monolith:
- RecipeListActivity: loadAll, search, delete, toggleFavorite
- RecipeDetailActivity: loadById, loadIngredients
- RecipeEditActivity: save, update

RULES: Interfaces ONLY (no implementations). All methods: suspend,
return Result<T>, use domain types (not Room/Android).
Package: com.example.recipes.domain.repository. Group by aggregate root.

Generated Repository Interface

// com.example.recipes.domain.repository.RecipeRepository
interface RecipeRepository {
    suspend fun getAll(): Result<List<Recipe>>
    suspend fun getById(id: Long): Result<Recipe>
    suspend fun search(query: String): Result<List<Recipe>>
    suspend fun save(recipe: Recipe): Result<Long>
    suspend fun update(recipe: Recipe): Result<Unit>
    suspend fun delete(id: Long): Result<Unit>
    suspend fun toggleFavorite(id: Long): Result<Recipe>
    suspend fun getByDifficulty(
        level: DifficultyLevel
    ): Result<List<Recipe>>
}

Watch: AI Leaking Implementation Details

AI frequently adds Room-specific methods to domain interfaces: fun getRecipeWithIngredients() (a Room @Relation concept) or fun observeAll(): Flow<List<RecipeEntity>> (Room entity type). If you see Entity, Dao, @Query, or any Android import in :core:domain, reject immediately. The domain interface defines what the app needs, not how Room provides it.

Creating Use Cases

Use cases are the verbs of your domain. Each one represents a single action the app can perform. The pattern is deliberately rigid: one class, one public method, injected dependencies.

Migration: Inline Logic → Use Cases

Before: Business logic in RecipeListActivity
// In RecipeListActivity.kt
fun onSearchTextChanged(query: String) {
    val trimmed = query.trim()
    if (trimmed.length < 2) {
        loadAllRecipes()  // Too short, show all
        return
    }
    val cursor = db.rawQuery(
        "SELECT * FROM recipes WHERE title LIKE ? OR description LIKE ?",
        arrayOf("%$trimmed%", "%$trimmed%")
    )
    val results = mutableListOf<Map<String, Any>>()
    while (cursor.moveToNext()) {
        results.add(mapOf(
            "id" to cursor.getLong(0),
            "title" to cursor.getString(1),
            // ... more fields
        ))
    }
    cursor.close()
    adapter.updateData(results)
}
After: SearchRecipesUseCase in :core:domain
// com.example.recipes.domain.usecase.SearchRecipesUseCase
class SearchRecipesUseCase(
    private val recipeRepository: RecipeRepository
) {
    suspend operator fun invoke(
        query: String
    ): Result<List<Recipe>> {
        val trimmed = query.trim()
        if (trimmed.length < 2) {
            return recipeRepository.getAll()
        }
        return recipeRepository.search(trimmed)
    }
}

The use case captured the business rule (minimum 2 characters for search) that was buried in an Activity. The database mechanics moved to the repository. The UI update moved to the presentation layer. Each layer handles exactly its concern.

Validation in the Domain

The monolith validates in the UI — checking editText.text.isNotEmpty() in an onClick handler. The clean version validates in the domain, where the rules actually belong.

Domain Validation in Use Cases

class SaveRecipeUseCase(
    private val recipeRepository: RecipeRepository
) {
    suspend operator fun invoke(
        recipe: Recipe
    ): Result<Long> {
        // Business rules live HERE, not in the UI
        if (recipe.title.isBlank()) {
            return Result.Error("Recipe title cannot be empty")
        }
        if (recipe.ingredients.isEmpty()) {
            return Result.Error("Recipe must have at least one ingredient")
        }
        if (recipe.servings < 1) {
            return Result.Error("Servings must be at least 1")
        }
        return recipeRepository.save(recipe)
    }
}

This is a pattern AI handles well — if you tell it where validation lives. Without the domain rules as context, AI will put validation in the ViewModel, the Composable, or both. With rules loaded, it consistently places validation in the use case.

Testing the Domain Layer

Domain tests are the easiest to write — everything is pure Kotlin, no Android test runners needed. AI generates these well because the inputs and outputs are completely predictable.

Good: Domain Test Generation Prompt

Generate unit tests for :core:domain use cases.

RULES: JUnit 5, fakes over mocks (FakeRecipeRepository), test behavior
not implementation. Naming: {UseCase}Test. Package: .domain.usecase (test).

For each use case test: 1) happy path (Success), 2) edge cases (empty,
zero, negative), 3) error propagation (repo Error → use case Error),
4) business rule validation (correct error message).

CONTEXT: [paste use case code + repository interface]

Generated Test: SearchRecipesUseCase

class SearchRecipesUseCaseTest {
    private val fakeRepository = FakeRecipeRepository()
    private val useCase = SearchRecipesUseCase(fakeRepository)

    @Test
    fun `search with valid query returns matching recipes`() = runTest {
        fakeRepository.addRecipe(Recipe(id = 1, title = "Pasta Carbonara", ...))
        fakeRepository.addRecipe(Recipe(id = 2, title = "Greek Salad", ...))
        val result = useCase("Pasta")
        assertIs<Result.Success<List<Recipe>>>(result)
        assertEquals(1, result.data.size)
    }

    @Test
    fun `search with short query returns all recipes`() = runTest {
        fakeRepository.addRecipe(Recipe(id = 1, title = "Pasta", ...))
        fakeRepository.addRecipe(Recipe(id = 2, title = "Salad", ...))
        val result = useCase("P")  // Less than 2 chars
        assertIs<Result.Success<List<Recipe>>>(result)
        assertEquals(2, result.data.size)  // Returns all, not filtered
    }

    @Test
    fun `search propagates repository error`() = runTest {
        fakeRepository.shouldReturnError = true
        val result = useCase("Pasta")
        assertIs<Result.Error>(result)
    }
}

The FakeRecipeRepository implements RecipeRepository with in-memory storage and lives in :core:testing. It's deterministic, fast, and requires zero Android infrastructure. For more on fakes vs mocks, see AI-Advanced Chapter 8.

Common AI Mistakes in Domain Extraction

Mistake What AI Does How to Catch & Fix
Android leak Adds import android.os.Parcelable or @Parcelize DOM-001 review: grep for android.* imports. Move Parcelable to :feature or :core:common
Anemic use cases Creates GetRecipesUseCase that just calls repository.getAll() Ask: does this use case add business logic? If it's just a pass-through, consider whether you need it
Room types in domain Uses RecipeEntity as domain model DOM-001: domain models are NOT Room entities. Mappers live in :data:repository
Missing Result wrapping Returns Recipe? instead of Result<Recipe> DOM-002: all repository methods return Result<T>. No nullable returns for errors
Stateful use cases Adds private var lastQuery = "" to a use case DOM-003: use cases are stateless. State belongs in ViewModel
Multiple public methods Adds validate() alongside invoke() DOM-003: one public method only. Validation is internal to invoke()

Try It: Extract Your Domain Layer

Exercise: From Monolith to :core:domain

  1. Run the archaeology prompt: Feed your monolith's key files to AI. Get the list of domain concepts to extract.
  2. Generate the Result type: Create your custom Result sealed class with Success/Error variants and extension functions.
  3. Generate domain models: Convert implicit data structures to explicit data classes and enums. Verify: zero Android imports.
  4. Generate repository interfaces: Define the contracts. Verify: all methods are suspend, all return Result<T>.
  5. Generate use cases: Extract business logic into single-method classes. Verify: stateless, single operator fun invoke().
  6. Run the AI review: Use the domain review checklist. Fix any violations.
  7. Generate tests: Use fakes, not mocks. Cover happy path, edge cases, and error propagation.
  8. Update the migration tracker: Mark all extracted domain concepts as MIGRATED.

Deliverable: A :core:domain module with models, repository interfaces, use cases, and tests. Zero Android dependencies. All tests passing.
