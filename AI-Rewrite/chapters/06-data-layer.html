Chapter 6

Building the Data Layer With AI

Room entities, DAOs, mappers, and repository implementations

Running the Plumbing

The foundation is poured. In Chapter 5, you defined what the app needs — domain models, repository interfaces, use cases. Now it's time to define how the data actually flows.

In renovation terms, the data layer is the plumbing. Nobody sees it — until it leaks. Room entities, DAOs, mappers, and repository implementations are the pipes connecting your domain to the database. Two modules get filled: :data:local (Room database, entities, DAOs) and :data:repository (implementations, mappers).

The Data Layer Rules

Before AI generates a single entity, load the data rules from Chapter 2. These prevent the most common Room mistakes AI makes.

Rule DATA-001: Explicit @ColumnInfo

Every Room entity field MUST have an explicit @ColumnInfo(name = "...") annotation. Never rely on Kotlin property names for column names. Column names use snake_case. This prevents silent column name changes when refactoring.

Rule DATA-002: Enums as Strings

Store enums as String in Room, not as ordinals. Use @TypeConverter with enum.name / enumValueOf(). Ordinal storage breaks silently when someone reorders the enum.

Rule DATA-003: @Transaction on @Relation

Any DAO method that returns a type with @Relation MUST be annotated with @Transaction. Without it, Room can return inconsistent data if a write happens mid-query.

Load All Three Into Every Prompt

AI consistently forgets DATA-001 (implicit column names) and DATA-003 (@Transaction). Include these rules verbatim in every data layer prompt. It takes 5 seconds and saves 30 minutes of debugging.

From Raw SQLite to Room Entities

The monolith uses SQLiteOpenHelper with raw SQL. AI needs to see the original table creation to produce correct Room entities.

Migration: Raw SQLite → Room Entities

Before: Raw SQL in DatabaseHelper.kt
// DatabaseHelper.kt — the monolith's data layer
class DatabaseHelper(context: Context) : SQLiteOpenHelper(context, "recipes.db", null, 1) {
    override fun onCreate(db: SQLiteDatabase) {
        db.execSQL("""
            CREATE TABLE recipes (
                id INTEGER PRIMARY KEY AUTOINCREMENT,
                title TEXT NOT NULL,
                description TEXT,
                difficulty INTEGER DEFAULT 0,
                servings INTEGER DEFAULT 1,
                is_favorite INTEGER DEFAULT 0,
                cookbook_id INTEGER,
                created_at INTEGER
            )
        """)
        db.execSQL("""
            CREATE TABLE ingredients (
                id INTEGER PRIMARY KEY AUTOINCREMENT,
                recipe_id INTEGER NOT NULL,
                name TEXT NOT NULL,
                amount REAL,
                unit TEXT,
                FOREIGN KEY (recipe_id) REFERENCES recipes(id) ON DELETE CASCADE
            )
        """)
    }
}
After: Room entities in :data:local
// com.example.recipes.data.local.entity.RecipeEntity
@Entity(tableName = "recipes")
data class RecipeEntity(
    @PrimaryKey(autoGenerate = true)
    @ColumnInfo(name = "id") val id: Long = 0,
    @ColumnInfo(name = "title") val title: String,
    @ColumnInfo(name = "description") val description: String?,
    @ColumnInfo(name = "difficulty") val difficulty: String,
    @ColumnInfo(name = "servings") val servings: Int,
    @ColumnInfo(name = "is_favorite") val isFavorite: Boolean,
    @ColumnInfo(name = "cookbook_id") val cookbookId: Long?,
    @ColumnInfo(name = "created_at") val createdAt: Long
)

@Entity(
    tableName = "ingredients",
    foreignKeys = [ForeignKey(
        entity = RecipeEntity::class,
        parentColumns = ["id"],
        childColumns = ["recipe_id"],
        onDelete = ForeignKey.CASCADE
    )]
)
data class IngredientEntity(
    @PrimaryKey(autoGenerate = true)
    @ColumnInfo(name = "id") val id: Long = 0,
    @ColumnInfo(name = "recipe_id") val recipeId: Long,
    @ColumnInfo(name = "name") val name: String,
    @ColumnInfo(name = "amount") val amount: Double,
    @ColumnInfo(name = "unit") val unit: String
)

Notice: difficulty is stored as String (not Int), matching DATA-002. Every field has explicit @ColumnInfo, matching DATA-001. The foreign key constraint is preserved from the original SQL.

Generating DAOs

DAOs are where AI shines — the CRUD patterns are predictable and the Room annotation API is well-documented in AI training data. But you still need constraints to prevent common mistakes.

Good: DAO Generation Prompt

Generate Room DAOs for :data:local. One DAO per entity.
RULES: suspend for one-shot, Flow for observable, @Transaction on @Relation,
@Insert(onConflict = REPLACE). Package: .data.local.dao

RecipeDao: getAll(Flow), getById, search(Flow), insert, update, delete,
  toggleFavorite, getWithIngredients(@Relation+@Transaction)
IngredientDao: getByRecipeId, insertAll, deleteByRecipeId

Generated RecipeDao

@Dao
interface RecipeDao {
    @Query("SELECT * FROM recipes ORDER BY created_at DESC")
    fun getAll(): Flow<List<RecipeEntity>>

    @Query("SELECT * FROM recipes WHERE id = :id")
    suspend fun getById(id: Long): RecipeEntity?

    @Query("SELECT * FROM recipes WHERE title LIKE '%' || :query || '%'")
    fun search(query: String): Flow<List<RecipeEntity>>

    @Transaction
    @Query("SELECT * FROM recipes WHERE id = :id")
    suspend fun getWithIngredients(id: Long): RecipeWithIngredients?

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insert(recipe: RecipeEntity): Long

    @Update
    suspend fun update(recipe: RecipeEntity)

    @Query("DELETE FROM recipes WHERE id = :id")
    suspend fun deleteById(id: Long)

    @Query("UPDATE recipes SET is_favorite = NOT is_favorite WHERE id = :id")
    suspend fun toggleFavorite(id: Long)
}

Watch: Missing @Transaction

AI frequently generates getWithIngredients() without @Transaction. This is the DATA-003 rule. Without it, a concurrent write between the parent query and the relation query can return a recipe with someone else's ingredients. Always check @Relation methods.

The @Relation Pattern

For the "recipe with ingredients" query, Room uses the @Relation annotation on a data class that embeds the parent and children.

RecipeWithIngredients

// com.example.recipes.data.local.entity.RecipeWithIngredients
data class RecipeWithIngredients(
    @Embedded val recipe: RecipeEntity,
    @Relation(
        parentColumn = "id",
        entityColumn = "recipe_id"
    )
    val ingredients: List<IngredientEntity>
)

This is a data layer type — it lives in :data:local, never in :core:domain. The mapper will convert this to a domain Recipe with its ingredients list.

Type Converters and Database Class

Room needs @TypeConverter for enums. Following DATA-002, store as strings. Register converters on the database class:

Database + Converters

@Database(entities = [RecipeEntity::class, IngredientEntity::class], version = 1)
@TypeConverters(Converters::class)
abstract class RecipesDatabase : RoomDatabase() {
    abstract fun recipeDao(): RecipeDao
    abstract fun ingredientDao(): IngredientDao
}

class Converters {
    @TypeConverter fun fromDifficulty(value: String): String = value
    @TypeConverter fun toDifficulty(value: String): String = value
}

Why strings? If someone adds BEGINNER at ordinal 0, all existing rows shift. "EASY" as a string is immune to reordering. AI defaults to ordinals — always enforce DATA-002.

Going to Room 3.0 (Optional, KMP)

Sidebar: Room 3.0 — Multiplatform-First, Coroutine-First

Room 3.0 shipped its first alpha in March 2026 (3.0.0-alpha03 as of April). It's a breaking version: the package moves from androidx.room to androidx.room3, the runtime artifact becomes androidx.room3:room3-runtime, KSP is the only supported processor (no KAPT), and DAO functions must be suspend unless they return reactive types like Flow. SupportSQLite is replaced by androidx.sqlite driver APIs. The payoff: full Kotlin Multiplatform support — Android, iOS, JVM, JS, WasmJS — so your :data:local module becomes reusable in a future KMP rewrite.

For this chapter, the good news: @Entity, @Dao, @Relation, and @Transaction patterns are unchanged in 3.0. For an Android-only Clean Architecture rewrite in 2026, stay on Room 2.x stable (2.8.4+, as pinned in Ch 4 and Ch 9). Re-evaluate Room 3.0 once it goes stable — especially if KMP is on your roadmap.

Data Mappers: The Boundary Layer

Room entities are not domain models. They exist in :data:local. Domain models exist in :core:domain. Mappers convert between them at the boundary, living in :data:repository.

Migration: Direct Entity Usage → Mapper Pattern

Before: Entity types leak into the UI
// In a bad rewrite — entity used directly in ViewModel
class RecipeViewModel : ViewModel() {
    val recipes: LiveData<List<RecipeEntity>> = recipeDao.getAll()
    //                         ^^^^^^^^^^^^
    // Room type in presentation layer = architecture violation
}
After: Mapper converts at the repository boundary
// com.example.recipes.data.repository.mapper.RecipeMapper
object RecipeMapper {
    fun RecipeEntity.toDomain(): Recipe = Recipe(
        id = id,
        title = title,
        description = description ?: "",
        difficulty = DifficultyLevel.valueOf(difficulty),
        servings = servings,
        isFavorite = isFavorite,
        cookBookId = cookbookId,
        createdAt = createdAt
    )

    fun Recipe.toEntity(): RecipeEntity = RecipeEntity(
        id = id,
        title = title,
        description = description,
        difficulty = difficulty.name,
        servings = servings,
        isFavorite = isFavorite,
        cookbookId = cookBookId,
        createdAt = createdAt
    )

    fun RecipeWithIngredients.toDomain(): Recipe = recipe.toDomain().copy(
        ingredients = ingredients.map { it.toDomain() }
    )
}

object IngredientMapper {
    fun IngredientEntity.toDomain(): Ingredient = Ingredient(
        id = id, name = name,
        amount = amount,
        unit = UnitOfMeasure.valueOf(unit)
    )

    fun Ingredient.toEntity(recipeId: Long): IngredientEntity = IngredientEntity(
        id = id, recipeId = recipeId,
        name = name, amount = amount, unit = unit.name
    )
}

The mapper is pure conversion logic — no business rules, no side effects. It's the translator standing at the border between two countries that speak different languages.

Good: Mapper Generation Prompt

Entity.toDomain() and DomainModel.toEntity(). Enums via .name/.valueOf().
Nullable fields: provide defaults. Stateless objects, no dependencies.

Repository Implementations

The repository implementation is the bridge. It takes the domain interface from Chapter 5 and fulfills the contract using Room DAOs and mappers. This is where dependency inversion becomes concrete.

RecipeRepositoryImpl

class RecipeRepositoryImpl(
    private val recipeDao: RecipeDao,
    private val ingredientDao: IngredientDao
) : RecipeRepository {

    override suspend fun getAll(): Result<List<Recipe>> = try {
        val entities = recipeDao.getAll().first()
        Result.Success(entities.map { it.toDomain() })
    } catch (e: Exception) { Result.Error("Failed to load recipes", e) }

    override suspend fun getById(id: Long): Result<Recipe> = try {
        val entity = recipeDao.getWithIngredients(id)
            ?: return Result.Error("Recipe not found")
        Result.Success(entity.toDomain())
    } catch (e: Exception) { Result.Error("Failed to load recipe", e) }

    override suspend fun save(recipe: Recipe): Result<Long> = try {
        val id = recipeDao.insert(recipe.toEntity())
        recipe.ingredients.forEach { ingredientDao.insert(it.toEntity(id)) }
        Result.Success(id)
    } catch (e: Exception) { Result.Error("Failed to save recipe", e) }

    override suspend fun delete(id: Long): Result<Unit> = try {
        recipeDao.deleteById(id)
        Result.Success(Unit)
    } catch (e: Exception) { Result.Error("Failed to delete recipe", e) }
}

Every method follows the same pattern: call DAO, map to domain, wrap in Result. Errors are caught at the boundary and converted to Result.Error. No exceptions escape the data layer.

Room Database Migration

If replacing SQLiteOpenHelper with Room, you may need a migration. For the recipes app the schema matches exactly — Room takes over the existing file. But if columns change:

Explicit Migration

val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(db: SupportSQLiteDatabase) {
        db.execSQL("ALTER TABLE recipes ADD COLUMN prep_time INTEGER DEFAULT 0")
    }
}

AI Migration Pitfall

AI loves to generate fallbackToDestructiveMigration() because it "works." This silently deletes all user data. Never allow it. Always write explicit migrations. Add // FORBIDDEN: fallbackToDestructiveMigration() to CLAUDE.md.

Error Handling at the Data Boundary

The data layer is where raw exceptions from Room, SQLite, and the filesystem get translated into domain-friendly Result.Error values.

Error Translation Pattern

override suspend fun save(recipe: Recipe): Result<Long> = try {
    val id = recipeDao.insert(recipe.toEntity())
    ingredientDao.insertAll(recipe.ingredients.map { it.toEntity(id) })
    Result.Success(id)
} catch (e: SQLiteConstraintException) {
    Result.Error("Recipe with this title already exists", e)
} catch (e: SQLiteFullException) {
    Result.Error("Storage full — cannot save recipe", e)
} catch (e: Exception) {
    Result.Error("Failed to save recipe", e)
}

Specific exceptions get specific messages. The domain layer never sees SQLiteConstraintException — it sees a Result.Error with a human-readable message.

Testing the Data Layer

Data layer tests come in two flavors: fake repositories for consumers, and instrumented tests for Room itself.

Fake Repository for Consumers

// :core:testing — shared by all test sourceSets
class FakeRecipeRepository : RecipeRepository {
    private val recipes = mutableListOf<Recipe>()
    var shouldReturnError = false

    override suspend fun getAll(): Result<List<Recipe>> {
        if (shouldReturnError) return Result.Error("Test error")
        return Result.Success(recipes.toList())
    }

    override suspend fun save(recipe: Recipe): Result<Long> {
        if (shouldReturnError) return Result.Error("Test error")
        recipes.add(recipe.copy(id = (recipes.size + 1).toLong()))
        return Result.Success(recipes.last().id)
    }
    // ... other methods follow same pattern
    fun addRecipe(recipe: Recipe) { recipes.add(recipe) }
}

Have AI generate one fake per repository interface. For Room DAO tests, you need instrumented tests (device/emulator), but fake repositories cover more test surface faster. Focus AI effort on fakes first.

Common AI Mistakes in the Data Layer

Mistake What AI Does Rule / Fix
Implicit column names Omits @ColumnInfo, relies on property name DATA-001: always explicit
Ordinal enum storage Stores DifficultyLevel.ordinal as Int DATA-002: store as String via .name
Missing @Transaction @Relation query without @Transaction DATA-003: always @Transaction on @Relation
Entity leak Returns RecipeEntity from repository to ViewModel Mapper at boundary. Domain types only in return values.
Destructive migration fallbackToDestructiveMigration() FORBIDDEN. Write explicit migrations.
Business logic in repo Puts validation in repository instead of use case Repository = data access only. Logic in domain.

Exercise: From Raw SQL to Room + Repository

  1. Feed AI the raw SQL schema: Paste your SQLiteOpenHelper.onCreate() or schema dump. Ask for Room entities with DATA-001, DATA-002, DATA-003 rules.
  2. Generate DAOs: One per entity. Verify: @Transaction on all @Relation methods, Flow for observable queries, suspend for one-shot queries.
  3. Generate the database class: @Database with entities, @TypeConverters, abstract DAO accessors. No fallbackToDestructiveMigration().
  4. Generate mappers: Entity.toDomain() and DomainModel.toEntity() as extension functions. Verify: enum conversion uses .name / .valueOf().
  5. Generate repository implementations: Implement domain interfaces. Every method: try/catch → Result. Mapper at every boundary.
  6. Generate fake repositories: In :core:testing. In-memory, deterministic, with error toggle.
  7. Run the review checklist: Check DATA-001, DATA-002, DATA-003, no entity leaks, no destructive migration.
