Chapter 11

The Integration Sprint

Connecting all layers, behavior comparison, performance validation, and the final build

Moving Day

The house is built and inspected. Every room is finished, every circuit tested. Now comes moving day — the moment when furniture goes in, doors open, and real people walk through the hallways for the first time.

Integration is where theory meets reality. Individual layers worked in isolation. Now they must work together. The ViewModel calls the use case which calls the repository which queries Room which returns data that flows through mappers back to the Composable. Every handoff is a potential break point.

This chapter is a sprint: integrate → test → fix → repeat. Daily. Until the rewritten app matches the monolith's behavior exactly — then surpasses it.

The Layer Integration Checklist

Connect the Layers — In Order

  • Domain ← Data: Repository implementations inject DAOs, return domain models via mappers. Verify: use case calls work end-to-end with real Room database.
  • Presentation ← Domain: ViewModels inject use cases via Hilt. Verify: events trigger state changes with real data flow.
  • Navigation ← Presentation: NavHost connects screens. Effects trigger navigation callbacks. Verify: full user flows work (list → detail → edit → back).
  • DI ← All: Hilt compiles all modules. Every @Inject resolved. No missing bindings. Verify: app starts without crash.
  • App ← Everything: :app module assembles NavHost, applies theme, handles deep links. Verify: the app does what the monolith did.

Rule INT-001: Integrate Bottom-Up

Connect layers from the bottom: data → domain → presentation → navigation → app. Never start from the UI and work down. Bottom-up integration catches boundary mismatches early — a mapper that returns the wrong type surfaces immediately, not after three layers of debugging.

Tip: Always Paste the Full Error

AI can't diagnose "it doesn't compile." Paste the complete error output — the module, the file, the line, the message. The more precise the error, the more precise the fix. Vague input yields vague suggestions.

// Fix 3: Repository returning entity instead of domain model

// Fix 4: Add @HiltAndroidApp to Application class
@HiltAndroidApp class RecipesApp : Application()

// Fix 5: Route not registered in NavHost
// In :app NavHost: composable<RecipeDetail> { ... }  // ← was missing

Every one of these is something AI generated correctly in isolation but failed at the boundary. The missing @InstallIn only surfaces when Hilt tries to assemble the full dependency graph. The mapper mismatch only appears when the repository and ViewModel are in the same build.

AI-Assisted Error Resolution Cycle

1

Build

Run ./gradlew :app:assembleDebug. Collect all errors.

2

Diagnose

Paste errors to AI with module graph and architecture rules. AI identifies which module needs the fix.

3

Fix

Apply fixes in the correct module. Never fix a data layer issue in the presentation layer.

4

Test

Run ./gradlew test. If any existing test breaks, the fix introduced a regression. Revert and try again.

5

Repeat

Go to step 1. Each cycle should reduce errors. If error count increases, the fix was wrong.

Warning: Don't Let AI "Fix" Errors by Breaking Architecture

AI's fastest fix for a missing binding is often to dump everything in :app. For a visibility issue, it makes everything public. For a mapper mismatch, it skips the mapper. Every "fix" must be validated against your architecture rules. A fast fix that violates DI-001 or PKG-001 is not a fix — it's debt.

Running the Full Test Suite

The Integration Test Sequence

# Step 1: Unit tests (should be green from Chapter 10)
./gradlew test

# Step 2: Android instrumented tests (Room, Compose UI)
./gradlew connectedAndroidTest

# Step 3: Lint and static analysis
./gradlew lint detekt

# Step 4: Full build
./gradlew :app:assembleRelease

If unit tests fail, the fix is in the layer that owns the test. If instrumented tests fail, the fix is usually in the data layer (Room schema) or navigation (missing routes). If lint fails, check for unused imports or deprecated APIs that AI introduced.

Behavior Comparison: Monolith vs Rewrite

The rewritten app must do everything the monolith did. No dropped features, no behavior differences. Feature parity is the minimum bar.

Feature Parity Checklist: Recipes App

  • Recipe list: Shows all recipes, sorted by title. Pull-to-refresh works. Empty state shown when no recipes.
  • Recipe detail: Shows title, ingredients, instructions, servings. Edit button navigates to edit screen.
  • Recipe edit: Create new recipe, edit existing. Validation (title required, servings > 0). Save persists to database.
  • Cookbook list: Shows cookbooks with recipe count. Create new cookbook.
  • Grocery list: Auto-generated from selected recipes. Check off items. Persists across app restarts.
  • Search: Filters recipes by title. Updates in real-time as user types.
  • Favorites: Toggle favorite on recipe. Filter by favorites.
  • Delete: Swipe to delete recipe. Undo via snackbar.

Tip: Compose Recomposition Tracking

Enable recomposition highlighting in Android Studio (Layout Inspector → Show Recomposition Counts). If a Composable recomposes on every frame, the state isn't stable. Common cause: creating new lambda instances in the Composable body instead of using remember or passing stable references.

Edge Cases AI Missed

AI generates the happy path well. But real apps live in the edge cases. The manual review pass catches what AI didn't think about.

Edge CaseWhat BreaksFix
Empty database on first launch List shows loading forever Handle empty state in ViewModel: if list is empty + no error → show empty state
Configuration change during save Save operation lost SaveRecipeUseCase runs in viewModelScope — survives config change
Network timeout during sync Infinite loading spinner OkHttpClient timeout configured. CallAdapter returns ApiResult.NetworkError
Deep link to deleted recipe Crash: RecipeNotFoundException ViewModel handles Result.Error(NotFound) → shows "Recipe not found" screen
Rapid back button presses Navigation crash: destination not found NavController.popBackStack() with currentDestination null check
Process death and restore Lost ViewModel state SavedStateHandle for critical state. Room persists everything else
Good Prompt: Find Edge Cases
Context: Recipes app with MVI architecture. 3 features: recipe (list,
detail, edit), books (cookbook list, detail), grocery (grocery list).

I've verified the happy paths. Now find edge cases I might have missed.
For each screen, consider:
- Empty states (no data, first launch)
- Error states (network, database, not found)
- Lifecycle events (config change, process death, background → foreground)
- Navigation edge cases (deep links, back stack, rapid navigation)
- Input edge cases (very long text, special characters, zero values)
- Concurrent operations (save while navigating, delete while loading)

For each edge case: describe the scenario, what currently breaks, and
what the fix should be (which module, which component).

The Daily Integration Cycle

AM

Build + Smoke Test

Assemble the app. Launch on device. Walk through every screen. Does it crash? Does data appear? Do transitions work?

MID

Feature Parity Check

Run the feature parity checklist. Compare monolith and rewrite side-by-side. Note any differences in behavior.

PM

Fix + Test + Commit

Fix integration issues. Run full test suite. Commit working state. Never end the day with a broken build.

The Final Build

Rule INT-002: Release Build Validation

The integration sprint is not done until you've produced a signed release APK, installed it on a real device, and verified all features work. Debug builds hide ProGuard issues, missing keep rules, and performance problems. Release is the only truth.

Release Build Checklist

# 1. ProGuard/R8 rules
# AI-generated modules may need keep rules for:
-keep class com.recipes.core.domain.model.** { *; }  # Serialized models
-keep class * extends androidx.room.RoomDatabase      # Room

# 2. Build the release APK
./gradlew :app:assembleRelease

# 3. Install on real device
adb install -r app/build/outputs/apk/release/app-release.apk

# 4. Test every feature on the real device
# ProGuard can strip code that worked in debug!

# 5. Compare APK sizes
ls -la app/build/outputs/apk/release/   # rewrite
ls -la monolith/app/build/outputs/apk/  # original

Warning: ProGuard Strips AI-Generated Code

AI generates data classes with @Serializable for navigation routes. ProGuard strips the serialization metadata unless you add keep rules. If navigation crashes only in release builds, missing keep rules are the culprit. Test every navigation route in the release APK.

Exercise: Your Integration Sprint

Exercise 11: Ship the Integration

  1. Assemble: Run ./gradlew :app:assembleDebug. Fix all compile errors using the AI-assisted error resolution cycle from this chapter. Don't break architecture rules to fix errors.
  2. Test suite: Run ./gradlew test. All 128+ tests must pass. For failures, use AI to diagnose — paste the full error with module context.
  3. Feature parity: Walk through every feature in the monolith. Does the rewrite do the same thing? Use the feature parity checklist. Note any gaps.
  4. Edge cases: Run the edge case prompt from this chapter. Fix the top 5 most critical edge cases (process death, deep links, empty states, config changes, timeouts).
  5. Release build: Build the signed release APK. Install on a real device. Verify every feature works with ProGuard enabled.

Success criteria: Clean release build. All tests green. Feature parity verified. Top edge cases handled. APK size within 15% of monolith.
