Chapter 2

Teaching AI Your Standards

Write the dual-format architecture rules BEFORE you touch a single line of code

Blueprints Before Demolition

In Chapter 1, you assembled the renovation crew and surveyed the damage. You have a migration inventory, a manifesto, and a target architecture. But here's the thing about AI: it's an incredibly fast builder that has never seen your blueprints.

Without explicit rules, AI will generate code that looks correct but violates your standards in subtle ways. It'll put a DAO in the feature module. It'll expose Room entities to the UI layer. It'll create a ViewModel that directly calls a repository instead of going through a use case.

Each violation is small. Accumulated across a 10-module rewrite, they become a new mess — just a cleaner-looking one.

The Rule: Architecture rules are written before any migration code. Not after the first PR. Not "when we get to that layer." Before. They're the blueprints your AI contractor reads on day one.

This chapter teaches you to write rules in two formats: review checklists for human reviewers, and .ctx files for AI context. Same rules, different audiences. By the end, you'll have a complete rule library covering all 10 areas of your architecture.

Why Rules Before Code

You might be tempted to start migrating and "figure out the rules as you go." Here's what happens:

Rules After Code

  • AI generates 40 files in the first sprint
  • You notice violations in PR review
  • You fix 40 files manually
  • AI generates 40 more with the same violations
  • You write the rule... and still have 40 files to fix

Rules Before Code

  • You spend 2-3 hours writing rules
  • Rules load into every AI session automatically
  • AI generates 40 files that follow the rules
  • PR review catches edge cases, not systemic violations
  • You update the rule once, all future code follows it

Even allowing for that variance, rules are the highest-ROI investment in your entire rewrite.

The 10 Architecture Rule Areas

Every Android app rewrite needs rules in these 10 areas. Some will be heavier than others for your project, but you need at least basic coverage in all of them.

Area Rule Prefix What It Governs
Package Structure PKG- Module boundaries, dependency direction, visibility
Presentation PRES- MVI contracts, ViewModel structure, Compose patterns
Domain DOM- Use cases, entities, repository interfaces, Result pattern
Data DATA- Room entities, DAOs, mappers, repository implementations
Error Handling ERR- Error hierarchy, layer boundaries, UI error mapping
Network NET- API layer, CallAdapter, authentication, OkHttp config
Testing TEST- Test structure, fakes vs mocks, naming, coverage targets
Dependency Injection DI- Hilt modules, scoping, @Binds vs @Provides, qualifiers
Gradle GRADLE- Convention plugins, version catalog, KSP configuration
Navigation NAV- Route definitions, type-safe navigation, deep links
Rule prefixes matter. When AI generates a file and you need to reference a violation, PRES-003 is unambiguous. "That thing about ViewModels" is not. Prefixes also let you filter rules by area when loading context.

Anatomy of a Rule: MUST / SHOULD / PREFER

Not all rules carry equal weight. Use three severity levels:

MUST
→
SHOULD
→
PREFER
  • MUST — Violation is a blocker. Code cannot be merged. Example: "Feature modules MUST NOT depend on other feature modules."
  • SHOULD — Violation requires justification. You need a reason to break it. Example: "Use cases SHOULD have a single public method (operator fun invoke)."
  • PREFER — Recommended but flexible. Example: "PREFER @Binds over @Provides when binding an interface to its implementation."

Each rule in the checklist follows this template:

Rule Template: Before and After

Vague Rule (Useless)

- Use proper architecture in ViewModels
- Don't put business logic in the UI
- Follow Clean Architecture principles

Specific Rule (Actionable)

PRES-001 [MUST] ViewModel MUST expose exactly one StateFlow<State>
  for the UI. The State MUST be a data class defined in the
  Contract interface alongside the Event sealed interface.

  GOOD:
    class RecipeListViewModel : ViewModel() {
        private val _state = MutableStateFlow(State())
        val state: StateFlow<State> = _state.asStateFlow()

        data class State(val recipes: List<Recipe> = emptyList())
        sealed interface Event { ... }
    }

  BAD:
    class RecipeListViewModel : ViewModel() {
        val recipes = MutableLiveData<List<Recipe>>()  // LiveData
        val isLoading = MutableStateFlow(false)  // Multiple flows
        val error = MutableStateFlow<String?>(null)  // Scattered state
    }

The difference is night and day. The specific rule gives AI a pattern to follow, a pattern to avoid, and a severity level. When AI generates code, it can check its own output against concrete examples.

META-RULE Every architecture rule MUST include: (1) a unique ID with area prefix, (2) severity level [MUST/SHOULD/PREFER], (3) one sentence stating the rule, (4) a GOOD example, (5) a BAD example showing the specific anti-pattern.

Writing the Review Checklist

The review checklist is the human-readable format. It's what you use during PR review, and it's what you hand to teammates. Here's how to write the rules for each of the 10 areas.

Package Structure Rules (PKG-)

Effective Prompt for Generating PKG Rules
I'm rewriting an Android app into 10 modules: :app, :core:common, :core:design-system, :core:domain, :core:testing, :data:local, :data:repository, :feature:recipe, :feature:books, :feature:grocery Generate package structure rules covering: 1. Which modules can depend on which (dependency graph) 2. What each module is allowed to contain (classes, resources) 3. Forbidden dependencies (feature → feature, data → presentation) 4. Visibility rules (internal vs public API) Format each rule as: [ID] [SEVERITY] description + GOOD example + BAD example. Use IDs like PKG-001, PKG-002, etc.
Gives AI the complete module list and asks for concrete rules with examples. The format constraint ensures machine-readable output.

Here's what the output should look like for a few critical PKG rules:

PKG-001 [MUST] Feature modules MUST NOT depend on other feature modules.
  Inter-feature communication goes through :core:domain interfaces.
  GOOD: :feature:recipe depends on :core:domain
  BAD:  :feature:recipe depends on :feature:books

PKG-002 [MUST] :core:domain MUST NOT have Android framework dependencies.
  Pure Kotlin only. No Context, no Lifecycle, no Room annotations.
  GOOD: data class Recipe(val id: Long, val title: String)
  BAD:  data class Recipe(val id: Long, @ColumnInfo val title: String)

PKG-003 [MUST] Data layer entities MUST NOT leak to the presentation layer.
  Use mappers at the repository boundary. Domain models cross boundaries.
  GOOD: repository.getRecipes() returns List<Recipe> (domain model)
  BAD:  repository.getRecipes() returns List<RecipeEntity> (Room entity)
Aim for 5-10 rules per area. More than 15 and AI starts ignoring the later ones. Fewer than 3 and you'll miss critical violations. The sweet spot is coverage without overload.

Presentation Rules (PRES-)

Presentation rules are typically the longest set because the UI layer has the most ways to go wrong. Key rules to write:

PRES-001 [MUST] Single StateFlow<State> per ViewModel. No scattered state.
PRES-002 [MUST] State is a data class. Events are a sealed interface.
PRES-003 [MUST] Side effects go through a Channel<Effect>, not SharedFlow.
PRES-004 [SHOULD] Reducer is a pure function: (State, Event) → State + Effects.
PRES-005 [MUST] No business logic in Composables. Composables render state.
PRES-006 [SHOULD] Composable previews for every screen and major component.
PRES-007 [PREFER] State hoisting: Composables receive state, emit events.

Domain Rules (DOM-)

DOM-001 [MUST] Use cases have a single public method: operator fun invoke().
DOM-002 [MUST] Repository interfaces live in :core:domain, implementations in :data.
DOM-003 [MUST] Domain models use the Result sealed class for error handling.
DOM-004 [MUST] No Android dependencies in :core:domain. Pure Kotlin only.
DOM-005 [SHOULD] Use cases accept and return domain models, never entities.

Data Rules (DATA-)

DATA-001 [MUST] Room entities use explicit @ColumnInfo for every field.
DATA-002 [MUST] Enums are stored as String, not ordinal.
DATA-003 [MUST] Queries with @Relation MUST use @Transaction.
DATA-004 [MUST] Mappers at the boundary: Entity ↔ Domain Model.
DATA-005 [SHOULD] Repository catches exceptions and returns Result.
Don't skip the GOOD/BAD examples when writing your real rules. The one-liners above are summaries. In your actual checklist, each rule needs concrete code showing the right way and the wrong way. AI learns more from examples than from descriptions.

Condensing to .ctx Files

Review checklists are great for humans. But AI context windows have limits. You can't paste 200 rules with full examples into every prompt. That's where .ctx files come in.

A .ctx file is a compressed version of a rule area — designed to fit in ~25 lines while preserving the essential constraints. Think of it as the blueprint summary that hangs on the wall at the construction site.

Checklist → .ctx Compression

Full Checklist (presentation-rules.md) — 80 lines

PRES-001 [MUST] ViewModel MUST expose exactly one StateFlow<State>
  for the UI. The State MUST be a data class defined in the
  Contract interface alongside the Event sealed interface.

  GOOD:
    class RecipeListViewModel : ViewModel() {
        private val _state = MutableStateFlow(State())
        val state: StateFlow<State> = _state.asStateFlow()
        data class State(...)
        sealed interface Event { ... }
    }

  BAD:
    class RecipeListViewModel : ViewModel() {
        val recipes = MutableLiveData<List<Recipe>>()
        val isLoading = MutableStateFlow(false)
        val error = MutableStateFlow<String?>(null)
    }

PRES-002 [MUST] ...
... (60 more lines)

Compressed .ctx File (presentation.ctx) — 25 lines

# Presentation Layer Rules
# Full checklist: /rules/presentation-rules.md

- Single StateFlow<State> per VM, State = data class in Contract
- Events = sealed interface in Contract, Effects via Channel
- Reducer: pure (State, Event) → State + Effects, no side effects
- Composables render state only, no business logic
- State hoisting: receive state, emit events up
- Preview for every screen and major component
- No LiveData, no multiple StateFlows, no mutable state in UI
- ViewModel calls use cases, never repositories directly
- Navigation events are Effects, not direct nav calls
- Loading/error/content modeled in State, not separate flows

Anti-patterns: LiveData, scattered flows, logic in Composables,
direct repo calls from VM, mutable state exposure, navigation
in Composables, SharedFlow for side effects.

The .ctx file sacrifices examples for density. That's intentional — AI has seen thousands of MVI implementations. It doesn't need a full code sample to know what "single StateFlow per VM" means. It needs the constraint.

CTX-001 Each .ctx file MUST be 25 lines or fewer. It MUST reference the full checklist path. It MUST list anti-patterns as a final section. One .ctx file per architecture area.

The Dual-Format System

Here's how the two formats work together in your project:

Rule File Structure

your-project/
├── rules/
│   ├── checklists/
│   │   ├── package-structure-rules.md    (PKG-001 through PKG-010)
│   │   ├── presentation-rules.md         (PRES-001 through PRES-015)
│   │   ├── domain-rules.md               (DOM-001 through DOM-008)
│   │   ├── data-rules.md                 (DATA-001 through DATA-012)
│   │   ├── error-handling-rules.md       (ERR-001 through ERR-006)
│   │   ├── network-rules.md              (NET-001 through NET-008)
│   │   ├── testing-rules.md              (TEST-001 through TEST-010)
│   │   ├── di-rules.md                   (DI-001 through DI-007)
│   │   ├── gradle-rules.md               (GRADLE-001 through GRADLE-005)
│   │   └── navigation-rules.md           (NAV-001 through NAV-006)
│   └── ctx/
│       ├── package-structure.ctx
│       ├── presentation.ctx
│       ├── domain.ctx
│       ├── data.ctx
│       ├── error-handling.ctx
│       ├── network.ctx
│       ├── testing.ctx
│       ├── di.ctx
│       ├── gradle.ctx
│       └── navigation.ctx
└── CLAUDE.md  (references both formats)

Checklists are for human PR review and onboarding. .ctx files are for AI context — loaded via CLAUDE.md, pasted into prompts, referenced by hooks.

Your CLAUDE.md should reference .ctx files, not full checklists. Embedding full checklists eats 2,000+ tokens on rules alone. The .ctx files keep it under 300 tokens per area.

Loading Rules Into CLAUDE.md

Your CLAUDE.md is the single source of truth that AI reads at the start of every session. Here's how to wire up the architecture rules.

CLAUDE.md Architecture Rules Section
## Architecture Rules This project follows strict architecture rules. Before generating or modifying any code, read the relevant .ctx files: - Package structure: `rules/ctx/package-structure.ctx` - Presentation (MVI): `rules/ctx/presentation.ctx` - Domain layer: `rules/ctx/domain.ctx` - Data layer: `rules/ctx/data.ctx` - Error handling: `rules/ctx/error-handling.ctx` - DI (Hilt): `rules/ctx/di.ctx` Full checklists with examples are in `rules/checklists/`. ### Critical Rules (Always Active) - Feature modules MUST NOT depend on other feature modules (PKG-001) - :core:domain MUST be pure Kotlin, no Android dependencies (PKG-002) - Single StateFlow<State> per ViewModel (PRES-001) - Use cases have one public method: operator fun invoke() (DOM-001) - Room entities use explicit @ColumnInfo (DATA-001) ### Forbidden Patterns - No LiveData anywhere in the project - No Room entities exposed outside :data modules - No business logic in Composables - No direct repository calls from ViewModels (use Use Cases)
Lists .ctx file paths for AI to read. Inlines the 5 most critical rules. Adds a "Forbidden Patterns" section as a hard stop.

The pattern: reference files for detail, inline the non-negotiables. AI sees the inlined rules immediately and reads .ctx files when working on the relevant layer.

The Rule Compliance Test

How do you know AI actually follows your rules? Test it. Before starting the real migration, ask AI to generate a small piece of code and check compliance.

Rule Compliance Test Prompt
Generate a complete ViewModel for the recipe list screen in :feature:recipe. It should: - Load recipes by category from GetRecipesByCategoryUseCase - Handle loading, success, and error states - Support pull-to-refresh - Support search filtering Follow ALL rules in the presentation .ctx file and the domain .ctx file. After generating, list which rules you followed and verify compliance.
A focused test: one ViewModel, multiple rules applicable. The "list which rules" instruction forces AI to self-check.

AI Self-Review Output

PRES-001: Single StateFlow<State> ✓
PRES-002: State is data class, Events sealed interface ✓
PRES-003: Effects via Channel ✓
PRES-004: Reducer is pure function ✓
PRES-005: No business logic in Composables ✓
DOM-001: Use case called via invoke() ✓
DOM-003: Result pattern used ✓
PRES-006: Missing Composable previews ✗

One failure out of eight. The rule is clear but not prominent enough. Fix: move PRES-006 into "Critical Rules" in CLAUDE.md, or add "No screen Composable without a @Preview" to Forbidden Patterns.

Run this for each layer: one ViewModel, one use case, one repository, one DAO. Four tests, 30 minutes, and you'll know which rules need reinforcement before the real migration.

The Rule Evolution Cycle

Rules aren't written once and forgotten. They evolve as you discover edge cases during the rewrite. Here's the cycle:

Write Rules
→
Compliance Test
→
Migrate
→
Discover Edge Case
→
Update Rule

The key discipline: when you find a new violation pattern, update the rule immediately. Don't fix the code and move on. Fix the code, add the anti-pattern to the rule, update the .ctx file. Future AI sessions get the updated constraint automatically.

Common trap: fixing violations in code review but never updating the rule. AI will make the same mistake in the next file, and the next, and the next. The rule is your leverage — use it.

Example: during the data layer migration, you discover AI keeps generating Room queries without @Transaction on @Relation methods. You fix three files manually. Then you update DATA-003 with the specific anti-pattern and a GOOD/BAD example. Now every future AI session gets the fix. The violation stops propagating.

Generating Your Full Rule Library With AI

You don't have to write every rule from scratch. AI can generate the initial set — you review and customize.

Rule Library Generation Prompt
Generate architecture rules for an Android app rewrite. The target architecture: - 10 Gradle modules (list them) - Presentation: MVI with Contract pattern, Compose, ViewModels - Domain: pure Kotlin, use cases with operator invoke, Result pattern - Data: Room with explicit @ColumnInfo, repository pattern, entity mappers - DI: Hilt with per-module @Module, @Binds preferred over @Provides - Testing: JUnit 5, fakes over mocks, ViewModel contract tests For each of the 10 areas (PKG, PRES, DOM, DATA, ERR, NET, TEST, DI, GRADLE, NAV): 1. Generate 5-10 rules with ID, severity [MUST/SHOULD/PREFER], description 2. Include GOOD and BAD code examples for each MUST rule 3. List anti-patterns at the end of each section Then generate a compressed .ctx version (25 lines max) for each area.
One prompt generates the entire initial rule library. You review, customize, and remove rules that don't apply to your project.
Review every generated rule. AI will include rules that don't match your project. No network layer? Delete NET rules. Using Kotlin Serialization? Update DATA rules. The generated library is a starting point, not a finished product.

Try It: Build Your Rule Library

Exercise: Write Rules for Your Rewrite

Using the migration inventory from Chapter 1 and the target architecture you defined:

  1. Generate the initial rules: Use the rule library generation prompt above. Customize it with your actual module names and tech stack.
  2. Review and prune: Remove rules that don't apply. Tighten rules that are too vague. Add rules for patterns specific to your project.
  3. Create .ctx files: Compress each area's checklist into a 25-line .ctx file. Reference the full checklist path.
  4. Wire into CLAUDE.md: Add the architecture rules section with .ctx references and critical rules inlined.
  5. Run the compliance test: Generate one ViewModel, one use case, one repository, one DAO. Check each against your rules.
  6. Fix failures: For any rule AI didn't follow, strengthen it: add to Critical Rules, add to Forbidden Patterns, or improve the GOOD/BAD examples.

Deliverable: A rules/ directory with checklists and .ctx files for all 10 areas, and a CLAUDE.md section that references them. This becomes the foundation for every chapter that follows.
