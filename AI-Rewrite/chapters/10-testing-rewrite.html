Chapter 10

Testing the Rewrite

AI-generated test suites, ViewModel contract tests, fakes, and the 128+ test target

The Building Inspector

Before anyone moves in, the building inspector arrives. Every beam, pipe, and circuit — tested against code. The monolith had zero tests. The rewritten app will have 128+. AI generates the suite, but you set the rules. A test that checks implementation details locks you into current code while catching nothing.

The order: domain first (pure functions), then data (fakes), then presentation (ViewModel contracts), then integration. For TDD with AI, see AI-Advanced Chapter 8.

The Testing Rules

Rule TEST-001: Unit Tests Dominate

80% unit tests, 15% integration tests, 5% UI tests. Unit tests run in milliseconds, catch logic errors, and provide fast feedback. Don't invert the pyramid — AI loves generating Espresso tests because they look impressive, but they're slow, flaky, and test the wrong thing.

Rule TEST-002: Fakes Over Mocks

Use hand-written fakes (in :core:testing) instead of Mockito/MockK. Fakes implement the same interface as the real class and maintain in-memory state. Mocks verify calls were made — fakes verify behavior is correct. AI generates mocks by default; override this.

Rule TEST-003: JUnit 5 + Turbine + Truth

JUnit 5 for lifecycle and parameterized tests. Turbine for testing Flows and Channels. Truth for assertions (readable, expressive). No JUnit 4 @Rule leftovers. No assertEquals — use assertThat().isEqualTo().

The Test Generation Order

1

Domain Layer

Use cases and domain logic. Pure Kotlin functions — no Android, no coroutines for most tests. Input → output. The easiest layer to test.

2

Data Layer (Fakes)

Generate fake repository implementations in :core:testing. These are used by presentation tests. Also test real repositories with in-memory Room databases.

3

Presentation Layer

ViewModel contract tests: given initial state + event → assert new state + effects. Uses fakes from step 2. Turbine for Flow/Channel assertions.

4

Integration Tests

Full stack: real ViewModel + real use case + fake repository. Verify that layers connect correctly. Fewer tests, higher confidence.

Testing the Domain Layer

Good Prompt: Generate Domain Tests
Context: :core:domain module, pure Kotlin, JUnit 5 + Truth.
Rules: TEST-001 (unit only, no Android), TEST-002 (FakeRecipeRepository),
TEST-003 (JUnit 5 + Truth assertThat(), no JUnit 4)

Classes: GetRecipesUseCase, GetRecipeByIdUseCase, SaveRecipeUseCase,
ValidateRecipeUseCase (title non-empty, servings > 0)

Per use case test: success path, error path (repo returns Result.Error),
edge cases (empty list, not found, validation failures).
Pattern: arrange (set up fake) → act (invoke) → assert.

Migration: Untested Monolith → Tested Domain

Before: Monolith (zero tests)
// RecipeListActivity.kt — business logic buried in UI
fun loadRecipes() {
    val cursor = db.rawQuery("SELECT * FROM recipes WHERE deleted = 0", null)
    val recipes = mutableListOf<Recipe>()
    while (cursor.moveToNext()) {
        recipes.add(Recipe(cursor.getLong(0), cursor.getString(1),
            cursor.getInt(2), cursor.getString(3) == "1"))
    }
    cursor.close()
    adapter.submitList(recipes.sortedBy { it.title })
}
// Untestable: requires Activity, database, and adapter.
After: Domain Tests (pure Kotlin)
class GetRecipesUseCaseTest {
    private val fakeRepo = FakeRecipeRepository()
    private val useCase = GetRecipesUseCase(fakeRepo)

    @Test fun `returns recipes sorted by title`() = runTest {
        fakeRepo.addRecipes(
            recipe(1, "Zucchini Bread"), recipe(2, "Apple Pie"), recipe(3, "Banana Pudding"))
        val recipes = (useCase() as Result.Success).data
        assertThat(recipes.map { it.title })
            .containsExactly("Apple Pie", "Banana Pudding", "Zucchini Bread").inOrder()
    }

    @Test fun `returns empty list when no recipes`() = runTest {
        assertThat((useCase() as Result.Success).data).isEmpty()
    }

    @Test fun `propagates repository error`() = runTest {
        fakeRepo.setShouldError(true)
        assertThat(useCase()).isInstanceOf(Result.Error::class.java)
    }
}

Three tests, three behaviors. No mocking framework. No Android dependency. Runs in milliseconds.

The Fake Repository Pattern

Good Prompt: Generate Fake Repository
Context: :core:testing, implements RecipeRepository from :core:domain.
Rules: TEST-002 (fake, not mock). MutableList storage + MutableSharedFlow.
Supports: addRecipes(), setShouldError(), clear().
When shouldError=true, all methods return Result.Error.
Under 50 lines. No Mockito, no MockK.

:core:testing — FakeRecipeRepository.kt

class FakeRecipeRepository : RecipeRepository {
    private val recipes = mutableListOf<Recipe>()
    private val recipesFlow = MutableSharedFlow<List<Recipe>>(replay = 1)
    private var shouldError = false

    fun addRecipes(vararg items: Recipe) { recipes.addAll(items); emit() }
    fun setShouldError(value: Boolean) { shouldError = value }
    fun clear() { recipes.clear(); emit() }
    private fun emit() { recipesFlow.tryEmit(recipes.toList()) }

    override fun observeRecipes(): Flow<List<Recipe>> = recipesFlow
    override suspend fun getRecipes(): Result<List<Recipe>> =
        if (shouldError) err() else Result.Success(recipes.toList())
    override suspend fun getRecipeById(id: Long): Result<Recipe> =
        if (shouldError) Result.Error(AppException.NotFound("Recipe"))
        else recipes.find { it.id == id }?.let { Result.Success(it) }
            ?: Result.Error(AppException.NotFound("Recipe"))
    override suspend fun saveRecipe(recipe: Recipe): Result<Unit> =
        if (shouldError) err() else { recipes.removeAll { it.id == recipe.id }
            recipes.add(recipe); emit(); Result.Success(Unit) }
    override suspend fun deleteRecipe(id: Long): Result<Unit> =
        if (shouldError) err() else { recipes.removeAll { it.id == id }
            emit(); Result.Success(Unit) }
    private fun err() = Result.Error(AppException.Database(RuntimeException("Fake")))
}

Tip: Fakes in :core:testing, Not in Each Module

All fakes live in :core:testing, which depends on :core:domain. Any module that needs fakes depends on :core:testing as a testImplementation. This avoids duplicating fakes across :feature:recipe, :feature:books, etc.

Why Fakes Beat Mocks

Mock (Fragile)
@Test fun `loads recipes`() = runTest {
    whenever(mockRepo.getRecipes())
        .thenReturn(Result.Success(testRecipes))
    viewModel.onEvent(LoadRecipes)
    verify(mockRepo).getRecipes()  // tests HOW
    verify(mockRepo, never()).deleteRecipe(any())
}
Fake (Resilient)
@Test fun `loads recipes into state`() = runTest {
    fakeRepo.addRecipes(apple, banana, cherry)
    viewModel.onEvent(LoadRecipes)
    assertThat(viewModel.state.value.recipes)
        .containsExactly(apple, banana, cherry)
    assertThat(viewModel.state.value.isLoading).isFalse()
}

Mock tests break on refactoring. Fake tests break only on behavior change. Always specify TEST-002: fakes over mocks.

Warning: AI's Mock Addiction

Even with TEST-002, AI generates @Mock and verify(). Scan for import org.mockito or import io.mockk. If found, reject and regenerate.

ViewModel Contract Tests

ViewModel tests verify the contract: given a state and an event, assert the resulting state and effects. This is MVI's killer testing advantage — every interaction is a pure state transition.

Good Prompt: Generate ViewModel Tests
Context: :feature:recipe tests, JUnit 5 + Turbine + Truth.
Rules: TEST-001/002/003. ViewModel: RecipeListViewModel.
Contract: RecipeListContract (State, Event, Effect).
Uses FakeRecipeRepository from :core:testing.

Scenarios: initial state (loading), LoadRecipes success/error,
SearchQueryChanged, RecipeClicked (→ Effect.NavigateToDetail),
DeleteRecipe success/error.

Pattern: arrange (fake) → act (onEvent) → assert state via
viewModel.state.value, assert effects via Turbine awaitItem().

ViewModel Contract Test Example

class RecipeListViewModelTest {
    private val fakeRepo = FakeRecipeRepository()
    private val getRecipes = GetRecipesUseCase(fakeRepo)
    private val deleteRecipe = DeleteRecipeUseCase(fakeRepo)
    private lateinit var viewModel: RecipeListViewModel

    @BeforeEach fun setup() {
        fakeRepo.addRecipes(recipe(1, "Apple Pie"), recipe(2, "Banana Bread"))
        viewModel = RecipeListViewModel(getRecipes, deleteRecipe)
    }

    @Test fun `load recipes populates state`() = runTest {
        viewModel.onEvent(RecipeListEvent.LoadRecipes)
        assertThat(viewModel.state.value.isLoading).isFalse()
        assertThat(viewModel.state.value.recipes).hasSize(2)
    }

    @Test fun `recipe click emits navigation effect`() = runTest {
        viewModel.effects.test {
            viewModel.onEvent(RecipeListEvent.RecipeClicked(1L))
            assertThat(awaitItem()).isEqualTo(RecipeListEffect.NavigateToDetail(1L))
        }
    }

    @Test fun `load error shows ui error`() = runTest {
        fakeRepo.setShouldError(true)
        viewModel = RecipeListViewModel(getRecipes, deleteRecipe)
        viewModel.onEvent(RecipeListEvent.LoadRecipes)
        assertThat(viewModel.state.value.error).isNotNull()
    }

    @Test fun `delete recipe removes from state`() = runTest {
        viewModel.onEvent(RecipeListEvent.LoadRecipes)
        viewModel.onEvent(RecipeListEvent.DeleteRecipe(1L))
        assertThat(viewModel.state.value.recipes).hasSize(1)
    }
}

Four tests, four behaviors. Turbine's awaitItem() makes Channel testing trivial.

Repository Tests With In-Memory Room

Testing Real Repositories Against Room

class RecipeRepositoryImplTest {
    private lateinit var db: RecipesDatabase
    private lateinit var repo: RecipeRepositoryImpl

    @BeforeEach fun setup() {
        db = Room.inMemoryDatabaseBuilder(context, RecipesDatabase::class.java)
            .allowMainThreadQueries().build()
        repo = RecipeRepositoryImpl(db.recipeDao(), RecipeMapper())
    }
    @AfterEach fun tearDown() { db.close() }

    @Test fun `save and retrieve recipe`() = runTest {
        val saveResult = repo.saveRecipe(recipe(id = 0, title = "Test Recipe"))
        assertThat(saveResult).isInstanceOf(Result.Success::class.java)
        val getResult = repo.getRecipes() as Result.Success
        assertThat(getResult.data).hasSize(1)
        assertThat(getResult.data.first().title).isEqualTo("Test Recipe")
    }

    @Test fun `get non-existent recipe returns not found`() = runTest {
        val result = repo.getRecipeById(999L) as Result.Error
        assertThat(result.exception).isInstanceOf(AppException.NotFound::class.java)
    }
}

Repository tests use in-memory Room — real SQLite, real DAOs, real mappers. Domain and presentation tests use fakes (fast, no Android). If a ViewModel test needs Room, you're testing the wrong thing.

Reviewing AI-Generated Tests

AI generates tests that look right but test the wrong thing. Here's the review checklist:

AI Test Quality Review

CheckPassFail
Tests behavior, not implementation? Asserts state/output Uses verify() to check method calls
Uses fakes, not mocks? FakeRecipeRepository @Mock or mockk()
Tests edge cases? Empty list, error, not found Only happy path
Readable test names? `delete recipe removes from state` test1(), testDelete()
Independent tests? Each test sets up its own state Tests depend on execution order
No sleeping? Uses runTest + Turbine Thread.sleep(1000) / delay(1000)
Correct assertion library? Truth: assertThat(x).isEqualTo(y) JUnit 4: assertEquals(x, y)
Bad: AI-Generated Test (Common Output)
// REJECT — violates TEST-001/002/003
@Test fun testGetRecipes() {
    val mockRepo = mockk<RecipeRepository>()                    // mock, not fake
    coEvery { mockRepo.getRecipes() } returns Result.Success(listOf())
    runBlocking {                                                // not runTest
        assertEquals(Result.Success(listOf<Recipe>()), useCase()) // not Truth
    }
    coVerify { mockRepo.getRecipes() }                           // tests HOW
}

MockK, runBlocking, assertEquals, verify — every rule violated. For more on reviewing AI-generated tests, see AI-CodeReview Chapter 3.

Compose UI Tests (The 5%)

UI tests are the smallest slice of the pyramid. They verify that the Composable renders correctly for a given state — not that business logic works (that's what ViewModel tests are for).

Compose UI Test: RecipeListScreen

class RecipeListScreenTest {
    @get:Rule val composeRule = createComposeRule()

    @Test fun `recipe list with items`() {
        composeRule.setContent {
            RecipeListScreen(
                state = RecipeListState(
                    recipes = listOf(recipe(1, "Apple Pie"), recipe(2, "Banana Bread")),
                    isLoading = false),
                onEvent = {})
        }
        composeRule.onNodeWithText("Apple Pie").assertIsDisplayed()
        composeRule.onNodeWithText("Banana Bread").assertIsDisplayed()
    }

    @Test fun `loading state shows indicator`() {
        composeRule.setContent {
            RecipeListScreen(state = RecipeListState(isLoading = true), onEvent = {})
        }
        composeRule.onNodeWithTag("loading_indicator").assertIsDisplayed()
    }
}

Because Composables are pure functions of State, UI tests are straightforward: create state → render → assert. No navigation, no ViewModels, no fakes.

Integration Tests: The Full Stack

Full-Stack Integration Test

class RecipeFlowIntegrationTest {
    private val fakeRepo = FakeRecipeRepository()
    private val getRecipes = GetRecipesUseCase(fakeRepo)
    private val saveRecipe = SaveRecipeUseCase(fakeRepo)
    private val deleteRecipe = DeleteRecipeUseCase(fakeRepo)

    @Test fun `full recipe lifecycle - create, list, delete`() = runTest {
        val recipe = Recipe(id = 0, title = "Test", servings = 4, isFavorite = false)
        assertThat(saveRecipe(recipe)).isInstanceOf(Result.Success::class.java)
        val listed = (getRecipes() as Result.Success).data
        assertThat(listed).hasSize(1)
        assertThat(deleteRecipe(listed.first().id)).isInstanceOf(Result.Success::class.java)
        assertThat((getRecipes() as Result.Success).data).isEmpty()
    }
}

Real use cases, fake repository. The test reads like a user story: create → list → delete → verify empty. Integration tests verify that layers connect correctly.

AI Testing Mistakes

MistakeWhat AI DoesRule / Fix
Mocks everywhere @Mock, mockk(), verify() TEST-002: fakes in :core:testing
Happy path only Tests success, ignores errors Every test class needs error + edge cases
runBlocking instead of runTest runBlocking { ... } TEST-003: use runTest (virtual time)
Thread.sleep() Waits for async operations Use Turbine's awaitItem()
JUnit 4 leftovers @Rule, assertEquals TEST-003: JUnit 5 + Truth
Testing implementation Verifies internal method calls Test outputs and state, not call chains

Exercise: Generate Your Test Suite

Exercise 10: The Test Generation Sprint

  1. Fakes first: Generate FakeRecipeRepository in :core:testing with TEST-002. Verify no Mockito/MockK imports.
  2. Domain tests: Generate tests for all use cases with TEST-001/003. Success + error + edge cases. Target: 35+ tests.
  3. ViewModel tests: Generate contract tests for all ViewModels. Provide Contract + fake + require Turbine. Target: 25+ per feature.
  4. Review: Apply the AI Test Quality Review table. Reject mocks, verify(), runBlocking, assertEquals.
  5. Count: Total all tests. If under 128, generate additional edge case tests.

Success criteria: 128+ tests. Zero mocks. Suite runs in under 5 seconds. Every test tests behavior, not implementation.
