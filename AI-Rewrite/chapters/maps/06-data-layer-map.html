The Big Idea

Implement the repository interfaces from Chapter 5 using Room entities, DAOs, and mappers — with strict rules preventing the most common AI data layer mistakes. The plumbing nobody sees, until it leaks.

📏 The Three Data Rules

🏷️
DATA-001: Explicit @ColumnInfo
Every entity field MUST have @ColumnInfo(name = "snake_case"). Never rely on property names. Prevents silent breaks on refactor.
🔤
DATA-002: Enums as String
Store enums with .name (not ordinal). Ordinals break when someone reorders the enum. AI defaults to ordinals — always enforce.
🔒
DATA-003: @Transaction on @Relation
Any DAO method returning @Relation types MUST be @Transaction. Without it, concurrent writes return inconsistent data.

🔧 The Data Layer Assembly Line

  1. 🗄️ Room Entities

    Feed raw SQL schema to AI. Get @Entity classes with explicit @ColumnInfo, String enums, foreign keys preserved.

  2. 📋 DAOs

    One per entity. Flow for observable queries, suspend for one-shot. @Transaction on all @Relation methods.

  3. 🔄 Type Converters

    Enum ↔ String converters. Register on @Database class with @TypeConverters.

  4. 🗺️ Mappers

    Extension functions: Entity.toDomain() and DomainModel.toEntity(). Stateless objects in :data:repository.

  5. 🔌 Repository Implementations

    Implement domain interfaces. Every method: try/catch → Result<T>. Mapper at every boundary.

  6. 🧪 Fake Repositories

    In :core:testing. In-memory, deterministic, shouldReturnError flag. Shared across all test sourceSets.

📜 The Repository Implementation Pattern

Input
Domain types from use cases (Recipe, Long, String)
Convert
Map domain → entity with .toEntity() before DAO calls

Call DAO methods (suspend / Flow)
Convert back
Map entity → domain with .toDomain() for return values
Wrap
try/catch everything → Result.Success or Result.Error
Error translation
SQLiteConstraintException → "already exists", SQLiteFullException → "storage full"

Missing @Transaction Inconsistent @Relation reads DATA-003: always on @Relation

🧪 Quick Quiz

Why must every entity field have explicit @ColumnInfo?

Renaming a Kotlin property without @ColumnInfo silently changes the database column name, breaking existing data. Explicit names decouple property names from schema.

Why store enums as String instead of ordinal?

Reordering enum values changes ordinal positions, silently corrupting all existing rows. String storage ("EASY", "MEDIUM") is immune to reordering.

Where do mappers live and why?

In :data:repository. They convert between Room entities (:data:local) and domain models (:core:domain). Neither module knows about the other's types — the mapper is the boundary translator.

Why is fallbackToDestructiveMigration() forbidden?

It silently deletes ALL user data when the schema changes. AI generates it because it's easy. Production apps MUST use explicit Migration objects.

What pattern does every repository method follow?

Map input to entity → call DAO → map output to domain → wrap in Result<T>. All inside try/catch so no raw exceptions escape the data layer.

🗝️ Key Takeaways

  • Two modules: :data:local (Room) and :data:repository (implementations + mappers).
  • Three rules: explicit @ColumnInfo (DATA-001), String enums (DATA-002), @Transaction on @Relation (DATA-003).
  • Mappers convert at the boundary. Entity types NEVER cross into domain or presentation.
  • Every repository method: try/catch → Result<T>. No raw exceptions escape the data layer.
  • Specific SQLite exceptions → specific error messages. Not just "something failed."
  • fallbackToDestructiveMigration() is FORBIDDEN. Always write explicit migrations.
  • Fake repositories in :core:testing for fast, deterministic tests without Room.
  • AI's sneakiest mistake: leaking entity types into ViewModels. Mappers exist for a reason.
  • Room 3.0 (alpha) is the KMP future. Stay on Room 2.x for the Android rewrite; re-evaluate when 3.0 stabilizes.
