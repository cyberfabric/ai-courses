The Big Idea

One Hilt @Module per Gradle module, with correct scopes and @Binds for interfaces. The electrical wiring that connects every layer — resolved entirely at compile time.

📏 The Four DI Rules

📦
DI-001: One @Module Per Gradle Module
DataLocalModule, DataRepositoryModule, AppModule. No giant dump-all-in-:app module.
🎯
DI-002: Scope Matching
@Singleton for DB + repos. Unscoped for use cases (stateless). @HiltViewModel implicit for ViewModels.
🔗
DI-003: @Binds Over @Provides
Interface → implementation = @Binds (abstract, zero overhead). @Provides only when construction needs logic.
🏷️
DI-004: Qualifier Location
Qualifiers defined in the module that defines the type (:core:common), not the provider (:app).

⚖️ @Binds vs @Provides

Aspect @Binds @Provides
Use when Interface → single implementation Construction needs builder logic
Module class abstract class object
Function abstract fun (no body) fun with return expression
Overhead Zero runtime — compile-time only Function call at runtime
Examples RecipeRepository ← RecipeRepositoryImpl Room.databaseBuilder(), OkHttpClient.Builder()

🎯 Scoping Decision Tree

  1. Is it a database or network client?

    Yes → @Singleton. Expensive to create, shared globally.

  2. Is it a repository implementation?

    Yes → @Singleton. Stateful (holds DAO refs), shared across features.

  3. Is it a use case?

    No scope. Stateless, cheap to create. New instance per injection.

  4. Is it a ViewModel?

    @HiltViewModel — scoped to the ViewModel lifecycle automatically.

  5. Is it a mapper or utility?

    No scope. Stateless objects need no lifecycle management.

🧪 Quick Quiz

Why one @Module per Gradle module instead of one giant AppModule?

Each module's bindings live where its types live. This keeps dependencies local, makes modules independently compilable, and prevents :app from becoming a dependency magnet.

When should you use @Binds vs @Provides?

@Binds: interface → single implementation (abstract function, zero overhead). @Provides: when construction needs logic (Room.databaseBuilder, OkHttpClient.Builder).

Why shouldn't use cases be @Singleton?

Use cases are stateless — they hold no data, just a repository reference. @Singleton keeps them in memory forever for no benefit. Unscoped = created when needed, garbage collected when done.

Why does :core:domain have no Hilt module?

Use cases only need @Inject constructor. Hilt sees the constructor, resolves the repository parameter from DataRepositoryModule, and creates the use case on demand. No @Module needed.

Where should qualifiers be defined?

In the module that defines the type (:core:common), not the module that provides the instance (:app). This way any module can reference the qualifier without depending on the provider.

🗝️ Key Takeaways

  • DI is the electrical wiring — connects domain ← data ← presentation at compile time.
  • One @Module per Gradle module. Never a giant dump-all AppModule.
  • @Binds for interfaces (zero overhead). @Provides only when construction needs logic.
  • @Singleton for DB + repos. Unscoped for use cases. @HiltViewModel for ViewModels.
  • :core:domain and :feature:* need no Hilt modules — @Inject constructor is enough.
  • Qualifiers defined where the type is defined, not where it's provided.
  • Convention plugins must include Hilt plugin + KSP in every module with Hilt annotations.
  • AI's default: @Provides everything, @Singleton everything, dump in AppModule. Override all three.
