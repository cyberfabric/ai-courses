Chapter 5

Extracting the Domain Layer

Memory Map
💡

The Big Idea

Extract pure Kotlin domain models, repository interfaces, and use cases into :core:domain — the module that depends on nothing and that everything else depends on. Get the heart of the architecture right before any data or UI code moves.

🏠 The Renovation Analogy

The domain layer is the load-bearing structure — beams and columns. Install framing before drywall. Fill :core:domain before anything else.

🧱 What Lives in :core:domain

📦
Domain Models
Immutable data classes (Recipe, Ingredient, CookBook, GroceryItem) and enums (DifficultyLevel, MealType, UnitOfMeasure).
🔀
Result Sealed Class
Custom Result<T> with Success/Error. Standardizes error handling across the entire domain boundary.
📜
Repository Interfaces
Contracts only — RecipeRepository, CookBookRepository, GroceryRepository. All methods suspend, all return Result<T>.
⚡
Use Cases
Single-action classes: operator fun invoke(). Stateless, constructor-injected, pure business logic.
✅
Validation
Business rules live in use cases (non-empty title, ≥1 ingredient, servings ≥ 1), NOT in the UI.
🚫
What Does NOT Belong
Zero Android imports. No Room, Hilt, Compose, or framework types. Pure Kotlin only.

📏 The Three Domain Rules

DOM-001: Pure Kotlin Only
Zero Android imports in :core:domain. If import android.* appears, the extraction has failed.
DOM-002: Result Sealed Class
All repository methods return Result<T>. No raw exceptions. No nullable returns for error cases.
DOM-003: Use Case Pattern
One public method: operator fun invoke(). Constructor injection. No state. No side effects.

⚡ The 8 Recipes App Use Cases

  1. GetRecipesUseCase

    Returns all recipes from the repository.

  2. GetRecipeByIdUseCase

    Returns a single recipe with its ingredients.

  3. SearchRecipesUseCase

    Search with min 2-character rule. Falls back to all recipes for short queries.

  4. SaveRecipeUseCase

    Validates: non-empty title, ≥1 ingredient, servings ≥ 1. Then saves.

  5. DeleteRecipeUseCase

    Deletes by ID, returns Result<Unit>.

  6. ToggleFavoriteUseCase

    Toggles and returns updated Recipe.

  7. GetGroceryListForRecipesUseCase

    Aggregates ingredients from selected recipes into a grocery list.

  8. AddRecipeToCookBookUseCase

    Associates a recipe with a cookbook.

⚠️ Common AI Mistakes

Mistake What AI Does Rule / Fix
Android leak Adds @Parcelize or import android.* DOM-001: grep for android imports
Anemic use cases Pass-through to repository with no logic Ask: does this add business value?
Room types in domain Uses RecipeEntity as model Mappers live in :data:repository
Missing Result wrapping Returns Recipe? for errors DOM-002: always Result<T>
Stateful use cases Adds var lastQuery DOM-003: stateless only
Multiple public methods Adds validate() alongside invoke() DOM-003: one public method

🧪 Quick Quiz

Why is :core:domain the first module you fill?

It depends on nothing and everything else depends on it. You can extract, test, and validate in complete isolation — pure Kotlin, no Android.

What does "code archaeology" mean in this context?

Using AI to excavate buried domain concepts from monolith code — data classes hidden in Cursor rows, business rules inline in onClick handlers, enums represented as magic integers.

Why use a custom Result instead of Kotlin's built-in kotlin.Result?

Kotlin's built-in Result can't be used as a normal return type (compiler restriction). A custom sealed class gives full control over the error model and extension functions.

Where does validation live in clean architecture?

In use cases, not in the UI. Business rules like "non-empty title" and "at least 1 ingredient" are domain concerns that belong in the domain layer.

What's the sneakiest AI mistake in domain extraction?

Leaking Android types — @Parcelize, Room entities, or android.* imports. Also: anemic use cases that are just pass-throughs to the repository with no actual business logic.

🗝️ Key Takeaways

  • :core:domain goes first — it depends on nothing, everything depends on it.
  • Use code archaeology prompts to excavate buried domain concepts from monolith code.
  • Three rules: pure Kotlin (DOM-001), Result<T> everywhere (DOM-002), single invoke() per use case (DOM-003).
  • Repository interfaces define contracts in domain. Implementations live in :data:repository.
  • Validation lives in use cases, not in the UI — business rules are domain concerns.
  • Test with fakes (FakeRecipeRepository in :core:testing), not mocks.
  • The sneakiest AI mistake: leaking Android types into pure Kotlin domain code.
  • End state: 20 files, 3 packages (model/repository/usecase), 0 Android imports, monolith untouched.
