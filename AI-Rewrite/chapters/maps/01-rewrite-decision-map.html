The Big Idea

AI doesn't just make rewrites faster — it makes them feasible for the first time, by compressing months of mechanical transformation into days of guided generation and review.

🧩 Core Concepts

🏚️
The Renovation Analogy
Peeling paint = refactor. Cracked foundation = rewrite. If the architecture is the debt, patches just spread the pain.
⚖️
Rewrite vs Refactor
6 signals decide: pain scope, test coverage, architecture soundness, team velocity, onboarding time, debt ratio. 4+ "rewrite" = do it.
🤖
AI Changes the Calculus
Team-reported 6-12 months → 4-8 weeks for a recipes-sized rewrite. AI maps dependencies, generates code from rules, and writes tests before you touch legacy code. Outcomes vary with codebase size and team practices.
🍳
The Recipes Monolith
Running case study: God Activities, raw SQL, no DI, no tests. One Activity = UI + business logic + data access in one file.
🏗️
10-Module Target
:app, :core:common, :core:design-system, :core:domain, :core:testing, :data:local, :data:repository, :feature:recipe, :feature:books, :feature:grocery.
🩺
AI Health Check
Prompt AI with specific classifications (God class, coupled, untestable, raw SQL) and target modules. Output = migration inventory.
📜
The Rewrite Manifesto
Non-negotiable contract: no new code in monolith, tests before deletion, feature parity preserved, rules loaded every session.
⚠️
Risk Matrix
HIGH: database + navigation. MEDIUM: UI migration. LOW: domain extraction + DI. Start low-risk, build safety net first.

⚖️ Rewrite vs Refactor Signals

Signal 🔧 Refactor 🏗️ Rewrite
Pain scope Localized to one module Systemic — everywhere
Tests Existing tests catch regressions Zero tests — flying blind
Architecture Sound, just needs cleanup No architecture at all
Velocity Slowing but shipping Features take 3x longer
Onboarding Week to ramp up Months of tribal knowledge
Tech debt Isolated pockets Debt is the architecture

📋 The Rewrite Process

  1. 🩺 AI Health Check

    Audit every file: classify as God class, coupled, untestable, raw SQL. Output a migration inventory table.

  2. 📏 Scope Estimation

    Feed inventory to AI. Get file counts, LOC estimates, test counts, risk levels, and dependency ordering per module.

  3. 📜 Write the Manifesto

    Three columns: what changes, what stays, what gets deleted. This is the gate for every AI session.

  4. 📐 Define Architecture Rules

    Before touching code: write the rules that constrain AI output. Rule IDs, severity levels, anti-patterns. (Chapter 2)

  5. 🏗️ Execute Module by Module

    Follow the migration order. Low-risk first, high-risk last. Tests before every deletion. (Chapters 4-9)

🧪 Quick Quiz

When should you rewrite instead of refactor?

When the problems are systemic (4+ rewrite signals): no tests, no architecture, extreme coupling, slow velocity, long onboarding, and debt baked into the structure itself.

How does AI change the rewrite risk equation?

AI compresses timelines (months → weeks), generates tests for legacy code before rewriting, maps dependency graphs automatically, and produces migration code from architecture rules.

What is the correct module migration order?

:core:domain → :data:local → :data:repository → :feature:* → :app. Domain first because everything depends on it. Low-risk first, high-risk last.

What are the four points of RULE RW-001 (the Rewrite Manifesto)?

(1) No new code in the monolith. (2) Every migrated file gets tests before deletion. (3) Feature parity is non-negotiable. (4) Architecture rules loaded into every AI session.

Why should you NOT start with database migration?

It's HIGH risk (data loss potential, SQLite quirks). Start with LOW risk modules (domain, DI) to build confidence and a test safety net before touching dangerous parts.

🗝️ Key Takeaways

  • Rewrite when the architecture is the problem — refactoring rearranges deck chairs on a sinking ship.
  • AI can compress rewrites from team-reported 6-12 months to 4-8 weeks for a recipes-sized codebase — treat the timeline as a feasibility signal, not a delivery promise.
  • The killer feature: AI generates tests for legacy code before you rewrite, so you're never flying blind.
  • Run the AI Health Check first: structured audit → migration inventory → scope estimation.
  • Write the Rewrite Manifesto before touching code: what changes, what stays, what dies.
  • Migrate low-risk first (:core:domain), high-risk last (database, navigation). Build the safety net before the high-wire act.
  • RULE PKG-001: Features never depend on features. RULE RW-001: The four non-negotiables of the manifesto.
