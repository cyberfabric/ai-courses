Chapter 9

Navigation, Errors, and Cross-Cutting Concerns

Memory Map
💡

The Big Idea

Cross-cutting concerns — navigation, errors, network, Gradle — are the hallways, fire alarms, city plumbing, and building codes that connect every room. They span all layers, and AI gets them wrong precisely because they require understanding boundaries, not just individual components.

🏗️ The Four Cross-Cutting Domains

🚪
Navigation
Hallways connecting rooms. Type-safe routes, feature-owned, assembled in :app. No raw strings, no navController leaks.
🚨
Error Handling
Fire alarms. Three-tier translation: ApiResult → AppException → UiError. Each layer translates, never passes raw signals.
🌐
Network Layer
City water supply. CallAdapter wraps all responses. Dual OkHttpClient: @Authenticated + @Public. Zero try/catch in repos.
📐
Gradle Conventions
Building codes. Convention plugins in build-logic/. Version catalog only. KSP-only, no KAPT. Feature build.gradle.kts under 10 lines.

🚪 The Navigation Rules

NAV-001: Route Constants in Owning Feature
:feature:recipe owns RecipeRoutes. :feature:books owns BookRoutes. :app only assembles the NavHost.
NAV-002: Type-Safe Routes Only
@Serializable data classes for routes. No raw strings like navigate("recipe/$id").
NAV-003: Features Don't Know Each Other
Cross-feature navigation via Effect → callback lambda → :app's NavHost resolves the target.

Navigation 3 (stable since Nov 2025, 1.1.x): Compose-first, NavMetadata DSL, shared elements. The Navigation 2.x type-safe routes pattern remains supported — adopt Nav 3 as a follow-up migration, not during the rewrite.

🌐 Network Layer Rules

NET-001: CallAdapter for ApiResult
Custom CallAdapter wraps every Retrofit response in ApiResult. Repositories never write try/catch around network calls.
NET-002: Dual OkHttpClient
@Authenticated (token interceptor) + @Public (no auth). Qualifiers defined in :core:common per DI-004.

📐 Gradle Convention Stack

  1. recipes.jvm.library

    Pure Kotlin modules: :core:domain, :core:common. No Android dependencies.

  2. recipes.android.library

    Base Android config: compileSdk, minSdk, test setup. Used by :data:* modules.

  3. recipes.android.hilt

    Hilt plugin + KSP for Hilt compiler. Applied by any module with DI.

  4. recipes.android.feature

    Extends library + hilt + Compose. Adds :core:domain, :core:common, :core:design-system, navigation, lifecycle.

  5. recipes.android.application

    :app module. Full Android app config, all feature modules, NavHost assembly.

Result: feature module build.gradle.kts is under 10 lines. All versions in libs.versions.toml.

🧪 Quick Quiz

Where do route constants live — :app, :core:navigation, or the owning feature?

In the owning feature module. :feature:recipe owns RecipeRoutes. :app only assembles the NavHost. A shared :core:navigation creates hidden coupling between all features.

How does :feature:recipe navigate to :feature:books without depending on it?

ViewModel emits an Effect → LaunchedEffect calls a callback lambda → the lambda was provided by :app's NavHost, which resolves the route to the books feature. No feature-to-feature dependency.

What are the three tiers of the error hierarchy?

Tier 1: ApiResult (network, in :data:repository). Tier 2: AppException (domain, in :core:domain). Tier 3: UiError (user strings, in :feature:*). Each layer translates — no tier is skipped.

Why does the CallAdapter eliminate try/catch in repositories?

The CallAdapter wraps every Retrofit response (success, HTTP error, network error) in ApiResult automatically. Repositories just call .toResult() to map to domain Result — no exception handling needed.

Why KSP over KAPT?

KSP is Kotlin-native (no Java stub generation), 2× faster compilation, better incremental build support, and is the recommended path for Hilt, Room, and Moshi annotation processing.

🗝️ Key Takeaways

  • Navigation = hallways. Type-safe @Serializable routes in owning features. NavHost assembles in :app.
  • Cross-feature navigation: Effect → callback lambda → :app resolves. Never feature → feature dependency.
  • Error handling = fire alarms. Three tiers: ApiResult → AppException → UiError. Each layer translates.
  • Network: CallAdapter wraps all calls in ApiResult. Dual OkHttpClient with @Authenticated/@Public qualifiers.
  • Gradle: Convention plugins per module type. Version catalog only. KSP everywhere, KAPT nowhere.
  • PKG-001: No forbidden imports. domain ← features → domain ← data. Validate after every AI pass.
  • AI's biggest cross-cutting mistake: ignoring module boundaries. It puts types in the wrong module and creates hidden coupling.
