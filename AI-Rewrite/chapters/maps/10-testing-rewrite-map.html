Chapter 10

Testing the Rewrite

Memory Map
💡

The Big Idea

The monolith had zero tests. The rewrite gets 128+. AI generates the suite, but you set the rules: fakes over mocks, behavior over implementation, unit tests dominate. The building inspector checks every room — systematically, not randomly.

📏 The Three Testing Rules

🏗️
TEST-001: Unit Tests Dominate
80% unit, 15% integration, 5% UI. Don't invert the pyramid. AI loves generating Espresso — override it.
🎭
TEST-002: Fakes Over Mocks
Hand-written fakes in :core:testing implement real interfaces. Mocks verify calls; fakes verify behavior.
🧰
TEST-003: JUnit 5 + Turbine + Truth
JUnit 5 lifecycle. Turbine for Flow/Channel. Truth for assertions. No JUnit 4 @Rule. No assertEquals.

📋 Test Generation Order

  1. Domain Layer

    Use cases + validation. Pure Kotlin, no Android. Input → output. Easiest to test.

  2. Data Layer (Fakes)

    Generate FakeRecipeRepository in :core:testing. Also test real repos with in-memory Room.

  3. Presentation Layer

    ViewModel contract tests: state + event → new state + effects. Uses fakes from step 2.

  4. Integration Tests

    Full stack: real ViewModel + real use case + fake repo. Verify layers connect correctly.

🧩 The Fake Repository Pattern

📦
Lives in :core:testing
All fakes in one module, shared via testImplementation. No duplication across features.
📋
Implements Real Interface
FakeRecipeRepository : RecipeRepository. Same contract as the real implementation.
💾
MutableList Storage
Stores data in-memory. addRecipes(), clear(), setShouldError() for test control.
💥
Error Simulation
setShouldError(true) makes all methods return Result.Error. Simulates failures without infrastructure.

⚠️ Common AI Mistakes

🤖
Mock Addiction
AI defaults to @Mock + verify(). Scan for import org.mockito / io.mockk — reject if found.
😊
Happy Path Only
Tests success, ignores errors. Every test class needs error + edge case tests.
⏰
runBlocking + sleep
Uses runBlocking (not runTest) and Thread.sleep() (not Turbine). Both are wrong.
🔍
Testing Implementation
Verifies internal method calls instead of asserting outputs. Breaks on every refactor.

🧪 Quick Quiz

Why fakes over mocks?

Fakes test behavior (WHAT the code does), mocks test implementation (HOW it does it). Fakes survive refactoring; mocks break when you change internal call patterns even if behavior is identical.

What's the test generation order and why?

Domain → Data → Presentation → Integration. Domain is pure Kotlin (easiest). Data fakes are needed by presentation tests. Presentation uses fakes from data. Integration combines everything.

Where do fakes live and why?

In :core:testing, which depends on :core:domain. Any module adds :core:testing as testImplementation. This avoids duplicating FakeRecipeRepository in every feature module.

How do you test ViewModel effects (navigation, snackbar)?

Turbine: viewModel.effects.test { awaitItem() }. Turbine suspends until the Channel emits, then you assert the Effect type. No Thread.sleep() needed.

What's wrong with: verify(mockRepo).getRecipes()?

It tests HOW (was getRecipes called?) not WHAT (are recipes in the state?). If you refactor the ViewModel to batch-load recipes differently, the test breaks even though behavior is correct.

🗝️ Key Takeaways

  • Building inspector analogy: systematic room-by-room inspection, not random spot checks.
  • TEST-001: 80% unit, 15% integration, 5% UI. Don't invert the pyramid.
  • TEST-002: Fakes in :core:testing. MutableList storage + error simulation. No Mockito, no MockK.
  • TEST-003: JUnit 5 + Turbine (Flows/Channels) + Truth (assertions). No JUnit 4 leftovers.
  • Order: domain (pure) → data (Room + fakes) → presentation (ViewModel contracts) → integration.
  • ViewModel tests: arrange (fake) → act (onEvent) → assert (state.value + Turbine effects).
  • 128+ tests, ~85% AI-generated, full suite under 2 seconds.
  • AI's default: mocks, happy-path-only, runBlocking, assertEquals. Override all four.
