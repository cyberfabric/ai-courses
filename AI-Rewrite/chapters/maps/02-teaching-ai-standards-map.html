The Big Idea

Architecture rules written before code are the highest-ROI investment in your rewrite. 2-3 hours of rules save 40-60 hours of fixing violations.

📊 The Rules-First ROI

Approach ⏰ Time 📋 Result
Rules After Code 40-60 hrs fixing AI repeats violations in every file; you fix retroactively
Rules Before Code 2-3 hrs writing AI follows rules from day one; PR review catches edge cases only

Hours are illustrative for a recipes-sized rewrite; the direction holds — rules up front beats retrofitting.

🏗️ The 10 Architecture Rule Areas

📦
PKG- Package Structure
Module boundaries, dependency direction, visibility. Features never depend on features.
🖥️
PRES- Presentation
MVI contracts, single StateFlow, Channel for effects, no logic in Composables.
🧠
DOM- Domain
Pure Kotlin use cases, operator invoke(), Result pattern, repository interfaces.
🗄️
DATA- Data
Explicit @ColumnInfo, enums as String, @Transaction on @Relation, entity mappers.
🚨
ERR- Error Handling
Error hierarchy, layer-boundary transformation, UI error mapping.
🌐
NET- Network
CallAdapter for ApiResult, dual OkHttpClient, token auth.
🧪
TEST- Testing
JUnit 5, fakes over mocks, ViewModel contract tests, coverage targets.
💉
DI- Dependency Injection
Per-module Hilt @Module, scope matching, prefer @Binds over @Provides.
⚙️
GRADLE- Build
Convention plugins, version catalog, KSP-only (no KAPT).
🧭
NAV- Navigation
Type-safe routes, route constants in owning feature, no raw strings.

⚖️ Rule Severity Levels

MUST
→
SHOULD
→
PREFER
MUST
Blocker. Code cannot merge if violated. "Feature modules MUST NOT depend on other feature modules."
SHOULD
Requires justification to break. "Use cases SHOULD have a single public method."
PREFER
Recommended but flexible. "PREFER @Binds over @Provides for interface bindings."

🔬 Anatomy of a Good Rule

  1. 🏷️ Unique ID

    Area prefix + number: PRES-001, DATA-003, PKG-002

  2. ⚖️ Severity Level

    [MUST], [SHOULD], or [PREFER] — sets enforcement expectations

  3. 📋 One-Sentence Description

    Clear, actionable constraint: "ViewModel MUST expose exactly one StateFlow<State>"

  4. ✅ GOOD Example

    Concrete code showing the correct pattern

  5. ❌ BAD Example

    Concrete code showing the specific anti-pattern to avoid

🧪 Quick Quiz

Why write rules BEFORE starting the migration?

Rules loaded upfront prevent systemic violations. Without them, AI repeats the same mistakes across every file, costing 40-60 hours of retroactive fixes vs 2-3 hours of rule-writing.

What are the three rule severity levels?

MUST (blocker, cannot merge), SHOULD (requires justification to break), PREFER (recommended but flexible).

What's the difference between a checklist and a .ctx file?

Checklists are full human-readable rules with GOOD/BAD code examples (for PR review). .ctx files are compressed to ≤25 lines with constraints only (for AI context windows).

What five elements make up a well-formed rule?

(1) Unique ID with area prefix, (2) severity [MUST/SHOULD/PREFER], (3) one-sentence description, (4) GOOD code example, (5) BAD code example showing the anti-pattern.

What should you do when you discover a new violation pattern during migration?

Fix the code AND update the rule immediately. Add the anti-pattern, update the .ctx file. Future AI sessions get the constraint automatically — the violation stops propagating.

📏 Key Rules Introduced

META-RULE
Every rule needs: unique ID, severity, description, GOOD example, BAD example.
CTX-001
.ctx files ≤25 lines, must reference full checklist, must list anti-patterns.
5-10 rules per area
Sweet spot: >15 and AI ignores late rules, <3 and you miss critical violations.
.ctx ≤ 300 tokens per area
Full checklists in CLAUDE.md eat 2,000+ tokens. .ctx files keep it lean.

🗝️ Key Takeaways

  • Write architecture rules before any migration code — 2-3 hours saves 40-60 hours.
  • Cover all 10 areas: PKG, PRES, DOM, DATA, ERR, NET, TEST, DI, GRADLE, NAV.
  • Three severity levels: MUST (blocker), SHOULD (justify), PREFER (flexible).
  • Every rule needs: ID + severity + description + GOOD example + BAD example.
  • Dual format: checklists for human review, .ctx files (≤25 lines) for AI context.
  • CLAUDE.md references .ctx files + inlines top 5 critical rules + lists forbidden patterns.
  • Run compliance tests before real migration: one ViewModel, use case, repo, DAO.
  • Rules evolve: fix violations AND update the rule immediately so AI never repeats them.
