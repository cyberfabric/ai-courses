The Big Idea

Moving day — the furniture goes in, doors open, and real people walk through. Individual layers worked in isolation. Now they must work together. Integrate bottom-up, test at every handoff, and never let AI "fix" errors by breaking architecture.

🔥 The Five Integration Failures

🔌
Missing @InstallIn
Hilt module compiles alone but fails when assembled. Fix: add @InstallIn(SingletonComponent::class).
🔒
Visibility Issues
Entity marked internal in :data:local, needed by :data:repository. Fix: make public.
🔄
Mapper Mismatch
Repository returns Entity instead of domain model. Fix: add .toDomain() mapper call.
📱
Missing @HiltAndroidApp
Application class has no annotation. Fix: @HiltAndroidApp class RecipesApp : Application().
🧭
Route Not Registered
Navigation destination not in NavHost graph. Fix: add composable<Route> { } in NavHost.

🔧 Error Resolution Cycle

  1. Build

    ./gradlew :app:assembleDebug — collect all errors.

  2. Diagnose

    Paste full errors + module graph to AI. AI identifies which module needs the fix.

  3. Fix

    Apply fix in the correct module. Never fix data issues in presentation.

  4. Test

    ./gradlew test — if existing tests break, the fix introduced a regression. Revert.

  5. Repeat

    Each cycle should reduce errors. If count increases, the fix was wrong.

📦 Release Build Checklist (INT-002)

ProGuard/R8 rules
Keep serialized models and Room database classes. AI-generated @Serializable routes need keep rules.
Build release APK
./gradlew :app:assembleRelease — debug builds hide ProGuard issues.
Test on real device
Install the signed APK. ProGuard can strip code that worked in debug. Test every navigation route.
Compare APK sizes
Rewrite APK should be within 15% of monolith. Multi-module + R8 often makes it smaller.
Compose recomposition check
Layout Inspector → recomposition counts. If Composable recomposes every frame, state isn't stable.

🧪 Quick Quiz

Why integrate bottom-up instead of top-down?

Bottom-up catches boundary mismatches (wrong types, missing mappers, broken contracts) at the lowest level first. Top-down hides these until three layers of debugging later.

What's the most common Hilt integration error?

Missing @InstallIn on a @Module. The module compiles independently but Hilt can't find it when assembling the full dependency graph in :app.

Why must you test the release APK, not just debug?

ProGuard/R8 strips and obfuscates code in release builds. @Serializable navigation routes, Room database classes, and reflection-based APIs can break when their metadata is stripped. Debug builds don't apply these optimizations.

How do you handle AI "fixes" that break architecture?

Reject them. If AI dumps bindings in :app to fix a missing binding, that violates DI-001. The correct fix is adding @InstallIn to the module that owns the binding. Every fix must be validated against rules.

What's the most overlooked edge case in a rewrite?

Process death and restore. ViewModels are cleared, state is lost. SavedStateHandle preserves critical state; Room persists data. AI almost never handles this without explicit prompting.

🗝️ Key Takeaways

  • Moving day analogy: furniture goes in, real people walk through. Integration is where theory meets reality.
  • INT-001: Integrate bottom-up (data → domain → presentation → navigation → app).
  • Five common failures: missing @InstallIn, visibility issues, mapper mismatches, missing @HiltAndroidApp, unregistered routes.
  • Error cycle: build → diagnose (paste full error to AI) → fix in correct module → test → repeat.
  • Never let AI "fix" errors by breaking architecture. Dump-in-:app, make-everything-public, skip-the-mapper are anti-patterns.
  • Feature parity checklist: every monolith feature must exist in the rewrite. Compare side-by-side.
  • Edge cases AI misses: empty DB, config change, timeout, deep links to deleted items, process death.
  • INT-002: Release build is the only truth. ProGuard strips AI-generated code. Test every route in release APK.
