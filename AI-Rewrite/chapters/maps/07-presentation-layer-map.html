Chapter 7

The Presentation Layer: MVI With AI

Memory Map
💡

The Big Idea

Every screen follows one rigid pattern: Contract (State + Event + Effect) → ViewModel (single StateFlow + Channel) → Composable (pure function of State). Give AI the Contract template and presentation rules, and it produces consistent, reviewable MVI code every time.

📜 The MVI Contract

📊
State
Single immutable data class. Everything the screen displays. Single source of truth via StateFlow.
👆
Event
Sealed interface. Every user action (click, type, swipe) is an Event. Dispatched via onEvent().
⚡
Effect
Sealed interface. One-shot side effects (navigation, snackbar). Via Channel, never State — avoids replay on config change.
🧠
ViewModel
Owns State + Effects. Receives Events via onEvent(). Delegates to use cases. No Android imports beyond ViewModel.
🎨
Composable
Pure function of State. collectAsStateWithLifecycle(). All actions → onEvent(). Zero business logic.
🔄
Reducer (optional)
Pure function: reduce(state, event) → state + effects. Extract for complex screens (6+ events). Testable without coroutines.

📏 The Three Presentation Rules

PRES-001: Contract Pattern
Every screen MUST have a Contract with State data class + Event sealed interface + Effect sealed interface.
PRES-002: Single State Source
One MutableStateFlow<State>, exposed as StateFlow. No scattered flows. No LiveData. No mutableStateOf() in ViewModel.
PRES-003: Effects via Channel
One-shot effects via Channel<Effect>, collected as Flow. Never put one-shots in State — they replay on config change.

✅ State Update Patterns

✅
Atomic copy()
_state.update { it.copy(isLoading = true, error = null) } — single atomic update, no intermediate renders.
❌
Multiple Updates
Two separate _state.value = ... calls render an intermediate invalid state between them.
✅
Derived in State
val filteredRecipes get() = ... as a computed property on the State data class.
❌
Derived in Composable
remember { filter... } in the Composable puts logic in the UI layer.

📁 Per-Screen File Pattern

  1. Contract.kt

    State + Event + Effect. The spec that constrains everything else.

  2. ViewModel.kt

    StateFlow + Channel + onEvent(). Delegates to use cases.

  3. Reducer.kt (optional)

    Pure function for complex screens. Extract when 6+ events.

  4. Screen.kt

    Composable. Collects state. Sends events. Zero logic.

Recipes app: 3 screens × 3-4 files = RecipeList, RecipeDetail, RecipeEdit

⚠️ Common AI Mistakes

Mistake What AI Does Fix
Mutable state mutableStateOf() in ViewModel PRES-002: single StateFlow<State>
SharedFlow effects SharedFlow replays on config change PRES-003: Channel prevents replay
Logic in Composable Filtering, sorting, formatting in UI Composables render State only
Multiple StateFlows Separate flows for each field Single State data class
Direct navigation navController in ViewModel Effect → callback lambda
Wrong lifecycle collectAsState() without lifecycle collectAsStateWithLifecycle()

🧪 Quick Quiz

What are the three parts of an MVI Contract?

State (single immutable data class), Event (sealed interface for user actions), Effect (sealed interface for one-shot side effects like navigation).

Why Channel instead of SharedFlow for effects?

SharedFlow replays cached values on configuration change (e.g., rotation). A navigation effect would fire twice. Channel delivers each effect exactly once.

When should you extract a Reducer?

When the screen has 6+ events or complex state transitions. The reducer is a pure function (state + event → new state + effects) testable without coroutines.

What's the #1 AI mistake in the presentation layer?

Putting business logic in Composables — filtering, sorting, formatting, conditional logic. Composables should be a pure function of State with zero logic.

Why use collectAsStateWithLifecycle() instead of collectAsState()?

It stops collecting when the lifecycle is below a threshold (e.g., app in background), preventing unnecessary recompositions and wasted resources.

🗝️ Key Takeaways

  • Every screen: Contract (State + Event + Effect) → ViewModel → Composable. No exceptions.
  • PRES-001: Contract pattern. PRES-002: single StateFlow. PRES-003: Channel for effects.
  • State updates must be atomic: _state.update { it.copy(...) }. Never multiple separate updates.
  • Composables are pure functions of State. Zero business logic. All actions → onEvent().
  • Navigation flows through Effects: ViewModel → Channel → LaunchedEffect → lambda callback.
  • Extract Reducer for complex screens (6+ events). Pure function, testable without coroutines.
  • Derive display values in the State data class, not in Composables with remember {}.
  • AI's sneakiest mistake: logic in Composables. If you see filtering, sorting, or conditionals — reject.
