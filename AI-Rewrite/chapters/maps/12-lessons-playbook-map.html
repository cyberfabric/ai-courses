The Big Idea

The real deliverable isn't the app — it's the blueprint. An eight-step, repeatable process for transforming any monolith into clean architecture using AI. Rules before code. Bottom-up generation. Fakes over mocks. AI is the laborer; you're the architect.

📋 The Eight-Step Rewrite (PLAY-001)

  1. Audit & Decide

    AI health check on monolith. Rewrite vs refactor. Define target architecture. (Ch 1)

  2. Write Rules

    50+ architecture rules, dual-format: review checklists + .ctx files. Load into CLAUDE.md. (Ch 2-3)

  3. Create Module Structure

    Multi-module Gradle. Convention plugins + version catalog. Empty modules that compile. (Ch 4)

  4. Extract Domain

    Pure Kotlin models, repository interfaces, use cases. Zero Android dependencies. (Ch 5)

  5. Build Data Layer

    Room entities, DAOs, mappers, repository implementations. Error handling at boundaries. (Ch 6)

  6. Create Presentation

    MVI Contracts, ViewModels, Composables, Hilt modules. Built on tested foundations. (Ch 7-8)

  7. Wire Cross-Cutting

    Navigation, errors, network, Gradle conventions. Connective tissue between layers. (Ch 9)

  8. Test & Integrate

    128+ tests. Feature parity. Performance validation. Release build. Ship it. (Ch 10-11)

🔍 Retrospective

✅
What Worked
Rules before code. Dual-format rules. Bottom-up generation. Fakes over mocks. MVI Contract testing.
❌
What Didn't
Cross-cutting on first pass. Trusting AI for module placement. Tests without explicit rules. One-shot migration prompts.
😮
What Surprised
AI excels at mechanical extraction. Worst at boundaries. Rules compound exponentially. Convention plugins were underrated.

🧪 Quick Quiz

What are the eight steps of the rewrite playbook?

1. Audit & Decide, 2. Write Rules, 3. Create Module Structure, 4. Extract Domain, 5. Build Data Layer, 6. Create Presentation, 7. Wire Cross-Cutting, 8. Test & Integrate.

What's AI's biggest strength and biggest weakness in a rewrite?

Strength: mechanical extraction (models from Activities, Room from SQL, mappers, boilerplate). Weakness: understanding boundaries (module placement, layer boundaries, error translation boundaries).

Why do rules compound during a rewrite?

Each chapter's rules build on previous ones. By Chapter 9, AI had 50+ rules as context and followed them. The investment in writing rules early pays exponential dividends in later chapters.

What transfers to other platforms?

The 8-step process, rules-first approach, bottom-up generation, domain layer purity, fakes over mocks, and contract testing. Only the specific tools change (Room → Core Data, Hilt → Swinject, etc.).

What's the difference between AI-Rewrite and AI-Factory?

Rewrite transforms an existing monolith (extract, migrate, replace). Factory builds from zero (design, build, create). Same rules, same prompting patterns, same architecture — different starting point.

🗝️ Key Takeaways

  • The blueprint is the real deliverable — not the app. An 8-step repeatable process for any rewrite.
  • Rules before code. 50+ architecture rules compound exponentially across chapters.
  • AI is the laborer, you're the architect. 85% AI-generated, but the 15% manual work is where quality comes from.
  • AI excels at extraction and boilerplate. AI fails at boundaries and module placement.
  • What worked: dual-format rules, bottom-up generation, fakes, MVI contracts.
  • What didn't: one-shot prompts, trusting AI for module placement, tests without rules.
  • 25+ rules across 10 areas = your team's shared vocabulary. Tool-agnostic, universally enforceable.
  • The playbook transfers to iOS, web, backend, KMP. Only the tools change — the process stays.
  • Next step: AI-Factory — same process, greenfield. If you can rewrite, building from scratch is easier.
