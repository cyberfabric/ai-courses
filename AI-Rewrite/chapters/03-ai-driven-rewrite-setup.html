Building the Workshop

Your CLAUDE.md from Chapter 2 already has architecture rules. Now you need rewrite-specific sections that tell AI about the migration context: what's been migrated, what hasn't, and what the rules of engagement are.

Rewrite-Specific Skills

Skills are reusable prompt templates that you invoke with a slash command. For a rewrite, you need four core skills that automate the repetitive parts of migration. Each lives in .claude/commands/.

Skills Directory Structure

.claude/
  commands/
    migrate-to-module.md      — Move a file to its target module
    extract-domain.md         — Extract domain models from legacy
    add-di.md                 — Generate Hilt module for a Gradle module
    generate-tests-for-migration.md  — Tests for code about to be deleted

Skill: /migrate-to-module

This is the workhorse skill. It takes a legacy file and moves it to the correct target module, transforming it along the way.

/migrate-to-module Skill Template
Given the legacy file at `$FILE_PATH`: 1. Read the file and identify which target module it belongs to (check docs/migration-tracker.md for the mapping) 2. Read the target module's .ctx rules 3. Create the new file in the target module following: - Package naming: com.recipes.{module}.{feature} - Architecture rules from the relevant .ctx files - Naming conventions from CLAUDE.md 4. Create or update the mapper if crossing a layer boundary 5. Update the migration tracker: mark this file as MIGRATED 6. Generate a test file for the migrated code 7. List all files in the legacy module that still reference this file DO NOT delete the legacy file yet — that happens after tests pass.
Seven-step workflow in one command. AI reads the mapping, applies rules, creates the file, writes tests, and updates tracking — all from a single /migrate-to-module invocation.

Skill: /extract-domain

This skill reads a God Activity or legacy class and extracts the pure domain logic: data classes, repository interfaces, and use cases.

/extract-domain Skill Template
Analyze `$LEGACY_FILE` and extract domain concepts: 1. Identify all data structures → create domain data classes in :core:domain 2. Identify all data operations → create repository interface in :core:domain 3. Identify all business logic → create use cases in :core:domain 4. Apply DOM-* rules from rules/ctx/domain.ctx 5. Every use case: single operator fun invoke(), returns Result 6. Every domain model: pure Kotlin, no Android annotations 7. Generate unit tests for each use case in :core:testing Output: list of created files with their full paths.
Targets domain extraction specifically. References the DOM-* rules. Produces testable output with a manifest of created files.

Skills: /add-di and /generate-tests-for-migration

/add-di

  • Reads a Gradle module's public API
  • Generates a Hilt @Module in the di/ package
  • Uses @Binds for interface bindings
  • Matches scope to module type (Singleton for data, ViewModelScoped for features)
  • Validates against DI-* rules

/generate-tests-for-migration

  • Reads a legacy file about to be deleted
  • Generates tests for its replacement in the new module
  • Focuses on behavior verification, not implementation
  • Uses fakes, not mocks (per TEST-* rules)
  • Tests must pass before the legacy file is removed
Skills are just markdown files with prompt templates. You can create new ones as you discover repetitive patterns during the rewrite. If you're doing the same thing three times, make it a skill.

Quality Gate Hooks

Skills tell AI what to do. Hooks tell AI what it can't do. A quality gate hook runs automatically on certain events and blocks the action if it violates a rule.

For a rewrite, you need three hooks:

1
No new files in legacy module
2
No forbidden patterns in new code
3
Tests required for every migration

Hook 1: Block New Legacy Code

This is the most critical hook. It prevents AI from adding new files to the monolith module during the rewrite.

settings.json Hook Configuration
{ "hooks": { "PreFileWrite": [ { "matcher": "app/src/main/java/legacy/**", "command": "echo 'BLOCKED: No new files in legacy module. Migrate to the target module instead.' && exit 1" } ] } }
Any attempt to create or modify files in the legacy path gets blocked with a clear message directing AI to the target module.

Hook 2: Forbidden Pattern Scanner

Runs after file creation and scans for patterns that should never appear in new code:

#!/bin/bash
# hooks/check-forbidden-patterns.sh
FILE="$1"

# Skip legacy files (they'll be deleted anyway)
if [[ "$FILE" == *"/legacy/"* ]]; then exit 0; fi

VIOLATIONS=""

# No LiveData in new code
if grep -q "LiveData" "$FILE" 2>/dev/null; then
  VIOLATIONS+="FORBIDDEN: LiveData found. Use StateFlow.\n"
fi

# No SQLiteDatabase in new code
if grep -q "SQLiteDatabase" "$FILE" 2>/dev/null; then
  VIOLATIONS+="FORBIDDEN: Raw SQLite found. Use Room.\n"
fi

# No Fragment in new code
if grep -q "extends Fragment\|: Fragment()" "$FILE" 2>/dev/null; then
  VIOLATIONS+="FORBIDDEN: Fragment found. Use Compose.\n"
fi

if [ -n "$VIOLATIONS" ]; then
  echo -e "$VIOLATIONS"
  exit 1
fi
Hooks are your automated reviewers. They catch the obvious violations before you even see the code. But they can't catch everything — architectural violations like "ViewModel calling repository directly" need human review against the checklist.

Hook 3: Migration Test Gate

Before any legacy file is deleted, verify that replacement tests exist and pass:

#!/bin/bash
# hooks/check-migration-tests.sh
FILE="$1"

# Only trigger on deletions in legacy module
if [[ "$FILE" != *"/legacy/"* ]]; then exit 0; fi

BASENAME=$(basename "$FILE" .kt)
# Check if tests exist for the replacement
TEST_COUNT=$(find . -path "*/test/*" -name "*${BASENAME}*Test.kt" | wc -l)

if [ "$TEST_COUNT" -eq 0 ]; then
  echo "BLOCKED: Cannot delete $BASENAME — no replacement tests found."
  echo "Run /generate-tests-for-migration first."
  exit 1
fi

The Migration Tracking System

You need to know, at any moment: what's been migrated, what's in progress, and what's remaining. This is the migration tracker — a markdown file that AI reads and updates during every migration operation.

docs/migration-tracker.md Template
## Migration Tracker — Recipes App Rewrite ### Summary | Status | Count | |--------|-------| | Migrated | 0 | | In Progress | 0 | | Remaining | 47 | ### Files | Legacy File | Target Module | Status | New Path | Tests | |-------------|---------------|--------|----------|-------| | RecipeActivity.kt | :feature:recipe | REMAINING | — | — | | RecipeDbHelper.kt | :data:local | REMAINING | — | — | | RecipeAdapter.kt | :feature:recipe | REMAINING | — | — | | NetworkUtils.kt | :core:common | REMAINING | — | — | | BookListActivity.kt | :feature:books | REMAINING | — | — | | GroceryActivity.kt | :feature:grocery | REMAINING | — | — | | ... | | | | | ### Migration Log (Skills append entries here as files are migrated) - [date] Migrated RecipeEntity → :data:local (3 tests added)
Machine-readable markdown table. AI updates the status column and new path as it migrates. The summary counts update automatically. The log provides an audit trail.

Every rewrite skill reads this tracker at the start and updates it at the end. You always know where the rewrite stands.

SETUP-002 Every migration skill MUST update docs/migration-tracker.md when it completes. Status transitions: REMAINING → IN_PROGRESS → MIGRATED. No file is deleted until status is MIGRATED and tests pass.

Branch Strategy for AI-Driven Rewrites

A rewrite touches every part of the codebase. You can't do it in one giant branch. Here's the strategy that keeps things manageable:

Branch Structure

main
 └── rewrite/setup          ← Module structure + empty modules (Ch 4)
      └── rewrite/domain     ← :core:domain extraction (Ch 5)
           └── rewrite/data  ← :data:* implementation (Ch 6)
                └── rewrite/presentation  ← :feature:* + ViewModels (Ch 7-8)
                     └── rewrite/navigation  ← Navigation + errors (Ch 9)
                          └── rewrite/tests  ← Full test suite (Ch 10)
                               └── rewrite/integration  ← Final cleanup (Ch 11)

Each branch maps to a course chapter and a layer of the architecture. The pattern:

  1. Create the branch from the previous rewrite branch
  2. Run the relevant skills to migrate that layer
  3. Run tests to verify the migration
  4. PR review against the layer's checklist
  5. Merge and create the next branch
One branch per layer, not one branch per file. AI can migrate dozens of files in a single session. The branch granularity matches the architecture layers, not individual files.
Never rewrite directly on main. Each branch should compile and pass tests independently. If the rewrite stalls, you can always ship from the last merged branch.

The Rewrite Dashboard

The migration tracker gives you file-level detail. The dashboard gives you the big picture — a prompt you run at the start of each session to see where things stand.

Dashboard Prompt
Read docs/migration-tracker.md and give me a rewrite status report: 1. Summary: files migrated / in progress / remaining (with percentages) 2. Per-module breakdown: how many files does each target module have? 3. Test coverage: how many migrated files have tests vs don't? 4. Blockers: any files marked IN_PROGRESS for more than one session? 5. Recommended next action: what should I migrate next based on dependency order and risk level? Keep the report under 30 lines.
Structured status check that AI generates from the tracker. The "recommended next action" turns a status report into a decision tool.
Pro tip: Run the dashboard prompt at the start of every AI session. It takes 10 seconds and ensures you and AI are aligned on what's been done and what's next. No more "wait, didn't we already migrate that file?"

Putting It All Together

Here's how the four systems interact during a typical migration session:

Dashboard
→
Pick File
→
Run Skill
→
Hooks Check
→
Tracker Update
  1. Dashboard: Run the status prompt. See 12 files remaining in :core:domain.
  2. Pick file: Choose RecipeRepository.kt from the tracker.
  3. Run skill: /extract-domain RecipeRepository.kt — AI creates the interface in :core:domain, the implementation stays in :data:repository for later.
  4. Hooks check: PreFileWrite hook verifies the new file isn't in the legacy path. Forbidden pattern scanner verifies no LiveData or raw SQL.
  5. Tracker update: Skill marks RecipeRepository.kt as MIGRATED in the tracker.

Repeat 10-15 times per session. That's a day's worth of manual migration work done in 2 hours.

Try It: Configure Your Rewrite Command Center

Exercise: Set Up Your Rewrite Infrastructure

Using your project from Chapters 1-2:

  1. Expand CLAUDE.md: Add a "Rewrite Status" section, a "Forbidden During Rewrite" section, and an "Available Skills" section. Reference your rules/ctx/ files.
  2. Create the skills: Write at least /migrate-to-module and /extract-domain in .claude/commands/. Use the templates above as starting points, customized with your module names.
  3. Configure hooks: Set up the "no new legacy code" hook in settings.json. Test it by asking AI to create a file in the legacy path — it should be blocked.
  4. Build the tracker: Create docs/migration-tracker.md with every file in your legacy module. Use the health check output from Chapter 1 to populate the target module column.
  5. Test the workflow: Run the dashboard prompt, pick one simple file, and migrate it using your skill. Verify the tracker updates and hooks fire.

Deliverable: A fully configured rewrite command center — CLAUDE.md, skills, hooks, and tracker — validated with one end-to-end migration.
