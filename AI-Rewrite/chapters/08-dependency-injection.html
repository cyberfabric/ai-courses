Chapter 8

Dependency Injection: Hilt Modules With AI

Per-module Hilt setup, scoping rules, and multi-module DI configuration

The Electrical Wiring

You've built the foundation (domain), installed the plumbing (data), and painted the walls (presentation). But nothing is connected. The ViewModel asks for a use case — where does it come from? The use case needs a repository — which implementation? The repository needs DAOs — from which database instance?

Dependency injection is the electrical wiring that connects every room to the panel. Hilt generates the wiring at compile time — but only if you tell AI the correct wiring diagram. One wrong scope, one missing @InstallIn, and the circuit doesn't close.

This chapter generates one Hilt @Module per Gradle module, with correct scopes, bindings, and qualifier placement.

The DI Rules

Rule DI-001: One @Module Per Gradle Module

Each Gradle module gets exactly one Hilt @Module in a di/ package. The module name follows the pattern {GradleModule}Module: DataLocalModule, DataRepositoryModule, FeatureRecipeModule. No dumping all bindings in :app.

Rule DI-002: Scope Matching

@Singleton for data and core modules (database, repositories). No scope annotation (unscoped) for use cases — they're stateless. @ViewModelScoped is implicit via @HiltViewModel. Never scope a stateless class.

Rule DI-003: @Binds Over @Provides

When binding an interface to its implementation, use @Binds (abstract function). Use @Provides only when construction needs logic (Room database builder, OkHttpClient, etc.). AI defaults to @Provides for everything — always enforce @Binds.

Why These Rules Matter for AI

Without DI-001, AI dumps 30 bindings into one massive AppModule. Without DI-002, it Singleton-scopes everything (including stateless use cases, wasting memory). Without DI-003, it creates unnecessary object instances via @Provides when @Binds would generate zero overhead. Load all three in every DI prompt.

The Monolith Had No DI

In the monolith, dependencies are created wherever they're needed. Activities instantiate helpers directly. There's no injection, no inversion, no testability.

Migration: Manual Construction → Hilt Injection

Before: Manual dependency creation in Activity
class RecipeListActivity : AppCompatActivity() {
    // Dependencies created inline — untestable, tightly coupled
    private lateinit var db: DatabaseHelper
    private lateinit var adapter: RecipeAdapter

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        db = DatabaseHelper(this)            // Direct construction
        adapter = RecipeAdapter(this)         // Direct construction
        loadRecipes()
    }

    private fun loadRecipes() {
        val cursor = db.readableDatabase.rawQuery(  // Direct DB access
            "SELECT * FROM recipes", null
        )
        // ... populate adapter with cursor data
    }
}
After: Hilt-injected ViewModel with use cases
@HiltViewModel
class RecipeListViewModel @Inject constructor(
    private val getRecipes: GetRecipesUseCase,
    private val searchRecipes: SearchRecipesUseCase,
    private val toggleFavorite: ToggleFavoriteUseCase,
    private val deleteRecipe: DeleteRecipeUseCase
) : ViewModel() {
    // Use cases injected — each backed by repo → DAO → database
    // All wiring handled by Hilt at compile time
    // Fully testable — swap use cases for fakes

    private val _state = MutableStateFlow(RecipeListContract.State())
    val state: StateFlow<RecipeListContract.State> = _state.asStateFlow()

    init { loadRecipes() }

    private fun loadRecipes() {
        viewModelScope.launch {
            _state.update { it.copy(isLoading = true) }
            getRecipes().onSuccess { recipes ->
                _state.update { it.copy(recipes = recipes, isLoading = false) }
            }
        }
    }
}

The Hilt Module Map

Here's the wiring diagram — which Gradle module gets which Hilt module, and what it binds.

Hilt Module Per Gradle Module

:data:local     → DataLocalModule      → @Provides RecipesDatabase, DAOs
:data:repository → DataRepositoryModule → @Binds RecipeRepository ← RecipeRepositoryImpl
:core:domain    → (no Hilt module)      → Use cases are unscoped, injected directly
:feature:recipe → (no Hilt module)      → @HiltViewModel handles injection
:feature:books  → (no Hilt module)      → @HiltViewModel handles injection
:app            → AppModule             → @Provides OkHttpClient, app-level singletons

Notice: :core:domain has no Hilt module. Use cases take repository interfaces via constructor — Hilt resolves them from DataRepositoryModule. Feature modules have no Hilt module either — @HiltViewModel handles ViewModel injection automatically.

Generating the Data Local Module

The :data:local module provides the Room database and DAOs. These need @Provides because they require builder logic.

Good: Data Local Hilt Module Prompt

Generate a Hilt @Module for :data:local.

BINDINGS NEEDED:
- RecipesDatabase (Room) — @Singleton, built with Room.databaseBuilder
- RecipeDao — from database.recipeDao()
- IngredientDao — from database.ingredientDao()

DI RULES:
- @Module @InstallIn(SingletonComponent::class)
- @Singleton for database instance
- @Provides for all (Room needs builder logic)
- Package: com.example.recipes.data.local.di
- Class name: DataLocalModule

Generated DataLocalModule

@Module
@InstallIn(SingletonComponent::class)
object DataLocalModule {

    @Provides
    @Singleton
    fun provideDatabase(
        @ApplicationContext context: Context
    ): RecipesDatabase = Room.databaseBuilder(
        context,
        RecipesDatabase::class.java,
        "recipes.db"
    ).build()

    @Provides
    fun provideRecipeDao(db: RecipesDatabase): RecipeDao = db.recipeDao()

    @Provides
    fun provideIngredientDao(db: RecipesDatabase): IngredientDao = db.ingredientDao()
}

DAOs are not @Singleton — they're lightweight accessors created from the database instance. Only the database itself is Singleton-scoped.

Generating the Data Repository Module

The :data:repository module binds repository interfaces to their implementations. This is where @Binds shines.

Generated DataRepositoryModule

@Module
@InstallIn(SingletonComponent::class)
abstract class DataRepositoryModule {

    @Binds
    @Singleton
    abstract fun bindRecipeRepository(
        impl: RecipeRepositoryImpl
    ): RecipeRepository

    @Binds
    @Singleton
    abstract fun bindCookBookRepository(
        impl: CookBookRepositoryImpl
    ): CookBookRepository

    @Binds
    @Singleton
    abstract fun bindGroceryRepository(
        impl: GroceryRepositoryImpl
    ): GroceryRepository
}

AI Will Use @Provides Here — Don't Let It

AI's default for interface binding is @Provides fun bindRepo(): RecipeRepository = RecipeRepositoryImpl(dao). This creates a new instance manually. @Binds is an abstract function that tells Hilt "when someone asks for RecipeRepository, give them RecipeRepositoryImpl" — zero runtime overhead, the binding is resolved at compile time.

Note: RecipeRepositoryImpl needs an @Inject constructor for this to work:

Constructor Injection on Implementation

class RecipeRepositoryImpl @Inject constructor(
    private val recipeDao: RecipeDao,
    private val ingredientDao: IngredientDao
) : RecipeRepository {
    // ... implementation from Chapter 6
}

Use Cases: No Module Needed

Use cases are stateless. They don't need a Hilt module or a scope — just an @Inject constructor.

Use Case with @Inject Constructor

class GetRecipesUseCase @Inject constructor(
    private val recipeRepository: RecipeRepository
) {
    suspend operator fun invoke(): Result<List<Recipe>> =
        recipeRepository.getAll()
}

Hilt sees the @Inject constructor, knows how to provide RecipeRepository (from DataRepositoryModule), and creates the use case on demand. No scope means a new instance every time it's injected — which is fine because it's stateless.

Don't Singleton-Scope Use Cases

AI loves to add @Singleton to everything. Use cases are stateless functions. Singleton-scoping them wastes memory by keeping them alive for the entire app lifetime. Unscoped = created when needed, garbage-collected when done.

ViewModels: @HiltViewModel

Feature module ViewModels don't need a Hilt module either. @HiltViewModel + @Inject constructor is all Hilt needs.

ViewModel with @HiltViewModel

@HiltViewModel
class RecipeListViewModel @Inject constructor(

    private val searchRecipes: SearchRecipesUseCase,
    private val toggleFavorite: ToggleFavoriteUseCase,
    private val deleteRecipe: DeleteRecipeUseCase

The full dependency chain: RecipeListViewModel ← use cases ← RecipeRepository (bound via @Binds) ← RecipeRepositoryImpl ← DAOs ← RecipesDatabase. All resolved at compile time by Hilt.

Qualifier Placement

When you have multiple implementations of the same type (e.g., two OkHttpClients for authenticated and unauthenticated requests), qualifiers disambiguate.

Qualifier Definition and Usage

// Define qualifiers in the module that DEFINES the type
// :core:common/di/Qualifiers.kt
@Qualifier
@Retention(AnnotationRetention.BINARY)
annotation class AuthenticatedClient

@Qualifier
@Retention(AnnotationRetention.BINARY)
annotation class UnauthenticatedClient

// Use in the module that PROVIDES the instances
// :app/di/AppModule.kt
@Module
@InstallIn(SingletonComponent::class)
object AppModule {
    @Provides @Singleton @AuthenticatedClient
    fun provideAuthClient(interceptor: AuthInterceptor): OkHttpClient = ...

    @Provides @Singleton @UnauthenticatedClient
    fun providePublicClient(): OkHttpClient = ...
}

Rule DI-004: Qualifier Location

Qualifiers are defined in the module that defines the type (usually :core:common), not the module that provides the instance. This ensures any module can reference the qualifier without depending on the provider.

Multi-Module Discovery

In a multi-module project, Hilt must discover all @Module classes across all Gradle modules. This works automatically as long as every module applies the Hilt plugin.

Hilt Plugin in Convention Plugins

// build-logic/convention/src/main/kotlin/recipes.android.feature.gradle.kts
plugins {
    id("com.android.library")
    id("dagger.hilt.android.plugin")  // Enables Hilt in this module
    id("com.google.devtools.ksp")      // KSP for Hilt code generation
}

dependencies {
    implementation(libs.hilt.android)
    ksp(libs.hilt.compiler)
}

If a module has @Module or @HiltViewModel but doesn't apply the Hilt plugin, the bindings are invisible. AI sometimes generates bindings in a module without checking that Hilt is enabled there. Always verify the convention plugin chain.

Testing DI Configuration

After generating all Hilt modules, verify the dependency graph resolves correctly. The simplest test: compile and run.

DI Verification Steps

// 1. Compile — catches most binding errors at build time
./gradlew assembleDebug

// 2. Run the app — catches runtime-only issues
//    Look for: MissingBinding, DuplicateBindings, CyclicDependency

// 3. For complex graphs, use Hilt's component tree visualization:
// Build → Analyze APK → search for "Hilt" generated sources

Hilt's compile-time verification catches most errors. If the project compiles with all @Module classes in place, the wiring is correct. Runtime crashes are rare — but when they happen, it's usually a missing @Module or a scope mismatch.

Try It: Wire Your Modules

Exercise: Generate Hilt Modules for Your Project

  1. Map the dependency graph: For each Gradle module, list what it needs injected and what it provides to others.
  2. Generate DataLocalModule: @Provides for Room database and DAOs. @Singleton for the database.
  3. Generate DataRepositoryModule: @Binds for each repository interface → implementation. @Singleton scope.
  4. Add @Inject constructors: Repository implementations, use cases. No Hilt modules needed for these.
  5. Add @HiltViewModel: To each ViewModel. Verify @Inject constructor with use case parameters.
  6. Verify convention plugins: Confirm Hilt plugin + KSP in every module that has Hilt annotations.
  7. Compile: ./gradlew assembleDebug. Fix binding errors until it builds clean.
  8. Run the review: Check DI-001 through DI-004. No giant AppModule, no @Provides for interfaces, no scoped use cases.

Deliverable: All modules wired with Hilt. Compile succeeds. The full chain resolves: ViewModel ← use cases ← repositories ← DAOs ← database.
