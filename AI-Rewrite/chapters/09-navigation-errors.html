Chapter 9

Navigation, Errors, and Cross-Cutting Concerns

Type-safe navigation, error hierarchy, network layer, and Gradle rules

Connecting the Rooms

The house is built. Foundation (domain), plumbing (data), walls (presentation), electrical wiring (DI) — all in place. But you can't get from the kitchen to the bedroom. There are no hallways, no doors, no corridors connecting the rooms.

Navigation is the hallway system. Error handling is the fire alarm — wired through every room, translating low-level sensor data into messages humans understand. The network layer is the plumbing that connects to the city water supply. And Gradle conventions are the building codes that keep everything up to standard.

This chapter tackles the cross-cutting concerns that span all layers. They're the unglamorous parts of architecture that AI loves to get wrong — because they require understanding boundaries, not just individual components.

The Navigation Rules

Rule NAV-001: Route Constants in Owning Feature

Each feature module defines its own route constants. :feature:recipe owns RecipeRoutes. :feature:books owns BookRoutes. The :app module only assembles the NavHost — it never invents route strings.

Rule NAV-002: Type-Safe Routes Only

No raw strings for navigation arguments. Use Kotlin serialization-based type-safe routes or sealed classes with typed parameters. If AI generates navController.navigate("recipe/$id"), reject it.

Rule NAV-003: Features Don't Know Each Other

Feature modules never depend on each other. Cross-feature navigation uses route interfaces defined in :core:common and resolved in :app. If :feature:recipe needs to navigate to :feature:books, it emits an Effect — the app-level NavHost handles the routing.

From startActivity() to Type-Safe Routes

Migration: Intent-Based Navigation → Compose Navigation

Before: Monolith (RecipeListActivity.kt)
// Scattered throughout Activities
btnViewRecipe.setOnClickListener {
    val intent = Intent(this, RecipeDetailActivity::class.java)
    intent.putExtra("RECIPE_ID", recipe.id)
    intent.putExtra("SOURCE", "list")
    startActivity(intent)
}

// In RecipeDetailActivity
btnEditRecipe.setOnClickListener {
    val intent = Intent(this, RecipeEditActivity::class.java)
    intent.putExtra("RECIPE_ID", recipeId)
    startActivityForResult(intent, EDIT_REQUEST_CODE)
}

// In RecipeDetailActivity.onActivityResult
override fun onActivityResult(requestCode: Int, resultCode: Int, data: Intent?) {
    if (requestCode == EDIT_REQUEST_CODE && resultCode == RESULT_OK) {
        loadRecipe() // reload
    }
}
After: :feature:recipe — RecipeRoutes.kt
// Route definitions in the owning feature
@Serializable data class RecipeList(val filter: String? = null)
@Serializable data class RecipeDetail(val recipeId: Long)
@Serializable data class RecipeEdit(val recipeId: Long)

// In :app — NavHost assembly
NavHost(navController, startDestination = RecipeList()) {
    composable<RecipeList> { entry ->
        RecipeListScreen(
            filter = entry.toRoute<RecipeList>().filter,
            onRecipeClick = { navController.navigate(RecipeDetail(it)) },
            onAddClick = { navController.navigate(RecipeEdit(recipeId = 0L)) })
    }
    composable<RecipeDetail> { entry ->
        RecipeDetailScreen(
            recipeId = entry.toRoute<RecipeDetail>().recipeId,
            onEditClick = { navController.navigate(RecipeEdit(it)) },
            onBack = { navController.popBackStack() })
    }
    composable<RecipeEdit> { entry ->
        RecipeEditScreen(
            recipeId = entry.toRoute<RecipeEdit>().recipeId,
            onSaved = { navController.popBackStack() },
            onBack = { navController.popBackStack() })
    }
}

Notice: startActivityForResult is gone. The edit screen calls onSaved, which pops the back stack. The list screen recomposes automatically because it collects from a StateFlow that Room keeps updated.

Cross-Feature Navigation Pattern

When :feature:recipe needs to navigate to :feature:books, it can't depend on that module. The pattern:

The Inter-Feature Navigation Flow

  1. ViewModel emits Effect.NavigateToCookbook(recipeId)
  2. Screen's LaunchedEffect calls the navigation callback lambda
  3. The lambda was provided by :app's NavHost: onAddToCookbook = { id -> navController.navigate(CookbookDetail(id)) }
  4. No feature module references any other feature module

Tip: AI Loves to Create Shared Navigation Modules

AI frequently generates a :core:navigation module that imports all feature routes. This creates a hidden coupling — every feature transitively depends on every other. Keep route definitions in owning features. The only assembler is :app.

What about Navigation 3? Jetpack Navigation 3 went stable in November 2025 (1.1.x as of April 2026) with a Compose-first design, NavMetadata DSL, and shared elements. The Navigation 2.x type-safe routes pattern this chapter teaches remains fully supported and is the right choice for an incremental rewrite; consider Navigation 3 as a follow-up migration once the rewrite is complete.

The Error Hierarchy

Errors are fire alarms. The smoke detector (network layer) reports "HTTP 401." The alarm panel (repository) translates it to "authentication failed." The room display (UI) shows "Please sign in again." Each layer translates, never passes raw signals upstream.

Rule ERR-001: Three-Tier Error Hierarchy

Network errors → ApiResult. Repository errors → AppResult / domain Result<T>. UI errors → UiError (user-visible strings). No layer skips a tier. No IOException reaches the ViewModel. No HTTP status code reaches the Composable.

Migration: Raw Exceptions → Three-Tier Error Handling

Before: Monolith (RecipeListActivity.kt)
// Raw exceptions everywhere
try {
    val recipes = api.getRecipes()
    adapter.submitList(recipes)
} catch (e: IOException) {
    Toast.makeText(this, "Network error: ${e.message}", Toast.LENGTH_LONG).show()
} catch (e: HttpException) {
    if (e.code() == 401) {
        startActivity(Intent(this, LoginActivity::class.java))
    } else {
        Toast.makeText(this, "Server error: ${e.code()}", Toast.LENGTH_LONG).show()
    }
} catch (e: Exception) {
    Toast.makeText(this, "Something went wrong", Toast.LENGTH_LONG).show()
}
After: Three Tiers
// Tier 1: Network layer (:data:repository) — CallAdapter wraps responses
sealed interface ApiResult<out T> {
    data class Success<T>(val data: T) : ApiResult<T>
    data class ApiError(val code: Int, val message: String) : ApiResult<Nothing>
    data class NetworkError(val exception: IOException) : ApiResult<Nothing>
}

// Tier 2: Repository — translates to domain Result<T>
override suspend fun getRecipes(): Result<List<Recipe>> = when (val response = api.getRecipes()) {
    is ApiResult.Success -> Result.Success(response.data.map { it.toDomain() })
    is ApiResult.ApiError -> Result.Error(AppException.Server(response.code, response.message))
    is ApiResult.NetworkError -> Result.Error(AppException.Network(response.exception))
}

// Tier 3: ViewModel — translates to UiError for display
private fun handleError(error: AppException): UiError = when (error) {
    is AppException.Network -> UiError("Check your connection and try again")
    is AppException.Server -> if (error.code == 401) UiError("Session expired. Please sign in.")
        else UiError("Server error. Try again later.")
    is AppException.NotFound -> UiError("Recipe not found")
    is AppException.Validation -> UiError(error.message)
}

Generating the Error Hierarchy

Good Prompt: Generate Error Types
Context: Multi-module Android app with Clean Architecture.
Modules: :data:repository (network + DB), :core:domain, :feature:*

Rules: ERR-001 (three-tier error hierarchy). No raw exceptions cross boundaries.
- Tier 1: ApiResult<T> in :data:repository — wraps Retrofit responses
- Tier 2: AppException in :core:domain — Network, Server(code, msg),
  NotFound, Validation, Database, Authentication
- Tier 3: UiError in :feature:* ViewModels — user-friendly strings

Generate: ApiResult sealed interface, AppException hierarchy, toResult()
extension, and mapToUiError() helper. Tell me which module each file goes in.

The key: tell AI exactly which types live in which module. Without module placement, it dumps everything in :app or creates circular dependencies.

Warning: AI Puts Error Types in the Wrong Module

AI commonly puts ApiResult in :core:domain (leaks network concepts into domain) or AppException in :data:repository (feature modules can't see it). AppException lives in :core:domain because both data and feature modules reference it. Review module placement for every generated error type.

The Network Layer Rules

Rule NET-001: CallAdapter for ApiResult

Never let raw Retrofit Response<T> escape to repository code. A custom CallAdapter wraps every response in ApiResult<T> automatically. The repository never writes try/catch around network calls — the adapter handles it.

Rule NET-002: Dual OkHttpClient

Two OkHttpClient instances: one for authenticated requests (with token interceptor), one for public endpoints. Qualified with @Authenticated and @Public in Hilt. AI defaults to a single client — override this.

CallAdapter: Zero Try/Catch in Repositories

// :data:repository — ApiResultCallAdapter.kt
class ApiResultCallAdapter<T>(private val resultType: Type) : CallAdapter<T, Call<ApiResult<T>>> {
    override fun responseType(): Type = resultType
    override fun adapt(call: Call<T>): Call<ApiResult<T>> = ApiResultCall(call)
}

// The repository becomes clean:
override suspend fun getRecipes(): Result<List<Recipe>> {
    return api.getRecipes().toResult { it.map { dto -> dto.toDomain() } }
}
// No try/catch. ApiResult handles network + HTTP errors. toResult() maps to domain Result.

Compare that to the monolith's try/catch nesting. The CallAdapter eliminates an entire category of boilerplate — and an entire category of AI mistakes.

Dual OkHttpClient Setup

Good Prompt: Generate Network Module
Context: :data:repository module, Hilt DI, OkHttp + Retrofit.

Rules: NET-001 (CallAdapter wraps all calls in ApiResult), NET-002 (two
OkHttpClient instances: @Authenticated with token interceptor, @Public
without), DI-004 (qualifiers in :core:common, not :data:repository)

Generate: @Authenticated/@Public qualifiers in :core:common, TokenInterceptor,
NetworkModule with dual OkHttpClient + Retrofit, ApiResultCallAdapterFactory.

Following DI-004, the @Authenticated and @Public qualifiers live in :core:common, not in :data:repository. This way any module can reference them without depending on the data layer.

Package Structure Validation

Rule PKG-001: No Forbidden Dependencies

:core:domain must never import from :data:* or :feature:*. :feature:* must never import from another :feature:*. The dependency graph is strictly unidirectional: features → domain ← data. Validate this after every AI generation pass.

AI doesn't inherently understand module boundaries. After generating cross-cutting code, run a validation pass:

Good Prompt: Validate Package Dependencies
Scan all .kt files for forbidden imports. Allowed graph:
  :feature:* → :core:domain, :core:common, :core:design-system
  :data:local → :core:domain, :core:common
  :data:repository → :data:local, :core:domain, :core:common
  :app → all modules
Forbidden: :core:domain → :data:*/:feature:*, :feature:* → :feature:*,
  :data:local → :feature:*/:data:repository

Report: file path, line number, forbidden import, rule violated.

The Gradle Rules

Rule GRADLE-001: Convention Plugins + Version Catalog

All build configuration via convention plugins in build-logic/. Dependencies via libs.versions.toml version catalog. No hardcoded version strings in any build.gradle.kts. No buildscript blocks. KSP only — no KAPT.

Bad: Hardcoded Versions
// :feature:recipe/build.gradle.kts
plugins {
    id("com.android.library")
    id("org.jetbrains.kotlin.android")
    kotlin("kapt") // KAPT!
    id("dagger.hilt.android.plugin")
}
dependencies {
    implementation("androidx.room:room-runtime:2.8.4")
    ksp("com.google.dagger:hilt-compiler:2.59.2")
}
Good: Convention Plugin + Catalog
// :feature:recipe/build.gradle.kts
plugins {
    id("recipes.android.feature")  // convention plugin
}
dependencies {
    implementation(libs.room.runtime)
    ksp(libs.hilt.compiler)  // KSP, not KAPT
}

// Convention plugin handles: Android config, Compose,
// Hilt plugin, KSP, test dependencies, etc.
Good Prompt: Generate Convention Plugins
Context: Multi-module Android project, composite builds for build-logic.
Rules: GRADLE-001 (convention plugins + version catalog, KSP only, no KAPT)

Plugin types: recipes.android.library (base), recipes.android.feature
(library + Compose + Hilt + KSP), recipes.android.application (app),
recipes.jvm.library (pure Kotlin: :core:domain, :core:common)

Generate: build-logic build files, convention plugin .kt files per type,
libs.versions.toml entries. Minimum plugins per type. Feature plugin adds

Convention Plugin: Feature Module

build-logic — RecipesAndroidFeaturePlugin.kt

class RecipesAndroidFeaturePlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        pluginManager.apply {
            apply("recipes.android.library")      // base Android config
            apply("recipes.android.hilt")          // Hilt + KSP
            apply("org.jetbrains.kotlin.plugin.compose")
        }
        dependencies {
            "implementation"(project(":core:domain"))
            "implementation"(project(":core:common"))
            "implementation"(project(":core:design-system"))
            "implementation"(libs.findLibrary("androidx.lifecycle.viewmodel.compose").get())
            "implementation"(libs.findLibrary("androidx.navigation.compose").get())
            "implementation"(libs.findLibrary("androidx.hilt.navigation.compose").get())
        }
    }
}

// Usage in :feature:recipe/build.gradle.kts — just one line:
// plugins { id("recipes.android.feature") }

The plugin declares the dependencies every feature needs. Individual feature modules only add feature-specific dependencies. AI generates one convention plugin per module type — not per module.

Warning: AI Ignores Your Version Catalog

Even when you provide the catalog, AI will generate implementation("group:artifact:version") with hardcoded strings. Add this to your rules: "All dependencies MUST use the version catalog. Hardcoded version strings are forbidden." Check every generated build.gradle.kts.

The Cross-Cutting Review Checklist

After every AI generation pass, verify:

  • Navigation: All routes are @Serializable data classes, defined in owning feature?
  • Navigation: NavHost is in :app only? No navController refs outside NavHost?
  • Navigation: Cross-feature navigation uses Effect → callback, not direct navigation?
  • Errors: Network errors wrapped in ApiResult by CallAdapter?
  • Errors: Repositories return Result<T> with AppException, never raw exceptions?
  • Errors: ViewModels translate to UiError — no HTTP codes in UI?
  • Network: Dual OkHttpClient with @Authenticated/@Public qualifiers?
  • Gradle: Convention plugins for all module types? No raw plugin application?
  • Gradle: KSP everywhere, KAPT nowhere?
  • Gradle: All versions in libs.versions.toml, none hardcoded?
  • Packages: No forbidden imports across module boundaries?

Exercise: Wire Your Cross-Cutting Concerns

Exercise 9: The Cross-Cutting Audit

  1. Navigation: Generate type-safe routes for your app's screens. Provide AI with NAV-001/002/003 rules, your feature module list, and which screens need cross-feature navigation. Verify no raw string routes appear.
  2. Errors: Generate the three-tier error hierarchy. Provide AI with ERR-001, your module structure, and the list of error scenarios (network timeout, 401, 404, validation failure, database error). Verify each type lives in the correct module.
  3. Network: Generate the CallAdapter and dual OkHttpClient. Provide AI with NET-001/002 and DI-004. Verify the qualifiers are in :core:common, not :data:repository.
  4. Gradle: Generate convention plugins for your module types. Provide AI with GRADLE-001 and your version catalog. Verify every feature module's build.gradle.kts is under 10 lines.
  5. Validate: Run the package dependency scan prompt from this chapter. Fix every violation before proceeding.

Success criteria: Zero raw string routes. Zero raw exceptions in ViewModels. Zero KAPT references. Zero hardcoded versions. Zero forbidden imports.
