Chapter 11

Building Custom Workflows & Integrations

Plugins, webhooks, and multi-service orchestration — make your agent do anything

The Integration Mindset

In the last chapter you locked down your agent's security. Now it's time to unlock its full potential.

Up to this point, you've been using OpenClaw's built-in integrations: Home Assistant, email, calendar, weather. Those cover the basics beautifully. But here's the thing — every service with an API is a potential agent tool.

Your favorite restaurant has an online reservation system? That's an API. Your building's package locker sends email notifications? That's a trigger. Your bank sends SMS alerts? That's data. Your electric company publishes a usage dashboard? That's an integration waiting to happen.

Think of your agent as a universal remote control. Out of the box it controls a few devices. But with plugins, it can control anything.

What You'll Build in This Chapter

By the end of this chapter, you'll understand how to connect your agent to any service, chain multiple actions into workflows, and trigger automations from external events. You'll go from "my agent does what it was built to do" to "my agent does whatever I need."

OpenClaw's Plugin System

OpenClaw extends its capabilities through plugins (also called tools). Each plugin teaches the agent a new skill. The built-in ones handle weather, reminders, and lists. Custom plugins handle everything else.

Anatomy of a Plugin

A plugin is a simple file that defines: what the tool does (so the LLM knows when to use it), what inputs it needs, and what code runs when it's called.

// plugins/restaurant-reservation.js
module.exports = {
  name: "reserve_restaurant",
  description: "Make a restaurant reservation. Use when the user wants to book a table.",
  parameters: {
    restaurant: { type: "string", description: "Restaurant name" },
    date: { type: "string", description: "Date in YYYY-MM-DD format" },
    time: { type: "string", description: "Time in HH:MM format" },
    party_size: { type: "number", description: "Number of guests" }
  },
  async execute({ restaurant, date, time, party_size }) {
    // Call the restaurant's booking API
    const response = await fetch("https://api.opentable.com/v2/reserve", {
      method: "POST",
      headers: { "Authorization": `Bearer ${process.env.OPENTABLE_KEY}` },
      body: JSON.stringify({ restaurant, date, time, covers: party_size })
    });
    const result = await response.json();
    return result.confirmation
      ? `Reservation confirmed: ${restaurant}, ${date} at ${time} for ${party_size}`
      : `Could not book: ${result.error}. Try a different time?`;
  }
};

That's it. The LLM reads the name and description to decide when to use it. The parameters tell it what information to collect from you. The execute function does the actual work.

You don't need to be a JavaScript expert to write plugins. The pattern is always the same: describe the tool, define what inputs it needs, call an API, return the result. If you can make a fetch call, you can write a plugin.

Registering Plugins

# config.yaml
plugins:
  directory: ./plugins
  enabled:
    - restaurant-reservation
    - package-tracker
    - electricity-usage
    - pet-vet-appointments

Drop a plugin file in the plugins/ directory, add it to the enabled list, restart OpenClaw. Your agent now has a new skill.

Building a Custom Integration Step by Step

Let's build a real plugin from scratch: a package tracker that checks delivery status when you ask "where's my package?"

Step 1: Define the Tool

// plugins/package-tracker.js
module.exports = {
  name: "track_package",
  description: "Track a package delivery status. Use when the user asks about a delivery or package.",
  parameters: {
    tracking_number: {
      type: "string",
      description: "The tracking number. If not provided, check the most recent package."
    }
  },
  async execute({ tracking_number }) {
    // Use a tracking API (e.g., AfterShip, 17track, or carrier-specific)
    const response = await fetch(
      `https://api.aftership.com/v4/trackings/${tracking_number}`,
      { headers: { "aftership-api-key": process.env.AFTERSHIP_KEY } }
    );
    const data = await response.json();
    const tracking = data.data.tracking;

    return {
      carrier: tracking.slug,
      status: tracking.tag,
      location: tracking.checkpoints[0]?.location || "Unknown",
      expected: tracking.expected_delivery,
      last_update: tracking.checkpoints[0]?.message || "No updates yet"
    };
  }
};

Step 2: Test It

Where's my package from Amazon?
I found your most recent tracking number (from the order confirmation email last Tuesday): 📦 Tracking: BG123456789 Carrier: Speedy Status: Out for delivery Location: Sofia distribution center Expected: Today by 6 PM Last update: "Package loaded on delivery vehicle" (8:47 AM) Looks like it'll arrive this afternoon! Want me to remind you to check the locker at 6 PM?

Notice the agent pulled the tracking number from a previous email (using the email integration from Chapter 7). That's the power of connecting integrations — they build on each other.

Webhook-Driven Workflows

Plugins let your agent pull information. Webhooks let external services push information to your agent. A webhook is just a URL that, when hit, triggers an action.

# config.yaml
webhooks:
  enabled: true
  port: 3701
  endpoints:
    - path: /package-delivered
      action: "A package was just delivered. Tell me what it is and log the expense."
    - path: /doorbell
      action: "Someone rang the doorbell. Check the camera and tell me who it is."
    - path: /smoke-alarm
      action: "URGENT: Smoke alarm triggered! Alert all household members immediately."

Now your smart doorbell can hit http://localhost:3701/doorbell when someone rings, and your agent sends you a Telegram message with the camera snapshot and a guess at who it is (delivery person? Neighbor? Unknown?).

Never expose webhook endpoints to the internet. Keep them on your local network only. If you need external services to trigger webhooks, use a secure tunnel (like Cloudflare Tunnel) with authentication, not a plain port forward.

Chaining Actions: The Real Magic

Individual plugins are useful. Chained plugins are transformative. Chaining means one event triggers a sequence of actions across multiple services.

The "Package Delivered" Chain

You get a package delivery notification. Here's what your agent does automatically:

Delivery Email
→
Update Shopping List
→
Log Expense
→
Notify You
  1. Email detected: Agent reads "Your order has been delivered" from Amazon
  2. Shopping list updated: The headphones you ordered are removed from your wishlist
  3. Expense logged: 459 BGN → Electronics category → budget updated
  4. You're notified: "Your Roborock Q7 Max was delivered! Expense logged (459 BGN, Electronics). This month's shopping budget: 659/800 BGN."

Four actions from one email, zero effort from you. And if the expense pushes you over the shopping budget, the agent flags it: "Heads up — you're over your monthly shopping budget by 59 BGN. Want to adjust next month's allocation?"

Defining Chains in Config

# Workflow chains
workflows:
  package_delivered:
    trigger: "email with subject containing 'delivered' or 'доставена'"
    steps:
      - action: "Identify what was delivered and the cost"
      - action: "Remove item from wishlist or shopping list if present"
      - action: "Log the expense to the correct category"
      - action: "Send me a summary notification"
      - condition: "If this pushes any budget over 90%, include a warning"

  grocery_receipt:
    trigger: "photo sent with caption containing 'receipt' or 'касова бележка'"
    steps:
      - action: "Scan receipt and extract items with prices"
      - action: "Log total to Groceries expense category"
      - action: "Check items against the shopping list, mark purchased"
      - action: "If any item is significantly more expensive than usual, flag it"

Scheduled Workflows

You've seen scheduled tasks in previous chapters (morning briefings, weekly health summaries). Let's look at the full power of scheduled workflows.

schedules:
  daily_briefing:
    cron: "0 7 * * 1-5"    # Weekday mornings at 7 AM
    action: |
      Morning briefing:
      1. Weather today and tomorrow
      2. Calendar events for today
      3. Unread urgent emails (from last night)
      4. Budget status (anything approaching limit?)
      5. Medication reminders due today
      6. One item from my reading list to motivate me

  weekly_review:
    cron: "0 19 * * 0"     # Sunday 7 PM
    action: |
      Weekly life review:
      1. Financial summary (spending by category, budget status)
      2. Health summary (weight trend, workouts, sleep, medication)
      3. Calendar preview for next week
      4. Open tasks and incomplete reminders
      5. Subscription renewal alerts for next week
      6. One positive observation about this week

  monthly_report:
    cron: "0 20 1 * *"     # First day of month at 8 PM
    action: |
      Monthly report:
      1. Total spending vs budget (with chart description)
      2. Top 5 expenses this month
      3. Subscription cost total and any unused services
      4. Health trends (weight, workout consistency)
      5. Books read, shows watched, goals progress
      6. Comparison to last month
Start with just the daily briefing. Add the weekly review after a week. Add the monthly report after a month. Don't configure everything at once — you'll get overwhelmed by notifications. Build up gradually as each one proves its value.

Conditional Logic

Workflows become truly smart when they include conditions — if/then logic that adapts behavior based on data.

workflows:
  smart_grocery_planning:
    trigger: "Sunday at 5 PM"
    steps:
      - action: "Check this week's grocery spending"
      - condition: "If over 150 BGN"
        then: "Suggest budget-friendly meal plan for next week using pantry staples"
        else: "Suggest normal meal plan for next week"
      - action: "Generate shopping list from the meal plan"
      - condition: "If any ingredient is on sale at Kaufland or Lidl"
        then: "Note the deal and suggest buying extra"

  weather_aware_commute:
    trigger: "Weekday mornings at 6:30 AM"
    steps:
      - action: "Check today's weather"
      - condition: "If rain probability > 50%"
        then: "Remind me to take an umbrella and suggest leaving 10 min early"
      - condition: "If temperature drops below 5°C"
        then: "Remind me to grab a heavy coat"
      - condition: "If there are traffic alerts on my route"
        then: "Suggest an alternative route or earlier departure"
[Sunday 5 PM — automatic trigger]
📊 Weekly grocery check: You spent 168 BGN this week (18 over your 150 BGN budget). Since we're over budget, here's a budget-friendly meal plan for next week using what's in your pantry: Mon: Pasta aglio e olio (pantry staples, ~3 BGN) Tue: Rice and bean bowls with whatever veggies are cheapest (Sam-friendly) Wed: Egg fried rice with frozen peas (pantry + freezer) Thu: Lentil soup — book club night, make a big pot (fast, cheap) Fri: Homemade flatbread pizzas (flour, tomato paste, whatever toppings) Estimated grocery cost: ~65 BGN (well under budget to compensate). Also: chicken breast is on sale at Lidl this week (7.99 BGN/kg instead of 12.99). Worth stocking up? Want me to finalize this plan and generate the list?

The agent checked spending, detected an overage, automatically switched to budget mode, prioritized pantry ingredients, and spotted a deal. That's conditional logic working across finance, meal planning, and shopping in one workflow.

Multi-Service Orchestration

The most powerful workflows span 3 or more services. Here are examples that connect everything you've built across this course:

The "Dinner Party" Workflow

Calendar Event
→
Guest Dietary Check
→
Meal Plan
→
Shopping List
→
Home Setup
workflows:
  dinner_party:
    trigger: "calendar event with 'dinner party' in title, 4 hours before"
    steps:

      - action: "Verify meal plan accommodates all guests"
      - action: "Check if all ingredients were purchased (cross-ref shopping list)"

workflows:

    trigger: "I say 'I'm sick' or 'not feeling well'"
    steps:
      - action: "Log symptom to health diary"
      - action: "Cancel or reschedule today's non-essential calendar events"
      - action: "Set thermostat to 23°C (comfort mode)"
      - action: "Suggest easy comfort food from what's in the fridge"
      - action: "Remind me to stay hydrated every 2 hours"
      - action: "If symptoms persist 48 hours, suggest calling the doctor"

Error Handling

Workflows fail. APIs go down. Credentials expire. Your agent needs to handle errors gracefully instead of silently breaking.

# Error handling rules in system prompt
When a workflow step fails:
1. DO NOT silently skip the step — tell me what failed and why
2. Continue with remaining steps if possible
3. Suggest a manual alternative for the failed step
4. If the failure is critical (security, health), escalate immediately
5. Log the error for debugging

Example: If the weather API is down during morning briefing,
say "Weather check failed (API timeout). I'll try again in
10 minutes. Here's the rest of your briefing..."
Add a "workflow health check" to your weekly review: "Were there any workflow failures this week? If so, list them with causes." This catches recurring issues before they become permanent blind spots.

Testing and Debugging Workflows

Before relying on a workflow, test it. OpenClaw has a dry-run mode that simulates the workflow without executing actions:

# Test a workflow without executing
openclaw workflow test dinner_party --dry-run

# Output:
# Step 1: [CHECK] Guest dietary restrictions → Would query contacts for "Mila, Stefan"
# Step 2: [CHECK] Meal plan → Would verify chicken stir-fry (Mila: no allergies, Stefan: vegetarian ⚠️)
# Step 3: [CHECK] Ingredients → Would cross-reference shopping list (3 items missing)
# Step 4: [SKIP] Home scene → Dry run, would set lights + music
# Step 5: [SKIP] Pre-heat oven → Dry run, would send HA command
# Result: ⚠️ Stefan is vegetarian — meal plan needs adjustment!

The dry run caught that your dinner plan doesn't work for a vegetarian guest. Better to discover this in testing than when Stefan is sitting at your table.

Workflows and Privacy Creep

As you build more integrations, your agent accumulates access to more services and more data. This is "privacy creep" — the gradual expansion of what your AI knows about you.

  • Audit your integrations quarterly. List every service your agent connects to. Do you still use all of them? Disconnect what you don't need
  • Review webhook endpoints. Each endpoint is an attack surface. Disable unused ones
  • Check plugin permissions. Does your restaurant plugin really need access to your contacts? Limit each plugin to the minimum data it needs
  • Workflow chains amplify risk. A chain that goes Email → Finance → Home Assistant means a compromised email integration could trigger home actions. Review chains with a security mindset from Chapter 10

Exercise: Build a 3-Service Workflow

Create a custom workflow that connects at least 3 services:

  1. Choose a trigger. Pick something that happens regularly: a delivery email, a calendar event, a time of day, or a phrase you say
  2. Design the chain. Write out 3-5 steps that should happen when the trigger fires. Each step should use a different service (email, calendar, finance, home, health, shopping, etc.)
  3. Implement it. Add the workflow to your config.yaml or create a plugin if needed
  4. Test with dry-run first to verify logic
  5. Run it for real and verify each step completes correctly
  6. Add error handling: What should happen if step 2 fails? Add fallback instructions

Suggested starter workflow: "When I get a delivery email → extract what was delivered → log the expense → update my shopping list → send me a summary." This touches email, finance, and shopping — three services you've already set up.
