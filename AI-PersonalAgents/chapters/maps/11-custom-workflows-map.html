Building Custom Workflows & Integrations

Memory Map

The Big Idea

Every service with an API is a potential agent tool. Your agent is a universal remote control — out of the box it handles a few things, but with plugins it can control anything. The real magic is chaining actions: one event triggers a cascade across multiple services with zero effort from you.

🔌 How a Plugin Works

📝 Define tool (name + description)
→
📋 Declare parameters
→
⚡ Execute (call API)
→
💬 Return result

The LLM reads name + description to decide when to use it. Parameters tell it what to collect. Execute does the work.

🧩 The Building Blocks

🧩
Plugins (Pull)
Your agent reaches out to a service on demand. "Where's my package?" → agent calls tracking API. One JS file per skill.
🔔
Webhooks (Push)
External services push events to your agent. Doorbell rings → hit localhost:3701/doorbell → agent sends you a camera snapshot.
⛓️
Chains
One trigger fires a sequence across services. Delivery email → update list → log expense → notify you. Four actions, zero effort.
⏰
Scheduled Workflows
Cron-triggered automations: daily briefings (7 AM), weekly reviews (Sunday 7 PM), monthly reports (1st of month). Build up gradually.
🔀
Conditional Logic
If/then branching: over grocery budget? → suggest budget meals. Rain forecast? → remind umbrella + leave early. Workflows adapt to data.
⚠️
Error Handling
Never silently skip failed steps. Report what broke, continue remaining steps, suggest manual alternatives, log for debugging.

📖 Key Terms

Plugin (Tool)
A file that teaches the agent a new skill: name, description, parameters, and an execute function that calls an API.
Webhook
A URL endpoint on your agent that external services hit to push events. Doorbell → /doorbell → agent notifies you.
Chain
A sequence of actions triggered by one event across multiple services. One email triggers 4+ automated steps.
Conditional Logic
If/then branching in workflows. Over budget → suggest cheap meals. Rain forecast → remind umbrella.
Dry Run
Testing a workflow without executing actions. Catches logic errors (like a vegetarian guest at a chicken dinner) before they happen.
Privacy Creep
The gradual, often unnoticed expansion of services and data your agent accesses. Audit quarterly and disconnect unused integrations.
Cron
Scheduled timing syntax: "0 7 * * 1-5" = 7 AM weekdays. Used for briefings, reviews, and reports.

🧪 Quick Quiz

What's the difference between a plugin and a webhook?

A plugin lets your agent pull information on demand (agent calls an API). A webhook lets external services push events to your agent (service hits a URL on your machine). Pull vs push.

Why should you never expose webhook endpoints to the internet?

Each webhook endpoint is an attack surface. Anyone who can hit it can trigger agent actions. Keep them on your local network only. If external access is needed, use a secure tunnel with authentication.

What's the "universal remote control" analogy?

Out of the box, your agent controls a few things (built-in integrations). With plugins, it can control anything that has an API — just like adding devices to a universal remote. Every service with an API is a potential agent tool.

Why use dry-run mode before deploying a workflow?

Dry run simulates the workflow without executing actions. It catches logic errors — like discovering Stefan is vegetarian and your dinner plan is chicken — before they cause real problems.

What is "privacy creep" and how do you prevent it?

The gradual expansion of services your agent connects to and data it accesses. Prevent it by auditing integrations quarterly, disconnecting unused ones, reviewing webhook endpoints, and limiting each plugin to minimum required data.

🎯 Key Takeaways

  • Every service with an API is a potential agent tool — your agent is a universal remote you can extend infinitely.
  • Plugins pull data on demand. Webhooks push events from external services. Both are simple to build.
  • Chains are the real magic: one trigger (email, calendar, phrase) fires a cascade across 3+ services automatically.
  • Conditional logic makes workflows smart: budget-aware meal plans, weather-reactive commutes, guest-aware dinner prep.
  • Build up gradually: daily briefing first, weekly review after a week, monthly report after a month.
  • Always dry-run before deploying. Test catches logic errors before they embarrass you at a dinner party.
  • Error handling is mandatory: never silently skip, report what failed, continue remaining steps, suggest alternatives.
  • Watch for privacy creep — audit integrations quarterly, disconnect unused services, limit plugin permissions.
