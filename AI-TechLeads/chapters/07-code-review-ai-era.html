Chapter 7

Code Review in the AI Era

New checklists, new smells, new strategies for reviewing AI-generated code

Why Traditional Code Review Breaks Down

In Chapter 6, you secured your AI tools. Now let's talk about the code they produce — because AI-generated code is breaking your review process in ways you probably haven't noticed yet.

Traditional code review was designed for human-written code: a developer thinks about a problem, writes a solution, and submits it for peer review. The reviewer reads the code, understands the reasoning, and checks for mistakes. This worked because humans write code at a pace humans can review.

AI changed the equation. A developer can now generate hundreds of lines in minutes. The code compiles, the tests pass, and it looks right. But code review pipelines weren't designed for this volume, and reviewers are burning out.

The core analogy: Reviewing AI code is like reviewing work from a very productive but inexperienced team member — the volume is great, but you need to verify the thinking, not just the output. The code may be syntactically perfect while being architecturally wrong.

The Review Problem in Numbers

  • 1.7x more issues in AI-generated code compared to human-written code (CodeRabbit 2026 report)
  • 35–40% increase in bug density for teams using AI without quality guardrails within 6 months
  • Review pipelines weren't built for AI-volume code — reviewer fatigue leads to more missed bugs, not fewer
  • Teams using AI-assisted code review report 40–60% less time on reviews while improving defect detection

The paradox: AI generates code faster, but reviewing it properly takes just as long (sometimes longer) because you can't assume the developer understands what was generated.

The "Looks Right" Trap

This is the most dangerous pattern in AI-era code review. AI-generated code has a unique property: it looks professionally written. It follows naming conventions, has reasonable structure, includes comments. It passes the casual scan test with flying colors.

But "looks right" isn't "is right." AI code has characteristic failure modes that are easy to miss:

👻
Plausible Hallucination. Code that references non-existent APIs, deprecated methods, or libraries that don't exist. It looks like a real API call — until runtime.
🔄
Pattern Mismatch. Code that follows a different pattern than your codebase. Works fine alone but creates inconsistency when integrated.
💣
Happy Path Only. AI generates the success case beautifully but skips error handling, edge cases, timeouts, and retry logic.
🛠️
Over-Engineering. AI adds abstractions, design patterns, and defensive code that aren't needed. Three layers of abstraction for a simple CRUD endpoint.
🕑️
Outdated Approaches. AI uses deprecated APIs, old library versions, or pre-2024 patterns because its training data includes historical code.
📋
Copy-Paste Proliferation. AI generates similar-but-slightly-different solutions for similar problems instead of reusing existing abstractions in your codebase.

Scenario: The PR That Passed Every Check

Tuesday morning. A mid-level developer submits a 400-line PR for a new notification service. Tests pass, linting is green, type-checking is clean. The reviewer scans it in 10 minutes — it looks well-structured, follows conventions, has good naming.

Approved and merged. Two days later, production alerts fire. The notification service is making database queries inside a loop, hitting the DB 200 times per batch instead of once. The AI had generated a working solution that was catastrophically inefficient at scale.

The code looked right. It worked in tests (with small data). But nobody asked: "Why is this querying inside a loop?" because the code was clean enough that it didn't trigger suspicion.

Discussion prompts: How would your current review process have caught this? What additional checks would have flagged the performance issue? Who should have been reviewing a new service — a peer or a senior engineer?

The New Code Review Checklist

Your old checklist still applies. But AI code needs additional scrutiny in specific areas. Add these 12 items to your review process:

The AI-Era Review Checklist (12 Items)

Architecture & Design

  • Does this follow our patterns? AI generates working code, but does it match your team's conventions, not just general best practices?
  • Is this the right abstraction level? AI loves adding layers. Does this need a factory pattern, or is a simple function sufficient?
  • Does it reuse existing code? Check if your codebase already has a utility, helper, or service that does what the AI reimplemented.

Correctness & Completeness

  • What's NOT there? Check for missing error handling, edge cases, null checks, timeouts, and retry logic.
  • Do the APIs exist? Verify that referenced libraries, methods, and endpoints are real and current.
  • Are the assumptions valid? AI makes assumptions about data shapes, concurrency, and ordering. Are they correct for your system?

Performance & Scale

  • N+1 queries? AI frequently generates database queries inside loops. Look for this pattern specifically.
  • Memory considerations? Does the code load entire datasets into memory when it should paginate or stream?
  • Concurrency safety? Is shared state accessed safely? AI often ignores thread safety.

Security & Dependencies

  • Security review applied? Run through the 10-item security checklist from Chapter 6.
  • Dependencies verified? Do all imported packages exist, and are they from trusted sources?
  • Secrets exposure? Any hardcoded values that should be environment variables?

Three Review Strategies That Work

Strategy 1: Focus on Architecture, Not Syntax

AI-generated code rarely has syntax errors. Spending your review time checking for typos or formatting is wasted effort. Instead, focus on the decisions:

Low-Value Review Focus

  • Variable naming consistency
  • Formatting and whitespace
  • Import ordering
  • Comment spelling
  • Bracket placement

Let linters and formatters handle these.

High-Value Review Focus

  • Why this approach over alternatives?
  • Does this fit our architecture?
  • What happens when this fails?
  • Will this scale with our data volume?
  • Does this reuse what we already have?

Only humans can assess these.

Strategy 2: Ask "Why This Approach?"

When a developer writes code themselves, they can explain their reasoning. When AI wrote it, the developer may not understand why that approach was chosen — they only know it works.

Tip: The "Explain It" Test

During code review, ask the PR author to explain the design decision behind a non-obvious section. If they say "that's what the AI generated" without being able to justify the approach, that's a red flag. The ownership principle from Chapter 5 applies: if you can't explain it, you shouldn't be merging it.

This isn't about catching people — it's about building a culture where developers engage critically with AI output. Over time, developers who practice this become dramatically better at both prompting and reviewing.

Strategy 3: Check What's NOT There

The most important review skill for AI code is looking for absences. AI is excellent at generating the happy path. It's terrible at anticipating what can go wrong.

Common things AI forgets:

  • Error handling for external API calls (timeouts, retries, circuit breakers)
  • Input validation at system boundaries
  • Logging for observability and debugging
  • Database transaction boundaries and rollback logic
  • Rate limiting and throttling
  • Graceful degradation when dependencies are unavailable
  • Cleanup and resource disposal (file handles, connections, streams)

Warning: The "Rubber Stamp" Problem

As AI-generated PRs become more frequent and consistently pass tests, some reviewers stop reading carefully. "The tests pass, the AI wrote it, it's probably fine." This is review fatigue, and it's the #1 code quality risk in AI-augmented teams.

Signs your team has this problem: review times dropping below 5 minutes for substantial PRs, "LGTM" with no comments becoming the norm, production bugs increasing despite CI passing. Address it by rotating reviewers, requiring at least one substantive comment per review, and running periodic "review quality audits."

Using AI to Review AI

Here's an idea that sounds circular but actually works: use AI-powered code review tools as the first pass, then have humans focus on what AI can't catch.

Step 1 Developer opens PR

Code ready for review.

→
Step 2 AI review (~90 sec)

Catches syntax, security, style, obvious bugs.

→
Step 3 Developer fixes

Addresses AI findings before human review.

→
Step 4 Human review

Architecture, intent, what AI can't assess.

This is the highest-leverage placement of AI in your review pipeline. The human reviewer gets a cleaner diff, freeing them to focus on design decisions, architecture fit, and business logic correctness.

Tip: AI Review Limitations

AI code review tools are excellent at catching: syntax issues, security vulnerabilities, style violations, and obvious bugs. They're poor at catching: intent mismatches (code does X correctly but X was the wrong thing to do), performance implications at scale, and cross-service dependencies. Use both AI and human review — they catch different things.

PR Etiquette in the AI Era

Your team needs updated PR conventions. Here's what works:

AI-Era PR Description Template

## What
[Brief description of the change]

## Why
[Business context — what problem does this solve?]

## AI Involvement
- Tool used: [Claude Code / Copilot / Cursor / None]
- AI role: [Initial scaffolding / Refactoring / Test generation / None]
- Human modifications: [What I changed, added, or removed from AI output]

## Review Focus
[What the reviewer should pay attention to — architecture decisions,
 performance concerns, security-sensitive areas]

## Testing
- [ ] Unit tests pass
- [ ] Integration tests pass
- [ ] Manually tested [specific scenarios]

The "AI Involvement" section isn't about policing — it's about giving reviewers context. If the reviewer knows AI scaffolded the implementation, they know to check for pattern consistency and missing edge cases. If the developer wrote it entirely, they can focus elsewhere.

The Review Size Problem

AI makes it easy to generate massive PRs. A developer who would have taken three days to write 200 lines can now generate 800 lines in an hour. This creates a review bottleneck.

The Anti-Pattern: Giant AI PRs

  • PRs over 400 lines get reviewed superficially (research consistently shows this)
  • Reviewers spend less time per line as PR size increases
  • AI-generated PRs are especially dangerous when large because the patterns look consistent even when they're wrong
  • "The AI wrote it in one session" is not a reason to submit it in one PR

The Solution: Decompose Before Generating

  • Break the task into logical commits before prompting the AI
  • One PR per logical change: data model, API endpoints, UI, tests
  • Set a team maximum PR size (200–400 lines is a good target)
  • If AI generates 800 lines, split it into 2–3 focused PRs
  • Each PR should be independently reviewable and deployable

Pair Review Sessions: Learning from AI Code

One of the most effective team practices is the pair review session: two developers reviewing an AI-generated PR together, discussing what the AI got right, what it got wrong, and why.

Leadership Insight: Reviews as Teaching Moments

AI-generated code creates a unique opportunity for team learning. When the whole team reviews an AI PR together (a "mob review"), everyone learns:

  • What AI code smells look like in your specific codebase
  • Which patterns the AI gets wrong for your architecture
  • How to ask better questions during review
  • What to add to your project context files to prevent recurring issues

Schedule a monthly 30-minute "AI Code Autopsy" — pick one AI-generated PR (preferably one that caused a bug) and review it together. These sessions build team-wide pattern recognition faster than any training program.

Tracking Review Quality

How do you know if your reviews are catching issues? Track these metrics:

Review Quality Metrics

  • Defect escape rate: Bugs found in production that should have been caught in review. Track separately for AI-generated vs human-written code.
  • Review depth: Average comments per PR. If it's trending toward zero, review fatigue is setting in.
  • Review time vs PR size: If large PRs get reviewed in the same time as small ones, reviewers are skimming.
  • Rework rate: How often does code get sent back for changes? Too low might mean rubber-stamping.
  • Time-to-first-review: How long PRs sit before anyone looks. AI increases PR volume, which can increase queue times.

Benchmark: What Good Looks Like

Based on 2026 industry data for teams using AI code generation:

  • Review cycle time: Under 4 hours from PR open to first substantive review
  • Defect escape rate: Under 5% (bugs that reach production vs caught in review)
  • Rework rate: 20–35% of PRs require changes (lower suggests rubber-stamping)
  • Review comments: Average 3–5 substantive comments per PR (not just "LGTM")

Exercise: Review an AI-Generated PR

Try It: Apply the New Checklist

Take a recent PR from your team that involved AI assistance. Re-review it using the 12-item checklist above. Compare what you find now vs what was caught in the original review.

Step 1: Select the PR

Pick a merged PR from the last 2 weeks where AI was used (ask the author if you're not sure). Ideally 100–300 lines.

Step 2: Re-Review with the Checklist

Go through all 12 items. For each, note:

  • Was this checked in the original review? (yes/no)
  • Did you find any new issues? (describe)
  • Would this have been caught by AI review tools? (yes/no)

Step 3: Assess the Gap

  • How many new issues did you find?
  • Which checklist areas were completely missed in the original review?
  • What single change to your review process would have the highest impact?

Step 4: Share with the Team

Present your findings in your next retro or team meeting. This is not about blame — it's about improving the process. The goal: make the checklist part of your team's standard review practice.

Pro tip: Do this exercise as a pair review session with another engineer. Two perspectives catch more than one, and the discussion itself builds review skills.
