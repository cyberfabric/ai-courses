Chapter 5

Setting AI Policies & Guidelines

Memory Map
💡

The Big Idea

AI policies are like coding standards — too strict and nobody follows them, too loose and you get chaos. Write a 2-page, developer-friendly document that emphasizes what IS allowed, not just what isn't.

🎮 The Policy Spectrum

🚫 AI Banned
Max control, zero gains, shadow AI
→
✅ AI Governed
Approved tools + guardrails
→
🌊 AI Free-for-All
Max speed, zero oversight

Your target: the middle. Productivity benefits with managed risk.

📜 The Five Policy Areas

📤
1. What Goes IN
Data classification: what's safe to send to AI tools, what needs approved tools, what's never allowed (secrets, PII, credentials).
📝
2. Before Merging
Developer owns every line. Same review standards as human code. Security-sensitive areas need senior review. No rubber-stamping.
🔧
3. Approved Tools
Company SSO accounts only. 2–3 vetted tools. Personal accounts banned for work code — even for the same tool.
©️
4. Ownership & IP
Copyright requires human authorship. Meaningful creative input = copyrightable. Employment agreements should cover AI-assisted work.
📄
5. Attribution
Practical middle ground: disclose the tool in PR descriptions, not line-by-line. "Assisted by Claude Code" — simple, auditable.

🔒 Data Classification for AI Context

Level What Rule
✅ Always Safe Open-source code, public docs, general questions Any AI tool
⚠️ Approved Tools Only Proprietary code, internal APIs, business logic Company SSO + zero-data-retention
🚫 Never API keys, PII, credentials, connection strings, compliance data No AI tool, ever

The "Newspaper Test": Would you be comfortable if this prompt appeared in a news article about your company?

©️ The IP & Licensing Landscape

📜
Copyright Law
US copyright requires human authorship. Pure AI output may not be copyrightable. Iterative human input = copyrightable.
🛡️
IP Indemnity
Only GitHub Copilot Business+ currently offers IP indemnity. Covers legal costs if AI code causes IP disputes.
🇧🇪
EU AI Act
High-risk AI enforcement begins August 2, 2026. Vendors must publish training data summaries and copyright compliance policies.
💥
Copyleft Risk
AI might generate code matching GPL sources. Use license scanning tools (FOSSA, Snyk, Black Duck) in CI/CD to catch contamination.

📄 Attribution: Two Schools of Thought

Aspect 📋 Mandatory Disclosure 👤 Ownership-Based ✅ Practical Middle
Requirement Tag AI-generated sections No distinction needed Note tool in PR description
Audit trail Full None Sufficient
Overhead High (hard to enforce) None Minimal
Reviewer signal Strong None Moderate

✍️ Good vs. Bad Policies

Trait ✅ Good Policy ❌ Bad Policy
Length 2 pages max 50+ pages
Language Specific: "Use company SSO only" Vague: "Use appropriate measures"
Focus Green-light: what you CAN do All restrictions, no permissions
Examples Concrete, real scenarios Abstract principles
Updates Quarterly review cadence Written once, never revisited

📋 The Policy Template (7 Sections)

  1. Approved Tools

    Company-managed accounts only. List specific tools, tiers, and SSO requirements.

  2. Data Rules

    Three tiers: always safe, approved tools only, never send. The "Newspaper Test."

  3. Code Ownership

    You own the code you commit. Same quality bar regardless of origin.

  4. Review Requirements

    Standard review for all AI code. Senior review for security-sensitive areas.

  5. Attribution

    Note tool in PR description. No line-by-line attribution required.

  6. Prohibited Uses

    Auth, crypto, payments, migrations, prod infra — expert review required.

  7. Reporting

    If AI exposes secrets, PII, or copyrighted code — report immediately.

❓ Common Edge Cases

🏠
Personal Projects at Work?
Yes, on personal time with personal accounts. Never mix personal and work AI accounts.
💬
AI for Commit Messages?
Yes. Low risk, high convenience. Just review what it says before pushing.
📋
Error Logs in AI?
Strip PII and secrets first. Logs often contain customer data and internal paths.
🚫
Client Says "No AI"?
Respect it. Tag the repo as AI-restricted. Disable tools for that project. Document the exception.

📖 Key Terms

Data Classification
Categorizing information by sensitivity level to determine what can be sent to AI tools.
IP Indemnity
Vendor guarantee to cover legal costs if AI-generated code causes intellectual property disputes.
Copyleft Contamination
Risk that AI-generated code matches GPL-licensed source, triggering obligations to share your proprietary code.
Shadow AI
Developers using personal, unapproved AI accounts for work code — creating security and compliance gaps.
Ownership Principle
The developer who commits the code is fully accountable, regardless of whether a human or AI generated it.
Newspaper Test
Before pasting data into AI: "Would I be comfortable if this appeared in a news article about our company?"
Zero-Data-Retention
Enterprise AI plan where the vendor does not store or train on your prompts and code.

🤔 Quick Quiz

What are the five policy areas every AI usage policy should cover?

(1) What goes into the AI (data classification), (2) merge requirements (review standards), (3) approved tools (SSO, enterprise accounts), (4) code ownership & IP, and (5) attribution & transparency.

Why are personal AI accounts banned for work code, even if it's the same tool?

Personal accounts have different data retention terms, no audit trail, no SSO, and no enterprise security features. It's the account type that matters, not the tool name.

What's the "practical middle ground" for AI attribution?

Disclose the tool, not the lines. PR descriptions mention "Assisted by Claude Code" but don't require line-by-line tagging. Gives an audit trail without creating busy work.

Who currently offers IP indemnity for AI-generated code?

Only GitHub Copilot Business and Enterprise. They cover legal costs if AI-generated code causes IP disputes. No other major AI coding tool offers this yet.

How long should an AI usage policy be?

Two pages maximum. A one-page quick reference (the TL;DR developers use daily) backed by a longer reference document for edge cases. If it's longer than 2 pages, nobody reads it.

📌 Key Takeaways

  • "Just be careful" isn't a policy. Write a 2-page document with clear rules, examples, and green lights.
  • Five areas to cover: data classification, merge requirements, approved tools, IP/ownership, and attribution.
  • The Ownership Principle: the developer who commits the code owns every line, AI-generated or not.
  • Company SSO accounts only. Personal AI accounts are never approved for work code.
  • Security-sensitive code (auth, crypto, payments) needs expert review even when AI-assisted.
  • The legal landscape is unsettled — only Copilot Business+ offers IP indemnity. Review your policy quarterly.
  • Get buy-in in 4 weeks: you draft → security reviews → legal reviews → team feedback → ship.
