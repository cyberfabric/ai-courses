Code Review in the AI Era

The Big Idea

Reviewing AI code is like reviewing work from a very productive but inexperienced team member — verify the thinking, not just the output. The code may be syntactically perfect while being architecturally wrong.

👻 The "Looks Right" Trap

👻
Plausible Hallucination
References non-existent APIs, deprecated methods, or libraries that don't exist. Looks like a real call — until runtime.
🔄
Pattern Mismatch
Follows general best practices but not YOUR team's conventions. Works alone, creates inconsistency when integrated.
💣
Happy Path Only
Beautiful success case, but skips error handling, edge cases, timeouts, and retry logic.
🛠️
Over-Engineering
Factory patterns, abstract layers, and defensive code that aren't needed. Three abstractions for a simple CRUD.
🕑️
Outdated Approaches
Uses deprecated APIs, old library versions, or pre-2024 patterns from historical training data.
📋
Copy-Paste Proliferation
Generates similar-but-different solutions instead of reusing existing abstractions in your codebase.

✅ The AI-Era Review Checklist (12 Items)

  1. Architecture & Design (3 items)

    Follows OUR patterns? Right abstraction level? Reuses existing code?

  2. Correctness & Completeness (3 items)

    What's NOT there? Do the APIs exist? Are assumptions valid?

  3. Performance & Scale (3 items)

    N+1 queries? Memory considerations? Concurrency safety?

  4. Security & Dependencies (3 items)

    Security checklist applied? Dependencies verified? Secrets exposure?

🎯 Three Strategies That Work

Strategy Focus Key Question
🎨 Architecture First Design decisions, not syntax "Does this fit our architecture?"
❓ Ask "Why?" Developer must explain reasoning "Why this approach over alternatives?"
👀 Check Absences What AI forgot to generate "What happens when this fails?"

👀 Common Things AI Forgets

💥
Error Handling
Timeouts, retries, circuit breakers for external API calls.
🛡️
Input Validation
Validation at system boundaries. AI trusts all inputs by default.
📋
Logging
Observability and debugging signals. AI writes code that works but is opaque.
🔄
Transaction Boundaries
Database transactions, rollback logic, and atomicity guarantees.

📖 Key Terms

Plausible Hallucination
AI code that references non-existent APIs or methods but looks like a real, valid call.
Review Fatigue
When reviewers stop reading carefully because AI PRs consistently pass tests. The #1 quality risk.
The "Explain It" Test
Ask the PR author to explain a non-obvious section. "The AI generated it" without justification is a red flag.
Defect Escape Rate
Percentage of bugs that reach production vs those caught in code review.
AI Code Autopsy
Monthly team session reviewing an AI-generated PR (ideally one that caused a bug) to build pattern recognition.
N+1 Query
Database queries inside a loop. AI's most common performance anti-pattern — works in tests, catastrophic at scale.
PR Decomposition
Breaking AI output into multiple focused PRs (200–400 lines each) instead of one giant submission.

🤔 Quick Quiz

What are the six AI code "failure modes" that pass casual review?

Plausible hallucination (fake APIs), pattern mismatch (not your conventions), happy path only (missing error handling), over-engineering (unnecessary abstractions), outdated approaches (deprecated APIs), copy-paste proliferation (duplicate instead of reuse).

What's the highest-leverage way to use AI in the review pipeline?

AI review before human review. AI runs in ~90 seconds, catches syntax, security, and style issues. Developer fixes those first. Human reviewer then sees a cleaner diff and focuses on architecture, intent, and business logic.

What is the "Explain It" test and when should you use it?

Ask the PR author to explain the reasoning behind a non-obvious design decision. If they say "that's what the AI generated" without being able to justify the approach, it's a red flag. The developer must understand what they're merging.

What are the signs of review fatigue in your team?

Reviews under 5 minutes for substantial PRs, "LGTM" with no comments becoming the norm, production bugs increasing despite CI passing. Fix: rotate reviewers, require substantive comments, run "review quality audits."

What maximum PR size should you set for AI-generated code?

200–400 lines per PR. PRs over 400 lines get reviewed superficially. If AI generates 800 lines, split into 2–3 focused PRs. Break the task into logical commits BEFORE prompting the AI.

📌 Key Takeaways

  • AI code "looks right" but has 6 characteristic failure modes. Verify the thinking, not just the output.
  • Use the 12-item AI-era checklist: architecture, correctness, performance, security — 3 items each.
  • Focus on high-value review: design decisions and what's missing. Let linters handle formatting.
  • AI review before human review is the highest-leverage pipeline placement (90-second first pass).
  • The "Explain It" test: if the developer can't justify the approach, they shouldn't merge it.
  • Review fatigue is the #1 quality risk. Watch for: fast approvals, no comments, rising production bugs.
  • Monthly "AI Code Autopsy" sessions build team-wide pattern recognition faster than any training.
