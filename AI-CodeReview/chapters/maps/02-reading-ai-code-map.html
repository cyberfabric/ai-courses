Chapter 2

Reading AI-Generated Code

Memory Map

💡 The Big Idea

🔍

AI Code Has Fingerprints

AI-generated code has recognizable "tells" — patterns, tics, and habits that signal where to focus your review. Learn the tells, and you'll instinctively know which 20% of the code hides the bugs.

🎯 The Seven AI Code Tells

🏗️
#1 Over-Abstraction
Factory for a builder for a strategy. Three files where one function would do. AI solves problems you don't have yet.
👻
#2 Hallucinated APIs
Methods that look real but don't exist. 1 in 5 AI samples reference non-existent libraries. Always check the docs.
🍳
#3 Kitchen Sink Imports
Imports more than it needs — entire modules for one function, types never referenced, deprecated packages alongside modern ones.
🛡️
#4 Overly Defensive
Try-catch everything, check for impossible nulls, add fallbacks that mask real errors instead of surfacing them.
📋
#5 Copy-Paste Duplication
Each AI prompt is a fresh conversation. Same validation logic inlined in three handlers instead of extracted once.
💀
#6 Dead Code
Unused functions, unreachable branches, 12 utility functions when only 3 are imported. AI gives you the whole toolkit.
💬
#7 Over-Commenting
Every line has a comment that restates the code. None explain why. Human devs comment context, AI comments syntax.

⚡ Speed-Reading a 500-Line Diff

  1. 📊 Stat View

    Run git diff --stat — see which files changed and how much. Shape before detail.

  2. 👁️ Read for Shape

    Scroll the whole diff quickly. Notice structure: new files, renames, deletes, boilerplate vs. targeted changes.

  3. 🔥 Identify Hot Spots

    Focus on high-risk files: auth, database queries, payments, config, public APIs. Do tests exist?

  4. 🔬 Deep-Read Hot Spots

    Line-by-line only on files that matter. Trace three scenarios: happy path, null/empty, boundary case.

Reviewer effectiveness collapses past 200–400 lines or 60–90 minutes (SmartBear/Cisco). AI diffs routinely exceed both.

🎭 Happy Path Trap

Aspect AI Does Well AI Gets Wrong
Happy path ✅ Almost always correct ❌ Rarely tests beyond it
Edge cases ✅ Handles some (often imaginary) ❌ Misses the real ones in your system
Error handling ✅ Syntactically correct try-catch ❌ Swallows errors, returns null silently
API usage ✅ Follows naming conventions ❌ May use APIs that don't exist
Code structure ✅ Clean, readable formatting ❌ Over-abstracts simple problems

Key insight: Resist tracing the happy path first. Go straight to: what happens when input is empty? Network times out? Two requests arrive simultaneously?

✅ When AI Code Is Actually Good

🔄
Follows Existing Patterns
Matches your codebase conventions: same naming, structure, error handling.
😴
It's Boring
Standard CRUD, simple data transforms, utility functions. Tedious to write, hard to get wrong.
🧪
Behavior-Focused Tests
Tests verify what the code does, not how. No implementation-detail coupling.
📦
Clean Imports
Only what's needed, no kitchen-sink imports, no deprecated packages.

The 80/20 rule: ~80% of AI code is boring and correct. Spend 80% of your time on the dangerous 20%.

🚨 Watch Out For

🚩 Red Flag: Untraced Changes
Code that doesn't map to any requirement, ticket, or conversation. AI agents frequently exhibit scope creep — you asked for an endpoint, and got a refactored DB layer too.
🤖 AI Tell: Phantom Edge Cases
A function handling 5 edge cases when only 2 are possible in your system. Ask: "Which checks protect against real scenarios?"
🧪 Quick Hallucination Check
Unfamiliar library API? Spend 30 seconds: grep -r "methodName" node_modules/lib/ or check the docs. This one habit catches the most dangerous AI bugs.
📝 The "Teach Me" Test
"Is there anything here I didn't know?" If yes — it's either worth learning, or something the AI invented. Either way, slow down.
📄 Drafts, Not Production
Treat large AI outputs as drafts. Run it, verify the approach, then reimplement in smaller pieces. The second draft is always cleaner.

❓ Quick Quiz

Test Your Understanding

Name 3 of the 7 AI code tells.

Over-abstraction, hallucinated APIs, kitchen-sink imports, overly defensive error handling, copy-paste duplication, dead code, over-commenting.

What's the "happy path trap"?

AI handles the happy path beautifully but crumbles on edge cases. Your instinct is to trace the happy path first — resist it. Go straight to empty inputs, timeouts, and race conditions.

What are the 4 steps for speed-reading a large AI diff?

1) Stat view (git diff --stat), 2) Read for shape (structure, not detail), 3) Identify hot spots (auth, data, config), 4) Deep-read only the hot spots line-by-line.

How do you spot a hallucinated API?

When AI uses a library method you don't recognize, spend 30 seconds checking: grep the node_modules or check official docs. The API may look plausible but not exist in any version.

When is AI code actually good and safe to quick-review?

When it follows existing patterns, is boring (standard CRUD/transforms), has behavior-focused tests, clean imports, and appropriate scope. Give it a sanity check and move on.

✅ Key Takeaways

If You Remember Nothing Else

  • AI code has 7 recognizable tells — learn them and you can't unsee them.
  • The most dangerous tell: hallucinated APIs that look real but don't exist. Always check the docs.
  • AI nails the happy path but crumbles on edge cases. Review the failure paths first.
  • Speed-read large diffs: stat view → shape → hot spots → deep read only what matters.
  • 80% of AI code is boring and correct. Spend your energy on the dangerous 20%.
  • Flag changes that don't trace back to a requirement — AI scope creep is real.
  • Treat large AI output as a draft, not production code. Reimplement in smaller, understood pieces.
