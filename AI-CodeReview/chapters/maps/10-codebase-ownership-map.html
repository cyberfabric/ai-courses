A City That Builds Faster Than Its Maps

42% of all code is AI-generated. Your codebase grows faster than your team's understanding of it. The bottleneck isn't writing code anymore — it's knowing what the code does, why it exists, and who can explain it. This invisible gap is called cognitive debt.

🏗️ Onboarding Flow

  1. 📐 Read ADRs & System Diagram

    Build the big picture before reading any code.

  2. 📄 Read CODEOWNERS

    Know who to ask about each area.

  3. 🔍 Trace One Request End-to-End

    Follow one user action from API to database and back.

  4. 🤖 Use Repo Intelligence Tools

    Ask Greptile/Sourcegraph "how does X work?" for code-grounded answers.

  5. 📝 Write a One-Page Doc

    If you can write it, you understand it. Share for corrections.

🔑 Key Terms

Cognitive Debt
Team-level erosion of shared understanding across a software system. Invisible until an incident exposes it.
Architecture Decision Record (ADR)
A document capturing one decision, its context, alternatives considered, and consequences. Answers "why" when the author is gone.
Bus Factor
Minimum team members who'd need to leave before a project is in trouble. AI amplifies this — context lives in prompt sessions, not code.
Architectural Drift
Inconsistent patterns across files because different AI sessions made different choices. Code works but has no coherence.
CODEOWNERS
File that auto-assigns reviewers based on which paths a PR touches. Ensures domain experts review AI-generated changes.
Documentation-as-Review
Writing docs for AI code as a comprehension exercise. Produces understanding + documentation in one step.
Repo Intelligence
Tools that index the full codebase for semantic understanding, catching cross-file issues that diff-only tools miss.

❓ Quick Quiz

Test Your Understanding

What is cognitive debt and how is it different from technical debt?

Cognitive debt is the erosion of shared understanding across a system. Unlike technical debt (measurable with linters/metrics), cognitive debt is invisible — tests pass, CI is green, but the team can't reason about the system. It surfaces during incidents.

Why does AI amplify the bus factor problem?

When the developer who prompted the AI leaves, they take context the code doesn't capture: the prompts used, alternatives rejected, edge cases discussed, and the "why" behind non-obvious decisions. The code and tests remain, but the reasoning is gone.

What's the difference between full-codebase-aware and diff-only review tools?

Full-codebase-aware tools index the entire repository and catch cross-file dependencies, broken contracts, and architectural violations (40-60% more issues). Diff-only tools only see changed lines and miss integration problems.

What's the "documentation-as-review" practice?

After reviewing significant AI-generated code, write a brief doc explaining what it does, why it exists, and how it connects to the system. If you can't write the doc, you don't understand the code well enough to approve it. Double benefit: comprehension + documentation.

How do you prevent architectural drift in an AI-heavy codebase?

Encode team standards in project rules (CLAUDE.md, .cursorrules), use linters as guardrails, review for patterns not just correctness, and refactor proactively when you spot inconsistency before it spreads.

📌 Key Takeaways

If You Remember Nothing Else

  • 42% of code is AI-generated. Your codebase grows faster than your team's understanding of it.
  • Cognitive debt is invisible — tests pass, CI is green, but nobody can reason about the system during incidents.
  • Use full-codebase-aware tools (Greptile, CodeRabbit) — diff-only analysis misses 40-60% of cross-file issues.
  • Write ADRs for every non-obvious decision. Code says "what." ADRs say "why."
  • Every module needs bus factor ≥ 2. Rotate ownership quarterly.
  • Encode conventions in project rules (CLAUDE.md), not tribal knowledge. Prevent architectural drift at the source.
  • Write documentation as a comprehension exercise — if you can't doc it, don't approve it.
  • Track metrics monthly: complexity, coupling, coverage, incident resolution time, bus factor.
