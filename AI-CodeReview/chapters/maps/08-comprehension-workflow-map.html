The Comprehension Workflow

Memory Map

💡 The Big Idea

🎓

If You Can't Teach It, You Don't Understand It

Developers spend 58% of their time on comprehension, not writing code (Xin Xia et al., 2018 IEEE TSE). With AI generating more code faster, the temptation is to cut comprehension time. The result: comprehension debt (Addy Osmani, March 2026) — code nobody truly owns. The fix is a systematic 5-step workflow applied to every non-trivial change.

📋 Step-by-Step Breakdown

  1. 📖 Read the Diff

    Start with git diff --stat for shape. Check: Does scope match intent? Unexpected files? New files duplicating existing code?

  2. 🎯 Identify the Intent

    Read the ticket/prompt before the code. Form a mental picture of the expected diff. Where reality diverges from expectation — that's where bugs live.

  3. 🔍 Trace the Data Flow

    Don't just read the function — trace how data enters, transforms, and exits. Where does input come from? What validation happens? Where does output go? What state is left on failure?

  4. ❓ Question the Assumptions

    AI makes more assumptions than humans because it doesn't know your system. Check: environment, input, concurrency, and state assumptions.

  5. 🧪 Test Your Understanding

    Without running the code, predict what happens with unexpected input. Write predictions down, then check. Where you're wrong = where bugs will emerge.

🧩 Core Concepts

📊
Comprehension Debt
The gap between code that exists and code humans actually understand. Invisible — tests pass, CI is green — but the team's mental model erodes.
👨‍🏫
Teach-It-Back
The Protégé Effect: preparing to teach produces deeper understanding than studying. Explain the code out loud in your own words — not "it adds retry logic" but how and why.
🍳
The Recipe Analogy
You can skim a soufflé recipe and think you get it. But "fold gently" and "stir in" produce completely different results. Skimming code is not comprehension.
🦆
Rubber Duck Approach
Explain the code to an inanimate object, a voice memo, or a written note. The act of articulating forces you to confront the parts you don't understand. GitHub Copilot CLI productized this April 2026 with a second-model "Rubber Duck" critic that closed 74.7% of the Sonnet→Opus gap.
📓
Comprehension Journal
After each review: what you learned, what surprised you, action taken. Over time, becomes a knowledge map of your codebase for rapid context rebuilding.
⏱️
Time-Boxing
Match comprehension investment to blast radius. A config bug affects one deploy. A payment bug affects every customer. If time runs out and you still don't understand — don't approve.

⚖️ Active Reading vs. Passive Scanning

Aspect 🔎 Active Reading 👀 Passive Scanning
Analogy Reading a contract Reading a news article
Approach Question every clause, look for what's missing Eyes move across lines, get general sense
Retention High — catches subtle bugs Low — retains almost nothing
Techniques Annotate, predict, question defaults, read tests first Skim and approve
Result Genuine understanding False confidence

⏳ Time-Boxing Framework

⚡
2 Minutes
Trivial: config, formatting, dependency bumps. Skim, check for surprises, approve.
🔧
10 Minutes
Standard: new endpoints, UI components, utilities. Run the full 5-step workflow.
🔒
30 Minutes
Complex/risky: auth, payments, migrations. Full workflow + line-by-line + teach-it-back.
🏗️
60+ Minutes
Architecture-level: new services, schema redesigns. Pair review recommended.

🔑 Key Terms

Comprehension Debt
The gap between code volume and human understanding of it. Unlike tech debt, it's invisible until an incident.
Protégé Effect
Preparing to teach material produces deeper understanding than just studying it. The basis of the teach-it-back technique.
Intent Mismatch
When the stated goal (ticket/prompt) diverges from what the code actually does. Where reality ≠ expectation, bugs live.
Blast Radius
The scope of damage if a bug exists. Determines how much time to invest in comprehension.
Active Reading
Reading code like a contract — questioning every clause, annotating, predicting behavior before reading the body.
Rubber Duck
Explaining code out loud to force yourself to confront gaps in understanding. No audience required.

A per-review log of what you learned, what surprised you, and actions taken. Becomes a knowledge map over time.

❓ Quick Quiz

Test Your Understanding

What are the 5 steps of the comprehension workflow?

Read (what changed?) → Intent (why?) → Trace (data flow) → Question (assumptions) → Test (predict edge cases).

What is comprehension debt?

The widening gap between the volume of code in a codebase and the volume any human engineer genuinely understands. Unlike tech debt, it's invisible — tests pass, CI is green, but the team's mental model erodes.

What's the difference between active reading and passive scanning?

Active reading = reading like a contract (question every clause, annotate, predict). Passive scanning = reading like a news article (general sense, low retention). Only active reading catches subtle bugs.

How do you know you actually understand a piece of code?

If you can teach it to someone else — explaining how it works, what triggers it, and why this approach was chosen — you understand it. If you can only say "it adds retry logic," you don't.

What should you do when the time-box runs out and you still don't understand the code?

Don't approve it. The code is too complex or poorly structured. Request the author to refactor into smaller, reviewable pieces. A PR you can't understand in a reasonable time shouldn't be merged as-is.

📌 Key Takeaways

If You Remember Nothing Else

  • "Mostly understand" is where bugs hide — apply the 5-step workflow to every non-trivial AI change.
  • Read → Intent → Trace → Question → Test — each step builds on the last.
  • If you can't teach it back out loud, you don't understand it.
  • Ask AI about decisions ("why this approach?"), not descriptions ("what does this do?").
  • Match comprehension time to blast radius: 2 min for trivial, 60+ min for architecture.
  • Keep a comprehension journal — it becomes your codebase knowledge map.
  • If the time-box expires and you still don't understand, don't approve.
