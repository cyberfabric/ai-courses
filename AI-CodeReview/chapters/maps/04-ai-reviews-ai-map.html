Using AI to Review AI

Memory Map

💡 The Big Idea

📝

AI Review = Spellcheck, Not Editor

AI review catches mechanical issues (90%+ accuracy on null pointers, off-by-ones, resource leaks). It cannot judge intent, architecture, or whether the code should exist at all. First pass, not final pass.

🌐 Full-Codebase vs. Diff-Only

🌐
Full-Codebase-Aware
Indexes your entire repo. Traces how a change in one file affects callers across 15 others. Catches 40-60% more cross-file issues.
📄
Diff-Only
Reviews only what changed. Fast and cheap, but misses broken callers, stale references, and cross-service dependencies.
💬
Ad-Hoc LLM Check
Paste git diff --cached into a different model with a focused prompt. Free, 60 seconds, catches real bugs.
📐
Rule-Based (SonarQube)
Not AI — deterministic rules. Fewer false positives. Best as a complement to AI review, not a replacement.

📖 Key Terms

Multi-Model Review
Using a different LLM to review than the one that wrote the code. Different training data = different blind spots. 7-15% accuracy improvement.
Iterative Consensus Ensemble (ICE)
Multiple LLMs critique each other until consensus. No fine-tuning needed. 7-15 point accuracy boost over best single model.
Full-Codebase-Aware
Tools that index your entire repo before reviewing. Catch 40-60% more cross-file issues than diff-only tools.
Signal-to-Noise Ratio
An unconfigured AI reviewer generates so much noise that devs stop reading. Worse than no review — false sense of security.
Dialog Pattern
AI #1 writes → AI #2 reviews → Human triages → Iterate. The human's job is judgment, not mechanical checking.

🚨 Watch Out For

🚩
Red Flag
Treating AI review as the final word. "The AI reviewed it" is not "the code is correct." AI catches 0% of intent mismatches.
🤖
AI Tell
AI reviewer leaves 4 comments about naming/style and misses the logic error on line 47. Optimizes for easy-to-detect, not important.
📢
Noise Problem
Unconfigured tools generate so many low-value comments that devs ignore all findings. Spend 2-4 hours on initial config.

❓ Quick Quiz

Test Your Understanding

Why use a different model to review than the one that wrote the code?

Different models have different blind spots. The generation model is less likely to catch its own errors because they share similar training biases. Cross-model review improves accuracy by 7-15 percentage points.

What's the 40-60% gap?

Full-codebase-aware tools catch 40-60% more cross-file issues than diff-only tools. They trace how a change in one file affects callers in other files.

What does AI review miss that humans must still check?

Business logic correctness, architectural fit, intent mismatches, institutional context, and whether the code should exist at all. These are judgment calls, not pattern-matching.

What's the "dialog pattern"?

AI #1 writes the code → AI #2 reviews it → Human triages findings and decides what to fix → Iterate until ready. The human's role is judgment and orchestration.

Why is an unconfigured AI reviewer worse than no reviewer?

It generates so many low-value comments that developers stop reading any findings. This creates a false sense of security: "the AI reviewed it" without any actual benefit.

✅ Key Takeaways

If You Remember Nothing Else

  • AI review is spellcheck, not editor. First pass, not final pass.
  • Use a different model to review than the one that wrote the code. Different blind spots = better coverage.
  • Full-codebase-aware tools catch 40-60% more cross-file issues than diff-only tools.
  • The ad-hoc LLM check (paste diff into a different model) is free, takes 60 seconds, and catches real bugs.
  • AI catches mechanical issues (90%+ accuracy). It catches 0% of intent mismatches.
  • Spend 2-4 hours configuring your AI review tool. Unconfigured = noise = developers ignore all findings.
  • Never reduce human review because "the AI handles it." AI review is advisory, not authoritative.
