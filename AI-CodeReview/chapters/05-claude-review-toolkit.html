Claude Code’s Review Toolkit

The multi-agent review fleet, REVIEW.md, and how to make automated review actually useful

From Overview to Deep Dive

In Chapter 4, you surveyed the AI review tool landscape — CodeRabbit, Greptile, Qodo, and the rest. Now we go deep on one specific tool: Claude Code’s built-in Code Review.

Why a whole chapter? Because if you’re using Claude Code daily (and you’re reading this book, so you probably are), the review system is already at your fingertips. No extra subscription. No third-party integration. Just a comment on your PR.

Think of this chapter like a mechanic’s guide to one specific diagnostic tool. Chapter 4 told you what tools exist in the shop. This chapter teaches you how to use the one already bolted to your workbench — its modes, its dials, its blind spots, and the situations where it shines.

How Code Review Works: The Fleet

When you trigger a review, Claude doesn’t run a single model pass over your diff. It dispatches a fleet of specialized agents that work in parallel. Each agent focuses on a different class of issue. Think of it as a code review panel, not a single reviewer.

The specialized agents cover distinct domains:

  • Correctness Agent — Verifies logic, checks for off-by-one errors, validates edge case handling, ensures the code does what the PR description claims.
  • Security Agent — Scans for injection vulnerabilities, authentication bypasses, insecure data handling, and dependency risks.
  • Performance Agent — Identifies N+1 queries, unnecessary re-renders, memory leaks, and algorithmic inefficiencies.
  • Style & Consistency Agent — Checks adherence to project conventions, naming patterns, and documentation standards.

The critical step is verification. After agents flag potential issues, a verification step checks each candidate against actual code behavior to filter out false positives. This is why Claude Code Review reports less than 1% of findings as incorrect in Anthropic’s internal testing — the verification layer weeds out the noise before you see it.

The results are deduplicated, ranked by severity, and posted as inline comments on the specific lines where issues were found. If no issues are found, Claude posts a short confirmation comment. Reviews complete in about 20 minutes on average.

Triggering Reviews: Automatic vs. Manual

Code Review offers three trigger modes, configured per repository by an admin:

Automatic Triggers
Once after PR creation — review runs when a PR opens or is marked ready. One review, one cost.

After every push — review runs on each push. Catches new issues as the PR evolves. Auto-resolves threads when you fix flagged issues. Higher cost.
Best for teams that want consistent coverage without manual effort.
Manual Trigger
Manual mode — reviews start only when someone comments @claude review or @claude review once on the PR. No automatic reviews.
Best for high-traffic repos where you want to opt specific PRs into review, or when you only want review on ready PRs.

The two comment commands work differently:

CommandWhat it does
@claude review Starts a review and subscribes the PR to push-triggered reviews going forward
@claude review once Starts a single review without subscribing to future pushes

When to Use “review once”

Use @claude review once for long-running PRs with frequent pushes, or when you want a one-off second opinion without triggering a review on every subsequent commit. It’s also useful for draft PRs — manual triggers work on drafts, unlike automatic triggers.

Reading the Review Output

Claude posts findings in three places. Know where to look:

1. Inline Comments on the Diff

The primary output. Each finding appears as a review comment on the exact line where the issue was found, tagged with a severity marker:

MarkerSeverityMeaning
🔴ImportantA bug that should be fixed before merging
🟡NitMinor issue, worth fixing but not blocking
🟣Pre-existingBug that exists in the codebase but wasn’t introduced by this PR

Each comment includes a collapsible extended reasoning section you can expand to see why Claude flagged the issue and how it verified the problem. This is where the real value is — don’t just read the summary, expand the reasoning.

2. Check Run Summary

The Claude Code Review check run appears alongside your CI checks. Click Details to see every finding in one table, sorted by severity:

Severity     | File:Line                 | Issue
-------------|---------------------------|--------------------------------------
🔴 Important | src/auth/session.ts:142   | Token refresh races with logout,
             |                           | leaving stale sessions active
🟡 Nit       | src/auth/session.ts:88    | parseExpiry silently returns 0
             |                           | on malformed input

The check run always completes with a neutral conclusion — it never blocks merging through branch protection rules. If you want to gate merges on findings, you can parse the machine-readable severity counts:

gh api repos/OWNER/REPO/check-runs/CHECK_RUN_ID \
  --jq '.output.text | split("bughunter-severity: ")[1]
  | split(" -->")[0] | fromjson'
# Returns: {"normal": 2, "nit": 1, "pre_existing": 0}

3. Files Changed Annotations

Findings also render as annotations in the Files changed tab, attached directly to diff lines. These persist even if GitHub rejects an inline comment on a line that moved.

Rate the Findings

Every review comment arrives with 👍 and 👎 buttons already attached. Click them. Anthropic collects reactions after the PR merges and uses them to tune the reviewer. This is how the system gets better over time — and it only works if you provide feedback.

REVIEW.md: Teaching Claude What to Care About

Out of the box, Code Review focuses on correctness: bugs that would break production. It does not flag formatting preferences or missing test coverage unless you tell it to. This is a good default — it avoids the noise problem from Chapter 4.

To customize what Claude checks, add a REVIEW.md file to your repository root. No configuration needed — Claude auto-discovers it.

# REVIEW.md — Example for a Node.js API project

## Always check
- New API endpoints have corresponding integration tests
- Database migrations are backward-compatible
- Error messages don't leak internal details to users
- Authorization checks on every route that handles user data

## Style
- Prefer early returns over nested conditionals
- Use structured logging (winston), not console.log or f-strings
- Async functions must have explicit error handling

## Skip
- Generated files under src/gen/
- Formatting-only changes in *.lock files
- Test fixture data files

The distinction between REVIEW.md and CLAUDE.md matters:

  • REVIEW.md — Review-only guidance. Read exclusively during code reviews. Use it for what to flag or skip during review.
  • CLAUDE.md — General project instructions that Claude uses for all tasks, including reviews. Use it for architectural context: “packages/auth owns all role and permission checks; apps/api is the only service allowed to mutate billing state.”

Think of it this way: CLAUDE.md tells the agents how your system is shaped. REVIEW.md tells them what to care about during review.

Don’t Skip REVIEW.md

Without a REVIEW.md, Claude applies generic correctness checks. With one, it applies your team’s correctness checks. The difference is significant. A generic reviewer doesn’t know that your project requires integration tests for every new endpoint, or that console.log is banned in production code. A configured reviewer does. Spend 15 minutes writing a REVIEW.md — it pays for itself on the first review.

GitHub Actions: Running Review in Your Own CI

The managed Code Review service runs on Anthropic’s infrastructure. If you need more control — custom triggers, integration with other CI steps, or self-hosted environments — use claude-code-action, the official GitHub Action.

# .github/workflows/claude-review.yml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          review_comment: true

The action runs the full Claude Code runtime in your GitHub Actions runner, giving Claude access to your repository, file structure, git history, and diff context. It posts findings directly to the PR, just like the managed service.

Key patterns you can build with the action:

  • Path-specific reviews — trigger deeper analysis only when security-critical files change (auth, payments, config)
  • Issue-to-PR workflows — mention @claude in an issue to have Claude implement and open a PR
  • Automated test generation — when new code lands without tests, Claude writes them and commits to the same branch
  • Custom review checklists — enforce project-specific rules beyond what REVIEW.md covers

The quickest setup path: run /install-github-app in your Claude Code terminal, which walks you through GitHub App installation and secret configuration.

Scaling Reviews: Lightweight vs. Deep

Not every PR deserves a 20-minute, $15–25 multi-agent review. The key is risk-based allocation:

Deep Review (Automatic)
• PRs touching auth, payments, or database migrations
• PRs over 200 lines of changed code
• PRs from AI agents or new contributors
• PRs modifying public API surfaces
Set these repos/paths to “After every push” trigger mode.
Skip or Manual Only
• Dependency bumps and lock file updates
• Documentation-only changes
• Auto-generated code (protobuf, OpenAPI stubs)
• Formatting-only PRs from linter auto-fixes
Set to “Manual” mode. Use @claude review once if you want a spot check.

What Code Review Doesn’t Catch

Let’s be honest about the limits. Claude Code Review is the best single-tool reviewer available in 2026. It’s still not sufficient by itself.

  • Intent mismatches — Claude can’t tell you the feature you built isn’t the one the customer needs. It checks correctness, not requirements.
  • Institutional context — “We tried this caching approach in Q3 2025 and it caused a cascading failure.” No amount of REVIEW.md captures tribal knowledge.
  • Architectural decisions — Should this be a microservice or a module? Should we use Redis or a database queue? These are design questions, not code quality questions.
  • Cross-service dependencies — While Claude sees the full codebase of the repo, it can’t trace calls across service boundaries in a distributed system.
  • Business logic validation — “Gold tier customers should get 15% off, not 10%.” Unless that rule is documented in CLAUDE.md, Claude has no way to know.

This is why the combined workflow exists. Claude catches the mechanical bugs so your human review catches the judgment-call bugs.

Claude Catches What Humans Miss: The Cross-File Bug

Here’s a real pattern where Claude Code Review shines — and a human reviewer probably wouldn’t catch it.

// PR changes src/api/orders.ts
export async function createOrder(userId: string, items: Item[]) {
  const order = await db.orders.create({ userId, items, status: 'pending' });
  await notifyUser(userId, order.id);  // <-- Claude flags this
  return order;
}

// Claude's finding (🔴 Important):
// notifyUser() was renamed to sendOrderNotification() in PR #847
// three weeks ago. This file still references the old name.
// The function won't be found at runtime — this will throw
// a ReferenceError when any order is created.

A human reviewer looking at this diff sees a clean function. The rename happened in a different PR, weeks ago. But because Claude examines the diff in the context of the full codebase, it traces the function call and discovers the reference is broken. This is the kind of cross-file bug that causes 2 AM pages — and that humans routinely miss.

Disabling Review Rules Because They’re “Too Noisy”

When Claude Code Review starts flagging the same type of issue repeatedly, the temptation is to add it to the “Skip” section of your REVIEW.md. Sometimes that’s right — if the findings are genuinely irrelevant (generated code, lock files).

But often, repeated findings mean you have a systemic problem. If Claude keeps flagging missing authorization checks, the fix isn’t to tell Claude to stop looking. The fix is to add an auth middleware that applies by default. If it keeps flagging N+1 queries, the fix is to add a query analyzer to your ORM setup.

Investigate before you silence. A noisy finding is a signal, not a nuisance. Disable it only after you’ve confirmed the pattern is genuinely not a problem in your system.

Setting Up Claude Code Review for Your Project

  • Enable Code Review in Claude admin settings (requires Team or Enterprise plan)
  • Select repositories to include — start with 1–2, not all of them
  • Choose trigger mode: “Once after PR creation” for most repos, “After every push” for security-critical repos
  • Create a REVIEW.md with your team’s “Always check,” “Style,” and “Skip” rules
  • Ensure your CLAUDE.md describes the system architecture (which packages own what)
  • Open a test PR and verify the Claude Code Review check run appears
  • Read the first few reviews carefully — expand the reasoning sections, rate with 👍/👎
  • Establish team workflow: Claude reviews first, humans review second using the 10-point checklist
  • Monitor costs in the analytics dashboard — adjust trigger modes if costs are too high
  • Review your REVIEW.md monthly — add new patterns you keep catching manually, remove rules that are no longer relevant

Run Your First Claude Code Review

If you have a Claude Team or Enterprise plan, do this now. It takes 5 minutes to set up.

  1. Go to claude.ai/admin-settings/claude-code and enable Code Review
  2. Select one repository — pick one with a recent or open PR
  3. Set the trigger to “Manual” for now (you control when it runs)
  4. Go to an open PR and comment: @claude review once
  5. Wait ~20 minutes for the review to complete
  6. Read every finding:
    • Expand the extended reasoning — do you agree with the verification logic?
    • Rate each comment 👍 or 👎
    • How many findings are real bugs vs. noise?
  7. Compare: would you have caught these issues in your own manual review?

If you don’t have Team/Enterprise: Use the code-review plugin locally. Run /code-review in your Claude Code terminal on a local branch to get a similar analysis without the managed service.

Write Your REVIEW.md

Create a REVIEW.md for your main project repository. This is the single highest-value configuration step.

  1. Create REVIEW.md in your repository root
  2. Add an “Always check” section with your team’s non-negotiable rules:
    • What must every new endpoint have? (tests? auth? rate limiting?)
    • What patterns are banned? (console.log in production? direct DB queries outside the data layer?)
    • What files require extra scrutiny? (auth, payments, config)
  3. Add a “Style” section with conventions your linter doesn’t enforce:
    • Prefer early returns? Structured logging? Specific error handling patterns?
  4. Add a “Skip” section for files Claude should ignore:
    • Generated code, lock files, fixture data, third-party vendored code
  5. Commit the file and open a PR to test it — does the next review reflect your rules?
