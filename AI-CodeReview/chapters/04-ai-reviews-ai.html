Using AI to Review AI

How to use AI review tools as your first-pass filter — spellcheck, not editor

The Spellcheck Analogy

In Chapter 3 you built the checklist — ten points, applied every time. That works. It also takes time. Now let’s talk about making it faster.

Think of AI review tools like spellcheck in a word processor. Spellcheck catches typos, grammar slips, and repeated words. It does not check whether your argument makes sense, whether the essay answers the prompt, or whether you plagiarized the second paragraph. You’d never submit a paper based solely on “spellcheck found no errors.” But you’d also never skip spellcheck — because manually hunting for typos is a waste of your editorial brain.

AI code review works the same way. It catches the mechanical stuff — null pointer dereferences, missing error handling, unused imports, off-by-one errors — so your human review time goes to the things that actually matter: architecture, business logic, security, and whether the code should exist at all.

The Multi-Model Review Strategy

Here’s something most developers don’t think about: the model that wrote the code has blind spots, and a different model has different blind spots. When the generation model and review model share similar training data and biases, the review model is less likely to catch errors that the generation model introduced.

The fix is simple: use a different model to review than the one that wrote the code.

  • Copilot wrote it? Review with Claude.
  • Claude wrote it? Review with GPT or Gemini.
  • Cursor wrote it? Run it through CodeRabbit or Qodo.

Research on the Iterative Consensus Ensemble (ICE) approach — where multiple LLMs critique each other until they reach consensus — shows 7–15 percentage points improvement in accuracy over the best single model, with no fine-tuning required. You don’t need a fancy framework to get this benefit. Just ask a second model to review.

The Quick Cross-Model Check

Before you push an AI-generated change, run git diff --cached and paste it into a different model than the one that wrote it. Ask: “Review this diff. Focus on logic errors, security issues, and missing edge cases. Be specific.” This 60-second habit catches bugs that the original model would never flag on its own code.

Full-Codebase-Aware vs. Diff-Only: Why It Matters

Imagine you rename a utility function in utils/format.ts. A diff-only tool sees the rename and says “looks good.” A full-codebase-aware tool traces every caller of that function across 15 files and tells you that 3 of them still reference the old name.

That’s the difference. Diff-only tools review what changed. Codebase-aware tools review what changed in the context of everything it touches.

The 40–60% Gap

Full-codebase-aware tools catch 40–60% more cross-file issues than diff-only tools. For monorepos and large codebases, this isn’t a nice-to-have — it’s the difference between useful and useless AI review. If your project has significant cross-file dependencies, prioritize a codebase-aware tool.

The review prompt matters. Here’s what works:

Vague Prompt
“Review this code.”
Too broad. You’ll get generic style suggestions and miss the real issues.
Focused Prompt
“Review this diff. This is an AI-generated change to our payment service. Focus on: (1) logic errors in the discount calculation, (2) missing auth/validation, (3) edge cases with empty carts or zero-amount orders. Be specific — cite line numbers.”
Gives context, narrows focus to high-risk areas, and asks for specificity. This gets useful feedback.

This costs nothing, takes 60 seconds, and catches real bugs. It’s especially useful when you’re working late, your team is in a different time zone, and you want a sanity check before pushing.

What AI Review Tools Actually Catch (and What They Miss)

Let’s be honest about the limits. AI review tools are excellent at pattern matching and terrible at judgment. Here’s the split:

What AI Review Catches Well
• Null pointer dereferences and missing null checks
• Off-by-one errors in loops
• Unused imports and dead code
• Missing error handling (uncaught exceptions, silent failures)
• Common security anti-patterns (hardcoded secrets, SQL injection)
• Style violations and formatting inconsistencies
• Resource leaks (unclosed connections, file handles)
• Duplicate code blocks
Mechanical issues. Pattern-matchable. High confidence.
What AI Review Misses
• Business logic correctness (“does this calculate the discount right?”)
• Architectural coherence (“does this fit our system design?”)
• Intent mismatches (“this works, but is it what we asked for?”)
• Cross-service dependencies (unless full-codebase-aware)
• Performance implications at scale (it sees the code, not the traffic)
• Institutional context (“we tried this approach in 2024 and it failed”)
• Whether the code should exist at all
Judgment calls. Context-dependent. Requires human experience.

This is why AI review is first-pass, not final-pass. It handles checklist points 3–6 from Chapter 3 (edge cases, security patterns, performance anti-patterns, duplication) with decent accuracy. Points 1, 2, 7, and 10 (intent, logic, integration, explain-it-back) still require a human brain.

AI Reviewer Flags Style, Misses Logic

Here’s a pattern you’ll see repeatedly: the AI reviewer leaves four comments about variable naming and bracket style, and completely misses the logic error on line 47.

// AI reviewer flagged this:
-  const usr = await getUser(id);    // "Consider renaming to 'user'"
+  const user = await getUser(id);

// AI reviewer missed this:
   function applyDiscount(order) {
     if (order.coupon && order.total > 50) {
       return order.total * 0.9;    // 10% off — correct
     }
     if (order.loyaltyTier === 'gold') {
       return order.total * 0.9;    // Also 10% off — but gold gets 15%!
     }
     return order.total;
   }

The AI caught a naming issue (low value) and missed a business logic error (high value). The gold tier discount should be 15% (* 0.85), not 10%. AI review tools optimize for what’s easy to detect, not what’s important. Always read the findings with one eye on what’s NOT flagged.

The human’s job isn’t to fix every issue — it’s to triage. The AI reviewer will flag 10 things. Maybe 5 are real issues, 3 are style preferences, and 2 are false positives. Your judgment determines which findings matter. That’s the part AI can’t do.

Configuring Your AI Review Stack

Out-of-the-box AI review tools generate noise. The teams that get real value spend 2–4 hours on initial configuration: defining custom rules, setting severity thresholds, and telling the tool what to prioritize.

Key configuration decisions:

  • Disable style-only comments unless your team has no linter. AI review time is better spent on logic and security than bracket placement.
  • Set severity thresholds. Block merges on critical/high findings. Warn on medium. Ignore low/info by default.
  • Enable security scanning. This is the highest-value AI review category — hardcoded secrets, injection vectors, insecure defaults.
  • Tune for your stack. If you’re a Python shop, enable Python-specific rules and disable the generic ones. Most tools support .coderabbit.yaml, qodo.yaml, or similar config files.
  • Map risky files. Tell the tool which files are security-critical (auth, payments, config) so it applies deeper analysis there.

The Noise Problem Is Real

An unconfigured AI reviewer generates so many low-value comments that developers stop reading them. This is worse than no AI review at all — because now you have a false sense of security (“the AI reviewed it”) without any actual benefit. Invest the setup time. Signal-to-noise ratio is everything.

Trusting AI Review as the Final Word

The most dangerous pattern in 2026: teams that deploy an AI review tool and then reduce human review because “the AI catches everything.”

It doesn’t. AI review tools catch 70–80% of mechanical issues. They catch approximately 0% of intent mismatches. They cannot tell you that the feature you built isn’t the feature the customer needs. They cannot tell you that a similar approach failed last quarter for reasons not captured in the code.

AI review is advisory, not authoritative. A clean AI review report means “no pattern-matchable issues found.” It does not mean “this code is correct.” The day your team treats an AI “LGTM” as a real LGTM is the day your review process stopped working.

Setting Up Your AI Review Stack

  • Choose a primary AI review tool (CodeRabbit, Qodo, or Copilot Code Review for starters)
  • Spend 2–4 hours on initial configuration — disable style-only comments, set severity thresholds
  • Enable security scanning as the highest priority category
  • Map your security-critical files (auth, payments, config) for deeper analysis
  • Add a complementary tool: SonarQube for deterministic quality gates, or a different LLM for cross-model review
  • Establish the team rule: AI review is first-pass, not final-pass — human review is still required
  • Build the ad-hoc habit: git diff --cached | pbcopy + paste into a different model before pushing
  • Review AI review tool findings weekly for the first month — calibrate signal vs. noise
  • Track false positive rate — if it’s above 20%, reconfigure or switch tools
  • Never reduce human review scope because “the AI handles it”

The Two-Tool Comparison

This exercise takes 15 minutes and shows you exactly what AI review can and can’t do.

  1. Pick your most recent AI-generated PR (or the last significant commit)
  2. Run it through two different review methods:
    • Method A: Use a formal tool (CodeRabbit free tier, Copilot code review, or gh copilot review)
    • Method B: Paste the diff into a chat model (Claude, GPT, Gemini) with the focused review prompt from this chapter
  3. Compare the findings:
    • What did Tool A catch that Tool B missed?
    • What did Tool B catch that Tool A missed?
    • How many false positives did each produce?
    • Which findings would have actually prevented a bug?
  4. Write down which approach you’ll use for which situations going forward.

Build Your AI Review Config

Set up an AI review tool on one of your active repositories.

  1. Pick a tool: CodeRabbit (free tier), Copilot code review (if you have a subscription), or Qodo PR-Agent (open source)
  2. Install it on one repository — not all of them. Start small.
  3. Configure it:
    • Disable purely stylistic comments (you have a linter for that)
    • Set severity thresholds: block on critical, warn on medium, ignore info
    • Enable security scanning
    • Add any project-specific review instructions (e.g., “this project uses PostgreSQL, not MySQL”)
  4. Open your next 3 PRs with the tool active
  5. After 3 PRs, assess: How many findings were useful? How many were noise? Adjust the config accordingly.
