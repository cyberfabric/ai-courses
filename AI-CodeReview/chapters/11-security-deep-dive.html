Chapter 11

Security Review for AI Code

The vulnerabilities AI introduces most — and how to catch them before production

The 2026 Security Landscape

Chapter 10 addressed codebase ownership. Now we focus on the area where AI-generated code poses the highest risk: security. Every other chapter in this course applies to all code. This chapter could save you from a breach.

Think of AI-generated code as a beautifully assembled house with unlocked doors. The structure looks great. The walls are straight, the paint is even, the floor plan makes sense. But the locks on the doors? Missing, or using cheap hardware the builder found convenient. AI produces code that looks secure — proper function names, reasonable structure, even comments about security — but the actual security logic has gaps that an attacker will find before your team does.

25–45% of AI-generated code contains confirmed vulnerabilities (across six major 2026 studies)
1.88× more likely to introduce vulnerabilities than human-written code (CodeRabbit analysis)
86% of AI-generated samples failed to defend against XSS (Veracode, 2026)
33% of all AI code vulnerabilities are injection flaws (AppSec Santa study)

These numbers are from 2026 — not hypothetical warnings, but measured results from production code. AppSec Santa’s February 2026 study tested 534 AI-generated code samples (89 prompts in Python and JavaScript) across six frontier LLMs — GPT-5.2, Claude Opus 4.6, Gemini 2.5 Pro, DeepSeek V3, Llama 4 Maverick, and Grok 4 — then ran the output through five SAST tools and confirmed findings against OWASP Top 10. GPT-5.2 had the lowest vulnerability rate at 19.1% (17 / 89). Claude Opus 4.6, DeepSeek V3, and Llama 4 Maverick tied for the worst at 29.2%. Across all models, 25.1% of samples shipped with confirmed vulnerabilities. DryRun Security’s March 2026 Agentic Coding Security Report (Claude / Codex / Gemini, 38 scans) found a complementary failure mode: AI agents produce authentication middleware correctly but frequently fail to wire it into subsequent components — 87% of agent PRs had at least one vulnerability, and WebSocket auth was missing in every final game codebase tested. The vulnerability lives in the integration, not any individual file.

The Security Paradox of AI Code

AI-generated code is syntactically clean but logically flawed. Traditional code scanners were built to catch sloppy code — missing brackets, obvious injection points. AI code passes those checks and fails on the subtler issues: missing trust boundaries, incomplete authorization checks, and insecure defaults that look intentional.

OWASP Top 10 Meets AI-Generated Code

AI doesn’t introduce new vulnerability categories — it amplifies the existing ones. Here are the OWASP Top 10 vulnerabilities most commonly introduced by AI, ranked by frequency in 2026 studies:

1. Injection Flaws (33% of AI vulnerabilities)

SQL injection, command injection, code injection. AI generates syntactically valid queries without reasoning about trust boundaries. It knows the shape of a query but not the threat model behind parameterization.

// AI-generated — spot the vulnerability
app.get('/users', (req, res) => {
  const query = `SELECT * FROM users
    WHERE name = '${req.query.name}'`;
  db.query(query, (err, results) => {
    res.json(results);
  });
});

The AI built a working endpoint. It also built a textbook SQL injection vector. A parameterized query would fix it — and any experienced developer would use one. But AI generates what it’s seen most often in training data, and unfortunately, training data includes millions of insecure examples.

2. Server-Side Request Forgery (SSRF)

The single highest-count vulnerability in the AppSec Santa study (32 instances). AI generates code that takes user-controlled URLs and fetches them without allowlists or IP blocking. The model doesn’t reason about internal network topology.

3. Broken Authentication & Authorization

This is the most dangerous category for AI code, because the vulnerability often isn’t in the auth function itself — it’s in the integration. AI writes a correct auth middleware, then generates a new endpoint without attaching it. The middleware exists. The endpoint is unprotected. Tests pass because they test the middleware in isolation.

4. Cross-Site Scripting (XSS)

Veracode’s 2026 testing found 86% of AI-generated samples failed to defend against XSS. AI renders user input into HTML without escaping, trusts frontend sanitization, or uses dangerouslySetInnerHTML in React without understanding why the name includes “dangerously.”

// AI-generated React component — XSS vulnerability
function UserProfile({ user }) {
  return (
    <div>
      <h2>{user.name}</h2>
      <div dangerouslySetInnerHTML={{ __html: user.bio }} />
    </div>
  );
}

If user.bio comes from user input without server-side sanitization, this is a stored XSS attack vector. AI uses dangerouslySetInnerHTML because it produces the “richest” output — it doesn’t consider that untrusted data rendered as HTML is one of the oldest attacks in web security.

5. Hardcoded Secrets

API keys, database credentials, JWT secrets embedded directly in source code. AI focuses on “code that works” over secure deployment patterns. It doesn’t know about your secrets manager or environment variable conventions. Aikido Security’s 2026 data shows AI-generated code now causes 1 in 5 enterprise security breaches, with hardcoded secrets being one of the leading entry points.

Common Security Patterns AI Gets Wrong

Beyond the OWASP categories, AI has specific patterns where it consistently fails. Learn to recognize these in review:

Authentication: The Integration Gap

// AI-generated auth middleware — correct in isolation
const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'No token' });
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);
    next();
  } catch (e) {
    return res.status(403).json({ error: 'Invalid token' });
  }
};

// Later in the same session, AI generates a new endpoint...
app.get('/admin/users', async (req, res) => {
  // No authMiddleware applied! The endpoint is unprotected.
  const users = await User.find({ role: 'admin' });
  res.json(users);
});

The auth middleware is correct. The admin endpoint is completely unprotected. If you review them separately (as AI often generates them), both look fine. The vulnerability is in the gap between them.

Input Validation: Partial Sanitization

AI sanitizes some inputs but not others. It validates the email field but trusts the display name. It checks the content type but doesn’t validate the request body size. Partial sanitization is worse than no sanitization — it creates a false sense of security.

Secret Management: The “Works Locally” Trap

// AI-generated — works in development, leaks in production
const payments = require('payments-sdk')('<your-test-secret-key-here>');

// Or slightly more subtle:
const API_KEY = process.env.API_KEY || 'default-api-key-12345';

The first is obvious. The second is insidious — if the environment variable isn’t set in production, the fallback exposes a default key. AI generates these patterns because they make the code “work” immediately during development.

The “Deprecated Crypto” Pattern

AI training data includes years of code using MD5, SHA-1, and ECB-mode encryption. AI will generate these without hesitation because they’re the most common patterns in its training set. In review, check every cryptographic function: is it using current algorithms (SHA-256+, AES-GCM, bcrypt/argon2 for passwords)? If you see MD5 or SHA-1 in new code, reject it immediately.

Static Analysis Tools: Your Security Safety Net

A critical 2026 finding from AppSec Santa’s study: across five SAST tools tested, the methodology turned 1,173 deduplicated raw findings into 175 confirmed positives — and the majority of those confirmed AI-code vulnerabilities were detected by only one of the five tools. No single tool catches everything. The recommendation is to run at least three tools for adequate coverage.

🔒

Uses semantic data flow analysis (from DeepCode acquisition) to catch complex vulnerabilities like second-order SQL injection across multiple files. Gartner Leader for Application Security Testing 2025. Best for: developer-first workflow, supply chain security.

Best for: enterprise teams with established DevOps, quality gate enforcement in CI/CD.

The Three-Tool Minimum

Run CodeQL + Semgrep + one language-specific scanner (e.g., Snyk for JavaScript/Python, SpotBugs for Java, Brakeman for Ruby). Each tool has different detection strengths. AppSec Santa’s 2026 study showed that most confirmed AI-code vulnerabilities were caught by only one of the five SAST tools tested — meaning any single tool will miss the majority of issues the others catch. Three complementary tools get you meaningfully closer to full coverage. These tools are complementary, not competing.

The “Security Sandwich”

The most effective security workflow for AI-generated code is a three-layer process:

1. Human Threat Model
Identify what could go wrong
before AI writes code
→
2. AI Implementation
Generate the code with
security constraints in the prompt
→
3. Human Security Review
Verify against threat model +
run SAST tools

The human bookends are non-negotiable. AI doesn’t understand your threat model — it doesn’t know which endpoints face the internet, which users are untrusted, or which data is sensitive. You provide that context before generation and verify it after.

Lightweight Threat Modeling for AI Changes

You don’t need a two-day workshop. For each security-relevant AI-generated change, answer four questions:

  1. What data does this code touch? (User input? Credentials? PII? Financial data?)
  2. Who are the threat actors? (Unauthenticated users? Authenticated users with limited access? Internal services?)
  3. What’s the blast radius? (One user? All users? The entire database? Financial loss?)
  4. What are the trust boundaries? (Where does trusted data end and untrusted data begin?)

Write the answers down before the AI generates the code. Then, after reviewing the AI’s output, check: does the implementation respect every trust boundary you identified? If not, that’s where the vulnerability is.

This lightweight threat model takes five minutes. It catches the class of bugs that no SAST tool can find — the architectural security gaps where individual components are correct but the system as a whole is vulnerable.

Security Review Automation in CI/CD

Manual security review doesn’t scale when AI generates code at volume. You need automated gates that catch the obvious issues before a human ever sees the PR.

# .github/workflows/security-gate.yml
name: Security Gate
on: [pull_request]
jobs:
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Semgrep scan
        uses: returntocorp/semgrep-action@v1
        with:
          config: p/owasp-top-ten p/secrets
      - name: Snyk Code
        uses: snyk/actions/node@master
        with:
          command: code test
      - name: Check for hardcoded secrets
        uses: gitleaks/gitleaks-action@v2

This workflow runs on every PR. Semgrep checks OWASP Top 10 patterns and secrets. Snyk scans for deeper vulnerabilities. Gitleaks catches hardcoded credentials. If any tool finds a critical issue, the PR fails — no manual review needed for the obvious stuff.

The key insight: automate the checklist, reserve human judgment for the hard problems. SAST tools catch the SQL injection AI introduced. Humans verify that the auth middleware is actually wired up to the new endpoint. Both are necessary; neither is sufficient alone.

The “High-Speed Intern” Standard

Addy Osmani’s framing: if code touches auth, payments, secrets, or untrusted input, treat AI as a high-speed intern and require:

  • A human threat model review
  • A SAST tool pass with zero critical findings
  • Line-by-line review of all security-relevant logic
  • Explicit verification that auth/authz middleware is applied to every endpoint

This isn’t about distrusting AI. It’s about recognizing that security is the one area where “mostly correct” equals “completely vulnerable.” A function that handles payments correctly 99% of the time and has a race condition the other 1% isn’t 99% secure — it’s compromised.

Dynamic Testing Matters Too

Static analysis catches patterns in code. Dynamic testing catches vulnerabilities in running code. After AI generates a web endpoint, run a quick DAST scan (OWASP ZAP is free) against the running application. AI-generated APIs frequently have vulnerabilities that only manifest at runtime — improper CORS headers, missing rate limiting, or response bodies that leak internal data structures. A 5-minute ZAP scan catches what SAST misses.

AI Using Deprecated Cryptographic Functions or Insecure Defaults

When you see these in AI-generated code, reject immediately:

// Red flags in AI-generated crypto
const hash = crypto.createHash('md5').update(password).digest('hex');
const cipher = crypto.createCipheriv('aes-128-ecb', key, null);
const token = jwt.sign(payload, 'secret123', { algorithm: 'HS256' });

MD5 for password hashing (broken since 2004). ECB mode for encryption (leaks patterns). A hardcoded JWT secret with a weak algorithm. AI generates these because they’re the most common patterns in its training data — millions of tutorials and Stack Overflow answers used these before better alternatives became standard.

What to use instead: bcrypt or argon2 for passwords, AES-256-GCM for encryption, RS256 with key rotation for JWTs, and secrets from environment variables or a secrets manager — never hardcoded.

AI-Generated Code Handling Auth or Payment Without Human Security Review

Scenario: A developer uses Claude Code to build a payment integration. The AI generates a Stripe checkout flow with proper API calls, webhook handling, and receipt generation. Tests pass. The developer opens a PR. Nobody on the team does a security-specific review because “it uses the Stripe SDK correctly.”

Three months later, the team discovers: the webhook endpoint doesn’t verify Stripe signatures, so anyone can POST fake payment confirmations. The idempotency key is the user ID (not the payment intent), so replayed requests create duplicate charges. The receipt endpoint leaks the customer’s full card details because AI included them in the response object “for completeness.”

The rule: Any AI-generated code touching auth, payments, PII, or secrets gets a dedicated security review — not just a code review. Every time. No exceptions. The cost of one security review is trivial compared to the cost of one breach.

Security Review for AI-Generated Code

  • Run at least two SAST tools (Semgrep + Snyk or CodeQL) — zero critical findings required
  • Check for hardcoded secrets — API keys, database URLs, JWT secrets, default passwords
  • Verify every endpoint has auth/authz middleware attached (not just defined)
  • Check all user inputs are validated and sanitized — especially in SQL queries, HTML rendering, and URL handling
  • Verify cryptographic functions use current algorithms (no MD5, SHA-1, ECB, or hardcoded keys)
  • Check for SSRF — does the code fetch user-controlled URLs? Are there allowlists?
  • Review error messages — do they leak stack traces, internal paths, or sensitive data?
  • Verify webhook endpoints validate signatures from the sending service
  • Check that environment variable fallbacks don’t expose default secrets
  • For payment/auth code: complete the 4-question threat model before approving

Run a SAST Scan on Your Latest AI PR

Do this now — it takes 10 minutes and will probably surprise you.

  1. Install Semgrep if you don’t have it: pip install semgrep (or brew install semgrep)
  2. Run it against your project with OWASP rules: 
    semgrep --config p/owasp-top-ten --config p/secrets .
  3. Review the findings. For each one:
    • Is it a real vulnerability or a false positive?
    • Was it in AI-generated code or human-written code?
    • Would your current review process have caught it?
  4. If Semgrep found zero issues — try adding Snyk Code as a second scanner. Remember: single-tool coverage catches less than 22% of actual vulnerabilities

Most developers running this for the first time find at least one hardcoded secret or one injection risk they didn’t know about. That’s not a failure — it’s exactly why automated scanning exists.

Build Your Security Gate

Set up automated security scanning for your team’s CI/CD pipeline:

  1. Copy the GitHub Actions workflow from this chapter (or adapt it for GitLab CI / Bitbucket Pipelines)
  2. Configure Semgrep with the p/owasp-top-ten and p/secrets rulesets
  3. Add Gitleaks for secret detection
  4. Set the pipeline to fail on critical findings but warn on medium findings (this prevents alert fatigue while blocking the worst issues)
  5. Run the pipeline on your last 5 PRs retroactively: 
    for commit in $(git log --oneline -5 --format=%H); do
  git checkout $commit && semgrep --config p/owasp-top-ten .
done
  6. Count how many issues the pipeline would have caught that your review process missed

Present the results to your team: “Here are the security issues our current review process missed. Here’s the automated gate that would catch them. It adds 2 minutes to CI.” Data-driven proposals get adopted faster than opinion-driven ones.
