Chapter 8

The Comprehension Workflow

A systematic process for understanding AI-generated code — not just shipping it

Why “Mostly Understand” Isn’t Enough

Chapter 7 showed you how to keep your skills sharp. This chapter is about applying those skills every single time AI-generated code lands in front of you. Because here’s the pattern that kills teams: you review the diff, you mostly get it, you approve and move on. Repeat fifty times. Now you own a codebase full of code you “mostly understand” — and “mostly” is where bugs hide.

Think of it like reading a recipe versus cooking from a recipe. You can skim a recipe and think “yeah, I get how soufflé works.” But the first time you actually make one, you discover that “fold gently” and “stir in” produce completely different results. The gap between reading code and understanding code works the same way. Skimming is not comprehension.

Addy Osmani coined the term comprehension debt in a March 15, 2026 article: the widening gap between the volume of code that exists in a codebase and the volume that any human engineer genuinely understands. Unlike technical debt, comprehension debt is invisible — tests pass, syntax is clean, CI is green — but the team’s mental model of the system quietly erodes.

Research backs this up. Xin Xia and colleagues’ 2018 IEEE field study (78 professional developers, 3,148 working hours) found developers spend roughly 58% of their time on program comprehension, not writing new code. With AI generating more code faster, that 58% is under pressure. The temptation is to cut comprehension time to ship faster. The result is code nobody owns.

The Core Principle

If you can’t teach someone what a piece of code does, you don’t understand it. And if you don’t understand it, you can’t debug it, refactor it, or extend it safely. The comprehension workflow exists to close that gap — systematically, every time.

The Five-Step Comprehension Workflow

This is the core of the chapter — a repeatable workflow you apply to every non-trivial AI-generated change. Five steps, each building on the last.

1. Read
What changed?

2. Intent
Why did it change?

3. Trace

Step 1: Read the Diff — What Changed?

Start with the shape, not the details. Use git diff --stat to see which files were touched and how much changed. This tells you the scope before you dive into specifics.

$ git diff --stat HEAD~1
 src/services/payment.ts    | 47 ++++++++++++++++-
 src/middleware/auth.ts      | 12 ++---
 src/utils/retry.ts          | 89 +++++++++++++++++++++++++++++++++
 tests/payment.test.ts       | 31 ++++++++++++
 4 files changed, 171 insertions(+), 8 deletions(-)

Three questions at this stage:

  • Scope check: Does the number of files and lines match the expected change? A “simple feature” touching 8 files deserves a raised eyebrow.
  • Unexpected files: Is anything modified that shouldn’t be? AI loves to “helpfully” refactor nearby code.
  • New files: retry.ts is new — does the project already have retry logic elsewhere? Duplication is a classic AI tell.

Step 2: Identify the Intent — What Was the AI Trying to Do?

Before reading line-by-line, answer: what was the goal? Check the commit message, the ticket, or the prompt that generated this code. If none of those exist, that’s already a problem — code without stated intent is code without accountability.

Compare the stated intent with the actual diff. Does the change accomplish what it claims? AI often achieves the stated goal while also introducing unrelated changes, over-engineering the solution, or solving a slightly different problem than what was asked.

The Intent Mismatch Test

Read the ticket or prompt first, form a mental picture of what the diff should look like, then read the actual diff. Where reality diverges from expectation is where bugs live. If the AI was asked to “add retry logic to the payment service” and the diff also modifies auth middleware — that’s scope creep that needs investigation.

Step 3: Trace the Data Flow — How Does Data Move?

This is where most reviewers stop too early. Don’t just read the function — trace how data enters, transforms, and exits.

For the payment service example:

  1. Where does the payment request come from? (API endpoint, queue, scheduled job?)
  2. What validation happens before it reaches this function?
  3. How does the retry logic interact with the payment gateway’s idempotency?
  4. What happens to the response? Where does it go next?
  5. If the function fails, who gets notified? What state is left behind?

AI-generated code often handles the function in isolation beautifully but ignores how it connects to the rest of the system. Tracing data flow catches these integration gaps.

Step 4: Question the Assumptions — What Does the AI Take for Granted?

Every piece of code makes assumptions. AI code makes more assumptions than human code because it doesn’t know your system. Look for:

  • Environment assumptions: Does it assume a specific database version, network configuration, or environment variable?
  • Input assumptions: Does it assume inputs are always valid, non-null, within expected ranges?
  • Concurrency assumptions: Does it assume single-threaded execution when your system is multi-threaded?
  • State assumptions: Does it assume the database or external service is always available?
// AI wrote this retry logic — spot the assumption
async function retryPayment(paymentId: string, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await paymentGateway.charge(paymentId);
    } catch (err) {
      if (i === maxRetries - 1) throw err;
      await sleep(1000 * Math.pow(2, i));
    }
  }
}

The assumption: retrying a payment is safe. But is the payment gateway idempotent? If the first call succeeded but the response was lost, retrying could charge the customer twice. The AI doesn’t know your gateway’s idempotency guarantees — you do.

Step 5: Test Your Understanding — Can You Predict Edge Cases?

The final check: without running the code, predict what happens with unexpected input. If you can predict correctly, you understand the code. If you can’t, you have gaps.

For the retry function above:

  • What happens if paymentId is an empty string?
  • What happens if the network times out after 30 seconds but the retry waits 4 seconds?
  • What happens if maxRetries is 0?
  • What happens if two instances of your service retry the same payment simultaneously?

Write your predictions down. Then check against the code. Where your predictions were wrong, that’s where your understanding was incomplete — and where bugs will eventually emerge.

The “Teach It Back” Technique

The single most effective comprehension technique is the oldest one in education: if you can teach it, you understand it. This is the Protégé Effect — research shows that preparing to teach material produces deeper understanding than simply studying it.

In practice, this means: after reviewing AI-generated code, explain it to someone. Out loud. In your own words. Not “it adds retry logic” — actually explain how the retry works, what triggers it, and why this approach was chosen over alternatives.

No colleague available? Use the rubber duck approach. Explain the code to an inanimate object, a voice memo, or a written note. The act of articulating forces you to confront the parts you don’t actually understand. GitHub Copilot CLI’s experimental “Rubber Duck” feature, announced April 6, 2026, productizes the same idea: a second model from a different family (GPT-5.4 when orchestrating with Claude) reviews the agent’s plan and work at three checkpoints — after drafting the plan, after a complex implementation, and after writing tests before executing them. The pairing closed 74.7% of the performance gap between Sonnet 4.6 and Opus 4.6 alone. The original technique (just talking out loud) still works just as well for human reviewers.

Teach-It-Back (Deep Understanding)
"This function retries failed payments using exponential backoff. It starts with a 1-second delay, doubles each time, and gives up after 3 attempts. The catch block only rethrows on the final attempt. But there's a gap — it doesn't check idempotency, so a timeout could cause a double charge."
You articulated the mechanism, the behavior, and a limitation. You understand this code.
Skim-and-Approve (Surface Understanding)
"It retries payments. Looks fine. The tests pass."
You described the label, not the behavior. You won’t catch the double-charge bug until it hits production.

Asking AI to Explain Its Decisions

One of the best uses of AI in your comprehension workflow is asking it to explain code it generated. But the quality of the explanation depends entirely on the quality of your prompt.

Prompts That Produce Useful Explanations
"Explain the retry logic in src/services/payment.ts. Why did you use exponential backoff instead of fixed intervals? What assumptions does this code make about the payment gateway's idempotency? What happens if two instances retry the same payment simultaneously?"
Specific, targeted questions that probe assumptions and alternatives. The explanation will reveal whether the AI considered edge cases or just generated a common pattern.
Prompts That Produce Useless Explanations
"Explain this code."
You’ll get a line-by-line narration that restates the code in English. That’s not understanding — it’s translation. You need to ask about decisions, not descriptions.

The key is to ask about decisions, not descriptions. “Why this approach?” is more valuable than “what does this do?” Because the “why” reveals whether the AI made a deliberate choice or just generated the most common pattern from its training data.

Active Reading vs. Passive Scanning

There are two ways to read code. Most developers default to the wrong one.

Passive scanning is reading code the way you read a news article — your eyes move across the lines, you get a general sense, and you move on. It feels productive. It isn’t. You retain almost nothing, and you miss subtle bugs.

Active reading is reading code the way you read a contract — you question every clause, you look for what’s missing, and you flag anything you don’t fully understand. It’s slower. It’s also the only way to actually comprehend what you’re reviewing.

Techniques for active reading:

  • Annotate as you go: Write quick notes next to each function — “this fetches user data,” “this validates the token.” If you can’t write a note, you don’t understand the function.
  • Predict before reading: Before reading a function body, read just the name and signature. Predict what it does. Then read the body and check your prediction.
  • Question defaults: When you see timeout = 30000, ask: why 30 seconds? Is that appropriate for this context? AI loves plausible-looking defaults that are actually arbitrary.
  • Read tests first: Tests tell you what the code is supposed to do. Reading tests before implementation gives you a spec to verify against.

When to Read Line-by-Line vs. Skim for Structure

Not every line deserves equal attention. Use a risk-based approach:

Read Line-by-Line Skim for Structure
Authentication and authorization logic UI layout and styling changes
Payment and financial calculations Generated boilerplate (config, imports)
Data validation and sanitization Test scaffolding (setup/teardown)
Concurrency and state management Documentation updates
Database migrations and queries Dependency version bumps

Osmani’s principle: “If code touches auth, payments, secrets, or untrusted input, treat AI as a high-speed intern.” Everything else can get a lighter pass — but never zero attention.

Building Mental Models of Unfamiliar Code

AI generates code in patterns you might not have chosen yourself. That unfamiliarity is actually an opportunity — if you engage with it rather than skip past it.

When you encounter an unfamiliar pattern, don’t just accept or reject it. Build a mental model of how it works:

  1. Name the pattern: Is this a retry with backoff? A circuit breaker? A saga? Naming it connects it to your existing knowledge.
  2. Draw the flow: Even a rough sketch on paper — boxes for components, arrows for data flow — makes abstract code concrete.
  3. Identify the invariants: What must always be true for this code to work correctly? Those invariants are your guardrails.
  4. Find the failure modes: How does this code fail? Gracefully? Silently? Catastrophically? The failure mode tells you the real risk.

The “Explain It Differently” Test

If you can only explain code using the same words and structure as the code itself, you’re describing, not understanding. Try explaining it using an analogy, a real-world equivalent, or a diagram. If you can translate it into a completely different representation, you genuinely understand it.

The Comprehension Journal

Chapter 7 introduced the learning journal for tracking knowledge gaps. The comprehension journal is its complement — a record of what you learned from each review, not just what you didn’t know.

## Comprehension Journal — 2026-04-10

**PR reviewed:** #347 — Add retry logic to payment service
**AI-generated:** Yes (Claude Code session)
**Time spent understanding:** 15 minutes

**What I learned:**
- Exponential backoff with jitter prevents thundering herd
- Our payment gateway IS idempotent (confirmed with team lead)
- The retry util duplicates logic in src/utils/http-client.ts

**What surprised me:**
- AI created a new retry utility instead of using existing one
- The backoff cap was set to 30s — our gateway times out at 15s

**Action taken:**
- Requested change: use existing retry utility
- Filed follow-up to align timeout with gateway SLA

Over time, these entries become a knowledge map of your codebase. When you need to touch the payment service six months later, you don’t start from scratch — you read your journal entries and rebuild context in minutes instead of hours.

Time-Boxing Comprehension

The uncomfortable truth: you can’t spend 45 minutes understanding every AI-generated function. Shipping matters. The goal is to spend the right amount of time — not zero, and not infinite.

Here’s a practical time-boxing framework:

2 min Trivial changes: config, formatting, dependency bumps. Skim the diff, check for surprises, approve.
10 min Standard features: new endpoints, UI components, utility functions. Run the 5-step workflow at normal speed.
30 min Complex or risky changes: auth, payments, data migrations. Full workflow, line-by-line reading, teach-it-back.
60+ min Architecture-level changes: new services, schema redesigns, cross-cutting concerns. Pair review recommended.

The key insight: match your comprehension investment to the blast radius. A bug in a config file affects one deployment. A bug in payment logic affects every customer. Allocate your time accordingly.

When You’re Over the Time Box

If you hit your time limit and still don’t understand the code, that’s not a sign to approve anyway. It’s a sign the code is too complex, poorly structured, or that it needs to be broken into smaller changes. Request the author (or the AI) to refactor into smaller, reviewable pieces. A PR you can’t understand in a reasonable time is a PR that shouldn’t be merged as-is.

Code That Solves the Problem in a Way You Wouldn’t Have Thought Of

Not all AI tells are red flags. Sometimes AI generates a solution using a pattern or technique you’ve never seen. Maybe it uses a bitwise operation where you would have used a conditional. Maybe it restructures a loop into a functional pipeline you didn’t know the language supported.

This is where the most learning happens. When AI surprises you, don’t just approve it because it’s clever. Run it through the 5-step workflow. Understand why this approach works. Add it to your comprehension journal. Next time, you’ll recognize the pattern — and you’ll be the one teaching it to a colleague.

The tell: you find yourself thinking “I would never have written it this way.” That’s not necessarily bad — it’s an invitation to learn.

Merging Code You “Mostly Understand”

Scenario: It’s Friday afternoon. You’ve been reviewing a 400-line AI-generated PR for 20 minutes. You understand 80% of it. The 20% you don’t fully grasp is in the error handling — some retry logic with cancellation tokens that you’ve never used before. Tests pass. CI is green. You think: “It probably works. I’ll learn it later.”

You won’t learn it later. You’ll forget about it until the error handling triggers in production and you can’t debug it because you never understood it in the first place.

The rule: If you can’t explain the error handling and edge case logic, the PR isn’t ready to merge. Those are exactly the parts that matter most during incidents. The happy path will work fine; it’s the 20% you skipped that will wake you up at 3 AM.

The Comprehension Workflow Checklist

  • Read the diff shape first (git diff --stat) — does the scope match the intent?
  • Identify the intent — read the ticket/prompt before the code
  • Trace data flow through the change — where does data enter, transform, and exit?
  • List the assumptions the code makes about the system
  • Predict edge case behavior before running the code
  • Explain the change out loud (teach-it-back) — can you articulate how it works?
  • Ask AI targeted “why” questions, not generic “explain this”
  • Time-box your review based on blast radius (2 / 10 / 30 / 60 min)
  • Log what you learned in your comprehension journal
  • If you can’t explain the error handling, don’t approve the PR

Write the Paragraph

Do this right now. It takes five minutes and tells you exactly how well your comprehension workflow is working.

  1. Find your most recent AI-generated change: git log --oneline -5
  2. Pick one non-trivial commit
  3. Close all AI tools
  4. In a scratch file or on paper, write a one-paragraph explanation of:
    • What the change does
    • Why it was made (the intent)
    • How data flows through it
    • One assumption it makes that could break
  5. Now open the actual diff and compare. What did you miss?

If your paragraph was accurate and complete — your comprehension is solid. If you missed the data flow or couldn’t name an assumption — use the 5-step workflow on this change right now. That’s real practice.

Build Your Comprehension Workflow Habit

Turn the 5-step workflow into a repeatable habit over the next two weeks:

  1. Week 1: Print the 5-step workflow (or stick it on a monitor note). For every AI-generated PR you review, physically check off each step. Yes, every one.
  2. Week 2: Start a comprehension journal. After each review, write a 3-line entry: what you learned, what surprised you, and one action you took.
  3. Ongoing: At the end of week 2, review your journal. Look for patterns:
    • Which steps do you skip most often? (That’s where your blind spot is)
    • What surprises you most frequently? (That’s where your knowledge gaps are)
    • How long does each review actually take? (Calibrate your time boxes)

The goal isn’t perfection — it’s building the muscle memory so the 5-step workflow becomes automatic, like checking your mirrors before changing lanes.
