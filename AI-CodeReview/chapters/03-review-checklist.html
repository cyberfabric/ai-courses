A 10-point system for reviewing any AI-generated change — from intent to understanding

From Instinct to System

In Chapter 2, you trained your eye. You can spot the tells — the over-abstraction, the hallucinated APIs, the phantom edge-case handlers. That’s a real skill.

But here’s the problem: instinct is unreliable under pressure. It’s 4:30 PM on a Friday. You have three PRs in your review queue. The AI-generated one “looks fine.” You approve it. Monday morning, on-call pages you because the “fine” PR had an unguarded auth path.

Think about airline pilots. A pilot with 10,000 hours of flight time could probably eyeball the instrument panel and say “looks good.” They don’t. They follow a pre-flight checklist. Every single time. Not because they’re inexperienced — because the human brain skips steps under pressure, fatigue, and routine. The checklist doesn’t replace expertise. It prevents expertise from cutting corners.

Why does this matter? Because without context, you’re reviewing blind. The AI that wrote the code cannot answer questions during review. The PR description is the only voice it gets. If that description says nothing useful, you have zero leverage.

Here’s what the difference looks like:

Bad PR Description
“Added user search feature.”
No intent, no proof, no risk assessment, no focus areas. You’re on your own.
Good PR Description
“Implements user search (TICKET-234). AI-generated via Claude Code. Tests cover exact match, partial match, and empty results. Risk: new DB index on users.name — check migration. Focus review on: search ranking logic in search.ts:45-80.”
Intent, proof, AI disclosure, risk, and focus — all in four sentences. Now you know exactly where to spend your review time.

The Single Highest-Impact Habit

If you take one thing from this chapter, make it this: require every AI-generated PR to include an intent statement and a risk assessment. This single habit catches more issues than any review tool. A developer who can articulate the intent and risks has already thought about the code. One who can’t — hasn’t.

The 10-Point AI Code Review Checklist

Here it is — the checklist. Ten questions you apply to every AI-generated change. Not ten optional suggestions. Ten mandatory checkpoints. Print it. Bookmark it. Make it a PR template.

The 10-Point AI Code Review Checklist

  • 1. Intent clarity — Does the change match what was requested? Can I state in one sentence what problem this solves?
  • 2. Logic correctness — Does the algorithm actually do what it claims? Trace through with a concrete example, not just a code read.
  • 3. Edge cases — What happens with null, empty, boundary values, concurrent access? Did the AI handle them or pretend they don’t exist?
  • 4. Security — Auth checks present? Input validated? Secrets hardcoded? XSS vectors? SQL injection? (AI gets this wrong 2.74× more often.)
  • 5. Performance — Unnecessary loops? N+1 queries? Memory leaks? Missing pagination? Excessive I/O? (AI produces 8× more I/O issues.)
  • 6. Duplication — Does this duplicate existing code in the codebase? AI doesn’t know your utils folder exists.
  • 7. Integration — Does it work with the rest of the system, not just in isolation? Does it respect existing error handling patterns, logging conventions, config structures?
  • 8. Tests — Are the tests testing behavior or just achieving coverage? Do they test what should happen, or just confirm what does happen?
  • 9. Maintainability — Will someone understand this in 6 months? Clear variable names, reasonable complexity, no magic numbers?
  • 10. The “explain it back” test — Can I (or the author) explain every function’s purpose and every non-obvious decision? If not, the PR isn’t ready.

Let’s walk through the points that catch the most bugs in practice.

Intent Clarity in Action

You open a PR. The AI added a Redis caching layer to the user profile endpoint. The diff is clean, the code compiles, the tests pass. Approved?

Wait. Why was caching added? Was there a performance issue? A requirement? Or did the AI just decide the endpoint “needed” caching because caching is a common pattern in its training data?

Without intent, you cannot evaluate correctness. A cache that nobody asked for is technical debt — it introduces invalidation complexity, stale data risks, and an infrastructure dependency. The code might be perfect. The decision might be wrong.

# PR comment that surfaces the intent problem:
> This adds Redis caching to /api/users/:id with a 5-minute TTL.
>
> Question: what's the motivation? I don't see a performance
> ticket for this endpoint. Our current p99 is 45ms, which is
> within SLA. If this was AI-suggested, we should evaluate
> whether the caching complexity is worth it.

That comment takes 30 seconds to write. It might save you a week of debugging stale-cache issues later.

Logic Correctness: Trace, Don’t Skim

Checklist point #2 says “trace through with a concrete example.” This is the difference between reading code and reviewing code. Here’s what that looks like:

// AI-generated sorting for a task dashboard
function sortTasks(tasks) {
  return tasks.sort((a, b) =>
    new Date(b.createdAt) - new Date(a.createdAt)
  );
}

The code is valid JavaScript. The sort works. But the ticket said: “sort tasks by priority, then by creation date within the same priority.” The AI sorted by createdAt only. Priority is ignored.

A code reader sees: “sorting function, looks correct.” A code reviewer traces the requirement to the implementation and catches the gap. This is why you check intent first — it gives you something to trace against.

Security: The 2.74× Problem

CodeRabbit’s 2026 analysis found that security issues appear at 2.74× the frequency in AI-generated code. XSS, missing auth checks, hardcoded secrets, insecure defaults. The AI produces functional code that forgets to ask “who should be allowed to run this?”

// AI-generated endpoint — spot the problem
app.get('/api/admin/users', async (req, res) => {
  const users = await db.users.findAll({
    include: ['email', 'role', 'lastLogin']
  });
  res.json(users);
});

// What's missing: authentication AND authorization
// This endpoint exposes every user's email and role to anyone
// who can guess the URL. No middleware, no token check, nothing.

This is a pattern AI produces constantly. It builds the feature. It forgets the guard. When you hit checklist point #4, you’re specifically looking for this: does every endpoint that handles sensitive data verify who’s asking?

Tests: The Coverage Trap

Checklist point #8 distinguishes between coverage and quality. AI is excellent at generating tests that achieve 90% code coverage. The problem: those tests often test how the code works, not what it should do. When you refactor the implementation, the tests break — even though the behavior is unchanged.

This is so common it deserves its own red flag.

AI-Generated Tests That Test Implementation, Not Behavior

Here’s the pattern. The AI writes a calculateTotal function that uses Array.reduce(). Then it generates a test:

// AI-generated test — testing implementation
test('calculateTotal uses reduce', () => {
  const spy = jest.spyOn(Array.prototype, 'reduce');
  calculateTotal([{ price: 10, qty: 2 }]);
  expect(spy).toHaveBeenCalled();
  spy.mockRestore();
});

// What the test SHOULD check — testing behavior
test('calculateTotal returns correct sum', () => {
  expect(calculateTotal([{ price: 10, qty: 2 }])).toBe(20);
  expect(calculateTotal([])).toBe(0);
  expect(calculateTotal([{ price: 0, qty: 5 }])).toBe(0);
});

The first test breaks if you refactor to a for loop. The second test doesn’t care how you calculate the total — it only cares that the answer is right. If refactoring the implementation breaks the test but the output is identical, the test is testing the wrong thing.

Tests That Mirror Code Structure

A subtler version of the same problem. The AI reads the implementation and writes a test that simply recapitulates it:

// Implementation
function getUserById(id) {
  return db.users.findOne({ where: { id } });
}

// AI-generated "test" — it's tautological
test('getUserById returns user', async () => {
  const result = await getUserById(1);
  const expected = await db.users.findOne({ where: { id: 1 } });
  expect(result).toEqual(expected);
});

This test will always pass because it does the exact same thing as the function it’s testing. It’s circular: “Does the code do what the code does? Yes.” A real test would seed a known user and verify the returned object matches the seeded data — testing against a known expectation, not against the implementation itself.

Security-Critical Review: The High-Speed Intern Rule

In Chapter 1, we introduced the “high-speed intern” mental model. Nowhere is it more important than in security-critical code.

The rule: Any AI-generated code touching authentication, authorization, payments, secrets, or user data gets treated as a first draft from a developer who has never seen your security model. Every assumption is questioned. Every guard is verified.

The 45% Stat Should Change Your Behavior

Veracode’s 2026 analysis found that 45% of AI-generated code contains security vulnerabilities — reaching 72% in Java codebases. For security-critical paths, the cost of missing one vulnerability is orders of magnitude higher than the time saved by AI generation. Slow down.

When reviewing security-critical AI code, check these specifically:

  • Every authorization path — does every endpoint that handles sensitive data verify the user has permission? Not just authentication (who are you?) but authorization (are you allowed to do this?).
  • Hardcoded secrets — search the diff for API keys, connection strings, tokens. AI loves to put placeholder values that look like real secrets: const API_KEY = 'sk-abc123...'.
  • Input sanitization — every user-facing input should be validated and sanitized. AI often validates format but not content (e.g., checks that an email looks like an email but doesn’t prevent SQL injection in the username field).
  • Crypto standards — AI uses deprecated cryptographic functions. Watch for MD5 for hashing, SHA1 for passwords, or Math.random() for tokens.

Incrementalism: Breaking AI Output into Reviewable Pieces

Here’s a scenario you know well: you ask AI to build a feature, and it generates 800 lines in one session. A data model, a service layer, three API routes, and tests. One giant commit. One giant review.

You cannot review 800 lines effectively. Google’s review research (PRs under 100 lines clear in under an hour, PRs of 100–500 lines take ~4 hours) and Microsoft’s PR-size analysis (PRs under 300 lines get 60% more thorough reviews; automated warnings on PRs over 400 lines drove a 35% drop in post-merge defects) converge on the same finding: teams keeping PRs under 400 lines see a 30–50% reduction in time-to-first-review and a 40–60% reduction in total review cycle time. The problem isn’t the code — it’s the batch size.

The AI generated a database model, a service layer, and route handlers? Those are three commits at minimum. Review the model first — does the schema make sense? Then the service layer — does the business logic match requirements? Then the routes — are the endpoints secure and well-integrated?

The Batch Size Mismatch

The biggest mistake teams make with AI-generated code is reviewing it in the same batch size the AI produced it. AI doesn’t know your review capacity. AI doesn’t get fatigued at line 400. You do. Break it down. Your future self will thank you at 2 AM when the on-call page comes in and you can pinpoint exactly which small, well-reviewed commit introduced the issue.

The “Explain It Back” Test

Checklist point #10 is special. It’s not a code check — it’s a comprehension check. The best-performing engineering teams in 2026 are now requiring AI-generated PRs to include explicit explanations: what the approach is, what edge cases were considered, and why this implementation was chosen.

The test works like this: before you approve an AI-generated PR, you must be able to explain three things:

  1. The intent — what problem does this solve, in your own words?
  2. The edge cases — what does it handle, and what does it not handle?
  3. The implementation choice — why this approach over alternatives?

How to do it with your own AI-generated code: Close the AI chat. Open the diff. Explain each function’s purpose without looking at the AI’s explanation. If you get stuck on a function, that’s the part that needs deeper review — or the part you don’t actually understand yet.

How to do it in a team review: Instead of leaving “LGTM,” leave a summary. That summary proves you understood the code AND often surfaces real concerns:

Rubber-Stamp Review
“LGTM, tests pass.”
Proves nothing. The reviewer might have glanced at it for 14 seconds. No evidence of comprehension.
Explain-It-Back Review
“This adds retry logic to the payment service — up to 3 retries with exponential backoff. Question: should we add a circuit breaker to avoid hammering a failing upstream? Also, the retry catches all errors. Should we only retry on 5xx/timeout, not on 4xx (client errors)?”
The reviewer understood the code and spotted two real issues: missing circuit breaker and overly broad retry scope. This is review that adds value.

When a developer cannot fill in “edge cases considered,” they do not understand what they shipped. That is the signal — and it surfaces before the reviewer has read a single line of code.

Apply the 10-Point Checklist to Your Next PR

This takes 10 minutes. Do it on your very next AI-generated PR — not someday, next time.

  1. Open your recent commits: git log --oneline -10
  2. Find the most recent AI-generated change
  3. Run git show <commit-hash> to see the full diff
  4. Go through all 10 checklist points. Write a one-line answer for each:
    • Intent? Logic? Edge cases? Security? Performance?
    • Duplication? Integration? Tests? Maintainability? Explain it back?
  5. Which points were easy to answer? Those are your review strengths.
  6. Which points made you pause or say “I’m not sure”? That’s where bugs are hiding.

Make this your default process for the next two weeks. It starts at 10 minutes per PR and gets faster with practice. Most developers report it drops to 5 minutes within a week.

- [ ] Edge cases handled (null, empty, boundary)
- [ ] Security: auth, validation, no hardcoded secrets
- [ ] Performance: no N+1, no unnecessary loops
- [ ] No duplication of existing code
- [ ] Integrates with existing patterns
- [ ] Tests check behavior, not implementation
- [ ] Maintainable in 6 months
- [ ] I can explain every function in this PR
  4. Use this template for your next 5 PRs. After 5, note which items consistently surface issues and adjust.
