Chapter 2

Reading AI-Generated Code

How to spot the “tells,” triage a 500-line diff, and know when the AI actually nailed it

Training Your Eye

In Chapter 1, you learned why reviewing AI code demands a different mindset. Now we need to train your eye to actually see the problems.

AI-generated code has fingerprints. Not obvious ones — it won’t say // written by Claude at the top. But there are patterns, tics, and habits that AI consistently produces. Once you learn to recognize them, you’ll scan a diff and instinctively know which parts need close attention and which parts are solid.

Think of it like a food inspector at a restaurant. A novice inspector checks every dish the same way. An experienced one walks in, sees three things out of place, and knows exactly where to focus. That’s what this chapter gives you — the trained instinct to know where the problems hide in AI output.

The Seven AI Code “Tells”

These are the patterns that consistently appear in AI-generated code across tools — Claude Code, Copilot, Cursor, Windsurf. Once you know them, you can’t unsee them.

Tell #1: Over-Abstraction

AI loves abstraction. Give it a simple task, and it’ll build a factory for a builder for a strategy. Three files where one function would do.

// AI wrote this for a simple config lookup
class ConfigurationProvider {
  constructor(private readonly loader: ConfigLoader) {}
  async getConfig<T>(key: string): Promise<T> {
    const raw = await this.loader.load(key);
    return this.deserialize<T>(raw);
  }
  private deserialize<T>(raw: string): T {
    return JSON.parse(raw) as T;
  }
}

// What you actually needed
const config = JSON.parse(fs.readFileSync('config.json', 'utf8'));

The AI version isn’t wrong — it’s just solving a problem you don’t have yet. If you have one config file, you don’t need a generic deserialization provider with dependency injection. That complexity will confuse the next developer who touches this code.

Tell #2: Hallucinated APIs

This is one of the most dangerous patterns. AI invents methods that look real but don’t exist, or uses patterns from an older version of a framework. Lasso Security’s package-hallucination research (Bar Lanyado, 2024), corroborated by a USENIX 2024 study covering 576,000 code samples across 16 LLMs, found that roughly 1 in 5 AI code samples includes hallucinated references to non-existent libraries or APIs — with rates climbing to 35% for Python.

// AI generated this for a Next.js app
import { useServerAction } from 'next/server';  // This doesn't exist

export default function Dashboard() {
  const data = useServerAction(fetchDashboard);  // Hallucinated API
  return <div>{data.summary}</div>;
}

The code reads perfectly. The import looks plausible. The function name follows Next.js naming conventions. But useServerAction doesn’t exist in any version of Next.js. The AI blended patterns from React Server Components, Server Actions, and client hooks into something that looks right but won’t compile. Models are trained with a data cutoff, and if a library released breaking changes after that cutoff, the model generates code for the old API — or worse, an API that never existed at all.

Quick Hallucination Check

When reviewing AI code that uses a library API you don’t recognize, spend 30 seconds checking the docs. Run grep -r "useServerAction" node_modules/next/ or check the official docs. This single habit catches the most dangerous class of AI bugs.

Tell #3: The “Kitchen Sink” Import

AI imports more than it needs. It pulls in entire modules when it uses one function. It imports types it never references. Look at the top of any AI-generated file and count the unused imports.

import {
  useState, useEffect, useCallback, useMemo,
  useRef, useContext, useReducer, useLayoutEffect  // Only useState and useEffect are used
} from 'react';
import axios from 'axios';       // fetch() is used instead, 30 lines down
import _ from 'lodash';          // Only _.debounce is used — import { debounce } instead

Unused imports aren’t just messy — they increase bundle size, add unnecessary dependencies, and signal that the AI didn’t actually think about what it needed.

Tell #4: Overly Defensive Error Handling

AI wraps everything in try-catch blocks, checks for impossible nulls, and adds fallbacks that mask real errors instead of surfacing them.

async function getUser(id: string) {
  try {
    const user = await db.users.findById(id);
    if (user === null || user === undefined) {  // findById already returns null
      return null;
    }
    if (typeof user.name !== 'string') {  // Schema guarantees this is a string
      user.name = String(user.name ?? '');
    }
    return user;
  } catch (error) {
    console.log('Error fetching user:', error);  // Swallowing the error
    return null;                                  // Caller has no idea what went wrong
  }
}

Every check here is technically “safe.” But the redundant null check, the type coercion on a typed field, and the error-swallowing catch block all add noise without adding safety. Worse, that return null in the catch block means upstream code will get null on a database connection failure and silently misbehave instead of failing loudly.

Tell #5: Copy-Paste Duplication

AI repeats similar code blocks instead of extracting shared utilities. You’ll see three functions that differ by one parameter, or two API routes with identical validation logic inlined in both.

// AI generated separate handlers with near-identical logic
app.post('/api/users', async (req, res) => {
  if (!req.body.name || req.body.name.length < 2) return res.status(400).json({ error: 'Invalid name' });
  if (!req.body.email || !req.body.email.includes('@')) return res.status(400).json({ error: 'Invalid email' });
  // ... create user
});

app.put('/api/users/:id', async (req, res) => {
  if (!req.body.name || req.body.name.length < 2) return res.status(400).json({ error: 'Invalid name' });
  if (!req.body.email || !req.body.email.includes('@')) return res.status(400).json({ error: 'Invalid email' });
  // ... update user (same validation, copy-pasted)
});

Each AI prompt is a fresh conversation. The AI doesn’t think “I already wrote this validation — let me extract it.” It solves each task independently, producing duplication that a human developer would naturally refactor.

Tell #6: Dead Code and Unreachable Branches

AI generates functions that are never called, enum values that are never used, and conditional branches that can never execute. It’s hedging — covering scenarios that might exist in some codebase, but don’t exist in yours.

// AI generated a utility file with 12 functions
// Only 3 are imported anywhere in the codebase
export function formatDate(date: Date): string { ... }     // Used
export function formatTime(date: Date): string { ... }     // Used
export function formatDateTime(date: Date): string { ... } // Used
export function formatRelative(date: Date): string { ... } // Never imported
export function formatDuration(ms: number): string { ... } // Never imported
export function parseDate(str: string): Date { ... }       // Never imported
// ... 6 more unused functions

This is the AI being “helpful” — you asked for date formatting, so it gave you the whole toolkit. But dead code is maintenance weight. It gets imported accidentally, confuses new team members, and shows up in search results when someone is looking for the function they actually need. Remove it.

Tell #7: The Over-Commenting Pattern

We covered this in Chapter 1 as an AI tell, but it’s worth repeating as a reading skill: when every line has a comment that restates the code, you’re looking at AI output. Human developers comment the why, not the what.

AI-Style Comments
// Check if the user is an admin
if (user.role === 'admin')
Restates what the code does. Adds zero information. Classic AI tell.
Human-Style Comments
// Admin bypass: skip rate limiting per SEC-2847
if (user.role === 'admin')
Explains why this check exists and links to the decision. This is what useful comments look like.

The Phantom Edge Case Handler

One of the most distinctive AI tells: a function that handles 5 edge cases when only 2 are possible in your system.

function calculateDiscount(order) {
  if (!order) return 0;                           // Can't happen: caller validates
  if (order.items.length === 0) return 0;         // Can't happen: empty carts are blocked
  if (order.total < 0) return 0;                  // Can't happen: totals are unsigned
  if (order.currency !== 'USD') {                 // Can't happen: single-currency system
    return convertAndDiscount(order);
  }
  // The actual discount logic finally starts here
  return order.total > 100 ? order.total * 0.1 : 0;
}

The AI doesn’t know your system. It imagines all the ways things could go wrong in any system and defends against all of them. The result: code that’s harder to read, tests that are harder to write, and a false sense of robustness. When you spot this, ask: “Which of these checks actually protect against real scenarios in our system?”

How AI Hallucinates: The “Looks Smart but Is Fragile” Pattern

Beyond individual tells, there’s a broader pattern to understand: AI writes code that handles the happy path beautifully but crumbles on edge cases.

This happens because LLMs predict the most likely next token. The happy path is the most common pattern in training data. Edge cases, by definition, are rare — so the model either ignores them or invents plausible-sounding handling that doesn’t actually work.

The Happy Path Trap

When reviewing AI code, your instinct will be to trace the happy path. Resist it. The happy path is almost always fine. Instead, immediately ask: what happens when the input is empty? What happens when the network times out? What happens when two requests arrive at the same time? That’s where AI code breaks.

Addy Osmani recommends treating large AI-generated outputs as drafts, not production code. Run it to verify the approach works, then reimplement in smaller, comprehensible pieces. The second draft is always cleaner, and — critically — you actually understand it.

Speed-Reading Large AI Diffs

AI-generated PRs are 154% larger than human PRs on average. You’re going to see 500-line diffs regularly. You cannot read every line with the same attention. You need a triage strategy.

The classic SmartBear/Cisco peer-review study — still the largest of its kind, with 2,500 reviews of 3.2M lines — established that reviewer effectiveness collapses past 200–400 lines or 60–90 minutes per session. GitClear’s 2024 analysis of 12,638 PRs found the median developer has only 2–5 hours per week to spend on review across all incoming PRs. That’s not enough attention for AI-generated diffs that routinely blow past 500 lines. Here’s a triage approach that fits the time you actually have:

1. Stat View
git diff --stat
Which files changed?
→
2. Shape
Skim structure:
new files, renames, deletes
→
3. Hot Spots
Focus on: auth,
data, config, tests
→
4. Deep Read
Line-by-line only
on hot spot files

Step 1: Start with git diff --stat. This shows you which files changed and how many lines were added or removed. A 500-line diff that touches 20 files is very different from one that touches 2 files.

Step 2: Read for shape, not detail. Scroll through the entire diff quickly. Don’t read the code — notice the structure. Are there new files? Renamed files? Deleted files? Large blocks of generated boilerplate vs. small targeted changes?

Step 3: Identify hot spots. These are the files that matter most: anything touching authentication, database queries, payment logic, configuration, or public API surfaces. Also look for test files — do they exist? Do they test the right things?

Step 4: Deep-read only the hot spots. Now go line-by-line, but only on the files that matter. For every function in those files, mentally trace through three scenarios: the happy path, the empty/null input, and the boundary case.

Here’s what this looks like in practice:

$ git diff --stat HEAD~1
 src/api/auth.ts          | 45 +++++++++++++++
 src/api/users.ts         | 12 ++++
 src/utils/validation.ts  | 89 +++++++++++++++++++++++++++++++
 src/types/index.ts       |  8 +++
 tests/auth.test.ts       | 67 +++++++++++++++++++++++
 package.json             |  2 +-
 6 files changed, 223 insertions(+), 1 deletion(-)

From this stat view alone, you already know: auth.ts is a hot spot (auth = high risk), validation.ts is the biggest change (89 lines — but validation code is often boilerplate), and package.json changed (check what dependency was added). Start your deep read with auth.ts, then check package.json for supply chain risk, then skim the rest.

The “Does This Code Teach Me Anything?” Test

As you scan AI-generated code, ask yourself: “Is there anything here I didn’t already know?” If the code uses a pattern or technique that’s new to you, slow down. That’s either something worth learning — or something the AI invented that doesn’t actually work. Either way, it deserves your full attention.

When AI Code Is Actually Good

Not all AI code is suspicious. Part of being a good reviewer is knowing when to not waste time. Here are signs that the AI nailed it:

  • It follows existing patterns. The code matches the conventions already in your codebase — same naming, same structure, same error handling approach.
  • It’s boring. Standard CRUD, straightforward data transformations, simple utility functions. The kind of code that’s tedious to write but hard to get wrong.
  • The tests are behavior-focused. Instead of testing implementation details, the tests verify what the code does, not how it does it.
  • It’s appropriately sized. Small, focused functions. No unnecessary abstraction layers. No dead code.
  • The imports are clean. Only what’s needed, no kitchen-sink imports, no deprecated packages.

When you see these signals, give a quick sanity check and move on. Save your deep-review energy for the code that triggers your tells.

The 80/20 Rule of AI Code Review

Roughly 80% of AI-generated code is boring, correct, and not worth agonizing over. The other 20% contains the bugs, security holes, and architectural mismatches that cause real problems. Your job as a reviewer is to spend 80% of your time on the dangerous 20% — and the tells are how you find it fast.

Code You Can’t Trace Back to a Requirement

The biggest red flag when reading AI-generated code: changes that don’t map to any requirement, ticket, or conversation.

AI agents frequently exhibit scope creep. You asked for a new API endpoint, and the PR also includes a refactored database layer, two new utility files, and a modified config. None of that was requested. None of it was reviewed as part of the original design.

When you see changes that don’t trace back to the task at hand, flag them immediately. Either they should be in a separate PR, or they shouldn’t exist at all. Untracked changes are how bugs, security holes, and technical debt sneak into production.

Reading AI Code Efficiently

Here’s your checklist for reading AI-generated code. Use it as a quick pre-flight before every review.

Reading AI Code: The Quick Scan

  • Run git diff --stat first — understand the shape before the detail
  • Check imports — are they all used? Any deprecated packages?
  • Scan for hallucinated APIs — do all method calls actually exist?
  • Look for copy-paste duplication — are there similar blocks that should be extracted?
  • Check error handling — is it masking errors or surfacing them?
  • Count the edge-case guards — are they protecting against real scenarios?
  • Read the comments — do they explain why or just restate the code?
  • Trace the happy path, then trace the failure path
  • Verify scope — does every change map to the original requirement?
  • Check for dead code — unused functions, unreachable branches, commented-out blocks

Spot the Tells

This exercise builds your pattern recognition. Do it now — it takes ten minutes.

  1. Open your recent git history: git log --oneline -20
  2. Find a commit or PR where AI generated most of the code
  3. Run git show <commit-hash> to see the full diff
  4. Read through the diff and write down every “tell” you spot:
    • Over-abstraction?
    • Hallucinated API?
    • Kitchen-sink imports?
    • Overly defensive error handling?
    • Copy-paste duplication?
    • Dead code?
    • Over-commenting?
    • Phantom edge cases?
  5. Count your tells. If you found 3 or more, your pattern recognition is already developing.

The AI Tell Catalogue

Build a personal catalogue of AI tells you encounter in your codebase. This becomes a reference you can use (and share with your team) during reviews.

  1. Create a file called ai-tells.md in your notes
  2. For each tell you find in your next 5 code reviews, record:
    • Pattern: what did the AI do? (e.g., “imported lodash for a single debounce”)
    • File: where did you find it?
    • Risk: low / medium / high
    • Fix: what should it have been instead?
  3. After 5 reviews, look for patterns. Which tells appear most often in your codebase?
  4. Share the catalogue with your team — it becomes a shared vocabulary for review discussions
