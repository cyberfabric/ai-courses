Chapter 6

Understanding What Changed

Git-based techniques for tracing, understanding, and reconstructing AI-generated changes

Why Git Archaeology Matters Now

Chapters 3–5 gave you tools for reviewing code as it lands. But there’s a different problem: understanding code after it landed. What did the AI agent do during that 45-minute session yesterday? What changed across those 12 commits? Why does auth.ts look different than you remember?

Think of git history like sedimentary rock layers. Each commit is a layer. A geologist reads rock layers to understand what happened millions of years ago. You read git history to understand what happened last Tuesday. The difference in 2026 is that AI agents deposit layers fast — sometimes 20 commits in a single session — and the layers don’t always explain themselves.

Addy Osmani calls this comprehension debt: the growing gap between how much code exists in your system and how much of it any human genuinely understands. Tests stay green, syntax stays clean, but the team’s mental model of the system quietly erodes. Git archaeology is how you fight that erosion.

The Four Git Commands You Need

You don’t need to be a git wizard. Four commands cover 90% of AI code archaeology. Learn them well.

1. git diff --stat — The Shape of the Change

Before you read any code, see the shape. How many files changed? How big are the changes? Where’s the weight?

$ git diff --stat HEAD~5..HEAD
 src/api/auth.ts           | 142 +++++++++++++++++++++
 src/api/users.ts          |  18 ++-
 src/middleware/rate-limit.ts | 89 ++++++++++++++
 src/utils/crypto.ts       |  34 ++++++
 src/utils/validation.ts   |   6 +-
 tests/auth.test.ts        |  67 +++++++++++
 package.json              |   3 +-
 7 files changed, 351 insertions(+), 8 deletions(-)

From this alone, you know: the bulk of the work is in auth.ts (142 lines, high risk) and rate-limit.ts (89 lines, new file). The validation.ts change is tiny (6 lines, probably safe). Start your investigation with auth.ts.

2. git log --oneline --graph — The Story

See the sequence. How did the AI work? Was it one massive commit or a logical progression?

$ git log --oneline --graph -10
* a3f8c21 Add rate limiting middleware
* 7b2e459 Add auth endpoint tests
* 1d4c890 Implement token refresh logic
* 9e7f312 Add crypto utilities for JWT signing
* 5c1a0b8 Set up auth route structure
* 82d6f1a (origin/main) Update README

This tells a story: the AI built from the bottom up — structure first, then crypto, then logic, then tests, then middleware. That’s a logical sequence. If instead you see one giant commit titled “Add auth system,” you have a problem — there’s no history to investigate when something breaks.

3. git show <commit> — The Detail

Once you know which commit matters, read it. git show displays the full diff for a single commit, with the commit message at the top.

$ git show 1d4c890
commit 1d4c890...
Author: Claude Code <noreply@anthropic.com>
Date:   Wed Apr 9 14:23:41 2026

    Implement token refresh logic

    Adds refresh token rotation with 7-day expiry.
    Revokes old refresh token on use to prevent replay.

diff --git a/src/api/auth.ts b/src/api/auth.ts
...

Good commit messages tell you what and why. Bad ones tell you nothing. We’ll cover how to enforce good ones shortly.

4. git blame — Who Wrote This Line?

When you’re debugging a specific function and need to know when it was introduced — and by whom (or what):

$ git blame src/api/auth.ts -L 45,55
1d4c890 (Claude Code 2026-04-09 14:23) 45| async function refreshToken(token: string) {
1d4c890 (Claude Code 2026-04-09 14:23) 46|   const payload = verifyRefreshToken(token);
1d4c890 (Claude Code 2026-04-09 14:23) 47|   if (!payload) throw new AuthError('Invalid refresh token');
a3f8c21 (Claude Code 2026-04-09 14:41) 48|   await revokeToken(token);  // Added in rate-limit commit??
1d4c890 (Claude Code 2026-04-09 14:23) 49|   const newAccess = signAccessToken(payload.userId);
1d4c890 (Claude Code 2026-04-09 14:23) 50|   const newRefresh = signRefreshToken(payload.userId);

Line 48 stands out — it was added in a different commit (a3f8c21, the rate-limiting commit). Why is token revocation in a rate-limiting commit? That’s a sign of scope creep — the AI agent drifted from its task.

Blame with Context

Use git blame -L 45,55 to focus on specific line ranges instead of blaming an entire file. For long files, this turns a wall of text into a focused investigation. Add -w to ignore whitespace changes — helpful when the AI reformatted code without changing logic.

Advanced Diff Strategies

When AI refactors code — moving functions between files, renaming variables, restructuring modules — the standard diff is almost unreadable. These flags help:

--word-diff — See What Actually Changed

$ git diff --word-diff HEAD~1
const [-timeout-]{+requestTimeout+} = config.get('timeout');
const result = await fetch(url, { [-timeout-]{+signal: AbortSignal.timeout(requestTimeout)+} });

Instead of showing entire lines as added/removed, --word-diff highlights the specific words that changed. Perfect for AI renames and small refactors.

--color-moved — Track Relocated Code

$ git diff --color-moved=dimmed-zebra HEAD~1

When AI moves a function from one file to another, the standard diff shows it as deleted from file A and added to file B — two big blocks of change that look like new code. With --color-moved, moved code is colored differently from truly new code. You instantly see: “this block wasn’t written, it was relocated.”

--name-only — Just the Files

$ git diff --name-only HEAD~5..HEAD
src/api/auth.ts
src/api/users.ts
src/middleware/rate-limit.ts
src/utils/crypto.ts
src/utils/validation.ts
tests/auth.test.ts
package.json

When you need a quick inventory without the noise. Combine with --diff-filter=A to see only added files, or --diff-filter=M for only modified files.

The AI Refactor Diff Stack

When reviewing an AI-driven refactor, run these three commands in order:
git diff --stat — see the shape
git diff --color-moved=dimmed-zebra — separate moved code from new code
git diff --word-diff — see the actual changes within lines
This trio turns an intimidating 500-line refactor diff into something you can read in 10 minutes.

Asking AI to Explain Its Own Changes

Here’s a technique most developers underuse: ask the AI to explain what it just did. After an AI agent session, before you commit or push:

Good Prompt
“Explain the diff between HEAD and HEAD~5. For each commit: what changed, why, and what assumptions did you make about the existing codebase?”
Forces the AI to articulate its reasoning per commit. You can verify each assumption against your knowledge of the system.
Weak Prompt
“What did you change?”
Too vague. The AI will summarize at a high level and skip the details that matter for review.

This is especially valuable when you pick up a session someone else started, or when you’re reviewing commits from an AI agent that ran autonomously overnight. The AI’s explanation won’t be perfect — it can only explain from its context window — but it gives you a starting point that’s much faster than reading raw diffs.

Annotating Commits: The AI Provenance Trail

In 2026, knowing which code was AI-generated is no longer optional. When something breaks at 2 AM, the first question is: “Who wrote this?” If the answer is “an AI agent, three weeks ago, and nobody reviewed it closely,” you need to know that before you start debugging.

Commit Message Conventions

The simplest approach: use git trailers in your commit messages.

feat: add input validation for user email fields

Adds server-side validation for email format and domain
allowlist. Rejects disposable email providers.

Generated-with: Claude Code
Reviewed-by: J. Santos
Edge-cases-tested: empty string, unicode, 255+ chars

Trailers appear cleanly in git log output and are machine-parseable. You can filter for AI-generated commits:

$ git log --all --grep="Generated-with:" --oneline
a3f8c21 Add rate limiting middleware
7b2e459 Add auth endpoint tests
1d4c890 Implement token refresh logic

2026 Provenance Tools

The tooling has matured significantly. Several tools now provide line-level AI attribution:

🔎
Git AI / AI Blame
A drop-in replacement for git blame that shows AI attribution for each line. Uses Git Notes to track provenance. Stores transcripts locally or in the cloud. Run git ai blame src/auth.ts to see which lines were AI-generated and which were human-written.
📋
Agent Trace (Cursor open standard)
An open RFC specification for recording AI code attribution data. Supported agents report exactly which lines they wrote — no guessing. Attributions survive rebases, squashes, amends, merges, and cherry-picks.
🔗
Agent Blame (Mesa)
Adds line-level AI attribution by intercepting hooks from Claude Code, Cursor, and OpenCode. Normalizes data from different agents into a common format. Attributions stored in Git Notes and preserved through history rewrites.

These tools answer the question: “Was this line written by a human or an AI?” That distinction matters for debugging (AI code has different failure modes) and for compliance (some organizations require human review of all AI-generated code).

AI Commits That Touch Unrelated Files

One of the most common AI tells in git history: scope creep across commits. You asked the agent to add rate limiting. It added rate limiting — and also refactored the logging utility, renamed a variable in an unrelated controller, and added a helper function “for future use.”

$ git diff --stat HEAD~1
 src/middleware/rate-limit.ts  |  89 ++++++++++++++++++++
 src/utils/logger.ts           |  23 +++---    <-- Why?
 src/api/orders.ts             |   4 +-        <-- Why?
 src/utils/helpers.ts          |  15 ++++     <-- Why?
 4 files changed, 116 insertions(+), 15 deletions(-)

Only rate-limit.ts was requested. The other three files are scope creep. The AI improved things it noticed along the way — which sounds helpful until one of those “improvements” introduces a regression. When you see unrelated files in a commit, investigate each one. If it doesn’t belong, revert it or move it to a separate commit with its own review.

The Massive Unexplained Commit

The biggest red flag in git history: a single commit with hundreds of changed lines and a commit message like:

commit 4f8a2bc
Author: dev@company.com
Date:   Wed Apr 9 16:45:00 2026

    AI changes

No explanation of what changed. No breakdown of why. No AI disclosure. No test evidence. This commit is a black box. When it breaks — and at some point, something in those hundreds of lines will break — you’ll have to reverse-engineer the entire change from scratch.

The rule: If a commit has more than 100 changed lines and a commit message shorter than 10 words, it needs to be broken up and explained before merging. No exceptions for AI-generated code — especially not for AI-generated code.

Building a Mental Model from Git History

Git history isn’t just for debugging. It’s a tool for understanding. When you join a project or return to code you haven’t touched in months, git history is your map.

1. Overview
git log --oneline -30
What happened recently?
→
2. Focus
git log --oneline -- src/api/
What changed in this area?
→
3. Detail
git log -p -- src/api/auth.ts
How did this file evolve?
→
4. Context
git show <commit>
What else changed with this?

This is how experienced developers onboard onto unfamiliar code. In the AI era, it’s also how you maintain understanding of code that AI wrote while you were in a meeting. The Anthropic study found a 17% comprehension drop when developers delegated to AI without actively understanding the output. Git history is your safety net against that drift.

The Friday Afternoon Scan

Every Friday, spend 10 minutes running git log --oneline --since="5 days ago" on your main branch. Scan the commits. For any AI-generated commit you don’t recognize, run git show on it. This habit keeps your mental model current and catches comprehension debt before it compounds.

Interactive Exploration Commands

A few more commands for specific scenarios:

Find When a Function Was Introduced

$ git log -S "function refreshToken" --oneline
1d4c890 Implement token refresh logic

The -S flag (called “pickaxe”) finds the commit that added or removed a specific string. When you encounter an unfamiliar function in AI-generated code, this tells you exactly when and in what context it appeared.

See All Changes by a Specific Author

$ git log --author="Claude" --oneline --since="2 weeks ago"
a3f8c21 Add rate limiting middleware
7b2e459 Add auth endpoint tests
1d4c890 Implement token refresh logic
9e7f312 Add crypto utilities for JWT signing

Filter by author to see all AI-generated commits in a time range. Combine with --stat to see the scope of AI’s recent changes.

Compare Branches

$ git diff main...feature/auth --stat
$ git log main...feature/auth --oneline

The three-dot syntax shows what the feature branch added that main doesn’t have. Essential for understanding the full scope of an AI agent’s work before a merge.

Git Commands for Understanding AI Changes

  • git diff --stat — see the shape: which files changed, how much
  • git log --oneline --graph — see the story: sequence and structure of commits
  • git show <commit> — see the detail: full diff for one commit
  • git blame -L <start>,<end> <file> — see the origin: who wrote each line, when
  • git diff --word-diff — see word-level changes (renames, small refactors)
  • git diff --color-moved=dimmed-zebra — separate moved code from new code
  • git log -S "function_name" — find when a function was introduced
  • git log --author="Claude" — filter to AI-generated commits
  • git log --all --grep="Generated-with:" — find commits with AI provenance trailers
  • git diff main...HEAD --stat — see full scope of a feature branch

Reconstruct an AI Agent Session

Do this right now. It takes 10 minutes and teaches you more about git archaeology than reading ever will.

  1. Find your most recent AI coding session: git log --oneline -20
  2. Identify the range of commits the AI produced (look for patterns: same author, close timestamps, related messages)
  3. See the shape: git diff --stat <first-commit>^..<last-commit>
  4. Read the story: git log --oneline <first-commit>^..<last-commit>
  5. For each commit, ask yourself:
    • Does the message explain what and why?
    • Does it touch only files related to its stated purpose?
    • Is AI involvement disclosed?
  6. Pick the commit with the most changed lines: git show <commit>
  7. Can you explain what this commit does without asking the AI?

If you found scope creep, unexplained commits, or code you can’t explain — congratulations, you’ve just identified your comprehension debt.

Establish Your Team’s Git Conventions for AI Code

Create a shared agreement for how your team annotates AI-generated commits.

  1. Draft a commit message template: 
    <type>: <short description>

<what changed and why, 2-3 sentences>

Generated-with: <tool name>
Reviewed-by: <reviewer>
Edge-cases-tested: <list>
  2. Add the template to your project: git config commit.template .gitmessage
  3. Agree on scope rules:
    • One concern per commit — if the AI drifted, split the commit
    • No commit over 200 lines without a detailed message body
    • AI provenance trailers required on all AI-generated commits
  4. Share this with your team and enforce it for two weeks
  5. After two weeks, review: are commit messages improving? Is scope creep easier to spot?
