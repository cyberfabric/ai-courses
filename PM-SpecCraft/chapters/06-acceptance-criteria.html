Chapter 6

Acceptance Criteria & Edge Cases

The smallest section in the PRD is the one engineering will live inside for two sprints. Make every line testable, generate the edge cases you’d miss alone, and keep the list short enough to actually ship.

Where the spec gets tested for real

The PRD is open. Eight of the nine sections are filled. Goals names a number; Risks lists five with kill switches; Scope says what ships. The cursor sits under heading six: Acceptance Criteria. It’s the smallest section on the page. It’s also the one engineering will read forty times over the next two sprints, and the one QA will read every day until launch.

A sentence in the Goals section can be a little hand-wavy and survive review. A sentence in the AC section cannot. AC are the spec’s testable surface — the place where ambiguity stops being a writing problem and starts being a build problem. A criterion that reads beautifully but can’t be checked is a wish dressed up as a requirement. Engineers will find that sentence on day three and ask you what it means. You don’t want to be answering that question for the first time in a code review.

The good news: AC are also the section where AI helps the most, and where careful human judgment matters the most. AI can generate fifty edge cases in thirty seconds, in shapes you wouldn’t have thought of. You decide which five matter for this release, which five you defer, and which forty you politely thank the model for and ignore. That triage — not the generation — is the work.

What “good AC” looks like

Three properties make a criterion useful. They’re short enough to memorize, and you’ll come back to them every time you read a draft.

  • Testable. A person or a script can run a check and get an unambiguous yes-or-no. “Loads quickly” isn’t testable. “Renders within 500ms p95” is.
  • Observable. The signal lives somewhere visible — a UI state, a log line, an analytics event, a database row. If the only proof lives in someone’s head, it isn’t a criterion.
  • Unambiguous. Two reasonable engineers reading the criterion build the same thing. If the AC could plausibly justify three different implementations, it isn’t done yet.

Most vague AC fail two of the three at once. Here’s a typical first draft and the rewrite that turns it into something you can defend.

ACCEPTANCE CRITERIA — first draft • Search results should load fast and feel responsive. • Empty results should be handled gracefully. • The feature should work well on mobile. • Errors should be communicated clearly to the user.
ACCEPTANCE CRITERIA — rewritten • Search results render within 500ms p95 on the standard test corpus (1,000 items, 50 concurrent users). • A query with zero results renders the empty state from Figma frame F-12 within the same 500ms budget, with the suggested-searches block populated from the top-3 fallback list. • On viewports ≤414px wide, results stack single-column and the search input remains visible while scrolling (sticky header, tested on iOS Safari and Chrome Android). • A failed search request shows the error state from F-15, includes a retry control, and emits a `search.error` analytics event with the error code.

The before version is friendly and unobjectionable; the after version is sharp enough that an engineer can build it and a tester can check it. Notice what changed: numbers, references to specific design frames, named platforms, named events, named fallback content. Specifics are the proof that judgment was applied.

One thing to notice: the rewrite is longer. That’s fine. AC are not the place to optimize for word count. They’re the place to optimize for “if I came back to this in three weeks, would I know exactly what we agreed?”

Three formats, all valid

You’ll see AC written in three common shapes. None of them is universally right. The shape that fits depends on what kind of behavior you’re describing.

Given / When / Then. The most structured of the three. Each criterion names the precondition (Given), the action or trigger (When), and the observable outcome (Then). Strong fit for user-flow features — anything where there’s a sequence of states.

GIVEN / WHEN / THEN Given a logged-in user with no scheduled meetings, When they open the “Schedule a 1:1” panel, Then the panel shows the empty-state copy and the “Pick a time” CTA is focused by default.

Checklist style. A flat list of conditions that must all be true before the feature counts as done. Strong fit for features that are mostly about presence or correctness rather than sequence — settings panels, configuration screens, audit logs.

CHECKLIST • All four notification categories appear in the panel. • Each toggle defaults to ON for existing accounts. • Toggling persists immediately (no separate Save button). • Each toggle change emits a `notification.toggle` event.

Behavior list. Plain-language statements of what the feature does and doesn’t do, in present tense. Strong fit for small features where structure adds more weight than clarity, or for the high-level behavior summary at the top of a longer AC list.

BEHAVIOR LIST • The panel opens within the existing billing settings page; no new top-level navigation is added. • The owner sees all admins; admins see only themselves. • Settings sync across the owner’s active sessions in ≤5s.
You can mix formats inside one PRD. A common pattern: behavior list for the high-level summary, Given/When/Then for the user-flow happy path, checklist for the “and these things must also be true” tail. Mixing isn’t messy — it’s a way of letting each criterion choose the shape that makes it clearest. What’s messy is forcing twenty Given/When/Thens just because you started with one.

Sharpening vague AC into testable AC, in one loop

You don’t need a long workflow to turn vague AC into sharp ones. A single back-and-forth with an AI tool gets you most of the way there, as long as the prompt asks for the right two things: a rewrite, and a flag for anything the AI couldn’t make testable without more information.

The flag is the part most PMs skip. It’s also the part that earns the AI its keep on this task. A good model, asked “rewrite each AC as testable, and tell me which ones you can’t make testable without more from me,” will return both halves — and the second half is a free list of decisions you still need to make.

Here’s the AC section from a PRD draft. Rewrite each criterion as testable Given/When/Then. For any criterion you can’t make testable without more information from me, don’t guess — flag it and tell me what you’d need to know.

AC draft:
• Search results should load fast and feel responsive.
• Empty results should be handled gracefully.
• The feature should work well on mobile.
• Errors should be communicated clearly to the user.
Three rewritten, one flagged.

(1) Given a user on the search page, when they submit a query, then results render within [LATENCY BUDGET]ms p95 on the standard test corpus. Flag: I don’t know your latency budget. Common targets are 500ms or 1s p95. What’s yours?

(2) Given a query that returns zero results, when the response loads, then the empty state from your design system renders, with a suggested-searches block populated from your top-fallback rules. Flag: I don’t know your empty-state design or your fallback rule. Can you point me to the Figma frame and the fallback policy?

(3) Given a viewport ≤414px wide, when results render, then they stack single-column and the search input remains visible while scrolling (tested on iOS Safari and Chrome Android).

(4) Given a search request fails, when the failure response returns, then the error state renders with a retry control and emits an analytics error event. Flag: I don’t know your error-event schema or which states you treat as “failure” (timeout vs 5xx vs network).
Latency budget is 500ms p95. Empty state is Figma frame F-12; the fallback rule is “top three queries from the last 7 days, scoped to the user’s workspace”. The error event is `search.error` with fields {code, query, latency_ms}. Treat timeout and 5xx as failure; treat network drop as a separate state with its own copy.
Updated. (1) results render within 500ms p95 on the standard test corpus. (2) empty state renders from Figma F-12 with the suggested-searches block populated from top-3 queries in the last 7 days, workspace-scoped. (4) split into 4a (timeout/5xx error state with retry, emits `search.error` with {code, query, latency_ms}) and 4b (network-drop state with its own copy from F-16, emits `search.network_drop` event).
Good. Hold there. I’ll add the network-drop variant to the design ticket.

The pattern in that exchange is worth naming. The PM didn’t ask the AI to make the AC sharp and guess the missing details. They asked for sharpening with explicit flags for ambiguity. The model returned a rewrite plus three concrete questions. The PM answered the questions in one turn. The model produced clean AC. Total elapsed time: maybe four minutes.

Compare that to the alternative, where the model fills in plausible numbers and design references it invented. You’d get sharp-looking AC that quietly committed the team to a 1s budget you never agreed to and a fallback rule that doesn’t exist. The flag-don’t-guess instruction is the difference between the two outcomes. It’s a tiny prompt change with outsized effects. Use it every time.

Generating edge cases — the eight buckets

AC sharpening covers the happy path. Edge cases cover everything else. Most missed bugs in production aren’t exotic; they’re from one of eight predictable corners that PMs and AI tools alike forget when the focus is on the main flow. Asking explicitly for those eight corners catches most of them up front.

The eight buckets, in the order most worth asking for:

  1. Empty states. First-use, no data, zero results, recently deleted, recently archived.
  2. Error states. Server errors, validation errors, timeouts, partial failures, retry behavior.
  3. Slow / no network. Slow connections, offline, intermittent drops, request timeouts mid-flow.
  4. Accessibility. Keyboard-only navigation, screen-reader announcements, focus management, color contrast, motion-reduced preferences.
  5. Permissions. Different roles, different plan tiers, expired sessions, revoked access mid-session.
  6. Multi-language & RTL. Long translated strings overflowing UI, right-to-left layout, locale-specific number/date formatting.
  7. Time zones & DST. Cross-timezone scheduling, daylight-saving transitions, dates near midnight, leap days.
  8. Large data & pagination. Thousand-row lists, deep pagination, very long strings, very small or very large numbers.

Asking your AI tool to enumerate edge cases for each bucket, against your specific feature, takes one prompt. The output is usually 30–60 candidates. Most of them you’ll defer or ignore. A few will change your release plan. The point of the exercise is to surface the few without having to remember the eight categories from memory at 4 PM on a Friday.

Here’s the exercise on a small but real feature: a “Schedule a 1:1” flow inside a workspace product, where the person scheduling can pick a colleague and a time slot. The PM is using Copilot inside their editor, with the PRD draft open in the same workspace.

Feature: Schedule a 1:1. Two users (organizer and invitee), 30-minute slots, organizer picks an invitee from their workspace, picks a time, sends an invite. Invitee accepts or declines. Calendar event is created on accept.

Walk through the eight edge-case buckets — empty states, error states, slow/no network, accessibility, permissions, multi-language/RTL, time zones/DST, large data/pagination. For each bucket, give me 1–2 specific edge cases this feature will hit. Be concrete; no generic platitudes.
Twelve edge cases across eight buckets.

Empty states. (1) Organizer’s workspace has only one member — them. The invitee picker should show a meaningful empty state, not a blank dropdown.
Error states. (2) Calendar API fails after the invitee accepts. The invite was sent, the event wasn’t created — what state does the user see, and what’s the recovery? (3) The invitee was deactivated between sending and accepting. Decline by default? Show error?
Slow / no network. (4) Organizer hits “Send” on a flaky connection; request times out but invite was actually sent. Duplicate-send risk on retry.
Accessibility. (5) Time-slot picker keyboard navigation: can a screen-reader user announce the selected slot? (6) Color-only indication of “available vs busy” slots fails for color-blind users.
Permissions. (7) Guest accounts in the workspace — can they be scheduled with? Can they schedule? (8) Organizer’s plan was downgraded mid-session and they no longer have scheduling access.
Multi-language / RTL. (9) German translation of “Schedule a 1:1” is much longer; CTA button overflow risk.
Time zones / DST. (10) Organizer in PT, invitee in CET, meeting on a DST transition Sunday — what time does each side see? (11) Slot near midnight crosses a date boundary for the invitee.
Large data / pagination. (12) Workspace has 5,000 members; invitee picker performance and search behavior at scale.
Mark the four that ship as AC for this milestone. Defer the rest with a one-line reason each.
Suggested ship: (2) calendar API failure recovery — high impact, frequent enough to matter. (4) duplicate-send protection — data corruption risk, low cost to add. (10) cross-timezone DST display — correctness, will create support tickets if wrong. (12) invitee picker at scale — affects the largest customers, who are the loudest.

Defer: (1) one-member workspace, low frequency. (3) deactivated invitee, edge case worth a follow-up ticket. (5) screen-reader slot picker, accessibility audit ticket separately. (6) color-only indication, design fix in next sprint. (7) guest scheduling, separate spec. (8) mid-session downgrade, broader auth work. (9) German overflow, batch with the next localization pass. (11) midnight slot, low frequency, follow-up.
Mostly agree. Pull (5) into ship — we have a partner deal contingent on accessibility audits. Move (4) to defer — the back-end already has idempotency on send. Final ship: 2, 5, 10, 12.

Two things to notice. First, the AI did the bucket enumeration faster than any human would, and it caught the DST one and the German-overflow one without being prompted — both of which would have surfaced in production otherwise. Second, the PM made the final cut. The model’s suggested ship list was reasonable but didn’t know about the partner accessibility deal or the existing back-end idempotency. The PM’s context overrode the model’s defaults. That’s the loop working as designed.

Pruning — which edge cases ship in this release

An edge-case list with thirty entries is a planning hazard, not a planning aid. The list is only useful after it’s been pruned. The pruning is yours to do, and it’s the highest-leverage minute you’ll spend on the AC section.

A simple three-question filter handles most cases. Run each candidate through it, in order. The candidates that survive all three become AC; the rest become deferred tickets, follow-ups, or quietly ignored.

  • How often does this happen? A bug that hits 0.1% of users once a week is different from a bug that hits 30% of users every session. Frequency × reach is a rough first cut.
  • How bad is it when it does? Data loss and revenue impact go to the top. Cosmetic drift and minor copy issues go to the bottom. A confused user who recovers in five seconds is different from one who never finds the feature again.
  • How cheap is it to handle in this release? A two-line fix during the same sprint is almost always worth doing. A two-week fix that pushes the launch is almost never worth it for a low-frequency edge case.

The candidates that score “often, bad, cheap” are no-brainers — ship as AC. The ones that score “rare, mild, expensive” are easy defers — one-line follow-up tickets. The interesting middle is where judgment lives. A rare-but-catastrophic edge case (data corruption affecting 0.1% of users) probably ships even if expensive. A common-but-mild one (slight layout drift on a niche browser) probably defers even if cheap, just to keep the AC list short.

Worth naming what “cheap to handle” really means, because it’s the question PMs most commonly get wrong. Cheap isn’t about lines of code. Cheap is about what the change touches: code paths, design, copy, analytics, QA, rollout. A two-line back-end change that requires a new design, new copy review, a new analytics event, and a new QA pass isn’t cheap — it’s expensive in disguise. A larger code change that lives entirely inside one already-tested module, with no design or copy change, is genuinely cheap. The right person to ask is the engineer working on the spec; their answer to “what does this touch?” is the cost estimate that matters.

Yours to decide: the AI can list edge cases forever and rank them with reasonable defaults. Only you know which ones change the release plan, which contractual obligations are in play, and which deferred items will come back to bite the team three months from now. Every AC you accept is a check the team has to pass before shipping; every AC you defer is a ticket someone has to remember. Both have costs. The triage is the work, and the work is yours.

Definition of Done

AC describe what this feature does. Definition of Done describes what every feature does, regardless of which one. It’s the team’s standing bar — the things that must be true before any spec ships, not just this one.

The DoD usually covers a small, predictable set of categories. They don’t belong in every PRD as separate AC, because they’d clutter every spec with the same six bullets. They belong as a separate, agreed-once document the whole team references.

DEFINITION OF DONE — team standing bar (rev 2026-Q1) • All AC pass in QA on staging. • Accessibility check: keyboard navigation works, color contrast meets WCAG AA, screen-reader pass on the new flow. • Performance budget: no regression beyond +10% on existing p95 for the affected endpoints. • Observability: new flows emit the analytics events listed in the spec; error logs include user/session context. • Copy reviewed by content team for any user-facing strings. • Feature flag in place; rollout plan approved. • Rollback plan documented and tested in staging.

The PM’s role with the DoD isn’t to negotiate it per ticket — that path leads to ten different DoDs across ten specs and a team that quietly stops trusting any of them. The PM’s role is to agree the DoD with engineering and design once a quarter, write it down somewhere everyone can find it, and reference it from every PRD with a single line: “Standard team DoD applies; see [link].”

If a particular feature has a DoD requirement above the standard bar — say, a partner integration needs a security review that other features don’t — that goes in the PRD as a one-off addition: “In addition to standard DoD: security review by partner-eng before launch.” The standing bar stays standing. The exceptions are visible.

Two failure modes to recognize. One: the DoD has thirty bullets and the team has stopped reading it. The cure is to cut to the seven that actually block ship. Two: the DoD has two bullets and the team is rebuilding it from scratch in every PRD. The cure is to take a quiet hour with engineering and write the missing five together.

Smell tests for AC

Two quick checks you can run on any AC list before sharing it with engineering. Both take less than a minute. Both will catch most of the issues that would otherwise surface on day three of the sprint.

The QA-sentence test. For each AC, ask: “Could I describe in one sentence how QA would test this?” If the answer is “not really”, or if it requires multiple sentences with words like “basically” or “ideally”, the AC isn’t testable yet. Push it back into the sharpening loop. AC that pass the QA-sentence test usually have specifics, observable signals, and a clear pass/fail. AC that fail are still wishes.
The uniform-confidence test. Re-read your AC list. Do they all sound about equally confident? If yes, that’s often a signal that the easy ones got the same treatment as the hard ones, and the hard ones are hiding edge cases. The hard AC should feel slightly harder to write than the easy AC — more conditions, more specifics, more flagged ambiguity. Uniform polish is uniform tone, not uniform rigor. Trust the discomfort; let the hard AC look harder.

Both tests are encouraging in tone, not punishing. They’re checks you run on yourself, not weapons you wave at others’ specs. The point is to catch your own draft before it travels into a review where catching it is more expensive.

Common patterns to watch for

A few patterns show up often enough that naming them helps you spot them in your own drafts. None of these are mistakes; they’re just shapes that AC tend to fall into when you’re moving fast. The high-leverage move is to notice the shape and reach for a sharper alternative.

Untestable verbs. “Intuitive”, “clear”, “fast”, “robust”, “seamless”. These verbs feel meaningful and aren’t. The fix: replace each with an observable. “Intuitive” becomes “a new user completes the flow without opening the help menu, in user testing with five participants”. “Fast” becomes “loads within 500ms p95”. The replacements are usually longer and always sharper. If you want a fast scan for these, ask Windsurf or ChatGPT: “find every untestable verb in this AC list and propose a specific behavior it could be replaced with.”

Happy-path-only AC. The list reads like a description of the feature working perfectly. No errors, no empty states, no edge cases. This is the most common pattern, because the happy path is what you’re excited to build. The fix: run the eight-bucket prompt explicitly before declaring the AC list done. If you wrote the AC without thinking about errors, the AI will catch it for you in one prompt.

AC that smuggle in solution choices. “The user sees a modal” sounds like an AC, but it commits the implementation to a modal — which the spec might not have decided. If the AC is “the user gets a clear confirmation before charges occur”, that’s testable behavior; the modal-vs-inline question stays open for design. Read each AC and ask whether you accidentally chose an implementation when you meant to specify a behavior. Behaviors travel; specific implementations lock you in.

AC that grow in review. The first draft has six AC. After three rounds of review, there are nineteen. Most of the new ones are good, but some are scope expansion that nobody explicitly approved. The fix: at each round, add a one-line note next to the new AC saying who asked for it and why. If a new AC arrives without that note, ask “is this in scope for this milestone?” explicitly. Sometimes the answer is yes; sometimes the answer is “ah, you’re right, that’s a follow-up.”

The single mega-AC. One AC bullet that’s six lines long and contains four conditions joined by “and”. Almost always, this is four AC pretending to be one. Split it. Each AC should be checkable independently; a compound AC means QA has to invent which sub-conditions to test in which order, which they’ll do differently than you expected.

Naming these patterns isn’t about catching anyone out. It’s about giving yourself a vocabulary for the small shapes your drafts will fall into when you’re tired or in a rush. The vocabulary makes the drafts easier to fix.

Writing AC live with one engineer

The AC section is the single highest-leverage place to invite an engineer into the drafting process. Not at review time, when changes are expensive — at drafting time, when changes are cheap. A 30-minute live session with one engineer and your AI tool open will produce sharper AC than three days of solo iteration, and it will save the much more painful conversation that happens when AC turn out to be uncheckable in the team’s actual stack.

The pattern that works: open a shared screen with the PRD draft and your AI tool side by side. The PM types the first-draft AC. The engineer reads each one and reacts. Where the engineer says “we can’t observe that the way you’ve written it”, the AC gets rewritten on the spot — sometimes by the PM, sometimes by asking the AI to propose three observable alternatives and picking the one the engineer can implement cheaply. Where the engineer says “there’s a tricky edge case here that you don’t know about yet”, the AC list grows by one. Both moves take seconds. Neither would happen reliably in async review.

Two specific moments are worth naming because they tend to recur. First, the engineer flags a measurement gap: “we don’t emit the analytics event you want to assert on; adding it is a separate ticket.” That’s a fork in the road — either the AC changes shape (assert on a different signal that already exists), or the spec gains a small “new analytics event required” line that becomes part of the build. Either is fine; what matters is that the choice gets made now, not at QA time.

Second, the engineer surfaces a feasibility cost the spec didn’t know about: “this AC implies we recompute permissions on every render, which we do not currently do.” That’s the moment to either rewrite the AC to fit the existing model or to accept the cost explicitly — ideally with the AI summarizing the trade-off in one paragraph that becomes a Risks-section line. Don’t let either path happen silently. Silent acceptance of a hidden cost is how sprints expand by 40% mid-flight.

The thing to watch in a paired session: the AI is not the third voice in the conversation. The AI is a typist and a sharpener. If you and the engineer disagree about whether an edge case ships in this milestone, asking the model to break the tie produces a reasonable-sounding compromise that satisfies neither of you and ages badly. Have the disagreement directly. The PM owns the call. The engineer owns the cost estimate. The AI helps express what you both already decided.

How long should the AC list be?

A useful rule of thumb: the AC list for a sprint-sized feature has roughly five to twelve criteria. Below five, you’re probably missing edge cases. Above twelve, you’re probably either describing too big a feature for one PRD or adding things that belong in DoD or in a follow-up ticket.

If your list is at twenty-five and growing, a few moves help. Pull DoD-flavored items (“analytics events fire correctly”, “accessibility audit passes”) out into the standing DoD reference. Bundle small variations of the same behavior into a single AC with a sub-list. Defer rare edge cases to follow-up tickets with explicit one-line reasoning. The goal isn’t a small list for its own sake; it’s a list that engineering and QA can hold in their heads at the same time.

If your list is at three, a few moves also help. Run the eight-bucket prompt and see if the model surfaces edge cases you missed. Re-read the discovery doc — what did the user pain point imply about failure modes? Walk the happy path mentally and ask “and what if this step doesn’t happen as expected?” at every transition. The AC list is usually richer than the first draft suggests; the second draft just has to hunt for the missing ones.

AC and the spec’s change history

AC drift more than any other section in a PRD. They get added in review, sharpened in pair sessions, deferred when a sprint runs hot, occasionally resurrected when a partner deal shifts the priorities. The drift is healthy — it’s the spec responding to reality — but only if the changes are visible. AC that quietly appear or disappear between v0.4 and v0.5 are a trust hazard, even when each individual change was the right call.

The cheap habit that handles this: every time the AC list changes, add one line to the spec’s decision history at the top. “v0.6: added AC 7 (admin-on-other-workspace) after eng review.” “v0.7: deferred AC 4 (large-data pagination) to follow-up ticket; back-end says it’s a separate refactor.” “v0.8: replaced AC 2 with two narrower AC after partner accessibility note.” The lines take ten seconds to write. They make the AC list legible six weeks from now, when nobody on the team can remember why number 7 was added or why number 4 isn’t there.

The AI tools help with this too, in a small, concrete way. After a pair session that produced six AC changes, ask Claude or ChatGPT: “summarize the diff between this AC list and the previous one in three bullet points suitable for the decision history.” You’ll get tidy lines back that you can paste straight in. It’s a sixty-second move that pays for itself the first time someone asks “wait, when did we add this one?”

AC for fuzzy outcomes

Some features don’t have a binary outcome. A search ranking change doesn’t pass or fail one test — it shifts a distribution. A recommendation panel doesn’t have a single right answer for any given user. An AI-powered summary doesn’t have one correct output. The AC mindset still applies, but the shape of the criteria has to change to match the shape of the outcome.

The move that works: instead of asserting a single state, assert on aggregate behavior, on bounds, and on negative cases. “The recommendation panel surfaces an item the user clicks within the first three slots in ≥30% of sessions” is testable in aggregate. “The summary is never longer than four sentences, and never references information not present in the source document” is testable per output. “The ranking change does not regress the top-decile query latency by more than 5%” is testable as a guardrail. The trick is to translate “the feature works” into “these aggregate signals stay within these bounds, and these specific failure shapes do not occur.”

Asking the AI to help is straightforward: paste the feature description, name the fuzzy outcome, and ask “rewrite as AC that test on aggregate signals and explicit guardrails, not on single-state pass/fail.” The model is good at this translation; it’s seen many ML spec patterns. Where it tends to slip is on the negative cases — the “and these failure shapes must not occur” half. That half is the part you, as the PM, are best positioned to write, because you know which failure shapes would be embarrassing to your specific customer base. Examples: the summary fabricating a number that wasn’t in the source. The recommendation surfacing a competitor’s product. The ranking putting a deprecated item in the top slot. Engineering doesn’t always know which ones matter; you do.

Fuzzy-outcome AC are also where the line between AC and Metrics blurs. A useful rule of thumb: the AC version is the floor (“must be at least this”), and the Metrics version is the goal (“we’re aiming for this”). Both can reference the same number, with different framing. AC at 30% click-through-on-top-three is a release gate; Metrics at 45% is the success target. The release passes the gate, then we work toward the target. Treating the two separately keeps the launch crisp and the success criteria honest.

Try it this week

The Edge-Case Bucket Drill. Pick a current ticket from your backlog — ideally one that’s about to enter sprint planning. The smaller and more concrete the better.

  1. Write down the feature in one sentence: who does what, in what state, with what outcome.
  2. In your AI tool of choice (Claude, ChatGPT, Gemini, Copilot, Windsurf — whichever you reach for first), paste the one-sentence description and ask: “Walk through these eight edge-case buckets — empty states, error states, slow/no network, accessibility, permissions, multi-language/RTL, time zones/DST, large data/pagination — and give me 1–2 specific edge cases this feature will hit per bucket. Be concrete; no generic platitudes.”
  3. Read the list. Most likely it’s 10–16 items.
  4. Mark each item with one of three labels: SHIP (becomes AC for this milestone), DEFER (becomes a follow-up ticket), IGNORE (out of scope for this product or this team). Run the three-question filter from earlier in this chapter on each.
  5. Circle the three items in your SHIP set that most change the release plan — new design work, new copy, new back-end behavior, new test fixtures. Take those three to your next sprint planning conversation.

Bonus credit: do the same exercise with a different AI tool and compare the lists. The eight buckets are the same; the model’s default emphasis differs. You’ll often find that one model leans on accessibility, another on permissions, another on data scale — useful to know which tool to reach for when you want a specific lens.

What you should end up with

By the end of an AC pass — sharpening, edge-case generation, pruning, DoD reference — you should have a section that fits on one screen, contains five to twelve testable criteria, references a Figma frame or two for any visual states, names the analytics events you expect to see, and is clear enough that a stranger could write a test plan from it without asking you a follow-up question.

You should also have a small list of deferred items, each with a one-line reason. That list isn’t a regret; it’s a sign you triaged. Specs without deferred items are usually specs where the triage didn’t happen, which means the engineering team will do it implicitly when they hit each edge case in code — and they’ll do it without your context.

And you should have a quiet sense that any sentence in the AC section could be picked up, handed to QA cold, and turned into a test that passes or fails. If a sentence triggers the “but how would we test that?” question in your own head, fix it now. Future you will thank present you when the sprint starts.

One last small habit worth carrying forward: read the AC section out loud once before you share the spec. Reading aloud catches the soft sentences faster than reading silently does — the ear notices vagueness the eye lets slide. It takes ninety seconds. It saves hours.
