Chapter 11

Your Personal AI PM Kit

Memory Map
🧰

The Big Idea

Spec one took a week. Spec two doesn’t have to. The kit closes the gap between “the craft I figured out” and “the craft I have ready next time.” Four parts — templates, context packs, a prompt library, naming conventions — built once, refined monthly, kept in plain markdown so it travels across whichever AI tool you happen to use this quarter. The kit doesn’t make the assistant smarter; it makes the starting position better.

🧱 The Four Parts of the Kit

📐
Templates
Reusable document skeletons — PRD outline, AC checklist, edge-case prompt, risk register, retro one-pager. Structures you fill in, not prose you copy.
📦
Context packs
A one-page brief about your product, pasted in before real work. The assistant’s onboarding doc. Largest payoff for the least effort.
🎯
Prompt library
10–15 named prompts, each with a when-to-use, an input shape, and the prompt itself. The shelf where you actually browse.
🗂️
Naming & storage
Where the kit lives, how files are named, how versions are tracked. Boring on the surface; load-bearing in practice.

All four parts together fit in fewer pages than a single PRD. The discipline is to keep them small. The temptation is to grow each part until it’s too heavy to maintain — and a heavy kit is one you stop opening on Monday morning.

📦 The Context Pack — Five Sections, One Page

🏢 What we are
product, customer, team, stage
👥 Who uses it
2–4 segments, one line each
📦 Context pack
(one page)
📖 Glossary
10–20 product-specific terms
🪧 Prior decisions
5–10 “X over Y because Z”

Plus a fifth section most people skip and that pays back the most: “What we don’t do.” Five to ten things explicitly out of scope as a product. Without it, the assistant cheerfully suggests features you’ve already ruled out. With it, the assistant knows the boundaries of your product and stays inside them. Refresh quarterly; archive the old one for two months.

🎯 The Prompt Library — Same Shape, Every Entry

The three-part prompt template When to use: One sentence. Future-you scans the library by use case, not by prompt text. Input shape: What to paste under the prompt. Forces you to gather the right material before pasting — half of what makes a prompt produce a good answer. The prompt: Short, specific, ends with one explicit constraint. (e.g. “push back if any of them is the same statement in different words.”)

Starter library, ten prompts, all in this shape: sharpen-problem-statement, draft-prd-section, generate-edge-cases, classify-comments, propose-diff-edits, translate-for-audience, run-smell-tests, draft-handoff-summary, summarize-decision-log, prune-stale-questions. Soft cap of 15. If you want a 16th, retire one first.

🔌 Tool-Agnostic Portability — Three Attachment Shapes

📋 Paste-into-chat 📌 Project files / system prompts 📁 Repo-attached docs
Works in every chat assistant: Claude, ChatGPT, Gemini, Copilot Chat Pin a context document so it’s always available without re-pasting Tools like Windsurf and Copilot in IDE mode read files from a repo
Three blocks in order: context pack → prompt template → input Claude Projects, ChatGPT Custom GPTs, Gemini Gems Reference context pack and prompt files directly; assistant pulls on demand
Verbose but universal. The lowest common denominator that always works Heavy lifting pre-attached; prompt library still pastes per session Suits PMs already working in a code-adjacent environment

The kit’s content doesn’t change — only the wrapper does. A context pack written for paste-into-chat works just as well as a Claude Project file or a repo-attached markdown doc. You don’t maintain three versions; you maintain one and adapt how you load it. Plain markdown is the design constraint, because the AI tool you use this quarter is probably not the one you’ll use next quarter.

PMs who run it consistently hit v1.0 by the second quarter and stay there.

📈 The Maturity Arc: v0.1 → v1.0

  1. 🌱 v0.1 (week one)

    Context pack mostly right, five prompts saved, three templates. Index file bare. Half your sessions still feel like freelancing because you forget the kit is there. The kit is helping, but you’re not yet leaning on it.

  2. 🌿 v0.3 (month two)

    Library up to ten entries. Glossary matches your team’s slang. Two prompts archived, three new ones added from recent specs. Reaching for the kit starts to feel automatic. Sessions ramp faster.

  3. 🌳 v0.7 (month four)

    Library at the soft cap of fifteen. Context pack refreshed once. Switched between two AI tools without rebuilding anything — the kit is plain markdown. Monthly review takes twenty minutes. New specs feel substantially easier than six months ago.

  4. 🌲 v1.0 (month six and beyond)

    The kit is steady. Add a prompt every month or two; retire one every month or two; refresh the context pack quarterly. Monthly review is mostly “yep, still good.” Doing maybe a quarter of the typing on every spec; the spec still reads as yours. Colleagues start asking about it.

⏱️ The 3-Hour Build — Kit v0.1 in an Afternoon

The afternoon plan Hour 1 — Context pack (60 min) Fresh markdown file. Five sections from memory: what we are / who uses it / glossary / prior decisions / what we don’t do. Don’t research. Have Claude or ChatGPT flag thin spots. Hour 2 — Prompt library (60 min) Pull last 3–4 spec sessions. Pick the 5–6 prompts you used most heavily. Copy each into a file with the three-part shape. Don’t over-edit; what worked already worked. Hour 3 — Templates & index (60 min) Pull PRD outline, AC checklist, risk register from your last shipped spec. Strip spec-specific content. Save as templates. Write a one-page index.md listing every file. The first version doesn’t need polish; it needs to exist.

👃 The Smell Test for Kit Health

The tell. You open the kit on Monday morning, scroll through the prompt library, and notice a prompt you’ve never used — you don’t even remember writing it. Or you spot two templates that do the same thing in slightly different words. Or the context pack still mentions a customer segment you sunset two quarters ago.

The check. Ask of each item: have I used this in the last 30 days? Would I describe it the same way today? Does it duplicate something already in the kit? If the answer to any of those is no, the item is owing you a decision.

The fix. Archive aggressively. A small kit you use is worth ten times a big kit you don’t. Move unused items to archive/; trust that anything you genuinely need will come back in the next monthly review.

🎚️ Where the Kit Shines — and Where It Doesn’t

✨ The kit shines on the predictable parts 🚫 The kit can’t help with judgment
Sharpening a fuzzy brief Whether the problem is worth solving
Drafting a PRD section Which user matters this quarter
Generating edge cases When to push back on a senior reviewer
Classifying review comments When to declare the spec frozen
Translating for a different audience None of these belong in a template

Mechanical work in the kit; judgment work outside it. Same split this whole course has been about. AI accelerates the mechanical layer; you stay in the driver’s seat for the judgment layer. The kit just makes the acceleration repeatable.

What’s yours: this kit, not the team’s standard. The kit is personal. It reflects your taste, your voice, your judgment about which prompts produce output you can actually use. It is not the team’s style guide, not a standard your colleagues should adopt, not something to push into a shared template library before you’ve lived with it for three months. Right now, the work is to build the kit that makes your spec work faster and more consistent, in your voice. The kit you maintain for yourself is always sharper than a kit you maintain for an audience.

❓ Quick Quiz

What are the four parts of the kit?

Templates (reusable document skeletons), context packs (one-page product briefs), prompt library (10–15 named prompts), and naming conventions and storage (where the kit lives, how files are named, how versions are tracked). Together they fit in fewer pages than a single PRD. The discipline is to keep them small.

What are the five sections of a context pack?

What we are (product, customer, team, stage), who uses it (2–4 segments, one line each), glossary (10–20 product-specific terms), prior decisions (5–10 “X over Y because Z” entries), and what we don’t do (5–10 things explicitly out of scope). The fifth section is the one most people skip and the one that pays back the most.

Why does each prompt in the library use the same three-part shape?

Uniformity is what makes the library scannable. When-to-use saves time later because you scan by use case, not by prompt text. Input shape forces you to gather the right material before pasting (half of what makes a prompt produce a good answer). The prompt itself is short, specific, and ends with one explicit constraint. After a few months, you’ve internalised the shape so well that drafting a new entry takes about five minutes.

What are the three attachment shapes for tool portability?

Paste-into-chat (verbose but universal — works in every chat assistant: Claude, ChatGPT, Gemini, Copilot Chat), project files / system prompts (Claude Projects, Custom GPTs, Gemini Gems — pin a context document so it’s always available without re-pasting), and repo-attached docs (Windsurf and Copilot in IDE mode read from a repo). Same content, different wrapper. You don’t maintain three versions.

What does the 30-minute monthly review do?

Three passes: prune (any prompt or template unused in 30 days → archive/ subfolder, not delete), add (any prompt you typed twice from memory becomes a library entry; any context fact you re-explained becomes a context-pack line), refresh (scan the context pack for staleness — sunset segments, overturned decisions, team changes). Thirty minutes, three passes, once a month. What separates the v0.3 plateau from the v1.0 steady state.

What does kit v1.0 look like?

Steady state, around month six. You add a new prompt every month or two; retire one every month or two; refresh the context pack quarterly. The monthly review is mostly “yep, still good.” The kit is doing maybe a quarter of the typing on every spec, the spec still reads as yours, and you can’t imagine working without it. Your colleagues have started asking about it.

What are the two main anti-patterns?

The bloated kit: 47 saved prompts, you remember 8, the rest is dead weight that slows every scan. Cure: the monthly prune. Prevention: soft cap of 15 — want a 16th, retire one first. The brittle kit: prompts assume Claude’s tone; context pack uses ChatGPT syntax; templates rely on Gemini quirks. When the tool changes, the kit needs a full rewrite. Cure: plain markdown, generic prompts. Test: paste a prompt into a tool you don’t usually use; if quality drops sharply, the prompt is too brittle.

How should you share the kit when colleagues ask?

Two paths work, one doesn’t. Works: send a snapshot framed as “fork it, prune ruthlessly, adapt to your voice” — they keep a third, rewrite the rest, no maintenance burden for you. Also works: extract two or three generic templates (PRD outline, AC checklist) and contribute them to a shared team folder. Doesn’t work: declaring your personal kit the team standard. The moment a kit has multiple owners with different tastes, it converges to a lowest-common-denominator that helps nobody.

What’s the portability test for adding something to the kit?

If a thing you’re considering adding only makes sense for one specific spec, it doesn’t belong in the kit. If it’d be useful on the next three specs you’ll write, it does. Templates pass by definition. Context packs pass because the product context is the same across specs. Prompts pass when they’re generic enough to work on multiple problem shapes. What fails: clever one-off prompts that worked great on one weird spec but won’t come up again. Capture them in the spec’s notes; don’t bury them in the kit.

Where does the kit shine, and where doesn’t it help?

Shines on the predictable parts: sharpening a fuzzy brief, drafting a PRD section, generating edge cases, classifying review comments, translating a paragraph for a different audience. Doesn’t help with judgment: whether the problem is worth solving, which user matters this quarter, when to push back on a senior reviewer, when to declare the spec frozen. Mechanical work in the kit; judgment work outside it. The kit speeds up the mechanical layer so you have more attention for the layer where attention is the only thing that matters.
