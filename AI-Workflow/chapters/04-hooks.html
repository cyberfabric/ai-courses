Chapter 4

Hooks That Work For You

Event-driven automation that fires without you lifting a finger

From On-Demand to Always-On

In Chapter 3, you built skills — powerful /commands that save time every time you invoke them. But there’s a catch: you still have to remember to invoke them.

Hooks eliminate that last manual step. They’re event listeners for Claude Code. When Claude Code edits a file, a hook can auto-lint it. When it writes Kotlin, a hook can auto-format it. When it runs a dangerous command, a hook can block it.

Think of hooks like a home automation system. You don’t flip the porch light on every evening — the motion sensor does it for you. Hooks are the motion sensors of your dev workflow.

How Hooks Work

A hook is a shell command that runs when a specific Claude Code event fires. Hooks live in settings.json — user-level or project-level. Each hook receives a JSON payload on stdin describing the event (tool name, file path, command, cwd, session ID, and so on), and your script uses jq to grab the fields it cares about.

Claude Code Event
→
Matcher Checks
→
Hook Runs
→
Result Fed Back

The basic structure in settings.json:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/your-script.sh"
          }
        ]
      }
    ]
  }
}

The schema: an event (e.g. PostToolUse), an optional matcher, and one or more handlers under the inner hooks array. Each handler has a type (almost always command) and a command string. $CLAUDE_PROJECT_DIR points at your project root — always quote it so paths with spaces work.

The JSON your script sees on stdin includes hook_event_name, tool_name, cwd, session_id, and a tool_input object. For an Edit, that’s tool_input.file_path, old_string, new_string; for Bash it’s tool_input.command. So when you see jq -r '.tool_input.file_path' later in the chapter, that’s how a script grabs the file Claude just edited.

Hook Types: When Things Fire

Claude Code fires hooks at many points in a session. Here are the ones you’ll reach for most often:

Hook Type Fires When Can Block? Best For
PreToolUse Before a tool is used Yes Guardrails, validation, blocking dangerous actions
PostToolUse After a tool is used No Linting, formatting, testing, logging
Notification Claude Code sends a notification No Alerts to Slack, sounds, desktop notifications
Stop Claude Code finishes its turn No Session summaries, final checks, logging
SubagentStop A subagent finishes No Collecting results from parallel agents
UserPromptSubmit You submit a prompt, before Claude reads it Yes Prompt rewriting, guardrails, injecting context
SessionStart A new or resumed session begins No Setting env vars, loading project state, warm-up checks
SessionEnd The session terminates No Final logs, cleanup, off-boarding metrics
PreCompact Before Claude compacts the conversation Yes Persisting context to disk before it’s summarized

Even more events exist (UserPromptExpansion, PostCompact, SubagentStart, PostToolUseFailure, CwdChanged, and others) — see the hooks reference for the full list. Most workflows only need the rows above.

PreToolUse is special. It’s the headline hook for blocking actions. Traditional way: exit 2 and the action is blocked, with stderr shown to Claude. Modern way: exit 0 and print a JSON object with {"hookSpecificOutput": {"hookEventName": "PreToolUse", "permissionDecision": "deny", "permissionDecisionReason": "..."}}. JSON is preferred — it also supports "allow", "ask", and rewriting the tool input via updatedInput. Pick one approach per script; don’t mix.

Matchers: Targeting Specific Tools

Without a matcher, a hook fires on every tool use. That’s rarely what you want. Matchers let you target specific tools:

{ "matcher": "Edit",       "hooks": [...] }   // exact tool
{ "matcher": "Edit|Write", "hooks": [...] }   // pipe = OR
{ "matcher": "Bash",       "hooks": [...] }   // any Bash command
{ "matcher": "*",          "hooks": [...] }   // every tool

The matcher checks the tool name. Most common: Edit (modify a file), Write (create a new file), Bash (run a shell command), Read (read a file). It also accepts regex (^Notebook) and MCP-style globs (mcp__memory__.*).

Your First Hook: Lint on Save

Let’s build the most universally useful hook: auto-linting every file Claude Code touches.

Setting Up Auto-Lint with ktlint

  1. Create the lint script. Save this as scripts/lint-changed.sh in your project:
#!/bin/bash
# scripts/lint-changed.sh — Lint a file if it's Kotlin
FILE=$(jq -r '.tool_input.file_path')
if [[ "$FILE" == *.kt || "$FILE" == *.kts ]]; then
  ktlint --format "$FILE" 2>&1
  if [ $? -ne 0 ]; then
    echo "ktlint found issues in $FILE"
  fi
fi
  1. Make it executable: chmod +x scripts/lint-changed.sh
  2. Add the hook. In .claude/settings.json:
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/lint-changed.sh"
          }
        ]
      }
    ]
  }
}
  1. Test it. Ask Claude Code to edit a Kotlin file. After the edit, the hook runs ktlint automatically. If there are issues, Claude Code sees them and can fix them immediately.

Without the Lint Hook

  1. Claude Code edits UserViewModel.kt
  2. You remember to run ktlint (or forget)
  3. Linter finds 3 style issues
  4. You ask Claude Code to fix them
  5. You run ktlint again to verify
  6. Repeat for the next file

3–5 minutes per file, easy to skip

With the Lint Hook

  1. Claude Code edits UserViewModel.kt
  2. Hook auto-lints — issues fixed inline
  3. You never think about linting again

0 seconds of your time, never skipped

Format on Save

Linting catches issues. Formatting fixes style. If your team uses Spotless (common in Android projects), you can auto-format every file Claude Code touches:

// In .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/format-changed.sh" }
        ]
      }
    ]
  }
}
#!/bin/bash
# scripts/format-changed.sh — Format a file if it's Kotlin or Java
FILE=$(jq -r '.tool_input.file_path')
if [[ "$FILE" == *.kt || "$FILE" == *.java ]]; then
  # Use spotless to format just this file
  ./gradlew spotlessApply -PspotlessFiles="$FILE" --quiet 2>&1
fi

Combine lint and format into one hook. Rather than having two separate PostToolUse hooks, create a single scripts/post-edit.sh that runs format first, then lint. Order matters — format fixes style, then lint catches logic issues.

Test on Change

This one is transformative. Every time Claude Code modifies a source file, automatically find and run the relevant tests:

#!/bin/bash
# scripts/test-changed.sh — Run tests related to a changed file
FILE=$(jq -r '.tool_input.file_path')

# Only for Kotlin source files (not test files themselves)
if [[ "$FILE" == *.kt && "$FILE" != *Test.kt && "$FILE" != *test* ]]; then
  # Derive the test file path
  TEST_FILE="${FILE/main/test}"
  TEST_FILE="${TEST_FILE%.kt}Test.kt"

  if [ -f "$TEST_FILE" ]; then
    # Extract the test class name for Gradle
    CLASS_NAME=$(basename "$TEST_FILE" .kt)
    echo "Running tests: $CLASS_NAME"
    ./gradlew test --tests "*$CLASS_NAME" --quiet 2>&1
  else
    echo "No test file found for $FILE"
  fi
fi
// In .claude/settings.json — add to existing PostToolUse array
{
  "matcher": "Edit|Write",
  "hooks": [
    { "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/test-changed.sh" }
  ]
}

The test-on-change hook creates a tight feedback loop. Claude Code edits a file → tests run instantly → if they fail, Claude Code sees the failure and can fix it immediately, in the same session. No more “I’ll run tests later” — later is right now, automatically.

Safety Guardrails: PreToolUse Hooks

PostToolUse hooks react to what happened. PreToolUse hooks prevent things from happening. This is your safety net.

Block force-push to main

#!/bin/bash
# scripts/guard-main.sh — Prevent dangerous operations on main/master
# Uses the JSON-decision pattern: exit 0 with permissionDecision=deny.
COMMAND=$(jq -r '.tool_input.command')
deny() {
  jq -n --arg reason "$1" '{hookSpecificOutput:{hookEventName:"PreToolUse",permissionDecision:"deny",permissionDecisionReason:$reason}}'
  exit 0
}

if echo "$COMMAND" | grep -qE 'git\s+push\s+.*--force.*(main|master)'; then
  deny "Force push to main/master is not allowed. Use a feature branch and open a PR."
fi

BRANCH=$(git branch --show-current 2>/dev/null)
if [[ ("$BRANCH" == "main" || "$BRANCH" == "master") ]] && echo "$COMMAND" | grep -qE 'git\s+commit'; then
  deny "Direct commits to $BRANCH are not allowed. Create a feature branch first."
fi
// In .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/guard-main.sh" }
        ]
      }
    ]
  }
}

Now Claude Code physically cannot force-push to main or commit directly to it. The hook blocks the action before it happens and explains why.

Enforce conventional commits

#!/bin/bash
# scripts/check-commit-msg.sh — Enforce conventional commits format
COMMAND=$(jq -r '.tool_input.command')

# Only check git commit commands
if echo "$COMMAND" | grep -qE 'git\s+commit'; then
  # Extract the commit message
  MSG=$(echo "$COMMAND" | grep -oP '(?<=-m\s")[^"]+')
  if [ -n "$MSG" ]; then
    # Check conventional commits format: type(scope): description
    if ! echo "$MSG" | grep -qE '^(feat|fix|docs|style|refactor|test|chore|ci|perf)(\(.+\))?: .+'; then
      echo "BLOCKED: Commit message doesn't follow conventional commits format." >&2
      echo "Expected: type(scope): description" >&2
      echo "Examples: feat(auth): add login screen" >&2
      echo "          fix(api): handle timeout errors" >&2
      exit 2  # exit 2 = block the tool call, send stderr back to Claude
    fi
  fi
fi

Keep PreToolUse hooks fast. They run before every matched tool use, so a slow hook slows down everything. Aim for under 100ms. If your check needs to do something heavy (like running Gradle), use a PostToolUse hook instead — it won’t block the action.

Combining Multiple Hooks

You can (and should) stack multiple handlers per matcher — they run in the order listed. Cluster all your Edit|Write reactions into a single inner hooks array:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/format-changed.sh" },
          { "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/lint-changed.sh" },
          { "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/test-changed.sh" }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/guard-main.sh" },
          { "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/check-commit-msg.sh" }
        ]
      }
    ]
  }
}

Hook Debugging

When a hook doesn’t do what you expect, here’s how to figure out why:

1. Add logging to your hook script

#!/bin/bash
# Capture stdin once, log it, then re-feed it to your real logic.
INPUT=$(cat)
echo "[HOOK $(date)] $0 — $INPUT" >> /tmp/claude-hooks.log
echo "$INPUT" | jq .

2. Common issues and fixes

Symptom Likely Cause Fix
Hook never fires Wrong matcher or settings file location Check matcher matches the tool name exactly; verify settings.json path
Hook fires but does nothing Script not executable or wrong path Run chmod +x on the script; use relative path from project root
Hook breaks Claude Code flow Script has an error or hangs Test the script manually first: ./scripts/lint-changed.sh test.kt
PreToolUse blocks unexpectedly Matcher is too broad Make the grep pattern more specific, or add an early exit for safe commands

3. Test hooks in isolation

Before adding a hook to settings.json, always test the script by hand — pipe in a fake JSON payload that mimics what Claude Code would send:

# Test the lint hook with a fake Edit payload
echo '{"tool_input":{"file_path":"src/main/java/com/myapp/UserViewModel.kt"}}' \
  | ./scripts/lint-changed.sh

# Test the guard hook with a simulated Bash payload
echo '{"tool_input":{"command":"git push --force origin main"}}' \
  | ./scripts/guard-main.sh
# Should print a JSON deny decision and exit 0

Hook Performance: Keep It Snappy

Hooks run synchronously. A slow hook means Claude Code pauses every time it edits a file. Here’s how to keep things fast:

Hook Performance Rules

PreToolUse hooks: Must be under 100ms. Use simple grep/regex checks, never Gradle or network calls.

PostToolUse hooks: Aim for under 5 seconds. Lint one file, not the whole project. Run one test class, not the full suite.

Stop hooks: Can take longer since they run at the end. Good for summaries and full-suite tests.

  • Lint one file, not the project. ktlint "$FILE" takes 200ms. ktlint on the whole project takes 10 seconds.
  • Run targeted tests, not the full suite. --tests "*UserViewModelTest" takes 3 seconds. The full suite takes 2 minutes.
  • Early-exit when irrelevant. If the file isn’t Kotlin, exit immediately. Don’t run any checks.
  • Use --quiet flags. Verbose output from Gradle or linters slows things down and clutters the context.

Project-Level vs. User-Level Hooks

Setting Location Shared? Use for
Project hooks .claude/settings.json Yes (checked in) Team standards: lint, format, guardrails
User hooks ~/.claude/settings.json No (personal) Personal preferences: notifications, logging

Team guardrails (no force-push to main, conventional commits) belong in project-level settings. Personal preferences (desktop notification sounds, session logging) belong in user-level settings.

Real-World Hook Recipes

Notification when Claude Code finishes a long task

// User-level: ~/.claude/settings.json
{
  "hooks": {
    "Notification": [
      { "hooks": [
        { "type": "command",
          "command": "bash -c 'MSG=$(jq -r .message); notify-send \"Claude Code\" \"$MSG\"'" }
      ] }
    ]
  }
}

Log every Claude Code session

// .claude/settings.json — fire SessionEnd to capture exit time
{
  "hooks": {
    "SessionEnd": [
      { "hooks": [
        { "type": "command",
          "command": "bash -c 'echo \"$(date +%Y-%m-%d\\ %H:%M) — done\" >> \"$CLAUDE_PROJECT_DIR\"/.claude/activity.log'" }
      ] }
    ]
  }
}
