Documentation That Writes Itself

Auto-generate READMEs, API docs, ADRs, and onboarding guides — and keep them forever in sync

The Documentation Graveyard

Every project has one. A README.md that describes the setup from two years ago. An API doc that lists endpoints that no longer exist. A wiki page titled “Architecture Overview” that matches approximately nothing in the current codebase.

The problem isn’t that nobody writes docs. Someone always writes the initial version. The problem is nobody updates them. Code changes twenty times a day. Documentation changes twice a quarter, if you’re lucky.

60% of developers say their project docs are outdated
45 min average time to write a good README from scratch
0 min time most devs spend updating docs after code changes

The solution isn’t “be more disciplined about documentation.” That’s never worked. The solution is making documentation generation so easy that it happens without thinking about it — just like linting and formatting happen via hooks.

Auto-Generating READMEs

The README is your project’s front door. Let’s build a skill that writes it by reading your actual code.

Building the /generate-readme Skill

  1. Create: .claude/commands/generate-readme.md
  2. Add this content:
# Generate README

Generate or update the project README.md.

## Steps
1. Read the current README.md (if it exists) to understand what's there
2. Read CLAUDE.md for project description and conventions
3. Scan the project structure (directories, key files, build files)
4. Read build.gradle.kts for dependencies and SDK versions
5. Check for existing setup scripts or Makefiles

## Sections to include
### Project Title & Description
- One-paragraph summary of what the project does
- Tech stack badges (Kotlin, Compose, min SDK, etc.)

### Prerequisites
- Required tools and versions (Android Studio, JDK, etc.)
- Required environment variables or config files

### Getting Started
- Clone instructions
- How to open in Android Studio
- How to build and run (exact Gradle commands)
- How to run tests

### Architecture
- High-level architecture pattern (Clean Architecture layers, etc.)
- Key modules/packages and their responsibilities
- Diagram if the project has one, or describe the data flow

### Contributing
- Branch naming convention
- PR process (reference CLAUDE.md if it has PR standards)
- Code style (link to CLAUDE.md rules)
- How to run linting and formatting

### Available Claude Code Skills
- List all skills in .claude/commands/ with one-line descriptions
- Show example usage for the most important ones

## Rules
- Base everything on the actual code, not assumptions
- If something can't be determined from the code, add a TODO marker
- Keep it concise — a README should be scannable in 2 minutes
- Use markdown formatting: headers, code blocks, bullet points
  1. Run it: /generate-readme
  2. Review and commit the generated README

README Without Automation

  1. Someone writes a README at project start
  2. Dependencies change — README not updated
  3. Architecture evolves — README not updated
  4. New team member follows outdated setup steps
  5. Wastes 2 hours on issues the README didn’t mention
  6. Nobody updates the README because it’s “too much work”

Result: a README that actively misleads.

README With /generate-readme

  1. Run /generate-readme anytime (takes 30 seconds)
  2. README is based on actual current code
  3. Setup steps match real build configuration
  4. New team member follows steps that actually work
  5. Scheduled agent flags when README goes stale

Result: a README that reflects reality.

API Documentation from Code

If your Android app has a backend, or if you maintain an SDK that other teams consume, API docs are critical. Build a skill that generates them from your actual endpoint definitions:

# File: .claude/commands/generate-api-docs.md

Generate API documentation from the codebase.

## Steps
1. Find all Retrofit interface definitions (or Ktor routes, etc.)
2. Read each endpoint: HTTP method, path, request body, response type
3. Read the data classes used in requests and responses
4. Check for existing API docs to preserve any manual notes

## For each endpoint, document
- **Method & Path**: `GET /api/v1/users/{id}`
- **Description**: What this endpoint does
- **Request**: Headers, path params, query params, body (with example JSON)
- **Response**: Success response (with example JSON), error responses
- **Authentication**: Required? What type?
- **Rate limits**: If documented in code comments

## Output
Generate a markdown file: `docs/API.md`
Group endpoints by feature area (users, payments, etc.)
Include a table of contents at the top.

Generate docs from code, not the other way around. If your Retrofit interfaces define @GET("/api/v1/users/{id}") and your data classes define the response shape, Claude Code can read those and generate accurate docs. The code is always the source of truth — let the docs flow from it.

KDoc Generation

Public APIs need documentation. But adding KDoc to every public class and function is tedious. Automate it:

# File: .claude/commands/generate-kdoc.md

Add KDoc documentation to undocumented public APIs in: **$ARGUMENTS**

## Steps
1. Read the specified file (or all files in the specified directory)
2. Find every public class, function, and property without KDoc
3. Read the implementation to understand what each element does
4. Add KDoc comments

## KDoc format
- Classes: purpose, usage example, thread safety if relevant
- Functions: what it does, @param for each parameter,
  @return description, @throws for declared exceptions
- Properties: what it represents, valid value range if applicable

## Rules
- Don't modify existing KDoc — only add where missing
- Don't document obvious things (getters, data class properties)
- DO document: complex logic, non-obvious behavior, side effects
- Keep descriptions concise — one sentence for simple elements
- Include @param and @return for public functions with non-obvious signatures
- Use the style of existing KDoc in the project as a reference

Run /generate-kdoc src/domain/ and every public API in your domain layer gets documented in seconds.

Architecture Decision Records

ADRs capture why you made architectural decisions. They’re invaluable six months later when someone asks “why did we use Room instead of SQLDelight?” But nobody writes them because the decision is fresh in your mind and it feels unnecessary. Until it isn’t.

# File: .claude/commands/generate-adr.md

Write an Architecture Decision Record for: **$ARGUMENTS**

## Steps
1. Read the relevant code and recent commits for context
2. If a PR number is provided, read the PR discussion for rationale

## ADR Template (follow exactly)
```markdown
# ADR-{next number}: {Title}

## Status
Accepted

## Date
{today's date}

## Context
What was the problem or need that prompted this decision?

## Decision
What was decided, with enough detail to understand the approach.

## Alternatives Considered
What else was evaluated and why it was rejected.

## Consequences
### Positive
- What we gain from this decision

### Negative
- Trade-offs we're accepting

### Risks
- What could go wrong and how we'd handle it
```

## Output
Save to docs/adr/{next-number}-{slug}.md
Update docs/adr/index.md with a link to the new ADR.

Generate ADRs at decision time, not later. When you’re in a Claude Code session discussing architectural options, run /generate-adr "Use Room for local persistence" right then. Claude Code captures the context while it’s still fresh — the alternatives you discussed, the trade-offs you weighed. It takes 30 seconds instead of the 30 minutes you’d never spend writing it later.

Keeping Docs in Sync

Generating docs is step one. Keeping them current is step two. Build a hook that flags stale documentation:

#!/bin/bash
# scripts/check-doc-freshness.sh — Flag when code changes may make docs stale
FILE="$1"

# Map source files to their documentation
declare -A DOC_MAP
DOC_MAP["src/data/api/"]="docs/API.md"
DOC_MAP["src/domain/"]="docs/architecture.md"
DOC_MAP["build.gradle"]="README.md"
DOC_MAP["settings.gradle"]="README.md"

for pattern in "${!DOC_MAP[@]}"; do
  if [[ "$FILE" == *"$pattern"* ]]; then
    DOC_FILE="${DOC_MAP[$pattern]}"
    if [ -f "$DOC_FILE" ]; then
      echo "NOTE: You changed $FILE — $DOC_FILE may need updating."
    fi
  fi
done
// In .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash -c './scripts/check-doc-freshness.sh \"$CLAUDE_FILE\"'"
      }
    ]
  }
}

Now every time Claude Code edits an API file, it sees a reminder: “docs/API.md may need updating.” It can then update the docs in the same session.

Onboarding Documentation

New team member joins. They need to understand the project. Instead of pointing them at an outdated wiki, generate a fresh onboarding guide:

# File: .claude/commands/generate-onboarding.md

Generate a new-developer onboarding guide for this project.

## Steps
1. Read CLAUDE.md, README.md, and any docs/ files
2. Scan the project structure and build configuration
3. Read the most important files (entry points, DI setup, navigation)

## Sections
### Project Overview
- What the app does, who uses it, key features

### Setup Guide
- Step-by-step from zero to running the app locally
- Common setup issues and how to fix them

### Architecture Map
- The layers and how data flows through them
- Key files: "start here" pointers for each layer

### Codebase Tour
- Top 10 files to read first, in order, with one-line explanations
- Key patterns to understand (DI, navigation, state management)

### Development Workflow
- Branch naming, PR process, review standards
- Available Claude Code skills and when to use them
- Hooks that run automatically

### Who to Ask
- Areas of expertise by team member (if findable in git history)

## Rules
- Write for someone who knows Android but not this project
- Include actual file paths, not generic descriptions
- Add copy-paste commands wherever possible

Diagrams from Code

A picture of your architecture is worth a thousand lines of documentation. Generate Mermaid diagrams that you can embed in your README or docs:

# File: .claude/commands/generate-diagram.md

Generate a Mermaid architecture diagram for: **$ARGUMENTS**

## Steps
1. Read the relevant source files
2. Identify components, their relationships, and data flow
3. Generate a Mermaid diagram

## Diagram types (based on $ARGUMENTS)
- "architecture" → high-level module/layer diagram
- "data-flow" → how data moves from API to UI
- "navigation" → app navigation graph
- "di" → dependency injection module relationships

## Output
```mermaid
graph TD
  A[UI Layer] --> B[ViewModel]
  B --> C[UseCase]
  C --> D[Repository]
  D --> E[Remote API]
  D --> F[Local DB]
```

Output the Mermaid code in a fenced code block.
Include a title and brief description above the diagram.

Mermaid renders natively on GitHub. If you put a Mermaid code block in your README.md or any markdown file on GitHub, it renders as a visual diagram automatically. No images to maintain, no external tools — the diagram is code that stays in sync when you regenerate it.

Documentation Freshness Checks

Combine the freshness concept with the scheduled agents from Chapter 7 for a weekly doc audit:

name: Documentation Freshness Check
on:
  schedule:
    - cron: '0 9 * * 1'  # Every Monday 9 AM UTC

jobs:
  check-docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Audit documentation
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude -p "Audit documentation freshness for this project.
            For each doc file (README.md, docs/*, CLAUDE.md):
            1. When was it last modified? (git log)
            2. What source files have changed since then?
            3. Are there any obvious mismatches between docs and code?
            4. Rate freshness: CURRENT / STALE / OUTDATED

            Output as a markdown report with specific update recommendations." \
            --allowedTools "Read,Bash(git log*),Bash(find*)" \
            > freshness-report.md

      - name: Create issue if stale docs found
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const report = fs.readFileSync('freshness-report.md', 'utf8');
            if (report.includes('STALE') || report.includes('OUTDATED')) {
              github.rest.issues.create({
                owner: context.repo.owner,
                repo: context.repo.repo,
                title: `Docs freshness audit — ${new Date().toISOString().split('T')[0]}`,
                body: report,
                labels: ['documentation']
              });
            }

The Documentation Workflow

Write Code
→
Hook Flags Stale Docs
→
Regenerate Docs
→
Review
→
Weekly Audit
  • Write code — you make changes as usual
  • Hook flags stale docs — the PostToolUse hook tells Claude Code which docs may be affected
  • Regenerate docs — run the relevant /generate skill (or Claude Code does it in the same session)
  • Review — docs go through the same PR review as code
  • Weekly audit — scheduled agent catches anything that slipped through

The Documentation Golden Rule

Treat documentation like test coverage: if it can be generated from source, it should be. If it must be hand-written (architecture rationale, onboarding context), generate the skeleton and fill in the human judgment parts. Never write boilerplate documentation by hand.
