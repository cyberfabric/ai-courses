AI-Powered Code Review

Consistent, thorough reviews on every PR — in minutes, not days

The Code Review Bottleneck

You know the pattern. You open a PR at 2 PM. Your reviewer is in meetings until 4. They start looking at 4:30 but get pulled into a production issue. Next morning they leave a few comments. You address them, push, and wait again. The PR that took an hour to write takes three days to merge.

Code review is the single biggest bottleneck in most development workflows. Not because reviews aren’t valuable — they absolutely are — but because they depend on a scarce resource: another human’s focused attention.

24–72h average time to first review comment
60% of review comments are about style, not logic
4–6 context switches per PR review cycle

AI doesn’t replace human reviewers. But it handles the mechanical parts — style checks, pattern violations, common bugs, security scanning — so your human reviewer can focus on architecture, design, and the stuff only a human understands.

Quick Review from the Command Line

The simplest way to get an AI review: pipe your diff into Claude Code.

# Review your current changes against main
git diff main...HEAD | claude -p "Review this diff. For each issue found, report:
- File and approximate location
- Severity: critical / warning / suggestion
- What the issue is
- How to fix it

Focus on: bugs, security issues, performance problems, and Android-specific pitfalls."

That’s a useful one-liner, but it’s not repeatable or customizable. Let’s make it a proper skill.

Building the /review Skill

Creating a Comprehensive /review Skill

  1. Create the file: .claude/commands/review.md
  2. Add this content:
# Code Review

Review the current branch's changes against main.

## Steps
1. Run `git diff main...HEAD` to get all changes
2. Run `git log main..HEAD --oneline` for commit context
3. Read CLAUDE.md for team review standards

## Review Checklist

### Logic & Correctness
- Are there any bugs or logic errors?
- Are edge cases handled (null, empty, boundary values)?
- Do error paths behave correctly?

### Security
- SQL injection, XSS, or command injection risks?
- Secrets or API keys in the diff?
- Proper input validation on user-facing inputs?
- Correct permission checks?

### Android-Specific
- Lifecycle awareness — will this leak or crash on config change?
- Threading — are coroutines scoped correctly (viewModelScope, not GlobalScope)?
- Memory — large bitmaps, unbounded caches, or retained references?
- Compose — unnecessary recompositions? Missing `remember` or `key`?

### Style & Conventions
- Matches patterns in CLAUDE.md?
- Consistent naming?
- No unnecessary complexity?

### Performance
- N+1 queries or redundant network calls?
- Missing pagination on large lists?
- Heavy work on the main thread?

## Output Format
For each issue, report:
```
### [SEVERITY] File: path/to/file
**Line area:** ~line number or function name
**Issue:** What's wrong
**Fix:** How to fix it
```

Severity levels: CRITICAL (must fix), WARNING (should fix), SUGGESTION (nice to have)

End with a summary: total issues by severity, and an overall assessment.
  1. Test it: /review in Claude Code on any branch with changes

Now every developer on your team has the same thorough review checklist. No more “I forgot to check for lifecycle leaks” — the skill checks every time.

Domain-Specific Review Checklists

One size doesn’t fit all. A UI change needs different checks than a database migration. Build specialized review skills:

/review:android-ui — Compose and UI changes

# File: .claude/commands/review/android-ui.md

Review the current changes, focusing on UI and Compose code.

Check for:
- **Recomposition**: Are composables stable? State hoisted correctly?
- **Accessibility**: contentDescription on all interactive elements, touch targets >= 48dp
- **Theming**: Using MaterialTheme values, not hardcoded colors/dimensions
- **Responsiveness**: Works on different screen sizes? Handles landscape?
- **State management**: collectAsStateWithLifecycle() not collectAsState()
- **Previews**: Every new screen has @Preview with light and dark themes
- **Navigation**: Correct back stack behavior, no leaked screens

Flag any hardcoded strings (should be in strings.xml).

/review:security — Security-focused review

# File: .claude/commands/review/security.md

Security review of the current branch changes.

## OWASP Mobile Top 10 Checks
1. **Insecure Data Storage**: SharedPreferences for sensitive data? Should use EncryptedSharedPreferences
2. **Insecure Communication**: HTTP instead of HTTPS? Certificate pinning missing?
3. **Insecure Authentication**: Tokens stored in plain text? Missing token refresh?
4. **Insufficient Input Validation**: User input passed to SQL, WebView, or Intents unvalidated?
5. **Insufficient Cryptography**: Weak algorithms? Hardcoded keys?
6. **Insecure Authorization**: Client-side-only permission checks?
7. **Client Code Quality**: Debug logging with sensitive data? Stack traces in production?
8. **Code Tampering**: ProGuard/R8 rules for new classes?
9. **Reverse Engineering**: Sensitive logic that should be server-side?
10. **Extraneous Functionality**: Test endpoints, debug flags, or backdoors?

## Also check
- Intent filters: are exported components properly secured?
- Deep links: validated before processing?
- WebView: JavaScript enabled without need? loadUrl with user input?

Report each finding with severity and remediation steps.

Review skills encode your team’s hard-won knowledge. Every security incident, every production bug, every “we should check for that in review” moment — capture it in a review skill. Over time, your /review commands become a living encyclopedia of everything your team has learned the hard way. New developers get the benefit of that institutional knowledge from day one.

Automated PR Descriptions

You built a /pr-description skill in Chapter 3. Let’s level it up by making it part of your review workflow. The idea: generate the PR description and a self-review in one step.

# File: .claude/commands/ship-pr.md

Prepare this branch for a pull request.

## Steps
1. Run `git diff main...HEAD` for all changes
2. Run `git log main..HEAD --oneline` for commit history
3. Read CLAUDE.md for PR template and conventions

## Generate

### PR Description
Write a complete PR description following the team template:
- Title (imperative mood, under 70 characters)
- ## What — bullet list of changes
- ## Why — motivation and context
- ## Testing — specific verification steps
- ## Breaking Changes — if any (with migration notes)

### Self-Review Findings
Run through the review checklist from CLAUDE.md and flag
anything the author should fix before requesting review.

## Output
Print the PR description in a markdown code block (ready to copy-paste),
followed by any self-review findings that need attention.

Self-review before human review. Having Claude Code review your own PR before you request human review catches the easy stuff early. Your reviewer spends less time on nitpicks and more time on meaningful feedback. Everyone wins.

Encoding Review Standards in CLAUDE.md

Skills define what to check. CLAUDE.md defines what good looks like. When both work together, reviews are consistent across the entire team.

# In your project's CLAUDE.md — add a Review Standards section

## Review Standards

### Must-fix (block merge)
- Any failing test
- Security vulnerabilities (OWASP top 10)
- Memory leaks or lifecycle issues
- Missing null checks on external API responses
- Hardcoded secrets or API keys

### Should-fix (fix before next release)
- Missing KDoc on public APIs
- Hardcoded strings (should be in strings.xml)
- Complex functions without tests (> 20 lines of logic)
- TODO comments without ticket numbers

### Nice-to-have (suggestion only)
- Variable naming improvements
- Extracting reusable functions
- Reducing nesting levels
- Adding Compose previews for new components

Now when any Claude Code session runs a review — whether through a skill, a CI step, or an interactive conversation — it knows your team’s severity levels.

Review in GitHub Actions

The ultimate automation: every PR gets an AI review automatically, with comments posted directly to GitHub. No human has to remember to run anything.

The cleanest way is Anthropic’s official action, anthropics/claude-code-action@v1. It runs Claude Code in CI, posts comments to the PR for you, and handles authentication — no manual install or extra github-script step needed:

# .github/workflows/ai-review.yml
name: AI Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            Review this PR's diff for an Android project. Check for:
            - Bugs and logic errors
            - Security issues (OWASP mobile top 10)
            - Android pitfalls (lifecycle, threading, memory)
            - Style violations

            For each issue, report the file, severity (critical/warning/suggestion),
            the problem, and the fix. Format as a markdown checklist; end with a summary.
          claude_args: --tools "Read,Grep,Bash(git *)"

--tools vs --allowedTools. In CI you usually want to restrict which tools are available at all — that’s --tools "Read,Grep,Bash(git *)". The older --allowedTools flag still works, but it controls auto-approval, not availability; it doesn’t stop a tool from running, just skips the prompt. For deny-by-default review jobs, prefer --tools (or --disallowedTools to block specific tools).

Need raw control? The older manual pattern still works — npm install -g @anthropic-ai/claude-code, then git diff … | claude -p "…" > review.md, then post the file with actions/github-script@v9. Reach for it when you need to script multiple Claude calls or pipe output through other tools before commenting.

Code Review Without Automation

  1. Developer opens PR at 2 PM
  2. Reviewer starts looking next morning
  3. Leaves 12 comments (8 are style nitpicks)
  4. Developer fixes, pushes, waits again
  5. Second review round: 2 more comments
  6. Approved after 2–3 days

Elapsed time: 2–3 days. Developer context-switches: 4–6 times.

Code Review With AI Automation

  1. Developer runs /ship-pr for self-review
  2. Fixes issues before opening PR
  3. Opens PR — AI review posts in 3 minutes
  4. Addresses AI findings, pushes
  5. Human reviewer sees clean PR, focuses on design
  6. Approved same day, often within hours

Elapsed time: hours, not days. Style nitpicks: zero (caught by AI).

Handling Review Output

A good review is only useful if it’s easy to act on. Structure your review output for scanability:

Hard to scan

The code looks mostly fine but there's an issue in UserRepository where the token might be null if the refresh fails, also the ViewModel doesn't handle the loading state correctly and there's a potential memory leak in the observer...

Wall of text. Hard to track which issues are fixed and which aren’t.

Easy to act on

### CRITICAL: UserRepository.kt **Function:** fetchUser() **Issue:** Null token after refresh failure causes crash **Fix:** Add null check before API call ### WARNING: HomeViewModel.kt **Function:** loadData() **Issue:** Loading state never set to false on error path **Fix:** Add `_uiState.value = Loaded` in catch block ### SUGGESTION: HomeScreen.kt **Issue:** Missing remember{} around derived state **Fix:** Wrap calculation in remember(key)

Structured, scannable, actionable. Each issue is independent and trackable.

Review for Specific File Types

Not every file needs the same review. Gradle files need dependency checks. XML needs accessibility checks. Kotlin needs lifecycle awareness. Use the file type to focus the review:

# File: .claude/commands/review/focused.md

Review only the files matching: **$ARGUMENTS**

## Steps
1. Run `git diff main...HEAD -- $ARGUMENTS` for targeted diff
2. Apply the appropriate review lens:

### For *.kt files
- Coroutine scoping, lifecycle awareness, null safety, Compose stability

### For *.gradle.kts / *.toml files
- New dependencies: are they necessary? Actively maintained? Any CVEs?
- Version conflicts with existing dependencies
- ProGuard rules needed for new libraries?

### For *.xml files (layouts)
- Accessibility: contentDescription, importantForAccessibility
- Hardcoded dimensions (should use dimens.xml)
- Deprecated attributes

### For AndroidManifest.xml
- New permissions: are they necessary? Explained in privacy docs?
- Exported components: properly secured?
- Intent filters: validated?

Output findings grouped by file, with severity levels.

The Review Workflow Stack

Here’s how all the review pieces fit together:

CLAUDE.md Standards
→
/ship-pr Self-Review
→
CI AI Review
→
Human Review
  • CLAUDE.md defines what “good code” means for your team
  • /ship-pr catches issues before the PR is even opened
  • CI AI review runs automatically on every push, posts findings as comments
  • Human review focuses on design, architecture, and context that only a teammate understands

Each layer catches different things. By the time a human looks at the code, the mechanical issues are already gone.

AI review augments humans, it doesn’t replace them. Claude Code is excellent at catching bugs, style issues, and known vulnerability patterns. It’s not great at judging whether the overall design is right, whether a feature matches the product intent, or whether the approach will create maintenance headaches six months from now. Keep humans in the loop for the high-judgment calls.

Exercise: Build Your Review Pipeline

Exercise: Set Up AI Code Review in Three Layers

Build this end-to-end. Total time: about 20 minutes.

Layer 1: Review standards in CLAUDE.md

  1. Open your project’s CLAUDE.md
  2. Add a “## Review Standards” section with must-fix, should-fix, and nice-to-have categories
  3. Include at least 5 rules specific to your project (Android lifecycle? API patterns? Testing requirements?)

Layer 2: The /review skill

  1. Create .claude/commands/review.md using the template from this chapter
  2. Customize the Android-specific checks for your project’s patterns
  3. Test it on an existing branch: run /review and check if the output matches what a human reviewer would catch

Layer 3: Self-review before PR

  1. Create .claude/commands/ship-pr.md
  2. On your next feature branch, run /ship-pr before opening the PR
  3. Fix any findings, then open the PR
  4. Notice how much cleaner the human review goes

Bonus: If your project uses GitHub Actions, set up the ai-review.yml workflow from this chapter. Watch it post automated review comments on your next PR.
