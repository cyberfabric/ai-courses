Chapter 13

Plugins & the Marketplace

Bundle your skills, hooks, agents, and MCP servers into one shareable unit — and discover what others have built.

The Last Mile of “Share With Your Team”

Every chapter so far has had the same nagging undertone. You build a great /pr-description skill (Chapter 3). You write a sharp lint-on-save hook (Chapter 4). You stand up an MCP server for your team’s deploy dashboard (Chapter 10). And then someone on Slack says “wait, that’s cool, how do I get that?” — and you find yourself zipping up a .claude/ directory, writing a setup README, and walking three teammates through copying files into the right places.

Plugins are the answer. A plugin packages slash commands, skills, hooks, MCP servers, subagents, and statusline scripts into one self-contained directory. A marketplace is a JSON catalog that points at a bunch of plugins. Together they’re the package manager Claude Code didn’t have until late 2025.

If Chapters 1–12 taught you to build automations, this chapter teaches you to distribute them.

What a Plugin Actually Is

Forget marketplaces for a minute. A plugin is just a directory with a tiny manifest file. Everything else — the skills, the hooks, the MCP servers — auto-discovers from standard subdirectories.

my-plugin/
├── .claude-plugin/
│   └── plugin.json          # The only required file. Just `name`.
├── commands/                # Slash commands (.md files)
│   └── pr-description.md
├── skills/                  # SKILL.md skills (model-invokable)
│   └── scaffold-feature/
│       └── SKILL.md
├── agents/                  # Subagents (Chapter 14)
│   └── code-reviewer.md
├── hooks/                   # Hook scripts referenced from .claude/settings.json
│   └── lint-changed.sh
└── .mcp.json                # MCP server registrations

The manifest path is hyphenated. It’s .claude-plugin/plugin.json, not .claude/plugin.json. Same trick on the marketplace side: the catalog lives at .claude-plugin/marketplace.json in the marketplace repo. Easy to typo, easy to fix.

A minimal plugin.json is genuinely tiny:

{
  "name": "team-android-toolkit",
  "version": "0.3.0",
  "description": "Skills and hooks for Android dev: scaffolding, linting, PR review",
  "author": { "name": "Mobile Platform" },
  "license": "MIT",
  "homepage": "https://github.com/acme/team-android-toolkit",
  "repository": "https://github.com/acme/team-android-toolkit",
  "keywords": ["android", "kotlin", "compose", "review"]
}

Only name is required. The other fields exist so the marketplace UI can show useful context and so /plugin update can detect that you’ve published a new version.

Your plugin is mostly stuff you’ve already built. If you followed Chapters 3, 4, and 10, your .claude/commands/, .claude/settings.json hooks, and a couple of MCP servers are already 80% of a plugin. The only new artifact is the manifest. Bundling them is less “build a thing” and more “wrap what you have in a directory and write five lines of JSON.”

Browsing & Installing From the Official Marketplace

The Anthropic-managed marketplace, claude-plugins-official, is auto-available the moment you start Claude Code — you don’t have to add it. To browse:

# Open the interactive plugin manager
/plugin

# You'll see three tabs: Discover, Installed, and Updates.
# Discover shows everything available across all your marketplaces.
# Press Enter on a plugin to see options.

If you already know what you want, you can install it directly from the command line:

# Install from the official marketplace
/plugin install claudemd-keeper@claude-plugins-official

# Format: <plugin-name>@<marketplace-name>
# The @marketplace suffix disambiguates when the same name appears in
# multiple marketplaces.

Install scopes: where the plugin lives

When you install a plugin, Claude Code asks (or you specify) where to put it:

ScopeWhereWhen to use
user (default) Your home directory — available across all your projects. Personal productivity stuff: shell aliases, code-archaeology skills, your private morning-digest plugin.
project Committed to the current repo (.claude/plugins/ or .mcp.json). Team-shared automations. Anyone who clones the repo gets them after running /plugin sync.
local This repo, your machine only — not committed. Plugins you want to try out before deciding whether to share them with the team.

Adding Your Own Marketplaces

The official marketplace is a great starting point, but the real power kicks in when you add your team’s private marketplace, or a community-maintained one.

# GitHub repo (most common form)
/plugin marketplace add anthropics/claude-plugins-official

# Any git URL
/plugin marketplace add https://gitlab.acme.internal/platform/plugins.git

# Local directory (great for plugin development)
/plugin marketplace add ./local-marketplaces/team-android

# Remote URL pointing at a marketplace.json
/plugin marketplace add https://plugins.acme.com/marketplace.json

A marketplace itself is just a JSON file at .claude-plugin/marketplace.json in the marketplace repo. It looks roughly like this:

{
  "name": "acme-platform-plugins",
  "owner": { "name": "Acme Platform Team" },
  "plugins": [
    {
      "name": "android-toolkit",
      "source": "github:acme/android-toolkit-plugin",
      "version": "0.3.0",
      "description": "Skills and hooks for Android dev"
    },
    {
      "name": "deploy-dashboard",
      "source": "github:acme/deploy-dashboard-plugin",
      "version": "1.0.2",
      "description": "MCP server + status skill for our deploy pipeline"
    }
  ]
}

Marketplace = catalog, plugin = code. The marketplace JSON points at where to fetch each plugin, but doesn’t contain the plugin code itself. This split (covered in “Team-shared marketplaces” below) is what lets you curate an approval list without forking every plugin.

Building Your First Plugin (15 Minutes)

Time to turn what you have into something installable. We’ll bundle the /pr-description skill from Chapter 3 with the lint-on-save hook from Chapter 4 into a single plugin called android-toolkit.

From scattered files to one shareable plugin

  1. Create the plugin directory. 
    mkdir -p android-toolkit/.claude-plugin
mkdir -p android-toolkit/commands
mkdir -p android-toolkit/hooks
cd android-toolkit
  2. Write the manifest. Save as .claude-plugin/plugin.json: 
    {
  "name": "android-toolkit",
  "version": "0.1.0",
  "description": "PR descriptions and lint-on-save for Android repos",
  "author": { "name": "Your Name" },
  "license": "MIT"
}
  3. Drop in your existing skill. Copy .claude/commands/pr-description.md from your project into commands/pr-description.md. No edits needed — the file is the same.
  4. Drop in your existing hook script. Copy scripts/lint-changed.sh into hooks/lint-changed.sh. Reference it from a hooks/hooks.json file: 
    {
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "${CLAUDE_PLUGIN_ROOT}/hooks/lint-changed.sh" }
        ]
      }
    ]
  }
}
    The $CLAUDE_PLUGIN_ROOT env var resolves to the plugin’s install directory at runtime — portable across machines.
  5. Test it locally. From your project (not the plugin): 
    /plugin marketplace add /absolute/path/to/parent-of-android-toolkit
/plugin install android-toolkit
    Now /pr-description appears in your slash-command palette and the lint hook fires on file edits.
  6. Push to GitHub. git init, commit, push to github.com/<you>/android-toolkit-plugin. Anyone can now install it with /plugin marketplace add <you>/android-toolkit-plugin.

Versioning & Updates

The version field in plugin.json isn’t cosmetic — it’s how Claude Code decides whether to fetch new code on /plugin update. Treat it like any semver:

  • Patch (0.3.0 → 0.3.1) — bug fix in a hook script, typo in a skill prompt.
  • Minor (0.3.1 → 0.4.0) — new skill, new hook, new MCP server. Backwards compatible.
  • Major (0.4.0 → 1.0.0) — renamed a slash command, removed a skill, changed a hook’s required env. Things that break callers.

If you change code but forget to bump version, your users won’t see the change. Claude Code caches the resolved plugin contents and only re-fetches when the version changes. This bites people the first time they ship a fix and wonder why nobody’s reporting it as fixed. Bump the patch number on every meaningful change.

Team-Shared Marketplaces: The Two-Repo Pattern

For a team marketplace, resist the urge to put your marketplace JSON and your plugin code in the same repo. The mature pattern is two repos:

  • acme-plugins-marketplace — just .claude-plugin/marketplace.json plus approval metadata. The platform team owns this.
  • acme-android-toolkit-plugin, acme-deploy-dashboard-plugin, ... — one repo per plugin. Anyone can own these.

This split is what lets the platform team revoke a problematic plugin (remove it from the marketplace JSON) without touching plugin source, and lets plugin authors ship updates without coordinating with the marketplace owner.

For developers on the team, the install dance is one-time:

# Once, anywhere
/plugin marketplace add acme/acme-plugins-marketplace

# Then, in any project
/plugin install android-toolkit@acme-plugins-marketplace

The Colon: What Plugin Namespacing Actually Does

You’ll occasionally see Claude Code references using a colon in skill names — android-toolkit:pr-description or plugin:skill. This is plugin namespacing, and it’s the only place a colon shows up in slash-command names today.

It’s for disambiguation. If you have your own /pr-description in .claude/commands/ and the android-toolkit plugin also exposes /pr-description, calling /android-toolkit:pr-description picks the plugin’s version specifically. Most of the time you don’t need to namespace — the unqualified /pr-description works as long as there’s only one of them.

The old subdirectory-namespacing rumor. If you’ve read older Claude Code docs or blog posts, you may remember a claim that .claude/commands/android/scaffold.md becomes /android:scaffold. That was never true (or stopped being true very quickly). Subdirectories under commands/ just organize files — they don’t prefix the slash-command name. Plugin namespacing is the only colon-syntax that’s real.

Security: What Plugins Can (and Can’t) Do

Installing a plugin gives it everything Claude Code has: file access, shell execution via hooks, network calls via MCP servers, and the ability to read your project context. That’s a non-trivial trust boundary. A few realities to keep in mind for 2026:

  • No binary signing. Plugins are not cryptographically signed. The marketplace JSON is the trust anchor — if the marketplace owner is reputable, you trust their curation; otherwise you don’t.
  • Static analysis only catches obvious things. A plugin’s hook scripts and MCP server source are visible — you can read them. But a plugin that uses an LLM to generate shell commands at runtime can do whatever the runtime LLM tells it to. Static review won’t catch that.
  • Read the manifest before you install. Open the plugin’s GitHub repo. Skim the hook scripts. Check what the MCP servers connect to. It takes 2 minutes for a small plugin and is the single best safeguard.
  • Prefer project scope for team plugins. When a plugin is committed to the repo, every change goes through PR review — the same gate your normal code passes. User-scope installs bypass that.

The plugin you trust today might be a different plugin tomorrow. A reputable repo gets compromised. A maintainer hands the project off. /plugin update fetches new code. Your trust decision needs to either pin to a version ("version": "0.3.0" exact in your marketplace entry) or trust the maintainer ongoingly. Both are valid; pick one consciously.

Migration: From .claude/commands/ to a Plugin

Sharing automations the old way

  1. Zip up your .claude/ directory and scripts/ folder.
  2. Write a SETUP.md explaining where each file goes.
  3. Drop the zip in a Slack thread.
  4. Walk three teammates through the unzip-and-copy dance.
  5. Two months later: someone’s on an outdated version, someone else has local edits, no one remembers who has what.

Friction wins. Adoption stalls.

Sharing as a plugin

  1. Move files into a plugin directory and write a 5-line plugin.json.
  2. Push to a GitHub repo.
  3. Add to your team’s marketplace JSON.
  4. Anyone runs /plugin install android-toolkit@acme-plugins — done.
  5. Updates propagate via /plugin update on a single bump of version.

One install command. Versioned. Reversible.

The migration itself is mechanical — you’re moving files, not rewriting them. The skills, hooks, and MCP server config you’ve already written work unchanged inside a plugin.

Exercise: Ship Your First Plugin

By the end of this exercise you’ll have a real plugin installed in your real project, sourced from your real GitHub account. Plan: 30 minutes.

  1. Pick two artifacts from this course you’ve already built: one slash command (e.g. /pr-description, /standup, or /review) and one hook (e.g. lint-on-save or guard-main-branch).
  2. Create a plugin directory in a new repo: my-toolkit/.claude-plugin/plugin.json with the minimal manifest from the recipe above. Use your real name and a real GitHub URL.
  3. Move the artifacts into commands/ and hooks/. Add a hooks/hooks.json referencing your hook script with ${CLAUDE_PLUGIN_ROOT}.
  4. Local test: from your project, run /plugin marketplace add /absolute/path/to/my-toolkit then /plugin install my-toolkit. Verify the slash command appears and the hook fires.
  5. Publish: git init, push to GitHub. Run /plugin marketplace add <you>/my-toolkit from a different machine (or have a teammate try it).
  6. Document the manifest: open the plugin.json in your editor. Add description, keywords, and repository. Bump version to 0.1.1. Push. Run /plugin update — confirm the change is picked up.
  7. Bonus: create a private marketplace repo (my-team-plugins) with just a marketplace.json pointing at your toolkit. Add it to two teammates’ setups. Tell them about the rest of this course while they install.

The Big Idea

A plugin is a directory plus a manifest. A marketplace is a JSON catalog. Together they turn your personal automation collection into something a teammate can install with one command and update with another. Everything you built in Chapters 3–11 was waiting for this layer to exist; now that it does, the natural unit of sharing isn’t a Slack thread — it’s an installable plugin.
