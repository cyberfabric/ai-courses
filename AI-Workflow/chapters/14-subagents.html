Chapter 14

Subagents — Specialized Agents for Specialized Jobs

Delegate heavy work to focused agents with their own context window, tools, and model — and stop drowning your main conversation in logs.

The Context-Window Problem You Didn’t Know You Had

You’re working on a feature. You ask Claude to find every place a deprecated API is called. It runs ten greps, reads thirty files, and spits a 4,000-line wall of search results into the conversation. You needed three lines of summary; you got the entire codebase. Now your main conversation is bloated with logs you’ll never reference again, the next message costs more tokens, and Claude’s next decision has to wade through search noise to find your actual intent.

Subagents fix this. A subagent runs in its own context window, with its own system prompt, its own tool allowlist, and (optionally) its own model. It does the heavy work somewhere else and returns a summary — nothing else. Your main conversation stays clean. Your costs stay sane. And because the subagent has a focused prompt, it often does the job better than the main agent would have.

This is the “delegate to a focused agent” primitive that Chapter 11 hinted at when it talked about orchestration. We’re going to name it, build a few, and see where it fits in the broader automation stack you’ve assembled across this course.

What a Subagent Actually Is

A subagent is a markdown file. That’s it. The file lives in .claude/agents/<name>.md (project-level, shared with the team) or ~/.claude/agents/<name>.md (user-level, available across all your projects). It has YAML frontmatter on top and a system prompt below. Claude Code discovers them automatically.

---
name: code-reviewer
description: Use for thorough code reviews on staged changes. Invoke before opening a PR.
tools: Read, Grep, Glob, Bash
model: sonnet
---

You are a senior Android code reviewer. Your job is to review the staged
changes against the team's checklist:

1. Lifecycle: are coroutines scoped to viewModelScope or lifecycleScope?
2. Threading: any blocking calls on Dispatchers.Main?
3. Memory: any retained references that could leak across config changes?
4. Tests: does each new public function have at least one unit test?
5. Style: does the code match the conventions in CLAUDE.md?

For each issue, output:
- Severity (blocker / nit)
- File and line
- The problem in one sentence
- The suggested fix as a code snippet

End with a one-line PR verdict: ship / fix-then-ship / rework.

Drop that into .claude/agents/code-reviewer.md, commit it, and every engineer on your team has a code-reviewer agent they can invoke.

Frontmatter Reference

Only name and description are required. The rest are optional but high-leverage:

FieldRequired?What it does
name Yes The handle. Used in @agent-name mentions and in the Task tool. Must be unique within the project.
description Yes The trigger that Claude matches against your request to decide whether to delegate. Write it in “use when” form. Vague descriptions = vague delegation.
tools No Comma-separated allowlist of tools the subagent can use. If you omit it, the subagent inherits the parent thread’s tools. Whitelist for tight control.
model No Model alias (haiku, sonnet, opus) or full model ID. inherit uses whatever the main conversation is using. Defaults to the parent model if omitted.

The description is the most important field. Claude uses it to decide when to delegate. “Use for code reviews” is OK; “Use for thorough Android code reviews on staged changes — checks lifecycle, threading, memory leaks; invoke before opening a PR” is much better. Phrases like “use when”, “best for”, “invoke before” pattern-match well to user intent.

How Delegation Actually Works

There are three ways a subagent gets invoked, and they shade into each other:

  1. Auto-delegation by description. Claude reads every available subagent’s description against your request and decides whether to delegate. This is why descriptions matter so much — they’re the only signal Claude has at decision time.
  2. Explicit @-mention. Type @code-reviewer in your prompt and Claude routes that work to the named subagent unconditionally. April 2026 added typeahead for this — start typing @ and you’ll get a picker. Use it when you want certainty.
  3. Programmatic via the Task tool. When you orchestrate workflows from skills, slash commands, or the Claude Agent SDK, you can spawn subagents directly with the Task tool, passing a subagent_type field. This is how Chapter 11’s pipelines compose specialists.

Auto-delegation is the magic; @-mention is the safety net. If you find yourself @-mentioning the same agent every time, your description is too vague — tighten it and let auto-delegation do its job.

The Three Built-In Subagents

You’ve been using subagents already, even before reading this chapter. Claude Code ships with three:

🔍
Explore
Read-only codebase search. Finds files, traces symbols, answers “how does X work?” questions across many files. Use when scope is uncertain or many areas of the codebase are involved. Returns a digest, not a wall of code.
📋
Plan
Software-architect agent. Designs implementation plans, identifies critical files, considers trade-offs. Reach for it before non-trivial implementation work — it’s the difference between “start coding and hope” and “know where you’re going.”
🤖
general-purpose
The default catch-all. Multi-step tasks, complex research, anything that doesn’t fit Explore or Plan. When in doubt, this is the one Claude reaches for.

Custom subagents fill the gaps the built-ins don’t cover — team-specific reviewers, domain-flavored auditors, repeated workflows.

Writing Your First Subagent

Time to build one. We’ll create a code-reviewer that runs your team’s checklist before every PR.

Build a code-reviewer subagent in 10 minutes

  1. Use the interactive helper. Inside Claude Code, run /agents. The interactive UI walks you through name, description, tools, and model. It can even draft a first system prompt from a one-line description.
  2. Or write the file directly. Create .claude/agents/code-reviewer.md with the frontmatter from the example above (or a variant for your stack — Compose, KMM, server-side, whatever you ship).
  3. Tighten the tools. A reviewer is read-only. Restrict to Read, Grep, Glob, Bash — Bash only because reviewers often need to run a single linter or test. Don’t give it Write or Edit: a code-reviewer that edits files is no longer a reviewer.
  4. Pick the right model. Code review benefits from strong reasoning — sonnet is the safe default, opus when stakes are high. haiku is too small for nuanced review.
  5. Test it. Stage a real diff, then in your main session: @code-reviewer review the staged changes. The @ mention (added April 2026) gives you direct, reliable invocation; you can also let Claude auto-delegate based on the description.
  6. Iterate the system prompt. If reviews come back too lenient, add “assume the change must be production-ready”. Too pedantic, add “skip nits in test files”. The system prompt is your knob.
  7. Commit it. git add .claude/agents/code-reviewer.md, push, and your whole team gets the same reviewer. Project-level agents are a team standard, not a personal toy.

Subagents are reusable system prompts with batteries. Every time you find yourself pasting the same “OK now act as a careful reviewer” preamble into Claude, that’s a subagent waiting to be born. The frontmatter is the “batteries”: tool allowlist, model choice, auto-delegation. Move the prompt into .claude/agents/ and you stop retyping it forever.

Subagent vs. Skill vs. Plugin

Three customization primitives, easy to confuse. Here’s when each one wins:

PrimitiveLives inBest for
Skill / slash command .claude/commands/*.md or .claude/skills/<name>/SKILL.md Reusable prompts you invoke as /name. Run in your main context. Great for code generation, doc writing, PR descriptions.
Subagent .claude/agents/<name>.md Tasks that produce a lot of intermediate output (reading files, running tools) you don’t want polluting your main context. Reviewers, auditors, researchers, test runners.
Plugin Installed via /plugin install The distribution layer. Plugins bundle skills + subagents + hooks + MCP servers into one shareable unit. See Chapter 13.

The decision rule is mostly about output volume: does the task generate a bunch of stuff you don’t want to see in the main thread? If yes, subagent. If no, skill.

Cost Optimization: Right-Sizing the Model

Every subagent invocation is its own model call. The model field gives you a knob most people forget about.

  • Routine, structural work → Haiku. Boilerplate generation, mechanical refactors, rote test scaffolding. Haiku is fast and cheap; on these tasks it’s indistinguishable from Sonnet to a careful reviewer.
  • Reasoning, nuance, judgment → Sonnet. Code review, design feedback, ambiguity-resolving research. Sonnet is the daily driver default.
  • Tricky novel work → Opus. Tough debugging, architecture design, security audits where one missed detail matters. Use sparingly; the cost difference is real.
  • model: inherit — explicitly opt into “use whatever the main session is using.” Useful for orchestration agents whose job is just to coordinate; you want the orchestrator to match the user’s expected quality bar.

Running Subagents in Parallel

Here’s the unlock: subagents are stateless and independent, so you can fire several at once. When Claude is investigating something with multiple unrelated angles — database schema, API surface, UI flow — it can spawn three subagents in one go and merge the summaries.

You don’t do anything special; Claude handles parallel spawning when the work is genuinely independent. What you can do is design subagents that are fit for parallel work:

  • Narrow scope — one subagent shouldn’t need to know what the others are doing.
  • Pure-research role — if a subagent edits files, parallel runs can race.
  • Bounded output — each summary should be small enough that merging three of them still fits comfortably in the main context.

Subagents in CI

The same subagent file works in CI without modification. Anthropic’s claude-code-action@v1 picks up .claude/agents/ from the repo automatically, which means CI can spawn the exact specialists your local Claude Code uses.

# .github/workflows/pr-checks.yml
name: PR specialist review
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: anthropics/claude-code-action@v1
        with:
          prompt: |
            @code-reviewer review the diff against main.
            @security-auditor scan for OWASP issues.
            @test-runner run the test suite and report failures only.

Three specialists, three context windows, one merged review comment on the PR. The code-reviewer.md, security-auditor.md, and test-runner.md files in .claude/agents/ are all the wiring you need.

Project-Level vs. User-Level Subagents

ScopePathUse For
Project .claude/agents/ (committed to repo) Team standards: the team’s reviewer, the team’s test runner, the team’s security auditor. PR-reviewed like any other code.
User ~/.claude/agents/ (your home dir) Personal specialists: a code-archaeology agent for legacy investigations, a CSS-untangler, anything you bring to every project.

When both define the same name, the project-level agent wins — teams should be able to override your personal preferences with shared standards.

Common Mistakes

Over-fragmentation. Twenty tiny subagents (one for each linter, one for each file type) means Claude has to read twenty descriptions to delegate one task. Aim for half a dozen strong specialists, not thirty weak ones. If two subagents have descriptions that overlap, merge them.

Wrong tool allowlist. A reviewer with Write access becomes a fixer-upper that quietly mutates your files. A research agent without WebSearch can’t do its job. Be deliberate about the tools field — restrict reviewers to read-only, give researchers web access, and never give a subagent more capability than its role requires.

Vague descriptions. “Use for reviews” collides with the built-in general-purpose agent and your subagent never gets picked. Specific, concrete descriptions win delegation.

With a research subagent

  3. Your main conversation gets only the 12 lines.
  4. Token usage stays flat. Claude’s next decision is on signal, not noise.

One summary in. Heavy lifting elsewhere.

Exercise: Build Three Specialists for Your Project

Plan: 45 minutes. Build a real trio that solves real PRs in your real repo.

  1. Pick three roles that recur in your team’s work. Suggested defaults: code-reviewer, test-runner, security-auditor. Substitute domain-specific roles if those don’t fit (database-migrator, query-perf-checker, accessibility-auditor).
  2. Run /agents for each one. Use the interactive helper to draft each agent’s system prompt; refine afterwards.
  3. Tighten frontmatter: for each agent, set tools explicitly (don’t inherit) and pick the smallest model that still does the job well.
  4. Test on a real PR. Stage a non-trivial diff. In one prompt: “@code-reviewer review the diff. @test-runner run the suite. @security-auditor scan it.” All three run in parallel; you read three summaries.
  5. Wire into CI. Add the three to a GitHub Actions workflow using anthropics/claude-code-action@v1 with a similar three-mention prompt. Confirm a PR triggers all three with reasonable cost.
  6. Commit project-level. Push .claude/agents/* to the repo. Tell two teammates. Have them run @code-reviewer on their next PR.
  7. Bonus: bundle the three agents into a plugin (Chapter 13) and publish via your team’s marketplace. Now any project can install them with one command.

The Big Idea

A subagent is a markdown file that gives Claude a focused colleague: own context, own tools, own model. Use one when a task would otherwise drown your main thread in logs you don’t need. Three good specialists beat thirty weak ones. The frontmatter is the lever — right-size the tools, right-size the model, write a description specific enough to win delegation.
