Your Automation Toolkit

Every building block Claude Code gives you for automation — and when to reach for each one

From “Why” to “How”

In Chapter 1, you decided to stop doing things manually. You ran your automation audit. You know what to automate. Now you need to know what tools you have to do it.

Claude Code isn’t just a chat interface. It’s a full automation platform with five distinct building blocks, each designed for a different kind of automation. Think of them like tools in a physical workshop — you wouldn’t use a screwdriver to hammer a nail.

By the end of this chapter, you’ll know exactly which tool to reach for when you spot an automation opportunity.

Each layer builds on the ones before it. CLAUDE.md shapes how Claude Code behaves. Skills give you reusable commands. Hooks trigger actions automatically on events. Shell scripts combine Claude Code with traditional tooling. CI/CD runs it all in your pipeline without you present.

Let’s break each one down.

CLAUDE.md: Your Configuration Layer

Before you write a single skill or hook, you need to understand CLAUDE.md. It’s the simplest automation tool — and the one most people underuse.

CLAUDE.md is a markdown file that tells Claude Code how to behave in your project. It’s like a .editorconfig but for an AI assistant. Drop it in your repo root, and every Claude Code session in that project follows its rules.

What goes in CLAUDE.md?

  • Coding conventions — “Use Kotlin coroutines, not RxJava” or “All ViewModels extend BaseViewModel”
  • Architecture rules — “Follow Clean Architecture with domain/data/presentation layers”
  • Review standards — “Every public function needs KDoc” or “No nullable types in domain models”
  • Project context — “This is an Android app using Jetpack Compose, Hilt, and Room”
  • Forbidden patterns — “Never use GlobalScope” or “Don’t add new Gradle dependencies without checking with the team”

Example CLAUDE.md

# MyApp Android Project ## Tech Stack - Kotlin, Jetpack Compose, Hilt, Room, Retrofit - Min SDK 26, Target SDK 35 - Gradle with version catalogs ## Conventions - Use `StateFlow` for UI state, never `LiveData` - Repository pattern: every data source has a repository interface in `domain/` - Tests use JUnit 5 + MockK + Turbine for Flow testing - Compose previews required for every screen composable ## Rules - No `GlobalScope` — use `viewModelScope` or a supervised scope - No wildcard imports - All network calls go through the `ApiResult` wrapper - PR descriptions must include: what, why, testing steps

Every Claude Code session in this project will follow these rules automatically. No need to repeat yourself.

CLAUDE.md levels

CLAUDE.md files work at multiple levels, from broadest to most specific:

Location Scope Use for
~/.claude/CLAUDE.md All your projects Personal preferences, coding style, your role
/project/CLAUDE.md This project (checked in) Team conventions, architecture, tech stack
/project/.claude/CLAUDE.md This project (local only) Personal project notes, WIP context
/project/subdir/CLAUDE.md Specific directory Module-specific rules (e.g., API layer conventions)

CLAUDE.md is your cheapest automation. Every rule you put in CLAUDE.md is a rule Claude Code follows automatically in every session, for every team member. No skills, no hooks, no scripts — just a markdown file. Start here before building anything else. Most teams have ten rules they repeat in every code review. Put them in CLAUDE.md and they become automatic.

Skills: Your Custom /Commands

Skills are reusable prompt templates that live in your project (or your home directory). You invoke them with a slash — /pr-description, /generate-tests, /scaffold-feature. They’re the automation equivalent of bash aliases, but powered by AI.

How skills work

A skill is a markdown file in .claude/commands/. The filename becomes the command name. The file contents become the prompt.

# File: .claude/commands/pr-description.md

Read the current git diff (staged and unstaged changes).
Write a PR description with:
- A concise title (under 70 characters)
- A "## What" section explaining the changes
- A "## Why" section explaining the motivation
- A "## Testing" section with verification steps

Follow the PR template in CLAUDE.md.
Output the description in markdown.

Now you type /pr-description in Claude Code and get a complete PR body in seconds.

Project skills vs. user skills

Type Location Shared? Best for
Project skill .claude/commands/ Yes (checked into git) Team workflows, project-specific commands
User skill ~/.claude/commands/ No (personal) Personal shortcuts, cross-project utilities

Skill naming convention: Use lowercase, hyphenated names. Subdirectories help with organization — .claude/commands/android/scaffold-feature.md still invokes as /scaffold-feature, but Claude Code shows the subdirectory as a description suffix (e.g. /scaffold-feature (project:android)). Colon-namespaced commands like plugin-name:skill-name are reserved for plugins (see Chapter 13). Group related skills by category to keep things organized.

Skills 2.0: Anthropic has merged slash commands and the newer Skills feature into one system. The bare-markdown form above (.claude/commands/foo.md → /foo) still works exactly as shown. The recommended form for anything you want to share or model-invoke is now a directory with a SKILL.md file at .claude/skills/foo/SKILL.md, plus YAML frontmatter (name, description, allowed-tools) and any supporting files. Both forms produce the same /foo invocation. We cover the SKILL.md form in detail in Chapter 3.

We’ll build skills hands-on in Chapter 3. For now, just know: if you do it interactively more than twice, it should be a skill.

Hooks: Event-Driven Automation

Skills are great, but you still have to type the command. Hooks remove even that step. They fire automatically when Claude Code does something — edits a file, runs a command, finishes a task.

Hooks are configured in your settings.json file (either ~/.claude/settings.json for user-level or .claude/settings.json for project-level).

Hook types

Hook Type When It Fires Example Use
PreToolUse Before Claude Code uses a tool Block force-push to main, validate file paths
PostToolUse After Claude Code uses a tool Auto-lint after file edits, run tests after code changes
Notification When Claude Code wants to notify you Send a Slack message, play a sound
Stop When Claude Code finishes a task Generate a summary, log the session
SubagentStop When a subagent finishes Collect results from parallel agents
UserPromptSubmit Right after you press enter, before Claude Code processes the prompt Prompt rewriting, guardrails, injecting extra context
SessionStart When Claude Code opens a session (or resumes one) Setup automations — warm caches, export env vars, write a session log header
SessionEnd When the session terminates Teardown, summary reports, commit any session-state files
PreCompact Just before Claude Code compacts the conversation Snapshot state, pin critical context to disk before it gets summarized away

That’s the headline set. Claude Code also exposes more specialized hooks (PostToolUseFailure, PostToolBatch, PermissionRequest, TaskCreated, FileChanged, WorktreeCreate, and others) for advanced automation. The full list lives in the official hooks reference — but the eight above cover 90% of what you’ll build in Chapter 4.

Anatomy of a hook

// In .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash -c './scripts/lint-changed.sh \"$CLAUDE_FILE\"'"
      }
    ]
  }
}

This hook runs your linting script every time Claude Code edits or writes a file. No manual step. No forgetting. It just happens.

Lint Workflow Without Hooks

  1. Claude Code edits a Kotlin file
  2. You remember to run ./gradlew ktlintCheck
  3. Linter finds 3 issues
  4. You ask Claude Code to fix them
  5. You run the linter again to verify

5 steps, 3–5 minutes, easy to forget

Lint Workflow With Hooks

  1. Claude Code edits a Kotlin file
  2. Hook runs linter automatically
  3. If issues found, Claude Code sees them and fixes immediately

1 step (for you: zero), always consistent

Chapter 4 is dedicated entirely to hooks. For now, remember the pattern: hooks = “when X happens, automatically do Y.”

The Claude Code CLI: Your Automation Engine

Everything we’ve discussed so far works inside an interactive Claude Code session. But the real automation power comes when you use Claude Code non-interactively — from scripts, cron jobs, and CI/CD pipelines.

The key flag is -p (print mode). It sends a prompt, gets a response, and exits. No interactive session, no human in the loop.

# Generate a PR description from your current diff
claude -p "Read the git diff and write a PR description following our template"

# Explain a complex file for onboarding
claude -p "Explain what src/data/NetworkModule.kt does, for a new team member"

# Generate tests for a specific class
claude -p "Generate JUnit 5 + MockK tests for src/domain/usecase/SyncDataUseCase.kt"

Key CLI flags for automation

Flag What it does Automation use
-p "prompt" Non-interactive mode, prints result Scripts, CI/CD, cron jobs
--output-format json JSON output for machine consumption Piping to jq, parsing in scripts
--allowedTools Auto-approve specific tools without prompting CI runs that should proceed without human approval (e.g., "Read,Bash(git diff)")
--disallowedTools Remove specific tools from Claude Code’s context entirely Hard safety: forbid Bash(rm:*) or Bash(sudo:*) in any automated run
--model Choose the model Use faster models for simple tasks
-c Continue the last conversation Multi-step workflows

Two flags, two jobs. --allowedTools is about auto-approval — the listed tools run without a permission prompt, but other tools are still available (Claude Code will just ask). --disallowedTools is about availability — the listed tools are stripped from the model’s context entirely. Use the second one for hard safety boundaries; use the first for friction-free CI.

Installing Claude Code: The npm install (npm install -g @anthropic-ai/claude-code) still works, but Anthropic now ships a native installer as the canonical path — on most systems you can run claude install stable or use the platform-specific installer from the official setup guide. Both approaches give you the same claude binary; native is faster to update via claude update.

Piping and composition

Because claude -p reads from stdin and writes to stdout, you can compose it with standard Unix tools:

# Pipe git log into Claude Code for a changelog
git log --oneline v1.2.0..HEAD | claude -p "Write a user-facing changelog from these commits"

# Review only the changed files
git diff --name-only | claude -p "For each file listed, summarize what changed and flag potential issues"

# Chain with jq for structured output
claude -p "List the top 5 tech debt items in this repo as JSON" --output-format json | jq '.result'

Exit codes matter. claude -p returns exit code 0 on success and non-zero on failure. This means you can use it in bash conditionals: if claude -p "Are there any security issues in this diff?" | grep -q "CRITICAL"; then echo "Blocking merge"; fi

Shell Scripting Refresher

You don’t need to be a bash wizard, but you need the basics to wire Claude Code into your workflow. Here’s the minimum viable shell knowledge for automation:

Variables and arguments

#!/bin/bash
FILE="$1"                     # First argument passed to the script
BRANCH=$(git branch --show-current)
DIFF=$(git diff --cached)

# Pass variables into Claude Code
claude -p "Review this diff from branch $BRANCH: $DIFF"

Conditionals

# Only run if there are staged changes
if [ -n "$(git diff --cached)" ]; then
  claude -p "Review my staged changes for issues"
else
  echo "Nothing staged to review"
fi

The tools you’ll use most

  • jq — parse JSON output from Claude Code
  • git diff / git log — feed code context into prompts
  • xargs — run Claude Code on each file in a list
  • tee — save output to a file while also displaying it
  • pipes (|) — chain commands together

Watch your quoting. When passing code or diffs to claude -p, always use double quotes around variables ("$DIFF" not $DIFF). Unquoted variables with special characters will break your scripts in mysterious ways.

GitHub Actions: Automation in the Cloud

Everything we’ve covered so far runs on your machine. GitHub Actions takes it to the cloud — Claude Code running in your CI/CD pipeline, triggered by pushes, PRs, schedules, or webhooks.

Here’s the shape of a workflow that runs Claude Code on every PR:

# .github/workflows/ai-review.yml
name: AI Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Run AI Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          git diff origin/main...HEAD | claude -p \
            "Review this diff. Flag bugs, security issues, and style violations.
             Output as a markdown checklist." \
            > review.md

      - name: Post Review Comment
        uses: actions/github-script@v9
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: review
            });

The official action. Anthropic now publishes anthropics/claude-code-action@v1 as the recommended way to run Claude Code in GitHub Actions — it handles the install, the API key wiring, and the result-posting automatically. The hand-rolled npm install + claude -p pattern above still works (and is useful when you need raw control), but for most jobs the official action is shorter, safer, and self-updating. Chapter 7 covers it in depth.

We’ll build this out fully in Chapter 7. For now, just understand the pattern: GitHub Action triggers → Claude Code runs non-interactively → results posted back to GitHub.

The Right Tool for the Job

This is the decision framework you’ll use throughout the rest of the course. When you spot an automation opportunity, ask yourself:

The Automation Decision Tree

Is it a behavior rule? → Put it in CLAUDE.md

Is it a task you trigger manually? → Make it a skill

Should it happen automatically on an event? → Make it a hook

Does it combine Claude Code with other tools? → Write a shell script

Should it run without you present? → Put it in CI/CD or a scheduled agent

Let’s see this in action with real examples:

Automation Right Tool Why
“Always use Kotlin coroutines, not RxJava” CLAUDE.md It’s a rule, not an action
Generate PR description from diff Skill You trigger it when you’re ready to open a PR
Auto-lint after every file edit Hook Should happen every time, no manual trigger
Morning repo summary + standup draft Shell script (cron) Combines git log, Claude Code, and file output
Review every PR automatically CI/CD Runs on GitHub, no developer present
Weekly documentation freshness check Scheduled agent Runs on a schedule, reports results

Putting It All Together: The Automation Stack Diagram

Here’s how all five layers work together in a real workflow:

Example: A Complete Feature-to-PR Automation Stack

  1. CLAUDE.md defines your team’s coding standards, architecture rules, and PR template. Every Claude Code session follows these automatically.
  2. You code the feature with Claude Code, which follows CLAUDE.md rules as it writes.
  3. Hooks auto-lint and auto-format every file Claude Code touches. Tests run automatically after code changes.
  4. You run /pr-description (a skill) to generate the PR body from the diff.
  5. You push. A GitHub Action triggers Claude Code to review the PR, post comments, and check for security issues.
  6. A scheduled agent checks weekly whether the documentation is still in sync with the code you changed.

That’s the full stack. CLAUDE.md sets the rules. Skills handle on-demand tasks. Hooks handle real-time events. Shell scripts combine tools. CI/CD handles everything that runs without you.

Exercise: Set Up Your Automation Foundation

Exercise: Create Your Project’s CLAUDE.md and First Skill

Do this now — it takes 10 minutes and sets the foundation for everything in this course.

Part 1: Create CLAUDE.md

  1. Open your main project directory
  2. Create a CLAUDE.md file at the root
  3. Add your tech stack, coding conventions, and at least 5 rules that you’d normally repeat in code reviews
  4. Commit it to your repo so the whole team benefits

Quick start: Ask Claude Code to help you write it:

claude -p "Look at this project's code and generate a CLAUDE.md with:
- Tech stack summary
- Coding conventions you can infer from the code
- Architecture patterns in use
- At least 5 rules for code quality"

Part 2: Create Your First Skill

  1. Create the directory: mkdir -p .claude/commands
  2. Create .claude/commands/explain.md with this content:
Read the file at $ARGUMENTS.
Explain what it does in plain English, suitable for a new team member.
Include:
- Purpose of the file
- Key functions/classes and what they do
- How it fits into the broader architecture
- Any non-obvious patterns or gotchas
  1. Test it: open Claude Code and type /explain src/main/java/com/myapp/data/NetworkModule.kt
  2. If it works, commit it. Your whole team now has this command.
