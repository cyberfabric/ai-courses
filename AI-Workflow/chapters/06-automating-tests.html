Automating Tests with AI

Generate, maintain, and improve your test suite — without the dread

Why Nobody Writes Enough Tests

Be honest. When was the last time you looked at a class, thought “I should write tests for this,” and actually did it right then? Not later. Not “before the PR.” Right then.

Testing is the broccoli of software development. Everyone knows it’s good for you. Nobody’s excited about it. And the gap between “we should have tests” and “we have tests” is usually measured in sprints.

40% of developers say they skip tests due to time pressure
15 min average time to write a decent unit test by hand
30 sec for Claude Code to generate a comprehensive test

The problem isn’t laziness. It’s friction. Writing a test means: figure out the testing pattern for this type of code, set up mocks, handle coroutine dispatchers, deal with Hilt test rules, and write assertions for all the edge cases. By the time you’ve done the setup, you’ve lost your flow.

AI eliminates the friction. Claude Code reads the source, understands the patterns, generates tests that match your project’s conventions, and gets it done in seconds. Let’s build this.

Quick Test Generation from the CLI

The simplest approach — and a great starting point:

# Generate tests for a specific file
claude -p "Generate comprehensive unit tests for src/domain/usecase/SyncDataUseCase.kt.
Use JUnit 5, MockK for mocking, and Turbine for Flow testing.
Follow the patterns in the existing test files in this project."

# Generate tests for everything that changed
git diff main --name-only -- '*.kt' | grep -v Test | while read f; do
  claude -p "Generate unit tests for $f using the project's testing patterns"
done

This works, but it’s ad-hoc. Let’s make it a proper, reusable skill.

Building the /generate-tests Skill

Creating a Comprehensive /generate-tests Skill

  1. Create the file: .claude/commands/generate-tests.md
  2. Add this content:
# Generate Tests

Generate comprehensive unit tests for: **$ARGUMENTS**

## Steps
1. Read the source file to understand its behavior
2. Read CLAUDE.md for testing conventions
3. Find existing test files in the project to match patterns
4. Identify the testing framework in use (check build.gradle.kts)

## Test Structure
- One test class per source class
- Use descriptive test names: `should do X when Y`
- Group tests by function using @Nested classes
- Use the Arrange-Act-Assert pattern

## What to Test
- **Happy path**: normal inputs produce expected outputs
- **Edge cases**: null, empty, zero, boundary values
- **Error paths**: exceptions, error states, network failures
- **State transitions**: for ViewModels, test each state change

## Android-Specific Patterns
- **ViewModels**: Use `runTest`, `Turbine` for StateFlow assertions,
  `StandardTestDispatcher` for coroutine control
- **Repositories**: Mock the data sources, test mapping logic,
  verify error wrapping with ApiResult
- **Use Cases**: Test business logic in isolation, mock repositories
- **Compose**: Use `createComposeRule()` for UI tests,
  test state rendering and user interactions

## Rules
- Match the testing framework already in the project
- Use MockK for mocking (not Mockito)
- Use Turbine for Flow/StateFlow testing
- Don't test private functions — test through public API
- Don't test simple getters/setters or data classes
- Add a comment at the top: "Generated by /generate-tests — review before committing"
  1. Test it: /generate-tests src/feature/home/HomeViewModel.kt

Writing Tests Manually

  1. Open the source file, study the logic
  2. Create a test file, set up the class
  3. Figure out the MockK syntax for this case
  4. Set up coroutine test dispatcher
  5. Write the happy path test (5 min)
  6. Think about edge cases (3 min each)
  7. Handle error cases (3 min each)
  8. Run and debug (5 min)

Time: 25–45 minutes per class. Often skipped.

With /generate-tests

  1. Run /generate-tests HomeViewModel.kt
  2. Review the generated tests (2 min)
  3. Tweak any edge cases (2 min)
  4. Run the suite to verify (1 min)

Time: 5 minutes per class. Never skipped.

Android-Specific Test Patterns

Android testing has its own quirks. Here’s what Claude Code generates for each layer when your skill is well-configured:

ViewModel tests

@OptIn(ExperimentalCoroutinesApi::class)
class HomeViewModelTest {

    private val testDispatcher = StandardTestDispatcher()
    private val repository = mockk<HomeRepository>()
    private lateinit var viewModel: HomeViewModel

    @BeforeEach
    fun setup() {
        Dispatchers.setMain(testDispatcher)
        viewModel = HomeViewModel(repository)
    }

    @AfterEach
    fun tearDown() {
        Dispatchers.resetMain()
    }

    @Nested
    inner class LoadData {
        @Test
        fun `should emit Loading then Success when repository returns data`() = runTest {
            // Arrange
            coEvery { repository.fetchItems() } returns ApiResult.Success(testItems)

            // Act & Assert
            viewModel.uiState.test {
                assertEquals(HomeUiState.Loading, awaitItem())
                viewModel.loadData()
                assertEquals(HomeUiState.Success(testItems), awaitItem())
            }
        }

        @Test
        fun `should emit Error when repository fails`() = runTest {
            // Arrange
            coEvery { repository.fetchItems() } returns ApiResult.Error("Network error")

            // Act & Assert
            viewModel.uiState.test {
                skipItems(1) // Skip initial Loading
                viewModel.loadData()
                val state = awaitItem()
                assertIs<HomeUiState.Error>(state)
                assertEquals("Network error", state.message)
            }
        }
    }
}

Show Claude Code your existing tests. The best way to get tests that match your patterns is to have existing tests in the project. Claude Code reads them and follows the same setup, naming conventions, and assertion style. Even two or three well-written test files set the pattern for the whole project.

Finding Coverage Gaps

Writing tests for new code is one thing. But what about the existing code that never got tested? Build a coverage gap detector:

# File: .claude/commands/find-coverage-gaps.md

Analyze test coverage gaps in the project.

## Steps
1. List all Kotlin source files in src/main (excluding generated code)
2. List all test files in src/test
3. Match source files to their test files by naming convention
4. Identify source files with NO corresponding test file
5. For source files WITH tests, read both and check if major
   public functions are covered

## Output
### Untested Files (no test file exists)
- List each file with a brief description of what it does
- Prioritize by risk: ViewModels and UseCases first, then Repositories, then UI

### Partially Tested Files (test exists but gaps found)
- List each file with the specific untested functions
- Note which functions have complex logic that most needs testing

### Summary
- Total source files: X
- Files with tests: Y (Z%)
- Suggested next 5 files to test (highest impact)

Run coverage gap detection weekly. Set up a cron job or scheduled agent (Chapter 7) that runs /find-coverage-gaps every Monday morning and posts the results to your team channel. Visibility creates accountability — when everyone sees the coverage gaps, they get fixed faster. Pair it with /generate-tests to close gaps in minutes.

Maintaining Tests When Code Changes

Tests break when code changes. That’s normal. But manually updating five test files after a refactor is painful enough that some developers skip the tests and just delete the broken ones. Don’t be that developer.

# File: .claude/commands/update-tests.md

Update tests to match recent code changes.

## Steps
1. Run `git diff main...HEAD -- '*.kt'` to see all changes
2. For each changed source file, find its corresponding test file
3. Read both the source and test files
4. Identify tests that are now broken or outdated because of the changes
5. Update the tests to match the new code behavior

## Rules
- Don't delete tests — update them to match new behavior
- If a function was renamed, rename the test
- If a function's signature changed, update the test setup
- If new behavior was added, add new test cases for it
- If behavior was removed, remove the corresponding tests
- Keep the same testing patterns and style
- Run the updated tests and fix any failures

After a refactor, just run /update-tests and the test suite is current again.

Flaky Test Diagnosis

Flaky tests — tests that pass sometimes and fail sometimes — are one of the most frustrating problems in development. They erode trust in your test suite and waste hours of debugging time.

# File: .claude/commands/diagnose-flaky.md

Diagnose and fix the flaky test: **$ARGUMENTS**

## Steps
1. Read the test file
2. Read the source code it tests
3. Analyze for common flakiness causes:

### Timing Issues
- Race conditions in coroutine tests (missing advanceUntilIdle?)
- Hardcoded delays instead of proper synchronization
- Tests depending on execution speed

### State Issues
- Shared mutable state between tests (missing @BeforeEach reset?)
- Tests that depend on execution order
- Static/singleton state not cleaned up

### External Dependencies
- Network calls in unit tests (should be mocked)
- File system access without cleanup
- Database state leaking between tests

### Android-Specific
- Missing Dispatchers.setMain/resetMain
- Compose test rule timing issues
- LiveData not observed (use InstantTaskExecutorRule)

## Output
- **Root cause**: What's causing the flakiness
- **Fix**: Exact code changes to make it stable
- **Prevention**: Pattern to follow to avoid this in future tests

The Flaky Test Triage Workflow

When CI fails with a flaky test: (1) run /diagnose-flaky TestClassName#testName, (2) apply the fix, (3) run the test 10 times locally to verify stability: for i in {1..10}; do ./gradlew test --tests "*TestName" || break; done. If it passes all 10, it’s stable.

Mutation Testing: Are Your Tests Actually Good?

Having tests isn’t enough. Tests that pass no matter what the code does are worse than useless — they give false confidence. Mutation testing checks if your tests actually catch bugs.

The concept: change the source code slightly (a “mutation”), then run the tests. If the tests still pass, they’re not catching the kind of bug that mutation represents.

# File: .claude/commands/mutation-check.md

Check test quality for: **$ARGUMENTS** using mutation analysis.

## Steps
1. Read the source file
2. Read its test file
3. Identify 5-8 meaningful mutations:
   - Change `>` to `>=` in conditionals
   - Replace `true` with `false` in boolean returns
   - Remove null checks
   - Change `+` to `-` in calculations
   - Remove function calls (especially side effects)
   - Change `&&` to `||` in compound conditions

For each mutation:
4. Would the existing tests catch this change?
5. If not, what test is missing?

## Output
### Mutations Caught (tests would fail) ✓
- List each mutation and which test catches it

### Mutations Survived (tests would NOT fail) ✗
- List each mutation
- Why it survived (missing test case)
- Suggest a specific test to add

### Test Quality Score
X out of Y mutations caught = Z% mutation score

Don’t aim for 100% mutation score. Some mutations are trivial (logging changes, cosmetic stuff). Focus on mutations in business logic, error handling, and security checks. A 70–80% mutation score on critical code is excellent.

Test Quality Review

Just like production code gets reviewed, test code should too. Build a specialized review skill:

# File: .claude/commands/review/tests.md

Review the test code in the current diff.

## Check for
- **False positives**: Tests that pass regardless of implementation
  (e.g., assertions on mock return values instead of behavior)
- **Missing edge cases**: Null inputs, empty collections, boundary values
- **Brittle tests**: Tests coupled to implementation details that will
  break on refactor without indicating a real bug
- **Incomplete mocking**: Mocks that don't verify important interactions
- **Readability**: Can someone understand what's being tested from the
  test name and structure alone?
- **Test isolation**: Each test should pass independently, in any order

## Android-Specific
- Coroutine tests: using runTest? Proper dispatcher management?
- Flow tests: using Turbine? Testing emissions in order?
- Compose tests: using proper finders and assertions?

Report findings with the same severity format as /review.

The Testing Automation Pipeline

Here’s how all the testing pieces chain together into a pipeline:

Code Change
→
/generate-tests
→
/review:tests
→
Run Suite
→
Coverage Check
  • Code change — you write or modify production code
  • /generate-tests — generates or updates tests for the changed files
  • /review:tests — validates the generated tests are actually good
  • Run suite — execute the tests (hook from Chapter 4 can do this automatically)
  • Coverage check — CI verifies coverage didn’t drop

With the hook from Chapter 4 running tests on every edit, and /generate-tests handling test creation, you go from “tests are a chore” to “tests just happen.”
