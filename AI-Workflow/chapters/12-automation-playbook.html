Your Automation Playbook

A strategy for continuous automation improvement — and your 30-day challenge

You Have the Tools. Now What?

Over eleven chapters, you’ve built an automation arsenal: skills that generate code, tests, and PR descriptions. Hooks that lint, format, and test on every edit. CI/CD workflows that review PRs and write changelogs. Productivity agents for standups and digests. MCP servers that connect Claude Code to your internal tools. Multi-step pipelines that handle entire processes end-to-end.

That’s a lot of capability. But capability without strategy is just a collection of toys. This final chapter gives you the strategy: how to decide what to automate next, how to maintain what you’ve built, how to bring your team along, and how to keep improving.

Think of this as your automation operating manual — the playbook you’ll reference long after you finish this course.

The Automation Audit (Revisited)

In Chapter 1, you ran your first automation audit. Now you have the experience to do it properly. The audit is something you should repeat quarterly — your workflow changes, new tools appear, and new automation opportunities emerge.

The Quarterly Automation Audit

  1. Track for one week. Every time you do something repetitive, log it: what it was, how long it took, how often you do it.
  2. Review existing automations. Are they still working? Still needed? Are any broken or outdated?
  3. Identify new candidates. What repetitive tasks appeared since the last audit?
  4. Score each candidate on the effort-vs-impact matrix (see below).
  5. Pick your top 3. Add them to your automation backlog.
  6. Retire what’s broken. Delete automations that no longer serve a purpose. Dead code applies to automation too.

The Effort-vs-Impact Matrix

Not all automations are created equal. Use this matrix to decide what to build next:

Low Effort (< 30 min) High Effort (> 2 hours)
High Impact (daily use, saves 5+ min) Do first. Quick wins with big payoff. Shell aliases, simple skills, CLAUDE.md rules. Do second. Worth the investment. CI/CD workflows, MCP servers, multi-step pipelines.
Low Impact (weekly use, saves < 2 min) Do when bored. Low-cost experiments. Maybe they’ll surprise you. Don’t bother. The effort exceeds the lifetime savings. Just do it manually.

Start in the top-left corner. Low effort, high impact. Always. It’s tempting to build the flashy multi-repo pipeline first, but three simple skills that save you 5 minutes each day are worth more than one complex pipeline you use monthly.

Four flavors of customization in .claude/. commands/ holds bare-markdown slash commands (user-invoked). skills/<name>/SKILL.md packages the same idea with YAML frontmatter so Claude can invoke it on its own when the description matches. agents/<name>.md defines subagents with their own context window, tool allowlist, and model. plugins/ is where installed plugins land — a plugin can bundle any combination of the other three plus hooks and MCP servers.

Document your automations with a skill. Create /list-automations that scans your project for all skills, hooks, scripts, MCP servers, and CI/CD workflows, and generates a summary. When a new team member asks “what automations do we have?” the answer is one command away.

Maintenance: Keeping Automations Alive

Automations rot just like documentation. Your codebase changes, APIs evolve, tools update. An automation that worked last month might break silently next month.

  • Run the quarterly audit. Review every automation. Delete the dead ones. Update the stale ones.
  • Monitor CI/CD automations. Check your GitHub Actions run history monthly. Failed runs mean broken automations.
  • Version your skills. They’re in git. Use meaningful commit messages when you update them so you can understand the history.
  • Test MCP servers after upgrades. When the MCP SDK or your internal tools update, verify your servers still work.
  • Keep hook scripts fast. Performance degrades over time as projects grow. Re-benchmark your hooks quarterly.

The “automate and forget” anti-pattern. The most dangerous automation is the one you set up, forgot about, and assumed was still working. A broken weekly maintenance agent is worse than no agent — it gives you false confidence that someone is watching. Monitor your automations or they’ll eventually lie to you.

Measuring ROI

How do you know if your automations are actually worth it? Measure three things:

Time Minutes saved per day/week. Track in your automation log.
Errors Bugs caught by automated review that humans missed before.
Velocity PRs merged per week, time from PR open to merge. 
# Add to your automation scripts for time tracking
START=$(date +%s)
# ... automation runs ...
END=$(date +%s)
DURATION=$((END - START))
echo "$(date +%Y-%m-%d),automation-name,$DURATION" >> ~/.claude/automation-times.csv

After a month, run a quick analysis:

claude -p "Read ~/.claude/automation-times.csv.
Calculate:
1. Total time spent running automations
2. Estimated time saved (compare to manual estimates)
3. Net time saved
4. Top 3 most time-saving automations
5. Any automations that take too long and should be optimized"

Team Adoption Playbook

Your automations are only as valuable as the number of people using them. Here’s how to introduce automation to your team without resistance:

Phase 1: Lead by example (weeks 1–2)

  • Set up CLAUDE.md and a few key skills in your project repo
  • Use them yourself visibly — in PRs, in standups, in code reviews
  • When someone says “nice PR description,” mention “thanks, /pr-description wrote it”

Phase 2: Share the quick wins (weeks 3–4)

  • Demo 2–3 automations in a team meeting (5 minutes max)
  • Focus on the ones that solve pain everyone feels (PR descriptions, test generation)
  • Commit the skills to the repo so everyone gets them immediately

Phase 3: Invite contributions (weeks 5+)

  • Ask teammates to suggest automations: “What do you do repeatedly that annoys you?”
  • Pair with someone to build their first skill
  • Celebrate when someone adds a new skill to the shared library

Introducing Automation Badly

  1. “We should all use these 15 new automation tools I built”
  2. Team feels overwhelmed and pressured
  3. Nobody uses any of them
  4. Automations rot and get deleted

Result: wasted effort, team resistance.

Introducing Automation Well

  1. You use /pr-description and people notice better PRs
  2. “How did you do that?” — “It’s a skill, you have it too”
  3. One teammate tries it, likes it, tells another
  4. Team organically starts contributing their own skills

Result: natural adoption, shared ownership.

Automation Anti-Patterns

After eleven chapters of “automate everything,” here’s the counterpoint. These are the ways automation goes wrong:

Over-automation

Automating things that don’t need automation. If you spend an hour building a skill you’ll use once, you lost time. Apply the effort-vs-impact matrix honestly.

Brittle pipelines

A 10-step pipeline where any step can fail silently. Build error handling, add monitoring, and keep pipelines as short as possible. Three reliable steps beat ten fragile ones.

Automate and forget

Setting up a scheduled agent and never checking if it’s still running. Broken automations give false confidence. Monitor everything.

Premature automation

Automating a process that’s still changing. Wait until the process stabilizes before investing in automation. Use interactive Claude Code sessions for evolving workflows.

Security shortcuts

Giving MCP servers more access than they need. Skipping input validation. Storing secrets in code. Connecting to production databases. Security is never worth sacrificing for convenience.

Staying Current

Claude Code evolves. New features appear, APIs change, capabilities expand. Stay current without making it a full-time job:

  • Check release notes monthly. New hooks, new CLI flags, new MCP capabilities — each one might enable an automation you couldn’t build before.
  • Follow the community. Other developers share skills and MCP servers. Adapt their work instead of building from scratch.
  • Revisit CLAUDE.md quarterly. Your project’s conventions evolve. Keep CLAUDE.md in sync or it becomes misleading.
  • Test after updates. When Claude Code updates, run your key automations to verify they still work as expected.

The next horizon

This course covered the foundation. The frontier keeps moving — here are the five surfaces worth tracking right now:

🧩
Plugins & the marketplace
A plugin packages slash commands, skills, hooks, MCP servers, and subagents into one installable bundle. Add a marketplace with /plugin marketplace add user/repo, then /plugin install <name>. Manifest lives at .claude-plugin/plugin.json; marketplaces declare their catalog in .claude-plugin/marketplace.json. This is the answer to “how do I share my whole automation stack with the team?”
🤖
Subagents
Specialized agents in .claude/agents/<name>.md with their own context window, system prompt, tool allowlist, and model. Claude delegates to them by description, or you invoke them with @agent-name. Reach for them when a step would otherwise flood your main context with logs or search results — code-reviewer, test-runner, security-auditor are the classic three.
📦
Skills (SKILL.md)
.claude/skills/<name>/SKILL.md with YAML frontmatter (name, description, allowed-tools) and supporting files. Unlike bare slash commands, Skills are model-invokable — Claude picks them up on its own when your description matches the situation, with progressive disclosure of supporting files.
🌐
Claude Code on the Web
Claude Code now runs in a managed sandboxed environment at claude.ai/code, plus the Anthropic mobile app. Sessions persist across surfaces; long-running automations can move off your laptop entirely. Useful for cron-style agents you want running without “is my machine awake?” as a dependency.
🛠️
Claude Agent SDK
The Claude Code SDK was renamed to the Claude Agent SDK in early 2026 to reflect its broader scope. It’s the programmatic foundation for embedding Claude Code’s agent loop in your own apps and services — same primitives (tools, subagents, hooks), exposed as a library. Worth a look once your automations grow beyond Claude Code itself.

The Finish Line (That’s Actually a Starting Line)

You started this course as a developer who uses AI to write code. You’re ending it as a developer who uses AI to run your entire workflow.
