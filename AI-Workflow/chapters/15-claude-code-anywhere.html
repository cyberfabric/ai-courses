Chapter 15

Claude Code on the Web and Mobile

Your automations don’t have to live on the laptop you’re typing on — sandboxed cloud sessions, mobile dashboards, and surfaces that survive a lid close.

The Surface Was the Limitation

Every chapter so far has quietly assumed the same setup: you, sitting at a laptop, running claude in a terminal. That’s the canonical Claude Code surface, and it’s great. But it’s also the bottleneck for a lot of automation patterns we covered: the weekly maintenance agent (Ch 11) only runs while your machine is awake; the build-failure diagnoser (Ch 7) needs CI infrastructure separate from Claude itself; the morning-digest skill (Ch 9) is useless on the train when your laptop is closed.

Claude Code now runs in three places, and they’re designed to interoperate. This chapter is about where your automations run — and how that choice changes the design space.

The Three Surfaces

💻
Terminal (the original)
The claude CLI on your machine. Full filesystem access. Best for active development, hooks, and anything that needs to read or modify files outside the repo. Subagents, skills, and plugins all started here.
🌐
Web — claude.ai/code
Anthropic-managed sandboxed cloud. Each session runs in a fresh VM with your repo cloned, common toolchains pre-installed (Python, Node, Java, Go, Rust, Ruby, PHP, C++, Docker, Postgres, Redis), and persistent state across browser closes. In research preview for Pro, Max, Team, and Enterprise users.
📱
Mobile (Anthropic app)
Monitor cloud sessions, kick off new ones, and approve PR fixes from your phone. Pairs with Remote Control to monitor a local session running on your machine. Closing your laptop doesn’t kill anything — you just check it from the bus.

The surfaces aren’t silos. Sessions move between them with two distinct primitives that are easy to confuse.

--remote vs. --teleport vs. Remote Control

These three names sound interchangeable. They aren’t. Read this table once and you’ll save yourself an hour of confusion.

PrimitiveDirectionUse it when
claude --remote "..." Terminal → cloud (creates a new cloud session) You want to fire-and-forget a task. Push first, then claude --remote "fix the failing tests". The cloud VM clones your branch, runs the work, you check progress with /tasks.
claude --teleport / /teleport Cloud → terminal (pulls an existing cloud session local) A cloud session has done good work and you want to continue locally. Picks the session, fetches its branch, loads conversation history. One-way: you can’t push a terminal session back to the cloud.
/remote-control Local CLI → phone/web (read-only mirror) You started a local session and want to monitor it from your phone or claude.ai. The work still runs on your laptop — you just see and steer it remotely. Auto-reconnects if the network drops.

The trap to avoid. --remote creates a new cloud session; --remote-control exposes a local session. They share six letters and do opposite things. If you only remember one rule: "remote = cloud, remote-control = phone."

Sessions Survive Surfaces

Cloud sessions persist independently of the device that started them. Close your browser, switch laptops, hand off to a teammate — the session is still there. This is genuinely new behavior; pre-2026, “closing the lid” meant “killing the agent.”

Two distinct “resume” verbs to know:

  • claude --resume — reopens a conversation from this machine’s local history. Cloud sessions don’t appear here.
  • claude --teleport — pulls a cloud session into your terminal. Local history isn’t listed.

Inside an existing CLI session, /teleport (or the alias /tp) opens the same picker without restarting Claude Code. /tasks lists every running cloud session you own and lets you press t to teleport into one directly.

What the Cloud VM Has — and Doesn’t

Every cloud session starts in a fresh VM. The repo is cloned; common runtimes are pre-installed. Configuration that lives in the repo carries over. Configuration that lives on your machine doesn’t.

Comes alongStays home
CLAUDE.md (in repo) ~/.claude/CLAUDE.md (your user-level)
.claude/settings.json hooks User-level hooks in ~/.claude/settings.json
.claude/skills/, .claude/agents/, .claude/commands/ User-scoped plugins, MCP servers added with claude mcp add
.mcp.json at repo root Static API tokens (no secrets store yet)
Plugins declared in .claude/settings.json Interactive auth flows (AWS SSO, etc.)

Every primitive you spent the last 14 chapters learning about — skills, hooks, subagents, plugins, MCP servers — works in the cloud as long as it’s committed to the repo. The cloud sandbox is a strong forcing function: if your automation lives in ~/.claude/, it’s yours alone; if it lives in .claude/ in the repo, the whole team gets it everywhere.

The cloud is the team-sharing test. If your team’s most important automation doesn’t work in a fresh cloud session, it doesn’t really work for new hires either. Run a cloud session occasionally as a sanity check — if your reviewer subagent fires correctly there, it’ll fire correctly for the engineer who joined yesterday.

The Sandbox: What’s Isolated, What’s Allowed

Sandboxing is the headline security feature, and it changes how you design hooks and tool-using automations.

The sandbox is built on OS-level primitives: Linux bubblewrap, macOS seatbelt. It enforces two boundaries:

  • Filesystem. Default writes go only to the current working directory and its subdirectories. Reads are broader by default. You can tighten or loosen via /sandbox in the CLI.
  • Network. Outbound traffic flows through a proxy with one of four levels — None, Trusted (curated allowlist of package registries, GitHub, cloud SDKs — the default), Full, or Custom (your own list).
  • Permission prompts. Anthropic reports an 84% reduction in permission interruptions in their internal usage. Hooks that previously fired a confirmation per shell command now batch into a single “OK to proceed in this sandbox?”

Sandboxes are not bulletproof. CVE-2026-39861 (March 2026) demonstrated a sandbox escape via symlinks; a separate write-up showed Claude Code’s own denylist could be bypassed with path tricks like /proc/self/root/usr/bin/npx. The team patched both, and the sandbox is genuinely safer than running everything raw — but it’s a defense in depth layer, not a substitute for not running untrusted code. Treat it like AppArmor: useful, not magic.

When Cloud, When Local

Both surfaces work for most tasks. The decision is pragmatic:

Choose cloud when…Choose local when…
The task is long-running and you want to close your laptop. You’re in active development and need fast iteration on the same files.
The work needs CPU/RAM your laptop hates (4 vCPU, 16 GB RAM, 30 GB disk per cloud session). You need access to local files outside the repo, or to apps your laptop has installed (Xcode, Android Studio).
You’re running multiple parallel tasks (each --remote spawns its own VM). You need interactive auth (AWS SSO, browser-based vendor login).
You want a clean reproducible environment that the rest of the team will get too. You’re iterating on hook scripts that need to be tested with your local file watcher.

You don’t pick once and stick with it. The two surfaces compose: plan locally, execute remotely; investigate remotely, fix locally.

Setup Scripts vs. SessionStart Hooks

There’s a subtle distinction worth getting right because it bites people the first time. Both run when a session starts; they belong to different layers.

Setup scriptSessionStart hook
Lives in Cloud environment UI (claude.ai) .claude/settings.json in your repo
Runs in Cloud sessions only Both local and cloud
Cached? Yes — setup script result is snapshotted for ~7 days No — runs every session start and resume
Use for apt install gh, fetching cloud-specific tools npm install, project-level setup for everyone

For cloud-only behavior in a SessionStart hook, gate on the $CLAUDE_CODE_REMOTE env var:

#!/bin/bash
if [ "$CLAUDE_CODE_REMOTE" != "true" ]; then
  exit 0
fi
npm install
pip install -r requirements.txt

Mobile Use Cases

You don’t need a phone keyboard for serious coding work. What you need is a phone view into things that are running elsewhere.

  • Monitor cloud sessions. The Anthropic app lists every running session, with status and recent output. Tap one to step through Claude’s reasoning, leave a comment, or approve the next step.
  • Approve auto-fix actions. If a cloud session is auto-fixing a PR (more on this in a moment) and Claude wants to make a non-trivial change, you can approve from the phone instead of waiting until you’re back at your laptop.
  • Kick off long-running work. “Start a cloud session that runs the full migration; tell me when it’s done.” The session runs while you’re in a meeting; the notification arrives when it lands.
  • Remote Control your laptop. The session runs on your machine but you steer from the phone — useful for kicking a stuck workflow without finding a desk.

Routines: Cron Without Your Laptop

The hardest part of Chapter 11’s scheduled-agent pattern was keeping the laptop awake. Routines move the cron loop into the cloud. Define a prompt, attach it to a repo and an environment, set a schedule — and a fresh cloud session fires that prompt on cadence, with full toolchain access, no laptop required.

# In your terminal, while inside Claude Code
/schedule

# Or define a routine that fires on a GitHub event:
# - new PR opened → spawn a /review session
# - issue labeled "claude-investigate" → spawn a research session
# - daily at 9am UTC → spawn the morning-digest session

Routines are the cloud equivalent of every “weekly maintenance script” you wrote in Chapter 11, except they run on Anthropic’s infrastructure and report back via mobile notifications. They’re the answer to “how do I run automation that doesn’t depend on whether my MacBook is plugged in?”

Auto-Fix: PR Maintenance Without You

Auto-fix is the cloud feature most worth knowing for a course about automation. It lets a cloud session subscribe to a PR’s GitHub events — CI failures, review comments — and respond automatically.

  • Reviewer leaves a comment with a clear suggestion: Claude makes the change, pushes, replies on the comment.
  • CI fails on the new branch: Claude reads the logs, attempts a fix, pushes again.
  • Comment is ambiguous or architectural: Claude asks before acting.

Turn it on three ways: (1) on a PR created in the web UI, click the CI status bar → Auto-fix; (2) from your terminal on a PR’s branch, run /autofix-pr; (3) from the mobile app, just tell Claude “watch this PR and fix any CI failures or review comments.”

Caution with comment-triggered automation. If your repo uses Atlantis, Terraform Cloud, or a similar bot that runs on issue_comment events, Claude’s auto-replies can trigger your downstream automation. Review what your issue_comment workflows actually do before enabling auto-fix on infra repos. The integration is convenient; the blast radius is real.

Combining Cloud Claude Code With the GitHub Action

You met anthropics/claude-code-action@v1 in Chapter 7. It runs Claude Code inside GitHub Actions runners. Cloud Claude Code is similar but different: it runs on Anthropic-managed infrastructure with a richer toolchain pre-installed.

Use the GitHub Action when…Use Cloud Claude Code when…
The work is part of your CI pipeline (changelog, AI review, PR description). The work is one-off or interactive (auto-fix, ultraplan, exploratory tasks).
You need GitHub-native triggers (push, PR open, schedule). You want a chat-driven session you can monitor from anywhere.
You’re on a free GitHub Actions tier and prefer to use those minutes. You’re on Pro/Max/Team/Enterprise and your laptop can’t spare the cycles.

They’re complementary. Many teams use both: Action for routine CI work, Cloud Claude Code for ad-hoc heavy lifting.

Before: laptop-bound automation

  1. Reviewer leaves a comment on your PR while you’re commuting.
  2. CI starts failing twenty minutes later.
  3. You see both at your desk three hours from now.
  4. You spend the first 90 minutes back catching up to the PR you thought was nearly done.

Workflow paused. Context lost. Day spoiled.

After: cloud + auto-fix + mobile

  1. Auto-fix is enabled on the PR (one click, before you left the desk).
  2. Comment arrives: cloud session reads it, makes the suggested change, pushes.
  3. CI fails on a flaky test: cloud session retries, fix doesn’t land, leaves a note.
  4. You arrive home, glance at the phone, see “5 of 6 issues resolved; 1 needs your call.”

Workflow continued without you.

What Doesn’t Work in the Cloud

The cloud is great. It is not magical. Things that genuinely don’t work or work badly:

  • Interactive auth flows — AWS SSO, browser-based vendor logins. The VM has no browser to redirect to. Use static tokens via env vars instead, with the visibility caveat (env vars aren’t a real secrets store yet).
  • Memory-heavy builds — 16 GB RAM and 4 vCPUs is plenty for most projects but not for a Bazel-on-monorepo job. For workloads beyond cloud limits, run via Remote Control on your own hardware.
  • Custom Docker base images — you can’t replace the cloud VM’s base image. Use a setup script to install on top, or run your image as a sidecar with docker compose.
  • User-only configuration — if it’s in ~/.claude/ instead of the repo, the cloud doesn’t see it. Move things to .claude/ in the repo to bring them along.
  • Bun — known proxy compatibility issues for package fetching. Other Node tooling works fine; Bun specifically struggles with the security proxy.

Recipe: Move Your Weekly Maintenance Agent to the Cloud

From cron-on-laptop to scheduled cloud session

  1. Audit what’s in ~/.claude/. Anything your weekly maintenance script depends on — CLAUDE.md rules, hooks, MCP servers — needs to live in the repo, not your home dir.
  2. Commit it. Move user-level CLAUDE.md content to repo CLAUDE.md (or a .claude/rules/ file). Re-declare MCP servers in .mcp.json. Re-declare plugins in .claude/settings.json’s enabledPlugins.
  3. Configure the cloud environment. On claude.ai/code, add an environment named “maintenance.” Set the network access level to Trusted (the default) plus any custom domains your scripts need. Add a setup script if you need anything beyond pre-installed tooling.
  4. Set up a routine. Use /schedule in the terminal (or the routines UI on claude.ai) to schedule the maintenance prompt to run weekly against your repo. This is the cloud equivalent of cron-on-laptop — only the laptop isn’t involved.
  5. Test once interactively. Run claude --remote "run the weekly maintenance routine on this repo". Watch progress with /tasks. Fix anything that breaks because of the cloud / local difference (paths, missing local-only deps).
  6. Switch off the laptop cron. Once the cloud routine has run cleanly twice, retire the local cron job. Your laptop no longer has to be awake on Sundays.
  7. Add mobile monitoring. Open the Anthropic app, tap the maintenance session whenever you remember. If it ever stops succeeding silently, you’ll see it during a 10-second phone glance.

Exercise: One Cloud Session, Three Surfaces

The point of this exercise is to feel session continuity across surfaces. 30 minutes.

  1. Start in the terminal. Pick a real-but-low-stakes task (“modernize this README,” “add JSDoc to utils/”). Push your branch first. Run claude --remote "<the task>".
  2. Check progress on web. Open claude.ai/code. Find your session in the sidebar. Watch a few steps of reasoning. Leave one inline comment on the diff.
  3. Check from mobile. Open the Anthropic app. Find the same session. Read the latest message. Verify it matches what you saw on web.
  4. Pull it home. When the session is in a good state, run claude --teleport from your terminal. Pick the session. Confirm Claude fetched the branch, loaded conversation history, and is ready for local work.
  5. Continue locally. Make one small follow-up edit interactively. Push. Mark the PR ready for review.
  6. Bonus: turn on auto-fix on the PR. Leave a deliberate review comment that asks for a clear, narrow change. Watch the cloud session pick it up and push the fix.

The Big Idea

Three surfaces, one session model. --remote creates cloud sessions; --teleport pulls them home; Remote Control mirrors a local session to your phone. Everything you committed to .claude/ in the repo — skills, hooks, subagents, plugins, MCP servers — runs identically across surfaces. The cloud isn’t a different kind of Claude Code; it’s the same Claude Code somewhere else, with a sandbox and 4 vCPUs and a phone-shaped window into it.

Fifteen chapters in, you’ve built every layer of the modern Claude Code automation stack: rules in CLAUDE.md, on-demand skills, event-driven hooks, AI-augmented code review, AI-generated tests, CI/CD that thinks, self-writing docs, personal productivity agents, MCP servers for your team, multi-step orchestration, a written automation playbook, plugins for distribution, subagents for specialization, and now the surfaces that let it all run wherever you happen to be.

The course ends. Your automation backlog doesn’t. Open the playbook from Chapter 12, run the audit, and pick the next thing — the best automation you’ll build is the one you start tomorrow when you catch yourself doing something twice. Build the third one as an installable plugin (Ch 13), give it a focused subagent (Ch 14), and ship it to the cloud (Ch 15). The infrastructure is finally ready for you.
