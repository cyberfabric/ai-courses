Chapter 3

Custom Claude Code Skills

Build reusable /commands that turn repetitive workflows into one-liners

From Typing to Commanding

In Chapter 2, you learned that skills become /commands. You even created a simple /explain skill. Now we’re going deep — including the modern SKILL.md form that lets Claude Code invoke your skills automatically when relevant.

By the end of this chapter, you’ll have a library of practical skills that save real time — scaffolding features, writing PR descriptions, explaining legacy code, and converting XML layouts to Compose. All copy-paste-ready, all shareable with your team.

Think of skills as your personal army of junior developers who each know how to do exactly one thing perfectly.

Anatomy of a Skill File

A skill comes in one of two forms — and both produce the same /<name> command. The simple form is a single markdown file under .claude/commands/. The richer form is a directory under .claude/skills/ with a SKILL.md inside, which unlocks YAML frontmatter, supporting files, and automatic invocation by Claude.

# Form 1: a single markdown file (simplest, still fully supported)
.claude/commands/pr-description.md        →  /pr-description
.claude/commands/scaffold.md              →  /scaffold
~/.claude/commands/standup.md             →  /standup (personal, all projects)

# Form 2: a directory with SKILL.md (recommended for new skills)
.claude/skills/pr-description/SKILL.md    →  /pr-description
~/.claude/skills/standup/SKILL.md         →  /standup (personal, all projects)

Custom commands and skills have been merged. A file at .claude/commands/deploy.md and a skill at .claude/skills/deploy/SKILL.md both create /deploy and behave the same when you invoke them. Your existing commands/ files keep working forever — nothing to migrate. The SKILL.md form just adds optional features when you want them: a directory for supporting files, frontmatter that lets Claude decide when to invoke the skill, and tool pre-approval.

Frontmatter unlocks the good stuff

Frontmatter is the YAML block between --- markers at the top of the file. It’s what turns a skill from “a snippet I type /foo to run” into “a capability Claude Code knows about and can decide to use on its own.”

# File: .claude/skills/pr-description/SKILL.md

---
name: pr-description
description: Generate a PR description from the current diff. Use when the user finishes a branch and is ready to open a pull request.
allowed-tools: Bash(git diff *) Bash(git log *) Read
---

You are writing a pull request description.
…

The fields you’ll use most:

  • name — The slash command. Lowercase letters, numbers, and hyphens only, max 64 characters. If omitted, the directory name is used.
  • description — What the skill does and when to use it. Claude reads this to decide whether to invoke the skill automatically. Front-load the key use case — the listing is capped at 1,536 characters.
  • allowed-tools — Tools Claude can use without asking for permission while this skill is active.
  • disable-model-invocation: true — You-only invocation. Use this for skills with side effects (deploy, commit, send-slack) so Claude can’t trigger them on its own.
  • argument-hint — Hint shown in autocomplete, e.g. [component-name].

When to reach for SKILL.md. Pick the bare-markdown form when the skill is yours, you’ll only ever invoke it manually, and you don’t need supporting files. Pick SKILL.md when you want Claude to invoke the skill on its own (e.g. an /explain-code skill that should kick in whenever someone asks “how does this work?”), when you need to bundle scripts or templates next to the prompt, or when you want to pre-approve specific tools so the skill runs without permission prompts.

The three parts of a good skill

  1. Context — Tell Claude Code what it’s looking at and what role to play
  2. Instructions — Step-by-step what to do, in the order you want it done
  3. Output format — What the result should look like

Well-structured skill

# /pr-description You are writing a pull request description. ## Steps 1. Read the current git diff (staged + unstaged) 2. Read CLAUDE.md for the PR template and conventions 3. Identify what changed and why (infer from code and commit messages) ## Output Write a PR description in markdown with: - **Title**: concise, under 70 characters, imperative mood - **## What**: bullet list of changes - **## Why**: motivation and context - **## Testing**: how to verify the changes - **## Screenshots**: placeholder if UI changed

Clear context, numbered steps, explicit output format. Claude Code knows exactly what to do.

Vague skill

Write a good PR description for my changes.

No structure, no format specification, no context about your team’s conventions. You’ll get a generic result every time.

Your First Real Skill: /generate-boilerplate

Let’s build something useful. Every Android developer creates the same set of files for a new feature: a ViewModel, a Repository interface, a Repository implementation, a UI screen, and maybe a use case. That’s five files with predictable boilerplate. Let’s automate it.

Building the /generate-boilerplate Skill

  1. Create the file: .claude/commands/generate-boilerplate.md
  2. Add this content:
# Generate Feature Boilerplate

Generate the boilerplate files for a new Android feature: **$ARGUMENTS**

## Steps
1. Read CLAUDE.md for project conventions and architecture patterns
2. Create these files in the appropriate package:

### ViewModel
- File: `feature/{name}/{Name}ViewModel.kt`
- Extends the project's base ViewModel
- Uses StateFlow for UI state
- Includes a sealed interface for UI events

### Repository
- Interface: `domain/repository/{Name}Repository.kt`
- Implementation: `data/repository/{Name}RepositoryImpl.kt`
- Uses the project's ApiResult wrapper for network calls

### UI Screen
- File: `feature/{name}/{Name}Screen.kt`
- Jetpack Compose screen with preview
- Collects state from ViewModel
- Handles UI events

### Use Case (if the feature involves business logic)
- File: `domain/usecase/{Name}UseCase.kt`
- Single responsibility, injectable via Hilt

## Rules
- Follow existing code style in the project
- Add @Inject and @HiltViewModel annotations
- Use the naming conventions from CLAUDE.md
- Add KDoc to public classes
  1. Test it: /generate-boilerplate UserProfile
  2. Claude Code creates all the files, following your project’s patterns

That’s five files created in seconds, all following your team’s conventions, all consistent with each other. No copy-pasting from an old feature, no forgetting the Hilt annotation, no creating the ViewModel but forgetting the Repository interface.

The Power of $ARGUMENTS

$ARGUMENTS is the magic variable that makes skills flexible. Whatever the user types after the command name gets injected into the prompt as $ARGUMENTS.

# When you type:
/generate-boilerplate UserProfile

# Claude Code sees:
# $ARGUMENTS = "UserProfile"

This lets you build skills that work for any input, not just a hardcoded case:

# A skill that works for any file
Explain the file at $ARGUMENTS for a new team member.

# A skill that works for any component type
Generate tests for the component described by: $ARGUMENTS

# A skill that accepts multiple parameters
# /deploy staging v2.1.0
Deploy version to environment. Arguments: $ARGUMENTS
Parse the first word as environment and second as version.

Pro tip: You can use $ARGUMENTS anywhere in your skill file — in headings, in code blocks, even in conditional instructions. Claude Code does the substitution before processing the prompt, so it works everywhere.

Positional and named arguments

For skills that take more than one input, you have two ergonomic upgrades over plain $ARGUMENTS. Grab specific arguments by position with $ARGUMENTS[0], $ARGUMENTS[1], … or the shorthand $0, $1, … for the first, second, and so on. Even better, when you use the SKILL.md form you can declare named arguments in frontmatter and reference them by name — which reads a lot like a function signature:

# File: .claude/skills/migrate-component/SKILL.md

---
name: migrate-component
description: Migrate a UI component from one framework to another
arguments: [component, from, to]
---

Migrate the $component component from $from to $to.
Preserve all existing behavior and tests.

Running /migrate-component SearchBar React Vue expands $component to SearchBar, $from to React, and $to to Vue. Multi-word values? Wrap them in quotes: /migrate-component "User Profile" React Vue.

Skill Gallery: Four Skills Every Team Needs

Here are four production-ready skills you can copy into your project right now. Each one addresses a task from the automation audit you did in Chapter 1.

1. /scaffold-feature — Full Feature Generator

# File: .claude/commands/scaffold-feature.md

Generate a complete feature module for: **$ARGUMENTS**

## What to create
Read the project structure and CLAUDE.md first. Then generate:

1. **Data layer**: Repository interface + implementation, DTOs, mapper
2. **Domain layer**: Use case(s), domain model
3. **Presentation layer**: ViewModel with UiState, Screen composable with preview
4. **DI**: Hilt module binding the repository
5. **Navigation**: Add the route to the nav graph
6. **Tests**: Unit tests for ViewModel and Repository (JUnit 5 + MockK)

## Rules
- Match existing package structure exactly
- Use existing base classes (BaseViewModel, ApiResult, etc.)
- Follow naming patterns from similar features in the codebase
- Every public class gets KDoc
- Compose previews use the project's preview theme wrapper

Scaffolding a Feature Manually

  1. Find a similar feature in the codebase
  2. Copy 6–8 files to new locations
  3. Rename every class, function, and variable
  4. Update package declarations
  5. Wire up Hilt bindings
  6. Add navigation route
  7. Fix the inevitable typos

Time: 20–30 minutes, error-prone

With /scaffold-feature

  1. Type /scaffold-feature PaymentHistory
  2. Review the generated files
  3. Start implementing the actual logic

Time: 2 minutes, consistent every time

2. /pr-description — PR Body from Diff

# File: .claude/commands/pr-description.md

Generate a pull request description for the current changes.

## Steps
1. Run `git diff main...HEAD` to see all changes on this branch
2. Run `git log main..HEAD --oneline` for commit history
3. Read CLAUDE.md for the PR template

## Output format
```markdown
## What
- [bullet list of what changed, grouped by area]

## Why
[1-2 sentences on motivation — what problem this solves]

## How
[Brief technical approach — key decisions, trade-offs]

## Testing
- [ ] [Specific test steps a reviewer can follow]
- [ ] [Include relevant test commands]

## Screenshots
[If UI changes, note where screenshots should go]
```

## Rules
- Title should be imperative mood, under 70 characters
- Be specific — "Add retry logic to PaymentRepository" not "Update code"
- If there are breaking changes, add a ⚠️ Breaking section
- Mention any new dependencies added

3. /explain-code — Onboarding-Friendly Explanations

# File: .claude/commands/explain-code.md

Explain **$ARGUMENTS** for a developer who is new to this codebase.

## Steps
1. Read the specified file or component
2. Read related files (imports, parent classes, interfaces it implements)
3. Check CLAUDE.md for architecture context

## Explain
- **What it does** — purpose in plain English, no jargon
- **How it fits** — where it sits in the architecture, what calls it, what it calls
- **Key patterns** — design patterns used (Repository, Factory, etc.) and why
- **Data flow** — how data moves through this component
- **Gotchas** — non-obvious behavior, edge cases, things that trip people up
- **Related files** — what to read next to understand the full picture

## Style
- Use analogies where they help
- Short paragraphs, bullet lists for technical details
- Assume the reader knows Kotlin/Android but not this specific codebase

4. /refactor-to-compose — XML to Compose Converter

# File: .claude/commands/android/refactor-to-compose.md

Convert the XML layout at **$ARGUMENTS** to Jetpack Compose.

## Steps
1. Read the XML layout file
2. Read the Activity/Fragment that uses this layout to understand data binding and click handlers
3. Read the ViewModel for state management patterns
4. Read CLAUDE.md for Compose conventions

## Generate
1. A `@Composable` function matching the layout's visual structure
2. Replace XML views with Compose equivalents:
   - `TextView` → `Text`
   - `RecyclerView` → `LazyColumn`/`LazyRow`
   - `ConstraintLayout` → `Column`/`Row` or `ConstraintLayout` composable
   - `ImageView` → `AsyncImage` (Coil) or `Image`
3. State hoisting — UI state comes from ViewModel via `collectAsStateWithLifecycle()`
4. A `@Preview` composable with sample data

## Rules
- Use Material 3 components where applicable
- Follow project's Compose conventions from CLAUDE.md
- Preserve all accessibility — contentDescription, semantics
- Keep the same visual behavior (margins → padding, weights → Modifier.weight)
- Add a TODO comment for any behavior that doesn't have a direct Compose equivalent

Skills are team knowledge, codified. That senior developer who always writes perfect PR descriptions? Their pattern is now a /pr-description skill that everyone gets. The architect who knows how to structure features? Their approach lives in /scaffold-feature. Skills capture expertise and make it available to the entire team, instantly.

Skill Chaining: Composing Workflows

Individual skills are powerful. Chained skills are transformative. You can reference other skills in your skill file, and you can build “meta-skills” that orchestrate multiple steps.

# File: .claude/commands/ship-feature.md

Complete the full shipping workflow for the current branch.

## Steps (execute in order)
1. Run the project's linter and fix any issues
2. Generate tests for any new or changed source files that don't have tests yet
3. Run the full test suite and fix any failures
4. Generate a PR description from the diff (follow the format in CLAUDE.md)
5. Create the PR on GitHub using the gh CLI

## Rules
- Stop and report if tests fail after 2 fix attempts
- Include test coverage for all new public functions
- PR title should reference the ticket number if found in branch name

One command. Lint, test, describe, create PR. What used to be a 15-minute manual process is now /ship-feature.

Organizing Your Skill Library

As your skill collection grows, organization matters. Here’s a structure that scales:

.claude/commands/
├── pr-description.md           # General workflow
├── explain-code.md             # General workflow
├── ship-feature.md             # General workflow
├── android/
│   ├── scaffold-feature.md     # Android-specific
│   ├── refactor-to-compose.md  # Android-specific
│   └── generate-tests.md       # Android test generation
├── review/
│   ├── security-review.md      # Security-focused review
│   ├── performance-review.md   # Performance-focused review
│   └── accessibility-review.md # Accessibility check
└── docs/
    ├── generate-readme.md      # Documentation
    └── generate-adr.md         # Architecture Decision Records

Subdirectories help you organize a growing library, but they don’t change the slash command name. A file at .claude/commands/android/scaffold-feature.md still produces /scaffold-feature — the subdirectory just shows up in the autocomplete description so you can see how the skill is grouped. Colon syntax like plugin-name:skill-name is reserved for skills shipped via plugins, where it prevents naming collisions across third-party packages. We’ll meet plugins later in the course.

Discoverability matters. Name your skills so that a new team member can guess what they do. /sf saves keystrokes but /scaffold-feature tells you what it does. Future you (and your teammates) will thank present you.

Debugging Skills

Your skill doesn’t always produce what you expect on the first try. Here’s how to fix it:

Common problems and fixes

Problem Likely Cause Fix
Output is too generic Skill doesn’t reference project context Add “Read CLAUDE.md” and “Follow existing patterns”
Wrong file locations Skill doesn’t read project structure Add “Read the project structure first” as step 1
Inconsistent style No style reference Add “Match the style of existing files in this directory”
Missing imports or DI Skill doesn’t check dependencies Add “Check existing Hilt modules and update if needed”
$ARGUMENTS not working Typo or wrong case Must be exactly $ARGUMENTS (all caps, with $)

Iterate, don’t overengineer. Start with a simple skill. Use it a few times. Notice what’s missing. Add instructions to fix the gaps. The best skills evolve through use, not through upfront design. Three rounds of “use it, tweak it” beats an hour of trying to write the perfect prompt.

Skill Versioning and Evolution

Skills live in your repo. They get committed, reviewed, and versioned just like code. This means:

  • PRs for skill changes — When someone improves a skill, the team can review the new prompt
  • Git blame for context — Why was this instruction added? Check the commit message
  • Branch-specific skills — Try a new version on a feature branch before merging

When to Update a Skill

Update your skills when: your architecture changes, your team adopts new conventions, you notice repeated manual fixes after running the skill, or a new team member finds the output confusing. A skill that isn’t evolving is a skill that’s falling behind your codebase.

Team Skill Libraries

The real multiplier is when skills become shared. Every developer on your team gets the same /commands, producing consistent results.

  • Project skills (.claude/commands/) are checked into git — everyone gets them
  • CLAUDE.md rules mean skills produce team-consistent output
  • New team members get the full skill library on day one
  • Skill improvements benefit everyone via normal git workflow

Onboarding goes from “here’s a wiki page about our conventions, good luck” to “run /scaffold-feature and everything follows our patterns automatically.”
