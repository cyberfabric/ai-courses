Chapter 10

MCP Servers for Your Team

Give Claude Code superpowers by connecting it to your internal tools and data

Beyond the Filesystem

Everything you’ve built so far — skills, hooks, CI/CD workflows — works with what Claude Code can already see: files, git, and shell commands. But your development workflow doesn’t live entirely in the filesystem.

Your tickets are in Jira. Your conversations are in Slack. Your deployment status is in an internal dashboard. Your staging database has the data you need to debug that weird production bug. And right now, to get any of that information into Claude Code, you have to copy-paste it manually.

MCP (Model Context Protocol) changes that. It lets you build servers that give Claude Code direct access to any tool, API, or data source your team uses. Ask Claude Code “what’s the status of JIRA-1234?” and it fetches the answer from Jira. Ask “is staging healthy?” and it checks your deployment dashboard.

What MCP Is (Simply)

Think of MCP as a USB port for Claude Code. Just like USB lets you plug any peripheral into your computer, MCP lets you plug any data source into Claude Code.

Claude Code
→
MCP Protocol
→
Your MCP Server
→
Your Internal Tool

An MCP server is a small program that:

  1. Declares tools — tells Claude Code what it can do (“I can look up Jira tickets”)
  2. Handles requests — when Claude Code calls a tool, the server executes it
  3. Returns results — sends the data back to Claude Code in a format it understands

You don’t need to understand the protocol details. You just need a server that talks to your tool and exposes it as named functions.

MCP servers come in two flavours, and Claude Code speaks both:

  • Local (stdio). A process Claude Code spawns on your machine and talks to over stdin/stdout. Best for talking to internal systems, local databases, or anything that needs filesystem or VPN access.
  • Remote (HTTP / SSE). A URL Claude Code calls over HTTPS. Most vendor-hosted servers (Stripe, Notion, Linear, Sentry, GitHub, Atlassian, …) ship as HTTP endpoints — claude mcp add --transport http stripe https://mcp.stripe.com and you’re done. No hosting, no install.

You’ll write your own stdio servers for internal tools and lean on hosted servers for popular SaaS.

Anatomy of an MCP Server

An MCP server has three main parts:

Component What It Does Example
Tools Functions Claude Code can call get_ticket, query_database, check_deploy
Resources Data Claude Code can read API docs, schema definitions, configuration files
Prompts Pre-built prompt templates “Summarize this ticket”, “Debug this query”

Most servers focus on tools — that’s where the automation power is.

Building Your First MCP Server

Let’s build a practical MCP server: one that checks your deployment status. We’ll use TypeScript with the official MCP SDK.

Building a Deployment Status MCP Server

  1. Set up the project:
mkdir mcp-deploy-status && cd mcp-deploy-status
npm init -y
npm install @modelcontextprotocol/sdk zod
npm install -D typescript @types/node
npx tsc --init
  1. Create the server: src/index.ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "deploy-status",
  version: "1.0.0",
});

// Tool: Check deployment status
server.registerTool(
  "check_deploy_status",
  {
    title: "Check Deployment Status",
    description: "Check the current deployment status for an environment",
    inputSchema: {
      environment: z.enum(["staging", "production"])
        .describe("Which environment to check"),
    },
  },
  async ({ environment }) => {
    const response = await fetch(
      `https://deploy.internal.company.com/api/status/${environment}`,
      { headers: { Authorization: `Bearer ${process.env.DEPLOY_TOKEN}` } }
    );
    const status = await response.json();
    return {
      content: [{
        type: "text",
        text: `## ${environment} Deployment Status
**Version:** ${status.version}
**Deployed at:** ${status.deployedAt}
**Status:** ${status.healthy ? "Healthy" : "UNHEALTHY"}
**Commit:** ${status.commitSha}
**Deployer:** ${status.deployedBy}`
      }]
    };
  }
);

// Tool: List recent deployments
server.registerTool(
  "list_deployments",
  {
    title: "List Recent Deployments",
    description: "List recent deployments for an environment",
    inputSchema: {
      environment: z.enum(["staging", "production"]),
      count: z.number().int().positive().default(5),
    },
  },
  async ({ environment, count }) => {
    const response = await fetch(
      `https://deploy.internal.company.com/api/history/${environment}?limit=${count}`,
      { headers: { Authorization: `Bearer ${process.env.DEPLOY_TOKEN}` } }
    );
    const history = await response.json();
    const rows = history.map((d: any) =>
      `| ${d.version} | ${d.deployedAt} | ${d.deployedBy} | ${d.status} |`
    ).join("\n");
    return {
      content: [{
        type: "text",
        text: `| Version | Deployed | By | Status |\n|---|---|---|---|\n${rows}`
      }]
    };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);
  1. Build it: npx tsc
  2. Register with Claude Code (see next section)

API note: the modern SDK uses server.registerTool(name, { description, inputSchema, … }, handler) with Zod schemas as the input shape. The legacy four-argument form (server.tool(name, desc, schema, handler)) still compiles but is deprecated — new servers should use registerTool.

Now you can ask Claude Code: “Is staging healthy?” or “Who deployed to production last?” and it calls your server to get real answers.

Registering MCP Servers

The fastest way is the claude mcp add CLI — it figures out the config file for you and writes the right shape. The --scope flag picks where the entry lands:

# Project scope: writes to .mcp.json at the project root.
# Designed to be checked into git so the whole team gets the server.
claude mcp add deploy-status --scope project --transport stdio \
  --env DEPLOY_TOKEN=$DEPLOY_TOKEN \
  -- node ./mcp-deploy-status/dist/index.js

# User scope: writes to your user config — applies across every project.
# Good for personal credentials and personal-only servers.
claude mcp add my-notes --scope user --transport stdio -- node ~/tools/notes.js

# Local scope (default): scoped to this project + this machine only,
# stored in your local user settings — never committed.
claude mcp add scratch --transport stdio -- node ./scratch-server.js

For project-shared servers, the resulting .mcp.json at your project root looks like:

// .mcp.json — checked into git for the team
{
  "mcpServers": {
    "deploy-status": {
      "type": "stdio",
      "command": "node",
      "args": ["./mcp-deploy-status/dist/index.js"],
      "env": {
        "DEPLOY_TOKEN": "${DEPLOY_TOKEN}"
      }
    }
  }
}

The top-level mcpServers key, the ${VAR} env-variable expansion, and the project-root location are all part of the spec — .mcp.json is the canonical project-shared registration path. You can hand-edit it, or stick with claude mcp add and let the CLI manage it. There’s also claude mcp add-json for pasting a JSON blob and claude mcp add-from-claude-desktop for importing servers from Claude Desktop.

Don’t commit secrets. Use the ${ENV_VAR} expansion in .mcp.json so the file references variables from your shell or .env file. Then add the actual values via your usual secret store (1Password, direnv, OS keychain) — never hardcode them. Project-scoped servers also prompt for approval the first time, so a malicious checked-in .mcp.json can’t silently exfiltrate data.

MCP Server for Jira / Issue Trackers

Want Claude Code to read your tickets while working on code? Build an issue tracker MCP server:

// Key tools to implement in a Jira MCP server
import { z } from "zod";

server.registerTool(
  "get_ticket",
  {
    description: "Get details of a Jira ticket",
    inputSchema: { ticketId: z.string().describe("e.g., PROJ-1234") },
  },
  async ({ ticketId }) => {
    const ticket = await jiraClient.getIssue(ticketId);
    return {
      content: [{
        type: "text",
        text: `## ${ticket.key}: ${ticket.fields.summary}
**Status:** ${ticket.fields.status.name}
**Assignee:** ${ticket.fields.assignee?.displayName || "Unassigned"}
**Priority:** ${ticket.fields.priority.name}
**Description:**
${ticket.fields.description}`
      }]
    };
  }
);

server.registerTool(
  "search_tickets",
  {
    description: "Search Jira tickets with JQL",
    inputSchema: { query: z.string().describe("JQL query or plain text search") },
  },
  async ({ query }) => {
    const results = await jiraClient.searchJira(query);
    // Format and return results
  }
);

server.registerTool(
  "add_comment",
  {
    description: "Add a comment to a Jira ticket",
    inputSchema: { ticketId: z.string(), comment: z.string() },
  },
  async ({ ticketId, comment }) => {
    await jiraClient.addComment(ticketId, comment);
    return { content: [{ type: "text", text: `Comment added to ${ticketId}` }] };
  }
);

With this server, you can say: “Read PROJ-1234 and implement the feature described in the ticket.” Claude Code fetches the ticket, reads the description, and starts coding. No tab-switching, no copy-pasting.

Working with Jira Without MCP

  1. Open Jira in the browser
  2. Find the ticket
  3. Read the description and acceptance criteria
  4. Switch back to Claude Code
  5. Paste the relevant details into your prompt
  6. Repeat when you need more context

Context switches: 4+. Time lost: 5–10 min per ticket.

Working with Jira With MCP

  3. Codes the feature with full ticket context

MCP Server for Database Queries

Give Claude Code read-only access to your staging database. Perfect for debugging data issues or understanding schema relationships:

server.registerTool(
  "query_database",
  {
    description: "Run a read-only SQL query on staging",
    inputSchema: { query: z.string().describe("SQL SELECT query") },
  },
  async ({ query }) => {
    // Safety: only allow SELECT queries
    if (!query.trim().toUpperCase().startsWith("SELECT")) {
      return {
        content: [{
          type: "text",
          text: "ERROR: Only SELECT queries are allowed. No mutations permitted."
        }]
      };
    }

    const result = await pool.query(query);
    // Format as markdown table
    const headers = Object.keys(result.rows[0] || {});
    const headerRow = `| ${headers.join(" | ")} |`;
    const separator = `| ${headers.map(() => "---").join(" | ")} |`;
    const dataRows = result.rows.slice(0, 50).map(row =>
      `| ${headers.map(h => row[h]).join(" | ")} |`
    ).join("\n");

    return {
      content: [{
        type: "text",
        text: `${headerRow}\n${separator}\n${dataRows}\n\n(${result.rowCount} rows)`
      }]
    };
  }
);

server.registerTool(
  "describe_table",
  {
    description: "Get the schema of a database table",
    inputSchema: { table: z.string() },
  },
  async ({ table }) => {
    const result = await pool.query(
      "SELECT column_name, data_type, is_nullable FROM information_schema.columns WHERE table_name = $1",
      [table]
    );
    // Format and return schema
  }
);

Read-only access only. No exceptions. Your database MCP server must enforce read-only at the server level. Don’t rely on Claude Code to be careful — validate every query before executing. Use a database user with SELECT-only permissions as a second layer of defense. Never connect to production — staging only.

Team-Shared MCP Configurations

MCP servers become really powerful when the whole team uses them. Commit a single .mcp.json at the project root and every teammate gets the same servers on their next session:

// .mcp.json — checked into the repo
{
  "mcpServers": {
    "jira": {
      "type": "stdio",
      "command": "node",
      "args": ["./tools/mcp-jira/dist/index.js"],
      "env": {
        "JIRA_URL": "https://yourcompany.atlassian.net",
        "JIRA_TOKEN": "${JIRA_API_TOKEN}"
      }
    },
    "deploy-status": {
      "type": "stdio",
      "command": "node",
      "args": ["./tools/mcp-deploy/dist/index.js"]
    },
    "staging-db": {
      "type": "stdio",
      "command": "node",
      "args": ["./tools/mcp-db/dist/index.js"],
      "env": {
        "DB_HOST": "staging-db.internal.company.com",
        "DB_NAME": "myapp_staging"
      }
    },
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/mcp"
    }
  }
}

The Sentry entry shows the second flavour: a hosted HTTP server. No local process to run, no token to wire up — Claude Code authenticates with the vendor on first use.

MCP servers are a force multiplier for every automation you’ve built. Your /standup skill can now include ticket status from Jira. Your /review skill can check deployment history. Your /morning-digest can include database health metrics. Every skill and hook gains access to every tool your MCP servers expose. The automation possibilities multiply with each server you add.

Security Best Practices

MCP servers bridge Claude Code to your internal systems. Security is non-negotiable:

  • Read-only by default. Start with read-only tools. Add write capabilities only when explicitly needed and approved.
  • Principle of least privilege. Each server gets the minimum permissions it needs. The DB server gets SELECT only. The Jira server gets read and comment, not admin.
  • No production database access. Staging only. Always.
  • Secrets in environment variables. Never hardcode tokens. Reference them via ${ENV_VAR} expansion in .mcp.json; keep the actual values in your shell, .env, or OS keychain.
  • Input validation. Sanitize every input before passing to APIs or databases. Prevent injection attacks.
  • Audit logging. Log every tool call with timestamp, user, and parameters for security review.

Testing MCP Servers

Before rolling out to the team, verify your server works correctly:

# 1. Test the server starts without errors
node dist/index.js
# Should start and wait for input (Ctrl+C to stop)

# 2. Test with Claude Code interactively
# Just start a Claude Code session with the server configured
# and ask it to use the tools:
# "Check the staging deployment status"
# "Describe the users table in the staging database"
# "Look up ticket PROJ-1234"

# 3. Test error handling
# Ask Claude Code to do something the server should reject:
# "Run DELETE FROM users" → should be blocked
# "Check production database" → should be denied

Build incrementally. Start with one tool per server. Get it working, test it with the team, then add more. A server with one reliable tool is better than a server with ten untested ones.

MCP Server Ideas for Your Team

Before you write your own, check whether the vendor already ships one. As of 2026 there are official hosted MCP servers for Stripe, Linear, Sentry, Notion, GitHub, Atlassian, HubSpot, Slack, Asana, Figma, Vercel, Neon, Supabase, PayPal and dozens more. The Anthropic MCP registry lists current ones, and a one-liner like claude mcp add --transport http stripe https://mcp.stripe.com is usually enough to add them. Build your own only for internal tools the vendor list doesn’t cover — that’s where the gallery below comes in.

S
Slack Server
Read channel messages, post updates, search conversations. Your standup skill can post directly to Slack. (Hosted version available; build your own only if you need internal-channel-only access.)
F
Firebase / Analytics Server
Query crash reports, check analytics events, monitor error rates. Debug production issues without leaving Claude Code.
C
Confluence / Wiki Server
Search and read team documentation, architecture pages, runbooks. Claude Code gets full context from your knowledge base.
G
Grafana / Monitoring Server
Check dashboards, query metrics, read alerts. When debugging performance, Claude Code sees what the monitoring sees.
