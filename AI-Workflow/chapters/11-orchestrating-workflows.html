Chapter 11

Orchestrating Multi-Step Workflows

Chain your automations into end-to-end pipelines that handle complex tasks in one shot

From Pieces to Pipelines

Over the last ten chapters, you’ve built an impressive arsenal: skills for code generation and review, hooks for automatic linting and testing, CI/CD workflows for changelogs and deployment checks, productivity agents for standups and digests, and MCP servers for connecting to internal tools.

Each automation is powerful on its own. But the real transformation happens when you chain them together into multi-step workflows that handle entire processes end-to-end.

Think of it like cooking. You’ve learned to chop, sauté, and season. Now it’s time to follow a recipe that turns all those techniques into a complete meal.

The Feature-to-PR Pipeline

This is the crown jewel — a single command that takes you from a ticket to a ready-for-review PR:

Building the /feature-pipeline Skill

  1. Create: .claude/commands/feature-pipeline.md
  2. Add this content:
# Feature Pipeline

Execute the full feature workflow for: **$ARGUMENTS**

This is a multi-step pipeline. Execute each step in order.
If any step fails, stop and report the failure.

## Step 1: Understand the Task
- If $ARGUMENTS looks like a ticket ID (e.g., PROJ-1234),
  fetch the ticket details using the Jira MCP tool
- If it's a description, use it directly
- Summarize: what needs to be built?

## Step 2: Create the Branch
- Derive a branch name from the task: `feature/{ticket-id}-{short-description}`
- Run `git checkout -b {branch-name}` from the latest main

## Step 3: Scaffold the Code
- Read CLAUDE.md for architecture patterns
- Create the necessary files (ViewModel, Repository, UI, etc.)
- Follow existing patterns in the codebase

## Step 4: Implement the Feature
- Write the actual implementation based on the task description
- Follow all rules in CLAUDE.md

## Step 5: Generate Tests
- Create unit tests for all new classes
- Use the project's testing patterns (JUnit 5, MockK, Turbine)
- Run the tests and fix any failures

## Step 6: Lint and Format
- Run the project linter on all changed files
- Fix any issues

## Step 7: Create the PR
- Commit all changes with a conventional commit message
- Push the branch
- Generate a PR description from the diff
- Create the PR using `gh pr create`

## Output
Report the result of each step.
End with the PR URL.
  1. Test it: /feature-pipeline PROJ-1234 or /feature-pipeline "Add user profile caching"

Feature Development Manually

  1. Read the Jira ticket (5 min)
  2. Create a branch (1 min)
  3. Scaffold files manually (15 min)
  4. Implement the feature (30+ min)
  5. Write tests (20 min)
  6. Run linter, fix issues (5 min)
  7. Write PR description (5 min)
  8. Push and create PR (2 min)

Overhead (excluding implementation): ~50 minutes

With /feature-pipeline

  1. Run /feature-pipeline PROJ-1234
  2. Review the generated code and tests
  3. Adjust implementation details
  4. PR is already created and ready

Overhead: ~5 minutes of review. Pipeline handles the rest.

Review the output. A pipeline that generates code, tests, and a PR in one shot is powerful — and dangerous if you don’t review it. Always check the generated code before approving the PR. The pipeline handles the mechanical work; you provide the judgment.

Composing Skills into Workflows

The feature pipeline works because it calls multiple automations in sequence. You can build the same pattern for any multi-step process:

The release workflow

# File: .claude/commands/release.md

Execute the full release workflow for version: **$ARGUMENTS**

## Steps (in order)
1. **Version bump**: Update version in build.gradle.kts and version.txt
2. **Changelog**: Generate changelog entry from commits since last tag
3. **Release notes**: Generate user-facing release notes for the Play Store
4. **Validation**: Run full test suite, lint, and deployment pre-checks
5. **Tag**: Create a git tag for the version
6. **Commit and push**: Commit version bump + changelog, push with tag

## Error handling
- If tests fail: STOP. Report failures. Do not tag or push.
- If lint fails: auto-fix and re-run. If still failing, STOP.
- If pre-checks fail: STOP. Report what's not ready.

## Output
Report each step's result. End with:
- The git tag created
- A summary of what's in this release
- The Play Store release notes (ready to paste)

The tech debt sprint workflow

# File: .claude/commands/debt-sprint.md

Run a focused tech debt reduction session.

## Steps
1. Run the tech debt scan to find current items
2. Filter to "quick fix" items (< 30 min each)
3. Pick the top 5 by severity
4. For each item:
   a. Create a fix on the current branch
   b. Run relevant tests to verify the fix
   c. Commit with message: "chore: fix {description}"
5. Generate a summary of all fixes

## Rules
- Only fix items that are clearly safe (no behavioral changes)
- Skip anything that requires design decisions
- Run tests after each fix, revert if tests break
- Stop after 5 items or 30 minutes of work

Skill chains vs. subagents. A composed skill (like the ones above) runs every step inside your main conversation — great for sequential pipelines you want to watch. When a step would flood your context with logs or search results, or when you want a step to use its own tool allowlist and even its own model, use a subagent instead. Subagents live in .claude/agents/<name>.md (project-shared) or ~/.claude/agents/<name>.md (personal), with frontmatter for name, description, tools, and model. Claude delegates to them automatically based on the description, or you can invoke one directly with @agent-name. They’re the natural next layer above skill chains for jobs like code review, security audits, or test generation that benefit from context isolation.

Scheduled Agents for Maintenance

Some workflows should run on their own, on a schedule. You set them up once and they keep your project healthy without any input:

#!/bin/bash
# scripts/weekly-maintenance.sh — Run every Monday via cron

echo "=== Weekly Maintenance: $(date) ===" >> maintenance.log

# 1. Dependency check
claude -p "Check for outdated or vulnerable dependencies in this project.
Report any that need updating." --allowedTools "Read,Bash(./gradlew*),Bash(cat*)" \
  >> maintenance.log 2>&1

# 2. Documentation freshness
claude -p "Check if documentation files (README.md, docs/*) are still
accurate compared to the current code. Report any stale sections." \
  --allowedTools "Read,Bash(git log*)" \
  >> maintenance.log 2>&1

# 3. Test coverage gaps
claude -p "Find source files that have no corresponding test files.
List the top 5 most critical untested files." \
  --allowedTools "Read,Bash(find*)" \
  >> maintenance.log 2>&1

# 4. Tech debt scan
claude -p "Count TODO, FIXME, and HACK comments in the codebase.
Compare with last week's count if available in maintenance.log." \
  --allowedTools "Read,Bash(grep*)" \
  >> maintenance.log 2>&1

echo "=== Maintenance Complete ===" >> maintenance.log
# Add to crontab: crontab -e
# Run every Monday at 8 AM
0 8 * * 1 cd /home/user/projects/myapp && bash scripts/weekly-maintenance.sh

Flag cheat-sheet for one-off automations. --allowedTools auto-approves tools you list (skipping the permission prompt). --tools restricts which tools are even available. --disallowedTools is an explicit blocklist. For headless scripts running in cron, prefer --tools for a tight allowlist over an empty --allowedTools "".

Remote Triggers: Event-Driven Workflows

Scheduled agents run on a clock. Remote triggers run on events — a Slack message, a Jira ticket transition, a webhook from your monitoring system.

# .github/workflows/on-issue-created.yml
name: Auto-Triage New Issues
on:
  issues:
    types: [opened]

jobs:
  triage:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6

      - name: Triage with Claude Code
        id: triage
        uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          claude_args: '--tools "Read,Bash(git log*)"'
          prompt: |
            Triage this GitHub issue:
              Title: ${{ github.event.issue.title }}
              Body: ${{ github.event.issue.body }}

              1. Categorize: bug / feature request / question / documentation
              2. Estimate severity: critical / high / medium / low
              3. Suggest labels
              4. If it's a bug, suggest which area of code to investigate
              5. If it's a feature, note if similar features exist

              Save the result to triage.md as a concise comment.

      - name: Post triage comment
        uses: actions/github-script@v9
        with:
          script: |
            const fs = require('fs');
            const triage = fs.readFileSync('triage.md', 'utf8');
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: `## Auto-Triage\n\n${triage}\n\n---\n*Automated by Claude Code*`
            });

The official action vs. the manual install. anthropics/claude-code-action@v1 bundles install + run + auth into a single step and accepts prompt and claude_args inputs. Use claude_args to pass any CLI flag — --tools to restrict tool availability, --disallowedTools for an explicit blocklist, --max-turns, --model, etc. --allowedTools still works (it controls auto-approval), but --tools is the modern flag for limiting which tools the agent can invoke at all.

The best workflows are invisible. When an issue is created, it’s automatically triaged. When a PR is merged, the changelog updates itself. When a dependency has a CVE, an issue is created with the fix. You don’t run these workflows — they run themselves. That’s the goal of orchestration: a project that maintains itself while you focus on building features.

Error Recovery Patterns

Multi-step workflows will fail. A test breaks, a push is rejected, a network call times out. Good orchestration handles failure gracefully:

Pattern 1: Stop and report

# In your skill: the safest default
If any step fails:
- Stop execution immediately
- Report which step failed and why
- List what was completed successfully
- Suggest how to fix the failure and resume

Pattern 2: Retry with backoff

#!/bin/bash
# Retry a flaky step up to 3 times
retry() {
  local max_attempts=3
  local attempt=1
  while [ $attempt -le $max_attempts ]; do
    if "$@"; then
      return 0
    fi
    echo "Attempt $attempt failed. Retrying in $((attempt * 5))s..."
    sleep $((attempt * 5))
    attempt=$((attempt + 1))
  done
  echo "Failed after $max_attempts attempts"
  return 1
}

# Usage
retry ./gradlew test --tests "*UserViewModelTest"

Pattern 3: Fallback

# In your skill
If the full test suite times out:
- Fall back to running only the tests for changed files
- Note in the PR description that full suite was not run
- Add a CI check that runs the full suite asynchronously

Default to “stop and report.” Automatic retries and fallbacks are useful for known-flaky operations (network calls, CI runners), but for most failures, you want to know about them immediately. A pipeline that silently retries a broken test three times is hiding a real problem.

Multi-Repo Automation

Some changes span multiple repositories — updating an SDK and all the apps that consume it, or changing an API and its clients simultaneously:

# File: .claude/commands/multi-repo-update.md

Coordinate a change across multiple repositories.

## Context
$ARGUMENTS describes the change that needs to happen across repos.

## Steps
1. Read the change description
2. Identify which repos are affected (check the repos listed below)
3. For each affected repo:
   a. Clone or navigate to it
   b. Create a feature branch with the same name
   c. Make the required changes
   d. Run tests
   e. Create a PR with a description that links to the other PRs
4. Create a tracking issue that links all PRs together

## Our repositories
- `myapp-android` — Android client
- `myapp-backend` — API server
- `myapp-shared` — Shared models and utilities

## Rules
- All PRs should have the same branch name for traceability
- Each PR description links to the tracking issue
- Note any deployment ordering requirements

Monitoring Your Automations

As you build more workflows, you need to know when they succeed and when they fail. Build a simple monitoring layer:

#!/bin/bash
# scripts/automation-monitor.sh — Log automation results
# Source this in your automation scripts

log_automation() {
  local name="$1"
  local status="$2"  # success or failure
  local details="$3"

  echo "$(date +%Y-%m-%d\ %H:%M) | $name | $status | $details" \
    >> ~/.claude/automation.log
}

# Usage in your scripts:
# source scripts/automation-monitor.sh
# log_automation "weekly-maintenance" "success" "4 items checked"
# log_automation "feature-pipeline" "failure" "Tests failed at step 5"
# File: ~/.claude/commands/automation-status.md

Review the automation health dashboard.

## Steps
1. Read ~/.claude/automation.log
2. Summarize the last 7 days of automation runs

## Report
- **Total runs**: X automations executed
- **Success rate**: X%
- **Failures**: list each with timestamp, name, and reason
- **Most used**: top 3 automations by frequency
- **Suggestions**: any automations that fail frequently and need attention

When NOT to Automate

Not everything should be a pipeline. Automation adds value when it eliminates repetitive, well-defined work. It adds complexity when the process is still evolving or requires human judgment at every step.

Don’t Automate When...

The process changes weekly. If you’re still figuring out the right workflow, automating it locks in a process that’s not ready to be locked in.

Every case is unique. If the task requires different judgment calls each time, a pipeline will either be too rigid or too complex.

The setup cost exceeds the lifetime savings. A workflow you run once a quarter doesn’t need a pipeline. A one-liner with claude -p is fine.

Failure is catastrophic. Automated deployments to production need extensive guardrails. If you’re not ready to build those guardrails, keep a human in the loop.

The Orchestration Maturity Model

Manual
→
Single Skills
→
Chained Workflows
→
Event-Driven
→
Self-Maintaining
  • Manual: You do each step by hand
  • Single skills: Each step is a /command, but you invoke them individually
  • Chained workflows: A master skill executes multiple steps in sequence
  • Event-driven: Workflows trigger on events (PRs, issues, schedules) without human initiation
  • Self-maintaining: Automations monitor each other, report health, and flag when they need updating

You don’t have to reach the top of this ladder for everything. Most workflows should live at “chained” or “event-driven.” Self-maintaining is for your most critical, most stable processes.

Exercise: Create Your Feature-to-PR Pipeline

Build a multi-step workflow that matches your actual development process. About 20 minutes.

Step 1: Design your pipeline

  1. Write down the steps you follow from “start feature” to “PR ready for review”

  1. Pick a small, well-defined task

  4. Note what the pipeline got right and what needs manual adjustment
  5. Iterate on the skill based on what you learned
