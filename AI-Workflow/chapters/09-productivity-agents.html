Personal Productivity Agents

Automate the meta-work — standups, digests, triage, and everything that isn’t coding

The Invisible Time Sink

You sit down at 9 AM. You open Slack. Fourteen notifications. You check GitHub — three PRs to review, two with new comments on yours. You write your standup update. You check CI for the PR you pushed yesterday. You scan the repo for what changed overnight. You look at the sprint board.

It’s 10 AM. You haven’t written a single line of code.

This is meta-work — the work about the work. It’s necessary, but it’s also the biggest hidden tax on your day. And unlike coding tasks, most developers don’t even think of meta-work as something that can be automated.

1.5 hrs average daily meta-work for a developer
23 min to regain focus after a context switch
375 hrs per year spent on meta-work (1.5h × 250 days)

This chapter builds the automations that reclaim your mornings. By the end, you’ll have a single command that gives you everything you need to start your day: what changed, what’s waiting for you, and what your standup update should say.

Automated Standup Summaries

The daily standup: “What did I do yesterday? What am I doing today? Any blockers?” Simple questions, but they require you to remember. And at 9:15 AM, your memory of yesterday is fuzzy at best.

Git remembers everything. Let it write your standup.

Building the /standup Skill

  1. Create: ~/.claude/commands/standup.md (user-level — works across all projects)
  2. Add this content:
# Generate Standup Update

Generate my daily standup update.

## Steps
1. Run `git log --author="$(git config user.email)" --since="yesterday" --oneline`
   to see what I committed yesterday
2. Run `git log --author="$(git config user.email)" --since="yesterday" --stat`
   for details on what changed
3. Run `git branch --show-current` to see what I'm working on now
4. Run `git stash list` to check for any stashed work-in-progress
5. Run `gh pr list --author=@me --state=open` to see my open PRs

## Output format
**Yesterday:**
- [Bullet list of what I accomplished, grouped by feature/task]

**Today:**
- [What I'm likely working on based on current branch and open PRs]

**Blockers:**
- [Any PRs waiting for review, or note "None"]

## Rules
- Be specific: "Implemented user profile caching in UserRepository" not "Worked on code"
- Group related commits into one bullet point
- If there are open PRs awaiting review, list them as blockers
- Keep it concise — 3-5 bullets per section max
  1. Test it every morning: /standup

Writing Your Standup Manually

  1. Try to remember what you did yesterday
  2. Scroll through Slack and GitHub for reminders
  3. Write vague bullets: “Worked on user profile feature”
  4. Forget to mention the bug fix you spent an hour on
  5. Copy-paste into Slack/Teams/standup tool

Time: 5–10 minutes. Accuracy: low.

With /standup

  1. Run /standup
  2. Copy the output into your standup channel
  3. Every commit, branch, and PR is accounted for

Time: 30 seconds. Accuracy: complete.

The Daily Digest

What happened in the repo overnight? Who pushed what? Are there PRs waiting for your review? Build a morning briefing:

# File: ~/.claude/commands/morning-digest.md

Generate my morning development digest.

## Steps
1. Run `git log --since="16 hours ago" --oneline --all` for recent repo activity
2. Run `gh pr list --state=open --json number,title,author,reviewRequested`
   for open PRs
3. Run `gh pr list --search "review-requested:@me"` for PRs needing my review
4. Run `gh issue list --assignee=@me --state=open` for my open issues

## Sections

### Repo Activity (last 16 hours)
- Summary of commits by other team members
- Any new branches created
- PRs merged overnight

### Waiting For Me
- PRs that need my review (with links)
- Issues assigned to me
- PR comments I haven't responded to

### My Open Work
- My open PRs and their status (approved? changes requested? CI status?)
- My current branch and last commit

### Today's Priorities
Based on the above, suggest a prioritized task list for today:
1. Reviews waiting (blocking others)
2. Addressing PR feedback on my work
3. Continuing feature work

Make it a shell alias. Add alias morning='cd ~/projects/myapp && claude /morning-digest' to your .bashrc. Now you type morning and get your entire briefing. Even faster: add it to a cron job that runs at 8:55 AM and saves the output to a file you open when you sit down.

Code Archaeology

Why does this function do that? Who wrote this code, and what were they thinking? When you’re working in a legacy codebase (which is every codebase older than six months), understanding the why behind the code is half the battle.

# File: .claude/commands/archaeology.md

Investigate the history and rationale behind: **$ARGUMENTS**

## Steps
1. Read the specified file or function
2. Run `git log --follow -p -- "$ARGUMENTS"` for the full change history
3. Run `git blame "$ARGUMENTS"` to see who changed each line and when
4. For the most significant commits, read the full commit messages
5. Check if any PRs reference this file (search commit messages for PR numbers)

## Report
### Current State
- What the code does now

### History
- When was it first written? By whom?
- What were the major changes and why? (from commit messages)
- Any patterns: was this code rewritten multiple times? Recently stable?

### Key Decisions
- Why was it written this way? (infer from commit messages and PR context)
- Are there any TODO or FIXME comments with history?
- Any workarounds or hacks with explanations?

### Recommendations
- Is this code safe to refactor?
- What would break if this changed?
- Who should you talk to for context? (most recent significant contributor)

Run /archaeology src/data/sync/SyncManager.kt and get the full story behind that mysterious file everyone is afraid to touch.

PR Status Dashboard

Instead of checking each PR individually, generate a team dashboard:

# File: .claude/commands/pr-dashboard.md

Generate a PR status dashboard for the team.

## Steps
1. Run `gh pr list --state=open --json number,title,author,
   createdAt,reviewDecision,statusCheckRollup,labels`
2. For each PR, check: how old is it? What's the review status?
   Are CI checks passing?

## Dashboard format

### Needs Attention (oldest first)
| PR | Author | Age | Reviews | CI | Status |
|....|....|....|....|....|....|

### Ready to Merge (approved + CI green)
- List PRs that can be merged right now

### Blocked
- PRs with failing CI or changes requested
- Include the specific blocker for each

### Stats
- Total open PRs
- Average PR age
- PRs opened this week vs merged this week

## Rules
- Flag any PR older than 3 days as "aging"
- Flag any PR older than 7 days as "stale"
- Highlight PRs that are approved but not merged (forgotten?)

Tech Debt Tracking

Tech debt is invisible until it bites you. Build a skill that makes it visible:

# File: .claude/commands/tech-debt-scan.md

Scan the project for technical debt and catalogue it.

## Steps
1. Search for TODO, FIXME, HACK, WORKAROUND, XXX comments in source code
2. Search for deprecated API usage
3. Check for suppressed warnings (@Suppress, @SuppressLint)
4. Look for hardcoded values that should be configurable
5. Check build.gradle.kts for outdated dependency patterns

## For each finding, report
- **Location**: File and line
- **Type**: TODO / Deprecated API / Suppressed Warning / Hardcoded Value / Other
- **Severity**: Critical (will break) / High (should fix soon) / Medium / Low
- **Effort**: Quick fix (< 30 min) / Medium (1-4 hours) / Large (> 1 day)
- **Description**: What the debt is and why it matters

## Output
Group by severity. For each severity level, sort by effort (quick fixes first).
End with a summary: total items by severity, suggested sprint allocation
(e.g., "dedicate 2 hours per sprint to quick-fix items").

Run tech debt scans monthly and track the trend. Create a docs/tech-debt.md file and update it each month with the scan results. Over time, you’ll see whether debt is growing or shrinking. This single metric can justify dedicating sprint time to debt reduction better than any argument.

Time-Saving Shell Aliases

Not everything needs to be a full skill. For quick, one-off automations, shell aliases with claude -p are lightning fast:

# Add to ~/.bashrc or ~/.zshrc

# Explain any error message
alias explain='claude -p "Explain this error and suggest fixes:"'
# Usage: some-command 2>&1 | explain

# Quick commit message from staged changes
alias commitmsg='git diff --cached | claude -p "Write a conventional commit message for this diff. Output ONLY the message, nothing else."'

# Summarize a file in one sentence
alias tldr='claude -p "Summarize this file in one sentence:"'
# Usage: cat some-file.kt | tldr

# What does this command do?
alias howto='claude -p "Explain what this command does, step by step:"'
# Usage: echo "find . -name '*.kt' -exec grep -l 'GlobalScope' {} +" | howto

# Review just my changes on this branch
alias myreview='git diff main...HEAD | claude -p "Quick review of these changes. Flag any bugs or issues. Be brief."'

Aliases are the gateway drug to automation. Every alias that saves you time is a candidate for becoming a full skill later. Start with aliases for your most common one-liners, then graduate them to skills when you need more structure or want to share them with the team.

Notification Triage

GitHub notifications pile up. Most are noise. A few are critical. Let AI sort them:

# File: ~/.claude/commands/triage-notifications.md

Triage my GitHub notifications.

## Steps
1. Run `gh api notifications --jq '.[] | {reason, subject, updated_at, repository}'`
2. Group and prioritize:

## Priority Levels

### P0 — Act Now
- Review requests on PRs (you're blocking someone)
- CI failures on your PRs
- Direct mentions in issues or PRs

### P1 — Today
- Comments on your open PRs
- Issues assigned to you
- PR approvals (you can merge)

### P2 — This Week
- Team PR activity (stay informed)
- Issues in your repos
- Release notifications

### P3 — Archive
- Bot notifications (Dependabot, etc.)
- Subscribed threads with no action needed
- Old notifications (> 7 days)

## Output
List by priority level. For P0 and P1, include direct links.
For P3, suggest bulk-marking as read.

The Morning Routine: Combining It All

Individual productivity skills are good. Combined into a morning routine, they’re transformative.

#!/bin/bash
# scripts/morning-routine.sh — Your complete morning briefing

echo "=== Morning Routine: $(date +%A,\ %B\ %d) ==="
echo ""

# 1. Repo digest
echo "--- REPO DIGEST ---"
cd ~/projects/myapp
claude -p "$(cat ~/.claude/commands/morning-digest.md)" --allowedTools "Bash,Read" 2>/dev/null

echo ""

# 2. Standup draft
echo "--- STANDUP DRAFT ---"
claude -p "$(cat ~/.claude/commands/standup.md)" --allowedTools "Bash" 2>/dev/null

echo ""

# 3. Notification triage
echo "--- NOTIFICATIONS ---"
claude -p "$(cat ~/.claude/commands/triage-notifications.md)" --allowedTools "Bash" 2>/dev/null

echo ""
echo "=== Ready to code ==="

One Command, Zero Morning Fog

Add alias goodmorning='bash ~/scripts/morning-routine.sh' to your shell config. Tomorrow morning, type goodmorning and get your digest, standup, and triaged notifications in one shot. The 30-minute morning fog? Gone. Replaced by a 2-minute briefing.

Personal Knowledge Base

Over time, you build up knowledge about your codebase — patterns, gotchas, why certain things are the way they are. Most of this lives in your head and disappears when you switch projects. Capture it:

# File: .claude/commands/note.md

Add a note to the project knowledge base.

## Steps
1. Read the note content from $ARGUMENTS
2. Read docs/knowledge-base.md (create if it doesn't exist)
3. Add the new note with today's date and the current file context
4. Organize notes by category (architecture, gotchas, patterns, decisions)

## Format for each note
### [Category] Title
**Date:** YYYY-MM-DD
**Context:** [file or area of codebase]
[The note content]

## Rules
- Keep notes concise — 2-3 sentences max
- Include file paths when relevant
- Don't duplicate existing notes — update if similar exists

Run /note "The SyncManager retry logic uses exponential backoff with a max of 3 attempts. Don’t change the max without checking the server rate limits." and the knowledge is preserved.
