The Big Idea

AI handles the mechanical review work — style, bugs, security scanning — so human reviewers can focus on design, architecture, and the high-judgment calls. PRs go from days to hours.

🚧 The Code Review Bottleneck

⏳
24–72 Hours
Average time to first review comment. That’s days of waiting on another human’s attention.
🎨
60% Style Nitpicks
Most review comments are about style, not logic. AI catches these instantly and consistently.
🔄
4–6 Context Switches
Each review cycle forces both author and reviewer to reload context. Expensive for everyone.

🚦 Review Severity Levels (for CLAUDE.md)

  1. 🔴 CRITICAL — Must Fix (Block Merge)

    Failing tests, security vulns, memory leaks, lifecycle crashes, hardcoded secrets.

  2. 🟡 WARNING — Should Fix (Before Next Release)

    Missing KDoc, hardcoded strings, complex functions without tests, TODOs without tickets.

  3. 🟢 SUGGESTION — Nice to Have

    Naming improvements, extracting reusable functions, reducing nesting, adding Compose previews.

🧪 Quick Quiz

What percentage of review comments are typically about style rather than logic?

About 60%. AI catches these instantly, freeing human reviewers for design and architecture feedback.

What are the four layers of the review pipeline?

CLAUDE.md standards → /ship-pr self-review → CI AI review (automatic) → Human review (design & context).

Why run /ship-pr before opening a PR?

It catches mechanical issues (style, bugs, missing tests) before the human reviewer sees them, making human review faster and more focused on high-value feedback.

What should AI review NOT be trusted for?

Architecture decisions, product intent alignment, and long-term maintenance implications. These require human judgment and context.

How do you restrict which tools Claude Code can use in CI?

Use --tools "Read,Grep,Bash(git *)" to limit what’s available. --allowedTools only auto-approves listed tools (it doesn’t block others). --disallowedTools explicitly denies. For the anthropics/claude-code-action@v1 action, pass these via claude_args:.

📌 Key Takeaways

  • AI handles mechanical review (style, bugs, security) so humans focus on design and architecture.
  • Four-layer pipeline: CLAUDE.md standards → self-review → CI auto-review → human review.
  • Build domain-specific review skills: /review:android-ui, /review:security, /review:focused.
  • Review skills encode your team’s hard-won knowledge — every incident becomes a check.
  • Structure review output: one block per issue with file, severity, problem, and fix.
  • GitHub Actions + anthropics/claude-code-action@v1 = automated review comments on every PR. Pass --tools via claude_args to restrict what Claude can do.
  • AI augments humans, doesn’t replace them. Keep humans for high-judgment calls.
