The Big Idea

Testing is the broccoli of software development — everyone knows it’s good, nobody’s excited. AI eliminates the friction (setup mocks, dispatchers, edge cases) that makes devs skip tests. From “I’ll add tests later” to “tests just happen.”

🧰 The Testing Skill Arsenal

🧪
/generate-tests
Read source file → match project patterns → generate tests with happy path, edge cases, error paths. 25 min → 5 min.
🔍
/find-coverage-gaps
List all sources, match to test files, identify untested and partially-tested files. Prioritize by risk.
🔄
/update-tests
After a refactor, update tests to match new behavior. Don’t delete broken tests — fix them.
🩺
/diagnose-flaky
Analyze flaky test for timing, state, or external dependency issues. Root cause + fix + prevention pattern.
🧬
/mutation-check
Mutate source code, check if tests catch it. Tests that pass regardless of changes give false confidence.
📋
/review:tests
Review test quality: false positives, missing edges, brittle tests, isolation, readability.

🎲 Flaky Test Causes & Fixes

  1. ⏱️ Timing Issues

    Race conditions, missing advanceUntilIdle(), hardcoded delays instead of synchronization.

  2. 📦 State Issues

    Shared mutable state, tests depending on order, singletons not cleaned up in @BeforeEach.

  3. 🌐 External Dependencies

    Real network calls in unit tests (should be mocked), file system or database state leaking.

  4. 📱 Android-Specific

    Missing Dispatchers.setMain/resetMain, Compose rule timing, LiveData without InstantTaskExecutorRule.

Triage: /diagnose-flaky → apply fix → run 10x to verify stability

🧬 Mutation Testing: Are Your Tests Actually Good?

Mutation Testing
Change the source slightly (a “mutation”), run tests. If tests still pass, they’re not catching that kind of bug.
Example Mutations
Change > to >=, swap true/false, remove null checks, change + to -, swap &&/||.
Mutation Killed ✅
A test caught the mutation — your test works.
Mutation Survived ❌
No test caught it — you’re missing a test case.
Target Score
70–80% on critical code is excellent. Don’t aim for 100% — some mutations are trivial.

🧪 Quick Quiz

Why do 40% of developers skip writing tests?

Time pressure and friction. Setting up mocks, dispatchers, Hilt rules, and edge cases kills your flow. AI removes that friction.

What are the four categories of things to test?

Happy path (normal inputs), edge cases (null/empty/boundary), error paths (exceptions/failures), and state transitions (Loading → Success → Error).

What is mutation testing and what score should you aim for?

Change source code slightly, see if tests catch it. 70–80% mutation score on critical code is excellent. 100% isn’t worth chasing.

How do you verify a flaky test fix is actually stable?

Run the test 10 times locally: for i in {1..10}; do ./gradlew test --tests "*TestName" || break; done. All 10 pass = stable.

What’s the best way to get AI-generated tests that match your project’s patterns?

Have existing well-written test files in the project. Claude Code reads them and follows the same setup, naming, and assertion style.

📌 Key Takeaways

  • AI eliminates test-writing friction: 15 min manual → 30 sec generated. Tests stop being skipped.
  • Six testing skills: /generate-tests, /find-coverage-gaps, /update-tests, /diagnose-flaky, /mutation-check, /review:tests.
  • Always test: happy path, edge cases, error paths, and state transitions.
  • Android patterns: runTest + Turbine for ViewModels, MockK for Repositories, createComposeRule for UI.
  • Flaky tests: diagnose cause (timing, state, external, Android-specific) → fix → run 10x to verify.
  • Mutation testing validates test quality — tests that pass no matter what give false confidence.
  • Full pipeline: Code Change → /generate-tests → /review:tests → Run Suite → Coverage Check.
