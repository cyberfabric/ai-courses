The Big Idea

Skills are markdown that becomes /commands — either a single file under .claude/commands/ or a directory with a SKILL.md. Both produce the same slash command. The directory form adds frontmatter that lets Claude invoke skills automatically when relevant.

🧬 Anatomy of a Skill

Two equivalent forms

Form Path What it adds
Bare markdown .claude/commands/<name>.md Simplest. You-invoked only.
SKILL.md directory .claude/skills/<name>/SKILL.md Frontmatter, supporting files, model invocation.

The frontmatter fields you’ll use

name
The slash command. Lowercase + hyphens, max 64 chars.
description
What it does and when to use it. Claude reads this to decide whether to invoke automatically.
allowed-tools
Tools that run without permission prompts while the skill is active.
disable-model-invocation: true
You-only. Use for skills with side effects (deploy, commit, send-slack).
argument-hint
Hint shown in autocomplete, e.g. [component-name].

🏗️ The Three Parts of a Good Skill

🎯
1. Context
Tell Claude Code what it’s looking at and what role to play. “You are writing a PR description.”
📋
2. Instructions
Step-by-step what to do, in order. Numbered lists work best. Include “Read CLAUDE.md” as a step.
📤
3. Output Format
What the result should look like. Be explicit: markdown template, sections, length constraints.

🖼️ Four Skills Every Team Needs

🏗️
/scaffold-feature
Generates ViewModel, Repository, UI Screen, Use Case, DI module, and tests for a new feature. 20 min → 2 min.
📝
/pr-description
Reads git diff + commit history, outputs What/Why/How/Testing sections following your team’s template.
🔍
/explain-code
Onboarding-friendly explanation: what it does, how it fits, key patterns, data flow, gotchas, related files.
🔄
/refactor-to-compose
Converts XML layouts to Jetpack Compose: reads layout + Activity + ViewModel, generates composable with preview.

🔑 The $ARGUMENTS Magic Variable

$ARGUMENTS
Everything the user types after the /command name. Substituted into the prompt before processing.
$0, $1, $2 …
Positional shorthand. $0 = first argument, $1 = second, etc. (Same as $ARGUMENTS[0].)
Named arguments (SKILL.md)
Declare arguments: [component, from, to] in frontmatter, then write $component, $from, $to in the body. Reads like a function signature.
Quoting multi-word values
/migrate "User Profile" React Vue → $0 = "User Profile".
Common gotcha
Must be exactly $ARGUMENTS — all caps, with $. Wrong case = no substitution.

Subdirs organize files but don’t change the slash command name. Colon syntax (plugin-name:skill-name) is reserved for plugin skills.

🐛 Debugging Skills: Common Fixes

Problem Cause Fix
Output too generic No project context Add “Read CLAUDE.md first”
Wrong file locations Didn’t read structure Add “Read project structure” as step 1
Inconsistent style No style reference Add “Match existing file style”
Missing DI / imports Didn’t check deps Add “Update Hilt modules if needed”
$ARGUMENTS empty Typo or wrong case Must be $ARGUMENTS exactly

🔄 The Skill Lifecycle

  1. Write a Simple Version

    Start minimal. Context + basic instructions + output format. 5 minutes.

  2. Use It a Few Times

    Notice what’s missing, what’s wrong, what needs tweaking.

  3. Iterate and Refine

    Add instructions to fix gaps. Three rounds of “use it, tweak it” beats an hour of upfront design.

  4. Share with Team

    Commit to repo. PRs for skill changes. Git blame for why an instruction was added.

  5. Evolve with Codebase

    Update when architecture changes, conventions shift, or new team members find output confusing.

🧪 Quick Quiz

Where do project-level skill files live?

In .claude/commands/ at the project root. They’re checked into git so the whole team gets them.

What are the three parts of a well-structured skill?

Context (role & situation), Instructions (numbered steps), and Output Format (what the result should look like).

How do subdirectories affect skill names?

They don’t. .claude/commands/android/scaffold.md still produces /scaffold — the subdirectory just shows up in autocomplete to help you see how the skill is grouped. Colon syntax (plugin-name:skill-name) is reserved for skills shipped via plugins.

What does the SKILL.md form give you that bare markdown doesn’t?

YAML frontmatter (model invocation, tool pre-approval, named arguments), a directory for supporting files, and the option for Claude to invoke the skill automatically when its description matches the situation.

What’s the best way to debug a skill that produces generic output?

Add “Read CLAUDE.md” and “Follow existing patterns in this directory” as early steps. Give it project context.

What is a “meta-skill” or chained skill?

A skill that orchestrates multiple steps in sequence — like /ship-feature which lints, tests, writes a PR description, and creates the PR.

📌 Key Takeaways

  • A skill is markdown that becomes /<name>. Two equivalent forms: bare file under .claude/commands/, or .claude/skills/<name>/SKILL.md directory.
  • SKILL.md frontmatter (name, description, allowed-tools, disable-model-invocation) is what unlocks model-invoked skills and tool pre-approval.
  • Structure the body in three parts: Context, Instructions, Output Format.
  • $ARGUMENTS works everywhere; $0/$1 grabs by position; named arguments via arguments: frontmatter read like function signatures.
  • Subdirectories organize files but don’t namespace command names. Colon syntax is reserved for plugin skills.
  • Four essential skills: /scaffold-feature, /pr-description, /explain-code, /refactor-to-compose.
  • Chain skills into meta-skills for end-to-end workflows like /ship-feature.
  • Skills are team knowledge codified — commit them, review them, evolve them like code.
