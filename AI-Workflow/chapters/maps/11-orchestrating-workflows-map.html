📦 Workflow Patterns

👑
/feature-pipeline
Ticket → branch → code → tests → lint → PR. The crown jewel of workflow orchestration.
🏷️
/release
Version bump → changelog → release notes → tests → tag → push. Full release in one shot.
🧹
/debt-sprint
Scan tech debt → pick top 5 quick fixes → fix → test → commit each. Safe, automated cleanup.
🔧
/weekly-maintenance
Dependency check → doc freshness → coverage gaps → tech debt count. Runs every Monday via cron.
🔀
/multi-repo-update
Coordinate changes across repos: same branch name, linked PRs, tracking issue.
📊
/automation-status
Read the automation log → summarize runs, success rate, failures, and suggestions.

🛡️ Error Recovery Patterns

🛑
Stop & Report
The safest default. Stop immediately, say what failed, list what succeeded, suggest the fix.
🔄
Retry with Backoff
For known-flaky operations (network, CI runners). Retry up to 3 times with increasing delay.
🔀
Fallback
Full suite times out? Run only changed-file tests. Note the fallback in the PR description.

📈 Orchestration Maturity Model

  1. Manual

    Every step done by hand. Where you started.

  2. Single Skills

    Each step is a /command, but you invoke them one at a time.

  3. Chained Workflows

    A master skill executes multiple steps in sequence. Most workflows belong here.

  4. Event-Driven

    Workflows trigger on events (PRs, issues, schedules) without human initiation.

  5. Self-Maintaining

    Automations monitor each other, report health, flag when updates are needed. Reserved for critical, stable processes.

🚫 When NOT to Automate

🔄
Process Still Changing
If your workflow changes weekly, automating locks in something that isn’t ready to be locked in.
🤔
Every Case Is Unique
Tasks requiring different judgment calls each time make rigid pipelines or overly complex ones.
📉
Cost Exceeds Savings
Quarterly tasks don’t need a pipeline. A one-liner with claude -p is enough.
💥
Failure Is Catastrophic
Production deploys need extensive guardrails. No guardrails yet? Keep a human in the loop.

📖 Key Terms

Pipeline / Workflow
A master skill that executes multiple automation steps in sequence to complete an end-to-end process.
Scheduled Agent
A Claude Code script run on a cron schedule — e.g., weekly maintenance, daily digest.
Remote Trigger
A workflow invoked by an external event — GitHub issue created, Slack message, webhook.
Stop & Report
The default error pattern: halt on failure, report what broke, suggest the fix.
Retry with Backoff
Re-attempt flaky operations with increasing delay between attempts (5s, 10s, 15s).
Fallback
Switch to a lighter alternative when the primary step fails (e.g., run changed-file tests only).
Multi-Repo Automation
Coordinating changes across repositories with same branch names, linked PRs, and tracking issues.
Automation Monitoring
Logging every automation run to a file and reviewing health with an /automation-status skill.
Subagent
A specialized agent in .claude/agents/<name>.md with its own context, tool allowlist, and model. Invoked by @agent-name or auto-delegated.
claude-code-action@v1
Anthropic’s official GitHub Action. Bundles install + auth + run. Inputs: prompt, claude_args, anthropic_api_key.
--tools vs --allowedTools
--tools restricts which tools are available; --allowedTools auto-approves listed tools (skips the prompt). --disallowedTools is an explicit blocklist.

🧪 Quick Quiz

What analogy does the chapter use for orchestration?

Cooking. You’ve learned to chop, sauté, and season — now follow a recipe that turns techniques into a complete meal.

What is the “crown jewel” workflow?

The feature-to-PR pipeline — /feature-pipeline PROJ-1234 goes from ticket to ready-for-review PR in one command.

What’s the safest default error recovery pattern?

Stop and report. Halt immediately, say what failed, list what succeeded, suggest the fix. Automatic retries can hide real problems.

What’s the difference between scheduled agents and remote triggers?

Scheduled agents run on a clock (cron). Remote triggers run on events (issue created, webhook, Slack message).

Where should most workflows live on the maturity model?

At “chained workflows” or “event-driven.” Self-maintaining is only for your most critical, most stable processes.

When do you reach for a subagent over a composed skill?

When a step would flood your main context with logs or search results, when you want a different tool allowlist or model for that step (e.g., Haiku for cheap work), or when a clearly-scoped specialist (code-reviewer, security-auditor) gets reused often. Subagents live in .claude/agents/.

What’s the modern way to run Claude Code in GitHub Actions?

Use anthropics/claude-code-action@v1 with prompt and claude_args inputs. It replaces manual npm install -g @anthropic-ai/claude-code + claude -p patterns. Pair with actions/checkout@v6 and actions/github-script@v9.

Key Takeaways

  • Chain skills, hooks, CI/CD, and MCP servers into multi-step pipelines that handle entire processes.
  • The feature-to-PR pipeline is the crown jewel: ticket → branch → code → tests → lint → PR in one command.
  • Three trigger types: manual (/command), scheduled (cron), and event-driven (webhooks, GitHub events).
  • Default to stop & report on failure. Only retry known-flaky operations.
  • Most workflows belong at “chained” or “event-driven” maturity. Self-maintaining is for critical, stable processes only.
  • Don’t automate processes that change weekly, require unique judgment, cost more than they save, or fail catastrophically.
  • Always review pipeline output — the pipeline handles mechanics, you provide judgment.
  • For event-driven workflows, prefer anthropics/claude-code-action@v1 over manual install + claude -p.
  • When a step needs context isolation, its own tool allowlist, or a cheaper model, reach for a subagent (.claude/agents/) instead of another skill in the chain.
