Chapter 15

Claude Code on the Web and Mobile

Memory Map
🌐

The Big Idea

Three surfaces, one session model. --remote creates cloud sessions; --teleport pulls them home; Remote Control mirrors a local session to your phone. Anything in your repo’s .claude/ runs identically across surfaces.

📡 The Three Surfaces

💻
Terminal
The original. Full filesystem, hooks, active dev. Best for fast iteration on the same files.
🌐
Web (claude.ai/code)
Anthropic-managed sandboxed cloud. Fresh VM per session, repo cloned, common toolchains pre-installed. 4 vCPU / 16 GB RAM / 30 GB disk.
📱
Mobile
Anthropic app. Monitor cloud sessions, kick off new ones, approve auto-fix actions, Remote Control your laptop session.

🔀 Three Primitives, Three Directions

PrimitiveDirectionWhat it does
--remote Terminal → cloud Creates a new cloud session running on Anthropic infra
--teleport Cloud → terminal Pulls an existing cloud session into your local CLI
/remote-control Local → phone/web Mirrors a local CLI session for monitoring elsewhere

The trap: --remote creates cloud, --remote-control mirrors local. Six shared letters, opposite jobs.

📦 What the Cloud VM Has — and Doesn’t

✅
Comes Along (in repo)
Repo CLAUDE.md, .claude/settings.json, .claude/skills/, .claude/agents/, .claude/commands/, .mcp.json, plugins declared in repo settings.
❌
Stays Home (user-level)
~/.claude/CLAUDE.md, user MCPs added with claude mcp add, user-only plugins, static API tokens (no secrets store yet).

The cloud is the team-sharing test. If your automation doesn’t work in a fresh cloud session, it doesn’t work for new hires either.

🛡️ The Sandbox

  1. OS-level primitives

    Linux bubblewrap, macOS seatbelt. Real OS isolation, not just convention.

  2. Filesystem boundary

    Default writes only to cwd + subdirs. Reads broader by default. Tighten via /sandbox.

  3. Network boundary

    Four levels: None, Trusted (default allowlist), Full, Custom.

  4. 84% fewer permission prompts

    Anthropic’s internal usage. Hooks no longer interrupt for every shell call.

Defense in depth, not magic. CVE-2026-39861 (March 2026) showed sandbox escapes are real. Treat like AppArmor.

🚧 What Doesn’t Work in the Cloud

🔐
Interactive Auth
AWS SSO, browser-based logins. No browser to redirect to. Use static tokens.
🐘
Memory-Heavy Builds
16 GB RAM ceiling. Bazel monorepos may not fit. Use Remote Control on your hardware.
🐳
Custom Base Images
Can’t replace cloud VM’s base. Install on top via setup script, or run as sidecar with docker compose.
🏠
User-Only Config
~/.claude/ is invisible to cloud. Move things into the repo to bring them along.

📖 Key Terms

--remote
CLI flag that creates a NEW cloud session on Anthropic infra. The cloud VM clones your branch from GitHub.
--teleport / /teleport / /tp
CLI command that pulls an existing cloud session into your terminal. One-way: cloud → local.
Remote Control
Feature (Feb 2026) that mirrors a local CLI session to phone/web for monitoring. Local → mobile.
Routine
Scheduled cloud session. Cron-style automation that doesn’t depend on your laptop being awake. Configured via /schedule.
Auto-fix
Cloud session subscribed to a PR’s GitHub events. Reads CI failures and review comments, makes clear fixes automatically.
Setup script
Bash script attached to a cloud environment, runs once per environment, cached ~7 days. For installing cloud-only tools.
$CLAUDE_CODE_REMOTE
Env var set to true in cloud sessions. Use it in SessionStart hooks to skip local execution.
Trusted network access
Default cloud network level. Curated allowlist of package registries, GitHub, cloud SDKs. Most workflows just work.

🧪 Quick Quiz

What’s the difference between --remote and --remote-control?

--remote creates a cloud session on Anthropic infrastructure. --remote-control mirrors a local CLI session to your phone/web. Six shared letters, opposite directions.

Which way does --teleport move a session?

Cloud → terminal. One-way pull. You can’t push a terminal session to the cloud with this flag. Use --remote to start a new cloud session instead.

What lives in your ~/.claude/ that won’t make it to the cloud?

User-level CLAUDE.md, user-scoped plugins, MCP servers added with claude mcp add, and any auth config. Move these to repo-level (CLAUDE.md, .claude/settings.json, .mcp.json) to bring them along.

What two boundaries does the sandbox enforce?

Filesystem (default writes to cwd only) and network (proxy with allowlist levels: None / Trusted / Full / Custom). Built on Linux bubblewrap and macOS seatbelt.

When should you use a setup script vs. a SessionStart hook?

Setup script for cloud-specific things (lives in cloud env UI, cached ~7 days, runs only in cloud). SessionStart hook for project setup that should run everywhere (lives in .claude/settings.json, runs every start in both cloud and local).

What’s the cloud VM resource ceiling?

4 vCPUs, 16 GB RAM, 30 GB disk. Plenty for most projects, not enough for Bazel-on-monorepo. For workloads above the ceiling, use Remote Control on your own hardware.

Key Takeaways

  • Three surfaces: terminal, web (claude.ai/code), mobile app. Sessions move between them with named primitives.
  • --remote = create cloud session. --teleport = pull cloud session local. /remote-control = mirror local to phone. Three different jobs.
  • Cloud sessions persist across browser closes and laptop sleep. --resume reopens local history; --teleport picks up cloud sessions.
  • Anything in .claude/ in the repo (skills, agents, hooks, plugins, MCP) runs identically across surfaces. ~/.claude/ stays home.
  • Sandbox = OS-level isolation (bubblewrap / seatbelt). 84% fewer permission prompts. Defense in depth, not magic — sandbox escapes have happened.
  • Use cloud for long-running, parallel, or laptop-can’t-spare work. Use local for fast iteration and anything outside the repo.
  • Routines + auto-fix + mobile = workflow continuity without you. PRs progress while you commute.
  • Cloud Claude Code and the GitHub Action are complementary. CI = Action; ad-hoc heavy lifting = Cloud.
