Hooks That Work For You

Memory Map

The Big Idea

Hooks are motion sensors for your dev workflow. They fire when Claude Code does something — lint on edit, block force-push, run tests on change — and your script reads the event details as JSON on stdin. You never have to remember; it just happens.

🪝 Hook Types You’ll Use Most

Type When Blocks? Best For
🛡️ PreToolUse Before a tool runs ✅ Yes Guardrails, validation, blocking danger
✅ PostToolUse After a tool runs ❌ No Lint, format, test, log
📝 UserPromptSubmit You submit a prompt ✅ Yes Prompt rewriting, guardrails, context injection
🚀 SessionStart Session begins/resumes ❌ No Env vars, project warm-up, loading state
🏁 SessionEnd Session terminates ❌ No Final logs, cleanup, metrics
🗜️ PreCompact Before context compacts ✅ Yes Persisting context before it’s summarized
🔔 Notification Claude notifies ❌ No Slack alerts, desktop notifications
🤖 SubagentStop Subagent finishes ❌ No Collecting parallel results

More events exist (PostCompact, UserPromptExpansion, CwdChanged, …) — see the hooks reference.

🎯 Matchers: Targeting Specific Tools

"matcher": "Edit"
Fires only when Claude Code modifies an existing file.
"matcher": "Write"
Fires when Claude Code creates a new file.
"matcher": "Edit|Write"
Fires on either — the most common combo for lint/format hooks.
"matcher": "Bash"
Fires on shell commands — use with PreToolUse for guardrails.
No matcher
Fires on every tool use. Rarely what you want; be specific.

Two ways to block: exit 2 with stderr, or exit 0 with JSON {permissionDecision: "deny"}. JSON also supports "allow", "ask", and rewriting input.

⚡ Hook Performance Rules

  1. PreToolUse: Under 100ms

    Runs before every matched action. Simple grep/regex only. Never Gradle or network calls.

  2. PostToolUse: Under 5 seconds

    Lint one file, not the project. Run one test class, not the suite. Use --quiet.

  3. Stop: Can take longer

    Runs once at the end. Good for full test suites, summaries, and session logs.

  4. Always: Early exit when irrelevant

    Check file extension first. If it’s not Kotlin, exit immediately. Don’t waste time on unrelated files.

🐛 Debugging Hooks

📝
Add Logging
Capture stdin once: INPUT=$(cat); echo "$INPUT" >> /tmp/claude-hooks.log — then re-feed it to your real logic.
🧪
Test in Isolation
Pipe a fake JSON payload: echo '{"tool_input":{"file_path":"x.kt"}}' | ./scripts/lint-changed.sh
🔑
Check Permissions
Script not running? chmod +x it. Verify the path is relative to project root.
🎯
Narrow the Matcher
Blocking too much? Make grep patterns more specific. Add early exits for safe commands.

🧪 Quick Quiz

Which hook types can block actions, and how?

PreToolUse, UserPromptSubmit, and PreCompact can all block. Two patterns: exit 2 with a stderr message (traditional), or exit 0 and emit JSON like {"hookSpecificOutput": {"permissionDecision": "deny", "permissionDecisionReason": "…"}} (modern; also supports allow, ask, and rewriting input).

How does a hook script get the file path Claude just edited?

It reads JSON from stdin and parses the tool_input.file_path field with jq: FILE=$(jq -r '.tool_input.file_path'). There is no $CLAUDE_FILE env var.

What matcher should you use for a lint-on-edit hook?

"matcher": "Edit|Write" — catches both file modifications and new file creation.

Why should PreToolUse hooks run under 100ms?

They run synchronously before every matched tool use. A slow hook pauses Claude Code on every single edit or command.

What’s the home automation analogy for hooks?

Hooks are motion sensors. You don’t flip the porch light every evening — the sensor does it automatically when it detects the event.

Where do team guardrail hooks belong — project or user settings?

Project-level (.claude/settings.json) so they’re checked into git and enforced for everyone.

📌 Key Takeaways

  • Hooks = event listeners. settings.json schema: { matcher, hooks: [{ type: "command", command }] }.
  • Hook scripts read a JSON payload from stdin — parse with jq (.tool_input.file_path, .tool_input.command, …). There is no $CLAUDE_FILE or $CLAUDE_COMMAND — only $CLAUDE_PROJECT_DIR and friends.
  • Many event types: PreToolUse, PostToolUse, UserPromptSubmit, SessionStart/End, PreCompact, Notification, SubagentStop, …
  • PreToolUse blocks two ways: exit 2, or JSON permissionDecision: "deny". JSON also supports "allow", "ask", and rewriting input.
  • PostToolUse is your quality net — auto-lint, auto-format, auto-test after every edit.
  • Keep hooks fast: Pre <100ms, Post <5s. Lint one file, test one class, early-exit on irrelevant files.
  • Team standards go in project hooks. Personal prefs go in user hooks.
  • Test scripts in isolation by piping fake JSON: echo '{"tool_input":{...}}' | ./script.sh.
