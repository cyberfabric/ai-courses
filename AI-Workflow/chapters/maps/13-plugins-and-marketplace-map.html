Plugins & the Marketplace

Memory Map
🧩

The Big Idea

A plugin is a directory plus a tiny manifest. A marketplace is a JSON catalog of plugins. Together they turn your personal collection of skills, hooks, MCP servers, and subagents into something a teammate can install with one command and update with another.

📦 Anatomy of a Plugin

📋
.claude-plugin/plugin.json
The only required file. Only name is mandatory. version, description, author, license, repository, keywords are optional but recommended.
⚡
commands/
Slash commands as bare-markdown files. Auto-discovered.
📦
skills/<name>/SKILL.md
Frontmatter-gated, model-invokable skills with optional supporting files.
🤖
agents/
Subagents (Chapter 14) — specialized agents with own context, tools, model.
🪝
hooks/
Hook scripts. Reference them from hooks.json using ${CLAUDE_PLUGIN_ROOT} for portable paths.
🔌
.mcp.json
MCP server registrations bundled with the plugin.

⚙️ Install Workflow

  1. Add a Marketplace

    /plugin marketplace add owner/repo — or git URL, local path, or remote URL.

  2. Browse

    /plugin opens the manager. Discover tab shows everything across all your marketplaces.

  3. Install

    /plugin install <name>@<marketplace> — or pick interactively. Choose scope: user / project / local.

  4. Update

    /plugin update — refetches plugins whose version in plugin.json has changed.

📍 Install Scopes

Scope Where Use For
user (default) Home dir, all your projects Personal skills, productivity tools
project Committed to the repo Team-shared automations, peer-reviewed via PRs
local This repo, your machine only Trying things out before sharing

🏛️ The Two-Repo Pattern

📋 marketplace.json (catalog only)
📦 plugin-A repo
🏛️ Acme Plugins Marketplace
📦 plugin-B repo
📦 plugin-C repo

Marketplace repo holds the catalog; plugin repos hold the code. Lets the platform team curate without forking, and lets plugin authors ship updates without coordination.

⚠️ Security Realities (2026)

🚫
No Binary Signing
Plugins aren’t cryptographically signed. The marketplace owner is your trust anchor.
🔍
Static Analysis Only
You can read hook scripts, but a plugin that uses an LLM at runtime can do whatever the runtime LLM tells it to.
📖
Read the Manifest
Skim the plugin.json and hook scripts before installing. 2 minutes of review = best safeguard.
📌
Pin or Trust
Either pin a specific version in your marketplace entry or trust the maintainer ongoingly. Pick one consciously.

🤔 Build, Install, or Both?

📥
Install
When a vetted plugin already does what you need. claudemd-keeper, browser-control, repomix — don’t reinvent these.
🛠️
Build
When the automation is team-specific: your conventions, your tools, your workflows. The artifacts you wrote in Ch 3, 4, 10 belong here.
🪞
Fork
When a community plugin is 80% right. Fork, adjust, version with a custom suffix (1.2.0-acme), publish to your team marketplace.

🏷️ Versioning Discipline

BumpExampleWhen
Patch0.3.0 → 0.3.1Bug fix, typo, prompt tweak
Minor0.3.1 → 0.4.0New skill / hook / MCP server. Backwards compatible.
Major0.4.0 → 1.0.0Renamed or removed a slash command. Breaks callers.

Forget to bump version → users won’t see your fix. Caching footgun.

📖 Key Terms

Plugin
A directory containing .claude-plugin/plugin.json plus any combination of commands/, skills/, agents/, hooks/, and .mcp.json.
Marketplace
A JSON catalog (.claude-plugin/marketplace.json) listing plugins and where to fetch them.
claude-plugins-official
Anthropic’s managed marketplace. Auto-available — no marketplace add needed.
Plugin namespacing
The plugin-name:skill-name syntax. Used to disambiguate when two plugins expose the same slash command. The only place a colon appears in slash-command names.
$CLAUDE_PLUGIN_ROOT
Env var that resolves to a plugin’s install directory at runtime. Use it in hooks.json for portable script paths.
Two-repo pattern
Splitting marketplace JSON and plugin source into separate repos. Lets curation and plugin development evolve independently.
Install scope
Where a plugin is installed: user (across projects), project (committed to repo), or local (this repo, your machine only).

🧪 Quick Quiz

What’s the only required field in plugin.json?

name. Everything else is optional, including the manifest itself in some cases — though you should always set version so updates work.

Where does the manifest file actually live?

.claude-plugin/plugin.json — with a hyphen, not .claude/plugin.json. Marketplace catalogs follow the same pattern: .claude-plugin/marketplace.json.

What does the colon in android-toolkit:pr-description mean?

Plugin namespacing. It picks the /pr-description from the android-toolkit plugin specifically, in case another plugin or your own .claude/commands/ exposes the same name.

What happens if you ship a fix but forget to bump version?

Users won’t see it. Claude Code caches resolved plugin contents and only re-fetches when version changes. Bump the patch number on every meaningful change.

Why split the marketplace JSON and the plugin code into two repos?

It lets the platform team revise the approval list without touching plugin source, and lets plugin authors ship updates without coordinating with the marketplace owner. The marketplace stays small and stable; plugin repos churn at their own pace.

Which install scope is best for team-shared plugins?

Project scope. The plugin gets committed to the repo, so every change goes through PR review. Anyone who clones the repo gets it after running /plugin sync.

Key Takeaways

  • A plugin is a directory + .claude-plugin/plugin.json. Skills, hooks, agents, MCP servers auto-discover from standard subdirectories.
  • A marketplace is a JSON catalog — .claude-plugin/marketplace.json — not a code repo. Marketplace points at plugins.
  • Install via /plugin install <name>@<marketplace>. Three scopes: user (default), project (committed), local (this machine only).
  • Bump version on every meaningful change. Forget → users don’t see the fix.
  • Use the two-repo pattern for team marketplaces: catalog in one repo, plugin source in others. Curation and code evolve separately.
  • Plugin namespacing (plugin-name:skill-name) is the only colon-syntax in slash commands. The old subdirectory rumor was wrong.
  • No binary signing in 2026. Read the manifest. Trust the maintainer. Pin versions when stakes are high.
  • Plugins don’t replace Ch 3–11 — they bundle them. Same skills, same hooks, same MCP servers; one install command.
