CI/CD Meets AI

Claude Code in your pipeline — changelogs, reviews, and deployment checks that run themselves

Taking AI Out of Your Terminal

Everything up to this point has run on your machine, in your Claude Code session. You type a command, you get a result. Powerful, but limited — it only works when you’re at the keyboard.

This chapter moves Claude Code into the cloud. Into GitHub Actions workflows that trigger on every push, every PR, every scheduled cron. Into pipelines that run while you’re asleep, reviewing PRs, writing changelogs, diagnosing build failures, and checking deployment readiness.

Your pipeline is about to get an AI brain.

Why Put AI in Your Pipeline?

100% of PRs reviewed (not just the ones a human gets to)
3 min from push to AI review comment
0 changelogs forgotten at release time

CI/CD AI automation isn’t about replacing your existing pipeline. It’s about adding intelligence to the steps that currently require human judgment. Your linter already runs in CI. Now your AI reviewer does too.

The Easy Path: anthropics/claude-code-action@v1

The shortest path to Claude Code in CI is Anthropic’s official GitHub Action. It runs Claude Code, posts comments to PRs natively, and handles authentication — no manual install, no separate actions/github-script step.

- uses: anthropics/claude-code-action@v1
  with:
    anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
    prompt: |
      Review this PR's diff for bugs, security issues, and Android pitfalls.
      Post findings as PR review comments.
    claude_args: --max-turns 5 --tools "Read,Grep,Bash(git *)"

Two key inputs:

  • prompt — instructions for Claude. Plain text or a skill name (your /scaffold-feature and /review skills work in CI too).
  • claude_args — anything you’d pass to the claude CLI: --max-turns, --model, --tools, --mcp-config, …

The action auto-detects mode. With an issue_comment trigger, it listens for @claude mentions; with a pull_request trigger and a prompt: set, it runs immediately. No mode parameter to configure.

Quick setup: run /install-github-app in your Claude Code terminal. It installs the GitHub App, sets up the API key secret, and writes a starter workflow — all in one go.

The Manual Approach (For Raw Control)

The recipes in this chapter mostly use the manual pattern: install Claude Code via npm, then invoke claude -p directly. That pattern still works and gives you raw control — piping output through jq, branching on exit codes, chaining multiple Claude calls in one job. Reach for it when the official action’s surface area isn’t enough.

# The universal setup step
- name: Install Claude Code
  run: npm install -g @anthropic-ai/claude-code

# The universal execution pattern
- name: Run Claude Code
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
  run: claude -p "your prompt here" --tools "Read,Grep" > output.md

Pick the right tool flag. --tools "Read,Grep" restricts which tools are available at all — the safest pattern for CI. --allowedTools "Read,Grep" auto-approves those tools but doesn’t block others; use it when you want fewer permission prompts but still want every tool reachable. --disallowedTools "Bash(rm *),Write" explicitly blocks specific tools and takes precedence over allow-rules — great for “everything except” setups. For deny-by-default review jobs, prefer --tools.

Automated Changelog Generation

Nobody likes writing changelogs. But users and stakeholders love reading them. Let’s automate the entire thing.

Auto-Generate CHANGELOG.md on Every Release

  1. Create the workflow file: .github/workflows/changelog.yml
  2. Add this content:
name: Generate Changelog
on:
  push:
    tags:
      - 'v*'  # Triggers on version tags like v1.2.0

jobs:
  changelog:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Get previous tag
        id: prev_tag
        run: |
          PREV=$(git describe --tags --abbrev=0 HEAD^ 2>/dev/null || echo "")
          echo "tag=$PREV" >> "$GITHUB_OUTPUT"

      - name: Generate changelog entry
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          RANGE="${{ steps.prev_tag.outputs.tag }}..HEAD"
          git log --oneline $RANGE | claude -p \
            "Write a changelog entry for version ${{ github.ref_name }}.
             Group changes into: Features, Bug Fixes, Performance,
             Breaking Changes (if any).
             Use bullet points. Be concise but specific.
             Start with ## ${{ github.ref_name }} - $(date +%Y-%m-%d)" \
            --tools "" \
            > new-entry.md

      - name: Prepend to CHANGELOG
        run: |
          if [ -f CHANGELOG.md ]; then
            cat new-entry.md <(echo -e "\n") CHANGELOG.md > temp.md
            mv temp.md CHANGELOG.md
          else
            mv new-entry.md CHANGELOG.md
          fi

      - name: Commit and push
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add CHANGELOG.md
          git commit -m "docs: update changelog for ${{ github.ref_name }}"
          git push origin HEAD:main
  1. Add your ANTHROPIC_API_KEY to GitHub repository secrets
  2. Tag a release: git tag v1.3.0 && git push --tags
  3. Watch the changelog appear automatically

Release Notes from Commits

Changelogs are for developers. Release notes are for users. Same data, different audience:

- name: Generate release notes
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
  run: |
    git log --oneline ${{ steps.prev_tag.outputs.tag }}..HEAD | claude -p \
      "Write user-facing release notes for a mobile app.
       Audience: non-technical users.
       Tone: friendly, clear, no jargon.
       Group into: What's New, Improvements, Bug Fixes.
       Max 8 bullet points total — combine related changes.
       Each bullet should explain the user benefit, not the code change.
       Example: 'Faster app launch time' not 'Optimized DI initialization'" \
      --tools "" \
      > release-notes.md

Version Bump Automation

If you use conventional commits (feat:, fix:, breaking:), Claude Code can determine the right version bump:

- name: Determine version bump
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
  run: |
    CURRENT_VERSION=$(cat version.txt)
    git log --oneline ${{ steps.prev_tag.outputs.tag }}..HEAD | claude -p \
      "Current version: $CURRENT_VERSION
       Based on these conventional commits, determine the semantic version bump:
       - feat: → minor bump
       - fix: → patch bump
       - Any BREAKING CHANGE or ! after type → major bump

       Output ONLY the new version number, nothing else." \
      --tools "" \
      > new-version.txt

Build Failure Diagnosis

When CI fails, someone has to read the logs, understand the error, and figure out the fix. Usually that takes 10–20 minutes of scrolling through verbose output. Let AI do the scrolling:

name: Diagnose Build Failure
on:
  workflow_run:
    workflows: ["Build"]
    types: [completed]

jobs:
  diagnose:
    if: ${{ github.event.workflow_run.conclusion == 'failure' }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6

      - name: Download build logs
        uses: actions/github-script@v9
        with:
          script: |
            const logs = await github.rest.actions.downloadWorkflowRunLogs({
              owner: context.repo.owner,
              repo: context.repo.repo,
              run_id: context.payload.workflow_run.id
            });
            require('fs').writeFileSync('build-logs.txt',
              Buffer.from(logs.data));

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Diagnose failure
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          cat build-logs.txt | tail -500 | claude -p \
            "This CI build failed. Analyze the logs and report:
             1. **Root cause**: What specifically failed and why
             2. **Error location**: Which file/line/test
             3. **Suggested fix**: Exact steps to resolve
             4. **Category**: compilation / test failure / dependency /
                config / infrastructure

             Be specific. Quote the relevant error messages." \
            --tools "" \
            > diagnosis.md

      - name: Post diagnosis to PR
        if: github.event.workflow_run.pull_requests[0]
        uses: actions/github-script@v9
        with:
          script: |
            const fs = require('fs');
            const diagnosis = fs.readFileSync('diagnosis.md', 'utf8');
            const pr = context.payload.workflow_run.pull_requests[0];
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: pr.number,
              body: `## Build Failure Diagnosis\n\n${diagnosis}\n\n---\n*Automated by Claude Code*`
            });

Build failure diagnosis changes the feedback loop. Instead of a developer seeing “CI failed” and spending 15 minutes figuring out why, they see a comment that says “Compilation failed in UserRepository.kt line 42 because the new ApiResult.Loading state isn’t handled in the when expression. Add the missing branch.” Fix time drops from 15 minutes to 2 minutes.

Scheduled Agents

Not everything should run on push or PR. Some tasks need to run on a schedule — daily, weekly, or monthly. Claude Code supports scheduled agents that run on cron:

name: Weekly Maintenance
on:
  schedule:
    - cron: '0 9 * * 1'  # Every Monday at 9 AM UTC

jobs:
  maintenance:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Dependency audit
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          ./gradlew dependencies --configuration releaseRuntimeClasspath \
            > deps.txt 2>&1
          cat deps.txt | claude -p \
            "Analyze these Android project dependencies:
             1. Flag any with known CVEs (check version numbers)
             2. Flag any that are 2+ major versions behind latest
             3. Flag deprecated libraries with suggested replacements
             4. Summarize as a markdown report" \
            --tools "" \
            > dep-report.md

      - name: Create issue if findings
        uses: actions/github-script@v9
        with:
          script: |
            const fs = require('fs');
            const report = fs.readFileSync('dep-report.md', 'utf8');
            if (report.includes('CVE') || report.includes('deprecated')) {
              github.rest.issues.create({
                owner: context.repo.owner,
                repo: context.repo.repo,
                title: `Weekly dependency audit — ${new Date().toISOString().split('T')[0]}`,
                body: report,
                labels: ['maintenance', 'dependencies']
              });
            }

Other scheduled agent ideas

  • Documentation freshness check — weekly audit of docs vs. code
  • Test coverage report — weekly coverage gap detection with /find-coverage-gaps
  • Tech debt inventory — monthly scan for TODOs, FIXMEs, and deprecated API usage
  • Security scanning — daily check for new CVEs in your dependency tree

Deployment Pre-Checks

Before a release goes out, verify that everything is in order:

- name: Deployment readiness check
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
  run: |
    claude -p "Deployment readiness check for Android release.
      Verify:
      1. Read version.txt — is version bumped from last tag?
      2. Read CHANGELOG.md — does it have an entry for this version?
      3. Check git log for any 'WIP' or 'fixup' commits
      4. Check AndroidManifest.xml — any debug-only permissions left?
      5. Check build.gradle.kts — is debuggable set to false for release?
      6. Any TODO comments with 'BEFORE_RELEASE' markers?

      Output a GO / NO-GO decision with details." \
      --allowedTools "Read,Bash(git log*),Bash(grep*)" \
      > readiness.md

Be careful with CI costs. Claude Code API calls cost money. A review on every push to every branch adds up. Cost-control levers, in order of impact:

  • Lean on prompt caching. Claude Code automatically caches stable prompt prefixes (CLAUDE.md, skill content, large diffs you reference repeatedly). Cache hits cost about 10% of standard input price, so a CI job that re-runs the same review prompt across many PRs gets dramatically cheaper. The current TTL is 5 minutes, so back-to-back jobs benefit most.
  • Cap iterations. Pass --max-turns 5 (or lower) in claude_args to stop runaway loops. Most CI prompts only need 1–3 turns.
  • Filter triggers. Run AI review only on PRs to main, not every push to every branch. Use paths: filters to skip docs-only changes.
  • Use the right model size. Sonnet for review, Haiku for simple classification (e.g. version-bump decisions). Override per-job via claude_args: --model claude-haiku-4-5.
  • Add concurrency limits. A concurrency: { group: review-${{ github.ref }}, cancel-in-progress: true } block stops parallel runs on the same PR from piling up.
  • Set workflow timeouts. timeout-minutes: 10 on the job is your circuit breaker.

Monitor your Anthropic API usage weekly and set billing alerts on the console.

    git log --oneline $PREV_TAG..HEAD | claude -p \
      "Write Google Play Store release notes.
       Requirements:
       - Max 500 characters

       - No technical jargon
       - Focus on top 3-4 changes users will notice
       - End with a positive note encouraging updates" \
      --tools "" \
      > play-store-notes.txt

PR merge automation

name: Auto-Merge
on:
  pull_request_review:
    types: [submitted]

jobs:
  auto-merge:
    if: github.event.review.state == 'approved'
    runs-on: ubuntu-latest
    steps:
      - name: Check all status checks passed
        uses: actions/github-script@v9
        with:
          script: |
            const { data: checks } = await github.rest.checks.listForRef({
              owner: context.repo.owner,
              repo: context.repo.repo,
              ref: context.payload.pull_request.head.sha
            });
            const allPassed = checks.check_runs.every(
              c => c.conclusion === 'success' || c.conclusion === 'skipped'
            );
            if (allPassed) {
              await github.rest.pulls.merge({
                owner: context.repo.owner,
                repo: context.repo.repo,
                pull_number: context.payload.pull_request.number,
                merge_method: 'squash'
              });
            }
