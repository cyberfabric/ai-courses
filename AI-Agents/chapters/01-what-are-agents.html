Chapter 1

What Are AI Agents, Really?

Beyond the hype — what agents actually are, what they aren’t, and why they matter now

The Agent Hype Cycle

If you’ve been anywhere near the tech world in 2025, you’ve heard the word “agent” roughly ten thousand times. Every startup has an “AI agent.” Every framework promises “agentic capabilities.” Every conference keynote ends with a demo of some autonomous system doing something impressive and slightly terrifying.

The problem? Nobody agrees on what the word actually means.

Some people use “agent” to mean a chatbot that can call a function. Others mean a fully autonomous system that can browse the web, write code, and deploy it — without a human in the loop. Still others use it as a marketing term for “our product, but with more AI.”

In this chapter, we’re going to cut through the noise. By the end, you’ll have a clear, precise definition of what an AI agent is, understand how it differs from the tools you already use, and know why this matters for your work as a developer.

So What IS an AI Agent?

Here’s the definition we’ll use throughout this book:

An AI agent is a system that uses a language model to observe its environment, reason about what to do, take actions using tools, and repeat this loop until a goal is achieved.

The key word is loop. A chatbot responds once. An agent keeps going.

Let’s break that down into the four essential pieces:

  • Observe — the agent perceives its current state: user instructions, tool outputs, error messages, file contents, API responses
  • Reason — the LLM thinks about what to do next, based on everything it has observed so far
  • Act — the agent executes a tool: runs a search, writes a file, calls an API, executes code
  • Loop — the agent checks the result, decides if the goal is met, and if not, goes back to step 1

This is the observe-think-act loop, and it’s the beating heart of every agent system ever built. Without the loop, you just have a chatbot. With it, you have something that can actually do work.

The Observe-Think-Act Loop

Every agent, from the simplest script to a multi-agent orchestra, follows this fundamental cycle:

Observe → Think → Act → Check → Repeat or Stop

This is analogous to an event loop in a server or a game loop in game development. The agent doesn’t just respond — it keeps running until the job is done or it hits a limit. The LLM is the “brain” that decides what happens at each step, but the loop itself is just regular code you write.

The Spectrum: From Chatbot to Autonomous System

Not everything with an LLM inside it is an agent. There’s a clear spectrum, and understanding where each tool sits on it will save you from a lot of confusion:

Type Who Steers? Can Use Tools? Loops? Example
Chatbot The conversation No (or limited) No Customer support chat, basic ChatGPT
Copilot The human Yes, but human-initiated Rarely GitHub Copilot autocomplete, inline chat
Agent The agent (with guardrails) Yes, autonomously Yes Claude Code, Cursor Agent Mode
Autonomous System The system itself Yes, with minimal oversight Yes, continuously Self-driving research agents, trading bots

The key distinction is who decides what happens next. In a chatbot, you type the next message. In a copilot, you accept or reject each suggestion. In an agent, the system decides what to do next — within boundaries you set.

Think of it like the difference between GPS navigation (copilot — suggests turns, you drive) and a self-driving car (agent — drives itself, you set the destination). The self-driving car still has limits — it won’t drive off a cliff — but it makes the moment-to-moment decisions autonomously.

What Makes an Agent an Agent: The Five Capabilities

A system needs most of these capabilities to qualify as an agent:

1. Tool Use

The agent can call external functions — search the web, read files, execute code, make API calls. Without tools, an LLM can only generate text. With tools, it can do things.

2. Planning

The agent can break a complex goal into steps and decide the order of execution. It doesn’t just react — it strategizes.

3. Memory

The agent remembers what it’s already done, what worked, what failed, and what the user said earlier. This can be as simple as the conversation history or as complex as a vector database.

4. Self-Correction

When something goes wrong (and it will), the agent can detect the error, adjust its approach, and try again. A chatbot just gives you a wrong answer. An agent notices and fixes it.

5. Autonomous Execution

The agent runs multiple steps without requiring human input at each step. You give it a goal; it figures out the path.

Don’t confuse “can call a function” with “is an agent.” A single API call that runs a tool and returns a result is just function calling. An agent uses function calling inside a loop, deciding which functions to call, evaluating results, and continuing until the task is done. The loop is what makes it agentic.

See It in Action: An Agent Trace

The best way to understand an agent is to watch one work. Here’s a trace of a coding agent (like Claude Code) handling a simple bug fix request:

Think The user wants me to fix the login bug. Let me first find the relevant code by searching for the authentication logic.
Tool grep -r "authenticate" src/ --include="*.py"
Result Found src/auth/login.py:42 — the authenticate() function
Think Let me read this file to understand the current implementation and find the bug.
Tool read_file("src/auth/login.py")
Result Found it — line 47 compares password instead of hashed_password. That’s the bug.
Think I need to fix the comparison to use the hashed version. Let me also check if there are tests I should update.
Tool edit_file("src/auth/login.py", line=47, new="if verify_hash(password, stored_hash):")
Result File updated successfully.
Tool run_tests("tests/test_auth.py")
Result All 12 tests pass.
Respond Fixed the login bug — line 47 was comparing plaintext instead of hashed passwords. Tests pass.

Notice what happened: the agent made six decisions without the user doing anything. It searched, read, analyzed, fixed, tested, and reported. That’s the observe-think-act loop in practice.

Why Agents Became Possible Now

Developers have dreamed about autonomous coding assistants since the 1960s. So why did agents suddenly become real in 2024–2025? A few breakthroughs converged:

1M Token context windows (Claude, Gemini) — agents can hold entire codebases in memory
92% Of US developers now use AI coding tools (GitHub Octoverse 2025)
97M+ Monthly MCP SDK downloads — tools are becoming standardized

Reasoning Models

Models like Claude Opus 4, GPT-4o, and the “reasoning” series (o1, o3, o4-mini) can now think through multi-step problems before acting. Earlier models would jump to conclusions; modern models can plan.

Formalized Tool Use

In 2023, calling tools from an LLM was a hack — you’d parse JSON from markdown output and pray. By 2025, every major API has native tool use: structured JSON schemas, guaranteed valid calls, and built-in support for multiple tool calls in a single response.

The MCP Standard

Anthropic introduced the Model Context Protocol (MCP) in November 2024, and it became the “USB-C for AI tools.” MCP gives agents a universal way to connect to tools — databases, APIs, file systems — without custom integration code for each one. By early 2026, MCP had been donated to the Linux Foundation’s new Agentic AI Foundation, with backing from OpenAI, Google, Microsoft, and AWS. It’s the HTTP of the agent era.

Production Infrastructure

Monitoring, tracing, guardrails, and deployment tools caught up. You can now observe what an agent is doing, set budgets, and kill it when things go wrong. This moved agents from “cool demo” to “thing you can actually deploy.”

Agent vs Workflow: When You Actually Need an Agent

Here’s an uncomfortable truth: most tasks don’t need an agent. A well-written script, a CI/CD pipeline, or a simple API call will handle 80% of automation needs. Agents are for the other 20% — the messy, ambiguous, judgment-heavy stuff.

The “Could a Bash Script Do This?” Test

Before building an agent, ask yourself: “Could I solve this with a deterministic script?”

  • If yes → use the script. It’s faster, cheaper, and more predictable.
  • If “yes, but it would be 10,000 lines of if/else” → consider an agent. The LLM handles the judgment calls those if/else blocks are trying to encode.
  • If “no, because the task requires understanding context and making judgment calls” → this is agent territory.

Use this pattern when: deciding whether to build an agent vs a traditional automation. The goal isn’t to use agents everywhere — it’s to use them where they actually outperform deterministic code.

Good use of an agent
“Review this PR for security issues, code quality, and style violations. Suggest fixes and explain your reasoning.”
Requires reading code, understanding context, applying judgment, and generating natural language explanations. A script can’t do this.
Bad use of an agent
“Run our linter and format the code with prettier.”
This is two CLI commands. An agent would be slower, more expensive, and less reliable than eslint . && prettier --write .

The Risks: Why Agents Need Guardrails

With great autonomy comes great potential for disaster. Here’s what can go wrong:

Hallucination with consequences. When a chatbot hallucinates, you get a wrong answer. When an agent hallucinates, it might delete the wrong file, call the wrong API, or commit broken code. The loop amplifies mistakes — each bad decision feeds into the next step.

Runaway Costs

An agent stuck in a loop can burn through hundreds of dollars in API calls in minutes. Every iteration costs tokens, and a confused agent will iterate a lot. Always set token budgets and step limits.

Unintended Actions

An agent with access to your production database can do real damage. An agent with access to your email can send messages you didn’t authorize. The tools you give an agent define its blast radius — choose carefully.

The Confidence Problem

LLMs are confidently wrong just as often as they’re confidently right. An agent won’t say “I’m not sure” and ask for help — unless you explicitly build that behavior in. Chapter 6 covers human-in-the-loop patterns that address this.

The “give it all the tools” anti-pattern. New agent builders often give their agent every possible tool “just in case.” This is like giving an intern admin access to production on day one. Start with the minimum tools needed for the task. You can always add more — you can’t un-delete a database.

The Agent Analogy

Here’s the mental model that will serve you best throughout this book:

An agent is like a new developer on your team.

You give them a task (“fix this bug”), they have a specific set of skills (the LLM), access to tools (IDE, terminal, docs), and they work through the problem step by step. They make mistakes. They sometimes need to ask for help. They get faster as they learn your codebase. And you definitely don’t give them sudo rm -rf / on day one.

This analogy scales. A single agent is one developer. A multi-agent system (Chapter 7) is a team. An agent with human-in-the-loop (Chapter 6) is a developer with a tech lead reviewing their work. An agent with memory (Chapter 5) is a developer who takes notes.

The Core Code: What an Agent Looks Like

Before we go deeper in Chapter 2, let’s see the simplest possible agent in code. This isn’t production-ready — it’s the skeleton that every agent is built on:

import anthropic

client = anthropic.Anthropic()

tools = [
    {
        "name": "search_web",
        "description": "Search the web for information",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "Search query"}
            },
            "required": ["query"]
        }
    }
]

messages = [{"role": "user", "content": "What's the weather in Berlin?"}]

# THE AGENT LOOP — this is the whole idea
while True:
    response = client.messages.create(
        model="claude-sonnet-4-6-20250514",
        max_tokens=1024,
        tools=tools,
        messages=messages
    )

    # If the model wants to use a tool, execute it
    if response.stop_reason == "tool_use":
        # Extract tool call, execute it, feed result back
        tool_block = next(b for b in response.content if b.type == "tool_use")
        result = execute_tool(tool_block.name, tool_block.input)

        messages.append({"role": "assistant", "content": response.content})
        messages.append({
            "role": "user",
            "content": [{
                "type": "tool_result",
                "tool_use_id": tool_block.id,
                "content": result
            }]
        })
    else:
        # Model is done — print the final response
        print(response.content[0].text)
        break

That while True loop is the entire secret. The LLM decides whether to call a tool or respond. If it calls a tool, you execute it, feed the result back, and let the LLM decide again. That’s it. Every agent framework in existence is a dressed-up version of this loop.
