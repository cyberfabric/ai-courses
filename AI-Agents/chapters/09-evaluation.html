Chapter 9

Agent Evaluation & Testing

How do you know if your agent is good? The hardest question in agent development.

The Evaluation Problem

With traditional software, testing is straightforward: give input X, expect output Y. If you get Y, the test passes. If you get something else, it fails. Nice, clean, deterministic.

Agents break this model completely. Ask your research agent “What are the top AI frameworks in 2026?” and it might search different queries each time, read different pages, and produce different (but equally valid) answers. The path is non-deterministic, and there’s often no single “correct” answer.

So how do you test something that gives different answers every time? You change what you measure.

Evaluating agents is like evaluating employees

You don’t just check if they finished the task. You look at quality (was the result good?), efficiency (how long did it take?), process (did they follow the right steps?), and cost (what resources did they use?). Agent evaluation works the same way — multiple dimensions, not just pass/fail.

What to Measure

Every agent evaluation should capture these five dimensions:

Completion Did the agent finish the task without human intervention?
Accuracy Was the result correct and complete?
Efficiency How many steps and tokens did it use?
Metric What It Measures How to Calculate
Task completion rate % of tasks finished successfully successful_tasks / total_tasks
Accuracy score Quality of the result (0–1) LLM-as-judge or human rating
Step count How many tool calls to finish Count tool_use blocks per task
Cost per task Dollar cost per successful task total_tokens × price_per_token
Cost per success Accounting for failed attempts total_spend / successful_tasks
Latency Wall-clock time per task end_time - start_time
Cost per success is more revealing than cost per task. If your agent costs $0.05 per run but fails 50% of the time, your real cost is $0.10 per successful result. Track both.

End-to-End Evaluation

The most important eval: run the agent on a set of tasks and measure outcomes. This is your integration test — does the agent actually work?

import json
from datetime import datetime

class EvalSuite:
    """A test suite for evaluating agent performance."""

    def __init__(self, name: str):
        self.name = name
        self.test_cases = []
        self.results = []

    def add_case(self, query: str, expected_contains: list = None,
                 expected_tools: list = None, max_steps: int = 15):
        """Add a test case with expected outcomes."""
        self.test_cases.append({
            "query": query,
            "expected_contains": expected_contains or [],
            "expected_tools": expected_tools or [],
            "max_steps": max_steps,
        })

    def run(self, agent_fn):
        """Run all test cases against an agent function."""
        print(f"\nRunning eval suite: {self.name}")
        print(f"{'='*60}")

        for i, case in enumerate(self.test_cases, 1):
            print(f"\n  Test {i}/{len(self.test_cases)}: {case['query'][:60]}...")
            start = datetime.now()

            try:
                result = agent_fn(case["query"], max_steps=case["max_steps"])
                elapsed = (datetime.now() - start).total_seconds()

                # Score the result
                score = self._score(result, case)
                self.results.append({
                    "case": case["query"],
                    "passed": score["passed"],
                    "score": score["score"],
                    "elapsed": elapsed,
                    "details": score["details"],
                })
                status = "PASS" if score["passed"] else "FAIL"
                print(f"  [{status}] Score: {score['score']:.0%} "
                      f"({elapsed:.1f}s)")

            except Exception as e:
                self.results.append({
                    "case": case["query"],
                    "passed": False,
                    "score": 0,
                    "error": str(e),
                })
                print(f"  [ERROR] {e}")

        self._print_summary()

    def _score(self, result: str, case: dict) -> dict:
        """Score a result against expected outcomes."""
        checks = []
        result_lower = result.lower()

        for keyword in case["expected_contains"]:
            found = keyword.lower() in result_lower
            checks.append({"check": f"contains '{keyword}'", "passed": found})

        passed_count = sum(1 for c in checks if c["passed"])
        total = max(len(checks), 1)

        return {
            "passed": passed_count == total,
            "score": passed_count / total,
            "details": checks,
        }

    def _print_summary(self):
        passed = sum(1 for r in self.results if r["passed"])
        total = len(self.results)
        avg_score = sum(r["score"] for r in self.results) / max(total, 1)
        print(f"\n{'='*60}")
        print(f"Results: {passed}/{total} passed ({avg_score:.0%} avg score)")
        print(f"{'='*60}")

Step-Level Evaluation

End-to-end tells you if the agent works. Step-level tells you how it works. Did it pick the right tool? Pass the right parameters? Avoid unnecessary steps?

The Golden Trajectory Approach

Define an ideal sequence of tool calls for a task (the “golden trajectory”), then compare the agent’s actual path against it:

Define golden path → Run agent → Compare trajectories → Score alignment

You’re not checking for an exact match — the agent might take a different but equally valid path. Instead, check: Did the agent use all the key tools? Did it avoid unnecessary calls? Did it call tools in a sensible order?

def evaluate_trajectory(actual_tools: list, golden: dict) -> dict:
    """Compare agent's tool calls against a golden trajectory."""
    scores = {}

    # 1. Were all required tools used?
    required = set(golden.get("required_tools", []))
    used = set(actual_tools)
    scores["required_coverage"] = len(required & used) / max(len(required), 1)

    # 2. Were any unnecessary tools called?
    allowed = set(golden.get("allowed_tools", actual_tools))
    unnecessary = used - allowed
    scores["no_unnecessary"] = 1.0 if not unnecessary else \
        max(0, 1 - len(unnecessary) / len(used))

    # 3. Was the step count reasonable?
    max_steps = golden.get("max_expected_steps", 10)
    scores["efficiency"] = min(1.0, max_steps / max(len(actual_tools), 1))

    # Overall score
    scores["overall"] = sum(scores.values()) / len(scores)
    return scores

# Define a golden trajectory
golden = {
    "required_tools": ["search_web", "read_page"],
    "allowed_tools": ["search_web", "read_page", "save_report"],
    "max_expected_steps": 6,
}

# After running the agent, evaluate:
actual = ["search_web", "read_page", "search_web", "read_page", "save_report"]
scores = evaluate_trajectory(actual, golden)
print(f"Trajectory score: {scores['overall']:.0%}")

LLM-as-Judge

For evaluating output quality (not just presence of keywords), use another LLM as a judge. This is the dominant evaluation method in 2026, used by 53% of organizations:

def llm_judge(question: str, agent_answer: str,
              reference: str = None) -> dict:
    """Use an LLM to evaluate the quality of an agent's answer."""
    judge_prompt = f"""Evaluate this agent's answer on a scale of 1-5 for each criterion:

Question: {question}

Agent's Answer: {agent_answer}

{f"Reference Answer: {reference}" if reference else ""}

Criteria:
1. ACCURACY (1-5): Is the information factually correct?
2. COMPLETENESS (1-5): Does it cover the key aspects of the question?
3. CLARITY (1-5): Is it well-organized and easy to understand?
4. SOURCING (1-5): Does it cite sources or provide evidence?

Respond in JSON format:
{{"accuracy": N, "completeness": N, "clarity": N, "sourcing": N, "reasoning": "..."}}"""

    response = client.messages.create(
        model="claude-sonnet-4-6-20250514",
        max_tokens=500,
        messages=[{"role": "user", "content": judge_prompt}],
    )
    return json.loads(response.content[0].text)
LLM judges have biases. They tend to prefer longer answers, more formal language, and outputs that look like their own training data. Mitigate this by: (1) using specific, numerical rubrics, (2) including reference answers when possible, (3) rotating judge models, and (4) calibrating scores against human ratings on a small sample.

Building Your Own Evaluation Set

Generic benchmarks tell you how the model performs on generic tasks. But your agent has a specific domain, specific tools, and specific users. You need domain-specific evals.

Evaluation-Driven Development (EDD)

Concept: Write your evals before building the agent, just like writing tests before code in TDD.

Process:

  1. Define 10–20 representative tasks your agent should handle
  2. For each task, specify: expected outcome, required tools, maximum steps, quality criteria
  3. Set a target score (e.g., “pass 80% of cases”)
  4. Build the agent, run the eval, iterate until you hit the target
  5. Every change to the agent must not regress the eval score

Key difference from TDD: Agent evals use thresholds instead of binary pass/fail, because agents are non-deterministic. A test that passes 9 out of 10 runs might be fine. A test that passes 6 out of 10 needs work.

Structure your eval set with three types of cases:

  • Happy path (40%) — straightforward tasks the agent should handle easily
  • Edge cases (40%) — tricky inputs, ambiguous queries, missing information
  • Adversarial (20%) — inputs designed to confuse: contradictory requests, prompt injection attempts, impossible tasks

Regression Testing

Every improvement to your agent risks breaking something else. Run your full eval suite after every change:

def regression_check(new_results: list, baseline_results: list,
                      tolerance: float = 0.05) -> dict:
    """Compare new eval results against a baseline."""
    regressions = []

    for new, base in zip(new_results, baseline_results):
        if new["score"] < base["score"] - tolerance:
            regressions.append({
                "case": new["case"],
                "baseline_score": base["score"],
                "new_score": new["score"],
                "delta": new["score"] - base["score"],
            })

    new_avg = sum(r["score"] for r in new_results) / len(new_results)
    base_avg = sum(r["score"] for r in baseline_results) / len(baseline_results)

    return {
        "passed": len(regressions) == 0,
        "regressions": regressions,
        "avg_score_change": new_avg - base_avg,
        "summary": f"{'PASS' if not regressions else 'FAIL'}: "
                   f"{len(regressions)} regressions, "
                   f"avg score {new_avg:.0%} (was {base_avg:.0%})"
    }
Run evals multiple times. Because agents are non-deterministic, a single run isn’t reliable. Run each test case 3–5 times and use the median score. A test that passes 3/5 times at 80%+ is more informative than one perfect run that might have been lucky.

A/B Testing Agent Versions

When you have two versions of your agent (different model, different prompt, different tools), compare them head-to-head on the same tasks:

def ab_test(agent_a, agent_b, test_cases: list, runs_per_case: int = 3):
    """Compare two agent versions on the same tasks."""
    results = {"a": [], "b": []}

    for case in test_cases:
        for run in range(runs_per_case):
            score_a = run_and_score(agent_a, case)
            score_b = run_and_score(agent_b, case)
            results["a"].append(score_a)
            results["b"].append(score_b)

    avg_a = sum(r["score"] for r in results["a"]) / len(results["a"])
    avg_b = sum(r["score"] for r in results["b"]) / len(results["b"])
    cost_a = sum(r["cost"] for r in results["a"]) / len(results["a"])
    cost_b = sum(r["cost"] for r in results["b"]) / len(results["b"])

    print(f"Agent A: {avg_a:.0%} accuracy, ${cost_a:.4f}/task")
    print(f"Agent B: {avg_b:.0%} accuracy, ${cost_b:.4f}/task")
    print(f"Winner: {'A' if avg_a > avg_b else 'B'} "
          f"(by {abs(avg_a - avg_b):.0%})")

    return results

Cost vs Quality Tradeoffs

A common dilemma: a cheaper model takes more steps but costs less per step. An expensive model finishes in fewer steps but each step costs more. Which is better?

Claude Sonnet (fewer steps)
3 steps, $0.04/task, 92% accuracy Cost per success: $0.043
Claude Haiku (more steps)
7 steps, $0.008/task, 78% accuracy Cost per success: $0.010

The answer depends on your priorities. If accuracy matters most, Sonnet wins. If you’re processing thousands of tasks and can tolerate some failures, Haiku’s cost per success might be better. The eval suite answers this question for you — don’t debate, measure.
