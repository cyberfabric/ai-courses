Chapter 2

Agent Architectures

The blueprints that determine how your agent thinks, plans, and acts

Why Architecture Matters

In Chapter 1, you saw the simple while True loop that powers every agent. That loop is the engine — but the architecture is the driving style. The same LLM, given the same tools and the same task, will behave completely differently depending on how you structure the loop around it.

Think of it this way: if you asked three managers to get a project done, one might micromanage every step, another would delegate tasks and check in periodically, and the third would create a full project plan before anyone writes a line of code. Same goal, same team, wildly different outcomes depending on the management style.

Agent architectures are those management styles. In this chapter, you’ll learn the five major patterns, understand when to use each one, and implement the most important one — ReAct — from scratch.

Architecture 1: The Simple Tool-Use Loop

This is what you saw in Chapter 1. It’s the simplest possible agent architecture:

Call LLM → Tool call? → Execute tool → Feed result back → Repeat

The LLM decides on its own what to do. You don’t give it a strategy or ask it to plan. You just let it call tools until it produces a final answer. This is what most API tool-use examples demonstrate.

When to use it: Simple, well-defined tasks with 1–3 steps. Looking up information, formatting data, answering a factual question with one search.

When it fails: Complex tasks where the LLM needs to reason through multiple steps. Without explicit reasoning, the model tends to jump to action too fast and miss important context.

The “just let it figure it out” trap. The simple tool-use loop works great for demos but falls apart on real tasks. Without any reasoning structure, the agent tends to: (1) call the first tool that seems relevant without thinking, (2) get confused by ambiguous results, (3) loop endlessly when it doesn’t know what to do. For anything beyond trivial tasks, you need a pattern that encourages the model to think before it acts.

Architecture 2: ReAct (Reasoning + Acting)

ReAct is the workhorse of the agent world. Introduced in a 2022 research paper, it’s become the default architecture for production agents in 2025–2026. The core idea is beautifully simple: force the agent to think out loud before every action.

The ReAct Loop

ReAct interleaves reasoning and acting in an explicit cycle:

  1. Thought — the agent reasons about the current state, what it knows, and what it should do next
  2. Action — the agent calls a tool based on its reasoning
  3. Observation — the agent sees the result of the tool call
  4. Repeat — back to Thought, incorporating the observation

The key difference from the simple loop: the Thought step is explicit. The agent doesn’t just silently decide to call a tool — it writes out its reasoning first. This creates an audit trail, improves decision quality, and makes debugging possible. In regulated industries (finance, healthcare), this reasoning trace is often a compliance requirement.

Here’s what a ReAct trace looks like for a research question:

Thought The user wants to know the current market cap of NVIDIA. I should search for the latest financial data. Let me use a web search.
Action search_web("NVIDIA market cap April 2026")
Observation Results show NVIDIA market cap is approximately $3.2 trillion as of Q1 2026, but some sources show different dates. Let me verify with a more specific source.
Thought The search results are from different dates. I should check a financial data source directly to get the most current number.
Action search_web("NVIDIA NVDA market cap today site:finance.yahoo.com")
Observation Yahoo Finance shows NVIDIA (NVDA) market cap at $3.18T.
Thought I now have a reliable source with a current number. I can give the user a confident answer with the source.
Respond NVIDIA’s market cap is approximately $3.18 trillion (source: Yahoo Finance).

Notice how the Thought steps add crucial value: the agent notices the conflicting dates, decides to verify, and explains its reasoning. Without ReAct, it might have just returned the first number it found.

Implementing ReAct: The System Prompt Is the Architecture

Here’s the key insight: ReAct is not a framework feature — it’s a system prompt pattern. You implement it by telling the model how to think. With modern models that support extended thinking (like Claude’s thinking blocks), the reasoning happens naturally within the model’s response. But you can also enforce it explicitly:

SYSTEM_PROMPT = """You are a research assistant with access to tools.

For every task, follow this process:
1. THOUGHT: Reason about what you know, what you need, and what to do next.
2. ACTION: Call the appropriate tool.
3. OBSERVATION: Analyze the result.
4. Repeat until you have enough information to give a complete answer.

Always think before acting. Never call a tool without first explaining
why you're calling it and what you expect to learn."""

That’s it. The system prompt is the architecture. Modern models like Claude Opus and Sonnet follow these instructions reliably enough that this simple prompt turns a basic tool-use loop into a ReAct agent.

With Claude’s extended thinking feature, the “Thought” step happens inside a dedicated thinking block that you can inspect but that doesn’t count against the output token limit. This gives you the audit trail of ReAct without the token cost of writing thoughts into the main output.

Architecture 3: Plan-and-Execute

ReAct is great for tasks where you can figure things out as you go. But some tasks benefit from making a plan first:

Analyze Task → Create Plan → Execute Step 1 → Execute Step 2 → ... → Re-plan if needed

The key difference: the agent creates a full plan before executing anything. Then it works through the plan step by step, potentially re-planning when things don’t go as expected.

A Modern Twist: Planning as a Tool

Here’s a pattern that emerged from production agent systems in 2025: make the plan itself a tool call. Instead of a separate planning phase, the agent has a “create_plan” tool that it can call at any point:

tools = [
    {
        "name": "create_plan",
        "description": "Create or update a task plan. Call this before starting "
                       "complex multi-step work. List the steps needed, their "
                       "order, and expected outputs.",
        "input_schema": {
            "type": "object",
            "properties": {
                "goal": {"type": "string", "description": "The overall goal"},
                "steps": {
                    "type": "array",
                    "items": {
                        "type": "object",
                        "properties": {
                            "step": {"type": "string"},
                            "status": {"type": "string", "enum": ["pending", "done", "failed"]}
                        }
                    },
                    "description": "Ordered list of steps to achieve the goal"
                }
            },
            "required": ["goal", "steps"]
        }
    },
    # ... other tools like search_web, read_file, etc.
]

The clever part: the plan tool doesn’t actually do anything. It’s a no-op — it just stores the plan in the conversation context. But that’s the point. By writing out a plan, the agent commits to a structure, and the plan stays in the context window where the model can reference it at every subsequent step. This is a form of context engineering: shaping what’s in the model’s context to improve its behavior.

Claude Code uses exactly this approach internally — a “TodoList” tool that keeps the agent focused on the current task and prevents it from wandering off-track.

Planning as a Tool

Problem: The agent loses track of what it’s doing on complex, multi-step tasks.

Solution: Give the agent a “plan” tool that writes a structured task list into the conversation context. The tool itself does nothing — its value is purely as context engineering.

When to use: Any task that takes more than 5–7 tool calls to complete. Tasks where the agent needs to remember what it already tried. Multi-file edits, research with multiple sources, complex debugging sessions.

Trade-off: Uses extra tokens for the plan, but dramatically reduces wasted cycles from the agent repeating work or forgetting its progress.

Architecture 4: Reflexion (Self-Correction)

What if the agent could review its own work and fix mistakes? That’s Reflexion — an architecture where the agent adds a critique step after producing output:

Act → Produce output → Critique output → Fix issues → Final output

You’ve probably done this yourself: write code, run it, see the error, fix it, run again. Reflexion formalizes this natural debugging cycle. Modern LLMs show emergent self-correction — they can write code, see it fail, and adjust without explicit prompting. But you can also make it explicit:

def reflexion_loop(agent, task, max_attempts=3):
    for attempt in range(max_attempts):
        # Generate a solution
        result = agent.run(task)

        # Ask the agent to critique its own output
        critique = agent.run(
            f"Review this output critically. Is it correct? Complete? "
            f"Are there edge cases missed?\n\nOutput:\n{result}"
        )

        # If the critique finds no issues, we're done
        if "no issues" in critique.lower() or "looks correct" in critique.lower():
            return result

        # Otherwise, have the agent fix the issues
        task = f"Fix these issues with the previous output:\n{critique}\n\nOriginal output:\n{result}"

    return result  # Return best effort after max attempts

This is simple but powerful. Research shows that even basic reflection prompts (“did I make a mistake?”) noticeably improve agent output quality.

Reflexion doubles your token cost. Every critique step is another full LLM call. For simple tasks, the extra cost isn’t worth it. Use Reflexion for high-stakes outputs where correctness matters: code generation, data analysis, content that will be published. Skip it for quick lookups or one-off tasks.

Architecture 5: Tree of Thoughts

What if, instead of following one path, the agent explored multiple paths simultaneously?

Tree of Thoughts (ToT) generates several possible next steps, evaluates each one, and pursues the most promising path. Think of it like a chess engine that considers multiple moves before choosing the best one.

def tree_of_thoughts(agent, problem, branches=3, depth=3):
    """Explore multiple reasoning paths and pick the best one."""
    candidates = []

    for _ in range(branches):
        # Generate a different approach to the problem
        approach = agent.run(
            f"Propose ONE unique approach to solve this. "
            f"Be creative and different from obvious solutions.\n\n{problem}"
        )
        candidates.append(approach)

    # Evaluate all approaches
    evaluation = agent.run(
        f"Compare these {branches} approaches. Score each 1-10 "
        f"for feasibility, correctness, and efficiency. "
        f"Pick the best one and explain why.\n\n"
        + "\n\n".join(f"Approach {i+1}: {c}" for i, c in enumerate(candidates))
    )

    return evaluation

When to use it: Brainstorming, creative problem solving, puzzles with non-obvious solutions. Design tasks where you want to compare alternatives.

When to skip it: Most practical agent tasks. ToT is expensive (N branches × depth LLM calls) and overkill for well-defined problems. If you know the general approach and just need execution, ReAct or Plan-and-Execute is better.

Architecture 6: The Router Pattern

As tasks get complex, a single agent with 50 tools becomes unwieldy. The router pattern solves this with specialization: a lightweight “router” agent decides which specialist agent should handle the task.

The Router Architecture

A manager agent receives the user’s request and routes it to the appropriate specialist:

User Request → Router Agent → Specialist Agent → Response

The router is typically a fast, cheap model (like Claude Haiku) with a simple job: read the request, decide which specialist to call. Each specialist is a full agent with its own tools, system prompt, and expertise. This is how OpenAI’s Agents SDK implements handoffs: routing is represented as a tool call (transfer_to_refund_agent), making it native to the tool-use loop.

Two flavors of the router pattern exist in modern frameworks:

  • Handoff — the specialist takes over the conversation. The router steps aside. Used in OpenAI Agents SDK: handoffs=[refund_agent, billing_agent]
  • Agent-as-tool — the specialist handles a subtask and returns results to the router, which keeps control. Used in Claude Agent SDK via sub-agent spawning with context isolation.

We’ll go deep on multi-agent patterns in Chapter 7. For now, just know that the router is the gateway drug to multi-agent systems.

Comparing Architectures

Here’s the cheat sheet for picking the right architecture:

Architecture Best For Token Cost Reliability Complexity
Simple Loop 1–3 step tasks Low Medium Trivial
ReAct Research, debugging, open-ended tasks Medium High Low
Plan-and-Execute Complex, multi-step projects Medium-High High Medium
Reflexion High-stakes outputs, code gen High (2x) Very High Low
Tree of Thoughts Creative/design problems Very High (Nx) Variable Medium
Router Multi-domain, many tools Medium High Medium-High

The Trade-Off Triangle

Every agent architecture trades between three things:

Autonomy How much can the agent do on its own without human input?
Reliability How often does the agent produce the correct result?
Cost How many tokens and API calls does it consume?

Pick two. High autonomy + high reliability = expensive (more reasoning, self-correction, multiple paths). High autonomy + low cost = unreliable (the model just guesses). High reliability + low cost = low autonomy (more human oversight, less agent freedom). The production consensus in 2026: start simple, add complexity only when the simpler approach fails.

The System Prompt: Your Agent’s Operating System

Across all these architectures, one thing has more impact on agent behavior than any other design choice: the system prompt.

The system prompt is not just “instructions.” It’s the lens through which the model sees everything. It defines the agent’s personality, decision-making style, tool preferences, and failure modes. Two agents with identical code and tools but different system prompts will behave like completely different systems.

Effective system prompt
You are a code reviewer. For each file: 1. Check for security vulnerabilities first 2. Then check correctness 3. Then check style If you find a critical security issue, STOP and report it immediately. Never suggest changes you aren't confident about.
Clear priorities, specific behavior rules, defined stopping conditions.
Vague system prompt
You are a helpful assistant that reviews code. Be thorough and careful. Use the tools available to you.
No priorities, no strategy, no guardrails. The agent will wander aimlessly.
Your system prompt should answer three questions: (1) What is the agent’s goal? (2) How should it approach the task? (the architecture, in natural language) (3) What should it NOT do? (guardrails, boundaries, escalation rules). If your system prompt doesn’t answer all three, your agent will surprise you — and not in a good way.

Putting It All Together: A ReAct Agent from Scratch

Let’s build a complete ReAct agent in Python. This agent can search the web and answer research questions. We’ll use the Anthropic SDK, but the pattern works with any LLM API:

import anthropic
import json

client = anthropic.Anthropic()

# Define tools the agent can use
tools = [
    {
        "name": "search_web",
        "description": "Search the web for current information. Use this when "
                       "you need facts, data, or recent events.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {
                    "type": "string",
                    "description": "The search query. Be specific."
                }
            },
            "required": ["query"]
        }
    },
    {
        "name": "read_page",
        "description": "Read the contents of a web page. Use this when you "
                       "have a URL and need the full text.",
        "input_schema": {
            "type": "object",
            "properties": {
                "url": {
                    "type": "string",
                    "description": "The URL to read"
                }
            },
            "required": ["url"]
        }
    }
]

SYSTEM_PROMPT = """You are a research agent. Your job is to answer questions
accurately by searching the web and reading sources.

PROCESS:
1. Think about what information you need.
2. Search for it using search_web.
3. If needed, read specific pages with read_page for more detail.
4. Once you have enough information, provide a clear, sourced answer.

RULES:
- Always cite your sources.
- If sources conflict, note the disagreement.
- If you can't find reliable information, say so honestly.
- Maximum 5 search queries per question — be efficient."""

def execute_tool(name, inputs):
    """Execute a tool and return its result."""
    if name == "search_web":
        # In production, call a real search API
        return f"[Search results for: {inputs['query']}]"
    elif name == "read_page":
        return f"[Contents of: {inputs['url']}]"
    return "Unknown tool"

def run_agent(question, max_steps=10):
    """Run a ReAct agent to answer a question."""
    messages = [{"role": "user", "content": question}]

    for step in range(max_steps):
        response = client.messages.create(
            model="claude-sonnet-4-6-20250514",
            max_tokens=2048,
            system=SYSTEM_PROMPT,
            tools=tools,
            messages=messages
        )

        # Check if the agent wants to use tools
        if response.stop_reason == "tool_use":
            # Process all tool calls in the response
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    print(f"  [Step {step+1}] Tool: {block.name}({block.input})")
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result
                    })

            messages.append({"role": "assistant", "content": response.content})
            messages.append({"role": "user", "content": tool_results})
        else:
            # Agent is done — return the final answer
            final = next(b.text for b in response.content if b.type == "text")
            print(f"  [Done in {step+1} steps]")
            return final

    return "Agent hit step limit without finishing."

# Run it!
answer = run_agent("What are the top 3 AI agent frameworks in 2026?")
print(answer)

This is a real, runnable ReAct agent in about 80 lines. Swap in actual search/read implementations and you have a working research assistant. Chapter 4 will build a more complete version with error handling and user interaction.

Hybrid Architectures: The Production Reality

In practice, production agent systems in 2026 rarely use a single pure architecture. They use hybrids: a Plan-and-Execute outer loop with ReAct inside each step. Or a Router that dispatches to specialists, each using Reflexion on their output.

The winning pattern for production is: deterministic workflows with embedded AI decision points. Not pure autonomous agents, but structured pipelines where the LLM handles the judgment calls and regular code handles the predictable parts.

The Production Rule of Thumb

Start with the simplest architecture that could work. Add complexity only when you have evidence the simple version fails. Most agents in production use ReAct or simple tool-use loops. Plan-and-Execute for complex tasks. Multi-agent for specialized domains. Tree of Thoughts almost never.
