Chapter 10

Agent Frameworks & SDKs

Choosing the right tool for your agent project — because you don’t have to build everything from scratch

Why Frameworks Exist

You’ve been building agents from scratch since Chapter 4. You wrote the tool-use loop, the memory system, the approval gates, the error handling. You understand exactly how it all works. Now the question is: do you want to keep writing all of that for every project?

Agent frameworks are like web frameworks — you can build everything from scratch with raw HTTP, but Django, Express, or Rails give you patterns, conventions, and an ecosystem. The question is always: do the abstractions match your problem?

In this chapter, we’ll survey the six major frameworks in the 2026 landscape, build the same agent in two of them, and give you a decision matrix for choosing the right one.

Framework 1: Claude Agent SDK

Anthropic’s framework, built on the same infrastructure that powers Claude Code. If you’re building on the Anthropic ecosystem, this is the most natural fit.

CA
Claude Agent SDK
Philosophy: MCP-native, safety-first. Tools are implemented as in-process MCP servers. Built-in permission modes (allow/deny/auto-approve), lifecycle hooks, and sub-agent spawning with context isolation.
# Claude Agent SDK — research agent
from claude_agent_sdk import Agent, tool

@tool
def search_web(query: str) -> str:
    """Search the web for current information."""
    # Implementation here
    return search_api.search(query)

@tool
def read_page(url: str) -> str:
    """Read the text content of a web page."""
    return fetch_and_extract(url)

agent = Agent(
    model="claude-sonnet-4-6-20250514",
    system_prompt="You are a research agent. Search the web to answer "
                  "questions accurately. Cite your sources.",
    tools=[search_web, read_page],
)

result = agent.query("What are the top AI trends in 2026?")
print(result)

Best for: Anthropic-stack projects, Claude Code extensions, safety-critical agents, MCP-heavy tool ecosystems.

Framework 2: OpenAI Agents SDK

Despite the name, the OpenAI Agents SDK is provider-agnostic — it works with 100+ LLMs. It’s the simplest framework to get started with.

OA
OpenAI Agents SDK
Philosophy: Minimal primitives, maximum flexibility. Three core concepts: Agents, Handoffs, and Guardrails. Built-in tracing enabled by default. Handoffs make multi-agent routing trivial.
# OpenAI Agents SDK — research agent
from agents import Agent, Runner, function_tool

@function_tool
def search_web(query: str) -> str:
    """Search the web for current information."""
    return search_api.search(query)

@function_tool
def read_page(url: str) -> str:
    """Read the text content of a web page."""
    return fetch_and_extract(url)

agent = Agent(
    name="researcher",
    instructions="You are a research agent. Search the web to answer "
                 "questions accurately. Cite your sources.",
    tools=[search_web, read_page],
)

result = Runner.run_sync(agent, "What are the top AI trends in 2026?")
print(result.final_output)

Best for: Quick prototypes, multi-provider setups, projects that need handoffs between specialized agents.

Framework 3: LangGraph

LangGraph models agent workflows as directed graphs with explicit state. Each node is a function, edges define the flow, and state is managed as a typed schema. The most production-mature framework in 2026.

LG
LangGraph
Philosophy: Agents are state machines. Define your state, define your nodes, define your edges. Built-in checkpointing (Postgres, SQLite, Redis), human-in-the-loop interrupts, and time-travel debugging via LangSmith.
# LangGraph — research agent with state management
from langgraph.graph import StateGraph, MessagesState
from langgraph.prebuilt import ToolNode, tools_condition
from langchain_anthropic import ChatAnthropic

model = ChatAnthropic(model="claude-sonnet-4-6-20250514")
model_with_tools = model.bind_tools([search_web, read_page])

def call_model(state: MessagesState):
    response = model_with_tools.invoke(state["messages"])
    return {"messages": [response]}

# Build the graph
graph = StateGraph(MessagesState)
graph.add_node("agent", call_model)
graph.add_node("tools", ToolNode(tools=[search_web, read_page]))
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", tools_condition)
graph.add_edge("tools", "agent")

app = graph.compile()
result = app.invoke({"messages": [
    ("user", "What are the top AI trends in 2026?")
]})

Best for: Complex stateful workflows, production deployments needing checkpointing, teams that want visual debugging via LangSmith.

Framework 4: CrewAI

CrewAI takes a role-based approach: you define agents with roles, goals, and backstories, then organize them into crews that collaborate.

CR
CrewAI
Philosophy: Agents are team members with roles. Define who they are, what they’re good at, and let them collaborate. Two modes: Crews (autonomous multi-agent teams) and Flows (event-driven deterministic workflows).
# CrewAI — research team
from crewai import Agent, Task, Crew

researcher = Agent(
    role="Research Analyst",
    goal="Find accurate, current information on given topics",
    backstory="You're a senior research analyst who always cites sources.",
    tools=[search_web, read_page],
)

writer = Agent(
    role="Technical Writer",
    goal="Synthesize research into clear, well-structured articles",
    backstory="You turn complex research into readable content.",
)

research_task = Task(
    description="Research the top AI trends in 2026",
    agent=researcher,
    expected_output="Detailed findings with sources",
)

writing_task = Task(
    description="Write a summary article based on the research",
    agent=writer,
    expected_output="A well-structured article",
)

crew = Crew(agents=[researcher, writer], tasks=[research_task, writing_task])
result = crew.kickoff()

Best for: Multi-agent collaboration, role-based task delegation, teams that think in terms of “who does what.”

Framework 5: AG2 / Microsoft Agent Framework

AG
AG2 (AutoGen successor)
Philosophy: Conversational multi-agent. Agents chat with each other to solve problems. AG2 is the independent open-source fork; Microsoft merged AutoGen with Semantic Kernel into the Microsoft Agent Framework (GA Q1 2026) for enterprise Azure deployments.

Best for: Microsoft/Azure stack, conversational agent patterns, enterprise deployments requiring C#/Java support alongside Python.

Framework 6: Google ADK

GK
Google Agent Development Kit (ADK)
Philosophy: Code-first Python toolkit, v1.0 stable. Optimized for Gemini but model-agnostic. Built-in workflow agents (Sequential, Parallel, Loop), eval tools, web UI (adk web), and native A2A protocol support for agent-to-agent communication.

Best for: Google Cloud/Vertex AI, multimodal agents (vision + audio), teams using Gemini models, A2A interoperability.

Framework vs No Framework

The Decision Framework

Ask these questions to decide if you need a framework:

  • Single agent, <5 tools, one-off project? → No framework. Your Chapter 4 code is fine.
  • Need multi-agent orchestration? → Framework. Don’t reinvent the coordination wheel.
  • Need persistence/checkpointing? → Framework (LangGraph excels here).
  • Need built-in observability? → Framework. Tracing is hard to retrofit.
  • Team project with multiple developers? → Framework. Shared patterns beat everyone rolling their own.
  • Learning how agents work? → No framework. Build from scratch first (which you’ve done!).
The abstraction trap. Some frameworks hide so much that debugging becomes impossible. When your agent fails, you need to see the raw tool calls, the full conversation history, and the model’s reasoning. If a framework doesn’t let you inspect these, it will cost you more time than it saves. Always verify you can access the underlying tool-use loop before committing to a framework.

Same Agent, Two Frameworks

Let’s compare how the same task — a research agent that answers a question — looks in two frameworks side by side:

OpenAI Agents SDK (~15 lines)
agent = Agent( name="researcher", instructions="Research and answer questions.", tools=[search_web, read_page], ) result = Runner.run_sync( agent, "Top AI trends 2026?" ) print(result.final_output)
Minimal code. Framework handles the loop, tool execution, and response extraction.
LangGraph (~20 lines)
graph = StateGraph(MessagesState) graph.add_node("agent", call_model) graph.add_node("tools", ToolNode(tools)) graph.set_entry_point("agent") graph.add_conditional_edges( "agent", tools_condition ) graph.add_edge("tools", "agent") app = graph.compile() result = app.invoke(messages)
More explicit. You define the graph structure, which gives you fine control over flow, branching, and state.

Both produce the same result. OpenAI’s SDK is simpler for basic agents. LangGraph gives you more control over the flow, which matters when you need conditional branches, parallel execution, or checkpointing.

Migration: Getting Out of a Framework

The biggest risk with frameworks is lock-in. Here’s how to minimize it:

  • Keep tools framework-agnostic — implement tools as plain Python functions, then wrap them for your framework
  • Keep system prompts in config files — not hardcoded in framework-specific classes
  • Use MCP for tool distribution — MCP tools work with any framework
  • Avoid framework-specific state — use standard formats (JSON, SQLite) for persistence
  • Keep the core logic portable — if you stripped the framework away, could you reconstruct the agent with your Chapter 4 loop?
The frameworks that survive tend to become thinner over time, not thicker. MCP standardized tools. A2A is standardizing agent communication. The unique value of each framework is shrinking toward orchestration patterns and developer experience. Choose based on how well the framework fits your workflow today, not on how many features it promises.

Agent Trace: Framework Abstractions

Here’s what happens under the hood when you call Runner.run_sync() or graph.compile().invoke() — the same loop you built in Chapter 4:

Framework Serialize tools to JSON Schema, compose system prompt, initialize message history.
LLM Call client.messages.create(model, system, tools, messages) — the same API call you wrote
Check response.stop_reason == "tool_use"? — same check you wrote
Execute Framework calls your tool function, serializes result, appends to messages — same pattern
Loop Back to LLM Call. Repeat until stop_reason == "end_turn" — the while True you wrote
Return Extract final text, return to caller. Frameworks add: tracing, token counting, checkpoint saves.

The framework adds convenience (auto-schema generation, tracing, checkpoints) but the core is identical to your handwritten code. That’s why Chapter 4 came before this one — you understand the foundation.
