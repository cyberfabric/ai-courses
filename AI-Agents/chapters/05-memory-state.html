Chapter 5

Memory & State Management

How agents remember — because the conversation history is not enough

The Memory Problem

Your Chapter 4 agent has a dirty secret: it’s an amnesiac. Run it twice with the same question and it starts from scratch both times. Ask it to “continue where you left off” and it has no idea what you’re talking about. Ask it to reference research from yesterday and it draws a complete blank.

This is the memory problem. LLMs are stateless by design — every API call starts with a blank slate. The only reason your agent “remembers” anything within a single session is because you keep appending to the messages list and sending the full history back with each call. That works until it doesn’t.

In this chapter, you’ll learn the three types of agent memory, how to manage the context window as it grows, and how to give your agent persistent memory that survives across sessions.

Agent memory is like a developer’s workflow

You have your current task in your head (working memory), your Slack thread with the team (short-term memory), and your wiki/docs where you write things down for next week (long-term memory). An agent needs all three levels too.

The Three Types of Agent Memory

Memory Architecture for Agents

Every agent system has three distinct memory layers, each serving a different purpose:

Working Memory → Short-Term Memory → Long-Term Memory
  • Working memory — variables and scratchpads the agent uses during a single task. Lives in code, not in the conversation.
  • Short-term memory — the conversation history (messages list). Grows with every step. Limited by the context window.
  • Long-term memory — persistent storage (files, databases, vector stores) that survives across sessions.

The key insight: context quality matters more than context quantity. A 1M-token context window doesn’t help if 90% of it is irrelevant tool output from 50 steps ago. Memory management is about curating what’s in the context — keeping useful information close and pushing stale data out.

Working Memory: The Agent’s Scratchpad

Working memory is the simplest form: variables in your code that the agent uses during a single task. The agent doesn’t directly see or control these — your code manages them.

class AgentState:
    """Working memory for a single agent run."""
    def __init__(self, task: str):
        self.task = task
        self.steps_taken = 0
        self.tools_called = []
        self.total_tokens = 0
        self.findings = []        # Intermediate results
        self.errors = []          # Errors encountered
        self.plan = []            # Current plan steps

    def log_step(self, tool_name: str, tokens: int):
        self.steps_taken += 1
        self.tools_called.append(tool_name)
        self.total_tokens += tokens

    def is_over_budget(self, max_tokens: int = 100_000) -> bool:
        return self.total_tokens > max_tokens

This state object tracks what the agent has done, how much it’s spent, and whether it’s time to stop. The agent loop checks state.is_over_budget() at each step and can use state.findings to inform its next move.

Working memory is cheap and reliable because it’s just Python variables. Use it for anything the agent needs to track within a single run: step counters, token budgets, error counts, lists of files already processed.

Short-Term Memory: The Conversation History

Short-term memory is the messages list you built in Chapter 4. Every user message, assistant response, tool call, and tool result gets appended. The LLM sees this full history at every step, which is how it “remembers” what happened earlier.

The problem: it grows without bound.

~5KB Average web page content returned by read_page
10 pages Agent reads during a typical research task
50KB+ Context consumed — and growing with every step

Even with Claude’s 200K-token context window (or 1M with extended context), you’ll hit problems. Models suffer from the “lost in the middle” effect — information in the middle of the context gets less attention than information at the start or end. After ~32K tokens of context, accuracy starts to drop.

Strategy 1: Truncate Tool Outputs

The simplest fix: don’t let tool outputs bloat the context. We already did this in Chapter 4 with text[:5000], but you can be smarter:

def truncate_result(result: str, max_chars: int = 3000) -> str:
    """Truncate tool output, keeping the start and end."""
    if len(result) <= max_chars:
        return result
    half = max_chars // 2
    return (result[:half]
            + f"\n\n[...{len(result) - max_chars} chars truncated...]\n\n"
            + result[-half:])

Strategy 2: Observation Masking

A technique from the LangChain “Deep Agents” framework: replace old tool outputs with short placeholders while keeping the tool call metadata visible. The agent can see what it did (called search_web with query X) without the full 5KB result clogging the context.

def mask_old_observations(messages: list, keep_recent: int = 4) -> list:
    """Replace old tool results with summaries to save context."""
    masked = []
    tool_result_count = sum(
        1 for m in messages
        if isinstance(m.get("content"), list)
        and any(isinstance(b, dict) and b.get("type") == "tool_result"
                for b in m["content"])
    )

    results_seen = 0
    for msg in messages:
        if (isinstance(msg.get("content"), list)
            and any(isinstance(b, dict) and b.get("type") == "tool_result"
                    for b in msg["content"])):
            results_seen += 1
            if results_seen <= tool_result_count - keep_recent:
                # Replace old results with placeholders
                masked_content = []
                for block in msg["content"]:
                    if isinstance(block, dict) and block.get("type") == "tool_result":
                        masked_content.append({
                            **block,
                            "content": "[previous result — see summary above]"
                        })
                    else:
                        masked_content.append(block)
                masked.append({**msg, "content": masked_content})
                continue
        masked.append(msg)
    return masked

Strategy 3: Summarize and Compress

When the context hits 70–80% capacity, ask the LLM to summarize the conversation so far, then replace the old messages with the summary:

def compress_history(client, messages: list, threshold: int = 50_000) -> list:
    """Compress conversation history when it gets too long."""
    # Estimate token count (rough: 1 token ≈ 4 chars)
    total_chars = sum(len(str(m)) for m in messages)
    if total_chars < threshold:
        return messages  # Not yet needed

    # Ask the LLM to summarize
    summary_response = client.messages.create(
        model="claude-haiku-4-5-20251001",  # Use a cheap model
        max_tokens=1000,
        messages=[{
            "role": "user",
            "content": (
                "Summarize the key findings and decisions from this "
                "conversation. Include: what was searched, what was found, "
                "what conclusions were reached, and what remains to be done.\n\n"
                + str(messages)
            )
        }]
    )
    summary = summary_response.content[0].text

    # Replace old messages with the summary
    return [
        {"role": "user", "content": f"[Previous conversation summary]\n{summary}"},
        *messages[-4:]  # Keep the last 4 messages for continuity
    ]
Don’t summarize too aggressively. If you compress too early or too much, the agent loses critical context and starts repeating work or making contradictory decisions. The sweet spot: keep the last 4–6 tool interactions verbatim, summarize everything older. And always keep the original user request in full.

Long-Term Memory: Surviving Across Sessions

Short-term memory dies when the process ends. For agents that need to remember things across sessions — past research, user preferences, learned patterns — you need long-term memory.

Option 1: File-Based Memory (Simplest)

Claude Code uses this approach: a MEMORY.md file that the agent reads at the start of each session and updates during work. It’s simple, versioned (it’s just a file in your repo), and surprisingly effective.

The Memory Tool Pattern

Problem: The agent needs to remember information across sessions but you don’t want to build a complex storage system.

Solution: Give the agent two tools — save_memory and recall_memory — that read/write to local files. The agent decides what’s worth remembering.

How it works:

  1. At session start, the agent reads memory files into its context
  2. During work, it can call save_memory to persist a fact or finding
  3. When it needs past information, it calls recall_memory with a query

When to use: Single-user agents, prototypes, agents that need to remember user preferences or project-specific knowledge. This is how Claude Code manages per-project context.

import os, json
from pathlib import Path

MEMORY_DIR = Path("agent_memory")
MEMORY_DIR.mkdir(exist_ok=True)

memory_tools = [
    {
        "name": "save_memory",
        "description": (
            "Save a piece of information to persistent memory. Use this "
            "to remember important findings, user preferences, or facts "
            "you'll need in future sessions. Each memory has a topic "
            "(key) and content (value)."
        ),
        "input_schema": {
            "type": "object",
            "properties": {
                "topic": {
                    "type": "string",
                    "description": "Short topic key, e.g. 'user_preference_language'"
                },
                "content": {
                    "type": "string",
                    "description": "The information to remember"
                }
            },
            "required": ["topic", "content"]
        }
    },
    {
        "name": "recall_memory",
        "description": (
            "Recall information from persistent memory. Returns all "
            "saved memories if no topic specified, or a specific memory "
            "by topic. Use this at the start of a session or when you "
            "need context from past work."
        ),
        "input_schema": {
            "type": "object",
            "properties": {
                "topic": {
                    "type": "string",
                    "description": "Topic to recall. Omit to list all memories."
                }
            }
        }
    }
]

def save_memory(topic: str, content: str) -> str:
    filepath = MEMORY_DIR / f"{topic}.json"
    data = {"topic": topic, "content": content, "saved_at": str(datetime.now())}
    filepath.write_text(json.dumps(data, indent=2))
    return json.dumps({"status": "success", "message": f"Saved memory: {topic}"})

def recall_memory(topic: str = None) -> str:
    if topic:
        filepath = MEMORY_DIR / f"{topic}.json"
        if filepath.exists():
            return filepath.read_text()
        return json.dumps({"status": "not_found", "message": f"No memory for '{topic}'"})
    # List all memories
    memories = []
    for f in MEMORY_DIR.glob("*.json"):
        data = json.loads(f.read_text())
        memories.append({"topic": data["topic"], "saved_at": data["saved_at"]})
    return json.dumps({"status": "success", "memories": memories})

Add these tools to your Chapter 4 agent and it can now remember things across runs. The agent decides what’s worth saving — you don’t need to wire up any triggers.

Option 2: Vector Store Memory (For Search)

When the agent has hundreds of memories, file lookup by exact topic breaks down. You need semantic search — finding memories by meaning, not by exact key. That’s where vector stores come in.

# pip install chromadb
import chromadb

# Create a persistent vector store
chroma = chromadb.PersistentClient(path="./agent_memory_db")
collection = chroma.get_or_create_collection("agent_memories")

def save_to_vector_store(text: str, metadata: dict = None):
    """Save a memory with semantic search capability."""
    doc_id = f"mem_{collection.count()}"
    collection.add(
        documents=[text],
        metadatas=[metadata or {}],
        ids=[doc_id]
    )

def recall_from_vector_store(query: str, n_results: int = 3) -> list:
    """Find memories semantically similar to the query."""
    results = collection.query(query_texts=[query], n_results=n_results)
    return results["documents"][0] if results["documents"] else []

ChromaDB embeds your text automatically and stores it locally. When the agent calls recall_from_vector_store("What did we learn about authentication?"), it returns the most semantically similar memories, even if they don’t contain the exact word “authentication.”

Start with files, graduate to vectors. File-based memory is perfect for agents with fewer than ~100 memories. Once you need semantic search across hundreds of entries, switch to a vector store. Don’t start with a vector store — it’s unnecessary complexity for most agents.

Structured State: Tracking Progress

For complex tasks, the agent needs to track not just what it remembers but where it is in the task. This is structured state — an explicit record of completed steps, pending work, and intermediate results.

Remember the “planning as a tool” pattern from Chapter 2? Combine it with state tracking:

task_state = {
    "goal": "Research and compare 3 AI agent frameworks",
    "steps": [
        {"step": "Search for framework comparisons", "status": "done",
         "result": "Found 5 relevant articles"},
        {"step": "Read and analyze top 3 articles", "status": "in_progress",
         "result": "Read 2 of 3"},
        {"step": "Synthesize findings into report", "status": "pending"},
        {"step": "Save report to file", "status": "pending"}
    ],
    "findings": {
        "LangGraph": {"stars": "44K", "strengths": "state management, persistence"},
        "CrewAI": {"stars": "44.6K", "strengths": "role-based multi-agent"}
    }
}

Checkpointing: Resume Where You Left Off

What if the agent crashes mid-task? Or hits a rate limit and needs to pause? Checkpointing saves the full agent state so you can resume later.

import json
from datetime import datetime

def save_checkpoint(messages, state, checkpoint_file="checkpoint.json"):
    """Save agent state so we can resume later."""
    # Convert message objects to serializable format
    serializable_messages = []
    for msg in messages:
        if isinstance(msg.get("content"), list):
            # Handle content blocks (tool results, etc.)
            serializable_messages.append({
                "role": msg["role"],
                "content": [
                    block if isinstance(block, dict)
                    else {"type": "text", "text": str(block)}
                    for block in msg["content"]
                ]
            })
        else:
            serializable_messages.append(msg)

    checkpoint = {
        "timestamp": str(datetime.now()),
        "messages": serializable_messages,
        "state": state.__dict__ if hasattr(state, '__dict__') else state,
    }
    with open(checkpoint_file, "w") as f:
        json.dump(checkpoint, f, indent=2, default=str)
    print(f"  [Checkpoint saved: {checkpoint_file}]")

def load_checkpoint(checkpoint_file="checkpoint.json"):
    """Load a saved checkpoint to resume work."""
    with open(checkpoint_file) as f:
        return json.load(f)
Checkpoints include conversation history. That means they can contain sensitive information from tool outputs — API responses, file contents, search results. Store checkpoint files securely and clean them up when no longer needed.

Agent Trace: Memory in Action

Here’s what a memory-aware agent looks like during a second session:

Think The user wants me to continue researching AI frameworks. Let me check if I have any previous findings.
Tool recall_memory({}) — list all saved memories
Result Found 2 memories: “framework_comparison” (saved yesterday), “user_preference_detail_level” (saved 3 days ago)
Tool recall_memory({"topic": "framework_comparison"})
Result Previous findings: compared LangGraph and CrewAI. Haven’t covered Claude Agent SDK or OpenAI Agents SDK yet.
Think Great, I already have data on LangGraph and CrewAI. I need to research Claude Agent SDK and OpenAI Agents SDK to complete the comparison. The user prefers detailed analysis.
Tool search_web({"query": "Claude Agent SDK features 2026"})
Respond Continuing from where we left off — I already have notes on LangGraph and CrewAI. Now researching the remaining frameworks…

Notice how the agent doesn’t repeat work. It checks memory first, finds previous findings, and continues from where it stopped. This is the difference between a tool and an assistant.

When NOT to Use Memory

Memory adds complexity, cost, and potential bugs. Keep your agent stateless when possible:

  • One-shot tasks — if the agent runs once and is done, it doesn’t need memory
  • Idempotent operations — if the agent can re-derive everything from the input, memory is redundant
  • Security-sensitive contexts — memory persists data, which means it can leak if not secured
  • High-throughput systems — memory lookups add latency to every step

The rule: start stateless, add memory only when the agent clearly needs it. If users complain that the agent “forgot” something important, that’s the signal to add memory.
