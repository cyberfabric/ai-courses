Chapter 6

Human-in-the-Loop Patterns

Keeping humans in control — the most critical pattern for production agents

Why Full Autonomy Is (Usually) a Bad Idea

In Chapter 5, your agent learned to remember things. In Chapter 4, it learned to act. Put those together and you have a system that can take action based on what it “remembers” you wanted — without checking first. That’s either incredibly useful or terrifying, depending on whether it does the right thing.

Here’s the uncomfortable math: if your agent makes the right decision 95% of the time, that sounds great. But if it runs 100 steps per day, that’s 5 mistakes per day. If any of those mistakes involve deleting files, sending emails, or modifying production data, you have a serious problem.

Human-in-the-loop is like the permissions model on your phone

Apps can access the internet freely (low risk), need approval for your camera or location (medium risk), and are blocked from accessing other apps’ data (high risk). Your agent needs the same tiered permission model — some actions are free, some need approval, and some are always blocked.

The Autonomy Spectrum

Human oversight isn’t binary. There’s a whole spectrum between “approve every step” and “let it run wild”:

Level Human Role Agent Freedom Use Case
Fully Manual Approves every action None Learning, debugging
Approval Gates Approves risky actions only Low-risk actions free Early production
Supervised Autonomy Monitors, intervenes on errors Most actions free Trusted production
Full Autonomy Reviews outputs after the fact All actions free Low-stakes, well-tested

Most production agents in 2026 live at Approval Gates or Supervised Autonomy. Full autonomy is rare and reserved for low-stakes tasks with extensive test coverage. The industry term is “bounded autonomy” — the agent runs freely within defined limits, with mandatory escalation for high-stakes decisions.

Pattern 1: Approval Gates

The simplest and most effective pattern: require human approval before specific actions. You define which tools are “sensitive,” and the agent pauses to ask before calling them.

The Approval Gate Pattern

Problem: The agent can take actions with real-world consequences (file writes, API calls, database changes) and you want to prevent mistakes.

Solution: Categorize tools into tiers (free, needs-approval, blocked). Before executing a tool in the “needs-approval” tier, pause and ask the human to approve or deny.

When to use: Any agent that performs write operations, sends communications, or modifies external systems. This is the minimum viable safety pattern for production agents.

class ApprovalGate:
    """Configurable approval system for agent tool calls."""

    def __init__(self):
        # Tools that run freely
        self.auto_approve = {"search_web", "read_page", "read_file", "recall_memory"}

        # Tools that require human approval
        self.needs_approval = {"write_file", "save_report", "send_email",
                               "delete_file", "create_pr", "save_memory"}

        # Tools that are always blocked
        self.blocked = {"drop_table", "rm_rf", "push_to_production"}

    def check(self, tool_name: str, tool_input: dict) -> str:
        """Returns 'allow', 'deny', or 'ask'."""
        if tool_name in self.blocked:
            return "deny"
        if tool_name in self.auto_approve:
            return "allow"
        if tool_name in self.needs_approval:
            return "ask"
        # Unknown tools default to asking
        return "ask"

    def request_approval(self, tool_name: str, tool_input: dict) -> bool:
        """Ask the human for approval."""
        print(f"\n  {'='*50}")
        print(f"  APPROVAL REQUIRED")
        print(f"  Tool:   {tool_name}")
        print(f"  Input:  {json.dumps(tool_input, indent=2)[:300]}")
        print(f"  {'='*50}")
        response = input("  Allow this action? (y/n/details) > ").strip().lower()
        if response == "details":
            print(f"  Full input: {json.dumps(tool_input, indent=2)}")
            response = input("  Allow? (y/n) > ").strip().lower()
        return response == "y"


gate = ApprovalGate()

# In your agent loop, before executing a tool:
def execute_with_approval(tool_name, tool_input):
    decision = gate.check(tool_name, tool_input)

    if decision == "deny":
        return '{"status": "blocked", "message": "This action is not permitted."}'

    if decision == "ask":
        if not gate.request_approval(tool_name, tool_input):
            return '{"status": "denied", "message": "User declined this action."}'

    # Execute the tool
    return tool_functions[tool_name](tool_input)

When the agent tries to write a file, the human sees exactly what’s about to happen and can approve or deny. The agent gets structured feedback either way, so it can adjust its approach.

Case Study: Claude Code’s Permission Model

Claude Code is the best real-world example of approval gates done right. It uses a three-tier system that every agent builder should study:

Claude Code’s Permission Architecture

Claude Code evaluates every tool call through a permission stack, applied in strict priority order:

  1. Deny rules (highest priority) — always block, regardless of other settings. Example: Bash(rm -rf /)
  2. Ask rules — prompt the user for approval. Example: file writes, git pushes
  3. Allow rules (lowest priority) — auto-approve without prompting. Example: file reads, searches

Rules can target specific tools, specific arguments (e.g., allow Bash(npm test) but ask for Bash(npm publish)), or categories. Users configure rules at project and global levels. The system also supports hooks — shell commands that run before or after tool calls, enabling custom validation logic beyond simple allow/deny.

The key insight: permissions are per-tool and per-argument, not just per-tool. Reading /etc/passwd and reading src/app.py are the same tool call but very different risk levels. Good permission systems let you express that difference.

Pattern 2: Escalation

Approval gates handle known risky actions. But what about situations where the agent itself is uncertain? That’s where escalation comes in: the agent decides to ask for help.

ESCALATION_PROMPT = """
You have access to an escalation tool. Use it when:
1. You are less than 80% confident in your next action
2. You encounter conflicting information from multiple sources
3. The task seems to require access or permissions you don't have
4. You've failed the same approach twice
5. The user's request is ambiguous and you could interpret it multiple ways

When in doubt, escalate. A brief interruption is better than a wrong action."""

escalate_tool = {
    "name": "ask_human",
    "description": (
        "Ask the human operator for guidance. Use this when you're "
        "uncertain, stuck, or the task requires a judgment call. "
        "Provide clear context about what you need help with."
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "question": {
                "type": "string",
                "description": "What you need help with — be specific"
            },
            "context": {
                "type": "string",
                "description": "What you've tried so far and why you're stuck"
            },
            "options": {
                "type": "array",
                "items": {"type": "string"},
                "description": "If applicable, the options you're choosing between"
            }
        },
        "required": ["question"]
    }
}

The agent can now choose to ask for help, and it provides structured context about why. This is far better than the agent guessing and getting it wrong.

Think The user asked me to “clean up the old branches.” I found 12 branches. Some look like active work (“feature/new-auth” has commits from yesterday). I’m not confident I should delete these without checking.
Tool ask_human({"question": "Which branches should I delete?", "context": "Found 12 branches. 3 appear active (commits within the last week): feature/new-auth, fix/login-bug, feature/dashboard-v2. The other 9 have no commits in 30+ days.", "options": ["Delete only the 9 inactive branches", "Delete all 12", "Let me review the list first"]})
Result User: “Delete the 9 inactive ones. Leave the active branches alone.”
Think Clear instructions. I’ll delete only the 9 branches with no recent activity.
Tool delete_branches({"branches": ["old-feature-1", "old-feature-2", ...], "confirm": true})
Respond Deleted 9 inactive branches. Kept feature/new-auth, fix/login-bug, and feature/dashboard-v2 active.

Without escalation, the agent might have deleted all 12 branches — including someone’s active work. The two-second interruption saved hours of frustration.

Pattern 3: The Budget Pattern

Instead of approving individual actions, give the agent a resource budget. It runs freely until the budget is exhausted, then must check in:

class BudgetController:
    """Give the agent a budget and require check-in when it's spent."""

    def __init__(self, max_tokens=50_000, max_tool_calls=20, max_minutes=5):
        self.max_tokens = max_tokens
        self.max_tool_calls = max_tool_calls
        self.max_minutes = max_minutes
        self.tokens_used = 0
        self.tool_calls_made = 0
        self.start_time = time.time()

    def record_usage(self, tokens: int):
        self.tokens_used += tokens
        self.tool_calls_made += 1

    def check_budget(self) -> dict:
        """Check if any budget is exhausted."""
        elapsed = (time.time() - self.start_time) / 60

        if self.tokens_used >= self.max_tokens:
            return {"exhausted": "tokens", "used": self.tokens_used,
                    "limit": self.max_tokens}
        if self.tool_calls_made >= self.max_tool_calls:
            return {"exhausted": "tool_calls", "used": self.tool_calls_made,
                    "limit": self.max_tool_calls}
        if elapsed >= self.max_minutes:
            return {"exhausted": "time", "used": round(elapsed, 1),
                    "limit": self.max_minutes}

        return None  # Budget is fine

    def request_extension(self) -> bool:
        status = self.check_budget()
        print(f"\n  BUDGET CHECK: {status['exhausted']} limit reached "
              f"({status['used']}/{status['limit']})")
        print(f"  Tokens: {self.tokens_used}/{self.max_tokens} | "
              f"Calls: {self.tool_calls_made}/{self.max_tool_calls}")
        response = input("  Continue with extended budget? (y/n) > ")
        if response.lower() == "y":
            # Double the limits
            self.max_tokens *= 2
            self.max_tool_calls *= 2
            self.max_minutes *= 2
            return True
        return False

The budget pattern is great for cost control. The agent works autonomously within its budget, then pauses for a human checkpoint. This is particularly important when agents are running against paid APIs.

Combine patterns. Budget + approval gates is the most common production setup. The budget prevents runaway costs while approval gates prevent dangerous actions. They work on different dimensions (resources vs risk) so they complement each other naturally.

Pattern 4: Audit Trails

Not all oversight happens in real time. Audit trails let you review what the agent did after the fact — essential for debugging, compliance, and building trust.

import json
from datetime import datetime

class AuditLogger:
    """Log every agent action for post-hoc review."""

    def __init__(self, log_file="agent_audit.jsonl"):
        self.log_file = log_file

    def log(self, event_type: str, data: dict):
        entry = {
            "timestamp": datetime.now().isoformat(),
            "event": event_type,
            **data
        }
        with open(self.log_file, "a") as f:
            f.write(json.dumps(entry) + "\n")

    def log_tool_call(self, tool_name, tool_input, result, approved_by=None):
        self.log("tool_call", {
            "tool": tool_name,
            "input": tool_input,
            "result_preview": str(result)[:200],
            "approved_by": approved_by,  # "auto", "human", or "denied"
        })

    def log_decision(self, decision, reasoning):
        self.log("decision", {
            "decision": decision,
            "reasoning": reasoning,
        })


audit = AuditLogger()

# In your agent loop:
# audit.log_tool_call("write_file", {"path": "report.md"}, result, "human")
# audit.log_decision("escalate", "Conflicting info from 2 sources")

The JSONL format (one JSON object per line) makes audit logs easy to grep, filter, and analyze. Every entry has a timestamp and event type so you can reconstruct exactly what happened.

Audit logs may contain sensitive data. Tool inputs and results can include API keys, personal information, or proprietary content. Treat audit logs with the same security as your application data. Implement retention policies and access controls.

Pattern 5: Notification Without Blocking

Sometimes you want to know what the agent is doing without blocking it. Notifications let the agent continue working while alerting humans to important decisions:

class NotificationSystem:
    """Alert humans without blocking agent execution."""

    def __init__(self):
        self.notifications = []

    def notify(self, level: str, message: str, details: dict = None):
        entry = {
            "level": level,  # "info", "warning", "critical"
            "message": message,
            "details": details,
            "timestamp": datetime.now().isoformat()
        }
        self.notifications.append(entry)

        # In production, send to Slack, email, PagerDuty, etc.
        if level == "critical":
            print(f"\n  *** CRITICAL: {message} ***")
            # send_slack_alert(message)
        elif level == "warning":
            print(f"\n  [!] Warning: {message}")

    def get_summary(self) -> str:
        """End-of-run summary of all notifications."""
        warnings = [n for n in self.notifications if n["level"] == "warning"]
        critical = [n for n in self.notifications if n["level"] == "critical"]
        return (f"Agent completed. "
                f"{len(critical)} critical alerts, "
                f"{len(warnings)} warnings.")

The agent runs without interruption, but humans get alerts for unusual situations. This works well for background agents that process batches or monitor systems.

Graceful Degradation: When the Human Doesn’t Respond

What happens when your agent asks for approval and nobody’s there? You need a fallback strategy:

  • Timeout with safe default — wait N seconds, then skip the action and continue with read-only operations
  • Queue for later — save the pending action and move on to other work
  • Partial execution — do the safe parts, skip the risky parts, report what was skipped
  • Complete stop — checkpoint state and halt until a human responds
Don’t default to “allow” on timeout. If the human isn’t responding, the worst thing to do is auto-approve the action they were supposed to review. Default to the safest option: skip the action, log it, and continue with non-destructive work. You can always run the skipped actions later with approval.

Building Trust Gradually

The best human-in-the-loop systems evolve over time. Start tight, loosen as the agent proves itself:

Week 1: Approve everything → Week 2: Auto-approve reads → Month 2: Auto-approve safe writes → Month 6: Supervised autonomy

This mirrors how you onboard a new team member. Day one: pair on everything. Week two: review all PRs. Month two: review only complex changes. Month six: they’re autonomous with spot checks. The agent earns trust through a track record, not a promise.

In practice, you implement this by adjusting your ApprovalGate configuration over time — moving tools from needs_approval to auto_approve as you gain confidence in the agent’s behavior on those specific actions.

Putting It All Together

Here’s how the patterns combine in a production-ready agent loop:

def run_supervised_agent(question, gate, budget, audit, notifications):
    """Agent loop with full human-in-the-loop support."""
    messages = [{"role": "user", "content": question}]

    while True:
        # Check budget before each step
        budget_status = budget.check_budget()
        if budget_status and not budget.request_extension():
            notifications.notify("info", "Agent stopped: budget exhausted")
            break

        response = client.messages.create(
            model="claude-sonnet-4-6-20250514",
            max_tokens=4096, system=SYSTEM_PROMPT,
            tools=tools, messages=messages,
        )
        budget.record_usage(response.usage.input_tokens + response.usage.output_tokens)

        if response.stop_reason == "tool_use":
            tool_results = []
            for block in response.content:
                if block.type != "tool_use":
                    continue

                # Check permissions
                result = execute_with_approval(block.name, block.input)
                approval_type = gate.check(block.name, block.input)

                # Log everything
                audit.log_tool_call(block.name, block.input, result, approval_type)

                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result,
                })

            messages.append({"role": "assistant", "content": response.content})
            messages.append({"role": "user", "content": tool_results})
        else:
            audit.log_decision("complete", "Agent finished task")
            print(notifications.get_summary())
            return response.content[0].text
