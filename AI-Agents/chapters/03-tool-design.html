Chapter 3

Tool Design for Agents

The most underestimated skill in agent building — tools make or break your agent

Tools Matter More Than Models

Here’s a controversial claim: your choice of tools matters more than your choice of model. A great model with poorly designed tools will fail on basic tasks. A decent model with well-designed tools will handle complex work reliably.

Why? Because tools are the agent’s hands. The LLM is the brain — it can think, reason, plan — but without tools, it can only produce text. And when tools are confusing, ambiguous, or poorly documented, the brain makes bad decisions about which hand to use and how.

Think of it like designing an API for a junior developer who can only read your documentation — no Stack Overflow, no asking a colleague. If your API docs are vague, they’ll misuse the API. If the error messages are cryptic, they’ll get stuck. That’s exactly how an LLM experiences your tools.

Anatomy of a Tool

Every tool an agent uses has four parts:

The Four Parts of a Tool

  1. Name — a clear, descriptive identifier: search_web, read_file, create_issue
  2. Description — what the tool does, when to use it, and what to expect. This is what the LLM reads to decide whether to call it.
  3. Parameters — the inputs, defined as a JSON Schema with types, descriptions, required fields, and constraints
  4. Return format — what comes back: structured data, natural language, or an error message

The LLM sees the name, description, and parameter schema. It never sees your implementation code. So everything the model needs to know must be in those three fields. This is the fundamental insight of tool design: the interface IS the tool, as far as the agent is concerned.

The Golden Rules of Tool Design

Rule 1: Names Should Be Verbs

Tool names should describe an action. The LLM uses the name as its first signal for what the tool does.

Good names
search_web read_file create_pull_request list_directory delete_branch
Bad names
web_tool file_handler github_api utils do_stuff

Rule 2: Descriptions Are Documentation

The description is the most important part of the tool. It’s what the LLM reads to decide: “Should I use this tool? What parameters should I pass? What will I get back?”

Good description
Search the web for current information. Returns up to 5 results, each with a title, URL, and snippet. Use specific, detailed queries for better results. Example: "Python 3.12 new features" instead of just "Python".
Bad description
Searches the web.

A good description answers: What does it do? When should I use it? What does it return? Any tips for using it well?

Rule 3: Parameters Need Types, Descriptions, and Constraints

Every parameter should have a type, a description, and constraints where applicable. Don’t make the LLM guess what format you expect.

# Good: explicit types, descriptions, constraints
{
    "name": "search_issues",
    "description": "Search for issues in a GitHub repository. Returns "
                   "matching issues with title, number, state, and labels.",
    "input_schema": {
        "type": "object",
        "properties": {
            "repo": {
                "type": "string",
                "description": "Repository in 'owner/name' format, e.g. 'facebook/react'"
            },
            "query": {
                "type": "string",
                "description": "Search terms to match against issue title and body"
            },
            "state": {
                "type": "string",
                "enum": ["open", "closed", "all"],
                "description": "Filter by issue state. Defaults to 'open'."
            },
            "max_results": {
                "type": "integer",
                "description": "Maximum number of results to return (1-100). Default: 10."
            }
        },
        "required": ["repo", "query"]
    }
}

Notice the use of enum for the state parameter. When you have a fixed set of valid values, always use an enum. It prevents the model from inventing invalid values like “active” or “pending.”

Include example values in parameter descriptions. Instead of "The repository name", write "Repository in 'owner/name' format, e.g. 'facebook/react'". The example removes all ambiguity about the expected format.

Rule 4: Return Structured Data

When your tool returns data, prefer structured formats over free-form text. The LLM can parse JSON reliably; it struggles with ambiguous prose.

Structured return
{"results": [ {"title": "Issue #42", "state": "open", "labels": ["bug"]}, {"title": "Issue #37", "state": "open", "labels": ["feature"]} ], "total_count": 2}
Unstructured return
Found 2 issues. The first one is Issue #42 which is open and has the bug label. The second is Issue #37, also open, labeled as feature.

Exception: when the tool returns content meant to be read (like a web page or document), natural language is fine. The rule is: data should be structured, content can be natural.

Writing Descriptions That LLMs Understand

This is where most tool designers get it wrong. They write descriptions for humans and forget that an LLM reads them differently. Here’s what LLMs need:

  • Be explicit about edge cases: “Returns an empty array if no results are found, not an error.”
  • Specify what NOT to do: “Do not use this for binary files. Use read_binary_file instead.”
  • Include format examples: “Date format: YYYY-MM-DD. Example: 2026-04-02.”
  • Describe the relationship between tools: “Use search_web first to find URLs, then use read_page to get the full content.”

The Description Template

Use this structure for every tool description:

  1. What it does (one sentence)
  2. When to use it (one sentence about the use case)
  3. What it returns (the format and key fields)
  4. Usage tips (optional — examples, edge cases, related tools)

Example: “Execute a SQL query against the application database. Use this when you need to look up data, check records, or verify database state. Returns a JSON object with rows (array of objects) and row_count (integer). Only SELECT queries are allowed — for writes, use the write_record tool instead.”

Error Handling: Talk to the Agent, Not the Developer

When a tool fails, the error message goes straight to the LLM. A stack trace is useless. The agent needs to know: what happened, why, and what to do next.

# Bad: raw error the agent can't use
def search_web(query):
    try:
        results = api.search(query)
        return results
    except Exception as e:
        return str(e)  # "ConnectionError: HTTPSConnectionPool..."

# Good: structured error the agent can act on
def search_web(query):
    try:
        results = api.search(query)
        if not results:
            return {
                "status": "no_results",
                "message": f"No results found for '{query}'.",
                "suggestion": "Try broader search terms or different keywords."
            }
        return {"status": "success", "results": results}
    except RateLimitError:
        return {
            "status": "error",
            "message": "Search API rate limit exceeded.",
            "suggestion": "Wait a moment and try again, or use cached results."
        }
    except ConnectionError:
        return {
            "status": "error",
            "message": "Cannot connect to search API.",
            "suggestion": "The search service may be down. Try an alternative approach."
        }

The pattern: status + message + suggestion. The status lets the agent branch programmatically. The message explains what happened. The suggestion tells the agent how to recover. This is the difference between an agent that gets stuck on errors and one that works around them.

Tool Granularity: The Goldilocks Problem

How many tools should your agent have? How specific should each one be? This is the hardest design choice in agent tool design.

Too many tools overwhelms the model. LLMs degrade when presented with large tool sets. With 5–10 tools, models pick accurately. With 50+ tools, they start choosing wrong tools, hallucinating parameters, or ignoring tools entirely. If you need many tools, use the router pattern (Chapter 2) to split them across specialized agents.

Too Fine-Grained: Death by a Thousand Tools

Imagine giving your agent these tools for file operations:

# Too fine-grained — 8 tools for basic file operations
tools = [
    "open_file",       # Open a file handle
    "read_line",       # Read one line
    "read_all",        # Read entire file
    "write_line",      # Write one line
    "write_all",       # Write entire file
    "close_file",      # Close file handle
    "check_exists",    # Check if file exists
    "get_file_info"    # Get file metadata
]

The agent now has to figure out that it needs check_exists → open_file → read_all → close_file just to read a file. That’s four tool calls for one logical operation — wasted tokens, wasted time, more chances for errors.

Too Coarse: The God Tool

# Too coarse — one tool that does everything
tools = [
    {
        "name": "file_operation",
        "description": "Perform any file operation",
        "parameters": {
            "operation": {"type": "string"},  # "read", "write", "delete", "list", ...
            "path": {"type": "string"},
            "content": {"type": "string"},
            "options": {"type": "object"}     # ???
        }
    }
]

Now the agent has to figure out which “operation” string to pass, what “options” means, and which parameters apply to which operations. This is equally bad.

Just Right: Task-Oriented Tools

# Just right — one tool per meaningful action
tools = [
    "read_file",      # Read an entire file's contents
    "write_file",     # Write content to a file (creates or overwrites)
    "list_directory",  # List files in a directory
    "delete_file",     # Delete a file (with confirmation)
    "search_files"     # Search for files by name or content
]

Five tools, each doing one clear thing at a meaningful level of abstraction. The agent doesn’t need to manage file handles or choose operations from a menu. Each tool name tells the model exactly what it does.

The sweet spot is 5–15 tools per agent. Enough to be useful, few enough for the model to reason about. If you need more, split your tools across specialized agents connected by a router.

Dangerous Tools: When Tools Can Hurt

Some tools can do real damage: deleting files, sending emails, executing database writes, making purchases. These need special treatment.

Strategy 1: Confirmation Parameters

{
    "name": "delete_file",
    "description": "Delete a file permanently. This action cannot be undone. "
                   "Set confirm=true to actually delete; false for a dry run "
                   "that shows what would be deleted.",
    "input_schema": {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "Path to the file to delete"},
            "confirm": {
                "type": "boolean",
                "description": "Must be true to actually delete. false = dry run."
            }
        },
        "required": ["path", "confirm"]
    }
}

Strategy 2: Read-Only by Default

Give the agent read-only tools by default. Only add write tools when the task requires them. An agent with read_file and search_code can investigate safely. Add write_file only when you actually want it to make changes.

Strategy 3: Tool Annotations (MCP)

MCP introduced tool annotations — metadata that describes a tool’s behavior characteristics:

# MCP tool with annotations
{
    "name": "drop_table",
    "description": "Drop a database table and all its data.",
    "inputSchema": { ... },
    "annotations": {
        "readOnlyHint": false,
        "destructiveHint": true,
        "idempotentHint": false,
        "openWorldHint": false
    }
}

The destructiveHint flag tells the agent (and any approval system) that this tool can cause irreversible changes. Frameworks can automatically require human approval for tools marked as destructive — which is exactly what Claude Code does with its permission model.

Agent Trace: Good Tools vs Bad Tools

Let’s see the difference tool design makes. Here’s an agent trying to find and fix a bug, first with bad tools, then with good ones:

Think I need to find the file with the login bug. Let me search for it.
Tool file_operation(operation="search", path="src/", content="login") — using a poorly-designed god tool
Result Error: 'content' parameter not valid for 'search' operation. Use 'query' instead.
Tool file_operation(operation="search", path="src/", query="login") — guessing the right param name
Result Found 47 matches — raw count, no file paths or context
Think I got a count but no details. I don’t know which files matched. Let me try a different approach…

Three tool calls and the agent still has nothing useful. Now with well-designed tools:

Think I need to find the file with the login bug. Let me search the codebase.
Tool search_files(query="authenticate", path="src/", max_results=5)
Result {"results": [{"file": "src/auth/login.py", "line": 42, "snippet": "def authenticate(user, password):"}], "total": 1}
Tool read_file(path="src/auth/login.py")
Result Full file contents with line numbers. Bug found at line 47.
Respond Found the bug in src/auth/login.py:47 — ready to fix it.

Two tool calls, clear results, bug located. That’s the power of good tool design.

Tool Composition: Building Complex from Simple

Instead of building one massive tool that does everything, build simple tools that compose well together. The agent figures out the composition.

# Simple, composable tools
tools = [
    "search_web",       # Find information online
    "read_page",        # Get full content of a URL
    "extract_data",     # Pull structured data from text
    "save_to_file",     # Save content to local file
]

# The agent composes them naturally:
# search_web("NVIDIA quarterly earnings 2026")
#   → read_page("https://investor.nvidia.com/...")
#     → extract_data(text=page_content, fields=["revenue", "profit", "guidance"])
#       → save_to_file(path="report.json", content=extracted_data)

Each tool is simple and testable on its own. Together, they handle complex workflows. The LLM figures out the chaining — that’s exactly the kind of judgment call agents excel at.

MCP: The Standard for Tool Distribution

Until 2024, every agent framework had its own way to define and connect tools. MCP (Model Context Protocol) changed that.

MCP = USB-C for AI Tools

MCP is an open standard that lets any tool connect to any agent through a common protocol. Define your tools once as an MCP server, and any MCP-compatible agent (Claude, Cursor, VS Code, custom agents) can use them. In 2025, MCP was donated to the Linux Foundation’s Agentic AI Foundation with backing from OpenAI, Google, Microsoft, and AWS.

An MCP server is just a process that exposes tools over a standardized protocol. Here’s a minimal example:

from mcp.server import FastMCP

app = FastMCP("my-tools")

@app.tool()
def search_issues(repo: str, query: str, state: str = "open") -> dict:
    """Search for GitHub issues in a repository.

    Args:
        repo: Repository in 'owner/name' format, e.g. 'facebook/react'
        query: Search terms to match against issue title and body
        state: Filter by state: 'open', 'closed', or 'all'. Default: 'open'

    Returns:
        Dict with 'issues' array and 'total_count' integer.
    """
    # Your implementation here
    issues = github_api.search_issues(repo, query, state)
    return {"issues": issues, "total_count": len(issues)}

app.run()

The FastMCP helper generates the JSON Schema from your Python type hints and docstring. One definition, usable everywhere. The 2026 MCP roadmap adds OAuth 2.1 authentication, agent-to-agent coordination, and a verified server registry.

Testing Tools Before Giving Them to Agents

Always test your tools independently before giving them to an agent. If a tool has a bug, the agent will waste cycles trying to work around it — and you’ll blame the model when the real problem is your tool.

  • Call each tool with valid inputs — does it return the expected format?
  • Call each tool with edge-case inputs — empty strings, nulls, extreme values
  • Call each tool with invalid inputs — does it return a useful error message?
  • Verify the description is complete — could a stranger use the tool from the description alone?
  • Check parameter types match actual behavior — if you say “integer,” does the tool fail on “42” (string)?
The description test: show your tool definition (name, description, parameters) to a colleague without any other context. Can they use the tool correctly? If not, neither can the LLM.
