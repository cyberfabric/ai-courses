Chapter 12

Building a Real Agent System

Capstone project — design, build, and deploy a multi-agent code review system

The Capstone Challenge

You’ve learned every piece of the puzzle: architectures (Ch. 2), tool design (Ch. 3), the agent loop (Ch. 4), memory (Ch. 5), human-in-the-loop (Ch. 6), multi-agent orchestration (Ch. 7), error handling (Ch. 8), evaluation (Ch. 9), frameworks (Ch. 10), and production deployment (Ch. 11). Now we put it all together.

We’re building a multi-agent code review system. When someone opens a PR, our system automatically analyzes it from three angles — security, code quality, and logic — then merges the findings into a unified review. A human reviewer approves before anything gets posted.

This is a real system you can deploy on real repositories.

System Design

Code Review Agent Architecture

Four agents, coordinated by a supervisor using the pattern from Chapter 7:

PR Diff → Orchestrator → Security Agent
Orchestrator → Quality Agent → Orchestrator → Merge & Review

The Orchestrator (Sonnet) parses the PR, creates a review plan, delegates to specialists (Haiku), merges results, and presents the final review. A human gate approves before posting. Each specialist has a focused system prompt and sees only the relevant code.

Agent Roles

O
Orchestrator
Receives the PR diff, creates a review plan, delegates chunks to specialists, merges findings, deduplicates, ranks by severity, and produces the final review. Uses Claude Sonnet for reasoning.
S
Security Agent
Scans for vulnerabilities: SQL injection, XSS, hardcoded secrets, insecure dependencies, auth bypasses. Outputs findings with severity (critical/high/medium/low), file, line, and fix suggestion.
Q
Quality Agent
Checks code quality: naming conventions, code duplication, complexity, design patterns, error handling. References the project’s style guide if provided.

def get_pr_diff(base_branch: str, pr_branch: str) -> str:
    result = subprocess.run(
        ["git", "diff", f"{base_branch}...{pr_branch}"],
        capture_output=True, text=True, timeout=30
    )
    if result.returncode != 0:
        return json.dumps({"status": "error", "message": result.stderr})
    diff = result.stdout
    if len(diff) > 15000:
        diff = diff[:15000] + "\n\n[...diff truncated — too large]"
    return json.dumps({"status": "success", "diff": diff})

def read_file(path: str) -> str:
    try:
        with open(path) as f:
            content = f.read()
        if len(content) > 8000:
            content = content[:8000] + "\n[...truncated]"
        return json.dumps({"status": "success", "content": content})
    except FileNotFoundError:
        return json.dumps({"status": "error", "message": f"File not found: {path}"})

Step 2: Build the Specialist Agents

Each specialist gets a focused system prompt that tells it exactly what to look for and how to report findings:

SECURITY_PROMPT = """You are a security review specialist. Analyze code
diffs for security vulnerabilities.

CHECK FOR:
- SQL injection, XSS, command injection
- Hardcoded secrets (API keys, passwords, tokens)
- Insecure authentication or authorization
- Missing input validation on user-facing endpoints
- Insecure dependencies or deprecated crypto

OUTPUT FORMAT for each finding:
- severity: critical | high | medium | low
- file: the file path
- line: approximate line number
- issue: one-sentence description
- fix: suggested remediation

If you find NO issues, say so explicitly. Don't invent problems."""

QUALITY_PROMPT = """You are a code quality reviewer. Analyze code diffs
for quality and maintainability issues.

CHECK FOR:
- Unclear or inconsistent naming
- Functions longer than 50 lines
- Missing error handling
- Code duplication (copy-pasted logic)
- Overly complex logic that could be simplified
- Missing or incorrect type hints

OUTPUT FORMAT for each finding:
- severity: high | medium | low
- file: the file path
- line: approximate line number
- issue: one-sentence description
- suggestion: how to improve it

Focus on substantive issues. Don't flag style preferences."""

def run_specialist(system_prompt: str, code_context: str,
                   model: str = "claude-haiku-4-5-20251001") -> str:
    """Run a specialist review agent."""
    response = client.messages.create(
        model=model,
        max_tokens=2048,
        system=system_prompt,
        messages=[{"role": "user", "content": code_context}],
    )
    return response.content[0].text
Notice how each prompt ends with “Don’t invent problems” or “Don’t flag style preferences.” Without these guardrails, LLMs tend to produce false positives — flagging every line as a potential issue. Precision matters more than recall in code review.

Step 3: Build the Orchestrator

ORCHESTRATOR_PROMPT = """You are a code review orchestrator. Your job is to
coordinate a thorough code review by delegating to specialists.

PROCESS:
1. Get the PR diff and list of changed files
2. Delegate security review to the security specialist
3. Delegate quality review to the quality specialist
4. Merge all findings, remove duplicates, rank by severity
5. Produce a final review summary

RULES:
- Pass relevant code context to each specialist — not the entire diff
- Group findings by file
- Start with critical/high issues, then medium, then low
- Include an overall assessment: APPROVE, REQUEST_CHANGES, or NEEDS_DISCUSSION
- Be concise — reviewers don't want to read a novel"""

def run_orchestrator(base_branch: str, pr_branch: str) -> dict:
    """Run the full code review pipeline."""
    print("  [1/5] Getting PR diff...")
    diff = get_pr_diff(base_branch, pr_branch)
    diff_data = json.loads(diff)

    if diff_data["status"] != "success":
        return {"status": "error", "message": "Could not get diff"}

    code = diff_data["diff"]

    print("  [2/5] Security review...")
    security_findings = run_specialist(SECURITY_PROMPT, code)

    print("  [3/5] Quality review...")
    quality_findings = run_specialist(QUALITY_PROMPT, code)

    print("  [4/5] Merging findings...")
    merge_prompt = f"""Merge these review findings into a single, organized
code review. Remove duplicates. Group by file. Rank by severity.

SECURITY FINDINGS:
{security_findings}

QUALITY FINDINGS:
{quality_findings}

Produce a final review with:
1. Overall verdict: APPROVE, REQUEST_CHANGES, or NEEDS_DISCUSSION
2. Summary (2-3 sentences)
3. Findings grouped by file, ordered by severity
4. Each finding with: severity, line, issue, suggestion"""

    final_review = run_specialist(
        "You merge code review findings into a clear, actionable review.",
        merge_prompt,
        model="claude-sonnet-4-6-20250514",  # Smart model for merging
    )

    print("  [5/5] Review complete.")
    return {"status": "success", "review": final_review}

Step 4: Add Human-in-the-Loop

The reviewer sees the findings and decides what to post (from Chapter 6):

def human_review_gate(review: str) -> dict:
    """Let a human approve, edit, or reject the review."""
    print("\n" + "="*60)
    print("AGENT CODE REVIEW — PENDING APPROVAL")
    print("="*60)
    print(review)
    print("="*60)
    print("\nOptions:")
    print("  [a] Approve and post as-is")
    print("  [e] Edit before posting")
    print("  [r] Reject — do not post")
    print("  [d] Discard specific findings")

    choice = input("\nYour choice (a/e/r/d) > ").strip().lower()

    if choice == "a":
        return {"action": "post", "review": review}
    elif choice == "e":
        print("\nPaste your edited review (end with an empty line):")
        lines = []
        while True:
            line = input()
            if line == "":
                break
            lines.append(line)
        return {"action": "post", "review": "\n".join(lines)}
    elif choice == "d":
        print("Enter finding numbers to remove (comma-separated):")
        # In production, parse and filter findings
        return {"action": "post", "review": review}
    else:
        return {"action": "reject"}

Step 5: Handle Errors Gracefully

What happens when one specialist fails? We don’t want to lose the entire review (from Chapter 8):

Partial Results Pattern

Problem: In a multi-agent system, one agent fails. Do you retry everything or lose the work?

Solution: Collect results independently. If one agent fails, still present results from the others with a note about what’s missing.

def run_with_partial_results(specialists: dict, code: str) -> dict:
    results = {}
    failures = []

    for name, prompt in specialists.items():
        try:
            results[name] = run_specialist(prompt, code)
        except Exception as e:
            failures.append({"agent": name, "error": str(e)})
            results[name] = f"[{name} review unavailable: {e}]"

    return {
        "results": results,
        "failures": failures,
        "complete": len(failures) == 0,
        "note": f"{len(failures)} agent(s) failed" if failures else None,
    }

When to use: Any multi-agent system where partial output is better than no output. Code review with only security findings is still valuable, even without quality analysis.

Putting It All Together

def review_pr(base_branch: str = "main", pr_branch: str = None):
    """Full code review pipeline with human approval."""
    if not pr_branch:
        pr_branch = subprocess.run(
            ["git", "branch", "--show-current"],
            capture_output=True, text=True
        ).stdout.strip()

    print(f"\n  Reviewing: {pr_branch} → {base_branch}")

    # Run the orchestrator
    result = run_orchestrator(base_branch, pr_branch)

    if result["status"] != "success":
        print(f"  Error: {result['message']}")
        return

    # Human approval gate
    decision = human_review_gate(result["review"])

    if decision["action"] == "post":
        print("\n  Review approved. In production, this would be posted "
              "as a PR comment via the GitHub API.")
        # gh_api.post_review_comment(pr_number, decision["review"])
    else:
        print("\n  Review rejected. No comments posted.")

# Run it on the current branch
review_pr()

Step 2 Security Agent analyzes diff → Found 2 issues: hardcoded JWT secret in auth.py:15 (CRITICAL), missing CSRF token validation in views.py:82 (HIGH)
Step 3 Quality Agent analyzes diff → Found 3 issues: authenticate() is 67 lines (HIGH), inconsistent error returns (MEDIUM), missing type hints on public API (LOW)
Step 4 Orchestrator merges: 5 unique findings, verdict REQUEST_CHANGES (critical security issue found)
Step 5 Human gate: Reviewer sees findings. Approves security and quality items, removes the type hints suggestion (team doesn’t enforce this).
Posted 4 findings posted as PR comment. Total cost: $0.08. Time: 18 seconds.

Cost and Latency Budget

$0.05–0.15 Cost per review (Haiku specialists + Sonnet orchestrator)
15–30s Wall-clock time for a typical 200-line PR
3 agents Security + Quality + Merger (can run security & quality in parallel)

Running specialists in parallel (security + quality simultaneously) cuts latency by ~40%. With prompt caching on the system prompts, cost drops another 30% for subsequent reviews.

Evaluation: Testing the System

Use the evaluation techniques from Chapter 9 to measure your review system:

review_eval = EvalSuite("Code Review System v1")

# Test: known vulnerability should be caught
review_eval.add_case(
    query="diff with SQL injection in user_query = f'SELECT * FROM users WHERE id={id}'",
    expected_contains=["SQL injection", "parameterized", "critical"],
)

# Test: clean code should get APPROVE
review_eval.add_case(
    query="diff with well-structured code, proper error handling, typed params",
    expected_contains=["APPROVE"],
)

# Test: should not flag style preferences
review_eval.add_case(
    query="diff using single quotes instead of double quotes in Python",
    expected_contains=[],  # Should find no issues
)
Measure precision, not just recall. A review system that flags everything has 100% recall but is useless — developers will ignore it. Aim for high precision (>80% of flagged issues are real issues) even if it means missing some edge cases. A trusted reviewer that catches 70% of real issues is more valuable than a noisy one that catches 95% but has a 50% false positive rate.

Lessons Learned

After building this system, here’s what you’ll discover:

Diff context is not enough. A diff shows what changed but not why. The security agent might see password = request.form["password"] and flag it, not realizing it’s immediately hashed on the next line (which isn’t in the diff). Give specialists access to read_file so they can check surrounding context. This costs more tokens but dramatically reduces false positives.
  • Smaller PRs get better reviews — under 200 lines, agents catch most real issues. Over 500 lines, quality drops sharply
  • System prompts need iteration — your first draft will produce too many false positives. Iterate based on real PR feedback
  • The merger agent is critical — without deduplication and ranking, the review is a noisy list instead of actionable feedback
  • Human-in-the-loop builds trust — developers accept AI reviews much faster when they know a human approved each one
  • Start with one specialist — get security review working and trusted before adding quality and performance

The Agent Builder’s Mindset

As you build more agent systems, keep these principles close:

Seven Principles for Agent Builders

  1. Start simple, add complexity only when the simple version fails — a single agent with good tools beats a complex multi-agent system you can’t debug
  2. Tools matter more than models — invest time in tool design (Ch. 3), not model selection
  3. The system prompt IS the architecture — a clear prompt turns a generic LLM into a focused specialist
  4. Always have an escape hatch — humans must be able to intervene, override, and shut down
  5. Measure everything — cost, accuracy, latency. You can’t improve what you don’t measure
  6. Errors are normal — design for recovery, not perfection. 85% per step is realistic; 100% is fantasy
  7. Build the loop first, then the framework — understand while True before reaching for abstractions
