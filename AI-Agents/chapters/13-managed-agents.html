Managed Agents: Deploy on Anthropic’s Infrastructure

Zero-infrastructure agent deployment — Anthropic hosts the harness, you define the agent

From “Go Build Something Amazing” to “Go Ship It”

In Chapter 12, you built a multi-agent code review system. In Chapter 11, you learned what it takes to deploy agents to production yourself — containers, scaling, monitoring, security, the works. It was a lot.

What if you could skip most of that?

Anthropic’s Managed Agents launched in public beta in April 2026 with a simple proposition: you define what your agent does (model, tools, system prompt), and Anthropic runs it in their cloud. No containers to manage. No scaling to configure. No sandbox to secure. You get a production-ready agent harness with a single API call.

Think of it this way: Chapter 11 was AWS EC2 — full control, full responsibility. This chapter is Heroku — push your config and go.

Why this chapter exists: The course was planned with 12 chapters, but Managed Agents changed the deployment story so fundamentally that it deserved its own chapter. If you’re building agents in 2026, this is likely how you’ll deploy your first production system.

The Deployment Problem, Revisited

Remember Chapter 11? Sandboxing, container orchestration, the tool-use loop, credential management, session persistence, scaling, observability — that’s a lot of infrastructure for “I just want my agent to run a task.” Managed Agents handles all of it. The trade-off? Less customization of the runtime. But for the vast majority of use cases, the built-in capabilities are more than enough.

The Four Core Concepts

Managed Agents Architecture

Everything in Managed Agents revolves around four concepts:

Agent — The configuration. Your model choice, system prompt, tool definitions, and MCP server attachments. Created once, referenced by ID. Think of it as a deployment manifest.

Environment — The container template. Which packages to pre-install, networking rules, file mounts. This defines what the agent’s sandbox looks like when it boots up.

Session — A running instance. When you start a session, Anthropic spins up the environment, loads the agent config, and begins executing. Each session is isolated — its own filesystem, its own state.

Events — The communication channel. Your application and the agent exchange messages via server-sent events (SSE). You send tasks in; the agent streams responses, tool calls, and status updates out.

The relationship: an Agent runs inside an Environment, creating a Session, which communicates through Events.

Under the Hood: Brain, Harness, Sandbox

Anthropic’s engineering blog describes the internal architecture as three decoupled components:

  • Session (the Brain) — An append-only event log that stores every message, tool call, and result. This is the source of truth. If the container crashes, the session survives.
  • Harness (the Logic) — The stateless loop that calls Claude, parses tool calls, routes them to the sandbox, and feeds results back. This is the while True loop you built in Chapter 4 — but managed.
  • Sandbox (the Hands) — The isolated container where code actually runs. File operations, bash commands, web fetches — all happen here. Containers are replaceable; if one fails, another spins up with the same session state.

This separation is why Managed Agents can offer reliability guarantees that are hard to replicate yourself. The session (your data) is decoupled from the container (the execution), so infrastructure failures don’t lose your work.

Creating Your First Managed Agent

Let’s get to code. First, install the Anthropic SDK:

pip install anthropic

Now create an agent. This is just a configuration — nothing is running yet:

from anthropic import Anthropic

client = Anthropic()  # Uses ANTHROPIC_API_KEY env var

# Step 1: Create the agent configuration
agent = client.beta.agents.create(
    name="Research Assistant",
    model="claude-sonnet-4-6",
    system="""You are a research assistant. When given a topic:
1. Search the web for current information
2. Read relevant pages
3. Synthesize findings into a clear summary
4. Save results to a file

Be thorough but concise. Cite your sources.""",
    tools=[
        {"type": "agent_toolset_20260401"},  # All built-in tools
    ],
)

print(f"Agent created: {agent.id}")
The beta header: Managed Agents requires the header anthropic-beta: managed-agents-2026-04-01. The Python and TypeScript SDKs set this automatically when you use client.beta.agents. If you’re using raw HTTP, add it yourself.

Configuring the Environment

The environment defines what the agent’s container looks like. Need pandas for data analysis? Numpy for computations? Specify them here:

# Step 2: Create the environment
environment = client.beta.environments.create(
    name="research-env",
    config={
        "type": "cloud",
        "packages": {
            "pip": ["requests", "beautifulsoup4", "pandas"],
            "npm": ["marked"],       # Node.js packages too
            "apt": ["jq", "curl"],   # System packages
        },
        "networking": {
            "type": "unrestricted",  # Full outbound access
        },
    },
)

print(f"Environment created: {environment.id}")

Networking has two modes: unrestricted (default — full outbound access minus a safety blocklist) and limited (restricted to a specific allowed_hosts list). Use limited when you want to lock down exactly which external services the agent can reach:

# Locked-down environment for sensitive tasks
secure_env = client.beta.environments.create(
    name="secure-env",
    config={
        "type": "cloud",
        "packages": {"pip": ["anthropic"]},
        "networking": {
            "type": "limited",
            "allowed_hosts": ["api.github.com", "pypi.org"],
            "allow_mcp_servers": True,
            "allow_package_managers": True,
        },
    },
)

Starting a Session and Streaming Events

Now the fun part — running the agent. You create a session, open an SSE stream, send a message, and watch the agent work:

# Step 3: Start a session
session = client.beta.sessions.create(
    agent=agent.id,
    environment_id=environment.id,
    title="Research: AI agent frameworks 2026",
)

print(f"Session started: {session.id}")

# Step 4: Stream events
with client.beta.sessions.events.stream(session.id) as stream:
    # Send the task
    client.beta.sessions.events.send(
        session.id,
        events=[{
            "type": "user.message",
            "content": [{
                "type": "text",
                "text": "Research the top 5 AI agent frameworks "
                       "in 2026. Compare their features, maturity, "
                       "and best use cases. Save a report to report.md",
            }],
        }],
    )

    # Process the stream
    for event in stream:
        match event.type:
            case "agent.message":
                for block in event.content:
                    if hasattr(block, "text"):
                        print(block.text, end="")
            case "agent.tool_use":
                print(f"\n🔧 Using tool: {event.name}")
            case "session.status_idle":
                print("\n\n✅ Agent finished.")
                break

That’s it. About 40 lines of Python to go from “I want an agent” to “I have an agent running in production infrastructure.” Compare that to the hundreds of lines of container orchestration, sandbox setup, and harness code from Chapter 11.

Built-in Tools: What Your Agent Gets for Free

When you specify agent_toolset_20260401, your agent gets a full development toolkit:

bash
Execute shell commands in the container
read
Read files from the sandbox filesystem
write
Create and overwrite files
edit
String replacements in existing files
glob
Find files by pattern matching
grep
Search file contents with regex
web_fetch
Fetch content from any URL
web_search
Search the web ($10/1K searches)

You can enable or disable individual tools. Need a code-only agent that can’t browse the web? Disable web_fetch and web_search:

agent = client.beta.agents.create(
    name="Code-Only Agent",
    model="claude-sonnet-4-6",
    system="You write and test Python code. No web access.",
    tools=[{
        "type": "agent_toolset_20260401",
        "configs": [
            {"name": "web_fetch", "enabled": False},
            {"name": "web_search", "enabled": False},
        ],
    }],
)

Or flip the default — disable everything, then allowlist only what you need by setting "default_config": {"enabled": False} and explicitly enabling individual tools.

Custom Tools: Your Code, Their Infrastructure

Built-in tools cover a lot, but what about your own APIs? Custom tools let you define tools that your application executes when the agent calls them. The agent sees the tool definition; when it makes a call, you receive an event, run your logic, and send the result back.

Pattern: The Custom Tool Loop

Problem: Your agent needs to interact with external systems (databases, internal APIs, third-party services) that can’t run inside the sandbox.

Solution: Define custom tools with JSON schemas. When the agent invokes one, your application receives an agent.tool_use event, executes the logic locally, and sends back a user.custom_tool_result event.

Why it works: The agent thinks it’s calling a tool like any other. Your app acts as the bridge between the managed sandbox and your own infrastructure. Credentials never touch the agent’s environment.

When to use: Any time the agent needs to reach systems outside its sandbox — databases, payment APIs, internal microservices, proprietary tools.

# Define an agent with a custom tool
agent = client.beta.agents.create(
    name="DB Query Agent",
    model="claude-sonnet-4-6",
    system="You help users query the product database.",
    tools=[
        {"type": "agent_toolset_20260401"},
        {
            "type": "custom",
            "name": "query_database",
            "description": (
                "Execute a read-only SQL query against the product "
                "database. Returns rows as JSON. Only SELECT "
                "statements are allowed. Use this to answer "
                "questions about products, inventory, and pricing."
            ),
            "input_schema": {
                "type": "object",
                "properties": {
                    "sql": {
                        "type": "string",
                        "description": "The SQL SELECT query",
                    },
                },
                "required": ["sql"],
            },
        },
    ],
)

# Handle custom tool calls in the event stream
with client.beta.sessions.events.stream(session.id) as stream:
    client.beta.sessions.events.send(session.id, events=[{
        "type": "user.message",
        "content": [{"type": "text",
                      "text": "How many products are priced over $100?"}],
    }])

    for event in stream:
        if event.type == "agent.tool_use" and event.name == "query_database":
            # YOUR code runs the query (not the agent's sandbox)
            result = run_query(event.input["sql"])

            # Send the result back to the agent
            client.beta.sessions.events.send(session.id, events=[{
                "type": "user.custom_tool_result",
                "tool_use_id": event.id,
                "content": json.dumps(result),
            }])

        elif event.type == "agent.message":
            for block in event.content:
                if hasattr(block, "text"):
                    print(block.text, end="")

        elif event.type == "session.status_idle":
            break
Security note: Custom tools execute in YOUR application, not the sandbox. This means the agent’s SQL query runs against your real database. Always validate inputs — enforce read-only queries, parameterize inputs, and apply the same security practices you would for any user-facing API.

Connecting MCP Servers

Managed Agents supports MCP (Model Context Protocol) natively. Set "allow_mcp_servers": true in your environment’s networking config, add the MCP server host to allowed_hosts, and your agent discovers its tools at session startup. MCP servers let you add GitHub integration, Slack messaging, database access, or any custom capability through a standardized protocol — no custom tool boilerplate needed.

Multi-Agent on Managed Infrastructure

Remember the supervisor pattern from Chapter 7? Managed Agents supports it natively through callable_agents. Define specialist agents, then create an orchestrator that can delegate to them:

# Specialist agents
reviewer = client.beta.agents.create(
    name="Code Reviewer",
    model="claude-sonnet-4-6",
    system="You review code for quality, security, and best "
           "practices. Be specific and cite line numbers.",
    tools=[{"type": "agent_toolset_20260401",
            "configs": [{"name": "bash", "enabled": False}]}],
)

test_writer = client.beta.agents.create(
    name="Test Writer",
    model="claude-sonnet-4-6",
    system="You write comprehensive unit tests in pytest. "
           "Cover edge cases and error paths.",
    tools=[{"type": "agent_toolset_20260401"}],
)

# Orchestrator that delegates to specialists
lead = client.beta.agents.create(
    name="Engineering Lead",
    model="claude-sonnet-4-6",
    system="You coordinate engineering tasks. Delegate code "
           "review to Code Reviewer and test writing to "
           "Test Writer. Synthesize their results into a "
           "final report.",
    tools=[{"type": "agent_toolset_20260401"}],
    callable_agents=[
        {"type": "agent", "id": reviewer.id,
         "version": reviewer.version},
        {"type": "agent", "id": test_writer.id,
         "version": test_writer.version},
    ],
)

When the orchestrator delegates, each specialist runs in its own thread within the session — isolated conversation history, separate execution context. You can monitor delegation via SSE events: session.thread_created fires when a specialist is spawned, and session.thread_idle fires when it finishes.

Delegation is one level deep: The orchestrator can delegate to specialists, but specialists cannot delegate further. This keeps execution predictable and costs bounded. If you need deeper hierarchies, create multiple orchestrator layers.

Managed vs Self-Hosted: The Decision Framework

When should you use Managed Agents vs the self-hosted approach from Chapter 11? Here’s the honest comparison:

The rule of thumb: Start with Managed Agents. Move to self-hosted when you hit a limitation that matters — not before. Most teams never hit that point. The ones that do usually need multi-provider support or strict data residency requirements.

What It Costs

Managed Agents billing has three components: $0.08 per agent-hour (millisecond precision, only while session status is running), standard Claude API token pricing, and $10 per 1,000 web searches if enabled. No monthly base fee — you pay for what you use. Idle time and terminated sessions don’t accrue runtime costs.

Common Pitfalls

Pitfall: Leaving sessions running

Sessions accrue runtime costs while active. If your code crashes before closing the session, the agent keeps running (and billing). Always use try/finally or context managers to ensure cleanup.

Pitfall: Not validating custom tool inputs

Custom tool calls are effectively untrusted input. An agent might craft a SQL query that deletes data, or call an API with unexpected parameters. Validate custom tool inputs as strictly as you would user input from a web form.

Putting It All Together: The Research Agent, Managed

Remember the research agent from Chapter 4? You built it from scratch — the tool-use loop, the tool definitions, the error handling. Here’s the same agent, deployed on Managed Agents:

from anthropic import Anthropic

client = Anthropic()

agent = client.beta.agents.create(
    name="Research Agent v2",
    model="claude-sonnet-4-6",
    system="""You are a research agent. Given a topic:
1. Search the web for authoritative sources
2. Read the most relevant pages
3. Synthesize findings into a structured summary
4. Save the summary to research_output.md
Always cite sources with URLs. Be thorough but concise.""",
    tools=[{"type": "agent_toolset_20260401"}],
)

environment = client.beta.environments.create(
    name="research-env",
    config={"type": "cloud",
            "networking": {"type": "unrestricted"}},
)

def run_research(topic: str) -> str:
    session = client.beta.sessions.create(
        agent=agent.id, environment_id=environment.id,
        title=f"Research: {topic}",
    )
    result_text = []
    with client.beta.sessions.events.stream(session.id) as stream:
        client.beta.sessions.events.send(session.id, events=[{
            "type": "user.message",
            "content": [{"type": "text",
                         "text": f"Research: {topic}"}],
        }])
        for event in stream:
            if event.type == "agent.message":
                for block in event.content:
                    if hasattr(block, "text"):
                        result_text.append(block.text)
            elif event.type == "session.status_idle":
                break
    return "".join(result_text)

summary = run_research("quantum computing breakthroughs 2026")
print(summary)

Compare this to Chapter 4’s implementation: no manual tool-use loop, no tool function definitions, no JSON schema writing, no error-handling boilerplate. The managed harness handles all of that. Your code focuses entirely on what the agent should do, not how the loop works.

Build It: Deploy Your Research Agent

Take the research agent above and extend it into a production-ready deployment:

  1. Create the agent and environment using the code above. Run a test query and verify results.
  2. Add a custom tool — define save_to_notion (or any service) as a custom tool. Handle the agent.tool_use event and return results via user.custom_tool_result.
  3. Build a multi-agent version — create a “Fact Checker” specialist and wire it via callable_agents. The researcher delegates fact-checking before finalizing.
  4. Measure costs — run 5 queries and track token usage plus runtime hours. Compare to Chapter 4 self-hosted costs.
