Chapter 8

Error Handling & Recovery

Making agents robust — because in production, everything that can fail will fail

Why Agent Errors Are Different

Traditional software bugs are deterministic: the same input produces the same error every time. Agent failures are a different beast entirely. The same input can produce different results on different runs. An agent might succeed 9 times and fail on the 10th because the LLM chose a slightly different reasoning path, or a web page loaded differently, or the context window was arranged just differently enough to cause a hallucination.

Here’s the math that makes this urgent: if each step of your agent has an 85% success rate (quite good for an LLM), then a 10-step task has a success rate of 0.8510 = 20%. That’s right — a “good” agent fails 80% of the time on complex tasks without error handling. Robust recovery isn’t a nice-to-have — it’s the difference between a demo and a product.

Error handling in agents is like error handling in distributed systems

You need retries, circuit breakers, fallbacks, and monitoring — because no single step is guaranteed to succeed. If you’ve built microservices, you already know these patterns. The difference is that your “service” makes non-deterministic decisions.

Categories of Agent Failure

Every agent failure falls into one of these categories:

Category Example Recovery Strategy
Tool error API timeout, rate limit, invalid URL Retry with backoff, try alternative tool
LLM hallucination Invents a function that doesn’t exist, fabricates data Output validation, self-correction prompt
Format violation Returns text instead of JSON, malformed tool calls Retry with format reminder, parse defensively
Infinite loop Agent repeats the same action endlessly Step limit, loop detection, circuit breaker
Cost overrun Token budget exhausted, API credits drained Budget controller (Ch. 6), tiered model routing
Refusal Model declines to perform a legitimate action Rephrase the request, provide more context

Retry Strategies

Simple Retry with Backoff

The most common recovery: try again, with increasing delays between attempts. This handles transient errors like API timeouts and rate limits.

import time
import random

def retry_with_backoff(fn, max_retries=3, base_delay=1.0):
    """Retry a function with exponential backoff and jitter."""
    for attempt in range(max_retries):
        try:
            return fn()
        except Exception as e:
            if attempt == max_retries - 1:
                raise  # Last attempt — give up
            delay = base_delay * (2 ** attempt) + random.uniform(0, 1)
            print(f"  [Retry {attempt + 1}/{max_retries}] "
                  f"Error: {e}. Waiting {delay:.1f}s...")
            time.sleep(delay)

Retry with Modified Prompt

When the error is from the LLM (bad format, hallucination), retrying with the same input often produces the same error. Instead, modify the prompt to include the error:

def retry_with_feedback(client, messages, tools, error_msg, max_retries=2):
    """Retry an LLM call with error feedback injected."""
    for attempt in range(max_retries):
        if attempt > 0:
            # Add the error as context for the next attempt
            messages.append({
                "role": "user",
                "content": (
                    f"Your previous attempt had an error: {error_msg}\n"
                    f"Please try again, avoiding this issue."
                )
            })

        response = client.messages.create(
            model="claude-sonnet-4-6-20250514",
            max_tokens=4096,
            tools=tools,
            messages=messages,
        )

        # Validate the response
        error_msg = validate_response(response)
        if error_msg is None:
            return response  # Valid response

    return None  # All retries failed
LLMs tend to repeat the same mistake if you just retry. The key is feeding the error back so the model can reason about what went wrong. This is a lightweight form of the Reflexion pattern from Chapter 2.

Fallback Patterns

When one approach fails completely, try an alternative:

The Fallback Chain

Problem: A tool or approach fails and retrying won’t help (e.g., an API is down, a page is blocked).

Solution: Define a chain of alternatives. Try tool A, then tool B, then escalate to a human.

Implementation:

def search_with_fallback(query: str) -> str:
    """Try multiple search approaches before giving up."""
    strategies = [
        ("primary_search", lambda: search_api.search(query)),
        ("backup_search", lambda: duckduckgo_search(query)),
        ("cached_search", lambda: check_cache(query)),
    ]

    for name, strategy in strategies:
        try:
            result = strategy()
            if result and result.get("status") == "success":
                return result
            print(f"  [{name}] No results, trying next...")
        except Exception as e:
            print(f"  [{name}] Failed: {e}, trying next...")

    # All strategies failed — escalate
    return {
        "status": "all_failed",
        "message": "All search methods failed.",
        "suggestion": "Ask the human for the information directly."
    }

When to use: Any tool that depends on external services (APIs, web scraping, databases). Always have at least one fallback for critical tools.

Self-Correction

The most powerful error recovery pattern: the agent detects and fixes its own mistakes. This is the Reflexion architecture from Chapter 2, applied specifically to error handling.

Tool write_file({"path": "calc.py", "content": "def avg(nums):\n return sum(nums) / len(nums)"})
Tool run_tests({"file": "test_calc.py"})
Result FAILED: test_avg_empty_list — ZeroDivisionError: division by zero
Think The test failed because I didn’t handle the empty list case. I need to add a guard clause for when len(nums) == 0.
Tool write_file({"path": "calc.py", "content": "def avg(nums):\n if not nums:\n return 0\n return sum(nums) / len(nums)"})
Tool run_tests({"file": "test_calc.py"})
Result All 5 tests passed.
Respond Fixed the empty list bug and all tests pass now.

This “write, test, fix” cycle is emergent self-correction — modern LLMs do it naturally when they can see test output. But you can also make it explicit with a validation step:

def self_correcting_action(agent, action, validator, max_attempts=3):
    """Execute an action with self-correction on failure."""
    for attempt in range(1, max_attempts + 1):
        result = agent.execute(action)

        # Validate the result
        validation = validator(result)
        if validation["valid"]:
            return result

        print(f"  [Attempt {attempt}] Validation failed: {validation['error']}")

        # Ask the agent to fix its output
        action = (
            f"Your previous output had this error: {validation['error']}\n"
            f"Previous output: {result}\n"
            f"Please fix the issue and try again."
        )

    # All attempts failed
    return {"status": "failed", "last_error": validation["error"]}
LLMs tend to repeat the same flawed reasoning across attempts. If the agent makes the same mistake three times in a row, a fourth retry won’t help. Instead, try a different perspective: use a different model, change the system prompt, or simplify the task. Multi-Agent Reflexion — having a different agent review the work — outperforms single-agent self-correction significantly.

Guardrails: Validate Before Acting

Don’t wait for errors — prevent them. Guardrails validate agent outputs before they reach the real world.

The Three Guardrail Layers

  1. Input guardrails — validate the user’s request before the agent starts. Catch prompt injection, off-topic requests, or requests for blocked actions.
  2. Execution guardrails — validate tool calls before executing them. Check parameters, verify permissions, enforce rate limits.
  3. Output guardrails — validate the agent’s final response before returning it. Check for PII, hallucinated URLs, format compliance.

These layers run in parallel with agent execution for minimal latency impact. OpenAI Agents SDK implements this natively — guardrails fail fast and immediately stop the agent if triggered. Production systems typically add 50–200ms of latency for guardrail checks.

def validate_tool_call(tool_name: str, tool_input: dict) -> dict:
    """Pre-execution guardrail: validate before running the tool."""
    errors = []

    # Check for obviously wrong parameters
    if tool_name == "write_file":
        path = tool_input.get("path", "")
        if ".." in path or path.startswith("/"):
            errors.append("Path traversal detected — blocked for security")
        if path.endswith((".env", ".pem", ".key")):
            errors.append("Writing to sensitive file type — blocked")

    if tool_name == "search_web":
        query = tool_input.get("query", "")
        if len(query) < 3:
            errors.append("Search query too short — provide more detail")

    if tool_name == "run_command":
        cmd = tool_input.get("command", "")
        dangerous = ["rm -rf", "drop table", "format", "mkfs"]
        if any(d in cmd.lower() for d in dangerous):
            errors.append(f"Dangerous command blocked: {cmd}")

    if errors:
        return {"valid": False, "errors": errors}
    return {"valid": True}

validation = validate_tool_call(block.name, block.input)

The Pre-Flight Check Pattern

Before executing a plan, verify that the conditions for success are met:

def preflight_check(plan: list, available_tools: list) -> dict:
    """Verify a plan is executable before starting."""
    issues = []

    for step in plan:
        # Check that required tools exist
        if step.get("tool") and step["tool"] not in available_tools:
            issues.append(f"Step '{step['description']}' needs "
                          f"tool '{step['tool']}' which is not available")

        # Check for dependencies
        if step.get("depends_on"):
            dep = step["depends_on"]
            if dep not in [s["id"] for s in plan]:
                issues.append(f"Step '{step['description']}' depends on "
                              f"'{dep}' which doesn't exist in the plan")

    if issues:
        return {"ready": False, "issues": issues}
    return {"ready": True}

Circuit Breakers

A circuit breaker stops the agent when the error rate gets too high, preventing catastrophic damage from runaway failures:

class CircuitBreaker:
    """Stop the agent when too many errors occur."""

    def __init__(self, max_errors=5, window_size=10, cooldown=60):
        self.max_errors = max_errors
        self.window_size = window_size
        self.cooldown = cooldown
        self.results = []       # Recent results: True=success, False=failure
        self.tripped_at = None

    def record(self, success: bool):
        self.results.append(success)
        if len(self.results) > self.window_size:
            self.results.pop(0)

    def is_open(self) -> bool:
        """Check if the circuit breaker has tripped."""
        if self.tripped_at:
            if time.time() - self.tripped_at < self.cooldown:
                return True  # Still in cooldown
            self.tripped_at = None  # Reset after cooldown

        recent_errors = self.results.count(False)
        if recent_errors >= self.max_errors:
            self.tripped_at = time.time()
            return True
        return False

    def check_or_raise(self):
        if self.is_open():
            raise RuntimeError(
                f"Circuit breaker tripped: {self.results.count(False)} "
                f"errors in last {self.window_size} operations. "
                f"Agent halted for {self.cooldown}s."
            )


breaker = CircuitBreaker(max_errors=3, window_size=5)

Real cost of missing circuit breakers: a reported production incident saw an inter-agent loop escalate costs from $127/week to $47,000 over four weeks. Circuit breakers would have caught this in the first hour. Always set hard limits on both error rates and spending.

Timeout Management

Three levels of timeout to control:

Per-Tool Each tool call has a timeout (10-30s for APIs, 60s for heavy operations)
Per-Task Total time for one agent task (5-15 minutes typical)
Per-Session Total budget for a user session (30-60 minutes, then checkpoint and pause)

Logging and Observability

You can’t fix what you can’t see. Every production agent needs structured logging that captures the reasoning chain, not just the inputs and outputs:

  • Every LLM call: model, input tokens, output tokens, latency, cost
  • Every tool call: name, parameters, result status, duration
  • Every decision point: why the agent chose this tool, what alternatives were considered
  • Every error: type, message, recovery action taken
  • Session totals: steps, tokens, cost, success/failure

The observability landscape in 2026 has converged around OpenTelemetry as the collection standard, with specialized platforms for analysis:

LangSmith
Deep LangGraph integration with near-zero overhead. Best if you’re using LangChain/LangGraph. Traces reasoning chains, tool calls, and agent state transitions.

Open-source (MIT, 19K+ GitHub stars), self-hostable, with OpenTelemetry support. Framework-agnostic. Best for teams that want full control of their observability data.

Braintrust
Framework-agnostic with strong CI/CD integration. Focus on evaluation + observability in one platform. Best for teams running agent evals in CI pipelines.

Dead Letter Queues

What happens to tasks that repeatedly fail? Don’t just drop them — save them for later analysis:

class DeadLetterQueue:
    """Store tasks that repeatedly fail for later review."""

    def __init__(self, file="dead_letters.jsonl"):
        self.file = file

    def add(self, task, error, attempts, context=None):
        entry = {
            "timestamp": datetime.now().isoformat(),
            "task": task,
            "error": str(error),
            "attempts": attempts,
            "context": context,
        }
        with open(self.file, "a") as f:
            f.write(json.dumps(entry) + "\n")
        print(f"  [DLQ] Task moved to dead letter queue after "
              f"{attempts} failed attempts")

Your Turn

Add Comprehensive Error Handling to Your Agent

Take your Chapter 4 research agent and make it production-robust:

  1. Add retry with backoff for all tool calls — 3 retries with exponential backoff for transient errors
  2. Add a fallback chain for web search — if the primary search fails, try an alternative, then return a helpful error
  3. Add output guardrails — validate that search results contain actual URLs, that read_page returns meaningful content (not just error pages)
  4. Add a circuit breaker — trip after 3 errors in 5 operations, with a 30-second cooldown

  6. Add a dead letter queue — save failed tasks for review

Test by intentionally breaking things: disconnect your internet mid-task, feed the agent a URL that returns 404, give it a question that requires a tool you haven’t defined. Does it recover gracefully?
