Chapter 7

Multi-Agent Orchestration

When one agent isn’t enough — patterns for agents that work together

When One Agent Isn’t Enough

Your Chapter 4 agent is great at research. But what if you need it to also analyze data, write code, create a presentation, and review its own work? You could give one agent 30 tools and a 2,000-word system prompt, but that’s like hiring one person to be the researcher, writer, designer, and QA tester. It’s possible, but the quality suffers.

Multi-agent orchestration solves this with specialization: multiple agents, each with a focused role and a small set of tools, coordinated by an orchestration pattern. It’s the difference between a freelancer doing everything and a team where each person excels at their part.

Multi-agent orchestration is like a software team

You have a tech lead (supervisor agent) who breaks down the project, specialists (frontend, backend, QA agents) who do the work, and they coordinate through standups and shared docs (message passing and shared state). Nobody does everything — everyone does their thing well.

The Four Orchestration Patterns

Almost every multi-agent system uses one of these four patterns (or a combination):

Supervisor One manager delegates to specialists
Pipeline Agents in a chain, each processing in order
Swarm Peers collaborate, routing work dynamically

Pattern 1: The Supervisor

The most common pattern. One “manager” agent receives the task, breaks it into subtasks, delegates each to a specialist, and synthesizes the results.

The Supervisor Architecture

User Task → Supervisor → Specialist A
Supervisor → Specialist B → Supervisor → Final Result

The supervisor is typically a reasoning-heavy model (like Claude Opus or GPT-4) that plans and coordinates. Specialists can be cheaper, faster models (Claude Haiku, GPT-4o-mini) since they have narrow, well-defined tasks. This heterogeneous model selection can cut costs by up to 90% compared to using the expensive model for everything.

Here’s a supervisor implementation using the “agent as tool” pattern — each specialist agent is wrapped as a tool the supervisor can call:

import anthropic
import json

client = anthropic.Anthropic()

def run_specialist(name: str, system_prompt: str, task: str,
                   model: str = "claude-haiku-4-5-20251001") -> str:
    """Run a specialist agent and return its output."""
    response = client.messages.create(
        model=model,
        max_tokens=2048,
        system=system_prompt,
        messages=[{"role": "user", "content": task}],
    )
    return response.content[0].text


# Define specialist agents as tools for the supervisor
supervisor_tools = [

SPECIALISTS = {

    "writer": "You are a writing specialist. Given research findings, "

}

SUPERVISOR_PROMPT = """You are a project supervisor. Your job is to coordinate
specialists to complete complex tasks. You have three specialists available:

- researcher: gathers information on a topic
- writer: creates structured content from research
- reviewer: checks quality and accuracy

PROCESS:
1. Break the task into subtasks for each specialist
2. Start with research, then writing, then review
3. If the reviewer finds issues, send the content back to the writer
4. Synthesize the final result and present it to the user

Be efficient — only involve specialists when their expertise is needed."""

The supervisor calls specialists as tools. Each specialist runs in its own context with its own system prompt. The supervisor sees only the result, not the specialist’s internal reasoning.

Running the Supervisor System

def run_supervisor(task: str, max_steps: int = 10) -> str:
    """Run the supervisor with its specialist team."""
    messages = [{"role": "user", "content": task}]

    for step in range(1, max_steps + 1):
        response = client.messages.create(
            model="claude-sonnet-4-6-20250514",  # Smart model for supervisor
            max_tokens=4096,
            system=SUPERVISOR_PROMPT,
            tools=supervisor_tools,
            messages=messages,
        )

        if response.stop_reason == "tool_use":
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    specialist_name = block.name
                    specialist_task = block.input["task"]
                    print(f"  [{step}] Supervisor → {specialist_name}: "
                          f"{specialist_task[:80]}...")

                    # Run the specialist agent
                    result = run_specialist(
                        specialist_name,
                        SPECIALISTS[specialist_name],
                        specialist_task,
                    )
                    print(f"  [{step}] {specialist_name} → Supervisor: "
                          f"{result[:80]}...")

                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result,
                    })

            messages.append({"role": "assistant", "content": response.content})
            messages.append({"role": "user", "content": tool_results})
        else:
            print(f"\n  [Done in {step} steps]")
            return response.content[0].text

    return "Supervisor hit step limit."

# Run it
result = run_supervisor(
    "Write a comparison of Rust vs Go for CLI tools in 2026. "
    "Research both, write a structured analysis, and review for accuracy."
)
print(result)

Pattern 2: The Pipeline

Agents in a chain, each processing and passing results to the next. Think of it like an assembly line.

Input → Agent 1: Extract → Agent 2: Analyze → Agent 3: Format → Output 
def run_pipeline(input_data: str, stages: list) -> str:
    """Run agents in a pipeline — each stage's output feeds the next."""
    current_data = input_data

    for i, stage in enumerate(stages, 1):
        print(f"  Pipeline stage {i}/{len(stages)}: {stage['name']}")
        current_data = run_specialist(
            name=stage["name"],
            system_prompt=stage["system_prompt"],
            task=f"{stage['instruction']}\n\nInput:\n{current_data}",
        )
        print(f"  → Output: {current_data[:100]}...")

    return current_data

# Example: content creation pipeline
pipeline = [
    {
        "name": "outliner",
        "system_prompt": "You create detailed outlines for articles.",
        "instruction": "Create a detailed outline for an article about:"
    },
    {
        "name": "writer",
        "system_prompt": "You write engaging articles from outlines.",
        "instruction": "Write a full article based on this outline:"
    },
    {
        "name": "editor",
        "system_prompt": "You edit articles for clarity and style.",
        "instruction": "Edit this article for clarity, grammar, and flow:"
    }
]

result = run_pipeline("The future of remote work in 2026", pipeline)

When to use: Tasks with clear, sequential stages where each stage transforms the output. Content creation, data processing, ETL pipelines.

Limitation: No backtracking. If stage 2 finds a problem with stage 1’s output, it can’t send it back. For tasks that need iteration, use the supervisor pattern instead.

Pattern 3: The Swarm

In a swarm, agents are peers that route work to each other dynamically. There’s no central supervisor — each agent decides whether to handle a task itself or hand it off to a more appropriate peer.

The Handoff Pattern (OpenAI Agents SDK Style)

Problem: You have multiple specialized agents and need requests to flow to the right one automatically.

Solution: Each agent has a list of agents it can “hand off” to. When an agent determines the task is outside its expertise, it transfers the conversation to a more suitable peer. The handoff is represented as a tool call.

When to use: Customer service (billing agent, tech support agent, sales agent), multi-domain Q&A, any system where the right specialist depends on the user’s actual question.

# Swarm-style handoffs — each agent can route to peers
def create_swarm_agent(name, system_prompt, peers):
    """Create an agent that can hand off to peers."""
    handoff_tools = [
        {
            "name": f"transfer_to_{peer}",
            "description": f"Transfer this conversation to the {peer} specialist. "
                           f"Use this when the task is better handled by {peer}.",
            "input_schema": {
                "type": "object",
                "properties": {
                    "reason": {
                        "type": "string",
                        "description": "Why you're transferring"
                    }
                },
                "required": ["reason"]
            }
        }
        for peer in peers
    ]

    return {
        "name": name,
        "system_prompt": system_prompt,
        "tools": handoff_tools,
    }

# Define a customer service swarm
billing = create_swarm_agent(
    "billing", "You handle billing questions, refunds, and payment issues.",
    peers=["tech_support", "sales"]
)
tech_support = create_swarm_agent(
    "tech_support", "You handle technical problems, bugs, and setup issues.",
    peers=["billing", "sales"]
)
sales = create_swarm_agent(
    "sales", "You handle upgrade requests, pricing, and new subscriptions.",
    peers=["billing", "tech_support"]
)

agents = {"billing": billing, "tech_support": tech_support, "sales": sales}

def run_swarm(initial_agent: str, user_message: str):
    """Run a swarm — agents hand off to each other as needed."""
    current_agent = initial_agent
    messages = [{"role": "user", "content": user_message}]

    for step in range(10):
        agent = agents[current_agent]
        response = client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=1024,
            system=agent["system_prompt"],
            tools=agent["tools"],
            messages=messages,
        )

        # Check for handoff
        for block in response.content:
            if block.type == "tool_use" and block.name.startswith("transfer_to_"):
                next_agent = block.name.replace("transfer_to_", "")
                print(f"  {current_agent} → {next_agent}: {block.input['reason']}")
                current_agent = next_agent
                break
        else:
            # No handoff — agent handled the request
            return response.content[0].text

    return "Swarm hit routing limit."

Pattern 4: The Debate

Multiple agents propose different solutions. A judge agent evaluates them and picks the best one. This is useful when quality matters more than speed.

def run_debate(question: str, n_debaters: int = 3) -> str:
    """Multiple agents propose answers, a judge picks the best."""
    proposals = []
    for i in range(n_debaters):
        proposal = run_specialist(
            f"debater_{i+1}",
            f"You are debater #{i+1}. Provide a unique, well-reasoned "
            f"answer. Think creatively — don't give the obvious answer.",
            question,
        )
        proposals.append(proposal)
        print(f"  Debater {i+1}: {proposal[:100]}...")

    # Judge evaluates all proposals
    judge_input = "\n\n".join(
        f"## Proposal {i+1}\n{p}" for i, p in enumerate(proposals)
    )
    verdict = run_specialist(
        "judge",
        "You are a judge. Evaluate the proposals for accuracy, "
        "completeness, and reasoning quality. Pick the best one "
        "and explain why. Combine strengths from multiple proposals "
        "if appropriate.",
        f"Question: {question}\n\n{judge_input}",
        model="claude-sonnet-4-6-20250514",  # Smart model for judging
    )
    return verdict

When to use: Design decisions, complex analysis, any task where exploring multiple approaches yields better results. When to skip: Simple tasks, time-sensitive work (debate multiplies latency).

Agent Communication: How Agents Share Information

Agents need to exchange data. Three approaches:

Method How It Works Best For
Tool returns Specialist output flows back through tool results Supervisor pattern, simple handoffs
Shared state All agents read/write to a common state object or file Pipeline, collaborative editing
Message passing Agents send structured messages to each other via a queue Async work, distributed systems

For most projects, tool returns are enough. The supervisor calls a specialist, gets the result, and passes relevant parts to the next specialist. Shared state and message passing add complexity — use them only when agents genuinely need to work concurrently on shared data.

The Coordination Problem

Multi-agent systems introduce coordination challenges that don’t exist with single agents:

Duplicate work. Without coordination, two agents might research the same topic independently, wasting tokens and time. The supervisor pattern prevents this naturally (the supervisor assigns distinct tasks), but swarm patterns need explicit deduplication — a shared “already done” list that agents check before starting work.
Cost multiplication. Multi-agent systems multiply API costs. A 3-agent supervisor system uses at least 4x the tokens of a single agent (supervisor + 3 specialists). Strategies to manage this: use cheap models for specialists, keep specialist prompts short, limit the number of specialist calls, and cache results aggressively.

When Multi-Agent vs. Single Agent

Good use of multi-agent
“Review this PR for security, code style, and performance.” → Three specialist agents, each focused on one aspect, run in parallel.
Different expertise needed. Specialists can run concurrently. Each has a clear, narrow scope.
Overkill for multi-agent
“Summarize this article.” → No need for a researcher, writer, and reviewer. One agent does this fine.
Simple task, single expertise. Multi-agent adds cost and latency without benefit.
The decision rule: use multi-agent when the task requires different types of expertise or when subtasks can run in parallel. If one agent with the right tools can do it in a few steps, multi-agent is overhead, not help.

Real-World Example: Code Review System

Here’s how a production code review system might work with three specialist agents:

S
Security Agent
Scans for vulnerabilities: SQL injection, XSS, hardcoded secrets, insecure dependencies. Uses a security-focused system prompt and has access to vulnerability databases.
Q
Quality Agent
Checks code quality: naming conventions, code duplication, complexity metrics, design patterns. Reviews against the project’s style guide.
P
Performance Agent
Identifies performance issues: N+1 queries, missing indexes, unnecessary allocations, algorithmic complexity. Suggests optimizations.

A supervisor agent coordinates all three, merges their findings, resolves conflicts (e.g., when the security agent wants a change that hurts performance), and produces a unified review. This is exactly the system we’ll build in the Chapter 12 capstone project.

What’s Next

Multi-agent systems are powerful but fragile. When one agent in a three-agent pipeline fails, the whole system can stall. When a specialist returns garbage, the supervisor has to deal with it. These systems need robust error handling — and agent errors are fundamentally different from traditional software bugs.
