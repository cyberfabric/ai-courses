Chapter 8

Error Handling & Recovery

Memory Map
📉

The Big Idea

If each step has 85% success, a 10-step task succeeds only 20% of the time (0.8510). Robust error recovery isn’t a nice-to-have — it’s the difference between a demo and a product.

🚨 Six Categories of Agent Failure

Category Example Recovery
🔧 Tool Error API timeout, rate limit, 404 Retry with backoff, fallback tool
🌀 Hallucination Invents functions, fabricates data Output validation, self-correction
📝 Format Violation Text instead of JSON, malformed calls Retry with format reminder
♾️ Infinite Loop Repeats the same action endlessly Step limit, loop detection, circuit breaker
💸 Cost Overrun Token budget exhausted Budget controller, tiered model routing
🚫 Refusal Model declines a legitimate action Rephrase request, add context

Like distributed systems: retries, circuit breakers, fallbacks, and monitoring. The difference is your “service” makes non-deterministic decisions.

🛡️ Recovery Patterns

🔁
Retry with Backoff
Exponential delay + jitter. Handles transient errors (timeouts, rate limits). 3 attempts is the sweet spot.
💬
Retry with Feedback
Feed the error message back to the LLM: “Your previous attempt had this error…” Lightweight Reflexion for format/logic errors.
🔀
Fallback Chain
Try tool A → tool B → cache → escalate. Always have at least one fallback for critical tools that depend on external services.
🪞
Self-Correction
Agent detects its own mistakes (via tests, validation) and fixes them. The write → test → fix cycle. Multi-agent review outperforms single-agent.
⚡
Circuit Breaker
Trip after N errors in a window. Halt the agent with a cooldown. Prevents catastrophic cascading failures. $47K incident without one.
📬
Dead Letter Queue
Tasks that fail all retries get saved to JSONL for later review. Patterns in DLQ reveal systemic issues.

🛑 Three Guardrail Layers

  1. 🚪 Input Guardrails

    Validate the user’s request before the agent starts. Catch prompt injection, off-topic requests, blocked actions.

  2. ⚙️ Execution Guardrails

    Validate tool calls before executing them. Check parameters, verify permissions, block path traversal and dangerous commands.

  3. 📤 Output Guardrails

    Validate the agent’s final response before returning it. Check for PII, hallucinated URLs, format compliance.

Run guardrails in parallel with agent execution. Typical overhead: 50–200ms.

You Can’t Fix What You Can’t See

Log every LLM call (model, tokens, latency, cost), every tool call (name, params, result, duration), every decision point, every error. Converge on OpenTelemetry for collection; use LangSmith, Langfuse, or Braintrust for analysis.

📖 Key Terms

Exponential Backoff
Increasing wait time between retries (1s → 2s → 4s + jitter). Prevents hammering a failing service.
Fallback Chain
An ordered list of alternative strategies: try primary tool, then backup, then cache, then escalate to human.
Self-Correction
The agent detects its own mistakes (via tests or validation) and autonomously fixes them. Reflexion applied to error handling.
Guardrails
Validation layers (input, execution, output) that check agent behavior before it reaches the real world.
Circuit Breaker
Halts the agent when error rate exceeds a threshold in a sliding window. Resets after a cooldown period.
Pre-Flight Check
Verifying that conditions for success are met before executing a plan (required tools available, dependencies satisfied).
Dead Letter Queue
Storage for tasks that fail all retry attempts. Reviewed periodically to find systemic issues.

🧪 Quick Quiz

Why does 85% per-step accuracy become 20% for a 10-step task?

Probabilities multiply: 0.8510 = 0.20. Each step’s success rate compounds. Without error recovery, complex tasks almost always fail.

Why is “retry with feedback” better than simple retry for LLM errors?

LLMs tend to repeat the same mistake with the same input. Feeding the error back gives the model context to reason about what went wrong and try a different approach.

What are the three guardrail layers?

Input (validate user request before starting), Execution (validate tool calls before running), Output (validate response before returning). Run in parallel for minimal latency.

When should a circuit breaker trip?

When the error rate exceeds a threshold in a sliding window (e.g., 3 errors in last 5 operations). The agent halts for a cooldown period, preventing cascading failures and runaway costs.

What’s the value of a dead letter queue?

Saves failed tasks for later review instead of dropping them. Patterns in the DLQ reveal systemic issues: a consistently failing tool, a confusing question type, or a guardrail edge case.

📌 Key Takeaways

  • Agent errors are non-deterministic. The same input can succeed 9 times and fail on the 10th.
  • Six failure types: tool error, hallucination, format violation, infinite loop, cost overrun, refusal.
  • Retry with feedback > simple retry for LLM errors. Feed the error back so the model can reason about it.
  • Always have fallback chains for critical tools that depend on external services.
  • Three guardrail layers (input, execution, output) validate before the agent acts.
  • Circuit breakers prevent catastrophic failures. Set hard limits on error rates and spending.
  • Log everything. You can’t debug what you can’t see. Use OpenTelemetry + a specialized platform.
