Agent Frameworks & SDKs

Memory Map

The Big Idea

Agent frameworks are like web frameworks — you can build everything from scratch, but patterns, conventions, and ecosystems save time. The question is always: do the abstractions match your problem? Every framework runs the same tool-use loop you built in Chapter 4.

🏗️ The Six Frameworks

🟠
Claude Agent SDK
MCP-native, safety-first. Tools as in-process MCP servers, permission modes, sub-agent spawning. Best for: Anthropic stack, MCP ecosystems.
🟢
OpenAI Agents SDK
Minimal primitives: Agents, Handoffs, Guardrails. Provider-agnostic (100+ LLMs). Tracing by default. Best for: Quick starts, multi-provider setups.
🔵
LangGraph
Agents as state machines. Directed graphs with typed state, checkpointing (Postgres/SQLite/Redis), time-travel debugging. Best for: Complex stateful workflows.
🟣
CrewAI
Role-based multi-agent. Define agents with roles, goals, backstories. Crews (autonomous) + Flows (deterministic). Best for: Team-style collaboration.
🔷
AG2 / Microsoft
Conversational multi-agent. AutoGen successor merged with Semantic Kernel for Azure enterprise. Best for: .NET + Python, Azure stack.
🔴
Google ADK
Code-first Python, v1.0 stable. Built-in workflow agents, eval tools, web UI. A2A protocol support. Best for: Google Cloud, multimodal agents.

🤔 Framework vs No Framework

  1. ✅ Use a Framework When…

    Multi-agent orchestration, persistence/checkpointing, built-in observability, team projects with multiple developers.

  2. ❌ Skip the Framework When…

    Single agent with <5 tools, one-off projects, learning how agents work (build from scratch first!).

  3. ⚠️ Watch for the Abstraction Trap

    If you can’t see raw tool calls, full conversation history, and model reasoning — the framework will cost more time than it saves.

🧪 Quick Quiz

What do all six frameworks have in common?

They all run the same tool-use loop underneath (call LLM → check tool calls → execute → return results → repeat). They all support MCP for tool integration. The differences are in orchestration patterns, state management, and developer experience.

Which framework is best for complex stateful workflows?

LangGraph. It models agents as state machines with typed state schemas, built-in checkpointing (Postgres/SQLite/Redis), and time-travel debugging via LangSmith. Highest learning curve, but most control over flow.

When should you NOT use a framework?

Single agent with <5 tools, one-off projects, or when you’re learning how agents work. Build from scratch first to understand the foundation. Framework value emerges with multi-agent, persistence, observability, or team collaboration needs.

What is the “abstraction trap”?

Frameworks that hide too much, making debugging impossible. If you can’t inspect raw tool calls, conversation history, and model reasoning when something fails, the framework costs more time than it saves.

How do you minimize framework lock-in?

Keep tools as plain Python functions (wrap for framework). Store prompts in config files. Use MCP for tool distribution. Use standard formats for persistence. Ask: could I reconstruct this agent with my Chapter 4 loop?

📌 Key Takeaways

  • Six production frameworks in 2026. All run the same tool-use loop underneath. All support MCP.
  • Simplest start → OpenAI Agents SDK. Complex state → LangGraph. Role-based teams → CrewAI. Anthropic stack → Claude Agent SDK.
  • Use a framework for multi-agent, persistence, observability, or team projects. Skip it for simple single-agent tasks.
  • Beware the abstraction trap: always verify you can inspect raw tool calls and conversation history.
  • Minimize lock-in: plain Python tools, config-file prompts, MCP distribution, standard state formats.
  • Frameworks are getting thinner as MCP standardizes tools and A2A standardizes communication.
  • If unsure, start with OpenAI Agents SDK (lowest friction) or CrewAI (most intuitive). Migrate later if needed.
