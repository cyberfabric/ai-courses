Chapter 9

Agent Evaluation & Testing

Memory Map
📏

The Big Idea

Agents give different answers every time — traditional pass/fail testing breaks down. Change what you measure: not “did it return X?” but “was the result good, efficient, and cost-effective?” Evaluate agents like employees — quality, efficiency, process, and cost.

📊 What to Measure

✅
Completion
Did the agent finish without human intervention? Track successful_tasks / total_tasks.
🎯
Accuracy
Was the result correct and complete? Score 0–1 via keyword checks, LLM-as-judge, or human rating.
⚡
Efficiency
How many steps and tokens? Fewer steps = lower cost, lower latency. Count tool_use blocks.
💰
Cost per Success
total_spend / successful_tasks. More revealing than cost per task — accounts for failed attempts.
⏱️
Latency
Wall-clock time per task. Matters for user-facing agents. End-to-end, not just per-step.

🔍 Evaluation Methods

Method What It Tests Best For
🧪 End-to-End Does the agent produce a good result? Integration testing, overall quality
👣 Golden Trajectory Did it take the right steps? Tool selection, efficiency analysis
⚖️ LLM-as-Judge Quality beyond keyword matching Accuracy, completeness, clarity scoring
🔄 A/B Testing Which version is better? Model selection, prompt comparison
📉 Regression Did changes break anything? Continuous improvement safety net

→
🤖 Run agent
→
🔀 Compare trajectories
→
📊 Score alignment

Not exact-match — check: were required tools used? Were unnecessary calls avoided? Was the step count reasonable?

🏗️ Evaluation-Driven Development (EDD)

  1. Define Test Cases (10–20)

    Representative tasks your agent should handle. Mix: 40% happy path, 40% edge cases, 20% adversarial.

  2. Specify Expected Outcomes

    Per case: keywords, required tools, max steps, quality criteria. Use thresholds, not binary pass/fail.

  3. Set Target Score

    e.g., “pass 80% of cases.” Run each case 3–5 times, use median score (agents are non-deterministic).

  4. Build → Run → Iterate

    Build the agent, run the eval, improve until target is hit. Every change must not regress the eval score.

📖 Key Terms

End-to-End Evaluation
Running the agent on a full test suite and measuring task outcomes (completion, accuracy, cost).
Golden Trajectory
An ideal sequence of tool calls for a task, used as a reference to evaluate agent efficiency and tool selection.
LLM-as-Judge
Using an LLM to score agent output quality on criteria like accuracy, completeness, and clarity. Used by 53% of organizations.
Evaluation-Driven Development (EDD)
Writing evals before building the agent, like TDD for agents. Uses thresholds instead of binary pass/fail.
Cost per Success
total_spend / successful_tasks. Accounts for failed attempts, more revealing than raw cost per task.
Regression Testing
Running the full eval suite after every change to ensure improvements don’t break existing capabilities.
A/B Testing
Comparing two agent versions head-to-head on the same tasks with multiple runs per case.

🧪 Quick Quiz

Why can’t you use traditional pass/fail tests for agents?

Agents are non-deterministic — the same input can produce different (but equally valid) answers each time. You need thresholds and scoring (e.g., “passes 80% of runs”) instead of binary pass/fail.

Why is “cost per success” better than “cost per task”?

If an agent costs $0.05/run but fails 50% of the time, the real cost is $0.10 per successful result. Cost per success accounts for wasted attempts.

What are the three categories of eval test cases?

Happy path (40%) — straightforward tasks. Edge cases (40%) — tricky/ambiguous inputs. Adversarial (20%) — designed to confuse (contradictions, prompt injection, impossible tasks).

What are LLM-as-judge biases to watch for?

LLM judges prefer longer answers, formal language, and outputs similar to training data. Mitigate with: specific numerical rubrics, reference answers, rotating judge models, and calibrating against human ratings.

How many times should you run each eval case?

3–5 times, using the median score. A single run isn’t reliable because agents are non-deterministic. One perfect run might just be lucky.

📌 Key Takeaways

  • Evaluate agents on five dimensions: completion, accuracy, efficiency, cost, and latency.
  • Cost per success (> cost per task) is the metric that reveals true economics.
  • Golden trajectories evaluate the process; LLM-as-judge evaluates the output. Use both.
  • Write evals before building (EDD). Mix: 40% happy path, 40% edge cases, 20% adversarial.
  • Run each case 3–5 times, use medians. Non-determinism means one run proves nothing.
  • Regression test after every change. Set a tolerance threshold (e.g., 5%).
  • Don’t debate model choices — A/B test them on your eval suite and let the numbers decide.
