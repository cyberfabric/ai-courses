Tool Design for Agents

Memory Map
💡

The Big Idea

Tool design matters more than model selection. A great model with bad tools fails. A decent model with great tools succeeds. The interface IS the tool — the LLM never sees your code, only the name, description, and schema.

The LLM sees the first three. It never sees your implementation. Everything the model needs must live in the interface.

✨ The Four Golden Rules

📛
1. Names = Verbs
✅ search_web, read_file, create_issue
❌ web_tool, utils, do_stuff
📝
2. Descriptions = Docs
Answer: What does it do? When to use? What it returns? Usage tips? Include examples in descriptions.
📐
3. Typed Parameters
Types, descriptions, and constraints for every param. Use enum for fixed value sets. Include example values.
📊
4. Structured Returns
Return JSON for data, natural language for content. The LLM parses structured data reliably; it struggles with prose.

📋 The Description Template

  1. What It Does

    One sentence: “Execute a SQL query against the application database.”

  2. When to Use It

    The use case: “Use when you need to look up data or verify database state.”

  3. What It Returns

    Format and fields: “Returns JSON with rows (array) and row_count (integer).”

  4. Usage Tips

    Edge cases, examples, related tools: “Only SELECT — for writes, use write_record.”

🚨 Error Handling Pattern

⚠️ Status
+
💬 Message
+
💡 Suggestion

Status lets the agent branch. Message explains what happened. Suggestion tells the agent how to recover. Never return raw stack traces — they’re useless to an LLM.

⚖️ The Goldilocks Problem: How Many Tools?

Granularity Problem Example
🔸 Too Fine 4 tool calls for one logical action; wasted tokens & errors open_file + read_line + close_file just to read
🔸 Too Coarse “God tool” — agent guesses operations & params file_operation(operation="search", ...)
✅ Just Right One tool per meaningful action; clear & composable read_file, write_file, search_files
🎯

The Sweet Spot: 5–15 Tools Per Agent

Models pick accurately with 5–10 tools. At 50+, they hallucinate parameters and choose wrong tools. Need more? Use the router pattern to split across specialist agents.

⚠️ Handling Dangerous Tools

✋
Confirmation Params
Add confirm: boolean — false = dry run, true = actually execute. Force the agent to be explicit.
👁️
Read-Only Default
Give read tools by default. Only add write tools when the task actually requires changes.
🏷️
MCP Annotations
destructiveHint: true flags irreversible tools. Frameworks auto-require human approval for these.

🧩 Composition Over Complexity

🔍 search_web
→
📖 read_page
→
📊 extract_data
→
💾 save_to_file

Build simple, testable tools. The agent figures out the chaining — that’s what LLMs are good at. Don’t build one massive tool that does everything.

📖 Key Terms

JSON Schema
The standard format for defining tool parameters — types, descriptions, required fields, enums, constraints.
Tool Granularity
How specific each tool is. Sweet spot: one tool per meaningful action, 5–15 tools per agent.
God Tool
Anti-pattern: one tool that tries to do everything via operation parameters. Confuses the LLM.
Confirmation Parameter
A boolean confirm param that forces dry-run by default for dangerous operations.
Tool Annotations
MCP metadata (readOnlyHint, destructiveHint, etc.) that describe a tool’s safety characteristics.
FastMCP
Python helper that generates MCP tool schemas from type hints and docstrings. Define once, use everywhere.
Tool Composition
Building complex workflows from simple, independent tools. The LLM handles the chaining.

🎭 The Core Analogy

👨‍💻

Designing an API for a Junior Dev Who Can Only Read Docs

No Stack Overflow. No asking colleagues. Just your documentation. If the docs are vague, they’ll misuse the API. If errors are cryptic, they’ll get stuck. That’s how an LLM experiences your tools.

🧪 Quick Quiz

Why does tool design matter more than model selection?

Because tools are the agent’s hands. The LLM can only generate text without them. Poorly designed tools cause the “brain” to make bad decisions about what to do and how, regardless of model quality.

What four things should a tool description answer?

What does it do? When should I use it? What does it return? Any tips for using it well?

What’s the ideal number of tools per agent?

5–15. Models pick accurately with 5–10. At 50+, they hallucinate and choose wrong tools. Use a router to split large tool sets across specialist agents.

How should tool errors be structured?

Status + Message + Suggestion. Status for branching, message for what happened, suggestion for how to recover. Never return raw stack traces.

What is the “description test” for tool quality?

Show the tool definition to a colleague with no other context. If they can’t use it correctly from the description alone, neither can the LLM.

📌 Key Takeaways

  • Tools > models. The interface IS the tool — the LLM never sees your implementation.
  • Four golden rules: verb names, descriptions-as-docs, typed parameters with constraints, structured returns.
  • Error pattern: status + message + suggestion. Never return stack traces.
  • Goldilocks granularity: 5–15 task-oriented tools per agent. Too fine or too coarse both fail.
  • Dangerous tools need: confirmation params, read-only defaults, or MCP destructiveHint annotations.
  • Compose simple tools — let the agent figure out the chaining.
  • Test tools independently before giving them to an agent. The “description test”: if a human can’t use it from docs alone, neither can the LLM.
