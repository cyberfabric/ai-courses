🤖 The Four Agent Roles

🎯
Orchestrator
Parses the PR, creates a review plan, delegates to specialists, merges findings, deduplicates, ranks by severity. Uses Sonnet for reasoning.
🔒
Security Agent
Scans for SQL injection, XSS, hardcoded secrets, auth bypasses, insecure deps. Reports severity + file + line + fix suggestion.
✨
Quality Agent
Checks naming, code duplication, complexity, error handling, design patterns. Focuses on substantive issues, not style preferences.
🔀
Merger Agent
Combines all specialist findings, removes duplicates, groups by file, ranks by severity, produces final verdict: APPROVE / REQUEST_CHANGES / NEEDS_DISCUSSION.

⏱️ Review Pipeline Step-by-Step

  1. 📄 Get PR Diff

    Fetch the diff and list of changed files using get_pr_diff and list_files tools. Truncate diffs >15K chars.

  2. 🔒 Security Review

    Security Agent (Haiku) scans for vulnerabilities. Each finding: severity, file, line, issue, fix.

  3. ✨ Quality Review

    Quality Agent (Haiku) checks code quality. Can run in parallel with Security for ~40% latency savings.

  4. 🔀 Merge & Rank

    Orchestrator (Sonnet) deduplicates, groups by file, ranks by severity, produces verdict.

  5. 👤 Human Approval

    Reviewer sees findings. Options: approve as-is, edit, discard specific findings, or reject entirely.

🧩 Key Patterns Applied

Supervisor Pattern (Ch.7)
The orchestrator delegates to specialists and merges their results — the core multi-agent coordination pattern.
Partial Results (Ch.8)
If one specialist fails, present results from the others with a note. A security-only review is better than no review.
Human Gate (Ch.6)
Four options: approve, edit, discard findings, or reject. Builds developer trust in AI reviews.
Tiered Model Routing (Ch.11)
Sonnet for orchestration/merging (needs judgment), Haiku for specialists (needs speed, cost efficiency).
Focused System Prompts (Ch.4)
Each specialist has a narrow prompt ending with “don’t invent problems” — precision over recall.
Eval-Driven Quality (Ch.9)
Test suite checks: known vulns are caught, clean code gets APPROVE, style nits aren’t flagged.

🏛️ The Agent Builder’s Seven Principles

1️⃣
Start Simple
Add complexity only when the simple version fails.
2️⃣
Tools > Models
Invest in tool design, not model selection.
3️⃣
Prompt = Architecture
A clear system prompt turns a generic LLM into a focused specialist.
4️⃣
Escape Hatch
Humans must always be able to intervene, override, and shut down.
5️⃣
Measure Everything
Cost, accuracy, latency. You can’t improve what you don’t measure.
6️⃣
Errors Are Normal
Design for recovery, not perfection. 85% per step is realistic.
7️⃣
Loop First
Understand while True before reaching for frameworks.

⚠️ Hard-Won Lessons

🔍
Diff Context Isn’t Enough
Give specialists read_file access to check surrounding code. Costs more tokens but slashes false positives.
🎯
Precision > Recall
A trusted reviewer catching 70% of issues beats a noisy one catching 95% with 50% false positives. Aim for >80% precision.
🔀
Merger Is Critical
Without deduplication and ranking, the review is noise. The merger agent turns a list into actionable feedback.
🚶
Start With One Specialist
Get security review working and trusted before adding quality, performance, etc.

🧪 Quick Quiz

Why does the orchestrator use Sonnet while specialists use Haiku?

The orchestrator needs reasoning and judgment (planning, merging, deduplication, ranking). Specialists do focused, narrow analysis where Haiku’s speed and low cost win. This tiered routing saves 60–90% on costs.

What is the “partial results” pattern and why does it matter?

If one specialist fails (e.g., quality agent errors out), the system still presents results from the others (security findings) with a note about what’s missing. Partial output is better than no output in multi-agent systems.

Why do specialist prompts end with “don’t invent problems”?

Without this guardrail, LLMs produce excessive false positives — flagging every line as a potential issue. In code review, precision matters more than recall. Developers will ignore a noisy reviewer.

What’s the key difference between a 200-line and 500-line PR for agent review?

Under 200 lines, agents catch most real issues. Over 500 lines, review quality drops sharply — too much context for the agent to process effectively. Smaller PRs get dramatically better reviews.

📌 Key Takeaways

  • The capstone builds a multi-agent code review system using every concept from chapters 1–11.
  • Four roles: Orchestrator (plan & merge), Security Agent, Quality Agent, Merger — coordinated via supervisor pattern.
  • Tiered model routing: Sonnet for orchestration, Haiku for specialists. Cost: $0.05–0.15 per review.
  • Partial results pattern: if one agent fails, still present the others’ findings.
  • Human gate with four options (approve/edit/discard/reject) builds trust in AI reviews.
  • Precision > recall: aim for >80% of flagged issues being real. “Don’t invent problems” in prompts.
  • Sweet spot: PRs under 200 lines. Quality drops sharply above 500 lines.
  • The seven principles: start simple, tools > models, prompt = architecture, escape hatch, measure everything, errors are normal, loop first.
