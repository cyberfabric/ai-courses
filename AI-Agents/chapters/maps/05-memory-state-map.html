Memory & State Management

Memory Map

The Big Idea

LLMs are stateless — memory is infrastructure you build, not something you get for free. The conversation history is not enough. Context quality matters more than context quantity.

🏗️ The Three Memory Layers

Think of it like a developer’s workflow: the task in your head (working), the Slack thread with your team (short-term), and the wiki/docs you check next week (long-term).

📦 Memory Types Unpacked

📋
Working Memory
Python variables and scratchpads used during a single run. The AgentState class tracks steps taken, tokens used, findings, and errors. Cheap, reliable, gone when the task ends.
💬
Short-Term Memory
The messages list — every user message, assistant response, and tool result appended each step. Grows without bound. Suffers “lost in the middle” after ~32K tokens.
🗄️
Long-Term Memory
Persistent storage that survives across sessions. Two options: file-based (JSON/Markdown like Claude Code’s MEMORY.md) or vector stores (ChromaDB) for semantic search at scale.

🔧 Three Strategies to Manage Context

  1. ✂️ Truncate Tool Outputs

    Keep the start and end of long results, cut the middle. Use truncate_result() with a character limit. Agent still sees structure without the bloat.

  2. 🎭 Observation Masking

    Replace old tool results with short placeholders: “[Previously searched: Python frameworks — found 5 results]”. Agent remembers what it did without the full payload.

  3. 📝 Summarize & Compress

    At 70–80% context capacity, ask a cheap model (Haiku) to summarize older messages. Keep last 4–6 tool interactions verbatim. Always preserve the original user request.

⚖️ Long-Term Memory: File vs Vector Store

Aspect 📄 File-Based (MEMORY.md) 🔍 Vector Store (ChromaDB)

📊 State Beyond Memory

📋
Structured State
Track goal, steps[] with status (done/in_progress/pending), and findings{}. The agent always knows where it is in a complex, multi-step task.
💾
Checkpointing
Serialize messages + state to JSON. Resume after crashes or rate limits. Warning: checkpoints may contain sensitive data — store securely.
⚙️
AgentState Class
Working memory in code: steps_taken, tools_called, total_tokens, is_over_budget(). Your safety net against runaway agents.
⚠️

When NOT to Use Memory

Start stateless. Add memory only when the agent clearly needs it. Skip memory for: one-shot tasks, idempotent operations (can re-derive from input), security-sensitive contexts (memory persists = leak risk), and high-throughput systems (memory lookups add latency).

📖 Key Terms

Working Memory
Variables in code (AgentState) used during a single agent run. Gone when the task ends.
Short-Term Memory
The messages list — conversation history sent to the LLM on every API call.
Long-Term Memory
Persistent storage (files, vector stores) that survives across sessions and restarts.
Lost in the Middle
LLMs attend less to information in the middle of long contexts. After ~32K tokens, accuracy drops.
Observation Masking
Replacing old tool outputs with short placeholders to save context space while preserving metadata.
Checkpointing
Serializing full agent state (messages + working memory) to disk for save/resume capability.
MEMORY.md
File-based long-term memory pattern (used by Claude Code). Simple, versioned, surprisingly effective.

🧪 Quick Quiz

What are the three types of agent memory?

Working memory (scratchpad variables in code), short-term memory (the messages list / conversation history), and long-term memory (persistent storage like files or vector databases).

Why does “lost in the middle” matter for agents?

LLMs attend less to information placed in the middle of long contexts. After ~32K tokens, accuracy drops significantly. This is why you must truncate, mask, or compress — not just append endlessly.

When should you use file-based vs vector store memory?

Files for <100 memories, exact-topic lookup, prototypes, and git-friendly versioning. Vector stores for semantic search across hundreds+ entries, RAG-style retrieval, and production-scale agents.

What two tools give an agent persistent memory?

save_memory (persist a fact by topic) and recall_memory (retrieve by topic or list all). The agent decides what’s worth remembering.

Name three situations where you should NOT add memory.

One-shot tasks (nothing to remember), idempotent operations (can re-derive from input), and security-sensitive contexts (persisted memory = potential data leak). Also: high-throughput systems where memory lookups add unacceptable latency.

📌 Key Takeaways

  • LLMs are stateless. Memory is infrastructure you build: working, short-term, and long-term.
  • Working memory = Python variables (AgentState). Cheap, single-run, tracks steps/tokens/findings.
  • Short-term memory = the messages list. Grows without bound — manage it or accuracy drops.
  • Three context strategies: truncate outputs, mask old observations, summarize and compress.
  • Long-term memory: start with files (MEMORY.md pattern), graduate to vector stores when you need semantic search.
  • The Memory Tool Pattern: save_memory + recall_memory — the agent decides what to remember.
  • Start stateless. Add memory only when the agent clearly needs it.
