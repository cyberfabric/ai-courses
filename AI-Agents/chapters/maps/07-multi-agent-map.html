Chapter 7

Multi-Agent Orchestration

Memory Map
👥

The Big Idea

One agent with 30 tools is a freelancer doing everything — possible, but quality suffers. Multiple specialized agents, each with a focused role and small toolset, work like a software team: tech lead delegates, specialists excel, and they coordinate through shared protocols.

🏗️ The Four Orchestration Patterns

👔
Supervisor
One manager agent breaks the task down, delegates to specialists, and synthesizes results. The most common pattern. Specialists are wrapped as tools.
⛓️
Pipeline
Agents in a chain — each processes and passes output to the next. Like an assembly line. Simple but no backtracking.
🐝
Swarm
Peer agents hand off work dynamically. No central supervisor — each agent decides if it should handle the task or transfer to a peer.
⚖️
Debate
Multiple agents propose competing solutions. A judge agent evaluates and picks the best (or combines strengths). Quality over speed.

📊 Choosing the Right Pattern

Pattern Best For Limitation Cost
👔 Supervisor Complex tasks needing varied expertise Supervisor is a bottleneck 4x+ (supervisor + specialists)
⛓️ Pipeline Sequential stages (extract → analyze → format) No backtracking if stage fails N× (one per stage)
🐝 Swarm Dynamic routing (customer service, multi-domain Q&A) Harder to debug, risk of loops Varies by routing depth
⚖️ Debate Design decisions, complex analysis Multiplies latency (N proposals + judge) (N+1)× debaters + judge

🛠️ Building a Supervisor System

  1. Define Specialist Agents

    Each specialist has a focused system prompt and narrow expertise. Wrap each as a tool the supervisor can call (“agent as tool” pattern).

  2. Create the Supervisor

    A reasoning-heavy model that plans, delegates, and synthesizes. Sees specialist tools but not their internal reasoning.

  3. Implement the Loop

    Supervisor receives task → calls specialist tools → gets results → decides next step or produces final output.

  4. Handle Iteration

    If reviewer finds issues, supervisor sends content back to writer with feedback. The loop supports multiple rounds.

🎯

When to Go Multi-Agent

Use multi-agent when the task requires different types of expertise or when subtasks can run in parallel. If one agent with the right tools can do it in a few steps, multi-agent is overhead, not help.

⚠️ Coordination Challenges

👯
Duplicate Work
Without coordination, agents may research the same topic. Supervisors prevent this naturally; swarms need an “already done” list.
💸
Cost Multiplication
A 3-agent supervisor uses 4x+ tokens. Mitigate with cheap specialist models, short prompts, call limits, and aggressive caching.
🔀
State Conflicts
Concurrent agents writing to shared state can conflict. Prefer tool returns over shared state where possible.

📖 Key Terms

Supervisor Pattern
A manager agent delegates subtasks to specialists and synthesizes their results into a final output.
Pipeline Pattern
Sequential chain of agents where each stage’s output feeds the next. No backtracking.
Swarm / Handoff Pattern
Peer agents route work dynamically to each other. Each agent decides whether to handle a task or transfer it.
Debate Pattern
Multiple agents propose competing solutions; a judge evaluates and picks the best.
Agent as Tool
Wrapping an entire specialist agent as a tool callable by a supervisor. The core technique for the supervisor pattern.
Heterogeneous Model Selection
Using expensive models (Opus/Sonnet) for supervisors and cheap models (Haiku) for specialists to cut costs up to 90%.
Handoff
A tool call where one agent transfers the conversation to a more suitable peer agent (swarm pattern).

🧪 Quick Quiz

What are the four multi-agent orchestration patterns?

Supervisor (manager delegates to specialists), Pipeline (sequential chain), Swarm (peer handoffs), and Debate (competing proposals + judge).

What is the “agent as tool” pattern?

Wrapping a specialist agent as a tool that the supervisor can call. The supervisor sees only the specialist’s output, not its internal reasoning. This is how supervisor systems delegate work.

How does heterogeneous model selection save money?

Use an expensive reasoning model (Opus/Sonnet) for the supervisor that plans and coordinates, and cheap fast models (Haiku) for specialists with narrow tasks. Cuts costs up to 90%.

When should you NOT use multi-agent?

When one agent with the right tools can complete the task in a few steps. Multi-agent adds cost and latency. Only use it when the task requires different types of expertise or parallelizable subtasks.

What’s the main limitation of the pipeline pattern?

No backtracking. If stage 2 finds a problem with stage 1’s output, it can’t send it back. For tasks that need iteration, use the supervisor pattern instead.

📌 Key Takeaways

  • Multi-agent = specialization. Each agent has a focused role and small toolset, coordinated by an orchestration pattern.
  • Four patterns: Supervisor (most common), Pipeline (sequential), Swarm (dynamic handoffs), Debate (competing proposals).
  • The “agent as tool” pattern: wrap specialists as tools for the supervisor to call.
  • Use heterogeneous models: expensive supervisor, cheap specialists. Cuts costs up to 90%.
  • For communication, tool returns are usually enough. Add shared state or message passing only when needed.
  • Decision rule: go multi-agent when you need different expertise or parallelizable subtasks. Otherwise, keep it simple.
  • Watch for cost multiplication (4x+ tokens) and duplicate work. Cache aggressively, limit specialist calls.
