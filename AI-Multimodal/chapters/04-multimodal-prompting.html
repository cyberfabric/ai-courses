CHAPTER 4

Multimodal Prompting: Sending Images to LLMs

The practical chapter — API calls, prompt patterns, and the things that trip everyone up

Theory's Over. Let's Build.

For three chapters, we've been building foundations — what multimodal means, how encoders work, how fusion combines modalities. Now it's time to get your hands dirty.

This chapter is about the thing you'll do most often with multimodal AI: sending images to an LLM and getting useful answers back. Whether you're building an image analysis feature, a visual code reviewer, or a screenshot-to-bug-report tool, it all starts here.

The well-read friend analogy: Sending an image to an LLM is like showing a photo to a friend who has read every book, article, and manual ever written — but has never actually been outside. They can identify and describe almost everything, draw on vast knowledge to interpret what they see, and make impressive inferences. But sometimes they "fill in" details that aren't actually there, because their understanding comes from reading about the world, not from experiencing it firsthand.

Keep this analogy in mind. It'll help you understand both the impressive capabilities and the frustrating limitations.

How Images Get to the Model

Before we talk about prompting strategy, let's cover the plumbing. There are two ways to send an image to an LLM via API:

Base64 Encoding

Convert the image file into a text string (base64) and include it directly in the API request. The image travels inside the JSON payload.

Pros: Works everywhere, no hosting needed, the image is self-contained

Cons: Makes API requests larger (~33% bigger than the file), slower to transmit

URL Reference

Pass a URL pointing to the image. The model's server fetches the image itself.

Pros: Smaller API requests, faster for large images

Cons: Image must be publicly accessible (or the API must support signed URLs)

In practice, base64 is the most common approach — it works reliably and doesn't require hosting images anywhere. Let's see how it works across the three major APIs.

Sending Images: Claude vs GPT-4 vs Gemini

All three major multimodal APIs follow the same basic pattern: send a message that contains both text and image content. The syntax differs, but the concept is identical.

Claude (Anthropic)

import anthropic, base64

client = anthropic.Anthropic()

with open("screenshot.png", "rb") as f:
    image_data = base64.standard_b64encode(f.read()).decode("utf-8")

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": [
            {"type": "image", "source": {
                "type": "base64",
                "media_type": "image/png",
                "data": image_data}},
            {"type": "text", "text": "What's in this image?"}
        ]
    }]
)

GPT-4 (OpenAI)

from openai import OpenAI
import base64

client = OpenAI()

with open("screenshot.png", "rb") as f:
    image_data = base64.standard_b64encode(f.read()).decode("utf-8")

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{
        "role": "user",
        "content": [
            {"type": "image_url", "image_url": {
                "url": f"data:image/png;base64,{image_data}"}},
            {"type": "text", "text": "What's in this image?"}
        ]
    }]
)

Gemini (Google)

import google.generativeai as genai
from PIL import Image

genai.configure(api_key="YOUR_KEY")
model = genai.GenerativeModel("gemini-2.0-flash")

image = Image.open("screenshot.png")
response = model.generate_content([
    "What's in this image?",
    image
])
Notice the pattern: Every API uses the same concept — a message with mixed content types (text + image). The image is always alongside the text prompt, never separate. This is because the model uses cross-attention (Chapter 3) to relate your question to the image content.

What Vision Models Can Actually Do

You might be surprised by the range. Modern vision models don't just "describe pictures." They can:

Vision Model Capabilities

  • Describe — "What's in this image?" — detailed scene description
  • Analyze — "Is this UI accessible?" — quality and compliance assessment
  • Extract — "What does the receipt total say?" — read and extract text/data
  • Compare — "What changed between these two screenshots?" — diff analysis
  • Reason — "Why might this code crash based on the error dialog?" — inference and logic
  • Count — "How many people are in this photo?" — enumeration (with caveats)
  • Locate — "Where is the submit button?" — spatial reasoning (with caveats)
  • Generate code — "Write HTML/CSS that recreates this design" — from mockup to code

That's a lot of capabilities from a single API call. But capabilities alone don't guarantee good results — the prompt makes a huge difference.

Prompt Engineering for Images

The way you phrase your question dramatically affects the quality of vision responses. Here are the patterns that work:

Be Specific About What You Want

Vague prompt

Prompt: "Look at this image."

Result: A generic description. The model doesn't know what you need, so it describes everything at surface level.

Specific prompt

Prompt: "This is a screenshot of our app's checkout page. List every form field you can see, its current value (if any), and whether it appears to have a validation error."

Result: A focused, structured analysis of exactly what you need.

Give Context About What the Image Is

The model doesn't know if your image is a mockup, a production screenshot, a medical scan, or a cat photo. Tell it.

Context-rich prompt

Prompt: "This is an architecture diagram for our microservices system. Identify all services shown, their connections, and any potential single points of failure."

By saying "architecture diagram" and "microservices," you've activated the model's knowledge of software architecture, making the response far more relevant than a generic "I see boxes connected by arrows."

Request Structured Output

Structured output prompt

Prompt: "Extract the data from this chart and return it as a JSON object with keys: title, x_axis_label, y_axis_label, and data_points (array of {x, y} objects)."

When you specify the exact output format, the model is remarkably good at following it — even extracting numerical values from bar charts and line graphs.

The golden rule of multimodal prompting: Tell the model (1) what the image is, (2) what you want to know about it, and (3) what format you want the answer in. The more specific you are on all three, the better the result.

Multi-image comparison example

# Send two images for comparison (Claude API)
messages=[{
    "role": "user",
    "content": [
        {"type": "image", "source": {"type": "base64",
            "media_type": "image/png", "data": before_img}},
        {"type": "image", "source": {"type": "base64",
            "media_type": "image/png", "data": after_img}},
        {"type": "text", "text":
            "Image 1 is before the fix, Image 2 is after. "
            "What visual differences do you see?"}
    ]
}]
Watch the token cost. Each image uses tokens. Two high-resolution images can easily cost 2,000-3,000 tokens of input. For Claude, a 1080p image costs roughly 1,600 tokens. For GPT-4o, image cost depends on the detail level you choose (low/high). Always check your provider's pricing — multi-image requests can get expensive fast.

Image Resolution and Token Cost

Bigger images cost more tokens, take longer to process, and don't always improve results. Here's what you need to know:

~1,600
tokens for a 1080p image in Claude
85 / 1,105
tokens for GPT-4o low / high detail mode
258
tokens per image in Gemini Flash
  • Resize before sending — a 4K screenshot adds cost but rarely adds understanding. Resize to 1080p or even 720p for most tasks.
  • Crop to the relevant area — if you only care about a button in the corner, crop the image to just that region. Fewer tokens, better focus.
  • Use low-detail mode when it's enough — GPT-4o's "low" detail mode uses only 85 tokens per image. Perfect for "what kind of image is this?" but not for reading small text.
  • JPEG vs PNG — the file format doesn't matter for token count (it's based on pixel dimensions), but JPEG is smaller for upload, saving bandwidth.
Cost optimization trick: For batch processing (analyzing 100 screenshots), first send each image at low resolution with a simple question ("Does this screenshot contain an error dialog?"). Only send the ones that match to high-resolution analysis. This can cut your costs by 80%+.

The Hallucination Problem: What Models Get Wrong

Here's where the "well-read friend who's never been outside" analogy really clicks. Vision models have specific, predictable failure modes:

Surprisingly Good At

  • Describing scenes and objects
  • Reading printed text (even at angles)
  • Understanding charts and graphs
  • Analyzing UI layouts
  • Recognizing well-known logos and brands
  • Interpreting diagrams and flowcharts

Frustratingly Bad At

  • Counting objects precisely (especially > 10)
  • Reading very small or blurry text
  • Exact spatial reasoning ("is A left of B?")
  • Distinguishing very similar images
  • OCR on handwritten text (inconsistent)
  • Understanding perspective and 3D layout

The Biggest Danger: Confident Hallucinations

Vision models don't say "I'm not sure." They will confidently report text that doesn't exist in the image, count 7 objects when there are 5, or describe colors incorrectly. This isn't uncertainty — it's hallucination, and it looks exactly like a correct answer.

Hallucination trap

Prompt: "Read the serial number printed on this device."

What happens: If the serial number is too small or blurry to read, the model may invent a plausible-looking serial number instead of saying "I can't read it." It'll look completely real.

Hallucination-resistant prompt

Prompt: "Read the serial number on this device. If you cannot read it clearly, say 'UNREADABLE' instead of guessing. Accuracy is critical — do not infer or approximate."

Why this works: You've explicitly given the model permission to say "I don't know" and told it that guessing is worse than admitting uncertainty.

Never trust vision model output for safety-critical data without verification. Serial numbers, medical readings, financial figures, legal text — always validate extracted data against the source. Treat vision model output like OCR output: useful first pass, not ground truth.

Practical Patterns You'll Use All the Time

Pattern 1: Screenshot Analysis

The dev's best friend

Prompt: "This is a screenshot of our Android app. Identify: (1) any UI elements that look misaligned or overlapping, (2) any text that appears truncated, (3) whether the spacing and padding look consistent. Format as a numbered list."

Use this for QA, design reviews, and catching visual regressions. Faster than manually checking every screen.

Pattern 2: Chart Interpretation

Data extraction from visuals

Prompt: "This is a line chart from our analytics dashboard. Extract the approximate values for each data point and identify the overall trend. Return as JSON with 'data_points' and 'trend_summary' keys."

Surprisingly accurate for bar charts and line graphs. Less reliable for complex scatter plots or overlapping data series.

Pattern 3: Code from Screenshots

When someone sends you a screenshot of code instead of the actual code

Prompt: "Transcribe the code visible in this screenshot exactly as written, preserving indentation. Then explain what it does and identify any bugs."

Every developer's been there: someone pastes a screenshot of an error in Slack. Now you can paste it straight into Claude.

Building a Simple Image Analysis Tool

Let's put it all together. Here's a complete, runnable Python script that takes an image and answers questions about it:

→
Parse Response

Extract and display the answer

image_qa.py — a reusable image analysis function

import anthropic, base64, sys

def ask_about_image(image_path, question):
    client = anthropic.Anthropic()
    with open(image_path, "rb") as f:
        img = base64.standard_b64encode(f.read()).decode("utf-8")
    ext = image_path.split(".")[-1].replace("jpg", "jpeg")
    resp = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        messages=[{"role": "user", "content": [
            {"type": "image", "source": {"type": "base64",
                "media_type": f"image/{ext}", "data": img}},
            {"type": "text", "text": question}
        ]}])
    return resp.content[0].text

if __name__ == "__main__":
    print(ask_about_image(sys.argv[1], sys.argv[2]))

Usage: python image_qa.py screenshot.png "What errors do you see?"

That's a multimodal AI tool in 15 lines. You can wrap this in a web API, plug it into a Slack bot, or use it in a CI pipeline to analyze screenshots from automated tests.

Get Your Hands Dirty

Try It Now: The IDE Screenshot Challenge

Take a screenshot of your IDE right now — with some code visible. Upload it to Claude or ChatGPT and try these prompts in order:

  1. General: "What programming language is this? What does this code do?"
  2. Specific: "Are there any potential bugs or issues in this code?"
  3. Extraction: "Transcribe this code as text, preserving formatting."
  4. Structured: "List all function/method names visible in this screenshot as a JSON array."

Notice how the model handles each task differently. The general description is usually good. Bug detection is hit-or-miss (it may flag style issues but miss logic bugs). Transcription is surprisingly accurate for readable code. Structured extraction works well with clear output format instructions.

Bonus experiment: Try the same screenshot at different resolutions (resize to 50%, 25%). At what point does the model start struggling to read the code?

Structured Output from Images

One of the most powerful patterns is extracting structured data (JSON, tables, lists) from images. This is the bridge between "the AI can see the image" and "my code can use the data."

Extracting structured data from a UI screenshot

Prompt:

Analyze this mobile app screenshot and return JSON:
{
  "screen_name": "...",
  "navigation_elements": ["..."],
  "interactive_elements": [
    {"type": "button|input|toggle", "label": "...", "state": "..."}
  ],
  "visible_text": ["..."],
  "potential_issues": ["..."]
}

The model will fill in the JSON structure based on what it sees. You can then parse this in your code and use it programmatically — automated UI auditing, test generation, accessibility checks.

Pro tip: When extracting structured data, include an example of the expected output in your prompt. This "few-shot" approach dramatically improves consistency, especially for complex schemas. The model follows the pattern of your example.
