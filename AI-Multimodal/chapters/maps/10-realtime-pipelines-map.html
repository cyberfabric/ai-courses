Real-Time Multimodal Pipelines

Memory Map

The Big Idea

Real-time multimodal AI eliminates the text intermediary. Instead of chaining STT → LLM → TTS (7 steps, 1-2.5s), native APIs like GPT-4o Realtime process audio directly — audio in, audio out — in 3 steps and ~300-500ms. The gap between "tool you type at" and "colleague who talks with you" is closing fast.

📊 Where Latency Lives (Chained Pipeline)

  1. 🔇 Silence Detection: 300-800ms

    The hidden killer. Waiting to be sure the user stopped talking. VAD models can reduce this.

  2. 📡 Network Upload: 50-150ms

    Sending audio to the server. Geography matters — Sydney to Virginia = 200ms from light speed alone.

  3. 🔤 STT Processing: 200-500ms

    Whisper API. Deepgram cuts this to ~100ms for streaming.

  4. 🧠 LLM Time-to-First-Token: 200-600ms

    Varies by model and prompt length. Streaming helps — start TTS as soon as first sentence arrives.

  5. 🔊 TTS First Chunk: 100-300ms

    ElevenLabs ~300ms, OpenAI TTS slightly faster. Streaming is critical.

Total: 900ms-2.5s. Key optimization: pipeline parallelism — stream LLM tokens directly to TTS, don't wait for full response.

🧩 Core Concepts

🔌
WebSockets
Persistent bidirectional connection. Like a phone call vs sending letters (HTTP). Used by most real-time AI APIs. Best for server-side apps.
📹
WebRTC
Browser-native real-time media. Peer-to-peer, built-in echo cancellation & noise suppression. Best for browser apps. OpenAI Realtime supports both.
✋
Interruption Handling
User talks while AI speaks → detect speech, stop playback, cancel generation, capture new input. Native APIs handle this; custom pipelines need explicit logic.
🔁
Echo Cancellation
If the mic picks up AI's audio output, the system interrupts itself in a loop. WebRTC handles this; WebSocket apps need explicit filtering.
🛠️
Function Calling (Voice)
"What's the weather in Berlin?" → model calls get_weather mid-conversation → speaks the result naturally. Actions during live voice chat.
📷
Live Camera + AI
Camera frames (1-5 FPS) streamed alongside audio. AI sees what you see + hears what you say. Gemini Live does this natively. Screen sharing for devs.

🧰 Real-Time Toolkit

🔴 OpenAI Realtime API
Native audio-to-audio. WebSocket + WebRTC. Built-in VAD, interruption, function calling. Powers ChatGPT Voice Mode.
🟢 Gemini Live API
Native multimodal streaming: audio + video + screen sharing. "Pair programming with an AI that sees your screen."
🟣 LiveKit
Open-source real-time audio/video platform. "Agents" framework for AI voice assistants. Most mature production option.
🔵 Pipecat
Open-source pipeline framework by Daily. Plug in STT + LLM + TTS components. Handles VAD, interruption, turn-taking.
⚡ Deepgram
~100ms streaming STT. When Whisper is too slow for your chained pipeline. Also offers TTS.

🧪 Quick Quiz

Test Your Understanding

Why is the chained STT→LLM→TTS pipeline slow compared to native real-time?

7 sequential steps vs 3, each adding latency. Plus, voice emotion is lost when transcribed to text. Native APIs process audio directly (audio in → model → audio out), eliminating STT and TTS steps entirely. Result: ~300-500ms vs 1-2.5s.

What is "pipeline parallelism" and why does it help?

Instead of waiting for the full LLM response before starting TTS, you stream LLM tokens directly to the TTS engine. TTS starts synthesizing as soon as the first sentence arrives. This can cut perceived latency roughly in half.

What's the "silence detection" problem?

The system waits 500-800ms of silence to confirm the user has stopped speaking — before any AI processing even begins. This "hidden" latency is often the biggest single contributor. Voice Activity Detection (VAD) models can detect end-of-speech faster and more accurately.

Why is echo cancellation critical for voice AI?

If the microphone picks up the AI's audio output, the system thinks the user is speaking and interrupts itself — creating an infinite self-interruption loop. WebRTC handles this automatically; custom WebSocket implementations need explicit echo cancellation.

When should you use WebSockets vs WebRTC?

WebSockets for server-side apps (Python, backend, mobile). WebRTC for browser-based apps — it provides built-in echo cancellation, noise reduction, and codec negotiation that you'd otherwise build yourself.

Key Takeaways

  • Target: 200ms response time for natural conversation. Over 500ms feels slow. Over 1s feels broken.
  • Chained pipeline (STT→LLM→TTS) = 7 steps, 1-2.5s. Optimize with pipeline parallelism — stream LLM tokens to TTS.
  • Native real-time APIs (GPT-4o Realtime, Gemini Live) = 3 steps, 300-500ms. Audio in, audio out, no text intermediary. Preserves voice emotion.
  • Silence detection is the hidden latency killer (500-800ms). VAD models help. Echo cancellation prevents self-interruption loops.
  • Hybrid edge+cloud wins: on-device for wake word and quick feedback, cloud for complex reasoning.
  • Use LiveKit or Pipecat for production real-time AI. Don't build the plumbing from scratch.
