Chapter 5

Document Understanding & OCR 2.0

Memory Map
💡

The Big Idea

Traditional OCR reads characters. Multimodal AI understands documents. Send a document image to an LLM and get structured, meaningful data back — no regex, no templates, no custom parsers.

🧩 Core Concepts

🔤
Traditional OCR
Reads characters from images. Fast and cheap, but produces raw text with no understanding of meaning, layout, or relationships.
🧠
OCR 2.0 (Multimodal AI)
Sees layout AND understands meaning. Returns structured JSON directly. Works across layouts, languages, and document types.
🧾
Receipt / Invoice Extraction
One API call extracts merchant, items, prices, totals as structured data. Include a JSON example in your prompt as a schema definition.
📊
Chart Understanding
Vision models can read bar charts, line graphs, and pie charts. Tell the model to approximate values — don't let it hallucinate precision.
📋
Table Extraction
Convert visual tables to JSON arrays of objects. Provide example format with column headers as keys. Watch out for merged cells.
✍️
Handwriting & Diagrams
Neat handwriting ~85% accurate, messy ~60%, cursive ~40%. For diagrams, ask for structured output (Mermaid, PlantUML, step lists).
📄
Multi-Page PDFs
Three strategies: page-by-page, key page selection, or multi-image single request. Balance cross-page context vs token cost.
🔄
Hybrid Approach
Use fast/cheap OCR for high-volume simple text. Use multimodal LLMs for complex documents needing semantic understanding.

🎯 The Key Analogy

Traditional OCR is like hiring a typist — they can read your receipts but don't understand the numbers. Multimodal AI is like hiring an accountant — they read AND understand what everything means.

📑 Multi-Page Document Strategies

  1. Page-by-Page

    Convert each page to image, process independently. Simple but loses cross-page context.

  2. Key Page Selection

    Only send the pages that matter (first for summary, last for signatures). Saves tokens, focused results.

  3. Multi-Image Single Request

    Send multiple pages in one API call. Best cross-page context, but token costs add up. Good for 5–10 pages.

⚠️ Pitfalls That Will Bite You

🔢 Hallucinated Numbers
"3" becomes "8", decimals shift. Always validate: do line items add up to the subtotal?
📅 Date Format Ambiguity
"01/02/2024" — is that Jan 2 or Feb 1? Always specify ISO 8601 (YYYY-MM-DD) in your prompt.
🌫️ Overconfidence on Blurry Text
Models guess instead of saying "I can't read this." Add "use null for unclear values" to every prompt.
💰 Token Cost Surprises
A high-res A4 scan = 2,000+ tokens. At 1,000 docs/day, that's 2M tokens just for images.
🧩 Malformed JSON
Models occasionally return broken JSON or add markdown fences. Always wrap parsing in try/except.

✨ Prompting Best Practices

📐
Include a JSON Schema
Show the exact output shape you want. Field names, data types, nesting — the model follows the pattern.
📏
Specify Data Formats
Dates as YYYY-MM-DD, numbers not strings, null for missing fields. Remove ambiguity upfront.
🔍
Set Precision Expectations
For charts: "approximate to nearest whole number." For financial: "be precise with numbers."
🚫
Handle Uncertainty
"Use null instead of guessing for unclear text." Prevents the model from confidently hallucinating.

❓ Quick Quiz

What's the key difference between traditional OCR and multimodal document understanding?

OCR reads characters (raw text you must parse). Multimodal AI understands meaning and returns structured data — it knows "Total: $42.50" is the amount owed, not just a string.

Why should you include a JSON example in your document extraction prompt?

The model follows patterns. The example acts as a schema definition — it ensures consistent field names, data types, and nesting across all documents.

What DPI should you use when converting PDF pages to images?

150 DPI — readable text without enormous images. 300 DPI rarely improves results but doubles token cost. 72 DPI is too low for most text.

How do you prevent hallucinated precision when extracting chart data?

Tell the model to "approximate to the nearest whole number" or "round to one decimal place." This manages expectations and prevents fake precision.

When should you use traditional OCR instead of a multimodal LLM?

When you need high-volume (>millions/day), low-cost processing for simple text extraction. Traditional OCR is ~50× cheaper and much faster for straightforward character recognition.

✅ Production Checklist

  • Validate extracted numbers against internal consistency (line items vs totals)
  • Specify date formats explicitly (ISO 8601)
  • Include "use null for unclear values" in every extraction prompt
  • Resize images to 150 DPI before sending
  • Wrap JSON parsing in error handling (try/except)
  • Log raw model responses for debugging
  • Use hybrid approach: fast OCR for volume, LLMs for complexity

Key Takeaways

  • Multimodal AI doesn't just read documents — it understands them. One API call replaces regex + custom parsers.
  • Always include a JSON example in your prompt as a schema definition for consistent, structured output.
  • Receipts, invoices, charts, tables, diagrams, handwriting — all handled by the same vision + language approach.
  • Trust but verify: validate numbers, specify date formats, and handle malformed JSON gracefully.
  • Use the hybrid approach: fast/cheap OCR for high-volume simple extraction, multimodal LLMs for complex understanding.
  • 150 DPI is the sweet spot for PDF-to-image conversion — balances quality and token cost.
