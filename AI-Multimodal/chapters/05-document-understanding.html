CHAPTER 5

Document Understanding & OCR 2.0

When AI doesn't just read the letters — it understands the whole document

OCR Is Dead. Long Live OCR.

In the last chapter, you learned how to send images to LLMs and get useful answers back. Now let's apply that skill to one of the most immediately valuable use cases in production software: understanding documents.

For decades, OCR (Optical Character Recognition) was the go-to technology for extracting text from images. Scan a receipt, get the characters. Photograph a page, get the words. It worked. Sort of.

The problem? Traditional OCR is like a person who can read but doesn't understand. It sees "Total: $42.50" and gives you back the string "Total: $42.50." Great — but it has no idea that's the amount someone owes. It doesn't know it's at the bottom of a receipt. It can't tell you what was purchased or calculate whether the line items actually add up to that total.

The OCR 2.0 Paradigm

Traditional OCR reads characters. Multimodal AI understands documents. Instead of getting raw text that you then have to parse with regex and heuristics, you send the document image to an LLM and get structured, meaningful data back. The model sees layout, hierarchy, relationships between fields — just like a human would.

This isn't a marginal improvement. It's the difference between hiring a typist and hiring an accountant. Both can read your receipts — but only one understands what the numbers mean.

Traditional OCR vs. Multimodal Understanding

Let's make the difference concrete. Imagine you photograph a restaurant receipt:

Traditional OCR output
BELLA ITALIA Via Roma 15 2x Margherita 18.00 1x Tiramisu 8.50 1x Espresso 3.00 Subtotal 29.50 Tax (10%) 2.95 Total 32.45 Card ****1234

Raw text. You now need regex or custom parsing to extract merchant name, items, prices, and total. Good luck when the next receipt has a different layout.

Multimodal AI output
{ "merchant": "Bella Italia", "address": "Via Roma 15", "items": [ {"name": "Margherita", "qty": 2, "price": 18.00}, {"name": "Tiramisu", "qty": 1, "price": 8.50}, {"name": "Espresso", "qty": 1, "price": 3.00} ], "subtotal": 29.50, "tax": 2.95, "total": 32.45, "payment_method": "Card ending 1234" }

Structured JSON. Ready to use in your code. Works regardless of receipt layout because the model understands what each field means.

Notice the difference? The OCR approach gives you text you need to interpret. The multimodal approach gives you data you can use. And the multimodal version generalizes across layouts, languages, and formats without custom parsing rules.

Vision + Language = Document Intelligence

Document understanding is a perfect showcase for multimodal fusion (Chapter 3). The vision encoder sees the layout — headers, tables, columns, alignment, font sizes. The language model understands the meaning — "Total" is a sum, "Date" is a timestamp, a table has headers and rows. Together, they don't just read the document — they comprehend it. This is cross-attention at work: the language model attends to the visual layout to figure out which numbers belong to which labels.

Practical: Processing Receipts

Receipts are the "Hello World" of document understanding. They're everywhere, they're messy, and they're the first thing most teams try to automate. Here's how to do it properly:

import anthropic, base64, json

def extract_receipt(image_path):
    client = anthropic.Anthropic()
    with open(image_path, "rb") as f:
        img = base64.standard_b64encode(f.read()).decode("utf-8")
    ext = image_path.split(".")[-1].replace("jpg", "jpeg")

    resp = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        messages=[{"role": "user", "content": [
            {"type": "image", "source": {"type": "base64",
                "media_type": f"image/{ext}", "data": img}},
            {"type": "text", "text": """Extract receipt data as JSON:
{"merchant":"...", "date":"...", "items":[
  {"name":"...", "qty":1, "price":0.00}],
"subtotal":0.00, "tax":0.00, "total":0.00}
If a field isn't visible, use null."""}
        ]}])
    return json.loads(resp.content[0].text)

That's it. No regex. No layout templates. No training a custom model. One API call, and you get structured data from any receipt, in any language, with any layout.

Why include a JSON example in the prompt? The model follows patterns. By showing the exact shape you want (field names, data types, nesting), you get consistent output every time. Without the example, the model might use "store_name" one time and "merchant" the next. The example is your schema definition.

Invoices and Business Documents

Receipts are simple. Invoices are where document understanding really earns its keep. An invoice has headers, line items, tax calculations, payment terms, sender and recipient addresses — a complex, hierarchical structure.

Invoice extraction prompt
This is a scanned invoice. Extract the following as JSON: - sender (company name, address) - recipient (company name, address) - invoice_number, date, due_date - line_items (description, quantity, unit_price, total) - subtotal, tax_rate, tax_amount, grand_total - payment_terms, bank_details (if visible) Use null for any field not present. For amounts, use numbers (not strings). Dates in YYYY-MM-DD format.

Be explicit about data types and date formats. Without this, you'll get inconsistent formatting across different invoices — "March 5, 2024" vs "05/03/2024" vs "2024-03-05".

The model handles multi-column layouts, varying fonts, logos in the header, and even invoices in languages you don't speak. This is where multimodal AI completely outclasses traditional OCR — the understanding layer is language-agnostic because the model actually knows what an invoice is.

Verify financial data. Models can misread digits, especially in low-quality scans. A "3" can become an "8." A decimal point can shift. For any financial processing pipeline, add a validation step: do the line items add up to the subtotal? Does the tax calculation check out? Flag discrepancies for human review.

Charts and Graphs: Reading Visual Data

Here's something that feels like magic the first time you try it: send a chart image to a multimodal LLM and ask it to extract the data.

Bar charts, line graphs, pie charts — the model can read them all and convert the visual information back into numbers. It's not perfect (more on that in a moment), but it's shockingly good for a system that's essentially "looking" at colored shapes.

Chart Understanding Pipeline

Chart Image → Vision Encoder → Identify Chart Type → Read Axes & Labels → Extract Data Points → Structured Output
Chart data extraction
This is a bar chart from a quarterly report. Extract the data as JSON: { "title": "...", "x_axis": "...", "y_axis": "...", "data": [{"label": "...", "value": 0}], "trend": "increasing|decreasing|stable|mixed" } Approximate values to the nearest whole number.
Works well

Bar charts — clear bars, distinct labels

Line graphs — single or few lines, labeled axes

Pie charts — labeled slices with percentages

Simple tables rendered as images

Struggles with

Overlapping data series — multiple stacked bars

Dense scatter plots — hundreds of data points

3D charts — perspective distortion confuses values

Tiny axis labels — low resolution kills accuracy

Pro tip: Tell the model to "approximate to the nearest whole number" or "round to one decimal place." This manages expectations and reduces hallucinated precision. A model claiming a bar is exactly "47.3" when the axis doesn't have that granularity is hallucinating precision, not reading data.

Table Extraction: The Crown Jewel

Tables in images are traditionally one of the hardest problems in document processing. Traditional OCR can read the text in each cell, but figuring out which cell belongs to which row and column? That requires understanding the visual layout — exactly what multimodal AI excels at.

Table extraction prompt
This image contains a table. Extract it as a JSON array of objects where each object is a row, and the keys are the column headers. Example format: [{"Name": "Alice", "Age": 30, "City": "NYC"}, {"Name": "Bob", "Age": 25, "City": "LA"}] Preserve the exact values. Use null for empty cells.

By giving an example shape, the model knows to use column headers as keys. Without this, it might output a flat list, a markdown table, or something else entirely.

This works remarkably well for:

  • Spreadsheet screenshots (when you can't get the actual file)
  • Tables in PDF reports (especially older PDFs where text extraction fails)
  • Handwritten tables on whiteboards
  • Comparison tables on websites (screenshot and extract)
Watch for merged cells and multi-row headers. Tables with complex merged cells, nested headers, or spanning rows will confuse even the best models. For these cases, add explicit instructions: "The first two rows are headers. Row 1 is the category, row 2 is the sub-category." Guide the model through the structure.

Handwriting, Diagrams, and the Messy Real World

Handwriting Recognition

Can multimodal AI read handwriting? Yes — better than you'd expect, worse than you'd hope. Clean, neat handwriting in a well-lit photo? Usually fine. A doctor's prescription on a crumpled napkin? Expect hallucinations.

~85% accuracy on neat handwriting
~60% accuracy on messy handwriting
~40% accuracy on cursive or shorthand

The trick with handwriting is to give the model context. "This is a handwritten shopping list" triggers very different expectations than "this is a handwritten chemistry formula." Context helps the model resolve ambiguous letters.

Diagram Understanding

Flowcharts, architecture diagrams, wireframes, mind maps — multimodal models can describe what they see, identify elements and connections, and even convert diagrams to structured representations.

Diagram-to-structure prompt
This is a flowchart diagram. Describe the flow as a structured list of steps with decision points. Format: 1. [Start] → description 2. [Decision] condition → Yes: step / No: step 3. [Process] description 4. [End] outcome Include all branches and paths.

The model "sees" boxes, arrows, diamonds, and text. By asking for a structured list with specific markers ([Decision], [Process]), you get a consistent, parseable output instead of a rambling paragraph.

Whiteboard photos are surprisingly useful. After a design session, photograph the whiteboard and ask the model to: (1) transcribe all text, (2) describe the diagram structure, (3) convert it to Mermaid or PlantUML syntax. You'll have a digital version of your whiteboard discussion in seconds.

Multi-Page Documents: The PDF Problem

Real documents aren't single pages. Reports, contracts, manuals — they're multi-page PDFs. How do you handle that with vision models?

Strategy 1: Page-by-Page Processing

Convert each page to an image and process independently. Simple, but you lose cross-page context (a table that spans two pages, a paragraph that continues on the next page).

Strategy 2: Key Page Selection

Don't process every page. Identify the pages that matter (first page for summary, last page for signatures, specific pages for data) and only send those. This saves tokens and focuses the model's attention.

Strategy 3: Multi-Image Single Request

Send multiple pages in a single API call (as multiple images) so the model has cross-page context. This is the best approach for short documents (5-10 pages), but token costs add up quickly.

Multi-Page Document Processing Pipeline

PDF Input → Convert Pages to Images → Classify Each Page → Send Relevant Pages to LLM → Merge Results 
from pdf2image import convert_from_path

# Convert PDF to images (one per page)
pages = convert_from_path("report.pdf", dpi=150)

# Process each page (or selected pages)
for i, page in enumerate(pages):
    page.save(f"page_{i+1}.png", "PNG")
    result = extract_document(f"page_{i+1}.png",
        f"This is page {i+1} of {len(pages)}. "
        "Extract key data from this page.")
DPI matters. 150 DPI is usually the sweet spot — readable text without enormous images. 300 DPI doubles the pixel count (and token cost) but rarely improves results unless the document has tiny text. 72 DPI is too low for most text.

When to Use What: OCR Tools vs. Multimodal LLMs

Multimodal LLMs aren't always the right choice. Here's when to use each approach:

Criteria Traditional OCR Multimodal LLM
Speed Fast (<1 second) Slower (2-10 seconds)
Cost per document Very cheap ($0.001) More expensive ($0.01-0.05)
Understanding Characters only Full semantic understanding
Layout handling Needs templates/rules Automatic, layout-agnostic
Structured output You build the parser Built-in JSON extraction
Volume Millions/day, no problem Rate-limited, costly at scale
Best for Simple text extraction, high volume, low cost Complex documents, varied layouts, meaning extraction
T
Tesseract OCR
Free, open-source, runs locally. Best for simple text extraction at high volume. No understanding — pure character recognition. Use when you just need raw text and will handle parsing yourself.
A
AWS Textract
Cloud OCR with table and form detection built in. Returns structured data with bounding boxes. Good middle ground between raw OCR and full LLM understanding. Tables and forms are its sweet spot.
G
Google Document AI
Specialized document processors for invoices, receipts, IDs, and more. Pre-trained for specific document types. Higher accuracy than general OCR, lower cost than LLM calls. Great when your document type matches a supported processor.
C
Claude / GPT-4o / Gemini Vision
Full multimodal understanding. Handles any document type without training. Returns structured data with semantic understanding. Best for varied, complex documents where building custom parsers isn't feasible.

The smart play? Use a hybrid approach. Use fast, cheap OCR (Tesseract or cloud OCR) for simple, high-volume text extraction. Use multimodal LLMs for complex documents, new layouts, and cases where you need semantic understanding — not just character recognition.

Building a Receipt Scanner: End-to-End

Let's build something real. Here's a complete receipt scanner that processes an image and returns validated, structured data:

import anthropic, base64, json

def scan_receipt(image_path):
    client = anthropic.Anthropic()
    with open(image_path, "rb") as f:
        img = base64.standard_b64encode(f.read()).decode("utf-8")

    resp = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        messages=[{"role": "user", "content": [
            {"type": "image", "source": {"type": "base64",
                "media_type": "image/jpeg", "data": img}},
            {"type": "text", "text": """Extract this receipt
as JSON: {"merchant":"...", "date":"YYYY-MM-DD",
"items":[{"name":"...","qty":1,"price":0.00}],
"subtotal":0.00,"tax":0.00,"total":0.00}
Use null for missing fields. Be precise with numbers."""}
        ]}])
    data = json.loads(resp.content[0].text)
    # Validate: do line items add up?
    if data.get("items") and data.get("subtotal"):
        calc = sum(i["qty"] * i["price"] for i in data["items"])
        data["_validation"] = "pass" if abs(calc - data["subtotal"]) < 0.02 else "check"
    return data

Notice the validation step at the end. We calculate whether the line items add up to the subtotal. If they don't match (within a 2-cent tolerance for rounding), we flag it. This is the kind of defensive programming you need when building on top of AI output — trust but verify.

Production tip: Always parse AI output inside a try/except block. Even with JSON instructions, models occasionally return malformed JSON, add explanatory text before the JSON, or wrap it in markdown code fences. Strip those before parsing. A robust pipeline expects and handles imperfect output.

Pitfalls That Will Bite You

After building a few document understanding features, these are the traps everyone falls into:

1. Hallucinated Numbers

The model might "read" $1,250 as $1,260 or flip a digit in an account number. Always validate numerical output when accuracy matters. Cross-reference totals with line items. Flag impossible values (negative prices, dates in the future).

2. Language Assumptions

If your prompt says "extract the date," the model might interpret "01/02/2024" as January 2 (US) or February 1 (EU). Specify the format: "Dates in YYYY-MM-DD format, ISO 8601."

3. Overconfidence with Blurry Text

Blurry, faded, or partially obscured text is where hallucinations peak. The model won't say "I can't read this" — it'll guess. Add to your prompt: "If text is unclear or unreadable, use null instead of guessing."

4. Token Cost Surprises

A high-resolution scan of an A4 document can cost 2,000+ tokens as input. Processing 1,000 documents a day? That's 2 million tokens just for images. Budget for it.

  • Always validate extracted numbers against internal consistency
  • Specify date formats explicitly (ISO 8601 preferred)
  • Include "use null for unclear values" in every extraction prompt
  • Resize images to 150 DPI before sending (save tokens without losing quality)
  • Wrap JSON parsing in error handling
  • Log raw model responses for debugging failed extractions
