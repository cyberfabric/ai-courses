CHAPTER 10

Real-Time Multimodal Pipelines

Streaming audio, video, and text to AI models — with virtually no delay

From Batch to Live

Everything we've built so far in this course has been batch processing. You take a photo, send it to an API, wait a few seconds, get a response. You record audio, upload it to Whisper, get a transcript. You extract video frames, send them to Claude, get a summary.

That's incredibly useful — but it's not how humans communicate. We don't record a sentence, wait three seconds, then listen to a reply. We talk and listen at the same time. We see and hear simultaneously. We interrupt, point at things, react in real-time.

Real-time multimodal AI is about making machines work the same way — processing audio, video, and text simultaneously, with latency so low it feels like a natural conversation.

The Simultaneous Interpreter

Real-time multimodal AI is like a simultaneous interpreter at the UN. They listen to the speaker in one language, understand the meaning, translate in their head, and speak the translation — all at the same time, with virtually no delay. Now imagine that interpreter can also see what the speaker is pointing at, read the slides behind them, and adjust their translation based on the audience's facial expressions. That's what we're building toward.

If you've tried ChatGPT's Advanced Voice Mode or Gemini Live, you've already experienced this. You talk to the AI like you would talk to a person. You can interrupt it mid-sentence. It responds in a natural voice with emotion and pacing. It feels alive in a way that typing prompts and reading responses simply doesn't.

In this chapter, you'll learn how these systems work under the hood, what makes real-time processing so challenging, and how to build your own streaming pipelines.

Why Real-Time Is Hard

Processing multimodal data in real-time is fundamentally different from batch processing. Here's why:

The Latency Budget

In a natural conversation, the gap between one person finishing a sentence and the other person starting to respond is about 200-300 milliseconds. That's your target. If an AI takes longer than about 500ms to start responding, the conversation feels sluggish and unnatural. Over 1 second? It feels broken.

Now think about what needs to happen in those 200-300ms:

  1. Capture audio from the microphone
  2. Send it to a server (network latency)
  3. Convert speech to text (or process raw audio)
  4. Run the language model to generate a response
  5. Convert the response text to speech
  6. Send the audio back to the client
  7. Play it through the speaker

That's seven steps, and each one takes time. If any single step is too slow, the whole experience falls apart.

200ms Target end-to-end latency for natural conversation
500ms Maximum latency before users notice delay
7+ Pipeline steps that must fit inside that budget

The Three Enemies of Real-Time

1. Network latency. Every round trip to a cloud server adds 20-100ms. If your pipeline requires multiple sequential API calls (STT → LLM → TTS), those round trips stack up. A user in Sydney hitting a server in Virginia might add 200ms just from the speed of light.

2. Processing time. LLMs take time to generate tokens. Whisper takes time to transcribe. TTS takes time to synthesize. Even fast models need some time to think.

3. Buffering. Audio and video data comes in chunks. You need enough data to make sense of it (you can't transcribe half a syllable), but waiting for bigger chunks means more delay. Finding the right chunk size is a constant balancing act.

Audio: The Most Latency-Sensitive Modality

Humans are far more sensitive to audio latency than visual latency. A 100ms delay in a video stream is barely noticeable — we're used to video buffering. But a 100ms echo or delay in audio during a conversation is immediately jarring. This is why voice AI systems prioritize audio latency above everything else.

Think about it: you can have a perfectly natural phone call at 360p video quality, but add 300ms of audio delay and the conversation becomes painful. Audio latency is the bottleneck that defines real-time multimodal quality.

The Traditional Pipeline: STT → LLM → TTS

Before native multimodal real-time APIs existed, the standard approach was to chain three separate models together. This is still a valid architecture, and understanding it helps you appreciate why the newer approaches are such a big deal.

The Classic Voice Assistant Pipeline

Microphone → STT (Whisper) → LLM (Claude/GPT) → TTS (ElevenLabs) → Speaker

Step 1: Capture audio from the user's microphone. Step 2: Send audio chunks to a speech-to-text model (Whisper, Deepgram, AssemblyAI). Step 3: Send the transcribed text to an LLM for a response. Step 4: Stream the LLM's text output to a TTS engine to generate speech. Step 5: Play the audio back to the user.

Here's what a minimal implementation looks like:

import pyaudio, io, anthropic
from openai import OpenAI
from elevenlabs import stream as el_stream

oai = OpenAI()
claude = anthropic.Anthropic()

# 1. Record audio until silence detected
audio_data = record_until_silence()

# 2. Transcribe with Whisper
transcript = oai.audio.transcriptions.create(
    model="whisper-1",
    file=io.BytesIO(audio_data)
)

# 3. Get LLM response (streaming)
response = claude.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=300,
    messages=[{"role": "user",
               "content": transcript.text}],
    stream=True
)

# 4. Stream text to TTS and play audio
el_stream(text_stream=response)
The key optimization in this pipeline is streaming at every step. Don't wait for the full LLM response before starting TTS. Instead, stream the LLM's tokens directly to the TTS engine, which starts synthesizing speech as soon as it receives the first sentence. This technique is called pipeline parallelism, and it can cut perceived latency in half.

Where the Latency Lives

Let's break down the timing for a typical request:

Step Typical Latency Notes
Audio capture + silence detection 300-800ms Waiting for the user to stop speaking
Network: audio upload 50-150ms Depends on audio length and connection
STT processing 200-500ms Whisper API; faster with Deepgram (~100ms)
LLM time-to-first-token 200-600ms Varies by model and prompt length
TTS first-chunk generation 100-300ms Time until first audio chunk is ready
Network: audio playback start 50-100ms Streaming the first audio chunk back

Total: 900ms - 2.5 seconds. That's noticeably slow for a conversation. Usable, but not natural. You can optimize individual steps (use Deepgram instead of Whisper for faster STT, use a smaller LLM for faster responses), but the fundamental problem is sequential processing: each step waits for the previous one to finish.

Silence detection is the hidden latency killer. How does your system know the user has finished speaking? Most systems wait for a gap in the audio (typically 500-800ms of silence). That means even before any AI processing starts, you've already burned half a second just waiting to be sure the user is done talking. Advanced systems use Voice Activity Detection (VAD) models to detect end-of-speech more quickly and accurately.

The Breakthrough: Native Real-Time APIs

The chained pipeline works, but it has an inherent limitation: the LLM only sees text. All the richness of the human voice — tone, emotion, emphasis, hesitation — gets flattened into a plain text transcript before the model ever sees it. And the model's response is also flat text that the TTS engine must bring to life.

Native real-time APIs change the game by processing audio (and sometimes video) directly, without the text intermediary.

O
OpenAI Realtime API (GPT-4o)
Audio in, audio out — no separate STT or TTS step. The model hears your voice directly and generates speech directly. It understands tone, emotion, and emphasis in your voice. It can be interrupted mid-sentence. Supports function calling during voice conversations. Uses WebSocket connections for low-latency streaming. This is what powers ChatGPT's Advanced Voice Mode.
G
Google Gemini Live API
Native multimodal streaming with audio and video input. You can have a voice conversation while sharing your camera feed — the model sees what you see and hears what you say, simultaneously. Supports real-time screen sharing, camera input, and voice interaction. The closest thing to "talking to an AI that's sitting next to you looking at your screen."

How Native Real-Time Works

Instead of the chained STT → LLM → TTS pipeline, native real-time models work like this:

Native Real-Time Pipeline

Microphone → Audio Chunks (WebSocket) → Multimodal Model → Audio Chunks (WebSocket) → Speaker

The model receives raw audio (and optionally video frames) via a persistent WebSocket connection. It processes the audio directly — no transcription step. It generates audio output directly — no TTS step. The result is dramatically lower latency because you've eliminated two entire pipeline stages.

The difference in latency is significant:

Native Real-Time

Time to first audio response: ~300-500ms Pipeline steps: 3 (capture → model → playback) Voice quality: Natural, emotional, preserves speaker's tone context Interruption: Built-in support

Fewer steps, lower latency, richer understanding of voice characteristics.

Chained STT → LLM → TTS

Time to first audio response: ~1000-2500ms Pipeline steps: 7 (capture → upload → STT → LLM → TTS → download → playback) Voice quality: TTS voice only, loses speaker's emotional context Interruption: Custom implementation

More steps, higher latency, voice information lost in transcription.

The Transport Layer: WebSockets and WebRTC

Real-time streaming needs a fast, persistent connection between the client and server. Two technologies dominate this space:

WebSockets

WebSockets provide a persistent, bidirectional connection between client and server. Unlike HTTP (which is request-response), a WebSocket stays open, letting both sides send data at any time. This is what most real-time AI APIs use.

Think of HTTP as sending letters — you write a letter, mail it, wait for a reply. WebSockets are like a phone call — the line stays open and either side can talk at any time.

import asyncio, websockets, json, base64

async def realtime_session():
    url = "wss://api.openai.com/v1/realtime"
    headers = {
        "Authorization": "Bearer YOUR_KEY",
        "OpenAI-Beta": "realtime=v1"
    }
    async with websockets.connect(
        url, extra_headers=headers
    ) as ws:
        # Send audio chunks as they're captured
        async for chunk in mic_stream():
            await ws.send(json.dumps({
                "type": "input_audio_buffer.append",
                "audio": base64.b64encode(
                    chunk).decode()
            }))
        # Receive and play response audio
        async for msg in ws:
            data = json.loads(msg)
            if data["type"] == "response.audio.delta":
                play_audio(data["delta"])

WebRTC

WebRTC (Web Real-Time Communication) is the technology behind video calls in your browser. It's designed for ultra-low-latency audio and video streaming, with built-in support for echo cancellation, noise suppression, and adaptive bitrate.

While WebSockets send data through a server, WebRTC can establish peer-to-peer connections, reducing latency even further. OpenAI's Realtime API also supports WebRTC connections, which is the recommended approach for browser-based applications.

When to use which? Use WebSockets for server-side applications (Python scripts, backend services, mobile apps). Use WebRTC for browser-based applications where you need the lowest possible audio/video latency and built-in media handling. WebRTC handles echo cancellation, noise reduction, and codec negotiation automatically — things you'd have to build yourself with WebSockets.

Interruption Handling: The Conversation Killer

Here's a scenario that seems simple but is surprisingly hard to get right: the user starts talking while the AI is still speaking. In a human conversation, we handle interruptions naturally — we stop talking when someone else starts. But for an AI system, this requires careful engineering.

Why Interruptions Are Tricky

When the AI is generating a response, multiple things are happening simultaneously:

  • The LLM is still generating tokens
  • The TTS engine is still converting text to audio
  • Audio chunks are being sent to the client
  • The speaker is playing those chunks

When the user interrupts, you need to:

  1. Detect that the user is speaking (not just coughing or a background noise)
  2. Stop the current audio playback immediately
  3. Cancel the in-progress LLM generation (why keep generating a response nobody will hear?)
  4. Capture the user's new input
  5. Process the new input with context of what was already said

Native real-time APIs like OpenAI's handle this automatically — they detect when the user starts speaking and stop their own output. But if you're building a custom pipeline, you need to implement this yourself.

The echo problem. If the AI's audio output is picked up by the user's microphone, the system thinks the user is speaking and interrupts itself. This creates an infinite loop of self-interruption. The solution is echo cancellation — filtering out audio that matches what the system is currently playing. WebRTC handles this automatically; custom WebSocket implementations need explicit echo cancellation logic.

Function Calling in Voice Conversations

One of the most powerful features of real-time APIs is function calling during voice conversations. The model can decide, mid-conversation, that it needs to call an external function — check the weather, look up a database, control a smart home device — and seamlessly integrate the result into its spoken response.

Imagine saying: "Hey, what's the weather like in Berlin tomorrow?" The model:

  1. Hears your question
  2. Decides it needs to call a get_weather function
  3. Sends the function call to your server
  4. Receives the weather data
  5. Speaks: "Tomorrow in Berlin it'll be 18 degrees and partly cloudy — nice day for a walk!"

All of this happens in about a second, during a natural voice conversation. No typing, no waiting for a text response.

# Define tools for the real-time session
tools = [{
    "type": "function",
    "name": "get_weather",
    "description": "Get current weather",
    "parameters": {
        "type": "object",
        "properties": {
            "location": {"type": "string"}
        },
        "required": ["location"]
    }
}]

# When the model calls a function:
if event["type"] == "response.function_call":
    result = get_weather(event["arguments"])
    # Send the result back to continue
    await ws.send(json.dumps({
        "type": "conversation.item.create",
        "item": {"type": "function_call_output",
                 "output": json.dumps(result)}
    }))

Live Camera + AI: Seeing in Real-Time

Voice conversations are impressive, but the next frontier is adding vision to the real-time loop. Imagine an AI that can see what your phone's camera sees, hear what you're saying, and respond in real-time. That's already possible.

How Live Vision Works

The approach is conceptually simple: capture video frames from the camera, send them alongside audio to the model, and get a response that accounts for both what it sees and what it hears.

In practice, you can't send every frame — that would overwhelm both the network and the model. Instead, you sample frames at a reasonable rate (typically 1-5 frames per second) and send them as part of the streaming session.

Live Multimodal Pipeline

Camera + Mic → Audio Stream + Frame Samples → Multimodal Model → Audio + Text Response → Speaker + Screen

The camera feed is sampled at 1-5 FPS (much lower than the full video rate). These frames are sent to the model alongside the continuous audio stream. The model processes both inputs and generates a response that references what it sees and hears. Gemini Live does this natively; for other models, you'd combine a frame-capture loop with the audio streaming pipeline.

Use Cases for Live Vision + Voice

  • Visual assistance for the blind: "What's in front of me? Is the traffic light green?" — an AI with your phone's camera feed providing real-time narration of the world
  • Live tech support: "My printer is showing an error" — point your camera at the printer and the AI reads the error code, looks it up, and talks you through the fix
  • Cooking assistance: Point your camera at your cutting board — "Am I dicing these onions finely enough?" The AI sees your technique and gives live feedback
  • Learning and training: "Show me how to tie this knot" — the AI watches your hands and corrects your technique in real-time
  • Real-time translation: Point your camera at a sign in a foreign language while the AI reads it aloud in your language

Screen Sharing: The Developer's Use Case

For developers, the most immediately useful live vision feature is screen sharing with an AI. Share your screen (or IDE), talk to the AI about your code, and it can see exactly what you see. "Why is this test failing?" becomes a much richer question when the AI can read the error message, see the code, and look at the terminal output — all while you explain the context verbally.

Gemini Live already supports this. You share your screen, talk to Gemini, and it responds based on what it sees and hears. This is essentially pair programming with an AI that can see your screen.

Building a Simple Real-Time Assistant

Let's build a basic real-time voice assistant using the chained approach. While not as smooth as native real-time APIs, this gives you full control over each step and works with any combination of models.

import pyaudio, wave, io, anthropic
from openai import OpenAI

oai = OpenAI()
claude = anthropic.Anthropic()
history = []

def listen(seconds=5):
    """Record audio from microphone."""
    p = pyaudio.PyAudio()
    stream = p.open(format=pyaudio.paInt16,
        channels=1, rate=16000, input=True,
        frames_per_buffer=1024)
    frames = [stream.read(1024)
              for _ in range(int(16000/1024*seconds))]
    stream.stop_stream(); stream.close()
    buf = io.BytesIO()
    wf = wave.open(buf, 'wb')
    wf.setnchannels(1); wf.setsampwidth(2)
    wf.setframerate(16000); wf.writeframes(
        b''.join(frames)); wf.close()
    buf.seek(0)
    return buf

def assistant_loop():
    """Main conversation loop."""
    while True:
        print("Listening...")
        audio = listen(seconds=5)
        # STT
        text = oai.audio.transcriptions.create(
            model="whisper-1", file=audio).text
        print(f"You: {text}")
        # LLM
        history.append({"role": "user",
                        "content": text})
        resp = claude.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=200, messages=history)
        reply = resp.content[0].text
        history.append({"role": "assistant",
                        "content": reply})
        print(f"AI: {reply}")
        # TTS
        speech = oai.audio.speech.create(
            model="tts-1", voice="nova",
            input=reply)
        play_audio(speech.content)

assistant_loop()
This is a simplified example. A production-quality version would add: Voice Activity Detection (VAD) to know when the user has stopped speaking, streaming at each step (not waiting for full responses), interruption handling, error recovery, and a proper audio playback queue. Libraries like LiveKit, Pipecat, and Vocode handle all of this for you.

Edge vs Cloud: Where to Process

One of the biggest architecture decisions for real-time multimodal is: where does the processing happen?

Cloud Processing

Pros: + Most capable models (GPT-4o, Gemini) + No device constraints + Easy to update and improve Cons: - Network latency (20-200ms per hop) - Requires internet connection - Privacy: audio/video sent to servers - Cost per API call

Edge/On-Device Processing

Pros: + Near-zero network latency + Works offline + Privacy: data stays on device + No per-call API costs Cons: - Limited model capability - Constrained by device hardware - Harder to update - Battery/thermal concerns on mobile

The practical answer is usually both. Use on-device models for latency-critical, privacy-sensitive tasks (wake word detection, basic voice commands, echo cancellation) and cloud models for the heavy thinking (complex reasoning, long conversations, multimodal understanding).

This is exactly how Siri and Google Assistant work: a small on-device model detects "Hey Siri" with near-zero latency, then the cloud handles the actual request. Apple's on-device models handle simple tasks locally, and only escalate to the cloud for complex queries.

The hybrid approach is winning. Whisper Tiny runs on-device for fast, rough transcription to provide immediate feedback ("I heard you say..."), while the full request is processed in the cloud for an accurate response. The user sees instant acknowledgment (low perceived latency) while the real processing happens in the background.

Where This Is Heading

Real-time multimodal AI is evolving incredibly fast. Here's what's coming:

Always-On Multimodal AI

Smart glasses (Meta Ray-Ban, etc.) with cameras and microphones that feed an always-on AI. You look at a restaurant menu in Japanese, and the AI whispers the English translation in your ear. You're in a meeting and forget someone's name — the AI recognizes their face and quietly reminds you. This is the "AI companion" vision, and the hardware is nearly there.

Real-Time Video Generation

Today, models understand video in real-time. Soon, they'll generate video in real-time. Imagine a video call where the AI renders itself as a realistic avatar, complete with facial expressions and gestures that match its speech. Or real-time game environments generated entirely by AI based on your actions.

Multimodal Agents

AI agents that can see your screen, hear your voice, control your computer, and carry out complex tasks autonomously. "Book me a flight to Berlin next Thursday, the cheapest option" — and the agent navigates the browser, fills in forms, and asks you verbally for confirmation before booking. This combines real-time multimodal understanding with the ability to act.

The "AI as a Colleague" Milestone

We're moving from AI as a tool (type a prompt, get a response) to AI as a colleague (talks to you, sees what you see, works alongside you in real-time). The technical pieces — real-time voice, live vision, function calling, low latency — are all landing now. The next 2-3 years will be about combining them into experiences that feel genuinely collaborative, not just responsive.

Your Real-Time Toolkit

Building real-time multimodal systems from scratch is hard. These frameworks handle the plumbing so you can focus on the experience:

LK
LiveKit
Open-source platform for real-time audio and video. Handles WebRTC, room management, and media routing. Has an "Agents" framework specifically for building AI voice/video assistants. Integrates with OpenAI Realtime API, Deepgram, ElevenLabs, and more. Probably the most mature option for production real-time AI.
PC
Pipecat
Open-source framework by Daily for building voice and multimodal AI agents. Provides a pipeline abstraction where you plug in STT, LLM, and TTS components. Handles audio transport, interruption, VAD, and turn-taking out of the box. Great for prototyping real-time voice assistants quickly.
DG
Deepgram
Ultra-fast speech-to-text API (~100ms latency for streaming transcription). If you're building a chained pipeline and Whisper is too slow, Deepgram's streaming STT can dramatically reduce your STT latency. Also offers TTS. Purpose-built for real-time use cases.
RT
OpenAI Realtime API
Native audio-to-audio model API. WebSocket and WebRTC support. Built-in VAD, interruption handling, and function calling. The simplest path to a high-quality real-time voice assistant if you're in the OpenAI ecosystem. Currently supports GPT-4o and GPT-4o-mini.

Tips and Common Pitfalls

Start with a managed API. Don't build a real-time pipeline from scratch unless you have a specific reason. Use OpenAI's Realtime API or Gemini Live for prototyping. You can always swap in custom components later once you understand the latency and quality requirements.
Measure latency at every step. Add timestamps at each pipeline stage. If your total latency is 1.5 seconds, you need to know exactly where that time is going. Is it silence detection? STT? LLM? TTS? You can only optimize what you measure.
Real-time is expensive. A voice conversation uses continuous API calls for the duration of the conversation. A 10-minute voice chat can cost significantly more than a text conversation of the same length. Monitor costs carefully and set session duration limits. OpenAI charges per-minute for Realtime API sessions.
Don't forget the non-AI parts. The best AI model in the world doesn't matter if your audio capture has glitches, your WebSocket drops, or your playback buffer underruns. Reliable audio I/O, stable network connections, and proper error handling are just as important as the AI model itself. Real-time systems have no room to retry — when something fails, the user hears silence.
