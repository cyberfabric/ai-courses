Chapter 5

Vector Databases: Your AI's Memory

Where embeddings live and how similarity search works at scale

From a Python List to a Million Vectors

In Chapter 3, we stored our embeddings in a plain Python list and searched with a for loop. It worked beautifully for 50 chunks. But imagine your company has 100,000 documents, each split into 10 chunks. That's a million vectors. Your for loop now compares the user's query against every single one of them, one by one.

That's like searching for a book in a library by walking down every aisle, pulling out every book, and checking the cover. It works — technically — but you'd be there all day.

A vector database is the catalog system. It organizes your embeddings so that when a query comes in, it jumps straight to the right neighborhood and pulls the best matches in milliseconds — even across millions of vectors.

The one-sentence version: A vector database stores your embeddings, indexes them for lightning-fast similarity search, and returns the nearest neighbors to any query vector.

What a Vector Database Actually Does

At its core, a vector database does three things:

  1. Stores vectors — your embeddings plus any metadata you attach (source file, date, category, page number)
  2. Indexes them — builds a data structure that makes similarity search fast (we'll cover the algorithms shortly)
  3. Searches them — given a query vector, finds the k most similar vectors in the index

Think of it like a regular database, but instead of looking up rows by exact key matches (WHERE id = 42), you're asking "which rows are closest to this point in 1536-dimensional space?"

The vector database bridges indexing time and query time. At indexing time, you fill it with embeddings. At query time, you search it. It's the persistent memory of your RAG system.

The Speed Problem: Why You Can't Just Loop

Let's put numbers on it. Say you have 1 million vectors of 1536 dimensions each. A brute-force search computes cosine similarity against every single vector:

# Brute force: O(n) comparisons
import numpy as np

def brute_force_search(query_vector, all_vectors, top_k=5):
    """Search 1M vectors one by one. Accurate, but painfully slow."""
    similarities = np.dot(all_vectors, query_vector)
    top_indices = np.argsort(similarities)[-top_k:][::-1]
    return top_indices

# With 1M vectors: ~200-500ms per query
# With 10M vectors: ~2-5 seconds per query
# With 100M vectors: go get lunch

For a chatbot answering user questions, 2-5 seconds just for retrieval is unacceptable. Users expect answers in under a second. You need a way to search without checking every vector.

The solution? Approximate Nearest Neighbors (ANN).

Approximate Nearest Neighbors: Close Enough Is Fast Enough

Here's the key insight that makes vector databases possible: you don't need the exact nearest neighbors. You need very good nearest neighbors, and you need them fast.

Imagine you're in a new city looking for a good Italian restaurant. You could visit every single restaurant, read every menu, and pick the best one. Or you could go to the neighborhood known for Italian food and pick from the top three you find there. You might miss the absolute best restaurant hidden in a random alley — but you'll find a great one in minutes instead of days.

That's what ANN algorithms do. They trade a tiny bit of accuracy (maybe 95-99% recall instead of 100%) for massive speed gains (milliseconds instead of seconds).

The three major ANN indexing algorithms:

1. IVF (Inverted File Index) — "Divide the City into Neighborhoods"

IVF clusters all your vectors into groups (typically 100-1000 clusters). When a query comes in, it first figures out which clusters are closest, then only searches vectors within those clusters.

  • Analogy: A city divided into districts. Looking for Italian food? Check the restaurant district, skip the industrial zone.
  • Speed: Instead of searching 1M vectors, you search maybe 10,000 (the nearest cluster).
  • Tradeoff: Search more clusters (nprobe) for better accuracy but slower speed.

2. HNSW (Hierarchical Navigable Small Worlds) — "A Highway System for Vectors"

HNSW builds a multi-layer graph where each vector is a node connected to its neighbors. The top layer has few nodes with long-range connections (highways). Each lower layer adds more nodes with shorter connections (local roads). To search, you start at the top and zoom in.

  • Analogy: Flying from New York to a small town in France. You take a transatlantic flight (top layer), then a regional flight (middle layer), then a taxi (bottom layer). You don't walk the whole way.
  • Speed: Searches in O(log n) time — even 100M vectors take only milliseconds.
  • Tradeoff: Uses more memory (1.5-2x the raw vector data) to store the graph structure.

3. Product Quantization (PQ) — "Compress to Save Space"

PQ compresses each vector by splitting it into sub-vectors and replacing each sub-vector with the ID of its nearest centroid from a codebook. This can shrink a 1536-dimension float32 vector from 6KB to under 100 bytes.

  • Analogy: Instead of storing every photo at full resolution, you store thumbnails. You lose some detail, but you can fit millions more photos on your phone.
  • Speed: Searching compressed vectors is much faster because they fit in CPU cache.
  • Tradeoff: Lower accuracy — PQ is usually combined with IVF or HNSW, not used alone.
Which algorithm should you care about? In practice, most vector databases use HNSW as their default. It's the best general-purpose algorithm for datasets up to hundreds of millions of vectors. You rarely need to configure the algorithm yourself — the database picks a good default. But understanding how it works helps you debug performance issues and tune parameters when needed.

The Big Players: Choosing a Vector Database

The vector database landscape in 2026 has matured significantly. Here are the five databases you'll encounter most often, each with a distinct personality:

🟢
ChromaDB
The "Just Get Started" database. Install with pip install chromadb, create a collection, and you're storing and querying vectors in under five minutes. No infrastructure, no configuration files. Uses HNSW under the hood. Perfect for prototyping and small-to-medium production workloads. Starts to hit performance limits around 5-10 million vectors.
🌲
Pinecone
The "I don't want to manage infrastructure" database. Fully managed cloud service — get an API key, create an index, start querying. No servers to provision, no indexes to tune. Serverless pricing means you pay for what you use. The easiest path from prototype to production, though you're locked into their platform and pricing.
🦀
Qdrant
The "Performance matters" database. Written in Rust, Qdrant delivers exceptional speed — P95 latency of 22ms at 10 million vectors. Powerful metadata filtering with payload indexes. Available as open-source (self-host) or managed cloud. Great choice when you need fast filtered search at scale.
🔷
Weaviate
The "batteries included" database. Open-source with built-in modules for generating embeddings — insert raw text and Weaviate handles the vectorization. Native hybrid search combines vector and keyword search in a single query. Best when you want vectorization and search tightly integrated.
🐘
pgvector
The "use what you already have" option. A PostgreSQL extension that adds vector columns and similarity search to your existing Postgres database. Supports both HNSW and IVF indexing. No new infrastructure to manage, full SQL power, transactional consistency. The default recommendation if you already run Postgres.

Hands-On: Your First Vector Database with ChromaDB

Enough theory. Let's store and search some vectors. We'll use ChromaDB because you can go from zero to searching in about 10 lines of code.

# Install: pip install chromadb

import chromadb

# 1. Create a client (in-memory for prototyping)
client = chromadb.Client()

# 2. Create a collection (like a table in a regular database)
collection = client.create_collection(
    name="company_docs",
    metadata={"hnsw:space": "cosine"}  # Use cosine similarity
)

# 3. Add documents — Chroma embeds them automatically!
collection.add(
    ids=["doc1", "doc2", "doc3", "doc4", "doc5"],
    documents=[
        "Our refund policy allows returns within 30 days of purchase.",
        "Employees get 20 days of paid vacation per year.",
        "The engineering team uses Python and TypeScript primarily.",
        "Our office is located at 123 Main Street, San Francisco.",
        "Customer support is available 24/7 via chat and email."
    ],
    metadatas=[
        {"department": "sales", "updated": "2026-01"},
        {"department": "hr", "updated": "2026-03"},
        {"department": "engineering", "updated": "2026-02"},
        {"department": "operations", "updated": "2025-11"},
        {"department": "support", "updated": "2026-04"}
    ]
)

# 4. Query — Chroma embeds the question and finds similar documents
results = collection.query(
    query_texts=["How do I return a product?"],
    n_results=2
)

print(results["documents"])
# [['Our refund policy allows returns within 30 days of purchase.',
#   'Customer support is available 24/7 via chat and email.']]

print(results["distances"])
# [[0.312, 0.891]]  — lower distance = more similar

That's it. Five documents in, one query out, and the most relevant document comes back first. No for loop, no numpy, no manual embedding. ChromaDB handles the embedding, indexing, and searching for you.

Persisting to Disk

The in-memory client is great for experiments, but your vectors disappear when the process exits. To persist them:

# Use PersistentClient to save to disk
client = chromadb.PersistentClient(path="./chroma_data")

# Everything else stays the same!
collection = client.get_or_create_collection("company_docs")

# Your vectors survive restarts

Metadata Filtering: Narrowing the Search

Real-world RAG systems don't just search by similarity — they also filter by metadata. Want to search only documents from the HR department? Only documents updated in the last month? Metadata filtering lets you narrow down candidates before the vector search happens.

Metadata filtering in action: Think of it as a two-step search. First, the database eliminates vectors that don't match your filter (like removing all non-HR documents). Then, it runs similarity search only on what's left. This is faster and more relevant than searching everything and filtering after. 
# Search only within the HR department
results = collection.query(
    query_texts=["How many vacation days do I get?"],
    n_results=3,
    where={"department": "hr"}  # Pre-filter by metadata
)

# Combine multiple filters
results = collection.query(
    query_texts=["What tools does the team use?"],
    n_results=3,
    where={
        "$and": [
            {"department": "engineering"},
            {"updated": {"$gte": "2026-01"}}
        ]
    }
)

Metadata filtering is essential for multi-tenant RAG systems where each user should only see their own documents, or for time-sensitive data where you want to prioritize recent content.

Collections and Namespaces: Organizing Your Vectors

As your RAG system grows, you'll have different types of documents — support articles, product docs, internal wikis, meeting notes. You can dump them all into one collection, or organize them into separate collections.

Think of collections like different shelves in a library. The fiction shelf, the reference shelf, the periodicals shelf. Each has its own index and can be searched independently.

# Separate collections for different document types
support_docs = client.create_collection("support_articles")
product_docs = client.create_collection("product_documentation")
internal_wiki = client.create_collection("internal_wiki")

# Search only support articles
results = support_docs.query(
    query_texts=["How do I reset my password?"],
    n_results=3
)

# Or search across collections by querying each one
# and merging results (more on this in Chapter 7)
One collection or many? Start with one collection and use metadata filtering to separate document types. Split into multiple collections only when you have distinct embedding models for different content types, or when you need completely isolated search scopes (e.g., per-tenant data).

Upserting, Updating, and Deleting: Keeping Your Index Fresh

Documents change. Policies get updated, products get discontinued, new FAQs get written. Your vector database needs to stay in sync with your source data.

# Upsert: insert or update if the ID already exists
collection.upsert(
    ids=["doc1"],
    documents=["Our refund policy allows returns within 60 days of purchase."],
    metadatas=[{"department": "sales", "updated": "2026-04"}]
)

# Delete by ID
collection.delete(ids=["doc4"])

# Delete by metadata filter
collection.delete(where={"department": "operations"})

The key concept is upsert — "update if exists, insert if new." This is how you keep your index fresh without worrying about duplicates. Most vector databases support upsert as a first-class operation.

Vector databases are not traditional databases. Don't try to do SQL-style joins, aggregations, or complex relational queries. A vector database answers one question well: "Which vectors are most similar to this one?" It stores metadata for filtering, not for relational modeling. If you need joins and transactions alongside vector search, look at pgvector, which gives you both worlds in PostgreSQL.
Index build time vs. query time is a real tradeoff. When you add vectors, the database needs to update its index. HNSW indexes are fast to query but slower to build — inserting 1 million vectors can take minutes. If you're doing bulk ingestion, check if your database supports batch inserts (most do) and consider building the index after all data is loaded rather than incrementally.

A Complete Example: Building a FAQ Search Engine

Let's put it all together with a more realistic example. We'll build a FAQ search engine that indexes questions and answers, then retrieves the best match for a user query:

import chromadb

# Initialize persistent storage
client = chromadb.PersistentClient(path="./faq_db")
faq = client.get_or_create_collection("faq", metadata={"hnsw:space": "cosine"})

# Our FAQ data (in practice, you'd load this from a file or API)
faqs = [
    {"q": "How do I cancel my subscription?",
     "a": "Go to Settings > Billing > Cancel Plan. You'll keep access until the end of your billing period."},
    {"q": "What payment methods do you accept?",
     "a": "We accept Visa, Mastercard, American Express, and PayPal."},
    {"q": "Can I upgrade my plan mid-cycle?",
     "a": "Yes! Go to Settings > Billing > Change Plan. You'll be charged a prorated amount."},
    {"q": "How do I export my data?",
     "a": "Go to Settings > Data > Export. You can export as CSV or JSON."},
    {"q": "Is there a mobile app?",
     "a": "Yes, available on iOS and Android. Search for 'OurApp' in your app store."},
]

# Index the FAQs — embed the question for matching, store the answer as metadata
faq.upsert(
    ids=[f"faq_{i}" for i in range(len(faqs))],
    documents=[f["q"] for f in faqs],       # Embed the questions
    metadatas=[{"answer": f["a"]} for f in faqs]  # Store answers as metadata
)

# Search
def ask_faq(question, top_k=2):
    results = faq.query(query_texts=[question], n_results=top_k)
    for i, (doc, meta, dist) in enumerate(zip(
        results["documents"][0],
        results["metadatas"][0],
        results["distances"][0]
    )):
        print(f"\n--- Match {i+1} (distance: {dist:.3f}) ---")
        print(f"Q: {doc}")
        print(f"A: {meta['answer']}")

ask_faq("I want to stop paying")
# --- Match 1 (distance: 0.287) ---
# Q: How do I cancel my subscription?
# A: Go to Settings > Billing > Cancel Plan. ...
#
# --- Match 2 (distance: 0.614) ---
# Q: Can I upgrade my plan mid-cycle?
# A: Yes! Go to Settings > Billing > Change Plan. ...

Notice how "I want to stop paying" matched "How do I cancel my subscription?" even though they share no words in common. That's the power of semantic search through embeddings — and the vector database made it fast.

Using a Custom Embedding Model

ChromaDB's default embedding model is fine for getting started, but for production you'll want to choose your own. Here's how to use OpenAI embeddings with ChromaDB:

import chromadb
from chromadb.utils.embedding_functions import OpenAIEmbeddingFunction

# Use OpenAI's embedding model
openai_ef = OpenAIEmbeddingFunction(
    api_key="your-api-key",
    model_name="text-embedding-3-small"
)

# Pass the embedding function when creating the collection
collection = client.create_collection(
    name="docs_with_openai",
    embedding_function=openai_ef
)

# Now all add/query operations use OpenAI embeddings automatically
Never mix embedding models in the same collection. If you indexed documents with OpenAI's text-embedding-3-small, you must query with the same model. Vectors from different models live in completely different spaces — comparing them is like comparing GPS coordinates from two different planets.

When to Use Which Database: A Decision Tree

Use ChromaDB when:
  • You're prototyping or building a personal project
  • You have fewer than 5 million vectors
  • You want zero configuration
  • You need something working in 5 minutes
Use pgvector when:
  • You already run PostgreSQL
  • You need transactional consistency
  • You want SQL power alongside vector search
  • Your team doesn't want another database to manage
Use Pinecone when:
  • You have no infrastructure team
  • You want managed everything
  • You need to scale to billions of vectors
  • You'd rather pay than operate
Use Qdrant or Weaviate when:
  • You need high-performance filtered search
  • You're building a multi-modal or hybrid search system
  • You want open-source with cloud-hosted options
  • You need fine-grained control over indexing
Build a vector-powered document search.
  1. Install ChromaDB: pip install chromadb
  2. Pick a set of 20+ text documents (a README, blog posts, a FAQ, meeting notes — anything text-based)
  3. Split each document into chunks of roughly 200-300 words (manually is fine for now)
  4. Index all chunks into a ChromaDB collection with metadata (source filename, chunk number)
  5. Write 5 test queries and run them against your collection
  6. For each query, print the top 3 results with their distance scores
  7. Bonus: Try filtering by source file — search only within a specific document

This exercise gives you hands-on experience with the full vector database workflow: indexing, querying, and filtering. You'll start to develop intuition for what distance scores "feel like" — essential for tuning retrieval quality in later chapters.
