The Big Idea

No single retrieval method is best for every query. The secret is combining keyword search (precision on exact terms) + semantic search (understanding meaning) + re-ranking (quality check) into a pipeline that handles whatever users throw at it.

🔍 Three Ways to Find a Needle in the Haystack

Aspect 🔤 Keyword (BM25) 🧠 Semantic Search ⚡ Hybrid Search
How it works Matches exact words (TF-IDF with length normalization) Compares embedding vectors for meaning similarity Runs both, merges results with RRF
Wins when Error codes, product names, CLI commands, exact IDs Paraphrased questions, synonyms, conceptual queries Mixed query types in production
Fails when Different words, same meaning ("cancel" vs "terminate") Exact identifiers, domain jargon, confusable topics Rarely — that's the point
Relevant in top 10 62% 71% 87% (with re-ranking)
Speed Very fast Fast (pre-computed embeddings) Slightly slower (two searches + fusion)

🧩 Core Concepts

🔤
BM25
Ctrl+F on steroids. Scores by term frequency (diminishing returns), inverse document frequency (rare words matter more), and length normalization. The classic keyword algorithm.
🔀
Reciprocal Rank Fusion
Merges BM25 and vector results by rank position, not raw scores. Formula: 1/(k+rank) with k=60. If a chunk ranks high in both methods, it bubbles to the top.
🎯
Cross-Encoder Re-ranking
Feeds query + document together into one model for precise scoring. Much more accurate than bi-encoder (separate embeddings) but too slow for first-pass search. The "hiring manager" after the recruiter.
🏷️
Metadata Filtering
Narrow search space before vector search using labels (version, date, department). Like telling the librarian "only look in Science." Don't over-filter or you'll exclude relevant chunks.
🔄
Multi-Query Retrieval
LLM rephrases the user's question 3+ ways, searches with all variants, merges results. Improves recall for vague or ambiguous queries. Tradeoff: more API calls.
⚖️
Bi-Encoder vs Cross-Encoder
Bi-encoder: embed query and doc separately, compare. Fast, pre-computable. Cross-encoder: feed both together, output score. Slow, precise. Use bi-encoder for search, cross-encoder for re-ranking.

📈 Build Up Complexity Gradually

  1. 🟢 Start: Pure Semantic Search

    Simple, effective, good enough for prototypes and most early use cases.

  2. 🔤 Add BM25 → Hybrid Search

    When you notice keyword queries failing (exact terms, IDs, error codes).

  3. 🎯 Add Cross-Encoder Re-ranking

    When precision matters (legal, medical, compliance). Scores top 20 candidates, returns best 3-5.

  4. 🔄 Add Multi-Query Retrieval

    When users ask vague or ambiguous questions. LLM rephrases the query 3+ ways.

  5. 🏷️ Add Metadata Filtering

    When data has natural categories (version, department, date). Filter before search.

📖 Key Terms

BM25 (Best Match 25)
Keyword search algorithm that scores by term frequency, inverse document frequency, and length normalization.
Reciprocal Rank Fusion (RRF)
Combines ranked lists from different search methods using rank positions, not raw scores. Formula: 1/(k+rank), k=60.
Hybrid Search
Running both keyword (BM25) and semantic (vector) search on every query, then fusing results. The production standard for RAG in 2026.
Cross-Encoder
A model that takes query+document together as input and outputs a relevance score. More accurate than bi-encoders but too slow for first-pass search.
Bi-Encoder
Embeds query and document separately, then compares vectors. Fast and pre-computable, but misses fine-grained query-document interactions.
Re-ranking
A second-pass scoring step that uses a cross-encoder to re-order the top candidates from initial retrieval. The "quality check."
Multi-Query Retrieval
Using an LLM to rephrase the user's question multiple ways, searching with all variants to improve recall.
Precision@k
Of the top k results returned, how many are actually relevant? The key metric for retrieval quality.

🧪 Quick Quiz

Why can't you just add BM25 scores and cosine similarity scores together?

They're on completely different scales (BM25 might give 14.7 while cosine gives 0.82). RRF solves this by ignoring raw scores entirely and using only rank positions to combine results.

Why is re-ranking done as a second pass instead of being the primary search method?

Cross-encoders feed query + document together into one model, making them very accurate but very slow. You can't pre-compute anything. So you use fast approximate search first (top 20-100), then re-rank only those candidates.

When would semantic search return a confidently wrong result?

When topics are superficially similar but unrelated: "return policy" vs "return types in TypeScript," or "Python memory management" vs "snake species memory." The embedding sees topical overlap that isn't actually relevant.

Why is retrieving 3-5 chunks better than 20 chunks?

Stuffing too much context causes the LLM to get confused by contradictory chunks, buries important info in noise, and wastes tokens/money. Research shows 3-5 high-quality chunks consistently beat 10-20 mediocre ones.

What's the first retrieval upgrade you should add to a basic semantic search?

Add BM25 for hybrid search. It catches keyword queries that semantic search misses (error codes, exact IDs, CLI commands) and typically improves top-10 recall from ~71% to ~87% (with re-ranking).

⚠️ Watch Out

🎭
Semantic ≠ Relevant
Topically similar isn't always actually relevant. "Return policy" and "return types in TypeScript" may score high together. Hybrid search + re-ranking catches these.
📚
More Chunks ≠ Better
3-5 high-quality chunks beat 10-20 mediocre ones. Too much context confuses the LLM and wastes tokens.
🚫
Don't Over-Filter
Strict metadata filters can exclude the best chunk. The answer to "v3.2 API changes" might live in general API docs, not tagged as v3.2.

📌 Key Takeaways

  • Hybrid search (BM25 + semantic + RRF) is the production standard — 87% relevant in top 10 vs 62-71% alone.
  • BM25 wins on exact terms (error codes, IDs, commands). Semantic wins on meaning (synonyms, paraphrases).
  • RRF merges results by rank position (not scores), using 1/(60+rank). Simple and rarely needs tuning.
  • Re-ranking with a cross-encoder is the biggest single precision upgrade. Slow but accurate — run it on the top 20, not all candidates.
  • 3-5 high-quality chunks beat 10-20 mediocre ones. Focus on precision, not volume.
  • Build up complexity gradually: semantic only → add BM25 → add re-ranking → add multi-query → add metadata filters.
