The Big Idea

Production RAG balances three forces: quality (better answers), speed (faster responses), and cost (cheaper operations). Improve one, the others suffer — unless you're clever about caching, streaming, and access control.

🧊 Three Caching Layers

  1. Layer 1: Embedding Cache

    Cache query text → embedding vector. Normalize text first (lowercase, strip whitespace). Saves embedding API calls for repeated queries.

  2. Layer 2: Retrieval Cache

    Cache query embedding → chunk IDs + scores. Skips the vector DB round-trip entirely for repeated queries.

  3. Layer 3: Semantic Cache

    The big one. "What's the refund policy?" and "How do I get my money back?" are the same question. Embed queries, find similar cached ones (threshold ~0.92), return cached answers.

🔄 Keeping the Index Fresh

Strategy When to Use Tradeoff
Full re-index Changing embedding model, major restructuring Expensive but guarantees consistency
Incremental upsert New/updated documents Cheap, but needs change detection
Scheduled refresh Sources without webhooks Simple, but introduces staleness window
🔑

Pro Tip: Hash Your Content

Store a content hash alongside each vector. Compare hashes to detect changes — hash comparison is free, re-embedding is an API call.

🔐 Security & Access Control

🛡️
Cardinal Rule
Enforce authorization before augmentation. If an unauthorized doc enters the LLM context, the answer is compromised. Filter at retrieval time, not generation time.
🏷️
Metadata Filtering
Store permission metadata at ingestion. Filter by department, role, user ID during search. All major vector DBs support this.
📋
Audit Logging
Log every access: who queried, which chunks returned, when. Essential for compliance and debugging.
💉
Prompt Injection
Malicious docs can contain "ignore previous instructions." Sanitize content, warn the LLM, and limit document access. Defense in depth.

📈 Scaling the Vector Database

Sharding
Split index across multiple nodes. Managed DBs handle this automatically; self-hosted requires config.
Read Replicas
Write to one node, read from many. Serves more concurrent queries.
Index Tuning (HNSW)
Higher ef_construction & M = better recall but slower indexing and more memory.
Quantization
Compress float32 vectors to int8 or binary. Cuts memory 4-8x with small accuracy loss.
Matryoshka Embeddings
Truncate vectors to fewer dimensions (e.g., 256 instead of 1536). Cuts storage 6x at 1M+ vectors.

🧪 Quick Quiz

What's the biggest cost component in a production RAG system?

LLM generation — it accounts for ~97% of costs. Embedding and vector DB costs are comparatively small.

What is a semantic cache, and why is it so effective?

A semantic cache finds similar questions (not just identical ones) by comparing query embeddings. "What's the refund policy?" matches "How do I get my money back?" — delivering sub-100ms responses vs multi-second LLM calls.

Why must access control happen at retrieval time, not generation time?

If an unauthorized document enters the LLM's context, the response may leak that information — even if you try to filter the output afterward. The LLM has already "seen" the document.

What's the difference between the silo and pool multi-tenancy models?

Silo = separate vector index per tenant (strong isolation, higher cost). Pool = shared index with mandatory tenant_id metadata filter (lower cost, but a filter bug = data leak).

How do you monitor faithfulness in production without judging every response?

Sample 1-5% of queries and run them through your faithfulness evaluator. Track the score over time — a drop signals data corruption, model changes, or retrieval degradation.

📌 Key Takeaways

  • Production RAG balances quality, speed, and cost — LLM generation is ~97% of your bill.
  • The 3-second latency budget: embed (50-100ms) + search (10-50ms) + re-rank (100-200ms) + generate (500-2000ms). Stream to cut perceived latency.
  • Three caching layers (embedding, retrieval, semantic) are the single most impactful production optimization — up to 65x faster on cache hits.
  • Hash your content to detect changes cheaply. Use incremental upserts for day-to-day freshness, full re-index only when changing embedding models.
  • Enforce access control at retrieval time, never at generation time. Store permissions as metadata, filter during search.
  • Monitor four layers: infrastructure, retrieval quality, generation quality, and user satisfaction. Sample faithfulness checks are your canary.
  • Analogy: prototype = dinner party for 10 friends. Production = restaurant serving 500 a night. Same recipes, everything else changes.
