Chunking: The Art of Splitting Documents

Memory Map
💡

The Big Idea

A chunk is the unit of retrieval. How you split your documents is the single biggest lever in RAG quality — more impactful than your choice of embedding model or vector database. Get chunking right and even a simple pipeline shines.

⚖️ The Chunking Dilemma

Aspect 🔬 Too Small (50-100 tokens) 🎯 Sweet Spot (200-512 tokens) 🗞️ Too Big (2000+ tokens)
Content Sentence fragments Complete paragraphs Multiple pages
Retrieval Precise match but no context Relevant match with context Lots of irrelevant noise
LLM answer "Yes." (no explanation) Complete, grounded answer Buries the answer in noise
Embedding quality Muddled, incomplete meaning Focused, clean vectors Diluted across many topics

✂️ Six Chunking Strategies

  1. 📏 Fixed-Size Chunking

    Split every N characters. Simple, predictable. Good for prototypes and uniform text (logs, transcripts). Will cut sentences mid-word without overlap.

  2. 🔄 Recursive Character Splitting ⭐

    The workhorse — your default choice. Tries paragraph boundaries first, falls back to sentences, then words. Preserves natural structure. 512 tokens + 64 overlap is the recommended baseline.

  3. 📝 Sentence-Based Chunking

    Splits on sentence boundaries using NLP (NLTK). Guarantees complete thoughts. Great for well-written prose; struggles with code, tables, and bullet lists.

  4. 🏗️ Document-Structure-Aware

    Uses headings, sections, and HTML/Markdown structure. Each chunk carries hierarchy metadata (e.g., h2: Refund Policy). Best for docs, wikis, and API references.

  5. 🧠 Semantic Chunking

    Embeds every sentence, splits where meaning shifts. Sounds smart, but benchmarks show 512-token recursive splitting often wins. Always test with your data.

  6. 🔮 Late Chunking

    Embed the entire document first, then split token embeddings into chunks. Pronouns and references keep their context. Supported by Jina jina-embeddings-v3.

🧩 Key Concepts

🔗
Overlap
Consecutive chunks share 10-20% of text so boundary sentences aren't lost. Not always beneficial — SPLADE study found zero benefit in some setups. Test it.
🏷️
Chunk Metadata
Always attach: source file, chunk index, section heading, last updated date. Essential for filtering, attribution, and debugging.
📊
Tokens vs Characters
Always think in tokens — that's what models consume. 512 tokens ≈ 380 words ≈ 1,900 characters. Use tiktoken to count.
🧹
Preprocessing Matters
Strip boilerplate before chunking. PDFs need header/footer removal. HTML needs nav/sidebar stripping. Markdown is the easiest format to chunk.
🐛
Tiny Chunk Warning
Chunks under ~20 words ("See next section") match many queries but provide zero context. Filter or merge them with neighbors.
📐
Quality Sanity Check
Read 10 random chunks (do they make sense alone?), run 10 test queries (do top-3 contain the answer?), check size distribution (spike of tiny chunks = problem).

🧪 Quick Quiz

What's the recommended baseline chunk size for most RAG projects?

400-512 tokens with 10-15% overlap, using recursive character splitting. This covers ~80% of use cases. Benchmark and adjust from there.

Why does recursive splitting outperform fixed-size splitting?

Recursive splitting tries paragraph boundaries first, then sentences, then words. This keeps complete thoughts together, producing cleaner embeddings. Fixed-size will happily cut a sentence in half, creating muddled vectors that match queries poorly.

When would you choose structure-aware chunking over recursive?

When your documents have clear heading structure (Markdown, HTML, wikis, API docs). Each chunk gets metadata about its position in the hierarchy (e.g., h2: Refund Policy), which is a strong retrieval signal.

What's the key difference between traditional chunking and late chunking?

Traditional: chunk first, then embed each chunk independently. Late: embed the entire document first (so all tokens see full context via attention), then split the token embeddings into chunks. Late chunking preserves pronoun references and cross-section context.

Why should you be skeptical of semantic chunking despite it sounding "smarter"?

Benchmarks (NAACL 2025, Vectara 2026) found that semantic chunks averaged only 43 tokens — too little context for good LLM answers. Plain recursive 512-token splitting matched or beat semantic chunking on end-to-end accuracy. "Smarter" ≠ "better." Always benchmark.

⚠️ Common Pitfalls

📑
PDF Headers & Footers
Repeat on every page, polluting chunks with "Page 47 | Company Confidential." Strip them before chunking or your embeddings will be noisy.
🤖
"Smarter" ≠ Better
Semantic chunking sounds advanced but benchmarks show 512-token recursive splitting often wins. Reason: semantic chunks are too small for the LLM to generate good answers.
📏
Characters ≠ Tokens
500 characters of code ≠ 500 characters of prose. Always set chunk sizes in tokens — that's what models actually process.

📌 Key Takeaways

  • Chunking is the #1 lever for RAG quality — more impactful than embedding model or vector DB choice.
  • Start with recursive character splitting at 512 tokens with 10-15% overlap. It covers 80% of projects.
  • Use structure-aware splitting for documents with headings (Markdown, HTML, wikis).
  • The sweet spot is 200-512 tokens per chunk — enough for a complete thought, not enough to drown in noise.
  • Always preprocess before chunking: strip PDF headers, HTML boilerplate, and convert to Markdown when possible.
  • Attach metadata to every chunk: source file, chunk index, section heading, last updated date.
  • Don't trust "smart" ≠ "better" — benchmark your chunking strategy against your actual data and queries.
