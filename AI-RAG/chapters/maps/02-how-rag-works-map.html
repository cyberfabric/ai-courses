Chapter 2

How RAG Actually Works

Memory Map
💡

The Big Idea

RAG has two phases: indexing (prepare your data once) and query (answer questions in real time). Like a restaurant — prep work happens before service, not when the order arrives.

🍳 The Restaurant Analogy

Aspect 🔪 Indexing (Prep Work) 🍽️ Query (Service Time)
When Offline, done once or periodically Online, every time a user asks
Speed Minutes to hours (no rush) Under 2 seconds (real-time)
Steps Load → Chunk → Embed → Store Embed Q → Search → Prompt → Generate
Analogy Chopping vegetables, making sauces Taking the order, cooking, plating
Runs Once per document set Once per user question

  1. 📄 Load Documents

    Read text from PDFs, Markdown, Confluence, Google Docs. In production, this is the messiest step — OCR, boilerplate stripping, API rate limits.

  2. ✂️ Split Into Chunks

    Break documents into small, meaningful pieces (~200–500 tokens). Use overlap (10–20%) so sentences at boundaries aren't lost. Chunk size is the #1 lever for RAG quality.

  3. 🔢 Create Embeddings

    Convert each chunk into a vector (list of numbers) that captures its meaning. Similar meanings = nearby vectors. One API call per batch.

  4. 💾 Store in Vector Database

    Save vectors + metadata (source file, page) in a database optimized for similarity search. Built once, used for every query. Like building a library catalog.

🔍 Phase 2: Query Pipeline

❓ User Asks
"How many vacation days?"
→
🔢 Embed Q
~50–100ms
→
🔍 Search
~10–50ms
→
📋 Build Prompt
<1ms
→
✍️ Generate
~500–2000ms
→
📨 Answer
With sources

  1. 🔢 Embed the Question

    Convert the user's question into a vector using the same model used during indexing. Question and chunks must live in the same "meaning space."

  2. 🔍 Search for Similar Chunks

    Find the top-k nearest vectors in the database. "Relevant text" becomes "nearby points." Semantic search finds meaning, not just matching words.

  3. 📋 Build the Augmented Prompt

    Combine retrieved chunks + user question into a prompt that tells the LLM: "Answer from this context ONLY. Cite your sources."

  4. ✍️ Generate the Answer

    LLM reads the retrieved context and generates a grounded answer with source citations. It synthesizes across chunks, not just copy-pastes.

🚧 Where Each Stage Can Fail

📄
Loading
Corrupt PDFs, garbled text extraction, missing pages. Sign: chunks contain gibberish.
✂️
Chunking
Too small = lost context. Too big = diluted relevance. Sign: correct doc exists but isn't retrieved.
🔢
Embedding
Wrong model choice or mixing models. Sign: related chunks aren't near each other.
🔍
Retrieval
Returns topically related but factually irrelevant chunks. Sign: wrong answer despite right docs in DB.
✍️
Generation
LLM ignores context or hallucinates beyond it. Sign: answer contradicts retrieved documents.

⚠️ 80%+ of RAG failures are upstream of generation — don't blame the LLM first!

⏱️ Latency Budget (Per Query)

Step Time % of Total
Embed question 50–100ms ~5%
Vector search 10–50ms ~2%

Total: under 2 seconds. LLM generation dominates the latency budget.

📝 Key Terms

Indexing Phase
The offline prep step: load docs, chunk, embed, store. Done once per document set.
Query Phase
The real-time answer step: embed question, search, build prompt, generate. Runs per question.
Chunk
A small, meaningful piece of a document (~200–500 tokens). The unit of retrieval in RAG.
Overlap
Chunks share 10–20% of content at boundaries so important sentences aren't cut in half.
Similarity Search
Finding the k vectors closest to the query vector — turning "find relevant text" into "find nearby points."
Augmented Prompt
The prompt sent to the LLM combining: system instructions + retrieved chunks + user question.
Source Attribution
Including references in the answer so users can verify which document the information came from.

🧪 Quick Quiz

What are the two phases of RAG and when does each run?

Indexing runs offline/once: load, chunk, embed, store. Query runs in real time per question: embed question, search, build prompt, generate answer.

Why must you use the same embedding model for indexing and querying?

Question and document vectors must live in the same "meaning space" so similarity search works. Different models create different spaces — vectors from one model are meaningless to another.

Why not just stuff all your documents into the LLM's context window?

Three reasons: (1) context windows have limits, (2) cost scales linearly with token count, (3) accuracy drops when relevant info is buried in long context. RAG's indexing makes it scalable.

Where do 80%+ of RAG failures originate?

Upstream of generation — bad loading, poor chunking, wrong embedding model, or irrelevant retrieval. Always check what was retrieved before blaming the LLM.

What takes up ~93% of the query latency budget?

LLM generation (500–2000ms). Embedding and vector search together are under 150ms.

🎒 Key Takeaways

  • RAG = two phases: indexing (prep once) + query (answer in real time). Like a restaurant's prep kitchen and service line.
  • Indexing: Load → Chunk → Embed → Store. Query: Embed Q → Search → Build Prompt → Generate.
  • Chunk size (~200–500 tokens with overlap) is the #1 lever for RAG quality.
  • Use the same embedding model for indexing and querying — different models create incompatible vector spaces.
  • 80%+ of RAG failures happen upstream of generation — check retrieval before blaming the LLM.
  • Total query time is under 2 seconds, dominated by LLM generation (~93% of latency).
