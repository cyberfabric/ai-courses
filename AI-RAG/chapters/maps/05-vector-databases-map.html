Vector Databases: Your AI's Memory

Memory Map

The Big Idea

A vector database is the catalog system for your embeddings — it organizes millions of vectors so similarity search takes milliseconds instead of minutes. It's the persistent memory that bridges indexing time and query time in every RAG system.

🧩 Core Concepts

🗄️
What a Vector DB Does
Three jobs: store vectors + metadata, index them for fast search, search for the k nearest neighbors to any query vector.
⚡
ANN Search
Approximate Nearest Neighbors — trade a tiny bit of accuracy (95-99% recall) for massive speed gains. Like picking a great restaurant in the Italian district instead of visiting every restaurant in the city.
🏘️
IVF (Inverted File Index)
Clusters vectors into neighborhoods. Search only the closest clusters instead of every vector. Tune nprobe to balance speed vs accuracy.
🛣️
HNSW
A multi-layer graph — highways at the top, local roads at the bottom. Searches in O(log n) time. The default algorithm in most vector DBs. Uses ~1.5-2x memory.
🗜️
Product Quantization
Compresses vectors from ~6 KB down to <100 bytes by replacing sub-vectors with codebook IDs. Like storing thumbnails instead of full-res photos. Usually combined with IVF or HNSW.
🏷️
Metadata Filtering
Narrow candidates before vector search using metadata (department, date, category). Two-step: filter first, then similarity search on what remains. Essential for multi-tenant systems.
📚
Collections
Like shelves in a library — organize vectors by document type. Start with one collection + metadata filtering; split only when you need separate embedding models or isolated scopes.
🔄
Upsert
"Update if exists, insert if new." The key operation for keeping your index in sync with source data without creating duplicates.

🔍 The Three ANN Algorithms

  1. 🏘️ IVF — "Divide the City into Neighborhoods"

    Cluster all vectors into 100-1000 groups. Query hits nearest clusters only. Fast setup, tunable via nprobe.

  2. 🛣️ HNSW — "A Highway System for Vectors"

    Multi-layer graph: transatlantic flight → regional flight → taxi. O(log n) search. Best general-purpose algorithm. Higher memory cost.

  3. 🗜️ PQ — "Compress to Save Space"

    Split vectors into sub-vectors, replace each with a codebook centroid ID. Massive compression. Usually paired with IVF or HNSW, not used alone.

📖 Key Terms

Vector Database
A database optimized for storing, indexing, and searching high-dimensional vectors (embeddings) by similarity.
Approximate Nearest Neighbors (ANN)
Algorithms that find "close enough" matches in milliseconds instead of exact matches in seconds. The core trick behind fast vector search.
IVF (Inverted File Index)
Partitions vector space into clusters; searches only the nearest clusters. Tune nprobe for speed vs accuracy.
HNSW (Hierarchical Navigable Small Worlds)
A multi-layer graph that routes from coarse (highways) to fine (local roads). O(log n) search time. The most popular default algorithm.
Product Quantization (PQ)
Compresses vectors by replacing sub-vectors with codebook centroid IDs. Shrinks storage dramatically but reduces accuracy.
Collection
A named group of vectors in a vector database — like a table in a relational DB or a shelf in a library.
Upsert
"Update if exists, insert if new." The standard operation for keeping your vector index in sync with changing source data.
Metadata Filtering
Pre-filtering vectors by attached metadata (department, date, category) before running similarity search.
Cosine Similarity
Measures the angle between two vectors. Closer to 1 = more similar. The default similarity metric in most vector databases.

🧪 Quick Quiz

Why can't you just use a Python list and a for loop for vector search in production?

Brute-force search is O(n) — it checks every single vector. With 1M+ vectors, that's 200-500ms per query, far too slow for interactive use. Vector databases use ANN algorithms (like HNSW) to search in milliseconds.

What does "approximate" mean in Approximate Nearest Neighbors?

ANN algorithms trade a tiny bit of accuracy (95-99% recall) for massive speed gains. You might miss the absolute best match, but you'll find a very good one in milliseconds instead of seconds.

Can you mix embeddings from different models in the same collection?

No! Vectors from different embedding models live in completely different spaces. Comparing them is like comparing GPS coordinates from two different planets. Always use the same model for indexing and querying.

What is an "upsert" and why is it important for RAG?

Upsert = "update if exists, insert if new." It keeps your vector index in sync with changing source data (updated policies, new FAQs, etc.) without creating duplicates.

When should you use metadata filtering instead of multiple collections?

Start with one collection + metadata filtering (by department, date, etc.). Split into multiple collections only when you need separate embedding models for different content types, or completely isolated search scopes like per-tenant data.

⚠️ Watch Out

🚫
Not a Relational DB
Don't try SQL joins, aggregations, or complex relational queries. Vector DBs answer one question: "Which vectors are closest?" For both worlds, use pgvector.
⏱️
Index Build Time
HNSW indexes are fast to query but slower to build. For bulk ingestion, use batch inserts and build the index after all data is loaded.
🔀
Never Mix Models
If you indexed with text-embedding-3-small, you must query with the same model. Different models = different vector spaces = meaningless comparisons.

📌 Key Takeaways

  • A vector database stores embeddings, indexes them for fast ANN search, and returns the k nearest neighbors in milliseconds.
  • HNSW is the default algorithm in most databases — O(log n) search via a multi-layer graph structure.
  • ANN trades a tiny bit of accuracy (95-99% recall) for massive speed gains over brute-force search.
  • Start with ChromaDB (prototyping) or pgvector (if you have Postgres). Scale to Qdrant/Pinecone/Weaviate when needed.
  • Use metadata filtering to narrow search scope — essential for multi-tenant and time-sensitive data.
  • Never mix embedding models in the same collection — always index and query with the same model.
  • Use upsert to keep your index in sync with changing source documents.
