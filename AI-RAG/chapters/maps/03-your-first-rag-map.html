Your First RAG System in 30 Minutes

The Big Idea

A complete, working RAG system fits in ~60 lines of Python with just 3 packages. No frameworks, no vector database — just the raw pattern: load, chunk, embed, search, prompt, generate.

🧰 The Setup

🐍
Python 3.9+
The runtime. You already have it.
🔑
OpenAI API Key
For embeddings. $0.02/million tokens — pennies.
🔑
Anthropic API Key
For LLM generation. (Or use OpenAI for both.)
📦
3 Packages
openai + anthropic + numpy. No LangChain, no LlamaIndex.

🔨 The 7 Build Steps

  1. 📁 Create Knowledge Base

    Make a docs/ directory with .txt files (pricing, refunds, features). Real data, plain text.

  2. 📄 Load Documents

    Read each .txt file into a dict with content and source fields.

  3. ✂️ Chunk Documents

    Split into 500-char chunks with 50-char overlap. Naive but functional.

  4. 🔢 Create Embeddings

    Call text-embedding-3-small — one API call turns chunks into 1536-dim vectors.

  5. 🔍 Search with Cosine Similarity

    Embed the question, compare against every chunk with numpy. Return top-k matches.

  6. 📋 Build Prompt & Generate

    Stuff retrieved chunks into a prompt: "Answer ONLY from context. Cite sources." Send to Claude.

  7. 🧪 Test It!

    Interactive REPL: ask questions, see answers with sources and similarity scores.

✅ What Works vs ❌ What Breaks

Aspect ✅ Works Well ❌ Breaks
Direct questions Accurate, sourced answers —
Paraphrased questions Embeddings capture meaning —
Out-of-scope Proper "I don't know" response —
Chunk boundaries — Info split across chunks is missed
Exact keywords — "SOC 2" may rank low in semantic search
Multi-chunk reasoning — Incomplete answers when facts are scattered

🗺️ What We Skipped & Where We Fix It

✂️
Smart Chunking
Fixed-size splits break mid-sentence. Sentence/paragraph-aware chunking boosts quality 20–30%. → Ch. 6
💾
Vector Database
Numpy search is O(n) and doesn't persist. Real DBs are O(log n) and durable. → Ch. 5
🔀
Hybrid Search
Pure semantic misses exact keywords (product IDs, error codes). Combine with BM25. → Ch. 7
🏆
Re-ranking
Top-k results aren't always best. A cross-encoder reorders for higher precision. → Ch. 7
📊
Evaluation
"It seems to work" isn't a metric. Need precision, recall, faithfulness scores. → Ch. 10
📄
Real Document Loaders
Real data lives in PDFs, HTML, Confluence — not clean .txt files. → Ch. 8

📝 Key Terms

text-embedding-3-small
OpenAI's embedding model: 1536 dimensions, $0.02/million tokens. Best balance of quality and cost for most RAG use cases.
Cosine Similarity
Measures how close two vectors point in the same direction. Score of 1.0 = identical meaning, 0.0 = unrelated. Above 0.7 = strong match.
top_k
The number of most-similar chunks returned by search. Default 3 in this demo. Too few = missing info, too many = noise.
Augmented Prompt
The prompt sent to the LLM: system instructions + retrieved context + user question. "Answer ONLY from context. Cite sources."
Brute-force Search
Comparing query against every chunk. O(n) complexity. Works for small datasets; vector databases needed at scale.
Chunk Overlap
Chunks share characters at boundaries (50 chars in this demo) so sentences at edges aren't split in half.

🧪 Quick Quiz

How many pip packages does the minimal RAG system need?

Three: openai (embeddings), anthropic (generation), and numpy (cosine similarity). No frameworks required.

Why do we use chunk overlap?

So important sentences at chunk boundaries aren't cut in half. Without overlap, "refund policy is 14" might end one chunk while "days for annual" starts the next — neither has the complete info.

What does a cosine similarity score of 0.85 mean?

The question and chunk vectors point in nearly the same direction — a strong semantic match. Scores above 0.7 are typically good matches for RAG retrieval.

Why doesn't this demo use a vector database?

For educational clarity. Numpy brute-force search works for small datasets and lets you see exactly what's happening. Real vector databases are needed for scale (millions of chunks), persistence, and O(log n) search speed.

What's the single biggest improvement you could make to this system?

Smarter chunking. Better chunking alone can boost answer quality 20–30%. The naive 500-char split breaks mid-sentence and loses context.

🎒 Key Takeaways

  • A working RAG system needs only ~60 lines of Python and 3 packages — no frameworks required.
  • The build steps: create docs → load → chunk → embed → search (cosine similarity) → prompt → generate.
  • OpenAI's text-embedding-3-small is the go-to: 1536 dims, cheap, effective for most use cases.
  • Semantic search works great for paraphrased questions but struggles with exact keywords and chunk boundary issues.
  • The #1 improvement: smarter chunking (Ch. 6). Then: vector DB (Ch. 5), hybrid search (Ch. 7), evaluation (Ch. 10).
  • Every production RAG system follows this same core pattern — the rest of the course makes each step better.
