Evaluating Your RAG System

The Big Idea

You can't improve what you can't measure. RAG evaluation has three layers — retrieval, generation, and end-to-end — and you need metrics for each because a failure in any layer looks different and needs a different fix.

🔍
Retrieval Failure
Wrong chunks retrieved. The LLM answers from wrong context. Fix: better chunking, embeddings, or hybrid search.
📝
Generation Failure
Right chunks retrieved, but LLM hallucinated or ignored context. Fix: stronger prompts, better models, compression.
🎯
End-to-End Failure
Retrieval and generation were correct, but didn't address what the user actually wanted. Fix: query rewriting, intent understanding.

📏 Retrieval Metrics

Precision@k
Of the k chunks retrieved, how many were relevant? High precision = less noise in the context window.
Recall@k
Of all relevant chunks that exist, how many were retrieved? High recall = fewer missed answers.
MRR (Mean Reciprocal Rank)
How far down do you have to look to find the first relevant chunk? MRR of 0.8 = usually in top 1-2 positions.
nDCG
The gold standard — handles graded relevance (highly relevant > somewhat relevant) and penalizes bad ranking.

✍️ Generation Metrics

🛡️
Faithfulness
Every claim in the answer is supported by retrieved context. The single most important generation metric — unfaithful = hallucination.
🎯
Relevance
The answer actually addresses what was asked. A faithful summary of the wrong topic is still useless.
📋
Completeness
The answer covers all aspects of the question. Answering half the question faithfully still leaves the user unsatisfied.

📦 Building an Evaluation Dataset

  1. Start: 50 Hand-Crafted Questions

    Real user queries with expected chunks and reference answers. Include simple, multi-hop, ambiguous, and "I don't know" types.

  2. Expand: Synthetic Generation

    Feed chunks to an LLM, generate 3 questions per chunk. RAGAS has a built-in synthetic test generator.

  3. Validate: Keep Real Questions

    Synthetic questions use doc vocabulary. Always keep a core set of real user queries from support tickets and chat logs.

❌
<20 questions
Scores swing wildly between runs — too few to be reliable.
✅
50+ questions
Stable metrics you can track over time. The sweet spot to start.
🚀
200-500 questions
Comprehensive coverage with synthetic generation. For mature systems.

🔬 Debugging Retrieval Failures

Symptom Likely Cause Fix
Right topic, wrong details Chunks too large Smaller chunks or parent-child
Completely irrelevant results Vocabulary mismatch Hybrid search or query rewriting
Right chunk at position 8-10 Ranking issue Add a cross-encoder re-ranker
Chunk not in index at all Ingestion bug Check ingestion pipeline
Bad on complex queries only Needs decomposition Multi-query or step-back

🔬 Debugging Generation Failures

🙈
Context Ignored
LLM uses own knowledge instead of context. Fix: stronger system prompt — "answer ONLY from provided context."
⚔️
Contradictory Context
Different chunks disagree. LLM picks one, maybe wrong. Fix: metadata filtering to prefer recent docs.
🔎
Subtle Confabulation
"30 days" becomes "30 business days." Hardest to catch manually — automated faithfulness scoring is essential.

🧪 Quick Quiz

What are the three layers of RAG evaluation?

Retrieval quality (did you find the right chunks?), generation quality (is the answer correct and grounded?), and end-to-end quality (did the user get what they needed?).

What's the difference between precision@k and recall@k?

Precision@k = of the k chunks retrieved, how many were relevant. Recall@k = of all relevant chunks, how many were retrieved. In RAG, recall usually matters more — missing the right chunk is worse than including noise.

Why is faithfulness the single most important generation metric?

An unfaithful answer is a hallucination — and eliminating hallucinations is the whole reason you're building RAG. If the answer isn't grounded in retrieved context, the system is making things up.

Why should you use a different LLM model as judge than the one that generated the answer?

LLM judges are biased toward their own generation style. Using a different model (e.g., GPT-4 to judge Claude's output) reduces this bias and gives more objective evaluation.

When should you use RAGAS vs DeepEval?

RAGAS for exploratory evaluation ("how good is my system now?"). DeepEval for automated CI/CD testing ("did this change make things worse?"). Use both in a mature project.

📌 Key Takeaways

  • RAG evaluation has three layers: retrieval, generation, and end-to-end — each layer can fail independently and needs its own metrics.
  • Start with precision@5 and recall@5 for retrieval, faithfulness for generation — these cover 80% of what you need.
  • LLM-as-judge achieves ~80% agreement with human evaluators — about as reliable as a second human opinion.
  • Build a 50-question eval dataset first (real user queries), then expand with synthetic generation.
  • RAGAS for exploration, DeepEval for CI/CD — use both in production.
  • High retrieval scores + low faithfulness = generation problem (fix prompts). Low retrieval scores = retrieval problem (fix chunking/search).
  • The evaluation loop: change one variable → run eval → compare scores → keep the winner → repeat.
