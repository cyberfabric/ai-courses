Chapter 9

Advanced RAG Patterns

Memory Map
💡

The Big Idea

Basic RAG retrieves chunks similar to the question. Advanced RAG transforms the question, restructures the chunks, and adds feedback loops — dramatically improving answers without changing your data.

🚧 Three Weaknesses of Basic RAG

🔤 Query Mismatch
User words ≠ doc words
🧩 Context Fragmentation
Answer spans multiple chunks

🧰 The Advanced RAG Toolkit

✍️
Query Rewriting
LLM rephrases vague questions into precise search queries. Like a librarian who reformulates "that thing about leaving" into "employee offboarding procedure."
🔮
HyDE
Generate a hypothetical answer, embed that instead of the question. Works because answers look like documents — bridging the question-document gap in embedding space.
🔭
Step-Back Prompting
Too-specific questions retrieve poorly. Ask a broader question first for context, then answer the specific one. Like understanding the HVAC system before debugging a cold bathroom.
👨‍👧
Parent-Child Retrieval
Index small child chunks (100 tokens) for precision, return parent chunks (500 tokens) for context. Like using a book's index to find the right page.
🪟
Sentence-Window
Index individual sentences, but expand to surrounding sentences on retrieval. Lighter than parent-child — great for flat text like transcripts, emails, and chat logs.
🔗
Multi-Hop RAG
Break complex questions into sub-questions, retrieve for each, then synthesize. Each hop can build on previous answers — like following a chain of references.
🪞
Self-RAG
The system reflects: Do I need retrieval? Are these chunks relevant? Is my answer grounded? Adds 5-10 LLM calls but catches errors before the user sees them.
🗜️
Contextual Compression
After retrieval, extract only the relevant sentences from each chunk. Reduces 2500 tokens of mixed-relevance context down to 500 tokens of pure signal.
🛡️
CRAG
Corrective RAG — a quality gate after retrieval. Evaluates chunks as Correct / Ambiguous / Incorrect. Falls back to web search if retrieval is poor.

📊 Pattern Comparison

Pattern Best For Cost Complexity
✍️ Query Rewriting Vague / conversational questions +1 LLM call Low
🔮 HyDE Vocabulary mismatch with docs +1 LLM call Low
🔭 Step-Back Overly specific questions +1 LLM call, +1 search Low
👨‍👧 Parent-Child Structured docs (sections, headings) +indexing time Medium
🪟 Sentence Window Flat text (transcripts, emails) +indexing time Medium
🔗 Multi-Hop Complex multi-source questions +1 LLM call, +N searches Medium
🪞 Self-RAG High-stakes answers (legal, medical) +5-10 LLM calls High
🗜️ Compression Long chunks with mixed relevance +N LLM calls Low
🛡️ CRAG Unreliable retrieval quality +1 LLM call + fallback Medium

🛤️ Recommended Adoption Order

  1. Start: Query Rewriting

    Highest bang-for-the-buck. One cheap LLM call, dramatic improvement on vague queries. Almost always worth adding first.

  2. Add: Parent-Child Retrieval

    Re-index with small children for precision, return parents for context. Biggest impact on answer completeness.

  3. Add: CRAG Quality Gate

    Catches irrelevant retrieval before it reaches the LLM. Prevents the most common RAG failure: confident answers from wrong context.

  4. If Needed: Self-RAG

    For high-stakes domains (legal, medical, financial) where a wrong answer is worse than a slow one. Adds significant latency.

🏗️ Real-World Pattern Stacks

🏢 Internal Knowledge Base
Query Rewriting + Parent-Child — users tolerate an extra second for better answers
💬 Customer Chatbot
HyDE or Query Rewriting only — one extra call, minimal latency impact
⚖️ Legal / Compliance
Self-RAG + CRAG — accuracy is everything, latency doesn't matter
⚡ Real-Time Search
Basic RAG only — no room for extra LLM calls in autocomplete/suggestions
🔧 Practical Production Stack
Query Rewriting → Parent-Child → CRAG → Contextual Compression → Generate (3-4 extra LLM calls)

⚠️ Watch Out For

🔮
HyDE Can Mislead
If the hypothetical answer is off-base, you'll search in the wrong direction. Always keep the original query as fallback.
🪞
Self-RAG Latency
5-10 LLM calls per query. Only use when accuracy matters more than speed.
🛡️
CRAG Evaluator Quality
The evaluator LLM can misjudge relevance. Use a capable model for evaluation even if you use a faster one for generation.

🧪 Quick Quiz

What are the three fundamental weaknesses of basic RAG?

Query mismatch (user words ≠ doc words), context fragmentation (answer spans multiple chunks), and retrieval overconfidence (always returns something, even when nothing relevant exists).

How does HyDE differ from query rewriting?

Query rewriting rephrases the question into better search queries. HyDE generates a hypothetical answer and embeds that instead, because answers are linguistically closer to documents than questions are.

When should you use parent-child retrieval vs sentence-window retrieval?

Parent-child for documents with clear hierarchical structure (sections, headings). Sentence-window for flat, flowing text (meeting transcripts, chat logs, emails).

What does CRAG do when it evaluates retrieved documents as "Ambiguous"?

It supplements the retrieved chunks with results from a web search (or alternative retrieval strategy), combining both sources before generating the answer.

Which advanced pattern should you almost always add first, and why?

Query rewriting — it's cheap (+1 LLM call), low complexity, and has the highest bang-for-the-buck, especially for vague or conversational user queries.

📌 Key Takeaways

  • Basic RAG struggles with vague queries, multi-chunk answers, and irrelevant retrieval — advanced patterns fix each of these.
  • Query rewriting is the first pattern to add — cheap, simple, and dramatically improves retrieval quality.
  • HyDE bridges the question-document gap by embedding a hypothetical answer instead of the question.
  • Parent-child retrieval gives you precision (small chunks for search) AND context (big chunks for the LLM).
  • Self-RAG and CRAG add reflection and quality gates — use them when accuracy matters more than latency.
  • Don't implement all patterns at once — measure failures first, then add the pattern that fixes your most common failure mode.
  • In production, 2-3 combined patterns (e.g., query rewriting + parent-child + CRAG) covers most use cases.
