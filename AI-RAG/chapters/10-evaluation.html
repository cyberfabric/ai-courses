Chapter 10

Evaluating Your RAG System

Because "it seems to work" is not a metric — how to measure retrieval quality, answer accuracy, and know when your RAG is good enough

The big idea: You can't improve what you can't measure. RAG evaluation breaks down into three layers — did you retrieve the right chunks, did the LLM generate a faithful answer, and did the user get what they needed? Each layer has its own metrics, and you need all three to build a system you can trust.

In the last chapter, you learned nine advanced RAG patterns — query rewriting, HyDE, parent-child retrieval, CRAG, and more. Each one promises better answers. But here's the uncomfortable question: how do you know if they actually help?

"The answers feel better" is not engineering. "Users seem happier" is not a metric. If you can't put a number on it, you can't improve it — and you definitely can't tell your manager why the new chunking strategy was worth two weeks of work.

Think of it this way: you wouldn't ship a web app without checking if the pages load, the forms submit, and the database queries return correct results. RAG systems need the same discipline. The difference is that RAG has three things that can go wrong instead of one — and each needs its own test.

Let's build a measurement system that turns "it seems to work" into real numbers.

The Three Layers of RAG Evaluation

Where evaluation fits in the pipeline: Evaluation is not a step you add at the end — it's a feedback loop that wraps around your entire pipeline. You evaluate retrieval quality (did you find the right chunks?), generation quality (is the answer correct and grounded?), and end-to-end quality (did the user get what they needed?). Each layer tells you something different about where your system is failing.

Imagine you ask your RAG system "What is our refund policy?" and it answers "Our refund policy allows returns within 60 days." There are three distinct ways this could go wrong:

  1. Retrieval failure: The system retrieved chunks about shipping policy instead of refund policy. The LLM answered based on the wrong context. Fix: better chunking, better embeddings, or hybrid search.
  2. Generation failure: The system retrieved the right chunks (which say "30 days"), but the LLM hallucinated "60 days." Fix: better prompts, stronger models, or contextual compression.
  3. End-to-end failure: The retrieval was correct, the generation was faithful, but the answer didn't address what the user actually wanted (they wanted to know how to get a refund, not the time limit). Fix: query rewriting, better understanding of user intent.

You need different metrics for each layer. Let's start with retrieval.

Retrieval Metrics: Did You Find the Right Chunks?

Retrieval evaluation measures whether your search step returns the documents that actually contain the answer. This is the foundation — if retrieval fails, nothing downstream can save you. The key metrics are precision (how much of what you retrieved is relevant?) and recall (how much of what's relevant did you retrieve?).

To evaluate retrieval, you need a ground truth — a set of questions where you already know which chunks contain the correct answer. Then you compare what your system retrieves against what it should retrieve.

Precision@k: Are the Retrieved Chunks Relevant?

Precision@k answers: "Of the k chunks I retrieved, how many were actually relevant?"

def precision_at_k(retrieved_ids: list[str], relevant_ids: set[str], k: int) -> float:
    """What fraction of retrieved chunks are relevant?"""
    top_k = retrieved_ids[:k]
    relevant_in_top_k = sum(1 for doc_id in top_k if doc_id in relevant_ids)
    return relevant_in_top_k / k

# Example: retrieved 5 chunks, 3 are relevant
retrieved = ["doc_1", "doc_7", "doc_3", "doc_12", "doc_5"]
relevant = {"doc_1", "doc_3", "doc_5", "doc_9"}

print(precision_at_k(retrieved, relevant, k=5))  # 3/5 = 0.6

A precision@5 of 0.6 means 60% of what you retrieved was useful. The other 40% was noise — wasting context window tokens and potentially confusing the LLM.

Recall@k: Did You Find Everything Relevant?

Recall@k answers: "Of all the relevant chunks that exist, how many did I retrieve?"

def recall_at_k(retrieved_ids: list[str], relevant_ids: set[str], k: int) -> float:
    """What fraction of relevant chunks did we retrieve?"""
    top_k = retrieved_ids[:k]
    relevant_in_top_k = sum(1 for doc_id in top_k if doc_id in relevant_ids)
    return relevant_in_top_k / len(relevant_ids) if relevant_ids else 0.0

print(recall_at_k(retrieved, relevant, k=5))  # 3/4 = 0.75

A recall@5 of 0.75 means you found 3 out of 4 relevant chunks. The missing one might contain the key detail the LLM needs to give a complete answer.

Precision vs recall tradeoff: Retrieving more chunks (higher k) usually improves recall but hurts precision. Retrieving fewer chunks improves precision but risks missing relevant content. In RAG, you typically care more about recall — it's better to include some noise than to miss the chunk that contains the answer. A common sweet spot: retrieve 5-10 chunks, then use re-ranking to push the best ones to the top.

MRR and nDCG: Ranking Quality Matters

MRR (Mean Reciprocal Rank) answers: "How far down do I look to find the first relevant chunk?" An MRR of 0.8 means the first relevant chunk is usually in the top 1-2 positions. For RAG, this matters a lot — if the best chunk is buried at position 8, the LLM gets the wrong context first.

nDCG (Normalized Discounted Cumulative Gain) is the gold standard. Unlike precision/recall (binary: relevant or not), nDCG handles graded relevance — "highly relevant" scores higher than "somewhat relevant," and results near the top count more than those at the bottom.

Which retrieval metric should you use? Start with precision@5 and recall@5 — they're easy to understand and compute. Add MRR when you care about ranking quality. Graduate to nDCG when you need graded relevance. For most teams, precision + recall at k=5 tells you 80% of what you need to know.

Generation Metrics: Is the Answer Any Good?

Even with perfect retrieval, the LLM can still produce bad answers. Generation evaluation checks three qualities:

Faithfulness: Does the Answer Stick to the Evidence?

Faithfulness (also called "groundedness") measures whether every claim in the answer is supported by the retrieved context. This is the single most important generation metric — an unfaithful answer is a hallucination, and hallucinations are the whole reason you're building RAG in the first place.

def evaluate_faithfulness(answer: str, context: str, client) -> dict:
    """Use an LLM to check if the answer is faithful to the context."""
    response = client.messages.create(
        model="claude-sonnet-4-6-20250514",
        max_tokens=500,
        messages=[{
            "role": "user",
            "content": f"""Evaluate the faithfulness of this answer.

CONTEXT (retrieved documents):
{context}

ANSWER:
{answer}

For each claim in the answer:
1. List the claim
2. Is it supported by the context? (YES / NO / PARTIAL)
3. If NO, what did the answer say vs what the context says?

Then give an overall faithfulness score from 0.0 to 1.0."""
        }]
    )
    return response.content[0].text

Analogy: faithfulness is like checking a student's essay against their source material. Did they cite their sources correctly, or did they make things up?

Answer Relevance and Completeness

Relevance: A faithful answer can still be useless. If you ask "How do I get a refund?" and the system responds with a faithful summary of refund statistics for Q3, it's grounded but doesn't answer the question. Relevance measures whether the answer actually addresses what was asked.

Completeness: "What are the requirements and deadlines for the internship program?" needs both requirements and deadlines. An answer covering only requirements scores high on faithfulness and relevance, but low on completeness.

LLM-as-Judge: Your Automated Evaluator

You might have noticed: most generation metrics require an LLM to evaluate another LLM. This is the LLM-as-judge pattern, and it's the backbone of modern RAG evaluation.

The idea sounds circular — "using AI to grade AI" — but it works surprisingly well. Research shows that strong LLMs (Claude, GPT-4) achieve roughly 80% agreement with human evaluators, which matches the consistency between two different human evaluators. In other words, an LLM judge is about as reliable as a second human opinion.

Why LLM-as-judge works: Evaluating an answer is an easier task than generating one. It's like how a film critic doesn't need to be able to direct a movie to judge one. The LLM judges whether claims are supported by evidence, whether the answer addresses the question, and whether key points are missing — all structured evaluation tasks where LLMs excel.

Here's a practical LLM-as-judge implementation that scores a RAG response on all three dimensions:

from anthropic import Anthropic

client = Anthropic()

def judge_rag_response(question: str, answer: str, context: str,
                       reference_answer: str = None) -> dict:
    """Score a RAG response on faithfulness, relevance, and completeness."""
    ref_section = ""
    if reference_answer:
        ref_section = f"\nREFERENCE ANSWER (ground truth):\n{reference_answer}\n"

    response = client.messages.create(
        model="claude-sonnet-4-6-20250514",
        max_tokens=400,
        messages=[{
            "role": "user",
            "content": f"""You are evaluating a RAG system's response. Score each
dimension from 0.0 to 1.0. Reply in this exact format:
faithfulness: [score]
relevance: [score]
completeness: [score]
explanation: [one sentence summary]

QUESTION: {question}

RETRIEVED CONTEXT:
{context}

GENERATED ANSWER:
{answer}
{ref_section}
Score each dimension:
- faithfulness: Is every claim in the answer supported by the context?
- relevance: Does the answer address what was asked?
- completeness: Does it cover all aspects of the question?"""
        }]
    )

    # Parse scores from response
    text = response.content[0].text
    scores = {}
    for line in text.strip().split("\n"):
        if ":" in line:
            key, value = line.split(":", 1)
            key = key.strip().lower()
            try:
                scores[key] = float(value.strip())
            except ValueError:
                scores[key] = value.strip()
    return scores

LLM judges have biases. They tend to prefer longer answers (even when shorter ones are better), they're biased toward their own generation style, and they can be overly generous on faithfulness. Mitigate this by: (1) being very explicit in your evaluation prompt about what counts as faithful, (2) asking the judge to list each claim and check it individually (chain-of-thought), and (3) using a different model as judge than the one that generated the answer. If Claude generated the answer, use GPT-4 to judge it, or vice versa.

Building an Evaluation Dataset

Metrics are useless without data to measure. You need an evaluation dataset — a set of question-answer-context triples that represent real queries your system should handle.

Think of it like building a test suite for your API. Each "test case" has:

  • Question: What the user asks
  • Expected context: Which chunks should be retrieved (for retrieval metrics)
  • Reference answer: The correct answer (for generation metrics)
# evaluation_dataset.json
[
    {
        "question": "What is our refund policy for online orders?",
        "expected_chunk_ids": ["refund-policy-001", "refund-policy-002"],
        "reference_answer": "Online orders can be refunded within 30 days of purchase. Items must be unused and in original packaging. Refunds are processed to the original payment method within 5-7 business days."
    },
    {
        "question": "How do I set up SSO with Okta?",
        "expected_chunk_ids": ["sso-setup-001", "okta-config-001", "sso-setup-003"],
        "reference_answer": "To set up SSO with Okta: 1) Navigate to Admin > Identity Providers, 2) Click 'Add Provider' and select Okta, 3) Enter your Okta domain and client ID, 4) Configure the callback URL in your Okta dashboard."
    }
]

How Many Questions Do You Need?

<20
Scores swing wildly between runs — too few
50+
Stable metrics to track changes over time
200-500
Comprehensive coverage with synthetic generation

Start with 50 hand-crafted questions that represent real user queries. Prioritize diversity: include simple factual questions, multi-hop questions, questions with ambiguous wording, and questions that should return "I don't know." Quality matters more than quantity — 50 well-crafted questions beat 500 sloppy ones.

Generating Synthetic Test Data

Once you have 50 golden questions, expand your dataset with an LLM: feed each chunk to the model and ask it to generate 3 questions that the chunk should answer, along with expected answers. RAGAS also has a built-in synthetic test generator that creates questions with different difficulty levels (simple, multi-hop, reasoning).

Synthetic questions have a blind spot: they're generated from your documents, so they use the same vocabulary and framing. Real users don't talk like your documentation. Always keep a core set of real user questions in your eval dataset — harvest them from support tickets, chat logs, or user interviews. Synthetic data expands your coverage; real data keeps you honest.

RAGAS: The RAG Evaluation Toolkit

RAGAS (Retrieval Augmented Generation Assessment) is the most popular open-source framework for evaluating RAG systems. Instead of building everything from scratch, RAGAS gives you pre-built metrics and a clean API.

# pip install ragas

from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)
from datasets import Dataset

# Prepare your evaluation data
eval_data = {
    "question": [
        "What is our refund policy?",
        "How do I set up SSO?",
    ],
    "answer": [
        "Refunds are available within 30 days of purchase.",
        "Go to Admin > Identity Providers and add your SSO config.",
    ],
    "contexts": [
        ["Our refund policy allows returns within 30 days..."],
        ["To configure SSO, navigate to Admin > Identity Providers..."],
    ],
    "ground_truth": [
        "Online orders can be refunded within 30 days.",
        "Navigate to Admin > Identity Providers, click Add Provider.",
    ],
}

dataset = Dataset.from_dict(eval_data)

# Run evaluation
results = evaluate(
    dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall],
)

print(results)
# {'faithfulness': 0.92, 'answer_relevancy': 0.88,
#  'context_precision': 0.85, 'context_recall': 0.90}

What RAGAS Measures

Metric What It Checks Needs Ground Truth?
Faithfulness Are claims in the answer supported by retrieved context? No
Answer Relevancy Does the answer address the question? No
Context Precision Are the retrieved chunks relevant to the question? Yes
Context Recall Were all needed chunks retrieved? Yes

The first two metrics (faithfulness and answer relevancy) are reference-free — they don't need ground truth answers, which makes them perfect for quick evaluation. Context precision and recall need ground truth (expected chunk IDs), so they require a curated eval dataset.

RAGAS gotcha: NaN scores. RAGAS uses an LLM internally to compute metrics. If the LLM returns invalid JSON or an unexpected format, RAGAS silently returns NaN instead of a score. Always check your results for NaN values before averaging. A common fix: use a more capable model (Claude Sonnet or GPT-4) as the RAGAS evaluator, and increase the retry count in the RAGAS config.

DeepEval: Test-Driven RAG Evaluation

If RAGAS is like running a Jupyter notebook, DeepEval is like writing unit tests. It integrates with pytest so you can run RAG evaluations as part of your CI/CD pipeline.

# pip install deepeval

from deepeval import assert_test
from deepeval.test_case import LLMTestCase
from deepeval.metrics import (
    FaithfulnessMetric,
    AnswerRelevancyMetric,
    ContextualRelevancyMetric,
)

def test_refund_policy_question():
    test_case = LLMTestCase(
        input="What is our refund policy?",
        actual_output="Refunds are available within 30 days of purchase.",
        retrieval_context=[
            "Our refund policy allows returns within 30 days of purchase. "
            "Items must be in original packaging."
        ],
    )

    faithfulness = FaithfulnessMetric(threshold=0.8)
    relevancy = AnswerRelevancyMetric(threshold=0.7)

    assert_test(test_case, [faithfulness, relevancy])

Run it with pytest test_rag.py and you get pass/fail results just like any other test. This is powerful for regression testing — if a code change breaks your RAG quality, your CI pipeline catches it before it ships.

When to use which: Use RAGAS for exploratory evaluation — "how good is my system right now?" Use DeepEval for automated testing — "did this change make things worse?" Use both in a mature RAG project: RAGAS for experimentation, DeepEval for CI/CD quality gates.

Debugging Retrieval Failures

Your evaluation scores show low context precision or recall. Now what? Here's a systematic debugging approach:

Start by inspecting what was actually retrieved vs what should have been. Print each retrieved chunk with its score, mark whether it was expected, and list any missed chunks. This immediately shows whether the problem is wrong results or missing results.

Common Retrieval Failure Patterns

Symptom Likely Cause Fix
Right topic, wrong details Chunks too large — the answer is diluted Smaller chunks or parent-child retrieval
Completely irrelevant results Vocabulary mismatch between query and docs Hybrid search (add BM25) or query rewriting
Right chunk at position 8-10 Ranking issue — relevant chunk scored low Add a re-ranker (cross-encoder)
Relevant chunk not in index at all Ingestion bug — document wasn't indexed Check your ingestion pipeline for errors
Good for simple queries, bad for complex ones Multi-hop questions need decomposition Multi-query retrieval or step-back prompting

Debugging Generation Failures

The right chunks were retrieved, but the answer is still wrong. Here's how to diagnose:

The Context Was Retrieved but Ignored

The LLM generated an answer from its own knowledge instead of the provided context. This usually means your system prompt isn't strong enough.

Weak system prompt
Answer the user's question using the following context. Context: {chunks}
The LLM may use its own knowledge when the context doesn't seem sufficient.
Strong system prompt
Answer the user's question based ONLY on the provided context. If the context does not contain enough information to answer, say "I don't have enough information to answer this." Do NOT use any outside knowledge. Context: {chunks}
Explicit constraints force the LLM to stay grounded in the retrieved context.

The Context Was Contradictory

Different chunks say different things (outdated docs, conflicting sources). The LLM picks one, and it might pick wrong. Fix: add metadata filtering to prefer recent documents, or use contextual compression to extract only the most authoritative source.

The Answer Confabulates Details

The context says "within 30 days" and the LLM says "within 30 business days." Subtle but wrong. This is the hardest failure to catch manually — it's why automated faithfulness scoring is so important.

A/B Testing RAG Configurations

Now you have metrics and an eval dataset. The real power is using them to compare configurations. Changed your chunk size from 500 to 300 tokens? Run both through the eval and compare.

def ab_test_rag(eval_dataset: list[dict], pipeline_a, pipeline_b,
                judge_client) -> dict:
    """Compare two RAG pipeline configurations."""
    scores_a = {"faithfulness": [], "relevance": [], "retrieval_precision": []}
    scores_b = {"faithfulness": [], "relevance": [], "retrieval_precision": []}

    for item in eval_dataset:
        question = item["question"]
        expected_chunks = set(item.get("expected_chunk_ids", []))

        # Run both pipelines
        result_a = pipeline_a.query(question)
        result_b = pipeline_b.query(question)

        # Score retrieval
        if expected_chunks:
            prec_a = precision_at_k(result_a.chunk_ids, expected_chunks, k=5)
            prec_b = precision_at_k(result_b.chunk_ids, expected_chunks, k=5)
            scores_a["retrieval_precision"].append(prec_a)
            scores_b["retrieval_precision"].append(prec_b)

        # Score generation (LLM-as-judge)
        judge_a = judge_rag_response(question, result_a.answer, result_a.context)
        judge_b = judge_rag_response(question, result_b.answer, result_b.context)

        scores_a["faithfulness"].append(judge_a.get("faithfulness", 0))
        scores_b["faithfulness"].append(judge_b.get("faithfulness", 0))

    # Compare averages
    report = {}
    for metric in scores_a:
        avg_a = sum(scores_a[metric]) / len(scores_a[metric]) if scores_a[metric] else 0
        avg_b = sum(scores_b[metric]) / len(scores_b[metric]) if scores_b[metric] else 0
        report[metric] = {"pipeline_a": avg_a, "pipeline_b": avg_b,
                          "delta": avg_b - avg_a}
    return report

The evaluation loop in practice: Change one variable (chunk size, embedding model, retrieval strategy) → run the eval dataset through both configurations → compare scores → keep the winner → repeat. This is how you systematically improve a RAG system instead of guessing. Every improvement is backed by numbers, and you can explain exactly why one approach beats another.

Quality Thresholds: When Is Good Enough?

Use Case Faithfulness Relevance Context Recall
Internal knowledge base ≥ 0.80 ≥ 0.75 ≥ 0.70
Customer-facing chatbot ≥ 0.85 ≥ 0.80 ≥ 0.80
Legal / compliance ≥ 0.95 ≥ 0.90 ≥ 0.90
Medical / safety-critical ≥ 0.95 ≥ 0.90 ≥ 0.95

High retrieval precision doesn't guarantee good answers. You can retrieve 5 perfectly relevant chunks and still get a bad answer if: the LLM hallucinates details, the context is contradictory, or the question requires reasoning the LLM can't do. Always evaluate retrieval and generation separately. A drop in faithfulness with high retrieval scores means your problem is in the generation step, not retrieval. Treating them as one metric hides where things break.

Try It: Build a 20-Question Evaluation Suite

  1. Create your eval dataset: Write 20 questions that real users would ask of your document collection. For each, note which chunks should be retrieved and what the correct answer is.
  2. Mix question types: Include 5 simple factual questions, 5 that require combining information, 5 that use different wording than the docs, and 5 that should return "I don't know."
  3. Score with RAGAS: Run your eval dataset through RAGAS and get faithfulness, relevance, context precision, and context recall scores.
  4. Find your weakest metric: Which score is lowest? If it's context recall, focus on retrieval improvements. If it's faithfulness, focus on your prompt engineering.
  5. Run an A/B test: Pick one change (e.g., add query rewriting from Chapter 9) and compare scores before and after.

Your goal: get all four RAGAS metrics above 0.80. If any metric is below 0.70, that's where your biggest improvement opportunity is.
