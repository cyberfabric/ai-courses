Beyond Text: The Future of RAG

Multi-modal RAG, code retrieval, knowledge graphs, agentic RAG, and where this entire field is heading

The big idea: Everything you've learned about RAG — chunking, embedding, retrieving, generating — applies far beyond plain text. The same pattern powers image search, code assistants, database querying, and autonomous AI agents. This final chapter expands your mental model from "text Q&A" to "any data, any modality, any reasoning pattern."

In the last chapter, you learned to run RAG in production — managing costs, optimizing latency, securing access, and monitoring everything. Your text-based RAG system is solid.

But here's the thing: the real world isn't just text. Your company has architecture diagrams, database tables, code repositories, meeting recordings, and product screenshots. A RAG system that only understands text is like a librarian who can only read books but ignores maps, charts, and blueprints.

The good news? The core RAG pattern — retrieve relevant context, then generate — works for all of these. The retrieval mechanism changes, but the idea is the same. Let's explore what's possible.

Multimodal RAG: When Text Isn't Enough

Imagine you're building a support system for a hardware product. The manual has wiring diagrams, the FAQ has screenshots, and the troubleshooting guide has flowcharts. A text-only RAG system would miss all of that visual information — or worse, try to answer with text that says "see Figure 3" without actually seeing Figure 3.

Multimodal RAG indexes and retrieves images, diagrams, tables, and other non-text content alongside regular text. Think of it as giving your RAG system eyes in addition to its reading ability.

Where multimodal fits in the pipeline: The core pipeline stays the same — ingest, index, retrieve, generate. What changes is that the ingest step now extracts images and tables (not just text), the index step stores multimodal embeddings, and the retrieve step can return a mix of text chunks and visual content. The generate step uses a vision-capable LLM (like GPT-4o or Claude) to reason over both.

Three Approaches to Multimodal RAG

Approach How It Works Best For
Caption & Embed Use a vision model to generate text descriptions of images, then embed the captions alongside text chunks Simple setups; works with any text embedding model
Native Vision Embeddings Use multimodal embedding models (like Cohere Embed 4 or Voyage Multimodal) that natively embed both text and images into the same vector space Rich visual content where captions lose too much detail
Late Interaction (ColPali) Treat entire document pages as images and use vision-language models to match queries against page screenshots Complex layouts (PDFs with tables, charts, mixed formatting)

The "caption and embed" approach is the easiest to implement and works surprisingly well for many use cases:

import anthropic
from pathlib import Path
import base64

client = anthropic.Anthropic()

def caption_image(image_path: str) -> str:
    """Use a vision model to generate a text caption for an image."""
    image_data = base64.standard_b64encode(
        Path(image_path).read_bytes()
    ).decode("utf-8")

    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=300,
        messages=[{
            "role": "user",
            "content": [
                {"type": "image", "source": {
                    "type": "base64",
                    "media_type": "image/png",
                    "data": image_data
                }},
                {"type": "text", "text":
                    "Describe this image in detail for a "
                    "technical knowledge base. Include all "
                    "text, labels, relationships, and data "
                    "visible in the image."
                }
            ]
        }]
    )
    return response.content[0].text


# Index: caption each image, then embed the caption
for img in document_images:
    caption = caption_image(img.path)
    embedding = embed(caption)  # Same embedding model as text
    vector_db.upsert(
        id=img.id,
        vector=embedding,
        metadata={
            "type": "image",
            "caption": caption,
            "source_page": img.page_number,
            "image_path": img.path
        }
    )

The ColPali approach is a game-changer for PDFs. Instead of extracting text from PDFs (which loses tables, charts, and layout), ColPali-style models treat each page as an image and create embeddings directly from the visual representation. Tools like RAG-Anything (open-sourced in 2025) combine this with knowledge graph construction to handle scanned PDFs, embedded images, LaTeX equations, and charts — all without traditional text extraction.

Gotcha: Multimodal embeddings aren't interchangeable. You can't mix text embeddings from one model with image embeddings from another — they live in different vector spaces. If you use the "native vision embeddings" approach, every piece of content (text and images) must be embedded with the same multimodal model. The "caption and embed" approach avoids this problem by converting everything to text first.

Code RAG: How Your AI Coding Assistant Works

Every time you use Cursor, GitHub Copilot, or Claude Code, you're using Code RAG. These tools don't just predict the next token — they retrieve relevant code from your project and feed it to the LLM as context. That's why Copilot's suggestions get dramatically better when you have related files open.

Code retrieval is RAG with a twist. Instead of searching a document corpus, code RAG searches your codebase — function definitions, type signatures, import graphs, docstrings, and test files. The retrieval signals are different too: code structure (AST), symbol references, and file proximity matter more than pure semantic similarity. When you type a function call, the assistant retrieves the function's definition, its callers, and related tests — not just "semantically similar" code snippets.

How Code Assistants Retrieve Context

1
Codebase Indexing

The tool indexes your repository: file tree, symbols, function signatures, import graphs. Some tools use embeddings; others use AST-based indexing.

↓
2
Context Gathering

When you type, the tool collects signals: cursor position, open files, recent edits, imported symbols, the current function's signature.

↓
3
Retrieval

Using those signals, it retrieves the most relevant code: the function you're calling, related type definitions, similar patterns elsewhere in the codebase.

↓
4
Fill-in-the-Middle (FIM)

The LLM receives the code before your cursor (prefix), after your cursor (suffix), and the retrieved context. It generates the code that fits in the middle.

# Simplified Code RAG — what happens behind the scenes
# when your AI assistant suggests a function body

def build_code_context(cursor_position, project):
    """Gather context for a code completion request."""
    context_chunks = []

    # 1. Current file — code before and after cursor
    current_file = project.get_file(cursor_position.file)
    prefix = current_file[:cursor_position.offset]
    suffix = current_file[cursor_position.offset:]

    # 2. Open tabs — the user chose to open these, so they're likely relevant
    for tab in project.open_tabs:
        if tab != current_file:
            context_chunks.append({
                "source": tab.path,
                "content": tab.content[:2000],  # First 2K chars
                "relevance": "open_tab"
            })

    # 3. Symbol retrieval — find definitions of symbols used near cursor
    symbols_near_cursor = extract_symbols(prefix[-500:])
    for symbol in symbols_near_cursor:
        definition = project.find_definition(symbol)
        if definition:
            context_chunks.append({
                "source": definition.file,
                "content": definition.code,
                "relevance": "symbol_definition"
            })

    # 4. Semantic retrieval — find similar code patterns
    query = prefix[-200:]  # Use recent code as search query
    similar = project.vector_index.search(query, top_k=3)
    context_chunks.extend(similar)

    return prefix, suffix, context_chunks

This is why "open the relevant file" improves AI suggestions. Most code assistants use open tabs as a strong retrieval signal. If you're working on a REST handler and the data model file is open, the assistant can see the field names and types. Close that tab, and it has to guess — or retrieve less relevant code from the index.

Structured Data RAG: Talking to Databases

What if your data lives in a PostgreSQL database, not in documents? You can't chunk a SQL table into paragraphs. This is where Text-to-SQL RAG comes in: the user asks a natural language question, and the system converts it into a SQL query, executes it, and generates a human-readable answer.

Think of it like having a data analyst on call. Instead of writing SELECT department, AVG(salary) FROM employees GROUP BY department, the user asks: "What's the average salary by department?"

Two Approaches to Database RAG

Approach How It Works Best For
Text-to-SQL LLM generates a SQL query from the natural language question. The query runs against the database. Results are fed back to the LLM for a natural language answer. Aggregations, filters, computations, joins — anything SQL is good at
Embed & Retrieve Rows Embed database rows (or their text representations) into a vector store. Retrieve similar rows via semantic search. Free-text fields, conceptual queries ("find customers like X") 
# Text-to-SQL RAG — simplified pipeline

def text_to_sql_rag(question: str, db_schema: str, llm, db_conn):
    """Convert a natural language question to SQL, execute, and explain."""

    # Step 1: Generate SQL from the question + schema
    sql_prompt = f"""You are a SQL expert. Given this database schema:

{db_schema}

Convert this question to a SQL query:
Question: {question}

Return ONLY the SQL query, no explanation."""

    sql_query = llm.generate(sql_prompt).strip()

    # Step 2: Execute the query (with safety checks!)
    if not sql_query.upper().startswith("SELECT"):
        return "I can only run read-only queries for safety."

    results = db_conn.execute(sql_query).fetchall()

    # Step 3: Generate a natural language answer
    answer_prompt = f"""Based on this data:

Question: {question}
SQL Query: {sql_query}
Results: {results[:50]}  # Limit rows sent to LLM

Provide a clear, concise answer to the question.
Include the key numbers and cite the data."""

    return llm.generate(answer_prompt)

Never let an LLM run arbitrary SQL against your production database. Always: (1) whitelist only SELECT statements, (2) use a read-only database connection, (3) set query timeouts, and (4) limit result set sizes. LLMs can generate valid but destructive queries — DROP TABLE is syntactically correct too.

Knowledge Graphs + RAG: GraphRAG

Standard vector RAG treats each chunk as an isolated island. It finds the most similar chunks but doesn't understand how concepts relate to each other. Ask "How does the marketing team's strategy connect to the engineering roadmap?" and vector search might retrieve chunks about marketing and chunks about engineering — but it can't trace the relationship between them.

GraphRAG solves this by building a knowledge graph from your documents. Instead of just storing chunks, it extracts entities (people, products, concepts) and relationships ("manages," "depends on," "competes with"), then uses that structure for retrieval.

How GraphRAG works (the Microsoft approach): First, the system uses an LLM to extract entities and relationships from your text. Then it builds a graph and detects "communities" — clusters of closely related entities. For each community, it generates a summary. When you ask a question, GraphRAG can traverse the graph to find related entities, use community summaries for high-level questions, or combine graph traversal with vector search for the best of both worlds. In benchmarks, it shows 3.4x better accuracy than traditional vector RAG for questions that require connecting information across multiple documents.

# GraphRAG — conceptual pipeline (using the Microsoft graphrag library)
# pip install graphrag

# Step 1: Extract entities and relationships from documents
# GraphRAG uses an LLM to identify entities and how they connect

"""
Input text: "Alice manages the payments team. The payments team
depends on the auth service maintained by Bob's infrastructure team."

Extracted entities:
  - Alice (person, role: manager)
  - Payments Team (team)
  - Auth Service (service)
  - Bob (person)
  - Infrastructure Team (team)

Extracted relationships:
  - Alice --manages--> Payments Team
  - Payments Team --depends_on--> Auth Service
  - Auth Service --maintained_by--> Infrastructure Team
  - Bob --member_of--> Infrastructure Team
"""

# Step 2: Query the graph
# "Who is affected if the auth service goes down?"
# Graph traversal: Auth Service <-- depends_on -- Payments Team
#                  Payments Team <-- manages -- Alice
# Answer: "The payments team (managed by Alice) depends on the
#          auth service, so they would be directly affected."

When to Use GraphRAG vs Vector RAG

Question Type Vector RAG GraphRAG
"What is our refund policy?" Great — find the policy chunk Overkill
"How are teams X and Y connected?" Struggles — no relationship awareness Excellent — traverses relationships
"Summarize all Q1 initiatives" Retrieves random chunks Uses community summaries
"Find docs about authentication" Great — semantic similarity Good, but slower

Gotcha: GraphRAG is expensive to build. Extracting entities and relationships requires sending every chunk through an LLM, which can cost 10–50x more than just embedding the text. For a 10,000-document corpus, vector indexing might cost $5 in embedding API calls, while GraphRAG entity extraction could cost $50–$200 in LLM calls. Use GraphRAG when your questions genuinely require relationship reasoning — not as a default replacement for vector search.

Agentic RAG: When the System Thinks for Itself

Everything we've built so far follows a fixed pipeline: embed the question, retrieve chunks, generate an answer. But what if the question is ambiguous? What if the first retrieval misses? What if the answer requires multiple searches?

Agentic RAG replaces the fixed pipeline with an autonomous agent that decides what to do at each step. The agent can rephrase the query, choose which index to search, evaluate whether retrieved chunks are relevant, search again if needed, and combine information from multiple sources.

The shift from pipeline to agent: In traditional RAG, the retrieval strategy is hard-coded: embed, search, return top-k. In agentic RAG, the agent dynamically chooses its retrieval strategy based on the question. A factual question might trigger a single vector search. A complex question might trigger keyword search first, then semantic search, then a re-ranking step. The agent observes the results at each step and decides what to do next — following a ReAct loop (Reason → Act → Observe → Repeat).

# Agentic RAG — the agent decides its own retrieval strategy

from anthropic import Anthropic

client = Anthropic()

tools = [
    {
        "name": "keyword_search",
        "description": "Search by exact keyword match (BM25). "
                       "Best for names, IDs, error codes, jargon.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string"},
                "collection": {"type": "string",
                               "enum": ["docs", "code", "tickets"]}
            },
            "required": ["query"]
        }
    },
    {
        "name": "semantic_search",
        "description": "Search by meaning similarity. "
                       "Best for conceptual questions.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string"},
                "collection": {"type": "string",
                               "enum": ["docs", "code", "tickets"]}
            },
            "required": ["query"]
        }
    },
    {
        "name": "sql_query",
        "description": "Query structured data. "
                       "Best for numbers, aggregations, filters.",
        "input_schema": {
            "type": "object",
            "properties": {
                "question": {"type": "string"}
            },
            "required": ["question"]
        }
    }
]

# The agent decides which tools to call and in what order
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    system="You are a research assistant with access to multiple "
           "data sources. Use the available tools to find information "
           "needed to answer the user's question. Search multiple "
           "sources if needed. If results seem incomplete, try a "
           "different search strategy.",
    tools=tools,
    messages=[{"role": "user", "content": user_question}]
)
# The agent might: semantic_search("docs") -> read results ->
# keyword_search("tickets") -> combine -> answer

RAG as one tool among many. In a fully agentic system, RAG isn't the whole application — it's one tool the agent can use. The agent might also call APIs, run calculations, query databases, or browse the web. The agent decides when retrieval is needed, and it might not need it at all for some questions. This is the pattern behind tools like Claude Code, Cursor's agent mode, and enterprise AI assistants.

Conversational RAG: Handling Follow-Up Questions

Real users don't ask isolated questions. They have conversations: "What's our refund policy?" followed by "What about for international orders?" followed by "How long does that take?" Each follow-up depends on the previous context.

The challenge: "How long does that take?" means nothing on its own. Your retrieval system needs to know that "that" refers to "international refunds."

The Solution: Query Contextualization

# Conversational RAG — rewrite follow-up questions using chat history

def contextualize_query(chat_history: list, new_question: str, llm) -> str:
    """Rewrite a follow-up question to be self-contained."""
    prompt = f"""Given this conversation history:

{format_history(chat_history)}

The user's latest question is: "{new_question}"

Rewrite this question to be fully self-contained — it should
make sense without the conversation history. If the question
is already self-contained, return it unchanged.

Return ONLY the rewritten question."""

    return llm.generate(prompt)

# Example:
# History: [("What's our refund policy?", "Our refund policy is..."),
#           ("What about international orders?", "For international...")]
# New question: "How long does that take?"
# Rewritten: "How long does the refund process take for international orders?"

# Now search with the rewritten query — much better retrieval!

Gotcha: Don't stuff the entire chat history into every retrieval query. If you embed the full conversation as your search query, the embedding will be diluted by irrelevant earlier turns. Instead, use the LLM to rewrite just the latest question into a self-contained query, then search with that. This keeps retrieval focused and accurate.

When NOT to Use RAG

RAG isn't always the answer. Here are the alternatives and when they win:

Alternative When It Wins Over RAG Tradeoff
Long Context Windows Your entire knowledge base fits in the context window (e.g., Claude's 200K tokens). Just paste everything in — no retrieval needed. Expensive per query. Doesn't scale beyond the context limit. Slower.
Fine-Tuning You need the model to adopt a specific style, follow strict formatting rules, or internalize domain jargon. Knowledge that changes rarely. Expensive to train. Can't update knowledge without retraining. Still hallucinates.
Prompt Caching You have a stable set of reference documents that every query needs. Cache them as a system prompt prefix. Limited to what fits in the prompt. Not dynamic.
Direct API/Tool Calls The answer is in a structured system (database, API, calendar). Don't embed it — just call the API. Requires API access and integration work.

The decision tree is simple: Can you fit all the relevant data in the context window and afford the cost? → Use long context. Is the knowledge stable and about style/format, not facts? → Fine-tune. Is the data in a structured system with an API? → Call the API. For everything else — large, changing, unstructured knowledge bases — RAG is still the best pattern.
