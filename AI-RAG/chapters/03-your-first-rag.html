Chapter 3

Your First RAG System in 30 Minutes

Build a working RAG pipeline from scratch with just Python and an API key

Time to Build

Enough theory. In Chapters 1 and 2, you learned why RAG exists and how it works. Now you're going to build one — a complete, working RAG system that answers questions from your own documents.

No frameworks, no magic libraries, just Python, an API key, and about 60 lines of code.

What We're Building

A command-line Q&A bot that reads text files (a company FAQ), indexes them with embeddings, and answers questions by retrieving the most relevant passages. No vector database — just Python lists and numpy. Simple, transparent, and educational.

Setup: What You Need

You need three things:

  • Python 3.9+ — you almost certainly have this already
  • An OpenAI API key — for creating embeddings ($0.02 per million tokens — pennies)
  • An Anthropic API key — for the LLM generation step (or use OpenAI for both)

Install the dependencies:

pip install openai anthropic numpy

That's it. Three packages. No LangChain, no LlamaIndex, no vector database. We're building from scratch so you understand every line.

You can use OpenAI for both embeddings and generation if you prefer. We use OpenAI for embeddings (excellent and cheap) and Claude for generation (great at following instructions to answer from context). Mix and match as you like.

Step 2: Load and Chunk the Documents

Create rag.py and start with loading and chunking:

import os
import numpy as np
from openai import OpenAI
import anthropic

# Initialize API clients
openai_client = OpenAI()   # uses OPENAI_API_KEY env var
claude_client = anthropic.Anthropic()  # uses ANTHROPIC_API_KEY

# --- Load documents ---
def load_documents(directory):
    documents = []
    for filename in sorted(os.listdir(directory)):
        if filename.endswith('.txt'):
            path = os.path.join(directory, filename)
            with open(path, 'r') as f:
                documents.append({
                    'content': f.read(),
                    'source': filename
                })
    return documents

# --- Chunk documents ---
def chunk_document(doc, chunk_size=500, overlap=50):
    text = doc['content']
    chunks = []
    start = 0
    while start < len(text):
        end = min(start + chunk_size, len(text))
        chunks.append({
            'text': text[start:end].strip(),
            'source': doc['source']
        })
        start += chunk_size - overlap
    return chunks

We load text files and split them into 500-character chunks with 50 characters of overlap. In Chapter 6, we'll learn much smarter chunking. For now, this works.

Step 3: Create Embeddings

Turn each chunk into a vector — a list of numbers that captures its meaning:

# --- Create embeddings ---
def get_embeddings(texts):
    response = openai_client.embeddings.create(
        model="text-embedding-3-small",
        input=texts
    )
    return [item.embedding for item in response.data]

One function, three lines. OpenAI's text-embedding-3-small returns a 1536-dimensional vector per input. Embedding our three sample documents costs a fraction of a cent.

Why text-embedding-3-small? It's the best balance of quality and cost for most RAG use cases. 1536 dimensions, up to 8,191 tokens per input, $0.02 per million tokens. For higher quality at 3x cost, there's text-embedding-3-large (3072 dimensions). We'll compare models in Chapter 4.

Step 4: Search with Cosine Similarity

Instead of a vector database, we use plain numpy to find the most similar chunks:

# --- Similarity search ---
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

def search(query, chunks, chunk_embeddings, top_k=3):
    query_embedding = get_embeddings([query])[0]

    similarities = [
        cosine_similarity(query_embedding, emb)
        for emb in chunk_embeddings
    ]

    top_indices = np.argsort(similarities)[-top_k:][::-1]

    return [{
        'text': chunks[idx]['text'],
        'source': chunks[idx]['source'],
        'score': similarities[idx]
    } for idx in top_indices]
Cosine similarity measures how close two vectors point in the same direction. A score of 1.0 means identical meaning, 0.0 means unrelated. For RAG, scores above 0.7 are usually strong matches. We're comparing the question against every chunk — this brute-force approach works for small datasets but doesn't scale to millions (that's why vector databases exist, Chapter 5).

Step 5: Build the Prompt and Generate

Take the retrieved chunks, build a prompt, and let the LLM answer:

# --- Build prompt and generate ---
def build_prompt(question, results):
    context_parts = []
    for r in results:
        context_parts.append(f"[Source: {r['source']}]\n{r['text']}")
    context = "\n\n---\n\n".join(context_parts)

    return f"""You are a helpful customer support assistant.
Answer the user's question based ONLY on the provided context.
If the context doesn't contain the answer, say
"I don't have that information in my knowledge base."
Always cite which source document(s) your answer comes from.

Context:
{context}

Question: {question}

Answer:"""

def ask(question, chunks, chunk_embeddings):
    # Retrieve relevant chunks
    results = search(question, chunks, chunk_embeddings)
    # Build augmented prompt and generate
    prompt = build_prompt(question, results)
    response = claude_client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.content[0].text, results
This is the complete RAG pipeline in one function. The ask() function does exactly what we described in Chapter 2: embed the question, retrieve similar chunks, build an augmented prompt, and generate an answer. Every production RAG system follows this same pattern. The rest of this course is about making each step better.

Step 6: Put It All Together

# --- Main: wire everything up ---
def main():
    print("Loading documents...")
    docs = load_documents("docs/")
    print(f"  Loaded {len(docs)} documents")

    print("Chunking...")
    all_chunks = []
    for doc in docs:
        all_chunks.extend(chunk_document(doc))
    print(f"  Created {len(all_chunks)} chunks")

    print("Creating embeddings...")
    texts = [c['text'] for c in all_chunks]
    embeddings = get_embeddings(texts)
    print(f"  Created {len(embeddings)} embeddings")
    print(f"  Dimensions: {len(embeddings[0])}")

    print("\nRAG system ready! Ask questions ('quit' to exit).\n")

    while True:
        question = input("You: ").strip()
        if question.lower() in ('quit', 'exit', 'q'):
            break
        if not question:
            continue

        answer, sources = ask(question, all_chunks, embeddings)
        print(f"\nAssistant: {answer}")
        print(f"\n  Sources:")
        for s in sources:
            print(f"    - {s['source']} (score: {s['score']:.3f})")
        print()

if __name__ == "__main__":
    main()

What Works and What Doesn't

Play with the system for a few minutes. You will notice some patterns:

What works well

  • Direct questions about documented topics get accurate, sourced answers
  • Paraphrased questions work because embeddings capture meaning, not just keywords
  • Out-of-scope questions get a proper "I don't know" response
  • Cross-document synthesis works when relevant chunks are retrieved from multiple files

What breaks

  • Questions about details split across chunk boundaries may miss information
  • Very specific keyword searches (like "SOC 2") may rank lower than conceptually related but lexically different chunks
  • Questions requiring reasoning across many chunks may get incomplete answers
Your chunking strategy is already showing its limits. Our naive 500-character split does not respect paragraph or section boundaries. A chunk might start mid-sentence or split a list item in half. This is the #1 thing we will improve in Chapter 6. Better chunking alone can boost answer quality by 20-30%.

What We Skipped (and Why It Matters)

This system works, but it cuts every corner. Here is what we deliberately skipped and where we will fix it:

What We Skipped Why It Matters Where We Fix It
Smart chunking Our fixed-size split breaks mid-sentence, losing context Chapter 6
Vector database numpy search is O(n) and doesn't persist; a real DB is O(log n) and durable Chapter 5
Hybrid search Pure semantic search misses exact keyword matches (product IDs, error codes) Chapter 7
Re-ranking The top-k results aren't always the best; a cross-encoder can reorder them Chapter 7
Evaluation "It seems to work" is not a metric. You need precision, recall, and faithfulness scores Chapter 10
Document loaders Real data lives in PDFs, HTML, Confluence, Notion — not clean .txt files Chapter 8
Don't be discouraged by what we skipped. You just built a working RAG system from scratch in under 100 lines of code. Every improvement from here is incremental — we are not throwing this away, we are making it better, one piece at a time.
Extend your RAG system. Now that you have a working system, try these experiments:
  1. Add your own documents: Drop a few .txt files from your own work (a README, a FAQ, meeting notes) into the docs/ directory and re-run. See how it handles your real data.
  2. Experiment with chunk size: Change chunk_size from 500 to 200 and then to 1000. Ask the same questions and compare answer quality. Which size works best?
  3. Change top_k: Modify the top_k parameter in the search function from 3 to 1, then to 5. How does the number of retrieved chunks affect answer quality?

These experiments build intuition you will use throughout the rest of the course.
