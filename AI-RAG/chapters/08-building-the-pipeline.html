Chapter 8

Building a Production RAG Pipeline

From notebook prototype to a system that handles real users, messy documents, and everything in between

Over the last four chapters, you've collected every building block: embeddings, vector databases, chunking strategies, and retrieval methods. You have all the LEGO bricks. Now it's time to build the actual house.

This chapter is about the jump from "it works in a Jupyter notebook" to "it works reliably for real users." That gap is bigger than most people expect. In production, your documents aren't clean text files — they're messy PDFs, HTML pages with boilerplate, and Confluence wikis with broken formatting.

The Big Idea

A production RAG system is two pipelines stitched together: an ingestion pipeline that turns messy documents into searchable chunks, and a query pipeline that turns user questions into grounded answers. Getting each one right — and connecting them cleanly — is what separates a demo from a product.

Two Pipelines, One System

Every RAG system, from a weekend prototype to an enterprise knowledge base, has the same two-phase architecture:

The RAG Pipeline Architecture

Ingestion pipeline (runs ahead of time, often on a schedule):

Load documents → Preprocess & clean → Chunk → Embed → Store in vector DB

Query pipeline (runs in real time, when a user asks a question):

Receive question → Embed question → Retrieve chunks → Build prompt → Generate answer → Return with sources

The ingestion pipeline is your kitchen prep — washing, chopping, and organizing ingredients before the restaurant opens. The query pipeline is the cook assembling a dish to order. Both matter, but if your prep is sloppy, no amount of cooking skill will save the meal.

Let's build each one, piece by piece.

The Ingestion Pipeline

The ingestion pipeline takes raw documents and turns them into searchable vectors. Here's the full flow:

1. Load

Read documents from their source

→
2. Preprocess

Clean, normalize, extract text

→
3. Chunk

Split into retrieval-sized pieces

→
4. Embed

Convert chunks to vectors

→
5. Store

Upsert into vector database

Sounds simple on paper. The devil is in step 1 and 2.

Document Loaders: The Messy Reality

In a tutorial, your documents are clean .txt files. In the real world, they're a chaotic mix of formats — each needing special handling. Here's how to tackle the most common ones:

PDFs — The Hardest Format

PDFs are the bane of every RAG engineer's existence. They look nice to humans but are a nightmare for machines — text can be out of order, tables are rendered as positioned elements, and headers are indistinguishable from body text.

import pymupdf  # PyMuPDF — fast, reliable PDF extraction

def load_pdf(file_path: str) -> list[dict]:
    """Extract text from a PDF, page by page."""
    doc = pymupdf.open(file_path)
    pages = []
    for page_num, page in enumerate(doc):
        text = page.get_text("text")  # Plain text extraction
        if text.strip():  # Skip blank pages
            pages.append({
                "content": text,
                "metadata": {
                    "source": file_path,
                    "page": page_num + 1,
                    "total_pages": len(doc)
                }
            })
    return pages

When PDFs Fight Back

For complex PDFs with tables, multi-column layouts, or embedded images, plain text extraction won't cut it. Use Unstructured.io or OpenDataLoader — they understand document structure and can extract tables as Markdown, preserve reading order, and even handle scanned documents with OCR. In 2026, these tools have gotten remarkably good at producing clean, structured output from messy PDFs.

HTML — Strip the Boilerplate

from bs4 import BeautifulSoup

def load_html(file_path: str) -> dict:
    """Extract meaningful content from HTML, ignoring nav, footer, ads."""
    with open(file_path, "r", encoding="utf-8") as f:
        soup = BeautifulSoup(f.read(), "html.parser")

    # Remove non-content elements
    for tag in soup.select("nav, footer, header, script, style, .sidebar, .ads"):
        tag.decompose()

    # Extract text from the main content area
    main = soup.select_one("main, article, .content, #content")
    text = main.get_text(separator="\n", strip=True) if main else soup.get_text(separator="\n", strip=True)

    return {
        "content": text,
        "metadata": {
            "source": file_path,
            "title": soup.title.string if soup.title else ""
        }
    }

Markdown and Plain Text — The Easy Ones

Markdown is straightforward — read the file, optionally extract the title from the first # heading. Plain text is even simpler. These are the formats where you can skip straight to chunking.

Enterprise Sources: Confluence, Notion, Google Docs

Enterprise data lives in wikis and SaaS tools. The pattern is always: export or API call (pull as HTML/Markdown) → clean (strip platform boilerplate) → normalize (convert to a consistent format). LangChain and LlamaIndex ship with dozens of connectors for these sources — don't reinvent them.

Preprocessing: Cleaning the Mess

Raw extracted text is rarely ready for chunking. You need a preprocessing step that cleans, normalizes, and enriches the text:

import re

def preprocess(text: str) -> str:
    """Clean extracted text for chunking."""
    text = re.sub(r"\n{3,}", "\n\n", text)          # Collapse blank lines
    text = re.sub(r"[\u200b\u200c\u200d\ufeff]", "", text)  # Remove invisible chars
    text = re.sub(r"[^\S\n]+", " ", text)            # Normalize whitespace
    text = re.sub(r"\n\s*Page \d+ of \d+\s*\n", "\n", text)  # Remove page numbers
    return "\n".join(line.strip() for line in text.split("\n")).strip()

Tables Are a Special Kind of Pain

When you extract text from a PDF table, the columns often get smashed together or read in the wrong order. "Revenue | Q1 | $10M" might become "Revenue Q1 $10M" or worse, "Revenue $10M Q1". If your documents contain important tables, use a table-aware parser like Unstructured.io or PyMuPDF's table extraction, and convert tables to Markdown format before chunking. A Markdown table preserves structure that a vector search can actually work with.

Putting the Ingestion Pipeline Together

Here's a complete ingestion pipeline that handles multiple file types, preprocesses, chunks, embeds, and stores:

import os
from pathlib import Path
from openai import OpenAI
import chromadb
from langchain_text_splitters import RecursiveCharacterTextSplitter

client = OpenAI()
chroma = chromadb.PersistentClient(path="./chroma_db")
collection = chroma.get_or_create_collection("my_docs")

# Configure chunking
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", ". ", " ", ""]
)

def ingest_directory(docs_dir: str):
    """Ingest all supported documents from a directory."""
    loaders = {
        ".pdf": load_pdf,
        ".html": load_html,
        ".md": load_markdown,
        ".txt": lambda p: {"content": Path(p).read_text(), "metadata": {"source": p}}
    }

    for file_path in Path(docs_dir).rglob("*"):
        ext = file_path.suffix.lower()
        if ext not in loaders:
            continue

        print(f"Processing: {file_path}")

        # Load — returns a dict or list of dicts (PDFs return per-page)
        result = loaders[ext](str(file_path))
        docs = result if isinstance(result, list) else [result]

        for doc in docs:
            # Preprocess
            clean_text = preprocess(doc["content"])
            if not clean_text:
                continue

            # Chunk
            chunks = splitter.split_text(clean_text)

            # Embed all chunks in one batch
            response = client.embeddings.create(
                model="text-embedding-3-small",
                input=chunks
            )
            embeddings = [e.embedding for e in response.data]

            # Store with metadata
            ids = [f"{file_path.stem}_chunk_{i}" for i in range(len(chunks))]
            metadatas = [{**doc["metadata"], "chunk_index": i} for i in range(len(chunks))]

            collection.upsert(
                ids=ids,
                embeddings=embeddings,
                documents=chunks,
                metadatas=metadatas
            )

    print(f"Ingestion complete. Total chunks: {collection.count()}")

# Run it
ingest_directory("./documents")

Batch Your Embedding Calls

Embedding APIs accept batches of text. Instead of embedding one chunk at a time (slow, expensive in API overhead), collect all chunks from a document and embed them in a single call. OpenAI's text-embedding-3-small accepts up to 2048 inputs per batch. This alone can make your ingestion 10-50x faster.

The Query Pipeline

The query pipeline runs in real time — a user asks a question, and your system needs to return a grounded answer in under 2 seconds. Here's the full flow:

1. Question

User asks something

→
2. Embed

Convert question to vector

→
3. Retrieve

Find top-K similar chunks

→
4. Build Prompt

Assemble context + question

→
5. Generate

LLM produces grounded answer

Let's implement it:

import anthropic

llm = anthropic.Anthropic()

def query(question: str, top_k: int = 5) -> dict:
    """Run the full query pipeline: embed, retrieve, generate."""

    # Step 1: Embed the question
    q_embedding = client.embeddings.create(
        model="text-embedding-3-small",
        input=question
    ).data[0].embedding

    # Step 2: Retrieve similar chunks
    results = collection.query(
        query_embeddings=[q_embedding],
        n_results=top_k
    )
    chunks = results["documents"][0]
    sources = results["metadatas"][0]

    # Step 3: Build the prompt with context
    context = "\n\n---\n\n".join(
        f"[Source: {s.get('source', 'unknown')}, chunk {s.get('chunk_index', '?')}]\n{chunk}"
        for chunk, s in zip(chunks, sources)
    )

    # Step 4: Generate a grounded answer
    response = llm.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        system=RAG_SYSTEM_PROMPT,
        messages=[{
            "role": "user",
            "content": f"Context:\n{context}\n\nQuestion: {question}"
        }]
    )

    return {
        "answer": response.content[0].text,
        "sources": sources
    }

Prompt Engineering for RAG

The system prompt is the most underrated part of a RAG pipeline. A bad system prompt lets the LLM hallucinate freely. A good one enforces grounding — the model answers only from the provided context.

Context Engineering > Prompt Engineering

In 2026, the RAG community talks about context engineering rather than prompt engineering. The insight: what matters most isn't how cleverly you phrase your instructions — it's the quality and relevance of the context you put in front of the model. A mediocre prompt with excellent context beats a brilliant prompt with garbage context every time. Focus your effort on retrieval quality first, then tune the prompt.

That said, the system prompt still matters. Here's one that works well in production:

Production RAG System Prompt

RAG_SYSTEM_PROMPT = """You are a helpful assistant that answers
questions based on the provided context.

RULES:
1. Answer ONLY based on the provided context. Do not use
   prior knowledge.
2. If the context does not contain enough information to
   answer the question, say: "I don't have enough
   information in the available documents to answer this."
3. When you use information from the context, cite the
   source using [Source: filename] format.
4. Be concise but complete. Don't pad your answer.
5. If the context contains contradictory information,
   acknowledge the contradiction and cite both sources.
"""

Without these rules, a generic prompt like "You are a helpful assistant" lets the LLM fill gaps from its training data — you'll get confident-sounding answers with no basis in your actual documents.

Context Placement Matters

Where you put the retrieved context in the prompt affects quality. Research consistently shows: put the context before the question, not after. LLMs process text sequentially, and having the context fresh in their "working memory" when they encounter the question produces more grounded answers. The format Context: [chunks]\n\nQuestion: [user's question] outperforms Question: [question]\n\nContext: [chunks].

Source Attribution: Trust Through Transparency

Users need to verify your system's answers. Source attribution turns "the AI said so" into "the AI said so, and here's where it found that information."

There are two levels: document-level ("Based on refund-policy.pdf") and chunk-level ("refund-policy.pdf, page 3, section 'Enterprise Accounts'"). The key is storing rich metadata during ingestion — source path, page number, section heading, last-updated date. Then when the LLM cites a source, your frontend can link directly to the original document or even highlight the specific passage. This is the difference between a RAG system people tolerate and one they trust.

Streaming Responses: UX Matters

LLM generation is the slowest part of the query pipeline — often 1-3 seconds before the full answer is ready. Streaming shows words as they're generated, turning a frustrating wait into a fluid experience.

# The key change: use messages.stream() instead of messages.create()
with llm.messages.stream(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    system=RAG_SYSTEM_PROMPT,
    messages=[{
        "role": "user",
        "content": f"Context:\n{context}\n\nQuestion: {question}"
    }]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)

Streaming + Source Attribution Is Tricky

When you stream the answer, the LLM hasn't finished generating yet — so you can't show source citations until the stream completes. The common pattern: stream the answer text, then append a "Sources" section at the end once the full response is available. Some teams parse the LLM's inline citations (like [Source: file.pdf]) from the stream in real time and render them as footnotes — but this requires careful parsing of partial token streams.

Caching: Don't Repeat Expensive Work

Three types of caching can dramatically reduce cost and latency in a RAG pipeline:

Embedding Cache
Cache computed embeddings to avoid re-embedding the same text. Essential during ingestion when you re-process documents.
Query Cache
Cache full query results (question → answer). If someone asks the same question twice, return the cached answer instantly.
Semantic Cache
Cache by meaning, not exact match. "What's the refund policy?" and "How do refunds work?" hit the same cache entry.

A simple query cache: hash the question, store the result with a timestamp, return cached answers for duplicates within a TTL (e.g., 1 hour). Use Redis or even a Python dict for small-scale systems.

Semantic Caching in Practice

For semantic caching, embed the incoming question and check if any cached question vector is within a similarity threshold (cosine similarity > 0.95). Libraries like GPTCache handle this out of the box. Semantic caching can reduce LLM calls by 30-50% in scenarios with repetitive queries — like customer support where many people ask the same questions in different words.

Error Handling: When Things Go Wrong

Production systems encounter failures. Here are the three most common and how to handle each:

1. Empty Retrieval

The vector search returns no relevant chunks (or chunks with very low similarity scores).

# After retrieving, check distances (ChromaDB: lower = more similar)
results = collection.query(
    query_embeddings=[embed(question)],
    n_results=5,
    include=["documents", "metadatas", "distances"]
)

# Filter out low-relevance chunks
relevant = [
    (doc, meta) for doc, meta, dist
    in zip(results["documents"][0], results["metadatas"][0], results["distances"][0])
    if dist < 0.7  # Threshold — tune for your data
]

if not relevant:
    return {"answer": "I couldn't find relevant information. "
                      "Could you rephrase your question?", "sources": []}

2. API Failures

Embedding or LLM APIs can be slow or temporarily down. Use retries with exponential backoff — the tenacity library makes this trivial, or write a simple loop with time.sleep(base_delay * 2**attempt). Always set a max retry count (3 is typical) so you fail fast rather than hanging.

3. The LLM Ignores Context

When the LLM Goes Rogue

If your LLM consistently ignores the provided context, check these in order: (1) Is the system prompt explicitly requiring grounded answers? (2) Is the context placed before the question? (3) Are you retrieving too many chunks, burying the relevant ones in noise? (4) Is the question ambiguous enough that the model's prior knowledge is "more confident" than the context? Reducing top-K from 10 to 3-5 and strengthening the grounding instructions usually fixes this.

Pipeline Orchestration: Build vs Framework

You have three options for structuring your pipeline:

Choosing Your Orchestration Layer

Approach Best For Trade-offs
Custom Python Simple pipelines, full control, small teams You own every line — easy to debug, but you build everything yourself
LangChain / LangGraph Complex agent workflows, multi-step reasoning, tool use Rich ecosystem and connectors; LangGraph adds stateful graph-based orchestration for production
LlamaIndex Document-heavy RAG, complex data sources, knowledge indices Best-in-class data connectors and index types; Workflows API for production pipelines

Honest advice: start custom, adopt a framework when the complexity justifies it. If you have a straightforward query flow — custom Python is simpler and easier to debug. You've already seen the code in this chapter: it's about 150 lines total. Adopt LangChain/LangGraph when you need complex agent workflows. Adopt LlamaIndex when your data complexity grows — dozens of sources, multiple index types, sophisticated document relationships.

The Framework Trap

Don't pick a framework because it has the most GitHub stars. Pick it because it solves a problem you actually have. Many production RAG systems are 200 lines of Python with openai, chromadb, and anthropic — no framework needed. Frameworks add value when your pipeline becomes complex enough that you're reinventing their abstractions anyway. If you catch yourself writing your own retry logic, state management, and connector library, that's the signal.

The Complete Picture

Let's zoom out. Here's a production-ready pipeline architecture with all the components we've covered:

  • Ingestion: Document loaders for each format (PDF, HTML, MD) → Preprocessing & cleaning → Recursive chunking (500 tokens, 50 overlap) → Batch embedding → Upsert to vector DB with rich metadata
  • Query: Receive question → Embed question → Retrieve top-5 chunks → Score filter (drop low-relevance) → Build grounded prompt → Stream LLM response → Return answer with source citations
  • Caching: Embedding cache for ingestion → Query cache for repeat questions → Optional semantic cache for similar questions
  • Error handling: Retry with backoff for API calls → Fallback message for empty retrieval → Score threshold to filter noise
  • Monitoring: Log queries, retrieved chunks, and user feedback → Track retrieval quality over time

Every component here is something you already understand from previous chapters. The pipeline is just these pieces working together.
