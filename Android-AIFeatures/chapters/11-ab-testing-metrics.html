Chapter 11

A/B Testing & Measuring AI Features

Metrics that matter, controlled experiments, prompt testing, and knowing when to kill a feature

Why AI Features Need Different Metrics

Traditional features have binary outcomes: the button works or it doesn't. The page loads or it crashes. You measure success with simple metrics: crash rate, load time, conversion rate.

AI features have a quality gradient. The AI suggestion was helpful... somewhat helpful... useless... or actively harmful. There's no single "it works" threshold. And the same feature can be amazing for one user and terrible for another.

This means you need new ways to measure. This chapter gives you the complete measurement toolkit for AI features.

Scenario: Your team shipped an "AI Summarize" button in the note-taking app two months ago. The PM asks: "Is it working?" You check the analytics. 40% of users have tried it. But is that good? Are the summaries actually helpful? Are users editing them heavily? Is it worth the $200/month in API costs?

Without the right metrics, "Is it working?" is unanswerable. With the right metrics, you can say: "78% of summaries are accepted without edits, users who use it retain 23% better, and each summary costs $0.0003. It's our highest-ROI feature."

The AI Quality Scorecard

Every AI feature should be measured across five dimensions:

Accuracy How often is the AI right?
Latency How fast does it respond?
Cost How much per user per month?
Dimension How to Measure Target
Accuracy Acceptance rate, edit distance, user corrections Depends on feature (70-95%)
Latency P50 and P95 response time <2s for generation, <100ms for suggestions
Cost API spend per user per month <$0.05/user/month for most features
Satisfaction Thumbs up/down ratio, explicit feedback >80% positive
Coverage % of requests that get a useful response (not "I don't know") >90%

Pick your north star metric. For a summarization feature, it's probably acceptance rate (did the user use the summary?). For a chatbot, it's task completion rate (did the user accomplish what they came for?). For recommendations, it's click-through rate. One metric rules them all — the rest are supporting evidence.

The Edit Distance Trick

Edit distance is the most underrated AI metric. It tells you how much the user changed the AI's output:

fun calculateEditDistance(aiOutput: String, userFinal: String): Float {
    if (aiOutput == userFinal) return 0f // perfect — no edits
    val maxLen = maxOf(aiOutput.length, userFinal.length)
    if (maxLen == 0) return 0f
    return levenshteinDistance(aiOutput, userFinal).toFloat() / maxLen
}

// Log after user saves
analytics.log("ai_edit_distance", mapOf(
    "feature" to "summarize",
    "distance" to calculateEditDistance(aiSummary, userSavedText),
    "ai_model" to "gemini-flash"
))

An edit distance of 0 means the AI nailed it. Under 0.1 means minor tweaks. Over 0.5 means the user rewrote most of it — the AI wasn't helpful. Track the distribution, not just the average.

A/B Testing AI Features

The gold standard of measurement: randomly assign users to groups and compare outcomes.

The AI A/B Test Framework

For AI features, you'll run two types of A/B tests:

  1. Feature test: AI-enabled vs AI-disabled. Does the AI feature make users more productive, engaged, or satisfied? Assign 50% of users to see the AI feature and 50% to see the app without it. Measure task completion time, error rate, and retention.
  2. Quality test: Model A vs Model B. Same feature, different AI under the hood. Test Gemini Flash vs GPT-4o-mini for summarization, or prompt variant A vs prompt variant B. Measure quality metrics (acceptance rate, edit distance) and cost.

Critical rule: Only change one variable at a time. If you change the model and the prompt simultaneously, you can't tell which one affected the results.

Firebase A/B Testing Setup

Firebase Remote Config + A/B Testing gives you the infrastructure to run experiments on Android without app updates.

Recipe: A/B Testing Your AI Feature with Firebase

Goal: Test two different summarization prompts and measure which produces better user outcomes.

Step 1: Define the Remote Config parameter:

// In Firebase Console: create parameter "summary_prompt_variant"
// Default value: "A"
// A/B test: 50% get "A", 50% get "B"

// In your app, fetch the variant:
val remoteConfig = Firebase.remoteConfig
val promptVariant = remoteConfig.getString("summary_prompt_variant")

Step 2: Use the variant to select the prompt:

fun getSummaryPrompt(text: String, variant: String): String = when (variant) {
    "A" -> "Summarize in 3 bullet points:\n$text"
    "B" -> """Summarize the following text.
              Format: exactly 3 bullet points.
              Each bullet: one sentence, max 20 words.
              Focus on decisions and action items.
              Text: $text""".trimIndent()
    else -> getSummaryPrompt(text, "A") // fallback
}

Step 3: Log the outcome events that Firebase will measure:

// When user taps "Summarize"
Firebase.analytics.logEvent("ai_summary_requested") {
    param("variant", promptVariant)
}

// When the summary is shown
Firebase.analytics.logEvent("ai_summary_displayed") {
    param("variant", promptVariant)
    param("latency_ms", latency)
}

// When user accepts the summary (closes without editing)
Firebase.analytics.logEvent("ai_summary_accepted") {
    param("variant", promptVariant)
    param("edit_distance", editDistance)
}

// When user dismisses the summary
Firebase.analytics.logEvent("ai_summary_dismissed") {
    param("variant", promptVariant)
}

Step 4: In Firebase Console, set up the A/B test with "ai_summary_accepted" as the primary success event. Firebase calculates statistical significance automatically and tells you which variant wins.

Wait for statistical significance. Don't peek at results after 2 days and declare a winner. Firebase will tell you when the results are statistically significant (typically 95% confidence). For AI features, this usually takes 1-2 weeks with a few thousand users.

Prompt A/B Testing

Changing a few words in your prompt can dramatically change output quality. Prompt A/B testing is the fastest way to improve AI features without changing any code.

Prompt Variant A

Summarize in 3 bullet points.

Simple, short. Works well for straightforward content. May produce inconsistent format for complex documents.

Prompt Variant B

Summarize the following text in exactly 3 bullet points. Each bullet should be one concise sentence (max 20 words). Focus on action items and decisions.

Detailed constraints. More consistent output format. May be over-constrained for creative content.

Things worth testing in prompts:

  • Output format — "3 bullets" vs "1 paragraph" vs "key: value pairs"
  • Length constraints — "under 50 words" vs "2-3 sentences" vs unconstrained
  • Focus instructions — "focus on action items" vs "focus on key facts" vs "focus on conclusions"
  • Tone — "professional" vs "casual" vs "concise"
  • Examples — Including a sample output in the prompt vs not including one

Cost Monitoring and Budgeting

AI features have a running cost that traditional features don't. You need to track it like you track server costs.

Setting Cost Alerts

// Check costs periodically (e.g., in a WorkManager task)
fun checkCostAlerts() {
    val todayCost = aiUsageDao.getTotalCostSince(todayStart())
    val monthCost = aiUsageDao.getTotalCostSince(monthStart())

    if (todayCost > DAILY_BUDGET) {
        alertOps("AI daily budget exceeded: $${"%.2f".format(todayCost)}")
        enableRateLimiting() // reduce AI availability
    }
    if (monthCost > MONTHLY_BUDGET * 0.8f) {
        alertOps("AI monthly budget at 80%: $${"%.2f".format(monthCost)}")
    }
}

Set the budget alert before you launch, not after the first bill. A viral moment can 10x your traffic overnight. Without a circuit breaker, a $200/month feature becomes a $2,000 surprise. Chapter 12 covers rate limiting in detail.

Quality Monitoring in Production

You can't manually review every AI response. But you can sample and automate quality checks:

Automated Quality Gates

  • Response length check: Is the summary within expected bounds? (Too short = empty response; too long = model ignored constraints)
  • Format compliance: If you asked for 3 bullets, did you get 3 bullets? Parse and validate
  • Language check: Is the response in the expected language?
  • Toxicity scoring: Run responses through a safety classifier before showing to users
  • Latency tracking: P50, P95, P99 response times. Alert if P95 exceeds your budget

Human Review Sampling

Log a random 1-5% of AI inputs/outputs (with user consent and privacy controls) for periodic human review. This is how you catch subtle quality issues that automated checks miss:

The Weekly AI Quality Review

Every week, review 50-100 sampled AI interactions. Look for: hallucinations (facts not in the source), tone issues (too formal, too casual), format drift (model started ignoring your constraints), and edge cases (unusual inputs that produce bad outputs). This 30-minute ritual catches problems before they affect thousands of users.

User Feedback Systems

Explicit feedback (thumbs up/down) and implicit feedback (acceptance/dismissal) are both valuable. Here's how to implement them without causing survey fatigue:

The Feedback Hierarchy

Signal Effort for User Data Quality When to Use
Implicit (accept/dismiss) Zero Good Every interaction
Edit distance Zero Great Every editable output
Thumbs up/down Low Great Important outputs (summaries, long generations)
"Was this helpful?" Medium Good Weekly prompt, not every interaction
Free-text feedback High Excellent Only on "thumbs down" follow-up

The golden rule: collect implicit signals from every interaction, explicit feedback from important ones, and detailed feedback only when something went wrong.

When to Kill an AI Feature

Sometimes the honest answer is: this feature isn't working. Knowing when to sunset an AI feature is as important as knowing when to ship one.

  • Low adoption: Less than 10% of eligible users try it after 4 weeks of availability
  • Low retention: Users try it once and never come back. One-time trial rate >70%
  • High cost, low value: API costs exceed $0.10/user/month with no measurable impact on retention or revenue
  • Quality plateau: You've iterated on prompts and models 5+ times with no meaningful quality improvement
  • Negative feedback: Thumbs-down ratio consistently above 30%

Killing a feature isn't failure — it's good product management. You learned what doesn't work, freed up API budget for features that do work, and reduced maintenance burden. Document what you learned (why it failed, what surprised you) so the next AI feature attempt starts smarter.
