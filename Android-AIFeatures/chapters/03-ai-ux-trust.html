Chapter 3

Designing AI UX That Users Trust

Loading states, confidence, error handling, and user control — the patterns that earn trust

Trust Is Earned, Never Assumed

You've picked your AI approach. You've got an API key. You're ready to build. But here's the thing: the technical implementation is maybe 40% of shipping a great AI feature. The other 60% is UX.

Think about it like meeting a new colleague. You don't trust their recommendations on day one — they have to earn it by being right consistently, being transparent when they're unsure, and admitting when they're wrong. AI features work the same way.

1 Bad hallucination to destroy trust in a feature forever
5-10 Correct results needed to build initial trust
73% Of users abandon an AI feature after 3 bad experiences

This chapter gives you the complete UX toolkit for AI features — every state, every edge case, every pattern you need to make users feel safe and in control.

Loading States: The Three Phases

When a user triggers an AI feature, there's a gap between "I asked" and "I got an answer." How you fill that gap determines whether the user feels patient or frustrated.

Never, ever show a bare spinner. Users staring at a spinner with no context think the app is frozen. Instead, use three-phase loading:

Initiating
→
Processing
→
Completing

Phase 1: Initiating

Tell the user what's happening. "Analyzing your photo..." "Summarizing the article..." "Finding similar items..." This sets expectations and proves the app isn't frozen.

Phase 2: Processing

Show progress. For streaming LLM responses, show text as it arrives. For image analysis, show a skeleton screen or pulsing placeholder. For multi-step processes, show which step you're on: "Reading the document... Extracting key points... Generating summary..."

Phase 3: Completing

Reveal the result with a smooth transition — not a jarring pop. A subtle fade-in or slide-up animation tells the user "this is the answer" without being distracting.

The Contextual Loading Pattern

The best AI loading states are contextual — they tell the user what the AI is doing, not just that it's busy. Compare:

  • Bad: A spinning circle. (What's happening? Is it broken? Should I tap again?)
  • Better: "Loading..." (OK, something is happening, but what?)
  • Good: "Analyzing your receipt..." (I know what it's doing.)
  • Great: "Reading the text... Found merchant name... Extracting total..." (I can see it working step by step.)

Real-world example: ChatGPT shows the streaming text as it generates. Gemini shows "Thinking..." with animated dots before the response starts. Both set expectations and keep the user engaged during the wait.

Streaming Responses in Compose

For LLM features (summarization, chat, content generation), streaming the response token by token is the gold standard. It feels faster than waiting for the full response, and it sets realistic expectations for generation time.

Recipe: Streaming AI Text in Compose

Goal: Show a Gemini response word by word as it arrives, like the ChatGPT typing effect.

Step 1: Create a ViewModel that collects the stream:

class SummaryViewModel(private val model: GenerativeModel) : ViewModel() {
    private val _text = MutableStateFlow("")
    val text: StateFlow<String> = _text

    private val _isGenerating = MutableStateFlow(false)
    val isGenerating: StateFlow<Boolean> = _isGenerating

    fun summarize(input: String) {
        viewModelScope.launch {
            _isGenerating.value = true
            _text.value = ""
            model.generateContentStream("Summarize in 3 bullets:\n$input")
                .collect { chunk ->
                    _text.value += chunk.text.orEmpty()
                }
            _isGenerating.value = false
        }
    }
}

Step 2: Build the streaming UI in Compose:

@Composable
fun StreamingSummary(viewModel: SummaryViewModel) {
    val text by viewModel.text.collectAsStateWithLifecycle()
    val isGenerating by viewModel.isGenerating.collectAsStateWithLifecycle()

    Column(modifier = Modifier.padding(16.dp)) {
        if (text.isNotEmpty()) {
            Text(text, style = MaterialTheme.typography.bodyLarge)
        }
        if (isGenerating) {
            Row(verticalAlignment = Alignment.CenterVertically) {
                CircularProgressIndicator(
                    modifier = Modifier.size(12.dp),
                    strokeWidth = 2.dp
                )
                Spacer(Modifier.width(8.dp))
                Text("Generating...",
                    color = MaterialTheme.colorScheme.outline,
                    style = MaterialTheme.typography.labelSmall)
            }
        }
    }
}

The user sees text appearing in real time, with a small spinner at the bottom while generation continues. It feels responsive even if the full response takes 3 seconds.

Communicating Confidence

In Chapter 1, we introduced the confidence spectrum. Now let's implement it. The core rule: translate confidence into behavior, not numbers.

Exposing Raw Confidence

"This is a receipt (67.3% confidence)" Users think: Is 67% good? Should I trust this? What happens at 66%?

Raw numbers create anxiety. Users don't have a mental model for what "67% confidence" means.

Confidence as Behavior

High: "Receipt detected" — just show the result Medium: "This looks like a receipt" — softer language Low: "Is this a receipt?" — ask the user to confirm

The AI's uncertainty is expressed through language and interaction, not numbers. Users naturally understand the difference.

Here's how to implement confidence tiers in your code:

sealed class AiResult<T> {
    data class HighConfidence<T>(val value: T) : AiResult<T>()
    data class MediumConfidence<T>(val value: T) : AiResult<T>()
    data class LowConfidence<T>(val options: List<T>) : AiResult<T>()
    data class NoResult<T>(val reason: String) : AiResult<T>()
}

// In your ViewModel, map API confidence to UX tiers
fun classifyConfidence(score: Float, result: String): AiResult<String> = when {
    score > 0.9f  -> AiResult.HighConfidence(result)
    score > 0.7f  -> AiResult.MediumConfidence(result)
    score > 0.5f  -> AiResult.LowConfidence(topResults)
    else           -> AiResult.NoResult("Couldn't determine the content")
}

The thresholds (0.9, 0.7, 0.5) aren't magic numbers — tune them based on user testing. If users are frequently correcting "high confidence" results, your threshold is too low. If they're rarely seeing results because everything falls into "no result," it's too high.

The "AI Suggested" Badge

When AI generates or suggests content, users need to know. Not because they'll distrust it — but because they need to know they can change it.

Scenario: Diego opens his email app and starts replying to a message. The app shows three suggested quick replies: "Sounds good!", "Thanks, I'll look into it", and "Let me get back to you on that." Each suggestion has a subtle "AI" chip next to it.

Diego taps "Thanks, I'll look into it" — the text fills into the compose field, editable. He changes "look into it" to "check with the team" and sends. The AI got him 80% there; he did the last 20%.

Without the "AI" badge, Diego might wonder: did the app read my mind? Is it canned text? Is it being sent automatically? The badge removes ambiguity and gives him confidence to edit.

Implementation in Compose is simple — a small chip or label:

@Composable
fun AiSuggestionChip(text: String, onClick: () -> Unit) {
    AssistChip(
        onClick = onClick,
        label = { Text(text) },
        leadingIcon = {
            Icon(
                Icons.Outlined.AutoAwesome,
                contentDescription = "AI suggestion",
                modifier = Modifier.size(16.dp)
            )
        }
    )
}

The AutoAwesome sparkle icon has become the universal symbol for "AI did this." Google uses it, Samsung uses it, most apps use it. Stick with the convention — users already know what it means.

User Control and Override

This is the single most important UX principle for AI features: the user must always feel in control. Every AI suggestion, generation, or action needs an escape hatch.

  • Dismiss: "Not now" / swipe away / tap outside to close
  • Edit: Modify the AI's suggestion before accepting it
  • Undo: Reverse an AI action after accepting it
  • Feedback: "Not helpful" / thumbs down / "wrong answer"
  • Disable: Turn off AI suggestions for this feature entirely

Never auto-apply AI results without user confirmation. An AI that auto-corrects your text, auto-sends a message, or auto-deletes a photo will terrify users. AI should suggest, not act. The user confirms, the app executes.

The Suggestion-Confirmation Pattern

The safest pattern for most AI features follows three steps:

  1. AI suggests — Show the AI's output in a clearly labeled, non-destructive way
  2. User reviews — The user can see, edit, or dismiss the suggestion
  3. User confirms — Only after explicit user action does the suggestion become real

Gmail Smart Compose nails this: gray text appears ahead of your cursor (suggestion), you keep typing to ignore it or press Tab to accept (confirmation). At no point does the AI take control.

Error Communication

AI errors are different from regular app errors. A network error is binary — it worked or it didn't. An AI error is a spectrum — the result might be slightly wrong, confidently wrong, or completely nonsensical.

The AI Error Taxonomy

Error Type What Happened What to Show
Infrastructure failure API down, timeout, no network "AI is temporarily unavailable. Try again in a moment."
Low confidence AI can't determine the answer "I'm not sure about this. Here are some possibilities..." or show nothing
Content filtered Safety filter triggered "I can't help with that request." (No details about why — prevents gaming)
Rate limited User hit usage cap "You've reached today's AI limit. Resets at midnight." with manual fallback
Wrong result AI is confidently incorrect This is the hardest — the AI doesn't know it's wrong. You need feedback mechanisms.

The Golden Rule of AI Errors

"I'm not sure" is always better than a wrong answer presented with confidence. If your model's confidence is below your threshold, show nothing, ask for clarification, or present multiple options. A confident hallucination damages trust far more than an honest "I don't know."

Feedback Loops

Feedback is how your AI feature gets better over time — and how users feel heard when things go wrong. But feedback collection has its own UX challenge: you need data without annoying users.

Explicit Feedback

Thumbs up/down buttons, "Was this helpful?" prompts, star ratings. Use sparingly — showing this after every AI interaction causes survey fatigue. Good rules of thumb:

  • Show feedback buttons inline, not in a popup
  • Ask after critical interactions (long summaries, important classifications) but not routine ones (autocomplete, quick suggestions)
  • Make it one tap — thumbs up/down, not a 5-star scale with a text field

Implicit Feedback

The most powerful feedback doesn't require the user to do anything extra:

  • Accepted vs dismissed — Did the user tap the AI suggestion or swipe it away?
  • Edit distance — The user accepted the draft but rewrote 80% of it? The AI didn't do a great job.
  • Time to action — Did the user accept the suggestion immediately (high quality) or stare at it for 10 seconds before accepting (uncertain)?
  • Re-queries — Did the user search again right after seeing results? The first results were probably bad.

Log implicit signals from day one, even if you don't use them yet. When you want to improve your AI feature in three months, this data will be gold. Chapter 11 covers measurement and A/B testing in detail.

Accessibility for AI Features

AI features introduce unique accessibility challenges. Dynamic content, streaming text, and visual overlays need special attention.

Screen Readers and Dynamic Content

When AI results load, screen readers need to know. Use LiveRegion in Compose to announce new content:

Text(
    text = aiResult,
    modifier = Modifier.semantics {
        liveRegion = LiveRegionMode.Polite
    }
)

Polite waits for the current announcement to finish before reading the AI result. Use Assertive only for critical AI results (like error messages).

Other Accessibility Considerations

  • Streaming text: Don't announce every token — wait for a sentence boundary or announce the complete result when streaming finishes
  • AI-generated images: Always provide alt text. If AI generates an image, have the AI also generate a description
  • Motion: Respect prefers-reduced-motion — skip streaming animations for users who find them disorienting
  • Confidence indicators: If you use color to show confidence (green/yellow/red), also use text or icons for color-blind users

Progressive Disclosure

Don't dump every AI capability on the user at once. Layer it:

The Three-Layer AI Disclosure Pattern

  1. Layer 1 — Simple action: A single button or auto-triggered suggestion. "Summarize" button, autocomplete text, smart sort. One tap, instant value, no configuration.
  2. Layer 2 — Customization: After the user has used and trusted the basic feature, reveal options. "Summarize as bullet points / paragraph / key takeaways." "Search in: all notes / this folder / last 7 days."
  3. Layer 3 — Power user: Advanced settings for users who want control. Prompt customization, model selection, temperature adjustment, output format. Hidden behind a "gear" icon or advanced settings.

Key insight: Most users will live on Layer 1 forever, and that's fine. Design Layer 1 to work beautifully with zero configuration. The other layers exist for the 15% who want more control.

Dark Patterns to Avoid

AI features create new opportunities for manipulative UX. Don't be that app.

Fake Urgency

"AI detected that this deal expires soon!" — when the AI didn't actually analyze anything and the "urgency" is manufactured to drive clicks.

Manufactured Authority

"Our AI recommends this product" — when the recommendation is actually a paid placement. AI should never be a disguise for advertising.

Engagement Traps

"You asked the AI 3 questions today. Upgrade to ask more!" — when the free limit is artificially low to frustrate users into paying. Limits should reflect real costs, not manipulation.

Invisible AI

Using AI to filter, sort, or prioritize content without telling the user. If AI is shaping what the user sees, they should know — and be able to switch to chronological or manual sorting.

The test: Would you be embarrassed if a journalist wrote about how your AI feature works? If the answer is yes, redesign it. Transparency builds long-term trust; manipulation destroys it.

Putting It All Together: A Complete UX Flow

Let's trace the complete user experience for an "AI Summarize" button on an article reader app:

State What the User Sees Implementation Detail
Idle Sparkle icon + "Summarize" button at the top of the article Only show if article length > 500 words
Initiating Button dims, text changes to "Reading article..." Disable button to prevent double-tap
Processing Summary card appears with streaming bullet points generateContentStream() with Flow collection
Complete Full summary with "AI Summary" badge + thumbs up/down Cache the result; show feedback buttons inline
Error "Couldn't summarize. Tap to retry." with a retry icon Log the error type; show manual fallback if repeated failure
Dismissed Summary card collapses; "Summarize" button returns Log dismissal as implicit feedback

Every state is covered. The user is never confused, never stuck, and always in control.

Your Turn: Design the Full Flow

Exercise: Design an "AI Email Reply" Feature

You're building an email app. When the user opens an email, they see an "AI Reply" button that generates three suggested replies. Design the complete UX flow:

  1. Sketch every state: Idle, loading, results, error, user edit, and feedback
  2. Define confidence behavior: What does the AI show when it's very confident vs. unsure?
  3. Plan error cases: What if the API is down? What if the email is in a language the AI doesn't support? What if the email content triggers a safety filter?
  4. Add user control: How does the user dismiss, edit, regenerate, or provide feedback?
  5. Consider accessibility: How does a screen reader user experience the suggested replies?

Bonus: For each suggested reply, decide — should it auto-fill the compose field on tap, or show a preview first? What's the tradeoff?
