Chapter 7

Image Understanding & Visual Search

Give your app eyes — categorize photos, search visually, describe scenes, and extract structured data from images

Your App Can See Now

Up through Chapter 6, every AI feature we built worked with text. Search, summarization, chat — all words. But users don't live in a text-only world. They take photos, scan documents, browse product images, and expect your app to understand what's in those images, not just that an image file exists.

Image understanding is one of the most mature areas of AI on mobile. Google Photos has been doing it for years. ML Kit makes the basics trivial. And multimodal models like Gemini Vision can describe, analyze, and extract structured data from any image you throw at them.

This chapter covers the full spectrum: from simple on-device classification to powerful cloud-based image analysis.

On-Device Image Classification with ML Kit

The fastest way to add image understanding: ML Kit's Image Labeling API. It runs on-device, needs no internet, and identifies objects, scenes, and activities in photos.

Recipe: Auto-Tag Photos with ML Kit

Goal: Automatically categorize photos as "food," "nature," "people," "document," etc.

Step 1: Add ML Kit to your project:

implementation("com.google.mlkit:image-labeling:17.0.9")

Step 2: Create the labeling helper:

class ImageLabeler {
    private val labeler = ImageLabeling.getClient(
        ImageLabelerOptions.Builder()
            .setConfidenceThreshold(0.7f) // only return labels with 70%+ confidence
            .build()
    )

    suspend fun labelImage(bitmap: Bitmap): List<String> {
        val image = InputImage.fromBitmap(bitmap, 0)
        return suspendCoroutine { cont ->
            labeler.process(image)
                .addOnSuccessListener { labels ->
                    cont.resume(labels.map { it.text })
                }
                .addOnFailureListener { cont.resume(emptyList()) }
        }
    }
}

Step 3: Use it when photos are added:

// When user adds a photo to the gallery
val labels = imageLabeler.labelImage(photoBitmap)
// labels = ["Food", "Dessert", "Baked goods", "Cake"]
photoDao.update(photo.copy(tags = labels))

Step 4: Display tags in the UI:

@Composable
fun PhotoTags(tags: List<String>) {
    FlowRow(horizontalArrangement = Arrangement.spacedBy(4.dp)) {
        tags.forEach { tag ->
            AssistChip(onClick = { /* filter by tag */ },
                label = { Text(tag, style = MaterialTheme.typography.labelSmall) })
        }
    }
}

That's it — on-device, instant, free, private. ML Kit recognizes 400+ label categories out of the box.

Multimodal AI for Deep Image Understanding

ML Kit tells you "there's a cat in this photo." Gemini Vision tells you "there's an orange tabby cat sleeping on a blue couch next to a half-read book about gardening." When you need understanding, not just labels, you need a multimodal model.

Scenario: Tomasz is building an inventory management app for a small bookstore. The owner photographs each new shipment of books. Tomasz's app uses Gemini Vision to analyze each photo: it identifies the book titles, authors, ISBNs (from the barcode), and even the condition ("slight wear on the spine"). All from a single photo, extracted into structured data that auto-populates the inventory form.

The bookstore owner saves 2 hours per shipment. That's the power of multimodal — the AI sees the image the way a human would, and can answer questions about it.

val model = GenerativeModel(
    modelName = "gemini-2.0-flash",
    apiKey = BuildConfig.GEMINI_API_KEY
)

suspend fun analyzeImage(bitmap: Bitmap, question: String): String {
    val response = model.generateContent(
        content {
            image(bitmap)
            text(question)
        }
    )
    return response.text ?: "Could not analyze the image"
}

// Usage examples:
// analyzeImage(photo, "Describe this image in one sentence")
// analyzeImage(photo, "What product is this? Return: name, brand, color")
// analyzeImage(receipt, "Extract: merchant, date, total, line items as JSON")

The prompt matters as much as the image. "Describe this image" gives you a paragraph. "List the 3 main objects in this image" gives you a focused list. "Extract the text and return it as JSON with fields: merchant, date, total" gives you structured data. Be specific about what you want back.

Building Visual Search

Visual search is the image equivalent of semantic text search from Chapter 4. Instead of embedding text into vectors, you embed images into vectors. Similar images have similar vectors.

The "Find More Like This" Pattern

Visual search unlocks a powerful product pattern: the user points at an image and says "find more like this." No typing, no describing, no keyword guessing.

  1. User taps a photo in their gallery, a product in a catalog, or snaps a picture
  2. App embeds the image into a vector using CLIP or a similar model
  3. App searches for nearest vectors in the content database
  4. Results appear — visually similar images, ranked by similarity

Real-world example: Pinterest's "Visual Search" lets you draw a box around part of an image and find similar items. Google Lens identifies objects and finds products, places, and information. Both use image embeddings under the hood.

The architecture mirrors text semantic search, but with image embeddings:

class VisualSearchEngine(private val clipModel: Interpreter) {
    // Embed an image into a 512-dimension vector
    fun embedImage(bitmap: Bitmap): FloatArray {
        val resized = Bitmap.createScaledBitmap(bitmap, 224, 224, true)
        val input = preprocessForClip(resized)
        val output = Array(1) { FloatArray(512) }
        clipModel.run(input, output)
        return output[0]
    }

    // Find visually similar images
    fun findSimilar(query: Bitmap, library: List<ImageEntry>): List<ImageEntry> {
        val queryVec = embedImage(query)
        return library
            .map { it to cosineSimilarity(queryVec, it.embedding) }
            .sortedByDescending { it.second }
            .take(10)
            .map { it.first }
    }
}

CLIP models (Contrastive Language-Image Pre-Training) are special — they can embed both images and text into the same vector space. This means you can search images using text queries ("sunset at the beach") or image queries (a photo of a sunset). Same engine, two input types.

Scene Description for Accessibility

One of the most impactful uses of image AI: generating descriptions for visually impaired users. Every image in your app should have meaningful alt text, and AI can generate it automatically.

Two-Tier Accessibility

Tier 1: ML Kit (On-Device, Basic)

Labels: "Food, Dessert, Cake, Plate" Alt text: "Image contains: food, dessert, cake" Fast, free, works offline. Good enough for basic context.

Tier 2: Gemini Vision (Cloud, Rich)

Prompt: "Describe this image for a visually impaired user in one sentence." Alt text: "A slice of chocolate layer cake on a white plate, topped with fresh strawberries and dusted with powdered sugar." Slower, costs money, but gives a meaningful description.

Use Tier 1 as the default (immediate, free). Offer Tier 2 as a "Get detailed description" option, or pre-generate Tier 2 descriptions in the background with WorkManager.

ML Kit's text recognition extracts text from images. That's useful but limited — you get a blob of strings with no structure. The real power comes from combining OCR with an LLM to understand the text.

val recognizer = TextRecognition.getClient(TextRecognizerOptions.DEFAULT_OPTIONS)
val rawText = suspendCoroutine { cont ->
    recognizer.process(InputImage.fromBitmap(bitmap, 0))

""".trimIndent())

// Result: {"merchant": "Starbucks", "date": "2026-03-28", "total": "8.45",
//          "items": [{"name": "Latte", "price": "5.95"}, ...]}

The OCR is free and on-device. The LLM parsing costs a fraction of a cent. Together, they turn a photo of a receipt into structured data your app can use.

Image Preprocessing

Before sending an image to any AI model, you need to prepare it. Android's image format zoo (JPEG, PNG, HEIC, WebP, different orientations, different sizes) can cause issues.

  • Resize: Cloud APIs accept large images but charge by size. Resize to 1024px max dimension for most tasks
  • Orientation: Read EXIF data and rotate before processing. A sideways photo gives sideways results
  • Format: Convert HEIC to JPEG before sending to APIs that don't support HEIC
  • Compression: Compress to 80% JPEG quality — minimal quality loss, significant size reduction
  • Strip metadata: Remove EXIF GPS data before sending to cloud APIs (privacy)
fun prepareImageForApi(context: Context, uri: Uri): Bitmap {
    val source = ImageDecoder.createSource(context.contentResolver, uri)
    val bitmap = ImageDecoder.decodeBitmap(source)

    // Fix orientation from EXIF
    val rotated = fixOrientation(context, uri, bitmap)

    // Resize if too large
    val maxDim = 1024
    val scale = minOf(maxDim.toFloat() / rotated.width, maxDim.toFloat() / rotated.height, 1f)
    return Bitmap.createScaledBitmap(rotated,
        (rotated.width * scale).toInt(),
        (rotated.height * scale).toInt(), true)
}

Batch Processing with WorkManager

Analyzing photos one at a time is fine for user-triggered actions. But for background tasks (categorizing an entire photo library, pre-generating alt text), use WorkManager:

class PhotoAnalysisWorker(
    context: Context, params: WorkerParameters
) : CoroutineWorker(context, params) {
    override suspend fun doWork(): Result {
        val untaggedPhotos = photoDao.getUntagged(limit = 20) // batch of 20
        for (photo in untaggedPhotos) {
            val bitmap = loadBitmap(photo.uri)
            val labels = imageLabeler.labelImage(bitmap)
            photoDao.update(photo.copy(tags = labels))
        }
        return if (photoDao.hasUntagged()) Result.retry() else Result.success()
    }
}

// Schedule with constraints
val request = PeriodicWorkRequestBuilder<PhotoAnalysisWorker>(1, TimeUnit.HOURS)
    .setConstraints(Constraints.Builder()
        .setRequiresBatteryNotLow(true) // don't drain the battery
        .build())
    .build()
WorkManager.getInstance(context).enqueueUniquePeriodicWork(
    "photo-analysis", ExistingPeriodicWorkPolicy.KEEP, request
)

Batch processing + memory = trouble. Loading 100 full-resolution bitmaps will crash your app. Process one image at a time, recycle the bitmap after each one, and use limit to process in small batches. Monitor memory with Runtime.getRuntime().freeMemory().

Privacy Considerations

Photos are the most personal data on someone's phone. Treat them accordingly.

The Photo Privacy Checklist

On-device first: Use ML Kit for categorization and basic labeling — photos never leave the device.

Explicit consent for cloud: Before sending any photo to a cloud API, tell the user: "This photo will be sent to [provider] for analysis." Let them opt in, not opt out.

Strip metadata: Remove GPS coordinates and EXIF data before cloud upload. Nobody's AI analysis needs to know where a photo was taken.

Don't store remotely: Send the image for analysis, get the result, and don't keep a copy on the server. Implement transient processing.

Cost of Image APIs

Image analysis is more expensive than text. Budget accordingly:

Operation Approx. Cost Strategy
ML Kit labeling $0 (on-device) Use for all basic classification
Gemini Flash (image input) ~$0.002 per image Use for detailed analysis, OCR parsing
GPT-4o Vision ~$0.01-0.03 per image Use only when Gemini quality isn't enough
Cloud embedding (CLIP) ~$0.001 per image Embed once at upload, search is free after

The cost-saving stack: Use ML Kit (free) for everything it can handle. Use Gemini Flash for detailed analysis. Use GPT-4o Vision only as a fallback for complex cases. Cache every result. This keeps your average cost per image under $0.005.
