Intelligent Notifications & Recommendations

AI-driven timing, personalized content, and behavior prediction — helping users before they ask

The Notification Paradox

Notifications are the most powerful and most dangerous tool in your app. A well-timed, relevant notification drives engagement. A badly timed, irrelevant one drives uninstalls.

Think of notifications like a colleague tapping you on the shoulder. If they interrupt your focus to tell you something urgent and relevant, you're grateful. If they interrupt you during a meeting to say "hey, we have new posts!" you want to throw your phone out the window.

AI can fix this by answering three questions: What to notify (is this content worth interrupting for?), When to notify (is now a good time for this user?), and How to say it (what message will resonate?).

50% Of users disable notifications within the first week
4x Higher open rate for personalized vs generic notifications
28% Of uninstalls are triggered by annoying notifications

Smart Notification Timing

Sending a notification at 3am because that's when a new article was published is lazy engineering. Sending it at 7:30am because that's when this user typically opens the app is smart engineering.

On-Device Behavior Signals

You don't need a cloud ML pipeline to figure out when a user is available. Android gives you signals right on the device:

Signal API What It Tells You
App usage patterns UsageStatsManager When the user typically opens your app (morning? evening? lunch?)
Activity recognition ActivityRecognitionClient Is the user walking, driving, or still? (Don't notify while driving)
Do Not Disturb NotificationManager Is the device in DND mode? Respect it.
Screen state PowerManager Is the screen on? If so, the user is active and more likely to engage

Recipe: Learning the User's Active Hours

Goal: Figure out when the user typically opens your app and schedule notifications for those windows.

Step 1: Log app open times locally:

@Entity
data class AppOpenEvent(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val hourOfDay: Int,     // 0-23
    val dayOfWeek: Int,     // 1-7 (Monday-Sunday)
    val timestamp: Long = System.currentTimeMillis()
)

// In your Application.onCreate() or main Activity
appOpenDao.insert(AppOpenEvent(
    hourOfDay = LocalTime.now().hour,
    dayOfWeek = LocalDate.now().dayOfWeek.value
))

Step 2: Calculate the optimal notification window:

fun getOptimalNotificationHour(events: List<AppOpenEvent>): Int {
    if (events.size < 7) return 9 // default to 9am until we have enough data

    // Find the most common hour the user opens the app
    return events
        .groupBy { it.hourOfDay }
        .maxByOrNull { it.value.size }
        ?.key ?: 9
}

// For day-specific optimization:
fun getOptimalHourForDay(events: List<AppOpenEvent>, dayOfWeek: Int): Int {
    val dayEvents = events.filter { it.dayOfWeek == dayOfWeek }
    if (dayEvents.size < 3) return getOptimalNotificationHour(events)
    return dayEvents.groupBy { it.hourOfDay }.maxByOrNull { it.value.size }?.key ?: 9
}

Step 3: Schedule notifications using WorkManager:

fun scheduleSmartNotification(content: NotificationContent) {
    val optimalHour = getOptimalHourForDay(
        appOpenDao.getRecentEvents(days = 30),
        LocalDate.now().plusDays(1).dayOfWeek.value
    )
    val delay = calculateDelayUntilHour(optimalHour)

    val request = OneTimeWorkRequestBuilder<NotificationWorker>()
        .setInitialDelay(delay, TimeUnit.MILLISECONDS)
        .setInputData(workDataOf("content" to content.toJson()))
        .build()
    WorkManager.getInstance(context).enqueue(request)
}

After two weeks of data, your app knows that this user opens it at 8am on weekdays and 10am on weekends. Notifications arrive right when the user is likely to see them.

Notification Content Personalization

The same notification can perform very differently based on how you phrase it. AI can help you tailor the message to each user's preferences:

Generic Notification

"New recipes available" Every user gets the same message. Low open rate. Feels like spam.

Personalized Notification

"3 new chicken recipes — including one under 20 minutes" Tailored to what this user likes (chicken) and cares about (quick meals). 4x higher open rate.

You don't need an LLM for every notification. A template system with user preferences works great:

fun buildNotificationText(user: UserPreferences, newContent: List<Content>): String {
    val relevant = newContent.filter { it.matchesPreferences(user) }
    if (relevant.isEmpty()) return "" // don't notify if nothing is relevant

    val topCategory = relevant.groupBy { it.category }.maxByOrNull { it.value.size }
    val count = relevant.size
    val highlight = relevant.firstOrNull { it.isQuick }?.let {
        " — including one under ${it.cookTime} minutes"
    } ?: ""

    return "$count new ${topCategory?.key ?: "items"}$highlight"
}

The personalization spectrum: Start with template-based personalization (cheap, fast, deterministic). If you need more creativity — like summarizing 10 different updates into one engaging sentence — use an LLM, but batch-generate notifications server-side, not on-device per-user.

Content Recommendation Engines

Recommendations are the proactive version of search. Instead of the user asking "what should I read/cook/buy?", your app answers that question before it's asked.

The Three Approaches

Approach How It Works Best For
Collaborative filtering "Users like you also liked X" Large user bases (>10K users) with shared content
Content-based "Similar to what you've liked before" Any user base; works with embeddings from Chapter 4
Hybrid Combines both for better accuracy Production systems with enough data for both

For most Android apps starting out, content-based is the way to go. You already have the tools from Chapter 4 — content embeddings and user preference vectors. Recommendations are just "find content closest to the user's preference vector."

fun getRecommendations(
    userPreference: FloatArray,  // average of recently engaged content embeddings
    allContent: List<Content>,
    alreadySeen: Set<Long>
): List<Content> {
    return allContent
        .filter { it.id !in alreadySeen }  // don't recommend what they've seen
        .map { it to cosineSimilarity(userPreference, it.embedding) }
        .sortedByDescending { it.second }
        .take(10)
        .map { it.first }
}

The Proactive Assistant Pattern

The best recommendation features surface information before the user asks, at the moment they're most likely to need it. The pattern:

  1. Learn the user's routine: What do they do at what time? (Check weather at 7am, browse recipes at 5pm, review expenses on Sundays)
  2. Prepare relevant content: Pre-compute recommendations based on their patterns and preferences
  3. Surface at the right moment: Show a card on the home screen, a widget update, or a well-timed notification
  4. Make it dismissible: If the user doesn't want proactive suggestions, let them turn it off

Real-world examples: Google Maps shows your commute time without asking. A fitness app suggests today's workout at your usual exercise time. A news app surfaces your preferred topics as a morning digest.

On-Device Recommendation Models

For privacy-sensitive apps, you can run the entire recommendation engine on-device. No user data leaves the phone.

Scenario: Marco uses a health and wellness app that tracks his meals and suggests recipes. The app never sends his dietary data to a server. Instead, a lightweight TFLite recommendation model runs locally, learning from his meal history that he prefers Mediterranean cuisine, avoids dairy, and usually cooks quick meals on weekdays.

Every evening at 5pm, a card appears on his home screen: "Tonight's suggestion: 20-Minute Lemon Herb Salmon." It's relevant, timely, and private. Marco taps it and starts cooking.

The architecture is simple: train a small recommendation model on aggregate, anonymized data (server-side). Deploy it to devices. Let it personalize using the user's local history.

class OnDeviceRecommender(context: Context) {
    private val model = Interpreter(
        FileUtil.loadMappedFile(context, "recommender.tflite")
    )

    fun recommend(
        userHistory: List<Long>,   // IDs of recently engaged items
        candidateIds: List<Long>    // IDs of items to score
    ): List<Pair<Long, Float>> {
        val input = preprocessHistory(userHistory) // encode as model input
        val output = Array(1) { FloatArray(candidateIds.size) }
        model.run(input, output)

        return candidateIds.zip(output[0].toList())
            .sortedByDescending { it.second }
    }
}

Behavior Prediction

Beyond recommendations, you can predict what the user wants to do next based on patterns. This doesn't require fancy ML — often, simple rules work beautifully:

data class UserPattern(
    val action: String,        // "check_weather", "browse_recipes", "log_expense"
    val typicalHour: Int,      // most common hour for this action
    val typicalDay: Int?,      // null = every day, or specific day of week
    val confidence: Float      // how consistent is this pattern
)

fun detectPatterns(events: List<UserEvent>): List<UserPattern> {
    return events
        .groupBy { it.action }
        .mapNotNull { (action, actionEvents) ->
            val hourCounts = actionEvents.groupBy { it.hourOfDay }
            val peakHour = hourCounts.maxByOrNull { it.value.size }
            val consistency = (peakHour?.value?.size?.toFloat() ?: 0f) / actionEvents.size

            if (consistency > 0.3f && actionEvents.size > 10) {
                UserPattern(action, peakHour!!.key, null, consistency)
            } else null
        }
}

"You usually check weather at 7am" → show a weather card at 6:55am. "You log expenses on Sundays" → send a gentle reminder on Sunday morning. Simple patterns, big impact.

Notification Grouping and Summarization

When 12 notifications arrive while the user is away, don't show 12 individual notifications. Group and summarize them:

12 Raw Notifications
→
Group by Source
→
AI Summarize
→
1-3 Smart Notifications
fun summarizeNotificationBatch(notifications: List<RawNotification>): List<SmartNotification> {
    val grouped = notifications.groupBy { it.source }

    return grouped.map { (source, items) ->
        if (items.size <= 2) {
            // Few items — show individually
            items.map { SmartNotification(title = source, body = it.preview) }
        } else {
            // Many items — summarize
            val summary = when (source) {
                "messages" -> "${items.size} messages from ${items.distinctBy { it.sender }.size} people"
                "news" -> "${items.size} articles about ${items.first().category}"
                else -> "${items.size} updates from $source"
            }
            listOf(SmartNotification(title = source, body = summary))
        }
    }.flatten()
}

For richer summaries, send the batch to an LLM: "Summarize these 8 messages into one notification preview" → "5 messages from Anna about the project deadline, 3 from the team chat about lunch plans."

Engagement Metrics That Matter

To improve your notifications and recommendations, you need to measure the right things:

Metric What It Measures Target
Open rate % of notifications tapped >15% (industry average is ~5%)
Dismissal rate % swiped away without opening <60%
Action rate % that led to meaningful in-app action >8%
Opt-out rate % of users disabling notifications <10% in first month
Recommendation CTR % of recommended items tapped >5% (varies by domain)

High open rate + high uninstall rate = manipulation. If your notifications are clickbait that gets opens but doesn't deliver value, users will eventually leave. Optimize for action rate and retention, not just opens.

Avoiding the Filter Bubble

A recommendation engine that only shows what users already like creates an echo chamber. Users get stuck in a loop of similar content and never discover anything new.

Diversification Strategies

  • Serendipity slot: Reserve 1-2 positions in every recommendation list for content outside the user's usual preferences. "You might also like..." from a different category
  • Exploration bonus: Boost the score of content the user hasn't interacted with before, even if it's less similar to their preferences
  • Category cap: No single category can take more than 40% of recommendation slots. Force variety
  • User control: Let users explicitly say "show me less of this" or "explore more categories." Put the user in the driver's seat

The Recommendation Ethics Test

Ask yourself: "Would I be comfortable if the user could see exactly why this was recommended?" If the answer is "because they'll click and spend more time in the app," that's a manipulative recommendation. If the answer is "because it matches their interests and saves them time finding it," that's a helpful recommendation. Build the latter.
