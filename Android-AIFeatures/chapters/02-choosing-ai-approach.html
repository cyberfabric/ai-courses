Chapter 2

Choosing the Right AI Approach

Cloud APIs vs on-device vs hybrid — a decision framework for every feature

The Three Paths

In Chapter 1, we previewed the three ways to add AI to your Android app. Now it's time to go deep. Every AI feature you build will use one of these paths — and picking the wrong one can mean burning money, frustrating users, or hitting a wall you can't climb over.

Think of it like choosing transportation. Cloud APIs are taxis: fast to hail, go anywhere, but the meter is always running. On-device models are bicycles: free to ride, always available, but they can't carry heavy loads. Hybrid is owning a car: flexible, powerful, but you need to maintain it.

Cloud AI APIs: The Power Taxi

You send data to a server, a powerful model processes it, and you get results back. This is the fastest path from "I have an idea" to "users are using it."

The Major Players in 2026

Provider Best For Pricing Model
Gemini API (Google) Multimodal (text + images + video), tight Android integration Per-token (input + output)
OpenAI API Text generation, function calling, strong ecosystem Per-token (input + output)
Claude API (Anthropic) Long context, nuanced reasoning, safety-focused Per-token (input + output)
Azure AI Enterprise compliance, hosted OpenAI models Per-token or provisioned throughput
AWS Bedrock Multi-model access, AWS ecosystem integration Per-token, varies by model

For most Android developers starting out, Gemini API is the natural first choice — it has a generous free tier, great multimodal support, and first-party Android SDKs. But always abstract your AI provider behind an interface (more on that later).

When Cloud Makes Sense

  • Complex reasoning — Summarizing a 10-page document, generating a thoughtful reply, answering open-ended questions
  • Multimodal analysis — Understanding images, extracting data from documents, analyzing video
  • Rapid prototyping — You can have an AI feature working in an afternoon, no ML expertise needed
  • Small user base — When you have 1,000 users, API costs are pocket change

When Cloud Hurts

  • Real-time features — Network round-trip adds 200-800ms; autocomplete and camera overlays can't wait that long
  • Scale costs — At 1M users, even cheap API calls add up fast
  • Offline scenarios — No internet = no AI feature, frustrated user
  • Privacy-sensitive data — Health records, financial data, private messages — sending to a third-party server has legal implications

On-Device Models: The Free Bicycle

The model runs right on the user's phone. No network call, no API bill, no privacy concerns about data leaving the device. The tradeoff? Smaller, less capable models.

The On-Device Toolkit

ML
ML Kit (Google)
Pre-built APIs for text recognition, face detection, barcode scanning, language identification, and more. No ML expertise needed — just call the API.
TF
TensorFlow Lite
Run custom ML models on Android. You bring the model (or pick from the model zoo), TFLite handles the inference. Great for custom classifiers and embeddings.
MP
MediaPipe
Real-time ML pipelines for camera features — object detection, hand tracking, pose estimation, face mesh. Optimized for mobile frame rates.
GN
Gemini Nano (via AICore)
Google's on-device LLM for Pixel and Samsung flagship devices. Summarization, smart reply, rewriting — actual language understanding without the cloud.

When On-Device Makes Sense

  • Real-time processing — Camera overlays, live translation, keyboard predictions need <50ms response
  • Privacy-first features — Health data, private messages, financial info that should never leave the device
  • Scale economics — 1M users making 10 requests/day = 10M daily inferences = $0 in API costs
  • Offline support — The app works on a plane, in a tunnel, or in areas with bad connectivity

When On-Device Hurts

  • Complex tasks — On-device models can't match cloud models for reasoning, long-form generation, or nuanced understanding
  • Model size constraints — Users won't download a 2GB model for your note-taking app; 20-50MB is the practical limit for most features
  • Device fragmentation — Gemini Nano only runs on flagship devices; your mid-range users get nothing (unless you fall back to simpler models)
  • Model updates — Updating a cloud model is instant; updating an on-device model means shipping a new APK or downloading in the background

The Decision Matrix

Here's the framework. For each AI feature you're considering, score these five dimensions:

Dimension Cloud Wins On-Device Wins
Latency >500ms acceptable (summarization, search) <100ms needed (autocomplete, camera)
Privacy Public/non-sensitive data Personal, health, financial data
Cost at Scale Small user base (<50K DAU) Large user base (>100K DAU)
Offline Need Always-connected use cases Must work without internet
Capability Complex reasoning, generation Classification, detection, simple NLP

If most answers land in one column, your choice is clear. If they're split, you're looking at a hybrid approach.

The Hybrid Architecture

The Tiered Intelligence Pattern

The best AI-powered apps don't choose between cloud and on-device — they use both. The pattern works like this:

  1. On-device for fast, frequent, private tasks — Text classification ("is this a question or a command?"), image preprocessing, keyboard predictions, basic entity extraction
  2. Cloud for heavy, occasional, high-value tasks — Summarizing a long document, generating a draft email, answering complex questions, analyzing multiple images
  3. Smart routing — A thin layer that decides which path to take based on task complexity, connectivity, and user preferences

Real-world example: Google Translate uses on-device models for instant camera translation (speed) but cloud models for high-quality document translation (accuracy). The user never knows the difference — it just works.

Cost Modeling: The Math That Matters

Let's do real math. Say you're building a note-taking app with an "AI Summarize" button.

Scenario: Your note-taking app has 100K daily active users (DAU). Each user summarizes an average of 3 notes per day. Each note is ~500 words (roughly 650 tokens input), and the summary is ~75 words (roughly 100 tokens output).

Daily API calls: 100,000 × 3 = 300,000 requests/day

Daily tokens: 300K × 650 input = 195M input tokens. 300K × 100 output = 30M output tokens.

Monthly cost (Gemini Flash): ~$15/month input + ~$9/month output = ~$24/month. Very manageable.

Monthly cost (GPT-4o): ~$490/month input + ~$300/month output = ~$790/month. Ow.

Monthly cost (on-device): $0. But quality is lower and only works on supported devices.

Always model costs before you commit. The formula is simple: DAU × requests_per_user × tokens_per_request × price_per_token × 30. Run it for your current user count and for 10x growth. If 10x growth breaks the bank, plan your fallback now.

Latency Budgets: What Users Tolerate

Different features have different speed requirements. Here's a practical guide:

<100ms Autocomplete, keyboard predictions, live camera — on-device only
100-500ms Search results, quick classifications — fast cloud or on-device
1-3 seconds Summarization, content generation — cloud with loading UI

The key insight: latency isn't just about speed — it's about perceived speed. A 2-second summarization feels fast if you show a streaming animation. A 500ms search feels slow if the screen goes blank. We'll cover loading UX in Chapter 3.

Privacy and Compliance

Sending user data to a cloud API has legal implications that your product and legal teams care about deeply:

  • GDPR (EU) — Sending personal data to a US-based API requires a data processing agreement (DPA) and potentially a transfer impact assessment. Cloud providers offer DPAs, but you need to sign them.
  • Play Store privacy labels — If you send photos to a cloud API, you must declare "Photos" in your data safety section. Users see this before they install.
  • Health data (HIPAA) — If your app handles health information, most cloud AI APIs are not HIPAA-compliant by default. On-device is often the only safe option.
  • Children's data (COPPA) — If your app targets kids, sending their data to AI APIs has strict requirements.

When in doubt, process on-device. No amount of legal paperwork can undo the trust damage from a headline that says "App sends your private messages to AI servers." If the data is sensitive, keep it local. Chapter 12 covers privacy compliance in detail.

API Key Security on Android

This is the section that prevents your first security incident. Pay attention.

Never Do This

val apiKey = "sk-abc123..." // hardcoded in Kotlin // or in BuildConfig from local.properties // Both end up in the APK, which anyone can decompile

API keys in the APK can be extracted in minutes. Someone will find it, abuse it, and you'll get a $10,000 bill.

Do This Instead

// Your app calls YOUR backend api.summarize(noteText) // Your backend calls the AI API with the key // The key never touches the client device

Your backend acts as a proxy. The API key lives on your server, protected and revocable.

Recipe: Secure AI API Proxy with Your Backend

Goal: Call a cloud AI API without exposing the API key in your Android app.

Step 1: Define a simple API interface in your Android app:

interface AiApiService {
    @POST("api/summarize")
    suspend fun summarize(
        @Body request: SummarizeRequest
    ): SummarizeResponse
}

@Serializable
data class SummarizeRequest(val text: String)

@Serializable
data class SummarizeResponse(val summary: String)

Step 2: On your backend (e.g., a simple Cloud Function), receive the request and forward to the AI API:

// Cloud Function (Node.js example)
exports.summarize = async (req, res) => {
  const { text } = req.body;
  const response = await fetch("https://generativelanguage.googleapis.com/...", {
    method: "POST",
    headers: { "x-goog-api-key": process.env.GEMINI_API_KEY },
    body: JSON.stringify({ contents: [{ parts: [{ text }] }] })
  });
  const data = await response.json();
  res.json({ summary: data.candidates[0].content.parts[0].text });
};

Step 3: Add abuse prevention with Firebase App Check:

// In your Android app initialization
Firebase.appCheck.installAppCheckProviderFactory(
    PlayIntegrityAppCheckProviderFactory.getInstance()
)

// Your backend verifies the App Check token
// Rejects requests without valid tokens

Now your API key is safe on the server, and App Check prevents unauthorized clients from abusing your endpoint.

Vendor Lock-In: The Abstraction Layer

Today you're using Gemini Flash. Next month, GPT-5 launches and it's better for your use case. Or Gemini raises prices. Or your legal team says you need to switch to an EU-hosted provider. If your app calls Gemini directly in 47 places, you're stuck.

The fix is simple: abstract the AI provider behind an interface.

interface AiProvider {
    suspend fun summarize(text: String): String
    suspend fun classify(text: String, categories: List<String>): String
    suspend fun generateReply(context: String, tone: String): String
}

class GeminiProvider(private val model: GenerativeModel) : AiProvider {
    override suspend fun summarize(text: String): String {
        val response = model.generateContent("Summarize: $text")
        return response.text ?: "Unable to summarize"
    }
    // ... other methods
}

class OpenAiProvider(private val client: OpenAI) : AiProvider {
    // Same interface, different implementation
}

Now switching providers is a one-line change in your dependency injection setup. Your features don't know or care which AI is behind the curtain.

This abstraction also makes testing trivial. Create a FakeAiProvider that returns canned responses, and you can test your entire AI feature flow without making a single API call.

Fallback Strategies: When Things Break

Cloud APIs go down. Devices run out of memory. Users lose connectivity on the subway. Your AI feature needs a plan for each scenario.

  • Cloud API is down: Show cached results if available, offer manual workflow, display "AI is temporarily unavailable" (never a generic error)
  • Device can't run the model: Fall back to cloud, or degrade gracefully (show keyword search instead of semantic search)
  • User is offline: Queue requests for later, use on-device for what you can, be transparent about what's unavailable
  • Rate limit hit: Show a friendly "You've used your AI quota for today" with upgrade option, never a raw HTTP 429

The Fallback Principle

Every AI feature should degrade to a usable — if less magical — experience. If your "smart search" cloud API is down, show basic keyword search. If your "AI compose" is offline, show the empty text field. The user should always be able to complete their task, just with more effort.

Putting It All Together

Here's a quick-reference cheat sheet for common AI features:

Feature Recommended Approach Why
Autocomplete / Predictions On-device Needs <100ms, high frequency, privacy matters
Photo categorization On-device (ML Kit) Privacy, offline support, well-solved problem
Document summarization Cloud (or hybrid) Needs powerful model, tolerates 1-2s latency
In-app chatbot Cloud Needs reasoning, context, nuanced responses
Camera object detection On-device (MediaPipe) Real-time requirement, no network dependency
Email draft generation Hybrid On-device for quick reply, cloud for long drafts
Receipt OCR + parsing Hybrid ML Kit for OCR (on-device), LLM for structuring (cloud)
