The Big Idea

Keyword search matches strings. Semantic search matches meaning. By turning text into vectors (embeddings), "something spicy with chicken" finds "Sriracha Stir-Fry" — even though the words don't overlap at all.

⌨️ Two-Tier Autocomplete

Tier How It Works Speed Example
🏃 Tier 1: Prefix Match items starting with typed text <5ms (instant) "chic" → "Chicken Parmesan"
🧠 Tier 2: Semantic Embed query, find meaning matches 30-100ms (debounced) "chic" → also "Poultry Stir-Fry"

Show Tier 1 instantly. Blend Tier 2 as it arrives. Search gets smarter in real time.

🏗️ The Hybrid Search Stack

  1. Layer 1: Instant (<5ms)

    Prefix matching + cached popular queries. Direct title matches, recent searches. On-device, no AI.

  2. Layer 2: Smart (30-100ms)

    Semantic embedding search + fuzzy matching. Meaning-based results, typo tolerance. On-device AI.

  3. Layer 3: Deep (200-500ms, optional)

    Cloud LLM re-ranking of top 20 results. Maximum relevance for complex queries.

📖 Key Terms

Text Embedding
Converting text into a vector of numbers (384-768 dims) that captures semantic meaning.
Cosine Similarity
A measure of how similar two vectors are (0 = unrelated, 1 = identical meaning).
Semantic Search
Finding content by meaning, not keywords. "Blue jacket" matches "Navy Denim Jacket."
Prefix Matching
Instant matching of items that start with the typed text. No AI needed, <5ms.
Levenshtein Distance
Number of character edits to transform one string into another. Catches severe typos.
Re-ranking
Adjusting search result order using multiple signals (similarity, recency, user history, popularity).
User Preference Vector
Average embedding of recently clicked items. Personalizes search without sending data to the cloud.
Debouncing
Waiting for the user to stop typing (300ms) before running the expensive search. Prevents wasted work.

🧪 Quick Quiz

Test Your Understanding

Why does keyword search fail for "something spicy with chicken"?

No recipe title literally contains "something spicy with chicken." Keyword search matches strings, not meaning. Semantic search converts both the query and recipe titles into vectors and finds matches based on meaning similarity, so it can connect "spicy chicken" to "Sriracha Stir-Fry."

What are text embeddings?

Vectors of 384-768 numbers that represent text meaning in a high-dimensional space. Similar meanings have nearby vectors ("happy" near "joyful"), different meanings are far apart ("happy" far from "socket wrench"). Cosine similarity measures the distance.

Why use two-tier autocomplete instead of just semantic search?

Tier 1 (prefix matching) is instant (<5ms) and gives users immediate feedback. Tier 2 (semantic) takes 30-100ms and adds meaning-based results. Showing Tier 1 immediately keeps the UI responsive while smarter results blend in seamlessly.

How does on-device personalized search work without sending data to the cloud?

Track which search results the user clicks. Average the embeddings of recently clicked items to create a user preference vector. Blend it with the query vector (80% query + 20% preference) for personalized ranking. All on-device, all private.

What's the "no-dead-end rule" for search?

Never show "No results" with nothing else. Always offer an escape: alternative query suggestions, category browsing, popular items, or web search fallback. The user should never hit a wall.

Key Takeaways

  • Semantic search matches meaning, not strings — powered by text embeddings (text → vectors).
  • On-device embeddings (all-MiniLM, 22MB) give you semantic search with zero API cost and full privacy.
  • Two-tier autocomplete: instant prefix matching + debounced semantic search for the best of both.
  • Re-rank results with multiple signals: similarity (60%), recency (15%), user affinity (15%), popularity (10%).
  • Typo tolerance comes mostly free with embeddings; add Levenshtein distance for severe typos.
  • Personalize search on-device by averaging embeddings of clicked items — no data leaves the phone.
  • Performance essentials: debounce queries, cache embeddings, pre-compute popular results.
  • Never show "No results" with nothing else — always offer an escape route.
