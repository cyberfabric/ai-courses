Chapter 12

Shipping AI Features Safely

Memory Map
🛡️

The Big Idea

A button that works in testing will work in production. An AI feature that works in testing might hallucinate, get exploited, bankrupt your API budget, or violate privacy regulations in the real world. This checklist is the gap between "it works on my machine" and "it's safe for millions."

✅ The Pre-Launch Safety Checklist

🔒
Content Safety
AI outputs filtered for harmful, offensive, or misleading content. Three layers: model settings, output validation, domain filters.
💰
Cost Safety
Per-user and global rate limits prevent runaway API spend. Circuit breaker kills AI when cost spikes.
🔐
Privacy Safety
User data handled per policy and regulations. Consent for cloud processing. Data retention enforced.
🛡️
Abuse Safety
Bad actors can't exploit for spam generation, content farming, data extraction, or free API access.
🔄
Fallback Safety
App works gracefully when AI is unavailable, slow, or wrong. Manual fallbacks always available.
🔴
Kill Switch
Disable the feature instantly via Firebase Remote Config — no app update needed. Non-negotiable.

⚠️ If any box is unchecked, you're not ready to ship.

⏱️ Rate Limiting Layers

Layer Limit Purpose User Message
Per-minute 10 requests Prevent rapid-fire abuse "Hold on — give me a moment!"
Per-hour 50 requests Limit heavy sessions "You've been busy! Resets in [time]."
Per-day 200 requests Daily cost cap per user "Today's AI limit reached. Resets at midnight."
Global circuit breaker Cost > $X/hour Emergency protection "AI features temporarily unavailable." + alert ops

💡 Rate limit messages should be friendly, not punishing. The user isn't doing anything wrong. Frame limits as positive ("you've been busy!"), show when limits reset, and always offer manual fallbacks.

🧱 Content Filtering: Defense in Depth

  1. Layer 1: Model Safety Settings

    Built-in API safety filters (Gemini SafetySetting). Block harassment, hate speech, explicit content, dangerous content at MEDIUM_AND_ABOVE.

  2. Layer 2: Output Validation

    Check AI output before showing to user: PII detection, max length enforcement, blank response handling.

  3. Layer 3: Domain-Specific Filters

    Your app's context matters. A medical app filters differently than a cooking app. Add use-case-specific checks.

One layer failing doesn't mean harmful content reaches the user. That's the point of depth.

The 3 AM Incident

A viral TikTok video 20x'd your downloads overnight. Every new user tried the chatbot. Your $200/month API budget burned $200/hour. You opened Firebase Console on your phone, set rollout to 10%, and the bleeding stopped in minutes. Without the kill switch, you'd wake up to a $4,800 bill.

📋 Privacy Compliance Checklist

Data Processing Disclosure
Tell users their content is sent to AI APIs. Required by GDPR and CCPA.
Consent for Cloud Processing
Opt-in before first cloud AI use: "Your text will be processed by Google's AI." Required by GDPR.
Data Retention
Don't store AI inputs/outputs longer than needed. Define and enforce retention periods.
Right to Deletion
When a user deletes their account, delete all AI interaction logs too.
Play Store Data Safety
Declare what data you collect and share in the Data Safety section.
AI Content Disclosure
Label AI-generated content as AI-generated. Google Play policy for generative AI.

🚨 Incident Response Playbook

  1. 🔍 Detect

    Automated alert fires: cost spike, quality drop, error rate, or user report.

  2. 📊 Assess (30 seconds)

    How many users affected? Harmful content, cost overrun, or quality issue?

  3. 🔧 Mitigate (2 minutes)

    Kill switch if harmful content. Rate limit if cost spike. Rollback prompt if quality issue.

  4. 📢 Communicate (10 minutes)

    In-app status if user-facing. Notify leadership if significant.

  5. 🔬 Root Cause (hours/days)

    Model update? Prompt regression? Attack? Find the why.

  6. 🛡️ Prevent (days)

    Add monitoring, tests, or guardrails. Update the playbook.

📌 Model & API Versioning Rules

  • Pin model versions: Use gemini-2.0-flash, not gemini-flash-latest. "Latest" changes without warning
  • Test before upgrading: Run quality tests (Ch. 11) on new versions before switching
  • Prompts in Remote Config: Update prompts without app updates. A/B testable instantly
  • Graceful deprecation: Fall back to replacement APIs. Never crash on deprecation

🧪 Quick Quiz

Why should you never ship an AI feature to 100% of users on day one?

AI features can behave unexpectedly with real data, real edge cases, and real bad actors. Gradual rollout (1% → 10% → 50% → 100%) catches problems at small scale before they become headlines or huge bills. An instant kill switch lets you dial back at any stage.

What are the three layers of content filtering and why do you need all three?

Layer 1: Model safety settings (API-level). Layer 2: Output validation (format, PII, length). Layer 3: Domain-specific filters (your app's context). One layer failing doesn't mean harmful content reaches the user — that's defense in depth.

What's the most important prompt injection defense?

Instruction/data separation — use the API's message roles (system, user) rather than concatenating user input into the prompt string. This prevents user text from being interpreted as instructions.

Why should rate limit messages be friendly, not punishing?

The user isn't doing anything wrong — they're just using the feature a lot. Framing limits positively ("you've been busy!"), showing reset times, and offering manual fallbacks keeps the user experience positive even at the limit.

🎯

The One Thing to Remember

AI is a tool, not a product. The product is the experience you build for your user. The best AI features are invisible — the user doesn't think "wow, what great AI." They think "wow, this app understands me."

Key Takeaways

  • Complete the 6-point safety checklist (content, cost, privacy, abuse, fallback, kill switch) before shipping any AI feature.
  • Implement multi-layer rate limiting (per-minute, per-hour, per-day, global circuit breaker) with friendly user messages.
  • Use defense in depth for content filtering: model settings + output validation + domain-specific filters.
  • Gradual rollout (1% → 10% → 50% → 100%) with Firebase Remote Config and an instant kill switch. Never go 0-to-100.
  • Have an incident response playbook ready before launch: detect → assess → mitigate → communicate → root cause → prevent.
