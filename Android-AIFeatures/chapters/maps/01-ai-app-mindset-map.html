The Big Idea

AI features are probabilistic, not deterministic. They will be wrong sometimes — and that's OK. Design for the error case first, keep the cost of being wrong low, and accelerate things users already want to do.

🔀 The Fundamental Shift

Aspect Traditional (Deterministic) AI-Powered (Probabilistic)
Output Same every time (2+2=4) "Probably" correct (94% cat)
Errors Bugs — fix once, gone forever Inherent — always possible, design around them
Testing Assert exact results Measure accuracy, monitor quality over time
Lifecycle Build → Ship → Done Build → Ship → Monitor → Improve → ♻️
UX Show the result Show the result + escape hatch + feedback loop

🎯 The Four AI Feature Patterns

👻
Invisible Intelligence
AI works behind the scenes — users never think about it. Gmail spam filter: it just works.
🦸
Augmented Capability
Gives users a superpower they didn't have. Google Photos: search by "dog at the park."
⚡
Reduced Friction
Removes tedious steps. Smart Compose: tap to accept the predicted sentence.
🔮
Proactive Assistance
Anticipates needs before you ask. Google Maps: "Leave by 5:15 PM to arrive on time."

📊 The AI Feature Spectrum

  1. Classification (Low Risk)

    Is this a receipt? Yes/No. Simple, safe, great starting point.

  2. Autocomplete (Low-Med Risk)

    Search suggestions, word predictions. Wrong guess? User just keeps typing.

  3. Summarization (Medium Risk)

    TL;DR for articles. User should always see the original too.

  4. Content Generation (Med-High Risk)

    Draft emails and replies. AI creates a starting point, user edits.

  5. Conversational AI (High Risk)

    In-app chatbot. Open-ended, hard to control, expensive.

  6. Autonomous Agents (Very High Risk)

    AI takes actions on behalf of the user. Maximum capability, maximum danger.

🚫 Five Anti-Patterns to Avoid

🤷
AI for AI's Sake
Adding AI where a dropdown would work better. Always ask: is AI the best tool?
📦
The Black Box
AI decides with no explanation. Users need enough context to trust recommendations.
💤
Set and Forget
Ship and never look again. Without monitoring, you won't know when quality degrades.
🎪
The Over-Promise
"AI-powered" label on a keyword match. Users who've seen real AI will notice.
🍳
The Kitchen Sink
Five AI features at once instead of nailing one. Ship one excellent feature first.

📖 Key Terms

Deterministic
Same input always produces the same output. Traditional features work this way.
Probabilistic
Output is a "best guess" with a confidence level. AI features work this way.
Probabilistic UX
Designing interfaces that work gracefully when the AI is wrong.
Confidence Score
How sure the model is about its prediction (0-100%). Translate into behavior, not numbers.
On-Device Inference
Running the AI model locally on the phone. Fast, free, private, but limited capability.
Cloud AI API
Sending data to a remote server for AI processing. Powerful but costs money and needs internet.
Escape Hatch
A way for the user to dismiss, correct, or override the AI's suggestion. Every AI feature needs one.
Feedback Loop
Mechanism for the AI to learn from user corrections (thumbs up/down, accept/dismiss).

🧪 Quick Quiz

Test Your Understanding

What's the difference between deterministic and probabilistic features?

Deterministic features produce the same output every time (2+2=4). Probabilistic features produce a "best guess" that may vary and can be wrong ("this is probably a cat, 94% confidence").

Name the four AI feature patterns.

Invisible Intelligence (works behind the scenes), Augmented Capability (gives superpowers), Reduced Friction (removes tedious steps), and Proactive Assistance (anticipates needs).

Why should you "design for the error case first"?

Because AI will be wrong regularly. If you design what happens when the AI is wrong before building, you ensure the experience stays good even when the AI isn't perfect. Five questions to answer: what does the user see when AI is right/wrong, how do they correct it, does AI learn, is there a fallback?

What's the "magic vs. useful" trap?

Building an impressive demo that wows stakeholders but nobody actually uses in production. A 95%-accurate classifier solving a real pain point beats a flashy chatbot nobody needs. The test: does it save the user time, effort, or frustration?

Should you show raw confidence percentages to users?

No. Users think in "right or wrong," not "73.4% confident." Translate confidence into behavior: high confidence = just show the result, medium = add an "AI suggested" badge, low = show options, very low = don't show or ask for clarification.

Key Takeaways

  • AI features are probabilistic — they will be wrong, and your UX must handle that gracefully.
  • The four patterns: Invisible Intelligence, Augmented Capability, Reduced Friction, Proactive Assistance.
  • Start low on the complexity spectrum (classification, autocomplete) and work up.
  • Users expect speed (<2s), accuracy, and control (dismiss, edit, undo).
  • Design for the error case first — what happens when the AI is wrong?
  • The golden rule: AI should accelerate something users already want to do.
  • AI features are gardens, not buildings — they need continuous monitoring and improvement.
  • Avoid anti-patterns: AI for AI's sake, black boxes, set-and-forget, over-promising, kitchen sink.
