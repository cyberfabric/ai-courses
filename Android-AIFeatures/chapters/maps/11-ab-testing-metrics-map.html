Chapter 11

A/B Testing & Measuring AI Features

Memory Map
📊

The Big Idea

Traditional features are binary — they work or they don't. AI features have a quality gradient: helpful, somewhat helpful, useless, or harmful. You need new metrics, new experiments, and honest kill criteria to know if your AI feature actually works.

🏆 The AI Quality Scorecard

Every AI feature measured across five dimensions:

🎯
Accuracy
How often is the AI right? Measure via acceptance rate, edit distance, user corrections. Target: 70-95% depending on feature.
⚡
Latency
P50 and P95 response time. Target: <2s for generation, <100ms for suggestions.
💰
Cost
API spend per user per month. Target: <$0.05/user/month for most features. Track daily + monthly.
😊
Satisfaction
Thumbs up/down ratio, explicit feedback. Target: >80% positive.
📡
Coverage
% of requests that get a useful response (not "I don't know"). Target: >90%.

💡 Pick one north star metric. Summarization → acceptance rate. Chatbot → task completion. Recommendations → click-through rate. One metric rules; the rest are supporting evidence.

📏 North Star by Feature

Feature North Star Metric How to Measure
🔍 Smart Search Search success rate % of searches followed by a click (not a re-search)
📝 Summarization Acceptance rate % of summaries used without editing
💬 Chat Task completion % of conversations with intended action performed
📷 Image AI Classification accuracy % of auto-tags the user doesn't correct
🔔 Recommendations Click-through rate % of recommended items engaged with
📋 Smart Forms Edit distance How much the user changes AI-filled fields

📐 The Edit Distance Trick

The most underrated AI metric — measures how much the user changed the AI's output:

AI Output
→
User Edits
→
Final Version
→
Levenshtein Distance ÷ Max Length
Edit Distance Meaning Verdict
0 No edits at all ✅ AI nailed it
<0.1 Minor tweaks ✅ AI was helpful
0.1-0.5 Significant changes ⚠️ AI gave a starting point
>0.5 User rewrote most of it ❌ AI wasn't helpful

🔥 Firebase A/B Testing Pipeline

  1. Define Remote Config Parameter

    Create "summary_prompt_variant" in Firebase Console. Default "A", experiment splits 50/50.

  2. Branch on Variant in Code

    Fetch the variant with remoteConfig.getString(). Use it to select the prompt or model.

  3. Log Outcome Events

    ai_summary_requested, ai_summary_displayed, ai_summary_accepted, ai_summary_dismissed — all tagged with variant.

  4. Wait for Significance

    Firebase calculates statistical significance automatically. Typically 1-2 weeks with a few thousand users. Don't peek early.

👍 The Feedback Hierarchy

Signal User Effort Data Quality Frequency
Implicit (accept/dismiss) Zero Good Every interaction
Edit distance Zero Great Every editable output
Thumbs up/down Low Great Important outputs
"Was this helpful?" Medium Good Weekly, not every time
Free-text feedback High Excellent Only on thumbs-down follow-up

Golden rule: implicit signals from every interaction, explicit feedback from important ones, detailed feedback only when something went wrong.

💸 Set the budget alert BEFORE launch, not after the first bill. A viral moment can 10x traffic overnight. Without a circuit breaker, a $200/month feature becomes a $2,000 surprise.

🔬 Production Quality Gates

  • Response length check: Within expected bounds? Too short = empty; too long = constraints ignored
  • Format compliance: Asked for 3 bullets → got 3 bullets? Parse and validate
  • Language check: Response in expected language?
  • Toxicity scoring: Safety classifier before showing to users
  • Latency tracking: P50, P95, P99 — alert if P95 exceeds budget
  • Weekly human review: Sample 50-100 interactions, review for hallucinations, tone, and format drift

💀 When to Kill an AI Feature

Low Adoption (<10%)
Less than 10% of eligible users try it after 4 weeks. The feature isn't compelling enough.
Low Retention (one-time >70%)
Users try it once and never come back. Novelty without lasting value.
High Cost, Low Value (>$0.10/user/month)
API costs exceed thresholds with no measurable impact on retention or revenue.
Quality Plateau
5+ iterations on prompts and models with no meaningful improvement. You've hit the ceiling.
Negative Feedback (>30% thumbs-down)
Consistently bad ratings. Users are telling you — listen.

💡 Killing a feature isn't failure — it's good product management. Document what you learned, free up API budget for features that work, reduce maintenance burden. The next AI feature attempt starts smarter.

🧪 Quick Quiz

Why can't you measure AI features with traditional binary success metrics?

AI features have a quality gradient (helpful → somewhat helpful → useless → harmful), not a binary pass/fail. The same feature can work great for one user and fail for another. You need metrics like acceptance rate, edit distance, and satisfaction scores.

What's the most underrated AI metric and why?

Edit distance. It measures how much the user changed the AI's output, giving you a precise, zero-effort quality signal. Distance of 0 = perfect. Under 0.1 = minor tweaks. Over 0.5 = user rewrote everything.

Why shouldn't you change the model and prompt simultaneously in an A/B test?

You can't isolate which change caused the effect. If quality improves, was it the better model or the better prompt? Change one variable at a time for clean, actionable results.

When should you collect free-text feedback from users?

Only as a follow-up to thumbs-down. Free-text feedback has high user effort, so use it sparingly. Collect implicit signals (accept/dismiss) from every interaction, and reserve detailed feedback for when something went wrong.

Key Takeaways

  • AI features need a quality scorecard: accuracy, latency, cost, satisfaction, and coverage — with one north star metric per feature.
  • Edit distance is the most precise, zero-effort quality metric. Track the distribution, not just the average.
  • Run two types of A/B tests: feature tests (AI on vs off) and quality tests (model/prompt A vs B). One variable at a time.
  • Set cost alerts before launch. Log every API call. Circuit breaker at budget limit. A viral moment can 10x your bill overnight.
  • Know when to kill a feature: low adoption, one-time trial, high cost with no impact, quality plateau, or persistent negative feedback.
