Smart Search & Autocomplete

Search that understands intent, not just keywords — semantic search, suggestions, and AI-powered ranking

Beyond Keyword Matching

Your app has a search bar. A user types "that blue jacket from last week." Your database has an item called "Navy Denim Jacket — Added March 24." Traditional keyword search returns zero results because no field contains the words "blue," "that," or "last week."

This is the fundamental limitation of keyword search: it matches strings, not meaning. And users don't think in database column names — they think in fuzzy, human, contextual language.

Scenario: Priya uses a recipe app with 2,000 saved recipes. She's hungry and types "something spicy with chicken that's quick." Keyword search finds nothing — no recipe title contains that exact phrase. Semantic search understands the intent and returns "15-Minute Sriracha Chicken Stir-Fry," "Quick Tandoori Chicken Bites," and "Spicy Chicken Quesadilla." Priya taps the first result and starts cooking.

That's the difference between search that matches text and search that understands meaning.

AI-powered search isn't just nicer — it's a fundamentally different approach. And it's more accessible than you think.

Text Embeddings 101

The foundation of semantic search is text embeddings — turning words into numbers that capture meaning.

Imagine a giant map where every word and phrase has a location. "Happy" and "joyful" are neighbors. "Happy" and "socket wrench" are on opposite sides. An embedding model converts text into a point on this map — technically a vector of 384-768 numbers.

The magic: you can measure the distance between two points. Similar meanings = close together. Different meanings = far apart. Search becomes "find the content closest to the user's query on the map."

384 Dimensions in a typical small embedding (all-MiniLM)
22MB Model size for on-device semantic search
<50ms Time to embed a query on a modern phone

Building a Local Semantic Search Index

The simplest approach to semantic search works entirely on-device. Here's the architecture:

Content
→

→
Store Vectors
→
Query
→
Rank by Similarity

Step 1: Embed your content. When the user adds content (recipes, notes, products), run each item through the embedding model and store the vector alongside the item in your database.

Step 2: Embed the query. When the user types a search, embed their query into the same vector space.

Step 3: Find nearest neighbors. Calculate the distance between the query vector and every content vector. Return the closest matches.

Let's build this for real.

Recipe: On-Device Semantic Search with TFLite

Goal: Search recipes by meaning, not keywords, using an on-device embedding model.

Step 1: Add TFLite to your project:

implementation("org.tensorflow:tensorflow-lite:2.16.1")
implementation("org.tensorflow:tensorflow-lite-support:0.4.4")

Step 2: Load the embedding model and create a helper:

class EmbeddingHelper(context: Context) {
    private val interpreter = Interpreter(
        FileUtil.loadMappedFile(context, "all-MiniLM-L6-v2.tflite")
    )

    fun embed(text: String): FloatArray {
        val input = tokenize(text) // tokenizer for MiniLM
        val output = Array(1) { FloatArray(384) }
        interpreter.run(input, output)
        return output[0]
    }
}

Step 3: Store embeddings alongside your content in Room:

@Entity
data class Recipe(
    @PrimaryKey val id: Long,
    val title: String,
    val ingredients: String,
    val embedding: FloatArray // store as BLOB
)

// When adding a recipe:
val embedding = embeddingHelper.embed("$title $ingredients")
dao.insert(recipe.copy(embedding = embedding))

Step 4: Search by cosine similarity:

fun search(query: String, recipes: List<Recipe>): List<Recipe> {
    val queryVec = embeddingHelper.embed(query)
    return recipes
        .map { it to cosineSimilarity(queryVec, it.embedding) }
        .sortedByDescending { it.second }
        .take(10)
        .map { it.first }
}

fun cosineSimilarity(a: FloatArray, b: FloatArray): Float {
    val dot = a.zip(b).sumOf { (x, y) -> (x * y).toDouble() }
    val normA = sqrt(a.sumOf { (it * it).toDouble() })
    val normB = sqrt(b.sumOf { (it * it).toDouble() })
    return (dot / (normA * normB)).toFloat()
}

Now "something spicy with chicken" will match recipes about spicy chicken dishes, even if the title says "Sriracha Stir-Fry." The vectors capture the semantic relationship.

Where to get the model: Download all-MiniLM-L6-v2 in TFLite format from Hugging Face or the TensorFlow Hub. It's ~22MB — small enough to bundle in your APK or download on first launch.

Cloud-Powered Semantic Search

On-device embeddings work great for small-to-medium datasets (up to a few thousand items). For higher quality or larger datasets, use a cloud embedding API:

Provider Model Dimensions Best For
Google Gemini Embedding 768 Multimodal (text + images), Google ecosystem
OpenAI text-embedding-3-small 1536 High quality text embeddings, widely adopted
Cohere embed-v3 1024 Multilingual, search-optimized

The typical pattern: batch-embed your content when it's created or updated (server-side), store the vectors in a database, and embed the user's query in real time for search.

Don't embed on every search. Embedding the content only happens once (when content is added). Only the search query needs real-time embedding. For a recipe app with 2,000 recipes, you embed 2,000 items once and only embed the query at search time.

Autocomplete That Predicts Intent

Great search doesn't wait for the user to finish typing. It starts suggesting as they go.

The Two-Tier Autocomplete Pattern

The best autocomplete systems combine two approaches:

  1. Tier 1 — Prefix matching (instant, on-device): As the user types "chic", show items that start with "chic" — "Chicken Parmesan," "Chickpea Curry." This is instant (<5ms) and requires no AI.
  2. Tier 2 — Semantic prediction (fast, AI-powered): After a short debounce (200-300ms), run the query through your embedding model to find semantically related results. "chic" now also surfaces "Poultry Stir-Fry" and "Hen Roast" — items that don't start with "chic" but are related.

Show Tier 1 results immediately. Blend in Tier 2 results as they arrive. The user sees instant suggestions that get smarter as they keep typing.

Real-world example: Google Search does exactly this — instant prefix suggestions appear immediately, then AI-powered suggestions (trending queries, personalized predictions) blend in within a few hundred milliseconds.

Typo Tolerance and Fuzzy Matching

Users make typos. A lot. "receipe" should find "recipes." "kotlin coroutenes" should find "Kotlin Coroutines." If your search is strict, it feels broken.

The good news: semantic search already handles many typos naturally, because the embedding model is trained on noisy text. "reciepe" and "recipe" have similar vectors. But for extreme typos, combine with Levenshtein distance:

fun fuzzyMatch(query: String, candidates: List<String>): List<String> {
    return candidates
        .map { it to levenshteinDistance(query.lowercase(), it.lowercase()) }
        .filter { it.second <= 2 } // allow up to 2 character edits
        .sortedBy { it.second }
        .map { it.first }
}

// Combine with semantic search for the best of both worlds
fun hybridSearch(query: String, items: List<Recipe>): List<Recipe> {
    val semantic = semanticSearch(query, items)      // meaning-based
    val fuzzy = fuzzyTitleSearch(query, items)        // typo-tolerant
    return mergeResults(semantic, fuzzy).distinct()   // deduplicated
}

The 80/20 rule of typo tolerance: Semantic embeddings handle 80% of typos automatically. Levenshtein distance catches the remaining 20% (severe typos, transposed characters). You rarely need anything more sophisticated.

Search Result Ranking with AI

Raw semantic similarity gets you most of the way, but the best search results also consider context. A re-ranking layer can boost results based on multiple signals:

Signal Weight Example
Semantic similarity Primary How closely the content matches the query's meaning
Recency Moderate Recently added or viewed items rank higher
User history Moderate Items similar to what the user has liked/clicked before
Popularity Light Items that other users have engaged with
Completeness Light Items with photos, full descriptions rank higher than stubs 
fun rerank(results: List<SearchResult>, userId: String): List<SearchResult> {
    return results.map { result ->
        val score = result.semanticScore * 0.6f +
            recencyBoost(result.updatedAt) * 0.15f +
            userAffinityScore(result, userId) * 0.15f +
            popularityScore(result) * 0.1f
        result.copy(finalScore = score)
    }.sortedByDescending { it.finalScore }
}

The weights are a starting point — tune them with A/B testing (Chapter 11). The important thing is that re-ranking happens after semantic search narrows the candidates, so it's fast even with complex scoring.

Building the Search UI in Compose

A beautiful search experience needs more than just results. Here's the complete UI architecture:

The Search Bar

@OptIn(ExperimentalMaterial3Api::class)
@Composable
fun SmartSearchBar(viewModel: SearchViewModel) {
    var query by remember { mutableStateOf("") }
    val suggestions by viewModel.suggestions.collectAsStateWithLifecycle()
    val results by viewModel.results.collectAsStateWithLifecycle()
    var active by remember { mutableStateOf(false) }

    SearchBar(
        query = query,
        onQueryChange = { newQuery ->
            query = newQuery
            viewModel.onQueryChanged(newQuery) // debounced search
        },
        onSearch = { viewModel.search(query) },
        active = active,
        onActiveChange = { active = it },
        placeholder = { Text("Search recipes...") },
        leadingIcon = { Icon(Icons.Default.Search, "Search") }
    ) {
        // Show suggestions while typing
        suggestions.forEach { suggestion ->
            ListItem(
                headlineContent = { Text(suggestion) },
                leadingContent = { Icon(Icons.Default.History, null) },
                modifier = Modifier.clickable {
                    query = suggestion
                    viewModel.search(suggestion)
                }
            )
        }
    }
}

Essential Search States

Every search UI needs to handle these states gracefully:

State What to Show
Empty (no query) Recent searches, trending items, or personalized suggestions
Typing Real-time autocomplete suggestions (Tier 1 + Tier 2)
Loading Skeleton cards or subtle shimmer (not a spinner)
Results Matching items with highlighted match context
No results "No matches for X. Try: [related suggestions]" — never a dead end
Error "Search is temporarily unavailable. Try again?" with fallback to keyword search

The No-Dead-End Rule

A search that returns "No results" with nothing else is a dead end. Always offer an escape route: suggested alternative queries, category browsing, popular items, or a "Search the web?" fallback. The user should never hit a wall.

Personalized Search

Over time, your search should learn what this user cares about. If a user always picks spicy recipes, "chicken" should rank "Spicy Chicken" above "Chicken Salad." This doesn't require a fancy recommendation engine — a simple affinity model works:

// Track what the user clicks in search results
@Entity
data class SearchClick(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val query: String,
    val clickedItemId: Long,
    val timestamp: Long
)

// Build a lightweight preference vector
fun buildUserPreference(clicks: List<SearchClick>, items: Map<Long, Recipe>): FloatArray {
    val recentClicks = clicks.sortedByDescending { it.timestamp }.take(50)
    val embeddings = recentClicks.mapNotNull { items[it.clickedItemId]?.embedding }
    // Average the embeddings of recently clicked items
    return averageVectors(embeddings)
}

The user's preference vector is the average of what they've recently clicked. Blend it with the query vector (e.g., 80% query + 20% preference) for personalized ranking — all on-device, no data leaves the phone.

Privacy win: On-device personalization means the user's search behavior and preferences stay on their phone. No tracking servers, no user profiles in your database, no GDPR headaches. The search gets personal without being creepy.

Performance Optimization

Search needs to feel instant. Here's how to keep it fast on mid-range devices:

Debouncing

Don't run semantic search on every keystroke. Wait for the user to pause:

// In your ViewModel
private val queryFlow = MutableStateFlow("")

init {
    queryFlow
        .debounce(300) // wait 300ms after typing stops
        .filter { it.length >= 2 } // minimum 2 characters
        .distinctUntilChanged()
        .onEach { query -> performSearch(query) }
        .launchIn(viewModelScope)
}

Embedding Cache

Cache query embeddings for repeated searches. Users often search the same things:

private val embeddingCache = LruCache<String, FloatArray>(100)

fun getEmbedding(text: String): FloatArray {
    return embeddingCache.get(text) ?: embeddingHelper.embed(text).also {
        embeddingCache.put(text, it)
    }
}

Pre-compute Popular Queries

If you know the top 50 queries (from analytics), pre-compute their results and store them. When a user types a common query, serve the cached results instantly while semantic search runs in the background.

Watch your memory. Each 384-dimension embedding is ~1.5KB. For 10,000 items, that's ~15MB of vectors in memory. For larger datasets, load embeddings in batches or use an on-device vector database like ObjectBox with vector search support.

The Hybrid Search Stack

For production apps, the best search combines multiple approaches. Here's the stack, from fastest to smartest:

Layer 1: Instant (on-device)

Prefix matching + cached popular queries Response time: <5ms Shows: direct title matches, recent searches

Layer 2: Smart (on-device AI)

Semantic embedding search + fuzzy matching Response time: 30-100ms Shows: meaning-based matches, typo-corrected results

Layer 1 shows results immediately. Layer 2 blends in within a fraction of a second. The user sees the search getting smarter in real time as they type.

For apps that need even higher quality, add a Layer 3: a cloud re-ranking step that uses a large language model to re-score the top 20 results. This adds 200-500ms but can dramatically improve relevance for complex queries.
