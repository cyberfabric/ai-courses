Chapter 6

In-App Chat & Conversational UI

Build a chatbot users actually want to talk to — Compose UI, streaming, memory, and function calling

When a Chatbot Makes Sense

In Chapter 5, you built one-shot AI features — user asks, AI answers, done. A chatbot is different: it's a conversation. The user asks a follow-up, references something from earlier, changes their mind. This is harder to build, more expensive to run, and more ways to go wrong.

So before you build a chatbot, ask: does this problem actually need a conversation?

Chatbot Makes Sense

Customer support: "I can't find my order" → "Which order?" → "The one from last Tuesday" → "Found it, here's the status..." Product discovery: "I need a gift for my mom" → "What does she like?" → "She's into gardening" → "Here are 5 options..." Learning/tutoring: "Explain coroutines" → "What about structured concurrency?" → "Show me an example"

Chatbot Is Overkill

Settings: "Turn on dark mode" → just put a toggle switch Binary choices: "Should I save this?" → Yes/No buttons are faster Simple lookup: "What's the weather?" → show a card, don't start a conversation

If the user's need can be satisfied in one interaction, a chatbot adds friction instead of removing it.

The rule of thumb: use a chatbot when the interaction requires clarification, exploration, or multi-step reasoning. For everything else, use a button, a form, or a one-shot AI feature from Chapter 5.

The Chat Data Model

Every chat feature starts with a solid data model. Here's what you need:

enum class MessageRole { USER, ASSISTANT, SYSTEM }
enum class MessageStatus { SENDING, STREAMING, COMPLETE, ERROR }

@Entity
data class ChatMessage(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val conversationId: String,
    val role: MessageRole,
    val content: String,
    val timestamp: Long = System.currentTimeMillis(),
    val status: MessageStatus = MessageStatus.COMPLETE
)

@Entity
data class Conversation(
    @PrimaryKey val id: String = UUID.randomUUID().toString(),
    val title: String = "New Chat",
    val createdAt: Long = System.currentTimeMillis(),
    val systemPrompt: String = ""
)

Key design decisions: each message has a role (so the AI knows who said what), a status (so the UI can show streaming or error states), and a conversationId (so you can persist multiple conversations).

Building the Chat UI in Compose

A chat UI has three core pieces: the message list, the input field, and the typing indicator. Let's build each one.

The Message List

@Composable
fun ChatMessageList(messages: List<ChatMessage>, listState: LazyListState) {
    LazyColumn(
        state = listState,
        modifier = Modifier.fillMaxSize(),
        contentPadding = PaddingValues(16.dp),
        verticalArrangement = Arrangement.spacedBy(8.dp)
    ) {
        items(messages, key = { it.id }) { message ->
            ChatBubble(message)
        }
    }
}

Message Bubbles

@Composable
fun ChatBubble(message: ChatMessage) {
    val isUser = message.role == MessageRole.USER
    Row(
        modifier = Modifier.fillMaxWidth(),
        horizontalArrangement = if (isUser) Arrangement.End else Arrangement.Start
    ) {
        Surface(
            shape = RoundedCornerShape(16.dp),
            color = if (isUser) MaterialTheme.colorScheme.primary
                    else MaterialTheme.colorScheme.surfaceVariant,
            modifier = Modifier.widthIn(max = 280.dp)
        ) {
            Text(
                text = message.content,
                modifier = Modifier.padding(12.dp),
                color = if (isUser) MaterialTheme.colorScheme.onPrimary
                        else MaterialTheme.colorScheme.onSurfaceVariant
            )
        }
    }
}

The Input Field

@Composable
fun ChatInput(onSend: (String) -> Unit, enabled: Boolean) {
    var text by remember { mutableStateOf("") }
    Row(
        modifier = Modifier.fillMaxWidth().padding(8.dp),
        verticalAlignment = Alignment.CenterVertically
    ) {
        OutlinedTextField(
            value = text,
            onValueChange = { text = it },
            modifier = Modifier.weight(1f),
            placeholder = { Text("Type a message...") },
            enabled = enabled,
            maxLines = 4
        )
        Spacer(Modifier.width(8.dp))
        IconButton(
            onClick = { if (text.isNotBlank()) { onSend(text); text = "" } },
            enabled = enabled && text.isNotBlank()
        ) {
            Icon(Icons.AutoMirrored.Default.Send, "Send")
        }
    }
}

Auto-scroll to the latest message. When a new message arrives or the AI is streaming, scroll to the bottom automatically. But if the user has scrolled up to read history, don't yank them back down. Track scroll position and only auto-scroll when the user is already near the bottom.

Streaming Chat Responses

A chatbot that makes users wait 3 seconds staring at a spinner feels dead. Streaming the response token by token makes it feel alive and responsive. Here's the complete pipeline:

Recipe: Streaming Chat with Gemini API

Goal: Send a message with conversation history, stream the AI's response into the chat bubble in real time.

Step 1: Set up the Gemini chat model:

val model = GenerativeModel(
    modelName = "gemini-2.0-flash",
    apiKey = BuildConfig.GEMINI_API_KEY,
    systemInstruction = content { text(systemPrompt) }
)
// Start a chat session that tracks history automatically
val chat = model.startChat(history = existingMessages.toGeminiHistory())

Step 2: Send a message and stream the response in your ViewModel:

fun sendMessage(userText: String) {
    viewModelScope.launch {
        // Add user message to the list
        addMessage(ChatMessage(role = MessageRole.USER, content = userText))

        // Create a placeholder for the AI response
        val aiMessage = ChatMessage(
            role = MessageRole.ASSISTANT,
            content = "",
            status = MessageStatus.STREAMING
        )
        addMessage(aiMessage)

        try {
            var fullResponse = ""
            chat.sendMessageStream(userText).collect { chunk ->
                fullResponse += chunk.text.orEmpty()
                updateLastMessage(fullResponse, MessageStatus.STREAMING)
            }
            updateLastMessage(fullResponse, MessageStatus.COMPLETE)
        } catch (e: Exception) {
            updateLastMessage("Sorry, something went wrong.", MessageStatus.ERROR)
        }
    }
}

Step 3: Show the streaming indicator in the bubble:

@Composable
fun ChatBubble(message: ChatMessage) {
    // ... existing bubble code ...
    if (message.status == MessageStatus.STREAMING) {
        Spacer(Modifier.height(4.dp))
        LinearProgressIndicator(
            modifier = Modifier.fillMaxWidth().height(2.dp)
        )
    }
}

The Gemini SDK's startChat() handles conversation history automatically — each call to sendMessageStream() includes the full conversation context.

Conversation Memory

The magic of a chatbot is context. "Show me more like that" only works if the AI remembers what "that" refers to. But every AI model has a context window limit, and sending the entire conversation history gets expensive.

The Memory Problem

1M Gemini Flash context window (tokens) — roughly 750K words
128K GPT-4o context window — roughly 96K words
$$$ Cost grows linearly with conversation length

For most conversations (under 50 messages), just send the full history. But for long conversations, use the sliding window + summary approach:

fun buildConversationContext(
    messages: List<ChatMessage>,
    maxMessages: Int = 20
): List<ChatMessage> {
    if (messages.size <= maxMessages) return messages

    // Keep the system prompt + first message for context
    val anchor = messages.take(2)
    // Summarize the middle
    val middle = messages.drop(2).dropLast(maxMessages - 2)
    val summary = ChatMessage(
        role = MessageRole.SYSTEM,
        content = "Summary of earlier conversation: ${summarize(middle)}"
    )
    // Keep the most recent messages
    val recent = messages.takeLast(maxMessages - 3)

    return anchor + summary + recent
}

This gives the AI the full arc of the conversation (via summary) plus recent context (via last N messages), without sending hundreds of messages on every request.

System Prompts That Work

The system prompt defines your chatbot's personality, knowledge boundaries, and behavior. This is the single most important piece of your chatbot — get it right, and the chatbot feels helpful and consistent. Get it wrong, and it feels chaotic.

The Scoped Assistant Pattern

The most successful in-app chatbots are scoped — they have a clear domain, a defined personality, and firm boundaries about what they will and won't do.

  1. Define the domain: "You are a recipe assistant for a cooking app." Not "You are a helpful AI."
  2. Set boundaries: "Only answer questions about cooking, recipes, and ingredients. For anything else, politely say you can only help with cooking."
  3. Define the personality: "Be friendly, encouraging, and concise. Use casual language. Suggest alternatives when possible."
  4. Specify output format: "Keep responses under 3 sentences unless the user asks for detail. Use bullet points for ingredient lists."

Why this works: A scoped assistant is predictable. Users quickly learn what it can do, how to talk to it, and what to expect. An unscoped chatbot is a roulette wheel — sometimes helpful, sometimes bizarre, never consistent.

Vague System Prompt

You are a helpful assistant.

No personality, no boundaries, no format guidance. The chatbot will do anything — including going off-topic, being verbose, and hallucinating.

Scoped System Prompt

You are a recipe assistant for CookMate, a cooking app. Your job: help users find recipes, suggest substitutions, and explain cooking techniques. Boundaries: only answer cooking-related questions. If asked about other topics, say "I'm your cooking assistant — I can help with recipes and cooking tips!" Style: friendly, concise (under 3 sentences unless asked for detail). Use bullet points for lists. Never invent recipes. If unsure, say "I'm not sure about that — try searching the app for more options."

Clear domain, firm boundaries, defined personality, format rules, and an honesty instruction. Consistent results every time.

Tool Use / Function Calling

The most powerful chatbot feature: letting the AI do things in your app, not just talk. When a user says "Show me nearby Italian restaurants," the AI can call a function that queries your database and returns real results.

Scenario: Lena opens her cooking app and asks the chat assistant: "What can I make with chicken and broccoli?" The AI doesn't just generate a generic answer — it calls a searchRecipes function that queries the app's actual recipe database. It returns three real recipes from the app with photos, cook times, and a "Start Cooking" button right in the chat.

Lena taps "Start Cooking" on the stir-fry recipe. The app navigates to the step-by-step cooking screen. The chatbot bridged conversation and action seamlessly.

Here's how function calling works with Gemini:

// Define the tools the AI can use
val tools = listOf(
    Tool(functionDeclarations = listOf(
        FunctionDeclaration(
            name = "searchRecipes",
            description = "Search for recipes by ingredients or keywords",
            parameters = mapOf("query" to Schema.string("Search query"))
        ),
        FunctionDeclaration(
            name = "getRecipeDetails",
            description = "Get full details for a specific recipe",
            parameters = mapOf("recipeId" to Schema.integer("Recipe ID"))
        )
    ))
)

val model = GenerativeModel(
    modelName = "gemini-2.0-flash",
    tools = tools,
    // ... other config
)

When the AI decides to call a function, you execute it locally and return the result:

// In your message handling loop
val response = chat.sendMessage(userText)
for (part in response.candidates.first().content.parts) {
    when (part) {
        is TextPart -> displayText(part.text)
        is FunctionCallPart -> {
            val result = when (part.name) {
                "searchRecipes" -> recipeRepo.search(part.args["query"]!!)
                "getRecipeDetails" -> recipeRepo.getById(part.args["recipeId"]!!)
                else -> "Unknown function"
            }
            // Send the function result back to the AI
            chat.sendMessage(content { functionResponse(part.name, result) })
        }
    }
}

Start with 2-3 tools, not 20. Each tool adds complexity and potential for the AI to make wrong calls. Start with the most useful actions (search, get details, add to favorites) and expand based on user feedback.

Rich Messages Beyond Text

A chat that only shows plain text is boring and limited. Rich messages make the chatbot feel integrated with your app:

sealed class ChatContent {
    data class Text(val body: String) : ChatContent()
    data class RecipeCard(
        val id: Long, val title: String,
        val imageUrl: String, val cookTime: String
    ) : ChatContent()
    data class ActionChips(val options: List<String>) : ChatContent()
    data class ImageMessage(val url: String, val caption: String) : ChatContent()
}

// Render different content types
@Composable
fun RichChatBubble(content: ChatContent) {
    when (content) {
        is ChatContent.Text -> Text(content.body)
        is ChatContent.RecipeCard -> RecipeCardBubble(content)
        is ChatContent.ActionChips -> ChipRow(content.options)
        is ChatContent.ImageMessage -> ImageBubble(content)
    }
}

Action chips are especially useful: after the AI responds, show tappable suggestions like "Show me more," "Something easier," "Add to favorites." This guides the conversation and reduces typing.

Handling Edge Cases

Empty Conversations

A blank chat screen is intimidating. Show a greeting and suggested prompts:

if (messages.isEmpty()) {
    Column(horizontalAlignment = Alignment.CenterHorizontally) {
        Text("Hi! I'm your cooking assistant.")
        Text("Try asking me:", style = MaterialTheme.typography.labelMedium)
        listOf("What can I make with pasta?",
               "Quick dinner ideas",
               "How do I poach an egg?"
        ).forEach { suggestion ->
            SuggestionChip(onClick = { onSend(suggestion) },
                label = { Text(suggestion) })
        }
    }
}

Very Long Conversations

After 200+ messages, LazyColumn performance can degrade. Paginate: load the latest 50 messages, and load older ones when the user scrolls to the top.

Network Errors Mid-Stream

If the connection drops while the AI is streaming, keep the partial response and show a "Connection lost. Tap to retry" button. Don't discard what was already received.

Content filtering is non-negotiable. Users will test your chatbot with offensive, harmful, or manipulative input. Use the model's built-in safety settings, add your own keyword filters for your domain, and always have a graceful "I can't help with that" response. Never let the chatbot be weaponized.

Rate Limiting and Cost Control

A chatbot is the most expensive AI feature to run. Each message sends the entire conversation history, and talkative users can generate hundreds of API calls per session.

Limit Type Recommended Value User Message
Per-minute 10 messages "Slow down a bit — I need a moment to think!"
Per-day (free tier) 50 messages "You've used today's messages. Resets at midnight."
Per-conversation 100 messages "This conversation is getting long. Start a new one?"
Global circuit breaker Cost > $X/hour "Chat is temporarily unavailable." (internal alert)

Cost Math for Chat

A 20-message conversation with Gemini Flash costs roughly $0.001-0.005 (depending on message length). At 10K DAU with 3 conversations/day, that's $30-150/month. Manageable. But at 100K DAU, it's $300-1,500/month. Budget before you ship, and set the circuit breaker before you need it.

Offline Chat

For devices with Gemini Nano (Pixel 8+, Samsung S24+), you can offer a limited offline chat experience. The quality is lower and context is shorter, but it works without internet:

suspend fun sendChatMessage(text: String): Flow<String> {
    return if (isOnline()) {
        geminiFlashChat.sendMessageStream(text)
    } else if (isGeminiNanoAvailable()) {
        geminiNanoChat.sendMessageStream(text)
    } else {
        flowOf("I need an internet connection to chat. Try again when you're online.")
    }
}

Show a subtle indicator when in offline mode: "Offline mode — responses may be shorter." Set expectations before the user notices the quality difference.
