Chapter 4

RAG: Teaching AI Your Data

Memory Map
💡

The Big Idea

RAG = look up the relevant info, paste it into the prompt, let the model answer from it. You're not changing the model — you're changing what it sees. Like giving a smart person the right reference book before asking a question.

📖 What RAG Stands For

Retrieval
Find relevant pieces
from your docs
→
Augmented
Add them to
the prompt
→
Generation
LLM answers using
retrieved context

🔧 The RAG Pipeline

Two phases: ingestion (prepare once) and querying (run per question).

  1. 📄 Chunk

    Break documents into small, self-contained pieces (200-1000 tokens). Split by paragraph/section, add overlap.

  2. 🔢 Embed

    Turn each chunk into a vector (list of numbers) using an embedding model. Similar meaning = similar vectors.

  3. 💾 Store

    Save chunks + vectors in a vector database. Enables searching by meaning, not keywords.

  4. 🔍 Query

    User asks a question → embed the question → find the top 3-5 most similar chunks.

  5. 🤖 Generate

    Send retrieved chunks + question to the LLM. It reads the context and generates an answer.

⚖️ RAG vs Fine-Tuning

Factor RAG Fine-Tuning
Purpose Model needs to KNOW things Model needs to DO things differently
Example "Answer from our docs" "Use our brand voice"
Data freshness ✅ Always current ❌ Frozen at training time
Setup cost Low (hours) High (days + GPU)
When data changes Re-index (minutes) Retrain (hours/days)
Can cite sources? ✅ Yes, verifiable ❌ Internalized, may hallucinate

Need both? Fine-tune for behavior, RAG for data access. Most powerful combination.

⚠️ When RAG Fails (and How to Fix It)

✂️
Bad Chunking
Chunks break mid-sentence or mix topics. Fix: use paragraph boundaries, add overlap.
🎯
Irrelevant Retrieval
Results are vaguely related but don't answer the question. Fix: better embedding model, increase k.
👻
Missing Info
Answer exists but embedding doesn't match it. Fix: hybrid search (vector + keyword), add summaries.
💥
Context Overflow
Too many chunks for the context window. Fix: fewer, more relevant chunks. Larger context model.

#1 mistake: blaming the LLM when the real problem is retrieval. Print your chunks first!

🧪 Quick Quiz

What does RAG stand for and what's the one-line explanation?

Retrieval-Augmented Generation. Find relevant docs, paste them into the prompt, let the LLM answer from them.

Your RAG chatbot gives bad answers. What should you check FIRST?

The retrieved chunks. Print what the vector search returns. If the chunks are irrelevant, fix retrieval before touching the LLM prompt.

"I want my AI to always respond in our brand voice." Is that RAG or fine-tuning?

Fine-tuning. RAG gives the model access to data. Fine-tuning changes how it behaves (tone, format, style).

What's a good starting chunk size?

300-500 tokens with 50-token overlap. Too small = no context. Too large = diluted relevance.

Which vector database should you start with for prototyping?

Chroma. Zero setup, runs in-process with Python. Move to pgvector or Pinecone for production.

📌 Key Takeaways

  • RAG = retrieve relevant docs + paste into prompt + generate. You don't change the model, you change what it sees.
  • The pipeline: Chunk → Embed → Store → Query → Retrieve → Generate.
  • RAG for data access, fine-tuning for behavior. Need both? Combine them.
  • Start with Chroma (vector DB) + nomic-embed-text (embedding) for prototyping.
  • When results are bad, check retrieval first — bad chunks are usually the culprit, not the LLM.
  • Chunk size matters: 300-500 tokens, paragraph boundaries, overlap.
  • For Android: your app makes an HTTP request to a backend that runs RAG. The app doesn't touch vectors.
  • RAG is the most practical AI pattern for app developers. Every "chat with your docs" product uses it.
