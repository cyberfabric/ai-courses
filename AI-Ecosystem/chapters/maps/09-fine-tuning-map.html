Chapter 9

Fine-Tuning: LoRA & QLoRA

Memory Map
💡

The Big Idea

RAG changes what the model sees. Fine-tuning changes how the model behaves. They solve different problems. LoRA and QLoRA made fine-tuning go from "you need a GPU cluster" to "single GPU, under $10, over lunch." The base model is a trained chef — LoRA just gives it a recipe card with your preferences.

🎯 Five Signs You Need Fine-Tuning

📋
Consistent Format
Specific JSON schema, report template, or structure. Prompting gets 90%; fine-tuning gets 99%.
🏥
Domain Language
Medical, legal, or technical terminology the model gets subtly wrong.
🗣️
Brand Voice
Responses should sound like your company, not generic ChatGPT-ish.
⚙️
Specific Behavior
"Always bullet points," "never use markdown." When system prompts are unreliable.
🏎️
Smaller, Faster Model
A fine-tuned 7B can match a 70B on your specific task. 10x faster, 1/10th the cost.

Start with QLoRA. The quality difference is negligible; the memory savings are massive.

⚖️ RAG vs Fine-Tuning: The Decision

Question Answer Solution
Model doesn't know the answer Knowledge problem ✅ RAG
Model knows but says it wrong Behavior problem ✅ Fine-tuning
Needs your data AND your style Both ✅ RAG + Fine-tuning
Data changes frequently Live data needed ✅ RAG (always current)
Need consistent format/tone Style problem ✅ Fine-tuning
Need to cite sources Verifiability ✅ RAG (sources are retrievable)

🌍 Common Use Cases

📋 Structured output
Always respond in a specific JSON format. ~200+ input/output pairs.
🗣️ Brand voice
Match your company's writing style. ~500+ examples of your actual content.
🏥 Domain expertise
Medical, legal, or technical terminology. ~1000+ domain Q&A pairs.
💻 Code style
Follow your team's coding conventions. ~500+ code samples.
🏷️ Classification
Categorize inputs consistently. ~500+ labeled examples.
🔬 Model distillation
Teach a small model to mimic GPT-4 on your task. ~1000+ large-model responses as targets.

🧪 Quick Quiz

What's the difference between RAG and fine-tuning in one sentence?

RAG gives the model access to data it doesn't have (knowledge). Fine-tuning changes how the model behaves (style, format, tone).

What does LoRA actually do?

Adds tiny adapter matrices alongside the frozen model weights. Only the adapters are trained (0.1-1% of parameters). The base model stays unchanged. Output: a 10-100 MB adapter file.

How is QLoRA different from LoRA?

QLoRA loads the base model in 4-bit precision (75% less memory) while training the LoRA adapters in full precision. Same quality, much less VRAM. Start with QLoRA.

"Our chatbot needs to access company docs AND always respond in our brand voice." RAG, fine-tuning, or both?

Both. RAG for document access (knowledge problem). Fine-tuning for brand voice (behavior problem). This is the most powerful combination.

What's the #1 mistake in fine-tuning projects?

Bad training data. Data prep is 70-80% of the work. 200 perfect examples beat 5000 sloppy ones. If you don't have quality data, fine-tuning won't help.

📌 Key Takeaways

  • RAG = what the model sees. Fine-tuning = how it behaves. Different problems, different solutions.
  • LoRA trains tiny adapters (0.1-1% of parameters) instead of the whole model. 10-100 MB adapter files.
  • QLoRA adds 4-bit quantization. Fine-tune 7B on 8 GB VRAM, 13B on 16 GB. Start here.
  • The chef analogy: the base model is a trained chef. LoRA gives it a recipe card with your preferences.
  • Data prep is 70-80% of the work. 100-1000 high-quality examples needed. Quality over quantity.
  • Cost reality: QLoRA on 500 examples = 1-2 hours, under $10. Side project budget.
  • Decision tree: model doesn't know → RAG. Model knows but says it wrong → fine-tune. Both → use both.
  • For Android: fine-tuned backend models for better responses, or fine-tuned tiny models (1-3B) for on-device AI features.
