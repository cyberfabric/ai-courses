Chapter 7

Multi-Agent Systems: AutoGen & CrewAI

Memory Map
💡

The Big Idea

Instead of one AI doing everything, give each agent a role, tools, and a goal — then let them collaborate like a dev team. One writes, another reviews, a third tests. Multi-agent adds value when you genuinely need multiple perspectives. But a single well-prompted agent handles 90% of use cases.

🤖 What Makes an Agent

🎭
Role
"You are a senior code reviewer." A system prompt that defines the agent's identity and expertise.
🔧
Tools
Functions the agent can call: web search, run code, query databases. Each agent gets tools relevant to its role.
🎯
Goal
What the agent is trying to accomplish. A clear objective keeps the agent focused and on-track.

Think microservices: each has a focused responsibility, they communicate through defined interfaces, together they do more than one could alone.

🆚 AutoGen vs CrewAI

Aspect AutoGen (Microsoft) CrewAI

Analogy 🗣️ Slack brainstorm 📋 Project board with tickets
Structure Flexible, free-form Structured, opinionated
Killer feature Code execution in Docker sandbox Role/backstory/task pattern
Learning curve Moderate — more decisions to make Lower — more built-in guidance

Start with CrewAI for structured results. Use AutoGen for exploratory/code-heavy tasks.

✅ Where Multi-Agent Actually Helps

💻 Code Review Pipeline
Writer → Bug reviewer → Security reviewer. Separate agents catch more than one "write and review yourself" agent.
📚 Research & Synthesis
Agents specialize by source type (papers, news, forums). A synthesizer combines findings into a coherent report.
✍️ Content Creation
Researcher → Writer → Editor → Fact-checker. Each step has a quality gate that catches mistakes.
📊 Data Analysis
Data cleaner → Statistician → Visualizer → Narrator. Specialized tools per agent.

🎈 Hype vs Reality

Works Well ✅ Overhyped ❌
Tasks where self-review improves quality Simple tasks one LLM call handles fine
Pipelines with clear stages and handoffs When agents just agree (echo chamber)
Code gen with iterative testing When debate adds latency, not quality
Research from multiple source types When bottleneck is LLM knowledge, not perspective
Any task where "a second opinion" catches errors Demos that look impressive but cost $5/run

💰 The Cost Reality

3 agents × 5 conversation turns = ~15X the cost of a single LLM call:

🎯 The "Do I Need Multi-Agent?" Test

  1. Build it with a single agent first

    One well-prompted LLM with tools. See where the output falls short.

  2. Ask: where did it fail?

    "Didn't catch its own mistakes" → add a reviewer agent. "Missed a perspective" → add a specialist. "Didn't know enough" → use RAG, not more agents.

  3. Measure the improvement

    Can you prove multi-agent produces better results? If the improvement is marginal, stick with single-agent.

  4. Consider the cost

    Is this a high-value task (code review, research report) or a high-volume one (chatbot responses)? Multi-agent suits the former.

🧪 Quick Quiz

What are the three things that define an agent in multi-agent systems?

A role (system prompt defining identity), tools (functions it can call), and a goal (what it's trying to accomplish).

What's the key difference between AutoGen and CrewAI?

AutoGen is conversation-driven (agents chat freely, like a Slack brainstorm). CrewAI is task-driven (agents complete assigned tickets, like a project board).

A single agent gives mediocre answers because it doesn't know enough about your domain. Should you add more agents?

No. The bottleneck is knowledge, not perspective. Use RAG to give the agent access to your data. More agents with the same knowledge won't help.

Why is self-review by a separate agent better than asking one agent to "write and review"?

A single agent tends to be too forgiving of its own work. A separate reviewer agent with a different role and system prompt approaches the output with fresh eyes and catches mistakes the writer overlooked.

What's AutoGen's killer feature?

Code execution in a sandboxed Docker container. The coder agent writes Python, AutoGen runs it, the agent sees the actual output (including errors), and iteratively debugs until it works.

📌 Key Takeaways

  • Agent = role + tools + goal. Multiple agents collaborate like a dev team, each with a focused responsibility.
  • AutoGen = Slack brainstorm (conversation-driven). CrewAI = project board (task-driven). Start with CrewAI for structure.
  • Real value: code review, research synthesis, content pipelines — tasks where a second opinion catches errors.
  • The cost problem: 3 agents × 5 turns = ~15X cost. Only worth it for high-value tasks.
  • 90% rule: a single well-prompted agent with tools handles most use cases. Build single-agent first, add agents only when you can measure the improvement.
  • Knowledge gap? Use RAG, not more agents. Perspective gap? Add a reviewer agent.
  • Local agents are fine for development. Use cloud models for production, or hybrid (cloud lead + local support).
  • For Android: multi-agent runs on the backend. Your app consumes results through a standard API.
