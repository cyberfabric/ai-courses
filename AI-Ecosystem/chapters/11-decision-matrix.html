Chapter 11

The Decision Matrix

Which tool for which job — a practical framework for choosing the right AI stack

The Paradox of Choice

You've now learned about model hubs, local inference, RAG, frameworks, agents, multi-agent systems, UIs, fine-tuning, and automation. That's a lot of tools. And if you're feeling overwhelmed about which ones to actually use, that's completely normal.

Here's the secret: you will never use all of these tools on one project. Most AI features need 2-3 tools. The skill isn't knowing every tool deeply — it's knowing which 2-3 to reach for when you have a specific problem.

This chapter gives you that skill. We'll start with the problem, not the tool.

Start With the Problem

When someone says "let's add AI to our app," the first question isn't "which framework should we use?" It's "what problem are we solving?" The answer to that question narrows your toolset immediately.

The Decision Tree

"I need AI to know my data" → RAG (Ch. 4) + a vector database + an LLM

"I need AI to behave differently" → Fine-tuning (Ch. 9) with LoRA/QLoRA

"I need a multi-step AI workflow" → LangChain (Ch. 5) or LangGraph (Ch. 6)

"I need multiple AI perspectives" → AutoGen or CrewAI (Ch. 7)

"I need a quick demo" → Gradio (Ch. 8)

"I need business automation" → n8n (Ch. 10)

"I need privacy or zero cost" → Ollama (Ch. 3)

"I need to find or share models" → HuggingFace (Ch. 2)

Most projects map to one or two of these statements. If you're trying to address five of them at once, you're probably overcomplicating things.

Common Architecture Patterns

Instead of figuring out everything from scratch, start with a proven pattern and adapt it. Here are the four most common AI architectures you'll encounter:

Pattern 1: RAG Chatbot

Use case: "Chat with our docs" — a chatbot that answers questions about your specific content.

Documents → Embed (HuggingFace) → Vector DB (Chroma) → Retrieve + LLM → Answer
Component Budget Option Production Option
Embedding model nomic-embed-text (Ollama, free) text-embedding-3-small (OpenAI)
Vector database Chroma (in-process, free) pgvector or Pinecone
LLM Llama 3.1 (Ollama, free) GPT-4 or Claude (API)
Framework LangChain or raw Python LangChain + LangSmith
UI Gradio ChatInterface Custom frontend (React/Vue)

Complexity: Medium. Time to prototype: A day. Time to production: 1-2 weeks.

Pattern 2: AI-Powered Mobile Backend

Use case: Adding AI features to your Android/iOS app — smart search, content recommendations, natural language interface.

Mobile App → Your Backend API → Cloud LLM API → Response
Component Simple Advanced
Backend FastAPI + OpenAI SDK FastAPI + LangChain
LLM GPT-4 / Claude API Same + Ollama fallback
Data access System prompt with context RAG pipeline
Mobile client HTTP calls (Retrofit) Same + streaming (SSE)

Complexity: Low to medium. Key insight: From the mobile side, it's just a REST API. Start with direct API calls; add RAG or LangChain only when needed.

For most mobile AI features, the "simple" column is the right starting point. A FastAPI backend that calls the OpenAI API and returns JSON is something you can build in an afternoon. Don't add LangChain, vector databases, or agents until you've confirmed the basic approach works.

Pattern 3: Internal Automation

Use case: Automating repetitive business tasks — email classification, report generation, data extraction.

Trigger (Email/Webhook) → n8n Workflow → AI Processing (Ollama) → Output (DB/Slack/Email)
Component Privacy-First Quality-First
Automation n8n (self-hosted) n8n (cloud or self-hosted)
LLM Ollama (local, free) GPT-4 or Claude (API)
Data storage Your own database Your own database

Complexity: Low. Key insight: No code required. This is the fastest path from "I have a repetitive task" to "it's automated."

Pattern 4: Research / Agent Assistant

Use case: AI that researches, analyzes, and synthesizes — with tool use, iteration, and quality checks.

User Query → LangGraph Agent → Tools (Search, DB, APIs) → Evaluate & Refine → Answer

Complexity: High. Key insight: This is the most powerful pattern and the most expensive. Use it only when simpler approaches (direct API calls, RAG) aren't sufficient. The iteration loop is what makes it smart — and expensive.

Pattern 4 is where most developers over-engineer. Before building a LangGraph agent with tools, ask: would a simple RAG pipeline answer the question? Would a single LLM call with a good prompt work? Start at Pattern 1 or 2 and escalate to Pattern 4 only when you've confirmed simpler approaches don't cut it.

Cost Comparison

Money matters. Here's what different approaches actually cost:

Approach Cost Per 1,000 Queries Quality Setup Effort
GPT-4 API (direct) $10-30 Excellent Minutes
Claude API (direct) $8-25 Excellent Minutes
GPT-4 + RAG $15-40 Excellent + your data Days
Ollama (local, 8B model) $0 (electricity only) Good Minutes
Ollama + RAG $0 Good + your data Days
Fine-tuned 7B model (Ollama) $0 (after training cost of ~$10) Very good for specific tasks Days
Multi-agent (3 agents, GPT-4) $50-150 Excellent for complex tasks Weeks

Notice the pattern: simplicity is cheap, complexity is expensive. A direct API call costs cents. A multi-agent system costs dollars. Make sure the quality improvement justifies the cost increase.

Complexity Comparison

Simple Direct API calls, n8n workflows. Hours to build. Perfect for most features.
Medium RAG pipelines, LangChain chains. Days to build. When you need your own data.
Complex LangGraph agents, multi-agent, fine-tuning. Weeks to build. When nothing simpler works.

The "Keep It Simple" Rule

If a direct API call solves your problem, don't use LangChain.

If a chain solves it, don't use LangGraph.

If one agent solves it, don't use three.

If RAG solves it, don't fine-tune.

Every layer of complexity you add is a layer you have to debug, maintain, and pay for. Start simple. Add complexity only when you can point to the specific limitation that requires it.

Stack Recommendations by Situation

Solo Developer, Side Project

  • LLM: Ollama (free) for development, cloud API for production
  • Framework: raw Python + OpenAI SDK (keep it simple)
  • UI: Gradio for demos, simple web frontend for production
  • Avoid: LangChain (overkill), multi-agent (expensive), fine-tuning (unnecessary)

Startup Building an AI Feature

  • LLM: Cloud API (GPT-4 or Claude) for quality
  • Framework: LangChain if you need RAG or tools; raw Python if you don't
  • Data: RAG with Chroma or pgvector if you need your own data
  • UI: Custom frontend (your product needs its own look)
  • Prototype first with Gradio, validate with users, then build the real thing

Enterprise Adding AI to Existing Systems

  • LLM: Cloud API with enterprise agreements (Azure OpenAI, AWS Bedrock)
  • Framework: LangChain + LangSmith for observability
  • Automation: n8n for internal workflows (self-hosted for privacy)
  • Data: RAG with pgvector (if you already use Postgres) or Pinecone
  • Consider: Fine-tuning for domain-specific tasks, human-in-the-loop for sensitive actions

Android Developer Adding Smart Features

  • Start: Cloud API (GPT-4/Claude) + FastAPI backend — simplest path
  • If you need your data: Add RAG to your backend
  • If you need privacy: Ollama on your backend server
  • If you need on-device AI: Small fine-tuned model (1-3B) via ONNX Runtime or MediaPipe
  • Prototype: Gradio demo first, then build the native Android UI
  • Your Android code: Retrofit + coroutines, calling your AI backend like any other API

What to Learn First

You don't need to master everything at once. Here's a recommended learning path:

Order What to Learn Why First
1 Direct API calls (OpenAI/Anthropic SDK) Foundation — everything else builds on this
2 Ollama + local models Free practice environment, privacy
3 RAG (embeddings + vector DB) The most practical pattern for real applications
4 Gradio Show your work; get feedback fast
5 LangChain (if your app needs it) Only when raw Python becomes painful
6 Everything else (LangGraph, agents, fine-tuning, n8n) Learn on demand, when a specific project needs it
Steps 1-4 cover 80% of what you'll actually need in practice. Steps 5-6 are "learn when you need them." Don't feel pressured to know everything — knowing when to reach for a tool is more valuable than knowing every tool's API.
Design Three Tech Stacks

For each project below, choose your tools using the decision tree and patterns from this chapter. Name the specific tools and justify each choice:

  1. A customer support chatbot for your mobile app. It should answer questions from your help center documentation. Budget: moderate. Privacy: standard.
  2. An internal tool that classifies incoming bug reports by severity and routes them to the right team on Slack. Budget: minimal. Privacy: important (internal data).
  3. A mobile app feature that summarizes long articles into 3-sentence bullet points, works offline, and runs on the user's device. Budget: low ongoing costs.

Suggested answers:

1. Pattern 2 (mobile backend) + RAG. Cloud LLM API + RAG pipeline with Chroma/pgvector + FastAPI backend + Android client. Prototype with Gradio first.

2. Pattern 3 (internal automation). n8n (self-hosted) + Ollama for privacy. Text Classifier node + Slack integration. No code needed.

3. Fine-tuned small model (Phi-3 or Llama 3.2 1B), quantized, deployed on-device via ONNX Runtime. One-time fine-tuning cost, zero per-request cost.
