Chapter 6

LangGraph: Smarter Workflows

When chains aren't enough — graphs with loops, branches, and state for AI that can think in circles

The Limitation of Chains

In Chapter 5, you learned about LangChain's chains: prompt → LLM → parser → done. Clean, simple, one direction. But real-world AI applications aren't always that tidy.

Imagine you're building an AI research assistant. It needs to:

  1. Take a user's question
  2. Search for relevant information
  3. Read the results and decide if they're good enough
  4. If not, refine the search query and try again
  5. Once the results are good, write a summary
  6. Review the summary for accuracy
  7. If the summary has issues, go back and fix them

See the problem? Steps 4 and 7 go backwards. A chain can't do that. Chains are like a one-way street — they flow in one direction and they're done. What you need is a graph — a workflow that can loop, branch, and make decisions about where to go next.

That's LangGraph.

The One-Sentence Version

LangGraph is a framework for building stateful, graph-based AI workflows where steps can loop, branch, and make decisions about what to do next.

If LangChain chains are like a recipe (step 1, step 2, step 3, serve), LangGraph is like a flowchart — with diamonds for decisions, arrows that go backwards, and loops that repeat until a condition is met.

Most AI applications start as simple chains and then hit a wall when they need to do something smarter: retry on failure, validate their own output, take multi-step actions with feedback loops, or let humans approve intermediate steps. LangGraph gives you a structured way to build these complex workflows instead of hacking together ad-hoc retry logic and if-statements. If you're building anything more sophisticated than "prompt in, answer out," LangGraph is worth knowing about.

The Key Concepts

1. State — The Memory of Your Workflow

Every LangGraph workflow has a state — a Python object (usually a TypedDict) that holds all the information being passed between steps. Think of it as the "working memory" of your graph.

from typing import TypedDict, Annotated
from operator import add

class ResearchState(TypedDict):
    question: str                          # The user's original question
    search_queries: list[str]              # Queries we've tried
    results: Annotated[list[str], add]     # Search results (accumulated)
    summary: str                           # The final summary
    is_good_enough: bool                   # Quality check flag

Every node in the graph can read the current state, do work, and return an updated state. This is fundamentally different from chains, where data flows through but there's no persistent "memory" between steps.

2. Nodes — The Workers

A node is just a Python function that takes the current state, does something, and returns state updates. That's it.

def search_node(state: ResearchState) -> dict:
    """Search for information based on the current query."""
    query = state["search_queries"][-1]
    results = web_search(query)  # your search logic
    return {"results": results}

def evaluate_node(state: ResearchState) -> dict:
    """Check if results are good enough."""
    # Ask an LLM to evaluate quality
    is_good = llm_evaluate(state["results"], state["question"])
    return {"is_good_enough": is_good}

If you've done Android development, nodes are like individual ViewModel functions or UseCase classes — each one has a single job.

3. Edges — The Connections

Edges define the flow between nodes. There are two types:

  • Normal edges: always go from A to B (like chain links)
  • Conditional edges: look at the state and decide where to go next

Conditional edges are the magic. They're what make loops and branching possible:

def should_continue(state: ResearchState) -> str:
    """Decide: try again or move on?"""
    if state["is_good_enough"]:
        return "write_summary"    # Good results → move forward
    if len(state["search_queries"]) >= 3:
        return "write_summary"    # Tried 3 times → give up, use what we have
    return "refine_search"        # Bad results → loop back and try again

This is a routing function. After the evaluate node runs, LangGraph calls this function to decide which node to visit next. It can go forward, backward, or anywhere in the graph.

Putting It Together

Here's how you wire nodes and edges into a graph:

from langgraph.graph import StateGraph, END

graph = StateGraph(ResearchState)

# Add nodes
graph.add_node("search", search_node)
graph.add_node("evaluate", evaluate_node)
graph.add_node("refine_search", refine_node)
graph.add_node("write_summary", summary_node)

# Add edges
graph.set_entry_point("search")
graph.add_edge("search", "evaluate")
graph.add_conditional_edges("evaluate", should_continue)
graph.add_edge("refine_search", "search")   # Loop back!
graph.add_edge("write_summary", END)

app = graph.compile()

And the flow looks like this:

Search → Evaluate → Good enough? → Write Summary

↑ If not good enough, loop back to Search via Refine ↑

That loop — evaluate, decide, maybe go back — is impossible with a simple chain. It's the core reason LangGraph exists.

Why Cycles Matter

The ability to loop changes what's possible with AI applications. Here are patterns that require cycles:

Pattern What Happens Example
Self-correction LLM checks its own output and retries if it's wrong Code generator that runs tests and fixes errors
Iterative refinement Each pass improves the output Research assistant that deepens its search
Tool retry If a tool call fails, try a different approach Search with different keywords when first query fails
Multi-step planning Plan, execute, evaluate, replan Trip planner that adjusts based on availability
Validation loops Output must meet criteria before proceeding JSON output validated against a schema
The most common use of LangGraph in practice is the self-correcting agent: an LLM that tries something, checks if it worked, and retries with adjustments if it didn't. This pattern alone justifies learning LangGraph.

Agent Architectures

LangGraph makes it practical to implement well-known AI agent patterns. Here are the three you'll encounter most:

ReAct (Reason + Act)

The agent thinks about what to do, takes an action (calls a tool), observes the result, and repeats until it has an answer. This is a loop: think → act → observe → think → act → observe → ... → answer.

ReAct in Action

Thought: I need to find the user's order status. Action: call search_orders(user_id="123") Observation: Order #456, shipped 2 days ago, tracking: XYZ Thought: I have the info. Let me respond. Answer: Your order #456 shipped 2 days ago. Tracking: XYZ

The LLM decides which tool to call, reads the result, and decides what to do next. This loop is a graph cycle.

Plan-and-Execute

A planning step creates a list of tasks, then an execution step works through them one by one. After each task, the plan can be revised. Good for complex, multi-step problems.

Tool-Calling Agent

The simplest agent pattern: the LLM has access to tools and calls them as needed. Modern LLMs (GPT-4, Claude, Llama 3.1) natively support "tool calling" — they can output structured tool-call requests that your code executes.

Agent architectures are powerful but expensive. A ReAct loop might call the LLM 5-10 times for a single user query. At cloud API prices, that adds up quickly. Always consider whether a simpler approach (a single chain, a direct tool call) would work before reaching for an agent.

Human-in-the-Loop

One of LangGraph's most practical features: you can pause the graph at any point and wait for human approval before continuing.

Imagine a workflow that drafts an email response to a customer. Before sending, it pauses and shows the draft to a human agent. The human approves, edits, or rejects it. The graph resumes based on the human's decision.

from langgraph.checkpoint.memory import MemorySaver

# Compile with checkpointing (needed for interrupt)
app = graph.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["send_email"]  # Pause before this node
)

# Run until the interrupt point
result = app.invoke(initial_state, config={"configurable": {"thread_id": "1"}})
# → Graph pauses. Show draft to human. Get approval.

# Resume after human approves
app.invoke(None, config={"configurable": {"thread_id": "1"}})

This is crucial for production AI applications. You rarely want AI to take irreversible actions (sending emails, making purchases, updating databases) without human oversight. LangGraph makes this a first-class feature rather than something you hack together.

Checkpointing: Save and Resume

LangGraph can save the complete state of a running graph at every step. This gives you:

  • Fault tolerance: if the process crashes, resume from the last checkpoint
  • Time travel: replay the graph from any previous state (great for debugging)
  • Human-in-the-loop: pause for hours or days, then resume exactly where you left off
  • Branching: fork from a checkpoint to try different approaches

Checkpoints are stored in a backend of your choice (memory for development, SQLite or Postgres for production). Think of it like git for your workflow's state.

LangGraph vs LangChain: When to Use Which

Scenario Use LangChain Use LangGraph
Simple RAG pipeline Yes — retrieve, prompt, answer Overkill
Chatbot with memory Yes — chain + memory Only if it needs complex routing
Agent with tools Simple cases Yes — tool loops, error recovery
Self-correcting workflow Can't — no loops Yes — this is its sweet spot
Multi-step with human approval Not built-in Yes — interrupt/resume is native
Complex branching logic Awkward Yes — conditional edges are clean

The Simple Rule

If your workflow is a straight line (A → B → C), use LangChain chains. They're simpler, more transparent, and easier to debug.

If your workflow has decisions, loops, or human checkpoints, use LangGraph. The graph structure makes these patterns clean instead of hacky.

Most AI applications start as chains and graduate to graphs when they need to get smarter. That's a perfectly normal evolution.

When LangGraph Is Overkill

Let's be honest: most AI applications don't need LangGraph. If you're building:

  • A chatbot that answers questions from docs → a LangChain chain is fine
  • A text classifier → a single LLM call is fine
  • A summarizer → a single LLM call is fine
  • A translation feature → a single LLM call is fine

LangGraph adds complexity. If you don't need cycles, state management, or human-in-the-loop, that complexity buys you nothing. Don't use a graph when a straight line works.

A good heuristic: sketch your workflow on paper. If you draw any arrows that go backwards (loops) or diamonds for decisions, consider LangGraph. If it's all arrows going forward, stick with LangChain or raw Python.

Connection to Android Development

LangGraph workflows run on your backend. From the Android side, you're just calling an API. But the kind of features LangGraph enables are interesting for mobile:

  • Trip planner: search flights → check availability → adjust dates if needed → search hotels → create itinerary → let user approve → book. Multiple loops and a human-in-the-loop step.
  • Smart form filling: extract info from a document → validate → ask user to confirm missing fields → submit. The graph handles missing data gracefully.
  • Research assistant: user asks a question → search multiple sources → evaluate quality → synthesize answer. The iterative refinement produces much better results than a single-shot chain.
  • Multi-step onboarding: analyze user's profile → recommend settings → let user adjust → configure app. The graph adapts to user choices.

The key pattern: any feature where the AI needs to try, evaluate, and adjust benefits from LangGraph. Your Android app sees a streaming response or a series of status updates; the complexity lives in the backend graph.

Learning Curve: Honest Assessment

  • Understanding the concept (1 hour): If you understand flowcharts, you understand LangGraph's model.
  • Simple graph with 3-4 nodes (half a day): The StateGraph API is straightforward once you see a few examples.
  • Conditional edges and loops (1 day): Getting routing functions right takes some trial and error.
  • Human-in-the-loop and checkpointing (1-2 days): The interrupt/resume pattern requires understanding thread IDs and state serialization.
  • Production patterns (ongoing): Error handling, streaming, subgraphs, complex state management.

LangGraph is easier to learn if you already understand LangChain, because it builds on the same concepts (LLMs, tools, prompts). The new concepts — state, nodes, edges, conditional routing — are intuitive if you've ever drawn a flowchart.

  • Your AI workflow needs loops (retry, self-correction, iterative refinement)
  • You need human-in-the-loop approval before the AI takes action
  • You're building an agent that uses tools and makes multi-step decisions
  • You need fault tolerance (crash recovery, long-running workflows)
  • Your workflow has complex branching logic based on intermediate results
  • Your workflow is a straight line (retrieve → prompt → answer)
  • You're making single LLM calls (classification, summarization, translation)
  • You don't need loops or branching — a chain works fine
  • You're just getting started with AI apps — learn chains first
  • The added complexity isn't justified by your use case
Sketch a Graph

Grab a piece of paper (or a whiteboard app) and design a graph for one of these scenarios:

  1. Code review assistant: Takes a code diff, analyzes it for bugs, suggests fixes, checks if fixes are valid, and produces a review summary. If a fix creates new issues, loop back.
  2. Customer support agent: Classifies the issue, searches the knowledge base, drafts a response, checks if the response addresses the issue, and routes to a human if confidence is low.
  3. Content writer: Takes a topic, researches it, writes a draft, evaluates the draft against criteria, revises if needed, and produces a final version.

For your chosen scenario, identify:

  • What is the state? (What data is passed between nodes?)
  • What are the nodes? (What does each step do?)
  • Where are the conditional edges? (Where does the graph decide what to do next?)
  • Where would you add a human-in-the-loop checkpoint?

If your diagram has arrows going backwards, you've just designed a LangGraph workflow.
