Chapter 5

LangChain: Connecting the Pieces

The Swiss Army knife of AI development — chains, tools, memory, and why everyone has an opinion about it

The One-Sentence Version

LangChain is a framework for building applications powered by LLMs — it gives you ready-made components for prompts, chains, memory, tools, and retrieval, so you can connect them like Lego blocks instead of building everything from scratch.

In the last chapter, you saw a RAG pipeline that used LangChain imports for loading, splitting, and embedding documents. That was a taste. LangChain does much more than RAG — it's a general-purpose framework for any application that involves an LLM.

It's also the most controversial tool in this guide. Some developers love it. Others actively avoid it. By the end of this chapter, you'll understand both sides and be able to make your own call.

LangChain is the most popular framework for building AI applications, with over 100K GitHub stars and a massive ecosystem of integrations. Even if you decide not to use it, understanding LangChain teaches you the fundamental patterns of AI application development — chains, tools, memory, retrieval. These patterns appear in every AI framework, not just LangChain. Think of it as learning the design patterns of AI apps.

Why LangChain Exists

Calling an LLM API is easy. Three lines of Python, done. But building a real AI application involves much more:

  • You need prompt templates that accept variables (the user's name, their question, retrieved context)
  • You need to chain steps together: retrieve documents, format a prompt, call the LLM, parse the output
  • You need memory so the conversation has continuity ("Earlier you mentioned...")
  • You need tools so the LLM can do things — search the web, query a database, call an API
  • You need output parsing to extract structured data from the LLM's free-text response
  • You need to swap between different LLM providers (OpenAI, Anthropic, Ollama) without rewriting your code

You could build all of this yourself. LangChain gives you pre-built components for each piece, with 700+ integrations to databases, APIs, model providers, and tools.

If you're an Android developer, think of LangChain like Retrofit + Room + Dagger rolled into one, but for AI. You could do HTTP calls, database queries, and dependency injection manually, but frameworks handle the boilerplate.

The Core Concepts

LangChain has six key concepts. Let's walk through each one.

1. Prompt Templates

Instead of hardcoding prompts, you create templates with placeholders:

from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant for {app_name}. Answer based on: {context}"),
    ("human", "{question}")
])

# Fill in the variables
formatted = prompt.invoke({
    "app_name": "MyApp",
    "context": "Users can reset passwords via Settings > Account > Reset",
    "question": "How do I reset my password?"
})

This seems trivial, but templates are reusable, testable, and versionable. When you have 20 different prompts in your app, templates keep things organized.

2. Chains — Connecting Steps

A chain is a sequence of operations: prompt → LLM → output parser. You connect them with the | (pipe) operator:

from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

# Create a chain: prompt → LLM → parse output as string
chain = prompt | ChatOpenAI(model="gpt-4") | StrOutputParser()

# Run it
answer = chain.invoke({
    "app_name": "MyApp",
    "context": "Users can reset passwords via Settings > Account",
    "question": "How do I reset my password?"
})
print(answer)

The pipe syntax is called LCEL (LangChain Expression Language). It's the modern way to compose chains and it makes the data flow easy to read: data flows left to right through each component.

Input Variables → Prompt Template → LLM → Output Parser → Result

3. Memory — Conversation Continuity

LLMs are stateless — each API call is independent. If a user says "my name is Alex" and then asks "what's my name?", the LLM has no idea unless you resend the entire conversation history.

LangChain's memory components handle this by storing conversation history and automatically including it in each new prompt. Under the hood, it's doing exactly what you'd expect: appending previous messages to the prompt. But it saves you from managing that list yourself.

Memory in LangChain isn't magic — it's literally the "append everything to the prompt" trick. The conversation grows longer with each turn. For long conversations, LangChain offers strategies like summarizing old messages or keeping only the last N turns. But it's important to know: there's no real "memory" here, just prompt management.

4. Tools — Letting the LLM Take Actions

This is where things get powerful. A "tool" is a function the LLM can choose to call. You define what the tool does, and the LLM decides when to use it based on the user's question.

from langchain_core.tools import tool

@tool
def search_database(query: str) -> str:
    """Search the product database for items matching the query."""
    # Your actual database search logic here
    return f"Found 3 products matching '{query}'"

@tool
def get_weather(city: str) -> str:
    """Get the current weather for a city."""
    return f"Weather in {city}: 22°C, sunny"

# Give tools to the LLM — it decides when to call them
llm_with_tools = ChatOpenAI(model="gpt-4").bind_tools([search_database, get_weather])

When the user asks "What's the weather in Berlin?", the LLM recognizes it needs the get_weather tool, generates the function call, LangChain executes it, and feeds the result back to the LLM to generate the final answer.

This is how you build AI agents that do things rather than just say things. An LLM with tools can search databases, send emails, create tickets, check inventory — whatever functions you expose.

5. Output Parsers — Structured Responses

LLMs return free-form text. If your app needs structured data (JSON, a list, specific fields), output parsers extract it:

from langchain_core.output_parsers import JsonOutputParser

# The parser instructs the LLM to output JSON and then parses it
parser = JsonOutputParser()
chain = prompt | llm | parser

# You get a Python dict back, not a string
result = chain.invoke({"question": "List 3 features of Kotlin"})
# result = {"features": ["Null safety", "Coroutines", "Extension functions"]}

This is more reliable than hoping the LLM outputs valid JSON on its own. The parser adds formatting instructions to the prompt and validates the response.

6. Retrievers — The RAG Connection

Retrievers connect LangChain to vector databases (from Chapter 4). They're the bridge between your stored documents and your chains:

# Create a retriever from a vector store
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# Use it in a chain: question → retrieve docs → format prompt → LLM → answer
rag_chain = (
    {"context": retriever, "question": lambda x: x}
    | prompt
    | llm
    | StrOutputParser()
)

answer = rag_chain.invoke("How do I configure push notifications?")

This is the RAG pipeline from Chapter 4, but now it's a single composable chain. The retriever fetches relevant documents, the prompt template combines them with the question, and the LLM generates the answer.

The Love/Hate Relationship

Now for the elephant in the room. LangChain is the most popular AI framework and the most criticized. Here's both sides, honestly:

Why People Love It

• Fast prototyping: build a RAG chatbot in an hour • 700+ integrations: every DB, API, and model works • Huge community: answers to every question on Stack Overflow • Battle-tested patterns: don't reinvent common workflows • LCEL pipe syntax: clean, readable data flows

Why People Hate It

• Over-abstracted: 5 layers of indirection for a simple API call • Changes frequently: code from 6 months ago often doesn't work • Debugging is painful: errors are buried in deep call stacks • Opaque magic: hard to understand what's actually happening • Heavy dependency: pulls in a lot of packages you may not need

Both sides are right. LangChain genuinely accelerates prototyping and provides excellent integrations. But it also adds complexity that can make debugging harder and code less transparent.

My Honest Take

Use LangChain for prototyping and when you need lots of integrations. It's the fastest way to get an AI application working. But if your app is simple (one LLM call, one retriever, one output), consider whether you actually need a framework at all. Sometimes 20 lines of plain Python with the OpenAI SDK is clearer than 10 lines of LangChain that hide what's happening.

The sweet spot: use LangChain to learn the patterns, prototype quickly, and leverage integrations. But don't be afraid to drop down to raw API calls when the abstraction gets in your way.

LangChain vs Doing It Yourself

Let's compare the same task — a simple RAG query — with and without LangChain:

With LangChain

chain = ( {"context": retriever, "question": ...} | prompt | llm | parser ) answer = chain.invoke(question)

Concise, composable, integrations handled. But what's happening inside retriever?

Without LangChain (raw Python)

chunks = vector_db.search(question, k=3) context = "\n".join(chunks) response = openai.chat(messages=[ {"role": "system", "content": f"...{context}"}, {"role": "user", "content": question} ]) answer = response.choices[0].message

More explicit, every step visible. But you handle all integrations yourself.

For simple pipelines, the raw version is perfectly fine. For complex ones (multiple tools, branching logic, memory, multiple retrievers), LangChain starts to pay for itself.

LangSmith: Debugging and Observability

One of LangChain's strongest features is LangSmith — a platform for tracing, debugging, and evaluating your AI applications. When something goes wrong (and it will), LangSmith shows you exactly what happened at each step of your chain:

  • What prompt was actually sent to the LLM
  • What the LLM responded with
  • Which tools were called and what they returned
  • How long each step took
  • How much each call cost

If you've ever used Android's Network Inspector to debug API calls, LangSmith is the same idea for AI chains. It's free for development use and genuinely makes debugging 10x easier.

Even if you decide not to use LangChain in production, consider using LangSmith for observability. It supports non-LangChain applications too and gives you visibility into what your AI is actually doing.

The Ecosystem Around LangChain

LangChain isn't just one library. It's a family of packages:

Package What It Does When You'd Use It
langchain-core Base abstractions, LCEL, interfaces Always — the foundation
langchain Chains, agents, memory, generic components Most applications
langchain-openai OpenAI integration When using GPT models
langchain-community 700+ third-party integrations When connecting to specific tools/databases
langgraph Stateful, graph-based workflows Complex agent behaviors (Chapter 6)
langsmith Tracing, debugging, evaluation Debugging and monitoring
LangChain's import paths have changed significantly over time. If you find code examples online, check the date. Code from early 2024 often uses deprecated imports like from langchain.llms import OpenAI. Current code uses provider-specific packages like from langchain_openai import ChatOpenAI. Always check the official docs for current imports.

Connection to Android Development

LangChain runs on your backend, not in your Android app. The architecture looks like this:

Android App → Your Backend API → LangChain Pipeline → LLM (Cloud or Ollama)

Your Android app makes a simple HTTP request. Your backend (Python, probably FastAPI or Flask) runs the LangChain pipeline and returns the result. From the Android side, it's just another API call — you don't need to know or care that LangChain exists.

This is actually the pattern most AI-powered apps use, regardless of the mobile platform. The AI logic lives server-side; the mobile client is a thin UI layer.

Learning Curve: Honest Assessment

  • Basic chains (1-2 hours): prompt + LLM + output parser. If you can read Python, you can do this.
  • RAG pipelines (half a day): retrievers, vector stores, document loaders. The concepts from Chapter 4 translate directly.
  • Tools and agents (1-2 days): defining tools, understanding how the LLM decides when to call them, handling edge cases.
  • LCEL composition (a few days): complex chains with branching, parallel execution, and custom runnables. The syntax is powerful but takes practice.
  • Production patterns (ongoing): error handling, streaming, caching, evaluation. This is where LangChain's complexity shows.
  • You're building a RAG pipeline and want ready-made document loaders, splitters, and retrievers
  • You need to integrate with many services (vector DBs, APIs, model providers)
  • You're prototyping and want to move fast
  • You want observability with LangSmith
  • Your application needs tools, memory, and complex chains
  • Your app makes one or two simple LLM calls — just use the SDK directly
  • You need maximum control and transparency over every step
  • You're allergic to deep abstraction layers and magic behavior
  • Your team doesn't know Python (LangChain is Python-first; the JS version exists but lags behind)
  • You're building something so custom that a framework would constrain you
Read and Trace a LangChain Example

You don't need to run any code for this exercise. Instead, look at this chain and trace what happens at each step:

chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt_template
    | ChatOpenAI(model="gpt-4")
    | StrOutputParser()
)
result = chain.invoke("What's our refund policy?")
  1. What does the retriever do with the input?
  2. What does RunnablePassthrough() do?
  3. What does the prompt_template receive?
  4. What does the LLM receive?
  5. What does StrOutputParser do to the LLM's response?

If you can answer these five questions, you understand the LangChain data flow. Hint: RunnablePassthrough just forwards the input unchanged.

What's Next

LangChain chains flow in one direction: A → B → C → done. But what if your AI application needs to loop? What if the LLM needs to try something, check the result, and go back and try again? What if you need conditional branching — "if the answer is good, return it; if not, retry with a different approach"?

That's where LangGraph comes in. In Chapter 6, we'll explore how graphs with cycles, state, and conditional logic let you build AI applications that are smarter than simple chains. It's the difference between a script and a program.
