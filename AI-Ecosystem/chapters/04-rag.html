Chapter 4

RAG: Teaching AI Your Data

The most practical pattern in AI — make any model answer questions about your documents, your code, your world

The Problem

You've got an AI model running — maybe through a cloud API, maybe locally with Ollama. It's smart. It can write code, explain concepts, and have conversations. But ask it about your company's product documentation, your app's API, or the meeting notes from last Tuesday, and it draws a complete blank.

That's because AI models only know what they were trained on. GPT-4 was trained on internet data up to a cutoff date. Llama was trained on public datasets. Neither has ever seen your internal wiki, your codebase, or your customer support tickets.

So how do you make AI know about your stuff?

There are two approaches:

  • Fine-tuning (Chapter 9): Retrain the model on your data. Expensive, slow, and you need to redo it every time your data changes.
  • RAG: Find the relevant information at query time and paste it into the prompt alongside the user's question. Cheap, fast, and always up-to-date.

For most use cases, RAG wins. It's the single most practical AI pattern you'll learn in this guide.

RAG is how you build AI features that actually know things — a support chatbot that answers questions about your app, a search feature that understands natural language, a documentation assistant that helps developers onboard faster. Every "chat with your docs" product you've ever seen? That's RAG. And building one is simpler than you think.

What RAG Actually Stands For

Retrieval-Augmented Generation. Let's break that down:

  • Retrieval — find the relevant pieces of information from your documents
  • Augmented — add that information to the prompt (augment the context)
  • Generation — let the LLM generate an answer using the retrieved context

That's it. RAG is "look up the relevant info, paste it into the prompt, then ask the model to answer based on it." It sounds almost too simple to deserve a fancy acronym. But the devil is in the details of how you find the relevant info.

The RAG Pipeline, Step by Step

A RAG system has two phases: ingestion (preparing your documents) and querying (answering questions). Let's walk through both.

Documents → Chunk → Embed → Store → Query → Retrieve → Generate

Step 1: Chunking — Breaking Documents Into Pieces

Your documents are too long to paste into a prompt whole. A user manual might be 200 pages. An API reference might have thousands of endpoints. You need to break them into smaller, digestible pieces called chunks.

Chunking is like cutting a book into paragraphs or sections. Each chunk should be:

  • Small enough to fit in a prompt alongside other chunks (typically 200-1000 tokens)
  • Large enough to contain a complete thought or piece of information
  • Self-contained — a chunk should make sense on its own

Good Chunking

Chunk: "The UserProfile API endpoint accepts a JSON body with 'name' (string, required), 'email' (string, required), and 'avatar_url' (string, optional). Returns 201 on success with the created profile object."

Self-contained, useful — enough to answer a question about this endpoint.

Bad Chunking

Chunk: "string, optional). Returns 201 on success with the created profile object. ## Delete Profile The"

Cut mid-sentence, mixes two topics — useless for answering questions.

Most RAG tutorials use simple character-count splitting. That works for demos but creates terrible chunks in practice. Better approaches: split by paragraph/section, use overlap between chunks (so context isn't lost at boundaries), or use "semantic chunking" that detects topic changes. Start simple, then improve your chunking when the answers aren't good enough.

Step 2: Embedding — Turning Text Into Numbers

This is where the magic from "How AI Actually Works" pays off. Remember embeddings? They turn text into vectors — lists of numbers that capture the meaning of the text.

Each chunk gets converted into a vector. The key property: chunks with similar meaning have similar vectors. A chunk about "user authentication" and a question about "how do users log in" will have vectors that are close together, even though they use different words.

You do this using an embedding model — a specialized AI model that takes text in and produces a vector out. Popular choices:

Model Where Quality Notes
text-embedding-3-small OpenAI API Very good Cheap, fast, cloud-only
nomic-embed-text Ollama / HuggingFace Good Free, runs locally
all-MiniLM-L6-v2 HuggingFace Good Tiny, fast, great for getting started
BGE-large HuggingFace Excellent Top-tier open-source option

Step 3: Storing — Vector Databases

You now have thousands of chunks, each with a vector. You need somewhere to store them and — crucially — to search them by similarity. That's what a vector database does.

A vector database is like a regular database, but instead of searching by exact match (WHERE name = 'John'), you search by meaning (find the 5 chunks most similar to this question).

C
Chroma
The "SQLite of vector databases." Runs in-process with your Python code, zero setup. Perfect for development and small projects. Start here.
P
Pinecone
Fully managed cloud vector database. Zero ops, scales automatically. Great for production but costs money.
PG
pgvector
PostgreSQL extension that adds vector search. If you already use Postgres, this lets you keep everything in one database.
W
Weaviate
Open-source, feature-rich, self-hosted or cloud. Good middle ground between Chroma (simple) and Pinecone (managed).
My recommendation: start with Chroma for development and prototyping. It requires zero configuration and runs in your Python process. When you go to production, consider pgvector if you already use PostgreSQL, or Pinecone if you want zero ops.

Step 4: Querying — Finding Relevant Chunks

When a user asks a question, you:

  1. Embed the question using the same embedding model
  2. Search the vector database for chunks with similar vectors
  3. Get back the top 3-5 most relevant chunks

This is called similarity search or semantic search. It finds relevant content based on meaning, not keywords. "How do I reset my password?" will match a chunk about "account recovery steps" even though they share no words.

Step 5: Generating — The LLM Answers

Now you combine everything. You take the user's question, the retrieved chunks, and a system prompt, and send them all to the LLM:

What the LLM actually sees

System: Answer the user's question based ONLY on the provided context. If the context doesn't contain the answer, say "I don't know." Context: [Chunk 1: UserProfile API accepts name, email...] [Chunk 2: Authentication uses JWT tokens...] [Chunk 3: Password reset sends email to...] User: How do I reset a user's password?

The LLM generates an answer using the retrieved chunks as its source of truth.

The LLM doesn't "know" the answer — it reads the answer from the chunks you gave it. That's why it's called retrieval-augmented: you're augmenting the model's knowledge by giving it relevant documents at query time.

The Complete Picture in Code

Here's what a minimal RAG pipeline looks like. This isn't production code, but it shows the shape of the whole thing:

# Ingestion: prepare your documents (run once)
from langchain_community.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import OllamaEmbeddings

# 1. Load → 2. Chunk → 3. Embed & Store
docs = TextLoader("my_docs.txt").load()
chunks = RecursiveCharacterTextSplitter(chunk_size=500).split_documents(docs)
vectorstore = Chroma.from_documents(chunks, OllamaEmbeddings(model="nomic-embed-text"))

# Querying: answer a question (run per question)
results = vectorstore.similarity_search("How do I reset a password?", k=3)
context = "\n".join([r.page_content for r in results])
# Send context + question to your LLM (Ollama, OpenAI, etc.)

That's ~10 lines for a working RAG pipeline. The LangChain imports handle the boring parts (loading files, splitting text, talking to the vector database). Chapter 5 goes deeper on LangChain itself.

Why RAG Works (and Why It Sometimes Doesn't)

RAG works because you're not changing the model — you're changing what the model sees. The model's reasoning ability stays the same; you're just giving it better context. It's like giving a smart person the right reference book before asking them a question.

But RAG can fail, and when it does, it's usually one of these problems:

Problem 1: Bad Chunking

If your chunks break sentences mid-thought or mix unrelated content, the retrieved context will be confusing. Fix: improve your chunking strategy (use paragraph boundaries, add overlap).

Problem 2: Irrelevant Retrieval

The vector search returns chunks that are vaguely related but don't actually answer the question. Fix: try a better embedding model, increase the number of retrieved chunks (k), or add metadata filtering.

Problem 3: Missing Information

The answer exists in your documents, but the embedding model doesn't match the question to the right chunk. Fix: try rephrasing your chunks, use hybrid search (vector + keyword), or add summaries.

Problem 4: Context Window Overflow

You retrieve too many chunks and they don't fit in the model's context window. Fix: retrieve fewer, more relevant chunks. Or use a model with a larger context window.

The #1 mistake beginners make with RAG: they get bad results and blame the LLM. Usually the problem is retrieval — the right chunks aren't being found. Before tweaking your LLM prompt, check what the vector search actually returns. If it's irrelevant, no prompt engineering will save you.

RAG vs Fine-Tuning: The Decision Framework

People often confuse RAG and fine-tuning, or wonder which to use. Here's a clear framework:

Factor RAG Fine-Tuning
Purpose Give the model ACCESS to your data Change how the model BEHAVES
Example "Answer questions about our docs" "Always respond in our brand voice"
Data freshness Always current (reads live data) Frozen at training time
Setup cost Low (hours) High (days + GPU costs)
Per-query cost Higher (embedding + retrieval + generation) Lower (just generation)
When data changes Re-index the new docs (minutes) Retrain the model (hours/days)
Accuracy Can cite sources, verifiable Internalized, may hallucinate

The Simple Rule

RAG = "The model needs to KNOW things it doesn't know." Your docs, your API, your company data.

Fine-tuning = "The model needs to DO things differently." Your tone, your format, your specialized behavior.

Both = "The model needs to know your data AND behave in a specific way." Fine-tune for behavior, RAG for data access. This is the most powerful combination.

Real-World RAG Examples

To make this concrete, here are actual things people build with RAG:

  • Documentation chatbot: "How do I configure push notifications in our SDK?" → RAG retrieves the relevant SDK docs and the LLM answers based on them.
  • Code search: "Show me how we handle authentication" → RAG searches your codebase and returns relevant code snippets with explanations.
  • Customer support: "My order hasn't shipped" → RAG retrieves the customer's order data and shipping policies, LLM generates a personalized response.
  • Legal research: "What do our terms say about refunds?" → RAG finds the relevant contract clauses.
  • Internal knowledge base: "What's our deployment process?" → RAG searches your internal wiki and Confluence pages.

Connection to Android Development

As an Android developer, RAG opens up practical features:

  • In-app help: Users ask questions in natural language, your backend uses RAG to find answers in your documentation and FAQ. No more digging through help pages.
  • Smart search: Instead of keyword matching, your search feature understands meaning. "How to change my profile picture" matches content about "avatar settings" and "photo upload."
  • Onboarding assistant: New developers on your team ask questions about the codebase. RAG searches your code, READMEs, and architecture docs to give contextual answers.
  • Content recommendations: Embed your content items as vectors. When a user reads an article, find similar content by vector similarity.

The architecture is simple: your Android app sends a question to your backend. The backend runs the RAG pipeline (retrieve relevant chunks, send to LLM, get answer). The app displays the answer. Your Android code doesn't need to know anything about vectors or embeddings — it just makes an HTTP request.

  • You need AI to answer questions about your specific documents or data
  • Your data changes frequently and you need answers to stay current
  • You want the AI to cite sources ("Based on page 12 of the manual...")
  • You're building a search feature that understands natural language
  • You have a knowledge base and want to make it conversational
  • The model already knows the answers (general knowledge, public info)
  • You need to change the model's behavior or tone (use fine-tuning instead)
  • Your data is a few paragraphs — just paste it into the system prompt
  • You need real-time data (stock prices, live feeds) — use tool calling instead

Common Gotchas and Honest Advice

  • Start with the simplest possible pipeline. Load a text file, split by paragraphs, embed with a free model, store in Chroma. Get it working before optimizing.
  • Evaluate by reading the retrieved chunks. Before blaming the LLM, print out what the retrieval step actually returns. Is it relevant? If not, fix retrieval first.
  • Chunk size matters more than you think. Too small = no context. Too large = diluted relevance. 300-500 tokens with 50-token overlap is a decent starting point.
  • Don't over-engineer the vector database. Chroma or SQLite with vectors is fine for thousands of documents. You don't need Pinecone until you have millions of vectors or need zero-ops at scale.
  • Test with real questions. Don't test with "What is X?" questions that a simple search could answer. Test with the messy, ambiguous questions your actual users would ask.
Design a RAG Pipeline

You don't need to build anything for this exercise — just think through the design. Pick one of these scenarios (or use your own):

  • A chatbot that answers questions about your app's user documentation
  • A search tool for your team's internal wiki
  • A support assistant that knows your product's FAQ and troubleshooting guides

Now answer these questions:

  1. What documents would you ingest? (docs, wiki pages, code, FAQs?)
  2. How would you chunk them? (by paragraph? by section? by page?)
  3. Which embedding model would you use? (cloud or local?)
  4. Which vector database would you start with?
  5. Which LLM would generate the answers? (cloud API or Ollama?)
  6. What's the hardest question a user might ask? Would your pipeline handle it?

Write down your design. In later chapters, you'll learn the tools to actually build it.
