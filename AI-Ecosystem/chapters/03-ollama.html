Chapter 3

Ollama: Running AI on Your Machine

One command to download, run, and talk to AI models — no cloud, no API keys, no cost per request

The One-Sentence Version

Ollama is Docker for AI models — it lets you download and run large language models locally on your own machine with a single command.

In Chapter 2, we explored HuggingFace — the massive library of AI models. But browsing models on a website and actually running them are two very different things. Setting up a model from raw files involves installing Python libraries, managing dependencies, figuring out quantization, configuring GPU memory... it's a mess.

Ollama makes all of that disappear. Want to run Meta's Llama 3.1? Type ollama run llama3.1. That's it. Ollama downloads the model, sets up the runtime, and gives you an interactive chat — all in one command.

Running AI locally changes the game for development. No API keys to manage, no per-request costs during prototyping, no sending your code or data to someone else's servers. When you're building an AI-powered feature, you can iterate as fast as you can type — no rate limits, no billing surprises. It's like the difference between testing your Android app on a cloud device versus running it on the emulator right on your machine.

Why Run Models Locally?

Cloud APIs (OpenAI, Anthropic, Google) are powerful. They give you access to the best models in the world. So why would you run a smaller, weaker model on your own hardware?

Privacy

When you call a cloud API, your data travels to someone else's server. For personal projects, that's fine. For company codebases, customer data, medical records, or legal documents? That can be a dealbreaker. Local models keep everything on your machine — nothing leaves your network.

Cost

Cloud APIs charge per token. During development, you might call the API hundreds of times while iterating on a prompt. That adds up. A local model costs $0 per request after the initial hardware investment (which you've already made — it's your laptop).

Speed (for Development)

No network latency. No waiting for API rate limits. For short requests, a local model on a good GPU can respond faster than a cloud API because there's no round-trip over the internet.

Offline Access

Airplane? Train? Coffee shop with terrible WiFi? Your local model doesn't care. It works anywhere.

No Rate Limits

Cloud APIs limit how many requests you can make per minute. When you're stress-testing a prompt template or running batch processing, rate limits slow you down. Local models have no limits — you can hammer them as hard as your hardware allows.

The best workflow for most developers: prototype locally with Ollama, deploy with a cloud API. Use local models for fast iteration and testing, then switch to a cloud model (GPT-4, Claude, Gemini) for production where quality matters most.

Getting Started: It's Really Just One Command

Install Ollama from ollama.com (macOS, Linux, Windows — all supported), then:

# Download and run Llama 3.1 (8B parameters) — first run downloads ~4.7 GB
ollama run llama3.1

# You're now in a chat. Type anything:
>>> What is dependency injection in Android?

# Want a different model? Just run it:
ollama run mistral
ollama run phi3
ollama run codellama

# List your downloaded models:
ollama list

# Pull a model without starting a chat:
ollama pull llama3.1

That's not a simplified example. That's literally the entire process. The first time you run a model, Ollama downloads it (this takes a few minutes depending on your internet). After that, it starts in seconds.

What Happens Under the Hood

When you type ollama run llama3.1, here's what actually happens:

Check Local Cache → Download Model (if needed) → Load into RAM/VRAM → Start Inference Server → Interactive Chat

Ollama stores models in a local cache (like Docker images). It automatically detects whether you have a GPU and uses it if available. It handles quantization, memory management, and context windows without you configuring anything.

Behind the scenes, Ollama uses llama.cpp — a blazing-fast C++ inference engine that's become the standard for running models on consumer hardware. You don't need to know this, but it's why Ollama is fast.

The Model Library

Ollama maintains its own curated library of models. These are pre-packaged, tested, and optimized. Here are the ones you should know about:

Model Size Best For RAM Needed
llama3.1 8B General purpose — the default choice ~5 GB
llama3.2 1B / 3B Lightweight tasks, low-resource machines ~1-2 GB
mistral 7B Strong all-rounder, Apache 2.0 license ~4.5 GB
phi3 3.8B Surprisingly capable for its tiny size ~2.5 GB
codellama 7B / 13B Code generation and understanding ~4.5-8 GB
gemma2 2B / 9B Google's open model, good quality ~1.5-6 GB
qwen2.5 0.5B-72B Wide range of sizes, strong benchmarks Varies
deepseek-coder-v2 16B Code-focused, excellent for programming tasks ~10 GB
My recommendation for getting started: run ollama run llama3.1 first. It's the best balance of quality, speed, and resource usage. If your machine is older or low on RAM, try ollama run phi3 — it's shockingly good for a 3.8B model.

Hardware: What You Actually Need

This is the question everyone asks first. Here's the honest answer:

CPU Inference (No GPU)

Yes, you can run AI models on just your CPU. It works. It's slow — expect 5-15 tokens per second for a 7B model on a modern CPU. That's readable but not fast. Smaller models (1-3B) are much snappier on CPU.

GPU Inference

A GPU makes everything 5-10x faster. You don't need a $2000 gaming GPU — a mid-range NVIDIA card with 8GB VRAM can comfortably run 7-8B models at 30-60 tokens per second. Apple Silicon Macs (M1/M2/M3) are excellent for this — the unified memory architecture means the GPU can use all your RAM.

Your Hardware What You Can Run Experience
8 GB RAM, no GPU 1-3B models (Phi3, Llama 3.2 1B) Usable for simple tasks
16 GB RAM, no GPU 7-8B models (Llama 3.1, Mistral) Slow but functional
16 GB RAM + 8 GB GPU 7-8B models with GPU acceleration Fast, comfortable for development
Apple Silicon Mac (16 GB+) 7-13B models using unified memory Great experience, the sweet spot
32 GB RAM + 16-24 GB GPU 13-30B models Near-cloud quality locally
Don't try to run a 70B model on a laptop with 16 GB RAM. It will either crash or swap to disk and take minutes per response. Match your model size to your hardware. A fast 7B model is always better than a slow 70B model that takes 30 seconds per answer.

CPU vs GPU: The 10x Difference

When a model generates text, it does billions of math operations per token. CPUs do these operations one by one (or a few at a time). GPUs do thousands simultaneously — that's what they were designed for.

Practical numbers for a 7B model generating a paragraph:

With GPU (RTX 3060, 12GB)

Speed: ~40 tokens/second Feel: Like a fast typist A paragraph: ~2 seconds

CPU Only (Modern i7/Ryzen)

Speed: ~8 tokens/second Feel: Readable but you notice the lag A paragraph: ~10 seconds

Both are usable. The GPU is just much more pleasant. If you're using this for development (iterating on prompts, testing RAG pipelines), the speed difference matters a lot over the course of a day.

The API: Your App Talks to Ollama

Here's where it gets interesting for developers. Ollama isn't just a command-line toy — it runs a REST API server on localhost:11434. Any application can talk to it, in any language.

Even better: Ollama's API is compatible with the OpenAI API format. This means code written for OpenAI's API works with Ollama by just changing the base URL. Libraries, frameworks, and tools that support OpenAI can talk to your local model with a one-line config change.

import requests

# Ollama's chat API — same format as OpenAI
response = requests.post("http://localhost:11434/api/chat", json={
    "model": "llama3.1",
    "messages": [
        {"role": "user", "content": "Explain dependency injection in 2 sentences"}
    ],
    "stream": False
})

print(response.json()["message"]["content"])

Or, using the OpenAI-compatible endpoint (so all OpenAI SDKs and tools work):

from openai import OpenAI

# Point the OpenAI SDK at your local Ollama server
client = OpenAI(base_url="http://localhost:11434/v1", api_key="unused")

response = client.chat.completions.create(
    model="llama3.1",
    messages=[{"role": "user", "content": "What's MVVM?"}]
)
print(response.choices[0].message.content)

This is a Big Deal

The OpenAI-compatible API means you can develop with a free local model and deploy with a cloud model (or vice versa) by changing a single URL. Your Android app's backend calls the same endpoint either way. LangChain (Chapter 5), RAG pipelines (Chapter 4), and most AI tools support this pattern out of the box.

Ollama vs Cloud APIs: When Each Makes Sense

Scenario Use Ollama (Local) Use Cloud API
Development & prototyping Fast iteration, no cost —
Sensitive/private data Data never leaves your machine —
Production quality — Larger, better models available
Complex reasoning — Cloud models are smarter
Batch processing No rate limits, no cost —
Offline / air-gapped Works without internet —
Scaling to many users — Infinitely scalable
Testing prompts Free, fast, unlimited —

The short version: local for development and privacy, cloud for production quality and scale. Many teams use both — Ollama during development, a cloud API in production.

Open WebUI: A ChatGPT Interface for Your Local Models

The command line is great for developers, but sometimes you want a proper chat interface — with conversation history, markdown rendering, and the ability to switch between models. Open WebUI gives you exactly that.

It's a web application (runs locally via Docker) that connects to your Ollama server and provides a ChatGPT-like interface. Install it, point it at localhost:11434, and you have a private, free ChatGPT running on your own machine.

Open WebUI is particularly useful for non-technical team members. Set up Ollama + Open WebUI on a team server, and everyone can chat with AI models without API keys, without cost, and without data leaving the company network.

Connection to Android Development

As an Android developer, Ollama fits into your workflow in several ways:

  • Local development server: Run Ollama on your dev machine. Your Android app calls http://10.0.2.2:11434 (the emulator's localhost alias) during development — free, fast, private. Switch to a cloud API for production builds.
  • Backend integration: Your backend server runs Ollama for AI features that need privacy or don't justify cloud API costs. The Android app just sees a REST endpoint.
  • Prompt engineering: Use Ollama to rapidly iterate on prompt templates. When you find one that works, deploy it with a cloud model for better quality.
  • RAG development: Building a feature that searches your app's content? Develop the entire RAG pipeline (Chapter 4) locally with Ollama before paying for cloud inference.

Limitations: Being Honest

Ollama is great, but let's be upfront about what it can't do:

  • Quality gap: A local 8B model is noticeably worse than GPT-4 or Claude for complex reasoning, nuanced writing, and multi-step tasks. It's not close. For simple tasks (classification, summarization, Q&A), the gap is much smaller.
  • Context window: Local models typically support 4K-8K token contexts. Cloud models offer 128K-200K+. If your task involves long documents, local models struggle.
  • Multimodal: Cloud models handle images, audio, and video natively. Local multimodal models exist but are less mature.
  • One user at a time: Your laptop serves one conversation well. If multiple people need simultaneous access, you need a beefier server or cloud APIs.
Don't evaluate AI by testing only with a local 7B model. If your first experience with RAG or LangChain uses a local model and the results are mediocre, the tool might not be the problem — the model might be. Always validate with a cloud model before deciding a technique "doesn't work."
  • You're prototyping and don't want to pay per API call
  • Privacy matters — sensitive data shouldn't leave your machine
  • You want to iterate on prompts quickly without rate limits
  • You need offline access to AI
  • You're running batch processing on many inputs
  • You want a free "ChatGPT" for your team (with Open WebUI)
  • You need the best possible AI quality for production
  • Your use case requires very long context windows (100K+ tokens)
  • You don't have at least 8 GB of free RAM
  • You need to serve hundreds of concurrent users
  • You're building a quick one-off script and already have API keys
Try Ollama

This is the most hands-on exercise in the guide so far. Here's what to do:

  1. Install Ollama from ollama.com (takes 2 minutes)
  2. Run your first model: ollama run llama3.1 (or phi3 if you have limited RAM)
  3. Ask it a coding question: "Write a Kotlin function that validates an email address"
  4. Ask the same question to ChatGPT (or Claude). Compare the answers. Notice the quality difference — and notice where the local model is "good enough"
  5. Try the API: Open a new terminal and run: curl http://localhost:11434/api/generate -d '{"model":"llama3.1","prompt":"What is MVVM?","stream":false}'
  6. Try a different model: ollama run mistral — notice how different models have different "personalities"

This exercise takes about 15-20 minutes (plus download time). After this, you'll have a genuine feel for what local AI can and can't do.
