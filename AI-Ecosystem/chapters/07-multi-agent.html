Chapter 7

Multi-Agent Systems: AutoGen & CrewAI

When one AI isn't enough — agents collaborating, debating, and solving problems together

The Idea

In the last chapter, you saw how LangGraph lets a single AI agent loop, branch, and make decisions. That's powerful. But here's a thought experiment: what if instead of one agent doing everything, you had a team?

  • One agent writes code
  • Another agent reviews the code and finds bugs
  • A third agent tests it and reports results
  • The first agent fixes the bugs based on the feedback

That's multi-agent systems. Multiple LLM-powered agents, each with a specific role, collaborating on a task. It sounds like science fiction, but it works — sometimes impressively, sometimes expensively, and sometimes both.

This is the most hyped area in AI development right now. Let's cut through the hype and understand what it actually is, when it helps, and when it's overkill.

Multi-agent systems represent the frontier of what's possible with LLMs today. Even if you never build one, understanding the concept helps you evaluate tools and claims you'll encounter in the wild. And when you do have a genuinely complex task that benefits from multiple perspectives — code review, research synthesis, content creation pipelines — knowing how to set up a crew of agents is a powerful tool in your belt.

What Is an "Agent" Here?

In the multi-agent context, an agent is an LLM instance with three things:

  • A role: "You are a senior code reviewer" or "You are a research analyst"
  • Tools: functions it can call (search the web, run code, query a database)
  • A goal: what it's trying to accomplish

Each agent is basically a separate LLM conversation with a carefully crafted system prompt and access to specific tools. The magic is in how agents interact with each other.

Think of it like microservices: each service has a focused responsibility, they communicate through defined interfaces, and together they accomplish something bigger than any one could alone.

AutoGen: Conversation-Driven Agents

AG
AutoGen (by Microsoft)
A framework where agents collaborate by having conversations with each other. Agents talk back and forth until they solve the problem or reach a termination condition.

How AutoGen Works

AutoGen's core model is simple: agents send messages to each other in a conversation. You define who talks to whom, and the agents take turns responding until the task is done.

from autogen import AssistantAgent, UserProxyAgent

# Create agents with different roles
coder = AssistantAgent(
    name="Coder",
    system_message="You write Python code to solve problems. Write clean, tested code.",
    llm_config={"model": "gpt-4"}
)

reviewer = AssistantAgent(
    name="Reviewer",
    system_message="You review code for bugs, style, and edge cases. Be thorough but fair.",
    llm_config={"model": "gpt-4"}
)

# UserProxy can execute code and represent the human
user = UserProxyAgent(name="User", code_execution_config={"work_dir": "coding"})

# Start a conversation: User asks Coder, Coder writes, Reviewer reviews
user.initiate_chat(coder, message="Write a function that finds duplicate files by hash")

What happens: the User agent sends the task to Coder. Coder writes code. The conversation can then be routed to Reviewer, who critiques it. Coder revises based on feedback. This back-and-forth continues until the code is approved or a turn limit is reached.

Key AutoGen Features

  • Conversation-based: agents communicate through natural language messages, just like developers in a Slack channel
  • Code execution: agents can write and run code in a sandboxed environment, seeing the actual output
  • Human-in-the-loop: the UserProxyAgent can require human approval before executing code or taking actions
  • Group chat: multiple agents in the same conversation, with automated or custom speaker selection
  • Flexible topology: agents can talk one-on-one, in groups, or in custom patterns
AutoGen's code execution capability is its killer feature. The coder agent writes Python, AutoGen runs it in a sandboxed Docker container, and the agent sees the actual output (including errors). This means agents can iteratively debug their own code until it actually works.

CrewAI: Task-Driven Teams

CA
CrewAI
A framework for organizing AI agents into structured teams (crews) with defined roles, goals, and tasks. More structured than AutoGen, less free-form conversation.

How CrewAI Works

CrewAI is more opinionated than AutoGen. Instead of free-form conversation, you define a structured crew with explicit roles, tasks, and execution order.

from crewai import Agent, Task, Crew

# Define agents with roles and backstories
researcher = Agent(
    role="Research Analyst",
    goal="Find accurate, comprehensive information on the given topic",
    backstory="You're a senior analyst known for thorough, well-sourced research.",
    tools=[web_search, arxiv_search]
)

writer = Agent(
    role="Technical Writer",
    goal="Turn research findings into clear, engaging content",
    backstory="You write for developers — clear, practical, no fluff.",
    tools=[])

# Define tasks
research_task = Task(description="Research the state of on-device AI for mobile apps",
                     agent=researcher, expected_output="A detailed research brief")

writing_task = Task(description="Write a blog post based on the research",
                    agent=writer, expected_output="A 1000-word blog post",
                    context=[research_task])  # Writer receives researcher's output

# Create and run the crew
crew = Crew(agents=[researcher, writer], tasks=[research_task, writing_task])
result = crew.kickoff()

Key CrewAI Features

  • Role-based: each agent has a role, goal, and backstory (the backstory helps the LLM "stay in character")
  • Task-driven: work is organized into explicit tasks with expected outputs
  • Task dependencies: tasks can depend on other tasks' outputs (like a build pipeline)
  • Tool access: each agent gets specific tools relevant to their role
  • Process types: sequential (one after another) or hierarchical (a manager delegates to workers)

AutoGen vs CrewAI

Aspect AutoGen CrewAI
Interaction model Conversation-driven (agents chat) Task-driven (agents complete assignments)
Structure Flexible, free-form Structured, opinionated
Best analogy A group brainstorm in Slack A project board with assigned tickets
Code execution Built-in (Docker sandbox) Through tools (less integrated)
Learning curve Moderate — flexible means more decisions to make Lower — structured means more guidance
Best for Exploratory tasks, code generation, open-ended problems Defined workflows, content pipelines, structured deliverables

My Recommendation

Start with CrewAI if you want structured, predictable results. Its role/task/crew model is intuitive and produces consistent output. Use AutoGen if your task is more exploratory or requires agents to iteratively solve problems through conversation (especially code generation tasks).

Both are evolving rapidly. Check their current state before committing — the landscape may look different in six months.

Real Use Cases (Where Multi-Agent Actually Helps)

Let's be specific about when multiple agents genuinely outperform a single agent:

Code Review Pipeline

Agent 1 writes code. Agent 2 reviews for bugs. Agent 3 reviews for security. The separation of concerns means each reviewer is focused and thorough. A single agent asked to "write and review" tends to be too forgiving of its own work.

Research and Synthesis

Agent 1 searches academic papers. Agent 2 searches news articles. Agent 3 searches forums. Agent 4 synthesizes all findings into a coherent report. Each researcher can focus deeply on one source type.

Content Creation Pipeline

Researcher → Writer → Editor → Fact-checker. Each step has a clear deliverable and quality gate. The editor and fact-checker catch mistakes the writer makes.

Data Analysis

Agent 1 cleans and structures data. Agent 2 runs statistical analysis. Agent 3 creates visualizations. Agent 4 writes the narrative. Each agent has specialized tools and focus.

The Hype vs Reality

Let's be honest about the current state of multi-agent systems:

Where It Works Well

• Tasks where self-review genuinely improves quality • Pipelines with clear stages and handoffs • Code generation with iterative testing • Research from multiple source types • Any task where "a second opinion" catches errors

Where It's Overhyped

• Simple tasks that one LLM call handles fine • When agents just agree with each other (echo chamber) • When the "debate" adds latency without quality • Tasks where the bottleneck is the LLM's knowledge, not its perspective • Demos that look impressive but cost $5 per run

The Cost Problem

This is the elephant in the room. If a single LLM call costs X, a 3-agent system with 5 turns of conversation costs roughly 15X. For cloud APIs, that means:

1 Agent ~$0.01-0.05 per task
3 Agents ~$0.10-0.50 per task
5 Agents ~$0.50-2.00 per task

This is fine for high-value tasks (code review, research reports) but absurd for things like chatbot responses or simple classifications. Always ask: does the quality improvement justify the 10-15x cost increase?

The biggest mistake with multi-agent systems: using them because they're cool, not because they're needed. A single well-prompted LLM with tools (a LangGraph agent) handles 90% of use cases. Multi-agent adds value when you genuinely need separation of concerns and self-review. Test with a single agent first; add agents only when you can measure the quality improvement.

When Single-Agent Is Better

For most tasks, one agent is enough. Here's a useful rule of thumb:

  • Can one person do this task well? Then one agent is probably enough.
  • Would you assign this to a team in real life? Then consider multi-agent.
  • Is the bottleneck knowledge or perspective? If it's knowledge (the LLM doesn't know enough), adding agents won't help — use RAG instead. If it's perspective (the LLM misses things), a reviewer agent might help.

Most AI features in mobile apps — chatbots, search, classification, summarization — are single-agent tasks. Multi-agent shines in backend pipelines and content workflows, not in user-facing interactions.

A great way to test whether you need multi-agent: build the feature with a single agent first. Then ask yourself: "Where did the output fall short?" If the answer is "it didn't catch its own mistakes" or "it missed a perspective," that's where you add a second agent. Start minimal, add complexity only when you can point to the specific improvement.

Running Agents Locally

Both AutoGen and CrewAI support Ollama as a backend. This means you can run multi-agent systems entirely on your machine with zero API costs — great for development and experimentation.

The trade-off: local models (7-8B parameters) are noticeably weaker as agents than cloud models (GPT-4, Claude). They're more likely to go off-track, hallucinate tool calls, or produce mediocre output. For development and learning, local is fine. For production-quality results, you'll likely want cloud models for at least the key agents.

A practical middle ground: use a cloud model for the "lead" agent (the one making decisions) and local models for supporting agents (the ones doing simpler, well-defined tasks). This hybrid approach keeps costs down while maintaining quality where it matters most.

Connection to Android Development

Multi-agent systems are exclusively backend tools. Your Android app never interacts with agents directly. But the features they enable can be exposed through your API:

  • Code review in your CI/CD: A crew that reviews PRs, checks for security issues, and writes review comments. Your mobile team benefits from better code quality without running agents on devices.
  • Content pipeline: A crew that researches, writes, edits, and publishes content for your app. The Android app just fetches the finished content from your CMS.
  • Support escalation: A first agent handles common questions. If confidence is low, it escalates to a specialist agent. If still unresolved, it routes to a human. Your app shows the response regardless of which agent produced it.
  • Test generation: A crew that reads your API specs and generates test cases. Useful for ensuring your Android app's backend behaves correctly.

The pattern is always the same: agents run on the server, your app consumes the results through a standard API.

Learning Curve: Honest Assessment

  • Understanding the concept (30 minutes): If you've ever managed a team, you understand multi-agent. It's assigning roles and tasks.
  • Simple 2-agent setup (1-2 hours): CrewAI's role/task/crew pattern is intuitive. AutoGen's conversable agents are also quick to start.
  • Effective role design (ongoing): Crafting system prompts that make agents actually useful (and not just chatty) takes iteration.
  • Managing cost and quality (ongoing): Tuning the number of turns, knowing when to stop, balancing cost vs quality.
  • Production deployment (days): Error handling, fallbacks, monitoring, cost control.
  • Your task genuinely benefits from multiple perspectives (review, fact-checking)
  • You have a pipeline with distinct stages and clear handoffs
  • Code generation tasks where self-testing improves quality
  • Research tasks requiring synthesis from multiple source types
  • You can measure that multi-agent produces better results than single-agent
  • A single LLM call or simple chain handles the task well
  • Your task is latency-sensitive (users are waiting for a response)
  • Cost is a concern and the quality improvement is marginal
  • You're adding agents because it sounds impressive, not because it helps
  • The agents end up agreeing with each other without catching real issues
Design a 3-Agent Crew

Pick a task from your domain and design a crew of three agents. For each agent, define:

  1. Role: What is this agent's job title?
  2. Goal: What is it trying to accomplish?
  3. Tools: What can it do? (search, run code, query databases, etc.)
  4. Backstory: A sentence that helps the LLM stay in character

Then define the tasks:

  • What does each agent produce?
  • Which tasks depend on other tasks' output?
  • Where would you add a human checkpoint?

Finally, the honest question: would a single well-prompted agent with tools do 80% as well for 1/10th the cost? If yes, start with that and add agents only if needed.
