Skills vs Slash Commands

What skills actually are, where they live, what “modern” vs “legacy” means, and how the /name invocation system in Claude Code really works.

TL;DR

Skills and slash commands are the same /name invocation system. Anthropic merged them.

.claude/skills/<name>/SKILL.md is the modern form. .claude/commands/<name>.md is the legacy form — still works, not deprecated, just older. Both produce /<name>.

If you’ve ever read about Claude Code and walked away wondering “wait, are skills and slash commands the same thing or not?” — this guide is for you. The terminology is genuinely messy across the internet because Anthropic changed it: in 2026 they merged custom slash commands into a broader feature called skills. The old name still appears everywhere, the old file format still works, and tutorials written six months apart describe it differently.

The good news: the underlying mechanism is one thing, not two. Once you see that, everything else clicks.

“Custom commands have been merged into skills. A file at .claude/commands/deploy.md and a skill at .claude/skills/deploy/SKILL.md both create /deploy and work the same way. Your existing .claude/commands/ files keep working. Skills add optional features: a directory for supporting files, frontmatter to control whether you or Claude invokes them, and the ability for Claude to load them automatically when relevant.” — code.claude.com/docs/en/skills

What is a skill, really?

A skill is a piece of reusable instruction you can hand to Claude Code. It lives as a markdown file (or a directory containing one), and it does two things at once:

  1. It becomes a slash command. Type /your-skill and the skill content gets injected into the conversation.
  2. Claude can pick it up automatically. When the description matches what you’re asking for, Claude loads the skill on its own — no slash command needed.

That second behavior is the key thing that separates modern skills from the older “just a slash command” idea. A skill isn’t only a shortcut you type — it’s also context the model can reach for when relevant. Think of it like a recipe in a recipe box: you can grab a specific recipe by name (“make me /lasagna”), but if you just say “I’m hungry for Italian,” the model can flip through the box and find lasagna on its own.

The three things people actually mean by “commands”

When you read “command” in a tutorial or docs page, it can mean any of three different things. Most of the confusion in the wild comes from blurring them. Here’s the clean version:

Term What it actually is Examples Can you create one?
Built-in command Hard-coded behavior inside the Claude Code CLI. Not a file. Not customizable. /help, /clear, /compact, /model, /permissions, /config No
Bundled skill A prompt-based skill that Anthropic ships with Claude Code. Same mechanism as your own skills, just shipped by them. /simplify, /batch, /debug, /loop, /claude-api, /init, /review, /security-review No
Custom skill A skill you (or your project) author. Modern form: .claude/skills/<name>/SKILL.md. Legacy form: .claude/commands/<name>.md. Both produce /<name>. /deploy, /commit, anything you write Yes

The official commands reference page lists built-in commands and bundled skills together, with bundled skills marked Skill in the Purpose column. That table is the authoritative source for which is which.

Where do skills live?

A skill is just a directory on disk. Where the directory lives determines who can use the skill:

ScopePathApplies to
Enterprise Configured via managed settings All users in your organization
Personal ~/.claude/skills/<name>/SKILL.md All your projects, on your machine
Project .claude/skills/<name>/SKILL.md This project only (commit to git to share with the team)
Plugin <plugin>/skills/<name>/SKILL.md Wherever the plugin is enabled

Conflict resolution when two scopes define a skill with the same name: enterprise wins over personal, personal wins over project. Plugin skills are namespaced as plugin-name:skill-name so they can never collide. And if a skill at .claude/skills/foo/SKILL.md shares a name with a legacy command at .claude/commands/foo.md, the skill wins.

The modern form vs the legacy form — side by side

Both shapes produce a slash command. The difference is what each shape can contain.

Legacy form older, still works

.claude/commands/deploy.md

A single markdown file. The filename (without .md) becomes the slash command. Same YAML frontmatter as a skill. No supporting files, no bundled scripts.

---
description: Deploy to production
disable-model-invocation: true
---

Deploy $ARGUMENTS:
1. Run tests
2. Build
3. Push

Invoked as /deploy.

Modern form recommended

.claude/skills/deploy/SKILL.md

A directory. The directory name becomes the slash command. The directory can hold supporting files (templates, scripts, reference docs) loaded only when needed.

.claude/skills/deploy/
├── SKILL.md          # required
├── checklist.md      # loaded on demand
└── scripts/
    └── verify.sh     # runnable

Also invoked as /deploy.

What the modern form adds

  • Supporting files — templates, scripts, reference docs that load only when the skill references them. Keeps SKILL.md focused.
  • Bundled scripts — the ${CLAUDE_SKILL_DIR} variable resolves to the skill’s directory, so scripts work from any working directory.
  • Subagent execution — context: fork in the frontmatter runs the skill in an isolated subagent (great for research and refactors).
  • Monorepo discovery — nested .claude/skills/ directories are found automatically when you edit files in subpackages.
  • Live reload of new directories without restarting (after the first creation).

What both forms share

  • Same /name invocation.
  • Same YAML frontmatter fields.
  • Same $ARGUMENTS and ${CLAUDE_*} substitutions.
  • Same !`<command>` dynamic context injection (run shell, inline output before Claude sees the prompt).
  • Same automatic invocation by Claude when the description matches — unless you set disable-model-invocation: true.

How invocation works

There are two ways anything /foo gets triggered:

  1. You type /foo. Works for built-in commands, bundled skills, custom skills (modern or legacy), and MCP prompts (/mcp__server__prompt).
  2. Claude invokes it automatically. Only works for skills (custom or bundled), and only when the skill’s description matches what you’re asking for. Built-in commands cannot be auto-invoked.

Two frontmatter fields control who can invoke a skill:

FrontmatterYou can invokeClaude can invokeWhen to use
(default) Yes Yes Most reference skills (style guides, conventions, helpers)
disable-model-invocation: true Yes No Side-effect skills: /deploy, /commit, /send-slack-message — you don’t want Claude deciding to deploy because the code “looks ready.”
user-invocable: false No Yes Background knowledge a user shouldn’t manually trigger (e.g. a legacy-system-context skill Claude reads when relevant).

To see every skill currently available in your session, type /skills.

Frontmatter quick reference

The fields you’ll most often see in skill examples. The full table with every field is in the official frontmatter reference.

FieldWhat it does
nameDisplay name. Defaults to the directory (modern) or filename (legacy). Lowercase, numbers, hyphens, max 64 chars.
descriptionThe single most important field. What the skill does and when to use it. Claude uses this to decide whether to auto-invoke. Combined cap of 1,536 chars with when_to_use.
when_to_useTrigger phrases or example requests. Appended to description for matching.
argument-hintAutocomplete hint, e.g. [issue-number].
argumentsNamed positional args, e.g. arguments: [issue, branch] — lets you write $issue and $branch in the body.
disable-model-invocationtrue = only the user can invoke. Default false.
user-invocablefalse = hide from the / menu. Default true.
allowed-toolsTools Claude can use without asking permission while this skill is active.
contextfork = run in a subagent. Pair with agent.
agentWhich subagent type when context: fork. Built-ins: Explore, Plan, general-purpose.
pathsGlob patterns; auto-invocation only fires when working with matching files.

String substitutions inside the skill body

  • $ARGUMENTS — everything after the command name
  • $0, $1, … — positional args (shell-style quoting; wrap multi-word values in quotes)
  • $name — when declared in arguments: frontmatter
  • ${CLAUDE_SKILL_DIR} — directory containing SKILL.md (use this for bundled scripts)
  • ${CLAUDE_SESSION_ID} — current session ID
  • ${CLAUDE_EFFORT} — current effort level

Dynamic context injection

Both forms support running shell commands before Claude sees the skill content. The output replaces the placeholder in the rendered prompt. This is preprocessing — not a tool call Claude makes. 

---
description: Summarize uncommitted changes
---

## Current changes

!`git diff HEAD`

## Instructions

Summarize the diff in 2–3 bullets and flag risks.

For multi-line scripts, use a fenced block opened with ```! instead of the inline form.
