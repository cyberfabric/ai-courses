Chapter 3

Designing AI UX That iOS Users Trust

Loading states, streaming, confidence signals, error handling, and accessibility — the UX patterns that separate “feels magical” from “feels broken”

The Trust Equation on iOS

Here’s the thing about iPhone users: they paid $1,000+ for a device that just works. When your AI feature gets something wrong, it doesn’t just feel like a bug — it feels like the premium experience they paid for has broken down. The trust bar on iOS is higher than any other platform.

Trust is earned in tiny increments and lost in a single bad interaction. An AI feature that’s right 95% of the time but handles the other 5% poorly will be remembered for those failures. An AI feature that’s right 90% of the time but handles errors gracefully will be remembered as reliable.

Scenario: Priya opens her journaling app and taps “Summarize This Week.” She sees a gentle pulsing animation with the text “Reading your entries…” After a second, words begin streaming onto the screen, building her summary sentence by sentence. When it finishes, a subtle “AI Summary” badge appears, along with an “Edit” button. She tweaks one sentence, taps “Save,” and the summary is hers.

Now imagine the alternative: she taps the button, sees a generic spinner for 3 seconds, then a wall of text appears all at once with no indication it came from AI. Same output. Radically different experience.

This chapter is about the difference between those two experiences. The AI model is the same — the UX is what separates trust from frustration.

The Three Phases of AI Loading States

Traditional loading: show a spinner, wait, show the result. AI loading needs more nuance because the wait is unpredictable and the result is uncertain. Every AI interaction has three phases:

The Three-Phase AI Loading Pattern

  1. Initiating — The user tapped the button. Show a ProgressView with contextual text: “Analyzing your receipt…” not “Loading…” The user should know what the AI is doing, not just that something is happening.
  2. Processing — Tokens are arriving. Stream them to the screen with a smooth typing animation. The user sees progress in real time. This phase turns a wait into a reveal.
  3. Completing — The response is done. Transition smoothly from streaming to the final result. Show action buttons (Edit, Copy, Share), confidence indicators if needed, and the “AI generated” badge.

Never leave users staring at a spinner with no context. A contextual loading message (“Summarizing 3 entries…”) feels faster than a generic one (“Please wait…”), even when the actual wait time is identical.

Streaming Responses in SwiftUI

The ChatGPT-style token-by-token text reveal has trained users to expect it. When text appears all at once, it feels abrupt. When it streams word by word, it feels thoughtful — like the AI is composing in real time. Here’s how to build it:

Recipe: Streaming AI Text in SwiftUI

Goal: Display Foundation Models output token-by-token as it generates, with smooth animation.

import FoundationModels
import SwiftUI

struct StreamingTextView: View {
    let prompt: String
    @State private var streamedText = ""
    @State private var isStreaming = false
    @State private var isDone = false

    var body: some View {
        VStack(alignment: .leading, spacing: 12) {
            if isStreaming || isDone {
                Text(streamedText)
                    .font(.body)
                    .animation(.easeIn(duration: 0.15), value: streamedText)
            }
            if isStreaming {
                ProgressView()
                    .controlSize(.small)
            }
            if isDone {
                Label("AI Generated", systemImage: "sparkles")
                    .font(.caption)
                    .foregroundStyle(.secondary)
            }
        }
        .task { await generate() }
    }

    func generate() async {
        isStreaming = true
        let session = LanguageModelSession()
        do {
            let stream = session.streamResponse(to: prompt)
            for try await partial in stream {
                streamedText = partial.content
            }
        } catch { streamedText = "Couldn't generate a response." }
        isStreaming = false
        isDone = true
    }
}

The key insight: streamResponse(to:) returns an AsyncSequence of partial results. Each iteration gives you the full text so far, so you just assign it to your @State property and SwiftUI handles the rest.

Respect Reduce Motion. Some users have “Reduce Motion” enabled in Accessibility settings. When that’s on, skip the streaming animation and show the complete result at once. Check with UIAccessibility.isReduceMotionEnabled or SwiftUI’s @Environment(\.accessibilityReduceMotion).

Apple HIG for AI: The Gold Standard

Apple’s Human Interface Guidelines have a dedicated section on machine learning. The core principles are:

  • Subtle — AI should enhance, not dominate. Writing Tools appears as a quiet option in the text selection menu, not a flashing banner.
  • Integrated — AI features should feel like natural parts of the app, not bolt-on additions. Live Text doesn’t have its own screen — it’s just there when you point your camera at text.
  • User-controlled — Users must always feel in charge. Every AI suggestion can be dismissed, edited, or undone. No forced AI interactions.

Writing Tools: The Gold Standard

Study how Apple designed Writing Tools. It appears in the text selection menu — where you already are. It offers options (Rewrite, Proofread, Summarize) without requiring you to learn anything new. Changes are shown as tracked edits you can accept or reject individually. The AI is powerful, but the user is always in control. This is what your AI features should aspire to.

Communicating Confidence

One of the trickiest UX decisions: how do you tell users how confident the AI is? The answer depends on the confidence level — and raw numbers are almost never the right approach.

AI Confidence What to Show iOS Example
High (>90%) Just show the result — no qualification QuickType: suggestion appears, tap to accept
Medium (70–90%) Show with a subtle “AI suggested” badge Mail: suggested replies with “Siri Suggestion” label
Low (50–70%) Show multiple options, let the user pick Safari: alternative search suggestions
Very Low (<50%) Don’t show — ask for clarification or show nothing Siri: “I’m not sure I understand”

Confidence Done Right

Mail shows three suggested replies at the bottom of an email. No percentages, no confidence scores. If none fit, you just type normally. The AI's uncertainty is invisible.

Translate confidence into behavior (show more/fewer options), not numbers.

Confidence Done Wrong

"AI Confidence: 67%" displayed next to a suggestion. The user stares at it. Is 67% good? Bad? Should they trust it? They dismiss the suggestion they would have accepted without the number.

Raw confidence scores create anxiety. Users don't know how to interpret them.

The “AI Suggested” Pattern

When your app shows AI-generated content alongside user-created content, you need a clear signal. Users must know what came from them and what came from the AI. Apple uses subtle but consistent patterns:

  • Mail: Suggested replies have a distinct visual style (lighter background, different position)
  • Safari: AI summaries show an explicit “AI Summary” badge at the top
  • Writing Tools: Proposed changes appear as tracked edits with accept/reject controls
// A reusable AI badge for your SwiftUI views
struct AIBadge: View {
    var body: some View {
        Label("AI Generated", systemImage: "sparkles")
            .font(.caption2)
            .fontWeight(.medium)
            .padding(.horizontal, 8)
            .padding(.vertical, 4)
            .background(.ultraThinMaterial)
            .clipShape(Capsule())
    }
}

User Control and Override

This is non-negotiable: every AI feature needs an escape hatch. Users must always feel in control. If they can’t dismiss, edit, undo, or turn off an AI suggestion, it’s not a feature — it’s a trap.

  • Dismiss — Swipe away or tap X to remove the suggestion
  • Edit — Modify the AI’s output before using it
  • Undo — Revert to the original after accepting an AI change
  • Feedback — “Not helpful” button that teaches the system
  • Manual fallback — The non-AI way to do the same thing still exists
  • Global off switch — A setting to disable AI suggestions entirely

App Store Review cares about this. Apple specifically checks that AI features give users control. If your generative feature produces content with no way to edit, dismiss, or report it, expect a rejection. Build the escape hatches before you build the feature.

Error Communication

When the AI doesn’t know, it should say so. “I’m not sure” is always better than a confident wrong answer. Here’s the hierarchy:

Confident? Show result
→
Uncertain? Show options
→
Lost? Ask for clarification
→
Failed? Explain & offer fallback

Error messages for AI features should be:

  • Honest — “I couldn’t understand this text” not “Something went wrong”
  • Actionable — “Try a shorter paragraph” not “Please try again later”
  • Non-blaming — Never “Invalid input.” The user didn’t do anything wrong — the AI reached its limits
struct AIErrorView: View {
    let message: String
    let suggestion: String
    let onRetry: () -> Void

    var body: some View {
        VStack(spacing: 12) {
            Label(message, systemImage: "exclamationmark.bubble")
                .font(.subheadline)
                .foregroundStyle(.secondary)
            Text(suggestion)
                .font(.caption)
                .foregroundStyle(.tertiary)
            Button("Try Again", action: onRetry)
                .buttonStyle(.bordered)
        }
        .padding()
    }
}

// Usage:
AIErrorView(
    message: "Couldn't summarize this text",
    suggestion: "Try selecting a shorter section",
    onRetry: { Task { await regenerate() } }
)

Feedback Loops

Great AI features get better over time. To improve, you need user feedback — but asking for it is an art:

Explicit Feedback

Thumbs up/down buttons after AI output. Keep them small and unobtrusive. Don’t ask on every interaction — sampling (every 5th use) reduces fatigue.

Implicit Feedback

Did the user accept the suggestion or dismiss it? Did they edit it heavily or use it as-is? Did they tap “Regenerate”? These signals are gold — and they don’t require the user to do anything extra.

Collect implicit signals first. Acceptance rate (% of suggestions used as-is), edit distance (how much users changed the AI output), and dismissal rate tell you more about quality than a thumbs-up button ever will. And they’re free — no user friction at all.

Accessibility for AI Features

AI features introduce unique accessibility challenges. Dynamic content that appears, streams, and changes requires special attention:

VoiceOver

When AI content appears dynamically, VoiceOver users need to be notified. Use accessibility announcements and focus management:

// Announce when AI result is ready
UIAccessibility.post(
    notification: .announcement,
    argument: "AI summary ready. \(summaryText)"
)

// Move focus to the result
@AccessibilityFocusState private var focusOnResult: Bool

// After result loads:
focusOnResult = true

Dynamic Type

AI-generated text must respect the user’s preferred text size. Never use fixed font sizes for AI output — always use semantic styles (.body, .headline, .caption) that scale automatically.

Reduce Motion

Streaming animations should be disabled when Reduce Motion is on. Show the complete result at once instead:

@Environment(\.accessibilityReduceMotion) var reduceMotion

if reduceMotion {
    // Show full result immediately
    Text(fullResponse)
} else {
    // Stream token by token
    StreamingTextView(prompt: prompt)
}

Progressive Disclosure

Don’t overwhelm users on day one. AI features should reveal complexity gradually:

Level What the User Sees Example
Level 1: Simple One-tap AI action with sensible defaults “Summarize” button — tap it, get a summary
Level 2: Customize Options to adjust the AI’s behavior Choose summary length: “Brief” / “Detailed” / “Bullet Points”
Level 3: Power User Advanced settings and fine-tuning Custom instructions, tone preference, output format

Most users will stay at Level 1 forever. That’s fine — it means you got the defaults right. But power users should be able to dig deeper without the interface getting in the way of casual users.

Dark Patterns to Avoid

Apple’s App Store Review will reject AI features that manipulate, deceive, or trap users. But beyond review, these patterns erode trust:

Manipulate

"Our AI predicts you'll miss this deal in 2 hours!" — AI-generated fake urgency to pressure a purchase.

Using AI to manufacture urgency is deceptive. Apple will reject it.

Deceive

AI-generated product reviews displayed as real user reviews with no disclosure.

Presenting AI output as human-created content violates App Store guidelines and user trust.

  • No forced AI — Users must be able to turn off AI suggestions. “You can’t use this feature without AI” is a trap.
  • No hidden AI — If content is AI-generated, label it. Transparency is not optional.
  • No fake personalization — “Made just for you” when it’s the same for everyone is a lie.

Putting It All Together

Exercise: Design an “AI Email Reply” Feature

Design the complete UX flow for an AI feature that generates email reply suggestions. Describe (or sketch) every state:

  1. Empty — No AI reply yet. What does the user see? How do they trigger the AI?
  2. Initiating — The user tapped “Suggest Reply.” What contextual loading message do you show?
  3. Streaming — Tokens are arriving. How do you display them? What animation?
  4. Results — Three reply options are ready. How do you show confidence? How does the user pick one?
  5. Error — The AI couldn’t generate a reply. What message do you show? What’s the fallback?
  6. User Edit — The user picked a reply but wants to modify it. How does editing work?
  7. Feedback — After sending, how do you collect feedback? Explicit or implicit?

For each state, consider: What does VoiceOver announce? What if Reduce Motion is enabled? Does the feature work on older devices without Foundation Models?
