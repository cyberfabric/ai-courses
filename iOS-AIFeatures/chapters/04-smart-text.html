Chapter 4

Smart Text with Foundation Models

@Generable structured output, Writing Tools, classification, summarization — turning unstructured text into typed Swift data

Foundation Models 101

In Chapter 1, we showed you three lines of code to generate text. In Chapter 3, we streamed it to the screen with a typing animation. Now it’s time to go deep. This chapter is where you stop playing with the framework and start building real features.

The Foundation Models framework gives your app access to Apple’s ~3B parameter on-device language model. It’s the same model powering Writing Tools, Mail suggestions, and Safari summaries. Three key objects to know:

  • LanguageModelSession — Your conversation with the model. Create one, send prompts, get responses. It’s stateful — it remembers previous turns.
  • @Generable — A macro you put on Swift structs and enums. The model fills your types instead of returning raw text. This is the killer feature.
  • @Guide — Constraints on properties. Tell the model a rating must be 1–5, or a category must come from your enum. The model literally cannot produce invalid values.

Why @Generable Changes Everything

Without @Generable, you get a string back from the model and parse it yourself. With @Generable, you define a Swift struct, and the model fills it — guaranteed to match your schema through constrained decoding. No JSON parsing. No regex. No “the model returned the wrong format.” The compiler and the model work together.

The @Generable Macro

This is the feature that makes Foundation Models genuinely different from every other AI API. You define a Swift type, the model fills it. Let’s start with a practical example:

Recipe: Extract Meeting Details from Text

Goal: Turn “Let’s meet with Anna on Thursday at 2pm in the boardroom” into a typed Swift struct.

Step 1: Define the output type:

import FoundationModels

@Generable
struct MeetingDetails {
    @Guide(description: "The people mentioned in the meeting")
    var attendees: [String]

    @Guide(description: "The date and time of the meeting")
    var dateTime: String

    @Guide(description: "The location of the meeting")
    var location: String

    @Guide(description: "Brief summary of the meeting purpose")
    var purpose: String
}

Step 2: Ask the model to fill it:

let session = LanguageModelSession()
let text = "Let's meet with Anna on Thursday at 2pm in the boardroom to discuss Q3 goals"
let result = try await session.respond(
    to: "Extract meeting details from: \(text)",
    generating: MeetingDetails.self
)
// result.attendees → ["Anna"]
// result.dateTime → "Thursday at 2pm"
// result.location → "the boardroom"
// result.purpose → "Discuss Q3 goals"

That’s it. No JSON parsing. No string manipulation. The model returns a MeetingDetails instance with typed properties. If a property can’t be found, the model still returns a valid struct (with empty strings or arrays) — it never crashes.

@Guide Constraints: The Safety Net

What if you need the model to choose from a fixed set of values? That’s where @Guide shines. Combined with enums, it guarantees the model can only produce values you’ve defined:

@Generable
enum Priority: String {
    case high, medium, low
}

@Generable
enum ContentCategory: String {
    case work, personal, health, finance, travel
}

@Generable
struct NoteAnalysis {
    @Guide(description: "Priority based on urgency and importance")
    var priority: Priority

    @Guide(description: "The best category for this note")
    var category: ContentCategory

    @Guide(description: "A one-sentence summary")
    var summary: String
}

When you generate a NoteAnalysis, the priority will always be .high, .medium, or .low. Never "urgent", never "kinda important", never garbage. This is constrained decoding — the model’s token generation is restricted at inference time to only produce valid enum cases.

Use enums aggressively. Every time you’d write a String property and hope the model returns something reasonable, ask yourself: can I make this an enum? Categories, priorities, sentiments, ratings, statuses — all safer as enums.

Building a “TL;DR” Feature

Let’s build a complete feature end-to-end: the user reads a long article, taps “Summarize,” and sees a streaming AI summary with an “AI Summary” badge.

Scenario: Marcus is reading a 2,000-word article about climate policy in his news app. He’s interested but short on time. He taps the “TL;DR” button at the top. A gentle animation starts: “Summarizing…” Then, sentence by sentence, a 3-line summary streams onto the screen. Below it, a small “AI Summary” badge and an “Expand” button to see the full article. All of this happened on his device, for free, in under 2 seconds.

The TL;DR Pattern

  1. Trigger — A clear button or gesture (“Summarize,” “TL;DR,” or an icon). Never auto-summarize without the user asking.
  2. Loading — Contextual text: “Summarizing article…” (not “Loading…”).
  3. Streaming — Show the summary building word-by-word. Users trust text they watch being written.
  4. Result — Summary with an “AI Summary” badge, plus “Expand” to see the original. Optionally, thumbs up/down for feedback.
  5. Cache — Store the summary locally. If the user returns, show the cached version instantly.

This pattern works for articles, emails, notes, chat histories, documents — anything with text the user might want shorter.

Writing Tools Integration

Here’s a gift from Apple: if your app uses standard text views (TextEditor, UITextView, NSTextView), you get Writing Tools for free. Users can select text, tap “Writing Tools,” and access Rewrite, Proofread, and Summarize — all powered by Apple Intelligence, with zero code from you.

Want to go further? You can add custom Writing Tools actions specific to your app using App Intents. A legal app could add “Simplify Legal Language.” A marketing tool could add “Make It More Persuasive.” These show up alongside Apple’s built-in options in the Writing Tools menu.

Free features are the best features. Before building a custom text AI feature, check if Writing Tools already does what you need. Summarization, proofreading, and tone adjustments are built in. Don’t reinvent what Apple gives you for free — extend it.

Text Classification In-App

Classification is the sweet spot for @Generable: the model reads text and produces a category. It’s reliable, fast, and the constrained decoding means you never get unexpected values.

@Generable
enum Sentiment: String {
    case positive, neutral, negative
}

@Generable
struct FeedbackAnalysis {
    @Guide(description: "Overall sentiment of the feedback")
    var sentiment: Sentiment

    @Guide(description: "Main topics mentioned, up to 3")
    var topics: [String]

    @Guide(description: "Is this feedback actionable by the team?")
    var isActionable: Bool
}

// Usage
let session = LanguageModelSession()
let analysis = try await session.respond(
    to: "Analyze this customer feedback: \(feedbackText)",
    generating: FeedbackAnalysis.self
)

switch analysis.sentiment {
case .positive: showConfetti()
case .neutral: logForReview()
case .negative: alertSupportTeam()
}

Because Sentiment is a @Generable enum, the model can only return .positive, .neutral, or .negative. Your switch is exhaustive. No default case needed. No “what if the model returns something weird?”

Prompt Engineering for a ~3B Model

The on-device model is capable, but it’s not GPT-5. It’s a ~3 billion parameter model optimized for speed and efficiency. Writing good prompts for it requires a different approach than prompting a cloud model:

Good: Specific, Direct

Summarize this article in exactly 3 bullet points. Each bullet should be one sentence. Focus on the main argument, the key evidence, and the conclusion.

Clear structure, specific length, focused scope. The model knows exactly what you want.

Bad: Vague, Overloaded

Read this article carefully. Consider the author's perspective, the historical context, the counterarguments, and the implications. Then provide a comprehensive analysis with recommendations.

Too many sub-tasks for a ~3B model. It will produce a shallow attempt at all of them instead of a strong answer to one.

Tips for the On-Device Model

  • Be specific about output format — “in 3 bullet points” or “in one sentence” works better than “briefly”
  • One task per prompt — Classify OR summarize OR extract. Don’t ask for all three in one prompt (use @Generable for that instead)
  • Use instructions for persona — Set session instructions for consistent behavior: LanguageModelSession(instructions: "You are a helpful recipe assistant. Answer cooking questions only.")
  • Prefer @Generable over free text — When you know the output structure, always use @Generable. It’s faster and more reliable than parsing free text
  • Give examples in-prompt — “Classify the tone. Examples: ‘I love it!’ → positive, ‘It broke on day one’ → negative” — in-context examples dramatically improve accuracy on the ~3B model
  • Use #Playground in Xcode — Add #Playground to any code file and iterate on prompts with live preview in the canvas, just like SwiftUI previews

Session Instructions vs Prompts

Foundation Models separates instructions from prompts. Instructions define how the model behaves across all interactions in a session. Prompts are individual requests. Think of instructions as the model’s job description and prompts as individual tasks:

// Instructions set the persona (stays for the whole session)
let session = LanguageModelSession(
    instructions: """
    You are a nutrition assistant for a meal planning app.
    Only answer questions about food, nutrition, and recipes.
    Keep answers under 3 sentences.
    """
)

// Prompts are individual tasks
let response1 = try await session.respond(to: "Is avocado a fruit?")
let response2 = try await session.respond(to: "How much protein in chicken?")

The instructions persist across both prompts — the model stays in character as a nutrition assistant for the entire session. This is far more reliable than repeating the persona in every prompt.

Streaming @Generable Results

You can stream structured output too — not just free text. When generating a @Generable struct, partial results arrive as properties get filled:

let stream = session.streamResponse(
    to: "Analyze this note: \(noteText)",
    generating: NoteInsights.self
)
for try await partial in stream {
    // partial.summaryBullets may be populated before priority
    updateUI(with: partial)
}

This lets you show the summary bullets appearing one by one while the priority badge fills in a moment later — a polished, progressive reveal instead of waiting for the entire struct.

Content Quality Control

The model will sometimes produce weak results. That’s not a bug — it’s the nature of a ~3B model working on-device. Plan for it:

Generate
→
Validate
→
Show or Retry
→
Fallback to Cloud
  • Regenerate button — Let users tap “Try Again” to get a different output. Sometimes the second attempt is better.
  • Prompt fallback — If the first prompt produces weak results, try a simpler version. “Summarize in one sentence” is easier for the model than “Provide a comprehensive summary.”
  • Cloud escalation — For complex tasks, detect low quality and offer to process via a cloud API: “Want a more detailed analysis? This requires an internet connection.”

Don’t silently escalate to cloud. If you switch from on-device to cloud processing, tell the user. Their data is leaving the device — that’s a privacy decision they should make, not you.

Caching Generated Content

AI inference is free but not instant. If a user summarized an article yesterday, show the cached summary today instead of regenerating it:

struct CachedSummary {
    let sourceHash: String  // Hash of the original text
    let summary: String
    let generatedAt: Date
}

func getSummary(for text: String) async throws -> String {
    let hash = text.hashValue.description

    // Check cache first
    if let cached = SummaryCache.shared.get(hash: hash) {
        return cached.summary  // Instant!
    }

    // Generate fresh
    let session = LanguageModelSession()
    let response = try await session.respond(
        to: "Summarize in 3 bullet points: \(text)"
    )

    // Cache for next time
    SummaryCache.shared.store(
        CachedSummary(sourceHash: hash, summary: response.content, generatedAt: .now)
    )
    return response.content
}

Invalidate the cache when the source text changes. A simple hash comparison works: if the hash is different, regenerate. You can store cached results in SwiftData for persistence across app launches, or use a simple in-memory dictionary for session-only caching.

Cache aggressively for text features. AI summaries and classifications rarely change for the same input text. Caching turns a 1-second generation into a 0ms lookup. Users notice — returning to a previously summarized article and seeing the result instantly feels premium.

Real-World Examples

Third-party developers are already shipping production features with Foundation Models:

Signeasy — Contract Insights

Users can ask questions about their contracts and get plain-language answers. The app uses Foundation Models to extract key clauses, dates, and obligations — all on-device. Legal documents are deeply sensitive, so on-device processing is essential for user trust. The AI highlights what matters most: “This contract auto-renews on March 15” is far more useful than making the user read 20 pages of legalese.

Zoho Notebooks — Smart Summaries

When users accumulate dozens of notes, the app summarizes each one so they can scan their library without opening every note. A one-line summary per note, generated on-device, cached locally. This turns a chaotic note collection into a scannable library — a feature that wouldn’t be economically viable if each summary cost money via a cloud API, but is trivially free with Foundation Models.

Capture — Smart Categorization

As users type a note or task, the app suggests categories in real time. “Buy groceries” gets tagged as “Shopping.” “Call Dr. Kim” gets tagged as “Health.” Foundation Models with @Generable enum output makes this reliable and instant. The categorization is a perfect fit for constrained decoding — the model can only pick from the user’s existing categories, so it never invents a random new one.

Notice the pattern: all three apps use AI to turn unstructured text into structured data. That’s the sweet spot for Foundation Models + @Generable. If your feature is about understanding, extracting, or classifying text, you’re in the right place.
