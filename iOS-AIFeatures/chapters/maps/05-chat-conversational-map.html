The Big Idea

LanguageModelSession is already built for conversations — it maintains a transcript automatically. Combined with the Tool protocol, your chatbot can do more than generate text: it can call your app’s functions, search databases, and take real actions. That’s when chat goes from “cute demo” to “genuinely useful.”

🤔 Does Your Feature Need a Chatbot?

Chat Makes Sense Chat Doesn’t Make Sense
Customer support, product discovery Changing settings, toggling a preference
Guided workflows, tutoring Choosing between 3 options (use a picker)
Data exploration, trip planning Performing a single action (use a button)
Needs unfold over multiple turns Describable in one sentence

🍽️ The restaurant test: If a waiter could hand you a menu instead of asking you to describe your drink, use a picker — not a chatbot.

🔄 Multi-Turn Sessions

  1. Create Session with Instructions

    LanguageModelSession(instructions: "You are a travel assistant...") — persona persists across all turns.

  2. Send First Message

    session.respond(to:) or session.streamResponse(to:) — the session records everything in a Transcript.

  3. Follow-Up Turns

    The model automatically has context from prior turns. “What about restaurants near the shrine?” resolves without repeating “Kyoto.”

  4. Context Limit (~4,096 tokens)

    Older messages eventually fall out. Design conversations to be focused, not infinite. Save key facts if you need longer memory.

Tool protocol
Conform to Tool with a name, description, and call(arguments:) method. The framework auto-executes calls.
@Generable Arguments
Tool arguments are @Generable structs — constrained decoding ensures the model can never produce invalid arguments.
Autonomous Invocation
The model decides when and how to call tools based on conversation context. You register tools; it decides when to use them.
Instructions (System Prompt)
The session’s “constitution.” Defines personality, scope, and boundaries. Set once, persists across all turns.
Transcript
The session’s memory. Records all prompts, responses, and tool calls. Can be saved and restored for persistence.
Rich Messages
Beyond text: cards, images, action buttons via Swift enums with associated values. Use for structured data, not every message.

📡 On-Device vs Cloud Chat

Factor On-Device (Foundation Models) Cloud (OpenAI, Claude, Gemini)
💰 Cost $0 forever $0.15–$15 / 1M tokens
🔒 Privacy Data never leaves device Sent to third-party servers
📏 Context ~4,096 tokens 128K–1M+ tokens
🧠 Reasoning Good for simple tasks Excellent for complex reasoning
📶 Offline ✅ Works anywhere ❌ Requires internet

🔀 Hybrid approach: Route simple queries on-device, complex ones to cloud. Always tell users when data leaves the device.

⚠️ Edge Case Checklist

👋
Empty Conversation
Show starter prompts, never a blank screen. “Plan a trip,” “Find recipes,” “Help me decide.”
📏
Long Conversations
~4,096 token limit means older turns fall out. Save key facts in instructions for longer memory.
⚡
Rapid-Fire Messages
Queue messages, process sequentially. Show “Please wait for current response” if needed.
📡
Network Errors
Show partial response + “Tap to retry” badge. Don’t discard already-streamed text.
🛡️
Offensive Input
Foundation Models has built-in safety filters. For cloud: add your own. Respond gracefully, don’t lecture.
📱
Unsupported Devices
Check SystemLanguageModel.default.isAvailable. Fall back to cloud-only or disable with explanation.

🧪 Test Yourself

Quick Quiz

How does LanguageModelSession handle multi-turn context?

It maintains a Transcript of all prompts and responses automatically. You don’t resend history — the session remembers previous turns. The transcript can also be saved and restored for persistence across app restarts.

What are the three requirements of the Tool protocol?

A name (unique identifier), a description (so the model knows when to call it), and a call(arguments:) method. The arguments must be a @Generable type for constrained decoding.

When should you NOT use a chatbot?

When the interaction can be described in one sentence. Changing settings, toggling preferences, choosing between a few options — these are buttons and pickers, not conversations. The restaurant test: if a menu would be faster than typing, use a picker.

What’s the context window limit for on-device chat?

~4,096 tokens. Fine for 10–15 focused turns. For very long conversations, older messages fall out. Save key facts in instructions for longer memory, or route to a cloud API with a larger context window.

Should you stream short responses (under ~30 words)?

No. Short confirmations like “Done! Alarm set for 7am” should appear all at once with a fade-in. Streaming short text feels sluggish. Stream responses over ~30 words; show shorter ones instantly.

Key Takeaways

  • Not every feature needs a chatbot. If a button or picker works, use that instead.
  • LanguageModelSession is stateful — it remembers previous turns via its Transcript. No manual history management.
  • Stream long responses word-by-word. Show short responses (<30 words) all at once.
  • Tool calling turns a text generator into an app interface — search, create, navigate, all from chat.
  • Tool arguments use @Generable — same constrained decoding from Chapter 4, so arguments are always valid.
  • The ~4,096 token context window is fine for focused conversations. Design for focus, not infinity.
  • Hybrid routing: simple queries on-device ($0), complex ones to cloud. Always tell users when data leaves.
  • Conversation starters eliminate the blank-page problem. Never show an empty chat screen.
  • Persist conversations with SwiftData. Users expect chat history to survive app restarts.
  • Steal from Messages.app: bubble layout, scroll behavior, keyboard interaction, animation timing.
