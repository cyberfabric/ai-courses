Smart Text with Foundation Models

The Big Idea

@Generable turns the on-device LLM from a text generator into a typed data extractor. Define a Swift struct, the model fills it — guaranteed to match your schema. No JSON parsing. No regex. No wrong format. The compiler and the model work together.

🧱 The Three Key Objects

💬
LanguageModelSession
Your conversation with the model. Stateful — remembers previous turns. Set persona via instructions: parameter.
🏗️
@Generable
Macro on structs/enums. Model fills your types via constrained decoding. The killer feature of Foundation Models.
🛡️
@Guide
Constraints on properties. Descriptions help the model, enums restrict output to valid values only.

🎯 Common @Generable Patterns

Structured Extraction
Text → typed struct. "Meet Anna Thursday 2pm" → MeetingDetails(attendees: ["Anna"], dateTime: "Thursday 2pm")
Classification with Enum
Text → enum case. "I love it!" → Sentiment.positive. Exhaustive switch, no default needed.
Multi-field Analysis
Text → struct with mixed types. Summary (String) + priority (enum) + isActionable (Bool) in one call.
Streaming Structs
streamResponse(generating:) fills properties progressively. Show bullets first, then priority badge.

📝 The TL;DR Pattern

  1. Trigger

    Clear button or gesture. Never auto-summarize without asking.

  2. Loading

    Contextual text: "Summarizing article…" (not "Loading…").

  3. Streaming

    Show summary building word-by-word. Users trust text they watch being written.

  4. Result

    "AI Summary" badge + "Expand" button + optional feedback.

  5. Cache

    Store locally. Return visits show cached result instantly (0ms vs 1s).

✍️ Prompt Engineering for ~3B Models

Do This Not This
"Summarize in exactly 3 bullet points" "Briefly summarize"
One task per prompt (classify OR summarize) Multiple complex sub-tasks in one prompt
Give in-context examples: "'I love it!' → positive" Assume the model infers your categories
Use @Generable when you know the output shape Parse free-text responses with regex
Set persona via session instructions: Repeat the persona in every prompt
Iterate with #Playground in Xcode Guess at prompts without testing

🧪 Test Yourself

Quick Quiz

What does @Generable do that's different from a regular API call?

It constrains the model to fill a typed Swift struct or enum via constrained decoding. You get a MeetingDetails instance, not a String. The model literally cannot produce output that doesn't match your schema.

Why should you prefer enums over Strings for classification properties?

Enums restrict the model to your predefined values only. Priority.high is guaranteed — never "urgent" or "kinda important." Your switch is exhaustive with no default case needed.

What's the difference between session instructions and prompts?

Instructions = the model's job description (persists across all prompts in the session). Prompts = individual tasks. Set persona via instructions once, not in every prompt.

How do you get Writing Tools for free in your app?

Use standard text views (TextEditor, UITextView, NSTextView). Rewrite, Proofread, and Summarize appear automatically. Zero lines of code.

When should you cache AI-generated text?

Always for summaries and classifications. Hash the source text; if unchanged, return the cached result (0ms vs ~1s). Invalidate when the source changes.

Key Takeaways

  • @Generable turns unstructured text into typed Swift data. Define a struct, the model fills it.
  • @Guide with enums = constrained decoding. The model can only produce values you've defined.
  • Instructions set persona for the session. Prompts are individual tasks. Don't repeat the persona.
  • Standard text views get Writing Tools (Rewrite, Proofread, Summarize) for free. Don't reinvent it.
  • Prompt a ~3B model with one specific task, explicit format, and in-context examples.
  • Cache aggressively — summaries and classifications rarely change for the same input.
  • Quality control: Generate → Validate → Retry → Cloud fallback. Never silently escalate.
  • The sweet spot: turning unstructured text into structured data (extraction, classification, summarization).
