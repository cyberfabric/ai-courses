The Big Idea

Great AI features learn. They notice that you prefer short summaries, search for vegetarian recipes, and open the app at 8 AM. Three tools make it happen: LoRA adapters teach the model your domain language, MLUpdateTask personalizes models with user behavior, and preference scoring ranks content by interest. All on-device. All private. No data ever leaves the phone.

🔧 Three Personalization Tools

🧬
LoRA Adapters
~30 MB trainable layers added to the 3B base model. Teaches domain language (medical, legal, brand voice). Trained offline with Python toolkit, loaded at runtime.
📈
MLUpdateTask
Retrains Core ML models on-device with user interactions. The model improves for this specific user. Runs in background, takes seconds for small models.
⭐
Preference Scoring
Simple category weights (0.0–1.0) updated by taps (+0.05), saves (+0.15), time spent (+0.1), dismissals (−0.1). Sort feed by score. 25 lines of Swift.

🎯 Three Recommendation Approaches

Approach How It Works Best For

Collaborative Filtering “Users like you also liked X.” MLRecommender in Create ML. Large user bases
Hybrid Combine content features + user behavior. Weight each signal. Production systems

🔮 The Proactive Assistant Pattern

  1. Track Patterns

    Store last 30 app open times. Detect clusters: “opens between 7:45–8:15 AM on weekdays.”

  2. Predict Need

    5 minutes before the pattern, prepare content. Travel app → boarding pass at airport. Recipe app → dinner ideas at 5 PM.

  3. Surface Proactively

    Widget update, notification, or in-app card. Label clearly: “Your morning briefing.”

  4. Measure & Adjust

    Track if the user acts on proactive content. If dismissed repeatedly, reduce frequency.

🫧 Avoiding the Filter Bubble

🎲
Serendipity Injection
Reserve 10–20% of the feed for items outside user’s usual preferences. Label “Something New.”
📊
Category Cap
Never let one category dominate >40% of the visible feed, even if the model loves it.
⏳
Time Decay
Multiply scores by 0.95 weekly. December’s baking obsession fades by July.
🔘
Explicit Controls
“Show me less/more of this” buttons. Let users override the algorithm. Offer reset option.
❓
“Why This?” Labels
“Because you saved 3 Indian recipes.” Specific. Transparent. 30% more engagement than “Recommended for you.”

📖 Key Terms

LoRA (Low-Rank Adaptation)
PEFT technique: adds small trainable layers to transformer without changing base weights. ~30 MB adapter for the 3B model.
MLUpdateTask
Core ML API for on-device model retraining. Feed user interactions, get a personalized model. No data leaves the device.
MLRecommender
Create ML tool for collaborative filtering. Trains on user-item interactions, exports as .mlmodel for on-device inference.
Content-Based Filtering
Recommend items similar to what the user liked before. Uses item features (category, tags, keywords).
Collaborative Filtering
“Users like you also liked X.” Requires aggregate interaction data from many users.
Serendipity Injection
Intentionally showing 10–20% content outside user preferences to prevent echo chambers.
Time Decay
Multiplying preference scores by 0.95 weekly so old interests fade and recent behavior dominates.
Proactive Assistant
Surfacing relevant information before the user asks, based on time, location, and behavior patterns.

🧪 Test Yourself

Quick Quiz

When should you use a LoRA adapter instead of prompt engineering?

When your domain has specialized vocabulary (medical, legal, brand-specific) that the general model handles poorly, AND you have hundreds to thousands of training examples. For general tasks, simple classification, or one-off use, prompt engineering with @Guide constraints is cheaper and sufficient.

How does MLUpdateTask enable private personalization?

MLUpdateTask retrains a Core ML model directly on the user’s device using their interaction data (taps, saves, time spent). The updated model is better for this specific user. No data leaves the phone — training happens locally in seconds for small models.

What’s the simplest way to build a recommendation engine on iOS?

Preference scoring: maintain category weights (0.0–1.0), update on interactions (tap +0.05, save +0.15, dismiss −0.1), sort content by score. 25 lines of Swift, stored in SwiftData. Apply weekly time-decay of 0.95 so preferences evolve.

How do you avoid the filter bubble in recommendations?

Five strategies: serendipity injection (10–20% outside preferences), category cap (no category >40%), time decay (old preferences fade), explicit controls (“show less/more”), and “Why this?” labels for transparency. Users who understand why get 30% more engagement.

What’s the proactive assistant pattern?

Surface relevant information before the user asks. Track open-time patterns, predict when they’ll want content, and deliver 5 minutes early. Travel app → boarding pass at airport. Recipe app → dinner ideas at 5 PM. Label clearly so it feels helpful, not creepy.

Key Takeaways

  • Three personalization tools: LoRA adapters (domain language), MLUpdateTask (user behavior), preference scoring (content ranking).
  • LoRA adapters are ~30 MB, trained offline with Apple’s Python toolkit, loaded at runtime. Use for specialized vocabulary only.
  • MLUpdateTask retrains Core ML models on-device with user interactions. No data leaves the phone.
  • Simple preference scoring (category weights + interaction signals) works surprisingly well. 25 lines of Swift.
  • Three recommendation approaches: content-based, collaborative filtering (MLRecommender), and hybrid.
  • Proactive assistant: predict user needs from behavior patterns, surface content before they ask.
  • Smart notification timing: find peak hours from open history, schedule 1 hour before. Cap at 1–2/day.
  • Fight the filter bubble: serendipity injection (10–20%), category cap (40%), time decay (0.95/week), explicit user controls.
  • “Why this?” explainers are not optional. Specific reasons beat “Recommended for you” by 30%.
  • Measure: CTR (should rise), dismissal rate (should fall), opt-out rate (keep below 5%).
