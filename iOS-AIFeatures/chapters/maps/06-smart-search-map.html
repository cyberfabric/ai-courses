The Big Idea

Keyword search matches characters. Semantic search matches meaning. NLEmbedding turns text into 512-dimensional vectors on-device — “blue jacket” finds “navy parka” because their meanings are close. Combine with Foundation Models content generation to go from finding things to creating things.

📖 Key Terms

NLEmbedding
Apple’s Natural Language framework class for text embeddings. 512-dim vectors, 7 languages, on-device, iOS 13+.
Cosine Similarity
Measures how similar two vectors are (1.0 = identical meaning, 0 = unrelated). The math behind semantic search.
Sentence Embedding
NLEmbedding.sentenceEmbedding(for: .english) — embeds full sentences/phrases, not just individual words.
Two-Tier Autocomplete
Prefix matching (instant, free) + semantic suggestions (200ms debounce, NLEmbedding). Best of responsive + smart.
AI Draft Pattern
AI generates a starting point, user edits to make it theirs. Eliminates the blank page, preserves user voice.
Map-Reduce
Split long text into chunks, summarize each (map), then summarize the summaries (reduce). Overcomes the ~4,096 token limit.
Personalized Search
Boost results based on user behavior (category preferences, favorites, recent searches). All on-device, private.
.searchable
SwiftUI modifier for adding a search bar with suggestions. iOS 26 adds searchToolbarBehavior() and search tab role.

🧪 Test Yourself

Quick Quiz

Why does “blue jacket” find “navy parka” with semantic search?

Because NLEmbedding converts both into 512-dim vectors that capture meaning. “Blue jacket” and “navy parka” have similar vectors (high cosine similarity) because they mean similar things — even though they share zero words.

What are the two tiers in the autocomplete pattern?

Tier 1: Prefix matching (instant, free) shows items starting with the typed text. Tier 2: Semantic suggestions (200ms debounce) uses NLEmbedding to find conceptually related items. Show both: prefix first, semantic below.

When should you use cloud embeddings instead of NLEmbedding?

When you need more languages (NLEmbedding supports 7, cloud supports 100+), higher quality vectors, or search across millions of items in a server-side vector database. For personal/on-device content, NLEmbedding is sufficient.

How do you handle documents that exceed the ~4,096 token context window?

Map-reduce chunking: split into ~2,000-token chunks, summarize each chunk independently (map), then combine the summaries and produce a final cohesive summary (reduce). Two good summaries merged beats one truncated summary.

Why should you ground content generation in search results?

Generating content without real data risks hallucination — the model makes things up. Search finds real data first; generation makes it actionable. “Generate Shopping List” from a found recipe is grounded. “Write me a recipe” is not.

Key Takeaways

  • NLEmbedding turns text into 512-dim vectors on-device. Semantic search matches meaning, not characters.
  • Embed content once at creation time, store vectors, search by cosine similarity. ~1ms per item to index.
  • Two-tier autocomplete: prefix match (instant) + semantic suggestions (200ms debounce). Responsive AND smart.
  • NLEmbedding: 7 languages, iPhone 8+, free, offline. Cloud embeddings: 100+ languages, higher quality, costs money.
  • AI Draft pattern: AI creates the starting point, user refines. Never present AI content as final.
  • @Generable templates guarantee structured output: headline, body, highlights — typed fields, no parsing.
  • Map-reduce for long documents: chunk → summarize each → summarize the summaries.
  • Search + Generate is the killer combo. Ground generation in real data to avoid hallucination.
  • Personalize on-device: boost categories from user history, weight favorites. No data leaves the phone.
  • Don’t over-personalize — avoid the filter bubble with a “Discover Something New” section.
