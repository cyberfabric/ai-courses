Chapter 11

Smart Personalization & Recommendations

LoRA adapters, on-device learning, recommendation engines, and proactive features that anticipate what users need

In the previous chapters, your AI features treated every user identically. The same summary style. The same search ranking. The same suggestions. That’s like a barista who makes every coffee exactly the same, regardless of who walks in.

Great AI features learn. They notice that you prefer short summaries, that you always search for vegetarian recipes, that you open the app at 8 AM for news and 9 PM for relaxation. And they adapt — quietly, privately, on-device.

The Apple Way: Private Personalization

Apple Intelligence personalizes by processing data on-device using information from Calendar, Mail, Photos, and your app. No user data is sent to Apple. No interactions train Apple’s models. Users can see exactly what’s happening at Settings > Privacy & Security > Apple Intelligence Report. Your app should follow the same philosophy.

User Story: The App That Knows You

Priya uses a recipe app. On day one, she sees generic suggestions: pasta, salad, steak. Over the next two weeks, she taps on Indian food, saves vegetarian recipes, and spends the most time reading quick weeknight meals. By week three, the app’s home feed leads with “Quick Vegetarian Indian Dinners.” There’s a small “Because you love quick Indian recipes” label under the section. She never configured anything — the app just noticed.

Custom LoRA Adapters: Teaching the Model Your Language

Prompt engineering works for most tasks. But some domains have specialized vocabulary that a general-purpose 3B model doesn’t handle well: medical terminology, legal jargon, brand-specific style, niche industry terms. That’s where LoRA adapters come in.

LoRA (Low-Rank Adaptation) adds small trainable layers into the transformer without changing the base model. Think of it like teaching the model a new dialect without making it forget English.

~30 MB
Adapter size
A rank-16 adapter is tens of megabytes. The 3B base model stays frozen — you only ship the small adapter.
100%
On-device
Adapters load into Foundation Models at runtime. No cloud. Same privacy as the base model.
Python
Training toolkit
Apple provides a Python-based training toolkit with scripts, notebooks, and evaluation tools.

When to Use LoRA Adapters

Good Candidates for LoRA

  • Medical apps: Correct drug names, dosages, medical abbreviations
  • Legal apps: Contract terminology, legal citation formats
  • Brand voice: Company-specific tone and terminology
  • Specialized domains: Cooking techniques, fitness terminology, finance jargon

Don’t Use LoRA For

  • General text: Prompt engineering handles this fine
  • Simple classification: @Generable with enum is cheaper
  • Personalization: User preferences change too fast for adapter retraining
  • One-off tasks: Training cost only makes sense for repeated use

API Recipe: Loading a Custom LoRA Adapter

Goal: Load a domain-specific adapter into Foundation Models at runtime.

Step 1: Train the adapter (Python)

Use Apple’s adapter training toolkit (available at developer.apple.com/apple-intelligence/foundation-models-adapter/). Train on your domain data. Export the adapter file.

Step 2: Bundle the adapter in your app

Add the exported adapter file (typically .mlpackage or .adapters) to your Xcode project’s resources.

Step 3: Load and use in Swift

import FoundationModels

// Load the adapter from your app bundle
let adapterURL = Bundle.main.url(
    forResource: "MedicalTerms",
    withExtension: "adapters"
)!
let adapter = try LanguageModelAdapter(url: adapterURL)

// Create a session with the adapter
let session = LanguageModelSession(adapter: adapter)

// Generate with domain-specific understanding
let response = try await session.respond(
    to: "Summarize this patient note: \(patientNote)"
)

Result: The model now understands medical terminology, drug names, and clinical abbreviations. Same Foundation Models API, but with domain expertise.

Warning: Adapter Training Requires Data

You need hundreds to thousands of domain-specific examples to train an effective adapter. Poor training data produces poor results. If you don’t have enough domain data, stick with prompt engineering and @Guide constraints — they’re surprisingly effective for most use cases.

On-Device Learning with Create ML

LoRA adapters are trained offline and shipped with your app. But what about learning from individual user behavior in real time? That’s where MLUpdateTask comes in.

MLUpdateTask lets you retrain a Core ML model directly on the user’s device using their interactions. The model gets better for this specific user without any data leaving the phone.

How It Works

1

Ship a Base Model

Train a general classification model with Create ML. Ship it in your app bundle.

→
2

Collect Interactions

Track user behavior: taps, saves, time spent, dismissals. Store locally.

→
3

Update On-Device

MLUpdateTask retrains with user data. ~seconds for small models.

→
4

Replace the Model

Swap the updated model into your inference pipeline. Immediately better.

import CoreML

let updatedModel = try MLUpdateTask(
    forModelAt: modelURL,
    trainingData: userInteractions,  // MLBatchProvider
    configuration: updateConfig,
    completionHandler: { context in
        // Save the updated model
        let updatedURL = context.model.write(to: documentsURL)
        self.currentModel = try MLModel(contentsOf: updatedURL)
    }
)
updatedModel.resume()

Tip: Update in Background

Run MLUpdateTask when the app is idle or during background refresh, not during active use. Training takes a few seconds for small models but can spike CPU usage. Schedule it when the user puts the phone down, not when they’re scrolling.

Building a Recommendation Engine

Recommendation engines don’t have to be Netflix-scale. A simple on-device system that tracks user preferences and ranks content can dramatically improve engagement — and it costs nothing to run.

Three Approaches

ApproachHow It WorksBest For
Content-Based Recommend items similar to what the user liked before. Uses item features (category, tags, keywords). Small catalogs, new users with some history
Collaborative Filtering “Users like you also liked X.” Uses MLRecommender in Create ML. Large user bases, pre-trained on aggregate data
Hybrid Combine both: content features + user behavior. Weight each signal. Production systems, best accuracy

A Simple On-Device Preference Model

struct UserPreferences {
    // Category weights: higher = user likes it more
    var categoryScores: [String: Double] = [:]

    mutating func recordInteraction(
        category: String, signal: InteractionSignal
    ) {
        let current = categoryScores[category, default: 0.5]
        switch signal {
        case .tap:      categoryScores[category] = min(current + 0.05, 1.0)
        case .save:     categoryScores[category] = min(current + 0.15, 1.0)
        case .timeSpent(let seconds):
            if seconds > 30 {
                categoryScores[category] = min(current + 0.1, 1.0)
            }
        case .dismiss:  categoryScores[category] = max(current - 0.1, 0.0)
        }
    }

    func score(for item: ContentItem) -> Double {
        categoryScores[item.category, default: 0.5]
    }
}

Store this in SwiftData or UserDefaults. Sort your feed by score(for:). That’s a working personalization engine in 25 lines of Swift.

Tip: Decay Over Time

Preferences change. A user who loved baking content last month might be into salads now. Apply a time-decay factor: multiply scores by 0.95 weekly so old preferences gradually fade and recent behavior dominates.

Product Pattern: The Proactive Assistant

The pattern: Surface relevant information before the user asks for it, based on time, location, and behavior patterns.

Why it works: The best AI features feel like they read your mind. A travel app showing your boarding pass when you arrive at the airport. A fitness app suggesting your usual workout at your usual time. A recipe app surfacing dinner ideas at 5 PM.

Where to use it: Any app with content the user accesses in patterns — news at breakfast, recipes before dinner, workout at gym time, music during commute.

Implementation:

  • Track open times in a simple array (last 30 opens)
  • Detect clusters: “user opens between 7:45–8:15 AM on weekdays”
  • Schedule proactive content (widget update, notification) 5 minutes before the pattern
  • Label it clearly: “Your morning briefing” not just random push

Smart Notification Timing

Sending notifications at the wrong time is worse than not sending them at all. A notification at 3 AM gets dismissed; one that arrives exactly when the user was about to open the app feels magical.

Building a Timing Model

struct NotificationScheduler {
    // Track when the user typically opens the app
    var openTimes: [Date] = []

    func optimalSendTime() -> DateComponents? {
        let calendar = Calendar.current
        // Group opens by hour-of-day on weekdays
        let weekdayHours = openTimes
            .filter { calendar.isDateInWeekday($0) }
            .map { calendar.component(.hour, from: $0) }

        guard !weekdayHours.isEmpty else { return nil }

        // Find the most common hour
        let frequency = Dictionary(grouping: weekdayHours) { $0 }
            .mapValues { $0.count }
        let peakHour = frequency.max(by: { $0.value < $1.value })?.key

        guard let hour = peakHour else { return nil }
        var components = DateComponents()
        components.hour = hour - 1  // 1 hour before peak
        components.minute = 0
        return components
    }
}

Warning: Notification Fatigue Is Real

Personalized doesn’t mean frequent. Even perfectly timed notifications annoy users if they arrive too often. Cap at 1–2 proactive notifications per day. Track the opt-out rate — if it climbs above 5%, you’re pushing too hard.

Personalizing Notification Content

Same notification, different users, different copy. Foundation Models can generate personalized notification text that matches the user’s style.

Personalized Notifications

  • “3 new vegetarian Indian recipes this week” (matches Priya’s preferences)
  • “Your morning news: 5 stories about AI” (matches Marcus’s interests)
  • “Quick 20-min workout ready” (matches the user’s preferred duration)

Generic Notifications

  • “New recipes available!”
  • “Check out today’s news”
  • “Time for a workout!”

Measuring Personalization Quality

Personalization is only valuable if it actually helps. Measure these metrics to know if your system is working.

CTR
Click-through rate
% of recommended items the user taps. Should increase over time as the model learns.
Dismiss
Dismissal rate
% of recommendations dismissed without action. If this is rising, your model is getting worse.
Time
Time-to-action
How quickly the user acts on a recommendation. Faster = more relevant.
Opt-out
Opt-out rate
% of users who disable personalization. Above 5% = something’s wrong.

Avoiding the Filter Bubble

A recommendation engine that only shows what users already like creates an echo chamber. Last month’s obsession with pasta becomes a permanent pasta prison. Here’s how to break the bubble.

Tip: The “Why This?” Label Is Not Optional

Users who understand why they see a recommendation are 30% more likely to engage with it. “Because you saved 3 Indian recipes” is specific and helpful. “Recommended for you” is vague and suspicious. Foundation Models can generate these explanations from the preference model’s data.

Exercise: Build a Smart Feed

Build a personalized content feed that learns from user behavior:

  1. Track interactions: Record taps, saves, time spent, and dismissals for each content item. Store with category and timestamp in SwiftData.
  2. Build the preference model: Implement a UserPreferences struct that maintains category scores (0.0–1.0) based on interaction signals. Apply weekly time-decay of 0.95.
  3. Rank content: Sort your feed by the preference model’s score. Higher-scored categories appear first.
  4. Add serendipity: Reserve 15% of the feed for items outside the user’s top 3 categories. Label them “Discover Something New.”
  5. Add explainers: Show “Because you saved 3 items in [category]” under personalized sections. Use Foundation Models to generate natural-sounding explanations.
  6. Measure: Log CTR, dismissal rate, and time-to-action. Compare personalized vs non-personalized sections.

Stretch goal: Add an MLUpdateTask-powered classification model that learns the user’s preferred content complexity (beginner vs advanced) and adjusts the feed accordingly.
