Choosing Your AI Stack on iOS

Foundation Models vs Core ML vs Vision vs Cloud APIs — a decision framework that actually helps

The Four Paths

In Chapter 1, we previewed three approaches to AI on iOS. The reality is richer — there are four distinct paths, each with a sweet spot where it shines and a range where it struggles. Choosing wrong costs you months of rework. Choosing right means your feature ships faster, costs less, and delights users from day one.

Let’s break each one down, then build a decision framework that actually works in practice.

Path 1: Foundation Models Framework

This is Apple’s headline AI offering, introduced at WWDC 2025. It gives your app direct access to the same ~3B parameter on-device language model that powers Writing Tools, Mail suggestions, and Siri.

Scenario: David is building a recipe app. When a user pastes a long recipe from the web, the app extracts a structured ingredient list, cooking steps, and prep time — all in under a second, entirely on-device, with no API key and no server cost. His app works in airplane mode, in mountain cabins with zero signal, and on the subway. The 100K users who do this three times a week cost him exactly $0 in inference.

What It Can Do

  • Text generation — summaries, drafts, rewrites, translations
  • Structured output — the @Generable macro constrains the model to fill typed Swift structs and enums, guaranteeing structural correctness
  • Classification — sentiment, categories, priority levels — with @Guide constraints that restrict output to your predefined values
  • Tool calling — the model can invoke functions you define, bridging conversation with app actions
  • Multi-turn sessions — LanguageModelSession maintains conversation context automatically
  • Streaming — responses arrive as AsyncSequence, perfect for real-time SwiftUI display

What It Can’t Do (Well)

  • Complex multi-step reasoning (it’s a ~3B model, not GPT-5)
  • Processing very long documents (limited context window)
  • Image understanding or generation (text only)
  • Tasks requiring world knowledge beyond its training data
$0 Cost per inference — forever
<1s Typical response for short prompts
A17 Pro+ Minimum chip — iPhone 15 Pro or M1 Mac

Path 2: Apple’s Built-In Frameworks

Before Foundation Models existed, Apple already gave you a powerful suite of on-device AI through specialized frameworks. These are still the best choice for many tasks — they’re faster, lighter, and support far more devices.

Framework What It Does Key APIs
Vision Image analysis — OCR, face detection, document understanding, image classification, object tracking VNRecognizeTextRequest, RecognizeDocumentsRequest, VNClassifyImageRequest
Natural Language Text analysis — entities, sentiment, language detection, text embeddings for semantic search NLTagger, NLEmbedding, NLLanguageRecognizer
SpeechAnalyzer Speech-to-text — real-time transcription, long-form audio, distant microphone support SpeechAnalyzer (WWDC 2025, replaces SFSpeechRecognizer)
Sound Analysis Audio classification — 300+ sound categories (applause, sirens, music, speech) SNClassifySoundRequest, SNAudioStreamAnalyzer

Don’t overlook Apple frameworks. Need OCR? Use Vision — it’s faster and lighter than sending text to Foundation Models for extraction. Need semantic search? NLEmbedding is purpose-built for it. These frameworks run on devices going back to iPhone 8. Foundation Models is powerful, but it’s not always the right hammer.

Path 3: Core ML (Custom Models)

When Apple’s built-in models don’t fit your specific domain, Core ML lets you run your own machine learning models on-device. You train a model (or convert one from PyTorch/TensorFlow), package it as a .mlmodel or .mlpackage, and ship it inside your app bundle.

Common use cases:

  • Custom image classifiers — “Is this a skin condition worth seeing a doctor about?” (trained on your medical dataset)
  • Domain-specific NLP — legal clause detection, scientific paper parsing
  • Recommendation models — personalized content ranking trained on your app’s engagement data
  • On-device personalization — MLUpdateTask fine-tunes a model with the user’s data, entirely on their device

Core ML is the hard path. You need training data, ML expertise, model optimization, and ongoing maintenance. If an Apple framework or Foundation Models can do the job, prefer those. Save Core ML for when you need something truly custom. For deeper coverage, see iOS-OnDevice Chapters 3, 8, and 11.

Path 4: Cloud APIs

Sometimes on-device isn’t enough. When you need complex reasoning, massive context windows, photorealistic image generation, or capabilities that exceed a ~3B model, cloud APIs are the answer.

Cloud APIs give you the most powerful models available — but they come with real tradeoffs: they cost money per request, require internet connectivity, introduce latency, and send user data off-device (which means privacy nutrition labels light up).

The Decision Matrix

Now the practical part. When you’re staring at a feature spec, how do you pick the right path? Five factors matter:

The Five-Factor Decision Framework

  1. Latency — Does the feature need to respond in <100ms? On-device only. Autocomplete, real-time transcription, and camera features can’t tolerate a round trip to the cloud.
  2. Privacy — Does the feature touch sensitive data? Health, financial, personal photos, messages = on-device only. Users see your privacy nutrition label before they download.
  3. Cost at Scale — Foundation Models = $0 per inference. Cloud APIs at 100K DAU × 3 requests/day = 300K requests/day. Even at $0.001 per request, that’s $9,000/month. At $0.01 per request, it’s $90,000/month.
  4. Device Coverage — Foundation Models requires A17 Pro or later. That excludes iPhone 15 (non-Pro), iPhone 14, and all older devices. Vision and Natural Language frameworks go back much further.
  5. Capability — Can the on-device model handle the task? For complex reasoning, long documents, or image generation, you may need cloud.

Cost Modeling: The Numbers That Matter

Let’s make the cost argument concrete. Imagine you’re building an AI summary feature with 100K daily active users, each triggering 3 summaries per day:

Foundation Models (On-Device)

100K users × 3 requests/day = 300K inferences/day Cost per inference: $0 Daily cost: $0 Monthly cost: $0 Annual cost: $0 Plus: works offline, no server to maintain

The economics are transformative. Free inference changes what features are viable.

Cloud API (e.g., GPT-5.4 Mini)

100K users × 3 requests/day = 300K inferences/day ~500 tokens per request (in+out) Daily tokens: 150M Cost at $0.75/1M tokens: ~$112/day Monthly cost: ~$3,375 Annual cost: ~$40,500 Plus: server costs, monitoring, error handling

Not catastrophic, but $40K/year adds up — and that’s with the cheapest model. Scale to 500K DAU and it’s $200K/year.

The economics are clear: if Foundation Models can handle the task, use it. The $0 inference cost is not just a nice-to-have — it changes which features are financially viable. Features that would lose money as cloud API calls become free as on-device inference.

Device Compatibility: The Real-World Challenge

Here’s the uncomfortable question: Foundation Models requires iPhone 15 Pro or later. What about the rest of your users?

In 2026, a significant portion of active iPhones are still iPhone 12, 13, and 14 models. You have three strategies:

Strategy 1: Graceful Degradation

Check at runtime whether Foundation Models is available. If it is, show the AI feature. If not, show a simpler version or skip it entirely.

import FoundationModels

func isFoundationModelsAvailable() -> Bool {
    return SystemLanguageModel.default.isAvailable
}

// In your view
if isFoundationModelsAvailable() {
    AISummaryButton(article: article)
} else {
    // Fallback: show a manual "highlight key points" feature
    ManualHighlightView(article: article)
}

Strategy 2: Framework Fallback

Use Foundation Models on supported devices, fall back to Vision or Natural Language on older devices. For example, text extraction can use Foundation Models for smart parsing on new devices and VNRecognizeTextRequest for raw OCR on older ones.

Strategy 3: Cloud Fallback

Route to a cloud API on older devices. This adds cost and requires internet, but gives all users access to the AI feature. Use this sparingly — only for features where the AI is core to the experience.

Don’t gate your entire app on Foundation Models. Make AI features optional enhancements, not hard requirements. The best AI features feel like a bonus — the app works fine without them, but they’re delightful when present.

Private Cloud Compute

There’s a middle ground between on-device and third-party cloud: Apple’s Private Cloud Compute (PCC).

When a Foundation Models request exceeds what the on-device model can handle, Apple can route it to PCC — larger models running on Apple silicon servers in data centers. The key guarantees:

  • No data storage — your data is processed and discarded, never stored on the server
  • No remote access — Apple intentionally built PCC with no remote shell or debugging mechanisms
  • Cryptographically auditable — the server code is published for security researchers to verify
  • Same privacy model — the privacy nutrition label stays “No Data Collected”

From your code’s perspective, PCC is transparent — you make the same LanguageModelSession call, and the framework decides whether to process on-device or route to PCC. You don’t need to handle this routing yourself.

The Hybrid Architecture

In practice, most production apps use a hybrid approach. On-device handles fast, private, frequent tasks. Cloud handles complex, infrequent, heavy tasks. The routing logic lives in a simple Swift layer:

Recipe: Build a Hybrid AI Router

Goal: Route AI requests to the best available provider based on task complexity and device capability.

enum AIProvider {
    case foundationModels  // Free, on-device, private
    case appleFramework    // Free, on-device, wider support
    case cloudAPI          // Powerful, costs money
}

struct AIRouter {
    func bestProvider(for task: AITask) -> AIProvider {
        switch task {
        case .classify, .summarizeShort, .extractStructured:
            // Fast, simple tasks — on-device if available
            if SystemLanguageModel.default.isAvailable {
                return .foundationModels
            }
            return .appleFramework  // NLTagger, Vision, etc.

        case .ocrExtraction, .faceDetection, .imageClassification:
            // Specialized vision/NLP tasks — Apple frameworks are best
            return .appleFramework

        case .complexReasoning, .longDocument, .imageGeneration:
            // Heavy tasks — cloud API needed
            return .cloudAPI
        }
    }
}

This isn’t production code — it’s a mental model. In a real app, you’d add error handling, connectivity checks, and user preferences (some users may opt out of cloud processing). The key insight: make the routing decision transparent to the UI layer. Your SwiftUI views shouldn’t care which provider generated the result.

API Key Security on iOS

If you’re using cloud APIs, you need to handle API keys. And here’s the critical rule:

Never ship API keys in your app binary. An IPA file is a zip archive. Anyone can extract it, search for strings, and find your API key. They then use your key, you get the bill, and your key gets rate-limited or banned. This happens constantly.

The Right Way: Your Own Backend as a Proxy

The API Proxy Pattern

Your iOS app talks to your server. Your server talks to the cloud API. The API key lives on your server, never on the device. Your server also enforces rate limits, monitors costs, and can cut off abusive users instantly. This is the only production-safe pattern for cloud AI on iOS.

Additional security layers:

  • App Attest — Apple’s attestation API proves the request comes from a legitimate copy of your app, not a script
  • Per-user rate limits — cap cloud requests per user per day (e.g., 50/day for free tier, 200/day for premium)
  • Cost circuit breakers — if daily API spend exceeds a threshold, pause cloud features and fall back to on-device

Putting It All Together: The Decision Cheat Sheet

When you’re staring at a feature request, run through this checklist:

If your feature needs… Use this Why
Text summarization, classification, structured extraction Foundation Models Free, fast, private, @Generable for typed output
OCR, document scanning, image classification Vision framework Purpose-built, faster than LLM, wider device support
Semantic search, text similarity, entity extraction Natural Language NLEmbedding is optimized for this exact task
Speech-to-text, real-time transcription SpeechAnalyzer On-device, supports distant mic and long-form audio

Real-World Architecture Examples

Smart Email Client

Summarize emails: Foundation Models (free, on-device, handles short text well).
Extract contacts and dates: Natural Language (NLTagger — purpose-built, wide device support).
Generate detailed replies to complex threads: Cloud API (needs reasoning and context that exceeds on-device).
Result: 90% of requests are free and private. The 10% that hit cloud are the high-value cases where users will accept a brief wait.

Document Scanner App

OCR text extraction: Vision (VNRecognizeTextRequest — fast, accurate, wide device support).
Structured parsing: Foundation Models (@Generable to turn raw OCR text into typed receipt/invoice data).
Fallback for older devices: Vision for OCR + simple regex parsing instead of Foundation Models.
Result: Works on every iPhone. Smart structured parsing is a premium on supported devices.

Fitness Coaching App

Workout classification: Core ML (custom model trained on exercise video data).
Personalized tips: Foundation Models (generates coaching text based on workout patterns).
All health data stays on-device: Privacy nutrition label: “No Data Collected.”
Result: Users trust the app with heart rate, body metrics, and workout history because nothing ever leaves their phone.
