Chapter 9

Voice & Audio Intelligence

SpeechAnalyzer, sound classification, voice-to-action pipelines, and building audio features that feel like magic

In Chapter 8 you taught your app to see and create images. Now it’s time to teach it to listen.

Voice and audio are the most natural interfaces humans have. We spoke for thousands of years before we typed. Yet most apps still treat voice as a second-class citizen — a glorified dictation button stuck in a corner. That’s a missed opportunity, and Apple’s new SpeechAnalyzer framework makes it easier than ever to build voice-first features that feel effortless.

What Changed in iOS 26

SpeechAnalyzer replaces the decade-old SFSpeechRecognizer. It’s modular, Swift Concurrency-native, and entirely on-device. It supports long-form audio, distant microphones, automatic language detection, and is 2.2× faster than Whisper Large V3 Turbo. It already powers Notes, Voice Memos, and Journal.

User Story: The Meeting That Summarized Itself

Sarah finishes a 30-minute project meeting. She opens the app, taps “Meeting Summary,” and within seconds sees: a 5-bullet recap, a list of action items with owners, and three key decisions. She shares it to Slack with one tap. The entire pipeline — transcription, understanding, extraction — ran on her iPhone. No cloud. No subscription. No waiting.

The SpeechAnalyzer Architecture

The old SFSpeechRecognizer was a single monolithic class that tried to do everything. SpeechAnalyzer takes the opposite approach: it’s modular. You compose the capabilities you need by adding modules to a session.

1

SpeechAnalyzer

The session manager. Create one, add modules, start analyzing.

→
2

Modules

SpeechTranscriber, DictationTranscriber, or SpeechDetector — pick what you need.

→
3

AsyncSequence

Streaming partial results via Swift Concurrency. No delegates, no callbacks.

Three Modules, Three Use Cases

ModuleWhat It DoesBest For
SpeechTranscriber Clean transcription optimized for commands and structured input Voice commands, search queries, form filling
DictationTranscriber Natural dictation with punctuation, paragraph breaks, formatting Note-taking, email composition, long-form text
SpeechDetector Detects voice activity without full transcription Push-to-talk, voice activation, silence detection

Tip: DictationTranscriber vs SpeechTranscriber

If the user is dictating (composing text), use DictationTranscriber — it adds punctuation and formatting automatically. If the user is commanding (telling the app what to do), use SpeechTranscriber — it gives you clean, parseable text without extra formatting.

API Recipe: Real-Time Transcription with SpeechAnalyzer

Goal: Transcribe speech in real time and display streaming text in SwiftUI.

Step 1: Configure the analyzer

import Speech

let analyzer = SpeechAnalyzer()
let transcriber = SpeechTranscriber()
try await analyzer.add(transcriber)

Step 2: Start analyzing from the microphone

// Analyze live audio from the default microphone
try await analyzer.start()

Step 3: Stream partial results

// Results arrive as an AsyncSequence
for await result in transcriber.results {
    let text = result.transcription.formattedString
    await MainActor.run {
        self.transcribedText = text
    }
}

Step 4: Stop when done

analyzer.stop()

Result: A live transcription that updates word-by-word as the user speaks. Entirely on-device. No network required.

Building a Transcription UI in SwiftUI

A bare text field that updates isn’t good enough. Users need visual feedback that tells them: the app is listening, speech is being processed, and the result is building in real time.

Product Pattern: The Voice Note Experience

The pattern: Tap to record → live waveform + streaming text → tap to stop → instant summary + action items.

Why it works: Users can see that the app is listening (waveform), see the transcription build word-by-word (streaming text), and immediately get value (summary). No waiting, no “processing” spinner at the end.

Where to use it: Voice memos, meeting notes, journal entries, interview recording, lecture capture, field notes.

Key insight: The streaming text is the loading indicator. If users see words appearing, they trust the system is working. A waveform alone isn’t enough — they need to see understanding happening.

Structured Extraction with @Generable

import FoundationModels

@Generable
struct CalendarAction {
    @Guide(description: "The event title")
    var title: String
    @Guide(description: "Person or people involved")
    var attendees: [String]
    @Guide(description: "Date in ISO 8601 format")
    var date: String
    @Guide(description: "Time in HH:mm format")
    var time: String
}

let session = LanguageModelSession()
let action: CalendarAction = try await session.respond(
    to: "Extract the calendar event: \(transcribedText)",
    generating: CalendarAction.self
)

Vision sees pixels. SpeechAnalyzer hears words. Foundation Models understands meaning. The pipeline is always the same: capture → transcribe → structure → act.

Warning: Always Confirm Before Acting

Never execute an action silently after voice input. Show the extracted data (“Lunch with Anna, tomorrow 12:00 PM”) and let the user confirm or edit. Voice recognition is good — but not perfect. A misheard “Anna” could become “Hannah” and schedule the wrong person.

Sound Classification: Hearing the World

SpeechAnalyzer understands human speech. But what about everything else? Dogs barking, doorbells ringing, applause, sirens, music? That’s where the Sound Analysis framework comes in.

SNClassifySoundRequest can identify 300+ sound categories in real time using Apple’s built-in Core ML model. No training, no model management, no cost.

The Audio Pipeline

Sound classification uses the same audio engine pattern across iOS:

import AVFoundation
import SoundAnalysis

let engine = AVAudioEngine()
let inputNode = engine.inputNode
let bus = 0
let format = inputNode.outputFormat(forBus: bus)

let analyzer = SNAudioStreamAnalyzer(format: format)
let request = try SNClassifySoundRequest(classifierIdentifier: .version1)
try analyzer.add(request, withObserver: self)

inputNode.installTap(onBus: bus, bufferSize: 8192,
                     format: format) { buffer, time in
    analyzer.analyze(buffer, atAudioFramePosition: time.sampleTime)
}

try engine.start()

Results arrive via the SNResultsObserving protocol:

func request(_ request: SNRequest,
             didProduce result: SNResult) {
    guard let classification = result as? SNClassificationResult,
          let top = classification.classifications.first,
          top.confidence > 0.7 else { return }
    // top.identifier = "dog_bark", "siren", "music", etc.
    print("\(top.identifier): \(top.confidence)")
}

  • Wildlife: Classify bird songs during nature walks

Don’t Build These

  • Surveillance: Continuous background recording of conversations
  • Profiling: Classifying users by their environment sounds
  • Always-on: Battery-draining 24/7 listening without clear purpose
  • Covert: Sound analysis without the user knowing it’s active
  • Creepy: “We noticed you’re in a restaurant” notifications

Building a Voice Assistant

The most ambitious voice feature is a complete voice interaction loop: the user speaks, the app understands, takes action, and responds. Here’s the full architecture:

1

SpeechTranscriber captures and transcribes speech

→
2

Understand

Foundation Models interprets intent with tool calling

→
3

Execute

Tool call triggers app action (search, create, navigate)

→
4

Respond

AVSpeechSynthesizer speaks the result back to the user

Voice Response with AVSpeechSynthesizer

import AVFoundation

func speak(_ text: String) {
    let utterance = AVSpeechUtterance(string: text)
    utterance.voice = AVSpeechSynthesisVoice(language: "en-US")
    utterance.rate = AVSpeechUtteranceDefaultSpeechRate
    utterance.pitchMultiplier = 1.0
    let synthesizer = AVSpeechSynthesizer()
    synthesizer.speak(utterance)
}

Tip: Don’t Overdo Voice Responses

Spoken responses work for short confirmations (“Event created for tomorrow at noon”) but annoy users for long content. Show long results on screen and only speak a brief summary. Nobody wants to listen to a 200-word meeting summary read aloud.

→

@Generable
struct MeetingSummary {
    @Guide(description: "3-5 bullet summary of the meeting")
    var keyPoints: [String]
    @Guide(description: "Action items with owner names")
    var actionItems: [ActionItem]
    @Guide(description: "Key decisions that were made")
    var decisions: [String]
    @Guide(description: "Suggested title for the meeting")
    var suggestedTitle: String
}

@Generable
struct ActionItem {
    var description: String
    var owner: String
}

Warning: Context Window Limits

Foundation Models has a ~4,096 token context window. A 30-minute meeting transcript can easily exceed this. Use the map-reduce pattern from Chapter 6: chunk the transcript into segments, summarize each, then summarize the summaries. Two good summaries merged beats one truncated summary.

Multilingual Voice Features

SpeechAnalyzer supports automatic language detection — no need to ask the user what language they speak. The framework figures it out from the audio.

For multilingual apps, you can chain SpeechAnalyzer with Foundation Models for a complete pipeline:

  1. Transcribe in the user’s language (automatic detection)
  2. Translate with Foundation Models if needed
  3. Extract structured data in a language-agnostic format (@Generable)
  4. Display results in the user’s preferred language

Tip: Test with Accented Speech

Real-world speech includes accents, code-switching (mixing languages mid-sentence), and background noise. Test with diverse audio samples, not just studio-quality recordings. Apple’s model handles accents well, but edge cases exist — especially with technical jargon and proper nouns.

Privacy: The Microphone Is Sacred

The microphone is the most sensitive permission on iOS. Users grant camera access without much thought. They hesitate with the microphone. Here’s how to earn their trust.

The Permission Dialog Matters

Good Permission Context

Bad Permission Context

“Allow microphone access to enable audio features.”

Battery and Performance

Continuous audio processing is a battery killer. Users will blame your app when their phone dies at 2 PM. Here’s how to be responsible.

Power-Efficient Audio Strategies

  • Use SpeechDetector first: Only run full transcription when speech is detected. SpeechDetector is much lighter than SpeechTranscriber.
  • Stop promptly: When the user pauses speaking, stop the analyzer. Don’t keep it running “just in case.”
  • Respect Low Power Mode: Check ProcessInfo.processInfo.isLowPowerModeEnabled and reduce audio quality or disable non-essential features.
  • Handle interruptions: Phone calls, alarms, and other apps interrupt the audio session. Register for AVAudioSession.interruptionNotification and pause gracefully.
  • Background audio: Only request audio background mode if your feature genuinely needs it (like a meeting recorder). App Store Review scrutinizes this.

Audio Session Configuration

Before any audio recording, configure the audio session properly. Getting this wrong causes silent failures that are maddening to debug.

let session = AVAudioSession.sharedInstance()
try session.setCategory(.playAndRecord,
                        mode: .default,
                        options: [.defaultToSpeaker,
                                  .allowBluetooth])
try session.setActive(true)

Tip: Category Matters

Use .playAndRecord if your app both listens and speaks (voice assistant). Use .record if it only listens (transcription). Using the wrong category causes audio routing issues — Bluetooth headphones stop working, speaker output disappears, or the volume drops to near-silent.
