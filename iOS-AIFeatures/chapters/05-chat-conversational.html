Chapter 5

In-App Chat & Conversational AI

SwiftUI chat UI, multi-turn sessions, tool calling, streaming responses — making your chatbot actually useful

Does Your Feature Need a Chatbot?

Before you build a chat interface, ask yourself: does this feature actually benefit from a conversation? A chatbot is a powerful interaction pattern — but it’s also a complex one. Users have to think of what to type, read the response, then decide what to type next. That’s a lot of cognitive work compared to tapping a button.

Chat Makes Sense

Customer support, product discovery, guided workflows, learning/tutoring, data exploration, trip planning — anything where the user’s needs unfold over multiple turns and aren’t fully known upfront.

The user needs to refine their request, the AI needs to ask clarifying questions, or the conversation itself IS the feature.

Chat Doesn’t Make Sense

Changing settings, choosing between 3 options, toggling a preference, performing a single action — anything where a button, picker, or form would be faster.

If you can describe the entire interaction in one sentence, it’s a button, not a conversation.

The restaurant test: If a waiter asked you “What would you like to drink?” and you had to type your answer instead of picking from the menu, you’d be annoyed. Don’t make your users type when they could tap.

The Anatomy of a Chat UI in SwiftUI

A chat interface looks simple — bubbles going up and down. But a production chat UI has a dozen subtle behaviors: it scrolls to the bottom when new messages arrive, stays put when the user scrolls up to read history, handles the keyboard smoothly, and shows a typing indicator while the AI thinks. Let’s build one.

Scenario: Priya is planning a trip to Kyoto using a travel app. She types “What’s the best time to visit Kyoto?” A gentle typing indicator pulses for a moment, then text streams in bubble-by-bubble: “Late March to mid-April for cherry blossoms, or November for autumn foliage…” She follows up: “What about restaurants near Fushimi Inari?” The AI remembers her context — she’s planning a Kyoto trip — and gives relevant, specific answers. She never had to repeat herself.

The Message Model

Start with a clean data model. Every message has a role, content, a timestamp, and a status (for tracking whether AI responses are still streaming):

import SwiftData

enum MessageRole: String, Codable {
    case user, assistant, system
}

enum MessageStatus: String, Codable {
    case sending, streaming, complete, failed
}

@Model
class ChatMessage {
    var id: UUID
    var role: MessageRole
    var content: String
    var timestamp: Date
    var status: MessageStatus

    init(role: MessageRole, content: String) {
        self.id = UUID()
        self.role = role
        self.content = content
        self.timestamp = .now
        self.status = role == .user ? .complete : .sending
    }
}

Using @Model from SwiftData means conversations persist automatically. Close the app, come back tomorrow, and the chat history is right there. Users expect this — nobody wants to lose a 20-message conversation because they switched to Safari.

The Chat View

The core layout is a ScrollView with a LazyVStack for message bubbles and a text input bar pinned to the bottom:

struct ChatView: View {
    @State private var viewModel = ChatViewModel()
    @State private var inputText = ""

    var body: some View {
        VStack(spacing: 0) {
            ScrollView {
                LazyVStack(spacing: 12) {
                    ForEach(viewModel.messages) { message in
                        MessageBubble(message: message)
                    }
                }
                .padding()
            }
            .defaultScrollAnchor(.bottom)

            // Input bar
            HStack(spacing: 12) {
                TextField("Ask anything...", text: $inputText)
                    .textFieldStyle(.roundedBorder)
                Button { sendMessage() } label: {
                    Image(systemName: "arrow.up.circle.fill")
                        .font(.title2)
                }
                .disabled(inputText.isEmpty || viewModel.isGenerating)
            }
            .padding()
        }
    }
}

The key trick is .defaultScrollAnchor(.bottom) — this tells SwiftUI to anchor the scroll view at the bottom, so new messages appear without you manually calling scrollTo(). It’s the difference between a chat that feels native and one that feels like a homework project.

Keyboard avoidance is free in SwiftUI. The TextField inside a VStack automatically pushes up when the keyboard appears. Add .scrollDismissesKeyboard(.immediately) on the ScrollView so users can swipe down to dismiss the keyboard while scrolling through chat history.

Multi-Turn Sessions with Foundation Models

Here’s the beautiful part: LanguageModelSession is already built for conversations. It maintains a transcript of all exchanges — every prompt and every response — so the model automatically has context from earlier turns. You don’t need to resend the entire chat history with each message.

let session = LanguageModelSession(
    instructions: "You are a helpful travel assistant. Answer travel questions only."
)

// Turn 1
let r1 = try await session.respond(to: "Best time to visit Kyoto?")
// "Late March to mid-April for cherry blossoms..."

// Turn 2 — the model remembers we're talking about Kyoto
let r2 = try await session.respond(to: "What about restaurants near the shrine?")
// "Near Fushimi Inari, you'll find great street food stalls..."

The session remembers “Kyoto” and “the shrine” resolves to Fushimi Inari from the earlier context. No prompt engineering tricks needed — the framework’s built-in transcript handles it.

Context window: ~4,096 tokens. The on-device model has a limited context window. For short conversations (10–15 turns), this is fine. For very long conversations, older messages may fall out of context. Design your UX so conversations are focused, not infinite. If you need longer conversations, route to a cloud API with a larger context window.

Streaming Responses

Nobody wants to stare at a spinner for 3 seconds and then see a wall of text. Streaming makes the AI feel responsive and thoughtful — like it’s composing a reply in real time.

The Streaming Chat Pattern

  1. User sends message — Immediately add the user’s bubble to the chat. Don’t wait for the AI to respond before showing the user’s own message.
  2. Show typing indicator — A pulsing dot animation in the AI’s bubble area. This buys you ~500ms of perceived responsiveness.
  3. Start streaming — Replace the typing indicator with real text as soon as the first token arrives. Stream word-by-word into the AI bubble.
  4. Complete — When streaming finishes, update the message status to .complete. Show action buttons (copy, share) if appropriate.

This is how ChatGPT, Apple’s Writing Tools, and every polished chat app works. Users trust text they watch being written more than text that appears all at once.

Recipe: Streaming Chat with Foundation Models

Goal: Stream the AI’s response into a chat bubble, token by token.

Step 1: Create the ViewModel with streaming support:

@Observable
class ChatViewModel {
    var messages: [ChatMessage] = []
    var isGenerating = false
    private var session = LanguageModelSession(
        instructions: "You are a helpful assistant. Be concise."
    )

    func send(_ text: String) async {
        let userMsg = ChatMessage(role: .user, content: text)
        messages.append(userMsg)

        let aiMsg = ChatMessage(role: .assistant, content: "")
        aiMsg.status = .streaming
        messages.append(aiMsg)
        isGenerating = true

        do {
            let stream = session.streamResponse(to: text)
            for try await partial in stream {
                aiMsg.content = partial.content
            }
            aiMsg.status = .complete
        } catch {
            aiMsg.content = "I couldn't process that. Try rephrasing?"
            aiMsg.status = .failed
        }
        isGenerating = false
    }
}

Step 2: Wire it up to the view. The MessageBubble view observes the ChatMessage model, so as aiMsg.content updates, the bubble redraws with the new text — giving you the streaming effect automatically.

Step 3: Add a typing indicator. Show it when isGenerating is true and the AI message content is still empty:

if viewModel.isGenerating,
   let last = viewModel.messages.last,
   last.content.isEmpty {
    TypingIndicator()  // Pulsing dots animation
}

Tool Calling: When Chat Meets Your App

This is where chatbots go from “cute demo” to “genuinely useful.” Tool calling lets the model invoke your app’s actual functionality mid-conversation. The user says “Show me Italian restaurants nearby,” and the model calls your search function, gets real results, and weaves them into its response.

In Foundation Models, tools conform to the Tool protocol. You provide a name, description, and an implementation. The model decides when to call tools based on the conversation context:

import FoundationModels

struct SearchRestaurantsTool: Tool {
    let name = "search_restaurants"
    let description = "Search for nearby restaurants by cuisine type"

    @Generable
    struct Arguments {
        @Guide(description: "The type of cuisine to search for")
        var cuisine: String
        @Guide(description: "Maximum number of results")
        var limit: Int
    }

    func call(arguments: Arguments) async throws -> String {
        let results = await RestaurantService.search(
            cuisine: arguments.cuisine, limit: arguments.limit
        )
        return results.map { "\($0.name) — \($0.rating)⭐" }
            .joined(separator: "\n")
    }
}

Notice how the Arguments struct uses @Generable — the same constrained decoding from Chapter 4. The model can’t produce invalid tool arguments because they’re typed. Register tools when creating the session:

let session = LanguageModelSession(
    instructions: "You are a restaurant finder. Use the search tool to find restaurants.",
    tools: [SearchRestaurantsTool()]
)

When the user asks “Find Italian places near me,” the model autonomously decides to call search_restaurants with cuisine: "Italian", gets back real restaurant data, and composes a natural response around it. The framework executes the tool call automatically and feeds the result back into the conversation.

Why Tool Calling Matters

Without tools, a chatbot can only generate text based on its training data. With tools, it can check your calendar, search your database, fetch live prices, create reminders, toggle app settings — anything you can write a function for. The model becomes an interface to your app, not just a text generator.

System Prompts: Keeping Your Bot on Track

The instructions parameter is your chatbot’s constitution. It defines personality, boundaries, and response format. Get it right, and the bot stays focused. Get it wrong, and users will quickly find ways to make it say things you don’t want.

Good: Focused Instructions

You are a meal planning assistant for the FreshPlate app. Help users plan weekly meals, suggest recipes from our database, and create shopping lists. Only answer questions about food, cooking, and nutrition. If asked about other topics, politely redirect to meal planning.

Clear role, clear boundaries, clear behavior for off-topic requests.

Bad: Vague Instructions

You are a helpful assistant. Be nice and answer questions.

No boundaries, no personality, no guardrails. The bot will happily answer questions about politics, help write emails, and generally do everything poorly instead of one thing well.

Instruction Template

Here’s a template that works well for most in-app chatbots:

let instructions = """
    Role: You are [specific role] for [app name].
    Scope: Help users with [specific tasks]. Do not answer questions about [out-of-scope topics].
    Tone: [Friendly/professional/casual]. Keep responses under [N] sentences.
    Tools: When the user asks about [trigger], use the [tool name] tool.
    Boundaries: If you don't know something, say so. Never make up [data type].
    """

Rich Messages Beyond Text

A great chat interface isn’t just text bubbles. The AI can respond with cards, images, action buttons, and interactive elements. Model your messages to support multiple content types:

enum MessageContent {
    case text(String)
    case restaurantCard(Restaurant)
    case actionButtons([ActionButton])
    case image(URL)

    struct ActionButton: Identifiable {
        let id = UUID()
        let title: String
        let action: () -> Void
    }
}

When the search tool returns restaurants, instead of showing plain text, render each result as a card with the name, rating, photo, and a “Get Directions” button. The chat becomes an interactive workflow, not just a Q&A.

Don’t overdo rich content. Cards and buttons are great for structured data (search results, product recommendations). But most conversational responses should still be plain text. A chat where every message is a card feels like a cluttered dashboard, not a conversation.

On-Device vs Cloud Chat

Foundation Models gives you free, private, offline chat. But the ~3B on-device model has limitations: it handles simple conversations well but struggles with complex multi-step reasoning, long context, or specialized knowledge domains. Here’s how to decide:

Factor On-Device (Foundation Models) Cloud (OpenAI, Claude, Gemini)
Cost $0 forever $0.15–$15 per 1M tokens
Privacy Data never leaves device Data sent to third-party servers
Context window ~4,096 tokens 128K–1M+ tokens
Reasoning Good for simple tasks Excellent for complex reasoning
Offline Works anywhere Requires internet
Devices A17 Pro+ / M1+ Any device with internet

The Hybrid Chat Architecture

The best approach is a router that sends simple queries on-device and complex ones to the cloud:

func routeMessage(_ text: String) async throws -> AsyncStream<String> {
    let complexity = estimateComplexity(text)

    if complexity == .simple, SystemLanguageModel.default.isAvailable {
        // Free, private, instant
        return streamFromDevice(text)
    } else if hasCloudQuota() {
        // Powerful but costs money
        return streamFromCloud(text)
    } else {
        // Fallback: simpler on-device response
        return streamFromDevice(simplifyPrompt(text))
    }
}

Tell users when data leaves the device. If a message gets routed to a cloud API, show a subtle indicator: a small cloud icon on the message bubble, or a one-time notice: “This response uses our cloud AI for a more detailed answer.” Transparency builds trust — secrecy destroys it.

Handling Edge Cases

Chat interfaces have more edge cases than any other UI pattern. Here’s your checklist:

  • Empty conversation — Show a welcome message or suggested starter questions. Never show an empty screen with just a text field. Starter suggestions like “Plan a weekend trip,” “Find dinner ideas,” or “Help me decide” lower the barrier to first message.
  • Very long conversations — The ~4,096 token context window means older messages eventually fall out. For most use cases, this is fine — conversations are naturally focused. If you need longer memory, save key facts from earlier turns and inject them into the instructions.
  • Rapid-fire messages — Users sometimes send multiple messages before the AI responds. Queue them and process sequentially, or show a “Please wait for the current response” indicator.
  • Network errors (cloud mode) — If a cloud response fails mid-stream, show the partial response with an “AI response interrupted. Tap to retry.” badge. Don’t throw away what you already streamed.
  • Offensive input — Foundation Models has built-in safety filtering. For cloud APIs, implement your own content filter. When triggered, respond with: “I can’t help with that. Let me know if there’s something else I can assist with.”
  • Device not supported — If Foundation Models isn’t available (older iPhone), either disable the chat feature with a clear explanation, or fall back to cloud-only mode.

Persisting Conversations with SwiftData

Users expect chat history to survive app restarts. With SwiftData, this is almost free — our ChatMessage model already has the @Model macro. Group messages into conversations:

@Model
class Conversation {
    var id: UUID
    var title: String  // AI-generated from first message
    var messages: [ChatMessage]
    var createdAt: Date
    var updatedAt: Date

    init(title: String = "New Chat") {
        self.id = UUID()
        self.title = title
        self.messages = []
        self.createdAt = .now
        self.updatedAt = .now
    }
}

You can even restore the LanguageModelSession from a saved transcript, so the model remembers the conversation context even after an app restart. Load the previous transcript and pass it when creating a new session.

Rate Limiting and Cost Control

On-device inference is free, so rate limiting isn’t about cost — it’s about battery and responsiveness. But if you use cloud APIs as a fallback, cost matters:

50 Cloud messages/day per user — a reasonable default limit
10 Messages/minute rate limit to prevent abuse
$0 On-device messages cost. No limits needed.

When a user hits the cloud limit, don’t just show an error. Fall back gracefully: “Switching to on-device AI for faster responses.” Frame the limit as a benefit (faster, more private), not a restriction. For premium apps, offer unlimited cloud access as a subscription tier.

Message Bubble UX Details

The details matter. Here are the small touches that separate a polished chat from a prototype:

  • Distinct bubble styles — User messages on the right (app accent color), AI messages on the left (neutral gray/system background). This is table stakes — Messages.app has trained every iPhone user to expect this layout.
  • Timestamps — Show timestamps between message groups, not on every message. “Today 2:34 PM” between a cluster of messages is enough.
  • Copy/share — Long-press on an AI response to copy text or share. Use .contextMenu or the iOS 17+ .contentMenuPreview.
  • Haptic feedback — A subtle UIImpactFeedbackGenerator(.light) when the AI starts responding. Tiny detail, premium feel.
  • Accessibility — Each bubble should be a VoiceOver group. Announce new AI messages: AccessibilityNotification.Announcement("New response from assistant"). Support Dynamic Type in all bubble text.

Steal from Messages.app. Apple’s iMessage is the best chat UI on the planet — it’s what your users compare everything to. Match its scroll behavior, bubble sizing, keyboard interaction, and animation timing. Don’t try to reinvent chat chrome.

Conversation Starters and Onboarding

An empty chat screen with just a text field is intimidating. The user thinks: “What am I supposed to type?” Always provide conversation starters:

struct ConversationStartersView: View {
    let starters = [
        "Plan a weekend trip to...",
        "Suggest dinner recipes for tonight",
        "Help me write a thank-you note",
        "What's the weather like this week?"
    ]

    var body: some View {
        VStack(spacing: 8) {
            Text("How can I help?")
                .font(.headline)
            ForEach(starters, id: \.self) { starter in
                Button(starter) { onTap(starter) }
                    .buttonStyle(.bordered)
            }
        }
    }
}

Show these only when the conversation is empty. As soon as the first message is sent, they disappear and the full chat takes over. This pattern is directly from ChatGPT, Gemini, and Claude — it works because it eliminates the blank page problem.

When NOT to Stream

Streaming is great for long-form responses. But for short, action-oriented answers, it can feel sluggish. If the AI is responding with “Done! Your alarm is set for 7am,” don’t stream that word by word — just show it. Use response length as a heuristic:

Rule of thumb: If the response is under ~30 words, show it all at once with a brief fade-in animation. If it’s over ~30 words, stream it. This keeps short confirmations snappy while long explanations feel thoughtful.
