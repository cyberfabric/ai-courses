Smart Search & Content Generation

Semantic search with NLEmbedding, AI-powered autocomplete, content drafting, and template-based generation

Why Keyword Search Fails

Your user opens your recipe app, types “something spicy with chicken,” and gets… nothing. Zero results. Because no recipe in your database is literally called “something spicy with chicken.” There’s a “Thai Basil Chicken” and a “Sriracha Lime Wings” — both perfect matches — but keyword search only finds exact string matches.

This is the gap between what users mean and what they type. Semantic search closes that gap by understanding intent, not just characters.

Scenario: Leo is decluttering his notes app. He has 200+ notes accumulated over two years. He vaguely remembers writing something about “that blue jacket from the trip.” Keyword search finds nothing — the note is titled “Copenhagen Packing List” and mentions a “navy parka.” But with semantic search, “blue jacket from the trip” matches “navy parka” + “Copenhagen” context because the embeddings understand that “blue jacket” and “navy parka” are semantically similar. Leo finds his note in 2 seconds.

Text Embeddings on iOS

An embedding turns text into a list of numbers — a 512-dimensional vector that captures the meaning of the text. Similar meanings produce similar vectors. “Happy” and “joyful” are close together. “Happy” and “socket wrench” are far apart. This is the foundation of semantic search.

Apple provides NLEmbedding in the Natural Language framework, available since iOS 13 and running on any iPhone 8 or later. It supports word-level and sentence-level embeddings in seven languages: English, Spanish, French, Italian, German, Portuguese, and Simplified Chinese.

Why Embeddings Matter for Your App

Embeddings let you search by meaning instead of matching characters. They’re the building block for “find similar items,” “smart suggestions,” and “related content” features — all running on-device, for free, with no internet required.

Building a Local Semantic Search

Here’s the big picture: you embed all your app’s content once (at import or creation time), store those vectors, and then at search time, embed the user’s query and find the closest matches.

Recipe: Semantic Search with NLEmbedding

Goal: Search 100 recipes by meaning, not just keywords. “Something spicy with chicken” should find “Thai Basil Chicken.”

Step 1: Get the sentence embedding model:

import NaturalLanguage

guard let embedding = NLEmbedding.sentenceEmbedding(for: .english) else {
    // Embedding not available for this language
    return
}

Step 2: Embed your content at import time and store the vectors:

struct RecipeIndex {
    let recipe: Recipe
    let vector: [Double]  // 512 dimensions
}

func buildIndex(recipes: [Recipe]) -> [RecipeIndex] {
    guard let embedding = NLEmbedding.sentenceEmbedding(for: .english) else { return [] }

    return recipes.compactMap { recipe in
        let text = "\(recipe.title) \(recipe.ingredients.joined(separator: " "))"
        guard let vector = embedding.vector(for: text) else { return nil }
        return RecipeIndex(recipe: recipe, vector: vector)
    }
}

Step 3: At search time, embed the query and find nearest neighbors:

func search(query: String, in index: [RecipeIndex]) -> [Recipe] {
    guard let embedding = NLEmbedding.sentenceEmbedding(for: .english),
          let queryVector = embedding.vector(for: query) else { return [] }

    return index
        .map { (item: $0, distance: cosineSimilarity(queryVector, $0.vector)) }
        .sorted { $0.distance > $1.distance }  // Higher similarity first
        .prefix(10)
        .map(\.item.recipe)
}

Step 4: Compute cosine similarity between two vectors:

func cosineSimilarity(_ a: [Double], _ b: [Double]) -> Double {
    let dot = zip(a, b).map(*).reduce(0, +)
    let magA = sqrt(a.map { $0 * $0 }.reduce(0, +))
    let magB = sqrt(b.map { $0 * $0 }.reduce(0, +))
    guard magA > 0, magB > 0 else { return 0 }
    return dot / (magA * magB)
}

Now “something spicy with chicken” returns Thai Basil Chicken, Sriracha Lime Wings, and Chicken Tikka Masala — even though none of those titles contain the words “spicy” or “something.”

Embed once, search many times. Building the index takes ~1ms per item. Searching is even faster. Embed content when it’s created or imported, store the vectors in SwiftData or a local database, and reuse them for every search. Don’t re-embed on every keystroke.

Cloud-Powered Semantic Search

NLEmbedding is great for modest datasets (hundreds to low thousands of items) on-device. But if you need:

  • More languages — NLEmbedding supports 7 languages. Cloud models cover 100+.
  • Higher quality — Cloud embedding models (OpenAI text-embedding-3-small, Google text-embedding-005) produce more nuanced vectors.
  • Large datasets — Millions of items need a vector database (Pinecone, Qdrant) on your server.

The architecture: embed content on your server at ingestion time, store vectors in a vector database, and query from the app via your backend API. The app sends the search query; the server embeds it, searches the vector DB, and returns results.

Cloud search means the query leaves the device. If your app searches personal content (notes, photos, health data), keep it on-device with NLEmbedding. Cloud semantic search is for catalog content — products, articles, recipes — where the data is already on your server anyway.

Autocomplete That Predicts Intent

Great autocomplete doesn’t just match prefixes — it predicts what the user is looking for. The trick is a two-tier approach:

The Two-Tier Autocomplete Pattern

  1. Tier 1: Prefix Match (instant) — As the user types “chi,” immediately show items starting with “chi”: “Chicken Tikka,” “Chimichurri Steak.” This is free, instant, and works on any device. Use localizedStandardContains for efficient prefix matching.
  2. Tier 2: Semantic Suggestions (200ms delay) — After a 200ms debounce, run an NLEmbedding search for the full typed text. “Chi” might surface “Kung Pao Tofu” (Chinese food, semantically related). Show these below the prefix matches, labeled “Related.”

The prefix matches give immediate feedback (the app feels responsive), while the semantic suggestions add intelligence (the app feels smart). Together they create the best of both worlds.

Search UI in SwiftUI

SwiftUI’s .searchable modifier gives you a search bar with suggestions out of the box. Pair it with your semantic search engine:

struct RecipeSearchView: View {
    @State private var searchText = ""
    @State private var results: [Recipe] = []
    @State private var suggestions: [String] = []

    var body: some View {
        NavigationStack {
            List(results) { recipe in
                RecipeRow(recipe: recipe)
            }
            .searchable(text: $searchText) {
                ForEach(suggestions, id: \.self) { suggestion in
                    Text(suggestion)
                        .searchCompletion(suggestion)
                }
            }
            .onChange(of: searchText) { _, query in
                performSearch(query)
            }
            .overlay {
                if results.isEmpty && !searchText.isEmpty {
                    ContentUnavailableView.search(text: searchText)
                }
            }
        }
    }
}

New in iOS 26: the searchToolbarBehavior() modifier lets you minimize the search bar into a compact toolbar button when the user isn’t actively searching. And you can create a dedicated search tab with the .search role — useful for apps where search is a primary navigation pattern.

Never show an empty search result without help. Use ContentUnavailableView.search for the built-in “No Results” state. Even better: suggest alternative queries. “No results for ‘blue pasta.’ Try: ‘pesto,’ ‘seafood,’ or ‘Mediterranean.’”

Content Generation with Foundation Models

Search helps users find things. Content generation helps them create things. The pattern is always the same: the AI produces a draft, the user edits it, and the result is theirs. Never present AI-generated content as final — it’s always a starting point.

The AI Draft Pattern

AI creates the first draft. The user refines it. This works for emails, product descriptions, social posts, review responses, meeting notes, and any text the user needs to produce. The AI eliminates the blank page; the user adds their voice.

Drafting with @Generable Templates

For consistent, structured content, combine Foundation Models with @Generable templates. This gives you typed fields that constrain the AI’s output:

@Generable
struct ProductDescription {
    @Guide(description: "A catchy headline, max 10 words")
    var headline: String

    @Guide(description: "A compelling 2-3 sentence description")
    var body: String

    @Guide(description: "Three key selling points as bullet items")
    var highlights: [String]
}

let session = LanguageModelSession(
    instructions: "You write concise, engaging product descriptions for an online store."
)
let description = try await session.respond(
    to: "Describe this product: \(product.name), category: \(product.category)",
    generating: ProductDescription.self
)

The template guarantees structure: you always get a headline, body, and exactly three highlights. No parsing, no hoping the model formatted it correctly. The view can bind directly to the struct’s fields.

Multi-Format Generation

Let users choose the output format for the same content:

@Generable
enum OutputFormat: String {
    case brief      // 1-2 sentences
    case detailed   // Full paragraph
    case bullets    // Bullet points
    case social     // Social media post style
}

@Generable
struct GeneratedContent {
    @Guide(description: "The content in the requested format")
    var text: String

    @Guide(description: "Suggested title for this content")
    var suggestedTitle: String
}

A single “Generate Description” button with a format picker gives users control over the AI’s output without needing to learn prompt engineering.

Token Limits and Chunking

The on-device model has a ~4,096 token context window. If your source content is longer — a 5,000-word article, a lengthy document, a full email thread — it won’t fit in a single prompt. The solution is chunking with a map-reduce pattern:

Split into chunks
→
Summarize each
→
Combine summaries
→
Final summary
func summarizeLongDocument(_ text: String) async throws -> String {
    let chunks = text.splitIntoChunks(maxTokens: 2000)

    // Map: summarize each chunk
    var chunkSummaries: [String] = []
    for chunk in chunks {
        let session = LanguageModelSession()
        let summary = try await session.respond(
            to: "Summarize this section in 2-3 sentences: \(chunk)"
        )
        chunkSummaries.append(summary.content)
    }

    // Reduce: summarize the summaries
    let combined = chunkSummaries.joined(separator: "\n\n")
    let finalSession = LanguageModelSession()
    let final = try await finalSession.respond(
        to: "Create a cohesive summary from these section summaries: \(combined)"
    )
    return final.content
}

Check token count before sending. Use a rough estimate of ~4 characters per token for English. A 16,000-character text is ~4,000 tokens and fits in one prompt. Anything longer needs chunking. When in doubt, chunk — two good summaries merged beats one truncated summary.

Personalized Search

Search gets smarter when it learns from the user. Track what the user searches for, what they tap, and how long they spend on results. Over time, use this to boost relevant results:

struct SearchPreference {
    var categoryBoosts: [String: Double]  // "Italian": 1.5, "Dessert": 0.8
    var recentSearches: [String]
    var favoriteItemIDs: Set<String>
}

func personalizedSearch(query: String, preferences: SearchPreference) -> [Recipe] {
    let baseResults = semanticSearch(query)

    return baseResults
        .map { recipe in
            var score = recipe.similarityScore
            // Boost categories the user prefers
            if let boost = preferences.categoryBoosts[recipe.category] {
                score *= boost
            }
            // Boost favorited items
            if preferences.favoriteItemIDs.contains(recipe.id) {
                score *= 1.3
            }
            return (recipe, score)
        }
        .sorted { $0.1 > $1.1 }
        .map(\.0)
}

All of this runs on-device. No user data leaves the phone. Over time, the search results adapt to each user’s tastes without anyone else ever seeing their search history.

Don’t over-personalize. If the user always searches for Italian food, they might still want to discover Thai recipes. Keep a balance: boost preferred categories, but don’t completely bury everything else. A “Discover Something New” section can help avoid the filter bubble.

Combining Search + Generation

The most powerful pattern combines semantic search with content generation. The user searches → finds something → the AI helps them act on it:

Search + Generate: The Killer Combo

User searches “quick dinner for two.” Semantic search returns Thai Basil Chicken (25 min, serves 2). User taps it. Below the recipe, a “Generate Shopping List” button uses Foundation Models to extract ingredients, check what’s likely already in their pantry (salt, oil, rice), and produce a clean shopping list.

Search finds the content. AI makes it actionable. Neither is as useful alone.

Generation Without Search: Hallucination Risk

User asks “Write me a recipe for Thai Basil Chicken.” The model makes one up. Ingredient amounts may be wrong. Cooking times may be unsafe. No real recipe was consulted — it’s pure hallucination dressed up as expertise.

Always ground generation in real data. Search first, generate from results.

Search Performance Tips

  • Debounce search input — Don’t run a semantic search on every keystroke. Wait 200–300ms after the user stops typing. Prefix matching can run immediately (it’s cheap).
  • Pre-compute embeddings — Embed content at creation or import time, not at search time. Store vectors alongside your data model in SwiftData.
  • Use Accelerate for vector math — Apple’s Accelerate framework has SIMD-optimized dot product and norm functions. For 100+ items, this matters.
  • Index in the background — When new content arrives, embed it on a background queue. Use Swift concurrency with Task.detached(priority: .utility).
  • Cache search results — If the user searches the same query twice, return cached results instantly. Invalidate when content changes.
  • Spotlight integration — Index your content with CSSearchableItem so it appears in system-wide Spotlight search. Combine with your in-app semantic search for maximum discoverability.
