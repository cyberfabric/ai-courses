Chapter 5

Batch APIs: Half Price for Patient Workloads

Memory Map
📦

The Big Idea

Batch APIs are the overnight shipping of AI: submit a JSONL file, get answers within 24 hours, pay exactly half. Same model, same quality — just delayed. Every major provider offers it. The right question isn't "why batch?" — it's "why am I paying real-time for this?"

🧩 Core Concepts

✈️
Half Price for Patience
Flat 50% off input AND output, every major provider, April 2026. No secret "cheaper batch" hiding — they've all converged.
🕐
24-Hour SLA
Guaranteed window on Anthropic and OpenAI; "up to 24h, no SLA" on Gemini. Most real jobs finish in 1–4 hours.
📄
JSONL In, JSONL Out
Same file format everywhere. Each row is a request + custom_id. Cross-provider migrations are nearly plug-and-play.
🔗
Stacking Matters
Batch + caching = 5% of list on Anthropic / OpenAI. On Gemini, the two don't stack — you get the better of the two per request.
🎯
No Quality Difference
Same model, same output. The only variable is latency. Submit your evals, backfills, and reports without worry.
🏷️
Always Tag Your Batches
Without feature/env metadata, batch spend hits your bill as one lumpy line — impossible to attribute.

💰 Stacking — The Classifier Example

  1. Real-time, no cache

    100K docs × 15.5K tokens at $3/M = $4,665/day. The baseline.

  2. + Prefix caching

    15K-token system prompt cached. $615/day (87% off).

  3. + Batch only

    No caching, but 50% off. $2,333/day (50% off).

  4. Batch + caching (both)

    Prefix cached AND batch discount stacks. $308/day — 93% off list.

⚠️ Operational Gotchas

⚠️
Partial failures are normal
A 100K-row batch with 150 errors is expected. Parse the error file, always have a retry path.
🔀
Output order is random
Output JSONL is in completion order, not input order. Match by custom_id, never index.
🚫
No streaming
Batch doesn't support SSE. If you rely on streaming tokens, batch isn't viable for that feature.
🔁
No native idempotency
Crash mid-submit = possible duplicate batch. Hash the payload yourself before calling the API.

📖 Key Terms

Batch API
Asynchronous processing endpoint: submit a file of prompts, get results within ~24h, pay 50% of list.
JSONL
JSON Lines format — one JSON object per line. The universal batch input format across providers.
custom_id
A stable identifier you attach to each batch row so you can reconcile results back to your source data.
SLA
Service-level agreement: the worst-case time the provider commits to. Anthropic/OpenAI: 24h. Gemini: none.
Error file
The companion JSONL output that lists which rows failed and why. Parse it every time.
Stacking
Combining multiple discounts (batch + caching) multiplicatively. Works on Anthropic/OpenAI, not on Gemini.
Fallback path
The real-time execution route used when a batch urgently needs to be re-run outside its 24h window.
Reconciliation
Matching output rows back to source records by custom_id — idempotently, so replays don't double-write.

🔍 When Batch Really Shines

🧪
Eval Suites
10K-prompt regression runs per deploy. No user waits, evals can take an hour. Classic batch territory.
📚
Embedding Backfills
Re-embedding a new corpus or migrating embedding models. Millions of chunks, all background.
📰
Daily Reports
End-of-day summaries, digests, trend analyses. Submit at 10pm, arrive in inboxes at 6am.
🎨
Synthetic Data
Generating training examples for fine-tuning. Batch + caching is a 95%-off superpower here.

❓ Quick Quiz

What's the flat batch discount across all three major providers?

50% off both input and output — industry-standard in April 2026.

Why does batch + caching matter more on Anthropic/OpenAI than on Gemini?

On Anthropic and OpenAI, the two discounts stack multiplicatively — cached input in batch ≈ 5% of list. On Gemini, they don't stack; you only get the better of the two per request.

What's the universal batch file format?

JSONL — one JSON request per line, each with a custom_id so you can reconcile results.

Why should you never rely on output row ordering?

Batch results come back in completion order, not input order. Always match by custom_id.

What's the most reliable test of "is this a batch candidate?"

Is a user waiting on the HTTP response right now? If yes, real-time. If no, batch — almost always.

🔄 The Real-Time Fallback Playbook

  1. Parameterize the path

    Same business logic flips between batch and real-time via one config flag — not a rewrite.

  2. Chunk the fallback

    100K requests will shred your rate limit. Split into 500-request chunks with exponential backoff.

  3. Cost the fallback

    Real-time re-run costs 2x (no batch discount), sometimes 20x if it busts your cache too.

  4. Page a human if it recurs

    Multiple batches bouncing to real-time in a week = the batch SLA is broken; escalate to provider support.

Key Takeaways

  • Batch APIs are a flat 50% off on all three major providers (April 2026) — no hunting for cheaper.
  • Trade-off: up to 24h latency for 2x cheaper calls. Perfect for evals, reports, backfills, synthetic data.
  • Batch + caching stacks on Anthropic / OpenAI (≈ 5% of list); not on Gemini (best-of-one).
  • Same JSONL format everywhere. Cross-provider migrations are plug-and-play.
  • Always reconcile by custom_id. Always parse the error file. Always tag batches for cost attribution.
  • Default question flips: not "why batch?" but "why am I paying real-time for this?"
  • Design the fallback path early — sometimes a batch has to be re-run now, and real-time re-runs cost 2–20x.
