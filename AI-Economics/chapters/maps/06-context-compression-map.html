Chapter 6

Context Compression: Spend Less by Sending Less

Memory Map
🧳

The Big Idea

Most production prompts are overpacked suitcases — 40–80% of tokens contribute almost nothing to the answer. Token pruning, hierarchical summarization, and semantic chunking let you carry the same information in half the bag. Pure savings on the input side, if you measure quality before you ship.

🧩 The Five Sources of Dead Weight

📜
Verbose instructions
"Please respond politely and thoroughly…" when 6 tokens do the work of 45.
📚
Uncompressed RAG chunks
10 chunks × 500 tokens, half the signal overlapping. Re-rank and compress.
💬
Full chat history
40 turns × 500 tokens ships on every call. A 1K living summary replaces 30K.
📝
Over-long few-shots
Five 5-line examples when two 2-line examples already teach the format.
📖
Verbatim policies
20K-token ToS quoted whole when three clauses are all that matter.
🪶
Politeness padding
"For example," "In other words," "It is important to note" — almost always deletable.

🗺️ Where Compression Fits — and Where It Doesn't

Information-level tasks compress freely. Exact-byte tasks refuse the trade — leave them uncompressed.

📖 Key Terms

Token pruning
Dropping individual tokens the compressor judges low-information — per-token keep/drop decisions.
Hierarchical summarization
Summarize chunks, then summarize those summaries — repeat until you fit in context.
Living summary
A running summary of a chat, re-generated every N turns, replacing older raw history.
Semantic chunking
Splitting documents at natural topic boundaries (embedding similarity drops) instead of fixed token counts.
Re-ranker
A second-pass model (often cross-encoder) that re-scores retrieved chunks and keeps only the top K.
DTCRS
Dynamic Tree Construction for Recursive Summarization — builds a query-specific summary tree per question.
Compression ratio
Fraction of tokens kept after compression — 0.3 means keep 30%, drop 70%.
The elbow
The compression ratio where quality starts dropping sharply. Ship one step conservative of the elbow.

📐 Measuring Before You Ship

  1. Build an eval set

    50–200 representative inputs with known-good outputs. Non-negotiable.

  2. Baseline at ratio 1.0

    Measure exact-match / F1 / rubric / human preference at no compression.

  3. Sweep 0.8 → 0.2

    Quality-vs-ratio plot reveals the elbow where quality collapses.

  4. Deploy one step conservative

    If elbow is 0.3, ship at 0.4. Leave headroom for real-world drift.

  5. Monitor in production

    Log a sample of raw-vs-compressed pairs; alert when a cheap judge model rates the compressed answer worse >10% of the time.

❓ Quick Quiz

What compression ratio and accuracy retention does LLMLingua-2 claim?

Up to 20x compression with 95–98% accuracy retention on non-structured tasks. 3–6x faster than LLMLingua v1.

When should you use hierarchical summarization instead of token pruning?

When the raw source is so long (100K+ tokens) that pruning alone can't fit it in context. Pruning removes filler; hierarchical summarization restructures the whole input.

Why do living summaries keep chat costs bounded?

Every N turns, older raw history is replaced by a compact summary. The prompt size stops growing linearly with conversation length — it plateaus at summary + last N turns.

What kinds of tasks should NOT be compressed?

Exact-byte tasks: legal review, exact-quote extraction, numeric/tabular data, code syntax review, schema-following outputs. When fidelity > meaning, don't compress.

What's "the elbow" in compression tuning?

The compression ratio on a quality-vs-ratio plot where quality drops sharply. Ship one step more conservative than the elbow to leave headroom.

Key Takeaways

  • 40–80% of production prompt tokens are dead weight — identify where yours hides.
  • LLMLingua-2 prunes individual tokens task-agnostically — 20x compression possible on prose.
  • Hierarchical summarization handles 100K+ token sources that pruning alone can't compress.
  • Living summaries keep chat context bounded forever regardless of turn count.
  • Semantic chunking beats fixed-size chunking and improves retrieval quality while saving tokens.
  • Never compress exact-byte tasks (legal, quotes, code syntax, tables).
  • Measure quality with an eval set before and after. Find the elbow, ship conservative.
  • Compression stacks with caching, batch, and routing — use all four for compounding savings.
