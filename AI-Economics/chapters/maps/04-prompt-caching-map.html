Chapter 4

Prompt Caching: The 90% Discount You're Not Using

Memory Map
🅿️

The Big Idea

Prompt caching is the parking-meter loyalty deal of AI: pay full price the first time, then roughly 10% of input cost on every repeat within the next few minutes. Every major provider now sells it — if your bill contains any repeated context, this is the single highest-ROI lever in the course.

🧩 Core Concepts

🥪
The Sandwich Rule
Stable content first (bread), dynamic content last (filling). Prefix match stops at the first byte of difference.
🎯
1,024-token Minimum
Below ~1K tokens, no provider caches. Short prompts never benefit. Put your long context in deliberately.
💸
Write Premium vs Read Discount
Anthropic charges extra on write (1.25x or 2x) to afford the 90% discount on every read. OpenAI & Gemini have no write premium.
⏱️
Sliding TTL
Every cache hit resets the timer. Busy caches stay warm; idle caches expire. TTL options: 5 min (all) or 1 hour (Anthropic only).
🔁
Semantic Cache Layer
Skip the LLM entirely when a new query is semantically close to a cached one. Reduces calls by 40–70% on FAQ-style traffic.
🧰
Stacks With Everything
Caching × batch = ~5–10% of list price. Caching × compression × routing compounds further. Use multiple levers.

⚠️ Silent Cache Killers

␣
Trailing whitespace
One extra \n invalidates the whole prefix match. Normalize aggressively.
📅
"Current date" headers
A timestamp at the top of the system prompt busts the cache on every call, every day.
🎭
Template re-render drift
Jinja/Handlebars output that differs microscopically between calls (dict ordering, float formatting) kills matches.
🏁
Missing cache_control
On Anthropic, forgetting the marker on half your code paths = half the reads miss the cache.

💰 Real-World Savings Example

  1. Support bot, 20K system prompt, 1K calls/hour

    Sonnet 4.6. Monthly traffic: ~730 hours.

  2. Uncached: $51,465/month

    20K × 1K × 730 = 14.6B input tokens at $3/M — the system prompt dominates.

  3. Prefix-cached: $12,097/month

    1 write/hour + 999 reads/hour. Write cost $55, reads $4,377 — $39K saved.

  4. + Semantic cache (50% hit): ~$6,048/month

    Half the queries skip the LLM entirely. Total savings: 88% ($45K/month).

📖 Key Terms

KV-cache
The model's internal attention lookup table. What's reused under the hood when your prefix matches.
Prefix match
Cache only reuses bytes from the start of the prompt up to the first difference — strict, byte-level.
Sliding TTL
Expiration timer that resets on every cache access. Busy caches never expire; idle ones do.
cache_control
Anthropic's explicit marker on a message block indicating it should be cached (up to 4 per prompt).
Implicit caching
Gemini's automatic mode — no API flag, no storage fee, applies to eligible prefixes.
Explicit caching
Gemini's named-cache mode — 32,768 token minimum, user-set TTL, small hourly storage fee.
cached_tokens
The response field on OpenAI indicating how many input tokens were served from the cache. Your hit-rate meter.
Semantic cache
Vector-similarity lookup that returns a cached response when a new query is close enough to a prior one.
Similarity threshold
The cutoff (typically ~0.85) that decides whether two queries are "similar enough" to reuse an answer.

❓ Quick Quiz

What's the minimum prefix length for prefix caching on OpenAI and Anthropic?

1,024 tokens on both. Shorter prompts never cache. Gemini Pro's implicit cache needs 2,048; explicit cache needs 32,768.

Why is putting a timestamp at the top of your system prompt a disaster for caching?

Prefix caching matches byte-for-byte from position 0. The timestamp differs every call, so the match breaks immediately and the entire prompt is billed as fresh input every time.

On Anthropic's 5-minute TTL, how many reads do you need to break even on the write premium?

About 0.28 — even a single cache read pays back the 1.25x write premium. If a prefix repeats at all within 5 minutes, cache it.

Why use semantic caching on top of prefix caching instead of just one?

They're orthogonal. Prefix caching cuts the cost of each call; semantic caching cuts the number of calls. Combined, they can reduce a bill ~88% on FAQ-style workloads.

Where should you put dynamic content in your prompt?

At the very end — after all static content. The cacheable prefix extends up to the first dynamic byte, so dynamic-last maximizes how much gets cached.

🧰 The Caching Audit Checklist

1️⃣
Dump a typical prompt
Pick your highest-volume endpoint. Capture the full prompt — system, context, user message.
2️⃣
Token-count each section
Provider tokenizers (count_tokens, tiktoken, countTokens) separate the static from the dynamic parts.
3️⃣
Classify: static / semi / dynamic
Any section that changes call-to-call is dynamic. A timestamp, a UUID — dynamic.
4️⃣
Measure call cadence
Average gap between calls — under 5 min fits 5-min TTL, under 1 hour fits 1-hour TTL.
5️⃣
Compute savings ceiling
cacheable-prefix × (calls/month − 1) × 0.9 × input-rate = your upper bound on caching savings.
6️⃣
Add cache-hit dashboard
Log cached_tokens, alert when hit rate drops 20%+ (usually = a template-renderer change).

Key Takeaways

  • Prompt caching gives ~90% off input costs on every read — on all three major providers in 2026.
  • Put static stuff first, dynamic stuff last — the prefix match breaks at the first difference.
  • Anthropic charges a write premium (1.25x/2x); OpenAI & Gemini implicit are free to write.
  • Break-even on Anthropic 5-min is <1 read; even a single repeat pays for itself.
  • Watch the cache-hit-rate metric (cached_tokens) — it drifts when templates change.
  • Whitespace, timestamps, and template noise silently kill cache matches.
  • Layer semantic caching on top for FAQ-style traffic — skips the LLM entirely.
  • Combine caching + batch + routing for compound savings toward 5% of list price.
