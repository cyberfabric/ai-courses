Chapter 4

Prompt Caching: The 90% Discount You're Not Using

Cache writes, cache reads, TTLs and the break-even math behind the biggest single lever on your AI bill

The parking-meter receipt trick

Imagine you rent a parking meter downtown for $3/hour. The first hour, you pay full price. But the garage has an odd deal: if you come back within the next hour and park in the same slot, you pay 30 cents instead of $3. Same slot, same price list, one small loyalty trick.

Prompt caching works exactly like this. The first time you send a big chunk of context — a system prompt, a retrieved document, a conversation history — you pay full price, plus a tiny premium. Every time you re-send the same prefix shortly after, you pay roughly 10% of input cost. That's a 90% discount on the bulk of most real prompts.

If your feature sends any repeated context — and almost every production AI feature does — caching is probably the single highest-return lever in this entire course. We're going to learn the mechanics on each provider, the break-even math, the failure modes that silently kill the cache, and how to stack caching with other discounts.

The core insight: LLMs don't have memory between calls, but the infrastructure that runs them can keep a warm KV-cache of your prefix for a few minutes. All three major providers now sell that warm slot at a discount. Your job is to make sure your prefix is cache-friendly.

What's actually being cached

When you send a prompt, the model's attention layer computes a huge internal lookup — the "KV-cache" — for every token in order. That computation is the expensive part. If the first 10,000 tokens of your next request are byte-identical to what you sent before, the provider can skip recomputing that lookup and reuse the warm one. They pass most of that saving back to you.

The practical corollary: caching matches on the prefix, not the content. The cacheable portion always starts at the beginning of the message and ends where the first byte of difference is. If you put anything dynamic (a timestamp, a user ID, a session token) anywhere before your big system prompt, the whole cache benefit vanishes.

This is why "put dynamic content last" is the single most important caching rule on every provider.

Anthropic — explicit with cache_control

Anthropic's implementation is explicit: you place a cache_control marker on the message block you want to cache. Everything up to and including that block becomes the cacheable prefix. Two TTL options: the default 5-minute cache, and an extended 1-hour cache.1

Four things to internalize about Anthropic caching:

  • The TTL is sliding, not fixed. Every time the cache is hit, the timer resets. A busy cache stays warm; an idle cache expires at the interval.
  • You pay a small premium on the first call. That first write at 1.25x (or 2x) is what pays for the subsequent cheap reads.
  • Multiple breakpoints are free. You can drop cache_control on up to four different points in the prompt (e.g., system prompt, retrieved doc A, retrieved doc B, conversation history) so different prefixes can cache independently.
  • Workspace isolation is enforced. Since February 2026, caches don't leak across workspaces — good for multi-tenant setups, but your caches also won't share across dev/prod if you split them by workspace.

OpenAI — automatic, no flags required

OpenAI took the opposite design choice: prefix caching is automatic on every GPT-5.4 / GPT-5 and o-series request. You don't mark anything, you don't add any flags. If your prefix is longer than 1,024 tokens and the infrastructure still has it warm, you get the discount automatically.2

  • Discount: cached input tokens cost 10% of the normal input rate — a 90% discount.
  • Minimum prefix: 1,024 tokens. Prompts shorter than that never cache.
  • TTL: approximately 5–10 minutes after last use. Not guaranteed; OpenAI does not publish a fixed TTL.
  • Cache-write premium: none — writes are billed at the normal input rate.
  • Match granularity: prefix is hashed in 128-token chunks, so caching kicks in at 1,024, 1,152, 1,280, …

Because there's no write premium, OpenAI caching is almost always "free money" — if your prefix repeats, the discount shows up automatically. The downside: the unpublished TTL means your cache-hit rate can drift under load, and you'll only see it in billing.

Tip: OpenAI's response object includes usage.prompt_tokens_details.cached_tokens. Log it per request, aggregate by feature, and you've got a live cache-hit-rate dashboard with no extra infrastructure. If that number stays near zero for a feature with a big repeated prefix, you probably have a whitespace or dynamic-content bug.

Google Gemini — implicit + explicit, two modes

Gemini has two caching flavors that confuse almost everyone the first time they meet them.3

  • Implicit caching — automatic, like OpenAI. Prefixes of 1,024+ tokens (Flash) or 2,048+ tokens (Pro) qualify. No storage fee. Discount: 75% on Gemini 2.0, 90% on Gemini 2.5 and later.
  • Explicit context caching — you create a named cache object, reference it in subsequent requests. Minimum cache size: 32,768 tokens. You pay a small storage fee per hour per 1M cached tokens (billed while the cache lives). Same 75/90% discount on the read side.

Rule of thumb: for "normal" caching needs — system prompts, RAG context, chat history — stick with implicit. It's free, no plumbing, and covers the common cases. Use explicit only when you have a very large stable prefix (30K+ tokens — e.g., a full policy manual, a 500-page handbook, a big PDF) and you want to guarantee cache residency beyond the implicit TTL.

Watch out: explicit caches come with a storage bill. If you create a cache and forget to delete it (or pick a long TTL), the hourly storage fee can quietly add up. Always set a TTL that matches how long the cache is actually useful, and tear it down in a finally block.

Break-even math: when does caching actually pay off?

Caching isn't always a free lunch on Anthropic — you pay a premium on the write. If the cached prefix is only used once, you've spent 1.25x input for no return. The question is: how many reads before you break even?

Let X = number of times the same cached prefix is reused (read). For Anthropic 5-minute TTL: 

Without cache:  (X + 1) × 1.00x   = X + 1
With cache:     1.25x + X × 0.10x = 1.25 + 0.10X

Break-even: X + 1 = 1.25 + 0.10X
           0.90X = 0.25
           X = 0.28 reads

Translation: even a single cache read pays for the write premium on the 5-minute TTL. If your prefix is going to be reused at least once within five minutes, cache it.

For the 1-hour TTL on Anthropic: 

Without cache:  (X + 1) × 1.00x   = X + 1
With cache:     2.00x + X × 0.10x = 2.00 + 0.10X

Break-even: X + 1 = 2.00 + 0.10X
           0.90X = 1.00
           X ≈ 1.11 reads

You need the prefix reused about once within the hour to break even on the 1-hour TTL. Two or more reads and you're clearly in profit territory.

For OpenAI and Gemini implicit caching, there's no write premium, so the break-even is zero reads — any cache hit is pure savings. You can't lose money by "accidentally" caching a prefix that never repeats.

Real-numbers example: a chatbot with a 20K-token system prompt

Let's put it together. You're running a customer-support bot on Sonnet 4.6. The system prompt is 20,000 tokens (policies + tone guide + few-shot examples) and the user message averages 500 tokens. The bot handles 1,000 calls per hour — plenty of traffic to keep a cache warm.

What to cache, what NOT to cache

Not every long chunk of context benefits from caching. Use this rough taxonomy:

Cache these
System prompts (rules, tone, few-shot examples) Retrieved documents that repeat across users Policy manuals, schemas, code context Conversation history (when it stabilizes) Tool definitions and function schemas Translation glossaries, style guides
These are stable, often long, and reused across many calls. Classic cache hits.
Do NOT cache these
User IDs, session tokens, timestamps A/B test variant IDs Random seeds, nonces Freshly retrieved documents unique to one query Streaming chunks mid-generation Anything with PII you don't want retained
These break the prefix match, inflate cache churn, or leak identifiers across the cache infrastructure.

The ordering rule carries the whole design: stable stuff first, dynamic stuff last. If you're building a prompt, think of it as a sandwich — the bread (system + retrieved context) is cacheable, the filling (user's current message) goes at the end.

The silent killers: whitespace, invalidation, near-misses

Caching looks binary on paper — either your prefix matches or it doesn't — but real-world cache misses almost always trace to one of these five bugs. Most of them cost you money without any warning signs until you look at a cache-hit-rate graph.

  • Trailing whitespace or newline differences. One extra \n between the cached block and the next block invalidates the whole match. Normalize aggressively.
  • Template-engine re-rendering. If your system prompt is rendered from Jinja/Handlebars and the template renders even microscopically differently on each call (e.g., dict-ordering in Python <3.7, floating-point formatting), the prefix won't match.
  • Dynamic content inside a "static" section. The classic: a system prompt that starts with Current date: 2026-04-19 14:33:07. That one timestamp kills the entire cache, every call, every day.
  • Forgotten cache_control on subsequent calls. Anthropic requires the marker on every request that wants to read from the cache. If half your code paths set it and half don't, half miss.
  • Cache eviction under pressure. Under sustained load or provider capacity events, caches can be evicted early. Your hit rate drops silently; your bill creeps up. Monitor.
Tip: When you build a cached prefix, write a unit test that hashes the string and asserts the hash is stable. If the hash changes between runs without a real content change, your template is contaminating the cache. Catch it once in CI, never debug it in production.

Semantic caching — the 2026 complement

Prefix caching only helps when prefixes are byte-identical. But a huge portion of real traffic is semantically identical with small wording differences: "how do I reset my password" vs "I forgot my password, what do I do" vs "where's the password reset button?"

Semantic caching handles that case. You embed the incoming query, check a vector store for a sufficiently similar cached query, and return the cached response if similarity is above a threshold. When it hits, you skip the LLM entirely — the whole call costs pennies instead of dollars.4

The two popular stacks in 2026:

R
Redis Semantic Cache (LangChain RedisSemanticCache)
Redis + a vector-similarity index. Default distance threshold 0.2 (cosine). Recommended starting similarity threshold ~0.85. Drop-in wrapper for LangChain apps.
U
Upstash Semantic Cache
Serverless/edge-friendly, built on Upstash Vector. Good for Cloudflare Workers and Vercel edge deployments where you don't want to run Redis yourself.

Reported savings are real: semantic caching can skip 40–70% of LLM calls on help-desk, FAQ, and documentation workloads, with response latency dropping 40–50% because a vector lookup is much faster than a model call.

Warning — similarity threshold sensitivity: threshold too low and you return wrong answers (the cache "matches" queries that actually need different responses). Too high and the cache never hits. Start at 0.85 similarity (≈ 0.15 cosine distance), log false positives, and tune up or down based on real traffic. Always keep a fallback path to the LLM when confidence is low.

Same 1,000-calls/hour support bot from earlier, but now looking at three approaches over a month of traffic:

No caching, no optimizations

System prompt of 20K tokens re-sent on every call. User message + output unchanged.

  • Monthly input cost: $44,895
  • Monthly output cost: $6,570
  • Monthly total: $51,465
  • Effective cost per call: $0.071

Prefix caching + semantic cache (50% hit)

20K system prompt cached (1 write/hour, 999 reads/hour). 50% of user queries served from semantic cache — no LLM call at all.

  • Prefix caching saves ~$39K (see prior section)
  • 50% of calls skipped entirely → ~$6K more saved
  • Monthly total: ~$6,048
  • Effective cost per call: $0.008
→ Verdict: combining prefix + semantic caching cuts the bill by ~88% for this workload. $45,000/month saved on one feature.

The scenarios stack: prefix caching attacks the input tokens of the calls you do make; semantic caching attacks the number of calls you make at all. They're orthogonal — use both.

Stacking caching with the other levers

Prompt caching is one of the five levers. It gets even better when stacked:

  • Caching + batch (Chapter 5): batch is already 50% off; adding caching to long repeated prefixes in batch workloads gets you down toward an effective 5–10% of list price.
  • Caching + compression (Chapter 6): compress the dynamic part of the prompt so that the cacheable part is proportionally bigger. Small content changes don't bust the prefix.
  • Caching + routing (Chapter 7): different models have different cached system prompts — make sure your router doesn't cause each route to rewrite the cache constantly.
  • Caching + observability (Chapter 9): instrument cached_tokens per request, watch hit rate trend lines — a 20% drop in cache-hit rate typically signals a template change that broke the prefix.

Exercise: audit one feature's cache-ability

Pick one AI feature you ship. Spend 30 minutes on this:

  1. Dump a typical prompt. Open your most-called endpoint. Grab the full prompt structure that ships to the provider — system message, retrieved context, user message.
  2. Count the tokens in each section. Use the provider's tokenizer (Anthropic's count_tokens, OpenAI's tiktoken, Gemini's countTokens). Break out: system, context, user, total.
  3. Classify each section as static, semi-static, or dynamic. A section with any token changing call-to-call is dynamic.
  4. Compute the cacheable prefix length. That's the sum of everything static, up to the first dynamic section.
  5. Estimate your cache-read rate. Assume calls land every T seconds on average. If T < 300 (5 min), you're in 5-min-cache territory. If T > 3,600, caching won't help much without the 1-hour TTL.
  6. Compute monthly savings. (cacheable prefix tokens) × (calls/month − 1) × (0.9 × input_rate). That's your ceiling.
  7. If the ceiling exceeds 10% of the feature's monthly bill: this is a work item, not a "nice-to-have." Add it to next sprint.

Bonus: grep your code for timestamp formatting, UUIDs, or template-variable substitution inside what you intended to be a static prefix. Each hit is a potential cache killer.

The mental model to take away

Think of every prompt as a two-part package: a bread (static, cacheable prefix) and a filling (dynamic, per-call content). The engineering question is always: how thick is the bread, how thin is the filling, and how well-lined-up is the sandwich?

  • Put stable content first, dynamic content last.
  • Mark the cache explicitly on Anthropic; trust automatic on OpenAI and Gemini.
  • Verify cache hit rate via cached_tokens in the response.
  • Normalize whitespace and template output to prevent near-miss churn.
  • Layer semantic caching on top for high-similarity workloads.
  • Re-check the cache-hit-rate dashboard weekly — it drifts.
