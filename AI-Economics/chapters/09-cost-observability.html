Chapter 9

Cost Observability: You Can’t Optimize What You Can’t See

Tools, dashboards, and the 2026 gateway + eval-tool pattern for knowing what every AI feature actually costs.

The electricity-meter analogy

Imagine living in a house with no electricity meter. You get one invoice per month — sometimes $40, sometimes $400 — and the bill never tells you which appliance ate the watts. The fridge? The AC? The teenager gaming at 3 a.m.?

That’s your AI stack without observability. The provider sends you a number at the end of the month. You squint at it, shrug, and pay. Next month it’s double and nobody knows why.

Cost observability is the meter. It breaks that monthly number down to per-feature, per-user, and per-request cost — in real time, not at the end of the billing cycle. Without it, every optimization you’ve learned so far is guesswork.

The rule in one sentence: if you can’t point at a specific feature and say “this costs $X per request, $Y per user per month, and we could cut it by Z % by doing W,” you’re not cost-engineering. You’re hoping.

What to instrument (the “non-negotiable ten”)

Before you pick a tool, know what it has to capture. Any AI request that hits your logs should record at least these ten fields:

  1. Request ID — unique, correlates to app logs
  2. Feature tag — summarizer, support-bot, code-review (metadata field)
  3. User ID / tenant ID — for per-user and B2B showback
  4. Model used — and the provider it was routed to
  5. Input tokens and output tokens (separate)
  6. Cache read / cache write counts (for prompt-cache attribution)
  7. Cost in USD — computed, not inferred
  8. Latency — p50 / p95 / p99 breakdowns later
  9. Status — success, 429, 529, 5xx, user-cancelled
  10. Quality proxy — schema valid? user thumbs-up? eval score?

Feature tagging is the single highest-leverage decision. Without a feature attribute on every request, your dashboards are just a single giant line labelled “LLM spend.” With it, you get pre-built breakdowns per feature, per team, per user, per environment — the raw material for every optimization decision in this course.[1]

The gateway + eval-tool pattern (2026 best practice)

A year ago teams mixed and matched. In 2026 the de-facto architecture is two layers:

🧑‍💻
Your app
Calls LLM via SDK
→
🚪
Gateway
Portkey / OpenRouter
routing + retries + logs
→
🌐
Providers
Anthropic / OpenAI / Gemini
→
📊
Eval & trace
Langfuse / Braintrust / Phoenix
cost + quality + latency

The gateway catches every call (no matter which provider). The eval tool adds the quality layer on top. Together they give you unified cost attribution and quality tracking without having to wrap every SDK yourself.[8]

Why two layers, not one? Gateways are great at capturing raw traffic. Eval tools are great at scoring quality and running A/B tests. A gateway alone misses did this answer actually help?; an eval tool alone misses the requests your gateway routed via failover. Running both is cheap and covers both sides.

Per-feature cost attribution: the metadata trick

Every modern SDK supports a metadata bag on each request. Tag every call with feature, user_id, environment, and (optionally) version. That’s it — the rest is just filtering. 

client.messages.create(
    model="claude-haiku-4-5",
    messages=[...],
    metadata={
        "feature": "ticket-summarizer",
        "user_id": "u_42",
        "environment": "prod",
        "version": "v3",
    },
)

Once every request carries these tags, your Langfuse / Braintrust / Phoenix dashboard groups them automatically. You can ask the question every finance partner asks — “what does ticket-summarizer cost us this month?” — and get an answer in seconds instead of days.

A worked example: finding the hog

A team shipped three AI features and got a $12,000 monthly bill. Nobody could say where the money went. Here’s what instrumentation revealed:

Before instrumentation — $12,000/month, mystery

Bill grew from $4K to $12K over three months. No per-feature breakdown. Team assumed “users grew, so it grew.”

After Langfuse + feature tagging — the real picture

  • ticket-summarizer: 85 % of traffic but only $1,400 — cheap Haiku calls, well-cached
  • meeting-notes: 12 % of traffic, $3,200 — Opus calls, no caching
  • draft-assistant: 3 % of traffic but $7,400 — 30K-token context on every call, never cached, Opus-only
→ One feature (3 % of traffic) ate 62 % of the bill. Moving it to Sonnet + cache dropped total spend to $4,800/month — a $7,200 saving in one sprint.

This is the most important truth about cost observability: the distribution of spend is always weirder than the distribution of traffic. Find the outlier feature first. Fix that. Everything else is rounding error.

Per-user attribution for B2B SaaS

If you sell AI features to businesses, every customer’s usage is a line item in your unit economics. The 80/20 rule bites hard here — one enterprise customer can eat as much token volume as your next 50 customers combined.

  • Tag every request with the tenant/organization ID as a first-class field.
  • Dashboards grouped by tenant make outliers visible in one glance.
  • Alerts on per-user anomalies. If a single user’s daily token count jumps 10×, they’re either running an agent loop or doing something abusive. Either way, you want to know.
  • Showback reports. Send customers their own usage monthly. Upsell opportunity, fairness, and trust all in one pattern.

User IDs are PII — hash or pseudonymize them. Don’t send raw email addresses into your observability tool. Use a stable hash (e.g., SHA-256 of the user ID + a pepper). Most tools offer a PII-scrubbing layer; turn it on before day one.

Alerts that don’t wake you at 3 a.m.

Cost spikes are rarely sudden outages. They’re usually slow drifts — a bad prompt change, a retry loop on a flaky endpoint, an agent that starts hallucinating longer and longer outputs. You want alerts that catch the drift before it becomes a bill.

❌ Threshold alerts

“Alert if daily cost > $500.”

Problem: $500 yesterday was normal but today it’s a crisis. The threshold stays flat while traffic grows, so you get false positives or silent misses.

✅ Anomaly alerts

“Alert if cost per feature deviates more than 3σ from the 30-day rolling mean.”

Catches drift and scales with growth. Most modern tools (Langfuse, Braintrust, Phoenix) have this built-in.

Good alert types to wire up:

  • Per-feature cost anomaly — 3σ over baseline
  • Per-user runaway — single user exceeds their historical daily p99 by 5×
  • Cache hit rate collapse — drops below a floor (e.g., 70 % on a cached prompt)
  • Retry ratio spike — 429/529 rate rises above 1 %
  • Model drift — a feature suddenly routing 80 % to Opus when it used to be 10 %

The dashboard that matters

Build one page — a weekly cost review dashboard — with exactly five panels:

  1. Weekly cost trendline (30-day rolling, stacked by feature)
  2. Top-10 most expensive requests (drill-down to the prompt for debugging)
  3. Cache hit rate per feature (chart: currently-cached / should-be-cached)
  4. Cost per user / per tenant (sorted descending, filter outliers)
  5. Model-mix distribution (stacked bar: Haiku / Sonnet / Opus / OSS over time)

Look at this dashboard every Monday for 15 minutes. Most of the cost wins you find all year come from five to ten glances at this one page.

Build vs. buy: the 5-line SDK wrapper

Before paying for a SaaS, you can roll basic observability in an afternoon. Here’s a wrapper that handles feature tagging, cost attribution, and structured logging for Anthropic calls: 

def call_llm(messages, feature, user_id, model="claude-haiku-4-5"):
    start = time.time()
    resp = anthropic.messages.create(
        model=model, messages=messages,
        metadata={"feature": feature, "user_id": hashlib.sha256(user_id.encode()).hexdigest()[:16]},
    )
    cost = compute_cost(model, resp.usage)  # per-model rate table
    logger.info({
        "feature": feature, "user_hash": metadata["user_id"],
        "model": model, "in": resp.usage.input_tokens, "out": resp.usage.output_tokens,
        "cache_read": getattr(resp.usage, "cache_read_input_tokens", 0),
        "cost_usd": cost, "latency_ms": int((time.time() - start) * 1000),
        "status": "ok",
    })
    return resp

Ship this, pipe the logs to Datadog / CloudWatch / your data warehouse, and you have 80 % of what a cost-observability SaaS gives you. Graduate to Langfuse or Braintrust when your team starts asking for quality evals, A/B tests, and per-user dashboards.

🏗️ Roll your own

  • 1–2 eng-days to ship
  • $0 SaaS cost
  • You own the data and schema
  • No evals, no UI, no alerting — DIY
  • Best for: small teams, simple cost tracking, early-stage products

💳 Buy (Langfuse / Braintrust / Phoenix Cloud)

  • Under an hour to ship
  • $29–$249/month entry paid tiers
  • Built-in evals, dashboards, anomaly alerts
  • Quality tracking on day one
  • Best for: teams running evals, teams doing showback, B2B SaaS
Verdict: start with the wrapper. Migrate to a SaaS when evals or per-user dashboards become the bottleneck. The wrapper stays useful — many teams keep it even after adopting a tool.

🧪 Exercise: Instrument one feature this week

  1. Pick one AI feature currently running in production.
  2. Add feature + user_id metadata to every request.
  3. Log the 10 non-negotiable fields (request ID, model, tokens, cost, status, latency, cache, quality proxy, feature, user).
  4. Pipe logs into Langfuse Hobby (free) or your existing logging platform.
  5. Wait 7 days. Pull a dashboard showing cost per user, cost per model, and p95 latency.
  6. Identify the outlier — the one user, prompt, or model that’s disproportionately expensive.
  7. Write down the projected monthly saving if you fix it. Then fix it.

The production observability checklist

  • ✅ Every request carries feature, user_id, and environment metadata.
  • ✅ Cost is computed per request, not inferred from the bill.
  • ✅ Cache read / write counts are captured alongside tokens.
  • ✅ Quality proxy (schema pass, thumbs-up, eval score) logged on every request.
  • ✅ Weekly dashboard with 5 panels exists and gets looked at.
  • ✅ Anomaly alerts, not thresholds, on per-feature cost.
  • ✅ User IDs are hashed before leaving your app.
  • ✅ You’re NOT betting the farm on Helicone for new work (maintenance-mode).
  • ✅ Gateway + eval-tool layered architecture, or a deliberate reason not to.
