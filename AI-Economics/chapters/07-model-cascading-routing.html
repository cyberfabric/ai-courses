Chapter 7

Model Cascading & Routing: Right Model, Right Job

Why your cheapest model should answer most of the questions — and your expensive model should only show up when it’s actually needed.

The support-team analogy

Picture a well-run customer support team. A tier-1 agent picks up the phone for most calls — reset passwords, track orders, read the FAQ out loud. A tier-2 specialist handles refunds and weird edge cases. The CEO gets involved when a journalist calls about a lawsuit.

Nobody would dream of having the CEO take every call. It’s not just expensive — it’s wasteful. Most questions don’t need a CEO.

Now look at your AI stack. If every request goes to Claude Opus or GPT-5.4 Standard, you’re running a support team where the CEO picks up every call. Cascading and routing are how you build the tier-1 / tier-2 / tier-3 discipline into your API layer.

The rule in one sentence: send the easy 80 % of traffic to the cheapest model that works, and reserve the premium model for the 20 % that actually needs it. Done right, this is a 60–80 % cut on total spend without a visible quality drop.[1]

Cascading vs. routing: two flavors of the same idea

These two words get used interchangeably, but they aren’t the same pattern. The difference matters because it changes where you pay the overhead.

Cascading — try cheap first

Mechanic: call the cheap model. If its answer is confident / passes a check, ship it. Otherwise, retry on the expensive model.

Overhead: the easy 80 % pays for 1 cheap call. The hard 20 % pays for 1 cheap call plus 1 expensive call.

Best when: you can tell from the output whether the cheap model succeeded (structured output, self-rated confidence, schema check).

Routing — classify, then dispatch

Mechanic: a small classifier (another LLM call, a keyword rule, or an embedding similarity) looks at the request and picks the right model up front.

Overhead: every request pays for the classifier plus the chosen model.

Best when: the request type is predictable from the input — FAQ vs. coding, short vs. long, extraction vs. reasoning.

Cascade feels like a retry; routing feels like a switchboard. If you already have quality checks on your output (JSON-schema validators, downstream evals, a confidence field), cascading slots in cleanly. If your tasks split cleanly into buckets (support tickets by topic, queries by length), routing is simpler.

How cascading actually works

A cascade needs three ingredients: a cheap model, an escalation signal, and an expensive model. The signal is the tricky part. You have four practical options:

  1. Self-reported confidence. Ask the cheap model to score its own answer 0–1. Escalate if below a threshold (typical: 0.7).
  2. Structured-output check. If the task produces JSON, YAML, or SQL, fail the validator → escalate.
  3. Consistency vote. Call the cheap model twice with slight variation. If answers disagree, escalate.
  4. Downstream eval. Run the output through a cheap rubric (another small LLM call, or a deterministic test) → escalate on failure.

Self-reported confidence is poorly calibrated. A 2026 cascade paper (CascadeDebate) reported that static confidence thresholds are “brittle under distribution shifts” — models routinely say “0.9 confident” when they’re wrong, especially on edge cases.[2] Use confidence as one signal, not the only one.

A 2-level cascade in pseudocode

def answer(query):
    draft = call_haiku(query)           # cheap
    if draft.confidence >= 0.75
       and validator(draft) == "ok":
        return draft                     # ship it
    return call_sonnet(query)           # escalate

That’s the whole pattern. Ten lines of code can move most of your traffic from a $3-per-million-input model to a $1-per-million-input model.[3]

How routing actually works

Routing replaces the retry loop with a classifier. Before the “real” model gets involved, a smaller and faster step decides which one to use.

Three ways to build the classifier

  • Regex / keyword rules — nearly free, brittle. Fine for narrow domains (“if message contains refund, route to refund-model”).
  • Embedding similarity — cheap (~$0.02 per 1M tokens for OpenAI text-embedding-3-small), robust to rewording. Build a library of labeled example prompts, take cosine similarity, dispatch to the bucket with the nearest neighbor.
  • Tiny-LLM tagger — call Haiku 4.5 with a short classification prompt. Typical tagger cost: 300 tokens in, 10 tokens out = ~$0.0003 per request.[3] Worth it if rules and embeddings aren’t accurate enough.

The router-overhead break-even rule. A router is only profitable if its classification cost is smaller than the savings it creates. If the premium model costs $0.01 per request and the cheap model costs $0.001, a $0.0005 tagger pays for itself whenever it correctly routes one request per two to the cheap model.

The break-even math (worked example)

Imagine a customer-support bot receiving 1 million requests per month. Average input: 2,000 tokens. Average output: 300 tokens. Let’s cost this three ways on Anthropic’s Claude family at April 2026 rates: Haiku 4.5 at $1/$5 per 1M, Sonnet 4.6 at $3/$15 per 1M, Opus 4.7 at $5/$25 per 1M.[3]

Scenario A: Everything on Sonnet (the “safe” default)

  • Input: 1,000,000 req × 2,000 tokens = 2,000M tokens × $3 = $6,000
  • Output: 1,000,000 req × 300 tokens = 300M tokens × $15 = $4,500
  • Monthly total: $10,500

Scenario B: Cascade — Haiku first, Sonnet on 20 % failure

  • Haiku on 100 % of traffic: 2,000M in × $1 + 300M out × $5 = $3,500
  • Sonnet on 20 % escalations: 400M in × $3 + 60M out × $15 = $2,100
  • Monthly total: $5,600

Scenario C: Router with Haiku tagger — 80 % Haiku, 18 % Sonnet, 2 % Opus

  • Tagger (tiny-LLM): 1M req × 300 in + 10 out on Haiku = $305
  • Haiku 80 %: 1,600M in × $1 + 240M out × $5 = $2,800
  • Sonnet 18 %: 360M in × $3 + 54M out × $15 = $1,890
  • Opus 2 %: 40M in × $5 + 6M out × $25 = $350
  • Monthly total: $5,345
→ Sonnet-everything $10,500 vs. 3-tier router $5,345 = 49 % saved per month, or $61,860 over a year.

Scenario B (cascade) and Scenario C (router) land within 5 % of each other on pure cost. The tie-breaker is usually quality and engineering effort: cascades are easier to build but add latency on hard queries; routers are harder to calibrate but keep latency flat.

The 2026 tooling landscape

You don’t have to build this yourself. A handful of platforms offer drop-in routers and gateways that do the dispatch logic for you.

🛣️ OpenRouter

The most popular neutral aggregator. One OpenAI-compatible endpoint, 300+ models from 60+ providers, passthrough pricing (no markup on model rates) plus a 5.5 % fee on credit purchases. Its Auto Router meta-model (2M-token context) picks a model per request and bills at the routed model’s rate.[4]

🧠 Martian

One of the first production routers, with “model mapping” interpretability under the hood. Claims 20–97 % cost reductions with GPT-4-level quality. Free tier: 2,500 requests. Developer plan: $20/month for API access. Used at Amazon, Zapier, and through Accenture’s Project Spotlight.[5]

💎 Not Diamond

Routing-as-a-model: upload an evaluation dataset and Not Diamond returns a router tuned to your workload. Teams report 10–100× inference savings for narrow tasks. Bills on inference cost incurred during optimization (you only pay if the model you picked is actually improved).[6]

🔬 RouteLLM (open source)

Academic framework from LMSYS. Trains routers on preference data. Published benchmarks: up to 85 % cost reduction at 95 % of GPT-4’s MT-Bench score; 45 % reduction on MMLU; 35 % on GSM8K. Drop-in OpenAI-client replacement.[1]

🚪 Portkey Gateway

Full AI gateway — not just a router. 1,600+ models, weighted routing, automatic retry with exponential backoff (up to 5 attempts), per-provider fallback, observability, guardrails, PII detection. Free open-source core; paid plans add enterprise controls.[7]

The tool landscape still shifts fast. Routers that were hot two years ago have been acquired, shuttered, or turned into enterprise-only products. Before building a dependency on a third-party router, check their pricing page and their GitHub commit graph. If the last commit was six months ago, treat it as a red flag.

When the 60–80 % savings claim is real (and when it isn’t)

You’ll see “save 80 % with cascading!” all over marketing copy. The number is real in specific conditions and fictional in others. Know which side of the line you’re on.

✅ Where the big savings are real

  • FAQ / knowledge-base answers (90 % cheap-model-capable)
  • Intent classification & tagging
  • Structured extraction from clean inputs
  • Short summaries & rewrites
  • Translation between common languages
  • SQL generation for simple queries

❌ Where the savings evaporate

  • Multi-step agentic workflows (small errors compound)
  • Complex code reasoning across a codebase
  • Long-form creative writing
  • Legal or medical reasoning with edge cases
  • Tasks where quality degradation is silent and only surfaces in production
  • Workloads already dominated by a long system prompt (cache that first — see Chapter 4)
Verdict: cascading / routing wins when tasks are heterogeneous and most of them are easy. It loses on uniformly-hard workloads.

Failure modes to watch for

  • Silent quality drift. The cheap model starts handling more traffic over time because your escalation signal drifts. You notice only when a user complains. Mitigation: sample 1 % of requests and score both models in parallel, weekly.
  • Miscalibrated confidence. The cheap model says “0.9 confident” and is still wrong. Mitigation: pair confidence with a deterministic check (schema, test, embedding similarity to known-good answer).
  • Cache miss on escalation. If you cache the cheap model’s system prompt but not the expensive model’s, every escalation pays cache-write costs on the big model. Mitigation: cache both tiers’ system prompts symmetrically.
  • Latency tail. A cascade that escalates 20 % of the time adds a big number to p95 latency. Mitigation: time-box the cheap attempt; speculatively fire the expensive model in parallel on high-stakes requests.
  • Router feedback loop. If the router is trained on its own historical choices, it will keep routing the same way and never explore. Mitigation: inject 2–5 % random traffic to the other tiers and evaluate.

Build your own router in ten lines

Before paying for a third-party service, try the hand-rolled version. It’s usually enough. 

def route(query: str) -> str:
    length = len(query.split())
    has_code = "```" in query or "def " in query
    needs_reasoning = any(k in query.lower()
        for k in ["why", "explain", "prove", "compare"])

    if has_code or needs_reasoning or length > 500:
        return "sonnet-4-6"
    if length < 20:
        return "haiku-4-5"
    return "haiku-4-5"  # default cheap

answer = call_model(route(query), query)

That’s a working router. Not as smart as Not Diamond. Not as flexible as Martian. But free, deterministic, and auditable. Start here. Graduate to a paid service when your evals tell you you’re leaving money on the table.

Always measure before you migrate. The first router you ship should log every routing decision with the model chosen, the cost, and a quality proxy (schema pass? length reasonable? user thumbs-up?). Without those logs you can’t tell if the router is helping or quietly degrading the product.

Cascade depth: 2-tier vs 3-tier vs 4-tier

The cascade examples above use two tiers (cheap → expensive). In production, you’ll often want more rungs on the ladder. But each rung adds complexity and latency variance.

DepthTypical tiersWhen it winsWhat it costs you
2-tier Haiku → Sonnet Homogeneous traffic, clear easy/hard split Simplest. Escalates a lot for “medium” requests that don’t really need Sonnet.
3-tier Haiku → Sonnet → Opus Mixed workload with a small fraction of genuinely hard queries Catches the 2–5 % of really hard cases. Escalation logic gets 2× more code.
4-tier Free OSS → Haiku → Sonnet → Opus High-volume, cost-obsessed workloads (content moderation, embeddings) Free tiers bring quality floor risk and rate-limit ceilings. Monitor drift weekly.

Most production teams land at 2-tier or 3-tier. Going beyond that makes the routing logic harder to audit than the savings are worth. As a rule of thumb, each extra rung should save at least 15 % over the rung above it on its traffic slice — otherwise the complexity isn’t paying for itself.

Free-tier routing: a 2026 wildcard

OpenRouter exposes 29 free models as of April 2026 — DeepSeek R1, Llama 3.3 70B, Gemma 3, and a rotating cast — at zero marginal cost, rate-limited to 20 requests per minute and 200 per day.[4] That changes the cascade math on high-volume workloads.

Content-moderation classifier: 10M classifications / month

  • Naive (Sonnet-only): 10M req × 600 tokens in × $3 / 1M + 10M × 20 tokens out × $15 / 1M = $21,000
  • Haiku-only: 10M × 600 in × $1 + 10M × 20 out × $5 = $7,000
  • Cascade: free OSS (80 %) → Haiku (20 %): OSS free + 2M req × (600 × $1 + 20 × $5) / 1M = $1,400
→ Adding a free OSS tier in front of Haiku drops the monthly bill from $7,000 to $1,400 — an additional 80 % saving on top of the “cheap-only” baseline.

Free tiers come with a ceiling. 20 RPM / 200 RPD on OpenRouter free models means you can’t route more than ~6,000 daily requests through them per key. For higher volumes, you need self-hosted OSS (Chapter 10) or paid inference at bulk rates. Free is a great first rung — never your only rung.

Stacking with the other levers

Cascading and routing stack cleanly with everything in the earlier chapters:

  • + Caching (Chapter 4): the cheap model’s long system prompt caches beautifully because it’s called on 100 % of traffic — highest cache-hit rate in your stack.
  • + Batch (Chapter 5): route batchable work to the cheap model and batch it → 50 % discount on top of the already-cheap tier.
  • + Compression (Chapter 6): compress the context once, then feed the shrunken context through the router. Every tier enjoys the compression.
→ Stacking a router + caching + compression routinely clears 80 % total savings on support workloads.

🧪 Exercise: Profile & router-size your app

  1. Export a week of request logs from one AI endpoint in your app.
  2. Tag each request by complexity: trivial (FAQ, extraction), medium (summarization, short reasoning), hard (multi-step reasoning, code gen).
  3. Count the distribution. What percentage is trivial?
  4. Recompute monthly cost under a router: trivial → Haiku, medium → Sonnet, hard → Opus.
  5. Compare to your current bill. If the delta is over 30 %, schedule a routing sprint.

The production checklist

  • ✅ You’ve profiled request distribution (trivial / medium / hard).
  • ✅ You’ve defined an escalation signal that isn’t just self-reported confidence.
  • ✅ You log model choice + cost + quality proxy on every request.
  • ✅ You sample both tiers in parallel on ~1 % of traffic to catch drift.
  • ✅ Cache is configured on both tiers, not just the expensive one.
  • ✅ You have a latency budget for the cascade case (cheap + escalation).
  • ✅ If using a third-party router, you’ve verified the vendor is still active.
