Chapter 8

Rate Limits, Tiering & Capacity Planning

The provider’s ceiling is real. Here’s how high it is, how to climb it, and why 2026’s agent traffic might smash through it overnight.

The electricity-grid analogy

Your house has a main fuse. No matter how many appliances you own, you can’t run them all at once — the fuse trips. Providers work the same way. You can have the best-optimized cascade in the world, but if your requests slam into a 50-per-minute rate limit, you’re shedding traffic and losing users.

Rate limits are the fuse box of your AI stack. They’re boring, they’re invisible when things go well, and they become a five-alarm emergency the moment your product gets a write-up on Hacker News.

In 2026, the rules changed. Limits are now measured in tokens, not requests. Tiers unlock by cumulative spend, not by asking nicely. And a new traffic category — agents — can multiply your token usage 10–30× overnight. This chapter maps out the landscape and teaches you to forecast capacity before a bad Monday.

The one thing to remember: rate limits are a cost-engineering problem as much as a reliability one. When you hit the ceiling and fall back to a more-expensive model (or an entirely different provider), your cost per request jumps. Capacity planning is cost planning.

From requests-per-minute to tokens-per-minute

Old rate limits measured “API calls per minute.” That worked when every call was cheap and roughly the same size. In 2026 it’s a joke: one 100K-token summarization call costs the provider orders of magnitude more compute than one 200-token classification call, yet a per-request limit treats them identically.

Every major provider has now standardized on token-based limits, usually sliced three ways:

  • RPM — requests per minute (the old-school cap, still there as a floor)
  • ITPM — input tokens per minute (what you send)
  • OTPM — output tokens per minute (what you receive)
  • TPD / RPD — daily ceilings, often 10× the per-minute cap

All four apply simultaneously. You can run clean on TPM and still get a 429 because RPM spiked from a user loop-clicking a button.[1]

The math you should carry: token-based rate limits mean a single LLM request can consume anywhere from 100 to 100,000 tokens. Size your capacity by the average conversation, not the typical request, and leave a 2× headroom for the long-context outliers.[2]

Gemini’s per-project quota is a silent killer. If you rotate API keys for security, every key inside one Google Cloud project still shares the same token bucket. Splitting by key doesn’t give you more throughput — you need separate projects.[4]

The anatomy of a 429

When you hit a limit, you get HTTP 429 (rate_limit_error). When the provider is overloaded, you get HTTP 529 (overloaded_error). They look similar but need completely different responses.

HTTP 429 — you went over

The server is telling you: you sent too much. Backing off doesn’t fix the root cause.

Response: drop concurrency, raise caching, route easy traffic to a cheaper tier, request a tier bump.

HTTP 529 — they’re overloaded

The provider’s fleet is saturated (global, not your fault). Typically resolves in 2–5 minutes.[7]

Response: retry with exponential backoff + jitter. If it persists, fail over to a secondary provider.

Exponential backoff with jitter, not without. If five instances of your app all retry after exactly 1 second, you’ll reproduce the same thundering-herd spike. Add random jitter (e.g., sleep(base * 2**attempt + random(0, 1))) so the retries spread out. Anthropic’s Python SDK auto-retries 408/409/429/5xx twice by default with exponential backoff.[7]

Agents: the 2026 capacity wildcard

A chat message used to cost one API call. An agent task can cost thirty — maybe more. That’s not a rounding error; it changes your capacity math fundamentally.

A single agent task with a 5-step refinement loop re-reading a 25,000-token context can consume 125,000+ tokens per user action.[2] Multiply that across your user base and you’re not planning for traditional “1 call per user per minute.” You’re planning for agent-shaped traffic that is steadier, heavier, and more persistent than anything you’ve seen before.

The agent multiplier effect: one feature, two traffic profiles

Feature: a code assistant answering 1,000 developer questions per hour.

  • Chat-style baseline: 1 request, 2K input + 500 output = 2,500 tokens → 2.5M tokens/hour total
  • Agent-style (avg 8 steps): 8 requests, 25K input + 500 output each = 204K tokens/task → 204M tokens/hour total
  • Traffic multiplier: ~82× on the exact same user-facing feature.
→ The same 1,000 questions/hour that comfortably fit in Anthropic Tier 2 as chat will blow through Tier 4 ceilings as an agent.

Capacity forecasting: three numbers that matter

Capacity planning isn’t a spreadsheet exercise for the ops team — it’s a back-of-envelope calculation every AI engineer should do before shipping. Three numbers tell you whether you’re safe:

  1. Peak RPS — requests per second at the 95th percentile hour. Pull from your last month of logs.
  2. Tokens per request — average input + output, weighted by model. Don’t forget the agent multiplier.
  3. Growth factor — your expected multiple in 6 months. If you’re growing 3× / quarter, use 9×.

Multiply. Compare to your current tier’s TPM. If the result exceeds 70 % of your ceiling, you’re one marketing email from an outage.

Capacity check: a real-time support bot

  • Peak RPS: 15 requests/sec → 900 RPM
  • Tokens per request: 3,000 in + 400 out = 3,400
  • Peak TPM: 900 × 3,400 = 3.06M TPM
  • 6-month growth factor: 5× → projected 15.3M TPM
  • Current tier (OpenAI Tier 3): 4M TPM → will breach in ~3 months
  • Action: request Tier 4 bump now, or start routing 40 % of traffic to a cheaper/second provider

Read the headers, Luke

Every provider returns rate-limit state on every response. Reading those headers in production is how you see the wall approaching, instead of feeling it when you smack into it.

  • Anthropic: anthropic-ratelimit-requests-remaining, -tokens-remaining, -input-tokens-remaining, -output-tokens-remaining, plus -reset timestamps for each.
  • OpenAI: x-ratelimit-remaining-requests, x-ratelimit-remaining-tokens, x-ratelimit-reset-requests, x-ratelimit-reset-tokens.
  • Google Gemini: quota consumption is visible via Google Cloud Console; client headers are minimal, so instrument at your gateway layer.
  • Perplexity: standard Retry-After when you breach; leaky-bucket state is inferred from request timing.

Log the “remaining” values alongside your per-request cost. When the remaining ratio dips below 30 %, your app should start shedding or slowing non-critical traffic automatically.

Build a “ceiling proximity” dashboard. Chart 1 - (remaining / ceiling) over the past hour. Alert at 80 %. This one metric replaces a dozen reactive postmortems about “why did we suddenly start seeing 429s last night.”

Streaming, sockets, and the hidden capacity cost

Streaming responses feel instant to users but hold an open connection until the last token. Most providers count the full response against your TPM even if the client disconnects mid-stream. A user who gets impatient and closes a tab doesn’t refund your tokens.

This matters for capacity planning in three ways:

  • Long outputs × long context × many users = silent TPM burn. A 50K-token streamed answer to 1,000 concurrent users is 50M tokens in flight whether they read it or not.
  • Concurrency caps on streaming are tighter. Some providers impose a separate concurrent-streams limit; Perplexity’s Sonar Deep Research is 5–100 RPM precisely because each call holds a connection for ages.[5]
  • Cancel-on-disconnect is on you. If your server keeps streaming after the client closes the socket, you pay for tokens nobody reads. Wire up abort semantics end-to-end.

Max-tokens is your seatbelt. Always set max_output_tokens explicitly. An unbounded generation on a chatty reasoning model can produce 30K+ tokens you never wanted — billed at output rate, and counting against your TPM ceiling. Default to a sensible cap (say 2,000 for chat, 8,000 for reasoning) and raise only where the feature demands it.

  • Concurrency caps. A semaphore limits how many concurrent calls your app makes, turning 429s into queue-and-retry instead of user-facing errors.
  • Token budget per request. Cap max input + max output per feature. Agent tasks with runaway loops are your biggest risk.
  • Cross-provider failover. Keep a secondary provider warm. Cost may jump 20–40 %, but uptime is worth it. Portkey, OpenRouter, and your own hand-rolled fallback all do this.
  • Feature-level degradation. When the big model is flaky, drop streaming, shrink max_output, or serve a cached past response. Tell the user, don’t silently fail.

Do not flood the API with parallel retries. If your app retries 10 times concurrently on a 529, you’re making the overload worse and racing every other customer doing the same thing. One calm retry, then backoff. Anthropic’s docs explicitly call this out.[7]

Requesting a tier bump

When growth outruns auto-tiering, you can ask for a manual increase. Providers look at a short list of signals:

  • Clean history. Low 429 rate, low 5xx rate. If you’re already over-retrying, fix that first.
  • Steady usage pattern. Spiky traffic gets smaller bumps than steady climbs. Together AI’s dynamic tiering literally rewards sustained, consistent load.[6]
  • Deposit or committed spend. Top tiers often require a prepay. Anthropic Tier 4 starts at $400 deposit; OpenAI Tier 5 requires $1,000+ cumulative spend.[1]
  • Business context. Describe your workload, traffic shape, and any PR / launch spikes on the calendar.

Ask before you need it. Typical approval timelines are 3–10 business days. If you have a big launch in a week, it’s already too late. Request a bump as soon as your forecast shows a breach in the next quarter.

Multi-provider capacity: when the complexity pays off

Running across two providers is the cleanest way to double your effective ceiling. It’s also the messiest to maintain — different SDKs, different rate-limit headers, different behavior on streaming, and quality drift you have to monitor weekly.

✅ Multi-provider wins when…

  • You’re close to (or past) a single-provider tier ceiling
  • Downtime costs you real money (SaaS SLAs)
  • You’re already using a router / gateway (OpenRouter, Portkey)
  • Your prompts are portable across providers
  • You run regions where one provider has latency issues

❌ Multi-provider loses when…

  • You lean heavily on provider-specific features (cache_control, JSON mode flavors, tool calling idioms)
  • Team is small and already stretched
  • Your current tier has 5× headroom
  • Eval coverage is thin — quality drift between providers is invisible
Verdict: start by raising your primary tier. Go multi-provider only when your forecast and your reliability needs justify the engineering cost.

🧪 Exercise: Capacity fire-drill

  1. Pick your highest-traffic AI endpoint. Pull the last 7 days of logs.
  2. Compute peak RPS in your busiest 1-minute window.
  3. Compute average tokens-per-request (input + output).
  4. Multiply: peak_RPM × avg_tokens = peak_TPM.
  5. Look up your current provider tier. What’s the TPM ceiling?
  6. Compute the headroom: 1 − (peak_TPM / tier_TPM).
  7. If headroom < 30 %, you’re one viral moment from a 429. Schedule a tier bump or routing change this week.
  8. Repeat the calculation assuming 10× growth. Where does it break?

The production capacity checklist

  • ✅ You know your peak TPM and your current tier’s TPM ceiling.
  • ✅ You have a documented forecast at 3x and 10x growth.
  • ✅ Your app handles 429 and 529 differently (throttle vs. backoff-retry).
  • ✅ Exponential backoff uses jitter, not a fixed delay.
  • ✅ There’s a concurrency cap (semaphore) on outgoing LLM calls.
  • ✅ Token budgets per feature are enforced — agents can’t spiral into runaway loops.
  • ✅ You monitor 429 rate as a first-class metric, like error rate.
  • ✅ You have a failover provider configured (even if unused today).
  • ✅ Tier-bump conversations start 6–12 weeks before projected breach.
