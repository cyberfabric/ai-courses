Chapter 5

Batch APIs: Half Price for Patient Workloads

The flat 50% discount across providers — and how stacking batch + caching gets you to 95% off

Overnight shipping vs next-day air

Anyone who's ever shipped a package knows the rule: if you need it tomorrow morning, pay the premium; if you can wait until end-of-day, it's half the price. FedEx and UPS figured this out decades ago. Every major LLM provider in 2026 has adopted the same two-tier model — real-time and batch. Real-time costs sticker price, responds in seconds. Batch costs exactly half, responds within 24 hours.

The discount is the same on Anthropic, OpenAI, and Google. There's no "cheapest batch provider" to hunt for — they've all converged on 50%. So the question isn't which provider; it's which of your workloads are patient enough to move to batch. If you've never audited your workloads for batch-eligibility, you're probably paying double for 20–60% of your traffic.

The core principle: batch trades latency for cost — linearly, cleanly, no quality difference. If the user isn't waiting on the response in the current HTTP request, you almost always belong on batch.

What a Batch API actually is

You upload a file with many prompts. The provider processes them asynchronously, at their own pace, within 24 hours. You poll for status or receive a webhook. When it's done, you download a file with the answers. That's it. No WebSockets, no long-lived connections, no SSE streams. Just upload → wait → download.

The trade-off is straightforward. Because the provider has 24 hours to fill the order, they can schedule your requests on spare capacity — filling the troughs between the real-time traffic spikes. That's the cost saving they pass back. From your side, nothing about the response changes: same model, same quality, same output. Just delayed.

Every provider uses the same file format: JSONL (one JSON request per line). The schema is essentially the same request body you'd send to the real-time endpoint, plus a custom_id field so you can match responses back to your original request. This makes cross-provider batch migrations surprisingly painless.

Anthropic — Message Batches API

Anthropic's Message Batches API launched in 2024 and has remained unchanged in structure through April 2026: upload a JSONL file of Messages API requests, get a batch ID, poll until it's done, download the output file.1

  • Discount: 50% off both input and output tokens, on every Claude model.
  • SLA: results within 24 hours of submission. Often much faster in practice (often under 2 hours for moderate-sized batches).
  • File format: JSONL, one Messages request per line, with a custom_id per row.
  • Max batch size: 100,000 requests or ~256 MB per batch.
  • Caching stacks: prompt caching still applies inside batch. Cache reads in a batch = 0.1x input × 0.5 batch discount = 5% of normal input cost.

Anthropic is the provider where batch + caching stacking is most dramatic. If your batch workload has any repeated prefix (RAG, common system prompt, fixed few-shot examples), you can routinely pay 5–10% of the real-time rate.2

OpenAI — Batch API

OpenAI's Batch API pioneered the JSONL + Files pattern that everyone else copied. Upload through the Files API, create a batch job referencing the file, poll or use webhooks, download when complete.3

  • Discount: 50% off both input and output, on all supported models (GPT-5.4 family, GPT-5, GPT-4.1, o-series).
  • SLA: 24 hours.
  • Max batch size: 50,000 requests per file.
  • Files API flow: upload JSONL → create batch with the file ID → poll via batches.retrieve → download output and error files when status is completed.
  • Caching stacks: prompt caching still applies inside batch. Cached input tokens in batch = 0.1x input × 0.5 = 5% of list input. Uncached input is 50% of list.
Tip: OpenAI batch jobs can finish much faster than the 24-hour SLA — many overnight jobs complete in 1–4 hours. But never design for that: the SLA is 24h, and during high-demand periods (end-of-quarter, major model launches) your job can genuinely take most of a day. Build the pipeline to handle the worst case, not the best.

Google Gemini — Batch API (Developer & Vertex)

Google offers batch mode through both the Gemini Developer API and Vertex AI. Same core deal, same 50% discount, same 24-hour latency. One important asterisk, though: on Gemini, batch does NOT stack with context caching. The cache discount takes precedence; you get whichever of the two is cheaper for that specific request, not both.4

  • Discount: 50% off both input and output tokens.
  • SLA: "up to 24 hours" — explicitly no guaranteed SLA for batch mode on the Developer API.
  • Caching + batch: do not stack. If a request would hit cache (90% off), you pay the cache rate. If it wouldn't hit cache, you pay the batch rate (50% off). Never both.
  • Input options: Cloud Storage bucket for Vertex; inline JSON for the Developer API; BigQuery for Vertex with structured data.
Watch out: if you're migrating a batch-heavy workload between providers, the non-stacking rule on Gemini can catch you off guard. A workload that costs 5% of list on Anthropic (caching + batch combined) may cost 10% on Gemini (cache-only OR batch-only, whichever applies per request). Always model the real effective rate, not the headline percentages.

When batch fits (and when it doesn't)

The decision is nearly mechanical: is a human waiting on this response in the current HTTP request? If yes, real-time. If no, batch.

Batch-friendly workloads
Overnight evaluation / regression test runs Daily report generation Embedding backfills for new corpora Dataset labeling / classification Translation of static content libraries Summarization of long archives Synthetic data generation for fine-tuning Monthly content audits Email digest generation (send at 6am)
None of these block a user's interaction. They can all wait a few hours.
Not batch-friendly
Chat responses (user is waiting) Live search / autocomplete Agentic loops with tool-calling Real-time content moderation Streaming responses Interactive code completion Anything with a timeout < 1 min Anything driving a UI state transition
Users won't tolerate 4-hour waits; the workflow needs sub-second responses.

A common middle case: an async job triggered by a user action (e.g., "analyze my uploaded PDF"). If your SLA to the user is "we'll email you the result within an hour," that's borderline. Batch usually finishes in under an hour, but you can't rely on it. Either set expectations to 24h or use real-time with a worker queue. Don't pretend batch is "probably fast enough."

The crown jewel — stacking batch + caching

On Anthropic and OpenAI, batch and prompt caching multiply. Do the arithmetic: a cached read on Anthropic Sonnet 4.6 is 0.1x input, which is $0.30 per 1M tokens. Hit that in batch mode and you pay half again — $0.15 per 1M. That's 5% of the $3/M list price.

This is why, for any workload where you can reasonably batch, you almost always want to. Embedding generation, eval runs, synthetic data generation, daily reports — none of these need to be real-time. Almost all of them have heavily repeated prefixes. Almost all of them therefore belong on batch + cache.

Architecture — the patient pipeline

A production batch pipeline has five moving parts. None of them are complicated; all of them are important to get right the first time.

  1. Assemble: build the JSONL file. Each row gets a custom_id — a stable identifier you can match back to your data. Never rely on row ordering; always rely on custom_id.
  2. Submit: upload the file; create the batch; store the batch ID alongside your data in a table or queue. Treat the batch ID as the primary key for subsequent polling.
  3. Poll: check status every few minutes. Most providers now offer webhooks — use them if available, but always keep a poll fallback (webhooks miss, especially on network hiccups).
  4. Download: when status is completed, fetch both the output file and the error file. Error files are non-empty surprisingly often — always handle them.
  5. Reconcile: join output rows back to source data by custom_id. Write idempotently. Requeue failed rows to a real-time or next batch retry path.

Operational gotchas you'll hit

Batch looks simple on paper; the production reality has some corners. Here are the ones that bite teams in their first week.

  • Partial failures are normal. A 100K-row batch with 150 failed rows is expected. Some are malformed prompts, some are transient provider issues. Always parse the error file and have a retry path.
  • Result ordering is not guaranteed. The output JSONL comes back in whatever order the provider finished processing. Always key by custom_id.
  • Rate-limit interactions. Batch submissions still count against your account-level limits on some providers. If you submit 10 batches of 100K each in a minute, you may get throttled. Pace submissions.
  • No streaming responses. Batch doesn't support Server-Sent Events or streaming. If your prompt depends on streaming for long outputs, you'll need to adjust max_tokens or split the work.
  • Idempotency keys don't exist natively. If your orchestrator crashes mid-submit and restarts, you can accidentally submit the same batch twice. Use your own idempotency token (hash the payload + timestamp) before calling the API.
  • Error files can be large. A batch with 10% failures at 100K rows produces a 10K-row error file. Stream-parse it; don't load into memory wholesale.
Warning — cost attribution blind spot: batch invoices land in your billing dashboard as one lumpy line item per batch. If you don't tag your batches (via metadata or a naming convention), you'll have no way to attribute spend back to specific features. Tag every batch with the feature, environment, and requester — future-you will thank present-you.

Graceful degradation — when batch has to become real-time

Sometimes a workload lives on batch 95% of the time but occasionally needs a "right now" execution — a CEO asked for a re-run, a bug was discovered and last night's batch has to be redone before noon. Design for that from the start:

  • Parameterize the execution path. The same business logic should flip between "submit this to batch" and "run this now via the real-time endpoint" with one config.
  • Chunk the real-time fallback. 100K requests fired at the real-time endpoint will eat your rate limit. Split into 500-request chunks with backoff.
  • Compute the real cost of the fallback path. A rerun at real-time rates costs 2x (no batch discount) and sometimes 20x (if it also means busting the cache). Always know this number.
  • Page a human if fallback volume is high. Several batches bouncing to real-time in the same week usually means the batch SLA is genuinely broken and someone needs to call the provider's support.

Exercise: find one workload to move to batch

30-minute audit. Pick one AI feature in your stack and ask:

  1. Who's waiting on the response? If the answer isn't "a user watching a loading spinner," this is a batch candidate.
  2. What's the monthly call volume and token shape? Record (calls/month, avg input tokens, avg output tokens).
  3. Does it have a repeated prefix? System prompt, fixed few-shot, shared RAG context? Write down the prefix token count.
  4. Compute current monthly cost. calls × (input × in_rate + output × out_rate).
  5. Compute batch-only cost. Same formula, with both rates at 50% of list.
  6. Compute batch + caching cost. Prefix tokens × calls × 0.05x rate (batch × cache) + dynamic tokens × calls × 0.5x rate + output × calls × 0.5x rate.
  7. Decide. If batch + cache would save > 40% AND latency tolerance is > 30 min, add "move to batch" to your next sprint.

Bonus: identify three more eligible workloads. Most teams have at least five, and every one of them is leaking money on real-time rates it doesn't need.

The mental reframe

Most teams default to real-time because that's what the "quick-start" docs showed, not because the workload needs it. The instinctual question is: "why would I use batch?" That's the wrong question. The right question is: "why am I using real-time?" If the answer isn't "because a user is waiting," you're paying double for no reason.

  • Every provider offers a flat 50% batch discount in April 2026.
  • Batch + caching stacks on Anthropic and OpenAI (5% of list).
  • Batch + caching does NOT stack on Gemini (best-of, not both).
  • Default question: "why am I paying real-time for this?"
  • JSONL in, JSONL out, keyed by custom_id. Cross-provider plumbing is nearly identical.
  • Always parse error files, reconcile by custom_id, and tag batches for cost attribution.
