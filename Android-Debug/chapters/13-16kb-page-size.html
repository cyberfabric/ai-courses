Chapter 13

Debugging 16 KB Page Size Compatibility for Native Libraries

The Pixel 9 crash signature that’s gating Play Store submissions in 2026

The Crash Class That Came Out of Nowhere

Imagine: your app has shipped to millions of users for years. Stable. Boring. Then a Pixel 9 user files a one-star review — “crashes immediately on launch.” You reproduce it, and there it is in logcat, the line that ruined a thousand evenings:

java.lang.UnsatisfiedLinkError: dlopen failed: "/data/app/.../lib/arm64-v8a/libfoo.so"
    is not page-size compatible. Please rebuild library with new NDK.

You didn’t change anything. The library was built two years ago. Why is it crashing now?

Because Android changed underneath you. Some Android 15+ devices — the ones with more RAM, including the Pixel 8a/9/9a family with the 16 KB developer toggle — switched from the legacy 4 KB memory page size to 16 KB. Native libraries (`.so` files) are loaded with `mmap()` calls that require alignment to the device’s page size. A `.so` aligned to 4 KB cannot be mapped on a device with 16 KB pages. The dynamic linker refuses, and your process dies before `onCreate()` even runs.

Nov 1, 2025 Google Play deadline for 16 KB support on new submissions targeting Android 15+
May 31, 2026 Last automatic Play Console extension for the requirement
4× Page size jump (4096 → 16384 bytes) — the source of the misalignment

This chapter is about debugging that crash class. It’s the most common new-OS-restriction bug of 2026, it’s the easiest to miss in CI (no test you have today exercises 16 KB devices), and most teams hit it through a transitive dependency they didn’t even know was native.

Why It Happens: A 30-Second Mental Model

When the OS loads a `.so` file, it uses `mmap()` to project the file’s ELF segments into memory. `mmap()` can only operate at page-size granularity. If the segments are aligned for 4 KB pages but the OS is using 16 KB pages, the math doesn’t work — you can’t place a 4 KB-aligned segment cleanly on a 16 KB grid. The dynamic linker (`linker64`) detects this and fails with the signature above.

The fix lives in two places:

  • The compiler flags when the `.so` was built. NDK r28+ emits 16 KB-aligned segments by default. Older NDKs need linker flags to override the 4 KB default. (developer.android.com/guide/practices/page-sizes, 2026-04-26)
  • The APK packaging. AGP 8.5.1+ (and certainly AGP 9.x in 2026) zip-aligns the APK’s native libs with the 16 KB stride so the runtime can map them directly out of the APK without copying.

What Claude Code Finds

(1) The crash is pre-`onCreate()`. `loadLibrary` failed during a static initializer in `NativeImageProcessor`, which `MainApplication.onCreate()` indirectly triggers. There is no recoverable point — the process dies on launch. (2) The offending `.so` is `libimage_proc.so`, located at `lib/arm64-v8a/libimage_proc.so` inside the installed APK. (3) The linker warning spells out the math: segment alignment is `0x1000` (4096 bytes), system page size is `0x4000` (16384 bytes). 4 KB doesn’t divide cleanly into 16 KB at the segment offset, so the segment can’t be mapped. (4) The fix is upstream — rebuild `libimage_proc.so` with NDK r28+, or apply the legacy linker flags if you’re stuck on r27. (5) Triage hint: the path tells you whether this is your code (look in `app/src/main/cpp/`) or a dependency (look in any AAR’s `jni/` folder). For `libimage_proc.so` named like this, it’s almost certainly your own native module.

The two lines that matter are the linker warning and the `dlopen failed: unsupported page size 16384` text. AI gives you the diagnosis instantly because this is one of the most pattern-matchable Android crashes that exists — there is exactly one cause.

Catching It Before It Ships: APK Analyzer + Lint

The cheapest diagnosis is the one that runs before users see the crash. Three checks should be wired into every native-using project:

1. APK Analyzer (manual, but instant)

In Android Studio: Build → Analyze APK…, then drill into `lib/arm64-v8a/`. Each `.so` row has an Alignment column. Anything other than “16 KB” is a candidate to fix.

2. The official check_elf_alignment.sh script

# From the NDK or download from Google’s repo:
$ ./check_elf_alignment.sh app-release.apk

Checking APK app-release.apk
  lib/arm64-v8a/libnative-lib.so       ALIGNED   (16 KiB)
  lib/arm64-v8a/libimage_proc.so       UNALIGNED (4 KiB)   <-- BROKEN
  lib/arm64-v8a/libreactnative.so      UNALIGNED (4 KiB)   <-- BROKEN
  lib/arm64-v8a/libopencv_java4.so     ALIGNED   (16 KiB)
  lib/x86_64/libnative-lib.so          ALIGNED   (16 KiB)

3. llvm-objdump for surgical inspection of one library

$ANDROID_HOME/ndk/28.1.13356709/toolchains/llvm/prebuilt/linux-x86_64/bin/llvm-objdump \
    -p libimage_proc.so | grep LOAD

LOAD off 0x0000000000000000 vaddr 0x0000000000000000 align 2**12  # 2^12 = 4096 (BAD)
LOAD off 0x000000000004a000 vaddr 0x000000000004b000 align 2**12  # BAD
Look at the align column. 2**14 = 16384 = good. 2**13 = 8192 or 2**12 = 4096 = needs rebuild. The `check_elf_alignment.sh` script just wraps this `llvm-objdump` check across every `.so` in your APK so you don’t have to script it yourself.

4. zipalign -c -P 16 -v 4 for the packaging side

Even with 16 KB-aligned `.so` files, the APK itself must place them at 16 KB-aligned offsets (uncompressed) so the runtime can `mmap()` directly into the APK. AGP 8.5.1+ does this automatically; verify with:

$ANDROID_HOME/build-tools/35.0.0/zipalign -c -P 16 -v 4 app-release.apk
# Last line: "Verification successful" or it tells you which entries are misaligned.

Create .claude/commands/analyze-page-size.md:

# /analyze-page-size — 16 KB Page Size Audit

Take output from `check_elf_alignment.sh` (or the APK Analyzer)
and produce a structured remediation plan.

## Input
The user pastes the alignment report. Each line has the form:
  <path inside APK>  ALIGNED|UNALIGNED  (4 KiB|16 KiB)

## Analysis Steps
1. **Group findings:** ALIGNED count vs UNALIGNED count, broken
   down by ABI (arm64-v8a, armeabi-v7a, x86_64, x86).

4. **Suggest the cheapest fix** per library: rebuild local, bump
   dependency version, or apply legacy linker flags.

## Output Format
### Alignment Summary
| ABI | Aligned | Unaligned |

### Critical (launch crash) ⚠️
[List of unaligned `.so` files loaded eagerly]

### Non-critical (feature crash)
[List of unaligned `.so` files loaded lazily]

### Remediation Plan
[Per library: source classification + fix + estimated effort]

When the Culprit Is a Transitive Dependency

The hardest 16 KB bugs are the ones you didn’t write. Your APK pulls in a `.so` because some library you depend on bundles native code, and now you’re responsible for shipping it 16 KB-aligned. The frequent offenders in 2026:

Ecosystem What to bump Notes
React Native RN 0.76+ (Hermes & JSC pre-aligned in newer releases) Many third-party RN modules ship their own `.so`; check each one.
Flutter Flutter 3.27+ engine The Flutter engine itself is aligned in recent releases; plugin authors are mixed.
OpenCV (Android SDK) 4.10.0+ Earlier OpenCV releases shipped 4 KB-aligned `libopencv_java*.so`.
SQLCipher / SQLite native SQLCipher 4.6.0+ Android AAR Crypto libs are common offenders — check yours.
Conscrypt 2.5.4+ 2.5.3 was the famous Pixel 8 16 KB crash; bumped quickly. (github.com/google/conscrypt/issues/1413, 2026-04-26)
ML Kit / TFLite TFLite 2.17.0+ Google’s own libs were among the first to align.
Game engines (Unity, Unreal) Unity 6.0 LTS, Unreal 5.5+ Engine-shipped native libs; rebuild the export with the latest engine version.

The diagnostic workflow is the same regardless of source:

  1. Run check_elf_alignment.sh on a release APK.
  2. For each unaligned `.so`, find which AAR or Maven artifact contains it: ./gradlew :app:dependencies | grep -i <lib-name> or unzip the AAR and look in `jni/`.
  3. Check the dependency’s changelog for “16 KB” or “page size” mentions.
  4. If a fixed version exists: bump it. If not: file an issue, and decide whether to rebuild from source, drop the dependency, or use the legacy linker flags as a temporary workaround on a fork.

Rebuilding First-Party Native Code

If the `.so` is yours — built from `app/src/main/cpp/` — the fix is mostly a toolchain bump.

The happy path: NDK r28+ and AGP 8.5.1+

// app/build.gradle.kts
android {
    ndkVersion = "28.1.13356709"  // r28 or newer = 16 KB by default
    // ...
    externalNativeBuild {
        cmake {
            path = file("src/main/cpp/CMakeLists.txt")
            version = "3.22.1+"
        }
    }
}

Rebuild, run the alignment check, you’re done. NDK r28 generates 16 KB-aligned ELF segments by default, and AGP 8.5.1+ zip-aligns the APK’s `lib/` entries with the correct stride. (developer.android.com/guide/practices/page-sizes, 2026-04-26)

Stuck on NDK r27 or older? Linker flags.

If you can’t bump the NDK (CI image lock-in, gradle plugin compat), add the legacy linker flags. CMake:

# CMakeLists.txt
target_link_options(${CMAKE_PROJECT_NAME} PRIVATE
    "-Wl,-z,max-page-size=16384"
    "-Wl,-z,common-page-size=16384"
)

ndk-build (legacy):

# Android.mk
LOCAL_LDFLAGS += -Wl,-z,max-page-size=16384 -Wl,-z,common-page-size=16384
The legacy linker flags only fix the alignment of segments inside the `.so`. They do not fix the APK packaging. If you’re on an old AGP that doesn’t zip-align with a 16 KB stride, the apk-level zipalign -P 16 check will still fail. Bumping AGP is usually less painful than bumping NDK.

Runtime Code: Stop Hardcoding 4096

Even with aligned `.so` files, your C/C++ code may still assume 4 KB pages internally — and that assumption breaks when the OS hands out 16 KB pages.

// ❌ BAD — hardcoded 4 KB assumption
#define PAGE_SIZE 4096
void* mapped = mmap(NULL, PAGE_SIZE, PROT_READ, MAP_PRIVATE, fd, 0);

// ❌ ALSO BAD — magic number
char buffer[4096];

// ✅ GOOD — query the runtime
#include <unistd.h>
long page_size = sysconf(_SC_PAGESIZE);   // POSIX-correct
// or
int page_size = getpagesize();            // BSD-style, equivalent on Linux/Android

NDK r27 and higher actually undefine the PAGE_SIZE macro when 16 KB mode is enabled, so old code that referenced the macro will fail to compile rather than fail at runtime. That’s a feature: it forces the audit before users ever see the crash.

Root Cause Pattern: Buffer Sized to PAGE_SIZE

A common silent bug: a native ring buffer or memory pool sized as a multiple of 4096. On a 16 KB device, your “one-page allocation” is now quarter the size you reasoned about, and code that assumed page-aligned writes overlaps with the next allocation. Symptoms: data corruption, crashes deeper in the stack than the actual bug, sporadic JNI errors. The fix is always: query the page size at runtime; never bake it into a constant.

Testing Against a 16 KB Environment

You have two options. Pick whichever is closer to hand — both are necessary in CI for any team shipping native code.

Option A: Android Studio 16 KB emulator system images

  1. SDK Manager → SDK Platforms → check “Show Package Details.”
  2. Under Android 15 (or higher), select “Google APIs Experimental 16 KB Page Size ARM 64 v8a System Image.”
  3. Create a new AVD using that image.
  4. Boot, then verify: adb shell getconf PAGE_SIZE should print 16384.

Option B: Pixel 8/8a/9/9a developer toggle

On supported Pixels (Pixel 8, 8 Pro, 8a on Android 15 QPR1+; Pixel 9 family on QPR2+; Pixel 9a on Android 16+), open Settings → System → Software updates, find Boot with 16 KB page size, toggle it, and reboot. The device wipes its app data and comes back up with 16 KB pages.

Add a tagged emulator job to CI: spin up the 16 KB AVD, install the release APK, run a smoke test that exercises every code path that touches a `System.loadLibrary` call. If any `.so` is misaligned, it crashes immediately and the job fails. This is the cheapest possible regression gate — it costs you one extra emulator boot per build.

The Decision Matrix: Rebuild, Drop, or Wait

When a transitive dependency is the culprit, you have three options. Most teams pick wrong on the first pass.

Option When to choose Risk
Bump dependency version Upstream has shipped a 16 KB-aligned release. Always your first choice. API breakage in the dependency’s major-version bump.
Rebuild from source locally Open-source dependency, no upstream fix yet, you can vendor it. You now own the security patches for that lib until upstream catches up.
Drop the dependency The lib is not load-bearing; alternatives exist; it’s abandoned. Migration work, possible feature regression.
Wait for upstream Active maintainer, fix in a PR, you can extend Play deadline to May 31, 2026. The deadline is hard. Don’t wait past April 2026 without a fallback plan.
Patch with linker flags on a fork Closed-source binary that won’t be updated; vendor relationship blocking. Highest. You’re distributing a modified binary — check the license.

Fix Pattern: The Two-Line Bump That Saves the Day

// Before:
implementation("com.example.image-processing:imageproc-android:2.3.1")
// alignment check: lib/arm64-v8a/libimageproc.so  UNALIGNED (4 KiB)

// After:
implementation("com.example.image-processing:imageproc-android:2.6.0")
// alignment check: lib/arm64-v8a/libimageproc.so  ALIGNED (16 KiB)

For dependencies that have shipped a fix, this is all that’s needed. The pattern that wastes weeks is teams forking and patching when a version bump on a recently-released dependency would have done it. Always check the changelog first.

Building the /diagnose-page-size-crash Skill

The companion to `/analyze-page-size` (which audits an APK) is `/diagnose-page-size-crash` (which takes a single crash log and tells you what to fix). Create .claude/commands/diagnose-page-size-crash.md:

# /diagnose-page-size-crash — 16 KB Crash Triage

You are given an Android crash log. Determine if it is a 16 KB
page size compatibility crash, and if so, identify the offending
library and suggest a fix.

## Detection Heuristic
A crash is a page-size crash if the log contains ANY of:
  - "is not page-size compatible"
  - "unsupported page size 16384"
  - "LOAD segment alignment (0x1000) is smaller than system page size (0x4000)"
  - dlopen failure mentioning a `.so` with a linker warning above it

## Analysis Steps
1. **Confirm classification.** If the log is NOT a page-size crash,
   say so and route to /analyze-crash instead.
2. **Extract the offending `.so`** path from the dlopen line.
   The path is `/data/app/.../lib/<abi>/<libname>.so`.
3. **Classify source** based on the library name:
   - Looks like a vendor name (e.g., libreactnative, libopencv) → transitive
   - Looks like a project-specific name (e.g., libmyapp_native) → first-party
4. **Pinpoint the load site** from the Java/Kotlin stack:
   the `System.loadLibrary` call that triggered it.
5. **Suggest the fix** in priority order:
   a) For transitive: name the most likely Maven artifact + suggested
      version bump.
   b) For first-party: NDK r28+ rebuild OR linker flags.
6. **Note the user impact:** is this loaded eagerly (launch crash)
   or lazily (feature crash)?

## Output Format
- **Classification:** 16 KB page size crash (confidence: high|medium)
- **Offending library:** <name> (<source: first-party|transitive>)
- **Load site:** <file:line> in Java/Kotlin
- **User impact:** launch crash | feature crash | background crash
- **Recommended fix:** <1–3 sentence remediation>
- **Verification step:** <the exact alignment-check command to confirm>

Now when a teammate pastes a crash, you don’t play guess-the-cause — the skill spits out the library name, the source classification, the most likely fix, and the verification step.

How This Connects to the Rest of the Course

This is the same loop you’ve practiced for 12 chapters: signature → AI diagnosis → root cause → reusable skill → verification. The page-size crash is special only because it’s new (most teams haven’t seen it) and because the fix is almost always upstream (you can’t handle it in your own code). But the muscle memory carries over directly:

  • Use /analyze-crash (Ch 2) for the initial triage if you’re unsure of the class.
  • Use /diagnose-page-size-crash (this chapter) once you’ve confirmed the signature.
  • Use /analyze-page-size (this chapter) on every release APK as a CI gate.
  • Add a 16 KB emulator job to CI — this is the only test that actually exercises the code path.

Exercise: Introduce a Misaligned Native Lib, Then Fix It

  1. Create a tiny native module in a side project. app/src/main/cpp/native-lib.cpp with a single JNI function that returns a string.
  2. Force a 4 KB build: temporarily downgrade ndkVersion to "25.2.9519653" (NDK r25c) in app/build.gradle.kts. Do not add the linker flags.
  3. Build a release APK. Run ./check_elf_alignment.sh app-release.apk. You should see UNALIGNED (4 KiB) for `libnative-lib.so`.
  4. Spin up the 16 KB emulator from the SDK Manager. Install the APK. Launch it. Capture the crash with adb logcat -d -t 200 *:E > crash.log.
  5. Feed the log to your /diagnose-page-size-crash skill. Verify it identifies the offending library and recommends an NDK bump.
  6. Apply the fix: bump ndkVersion to "28.1.13356709" (or newer). Rebuild. Re-run the alignment check. Re-run the app on the 16 KB emulator. The launch crash should be gone.
  7. Bonus: repeat step 6 but instead of bumping NDK, add the legacy linker flags to CMakeLists.txt. Confirm the same outcome — this is the workaround you’ll need when an NDK bump is blocked.

By the end of this exercise, you’ve seen the crash, diagnosed it, and fixed it two different ways — which means the next time it happens to a real shipping app, you know exactly which lever to pull.
