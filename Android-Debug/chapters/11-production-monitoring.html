Chapter 11

Production Monitoring & Crash Reporting

Firebase Crashlytics meets AI — triage, prioritize, and fix production issues at scale

No Debugger, Only Telemetry

Every chapter so far has been about bugs you can reproduce on your machine. Production is different. You can’t attach a debugger. You can’t add a log statement. You can’t ask the user to “try rotating the device.” All you have is what your crash reporting tool sends back.

Chapter 10 fixed your test suite. This chapter tackles the crashes that got past it — the ones hitting real users on real devices in conditions you never imagined.

99.5% crash-free rate threshold for a “healthy” Android app
70% of production crashes come from the top 5 crash clusters
24 hrs target time-to-fix for a P0 production crash

Firebase Crashlytics Overview

Crashlytics is the industry standard for Android crash reporting. If you’re not using it, set it up now — it’s free and takes 10 minutes.

// build.gradle.kts (project level)
plugins {
    id("com.google.gms.google-services") version "4.4.2" apply false
    id("com.google.firebase.crashlytics") version "3.0.7" apply false
}

// build.gradle.kts (app level)
plugins {
    id("com.google.gms.google-services")
    id("com.google.firebase.crashlytics")
}

dependencies {
    implementation(platform("com.google.firebase:firebase-bom:34.12.0"))
    implementation("com.google.firebase:firebase-crashlytics")
    implementation("com.google.firebase:firebase-analytics")
}

Crashlytics automatically captures fatal crashes. But the real power comes from what you add on top: custom keys, breadcrumb logs, non-fatal exception recording, and user identifiers.

Setting Up Rich Crash Context

// Set user identifier (for tracking issues by account)
Firebase.crashlytics.setUserId("user_${userId}")

// Custom keys — add domain-specific context
Firebase.crashlytics.setCustomKey("subscription_tier", "premium")
Firebase.crashlytics.setCustomKey("cart_items_count", cartItems.size)
Firebase.crashlytics.setCustomKey("screen", "CheckoutActivity")
Firebase.crashlytics.setCustomKey("feature_flags", enabledFlags.joinToString(","))

// Breadcrumb logs — trace user journey before crash
Firebase.crashlytics.log("User tapped 'Add to Cart' for product ${product.id}")
Firebase.crashlytics.log("Navigated to CheckoutActivity")
Firebase.crashlytics.log("Payment method selected: credit_card")

// Non-fatal exception recording
try {
    processPayment(order)
} catch (e: PaymentDeclinedException) {
    Firebase.crashlytics.recordException(e)  // Logged, doesn't crash
    showPaymentError(e.userMessage)
}
Custom keys are searchable in the Crashlytics console. Add keys for: current screen, user tier/role, feature flags, and data state (e.g., cart size, logged-in status). When a crash hits, these keys tell you what the user was doing, not just what code failed.

What Claude Code Finds

(1) Regression: 0 events on v4.1.3, 2,847 on v4.2.0 — this crash was introduced in the latest release. (2) The crash happens after payment success — the payment completes, then the app tries to navigate to a confirmation screen via Fragment transaction. (3) checkStateLoss means the Fragment transaction is attempted after onSaveInstanceState — the Activity was already saving state (possibly because the user backgrounded the app during payment). (4) 4-second gap between “Pay Now” tap and crash suggests the user may have switched apps while waiting for the payment API.

Root Cause: Fragment Transaction After State Save

When the user taps “Pay Now” and then switches to another app while the payment processes, Android calls onSaveInstanceState(). When the payment API returns (4 seconds later), the callback tries to commit() a Fragment transaction, but the Activity has already saved state. commit() throws because it could lose the transaction.

This is a v4.2.0 regression because the previous version used commitAllowingStateLoss() or handled the navigation differently.

Fix: Use commitAllowingStateLoss or Lifecycle-Aware Navigation

// Option 1: commitAllowingStateLoss (quick fix)
// Safe when losing the navigation state is acceptable
parentFragmentManager.beginTransaction()
    .replace(R.id.container, ConfirmationFragment())
    .commitAllowingStateLoss()

// Option 2: Lifecycle-aware (better — defer until resumed)
class CheckoutFragment : Fragment() {
    private fun onPaymentSuccess(order: Order) {
        viewLifecycleOwner.lifecycleScope.launch {
            // Wait until the Fragment is at least STARTED
            viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
                // Safe to navigate — Activity is in foreground
                findNavController().navigate(
                    CheckoutFragmentDirections.toConfirmation(order.id)
                )
                return@repeatOnLifecycle
            }
        }
    }
}

Option 2 is the robust fix. repeatOnLifecycle(STARTED) ensures navigation only happens when the Activity is in the foreground. If the user backgrounded the app, navigation is deferred until they return.

App Quality Insights + Gemini: The IDE-Integrated View

The workflow above — copy a Crashlytics cluster from the web console, paste into Claude Code — is the deepest analysis path. But Android Studio also ships a lighter, IDE-integrated counterpart: the App Quality Insights tool window, which now uses Gemini to generate root-cause hints directly next to the offending stacktrace.

🪟
App Quality Insights (View → Tool Windows → App Quality Insights)
Pulls Crashlytics clusters and Android Vitals data into the IDE. Click any frame in the stacktrace to jump to your code. The “Insights” tab uses Gemini to suggest a root cause for the crash without leaving the editor — useful for a fast first read before you escalate to a Claude Code session for the deeper triage and fix-pattern work.

Treat the two as complementary: App Quality Insights is the in-IDE radar (browse the cluster, see the Gemini hint, navigate to the suspect code). Claude Code with the /triage-crash skill below is the deep workbench (severity assessment, regression check, fix-pattern code, recommended action).

Building a /triage-crash Skill

Create .claude/commands/triage-crash.md:

# /triage-crash — Production Crash Triage

Analyze a production crash report from Firebase Crashlytics and
produce a prioritized triage assessment.

## Input
The user will provide a Crashlytics crash cluster summary including:
- Crash count and affected users
- Versions affected
- Stacktrace
- Device distribution
- Custom keys and breadcrumbs (if available)

## Analysis Steps

1. **Assess severity:**
   - Critical: data loss, security issue, payments affected, >5% DAU
   - High: >1% DAU, core feature broken, regression from new release
   - Medium: <1% DAU, edge case, workaround exists
   - Low: cosmetic, rare devices only, non-core feature

2. **Check for regression:** compare affected versions. If the crash
   is new in the latest release, it's a regression — higher priority.

3. **Analyze the stacktrace** using the same techniques from
   /analyze-crash (Caused-by chain, crash classification).

4. **Use custom keys** to understand user context: what screen, what
   data state, what feature flags were active.

5. **Read breadcrumbs** to reconstruct the user journey that led to
   the crash.

6. **Estimate fix complexity:** trivial (one-line), moderate (refactor
   a method), complex (architectural change).

## Output Format

### Severity: [Critical/High/Medium/Low]

### User Impact
[How many users, what they experience, which feature is broken]

### Regression?
[Yes/No — which version introduced it]

### Root Cause
[Why it happens]

### Suggested Fix
```kotlin
[Code]
```

### Fix Complexity: [Trivial/Moderate/Complex]

### Recommended Action
[Hotfix immediately / Fix in next release / Monitor / Investigate more]

Non-Fatal Exception Analysis

Non-fatal exceptions are the silent signals. They don’t crash the app, but they indicate problems: excessive retries, fallback paths being taken, and error states that degrade the experience.

// Track non-fatals that indicate problems:
try {
    val products = api.getProducts()
    displayProducts(products)
} catch (e: SocketTimeoutException) {
    Firebase.crashlytics.recordException(e)
    Firebase.crashlytics.log("Timeout fetching products — showing cached data")
    displayCachedProducts()  // Works, but user gets stale data
}
Prompt
Here are our top 5 non-fatal exceptions from the last week: 1. SocketTimeoutException (api.getProducts) — 4,200 events 2. JsonSyntaxException (parsing /api/v2/promotions) — 890 events 3. SSLHandshakeException (api.example.com) — 340 events 4. IllegalStateException (Room migration) — 120 events 5. FileNotFoundException (loading cached images) — 95 events Which of these indicate real problems vs expected behavior? Prioritize them for investigation.
AI will flag: #1 is high volume and suggests server performance issues. #2 is likely a backend API change. #3 could be certificate rotation. #4 is a known pattern from Chapter 2. #5 is likely benign cache eviction.

Crash Trends and Regression Detection

The most valuable Crashlytics workflow: comparing crash rates across releases to catch regressions early.

Ship v4.2.0 → Monitor 24 hrs → Compare crash rate → Regression? Hotfix

// Feed this to Claude Code:

Monitor every release for 24–48 hours after rollout. Use a staged rollout (1% → 10% → 50% → 100%) so you can halt if crash rates spike. A 1.5% drop in crash-free rate like the example above should trigger an immediate investigation.

Custom Keys for Domain-Specific Analysis

Generic crash reports tell you what crashed. Custom keys tell you why it matters. Teach Claude Code what your custom keys mean:

Prompt
Here's a crash report with our custom keys. Let me explain what they mean for our domain: - subscription_tier: "free" / "premium" / "enterprise" (premium users pay $9.99/mo — crashes for them are higher priority) - cart_items_count: number of items in cart at crash time (high count = user was deep in a purchase flow — lost sale) - feature_flags: comma-separated list of enabled experiments (if a flag appears in most crash events, the experiment might be the cause) Crash data: [paste Crashlytics cluster with custom keys] Is the "new_checkout_flow" feature flag correlated with this crash?
By explaining the business meaning of custom keys, AI can assess business impact — not just technical severity.

The Production Debugging Mindset

Production debugging is about prioritization, not just diagnosis. You can’t fix every crash immediately. Use AI to triage: severity, user impact, regression status, and fix complexity. Fix the crash that affects 1,200 premium users before the one that affects 3 users on Android 8. The /triage-crash skill gives you this framework automatically — severity, impact, and recommended action in 10 minutes instead of 45.

Hands-On: Triage Your Production Crashes

  1. Open Crashlytics: Go to your Firebase Console → Crashlytics. Note your current crash-free rate and the top 3 crash clusters.
  2. Export a cluster: Pick the top crash. Copy: the stacktrace, affected versions, event count, user count, device distribution, and any custom keys/breadcrumbs.
  3. Run /triage-crash: Paste the crash data into Claude Code. Review the severity assessment and fix recommendation. Does AI’s priority match your intuition?
  4. Check for regressions: Filter crashes by version. Are any crashes new to your latest release? If so, these should be your highest priority.
  5. Generate a weekly report: Gather this week’s top 5 crashes and overall stats. Run /weekly-crash-report. Share the output with your team.
  6. Bonus: Add custom keys to your app if you haven’t already. Deploy, wait a week, then re-run the triage — notice how much richer the AI analysis becomes with business context.
