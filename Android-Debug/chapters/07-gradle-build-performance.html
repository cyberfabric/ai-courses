Chapter 7

Gradle Build Performance

Stop waiting for builds — let AI find the bottlenecks hiding in your Gradle configuration

The Build Time Tax

Chapters 1–6 optimized how your app runs. This chapter optimizes how your app builds. Because the fastest code in the world doesn’t help if you spend 4 minutes waiting for Gradle every time you change a line.

Build times are a developer productivity multiplier — or destroyer. A 30-second incremental build keeps you in flow. A 4-minute build sends you to check Slack, lose context, and take 10 minutes to get back on track.

35–50 builds per day for an active Android developer
2+ hrs daily wait time if each build takes 4 minutes
$50K+ annual cost per developer of slow builds (salary + CI)

Capturing Build Data

Before you optimize, you need to know what’s slow. Gradle offers three ways to capture build performance data:

Method 1: --profile (Quick Local Report)

# Generates an HTML report in build/reports/profile/
./gradlew assembleDebug --profile

# The report shows:
# - Configuration phase time
# - Task execution times (sorted by duration)
# - Dependency resolution time

Method 2: --scan (Detailed Cloud Report)

# Uploads a detailed build scan to scans.gradle.com
./gradlew assembleDebug --scan

# Provides:
# - Timeline visualization of all tasks
# - Build cache hit/miss details
# - Dependency download times
# - Configuration phase breakdown
# - Comparison between builds

Method 3: Build Analyzer (Android Studio)

After any build, click Build → Build Analyzer in Android Studio. It shows task impact, warnings about annotation processors, and configuration bottlenecks.

For AI analysis, --profile is the most practical — it generates a local HTML report you can summarize and paste into Claude Code. Build scans are better for team sharing and historical comparison.

What Claude Code Finds

(1) kapt is the dominant bottleneck — three kapt tasks total 80s (35% of build time). These are annotation processor runs, likely Hilt/Dagger, Room, and possibly Moshi/Gson. (2) Configuration phase is slow at 18.2s for 32 projects — configuration cache would nearly eliminate this. (3) Build cache hit rate of 72% is decent but the 18 executed tasks suggest some tasks aren’t cacheable or their inputs keep changing. (4) The single biggest win would be migrating from kapt to KSP, which processes annotations incrementally instead of from scratch.

Root Cause: kapt Runs From Scratch on Every Build

kapt (Kotlin Annotation Processing Tool) delegates to Java’s annotation processing API, which requires a full reprocessing pass on every build. Even changing one file triggers a complete kapt run. For Hilt/Dagger projects, this means regenerating the entire dependency graph every time.

KSP (Kotlin Symbol Processing) is a Kotlin-native replacement that supports incremental processing. It only reprocesses files that actually changed.

Fix: Migrate from kapt to KSP

// build.gradle.kts — BEFORE (kapt):
plugins {
    id("com.google.devtools.ksp") version "2.1.0-1.0.29"  // Add KSP
    // Keep kapt temporarily for libraries that don't support KSP yet
    kotlin("kapt")
}

dependencies {
    // Hilt: kapt → ksp
    // kapt("com.google.dagger:hilt-compiler:2.53")      // Remove
    ksp("com.google.dagger:hilt-compiler:2.53")           // Add

    // Room: kapt → ksp
    // kapt("androidx.room:room-compiler:2.7.0")          // Remove
    ksp("androidx.room:room-compiler:2.7.0")              // Add

    // Moshi: kapt → ksp
    // kapt("com.squareup.moshi:moshi-kotlin-codegen:1.15.1")  // Remove
    ksp("com.squareup.moshi:moshi-kotlin-codegen:1.15.1")      // Add
}

// Once all processors are migrated, remove kapt entirely:
// plugins { kotlin("kapt") }  ← Delete this line

Expected improvement: 40–60% reduction in annotation processing time. kapt runs of 62s typically drop to 15–25s with KSP, and incremental builds are dramatically faster.

Building an /analyze-build Skill

Create .claude/commands/analyze-build.md:

# /analyze-build — Gradle Build Performance Analysis

Analyze Gradle build profile data and suggest optimizations.

## Input
The user will provide one or more of:
- Gradle --profile report summary (task times, cache stats)
- Build scan URL or summary
- build.gradle.kts configuration files
- Gradle properties

## Analysis Steps

1. **Identify the top time consumers** — which tasks take the most
   wall-clock time? Focus on the top 5.

2. **Check annotation processing:**
   - Is kapt still in use? → Recommend KSP migration
   - Which processors are active? (Hilt, Room, Moshi, Glide)
   - Are any processors not KSP-compatible?

3. **Evaluate build cache:**
   - Cache hit rate (target: >85%)
   - Tasks that should be cached but aren't
   - Configuration cache status

4. **Review configuration phase:**
   - Time spent (target: <5s)
   - Number of projects configured
   - Configuration cache opportunity

5. **Check Gradle settings:**
   - Parallel execution enabled?
   - org.gradle.workers.max set appropriately?
   - org.gradle.jvmargs memory allocation
   - Gradle daemon keep-alive

6. **Dependency analysis:**
   - api vs implementation usage (api leaks cause cascade rebuilds)
   - Unused dependencies adding compilation time
   - Version conflicts requiring resolution

[Ranked list with time and percentage of total build]

### Recommended Optimizations
[Ranked by expected impact: High/Medium/Low]

```properties
[Recommended gradle.properties changes]
```

### Migration Steps
[If kapt→KSP or other migrations are recommended]

Common Build Killers

Killer #1: api vs implementation

Using api instead of implementation for dependencies exposes them to all dependent modules. When that dependency changes, everything recompiles.

// BAD: Exposes Retrofit to all modules that depend on :data
// If Retrofit updates, every module recompiles
dependencies {
    api("com.squareup.retrofit2:retrofit:2.11.0")
}

// GOOD: Only :data module sees Retrofit internally
// Changes to Retrofit only recompile :data
dependencies {
    implementation("com.squareup.retrofit2:retrofit:2.11.0")
}
Ask Claude Code to scan your Gradle files: “Review my dependency declarations and identify any api dependencies that should be implementation.” AI can trace which types are actually exposed in your module’s public API.

Killer #2: Disabled Build Cache

# gradle.properties — make sure these are set:
org.gradle.caching=true
org.gradle.configuration-cache=true
org.gradle.parallel=true

The build cache stores task outputs and reuses them when inputs haven’t changed. Configuration cache stores the result of the configuration phase. Together, they can cut incremental build times by 50–80%.

Configuration cache can break plugins that aren’t compatible. Enable it and run a build — Gradle will tell you which plugins need updating. In 2026 it’s Gradle’s preferred execution mode (since Gradle 9.0) — AGP 9.x, KGP, and most mainstream plugins ship with support, and Gradle plans to make it the only supported mode in a future major release.

Killer #3: Wrong JVM Arguments

# gradle.properties — default is often too low for Android builds:

# BAD: Default or too-small heap
org.gradle.jvmargs=-Xmx512m

# GOOD: Sized for a typical Android project
org.gradle.jvmargs=-Xmx4g -XX:+UseParallelGC -XX:MaxMetaspaceSize=512m

# For larger multi-module projects:
org.gradle.jvmargs=-Xmx6g -XX:+UseParallelGC -XX:MaxMetaspaceSize=1g

If the Gradle daemon runs out of heap, it triggers GC pauses that stall the build. Too much heap wastes memory. Feed your gradle.properties to Claude Code along with your project size (number of modules, total Kotlin files) for a tailored recommendation.

Dependency Analysis with AI

Unused dependencies add compilation time and bloat your APK. Claude Code can audit your dependency tree:

Prompt
Here's the dependency block from my app module's build.gradle.kts. I suspect some of these aren't used. Can you identify likely unused dependencies based on common usage patterns? [paste dependencies block] My app uses: Compose UI, Hilt, Room, Retrofit, Coil for images. We removed the XML layout system 6 months ago.
The mention of removing XML layouts tells AI to flag any remaining View-system dependencies (AppCompat widgets, RecyclerView, ConstraintLayout) as likely unused.

Modularization for Build Speed

A single-module app recompiles everything on every change. A well-modularized app only recompiles the module you changed and anything that depends on it.

Change in :feature-cart → Recompile :feature-cart → Relink :app → Done

vs. a monolith:

Change anywhere → Recompile everything → Relink everything → Done (slowly)

Claude Code can suggest module boundaries. Paste your project’s package structure and ask: “Based on this package layout, what module boundaries would give me the best build speed improvement?”

Module Structure
app/ ├── data/ ├── domain/ ├── ui/ │ ├── home/ │ ├── cart/ │ ├── profile/ │ └── checkout/ └── di/ One module. 847 Kotlin files.
Every change recompiles all 847 files. Incremental compilation helps, but kapt/KSP still processes everything.
Module Structure
:app (shell, navigation) :core:data (repositories, DB) :core:domain (use cases, models) :core:network (Retrofit, API) :core:ui (shared composables) :feature:home :feature:cart :feature:profile :feature:checkout 9 modules. Longest recompile: ~180 files.
Changes in :feature:cart only recompile that module. Core modules change rarely, so their cache stays warm.

CI-Specific Optimizations

Local builds and CI builds have different bottlenecks. CI often starts with a cold cache and no Gradle daemon.

# CI-specific gradle.properties:
org.gradle.caching=true
org.gradle.parallel=true
org.gradle.daemon=false  # CI: no daemon (fresh JVM each time)
org.gradle.jvmargs=-Xmx6g -XX:+UseParallelGC

# Remote build cache (share cache across CI runs and developers):
# settings.gradle.kts:
buildCache {
    remote<HttpBuildCache> {
        url = uri("https://your-cache-server.com/cache/")
        isPush = System.getenv("CI") != null  // Only CI pushes to cache
        credentials {
            username = System.getenv("CACHE_USER") ?: ""
            password = System.getenv("CACHE_PASS") ?: ""
        }
    }
}
A remote build cache is the single biggest CI optimization. When one CI run builds a task, the output is cached remotely. The next CI run (and local developers) can download the cached output instead of rebuilding. This can cut CI times by 40–70%.

Your Build Performance Checklist

  • Run ./gradlew assembleDebug --profile and identify top 5 slowest tasks
  • Migrate from kapt to KSP for all compatible annotation processors
  • Enable configuration cache (org.gradle.configuration-cache=true)
  • Enable build cache (org.gradle.caching=true)
  • Set JVM args appropriately (-Xmx4g minimum for Android projects)
  • Audit api vs implementation — minimize api usage
  • Remove unused dependencies
  • Consider modularization if your app has 300+ Kotlin files in one module
  • Set up a remote build cache for CI

The Build Speed Mindset

Every minute of build time is a minute of broken flow state. A 22-second incremental build lets you stay in the zone. A 2-minute build sends you to a browser tab. The optimizations in this chapter are a one-time investment that pays back every single day, for every developer on your team. Run the profile, feed it to Claude Code, and fix the top 3 bottlenecks. You’ll never look back.

Hands-On: Profile and Optimize Your Build

  1. Capture a baseline: Run ./gradlew clean assembleDebug --profile. Open the report in build/reports/profile/.
  2. Extract the summary: Copy the top 10 tasks by execution time, the configuration phase time, and the build cache hit rate.
  3. Run /analyze-build: Paste the summary into Claude Code. Also paste your gradle.properties and the plugins + dependencies blocks from your app’s build.gradle.kts.
  4. Apply the top recommendation: If AI says “migrate kapt to KSP”, do the Hilt and Room migrations first (they’re the biggest wins).
  5. Measure again: Run ./gradlew assembleDebug --profile (without clean) to measure incremental build time. Compare to your baseline.
  6. Bonus: Enable configuration cache, rebuild, and check if the configuration phase drops to under 1 second.
