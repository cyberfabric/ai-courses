Flaky Test Diagnosis

Systematically diagnosing tests that pass sometimes and fail other times

The Retry-and-Ignore Culture

Chapter 9 gave you profiling superpowers for runtime performance. This chapter tackles a different kind of performance problem: your test suite’s reliability. Because an unreliable test suite is worse than no test suite at all.

You know the pattern. CI goes red. You check the failure. It’s that test again — the one that fails once every 5 runs. You click “Retry.” It passes. You merge. And slowly, your team stops trusting the test suite entirely.

15–25% of test suites in Android projects contain at least one flaky test
4.2 hrs average weekly developer time wasted on flaky test investigation
3 retries typical CI configuration to “work around” flaky tests

Every retry costs CI minutes. Every ignored failure is a real bug waiting to slip through. AI can break this cycle by categorizing the flakiness pattern and pointing to the root cause.

The Four Categories of Flaky Tests

Flaky tests aren’t random. They fall into categories, and each category has a different fix:

Category Symptom Typical Cause
Timing-dependent Fails on slow machines, passes on fast ones Missing advanceUntilIdle(), hardcoded delays, race conditions
Order-dependent Fails when run after a specific other test Shared mutable state between tests, missing cleanup
Environment-dependent Passes locally, fails on CI Timezone, locale, emulator speed, screen density differences
Resource-dependent Fails intermittently under load Database locks, file handles, network ports, memory pressure
When you feed a flaky test failure to Claude Code, always include: the test name, the assertion that failed, the stacktrace, and how often it fails (e.g., “fails 1 in 5 runs”). The failure frequency is a strong signal for which category it falls into.

Timing Flakiness: Coroutine Tests

This is the #1 source of flaky Android unit tests. Coroutines are asynchronous, but tests often expect synchronous results.

Flaky Test Failure Log

CartViewModelTest > addToCart_updatesCartCount FAILED
    java.lang.AssertionError:
    Expected: 1
    Actual:   0
    at com.example.shopapp.ui.cart.CartViewModelTest
        .addToCart_updatesCartCount(CartViewModelTest.kt:47)

    // This test passes ~80% of the time locally,
    // but fails ~40% on CI.

The Test Code

@Test
fun addToCart_updatesCartCount() = runTest {
    val viewModel = CartViewModel(fakeRepository)

    viewModel.addToCart(testProduct)

    // FLAKY: The coroutine inside addToCart might not
    // have completed by the time we assert
    assertEquals(1, viewModel.cartCount.value)
}

What Claude Code Finds

Category: Timing-dependent. Problem: addToCart() launches a coroutine internally, but the test asserts immediately after calling it. On a fast machine, the coroutine completes before the assertion. On a slow CI emulator, it doesn’t. The 80%/40% split between local and CI is the classic signature of a timing-dependent flaky test.

Root Cause: Missing advanceUntilIdle() After Coroutine Launch

runTest provides a TestCoroutineScheduler that controls coroutine execution, but only if the coroutines use the test dispatcher. When CartViewModel.addToCart() launches a coroutine in viewModelScope, that coroutine runs on Dispatchers.Main, which runTest overrides — but the test still needs to explicitly advance the scheduler to let the coroutine complete.

Fix: Properly Advance the Test Dispatcher

// Option 1: advanceUntilIdle() — let all pending coroutines complete
@Test
fun addToCart_updatesCartCount() = runTest {
    val viewModel = CartViewModel(fakeRepository)

    viewModel.addToCart(testProduct)
    advanceUntilIdle()  // Wait for all coroutines to finish

    assertEquals(1, viewModel.cartCount.value)
}

// Option 2: Turbine for Flow testing (more precise)
@Test
fun addToCart_updatesCartCount() = runTest {
    val viewModel = CartViewModel(fakeRepository)

    viewModel.cartCount.test {
        assertEquals(0, awaitItem())        // Initial state
        viewModel.addToCart(testProduct)
        assertEquals(1, awaitItem())        // After add
    }
}

// IMPORTANT: Set up the test dispatcher in your test class:
@OptIn(ExperimentalCoroutinesApi::class)
class CartViewModelTest {
    @get:Rule
    val mainDispatcherRule = MainDispatcherRule()
    // MainDispatcherRule replaces Dispatchers.Main with a TestDispatcher
}

The MainDispatcherRule is critical. Without it, viewModelScope uses the real Dispatchers.Main, which doesn’t exist in unit tests. Here’s the rule:

class MainDispatcherRule(
    val dispatcher: TestDispatcher = UnconfinedTestDispatcher()
) : TestWatcher() {
    override fun starting(description: Description) {
        Dispatchers.setMain(dispatcher)
    }
    override fun finished(description: Description) {
        Dispatchers.resetMain()
    }
}

Building a /diagnose-flaky Skill

Create .claude/commands/diagnose-flaky.md:

# /diagnose-flaky — Flaky Test Diagnosis

Analyze a flaky test failure and identify the root cause.

## Input
The user will provide:
- Test name and file
- Assertion failure message and stacktrace
- Failure frequency (e.g., "fails 1 in 5 runs")
- Whether it fails locally, on CI, or both
- The test source code (if available)

## Analysis Steps

1. **Categorize the flakiness:**
   - Timing: fails more on slow machines → coroutine/delay issue
   - Order: fails only when run with other tests → shared state
   - Environment: fails only on CI → timezone/locale/emulator
   - Resource: fails under load → DB/file/network contention

2. **For timing issues, check:**
   - Is runTest used? Is advanceUntilIdle() called?
   - Is MainDispatcherRule set up?
   - Are there hardcoded delays (Thread.sleep, delay)?
   - Are Flows collected properly (Turbine)?

3. **For order issues, check:**
   - Static/companion object state between tests
   - Database not cleared between tests
   - Shared CoroutineScope or TestDispatcher

4. **For environment issues, check:**
   - Date/time assertions sensitive to timezone
   - String comparisons sensitive to locale
   - Pixel-based UI assertions sensitive to screen density
   - Animation timing on slow emulators

5. **For resource issues, check:**
   - Database accessed from multiple test threads
   - File locks not released
   - In-memory test database vs persistent

## Output Format

### Flakiness Category
[Timing / Order / Environment / Resource]

### Root Cause
[Specific explanation]

### Fix
```kotlin
[Code]
```

### Confidence
[High/Medium/Low — based on how well the symptoms match]

Compose UI Test Flakiness

Compose UI tests have their own synchronization model. The test framework waits for composition to settle, but it can’t always know when your async data loading is done.

// FLAKY: Assert immediately — data might not be loaded yet
@Test
fun productList_showsProducts() {
    composeTestRule.setContent {
        ProductListScreen(viewModel = testViewModel)
    }
    // If the ViewModel loads data asynchronously,
    // this might run before the list renders:
    composeTestRule.onNodeWithText("Running Shoes").assertExists()
}

// FIXED: Wait for the content to appear
@Test
fun productList_showsProducts() {
    composeTestRule.setContent {
        ProductListScreen(viewModel = testViewModel)
    }
    composeTestRule.waitUntilExactlyOneExists(
        hasText("Running Shoes"),
        timeoutMillis = 5_000
    )
}
Never use Thread.sleep() in UI tests. It’s the most common source of flakiness. If you need 2 seconds today, you’ll need 5 on a slow CI emulator, and 10 when the server is slow. Use waitUntil, waitUntilExactlyOneExists, or IdlingResource instead.

Espresso Test Flakiness

If you still have Espresso tests (View-based UI), the most common flakiness source is missing IdlingResource registration.

// The problem: Espresso doesn't know about your custom async work
// It thinks the UI is idle when it's actually loading data

// The fix: Register an IdlingResource for your async operations
class OkHttpIdlingResource(private val client: OkHttpClient) : IdlingResource {
    private var callback: IdlingResource.ResourceCallback? = null

    override fun getName() = "OkHttp"

    override fun isIdleNow(): Boolean {
        val idle = client.dispatcher.runningCallsCount() == 0
        if (idle) callback?.onTransitionToIdle()
        return idle
    }

    override fun registerIdleTransitionCallback(callback: ResourceCallback) {
        this.callback = callback
    }
}

// In your test setup:
@Before
fun setup() {
    val resource = OkHttpIdlingResource(okHttpClient)
    IdlingRegistry.getInstance().register(resource)
}

@After
fun teardown() {
    IdlingRegistry.getInstance().unregister(resource)
}

Order-Dependent Tests: Shared State

When tests share mutable state, they work in isolation but fail when run together. AI is excellent at spotting this because it can scan the test class for static state.

Prompt
This test passes when run alone but fails when the full test suite runs. It seems to fail specifically after UserRepositoryTest runs. Here's the test: [paste test code] And here's the test class setup: [paste @Before/@After and companion object] The test uses an in-memory Room database.
AI will check: is the database instance shared via a companion object? Is @After clearing all data? Is the TestDispatcher shared between test classes?
// COMMON BUG: Database shared between tests via singleton
companion object {
    val db = Room.inMemoryDatabaseBuilder(
        ApplicationProvider.getApplicationContext(),
        AppDatabase::class.java
    ).build()  // Same instance used by ALL tests in this class!
}

// FIX: New database per test
@get:Rule
val instantTaskRule = InstantTaskExecutorRule()

private lateinit var db: AppDatabase

@Before
fun setup() {
    db = Room.inMemoryDatabaseBuilder(
        ApplicationProvider.getApplicationContext(),
        AppDatabase::class.java
    ).allowMainThreadQueries()  // OK for tests
     .build()
}

@After
fun teardown() {
    db.close()  // Critical: release the database
}

CI-Specific Failures

Tests passing locally but failing on CI is infuriating. The differences are usually subtle:

Difference How It Causes Flakiness Fix
Emulator speed Animations and async operations take longer Disable animations: adb shell settings put global window_animation_scale 0
Timezone Date assertions fail (CI is UTC, you’re UTC+2) Use Clock abstraction, test with fixed timezone
Locale Number/date formatting differs (comma vs period decimal) Set locale explicitly in tests: Locale.setDefault(Locale.US)
Screen density Screenshot tests or pixel-based assertions fail Use semantic matchers (hasText) not pixel positions
Parallel execution Tests compete for shared resources (DB, files, ports) Use unique names per test (testName.methodName)
Add this to your CI emulator setup script to eliminate the most common CI-specific flakiness sources: 
adb shell settings put global window_animation_scale 0
adb shell settings put global transition_animation_scale 0
adb shell settings put global animator_duration_scale 0
adb shell settings put system time_12_24 24
adb shell settings put system date_format "yyyy-MM-dd"

Your Flaky Test Checklist

  • Every coroutine test uses runTest with advanceUntilIdle()
  • MainDispatcherRule is set up in all ViewModel test classes
  • No Thread.sleep() in UI tests — use waitUntil or IdlingResource
  • Each test creates its own database instance, closed in @After
  • No mutable state in companion object blocks shared between tests
  • CI emulator has animations disabled
  • Date/time tests use a fixed Clock, not System.currentTimeMillis()
  • Locale-sensitive tests set Locale explicitly

The Flaky Test Mindset

A flaky test is not “random.” It is a deterministic bug in your test that only manifests under certain conditions. Treat it like any other bug: categorize it, find the root cause, and fix it. Never retry your way past a flaky test — every retry is a ticking time bomb. Feed the failure log and test code to Claude Code, let AI categorize the flakiness, and apply the appropriate fix pattern.

Hands-On: Find and Fix Your Team’s Flakiest Test

  1. Identify the culprit: Check your CI dashboard for the test with the most retries or failures in the last month. If your CI doesn’t track this, run your test suite 10 times and note which tests fail inconsistently.
  2. Collect the evidence: Get the failure log (assertion message + stacktrace), the test source code, and note the failure frequency and whether it’s local, CI, or both.
  3. Run /diagnose-flaky: Paste everything into Claude Code. Let AI categorize the flakiness and suggest a fix.
  4. Apply the fix: Implement Claude Code’s suggestion. The fix will likely be one of: add advanceUntilIdle(), replace Thread.sleep with waitUntil, isolate shared state, or fix a timezone assumption.
  5. Verify: Run the test 20 times locally (./gradlew test --tests "YourTest" --rerun) and confirm it passes every time.
  6. Bonus: Scan all your test classes for Thread.sleep, companion object with mutable state, and missing @After cleanup. Feed the results to Claude Code for a bulk flakiness audit.
