Chapter 5

Compose Recomposition Performance

When your composables redraw 50 times during a scroll and you wonder where the jank comes from

The Invisible Performance Tax

Chapter 4 dealt with memory leaks — objects that silently accumulate. Recomposition issues are the CPU equivalent: your composables silently re-execute dozens of times when they should execute once, burning battery and dropping frames.

In the View system, a layout only redraws when you explicitly call invalidate() or change a property. In Compose, any state change can trigger recomposition of any composable that reads that state. This is powerful — and dangerous.

16 ms budget per frame at 60 FPS — exceed it and you drop frames
10–100x recomposition overhead from common Compose mistakes
#1 Compose performance issue: unnecessary recomposition

The Recomposition Mental Model

Think of recomposition like re-rendering a web page. If every keystroke in a search box re-rendered the entire page instead of just the search results, your browser would grind to a halt. Compose works the same way — it should only recompose the composables that actually depend on the changed state.

Recomposition is triggered when:

  • A State, MutableState, StateFlow, or other observable value changes
  • A composable reads that state (directly or indirectly)
  • Compose decides the composable’s inputs have changed (parameter equality check)

The problem arises when Compose thinks inputs have changed even though the meaningful data hasn’t. This happens because of unstable types, lambda allocations, and state read scoping mistakes.

Strong Skipping Mode (The Modern Default)

Before you go hunting for unstable parameters, know what the compiler already does for you. Since Kotlin 2.0.20 (Compose Compiler 1.5.4 introduced it; 2.0.20 made it default), strong skipping mode is on by default for every Compose project on a current Kotlin toolchain. It changes the rules you may have learned a couple of years ago.

What strong skipping does:

  • All restartable composables become skippable — even ones with unstable parameters. The old “restartable but NOT skippable” verdict in the compiler report mostly goes away.
  • Lambdas inside composables are auto-memoized. The compiler wraps them in remember with their captured values as keys. You no longer need to manually remember { { ... } } to stabilize a callback.
  • Equality semantics shift: stable parameters are compared with object equality (.equals()); unstable parameters are compared with instance equality (===).
// What the compiler does for you under strong skipping:
@Composable
fun ProductCard(product: Product, onClick: () -> Unit) {
    // 'onClick' is auto-memoized — same instance across recompositions
    // unless its captured values change.
    Card(onClick = onClick) { Text(product.name) }
}

So why do @Stable and @Immutable still matter? Because they switch the comparison from === (instance) to .equals() (content). For data flowing from Room, networking, or any source that reallocates objects on every emission, instance equality always reports “different” and the composable recomposes anyway. Annotating the type as @Immutable lets Compose see “same content → safe to skip.”

If you need to opt out for a specific case, use @NonSkippableComposable on a composable to disable skipping entirely, or @DontMemoize on a lambda to disable auto-memoization.

The trap patterns below still matter — strong skipping reduces but doesn’t eliminate the cost of unstable types and re-allocated lambdas. Treat them as the second optimization tier you reach for after profiling tells you a screen is still janky.

Measuring Recomposition

Before optimizing, you need data. Here’s how to see recomposition counts:

Method 1: Layout Inspector

Android Studio’s Layout Inspector shows recomposition counts in real-time:

  1. Run your app in debug mode
  2. Open Tools → Layout Inspector
  3. Enable “Show Recomposition Counts” in the toolbar
  4. Interact with the app — watch the numbers climb

Method 2: Compose Compiler Metrics

Since Kotlin 2.0, the Compose Compiler ships as the org.jetbrains.kotlin.plugin.compose Gradle plugin (versioned in lockstep with Kotlin), and metrics are configured through a dedicated DSL block instead of raw -P flags. Apply the plugin and add a composeCompiler block to your build.gradle.kts:

// app/build.gradle.kts
plugins {
    id("org.jetbrains.kotlin.plugin.compose")
}

composeCompiler {
    reportsDestination = layout.buildDirectory.dir("compose_compiler")
    metricsDestination = layout.buildDirectory.dir("compose_compiler")
}

After building, check app/build/compose_compiler/ for files like *-composables.txt and *-classes.txt. These show which composables are restartable, skippable, and which classes are stable or unstable.

Feed the Compose compiler metrics files directly to Claude Code. Ask: “Which composables in this report are not skippable, and what’s making their parameters unstable?” AI can parse these reports much faster than you can scan them manually.

The Top Recomposition Traps

AI catches these patterns instantly because they appear in almost every Compose codebase. Let’s walk through each one.

Before diving in: with strong skipping on, Compose will often skip these composables anyway, so you may never see the “NOT skippable” verdict the old reports complained about. The traps still cost CPU though — unstable parameters force === instance-equality checks (which a freshly allocated list or DTO will always fail), and re-allocated lambda captures invalidate the auto-remember keys the compiler generated. The same patterns are also exactly what you must fix if you opt out of skipping on a particular composable with @NonSkippableComposable. Read on for the fixes.

Trap 1: Unstable Parameters

Compose skips recomposition if a composable’s parameters haven’t changed. But it can only do this if the parameters are stable — meaning Compose can verify equality. Lists, non-data classes, and classes from external modules are often unstable.

Compose Compiler Report: Unstable Parameters

restartable but NOT skippable composable fun ProductCard(
  unstable product: Product,
  unstable onAddToCart: Function1<Product, Unit>
)

// From classes.txt:
unstable class Product {
  stable val id: String
  stable val name: String
  stable val price: Double
  unstable val tags: List<String>    // <-- List is unstable!
  stable val imageUrl: String
}

What Claude Code Finds

Problem: ProductCard is “restartable but NOT skippable.” This means every time the parent recomposes, ProductCard will re-execute — even if the product data hasn’t changed. Cause: The Product class has a List<String> property (tags), and List is an interface that Compose treats as unstable (it could be mutable). Also, the onAddToCart lambda is unstable because it’s a new allocation on each parent recomposition.

Root Cause: List Interface + Lambda Allocation = No Skipping

Compose’s stability system can’t guarantee that a List<String> hasn’t been mutated between recompositions (it could be a MutableList behind the interface). So it conservatively marks Product as unstable. Combined with a lambda parameter that’s re-allocated on every parent recomposition, Compose never skips ProductCard.

In a LazyColumn with 50 product cards, this means all 50 recompose on every state change in the parent — even if only a search bar text changed.

Fix: Make Parameters Stable

// Fix 1: Use @Immutable and ImmutableList from kotlinx.collections
@Immutable
data class Product(
    val id: String,
    val name: String,
    val price: Double,
    val tags: ImmutableList<String>,  // kotlinx.collections.immutable
    val imageUrl: String
)

// Fix 2: Stabilize the lambda with remember
@Composable
fun ProductList(viewModel: ProductViewModel = hiltViewModel()) {
    val products by viewModel.products.collectAsState()

    // Stabilize the lambda — same reference across recompositions
    val onAddToCart = remember(viewModel) {
        { product: Product -> viewModel.addToCart(product) }
    }

    LazyColumn {
        items(products, key = { it.id }) { product ->
            ProductCard(product = product, onAddToCart = onAddToCart)
        }
    }
}

// Now ProductCard IS skippable — Compose can skip it when
// the product and lambda haven't changed.

Key changes: (1) Replace List with ImmutableList from kotlinx-collections-immutable so Compose knows it can’t mutate. (2) Wrap the lambda in remember so the same reference is passed across recompositions. (3) Use key in the LazyColumn so Compose can track individual items.

Trap 2: Reading State in the Wrong Scope

// BAD: Reading state in the parent forces recomposition of everything
@Composable
fun SearchScreen(viewModel: SearchViewModel = hiltViewModel()) {
    val scrollState = viewModel.scrollPosition.collectAsState()

    Column {
        SearchBar(...)
        // scrollState is read HERE, in SearchScreen's scope.
        // Any scroll change recomposes SearchScreen AND SearchBar.
        ProductList(scrollOffset = scrollState.value)
    }
}

// GOOD: Read state only where it's needed
@Composable
fun SearchScreen(viewModel: SearchViewModel = hiltViewModel()) {
    Column {
        SearchBar(...)
        // Move state reading INTO ProductList
        ProductList(viewModel)
    }
}

@Composable
fun ProductList(viewModel: SearchViewModel) {
    val scrollState by viewModel.scrollPosition.collectAsState()
    // Now only ProductList recomposes on scroll changes.
    // SearchBar is untouched.
    LazyColumn(state = rememberLazyListState(scrollState)) { ... }
}
The rule: read state as late as possible, in the narrowest composable scope. The composable that reads the state is the one that recomposes. Push state reads down the tree.

Trap 3: Missing remember

// BAD: Creates a new DateFormatter on EVERY recomposition
@Composable
fun OrderDate(timestamp: Long) {
    val formatter = SimpleDateFormat("MMM dd, yyyy", Locale.US)
    Text(formatter.format(Date(timestamp)))
}

// GOOD: Create once, reuse across recompositions
@Composable
fun OrderDate(timestamp: Long) {
    val formatter = remember {
        SimpleDateFormat("MMM dd, yyyy", Locale.US)
    }
    Text(formatter.format(Date(timestamp)))
}

Without remember, expensive objects are recreated on every recomposition. For a SimpleDateFormat this is mild. For a regex pattern, a JSON parser, or a bitmap decoder, it can be devastating.

Building an /analyze-recomposition Skill

Create .claude/commands/analyze-recomposition.md:

# /analyze-recomposition — Compose Recomposition Analysis

Analyze Compose recomposition issues from metrics, Layout Inspector
data, or code review.

## Input
The user will provide one or more of:
- Compose compiler metrics (*-composables.txt, *-classes.txt)
- Recomposition counts from Layout Inspector
- Composable source code with performance concerns

## Analysis Steps

1. **Identify non-skippable composables** — any composable marked
   "restartable but NOT skippable" is recomposing unnecessarily.
   Note whether strong skipping is enabled (Kotlin 2.0.20+ default):
   under strong skipping, the relevant signal is unstable parameters
   that hurt skip *quality* (forcing `===` instance equality on
   freshly allocated objects), not skip *occurrence*.

2. **Find unstable parameters** (refinement layer over strong
   skipping — still actionable because instance equality on
   reallocated data ≈ "never skip"):
   - List, Map, Set interfaces (use kotlinx ImmutableList etc.)
   - Non-data classes without @Stable or @Immutable
   - Classes from external modules (Compose can't verify stability)
   - Lambda captures that change every recomposition (auto-memoize
     keys invalidate)

3. **Check state read scoping** — is state being read higher in the
   tree than necessary? Moving reads lower reduces blast radius.

4. **Check for missing remember** — expensive object creation inside
   composable bodies without remember.

5. **LazyColumn/LazyRow audit:**
   - Missing key parameter (forces recomposition of all items)
   - Heavy composables in item blocks
   - Nested scrolling containers

[List with reason for each]

### Unstable Types
[Classes that need @Immutable or property changes]

### State Scoping Issues
[State reads that should be moved lower]

### Optimization Suggestions
[Ranked by impact: High/Medium/Low]

### Code Fixes
```kotlin
[Specific fixes]
```

Advanced: derivedStateOf and snapshotFlow

Two power tools that AI often recommends for recomposition optimization:

derivedStateOf — Computed State That Filters Recomposition

// Without derivedStateOf: recomposes on EVERY scroll pixel
@Composable
fun ScrollIndicator(listState: LazyListState) {
    val showButton = listState.firstVisibleItemIndex > 0
    // This recomposes every time firstVisibleItemIndex changes:
    // 0→1, 1→2, 2→3... each triggers recomposition
    if (showButton) { ScrollToTopButton() }
}

// With derivedStateOf: only recomposes when the derived value changes
@Composable
fun ScrollIndicator(listState: LazyListState) {
    val showButton by remember {
        derivedStateOf { listState.firstVisibleItemIndex > 0 }
    }
    // Only recomposes when showButton flips: false→true or true→false
    // Not on every index change. Massive reduction.
    if (showButton) { ScrollToTopButton() }
}

snapshotFlow — Bridge Compose State to Coroutine Flows

// Useful when you need to react to state changes with side effects
@Composable
fun AnalyticsTracker(listState: LazyListState) {
    LaunchedEffect(listState) {
        snapshotFlow { listState.firstVisibleItemIndex }
            .distinctUntilChanged()
            .filter { it > 10 }
            .collect { index ->
                analytics.trackScroll(index)  // Only fires when needed
            }
    }
}
Use derivedStateOf when you want to reduce recomposition by computing a simpler value from frequently-changing state. Use snapshotFlow when you need to perform side effects (analytics, logging, network calls) in response to state changes.

Lazy List Performance

LazyColumn and LazyRow are where recomposition issues hurt the most — because they deal with many items and scrolling demands consistent 60 FPS.

The key Parameter

// BAD: No key — Compose tracks items by position
LazyColumn {
    items(products) { product ->
        ProductCard(product)
    }
}
// If a product is added at the top, EVERY item recomposes
// because their positions all shifted.

// GOOD: Key by unique ID — Compose tracks items by identity
LazyColumn {
    items(products, key = { it.id }) { product ->
        ProductCard(product)
    }
}
// Now only the new item composes. Existing items keep their state.

Heavy Item Composables

Keep item composables lightweight. If a ProductCard loads images, formats prices, and parses markdown descriptions on every recomposition, your scroll will jank.

// Move expensive operations into remember or the ViewModel
@Composable
fun ProductCard(product: Product) {
    val formattedPrice = remember(product.price) {
        NumberFormat.getCurrencyInstance().format(product.price)
    }
    // formattedPrice is only recalculated when price actually changes
}

Baseline Profiles

Baseline Profiles pre-compile your critical code paths so they run as optimized machine code from the first app launch, instead of being interpreted by the JIT compiler.

// Add to your app's build.gradle.kts:
dependencies {
    implementation("androidx.profileinstaller:profileinstaller:1.4.1")
    baselineProfile(project(":baselineprofile"))
}

// Create a :baselineprofile module with a test that exercises
// your critical UI paths:
@RunWith(AndroidJUnit4::class)
class BaselineProfileGenerator {
    @get:Rule
    val rule = BaselineProfileRule()

    @Test
    fun generateBaselineProfile() = rule.collect("com.example.myapp") {
        pressHome()
        startActivityAndWait()
        // Navigate through critical screens
        device.findObject(By.text("Products")).click()
        device.waitForIdle()
        // Scroll the product list
        device.findObject(By.res("product_list")).fling(Direction.DOWN)
    }
}

Your Compose Performance Checklist

  • Enable Compose compiler metrics to identify non-skippable composables
  • Mark data classes with @Immutable when all properties are immutable
  • Use ImmutableList / ImmutableMap from kotlinx-collections-immutable
  • Wrap callback lambdas in remember to stabilize references
  • Read state in the narrowest scope possible — push reads down the tree
  • Always provide key in LazyColumn / LazyRow items
  • Use derivedStateOf to filter frequent state changes
  • Use remember for expensive object creation inside composables
  • Generate Baseline Profiles for critical UI paths

The Recomposition Rule of Thumb

If a composable is recomposing and nothing visible on screen is changing, something is wrong. Every recomposition should produce a visible change. If it doesn’t, you’re burning CPU for nothing. Use Layout Inspector’s recomposition counts to find the offenders, feed the Compose compiler metrics to Claude Code, and fix the stability issues at their source.

Hands-On: Profile and Optimize a Compose Screen

  1. Enable compiler metrics: Add the Compose compiler flags to your build.gradle.kts and rebuild.
  2. Find unstable composables: Open the *-composables.txt file in app/build/compose_metrics/. Feed it to Claude Code with /analyze-recomposition. How many composables are “restartable but NOT skippable”?
  3. Measure recomposition: Open Layout Inspector, enable recomposition counts, and scroll through a list screen. Note which composables have the highest counts.
  4. Apply fixes: Pick the highest-count composable and apply the techniques from this chapter: stabilize parameters, add key to lazy lists, scope state reads.
  5. Measure again: Compare recomposition counts before and after. Did the counts drop?
  6. Bonus: Set up Baseline Profiles and run a Macrobenchmark to measure cold start and scroll performance.
