Chapter 4

Memory Leak Detection with AI

LeakCanary output meets Claude Code — find and fix memory leaks systematically

The Silent Killer

Crashes are loud. ANRs are visible. Memory leaks are neither. Your app doesn’t crash right away. It just gets slower. GC pauses get longer. Frame rates drop. The system starts killing your background processes more aggressively. And after a few screen rotations or navigation cycles, you get the dreaded OutOfMemoryError.

Chapter 3 showed you how ANRs freeze the app temporarily. Memory leaks are the slow version of death — like a faucet dripping into a bucket. Each drip is invisible. The overflow is catastrophic.

256 MB typical heap limit for an Android app (varies by device)
8x memory cost of keeping one leaked Activity vs. GC-ing it
#3 most common cause of OOM crashes on Android

The good news: memory leaks follow patterns, and AI is excellent at pattern recognition. LeakCanary gives you structured leak traces. Claude Code tells you exactly what’s holding the reference and how to break it.

LeakCanary 101

If you’re not using LeakCanary, stop reading and add it now. It’s the single most valuable debugging library for Android memory issues.

// build.gradle.kts (app module)
dependencies {
    debugImplementation("com.squareup.leakcanary:leakcanary-android:2.14")
}

That’s it. No initialization code. No configuration. LeakCanary automatically watches destroyed Activities, destroyed Fragments, cleared ViewModels, and dismissed Dialogs. When it detects one of these objects surviving garbage collection, it dumps a leak trace.

How LeakCanary Works

Object destroyed → Wait for GC → Still in memory? → Dump heap → Analyze → Show leak trace

LeakCanary uses WeakReference to track objects that should be garbage collected. If an object survives two GC cycles after it should have been collected, LeakCanary dumps the heap and finds the reference chain keeping it alive.

Reading a LeakCanary Trace

A leak trace is structured data that reads like a chain of custody — from the GC root (the thing that’s supposed to be alive) down to the leaking object (the thing that should have been collected).

What Claude Code Finds

The chain: LocationTracker (alive, singleton or long-lived) holds a callback reference to MapActivity (destroyed). The ~~~ under callback marks the suspected cause.

Translation: MapActivity registered itself as a location callback listener but never unregistered when the Activity was destroyed. The LocationTracker lives longer than the Activity, so it keeps the dead Activity in memory.

Impact: 14.2 MB retained across 2,847 objects. Every time the user opens and closes the map screen, another 14 MB leaks.

Root Cause: Unregistered Callback in Long-Lived Object

MapActivity registers itself as a LocationTracker.callback in onCreate() but never clears the reference in onDestroy(). Since LocationTracker is a singleton (or Application-scoped), it outlives the Activity and holds a strong reference to the destroyed Activity’s entire view hierarchy.

This is the “listener registration without cleanup” pattern — the single most common memory leak on Android.

Fix: Unregister in onDestroy (or Use Lifecycle-Aware Components)

// Option 1: Manual cleanup
class MapActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        locationTracker.callback = this::onLocationUpdate
    }

    override fun onDestroy() {
        super.onDestroy()
        locationTracker.callback = null  // Break the reference
    }
}

// Option 2: Lifecycle-aware (better)
class LocationTracker @Inject constructor() : DefaultLifecycleObserver {
    private var callback: ((Location) -> Unit)? = null

    fun observe(owner: LifecycleOwner, onLocation: (Location) -> Unit) {
        callback = onLocation
        owner.lifecycle.addObserver(this)
    }

    override fun onDestroy(owner: LifecycleOwner) {
        callback = null  // Automatically cleaned up
        owner.lifecycle.removeObserver(this)
    }
}

Option 2 is strictly better. By implementing DefaultLifecycleObserver, the cleanup happens automatically when the lifecycle owner is destroyed. No chance of forgetting the cleanup.

[Pattern name from the list above]

The Classic Android Memory Leaks

AI recognizes these patterns instantly because they appear in nearly every Android codebase. Let’s walk through each one.

Leak #1: Activity Reference in Static Field

// The leak:
companion object {
    var lastActivity: Activity? = null  // NEVER do this
}

override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    lastActivity = this  // Leaked forever
}

A companion object property lives for the entire process lifetime. Storing an Activity reference there prevents it from ever being garbage collected.

Leak #2: Inner Class Holding Outer Reference

// Anonymous inner classes hold an implicit reference to the outer class:
class ProfileActivity : AppCompatActivity() {
    fun startTimer() {
        // This Runnable holds a reference to ProfileActivity!
        handler.postDelayed(object : Runnable {
            override fun run() {
                updateUI()  // Implicitly: this@ProfileActivity.updateUI()
                handler.postDelayed(this, 1000)
            }
        }, 1000)
    }
    // If the Activity is destroyed while the Runnable is pending,
    // the Handler keeps the Activity alive via the Runnable reference.
}
Use lifecycleScope.launch instead of Handler.postDelayed. Coroutines launched with lifecycleScope are automatically cancelled when the lifecycle is destroyed — no manual cleanup needed, no leak possible.

Leak #3: ViewBinding in Fragments

This one catches even experienced developers. Fragment views are destroyed and recreated during configuration changes, but the Fragment instance itself survives.

// The leak:
class SettingsFragment : Fragment(R.layout.fragment_settings) {
    private var binding: FragmentSettingsBinding? = null

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        binding = FragmentSettingsBinding.bind(view)
    }
    // Missing: onDestroyView doesn't clear binding!
    // The Fragment outlives its view, holding onto the entire view tree.
}

// The fix:
class SettingsFragment : Fragment(R.layout.fragment_settings) {
    private var _binding: FragmentSettingsBinding? = null
    private val binding get() = _binding!!

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        _binding = FragmentSettingsBinding.bind(view)
    }

    override fun onDestroyView() {
        super.onDestroyView()
        _binding = null  // Critical! Release the view references
    }
}
This leak doesn’t apply to Jetpack Compose. Compose handles view lifecycle automatically. If you’re using ViewBinding in Fragments alongside Compose, though, you still need the cleanup pattern above.

Leak #4: ViewModel Holding View References

// The leak:
class ProductViewModel : ViewModel() {
    // ViewModels survive configuration changes.
    // Holding a View/Context reference = guaranteed leak.
    lateinit var recyclerView: RecyclerView  // NEVER do this
}

// The fix: ViewModels should only hold data, never views.
class ProductViewModel : ViewModel() {
    private val _products = MutableStateFlow<List<Product>>(emptyList())
    val products: StateFlow<List<Product>> = _products.asStateFlow()
    // Let the Fragment/Activity observe this and update the view.
}

Compose-Era Memory Considerations

Jetpack Compose eliminates many classic leaks (no ViewBinding, no manual view references), but introduces its own memory gotchas.

remember vs. rememberSaveable

// remember: survives recomposition but NOT configuration changes
val scrollState = remember { ScrollState(0) }

// rememberSaveable: survives recomposition AND configuration changes
val scrollPosition = rememberSaveable { 0 }

The memory concern with remember is storing large objects that should be scoped to a ViewModel instead:

// Potential memory issue:
@Composable
fun ProductList() {
    // This re-creates a large list on every composition!
    val products = remember { loadAllProducts() }  // Bad if loadAllProducts() is heavy
}

// Better: hoist to ViewModel
@Composable
fun ProductList(viewModel: ProductViewModel = hiltViewModel()) {
    val products by viewModel.products.collectAsState()
}

LaunchedEffect Scope Leaks

// Potential leak: LaunchedEffect capturing an Activity reference
@Composable
fun TrackScreen(activity: Activity) {
    LaunchedEffect(Unit) {
        // If this coroutine outlives the composable, it holds
        // the Activity reference via the captured parameter
        analytics.trackScreen(activity.localClassName)
    }
}

// Safe: pass only the data you need, not the Activity
@Composable
fun TrackScreen(screenName: String) {
    LaunchedEffect(Unit) {
        analytics.trackScreen(screenName)  // No Activity reference captured
    }
}
When feeding Compose memory issues to Claude Code, describe the composable hierarchy and what’s captured in remember and LaunchedEffect blocks. AI can spot closure captures that hold unexpected references.

Building a /scan-for-leaks Skill

Like our ANR prevention skill, this one scans your code proactively.

Create .claude/commands/scan-for-leaks.md:

# /scan-for-leaks — Memory Leak Pattern Scanner

Scan the codebase for known Android memory leak patterns.

## Patterns to Detect

1. **Static Activity/Context references** — companion object or
   top-level properties holding Activity, Fragment, View, or Context
2. **Missing onDestroyView cleanup** — Fragments with ViewBinding that
   don't null the binding in onDestroyView
3. **ViewModel holding View/Context** — ViewModels with View, Activity,
   Fragment, or Context properties
4. **Unregistered listeners** — registerListener/addCallback without
   matching unregister/removeCallback in onDestroy/onDestroyView
5. **Handler.postDelayed without removeCallbacks** — Runnables posted
   to Handlers without cleanup on destroy
6. **Anonymous inner classes in long-lived scopes** — object expressions
   in companion objects, singletons, or Application class that capture
   Activity/Fragment references
7. **Compose captures** — LaunchedEffect or remember blocks that
   capture Activity or other lifecycle-scoped objects

## Output Format
For each potential leak found:
- **File:line** — location
- **Severity** — High (guaranteed leak), Medium (likely leak),
  Low (potential leak under certain conditions)
- **Pattern** — which leak pattern
- **Fix** — specific code change

AI-Assisted Heap Dump Analysis

Sometimes LeakCanary isn’t enough. You need to dig into a full heap dump — for example, when your app uses too much memory overall, not just from a single leak.

Capturing and Summarizing Heap Data

# Capture heap dump from Android Studio:
# 1. Open Profiler > Memory
# 2. Click "Dump Java Heap"
# 3. Export as .hprof

# Or from command line:
adb shell am dumpheap com.example.myapp /data/local/tmp/heap.hprof
adb pull /data/local/tmp/heap.hprof

# Convert to standard format (required for some tools):
hprof-conv heap.hprof heap-std.hprof

You can’t paste a raw .hprof file into Claude Code. Instead, use Android Studio’s Memory Profiler to extract key metrics, then feed those to AI:

Prompt
Here's a summary from my heap dump after 10 minutes of normal app usage: Total heap: 187 MB / 256 MB limit Top allocations by retained size: - Bitmap: 72 MB (847 instances) - byte[]: 34 MB (12,432 instances) - Activity instances: 5 (should be 1 — user is on HomeActivity) - Fragment instances: 12 (should be ~3 for current navigation) - String: 18 MB (89,201 instances) The app has only one Activity visible. Why are there 5 Activity instances retained?
Extracted metrics + a specific question = AI can identify that 4 Activities leaked, likely during rotation or navigation, and suggest where to look.

Your Memory Leak Checklist

  • Add LeakCanary to your debugImplementation dependencies
  • Never store Activity, Fragment, View, or Context in static/companion fields
  • Always null ViewBinding in onDestroyView() for Fragments
  • Use DefaultLifecycleObserver for automatic listener cleanup
  • ViewModels hold data only — never views or contexts
  • Prefer lifecycleScope over Handler.postDelayed
  • In Compose, don’t capture Activity references in LaunchedEffect or remember
  • Set up /analyze-leak and /scan-for-leaks skills

The Leak-Free Mindset

Every time you pass this (an Activity or Fragment) to something that lives longer than it does — a singleton, a companion object, a callback, a ViewModel — you’re creating a potential leak. The question is always: “What happens to this reference when my Activity/Fragment is destroyed?” If the answer is “nothing,” you have a leak.

Hands-On: Create, Detect, and Fix a Memory Leak

  1. Add LeakCanary to your app if you haven’t already (one line in build.gradle.kts).
  2. Create a deliberate leak: Add a companion object property in any Activity and assign this to it in onCreate(): 
    companion object {
    var leakedActivity: Activity? = null
}
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    leakedActivity = this
}
  3. Trigger it: Open the Activity, then press Back. Rotate the device. Open it again. LeakCanary should notify you within 5–10 seconds.
  4. Capture the trace: Open the LeakCanary notification and copy the full leak trace.
  5. Run /analyze-leak: Paste the trace into Claude Code. Verify it identifies the companion object reference as the cause.
  6. Fix and verify: Remove the static reference, rebuild, and repeat the trigger steps. LeakCanary should stay silent.
  7. Bonus: Run /scan-for-leaks on your actual codebase. How many potential leaks does AI find?
