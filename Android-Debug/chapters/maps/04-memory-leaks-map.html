The Big Idea

Every time you pass this (Activity/Fragment) to something that lives longer — a singleton, companion object, callback, ViewModel — ask: "What happens to this reference when my Activity is destroyed?" If the answer is "nothing," you have a leak.

🔬 Anatomy of a Leak Trace

  1. GC Root (top)

    The long-lived object that's supposed to be alive — singleton, static field, Application scope.

  2. Suspected Cause (~~~ underline)

    The reference marked with ~~~ is where the retention chain should be broken.

  3. Leaking Object (bottom)

    The node marked Leaking: YES — the Activity/Fragment that should have been GC'd. Shows retained size.

Analogy: A chain of custody from something alive to something dead. Break the ~~~ link to fix the leak.

🗂️ The Classic Android Memory Leaks

🔗
#1 Unregistered Listener
Register callback in onCreate, forget to unregister in onDestroy. The long-lived owner keeps the dead Activity alive. Most common leak.
📌
#2 Static Activity Reference
companion object { var activity: Activity? = null } — lives for the entire process. Activity can never be GC'd.
🪆
#3 Inner Class Capture
Anonymous Runnable in Handler.postDelayed implicitly captures outer Activity. Fix: use lifecycleScope.launch instead.
🖼️
#4 ViewBinding in Fragments
Fragment outlives its view on config change. Must null _binding in onDestroyView(). Doesn't apply to Compose.
🧮
#5 ViewModel Holding Views
ViewModels survive config changes. Holding a View or Context = guaranteed leak. ViewModels hold data only.
🧪
#6 Compose Captures
LaunchedEffect capturing an Activity reference. Fix: pass only the data you need, not the Activity itself.

🛠️ Fix Patterns at a Glance

Leak Pattern Quick Fix Better Fix
Unregistered listener Null callback in onDestroy Use DefaultLifecycleObserver for auto-cleanup
Static Activity reference Don't. Just don't. Use Application context or dependency injection
Handler + Runnable removeCallbacks in onDestroy Replace with lifecycleScope.launch { delay() }
ViewBinding in Fragment _binding = null in onDestroyView Migrate to Compose (no binding needed)
ViewModel + View Remove the View reference Expose StateFlow, let UI observe
Compose LaunchedEffect capture Pass data, not Activity Hoist side effects to ViewModel

📖 Key Terms

LeakCanary
Debug library that auto-detects leaked Activities, Fragments, ViewModels, and Dialogs. One-line setup, zero config.
Leak trace / retention chain
Path from GC root → leaked object. The ~~~ underline marks the suspected cause reference.
GC Root
An object the garbage collector considers always reachable — static fields, thread stacks, JNI globals.
WeakReference
A reference that doesn't prevent GC. LeakCanary uses these to detect when objects survive collection.
DefaultLifecycleObserver
Interface for automatic lifecycle-aware cleanup. Implement onDestroy to break references automatically.
remember vs rememberSaveable
remember: survives recomposition. rememberSaveable: survives recomposition AND config changes.
.hprof
Heap dump file format. Capture via Android Studio Profiler or adb shell am dumpheap. Extract metrics for AI analysis.

🧪 Quick Quiz

What does the ~~~ underline in a LeakCanary trace mean?

It marks the suspected cause — the specific reference in the retention chain that should be broken to fix the leak.

Why must you null _binding in onDestroyView() for Fragments using ViewBinding?

The Fragment instance survives config changes, but its view is destroyed and recreated. Without nulling, the Fragment holds a reference to the old (dead) view tree, leaking the entire view hierarchy.

Why can't you store an Activity reference in a ViewModel?

ViewModels survive configuration changes (rotation). The old Activity is destroyed, but the ViewModel keeps it alive via the reference. Guaranteed leak on every rotation.

How does Compose avoid the ViewBinding leak problem?

Compose has no ViewBinding. View lifecycle is managed automatically by the Compose runtime. However, Compose introduces its own risks: LaunchedEffect capturing Activity references and heavy objects in remember.

What's the "lifecycle-aware" fix for listener leaks?

Implement DefaultLifecycleObserver. In observe(), register the callback and add the observer. In onDestroy(), null the callback. Cleanup happens automatically — no chance of forgetting.

📌 Key Takeaways

  • Add LeakCanary (debugImplementation) — it auto-detects leaked Activities, Fragments, ViewModels, and Dialogs with zero config.
  • The 6 classic leaks: unregistered listener, static reference, inner class capture, ViewBinding in Fragment, ViewModel + View, Compose LaunchedEffect capture.
  • Read leak traces bottom-up: find Leaking: YES, then follow the ~~~ underline to the cause.
  • Use DefaultLifecycleObserver for automatic cleanup instead of manual onDestroy null-outs.
  • ViewModels hold data only (StateFlow) — never Views, Contexts, or Activities.
  • In Compose, pass data to LaunchedEffect, not Activity references. Hoist heavy state to ViewModel.
