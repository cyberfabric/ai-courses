Production Monitoring & Crash Reporting

The Big Idea

Production debugging is about prioritization, not just diagnosis. You can't fix every crash immediately. Use AI to triage by severity, user impact, regression status, and fix complexity — fix the crash hitting 1,200 premium users before the one affecting 3 users on Android 8.

🧱 The Four Layers of Crash Context

  1. Stacktrace (automatic)

    Crashlytics captures this by default. The what and where of the crash. Upload mapping files for deobfuscation.

  2. Custom Keys (you add)

    Current screen, user tier, feature flags, cart size, logged-in status. The business context that tells you why it matters.

  3. Breadcrumb Logs (you add)

    Trace the user's journey: "tapped Pay Now" → "payment API started" → "API returned success" → CRASH. The story of how they got there.

  4. Non-Fatal Exceptions (you add)

    Caught errors that don't crash but indicate problems: timeouts, parsing failures, fallback paths. The silent signals of degraded experience.

🚦 Crash Severity Framework

Severity Criteria Action
🔴 Critical Data loss, security issue, payments affected, >5% DAU Hotfix immediately, halt rollout
🟠 High >1% DAU, core feature broken, regression from new release Fix in next release, consider hotfix
🟡 Medium <1% DAU, edge case, workaround exists Schedule for next sprint
🟢 Low Cosmetic, rare devices only, non-core feature Backlog, monitor trend

📖 Key Terms

Crash-Free Rate
Percentage of daily active users who don't experience a crash. Target: 99.5%+. A drop from 99.6% to 98.1% is a red alert.
Crash Cluster
Crashlytics groups identical crashes by stacktrace signature. Each cluster represents one unique bug across all affected users.
Staged Rollout
Releasing to 1% → 10% → 50% → 100% of users, monitoring crash rates at each stage. Lets you halt before a buggy release reaches everyone.
Custom Keys
Key-value pairs you set at runtime (setCustomKey). Searchable in Crashlytics. The bridge between technical crashes and business context.
Breadcrumbs
Sequential log messages (Firebase.crashlytics.log()) that reconstruct the user's journey leading to a crash.
Non-Fatal Exceptions
Caught exceptions reported via recordException(). Don't crash the app but indicate degraded functionality (timeouts, parse errors, fallbacks).
Android Vitals
Google Play Console metrics: crash rate, ANR rate, startup time, rendering. Affects your app's Play Store visibility.

🧪 Quick Quiz

A crash has 0 events on v4.1.3 but 2,847 on v4.2.0. What does this tell you?

It's a regression introduced in v4.2.0. This gets higher priority than an existing crash, because it means something in the latest release broke. Compare the diff between versions to find the change.

Why add custom keys like subscription_tier and cart_items_count?

They add business context to technical crashes. A crash during checkout for a premium user with 5 cart items is a lost sale worth investigating immediately. AI can assess business impact, not just code severity.

What's the difference between a crash and a non-fatal exception in Crashlytics?

A crash kills the app process (fatal). A non-fatal is a caught exception you report via recordException() — the app continues but something went wrong (timeout, parse error, fallback triggered). Non-fatals are silent signals of degraded UX.

The breadcrumbs show: "tapped Pay Now" → 4 seconds → "payment API success" → CRASH. What happened?

The user likely backgrounded the app during the 4-second payment wait. Android called onSaveInstanceState(). When the payment callback tried a Fragment transaction, it threw IllegalStateException. Fix: use lifecycle-aware navigation with repeatOnLifecycle(STARTED).

How often should you monitor crash rates after a new release?

Every release should be monitored for 24–48 hours. Use staged rollouts (1% → 10% → 50% → 100%) so you can halt if crash rates spike. A >1% drop in crash-free rate should trigger immediate investigation.

Key Takeaways

  • Production debugging = prioritization. Use AI to triage by severity, user impact, and fix complexity.
  • Enrich Crashlytics with custom keys, breadcrumbs, and non-fatal exceptions — the richer the context, the better the AI analysis.
  • 70% of crashes come from the top 5 clusters. Focus there first.
  • Compare crash rates across versions to catch regressions. 0 events on old version + thousands on new = regression.
  • Use staged rollouts (1%→10%→50%→100%) and monitor 24 hours after each release.
  • Build /triage-crash for individual incidents and /weekly-crash-report for team-level visibility.
