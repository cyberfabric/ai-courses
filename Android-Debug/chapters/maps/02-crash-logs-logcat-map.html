Chapter 2

Crash Logs, Logcat & Stacktraces

Memory Map
💡

The Big Idea

The quality of AI crash analysis is directly proportional to the quality of your input. Filter your logcat. Deobfuscate your stacktraces. Include context. Two minutes preparing saves twenty minutes of vague output.

🔬 Anatomy of a Stacktrace

  1. FATAL EXCEPTION: main

    Thread name — tells AI if it's a lifecycle, threading, or callback issue.

  2. Process: com.example.myapp

    Package name — helps AI filter your code from framework code.

  3. IllegalStateException: Fragment not attached

    Exception type + message — AI maps this to known Android patterns instantly.

  4. at com.example.myapp.ui.ProfileFragment.updateUI(:87)

    Your code entry point — the first frame with your package name.

  5. Caused by: (deepest block)

    The real culprit — like a crime scene, follow the chain to where the crime happened.

📟 Essential Logcat Commands

🎯
--pid Filter
adb logcat --pid=$(adb shell pidof your.pkg) — shows only your app, cuts 95% of noise.
💀
Fatal Only
adb logcat "*:E" | grep -A 20 "FATAL EXCEPTION" — just crashes.
🏷️
Tag Filter
adb logcat -s "TagName" — when you know the component.
💾
Dump to File
adb logcat -d > crash_log.txt — snapshot for AI analysis.
🧹
Clear & Capture
adb logcat -c && adb logcat --pid=... — fresh log before reproducing.

🗂️ The Crash Pattern Library

no such table		 Missing Room migration Add Migration(old, new) with CREATE TABLE

🧩 Key Concepts

⛓️
Caused-by Chain
The deepest Caused-by is the real culprit. Upper blocks are just wrappers that re-threw the exception.
🔐
R8 Deobfuscation
Release builds obfuscate names. Run retrace with mapping.txt before feeding to AI. No mapping = unreadable.
📱
Multi-Device Crashes
Same code, different behavior. Include both stacktraces + device/API info. AI knows Samsung, Xiaomi, Pixel quirks.
📦
Library Crashes
When the stacktrace is all OkHttp/Retrofit, paste your usage code alongside it. AI traces backward from the crash to your config.
⚡
/analyze-crash Skill
Reusable command that classifies crash type, traces to user code, identifies root cause, and rates severity.
🏷️
Crash Classification
Five types: Lifecycle, Threading, Data, Resource, Library. Each type needs a different class of fix.

📖 Key Terms

FATAL EXCEPTION
The logcat header for an uncaught exception that kills your process. Always include this line when pasting to AI.
Caused-by Chain
Nested exceptions. Read bottom-up — the deepest block is where the crime happened.
R8 / ProGuard
Code shrinker that obfuscates class/method names in release builds. Produces mapping.txt needed for deobfuscation.
retrace
Android SDK tool that converts obfuscated stacktraces back to readable form using mapping.txt.
viewLifecycleOwner.lifecycleScope
Coroutine scope tied to Fragment's view lifecycle. Auto-cancels on view destroy. Prevents "not attached" crashes.
/analyze-crash
Claude Code skill that classifies crashes into types and returns structured analysis with severity rating.
fallbackToDestructiveMigration()
Room option that wipes the DB on failed migration. Use in debug only — deletes all user data in production.

🧪 Quick Quiz

In a multi-cause stacktrace, which "Caused by" block should you focus on?

The deepest (last) Caused-by block. It's the root exception. The ones above are wrappers that caught and re-threw it.

Why must you deobfuscate a release stacktrace before feeding it to AI?

R8 renames classes/methods to a.b.c.d(). AI can't pattern-match on obfuscated names. Use retrace with mapping.txt first.

What causes "Fragment not attached to a context" crashes?

A coroutine launched with lifecycleScope (Activity-scoped) outlives the Fragment's view. When it completes and calls requireContext(), the Fragment is detached. Fix: use viewLifecycleOwner.lifecycleScope.

Why does "SQLiteException: no such table" only hit users who upgrade the app?

Fresh installs create all tables from scratch. Upgraders have the old DB schema and need a Migration to add new tables. Missing migration = table doesn't exist.

What five crash classifications does the /analyze-crash skill use?

Lifecycle, Threading, Data, Resource, and Library (third-party). Each type requires a different class of fix.

📌 Key Takeaways

  • Filter logcat with --pid before pasting — 30 focused lines beat 2000 lines of noise.
  • In chained exceptions, the deepest Caused-by is the real culprit — always read bottom-up.
  • Deobfuscate R8 stacktraces with retrace + mapping.txt before AI analysis. Keep mapping files for every release.
  • Use viewLifecycleOwner.lifecycleScope instead of lifecycleScope in Fragments to prevent lifecycle crashes.
  • Build the /analyze-crash skill for repeatable, classified crash analysis with severity ratings.
  • For library crashes, paste your usage code alongside the stacktrace so AI can trace backward to your misconfiguration.
