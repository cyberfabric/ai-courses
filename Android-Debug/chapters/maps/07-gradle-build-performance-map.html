💀 The Build Killers

🐌
kapt (Not KSP)
kapt reprocesses all annotations from scratch on every build. KSP is incremental — 40-60% faster. Migrate Hilt, Room, and Moshi first.
🔗
api vs implementation
api exposes dependencies to all dependents. When that dep changes, everything recompiles. Use implementation unless the type is in your public API.
💾
Disabled Caches
Build cache + configuration cache together cut incremental builds by 50-80%. Config cache alone dropped 18s → 0.4s in the chapter example.
📦
Monolith Module
847 files in one module = recompile everything on every change. Modularize: a change in :feature-cart only rebuilds that module.

⚙️ Essential gradle.properties

org.gradle.caching=true
Stores task outputs and reuses them when inputs haven't changed. Target: >85% cache hit rate.
org.gradle.configuration-cache=true
Caches the configuration phase result. Drops 18s config phase to <1s. Gradle 9’s preferred execution mode — AGP 9.x and most mainstream plugins ship with support; expected to become mandatory in a future major release.
org.gradle.parallel=true
Executes independent tasks across modules in parallel. Essential for multi-module projects.
org.gradle.jvmargs=-Xmx4g
Heap size for the Gradle daemon. Too low (512m) triggers GC pauses. Use 4g for typical projects, 6g for large ones.
org.gradle.daemon=false
CI-only: skip daemon overhead since each CI run starts fresh. Keep true (default) for local development.

🧪 Quick Quiz

Why is kapt slow and KSP fast?

kapt delegates to Java's annotation processing API, which requires a full reprocessing pass on every build. KSP is Kotlin-native and supports incremental processing — it only reprocesses files that actually changed.

What's the danger of using api instead of implementation?

api exposes the dependency to all dependent modules. When that dependency changes, every dependent module recompiles — creating cascade rebuilds. Use implementation unless the dependency's types appear in your module's public API.

What's a remote build cache and why is it the biggest CI win?

A remote build cache stores task outputs on a shared server. When one CI run builds a task, subsequent CI runs (and local devs) download the cached output instead of rebuilding. This can cut CI times by 40-70%.

Your config phase takes 18 seconds for 32 projects. How do you fix it?

Enable configuration cache: org.gradle.configuration-cache=true. It stores the result of the configuration phase and reuses it. In the chapter example, this dropped 18s to 0.4s.

Key Takeaways

  • Run ./gradlew assembleDebug --profile to identify your top 5 slowest tasks before optimizing anything.
  • Migrate from kapt to KSP — the single biggest win for most Android projects (40-60% annotation processing reduction).
  • Enable both build cache and configuration cache in gradle.properties — together they cut incremental builds by 50-80%.
  • Use implementation not api for dependencies unless the type is in your module's public API — prevents cascade recompilation.
  • Set up a remote build cache for CI — it's the single biggest CI optimization (40-70% reduction).
