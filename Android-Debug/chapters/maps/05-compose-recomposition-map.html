Chapter 5

Compose Recomposition Performance

Memory Map
💡

The Big Idea

If a composable recomposes and nothing visible on screen changes, you're burning CPU for nothing. Every recomposition should produce a visible change. Find the offenders with Layout Inspector, fix them with stability annotations and scoped state reads.

🔄 How Recomposition Triggers

The problem: Compose thinks inputs changed even when meaningful data hasn't — unstable types, re-allocated lambdas, and wrong state scoping are the culprits.

🚀 Strong Skipping Mode (Default since Kotlin 2.0.20)

✅
All restartable = skippable
Even composables with unstable parameters now skip. The old "restartable but NOT skippable" verdict mostly disappears.
🪄
Lambdas auto-memoized
The compiler wraps every lambda inside a composable in remember with its captures as keys. Manual remember { { ... } } is no longer required.
⚖️
Equality shifts
Stable params: object equality (.equals()). Unstable params: instance equality (===). Reallocated objects always fail === — that's why @Immutable still matters.
🚫
Opt-outs
@NonSkippableComposable on a function disables skipping. @DontMemoize on a lambda disables auto-memoization.

🪤 The Top Recomposition Traps

⚠️
Unstable Parameters
List, Map, non-data classes use === equality under strong skipping — reallocated objects never match, so the composable still recomposes. Fix: ImmutableList + @Immutable for object-equality semantics.
🎯
State Read Scoping
Reading state in a parent recomposes everything below it. Rule: read state as late as possible, in the narrowest composable scope.
🧠
Missing remember
Without remember, expensive objects (formatters, parsers, regex) are recreated on every recomposition. Wrap them.
🔗
Lambda Captures
Strong skipping auto-memoizes lambdas, but the keys are the captured values. If the captures change every recomposition, the auto-remember still invalidates. Stabilize the captures (or use @DontMemoize for hot paths).
🔑
Missing LazyColumn key
Without key, Compose tracks items by position. Adding an item at the top recomposes every item. Always use key = { it.id }.
📦
Heavy Item Composables
Image loading, price formatting, markdown parsing inside items kills scroll perf. Move expensive work into remember or the ViewModel.

🛠️ The 5-Step Optimization Workflow

  1. Enable Compose Compiler Metrics

    Apply org.jetbrains.kotlin.plugin.compose, then add a composeCompiler { reportsDestination = ...; metricsDestination = ... } block. Rebuild to generate *-composables.txt and *-classes.txt in app/build/compose_compiler/.

  2. Find Skip-Quality Issues

    Under strong skipping, look at unstable parameters that hurt skip quality (=== on reallocated objects = never matches), not skip occurrence. Feed the report to Claude Code with /analyze-recomposition.

  3. Measure with Layout Inspector

    Run app in debug mode → Tools → Layout Inspector → enable "Show Recomposition Counts" → interact and watch numbers climb.

  4. Apply Fixes

    Stabilize parameters (@Immutable, ImmutableList), add key to lazy lists, scope state reads lower, wrap lambdas in remember.

  5. Measure Again

    Compare recomposition counts before vs. after. Target: dramatic reduction. Then set up Baseline Profiles for startup and scroll performance.

📖 Key Terms

Recomposition
Compose re-executing a composable function when its inputs (state) change — the Compose equivalent of re-rendering.
Skippable
A composable that Compose can skip if all its parameters are equal to the previous call. Under strong skipping (Kotlin 2.0.20+ default), all restartable composables are skippable.
Strong Skipping Mode
Default Compose Compiler behavior since Kotlin 2.0.20. All restartable composables become skippable (even with unstable params), and lambdas inside composables are auto-memoized.
@Immutable
Annotation promising all properties of a class will never change after construction. Switches comparison from === to .equals() — lets reallocated-but-content-equal objects skip.
@Stable
Annotation promising that changes will be notified to Compose via MutableState. Weaker than @Immutable but also enables object-equality comparison.
@NonSkippableComposable / @DontMemoize
Opt-outs for strong skipping — disable skipping on a composable, or disable auto-memoization on a specific lambda.
ImmutableList
From kotlinx-collections-immutable. A List implementation Compose treats as stable (unlike the List interface).
derivedStateOf
Creates a derived state that only triggers recomposition when its computed value changes, not when the source state changes.
snapshotFlow
Converts Compose snapshot state reads into a coroutine Flow — useful for side effects like analytics in response to state changes.
Baseline Profile
A list of critical code paths that get AOT-compiled for faster startup and reduced jank on first launch.

🧪 Quick Quiz

Why is List<String> unstable in Compose?

List is an interface — it could be backed by a MutableList. Compose can't guarantee it hasn't been mutated between recompositions, so it conservatively marks it unstable. Use ImmutableList from kotlinx-collections-immutable instead.

What's the difference between derivedStateOf and snapshotFlow?

derivedStateOf computes a new state value that only triggers recomposition when the derived value changes (UI use). snapshotFlow converts state reads into a Flow for side effects (analytics, logging, network calls) inside a coroutine.

A composable is "restartable but NOT skippable." What does that mean — and is it still common in 2026?

Historically, it meant Compose could restart the composable but never skip it, usually because of unstable parameter types. Under strong skipping (default since Kotlin 2.0.20), this verdict is rare — restartable composables are skippable by default. The remaining concern is skip quality: unstable params use === on potentially reallocated objects, so they often fail equality and recompose anyway.

If strong skipping auto-memoizes lambdas, why do @Immutable and ImmutableList still matter?

Because they switch parameter comparison from === (instance equality) to .equals() (content equality). When data flows from Room or networking, every emission allocates new objects — === always reports "different," so the composable recomposes even when content is identical. @Immutable tells Compose it's safe to use object equality.

You add an item at the top of a LazyColumn with no key. What happens?

Every item recomposes because Compose tracks items by position. All positions shifted by one, so Compose thinks every item changed. Adding key = { it.id } lets Compose track by identity instead.

Key Takeaways

  • Recomposition is the #1 Compose performance issue — measure before you optimize using Layout Inspector and the modern composeCompiler {} DSL.
  • Strong skipping (default since Kotlin 2.0.20) makes all restartable composables skippable and auto-memoizes lambdas. @Immutable + ImmutableList still matter to enable object-equality comparison on reallocated objects.
  • Read state in the narrowest scope — the composable that reads the state is the one that recomposes. Push reads down the tree.
  • Always provide key in LazyColumn/LazyRow — without it, inserting one item recomposes all items.
  • Use derivedStateOf to filter rapid state changes into meaningful recomposition triggers (e.g., boolean from scroll index).
