Battery and Power Optimization

Memory Map

The Big Idea

Every time your code wakes the CPU, activates the radio, or requests a GPS fix, ask: "Is the user getting visible value from this work right now?" If not, defer it, batch it, or eliminate it. Users don't see your sync. They do see "24% battery used by ShopApp."

🔥 The Battery Drain Hall of Fame

📍
GPS Overuse
PRIORITY_HIGH_ACCURACY at 1-second intervals keeps GPS radio on continuously. ~15%/hour. Switch to BALANCED_POWER_ACCURACY (~0.5%/hour).
🔓
Wake Lock Leaks
acquire() without timeout + exception = CPU stays awake forever. Always use acquire(timeoutMs) and release() in a finally block.
🔄
Over-Frequent Sync
Syncing a daily-changing catalog every 2 minutes = 720 syncs/day. Match sync frequency to actual data change rate + add battery/network constraints.
📡
Radio Tail
Each network request keeps the cellular radio active ~20 seconds. 10 small requests in 10 min = radio on almost continuously. Batch them.
🔔
Polling Instead of Push
Polling every 30 seconds for new messages drains battery. Use FCM push notifications — the radio only activates when there's actual data.
⚙️
Wrong Scheduling API
Using AlarmManager (exact) for deferrable work wakes the device from Doze. Use WorkManager for anything that can wait.

📋 Which Scheduling API?

API Use When Battery Impact
WorkManager Any deferrable background work (default choice) 🟢 Low — respects Doze, batches intelligently
AlarmManager (inexact) Periodic triggers, timing not critical 🟡 Medium — system batches with other alarms
AlarmManager (exact) User-facing alarms only (medication, meetings) 🔴 High — wakes from Doze, needs permission
Foreground Service User-initiated ongoing tasks (music, navigation) 🟡 Varies — shows notification, user aware

🔐 The Safe Wake Lock Pattern

  1. Acquire with timeout

    wakeLock.acquire(10 * 60 * 1000L) — auto-release after 10 minutes even if your code crashes.

  2. Wrap work in try/finally

    try { doWork() } finally { if (wakeLock.isHeld) wakeLock.release() } — guaranteed cleanup.

  3. Prefer WorkManager

    WorkManager handles wake locks automatically. Only manage them yourself if you truly need custom wake lock behavior.

📖 Key Terms

Doze Mode
System-level power saver that defers alarms, jobs, syncs, and network when device is idle with screen off.
App Standby Buckets
5 tiers (Active → Restricted) that throttle your app's background work based on how recently the user interacted with it.
Wake Lock
Keeps the CPU running when the screen is off. Background partial wake locks count against the new Play Console quality metric. Always use a timeout.
Excessive Wake Locks Metric
Play Console quality bar (enforced March 1, 2026): >2h cumulative non-exempt wake locks in 24h, in >5% of sessions over 28 days = derank + battery-drain warning.
FGS Timeout (Android 15)
dataSync and mediaProcessing foreground services get 6h per 24h period. Service.onTimeout() fires at the limit — call stopSelf() within seconds.
Radio Tail
The ~20 seconds the cellular radio stays active after each network request. Batching requests eliminates redundant tail time.
Geofence
Hardware-accelerated location trigger that fires when the user enters/exits an area. Nearly zero battery cost vs. continuous location polling.
Battery Historian
Google's tool for visualizing battery usage timelines from a bug report. Shows wake locks, jobs, GPS, and network activity.
PRIORITY_BALANCED_POWER_ACCURACY
Location priority that uses cell towers and WiFi instead of GPS. ~30x less battery drain than HIGH_ACCURACY for most use cases.

⚡ The 2026 Battery Bar

📉
Excessive Wake Locks (Mar 1, 2026)
>2h cumulative background wake locks in 24h across >5% of sessions (28-day window) = Play Store derank + store-listing battery warning. Investigate any tag with P90/P99 > 60 min.
⏰
FGS 6-Hour Timeout (Android 15)
dataSync + mediaProcessing capped at 6h per 24h. Implement Service.onTimeout() & call stopSelf(). Test: adb shell am compat enable FGS_INTRODUCE_TIME_LIMITS.
🚫
No FGS from BOOT_COMPLETED
Starting an FGS from a BOOT_COMPLETED receiver throws ForegroundServiceStartNotAllowedException. Use WorkManager with setInitialDelay instead.
📊
Android 16 Job Quotas
Jobs running concurrent with an FGS now have runtime quotas. The “hide everything behind an FGS” pattern no longer buys unlimited compute. Design for interruption.

🧪 Quick Quiz

Why should you always call acquire(timeoutMs) instead of bare acquire()?

Bare acquire() has no safety net — if your code throws an exception or hangs before release(), the CPU stays awake indefinitely, draining the battery. The timeout overload auto-releases even if your code fails.

Your shopping app checks for new products every 2 minutes via AlarmManager. What's wrong?

Two problems: (1) Product catalogs change daily, not every 2 minutes — use a 24-hour WorkManager periodic task instead. (2) AlarmManager can wake the device from Doze; WorkManager respects Doze and adds battery/network constraints.

What's the "radio tail" and how do you avoid it?

The cellular radio stays active ~20 seconds after each request. Many small requests keep it on almost continuously. Fix: batch network requests (e.g., collect analytics events and send them all in one call via WorkManager).

When should you use PRIORITY_HIGH_ACCURACY for location?

Only for turn-by-turn navigation or features that genuinely need meter-level GPS accuracy. For store finders, delivery tracking, or weather apps, PRIORITY_BALANCED_POWER_ACCURACY (cell/WiFi) is sufficient and uses ~30x less battery.

Key Takeaways

  • Location is the #1 battery drain after screen — use BALANCED_POWER_ACCURACY and geofences instead of continuous GPS tracking.
  • Always use wakeLock.acquire(timeoutMs) with a try/finally release — never bare acquire().
  • Default to WorkManager for all background work. Reserve exact alarms for user-facing reminders and foreground services for user-initiated tasks.
  • Batch network requests to avoid the ~20-second radio tail per call. Use FCM push instead of polling.
  • Audit regularly with adb shell dumpsys batterystats --charged [package] — target under 2% battery per hour for non-navigation apps.
  • Battery is now a Play Store quality bar (March 2026): keep background wake locks below 2h/24h in >95% of sessions; dataSync/mediaProcessing FGS run at most 6h/24h.
