Chapter 3

ANR Diagnosis and Prevention

Memory Map
💡

The Big Idea

Every time you write code on the main thread, ask: "Could this ever take more than 100ms?" If yes — or even "maybe on a slow device" — move it to a background thread. The main thread has one job: respond to the user.

😱 Why ANRs Are Worse Than Crashes

🚪
47% Uninstall
Nearly half of users uninstall after experiencing an ANR. No crash dialog, no review — they just leave.
⚖️
2x Play Store Penalty
ANRs weigh twice as much as crashes in Play Store vitals. Bad-behavior threshold: 0.47% of sessions.
⏱️
5-Second Rule
Main thread blocked for 5 seconds = ANR dialog. That's all it takes.
👻
Silent Killer
No stacktrace, no crash report. The app just freezes. Hardest Android bugs to troubleshoot.

🎯 The Three ANR Triggers

ANR Type Trigger Timeout
Input dispatching (most common) Main thread doesn't respond to touch/key 5 seconds
BroadcastReceiver onReceive() doesn't complete 10s foreground / 60s background
Service onCreate/onStartCommand/onBind doesn't complete 20s foreground / 200s background

🧵 Reading the Main Thread State

  1. Blocked

    Waiting to acquire a lock held by another thread. Find waiting to lock <0x...> held by thread N, then find thread N.

  2. Runnable (doing heavy work)

    Main thread is running CPU-intensive code — JSON parsing, bitmap ops, large list sorting. Top stack frames reveal what.

  3. Waiting / Sleeping

    Main thread is waiting for a callback, future, or synchronous IPC that hasn't returned.

🗂️ Common ANR Patterns

💾
SharedPreferences Trap
getString() looks harmless but loads the entire XML from disk. First call + concurrent commit() = ANR. Fix: migrate to DataStore.
🔀
Wrong Dispatcher
viewModelScope.launch defaults to Dispatchers.Main. Room query inside = main thread IO. Fix: wrap in withContext(Dispatchers.IO).
🔒
Lock Contention
Main thread waits for a DB lock held by a long-running background sync. Two problems compound — sync query + direct main-thread access.
📡
Binder IPC Stall
System service call (getPackageInfo, etc.) blocks main thread. Look for BinderProxy.transactNative in the trace. Move call off main thread.
⌨️
Undebounced Input
DB query on every keystroke in afterTextChanged. Fix: use debounce(300) with StateFlow + Room's Flow return type.
🚫
allowMainThreadQueries()
Disabling Room's safety check is asking for ANRs. Room's check exists for a reason. Use suspend or Flow instead.

📖 Key Terms

ANR (Application Not Responding)
Android kills your UI responsiveness when the main thread is blocked beyond the timeout threshold.
traces.txt
Thread dump at /data/anr/traces.txt showing every thread's state at the moment of the ANR.
Lock contention
Main thread waiting for a lock held by a background thread. The ANR trace shows waiting to lock <0x...> held by thread N.
StrictMode
Debug-time detector for disk, network, and slow calls on the main thread. Enable in Application.onCreate().
DataStore
Fully async replacement for SharedPreferences. Won't block the main thread.
debounce()
Coroutine Flow operator that waits N ms after the last emission before forwarding. Prevents per-keystroke queries.
Binder IPC
Inter-process communication to Android system services. Can stall the main thread if the service is overloaded.

🧪 Quick Quiz

What are the three types of ANR triggers and their timeouts?

Input dispatching (5s) — main thread doesn't respond to touch. BroadcastReceiver (10s fg / 60s bg) — onReceive doesn't complete. Service (20s fg / 200s bg) — lifecycle methods don't complete.

In an ANR trace, the main thread is "Blocked" and "waiting to lock <0x02a3e5c8> held by thread 14". What's the next step?

Find thread 14 in the trace dump to see what it's doing. It holds the lock the main thread needs. The root cause is whatever long operation thread 14 is running.

Why is SharedPreferences.getString() an ANR risk?

It's synchronous. The first call loads the entire XML from disk. If another thread is calling commit(), getString() blocks until the commit finishes. On slow flash storage = ANR. Fix: migrate to DataStore.

What's wrong with viewModelScope.launch { dao.getAllUsers() }?

viewModelScope.launch defaults to Dispatchers.Main. The Room query runs on the main thread. Fix: wrap in withContext(Dispatchers.IO) or use Room's Flow/suspend return types.

What's the Play Store ANR bad-behavior threshold?

0.47% of daily sessions. Roughly 1 in 200 sessions with an ANR gets your app flagged, hurting visibility in the Play Store.

📌 Key Takeaways

  • ANRs are worse than crashes — 47% uninstall rate and 2x penalty in Play Store vitals.
  • The main thread blocked for 5 seconds = ANR. Ask "could this take >100ms?" for every main-thread operation.
  • ANR traces show every thread's state. Find "main" thread, check its state, follow the lock chain to the holder.
  • Top ANR causes: SharedPreferences on main thread, wrong dispatcher, lock contention, undebounced input queries.
  • Enable StrictMode in debug builds to catch violations before they become production ANRs.
  • Build /analyze-anr (diagnosis) and /check-anr-risks (prevention) skills for your toolkit.
