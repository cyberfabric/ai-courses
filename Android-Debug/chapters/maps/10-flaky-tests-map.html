Chapter 10

Flaky Test Diagnosis

Memory Map
💡

The Big Idea

A flaky test is not "random." It is a deterministic bug in your test that only manifests under certain conditions. Categorize it, find the root cause, fix it. Never retry your way past it.

🎯 The Four Categories of Flakiness

⏱️
Timing-Dependent
Fails on slow machines, passes on fast ones. Missing advanceUntilIdle(), hardcoded delays, race conditions between coroutines.
🔗
Order-Dependent
Passes alone, fails when run after another test. Shared mutable state in companion objects, database not cleared between tests.
🌍
Environment-Dependent
Passes locally, fails on CI. Timezone, locale, emulator speed, screen density, animation scale differences.
🏗️
Resource-Dependent
Fails intermittently under load. Database locks, file handles, network ports, memory pressure from parallel test execution.

🏠 Local vs CI: Why Tests Behave Differently

Difference How It Causes Flakiness Fix
Emulator speed Async operations take longer Disable animations via adb shell
Timezone Date assertions fail (CI is UTC) Use Clock abstraction, fixed timezone
Locale Number/date formatting differs Set Locale.setDefault(Locale.US)
Screen density Pixel-based assertions fail Use semantic matchers (hasText)
Parallelism Tests compete for shared resources Use unique names per test instance

🔧 Fix Patterns by Category

  1. Timing: Coroutine Tests

    Use runTest + advanceUntilIdle(). Set up MainDispatcherRule in every ViewModel test class. Use Turbine for Flow assertions.

  2. Timing: Compose UI Tests

    Replace Thread.sleep() with waitUntilExactlyOneExists(). Let the test framework handle synchronization — never hardcode delays.

  3. Timing: Espresso Tests

    Register IdlingResource for async operations (OkHttp, Room, custom loaders). Espresso can't wait for work it doesn't know about.

  4. Order: Shared State

    Create a new database instance per test in @Before, close it in @After. Never put mutable state in companion object.

  5. Environment: CI Setup

    Disable all animation scales. Set explicit locale and timezone. Use semantic UI matchers instead of pixel positions.

📖 Key Terms

advanceUntilIdle()
Tells the test coroutine scheduler to execute all pending coroutines before continuing. The #1 fix for timing-flaky coroutine tests.
MainDispatcherRule
A JUnit rule that replaces Dispatchers.Main with a TestDispatcher. Without it, viewModelScope coroutines can't be controlled in tests.
Turbine
A library for testing Kotlin Flows. Uses awaitItem() instead of collecting + asserting, making Flow tests deterministic.
IdlingResource
Tells Espresso about custom async work. Espresso waits until all registered IdlingResources are idle before performing actions.
waitUntilExactlyOneExists()
Compose test API that blocks until exactly one node matching the matcher appears. Replaces Thread.sleep().
Retry-and-Ignore Culture
The anti-pattern where flaky tests are retried instead of fixed. Costs CI minutes, erodes trust, and lets real bugs slip through.

🧪 Quick Quiz

A test passes 80% locally but only 60% on CI. What category is this?

Timing-dependent (or possibly environment-dependent). The CI emulator is slower, so race conditions in coroutines or animations manifest more often. Check for missing advanceUntilIdle() or Thread.sleep() instead of proper synchronization.

Why should you never use Thread.sleep() in a UI test?

The delay that works today on your machine will be too short on a slow CI emulator. Use waitUntil, waitUntilExactlyOneExists, or IdlingResource — they wait for the right condition, not for a guessed number of milliseconds.

A test passes alone but fails when the full suite runs. What should you check?

Shared mutable state: database singletons in companion object, static fields, test dispatchers reused across classes. Each test should create and tear down its own state.

What four pieces of information should you give Claude Code when diagnosing a flaky test?

Test name, assertion failure + stacktrace, failure frequency (e.g., "1 in 5 runs"), and whether it fails locally, on CI, or both. The frequency is a strong signal for the flakiness category.

What does MainDispatcherRule do, and what breaks without it?

It replaces Dispatchers.Main with a TestDispatcher so viewModelScope coroutines use the test scheduler. Without it, coroutines use the real Main dispatcher (which doesn't exist in unit tests) and crash or run uncontrolled.

Key Takeaways

  • Flaky tests fall into 4 categories: timing, order, environment, resource. Each has a different fix.
  • Timing flakiness (#1 cause): use runTest + advanceUntilIdle() + MainDispatcherRule for coroutine tests.
  • Never use Thread.sleep() in UI tests. Use waitUntil, IdlingResource, or Turbine instead.
  • Order-dependent tests: create a fresh database per test, never share mutable state via companion object.
  • CI flakiness: disable animations, set locale/timezone explicitly, use semantic matchers.
  • Feed Claude Code the test name, failure log, frequency, and source code — let AI categorize and suggest the fix pattern.
