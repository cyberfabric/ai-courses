Chapter 13

Debugging 16 KB Page Size Compatibility

Memory Map
💡

The Big Idea

If your app ships native code (`.so` files), every one of them must be aligned to 16 KB page boundaries to load on Android 15+ devices that opted into 16 KB mode. The crash is brutal — it’s a launch crash with a single, pattern-matchable signature. The fix is almost always a toolchain bump (NDK r28+, AGP 8.5.1+) or a dependency version bump, not a code change.

🚨 The Crash Signature

📛
dlopen failed
UnsatisfiedLinkError: dlopen failed: "<...>.so" is not page-size compatible.
⚠️
Linker warning
LOAD segment alignment (0x1000) is smaller than system page size (0x4000)
💥
Pre-onCreate launch crash
Fires from a static initializer or System.loadLibrary call — no recovery.
📵
100% reproducible
Pixel 9 with 16 KB toggle, or Android Studio 16 KB emulator. Always crashes — never flaky.

📅 The Deadline Timeline

  1. Nov 1, 2025

    Google Play starts requiring 16 KB compatibility for new submissions and updates targeting Android 15+ that include native code.

  2. Nov 2025 – Apr 2026

    Play Console grants automatic extensions on request. Use this window to migrate — not as an excuse to delay.

  3. May 31, 2026

    Last automatic extension date. After this, non-compliant updates are rejected.

  4. 2026 onward

    More 16 KB devices ship by default. Compatibility shifts from "ship-blocker" to "your existing users start crashing."

🛠️ The Fix Menu

Source of `.so` Fix Effort
First-party (your `cpp/`) Bump NDK to r28+, rebuild 15 min
First-party, locked NDK Add -Wl,-z,max-page-size=16384 + common-page-size=16384 in CMake 15 min
Transitive AAR with fix shipped Bump dependency version (one-line change) 30 min + test
Transitive, no upstream fix File issue, then: rebuild from source OR drop dependency OR fork-and-patch hours – days
Old AGP (<8.5.1) Bump AGP — needed for 16 KB zip-alignment of `lib/` entries 1 hour + test

🧠 Key Terms

Page size
The unit at which the OS hands out memory via mmap(). Historically 4 KB on Android; 16 KB on newer high-RAM devices.
ELF segment alignment
Each LOAD segment in a `.so` declares the page boundary it needs. Encoded as 2**N bytes — 2**14 = 16384 (good), 2**12 = 4096 (broken).
Zip alignment
Inside the APK, native libs must start at a 16 KB-aligned offset so the runtime can mmap() them directly without copying. AGP 8.5.1+ does this automatically.
NDK r28+
The first NDK that emits 16 KB-aligned segments by default. Earlier NDKs need explicit linker flags.
getpagesize() / sysconf(_SC_PAGESIZE)
The runtime-correct way to query page size in C/C++. Replaces hardcoded 4096 or the now-undefined PAGE_SIZE macro.
16 KB toggle
The Pixel 8/8a/9/9a developer setting (Settings → System → Software updates) that boots the device with 16 KB pages — the easiest way to test.

🔍 Detection Tools

📊
APK Analyzer
Android Studio → Build → Analyze APK. Look at the Alignment column on `lib/arm64-v8a/*.so` rows.
📜
check_elf_alignment.sh
Google’s script: scans every `.so` in an APK and prints ALIGNED / UNALIGNED.
🔬
llvm-objdump -p
Surgical inspection of one `.so`. Look for align 2**14 on LOAD lines.
📦
zipalign -c -P 16
Verifies the APK’s zip-level alignment. Should print "Verification successful."
📱
16 KB emulator
SDK Manager → Android 15+ → "Google APIs Experimental 16 KB Page Size" system image.
⚡
adb shell getconf PAGE_SIZE
Returns 16384 if you’re actually on a 16 KB device/emulator. Sanity-check before testing.

📚 Common Transitive Offenders

React Native (pre-0.76)
Hermes & multiple core `.so` files were 4 KB. RN 0.76+ ships aligned. Many third-party RN modules still need their own bump.
Flutter (pre-3.27)
Engine `.so` was 4 KB. Bump the Flutter SDK; plugin authors are mixed — audit each one.
OpenCV Android (pre-4.10.0)
libopencv_java*.so shipped 4 KB. 4.10.0+ is aligned.
Conscrypt (2.5.3)
The famous Pixel 8 launch crash — libconscrypt_jni.so was 4 KB. Fixed in 2.5.4+.
SQLCipher / SQLite native
Crypto libs are common offenders. SQLCipher 4.6.0+ is aligned.
Unity / Unreal exports
The engine ships its own native runtime. Re-export from Unity 6.0 LTS or Unreal 5.5+.

⚠️ The Hardcoded-4096 Trap

❌ #define PAGE_SIZE 4096
❌ char buf[4096]
Use sysconf(_SC_PAGESIZE)
or getpagesize()
❌ mmap(NULL, 4096, ...)
❌ Ring buffer sized to 4096

🧪 Quick Quiz

What does align 2**14 mean in llvm-objdump output?

2^14 = 16384 = 16 KB-aligned. Good. 2**12 = 4096 = 4 KB-aligned and needs to be rebuilt.

You bumped to NDK r28 and rebuilt, but the APK still fails the alignment check. What’s the most likely cause?

Your AGP is older than 8.5.1, so the APK isn’t being zip-aligned with a 16 KB stride even though the `.so` files themselves are aligned. Bump AGP.

The crash log names libopencv_java4.so. What’s the cheapest fix?

Bump the OpenCV Android dependency to 4.10.0 or newer. Don’t fork or patch — the upstream fix exists.

Why is PAGE_SIZE 4096 in your C++ code dangerous on a 16 KB device, even if your `.so` files are aligned?

Code that allocates "one page" gets 4 KB instead of the OS’s 16 KB, so adjacent allocations may overlap, ring buffers may corrupt data, and mmap() calls may behave unexpectedly. Use sysconf(_SC_PAGESIZE) at runtime.

How do you turn a Pixel 9 into a 16 KB test device?

Settings → System → Software updates → Boot with 16 KB page size. Toggle on, reboot. Verify with adb shell getconf PAGE_SIZE → 16384.

Key Takeaways

  • The 16 KB page size requirement is a hard Google Play gate as of Nov 1, 2025 (extendable to May 31, 2026) for any app shipping native code targeting Android 15+.
  • The crash signature is unmistakable: dlopen failed: "<...>.so" is not page-size compatible — always a launch crash, always 100% reproducible on a 16 KB device.
  • Audit your APK with check_elf_alignment.sh or APK Analyzer’s Alignment column. Anything other than 16 KiB needs to be fixed.
  • First-party code: bump NDK to r28+ (16 KB-aligned by default) and AGP to 8.5.1+ (16 KB zip-alignment). Or add the linker flags as a workaround.
  • Transitive offenders in 2026 include older React Native, Flutter, OpenCV (<4.10.0), Conscrypt 2.5.3, SQLCipher, and old Unity/Unreal exports. Bump the version — almost always the fix.
  • In your own C/C++ code: replace hardcoded 4096 and PAGE_SIZE with sysconf(_SC_PAGESIZE) or getpagesize().
  • Test on a real 16 KB environment: SDK Manager’s "Google APIs Experimental 16 KB Page Size" emulator image, or the Pixel 8/9 "Boot with 16 KB page size" toggle.
  • Ship two skills: /analyze-page-size for APK audits and /diagnose-page-size-crash for triaging crash logs.
