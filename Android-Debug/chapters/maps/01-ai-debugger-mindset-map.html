The Big Idea

Stop debugging alone. AI brings breadth (millions of crash patterns). You bring depth (your codebase and domain knowledge). Together, you debug in minutes what used to take hours.

📊 The Debugging Reality

⏱️
35–50% of Dev Time
Debugging and testing eat up to half your workday. It's the biggest time sink in Android development.
🧠
75% = Understanding
Three-quarters of debugging time is spent figuring out the problem, not writing the fix.
🔄
23 Min Focus Tax
Each debugging interruption costs 23 minutes to regain focus. Fewer guess-cycles = fewer interruptions.
⚡
60% Time Reduction
AI-augmented debugging cuts a typical crash investigation from ~43 minutes to ~17 minutes.

🧩 Core Concepts

🤖
AI Pattern Matching
AI reads stacktraces by matching across millions of known crash patterns simultaneously — exception type, call chain, library combos, API quirks.
📋
CLAUDE.md Setup
Give Claude Code project context: architecture, dependencies, common crash areas, key file paths. Better context = better analysis.
⚡
/debug Skill
A reusable command in .claude/commands/debug.md that takes a stacktrace and returns: exception, crash location, root cause, fix, prevention.
🔍
The 5 Whys
Keep asking "why?" to drill past symptoms to root cause. AI excels at this — it doesn't get impatient or jump to conclusions.
📓
Debugging Journal
Log every resolved bug: symptom, root cause, fix, prevention. Builds a searchable team knowledge base over time.
⚠️
When AI Is Wrong
Watch for: try-catch as fix, invented APIs, too-simple solutions, contradicts your architecture. Always verify before applying.

📖 Key Terms

Stacktrace
The call chain printed when an exception occurs — your primary input for AI-assisted debugging.
Root Cause
The underlying reason a bug happens, not just the line where it crashes. The "5th Why."
CLAUDE.md
Project instruction file that gives Claude Code context about your architecture, dependencies, and key files.
/debug Skill
A reusable Claude Code command stored in .claude/commands/debug.md for structured crash analysis.
5 Whys
Iterative questioning technique: keep asking "why?" to trace symptoms back to the real root cause.
StateFlow
Kotlin coroutine primitive for observable state. The fix pattern for "eager access of async data" crashes.
Debugging Journal
A living document of bugs, root causes, and fixes — your team's searchable knowledge base.
Eager Access of Async Data
Top-5 Android crash cause: synchronously accessing data that hasn't loaded yet (e.g., in a ViewModel constructor).

🔍 The 5 Whys in Action

  1. Why did the app crash?

    NullPointerException calling user.getName()

  2. Why was user null?

    Repository hadn't returned data when loadProfile() ran

  3. Why was loadProfile() called too early?

    It runs in the ViewModel constructor — synchronously

  4. Why does the constructor call it synchronously?

    Original code assumed the user was already cached in Room

  5. Why isn't the user cached?

    New users haven't completed onboarding — no local data exists yet

Real fix target: onboarding flow doesn't pre-populate user data — not just a null check on line 47.

🧪 Quick Quiz

What percentage of debugging time is spent understanding the problem vs. fixing it?

75% is spent understanding the problem. Only 25% is writing the actual fix. AI's biggest win is cutting this understanding phase.

What's the key difference between the traditional and AI-augmented debugging cycles?

The AI cycle replaces blind "guess → rebuild → reproduce" loops with a structured hypothesis. You go: Observe → Hypothesize with AI → Verify → Fix.

Where do you store a reusable /debug skill for Claude Code?

In .claude/commands/debug.md in your project root. Commit it to your repo so the whole team benefits.

Name three red flags that an AI-suggested fix might be wrong.

(1) It suggests try-catch as the fix. (2) It references an API that doesn't exist. (3) The fix is too simple for the problem (e.g., "just add a null check"). Also: it contradicts what you know about your architecture.

What four sections should a debugging journal entry contain?

Symptom (what the user/dev saw), Root Cause (the actual issue), Fix (what changed and why), Prevention (how to avoid this class of bug).

📌 Key Takeaways

  • Set up CLAUDE.md with your project's architecture, dependencies, and common crash areas before you start debugging.
  • Create a /debug skill in .claude/commands/debug.md for consistent, structured crash analysis.
  • Feed AI the full stacktrace + context (reproduction steps, device info, what you've checked) — like talking to a senior colleague.
  • Use the 5 Whys technique to drill past symptoms to the real root cause.
  • Always verify AI suggestions before applying — watch for try-catch fixes, invented APIs, and oversimplifications.
  • Log every resolved bug in a debugging journal — your team's searchable knowledge base grows with every fix.
