The Big Idea

Android Studio's profiler shows you beautiful graphs. The problem? Reading profiler output is a skill unto itself. Extract the key data — hot methods, allocation counts, janky frame breakdowns — feed it to Claude Code, and get a plain-English explanation of what's wrong and how to fix it.

🧩 Core Concepts

🕳️
The Profiler Data Gap
67% of Android devs have the profiler but rarely use it effectively. The gap isn't the tool — it's interpreting the output. AI bridges that gap.
🔥
CPU Method Traces
Record which methods run, how long each takes, and how often. Extract the "Top Down" view — method name, self time, total time, call count — and feed it to AI.
📦
Memory Allocation Tracking
CPU profiling says what's slow. Memory profiling says what's allocating too much. Together they paint a complete picture. Watch for GC pressure.
⚡
System Trace (Perfetto)
The nuclear option: see your app, system framework, GPU, and scheduler all at once. Green frames = on time, red = janky. Extract only the janky frames for AI.
🖼️
Frame Rendering Analysis
Dropped frames = jank users feel. Each frame has phases (input, animation, measure, layout, draw, GPU). Identify which phase blows the 16ms budget.
🧬
Code + Profiler = Full Picture
The power move: feed both profiler output AND source code to Claude Code. AI connects performance data to specific implementation choices — like a network call inside a composable.

🖼️ Frame Rendering Phases

Phase What It Does Slow? Common Cause
Input Process touch events Heavy work in onClick / onTouch handlers
Animation Update animation values Too many concurrent animations, complex interpolators
Measure / Layout Calculate sizes & positions Deep view hierarchy, nested weights, repeated measure passes
Draw Record draw commands Complex custom drawing, large bitmap operations
Sync / Upload Send textures to GPU Large/uncompressed bitmaps, too many layers
GPU / Render Execute draw commands Overdraw, complex shaders, large surface area

📖 Key Terms

Method Trace
A recording of which methods your app calls, how long each takes, and how many times — the raw data for CPU profiling.
Flame Chart
A visual representation of the call stack over time. Wide bars = methods consuming lots of CPU. AI works better with the "Top Down" text view.
Top Down View
A table of methods sorted by total time, showing the call hierarchy. This is what you extract and paste into Claude Code.
Allocation Pressure
Creating too many objects too fast, forcing the garbage collector to run frequently. Each GC pause directly causes dropped frames.
GC Pause
A garbage collection event that pauses your app. 12 GC pauses in 5 seconds = 156ms of lost frame time = visible jank.
Perfetto
Android's system-level tracing tool. Shows everything: your app, system framework, GPU, scheduler, and thread interactions.
Choreographer#doFrame
The system callback that drives each frame. Its duration tells you if the frame was on time (<16ms) or janky (>16ms).
16ms Budget
At 60 FPS, each frame gets ~16ms. At 120 FPS, ~8ms. Anything over = dropped frame = visible jank.
/profile-cpu Skill
A reusable Claude Code command that takes hot methods with call counts and durations and returns bottleneck analysis with fix suggestions.

🧪 Quick Quiz

What format should you extract from the CPU profiler for Claude Code?

The "Top Down" view — method names with self time, total time, and call counts. AI works better with structured text than flame chart screenshots.

Why did 48 visible items cause 96 Bitmap allocations in the chapter's example?

Each item decoded a bitmap from disk (48) AND created a second bitmap for rounded corners (48). Total: 96 allocations. The fix: use Coil/Glide with built-in rounded corner transformations.

When feeding Perfetto frame data to Claude Code, should you paste all 300 frames?

No. Only paste the janky frames (those exceeding 16ms) with their phase breakdowns. AI diagnoses root causes from the slow frames — the fast ones don't add useful signal.

What's the "power move" for getting the best AI profiling analysis?

Feed both the profiler output AND the relevant source code. This lets AI connect performance data to specific implementation choices — like spotting a network call inside a composable body.

A method called NumberFormat.getInstance() appears 48 times in a 5-second trace. What's wrong?

A new NumberFormat instance is created for every list item instead of reusing one. The fix: create a single instance as a class field and reuse it across all onBindViewHolder calls.

✅ Your Profiling Checklist

  1. Profile the specific interaction

    Don't profile the whole app session. Isolate the scroll, transition, or animation that feels slow. 5–10 seconds is enough.

  2. Extract structured data

    Top Down method summaries for CPU. Allocation counts sorted by class for memory. Janky frame breakdowns for Perfetto.

  3. Include source code with profiler data

    Paste both the hot method data AND the relevant code. AI can then connect "14ms in calculateShippingEstimate()" to "synchronous network call during composition."

  4. Fix and re-measure

    Apply the top fix, re-profile the same interaction, compare numbers. Every optimization should be measurable — no guessing.

  5. Build reusable skills

    Create /profile-cpu and similar skills so your team can run the same analysis without crafting prompts from scratch every time.

Key Takeaways

  • The profiler shows data; AI explains what it means. Bridge the gap by extracting structured text for Claude Code.
  • For CPU profiling, extract the "Top Down" view with method names, self time, total time, and call counts.
  • Memory profiling reveals allocation pressure — too many objects forcing GC pauses that cause jank.
  • Perfetto system traces show frame-level timing. Only feed janky frames (>16ms) to AI, not every frame.
  • The most powerful technique: feed both profiler output AND source code so AI can connect data to implementation.
  • Every optimization must be measurable — profile before, fix, profile after. No guessing.
