Crash Logs, Logcat & Stacktraces

Master the art of feeding Android log output to AI and getting actionable analysis back

From Mindset to Mechanics

In Chapter 1, you learned why AI-assisted debugging works. Now we get into the how. This chapter is about the raw material of Android debugging: crash logs, logcat output, and stacktraces. These are the artifacts your app produces when something goes wrong — and they’re exactly what AI is best at analyzing.

By the end of this chapter, you’ll have a reusable /analyze-crash skill that takes messy logcat output and returns a structured diagnosis every time.

Anatomy of an Android Stacktrace

Before we feed anything to AI, let’s make sure you know what each part of a stacktrace means. Think of a stacktrace as a snapshot of the call stack at the moment something exploded — like a freeze-frame of dominoes mid-fall.

FATAL EXCEPTION: main
Process: com.example.myapp, PID: 18432
java.lang.IllegalStateException: Fragment ProfileFragment{a1b2c3d}
  not attached to a context.
    at androidx.fragment.app.Fragment.requireContext(Fragment.java:952)
    at com.example.myapp.ui.profile.ProfileFragment.updateUI(ProfileFragment.kt:87)
    at com.example.myapp.ui.profile.ProfileFragment$loadData$1.invokeSuspend(ProfileFragment.kt:54)
    at kotlin.coroutines.jvm.internal.BaseContinuationImpl.resumeWith(ContinuationImpl.kt:33)
    at kotlinx.coroutines.DispatchedTask.run(DispatchedTask.kt:108)
    at android.os.Handler.handleCallback(Handler.java:958)
    at android.os.Looper.loop(Looper.java:230)
    at android.app.ActivityThread.main(ActivityThread.java:8232)

Here’s what each section tells you:

Part What It Means What AI Needs
FATAL EXCEPTION: main Crash on the main (UI) thread Thread context — main thread crashes have different causes than background thread crashes
Process: com.example.myapp Your app’s package name Helps AI filter your code from framework code
IllegalStateException The exception type AI maps this to known Android patterns instantly
Fragment...not attached to a context The specific error message Critical for narrowing down the cause
The stack frames Call chain from crash point to entry point AI traces this to find where your code triggered the issue
Always paste the complete stacktrace, including the FATAL EXCEPTION header and the thread name. AI uses the thread name to determine if this is a lifecycle issue (main thread), a threading bug (background thread), or a system callback issue.

Logcat: Filtering the Noise

Raw logcat is a firehose. On a busy device, you’ll see hundreds of lines per second from system services, other apps, and garbage collection. You need to filter before you analyze.

Essential Logcat Commands

# Filter by your app's package (most common)
adb logcat --pid=$(adb shell pidof com.example.myapp)

# Filter FATAL crashes only
adb logcat "*:E" | grep -A 20 "FATAL EXCEPTION"

# Filter by tag (when you know the component)
adb logcat -s "ProfileFragment" "ProfileViewModel"

# Save to file for AI analysis
adb logcat -d > crash_log.txt

# Clear and start fresh before reproducing a bug
adb logcat -c && adb logcat --pid=$(adb shell pidof com.example.myapp)

The -d flag dumps the current log buffer and exits — perfect for capturing a snapshot to paste into Claude Code. The --pid filter is your best friend: it shows only your app’s output, cutting out 95% of the noise.

Prompt
Here's my logcat: [paste 2000 lines of unfiltered logcat including system logs, GC events, and other apps]
Too much noise. AI will spend tokens parsing irrelevant system logs and may miss the actual crash.
Prompt
App crashes when navigating to the profile screen after login. Here's the filtered crash log from logcat (PID-filtered for my app only): [paste 30-50 lines of relevant logcat around the crash] Using: Fragment navigation, Hilt, coroutines for data loading.
Filtered output + context + reproduction steps = precise AI analysis.

Reading FATAL EXCEPTION Blocks

Not all crashes are simple single-exception affairs. Android loves to chain exceptions together with “Caused by” blocks. Here’s where things get interesting — and where AI really shines.

Multi-Cause Crash: Chained Exceptions

FATAL EXCEPTION: main
Process: com.example.shopapp, PID: 21503
java.lang.RuntimeException: Unable to start activity
  ComponentInfo{com.example.shopapp/.ui.checkout.CheckoutActivity}:
  android.database.sqlite.SQLiteException: no such table: cart_items
  (code 1 SQLITE_ERROR)
    at android.app.ActivityThread.performLaunchActivity(ActivityThread.java:3782)
    at android.app.ActivityThread.handleLaunchActivity(ActivityThread.java:3922)
    at android.app.servertransaction.LaunchActivityItem.execute(LaunchActivityItem.java:103)
    ...
Caused by: android.database.sqlite.SQLiteException: no such table:
  cart_items (code 1 SQLITE_ERROR)
    at android.database.sqlite.SQLiteConnection.nativeExecuteForCursorWindow(Native Method)
    at android.database.sqlite.SQLiteSession.executeForCursorWindow(SQLiteSession.java:838)
    at com.example.shopapp.data.db.CartDao_Impl$4.call(CartDao_Impl.java:112)
    at com.example.shopapp.data.db.CartDao_Impl$4.call(CartDao_Impl.java:97)
    at androidx.room.RoomDatabase.query(RoomDatabase.java:480)
    at com.example.shopapp.data.db.CartDao_Impl.getCartItems(CartDao_Impl.java:103)
    at com.example.shopapp.ui.checkout.CheckoutViewModel.<init>(CheckoutViewModel.kt:28)
    at com.example.shopapp.ui.checkout.CheckoutActivity.onCreate(CheckoutActivity.kt:34)
    ... 12 more

What Claude Code Finds

Claude Code reads the chain bottom-up: (1) The real error is in the Caused by block — SQLiteException: no such table: cart_items. (2) This is called from CartDao_Impl.getCartItems(), which is a Room-generated DAO. (3) The table doesn’t exist because a Room migration is missing — the cart_items table was added in a new version but the database wasn’t migrated. (4) This happens at Activity launch, inside the ViewModel constructor.

Root Cause: Missing Room Database Migration

The cart_items table was added to the Room @Database entity list, and the database version was bumped, but no Migration object was provided. Room tried to query a table that doesn’t exist in the old database schema.

This is the #1 Room crash pattern. It only hits users who upgrade the app — fresh installs work fine because Room creates all tables from scratch.

Fix: Add the Missing Migration

// In your AppDatabase companion object:
val MIGRATION_3_4 = object : Migration(3, 4) {
    override fun migrate(db: SupportSQLiteDatabase) {
        db.execSQL("""
            CREATE TABLE IF NOT EXISTS `cart_items` (
                `id` INTEGER PRIMARY KEY AUTOINCREMENT NOT NULL,
                `product_id` TEXT NOT NULL,
                `quantity` INTEGER NOT NULL DEFAULT 1,
                `added_at` INTEGER NOT NULL
            )
        """)
    }
}

// In your database builder:
Room.databaseBuilder(context, AppDatabase::class.java, "shop.db")
    .addMigrations(MIGRATION_3_4)
    .build()

Pro tip: Also add .fallbackToDestructiveMigration() in debug builds so you don’t crash during development, but never in release builds — it deletes all user data.

The “Caused by” Rule

When a stacktrace has multiple “Caused by” blocks, the last one is usually the real culprit. The ones above it are just wrappers that caught and re-threw the exception.

Think of it like a crime scene: the top of the stacktrace is where the body was found. The “Caused by” chain leads you back to where the crime actually happened.

Tell Claude Code to “focus on the deepest Caused by block” when pasting multi-cause crashes. This directs AI to the root exception instead of the wrapper.

ProGuard/R8 Obfuscated Stacktraces

Release builds run through R8 (Android’s code shrinker), which renames your classes and methods to things like a.b.c.d(). A stacktrace from a release build looks like gibberish:

java.lang.NullPointerException
    at c.a.b.f.a(SourceFile:12)
    at c.a.b.e.b(SourceFile:45)
    at c.a.a.d.onClick(SourceFile:8)

You must deobfuscate before feeding to AI. AI can’t pattern-match on obfuscated names.

Deobfuscation Steps

# 1. Find your mapping file (generated at build time)
# Location: app/build/outputs/mapping/release/mapping.txt

# 2. Use the retrace tool (bundled with Android SDK)
$ANDROID_HOME/cmdline-tools/latest/bin/retrace \
    app/build/outputs/mapping/release/mapping.txt \
    obfuscated_stacktrace.txt

# 3. Or use the ProGuard retrace tool directly
java -jar $ANDROID_HOME/tools/proguard/lib/retrace.jar \
    mapping.txt stacktrace.txt
Keep your mapping files! Store mapping.txt for every release build. Upload them to Firebase Crashlytics, or archive them alongside your APK/AAB. Without the mapping file, obfuscated stacktraces from production are unreadable.

Once you have the deobfuscated stacktrace, feed it to Claude Code exactly like any other crash log.

Building the /analyze-crash Skill

In Chapter 1, you built a general /debug skill. Now let’s build a specialized one for Android crash logs. This skill understands logcat format, Caused-by chains, and Android-specific patterns.

Create .claude/commands/analyze-crash.md:

# /analyze-crash — Android Crash Log Analysis

Analyze raw logcat output or a stacktrace from an Android app crash.

## Input
The user will paste one of:
- A FATAL EXCEPTION block from logcat
- A stacktrace from Crashlytics or Play Console
- A section of logcat output containing errors

## Analysis Steps

1. **Identify all exceptions** in the Caused-by chain. Focus on the
   deepest (last) Caused-by — that's usually the root exception.

2. **Classify the crash type:**
   - Lifecycle issue (Fragment not attached, Activity destroyed)
   - Threading issue (CalledFromWrongThreadException, main thread IO)
   - Data issue (NPE, SQLiteException, JsonSyntaxException)
   - Resource issue (OOM, BadTokenException, WindowLeaked)
   - Third-party library issue (crash inside OkHttp, Retrofit, Glide)

3. **Trace to user code** — find the first stack frame that belongs
   to the app (not androidx.*, kotlin.*, or java.*).

4. **Identify the root cause** — WHY did this happen, not just WHAT.

5. **Suggest a fix** with Kotlin code.

6. **Rate the severity**: Critical (data loss/security), High (crash
   for many users), Medium (crash for edge case), Low (cosmetic).

## Output Format

### Exception Chain
[List each exception in the chain, deepest first]

### Crash Classification
[Lifecycle / Threading / Data / Resource / Library]

### Your Code Entry Point
[File:line where your code triggered this]

### Root Cause
[Why this happened]

### Suggested Fix
```kotlin
[Code]
```

### Severity: [Critical/High/Medium/Low]

### Prevention
[How to prevent this class of bug in the future]
The crash classification step is what makes this skill powerful. By categorizing the crash type, AI can apply the right mental model: lifecycle bugs need lifecycle-aware fixes, threading bugs need dispatcher fixes, data bugs need null-safety fixes.

Multi-Device Crashes

The same code can crash on a Samsung Galaxy S24 and work fine on a Pixel 8. Welcome to Android fragmentation. AI helps here because it knows about manufacturer-specific quirks and API-level differences.

When reporting a multi-device crash to Claude Code, include:

This crash happens on Samsung devices running Android 13 but not
on Pixel devices running Android 14. Here are both stacktraces:

## Samsung Galaxy S24 (Android 13, One UI 6.0):
[paste stacktrace]

## Pixel 8 (Android 14):
Works fine — no crash.

Our minSdk is 24, targetSdk is 35.
We use FLAG_IMMUTABLE for PendingIntents.

Claude Code will look for manufacturer-specific behaviors: Samsung’s aggressive battery optimization, Xiaomi’s custom permission system, or API-level differences in how PendingIntent flags are handled.

Third-Party Library Crashes

Sometimes the stacktrace points entirely into library code. You see okhttp3.internal. or retrofit2.adapter. and none of your own code appears. These are tricky because you can’t just “fix the library.”

The approach: ask AI to work backward from the library crash to your usage.

Prompt
This crash happens inside OkHttp and none of my code appears in the stacktrace. But it only happens when I call the /api/checkout endpoint. Here's the stacktrace: [paste stacktrace] Here's my Retrofit interface for this endpoint: [paste CheckoutApi.kt] And here's my OkHttp client configuration: [paste NetworkModule.kt]
By providing your usage code alongside the library stacktrace, AI can spot the misconfiguration — maybe a wrong content type, a missing interceptor, or an incorrect timeout setting.

The Crash Pattern Library

AI recognizes certain Android crash patterns instantly because they’re so common. Here are the ones you’ll see most often:

Exception Common Cause AI’s Usual Diagnosis
IllegalStateException: Fragment not attached Coroutine completes after Fragment is destroyed Use viewLifecycleOwner.lifecycleScope instead of lifecycleScope
BadTokenException Showing dialog after Activity is finishing Check isFinishing and isDestroyed before showing dialogs
DeadObjectException IPC to a process that died (often System UI) Wrap in try-catch; usually transient and system-level
TransactionTooLargeException Saving too much data in onSaveInstanceState Use ViewModel to hold data; save only IDs in Bundle
SQLiteException: no such table Missing Room migration Add the migration or export/verify schema
SecurityException: Permission denied Missing runtime permission check on Android 6+ Add runtime permission request before the API call
InflateException Missing attribute, wrong theme, or broken custom View Check the nested Caused-by for the real layout error

When you feed any of these to Claude Code, it will typically nail the cause in seconds. These patterns are bread and butter for AI analysis.

Real-World Walkthrough: Fragment Lifecycle Crash

Let’s trace a full debugging session from logcat to fix. This is the crash that haunts every Android dev who uses coroutines with Fragments.

The Crash

FATAL EXCEPTION: main
Process: com.example.newsapp, PID: 9821
java.lang.IllegalStateException: Fragment ArticleFragment{e4f5a6b}
  not attached to a context.
    at androidx.fragment.app.Fragment.requireContext(Fragment.java:952)
    at com.example.newsapp.ui.article.ArticleFragment.showArticle(ArticleFragment.kt:72)
    at com.example.newsapp.ui.article.ArticleFragment$fetchArticle$1.invokeSuspend(ArticleFragment.kt:45)
    at kotlin.coroutines.jvm.internal.BaseContinuationImpl.resumeWith(ContinuationImpl.kt:33)
    at kotlinx.coroutines.DispatchedTask.run(DispatchedTask.kt:108)
    at android.os.Handler.handleCallback(Handler.java:958)
    at android.os.Looper.loop(Looper.java:230)
    at android.app.ActivityThread.main(ActivityThread.java:8232)

Claude Code’s Analysis

Exception: IllegalStateException — Fragment not attached to a context.

Classification: Lifecycle issue.

Entry point: ArticleFragment.kt:45 — a coroutine launched in fetchArticle() that calls showArticle() after the Fragment is detached.

Root cause: The coroutine was launched with lifecycleScope (Activity-scoped) instead of viewLifecycleOwner.lifecycleScope (view-scoped). When the user navigates away, the Fragment’s view is destroyed but the coroutine keeps running. When it completes and calls requireContext(), the Fragment is already detached.

Fix: Use viewLifecycleOwner.lifecycleScope

// BEFORE (crashes):
class ArticleFragment : Fragment() {
    private fun fetchArticle(id: String) {
        lifecycleScope.launch {
            val article = repository.getArticle(id)
            showArticle(article)  // Fragment might be detached!
        }
    }
}

// AFTER (safe):
class ArticleFragment : Fragment() {
    private fun fetchArticle(id: String) {
        viewLifecycleOwner.lifecycleScope.launch {
            val article = repository.getArticle(id)
            showArticle(article)  // Automatically cancelled when view is destroyed
        }
    }
}

viewLifecycleOwner.lifecycleScope is tied to the Fragment’s view lifecycle, not the Fragment itself. When the view is destroyed (on navigation), the coroutine is automatically cancelled. No more “not attached to a context” crashes.

Your Crash Analysis Checklist

  • Filter logcat output before pasting — use --pid or tag filters
  • Always include the full FATAL EXCEPTION header with thread name
  • For chained exceptions, look at the deepest Caused-by block
  • Deobfuscate R8/ProGuard stacktraces before feeding to AI
  • Include context: device info, reproduction steps, and relevant code
  • For multi-device crashes, provide stacktraces from both passing and failing devices
  • Set up the /analyze-crash skill for repeatable, structured analysis

The Key Takeaway

The quality of AI crash analysis is directly proportional to the quality of the input you give it. Filter your logcat. Deobfuscate your stacktraces. Include context. The 2 minutes you spend preparing the input saves 20 minutes of vague AI output and follow-up questions.

Hands-On: Build and Test Your /analyze-crash Skill

  1. Create the skill: Copy the /analyze-crash template from this chapter into .claude/commands/analyze-crash.md in your Android project.
  2. Capture a crash: Run your app, navigate to a screen, and either find an existing crash or trigger one. Use adb logcat --pid=$(adb shell pidof your.package) to capture only your app’s output.
  3. Run the skill: In Claude Code, type /analyze-crash and paste the crash log. Review the structured output — did it correctly classify the crash type?
  4. Test with a chained exception: If you have a Room database, try bumping the version number without adding a migration. Capture the SQLiteException and feed it to Claude Code. The AI should identify the missing migration immediately.
  5. Bonus — obfuscated crash: If you have a release build, capture a crash from it, then deobfuscate with retrace and feed both the obfuscated and deobfuscated versions to AI. Compare the analysis quality.
