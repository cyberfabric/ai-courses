What Performance Budgets Are

A performance budget is a concrete threshold: a number you commit to not exceeding. Not “the app should be fast” — that’s a wish. A budget is: “cold start must be under 800ms on a Pixel 6.”

<800 ms Cold start budget (time to first frame)
<16 ms Frame budget (P95 frame render time)
99.5% Crash-free rate budget

Budgets make performance measurable and enforceable. Without them, performance degrades with every release because nobody notices the extra 50ms here, the extra 2MB there. With budgets, CI fails when you exceed them.

Benchmark Libraries: Measuring Budgets Automatically

Budgets are useless if you only check them manually. Android’s Jetpack Benchmark libraries run performance tests in CI and fail when budgets are exceeded.

Macrobenchmark: Startup and Scroll

// Create a :macrobenchmark module
// build.gradle.kts:
plugins {
    id("com.android.test")
}

android {
    targetProjectPath = ":app"
    experimentalProperties["android.experimental.self-instrumenting"] = true
}

dependencies {
    implementation("androidx.benchmark:benchmark-macro-junit4:1.4.1")
}

// StartupBenchmark.kt:
@RunWith(AndroidJUnit4::class)
class StartupBenchmark {
    @get:Rule
    val benchmarkRule = MacrobenchmarkRule()

    @Test
    fun startupCold() = benchmarkRule.measureRepeated(
        packageName = "com.example.myapp",
        metrics = listOf(StartupTimingMetric()),
        iterations = 5,
        startupMode = StartupMode.COLD
    ) {
        pressHome()
        startActivityAndWait()
    }

    @Test
    fun scrollProductList() = benchmarkRule.measureRepeated(
        packageName = "com.example.myapp",
        metrics = listOf(FrameTimingMetric()),
        iterations = 5,
        startupMode = StartupMode.WARM
    ) {
        startActivityAndWait()
        device.findObject(By.res("product_list")).fling(Direction.DOWN)
        device.waitForIdle()
    }
}

Microbenchmark: Hot Code Paths

// For measuring specific functions:
@RunWith(AndroidJUnit4::class)
class PriceFormatterBenchmark {
    @get:Rule
    val benchmarkRule = BenchmarkRule()

    @Test
    fun formatPrice() = benchmarkRule.measureRepeated {
        PriceFormatter.format(49.99, "USD")
    }
}

What Claude Code Finds

(1) Cold start is 447ms over budget and 312ms worse than the previous run — a clear regression. This likely correlates with a recent dependency addition, initialization code in Application.onCreate(), or a new ContentProvider. (2) Scroll performance is within budget at P50 and P95, but P99 exceeds the frame budget — there are occasional janky frames. (3) The 3.1% jank rate suggests an intermittent issue, possibly GC pauses or one expensive item in the list.

Root Cause: New SDK Initialization in Application.onCreate()

The 312ms regression correlates with a recent commit that added a new analytics SDK. The SDK’s initialize() method runs synchronously on the main thread during Application.onCreate(), adding ~300ms to cold start. This is a common pattern: SDKs that look like one-line additions but carry hidden startup costs.

Fix: Defer SDK Initialization with App Startup Library

// BEFORE: Synchronous initialization blocks startup
class ShopApplication : Application() {
    override fun onCreate() {
        super.onCreate()
        AnalyticsSDK.initialize(this)     // 300ms blocking!
        CrashReporter.initialize(this)     // 50ms
        FeatureFlags.initialize(this)      // 80ms
        ImageLoader.initialize(this)       // 40ms
    }
}

// AFTER: Use App Startup for lazy/async initialization
// 1. Remove manual init from Application.onCreate()
// 2. Create initializers:
class AnalyticsInitializer : Initializer<AnalyticsSDK> {
    override fun create(context: Context): AnalyticsSDK {
        return AnalyticsSDK.initialize(context)
    }
    override fun dependencies() = emptyList<Class<Initializer<*>>>()
}

// 3. For non-critical SDKs, defer to background:
class ShopApplication : Application() {
    override fun onCreate() {
        super.onCreate()
        // Only critical SDK (crash reporter) on main thread
        CrashReporter.initialize(this)

        // Defer everything else
        lifecycleScope.launch(Dispatchers.Default) {
            AnalyticsSDK.initialize(this@ShopApplication)
            FeatureFlags.initialize(this@ShopApplication)
        }
    }
}

Result: Cold start drops by ~380ms because only the crash reporter blocks the main thread. Analytics and feature flags initialize in the background while the first screen renders.

Building a /check-budgets Skill

Create .claude/commands/check-budgets.md:

# /check-budgets — Performance Budget Check

Compare current benchmark results against performance budgets
and flag violations.

## Input
The user will provide benchmark results from Macrobenchmark,
Microbenchmark, or manual profiling, along with their budget targets.

## Analysis Steps

1. **Compare each metric against its budget** — flag anything over.
2. **Identify regressions** — compare against previous results if
   provided. Any increase > 10% is a regression worth investigating.
3. **Rank violations by user impact:**
   - Startup time: directly affects first impression
   - Frame time: affects perceived smoothness
   - APK size: affects install conversion
   - Memory: affects background kill priority
   - Crash rate: affects retention
4. **Suggest which budgets to tackle first** based on how far
   over budget and how many users are affected.

## Output Format

### Budget Status
| Metric | Budget | Actual | Status |
|--------|--------|--------|--------|
[Table of all metrics]

### Violations (ranked by impact)
[Each violation with: how far over, likely cause, suggested fix]

### Regressions
[Metrics that got worse since last check]

### Recommended Actions
[Top 3 things to fix this sprint]

APK Size Analysis

APK size is the easiest budget to exceed and the hardest to fix after the fact. Every library, every image asset, every unused class adds up.

# Analyze APK size breakdown:
$ANDROID_HOME/cmdline-tools/latest/bin/apkanalyzer \
    apk summary app-release.apk

# Top files by size:
$ANDROID_HOME/cmdline-tools/latest/bin/apkanalyzer \
    files list --order-by SIZE app-release.apk | head -20

# Check for unused resources:
$ANDROID_HOME/cmdline-tools/latest/bin/apkanalyzer \
    resources unused app-release.apk
Feed the apkanalyzer output to Claude Code: “Here are the top 20 files by size in my APK. Which ones can I reduce or eliminate?” AI often spots: uncompressed PNGs that should be WebP, unused native libraries for architectures you don’t need, and bundled fonts that could use system fonts instead.

The Debugging Decision Tree

When something goes wrong, you need to know which tool to reach for. Here’s the decision tree that routes you to the right chapter and skill:

Symptom Tool / Skill Chapter
App crashes with a stacktrace /analyze-crash Ch 2
App freezes / “not responding” /analyze-anr Ch 3
LeakCanary notification /analyze-leak Ch 4
UI is janky / dropped frames /analyze-recomposition or /profile-cpu Ch 5, 9
API call fails / 401 / timeout /analyze-network Ch 6
Build is slow /analyze-build Ch 7
Battery drain complaint /analyze-battery Ch 8
Test passes sometimes, fails sometimes /diagnose-flaky Ch 10
Production crash spike /triage-crash Ch 11
Benchmark exceeds budget /check-budgets Ch 12

The 30-Day Debugging Challenge

Knowledge without practice evaporates. Here’s a 30-day plan to make AI-assisted debugging a habit:

Week 1: Crash Analysis (Chapters 1–3)

  • Day 1–2: Set up CLAUDE.md and /debug skill in your project
  • Day 3–4: Triage 3 real crashes from Crashlytics using /analyze-crash
  • Day 5: Pull an ANR trace and run /analyze-anr on it

Week 2: Profiling (Chapters 4–6)

  • Day 6–7: Run LeakCanary, analyze any leaks with /analyze-leak
  • Day 8–9: Enable Compose compiler metrics, run /analyze-recomposition
  • Day 10: Capture OkHttp logs from a failing flow, run /analyze-network

Week 3: Infrastructure (Chapters 7–9)

  • Day 11–12: Profile your Gradle build, run /analyze-build, apply top fix
  • Day 13–14: Capture battery stats, run /analyze-battery
  • Day 15: CPU-profile a slow screen, run /profile-cpu

Week 4: Tests and Production (Chapters 10–12)

  • Day 16–17: Fix your team’s most notorious flaky test with /diagnose-flaky
  • Day 18–19: Generate your first /weekly-crash-report, share with team
  • Day 20: Set up performance budgets and Macrobenchmark for cold start
