Chapter 3

ANR Diagnosis and Prevention

The silent app-killer that users never forgive — diagnosed by AI in seconds

Worse Than a Crash

In Chapter 2, we mastered crash analysis — the app blows up, you get a stacktrace, AI tells you why. ANRs are a different beast entirely. The app doesn’t crash. It freezes. The UI stops responding. The user taps, swipes, presses Back — nothing happens. After 5 seconds, Android shows the dreaded “App isn’t responding” dialog.

And then the user uninstalls your app. Quietly. Without leaving a review. You never even know it happened.

47% of users uninstall after experiencing an ANR
2x penalty weight of ANRs vs crashes in Play Store vitals
5 sec is all it takes — main thread blocked for 5 seconds = ANR

Google Play Console tracks ANR rates alongside crashes, and a high ANR rate will hurt your app’s visibility in the Play Store. The bad-behavior threshold is 0.47% — meaning if roughly 1 in 200 daily sessions experiences an ANR, your app gets flagged.

The Three ANR Triggers

ANRs aren’t random. Android triggers them in exactly three scenarios:

ANR Type Trigger Timeout
Input dispatching Main thread doesn’t respond to a touch/key event 5 seconds
BroadcastReceiver Receiver’s onReceive() doesn’t complete 10 seconds (foreground) / 60 seconds (background)
Service Service’s onCreate(), onStartCommand(), or onBind() doesn’t complete 20 seconds (foreground) / 200 seconds (background)

The input dispatching ANR is by far the most common. It means your main thread is busy doing something it shouldn’t be — database queries, network calls, heavy computation, or waiting on a lock held by another thread.

Reading ANR Traces

When an ANR happens, Android dumps a thread trace to /data/anr/traces.txt. This file is the ANR equivalent of a crash stacktrace, except it shows every thread in your process and what each one was doing at the moment of the ANR.

How to Pull ANR Traces

# Pull the ANR trace file from the device
adb pull /data/anr/traces.txt

# On newer Android versions, you might need:
adb bugreport bugreport.zip
# Then extract and find the ANR traces inside

# For quick access during development:
adb shell cat /data/anr/traces.txt | head -200
On Android 11+, direct access to /data/anr/traces.txt may be restricted. Use adb bugreport instead, which bundles ANR traces along with system logs, memory info, and more. Claude Code can analyze the relevant sections from a bugreport too.

What Claude Code Finds

Thread “main” is in state Blocked, waiting to acquire a lock on the Room database (0x02a3e5c8). Thread “RoomQueryThread-2” (tid=14) holds that lock and is running a long inventory sync query. The main thread is calling searchProducts() directly from afterTextChanged — a synchronous database query triggered on every keystroke in the search bar.

Root Cause: Synchronous Database Query on Main Thread + Lock Contention

Two problems compound here: (1) SearchViewModel.onQueryChanged() runs a Room query synchronously on the main thread. (2) That query can’t proceed because the database lock is held by a background sync operation running a long query.

Either problem alone might not cause an ANR. Together — synchronous main-thread query + lock contention with a slow background query — the main thread stalls for 5+ seconds.

Fix: Move Query Off Main Thread + Debounce Input

class SearchViewModel @Inject constructor(
    private val productDao: ProductDao
) : ViewModel() {

    private val _query = MutableStateFlow("")

    val searchResults: StateFlow<List<Product>> = _query
        .debounce(300)  // Don't query on every keystroke
        .distinctUntilChanged()
        .flatMapLatest { query ->
            if (query.length < 2) flowOf(emptyList())
            else productDao.searchProductsFlow(query)  // Room Flow runs on IO
        }
        .stateIn(viewModelScope, SharingStarted.Lazily, emptyList())

    fun onQueryChanged(text: String) {
        _query.value = text  // Just update the flow, no blocking
    }
}

Key changes: (1) Use Room’s Flow return type, which automatically runs on a background thread. (2) Debounce by 300ms so we don’t fire a query on every keystroke. (3) The main thread never touches the database directly.

Building an /analyze-anr Skill

ANR traces are more complex than crash stacktraces because you need to analyze multiple threads and their interactions. Let’s build a skill that handles this.

Create .claude/commands/analyze-anr.md:

# /analyze-anr — ANR Trace Analysis

Analyze an ANR trace dump and identify the blocked thread, lock holder,
and root cause.

## Input
The user will paste an ANR trace (from /data/anr/traces.txt or a
bugreport). It may include multiple thread dumps.

## Analysis Steps

1. **Find the main thread** — look for `"main" prio=5 tid=1` and check
   its state (Blocked, Waiting, Sleeping, or Runnable doing heavy work).

2. **If Blocked:** identify what lock/monitor it's waiting on, and find
   which thread holds that lock. Follow the chain — sometimes thread A
   waits on B, which waits on C (deadlock potential).

3. **If Runnable:** the main thread is doing CPU-intensive work. Look at
   the top stack frames to identify what computation is blocking.

4. **If Waiting/Sleeping:** the main thread is waiting for a callback,
   future, or synchronous IPC that hasn't returned.

5. **Trace to user code** — find the stack frame in the app's package
   (not android.*, java.*, or androidx.*).

6. **Classify the ANR type:**
   - Database on main thread (Room, SQLite, SharedPreferences)
   - Network on main thread (OkHttp, HttpURLConnection)
   - Lock contention (main thread waiting for background thread's lock)
   - Heavy computation (JSON parsing, bitmap processing, list sorting)
   - Deadlock (two threads each waiting for each other's lock)
   - Binder IPC (slow system service call)

## Output Format

### Main Thread State
[State and what it's doing]

### Blocked By
[Which thread holds the lock / what operation is slow]

### ANR Classification
[Database / Network / Lock / Computation / Deadlock / IPC]

### Your Code Entry Point
[File:line in your code that started this]

### Root Cause
[Why the main thread is blocked]

### Suggested Fix
```kotlin
[Code showing the fix]
```

### Prevention
[How to prevent this class of ANR]

Common ANR Patterns AI Catches Instantly

After analyzing thousands of ANR traces, certain patterns emerge over and over. Here are the ones AI recognizes immediately:

Pattern 1: SharedPreferences on Main Thread

This is the sneakiest ANR cause because SharedPreferences looks harmless. A quick getString() — what could go wrong?

// This looks innocent but can ANR:
val token = prefs.getString("auth_token", null)

// Why? SharedPreferences.getString() is synchronous.
// First call loads the entire XML file from disk.
// If another thread is calling commit(), getString()
// blocks until the commit finishes.
// On a phone with slow flash storage = ANR.
Migrate from SharedPreferences to DataStore (Preferences DataStore) for any data accessed on the main thread. DataStore is fully asynchronous and won’t block.

Pattern 2: Dispatchers.Main Where Dispatchers.IO Should Be

// ANR waiting to happen:
viewModelScope.launch {  // Default: Dispatchers.Main
    val users = database.userDao().getAllUsers()  // Blocking IO!
    _users.value = users
}

// Safe version:
viewModelScope.launch {
    val users = withContext(Dispatchers.IO) {
        database.userDao().getAllUsers()
    }
    _users.value = users
}

viewModelScope.launch defaults to Dispatchers.Main. If your Room DAO returns a plain List<User> instead of Flow<List<User>>, that query runs on the main thread. Room will actually throw an exception by default, but some developers disable this check with allowMainThreadQueries() — which is just asking for ANRs.

Never use .allowMainThreadQueries() in production code. Room’s main-thread check exists to prevent exactly this class of ANRs. If you need a synchronous-looking API, use suspend functions or Flow.

Pattern 3: Binder IPC Stalls

Sometimes the main thread isn’t blocked by your code at all — it’s waiting for a system service. You’ll see stack frames like:

at android.os.BinderProxy.transactNative(Native Method)
at android.os.BinderProxy.transact(BinderProxy.java:584)
at android.content.pm.IPackageManager$Stub$Proxy.getPackageInfo(...)
at android.app.ApplicationPackageManager.getPackageInfo(...)

When Claude Code sees this pattern, it identifies it as a Binder IPC stall — the system’s PackageManager service is overloaded. This is often transient and device-specific, but if it happens frequently, the fix is to move the getPackageInfo() call off the main thread.

StrictMode: Your ANR Early Warning System

StrictMode detects accidental main-thread violations before they become ANRs. Think of it as a smoke detector — it alerts you to the fire before the building burns down.

// Add to your Application.onCreate() — debug builds only:
if (BuildConfig.DEBUG) {
    StrictMode.setThreadPolicy(
        StrictMode.ThreadPolicy.Builder()
            .detectDiskReads()
            .detectDiskWrites()
            .detectNetwork()
            .detectCustomSlowCalls()
            .penaltyLog()  // Log violations to logcat
            .penaltyFlashScreen()  // Flash the screen red (very visible!)
            .build()
    )
    StrictMode.setVmPolicy(
        StrictMode.VmPolicy.Builder()
            .detectLeakedClosableObjects()
            .detectLeakedRegistrationObjects()
            .detectLeakedSqlLiteObjects()
            .penaltyLog()
            .build()
    )
}

Once StrictMode is enabled, every disk read, disk write, or network call on the main thread gets logged. Feed these logs to Claude Code:

Prompt
StrictMode is flagging these violations in my app. Which ones are real ANR risks vs false positives? Here are the logcat entries: D/StrictMode: StrictMode policy violation: android.os.strictmode.DiskReadViolation at com.example.myapp.ui.home.HomeFragment.onViewCreated(HomeFragment.kt:38) at ... D/StrictMode: StrictMode policy violation: android.os.strictmode.DiskReadViolation at android.content.res.AssetManager.nativeGetResourceValue(Native Method) at ...
The first violation (your code reading from disk in onViewCreated) is a real risk. The second (AssetManager) is a framework-level read that's typically fast and safe — AI helps you triage which violations matter.

Building a /check-anr-risks Skill

Prevention is better than diagnosis. Let’s build a skill that scans your code for common ANR patterns before they ship.

Create .claude/commands/check-anr-risks.md:

# /check-anr-risks — Scan for Main-Thread Violations

Scan the codebase for code patterns that could cause ANRs.

## What to Look For

1. **Database on main thread:**
   - Room DAO calls not wrapped in withContext(Dispatchers.IO)
   - SharedPreferences.edit().commit() (use apply() instead)
   - SQLite operations outside coroutine scope

2. **Network on main thread:**
   - Synchronous OkHttp/Retrofit calls
   - URL.openConnection() outside background thread

3. **Heavy computation on main thread:**
   - JSON parsing (Gson/Moshi) in UI layer
   - Bitmap operations in onDraw/onBind
   - Large list sorting/filtering in adapters

4. **Lock contention risks:**
   - synchronized blocks accessed from both main and background threads
   - ReentrantLock usage shared across threads

5. **Dispatcher misuse:**
   - viewModelScope.launch without withContext for IO ops
   - lifecycleScope.launch with blocking calls

## Output Format
For each risk found, report:
- **File:line** — where the risky code is
- **Risk level** — High/Medium/Low
- **Pattern** — what type of ANR risk
- **Fix** — how to make it safe
Run /check-anr-risks as part of your pre-PR review routine. It takes 30 seconds and can catch ANR-causing code before it reaches production.

Your ANR Prevention Checklist

  • Enable StrictMode in debug builds to catch main-thread violations early
  • Never call allowMainThreadQueries() in production Room builds
  • Use SharedPreferences.apply() instead of commit() (or migrate to DataStore)
  • Default to Dispatchers.IO for all database and file operations in coroutines
  • Debounce text input that triggers database queries
  • Use Room’s Flow or suspend return types instead of plain lists
  • Set up the /analyze-anr and /check-anr-risks skills
  • Monitor ANR rates in Google Play Console — keep below 0.47%

The ANR Mindset

Every time you write code that runs on the main thread, ask yourself: “Could this ever take more than 100ms?” If the answer is yes, or even “maybe on a slow device,” move it to a background thread. The main thread has one job: respond to the user. Everything else belongs somewhere else.

Hands-On: Trigger, Capture, and Diagnose an ANR

  1. Trigger a deliberate ANR: In any Activity’s onCreate(), add Thread.sleep(6000). Build and run the app, then tap on the screen. Wait for the ANR dialog.
  2. Capture the trace: Run adb bugreport bugreport.zip, unzip it, and find the ANR trace section. Alternatively, try adb pull /data/anr/traces.txt.
  3. Run /analyze-anr: Paste the main thread’s trace into Claude Code using your new skill. Verify it identifies Thread.sleep() as the blocking call.
  4. Try a realistic ANR: Replace the Thread.sleep() with a synchronous database query or SharedPreferences commit() with a large payload. Capture and analyze the ANR trace again — is the AI analysis more nuanced?
  5. Run /check-anr-risks: Point Claude Code at your project and let it scan for main-thread violations. How many real risks does it find?

Remove the deliberate Thread.sleep() when you’re done! Commit the skills files, though — you’ll use them throughout this course.
