Chapter 8

Battery and Power Optimization

Your app works great — until the user notices it ate 30% of their battery by lunch

The Battery Is the Scoreboard

In Chapter 7, we optimized build times — the developer’s productivity bottleneck. This chapter tackles the user’s most visible resource: their battery. Users don’t know what a recomposition count is. They don’t care about your Gradle cache hit rate. But they absolutely notice when your app shows up in Settings → Battery as a top consumer.

And Android notices too. Starting with Android 9, the system aggressively restricts background behavior for apps it considers battery-heavy. Your perfectly working background sync can simply stop firing.

46% of users check battery usage stats at least weekly
5 App Standby Buckets that throttle your app’s background work
2h / 28d Play Console’s “excessive wake locks” threshold — can derank your store listing

Doze Mode and App Standby Buckets

Before debugging battery issues, you need to understand the two systems that control your app’s background access:

System What It Does Impact on Your App
Doze Mode Defers alarms, network, jobs, and syncs when device is idle Your AlarmManager alarms and JobScheduler jobs get batched into maintenance windows
App Standby Buckets Categorizes apps by usage frequency: Active, Working Set, Frequent, Rare, Restricted Rare/Restricted apps can only run jobs once per day; alarms are heavily deferred
Check your app’s current standby bucket with: adb shell am get-standby-bucket com.example.myapp. If you’re in “Rare” or “Restricted” during testing, the system is throttling your background work — which will mask real issues. Reset with: adb shell am set-standby-bucket com.example.myapp active.

The Excessive Wake Locks Metric (New for 2026)

In November 2025, Google Play exited beta on a new Android vitals metric: excessive partial wake locks. Starting March 1, 2026, this metric is part of the Play Store quality-enforcement system — meaning if you fail it, your app can be excluded from recommendations and discovery surfaces, and users may see a battery-drain warning on your store listing.

This is not a soft warning. It’s the same enforcement mechanism that already exists for crash rate and ANR rate. Battery is now a Play Console quality bar.

The bad-behavior threshold: A user session is flagged as excessive if your app holds non-exempt partial wake locks for more than 2 cumulative hours while the screen is off in a 24-hour period. If more than 5% of your sessions over the last 28 days are excessive, you cross the bad-behavior threshold and become subject to discovery suppression and the store-listing warning.

Android Studio’s App Quality Insights and the Play Console both surface this data. The Play Console shows a per-wake-lock-tag breakdown of P90 / P99 durations — Google’s own guidance is to investigate any wake lock with P90 or P99 above 60 minutes.

Some Wake Locks Are Exempt

Not every partial wake lock counts against the metric. Wake locks held during these states are exempt and do not contribute to the threshold:

  • While a foreground service is running
  • While the app is the top-visible app
  • While media is being played or recorded
  • While a user-initiated job is executing
  • While the device is charging

This is why the metric is targeted specifically at background wake-lock abuse — not the legitimate cases where the user is actively engaged with your app.

What Claude Code Finds

(1) LocationUpdateService wake lock is held for 42 minutes total across 2,531 acquisitions. This is the top battery drain — the service is requesting location updates far too frequently. (2) ProductSyncAdapter fired 847 times, which is excessive for a product catalog that changes daily. (3) GPS is active for 38 minutes, likely correlated with the location wake lock. (4) Total estimated usage of 8.7% is very high for an e-commerce app — target is under 2%.

Root Cause: Over-Aggressive Location Updates + Sync Frequency

Two compounding issues: (1) LocationUpdateService requests GPS updates every 5 seconds with PRIORITY_HIGH_ACCURACY, keeping the GPS radio active almost continuously. For a shopping app, this precision and frequency are unnecessary — store locations don’t move. (2) ProductSyncAdapter runs every 2 minutes to check for catalog updates. A product catalog that changes daily does not need minute-level sync.

Fix: Right-Size Location Requests + Reduce Sync Frequency

// BEFORE: Over-aggressive location request
val locationRequest = LocationRequest.Builder(
    Priority.PRIORITY_HIGH_ACCURACY,
    5_000L  // Every 5 seconds — way too frequent for store finder
).build()

// AFTER: Appropriate for a store finder feature
val locationRequest = LocationRequest.Builder(
    Priority.PRIORITY_BALANCED_POWER_ACCURACY,  // Cell/WiFi, not GPS
    300_000L  // Every 5 minutes — stores don't move
).setMinUpdateDistanceMeters(100f)  // Only if user moves 100m
 .setMaxUpdateDelayMillis(600_000L)  // Batch up to 10 min
 .build()

// BEFORE: Sync every 2 minutes
PeriodicWorkRequestBuilder<ProductSyncWorker>(
    2, TimeUnit.MINUTES  // 720 syncs per day!
).build()

// AFTER: Sync once per day, with constraints
PeriodicWorkRequestBuilder<ProductSyncWorker>(
    24, TimeUnit.HOURS
).setConstraints(
    Constraints.Builder()
        .setRequiredNetworkType(NetworkType.UNMETERED)  // WiFi only
        .setRequiresBatteryNotLow(true)
        .build()
).build()

Key changes: (1) PRIORITY_BALANCED_POWER_ACCURACY uses cell towers and WiFi instead of GPS — 10x less battery drain. (2) Update interval of 5 minutes with distance filter means the radio only activates when meaningful. (3) Sync drops from 720/day to 1/day, constrained to WiFi and sufficient battery.

WakeLock Audit

Wake locks are the #1 cause of unnecessary battery drain. They keep the CPU running when it should be sleeping. Every acquire without a matching release is a battery leak.

// The dangerous pattern:
val wakeLock = powerManager.newWakeLock(
    PowerManager.PARTIAL_WAKE_LOCK, "MyApp::SyncLock"
)
wakeLock.acquire()  // CPU stays awake...
doLongSync()        // If this throws, release is never called!
wakeLock.release()

// The safe pattern:
val wakeLock = powerManager.newWakeLock(
    PowerManager.PARTIAL_WAKE_LOCK, "MyApp::SyncLock"
)
wakeLock.acquire(10 * 60 * 1000L)  // 10 min timeout — auto-release!
try {
    doLongSync()
} finally {
    if (wakeLock.isHeld) wakeLock.release()
}
Always use a timeout on wakeLock.acquire(). The overload acquire(timeoutMs) automatically releases the lock after the timeout. This prevents battery-killing leaks if your code throws an exception or hangs.

WorkManager vs AlarmManager vs JobScheduler

Choosing the wrong scheduling API is a common source of battery drain. Here’s when to use each:

API Use When Battery Impact
WorkManager Any deferrable background work. Default choice for most tasks. Low — respects Doze, batches intelligently
AlarmManager (exact) User-facing alarms (medication reminders, meeting alerts) High — wakes device from Doze. Requires SCHEDULE_EXACT_ALARM permission.
AlarmManager (inexact) Periodic triggers where timing isn’t critical Medium — system batches with other alarms
Foreground Service User-initiated ongoing tasks (music playback, navigation, file upload) Varies — shows notification, user is aware
Prompt
Here's our background scheduling code. We use AlarmManager for product sync, a foreground service for checking new messages, and WorkManager for image upload. Are we using the right API for each task? [paste scheduling code for all three] Product sync: checks catalog every 2 hours, not user-visible Message check: polls server every 30 seconds for new messages Image upload: user takes a photo, upload can happen anytime
AI will likely flag: product sync should use WorkManager (not AlarmManager); message check should use FCM push instead of polling; image upload is correctly using WorkManager.

Location Services Battery Drain

Location is the single biggest battery drain after the screen. The difference between good and bad location usage can be 10x in battery impact.

Location Config
Priority: HIGH_ACCURACY (GPS) Interval: 1 second Smallest displacement: 0 meters Running in: foreground service, always on
GPS radio on continuously. Appropriate only for turn-by-turn navigation. Drains ~15% battery per hour.

Cell/WiFi positioning, only when user moves significantly. Geofences are hardware-accelerated and nearly free. Drains ~0.5% per hour.

Geofencing vs Continuous Location

// Instead of continuously tracking location to detect "near a store",
// use geofences — they're hardware-accelerated and battery-free:
val geofence = Geofence.Builder()
    .setRequestId("store-123")
    .setCircularRegion(lat, lng, 200f)  // 200m radius
    .setExpirationDuration(Geofence.NEVER_EXPIRE)
    .setTransitionTypes(Geofence.GEOFENCE_TRANSITION_ENTER)
    .build()

val request = GeofencingRequest.Builder()
    .addGeofence(geofence)
    .setInitialTrigger(GeofencingRequest.INITIAL_TRIGGER_ENTER)
    .build()

geofencingClient.addGeofences(request, geofencePendingIntent)

Foreground Service Audit

Android 14+ requires you to declare the type of foreground service, and some types require specific permissions. AI can audit your foreground service usage:

// AndroidManifest.xml — must declare type:
<service
    android:name=".service.SyncForegroundService"
    android:foregroundServiceType="dataSync"
    android:exported="false" />

// Question to ask Claude Code:
// "Do I really need a foreground service for data sync, or can
//  WorkManager handle it? Here's what the service does: [paste code]"
In most cases, WorkManager with a long-running worker is a better choice than a foreground service for background data operations. WorkManager handles wake locks, constraints, and retries automatically. Reserve foreground services for truly user-initiated ongoing tasks.

Android 15: The 6-Hour Timeout for dataSync and mediaProcessing

If your app targets Android 15 or higher, the system tracks how long dataSync and mediaProcessing foreground services run while the app is in the background. Cumulative 6 hours per 24-hour period, shared across all FGS instances of that type. After that, the system calls Service.onTimeout(int, int) and your service must call stopSelf() within a few seconds — or the system throws a RemoteServiceException.

// Handle the Android 15 timeout callback:
override fun onTimeout(startId: Int, fgsType: Int) {
    super.onTimeout(startId, fgsType)
    // Persist whatever state you need, then stop:
    saveProgress()
    stopSelf()
}

// Test the timeout without waiting 6 real hours:
// adb shell am compat enable FGS_INTRODUCE_TIME_LIMITS com.your.package
The timer is shared per type. If a dataSync service runs for 4 hours, then a different dataSync service starts, the second one gets only 2 hours before timeout. Bringing the app to the foreground resets the budget. The 6-hour budget is tracked separately for dataSync and mediaProcessing.

The BOOT_COMPLETED Trap

Since Android 14, starting a foreground service from a BOOT_COMPLETED broadcast receiver throws ForegroundServiceStartNotAllowedException for most FGS types. You can’t spin up a sync service the moment the device boots anymore — the system considers the app “not user-visible.” Use WorkManager with setInitialDelay, or a one-time OneTimeWorkRequest, instead.

// Test the BOOT_COMPLETED restriction without rebooting:
// adb shell am compat enable FGS_BOOT_COMPLETED_RESTRICTIONS com.your.package
// Then trigger your BOOT_COMPLETED handler with:
// adb shell am broadcast -a android.intent.action.BOOT_COMPLETED -p com.your.package

Android 16 adds further pressure: job runtime quotas apply to jobs that run concurrently with foreground services, so the “hide everything behind an FGS” pattern no longer buys unlimited compute. If a job exceeds its quota while an FGS is running, the system stops the job. The fix is the same direction as the Android 15 timeout: design your background work to be interruptible, save progress regularly, and prefer WorkManager’s constraint system over manually managing service lifecycles.

Network Batching

Each network request wakes the cellular radio for ~20 seconds (the “radio tail”). Ten small requests in 10 minutes means the radio is active almost continuously. Batching those into one request eliminates 90% of the radio time.

// BAD: Individual analytics events sent immediately
fun trackEvent(event: AnalyticsEvent) {
    scope.launch {
        api.sendEvent(event)  // Each call wakes the radio for ~20s
    }
}

// GOOD: Batch analytics events
class AnalyticsBatcher @Inject constructor(
    private val api: AnalyticsApi
) {
    private val batch = mutableListOf<AnalyticsEvent>()

    fun trackEvent(event: AnalyticsEvent) {
        synchronized(batch) { batch.add(event) }
    }

    // Flush periodically via WorkManager, not on every event
    suspend fun flush() {
        val events = synchronized(batch) {
            batch.toList().also { batch.clear() }
        }
        if (events.isNotEmpty()) {
            api.sendBatch(events)  // One request instead of many
        }
    }
}

Your Battery Optimization Checklist

  • Always use acquire(timeoutMs) for wake locks — never bare acquire()
  • Use PRIORITY_BALANCED_POWER_ACCURACY for location unless navigation is the feature
  • Use geofences instead of continuous location tracking for proximity detection
  • Default to WorkManager for all background work — reserve foreground services for user-initiated tasks
  • Add constraints to WorkManager jobs: network type, battery level, charging state
  • Batch network requests — especially analytics events
  • Use FCM push instead of polling for real-time updates
  • Audit with dumpsys batterystats periodically

The Battery Mindset

Every time your code wakes the CPU, activates the radio, or requests a GPS fix, you’re spending a shared resource — the user’s battery. Ask yourself: “Is the user getting visible value from this work right now?” If not, defer it, batch it, or eliminate it. The user doesn’t see your sync operation. They do see “24% battery used by ShopApp.”

Hands-On: Audit Your App’s Battery Usage

  1. Reset battery stats: Run adb shell dumpsys batterystats --reset.
  2. Use the app: Disconnect USB and use the app normally for 30 minutes. Navigate between screens, trigger background features, let it sit idle.
  3. Capture stats: Reconnect and run adb shell dumpsys batterystats --charged com.your.package. Extract the wake lock, job, sensor, and network sections.
  4. Run /analyze-battery: Paste the dump into Claude Code. Ask it to identify the top 3 battery drains and suggest fixes.
  5. Check location settings: Search your codebase for LocationRequest or PRIORITY_HIGH_ACCURACY. Ask Claude Code if the priority and interval are appropriate for your use case.
  6. Bonus: Set up Battery Historian with Docker and upload a bug report. Compare the visual timeline with the text dump analysis from Claude Code.
