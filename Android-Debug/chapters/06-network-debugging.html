Network Debugging with AI

OkHttp, Retrofit, SSL — diagnosing network-layer issues with AI assistance

When the Server Says No

In Chapter 5, we optimized how your app renders. Now let’s talk about where the data comes from. Network calls are the lifeline of almost every Android app, and when they break, the symptoms are maddening: silent failures, intermittent timeouts, 401s that only happen on certain devices, and response bodies that don’t match your data model.

The challenge with network bugs is that they span two systems — your app and the server. You can’t set a breakpoint on a remote API. But you can capture every HTTP request and response with OkHttp interceptors, and feed that data to Claude Code for analysis.

38% of mobile app bugs involve the network layer
3–5 layers involved in a single API call (app, OkHttp, TLS, server, CDN)
60% of network bugs are reproducible only intermittently

Setting Up OkHttp Logging

The first step in network debugging is seeing what’s actually going over the wire. OkHttp’s HttpLoggingInterceptor captures every request and response.

// build.gradle.kts
dependencies {
    debugImplementation("com.squareup.okhttp3:logging-interceptor:5.3.2")
}

// In your Hilt NetworkModule:
@Provides
@Singleton
fun provideOkHttpClient(): OkHttpClient {
    return OkHttpClient.Builder()
        .apply {
            if (BuildConfig.DEBUG) {
                addInterceptor(HttpLoggingInterceptor().apply {
                    level = HttpLoggingInterceptor.Level.BODY
                })
            }
        }
        .connectTimeout(15, TimeUnit.SECONDS)
        .readTimeout(30, TimeUnit.SECONDS)
        .writeTimeout(15, TimeUnit.SECONDS)
        .build()
}

OkHttp 5.x (5.0.0 shipped July 2025) is a major release with breaking changes for callers still on 4.x — OkHttp now ships split JVM and Android artifacts, MockWebServer relocated to the mockwebserver3 package and coordinate, and timeout APIs accept kotlin.time.Duration. The HttpLoggingInterceptor usage above is unchanged across the bump.

Never log request/response bodies in release builds. They can contain auth tokens, personal data, and payment information. Use Level.BODY only in debug builds. For release, use Level.BASIC at most (method + URL + response code).

Logging Levels

Level What It Logs When to Use
NONE Nothing Release builds
BASIC Method, URL, response code, timing Quick overview of call patterns
HEADERS Basic + all headers Auth, caching, content-type issues
BODY Headers + full request/response bodies Debugging data issues (debug only!)

Feeding HTTP Logs to Claude Code

Once logging is enabled, capture a failing API flow and feed it to Claude Code. Here’s what a real failing flow looks like:

OkHttp Interceptor Log: 401 After Token Refresh

--> POST https://api.example.com/auth/refresh

Content-Length: 87

{"access_token":"eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ1c2Vy...","expires_in":3600}

--> GET https://api.example.com/v2/user/profile
Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...

<-- 401 Unauthorized (52ms)
Content-Type: application/json

{"error":"invalid_token","message":"Token has been revoked"}

What Claude Code Finds

The refresh succeeded (200 OK) and returned a new access_token. But the profile request still uses the OLD token — look at the Authorization header. The Bearer token in the GET request matches the original expired token, not the new one from the refresh response.

Diagnosis: The token refresh interceptor updates the stored token, but the retried request was built before the refresh completed. The request chain doesn’t rebuild the Authorization header after refreshing.

Root Cause: Race Condition in Auth Interceptor

The Authenticator refreshes the token and stores it, but OkHttp’s retry uses the original request object. If the interceptor doesn’t build a new request with the fresh token, the retry sends the stale token and gets 401 again.

This is the most common auth-related network bug. It manifests as “random logouts” or “401s even though the user is logged in.”

Fix: Rebuild the Request with the New Token

class TokenAuthenticator @Inject constructor(
    private val tokenStore: TokenStore
) : Authenticator {

    override fun authenticate(route: Route?, response: Response): Request? {
        // Don't retry if we already refreshed (prevent infinite loops)
        if (response.request.header("X-Retry-Auth") != null) return null

        synchronized(this) {
            // Check if another thread already refreshed
            val currentToken = tokenStore.accessToken
            val requestToken = response.request.header("Authorization")
                ?.removePrefix("Bearer ")

            val freshToken = if (currentToken == requestToken) {
                // Token hasn't been refreshed yet — do it now
                tokenStore.refreshTokenSync() ?: return null
            } else {
                // Another thread already refreshed — use the new token
                currentToken
            }

            // Build a NEW request with the fresh token
            return response.request.newBuilder()
                .header("Authorization", "Bearer $freshToken")
                .header("X-Retry-Auth", "true")
                .build()
        }
    }
}

Key details: (1) synchronized prevents multiple threads from refreshing simultaneously. (2) Check if the token was already refreshed by another thread. (3) Build a new request with the fresh token. (4) X-Retry-Auth header prevents infinite retry loops.

Building an /analyze-network Skill

Create .claude/commands/analyze-network.md:

# /analyze-network — HTTP Traffic Analysis

Analyze OkHttp interceptor logs or HTTP traffic captures from an
Android app.

## Input
The user will paste one or more HTTP request/response pairs from
OkHttp logging interceptor output.

## Analysis Steps

1. **Map the request flow** — identify the sequence of calls and
   their dependencies (auth → data → follow-up).

2. **Check response codes:**
   - 4xx: client error — wrong URL, missing auth, bad payload
   - 5xx: server error — log it but look for client-side causes
   - Timeouts: check client timeout config vs server response time

3. **Inspect headers:**
   - Authorization: is the token present, fresh, correct format?
   - Content-Type: does it match what the server expects?
   - Cache-Control: is caching configured appropriately?
   - Accept: does it match what the server returns?

4. **Analyze timing:**
   - DNS resolution time (if available)
   - Connection time vs. response time
   - Calls that are suspiciously slow (>2s for simple endpoints)

5. **Check for patterns:**
   - Duplicate requests (same endpoint called multiple times)
   - Missing error handling (no retry on transient failures)
   - Token refresh race conditions
   - Unnecessary sequential calls that could be parallelized

## Output Format

### Request Flow
[Numbered sequence of HTTP calls]

### Issues Found
[Each issue with severity: Critical/High/Medium/Low]

### Root Cause
[Why the failure or inefficiency is happening]

### Suggested Fix
```kotlin
[Code]
```

### Optimization Opportunities
[Caching, batching, parallelization suggestions]

SSL/TLS Debugging

SSL errors are the most cryptic network failures. The stacktraces are full of security jargon, and the error messages are deliberately vague (for security reasons). AI excels here because it can map the cryptic exception to a specific misconfiguration.

Certificate Pinning Failures

// Setting up certificate pinning:
val client = OkHttpClient.Builder()
    .certificatePinner(
        CertificatePinner.Builder()
            .add("api.example.com", "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=")
            .add("api.example.com", "sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=")
            .build()
    )
    .build()

// When pinning fails, you get:
// javax.net.ssl.SSLPeerUnverifiedException:
//   Certificate pinning failure!
//   Peer certificate chain:
//     sha256/CCCCCCCCCCC... : CN=api.example.com
//     sha256/DDDDDDDDDDD... : CN=Let's Encrypt R3
//   Pinned certificates for api.example.com:
//     sha256/AAAAAAAAAA...
//     sha256/BBBBBBBBBB...
Prompt
I'm getting this certificate pinning failure on production but not on staging. The pinning was working until last week. Here's the exception: [paste SSLPeerUnverifiedException] And here's my CertificatePinner configuration: [paste config code] Did the server's certificate rotate?
Claude Code will compare the pinned hashes with the peer certificate chain, identify which certificate changed, and explain whether it was a leaf cert rotation or a CA chain change.
When pinning certificates, always pin at least two hashes: the current certificate and a backup. Pin the intermediate CA certificate, not the leaf — leaf certificates rotate frequently. Include a pin for a backup CA from a different provider in case your primary CA has an outage.

SSLHandshakeException Analysis

Common SSLHandshakeException causes AI identifies instantly:

Error Message Common Cause AI’s Usual Fix
CertPathValidatorException: Trust anchor not found Self-signed cert or missing intermediate CA Add custom TrustManager for staging; fix cert chain for production
SSLProtocolException: SSL handshake aborted TLS version mismatch (server requires TLS 1.3, device only supports 1.2) Configure ConnectionSpec with compatible TLS versions
CertificateException: No name matching Certificate CN/SAN doesn’t match the requested hostname Fix the server certificate or use the correct hostname

Timeout and Retry Strategies

Bad timeout configuration is the #1 cause of “the app is slow” complaints that aren’t actually slow code — they’re slow network handling.

OkHttp Config
OkHttpClient.Builder() .connectTimeout(60, SECONDS) .readTimeout(60, SECONDS) .writeTimeout(60, SECONDS) .build()
60-second timeouts mean users wait a full minute before seeing an error. Most users abandon after 8 seconds.
OkHttp Config
OkHttpClient.Builder() .connectTimeout(10, SECONDS) .readTimeout(30, SECONDS) .writeTimeout(15, SECONDS) .retryOnConnectionFailure(true) .addInterceptor(RetryInterceptor(maxRetries = 2)) .build()
Aggressive connect timeout (fail fast if server is unreachable), longer read timeout (for large responses), and retry logic for transient failures.

Ask Claude Code to review your timeout configuration by pasting your OkHttpClient.Builder code and describing your API patterns (mostly small JSON? large file downloads? real-time streaming?).

Caching Strategy Review

A well-configured HTTP cache can eliminate unnecessary network calls entirely. But most Android apps either don’t cache at all or cache incorrectly.

// Setting up OkHttp cache:
val cache = Cache(
    directory = File(context.cacheDir, "http_cache"),
    maxSize = 50L * 1024 * 1024  // 50 MB
)

val client = OkHttpClient.Builder()
    .cache(cache)
    .build()

// For endpoints that rarely change, add a cache interceptor:
class CacheInterceptor : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val response = chain.proceed(chain.request())
        val cacheControl = CacheControl.Builder()
            .maxAge(5, TimeUnit.MINUTES)
            .build()
        return response.newBuilder()
            .removeHeader("Pragma")
            .removeHeader("Cache-Control")
            .header("Cache-Control", cacheControl.toString())
            .build()
    }
}
Prompt
Here are the Cache-Control headers from our main API endpoints. Which ones could benefit from client-side caching, and what cache durations would you recommend? GET /api/v2/categories → Cache-Control: no-cache GET /api/v2/products?page=1 → Cache-Control: no-store GET /api/v2/user/profile → Cache-Control: private, max-age=0 GET /api/v2/config/features → no Cache-Control header Categories change weekly. Products change daily. Profile changes on user action. Feature flags change on deploy.
Giving AI the change frequency for each endpoint lets it suggest specific cache durations. Categories could cache for hours, products for minutes, features for the session.

API Call Deduplication

A common performance issue: the same API endpoint gets called multiple times unnecessarily. This happens when multiple screens or ViewModels independently request the same data.

// The problem: three ViewModels each call getUser()
// That's three identical network requests.

// Fix: Use a repository with in-flight request deduplication
class UserRepository @Inject constructor(
    private val api: UserApi
) {
    private var inFlightRequest: Deferred<User>? = null

    suspend fun getUser(): User {
        // If a request is already in flight, await it instead of starting a new one
        inFlightRequest?.let { return it.await() }

        return coroutineScope {
            val deferred = async { api.getUser() }
            inFlightRequest = deferred
            try {
                deferred.await()
            } finally {
                inFlightRequest = null
            }
        }
    }
}
Run /analyze-network on a logcat capture from a typical user session. Look for the same URL appearing multiple times within a short window. AI can identify which calls are redundant and suggest where to add caching or deduplication.

Your Network Debugging Checklist

  • Add HttpLoggingInterceptor with Level.BODY in debug builds only
  • Set realistic timeouts — 10s connect, 30s read, 15s write
  • Implement a thread-safe TokenAuthenticator that rebuilds requests
  • Pin intermediate CA certificates, not leaf certs; always have a backup pin
  • Configure OkHttp Cache with a reasonable size (50–100 MB)
  • Deduplicate in-flight requests in your repository layer
  • Set up the /analyze-network skill for repeatable HTTP diagnostics

The Network Debugging Approach

Network bugs feel random because you can’t see the wire. But with OkHttp logging, every request and response is visible. Capture the full HTTP conversation, paste it into Claude Code, and let AI trace the issue across headers, tokens, timing, and response bodies. The bug that took an hour to figure out from a vague “something went wrong” error takes 5 minutes when you have the full HTTP log.

Hands-On: Capture and Diagnose a Network Flow

  1. Enable logging: Add HttpLoggingInterceptor at Level.BODY to your OkHttp client (debug builds only).
  2. Capture a flow: Run your app and navigate through a feature that makes multiple API calls (e.g., login → load profile → load data). Capture the logcat output.
  3. Run /analyze-network: Paste the HTTP log sequence into Claude Code. Ask it to: identify the request flow, check for duplicate calls, review timeout configuration, and spot any error handling gaps.
  4. Test error handling: Enable airplane mode mid-flow, or point your app at a non-existent server. Capture the error logs and feed them to Claude Code. Does your app handle the failure gracefully?
  5. Audit caching: Paste the Cache-Control headers from your API responses and ask Claude Code which endpoints should be cached and for how long.
