The Profiler Data Gap

Chapters 1–8 tackled specific problems: crashes, ANRs, memory leaks, battery drain. But sometimes the bug report is just “the app feels slow.” No stacktrace. No crash. Just… jank. That’s when you reach for the profiler.

Android Studio’s profiler is powerful. It shows CPU flame charts, memory allocation timelines, network request waterfalls, and system-level traces. The problem? Reading profiler output is a skill unto itself. Most developers open the profiler, see a wall of colored bars, and close it again.

This is where AI shines. You extract the key data from the profiler, feed it to Claude Code, and get a plain-English explanation of what’s consuming your CPU, why your memory is spiking, and which frames are being dropped.

67% of Android devs have the profiler but rarely use it effectively
3 types of profiling: CPU, Memory, and System Trace (Perfetto)
16 ms per frame budget — anything over this causes visible jank

CPU Profiling: Method Traces

A CPU method trace records which methods your app calls, how long each takes, and how many times each is called. It’s a snapshot of where your CPU time goes.

Capturing a Method Trace

# From Android Studio:
# 1. Open Profiler (View → Tool Windows → Profiler)
# 2. Select your app process
# 3. Click the CPU section
# 4. Click "Record" → choose "Sample Java/Kotlin Methods"
# 5. Interact with the app (scroll, navigate, etc.)
# 6. Click "Stop" to end recording
# 7. Right-click the recording → "Export Trace"

# From command line (for CI or scripted captures):
# Start recording:
adb shell am profile start com.example.myapp /data/local/tmp/cpu.trace
# Use the app...
# Stop recording:
adb shell am profile stop com.example.myapp
adb pull /data/local/tmp/cpu.trace

Reading the Data for AI

The profiler shows a flame chart and a “Top Down” / “Bottom Up” method list. For Claude Code, extract the “Top Down” view — it shows the hot methods with call counts and wall-clock time.

CPU Profile: Hot Methods During Product List Scroll

Top methods by wall-clock time (5-second capture during scroll):

Method                                          Self Time   Total Time  Count
──────────────────────────────────────────────────────────────────────────
ProductAdapter.onBindViewHolder()                   2ms       847ms      48
  ├─ ProductImageLoader.loadImage()                12ms       612ms      48
  │    ├─ BitmapFactory.decodeStream()            389ms       389ms      48
  │    └─ ImageUtils.roundCorners()               198ms       198ms      48
  ├─ PriceFormatter.format()                        8ms       142ms      48
  │    └─ NumberFormat.getInstance()               127ms       127ms      48
  └─ ProductAdapter.calculateDiscount()             83ms        83ms      48

  └─ CanvasContext.draw()                         234ms       234ms     312

What Claude Code Finds

(1) BitmapFactory.decodeStream() is the biggest single consumer at 389ms — decoding images on every onBindViewHolder call means no image caching. Each scroll recycles views and re-decodes bitmaps from scratch. (2) NumberFormat.getInstance() is called 48 times at 127ms total — it’s creating a new NumberFormat instance for every item instead of reusing one. (3) 12 GC pauses in 5 seconds indicates heavy allocation pressure, likely from the bitmap decoding. (4) ImageUtils.roundCorners() creates a new bitmap per item — should use rounded corner clipping in the ImageView or Compose instead.

Root Cause: No Image Caching + Object Allocation in Bind

Every call to onBindViewHolder decodes a full bitmap from disk, rounds its corners (creating another bitmap), and instantiates a new NumberFormat. For a list of 48 visible items, this means 96 bitmap allocations and 48 NumberFormat instances per scroll gesture. The GC can’t keep up, causing 156ms of GC pauses that directly produce janky frames.

Fix: Use Coil/Glide for Image Caching + Reuse Formatters

// BEFORE: Manual bitmap loading in onBindViewHolder
override fun onBindViewHolder(holder: ViewHolder, position: Int) {
    val product = products[position]
    val bitmap = BitmapFactory.decodeStream(openImage(product.imageUrl))
    val rounded = ImageUtils.roundCorners(bitmap, 12f)
    holder.imageView.setImageBitmap(rounded)
    holder.price.text = PriceFormatter.format(product.price)
}

// AFTER: Coil for images, cached formatter
class ProductAdapter : ListAdapter<Product, ViewHolder>(...) {
    // Create once, reuse across all binds
    private val priceFormat = NumberFormat.getCurrencyInstance()

    override fun onBindViewHolder(holder: ViewHolder, position: Int) {
        val product = products[position]

        // Coil handles caching, decoding, and memory management
        holder.imageView.load(product.imageUrl) {
            crossfade(true)
            transformations(RoundedCornersTransformation(12f))
            memoryCachePolicy(CachePolicy.ENABLED)
        }

        holder.price.text = priceFormat.format(product.price)
    }
}

Key changes: (1) Coil (or Glide) handles bitmap decoding, memory caching, disk caching, and recycling. No manual BitmapFactory calls. (2) Rounded corners via Coil transformation instead of creating a new bitmap. (3) Single NumberFormat instance reused across all binds.

Building a /profile-cpu Skill

Create .claude/commands/profile-cpu.md:

# /profile-cpu — CPU Profile Analysis

Analyze CPU profiling data from Android Studio method traces.

## Input
The user will paste a "Top Down" or "Bottom Up" method summary from
the Android Studio CPU profiler, including method names, self time,
total time, and call counts.

## Analysis Steps

1. **Identify the top 5 methods by total time** — these are the
   hottest code paths.

2. **Look for allocation-heavy patterns:**
   - Object creation inside loops or RecyclerView.onBindViewHolder
   - New instances where a cached instance should be reused
   - Bitmap operations without image loading library

3. **Check for main-thread violations:**
   - IO operations (disk, network) on the main thread
   - JSON parsing or heavy computation during UI rendering

4. **Analyze GC pressure:**
   - Frequent GC pauses indicate allocation pressure
   - Each GC pause directly causes dropped frames

5. **Correlate with frame timing:**
   - RenderThread time > 16ms = dropped frames
   - Identify which user-code methods push over budget

## Output Format

### Top CPU Consumers
[Ranked list with time, count, and impact]

### Allocation Issues
[Objects being created where they shouldn't be]

### Main Thread Violations
[IO or heavy work on the main thread]

### Frame Budget Analysis
[Which methods push frames over 16ms]

### Optimization Suggestions
```kotlin
[Specific code changes, ranked by impact]
```

Memory Profiling: Allocation Tracking

CPU profiling tells you what’s slow. Memory profiling tells you what’s allocating too much. Together, they paint a complete picture.

Capturing Allocation Data

# In Android Studio Profiler:
# 1. Select the Memory section
# 2. Click "Record Java/Kotlin Allocations"
# 3. Perform the slow action (scroll, navigate, etc.)
# 4. Click "Stop"
# 5. Sort by "Shallow Size" or "Allocation Count"

Extract the top allocators for Claude Code:

Prompt
Here are the top allocations during a 5-second scroll of our product list: Class Alloc Count Shallow Size Bitmap 96 24.3 MB byte[] 1,847 8.2 MB String 3,421 1.8 MB ProductViewState 192 0.4 MB NumberFormat 48 0.1 MB Total allocations: 8,247 objects in 5 seconds GC pauses: 12 (total 156ms) Why are there 96 Bitmap allocations for a list with 48 visible items?
The specific question "why 96 for 48 items" tells AI to look for double-allocation patterns — likely original decode + rounded copy.

System Trace (Perfetto) Analysis

CPU and memory profilers show your app’s code. System traces (Perfetto) show everything: your app, the system framework, the GPU, the scheduler, and thread interactions. This is the nuclear option for “where are my frames going?”

Capturing a System Trace

# From Android Studio:
# Profiler → CPU → Record → "System Trace"

# From command line (Perfetto):
adb shell perfetto \
    -c - --txt \
    -o /data/misc/perfetto-traces/trace.perfetto-trace \
    <<EOF
buffers: {
    size_kb: 63488
    fill_policy: RING_BUFFER
}
data_sources: {
    config {
        name: "linux.ftrace"
        ftrace_config {
            ftrace_events: "sched/sched_switch"
            ftrace_events: "power/suspend_resume"
            ftrace_events: "sched/sched_wakeup"
            atrace_categories: "view"
            atrace_categories: "gfx"
            atrace_categories: "am"
            atrace_apps: "com.example.myapp"
        }
    }
}
duration_ms: 10000
EOF

adb pull /data/misc/perfetto-traces/trace.perfetto-trace

Open the trace at ui.perfetto.dev or in Android Studio. The timeline shows each frame as a colored block. Green = on time, yellow = slightly late, red = janky.

Extracting Perfetto Data for AI

You can’t paste a Perfetto trace file into Claude Code, but you can extract the frame timing summary:

# In Perfetto UI, look for:
# - "Choreographer#doFrame" durations
# - Any frame > 16ms (60 FPS) or > 8ms (120 FPS)
# - "RenderThread" durations

# Extract the janky frames:
Frame 1: Choreographer doFrame 8.2ms ✓
Frame 2: Choreographer doFrame 7.1ms ✓
Frame 3: Choreographer doFrame 42.8ms ✗ JANK
  └─ inflate: 18.3ms
  └─ measure: 12.4ms
  └─ layout: 8.7ms
  └─ draw: 3.4ms
Frame 4: Choreographer doFrame 6.9ms ✓
Frame 5: Choreographer doFrame 38.1ms ✗ JANK
  └─ bindViewHolder: 31.2ms
When feeding frame timing to Claude Code, include the breakdown for janky frames only. Don’t paste 300 frames that are fine — focus on the ones that exceeded the budget. AI can diagnose the root cause from the breakdown of the slow frames.

Frame Rendering Analysis

Dropped frames are what users feel as jank. Here’s how to systematically identify the cause:

Capture trace → Find janky frames → Extract breakdown → Feed to AI → Fix hotspot

The frame breakdown tells you which phase of rendering is over budget:

Phase What It Does Common Cause If Slow
Input Process touch events Heavy work in onClick, onTouch handlers
Animation Update animation values Too many concurrent animations, complex interpolators
Measure/Layout Calculate view sizes and positions Deep view hierarchy, nested weights, repeated measure passes
Draw Record draw commands Complex custom drawing, large bitmap operations
Sync/Upload Send textures to GPU Large/uncompressed bitmaps, too many layers
GPU/Render Execute draw commands on GPU Overdraw, complex shaders, large surface area

Combining Profiler Data with Code Context

The most powerful technique: feed both profiler output and the source code to Claude Code. This lets AI connect the performance data to specific implementation choices.

Prompt
This composable renders a product card in our LazyColumn. During scroll, each card takes 28ms to compose (should be < 8ms). Here's the CPU profile hot methods: onBindViewHolder: 28ms total ├─ ProductCard composition: 22ms │ ├─ calculateShippingEstimate(): 14ms (network lookup!) │ └─ formatRelativeDate(): 5ms └─ image decode: 6ms And here's the composable code: @Composable fun ProductCard(product: Product) { val shipping = calculateShippingEstimate(product.warehouse) val dateText = formatRelativeDate(product.createdAt) // ... UI code }
AI immediately spots: calculateShippingEstimate() should not be called during composition — it's a synchronous function that takes 14ms. Move it to the ViewModel and pass the result as a parameter.

Your Profiling Checklist

  • Profile the specific interaction that feels slow, not the whole app session
  • Extract top-down method summaries from CPU traces for Claude Code
  • Check allocation counts in memory profiler for GC pressure
  • Use System Trace (Perfetto) to see frame-level timing breakdown
  • Focus on janky frames only — don’t analyze frames that are already fast
  • Combine profiler output with source code when prompting AI
  • Reuse objects instead of allocating in onBindViewHolder or composable bodies
  • Never do network, disk IO, or heavy computation during rendering

The Profiling Workflow

Don’t profile and optimize blindly. Follow this cycle: (1) Identify the interaction that feels slow. (2) Capture a focused trace of that interaction (5–10 seconds). (3) Extract the hot methods and janky frame breakdowns. (4) Feed them to Claude Code with the relevant source code. (5) Apply the fix. (6) Profile again to verify the improvement. Every optimization should be measurable.

Hands-On: Profile a Screen Transition

  1. Pick a slow interaction: Find a screen transition, list scroll, or animation that feels janky in your app. If everything is smooth, intentionally add Thread.sleep(20) inside a LazyColumn item to simulate jank.
  2. Capture a CPU trace: Open the Profiler, start a “Sample Java/Kotlin Methods” recording, perform the interaction, and stop recording.
  3. Extract hot methods: Switch to “Top Down” view, sort by total time, and copy the top 10 methods with their self time, total time, and call counts.
  4. Run /profile-cpu: Paste the data into Claude Code. Ask it to identify the top 3 bottlenecks and suggest fixes.
  5. Apply and re-measure: Fix the top issue, re-profile the same interaction, and compare frame times.
  6. Bonus: Capture a System Trace of the same interaction. Extract the janky frames and their phase breakdowns (measure, layout, draw). Feed to Claude Code for a frame-level diagnosis.
