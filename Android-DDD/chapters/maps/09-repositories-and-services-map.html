Chapter 9

Repositories and Domain Services

Memory Map
📚

The Big Idea

A repository is an in-memory Set<Aggregate> from the domain's perspective — not a DAO with extra steps. A domain service is stateless logic that doesn't fit on a single entity — not a god-class called UserManager. Both live in :domain; both hide complexity behind small, domain-shaped contracts.

📥 Anatomy of a Repository

1️⃣
One per aggregate type
OrderRepository, not BillingRepository. The repository is the gateway to one consistency boundary.
📚
Collection-shaped
Tiny interface: findById, save, remove, maybe add. Not 30 query methods.
🌫️
Hides persistence
Domain code never knows whether the data came from Room, Retrofit, RAM, or a CSV.
📦
Returns aggregates
Whole aggregates with invariants intact — never DTOs, partial entities, or property bags.

⚖️ Repository vs DAO

AspectRepository ✅DAO ❌ (when called Repository)

Dependency arrow points inward: :data depends on :domain, never the reverse. Reversing it is the architectural sin Clean and Hexagonal both exist to prevent.

🧮 Domain Service vs Use Case vs Infrastructure Service

KindWhat it doesWhere it lives
Domain service Stateless domain logic that spans entities (ShippingFeeCalculator) :domain
Application service (use case) Orchestrates aggregates + repos + services to fulfill one user intent :domain or :feature-*
Infrastructure service Talks to outside world: HTTP, push, sensors. Implements a domain interface. :data / :platform

🧭 The Specification Pattern

ByCustomer(id)
WithStatus(s)
OrderSpec
+ and
PlacedAfter(t)
repo.find(spec)

A small composable predicate. Keeps the repository interface tiny while letting callers express any query. Reach for it when a 30-method interface is the alternative.

🚨 Anti-Patterns

🦣
Repository-as-DAO
getOrdersByStatusAndCustomerAndDate + 14 more. SQL with method calls. Take DTOs, return DTOs.
👹
God-service
UserManager, OrderHelper, PaymentService. After 18 months: 90 methods, 11 deps, 1 Mockito test.
📛
FooRepositoryImpl in :domain
Defeats dependency direction. Move impl to :data; rename RoomFooRepository — persistence in the name.
📦
Reflexive Result<T> wrap
"Not found" is just null. Result is for domain-meaningful failures, not generic exception routing.
🪞
Use-case-per-method
FindOrderUseCase that just calls repo.findById. Forwarding stubs earn nothing — ViewModel can call directly.
🧱
Anemic repository
Returns DTOs, not aggregates. Caller has to validate, map, and reconstitute everywhere.

🧪 Fakes Over Mocks

  1. Define the contract once

    FakeOrderRepository : OrderRepository — 20 lines, lives in test fixtures.

  2. Reuse across every test

    Same fake, different setups. The contract is in code, not in mock-script ceremony.

  3. Test by state, not call recording

    assertEquals(saved, fake.findById(id)) — not verify(mock).save(any()).

  4. Survive refactors

    Changing one method signature updates the fake once; mock-heavy tests would all break.

🚫 When You Don't Need a Repository

  • 📋 Small finite reference data (countries, currencies). Wrap in a typed accessor; no abstraction needed.
  • ⏳ Per-session in-memory aggregates (a wizard, an in-progress form). Pass through the use case directly.
  • ⚙️ Settings stores read once at startup. A SettingsHolder with a StateFlow is enough.

Reach for the repository when complex persistence needs to be hidden behind a domain-shaped contract. If there's nothing to hide, don't pretend.

📖 Key Terms

Repository
A collection-like interface for one aggregate type. Hides persistence; speaks domain.
Specification
A small predicate object that the repository can interpret. Keeps the interface tiny while supporting rich queries.
Domain Service
Stateless domain logic that doesn't fit on a single entity. Inputs and outputs are domain types.
Application Service (Use Case)
Orchestrates aggregates, repositories, domain services, and events to fulfill one user intent.
Infrastructure Service
Talks to the outside world. Lives in :data/:platform. Implementation of a domain interface.
Fake
A working in-memory implementation of an interface, used in tests. Captures the contract; survives refactors.
Hilt @Binds
The seam where :data's implementation is wired to :domain's interface at the application graph.

❓ Quick Quiz

Test yourself

How many methods should a typical repository interface have?

Around four: findById, save, remove, and maybe find(spec) or observe. If yours has 17, it's a DAO with a fancy name.

Why do repositories return aggregates instead of DTOs?

Because invariants and behavior travel with the aggregate. A DTO is a property bag — the caller has to revalidate, remap, and reconstitute the aggregate at every call site, which is exactly the work the aggregate was supposed to encapsulate.

What's the difference between a domain service and an application service?

A domain service expresses domain logic that doesn't fit on a single entity (e.g., ShippingFeeCalculator). An application service orchestrates aggregates, repositories, and domain services to fulfill one user intent (e.g., PlaceOrderUseCase). Both are stateless; the application service is closer to the UI.

Why fakes over mocks?

A fake captures the contract in code. A mock script captures only one test's specific calls. Fakes outlive refactors; mock scripts rot the moment the test changes.

Where does RoomOrderRepository live?

In :data, not :domain. The interface lives in :domain; the Room-aware implementation lives in :data. Hilt's @Binds wires them together. Naming the impl after the persistence (RoomOrderRepository) is a hint at its layer.

When should you skip the repository abstraction?

Small finite reference data, per-session in-memory aggregates, simple settings stores. The repository's job is to hide complex persistence; if there's nothing to hide, the abstraction earns nothing.

Key Takeaways

  • A repository is a collection-shaped contract for one aggregate type. Four methods, not seventeen.
  • Repositories return whole aggregates. DTOs and partial entities defeat the consistency boundary.
  • Interface in :domain, implementation in :data. Dependency arrow points inward.
  • Use the specification pattern when query growth would force a giant interface. Tiny composable predicates beat 30 methods.
  • A domain service is stateless logic that doesn't fit on one entity. PriceCalculator ✅, UserManager ❌.
  • Three "services": domain (logic), application (orchestration), infrastructure (outside world). Different layers, different rates of change.
  • Prefer hand-written fakes over mocking frameworks. The fake captures the contract; the contract survives refactors.
  • Skip repositories for small finite data, in-memory wizards, and trivial settings stores. The abstraction needs something to hide.
  • Repository as DAO and god-service are the two anti-patterns to spot in code review. Both have the same fix: rename and shed.
