Chapter 10

DDD on Android Architecture

Memory Map
🏗️

The Big Idea

A DDD-shaped Android codebase is one where the build graph itself enforces the boundaries. Each bounded context gets its own :domain, :data, and :feature modules; :domain is pure Kotlin, depends on nothing Android, and runs in milliseconds. The compiler refuses what the team would otherwise have to argue about in code review.

📦 The Canonical Module Layout

🧠
:domain:*
Pure Kotlin/JVM. Aggregates, value objects, repository interfaces, domain services, events. Zero Android.
💾
:data:*
Room, Retrofit, DTOs, mappers. Repository implementations. The ACL between API/DB and domain.
🎨
:feature:*
UI per user-facing flow. ViewModels, Compose, navigation. Depends on :domain only.
🏠
:app
The shell. Hilt graph wires :data implementations into :domain interfaces. Knows everyone.

➡️ Dependency Arrows (read top to bottom)

:domain:*
←
:data:*

Both :feature and :data point at :domain. :domain points at nothing app-specific. :feature never directly depends on :data — Hilt bridges them at :app.

Cross-context coupling at the domain layer is exactly what bounded contexts exist to prevent. Talk via domain events or an explicit ACL — never by importing another context's types.

Confusing the two is the silent killer: domain rules end up in the UI layer, UI quirks pollute the domain.

🎯 ViewModel = Application Service, Not Domain

  • ✅ Orchestrates: calls use cases, maps domain state to UI state, forwards user intents.
  • ✅ Holds StateFlow<UiState> — not domain logic.
  • ❌ Validates emails — that belongs in an EmailAddress value object.
  • ❌ Computes discounts — that belongs in a domain service.
  • ❌ Decides invariants — that belongs on the aggregate.

🔗 Coroutines and Flow Discipline

  • 📥 Repos may return Flow<T> — aggregate lifecycles are naturally observable.
  • 🧮 Domain services return values — no I/O, no time, no flow.
  • 📐 Aggregates expose plain methods — order.place(fee, now) returns an event.
  • 🚫 No kotlinx-coroutines-android in :domain — that's an Android dispatcher leak.
  • 🧵 withContext(Dispatchers.IO) belongs in :data, never in :domain.

🗄️ Room ↔ Aggregate Mapping Rules

  1. Room rows are not aggregates

    They're storage shape. Mappers in :data rebuild the aggregate from rows.

  2. Aggregate writes are transactional

    @Transaction on saveAggregate ensures root + children land atomically.

  3. Load whole or nothing

    Lazy @Relation across aggregate boundaries is a footgun. Half-loaded aggregates break invariants silently.

  4. Never expose Room types upward

    Flow<WorkoutWithSets> stays in :data. The repository's observe returns Flow<Workout?>.

🌍 Why KMP Pays Off

  • 📚 :domain becomes commonMain — aggregates and VOs shared with iOS unchanged.
  • 🔌 androidMain / iosMain hold actual implementations of the same interfaces.
  • ⚡ If :domain was Android-free from day one, KMP migration is a build-script change, not a rewrite.
  • 🎯 Same discipline pays off on Wear, Auto, desktop, and CLI — optionality is the win.

🛡️ Three Lines of Defense

⚙️
Compiler-enforced
:domain uses kotlin-jvm, not com.android.library. Android types literally aren't on the classpath.
🔍
Lint-enforced
detekt ForbiddenImport rule blocks android.* / androidx.* in :domain at static analysis.
🧪
CI-enforced
Konsist or ArchUnit test fails the build if any :domain file imports an Android package.

🚨 Android-Specific Anti-Patterns

📱
Domain logic in Activities
Validation in onClickListener, business rules in TextWatcher. Push down into VOs and aggregates.
🧱
Rules in Adapters / Composables
Discounts computed in a RecyclerView.Adapter. Pre-compute in domain or ViewModel; UI just renders.
🗑️
The :common magnet
Becomes :everything. Replace with intent-shaped :platform:logging, :designsystem, etc.
🤖
ViewModel as god-object
Validation + orchestration + UI logic concentrated in one un-testable class. Push rules into :domain.

📖 Key Terms

Pure-Kotlin domain module
A Gradle module using kotlin-jvm (or KMP), depending on zero Android artifacts.
Module per context per layer
One bounded context maps to a :domain:context, :data:context, and one or more :feature:* modules.
Inward dependency rule
All arrows point toward :domain. :data depends on :domain; never the reverse.
Application service (use case)
Orchestrator inside :domain or :feature. ViewModels delegate to these instead of implementing them.
ACL mapper
Translates DTOs (HTTP) or Room rows (DB) to domain aggregates. Lives in :data.
Konsist / ArchUnit
JVM tools that run as tests and assert structural rules (no Android in :domain, no cross-context imports).
commonMain
The KMP source set that becomes shared :domain across Android and iOS.

❓ Quick Quiz

Test yourself

Why must :domain avoid the com.android.library Gradle plugin?

Because then Android types are on its classpath, and a teammate can accidentally import android.util.Log in an aggregate without anything failing the build. Using kotlin-jvm makes the constraint compiler-enforced — the import literally won't resolve.

Should the domain state and the UI state ever be the same type?

No. The domain state is the business state machine (Cart.Empty / Active / Submitted); the UI state is what the screen renders right now (Loading / Showing / Error) plus derived values. Conflating them either pollutes the domain with UI concerns or buries domain rules in the ViewModel.

Where does the Retrofit-DTO mapper live?

In :data, alongside the DTO and the API interface. :domain only ever sees the aggregate and its value objects. The mapper is the ACL between the API context and your domain context.

Can :feature:checkout directly depend on :data:checkout?

No. :feature depends on :domain only. Hilt at the :app graph wires the :data implementation into the interfaces :feature requested. Direct :feature → :data coupling leaks Room and Retrofit upward and breaks fast unit tests.

Why is @Relation a footgun for aggregates?

It can lazily load child rows after the parent, leaving the aggregate half-loaded when domain methods access it. Always load the whole aggregate or nothing. If the aggregate is too big to load whole, it's too big — revisit aggregate sizing.

What's the cheapest way KMP discipline pays off even without an iOS app?

Tests run in milliseconds (no Robolectric), the same domain works on Wear/Auto/desktop/CLI, and the day someone says "we should ship on iOS" the migration is a build-script change instead of a rewrite. Optionality, not portability, is the real win.

Key Takeaways

  • :domain is pure Kotlin and depends on nothing Android. Compiler-enforced via kotlin-jvm, not com.android.library.
  • One bounded context per :domain/:data/:feature trio. Cross-context talk goes through events or explicit ACLs.
  • Dependency arrows point inward. :data → :domain. :feature → :domain. Never the reverse, never :feature → :data.
  • Domain state machines and UI state machines are different types. Don't conflate Cart.Active with CheckoutUiState.Showing.
  • ViewModels orchestrate; they don't enforce. Every business rule belongs in a value object, aggregate, or domain service in :domain.
  • Repository interfaces may return Flow; aggregates and domain services don't. Threading lives in :data, not :domain.
  • Room rows are not aggregates — mappers in :data rebuild them. Aggregate writes are transactional. Lazy collections across boundaries are forbidden.
  • Retrofit DTOs are the API context's vocabulary. The mapper in :data is the ACL that protects your domain from backend churn.
  • KMP is one config away if :domain stayed Android-free. Test speed, Wear/Auto reuse, and iOS optionality all share the same discipline.
  • Lock the rules in mechanically: kotlin-jvm plugin + detekt ForbiddenImport + Konsist test. Belt, suspenders, parachute.
  • Anti-patterns to spot in PR review: domain logic in Activities, business rules in adapters/composables, :common magnet modules, ViewModel-as-god-object.
