Why DDD for Android?

The Big Idea

Software for a complex domain should mirror the domain — not the database, not the screen layout, not the framework. When code and business speak the same words, lies stop compounding.

🚕 The "Six Orders" Story

  1. The growth

    A ride-sharing app grows from 4 to 22 engineers in 18 months.

  2. The drift

    Six classes appear for one concept: RideRequest, Ride, TripRecord, Job, OrderDto, OrderEntity. Each team picked its own.

  3. The bug

    State-machine bug ships. Drivers get "completed" notifications mid-trip. Fix touches 14 modules, 800 lines of diff.

  4. The fix

    One-page UBIQUITOUS_LANGUAGE.md. Three teams agree the word is Trip. Refactor becomes mechanical — and exposes two more bugs.

🗝️ Key Terms

Domain
The business problem your software addresses — ride-hailing, payments, fitness tracking.
Ubiquitous Language
Same word, same meaning, everywhere — conversation, tickets, code, DB columns, analytics events.
Bounded Context
A boundary inside which a model and its language are consistent. Different contexts can use the same word with different meanings.
Anemic Domain Model
A class that only holds data, with no behavior; the rules live in helpers and ViewModels. The default Android shape — and a trap.
Strategic DDD
The high-altitude practices: language, contexts, maps, subdomains.
Tactical DDD
The in-code patterns: entities, value objects, aggregates, repositories, services, events.
Blue Book
Eric Evans, 2003. The original DDD canon.
Red Book
Vaughn Vernon, 2013. The practical sequel with code.

🤖 DDD + AI

Why AI changes the calculus

DDD always had an adoption problem — the boring upkeep got squeezed out under deadline. You still own strategy: what's a context, what's the language, what's core. AI carries the throughput: rename the 83rd String to EmailAddress, propagate a context-map change across 50,000 lines.

❓ Quick Quiz

Test yourself

What's wrong with a data class that has 20 fields and no methods?

It's an anemic model. The business rules that constrain those fields end up scattered across ViewModels, helpers, and use cases — drifting independently and breaking in production.

Why do mobile teams feel domain drift more than backend teams?

No service contract sits between the app and millions of devices. Feature teams pick their own model shapes; flags pile up; the :common module becomes the dumping ground. Only team discipline keeps the model coherent.

Strategic vs tactical DDD — which should you adopt first?

Strategic. Ubiquitous language + bounded contexts + a context map give you 80% of the value for 20% of the ceremony. Tactical patterns (aggregates, value objects) amplify a strategically aligned model; without alignment, they're just more boilerplate.

When is DDD overkill?

Simple CRUD apps, prototypes, throwaway code, single-developer projects. If you're the only reader of the code, you already have a ubiquitous language — yours.

If a designer says "Trip" in Figma, what should the analytics event be called?

trip_* — not ride_*, not order_*, not journey_*. Same word, every layer. That's what "ubiquitous" means in practice.

🛠️ Your Exercise

Twenty minutes. Pick three terms in your codebase you suspect mean different things to different people (User, Account, Order, Cart, Item, Profile, Status). For each: list every class with that name, write its real meaning in one sentence, check whether the meanings agree. Disagreements = bounded-context boundaries you just found for free.

Key Takeaways

  • Code should mirror the domain, not the database or the screen.
  • Anemic models are the default Android shape — and the most common trap.
  • Same word, different meaning is normal — that's what bounded contexts are for.
  • Strategic DDD (language + contexts) gives 80% of the value for 20% of the ceremony.
  • Tactical DDD (aggregates, value objects, repositories) amplifies a strategically aligned model — not a substitute.
  • DDD pays off in regulated, long-lived, multi-team, or cross-platform codebases. Skip it for prototypes and CRUD.
  • You own strategy; AI carries the throughput. The two together beat either alone.
