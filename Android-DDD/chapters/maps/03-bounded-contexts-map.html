Chapter 3

Bounded Contexts

Memory Map
🧱

The Big Idea

There is no single User, no single Customer, no single Order. There are several — one per bounded context. Each owns its own model, language, team, and boundary. Trying to unify them produces an 800-line god-object that nobody can change without six teams' approval.

🧱 The Four Ingredients

Model
(entities + rules)
Language
(words inside)
Bounded
Context
Team
(decision owner)
Boundary
(enforced edge)

Skip any one of these and the boundary leaks.

🧍 Customer Is Not Customer

💼
Sales
A lead with a pipeline stage, deal size, last-contacted date. May never have ordered.
💳
Billing
An invoice recipient with a tax ID, payment method, credit limit. Audit forbids deletion.
🎧
Support
A ticket-holder with a tier, satisfaction score, interaction history.
📦
Shipping
A delivery address. Often not even a person — a building, locker, or depot.

Four classes called Customer, four modules, four teams. Honest. Durable.

🔍 How to Spot a Boundary

SignalWhat it looks like
🗣️ LinguisticSame word, different meaning across teams. Cleanest signal.
👥 TeamTwo squads, two backlogs, two PMs — the seam between them is almost always a context.
🚀 DeploymentThings shipped together belong together. Dynamic feature modules = their own context.
Don't draw boundaries by technology. "Room layer," "Network layer," "UI layer" are horizontal technical layers, not bounded contexts. Contexts cut vertically through the stack — a context owns its UI, data, persistence, and events.

💀 The :common Decay Timeline

  1. Day 1

    :common has one User.kt, used by 2 features. Reasonable.

  2. Day 90

    Now contains Order, Address, PaymentMethod, Money. "Just for now."

  3. Day 365

    Depended on by everything. User has 38 fields, 6 nullable, 4 contexts inferred from prefixes. Touching it triggers full rebuild and breaks 4 unrelated features. Nobody owns it.

  4. The fix

    Stop adding to it. Extract one type at a time into its real owning context. Keep only tiny shared value objects (:shared:money) that do one thing.

🏦 Worked Example: A Neo-Bank

🪪
Onboarding
Applicant, VerificationStep, IdentityDocument, RiskScore.
💸
Transactions
Account, Transfer, Card, Beneficiary. Knows nothing of KYC.
📈
Investments
Portfolio, Position, Order, Instrument. Has its own Account — a brokerage account, not the same as Transactions.
📄
Statements
Period, StatementDocument, TaxYear. Read-only consumer of events.
Deliberate duplication is the point. "Account" lives in both Transactions and Investments. They are not the same shape and they should not be unified. That's the bounded context working as designed.

🛤️ Migration Sequence

  1. Draw the map first

    Even if it's wrong. Without it, the team will redesign on every PR.

  2. Pick the smallest, least-coupled context

    Statements / exports / settings. Read-only contexts have the fewest inbound deps.

  3. Create modules empty + green CI

    Open a PR that just adds :statements:domain / :data / :feature. Nothing inside.

  4. Move one type at a time

    Cut from :common, paste into the new context, fix imports, commit.

  5. Build the boundary

    Outsiders ask via interfaces in :domain, never via implementations in :data.

🚨 Wrong-Boundary Signals

📞
Cross-context chatter
Every change in A requires a change in B. The real boundary isn't where you drew it.
🏓
Types ping-ponging through ACLs
Translator becomes a maintenance burden — the contexts probably want to merge.
🦠
Shared types creeping back
A new type appears in :common "just for now." The team doesn't believe in the boundary.
🔥
Cross-team release escalations
Multiple teams own pieces of one context. Split the context or merge the teams.

🗝️ Key Terms

Bounded Context
An explicit boundary inside which a model and its language are consistent. Same word may mean different things outside it.
Context Map
A one-page diagram of all your contexts and the arrows between them. Every engineer should be able to draw it from memory.
Big Ball of Mud
The single-shared-model anti-pattern. An 800-line god-class with twenty nullable fields and six teams blocking every change.
Anti-Corruption Layer (ACL)
Translation code at the edge of a context. Keeps another context's words from leaking in. Full treatment in chapter 4.
:common module
The architectural "I'll just put it here for now." Quietly grows into :everything. Hostile to bounded contexts.
Subdomain (preview)
A piece of the business. Different from a bounded context (a piece of the solution). Often 1:1, sometimes not. Chapter 5.
Conway's Law
Software shape mirrors team communication shape. Bounded contexts say: "fine, let's be deliberate about it."

❓ Quick Quiz

Test yourself

Why is one shared User class across all features a trap?

It accumulates fields from every team that touches it: creditScore, shippingAddress, fraudFlags, vipTier... 800 lines later, every nullable field is a state-machine bug and changing it requires sign-off from six teams. The fix isn't refactoring — it's admitting there are several Users.

Are "Room layer" and "UI layer" bounded contexts?

No — those are technical / horizontal layers. Bounded contexts cut vertically through the stack. A Billing context owns its own UI, its own data layer, its own persistence, its own events. Confusing technical layers with contexts is the most common DDD mistake on Android.

What's the build-graph rule that keeps a Gradle bounded context honest?

:billing:feature depends on the interface in :billing:domain, never on the implementation in :billing:data. And :billing:* never imports :sales:*. The compiler enforces what stand-up agreements can't.

Two contexts both have a class called Account. Should you unify them?

Almost certainly no. If a Transactions Account has a balance and a Investments Account has positions, they are different concepts. Unifying produces a class half its uses ignore. Let them be two classes in two modules — that's the bounded context working as designed.

What's the Android-specific module type that forces bounded-context thinking?

Dynamic feature modules (loadable on demand via Play) and KMP shared modules. Both make the cost of a leaky boundary unignorable: dynamic features must be loadable independently, and KMP shared code can't import Android types at all.

🛠️ Your Exercise

One hour. By yourself. A piece of paper. List every Gradle module. Write one sentence per module: "this module owns ___." Cluster modules that own the same concept — that's a candidate context. Box per context, label with the team. Arrow per cross-context dependency, label "uses ___" or "subscribes to ___." Photo, Slack, ask the team: "Is this what we look like to you?"

Key Takeaways

  • A bounded context = model + language + team + boundary. Skip one and it leaks.
  • Same word, different meaning is normal — that's exactly what contexts solve.
  • On Android: roughly one context per Gradle :domain/:data/:feature trio.
  • Spot contexts via linguistic, team, and deployment boundaries — never by technical layer.
  • Cut vertically through the stack, not horizontally. Each context owns its UI, data, events.
  • The :common module is the Big Ball of Mud growing in your build graph — stop adding to it.
  • Deliberate duplication (two Account classes in two contexts) is honest, not wasteful.
  • Generate the context map from the Gradle graph — it's the only version that can't drift.
