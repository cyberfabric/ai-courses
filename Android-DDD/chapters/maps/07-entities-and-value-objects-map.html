Chapter 7

Entities and Value Objects

Memory Map
🆔

The Big Idea

One question separates the two atoms of tactical DDD: if you swapped this object for another with the exact same fields, would the system care? Yes → entity (identity matters). No → value object (defined by attributes). A Kotlin data class is not automatically a value object — it's one ingredient out of four.

🪤 The data class Trap

✅
What it gives you
Structural equals/hashCode, toString, componentN. The equality piece is exactly right for value objects.
⚠️
What it doesn't
No invariant enforcement. copy() bypasses factory validation (but init still runs). val can hold mutable refs.
🛡️
The fix
private constructor + factory + init { require(…) }. Now copy() still validates because init runs on every construction.
🚫
For entities
data class for an entity is wrong by default — its generated equals compares all fields. Override equals/hashCode to use id only.

🔁 The Refactor: Primitives → Domain Types

  1. Spot the anemic record

    data class User(val id: Long, val email: String, val countryCode: String, ...). Every field begs the question: validated? trimmed? which encoding?

  2. Wrap one primitive

    @JvmInline value class UserId(val value: Long). Or EmailAddress with private constructor + factory.

  3. Push validation into constructor

    init { require(EMAIL_REGEX.matches(raw)) }. Now invalid instances cannot exist anywhere.

  4. Let the compiler walk every call site

    Each scattered manual validation collapses into one place. StringUtils.formatXyz() moves onto the value as a method.

  5. Replace entity equality

    Override equals/hashCode by id. Two snapshots of the same user at different times are still the same entity.

📦 Where Each Lives on Android

🧪
:domain (pure Kotlin)
Entities (classes), value objects (inline value classes / data classes). No Android. No androidx. JVM unit tests in milliseconds.
💾
:data (Android)
Room @Entitys (separate from domain entities!), Retrofit DTOs, mappers DTO ↔ domain. The mapper is your tiny ACL.
🎨
:feature-* (Compose, ViewModels)
Depends on :domain. Validation runs on construction — your TextWatcher just calls EmailAddress.parse(input).
⚠️
Room @Entity ≠ domain entity
Same name, different shape. Persistence is its own context. Map between them in the repository.

🚨 Android-Specific Mistakes

📝
Validation in TextWatcher
Validation belongs on the value object, not the UI. One source of truth, every screen + test agrees.
📦
Primitives across screens
Pass UserId, not Long. Inline value classes parcel the same way; compiler catches the swap.
🗄️
Room as domain
No-arg constructors, public fields, @PrimaryKey — Room shape is bad domain shape.
🏷️
"It's just a label"
"Why wrap orderId?" Because the day a junior passes userId to refundOrder(), the compiler says nothing.

📖 Key Terms

Identity
What makes one entity distinct from another regardless of its fields. Usually an ID.
Structural Equality
Two objects equal if all their fields are equal. Right semantics for value objects, wrong for entities.
Inline Value Class
@JvmInline value class — a compile-time wrapper that disappears at runtime. Type safety without allocation.
Self-Validation
Invalid instances cannot exist. The constructor (or init block) throws on bad input.
Side-Effect-Free Method
A method that returns a new value instead of mutating. Standard for value objects.
Anemic Model
A record-like class with only fields and no behavior. The opposite of a real domain type.
ID-only equals
Override equals/hashCode on an entity to compare only the id. The same entity at any point in its lifecycle equals itself.

❓ Quick Quiz

Test yourself

What's the litmus test for entity vs value object?

"If I swapped this object for another with the same fields, would the system care?" Yes = entity (identity matters). No = value object (defined by attributes alone).

Why isn't a Kotlin data class automatically a value object?

It gives you structural equality (one of four criteria) but not the others: no invariant enforcement, copy() bypasses factory validation, and val fields can hold mutable references. Add private constructor + factory + init { require(…) } to close the gaps.

How should you override equals on an entity?

By ID only. other is User && other.id == id. Two snapshots of the same entity at different times must still compare equal — only its identity defines equality.

What's the runtime cost of @JvmInline value class UserId(val value: Long)?

Almost zero. The wrapper is a compile-time fiction in most contexts; the JVM sees a Long. So there's no performance excuse to keep raw Longs for IDs.

Should a Room @Entity be your domain entity?

No. Room @Entitys have constraints (no-arg constructor, public fields, @PrimaryKey) that make them poor domain types. Keep them in :data, write a mapper to your real domain entity, treat the mapper as a tiny ACL.

Why is the init block your "value object insurance policy"?

It runs on every construction path: the primary constructor, copy(), deserialization. Put your require(…) calls there and no path produces an invalid instance.

Key Takeaways

  • Identity vs equality is the central tactical-DDD distinction. Entities have identity; value objects are defined by their attributes.
  • A Kotlin data class alone is not a value object. Add private constructor + factory + init { require(…) } to make one.
  • Inline value classes (@JvmInline) give you typed wrappers with zero runtime overhead. Use them for every ID and every bounded scalar.
  • Override equals/hashCode on entities to compare by ID only. data class default equality is wrong for entities.
  • The four value-object criteria: immutable, self-validating, equal by attributes, side-effect-free behavior.
  • Room @Entity ≠ domain entity. Different module, different shape, mapper between them.
  • Lift primitives to value objects when they have rules, operations, or confusion risk. Stay primitive when they're truly scalar with no domain meaning.
  • All of this is pure Kotlin in :domain — tests run in milliseconds with no Android dependency.
  • "It's just a label" is the lie that leads to passing userId to refundOrder(orderId) in production.
