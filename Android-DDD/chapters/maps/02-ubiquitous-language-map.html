Ubiquitous Language

The Big Idea

One word per concept. The same word in chat, tickets, code, DB columns, and analytics. If a PM doesn't recognize your class name, your language isn't ubiquitous — it's a private dialect with translation overhead at every boundary.

🧱 The Five-Part Entry Shape

Definition
One sentence. What this term means in the domain — not how it's stored.
Examples
2–3 sentences a domain expert would actually say.
Do not say
The banned synonyms. The most important section — new engineers grep for it.
Related
Other terms this one depends on or co-occurs with.
Owner
Which Gradle module / bounded context owns the canonical definition.

🚨 Anti-Vocabulary Smells

👔
Manager
Missing entity, repository, or service with a real verb name.
🩹
Helper
Helper to whom? Almost always a missing domain word.
🗑️
Util / Utils
"I didn't know where to put this." The universal noise suffix.
⚙️
Processor / Handler
The verb is hiding inside — chargePayment(), not PaymentProcessor.
📦
Info / Data / Object
Pure suffix noise. UserInfo is just User in a hat.
🔧
Service (with state)
Services should be stateless. If yours holds a cache, it's an entity.

🔄 The 90-Minute Workshop

  1. List terms

    Walk through one feature out loud. Every noun goes on a sticky. 20 min of dumping.

  2. Cluster synonyms

    "User / member / account / profile" go in one group. PM gets the deciding vote.

  3. Define the winners

    One sentence per term. Dependencies on other terms? Add them to the list too.

  4. Write the "do not say" list

    The losing synonyms become the banned list. Most-grepped section.

  5. Commit + open PR

    Tag a PM in the description. Their replies are the actual asynchronous workshop.

🗝️ Key Terms

Ubiquitous Language
Same word, same meaning, in every artifact — chat, tickets, code, DB, analytics.
Translation Tax
The hidden cost of every team using a different word for the same concept.
Naming Compass
Nouns → classes, verbs → methods, adjectives → flags / value objects / states.
Anti-Vocabulary
Manager, Helper, Util, Processor, Handler, Info, Data — the noises engineers make when no domain word exists yet.

The natural decay of a language doc as features ship and APIs change.

❓ Quick Quiz

Test yourself

What is the brutal test for whether your language is ubiquitous?

Pick a random class name in your code. Say it out loud to a PM. If they don't recognize it, your language isn't ubiquitous — it's an engineer-only dialect with translation overhead.

Why do mobile teams need to be more deliberate about language than backend teams?

Backends get vocabulary alignment by accident — schemas, API contracts, and admin tools force it. On Android, nothing forces you to call Trip "Trip" instead of "Ride." Only team discipline does.

You see a class called OrderProcessor in a PR. What's the smell?

"Processor" is a noun-shaped wrapper around a hidden verb. The real domain language is probably placeOrder(), chargeOrder(), or fulfillOrder() — and those should be methods on Order or named use-case classes.

Why is "Activity" or "Screen" a smell inside the :domain module?

Those are presentation concepts. Domain doesn't know it's running on Android (or anywhere). If UI words leak in, you've smuggled framework thinking into the model and you've broken testability + KMP-readiness.

What's the cheapest, highest-leverage drift-prevention tool?

Two checkboxes in PULL_REQUEST_TEMPLATE.md: "did you add new domain terms?" and "did you introduce any anti-vocabulary suffixes?" Ten seconds to fill, surfaces 90% of drift before merge.

🛠️ Your Exercise

Thirty minutes. Today. Create UBIQUITOUS_LANGUAGE.md at your repo root. Pick three core terms (the ones from chapter 1's exercise are perfect). Write the five-part entry for each. Open a PR titled "Add initial UBIQUITOUS_LANGUAGE.md." Tag a PM or designer in the description and ask: "Does this match how you talk about these things?" The PR comments are the real workshop.

Key Takeaways

  • One word per concept — in chat, tickets, code, DB, analytics, design.
  • Lives in UBIQUITOUS_LANGUAGE.md at the repo root, reviewed in PRs like any other file.
  • Five-part entry shape: definition, examples, do not say, related, owner.
  • Naming compass: nouns → classes, verbs → methods, adjectives → flags / states.
  • Manager, Helper, Util, Processor, Handler — noise. Find the real domain word.
  • No Activity, Fragment, Screen, or View inside :domain.
  • PR template + tagged domain expert = 90% of drift caught before merge.
  • Once committed, Claude Code auto-loads the doc on every task — the language propagates for free.
