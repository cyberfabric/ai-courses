Chapter 11

The /ubiquitous-language Skill

Memory Map
🗣️

The Big Idea

Hand-written glossaries rot. Generated ones don't. A /ubiquitous-language Claude Code skill turns "we should write a glossary" from a quarterly chore into a one-prompt command anyone can run. Commit the skill once; every teammate gets the same vocabulary enforcement on every PR.

📂 Anatomy of a Skill

📍
Lives at .claude/commands/
Project-level. Filename becomes the slash command: ubiquitous-language.md → /ubiquitous-language.
📋
Frontmatter
description: for the command picker, argument-hint: for tooltips. Optional but useful.
🧠
Prompt body
Steps, output format, constraints. Verbs and rules, not nouns — the nouns come from the codebase.
🤝
Shared via git
Commit the skill; teammates pull and get the same command instantly. No setup ceremony.

🔌 Wiring the Doc into Claude

Run
/ubiquitous-language
→
UBIQUITOUS_LANGUAGE.md
generated at repo root
→
Add @UBIQUITOUS_LANGUAGE.md
to top of CLAUDE.md
→
Claude reads glossary
on every future task

The @-import is the load-bearing line. Without it, the file exists but isn't pulled into the session. With it, every PR Claude touches uses the team's vocabulary by default.

🧬 Per-Term Output Schema

  • 📛 Term — exact casing as used in code
  • 📖 Definition — one or two sentences in plain English
  • 📍 Code anchor — :domain:workouts/Workout.kt
  • 💬 Examples — idiomatic phrases from PRs / chat
  • 🚫 Do not say — alternative names the team has used historically
  • 🔗 Related — 2-4 sibling terms in the same context

🔁 The Two-Skill Pair

/ubiquitous-language
(generate)
CLAUDE.md
(@-import)
UBIQUITOUS_LANGUAGE.md
(the contract)
/sync-ubiquitous-language
(reconcile)
CI hook
(fail on drift)

One skill creates; the other reconciles. Sync is "print before write" — diff first, ask, then save. Together they keep the doc honest without manual upkeep.

🔄 The Iteration Loop

  1. Run on a small slice

    /ubiquitous-language workouts — one context, six terms, fast to read.

  2. Spot the misfires

    WorkoutDao shouldn't be in. Rpe got missed. Definitions sound like Wikipedia.

  3. Tighten the prompt

    Add explicit exclusions: "Exclude *Dao, *Dto, *Row, *Mapper — persistence and ACL classes, not domain."

  4. Re-run and diff

    Stop when the diff is empty for two runs in a row on the same code.

  5. Commit the prompt change

    Git history is your skill changelog — one commit per tightening round.

✅ Belongs in the Prompt vs ❌ Doesn't

✅ Belongs❌ Doesn't
Output format (per-term schema) Specific list of terms (those go in the glossary)
Files to read first (CLAUDE.md, gradle) Hard-coded module paths from your project
Bounded-context grouping rule Reviewer names, ticket prefixes
Anti-hallucination constraint "Be helpful," "be concise" filler
Argument handling ($ARGUMENTS) Examples of every possible term

🚨 Common Mistakes

🌫️
Too generic
"Generate a glossary" produces a glossary for any project. Specify format, grouping, constraints.
🪄
No code anchoring
Claude invents plausible-sounding terms. Force it to read :domain:* sources and cite file paths.
🗑️
Not committing the output
The file is the contract. Skill = generator, file = persistent context. Commit both, plus the CLAUDE.md import.
🔄
Rename without sync
Code drifts to Session; glossary still says Workout. Run sync before merge — or wire it into CI.

📖 Key Terms

Claude Code skill
A project-level slash command stored at .claude/commands/<name>.md. Filename = command name.
@-import
Markdown reference like @UBIQUITOUS_LANGUAGE.md that pulls another file into Claude's loaded context.
Code anchor
A file path + class name attached to every glossary term so a reviewer can grep, verify, and spot drift.
Do-not-say list
Per-term alternatives the team has explicitly rejected. The "what we used to call this" memory.
Print-before-write
Discipline of diffing proposed changes first, getting approval, then saving. Critical for sync-style skills.
Few-shot in a file
Keeping example glossary entries in EXAMPLES.md referenced by the skill, instead of inline in the prompt body.
Curation pass
The human review that follows the first generation: tighten definitions, fix casing, drop bad terms.

❓ Quick Quiz

Test yourself

Where does a Claude Code skill live, and how is it invoked?

In .claude/commands/<name>.md at the repo root. The filename without the extension becomes the slash command, e.g. ubiquitous-language.md → /ubiquitous-language.

Why isn't UBIQUITOUS_LANGUAGE.md auto-loaded the way CLAUDE.md is?

Because Claude Code only auto-loads CLAUDE.md. To pull in the glossary, add the line @UBIQUITOUS_LANGUAGE.md to the top of CLAUDE.md. That import is what makes the glossary part of every session.

What's the single biggest difference between a hallucinated glossary and a useful one?

Anchoring on real code. The skill must read :domain:* sources and require a file path for every term. "Never invent terms; if you cannot find code or chat support for a term, leave it out" is the constraint that does the heavy lifting.

Why have a separate /sync-ubiquitous-language skill?

Generation is rare; reconciliation is frequent. Sync prints a diff (stale entries, new terms, contradictions) and waits for approval before writing — safer than re-running generation, and faster than manual review.

What's the iteration loop for tightening a skill prompt?

Run on a small slice (one context). Spot misfires. Tighten the prompt with explicit rules. Re-run and diff. Stop when the diff is empty for two consecutive runs on the same code. Each tightening is one commit; git history becomes the skill's changelog.

What are two other skills that follow the same pattern?

Any of: /context-map emits the bounded-context relationship doc; /aggregate-audit lists aggregates and flags oversized ones; /anti-corruption-layers reports DTO ↔ aggregate mappers; /event-catalog lists domain events with publishers and subscribers.

Key Takeaways

  • A Claude Code skill is just a Markdown file at .claude/commands/<name>.md. Filename becomes the slash command.
  • Hand-written glossaries rot; generated ones don't. Make the doc one prompt away — commit the generator, not just the output.
  • UBIQUITOUS_LANGUAGE.md isn't auto-loaded. Wire it in with a single @UBIQUITOUS_LANGUAGE.md line at the top of CLAUDE.md.
  • Per-term schema: term, definition, code anchor, examples, do-not-say, related. The code anchor is what makes the glossary verifiable.
  • Constrain hallucination explicitly: "never invent terms; require a file anchor." Without that line, Claude will invent.
  • Pair with a /sync-ubiquitous-language reconciler. Print before write — never silently overwrite the contract.
  • Commit the skill to .claude/commands/. Every teammate's Claude Code installation gets it on the next pull. Zero setup.
  • The prompt holds verbs and constraints; the codebase supplies the nouns. Don't bake project-specific terms into the skill itself.
  • Iterate: small slice, spot misfires, tighten, re-run, diff. One commit per tightening; git log is your changelog.
  • The same pattern scales: /context-map, /aggregate-audit, /anti-corruption-layers, /event-catalog. Each is a few-dozen-line skill that produces a fresh, never-stale doc.
