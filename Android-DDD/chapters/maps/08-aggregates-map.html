Chapter 8

Aggregates

Memory Map
🛡️

The Big Idea

An aggregate is a scope of consistency, not a "natural cluster of related stuff." Draw the boundary around an invariant — a rule that must always be true — and only as wide as that invariant requires. One root, one boundary, one transaction. Everything else is detail.

🧩 The Three Things That Make an Aggregate

🌳
One Root
Outsiders only hold a reference to the root entity. Internal entities and value objects are reached through the root.
🛡️
One Consistency Boundary
The root protects an invariant. The boundary is drawn around exactly what that invariant requires — nothing more.
⚛️
One Transaction
Loaded together, mutated together, saved together. No half-states ever leak out.

🪜 Vernon's Four Rules of Effective Aggregate Design

  1. Model true invariants in consistency boundaries

    Only put inside the aggregate what cannot tolerate being inconsistent for even a moment.

  2. Design small aggregates

    Most should be one root + a few value objects. Big "tree" aggregates are a smell.

  3. Reference other aggregates by ID only

    Order holds a CustomerId, never a Customer. Fewer locks, looser coupling.

  4. Use eventual consistency outside the boundary

    Cross-aggregate updates flow through domain events — never one giant transaction.

🤖 Aggregates on Android (Room)

📥
@Transaction is your friend
Wrap aggregate load and save in @Transaction. Atomicity at the persistence layer.
🚫
@Relation across boundaries = trap
Customer @Relation List<Order> silently glues two aggregates. Use @Relation only inside one aggregate.
🥚
Eager, complete, or null
Repository returns the full aggregate or nothing. No half-hydrated stubs, no lazy collections.
🆔
Mint IDs client-side
UUID/ULID at the moment of creation. Offline-first works; events carry real IDs from birth.

🎯 Finding Aggregates: Follow the Invariants

  1. List the rules in plain English

    "Total = sum of subtotals." "Cannot ship before paid." "A coupon is used at most once per order."

  2. Find the entity responsible for each rule

    That entity is your aggregate root. Things that share an invariant probably share an aggregate.

  3. Pressure-test "must be atomic" claims

    Most "we need atomic consistency" arguments dissolve when you ask: "is a 200ms stale read actually a problem?"

  4. Default to eventual + events

    If two aggregates can be eventually consistent (almost always), keep them separate and bridge with events.

📖 Key Terms

Aggregate
A cluster of entities + value objects treated as one unit for data changes.
Aggregate Root
The single entity that outsiders reference. The only entry point into the aggregate.
Consistency Boundary
The seam around an invariant. Everything inside is consistent at every commit; everything outside is eventually consistent.
Invariant
A rule that must always be true. The reason an aggregate exists.
One-aggregate-per-transaction rule
Mutate one aggregate per commit. Cross-aggregate updates use events.
Reference by ID
Holding another aggregate's ID instead of embedding it. Loose coupling, independent storage, no shared lock.
Eventual Consistency
Cross-aggregate state lines up "eventually" via event handlers, not atomically in one transaction.
Law of Demeter (here)
"Talk only to the root." Never reach order.lines[0].sku; call a method on the root instead.

🚨 Anti-Patterns

🌲
Graph aggregate
Owns half the schema (Customer→Orders→Lines→Refunds). Massive joins, every mutation locks the customer.
📭
Anemic aggregate
Root is a property bag; logic lives in OrderManager. If the root doesn't enforce invariants, it's a record — not a root.
💤
Lazy collection
Internal collection returned as Flow/lazy Sequence. Breaks the "loaded as a unit" promise.
🔗
Two-aggregate transaction
Loading + mutating two aggregates in one Room @Transaction. Doubles lock surface, hides invariants.

❓ Quick Quiz

Test yourself

What's the difference between an aggregate and a "natural cluster of related rows"?

An aggregate is defined by an invariant, not by foreign-key relationships. The boundary is drawn only as wide as the invariant requires. Without that contract, you have a graph — not an aggregate.

Why isn't LineItem its own aggregate?

Nothing outside the order references a specific line item independently, a line item can't exist without an order, and the invariant "total = sum of subtotals" lives on the order. All three signals point inward.

How do you keep Order and Customer consistent if they're separate aggregates?

Eventual consistency via events. Mutate one aggregate in transaction 1, emit an event, let a handler update the other aggregate in transaction 2. Idempotent handler + persisted event log handles the failure case.

Why is Room's @Relation across aggregate boundaries dangerous?

It silently couples two aggregates into one query and one transaction. Customer @Relation List<Order> turns "load one customer" into "load every order ever" and prevents the two from ever scaling independently. Use @Relation only inside a single aggregate.

Why mint aggregate IDs on the client?

UUID/ULID at creation time means events carry a real ID immediately, offline-first works naturally, and you skip the round-trip to the database to learn what to call the new thing.

What lives in the ViewModel vs the aggregate?

ViewModel: load via repository, call domain methods, save, publish events, project to UI state. Aggregate: enforce invariants, validate state transitions, decide what events to emit. The interesting work happens inside the aggregate.

Key Takeaways

  • An aggregate is a scope of consistency: one root, one boundary, one transaction.
  • Boundaries are drawn around invariants, not around natural row relationships.
  • Vernon's rules: small, by-ID references, eventual consistency outside, true invariants inside.
  • Cross-aggregate updates always flow through domain events — never one giant transaction.
  • Outsiders only talk to the root. Never reach order.lines[0].sku; call a method on the root.
  • On Room: wrap aggregate I/O in @Transaction; never use @Relation across aggregate boundaries.
  • Repositories return the full aggregate or null — never half-hydrated, never lazy-loaded.
  • Mint aggregate IDs client-side (ULID). Offline-first, event-friendly, no round-trip.
  • Anti-patterns: graph aggregate, anemic root, lazy collections, two-aggregate transactions. Spot them in code review.
