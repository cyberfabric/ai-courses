The Big Idea

Bounded contexts have to talk — the question is on whose terms. Eight named patterns (ACL, Shared Kernel, Customer-Supplier, Conformist, OHS/PL, Separate Ways, Big Ball of Mud) cover every cross-context relationship you'll ever draw. Label every arrow on your map. The label is the design.

🧭 The Eight Patterns

🛡️
Anti-Corruption Layer (ACL)
Translation layer at the boundary. Foreign DTOs in, clean domain types out. Your favorite weapon for legacy and third-party SDKs.
🤝
Shared Kernel
A tiny jointly-owned model fragment. Use for universal value objects (Money, Currency) only. High coupling cost.
📋
Customer-Supplier
Downstream team has real influence on the upstream backlog. Healthy default for cooperating teams inside one org.
🙇
Conformist
You accept the upstream model as-is. Cheap until upstream breaks, then catastrophic. Use only for genuinely stable foreign types.
🌐
Open Host Service (OHS)
An API designed for many downstream consumers. Internally on Android: a sealed event hierarchy + a small repo interface.
📜
Published Language (PL)
The versioned, documented format an OHS speaks. Treated as a real contract. Add a version field from day one.
↔️
Separate Ways
No integration at all. Two contexts have similar concepts but unifying costs more than duplicating. The most-undersold pattern.
🐘
Big Ball of Mud
The legacy elephant. Don't fix it — name it as a context and wrap it in an ACL. New work in clean contexts; ball stays muddy.

🛠️ The 4-Step ACL Recipe (Android)

  1. Foreign DTOs (internal)

    The vendor's ugly types. Kept internal to the data module.

  2. Foreign API client (internal)

    Retrofit interface, gRPC stub, vendor SDK wrapper. Also internal.

  3. Private mappers

    Pure functions: FooDto.toDomain(): Foo. Tested in isolation.

  4. Public domain interface

    Implements a Repository from :context:domain. The ONLY public thing in the package.

📖 Key Terms

Anti-Corruption Layer
Named translation layer at a context boundary. Contains mappers but is more than them.
Shared Kernel
Small jointly-owned model fragment. Both teams approve every change.
Customer-Supplier
Downstream team has real influence on the upstream team's backlog.
Conformist
Downstream accepts upstream model as-is. No translation.
Open Host Service
Upstream API deliberately designed for many downstream consumers.
Published Language
Versioned, documented format an OHS speaks.
Separate Ways
Deliberate decision not to integrate. Duplication beats wrong abstraction.
Big Ball of Mud
Legacy context with no internal boundaries. Wrap, don't refactor.

🚨 Stale-Map Signals

🔥
Cross-team escalations every release
Boundary is in the wrong place. Either split the context or merge the teams.
📞
Recurring outages at one integration
Contract is wrong. Upgrade conformist → ACL.
🏓
Types ping-pong through 3 ACLs
One of the boundaries is fake. Merge the contexts on either side.
🧐
Map disagrees with build graph
Run ./gradlew :module-graph. If they differ, the diagram is usually lying.

❓ Quick Quiz

Test yourself

What's the difference between an ACL and a mapper?

A mapper is a one-line type conversion. An ACL is a deliberate, named layer with its own vocabulary translation. The ACL contains mappers, but it shields your model from foreign concepts entirely — nothing outside the ACL ever sees a foreign type.

When is Conformist the right choice?

For genuinely stable foreign types that don't change for business reasons (e.g., JDK Instant, AndroidX Bitmap). Default to ACL for anything that looks like business data.

Your :common module has 30 types and is used by 6 contexts. Is that a Shared Kernel?

No. A real Shared Kernel is <10 types, jointly owned via CODEOWNERS, version-locked, and changes only with all teams' approval. :common with 30 types is a junk drawer — an unnamed context that needs splitting.

What does "U" on the upstream side and "D, ACL" on the downstream side of an arrow tell you?

The upstream context dictates its own model; the downstream context translates the upstream's model into its own through an anti-corruption layer. The downstream is shielded from upstream changes.

You inherited a 500-class :legacy module. What do you do?

Name it as a context ("Big Ball of Mud"), wrap it in an ACL, and do all new work in clean contexts that consume it through the ACL. Don't try to refactor the ball — bound its blast radius.

📱 Mobile-Specific Anti-Patterns

🪵
The "shared module" that grew up
:shared now has entities + repos + ViewModels and 6 teams committing to it. It's a context with no name. Rename, split, or accept it as the new main context.
📡
DTOs leaking into ViewModels
If UserDto from Retrofit reaches your ViewModel, you're a conformist to your backend. Add an ACL at :context:data.

Key Takeaways

  • Bounded contexts must talk — the question is on whose terms, not whether.
  • Eight patterns cover every relationship: ACL, SK, C/S, Conformist, OHS, PL, Separate Ways, Big Ball of Mud.
  • ACL is the most useful pattern: foreign DTO → API client → private mapper → clean domain interface.
  • A Shared Kernel is <10 types, jointly owned via CODEOWNERS, near-zero churn. Anything else is a misnamed context.
  • Conformist is cheap until upstream breaks. Use only for stable, non-business foreign types.
  • For legacy: name it a context, wrap it in an ACL, never refactor it directly.
  • Label every arrow with U/D + a pattern name. The label is the design.
  • Compare your context map to ./gradlew :module-graph. When they disagree, the diagram is usually lying.
