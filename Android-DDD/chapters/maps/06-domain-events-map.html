Chapter 6

Domain Events

Memory Map
⚡

The Big Idea

Stop modeling only nouns — model verbs too. A domain event is an immutable, past-tense, business-named fact captured at the moment it happens. Get the events right and boundaries, audits, replay, and cross-context coupling all get easier. If your event names all end in "Updated," you're doing CRUD with extra steps.

📜 The Three Rules

⏪
Past tense
OrderPlaced, not PlaceOrder. PaymentCaptured, not CapturePayment. The fact already happened.
🪨
Immutable
No edits. Corrections are NEW events (e.g., OrderCancelled). The history is append-only.
🗣️
Domain-named
A PM should recognize the name on a Slack screenshot. No vendor leaks (StripeChargeSucceeded ❌).

🔄 The Checkout Event Chain

  1. Tap "Pay now" → Command PlaceOrder

    ViewModel sends command to use case.

  2. Use case loads Cart aggregate, calls cart.place()

    Aggregate validates, returns an OrderPlaced event.

  3. Bus publishes OrderPlaced

    All subscribers learn the same fact at the same moment.

  4. Subscribers react independently

    Payment captures → PaymentCaptured. Notifications schedules a push. Statements records the line.

  5. Later: OrderShipped arrives

    Same notifications handler picks it up and schedules a delivery push. No producer/consumer coupling.

📡 Publishing on Android

⚡
SharedFlow (default)
In-process, hot. Subscribers in the same process get events as they happen. No persistence. The 95% case.
📚
Room-backed event log
SharedFlow + Room table. Events persist; subscribers can replay. Use for offline-first apps and bug forensics.
🌐
Backend broker
Kafka, NATS, SQS. Use only when events leave the device. App publishes via API; broker handles backend fanout.

📖 Key Terms

Domain Event
Immutable, past-tense, business-named fact. Captured at the moment it happens.
Command
An imperative request that may be rejected. Produces 0..N events.
Event Storming
3-hour wall-of-stickies workshop that surfaces events, commands, aggregates, and boundaries with the whole team.
Event Bus
The thing subscribers attach to. On Android, almost always a SharedFlow.
Replay
Rebuilding state by reading the event log from the start. Free audit trail + bug forensics.
Schema Version
An integer field on every event. Looks like overhead until your first migration, then it's gold.
Event-as-Published-Language
Events are how upstream contexts speak to downstream contexts. Each consumer translates at its ACL.

🚨 Anti-Patterns

🎭
"EntityChanged" disguise
UserUpdated + the entire user object = CRUD in costume. Real events are specific: UserEmailVerified.
🎒
Fat events
30 fields per event = model leak. Carry the minimum (ID + delta). Consumers query upstream if they need more.
🪁
Events for everything
If a single caller awaits a single result, that's a method call — not architecture theater.
⏳
Sync flow on async events
Faking request/response on top of an event bus = worst of both worlds.

✅ When to Use vs When to Skip

Use a domain event when…Skip when…
Multiple subscribers react independentlyOne caller, one return value
Producer doesn't (and shouldn't) know consumersSingle-actor synchronous flow
The interaction crosses a context boundaryInside one tiny context with one consumer
You want audit / replay / forensicsNo business meaning to the change
The fact has business meaning a PM would recognizeJust a UX signal ("user opened a screen")

🎯 Naming Events Well

  • ✅ Past tense — OrderPlaced, never OrderPlacement.
  • ✅ Subject + verb — InvoiceIssued, SubscriptionCancelled.
  • ✅ Specific — UserEmailVerified beats UserUpdated.
  • ✅ Domain words, not technical — PaymentCaptured, not StripeChargeSucceeded.
  • ✅ One verb per event — OrderPlacedAndPaid is two events.

❓ Quick Quiz

Test yourself

Why must events be in past tense?

Because they are facts, not requests. PlaceOrder is a wish that may be rejected. OrderPlaced is a historical fact you can't undo. The tense is the difference between a command and an event.

Your event is named UserUpdated. Why is that a smell?

It's CRUD wearing a costume. Real domain events name the specific business transition: UserEmailVerified, UserSubscriptionRenewed, UserNameChanged. If you can't pick a specific past-tense verb, you don't have a domain event — you have a database write.

Do you need Kafka for domain events on Android?

Almost never. SharedFlow covers 95% of in-process needs. Add a Room-backed log when you want persistence/replay. Reach for a backend broker only when the event truly needs to leave the device.

What's a "schema version" field on an event for?

It looks like overhead until you have to migrate stored events. Once you've done one event-shape migration, you'll never ship an event without a version field again.

What are the four output stages of an event-storming workshop?

(1) Events on orange stickies in timeline order. (2) Commands on blue stickies. (3) Aggregates on yellow stickies. (4) Context boundaries drawn through the wall.

Key Takeaways

  • Domain events are immutable, past-tense, business-named facts. Three rules — everything else is detail.
  • Events ≠ commands. Past tense vs imperative. Fact vs wish. Cannot fail vs can be rejected.
  • Event storming = orange-sticky workshop with PM + engineers + QA. 3 hours, one wall. Output is the domain map.
  • In Kotlin: a sealed class with one data class per event. Always include occurredAt and a version: Int.
  • On Android: SharedFlow is the default bus; Room is the persistence layer; a backend broker is a last resort.
  • Events crossing context boundaries = the canonical Published Language. Each consumer translates at its ACL.
  • Cross-aggregate consistency is always eventual, mediated by events. Never two aggregates in one transaction.
  • If your event names end in "Updated"/"Changed"/"Modified," you're doing CRUD with an extra step.
  • Skip events for single-actor request/response flows. Don't pay the indirection cost when a method call is clearer.
