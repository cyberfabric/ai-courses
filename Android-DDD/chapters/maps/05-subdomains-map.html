Chapter 5

Subdomains: Core, Supporting, Generic

Memory Map
🎯

The Big Idea

Not all the code in your repo deserves the same level of investment. Sort every subdomain into Core, Supporting, or Generic using one question: "if a competitor copied this exactly, would we lose?" The answer tells you where to put your senior engineers, what to ship boring, and what to buy from a vendor.

🪣 The Three Buckets

🏆
Core
Your competitive moat. Where you must outclass everyone. Pick at most 1-2. Senior engineers, deep tests, custom build, multi-quarter horizon.
🛠️
Supporting
Necessary, custom-shaped, but not differentiating. Mid-level engineers, simple architecture, ship fast, refactor when it bleeds.
🔌
Generic
Identical at every company. Auth, payments, email, push, analytics. Buy a vendor, wrap in ACL, never touch again.

📦 Subdomains in settings.gradle.kts

🏆
Core gets the trio
:workout-planning:domain + :data + :feature. Full DDD treatment. The heart of the codebase.
🛠️
Supporting collapses modules
:tracking trio for medium logic; :profiles:feature alone if there's barely any logic. No need to over-engineer.
🔌
Generic gets a :vendor suffix
:auth:vendor, :payments:vendor, :analytics:vendor. The suffix screams "thin wrapper, do not extend."

📋 Key Terms

Core Domain
The subdomain that gives the business its competitive advantage. Rare — pick at most 1-2.
Supporting Subdomain
Necessary, custom to your business, but not a market differentiator. Aim for "competently boring."
Generic Subdomain
Necessary work that's identical at every company. Auth, payments, email, push, file storage.
The Litmus Test
"If a competitor copied this exactly, would we lose?" YES = core. NO = supporting/generic.
Subdomain
A piece of the business problem. Lives in slide decks and OKRs.
Bounded Context
A piece of the solution. Lives in the build graph and the team org.
:vendor module suffix
Naming convention signaling "this is a generic-subdomain wrapper, don't extend it."

🚨 Common Mistakes

🎉
"Everything is core"
Five cores = budget spread thin = nothing wins. Force-rank, demote.
🛠️
Building generic stuff
"How hard can auth be?" Quarters disappear. Buy a vendor.
🤖
Buying core stuff
Off-the-shelf recommenders give you everyone-else's experience.
💎
"Most interesting" ≠ Core
Engineers love deep technical problems. The deep problem isn't always the moat.
🪦
Sunk-cost reasoning
"We built it ourselves" isn't evidence it's core. The decision to keep maintaining is fresh, every year.
🕰️
Letting classifications rot
2022's core is 2026's commodity (search, recommendations, feature flags). Re-classify yearly.

🗓️ The 30-Minute Subdomain Audit

  1. List every Gradle module

    Open settings.gradle.kts. Pull all modules into a spreadsheet.

  2. Map module → subdomain

    Modules serving multiple subdomains are a flag worth investigating.

  3. Apply the litmus test

    Out loud. Brutally honest. With a PM in the room.

  4. Tag CORE / SUPPORTING / GENERIC

    One label per subdomain.

  5. Force-rank the cores

    If you have >2 cores, demote until you have 1-2.

  6. For each Generic: ask "why aren't we using a vendor?"

    Usually the answer is "we didn't think about it" — a fixable problem.

  7. For each Core: count the senior engineers there

    If zero, you've found your strategic mismatch. Raise it next planning cycle.

❓ Quick Quiz

Test yourself

What's the litmus test for Core?

"If a competitor copied this exactly, would we lose?" If YES, it's core. The test is competitive, not technical — "this code is hard" doesn't qualify it as core.

How many cores should a typical company have?

One or two. Almost never five. If your spreadsheet shows seven cores, you flinched on the litmus test — force-rank and demote.

What's the difference between a subdomain and a bounded context?

A subdomain is a piece of the business problem ("we need to verify identities"). A bounded context is a piece of the solution (:onboarding:domain). The classification (core/supporting/generic) belongs to the subdomain.

Your team wants to write its own auth. What do you say?

Auth is generic. Buy a vendor (Auth0, Firebase Auth, Clerk), wrap behind an ACL. Reinvesting senior-engineer time on auth means under-investing in the actual moat.

When should you re-classify subdomains?

Yearly. Vendor commoditization (what was core in 2018 is SaaS in 2026), strategic pivots, and regulatory shifts all change the answer. Don't wait for the architecture review to discover the moat moved.

Key Takeaways

  • Three buckets: Core (moat), Supporting (necessary plumbing), Generic (everyone has one).
  • The litmus test is one sentence: "If a competitor copied this exactly, would we lose?"
  • Pick at most 1-2 cores. "Everything is core" is the most common and most expensive mistake.
  • Subdomain ≠ Bounded Context. Subdomains are business problems; contexts are code. Classification belongs to subdomains.
  • Build core, build supporting (default), buy generic (default). Don't reinvent auth.
  • Encode the classification in settings.gradle.kts: trios for core, slim modules for supporting, :vendor-suffixed wrappers for generic.
  • Different buckets get different architectures, owners, test depth, and refactor cadence. Treating them all the same is over- or under-engineering.
  • Re-classify yearly. Today's core is tomorrow's commodity once a vendor catches up.
