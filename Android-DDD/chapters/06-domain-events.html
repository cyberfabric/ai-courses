Chapter 6

Domain Events

From "what is" to "what happens" — modeling change as a first-class thing

So far this course has been about nouns. Bounded contexts contain entities. Subdomains contain models. Even chapter 4's context map was nouns connected by labeled lines. This chapter is about verbs.

Most Android codebases are over-built on nouns. There's a User, an Order, a Payment. There are tables for each, classes for each, repositories for each. What's missing — almost everywhere — is a clear, named, persisted record of what just happened. That gap is where domain events come in.

The shift in perspective

Look at any feature in your app and ask: "what is the most important thing that happens here?" In a checkout flow it's not "the cart exists" or "the order exists." It's the order was placed. In KYC it's not "the applicant has fields." It's the applicant was approved. In ride-hailing it's not "the ride exists." It's the driver accepted, then the rider was picked up, then the ride completed.

The verb-first lens is a different way to scope features. When you put events at the center of your model:

  • Boundaries between contexts become obvious — events are exactly what crosses them.
  • Audit and replay become free — the events are the history.
  • UI becomes a projection — "show me what's happened so far" is now a query, not a join.
  • Bugs surface earlier — if you can't name the event that just happened, you don't understand the feature yet.

What a domain event actually is

Domain Event — an immutable, named record of something significant that happened in the domain, in the past tense, captured at the moment it occurred. It belongs to the ubiquitous language: PMs and engineers should both call it the same thing.
Do not say "update," "change," "notification," or "callback." An update is CRUD; a change is technical; a notification is a delivery mechanism; a callback is a hook. A domain event is a fact — specifically, a fact that the business cares about, named the way the business names it, captured forever.

Three rules separate a domain event from a generic message:

⏪
Past tense. OrderPlaced, not PlaceOrder. PaymentCaptured, not CapturePayment.
🪨
Immutable. The fact already happened. It cannot be edited. Corrections are new events.
🗣️
Domain-named. A PM should recognize the name on a slack screenshot.

Events vs commands: tense matters

Events are facts. Commands are wishes. They are not the same thing, and conflating them is the most common modeling error in event-driven code.

Event — past tense, immutable
data class OrderPlaced( val id: OrderId, val placedAt: Instant, val total: Money ) // "Yes, this happened. End of story." // Cannot be rejected. Cannot fail. // Already a historical fact.
Command — imperative, can fail
data class PlaceOrder( val cartId: CartId, val paymentMethodId: PaymentMethodId ) // "Please make this happen, if valid." // Can be rejected ("out of stock"). // Produces zero, one, or many events. // Lives in the request layer.

The flow is always: command in → aggregate validates → event(s) out. The command may be rejected. The event, once emitted, is a fact you cannot undo. (You can compensate it with another event — OrderCancelled — but you can't pretend OrderPlaced never happened.)

Event storming: the workshop format

Event storming is a low-tech, high-bandwidth workshop format invented by Alberto Brandolini for discovering domain events with a whole team in one room. You do not need to be a DDD expert to run it. You need a long wall, a stack of orange sticky notes, and three hours.

The basic flow:

📝 Events on orange stickies
→
📅 Timeline order
→
🟦 Commands on blue stickies
→
🟨 Aggregates emerge
→
🟪 Boundaries drawn

Stage one: everyone in the room writes domain events on orange sticky notes — past tense, one event per sticky — and slaps them on the wall in rough chronological order. The PM, the senior engineers, the QA lead, and ideally the support lead all participate. Quantity over quality. You will end up with 60-200 stickies.

Stage two: the team merges duplicates ("OrderShipped" and "ShippingNotificationSent" might be the same event, or two distinct ones — argue it out), reorders for true timeline, and identifies hot spots (clusters where everyone has questions or disagreements).

Stage three: blue sticky notes for commands ("PlaceOrder," "CancelOrder") and yellow ones for aggregates (the things that emit events — the "Order" itself). By the end, you have a complete event-storming canvas: every fact the system records, every command that might trigger one, every aggregate that owns each emission.

Stage four: you draw context boundaries through the wall. The seams almost always reveal themselves — clusters of events that share vocabulary cluster naturally; events that need translation (because the same noun has different meanings on either side) reveal a context boundary.

Run an event-storming session before your next big feature. It costs three hours; it saves three months. The team alignment from "we all wrote on this wall together" is hard to overstate. You will leave with a shared vocabulary, an obvious feature scope, and a list of events you can paste directly into your sealed-class hierarchy.

Domain events in Kotlin

Events as code: a sealed class with one data-class subtype per event, all immutable, all named in the past tense. This is one of the few places in DDD where Kotlin's type system makes the canonical pattern almost trivial.

// :checkout:domain — pure Kotlin, no Android sealed class CheckoutEvent { abstract val cartId: CartId abstract val occurredAt: Instant abstract val version: Int // schema version, never 0 data class CartItemAdded( override val cartId: CartId, override val occurredAt: Instant, override val version: Int = 1, val item: LineItem ) : CheckoutEvent() data class OrderPlaced( override val cartId: CartId, override val occurredAt: Instant, override val version: Int = 1, val orderId: OrderId, val total: Money, val customerId: CustomerId ) : CheckoutEvent() data class PaymentCaptured( override val cartId: CartId, override val occurredAt: Instant, override val version: Int = 1, val orderId: OrderId, val captured: Money, val provider: PaymentProvider ) : CheckoutEvent() data class OrderShipped( override val cartId: CartId, override val occurredAt: Instant, override val version: Int = 1, val orderId: OrderId, val carrier: Carrier, val tracking: TrackingNumber ) : CheckoutEvent() }

Notice the discipline: every event has an aggregate id (cartId), a timestamp, and a schema version. The version field looks like overhead until you have to migrate stored events six months later, at which point you'll wonder how anyone shipped events without a version field.

Publishing events on Android: Flow and SharedFlow

You do not need Kafka, RabbitMQ, or any broker to do domain-events well on Android. In-process events delivered via SharedFlow cover 95% of mobile use cases — and the other 5% is "this event needs to leave the device," which is what your backend is for.

// :checkout:domain — the publisher interface interface DomainEventBus { val events: SharedFlow<CheckoutEvent> suspend fun publish(event: CheckoutEvent) } // :checkout:data — the implementation @Singleton class InMemoryDomainEventBus @Inject constructor() : DomainEventBus { private val _events = MutableSharedFlow<CheckoutEvent>( replay = 0, extraBufferCapacity = 64, onBufferOverflow = BufferOverflow.SUSPEND ) override val events: SharedFlow<CheckoutEvent> = _events.asSharedFlow() override suspend fun publish(event: CheckoutEvent) = _events.emit(event) } // In an aggregate / use case class PlaceOrderUseCase @Inject constructor( private val cartRepo: CartRepository, private val bus: DomainEventBus, private val clock: Clock ) { suspend operator fun invoke(cmd: PlaceOrder): Result<OrderId> { val cart = cartRepo.findById(cmd.cartId) ?: return Result.failure(...) val (order, event) = cart.place(clock.now()) cartRepo.save(cart) bus.publish(event) // emit OrderPlaced return Result.success(order.id) } }

Subscribers anywhere in the app collect from bus.events. The Statements ViewModel updates totals; the Notifications system schedules a delivery push; an Analytics observer logs to Mixpanel. None of them know about the others; all of them learn the same fact at the same moment.

Persistence and replay

Once events are immutable facts, you can persist them. Once you persist them, your event log is your audit log — and replaying them rebuilds your state from scratch. This is event sourcing in miniature, and it's a superpower for two scenarios on mobile:

  • Offline-first apps. Capture events locally as they happen, sync to the backend when online. The list of unsynced events is your queue. The backend simply replays them.
  • Bug forensics. A user complains the order looks wrong. You pull the event log for that cartId, replay it, and see exactly what happened in what order. No guessing from log lines.
// :checkout:data — Room storage for events @Entity(tableName = "checkout_events") data class CheckoutEventRow( @PrimaryKey val id: String, // ULID for chronological ordering val cartId: String, val type: String, // "OrderPlaced", "PaymentCaptured", ... val version: Int, val occurredAt: Long, // epoch ms val payload: String // JSON ) @Dao interface CheckoutEventDao { @Insert suspend fun insert(row: CheckoutEventRow) @Query("SELECT * FROM checkout_events WHERE cartId = :cartId ORDER BY id ASC") suspend fun forCart(cartId: String): List<CheckoutEventRow> } // Replay = read all rows for a cart, decode each, fold into state suspend fun rehydrate(cartId: CartId, dao: CheckoutEventDao): Cart = dao.forCart(cartId.value).map { decode(it) }.fold(Cart.empty(cartId)) { c, e -> c.apply(e) }

You don't have to go full event-sourcing — most apps shouldn't — but persisting events as a secondary structure (alongside the regular tables) gives you the audit and the replay without forcing you to rebuild every read query.

Events crossing context boundaries

Remember chapter 4's Open Host Service + Published Language? Domain events are the most natural Published Language. Each upstream context emits events on a schema; downstream contexts subscribe and translate at the boundary via an ACL.

Checkout emits CheckoutEvent.OrderPlaced. The Statements context subscribes via its own ACL and turns the event into StatementLineRecorded. The Notifications context subscribes too and turns it into OrderConfirmationScheduled. Three contexts, one published event, zero direct coupling between consumers.

Critically: each consumer translates the event into its own vocabulary. A OrderPlaced sitting in your Notifications domain code is a category error — it's vocabulary leak. The ACL converts the foreign event into a local concept the moment it crosses the boundary.

Events as the contract between aggregates

Within a single context, events are also the canonical way to keep two aggregates eventually consistent without violating the "one aggregate per transaction" rule (which we'll meet properly in chapter 8). When the Cart aggregate emits OrderPlaced, an event handler in the same context creates a fresh Order aggregate. Two aggregates, two transactions, one consistent flow.

This is a key DDD pattern: cross-aggregate consistency is always eventual, mediated by events. The temptation to load two aggregates into one transaction is exactly the temptation that produces aggregate sprawl. Events are the antidote.

Common pitfalls

The "EntityChanged" disguise

The most common anti-pattern: an event named UserUpdated with a payload of "the entire User object." That isn't a domain event — it's a CRUD update wearing a costume. Real domain events are specific: UserEmailVerified, UserSubscriptionRenewed, UserNamedChanged. They name the thing that happened, not "something changed somewhere in the user."

If your event names all look like XChanged, YUpdated, ZModified, you're doing CRUD with an extra step. The whole point of domain events is to capture the meaningful, business-named transitions — not the fact that a row got a new value. Force the name to be specific. If you can't pick a specific past-tense verb, you don't have a domain event yet; you have a database write.

Fat events

An event with 30 fields is a model leak. Events should carry the minimum payload needed for downstream consumers to act — usually an ID plus the deltas. If a consumer needs more, it queries the upstream context. Stuffing the entire Order into the OrderPlaced payload couples every consumer to every field of Order, which defeats the boundary.

Events used for everything

Not every interaction needs to be an event. If a ViewModel calls a use case and immediately uses the return value, that's just a function call. Events shine when:

  • Multiple subscribers need to react to the same fact independently.
  • The producer doesn't know (and shouldn't know) about the consumers.
  • The interaction crosses a context boundary.
  • You want an audit trail or replayability.

If none of those apply, a regular method call is clearer.

A worked example: the checkout flow

Let's trace one realistic interaction. The user taps "Pay now" in a Compose checkout screen. Here's the event chain that follows:

  1. ViewModel sends a PlaceOrder command to the use case.
  2. The use case loads the Cart aggregate, calls cart.place(), gets back an OrderPlaced event.
  3. The use case publishes OrderPlaced on the bus.
  4. A handler in the Checkout context captures payment and emits PaymentCaptured.
  5. A handler in the Notifications context schedules an order-confirmation push (via its ACL on the bus).
  6. A handler in the Statements context records the line item.
  7. The fulfillment context (downstream service) eventually emits OrderShipped.
  8. The same notifications handler picks up OrderShipped and schedules a delivery push.

Notice how thin the wiring is. Each handler subscribes to the events it cares about. Adding a new feature ("send a fraud-check job on every OrderPlaced") means adding one new subscriber. No existing code changes.

When NOT to use domain events

Events have a cost: indirection, harder debugging, eventual consistency, schema versioning. Don't pay that cost when:

  • The flow is single-actor and synchronous. A login screen calling AuthRepository.login(...) and getting a User back doesn't need an event — it's a request/response.
  • The interaction never crosses a context boundary. Inside one small context with one consumer, a method call is clearer.
  • There's no business meaning to the change. "The user opened a screen" might be useful for analytics, but it's not a domain event — it's a UX signal.
  • You need an immediate, transactional response. Don't fake synchronous flow on top of async events. If the caller blocks until the event is processed, you have a method call dressed up as architecture theater.

Tooling: in-process vs broker

For Android apps, the spectrum is:

⚡
SharedFlow — in-process, hot
The default. Subscribers in the same process get events as they happen. No persistence. Loss-tolerant if your app dies. Use for UI reactivity and intra-app integration.
📚
Room-backed event log
SharedFlow plus persistence. Events go to a Room table; subscribers can replay from any point. Use for offline-first apps, audit, and bug forensics.
🌐
Backend broker (Kafka, NATS, SQS)
When events need to leave the device. The Android side publishes via your normal API; the backend broker handles fanout to other services. From the app's perspective it's just one more downstream consumer of your local bus.

Most apps live in tier one with a touch of tier two. Tier three is rare on the device itself — you cross to the backend before reaching for a real broker.

Naming events well

Names matter more for events than almost anything else in DDD, because the events are what your PMs and customers will eventually see in dashboards, audit logs, and product specs. A few rules:

  • Past tense, always. OrderPlaced, not OrderPlace or OrderPlacement.
  • Subject + verb. What thing? Did what? InvoiceIssued, SubscriptionCancelled.
  • Specific, not generic. UserEmailVerified beats UserUpdated every time.
  • Domain words, not technical. PaymentCaptured beats StripeChargeSucceeded — the second name leaks vendor.
  • One verb per event. OrderPlacedAndPaid is two events: OrderPlaced, PaymentCaptured.

Exercise: list every event in one feature

One hour. One feature you know well. A piece of paper.

  1. Pick one feature in your app — checkout, onboarding, ride request, password reset, anything end-to-end.
  2. Walk through the feature in your head from the user's first tap to the final outcome.
  3. Write down every significant business fact that happens during the flow. Past tense, one per line. Aim for at least 8-12.
  4. For each, ask: would a PM recognize this name? If not, rename it.
  5. For each, ask: do we currently capture this fact anywhere — logs, analytics, audit table, push notification trigger? Mark "captured" or "lost forever."
  6. Look at the "lost forever" rows. These are the events you're not currently modeling. Which ones would the support team kill for, the next time a customer complains?
  7. Pick the one or two highest-value missing events and draft a sealed-class event for each. Show them to your team in the next stand-up.

You will discover, almost without exception, that your app captures CRUD updates everywhere and domain events nowhere — even though your PMs talk in domain events all day. Closing that gap is what makes downstream features (analytics, audit, replay, eventual-consistency-across-contexts) tractable.
