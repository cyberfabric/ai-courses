Chapter 4

Context Mapping

How bounded contexts talk — and the eight relationships you can label on the arrows

Last chapter you drew boxes. This chapter you label arrows.

If your map from chapter 3 has more than one bounded context, you already have a context map — you just don't have a vocabulary for what the arrows mean. By the end of this chapter you will, and you'll be able to look at any two of your contexts and answer the only question that matters: "who depends on whom, and on whose terms?"

Why bounded contexts have to talk

The dream of a "fully isolated context" doesn't survive a real product. Your Onboarding context completes a verification, and your Transactions context needs to know about it so the user can finally send money. Your Investments context places an order, and Statements needs to record it for the year-end PDF. Real apps cross context boundaries on every screen.

The trick is not to avoid the integration. The trick is to make the integration explicit, named, and one-directional whenever possible. Evans gave us the words for this; you don't have to invent them.

The vocabulary at a glance

There are eight context-relationship patterns in the Blue Book canon. Most teams never need more than four of them. Here's the lineup, ordered roughly by how often you'll reach for each on a real Android app:

ACL
Anti-Corruption Layer — your favorite weapon
C/S
Customer / Supplier — healthy default for inter-team work
OHS
Open Host Service — an API designed to be consumed

And four more you'll meet less often, but you need to be able to spot:

  • Shared Kernel — a tiny shared model between two contexts. Powerful, expensive.
  • Conformist — you accept the upstream model as-is, no translation. The cheap path.
  • Published Language — a deliberate, versioned format for communication (often paired with OHS).
  • Separate Ways — you decide not to integrate at all, and live with the duplication.
  • Big Ball of Mud — the elephant. What to do when you inherit one.

Let's take them in order.

Anti-Corruption Layer (ACL)

Anti-Corruption Layer — a translation layer at the boundary of your context that converts the foreign model into your model. The foreign side never knows it exists; your side never sees the foreign types. It is "Google Translate" between two languages.
Do not say "mapper," "adapter," or "wrapper" interchangeably with ACL. A mapper is a one-line type conversion; an ACL is a deliberate, named layer with its own vocabulary translation. The ACL contains mappers, but it is more than them — it shields your model from foreign concepts entirely.

This is the pattern you will use most. Whenever you depend on a third-party SDK, a legacy backend you can't change, or a context owned by another team that doesn't share your worldview, the answer is almost always an ACL.

The shape: foreign DTOs come in one side, clean domain types come out the other. Your ViewModel, your repository, your aggregates — nothing in your context ever sees the foreign type. If the foreign API renames user_id to userIdentifier, only the ACL changes.

A real Android example: a third-party identity SDK

You're integrating a popular KYC vendor's Android SDK. It returns this charming object:

// What the vendor SDK gives you data class VendorVerificationResult( val status: String, // "PASS", "FAIL", "REVIEW", "UNKNOWN" val flags: Map<String, Any?>, // a junk drawer val raw_score: Double?, // sometimes null, sometimes 0.0 = unverified val provider_ref: String, // their tracking id val errorCode: Int = 0, // 0 = success val timestamp_ms: Long )

Letting that type leak into your domain is how you end up with a 200-line OnboardingViewModel that branches on string status codes. The ACL converts it into something your Onboarding context can actually reason about:

// :onboarding:domain — pure, ours sealed class VerificationOutcome { data class Verified(val confidence: Confidence) : VerificationOutcome() data class Rejected(val reason: RejectionReason) : VerificationOutcome() data class NeedsReview(val flags: List<ReviewFlag>) : VerificationOutcome() data class Failed(val cause: VerificationError) : VerificationOutcome() } // :onboarding:data — the ACL lives here class VendorAcl(private val sdk: VendorSdk) { suspend fun verify(applicant: Applicant): VerificationOutcome { val raw = sdk.runCheck(applicant.toVendorRequest()) return when (raw.status) { "PASS" -> VerificationOutcome.Verified(raw.toConfidence()) "FAIL" -> VerificationOutcome.Rejected(raw.toRejectionReason()) "REVIEW" -> VerificationOutcome.NeedsReview(raw.toReviewFlags()) else -> VerificationOutcome.Failed(raw.toError()) } } }

Now your ViewModel calls VendorAcl.verify(...) and gets a sealed Kotlin type with four cases. Exhaustive when. No string codes leaking. When the vendor releases a v2 with a completely different shape, the diff is contained to one file.

The ACL is the single most useful pattern for legacy migrations. When you can't refactor a 5-year-old backend or a giant :legacy module in one go, you wrap it in an ACL and pretend it has the shape you want. The legacy code stays put; your new context lives in clean domain terms; the ACL absorbs the ugliness. This is also exactly how you incrementally adopt DDD without a rewrite (chapter 12 walks through it end-to-end).

Shared Kernel

Shared Kernel — a small, jointly-owned model fragment shared between two contexts. Both teams must agree on every change. High coupling, but useful for truly universal value objects.
Do not call your :common module a "shared kernel." A real shared kernel is <10 types, jointly owned, version-locked. :common is a junk drawer with neither shared ownership nor strict review.

A shared kernel sounds appealing — "we both need Money, why duplicate?" — and it is occasionally right. But every line of code in a shared kernel must be approved by every consumer. Change Money and you need sign-off from Billing, Transactions, and Investments. That is expensive.

Use it for genuinely universal value objects: Money, Currency, Locale, UserId (if your IDs really are global). Don't use it for domain entities or anything that has even a whiff of business logic.

// settings.gradle.kts — a thin, deliberate shared kernel include(":shared:money") // Money, Currency, Amount — ~5 types include(":shared:ids") // UserId, OrderId — type-safe IDs // :billing:domain depends on :shared:money // :transactions:domain depends on :shared:money, :shared:ids // :investments:domain depends on :shared:money // CODEOWNERS sets joint review on :shared/* // Any change requires approval from billing-team AND transactions-team AND investments-team

The CODEOWNERS rule is what makes the kernel shared. Without it you've just got :common with a fancy name.

Customer-Supplier

This is the healthiest pattern between two cooperating teams. The downstream context (Customer) tells the upstream context (Supplier) what it needs. The upstream team accepts it as a real requirement and prioritizes accordingly. There's a real backlog conversation, real sprint commitments, real SLAs.

It's not a code pattern; it's a team-relationship pattern. But it has code consequences: the supplier publishes a stable contract, the customer holds the supplier accountable, and changes are negotiated rather than imposed.

Statements (downstream / customer) needs the Transactions (upstream / supplier) team to publish a TransactionRecorded event with a specific schema, monthly aggregates, and a guarantee of replay-ability. The Transactions team adds a ticket to its backlog. The Statements team gets a stable contract. Both teams are happy.

What makes it customer-supplier rather than something looser is that the upstream team feels obligated. If you can't get that obligation — if upstream says "we'll consider it" — you don't have customer-supplier. You probably have conformist.

Conformist

Conformist is what happens when you depend on an upstream context that doesn't care about you. The platform team that ships a Maps SDK; the giant backend service owned by a different org; the OS layer itself. They don't take requirements from you. You either accept their model as-is or you don't integrate.

The conformist accepts the upstream model. No translation, no ACL, no pretense. Their User is your User. Their JSON is your JSON. Their breakage is your breakage.

Conformist is cheap, until it isn't. The day the upstream team makes a breaking change, the conformist context shatters. There's no cushion. If you're going to be a conformist, do it deliberately, document it, and price in the future re-write — or upgrade to ACL the moment the upstream model starts hurting.

Practical advice: be a conformist for genuinely stable foreign types (the JDK's Instant, AndroidX's Bitmap) and an ACL-er for everything that looks like business data. The line between them is "could this model change for business reasons?" If yes, ACL.

Open Host Service & Published Language

Open Host Service (OHS) — an API your context publishes specifically for other contexts to consume. The API is designed for consumption, not as an accident of internal implementation.

Published Language (PL) — the format the OHS speaks. Versioned, documented, treated as a real contract.
Do not call your internal Retrofit interface an "open host service." OHS is a deliberate design choice that says "we expect to be consumed by N contexts." If only one context calls it, it's just an internal API.

OHS + PL is what an upstream context offers when it knows it has many downstream consumers. On a server it's a REST or gRPC API. Inside a single Android app, it's typically a sealed event hierarchy and a small repository interface, both designed for foreign consumption from day one.

// :transactions:domain — the published language sealed class TransactionEvent { abstract val id: TransactionId abstract val occurredAt: Instant abstract val version: Int // schema version, never 0 data class Posted( override val id: TransactionId, override val occurredAt: Instant, override val version: Int = 1, val amount: Money, val direction: Direction, val accountRef: AccountRef ) : TransactionEvent() data class Reversed( override val id: TransactionId, override val occurredAt: Instant, override val version: Int = 1, val originalId: TransactionId, val reason: ReversalReason ) : TransactionEvent() } // :transactions:domain — the open host service interface TransactionFeed { fun events(since: Instant): Flow<TransactionEvent> }

Statements consumes TransactionFeed via the published language. Notifications does the same. So might a future Analytics context. Each consumer reads the events; each consumer is responsible for its own ACL inward, if needed. The upstream team has one contract to maintain instead of three.

Separate Ways

Sometimes the right answer is no integration at all. Two contexts have similar concepts but the cost of unifying them — through ACLs, shared kernels, or events — would be greater than the cost of letting both have their own. So they go their separate ways.

Separate Ways is the most-undersold pattern in the canon. It's the one that says "yes, you have two notion of Address, no, we will not unify them, the duplication is cheaper than the integration." It is a perfectly valid answer when the two contexts almost never need to talk.

Duplication is cheaper than wrong abstraction. If two contexts each have a five-field Address and they only ever exchange a fully-formatted string at one point, do not build an ACL or a shared kernel for them. Let both Address classes exist, document the choice, and move on. The team-time saved is enormous.

Big Ball of Mud

Eric Evans named this lovingly: it's the legacy system you inherited, the one with no boundaries, no language consistency, and 12 years of accumulated decisions. Most apps have at least one Big Ball of Mud somewhere in their dependency graph — a giant Retrofit backend, a 500-class :legacy module, a cursed feature that nobody dares touch.

The DDD answer is not to fix it. The DDD answer is to name it as a context ("Legacy Banking Backend") and wrap it in an ACL. You stop pretending it's clean. You document its boundary. New work happens in new contexts that consume the Big Ball of Mud through the ACL. The ball stays muddy; your new code stays clean. Over time the ball shrinks, or it doesn't — either way, your downside is bounded.

Drawing the map: arrows, U/D labels, and what they mean

A context map is two things: boxes (contexts) and arrows (relationships). The arrow direction matters. The labels matter even more. Here's the canonical notation:

Symbol Meaning Where it goes
U Upstream — this context dictates the model On the side of the dictator
D Downstream — this context accepts upstream's model (or translates) On the side of the consumer
ACL Anti-corruption layer between U and D On the downstream end of the arrow
OHS / PL Open Host Service / Published Language On the upstream end of the arrow
SK Shared Kernel between two contexts Drawn as a small overlap region between the two boxes

An arrow from Statements to Transactions, with D on the Statements side, U/OHS on the Transactions side, and ACL right next to the D, reads as: "Statements consumes Transactions' open-host event feed via an ACL." That single arrow has communicated more than a 12-page architecture doc.

A worked map: the fintech app from chapter 3

Remember the four contexts: Onboarding, Transactions, Investments, Statements. Here's how they probably relate, with labels:

+--------------+ OHS/PL +-------------------+ | Onboarding | -----------------> | Transactions | | (U) | ApplicantApproved | (D, conformist on | | | | the event shape) | +--------------+ +-------------------+ | | OHS/PL | OHS/PL | TransactionEvent v v +--------------+ +-------------------+ | Investments | | Statements | | (D, ACL) | <----- SK ------ | (D, ACL) | | | Money, | | | | UserId | | +--------------+ +-------------------+ LEGEND U = upstream D = downstream OHS/PL = published events on a stable schema ACL = anti-corruption layer on the downstream side SK = shared kernel (Money, UserId)

You can read every relationship out loud: "Onboarding publishes ApplicantApproved as an open host service; Transactions consumes it as a downstream conformist on the event shape. Transactions then publishes TransactionEvent for both Investments and Statements, which each have their own ACL on their downstream side. Investments and Statements share a tiny kernel of value objects (Money, UserId)."

That paragraph used every concept from this chapter. The diagram says it in 20 lines.

Building an ACL on Android: a complete example

Let's make this concrete. Your Investments feature integrates with a third-party brokerage API that returns wildly inconsistent JSON. The upstream context is, effectively, "the Brokerage Big Ball of Mud." You want your domain to stay clean. The recipe:

// :investments:data — the ACL package // 1) The foreign DTOs (kept private to the ACL) internal data class BrokeragePositionDto( val sym: String, val qty: String, // yes, a string val mkt_val: Map<String, Any?>, val acct: String, val ts: Long? ) // 2) The Retrofit interface (also internal) internal interface BrokerageApi { @GET("v3/positions") suspend fun positions(@Header("X-Acct") acct: String): List<BrokeragePositionDto> } // 3) The mappers (private; tested in isolation) private fun BrokeragePositionDto.toDomain(): Position = Position( symbol = TickerSymbol(sym.trim().uppercase()), quantity = Quantity(BigDecimal(qty)), marketValue = mkt_val.toMoneyOrZero(), accountRef = AccountRef(acct) ) // 4) The ACL: the only public thing in the package class BrokerageAcl @Inject constructor( private val api: BrokerageApi ) : PositionRepository { override suspend fun positionsFor(account: AccountRef): List<Position> = api.positions(account.value).map { it.toDomain() } }

The shape: nothing outside :investments:data ever sees a BrokeragePositionDto. The ViewModel asks PositionRepository for Position objects and gets them. If the brokerage API breaks, only the ACL changes. If the brokerage gets replaced, only the ACL is rewritten.

The four-step recipe is universal: foreign DTO → foreign API client → private mapper → clean domain interface. Anything else (caching, retries, rate-limiting) goes inside the ACL too — the boundary is its own little fortress.

ACL vs Shared Kernel: how to pick

Engineers love a shared kernel because "no mappers!" Engineers come to regret a shared kernel because "every change needs three teams' approval." A simple decision tree:

Shared Kernel works when
- The type is a value object (no behavior, no identity) - It is genuinely universal (Money, Currency) - < 10 types total - Joint code-ownership is in place - You expect almost zero churn
ACL is the right move when
- The other side has its own model and worldview - Foreign types have fields you don't care about - Foreign types lack invariants you require - The other team owns its evolution - You expect any change at all

If both lists describe your situation, default to ACL. It's almost always cheaper than maintaining a shared kernel that nobody has the authority to change.

When to redraw the map

Context maps are not write-once artifacts. They drift as the org changes, as features ship, and as new integrations appear. Watch for these signals that yours is stale:

  • Cross-team escalations on every release. Two teams that the map shows as separate keep blocking each other. The boundary is in the wrong place.
  • Recurring outages at one integration point. The arrow is right, but the contract is wrong — you probably need to upgrade a conformist relationship to an ACL.
  • Types ping-ponging through three ACLs. The same data crosses three context boundaries unchanged. One of those boundaries is fake; merge the contexts on either side.
  • A "shared" type that one team always changes. If Money in your shared kernel only ever gets PRs from one team, that team owns it — it should be in their context with an ACL or OHS, not in a kernel.
  • The map disagrees with the build graph. Run ./gradlew :module-graph and compare to your map. If there's an arrow on the diagram that doesn't exist in the build, or vice versa, somebody's lying. Usually it's the diagram.

A good cadence: redraw the map every quarter, or whenever a team reorganizes, or whenever a new context appears. Make it a 30-minute meeting, not a workshop. The point is alignment, not artistry.

The role of events in context mapping

You'll notice that several of the patterns in this chapter (OHS, Customer-Supplier, even some ACLs) are easier to implement when contexts communicate via events rather than direct calls. That's not a coincidence — events are the single best mechanism for one-directional, asynchronous, schema-stable integration between contexts. They make most of these patterns nearly trivial.

That's why chapter 6 (Domain Events) is a big deal even though it sits in the "tactical" half of the course. The strategic patterns from this chapter and the tactical ones from chapter 6 are the same idea seen from two different elevations.

Anti-patterns specific to mobile teams

The "shared module" that became a context. Your :shared module slowly accumulated entities, repositories, ViewModels, and now half the team commits to it on every PR. It is no longer a kernel. It is a context with no name and no owner. Pick one of: rename it, split it, or accept that it's the new "main" context and reorganize everything else around it.
Treating the network DTOs as the model. If your UserDto from Retrofit flows directly into your ViewModel, you are a conformist to your backend — whether you meant to be or not. Every backend change is now a UI change. The fix is an ACL right at the data layer: DTOs in :context:data, domain types in :context:domain, mappers between them.

Exercise: classify every cross-context arrow in your app

One hour. Pull out the map you drew at the end of chapter 3.

  1. For every arrow on your map, write down which side is upstream (U) and which is downstream (D).
  2. For each arrow, pick exactly one label: ACL, SharedKernel, Customer-Supplier, Conformist, OHS/PL, or SeparateWays.
  3. If you can't decide, you've found something: either you don't know how the contexts integrate (talk to the other team) or the integration is incoherent (it needs to be one of the patterns — pick which one and refactor toward it).
  4. Highlight every arrow labeled Conformist. For each, ask: "Should this really be an ACL? What's the cost of upgrading?"
  5. Highlight every SharedKernel. For each, count the number of types in it. Anything > 10? It's not a kernel; it's a shared context. Decide whose context it should be.
  6. Take a picture and compare to ./gradlew :module-graph. Where the diagram and the build graph disagree, one of them is wrong.

The output of this exercise is a labeled context map and a small list of "we should probably do something about this arrow" items. Pick one. That's your next refactor.
