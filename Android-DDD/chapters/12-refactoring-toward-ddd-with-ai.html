Chapter 12

Refactoring Toward DDD with AI

An end-to-end, six-step AI-assisted refactor of a legacy Android codebase — with the human still owning the strategic decisions

You have the DDD vocabulary, the Android module shape from chapter 10, and the /ubiquitous-language skill from chapter 11. Now we put them together. This chapter walks through a realistic six-step AI-assisted refactor of a legacy Android codebase toward a DDD-shaped architecture — the kind of codebase you almost certainly already have, with anemic data classes, god-ViewModels, and a :common module nobody can defend.

The point isn't to demonstrate AI virtuosity; it's to show where AI is genuinely the right tool for the job and where it absolutely is not. DDD's strategic decisions — what's a bounded context, what's the core domain, where the seams should go — are still yours. AI shines on the high-throughput, low-creativity work that fills the rest of the refactor: codebase-wide renames, extraction of value objects, generation of repository interfaces, scaffolding tests. Use it for what it's good at; keep your hands on the wheel for the rest.

The starting codebase

Picture a typical Android app you've inherited. Five or six years old. About 60K lines of Kotlin, sprinkled with Java leftovers. One :app module, one :common, one :network, and a handful of :feature-* modules with overlapping concerns. The model layer is anemic; logic is scattered across ViewModels, "use case" classes, and helpers; primitives like String and Long are everywhere.

// Where we're starting — representative pain data class User( val id: Long, val email: String, val name: String, val phone: String?, val tier: String, // "free" | "pro" | "enterprise" — stringly typed val isActive: Boolean, ) class UserManager @Inject constructor( private val api: UserApi, private val db: AppDatabase, private val analytics: Analytics, ) { // 600 lines. Validates emails. Caches users. // Decides eligibility. Writes events. Mocks // its own dependencies in tests with Mockito. } class CheckoutViewModel : ViewModel() { // 800 lines. Computes totals. Validates the // address. Decides if the user can submit. // Talks directly to UserApi, OrderApi, Room, // analytics, feature-flag service, and a // singleton SessionHolder. }

None of this is unique to your app. It is the universal Android starting point after a few years of growth. The good news: every problem in that snippet is fixable with the patterns from chapters 7–10, and a meaningful chunk of the mechanical work can be delegated to AI.

The six-step refactor (and what AI does in each)

The order matters. Steps 1–3 are exploratory and human-led; steps 4–6 are mechanical and AI-led. Doing them out of order — especially mechanizing renames before you've named the domain — is the most common failure mode for AI-assisted refactors.

1. Context map
Have Claude scan the repo and produce a candidate CONTEXT_MAP.md. Human refines.
→
2. Ubiquitous language
Run /ubiquitous-language per context. Human curates.
→
3. Aggregate boundaries
Claude proposes; human decides which invariants matter and where the consistency boundaries go.
→
4. Value objects
Codebase-wide renames: String email → EmailAddress. Claude does the grunt work.
→
5. Pull logic into :domain
Move business rules out of ViewModels and managers into aggregates and domain services.
→
6. Repositories & :data
Extract repository interfaces; move Room/Retrofit code into :data module implementations.

Step 1: Generate a context map

Open Claude Code at the repo root. Ask for a candidate context map based on the build graph and the domain types it can find. Claude is good at this because it can read all the files in parallel; you're good at this because you know which boundaries are organizational truth versus accidental.

// Prompt > Read settings.gradle.kts and every :feature-* / :*-data module. List > the candidate bounded contexts based on (a) module names, (b) the > aggregates each module owns, and (c) the domain events crossing > module edges. For each candidate context, name 3-5 core concepts. > Output as CONTEXT_MAP.md draft. Do not invent contexts; if a module > is unclear, mark it "unclear" and explain. // Output (excerpt) ## Candidate context: Onboarding - Aggregates: SignupSession, EmailVerification - Crosses to: Identity (passes a verified UserId) ## Candidate context: Identity - Aggregates: User, Profile - Cross-cutting: every other context reads UserId ## Candidate context: Checkout - Aggregates: Cart, Order, Payment (?) - Cross-cutting: emits OrderPlaced ## Unclear: :common Contains UserManager, AnalyticsHelper, NetworkUtils, DateFormatter, AvatarLoader, SessionHolder. No coherent context.

Read the output. Argue with it. Move SessionHolder from :common into Identity; argue whether Payment is its own context or part of Checkout; flag the modules that have no business being in :common. The human-in-the-loop step turns Claude's structural read into a real strategic map.

Step 2: Generate the glossary

Now invoke /ubiquitous-language from chapter 11, once per bounded context. Run it on Identity first, since that's the context every other one touches. The output is a glossary with code anchors; the curation pass is short because Claude has already done the bulk reading.

Curate before you commit. Read the glossary end-to-end before merging. Two or three terms will read like Wikipedia ("a User is a person who uses the application") — tighten them to your team's actual usage. One or two terms will be wrong (Claude included a DTO; Claude missed a value object). Fix those by hand. Commit the result. The first version doesn't have to be perfect; it has to be real.

Step 3: Identify aggregates

This is the most strategic step in the whole refactor. AI can propose aggregates by looking for invariants in the code, but the final boundary is a human call. Get Claude's read; argue with it; commit your decisions to a short doc per context.

// Prompt > In the Checkout context, propose aggregate boundaries. For each > candidate aggregate: > - name the root entity > - list invariants currently enforced anywhere in the code (search > ViewModels, use cases, "Manager" classes, validators) > - estimate aggregate size (entities + value objects) > - flag invariants currently NOT enforced anywhere — those are bugs > > Do not propose aggregates that span more than one current Gradle > module. If two modules co-own a candidate aggregate, list the split > and recommend which module the aggregate should live in. // Output (excerpt) ### Candidate aggregate: Cart - Root: Cart - Invariants enforced (in CheckoutViewModel.kt:213-260): - Cart cannot have negative quantities - Cart cannot exceed 50 line items - Coupon must match cart subtotal threshold - Invariants NOT enforced anywhere: - Cart for inactive users should fail to add items (only validated in :feature-checkout/AddItemUseCase, missed in :api/CartController) ### Candidate aggregate: Order - Root: Order - Invariants enforced: payment must be authorized before shipment - Size: Order + List<LineItem> + Address (~3 entities + 2 VOs) - Note: LineItem is a VALUE OBJECT (chapter 8); not its own aggregate

Notice the by-product: a list of invariants that aren't enforced anywhere. Those are latent bugs. The refactor is also a free audit; treat each unenforced invariant as a ticket and fix it as you move logic into the aggregate.

Step 4: Extract value objects with codebase-wide renames

This is where AI earns its keep. Replacing String email with EmailAddress across 60K lines is a soul-crushing manual edit; for Claude it's an afternoon. The trick is doing it safely, one value object at a time, with the test suite as your safety net.

// :domain:identity/EmailAddress.kt — the new value object @JvmInline value class EmailAddress private constructor(val value: String) { companion object { private val regex = Regex("^[^@\\s]+@[^@\\s]+\\.[^@\\s]+$") fun of(raw: String): Result<EmailAddress> = if (regex.matches(raw)) Result.success(EmailAddress(raw)) else Result.failure(IllegalArgumentException("invalid email: $raw")) } } // What you ask Claude to do, one rename at a time: > Replace every `email: String` parameter and field with > `email: EmailAddress` across the repo. For each call site that > passes a raw String, wrap it with EmailAddress.of(it).getOrThrow() > AT THE BOUNDARY ONLY (Retrofit DTO mappers, Room mappers, user input). > Do not wrap inside :domain; types should already be EmailAddress > once the boundary mappers are updated. Run :domain tests after > each module. Stop and ask if any test fails or if you encounter > ambiguity (e.g., a String field named "email" that holds a search > query, not a real email).

The discipline that matters: "AT THE BOUNDARY ONLY." If you let Claude wrap raw strings inside :domain, you've just moved the problem. Wrapping happens in mappers (chapter 4 ACL pattern), and from that point inward everything is typed. Same recipe for UserId, Money, OrderId, Iban, and any other primitive-disguised-as-domain-type.

The "rename without test run" disaster. The single biggest way this step goes wrong is letting Claude rename across the whole repo in one PR with no intermediate verification. The compiler catches type errors but doesn't catch semantic mistakes — e.g., a String email field that actually holds a search query. Force Claude to run tests after each module-scoped rename and stop on the first failure. The cost of an extra PR is nothing; the cost of a silent semantic bug shipping is enormous.

Step 5: Pull logic out of ViewModels and managers

Now you have value objects flowing inside :domain and crisp aggregates with named invariants. Time to move business rules out of UserManager, CheckoutViewModel, and the rest of the kitchen-sink classes from step 1's snapshot. The mechanic Claude is best at: identify a method, classify it (aggregate behavior vs domain service vs application service vs UI concern), and propose where it should move.

Before — logic in ViewModel
class CheckoutViewModel { fun submit() { val cart = cartRepo.get(cartId) if (cart.lines.isEmpty()) { _state.value = Error("empty cart") return } if (!user.isActive) { _state.value = Error("inactive user") return } val total = cart.subtotal + shippingFee if (total > user.creditLimit) { _state.value = Error("over limit") return } cartRepo.markSubmitted(cartId) analytics.log("submit", total) } }
After — rules in domain
// :domain class Cart { fun submit( user: User, fee: Money, clock: Clock, ): Result<OrderPlaced> { /* invariants */ } } // :feature-checkout class CheckoutViewModel { fun submit() = viewModelScope.launch { placeOrder(cartId).fold( onSuccess = { /* state via flow */ }, onFailure = { e -> _state.update { Error(e.message) } }, ) } }

Notice what disappeared from the ViewModel: every if that encoded a business rule. The rules now live on the Cart aggregate where they can be tested against a single object instead of mocked across five dependencies. The ViewModel's job shrank to "call use case, map result to UI state" — exactly what chapter 10 said it should be.

Step 6: Introduce repository interfaces and split :data

By this point your :domain module has aggregates, value objects, and domain services. The last structural move is to introduce repository interfaces in :domain, move every Room and Retrofit type to a new :data module that implements them, and wire them up with Hilt in :app. Claude can scaffold this almost entirely.

// Prompt > In :feature-checkout, find every direct call to OrderApi (Retrofit) > and OrderDao (Room). For each: > 1. Define an OrderRepository interface in :domain:checkout if it > doesn't already exist, with collection-shaped methods. > 2. Move the Retrofit/Room call into :data:checkout, behind a class > RoomRetrofitOrderRepository : OrderRepository. > 3. Replace the call site to use the interface. > 4. Add a @Binds in DataModule.kt. > 5. Add a FakeOrderRepository in :domain:checkout/testFixtures. > Run the test suite after each call site is migrated. Stop on red.

One PR per repository, one repository per aggregate, with the test suite as the canary. By the end of the step, :feature-* modules import only from :domain; the Hilt graph wires :data implementations into :domain interfaces; and the legacy :common module has shrunk by 80–90%. The remainder of :common is genuinely cross-cutting infrastructure (logging, theming) that you'll redistribute into :platform:* modules in a small follow-up PR.

Verifying the refactor

Tests are the safety net. Two layers matter:

  • Existing tests — the signal that nothing broke. If your test suite goes red after a step, stop, diff, and revert if you can't pinpoint the cause within 15 minutes.
  • New tests, generated by Claude — the signal that new behavior is captured. For each aggregate method moved into :domain, ask Claude to generate a test class against the public method using the test fakes (not mocks) you established in chapter 9.
    • Run tests between every Claude action, not at the end of the day. Granular test runs convert silent disasters into loud, easy-to-fix small failures. If your CI is slow, run only the affected modules locally; ./gradlew :domain:checkout:test :feature:checkout:test is fast even on big repos.

    Limits of AI-driven DDD

    Three things AI is genuinely bad at, and where you have to stay in the seat:

    🧭
    Strategic boundary calls. What is and isn't a bounded context. What is core vs supporting vs generic. AI will helpfully suggest answers; the suggestions are often plausible, sometimes wrong.
    🤝
    Talking to domain experts. The PM and the support team know things the codebase doesn't. AI can't sit in those meetings; it can only encode what you bring back.
    ⚖️
    Trade-off calls. "Should this be a separate aggregate or a value object?" depends on team size, deployment cadence, and domain stability. Those are judgment calls only humans have the context for.

    Conversely: AI is excellent at codebase-wide consistency, mechanical refactors, generating boilerplate, drafting tests, and producing documentation that's anchored on code. The split of labor is clear — you decide what should happen; Claude does the high-volume work of making it happen.

    Workflow tactics that keep the refactor safe

  • One aggregate per PR. A 50-file PR that touches three contexts is a horror to review. A 6-file PR that extracts EmailAddress across the Identity context is reviewable in 10 minutes.
  • Never let Claude touch two contexts in one PR. Mixing contexts hides cross-context coupling that a single-context PR would surface as a missing import.
  • Diff every change. Read the diff — not Claude's summary, not the test result. The diff is the truth; everything else is interpretation.
  • Commit small, often. Each commit is a checkpoint Claude can roll back to. git reset --hard HEAD~1 is your friend on bad runs.
  • Use the glossary as a guard. If Claude proposes a class named UserManager, your UBIQUITOUS_LANGUAGE.md already says "Do not say: Manager" — reject the suggestion.
  • Pause and reflect every five PRs. Are you converging on the target architecture or making lateral moves? If lateral, stop the AI and re-read your context map before continuing.

    • The Identity context: a real before/after sketch

    Inside the Identity context, the refactor moved roughly: UserManager (600 lines, 11 dependencies) became User aggregate (140 lines), UserRepository interface (4 methods), RoomRetrofitUserRepository implementation (90 lines), EmailVerifier domain service (25 lines), and SignupUseCase + SignInUseCase application services (35 lines each). Net line count: similar. Structure: night and day. Tests: 3× the count, all using fakes, all running in milliseconds.

    Common pitfalls

    Blindly applying patterns

    "Every entity needs a repository." No. Reference data, in-memory wizards, and small finite sets don't need the ceremony (chapter 9). Don't let Claude apply the pattern uniformly — review every introduction and ask "what is this hiding?" If the answer is "nothing complex," delete the abstraction.

    Refactoring without a test safety net

    If your test coverage is thin, the very first task isn't refactoring — it's writing characterization tests against the current behavior. Pin the behavior down before you change anything, then refactor under the safety net. Claude can scaffold characterization tests too; that's a perfectly valid first PR.

    Ignoring legacy constraints

    Sometimes a piece of code is shaped strangely because of a constraint you don't know about — an API that only accepts certain payloads, a deprecated SDK that breaks under coroutines, a data model the backend owns. Before refactoring, ask Claude to summarize the strange code in plain English. If your summary makes no sense, find the human who knows the constraint before changing anything.

    Exercise: refactor one feature this week

    One feature. One week. Six PRs.

    1. Day 1 — Map. Pick one feature in your app (one screen, one flow). Run step 1 (context map prompt) scoped to that feature's modules. Commit the resulting CONTEXT_MAP.md draft.
    2. Day 2 — Glossary. Run /ubiquitous-language on the feature's context. Curate. Commit.
    3. Day 3 — Aggregates. Run step 3 (aggregate proposal). Read it carefully. Decide the boundary; write a 5-line note in the context's README.
    4. Day 4 — Value objects. Pick the most-used primitive in this feature (probably an ID or a money type). Run step 4 to extract it. One PR.
    5. Day 5 — Logic move. Pick the largest method in this feature's ViewModel. Have Claude classify it and propose where it should live. Move it. One PR.
    6. Day 6 — Repository. Extract the repository interface for the feature's main aggregate. Move Room/Retrofit calls behind it. One PR.
    7. Day 7 — Reflect. Read the six diffs back-to-back. What got better? What's still ugly? Write a one-page retro and share it with the team. The retro is what makes the next refactor easier.

    You will not finish the whole codebase in a week. That's not the point. The point is to walk through the entire workflow once, end-to-end, on one feature — so the next feature is easier, and the third is muscle memory.

    Closing thoughts: DDD + AI is more than the sum

    DDD gives you the vocabulary; AI gives you the throughput. Without DDD, AI helpfully renames things in three different vocabularies and adds another layer of god-classes. Without AI, DDD takes a quarter to introduce because every refactor is a manual edit across thousands of files. Together, the discipline of DDD constrains what AI does and the speed of AI makes the discipline applicable to a real codebase on a real schedule. The team that combines both ships an architecture that is small, named, tested, and consistent — in months, not years.

    Two practices to take with you out of this chapter, if nothing else:

    One: AI is a power tool, not a colleague. It does the high-throughput work and surfaces options; the strategic calls are still yours. The split of labor isn't optional; mistaking AI suggestions for AI decisions is the most expensive way to use it.

    Two: The artifacts persist. The UBIQUITOUS_LANGUAGE.md, the CONTEXT_MAP.md, the slash commands in .claude/commands/ — these outlive any individual refactor session. The first time you run them you pay for them; for the rest of the project's life they pay you back, every time anyone (human or AI) opens the repo and asks "what does this word mean here?"
