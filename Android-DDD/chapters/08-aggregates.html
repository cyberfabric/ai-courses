Chapter 8

Aggregates

Consistency boundaries — the aggregate-root rule, transaction scope, and how to size them right

You now have entities and value objects. Useful, but on their own they don't tell you which objects belong to which transaction, who is allowed to mutate what, or where the consistency seams of your domain run. That's the job of the aggregate — arguably the most important tactical pattern in DDD, and the one most often misunderstood on Android.

The promise of this chapter: by the end, you can look at any feature in your app and answer three questions crisply. What's the aggregate? Where is its consistency boundary? What's the root? Once you can answer those, repositories, transactions, and event flows follow almost automatically.

What an aggregate actually is

Aggregate — a cluster of related entities and value objects treated as a single unit for the purpose of data changes. The aggregate has one entity designated as the aggregate root (the only entry point from the outside) and a consistency boundary (everything inside is kept consistent within a single transaction).
Do not say "model," "graph," or "tree." Those terms describe the shape of the data; "aggregate" describes the contract: who's responsible for keeping these things consistent together. Without the contract, it's just a collection of related rows.

Three things make an aggregate an aggregate:

🌳
One root. Outsiders only hold references to the root entity. Everything inside is reached through the root.
🛡️
One consistency boundary. The root protects an invariant: a rule that must always be true across the cluster.
⚛️
One transaction. Loaded together, mutated together, saved together. No half-states leak out.

An aggregate is a scope of consistency, not a "natural cluster of related stuff." That distinction is the source of every aggregate-sizing mistake in the wild. The boundary is drawn around an invariant — a rule that must always be true — and only as wide as that invariant requires.

The Order/LineItem worked example

The textbook example: an Order with multiple LineItems, with the invariant "the order total equals the sum of line-item subtotals." If you let an outside object reach inside the order and add a line item directly, the total is now wrong. The invariant is broken. So LineItem is inside the Order aggregate, and the only way to add one is through Order.addLine(...) — which knows to recompute the total.

// :checkout:domain class Order( val id: OrderId, val customerId: CustomerId, private val _lines: MutableList<LineItem> = mutableListOf(), private var _status: OrderStatus = OrderStatus.Draft, private var _total: Money = Money.ZERO_USD ) { val lines: List<LineItem> get() = _lines.toList() // defensive copy val status: OrderStatus get() = _status val total: Money get() = _total fun addLine(sku: Sku, qty: Quantity, unitPrice: Money): LineAdded { check(_status == OrderStatus.Draft) { "cannot modify a $_status order" } val line = LineItem(sku, qty, unitPrice) _lines += line _total = _lines.fold(Money.ZERO_USD) { acc, l -> acc + l.subtotal } return LineAdded(id, sku, qty) } fun place(now: Instant): OrderPlaced { check(_status == OrderStatus.Draft) { "already placed" } check(_lines.isNotEmpty()) { "empty order" } _status = OrderStatus.Placed return OrderPlaced(id, _total, now) } override fun equals(other: Any?) = other is Order && other.id == id override fun hashCode() = id.hashCode() } // LineItem is a value object — defined entirely by its attributes data class LineItem internal constructor( val sku: Sku, val qty: Quantity, val unitPrice: Money ) { init { require(qty.value > 0) { "qty must be positive" } } val subtotal: Money get() = unitPrice * qty.value }

Notice four things. Order is the root: it's the only thing outsiders touch. LineItem is a value object inside the aggregate — not its own aggregate, because nobody outside the order has a reason to reference one specific line item independently. The internal collection _lines is private and exposed only through a defensive copy. And every mutation goes through a method that re-establishes the invariant before returning.

Why LineItem isn't its own aggregate

This is the most common newbie question: "why isn't LineItem its own aggregate?" The test is the same one we used last chapter, but applied to identity:

  • Does anyone outside the order need to reference a specific line item by its own ID? No.
  • Can a line item exist without an order? No.
  • Is there an invariant that spans multiple line items (the total)? Yes — and it lives on the order.

All three answers point inward: LineItem is internal to the Order aggregate. If you discover later that you do need to reference a specific line independently — say, returns are tracked per-line and a return links to a specific one — you'd give LineItem an ID and either keep it as an internal entity (still inside the aggregate) or, if a return-line outranks an order-line for some reason, you might split. But you don't pre-emptively split for hypothetical needs.

Sizing aggregates: small is usually right

Vaughn Vernon's Effective Aggregate Design (the famous three-paper series) is the canonical guidance. The headline rules:

Vernon's rules of thumb:
  1. Model true invariants in consistency boundaries. Only put inside the aggregate what you cannot live with being inconsistent for even a moment.
  2. Design small aggregates. Most aggregates should hold one root + a few value objects. Big aggregates with collections-of-children are a smell.
  3. Reference other aggregates by ID only. An Order holds a CustomerId, not a Customer. The customer is a separate aggregate.
  4. Use eventual consistency outside the boundary. Cross-aggregate updates happen via domain events — never a single transaction.

The single most common sizing mistake is treating a graph of related rows as one aggregate. "Customer has Orders, each Order has LineItems, each LineItem has Refunds — let's make Customer the root." That aggregate spans half your schema. It locks the customer to mutate any line item. It loads megabytes of data to add one row. It cannot scale.

The smaller version: Customer is its own aggregate. Order is its own aggregate (referencing CustomerId). Refund is its own aggregate (referencing OrderId, LineItemId). Three small consistency boundaries instead of one giant one. Coupling between them happens via events, not foreign-key cascades.

Finding aggregates: follow the invariants

The technique that works: list the rules that must always be true in your domain, and group entities together only when an invariant connects them.

Try this on the back of a napkin (or in a comment in your code):

  1. Pick one feature ("checkout").
  2. List the invariants in plain English. "The order total equals the sum of line subtotals." "An order cannot ship before payment is captured." "A coupon can be used at most once per order."
  3. For each invariant, ask: which entity is responsible for enforcing this? If two entities share an invariant, they probably belong in the same aggregate.
  4. For invariants that span multiple "natural" aggregates, think about whether they truly need to be enforced atomically or whether eventual consistency would be acceptable. (Spoiler: it almost always is.)

Most "we need a giant aggregate" arguments dissolve when you pressure-test the invariant. "Customer credit limit" sounds like it has to be enforced atomically against every order — until you realize a 200-millisecond stale read is fine, and even banks tolerate a small window of over-extension to catch up later.

The transaction rule: one aggregate per transaction

The Vernon rule: one aggregate, one transaction, one commit. This is not arbitrary dogma; it's the lever that lets you scale and reason about consistency.

Two aggregates in one transaction is the usual bug. The instant you load Order and Customer into one Room transaction to mutate both, you've doubled the lock surface, prevented the two aggregates from ever living in different storage, and made the invariant "they update together or not at all" implicit in the SQL rather than explicit in the domain. The cost compounds quickly. Resist the urge.

The "mostly" caveat: Vernon allows that you may, in narrowly-scoped cases, mutate two aggregates in one transaction if and only if you have measured a real consistency requirement that cannot be served by eventual consistency. In a decade of mobile work you will hit this case maybe twice. Default to one aggregate per transaction; treat exceptions as "I have a written justification I'm willing to defend at code review."

Cross-aggregate consistency: events, always

If you can't update two aggregates in one transaction, how do you keep them in sync? Domain events — exactly the pattern from chapter 6. The flow is:

Tx 1: Mutate Aggregate A
→
A emits event
→
Bus delivers
→
Tx 2: Handler mutates Aggregate B

Concretely: in the checkout flow, Cart.checkout() emits OrderPlaced in transaction 1. A handler in the same context creates a fresh Order aggregate in transaction 2. Two aggregates, two transactions, one consistent end state — reached eventually, not atomically.

This is also the standard answer for "but what if the second transaction fails?" Persist the event log first; replay on failure; treat the handler as idempotent. None of these techniques are exotic on Android — a small Room table for unprocessed events plus a worker to drain it covers 95% of cases.

The Law of Demeter inside an aggregate

The Law of Demeter (don't talk to strangers) and the aggregate rule (talk only to the root) are the same idea wearing different hats. Outsiders never reach order.lines[0].sku. They call order.skuOfLine(0) if they need that, or more typically they don't reach inside at all because the aggregate exposes higher-level operations like order.total, order.canBePlaced(), order.addLine(...).

Bad — reaching inside
val total = order.lines .map { it.unitPrice * it.qty.value } .reduce { a, b -> a + b } order.lines.add(LineItem(...)) // 1) Recomputes the invariant outside. // 2) Mutates an internal collection. // 3) Anyone reading code has to reverse- // engineer the invariant.
Good — through the root
val total = order.total order.addLine(sku, qty, unitPrice) // Invariant lives in one place. // Internal collection stays private. // New devs can read the public API // and understand the contract.

Aggregates and Room: the persistence layer

Room is the dominant persistence library on Android, and it has opinions that quietly conflict with aggregate design if you're not careful. Two specific things to watch:

// :data — Room sees rows, not aggregates @Entity(tableName = "orders") data class OrderRow( @PrimaryKey val id: String, val customerId: String, val status: String, val totalCents: Long, val currency: String ) @Entity( tableName = "order_lines", foreignKeys = [ForeignKey( entity = OrderRow::class, parentColumns = ["id"], childColumns = ["orderId"], onDelete = ForeignKey.CASCADE )] ) data class OrderLineRow( @PrimaryKey(autoGenerate = true) val pk: Long = 0, val orderId: String, val sku: String, val qty: Int, val unitPriceCents: Long ) @Dao interface OrderDao { @Transaction @Query("SELECT * FROM orders WHERE id = :id") suspend fun loadWithLines(id: String): OrderWithLines? @Transaction suspend fun saveAggregate(order: OrderRow, lines: List<OrderLineRow>) { upsertOrder(order) deleteLines(order.id) upsertLines(lines) } }

The two key disciplines:

  • Always wrap aggregate I/O in @Transaction. Room's @Transaction is your atomicity primitive. Loading an aggregate and saving an aggregate must each happen inside one of these.
  • Don't lazy-load across the boundary. Room's @Relation annotation makes it tempting to model an Order with a @Relation to its lines and load both eagerly. That's fine. What's not fine is starting from Customer and @Relation'ing to List<Order>, which violates aggregate-by-ID and quietly couples the two.
Room's @Relation across aggregate boundaries is the silent killer. A Customer with @Relation'd List<Order> looks neat. It silently turns load one customer into load every order ever. It glues two aggregates into one transaction. It makes pagination painful. Use @Relation only for entities and value objects inside the same aggregate — never to traverse to a different aggregate.

Loading aggregates: eager, by ID, complete

An aggregate is loaded as a unit. You don't lazy-load its parts. The repository's findById returns either the full aggregate or null — never a half-hydrated stub.

// :data class RoomOrderRepository( private val dao: OrderDao ) : OrderRepository { override suspend fun findById(id: OrderId): Order? { val row = dao.loadWithLines(id.value) ?: return null return Order( id = OrderId(row.order.id), customerId = CustomerId(row.order.customerId), _lines = row.lines.map { it.toDomain() }.toMutableList(), _status = OrderStatus.valueOf(row.order.status), _total = Money(Cents(row.order.totalCents), Currency.getInstance(row.order.currency)) ) } override suspend fun save(order: Order) { dao.saveAggregate( order = order.toRow(), lines = order.lines.map { it.toRow(order.id) } ) } }

The repository here is one method: "give me the whole aggregate, or nothing." Mappers translate between Room rows and domain types. That keeps Room's relational shape from leaking into your domain entity definitions, and protects the invariant that an Order exists only as a complete object with its lines.

Aggregate context boundaries: same name, different shape

In the Checkout context, an Order aggregate has LineItems, a customerId, a total, and a status machine. In the Fulfillment context, the same conceptual order is modeled as a Shipment with a List<Parcel>, a carrier, and a tracking number. Two contexts, two aggregates, one underlying business concept — connected by events crossing between them, never by sharing a model.

This is the bounded-contexts lesson from chapter 3 made tactical: two contexts can have an aggregate about the same business concept while keeping completely different shapes. The translation between them happens in events and ACLs at the boundary, never by reaching across.

Anti-patterns to spot in code review

The "graph aggregate"

An aggregate that owns half the schema. Customer -> Orders -> LineItems -> Refunds as one giant tree. Symptoms: massive joins on every load, every mutation locks the customer row, the test setup for "place an order" requires a full customer + payment-method + address fixture.

The "anemic aggregate"

An aggregate root that is just a property bag. Order(val total, val status, val lines) with all logic in some OrderManager. By definition, the root must protect invariants — if it doesn't, it's not a root, it's a record. Push behavior onto the aggregate.

The "lazy collection"

An aggregate that returns a Flow or lazy Sequence for its internal collection, with the load happening on traversal. This breaks the "loaded as a unit" promise and makes consistency reasoning impossible. Aggregates are loaded fully or not at all.

The "two-aggregate transaction"

A use case that loads Order + Customer + Inventory in one Room transaction, mutates two of them, saves all three. Symptoms: deadlocks, slow tests, and reasoning failures when you ask "what happens if the third update fails?" Refactor to one mutation per transaction with events bridging.

Aggregates and Compose / MVVM / MVI

A common confusion: where do aggregates live in a UI architecture? The answer is the same regardless of which UI pattern you use. Aggregates live in :domain. ViewModels (in MVVM) or stores (in MVI) orchestrate them: load the aggregate via the repository, call domain methods on it, save it back, publish events. The aggregate itself never knows about StateFlow, Compose, or SavedStateHandle.

// :feature-checkout class CheckoutViewModel @Inject constructor( private val orders: OrderRepository, private val bus: DomainEventBus, private val clock: Clock, ) : ViewModel() { private val _state = MutableStateFlow<CheckoutState>(CheckoutState.Idle) val state: StateFlow<CheckoutState> = _state.asStateFlow() fun place(orderId: OrderId) = viewModelScope.launch { val order = orders.findById(orderId) ?: run { _state.value = CheckoutState.Error("not found") return@launch } runCatching { order.place(clock.now()) } .onSuccess { event -> orders.save(order) // single aggregate, single tx bus.publish(event) // OrderPlaced flows to other contexts _state.value = CheckoutState.Placed(orderId) } .onFailure { e -> _state.value = CheckoutState.Error(e.message ?: "failed") } } }

The ViewModel is thin: load, call, save, publish, project to UI state. The interesting work — checking _status, validating non-empty lines, computing the total, deciding what event to emit — happens inside the aggregate. That's the goal.

A word on aggregate IDs

Generate aggregate IDs on the client, not the server. UUIDs (or ULIDs — sortable, also unique) let the domain mint an ID at the moment of creation, which means events can carry a real ID immediately, offline-first works naturally, and the persistence layer doesn't need a round-trip to learn what to call the new thing.

// :domain @JvmInline value class OrderId(val value: String) { companion object { fun next(): OrderId = OrderId(Ulid.fast().toString()) } } // In a use case val order = Order(id = OrderId.next(), customerId = ..., ...)

Exercise: identify the aggregate root in your app

Forty-five minutes. One feature. A real codebase.

  1. Pick a feature in your app that has at least one cluster of related rows: a cart, an order, a chat thread, a workout session, a ride request, a checklist.
  2. Write down the invariants — what must always be true across the cluster? "Total = sum of subtotals." "Cannot ship before paid." "Empty session is not allowed."
  3. Identify the entity that is responsible for those invariants. That's your aggregate root.
  4. List everything else: which are value objects inside the aggregate? Which are references to other aggregates (held by ID, not embedded)?
  5. Look at the existing code. Does the root currently expose mutation methods that re-establish the invariant? Or is the logic scattered across a use case, a manager, and three ViewModels?
  6. Pick the one mutation pathway that's most scattered, and pull it onto the root. Run your tests. The scope of change should feel small — you're moving code, not rewriting it.
  7. Write down whether anywhere in the codebase loads two aggregates into a single Room transaction. If so, sketch how an event-driven version would look. Don't refactor yet; just sketch.

You're not aiming to perfectly DDD-ify the feature today. You're aiming to draw a single boundary, label it, and put one rule inside it. That's the move that compounds — the next time someone touches the feature, the boundary is visible and they don't accidentally violate it.
