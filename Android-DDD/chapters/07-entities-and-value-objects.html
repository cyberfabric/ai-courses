Chapter 7

Entities and Value Objects

Identity vs equality — and why a Kotlin data class is not automatically a value object

Welcome to tactical DDD. The previous six chapters were about where models live: contexts, maps, subdomains, events. Now we zoom into a single bounded context and look at what's inside. The two atoms of tactical DDD are entities and value objects. Get this distinction wrong and every downstream pattern — aggregates, repositories, domain services — falls apart. Get it right and the rest of the canon clicks into place.

The good news: Kotlin gives us tools for both that other languages would kill for — data class, @JvmInline value class, sealed hierarchies, immutability by default in val. The bad news: those same tools are easy to misuse. Most Android codebases ship "value objects" that aren't, and "entities" that should have been values. Let's draw the line crisply.

The central distinction: identity vs equality

One question separates an entity from a value object:

The litmus test

If you swapped this object for another with the exact same fields, would the system care? If yes — it has identity, it's an entity. If no — it's a value object.

Two $20 bills are interchangeable. You don't care which physical bill ends up in your wallet; only that it's $20. That's a value. Two users named "Alex Chen" with the same email are not interchangeable — they are two different humans, with different orders, different message histories, different two-factor secrets. That's identity. That's an entity.

This is not a coding distinction. It's a modeling distinction the domain forces on you. Money is value. Address is value. Email address is value. User is entity. Order is entity. Ride is entity.

Entity — a domain concept whose identity matters and persists across changes. Two entities are equal if and only if their IDs are equal, regardless of any other fields. An entity has a lifecycle: born, mutated through valid transitions, eventually retired.
Do not say "row," "record," or "model." A row is a database concept — entities exist in the domain whether or not they're stored. "Model" is dangerously vague (Android calls everything a model). The DDD term is entity, and using it forces you to ask "what's its identity?" — the only question that actually matters here.
Value Object — a domain concept defined solely by its attributes. Two value objects with identical fields are the same thing. Value objects are immutable, validate themselves on construction, and have no ID.
Do not say "DTO," "POJO," or "data holder." A DTO is a transport shape; a POJO is a Java relic; "data holder" implies dumb storage. A value object is a concept with rules: it validates itself, exposes domain methods, and never lets an invalid instance exist.

The Kotlin data class trap

Here is the most common refrain when a senior Android dev meets DDD: "Oh, value objects are just Kotlin data classes, right? We've been doing this all along." Almost. Specifically, three-quarters right and one-quarter dangerously wrong.

What data class gives you for free:

  • equals() and hashCode() based on all primary-constructor fields — structural equality, perfect for value objects.
  • toString() that prints all fields — great for logs.
  • componentN() for destructuring.

What data class does not give you, and what makes it dangerous as a value object by default:

  • It cannot enforce invariants in the constructor beyond what an init block does — and most teams forget the init block entirely.
  • copy() bypasses any validation you put on factory methods, because copy() calls the primary constructor directly.
  • val fields can still hold mutable references (a data class with a val items: MutableList<X> is not immutable).
  • Equality covers all primary-constructor fields, including ones you may not want compared (like a creation timestamp).
"It's a data class, so it's a value object" is the single most common DDD-on-Android lie. A data class with primitive-typed fields, no validation, a public constructor, and a leaky copy() is a structurally-equal struct — not a value object. The structural-equality part is one ingredient out of four. Get the other three or stop calling it a value object.

What a real value object looks like in Kotlin

A proper Kotlin value object satisfies four criteria:

  1. Immutable. All fields val; no mutable collections; deeply frozen.
  2. Self-validating. An invalid instance cannot exist — the constructor throws (or you expose only a factory that returns Result).
  3. Equal by attributes. Two instances with the same fields are the same.
  4. Side-effect-free behavior. Methods return new instances; they never mutate.

Here's the canonical worked example: EmailAddress as a real value object, not a String wearing a label.

// :domain — pure Kotlin, no Android @JvmInline value class EmailAddress private constructor(val raw: String) { companion object { private val EMAIL_REGEX = Regex( "^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,}$" ) operator fun invoke(input: String): EmailAddress { val normalized = input.trim().lowercase() require(normalized.isNotEmpty()) { "email is blank" } require(normalized.length <= 254) { "email too long" } require(EMAIL_REGEX.matches(normalized)) { "invalid email: $input" } return EmailAddress(normalized) } fun parse(input: String): Result<EmailAddress> = runCatching { invoke(input) } } val domain: String get() = raw.substringAfter('@') val localPart: String get() = raw.substringBefore('@') }

What's happening here:

  • @JvmInline value class — on the JVM this compiles down to the underlying String in most contexts, so you pay zero allocation overhead in hot paths. You get the type-safety of a wrapper without the boxing cost.
  • private constructor — you cannot call EmailAddress("garbage") from outside; you must go through invoke or parse, which validate.
  • Normalization in the factory — trim, lowercase. Two EmailAddress("Alex@Example.com") and EmailAddress("alex@example.com") compare equal, which is exactly the invariant the domain wants.
  • parse for total functions — when you're parsing untrusted input (a text field, an API response), you use parse and pattern-match the Result. When you have a literal you trust, the throwing invoke is fine.
  • Domain methods — domain and localPart are real domain operations, not utility functions hanging off some EmailUtils singleton.

Compare that to the typical Android data class User(val email: String), where email can be empty, malformed, untrimmed, or in mixed case — and you have to guess at every call site whether it's been validated yet. Lifting email into EmailAddress moves that question from "everywhere" to "exactly the constructor."

Inline value classes: the zero-cost wrapper

Inline value classes (@JvmInline value class) are the Kotlin feature that makes value objects practical at scale. Without them, wrapping every primitive in a domain type costs an allocation per value — rough on a list screen with 500 rows. With them, the wrapper exists at compile time and disappears at runtime in most situations. There's almost no performance excuse to keep Strings and Longs in your domain types.

Use inline value classes for any primitive that has domain meaning beyond "a number" or "a string." Some examples that should be in every Android codebase:

// IDs — never raw Long @JvmInline value class UserId(val value: Long) @JvmInline value class OrderId(val value: String) @JvmInline value class CartId(val value: String) // Money-like things @JvmInline value class Cents(val value: Long) { operator fun plus(other: Cents) = Cents(value + other.value) operator fun minus(other: Cents) = Cents(value - other.value) fun toDisplayString(currency: Currency): String = NumberFormat.getCurrencyInstance().apply { this.currency = currency }.format(value / 100.0) } // Bounded scalars @JvmInline value class Percent private constructor(val value: Int) { companion object { operator fun invoke(value: Int): Percent { require(value in 0..100) { "percent out of range: $value" } return Percent(value) } } }

The win here is not just type safety. It's that fun transferFunds(from: UserId, to: UserId, amount: Cents) is impossible to call wrong, while fun transferFunds(from: Long, to: Long, amount: Long) has been quietly silently shipping bugs since 2014.

Multi-field value objects: data class done right

For value objects with multiple fields — Money(amount, currency), Address, DateRange(start, end) — data class is the right tool, but only with discipline.

// :domain data class Money private constructor( val amount: Cents, val currency: Currency ) { init { require(amount.value >= 0) { "money cannot be negative: $amount" } } operator fun plus(other: Money): Money { require(currency == other.currency) { "cannot add $currency and ${other.currency}" } return Money(amount + other.amount, currency) } operator fun minus(other: Money): Money { require(currency == other.currency) require(amount.value >= other.amount.value) { "negative result" } return Money(amount - other.amount, currency) } companion object { operator fun invoke(amount: Cents, currency: Currency) = Money(amount, currency) val ZERO_USD = Money(Cents(0), Currency.getInstance("USD")) } }

Three things make this safe:

  1. private constructor forces all instantiation through invoke (which validates).
  2. init block validates — this also catches copy() abuse, because copy() still runs init.
  3. Operations return new instances — no mutation, no surprise side effects.
The init block is your value-object insurance policy. It runs whether the constructor was called directly, via copy(), or by a deserializer. Always put your invariants there. require reads naturally and throws IllegalArgumentException, which downstream code can catch or let propagate.

Entities: equality by ID, not by content

An entity is the opposite. Two entities with identical fields except IDs are different things. Two entities with identical IDs but different fields are the same thing at different points in time. data class for an entity is therefore wrong by default: its generated equals compares all fields, which is exactly the wrong semantics.

// :domain — an entity, NOT a data class class User( val id: UserId, val email: EmailAddress, val name: FullName, val createdAt: Instant, private var _status: UserStatus ) { val status: UserStatus get() = _status fun verifyEmail(at: Instant): UserEmailVerified { check(_status == UserStatus.PendingVerification) { "cannot verify in state $_status" } _status = UserStatus.Active return UserEmailVerified(id, at) } fun deactivate(reason: DeactivationReason): UserDeactivated { check(_status != UserStatus.Deactivated) { "already deactivated" } _status = UserStatus.Deactivated return UserDeactivated(id, reason) } // Equality by ID only — same entity at any point in its lifecycle override fun equals(other: Any?): Boolean = other is User && other.id == id override fun hashCode(): Int = id.hashCode() override fun toString(): String = "User(id=$id, email=$email, status=$_status)" }

Note four things:

  • Not a data class — we override equals and hashCode ourselves, by ID only.
  • Internal mutability is allowed — _status is a private var. The entity has a lifecycle, and pretending it doesn't is a worse trade than tightly scoping the mutability.
  • Methods return events — tying back to chapter 6, the entity emits the fact (UserEmailVerified) and the caller publishes it.
  • State transitions are guarded — check guards the legal transitions; you cannot deactivate twice or verify a deactivated user.

Mutability: a careful exception

Value objects are immutable. Period. Entities are more often mutable internally — they have a lifecycle. But the trend in modern DDD (and modern Kotlin) is toward immutable entities too: verifyEmail() returns a new User with the new status rather than mutating the existing one. Both styles are legitimate; pick one and be consistent.

Mutable entity (traditional)
class User(...) { private var _status: UserStatus = ... fun verifyEmail(at: Instant) { _status = UserStatus.Active } } // Mutates in place. Easy to write, // straightforward when you have // long-running aggregate sessions.
Immutable entity (functional)
data class User( val id: UserId, val status: UserStatus, ... ) { fun verifyEmail(at: Instant): User = copy(status = UserStatus.Active) } // Returns new instances. Plays well with // Flow, Compose, and time-travel debugging. // Override equals/hashCode for ID-only.

Both work. The immutable version maps elegantly onto Compose state and unidirectional data flow; the mutable version is closer to Evans' original Blue Book examples. We're going to lean toward immutable in the rest of this course because it composes better with the Android stack we already use.

Why primitives obscure the model: a worked refactor

Let's take a typical Android data class and lift its fields into proper value objects. This is the refactor you can do in your own codebase this afternoon.

Before — primitives everywhere
data class User( val id: Long, val email: String, val firstName: String, val lastName: String, val phone: String?, val countryCode: String, val createdAt: Long ) // id: which kind of ID? user? account? // email: validated? trimmed? normalized? // names: empty allowed? max length? // phone: format? country? // countryCode: ISO-2? ISO-3? "us" or "US"? // createdAt: epoch ms? seconds? nanos?
After — domain types
class User( val id: UserId, val email: EmailAddress, val name: FullName, val phone: PhoneNumber?, val country: CountryCode, val createdAt: Instant ) { // equals/hashCode by id, see above } // Every type knows its own rules. // Misuse caught at the type system. // Domain methods land in the right place.

Spend 30 minutes lifting one anemic record like the one on the left, and the difference is immediate: validation moves from scattered if statements to constructors, formatting moves from StringUtils to toDisplayString() on the value, and the compiler stops you from passing a countryCode where the function wanted a phone-number country prefix.

Anchoring on Android: where each lives in your modules

// :feature-checkout/build.gradle.kts dependencies { implementation(project(":domain")) // entities, value objects implementation(project(":data")) // repository impls implementation(libs.androidx.compose.ui) // No direct Room or Retrofit here } // :domain — pure Kotlin, no Android, no androidx // Contains: // - Entities (User, Order, Cart) as classes // - Value objects (EmailAddress, Money, UserId) as inline value classes // or data classes with private constructors // - Domain events (sealed classes from chapter 6) // - Repository <interface>s (full chapter 9) // // :data — Android, Room, Retrofit // Contains: // - Room entities (separate from domain entities!) // - Mappers DTO <-> domain // - Repository implementations

Notice the deliberate separation: a Room @Entity is not a domain entity. It's a persistence-layer concept that maps to a row. The domain entity is what your business logic talks about; the Room entity is what the database stores. Keep them in different modules and write a mapper between them. The mapper is your anti-corruption layer between persistence and domain.

Persistence says "User" and means a row with nullable strings, default values, and @PrimaryKey. Domain says "User" and means an entity with validated value-object fields and guarded state transitions. Same name, different shape. The mapper between them is the smallest, cleanest ACL you'll ever write — and it pays for itself the first time you change the database schema without breaking your domain.

Common Android-specific mistakes

Putting validation in TextWatcher

Validation belongs in the value object, not in the UI layer. If your EmailAddress validates itself, the TextWatcher just calls EmailAddress.parse(input) and shows an error on Result.failure. Now your validation rule is one place, used by every screen, every API endpoint, and every test — not duplicated across three forms.

Passing primitives across screens

The Bundle says you can pass Longs, so devs pass Longs — and the compiler will happily let you pass an orderId where the next screen expected a userId. Pass the inline value class instead (it serializes the same way in Parcelable) and the compiler catches the mistake.

Treating Room @Entity classes as domain

Room @Entity classes have constraints (no-arg constructors, public fields, no inheritance hierarchies, @PrimaryKey) that make them fundamentally bad domain entities. Use them as the persistence shape and map to a real domain entity in your repository implementation.

"It's just a label" syndrome

"Why wrap orderId in a class? It's just a label." Because in three months, when a junior dev passes userId to refundOrder(orderId), the compiler will silently accept it and the bug will surface in production. The cost of @JvmInline value class is one line per ID; the savings is one production incident.

When primitives are fine

Not everything needs to be wrapped. Lift to a value object when:

  • The primitive has domain rules (validation, normalization, formatting).
  • The primitive has domain operations (Money.plus, Percent.of, EmailAddress.domain).
  • The primitive is passed around enough that mix-ups are likely (IDs, especially — you have many ID types).
  • The primitive shows up in multiple function signatures, where confusion would be costly.

Stay with primitives when:

  • The value is purely scalar with no rules (a counter, an array index).
  • It's an internal implementation detail that never crosses the domain boundary.
  • You'd be the only consumer and you'd just be wrapping for the sake of wrapping.

A bigger worked example: the RidePickupRequest family

Pulling chapter 1's "ride request" thread: here's the same concept modeled with proper entities and value objects.

// Value objects @JvmInline value class RideId(val value: String) @JvmInline value class RiderId(val value: String) @JvmInline value class DriverId(val value: String) data class GeoPoint private constructor(val lat: Double, val lng: Double) { init { require(lat in -90.0..90.0) { "bad lat: $lat" } require(lng in -180.0..180.0) { "bad lng: $lng" } } companion object { operator fun invoke(lat: Double, lng: Double) = GeoPoint(lat, lng) } } data class PickupWindow private constructor(val from: Instant, val to: Instant) { init { require(from < to) { "empty window" } } companion object { operator fun invoke(from: Instant, to: Instant) = PickupWindow(from, to) } } // Entity class RidePickupRequest( val id: RideId, val rider: RiderId, val pickup: GeoPoint, val dropoff: GeoPoint, val window: PickupWindow, private var _state: RideState ) { val state: RideState get() = _state fun accept(driver: DriverId, at: Instant): DriverAccepted { check(_state is RideState.Requested) { "not requestable in $_state" } _state = RideState.Accepted(driver, at) return DriverAccepted(id, driver, at) } override fun equals(other: Any?) = other is RidePickupRequest && other.id == id override fun hashCode() = id.hashCode() }

That's the entity-versus-value distinction in concrete code: RidePickupRequest has identity (id) and a lifecycle (state transitions); GeoPoint and PickupWindow are values — defined entirely by their fields, immutable, self-validating. The RiderId and DriverId wrappers exist for one reason: to make sure you can never pass a rider where a driver was expected, no matter how tired you are during the next on-call rotation.

Naming conventions

A few quiet conventions that pay off:

  • IDs end in Id: UserId, OrderId, CartId. Not UserUid, UserKey, UserPk.
  • Entities are nouns: User, Order, Cart. Never UserModel, UserData, UserDto.
  • Value objects are named after the concept: EmailAddress, Money, PhoneNumber. Not EmailString, MoneyAmount, PhoneInfo.
  • Verbs go on the entity, not on a service: order.cancel(reason), not OrderCanceller.cancel(order, reason).

Testing entities and value objects

One huge upside: both are pure Kotlin (your :domain module has no Android dependency), so tests are JVM unit tests — instant, no Robolectric, no instrumentation.

// src/test/java/.../EmailAddressTest.kt class EmailAddressTest { @Test fun `lowercases on construction`() { assertEquals("alex@example.com", EmailAddress("Alex@Example.com").raw) } @Test fun `rejects malformed input`() { assertTrue(EmailAddress.parse("not an email").isFailure) } @Test fun `equal regardless of input casing`() { assertEquals(EmailAddress("a@b.com"), EmailAddress("A@b.com")) } } // src/test/java/.../UserTest.kt class UserTest { @Test fun `cannot verify a deactivated user`() { val user = User(...).also { it.deactivate(...) } assertFailsWith<IllegalStateException> { user.verifyEmail(Instant.now()) } } }

Tests like these run in milliseconds, never flake, and document the domain rules better than any wiki page.

Exercise: lift one field today

Thirty minutes. One field. Your real codebase.

  1. Open your largest data class (most teams have a User, Account, or Profile).
  2. Pick one primitive field with domain meaning: email, phoneNumber, countryCode, id, amountCents, anything.
  3. Create an inline value class (or data class if multi-field) for that field, with a private constructor and a validating factory.
  4. Replace the field type in the original data class.
  5. Let the compiler walk you through every call site — and at each one, decide: was this site validating the same thing manually? Probably yes. Delete that validation; the type now does it.
  6. Run the tests. Anything red? Either the validation rule was wrong, or you found a bug. Both are good outcomes.
  7. Commit the diff and pay attention to its size. You'll see your codebase shrink — scattered validations and string-handling utilities collapse into one place.

Bonus: pick one field that you thought was identity (an entity-only thing like Long userId), wrap it in UserId, and feel the relief when the compiler stops you from accidentally swapping it with orderId. That single refactor will pay for the rest of this chapter.
