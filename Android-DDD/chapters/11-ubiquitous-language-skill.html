Chapter 11

The /ubiquitous-language Skill

A custom Claude Code slash command that emits UBIQUITOUS_LANGUAGE.md — and turns AI into a teammate that speaks your domain

You have the structural picture from chapter 10: :domain, :data, :feature, :app, dependency arrows pointing inward, bounded contexts mapped to Gradle modules. Now we shift gears. The next two chapters are about how AI fits into a DDD-shaped codebase, and they go in the order I'd recommend you actually do the work: first build the language artifact (this chapter), then use AI to refactor toward DDD on top of it (chapter 12).

The reason for that order is simple. The single most useful thing you can hand Claude Code — or any AI tool — is a file that says "in this codebase, here is what every domain word means." Without it, Claude has to guess from variable names. With it, Claude reads, applies, and reinforces your team's vocabulary on every task. We're going to build that file generator in this chapter as a Claude Code skill, then commit it so every teammate gets the same behavior with one slash command.

Why a skill (and not just a doc)

You could write UBIQUITOUS_LANGUAGE.md by hand. Many teams have. They mostly stop updating it after the second PR. The doc rots, then someone deletes it during a directory cleanup, then the team has another lunch about how "we should write a glossary." We're going to skip that loop entirely by making the doc generated — runnable by anyone, on any branch, with one command. When the doc is one prompt away, it stays fresh.

Claude Code skill — a project-level slash command stored at .claude/commands/<name>.md. The filename becomes the command (ubiquitous-language.md → /ubiquitous-language). The file body is a prompt template Claude follows when you invoke the command. Skills committed to the repo are shared with every teammate automatically — one git pull and they have the same command.
Do not say "macro," "shortcut," or "agent." A skill is not a macro because it can read files and use tools; it is not a shortcut because it has its own prompt; it is not an agent because it runs in your main session, not in a separate sub-context. Call it a slash command or a skill.

Anatomy of a skill file

Every Claude Code skill is just a Markdown file. The simplest one is a single line of instructions; a useful one has frontmatter that hints at the command's intent and a prompt body that handles arguments and edge cases.

# .claude/commands/ubiquitous-language.md --- description: Generate or update UBIQUITOUS_LANGUAGE.md for this codebase argument-hint: "[bounded-context-name]" --- You are generating the ubiquitous-language glossary for this Android codebase. If an argument was passed, scope your work to the bounded context named "$ARGUMENTS". Otherwise, scan the whole repository. # Steps 1. Read CLAUDE.md, settings.gradle(.kts), and any existing UBIQUITOUS_LANGUAGE.md. 2. Identify bounded contexts from the Gradle module layout (:domain:*, :feature:*). 3. For each context, scan domain types: aggregates, value objects, repositories, events. 4. Cross-reference with PR titles, recent commit messages, and ticket prefixes for terms the team actually uses out loud. 5. Write or update UBIQUITOUS_LANGUAGE.md at the repo root, grouped by bounded context. # Output format (per term) - Term (exact casing as used in code) - Definition: one or two sentences in plain English - Code anchor: file path + class name where the term is defined - Examples: 1-2 idiomatic phrases used in PRs or chat - Do not say: 1-3 alternative names the team has used historically - Related: 2-4 related terms in the same context # Constraints - Never invent terms. If you cannot find code or chat support for a term, leave it out. - Group strictly by bounded context. Do not merge contexts. - Mark cross-context terms with the contexts they appear in. - Preserve any "Do not say" entries already in the file unless explicitly contradicted.

Three things make this work as a real skill: the frontmatter (helps Claude Code's command picker show the right tooltip), the structured steps (so the run is repeatable), and the constraints (so Claude doesn't hallucinate terms that aren't actually in the codebase). The prompt is a contract with the model, not a wish list.

Wiring the doc into Claude's context

A glossary file Claude doesn't read is a glossary file that doesn't exist. CLAUDE.md is auto-loaded into every Claude Code session; UBIQUITOUS_LANGUAGE.md is not — unless you tell CLAUDE.md to import it. The mechanism is the @-import.

# CLAUDE.md (excerpt at the top of the file) @UBIQUITOUS_LANGUAGE.md # Project conventions This is the FitTracker Android codebase. We follow DDD: pure-Kotlin :domain modules, one bounded context per Gradle module group, value objects over primitives. Read UBIQUITOUS_LANGUAGE.md (imported above) for the glossary; never invent a synonym for an existing term. When writing or refactoring code: - Match the casing and spelling in UBIQUITOUS_LANGUAGE.md exactly - Prefer domain types over primitives (Money, EmailAddress, WorkoutId) - Put repository interfaces in :domain and implementations in :data - Domain-layer code must not import android.* or androidx.*

The single line @UBIQUITOUS_LANGUAGE.md is what makes the glossary load. Claude will read it before doing any work. From that moment on, every PR, refactor, or feature implementation gets the team's vocabulary applied automatically — no per-prompt reminder needed.

Treat CLAUDE.md as the entry point and UBIQUITOUS_LANGUAGE.md as the dictionary. The two files have different rates of change. CLAUDE.md changes when conventions shift (rare); UBIQUITOUS_LANGUAGE.md changes every time the team coins a new domain term (frequent). Splitting them means you regenerate the glossary often and touch CLAUDE.md almost never.

The output format you want Claude to produce

Claude is good at structured output when you specify the structure. Here's the shape we ask for — one section per bounded context, one entry per term, with the same five-field schema for every entry.

# UBIQUITOUS_LANGUAGE.md (generated) ## Bounded Context: Workouts ### Workout **Definition:** A single training session by one Athlete, with a start time, a duration, and a list of Sets. **Code anchor:** `:domain:workouts/Workout.kt` **Examples:** - "log a workout" - "the user finished a workout at 7:42" **Do not say:** Session, Activity, Training, Log **Related:** Set, Athlete, Rpe ### Set **Definition:** A grouping of repetitions of one Exercise at one weight, recorded inside a Workout. **Code anchor:** `:domain:workouts/Set.kt` **Examples:** - "the third set of squats" - "set 4 was to failure" **Do not say:** Round, Series, Group **Related:** Exercise, Workout, Reps ### Rpe **Definition:** Rate of Perceived Exertion. A 1-10 self-reported difficulty score the Athlete attaches to a Set or Workout. **Code anchor:** `:domain:workouts/Rpe.kt` (value class wrapping Int) **Examples:** - "RPE 8 means two reps in reserve" **Do not say:** difficulty, hardness, intensity (those are different concepts) **Related:** Set, Workout ## Bounded Context: Billing ...

Notice what makes the format powerful at code-review time: every term has a code anchor. When a teammate adds UserManager, you can grep the glossary for UserManager, find no entry, and reject the PR with "this isn't in the language." When a term is in the glossary but the file path no longer exists, you've spotted language drift before the term goes extinct.

How Claude uses the file once it exists

The behavior change is real and immediate. Without UBIQUITOUS_LANGUAGE.md, prompting Claude with "add a method that finds workouts by athlete" might produce findUserSessions, getActivities, queryTrainingLogs, or any other plausible-sounding name. With the glossary loaded, the same prompt produces findWorkoutsForAthlete — matching the casing, the noun (Workout not Session), and the role (Athlete not User) the team agreed on.

Without glossary
User: "add a method to find workouts by athlete" Claude generates: fun findUserSessions(userId: Long): List<TrainingLog> Three different domain words. None match the codebase. Reviewer wastes 10 minutes renaming.
With glossary loaded
User: "add a method to find workouts by athlete" Claude generates: fun findWorkoutsForAthlete( id: AthleteId ): List<Workout> Same nouns the rest of the code uses. Value object input, aggregate output. Reviewer approves.

Multiply that by every PR Claude touches and the cumulative effect is a codebase that stays linguistically consistent — the exact thing the ubiquitous-language practice was always supposed to deliver, now without the human enforcement burden.

Sharing the skill across the team

This is where Claude Code earns its keep over a personal AI chat. The .claude/commands/ directory lives in the repo. Commit it. Now every teammate's Claude Code installation sees the same set of commands the moment they pull. No manual setup, no "did you install the skill?" Slack threads.

$ git status On branch main Untracked files: .claude/commands/ubiquitous-language.md .claude/commands/sync-ubiquitous-language.md $ git add .claude/commands/ $ git commit -m "Add /ubiquitous-language and /sync-ubiquitous-language skills" $ git push # Teammate pulls; their Claude Code instance now has both commands. $ claude > /ubiquitous-language > /sync-ubiquitous-language

Two skills, one PR, the whole team is aligned. This is the multiplier effect: a discipline that used to require quarterly glossary reviews and PR templates is now a runnable command anyone can invoke before opening a draft PR.

The companion: /sync-ubiquitous-language

The first skill generates the glossary. The companion skill reconciles it — surfaces drift, deletes stale entries, flags new domain terms that have crept in without making it into the doc. You run the first one rarely (after big features); you run this one on every meaningful PR.

# .claude/commands/sync-ubiquitous-language.md --- description: Reconcile UBIQUITOUS_LANGUAGE.md with the current codebase --- Reconcile the existing UBIQUITOUS_LANGUAGE.md with the current state of the code. Do NOT regenerate from scratch. # Steps 1. Read UBIQUITOUS_LANGUAGE.md. 2. For each term, verify the code anchor still exists and the type is unchanged. 3. Diff the current :domain:* modules against the glossary. Report: - Stale entries (anchor file removed or class renamed) - New domain types not yet in the glossary - Terms that contradict the "Do not say" list 4. Print a summary diff WITHOUT writing the file. 5. After the user approves the diff, write the updated UBIQUITOUS_LANGUAGE.md. # Constraints - Preserve all existing "Do not say" entries unless the user explicitly removes them. - Never silently delete a term; ask first. - Print the section being changed before each modification.

Notice the discipline this skill encodes: print before write. The glossary is too important to silently overwrite, so the skill diffs first. The user approves; only then does the file change. This is the kind of guardrail that turns "AI editing my docs" from a horror movie into a useful tool.

A demonstration: running it on a sample app

Let's walk through what an actual run looks like on a small fintech app with three contexts. Imagine your codebase has :domain:onboarding, :domain:transactions, and :domain:statements, plus the corresponding :data and :feature modules.

1. You invoke the skill
/ubiquitous-language in the terminal at the repo root. No argument — we want the full glossary across all contexts.
→
2. Claude reads the structure
settings.gradle.kts, CLAUDE.md, all :domain:* sources, recent git log messages.
→
3. Claude proposes terms
Per context: aggregates (Account, Transaction, Statement), value objects (Iban, Money, StatementPeriod), events (TransactionPosted).
→
4. Claude writes the file
UBIQUITOUS_LANGUAGE.md at the repo root, grouped by context. Cross-context terms (Money) are flagged.
→
5. You review and commit
Read the file. Tighten any definitions that read like Wikipedia. Commit it. Add the @UBIQUITOUS_LANGUAGE.md import to CLAUDE.md.

From this point forward, every Claude Code task in this repo — bug fixes, new features, refactors — speaks the right vocabulary by default. You did not pay for that with discipline; you paid for it with one prompt and one commit.

What goes inside the prompt body (and what doesn't)

A skill prompt is a balance: too generic and the output is mush; too prescriptive and the skill doesn't generalize across PRs. Two checklists:

Belongs in the prompt
- The exact output format (per-term schema) - File paths to read first (CLAUDE.md, gradle files) - Bounded-context grouping rule - Anti-hallucination constraint ("never invent terms") - Preservation rule for "Do not say" - Argument handling ($ARGUMENTS)
Does not belong
- A specific list of terms (those go in the glossary, not the skill) - Codebase-specific module paths hard-coded - Reviewer names, ticket prefixes, etc. - "Be helpful," "be concise" filler - Examples of every possible term

The prompt is the process; the glossary is the output. Don't conflate them. A good skill prompt is mostly verbs and constraints; the nouns come from the codebase Claude is reading.

Common mistakes

Skill prompts that are too generic

"Generate a glossary for this project" produces a glossary that could belong to any project. The prompt has to specify format, grouping, and constraints. Read your skill back to yourself and ask: "could I delete the project name and would this prompt still tell Claude exactly what to do?" If the answer is "yes, it works for any project," it's too generic. Add the constraints that make this codebase's glossary look unlike any other.

Not anchoring on real code

If the prompt says "list the important terms," Claude will helpfully invent them. Add the explicit instruction "never invent terms; if you cannot find code or chat support for a term, leave it out". Then make sure the steps include reading the code: :domain:* sources, sealed-class hierarchies, value classes, sealed interfaces, repository interfaces. Anchoring on real code is the single biggest difference between a hallucinated glossary and a useful one.

Not committing the output

The output is the point. Some teams generate UBIQUITOUS_LANGUAGE.md and forget to commit it because they're treating it as a throwaway artifact. The skill is the generator; the file is the persistent contract Claude reads on every subsequent task. Commit both. Add a CI check if you want to be loud: "if :domain:* changed but UBIQUITOUS_LANGUAGE.md didn't, fail the build with a hint to run /sync-ubiquitous-language".

The "rename without updating" trap. A teammate refactors Workout to Session in code. Claude Code happily renames every file because it knows nothing about the glossary's "Do not say: Session" line. Six months later the glossary still says "Workout" but every class is Session. The rename should have been a glossary update first, code update second — or rejected at PR review. The companion /sync-ubiquitous-language skill catches this drift, but only if you run it. A pre-merge CI hook that runs it automatically is a reasonable hard gate.

Beyond glossaries: the same pattern, applied elsewhere

Once you have one skill, you have a template. Variations on the pattern that pay off the same way:

  • /context-map — emits a CONTEXT_MAP.md with bounded contexts and the relationships between them (chapter 4 vocabulary made executable).
  • /aggregate-audit — lists every aggregate, its root, its invariants, and its size in lines of code; flags aggregates over a threshold.
  • /anti-corruption-layers — reports every :data mapper, the DTO it translates from, and the aggregate it builds; surfaces missing or thin ACLs.
  • /event-catalog — emits the list of every domain event in the codebase, with publishers, subscribers, and the contexts they cross.

    • Each is a few-dozen-line skill that produces a fresh, never-stale doc on demand. None require a separate tool, separate hosting, or a separate update process. Commit the skill, the doc is one prompt away forever.

    Iterating on the skill prompt

    The first version of any skill prompt is wrong in small ways. Claude will pick up a slightly off pattern (lower-casing your acronyms, including class names that aren't really domain terms, ignoring the "do not say" rule on the first pass) and you'll need two or three rounds of tightening to lock the behavior in. Here is the practical loop:

    1. Run on a small slice
    Start with one bounded context: /ubiquitous-language workouts. A 6-term output is fast to read; a 60-term output is easy to lose track of.
    →
    2. Spot the misfires
    "It included WorkoutDao as a domain term." "It missed Rpe." "Its definitions sound like Wikipedia."
    →
    3. Tighten the prompt
    Add explicit rules: "Exclude *Dao, *Dto, *Row, and *Mapper. They are persistence and ACL classes, not domain terms."
    →
    4. Re-run and diff
    Run /ubiquitous-language workouts again. Diff the new output against the previous one. Stop when the diff is empty for two consecutive runs on the same code.

    One trick to accelerate the loop: keep a small EXAMPLES.md next to the skill with two or three "this is what a great glossary entry looks like" examples, then have the skill prompt reference it. Few-shot in a file beats few-shot in the prompt body because the file is editable without touching the skill.

    Version your skill in the commit message, not in the file. The skill itself is just a Markdown file — let git history tell the story. Each tightening round is a commit: "skill: exclude Dao/Dto from glossary", "skill: require code anchor for every term", "skill: preserve do-not-say across runs". After a month, git log .claude/commands/ubiquitous-language.md reads like a changelog of how your team learned to talk to Claude.

    The Workouts context: a complete glossary entry

    Inside the Workouts context, the team agreed: a Workout is one Athlete's training session; Set is a grouping of reps of one Exercise; Rpe is the 1-10 perceived-effort score; Athlete is the user role inside Workouts (deliberately not "user," because "user" is a Billing-context concept). Cross-context: Money appears in Billing too, but the same value class is used in both — that's the (rare) shared kernel call.

    That paragraph is what a teammate would write the first time they joined the project — and exactly what the skill would emit on first run, modulo formatting. The skill writes the glossary so the team doesn't have to; the team curates it. The split of labor is what makes the practice survive past month one.

    Exercise: write your /ubiquitous-language skill today

    Ninety minutes. Real codebase. One commit.

    1. From your repo root, create .claude/commands/ubiquitous-language.md using the template earlier in this chapter as your starting point.
    2. Tighten the prompt to your codebase: name your bounded contexts (or your module groups, if contexts aren't named yet), reference the file paths a teammate would search first.
    3. Open Claude Code at the repo root. Run /ubiquitous-language. Read the resulting UBIQUITOUS_LANGUAGE.md end-to-end. Expect to fix two or three definitions; that's the curation pass.
    4. Add @UBIQUITOUS_LANGUAGE.md as the first non-comment line in your CLAUDE.md.
    5. Commit both: the skill (.claude/commands/ubiquitous-language.md) and the generated glossary (UBIQUITOUS_LANGUAGE.md) and the CLAUDE.md change — one PR, three files.
    6. Open a tiny follow-up Claude Code task in the same session: ask it to add a method, write a test, or refactor a class. Notice the vocabulary it uses. Notice what it doesn't rename. The behavior change is the proof.
    7. Optionally, also create .claude/commands/sync-ubiquitous-language.md using the second template. Commit it; teammates get it for free on next pull.

    The first run is the largest investment; subsequent runs are cheap. The compound effect is a codebase that stays in its own language as it grows — the exact thing the DDD canon promised, now mechanized.
