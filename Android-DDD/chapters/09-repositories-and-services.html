Chapter 9

Repositories and Domain Services

Repositories as in-memory collections — and where stateless domain logic actually belongs

You can find aggregates and protect their consistency. Now you need two more building blocks to make the model run end-to-end: repositories — the place where aggregates come from and go to — and domain services — the place where logic lives when it doesn't belong on a single entity.

Both terms are wildly overloaded in the Android world. "Repository" usually means "a thin wrapper around a Retrofit API and a Room DAO with caching glued in between." "Service" usually means "anything that ends in Service." Neither matches what Evans and Vernon meant, and the gap is one of the bigger reasons "we tried Clean Architecture" projects often feel like ceremony without payoff. Let's draw the lines crisply.

What a repository really is

Repository — a collection-like interface that gives you aggregates by their identity, accepts new aggregates, and persists changes to existing ones. From the domain's perspective it is an in-memory Set<Aggregate>. The persistence mechanism — Room, Retrofit, in-memory, KMP SQLDelight — is hidden behind the interface and replaceable.
Do not say "DAO," "data source," or "store." DAOs are per-table and per-row; data sources are persistence-layer concepts; "store" is vague. A repository is per-aggregate and exposes a domain-shaped contract: findById, save, remove — not selectByXAndY.

Three things define a repository:

1️⃣
One per aggregate type. OrderRepository, not BillingRepository. The repository is the gateway to one consistency boundary.
📚
Collection-shaped. Tiny interface: findById, save, remove, maybe add. Not 30 query methods.
🌫️
Hides persistence. Domain code never knows whether the data came from Room, Retrofit, RAM, or a CSV.

The minimal repository interface

A repository is mostly four methods. That's it. If yours has 17, it's a DAO with a fancy name.

// :domain — the contract, no Android interface OrderRepository { suspend fun findById(id: OrderId): Order? suspend fun save(order: Order) suspend fun remove(id: OrderId) fun observe(id: OrderId): Flow<Order?> // optional, for reactive UIs }

That's a complete repository for an aggregate. Notice what's not there:

  • No getOrdersByCustomerId(customerId, status, dateRange, limit, offset). That's a query, not a collection operation.
  • No DTOs. save(order: Order) takes the domain aggregate; the implementation maps to whatever the storage layer wants.
  • No exceptions for "not found." findById returns Order? — nullable is the canonical "absent" signal in Kotlin.
  • No partial entities. The repository returns whole aggregates or nothing — the chapter 8 contract.

The Room-backed implementation

// :data — the implementation @Singleton class RoomOrderRepository @Inject constructor( private val dao: OrderDao, private val ioDispatcher: CoroutineDispatcher, ) : OrderRepository { override suspend fun findById(id: OrderId): Order? = withContext(ioDispatcher) { val row = dao.loadWithLines(id.value) ?: return@withContext null row.toDomain() // mapper from Room shape to aggregate } override suspend fun save(order: Order) = withContext(ioDispatcher) { dao.saveAggregate( order = order.toRow(), lines = order.lines.map { it.toRow(order.id) } ) } override suspend fun remove(id: OrderId) = withContext(ioDispatcher) { dao.deleteOrder(id.value) } override fun observe(id: OrderId): Flow<Order?> = dao.observeWithLines(id.value) .map { row -> row?.toDomain() } .flowOn(ioDispatcher) }

The DAO and the mappers stay in :data. The interface lives in :domain. :feature-checkout depends on :domain and gets the implementation injected by Hilt at the application graph. The domain code never imports Room types — not even RoomDatabase, not even @Entity.

The dependency direction is the whole point. :domain never depends on :data. :data depends on :domain to satisfy the interface. The arrow points inward. Reverse the arrow and you've made :domain a sub-module of your persistence layer — which is the architectural sin Clean Architecture and Hexagonal both exist to prevent.

Why repositories return aggregates (and not DTOs)

This is one of those small rules that pays compounding dividends. The repository's return type is the aggregate. Not a Room @Entity, not a Retrofit DTO, not a "presentation model," not a partial. The reasons:

  1. Invariants travel with the aggregate. If you return a DTO, the caller gets a property bag without methods, without validation, without state-transition guards. The whole point of the aggregate (chapter 8) is wasted at the boundary.
  2. Type safety up the stack. When findById returns Order, every caller gets a value-object-rich, validated object. When it returns OrderDto, every caller has to remember to validate and convert.
  3. One mapping, in one place. The mapper from row to aggregate lives in the repository implementation. Not in the use case, not in the ViewModel, not in three places.

Beyond the four methods: the specification pattern

So what about real queries? "All open orders for this customer placed in the last 30 days." That doesn't fit a four-method interface, but you also don't want a 17-method one. The DDD answer is the specification pattern: a small object that encapsulates a query predicate, which the repository can interpret.

// :domain — the specification, plain Kotlin sealed interface OrderSpec { data class ByCustomer(val customerId: CustomerId) : OrderSpec data class WithStatus(val status: OrderStatus) : OrderSpec data class PlacedAfter(val instant: Instant) : OrderSpec data class And(val left: OrderSpec, val right: OrderSpec) : OrderSpec infix fun and(other: OrderSpec): OrderSpec = And(this, other) } interface OrderRepository { suspend fun findById(id: OrderId): Order? suspend fun find(spec: OrderSpec): List<Order> suspend fun save(order: Order) suspend fun remove(id: OrderId) } // Caller usage in a ViewModel / use case val openRecent = repo.find( OrderSpec.ByCustomer(customerId) and OrderSpec.WithStatus(OrderStatus.Open) and OrderSpec.PlacedAfter(now.minus(30, DAYS)) )

The specification keeps the repository interface small (one find method) while letting callers express any query. The implementation translates the spec tree into a Room query (or a Retrofit call, or a series of in-memory filters — doesn't matter to the caller).

You don't always need this. If your app has two queries for orders, just put them on the interface as named methods (findOpenForCustomer, findRecent). Reach for specifications when the combinatorial growth would otherwise force a 30-method interface.

Anti-pattern: the repository-as-DAO

Bad — DAO with extra steps
interface OrderRepository { suspend fun getById(id: Long): OrderDto suspend fun getOrdersByStatus(s: String): List<OrderDto> suspend fun getOrdersByDateRange(...): List<OrderDto> suspend fun getOrdersByCustomerAndStatus(...): List<OrderDto> suspend fun updateStatus(id: Long, s: String) suspend fun updateTotal(id: Long, total: Long) suspend fun deleteByCustomerId(c: Long) // ... 11 more methods } // Returns DTOs, takes primitives, // per-column updates, per-shape queries. // This is just SQL with method calls.
Good — collection-shaped
interface OrderRepository { suspend fun findById(id: OrderId): Order? suspend fun find(spec: OrderSpec): List<Order> suspend fun save(order: Order) suspend fun remove(id: OrderId) } // Returns aggregates, takes value objects, // whole-aggregate writes, queries via spec. // Domain language, hidden persistence.

If your repository looks like the left, you don't have a repository — you have a thinly-wrapped DAO. There's nothing wrong with a DAO; just don't pretend it's something it isn't, and don't push it into :domain where it leaks SQL semantics into your business logic.

Domain services: where stateless logic lives

Domain Service — a stateless operation that belongs to the domain but doesn't fit cleanly on a single entity or value object. It expresses domain logic, lives in :domain, and uses domain types as inputs and outputs.
Do not say "manager," "helper," or "util." Those names are vague enough to absorb anything; UserManager ends up doing twelve unrelated things. A domain service has a precise responsibility expressed in domain language — PriceCalculator, RouteFinder, FraudCheck.

The classic test: "this operation needs to happen, but I can't find a single entity it naturally belongs to." If you find yourself adding a method to Order that needs to consult three other aggregates, take it off Order. Make a domain service that takes the inputs explicitly.

// :domain — a domain service class ShippingFeeCalculator( private val rates: ShippingRateTable, private val zones: ShippingZoneCatalog, ) { fun feeFor(order: Order, address: Address): Money { val zone = zones.zoneOf(address) val rate = rates.rateFor(zone, order.totalWeight) return rate.applyTo(order.subtotal) } } // Used from a use case class CheckoutUseCase( private val orders: OrderRepository, private val fees: ShippingFeeCalculator, private val bus: DomainEventBus, private val clock: Clock, ) { suspend operator fun invoke(cmd: PlaceOrder): Result<OrderId> { val order = orders.findById(cmd.orderId) ?: return Result.failure(NoSuchElementException()) val fee = fees.feeFor(order, cmd.shipTo) val event = order.place(fee, clock.now()) orders.save(order) bus.publish(event) return Result.success(order.id) } }

Things to notice. The service is stateless — ShippingFeeCalculator has dependencies, but no fields that change between calls. It takes domain types (an Order, an Address) and returns a domain type (Money). The use case orchestrates the calculator and the repository — but the use case is not the same thing as the domain service.

Three kinds of "service" — tell them apart

Senior teams trip on this distinction more than any other in tactical DDD. There are three different things often called "service" in Android codebases, and they live in different layers.

KindWhat it doesWhere it lives
Domain service Stateless domain logic that doesn't fit on one entity. Uses domain types only. :domain
Application service (a.k.a. use case, interactor) Orchestrates aggregates, repositories, domain services, and events to fulfill one user intention. :domain or :feature-*
Infrastructure service Talks to the outside world: HTTP, push, file system, sensors. Implementation of a domain interface. :data / :platform

Concretely: ShippingFeeCalculator is a domain service. CheckoutUseCase is an application service. RetrofitOrdersApi is an infrastructure service. They have different shapes, different test strategies, and different rates of change. Mixing them is the fastest way to make a "Clean Architecture" project feel anything but clean.

Anti-pattern: the god-service

The UserManager / PaymentService / OrderHelper bloat. Once a class has a name like SomethingManager, every loosely-related method ends up there: validation, formatting, fetching, caching, logging, sometimes a side effect for analytics. After 18 months it has 90 methods, 11 dependencies, and exactly one test that uses Mockito to mock six of them. The fix is two-step: rename it (PriceCalculator, UserSession, OrderPlacer) and let the new name push out everything that doesn't belong. If the rename is hard, that's the smell — the class never had a real responsibility.

Where the interfaces live: :domain vs :data

The cleanest module split for repositories and services on a multi-module Android app:

// settings.gradle.kts (excerpt) include( ":domain", // pure Kotlin, no Android ":data", // Room, Retrofit, mappers ":feature-checkout", ":feature-orders", ":app", ) // :domain/src/main/kotlin/.../OrderRepository.kt — the interface interface OrderRepository { ... } // :data/src/main/kotlin/.../RoomOrderRepository.kt — the implementation class RoomOrderRepository(...) : OrderRepository { ... } // :data/src/main/kotlin/.../di/DataModule.kt — Hilt binding @Module @InstallIn(SingletonComponent::class) abstract class DataModule { @Binds abstract fun bindOrderRepository( impl: RoomOrderRepository ): OrderRepository } // :feature-checkout depends on :domain, NOT on :data. // At runtime, the Hilt graph wires the data binding into // the feature module's dependencies.

This split lets you do something powerful: swap implementations. A KMP variant could implement OrderRepository with SQLDelight; a test build could use an in-memory fake. Neither requires changing one line of :domain or :feature-*.

Testing: fakes, not mocks

Vernon and most modern DDD writers prefer hand-written fakes over mocking frameworks. The reason: a fake captures the contract of the repository in code, while a mock script captures only the specific calls one test makes. Fakes outlive tests; mock scripts rot the moment the test changes.

// :domain/src/testFixtures/.../FakeOrderRepository.kt class FakeOrderRepository : OrderRepository { private val store = mutableMapOf<OrderId, Order>() private val flows = mutableMapOf<OrderId, MutableStateFlow<Order?>>() override suspend fun findById(id: OrderId): Order? = store[id] override suspend fun save(order: Order) { store[order.id] = order flows[order.id]?.value = order } override suspend fun remove(id: OrderId) { store.remove(id) flows[id]?.value = null } override fun observe(id: OrderId): Flow<Order?> = flows.getOrPut(id) { MutableStateFlow(store[id]) } }

One file, 20 lines, used by every test in every feature module. The contract is explicit: save upserts, remove deletes, observe emits the latest. Testing a use case becomes: create the fake, prime it with an aggregate, run the use case, assert on the fake's state and on emitted events. No verify(mock).save(eq(captured)) ceremony.

When you don't need a repository

Not every aggregate-shaped concept needs the full repository ceremony. Skip it when:

  • The data is a small, finite set loaded once at startup (a settings store, a list of supported countries). A simple SettingsHolder with a StateFlow is fine.
  • The aggregate is per-session and lives in memory only (an in-progress wizard, a temporary form). Pass it through the use case directly.
  • The data is read-only reference data shipped with the app (an icon catalog, a static rates table). Wrap it in a typed accessor; calling it a repository is overkill.

The point of the repository abstraction is to hide complex persistence behind a domain-shaped contract. If the persistence is trivial, the abstraction has nothing to hide.

The Cart bounded context: a complete picture

Inside the Checkout context: Cart is the aggregate, CartRepository is the contract for getting and saving carts, PriceCalculator is a domain service that computes totals across coupons and shipping rules, and PlaceOrderUseCase is the application service that orchestrates them. Each has a single, narrow responsibility. The repository hides Room; the calculator has no I/O; the use case is the only thing that touches the bus.

That layout — aggregate + repository + a couple of domain services + a handful of use cases — is the entire tactical DDD pattern for one bounded context. Replicate it per context, draw the seams between contexts with events (chapter 6), and you have a real, working DDD-shaped Android app.

Common Android-specific traps

The "RepositoryImpl" reflex

Many Android codebases name implementations FooRepositoryImpl and put them next to the interface in :domain. That defeats the dependency-direction win — :domain now contains a class that imports Room. Move the implementation to :data; rename it RoomFooRepository or RetrofitFooRepository so the persistence is in the name; keep the interface in :domain alone.

The "Result wrapping" reflex

Some teams wrap every repository return in Result<T> "to handle errors." Most of the time the error is "not found," which is just null. Result is for operations that can fail in domain-meaningful ways — "cannot place an empty order" — not for routing exceptions across the layer boundary. If your repository can throw IOException, let it; catch it where the user actually sees the error.

The "useCase per method" reflex

One use-case-per-method makes thin classes that just call a repository. FindOrderUseCase, SaveOrderUseCase, DeleteOrderUseCase — if the use case is just a forwarding stub, the ViewModel can call the repository directly. Use cases earn their keep when they orchestrate more than one moving piece (load aggregate, call domain service, save, publish event).

Exercise: rewrite one DAO as a repository

One hour. One DAO. Real codebase.

  1. Open your largest @Dao-annotated interface. Count its methods.
  2. Identify which aggregate (or candidate aggregate — if you skipped chapter 8's exercise, do it now) the DAO serves.
  3. Create a new FooRepository interface in :domain (or your equivalent pure-Kotlin module). Start with four methods: findById, save, remove, and one observe or query method specific to your needs.
  4. Create RoomFooRepository in :data implementing the interface. The DAO becomes its private dependency. Mappers DTO ↔ aggregate live here too.
  5. Walk to one caller of the old DAO and switch it to the new interface. The compiler will help; expect the call site to shrink.
  6. Write a FakeFooRepository in your test fixtures, 15-20 lines.
  7. Move one piece of "this method needs three aggregates and a calculation" logic out of a ViewModel into a domain service in :domain. Inject it into the use case (or directly into the ViewModel for now).
  8. Run the test suite. Anything red? Either the original logic was wrong, or your refactor moved a behavior — both are good outcomes to discover.

You don't have to convert every DAO in one sitting. The win is having one example in your codebase that the team can point to: "this is what a repository looks like; this is what a domain service looks like; let's match this when we add new ones."
