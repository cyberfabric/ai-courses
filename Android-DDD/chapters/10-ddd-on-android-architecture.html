Chapter 10

DDD on Android Architecture

Mapping DDD onto Gradle modules, Clean Architecture, MVI/MVVM, Room, Retrofit, and coroutines

You now have the full DDD vocabulary: bounded contexts, ubiquitous language, subdomains, events, entities, value objects, aggregates, repositories, and domain services. The question this chapter answers is the practical one your team will ask the moment you open a PR: "Where does each of these things actually live in our IDE?"

The good news: a modern multi-module Android app already gives you most of the slots you need. The bad news: every team I've seen freelances the slot names and the dependency directions, and small naming inconsistencies metastasize into architectural confusion within a year. This chapter draws crisp lines — module by module, layer by layer — so the DDD vocabulary lands in real Gradle code instead of a slide deck.

The big-picture module layout

Every DDD-shaped Android codebase I've worked on has converged on roughly the same module shape. The names vary, the spirit doesn't.

// settings.gradle.kts — the canonical layout rootProject.name = "fittracker" include( // Pure Kotlin/JVM — no Android, no Room, no Retrofit ":domain:workouts", ":domain:profile", ":domain:billing", ":domain:shared", // tiny shared kernel (use sparingly) // Android-aware persistence and network — one per context ":data:workouts", ":data:profile", ":data:billing", // UI features, one per user-facing flow ":feature:workout-log", ":feature:workout-detail", ":feature:profile-edit", ":feature:billing-portal", // App shell — only thing that knows about the others ":app", )

Three layers, four module families, one direction of dependency. :feature depends on :domain and gets :data wired in by Hilt at the :app graph. Nothing in :domain depends on anything Android. That last constraint is the load-bearing one; everything else flows from it.

Why :domain must be Android-free

Pure-Kotlin domain module — a Gradle module that compiles with the org.jetbrains.kotlin.jvm plugin, depends on zero Android artifacts, and contains your aggregates, value objects, repository interfaces, domain services, and domain events. It runs in milliseconds inside a JVM unit test, ships as part of a KMP target without modification, and never imports a single class whose package starts with android. or androidx..
Do not say "core module" or "common module." "Core" already means "core domain" (chapter 5), and "common" is the magnet that turns into :everything. Call it :domain, or split it per bounded context: :domain:workouts, :domain:billing.

Why this matters in concrete terms:

⚡
Tests run in milliseconds. No Robolectric, no Android SDK boot, no androidTest ceremony. Pure JUnit + Kotlin = a 2,000-test suite in under 10 seconds.
🔁
Reusable across surfaces. Wear OS, Auto, Compose for Desktop, KMP iOS, a JVM CLI for ops — all consume the same domain module unchanged.
🧱
The compiler enforces the rule. If you accidentally import android.util.Log in :domain, the build breaks. No reviewer needed.

The mechanical guarantee comes from the Gradle plugin choice. Make :domain:* use org.jetbrains.kotlin.jvm (or multiplatform for KMP). Don't apply com.android.library. The Android types are then literally not on the classpath; the compiler will refuse to compile a domain class that mentions Context, SharedPreferences, or LiveData.

One bounded context, one Gradle module

Chapter 3 introduced bounded contexts as the strategic boundary. Gradle modules are how you make that boundary structural — not just a comment in a design doc. The cleanest mapping is one module per context per layer:

// Bounded context "Workouts": // :domain:workouts — aggregates, repos, services // :data:workouts — Room + Retrofit implementations // :feature:workout-log — UI for logging a workout // :feature:workout-detail — UI for viewing one workout // Dependency rules (Gradle build files): :feature:workout-log -> :domain:workouts :feature:workout-detail -> :domain:workouts :data:workouts -> :domain:workouts :domain:workouts -> (nothing app-specific) // What the Workouts context CANNOT do: :domain:workouts -X-> :domain:billing :data:workouts -X-> :data:billing :feature:workout-log -X-> :feature:billing-portal

If a feature genuinely needs to talk to another context, it does it through a domain event (chapter 6) or an explicit ACL (chapter 4) — never by directly importing types from another context's module. The build-graph enforces what the team agreement asks for.

DDD inside Clean Architecture (without the ceremony)

Clean Architecture and Hexagonal both arrived at the same insight Evans did: business logic in the middle, I/O at the edges, dependencies pointing inward. The DDD vocabulary slots in cleanly:

Clean Architecture layerWhat lives there (DDD terms)Module
Entities / Domain Aggregates, entities, value objects, domain events, domain services, repository interfaces :domain:*
Use cases / Application Application services orchestrating one user intention :domain:* or :feature:*
Interface adapters ViewModels, mappers, presentation models, repository implementations :feature:* + :data:*
Frameworks & drivers Room, Retrofit, OkHttp, Hilt, Compose, sensors :data:* and :app

You don't need a separate :usecase module or a :interactor module. Use cases are small and they belong with the aggregates they orchestrate. Putting them in their own module multiplies module count without adding architectural value.

DDD with MVI: state machines over a domain

MVI works beautifully with DDD because both think in terms of state transitions. The trick: the domain models the business state machine; the presentation models the UI state machine. They are not the same thing, and confusing them is one of the silent killers of Android architecture.

// :domain:checkout — the BUSINESS state machine sealed interface Cart { val id: CartId data class Empty(override val id: CartId) : Cart data class Active( override val id: CartId, val lines: List<LineItem>, val coupon: Coupon? = null, ) : Cart data class Submitted( override val id: CartId, val orderId: OrderId, ) : Cart } // :feature:checkout — the UI state machine sealed interface CheckoutUiState { data object Loading : CheckoutUiState data class Showing( val cart: Cart.Active, val totals: Money, val canSubmit: Boolean, ) : CheckoutUiState data class Submitted(val orderId: OrderId) : CheckoutUiState data class Error(val message: String) : CheckoutUiState }

The Cart sealed hierarchy is domain truth: a cart is either empty, active, or submitted. The CheckoutUiState hierarchy is what the screen renders right now: it includes loading/error states the domain doesn't care about, plus derived values (totals, canSubmit) that are computed at the presentation layer. The ViewModel holds a StateFlow<CheckoutUiState> and updates it in response to user intents.

DDD with MVVM: ViewModels are application services, not domain

The "ViewModel as god object" trap. The most common DDD-on-Android failure mode is letting the ViewModel become the domain. It accumulates business rules ("validate the email," "compute the discount," "decide whether to allow checkout"), then accumulates orchestration ("call the repo, then the analytics, then update state"), then accumulates UI logic ("if in edit mode show field X"). Each layer's concerns are reasonable; concentrated in one class, they make the ViewModel un-testable, un-reusable on a tablet, and un-portable to KMP. The fix: ViewModels orchestrate, they do not enforce. Push every business rule down into aggregates, value objects, and domain services in :domain. The ViewModel calls them.

The minimum-viable ViewModel after a DDD pass looks like this:

// :feature:checkout @HiltViewModel class CheckoutViewModel @Inject constructor( private val placeOrder: PlaceOrderUseCase, private val carts: CartRepository, private val pricing: PriceCalculator, private val savedState: SavedStateHandle, ) : ViewModel() { private val cartId: CartId = savedState.toRoute<CheckoutRoute>().cartId private val _state = MutableStateFlow<CheckoutUiState>(CheckoutUiState.Loading) val state: StateFlow<CheckoutUiState> = _state.asStateFlow() init { viewModelScope.launch { carts.observe(cartId).collect { cart -> _state.value = when (cart) { null, is Cart.Empty -> CheckoutUiState.Error("Cart not found") is Cart.Active -> CheckoutUiState.Showing( cart = cart, totals = pricing.totalsFor(cart), canSubmit = cart.lines.isNotEmpty(), ) is Cart.Submitted -> CheckoutUiState.Submitted(cart.orderId) } } } } fun submit() = viewModelScope.launch { placeOrder(cartId).fold( onSuccess = { /* state will arrive via the flow */ }, onFailure = { e -> _state.update { CheckoutUiState.Error(e.message ?: "Failed") } }, ) } }

Notice what's not in this ViewModel: business rules, validation, totals math, event publishing. Those all live in PlaceOrderUseCase, PriceCalculator, and the Cart aggregate. The ViewModel's job is to map domain state into UI state and forward intents into the application service. That's it.

Coroutines and Flow at the domain boundary

A common question: should :domain use Flow? It's a Kotlin-X library, not Android, so technically yes — and most teams pick this trade-off because the alternative (callbacks, RxJava holdouts, hand-rolled observers) is worse. But there's a discipline:

  • Repository interfaces may return Flow<T>. An aggregate's lifecycle is naturally observable; modeling it as a flow lets the UI stay reactive.
  • Domain services should return values, not flows. A PriceCalculator takes a cart and returns a total — no I/O, no time, no flow.
  • Aggregates should expose plain methods. order.place(fee, now) returns a domain event. It does not return a flow.
  • Avoid kotlinx-coroutines-android in :domain. That artifact pulls in the main dispatcher, which is Android. Use kotlinx-coroutines-core only.
  • Threading is an infrastructure concern. :domain code never calls withContext(Dispatchers.IO); the repository implementation in :data does.

    • Room as a persistence backend for aggregates

    Room is excellent at one-aggregate-at-a-time persistence. Where teams stumble is mapping the aggregate's internal structure (a root entity with nested value objects and a collection of child entities) to Room's relational shape.

    // :data:workouts — Room rows, NOT domain types @Entity(tableName = "workouts") data class WorkoutRow( @PrimaryKey val id: String, val athleteId: String, val startedAtEpochMs: Long, val durationSec: Int, val rpe: Int, // value object, flattened ) @Entity( tableName = "workout_sets", foreignKeys = [ForeignKey( entity = WorkoutRow::class, parentColumns = ["id"], childColumns = ["workoutId"], onDelete = ForeignKey.CASCADE )], indices = [Index("workoutId")] ) data class SetRow( @PrimaryKey val id: String, val workoutId: String, val exerciseId: String, val reps: Int, val weightGrams: Long, ) data class WorkoutWithSets( @Embedded val workout: WorkoutRow, @Relation(parentColumn = "id", entityColumn = "workoutId") val sets: List<SetRow>, ) @Dao interface WorkoutDao { @Transaction @Query("SELECT * FROM workouts WHERE id = :id") suspend fun loadWithSets(id: String): WorkoutWithSets? @Transaction suspend fun saveAggregate(workout: WorkoutRow, sets: List<SetRow>) { upsertWorkout(workout) deleteSetsFor(workout.id) insertSets(sets) } @Upsert suspend fun upsertWorkout(row: WorkoutRow) @Query("DELETE FROM workout_sets WHERE workoutId = :id") suspend fun deleteSetsFor(id: String) @Insert suspend fun insertSets(rows: List<SetRow>) }

    Three rules to absorb. One: Room rows are not aggregates. They are storage shape; the mapper in :data rebuilds the aggregate. Two: aggregate writes are transactional — @Transaction on saveAggregate ensures the workout and its sets land atomically. Three: never expose a Flow<WorkoutWithSets> to :domain — that would leak Room types upward. The repository's observe returns Flow<Workout?>; the mapping happens in the implementation.

    Lazy collections are the footgun. Room offers @Relation, which loads child rows when you ask for the parent. That sounds great until you realize you've loaded an aggregate without its full state — and then a domain method accesses the empty list and quietly does the wrong thing. Always load the whole aggregate or load nothing. If the aggregate is too big to load whole, it's too big — revisit chapter 8 on aggregate sizing.

    Retrofit and DTOs: the API as a separate context

    The backend API is its own bounded context. Its types are shaped for HTTP, JSON-friendly, and frequently uglier than your domain's types deserve. The cure is the ACL pattern from chapter 4: a mapper layer in :data that translates DTOs to aggregates and back.

    // :data:workouts — the API DTO (matches the backend contract) @Serializable data class WorkoutDto( val id: String, val athlete_id: String, val started_at: String, // ISO-8601, because backend val duration_sec: Int, val rpe: Int, val sets: List<SetDto>, ) // :data:workouts — the mapper (the ACL) internal fun WorkoutDto.toDomain(): Workout = Workout( id = WorkoutId(id), athleteId = AthleteId(athlete_id), startedAt = Instant.parse(started_at), duration = Duration.ofSeconds(duration_sec.toLong()), rpe = Rpe(rpe), sets = sets.map { it.toDomain() }, ) // :data:workouts — the Retrofit interface stays inside :data interface WorkoutsApi { @GET("athletes/{id}/workouts") suspend fun list(@Path("id") athleteId: String): List<WorkoutDto> }

    The WorkoutsApi, the DTOs, and the mappers all live in :data:workouts. :domain:workouts sees only Workout, WorkoutId, AthleteId, Rpe. The day the backend renames started_at to begin_ts, exactly one file changes — the mapper — and nothing else in the codebase notices.

    KMP: where DDD really pays off

    In a Kotlin Multiplatform setup, the :domain module becomes commonMain — aggregates, value objects, repository interfaces, domain services, all shared between Android and iOS. The platform-specific bits (Room, SQLDelight, Retrofit, Ktor) live in androidMain and iosMain as actual implementations of the same interfaces. If you've kept :domain Android-free from day one, the KMP migration is a build-script change, not a codebase rewrite. If you haven't, the migration is brutal — every android.util.Log in your domain is a paper cut.

    You don't need to be planning an iOS port to benefit. The same discipline that makes :domain KMP-ready also makes it Wear-OS-ready, Auto-ready, desktop-ready, and command-line-ready. Optionality is the win.

    Enforcing the rules: the build graph and detekt

    ":domain never imports Android" is a rule that holds for exactly as long as your reviewers are paying attention. Lock it in mechanically:

    // :domain:workouts/build.gradle.kts plugins { id("org.jetbrains.kotlin.jvm") // NOT com.android.library } dependencies { implementation(libs.kotlinx.coroutines.core) implementation(libs.kotlinx.datetime) // No androidx.* on this module's classpath. testImplementation(libs.junit) testImplementation(libs.kotlin.test) } // detekt rule (top-level detekt.yml) ForbiddenImport: active: true imports: - 'android.**' - 'androidx.**' - 'com.google.android.**' reason: ":domain modules must not depend on Android" // Optional: a Konsist test that fails CI if rule is violated @Test fun `domain modules do not import android`() { Konsist.scopeFromModule("domain") .files .assertFalse { it.imports.any { i -> i.name.startsWith("android.") || i.name.startsWith("androidx.") } } }

    Three lines of defense: the Gradle plugin choice (compiler-enforced), the detekt rule (lint-time-enforced), and a Konsist or ArchUnit test (CI-enforced). Belt, suspenders, parachute. Skip whichever feel excessive for your team, but at least one should be active.

    Anti-patterns specific to Android

    Domain logic in Activities or Fragments

    Validation in an onClickListener. A "save" button that decides whether to call the API based on the state of three other UI fields. A TextWatcher that triggers a domain rule. Every one of these is domain logic in the wrong layer. Move it down: into a value object's constructor, into an aggregate method, into a domain service. The Activity then becomes a glass shell over a state flow — which is what it always should have been.

    Business rules in RecyclerView.Adapter / Compose item composables

    An adapter computes a discounted price. A Compose item shows a "premium" badge using a complicated conditional. The view layer has just absorbed business knowledge. The fix: enrich the domain model so the answer is already pre-computed (line.discountedPrice, user.tier), or compute it in the ViewModel and pass it through the UI state.

    The :common magnet module

    A module called :common or :core that everything depends on. Inevitably it grows analytics, network helpers, string utilities, themes, navigation interfaces, base ViewModels, and a handful of business rules. It becomes the project's largest module and the slowest one to compile. Replace it with intent-shaped tiny modules: :platform:logging, :platform:network, :designsystem, :domain:shared (only if absolutely necessary). When in doubt, duplicate — small duplication is cheaper than wrong coupling.

    The complete dependency picture

    // Allowed dependency arrows (read top to bottom): // // :app // | // +---------------+---------------+ // | | | // :feature:workout-log | :feature:billing-portal // | | | // +---> :domain:workouts | // | | // +---> :domain:billing // | // :data:workouts <----+ // | | // :data:billing <-----+ // // :app builds the Hilt graph, binding interfaces // in :domain:* to implementations in :data:*. // :feature:* never directly depends on :data:*. // :domain:workouts never depends on :domain:billing.

    Two arrows you'll be tempted to add — resist them. (1) :feature depending directly on :data: feels convenient, but it leaks Room and Retrofit upward and breaks every test that doesn't have a real database. (2) :domain:workouts depending on :domain:billing: looks innocent, but cross-context coupling at the domain layer is exactly what bounded contexts exist to prevent. If you genuinely need shared types, put them in :domain:shared and treat that module as a tiny shared kernel (chapter 4) under the strictest review.

    Exercise: sketch your module graph

    Forty-five minutes. A whiteboard or a Mermaid diagram. Your real codebase.

    1. Open settings.gradle.kts (or settings.gradle). List every include(...) entry.
    2. For each module, label it: :domain, :data, :feature, :platform, :app, or "unclear". Don't lie to yourself — "unclear" is fine and it's the most useful label you'll write today.
    3. For each :domain-labeled module, open its build.gradle.kts. Does it apply com.android.library or kotlin-jvm? If com.android.library, that module is structurally not a pure-Kotlin domain — flag it.
    4. For each module, open the dependencies { } block. Draw an arrow from this module to each module it depends on. Include the standard Android/AndroidX libraries only if they're notable (you don't need to draw arrows to androidx.core).
    5. Look for arrows that shouldn't exist: :domain → Android, :feature → :data, cross-context :domain → :domain. Highlight each in red.
    6. Look for missing arrows: places where two modules should be talking but instead share state via a global singleton or a static manager. Highlight in orange.
    7. Pick the single worst red arrow. Write a one-paragraph plan to fix it: which dependency moves, which interface gets extracted, which file gets renamed. That paragraph is your next refactor PR.

    You don't need to fix everything at once. The goal is to see the architecture: most teams have never drawn this graph and are quietly surprised by what they find. The graph is the truth; the design doc is the aspiration.

    What "DDD-shaped" means in your IDE

    A DDD-shaped Android codebase is one where: each bounded context has its own :domain, :data, and :feature modules; :domain is pure Kotlin and depends on nothing Android; aggregates and value objects live in :domain; repositories are interfaces in :domain with implementations in :data; ViewModels are thin orchestrators in :feature; the build graph itself enforces the boundaries that the team agreed on. Tests run fast, contexts stay separate, KMP is one config away, and refactors stay local.
