Chapter 5

Natural Language, Speech, and Sound

Give Your App Ears and a Brain for Words

In the last chapter, we taught your app to see. Now we teach it to read, listen, and understand. Apple ships three frameworks that handle language and audio entirely on-device: Natural Language (text analysis), Speech (voice transcription with the new SpeechAnalyzer), and SoundAnalysis (identifying 300+ sounds). No cloud APIs. No API keys. No per-request billing. Just frameworks that ship with iOS.

Think of it this way: Vision gives your app eyes. These three frameworks give it ears and a reading brain. Together, they turn your iPhone into a multi-sensory AI device.

NLTagger: Your Text Intelligence Engine

The NLTagger class is the workhorse of Apple's Natural Language framework. Feed it any text, and it tags every word with rich linguistic information — named entities (people, places, organizations), parts of speech, sentiment, lemmas, and more. It runs in microseconds, not milliseconds. It's essentially free.

Named Entity Recognition

Named entity recognition (NER) finds the "important stuff" in text — names of people, places, organizations, and dates. Think of it as a highlighter that automatically marks the actionable parts of any sentence.

import NaturalLanguage

let text = "Tim Cook announced at WWDC in San Jose that iOS 26 ships in September."
let tagger = NLTagger(tagSchemes: [.nameType])
tagger.string = text

tagger.enumerateTags(in: text.startIndex..<text.endIndex,
                     unit: .word, scheme: .nameType,
                     options: [.omitWhitespace, .omitPunctuation, .joinNames]) {
    tag, range in
    if let tag = tag, tag != .otherWord {
        print("\(text[range]) — \(tag.rawValue)")
        // "Tim Cook" — PersonalName
        // "WWDC" — OrganizationName
        // "San Jose" — PlaceName
    }
    return true
}

That's it. No model files to download. No configuration. NLTagger uses Apple's built-in language models that ship with iOS.

Sentiment Analysis

Sentiment analysis scores text from -1.0 (very negative) to +1.0 (very positive). It's perfect for scanning user reviews, chat messages, or feedback in real time.

let sentimentTagger = NLTagger(tagSchemes: [.sentimentScore])
sentimentTagger.string = "This app is absolutely amazing!"

let (sentiment, _) = sentimentTagger.tag(at: sentimentTagger.string!.startIndex,
                                          unit: .paragraph,
                                          scheme: .sentimentScore)
let score = Double(sentiment?.rawValue ?? "0") ?? 0
// score ≈ 0.9 (very positive)
Tip: Sentiment works best on paragraph-sized chunks. Single words like "great" may not score as high as you'd expect because the model considers context. Feed it complete sentences for reliable results.

Language Detection

NLLanguageRecognizer automatically identifies the language of any text. It supports 60+ languages and works with as little as a single sentence. Useful for routing text to language-specific models or displaying the right UI.

let recognizer = NLLanguageRecognizer()
recognizer.processString("Bonjour, comment allez-vous?")
let language = recognizer.dominantLanguage  // .french

// Get probability distribution for mixed-language text
let hypotheses = recognizer.languageHypotheses(withMaximum: 3)
// [.french: 0.95, .italian: 0.03, .spanish: 0.02]

NLEmbedding: Semantic Similarity Without a Server

Here's where NLP gets really powerful. NLEmbedding turns words and sentences into 512-dimensional vectors (arrays of numbers). Words with similar meanings end up close together in this vector space. "Happy" and "joyful" are neighbors; "happy" and "refrigerator" are far apart.

Why should you care? Because this enables semantic search — finding things by meaning, not just exact text match. All running on-device, all private.

if let embedding = NLEmbedding.wordEmbedding(for: .english) {
    // Find distance between two words (0.0 = identical, 2.0 = maximally different)
    let distance = embedding.distance(between: "happy", and: "joyful")
    // distance ≈ 0.42 (very close)

    // Find the 5 nearest neighbors to a word
    embedding.enumerateNeighbors(for: "swift", maximumCount: 5) { neighbor, dist in
        print("\(neighbor): \(dist)")
        // "fast", "rapid", "quick", "speedy", "nimble"
        return true
    }
}
Tip: Sentence embeddings (NLEmbedding.sentenceEmbedding(for:)) are even more useful. They let you compare the meaning of entire sentences — perfect for finding duplicate support tickets, matching FAQ questions, or building a "related items" feature.

SpeechAnalyzer: The New Era of Speech-to-Text (iOS 26)

If you've ever used SFSpeechRecognizer, you know the pain — clunky delegate callbacks, limited offline support, and accuracy that left you wanting more. iOS 26 replaces all of that with SpeechAnalyzer, a completely redesigned speech API built on Swift concurrency.

SpeechAnalyzer is modular. You attach modules that each do one job:

  • SpeechTranscriber — converts speech to text (real-time or from files)
  • SpeechDetector — detects when someone is speaking (voice activity detection)

Think of SpeechAnalyzer as the conductor and the modules as instruments. The conductor manages audio input and coordinates the modules. Each module does its specialized work independently.

Transcribing a File

The simplest use case — transcribe an audio file sitting on disk:

import Speech

func transcribeFile(at url: URL) async throws -> String {
    let transcriber = SpeechTranscriber(
        locale: .current,
        preset: .offlineTranscription
    )
    let analyzer = SpeechAnalyzer(modules: [transcriber])

    var fullText = ""
    try await analyzer.analyzeSequence(from: url)

    for try await result in transcriber.results {
        if result.isFinal {
            fullText += result.bestTranscription.formattedString + " "
        }
    }
    return fullText.trimmingCharacters(in: .whitespaces)
}

That's the entire file transcription pipeline. No delegates, no callbacks, no audio session gymnastics. Just async/await.

Live Microphone Transcription

For real-time speech-to-text, you feed live audio into SpeechAnalyzer using an AsyncStream:

import Speech
import AVFoundation

let transcriber = SpeechTranscriber(
    locale: .current,
    reportingOptions: [.volatileResults]
)
let analyzer = SpeechAnalyzer(modules: [transcriber])

// Get the audio format the analyzer wants
let format = try analyzer.bestAvailableAudioFormat()

// Create an AsyncStream to feed audio buffers
let (stream, continuation) = AsyncStream<AVAudioPCMBuffer>.makeStream()

// Start the analysis pipeline
Task {
    try await analyzer.analyzeSequence(stream)
}

// Read results as they arrive
Task {
    for try await result in transcriber.results {
        if result.isVolatile {
            // Partial result — update UI in real time
            updateLiveText(result.bestTranscription.formattedString)
        } else if result.isFinal {
            // Final, stable result
            appendFinalText(result.bestTranscription.formattedString)
        }
    }
}
Tip: .volatileResults gives you partial transcriptions that update as the user speaks — perfect for showing a "live typing" effect. Final results arrive once the model is confident in the transcription. Use volatile for real-time UI, final for storing the transcript.
Language models are downloaded on demand. The first time a user runs SpeechTranscriber for a specific locale, iOS downloads the language model in the background. This is automatic and system-managed — you don't need to handle it — but be aware that first-use may require a network connection. Subsequent uses are fully offline.

SoundAnalysis: Identifying 300+ Sounds

Your iPhone doesn't just hear words — it can identify sounds. Dog barking, siren, applause, music genre, coughing, glass breaking, doorbell. Apple's SoundAnalysis framework ships with a built-in classifier that recognizes 300+ sound categories, all on-device.

The practical applications are everywhere: accessibility features for deaf users, baby monitoring, security alerts, health monitoring (persistent cough detection), smart home automation, and ambient scene detection.

The Sound Classification Pipeline

import SoundAnalysis
import AVFoundation

class SoundClassifier {
    private let audioEngine = AVAudioEngine()
    private var analyzer: SNAudioStreamAnalyzer?

    func startListening() throws {
        let inputNode = audioEngine.inputNode
        let format = inputNode.outputFormat(forBus: 0)

        analyzer = SNAudioStreamAnalyzer(format: format)

        // Use Apple's built-in sound classifier
        let request = try SNClassifySoundRequest(classifierIdentifier: .version1)
        request.windowDuration = CMTime(seconds: 1.5, preferredTimescale: 600)
        request.overlapFactor = 0.5

        try analyzer?.add(request, withObserver: self)

        inputNode.installTap(onBus: 0, bufferSize: 8192, format: format) {
            [weak self] buffer, time in
            self?.analyzer?.analyze(buffer, atAudioFramePosition: time.sampleTime)
        }

        try audioEngine.start()
    }
}

extension SoundClassifier: SNResultsObserving {
    func request(_ request: SNRequest, didProduce result: SNResult) {
        guard let classificationResult = result as? SNClassificationResult,
              let topResult = classificationResult.classifications.first,
              topResult.confidence > 0.6 else { return }
        print("Detected: \(topResult.identifier) (\(topResult.confidence))")
        // "Detected: dog_bark (0.87)"
    }
}
Tip: The windowDuration controls how much audio the classifier analyzes at once. Shorter windows (0.5s) react faster but may be less accurate. Longer windows (3s) are more accurate but introduce delay. 1.5 seconds is a good default for most use cases.

Using Custom Sound Models

Apple's built-in classifier covers 300+ general sounds, but you can train your own for specific use cases. In Chapter 9 (Create ML), we'll train custom sound classifiers. For now, know that the pattern is identical — just swap the classifier identifier for your Core ML model:

// Built-in Apple classifier
let builtIn = try SNClassifySoundRequest(classifierIdentifier: .version1)

// Your custom Core ML model
let customModel = try MLModel(contentsOf: myModelURL)
let custom = try SNClassifySoundRequest(mlModel: customModel)

Combining All Three: The Voice-to-Intelligence Pipeline

The real power comes from combining these frameworks. An app that simultaneously transcribes speech (SpeechTranscriber), extracts entities and sentiment (NLTagger), and classifies ambient sounds (SoundAnalysis) — all four streams running in parallel, all on-device, all in real time.

The pattern: SpeechTranscriber produces text → NLTagger analyzes it for entities and sentiment → SoundAnalysis adds ambient context. Three frameworks, one unified intelligence layer, zero cloud dependency. You'll build exactly this in the exercise below.

Multilingual Considerations

iOS runs on devices worldwide, and Apple's NLP frameworks reflect that. But not all features are equal across all languages.

Well-Supported (50+ languages)

  • Language detection
  • Tokenization
  • Lemmatization
  • Part-of-speech tagging
  • Script identification

Limited Language Support

  • Named entity recognition (7 languages)
  • Sentiment analysis (7 languages)
  • Word embeddings (7 languages)
  • Sentence embeddings (English only)
  • SpeechTranscriber (varies by locale)
Best practice: Always check language support at runtime with NLTagger.availableTagSchemes(for: .word, language: detectedLanguage). If your desired scheme (e.g., .nameType) isn't in the list, fall back to basic tokenization. This prevents silent failures where the tagger returns no results.

Performance: Surprisingly Fast

One of the best things about these frameworks is their performance. NLTagger operates in microseconds. Even SpeechTranscriber delivers results faster than you can speak.

Performance Benchmarks (iPhone 16 Pro, A18 Pro)

Operation Latency Memory Battery Impact
NLTagger (NER, 100 words) ~0.2 ms ~2 MB Negligible
NLTagger (sentiment, paragraph) ~0.1 ms ~2 MB Negligible
NLEmbedding (word lookup) ~0.5 ms ~80 MB model Negligible
NLLanguageRecognizer ~0.05 ms ~1 MB Negligible
SpeechTranscriber (real-time) ~200 ms first result ~100 MB Moderate
SpeechTranscriber (file, 1 min) ~5-8 s ~100 MB Low (burst)
SoundAnalysis (continuous) ~30 ms per window ~15 MB Low-Moderate

Notice the pattern: NLTagger is so fast it's essentially free — you can run it on every keystroke, every clipboard change, every message without any user-visible impact. SpeechTranscriber and SoundAnalysis are heavier but designed for sustained real-time use with acceptable battery drain.

Memory matters for Speech. SpeechTranscriber loads a language model (~100 MB) into memory. If your app is already memory-heavy (e.g., running a Core ML vision model), be mindful of total memory usage. Unload the speech model when not in use.

Permissions and Privacy

Audio-related frameworks require user permission. Here's your Info.plist checklist:

Required Info.plist Keys

  • NSMicrophoneUsageDescription — required for SpeechAnalyzer and SoundAnalysis (any microphone access)
  • NSSpeechRecognitionUsageDescription — required specifically for SpeechTranscriber/SpeechAnalyzer

Not Required

  • NLTagger, NLEmbedding, NLLanguageRecognizer — no permissions needed (text analysis, no hardware access)
Privacy win: Since all three frameworks run entirely on-device, you can truthfully tell users "Your voice and text never leave your iPhone." Highlight this in your App Store description and privacy nutrition labels.

SpeechAnalyzer vs SFSpeechRecognizer

If you have existing code using SFSpeechRecognizer, here's why and when to migrate:

SpeechAnalyzer (iOS 26+)

  • Native async/await
  • Modular architecture (Transcriber + Detector)
  • Better accuracy (matches mid-tier Whisper)
  • Optimized for long-form audio
  • System-managed language models
  • Built for offline-first

SFSpeechRecognizer (iOS 10+)

  • Delegate-based callbacks
  • Monolithic API
  • Lower accuracy offline
  • 60-second limit for live audio
  • Manual language model management
  • Cloud-first, offline optional
Migration advice: If your minimum deployment target is iOS 26, use SpeechAnalyzer for all new speech features. If you need to support older iOS versions, wrap both APIs behind a Swift protocol and select at runtime based on #available(iOS 26, *).

Exercise: Build an AI Listener

Hands-On: The AI Listener

Build a SwiftUI app that runs three AI streams simultaneously:

  1. Speech transcription — Use SpeechTranscriber to convert microphone input to live text. Display volatile (partial) results in gray and final results in black.
  2. Entity extraction — Feed each final transcript segment to NLTagger with .nameType. Display extracted people, places, and organizations as colored tags below the transcript.
  3. Sound classification — Run SoundAnalysis in parallel. Display the top detected ambient sound category and confidence score in a badge at the top of the screen.

Architecture Tips:

  • Use an @Observable class (AIListener) with @Published properties for transcript, entities, sentiment, and ambientSound
  • Run SpeechAnalyzer and SoundAnalysis on separate Tasks — they operate on the same microphone input but process independently
  • Use NLLanguageRecognizer to detect the transcript's language before running NER — skip entity extraction if the language isn't supported
  • Add a "recording" toggle that starts/stops both audio pipelines

Stretch Goals:

  • Color transcript text by sentiment: green (positive), red (negative), gray (neutral)
  • Log all results with timestamps for session review
