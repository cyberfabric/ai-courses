The Golden Rule: Start with Apple's Built-In APIs

Always check if Vision, NaturalLanguage, SoundAnalysis, or Foundation Models can solve your problem first. These frameworks are free, maintained by Apple, optimized for every device, and require zero model management. Only reach for custom models or MLX when the built-in APIs can't handle your specific domain.

Building a Reusable Inference Wrapper

Every on-device AI feature follows the same pattern: load model, preprocess input, run inference, postprocess output, handle errors, measure performance. Let's build a reusable protocol.

import CoreML
import os

protocol OnDeviceMLFeature {
    associatedtype Input
    associatedtype Output

    var modelName: String { get }
    func preprocess(_ input: Input) throws -> MLFeatureProvider
    func postprocess(_ result: MLFeatureProvider) throws -> Output
}

extension OnDeviceMLFeature {
    func predict(_ input: Input) async throws -> Output {
        let log = OSLog(subsystem: "com.app.ml", category: modelName)
        let signpostID = OSSignpostID(log: log)

        // Load with warm-up
        os_signpost(.begin, log: log, name: "Load",
                    signpostID: signpostID)
        let config = MLModelConfiguration()
        config.computeUnits = .all
        let model = try MLModel(contentsOf: modelURL,
                                configuration: config)
        os_signpost(.end, log: log, name: "Load",
                    signpostID: signpostID)

        // Preprocess + Infer
        os_signpost(.begin, log: log, name: "Inference",
                    signpostID: signpostID)
        let features = try preprocess(input)
        let prediction = try model.prediction(from: features)
        os_signpost(.end, log: log, name: "Inference",
                    signpostID: signpostID)

        return try postprocess(prediction)
    }
}

Every AI feature in your app implements this protocol. You get consistent error handling, performance telemetry, and compute unit management for free.

import Vision
import NaturalLanguage
import FoundationModels

struct DocumentProcessor {
    // Step 1: OCR with Vision
    func extractText(from image: CGImage) async throws -> String {
        let request = VNRecognizeTextRequest()
        request.recognitionLevel = .accurate
        let handler = VNImageRequestHandler(cgImage: image)
        try handler.perform([request])
        return request.results?
            .compactMap { $0.topCandidates(1).first?.string }
            .joined(separator: "\n") ?? ""

Pipeline Memory Budget

When chaining models, memory adds up. Plan your budget carefully:

Multi-Model Memory Budget (iPhone 15 Pro, 8 GB)

Component Memory Duration
iOS + background apps ~2.5 GB Always
Your app (non-ML) ~100 MB Always
Vision text recognition ~50 MB During OCR only
NaturalLanguage tagger ~10 MB During extraction
Foundation Models Managed by OS During generation
Custom Core ML model ~50-200 MB Load/unload per step
MLX 3B model ~2.5 GB Load/unload per step
Available headroom ~5 GB

Run pipeline steps sequentially, not in parallel. Unload each model before loading the next to stay within budget.

Sequential, Not Parallel

Never load multiple large models simultaneously. Run each pipeline step, collect its output, unload, then load the next model. Vision + NaturalLanguage are lightweight enough to coexist, but an MLX model should be loaded alone.

Edge Cases and Failure Modes

Real-world inputs are messy. Your AI features need to handle the unexpected gracefully.

What Users Will Throw at You

  • Photos in extreme lighting (pitch dark, direct sun glare)
  • Blurry images from shaky hands or motion
  • Objects the model has never seen before
  • Mixed-language text in a single document
  • Background noise drowning out speech
  • Adversarial inputs (rotated, occluded, cropped)

How to Handle Them

  • Set confidence thresholds (reject predictions below 70%)
  • Show "I'm not sure" instead of a wrong answer
  • Add an "other" class for unknown objects
  • Use NLLanguageRecognizer to detect language first
  • Check audio levels before starting transcription
  • Validate input dimensions before inference

The Model Evaluation Checklist

Before shipping any AI feature, run it through these six checkpoints:

Six Metrics Before You Ship

  • Accuracy — does it get the right answer? Test on 100+ real-world inputs, not just your curated test set. What's your acceptable error rate?
  • Latency — fast enough for real-time? Under 30ms for camera features, under 2 seconds for generative text. Measure p95, not just average.
  • Model size — fits the download budget? Under 20 MB for embedded, use ODR or Background Assets for larger. Users on cellular notice.
  • Memory — runs without jetsam? Test on lowest-tier device with memory pressure. Stay under 200 MB for Core ML, plan carefully for MLX.
  • Power — drains the battery? Run continuous inference for 5 minutes, check battery drain. Neural Engine is 10x more power-efficient than CPU.
  • Device coverage — what % of your users can run it? Check your analytics for device distribution. If 40% of users have iPhone 12 or older, Foundation Models won't work for them.

Privacy as a Competitive Advantage

On-device AI isn't just a technical choice — it's a business differentiator. While competitors send user photos to cloud APIs, your app processes everything locally.

Your Privacy Story

  • App Privacy Report: "Data Not Collected" for all AI features
  • GDPR/CCPA: on-device processing doesn't require consent for AI analysis
  • Marketing: "Your photos/documents/voice never leave your device"
  • Enterprise: on-device AI meets data residency requirements automatically
  • User trust: Apple Intelligence's Private Cloud Compute sets the expectation — users increasingly expect on-device processing

The Complete Capstone: Design Your AI Feature

Hands-On: Technical Design Document

Design an on-device AI feature for your own app (or an app you wish existed). Write a technical design document covering all aspects:

  1. Problem statement — what user problem does this feature solve? Why does it need AI? Why on-device?
  2. Framework selection — which Apple/open-source framework(s) will you use? Justify with the decision framework from this chapter.
  3. Data pipeline — draw the input → preprocess → inference → postprocess → output flow. Identify where each framework fits.
  4. Optimization strategy — what compression techniques (Ch. 8) will you apply? What's your size and latency target?
  5. Device coverage matrix — which devices can run this feature? What's the fallback for unsupported devices?
  6. Memory budget — estimate RAM usage for each pipeline stage. Will it fit on your lowest-tier target device?
  7. Error handling — what can go wrong? How does the UI degrade gracefully for each failure mode?
  8. Privacy considerations — does any data leave the device? What do you disclose in the App Privacy Report?
  9. Monitoring plan — what metrics will you track? What's the alert threshold for each?
  10. Model update strategy — how will you ship model improvements? Staged rollout plan?
