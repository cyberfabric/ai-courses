Chapter 9

Create ML — Training Your Own Models

Your Data, Your Model, No Python Required

Why Train Your Own Model?

Every chapter so far has used models that someone else trained. Apple's Vision framework, pre-trained MobileNet, Hugging Face checkpoints — they're fantastic for general tasks. But what happens when you need to classify your company's products? Or detect your specific machine parts? Or recognize your custom sound events?

That's where Create ML comes in. Think of it as the "iMovie of machine learning" — Apple's tool that lets you train custom models by dragging in your data and clicking Train. No Python. No Jupyter notebooks. No GPU rental. Just your Mac and some labeled data.

What Create ML Gives You

  • No-code training — the Create ML app has a drag-and-drop GUI
  • Code-based training — Create ML Components framework lets you train in Swift
  • Transfer learning — start from Apple's pre-trained feature extractors, fine-tune with as few as 50 images per class
  • On-device personalization — update models on the user's iPhone with their data via MLUpdateTask
  • Standard output — every trained model is a .mlmodel file, ready for Core ML

Which Model Type Should You Pick?

Image Classification is the most common starting point. If you need to label an entire photo ("is this a receipt or a business card?"), start here. If you need to find where something is in the image (bounding boxes), choose Object Detection. For everything else, match the data type to the model type.

Image Classification: The 10-Minute Custom Model

Let's build an image classifier. The data format is dead simple: one folder per class, images inside each folder.

TrainingData/
  ├── coffee_mug/          ← 30+ photos of coffee mugs
  │   ├── mug_001.jpg
  │   ├── mug_002.jpg
  │   └── ...
  ├── water_bottle/        ← 30+ photos of water bottles
  │   ├── bottle_001.jpg
  │   └── ...
  ├── keyboard/            ← 30+ photos of keyboards
  │   ├── kb_001.jpg
  │   └── ...
  └── headphones/          ← 30+ photos of headphones
      ├── hp_001.jpg
      └── ...

That's your entire "dataset." The folder names become the class labels. Now open Create ML, choose Image Classification, and drag the TrainingData folder into the Training Data well.

Configuring Augmentations

Before clicking Train, enable augmentations. These are automatic image transformations that create variations of your training photos — giving the model more examples to learn from without you taking more photos.

Recommended Augmentations

  • Crop — random crops force the model to recognize objects that aren't perfectly centered
  • Rotation — ±15° rotations handle tilted camera angles
  • Flip (horizontal) — doubles your effective dataset for symmetric objects
  • Blur — slight blur simulates motion and out-of-focus shots
  • Noise — random noise teaches robustness to low-light conditions
  • Exposure — brightness variations handle different lighting environments

With augmentations, your 30 images per class effectively become 200+. The model sees coffee mugs that are rotated, cropped, blurred, and at different exposures — making it much more robust in the real world.

Transfer Learning: The Secret Sauce

Create ML doesn't train from scratch. It uses transfer learning — starting from Apple's pre-trained feature extractor that already understands edges, textures, shapes, and objects from millions of images. Your training only fine-tunes the top layers to distinguish your specific classes.

This is why 30-50 images per class is enough. The model already knows what "round" and "shiny" and "handle" look like. You're just teaching it that "round + shiny + handle = coffee mug."

Transfer Learning (Create ML)

  • 30-50 images per class
  • 2-5 minutes training on M1+
  • 90-98% accuracy typical
  • Uses Apple's base feature extractor

Training from Scratch

  • 10,000+ images per class
  • Hours or days of GPU training
  • Requires ML expertise to tune
  • Needs powerful hardware

Evaluating Your Model

After training, Create ML shows you a confusion matrix — a grid showing where the model gets confused. If it keeps mistaking water bottles for coffee mugs, you know you need more diverse photos of each, or they're genuinely similar from certain angles.

Key metrics to check:

  • Training accuracy — how well it performs on data it trained on (should be 95%+)
  • Validation accuracy — how well it performs on held-out data (the real measure)
  • Per-class precision/recall — which specific classes are weakest

The Overfitting Trap

If training accuracy is 99% but validation accuracy is 70%, your model memorized the training photos instead of learning general patterns. Fix it: add more diverse photos, enable more augmentations, or reduce the number of training iterations.

Object Detection Training

Image classification labels the whole image. Object detection finds where things are — drawing bounding boxes around specific objects. Think: "there's a coffee mug at position (120, 80) to (280, 240)."

The data format is different. You need images plus annotations — JSON or CSV files that specify bounding box coordinates for each object in each image.

// annotations.json format for Create ML
[
  {
    "image": "photo_001.jpg",
    "annotations": [
      {
        "label": "coffee_mug",
        "coordinates": {
          "x": 150, "y": 200,
          "width": 120, "height": 140
        }
      }
    ]
  }
]

Create ML includes a built-in annotation tool — open your images and draw bounding boxes directly in the app. For larger datasets, tools like RoboFlow or Label Studio can export in Create ML's JSON format.

Object Detection Data Tips

You need more data than for classification — aim for 100+ annotated images with varied object positions, sizes, and backgrounds. Include images with multiple objects, partial occlusion, and different lighting. The model needs to learn not just what the object looks like, but where it can appear.

Sound and Text Classification

Sound classification uses the same folder-per-class structure as images — one folder per sound category, 50+ audio clips (1-5 seconds each) per folder. Create ML extracts mel-frequency features automatically and the resulting model integrates directly with SoundAnalysis from Chapter 5.

Text classification uses a JSON or CSV with text samples and labels. Create ML offers two backends: Maximum Entropy (fast, works with 100+ samples) and Transfer Learning (better accuracy, needs 200+ samples per class). For most apps, the transfer learning backend wins.

Create ML Components: Training in Swift

The Create ML app is great for prototyping, but what if you want to train models programmatically? The Create ML Components framework lets you build training pipelines in pure Swift.

import CreateMLComponents
import CoreImage

// Define a training pipeline for image classification
let pipeline = ImageFeaturePrint()
    .appending(LinearClassifier())

// Load training data
let trainingData = try AnnotatedFiles(
    labeledDirectoryAt: trainingURL
)

// Train the model
let trained = try await pipeline.train(
    trainingData,
    annotationKeyPath: \.label
)

// Export as Core ML model
let mlModel = try trained.export()
try mlModel.write(to: outputURL)

Create ML Components is composable — chain transformers and estimators like UNIX pipes: ImageFeaturePrint (feature extractor) → LinearClassifier (classification head). Other estimators include BoostedTreeClassifier for tabular data and TimeSeriesForecaster for time-series predictions.

When to Use the App vs Components

Use the Create ML app for exploring model types, quick prototyping, and one-off training sessions. Use Create ML Components when you need automated training pipelines, integration with CI/CD, or on-device training triggered by app logic.

On-Device Personalization with MLUpdateTask

Here's the killer feature: you can update a model on the user's iPhone using their data. The model improves for each user individually, and their data never leaves the device.

Think of it as "the model adapts to you." A handwriting recognizer that learns your letter shapes. A food classifier that learns your specific meal preferences. A gesture detector that calibrates to your hand movements.

The key constraint: only specific layers are marked as updatable during model creation. Typically, you freeze the feature extractor and only update the classification head. This keeps the update fast (seconds, not minutes) and prevents catastrophic forgetting.

import CoreML

func personalizeModel(
    with trainingData: MLBatchProvider
) async throws -> MLModel {
    let modelURL = Bundle.main.url(
        forResource: "UpdatableClassifier",
        withExtension: "mlmodelc"
    )!

    // Create the update task
    let updateTask = try MLUpdateTask(
        forModelAt: modelURL,
        trainingData: trainingData,
        configuration: nil,
        completionHandler: { context in
            // Save the updated model
            let updatedModel = context.model
            let documentsURL = FileManager.default
                .urls(for: .documentDirectory,
                      in: .userDomainMask).first!
            let savedURL = documentsURL
                .appendingPathComponent("PersonalModel.mlmodelc")
            try? context.model.write(to: savedURL)
        }
    )

    // Run the update
    updateTask.resume()
}

Making a Model Updatable

Not every .mlmodel supports on-device updates. The model must be specifically configured with updatable layers during creation — either in Create ML (check "Updatable" in training options) or via coremltools in Python with coremltools.models.neural_network.make_updatable(). You can't retroactively make a model updatable.

Training Performance Benchmarks

How fast does Create ML actually train? Here are real numbers for common scenarios:

Create ML Training Times (M1 MacBook Pro)

Task Dataset Size Training Time Model Size Accuracy
Image Classification (5 classes) 150 images (30/class) ~45 seconds 17 KB 94%
Image Classification (20 classes) 1,000 images (50/class) ~3 minutes 56 KB 91%
Object Detection 500 annotated images ~30 minutes 5.2 MB 82% mAP
Sound Classification (4 classes) 200 clips (50/class) ~2 minutes 42 KB 92%
Text Classification (3 classes) 600 samples (200/class) ~15 seconds 85 KB 88%

Image classification models are tiny because Create ML outputs just the classification head — the feature extractor is already on the device as part of the OS.

Why Are Image Models So Small?

A 17 KB model seems impossibly small. The trick: Create ML's image classifiers use ImageFeaturePrint, Apple's built-in feature extractor that ships with every iPhone. Your model only contains the lightweight classification layer on top. At runtime, Core ML automatically chains the built-in feature extractor with your custom classifier. Smaller model, same accuracy.

Integrating Your Trained Model

The output of Create ML is a standard .mlmodel file. Integration is identical to Chapter 3:

import CoreML
import Vision

// Load your custom model (Xcode auto-generates the class)
let model = try DeskObjectClassifier(configuration: .init())
let vnModel = try VNCoreMLModel(for: model.model)

// Create a classification request
let request = VNCoreMLRequest(model: vnModel) { request, error in
    guard let results = request.results
        as? [VNClassificationObservation],
          let top = results.first else { return }

    print("\(top.identifier): \(top.confidence * 100)%")
    // "coffee_mug: 96.3%"
}

// Run on an image
let handler = VNImageRequestHandler(cgImage: photo)
try handler.perform([request])

That's it. Your custom-trained model works exactly like any other Core ML model — same VNCoreMLRequest workflow, same performance characteristics, same Neural Engine acceleration.

Data Collection Best Practices

Your model is only as good as your training data. The golden rule: train on data that looks like what the model will see in production.

Good Training Photos

  • Varied lighting (indoor, outdoor, dim)
  • Multiple angles and distances
  • Different backgrounds and contexts
  • Some blur and partial occlusion
  • Balanced classes (±20% in size)
  • Separate validation set (20% held out)

Bad Training Photos

  • Same angle, same background every time
  • Perfect studio lighting in every shot
  • Same distance, same orientation
  • 200 dogs but only 20 cats (imbalanced)
  • Same photo in training AND validation
  • Only one color/variant of the object

The Minimal Viable Dataset

How little data can you get away with? It depends on the task and the visual similarity between your classes:

30
images/class minimum

For visually distinct objects (mug vs keyboard vs plant)

100
images/class recommended

For similar-looking objects (types of birds, car models)

200+
images/class for production

For customer-facing features with high accuracy requirements

Start with 30. Train. Test. If accuracy is below target, add more data for the weakest classes first — faster than collecting a massive dataset upfront.

The #1 Training Pitfall: Background Bias

If all your coffee mug photos are on a white desk, the model learns "white desk = coffee mug." When someone holds a mug against a dark background, it fails. The fix: vary backgrounds aggressively. Add an "other/negative" class for images with no target object. And if you need to know where something is, use object detection, not image classification.

Exercise: Build a Desk Object Classifier

Hands-On: Your First Custom Model

Build a custom image classifier for 5 objects on your desk using Create ML:

  1. Choose 5 objects — pick items that are visually distinct (coffee mug, keyboard, phone, headphones, notebook)
  2. Collect training data — take 30 photos of each object with your iPhone. Vary angles, lighting, backgrounds, and distance. Save in a folder-per-class structure.
  3. Hold out validation data — set aside 10 additional photos per object that you will NOT use for training
  4. Train in Create ML — open the Create ML app, create a new Image Classification project, drag in your training folder, enable crop + rotation + flip augmentations, click Train
  5. Evaluate — review the confusion matrix. Which classes are confused? Why?
  6. Test on validation set — drag your held-out photos into the Testing area. What's the real-world accuracy?
  7. Export and integrate — save the .mlmodel, drag into your Xcode project from Chapter 3, replace MobileNet with your custom model, run on your iPhone

Bonus Challenges:

  • Add a 6th class: "nothing" (photos of empty desk, walls, floor) — does accuracy change?
  • Try the same exercise with Create ML Components in Swift instead of the GUI app
