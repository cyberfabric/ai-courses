MLX Swift and Open-Source Models

Llama, Mistral, and Gemma on Your iPhone

Beyond Apple's Walled Garden

In Chapter 6, you met Apple's Foundation Models framework — a ~3B parameter model baked into iOS 26, guided generation, tool calling, the works. It's polished, private, and zero-setup. But it's also a black box: you can't choose the model, you can't fine-tune it, and it only runs on iPhone 15 Pro or newer.

What if you want to run Llama 3.2, Mistral, Gemma, or Phi on a user's device? What if you need a model fine-tuned for your domain? What if you need to support iPhone 14 or iPad Air? That's where the open-source ecosystem comes in.

Three runtimes let you run open-source models on Apple devices:

MLX
Apple's open-source framework

Native Swift, Metal GPU, unified memory

llama.cpp
Community C/C++ engine

Widest model support, GGUF format

ExecuTorch
Meta's PyTorch runtime

Core ML + MPS backends

MLX: Apple's Open-Source ML Framework

MLX is Apple's answer to PyTorch and NumPy — an array framework built specifically for Apple Silicon. Think of it as "Metal for ML": it runs tensor operations on the GPU via Metal, leverages unified memory (CPU and GPU share the same RAM, no copying), and has first-class Swift support.

Loading and Running a Model

The mlx-swift-lm package handles model downloading from Hugging Face, loading weights, and text generation:

import MLXLLM
import MLXLMCommon

// Load a quantized model from Hugging Face
let modelContainer = try await LLMModelFactory.shared.load(
    hub: .init(id: "mlx-community/Llama-3.2-1B-Instruct-4bit")
)

// Generate text
let prompt = "Explain Core ML in one sentence:"
let result = try await modelContainer.perform { model, tokenizer in
    let input = try await tokenizer.applyChatTemplate(
        messages: [["role": "user", "content": prompt]]
    )
    return try MLXLMCommon.generate(
        input: input,
        parameters: .init(temperature: 0.7),
        model: model,
        tokenizer: tokenizer
    ) { tokens in
        // Streaming callback — called for each new token
        if tokens.count >= 500 { return .stop }
        return .more
    }
}

print(result.output)

The mlx-community organization on Hugging Face hosts hundreds of models pre-converted and quantized for MLX. Search for any popular model + "mlx" to find ready-to-use versions: mlx-community/Llama-3.2-1B-Instruct-4bit, mlx-community/Mistral-7B-Instruct-v0.3-4bit, mlx-community/gemma-2-2b-it-4bit.

Building a Chat Interface

Here's the SwiftUI view model pattern for streaming chat:

import SwiftUI
import MLXLLM
import MLXLMCommon

@Observable
class ChatViewModel {
    var messages: [ChatMessage] = []
    var isGenerating = false
    private var modelContainer: ModelContainer?

    func loadModel() async throws {
        modelContainer = try await LLMModelFactory.shared.load(
            hub: .init(id: "mlx-community/Llama-3.2-1B-Instruct-4bit")
        )
    }

    func send(_ text: String) async throws {
        messages.append(.user(text))
        messages.append(.assistant(""))
        isGenerating = true

        try await modelContainer?.perform { model, tokenizer in
            let chatMessages = self.messages.map {
                ["role": $0.role, "content": $0.content]
            }
            let input = try await tokenizer.applyChatTemplate(
                messages: chatMessages
            )
            try MLXLMCommon.generate(
                input: input,
                parameters: .init(temperature: 0.7),
                model: model,
                tokenizer: tokenizer
            ) { tokens in
                let text = tokenizer.decode(tokens: tokens)
                Task { @MainActor in
                    self.messages[self.messages.count - 1]
                        .content = text
                }
                return tokens.count >= 1000 ? .stop : .more
            }
        }

        isGenerating = false
    }
}

llama.cpp: The Universal Engine

If MLX is Apple's first-party solution, llama.cpp is the community's battle-tested workhorse. Written in C/C++, it runs on everything from Raspberry Pi to iPhone to Linux servers. It supports the widest range of models and has the most active community.

Integration Options

There are several ways to use llama.cpp on iOS:

llama.swift (Recommended)

  • Pre-compiled XCFramework
  • Proper semantic versioning via SPM
  • Clean Swift API wrapping C++ internals
  • Metal GPU acceleration built-in

Direct llama.cpp Source

  • Requires unsafeFlags in SPM
  • Prevents semantic versioning
  • Complex build configuration
  • More control, more headaches

GGUF: The Model Format

llama.cpp uses the GGUF format (GPT-Generated Unified Format). Models come in quantization variants named by their bit depth and method:

  • Q4_K_M — 4-bit quantization, medium quality. Best balance of size and quality for phones.
  • Q5_K_M — 5-bit, slightly better quality, ~25% larger than Q4.
  • Q8_0 — 8-bit, near-original quality, 2x the size of Q4. For devices with plenty of RAM.
  • Q2_K — 2-bit, aggressive compression. Noticeable quality loss, but tiny.

How to Choose a GGUF Variant

For iPhones with 6GB RAM: use Q4_K_M for 1-3B models. For 8GB RAM devices (iPhone 15 Pro+): you can run Q4_K_M of 7B models. Always leave ~2GB for the OS and your app. A 7B Q4_K_M model needs about 4.5GB of RAM.

ExecuTorch: PyTorch Native

ExecuTorch is Meta's framework for deploying PyTorch models on mobile and edge devices. If your ML team trains in PyTorch and you want the most direct path to iPhone, ExecuTorch is worth considering.

The key advantage: ExecuTorch supports multiple hardware backends on Apple devices:

  • Core ML delegate — routes operations to CPU, GPU, or Neural Engine via Core ML
  • MPS delegate — Metal Performance Shaders for GPU-accelerated inference
  • XNNPACK — optimized CPU kernels as fallback

The tradeoff: ExecuTorch has a steeper setup curve than MLX or llama.cpp. You need to export your PyTorch model through torch.export, lower it for the target backend, and package the resulting .pte file. It's a better fit when your team already has a PyTorch training pipeline.

Memory Management for Large Models

Running a multi-billion parameter model on a phone is a memory tightrope act. Here's the reality:

iPhone RAM Constraints

  • iPhone SE / 14: 4GB total. OS uses ~1.5GB. You get ~2.5GB. Max model: 1B at Q4.
  • iPhone 15: 6GB total. ~4GB available. Comfortable for 1-3B at Q4.
  • iPhone 15 Pro / 16 Pro: 8GB total. ~5.5GB available. Can fit 7B at Q4.

Exceed the budget and iOS kills your app via jetsam — no warning, instant termination.

Memory-Safe Patterns

class ModelManager {
    private var modelContainer: ModelContainer?

    // Load on demand, not at launch
    func loadIfNeeded() async throws {
        guard modelContainer == nil else { return }
        modelContainer = try await LLMModelFactory.shared.load(
            hub: .init(id: "mlx-community/Llama-3.2-1B-Instruct-4bit")
        )
    }

    // Unload when backgrounded or under memory pressure
    func unload() {
        modelContainer = nil
        // Force Metal to release GPU memory
        MLX.GPU.clearCache()
    }

    // Listen for memory warnings
    func setupMemoryWarningHandler() {
        NotificationCenter.default.addObserver(
            forName: UIApplication.didReceiveMemoryWarningNotification,
            object: nil, queue: .main
        ) { [weak self] _ in
            self?.unload()
        }
    }
}

The KV Cache Trap

As a conversation grows, the KV cache (key-value cache for attention) consumes more memory. A 1B model with 2048 tokens of context uses ~200MB just for the KV cache. For long conversations, either limit context length or periodically reset the session.

Model Quantization Formats Compared

Each runtime uses its own model format. Here's how they compare:

MLX Quantized (.safetensors)

  • Native Metal GPU acceleration
  • Unified memory (no CPU-GPU copy)
  • 4-bit and 8-bit quantization
  • Download from mlx-community on HF
  • Best for: Apple-only projects

GGUF (.gguf)

  • Widest model compatibility
  • Many quant variants (Q2-Q8)
  • Metal acceleration in llama.cpp
  • Cross-platform (Android too)
  • Best for: maximum flexibility

Same Swift protocol, different backend

import FoundationModels

protocol TextGenerator {
    func generate(prompt: String) async throws -> String
}

// Foundation Models backend
struct AppleFMGenerator: TextGenerator {
    func generate(prompt: String) async throws -> String {
        let session = LanguageModelSession()
        let response = try await session.respond(to: prompt)
        return response.content
    }
}

// MLX backend
struct MLXGenerator: TextGenerator {
    let modelContainer: ModelContainer

    func generate(prompt: String) async throws -> String {
        try await modelContainer.perform { model, tokenizer in
            // ... MLX generation code
        }
    }
}

// Runtime selection
func bestAvailableGenerator() async throws -> TextGenerator {
    if #available(iOS 26, *),
       SystemLanguageModel.default.availability == .available {
        return AppleFMGenerator()
    }

    let ram = ProcessInfo.processInfo.physicalMemory
    let modelID = ram > 6_000_000_000
        ? "mlx-community/Llama-3.2-3B-Instruct-4bit"
        : "mlx-community/Llama-3.2-1B-Instruct-4bit"

    let container = try await LLMModelFactory.shared.load(
        hub: .init(id: modelID)
    )
    return MLXGenerator(modelContainer: container)
}

Thermal and Battery Considerations

Running a billion-parameter model generates serious heat. In Chapter 7 you learned about thermal throttling — the same principles apply here, but more aggressively.

Production Checklist for On-Device LLMs

  • Limit generation length — cap at 500-1000 tokens per response to prevent thermal runaway
  • Monitor thermal state — check ProcessInfo.processInfo.thermalState before starting generation
  • Unload when backgrounded — release model memory when the app goes to background
  • Show progress indicators — streaming tokens keeps users engaged during generation
  • Batch wisely — don't run inference in a tight loop; give the device time to cool
  • Test on lowest-tier device — if it works on iPhone SE, it works everywhere
