Chapter 7

Hardware Acceleration — Neural Engine and Metal

What Happens When Your Model Hits Silicon

Every model you've run so far — image classification, Vision requests, Foundation Models — was executed by hardware inside your iPhone. But which hardware? Your iPhone has three distinct compute engines that can run ML workloads, and the one that gets the job makes a 15x difference in speed.

This chapter cracks open Apple Silicon to show you exactly how the CPU, GPU, and Neural Engine work together. You'll learn to control which engine runs your model, benchmark each one, profile with Instruments, and handle the thermal throttling that hits during sustained workloads. The goal: turn "it works" into "it works fast."

The Three Engines Inside Your iPhone

Every Apple Silicon chip since A11 contains three compute units that can run ML inference. Think of them as three different workers with very different skills:

CPU
The Generalist

Handles anything but nothing fast. Best for tiny models, simple math, and fallback. Slowest for ML.

GPU
The Parallel Processor

Thousands of cores for parallel math. Great for custom Metal shaders, image processing, and models with unsupported Neural Engine ops.

ANE
The ML Specialist

Purpose-built for tensor operations. 16 cores doing nothing but matrix multiply. Fastest and most power-efficient for supported models.

ANE = Apple Neural Engine. You'll see it called "Neural Engine," "ANE," or just "NE" in documentation and profiling tools. Same thing. It's the dedicated ML accelerator on the chip.

Neural Engine: A Brief History

Apple has been building dedicated ML hardware into every chip since 2017. Each generation brings more cores and higher throughput:

Neural Engine Progression

Chip Year NE Cores TOPS Notable Capability
A11 Bionic 2017 2 0.6 First Neural Engine (Face ID)
A12 Bionic 2018 8 5 Core ML 2, real-time inference
A14 Bionic 2020 16 11 16-core NE becomes standard
A15 Bionic 2021 16 15.8 iPhone SE (3rd gen) baseline
A16 Bionic 2022 16 17 4nm process
A17 Pro 2023 16 35 First Apple Intelligence support
A18 / A18 Pro 2024 16 35 2x faster ML vs A16
A19 / A19 Pro 2025 16 ~45 GPU Neural Accelerators, 3nm

TOPS = Trillion Operations Per Second. The A19 Pro at ~45 TOPS can perform 45 trillion simple math operations every second. For context, a MobileNetV2 inference needs roughly 300 million operations — the Neural Engine can run it about 150,000 times per second in theory. In practice, memory bandwidth limits real throughput, but you get the idea: the hardware is fast.

A19 Pro's secret weapon: GPU Neural Accelerators. The A19 Pro embeds dedicated tensor processing units inside each GPU core. This means the GPU itself can now run ML workloads natively, delivering up to 3x the peak GPU compute of the A18 Pro. Metal 4 exposes these through its new tensor API.

MLComputeUnits: Controlling Where Your Model Runs

Core ML lets you choose which compute units run your model via MLModelConfiguration:

import CoreML

let config = MLModelConfiguration()

// Let Core ML decide (usually best)
config.computeUnits = .all

// Force specific compute units
config.computeUnits = .cpuOnly          // Slowest, always available
config.computeUnits = .cpuAndGPU        // Skip Neural Engine
config.computeUnits = .cpuAndNeuralEngine  // Skip GPU

let model = try MLModel(contentsOf: modelURL, configuration: config)

MLComputeUnits Options

.allCore ML picks the best engine per operation. Default and usually optimal. Some ops run on ANE, others on GPU, a few on CPU.
.cpuAndNeuralEngineSkip GPU. Use when GPU is busy rendering (games, AR). Avoids GPU contention.
.cpuAndGPUSkip Neural Engine. Use when a model has ops unsupported by ANE (causes fallback overhead).
.cpuOnlyCPU only. Slowest, but guaranteed available on every device. Useful as a fallback.
Don't outsmart Core ML. In most cases, .all is the best choice. Core ML's runtime analyzes your model and dispatches each operation to the fastest available engine. Only override if you've profiled and found a specific reason (GPU contention, ANE-unsupported ops, thermal budget).

Why the First Inference Is Always Slow

You run your model for the first time and it takes 500ms. You run it again — 3ms. What happened?

The first inference triggers model compilation: Core ML compiles the model for the specific hardware, allocates memory buffers, initializes the Neural Engine pipeline, and compiles any Metal shaders needed for GPU operations. This is a one-time cost that gets cached for subsequent runs.

// Warm-up pattern: run a few dummy inferences before measuring
func warmUp(model: VNCoreMLModel, request: VNCoreMLRequest) {
    let dummyImage = CIImage(color: .black).cropped(to: CGRect(x: 0, y: 0, width: 224, height: 224))
    let handler = VNImageRequestHandler(ciImage: dummyImage)

    for _ in 0..<3 {
        try? handler.perform([request])
    }
    // Now the model is warm — subsequent inferences will be fast
}
Warm up before measuring. Always run 3–5 dummy inferences before benchmarking. The first-run compilation cost can be 50–100x slower than steady-state performance. If your app has a splash screen or loading state, warm up the model there.

Building a Benchmark Harness

To truly understand your model's performance, you need to measure it systematically. Here's a minimal benchmark harness:

import CoreML

func benchmarkModel(
    modelURL: URL,
    computeUnits: MLComputeUnits,
    iterations: Int = 50
) async throws -> (p50: Double, p95: Double, p99: Double) {
    let config = MLModelConfiguration()
    config.computeUnits = computeUnits
    let model = try MLModel(contentsOf: modelURL, configuration: config)

    // Warm up
    let dummyInput = try model.prediction(from: MLDictionaryFeatureProvider(dictionary: [:]))
    _ = dummyInput  // discard

    // Measure
    var times: [Double] = []
    for _ in 0..<iterations {
        let start = CFAbsoluteTimeGetCurrent()
        _ = try model.prediction(from: createSampleInput(for: model))
        let elapsed = (CFAbsoluteTimeGetCurrent() - start) * 1000  // ms
        times.append(elapsed)
    }

    times.sort()
    return (
        p50: times[iterations / 2],
        p95: times[Int(Double(iterations) * 0.95)],
        p99: times[Int(Double(iterations) * 0.99)]
    )
}
Why p50/p95/p99? Average latency hides outliers. p50 (median) is your typical experience. p95 tells you what 1 in 20 users sees. p99 catches rare spikes. For real-time features (camera), p95 matters more than average.

Real Device Benchmarks

Here's what MobileNetV2 inference actually looks like across devices and compute units:

Device .all (ANE) .cpuAndGPU .cpuOnly ANE Speedup
iPhone SE (3rd, A15) 4 ms 14 ms 48 ms 12x vs CPU
iPhone 15 (A16) 3.5 ms 12 ms 45 ms 13x vs CPU

The pattern is clear: the Neural Engine is 12–20x faster than CPU and 3–5x faster than GPU for this typical classification model. And the gap widens with each generation.

Device diversity is your design constraint. Your fastest user (iPhone 17 Pro) gets 1.5ms inference. Your slowest supported user (iPhone SE 3rd gen) gets 4ms. That's a manageable 2.7x spread for MobileNetV2, but for larger models the gap grows dramatically. Always benchmark on your minimum supported device.

Metal 4: GPU-Accelerated ML

Metal 4 (iOS 26) introduces first-class tensor support in the GPU pipeline. This matters because you can now run ML inference inside a Metal shader — no round-trip through device memory between the GPU and ML model.

2

→
5

Before Metal 4, running ML on the GPU meant: render frame → copy to CPU memory → run Core ML → copy results back to GPU → render with results. Each copy added latency. Metal 4 eliminates those copies by keeping everything on the GPU.

When to Use Metal 4 for ML

Good Use Cases for Metal 4 ML

  • Real-time image effects that need per-pixel ML (style transfer, super-resolution)
  • Games with AI-driven graphics (upscaling, denoising)
  • AR pipelines where ML and rendering share data
  • Custom model architectures not supported by Core ML

Stick with Core ML + ANE

  • Standard classification or detection models
  • Vision framework requests (already optimized)
  • Foundation Models (managed by the system)
  • Any model that runs fine on Neural Engine
Metal 4 ML is advanced. Most iOS developers will never need to write Metal shaders for ML. Core ML + the Neural Engine handle 90%+ of use cases. Metal 4 is for the 10% where you need GPU-ML fusion — games, AR, and custom graphics pipelines.

Thermal Throttling: The Hidden Performance Killer

Your model runs at 2ms for the first 30 seconds, then suddenly slows to 8ms. The iPhone got warm and started thermal throttling — reducing clock speeds to prevent overheating.

This is the #1 surprise for developers who benchmark in short bursts but deploy for sustained use. A fitness app tracking body pose needs to run for 30+ minutes. A document scanner processes 50 pages in a row. A camera classifier runs continuously.

import Foundation

// Monitor thermal state
NotificationCenter.default.addObserver(
    forName: ProcessInfo.thermalStateDidChangeNotification,
    object: nil, queue: .main
) { _ in
    switch ProcessInfo.processInfo.thermalState {
    case .nominal:
        // Full speed — use your fastest model
        useHighQualityModel()
    case .fair:
        // Slightly warm — still OK
        useHighQualityModel()
    case .serious:
        // Getting hot — reduce workload
        useLightweightModel()
        reduceFrameRate(to: 15)
    case .critical:
        // About to throttle hard — minimize ML work
        pauseInference()
        showCooldownMessage()
    @unknown default: break
    }
}
Design for .serious, not .nominal. If your app does sustained ML inference (camera, audio, fitness), your steady-state performance is the .serious thermal state, not .nominal. Benchmark for 60+ seconds to see real-world performance, not just the first-second burst.

Profiling with Instruments

Xcode's Instruments includes dedicated tools for ML profiling:

Instruments for ML Profiling

  • Core ML Instrument — Shows model load time, prediction latency, and which compute units each operation uses. The first tool to reach for.
  • Neural Engine Instrument — Visualizes compute running on the Neural Engine. See when ops fall back to CPU or GPU.
  • Metal System Trace — GPU utilization, shader execution, and memory transfers. Use when profiling Metal 4 ML pipelines.
  • Core ML Performance Report — Generate without writing code: Xcode → Product → Profile → select your model. Shows per-operation breakdown.

Spotting Neural Engine Fallbacks

The Core ML Instrument shows you exactly which operations run where. If you see operations falling back to CPU (marked in a different color), that's your bottleneck. Common reasons for ANE fallback:

  • Unsupported operations — Some custom layers or exotic activations aren't supported by ANE
  • Dynamic shapes — ANE works best with fixed input dimensions
  • Very small tensors — The overhead of dispatching to ANE isn't worth it for tiny operations
  • Quantization format — Some quantization schemes aren't ANE-compatible
Quick profiling without code: In Xcode, click on your .mlmodel or .mlpackage file, then click the Performance tab. Xcode will run the model on a connected device and show you load time, prediction time, and compute unit usage per operation — no code needed.

Unified Memory: Apple Silicon's Secret Advantage

On traditional computers, the CPU and GPU have separate memory. Moving data between them takes time. Apple Silicon uses unified memory — CPU, GPU, and Neural Engine all share the same memory pool.

What this means for ML:

  • No memory copies between compute units. A texture rendered on the GPU can be fed directly to the Neural Engine.
  • Larger models fit because you're not duplicating buffers across memory pools.
  • Faster pipeline transitions when Core ML dispatches different operations to different engines.

This is why Apple Silicon devices punch above their weight in ML benchmarks compared to competitors with higher raw TOPS but separate memory architectures.

Practical Guidelines

Do

  • Use .all compute units by default
  • Warm up models with 3–5 dummy inferences
  • Benchmark for 60+ seconds (not just first inference)
  • Monitor thermal state for sustained workloads
  • Profile with Core ML Instrument before optimizing
  • Benchmark on your minimum supported device

Don't

  • Force a specific compute unit without profiling
  • Trust first-inference timing (compilation overhead)
  • Ignore thermal throttling in sustained workloads
  • Assume GPU is faster than ANE for ML
  • Benchmark only on your newest device
  • Optimize before profiling
