📊 Six Metrics Before You Ship

🎯
Accuracy
Test on 100+ real-world inputs. What's your acceptable error rate?
⚡
Latency
< 30 ms for camera, < 2 s for generation. Measure p95, not average.
📦
Model Size
< 20 MB embed, use ODR/Background Assets for larger. Users on cellular notice.
💧
Memory
< 200 MB for Core ML. Test on lowest-tier device under memory pressure.
🔋
Power
5 min continuous test. Neural Engine is 10x more power-efficient than CPU.
📱
Device Coverage
What % of users can run it? Check analytics for device distribution.

📚 Key Terms

Decision Framework
Start with built-in APIs (Vision, NL, Sound, FM). Only use custom models when built-ins can't handle your domain.
Multi-Model Pipeline
Chain frameworks sequentially: Vision (detect) → NL (extract) → FM (generate). Unload between large models.
OnDeviceMLFeature Protocol
Reusable Swift protocol: preprocess, predict, postprocess with built-in os_signpost telemetry and error handling.
Progressive Enhancement
Every device gets some AI; better devices get more. Runtime capability detection, not lowest common denominator.

🤔 Quick Quiz

Test Yourself

What's the golden rule for choosing an on-device AI framework?

Always start with Apple's built-in APIs (Vision, NaturalLanguage, SoundAnalysis, Foundation Models). They're free, Apple-maintained, optimized for every device, and require zero model management. Only reach for custom models or MLX when the built-ins can't handle your specific domain.

Why should multi-model pipeline steps run sequentially, not in parallel?

Memory. Loading multiple large models simultaneously can exceed the device's RAM budget and trigger jetsam termination. Run each step, collect its output, unload the model, then load the next. Vision + NaturalLanguage are lightweight enough to coexist, but MLX models should load alone.

What are the six metrics to check before shipping an AI feature?

1) Accuracy — right answers on real inputs. 2) Latency — fast enough for the UX (p95). 3) Model size — fits download budget. 4) Memory — no jetsam on lowest device. 5) Power — acceptable battery drain. 6) Device coverage — what % of users can run it.

How does device diversity work as a design constraint?

Progressive enhancement. iPhone SE/12 gets Vision + NaturalLanguage + SoundAnalysis (always available). iPhone 14/15 adds MLX 1B-3B models. iPhone 15 Pro+ adds Foundation Models and 7B models. iPad M-series gets everything. Every user gets value; better devices unlock more capabilities.

What framework changes should you watch for at WWDC 2026?

Core AI (rumored successor to Core ML unifying classical and generative AI), developer LoRA adapter APIs for Foundation Models fine-tuning, multimodal Foundation Model APIs (vision + language), and MLX ecosystem expansion with more models and official Apple support.

📋 Key Takeaways

If You Remember Nothing Else

  • Golden rule: start with Apple's built-in APIs. Custom models only when built-ins can't handle your domain.
  • Chain frameworks into multi-model pipelines: Vision → NL → Foundation Models. Run sequentially, unload between steps.
  • Six metrics before shipping: accuracy, latency (p95), model size, memory, power, device coverage.
  • Progressive enhancement: every device gets some AI. Better devices get more. Never lowest common denominator.
  • Confidence thresholds: "I'm not sure" beats a wrong answer. Reject predictions below 70%.
  • Privacy is a competitive advantage: on-device = "Data Not Collected" = simplified compliance.
  • Watch for: Core AI (WWDC 2026), LoRA adapter APIs, multimodal Foundation Models, MLX ecosystem growth.
  • The patterns are permanent: inference pipelines, hardware dispatch, memory management, progressive enhancement.
