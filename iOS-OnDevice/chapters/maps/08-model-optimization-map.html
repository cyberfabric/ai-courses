Chapter 8

Model Optimization — Making Models iPhone-Ready

Memory Map

💡 The Core Insight

💡

Like JPEG for neural networks — trade a tiny bit of precision for massive size reduction.

A 22 MB float32 model becomes 2.8 MB with 4-bit palettization and only 1.1% accuracy loss. Three techniques — quantization, palettization, pruning — all available through coremltools with a few lines of Python.

📱 iPhone Shipping Constraints

📦
< 50 MB Model Size
Ideally under 20 MB. Over 50 MB increases download time and App Store friction.
⚡
< 30 ms Latency
On Neural Engine. Under 33ms = real-time at 30fps for camera features.
📌
< 200 MB Memory
Runtime footprint. Over 200 MB risks jetsam termination on older devices.

🔧 The Three Techniques

📏
Quantization
Reduce precision: float32 → float16 (2x), int8 (4x), int4 (8x). Like RAW → JPEG. linear_quantize_weights()
🎨
Palettization
Cluster weights into 2N value palette + store indices. Like GIF indexed color. palettize_weights(). Apple's secret weapon.
✂️
Pruning
Zero out small weights. 50% sparsity = half the weights are zero. Best combined with palettization. prune_weights()

📏 Quantization Precision Levels

Format Bytes/Weight Size Reduction Typical Accuracy Loss
Float32 4 Baseline —
Float16 2 2x 🆕 FREE ~0%
Int8 1 4x < 1%
Int4 0.5 8x 1–3%

💡 Float16 is always your first step — 2x smaller with zero accuracy loss. Use compute_precision=ct.precision.FLOAT16.

🎨 Palettization Bit Depths

Bits Palette Size Compression vs FP16 Accuracy Loss
8-bit 256 values 2x < 0.5%
6-bit 64 values 2.7x < 1%
4-bit 🎯 16 values 4x 1–2%
2-bit 4 values 8x 3–8%

💡 4-bit palettization is the sweet spot for most apps. K-means clustering adapts to your model's weight distribution.

📊 Real Numbers: MobileNetV3-Large

Variant Size Accuracy Drop
Float32 (baseline) 22 MB 75.2% —
Float16 11 MB 75.2% 0.0%
Int8 quantized 5.5 MB 74.8% -0.4%
4-bit palettized 2.8 MB 74.1% -1.1%
4-bit + 50% pruned 1.8 MB 73.0% -2.2%

22 MB → 2.8 MB = 8x compression with only 1.1% accuracy drop. Joint compression pushes to 12x.

📚 Key Terms

Post-Training Quantization
Compress an already-trained model. No training data needed. Takes seconds. Use cto.coreml.linear_quantize_weights().
Palettization
Cluster weights into a lookup table of 2N values using k-means. Store palette indices instead of weights. Apple's unique approach.
Magnitude Pruning
Set the smallest weights (by absolute value) to zero. Sparsity = % of weights that are zero. Best combined with palettization.
Accuracy Budget
How much accuracy you can lose before users notice. Define this before optimizing. 1–2% is invisible for classification.
.mlpackage
Recommended directory bundle format. Supports all optimization types and larger models. Use for all new work.
.mlmodelc
Pre-compiled model. Skips first-run compilation (saves 100–500ms). Generate with xcrun coremlcompiler.

🤔 Quick Quiz

Test Yourself

What's the first optimization you should always apply?

Float16 conversion. It halves model size with zero accuracy loss. Use compute_precision=ct.precision.FLOAT16 during ct.convert(). It's free.

How does palettization differ from quantization?

Quantization maps weights to a fixed linear scale (e.g., int8 = 256 evenly spaced values). Palettization clusters weights with k-means into a custom palette adapted to the model's actual weight distribution, then stores palette indices. Palettization often preserves more accuracy at the same compression ratio.

Why combine pruning with palettization?

Pruning alone gives modest size reduction (zeros still take space). But when combined with palettization, the zeros compress extremely well, and the Neural Engine can skip zero-weight operations for faster inference. The combo gives both better compression and better speed.

What's an "accuracy budget" and why define it upfront?

It's the maximum accuracy loss acceptable for your use case before users notice. Image classification: 1–2% is usually fine. Medical apps: even 0.5% may be too much. Defining it upfront prevents over-optimizing or shipping degraded models.

What's the #1 bug when converting Hugging Face models?

Wrong scale and bias during conversion. Each model expects specific input normalization (listed on the Hugging Face model card). If you get these wrong, the model sees garbage input and predictions are meaningless.

📋 Key Takeaways

If You Remember Nothing Else

  • iPhone targets: <50 MB model size, <30 ms latency, <200 MB runtime memory.
  • Float16 is always step one — 2x compression, zero accuracy loss. It's free.
  • Three techniques: quantization (precision), palettization (lookup tables), pruning (zeroing weights).
  • 4-bit palettization = sweet spot. 8x compression, ~1% accuracy loss for most models.
  • Joint pruning + palettization = maximum compression (12x+). Best of both worlds.
  • Palettization often beats quantization at the same ratio because k-means adapts to your model.
  • Always measure accuracy on a real validation set. Define your accuracy budget before optimizing.
  • Use .mlpackage format. Pre-compile to .mlmodelc to skip first-run compilation.
