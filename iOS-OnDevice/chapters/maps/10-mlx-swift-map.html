Chapter 10

MLX Swift and Open-Source Models

Memory Map

💡 The Core Insight

💡

Apple's Foundation Models are polished but locked. Open-source gives you choice, control, and wider device support.

Three runtimes — MLX Swift, llama.cpp, and ExecuTorch — let you run Llama, Mistral, Gemma, and more on iPhones. MLX is Apple-native with Metal GPU and unified memory. llama.cpp has the widest model support. The production pattern: detect device capabilities, pick the best available backend.

⚡ Three Runtimes

🍎
MLX Swift
Apple's open-source framework. Native Swift, Metal GPU, unified memory (no CPU-GPU copy). SPM package: mlx-swift-lm. Best for Apple-only projects.
🦙
llama.cpp
Community C/C++ engine. Widest model support, GGUF format, Metal acceleration. Use via llama.swift XCFramework. Cross-platform.
🔥
ExecuTorch
Meta's PyTorch runtime. Core ML delegate (ANE), MPS delegate (GPU), XNNPACK (CPU). Best when team already trains in PyTorch.

📌 Decision Guide

🍎
Use Foundation Models
Text summarization, rewriting, structured output with @Generable, tool calling. Zero setup, ships with OS.
🔌
Use Open-Source (MLX)
Domain-specific chatbot, older device support, multi-language, fine-tuned models, maximum privacy control.

📚 Key Terms

MLX
Apple's open-source array framework for Apple Silicon. Metal GPU acceleration, unified memory, first-class Swift support.
mlx-swift-lm
Swift package for loading and running LLMs/VLMs with MLX. Handles Hugging Face downloads, tokenization, and generation.
GGUF
GPT-Generated Unified Format. llama.cpp's model format with quantization variants (Q2_K through Q8_0).
ExecuTorch
Meta's on-device inference framework for PyTorch models. Core ML delegate for ANE, MPS delegate for GPU.
Unified Memory
Apple Silicon shares RAM between CPU and GPU — no data copying. MLX exploits this for faster tensor operations.
KV Cache
Key-value cache for attention layers. Grows with conversation length. A 1B model + 2048 tokens ≈ 200 MB extra RAM.
Jetsam
iOS memory termination. Exceed your RAM budget and the OS kills your app instantly with no warning.
mlx-community
Hugging Face organization hosting hundreds of pre-quantized models ready for MLX Swift.

✅ Production Checklist

📏
Limit Generation
Cap at 500-1000 tokens per response to prevent thermal runaway and excessive memory.
🌡️
Monitor Thermal State
Check ProcessInfo.thermalState before generation. Skip or defer if .serious/.critical.
💤
Unload When Backgrounded
Release model + call MLX.GPU.clearCache(). Respond to memory warnings immediately.
📲
Test on Lowest-Tier Device
If it works on iPhone SE (4 GB), it works everywhere. Don't only test on Pro models.

🤔 Quick Quiz

Test Yourself

What's the key advantage of MLX over llama.cpp on Apple devices?

MLX is built specifically for Apple Silicon and exploits unified memory — CPU and GPU share the same RAM with no data copying. It also has first-class Swift support and native Metal GPU acceleration, making it the most natural fit for iOS/macOS apps.

Which GGUF quantization variant is the sweet spot for iPhones?

Q4_K_M — 4-bit quantization with medium quality. Best balance of model size, quality, and memory usage. A 7B model at Q4_K_M needs ~4.5 GB of RAM, fitting on 8 GB devices.

When should you choose Foundation Models over open-source?

When you need zero-setup text generation, @Generable guided generation for structured output, built-in tool calling, or when targeting only iPhone 15 Pro+ users. Foundation Models ship with iOS 26 — no download, no model management.

What is jetsam and how do you avoid it?

Jetsam is iOS's memory termination mechanism — if your app exceeds its RAM budget, iOS kills it instantly with no warning. Avoid it by: choosing model size based on device RAM (leave ~2 GB for OS), unloading models when backgrounded, responding to memory warnings, and calling MLX.GPU.clearCache().

What's the runtime model selection pattern?

Check Foundation Models availability first (SystemLanguageModel.default). If unavailable, check device RAM to pick model size (3B for 8 GB, 1B for 6 GB). Fall back to cloud API for 4 GB devices. Abstract everything behind a TextGenerator protocol so your app code doesn't care which backend runs.

📋 Key Takeaways

If You Remember Nothing Else

  • Three runtimes: MLX Swift (Apple-native), llama.cpp (widest compatibility), ExecuTorch (PyTorch path).
  • MLX Swift: add mlx-swift-lm via SPM, load models from mlx-community on Hugging Face.
  • GGUF Q4_K_M is the sweet spot for phones. 7B at Q4 ≈ 4.5 GB RAM.
  • Foundation Models: zero setup, @Generable, tool calling. Open-source: model choice, fine-tuning, wider device support.
  • iPhone RAM: 4 GB (1B max), 6 GB (1-3B), 8 GB (up to 7B). Exceed budget = jetsam kill.
  • Always unload models when backgrounded. Call MLX.GPU.clearCache().
  • KV cache grows with conversation — limit context or periodically reset sessions.
  • Use a TextGenerator protocol to abstract backends. Runtime-select the best available model.
