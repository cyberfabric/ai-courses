🚧 Failure Modes & Responses

  1. Model Not Found

    ODR download failed or file corrupt. Show retry button, disable feature gracefully.

  2. Insufficient Memory

    Model too large for device. Fall back to smaller model or built-in framework.

  3. Inference Timeout

    Thermal throttling or overloaded device. Return cached result or show "try again."

  4. Unexpected Output

    Wrong input format or model mismatch. Log error, show fallback UI.

  5. OS Version Mismatch

    API not available. Use #available checks, never force-unwrap.

📚 Key Terms

On-Demand Resources (ODR)
Apple-hosted assets downloaded when needed via NSBundleResourceRequest. iOS manages storage automatically.
Background Assets
Framework for silent background downloads of large content. 200 GB Apple hosting included. Best for models > 200 MB.
Progressive Enhancement
Every device gets some AI features; better devices get more. Detect capabilities at runtime with DeviceCapability.
os_signpost
Apple's instrumentation API. Wrap model load and inference to measure timing in Instruments and MetricKit.
MetricKit / MXSignpostMetric
Production telemetry. OS aggregates your signpost data into daily payloads with histograms and percentiles.
Jetsam
iOS memory termination. Exceed budget = instant kill. Unload models on background and memory warnings.
A/B Testing Models
Ship two model variants to different user segments. Compare technical metrics (latency) AND product metrics (adoption).
Staged Rollout
Deploy new model to 5% → 25% → 100% of users. Monitor for regressions at each stage before expanding.

🤔 Quick Quiz

Test Yourself

When should you use On-Demand Resources vs Background Assets for a model?

Use On-Demand Resources for models 20-200 MB that are tied to specific features — downloaded when the user opens that feature. Use Background Assets for models > 200 MB that need to be ready before the user needs them — downloaded silently in the background, even before first app launch.

Why is on-device AI a "privacy superpower" for App Store compliance?

If data never leaves the device, you can mark "Data Not Collected" in App Store privacy labels, skip GDPR consent for AI processing, and avoid third-party AI data sharing disclosures. This simplifies compliance and builds user trust.

What's the "47 Safari tabs" test?

Test your AI feature after opening dozens of Safari tabs, playing music, and running navigation. This simulates real-world memory pressure. A model that loads fine in a clean simulator but crashes on a memory-pressured device is a production bug.

What six metrics should you track for AI features in production?

Model load time (< 500ms), inference latency p50 (< 30ms), inference latency p95 (< 100ms), memory footprint (< 200MB), inference success rate (> 99.5%), and feature adoption rate. Use os_signpost + MetricKit for measurement.

How should you roll out a new model version?

Staged rollout: 5% → 25% → 100% of users. Monitor technical metrics (latency, errors) and product metrics (adoption, satisfaction) at each stage. Always keep the previous version available as rollback. Version your model files (e.g., ClassifierV3_4bit.mlpackage).

📋 Key Takeaways

If You Remember Nothing Else

  • 50+ iPhone models in the wild. 4-8 GB RAM. Test on lowest-tier device under memory pressure.
  • Bundling: < 20 MB embed, 20-200 MB On-Demand Resources, > 200 MB Background Assets.
  • Progressive enhancement: detect capabilities at runtime, give every device the best it can handle.
  • Memory lifecycle: load on foreground, unload on background, emergency unload on memory warning.
  • On-device = "Data Not Collected" in App Store = privacy superpower for compliance.
  • Track: load time, p50/p95 latency, memory, success rate, adoption. Use os_signpost + MetricKit.
  • Staged rollout (5% → 25% → 100%) with rollback plan. Never update models without monitoring.
  • Handle every failure: model not found, OOM, inference timeout, wrong output, OS mismatch.
