Chapter 2

How ML Models Work on Apple Silicon

Just enough theory to be dangerous — models, tensors, inference, and where to find them

Opening the Black Box (Just a Crack)

In Chapter 1, we said on-device AI runs a "model" on the Neural Engine. But what is a model? If you've ever thought of ML models as mysterious black boxes that magically turn images into labels, this chapter cracks that box open just enough for you to work with them confidently.

You won't become a machine learning researcher. You'll become an iOS developer who understands what's happening when you call model.prediction() — and that's exactly what you need.

What Is an ML Model, Really?

Think of an ML model as a compiled function you didn't write. It takes inputs and returns outputs. You can't read the source code because there isn't any — the "logic" is millions of numerical weights that were learned from data during training.

Here's an analogy every iOS dev will get: imagine a function with no if statements, no switch cases, no hand-written rules. Instead, it has millions of floating-point numbers arranged in layers. When you feed it an input (an image, a sentence, a sound), it multiplies, adds, and transforms those numbers through the layers until an output comes out the other end.

Traditional code
func isHotdog(_ image: UIImage) -> Bool {
    // Handwritten pixel rules?
    // Color analysis?
    // Shape matching?
    // Good luck writing this.
}
You'd need to hand-code rules for every possible hotdog photo. Lighting, angles, backgrounds, bun types — impossible to enumerate.
ML model approach
let model = try HotdogClassifier()
let prediction = try model.prediction(image: photo)
// prediction.classLabel == "hotdog"
// prediction.classProbability["hotdog"] == 0.97
The model learned from 10,000+ hotdog photos. It handles every angle, lighting, and background automatically. You never wrote a single rule.

The model file itself — the .mlmodel or .mlpackage you drag into Xcode — contains those millions of learned weights plus a description of the layer architecture. Core ML reads this file, compiles it for your specific chip, and runs it on the Neural Engine.

The Inference Loop

Remember the universal pattern from Chapter 1? Let's expand it with what actually happens at each step:

1. Load

Core ML reads .mlmodel and compiles for the target chip

→

2. Preprocess

Convert input to the exact format the model expects

→

3. Predict

Neural Engine runs the forward pass through all layers

→

4. Postprocess

Convert raw numbers into human-readable results

In Swift, this loop looks like:

// 1. Load: Xcode auto-generates the MobileNetV2 class from .mlmodel
let config = MLModelConfiguration()
config.computeUnits = .all  // Let Core ML pick the best hardware
let model = try MobileNetV2(configuration: config)

// 2. Preprocess: resize image to 224x224, normalize pixels
let pixelBuffer = try resizeAndConvert(image, to: CGSize(width: 224, height: 224))

// 3. Predict: Neural Engine runs the forward pass
let prediction = try model.prediction(image: pixelBuffer)

// 4. Postprocess: read the result
print("It's a \(prediction.classLabel)!")  // "golden retriever"
print("Confidence: \(prediction.classProbability[prediction.classLabel]!)")

That's the entire loop. Every on-device AI feature you build follows this pattern. The details of steps 2 and 4 change depending on the model, but the shape never does.

Loading is the slow part. Compiling the model for your chip takes 100–500ms on first load. After that, each prediction runs in milliseconds. Load the model once at startup or when the view appears, then reuse it for every prediction.

Core ML Model Formats

You'll encounter two model formats in the wild:

.mlmodel (Legacy)
Single file containing weights + architecture
Smaller models, simpler distribution
Uses the Neural Network spec (older)
Still works fine for models under ~100MB. You'll see this format in many tutorials and Apple's model gallery.
.mlpackage (Modern)
Directory bundle with separate weight files
Supports larger models, ML Programs spec
Better optimization, the default for coremltools conversions
The current standard. Uses the ML Program representation, which supports more operations and compiles more efficiently on newer hardware.

When you drag either format into Xcode, the result is the same: Xcode compiles it into a .mlmodelc bundle and auto-generates a Swift class. You never interact with the raw format at runtime.

Core ML Model Compilation

At build time: Xcode compiles your .mlmodel/.mlpackage into .mlmodelc (an optimized binary). This is included in your app bundle.

At runtime: Core ML loads the .mlmodelc and may further optimize it for the specific device (A17 Pro vs. A19, etc.). This is why the first load takes longer.

Auto-generated class: Xcode creates a Swift class matching your model's name. A file called MobileNetV2.mlmodel generates a MobileNetV2 class with typed prediction() methods.

Tensors: The Language Models Speak

Here's the concept that trips up most iOS devs: ML models don't understand images, text, or audio. They understand tensors — multi-dimensional arrays of numbers. Your job is to convert your input into a tensor and convert the output tensor back into something useful.

But you already know tensors. You just call them by different names:

iOS Concept You Know Tensor Shape What It Represents
A single number Scalar (0D) A confidence score: 0.97
An array 1D tensor Probabilities for 1,000 classes: [0.01, 0.02, ..., 0.97]
A grayscale image 2D tensor (H × W) Each pixel is a brightness value: 224 × 224
A color image (UIImage) 3D tensor (H × W × C) Height × Width × RGB channels: 224 × 224 × 3
A batch of images 4D tensor (N × H × W × C) N images stacked together for batch inference

In Core ML, tensors are represented by two types:

  • MLMultiArray — The Foundation-era type. Works like a C-style multi-dimensional array. Verbose but universal.
  • MLShapedArray — The modern Swift type. Generic, subscriptable, and fully compatible with MLMultiArray. Prefer this in new code.
// Creating a tensor with MLShapedArray
let probabilities = MLShapedArray<Float>(
    scalars: [0.01, 0.02, 0.97],  // 3 class probabilities
    shape: [3]                      // 1D tensor with 3 elements
)

// Accessing values — just like a Swift array
let topScore = probabilities[2].scalar!  // 0.97

// Converting from MLMultiArray (common in model outputs)
let shaped = MLShapedArray<Float>(multiArray)
You rarely create tensors manually. For image models, Core ML and Vision handle tensor conversion automatically via CVPixelBuffer and VNImageRequestHandler. You'll mostly read output tensors, not create input ones.

Input Preprocessing: Why You Can't Just Throw a UIImage at a Model

Models are picky about their input. A model trained on 224×224 pixel images will crash or produce garbage if you feed it a 4032×3024 iPhone photo. Preprocessing bridges the gap between your app's data and what the model expects.

For image models, preprocessing typically means:

Resize

Scale to the model's expected dimensions (e.g., 224×224)

→

Convert

UIImage → CGImage → CVPixelBuffer

→

Normalize

Scale pixel values (0–255 → 0.0–1.0 or -1.0–1.0)

→

Orient

Handle EXIF orientation so the image isn't rotated

The good news: when you use VNCoreMLRequest (from the Vision framework), most of this is handled automatically. Vision resizes, converts, and normalizes for you. That's why most Core ML tutorials use Vision as the front door:

// Vision handles preprocessing automatically
let vnModel = try VNCoreMLModel(for: MobileNetV2().model)
let request = VNCoreMLRequest(model: vnModel)

// Feed any image — Vision resizes and normalizes it
let handler = VNImageRequestHandler(cgImage: anyImage, orientation: .up)
try handler.perform([request])
Orientation matters. iPhone photos have EXIF orientation metadata. If you skip orientation handling, your model might see a sideways or upside-down image. Always pass the correct CGImagePropertyOrientation to VNImageRequestHandler.

Output Postprocessing: Making Sense of Raw Numbers

The model's raw output is just numbers. An image classifier doesn't return "golden retriever" — it returns an array of 1,000 floating-point probabilities. You need to interpret them:

// Raw model output: MLMultiArray of shape [1, 1000]
// Index 207 = golden retriever, value = 0.97
// Index 208 = Labrador retriever, value = 0.02
// ... 998 other classes near 0.0

// Vision wraps this for you as VNClassificationObservation:
guard let results = request.results as? [VNClassificationObservation] else { return }

// Already sorted by confidence, mapped to labels
let top3 = results.prefix(3)
for result in top3 {
    print("\(result.identifier): \(result.confidence * 100)%")
}
// golden retriever: 97.1%
// Labrador retriever: 2.3%
// Irish setter: 0.4%

Different model types produce different outputs:

Model Type Raw Output What You Extract
Image classifier Array of probabilities Top class label + confidence
Object detector Bounding boxes + class scores CGRect coordinates + labels per object
Image segmenter Pixel-level class map Mask image for each segment
Text model Token IDs or embeddings Decoded text string

Model Metadata: The Built-In README

Every Core ML model carries metadata that tells you exactly how to use it. When you drag a model into Xcode and click on it, the model viewer shows you everything:

What Xcode's Model Viewer Shows

  • Input spec: name, type, and shape (e.g., "image: Color 224×224")
  • Output spec: name, type, and shape (e.g., "classLabel: String", "classProbability: Dictionary")
  • Class labels: the list of categories the model can recognize
  • Model class: the auto-generated Swift class name
  • Size: file size on disk
  • Author & license: who created it and usage terms
  • Compute units: which hardware backends are supported

This metadata is your contract with the model. If it says the input is a 224×224 color image, that's what you must provide. If the output is a dictionary of class probabilities, that's what you'll get back. No guessing required.

Common Model Architectures You'll Encounter

You don't need to understand neural network internals, but knowing the major architecture families helps you pick the right model for your task:

MobileNet
Image classification. Tiny (2–5 MB), fast (<5 ms), designed for mobile. Your go-to for "what is this?"
YOLOv8
Object detection with bounding boxes. Finds multiple objects and draws rectangles around each. 6–30 MB.
DeepLab
Image segmentation. Labels every pixel — background vs. person vs. object. Powers portrait mode effects.
BERT
Text understanding. Sentiment, entity extraction, question answering. Available in mobile-optimized variants (DistilBERT).
EfficientNet
Image classification, larger and more accurate than MobileNet. Good when accuracy trumps speed (20–80 MB).
Whisper
Speech-to-text transcription. Multiple size variants. Convertible to Core ML with coremltools.

The Size-Accuracy Tradeoff

Here's the fundamental tension on mobile: bigger models are more accurate, but they use more storage, more memory, and more battery. The table below shows real tradeoffs for image classification:

Image Classification Models: Size vs. Accuracy on iPhone 16 Pro
Model Size Top-1 Accuracy Inference (ANE) Memory
MobileNetV2 3.4 MB 71.8% ~3 ms ~15 MB
MobileNetV3-Large 5.4 MB 75.2% ~4 ms ~20 MB
EfficientNet-B0 20 MB 77.1% ~8 ms ~45 MB
EfficientNet-B4 74 MB 82.9% ~25 ms ~150 MB

Notice: MobileNetV3-Large gives 75% accuracy in 5 MB and 4 ms. EfficientNet-B4 gives 83% accuracy but costs 74 MB and 25 ms. For most iOS features, MobileNet's speed and size win. Use larger models only when accuracy is critical and you can afford the resources.

Start small, measure, then upgrade. Begin with MobileNetV3. If accuracy isn't good enough for your specific categories, move up to EfficientNet-B0. Only reach for B4+ if you've exhausted optimization options (Chapter 8 covers quantization and palettization to shrink larger models).

This is your model's API documentation. The input spec tells you what to provide. The output spec tells you what you'll get back. The auto-generated Swift class wraps all of this in type-safe methods.

Xcode auto-generates a Swift class with typed prediction() methods and a typed output struct. If you pass the wrong type, the compiler catches it. No runtime surprises.

How Apple Silicon Runs Your Model

When you call prediction(), Core ML decides where to run the model based on your MLModelConfiguration.computeUnits setting:

Compute Units Where It Runs When to Use
.all (default) Core ML picks the best option per layer Almost always. Let Core ML decide.
.cpuAndNeuralEngine Neural Engine + CPU fallback When you want to leave the GPU free for rendering
.cpuAndGPU GPU via Metal + CPU fallback When the Neural Engine doesn't support your model's ops
.cpuOnly CPU only Debugging, or when you need deterministic results
Use .all and move on. Core ML's scheduler is remarkably good at splitting model layers across hardware. It can run some layers on the Neural Engine and fall back to GPU or CPU for unsupported operations — all automatically. Only override this when benchmarking (Chapter 7).

The Unified Memory Advantage

On traditional PCs, the CPU, GPU, and ML accelerator each have separate memory. Moving data between them costs time. Apple Silicon is different: unified memory architecture means the CPU, GPU, and Neural Engine all share the same physical memory pool.

What this means for you: no expensive data copies between processors. The Neural Engine starts working immediately after the CPU prepares data. GPU rendering and ML inference share the same buffers. Core ML splits a model across CPU + GPU + Neural Engine with minimal overhead. This is why the same model runs faster on Apple Silicon than on competing chips with similar TOPS ratings.

Try It: Explore a Real Core ML Model

Exercise: Inspect MobileNetV2 in Xcode

  1. Download MobileNetV2 from Apple's Core ML Models page (developer.apple.com/machine-learning/models). It's about 3.4 MB.
  2. Create a new Xcode project (iOS App, SwiftUI). Name it "ModelExplorer."
  3. Drag MobileNetV2.mlmodel into the project navigator. Make sure "Copy items if needed" and your target are selected.
  4. Click the model file in the navigator. Xcode's model viewer opens. Write down:
    • Input name and shape
    • Output names and types
    • Number of class labels
    • File size
  5. Open the Predictions tab in the model viewer. Drag a photo from your desktop onto the preview area. Xcode runs inference right there and shows the top predictions.
  6. Check the auto-generated code: build the project (Cmd+B), then Cmd-click on MobileNetV2 in any Swift file to jump to the generated interface. Note the prediction(image:) method signature.

Bonus: Download a second model (like SqueezeNet or Resnet50) and compare their input/output specs, sizes, and prediction results on the same photo.
