Chapter 6

Foundation Models and Apple Intelligence

Your App's Own LLM — No Cloud, No API Key, No Bill

Every framework we've used so far — Vision, Natural Language, SoundAnalysis — does one specific thing well. But what if you want your app to think? Summarize a document, answer a question, rewrite text in a different tone, or decide what to do next based on free-form input?

That's what generative AI is for. And with iOS 26, Apple gives you direct access to the same on-device language model that powers Apple Intelligence — a ~3 billion parameter LLM running entirely on the Neural Engine. No cloud. No API key. No per-token billing. Just import FoundationModels and start generating.

What Is the Foundation Models Framework?

The Foundation Models framework is Apple's Swift API for on-device generative AI. It wraps the language model at the heart of Apple Intelligence — the same model that powers Writing Tools, Mail summaries, and Siri's intelligence features — and exposes it to your app through a clean, type-safe Swift interface.

Your First Generation: Hello, LLM

The simplest possible use of Foundation Models is three lines of code:

import FoundationModels

let session = LanguageModelSession()
let response = try await session.respond(to: "Tell me a fun fact about the iPhone.")
print(response.content)  // "The iPhone's first prototype was..."

That's it. No model loading, no configuration, no preprocessing. The framework handles everything. But this is just plain text output — the real power comes from guided generation.

Checking Device Compatibility

Not every iPhone can run Foundation Models. Before using the framework, check availability at runtime:

import FoundationModels

let availability = SystemLanguageModel.default.availability

switch availability {
case .available:
    // Good to go — start generating
    break
case .unavailable(.deviceNotEligible):
    // Hardware too old (pre-iPhone 15 Pro)
    showFallbackUI()
case .unavailable(.appleIntelligenceNotEnabled):
    // User hasn't enabled Apple Intelligence in Settings
    promptToEnableAI()
case .unavailable(.modelNotReady):
    // Model is still downloading or updating
    showLoadingState()
default:
    showFallbackUI()
}
Always check availability. Foundation Models requires both compatible hardware (iPhone 15 Pro or newer) AND Apple Intelligence enabled in Settings. If either condition isn't met, your code will throw. Design your feature with a graceful fallback for unsupported devices.

Guided Generation: Typed Output with @Generable

Here's the killer feature. Instead of getting back raw text that you have to parse (and pray it's valid JSON), you define a Swift struct and the model guarantees its output matches your type. No parsing errors. No hallucinated structure. Just type-safe Swift objects.

import FoundationModels

@Generable
struct MovieRecommendation {
    @Guide(description: "The movie title")
    var title: String

    @Guide(description: "Release year", .range(1920...2026))
    var year: Int

    @Guide(description: "A one-sentence reason to watch it")
    var reason: String

    @Guide(description: "Rating out of 10", .range(1...10))
    var rating: Int
}

Now use it in a session:

let session = LanguageModelSession()
let recommendation = try await session.respond(
    to: "Recommend a sci-fi movie for someone who loved Interstellar.",
    generating: MovieRecommendation.self
)
print(recommendation.title)   // "Arrival"
print(recommendation.year)    // 2016
print(recommendation.rating)  // 9

The @Generable macro generates a schema at compile time. The model uses constrained decoding to guarantee the output matches your struct — the right types, the right ranges, the right number of items. This isn't "parse and hope" — it's structurally guaranteed.

What types can @Generable handle? String, Int, Float, Double, Bool, Optional, Array, and enums with String raw values. You can also nest @Generable structs inside other @Generable structs for complex output shapes.

The @Guide Macro

@Guide gives the model instructions about each property. Think of it as documentation that the model actually reads:

@Generable
struct QuizQuestion {
    @Guide(description: "The quiz question text")
    var text: String

    @Guide(description: "Four possible answers", .count(4))
    var choices: [String]

    @Guide(description: "The correct answer, must be one of the choices")
    var answer: String

    @Guide(description: "Why this answer is correct")
    var explanation: String
}

The .count(4) constraint guarantees exactly 4 choices. The .range(1...10) from the earlier example guarantees an integer in that range. No validation code needed — the model can't produce invalid output.

The mental model: Traditional LLM output = "here's some text, good luck parsing it." Guided generation = "here's a filled-in Swift struct, guaranteed to compile." It's the difference between getting a handwritten letter and getting a filled-in form.

Streaming Responses

For a responsive UI, you don't want to wait for the entire response. Stream partial results as they're generated:

let session = LanguageModelSession()

// Stream plain text
for try await partial in session.streamResponse(to: "Summarize the iOS 26 keynote.") {
    // partial.content grows with each token
    updateUI(text: partial.content)
}

// Stream typed output — partial snapshots with Optional properties
for try await snapshot in session.streamResponse(
    to: "Recommend a sci-fi movie.",
    generating: MovieRecommendation.self
) {
    // Properties fill in one by one as the model generates
    if let title = snapshot.title { showTitle(title) }
    if let reason = snapshot.reason { showReason(reason) }
}
Streaming sends snapshots, not deltas. Unlike ChatGPT's token-by-token streaming, Foundation Models streams partially-filled structs. Optional properties become non-nil as the model generates them. This is much easier to work with in SwiftUI — just bind to the snapshot.

Multi-Turn Sessions

A LanguageModelSession maintains conversation context. Each call to respond(to:) builds on the previous exchanges:

let session = LanguageModelSession(instructions: """
    You are a helpful cooking assistant. Give brief, practical answers.
    """)

let first = try await session.respond(to: "How do I make pasta al dente?")
// "Boil salted water, cook 1 minute less than package directions..."

let followUp = try await session.respond(to: "What sauce goes best with it?")
// Knows we're talking about pasta from the previous turn

The instructions parameter is your system prompt — it sets the model's personality and constraints for the entire session. Keep it short and specific. This is a ~3B model, not GPT-4 — shorter, more focused instructions work better.

Tool Calling: Give the Model Superpowers

Tool calling lets the model invoke functions in your app. The model decides when to call a tool and what arguments to pass. You define the tools; the model orchestrates them.

Think of it like giving an assistant access to your phone's apps: "Check the weather" calls one tool, "look up a recipe" calls another. The assistant decides which tool to use based on the conversation.

import FoundationModels

@Generable
struct WeatherInfo {
    var temperature: Int
    var condition: String
}

struct WeatherTool: Tool {
    let name = "getWeather"
    let description = "Gets the current weather for a city"

    @Generable
    struct Arguments {
        @Guide(description: "The city name")
        var city: String
    }

    func call(arguments: Arguments) async throws -> String {
        // Call your weather API or local data
        let weather = await fetchWeather(for: arguments.city)
        return "It's \(weather.temp)°F and \(weather.condition) in \(arguments.city)."
    }
}

// Create session with tools
let session = LanguageModelSession(
    instructions: "You help users plan their day based on weather.",
    tools: [WeatherTool()]
)

let response = try await session.respond(
    to: "Should I bring an umbrella in San Francisco today?"
)
// Model calls WeatherTool, gets result, then answers:
// "Yes! It's 55°F and rainy in San Francisco today. Definitely bring an umbrella."
Tool calling flow: You ask a question → the model decides it needs weather data → it calls your WeatherTool → your tool returns a result → the model incorporates that result into its answer. All of this happens in a single respond(to:) call.

Writing Tools Integration

Apple's system-wide Writing Tools (Rewrite, Proofread, Summarize) work automatically with any standard text view. If you use TextField, TextEditor, or UITextView, your users already have access to Writing Tools — no code needed.

But you can customize the behavior:

TextEditor(text: $userText)
    .writingToolsBehavior(.complete)  // Full Writing Tools support
    // .limited — only proofreading
    // .disabled — opt out entirely

For custom text views that don't use the standard components, adopt the WritingToolsCoordinator protocol to integrate Writing Tools into your own text rendering.

Image Playground and Genmoji

Foundation Models isn't just text. Apple also provides on-device image generation through Image Playground:

import ImagePlayground

// Present the Image Playground sheet
.imagePlaygroundSheet(isPresented: $showPlayground,
                      concept: "A cat wearing a space helmet",
                      onCompletion: { url in
    // url points to the generated image
    generatedImage = url
})

For programmatic image generation without the UI sheet, use ImageCreator:

import ImagePlayground

let creator = ImageCreator()
let image = try await creator.generateImage(
    ImageCreator.Concept.text("A peaceful mountain landscape at sunset")
)
// image is a CIImage you can display or save
Image generation is slow. Unlike text generation (~30 tokens/sec), image generation can take 10–30 seconds depending on the device and complexity. Always show a progress indicator and run generation on a background task. Never block the UI.

On-Device vs Private Cloud Compute

Most Foundation Models requests run entirely on-device. But for tasks that exceed the on-device model's capabilities, Apple can route requests to Private Cloud Compute — Apple's secure cloud infrastructure where your data is processed in encrypted memory, never stored, and never accessible to Apple.

On-Device (Default)

  • Runs on Neural Engine
  • ~3B parameter model
  • Zero latency from network
  • Works offline
  • Data never leaves iPhone

Private Cloud Compute

  • Runs on Apple Silicon servers
  • Larger model, more capable
  • Requires network connection
  • Encrypted, never stored
  • Publicly auditable software

You don't choose which mode runs — the system decides based on the complexity of the request. Simple tasks (summaries, rewrites) stay on-device. Complex tasks (multi-step reasoning, large context) may route to the cloud. Users can opt out of cloud processing in Settings.

Prompt Engineering for On-Device Models

The on-device model is ~3B parameters — much smaller than GPT-4 or Claude. Prompts that work on large cloud models may need adjustment:

Good: Short, Focused Prompts

// Clear, single task
"Summarize this email in one sentence."

// Specific format via @Generable
"Generate 3 quiz questions about Swift."

// Focused instruction
session = LanguageModelSession(
  instructions: "You summarize text. Be concise."
)

Bad: Overloaded Prompts

// Too many instructions
"Summarize this, then analyze sentiment,
 then suggest improvements, then translate
 to Spanish, keeping the tone formal but
 friendly..."

// Ambiguous personality
"Be creative but accurate, concise but
 thorough, professional but casual..."
The golden rule for on-device prompts: One task per prompt. Use guided generation (@Generable) instead of asking for structured output in the prompt text. Let the type system do the work, not the prompt.

At 30–40 tokens per second, the on-device model generates about 1–2 sentences per second. That's fast enough for real-time streaming in a chat UI, but you should always stream responses to keep the UI responsive. Never call respond(to:) and wait for the full result before showing anything to the user.

Building a Complete Feature

Let's put it all together. Here's a text transformation feature using guided generation and streaming:

import FoundationModels
import SwiftUI

@Generable
struct TextTransformation {
    @Guide(description: "The transformed text")
    var result: String

    @Guide(description: "Confidence from 1-10", .range(1...10))
    var confidence: Int

    @Guide(description: "What was changed and why")
    var explanation: String
}

@Observable
class TextTransformer {
    var output: TextTransformation?
    var isGenerating = false

    func transform(_ text: String, style: String) async {
        isGenerating = true
        defer { isGenerating = false }

        let session = LanguageModelSession(
            instructions: "You transform text as requested. Be faithful to the original meaning."
        )

        do {
            for try await snapshot in session.streamResponse(
                to: "Rewrite this text in a \(style) style: \(text)",
                generating: TextTransformation.self
            ) {
                output = snapshot
            }
        } catch {
            print("Generation failed: \(error)")
        }
    }
}

Exercise: Build an AI Writing Assistant

Hands-On: Smart Text Transformer

Build a SwiftUI app that uses Foundation Models with guided generation for text transformations:

  1. Text input — A TextEditor where users paste or type text
  2. Transformation picker — Segmented control with 4 options: Summarize, Formalize, Simplify, Expand
  3. Guided output — Define a @Generable struct with result (transformed text), confidence (1–10), and wordCount (Int)
  4. Streaming display — Use streamResponse to show the result filling in progressively
  5. Availability check — Check SystemLanguageModel.default.availability on appear. Show a message if Foundation Models isn't available.

Architecture Tips:

  • Create a new LanguageModelSession for each transformation (don't reuse sessions across unrelated tasks)
  • Show the confidence score as a colored badge (green ≥ 7, yellow 4–6, red ≤ 3)
  • Add a "Copy Result" button that copies only the transformed text
  • Disable the transform button while isGenerating is true

Stretch Goal:

  • Add a multi-turn "Refine" button that sends "Make it shorter" or "Make it more casual" as follow-up prompts in the same session
