Step 1: Setting Up the Xcode Project

Create a new Xcode project: iOS → App, SwiftUI interface, Swift language. Name it LiveClassifier. Select your physical iPhone as the run destination — the Simulator can't access the camera.

Before we write any code, we need a model. You have two options:

Option A: Use a pre-built model
Download MobileNetV2.mlmodel from Apple's Core ML Models page
Fastest path. Drag it into Xcode and skip to Step 3. Great for your first time.
Option B: Convert from PyTorch
Use coremltools to convert a torchvision model yourself
Teaches you the conversion workflow. Essential for custom models later. We cover this in Step 2.

Step 2: Converting a PyTorch Model with coremltools

Most ML models in the wild are PyTorch or TensorFlow. Apple's coremltools library converts them to Core ML format. You'll need Python installed (the one from Homebrew or python.org is fine).

# Install coremltools (one-time setup)
pip install coremltools torch torchvision

Now create a Python script called convert_mobilenet.py:

import torch
import torchvision
import coremltools as ct

# Load a pre-trained MobileNetV2 from torchvision
model = torchvision.models.mobilenet_v2(weights="IMAGENET1K_V1")
model.eval()

# Trace the model with a sample input
example = torch.rand(1, 3, 224, 224)
traced = torch.jit.trace(model, example)

# Convert to Core ML with image input type
mlmodel = ct.convert(
    traced,
    inputs=[ct.ImageType(
        name="image",
        shape=(1, 3, 224, 224),
        scale=1.0 / (0.226 * 255.0),
        bias=[-0.485/0.229, -0.456/0.224, -0.406/0.225],
        color_layout=ct.colorlayout.RGB
    )],
    minimum_deployment_target=ct.target.iOS16,
    classifier_config=ct.ClassifierConfig("imagenet_classes.txt"),
)

# Save as .mlpackage
mlmodel.save("MobileNetV2.mlpackage")
print("Done! Model saved.")
What's that scale and bias? ImageNet models expect normalized pixel values. The scale and bias tell Core ML how to transform raw pixel values (0–255) into the range the model was trained on. These specific numbers are the ImageNet normalization constants that PyTorch's torchvision uses.

Run the script: python convert_mobilenet.py. You'll get a MobileNetV2.mlpackage directory. Drag it into your Xcode project navigator. Make sure "Copy items if needed" is checked and your app target is selected.

Classifier labels file. The classifier_config parameter needs a text file with one class label per line (1,000 lines for ImageNet). You can find imagenet_classes.txt in many GitHub repos or generate it from torchvision. If you skip this, the model will output indices instead of readable labels.

Step 3: Loading the Model in Swift

When you drag the model into Xcode, it auto-generates a Swift class. Build the project (Cmd+B) so Xcode generates it, then you can use it:

import CoreML
import Vision

class ImageClassifier: ObservableObject {
    @Published var topLabel = ""
    @Published var confidence: Float = 0.0

    private let vnModel: VNCoreMLModel

    init() {
        let config = MLModelConfiguration()
        config.computeUnits = .all
        let coreMLModel = try! MobileNetV2(configuration: config).model
        vnModel = try! VNCoreMLModel(for: coreMLModel)
    }
}

We wrap the Core ML model in VNCoreMLModel so we can use it with Vision's request pipeline. As we learned in Chapter 2, Vision handles all the image preprocessing — resizing, normalizing, orientation — so we don't have to.

MLModelConfiguration

Key property: computeUnits controls which hardware runs the model.

  • .all — Core ML picks the best option (Neural Engine, GPU, or CPU). Use this.
  • .cpuAndNeuralEngine — Skips GPU. Use when GPU is busy with rendering.
  • .cpuOnly — Slowest. Only for debugging.

Available since: iOS 12. The auto-generated model class accepts it in its initializer.

Step 4: The VNCoreMLRequest Workflow

Now add the classification method to our ImageClassifier class:

func classify(pixelBuffer: CVPixelBuffer) {
    let request = VNCoreMLRequest(model: vnModel) { [weak self] request, error in
        guard let results = request.results as? [VNClassificationObservation],
              let top = results.first else { return }

        DispatchQueue.main.async {
            self?.topLabel = top.identifier
            self?.confidence = top.confidence
        }
    }

    // imageCropAndScaleOption controls how Vision resizes the image
    request.imageCropAndScaleOption = .centerCrop

    let handler = VNImageRequestHandler(cvPixelBuffer: pixelBuffer)
    try? handler.perform([request])
}

That's the entire inference pipeline. Let's break down what happens:

  1. VNCoreMLRequest wraps your model and specifies a completion handler
  2. The completion handler receives VNClassificationObservation results, already sorted by confidence
  3. imageCropAndScaleOption = .centerCrop tells Vision to center-crop the image to the model's expected aspect ratio
  4. VNImageRequestHandler takes the raw pixel buffer and runs the request
  5. Results arrive on a background thread, so we dispatch to main for UI updates

Step 5: Camera Integration with AVFoundation

Now we need a live camera feed. This is the most code-heavy part, but the pattern is standard iOS camera work:

import AVFoundation

class CameraManager: NSObject, ObservableObject {
    let session = AVCaptureSession()
    var onFrame: ((CVPixelBuffer) -> Void)?

    func start() {
        session.sessionPreset = .hd1280x720

        guard let device = AVCaptureDevice.default(
            .builtInWideAngleCamera, for: .video, position: .back
        ),
        let input = try? AVCaptureDeviceInput(device: device)
        else { return }

        session.addInput(input)

        let output = AVCaptureVideoDataOutput()
        output.setSampleBufferDelegate(self, queue: DispatchQueue(label: "camera"))
        output.alwaysDiscardsLateVideoFrames = true
        session.addOutput(output)

        DispatchQueue.global(qos: .userInitiated).async {
            self.session.startRunning()
        }
    }
}

extension CameraManager: AVCaptureVideoDataOutputSampleBufferDelegate {
    func captureOutput(_ output: AVCaptureOutput,
                       didOutput sampleBuffer: CMSampleBuffer,
                       from connection: AVCaptureConnection) {
        guard let pixelBuffer = CMSampleBufferGetImageBuffer(sampleBuffer) else { return }
        onFrame?(pixelBuffer)
    }
}

Key decisions in this code:

  • .hd1280x720 — High enough quality for classification, low enough for smooth performance. Don't use 4K for ML inference.
  • alwaysDiscardsLateVideoFrames = true — If inference is slower than the frame rate, drop frames instead of queuing them. This prevents memory buildup.
  • Background queue — Camera delivery happens on a dedicated serial queue. Never block the main thread with camera frames.
Camera permission required. Add NSCameraUsageDescription to your Info.plist with a message like "Camera is used for real-time image classification." Without this, your app will crash on launch.

Step 6: Displaying the Camera Preview in SwiftUI

SwiftUI doesn't have a native camera view, so we bridge AVCaptureVideoPreviewLayer using UIViewRepresentable:

import SwiftUI

struct CameraPreview: UIViewRepresentable {
    let session: AVCaptureSession

    func makeUIView(context: Context) -> UIView {
        let view = UIView(frame: .zero)
        let layer = AVCaptureVideoPreviewLayer(session: session)
        layer.videoGravity = .resizeAspectFill
        view.layer.addSublayer(layer)
        return view
    }

    func updateUIView(_ uiView: UIView, context: Context) {
        if let layer = uiView.layer.sublayers?.first as? AVCaptureVideoPreviewLayer {
            layer.frame = uiView.bounds
        }
    }
}

Step 7: Wiring It All Together

Now let's build the main view that combines camera preview with classification results:

struct ContentView: View {
    @StateObject private var camera = CameraManager()
    @StateObject private var classifier = ImageClassifier()

    var body: some View {
        ZStack(alignment: .bottom) {
            CameraPreview(session: camera.session)
                .ignoresSafeArea()

            VStack(spacing: 8) {
                Text(classifier.topLabel)
                    .font(.title2.bold())
                Text("\(Int(classifier.confidence * 100))%")
                    .font(.title3)
                    .foregroundStyle(.secondary)
            }
            .padding()
            .background(.ultraThinMaterial, in: RoundedRectangle(cornerRadius: 16))
            .padding(.bottom, 40)
        }
        .onAppear {
            camera.onFrame = { pixelBuffer in
                classifier.classify(pixelBuffer: pixelBuffer)
            }
            camera.start()
        }
    }
}

Run it on your iPhone. Point the camera at objects around you. You should see labels and confidence percentages updating in real time at the bottom of the screen.

What just happened?

Every camera frame flows through: AVCaptureSession (captures frame) → CVPixelBuffer (raw pixels) → VNImageRequestHandler (preprocesses) → VNCoreMLRequest (Neural Engine inference) → VNClassificationObservation (result) → SwiftUI Text (display). All on-device. All in milliseconds.

Step 8: Threading Strategy

There's a subtle problem with the code above: we're running inference on the camera's delivery queue, which can cause dropped frames if inference is slow. A better approach uses a dedicated inference queue with throttling:

class ImageClassifier: ObservableObject {
    @Published var topLabel = ""
    @Published var confidence: Float = 0.0

    private let vnModel: VNCoreMLModel
    private let inferenceQueue = DispatchQueue(label: "inference")
    private var isProcessing = false

    func classify(pixelBuffer: CVPixelBuffer) {
        // Skip if still processing previous frame
        guard !isProcessing else { return }
        isProcessing = true

        inferenceQueue.async { [weak self] in
            defer { self?.isProcessing = false }
            // ... VNCoreMLRequest code from Step 4 ...
        }
    }
}
Why throttle? The camera delivers 30 frames per second. MobileNetV2 on the Neural Engine runs in ~4ms, so it can keep up. But larger models or CPU-only inference might take 50–100ms. The isProcessing guard ensures we never queue up a backlog of frames. We always classify the most recent frame, not one from 10 frames ago.

Common Pitfalls and Fixes

Here are the issues that trip up every developer building their first Core ML camera app:

Problem Cause Fix
Crash on launch Missing NSCameraUsageDescription in Info.plist Add the key with a descriptive string
"Model not found" crash Model not added to app target Select the .mlmodel file → File Inspector → check your target
Wrong or random predictions Image orientation not handled Pass orientation: .right to VNImageRequestHandler (camera frames are landscape-right)
Dropped frames / stuttering Inference blocking camera queue Use a separate inference queue with isProcessing throttle
Labels are numbers, not words Missing classifier labels during conversion Re-convert with classifier_config parameter in coremltools
App killed in background Camera session not stopped Call session.stopRunning() in onDisappear
The orientation fix is the one everyone forgets. iPhone camera frames arrive in .right orientation (landscape-right). If you don't tell Vision about this, the model sees a rotated image and predictions are wrong. Always pass orientation explicitly: VNImageRequestHandler(cvPixelBuffer: buffer, orientation: .right)

Adding Top-5 Predictions

A single label isn't always enough. Let's display the top 5 predictions to give users more context:

struct Prediction: Identifiable {
    let id = UUID()
    let label: String
    let confidence: Float
}

// In the completion handler:
let predictions = results.prefix(5).map {
    Prediction(label: $0.identifier, confidence: $0.confidence)
}

// In the SwiftUI view:
ForEach(classifier.predictions) { p in
    HStack {
        Text(p.label.prefix(25))
        Spacer()
        Text("\(Int(p.confidence * 100))%")
            .foregroundStyle(.secondary)
    }
    .font(.subheadline)
}

The Complete Architecture

Here's how all the pieces fit together:

LiveClassifier App Architecture
Component Role Thread
CameraManager Captures frames via AVCaptureSession Camera queue (serial)
ImageClassifier Runs VNCoreMLRequest on each frame Inference queue (serial)
CameraPreview Displays live camera via AVCaptureVideoPreviewLayer Main (UIKit layer)
ContentView Overlays classification results on camera Main (SwiftUI)

Three threads working in parallel: camera captures, inference processes, UI displays. They communicate through @Published properties and closures. No thread blocks another.
