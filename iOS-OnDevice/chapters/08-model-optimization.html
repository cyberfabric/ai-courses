Chapter 8

Model Optimization — Making Models iPhone-Ready

Smaller, Faster, Almost as Accurate

You've found the perfect model. It classifies images with 95% accuracy, detects 80 object categories, and runs beautifully — on your MacBook Pro with 36 GB of RAM. Then you try to run it on an iPhone. The 500 MB model takes 3 seconds to load, eats 1.2 GB of memory, and gets your app killed by the system within a minute.

This chapter fixes that. We'll take models that are too big or too slow for mobile and compress them using three techniques: quantization (reducing numerical precision), palettization (Apple's unique lookup-table compression), and pruning (removing near-zero weights). The tool is coremltools, Apple's Python package for Core ML model conversion and optimization.

Why Optimization Matters

Here are realistic constraints for shipping a model in an iOS app:

< 50 MB
Model Size

Ideally under 20 MB. Over 50 MB increases download time and App Store friction.

< 30 ms
Inference Latency

On the Neural Engine. Under 33ms = real-time at 30fps.

< 200 MB
Runtime Memory

Total memory footprint during inference. Over 200 MB risks jetsam termination on older devices.

A standard float32 model stores every weight as a 32-bit floating point number. A model with 25 million parameters (typical for MobileNet-scale) takes 100 MB at float32. By reducing precision, we can shrink that dramatically while keeping almost all the accuracy.

Quantization: Reducing Numerical Precision

Quantization is the most common optimization technique. The idea is simple: instead of storing every weight as a 32-bit float (4 bytes), store it as a smaller number — 16-bit float (2 bytes), 8-bit integer (1 byte), or even 4-bit integer (0.5 bytes).

Think of it like image compression: a RAW photo is huge and perfectly detailed. A JPEG is much smaller and looks nearly identical. You're trading a tiny bit of precision for a massive size reduction.

Precision Levels

  • Float32 — full precision, 4 bytes/weight. Baseline.
  • Float16 — half precision, 2 bytes. 2x smaller, negligible accuracy loss. The easy win.
  • Int8 — 8-bit integer, 1 byte. 4x smaller, < 1% accuracy loss typical.
  • Int4 — 4-bit integer, 0.5 bytes. 8x smaller, 1–3% accuracy loss. Aggressive.

When Quantization Hurts

  • Models with very small weight ranges (sensitive to precision)
  • Regression tasks where small output differences matter
  • Very small models (< 1M params) — less redundancy to exploit
  • Already-compressed models (double compression degrades quality)

Post-Training Quantization with coremltools

The simplest approach — take an already-converted Core ML model and quantize it. No training data needed, takes seconds:

import coremltools as ct
import coremltools.optimize as cto

# Load your Core ML model
model = ct.models.MLModel("MyModel.mlpackage")

# Int8 weight quantization (4x smaller)
op_config = cto.coreml.OpLinearQuantizerConfig(
    mode="linear_symmetric",
    dtype="int8"
)
config = cto.coreml.OptimizationConfig(global_config=op_config)
quantized_model = cto.coreml.linear_quantize_weights(model, config)
quantized_model.save("MyModel_int8.mlpackage")

# Int4 weight quantization (8x smaller, more aggressive)
op_config_4bit = cto.coreml.OpLinearQuantizerConfig(
    mode="linear_symmetric",
    dtype="int4",
    granularity="per_block",
    block_size=32
)
config_4bit = cto.coreml.OptimizationConfig(global_config=op_config_4bit)
quantized_4bit = cto.coreml.linear_quantize_weights(model, config_4bit)
quantized_4bit.save("MyModel_int4.mlpackage")
Start with float16. If your model is currently float32, converting to float16 is essentially free accuracy-wise and halves the model size. coremltools does this automatically during ct.convert() — just pass compute_precision=ct.precision.FLOAT16.

Palettization: Apple's Unique Compression

Palettization is Apple's secret weapon for model compression. Instead of storing every weight individually, it clusters weights into a palette (lookup table) of 2N values, then stores just the index into the palette for each weight.

Think of it like indexed color in GIF images. A 24-bit color photo has millions of unique colors. An indexed-color GIF uses a palette of 256 colors and stores just the palette index for each pixel. Massive compression, and it still looks recognizable.

Palettization Bit Depths
Bits Palette Size Compression vs Float16 Typical Accuracy Loss
8-bit 256 values 2x < 0.5%
6-bit 64 values 2.7x < 1%
4-bit 16 values 4x 1–2%
3-bit 8 values 5.3x 2–4%
2-bit 4 values 8x 3–8%

Palettization with coremltools

import coremltools as ct
import coremltools.optimize as cto

model = ct.models.MLModel("MyModel.mlpackage")

# 4-bit palettization with k-means clustering
op_config = cto.coreml.OpPalettizerConfig(
    mode="kmeans",
    nbits=4,
    granularity="per_grouped_channel",
    group_size=16
)
config = cto.coreml.OptimizationConfig(global_config=op_config)
palettized_model = cto.coreml.palettize_weights(model, config)
palettized_model.save("MyModel_palette4.mlpackage")
Palettization vs Quantization — which is better? For Core ML models, palettization often gives a better accuracy-size tradeoff than quantization at the same compression ratio. This is because the k-means clustering adapts to the actual weight distribution of your model, while quantization uses a fixed linear mapping. Try both and compare.

Pruning: Removing Unimportant Weights

Pruning sets small, near-zero weights to exactly zero. A model with 50% sparsity has half its weights set to zero — which means those weights can be stored more compactly and the Neural Engine can skip them during computation.

import coremltools as ct
import coremltools.optimize as cto

model = ct.models.MLModel("MyModel.mlpackage")

# Magnitude pruning — zero out the smallest 50% of weights
op_config = cto.coreml.OpMagnitudePrunerConfig(
    target_sparsity=0.5
)
config = cto.coreml.OptimizationConfig(global_config=op_config)
pruned_model = cto.coreml.prune_weights(model, config)
pruned_model.save("MyModel_pruned50.mlpackage")
Pruning shines when combined. Pruning alone gives modest size reduction (the zeros still take space unless the format supports sparse storage). The real win is combining pruning with palettization or quantization — the zeros compress extremely well, and the Neural Engine skips zero-weight operations for faster inference.

Combining Techniques: Joint Compression

The biggest wins come from stacking techniques. A common recipe:

  1. Prune 50% of weights (set smallest to zero)
  2. Palettize remaining weights to 4-bit (16-value palette)
  3. Result: much smaller model, often with < 2% accuracy loss
# Joint pruning + palettization
pruner_config = cto.coreml.OpMagnitudePrunerConfig(target_sparsity=0.5)
palette_config = cto.coreml.OpPalettizerConfig(mode="kmeans", nbits=4)

config = cto.coreml.OptimizationConfig(
    global_config=[pruner_config, palette_config]
)

# Apply both in sequence
compressed = cto.coreml.prune_weights(model, config)
compressed = cto.coreml.palettize_weights(compressed, config)
compressed.save("MyModel_pruned_palette4.mlpackage")

Measuring Accuracy Impact

Optimization without accuracy measurement is reckless. You need to know: how much accuracy did I lose, and is it acceptable for my use case?

import coremltools as ct
import numpy as np

def measure_accuracy(original_model, optimized_model, test_images):
    """Compare top-1 accuracy between original and optimized models."""
    original_correct = 0
    optimized_correct = 0

    for image, label in test_images:
        orig_pred = original_model.predict({"image": image})
        opt_pred = optimized_model.predict({"image": image})

        if get_top_label(orig_pred) == label:
            original_correct += 1
        if get_top_label(opt_pred) == label:
            optimized_correct += 1

    orig_acc = original_correct / len(test_images) * 100
    opt_acc = optimized_correct / len(test_images) * 100
    print(f"Original: {orig_acc:.1f}%  Optimized: {opt_acc:.1f}%")
    print(f"Accuracy drop: {orig_acc - opt_acc:.1f}%")
    return orig_acc, opt_acc
Define your accuracy budget before optimizing. How much accuracy can you lose before users notice? For image classification, 1–2% drop is usually invisible. For medical or safety-critical apps, even 0.5% might be too much. Know your threshold first, then optimize to fit it.

Size-Accuracy Tradeoff: Real Numbers

Here's what optimization looks like on a real model (MobileNetV3-Large, ImageNet):

MobileNetV3-Large Optimization Results

Variant Size Top-1 Accuracy Accuracy Drop ANE Latency
Float32 (baseline) 22 MB 75.2% — 3.5 ms
Float16 11 MB 75.2% 0.0% 3.0 ms
Int8 quantized 5.5 MB 74.8% -0.4% 2.5 ms
6-bit palettized 4.1 MB 74.9% -0.3% 2.8 ms
4-bit palettized 2.8 MB 74.1% -1.1% 2.2 ms
Int4 quantized 2.8 MB 73.5% -1.7% 2.0 ms
4-bit + 50% pruned 1.8 MB 73.0% -2.2% 1.8 ms

Notice: float16 is a free 2x compression with zero accuracy loss. 4-bit palettization gives 8x compression with only 1.1% accuracy drop. Joint pruning + palettization pushes to 12x compression at 2.2% drop. For most apps, 4-bit palettization is the sweet spot.

Hugging Face to Core ML

Many models you'll want to use come from Hugging Face. The conversion workflow:

from transformers import AutoModelForImageClassification
import coremltools as ct
import torch

# Download from Hugging Face
hf_model = AutoModelForImageClassification.from_pretrained(
    "google/mobilenet_v2_1.0_224"
)
hf_model.eval()

# Trace and convert
example = torch.randn(1, 3, 224, 224)
traced = torch.jit.trace(hf_model, example)

mlmodel = ct.convert(
    traced,
    inputs=[ct.ImageType(name="image", shape=(1, 3, 224, 224),
                         scale=1/255.0, bias=[0, 0, 0])],
    compute_precision=ct.precision.FLOAT16
)
mlmodel.save("HFModel.mlpackage")
Check the model card. Hugging Face model cards list expected input normalization (mean, std). Get the scale and bias right during conversion or your model will produce garbage predictions. This is the #1 source of "my converted model doesn't work" bugs.

Model Format Considerations

.mlmodel (legacy)
Single compiled file. Older format, limited to smaller models. Still works but being phased out.
.mlpackage (recommended)
Directory bundle format. Supports larger models, metadata, and all optimization types. Use this for all new work.
.mlmodelc (compiled)
Pre-compiled for a specific device family. Faster to load (skips first-run compilation). Generate with MLModel.compileModel(at:).
Pre-compile for faster first launch. You can compile your model at build time and ship the .mlmodelc directly. This eliminates the 100–500ms compilation delay on first use. Add a Run Script build phase that calls xcrun coremlcompiler compile MyModel.mlpackage OutputDir.

Common Optimization Mistakes

Do

  • Start with float16 — it's free
  • Measure accuracy on a real validation set
  • Try both quantization and palettization
  • Benchmark on your minimum supported device
  • Define your accuracy budget upfront
  • Test edge cases (dark images, unusual angles)

Don't

  • Quantize without measuring accuracy impact
  • Assume int4 is always better than 4-bit palette
  • Optimize a model that's already fast enough
  • Apply double compression (quantize an already-quantized model)
  • Skip validation because "it worked on 3 test images"
  • Forget to benchmark latency, not just size

Exercise: The Optimization Shootout

Hands-On: Compress and Compare

Take a standard MobileNetV3 model and apply multiple optimization techniques, then compare the results:

  1. Get the baseline — Download MobileNetV3-Large from Apple's ML Gallery or Hugging Face. Convert to Core ML with float32. Measure: file size, top-1 accuracy on 20 test images, ANE inference latency.
  2. Float16 — Re-convert with compute_precision=ct.precision.FLOAT16. Measure all three metrics.
  3. Int8 quantization — Apply linear_quantize_weights with int8. Measure.
  4. 4-bit palettization — Apply palettize_weights with nbits=4. Measure.
  5. Joint compression — Apply 50% pruning + 4-bit palettization. Measure.
  6. Compare — Build a table of all five variants. Which gives the best size-accuracy tradeoff? Which is fastest on the Neural Engine?

Bonus:

  • Try 6-bit palettization — is it a better sweet spot than 4-bit for your use case?
  • Find the exact sparsity level where pruning starts to noticeably hurt accuracy
