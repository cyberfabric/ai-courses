From "Works on My Phone" to "Ships to a Million Users"

The Production Gap

You've built an image classifier, a document scanner, a chat interface with MLX. Everything works beautifully on your iPhone 16 Pro. Ship it, right?

Not so fast. Your users have iPhone SE with 4GB RAM. iPhone 12 running iOS 17. iPad Air without a Neural Engine powerful enough for your 3B model. "Works on my device" is not a deployment strategy — it's a demo.

This chapter is about the gap between a working prototype and a production feature that runs reliably on millions of devices.

Option 1: Embed in App Bundle

Drag the .mlmodel or .mlpackage into Xcode. It ships with your app binary. The user has it immediately on install — no downloads, no waiting.

Pros

  • Available immediately at launch
  • No download management code
  • Works offline from first launch
  • Simplest implementation

Cons

  • Increases app download size
  • App Store limit: 4 GB total binary
  • Can't update model without app update
  • All users download it, even those who don't use the feature

Option 2: On-Demand Resources (ODR)

Tag your model as an on-demand resource. Apple hosts it on their CDN. Your app downloads it when needed and iOS manages the storage automatically.

// Request the model when the user opens the AI feature
let request = NSBundleResourceRequest(tags: ["image-classifier"])

request.beginAccessingResources { error in
    if let error = error {
        print("Download failed: \(error)")
        return
    }

    // Model is now available in the bundle
    let model = try? ImageClassifier(configuration: .init())
    // ... use the model

    // Release when done (iOS may purge if storage is low)
    request.endAccessingResources()
}

ODR Best Practices

Tag models with descriptive resource tags in Xcode's Resource Tags panel. Set the initial install tags for critical models (downloaded with the app) and prefetch tags for models you want pre-downloaded after install. Keep download-on-demand for optional features.

Option 3: Background Assets Framework

For large models (100MB+), the Background Assets framework downloads content silently in the background, even before the user first opens your app. Apple hosts the assets on their CDN (200GB included with your developer account).

This is the recommended path for MLX models and large Core ML packages. The user never sees a loading spinner — the model arrives before they need it.

App Store Guidelines for AI

Apple has specific rules for AI-powered apps. Violate them and your app gets rejected.

The Privacy Advantage

On-device AI is a regulatory superpower. If data never leaves the device, you don't need GDPR consent dialogs for AI processing, you can mark "Data Not Collected" in App Store privacy labels, and you dodge the entire third-party AI data sharing disclosure. This is a genuine competitive advantage.

Device Capability Detection

Not all iPhones are equal. Your app needs to detect what's available at runtime and adapt.

import CoreML
import UIKit

struct DeviceCapability {
    /// Available RAM in GB
    static var ramGB: Double {
        Double(ProcessInfo.processInfo.physicalMemory)
            / 1_073_741_824
    }

    /// Does this device support the Neural Engine?
    static var hasNeuralEngine: Bool {
        MLModel.availableComputeDevices.contains(
            where: { $0.description.contains("neuralEngine") }
        )
    }

    /// Can this device run Foundation Models?
    static var supportsFoundationModels: Bool {
        if #available(iOS 26, *) {
            return SystemLanguageModel.default.availability
                == .available
        }
        return false
    }

    /// Recommended model tier based on device
    static var modelTier: ModelTier {
        if supportsFoundationModels { return .foundationModels }
        if ramGB >= 6 { return .mlxLarge }   // 3B model
        if ramGB >= 4 { return .mlxSmall }   // 1B model
        return .visionOnly                    // Built-in frameworks
    }
}

enum ModelTier {
    case foundationModels  // iPhone 15 Pro+, iOS 26
    case mlxLarge          // 6+ GB RAM, 3B model
    case mlxSmall          // 4+ GB RAM, 1B model
    case visionOnly        // Older devices, built-in only
}

Progressive Enhancement, Not Lowest Common Denominator

Don't disable AI features for everyone just because iPhone SE can't run them. Instead: iPhone SE gets Vision framework features (text recognition, classification). iPhone 15 adds MLX chat. iPhone 15 Pro gets Foundation Models with guided generation. Every user gets value; Pro users get more.

Error Handling and Graceful Degradation

Models fail. Downloads stall. Inference crashes. Your app needs to handle every failure gracefully.

func classifyImage(_ image: CGImage) async -> ClassificationResult {
    do {
        // Attempt model-based classification
        let model = try loadModel()
        let prediction = try await model.classify(image)
        return .success(prediction)
    } catch let error as MLModel.Error {
        // Model loading failed — corrupt or incompatible
        logger.error("Model load failed: \(error)")
        return .fallback(reason: "AI feature temporarily unavailable")
    } catch {
        // Unexpected inference error
        logger.error("Inference failed: \(error)")
        return .fallback(reason: "Could not analyze image")
    }
}

enum ClassificationResult {
    case success(Prediction)
    case fallback(reason: String)
}

Error Handling Checklist

  • Model not found — ODR download failed or file corrupt. Show retry button, disable feature gracefully.
  • Insufficient memory — model too large for device. Fall back to smaller model or built-in framework.
  • Inference timeout — model takes too long (thermal throttling). Return cached result or show "try again."
  • Unexpected output — model returns garbage (wrong input format). Log and show fallback UI.
  • OS version mismatch — API not available on user's iOS version. Use #available checks.

Memory Management in Production

A model that works in testing can still crash in production when the user has 47 Safari tabs open. Memory discipline is non-negotiable.

class ModelLifecycleManager {
    private var model: MLModel?

    init() {
        // Load when view appears
        NotificationCenter.default.addObserver(
            forName: UIApplication.willEnterForegroundNotification,
            object: nil, queue: .main
        ) { [weak self] _ in
            Task { try? await self?.loadModel() }
        }

        // Unload when backgrounded
        NotificationCenter.default.addObserver(
            forName: UIApplication.didEnterBackgroundNotification,
            object: nil, queue: .main
        ) { [weak self] _ in
            self?.unloadModel()
        }

        // Emergency unload on memory warning
        NotificationCenter.default.addObserver(
            forName: UIApplication.didReceiveMemoryWarningNotification,
            object: nil, queue: .main
        ) { [weak self] _ in
            self?.unloadModel()
        }
    }

    func unloadModel() {
        model = nil
        // Core ML releases memory, but nudge it
        autoreleasepool { }
    }
}

The 47-Safari-Tabs Rule

Test your AI feature after opening dozens of Safari tabs, playing music, and running navigation. That's what real user memory pressure looks like. If your model loads successfully in a clean simulator but crashes on a memory-pressured device, you have a production bug.

Performance Monitoring with os_signpost and MetricKit

You can't fix what you can't measure. Instrument your AI features to track real-world performance.

import os

let inferenceLog = OSLog(
    subsystem: "com.yourapp.ml",
    category: .pointsOfInterest
)

func measureInference<T>(
    _ name: StaticString,
    block: () throws -> T
) rethrows -> T {
    let signpostID = OSSignpostID(log: inferenceLog)
    os_signpost(.begin, log: inferenceLog,
                name: name, signpostID: signpostID)

    let result = try block()

    os_signpost(.end, log: inferenceLog,
                name: name, signpostID: signpostID)
    return result
}

// Usage
let prediction = try measureInference("ImageClassification") {
    try model.prediction(from: input)
}

For production telemetry, use MetricKit with MXSignpostMetric:

import MetricKit

class PerformanceReporter: NSObject, MXMetricManagerSubscriber {
    func didReceive(_ payloads: [MXMetricPayload]) {
        for payload in payloads {
            // Custom signpost metrics (aggregated by OS)
            if let signpost = payload.signpostMetrics?.first(
                where: { $0.signpostName == "ImageClassification" }
            ) {
                let p50 = signpost.signpostIntervalData
                    .histogrammedSignpostDurations
                    .bucketEnumerator
                // Send to your analytics backend
                analytics.track("ml_inference_p50",
                                value: p50)
            }
        }
    }
}

A/B Testing Models

Should you ship the 4-bit palettized model or the int8 quantized one? A/B testing lets you answer with real user data instead of guessing.

// Simple feature flag for model variant
let modelVariant: String = RemoteConfig.shared
    .string(forKey: "ml_model_variant") ?? "v2-4bit"

func loadModel() throws -> MLModel {
    switch modelVariant {
    case "v2-4bit":
        return try ClassifierV2_4bit(configuration: .init()).model
    case "v2-int8":
        return try ClassifierV2_int8(configuration: .init()).model
    default:
        return try ClassifierV2_4bit(configuration: .init()).model
    }
}

Track both technical metrics (latency, accuracy, errors) and product metrics (feature adoption, user satisfaction, task completion rate). A faster model that gives worse results isn't a win.

Version Your Models

Name your model files with versions: ClassifierV3_4bit.mlpackage. When using ODR or Background Assets, include the version in the resource tag. Always keep the previous version available as fallback in case the new version has issues you didn't catch in testing.

The Production Hardening Checklist

Ship-Ready Checklist

  • Device detection — runtime capability check selects appropriate model variant
  • Graceful degradation — fallback UI for unsupported devices, download failures, and inference errors
  • Memory lifecycle — load on foreground, unload on background, respond to memory warnings
  • Performance telemetry — os_signpost for inference latency, MetricKit for production aggregates
  • Error logging — OSLog for all model failures with enough context to debug
  • Thermal awareness — check thermalState before heavy inference; defer if .serious
  • App Store compliance — privacy labels, generated content guidelines, binary size within limits
  • Offline support — model available without network, or graceful handling when download required
  • Update path — model versioning, staged rollout, rollback mechanism
  • Real-device testing — tested on lowest-tier supported device under memory pressure

Exercise: Production-Harden an AI Feature

Hands-On: Ship-Ready AI

Take any AI feature you built in earlier chapters and add production hardening:

  1. Device capability check — implement the DeviceCapability struct from this chapter. Detect RAM, Neural Engine, and Foundation Models support. Choose model variant accordingly.
  2. Error handling — wrap model loading and inference in do-catch blocks. Return fallback results on failure. Log errors with OSLog.
  3. Memory lifecycle — implement ModelLifecycleManager. Load on foreground, unload on background, respond to memory warnings.
  4. Performance telemetry — add os_signpost around model loading and inference. Measure p50 latency on your device.
  5. Graceful degradation UI — build a fallback view that appears when the AI feature is unavailable (unsupported device, download pending, or inference failure).

Bonus:

  • Add a simple A/B test: switch between two model variants via a local toggle, track latency and accuracy for each
  • Simulate memory pressure in Simulator (Debug → Simulate Memory Warning) and verify your app survives
