Prompt Chaining: Breaking Big Tasks into Steps

Prompt chaining is the single most powerful advanced technique. The idea is simple: instead of one monster prompt, use a sequence of smaller prompts where each step's output feeds into the next.

Think of it as a factory assembly line. Each station does one thing well. The product moves from station to station, getting more refined at each step.

Why Chaining Beats Single Prompts

Single Monster Prompt

"Read this 200-line Kotlin class. Identify all bugs. For each bug, explain the root cause, suggest a fix, write the corrected code, and generate test cases. Also identify performance issues and suggest refactoring opportunities. Format as a structured report."

Too many tasks at once. The AI will rush through some, skip others, and the output quality drops on everything.

3-Step Chain

Step 1: "Read this Kotlin class. List every potential bug, one per line, with the line number and a one-sentence description."

Step 2: "For each bug listed below, explain the root cause and write the corrected code. [paste Step 1 output]"

Step 3: "Generate unit tests for each fix below. Use JUnit 5 and assertk. [paste Step 2 output]"

How to Design a Chain

Every good chain follows the same structure:

  1. Decompose — What are the distinct sub-tasks? Each should have a single, clear goal.
  2. Identify handoffs — What does each step need from the previous one? Define the output format.
  3. Add quality gates — After each step, review the output before passing it forward. One bad step poisons the rest.
  4. Keep context lean — Only pass what the next step needs. Don't carry the full history forward.

A Real-World Chain: Feature Spec to Code

1

Analyze Requirements

"Read this feature spec. List the core requirements, edge cases, and dependencies as bullet points."

→
2

Design Architecture

"Given these requirements, propose a class structure with responsibilities. Use MVVM. [Step 1 output]"

→
3

Implement

"Implement the ViewModel from this design. Include error handling. [Step 2 output]"

→
4

Test

"Write unit tests for this ViewModel. Cover happy path, error cases, and edge cases from the requirements. [Step 1 + Step 3 output]"

Each step produces focused, high-quality output. The chain takes 4 minutes instead of the 1 minute a single prompt would take — but the result is dramatically better.

When to chain vs single prompt: If your task has more than 2-3 distinct sub-goals, chain it. If the output of one task feeds another, chain it. If quality matters more than speed, chain it. For quick one-off questions, a single prompt is fine.

Meta-Prompting: Using AI to Write Prompts

Here's a mind-bender: you can use AI to write better prompts. It's prompts all the way down.

Meta-prompting is especially useful when you're struggling to articulate what you want, or when you need to create prompts for other people (like building a team prompt library).

Meta-Prompt 
I need a reusable prompt template for code review.

Context: Our Android team (4 devs) uses this for pull request reviews.
We care most about: security, performance, readability.
We use: Kotlin, Compose, MVVM, Coroutines.

Generate a prompt template that:
- Takes a code diff as input
- Produces a structured review with severity levels
- Catches the kinds of issues our tech stack is prone to
- Is specific enough to be useful but generic enough to
  work for any PR

Also explain WHY you structured the prompt that way —
what prompt engineering principles did you apply?
AI Output (abbreviated) 
Generated Template:

You are a senior Android reviewer specializing in Kotlin,
Compose, and MVVM architecture.

Review the following code diff. For each issue found:
- Category: Security | Performance | Readability | Bug Risk
- Severity: Critical | Warning | Suggestion
- Location: file:line
- Issue: what's wrong (1 sentence)
- Fix: what to do instead (1 sentence)

Focus areas for this tech stack:
- Coroutine scope leaks and missing cancellation
- Compose recomposition performance (unnecessary recomps)
- State hoisting violations
- SQL injection in Room queries
- Hardcoded secrets or API keys

After the issue list, provide:
1. Overall assessment (1 sentence)
2. The single most important thing to fix before merging

Code diff:
```
{{PASTE_DIFF_HERE}}
```

Why this works:
- Persona (senior Android reviewer) primes domain knowledge
- Structured output format ensures consistency across reviews
- Tech-stack-specific focus areas catch real issues, not generic ones
- Severity levels help prioritize (Ch 5: Controlling Output)
- "Single most important thing" forces prioritization (Ch 6: Patterns)
- Template variable {{PASTE_DIFF_HERE}} makes it reusable

The AI just wrote you a prompt that applies half the techniques from this course — and explained its reasoning. You can refine it, test it, and share it with your team.

Meta-prompting power move: After the AI generates a prompt, follow up with: "Now critique this prompt. What are its weaknesses? How could it fail? Improve it." You'll get a second, better version that addresses its own blind spots.

Self-Reflection: Making AI Check Its Own Work

Self-reflection is asking the AI to review and improve its own output. It's like having a built-in code reviewer — the AI generates something, then critiques it, then improves it.

Self-Reflection Chain (2 Steps) 
Step 1 (Generate):
Write a Kotlin data class for a User profile with
validation. Include: name, email, age, bio.

Step 2 (Reflect + Improve):
Review the code you just wrote. Check for:
1. Missing edge cases in validation
2. Thread safety issues
3. Potential null pointer exceptions
4. Whether it follows Kotlin best practices

List every issue you find, then output an improved version
that fixes all of them.
What Self-Reflection Catches

Issues found in self-review:

  • Email validation uses basic regex — doesn't handle all RFC 5322 cases. Fixed: using Android's Patterns.EMAIL_ADDRESS.
  • Bio length not validated — could accept a 10MB string. Fixed: added 500-char limit.
  • Age validation allows negative numbers. Fixed: range check 0-150.
  • Data class is mutable via copy(). Fixed: added init block validation so copies are also validated.

The reflection step consistently catches 2-5 issues that the initial generation misses. It takes 30 seconds extra and dramatically improves quality.

The Reflection Formula

Three components make reflection work:

  1. Specific criteria — "Check for edge cases, thread safety, null pointers" beats "review this"
  2. List issues first — Force the AI to articulate problems before jumping to fixes
  3. Improved version — Don't just list problems — output the corrected result

Without specific criteria, the AI tends to say "looks good" and make cosmetic changes.

Tree-of-Thought: Exploring Multiple Paths

Regular chain-of-thought follows one reasoning path. Tree-of-thought explores multiple paths before committing to an answer. It's like brainstorming three approaches and then picking the best one.

Tree-of-Thought Prompt 
I need to implement offline caching for our Android app's
product catalog (5,000 products, updated daily).

Generate THREE different approaches:

Approach A: [describe approach]
- Pros: [list]
- Cons: [list]
- Complexity: [Low/Medium/High]

Approach B: [describe approach]
- Pros: [list]
- Cons: [list]
- Complexity: [Low/Medium/High]

Approach C: [describe approach]
- Pros: [list]
- Cons: [list]
- Complexity: [Low/Medium/High]

Now compare all three. Given that we need to ship in 2 weeks
with a team of 2 Android devs, which approach do you
recommend and why?

This technique is especially powerful for architecture decisions, algorithm design, and any problem where the first solution isn't necessarily the best one.

When to use tree-of-thought: Architecture decisions, complex debugging (multiple hypotheses), project planning (multiple strategies), and any time you hear yourself say "there must be a better way."

Structured Reasoning: XML Tags and Frameworks

For complex analysis, you can give the AI an explicit reasoning framework using XML tags. This forces the AI to think through each component systematically instead of jumping to conclusions.

Structured Reasoning with XML Tags 
Analyze whether we should migrate from REST to GraphQL.

Structure your analysis using these sections:

<assumptions>
List every assumption you're making about our situation.
</assumptions>

<arguments_for>
Arguments in favor of migrating. Include evidence.
</arguments_for>

<arguments_against>
Arguments against migrating. Include evidence.
</arguments_against>

<risks>
What could go wrong during migration? What's the worst case?
</risks>

<recommendation>
Your recommendation with confidence level (High/Medium/Low)
and the single biggest factor behind your recommendation.
</recommendation>

The XML tags aren't magic — they're just a way to force the AI into a structured thinking pattern. The AI fills in each section, and you get a complete, balanced analysis instead of a one-sided argument.

Why XML tags work so well: They create "thinking slots" that the AI must fill. An empty <arguments_against> section forces the AI to consider downsides even when the upside is obvious. This fights the AI's tendency to agree with whatever direction your prompt implies.

Building a Team Prompt Library

Individual prompt skills are great. Team-wide prompt libraries are a multiplier. Instead of each person reinventing the wheel, you create a shared collection of battle-tested prompts.

What Goes in a Prompt Library

  • Code review prompt — Standardized template for PR reviews (tech-stack-specific)
  • Bug report generator — Rough notes → structured report (from Chapter 8)
  • Test case generator — Feature spec → categorized test cases with quotas
  • Sprint retro agenda — Template that adapts to the sprint's context
  • Architecture decision analysis — Structured comparison of technical options
  • Onboarding explainer — "Explain [codebase concept] to a new team member"
  • Status report formatter — Raw notes → polished update (from Chapter 9)

The CLAUDE.md Pattern

If you use AI coding tools like Claude Code, Cursor, or GitHub Copilot, you can embed prompts directly in your project via a CLAUDE.md file (or equivalent). This file tells the AI about your project's conventions, architecture, and rules — every time it's used.

# Project: PaymentService
## Architecture: Clean Architecture with MVVM
## Language: Kotlin 2.0, Compose, Coroutines
## Testing: JUnit 5, Turbine for Flow testing
## Rules:
- All public functions must have KDoc
- Use Result type for error handling, never throw
- ViewModels must not reference Android framework classes

This is a form of persistent prompt engineering — your prompts live in the codebase and apply automatically.

Automated Prompt Pipelines

Once you've mastered prompt chaining by hand, the next step is automation. Prompt pipelines connect AI to your existing tools — CI/CD, scripts, workflows.

1

PR Opened

Developer opens a pull request. CI triggers the pipeline.

→
2

AI Review

Script sends diff to AI with code review prompt. Gets structured feedback.

→
3

Post Comment

Script posts AI review as a PR comment. Developer sees it alongside human reviews.

→
4

Track Quality

Log AI findings. Track how many issues AI catches vs humans over time.

Other pipeline ideas: automated changelog generation from commit messages, test case generation from new feature branches, documentation updates when APIs change, and standup report drafting from git activity.

Automation rule: Never let AI take irreversible actions automatically. AI can suggest in a PR comment. It should not merge the PR. AI can draft a changelog. It should not publish a release. Keep a human in the loop for anything that goes to production.

Evaluation: Measuring Prompt Quality

How do you know if prompt A is better than prompt B? For a single use, you can eyeball it. But for prompts that run hundreds of times (in pipelines, chatbots, or team libraries), you need systematic evaluation.

Using AI to Evaluate AI Output 
You are a quality evaluator. Rate the following AI-generated
code review on these criteria (1-5 scale):

1. Accuracy: Are the identified issues real bugs?
2. Completeness: Did it miss any obvious issues?
3. Actionability: Can a developer fix these from the description?
4. Priority: Are severity levels appropriate?
5. Conciseness: Is it focused or full of filler?

Code being reviewed:
[original code]

AI's review:
[AI output to evaluate]

For each criterion, give a score and one-sentence justification.
Then give an overall score (1-5) and the single biggest
improvement the review prompt needs.

The A/B Testing Framework

To systematically improve a prompt:

  1. Create a test set — 10-20 inputs that represent the range of real usage
  2. Run both prompts on every input in the test set
  3. Evaluate outputs — Use the rubric above (or a simpler pass/fail check)
  4. Compare scores — Which prompt wins on average? On edge cases?
  5. Keep the winner, iterate — Make one change, re-run, compare again

This is how production AI systems are optimized. It's overkill for a one-off prompt but essential for anything that runs repeatedly.

Will Prompt Engineering Still Matter?

People ask this a lot. "Models are getting smarter. Won't they just understand me eventually?"

Partly, yes. Models are getting better at interpreting vague requests. But here's the thing:

Prompt engineering isn't about compensating for dumb AI. It's about communicating clearly. Even when AI can "read your mind," the person who can articulate exactly what they want — the constraints, the format, the audience, the quality bar — will always get better results than someone who says "just do the thing."

The techniques may evolve. You might not need to say "think step by step" when models reason natively. But the skill — thinking clearly about what you want and how to communicate it — that's permanent. It's a thinking skill that makes you better at delegating to humans too.

Staying
Clear task definition, context framing, format specification, quality criteria
Evolving
Chain-of-thought triggers, token-level formatting hacks, model-specific workarounds
Growing
Prompt chaining, pipeline automation, evaluation systems, CLAUDE.md-style project prompts

Exercise: Design a 3-Step Prompt Chain

Pick a complex task from your daily work — something where a single prompt gives mediocre results. Design a 3-step chain for it.

  1. Choose the task — Code review, feature planning, bug investigation, architecture decision, documentation, etc.
  2. Decompose it — What are the 3 distinct steps? What does each step produce?
  3. Define the handoffs — What output format does Step 1 need for Step 2 to work? Step 2 for Step 3?
  4. Write all 3 prompts
  5. Run the chain — Execute each step, manually passing output forward
  6. Add self-reflection — After Step 3, add a reflection step: "Review your output. What did you miss? Improve it."

Bonus challenges:

  • Use meta-prompting: ask AI to improve one of your chain's prompts
  • Try tree-of-thought: have Step 2 generate 3 approaches, then pick the best
  • Turn your chain into a reusable template for your team's prompt library
