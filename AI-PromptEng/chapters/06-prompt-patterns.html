Chapter 6

Prompt Patterns That Work

Reusable templates for everyday tasks — copy, adapt, and save hours

Stop Writing Prompts from Scratch

You've learned the building blocks (Chapter 2), the techniques (Chapter 3), the personas (Chapter 4), and the format controls (Chapter 5). You have every tool you need. But here's the thing: most of your AI interactions fall into the same 8-10 categories.

You explain things. You review things. You generate things. You convert things. You debug things. You compare things. You plan things. You summarize things.

So why write a fresh prompt from scratch every time?

A prompt pattern is a reusable template with fill-in-the-blank slots. Think of it like a function: the structure stays the same, but the arguments change. Once you have a library of patterns that work, prompting becomes a 10-second operation: grab the pattern, fill in the blanks, go.

Anatomy of a Pattern

Every pattern in this chapter follows the same structure:

Template The reusable skeleton with [SLOTS]
→
Example A real prompt with slots filled in
→
Output What the AI actually returns
→
Variations Role-specific twists

The [BRACKETED SLOTS] are the parts you swap out each time. Everything else stays the same. Let's build your library.

Pattern 1: The Explainer

Use when you need something explained at a specific level for a specific audience.

Template

Explain [TOPIC] as if I'm a [AUDIENCE LEVEL].
Focus on [SPECIFIC ASPECT].
Use [FORMAT]: [FORMAT DETAILS].
Include a real-world analogy.

Example: Filled In

Explain Kotlin coroutines as if I'm a Java developer who's never used them.
Focus on how they differ from threads and when to use each.
Use bullet points: max 5 bullets, each 1-2 sentences.
Include a real-world analogy.

AI Output

Think of threads as hiring full-time employees — each one costs
resources even when idle. Coroutines are like tasks on a to-do
list — one employee (thread) can work through many tasks, pausing
and resuming as needed.

- **Lightweight, not 1:1 with threads.** You can launch 100,000
  coroutines easily; 100,000 threads would crash your app.
- **Suspending, not blocking.** When a coroutine waits for I/O,
  the thread is freed to do other work — like await in JS.
- **Structured concurrency.** Coroutines have parent-child
  relationships, so cancellation propagates automatically.
- **Dispatchers replace thread pools.** Use Dispatchers.IO for
  network/disk, Dispatchers.Main for UI, Dispatchers.Default
  for CPU work.
- **Replace AsyncTask and RxJava.** For most Android use cases,
  coroutines are simpler and more readable than both.

Variations by Role

QA: "Explain [feature] as if I'm writing test cases for it. Focus on what can go wrong."
Manager: "Explain [tech decision] as if I'm presenting it to stakeholders. Focus on business impact, not implementation."
iOS dev: "Explain [Android concept] by comparing it to the iOS equivalent."

Pattern 2: The Reviewer

Use when you need critical feedback on code, documents, designs, or plans.

Template

Review this [THING] for [CRITERIA].
List issues as [FORMAT].
Prioritize by [METRIC].
For each issue, suggest a specific fix.

[PASTE THE THING TO REVIEW]

Example: Code Review

Review this Kotlin function for security issues and performance problems.
List issues as a numbered list with severity (Critical/High/Medium/Low).
Prioritize by severity, most critical first.
For each issue, suggest a specific fix.

fun getUserData(userId: String): Map<String, Any> {
    val query = "SELECT * FROM users WHERE id = '$userId'"
    val result = db.rawQuery(query)
    val data = mutableMapOf<String, Any>()
    data["name"] = result.getString("name")
    data["email"] = result.getString("email")
    data["password"] = result.getString("password_hash")
    data["ssn"] = result.getString("ssn")
    return data
}

AI Output

1. **Critical: SQL Injection.** String interpolation in the query
   allows arbitrary SQL execution. Fix: use parameterized queries
   — `db.rawQuery("SELECT * FROM users WHERE id = ?", arrayOf(userId))`

2. **Critical: Sensitive data exposure.** Returning password_hash
   and SSN in a generic Map exposes secrets to any caller. Fix:
   return only needed fields; create a UserProfile data class
   without sensitive fields.

3. **High: SELECT * is wasteful.** Fetches all columns including
   ones you don't use. Fix: `SELECT name, email FROM users`

4. **Medium: No null handling.** If the user doesn't exist,
   getString() will throw. Fix: check if result has rows before
   accessing; return null or throw a typed exception.

5. **Low: Untyped return.** Map<String, Any> loses type safety.
   Fix: return a data class like `UserProfile(name, email)`.

Notice how the prompt's structure — criteria, format, prioritization, and fix requirement — produces a review that's immediately actionable, not just a list of complaints.

Pattern 3: The Generator

Use when you need to produce multiple items: test cases, sample data, ideas, names, scenarios.

Template

Generate [NUMBER] [THINGS] that [CRITERIA].
Format as [FORMAT].
Include [SPECIFIC DETAILS] for each.
Make sure to cover [DIVERSITY REQUIREMENT].

Example: Test Data

Generate 8 realistic user profiles for testing an e-commerce app.
Format as a JSON array.
Include for each: name, email, age, country, accountType ("free" or "premium"), orderCount (number).
Make sure to cover: different countries, ages from 18-70, mix of free/premium, and at least one user with 0 orders.

AI Output (truncated)

[
  {
    "name": "Maria Santos",
    "email": "maria.santos@email.com",
    "age": 34,
    "country": "Brazil",
    "accountType": "premium",
    "orderCount": 47
  },
  {
    "name": "James O'Brien",
    "email": "j.obrien@email.com",
    "age": 62,
    "country": "Ireland",
    "accountType": "free",
    "orderCount": 0
  },
  ...
]

The Diversity Requirement Is the Secret Sauce

Without it, AI generates suspiciously similar items — all ages 25-35, all from the US, all with similar order counts. The "Make sure to cover..." line forces variety, which is exactly what you need for thorough testing.

Pattern 4: The Transformer

Use when you need to convert data or code from one format or language to another.

Template

Convert this [INPUT FORMAT] to [OUTPUT FORMAT].
Preserve [ASPECTS TO KEEP].
Ignore [ASPECTS TO DROP].
[ADDITIONAL RULES].

[PASTE THE INPUT]

Before: Vague Conversion

"Convert this Java code to Kotlin."

Result: Literal translation that reads like Java with Kotlin syntax. Misses data classes, extension functions, coroutines, scope functions.

After: Pattern-Based Conversion

"Convert this Java code to idiomatic Kotlin. Preserve the public API and all business logic. Ignore Java boilerplate (getters/setters, explicit null checks). Additional: use data classes, scope functions (let/apply/also), and replace callbacks with coroutines where appropriate."

Result: Clean, idiomatic Kotlin that a Kotlin developer would actually write.

The Transformer pattern works for much more than code:

  • JSON to YAML (or vice versa)
  • SQL to NoSQL queries
  • Technical spec to user story
  • Meeting notes to action items
  • English to formal documentation
  • Figma design to component hierarchy

Pattern 5: The Debugger

Use when something is broken and you need help figuring out why. This is the pattern that turns AI into your best debugging partner.

Template

Here's the problem: [DESCRIPTION]
Here's what I expected: [EXPECTED BEHAVIOR]
Here's what actually happened: [ACTUAL BEHAVIOR]
Here's the relevant code:

[CODE]

What's causing this? Suggest a fix with explanation.

Example: Android Bug

Here's the problem: My RecyclerView items are duplicating when I
rotate the device.

Here's what I expected: The list should show the same 10 items
after rotation.

Here's what actually happened: After rotation, the list shows 20
items (the original 10 appear twice). Each rotation adds another
copy.

Here's the relevant code:

class ItemsFragment : Fragment() {
    private val items = mutableListOf<Item>()

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        super.onViewCreated(view, savedInstanceState)
        viewModel.items.observe(viewLifecycleOwner) { newItems ->
            items.addAll(newItems)
            adapter.notifyDataSetChanged()
        }
    }
}

What's causing this? Suggest a fix with explanation.

AI Output

**Root cause:** `items.addAll(newItems)` appends to the list every
time the observer fires. On rotation, the Fragment is recreated,
`onViewCreated` runs again, the LiveData replays its last value,
and `addAll` appends a second copy to the list that was never cleared.

**Why it compounds:** The `items` list persists across the Fragment's
view lifecycle because it's a property of the Fragment instance
(which may be retained or recreated with existing ViewModel data).

**Fix:**

```kotlin
viewModel.items.observe(viewLifecycleOwner) { newItems ->
    items.clear()        // Clear before adding
    items.addAll(newItems)
    adapter.notifyDataSetChanged()
}
```

**Even better — use ListAdapter:**

```kotlin
// In your adapter, extend ListAdapter instead of RecyclerView.Adapter
// Then simply:
viewModel.items.observe(viewLifecycleOwner) { newItems ->
    adapter.submitList(newItems)  // Handles diffing automatically
}
```

This eliminates the manual list management entirely and gives
you efficient item animations for free.

Warning: The Debugger Pattern Needs Real Context

The more context you provide, the better the diagnosis. Vague bug descriptions get vague answers. Always include: the expected behavior, the actual behavior, and the relevant code. Skip any one of these and the AI is guessing — just like a human colleague would be.

Pattern 6: The Comparator

Use when you need to decide between options — libraries, architectures, approaches, tools.

Template

Compare [OPTION A] and [OPTION B] across these dimensions: [LIST].
Present as a table.
Add a row for "Best for" with a one-sentence recommendation.
Recommend which to use for [MY SPECIFIC SCENARIO].

Example: Library Comparison

Compare Retrofit and Ktor Client for Android HTTP networking across
these dimensions: learning curve, Kotlin idiomaticness, multiplatform
support, community size, testing support, performance.

Present as a markdown table.
Add a row for "Best for" with a one-sentence recommendation.
Recommend which to use for a new Kotlin Multiplatform project
targeting Android and iOS.

AI Output

| Dimension          | Retrofit              | Ktor Client              |
|--------------------|-----------------------|--------------------------|
| Learning curve     | Low (huge docs/SO)    | Medium (less community)  |
| Kotlin idiom.      | Java-first, Kotlin OK | Kotlin-native, coroutines|
| Multiplatform      | Android/JVM only      | Full KMP support         |
| Community          | Massive, mature       | Growing, Jetbrains-backed|
| Testing            | MockWebServer, mature | Built-in mock engine     |
| Performance        | Excellent (OkHttp)    | Excellent (pluggable)    |
| Best for           | Android-only projects | Kotlin Multiplatform     |

**Recommendation for your scenario:** Use **Ktor Client**. Since
you're building a KMP project targeting both Android and iOS,
Ktor's multiplatform support means one networking layer shared
across platforms. Retrofit would lock you into Android-only and
require a separate iOS solution.

The Comparator works for any decision: libraries, cloud providers, database types, architecture patterns, hiring candidates, even lunch spots (okay, maybe not that last one).

Pattern 7: The Planner

Use when you need a step-by-step plan for a project, migration, feature, or initiative.

Template

I need to [GOAL].
My constraints are: [LIST].
Create a step-by-step plan with [DELIVERABLE] at each step.
Estimate [TIME/EFFORT] for each step.
Flag any risks or dependencies.

Example: Migration Plan

I need to migrate our Android app from Java to Kotlin.
My constraints are: 2 developers, can't stop feature work, app has
120 Java files, CI must stay green throughout.

Create a step-by-step plan with a concrete deliverable at each step.
Estimate effort in developer-days for each step.
Flag any risks or dependencies.

AI Output (truncated)

## Java-to-Kotlin Migration Plan

### Phase 1: Foundation (3 dev-days)
- **Step 1:** Configure Kotlin in the project (build.gradle,
  kotlin-android plugin). Deliverable: Kotlin compiles alongside Java.
- **Step 2:** Convert 3 simple utility files using the auto-converter
  + manual cleanup. Deliverable: First Kotlin files merged to main.
- Risk: Auto-converter produces non-idiomatic code. Plan for manual
  review of every conversion.

### Phase 2: New Code in Kotlin (ongoing)
- **Step 3:** All new files written in Kotlin, no new Java files.
  Deliverable: Team agreement + lint rule enforcing this.
- Risk: Team needs Kotlin onboarding. Budget 1-2 days for learning.

### Phase 3: Bottom-Up Conversion (40 dev-days)
- **Step 4:** Convert data models and utilities first (no Android
  dependencies). Deliverable: ~30 files converted.
- **Step 5:** Convert business logic / repository layer.
  Deliverable: ~40 files converted.
- **Step 6:** Convert UI layer (Activities, Fragments) last.
  Deliverable: ~50 files converted.
- Dependency: Steps 4→5→6 must be sequential (UI depends on logic
  depends on models).
...

Tip: The Constraints Are the Power

A plan without constraints is a fantasy. The magic of the Planner pattern is in the constraints line. "2 developers, can't stop feature work, CI must stay green" forces the AI to produce a realistic plan, not an ideal-world one. Add budget, timeline, team size, technical limitations — the more constraints, the more practical the plan.

Pattern 8: The Summarizer

Use when you need to distill long content into something digestible for a specific audience.

Template

Summarize this [CONTENT TYPE] in [FORMAT].
Focus on [AUDIENCE / PERSPECTIVE].
Highlight [KEY ASPECTS].
Max length: [CONSTRAINT].

[PASTE THE CONTENT]

Example: Meeting Summary

Summarize these meeting notes in 5 bullet points.
Focus on what a developer needs to know (skip business/political context).
Highlight: decisions made, action items with owners, deadlines.
Max length: one line per bullet.

[Meeting notes: 45-minute architecture review discussing migration
from monolith to microservices. Attendees: CTO, 3 senior devs,
product manager. Discussed auth service extraction, API gateway
selection, timeline for Q3, budget concerns, team allocation...]

AI Output

- **Decision:** Auth service will be extracted first as the pilot
  microservice (unanimous agreement).
- **API Gateway:** Team will evaluate Kong vs AWS API Gateway;
  Sarah owns a comparison doc by Friday June 13.
- **Timeline:** Auth service MVP due end of Q3; full migration is
  a 2025 initiative, not 2024.
- **Action:** Mike to set up the new repo structure and CI pipeline
  for the auth service by June 20.
- **Blocker:** No budget approved yet for additional infrastructure;
  CTO escalating to VP Engineering this week.

The Summarizer's power is in the "Focus on [AUDIENCE]" line. The same meeting notes summarized for a developer, a PM, and a CTO would highlight completely different things.

Building Your Own Patterns

The eight patterns above cover the most common tasks, but your work has its own patterns too. Here's how to spot and build them:

1. Notice Repetition "I ask this type of question a lot..."
→
2. Find the Variables What changes each time? Those are your [SLOTS].
→
3. Write the Template Replace the variables with [BRACKETED NAMES].
→
4. Test & Refine Use it 3 times. Adjust what doesn't work.

Tip: Share Patterns with Your Team

Prompt patterns are even more valuable when shared. Keep them in a team wiki, a shared Notion page, your project's CLAUDE.md, or a Slack channel. When one person discovers a pattern that works, the whole team benefits. Some teams even do "prompt pattern reviews" alongside code reviews.

Quick Reference: All Eight Patterns

Pattern Use When... Key Slots
Explainer You need something explained at a specific level [TOPIC], [AUDIENCE], [ASPECT], [FORMAT]
Reviewer You need critical feedback [THING], [CRITERIA], [FORMAT], [METRIC]
Generator You need multiple items produced [NUMBER], [THINGS], [CRITERIA], [DIVERSITY]
Transformer You need format/language conversion [INPUT], [OUTPUT], [PRESERVE], [IGNORE]
Debugger Something is broken [PROBLEM], [EXPECTED], [ACTUAL], [CODE]
Comparator You need to decide between options [OPTIONS], [DIMENSIONS], [SCENARIO]
Planner You need a step-by-step roadmap [GOAL], [CONSTRAINTS], [DELIVERABLE]
Summarizer You need to distill long content [CONTENT], [FORMAT], [AUDIENCE], [ASPECTS]

Hands-On Exercise

Exercise: Build Your Personal Pattern Library

This is a two-part exercise designed to make these patterns stick.

Part 1: Use Three Patterns Today

  1. Pick a real task from your current work — a bug you're fixing, a feature you're planning, a decision you're making.
  2. Choose the most relevant pattern from the eight above.
  3. Fill in the slots and run it. Compare the result to what you would have gotten with an unstructured prompt.
  4. Repeat with two more tasks, using different patterns each time.

Part 2: Create Your Own Pattern

  1. Think of a task you do at least once a week with AI (code review? writing tests? drafting messages?).
  2. Write out the prompt you typically use.
  3. Identify the parts that change each time — those are your [SLOTS].
  4. Replace the changing parts with descriptive [SLOT NAMES].
  5. Test your pattern three times with different inputs. Refine until it consistently produces good output.

Bonus: Save your best patterns somewhere accessible — a note, a snippet library, your CLAUDE.md file, or a team wiki. Future you will thank present you.

Key Takeaways

  • Patterns are reusable prompt templates with fill-in-the-blank slots — like functions for prompting
  • 8 core patterns cover most daily tasks: Explainer, Reviewer, Generator, Transformer, Debugger, Comparator, Planner, Summarizer
  • The power is in the structure, not the specific words — criteria, format, constraints, and diversity requirements shape the output
  • Build your own patterns by noticing repetition, finding the variables, and templating them
  • Share patterns with your team — one person's discovery benefits everyone
