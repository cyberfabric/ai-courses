Chapter 5

Controlling Output Format

JSON, markdown, tables, bullet points — get exactly the format you need, every time

The Reformatting Tax

You've asked the AI a brilliant question. You used a persona (Chapter 4), structured your prompt with all five building blocks (Chapter 2), and even added chain-of-thought (Chapter 3). The AI's answer is... perfect. Except it's a wall of prose, and you needed a JSON object.

Now you're spending five minutes copying, reformatting, and restructuring the output to fit your actual workflow. That's the reformatting tax — and it's entirely avoidable.

The goal of this chapter: Make the AI's output slot directly into your workflow — no copy-paste surgery required. Whether you need JSON for an API, a markdown table for documentation, or bullet points for a Slack message, you'll learn to get it right the first time.

Why Format Is the Forgotten Building Block

Remember the five building blocks from Chapter 2? Task, Context, Constraints, Format, Examples. Most people nail the first three and completely forget Format. They get a great answer wrapped in the wrong package.

Think of it like ordering food delivery. The food might be delicious, but if it arrives in a garbage bag instead of proper containers, you've got extra work before you can eat. Format is the container.

The Format Spectrum

AI models can output in virtually any text format. Here's the spectrum from most structured to most freeform:

JSON / XML Machine-readable, parseable, strict structure
→
CSV / YAML Data-friendly, import into spreadsheets or configs
→
Tables Human-readable comparisons, structured but visual
→
Lists / Bullets Scannable, easy to copy into docs or Slack
→
Prose / Essay Freeform text, maximum flexibility

The further left you go, the more precisely you need to specify what you want. JSON needs a schema. Bullet points just need a topic. Let's master each one.

JSON: The Developer's Best Friend

If you're a developer, JSON output is probably what you want most often. Whether it's for API mocking, test data, configuration, or just structured information — JSON is king.

Before: Vague Format

List the top 5 HTTP status codes with their meanings.

After: JSON with Schema

List the top 5 HTTP status codes. Return as a JSON array where each object has: code (number), name (string), description (string, one sentence), category (string: "success", "redirect", "client_error", or "server_error").

Prompt

List the top 5 HTTP status codes used in REST APIs.

Return as a JSON array. Each object must have:
- "code": number
- "name": string
- "description": string (one sentence, max 15 words)
- "category": one of "success", "redirect", "client_error", "server_error"

Return ONLY the JSON array. No explanation, no markdown fences.

AI Output

[
  {
    "code": 200,
    "name": "OK",
    "description": "The request succeeded and the response body contains the result.",
    "category": "success"
  },
  {
    "code": 201,
    "name": "Created",
    "description": "A new resource was successfully created.",
    "category": "success"
  },
  {
    "code": 400,
    "name": "Bad Request",
    "description": "The server cannot process the request due to client error.",
    "category": "client_error"
  },
  {
    "code": 404,
    "name": "Not Found",
    "description": "The requested resource does not exist on the server.",
    "category": "client_error"
  },
  {
    "code": 500,
    "name": "Internal Server Error",
    "description": "The server encountered an unexpected condition preventing fulfillment.",
    "category": "server_error"
  }
]

The Schema Specification Technique

When requesting JSON, always specify the exact field names, data types, and allowed values. Don't say "return as JSON" — say "return as a JSON object with these fields." The more precise your schema, the more consistent the output.

Pro tip: If you have an existing JSON example, paste it and say "Follow this exact structure."

Tip: Stripping Markdown Fences

AI models love wrapping JSON in ```json ... ``` markdown fences. If you're parsing the output programmatically, add: "Return ONLY the raw JSON. No markdown code fences, no explanation." If the model still wraps it, try: "Do NOT wrap the output in backticks or code blocks."

Markdown: For Docs, READMEs, and Everything In Between

Markdown is the lingua franca of developer documentation. When you need formatted text that renders beautifully in GitHub, Notion, Confluence, or any markdown viewer, ask for it explicitly.

Prompt

Write a brief README section for a user authentication module.

Format as markdown with:
- An H2 heading
- A one-paragraph description
- A "Quick Start" H3 with a numbered list (3 steps)
- A "Configuration" H3 with a table (columns: Parameter, Type, Default, Description)
  Include at least 4 parameters.
- A "Note" callout using a blockquote

AI Output

## User Authentication Module

A lightweight authentication module that handles user login, registration,
and session management. Supports JWT tokens, OAuth 2.0, and basic
username/password authentication out of the box.

### Quick Start

1. Install the package: `npm install @myapp/auth`
2. Initialize with your config: `AuthModule.init({ provider: 'jwt' })`
3. Protect routes with the middleware: `app.use('/api', auth.guard())`

### Configuration

| Parameter     | Type    | Default | Description                          |
|---------------|---------|---------|--------------------------------------|
| `provider`    | string  | `"jwt"` | Auth strategy: "jwt", "oauth", "basic" |
| `tokenExpiry` | number  | `3600`  | Token lifetime in seconds            |
| `refreshable` | boolean | `true`  | Enable refresh token rotation        |
| `maxAttempts` | number  | `5`     | Failed login attempts before lockout |

> **Note:** Always set `tokenExpiry` to match your security policy.
> In production, use environment variables for sensitive configuration.

Notice how specifying exactly which markdown elements to use (H2, H3, numbered list, table, blockquote) gives you predictable, consistent output every time.

Tables: When You Need to Compare

Tables are perfect for comparisons, feature matrices, and structured data that humans need to scan quickly. The key is specifying your columns upfront.

Prompt

Compare the three main Android architecture patterns.

Present as a markdown table with these columns:
- Pattern (name)
- Complexity (Low / Medium / High)
- Testability (1-5 stars)
- Best For (one sentence)
- Drawback (one sentence)

Sort by complexity from low to high.

AI Output

| Pattern | Complexity | Testability | Best For | Drawback |
|---------|-----------|-------------|----------|----------|
| MVC     | Low       | ★★☆☆☆      | Simple apps and prototypes | Activities become bloated "god objects" |
| MVP     | Medium    | ★★★★☆      | Apps needing clear separation and unit testing | Lots of boilerplate interfaces |
| MVVM    | High      | ★★★★★      | Complex apps with reactive UI and data binding | Steeper learning curve, debugging data flows |

Tip: Column Control = Output Control

The columns you define determine what information you get. Want more detail? Add more columns. Want brevity? Limit columns and add "(one word)" or "(one sentence)" constraints to each. The table structure forces the AI to be concise and parallel.

The "Respond ONLY With..." Technique

This is the single most powerful format control technique. When you need clean, parseable output with zero fluff, this phrase is your best friend.

The ONLY Technique

End your prompt with "Respond ONLY with [format]. No explanations, no additional text." This strips away the AI's natural tendency to add helpful context, caveats, and preambles. Use it when the output goes directly into code, a spreadsheet, or another tool.

Without "ONLY"

"Convert these temperatures to Celsius: 32F, 72F, 212F"

Result: "Sure! Here are the conversions: 32°F is 0°C, which is the freezing point of water. 72°F is 22.2°C, which is a comfortable room temperature..."

With "ONLY"

"Convert these temperatures to Celsius: 32F, 72F, 212F. Respond ONLY with the values as a comma-separated list. No explanations."

Result: "0, 22.2, 100"

The "ONLY" technique works especially well for:

  • Generating test data that goes straight into a file
  • Creating SQL queries you'll paste into a database client
  • Producing JSON for API mock responses
  • Getting code without the surrounding explanation
  • Extracting specific values from text

Controlling Length: Say Exactly How Much

"Be concise" means nothing to an AI. It's like telling a chef "use some salt" — you'll get a different amount every time. Instead, give specific length constraints.

Vague Length

  • "Keep it short"
  • "Be brief"
  • "Don't be too verbose"
  • "Give me a summary"

Specific Length

  • "In exactly 3 bullet points"
  • "In under 50 words"
  • "One paragraph, max 4 sentences"
  • "In a single sentence"

Here's a handy reference for length constraints by use case:

Use Case Length Constraint Example Phrase
Slack message 1-2 sentences "Respond in one sentence."
Code comment Under 15 words "Write a comment in under 15 words."
PR description 3-5 bullet points "Summarize in 3-5 bullet points."
Documentation section 2-3 paragraphs "Write 2-3 short paragraphs."
Email Under 100 words "Keep the entire email under 100 words."
Commit message Under 72 characters "One line, max 72 characters."

Warning: AI Models Can't Count

AI models are notoriously bad at counting words and characters precisely. "Exactly 50 words" will usually give you 45-60. For strict limits, use structural constraints instead: "exactly 3 bullet points" or "a single paragraph of 2-3 sentences" is much more reliable than word counts. When exact character counts matter (like commit messages), always verify yourself.

Controlling Tone: How It Should Sound

Format isn't just about structure — it's also about voice. The same information can be wrapped in very different tones, and specifying the tone is as simple as naming it.

Prompt

Explain what a null pointer exception is.

Write four versions, each exactly 2 sentences:
1. Technical (for a senior developer)
2. ELI5 (for a complete beginner)
3. Formal (for documentation)
4. Casual (for a Slack message to a teammate)

AI Output

**Technical:** A NullPointerException occurs when the JVM attempts to
dereference an object reference that points to null. This typically
indicates a missing initialization, an incorrect assumption about
return values, or a race condition in concurrent code.

**ELI5:** Imagine you wrote down a friend's address to visit them, but
the paper is blank. When you try to go there, you're stuck because
there's no address to follow — that's a null pointer exception.

**Casual:** Hey, you're getting an NPE because something is null when
you try to use it. Check where that variable gets set — it's probably
not initialized on that code path.

**Formal:** A null pointer exception is a runtime error that occurs
when an application attempts to use an object reference that has not
been assigned a valid instance. This condition requires the developer
to ensure proper null-safety checks or initialization prior to access.

Common tone keywords that work well: technical, casual, formal, friendly, academic, sarcastic, empathetic, ELI5, executive-summary, no-nonsense.

Multi-Part Outputs: Getting Everything in One Shot

Sometimes you need multiple things from a single prompt: an analysis and a recommendation and action items. Instead of asking three times, structure your request with explicit sections.

Prompt

Review this Kotlin function for potential issues:

```kotlin
fun processUser(userId: String): User {
    val user = database.findById(userId)
    val profile = api.fetchProfile(user.email)
    return User(user.name, profile.avatar, user.role)
}
```

Structure your response as:

## Issues Found
List each issue as a numbered item with severity (Critical/Medium/Low).

## Recommended Fix
Show the corrected code with comments explaining changes.

## Testing Notes
3 bullet points: what to test after making these changes.

AI Output

## Issues Found

1. **Critical:** `database.findById()` may return null, causing NPE on
   `user.email` access. No null check present.
2. **Medium:** `api.fetchProfile()` is a network call with no error
   handling. Timeout or failure will crash the function.
3. **Low:** No input validation on `userId` — empty strings or
   malformed IDs will propagate to the database layer.

## Recommended Fix

```kotlin
fun processUser(userId: String): User? {
    require(userId.isNotBlank()) { "userId must not be blank" }

    val user = database.findById(userId)
        ?: return null  // Handle missing user gracefully

    val profile = try {
        api.fetchProfile(user.email)
    } catch (e: Exception) {
        ProfileDefaults.empty()  // Fallback on API failure
    }

    return User(user.name, profile.avatar, user.role)
}
```

## Testing Notes

- Test with a non-existent userId to verify null is returned
- Test with API timeout/failure to verify fallback behavior
- Test with blank/empty userId to verify input validation

The Section Headers Technique

When you want multi-part output, write the section headers yourself in the prompt using markdown (##). The AI will mirror your structure exactly. This works because the model sees your headers as a template to fill in. You're not requesting a format — you're defining it.

Delimiters: Fencing Your Output

Delimiters are boundaries that separate different parts of the AI's response. They're essential when you're parsing output programmatically or when you need the AI to distinguish between your instructions and the content it should process.

XML Tags as Delimiters

XML-style tags are surprisingly effective delimiters, especially with Claude:

Prompt

Analyze this error message and provide structured output:

<error>
java.lang.OutOfMemoryError: GC overhead limit exceeded
  at com.myapp.ImageLoader.loadBitmap(ImageLoader.java:142)
  at com.myapp.Gallery.onBindViewHolder(Gallery.java:87)
</error>

Respond using these tags:
<cause> one-sentence root cause </cause>
<severity> Critical / High / Medium / Low </severity>
<fix> recommended fix in 2-3 sentences </fix>
<prevention> how to prevent this in the future </prevention>

AI Output

<cause>The app is loading full-resolution bitmaps into memory during
RecyclerView scrolling, exhausting the heap.</cause>

<severity>Critical</severity>

<fix>Use an image loading library like Coil or Glide that handles
downsampling and caching automatically. Set explicit size constraints
matching the ImageView dimensions rather than loading full-resolution
images.</fix>

<prevention>Implement memory-aware image loading with configurable
bitmap pools, use thumbnail previews for gallery views, and add heap
monitoring to your CI pipeline.</prevention>

Other Useful Delimiters

Common Delimiters

  • XML tags: <result>...</result> — best for structured parsing
  • Triple backticks: ```...``` — best for code blocks
  • Markdown headers: ## Section — best for human reading
  • Dashes/equals: --- — best for simple separators
  • Numbered sections: [1]...[2]... — best for ordered output

Tip: Claude Loves XML Tags

If you're using Claude specifically, XML tags are a first-class formatting tool. Claude is trained to follow XML-delimited structures very reliably. Use them to separate instructions from data, define output sections, and create parseable structured responses. Other models handle XML tags well too, but Claude is especially consistent.

Code Output: Getting Clean, Usable Code

When you want code from the AI, you need to specify more than just the language. Think about what makes code immediately usable vs. code that needs editing.

Prompt

Write a Kotlin extension function that validates an email address.

Requirements:
- Return type: Boolean
- Use regex, no external libraries
- Handle edge cases: empty string, no @, multiple @, no domain
- Include KDoc comment
- No main function or usage examples — just the function itself

Code Output Checklist

When requesting code, always specify:

  • Programming language and version (if relevant)
  • Just the function, or a full file with imports?
  • Include comments or keep it clean?
  • Include tests or just the implementation?
  • Error handling approach (exceptions? Result type? null?)
  • Dependencies: can it use external libraries?

Warning: The "Helpful" Code Problem

AI models love to be helpful. Ask for a function and you'll get the function, a main() demo, three usage examples, an explanation of how it works, and suggestions for improvement. If you just need the code, be explicit: "Return ONLY the function. No examples, no explanation, no main function."

When the AI Ignores Your Format

Sometimes despite your best efforts, the AI returns the wrong format. Don't just re-ask the same way — diagnose and fix.

Common Causes and Fixes

The format instruction is buried at the end
Fix: Move it to the top. "Format your response as a JSON array. Here's the task: ..."
Conflicting instructions
Fix: "Explain in detail" + "respond in one sentence" = confused AI. Pick one or separate them into distinct sections.
The AI adds preamble ("Sure! Here's the...")
Fix: Add "Do NOT include any preamble, greeting, or explanation. Start directly with the [format]."
Markdown fences around JSON/code
Fix: "Return raw [format] without markdown code fences or backticks."
The AI "improves" your format
Fix: "Follow this format EXACTLY. Do not modify, add to, or improve the structure."

The Reinforcement Technique

State the format instruction twice: once at the top of your prompt and once at the very end. AI models weight the beginning and end of prompts more heavily than the middle. Sandwiching your format instruction gives it maximum impact.

Return as a JSON object with these fields: name, age, role.
[... rest of your prompt ...]
Remember: return ONLY the JSON object. No other text.

Here are ready-to-use format instructions for tasks you probably do every week:

"Format as: **Title** (one line), **Steps** (numbered), **Expected** (one line), **Actual** (one line), **Severity** (Critical/High/Medium/Low)"
📝

"Format as: **Summary** (2-3 sentences), **Changes** (bullet list), **Testing** (checkbox list), **Notes** (optional)"
📊

"Present as a markdown table. Columns: [list]. Rows: [items]. End with a one-sentence recommendation."
✅
Test Cases

Email
"Format as: **Subject line**, then the email body. Tone: [formal/friendly]. Max 100 words."
🔍
Code Review
"List issues as: line number, severity (🔴🟡🟢), issue description, suggested fix. Most critical first."

Hands-On Exercise

Exercise: The Perfect Test Plan

Your mission: get the AI to output a perfectly formatted test plan as a markdown table. Here's the scenario:

Feature: A "Forgot Password" flow for a mobile app. The user enters their email, receives a reset link, clicks it, sets a new password, and gets a confirmation.

Your task:

  1. Write a prompt that asks the AI to generate test cases for this feature.
  2. Specify the format as a markdown table with these columns: ID, Test Case, Type (Positive/Negative/Edge), Steps, Expected Result.
  3. Add constraints: at least 8 test cases, include at least 2 edge cases, sort by Type (positive first, then negative, then edge).
  4. Use the "ONLY" technique: no introduction, no summary — just the table.

Bonus challenges:

  • Get the same test plan as JSON instead of a table. Which was easier to specify?
  • Add a "Priority" column with values P0-P3. Does the AI assign sensible priorities?
  • Try the reinforcement technique: format instruction at the top AND bottom. Does it help?

What to look for: Did the AI follow your exact column order? Did it include the right number of test cases? Are the Types correctly categorized? If anything is off, tweak your prompt and try again — that iteration skill is what separates good prompters from great ones.

Key Takeaways

  • Always specify format — "return as JSON", "present as a table", "respond in 3 bullet points"
  • Define schemas for structured data — field names, types, and allowed values
  • "Respond ONLY with..." strips away fluff for clean, parseable output
  • Use specific length constraints — "3 bullet points" beats "be concise"
  • Write section headers yourself to define multi-part output structure
  • XML tags make great delimiters — especially with Claude
  • Reinforce format instructions at both the start and end of your prompt
  • When format breaks, move the instruction to the top, remove conflicts, and be more explicit
