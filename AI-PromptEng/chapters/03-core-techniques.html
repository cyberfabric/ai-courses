Chapter 3

Core Techniques: Zero-Shot, Few-Shot, and Chain-of-Thought

The three fundamental prompting strategies — when to use each and why they work

Three Strategies, Infinite Situations

In Chapter 2, you learned the five building blocks of a great prompt. Now it's time to learn the three strategies that determine how the AI processes your request. Think of the building blocks as ingredients, and these techniques as cooking methods — same ingredients, completely different dishes.

The three techniques are:

Zero-Shot
Just ask — no examples needed
Few-Shot
Show 2-5 examples first
CoT
Think step by step

Every prompt you've ever written falls into one of these categories (or a combination). Let's master each one.

Zero-Shot: Just Ask

Zero-shot means asking the AI to do something with no examples. You describe the task, and the AI figures out how to do it based on its training alone. This is how most people prompt by default.

When Zero-Shot Works Great

  • Common tasks the AI has seen millions of times: summarizing, translating, explaining code
  • Simple instructions with no ambiguity: "Convert this to JSON" or "Sort this list"
  • Well-known formats: the AI knows what a unit test, SQL query, or git commit message looks like
  • Speed over precision: when "good enough" on the first try is fine
Zero-shot example:

Write a SQL query that finds all users who signed up in the last 30 days and have placed at least one order. Use the tables: users (id, email, created_at) and orders (id, user_id, total, created_at). 
SELECT u.id, u.email, u.created_at, COUNT(o.id) AS order_count
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.created_at >= NOW() - INTERVAL '30 days'
GROUP BY u.id, u.email, u.created_at
HAVING COUNT(o.id) >= 1
ORDER BY u.created_at DESC;

Works perfectly. The AI knows SQL inside and out, the task is unambiguous, and the format is obvious. No examples needed.

When Zero-Shot Falls Apart

Zero-shot struggles when:

  • Your task has a specific format the AI can't guess (company-specific templates, naming conventions)
  • The task is nuanced or subjective ("rate these bug reports by severity" — by whose scale?)
  • You need consistent output across multiple prompts (the AI might format things differently each time)

When zero-shot gives you inconsistent or wrong results, it's time to upgrade to few-shot.

Few-Shot: Show, Don't Tell

Few-shot means giving the AI 2-5 examples before your actual request. Instead of explaining the pattern, you demonstrate it. The AI infers the rules from your examples and applies them to new input.

This is the technique we touched on in Chapter 2 when we discussed the Examples building block. Now let's go deeper.

The analogy: Teaching someone to cook by showing them a recipe (zero-shot) vs. cooking alongside them three times first (few-shot). The second approach produces better results because they've seen how you handle the tricky parts.

Zero-Shot: Describe the Pattern

Write commit messages for these changes. Use the conventional commit format with a type prefix, scope in parentheses, and a concise description in imperative mood. Keep it under 72 characters.

That's a lot of rules to follow. The AI might get some right, some wrong.

Few-Shot: Show the Pattern

Write commit messages for these changes. Follow this format:

Example 1:
Change: Added email validation to signup form
Commit: feat(auth): add email format validation to signup

Example 2:
Change: Fixed crash when user rotates screen on profile page
Commit: fix(profile): handle configuration change in ProfileFragment

Example 3:
Change: Moved API constants to a dedicated file
Commit: refactor(api): extract endpoint constants to ApiRoutes

Now write commits for:
1. Removed unused imports from MainActivity
2. Added dark mode toggle to settings screen
3. Fixed incorrect price calculation for discounted items

The AI now has the exact pattern. No ambiguity. Consistent output guaranteed.

1. refactor(main): remove unused imports from MainActivity
2. feat(settings): add dark mode toggle to settings screen
3. fix(cart): correct price calculation for discounted items

Notice how the AI picked up the type prefixes (feat, fix, refactor), the scope convention, the imperative mood, and the length constraint — all from three examples.

How to Pick Good Few-Shot Examples

Not all examples are created equal. Bad examples confuse the AI. Good examples make it almost impossible to get wrong.

The Three Rules of Great Examples

  1. Diversity: Show different variations, not three identical cases. If you're classifying bugs, show a critical bug, a minor bug, and an edge case — not three critical bugs.
  2. Edge cases: Include at least one tricky example. This teaches the AI how to handle ambiguity, not just easy cases.
  3. Format consistency: Every example must follow the exact same format. If example 1 uses bullet points and example 2 uses numbered lists, the AI won't know which to follow.

Bad Examples (All the Same)

Example 1: Bug: App crashes → Severity: High
Example 2: Bug: Screen goes blank → Severity: High
Example 3: Bug: Data is lost → Severity: High

All three are "High." The AI has no idea what "Medium" or "Low" looks like.

Good Examples (Diverse)

Example 1: Bug: App crashes on launch for all users → Severity: Critical
Example 2: Bug: Profile photo doesn't resize on tablets → Severity: Low
Example 3: Bug: Checkout button sometimes unresponsive on slow networks → Severity: Medium

Now the AI understands the full spectrum and can classify new bugs accurately.

How many examples? The sweet spot is 2-5. One example might be a coincidence. Two establish a pattern. Three confirm it. Beyond five, you're usually wasting tokens without improving quality. Exception: if your task has many edge cases, 5-7 examples can help.

Chain-of-Thought: Think Step by Step

Chain-of-thought (CoT) is the technique of asking the AI to show its reasoning before giving a final answer. Instead of jumping to a conclusion, the AI works through the problem step by step.

This is the closest thing prompt engineering has to a superpower for complex tasks.

The analogy: Imagine asking a junior developer to review a pull request. If they just say "LGTM" you don't trust it. But if they say "I checked the null safety, verified the edge cases, confirmed the tests cover the new path, and validated the API contract" — now you trust the answer because you can see the reasoning.

Without CoT: Jump to Answer

We have 3 Android developers and 2 QA engineers. Each developer produces about 4 PRs per week. Each PR needs 45 minutes of QA review. QAs work 6 hours per day on reviews. Can our QA team keep up?
Yes, your QA team should be able to keep up with the PR volume.

That's... not helpful. No reasoning, no math, no confidence in the answer.

With CoT: Show Your Work

We have 3 Android developers and 2 QA engineers. Each developer produces about 4 PRs per week. Each PR needs 45 minutes of QA review. QAs work 6 hours per day on reviews. Can our QA team keep up?

Think through this step by step. Show your calculations.
Let me work through this step by step:

1. Weekly PR volume:
3 developers × 4 PRs/week = 12 PRs per week

2. Weekly review time needed:
12 PRs × 45 min = 540 minutes = 9 hours/week

3. Weekly QA capacity:
2 QAs × 6 hours/day × 5 days = 60 hours/week

4. Utilization:
9 hours needed / 60 hours available = 15% utilization

Answer: Yes, easily. Your QA team is at only 15% capacity for PR reviews, leaving plenty of time for other testing activities. Even if PR volume doubled or tripled, they'd still have capacity.

Now you can verify the math, spot errors, and trust the conclusion.

Why "Think Step by Step" Actually Works

This isn't magic. There's a real mechanism behind it:

The Technical Reason

AI models generate text one token at a time. Each new token is influenced by all the tokens before it. When you ask the AI to "think step by step," it generates intermediate reasoning tokens that become context for the next step. Each step builds on the previous one, making the final answer more accurate.

Without CoT, the model has to jump from question to answer in a single leap. With CoT, it gets a running start.

When CoT Helps Most

  • Math and calculations
  • Multi-step logic problems
  • Complex code debugging
  • Architecture decisions
  • Risk analysis
  • Comparing trade-offs
  • Any task where the reasoning matters as much as the answer

When CoT Hurts

  • Simple factual questions ("What's the Kotlin keyword for...")
  • Code generation for straightforward tasks
  • Translation or formatting
  • Tasks where you just need the answer, not the reasoning
  • When token budget is tight (CoT is verbose)
Rule of thumb: If a human would need to think for more than 10 seconds before answering, the AI benefits from CoT. If a human would answer instantly, skip CoT.

CoT Variants: More Than "Think Step by Step"

There are several ways to trigger chain-of-thought reasoning. Some are subtle, some are explicit.

Zero-Shot CoT
"Think step by step" — simple trigger, no examples needed. Works surprisingly well.
Explicit CoT
"First analyze X, then evaluate Y, then recommend Z" — you define the reasoning steps.
Structured CoT
"Use this format: Observation, Analysis, Conclusion" — you template the thinking.

Let's see Explicit CoT in action with a code review scenario:

Explicit CoT for code review:

Review this Kotlin function for potential issues. Follow these steps:

1. Read: What does this function do? Summarize in one sentence.
2. Correctness: Are there any logical errors or edge cases that would produce wrong results?
3. Safety: Are there null safety issues, uncaught exceptions, or thread-safety concerns?
4. Performance: Any obvious performance problems (N+1 queries, unnecessary allocations, blocking calls)?
5. Verdict: Rate as PASS, MINOR ISSUES, or NEEDS REWORK. List specific changes needed.

```kotlin fun getActiveUsers(db: Database): List<User> { val all = db.query("SELECT * FROM users") val active = all.filter { it.lastLogin != null } val sorted = active.sortedByDescending { it.lastLogin } return sorted.take(100) } ```
1. Read: Fetches all users from the database, filters to those who have logged in, sorts by most recent login, and returns the top 100.

2. Correctness: Functionally correct, but loads ALL users into memory just to return 100. The filter/sort should happen at the database level.

3. Safety: No null safety issues (lastLogin check is fine). However, no error handling for database connection failures. The query uses SELECT * which might break if columns change.

4. Performance: Major issue. If the users table has 1M rows, this loads all of them into memory. The WHERE, ORDER BY, and LIMIT should be in the SQL query.

5. Verdict: NEEDS REWORK
- Move filter to SQL: WHERE last_login IS NOT NULL
- Move sort to SQL: ORDER BY last_login DESC
- Move limit to SQL: LIMIT 100
- Replace SELECT * with specific columns
- Add try/catch for database errors

By defining the thinking steps yourself, you get a thorough, structured review instead of a random collection of comments.

Combining Techniques: Few-Shot + CoT

The real power move? Combine few-shot and chain-of-thought. You show examples that include reasoning, so the AI learns both the format AND the thinking process.

Few-Shot CoT: The Power Combo

Instead of just showing input → output examples, show input → reasoning → output examples. The AI will follow both the thinking pattern and the output format.

Few-Shot CoT for bug triage:

Classify these bug reports by priority. Think through the impact before deciding.

Example 1:
Bug: "Payment button doesn't work on checkout page"
Reasoning: This directly blocks revenue. All users trying to pay are affected. No workaround exists.
Priority: P0 - Critical

Example 2:
Bug: "User avatar is slightly pixelated on retina displays"
Reasoning: Cosmetic issue only. Doesn't affect functionality. Users on non-retina displays are unaffected. Easy workaround: use vector icons.
Priority: P3 - Low

Now classify:
Bug: "Search results don't include items added in the last 24 hours"
Reasoning: Search is a core discovery feature. Users can't find recently added items, which affects sellers' revenue and buyer experience. Partial workaround: users can browse categories directly. Affects all users who search for recent content, which is likely a significant percentage.

Priority: P1 - High

The AI didn't just classify — it reasoned through the impact, just like the examples taught it to. The reasoning makes the classification trustworthy and reviewable.

Which Technique Should I Use?

Here's a quick decision guide:

Is it simple?
Common format, no ambiguity?
→
Zero-Shot
Just ask directly
Need specific format?
Custom conventions?
→
Few-Shot
Show 2-5 examples
Complex reasoning?
Math, logic, analysis?
→
CoT
"Think step by step"
Custom format + reasoning?
Both at once?
→
Few-Shot + CoT
Show examples with reasoning
Start simple, upgrade as needed. Try zero-shot first. If the output is wrong or inconsistent, add examples (few-shot). If the reasoning is flawed, add "think step by step" (CoT). If you need both consistency and good reasoning, combine them. Don't over-engineer your first attempt.

Real-World Examples by Role

Developer: Zero-Shot

Convert this data class to use Kotlin Serialization instead of Gson annotations. Keep the same field names.

Works great — the AI knows both formats well.

QA: Few-Shot

Generate test cases following our format:

Example:
ID: TC-LOGIN-001
Title: Verify login with valid credentials
Steps: 1) Open app 2) Enter valid email 3) Enter valid password 4) Tap Login
Expected: User sees home screen

Now generate 5 test cases for the password reset flow.

The format is company-specific, so examples are essential.

Manager: CoT

We need to decide whether to build push notifications in-house or use Firebase Cloud Messaging. Think step by step: consider cost, time to implement, maintenance burden, team expertise, and scalability. Then make a recommendation.

Complex trade-off analysis benefits from structured thinking.

iOS Dev: Few-Shot + CoT

Migrate these UIKit view controllers to SwiftUI views. For each one, first explain what the original does, then show the SwiftUI equivalent.

Example:
Original: LoginViewController (handles email/password input, validation, login button)
Analysis: Simple form with two text fields and a button. State: email, password, isLoading, error.
SwiftUI: [LoginView code]

Now migrate: ProfileViewController, SettingsViewController

Combines format consistency (few-shot) with reasoning (CoT).

Hands-On: The Three-Way Challenge

Exercise: Solve the Same Problem Three Ways

Pick one of these tasks (or use your own):

  • Dev: "Review this function for potential issues" (paste a real function from your project)
  • QA: "Generate edge-case test scenarios for a login feature"
  • Manager: "Should we adopt Kotlin Multiplatform for our next project?"

Step 1 — Zero-Shot: Just ask the question directly. No examples, no "think step by step."

Step 2 — Few-Shot: Add 2-3 examples showing the output format you want.

Step 3 — CoT: Add "Think through this step by step" (and optionally define the steps).

Compare the three outputs:

  1. Which was most accurate?
  2. Which was most useful?
  3. Which took the least effort to write?

Bonus: Try the power combo — few-shot + CoT. Compare it against the best individual technique. Was the combination worth the extra effort?
