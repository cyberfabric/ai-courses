Prompt Patterns That Work

The Big Idea

A prompt pattern is like a function — the structure stays the same, only the arguments change. Eight patterns cover most daily tasks. Stop writing prompts from scratch; grab a template, fill in the [SLOTS], go.

🎯 The Eight Patterns

📚
1. Explainer
Explain [TOPIC] as if I'm a [AUDIENCE]. Focus on [ASPECT]. Use [FORMAT]. Include an analogy.
🔍
2. Reviewer
Review [THING] for [CRITERIA]. List issues as [FORMAT]. Prioritize by [METRIC]. Suggest fixes.
🏭
3. Generator
Generate [N] [THINGS] with [CRITERIA]. Format as [FORMAT]. Cover [DIVERSITY REQUIREMENT].
🔄
4. Transformer
Convert [INPUT] to [OUTPUT]. Preserve [KEEP]. Ignore [DROP]. Additional rules: [RULES].
🐛
5. Debugger
Problem: [DESC]. Expected: [X]. Actual: [Y]. Code: [CODE]. What's causing this? Fix it.
⚖️
6. Comparator
Compare [A] and [B] across [DIMENSIONS]. Table format. Recommend for [MY SCENARIO].
🗺️
7. Planner
I need to [GOAL]. Constraints: [LIST]. Step-by-step plan with [DELIVERABLE] per step. Flag risks.
📋
8. Summarizer
Summarize [CONTENT] in [FORMAT]. Focus on [AUDIENCE]. Highlight [ASPECTS]. Max: [LENGTH].

📊 When to Use Which Pattern

Pattern Use When... Secret Sauce
Explainer You need something explained at a specific level [AUDIENCE] controls depth
Reviewer You need critical feedback on code, docs, plans [CRITERIA] + "suggest a fix"
Generator You need multiple items: test data, ideas, cases [DIVERSITY REQUIREMENT]
Transformer You need format or language conversion [PRESERVE] + [IGNORE]
Debugger Something is broken and you need diagnosis Expected vs Actual + Code
Comparator You need to decide between options [MY SCENARIO] for recommendation
Planner You need a step-by-step roadmap [CONSTRAINTS] force realism
Summarizer You need to distill long content [AUDIENCE] changes what's highlighted

⚡ Pattern Power Tips

Diversity Requirement (Generator)
"Make sure to cover..." forces variety. Without it, AI generates suspiciously similar items.
Constraints (Planner)
Budget, team size, timeline, tech limits — more constraints = more realistic plan.
Preserve/Ignore (Transformer)
Saying "convert to Kotlin" gives Java-with-Kotlin-syntax. "Preserve API, ignore boilerplate, use idiomatic Kotlin" gives real Kotlin.
Expected vs Actual (Debugger)
All three pieces (problem, expected, actual) are required. Skip one and the AI guesses.
Audience (Summarizer)
Same meeting notes for a dev, PM, and CTO highlight completely different things.

🔨 Building Your Own Patterns

  1. Notice Repetition

    "I ask this type of question a lot..."

  2. Find the Variables

    What changes each time? Those become your [SLOTS].

  3. Write the Template

    Replace variables with [DESCRIPTIVE NAMES].

  4. Test & Refine

    Use it 3 times with different inputs. Adjust what doesn't work.

Share patterns with your team — wiki, Notion, CLAUDE.md, Slack channel. One person's discovery benefits everyone.

❓ Quick Quiz

What makes the Generator pattern produce diverse output instead of samey items?

The [DIVERSITY REQUIREMENT] slot: "Make sure to cover different countries, ages 18-70, mix of free/premium." Without it, AI generates suspiciously similar data.

Why does the Planner pattern need constraints?

A plan without constraints is a fantasy. Constraints like "2 devs, can't stop feature work, CI must stay green" force the AI to produce a realistic plan, not an ideal-world one.

What's the difference between the Explainer and the Summarizer?

The Explainer teaches a concept from scratch for a specific audience. The Summarizer distills existing content (meeting notes, docs, articles) into a shorter form for a specific audience.

Why does "convert this Java to Kotlin" produce bad Kotlin?

The Transformer pattern needs [PRESERVE] and [IGNORE] slots. Without them, AI does a literal translation. Adding "preserve API, ignore boilerplate, use data classes and coroutines" produces idiomatic Kotlin.

How do you build your own prompt pattern?

Notice repetition → find what changes each time (variables) → replace variables with [SLOT NAMES] → test 3 times and refine. Save and share with your team.

📌 Key Takeaways

  • Prompt patterns are reusable templates with [SLOTS] — like functions for prompting
  • Eight patterns cover most daily tasks: Explainer, Reviewer, Generator, Transformer, Debugger, Comparator, Planner, Summarizer
  • Each pattern's power comes from its specific slots: diversity (Generator), constraints (Planner), preserve/ignore (Transformer), criteria (Reviewer)
  • Patterns combine with personas (Ch 4) and format controls (Ch 5) for maximum impact
  • Build your own patterns by spotting repetition and extracting the variables
  • Share patterns with your team — prompt libraries multiply everyone's productivity
