Prompting for QA and Testing

Memory Map

The Big Idea

QA engineers are natural prompt engineers. Thinking in edge cases, questioning assumptions, and specifying expected behavior precisely — those are exactly the skills that make great prompts. AI is a QA force multiplier, not a replacement.

🧰 The QA Prompting Toolkit

📋
Test Case Generator
Feature spec + category quotas (3 happy, 3 negative, 3 edge, 2 error, 1 a11y) = comprehensive coverage.
🔮
Edge Case Discovery
"Think like a tester who breaks things. List 15 edge cases developers often forget." Follow up with "5 more" rounds.
🐛
Bug Report Writer
Paste rough notes → get Title, Severity, Environment, Steps to Reproduce, Expected/Actual, Suggested Investigation.
🗃️
Test Data Generator
Realistic JSON data with specified edge cases: Unicode names, zero values, boundary ages, negative balances.
💥
Exploratory "Break It"
"If you were trying to break this, what would you try?" Unusual inputs, rapid actions, interrupted flows, timing attacks.
🔄
Regression Analysis
"Given these changes, what existing features might break?" Returns affected features + smoke tests.
🔌
API Test Cases
Endpoint spec → test cases with request body, expected status code, and response pattern for every scenario.
♿
Accessibility Review
UI description → WCAG 2.1 AA issues with criterion, problem, and fix. First pass only — verify with real AT.

🔍 Edge Case Probing Flow

  1. Initial Ask

    "List 15 edge cases that could cause bugs. Focus on scenarios developers forget."

  2. Follow-Up Round 1

    "5 more edge cases: timezone issues, Unicode in fields."

  3. Follow-Up Round 2

    "5 more: concurrent access, mobile-specific (rotation, backgrounding, low memory)."

  4. Result

    25+ edge cases in minutes. More than most manual brainstorming sessions.

Where a developer should look first. The bonus field devs love.

❓ Quick Quiz

Why do you need "category quotas" instead of just asking for "comprehensive test cases"?

Without quotas, AI generates 80% happy-path tests and calls it comprehensive. Specifying "3 happy, 3 negative, 3 edge, 2 error, 1 accessibility" forces real diversity in test coverage.

What's the "Break It" technique and when do you use it?

"If you were trying to break [feature], what would you try?" Used for exploratory testing. Prompt the AI to think like a creative, impatient user who doesn't read instructions. Great for finding scenarios nobody planned for.

Why is "Suggested Investigation" a valuable field in AI-generated bug reports?

It gives developers a starting point for debugging instead of blind guessing. The AI can infer likely root causes from the symptoms, saving the developer time triaging the issue.

Why can't AI replace real accessibility testing?

AI can catch common issues (contrast ratios, missing labels, touch targets) but can't test actual screen reader flow, keyboard navigation, or how real users with disabilities experience the app. Use AI as a first pass, then verify with TalkBack/VoiceOver.

How do you get better test data from AI?

Explicitly specify the edge cases you need in the data: "Include Unicode names, one user exactly 18 today, a zero balance, a negative balance, maximum-length strings." The diversity requirement from the Generator pattern (Ch 6) is essential.

📌 Key Takeaways

  • QA engineers are natural prompt engineers — edge-case thinking IS the core skill
  • Category quotas force diverse test coverage: specify how many happy/negative/edge/error/a11y
  • "Break It" prompting unlocks creative exploratory testing in minutes
  • AI turns rough bug notes into structured reports with Suggested Investigation
  • Always specify edge cases in test data requests — Unicode, zeros, boundaries, negatives
  • Feed actual code diffs for targeted regression analysis
  • The QA persona primes AI for thorough testing, not happy-path developer thinking
  • AI is a first pass for accessibility — always follow up with real assistive technology
