Chapter 2

Anatomy of a Great Prompt

The five building blocks every prompt needs — and a reusable template you can start using today

From "Why" to "How"

In Chapter 1, we established that how you write a prompt matters more than which AI model you use. Now it's time to get practical. What exactly goes into a great prompt?

Turns out, every effective prompt — whether you're asking for code, a test plan, or a meeting agenda — contains the same five building blocks. Miss one, and the output suffers. Include all five, and the AI feels like it's reading your mind.

By the end of this chapter, you'll have a reusable framework you can apply to any prompt, for any task, on any AI.

The Five Building Blocks

Think of a great prompt like a great work order. You wouldn't hand a contractor a sticky note that says "fix the kitchen." You'd tell them what to do, what you're working with, what constraints you have, how you want it delivered, and maybe show them a photo of what you're going for.

Same logic. Every prompt has five parts:

Not every prompt needs all five. Sometimes the task alone is enough. But when you're not getting good results, one of these is almost always what's missing. Let's look at each one.

Block 1: Task — What Do You Want?

The Task is the most important block. It answers one question: what should the AI do?

The secret? Use action verbs. Don't describe a situation — give an instruction.

The Action Verb Rule

Start your prompt with a verb: Write, Review, Explain, Generate, Compare, Debug, Refactor, Summarize, List, Create.

A clear verb instantly tells the AI what kind of output to produce. "The login system has issues" is a statement. "Debug the login system" is a task.

Vague Task

I have some code that deals with user authentication and it could probably be better.

What does "better" mean? Faster? More secure? More readable? The AI will guess — and probably guess wrong.

Clear Task

Review this authentication code for security vulnerabilities. Focus on SQL injection, XSS, and insecure token storage.

Now the AI knows exactly what to look for and what to ignore.

Pro tip: If you can't express your task in one sentence, you probably have multiple tasks. Split them up. We'll cover the "one prompt, one task" rule later in this chapter.

Block 2: Context — What Does the AI Need to Know?

Context is the background information the AI needs to give you a relevant answer. Without it, the AI has to guess your situation — and generic situations produce generic answers.

Think about it: if a colleague asks you "how should I handle this error?", your first question would be "what error? what language? what framework? what are you building?" Context is you answering those questions before the AI has to ask.

The Context Checklist — include what's relevant:
  • Your role — "I'm a QA engineer" / "I'm a tech lead"
  • Your tech stack — "Android, Kotlin, Jetpack Compose"
  • The situation — "We're migrating from REST to GraphQL"
  • What you've tried — "I already tried increasing the timeout"
  • The audience — "This email is going to the VP of Engineering"

You don't need all of these every time. Just ask yourself: "What would a smart colleague need to know before helping me with this?"

Good Context

I'm building an Android app for a banking client. We're using Kotlin with Jetpack Compose and Hilt for DI. The app needs to work offline and sync when connectivity returns. We're currently using Room for local storage.

Suggest an architecture for the offline-first data sync layer.

The AI now knows the platform, language, frameworks, requirements, and current state. It won't suggest Swift solutions or ignore the offline requirement.

Block 3: Constraints — What Are the Rules?

Constraints are the guardrails. They tell the AI what to avoid, what limits to respect, and what rules to follow. Without constraints, the AI will give you the most "average" answer — which is rarely what you want.

Constraints come in many flavors:

Scope
"Only use the standard library" / "Don't change the public API"
Length
"Under 200 words" / "Maximum 3 bullet points"
Tone
"Professional but friendly" / "Technical, no jargon"
Exclusions
"Don't use deprecated APIs" / "Avoid third-party libraries"

The #1 Constraint Mistake: Forgetting to Say What NOT to Do

AI models are people-pleasers. If you say "write a function to parse this JSON," the AI might add error handling, logging, retry logic, and a caching layer you never asked for. Adding "Keep it simple — just parse the JSON, no error handling or extra features" saves you from deleting half the output.

Constraints in action:

Write a Kotlin function that validates an email address.
Constraints:
- Use regex, not a library
- Return a Boolean, not a Result type
- Handle only standard email formats (no international domains)
- No comments in the code — the function name should be self-documenting
- Maximum 15 lines

Notice how each constraint eliminates a possible direction the AI might have taken. Fewer surprises, better output.

Block 4: Format — How Do You Want the Output?

The Format block is the difference between getting a useful answer and getting an answer you need to reformat yourself. Tell the AI exactly what shape the output should take.

Format Options for Dev Work

  • Kotlin/Swift/Java code block
  • JSON schema
  • Markdown table
  • Numbered step-by-step
  • Git commit message
  • PR description

Format Options for Everyone

  • Bullet-point summary
  • Email draft
  • Pros/cons table
  • Slide outline
  • One-paragraph executive summary
  • Checklist

Format Makes All the Difference

Compare Room and Realm for local storage in Android apps.

Format: Markdown table with columns: Feature, Room, Realm, Winner. Include rows for: learning curve, performance, Compose integration, migration support, community size, and offline-first support.
| Feature | Room | Realm | Winner | |---------|------|-------|--------| | Learning curve | Low (SQL-based, great docs) | Medium (proprietary query lang) | Room | | Performance | Great for most apps | Faster on large datasets | Realm | | Compose integration | Native (Google-maintained) | Community-driven | Room | | Migration support | Built-in migration tools | Auto-migration | Tie | | Community size | Massive (official Google) | Large but shrinking | Room | | Offline-first | Manual sync needed | Built-in sync (Atlas) | Realm |

Without the format instruction, you'd get five paragraphs of prose. With it, you get a ready-to-paste comparison table.

The magic phrase: "Format your response as..." followed by exactly what you want. Works on every AI model, every time.

Block 5: Examples — Show, Don't Tell

Examples are the most powerful building block. When you show the AI what you mean instead of describing it, you eliminate ambiguity completely.

Think of it this way: you could spend 100 words explaining the commit message format your team uses... or you could just show two examples. The AI will get it instantly.

The "Here's What I Mean" Technique

Instead of explaining the pattern, show 2-3 examples and say "follow this pattern." The AI will infer the rules from the examples — often better than if you described them.

Examples in action:

Convert these user stories into test case titles. Follow this pattern:

Example input: "As a user, I want to reset my password via email"
Example output: "TC-AUTH-001: Verify password reset email is sent within 30 seconds"

Example input: "As a user, I want to see my order history"
Example output: "TC-ORDER-001: Verify order history displays all orders from the last 12 months"

Now convert these user stories:
1. "As a user, I want to filter products by price range"
2. "As a user, I want to save items to a wishlist"
3. "As an admin, I want to export user data as CSV"
1. TC-PRODUCT-001: Verify product list filters correctly by minimum and maximum price range
2. TC-WISHLIST-001: Verify items can be saved to wishlist and persist across sessions
3. TC-ADMIN-001: Verify user data CSV export contains all required fields and proper formatting

The AI picked up the naming convention (TC-CATEGORY-NNN), the level of detail, and the verification-focused phrasing — all from just two examples. No lengthy explanation needed.

How many examples? Two to three is the sweet spot. One example might be a coincidence. Two-three examples establish a pattern. More than five is usually overkill. We'll dive deeper into this in Chapter 3 when we cover "few-shot" prompting.

Putting All Five Blocks Together

Let's see what happens when you combine all five building blocks in a single prompt. Here's a real-world example a developer might use:

Before: Missing Most Blocks

Write an API endpoint for user profiles.

Has only a vague Task. Missing Context, Constraints, Format, and Examples. The AI will guess your framework, database, auth strategy, response shape, and error handling — and probably get most of them wrong.

After: All Five Blocks

[Task] Write a GET endpoint that returns a user's profile by ID.

[Context] This is for a Spring Boot 3 REST API using Kotlin. We use PostgreSQL with JPA. The User entity has: id, email, displayName, avatarUrl, createdAt. Authentication is handled by a JWT filter that puts the userId in the SecurityContext.

[Constraints]
- Return 404 if user not found, not an exception
- Don't expose the email field in the response
- No service layer — keep it in the controller for now
- Use ResponseEntity, not @ResponseBody

[Format] Respond with: the DTO class, the controller method, and a curl example. Use code blocks with language labels.

[Example response shape]
{ "id": 42, "displayName": "Jane", "avatarUrl": "https://...", "memberSince": "2024-01" }

Now the AI has everything it needs. No guessing. The first response will be close to production-ready.

The Order Matters

AI models pay more attention to the beginning and end of a prompt than the middle (this is called the "primacy-recency effect"). Here's the order that works best:

1 Context first — set the stage so everything that follows makes sense
2 Task second — now the AI knows what to do with the context
3 Constraints third — boundaries and rules
4 Format fourth — shape of the output
5 Examples last — the AI's freshest memory is the pattern it should follow

That said — don't overthink this. Any order that includes all five blocks beats a prompt that's missing three of them. The order is a refinement, not a requirement.

The Instruction Hierarchy

If you've used tools like Claude, Cursor, or the OpenAI API, you may have noticed there are different "layers" where you can put instructions. Here's how they rank:

Priority Order (Highest to Lowest)

  1. System prompt — Set by the app developer or tool. Highest authority. In Cursor, this is the .cursorrules file. In Claude Code, it's the CLAUDE.md file.
  2. User prompt — What you type in the chat. This is where you apply the five building blocks.
  3. Conversation context — Previous messages in the same conversation. The AI remembers these, but they carry less weight than the current prompt.
  4. Training data — The AI's general knowledge. Lowest priority — your instructions override default behavior.

When instructions conflict, higher layers win. This is why a system prompt saying "always respond in JSON" will override your request for prose — and why your explicit instruction overrides the AI's default habits.

Practical tip: If the AI keeps ignoring something you asked for, check whether a system prompt or tool configuration is overriding it. In Cursor or Claude Code, check the rules/settings files first.

The "One Prompt, One Task" Rule

This is one of the most counterintuitive rules in prompt engineering: you'll get better results from three separate prompts than from one mega-prompt.

The Mega-Prompt (Don't Do This)

Review this code for bugs, then refactor it to use coroutines instead of callbacks, then write unit tests for it, and finally write a pull request description.

The AI will rush through each part to fit everything in, and the quality of each section will suffer. It's like asking a chef to cook, serve, and clean up simultaneously.

Split Into Steps (Do This)

Prompt 1: "Review this code for bugs. List each bug with the line number, severity, and fix."

Prompt 2: "Now refactor this code to use Kotlin coroutines instead of callbacks. Keep the same public API."

Prompt 3: "Write unit tests for the refactored version. Use JUnit 5 and MockK."

Prompt 4: "Write a PR description summarizing the bug fixes and the callback-to-coroutines migration."

Each prompt gets full attention. Each output is higher quality. And you can review/adjust at each step before moving on.

When should you split? If your prompt contains the words "and then" or "also" more than once, it's time to break it up.

Three Mistakes That Kill Prompts

Mistake 1: Burying the Task

Some people write three paragraphs of context and put the actual question at the very end. The AI may lose focus by then. Lead with your task, or put it in a clearly labeled section.

Mistake 2: Conflicting Constraints

"Write a comprehensive analysis. Keep it under 100 words." Pick one. The AI will try to satisfy both and fail at both. If you need brevity AND depth, try: "Write a 100-word executive summary, then a detailed 500-word analysis below it."

Mistake 3: Assuming Context

"Fix the bug in the service." Which service? What bug? The AI doesn't have your codebase, your Jira ticket, or your debugging session in front of it. If you're pasting code, include it. If there's an error message, paste it. If the AI can't see it, it doesn't exist.

Your Prompt Template

Here's a fill-in-the-blank template you can use starting right now. Copy it, save it, bookmark it. Use it any time you're writing a non-trivial prompt.

--- Prompt Template ---

Context: I'm a [your role] working on [project/situation]. We use [relevant tech/tools]. [Any other relevant background].

Task: [Action verb] [specific thing you want done].

Constraints:
- [What to avoid]
- [Length/scope limits]
- [Style/tone requirements]

Format: Respond as [format: table / bullet list / code block / JSON / etc.].

Example: Here's what a good output looks like: [example]

You don't need to fill in every section every time. For a quick question, Task alone might suffice. But when you're not getting good results, pull out this template and fill in the blanks. The missing block is usually the problem.

Power move: Save this template as a snippet in your IDE or text expander. Type /prompt and the template appears. You'll write better prompts simply because the structure is in front of you.

Hands-On: Rebuild a Bad Prompt

Exercise: Rewrite Using All Five Blocks

Here's a bad prompt. Your job is to rewrite it using all five building blocks.

The bad prompt:
Help me with my tests.

Your task:

  1. What's the Task? What kind of help? Writing tests? Fixing tests? Reviewing test coverage?
  2. What's the Context? What language? What framework? What feature are you testing?
  3. What are the Constraints? Testing framework? Mocking strategy? Coverage target?
  4. What's the Format? Individual test functions? A full test file? A test plan?
  5. Can you add an Example? Show what one good test looks like for this project.

Write your improved prompt and send it to your favorite AI. Compare the output from "Help me with my tests" vs your five-block version. The difference will be dramatic.

Hint: A good rewrite might look something like:
"Write unit tests for the CartRepository.calculateTotal() function in our Kotlin Android app. Use JUnit 5 with MockK. Test these scenarios: empty cart, single item, multiple items, item with zero quantity, and discount code applied. Format as individual @Test functions with descriptive names."
