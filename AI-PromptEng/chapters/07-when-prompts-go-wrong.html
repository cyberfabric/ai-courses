Chapter 7

When Prompts Go Wrong

Debugging bad outputs — common failures, systematic fixes, and the iteration mindset

It's (Probably) Your Prompt

You've used every technique in this book. You added context, set a persona, specified the format, even used a pattern from Chapter 6. And the AI returned... garbage. Confidently delivered, beautifully formatted garbage.

Welcome to prompt debugging. It's a real skill, and it's one you'll use constantly.

Here's the mindset shift that separates good prompters from frustrated ones:

When the output is wrong, the prompt is the first suspect. Not the AI, not the model, not bad luck. Your prompt either said something ambiguous, assumed context the AI didn't have, or asked for something contradictory. The good news? Prompts are easy to fix once you know what to look for.

Think of it like debugging code. You don't blame the compiler when your code doesn't work — you read the error, check your assumptions, and fix the logic. Prompt debugging works the same way.

The Five Failure Modes

Bad AI output almost always falls into one of five categories. Knowing which failure mode you're dealing with tells you exactly where to look.

Let's diagnose and fix each one.

Failure 1: The AI Misunderstood the Task

You asked for X and got Y. The AI confidently did the wrong thing — not badly, but the wrong thing entirely.

The Prompt

"Review this code for me."

What you wanted: Security review focusing on SQL injection and authentication issues.

What you got: A style review suggesting variable renaming and adding comments.

The Fix

"Review this code for security vulnerabilities. Focus on: SQL injection, authentication bypass, sensitive data exposure. Ignore code style, naming conventions, and formatting."

What you get: A focused security review finding real vulnerabilities.

Diagnosis: The Ambiguity Test

Read your prompt and ask: "Could a smart person reasonably interpret this differently than I intended?" If yes, you found the bug. The fix is always the same: replace the ambiguous part with something specific. "Review" becomes "review for security vulnerabilities." "Improve" becomes "improve performance by reducing database queries." "Fix" becomes "fix the null pointer exception on line 42."

Failure 2: The AI Lacks Context

The answer is plausible but wrong because you assumed the AI knew something it didn't. This is the most common failure mode — and the sneakiest, because the output looks right.

The Failing Prompt

Write a function to validate user input for the registration form.

What You Got

fun validateInput(name: String, email: String, password: String): Boolean {
    if (name.isBlank()) return false
    if (!email.contains("@")) return false
    if (password.length < 8) return false
    return true
}

Looks reasonable, right? But your registration form also has a phone number field (required in your country), a date of birth (must be 18+), and the password policy requires uppercase, lowercase, number, AND special character. The AI didn't know any of that.

The Fixed Prompt

Write a Kotlin function to validate user input for our registration form.

Fields and rules:
- name: required, 2-50 characters, letters and spaces only
- email: required, valid email format
- phone: required, Bulgarian format (+359XXXXXXXXX)
- dateOfBirth: required, user must be 18+ years old
- password: min 10 chars, must contain uppercase, lowercase,
  digit, and special character (!@#$%^&*)

Return a Map<String, String> of field name to error message.
Return empty map if all valid.

Tip: The "What Am I Assuming?" Check

Before sending a prompt, ask yourself: "What do I know about this that the AI doesn't?" Your project's conventions, your company's requirements, your tech stack's quirks, your team's preferences — none of this is in the AI's training data. If it matters for the answer, it belongs in the prompt.

Failure 3: The AI Was Too Literal

You asked for something and got exactly what you asked for — which wasn't actually what you wanted. The AI followed your instructions to the letter but missed the spirit.

Too Literal

"Write 5 test cases for the login function."

What you got: 5 test cases that all test successful login with slightly different valid credentials. Technically 5 test cases, but useless.

Intent-Aligned

"Write 5 test cases for the login function. Include: 1 happy path, 2 negative cases (invalid credentials), 1 edge case (expired account), and 1 security case (SQL injection attempt)."

What you get: A diverse, useful test suite.

The "too literal" failure is especially common when:

  • You ask for "N items" without specifying diversity
  • You ask to "improve" code without saying what "better" means
  • You ask for "help" without defining what kind of help
  • You ask to "simplify" without specifying for whom or how much

Warning: The Obedient Genie Problem

AI is like a genie that grants wishes literally. "Make this function faster" might remove error handling. "Make this shorter" might remove important comments. "Fix the bug" might suppress the error instead of fixing the root cause. Always specify what to change AND what to preserve.

Failure 4: The AI Hallucinated

The AI made something up and presented it as fact. A function that doesn't exist. A library parameter that was never supported. A "best practice" from a blog post that was never written.

Hallucination is the most dangerous failure mode because the output is confident. There's no hesitation, no "I'm not sure." Just wrong information delivered with the same certainty as correct information.

A Hallucination-Prone Prompt

What's the best Android library for real-time face mesh detection
with Kotlin Multiplatform support?

Potential Hallucinated Output

I recommend FaceMeshKMP (version 2.4.1), a Kotlin Multiplatform
library that wraps Google's ML Kit on Android and Apple's Vision
framework on iOS. Install with:

implementation("io.facemesh:facemesh-kmp:2.4.1")

It supports real-time detection at 30fps and has built-in
ARCore integration...

This library doesn't exist. The dependency coordinates are made up. The version number is fabricated. But it sounds completely plausible.

Anti-Hallucination Techniques

  1. "If you're not sure, say so." Add this to any factual prompt. It gives the AI permission to be uncertain instead of inventing.
  2. "Only recommend tools you're confident exist." For library/tool recommendations specifically.
  3. "Cite your reasoning." Asking the AI to explain why it recommends something exposes weak reasoning faster than asking what it recommends.
  4. Verify independently. For any critical factual claim, check the source yourself. AI is a research assistant, not an oracle.

Tip: Where Hallucination Hides

Hallucination is most common in: specific version numbers, library names, API parameters, recent events, statistics, quotes from people, and URLs. It's least common in: well-established concepts, widely-known algorithms, mainstream language syntax, and general best practices. Calibrate your trust accordingly.

Failure 5: The Output Format Is Wrong

The content is correct but wrapped in the wrong package. You wanted JSON, you got prose. You wanted a table, you got bullet points. You wanted code only, you got code plus a three-paragraph explanation.

This one's usually the easiest to fix. Chapter 5 covered this in depth, but here's the quick debugging checklist:

Format Debugging Checklist

  • Is the format instruction at the top of the prompt? (Not buried at the end?)
  • Is the format instruction specific enough? ("JSON" vs "JSON array with fields: ...")
  • Are there conflicting instructions? ("Explain in detail" + "respond in one line")
  • Did you use the "ONLY" technique? ("Respond ONLY with the JSON.")
  • Did you reinforce at the end? ("Remember: raw JSON, no markdown fences.")

The Five-Question Debugging Checklist

When any prompt gives bad output, run through these five questions. They'll identify the problem 90% of the time.

1. What did I actually ask? Re-read your prompt as a stranger would.
→
2. What context is missing? What do I know that the AI doesn't?
→
3. Is anything ambiguous? Could this be interpreted two ways?
→
4. Did I specify format? Did I say HOW to respond?
→
5. Am I trusting facts? Should I verify this independently?

The "Explain Your Reasoning" Technique

When the AI gives you a suspicious answer but you can't tell where it went wrong, ask it to show its work. This is like using a debugger — you step through the logic and find where the reasoning breaks.

Prompt

You recommended using SharedPreferences for storing the user's
authentication token. Explain your reasoning step by step:
- What alternatives did you consider?
- Why did you choose this over the alternatives?
- What are the security implications?

AI Output (Exposing the Flaw)

My reasoning:
1. I considered SharedPreferences, EncryptedSharedPreferences,
   and Android Keystore.
2. I chose SharedPreferences because it's the simplest to
   implement and widely used for key-value storage.
3. Security implications: SharedPreferences stores data in
   plain text XML on the device filesystem. On rooted devices,
   this is trivially readable. For authentication tokens,
   this is actually NOT secure enough.

**Correction:** I should have recommended EncryptedSharedPreferences
or Android Keystore for auth tokens. SharedPreferences is
appropriate for non-sensitive preferences only.

Why This Works

Asking "explain your reasoning" forces the AI to make its assumptions explicit. You can then spot exactly where the logic went wrong. It's the prompt engineering equivalent of adding log statements to your code. Works especially well when the AI's recommendation feels off but you can't articulate why.

Iterative Refinement: The Prompt → Review → Adjust Loop

Prompt engineering is not "write the perfect prompt on the first try." It's an iterative loop, just like writing code.

Write Draft your prompt
→
Run Send it to the AI
→
Review Evaluate the output
→
Diagnose Which failure mode?
→
Adjust Fix the specific issue

Most good prompts take 2-3 iterations. That's normal, not a sign of failure. The key is to make targeted adjustments, not random changes.

Random Adjustment

The output was too generic, so you... rewrite the entire prompt from scratch, add a persona, change the format, and double the context.

Problem: You changed 4 things. If the output improves, you don't know which change helped. If it gets worse, you can't undo one thing.

Targeted Adjustment

The output was too generic, so you... add one specific constraint: "Focus specifically on Android 14 behavior, not general Android advice."

Benefit: One change, one variable. You can measure whether this specific fix helped.

Tip: When to Start Over vs. Iterate

Keep iterating if the output is partially right — the AI understood the general task but needs refinement. Start over if the output is completely off-track — the AI fundamentally misunderstood what you're asking. A good rule: if 3 iterations haven't improved things, the prompt structure itself is wrong. Rewrite from scratch using a different approach or pattern.

Negative Prompting: What NOT to Do

Sometimes the clearest way to get what you want is to tell the AI what you don't want. This is negative prompting, and it's surprisingly effective.

Positive Only

"Write a technical explanation of dependency injection in Android."

Result: A textbook explanation full of jargon, abstract patterns, and no practical code.

With Negative Constraints

"Write a technical explanation of dependency injection in Android. Do NOT use abstract examples — use real Android code with Hilt. Do NOT explain what DI is in theory — assume I know the concept and want to see it in practice. Do NOT mention Dagger 1 or manual DI."

Result: Practical, code-heavy, focused explanation using Hilt.

Negative prompting works well for:

  • Blocking common but unhelpful AI defaults ("Don't add logging or comments")
  • Avoiding deprecated approaches ("Don't use AsyncTask or Java threads")
  • Skipping obvious content ("Don't explain what REST is")
  • Preventing scope creep ("Don't add features I didn't ask for")

Prompt Versioning: Track What Works

When you're iterating on an important prompt, keep track of what you tried. You'd never write code without version control — important prompts deserve the same discipline.

A Simple Version Log

## Bug Triage Prompt

### v1 (too vague)
"Analyze this bug report and categorize it."
→ Result: Generic categories, no actionable output.

### v2 (added structure)
"Analyze this bug report. Categorize as: UI / Logic / Crash / Performance.
Rate severity: P0-P3. Suggest the most likely root cause."
→ Result: Good categories but wrong severity for edge cases.

### v3 (added examples) ✅
"Analyze this bug report. [same as v2] + Here are examples of
each severity: P0 = app crash on launch, P1 = feature broken for
all users, P2 = feature broken for some users, P3 = cosmetic."
→ Result: Accurate categorization and severity. Using this one.

Tip: Where to Keep Prompt Versions

For personal prompts: a notes app or text file. For team prompts: your project wiki, Notion, or even a prompts/ directory in your repo. For production prompts (in code that calls an API): treat them like code — version control, code review, and testing.

The Temperature Factor

If you're using AI through an API (or a tool that exposes the setting), temperature is a dial that controls randomness. It doesn't fix bad prompts, but understanding it helps you get more consistent results.

Temperature Behavior Best For
0.0 Most deterministic, same input → same output Code generation, JSON output, factual answers
0.3-0.5 Slight variation, mostly consistent Technical writing, analysis, most work tasks
0.7-1.0 More creative, varied, surprising Brainstorming, creative writing, idea generation
1.5+ Highly random, potentially incoherent Rarely useful — avoid unless experimenting

Warning: Temperature Doesn't Fix Bad Prompts

If your prompt gives wrong answers at temperature 0.7, lowering to 0.0 won't make the answers right — it'll just make them consistently wrong. Fix the prompt first, then adjust temperature for the right amount of variation. When using chat interfaces (Claude.ai, ChatGPT), you typically can't control temperature directly, so prompt quality matters even more.

Hands-On Exercise

Exercise: Debug This Prompt

Here's a prompt that gives inconsistent, mediocre results. Your job: diagnose the failure modes and fix them systematically.

The Broken Prompt

Help me with my code. It's not working right and I need it
fixed. Also make it better. Here's the code:

fun loadData() {
    val data = api.getData()
    updateUI(data)
}

Step 1: Identify the failure modes. Which of the 5 failure modes does this prompt trigger? (Hint: it hits at least 3.)

Step 2: Apply the debugging checklist. Go through all 5 questions and write down what's missing.

Step 3: Rewrite the prompt. Fix every issue you found. Use techniques from this chapter: be specific about the task, add context, specify format, use negative prompting if needed.

Step 4: Compare results. Run both the original and your fixed version. How different are the outputs?

Bonus: Version your rewrite. If the first fix isn't perfect, iterate: make one targeted change at a time and track what works.

Key Takeaways

  • Five failure modes: misunderstood task, missing context, too literal, hallucination, wrong format
  • The ambiguity test: "Could a smart person interpret this differently?" — if yes, make it specific
  • The assumption check: "What do I know that the AI doesn't?" — if it matters, include it
  • "Explain your reasoning" exposes flawed logic, like adding log statements to your code
  • Iterate with targeted changes — fix one thing at a time, not everything at once
  • Negative prompting ("Don't...") blocks unwanted defaults and scope creep
  • Version your prompts — track what works, reuse what's proven
  • Temperature controls randomness, not quality — fix the prompt first
