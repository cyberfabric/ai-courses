Chapter 11

Model Differences and Cross-Platform Tips

Claude vs GPT vs Gemini — what works where, and how to write portable prompts

Not All AIs Are Created Equal

You've spent 10 chapters learning how to write great prompts. Here's the twist: the same prompt can give you brilliant results on one model and mediocre results on another.

It's like driving different cars. The steering wheel, pedals, and mirrors are in roughly the same place, but a Tesla handles differently from a Toyota. You can drive both, but you'll get the best performance if you understand each one's quirks.

This chapter is your map of the AI model landscape — what each major model does well, where they differ, and how to write prompts that work across all of them.

Why this matters right now: Most teams use multiple AI tools. You might use Claude in your IDE, ChatGPT for brainstorming, and Gemini for Google Workspace tasks. Writing prompts that only work on one model wastes the skills you've built.

Why the Same Prompt Gives Different Results

Every model is trained on different data, with different techniques, and with different goals. This creates real behavioral differences:

The Same Prompt on Three Models 
Review this Kotlin function for potential issues:

fun getUser(id: String): User {
    val result = db.query("SELECT * FROM users WHERE id = '$id'")
    return User(result.name, result.email, result.password)
}
Claude's Typical Response

Structured, thorough. Lists SQL injection vulnerability first, then password exposure, then missing null handling. Provides corrected code with parameterized queries. Mentions security best practices.

GPT's Typical Response

Covers similar issues but may also suggest ORM alternatives, link to documentation, and provide multiple solution approaches. Can be more verbose.

Gemini's Typical Response

Often includes security context from Google's perspective. May reference Android-specific patterns (Room, Content Providers) if it infers you're building for Android.

Open-Source (Llama) Response

Catches the major issues but may miss nuances. Less likely to proactively mention best practices you didn't ask about. Quality varies significantly by model size.

None of these responses are "wrong" — they just reflect different training priorities and styles.

Model Strengths: When to Use What

Claude (Anthropic)

🟣
Best at
  • Instruction following — Claude is exceptionally good at following complex, multi-step instructions precisely
  • Long documents — Up to 200K tokens of context. Great for analyzing codebases, long documents, or large datasets
  • Structured output — JSON, XML, tables, formatted reports — Claude follows format instructions reliably
  • Nuanced analysis — Code review, document review, decision analysis with balanced perspectives
  • Safety and honesty — More likely to say "I'm not sure" than hallucinate confidently

GPT (OpenAI)

🟢
Best at
  • Broad knowledge — Massive training data means strong performance across diverse domains
  • Tool use and plugins — The largest ecosystem of integrations (web browsing, DALL-E, code interpreter)
  • Creative writing — Generally strong at marketing copy, storytelling, and creative tasks
  • Reasoning models — o1/o3 models excel at math, logic, and complex multi-step reasoning
  • Image understanding and generation — Native multimodal with strong vision capabilities

Gemini (Google)

🔵
Best at
  • Google ecosystem — Deep integration with Gmail, Docs, Sheets, Calendar, Search
  • Multimodal — Native understanding of images, video, and audio in a single model
  • Large context — Up to 1M+ tokens for some versions, great for processing entire codebases
  • Speed — Flash models are extremely fast for real-time applications
  • Factual grounding — Can ground responses in Google Search results

Open-Source Models

🟠
Best at
  • Privacy — Run locally, no data leaves your machine. Critical for sensitive codebases
  • Customization — Fine-tune on your own data for domain-specific tasks
  • Cost at scale — No per-token API charges once you have the infrastructure
  • Offline use — Works without internet. Good for air-gapped environments
  • Transparency — You can inspect the model weights and training process

The Model Selection Shortcut

Don't overthink it. Here's a practical decision framework:

  • Code + structured analysis? → Claude or GPT
  • Complex reasoning or math? → GPT o1/o3 or Claude
  • Google Workspace integration? → Gemini
  • Image/video understanding? → Gemini or GPT
  • Privacy-sensitive? → Open-source (local)
  • Maximum context length? → Gemini or Claude
  • Not sure? → Use whatever you have access to. All major models are good enough for most tasks.

Model-Specific Quirks

Each model has behaviors that can surprise you if you're coming from a different one. Here are the ones that trip people up most:

Claude Quirk

The honest refusal. Claude is more likely to say "I don't have enough information to answer this accurately" instead of guessing. This is a feature, not a bug — but it can frustrate you if you're used to models that always give an answer (even when they shouldn't).

Fix: If you want Claude to make its best guess, say "Give your best assessment even if you're not fully certain. Note your confidence level."

GPT Quirk

The eager helper. GPT tends to try to answer everything, even when it should push back. It may generate plausible-sounding but incorrect code for obscure libraries.

Fix: Add "If you're not confident this API/method exists, say so" to prompts involving specific libraries or recent tools.

Gemini Quirk

The Google-thinker. Gemini sometimes defaults to Google-specific solutions (Firebase, GCP, Android Jetpack) even when you haven't specified a platform.

Fix: Be explicit about your tech stack. "I'm using AWS, not GCP" or "This is a React web app, not Android."

Open-Source Quirk

The instruction-dropper. Smaller open-source models (7B-13B) tend to "forget" instructions mid-response, especially with complex multi-step prompts.

Fix: Keep prompts shorter and simpler. Break complex tasks into multiple sequential prompts instead of one large one.

Writing Portable Prompts

Here's the good news: 80% of good prompting works on every model. If you follow the principles from this course, your prompts will work well everywhere. The remaining 20% is model-specific optimization.

Model-Dependent Prompt

Use your artifacts feature to create a React component. Enable extended thinking for this. Search the web for the latest React 19 patterns.

This only works on specific models with specific features enabled.

Portable Prompt

Create a React component for a user profile card.

Requirements:
- Display name, avatar, bio, and join date
- Responsive: stack vertically on mobile
- Use TypeScript with proper typing
- Follow React 19 patterns (server components if applicable)

Think through the component structure before coding.
Output: complete component code with comments explaining key decisions.

The Portable Prompt Checklist

  • Be explicit about format — Don't rely on model defaults. Specify JSON, markdown, table, etc.
  • Include context, don't assume it — Different models have different training cutoffs and knowledge gaps
  • Use "think step by step" — Chain-of-thought works on every major model
  • Specify what to include AND exclude — Negative prompting works universally
  • Don't reference model-specific features — "Artifacts," "Canvas," "Deep Research" are product features, not prompting techniques
  • Keep instructions at the top — All models give more weight to the beginning of the prompt
  • Use examples (few-shot) — Showing what you want works better than describing it on every model

System Prompts: The Differences Matter

System prompts (also called "system messages" or "developer instructions") work differently across models. If you're building an application, this matters a lot.

System Prompt Structure (Works on All Major Models) 
You are [role] for [company/product].

Your responsibilities:
- [responsibility 1]
- [responsibility 2]

Rules:
- [rule 1]
- [rule 2]

Format: [how to structure responses]

When in doubt: [fallback behavior]
Key differences in system prompt handling:
  • Claude — Strong system prompt adherence. Treats system prompt as highest-priority instructions. Supports long, detailed system prompts well.
  • GPT — Good system prompt following, but user messages can sometimes override system instructions more easily. The "developer message" role (new) is stronger than the older "system" role.
  • Gemini — System instructions are set separately from conversation. Generally follows them well but can be more flexible in interpretation.
  • Open-Source — System prompt adherence varies wildly by model. Some follow them strictly, others treat them more like suggestions.

API vs Chat Interface: Same Model, Different Behavior

Here's something that surprises many people: the same model behaves differently depending on how you access it.

Chat Interface (claude.ai, ChatGPT, etc.)

Optimized for conversation. The model has a hidden system prompt that makes it friendly, conversational, and safe. It may add disclaimers, ask clarifying questions, or pad responses with context you didn't ask for.

API Access

More raw and controllable. You set the system prompt. No hidden instructions (beyond safety training). Responses are more direct. You control temperature, token limits, and response format.

The Same Question, Two Interfaces

Chat interface response:
"Great question! SQL injection is a type of security vulnerability where... [3 paragraphs of explanation with examples and caveats]"

API response (with minimal system prompt):
"SQL injection occurs when user input is concatenated directly into SQL queries without sanitization, allowing attackers to execute arbitrary SQL commands."

Neither is "better" — they serve different purposes. The chat interface is better for exploration and learning. The API is better for integration and predictable output.

Practical implication: If you're testing prompts in ChatGPT's web interface and then deploying them via the API, expect differences. Test prompts in the same environment where they'll run in production.

Cost: How Prompts Affect Your Bill

If you're using AI through APIs (or your company is paying per-seat licenses), prompt design directly affects cost. Here's what you need to know:

Input
Your prompt (what you send). Charged per token. Long prompts = more expensive.
Output
AI's response (what you receive). Usually 2-5x more expensive per token than input.
Context
Previous conversation turns. Grows with each exchange. The hidden cost multiplier.
Tier
Bigger models cost more. Use the smallest model that gets the job done.

Cost-Saving Prompt Techniques

  • Be concise in context — Don't paste an entire file when the AI only needs 20 lines
  • Limit output length — "Respond in under 100 words" saves output tokens
  • Use smaller models for simple tasks — Haiku/Flash for classification, formatting. Opus/Pro for complex analysis
  • Cache system prompts — Claude and some other APIs support prompt caching, which reduces costs for repeated system prompts
  • Break long conversations — Start fresh instead of carrying 50 turns of context. The AI doesn't need the full history for every question
  • Batch similar requests — "Review these 5 functions" in one prompt is cheaper than 5 separate prompts with the same system instructions

The Quick Selection Guide

When you have access to multiple models, here's a practical guide for which to reach for:

Task First Choice Why
Code review Claude or GPT Strong instruction following, structured output
Complex math/logic GPT o1/o3 or Claude Reasoning-optimized models
Analyzing a 100-page doc Claude or Gemini Large context windows
Quick classification Haiku / Flash / GPT-4o-mini Fast, cheap, good enough
Image analysis GPT or Gemini Strong multimodal capabilities
Google Docs/Sheets work Gemini Native integration
Sensitive/private code Local open-source Data never leaves your machine
Creative brainstorming Any major model All are strong here; use what you have
The honest truth: For 80% of daily tasks, any major model will do a good job. Model selection matters most at the extremes: very long documents, very complex reasoning, very cost-sensitive deployments, or very privacy-sensitive data.

Exercise: Cross-Model Prompt Testing

Take one of your best prompts from an earlier chapter exercise and test it across multiple AI models. Here's how:

  1. Pick a prompt that you use regularly (code review, test generation, email drafting, etc.)
  2. Run it on 2-3 different models — at minimum, try one from a different provider than your usual
  3. Compare results using this scorecard:
    • Accuracy — Did it get the facts right?
    • Format — Did it follow your format instructions?
    • Completeness — Did it cover everything you asked for?
    • Usefulness — Could you use the output as-is?
    • Extras — Did it add anything valuable you didn't ask for?
  4. Identify gaps — If a model underperformed, what would you change in the prompt to fix it?

Bonus: Create a "universal" version of your prompt that works well on all the models you tested. What did you have to make more explicit?
