Chapter 8

Prompting for QA and Testing

Test case generation, bug report writing, exploratory testing — AI as your QA superpower

QA Engineers Are Natural Prompt Engineers

Here's a secret: if you're a QA engineer, you already think like a prompt engineer. You think in edge cases. You question assumptions. You imagine how things can break. You specify expected behavior with precision.

Those are exactly the skills that make great prompts.

This chapter gives you a toolkit of prompting techniques specifically for testing workflows. Whether you're a dedicated QA engineer, a developer who writes tests, or a tech lead reviewing test coverage — you'll find prompts you can use today.

What this chapter covers: Generating test cases from specs, finding edge cases, writing bug reports, creating test data, exploratory testing with AI, regression analysis, API testing, accessibility review, and the ultimate QA persona.

Generating Test Cases from a Feature Spec

The most common QA task with AI: turning a feature description into comprehensive test scenarios. The key is giving the AI enough context about the feature and being explicit about what "comprehensive" means.

Before: Lazy Prompt

"Write test cases for the search feature."

Result: 5 generic test cases that test "search works" and "search with no results." Misses everything interesting.

After: QA-Grade Prompt

"Write test cases for the product search feature. The search supports: text queries, category filters, price range, sort by (relevance/price/rating). Results paginate at 20 items. Include: happy path, negative cases, edge cases, performance scenarios, and accessibility. Format as a table: ID, Title, Type, Steps, Expected Result."

Result: 15+ diverse test cases covering real scenarios testers actually care about.

The Complete Test Case Generator Prompt

You are a senior QA engineer known for finding bugs nobody else finds.

Generate test cases for this feature:

**Feature:** User profile photo upload
**Platform:** Android mobile app
**Constraints:**
- Max file size: 5 MB
- Supported formats: JPG, PNG, WebP
- Photo is cropped to square before upload
- Upload happens over the network (can fail)
- User can delete their photo to revert to default avatar

Generate at least 12 test cases covering:
- Happy path (3 cases)
- Negative/validation (3 cases)
- Edge cases (3 cases)
- Error handling (2 cases)
- Accessibility (1 case)

Format as a markdown table:
| ID | Title | Type | Precondition | Steps | Expected Result |

AI Output (first 6 rows)

| ID | Title | Type | Precondition | Steps | Expected |
|----|-------|------|-------------|-------|----------|
| TC-01 | Upload valid JPG | Happy | Logged in, has camera roll access | 1. Tap profile photo 2. Select JPG <5MB 3. Crop 4. Confirm | Photo uploads, profile updates |
| TC-02 | Upload valid PNG | Happy | Logged in | 1. Tap photo 2. Select PNG 3. Crop 4. Confirm | Photo uploads successfully |
| TC-03 | Delete existing photo | Happy | Has uploaded photo | 1. Tap photo 2. Tap "Remove" 3. Confirm | Reverts to default avatar |
| TC-04 | Upload 6MB file | Negative | Logged in | 1. Select 6MB JPG | Error: "File exceeds 5MB limit" |
| TC-05 | Upload GIF file | Negative | Logged in | 1. Select GIF file | Error: "Unsupported format" or file not selectable |
| TC-06 | Upload during no network | Error | Airplane mode on | 1. Select valid photo 2. Confirm upload | Error message, photo not changed, retry option |

The Category Quota Technique

Don't just say "comprehensive test cases." Specify how many of each type you want: "3 happy path, 3 negative, 3 edge, 2 error, 1 accessibility." Without quotas, the AI generates 80% happy path and calls it comprehensive. With quotas, you get real coverage.

The Edge Case Discovery Prompt

Edge cases are where bugs live. The AI is surprisingly good at brainstorming them — if you ask the right way.

Prompt

I'm testing a "Schedule Meeting" feature in a calendar app.
Users pick a date, start time, end time, and invite participants.

Think like a tester who breaks things for a living. List 15 edge
cases that could cause bugs. Focus on scenarios that developers
often forget to handle.

For each edge case, write:
- The scenario (one sentence)
- Why it's tricky (one sentence)
- What could go wrong (one sentence)

AI Output (first 5)

1. **Meeting crosses midnight** ŌĆö User schedules 11 PM to 1 AM.
   Tricky: the end date is actually the next day.
   Could go wrong: stored as same-day, meeting shows as negative duration.

2. **Meeting during DST change** ŌĆö Scheduled for 2:30 AM on
   "spring forward" night (which doesn't exist).
   Tricky: that time literally doesn't exist in local timezone.
   Could go wrong: crash, wrong time displayed, or silent shift.

3. **Invite yourself** ŌĆö User adds their own email to attendees.
   Tricky: should they appear as organizer AND attendee?
   Could go wrong: duplicate entries, double notifications.

4. **100+ attendees** ŌĆö User invites entire company.
   Tricky: email sending limits, UI rendering with long lists.
   Could go wrong: timeout, truncated list, rate-limited emails.

5. **Past date selection** ŌĆö User picks yesterday's date.
   Tricky: should the app allow scheduling in the past?
   Could go wrong: no validation, confusing calendar entries.

Tip: Follow Up with "What Else?"

After the AI gives you edge cases, ask: "Now think of 5 more edge cases you missed. Focus on: timezone issues, Unicode in fields, concurrent access, and mobile-specific scenarios." Each follow-up probes a different dimension. You can do 3-4 rounds and end up with 40+ edge cases in minutes.

Turning Observations into Bug Reports

You found a bug. Now you need to write it up so the developer can actually reproduce and fix it. AI can transform your rough notes into a structured, actionable bug report.

Prompt

Turn these rough notes into a structured bug report:

"the app crashes when I rotate the phone while a video is playing
in the detail screen. happens on pixel 7 with android 14. doesn't
crash if video is paused. started after last week's update."

Format the bug report as:
**Title:** (clear, searchable, under 80 chars)
**Severity:** (Critical/High/Medium/Low)
**Environment:** (device, OS, app version)
**Steps to Reproduce:** (numbered, precise)
**Expected Result:**
**Actual Result:**
**Notes:** (any additional context)
**Suggested Investigation:** (where a developer should look first)

AI Output

**Title:** App crashes on device rotation during video playback
in detail screen

**Severity:** High (crash, but requires specific user action)

**Environment:** Pixel 7, Android 14, app version [last update]

**Steps to Reproduce:**
1. Open the app and navigate to any item with a video
2. Tap the video to start playback
3. While the video is actively playing, rotate the device
   from portrait to landscape (or vice versa)

**Expected Result:** Video continues playing in the new
orientation without interruption.

**Actual Result:** App crashes immediately upon rotation.

**Notes:**
- Does NOT crash if the video is paused before rotation
- Regression: started after the most recent update
- Implies the issue is related to active playback state
  during configuration change

**Suggested Investigation:** Check the video player's lifecycle
handling during `onConfigurationChanged` or Fragment/Activity
recreation. The playing state likely isn't being saved and
restored. Look at recent changes to the detail screen or
video player component.

From messy notes to a professional bug report in 10 seconds. The "Suggested Investigation" field is a bonus that developers especially appreciate — it gives them a starting point instead of blind guessing.

Test Data Generation

Good testing needs realistic data. AI is excellent at generating diverse, realistic-looking test data — especially when you specify the edge cases you need.

Prompt

Generate 10 realistic user profiles for testing a banking app.
Format as JSON array.

Each profile needs: firstName, lastName, email, dateOfBirth
(YYYY-MM-DD), accountBalance (number), accountType ("checking"
or "savings"), isVerified (boolean).

Requirements:
- Include at least 2 names with Unicode characters (accents, etc.)
- Include one user who is exactly 18 years old today
- Include one user with a 0.00 balance
- Include one user with a negative balance (-overdraft)
- Mix of verified and unverified accounts
- Realistic age distribution (18-80)
- Balances from -500 to 250000

Return ONLY the JSON. No explanation.

Tip: Specify the Edge Cases in Your Data

Don't just ask for "10 realistic users." Explicitly request the edge cases you need: zero balances, boundary ages, Unicode names, empty fields, maximum-length strings, negative values. The Generator pattern from Chapter 6 + explicit diversity requirements = perfect test data every time.

Exploratory Testing with AI

Exploratory testing is about creative destruction — trying things nobody planned for. AI makes an excellent brainstorming partner for this.

The "Break It" Technique

Use this prompt template to get the AI thinking like a creative tester: "If you were trying to break [feature], what would you try? Think like a user who's creative, impatient, and doesn't read instructions. Include: unusual inputs, rapid actions, interrupted flows, and device-specific scenarios."

Prompt

If you were trying to break a "Forgot Password" flow in a mobile
app, what would you try?

The flow: user enters email ŌåÆ receives a reset link ŌåÆ clicks
link ŌåÆ enters new password ŌåÆ confirmation.

Think like a user who's creative, impatient, and doesn't read
instructions. Include:
- Unusual inputs
- Rapid/repeated actions
- Interrupted flows
- Timing attacks
- Multi-device scenarios

List 15 things to try. For each, explain what might break.

AI Output (first 7)

1. **Request reset 20 times in 1 minute** ŌĆö no rate limiting?
   Floods user's inbox, potential DoS on email service.

2. **Click reset link on a different device** ŌĆö does the link
   know it was requested from Device A? Should it work on Device B?

3. **Click the link twice** ŌĆö first click resets password. Second
   click: does it show an error or allow another reset?

4. **Request reset, then change password normally, then click
   the reset link** ŌĆö is the old link invalidated?

5. **Enter the new password but kill the app before confirmation**
   ŌĆö was the password actually changed? Half-completed state?

6. **Use the reset link 48 hours later** ŌĆö does it expire?
   What's the error message if expired?

7. **Enter SQL injection in the email field** ŌĆö does the backend
   sanitize input, or does `'; DROP TABLE users;--` cause chaos?

Regression Analysis: What Might Break?

When the team ships changes, a QA engineer's first question is: "What existing functionality might this break?" AI can help you think through the blast radius.

Prompt

We're making these changes to our Android e-commerce app:
1. Migrating from RecyclerView to LazyColumn (Compose)
2. Adding a new "Wishlist" feature with heart icon on product cards
3. Changing the payment flow from 3 steps to 2 steps

For each change, list:
- 3 existing features most likely to be affected
- What specifically could break
- A quick smoke test to verify it still works

Format as a table: Change | Affected Feature | Risk | Smoke Test

AI Output (first 4 rows)

| Change | Affected Feature | Risk | Smoke Test |
|--------|-----------------|------|------------|
| RecyclerViewŌåÆLazyColumn | Product list scrolling | Scroll position lost on back navigation, performance regression on large lists | Scroll down 50+ items, navigate to detail, press back ŌĆö verify position |
| RecyclerViewŌåÆLazyColumn | Pull-to-refresh | Compose swipe-refresh may behave differently than SwipeRefreshLayout | Pull to refresh on product list, verify indicator + data reload |
| Wishlist heart icon | Product card tap target | Heart icon may intercept taps meant for the product card itself | Tap near the heart area ŌĆö verify product detail opens, not wishlist toggle |
| Payment 3ŌåÆ2 steps | Promo code entry | If promo code was on the removed step, where does it go now? | Apply a promo code in the new flow ŌĆö verify discount appears |

Tip: Feed It the Actual Diff

For even more targeted regression analysis, paste the actual code diff (or PR description) into the prompt. The AI can identify specific lines of code that might affect other features. This works especially well with Claude's large context window — you can paste entire files.

API Testing Prompts

AI is excellent at generating API test scenarios — especially the edge cases that are tedious to think of manually.

Prompt

Generate API test cases for this endpoint:

POST /api/v1/users
Request body: { "name": string, "email": string, "role": "admin" | "user" }
Responses: 201 (created), 400 (validation), 409 (duplicate email), 500 (server error)

For each test case, provide:
- Description
- Request body (as JSON)
- Expected status code
- Expected response pattern

Cover: valid requests, each validation error, boundary values,
malicious input, and missing fields. At least 10 test cases.

This generates test cases you can directly translate into Postman collections, REST Client files, or automated API tests.

Accessibility Testing with AI

Accessibility is often undertested because it requires specialized knowledge. AI can bridge that gap by reviewing UI descriptions for common WCAG issues.

Prompt

Review this Android screen for accessibility issues:

Screen: Product checkout
Elements:
- "Back" button (icon only, no text)
- Product image (decorative)
- Product name: "Nike Air Max 90" (16sp, #333333 on #FFFFFF)
- Price: "$129.99" (14sp, #FF6600 on #FFFFFF)
- Quantity selector: minus button, number, plus button
- "Place Order" button (#FF6600 background, white text, 44dp height)

Check against WCAG 2.1 AA. For each issue found, provide:
- The element
- The WCAG criterion violated
- Why it's a problem
- How to fix it

AI Output (first 3 issues)

1. **Back button (icon only)**
   Criterion: 1.1.1 Non-text Content
   Problem: Screen readers can't announce the button's purpose.
   Fix: Add `contentDescription="Navigate back"` to the ImageButton.

2. **Price color #FF6600 on #FFFFFF**
   Criterion: 1.4.3 Contrast (Minimum)
   Problem: Contrast ratio is 3.0:1 ŌĆö below the 4.5:1 minimum
   for text smaller than 18sp.
   Fix: Darken to #CC5200 (4.6:1) or increase font to 18sp+.

3. **Quantity minus/plus buttons**
   Criterion: 2.5.5 Target Size
   Problem: If buttons are smaller than 44x44dp, they're hard
   to tap for users with motor impairments.
   Fix: Ensure minimum 48x48dp touch target (Android default).

Warning: AI Can't Replace Real Accessibility Testing

AI can catch many common issues (contrast, labels, touch targets), but it can't test screen reader flow, keyboard navigation, or real assistive technology behavior. Use AI for the first pass, then test with TalkBack/VoiceOver and real users with disabilities.

The Ultimate QA Persona

Remember personas from Chapter 4? Here's a QA persona you can reuse across all your testing prompts. Save it somewhere accessible and paste it at the top of any QA-related prompt.

The QA Persona

You are a senior QA engineer with 12 years of experience in
mobile app testing (Android and iOS). You're known for:
- Finding bugs nobody else finds
- Thinking about edge cases obsessively
- Writing bug reports so clear that developers thank you
- Questioning every assumption and "happy path"
- Considering accessibility, performance, and security
  alongside functional correctness

Your testing philosophy: "If it can go wrong for a user, it
will. My job is to find it before they do."

When generating test cases, you always include negative cases,
boundary conditions, and at least one scenario involving
interrupted flows (network loss, app backgrounding, low memory).

Prepend this to any prompt in this chapter and watch the output quality jump. The persona primes the AI to think like a thorough, experienced tester rather than a developer writing happy-path checks.

Hands-On Exercise

Exercise: Generate a Complete Test Plan

Pick a feature you're currently working on (or use this one: "Add to Cart" button in an e-commerce app). Generate a complete test plan using the techniques from this chapter.

Step 1: Test cases. Use the Category Quota technique to generate at least 12 test cases: 3 happy path, 3 negative, 3 edge, 2 error handling, 1 accessibility.

Step 2: Edge case brainstorm. Use the "Break It" technique to find 10 additional edge cases the test cases missed.

Step 3: Test data. Generate 8 realistic product records as JSON, including edge cases: zero price, very long product name, special characters, out-of-stock item.

Step 4: Regression check. List 5 existing features that "Add to Cart" might affect, with smoke tests for each.

Bonus: Prepend the QA persona to all four prompts. Compare results with and without it — is there a noticeable difference?

Key Takeaways

  • QA engineers are natural prompt engineers — thinking in edge cases IS the skill
  • Category quotas force diverse test cases: specify how many happy/negative/edge/error/a11y
  • "Break It" prompting unlocks creative exploratory testing: "If you were trying to break this..."
  • Bug reports from notes — AI turns rough observations into structured, actionable reports
  • Specify edge cases in test data — "include Unicode names, zero balance, boundary ages"
  • Regression analysis — ask "what existing features might this change break?"
  • The QA persona primes the AI to think like a thorough tester, not a happy-path developer
  • AI is a first pass for accessibility — always follow up with real assistive tech testing
