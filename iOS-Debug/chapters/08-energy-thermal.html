Chapter 8

Energy & Thermal Optimization

Use AI to find the silent battery drains, thermal throttling triggers, and background energy hogs hiding in your app

In Chapter 7, we made builds fast. Now we’re tackling something users care about even more than launch time: battery life. Nobody writes a 1-star review saying “your build takes 90 seconds.” But “this app drains my battery in 2 hours”? That review writes itself.

Energy bugs are uniquely insidious because they’re invisible during development. Your phone is plugged into your Mac while you test. You never see the battery impact. Then users in the real world — on cellular, in low-signal areas, with background refresh enabled — experience a completely different energy profile.

Why Energy Optimization Matters

🔋
“Battery drain” is the #2 reason for 1-star reviews (after crashes)
🌡️
Thermal throttling can drop CPU performance by 50-70%
📱
iOS deprioritizes battery-hungry apps in multitasking
📊
Users check Settings > Battery and blame the top app by percentage
The thermostat analogy: Think of your app’s energy usage like a household thermostat. The goal isn’t to never turn on the heater — it’s to heat efficiently. Burst work (CPU spike then idle) is like quick heating. Constant low-level polling is like leaving the heater on 24/7 at low — it costs more overall and never lets the system rest.

The Energy Profiling Toolkit

📊

Xcode Energy Gauge

The quick-look tool. Debug Navigator > Energy Impact shows a real-time bar chart: CPU, Network, Location, GPU, Background. Red bars = high energy. Look here first during development.

⚡

Power Profiler (Xcode 26)

The new deep-dive instrument. Shows subsystem-level power attribution: CPU, GPU, Display, Networking, Location. Correlates energy use with code execution and thermal state. Use wireless debugging — cable charging zeroes out power metrics.

🏥

MetricKit Energy Metrics

Production data. MXMetricPayload includes MXCPUMetric, MXGPUMetric, MXDiskIOMetric, MXNetworkTransferMetric. Delivered daily from opted-in users. The truth about real-world energy use.

📈

Xcode Organizer — Energy Tab

Aggregated energy data from App Store users. Shows trends across versions. Catch energy regressions when a new build ships.

Critical setup for Power Profiler: You must use wireless debugging when profiling energy. When the device is plugged into your Mac via USB, it’s charging — and power metrics report zero. Connect via WiFi (Window > Devices and Simulators > Connect via network) to get accurate readings.

MetricKit: Energy Truth from the Field

MetricKit delivers daily summaries of your app’s resource usage from real users. Here’s how to collect energy metrics:

import MetricKit

class MetricsManager: NSObject, MXMetricManagerSubscriber {
    func didReceive(_ payloads: [MXMetricPayload]) {
        for payload in payloads {
            // CPU usage
            if let cpu = payload.cpuMetrics {
                logger.info("CPU time: \(cpu.cumulativeCPUTime)")
                logger.info("CPU instructions: \(cpu.cumulativeCPUInstructions)")
            }
            // GPU usage
            if let gpu = payload.gpuMetrics {
                logger.info("GPU time: \(gpu.cumulativeGPUTime)")
            }
            // Network transfer
            if let net = payload.networkTransferMetrics {
                logger.info("Upload: \(net.cumulativeCellularUpload)")
                logger.info("Download: \(net.cumulativeCellularDownload)")
            }
            // Disk I/O
            if let disk = payload.diskIOMetrics {
                logger.info("Writes: \(disk.cumulativeLogicalWrites)")
            }
        }
    }
}

Feed this data to Claude Code for trend analysis. A sudden spike in cumulativeCPUTime after a release means your new code is working the CPU harder — and draining battery faster.

Reading an Energy Profile with AI

Power Profiler Output — News Feed Browsing Session

Session: 5 minutes of scrolling news feed
Device: iPhone 15 Pro, iOS 18.4, WiFi, 72% battery

Subsystem Breakdown:
  CPU:        42% of total energy
  Networking: 31% of total energy
  GPU:        15% of total energy
  Display:    8% of total energy
  Location:   4% of total energy

CPU Wake Pattern:
  Average wakes/minute: 847
  Peak wakes/minute: 2,134 (during background refresh)
  Idle periods: 0 (CPU never sleeps during session)

Network Pattern:
  Requests: 312 in 5 minutes (1.04/second)
  Cellular fallback events: 0
  Average payload: 4.2 KB per request

Thermal State: .nominal → .fair at 3:42 mark

AI Analysis

Three energy issues identified:

  • CPU never idles (847 wakes/min): The CPU should have idle periods between scroll events. Constant waking suggests a timer or polling mechanism that runs even when the user isn’t interacting. Likely culprit: a Timer or CADisplayLink running in the background for animation that should pause when off-screen.
  • 312 network requests in 5 minutes: That’s 1 request per second. For a news feed, this suggests individual image requests without batching, or polling for new content instead of using push notifications or server-sent events.
  • Thermal state rose to .fair: After less than 4 minutes of scrolling, the device is warming up. This correlates with the high CPU wake count. When thermal state reaches .serious or .critical, iOS throttles CPU — your app gets slower the longer it runs.

The 5 Silent Battery Drains

Drain 1: Aggressive Location Tracking

Root Cause: desiredAccuracy = kCLLocationAccuracyBest

Symptom: Location Services indicator stays on continuously. Users report battery drain even when app is in background.

Cause: Using kCLLocationAccuracyBest activates GPS hardware continuously. A weather app doesn’t need GPS-level accuracy — city-level is fine.

Fix: Use kCLLocationAccuracyKilometer for city-level, kCLLocationAccuracyHundredMeters for neighborhood-level. Use startMonitoringSignificantLocationChanges() instead of continuous tracking. For background: allowsBackgroundLocationUpdates = false unless absolutely necessary.

Pattern: Location Accuracy by Use Case

// BEFORE: GPS-level accuracy for a weather app 🔋💀
locationManager.desiredAccuracy = kCLLocationAccuracyBest
locationManager.startUpdatingLocation()

// AFTER: City-level accuracy with significant changes
locationManager.desiredAccuracy = kCLLocationAccuracyKilometer
locationManager.distanceFilter = 1000  // Only update every 1km
locationManager.startMonitoringSignificantLocationChanges()

// For precise navigation needs only:
locationManager.desiredAccuracy = kCLLocationAccuracyBestForNavigation
locationManager.activityType = .automotiveNavigation
// And STOP when navigation ends:
locationManager.stopUpdatingLocation()

Drain 2: Timer/Polling That Never Stops

A Timer.scheduledTimer(withTimeInterval: 1.0, ...) that polls an API every second is the energy equivalent of checking the mailbox every 60 seconds. Use push notifications or WebSockets instead, and always invalidate timers when the view disappears.

Drain 3: Background App Refresh Abuse

BGAppRefreshTask should fetch only essential data. If your refresh handler downloads 20 MB of content that the user may never see, you’re wasting battery. Keep background fetches under 30 seconds and under 1 MB.

Drain 4: Unnecessary GPU Work

Complex gradients, blur effects (.blur(radius:)), real-time shadows, and continuous animations drain the GPU. If an animation isn’t visible (scrolled off-screen), pause it. Use drawingGroup() for complex but static visual compositions.

Drain 5: Uncompressed Network Payloads

Sending and receiving uncompressed JSON over cellular burns radio energy. Enable gzip compression on your server and use URLSessionConfiguration.default which automatically handles Accept-Encoding: gzip. For images, serve WebP or HEIC instead of PNG.

Thermal State: When Your App Makes the Phone Hot

iOS provides ProcessInfo.thermalState with four levels. Your app should respond:

Thermal State What It Means Your App Should
.nominal Everything is fine Full features, full quality
.fair Slightly elevated temperature Reduce background work, lower animation frame rate
.serious System is throttling CPU/GPU Disable non-essential features, stop prefetching, reduce image quality
.critical System may force-suspend apps Minimal operation only. Save state. Prepare for suspension. 
// Monitor thermal state changes
NotificationCenter.default.addObserver(
    forName: ProcessInfo.thermalStateDidChangeNotification,
    object: nil, queue: .main
) { _ in
    switch ProcessInfo.processInfo.thermalState {
    case .nominal:
        enableAllFeatures()
    case .fair:
        reduceBackgroundWork()
    case .serious:
        disableNonEssentialFeatures()
        reduceImageQuality()
    case .critical:
        enterMinimalMode()
        saveState()
    @unknown default:
        break
    }
}
Testing thermal states: You can simulate thermal pressure in the iOS Simulator (not available for real devices). In Xcode: Debug > Simulate Thermal State > choose a level. Test that your app degrades gracefully at each level.

Background Tasks: Energy-Efficient Scheduling

Task Type Use For Energy Budget
BGAppRefreshTask Quick data sync (headlines, badges, unread counts) ~30 seconds, <1 MB. System schedules at optimal times.
BGProcessingTask Heavy work (ML model training, database cleanup, backup) Minutes. Runs when charging + WiFi. Set requiresExternalPower = true.
BGContinuedProcessingTask (iOS 26) User-initiated tasks that continue in background with progress UI Extended. Shows progress indicator to user.

Pattern: Energy-Efficient Background Fetch

// Register in application(_:didFinishLaunchingWithOptions:)
BGTaskScheduler.shared.register(
    forTaskWithIdentifier: "com.myapp.refresh",
    using: nil
) { task in
    guard let refreshTask = task as? BGAppRefreshTask else { return }

    // Keep it quick — 30 seconds max
    let fetchTask = Task {
        do {
            let unreadCount = try await api.fetchUnreadCount()
            UNUserNotificationCenter.current()
                .setBadgeCount(unreadCount)
        } catch {
            // Don't retry in background — wait for next scheduled refresh
        }
    }

    refreshTask.expirationHandler = {
        fetchTask.cancel()
    }

    // Schedule next refresh
    scheduleAppRefresh()

    await fetchTask.value
    refreshTask.setTaskCompleted(success: true)
}

func scheduleAppRefresh() {
    let request = BGAppRefreshTaskRequest(
        identifier: "com.myapp.refresh"
    )
    request.earliestBeginDate = Date(timeIntervalSinceNow: 15 * 60)
    try? BGTaskScheduler.shared.submit(request)
}

Build It: The /analyze-energy Skill

/analyze-energy — Claude Code Skill

Create .claude/commands/analyze-energy.md:

# /analyze-energy

Analyze iOS energy profiling data. The user will provide one or more of:
Power Profiler output, MetricKit energy metrics, Xcode Energy Gauge
screenshots, or Swift source code for background tasks and location services.

## Analysis Steps

1. **Identify top energy consumers** — which subsystem (CPU, GPU,
   Network, Location) uses the most power?
2. **Check CPU wake patterns** — is the CPU sleeping between user
   interactions? High wake counts = timer/polling abuse.
3. **Audit location configuration** — accuracy level appropriate for
   use case? Background location enabled unnecessarily?
4. **Review background tasks** — BGTaskScheduler usage, fetch duration,
   data volume per refresh.
5. **Check thermal impact** — does the app respond to thermal state
   changes? Does usage cause thermal escalation?

## Output Format

For each issue found:
- **Subsystem:** [CPU/GPU/Network/Location]
- **Issue:** [what's draining energy]
- **Impact:** [estimated battery drain per hour]
- **Fix:** [specific code change]
- **Priority:** [high/medium/low based on user impact]

Exercise: Find Your App’s Battery Drain

Setup:

  1. Connect your iOS device via WiFi (wireless debugging — not USB)
  2. Open Instruments > Power Profiler template
  3. Start recording and use your app normally for 5 minutes (scroll, navigate, load content)

Steps:

  1. Stop recording and examine the subsystem breakdown (CPU, GPU, Network, Location)
  2. Look for the highest-energy subsystem and note when spikes occur
  3. Check if the CPU has idle periods between interactions (it should!)
  4. Export or screenshot the Power Profiler summary
  5. Feed the data to Claude Code with /analyze-energy
  6. Apply the top suggested optimization
  7. Re-profile the same 5-minute flow and compare energy usage

Bonus: Add ProcessInfo.thermalState monitoring and verify your app responds to .serious by disabling non-essential features.

Key Takeaways

  • Profile wirelessly — USB charging zeroes out Power Profiler metrics
  • The Power Profiler in Xcode 26 shows subsystem-level energy attribution correlated with code execution
  • MetricKit delivers real-world energy data daily — compare across versions to catch regressions
  • The top 5 drains: aggressive location, timer polling, background refresh abuse, GPU overwork, uncompressed payloads
  • Use significantLocationChanges over startUpdatingLocation unless you’re building a navigation app
  • Monitor ProcessInfo.thermalState and degrade gracefully at .serious and .critical
  • BGAppRefreshTask should be quick (<30s) and light (<1 MB). Use BGProcessingTask for heavy work.
  • Build the /analyze-energy skill for production energy audits across app versions
