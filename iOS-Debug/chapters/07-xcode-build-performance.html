Chapter 7

Xcode Build Performance

Use AI to analyze build timelines, find slow-to-compile expressions, and shave minutes off every build

In Chapter 6, we optimized the network layer. But every time you want to test a fix, you hit Cmd+B and wait… and wait… and wait. A 90-second build that runs 40 times a day costs you an hour of staring at the Xcode progress bar. Over a year, that’s 250 hours — more than six full work weeks — lost to compilation.

The good news: Xcode 26 introduced compilation caching that can cut build times by 24-36%. The better news: AI can analyze your build timeline and tell you exactly which files, expressions, and settings are costing you the most time.

Step 1: Read the Build Timeline

Xcode has a built-in build timeline that most developers never look at. Open it after a build: Report Navigator (Cmd+9) > select a build > click the timeline icon (the bar chart icon in the top-right).

The timeline shows every compilation task as a horizontal bar. You’re looking for:

  • Long bars: Files that take disproportionately long to compile. A single file taking 15 seconds in a 90-second build is a red flag.
  • Serialized tasks: Bars that don’t overlap. These are compilation tasks waiting for dependencies instead of running in parallel.
  • Link phase: The final linking step. If it’s longer than individual compilation, you may benefit from EAGER_LINKING.
  • Script phases: Custom build scripts (SwiftLint, code generation) that run sequentially and block compilation.
Quick win: Move SwiftLint and other linting scripts to a post-build action or a separate CI step. They don’t need to block compilation, and removing them from the critical path can save 5-10 seconds per build.

Step 2: Find Slow-to-Compile Expressions

The Swift compiler can tell you exactly which expressions take too long to type-check. Add these flags to your build settings under Other Swift Flags:

-Xfrontend -warn-long-function-bodies=200
-Xfrontend -warn-long-expression-type-checking=200

The 200 means “warn if type-checking takes more than 200ms.” After a build, you’ll see warnings like:

Xcode Build Warnings — Slow Type-Checking

⚠️ /Sources/Views/DashboardView.swift:47:12
  Instance method 'body' type-checked in 1284ms
  (threshold: 200ms)

⚠️ /Sources/Networking/ResponseMapper.swift:23:9
  Expression type-checked in 856ms
  (threshold: 200ms)

⚠️ /Sources/Models/PriceCalculator.swift:89:16
  Expression type-checked in 2341ms
  (threshold: 200ms)

AI Analysis

Pattern recognized: Complex type inference chains.

  • DashboardView.body (1284ms): SwiftUI views with deeply nested modifiers and conditional logic force the type checker to explore many possible types. Break into smaller sub-views.
  • ResponseMapper (856ms): Likely a chain of .map/.flatMap/.compactMap on collections where the compiler infers intermediate types. Add explicit type annotations to break the chain.
  • PriceCalculator (2341ms): The worst offender. Probably a long arithmetic expression or a ternary chain. The compiler explores exponential type combinations for complex expressions. Split into intermediate variables with explicit types.

Root Cause: Exponential Type Checking

Symptom: A single expression takes 2+ seconds to type-check. The file is unreasonably slow to compile.

Cause: The Swift compiler’s type checker has exponential worst-case behavior with complex expressions. When you write a + b * c - d / e + f with mixed types (Int, Double, CGFloat), the compiler explores every possible overload combination. Similarly, long chains of .map { }.filter { }.reduce { } without type annotations create cascading inference.

Fix: Break long expressions into intermediate let variables with explicit type annotations. This gives the type checker anchoring points and reduces the search space from exponential to linear.

The Type Annotation Fix

Pattern: Break Type Inference Chains

// BEFORE — 2341ms to type-check
let total = items
    .filter { $0.isActive }
    .map { $0.price * (1.0 - $0.discount) }
    .reduce(0, +) + shippingCost * taxRate

// AFTER — 12ms to type-check
let activeItems: [Item] = items.filter { $0.isActive }
let prices: [Double] = activeItems.map { item in
    item.price * (1.0 - item.discount)
}
let subtotal: Double = prices.reduce(0, +)
let total: Double = subtotal + shippingCost * taxRate

Rule of thumb: If a chain has 3+ operations or mixes numeric types, add explicit type annotations. The 2 seconds you save per build adds up to hours per week.

Xcode 26: Compilation Caching

The biggest build performance feature in years. Xcode 26 can cache compilation results and reuse them when source files haven’t changed — even across branch switches and clean builds.

1. Enable
Build Settings > COMPILATION_CACHE_ENABLE_CACHING = YES
→
2. Build
First build populates the cache (LLVM CAS storage)
→
3. Reuse
Subsequent builds skip unchanged units (sub-function granularity)

Before/After: Compilation Caching Impact

Before (No caching)
Clean build after branch switch: 4 min 12s
Incremental build (1 file changed): 47s
CI pipeline full build: 8 min 30s
After (Compilation caching enabled)
Clean build after branch switch: 2 min 41s (-36%)
Incremental build (1 file changed): 31s (-34%)
CI pipeline with shared cache: 5 min 15s (-38%)
Prerequisite: Compilation caching requires Explicit Modules to be enabled (the default in Xcode 26 for Swift targets). If you previously set SWIFT_ENABLE_EXPLICIT_MODULES=NO, you’ll need to re-enable it. Explicit modules give the build system better dependency information, which enables more precise caching.

Build Settings AI Should Audit

Feed your project’s build settings to Claude Code and let it find misconfigurations:

Setting Debug Value Release Value Why It Matters
SWIFT_COMPILATION_MODE singlefile (incremental) wholemodule Incremental for fast debug builds; whole module for optimized release
SWIFT_OPTIMIZATION_LEVEL -Onone -O or -Osize -Onone in Debug = faster compilation. Never optimize Debug builds.
DEBUG_INFORMATION_FORMAT dwarf dwarf-with-dsym dSYM generation is slow. Only generate for release/archive builds.
EAGER_LINKING YES YES Starts linking before all files finish compiling. Can save 10-20% on link time.
COMPILATION_CACHE_ENABLE_CACHING YES YES Xcode 26’s compilation cache. Saves 24-36% on clean builds.
SWIFT_ENABLE_BATCH_MODE YES YES Groups files for compilation. Reduces compiler process spawn overhead.
Common mistake: Setting SWIFT_COMPILATION_MODE = wholemodule for Debug builds. This disables incremental compilation — changing one line recompiles the entire module. Always use singlefile (incremental) for Debug.

SPM Resolution: The Hidden Time Sink

Swift Package Manager resolution can add 30-60 seconds to your first build. AI audits for these patterns:

  • Too many packages: Every package adds resolution time. Audit for packages you included for one function you could have written in 10 lines.
  • Version ranges instead of exact versions: .upToNextMajor(from: "5.0") forces SPM to check for newer versions. Pin exact versions for faster resolution.
  • Transitive dependencies: One popular package can pull in 15 transitive dependencies. Use swift package show-dependencies to see the full graph.
  • Binary targets: Use .binaryTarget for large pre-built frameworks (e.g., Firebase, AWS SDK) to skip compilation entirely.
  • Package.resolved in git: Always commit Package.resolved. Without it, every team member resolves independently, wasting time and risking version drift.

$ swift package show-dependencies --format json | \

Build It: The /analyze-build Skill

/analyze-build — Claude Code Skill

Create .claude/commands/analyze-build.md:

# /analyze-build

Analyze Xcode build performance data. The user will provide one or more of:
build timeline screenshots, compilation warnings from -warn-long-function-bodies,
build settings dumps, or SPM dependency graphs.

## Analysis Steps

1. **Identify bottlenecks** — which files take the longest to compile?
2. **Check for slow expressions** — type-checking warnings indicate
   complex inference chains that need explicit type annotations.
3. **Audit build settings** — verify Debug vs Release configurations
   are appropriate (no whole-module in Debug, no optimization in Debug).
4. **Review SPM dependencies** — excessive packages, missing binary
   targets, unpinned versions?
5. **Check for serialization** — build scripts, framework dependencies,
   or module maps that prevent parallel compilation.

## Output Format

Prioritized list of optimizations:
- **Issue:** [what's slow]
- **Impact:** [estimated time savings]
- **Fix:** [specific build setting change, code refactor, or SPM adjustment]
- **Risk:** [any downsides to this change]

Modularization: The Long-Term Fix

If your app is a single-target monolith, changing any file can trigger recompilation of everything that depends on it. Modularization breaks your code into independent modules so that changes in one module don’t recompile others.

Good Prompt — Asking AI for Module Boundaries

/analyze-build

Here's my project structure with 180 Swift files in a single target.
Build time: 90 seconds for a one-line change.

[paste file list grouped by directory]

Suggest module boundaries that would minimize recompilation cascades.
Constraints:
- We use SwiftUI + Combine
- Core Data model is shared across features
- We can't break App Store submission (single app target required)
- We want to start with 3-4 modules, not 20

Claude Code will analyze your import graph and suggest boundaries like: CoreModels (Core Data + DTOs), Networking (API layer + Codable types), FeatureHome (home screen feature), with the main app target as the composition root.

Extracting Build Timing Data for AI

# Export build timing summary
xcodebuild -scheme MyApp -configuration Debug \
  -destination 'platform=iOS Simulator,name=iPhone 16' \
  build 2>&1 | tee build.log

# Extract compilation times (requires Xcode build timing)
grep "CompileSwift" build.log | \
  awk '{print $NF, $0}' | sort -rn | head -20

Key Takeaways

  • Enable compilation caching in Xcode 26 — 24-36% faster clean builds with one build setting
  • Use -warn-long-function-bodies and -warn-long-expression-type-checking to find slow expressions
  • Break type inference chains with explicit type annotations — the #1 fix for slow Swift compilation
  • Never use wholemodule compilation mode in Debug — it kills incremental builds
  • Audit SPM dependencies — pin versions, commit Package.resolved, use binary targets for large frameworks
  • Move linting scripts to post-build or CI — they don’t need to block compilation
  • Modularize for long-term build speed — AI can suggest module boundaries from your file structure
  • Build the /analyze-build skill for holistic build performance audits
