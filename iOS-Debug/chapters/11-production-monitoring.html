Production Monitoring & Crash Reporting

No debugger. No LLDB. No Instruments. Just telemetry, crash reports, and AI to make sense of it all.

The Production Debugging Challenge

Everything you’ve learned so far happens in development — Xcode attached, simulator running, LLDB waiting for your commands. Production is a different planet. Your app is on 50,000 devices running iOS versions you didn’t test, on hardware you don’t own, in network conditions you can’t replicate. When it crashes, you get a report. When it’s slow, you get a metric. That’s it.

The good news: production data is structured. Crash reports, MetricKit payloads, Crashlytics clusters — all machine-readable, all perfect for AI analysis. The pattern is always the same: extract the data, feed it to AI, get back prioritized fixes.

Firebase Crashlytics for iOS

Crashlytics is the most popular crash reporting tool for iOS. It captures crashes automatically, groups them into clusters by root cause, and shows you which crashes affect the most users. The data it produces is exactly what AI analyzes best.

What Crashlytics Captures

  • Crash stacktraces — symbolicated with dSYMs you upload (automatic via Xcode build phase)
  • Non-fatal exceptions — caught errors you report with Crashlytics.crashlytics().record(error:)
  • Breadcrumbs — automatic screen_view events (via Google Analytics SDK) showing what the user did before the crash
  • Custom keys — key-value pairs you set to track state: Crashlytics.crashlytics().setCustomValue(userId, forKey: "user_id")
  • Device metadata — iOS version, device model, free disk space, free RAM, battery level
  • Crash clusters — grouped by root cause so you see “500 users hit this force-unwrap” not “500 individual crashes”

Adding Context for AI Analysis

// Add breadcrumb context that AI can analyze later
func configureForProductionDebugging() {
    let crashlytics = Crashlytics.crashlytics()

    // User context
    crashlytics.setUserID(currentUser.anonymizedId)
    crashlytics.setCustomValue(currentUser.subscriptionTier, forKey: "tier")

    // App state context
    crashlytics.setCustomValue(AppState.current.rawValue, forKey: "app_state")
    crashlytics.setCustomValue("\(ProcessInfo.processInfo.thermalState.rawValue)",
                               forKey: "thermal_state")

    // Feature flags
    crashlytics.setCustomValue(FeatureFlags.newCheckoutEnabled, forKey: "new_checkout")
}

// Log non-fatal errors that indicate degraded experience
func fetchProducts() async {
    do {
        products = try await api.getProducts()
    } catch {
        // Don't crash, but report the degradation
        Crashlytics.crashlytics().record(error: error)
        products = cachedProducts  // fallback
    }
}
Non-fatals are gold for AI: Crashes get all the attention, but non-fatal exceptions reveal degraded experiences — fallback code paths, retry storms, stale caches. Feed your top 10 non-fatal exceptions to Claude Code. Often they indicate bugs that haven’t crashed yet but will.

Root Cause: Force-Unwrap on Optional API Field in New Checkout Flow

Evidence: 100% of crashes have new_checkout: true. 74% are on iOS 17.6. Crash is at line 67 during “Add to Cart.”

Why: The new checkout flow accesses product.cartConfiguration! — a field added in API v3. Users on iOS 17.6 hit an older cached API response (v2) where this field is nil.

Fix: Replace force-unwrap with guard let. Add a fallback for missing cartConfiguration.

Building a /triage-crash Skill

The /triage-crash Skill

# .claude/commands/triage-crash.md

Triage this production crash report and provide an action plan.

## Input:
I'll provide crash data from Crashlytics, MetricKit, or Xcode Organizer:
- Crash stacktrace (symbolicated)
- Device and iOS version distribution
- Custom keys and breadcrumbs
- Event count and affected user count

## Analysis:
1. **Severity assessment:**
   - Critical: >1% of DAU affected, data loss risk
   - High: >0.1% of DAU, core flow blocked
   - Medium: <0.1% of DAU, workaround exists
   - Low: edge case, cosmetic impact

2. **User impact estimate:**
   - How many users are affected?
   - Is the crash in a core flow (launch, purchase, content)?
   - Is it growing or stable over time?

3. **Root cause hypothesis:**
   - What's the immediate cause (nil unwrap, OOB, etc.)?
   - What's the underlying cause (API change, race, state)?
   - Does device/OS distribution suggest a platform bug?

4. **Fix priority and plan:**
   - Immediate: hotfix needed within 24 hours
   - Next release: include in current sprint
   - Backlog: track but not urgent
   - Provide the fix with Swift code

MetricKit: Your Daily Production Health Check

While Crashlytics focuses on crashes, MetricKit gives you the full health picture — CPU, memory, disk I/O, network, launch time, hang diagnostics, and energy metrics. Apple delivers these payloads daily from consenting users. Think of it as a daily blood test for your app.

Setting Up MetricKit

import MetricKit

class MetricSubscriber: NSObject, MXMetricManagerSubscriber {
    static let shared = MetricSubscriber()

    func startReceiving() {
        MXMetricManager.shared.add(self)
    }

    // Daily metric payloads
    func didReceive(_ payloads: [MXMetricPayload]) {
        for payload in payloads {
            let json = payload.jsonRepresentation()
            // Upload to your backend for AI analysis
            AnalyticsService.upload(metricPayload: json)
        }
    }

    // Diagnostic payloads (crashes, hangs, disk writes)
    func didReceive(_ payloads: [MXDiagnosticPayload]) {
        for payload in payloads {
            let json = payload.jsonRepresentation()
            AnalyticsService.upload(diagnosticPayload: json)
        }
    }
}

What MetricKit Delivers

Payload Type Key Metrics What AI Looks For
MXMetricPayload CPU time, GPU time, memory peak, disk writes, network bytes, launch time, hang rate Regressions between versions. Sudden spikes in CPU or memory.
MXDiagnosticPayload Crash logs, hang call stacks, disk write exceptions, CPU exceptions New crash patterns, hang hotspots, excessive disk writes.
MXAppLaunchMetric Cold, warm, and hot launch times with histograms Launch time regressions. P95 vs P50 spread.
MXHangDiagnostic Hang duration, call stack at hang point Main-thread blocking operations in production.

Detecting Regressions with AI

v2.3.1 (Baseline)

  • Launch P50: 1.1s / P95: 2.3s
  • Hang rate: 0.3%
  • Memory peak: 180 MB
  • CPU time: 42s avg/session

v2.4.0 (Regression)

  • Launch P50: 1.8s / P95: 3.9s
  • Hang rate: 1.4%
  • Memory peak: 245 MB
  • CPU time: 71s avg/session

AI correlates the changes in v2.4.0 with the metric shifts: the image caching layer likely caused the 65 MB memory increase (caching full-resolution images), the Alamofire migration may explain the CPU increase (different threading model), and the SwiftUI onboarding flow probably caused the launch regression (eager view initialization).

Xcode Organizer Insights

Xcode Organizer aggregates production data from App Store users (who opted into sharing). It’s less detailed than MetricKit but gives you trends across versions without any code changes.

The Five Tabs That Matter

Crashes Tab

  • Top crash clusters with frequency
  • Symbolicated stacktraces
  • iOS version and device distribution
  • Trend lines across app versions

Hangs Tab

  • Main-thread hang call stacks
  • Duration distribution
  • Affected iOS versions
  • Useful when MetricKit isn’t set up

Energy & Launch Tabs

  • Energy impact trends per version
  • Launch time histograms (cold/warm)
  • Compare your app vs. category average

Disk Writes Tab

  • Excessive write exceptions
  • Call stacks of write-heavy operations
  • Often reveals logging or caching bugs
Organizer + AI workflow: Open Xcode Organizer, select the Crashes tab, click a crash cluster, and click “Open in Project” to jump to the code. If the fix isn’t obvious, copy the full stacktrace and paste it into Claude Code with your source file for a root cause analysis.

TestFlight and On-Device Crash Logs

TestFlight Beta Feedback

TestFlight crash reports are available in App Store Connect within minutes. The killer feature: beta testers can submit feedback alongside crash reports, telling you what they were doing when it crashed. Feed the crash report + user description to AI for much richer context.

On-Device .ips Files

When you can’t reproduce a crash and need the raw data, users can export crash logs: Settings > Privacy & Security > Analytics & Improvements > Analytics Data. Look for files matching your app name with .ips extension. These are the same format as Xcode Organizer crash reports.

Symbolication reminder: On-device .ips files are not symbolicated. You need the matching dSYM (same build UUID) to make them readable. Always archive your builds and keep dSYMs. Without them, you just see memory addresses.

Crash Trend Analysis with AI

Individual crashes tell you what broke. Trends tell you when and why. Feed AI crash rate data across versions and it spots regressions instantly.

Good Prompt
Here are crash-free rates from Crashlytics for our last 5 releases: v2.1.0: 99.2% crash-free (baseline) v2.2.0: 99.1% crash-free (minor drop) v2.3.0: 98.4% crash-free (significant drop) v2.3.1: 98.8% crash-free (partial recovery) v2.4.0: 97.1% crash-free (worst ever) v2.3.0 introduced Core Data migration. v2.3.1 was a hotfix for migration crash on empty databases. v2.4.0 added new checkout flow + switched networking library. What pattern do you see? Where should we focus?

The Version Comparison Pattern

When feeding crash data to AI, always include what changed in each version. AI can’t diagnose a regression from numbers alone — it needs the context of which code shipped. Changelog + crash rate + top crash clusters = AI can pinpoint which change caused the regression.

Xcode 26 AI vs. Claude Code for Production Issues

Use Xcode 26 AI When...

  • You’re in Organizer and want a quick crash explanation
  • You need to jump from crash to code fix inline
  • The crash is in your code and the fix is straightforward
  • You want AI to explain a MetricKit hang call stack

Use Claude Code When...

  • You need to triage 20+ crash clusters and prioritize
  • You want to correlate crashes with MetricKit metrics
  • The crash involves complex state across multiple files
  • You need to generate a fix plan with code for the team
  • You want automated regression detection via XcodeBuildMCP
