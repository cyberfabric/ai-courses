Chapter 2

Crash Reports, Logs & Symbolication

Master iOS crash report analysis, dSYM symbolication, and structured logging with AI assistance

The Crash Report Is Your Crime Scene

In Chapter 1, you learned to feed crash data to AI and get hypotheses back. But here's the thing — if the data you feed is garbage, the hypotheses will be garbage too.

An unsymbolicated crash report is like a crime scene where all the evidence labels are in hex. 0x104a23def means nothing to you or to AI. But ProfileViewController.viewDidAppear(_:) + 148? Now we're talking.

This chapter teaches you to produce the cleanest, most informative crash data possible — because that's what makes AI analysis actually useful.

Anatomy of an iOS Crash Report (.ips)

Every iOS crash generates a .ips file (JSON format since iOS 15). These files live on the device at Settings → Privacy & Security → Analytics & Improvements → Analytics Data, or you can pull them from Xcode Organizer.

Here are the sections that matter, in the order AI needs them:

Crash Report Structure — .ips File (Annotated)

{
  "app_name": "MyApp",
  "app_version": "2.4.1",
  "os_version": "iPhone OS 18.3 (22D60)",
  "incident_id": "A1B2C3D4-E5F6-...",

  // === SECTION 1: What happened ===
  "exception": {
    "type": "EXC_CRASH",
    "signal": "SIGABRT",
    "subtype": "KERN_INVALID_ADDRESS at 0x0000000000000000"
  },
  "termination": {
    "namespace": "SIGNAL",
    "code": 6,
    "reason": "Abort trap: 6"
  },

  // === SECTION 2: Where it happened ===
  "faultingThread": 0,
  "threads": [
    {
      "id": 0,
      "name": "main",
      "frames": [
        { "imageIndex": 0, "symbol": "specialized UserManager.fetchProfile()", "imageOffset": 2340 },
        { "imageIndex": 0, "symbol": "ProfileViewController.viewDidLoad()", "imageOffset": 892 },
        { "imageIndex": 1, "symbol": "-[UIViewController loadViewIfNeeded]", "imageOffset": 120 },
        { "imageIndex": 1, "symbol": "-[UINavigationController _startTransition:...]", "imageOffset": 456 }
      ]
    }
  ],

  // === SECTION 3: What was loaded ===
  "usedImages": [
    { "name": "MyApp", "uuid": "ABC123...", "base": 4398046511104 },
    { "name": "UIKitCore", "uuid": "DEF456...", "base": 7140073472 }
  ]
}

AI Analysis

Exception: EXC_CRASH (SIGABRT) — the app called abort() deliberately. In Swift, this typically means a forced unwrap of nil, fatalError(), or a precondition failure.

Faulting thread: Thread 0 (main). The crash is in UserManager.fetchProfile() called from viewDidLoad().

Key insight: SIGABRT + the call chain from viewDidLoad into a data fetch method strongly suggests a force-unwrapped optional that was nil at load time — likely a user property that wasn't set before the profile screen was pushed.

What AI Needs from a Crash Report

When pasting crash reports into Claude Code or Xcode AI, always include:

  • Exception type + signal — this tells AI the crash category
  • Faulting thread's full backtrace — not just the top frame, the whole chain
  • Your app's frames specifically — highlight which frames are yours vs. system frameworks
  • OS version + device — some crashes are iOS-version-specific

The binary images section is only needed if the report is unsymbolicated.

dSYM Symbolication: Making Crash Reports Readable

An unsymbolicated crash report shows memory addresses instead of function names. It's like reading a recipe that says "combine ingredient at address 0x7F with ingredient at address 0xA3." Technically correct, completely useless.

dSYM files (debug symbol files) are the Rosetta Stone. Xcode generates them during every build. They map memory addresses back to file names, function names, and line numbers.

Finding Your dSYMs

Where dSYMs Live

  • Xcode Archives: Window → Organizer → right-click your archive → Show in Finder → dSYMs/ folder
  • Derived Data: ~/Library/Developer/Xcode/DerivedData/MyApp-xxx/Build/Products/Debug-iphonesimulator/MyApp.app.dSYM
  • App Store Connect: If you uploaded with bitcode, download dSYMs from App Store Connect → TestFlight or the app's Activity tab
  • Xcode automatic download: Xcode → Settings → Accounts → Download Debug Symbols (happens automatically when viewing Organizer crashes)

Manual Symbolication with atos

The atos command-line tool converts addresses to symbols. Here's the real syntax:

# Syntax:
atos -arch arm64 -o MyApp.app.dSYM/Contents/Resources/DWARF/MyApp \
     -l 0x104000000 0x104a23def

# Example output:
ProfileViewController.viewDidAppear(_:) (in MyApp) (ProfileViewController.swift:47)

The -l flag is the load address (from the binary images section of the crash report), and the final argument is the address to symbolicate.

UUID Mismatch = Wrong Symbols

Every dSYM has a UUID that must match the binary that crashed. If you symbolicate with the wrong dSYM (different build), you'll get wrong function names — which is worse than no names at all, because you'll debug the wrong code.

Check UUIDs with: dwarfdump --uuid MyApp.app.dSYM

Compare against the crash report's usedImages → uuid field.

The symbolicatecrash Script

For bulk symbolication, Apple ships a hidden gem:

# Find the script:
find /Applications/Xcode.app -name symbolicatecrash -type f

# Run it (set DEVELOPER_DIR first):
export DEVELOPER_DIR="/Applications/Xcode.app/Contents/Developer"
/Applications/Xcode.app/Contents/SharedFrameworks/DVTFoundation.framework/\
Versions/A/Resources/symbolicatecrash crash.ips MyApp.app.dSYM > symbolicated.crash

This symbolicates the entire report in one shot. Feed the symbolicated output to AI — the quality of analysis jumps dramatically.

Structured Logging with os_log

Crash reports tell you what happened. Logs tell you why it happened — the sequence of events leading up to the crash. But print() statements are invisible outside the debugger. Enter os_log.

Apple's Logger API (iOS 14+) gives you structured, categorized, filterable logs that survive app termination and can be queried programmatically:

import OSLog

// Create loggers per subsystem
private let logger = Logger(subsystem: "com.myapp", category: "networking")
private let authLogger = Logger(subsystem: "com.myapp", category: "auth")

// Log with levels
logger.debug("Starting request to \(url, privacy: .public)")
logger.info("Response received: \(statusCode) in \(duration, format: .fixed(precision: 2))ms")
logger.error("Request failed: \(error.localizedDescription, privacy: .public)")
authLogger.critical("Token refresh failed — user will be logged out")

Privacy Matters in Logs

By default, os_log redacts dynamic values in production for privacy. Only mark values as .public when they contain no user data. Use .private(mask: .hash) when you need to correlate entries without exposing values:

logger.info("Fetching profile for user \(userId, privacy: .private(mask: .hash))")
// Output: "Fetching profile for user <hash: 8A3F2B>"

This lets you track which user requests are failing without logging their actual ID.

Querying Logs with OSLogStore

The killer feature for AI debugging: you can programmatically fetch logs and feed them directly to AI for analysis.

import OSLog

func fetchRecentLogs() throws -> [OSLogEntryLog] {
    let store = try OSLogStore(scope: .currentProcessIdentifier)
    let position = store.position(date: Date().addingTimeInterval(-300)) // Last 5 min

    let entries = try store.getEntries(at: position)
        .compactMap { $0 as? OSLogEntryLog }
        .filter { $0.subsystem == "com.myapp" }

    return entries
}

// Format for AI consumption:
for entry in try fetchRecentLogs() {
    print("[\(entry.date)] [\(entry.category)] [\(entry.level.rawValue)] \(entry.composedMessage)")
}

Build this into a debug menu or a diagnostic export feature, and you can paste the structured output straight into Claude Code.

Exception Types: The AI Cheat Sheet

The exception type is the single most important field in a crash report. It tells AI (and you) the category of crash before reading a single frame. Here's the reference:

Exception Signal What It Means Common Swift Causes
EXC_BAD_ACCESS SIGSEGV Invalid memory read/write Use-after-free, dangling unowned, bad pointer from C interop
EXC_BAD_ACCESS SIGBUS Misaligned memory access Corrupted data, bad UnsafePointer arithmetic
EXC_CRASH SIGABRT Deliberate abort Force-unwrap nil (!), fatalError(), preconditionFailure(), failed as! cast
EXC_BREAKPOINT SIGTRAP Trace/breakpoint trap Swift runtime trap: array out-of-bounds, implicit unwrap nil, concurrency precondition
EXC_BAD_INSTRUCTION SIGILL Illegal instruction Rare: compiler bug, corrupted binary, Swift runtime internal error
00000020 SIGKILL Killed by system Watchdog timeout (0x8BADF00D), memory pressure (jetsam), background task exceeded time

When you paste a crash report into AI, the exception type is the first thing it reads. SIGABRT immediately suggests force-unwrap or assertion failure. SIGSEGV with a near-zero address suggests a nil/dangling pointer. This classification drives the entire analysis.

Building an /analyze-crash Skill

Time to build this chapter's reusable skill. The /analyze-crash command takes a raw .ips crash report and returns structured analysis.

The /analyze-crash Skill

Create .claude/commands/analyze-crash.md:

Analyze this iOS crash report (.ips format or symbolicated crash log).

## Analysis Steps:
1. Read the exception type and signal — classify the crash category
2. Identify the faulting thread and read its full backtrace
3. Separate OUR frames (app binary) from system framework frames
4. If unsymbolicated, note that symbolication is needed and identify
   the binary UUIDs and load addresses for atos
5. Look for known crash patterns:
   - SIGABRT → force-unwrap, fatalError, precondition
   - SIGSEGV near 0x0 → nil/dangling pointer
   - SIGTRAP → Swift runtime trap (array bounds, concurrency)
   - 0x8BADF00D → watchdog timeout (hang)
6. Read the source file at the faulting frame for context
7. Check for similar patterns elsewhere in the codebase

## Output Format:
- **Crash type:** [exception type + signal + human explanation]
- **Faulting thread:** [thread number + name]
- **Faulting frame:** [our code, file:line if available]
- **Root cause:** [1-2 sentence explanation]
- **Affected component:** [module/feature area]
- **Confidence:** [high/medium/low based on available information]
- **Suggested fix:** [specific code change]
- **Prevention:** [how to prevent this class of crash]

## Input:
$ARGUMENTS

This skill is more specialized than the /debug skill from Chapter 1. It's tailored specifically for .ips crash reports and uses the exception type classification table above.

Common Crash Patterns AI Recognizes Instantly

After training on millions of crash reports, AI is remarkably good at recognizing these iOS patterns. Here's what it looks for:

Top 5 Swift Crash Patterns

1. Force-unwrap nil (!): SIGABRT with unexpectedly found nil while unwrapping an Optional value in the crash reason. The fix is always: replace ! with guard let or if let.

2. Array out-of-bounds: SIGTRAP with Index out of range. AI checks the array source (API response, Core Data fetch) and the index source (hardcoded, user-driven, off-by-one).

3. Unrecognized selector: SIGABRT with -[ClassName selectorName]: unrecognized selector sent to instance. Usually a stale Storyboard/XIB outlet or an @objc method that was renamed.

4. Swift concurrency precondition failure: SIGTRAP with Precondition failed: sending value of non-Sendable type or Data race detected. Swift 6 strict concurrency catches these at runtime.

5. KeyPath crash: SIGABRT during KVO observation with a keypath that no longer exists. Common after refactoring a model property that was observed via #keyPath or string-based KVO.

Multi-Device Crash Comparison

The same code can crash differently on different devices. An iPhone 15 with 6GB RAM might never OOM-jetsam, but an iPhone SE with 3GB will. iOS 17 might not enforce a concurrency check that iOS 18 does.

Good Prompt for Multi-Device Analysis

"Here are 3 crash reports for the same crash — one from iPhone SE (iOS 17.4), one from iPhone 15 Pro (iOS 18.2), and one from iPad Air (iOS 18.0). Compare the stacktraces and identify whether this is the same root cause or different bugs triggered by the same user flow."

Claude Code excels here because it can hold all three reports in context simultaneously, compare frame-by-frame, and identify the common thread. Xcode AI is limited to the single report open in the editor.

Third-Party Framework Crashes

Sometimes the stacktrace points entirely into UIKit, SwiftUI, or Core Data — none of your code appears in the faulting thread. This doesn't mean it's Apple's bug (usually).

Reading System Framework Crashes

When the crash is in system code, look for your code on other threads or your code earlier in the faulting thread's history. Common patterns:

  • UIKit crash: Your code modified UI off the main thread. The crash shows in UIKit, but the cause is your background thread updating a UILabel.
  • Core Data crash: Your code accessed a managed object across contexts. The crash is in NSManagedObjectContext, but the cause is missing perform {} blocks.
  • SwiftUI crash: Your @Observable model mutated state during a view update. The crash is in AttributeGraph, but the cause is a computed property with side effects.

Tell Claude Code: "The crash is in UIKit but look at Thread 3 and Thread 5 for my app's code that might have triggered it."

Xcode 26 AI + Crash Reports

Xcode 26's AI integration brings crash report analysis directly into the IDE. When you open a crash report in the Organizer:

  • Inline explanations: Swift Assist annotates the crashing frame with a plain-English explanation
  • Jump to code: Click any symbolicated frame to navigate to the source, where Xcode AI can explain the context
  • Fix suggestions: For common patterns (force unwrap, bounds check), Xcode offers one-click fix suggestions

For straightforward crashes, this is faster than copying to Claude Code. For complex multi-file bugs, use Claude Code with the /analyze-crash skill.

Using XcodeBuildMCP for Automated Crash Reproduction

With XcodeBuildMCP, you can ask Claude Code to reproduce a crash automatically:

Good Prompt

"Based on this crash report showing a nil force-unwrap in ProfileViewController.viewDidLoad(), write a unit test that reproduces this crash by constructing ProfileViewController without setting the userId property. Build and run it via XcodeBuildMCP to confirm it crashes with the same exception."

The AI writes the test, builds via MCP, runs it, and confirms whether the reproduction matches the original crash. That's a crash-to-test pipeline with zero manual Xcode interaction.

With AI-Assisted Analysis

Hands-On Exercise

Exercise: Analyze a Real Crash Report with AI

Let's walk through a complete crash analysis using both Xcode AI and Claude Code.

  1. Create a crashable scenario. Add this to a view controller: 
    class UserProfileVC: UIViewController {
    var userName: String!  // Implicitly unwrapped optional

    override func viewDidLoad() {
        super.viewDidLoad()
        // userName is nil — never set before presenting!
        let greeting = "Hello, \(userName!)"
        title = greeting
    }
}

    Present this VC without setting userName.

  2. Capture the crash. Run in the simulator. When it crashes, go to the Debug Navigator and copy the full thread backtrace.
  3. Try Xcode AI. Click the crashing line — see Swift Assist's inline explanation. Does it identify the nil implicitly-unwrapped optional?
  4. Use /analyze-crash. If you built the skill from this chapter, run: 
    claude
> /analyze-crash [paste the full crash report or backtrace]

    Compare Claude Code's structured output (crash type, root cause, fix, prevention) with Xcode AI's inline suggestion.

  5. Practice symbolication. If you have access to a real production crash (from Xcode Organizer or Crashlytics), try manually symbolicating one frame with atos: 
    atos -arch arm64 \
  -o /path/to/MyApp.app.dSYM/Contents/Resources/DWARF/MyApp \
  -l 0x104000000 \
  0x10400ABCD
  6. Set up structured logging. Replace one print() in your project with a Logger call using subsystem and category. Open Console.app and filter for your subsystem.

Bonus: Export 5 minutes of os_log output using OSLogStore, paste into Claude Code, and ask it to identify any warning or error patterns.

Key Takeaways

  • Symbolicate first, analyze second. Unsymbolicated crash reports waste everyone's time — including AI's.
  • The exception type + signal is the most important field. It classifies the crash before you read a single frame.
  • os_log with Logger is the production-grade replacement for print(). It survives app termination, supports privacy controls, and is queryable via OSLogStore.
  • Use Xcode AI for quick inline crash explanations. Use Claude Code's /analyze-crash for deep analysis, multi-device comparison, and third-party framework crashes.
  • Build a crash-to-test pipeline with XcodeBuildMCP: AI reads the crash, writes a reproduction test, builds and runs it automatically.
