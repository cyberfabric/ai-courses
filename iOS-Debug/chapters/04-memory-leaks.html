Memory Leak Detection with AI

Combine Memory Graph Debugger and Instruments with AI to find and fix retain cycles

The Slow Kill: Why Memory Leaks Matter on iOS

A crash is a bullet. A memory leak is poison. Your app doesn't die immediately — it gets slower, heavier, and hungrier until iOS kills it with a jetsam termination. The user never sees a crash dialog. The app just vanishes from the background, and next time they open it, all their state is gone.

OOM
Out-of-memory jetsam — iOS silently kills your app when memory pressure is high
EXC_RESOURCE
Exception when your app exceeds its memory limit — appears in crash reports
~50MB
Memory footprint where iOS starts watching your app closely for pressure
#1
Retain cycles are the most common cause of memory leaks in Swift apps

Unlike languages with garbage collection, Swift uses Automatic Reference Counting (ARC). ARC is deterministic and efficient, but it can't break retain cycles. Two objects pointing to each other with strong references will live forever — leaking memory until jetsam kicks in.

The Navigation Test for Leaks

The simplest leak test: push a view controller, pop it, then capture the memory graph. If the view controller (or its view model) still appears in the heap, you have a leak. The object should be deallocated when you pop — if it isn't, something is retaining it.

For SwiftUI: navigate to a detail view, go back, and check if the detail view's @Observable model is still alive.

Feeding Memory Graph Data to AI

The Memory Graph Debugger shows you visual arrows, but the real power comes from extracting the retention chain as text and feeding it to AI. Here's how:

In the Memory Graph Debugger, select a leaked object. The right panel shows the backtrace (where it was allocated) and the references (who retains it). Copy this text output:

Memory Graph Debugger — Retain Cycle Detected

Leaked Object: ProfileViewModel (0x600003a04b80)
  Allocated at: ProfileViewModel.init() — ProfileViewModel.swift:12
  Size: 256 bytes

Retention Chain:
  ROOT → ProfileViewController (strong)
    → ProfileViewController.viewModel (strong)
      → ProfileViewModel (strong)
        → ProfileViewModel.onUpdate (strong)  // closure
          → ProfileViewController (strong)    // captured self!

Cycle: ProfileViewController ↔ ProfileViewModel.onUpdate closure

Additional leaked objects held by ProfileViewModel:
  - UserProfileData (128 bytes)
  - [UIImage] (3 items, 4.2 MB total)

AI Analysis

Leak type: Classic closure-based retain cycle. ProfileViewModel holds a closure (onUpdate) that captures ProfileViewController strongly, while the VC holds the VM strongly.

Total leaked memory: ~4.6 MB (mostly the UIImage array held by the leaked VM). Each time the user visits the profile screen, another 4.6 MB leaks permanently.

Fix: Add [weak self] to the onUpdate closure's capture list in ProfileViewController, where the closure is assigned.

Secondary fix: Consider whether onUpdate should be a delegate pattern (with weak var delegate) instead of a closure, which would prevent this class of leak entirely.

The retention chain is structured data — AI reads it like a linked list and immediately identifies where the cycle breaks.

Instruments: Leaks and Allocations

The Memory Graph Debugger gives you a snapshot. Instruments gives you a timeline. Use both:

Tool What It Shows When to Use
Memory Graph Debugger Object graph with reference arrows Pinpointing which objects form a cycle
Instruments — Leaks Automatic leak detection over time Finding leaks during a usage session
Instruments — Allocations Heap growth, persistent vs transient objects Detecting memory that grows and never shrinks

In Instruments Allocations, use Generation Analysis (Mark Generation button): mark before a user action, mark after, and see what persisted. If objects survive that should have been freed, you've found a leak.

The Classic iOS Memory Leaks

AI recognizes these patterns instantly because they appear in virtually every iOS codebase:

The 6 Classic Retain Cycle Patterns

1. Closure capturing self: The most common. A closure stored as a property captures self strongly, creating a cycle. Fix: [weak self] or [unowned self] in the capture list.

2. Strong delegate: var delegate: ProfileDelegate instead of weak var delegate: ProfileDelegate?. The delegator and delegate retain each other. Fix: always declare delegates as weak.

3. Timer retain cycle: Timer.scheduledTimer(withTimeInterval:repeats:block:) retains the closure, and if the closure captures self, the timer keeps the object alive. The timer must be invalidated to break the cycle.

4. NotificationCenter observer leaks: Pre-iOS 9, observers weren't removed automatically. Even post-iOS 9, block-based observers with addObserver(forName:object:queue:using:) return a token that retains the closure. Store and remove it in deinit.

5. CADisplayLink strong target: CADisplayLink(target:selector:) strongly retains the target. If the target owns the display link, you have a cycle. Fix: use a weak proxy object or preferredFrameRateRange with a closure-based approach.

6. Combine sink + store(in:): publisher.sink { self.update($0) }.store(in: &cancellables) — the sink closure captures self strongly, and cancellables is owned by self. Fix: [weak self] in the sink closure.

weak vs unowned: When Each Is Correct

Aspect weak unowned
Reference becomes nil when object deallocates Dangling pointer (crash on access!)
Type Must be optional (T?) Non-optional (T)
Use when The referenced object may outlive or die first You're certain the object lives at least as long
Safety Safe — nil access is handled by optionals Unsafe — accessing after dealloc = EXC_BAD_ACCESS
Performance Tiny overhead (zeroing weak reference) No overhead

Default to weak

Unless you have a proven performance bottleneck from weak references (you won't), always use weak. An unexpected nil is debuggable. An unowned crash on a deallocated object is the same EXC_BAD_ACCESS you saw in Chapter 2 — and it happens at a completely unrelated time, making it extremely hard to trace.

Ask Claude Code: "Audit my project for unowned references and check whether each one is actually safe."

SwiftUI Memory Considerations

SwiftUI introduced new memory management patterns — and new leak vectors:

@StateObject vs @ObservedObject

@StateObject owns the object's lifecycle — it creates the object once and keeps it alive for the view's lifetime. @ObservedObject doesn't own it — if nothing else retains it, it can be deallocated and recreated on every view re-evaluation.

// CORRECT: Parent creates and owns the VM
struct ProfileView: View {
    @StateObject private var viewModel = ProfileViewModel()
    var body: some View { ProfileContent(viewModel: viewModel) }
}

// CORRECT: Child receives but doesn't own
struct ProfileContent: View {
    @ObservedObject var viewModel: ProfileViewModel
    var body: some View { Text(viewModel.name) }
}

// BUG: Creating @ObservedObject inline — recreated every re-evaluation!
struct BuggyView: View {
    @ObservedObject var viewModel = ProfileViewModel() // Leaked or recreated!
    var body: some View { Text(viewModel.name) }
}

@Observable and @State

With Swift's Observation framework, you store @Observable classes in @State:

@Observable class ProfileModel {
    var name = ""
    var avatar: UIImage?
}

struct ProfileView: View {
    @State private var model = ProfileModel()
    var body: some View { Text(model.name) }
}

Known Issue: @State + @Observable Leaks

There have been reports of @State with @Observable reference types leaking in certain iOS versions (particularly iOS 17 early releases). The objects are created but never released when the view is removed from the hierarchy. This was largely resolved in iOS 18, but always verify with Memory Graph Debugger when using this pattern. If your deployment target includes iOS 17, consider sticking with @StateObject + ObservableObject.

Combine Memory Leaks

Combine publishers and subscribers are a common leak source. The pattern is always the same: a sink closure captures self, and the AnyCancellable is stored on self:

class SettingsViewModel: ObservableObject {
    @Published var settings: [Setting] = []
    private var cancellables = Set<AnyCancellable>()

    init() {
        // BUG: sink captures self strongly, cancellables owned by self
        settingsPublisher
            .sink { self.settings = $0 }  // strong capture!
            .store(in: &cancellables)
    }
}

// FIX: Use [weak self]
settingsPublisher
    .sink { [weak self] newSettings in
        self?.settings = newSettings
    }
    .store(in: &cancellables)

Xcode 26 AI for Combine Leaks

Swift Assist in Xcode 26 can analyze Combine chains and flag missing [weak self] in sink closures. When you hover over a .sink call, Xcode shows whether self is captured and whether the cancellable's lifetime creates a cycle. This inline analysis is faster than running the Memory Graph Debugger for known patterns.

Hands-On Exercise

Exercise: Find and Fix a Retain Cycle

Create a deliberate retain cycle, visualize it, and fix it with AI assistance.

  1. Create the leak. Add this to your project: 
    class LeakyViewController: UIViewController {
    var onDismiss: (() -> Void)?

    override func viewDidLoad() {
        super.viewDidLoad()
        // Retain cycle: closure captures self, self owns closure
        onDismiss = {
            print("Dismissed \(self.title ?? "unknown")")
        }
    }

    deinit {
        print("LeakyViewController deallocated") // This never prints!
    }
}
  2. Present and dismiss. Push this VC onto a navigation stack, then pop it. Check the console — deinit should print but doesn't.
  3. Capture the memory graph. After popping, click the Memory Graph Debugger button. Find LeakyViewController — it'll have a purple ! marker. Copy the retention chain text.
  4. Analyze with Claude Code: 
    claude
> /analyze-leak
Leaked Object: LeakyViewController
Retention: self → onDismiss (closure) → captures self (strong)
  5. Apply the fix. Claude Code should suggest adding [weak self]: 
    onDismiss = { [weak self] in
    print("Dismissed \(self?.title ?? "unknown")")
}
  6. Verify. Run again, push and pop. Confirm deinit prints. Capture memory graph again — no more purple markers.

Bonus: Try the same exercise with a Combine sink closure. Create a publisher subscription that captures self strongly in a ViewModel, verify the leak with Memory Graph Debugger, and fix it with [weak self].

Key Takeaways

  • Memory leaks are silent killers. They don't crash your app — they slowly consume memory until iOS jetsams you out.
  • Use Memory Graph Debugger for pinpointing cycles (who retains whom) and Instruments Allocations for tracking growth over time.
  • The retention chain from Memory Graph Debugger is structured data AI reads perfectly — paste it into /analyze-leak.
  • Default to weak over unowned. The performance difference is negligible; the crash risk of unowned is not.
  • Watch for SwiftUI's @Observable + @State on iOS 17 — verify with Memory Graph Debugger.
  • Use /scan-for-leaks as a static analysis step in code reviews — catch leaks before they ship.
