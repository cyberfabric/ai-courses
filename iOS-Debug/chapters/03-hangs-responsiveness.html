Hangs & Responsiveness Diagnosis

Use AI to diagnose main thread hangs, watchdog terminations, and the bugs users hate most

Hangs Are Worse Than Crashes

A crash is dramatic — the app dies, you get a report, you fix it. A hang is insidious. Your app is technically alive, but the UI is frozen. The user taps the screen. Nothing happens. They tap harder (as if that helps). Still nothing. After a few seconds, they force-quit your app and leave a 1-star review.

53%
Users who abandon an app after a freeze lasting more than 3 seconds
250ms
The threshold where users perceive a UI as "laggy" (MetricKit reports hangs above this)
0x8BADF00D
The watchdog termination code — "ate bad food" — iOS kills your unresponsive app
20s
Watchdog timeout for app launch. After 20 seconds, iOS terminates your app.

Crashes show up in crash reports. Hangs often don't — they're invisible unless you're explicitly looking for them with MetricKit, Instruments, or Xcode Organizer. That's what makes them so dangerous.

The Watchdog: 0x8BADF00D

iOS runs a watchdog timer that monitors your app's main thread responsiveness. If the main thread is blocked for too long, iOS terminates your app with the exception code 0x8BADF00D (read it out loud: "ate bad food").

The timeout thresholds vary by app state:

App State Timeout What Triggers It
Launch ~20 seconds App doesn't finish didFinishLaunching in time
Foreground ~10 seconds Main thread blocked while app is visible
Background task ~5-10 seconds Background task handler doesn't call completion
Suspended → resume ~10 seconds applicationWillEnterForeground blocks too long

Watchdog Terminations Don't Look Like Crashes

A watchdog kill produces a different crash report format. The exception code is 0x8badf00d and the termination reason is scene-create, scene-update, or similar. The key clue: every thread's backtrace is captured at the moment of termination — you see exactly what the main thread was doing when it got killed.

This is gold for AI analysis. The entire system state is frozen in time.

MetricKit: MXHangDiagnostic

MetricKit is Apple's framework for receiving performance diagnostics directly from production devices. For hangs, MXHangDiagnostic delivers fully symbolicated call stacks showing exactly what was blocking the main thread — and it catches hangs as short as 250ms that never trigger the watchdog.

import MetricKit

class MetricSubscriber: NSObject, MXMetricManagerSubscriber {

    func didReceive(_ payloads: [MXDiagnosticPayload]) {
        for payload in payloads {
            if let hangDiagnostics = payload.hangDiagnostics {
                for hang in hangDiagnostics {
                    // hang.callStackTree contains the blocking call chain
                    let callStack = hang.callStackTree.jsonRepresentation()
                    logger.warning("Hang detected: \(String(data: callStack, encoding: .utf8) ?? "")")
                }
            }
        }
    }
}

// Register at app launch:
MXMetricManager.shared.add(subscriber)

MetricKit delivers diagnostics once per day (aggregated over 24 hours). The call stacks are already symbolicated — no dSYM needed. This is structured data that AI analyzes beautifully.

MetricKit + Claude Code Pipeline

Export your MXHangDiagnostic payloads to a file or logging service. Then paste them into Claude Code with your /analyze-hang skill (we'll build it below). 1  CoreData            NSSQLiteStatement.execute() + 156

4  MyApp               AppDelegate.setupCoreData() + 456
5  MyApp               AppDelegate.application(_:didFinishLaunchingWithOptions:) + 128
6  UIKitCore           -[UIApplication _handleDelegateCallbacksWithOptions:...] + 344

AI Analysis

Hang type: Watchdog termination during app launch (scene-create). The app exceeded the ~20-second launch time allowance.

Blocking operation: A Core Data migration (migrateV12ToV13) running synchronously on the main thread during didFinishLaunchingWithOptions. The migration involves SQLite operations that took longer than 20 seconds on this device.

Root cause: Heavy database migration executed on the main thread at launch time. On older devices with large databases, this migration exceeds the watchdog threshold.

Fix: Move the migration to a background queue. Show a "Migrating data..." screen on the main thread while the migration runs asynchronously. Only present the main UI once the migration completes.

Notice how the syslog entries (from os_log) give AI the narrative: "migration started, then the watchdog killed us." Without those logs, AI only sees the SQLite call — the logs tell the story.

Common Hang Patterns

AI has seen these patterns thousands of times. When you paste a hang report, it immediately checks for these culprits:

The 6 Most Common iOS Hang Patterns

1. Synchronous network on main thread: URLSession.shared.dataTask with a completion handler is async, but Data(contentsOf: url) or a semaphore-based wrapper blocks the main thread completely.

2. Heavy Core Data fetch on main thread: A NSFetchRequest with no predicate on a table with 100K+ rows. Or a migration running synchronously in didFinishLaunching.

3. Synchronous Keychain access: SecItemCopyMatching can take seconds when the device is locked or the Keychain is large. Always call from a background queue.

4. Large image decoding on main thread: Loading a 4000x3000 JPEG with UIImage(named:) and immediately displaying it. The decode happens lazily on first render — on the main thread.

5. Excessive Auto Layout recalculation: A complex view hierarchy with hundreds of constraints being invalidated repeatedly during a single layout pass. Common with dynamically-sized cells in a table view.

6. Lock contention: The main thread tries to acquire a lock (via DispatchSemaphore, NSLock, or os_unfair_lock) that a background thread holds for a long operation.

Swift Concurrency and Hangs

Swift concurrency (async/await) was supposed to eliminate main-thread blocking. And it does — when used correctly. But there are subtle ways to still hang your app:

Pitfall 1: Blocking Calls Inside @MainActor Context

@MainActor
class ProfileViewModel: ObservableObject {
    @Published var profile: UserProfile?

    func loadProfile() async {
        // BUG: This blocks the main thread!
        // Thread.sleep is synchronous — it doesn't suspend
        Thread.sleep(forTimeInterval: 2.0)

        // BUG: Synchronous file I/O on MainActor
        let data = try! Data(contentsOf: largeFileURL)

        profile = try? JSONDecoder().decode(UserProfile.self, from: data)
    }
}

The method is async, but Thread.sleep and Data(contentsOf:) are synchronous. They block the MainActor's thread, freezing the UI. The async keyword only means the function can suspend — it doesn't mean every line inside it is non-blocking.

Good Prompt for Concurrency Hang Analysis

"This ViewModel is @MainActor and all its methods are async, but users report the profile screen freezes for 2-3 seconds. Analyze the code for synchronous blocking calls that would hang the main thread despite the async context."

Pitfall 2: Task Priority Inversion

// High-priority UI task waits for low-priority background work
@MainActor
func refreshUI() async {
    // This Task inherits .userInitiated from the MainActor caller
    let data = await fetchData() // But fetchData runs at .background priority

    // The main thread is waiting for a low-priority task
    // that keeps getting preempted by higher-priority work
    updateViews(with: data)
}

Claude Code can detect these priority inversions by reading the task hierarchy and actor isolation attributes across your codebase — something that's nearly impossible to spot by reading a single file.

Pitfall 3: nonisolated Misuse

@MainActor
class SettingsManager {
    var cache: [String: Any] = [:]

    // This runs OFF the main thread — but accesses
    // MainActor-isolated 'cache' property!
    nonisolated func preloadSettings() {
        // Swift 6 catches this at compile time
        // Swift 5.x: silent data race + potential hang
        let value = heavyComputation()
        cache["result"] = value // Race condition!
    }
}

Xcode 26 AI for Concurrency Issues

Swift Assist in Xcode 26 is particularly good at concurrency analysis. When you see a purple Thread Checker warning or a Sendable compiler error, Xcode AI explains the isolation boundary violation and suggests whether to use nonisolated, @Sendable, or restructure the actor isolation. For hangs specifically, ask: "Is this blocking the MainActor?"

Using os_signpost to Measure Hang-Prone Code

Signposts let you mark the beginning and end of operations so Instruments (and AI) can see exactly how long each section takes:

import OSLog

private let signposter = OSSignposter(subsystem: "com.myapp", category: "performance")

func loadUserData() {
    let signpostID = signposter.makeSignpostID()

    // Mark the start
    let state = signposter.beginInterval("loadUserData", id: signpostID)

    // ... do the work ...

    // Mark the end
    signposter.endInterval("loadUserData", state)
}

// For async code:
func loadUserDataAsync() async {
    let signpostID = signposter.makeSignpostID()
    let state = signposter.beginInterval("loadUserDataAsync", id: signpostID)

    await fetchFromNetwork()
    await processResults()

    signposter.endInterval("loadUserDataAsync", state)
}

In Instruments, signposts appear as labeled intervals on the timeline. You can instantly see "this function took 4.2 seconds" and drill into why. Export the signpost data and feed it to Claude Code for automated analysis.

Hands-On Exercise

Exercise: Diagnose and Fix a Deliberate Hang

You'll create a hang, capture it, analyze it with AI, and refactor to async.

  1. Create the hang. Add this to a button action or viewDidAppear: 
    @MainActor
func loadData() async {
    // Deliberate hang: synchronous sleep on main thread
    Thread.sleep(forTimeInterval: 6.0)

    // Simulate sync file read
    let url = Bundle.main.url(forResource: "large", withExtension: "json")!
    let data = try! Data(contentsOf: url)
    let items = try! JSONDecoder().decode([Item].self, from: data)
    print("Loaded \(items.count) items")
}
  2. Run and observe. Tap the button. The UI freezes for 6+ seconds. On a debug build, Thread Checker may fire.
  3. Capture with Time Profiler. Open Instruments → Time Profiler. Record while reproducing the hang. Select the frozen interval and copy the main thread call tree.
  4. Analyze with Claude Code: 
    claude
> /analyze-hang [paste the Time Profiler call tree or describe the hang]
  5. Refactor to async. Claude Code should suggest something like: 
    @MainActor
func loadData() async {
    // Show loading state immediately
    isLoading = true

    // Move heavy work off main thread
    let items = await Task.detached(priority: .userInitiated) {
        let url = Bundle.main.url(forResource: "large", withExtension: "json")!
        let data = try! Data(contentsOf: url)
        return try! JSONDecoder().decode([Item].self, from: data)
    }.value

    // Back on main thread — update UI
    self.items = items
    isLoading = false
}
  6. Verify. Run the Time Profiler again. The main thread should now show near-zero blocking during the load. The UI stays responsive with a loading indicator.

Bonus: Add os_signpost markers around the data loading to see the exact timing in Instruments. Compare the before/after intervals.

Key Takeaways

  • Hangs are worse than crashes for user experience. Users abandon apps that freeze — and they don't come back.
  • The watchdog (0x8BADF00D) has different timeouts per app state: ~20s for launch, ~10s for foreground.
  • MetricKit MXHangDiagnostic catches hangs as short as 250ms in production — crashes miss these entirely.
  • async doesn't mean non-blocking. A synchronous call inside an async method on @MainActor still freezes the UI.
  • Use Instruments Time Profiler to capture the hang, Claude Code /analyze-hang to diagnose it, and os_signpost to verify the fix.
  • Enable Thread Checker in your scheme — catch main-thread violations before your users do.
