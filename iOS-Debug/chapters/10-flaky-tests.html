Flaky Test Diagnosis

When “retry and ignore” becomes your team’s default response to red builds, you’ve got a trust problem.

Why Flaky Tests Erode Everything

A flaky test is a test that passes sometimes and fails sometimes with no code change. It’s the “boy who cried wolf” of your CI pipeline. After a few false alarms, your team stops trusting the test suite. Soon, real failures get retried and merged. Bugs ship.

The insidious thing about flaky tests isn’t the tests themselves — it’s the culture they create. Once “just retry it” becomes normal, your entire CI safety net has a hole.

AI changes the equation. A flaky test produces structured data across multiple runs — failure logs, assertion messages, stack traces, timing patterns. Feed those runs to AI, and it spots the pattern a human would need 20 runs and an hour to see.

The Four Categories of Flaky Tests

Not all flakiness is the same. Before you can fix a flaky test, you need to know why it flakes. AI classifies these instantly when you feed it failure logs from multiple runs.

⏱️ Timing-Dependent

  • Passes on fast machines, fails on slow CI
  • Race conditions between threads
  • Hardcoded sleep() or tight timeouts
  • Fix: proper async waiting, not sleep

📋 Order-Dependent

  • Passes alone, fails when run after another test
  • Shared mutable state (singletons, UserDefaults)
  • Database not cleaned between tests
  • Fix: isolate state, proper tearDown

🖥️ Environment-Dependent

  • Passes on one machine, fails on another
  • Simulator version differences
  • Locale, timezone, or date-dependent logic
  • Fix: explicit locale/timezone, mock dates

📡 Resource-Dependent

  • Network calls to real servers
  • File system race conditions
  • Keychain access on CI (provisioning)
  • Fix: mock external resources, stub I/O

Feeding Test Failures to AI

The key to AI-powered flaky test diagnosis is multiple runs. One failure is ambiguous. Five failures with different stack traces tell a story.

Good Prompt
This test fails roughly 1 in 5 runs on CI but passes locally. Here are the failure logs from the last 4 CI failures: Run 1: XCTAssertEqual failed: ("0") is not equal to ("3") — testFetchProducts() line 42 Run 2: XCTAssertEqual failed: ("1") is not equal to ("3") — testFetchProducts() line 42 Run 3: Passed Run 4: XCTAssertEqual failed: ("0") is not equal to ("3") — testFetchProducts() line 42 Run 5: XCTAssertEqual failed: ("2") is not equal to ("3") — testFetchProducts() line 42 Here’s the test code: ```swift func testFetchProducts() async throws { let viewModel = ProductListViewModel(service: MockProductService()) viewModel.loadProducts() // Line 42: XCTAssertEqual(viewModel.products.count, 3) } ``` And the ViewModel: ```swift func loadProducts() { Task { products = try await service.fetchProducts() } } ``` Why does the count vary between 0, 1, 2, and 3?
Bad Prompt
My test is flaky. Fix it.

Diagnosing a Race Condition

XCTest Output — 5 runs of testFetchProducts()
Test Suite 'ProductListViewModelTests' started.

--- Run 1 (FAIL) ---
testFetchProducts(): XCTAssertEqual failed: ("0") is not equal to ("3")
  ProductListViewModelTests.swift:42

--- Run 2 (FAIL) ---
testFetchProducts(): XCTAssertEqual failed: ("1") is not equal to ("3")
  ProductListViewModelTests.swift:42

--- Run 3 (PASS) ---
testFetchProducts(): passed (0.004 seconds)

--- Run 4 (FAIL) ---
testFetchProducts(): XCTAssertEqual failed: ("0") is not equal to ("3")
  ProductListViewModelTests.swift:42

--- Run 5 (PASS) ---
testFetchProducts(): passed (0.003 seconds)
AI Analysis: Classic race condition. loadProducts() fires an unstructured Task { }, but the test asserts immediately without waiting for the Task to complete. The count varies (0, 1, 2, 3) because the assertion races against the async operation — sometimes none of the products have arrived, sometimes some, sometimes all. Passes locally because your M4 Mac finishes the mock fetch before the assertion runs. CI’s shared runners are slower.

Root Cause: Unstructured Task Without Test Synchronization

Evidence: The assertion count varies randomly between 0 and 3 across runs. The ViewModel uses Task { } which is fire-and-forget — the test has no way to wait for completion.

Category: Timing-dependent flakiness.

Why it passes locally: Fast hardware completes the mock fetch before the synchronous assertion runs. CI runners are slower and more variable.

Fix Pattern: Make the Async Operation Awaitable

// BEFORE: Fire-and-forget Task — test can't wait
func loadProducts() {
    Task {
        products = try await service.fetchProducts()
    }
}

// Test races against the Task:
func testFetchProducts() async throws {
    let vm = ProductListViewModel(service: MockProductService())
    vm.loadProducts()
    XCTAssertEqual(vm.products.count, 3) // 💥 race condition
}

// AFTER Option A: Make loadProducts() async
func loadProducts() async throws {
    products = try await service.fetchProducts()
}

// Test awaits properly:
func testFetchProducts() async throws {
    let vm = ProductListViewModel(service: MockProductService())
    try await vm.loadProducts()
    XCTAssertEqual(vm.products.count, 3) // ✅ deterministic
}

// AFTER Option B: If Task {} is required (e.g., called from SwiftUI),
// use XCTestExpectation or observe the published property:
func testFetchProducts() async throws {
    let vm = ProductListViewModel(service: MockProductService())
    let expectation = XCTestExpectation(description: "Products loaded")

    let cancellable = vm.$products
        .dropFirst()
        .sink { products in
            if products.count == 3 { expectation.fulfill() }
        }

    vm.loadProducts()
    await fulfillment(of: [expectation], timeout: 2.0)
    XCTAssertEqual(vm.products.count, 3) // ✅ waits for data
    cancellable.cancel()
}

Before

  • Pass rate: ~60% (3 of 5 runs)
  • CI retries per week: 12
  • Time wasted: ~2 hours/week

After

  • Pass rate: 100% (deterministic)
  • CI retries per week: 0
  • Time saved: 2 hours/week

XCUITest Synchronization Pitfalls

UI tests are flakier than unit tests by nature. They depend on animations completing, views appearing, system alerts popping up, and simulators behaving. Here are the patterns AI catches immediately.

The sleep() Antipattern

// ❌ NEVER do this — timing varies on CI
func testLogin() {
    app.textFields["email"].tap()
    app.textFields["email"].typeText("user@example.com")
    app.buttons["Login"].tap()
    sleep(3)  // "Wait for the API"
    XCTAssert(app.staticTexts["Welcome"].exists)  // Fails if API takes 3.1s
}

// ✅ Use waitForExistence instead
func testLogin() {
    app.textFields["email"].tap()
    app.textFields["email"].typeText("user@example.com")
    app.buttons["Login"].tap()
    let welcome = app.staticTexts["Welcome"]
    XCTAssertTrue(welcome.waitForExistence(timeout: 10))
}

System Alert Handling

The most common XCUITest flake on CI: a system permission alert (“Allow Notifications?”, “Allow Location?”) blocks your test. Locally you’ve already dismissed it. CI starts with a fresh simulator.

// Handle system alerts automatically
override func setUp() {
    super.setUp()
    continueAfterFailure = false
    app = XCUIApplication()

    // Dismiss system alerts when they appear
    addUIInterruptionMonitor(forAlertHandling: { alert in
        if alert.buttons["Allow"].exists {
            alert.buttons["Allow"].tap()
            return true
        }
        if alert.buttons["Allow While Using App"].exists {
            alert.buttons["Allow While Using App"].tap()
            return true
        }
        return false
    })

    app.launch()
}
XCUITest alert gotcha: addUIInterruptionMonitor only fires when your test interacts with the app after the alert appears. If the alert blocks your next tap(), the handler fires. But if you’re just asserting exists without tapping, the alert won’t be dismissed. Always follow alert-triggering actions with an interaction.

Test Plan Configuration for Flake Detection

Xcode’s test plans have built-in tools for finding and managing flaky tests. Most teams never configure them.

Test Repetition Modes

Mode What It Does When to Use
Until Failure Repeats a passing test until it fails (up to N times) Proving a test is flaky. Run 100 repetitions to find the failure.
Retry on Failure Retries a failing test up to N times CI stability (band-aid, not a fix). Marks tests as “expected flaky.”
Fixed Repetitions Runs every test exactly N times Statistical flake detection. Run 10x and check for any failure.
Flake hunting with repetitions: In your test plan, set repetition to “Until Failure” with 50 repetitions. Run the suspect test. If it fails once in 50, you’ve confirmed it’s flaky and you have the failure log. Feed that log to Claude Code for diagnosis.

Randomized Test Order

Enable “Randomize Execution Order” in your test plan. This catches order-dependent flakiness — tests that only pass because they happen to run after another test that sets up state they depend on. If tests suddenly fail with randomized order, they have hidden dependencies.

Parallel Execution

Xcode 26 enables parallel test execution by default in primary test plans. Parallel tests run in separate processes, which naturally exposes shared-state bugs. If a test that passes serially fails in parallel, it’s reading or writing shared mutable state.

Swift Concurrency Testing Challenges

Swift 6.2 and Xcode 26 introduce default actor isolation — new projects default to @MainActor for all code unless explicitly opted out. This changes how tests interact with async code and introduces new patterns for testing actor-isolated types.

Testing @MainActor Code

// The ViewModel is implicitly @MainActor (Swift 6.2 default)
class ProductListViewModel: ObservableObject {
    @Published var products: [Product] = []

    func loadProducts() async throws {
        products = try await service.fetchProducts()
    }
}

// Test must also run on MainActor
@MainActor
func testLoadProducts() async throws {
    let vm = ProductListViewModel(service: MockProductService())
    try await vm.loadProducts()
    XCTAssertEqual(vm.products.count, 3)
}
Swift 6.2 testing pitfall: With default actor isolation set to MainActor, your test methods are implicitly @MainActor too. If your test calls a nonisolated async function, you may get unexpected behavior because the continuation resumes on the main actor. Be explicit about isolation in tests — annotate with @MainActor or nonisolated to make intent clear.

Swift Testing Framework (Xcode 26)

The Swift Testing framework (import Testing) is the modern alternative to XCTest. It runs tests in parallel by default, has better async support, and includes test attachments for attaching screenshots, logs, or data to test results. Use the .serialized trait when tests must run sequentially:

import Testing

@Suite(.serialized) // Forces sequential execution
struct DatabaseTests {
    @Test func insertAndFetch() async throws {
        let db = try TestDatabase()
        try await db.insert(Product(name: "Widget"))
        let products = try await db.fetchAll()
        #expect(products.count == 1)
    }
}

Shared Mutable State: The Silent Flake Factory

The #1 cause of order-dependent flakiness is shared mutable state that leaks between tests. Here are the iOS-specific culprits:

Shared State Isolation Checklist

  • Singletons: Use dependency injection instead. If you must use singletons, reset them in tearDown()
  • UserDefaults: Use a test-specific suite: UserDefaults(suiteName: "TestSuite") and remove it in tearDown with UserDefaults.standard.removePersistentDomain(forName: "TestSuite")
  • Core Data: Use NSInMemoryStoreType for test stores. Never share a persistent container across tests.
  • Keychain: Use a test-specific access group or clear keychain items in tearDown. CI runners may share keychain state between test runs.
  • FileManager: Use a temporary directory per test: FileManager.default.temporaryDirectory.appendingPathComponent(UUID().uuidString)
  • NotificationCenter: Use a test-specific instance instead of .default. Leftover observers from previous tests can trigger unexpected behavior.
  • URLSession caches: Use URLSessionConfiguration.ephemeral in tests to avoid cached responses from previous runs.
Quick singleton isolation pattern: Instead of NetworkService.shared everywhere, accept the service as an init parameter with a default: init(network: NetworkServiceProtocol = NetworkService.shared). Tests pass a mock. No global state touched.

CI Simulator Issues

Half of all “flaky” tests aren’t actually flaky — they’re failing because of CI simulator problems. Before blaming your test code, check the environment.

Symptom

  • Tests time out waiting for simulator
  • Crash in CoreSimulator during test
  • Test passes locally, always fails on CI
  • XCUITest can’t find UI elements

Fix

  • Pre-boot simulator before test run: xcrun simctl boot
  • Delete and recreate simulator: xcrun simctl delete all
  • Check Xcode version matches CI runner version
  • Disable animations: app.launchArguments += ["-UIViewAnimationsEnabled", "NO"]

Building a /diagnose-flaky Skill

The /diagnose-flaky Skill

# .claude/commands/diagnose-flaky.md

Diagnose this flaky test and recommend a fix.

## Input:
I'll provide:
- Test name and file path
- Failure logs from multiple CI runs (the more the better)
- The test source code
- Any relevant production code the test exercises

## Analysis steps:
1. **Classify** the flakiness:
   - Timing-dependent (race condition, hardcoded delays)
   - Order-dependent (shared state, missing tearDown)
   - Environment-dependent (locale, timezone, simulator)
   - Resource-dependent (network, file system, keychain)

2. **Identify the root cause** from the failure pattern:
   - Do failure values vary randomly? → race condition
   - Does it fail only after a specific test? → order dependency
   - Does it fail only on CI, never locally? → environment issue
   - Does the assertion message reference nil/timeout? → async timing

3. **Recommend a fix** with concrete Swift code:
   - Replace sleep() with waitForExistence() or expectations
   - Add proper async/await synchronization
   - Isolate shared state (UserDefaults suite, in-memory Core Data)
   - Add system alert handling for XCUITest
   - Mock external resources

4. **Verify the fix** strategy:
   - Suggest test repetition count to confirm stability
   - Recommend test plan settings (randomized order, parallel)

Xcode 26’s Coding Intelligence can help with test failures directly in the IDE. When a test fails, Swift Assist can explain the assertion failure and suggest fixes inline. For quick, single-test fixes, this is faster than switching to Claude Code.

Use Claude Code when you need to:

  • Correlate test failures with recent code changes (git log)
  • Audit an entire test suite for flakiness patterns
  • Generate a prioritized flake remediation plan

Swift Testing Attachments (Xcode 26)

Swift Testing now supports test attachments — attach screenshots, logs, or arbitrary data to test results. For flaky XCUITests, attach a screenshot on failure. When the test flakes on CI, you’ll see exactly what the screen looked like. Feed the screenshot + failure log to Claude Code for visual + textual diagnosis.
