Production Monitoring & Crash Reporting

Memory Map

The Big Idea

Production is a different planet — no debugger, no LLDB, no Instruments. But production data is structured: crash reports, MetricKit payloads, Crashlytics clusters. Extract the data, feed it to AI, get back prioritized fixes. 80% of crashes come from 3-5 unique issues — find those and you fix most of the pain.

🔥 What Crashlytics Captures

  1. 💥 Crash Stacktraces

    Symbolicated with dSYMs. Grouped into clusters by root cause.

  2. ⚠️ Non-Fatal Exceptions

    Crashlytics.record(error:) — degraded experiences that haven’t crashed yet. Gold for AI.

  3. 🍞 Breadcrumbs

    Automatic screen_view events showing what user did before crash. Powered by Google Analytics SDK.

  4. 🔑 Custom Keys

    Your state context: user tier, feature flags, thermal state, app state. Essential for AI pattern matching.

📖 Key Terms

Crash cluster
Crashlytics groups crashes by root cause. “500 users hit this force-unwrap” not “500 individual crashes.”
Non-fatal exception
Caught errors reported via Crashlytics.record(error:). Reveals degraded experiences before they become crashes.
Breadcrumbs
Automatic screen_view events (via Google Analytics) showing user actions before a crash. Critical context for AI.
MXMetricPayload
Daily MetricKit delivery: CPU, memory, disk, network, launch, hangs. A health check-up report for your app.
MXDiagnosticPayload
MetricKit diagnostic data: crash logs, hang call stacks, disk write exceptions, CPU exceptions.
Crash-free rate
Percentage of sessions without crashes. Track across versions to spot regressions. Target: >99.5%.
.ips files
On-device crash logs from Settings > Privacy > Analytics. Must be symbolicated with matching dSYMs to be readable.
/triage-crash skill
Claude Code command: severity assessment, user impact estimate, root cause hypothesis, fix priority with code.

🧪 Quick Quiz

Why are non-fatal exceptions “gold” for AI analysis?

Non-fatals reveal degraded experiences — fallback code paths, retry storms, stale caches. They indicate bugs that haven’t crashed yet but will. Feed your top 10 non-fatals to AI to find hidden issues before users complain.

What three things does AI need to diagnose a version regression?

Changelog (what code shipped), crash rates per version, and MetricKit/Crashlytics metrics before and after. Numbers alone are ambiguous — AI needs the context of which changes went into each release.

Why can’t you read on-device .ips files without dSYMs?

On-device .ips files are unsymbolicated — they show raw memory addresses instead of function names. You need the matching dSYM (same build UUID) to translate addresses back to code. Always archive builds and keep dSYMs.

What’s the difference between MetricKit and Xcode Organizer?

MetricKit delivers raw payloads (JSON) in-app daily — you can upload them to your backend and run AI analysis. Xcode Organizer aggregates App Store data with visual trends but requires no code changes. MetricKit is more detailed; Organizer is more convenient.

When should you use Xcode 26 AI vs Claude Code for production crashes?

Xcode AI for quick inline explanations of a single crash in Organizer. Claude Code for triaging 20+ clusters, correlating crashes with MetricKit metrics, complex multi-file state issues, or generating team fix plans via XcodeBuildMCP automation.

📌 Key Takeaways

  • 80% of crashes come from 3-5 issues. Triage the top clusters first.
  • Non-fatal exceptions reveal degraded experiences before they become crashes. Report them.
  • Add custom keys + breadcrumbs to Crashlytics — they’re the context AI needs for pattern matching.
  • MetricKit = daily production health. Compare across versions to catch regressions instantly.
  • Version comparison pattern: changelog + crash rate + metrics = AI pinpoints which change caused the regression.
  • Build /triage-crash for severity-based prioritization with user impact and fix code.
  • Always keep dSYMs. Without them, crash reports are just memory addresses.
