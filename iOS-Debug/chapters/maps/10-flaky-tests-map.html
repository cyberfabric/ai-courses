Chapter 10

Flaky Test Diagnosis

Memory Map
💡

The Big Idea

A flaky test is the “boy who cried wolf” of your CI pipeline. After a few false alarms, teams stop trusting the test suite and merge through real failures. AI spots the pattern across multiple runs that a human would need an hour to find.

🏷️ The Four Categories of Flakiness

⏱️
Timing-Dependent
Race conditions, hardcoded sleep(), tight timeouts. Passes on fast machines, fails on slow CI. Fix: proper async waiting.
📋
Order-Dependent
Shared mutable state (singletons, UserDefaults, DB). Passes alone, fails after another test. Fix: isolate state, proper tearDown.
🖥️
Environment-Dependent
Simulator version, locale, timezone, date logic. Passes on one machine, fails on another. Fix: explicit locale, mock dates.
📡
Resource-Dependent
Network calls to real servers, file system races, keychain on CI. Fix: mock external resources, stub I/O.

🏁 The Classic Race Condition Pattern

  1. 🔥 The Bug

    ViewModel uses Task { } (fire-and-forget). Test asserts immediately without waiting. Count varies: 0, 1, 2, or 3.

  2. 🔍 The Clue

    Assertion values vary randomly across runs. Passes locally (fast Mac), fails on CI (slow runners).

  3. ✅ Fix A: Make It Awaitable

    Change loadProducts() to async. Test uses try await vm.loadProducts(). Deterministic.

  4. ✅ Fix B: Use Expectations

    If Task {} is required (SwiftUI), observe the @Published property with XCTestExpectation + fulfillment(of:).

📱 XCUITest Synchronization

Antipattern Problem Fix
sleep(3) Arbitrary wait. Fails if API takes 3.1s. waitForExistence(timeout: 10)
Missing alert handler System permission alert blocks test on CI (fresh simulator). addUIInterruptionMonitor in setUp
No animation disable Animations delay element availability. -UIViewAnimationsEnabled NO launch arg
Hardcoded element text Fails when locale or dynamic content changes. Use accessibility identifiers instead

⚙️ Test Plan Configuration

Setting Purpose When to Use
Until Failure Repeats passing test until it fails Proving a test is flaky (run 50-100 reps)
Retry on Failure Retries failing test up to N times CI stability band-aid (not a real fix)
Randomize Order Runs tests in random sequence Finding order-dependent flakes
Parallel Execution Tests run in separate processes Finding shared-state bugs (default in Xcode 26)

🧹 Shared State Isolation Checklist

🔄
Singletons
Use dependency injection. Accept service as init parameter with default: init(network: Protocol = .shared)
⚙️
UserDefaults
Use test-specific suite: UserDefaults(suiteName: "TestSuite"). Remove in tearDown.
💾
Core Data
Use NSInMemoryStoreType. Never share a persistent container across tests.
🔑
Keychain
Test-specific access group or clear items in tearDown. CI runners share keychain state.
📂
FileManager
Temp directory per test: temporaryDirectory.appending(UUID().uuidString)
📢
NotificationCenter
Use a test-specific instance, not .default. Leftover observers cause ghost failures.

📖 Key Terms

Flaky test
A test that passes and fails intermittently with no code change. Erodes CI trust and leads to “retry and merge” culture.
Race condition
Unpredictable execution order between threads. Assertion values vary randomly. Fix: await the async operation properly.
waitForExistence(timeout:)
XCUITest method that waits for an element to appear. Always use instead of sleep().
addUIInterruptionMonitor
Handles system alerts (permissions) in XCUITests. Only fires after a test interaction, not passive assertions.
Test repetition modes
Xcode test plan settings: Until Failure (prove flakiness), Retry on Failure (CI band-aid), Fixed Repetitions (statistical check).
Swift Testing (.serialized)
Trait in the Swift Testing framework that forces sequential execution. Use for tests with unavoidable shared state.
/diagnose-flaky skill
Claude Code command that classifies flakiness category, identifies root cause from multi-run failure logs, and recommends fixes.
Test attachments (Xcode 26)
Attach screenshots, logs, or data to Swift Testing results. On flaky XCUITest failure, see exactly what the screen showed.

🧪 Quick Quiz

A test asserts count == 3 but randomly gets 0, 1, 2, or 3. What category of flakiness is this?

Timing-dependent (race condition). The test asserts before the async operation completes. The varying values show the assertion races against data loading — sometimes none have arrived, sometimes all have. Fix: await the operation or use XCTestExpectation.

Why does addUIInterruptionMonitor sometimes not fire?

It only fires when the test interacts with the app (tap, swipe) after the alert appears. If you’re just asserting exists without interacting, the alert won’t be dismissed. Always follow alert-triggering actions with an interaction.

How do you prove a test is flaky?

In your test plan, set repetition to “Until Failure” with 50+ repetitions. If it fails once in 50 runs, it’s confirmed flaky — and you have the failure log to feed to AI.

A test passes individually but fails when run after testSetupUser(). What’s the likely cause?

Order-dependent flakiness from shared mutable state. testSetupUser() likely writes to a singleton, UserDefaults, or database that the failing test reads. Fix: isolate state with test-specific instances and proper tearDown.

Why does Xcode 26 enable parallel test execution by default?

Parallel tests run in separate processes, which naturally exposes shared-state bugs. Tests that pass serially but fail in parallel are reading or writing shared mutable state — a latent flakiness source that parallel execution surfaces early.

📌 Key Takeaways

  • Classify flakiness first: timing, order, environment, or resource. The fix depends on the category.
  • Feed AI multiple failure logs — one failure is ambiguous, three tell the story.
  • Never use sleep() in tests. Use waitForExistence(), expectations, or async/await.
  • Isolate shared state: test-specific UserDefaults suites, in-memory Core Data, DI over singletons.
  • Use “Until Failure” with 50 reps to prove flakiness and capture the failure log.
  • Enable randomized order + parallel execution to surface hidden dependencies.
  • Build /diagnose-flaky for systematic flake triage across CI runs.
