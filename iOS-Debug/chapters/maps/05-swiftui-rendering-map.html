SwiftUI Rendering Performance

Memory Map
💡

The Big Idea

A SwiftUI view that re-evaluates its body 47 times during a scroll is repainting the house every time someone opens the front door. The rendering is cheap — the evaluation is where performance dies. Use _printChanges() to see it, @Observable to fix it.

🔄 The Rendering Pipeline

📝 Evaluate body
→
🔍 Diff old vs new
→
🎨 Render changes

You control step 1. Steps 2 & 3 are fast. Unnecessary evaluations = wasted CPU.

🎯 The 6 Performance Traps

  1. 🏭 Objects Created in body

    DateFormatter() allocated every evaluation. Fix: static let property.

  2. 📡 Reading Too Many Observable Properties

    View reads 5 properties → re-evaluates when any of them change. Fix: split into focused child views.

  3. 🪪 Unstable View Identity

    .id(UUID()) or non-stable ForEach IDs. State resets, animations break. Fix: persistent IDs.

  4. 🧮 Heavy Computation in body

    Sorting/filtering large collections every evaluation. Fix: cached computed state or .task.

  5. 📌 Missing @State for Local State

    Plain var resets on parent rebuild. Fix: use @State for view-local mutable values.

  6. ⚖️ EquatableView Neglect

    Expensive body with complex inputs. Fix: Equatable conformance to skip body when inputs match.

📖 Key Terms

View body evaluation
SwiftUI calling your body property to get a new UI description. Expensive if body does heavy work.
_printChanges()
Undocumented debug method. Add to body to see what triggered re-evaluation. Remove before shipping.
@Observable
Observation framework macro (iOS 17+). Property-level tracking — only views reading the changed property re-evaluate.
Cause & Effect Graph
Xcode 26 Instruments visualization showing which state change caused which view body to re-evaluate.
Structural identity
View identity based on position in the view tree. if/else branches create different identities.
Explicit identity
Identity set via .id() modifier or ForEach with Identifiable. Changing ID destroys the view.
EquatableView
Wraps a view with Equatable conformance so SwiftUI skips body evaluation when inputs haven't changed.
Derived state pattern
Parent computes a simple value from a complex model, passes it to child. Child decoupled from broad observation.

🧪 Quick Quiz

What does @identity changed mean in _printChanges() output?

The view was destroyed and recreated — SwiftUI considers it a new view, not an update. All @State resets, animations break. Usually caused by unstable .id() or non-stable ForEach identifiers.

Why does migrating from ObservableObject to @Observable dramatically reduce re-evaluations?

ObservableObject notifies all observers when any @Published property changes (object-level). @Observable tracks at the property level — only views that read the specific changed property re-evaluate.

Why shouldn't you create a DateFormatter inside body?

body can be called many times per second. Each call allocates a new formatter. Use a static let property instead — allocated once, reused forever.

What's the derived state pattern and when should you use it?

Instead of passing a full ViewModel to a child view, the parent computes the specific value the child needs and passes it as a simple property. Use when a child only needs one piece of data from a complex observable.

How do you verify an AI-suggested SwiftUI optimization actually worked?

Check _printChanges() output before and after. If the evaluation count doesn't drop, the suggestion didn't work. Never trust the fix without measuring.

📌 Key Takeaways

  • _printChanges() is your #1 tool — always start here before Instruments.
  • Xcode 26 SwiftUI Instrument has a Cause & Effect Graph — use Swift Assist to explain it.
  • Migrate ObservableObject → @Observable for property-level tracking. Single biggest performance win (501 evaluations → 1).
  • Keep identity stable. No .id(UUID()). Unstable identity = destroyed state + broken animations.
  • Don't create objects in body. Formatters, sort operations, allocations all belong outside.
  • Use the derived state pattern to decouple child views from broad observables.
  • Always verify AI suggestions with _printChanges() — if the count doesn't drop, the fix didn't work.
