The Big Idea

Without measurable thresholds, performance degrades one PR at a time — 50ms here, 2MB there. A performance budget is a contract enforced in CI: when a PR violates the budget, the build fails. No exceptions, no “we’ll fix it later.”

📖 Key Terms

Performance budget
Concrete threshold for a metric (launch, memory, FPS, size). Three tiers: Target, Warning, Ceiling. CI fails at Ceiling.
XCTMetric
XCTest protocol for measuring performance. Implementations: ApplicationLaunchMetric, CPUMetric, MemoryMetric, ClockMetric, OSSignpostMetric.
Baseline
Reference measurement set after first performance test run. Future runs compare against baseline; regressions beyond stddev threshold fail.
App Thinning
Apple technology that delivers only device-specific code and assets. Use asset catalogs to maximize size reduction.
/check-budgets
Claude Code skill comparing current metrics against budget tiers. Returns status per metric + trend direction + fix suggestions.
.xcresult
Xcode result bundle containing test results and performance metrics. Extract with xcresulttool for CI automation.
Debugging decision tree
Flowchart routing symptoms (crash, hang, leak, stutter, etc.) to the right chapter, tool, and Claude Code skill.

🧪 Quick Quiz

Why set three budget tiers instead of just one threshold?

Target gives the team a goal. Warning signals degradation early (time to investigate). Ceiling blocks merges (hard stop). This gives room to iterate without instantly breaking the build, while still preventing regression from shipping.

How do XCTest performance tests detect regressions?

You run the test once and set a baseline. Future runs compare against the baseline. If the measured value exceeds the baseline by more than the standard deviation threshold, the test fails. Set baselines on consistent hardware (CI runner, not your laptop).

What’s the 200MB cellular download limit?

Apple blocks automatic cellular downloads for apps over 200MB. Users must be on WiFi or explicitly approve the download. This makes app size a conversion metric — large apps get fewer installs.

What’s the permanent pattern that won’t change even as tools evolve?

Capture structured data → feed it to AI → get actionable fixes → enforce budgets in CI. The specific tools change (Xcode 27, smarter AI), but this workflow is permanent.

📌 Key Takeaways

  • Set three-tier budgets (Target / Warning / Ceiling) for launch, memory, FPS, and app size.
  • XCTest performance tests with baselines catch regressions in CI before they ship.
  • Use the debugging decision tree to route symptoms to the right tool and skill.
  • 13 Claude Code skills cover the full debugging lifecycle: crashes → profiling → quality → production.
  • App size: asset catalogs, HEIF/WebP, dead code stripping, SPM audit. Stay under 200MB cellular limit.
  • The permanent pattern: capture structured data → feed to AI → actionable fixes → enforce in CI.
  • AI amplifies your debugging intuition. Always validate suggestions against the actual code.
