AI-Assisted Instruments Profiling

Memory Map

The Big Idea

Instruments captures structured numerical data — call trees, sample counts, allocation sizes, thread states. That’s exactly what AI analyzes best. Stop staring at flame graphs guessing “is 340 samples a lot?” and start getting answers like “This method accounts for 38% of main-thread CPU — here’s the fix.”

📊 Which Instrument for Which Question?

Instrument Answers Key Metric
⏱️ Time Profiler Where does CPU time go? Self Weight (ms) per method
🧠 Allocations Where does memory go? Persistent vs transient bytes
🧵 System Trace Is my thread running or blocked? Running vs blocked duration
⚡ Power Profiler Which subsystem drains energy? mW per subsystem (per-core)
🧩 Foundation Models How fast is on-device AI inference? Tokens/sec, Neural Engine usage

⏱️ Time Profiler Essentials

📏
Weight vs Self Weight
Weight = total time including callees. Self Weight = time in the function itself. High Weight + zero Self Weight = it’s just a wrapper.
🔥
Flame Graph
Wider bar = more CPU. Enable “Invert Call Tree” to see leaf functions (actual work) at top. Easier for AI to analyze.
📊
Compare Sessions
Xcode 26: View > Detail Area > Compare With. Shows which functions got faster or slower between runs.
📋
Copy Call Tree
Select time range > Call Tree > Cmd+A > Cmd+C. Tab-indented text with Weight, Self Weight, Symbol — perfect for AI.

🧠 Allocations: Transient vs Persistent

🎉 Transient = guests who leave
Temp buffers, intermediate strings. High churn = CPU overhead.
vs
🏠 Persistent = guests who stay
Cached images, singletons, leaks. Growth = memory leak.

Generation Analysis: Click “Mark Generation” at key moments. Anything that persists between marks is suspect.

🧵 System Trace: Running vs Blocked

Aspect CPU Bound (Running) I/O Bound (Blocked)
System Trace color Green bars Gray/red bars
Time Profiler shows it? Yes (samples running threads) No (misses blocked threads!)
Fix Optimize algorithm, cache, move to background Async I/O, reduce lock contention
Symptom High CPU in Time Profiler Low CPU but laggy UI

💻 xctrace: CLI Profiling for CI

  1. 📝 Record a Trace

    xcrun xctrace record --template "Time Profiler" --attach "MyApp" --time-limit 10s --output /tmp/profile.trace

  2. 📂 Export Data as XML

    xcrun xctrace export --input /tmp/profile.trace --xpath '...time-profile...' --output /tmp/data.xml

  3. 🤖 Feed to Claude Code

    Pipe exported XML to /profile-cpu skill for automated analysis. XcodeBuildMCP automates the full pipeline.

⚠️ Profiling Pitfalls

🐛
Debug Build Profiling
Debug disables optimizations (-Onone). A 50ms method in Debug may be 2ms in Release. Always profile Release builds.
🎯
Optimizing Wrappers
800ms Weight + 0ms Self Weight = just a container. Look at Self Weight to find the real bottleneck deeper in the tree.
😴
Ignoring Blocked Time
Low CPU but laggy? Time Profiler misses blocked threads. Use System Trace to find I/O waits and lock contention.
📱
Wrong Device
Profile on the lowest-spec device you support (iPhone SE). iPhone Pro Max brute-forces past inefficient code.

📖 Key Terms

Self Weight
Time spent in a function itself (not its callees). The key metric for finding actual bottlenecks in Time Profiler.
Flame graph
Visual call tree where bar width = CPU time. Invert Call Tree shows leaf functions (actual work) at top.
Generation analysis
Mark moments in Allocations instrument. Objects that persist between marks are suspect — leaks or unbounded caches.
Priority inversion
High-priority thread waits for low-priority thread’s lock. System Trace shows yellow “priority boosted” events. Can happen with Swift actors.
xctrace
CLI tool behind Instruments.app. Record traces, export as XML, automate in CI. Essential for XcodeBuildMCP integration.
Foundation Models Instrument
Xcode 26 instrument for profiling on-device AI inference: token rate, memory, Neural Engine utilization.
/profile-cpu skill
Claude Code command that takes a Time Profiler call tree and returns prioritized bottlenecks (red/yellow/green) with fixes.
Compare With (Xcode 26)
View > Detail Area > Compare With. Compares call trees across profiling sessions to see which functions got faster or slower.

🧪 Quick Quiz

What’s the difference between Weight and Self Weight in Time Profiler?

Weight is the total time spent in a function including all its callees. Self Weight is time in the function itself. A function with high Weight but 0 Self Weight is just a container — the real work is deeper in the call tree. AI uses Self Weight to find actual bottlenecks.

Why does Time Profiler miss blocked threads?

Time Profiler is a statistical sampler that only records threads when they’re actively running on the CPU. If a thread is blocked waiting for I/O, a lock, or a network response, it won’t appear in samples. Use System Trace to see blocked time.

Why must you always profile Release builds?

Debug builds disable compiler optimizations (-Onone), include extra runtime checks, and skip inlining. A method that takes 50ms in Debug might take 2ms in Release. Profiling Debug builds gives misleading results.

How does generation analysis help find memory leaks?

Click “Mark Generation” at key moments (e.g., navigate forward and back). Objects allocated between marks that persist after the next mark are suspect — they should have been deallocated but weren’t, indicating a leak or unbounded cache.

What’s the multi-instrument AI pattern?

Feed Time Profiler (CPU), Allocations (memory), System Trace (thread states), and Swift source code to Claude Code together. AI correlates across instruments: “This method is slow because it allocates 200 strings per call while holding a lock that blocks the main thread.”

📌 Key Takeaways

  • AI is a profiler translator — it turns sample counts and call trees into “fix this method, here’s how.”
  • Self Weight finds actual bottlenecks. High Weight + 0 Self Weight = just a wrapper.
  • Time Profiler misses blocked threads. Low CPU + laggy UI? Use System Trace.
  • Always profile Release builds on the lowest-spec device you support.
  • Combine instruments for AI: Time Profiler + Allocations + System Trace + source code = full picture.
  • xctrace CLI enables CI profiling. XcodeBuildMCP automates the record → export → AI pipeline.
  • Build /profile-cpu for reusable call tree analysis with red/yellow/green severity levels.
