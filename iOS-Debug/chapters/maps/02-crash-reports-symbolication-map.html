The Big Idea

An unsymbolicated crash report is like a crime scene with all evidence labels in hex. Symbolicate first, analyze second — then the exception type alone tells AI the crash category before reading a single frame.

🎯 Top 5 Swift Crash Patterns AI Recognizes

  1. Force-Unwrap Nil (!)

    SIGABRT + "unexpectedly found nil." Fix: replace ! with guard let / if let.

  2. Array Out-of-Bounds

    SIGTRAP + "Index out of range." Check array source (API?) and index source (hardcoded? off-by-one?).

  3. Unrecognized Selector

    SIGABRT + "unrecognized selector sent to instance." Stale Storyboard outlet or renamed @objc method.

  4. Concurrency Precondition Failure

    SIGTRAP + "sending non-Sendable type" or "Data race." Swift 6 strict concurrency catches at runtime.

  5. KeyPath / KVO Crash

    SIGABRT during observation. Property was renamed/removed but KVO observation wasn't updated.

📖 Key Terms

.ips file
JSON crash report format (iOS 15+). Contains exception, threads, and binary images. Found in device analytics or Xcode Organizer.
dSYM
Debug Symbol file. Maps memory addresses to source code locations. UUID must match the crashed binary.
Symbolication
Process of converting hex addresses (0x104a23def) into human-readable function names + line numbers.
atos
Apple command-line tool for manual symbolication. Takes arch, dSYM path, load address, and crash address.
Logger (os_log)
Apple's structured logging API (iOS 14+). Categorized by subsystem + category, privacy-aware, survives termination.
OSLogStore
Swift API (iOS 15+) for programmatic log retrieval. Filter by date, subsystem, category. Export for AI consumption.
SIGABRT vs SIGSEGV vs SIGTRAP
SIGABRT = deliberate abort (force unwrap). SIGSEGV = invalid memory access. SIGTRAP = Swift runtime trap (bounds, concurrency).
/analyze-crash
Claude Code skill that classifies exception type, reads backtrace, identifies root cause, and suggests fix with confidence level.

🧪 Quick Quiz

What's the difference between SIGABRT and SIGTRAP in Swift?

SIGABRT means the app called abort() deliberately — force-unwrap nil, fatalError(), preconditionFailure(). SIGTRAP is a Swift runtime trap — array out-of-bounds, implicit unwrap nil, concurrency precondition failure.

Why must the dSYM UUID match the crashed binary's UUID?

Different builds produce different address-to-symbol mappings. Using the wrong dSYM gives you wrong function names, causing you to debug the wrong code — worse than no symbols at all.

How do you manually symbolicate a single crash frame?

atos -arch arm64 -o MyApp.app.dSYM/Contents/Resources/DWARF/MyApp -l <load_address> <crash_address>. Load address comes from the binary images section of the crash report.

Why is os_log better than print() for debugging?

os_log survives app termination, supports subsystem/category filtering, has privacy controls (.public/.private), minimal performance overhead, and can be queried programmatically via OSLogStore for AI analysis.

When does a crash in UIKit/SwiftUI mean YOUR code is the cause?

Almost always. Look for your code on other threads (background UI updates) or earlier in the call chain. Common: modifying UI off main thread (UIKit crash), accessing managed objects across contexts (Core Data crash), mutating state during view update (SwiftUI/AttributeGraph crash).

📌 Key Takeaways

  • Symbolicate first, analyze second. Hex addresses waste everyone's time — including AI's.
  • The exception type + signal is the most important field. It classifies the crash before you read a single frame.
  • Use atos for single frames, symbolicatecrash for full reports. Always verify dSYM UUID matches.
  • Replace print() with Logger (os_log). Structured, private, queryable via OSLogStore.
  • Xcode AI for quick inline crash explanations. Claude Code /analyze-crash for deep analysis and multi-device comparison.
  • System framework crashes are usually caused by your code — check other threads and earlier call chain.
