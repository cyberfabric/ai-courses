Memory Leak Detection with AI

Memory Map
💡

The Big Idea

A crash is a bullet; a memory leak is poison. Retain cycles are the #1 cause of leaks in Swift. The retention chain from Memory Graph Debugger is structured data AI reads perfectly — paste it and get an instant diagnosis.

🔄 The 6 Classic Retain Cycle Patterns

  1. 🔗 Closure Capturing self

    Closure stored as property captures self strongly. Fix: [weak self] in capture list.

  2. 👥 Strong Delegate

    var delegate: MyDelegate instead of weak var. Delegator ↔ delegate retain each other.

  3. ⏰ Timer Retain Cycle

    Timer.scheduledTimer retains the closure → closure captures self. Must invalidate to break.

  4. 📢 NotificationCenter Observer

    Block-based addObserver returns token that retains closure. Store token, remove in deinit.

  5. 🖥️ CADisplayLink Target

    Strongly retains target. If target owns the link → cycle. Fix: weak proxy object.

  6. 🔀 Combine sink + store(in:)

    sink { self.update($0) }.store(in: &cancellables) — self owns cancellables, sink captures self. Fix: [weak self].

🖼️ SwiftUI Memory Pitfalls

✅
@StateObject (Owner)
Creates & owns the object. Use in the parent view that creates the ViewModel. Created once.
📎
@ObservedObject (Borrower)
Doesn't own. Pass from parent. Never initialize inline — gets recreated every re-evaluation.
⚠️
@Observable + @State
Known leak on iOS 17 early releases: objects created but never released. Verify with Memory Graph Debugger. Fixed in iOS 18.

📖 Key Terms

ARC (Automatic Reference Counting)
Swift's memory management. Deterministic and efficient, but can't break retain cycles.
Retain cycle
Two+ objects holding strong references to each other, preventing deallocation. #1 cause of iOS memory leaks.
Memory Graph Debugger
Xcode tool that snapshots the heap, shows reference arrows, and auto-detects leaks with purple markers.
Generation Analysis
Instruments technique: mark before/after an action. Objects that persist but shouldn't = leak evidence.
Capture list
[weak self] or [unowned self] in a closure declaration. Controls how the closure references external objects.
OOM Jetsam
iOS silently kills apps under memory pressure. No crash dialog. State lost. Users blame your app.
EXC_RESOURCE
Exception when your app exceeds its memory limit. Shows in crash reports as a resource violation.
deinit
Swift deinitializer. If it never prints, your object is leaking. The simplest leak test.

🧪 Quick Quiz

What's the simplest way to test if a view controller leaks?

Add a print() in deinit. Push the VC, pop it. If deinit never prints, something is retaining it. Then use Memory Graph Debugger to find what.

Why default to weak over unowned?

An unexpected nil from weak is debuggable (just an optional). An unowned access after deallocation is an EXC_BAD_ACCESS crash at an unrelated time — extremely hard to trace. The performance difference is negligible.

How does a Combine sink create a retain cycle?

sink { self.update($0) }.store(in: &cancellables) — the sink closure captures self strongly, and cancellables is a property owned by self. Fix: use [weak self] in the sink closure.

When should you use @StateObject vs @ObservedObject?

@StateObject in the parent that creates & owns the VM (created once). @ObservedObject in children that receive it. Never initialize @ObservedObject inline — it gets recreated on every view re-evaluation.

What makes Memory Graph Debugger data ideal for AI analysis?

The retention chain is structured data: object → reference → object → reference. AI reads it like a linked list and immediately identifies where the cycle is and which reference should be weak.

📌 Key Takeaways

  • Memory leaks are silent killers. No crash dialog — iOS just jetsams your app and users lose state.
  • Retain cycles are the #1 cause. Two objects with strong refs to each other = leaked forever.
  • Memory Graph Debugger for pinpointing cycles, Instruments Allocations for tracking growth over time.
  • The retention chain is structured data AI reads perfectly — paste into /analyze-leak.
  • Default to weak over unowned. Always. The crash risk isn't worth the zero performance gain.
  • Use /scan-for-leaks as a static analysis step in code reviews — catch leaks before they ship.
