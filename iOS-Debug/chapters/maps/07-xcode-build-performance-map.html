The Big Idea

A 90-second build × 40 builds/day = 1 hour lost daily. The real cost isn’t time — it’s flow state. A 15-second build keeps you in the zone; a 90-second build sends you to Twitter. Xcode 26’s compilation caching cuts 24-36% with one setting.

🐌 Finding Slow Expressions

  1. 🏳️ Add Compiler Flags

    -Xfrontend -warn-long-function-bodies=200 and -warn-long-expression-type-checking=200 (in ms).

  2. 🔨 Build and Check Warnings

    Xcode shows warnings for any expression or function body that exceeds the threshold.

  3. ✂️ Break Inference Chains

    Add explicit type annotations to intermediate let variables. Turns exponential type-checking into linear.

  4. 🤖 Feed to /analyze-build

    Paste warnings + build timeline + settings for a holistic build audit from Claude Code.

📖 Key Terms

Compilation caching (Xcode 26)
LLVM CAS-backed cache that reuses compilation results for unchanged code. Sub-function granularity. 24-36% faster clean builds.
Explicit modules
Default in Xcode 26. Better dependency info for the build system. Required for compilation caching.
Incremental compilation (singlefile)
Only recompiles files that changed. Essential for Debug builds. wholemodule in Debug = full recompile on every change.
EAGER_LINKING
Starts linking before all files finish compiling. Saves 10-20% on link phase.
Type inference chain
Long .map { }.filter { }.reduce { } chains force exponential type-checking. Fix with explicit type annotations.
-warn-long-function-bodies
Swift compiler flag that warns when a function body takes too long to type-check. Set threshold in ms (e.g., 200).
Modularization
Breaking a monolith into independent modules. Changes in one module don’t recompile others.
Package.resolved
SPM lock file. Commit it to prevent version drift and skip redundant resolution across team members.

🧪 Quick Quiz

Why should Debug builds never use wholemodule compilation?

wholemodule recompiles the entire module for any change. In Debug, you want singlefile (incremental) so only changed files recompile. Using wholemodule in Debug turns every one-line fix into a full rebuild.

How does Xcode 26’s compilation caching work?

It uses LLVM Content Addressable Storage (CAS) to cache compilation results at sub-function granularity. When source files haven’t changed, cached results are reused — even across branch switches and clean builds. Requires Explicit Modules.

Why do long type inference chains slow down compilation?

The Swift type checker has exponential worst-case behavior. .map { }.filter { }.reduce { } chains without type annotations force the compiler to explore all possible overload combinations. Explicit let variables with type annotations give anchoring points, reducing the search to linear.

What should you do with SwiftLint in your build pipeline?

Move it to a post-build action or CI step. SwiftLint runs as a build script phase that blocks compilation. Removing it from the critical path saves 5-10s per build without losing linting coverage.

Why commit Package.resolved to git?

Without it, every developer and CI run resolves SPM dependencies independently, wasting time and risking version drift. Committing the lock file ensures everyone uses the same exact versions.

📌 Key Takeaways

  • Enable compilation caching in Xcode 26 (COMPILATION_CACHE_ENABLE_CACHING=YES) for 24-36% faster clean builds.
  • Use -warn-long-function-bodies=200 and -warn-long-expression-type-checking=200 to find slow expressions.
  • Break type inference chains with explicit type annotations — the #1 fix for slow Swift compilation.
  • Never use wholemodule in Debug. Never optimize (-O) in Debug. Never generate dSYMs in Debug.
  • Audit SPM: pin versions, commit Package.resolved, use binary targets for large frameworks.
  • Move linting scripts to post-build or CI — they block the critical compilation path.
  • Modularize for long-term speed. AI suggests module boundaries from your file structure.
