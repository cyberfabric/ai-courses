Chapter 6

Network & URLSession Debugging

Use AI to diagnose SSL failures, timeout storms, async/await pitfalls, and the invisible network bugs that ruin your app

In Chapter 5, we made our SwiftUI views render efficiently. But the fastest view in the world is useless if it’s waiting 10 seconds for a JSON response that never arrives. Network debugging on iOS is uniquely painful: SSL, App Transport Security, Codable serialization failures, retry storms, and the new async/await threading model all conspire to make “it works on my Mac” the most dangerous sentence in mobile development.

The good news? Network issues produce structured data — request/response pairs, timing metrics, error codes — and structured data is exactly what AI analyzes best.

Why Network Debugging Is Uniquely Hard on iOS

SSL/TLS
Encrypted by default — you can’t see what’s in the request without a proxy
ATS
App Transport Security blocks non-HTTPS — silent failures if misconfigured
Codable
Type-safe decoding means one missing JSON key = crash or silent nil
async/await
New concurrency model changes where errors surface and how cancellation propagates

Step 1: Make Your Network Talk

Before you can debug, you need data. URLSession is remarkably quiet by default. Here are three ways to make it talk:

Option A: URLSessionTaskMetrics

Apple’s built-in performance data. Every request generates metrics with DNS lookup time, TLS handshake duration, request/response sizes, and connection reuse status:

func urlSession(_ session: URLSession, task: URLSessionTask,
                 didFinishCollecting metrics: URLSessionTaskMetrics) {
    for tx in metrics.transactionMetrics {
        let dns = tx.domainLookupEndDate?.timeIntervalSince(
            tx.domainLookupStartDate ?? .now) ?? 0
        let tls = tx.secureConnectionEndDate?.timeIntervalSince(
            tx.secureConnectionStartDate ?? .now) ?? 0
        let total = tx.responseEndDate?.timeIntervalSince(
            tx.fetchStartDate ?? .now) ?? 0

        logger.info("""
            \(tx.request.url?.path ?? "?") | \
            DNS: \(dns * 1000, format: .fixed(precision: 0))ms | \
            TLS: \(tls * 1000, format: .fixed(precision: 0))ms | \
            Total: \(total * 1000, format: .fixed(precision: 0))ms | \
            Reused: \(tx.isReusedConnection)
            """)
    }
}

Option B: os_log with Network Subsystem

import OSLog

let networkLogger = Logger(subsystem: "com.myapp", category: "network")

// Before request
networkLogger.info("→ \(request.httpMethod ?? "?") \(request.url?.absoluteString ?? "?")")

// After response
networkLogger.info("← \(statusCode) \(url) [\(duration)ms] \(bodySize)B")

Option C: Proxyman or Charles Proxy

For full request/response inspection including headers and bodies:

🦊

Proxyman

Native macOS/iOS app. Auto-configures simulator proxy. JSON syntax highlighting. iOS app uses local VPN for on-device capture. Easier setup than Charles. Export sessions as HAR files for AI analysis.

🔵

Charles Proxy

Industry standard. Java-based. Requires manual proxy configuration on device. Powerful SSL proxying with root cert install. Map Local/Remote for mocking. Throttle to simulate slow networks.

Simulator shortcut: Proxyman auto-detects the iOS Simulator and captures traffic without any proxy configuration. For real devices on iOS 26, you install the Proxyman CA certificate under Settings > General > VPN & Device Management, then enable it in Certificate Trust Settings.

AI Analysis

Pattern recognized: Timeout retry storm.

  • Problem: 10-second timeout × 4 attempts = 40 seconds of blocked loading. The user sees a blank screen for 40 seconds before getting an error.
  • No exponential backoff: Retries fire immediately after timeout. If the server is overloaded, this makes it worse.
  • Missing cancellation: If the user navigates away, the retries keep firing in the background.
  • Root cause hypothesis: DNS resolution failing (check if the domain resolves from device network) or server-side issue. The URLSessionTaskMetrics would tell us if DNS or TLS is where the time is spent.

NSURLError Cheat Sheet

These error codes are your first clue. AI recognizes them instantly:

Error Code Name Common Cause
-1001 NSURLErrorTimedOut Server not responding, DNS failure, too-short timeout
-1003 NSURLErrorCannotFindHost DNS resolution failed. Typo in hostname or no network.
-1009 NSURLErrorNotConnectedToInternet Device is offline. Check reachability before retrying.
-1200 NSURLErrorSecureConnectionFailed SSL/TLS handshake failed. Certificate issue or ATS violation.
-1202 NSURLErrorServerCertificateUntrusted Server cert not trusted. Expired cert, wrong domain, or missing intermediate.
-999 NSURLErrorCancelled Task was cancelled. Often expected — but if unexpected, check Task lifecycle.

SSL Pinning Failures: The Silent App Killer

SSL pinning is security that verifies the server’s certificate matches what your app expects. When it fails, the connection is silently refused — no helpful error dialog, just “network error.”

Root Cause: SSL Pinning Failure After Certificate Rotation

Symptom: App works fine in development, fails in production for 30% of users. Error: NSURLErrorSecureConnectionFailed (-1200).

Cause: The server’s SSL certificate was rotated to a new Certificate Authority. The app pins to the old CA’s public key. Users who got the new certificate from CDN edge servers get failures; users hitting cached old cert still work.

Fix: Pin to the public key hash (SPKI), not the full certificate. Include both the current and next CA’s key hashes. Always have a backup pin. Implement a kill-switch URL that disables pinning in emergencies.

Prevention: Claude Code’s /analyze-network skill flags single-pin configurations as a risk during code review.

SSL Pinning + Debug Proxies: If your app uses SSL pinning, Proxyman and Charles Proxy won’t work out of the box — the proxy’s certificate won’t match your pins. You need a #if DEBUG flag to disable pinning during development, or use Proxyman’s built-in SSL Pinning bypass for the iOS simulator.

async/await Networking Pitfalls

Swift concurrency changed how we write network code. It also introduced new categories of bugs:

Pitfall 1: Forgotten Cancellation

// This Task lives forever even if the view disappears
Task {
    let (data, _) = try await URLSession.shared.data(from: feedURL)
    let feed = try JSONDecoder().decode(Feed.self, from: data)
    self.items = feed.items  // May update a deallocated view
}

Fix: Use .task Modifier for View-Scoped Network Calls

// .task is automatically cancelled when the view disappears
.task {
    do {
        let (data, _) = try await URLSession.shared.data(from: feedURL)
        let feed = try JSONDecoder().decode(Feed.self, from: data)
        self.items = feed.items
    } catch is CancellationError {
        // Expected — view disappeared. Do nothing.
    } catch {
        self.errorMessage = error.localizedDescription
    }
}

Why this matters: Unscoped Task { } blocks don’t cancel when the view disappears. The request completes, the decode runs, and the result tries to update state on a view that no longer exists. With .task, SwiftUI cancels the task automatically.

Pitfall 2: Serial Requests That Should Be Parallel

// BAD: These run one after another — 3 network round trips
let user = try await fetchUser()
let posts = try await fetchPosts(for: user.id)
let notifications = try await fetchNotifications()
// Total time: fetchUser + fetchPosts + fetchNotifications

// GOOD: Parallel where possible
let user = try await fetchUser()
async let posts = fetchPosts(for: user.id)
async let notifications = fetchNotifications()
// posts and notifications run in parallel after user completes
let (p, n) = try await (posts, notifications)
// Total time: fetchUser + max(fetchPosts, fetchNotifications)

Pitfall 3: Codable Decoding on the Main Thread

By default, URLSession.shared.data(from:) returns on an arbitrary thread, but if you’re in a @MainActor context, the decoding happens on the main thread. Large JSON payloads can cause a noticeable hitch:

Fix: Decode Off the Main Actor

// Move heavy decoding off MainActor
func loadFeed() async throws -> Feed {
    let (data, _) = try await URLSession.shared.data(from: feedURL)

    // Decode on a background thread
    return try await Task.detached(priority: .userInitiated) {
        try JSONDecoder().decode(Feed.self, from: data)
    }.value
}

URLSessionConfiguration: The Hidden Settings Panel

Most iOS developers use URLSession.shared and never think about configuration. That’s like driving a car without adjusting the mirrors. Key settings AI audits:

  • timeoutIntervalForRequest — Default is 60 seconds. Too long for mobile. Set to 15-30 seconds for API calls.
  • timeoutIntervalForResource — Default is 7 days (!). The total time allowed for a resource transfer. Set this for file downloads.
  • waitsForConnectivity — Default is false. When true, URLSession waits for network instead of failing immediately. Great for offline-to-online transitions.
  • httpMaximumConnectionsPerHost — Default is 6. Reduce if your server has connection limits. Increase for CDN-backed APIs.
  • requestCachePolicy — Default is .useProtocolCachePolicy. Audit this if stale data is a problem.
  • multipathServiceType — Set to .handover for apps that need seamless WiFi-to-cellular transitions.

Caching: The Performance Multiplier

The fastest network request is one you never make. AI can audit your caching strategy:

Good Prompt — Caching Audit

/analyze-network

## URLSession configuration:
let config = URLSessionConfiguration.default
config.requestCachePolicy = .useProtocolCachePolicy
config.urlCache = URLCache(
    memoryCapacity: 10_485_760,   // 10 MB
    diskCapacity: 52_428_800,      // 50 MB
    diskPath: "network_cache"
)

## Server response headers for /api/v2/feed:
Cache-Control: no-store
Content-Type: application/json

## Question: The feed endpoint returns the same data for ~5 minutes
but the server sends no-store. We're making 12 requests/minute
to the same endpoint. What should we do?

Claude Code will identify that the server’s Cache-Control: no-store header prevents URLCache from caching, and suggest either: (1) asking the backend team to change to Cache-Control: max-age=300, or (2) implementing a client-side response cache with TTL if you can’t change the server.

Combine Networking Pitfalls

If your codebase still uses Combine for networking (pre-async/await), AI catches these patterns:

// Common Combine pitfall: retry without delay
URLSession.shared.dataTaskPublisher(for: request)
    .retry(3)                    // Retries IMMEDIATELY on failure
    .decode(type: Feed.self, decoder: JSONDecoder())
    .receive(on: DispatchQueue.main)
    .sink(receiveCompletion: { _ in }, receiveValue: { feed in
        self.items = feed.items  // Strong self capture — potential leak
    })
    .store(in: &cancellables)

AI identifies three issues: (1) .retry(3) fires immediately — add .delay for backoff, (2) strong self capture in sink — use [weak self], (3) receiveCompletion ignores errors — handle .failure case.

Migration tip: Ask Claude Code to convert Combine networking chains to async/await. Feed it the publisher chain and say /refactor to async/await with proper cancellation and error handling. It handles the sink→try await, retry→backoff loop, and [weak self]→structured concurrency conversions.

Exercise: Diagnose a Failing API Flow

Setup:

  1. Set up Proxyman (or Charles Proxy) to capture simulator traffic
  2. In your app, trigger a flow that makes at least 3 API calls (e.g., login → fetch profile → fetch feed)
  3. Introduce a failure: set the device to Airplane Mode after the first request succeeds

Steps:

  1. Capture the full request/response log in Proxyman (or use your os_log network logging)
  2. Export the session data (Proxyman: File > Export > HAR format)
  3. Paste the exported data into Claude Code with /analyze-network
  4. Review the AI analysis: Does it correctly identify which requests succeeded and failed?
  5. Apply the suggested fixes: proper error handling, retry with backoff, waitsForConnectivity
  6. Re-test: toggle Airplane Mode during the flow and verify graceful degradation

Bonus: Add URLSessionTaskMetrics logging and compare DNS, TLS, and total timing between WiFi and cellular connections.

Key Takeaways

  • Make your network talk — URLSessionTaskMetrics, os_log, or Proxyman. You can’t debug what you can’t see.
  • NSURLError codes are your first diagnostic clue — AI classifies them instantly
  • SSL pinning failures are silent killers — always pin to SPKI hash with backup pins
  • Use .task for view-scoped network calls — automatic cancellation when views disappear
  • Parallelize independent requests with async let — don’t pay serial round-trip costs
  • Audit URLSessionConfiguration — the defaults (60s timeout, 7-day resource timeout) are rarely what you want
  • Build the /analyze-network skill for bulk log analysis and configuration audits
