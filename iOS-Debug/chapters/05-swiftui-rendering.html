Chapter 5

SwiftUI Rendering Performance

Diagnose and fix excessive view re-evaluation using AI, _printChanges(), and the SwiftUI Instrument

In Chapter 4, we hunted memory leaks — objects that refuse to die. Now we’re hunting ghosts of a different kind: view bodies that re-evaluate for no good reason. A SwiftUI view that evaluates its body 47 times during a single scroll is the performance equivalent of repainting your house every time someone opens the front door.

SwiftUI’s declarative magic comes at a cost: if you don’t understand what triggers re-evaluation, you can accidentally create an app that’s doing 10x more work than necessary — and you won’t see a crash or an error. Just a subtle stutter that users feel but can’t articulate.

The SwiftUI Rendering Pipeline

Before we debug, we need the mental model. SwiftUI rendering has three distinct phases:

1. Evaluate
SwiftUI calls your view’s body property. This produces a new description of what the UI should look like.
→
2. Diff
SwiftUI compares the new description against the previous one. Only differences move forward.
→
3. Render
Only the changed parts get rendered to screen. This is cheap — Core Animation handles it.

Here’s the key insight: step 1 is what you control, and step 1 is where performance dies. If your body is called 50 times but nothing changed, steps 2 and 3 are fast (diff says “nothing new”), but you still paid the cost of evaluating body 50 times. And if body creates formatters, does math, or allocates objects — that cost adds up fast.

The Restaurant Analogy: Think of body as a waiter writing down the full order. The kitchen (renderer) only cooks what changed. But if the waiter rewrites the entire menu every time a customer asks for more water, you’ve got a staffing problem — even if the kitchen is efficient.

Your #1 Debugging Tool: Self._printChanges()

This one method will save you more SwiftUI debugging time than any other tool. Add it to any view’s body to see exactly what triggered the re-evaluation:

struct ProductRow: View {
    let product: Product
    @EnvironmentObject var cart: CartViewModel

    var body: some View {
        let _ = Self._printChanges()  // <-- Add this line
        HStack {
            Text(product.name)
            Spacer()
            Text(product.formattedPrice)
        }
    }
}

When the view re-evaluates, you’ll see output like this in the console:

Console Output — _printChanges() Diagnostics

ProductRow: @self changed.
ProductRow: @identity changed.
ProductRow: _cart changed.
ProductRow: @self, @identity, _cart changed.

AI Analysis

What each message means:

  • @self changed — The view’s value itself changed (a property of the struct differs from last evaluation). This is normal when parent rebuilds.
  • @identity changed — SwiftUI thinks this is a new view, not an update to the existing one. The view loses its state. This is almost always a bug.
  • _cart changed — The @EnvironmentObject named cart published a change. Every view reading cart re-evaluates — even if it only reads cart.itemCount and what changed was cart.promoCode.

The _cart changed line is the smoking gun. If CartViewModel uses ObservableObject, any @Published property change triggers all observers. With @Observable, only views reading the changed property re-evaluate.

Development only! _printChanges() is an undocumented internal API. It’s invaluable for debugging but must be removed before shipping. Never commit it to your main branch. Use a #if DEBUG guard or a Git pre-commit hook.

The SwiftUI Instrument (Xcode 26)

WWDC25 introduced a dedicated SwiftUI instrument in Instruments 26. This is a game-changer — no more guessing which views are hot. The instrument provides three tracking lanes:

🔴

Long View Body Updates

Flags body evaluations that take too long. Shows you exactly which view and how many milliseconds it consumed.

🟠

Long Representable Updates

Catches slow UIViewRepresentable and UIViewControllerRepresentable updates — the bridge between UIKit and SwiftUI.

🟡

Other Long Updates

All other SwiftUI work that’s taking too long: environment propagation, preference key aggregation, layout passes.

But the real star is the Cause & Effect Graph. This visualizes why a view updated: which state change caused which view body to re-evaluate. It turns “something triggered a re-render” into “changing cart.promoCode caused ProductRow to re-evaluate because it reads cart.”

Xcode AI + SwiftUI Instrument: In Xcode 26, you can ask Swift Assist to explain what the Cause & Effect Graph is showing. Select a slow view body update in the timeline, right-click, and choose “Explain with Swift Assist.” The AI reads the graph and tells you in plain English why the update happened and how to fix it.

The Biggest Win: @Observable vs ObservableObject

If you take one action from this chapter, migrate your view models from ObservableObject to @Observable. The performance difference is dramatic and the reason is simple: granularity.

Aspect ObservableObject (Legacy) @Observable (Modern)
Tracking Object-level: any @Published change notifies all observers Property-level: only views reading the changed property re-evaluate
List of 1000 items Changing item #42 re-evaluates all 1000 rows Changing item #42 re-evaluates only that row
View wrappers @StateObject, @ObservedObject, @EnvironmentObject @State, plain property, @Environment
Boilerplate @Published on every property None — all stored properties tracked automatically
Min deployment iOS 13+ iOS 17+

Before/After: ObservableObject → @Observable Migration

Before (@ObservedObject + ObservableObject)
List with 500 items, changing one item’s favorite status
View body evaluations: 501 (parent + all rows)
Frame time: 24ms (dropped frame on 120Hz)
After (@Observable + @State)
Same list, same change
View body evaluations: 1 (only the changed row)
Frame time: 1.2ms (smooth at 120Hz)

The 6 SwiftUI Performance Traps AI Catches

Feed your _printChanges() output or SwiftUI Instrument data to Claude Code, and it will spot these patterns instantly:

Trap 1: Creating Objects Inside body

Bad — DateFormatter Created Every Evaluation

var body: some View {
    let formatter = DateFormatter()       // Allocated EVERY body call
    formatter.dateStyle = .medium
    Text(formatter.string(from: date))
}

Good — Static Formatter

private static let formatter: DateFormatter = {
    let f = DateFormatter()
    f.dateStyle = .medium
    return f
}()

var body: some View {
    Text(Self.formatter.string(from: date))
}

Trap 2: Reading Too Many Properties from an Observable

Even with @Observable, if your view reads 5 properties, it re-evaluates when any of those 5 change. The fix: split into focused child views that each read only what they need.

Trap 3: Unstable View Identity

When _printChanges() shows @identity changed, your view is being destroyed and recreated instead of updated. Common cause: using .id(UUID()) or an unstable identifier in a ForEach.

Root Cause: Unstable Identity

Symptom: _printChanges() prints @identity changed on every parent rebuild. Animations break. @State resets unexpectedly.

Cause: The view’s identity changes every evaluation — usually a random .id() or a ForEach over items without stable IDs. SwiftUI treats it as a new view each time.

Fix: Use stable, unique identifiers. Model objects should have a persistent id property. Never use .id(UUID()) unless you intentionally want to reset the view.

Trap 4: Heavy Computation in body

Sorting, filtering, or mapping large collections inside body means that work runs on every re-evaluation. Move it to a computed property backed by cached state, or use .task for async computation.

Trap 5: Missing @State for Local State

Using a plain var for state that should be @State means SwiftUI can’t track it, leading to either missing updates or parent-triggered rebuilds resetting your local value.

Trap 6: EquatableView Neglect

For views with expensive bodies that receive complex data, conforming to Equatable and wrapping in EquatableView lets SwiftUI skip body evaluation entirely when inputs haven’t changed.

Lazy Stacks: Not a Silver Bullet

LazyVStack and LazyHStack only create views when they scroll into the visible area. But “lazy” doesn’t mean “free”:

  • Missing explicit id: Without stable IDs in ForEach, SwiftUI may recreate views instead of reusing them during scroll.
  • Heavy row views: Each row is still evaluated when it appears. If row body takes 8ms and 10 rows appear at once, that’s 80ms — a dropped frame.
  • Prefetching: LazyVStack in a ScrollView has no native prefetch API like UICollectionView. For image-heavy lists, load thumbnails in .task and show placeholders.
  • Pinned headers: Section headers inside lazy stacks can trigger unexpected re-evaluations of the entire section.
When to use List vs LazyVStack: List uses UITableView under the hood with cell reuse — it’s more memory-efficient for very long lists (10,000+ rows). LazyVStack in a ScrollView gives you more layout control but no cell reuse. For most apps, List is the better default.

Identity: The Hidden Performance Lever

SwiftUI uses identity to track views across updates. There are two kinds:

Type How It Works Performance Impact
Structural Position in the view tree. if/else branches create different identities. Switching branches destroys and recreates the view — state resets.
Explicit .id() modifier or ForEach with Identifiable items. Changing the ID destroys the old view and creates a new one. Stable IDs = stable views = smooth animations.

The performance rule: keep identity stable. Every time identity changes, SwiftUI throws away the view’s state, transitions, and animation context, then rebuilds from scratch. This is the most common cause of janky list scrolling and broken animations.

The Derived State Pattern

When a view only needs one computed value from a complex model, don’t read the model directly. Derive the value in the parent and pass it down:

Pattern: Derived State to Minimize Re-evaluation

// BEFORE: ProductRow reads entire cart
struct ProductRow: View {
    let product: Product
    @EnvironmentObject var cart: CartViewModel  // Re-evaluates on ANY cart change

    var body: some View {
        HStack {
            Text(product.name)
            if cart.contains(product) {     // Only needs this one boolean
                Image(systemName: "cart.fill")
            }
        }
    }
}

// AFTER: Parent derives the boolean, row is decoupled
struct ProductRow: View {
    let product: Product
    let isInCart: Bool                        // Simple value — no observation

    var body: some View {
        HStack {
            Text(product.name)
            if isInCart {
                Image(systemName: "cart.fill")
            }
        }
    }
}

// Parent (the only view that observes the cart):
ForEach(products) { product in
    ProductRow(
        product: product,
        isInCart: cart.contains(product)
    )
}

Now ProductRow only re-evaluates when isInCart actually changes for that product, not when any cart property changes.

When AI Gets SwiftUI Wrong

Common AI hallucinations for SwiftUI performance:
  • “Use .drawingGroup() to fix slow lists” — drawingGroup() rasterizes to a bitmap. It helps with complex drawing (gradients, masks) but does nothing for view body evaluation cost.
  • “Add .animation(nil) to prevent re-renders” — This disables animation, it doesn’t prevent body evaluation. The view still re-evaluates.
  • “Use @ObservedObject instead of @StateObject for better performance” — The opposite is often true. @ObservedObject doesn’t own the object, so it can be recreated. The observation cost is the same.
  • “Wrap everything in EquatableView” — Only helps if the equality check is cheaper than the body evaluation. For simple views, the overhead of Equatable conformance isn’t worth it.

Always verify with _printChanges() after applying any AI suggestion. If the evaluation count doesn’t drop, the suggestion didn’t work.

Key Takeaways

  • _printChanges() is your first tool — always start here before reaching for Instruments
  • The SwiftUI Instrument in Xcode 26 shows Cause & Effect graphs — use Swift Assist to explain them
  • Migrate from ObservableObject to @Observable for property-level tracking — the single biggest performance win
  • Keep view identity stable: no .id(UUID()), no unstable ForEach identifiers
  • Don’t create objects in body — formatters, date components, sort operations all belong outside
  • Use the derived state pattern to decouple child views from broad observables
  • Always verify AI suggestions with _printChanges() — if the count doesn’t drop, the fix didn’t work
