Chapter 1

Welcome to the Other Side

What this course covers, why it matters, and how each chapter builds toward your first PR

So, You're About to Learn Some Tech Skills

Take a deep breath. You're not switching careers. You're not going to become a developer. You're just picking up a handful of tools that will make your job as a Product Manager significantly easier — and a lot more fun.

This course exists because your role is changing. More and more, PMs are expected to work closer to the code — not writing code, but describing what should be built, in a structured way that developers and AI tools can act on. That's what you'll learn here.

By the end of this course, you'll be able to take a ticket from your backlog, write clear behavioral specs for it, and submit those specs as a Pull Request on BitBucket — all by yourself, without needing a developer to hold your hand.

If words like "Pull Request," "BitBucket," or "specs" mean absolutely nothing to you right now — perfect. That's exactly where you should be. We'll explain every single one of them, starting from zero.

Why Should a PM Learn This?

You already know how to define what a product should do. That's your superpower. The gap isn't in your thinking — it's in the delivery mechanism. Right now, your ideas travel from your brain to a Jira ticket to a conversation to (hopefully) the right code. A lot gets lost along the way.

With the workflow you'll learn here, you'll describe features in a structured format that both humans and AI can understand. You'll submit that description directly into the codebase. Developers pick it up from there. Less miscommunication, fewer surprises, faster delivery.

Think of it like this: right now, you're describing what you want verbally over the phone, and someone else is writing it down. This course teaches you to type it yourself — in a shared Google Doc that everyone can see and comment on. Same ideas, better delivery channel.

What You'll Learn (and What You Won't)

Let's set expectations clearly. Here's what this course covers:

14 Chapters
5 Tools
0 Lines of code to write

That last number is important. You will never write a single line of programming code in this course. The AI will do that part. Your job is to describe what the software should do, not how to build it.

You WILL learn:

  • How to use the Terminal for a few basic commands (just six, we promise)
  • How to navigate Windsurf, an AI-powered workspace
  • How to track changes with Git — entirely through a visual interface
  • How to collaborate on BitBucket Server using Pull Requests
  • How to describe features with OpenSpec, a spec-driven framework

You will NOT learn:

  • Programming (JavaScript, Python, or anything else)
  • Databases or system architecture
  • DevOps, deployment, or server management
  • Anything that requires a computer science degree

This isn't a developer bootcamp. It's a focused guide to the specific tools you need for your workflow.

Meet the Five Tools

Throughout this course, you'll get comfortable with five tools. Here's a quick preview of each one. Don't worry about remembering everything — each tool gets its own dedicated chapter (or several).

T
Terminal
A text-based way to talk to your Mac. You'll learn six commands — just enough to navigate folders and create files. Think of it as Finder, but you type instead of click.
W
Windsurf
An IDE (think: fancy text editor) with a built-in AI assistant called Cascade. It's where you'll open projects, edit files, and talk to the AI — all in one window.
G
Git
A system that tracks every change ever made to a project. Like "Track Changes" in Google Docs, but far more powerful. You'll use it entirely through Windsurf's visual interface.
B
BitBucket Server
A website hosted by your company where all the code lives. Think of it as Google Drive for code. You'll use it to share your work and submit it for review.
O
OpenSpec
A framework for describing what to build using structured behavioral scenarios. You write the "recipe" (what the dish should taste like); the developer does the cooking.

How This Course Works

Each chapter builds on the one before it. We start with the basics (what's a terminal?) and gradually work our way up to the full workflow (ticket to Pull Request). You don't need to memorize everything in one sitting — you can always come back to earlier chapters.

Along the way, you'll see special boxes like this one:

Try-it boxes are your "hands on the keyboard" moments. When you see one, open your Mac and follow along. You learn by doing, not just reading.

Screenshot boxes describe what you should see on your screen at a particular moment. Use them to check that you're on the right track.

Real-world boxes connect a technical concept to something you already know — like Google Docs, email, or filing cabinets. If a concept isn't clicking, look for this box.

Don't Panic boxes appear whenever something might look intimidating. They're a friendly reminder that what you're seeing is normal and manageable.

Tip boxes offer shortcuts, best practices, or things that'll save you time.

Warning boxes flag things to be careful about. Not scary — just "heads up."

The End Goal

By Chapter 12, you'll be able to execute this entire workflow on your own:

Receive ticket → Open project → Create branch → Write specs → Commit → Push → Create PR

If that looks like a foreign language right now, that's completely fine. By the time you reach Chapter 12, every single step will feel natural.

Here's the Deal

You won't become a developer. You'll become a PM who can speak the developer's language just enough to deliver specs directly into the codebase. That's a superpower most PMs don't have — and you're about to get it.

Your Practice Project: A Google Notes App

Learning tools without a real project is like learning to drive without a car. So in Chapters 9 and 10, you'll use OpenSpec to describe a small app — a simplified version of Google Keep (that sticky-notes app from Google).

You won't write the code for it. You'll describe what it should do:

  • A grid of colorful note cards
  • Creating and editing notes
  • A light/dark theme toggle
  • Random colors for new notes

You'll write these features as behavioral specs ("WHEN the user clicks the New Note button, THEN a blank note card appears in the grid"). The AI will handle the actual implementation. Your job is to describe what and why — the developer (or AI) figures out the how.

Think of it like ordering at a restaurant. You tell the waiter "I'd like the salmon, medium-rare, with the lemon sauce on the side." You don't go into the kitchen and cook it yourself. The specs are your order — clear, specific, and understandable by the person who'll prepare it.

What You Need Before Starting

Here's your pre-course checklist:

  • A Mac (this course is Mac-only)
  • An internet connection
  • Access to your company's BitBucket Server (ask your team lead if unsure)
  • Curiosity and a willingness to try new things

That's it. No prior technical experience needed. No software to pre-install (we'll walk you through installations in the relevant chapters).

One Last Thing Before We Start

You're going to see unfamiliar screens. You're going to encounter words you've never heard before. You might even see an error message or two. All of this is completely normal.

Developers see error messages every single day. It's not a sign that something is broken beyond repair — it's the computer telling you what it needs. You'll get comfortable with that feeling faster than you think.

Remember

Every developer you know was once a beginner who didn't know what a terminal was. The only difference between where you are now and where you'll be at Chapter 14 is 14 short chapters of practice. Let's get started.

In the next chapter, we'll open the Terminal on your Mac and learn six simple commands. It's going to be easier than you think — and by the end of it, you'll feel like you've unlocked a secret door.
