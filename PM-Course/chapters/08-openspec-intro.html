Chapter 8

OpenSpec: What It Is and Why You Care

The framework that turns your product knowledge into structured specs that AI and developers can act on

The Missing Bridge

You now have all the tools: Terminal, Windsurf, Git, BitBucket, Pull Requests. You can track changes, push code, and submit work for review. But what are you going to submit?

That's where OpenSpec comes in. It's a framework that gives you a structured way to describe what a feature should do — in plain language, using behavioral scenarios that both developers and AI can understand.

This is a conceptual chapter. No hands-on yet — that starts in Chapter 9. Right now, let's understand what OpenSpec is, why it exists, and what your role as a PM looks like.

The Problem OpenSpec Solves

Here's a scenario you probably know too well:

  1. You write a ticket: "Add dark mode to the app"
  2. A developer reads it and starts building
  3. Two weeks later, the result doesn't match what you had in mind
  4. Back and forth begins: "That's not what I meant" / "But the ticket said…"

The problem isn't that you wrote a bad ticket or the developer did bad work. The problem is the format. A ticket describes intent loosely. There's room for interpretation, and interpretation leads to divergence.

Imagine ordering a cake by saying "Make it nice and chocolate-y." You'll get a cake, but will it have three layers or one? Frosting or ganache? Will it be what you pictured? Now imagine handing over a recipe card: "Three layers, dark chocolate sponge, chocolate ganache between layers, topped with cocoa powder." Same cake — but now the baker knows exactly what you want.

OpenSpec is that recipe card. It gives you a structured format to describe features so precisely that there's no room for "that's not what I meant."

OpenSpec in One Sentence

The Definition

OpenSpec is a framework that organizes HOW you describe WHAT to build — using structured artifacts that guide the journey from idea to implementation.

The key idea is Spec-Driven Development: define the behavior first, then implement. Instead of jumping straight into code, you describe what the software should do in plain language. Then the developer (or AI) builds exactly that.

The Four Artifacts

Every feature in OpenSpec produces up to four files. Each one has a specific purpose:

proposal.md WHY and WHAT — the business case
specs/ WHAT the system does — behavioral scenarios
design.md HOW to implement — technical approach
tasks.md Implementation checklist — step by step

Let's look at each one:

1. proposal.md — The "Why" and "What"

This is the business case. It answers: Why are we building this? and What should it do at a high level? Think of it as the executive summary you'd write for any project — goals, scope, and context.

2. specs/ — The Behavioral Scenarios

This is the heart of OpenSpec. The specs/ folder contains files that describe what the system does using WHEN/THEN scenarios. These are plain-language descriptions of behavior — no code, no technical jargon.

Example WHEN/THEN Scenario

WHEN the user clicks the "New Note" button THEN a blank note card appears in the notes grid AND the cursor is placed in the note's title field

Plain language. Describes what the user sees, not how the code works.

3. design.md — The "How" (Developer-Owned)

This is the technical plan: which components to build, what architecture to use, how to structure the code. You don't write this. The developer does.

4. tasks.md — The Implementation Checklist (Developer-Owned)

A checklist of specific implementation steps that the developer (or AI) works through. You don't write this either. It's generated from the design.

Your Role as PM: Phase 1

Here's the exciting part: your team splits the work into two phases. As a PM, you own Phase 1.

Phase 1: PM → Handoff → Phase 2: Dev
PhaseWhoWhat They CreatePR
Phase 1 You (PM) proposal.md + specs/ PR #1
Phase 2 Developer design.md + tasks.md + code PR #2

You describe the what (proposal + specs). The developer handles the how (design + tasks + implementation). Clean separation, clear ownership.

Think of building a house. You're the client who says: "I want a three-bedroom house with an open kitchen and a south-facing garden" (the specs). The architect then creates blueprints (the design) and the contractor follows a build plan (the tasks). You don't need to know about load-bearing walls or plumbing — that's their expertise. But without your clear description, they'd be guessing.

What Good Specs Look Like

Specs are behavioral, not technical. They describe what a user sees and does — not how the code works under the hood.

Good Spec (Behavioral)

WHEN the user requests their profile THEN the system displays their name, email, and avatar image

Bad Spec (Too Technical)

WHEN the GET /api/users/:id endpoint is called THEN return a JSON object with name, email, and avatar_url fields

The good spec describes what the user experiences. The bad spec describes what the code does. You should always write the first kind. The developer figures out the second kind from your behavioral description.

A good rule of thumb: if your spec uses words that a QA tester couldn't understand, it's too technical. Specs should be testable by someone who doesn't read code.

How AI Helps

Here's where it gets really powerful: you don't write specs from scratch. You describe the feature to the AI (Cascade, in Windsurf), and it generates the proposal and specs for you.

Your job is then to review and refine. Does the AI's interpretation match what you had in mind? Are there scenarios it missed? Edge cases you know about from user research? This is where your PM expertise shines — the AI generates the structure, you bring the product judgment.

You're not expected to write perfect WHEN/THEN scenarios from memory. The AI does the heavy lifting. Your role is to describe the feature in your own words, review what the AI produces, and refine it until it's right. Think of it as editing, not writing.

OpenSpec Commands: A Preview

In Chapters 9–11, you'll use these commands. For now, just scan the list so the names aren't brand new when you see them:

CommandWhat It DoesWho Uses It
openspec init Initialize OpenSpec in a project (one-time setup) Anyone
openspec new change <name> Start a new feature/change directory PM or Dev
/opsx:propose Generate proposal + specs (AI creates the artifacts) PM
/opsx:apply Implement the tasks from specs Dev
/opsx:archive Archive working data; move behavioral specs to specs/ folder Dev
openspec status Check what's ready and what needs attention Anyone
openspec validate Validate that specs are well-formed Anyone
openspec list List all changes and specs Anyone

Notice that most of the PM-facing commands start with /opsx: — these are Cascade commands you type in the AI chat panel. The openspec commands are typed in the terminal. You'll learn all of them hands-on.

The Handoff: From Specs to Implementation

Once you've created your proposal and specs and submitted them as PR #1, the process looks like this:

  1. Your team reviews the specs in the PR
  2. After approval, the PR gets merged
  3. A developer picks up the change
  4. They create design.md and tasks.md
  5. They implement the feature using /opsx:apply
  6. They archive the change using /opsx:archive
  7. They submit everything as PR #2

After archiving, something important happens: the working files (proposal, design, tasks) move to an archive/ folder, and only the behavioral WHEN/THEN scenarios survive in the specs/ folder. The specs become the project's living documentation — a growing record of what the system does.

Think of a cookbook that grows over time. Each time you cook a new recipe (feature), the recipe card (specs) gets filed permanently. The shopping list and prep notes (proposal, design, tasks) get archived — useful for reference, but the recipe is what matters going forward.

You Now Know the Framework

  • OpenSpec is a framework for describing what to build using structured artifacts
  • Four artifacts: proposal.md, specs/, design.md, tasks.md
  • You own Phase 1: proposal + specs (PR #1)
  • Developers own Phase 2: design + tasks + code (PR #2)
  • Specs use WHEN/THEN format — plain language, behavioral, non-technical
  • AI generates the artifacts; you review and refine
