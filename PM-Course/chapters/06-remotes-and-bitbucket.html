Chapter 6

Remotes, BitBucket, and Working with Others

How teams share code, what forks are, and why Pull Requests exist

Beyond Your Mac

In Chapter 5, you learned to track changes on your own computer. You created commits, saw the U/M/D badges, and built up a history. But all of that happened locally — on your Mac, visible only to you.

In the real world, you're working with a team. Everyone needs to share their changes, review each other's work, and make sure nobody accidentally breaks the project. This chapter explains how that works.

This is a conceptual chapter — no hands-on work, just understanding the big picture. In Chapter 7, you'll do every step yourself.

Local vs. Remote

These two words come up constantly, so let's nail them down:

TermWhereWho Can See It
Local Your Mac Only you
Remote A server (BitBucket) Anyone with access

When you commit in Windsurf, the commit is local. Nobody else knows about it until you push it to a remote server. The remote is the shared space where the team's code lives.

Think of it like writing in a personal notebook (local) versus posting on a shared whiteboard (remote). Your notebook is yours — you can scribble, erase, and rewrite freely. The whiteboard is where you put things when you're ready for the team to see them.

What Is BitBucket Server?

BitBucket Server is a website that hosts Git repositories. It's where the team's code lives, where you browse files, create pull requests, and review each other's work.

The key word here is Server. Your company runs its own BitBucket installation on company-owned hardware. It's not a public website — it's an internal tool, accessible only through your company's network.

You access BitBucket Server through a URL in your browser, something like https://bitbucket.yourcompany.com. Ask your team lead for the exact URL if you don't have it. You log in with your company credentials.

Why not just use a public service?

Companies host their own BitBucket for three reasons:

  • Security — the code never leaves company-controlled servers
  • Control — the company decides who can see and edit what
  • Compliance — certain industries require that code stays on-premises

From your perspective as a user, it works the same as any website. You open it in a browser, log in, click around. The fact that it's self-hosted is mostly invisible to you.

What You'll See on BitBucket

When you log into BitBucket Server and open a repository, you'll see:

  • Files — the project's folders and files, like Finder
  • Commits — the history of changes (same as what you see in Windsurf)
  • Branches — a list of all active branches
  • Pull Requests — open, merged, and declined PRs
  • Settings — permissions and configuration (you won't need this)

It's a web-based view of the same Git repository you work with in Windsurf. Think of BitBucket as the "online view" and Windsurf as the "local editing view" — same project, two ways to look at it.

Why You Can't Edit the Main Repository Directly

The team's main repository is the source of truth. It contains the official, reviewed, approved version of the project. If anyone could edit it directly, you'd have chaos — untested changes, conflicting edits, broken features.

Imagine a company's master contract template stored in a locked filing cabinet. You don't scribble edits directly on the master copy. Instead, you photocopy it, make your edits on the copy, and submit the copy for legal review. Only after approval do the edits get incorporated into the master. The locked cabinet protects the master from accidental (or unauthorized) changes.

That's exactly how it works in Git. The main repository is "locked" — you can read it, but you can't push changes directly to it. Instead, you work through a fork.

Forks: Your Personal Copy

A fork is your own copy of the main repository, hosted on BitBucket. It's not on your Mac — it's on the server, but it belongs to you.

When you fork a repository:

  • BitBucket creates a complete copy of the main repo under your account
  • You can do whatever you want with your fork — create branches, push commits, experiment
  • Your changes don't affect the main repo until you explicitly ask (via a Pull Request)
  • You only need to fork once per repository — your fork stays there for future work

Forking sounds complicated, but on BitBucket it's literally one click. You open the main repository in your browser, click "Fork," and BitBucket handles everything. You'll do this yourself in Chapter 7.

The Fork Workflow: Step by Step

Here's the complete flow you'll follow every time you need to make changes. Read through it now; you'll do every step hands-on in Chapter 7.

1. Fork → 2. Clone → 3. Branch → 4. Commit → 5. Push → 6. Pull Request

Let's walk through each step:

Step 1: Fork the repository on BitBucket

Go to the main repository on BitBucket and click "Fork." This creates your personal copy on BitBucket. You only do this once per project.

Step 2: Clone your fork to your Mac

Copy your fork's URL from BitBucket, then use Windsurf to clone it (download it) to your Mac. Now you have a local copy to work with. You only do this once per project too.

Step 3: Create a branch

Before making changes, create a new branch in Windsurf. This isolates your work so you can always get back to a clean state. You create a new branch for each task.

Step 4: Make changes and commit

This is the cycle from Chapter 5: edit files, stage, write a message, commit. Do this as many times as you need for your task.

Step 5: Push to your fork

Upload your branch and commits to your fork on BitBucket. This makes your work visible on the server — but only on your fork, not the main repo.

Step 6: Create a Pull Request

On BitBucket, create a PR from your fork's branch to the main repository. This is you saying: "Hey team, please review these changes and merge them if they look good."

Steps 1 and 2 (fork and clone) are one-time setup. After that, your daily work is steps 3–6: branch, commit, push, PR. It becomes routine quickly.

Pull Requests: The Review Process

A Pull Request (PR) is the most important collaboration tool in the workflow. It's where your work meets the team's feedback.

When you create a PR, here's what happens:

  1. You fill in the details: a title describing the change, a description explaining what you did and why, and you add reviewers (team members who should look at it).
  2. Reviewers get notified: they see your PR on BitBucket and can browse the changes you made — what was added, edited, or removed.
  3. Reviewers leave comments: they might say "Looks good!" or "Can you clarify this scenario?" or "I think we should also cover the edge case where…"
  4. You respond and update: if there's feedback, you make changes on your branch, commit, and push again. The PR updates automatically.
  5. Reviewers approve: once everyone's happy, they click "Approve."
  6. The PR gets merged: a maintainer merges your changes into the main repository. Your work is now part of the official project.

It's like submitting a draft proposal via email. You write it, send it to your manager and a few colleagues, they reply with suggestions, you incorporate the feedback and resend, and eventually everyone signs off. A PR is that exact process, but structured and tracked in one place instead of scattered across email threads.

What Happens After a Merge

Once your PR is merged:

  • Your changes are now in the main repository — part of the official project
  • The branch you worked on has served its purpose — it can be deleted
  • Your fork still exists on BitBucket for your next task
  • Next time you start a new task, you'll create a fresh branch (step 3) and repeat

The main repo grows one PR at a time. Every feature, every bug fix, every set of specs enters through a Pull Request. It's the team's quality gate.

The Big Picture: Where Everything Lives

Main Repo The team's official project on BitBucket — read-only for you
Your Fork Your copy on BitBucket — push branches here
Your Mac Your local clone — where you edit and commit

Changes flow in one direction: Your Mac → Your Fork → Main Repo. You edit locally, push to your fork, and create a PR to the main repo. That's the entire flow.

The Photocopier Analogy

Forks and PRs in One Story

The team's master document sits in a locked glass case (main repo). You walk up to the photocopier and make a complete copy (fork). You take the copy to your desk (clone).

You start a new section on a fresh sheet (branch). You write your edits and staple the pages together (commits). You bring the edited copy back to the office and put it in the "for review" tray (push + PR).

Your colleague picks it up, reads it, writes "Approved" in the margin (review). The office manager takes the approved pages and adds them to the master document in the glass case (merge).

Common Questions

"Do I need to fork every time I start a new task?"

No. You fork once. After that, your fork stays on BitBucket. For each new task, you just create a new branch on your existing fork.

"What if someone else changes the main repo while I'm working?"

That's normal and expected. When you create a PR, BitBucket will tell you if there are conflicts (rare for spec files). We'll cover how to handle that in Chapter 13 (Troubleshooting).

"Can I see other people's Pull Requests?"

Yes! On BitBucket, go to the main repository and click "Pull Requests." You can browse everyone's PRs — open, merged, and declined. It's a great way to see what the team is working on.

"What if my PR gets declined?"

It happens. Usually the reviewer explains why. You can update your work and either reopen the PR or create a new one. A declined PR isn't a failure — it's feedback.

This chapter introduced a lot of concepts. If it feels abstract right now, that's completely normal. Everything clicks into place when you do it yourself — and that's exactly what the next chapter is for.
