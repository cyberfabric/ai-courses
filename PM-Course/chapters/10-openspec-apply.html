Chapter 10

Hands-On: Implementing, Archiving, and Adding Features

Apply your specs, archive the change, then add a second feature and watch specs evolve

Completing the Cycle

In Chapter 9, you created a proposal and specs for a Notes app. That was Phase 1 — the PM's contribution. Now you'll see what happens next: the implementation (Phase 2) and the archiving process.

In a real workflow, Phase 2 is the developer's job. But for learning purposes, you'll run the commands yourself so you understand the full cycle. Then you'll add a second feature and see something powerful: how OpenSpec reads your existing specs to keep new features consistent.

You're about to see code being generated. Don't worry about understanding the code itself — that's the developer's domain. Focus on the process: what commands you run, what happens to the files, and how specs evolve over time.

Quick Recap: Where You Left Off

Your project currently has: 

openspec/
  changes/
    notes-app-base/
      proposal.md       ← Your business case
      specs/             ← Your WHEN/THEN scenarios
  specs/                 ← Empty (no archived specs yet)
  archive/               ← Empty (nothing archived yet)

The proposal and specs are committed to Git. In a real team, this would have been submitted as PR #1 and merged. Let's continue from here.

Running /opsx:apply — Implementing the Features

This is the command that turns specs into working code. Open Cascade and type: 

/opsx:apply

Cascade reads your specs and starts generating code. You'll see it creating files — HTML, CSS, JavaScript — right in your project. It may also create or update a design.md and tasks.md inside your change directory.

The process takes a minute or two. Let it finish.

When it's done, your project will have actual working files alongside the OpenSpec directory. The AI built the app based on your behavioral specs.

Remember the restaurant analogy? You placed your order (the specs) in Chapter 9. The kitchen (AI) just prepared the dish. You don't need to understand every ingredient or cooking technique — you just need to verify that what arrived matches what you ordered.

What you don't need to understand

The generated code files (HTML, CSS, JavaScript) are the developer's territory. You don't need to read them, understand them, or worry about them. Your contribution was the specs. The code is the developer's (or AI's) response to those specs.

If you're curious, you can open the generated HTML file in your browser to see the Notes app in action. But this is optional — in your daily workflow, the developer handles the implementation review.

Committing the Implementation

After /opsx:apply finishes, commit everything:

  1. Open Source Control
  2. Stage all changes (click + next to "Changes")
  3. Write a message: Implement notes app from specs
  4. Click Commit

This captures the implementation as a snapshot. In a real workflow, this would be part of the developer's PR #2.

Running /opsx:archive — Finalizing the Change

The feature is implemented. Now it's time to archive the change. This is an important step that keeps your project organized. In Cascade, type: 

/opsx:archive

After archiving, look at the File Explorer. The structure has changed:

  • The changes/notes-app-base/ directory's working files (proposal, design, tasks) have moved to archive/
  • The behavioral WHEN/THEN scenarios from your specs have moved to the openspec/specs/ folder

What just happened?

Archiving does two things:

WhatWhere It GoesWhy
Working data (proposal, design, tasks) archive/ Preserved for reference, but no longer active
Behavioral scenarios (WHEN/THEN) specs/ Becomes the project's living documentation

Think of it like finishing a construction project. The blueprints and work orders go into the filing cabinet (archive). But the inspection certificate — the document that says "this building does X, Y, and Z" — gets posted permanently (specs). Future projects need to know what the building can do, not how it was built.

Commit after archiving:

  1. Stage all changes
  2. Message: Archive notes-app-base change
  3. Commit

The specs/ Folder: Your Project's Memory

Open openspec/specs/ in the File Explorer. You should now see your behavioral scenarios filed here. These are the surviving WHEN/THEN scenarios from your first feature.

This folder is special. It's the single source of truth for what your app does. Every time a feature is archived, its behavioral specs accumulate here. Over time, this folder becomes a complete description of the entire system.

Why This Matters

When you (or the AI) create specs for a new feature, OpenSpec reads the existing specs in this folder. This means new features are always aware of existing behavior. No contradictions, no accidentally breaking old features with new ones.

Adding Feature #2: Random Colors

Let's see this in action. You'll add a second feature to the Notes app: random background colors for note cards.

Create a new change

In the integrated terminal:

openspec new change random-note-colors

Generate the proposal and specs

In Cascade:

/opsx:propose

Describe the feature:

What to Tell the AI

Add random background colors to note cards. When a new note is created, it should be assigned a random pastel color from a predefined set (light yellow, light blue, light green, light pink, light purple). The color should persist — it doesn't change after creation.

Watch what happens: Cascade generates new specs for the color feature. But this time, it's aware of the existing notes specs. The new scenarios reference and build on the existing behavior instead of contradicting it.

Review the new specs. You might see scenarios like:

Example: Color Feature Specs

WHEN a new note is created THEN the note card is assigned a random background color from the set: light yellow, light blue, light green, light pink, light purple WHEN a note card is displayed THEN the card's background reflects its assigned color WHEN a note's content is edited THEN the card's background color does not change

Notice how these scenarios integrate with the existing note creation behavior. The AI didn't redefine what "creating a note" means — it added the color aspect on top of the existing behavior.

Apply and Archive the Second Feature

The cycle is the same:

Propose → Review → Apply → Archive → Commit
  1. Run /opsx:apply — the AI implements the color feature
  2. Commit: Implement random note colors
  3. Run /opsx:archive — archive the change
  4. Commit: Archive random-note-colors change

Specs Evolve: The Accumulation Effect

Open openspec/specs/ again. It now contains scenarios from both features — the original notes behavior and the new color behavior. 

openspec/
  specs/
    note-management/
      spec.md          ← Scenarios for creating, editing, deleting notes
    theme/
      spec.md          ← Scenarios for light/dark toggle
    note-colors/
      spec.md          ← Scenarios for random card colors (NEW)
  archive/
    notes-app-base/    ← First feature's working files
    random-note-colors/ ← Second feature's working files

The specs folder is like a living policy manual for your app. Each time you add a feature, the new policies (scenarios) get added to the manual. Anyone — a new PM, a new developer, or the AI — can open this folder and understand everything the app does, without reading a single line of code.

The Key Insight

Specs accumulate. They don't get replaced — they grow. Each feature adds its behavioral scenarios to the shared pool. When the AI creates specs for feature #3, it reads features #1 and #2. When it creates feature #10, it reads all nine that came before. This is how OpenSpec keeps large projects consistent over time.

Your mission: Add a third feature to the Notes app on your own.

Pick one of these ideas (or invent your own):

  • Pin notes — pinned notes always appear at the top of the grid
  • Search notes — a search bar that filters notes by title or content
  • Note timestamps — each note shows when it was last edited
  1. Create a new change: openspec new change <your-feature-name>
  2. Run /opsx:propose and describe your feature
  3. Review the specs — do they reference existing behavior correctly?
  4. Run /opsx:apply to implement
  5. Run /opsx:archive to finalize
  6. Commit after each major step
  7. Check openspec/specs/ — it should now have three features' worth of scenarios

What You Just Learned

  • /opsx:apply turns specs into working code
  • /opsx:archive moves working files to archive, keeps behavioral specs
  • The specs/ folder is the single source of truth
  • AI reads existing specs when creating new features
  • Specs accumulate over time, building a living documentation
  • The cycle repeats: propose → review → apply → archive → commit
