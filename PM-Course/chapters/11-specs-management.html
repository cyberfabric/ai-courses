Chapter 11

Managing Your Specs Like a Pro

Naming conventions, organizing changes, validating specs, and keeping your project tidy

From Practice to Professional

In Chapters 9 and 10, you built specs for a Google Notes app and saw the full cycle: propose, review, apply, archive. That was practice with made-up feature names like notes-app-base.

In the real world, your team has conventions. Change directories have ticket numbers. Multiple features happen simultaneously. Specs need to stay clean and non-technical. This chapter covers the day-to-day habits that keep everything organized.

Ticket-Prefixed Change Directories

In practice, your change directories won't be called notes-app-base. They'll include the ticket number from your project tracker (Jira, Linear, or whatever your team uses). 

openspec new change mob123-add-dark-mode

This creates: 

openspec/changes/mob123-add-dark-mode/

Why ticket prefixes matter

Ticket prefixes create a chain of traceability:

Ticket MOB-123 → Change mob123-add-dark-mode → Branch mob123-add-dark-mode → PR #42

Anyone can look at a Pull Request, see the ticket number, and trace it back to the original requirement. Anyone can look at a ticket and find the corresponding specs, branch, and PR. Everything connects.

It's like putting an invoice number on every document related to a purchase order. The PO, the invoice, the delivery receipt, and the payment confirmation all share the same number. If someone asks about invoice #4567, you can instantly find everything related to it.

Change names must start with a letter, not a number. So use mob123-feature (good) instead of 123-feature (won't work). The ticket prefix convention mob123- handles this naturally.

Multiple Changes at the Same Time

In real projects, you'll often be working on more than one feature at a time. Each feature gets its own change directory: 

openspec/changes/
  mob123-add-dark-mode/
    proposal.md
    specs/
  mob456-note-categories/
    proposal.md
    specs/
  mob789-search-feature/
    proposal.md
    specs/

Each change is independent. You can propose specs for MOB-456 while MOB-123 is still being implemented by a developer. There's no conflict — each lives in its own directory.

Remember to create a separate Git branch for each change too. Branch mob123-add-dark-mode gets the dark mode specs. Branch mob456-note-categories gets the categories specs. One branch per change, one PR per change.

The specs/ Folder as Single Source of Truth

You saw this in Chapter 10, but it's worth emphasizing because it's the most important concept in day-to-day spec management.

The openspec/specs/ folder contains the accumulated behavioral knowledge of your entire project. After each feature is archived, its WHEN/THEN scenarios move here. Over time, this folder grows into a complete description of what the system does.

changes/ Active work in progress — temporary
specs/ Permanent behavioral truth — grows over time
archive/ Completed work for reference — historical

When AI creates specs for a new feature, it reads everything in specs/. This ensures new features are consistent with existing ones. The bigger the specs folder gets, the smarter the AI becomes about your project.

Archiving in Detail

Let's look at exactly what happens when you archive a change, since this is the mechanism that keeps everything organized.

Before archive:

openspec/
  changes/
    mob123-add-dark-mode/
      proposal.md          ← Business case
      specs/
        dark-mode/
          spec.md          ← WHEN/THEN scenarios
      design.md            ← Technical approach
      tasks.md             ← Implementation checklist
  specs/
    note-management/
      spec.md              ← Existing specs from previous features
  archive/                 ← Previous archived changes

After /opsx:archive:

openspec/
  changes/                 ← mob123 directory is gone
  specs/
    note-management/
      spec.md              ← Still here (untouched)
    dark-mode/
      spec.md              ← MOVED here from the change
  archive/
    mob123-add-dark-mode/  ← proposal, design, tasks archived here

The pattern is simple:

  • Behavioral specs (WHEN/THEN) → move to specs/ (permanent)
  • Everything else (proposal, design, tasks) → move to archive/ (reference)
  • The change directory → removed from changes/

Think of it like finishing a legal case. The court ruling (behavioral specs) goes into the official case law database where it affects future decisions. The case file (proposal, arguments, evidence) goes into the archive room. The lawyer's desk (change directory) is cleared for the next case.

Three Commands for Day-to-Day Management

OpenSpec has three utility commands you'll use regularly to check on the state of your project. Run these in the integrated terminal (Ctrl + `).

openspec status

Shows what needs attention: which changes are in progress, which are ready to archive, which need specs, etc. Think of it as a dashboard.

You'll see output listing each active change and its state. If everything is clean (no active changes), it'll tell you that too.

openspec list

Shows an overview of all changes and specs in the project — both active and archived. Use this when you want to see the big picture.

openspec validate

Checks that your specs are well-formed. It looks for common issues: missing scenarios, inconsistent formatting, or specs that don't follow the configured rules.

Run openspec validate before committing your specs. It catches problems early, before your PR reviewers do. Think of it like spell-check for specs.

CommandWhen to Use ItAnalogy
openspec status Start of the day, before committing Checking your email inbox
openspec list When you need the big picture Looking at the project board
openspec validate Before committing specs Running spell-check

What Good Specs Look Like vs. Bad Specs

Your team's config.yaml contains rules that enforce non-technical, QA-friendly specs. But understanding the principles helps you write better specs even before validation catches issues.

Good: Behavioral & Testable

WHEN the user toggles dark mode THEN the background changes to a dark color AND all text changes to a light color AND the toggle icon changes to a sun

Bad: Technical Implementation

WHEN dark mode is toggled THEN set CSS variable --bg-color to #1a1a2e AND add class .dark-theme to the body element AND update localStorage with theme preference

Good: User-Focused

WHEN the user has not created any notes yet THEN the main screen shows a friendly message "No notes yet — click Add Note to get started"

Bad: Vague

WHEN there are no notes THEN show empty state

Rules of thumb for good specs:

  • Describe what the user sees, not what the code does
  • Be specific — "a friendly message" is vague; include the actual text
  • No CSS, HTML, or JavaScript references in specs
  • No database, API, or backend language
  • A QA tester should be able to verify it without reading code

You don't need to get specs perfect on the first try. The AI generates good drafts, openspec validate catches structural issues, and your PR reviewers catch anything else. It's a team effort.

Your mission: Run the management commands on your Notes app project.

  1. Open your Notes app project in Windsurf
  2. Open the integrated terminal (Ctrl + `)
  3. Run openspec status — what does it report?
  4. Run openspec list — can you see all your changes and specs?
  5. Run openspec validate — are your specs well-formed?

If validate reports any issues, read the messages carefully. They usually tell you exactly what to fix (e.g., "spec contains technical language" or "missing WHEN/THEN format").

Keeping Your Project Clean

As you work with OpenSpec daily, follow these habits:

  • Always use ticket-prefixed change names: mob123-feature-name
  • One change directory per ticket, one branch per change, one PR per branch
  • Run openspec validate before committing
  • Archive changes promptly after implementation — don't leave stale changes lingering
  • Review the specs/ folder periodically — it should always reflect reality
  • Keep specs behavioral: WHEN/THEN, no technical details
