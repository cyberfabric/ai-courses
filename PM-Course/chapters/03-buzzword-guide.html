Chapter 3

Speaking Developer: A Buzzword Survival Guide

Every term you'll hear in this course, explained in plain English

Why You Need This Chapter

In Chapter 2, you learned to navigate your Mac with six Terminal commands. Before we move on to real tools, let's build your vocabulary. Developers use words that sound intimidating but describe surprisingly simple ideas.

This chapter is your phrasebook. No hands-on work — just read, absorb, and refer back whenever you hear a term you've forgotten. By the end, you'll be able to follow any conversation about Git, repositories, branches, or pull requests without getting lost.

There are about a dozen terms here. You do not need to memorize them all right now. You'll encounter each one in practice over the next chapters, and that's when they'll really stick. This chapter is a reference you can return to anytime.

IDE

IDE stands for Integrated Development Environment. That's a mouthful, but the concept is simple: it's an app where developers write code.

Think of Microsoft Word, but for code instead of essays. Word gives you spell-check, formatting, and a table of contents. An IDE gives developers similar helpers — highlighting errors, auto-completing words, and organizing files. Windsurf is the IDE you'll use in this course, and it also has a built-in AI assistant.

You won't be writing code in it — but you will use Windsurf to edit spec files, manage versions, and talk to the AI. We'll cover all of that in Chapter 4.

Repository (Repo)

A repository — almost always shortened to repo — is a project folder with a superpower: it remembers every change ever made to the files inside it.

A regular folder on your Mac just stores files. If you change a file, the old version is gone. A repo stores files and a complete history of every edit, every deletion, and every addition — along with who made the change and when.

Imagine a filing cabinet where every time you update a document, the old version doesn't get thrown away — it gets filed in a time-stamped drawer. You can open any drawer and see exactly what the document looked like last Tuesday. That's a repo.

Git

Git is the technology that makes repositories work. It's the engine that tracks all those changes behind the scenes.

You don't need to understand how Git works internally. Just know this: Git is the system that turns a regular folder into a repo (a folder with memory). Every developer on the planet uses Git. It was created in 2005, and it became the industry standard almost immediately.

If a repo is a filing cabinet with history, Git is the librarian who organizes everything. You hand the librarian your updated document, they file it, label it, and make sure you can always find any previous version.

In this course, you'll use Git entirely through Windsurf's visual interface — clicking buttons, not typing commands. You'll never need to type git in the Terminal.

Commit

A commit is a save point. When you make changes to files in a repo and you're happy with them, you commit those changes. This tells Git: "Remember this moment. Snapshot everything as it is right now."

Every commit has two parts:

  • The changes — which files were added, edited, or deleted
  • A message — a short description of what you did

Commit messages are written by you. They're like sticky notes on a filing cabinet drawer: "Added the login button," "Fixed the typo in the header," "Removed the old footer."

Good Commit Messages

Add specs for note creation feature Fix typo in proposal title Remove outdated color specs

Bad Commit Messages

stuff updates fix asdfghjkl

Good commit messages are short and specific. They help your future self (and your teammates) understand what each change was about without having to open the files.

Branch

A branch is a parallel copy of the project where you can make changes without affecting the original.

Every repo has a default branch, usually called main (or sometimes master). This is the "official" version of the project — the one everyone trusts. When you want to make changes, you create a new branch, do your work there, and only bring it back to main after it's been reviewed.

Imagine you're editing a shared Google Doc. Instead of typing directly into the original, you make a copy, do your edits on the copy, and then ask someone to review your changes before pasting them back. The copy is the branch. The original is main.

Branches are cheap and easy to create. You'll make one every time you start a new task. It's completely normal to have several branches at the same time — one per task you're working on.

Merge

Merging is the act of combining changes from one branch back into another. When your work on a branch is done and approved, it gets merged into main.

After the merge, your changes are now part of the official project. The branch you worked on has served its purpose — you can delete it without losing anything, because the changes are now in main.

Pull Request (PR)

A Pull Request — usually called a PR — is how you ask the team to review your changes before they get merged.

You don't just merge your branch into main on your own. You create a PR, which is basically a message that says: "Hey team, I made these changes on my branch. Please take a look, leave comments if something needs fixing, and approve when you're happy with it."

Think of submitting a document for approval. You write a memo, attach it to an email, and send it to your manager with the subject line "Please review the Q3 proposal." They read it, suggest a few edits, you make them, and they sign off. A PR is that process, but for code changes — and it all happens on BitBucket.

PRs are the heartbeat of teamwork in software. As a PM, you'll create PRs containing your specs — the descriptions of what to build. Developers will review your specs the same way they review code.

Fork

A fork is your personal copy of someone else's repository, hosted on BitBucket.

In your team's workflow, there's one "main" repository that everyone shares. You can't edit it directly — that would be too risky (imagine 20 people editing the same Google Doc at the same time with no review process). Instead, you create a fork: your own copy on BitBucket, where you can do whatever you want without affecting the original.

When you're ready, you create a PR from your fork to the main repository. The team reviews it and merges it. Your fork is your safe sandbox.

Clone

Cloning means downloading a repository from BitBucket to your Mac.

Your fork lives on BitBucket (a website). To actually work with the files, you need them on your computer. Cloning copies everything — the files, the folders, and the entire change history — to a folder on your Mac.

It's like downloading an email attachment. The original stays in the sender's outbox, and now you have a copy on your machine that you can edit. Except with cloning, your copy stays connected to the original — you can send your changes back when you're done.

Push and Pull

These two are a pair. They describe the direction files travel between your Mac and BitBucket.

TermDirectionWhat It Means
Push Your Mac → BitBucket Upload your latest commits to BitBucket so others can see them
Pull BitBucket → Your Mac Download the latest changes from BitBucket to your Mac

Think of it this way: push = send, pull = receive. You push your work up when you're done. You pull others' work down when you need the latest version.

BitBucket

BitBucket Server is a website hosted by your company where repositories live. It's where you browse code, create forks, submit pull requests, and review your teammates' work.

Your company runs its own BitBucket server — it's not a public website. You access it through your company's internal URL in a browser, and you log in with your company credentials.

If Git is the librarian and a repo is the filing cabinet, then BitBucket is the library building itself — the central place where everyone's filing cabinets live. You can walk in, browse what's there, and check out a copy to work on at your desk.

How It All Connects

Here's the big picture. This is the flow you'll follow as a PM:

Fork → Clone → Branch → Commit → Push → Pull Request → Merge

Let's walk through it as a story:

  1. You fork the team's repository on BitBucket — creating your own copy.
  2. You clone your fork to your Mac — downloading it so you can work on it.
  3. You create a branch — a safe space for your changes.
  4. You make changes and commit them — saving snapshots of your work.
  5. You push your branch to BitBucket — uploading your commits.
  6. You create a Pull Request — asking the team to review.
  7. After approval, your changes get merged into the main repository.

That's the entire workflow. Every step will be covered hands-on in Chapters 5 through 7. For now, just know the sequence exists.

The Big Analogy: A Shared Recipe Book

Tying It All Together

Imagine your team maintains a master recipe book in a locked glass case at the office (the main repository on BitBucket).

You photocopy the entire book (fork + clone). On your copy, you start a new section called "Desserts" (branch). You write a recipe for chocolate cake and mark it as final (commit).

You bring your photocopied section back to the office (push) and leave a note: "Hey team, I added a dessert section — take a look?" (Pull Request).

Your colleague reads the recipe, suggests adding a pinch of salt, you update it, and they approve. The new section gets inserted into the master book (merge).
