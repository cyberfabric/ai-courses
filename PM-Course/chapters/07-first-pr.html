Chapter 7

Pushing Code and Creating Your First PR

The full cycle: clone, branch, commit, push, and create a Pull Request on BitBucket

From Theory to Practice

In Chapter 6, you learned the concepts: forks, clones, branches, pushing, and Pull Requests. Now you're going to do every single step yourself.

By the end of this chapter, you will have:

  • Cloned a forked repository to your Mac
  • Created a new branch
  • Made a change and committed it
  • Pushed your branch to BitBucket
  • Created a real Pull Request

This is the workflow you'll use daily once you start working with OpenSpec. Let's walk through it step by step.

Think of this chapter as a test drive. You've read the manual (Chapter 6), and now you're getting behind the wheel. We'll go slowly, one turn at a time. By the end of the drive, you'll be comfortable with the route.

Starting Point: You Have a Fork

Before this chapter, you (or your team lead) should have already created a fork on BitBucket. If you haven't forked yet, here's a quick refresher:

  1. Open BitBucket Server in your browser (your company's internal URL)
  2. Navigate to the team's main repository
  3. Click the Fork button (usually near the top-right of the repository page)
  4. Accept the defaults and click Fork repository

After forking, BitBucket takes you to your new fork. The page looks the same as the main repo, but notice the name — it now shows your username, something like ~yourname/project-name. That's your personal copy.

Remember: you only fork once per project. If you've already forked this repository, just navigate to your existing fork on BitBucket.

Step 1: Clone Your Fork to Your Mac

Your fork lives on BitBucket (the server). To work with the files, you need to download them to your Mac. That's cloning.

Get the clone URL

  1. On your fork's page in BitBucket, look for the Clone button
  2. Click it — a dropdown appears with a URL
  3. Make sure HTTPS is selected (not SSH)
  4. Click the copy icon to copy the URL to your clipboard

The URL looks something like: https://bitbucket.yourcompany.com/scm/~yourname/project-name.git

You'll paste this URL into Windsurf in the next step.

Clone in Windsurf

  1. Open Windsurf
  2. Go to File → New Window (if you have another project open)
  3. Press Cmd + Shift + P to open the Command Palette — a search bar at the top of the window
  4. Type Git: Clone and select it
  5. Paste the URL you copied from BitBucket
  6. Choose a location on your Mac to save it (e.g., your Documents folder)
  7. Click Select Repository Location

The Command Palette (Cmd + Shift + P) is Windsurf's search-for-anything bar. You can type any action and it finds it for you. If you ever forget where a button is, try the Command Palette first. It's your safety net.

Windsurf downloads the repository. When it finishes, it asks: "Would you like to open the cloned repository?" Click Open. You should now see the project's files in the File Explorer (left panel), just like any folder you've opened before.

That's it — the project is on your Mac, connected to your fork on BitBucket. You only need to do this once per project.

Step 2: Create a New Branch

Before making any changes, create a branch. This keeps your work isolated and makes it easy to submit a clean Pull Request later.

  1. Look at the bottom-left corner of the Windsurf window
  2. You'll see a branch name (probably main or master)
  3. Click on it
  4. A dropdown appears — select Create new branch…
  5. Type a descriptive name for your branch
  6. Press Enter

The branch indicator in the bottom-left now shows your new branch name instead of main. You've switched to your new branch. All changes you make from now on will happen on this branch only.

Naming your branch

Branch names should be short and descriptive. Use hyphens between words, no spaces.

Good branch names

add-login-specs mob456-dark-mode fix-note-creation-spec

Bad branch names

my stuff test Branch 1 aaa

If your team uses ticket numbers (like MOB-456), include it in the branch name: mob456-add-dark-mode. This makes it easy to trace the branch back to the original ticket.

Step 3: Make a Change and Commit

This is the cycle from Chapter 5 — nothing new here. Let's do it:

  1. Create a new file or edit an existing one in the File Explorer
  2. Type some content and save with Cmd + S
  3. Open Source Control (branch icon in the left sidebar)
  4. Stage the change (click the +)
  5. Write a commit message
  6. Click Commit

You can make multiple commits on your branch before pushing. Each commit is a save point. Commit whenever you've reached a meaningful milestone, however small.

Think of commits like paragraphs in a document. Each one captures a complete thought. When you push, you're sending the whole document (all your commits) to the server at once.

Step 4: Push Your Branch to BitBucket

Your commits are local — only on your Mac. Now let's upload them to your fork on BitBucket.

Since this is a new branch that doesn't exist on BitBucket yet, Windsurf will offer to publish it:

  1. After committing, look at the Source Control panel
  2. You'll see a button that says Publish Branch
  3. Click it

Windsurf may ask for your BitBucket credentials the first time. Enter your company username and password (or a personal access token if your team uses those). This is a one-time setup — Windsurf remembers your credentials for future pushes.

If you see an authentication error, don't worry. The most common cause is a mistyped password. Try again carefully. If it still fails, ask a developer on your team — they can help you set up the right credentials in a few minutes.

After the push succeeds, your branch and all its commits are now on your fork on BitBucket. Anyone with access to your fork can see them.

For future pushes on the same branch (after you've published it once), the button changes to Sync Changes or shows an upload icon. Click it to push new commits you've made since the last push.

Step 5: Create a Pull Request on BitBucket

Your branch is on BitBucket. Now let's create a Pull Request to ask the team to review your changes.

  1. Open BitBucket Server in your browser
  2. Navigate to your fork
  3. BitBucket often shows a banner: "Your branch branch-name was recently pushed. Create a pull request." — click it
  4. If you don't see the banner, click Pull Requests in the sidebar, then Create pull request

Filling in the PR form

You'll see a form with these fields:

  • Source: your fork / your branch (should be pre-filled)
  • Destination: the main repository / main branch (should also be pre-filled)
  • Title: a short description of your change
  • Description: more details about what you did and why
  • Reviewers: people who should review your work

Fill in the form:

  • Title — short and descriptive, like a good email subject line: Add specs for note creation feature
  • Description — explain what you changed and why. A few sentences is enough.
  • Reviewers — add at least one team member (ask your team lead who to add if unsure)

Click Create.

You just created your first Pull Request!

Your changes are now formally submitted for review. The reviewers will be notified, and they can see exactly what you changed. This is how every change enters the main project — through a reviewed, approved Pull Request.

What Happens After You Create the PR

Now the ball is in the reviewers' court. Here's what to expect:

  1. Reviewers look at your changes — BitBucket shows them exactly which files you changed and what's different.
  2. They may leave comments — questions, suggestions, or "Looks good!" You'll see these on the PR page.
  3. You may need to make updates — if they ask for changes, go back to Windsurf, edit, commit, and push again. The PR updates automatically.
  4. They approve — once satisfied, reviewers click "Approve."
  5. Someone merges — after approval, a maintainer clicks "Merge." Your changes are now in the main repository.

If you need to make updates after creating the PR: just edit the files in Windsurf, commit, and push. You don't need to create a new PR. The existing PR automatically picks up any new commits you push to the same branch.

The Complete Flow at a Glance

Fork (once) → Clone (once) → Branch → Edit & Commit → Push → Create PR

Steps 1 and 2 are one-time setup. Your daily workflow is steps 3 through 6 — and it gets faster every time you do it.

Practice: Walk Through the Entire Cycle

Your mission: Do the full workflow from clone to Pull Request.

  1. If you haven't forked the practice repository yet, fork it on BitBucket
  2. Clone your fork in Windsurf (Cmd + Shift + P → "Git: Clone" → paste URL)
  3. Create a new branch: click the branch name in the bottom-left, then "Create new branch"
  4. Create a new file (e.g., my-practice-note.txt) and add some text
  5. Save (Cmd + S), stage, write a commit message, and commit
  6. Click Publish Branch to push
  7. Open BitBucket in your browser and create a Pull Request

If your PR shows up on BitBucket with your changes visible — you've completed the entire workflow. Everything from here on builds on this foundation.

Common Questions

"I don't see the Publish Branch button."

Make sure you've committed at least one change first. The button only appears when there's a new branch with commits ready to push.

"BitBucket says I don't have permission."

You might be trying to push to the main repository instead of your fork. Check that you cloned your fork's URL, not the main repo's URL. When in doubt, ask a developer to verify your remote URL.

"Can I create multiple PRs at the same time?"

Yes! Each PR is tied to a branch. If you have two branches, you can have two PRs open simultaneously. They're completely independent.

Always create a new branch for each task. Never commit directly to main. If you accidentally committed to main, ask a developer to help you move those commits to a new branch. It's an easy fix.
