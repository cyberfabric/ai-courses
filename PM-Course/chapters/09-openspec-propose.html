Chapter 9

Hands-On: Starting a Google Notes App with OpenSpec

Initialize OpenSpec, describe a feature, and watch AI generate your first proposal and specs

From Concepts to Creation

In Chapter 8, you learned what OpenSpec is and what your role looks like. Now you're going to do it. By the end of this chapter, you'll have a complete proposal and a set of behavioral specs for a Google Notes app — generated by AI, reviewed and refined by you.

What You'll Build

You'll describe a simple notes app inspired by Google Keep:

  • A grid of note cards on the main screen
  • Creating new notes with a title and body
  • Editing existing notes
  • Deleting notes
  • A light/dark theme toggle

Remember: you're not writing code. You're describing what the app should do in plain language. The AI and developers handle the implementation.

Think of yourself as the person ordering a meal at a restaurant. You're going to describe exactly what you want ("grilled salmon, medium, lemon sauce on the side") and the kitchen (AI + developers) will prepare it. Your job is to be clear and specific about the order.

Setting Up: Create a Fresh Project

Let's start with a clean folder for our Notes app.

  1. Open Windsurf
  2. Go to File → Open Folder
  3. Navigate to your Documents folder
  4. Create a new folder called google-notes-app (you can do this in Finder or in the dialog)
  5. Select and open that folder

You should see an empty project in Windsurf — the File Explorer on the left shows no files. That's perfect. This is a blank canvas.

Initializing OpenSpec

Before you can use OpenSpec commands, you need to initialize it in your project. This is a one-time setup, similar to how you initialized a Git repo in Chapter 5.

  1. Open the integrated terminal: press Ctrl + `
  2. Type the following command and press Enter:
openspec init

After running the command, look at the File Explorer. A new folder called openspec/ has appeared in your project. Expand it and you'll see:

  • changes/ — where your active work lives
  • specs/ — where finalized behavioral scenarios accumulate
  • archive/ — where completed work gets stored
  • config.yaml — OpenSpec settings (you won't need to edit this)

This is one of the few times you'll type a command in the terminal. OpenSpec initialization is a one-time thing per project. After this, most of your interaction will be through Cascade (the AI chat), not the terminal.

If you see an error like "command not found," OpenSpec might not be installed yet. Ask a developer on your team to help you install it — it's a quick setup.

Creating a New Change

In OpenSpec, every feature lives in its own change directory. Think of it as a folder dedicated to one specific feature or task. Let's create one for our Notes app.

You have two options. You can create the change directory via the terminal: 

openspec new change notes-app-base

Or you can let the AI create it for you in the next step (when you run /opsx:propose). Either way works. For learning purposes, let's do it explicitly in the terminal so you can see what happens.

A new folder appears inside openspec/changes/: 

openspec/
  changes/
    notes-app-base/    ← your new change directory
  specs/
  archive/

This is where the proposal and specs for this feature will live.

Generating Your Proposal and Specs

Now for the main event. You're going to describe the Notes app to Cascade and let it generate the proposal and specs for you.

  1. Open Cascade (the AI chat panel on the right side of Windsurf)
  2. Type the following command:
/opsx:propose

Cascade will ask you to describe the feature. This is where your PM skills shine. Describe the Notes app clearly and specifically:

What to Tell the AI

I want to build a simple notes app similar to Google Keep. Features: - A main screen showing a grid of note cards - Each note has a title and a body text - Users can create a new note by clicking an "Add Note" button - Users can click on a note card to edit its title and body - Users can delete a note by clicking a delete icon on the card - The app has a light/dark theme toggle in the header - Notes should persist (not disappear when you refresh the page)

Be specific about what the user sees and does. Don't worry about technical details.

Cascade starts working. You'll see it generating text in the chat. After a moment, it creates two things in your change directory:

  • proposal.md — the business case and feature overview
  • specs/ folder — with one or more spec files containing WHEN/THEN scenarios

The AI might take a minute to finish. You'll see it writing in real-time. Don't interrupt it — let it complete. If something goes wrong, you can always run /opsx:propose again.

Reviewing the Proposal

Open openspec/changes/notes-app-base/proposal.md in the editor by clicking on it in the File Explorer.

The proposal should include:

  • Context — why this feature exists
  • Goals — what it aims to achieve
  • Scope — what's included and what's not
  • Description — a summary of the feature

Read through it. Does it capture what you described? Does the "why" make sense? Is anything missing or inaccurate?

The proposal is a high-level document. It doesn't need to be perfect — it just needs to accurately describe the intent. The real detail lives in the specs.

Reviewing the Specs

Now open the files inside openspec/changes/notes-app-base/specs/. This is where the behavioral scenarios live — the heart of your work as a PM.

You should see WHEN/THEN scenarios for each feature. Something like:

Example Generated Specs

WHEN the user opens the app THEN a grid of existing note cards is displayed AND each card shows the note's title and a preview of the body text WHEN the user clicks the "Add Note" button THEN a new blank note card appears in the grid AND the cursor is placed in the title field WHEN the user types a title and body text in a note AND clicks outside the note or presses save THEN the note is saved with the entered content WHEN the user clicks the delete icon on a note card THEN the note is removed from the grid AND the note is permanently deleted

What to Look For

As you review, ask yourself:

  • Completeness — are all the features covered? Did the AI miss the theme toggle? The persistence?
  • Accuracy — do the scenarios match what you had in mind?
  • Edge cases — what happens if a user creates a note with no title? What happens with a very long note body?
  • Clarity — could a QA tester read these and know exactly what to test?

This review step is where your product expertise matters most. The AI is good at generating structure, but you know the product. You know what users expect. You know the edge cases from support tickets and user research. The AI drafts; you sharpen.

Refining the Specs

If something is missing or wrong, tell Cascade. You can type follow-up messages directly in the chat:

Good Refinement Requests

"Add a scenario for what happens when a note has no title — it should show 'Untitled Note' as a placeholder" "The delete action should ask for confirmation before deleting" "Add scenarios for the light/dark theme toggle"

Avoid These

"Make it better" "Add more stuff" "Use React and localStorage"

Be specific about what you want changed. The AI will update the spec files accordingly. You can go back and forth as many times as you need.

There's no limit to how many times you can refine. This is an iterative process. Think of it as editing a document together with a very fast co-author. You direct, the AI writes, you review, repeat.

Understanding Your Change Directory

Let's take a moment to see what's in your change directory now: 

openspec/
  changes/
    notes-app-base/
      proposal.md        ← The WHY and WHAT
      specs/
        note-management/
          spec.md        ← WHEN/THEN scenarios for notes
        theme/
          spec.md        ← WHEN/THEN scenarios for theme toggle
  specs/                 ← Empty for now (populated after archiving)
  archive/               ← Empty for now

The exact file names and structure may vary depending on what the AI generated. The important thing is: your proposal and specs are organized inside the change directory, ready for review.

Committing Your Work

You have a proposal and specs ready. Time to commit them using the same Git workflow from Chapter 5.

  1. Open Source Control (branch icon in the left sidebar)
  2. You'll see all the new files listed with U badges (Untracked)
  3. Stage all files (click + next to "Changes")
  4. Write a commit message: Add proposal and specs for notes app
  5. Click Commit

All files move from "Changes" to the commit history. Your proposal and specs are now saved as a Git snapshot. This is what you'd submit as PR #1 in a real workflow.

If you haven't initialized a Git repo yet for this project, Windsurf will prompt you to do so. Click Initialize Repository first (just like in Chapter 5), then stage and commit.

What You Just Did

Phase 1 — Complete

You just completed the PM's side of the OpenSpec workflow:

  • Initialized OpenSpec in a project
  • Created a change directory for a new feature
  • Used /opsx:propose to generate a proposal and specs
  • Reviewed the WHEN/THEN scenarios
  • Refined the specs with follow-up instructions
  • Committed everything to Git

In a real workflow, you'd push this to BitBucket and create PR #1. The team would review your specs, and after approval, a developer would pick up from here.

Your mission: Create the proposal and specs yourself.

  1. Open an empty folder in Windsurf
  2. Run openspec init in the integrated terminal
  3. Run /opsx:propose in Cascade
  4. Describe the Notes app features in your own words
  5. Review every scenario in the generated specs
  6. Ask Cascade to add at least one missing scenario
  7. Commit the proposal and specs

Take your time reviewing. The better your specs, the better the implementation will be. This is where your PM judgment makes the biggest difference.
