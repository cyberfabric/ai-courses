Chapter 12

The Full PM Workflow: From Ticket to PR

Everything you've learned, in one smooth end-to-end flow

The Moment It All Comes Together

You've learned the Terminal. You know Windsurf. You can track changes with Git, push to BitBucket, and create Pull Requests. You've written specs with OpenSpec and managed them like a pro.

Now let's put it all together into the workflow you'll use every day. This chapter walks through the complete process from the moment you receive a ticket to the moment your specs PR is submitted for review. One smooth flow, ten clear steps.

The Complete Workflow

1. Ticket → 2. Open Project → 3. Branch → 4. New Change → 5. Propose
6. Review → 7. Commit → 8. Push → 9. Create PR → 10. Review & Merge

Let's walk through each step with a concrete example. Imagine you've just received this ticket:

Ticket MOB-456: Add Dark Mode to the Notes App

As a user, I want to toggle between light and dark themes so the app is comfortable to use at night. The toggle should be in the header and the preference should persist across sessions.

Step 1: Receive the Ticket

You've read the ticket in your project tracker (Jira, Linear, etc.). Before opening any tools, take a moment to think about:

  • What are the key behaviors? — toggle between themes, persist preference
  • What should the user see? — a toggle in the header, colors changing
  • What are the edge cases? — first visit (no saved preference), switching mid-use

This thinking happens before you touch any tool. It's the PM part — and it's the most valuable part.

Step 2: Open the Project in Windsurf

Open Windsurf and open your cloned project folder (File → Open Folder). If this is your first time, you'll need to clone your fork first (see Chapter 7).

Windsurf remembers recently opened folders. You can usually find your project in File → Open Recent without navigating to it manually.

Step 3: Create a Branch

Click the branch name in the bottom-left corner of Windsurf, select Create new branch, and name it: 

mob456-add-dark-mode

The bottom-left now shows mob456-add-dark-mode instead of main. You're on your new branch, ready to work.

Remember: one branch per ticket, always matching the ticket number.

Step 4: Create a New OpenSpec Change

Open the integrated terminal (Ctrl + `) and create a change directory: 

openspec new change mob456-add-dark-mode

This creates openspec/changes/mob456-add-dark-mode/ — the workspace for this ticket's specs.

Step 5: Propose Specs with AI

Open Cascade (the AI chat) and type: 

/opsx:propose

When Cascade asks you to describe the feature, use your thinking from Step 1:

Your Feature Description

Add a dark mode toggle to the Notes app. - A toggle button in the header switches between light and dark themes - In dark mode, the background is dark and text is light - In light mode, the background is light and text is dark - The toggle icon changes: moon icon for light mode, sun icon for dark mode - The user's theme preference is saved and persists when they reopen the app - On first visit, the app defaults to light mode

Cascade generates proposal.md and specs/ for this feature.

Step 6: Review and Refine

Open the generated files and review them carefully:

Check the proposal

  • Does the "why" reflect the ticket's intent?
  • Is the scope correct — not too broad, not too narrow?

Check the specs

  • Is every behavior covered? (toggle, visual changes, persistence, default)
  • Are there edge cases? (What if localStorage is cleared? What about system theme preferences?)
  • Are the scenarios testable by a QA person?
  • Are they free of technical language?

This is like proofreading a contract before sending it to a client. The AI drafted it, but you're the one who signs off. Read every scenario and ask: "Is this exactly what I want the dev to build?"

If something is missing, tell Cascade: 

"Add a scenario for what happens when the user has no saved preference — should default to light mode"

Run openspec validate in the terminal to check for structural issues.

Step 7: Commit

Once you're happy with the proposal and specs:

  1. Open Source Control (branch icon in sidebar)
  2. Stage all changes (+)
  3. Write a message: Add specs for dark mode (MOB-456)
  4. Click Commit

Include the ticket number in your commit message. This creates another traceability link: anyone reading the commit history can jump straight to the ticket.

Step 8: Push to BitBucket

Click Publish Branch (or the sync/upload icon if you've pushed before). Your branch and commits are now on your fork on BitBucket.

Step 9: Create a Pull Request

  1. Open BitBucket Server in your browser
  2. Navigate to your fork
  3. Click the banner to create a PR (or go to Pull Requests → Create)
  4. Fill in the form:
FieldWhat to Write
Title MOB-456: Add dark mode specs
Description Proposal and behavioral specs for the dark mode feature. Covers theme toggle, visual changes, persistence, and default behavior.
Reviewers Your team lead + at least one other team member

Click Create.

Your PR is now live on BitBucket. Reviewers can see your proposal and specs, leave comments, and approve when they're satisfied. This is your Phase 1 delivery.

Step 10: Team Review and Merge

After you create the PR:

  1. Reviewers read your specs and leave comments
  2. You address feedback: edit in Windsurf, commit, push — the PR updates automatically
  3. Once approved, the PR gets merged into the main repository
  4. A developer picks up the change for Phase 2 (design + tasks + implementation)

Your part is done. The developer now has clear, behavioral specs to work from.

What to Look for in the Developer's PR #2

After the developer finishes implementation, they'll submit PR #2. As a PM, you can review it to check:

  • Were all your specs implemented? — compare the features against your WHEN/THEN scenarios
  • Does the behavior match? — if you can see a preview or demo, test the scenarios yourself
  • Were specs archived correctly? — the behavioral scenarios should be in specs/

You don't need to review the code in PR #2 — that's for other developers. Focus on whether the implemented behavior matches your specs.

Tips for Writing Great Specs

Do This

Be specific about what the user sees Include edge cases and defaults Use exact text for messages and labels Think about first-time vs returning users Validate before committing

Avoid This

Vague descriptions ("make it look nice") Skipping edge cases Technical implementation details Assuming the developer "just knows" Rushing without reviewing

Common PM Mistakes (and How to Avoid Them)

MistakeWhy It HappensHow to Fix It
Specs are too vague Rushing through review Read each scenario and ask: "Could a QA tester verify this?"
Missing edge cases Only thinking about the happy path Ask: "What if the user does something unexpected?"
Technical language in specs Copying developer suggestions verbatim Run openspec validate and rewrite in user terms
Forgetting to commit before pushing Excitement to create the PR Check Source Control — if there are staged changes, commit first
Committing to main instead of a branch Forgetting to create a branch at the start Always check the bottom-left corner before starting work

Chapter 12 — Complete

You've just walked through the complete PM workflow from start to finish. This is what your work looks like from now on:

  1. Receive ticket
  2. Open project in Windsurf
  3. Create branch
  4. Create OpenSpec change
  5. Propose specs with /opsx:propose
  6. Review and refine
  7. Commit
  8. Push
  9. Create PR on BitBucket
  10. Address review feedback, get merged

It gets easier every time. By the third or fourth ticket, you'll move through these steps without even thinking about them.

Your mission: Walk through the entire workflow with a practice ticket.

Pretend you received this ticket:

MOB-789: Add a search bar to filter notes by title

  1. Create branch mob789-search-notes
  2. Create change openspec new change mob789-search-notes
  3. Run /opsx:propose and describe the search feature
  4. Review and refine the specs
  5. Run openspec validate
  6. Commit with message: Add specs for note search (MOB-789)
  7. Push and create a PR

If you can do all seven steps without looking back at earlier chapters — congratulations, you've internalized the workflow.
