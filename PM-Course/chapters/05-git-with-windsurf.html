Chapter 5

Your First Git Journey (No Terminal Required)

Initialize a repo, track changes, and commit — all from Windsurf's visual interface

Time to Turn On Change Tracking

In Chapter 4, you opened a project folder in Windsurf. Right now, that folder is just a regular folder — if you change a file, the old version is gone forever.

In this chapter, you'll turn that folder into a Git repository — a folder that remembers every change. And you'll do the whole thing through Windsurf's visual interface. No typing Git commands. Just clicking buttons.

Think of it like turning on "Track Changes" in Google Docs. Before you turn it on, edits just happen and nobody keeps a record. After you turn it on, every edit is logged: who changed what, when, and you can see the full history. Initializing a Git repo is exactly that — flipping the "Track Changes" switch for your entire project folder.

Finding the Source Control Panel

Windsurf has a dedicated panel for Git. It's called Source Control, and it's how you'll do all your Git work.

  1. Look at the left sidebar — the vertical strip of icons
  2. Find the icon that looks like a branching line (like a Y-shaped road)
  3. Click it

The left panel switches from the File Explorer to the Source Control view. If your folder isn't a Git repo yet, you'll see a message like "No source control providers registered" or a button that says Initialize Repository.

You can switch between the File Explorer and Source Control anytime by clicking their icons in the sidebar. The keyboard shortcut for Source Control is Ctrl + Shift + G.

Initializing Your Repository

This is the moment you flip the switch. Click the Initialize Repository button.

After clicking, the Source Control panel changes. Instead of the "Initialize" message, you now see a list of files under a section called Changes. Each file has a letter badge next to it:

U — next to every file in your folder.

That U stands for Untracked. It means Git now knows your folder is a repo, but it hasn't started tracking these specific files yet. They're new to Git. It's like saying: "I see these files, but you haven't told me to pay attention to them yet."

Nothing looks different in Finder. Your files are exactly where they were. Initializing a repo doesn't move, rename, or change any files. It just creates a hidden folder called .git inside your project that Git uses to keep track of history. You'll never need to touch that hidden folder.

The Three Badges You'll See

From now on, Git will show letter badges next to files that have changed. There are only three you need to know:

BadgeMeaningWhat Happened
U Untracked This is a brand-new file Git hasn't seen before
M Modified You edited a file that Git was already tracking
D Deleted You deleted a file that Git was tracking

These badges appear in both the Source Control panel and next to filenames in the File Explorer. They're Git's way of telling you: "Something changed since the last save point."

Staging: Telling Git What to Include

Before you can save a snapshot (a commit), you need to tell Git which changes to include. This step is called staging.

Think of it like packing a box for shipping. You've made several changes to different files, but maybe you only want to save some of them right now. Staging is putting the files you want into the box. Committing is sealing the box and labeling it.

To stage a file in Windsurf:

  1. In the Source Control panel, hover over a file name
  2. Click the + (plus) icon that appears

The file moves from the Changes section to a new section called Staged Changes. This means: "This file is ready to be included in the next commit."

To stage all changed files at once, hover over the Changes header and click the + icon there. This stages everything in one click.

If you accidentally stage a file, hover over it in the Staged Changes section and click the − (minus) icon to unstage it. No harm done.

Your First Commit

Now for the exciting part: your first save point. Let's walk through the complete flow.

Step 1: Stage your files

In the Source Control panel, stage all files by clicking the + next to the "Changes" header. All your files should move to the "Staged Changes" section.

Step 2: Write a commit message

At the top of the Source Control panel, there's a text box that says "Message" (or similar placeholder text). Click inside it and type a short description of what you're saving: 

Add initial project files

This is your commit message. It should describe what you did in one short sentence. Future-you (and your teammates) will read these messages to understand the project's history.

Step 3: Click Commit

Click the Commit button (or press Cmd + Enter).

The Staged Changes section disappears. The Source Control panel shows no more pending changes. Your files are committed — Git has taken a snapshot of everything as it is right now.

You just made your first commit!

That's a real milestone. You told Git "remember this moment," and it will. You can now change, break, or delete anything — and always get back to this exact state.

Editing a File: The "M" Badge

Now let's see what happens when you edit a tracked file.

  1. Switch to the File Explorer (click the top icon in the sidebar)
  2. Open one of your text files by clicking on it
  3. Add a new line of text: This file was edited after the first commit.
  4. Save with Cmd + S

Look at the file name — both in the File Explorer and in the Source Control panel. You'll see an M badge next to it. Git noticed you modified the file. It's telling you: "This file is different from the last commit."

Now commit this change, just like before:

  1. Open Source Control (branch icon in sidebar)
  2. Stage the modified file (click the +)
  3. Type a message: Update notes with new text
  4. Click Commit

Done. You now have two commits in your history — two snapshots of your project at two different moments.

Deleting a File: The "D" Badge

Let's see what happens when you delete a tracked file.

  1. In the File Explorer, right-click on a file
  2. Select Delete (or Move to Trash)
  3. Confirm the deletion

In the Source Control panel, the deleted file appears with a D badge. Git is saying: "This file existed in the last commit, but it's gone now. Do you want me to record this deletion?"

To commit the deletion:

  1. Stage the deleted file (click +)
  2. Type a message: Remove unused file
  3. Click Commit

The file is gone from your project, but Git remembers it existed. If you ever needed it back, you could find it in your commit history. Nothing is truly lost in Git.

The Pattern: Change → Stage → Message → Commit

You've now done it three times. Every time you work with Git, it's the same four-step cycle:

Make changes → Stage → Write message → Commit

That's it. Whether you're adding a single file or editing twenty files at once, the pattern is always the same. It will become second nature very quickly.

It's like the save cycle in a video game. You play (make changes), decide it's a good moment to save (stage), name your save file "Before the boss fight" (write a message), and hit save (commit). If things go wrong later, you can always load that save point.

Viewing Your Commit History

You've made three commits now. Let's see the timeline.

In Windsurf, you can view your commit history through the Source Control panel. Look for a section or link labeled commits, history, or a clock icon. Click it to see a list of all your commits, newest first.

You should see something like: 

Remove unused file
Update notes with new text
Add initial project files

Each line is one commit — one snapshot. The most recent is at the top. You can click on any commit to see exactly what changed in that snapshot.

This is the history that makes Git so powerful. Every change, every deletion, every addition is recorded with a message, a timestamp, and your name. Weeks from now, you (or your teammates) can look at this timeline and understand exactly what happened and when.

Practice: The Full Cycle

Your mission: Go through the complete Git cycle three times.

  1. Commit #1 — Create: Create a new file called shopping-list.txt in the File Explorer. Type a few items inside. Save. Stage. Write message: Add shopping list. Commit.
  2. Commit #2 — Edit: Open shopping-list.txt and add more items. Save. Notice the M badge. Stage. Write message: Add more items to shopping list. Commit.
  3. Commit #3 — Delete: Delete shopping-list.txt. Notice the D badge. Stage. Write message: Remove shopping list. Commit.

Now check your commit history. You should see three new entries, each with the message you wrote. If you can see them — you've mastered the Git basics.

Common Questions

"What if I forget to stage before committing?"

Windsurf will warn you. If there's nothing in Staged Changes, the Commit button won't do anything (or it may offer to stage everything for you automatically). Either way, no harm done.

"How often should I commit?"

Whenever you reach a meaningful stopping point. Finished writing a set of specs? Commit. Added a new file? Commit. Fixed a typo? Commit. There's no such thing as committing too often. Small, frequent commits are better than one giant commit at the end of the day.

"Can I undo a commit?"

Yes, but let's not worry about that now. For this course, just know that commits are safe — they record history, they don't erase it. We'll cover troubleshooting in Chapter 13.

Always save your files (Cmd + S) before staging and committing. Git tracks what's saved on disk, not what's showing in the editor. If you see a white dot on the file tab, the file has unsaved changes that Git won't pick up yet.
